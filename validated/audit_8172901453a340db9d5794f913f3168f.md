### Title
ICP Permanently Locked in CMC Subaccount When `notify_*` Is Not Called Before Block Purge — (File: rs/nns/cmc/src/main.rs)

---

### Summary

The Cycles Minting Canister (CMC) implements a two-step flow for converting ICP to cycles. In step 1, a user sends ICP to a CMC-controlled subaccount. In step 2, the user calls `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` referencing the ledger block index. If the user fails to complete step 2 before the block index is purged from the `blocks_notified` map (which occurs when the map exceeds `MAX_NOTIFY_HISTORY = 1,000,000` entries), the ICP is permanently locked in the CMC subaccount with no recovery mechanism.

---

### Finding Description

The CMC maintains a `BTreeMap<BlockIndex, NotificationStatus>` called `blocks_notified` to prevent double-spending of ICP payments. To bound memory, `purge_old_notifications` is called on every `notify_*` invocation and removes the oldest entries when the map exceeds `MAX_NOTIFY_HISTORY`: [1](#0-0) 

```
const MAX_NOTIFY_HISTORY: usize = 1_000_000;
const MAX_NOTIFY_PURGE: usize = 100_000;
``` [2](#0-1) 

Once a block index is purged, `last_purged_notification` advances monotonically. Any subsequent `notify_*` call for that block index hits the guard: [3](#0-2) 

```rust
if block_index <= state.last_purged_notification {
    return Some(Err(NotifyError::TransactionTooOld(
        state.last_purged_notification + 1,
    )));
}
```

The same guard exists in `notify_create_canister` and `notify_mint_cycles`: [4](#0-3) [5](#0-4) 

`TransactionTooOld` is explicitly documented as **non-retriable** in the interface: [6](#0-5) 

The ICP transferred in step 1 sits in the CMC subaccount (e.g., `AccountIdentifier::new(CMC_ID, Some(Subaccount::from(&canister_id)))`) indefinitely. The CMC has no heartbeat, no admin endpoint, and no public API to sweep or refund ICP from subaccounts whose corresponding block index has been purged. The recently added automatic-refund mechanism (`issue_automatic_refund_if_memo_not_offerred`) only triggers for **non-meaningful memos**; it does not help when the memo is correct (e.g., `MEMO_TOP_UP_CANISTER`) but `notify_*` was never called: [7](#0-6) 

```rust
if MEANINGFUL_MEMOS.contains(&memo) {
    // Not eligible for refund.
    return Ok(());
}
``` [8](#0-7) 

---

### Impact Explanation

Any ICP transferred to a CMC subaccount with a meaningful memo (`MEMO_TOP_UP_CANISTER`, `MEMO_CREATE_CANISTER`, `MEMO_MINT_CYCLES`) for which `notify_*` is never called — or is called only after the block index is purged — is **permanently locked** in the CMC subaccount. There is no on-chain recovery path: no withdrawal endpoint, no sweep mechanism, no governance-free refund. The funds are not burned (they remain in the subaccount) but are inaccessible to the original sender and to the CMC itself without a canister upgrade.

The analog to the Celo M01 class is exact: a two-step commit-then-complete flow where the commit (ICP transfer) is irreversible and the complete step (notify call) has an effective expiry (block purge), after which the committed funds are unrecoverable.

Unlike the Celo case, there is no stated security rationale for locking the ICP. The Celo protocol intentionally burned attestation fees to deter validator spam. The CMC's locking is a side-effect of bounding `blocks_notified` memory, not a deliberate anti-abuse measure.

---

### Likelihood Explanation

**Passive path (user error):** A user sends ICP to a CMC subaccount, loses the block index, or simply forgets to call `notify_*`. If 1,000,000 other notify calls accumulate before the user retries, the block is purged. On a high-throughput mainnet this window shrinks over time as `last_purged_notification` advances.

**Active path (griefing):** An adversary who controls many principals can submit 1,000,000 ICP transfers to distinct CMC subaccounts (each costing only the ledger transfer fee, ~0.0001 ICP) and call `notify_*` for each, advancing `last_purged_notification` by up to `MAX_NOTIFY_PURGE = 100,000` per call. This can be used to deliberately push a victim's block index below `last_purged_notification`, permanently locking the victim's ICP. The cost is bounded by 1,000,000 × 0.0001 ICP = 100 ICP plus execution fees — a realistic griefing budget for high-value targets. [9](#0-8) 

---

### Recommendation

1. **Add a subaccount-sweep endpoint**: Allow the original sender (identified by the subaccount derivation) to reclaim ICP from a CMC subaccount whose block index is `TransactionTooOld`, after verifying no successful notification was ever recorded for that subaccount.
2. **Extend the automatic-refund path**: When `fetch_transaction` is called for a block with a meaningful memo that has already been purged, attempt to refund the subaccount balance to the sender rather than returning `TransactionTooOld` with no recourse.
3. **Document the locking behavior**: At minimum, add explicit documentation to the DID and user-facing docs that ICP sent to a CMC subaccount is at risk of permanent loss if `notify_*` is not called promptly.

---

### Proof of Concept

```
1. Alice sends 10 ICP to AccountIdentifier::new(CMC_ID, Some(Subaccount::from(&canister_id)))
   with memo = MEMO_TOP_UP_CANISTER. Block index = B.

2. Alice loses her block index or is delayed.

3. Adversary (or organic traffic) submits 1,000,000 ICP transfers to distinct CMC
   subaccounts and calls notify_top_up for each, advancing last_purged_notification past B.

4. Alice calls notify_top_up({block_index: B, canister_id}).
   → CMC returns Err(NotifyError::TransactionTooOld(last_purged_notification + 1)).

5. Alice's 10 ICP remains in the CMC subaccount indefinitely.
   No public API exists to recover it.
```

```rust
    fn purge_old_notifications(&mut self, max_history: usize) {
        let mut last_purged = 0;
        let mut cnt = 0;
        // Remove elements from the beginning of self.blocks_notified until either
        // it is small enough, or MAX_NOTIFY_PURGE entries have been removed.
        while self.blocks_notified.len() > max_history && cnt < MAX_NOTIFY_PURGE {
            // pop_first is nightly only
            let block_height = *self.blocks_notified.iter().next().unwrap().0;
            self.blocks_notified.remove(&block_height);
            last_purged = block_height;
            cnt += 1;
        }
        // make sure this grows monotonically (a delayed callback might have added older status)
        self.last_purged_notification = last_purged.max(self.last_purged_notification);
    }
```
