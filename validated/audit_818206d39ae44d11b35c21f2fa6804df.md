### Title
Unrestricted `deploy_erc20_token` Allows Any NEAR Account to Permanently Hijack the NEP-141→ERC-20 Token Registry - (File: engine/src/contract_methods/connector.rs)

---

### Summary

The `deploy_erc20_token` NEAR contract entrypoint has no caller access control. Any NEAR account can invoke it with an arbitrary NEP-141 account ID, causing the engine to deploy a standard ERC-20 contract and permanently write the NEP-141→ERC-20 mapping into storage. Because `register_token` rejects re-registration with `TokenAlreadyRegistered` and no removal path exists, a malicious actor can front-run the legitimate deployment of any NEP-141 token's ERC-20 mirror, permanently preventing the legitimate deployer from registering the token at the expected ERC-20 address.

---

### Finding Description

The public NEAR entrypoint `deploy_erc20_token` in `engine/src/lib.rs` dispatches directly to `contract_methods::connector::deploy_erc20_token` with no access guard:

```rust
#[unsafe(no_mangle)]
pub extern "C" fn deploy_erc20_token() {
    let io = Runtime;
    let env = Runtime;
    let mut handler = Runtime;
    contract_methods::connector::deploy_erc20_token(io, &env, &mut handler)
        .map_err(ContractError::msg)
        .sdk_unwrap();
}
``` [1](#0-0) 

Inside `connector::deploy_erc20_token`, the only gate is `require_running` (contract not paused). There is no `require_owner_only` or any other caller check:

The `engine::deploy_erc20_token` function then calls `engine.register_token(address, nep141)`, which inserts the NEP-141→ERC-20 mapping into `nep141_erc20_map`:

```rust

    &mut self,
    erc20_token: Address,
    nep141_token: AccountId,
) -> Result<(), RegisterTokenError> {
    match get_erc20_from_nep141(&self.io, &nep141_token) {
        ...
        Ok(_) => return Err(RegisterTokenError::TokenAlreadyRegistered),
    }
    nep141_erc20_map(self.io).insert(&nep141_token, &erc20_token);

``` [3](#0-2) 

Once a NEP-141 token is registered, it can **never** be re-registered. There is no `unregister_token` or `update_token_mapping` function anywhere in the codebase.

The `WithMetadata` branch of `deploy_erc20_token` even contains an explicit comment acknowledging the intended restriction — "this transaction could be executed by the owner of the contract only" — but this is never enforced in either branch: [4](#0-3) 

The `require_owner_only` helper exists and is used by other sensitive functions (e.g., `mirror_erc20_token`, `set_erc20_metadata`, `attach_full_access_key`), but is absent here: [5](#0-4) 

---

### Impact Explanation

An attacker can:

1. Call `deploy_erc20_token` with `Legacy("target-nep141.near")` from any NEAR account before the legitimate deployer.
2. The engine deploys a standard `EvmErc20.bin` contract at an address determined by the attacker's EVM nonce and permanently stores the mapping `target-nep141.near → attacker_chosen_address`.
3. The legitimate deployer's subsequent call fails with `TokenAlreadyRegistered`.
4. Because no removal path exists, the legitimate deployer can **never** register the token at the expected ERC-20 address.
5. All users who bridge `target-nep141.near` to Aurora receive tokens at the attacker-registered address rather than the expected one, breaking integrations, front-ends, and downstream contracts that rely on the canonical ERC-20 address.

The deployed ERC-20 is always the standard Aurora ERC-20 with the engine as admin, so direct theft of bridged tokens is not possible. However, the permanent corruption of the NEP-141→ERC-20 registry constitutes a **permanent freezing of the bridging path** for any targeted token: the legitimate deployer is permanently locked out, and the ecosystem-wide canonical address is permanently wrong.

---

### Likelihood Explanation

The attack requires no special privileges, no tokens, and no gas beyond a standard NEAR transaction fee. Any NEAR account can call `deploy_erc20_token` with any NEP-141 account ID at any time. A motivated attacker can monitor the NEAR mempool for legitimate `deploy_erc20_token` calls and front-run them, or pre-emptively register any NEP-141 token that has not yet been bridged to Aurora.

---

### Recommendation

Add `require_owner_only` to `deploy_erc20_token`, consistent with the existing comment and with the pattern used by `mirror_erc20_token` and other sensitive functions:

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?; // ADD THIS
        ...
    })
}

### Proof of Concept

1. Attacker observes that `"usdc.near"` has not yet been registered on Aurora.
2. Attacker calls `deploy_erc20_token` with `Legacy("usdc.near")` from any NEAR account.
3. Engine deploys ERC-20 at address `X` (determined by attacker's EVM nonce) and stores `usdc.near → X` permanently via `nep141_erc20_map`.
4. Legitimate deployer calls `deploy_erc20_token("usdc.near")` → `register_token` returns `Err(TokenAlreadyRegistered)` → transaction fails.
5. No recovery path exists; `usdc.near` is permanently mapped to `X`.
6. All subsequent `ft_on_transfer` calls from `usdc.near` mint tokens at address `X`, not the expected canonical address, breaking all downstream integrations.

This is the direct analog of the Bribe report: a public function (`deploy_erc20_token`) with no access control calls an internal state-mutating function (`register_token`) that should only be callable by the owner, allowing any unprivileged caller to permanently corrupt a privileged registry.
