Audit Report

## Title
Unbounded `BouncerValue::Wants` Window in `CertifierBouncer::new_bouncer` Enables Unvalidated Pool Flooding — (`rs/consensus/certification/src/certifier.rs`)

## Summary
`CertifierBouncer::new_bouncer` returns `BouncerValue::Wants` for every `CertificationMessageId` at any height ≥ `cup_height` not already in `certified_heights`, with no upper-bound guard. Combined with `SLOT_TABLE_NO_LIMIT = usize::MAX` for the certification channel and an unvalidated pool with no size cap, a single Byzantine subnet peer can flood the unvalidated certification pool with syntactically valid but cryptographically invalid artifacts at arbitrarily large heights, causing unbounded memory growth and eventual OOM-induced replica crash.

## Finding Description

**1. Unbounded bouncer window** (`rs/consensus/certification/src/certifier.rs`, L86–95):
```rust
Box::new(move |id| {
    let height = id.height;
    if height < cup_height || certified_heights.contains(&height) {
        BouncerValue::Unwanted
    } else {
        BouncerValue::Wants   // no upper-bound check
    }
})
```
The consensus bouncer caps acceptance at `next_cup_height + ACCEPTABLE_NOTARIZATION_CUP_GAP` and uses `LOOK_AHEAD = 10` (`rs/consensus/src/consensus/priority.rs`, L36, L57–59). The IDkg bouncer uses `finalized_height + LOOK_AHEAD`. The certification bouncer has no equivalent guard.

**2. No slot-table limit for certification** (`rs/replica/setup_ic_network/src/lib.rs`, L75, L268):
```rust
const SLOT_TABLE_NO_LIMIT: usize = usize::MAX;
// ...
new_p2p_consensus.abortable_broadcast_channel(assembler, SLOT_TABLE_NO_LIMIT)
```
The slot-table enforcement in `rs/p2p/consensus_manager/src/receiver.rs` (L373) only drops new entries when `peer_slot_table_len >= slot_limit`; with `usize::MAX` that check never fires.

**3. Unvalidated pool has no size limit** (`rs/artifact_pool/src/certification_pool.rs`, L259–271): `insert` unconditionally inserts into an in-memory `BTreeMap` with no capacity check.

**4. Certifier only validates heights the state manager requests** (`rs/consensus/certification/src/certifier.rs`, L214–244): `on_state_change` calls `validate` only over `state_heights_to_aggregate_and_validate`, derived from `list_state_hashes_to_certify`/`list_state_heights_to_certify`. Artifacts at heights far above the execution tip are never validated and never removed from the unvalidated pool.

**5. Purger only removes artifacts *below* `cup_height − MINIMUM_CHAIN_LENGTH`** (`rs/consensus/certification/src/certifier.rs`, L463–474): `RemoveAllBelow(cup_height − MINIMUM_CHAIN_LENGTH)` never touches artifacts at heights far *above* `cup_height`.

**6. Download loop retries indefinitely** (`rs/p2p/artifact_downloader/src/fetch_artifact/download.rs`, L216–219): `with_max_elapsed_time(None)`. Download tasks for future-height adverts never become `Unwanted` because the bouncer keeps returning `Wants`.

**Exploit flow:**
1. Byzantine peer creates N `CertificationShare` messages at heights H >> `cup_height` (signed with its own legitimate key — no forgery needed).
2. Computes their hashes, advertises them as slot updates on slots 1…N.
3. With `SLOT_TABLE_NO_LIMIT`, the slot table never blocks new entries; N download tasks are spawned.
4. Peer serves the messages when fetched; each passes the `message.id() == id` check (peer pre-computed the hash) and is delivered via `UnvalidatedArtifactMutation::Insert`.
5. The certifier never validates or purges these artifacts; the unvalidated `BTreeMap` and `HeightIndex` grow without bound → OOM → replica crash.

## Impact Explanation
A single Byzantine subnet peer (below the f < n/3 fault threshold) can crash all replicas on a subnet by exhausting their memory. This matches the **High** impact category: *Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS* ($2,000–$10,000). The attack targets the protocol layer, not raw bandwidth.

## Likelihood Explanation
- Requires one compromised subnet node below the fault threshold — explicitly in scope.
- No threshold-majority corruption needed; a single node suffices.
- No cryptographic forgery required; the attacker uses its own legitimate signing key.
- The attack is mechanically simple: send many slot-update messages with distinct slot numbers and respond to fetch requests with crafted artifacts.
- Repeatable and can target all replicas on the subnet simultaneously.

## Recommendation
1. **Add an upper-bound guard to `CertifierBouncer::new_bouncer`** analogous to the consensus and IDkg bouncers. Return `MaybeWantsLater` for heights beyond `cup_height + LOOK_AHEAD` (or a certification-specific constant).
2. **Set a finite `SLOT_TABLE_LIMIT_CERTIFICATION`** (similar to `SLOT_TABLE_LIMIT_INGRESS = 50_000`) for the certification channel in `rs/replica/setup_ic_network/src/lib.rs`.
3. **Add a size cap to the unvalidated certification pool** in `CertificationPoolImpl::insert` and drop insertions that exceed it.
4. **Periodically validate and discard unvalidated artifacts at heights far above the current execution tip**, rather than waiting for the state manager to request them.

## Proof of Concept
```rust
// Unit test confirming unbounded Wants window
let cup_height = Height::from(0);
let certified_heights: HashSet<Height> = HashSet::new();
let bouncer = move |id: &CertificationMessageId| {
    let h = id.height;
    if h < cup_height || certified_heights.contains(&h) {
        BouncerValue::Unwanted
    } else {
        BouncerValue::Wants
    }
};
for h in [1u64, 1_000, 1_000_000, u64::MAX - 1] {
    let id = CertificationMessageId {
        height: Height::from(h),
        hash: CertificationMessageHash::CertificationShare(
            CryptoHashOf::from(CryptoHash(vec![]))
        ),
    };
    assert_eq!(bouncer(&id), BouncerValue::Wants);
}
// All pass: window is truly unbounded.
```
Integration test: configure a local subnet with a malicious peer that advertises `N = 100_000` slot updates at heights `cup_height + 1_000_000` through `cup_height + 1_100_000`, serves valid-ID `CertificationShare` artifacts, and observe unvalidated pool size growing monotonically with no purge.
