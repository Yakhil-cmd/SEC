Audit Report

## Title
ckERC20 Sent Transaction Resubmission Failure Silently Freezes Minter Withdrawal Pipeline — (`rs/ethereum/cketh/minter/src/withdraw.rs`, `rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

## Summary
Once a ckERC20 withdrawal transaction has been signed and placed in `sent_tx`, a gas price spike above the user's pre-paid `max_transaction_fee` causes `resubmit_transactions_batch` to receive a `ResubmitTransactionError::InsufficientTransactionFee` that is only logged — no reimbursement is emitted and the transaction is never removed from `sent_tx`. Because `create_resubmit_transactions` returns immediately on the first such error, every subsequent pending withdrawal (all nonces > N) is also blocked at the Ethereum level, freezing the entire minter withdrawal pipeline until gas prices fall below the original cap.

## Finding Description

**Root cause — two distinct code paths, only one fixed:**

A prior fix (`f420b4d6e`, shipped in `minter_upgrade_2024_08_05.md`) addressed the *pre-send* case: when `create_transactions_batch` receives `CreateTransactionError::InsufficientTransactionFee` for a request still in `pending_withdrawal_requests`, it calls `reschedule_withdrawal_request` to move it back to the queue. [1](#0-0) 

The *post-send* case is unaddressed. Once a transaction is in `sent_tx`, `resubmit_transactions_batch` handles the equivalent error with only a log statement:

```rust
Err(e) => {
    log!(INFO, "Failed to resubmit transaction: {e:?}");
}
``` [2](#0-1) 

**Why the cap is hit:** For `WithdrawalRequest::CkErc20`, `record_created_transaction` stores `ResubmissionStrategy::GuaranteeEthAmount { allowed_max_transaction_fee: ckerc20.max_transaction_fee }`. During resubmission, `SignedTransactionRequest::resubmit` enforces a hard ceiling:

```rust
if new_tx_price.max_transaction_fee() > self.resubmission.allowed_max_transaction_fee() {
    return Err(ResubmitTransactionError::InsufficientTransactionFee { ... });
}
``` [3](#0-2) 

ckETH withdrawals use `ResubmissionStrategy::ReduceEthAmount` and can always absorb higher fees by reducing the sent amount; ckERC20 withdrawals cannot. [4](#0-3) 

**Cascading freeze:** `create_resubmit_transactions` stops processing on the first `InsufficientTransactionFee` error, explicitly blocking all higher nonces:

```rust
// We stop on the first error since if a transaction with nonce n could not be resubmitted
// then the next transactions with nonces n+1, n+2, ... are blocked anyway
...
return transactions_to_resubmit;
``` [5](#0-4) 

**No escape hatch:** No function in the minter source removes an entry from `sent_tx` without a finalized Ethereum receipt, and no governance or admin endpoint to cancel a stuck sent transaction was found.

## Impact Explanation

The impact is twofold and concrete:

1. **User funds frozen**: The user's ckETH fee and ckERC20 tokens are burned at withdrawal time. With the transaction stuck in `sent_tx` and no reimbursement event emitted, those assets are inaccessible for the duration of the gas spike — potentially hours or days.

2. **Minter-wide withdrawal DoS**: Because Ethereum enforces sequential nonce ordering, a single stuck ckERC20 transaction at nonce N prevents every subsequent withdrawal (ckETH and ckERC20) from being mined. All users with pending withdrawals are affected. This matches the allowed impact class: *High — Application/platform-level DoS with concrete user or protocol harm* and *High — Significant Chain Fusion / ck-token security impact*.

## Likelihood Explanation

Ethereum gas price spikes are routine. Any user who submitted a ckERC20 withdrawal during a low-gas window with a conservative `max_transaction_fee` will trigger this condition during any subsequent spike. A malicious actor can deliberately exploit this: submit a withdrawal with a minimal fee during low gas (the `withdraw_erc20` endpoint is callable by any unprivileged principal), then wait for gas to rise. The cost to the attacker is the burned ckETH fee and ckERC20 amount, which can be minimised by choosing the smallest supported withdrawal. The resulting pipeline freeze affects all other users. [6](#0-5) 

## Recommendation

1. **Emit a reimbursement on stuck sent transactions**: When `resubmit_transactions_batch` receives `ResubmitTransactionError::InsufficientTransactionFee`, emit a `FailedErc20WithdrawalRequest` (or equivalent) event to schedule reimbursement of both the ckETH fee and the ckERC20 tokens, and remove the entry from `sent_tx` after a configurable expiry block count.
2. **Unblock the nonce**: After removing the stuck transaction, allow the minter to advance past that nonce so subsequent withdrawals can proceed.
3. **Fee buffer at withdrawal time**: Require `max_transaction_fee` to include a margin above the current estimate (e.g., ≥10%) to reduce the probability of hitting the cap during normal fluctuations — consistent with the safety-margin language already present in `ckerc20.adoc`. [7](#0-6)

    /// We stop on the first error since if a transaction with nonce n could not be resubmitted
    /// (e.g., the transaction amount does not cover the new fees),
    /// then the next transactions with nonces n+1, n+2, ... are blocked anyway
    /// and trying to resubmit them would only artificially increase their transaction fees.
    pub fn create_resubmit_transactions(
        &self,
        latest_transaction_count: TransactionCount,
        current_gas_fee: GasFeeEstimate,
    ) -> Vec<Result<(LedgerBurnIndex, Eip1559TransactionRequest), ResubmitTransactionError>> {
        // If transaction count at block height H is c > 0, then transactions with nonces
        // 0, 1, ..., c - 1 were mined. If transaction count is 0, then no transactions were mined.
        // The nonce of the first pending transaction is then exactly c.
        let first_pending_tx_nonce: TransactionNonce = latest_transaction_count.change_units();
        let mut transactions_to_resubmit = Vec::new();
        for (nonce, burn_index, signed_tx) in self
            .sent_tx
            .iter()
            .filter(|(nonce, _burn_index, _signed_tx)| *nonce >= &first_pending_tx_nonce)
        {
            let last_signed_tx = signed_tx.last().expect("BUG: empty sent transactions list");
            match last_signed_tx.resubmit(current_gas_fee.clone()) {
                Ok(Some(new_tx)) => {
                    transactions_to_resubmit.push(Ok((*burn_index, new_tx)));
                }
                Ok(None) => {
                    // the transaction fee is still up-to-date but because the transaction did not get included,
                    // we re-send it as is to be sure that it remains known to the mempool and hopefully be included at some point.
                    // Since we always re-send the last non-included transactions in sent_tx, there is nothing to do.
                }
                Err(crate::tx::ResubmitTransactionError::InsufficientTransactionFee {
                    allowed_max_transaction_fee,
                    actual_max_transaction_fee,
                }) => {
                    transactions_to_resubmit.push(Err(
                        ResubmitTransactionError::InsufficientTransactionFee {
                            ledger_burn_index: *burn_index,
                            transaction_nonce: *nonce,
                            allowed_max_transaction_fee,
                            max_transaction_fee: actual_max_transaction_fee,
                        },
                    ));
                    return transactions_to_resubmit;
