### Title
Owner Can Steal All ETH Locked in Bridge by Changing `eth_connector_contract_account` - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `set_eth_connector_contract_account` function can be called by the contract owner at any time, with no one-time lock or immutability guarantee. The stored connector account ID is the sole trust anchor used in `ft_on_transfer` to decide whether incoming NEP-141 transfers mint base ETH balances. By pointing this to a malicious NEP-141 token they control, the owner can mint arbitrary ETH balances on Aurora without depositing real ETH, then withdraw real ETH from the bridge — draining it entirely.

---

### Finding Description

`set_eth_connector_contract_account` in `engine/src/contract_methods/connector.rs` (lines 418–438) allows the owner to overwrite the `EthConnectorAccount` storage key at any time, unconditionally:

```rust
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();
        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }
        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;
        set_connector_account_id(io, &args.account);          // ← no one-time guard
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
        Ok(())
    })
}

### Impact Explanation

**Critical — Direct theft of all ETH locked in the bridge.**

The owner can mint unbacked ETH balances on Aurora and then redeem them against the real ETH held in the legitimate eth-connector contract, leaving the bridge insolvent. Every user who deposited ETH into the bridge loses their funds.

---

### Likelihood Explanation

The function is callable by the owner at any time with a single NEAR transaction. There is no time-lock, no one-time initialization guard, and no event that would alert users before the attack completes. The attack requires only two owner transactions (change connector → mint → change back → withdraw) and can be executed atomically within a short window.

---

### Recommendation

Apply a one-time initialization guard analogous to the original report's recommendation. Since the connector account starts unset, the check is straightforward:

```diff

    io: I,
    env: &E,
) -> Result<(), ContractError> {

        if is_private.is_err() {

        }
+       // Allow setting only once; value must not already be present
+       if get_connector_account_id(&io).is_ok() {
+           return Err(errors::ERR_CONNECTOR_ALREADY_SET.into());
+       }
        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;
+       if args.account.as_str().is_empty() {
+           return Err(errors::ERR_INVALID_ACCOUNT.into());
+       }
        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
        Ok(())

### Proof of Concept

1. Owner deploys a malicious NEP-141 token `evil.near` that they fully control.
2. Owner calls `set_eth_connector_contract_account` with `account = evil.near`.
3. Owner calls `ft_transfer_call` on `evil.near` targeting the Aurora engine, with `amount = N` (equal to total ETH locked in the real bridge) and `msg = <attacker_evm_address>`.
4. The NEAR runtime calls `ft_on_transfer` on Aurora with `predecessor_account_id = evil.near`.
5. Inside `ft_on_transfer`, `evil.near == get_connector_account_id()` is true, so `receive_base_tokens` is called and N ETH is credited to the attacker's EVM address — with no real ETH deposited.
6. Owner calls `set_eth_connector_contract_account` again, restoring the real eth-connector account.
7. Attacker calls `withdraw` on Aurora; the call is forwarded to the real eth-connector, which releases N real ETH to the attacker's Ethereum address.
8. The bridge is now insolvent: all user deposits are gone.
