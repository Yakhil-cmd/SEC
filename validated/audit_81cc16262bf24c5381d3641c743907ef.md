Audit Report

## Title
2-of-4 Provider Disagreement (2-2 Split) Can DOS ckETH/ckERC20 Deposit Scraping - (`File: rs/ethereum/cketh/minter/src/deposit.rs`)

## Summary

The ckETH minter's `scrape_block_range` applies `NoReduction` after querying the EVM RPC canister, which converts any `EvmMultiRpcResult::Inconsistent` response into an immediate `Err`. When `scrape_until_block` receives that error it returns early without advancing the scraped-block pointer, permanently stalling deposit detection until the provider disagreement resolves or a governance upgrade replaces the offending provider. A 2-of-4 provider split (2 providers returning one result, 2 returning another) is sufficient to trigger the halt; three documented mainnet emergency upgrades confirm this is an exploited-in-production failure mode.

## Finding Description

**Code path:**

`scrape_logs()` (deposit.rs L140-159) calls three sequential `scrape_until_block` passes. Inside `scrape_until_block` (L213-232), each block-range chunk is passed to `scrape_block_range`. At L251-263, `scrape_block_range` calls the EVM RPC canister and immediately applies `NoReduction`:

```rust
.reduce_with_strategy(NoReduction)
```

`NoReduction` (eth_rpc_client/mod.rs L177-183) is:

```rust
impl<T> ReductionStrategy<T> for NoReduction {
    fn reduce(&self, results: EvmMultiRpcResult<T>) -> Result<T, MultiCallError<T>> {
        consistent_result_or_reduce(results, |inconsistent| {
            Err(MultiCallError::InconsistentResults(inconsistent))
        })
    }
}
```

`consistent_result_or_reduce` (L244-259) passes any `EvmMultiRpcResult::Inconsistent` variant directly to the closure, which unconditionally returns `Err`. On a non-size error, `scrape_block_range` returns `Err(e)` (L301-303). `scrape_until_block` catches this at L222-231 and executes `return`, abandoning all remaining block ranges **without updating `last_scraped_block_number`**. The next timer tick retries the identical range, producing the same error in a perpetual loop.

**Important correction to the submitted claim:** The EVM RPC client is configured with `ConsensusStrategy::Threshold { total: Some(4), min: 3 }` (eth_rpc_client/mod.rs L56-63). When 3-of-4 providers agree, the EVM RPC canister returns `EvmMultiRpcResult::Consistent`, and `NoReduction` succeeds — the test `should_mint_when_1_error_with_3_out_of_4_strategy` (cketh.rs L266-289) confirms this. Therefore a **single** misbehaving provider (3-1 split) does **not** halt scraping under the current code. The actual trigger is a **2-2 split**: two providers returning one result and two returning another, which the EVM RPC canister cannot resolve to `Consistent` regardless of threshold, so it returns `Inconsistent`, and `NoReduction` converts that to an error. The three cited mainnet incidents all occurred before the 3-of-4 threshold upgrade was deployed (the 2024-09-11 upgrade notes explicitly state "The long term solution is to use a more robust strategy (e.g., agreement among 3 providers, when 4 were queried)"), so they are evidence of the pre-upgrade fragility, not the current code path. The current residual risk is the 2-2 split.

**Why existing checks are insufficient:** `NoReduction` has no tolerance for any inconsistency. `StrictMajorityByKey` and `MinByKey` exist in the same file (L221-242, L198-219) and could handle a 2-2 split by selecting the majority or minimum, but neither is used for log scraping.

## Impact Explanation

A sustained 2-2 provider split halts all three log-scraping passes (`ReceivedEthLogScraping`, `ReceivedErc20LogScraping`, `ReceivedEthOrErc20LogScraping`). No new ETH or ERC20 deposit events are detected; no ckETH or ckERC20 tokens are minted for users who have already sent funds to the helper contract. User funds are locked on Ethereum with no IC-side credit until the split resolves or a governance upgrade replaces a provider. This is a concrete, in-scope **Chain Fusion / ck-token application-level DoS with direct user-funds harm**, matching the High impact tier ($2,000–$10,000): "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation

A 2-2 split requires two of four independent external RPC providers to return mutually inconsistent results simultaneously. This is a realistic and observed failure mode: providers can diverge due to indexing lag, API bugs, protocol upgrades (Dencun), or network partitions. The three mainnet incidents demonstrate that provider disagreements occur in practice and require emergency governance responses. The 3-of-4 threshold upgrade reduced the attack surface (single-provider failures no longer halt the system) but a 2-2 split remains unmitigated. Likelihood is **Medium-High**: not trivially triggered by a single actor, but naturally occurring from independent provider failures.

## Recommendation

1. Replace `NoReduction` in `scrape_block_range` with `StrictMajorityByKey` (already implemented in eth_rpc_client/mod.rs L221-242) keyed on the log list, so a 3-1 majority is accepted and a 2-2 split selects the majority rather than halting.
2. Apply the same replacement to `finalized_transaction_count` and any other call sites using `NoReduction` for safety-critical queries.
3. On a non-size, non-recoverable error, consider advancing the scraped block pointer by a configurable minimum to avoid indefinite retry of the same range.

## Proof of Concept

The existing test `should_not_mint_when_logs_too_inconsistent` (cketh.rs L241-264) is a direct reproduction:

```rust
mock.respond_with(JsonRpcProvider::Provider1, block_pi_logs.clone())
    .respond_with(JsonRpcProvider::Provider2, public_node_logs.clone())
    .respond_with(JsonRpcProvider::Provider3, block_pi_logs.clone())
    .respond_with(JsonRpcProvider::Provider4, public_node_logs.clone())
// → .expect_no_mint()
```

With a 2-2 split, `NoReduction` returns `InconsistentResults`, `scrape_until_block` returns early, and `SyncedToBlock` remains at `LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL` (cketh.rs L785-787), confirming the stall. The complementary test `should_mint_when_1_error_with_3_out_of_4_strategy` (cketh.rs L266-289) confirms that a 3-1 split succeeds, bounding the exploitable scenario to the 2-2 case.
