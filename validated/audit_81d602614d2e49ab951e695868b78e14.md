Audit Report

## Title
Unbounded Slot Table and Task Growth via `SLOT_TABLE_NO_LIMIT` Enables Single-Peer OOM DoS — (`rs/p2p/consensus_manager/src/receiver.rs`, `rs/replica/setup_ic_network/src/lib.rs`)

## Summary

Five of six P2P artifact channels (consensus, certifier, dkg, idkg, https_outcalls) are configured with `slot_limit = usize::MAX`. The per-peer slot table guard `peer_slot_table_len < self.slot_limit` is therefore always true, allowing a single authenticated subnet peer to grow `slot_table`, `active_assembles`, and `artifact_processor_tasks` without bound by sending slot updates with unique slot numbers and artifact IDs while never retracting them or delivering the artifact body. This causes OOM or CPU exhaustion on every honest replica the attacker is connected to.

## Finding Description

In `rs/replica/setup_ic_network/src/lib.rs` lines 74–75, two constants are defined:

```rust
const SLOT_TABLE_LIMIT_INGRESS: usize = 50_000;
const SLOT_TABLE_NO_LIMIT: usize = usize::MAX;
```

The comment at line 72 explicitly acknowledges the threat model for ingress but the same protection is not applied to the other five channels. At lines 237, 246, 268, 279, 291, and 303, `abortable_broadcast_channel` is called with `SLOT_TABLE_NO_LIMIT` for consensus (both branches), certifier, dkg, idkg, and https_outcalls. Only ingress (line 257) uses `SLOT_TABLE_LIMIT_INGRESS`.

In `rs/p2p/consensus_manager/src/receiver.rs`, the guard at line 373 is:

```rust
Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit => { … }
Entry::Vacant(_) => { /* drop */ }
```

With `self.slot_limit = usize::MAX`, the condition `peer_slot_table_len < usize::MAX` is always true in practice — the drop branch is unreachable. Every new `(peer_id, slot_number)` pair is unconditionally inserted into `slot_table`.

When `to_add` is `true` and the artifact ID has not been seen before, lines 405–419 unconditionally insert into `active_assembles` and spawn a new Tokio task via `artifact_processor_tasks.spawn_on`. The spawned `process_slot_update` task (lines 448–539) blocks waiting for either the peer to retract the slot (`peer_rx.wait_for(|p| p.is_empty())`) or the artifact to be assembled. A malicious peer that never retracts and never delivers the artifact body keeps every task alive indefinitely.

The only cleanup path is `handle_topology_update` (line 542), which removes slot table entries for nodes that leave the subnet topology — but this requires a governance action and does not prevent the attack while the node remains a subnet member.

## Impact Explanation

Three in-memory structures grow without bound per malicious peer: `slot_table[peer_id]` (one entry per unique slot number), `active_assembles` (one entry per unique artifact ID), and `artifact_processor_tasks` (one live Tokio task per unique artifact ID). Sending N distinct slot numbers with N distinct artifact IDs allocates O(N) HashMap entries and spawns O(N) Tokio tasks. At scale this causes OOM on the victim replica, crashing the process and halting its participation in consensus. This matches the allowed High impact: **Application/platform-level DoS, crash, consensus blocking, or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000). A single malicious node can simultaneously target every honest peer it is connected to.

## Likelihood Explanation

The attacker must be an authenticated subnet node (QUIC/TLS mutual authentication). This is within the explicit scope of Byzantine protocol peer behavior below the consensus fault threshold — a single compromised or malicious node operator can execute this without any governance majority. The attack is trivially scriptable: send a tight loop of `SlotUpdate { slot_number: i, commit_id: i, update: Update::Id(random_unique_id()) }` messages over the existing P2P connection for any of the five unprotected artifact types. No victim mistake, phishing, or external dependency is required.

## Recommendation

Apply a finite per-peer slot limit to all artifact channels, not just ingress. The existing `SLOT_TABLE_LIMIT_INGRESS = 50_000` pattern is the correct model. Appropriate limits for each artifact type should be derived from the maximum number of in-flight artifacts that the protocol can legitimately produce per round (e.g., consensus has a bounded number of blocks/notarizations per height). Remove or rename `SLOT_TABLE_NO_LIMIT` to prevent future misuse. Additionally, consider bounding `active_assembles` independently of the slot table, since a single artifact ID can be referenced from multiple slots.

## Proof of Concept

```rust
// Attacker is a legitimate subnet node connected to victim replica R.
// Uses any of the five unprotected artifact types (e.g., Consensus).
for i in 0u64.. {
    send_slot_update(R, artifact_type = Consensus, SlotUpdate {
        slot_number: SlotNumber(i),
        commit_id:   CommitId(i),
        update:      Update::Id(unique_consensus_id(i)),
    });
    // Never send the artifact body; never retract the slot.
    // Each iteration: slot_table grows by 1, active_assembles grows by 1,
    // artifact_processor_tasks spawns 1 new task that blocks indefinitely.
}
// R's memory and task count grow until OOM kills the replica process.
```

A deterministic integration test can verify this by constructing a `ConsensusManagerReceiver` with `slot_limit = usize::MAX`, injecting N synthetic `SlotUpdate` messages with unique slot numbers and artifact IDs from a single peer, and asserting that `active_assembles.len() == N` and `artifact_processor_tasks.len() == N` after processing — confirming unbounded growth with no drop path exercised.
