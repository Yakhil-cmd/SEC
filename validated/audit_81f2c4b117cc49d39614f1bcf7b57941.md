### Title
Incorrect `MAX_BLOBS_PER_BLOCK` Constant (Pre-Pectra Value 6 vs Post-Pectra Value 9) Causes EIP-4844 Transaction Rejection and Incorrect Blob Gas Limit - (File: `zk_ee/src/system/constants.rs`)

---

### Summary

`zk_ee/src/system/constants.rs` defines `MAX_BLOBS_PER_BLOCK = 6`, the pre-Pectra (pre-EIP-7691) value. Post-Pectra, the correct maximum is 9. This constant is consumed by the ZK STF metadata layer to size the blob-hash storage array and to compute `blobs_gas_limit()`. As a result, any EIP-4844 transaction carrying 7–9 blobs — valid on Ethereum mainnet post-Pectra — is either rejected or causes an ArrayVec overflow in ZKsync OS, while the Ethereum block-header validation layer in the same codebase already uses the correct value of 9.

---

### Finding Description

Two separate locations in the codebase define `MAX_BLOBS_PER_BLOCK` with conflicting values:

**Incorrect (stale pre-Pectra) definition:**

`zk_ee/src/system/constants.rs` line 31:
```rust
pub const MAX_BLOBS_PER_BLOCK: usize = 6;
``` [1](#0-0) 

**Correct post-Pectra definitions (already present in the bootloader layer):**

`basic_bootloader/src/bootloader/block_flow/ethereum/mod.rs` line 9:
```rust
pub const MAX_BLOBS_PER_BLOCK: usize = 9;
``` [2](#0-1) 

`basic_bootloader/src/bootloader/block_flow/ethereum/block_header.rs` line 30:
```rust
const MAX_BLOBS_PER_BLOCK: usize = 9;
``` [3](#0-2) 

The ZK STF metadata layer (`zk_metadata.rs`) imports exclusively from `zk_ee::system::constants` (line 8: `use crate::system::constants::*;`), so it picks up the stale value of 6:

1. **`TxLevelMetadata.blobs` ArrayVec capacity is capped at 6:** [4](#0-3) 

2. **`blobs_gas_limit()` returns `6 × GAS_PER_BLOB = 786 432` instead of `9 × GAS_PER_BLOB = 1 179 648`:** [5](#0-4) 

3. **ZK transaction validation imports `MAX_BLOBS_PER_BLOCK` from `zk_ee::system` (= 6) and uses it for blob-count enforcement:** [6](#0-5) 

The Ethereum block-header chain-checker already uses `TARGET_BLOB_GAS_PER_BLOCK = GAS_PER_BLOB * 6` (target) and `MAX_BLOBS_PER_BLOCK = 9` (max), consistent with EIP-7691/Pectra: [7](#0-6) 

---

### Impact Explanation

**Vulnerability class:** Resource accounting bug / EVM semantic mismatch / valid-execution unprovability.

Three concrete effects:

| Effect | Root cause | Magnitude |
|---|---|---|
| `blobs_gas_limit()` returns 786 432 instead of 1 179 648 | `max_blobs()` returns 6 | 33 % undercount |
| `TxLevelMetadata.blobs` ArrayVec panics or truncates on 7–9 blob tx | ArrayVec capacity = 6 | Tx unprocessable |
| ZK validation rejects valid 7–9 blob transactions | Blob count check uses limit 6 | Tx rejected |

A user submitting a valid post-Pectra EIP-4844 transaction with 7, 8, or 9 blobs will have that transaction rejected by ZKsync OS even though it is fully valid on Ethereum L1. This is a state-transition divergence: the ZKsync OS state transition function disagrees with Ethereum's rules, breaking EVM equivalence for blob transactions and potentially causing valid L2 transactions to be unprovable or silently dropped.

---

### Likelihood Explanation

**High.** Post-Pectra (active on Ethereum mainnet since April 2025), any user or application that submits an EIP-4844 transaction with 7–9 blobs triggers this path. No special privilege is required — any unprivileged transaction sender can reach the blob-count validation in `validate_and_compute_fee_for_transaction`. The discrepancy is structural (a hardcoded constant) and is hit on every such transaction.

---

### Recommendation

Update `MAX_BLOBS_PER_BLOCK` in `zk_ee/src/system/constants.rs` from `6` to `9` to match the post-Pectra (EIP-7691) specification:

```rust
// zk_ee/src/system/constants.rs
pub const MAX_BLOBS_PER_BLOCK: usize = 9;  // EIP-7691 / Pectra
```

Additionally, consolidate the three separate definitions of this constant (`zk_ee/src/system/constants.rs`, `basic_bootloader/src/bootloader/block_flow/ethereum/mod.rs`, `basic_bootloader/src/bootloader/block_flow/ethereum/block_header.rs`) into a single authoritative source to prevent future drift.

---

### Proof of Concept

1. Construct a valid EIP-4844 transaction with 7 blob versioned hashes (valid post-Pectra, `7 × GAS_PER_BLOB = 917 504 < 1 179 648`).
2. Submit it to ZKsync OS ZK STF.
3. In `validate_and_compute_fee_for_transaction` (`zk/validation_impl.rs`), the blob count check uses `MAX_BLOBS_PER_BLOCK = 6` from `zk_ee::system::constants`.
4. The transaction is rejected (blob count 7 > 6) or the `TxLevelMetadata.blobs` ArrayVec panics on push, even though the transaction is valid under Ethereum's post-Pectra rules (`MAX_BLOBS_PER_BLOCK = 9`).
5. The Ethereum block-header verifier in the same binary accepts the same block as valid (it uses `MAX_BLOBS_PER_BLOCK = 9`), confirming the divergence is internal to ZKsync OS.
