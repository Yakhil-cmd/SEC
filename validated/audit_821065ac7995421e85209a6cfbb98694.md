### Title
Multiblock Batch Log Accumulator Overflow Causes Unprovable Batches - (`basic_bootloader/src/bootloader/block_flow/zk/batch_data.rs`)

---

### Summary

In the multiblock batch proving path, L2→L1 logs from each block are accumulated into a fixed-capacity `ArrayVec<Bytes32, 16384>`. The per-block log limit enforces at most `MAX_NUMBER_OF_LOGS = 16_384` logs per block, but there is no enforcement on the **total** log count across all blocks in a batch. When the cumulative log count across blocks exceeds 16,384, `apply_to_array_vec` calls `ArrayVec::push` on a full array, which panics in Rust, making the batch unprovable.

---

### Finding Description

`MAX_NUMBER_OF_LOGS` is defined as 16,384 — matching the capacity of the Merkle tree used for the batch's `l2_logs_tree_root`. [1](#0-0) 

The per-block limit check in `check_for_block_limits` correctly rejects any transaction that would push a single block's log count above `MAX_NUMBER_OF_LOGS`. [2](#0-1) 

This check uses `system.io.logs_len()`, which returns the log count for the **current block only** — it resets between blocks. [3](#0-2) 

In the multiblock batch post-op, after each block is finalized, its logs are appended to the batch-level accumulator: [4](#0-3) 

The accumulator is declared as a fixed-capacity `ArrayVec<Bytes32, 16384>`: [5](#0-4) 

The `apply_to_array_vec` method calls `array_vec.push(log.hash())` unconditionally, with no bounds check: [6](#0-5) 

`ArrayVec::push` panics when the array is full. If a batch contains two blocks each with 8,193 logs (both individually within the 16,384 per-block limit), the second block's `apply_to_array_vec` call will attempt to push beyond capacity 16,384, causing a panic in the RISC-V proving binary.

The batch-level `l2_logs_root` computation in `into_public_input` then never completes: [7](#0-6) 

---

### Impact Explanation

A multiblock batch whose total L2→L1 log count across all blocks exceeds 16,384 will panic during the proving phase. The batch public input (`l2_logs_tree_root`) cannot be computed, making the batch permanently unprovable. The operator must either exclude the offending transactions or abandon the batch, causing a liveness disruption. This is a **valid-execution unprovability** issue: all individual blocks execute and finalize correctly, but the batch proof cannot be generated.

---

### Likelihood Explanation

Any user can emit L2→L1 logs by calling the L1 Messenger system contract at `0x8008` with `sendToL1(bytes)`. The hook is reachable by any EOA. [8](#0-7) 

A single block allows up to 16,384 logs. A batch of just two blocks, each with ~8,193 logs, is sufficient to trigger the overflow. An attacker willing to pay gas for ~16,385 `sendToL1` calls across two blocks can reliably trigger this condition.

---

### Recommendation

1. **Enforce a batch-level log limit**: Before calling `apply_to_array_vec`, assert or check that `batch_data.logs_storage.len() + io.logs_storage.len() <= MAX_NUMBER_OF_LOGS`. If the limit would be exceeded, either reject the block from the batch or return an error.

2. **Use a checked push**: Replace the unchecked `array_vec.push(log.hash())` in `apply_to_array_vec` with `array_vec.try_push(log.hash()).expect(...)` or propagate an error, so overflow is caught with a meaningful message rather than a panic.

3. **Align the per-block limit with the batch capacity**: If a batch can contain N blocks, the per-block log limit should be `MAX_NUMBER_OF_LOGS / N` (or the batch capacity should be `N * MAX_NUMBER_OF_LOGS`).

---

### Proof of Concept

1. Deploy or use the existing L1 Messenger system contract at `0x8008`.
2. Submit Block 1 containing transactions that call `sendToL1(bytes)` enough times to emit 8,193 L2→L1 logs (within the 16,384 per-block limit).
3. Submit Block 2 in the same multiblock batch, also emitting 8,193 logs.
4. When the multiblock batch prover runs `post_op` for Block 2, it calls `io.logs_storage.apply_to_array_vec(&mut batch_data.logs_storage)`.
5. `batch_data.logs_storage` already holds 8,193 entries from Block 1. Pushing 8,193 more exceeds the `ArrayVec<Bytes32, 16384>` capacity.
6. `ArrayVec::push` panics. The RISC-V proving binary aborts. The batch public input is never written. The batch is unprovable. [6](#0-5) [5](#0-4) [4](#0-3)

pub(crate) fn send_to_l1_inner<S: EthereumLikeTypes>(
    calldata: &[u8],
    resources: &mut S::Resources,
    system: &mut System<S>,
) -> Result<Result<(), &'static str>, SystemError> {
    if calldata.len() < 20 {
        return Ok(Err(
            "L1 messenger failure: sendToL1 called with invalid calldata",
        ));
    }

    let address_sender = B160::try_from_be_slice(&calldata[0..20]).ok_or(
        SystemError::LeafDefect(internal_error!("Failed to create B160 from 20 byte array")),
    )?;

    let message = &calldata[20..];

    // emit L1 message (ignore returned hash)
    // TODO(EVM-1190): hash calculation is suboptimal, to be refactored in future
    system.io.emit_l1_message(
        // Gas should be charged by the L1Messenger system contract
        ExecutionEnvironmentType::NoEE,
        resources,
        &address_sender,
        message,
    )?;
