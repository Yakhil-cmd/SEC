### Title
Blake2F Precompile Unbounded `rounds` Loop Exhausts NEAR Gas Budget, Bypassing EVM Gas Accounting — (`engine-precompiles/src/blake2.rs`)

---

### Summary

The `Blake2F` precompile accepts an attacker-controlled `rounds` field (up to `u32::MAX = 4,294,967,295`) and executes a tight CPU loop proportional to that value with no NEAR gas checkpoint. Because the EVM gas check is satisfied before the loop runs, the NEAR runtime hard-aborts the transaction mid-execution when the 300 TGas budget is exhausted. This abort bypasses the EVM-level gas refund path entirely, and because the NEAR transaction is reverted, the attacker's ETH gas charge is also reverted — making the attack free to repeat indefinitely.

---

### Finding Description

**Gas cost calculation — no overflow, but no NEAR budget awareness**

`required_gas` computes cost as:

```rust
Ok(num_rounds * costs::F_ROUND)   // F_ROUND = EthGas::new(1)
```

The `Mul<EthGas> for u32` implementation widens to `u64` before multiplying:

```rust
EthGas(u64::from(self) * rhs.0)
```

For `rounds = u32::MAX`, this yields `EthGas(4_294_967_295)` — a valid `u64`, no overflow. The EVM gas check at lines 174–178 then passes if the transaction's `gas_limit` is set to `u32::MAX + intrinsic_gas` (a legal `u64` value):

```rust
if let Some(target_gas) = target_gas
    && cost > target_gas
{
    return Err(ExitError::OutOfGas);
}
```

**The unbounded loop**

After the check passes, `f()` is called with `rounds = 4_294_967_295`:

```rust
for i in 0..rounds {
    let s = &consts::SIGMA[usize::try_from(i).expect("...") % 10];
    g(&mut v, 0, 4, 8, 12, m[s[0]], m[s[1]]);
    // ... 7 more g() calls
}
```

Each iteration executes 8 `g()` calls, each performing 8 wrapping arithmetic and rotate operations on `u64` values. The benchmark data in the codebase shows 12 rounds of Blake2F costs ~4 GGas of NEAR gas. Extrapolating: `4_294_967_295 rounds × (4 GGas / 12 rounds) ≈ 1.43 × 10^18 gas`, which is roughly **4.8 million times** the 300 TGas NEAR transaction limit.

**NEAR hard-abort bypasses EVM accounting**

When NEAR WASM gas is exhausted, the NEAR runtime issues a hard trap — not a Rust panic that can be caught. The entire NEAR transaction is aborted and reverted. The `submit_with_alt_modexp` function in `engine/src/engine.rs` pre-charges gas at line 1101 and issues refunds at line 1148:

```rust
let prepaid_amount = match engine.charge_gas(...) { ... };
// ... EVM execution (aborted here by NEAR runtime) ...
refund_unused_gas(&mut io, &sender, gas_used, &prepaid_amount, ...)?;
```

Because the NEAR transaction is reverted before `refund_unused_gas` is reached, **both the EVM state changes and the ETH gas charge are rolled back**. The attacker's ETH balance is restored, making each attack iteration free.

---

### Impact Explanation

An attacker can submit repeated transactions that each consume the full 300 TGas NEAR budget at zero ETH cost. The relayer pays NEAR gas for each aborted transaction. During each such transaction, no other Aurora transactions can be processed (NEAR processes one transaction at a time per shard). This constitutes **temporary freezing of funds**: users cannot interact with their ETH or ERC-20 balances held in Aurora contracts for as long as the attacker sustains the DoS. The EVM-level invariant — that a transaction passing EVM gas validation must terminate with a proper EVM result and gas refund — is violated.

---

### Likelihood Explanation

The attack requires only:
1. A valid 213-byte Blake2 input (trivially constructed per EIP-152 format)
2. `rounds` field set to `0xFFFFFFFF`
3. `gas_limit` set to `u32::MAX + ~50_000` (covers intrinsic gas, fits in `u64`)
4. Sufficient ETH balance to pass `charge_gas` — but this is refunded on NEAR abort, so the net cost is zero ETH

No privileged access, no admin keys, no social engineering. Any EOA on Aurora can execute this.

---

### Recommendation

Cap `rounds` to a value that cannot exhaust the NEAR gas budget. The standard Blake2b uses 12 rounds; EIP-152 allows arbitrary values but Aurora's NEAR execution environment imposes a hard constraint. A safe upper bound can be derived from the NEAR gas budget: with ~4 GGas per 12 rounds, the maximum safe rounds is approximately `300 TGas / (4 GGas / 12) ≈ 900_000`. Enforce this in `required_gas` and `run`:

```rust
const MAX_ROUNDS: u32 = 900_000; // tuned to NEAR 300 TGas budget
if num_rounds > MAX_ROUNDS {
    return Err(ExitError::Other(Borrowed("ERR_BLAKE2F_ROUNDS_LIMIT")));
}
```

Alternatively, implement a NEAR gas checkpoint inside the `f()` loop that returns an EVM `OutOfGas` error before the NEAR budget is exhausted.
