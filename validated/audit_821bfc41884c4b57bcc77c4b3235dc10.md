### Title
Pre-EIP-155 Legacy Transaction Replay Across Aurora Chain Instances — (File: `engine/src/engine.rs`)

---

### Summary

Aurora Engine's `submit_with_alt_modexp` function conditionally validates the chain ID only when it is present in the transaction signature. Pre-EIP-155 legacy transactions (signed with `v = 27` or `v = 28`) carry no chain ID, so `NormalizedEthTransaction::chain_id` is `None` and the guard is silently skipped. Any such transaction submitted to one Aurora deployment (e.g., Testnet, chain ID `1313161555`) can be replayed verbatim on another (e.g., Mainnet, chain ID `1313161554`), bypassing the only cross-chain replay protection the engine provides.

---

### Finding Description

**Root cause — `engine/src/engine.rs`, chain-ID guard:** [1](#0-0) 

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

The guard fires only when `transaction.chain_id` is `Some(…)`. When it is `None` the entire block is skipped and execution continues unconditionally.

**Why `chain_id` is `None` for pre-EIP-155 transactions — `engine-transactions/src/legacy.rs`:** [2](#0-1) 

```rust
pub const fn chain_id(&self) -> Option<u64> {
    match self.v {
        0..=34 => None,          // v = 27 or 28 → no chain ID
        _ => Some((self.v - 35) / 2),
    }
}
```

**How `chain_id` propagates into `NormalizedEthTransaction` — `engine-transactions/src/lib.rs`:** [3](#0-2) 

```rust
Legacy(tx) => Self {
    address: tx.sender()?,
    chain_id: tx.chain_id(),   // None for v = 27 / 28
    ...
}
```

A pre-EIP-155 transaction is still accepted by the RLP decoder: [4](#0-3) 

```rust
} else {
    let legacy = legacy::LegacyEthSignedTransaction::decode(&Rlp::new(bytes))?;
    Ok(Self::Legacy(legacy))
}
```

The nonce check that follows the chain-ID guard does not prevent replay across chains because the sender's nonce on the target chain may independently match: [5](#0-4) 

---

### Impact Explanation

An attacker who observes a pre-EIP-155 transaction on Aurora Testnet (or any other Aurora deployment) can submit the identical raw bytes to Aurora Mainnet. If the sender's nonce and balance on Mainnet satisfy the transaction's requirements, the EVM executes it and transfers the sender's Mainnet ETH (or invokes an arbitrary contract) without the sender's knowledge or consent. This constitutes **direct theft of user funds at rest**.

---

### Likelihood Explanation

Pre-EIP-155 transactions are uncommon with modern wallets, but they remain valid and can be produced by any user or tool that omits the chain-ID field (e.g., raw RLP construction, older libraries, or deliberate crafting). The attacker's entry path requires only:

1. Monitoring the public Aurora Testnet mempool or block explorer for pre-EIP-155 transactions.
2. Submitting the same bytes to Aurora Mainnet before the sender's Mainnet nonce advances.

No privileged access, key compromise, or social engineering is required. The attack is fully permissionless and externally reachable via the public `submit` endpoint.

---

### Recommendation

Reject pre-EIP-155 legacy transactions outright at the engine boundary. The simplest fix is to treat a missing chain ID as an invalid chain ID:

```rust
// Validate the chain ID — reject pre-EIP-155 transactions that carry none.
let chain_id = transaction.chain_id.ok_or(EngineErrorKind::InvalidChainId)?;
if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

This mirrors the approach taken by EIP-2930, EIP-1559, and EIP-7702 transaction types, all of which mandate a chain ID field and therefore always produce `Some(chain_id)`. [6](#0-5) 

---

### Proof of Concept

1. **Craft a pre-EIP-155 transaction** targeting Aurora Testnet (chain ID `1313161555`). Sign it with `v = 27` or `v = 28` so no chain ID is embedded in the signature.
2. **Submit to Testnet.** The engine accepts it: `transaction.chain_id` is `None`, the guard at line 1055 is skipped, and the transaction executes normally.
3. **Replay on Mainnet.** Submit the identical RLP-encoded bytes to Aurora Mainnet (chain ID `1313161554`) via the `submit` entrypoint.
4. **Engine skips the guard again** because `transaction.chain_id` is still `None`.
5. **Nonce check passes** if the sender's Mainnet nonce equals the transaction nonce (e.g., both chains were funded from the same key at the same nonce).
6. **Execution proceeds**, transferring the sender's Mainnet ETH to the transaction's `to` address — funds the sender never intended to move on Mainnet. [7](#0-6) [8](#0-7)
