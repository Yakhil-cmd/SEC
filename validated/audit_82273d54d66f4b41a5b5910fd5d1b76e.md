Audit Report

## Title
Unsolicited Pong Unconditionally Disconnects Any Post-Handshake Bitcoin Peer Connection — (`rs/bitcoin/adapter/src/connectionmanager.rs`)

## Summary
`process_pong_message` sets `valid_pong = false` for any `Pong` received while `ping_state` is `PingState::Idle`, then unconditionally calls `conn.disconnect()`. Because `PingState::Idle` is the normal resting state of every post-handshake connection, any Bitcoin peer that completes the version handshake can immediately tear down its own connection by sending a single unsolicited `Pong` message. An attacker operating or controlling a malicious Bitcoin node can exploit this repeatedly to hold the adapter at zero active connections.

## Finding Description
In `process_pong_message` ( [1](#0-0) ), the match on `conn.ping_state()` evaluates the `PingState::Idle` arm to `false`, causing the `else` branch to call `conn.disconnect()` unconditionally. There is no guard on `conn.state()` before this logic executes.

`process_bitcoin_network_message` routes `NetworkMessage::Pong` directly to `process_pong_message` with no prior connection-state check: [2](#0-1) 

Every new `Connection` is initialized with `PingState::Idle` ( [3](#0-2) ) and returns to `PingState::Idle` after every valid pong via `conn.idle()` ( [4](#0-3) ). This means the normal resting state of every post-handshake connection is `Idle`.

`conn.disconnect()` sets `state = ConnectionState::NodeDisconnected` and aborts the task handle: [5](#0-4) 

`reap_disconnected()` then removes the entry from `self.connections` and calls `address_book.remove_from_active()`: [6](#0-5) 

The full exploit path is: complete Bitcoin version handshake → send one `Pong(any_nonce)` → `process_pong_message` sees `PingState::Idle` → `valid_pong = false` → `conn.disconnect()` → connection removed. No existing guard prevents this.

## Impact Explanation
The Bitcoin adapter is the ckBTC chain-fusion component responsible for relaying Bitcoin data to the IC subnet. An attacker who can repeatedly complete the version handshake and send a single `Pong` per connection can hold the adapter at zero active connections indefinitely. This prevents the IC's Bitcoin integration from functioning — blocking ckBTC minting, redemption, and all Bitcoin-dependent subnet operations. This matches the allowed High impact: **Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000).

## Likelihood Explanation
The Bitcoin version handshake is a public, unauthenticated protocol. Any party can run a Bitcoin node, get it into the adapter's address book (via `Addr` messages from other peers or seed resolution), complete the handshake, and immediately send `Pong`. No privileged access, key material, or majority corruption is required. The attack is repeatable: each time the adapter reconnects, the attacker completes the handshake again and sends another `Pong`. A single attacker controlling one or a few Bitcoin nodes can sustain the DoS.

## Recommendation
In `process_pong_message`, treat an unsolicited `Pong` (received while `PingState::Idle`) as a no-op rather than a disconnect trigger:

```rust
PingState::Idle { last_pong_at: _ } => {
    // Unsolicited pong — ignore silently.
    return Ok(());
}
```

Optionally log a warning for observability. The `conn.disconnect()` call should only be reached when the connection is in `PingState::ExpectingPong` and the nonce does not match.

## Proof of Concept
State-machine unit test (no network required):

1. Construct a `ConnectionManager` and inject a connection in `ConnectionState::HandshakeComplete` with `PingState::Idle` (the default post-construction state).
2. Call `process_bitcoin_network_message(addr, &NetworkMessage::Pong(0))` with any nonce.
3. Assert `conn.state()` is `ConnectionState::NodeDisconnected` — confirmed by the `PingState::Idle` arm at [7](#0-6) .
4. Call `reap_disconnected()`.
5. Assert `self.connections` is empty and `address_book.remove_from_active` was called.

Any nonce value triggers the bug; no nonce matching is required because the `Idle` arm never inspects the nonce.
