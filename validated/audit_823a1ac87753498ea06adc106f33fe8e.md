Audit Report

## Title
VIP Caller Starvation via Unconditional Slot-Guard in `try_borrow_slot` — (`rs/nervous_system/clients/src/management_canister_client.rs`)

## Summary

`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` evaluates the `available_slot_count == 0` early-exit guard unconditionally before applying the VIP exemption, causing NNS canisters (including Governance) to be rejected with `SysTransient` when all 167 slots are occupied. Because the public, auth-free `canister_status` endpoint on NNS Root allows any external principal to consume slots, an unprivileged attacker can sustain a slot-exhaustion state that persistently blocks NNS Governance's management canister calls.

## Finding Description

In `rs/nervous_system/clients/src/management_canister_client.rs` at L264–287, `try_borrow_slot` sets `used_slot_count = 0` for VIP callers (L265) to express the intent that VIPs never consume a slot. However, the guard at L269 (`if *available_slot_count == 0`) is evaluated before any VIP distinction is applied:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };  // L265

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if *available_slot_count == 0 {          // L269 — fires for VIPs too
                let code = RejectCode::SysTransient as i32;
                let message = "Unavailable. Maybe, try again later?".to_string();
                return Err((code, message));         // L275 — VIP incorrectly rejected
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

The attacker entry point is `canister_status` in `rs/nns/handlers/root/impl/canister/canister.rs` (L88–98), which carries no caller check and is explicitly documented as public. The slot pool is initialized to 167 (L50). VIP status is determined at L57–66 by checking `ALL_NNS_CANISTER_IDS`; NNS Governance is in that set and is therefore intended to be exempt from slot limits. The exemption is broken because the rejection branch is taken before `used_slot_count` is ever applied.

Exploit flow:
1. Attacker sends 167 concurrent `canister_status` update calls from unprivileged principals to NNS Root, each targeting a canister Root controls so the management canister does not immediately reject.
2. All 167 calls are simultaneously in-flight (awaiting management canister response), holding `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` at 0.
3. NNS Governance calls Root's `canister_status` (e.g., during a governance operation). `new_management_canister_client` sets `is_caller_vip = true` (L58–60), but `try_borrow_slot` hits the `== 0` guard at L269 and returns `Err(SysTransient, "Unavailable. Maybe, try again later?")`.
4. Attacker continuously replenishes slots as old calls complete, sustaining the condition.

## Impact Explanation

NNS Governance relies on `NnsRootCanisterClientImpl::canister_status` (via Root's public endpoint) for governance operations such as canister upgrade checks. Persistent slot exhaustion causes those calls to fail with `SysTransient`, degrading or blocking NNS governance operations. This matches the allowed High impact: **Application/platform-level DoS with concrete NNS governance harm, not based on raw volumetric DDoS** ($2,000–$10,000).

## Likelihood Explanation

No privileged access is required. The `canister_status` endpoint is explicitly public and auth-free. Maintaining 167 simultaneous in-flight calls is feasible: each call suspends at the `await` point while the management canister responds, so 167 concurrent messages suffice. The attacker can replenish slots continuously as completions occur, requiring only sustained concurrency rather than volumetric throughput. The code comment itself acknowledges the queue fills at 500; 167 is one-third of that threshold.

## Recommendation

Add a VIP bypass to the guard in `try_borrow_slot` so that the `== 0` check is skipped entirely for VIP callers:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if !self.is_caller_vip && *available_slot_count == 0 {
                let code = RejectCode::SysTransient as i32;
                let message = "Unavailable. Maybe, try again later?".to_string();
                return Err((code, message));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

This preserves the invariant that VIP callers are never rejected by the slot-limit guard while non-VIP callers remain subject to it.

## Proof of Concept

State-machine / PocketIC integration test:

1. Deploy NNS Root with `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT = 167`.
2. Submit 167 concurrent `canister_status` update calls from 167 distinct non-NNS principals, each targeting a canister controlled by Root. Do not deliver management canister responses yet, keeping all 167 calls in-flight simultaneously.
3. Submit a `canister_status` call from `GOVERNANCE_CANISTER_ID` (a VIP principal per `ALL_NNS_CANISTER_IDS`).
4. Assert the Governance call returns `Err((SysTransient, "Unavailable. Maybe, try again later?"))`.
