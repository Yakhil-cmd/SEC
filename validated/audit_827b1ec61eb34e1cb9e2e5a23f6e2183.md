Audit Report

## Title
Overly Broad Self-Approval Guard Blocks All Cross-Subaccount ICRC-2 Approvals — (`rs/ledger_suite/icrc1/ledger/src/main.rs`, `rs/ledger_suite/icp/ledger/src/main.rs`)

## Summary
`icrc2_approve_not_async` compares only the `owner` (Principal) field when checking for self-approval, causing an unconditional `ic_cdk::trap` for any approval where `from.owner == spender.owner`, even when `from.subaccount != spender.subaccount`. Under ICRC-1, two accounts with the same owner but different subaccounts are distinct accounts; ICRC-2 must permit approvals between them. The identical defect exists in the ICP ledger. Any user or protocol attempting a cross-subaccount approval hits the trap deterministically, permanently blocking a valid and documented ICRC-2 usage pattern.

## Finding Description
In `rs/ledger_suite/icrc1/ledger/src/main.rs` at lines 828–829:

```rust
if from_account.owner == arg.spender.owner {
    ic_cdk::trap("self approval is not allowed")
}
``` [1](#0-0) 

The guard compares only the `Principal` field of the two `Account` structs. An ICRC-1 `Account` is `{owner: Principal, subaccount: Option<Subaccount>}`; two accounts sharing the same `owner` but with distinct `subaccount` values are **different accounts** per the spec. The guard should compare full account equality.

The lower-level `AllowanceTable::approve` in `rs/ledger_suite/common/ledger_core/src/approvals.rs` correctly uses full-account equality:

```rust
if account == spender {
    return Err(ApproveError::SelfApproval);
}
``` [2](#0-1) 

This correct check is **never reached** for cross-subaccount same-principal calls because `ic_cdk::trap` at line 829 fires first, unwinding the entire call and returning `CanisterCalledTrap` to the caller rather than a graceful `Err(ApproveError::...)`.

The identical bug is present in the ICP ledger at `rs/ledger_suite/icp/ledger/src/main.rs` lines 1328–1329: [3](#0-2) 

The existing `test_approve_self` test only validates the true self-approval case (same principal, same default subaccount) and does not cover the cross-subaccount case: [4](#0-3) 

## Impact Explanation
This is a significant ledger correctness and security defect in the in-scope ICP/ICRC ledger financial integration. Any user or protocol that routes funds through same-principal subaccounts using ICRC-2 approvals — a standard, documented pattern — is permanently blocked. If a protocol has already locked assets under the assumption that cross-subaccount approvals are functional (a reasonable assumption given the ICRC-2 spec), those assets become inaccessible via the approval pathway with no recovery mechanism. This constitutes concrete user and protocol harm in an in-scope ledger component, qualifying as **High** impact: "Significant ledger security impact with concrete user or protocol harm."

## Likelihood Explanation
The entrypoint is the public `icrc2_approve` update method, reachable by any unprivileged ingress call. No special role, key, or privilege is required. The trigger condition — same principal, different subaccounts — is a standard and explicitly documented ICRC-1/ICRC-2 usage pattern. Any user or protocol attempting this pattern hits the trap deterministically and repeatably.

## Recommendation
Change the self-approval guard in both ledgers from a principal-only comparison to a full-account comparison:

```rust
// Before (incorrect):
if from_account.owner == arg.spender.owner {
    ic_cdk::trap("self approval is not allowed")
}

// After (correct):
if from_account == arg.spender {
    ic_cdk::trap("self approval is not allowed")
}
```

Apply the same fix to `rs/ledger_suite/icp/ledger/src/main.rs` lines 1328–1329. Add a state-machine regression test covering the cross-subaccount case (same principal, different subaccounts) to `rs/ledger_suite/tests/sm-tests/src/lib.rs`.

## Proof of Concept

```rust
let principal_a = PrincipalId::new_user_test_id(1);
let subaccount_1 = Some([1u8; 32]);
let subaccount_2 = Some([2u8; 32]);

let (env, canister_id) = setup(ledger_wasm, encode_init_args,
    vec![(Account { owner: principal_a.0, subaccount: subaccount_1 }, 1_000_000)]);

let approve_args = ApproveArgs {
    from_subaccount: subaccount_1,
    spender: Account { owner: principal_a.0, subaccount: subaccount_2 },
    amount: Nat::from(500_000_u64),
    ..default_approve_args()
};

// Per ICRC-2 spec these are distinct accounts — approval must succeed.
// Actual result: Err(CanisterCalledTrap("self approval is not allowed"))
let result = env.execute_ingress_as(
    principal_a, canister_id, "icrc2_approve", Encode!(&approve_args).unwrap());
assert!(result.is_ok(), "cross-subaccount approval must succeed");
```
