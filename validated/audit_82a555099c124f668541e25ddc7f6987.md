Audit Report

## Title
Arbitrary `buyer` Field in `refresh_buyer_tokens` Allows Third-Party Forced Registration and `confirmation_text` Consent Bypass — (File: `rs/sns/swap/canister/canister.rs`)

## Summary
The SNS Swap canister's `refresh_buyer_tokens` update endpoint resolves the effective buyer from a caller-controlled payload field rather than the authenticated ingress sender. Any unprivileged principal can register any other principal as a swap participant and satisfy the swap's `confirmation_text` consent gate on that principal's behalf. This allows forced participation of any principal who has pre-funded their swap subaccount, and renders the `confirmation_text` mechanism ineffective as an informed-consent gate.

## Finding Description
In `rs/sns/swap/canister/canister.rs` (L128–143), the handler unconditionally accepts an arbitrary `buyer` string from the request payload:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // caller-controlled
};
swap_mut()
    .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
    .await
```

No check is ever performed that `p == caller_principal_id()` when `buyer` is non-empty. The `RefreshBuyerTokensRequest` proto documents this as "If not specified, the caller is used," confirming the field is intentionally caller-supplied, but no guard prevents misuse.

Inside `refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs`, L1134–1163), the canister:
1. Calls `validate_confirmation_text(confirmation_text)` (L1150), which only checks that the supplied text matches the swap's configured text — it does not verify that the caller is the `buyer`, and it does not store per-buyer consent records.
2. Queries the ICP ledger balance of `Account { owner: swap_canister_id, subaccount: principal_to_subaccount(&buyer) }` where `buyer` is the attacker-supplied value (L1154–1163).
3. If the balance meets `min_participant_icp_e8s`, inserts `buyer` into `self.buyers` (L1285–1288) and updates total participation amounts.

`validate_confirmation_text` (`rs/sns/swap/src/swap.rs`, L363–384) performs only a text equality check against the swap's init-configured text. The configured text is publicly readable from the swap's state. There is no binding of the confirmation to the authenticated caller identity.

**Exploit flow (Attack Path A — zero-cost forced registration):**
1. Victim Bob transfers `min_participant_icp_e8s` ICP to the swap's subaccount derived from his principal (the standard first participation step).
2. Before Bob calls `refresh_buyer_tokens` himself, attacker Alice reads the swap's `confirmation_text` from public state and calls `refresh_buyer_tokens({ buyer: "<Bob's principal>", confirmation_text: Some("<swap text>") })`.
3. The canister resolves `p = Bob`, validates the confirmation text (passes, because Alice supplied the correct public text), reads Bob's subaccount balance (meets minimum), and inserts Bob into `self.buyers` with his ICP amount locked.
4. Bob's ICP is now committed to the swap; Bob never sent a message to the swap canister and never explicitly consented.

**Exploit flow (Attack Path B — attacker-funded forced registration):**
1. Alice transfers `min_participant_icp_e8s` ICP to the swap's subaccount for any target principal `T` (e.g., a canister, a governance neuron controller).
2. Alice calls `refresh_buyer_tokens({ buyer: "<T>", confirmation_text: Some("<swap text>") })`.
3. `T` is registered as a participant. Alice receives SNS neurons attributed to `T`'s principal at finalization — but the ICP is swept to SNS governance, so the attacker's net cost is the opportunity cost of locked ICP.

## Impact Explanation
**Consent bypass**: The `confirmation_text` field exists as an explicit informed-consent gate (documented in `rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto` L847–850 and `rs/sns/swap/src/swap.rs` L1123–1126). Any caller can satisfy this check on behalf of any other principal by supplying the publicly readable text, rendering the gate ineffective. SNS projects that rely on this mechanism for legal or compliance purposes are directly harmed.

**Forced participation**: A victim who has pre-funded their swap subaccount (the normal first step) can be registered as a participant — with their ICP locked — before they choose to participate and without them ever sending a message to the swap canister. While the victim can recover ICP via `error_refund_icp` if the swap aborts, their ICP is locked for the swap's duration without consent.

**Swap-outcome manipulation**: By registering many attacker-controlled principals (each funded with `min_participant_icp_e8s`), an attacker can push `self.buyers.len()` toward `min_participants` or push total ICP toward `max_direct_participation_icp_e8s`, potentially triggering early commitment and excluding legitimate participants who arrive after the cap is hit.

This constitutes a **High** severity finding: significant SNS security impact with concrete user harm (forced ICP lockup without consent, compliance mechanism bypass) affecting in-scope SNS framework canisters.

## Likelihood Explanation
`refresh_buyer_tokens` is a public, unauthenticated update call reachable by any ingress sender with no role, key, or privileged access required. Attack Path A costs the attacker nothing beyond a single update call; it only requires that the victim has already transferred ICP to their subaccount, which is the standard and documented first step of participation. The `confirmation_text` is publicly readable from the swap's state. Attack Path B requires real ICP but the attacker receives SNS neurons in return. Both paths are trivially repeatable.

## Recommendation
Enforce that when `buyer` is non-empty and differs from the caller, `confirmation_text` must not be supplied by the caller:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let specified = PrincipalId::from_str(&arg.buyer).unwrap();
    if specified != caller_principal_id() && arg.confirmation_text.is_some() {
        ic_cdk::trap("confirmation_text may not be submitted on behalf of another principal");
    }
    specified
};
```

Alternatively, require that `buyer`, when non-empty, must equal the caller, eliminating third-party registration entirely and matching the authentication pattern used by other IC canisters.

## Proof of Concept
Minimal deterministic integration test using PocketIC or `StateMachine`:

```rust
// 1. Open a swap with confirmation_text = "I agree to the terms."
// 2. Transfer min_participant_icp_e8s ICP from Bob's identity to
//    Account { owner: swap_canister_id, subaccount: principal_to_subaccount(&bob) }
// 3. From Alice's identity, call:
//    refresh_buyer_tokens({
//        buyer: bob.to_string(),
//        confirmation_text: Some("I agree to the terms.".to_string()),
//    })
// 4. Assert: swap.buyers contains Bob's principal with amount >= min_participant_icp_e8s
// 5. Assert: Bob never sent any message to the swap canister
//    (verify via ingress history / message log)
```

This test is directly expressible using the existing `StateMachine`-based test infrastructure in `rs/sns/swap/tests/swap.rs` and `rs/sns/test_utils/src/state_test_helpers.rs`, where `execute_ingress_as` allows specifying the sender identity independently of the `buyer` field in the request payload.
