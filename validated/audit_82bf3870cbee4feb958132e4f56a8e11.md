Audit Report

## Title
Permanent ckBTC Loss Without Reimbursement When `BuildTxError::AmountTooLow` Occurs in `submit_pending_requests` — (`rs/bitcoin/ckbtc/minter/src/lib.rs`)

## Summary

The ckBTC minter enforces the minimum withdrawal check in `retrieve_btc` using a stale cached fee estimate (`fee_based_retrieve_btc_min_amount`), then burns the user's ckBTC unconditionally before any Bitcoin transaction is built. When `submit_pending_requests` later fetches a fresh fee estimate and `build_unsigned_transaction` returns `BuildTxError::AmountTooLow` due to a fee spike, the request is finalized as `FinalizedStatus::AmountTooLow` via `remove_retrieve_btc_request` with no reimbursement mint issued. The burned ckBTC is permanently lost.

## Finding Description

**Root cause — stale minimum check in `retrieve_btc`:**

`retrieve_btc` reads `s.fee_based_retrieve_btc_min_amount` from cached state, last written by a prior periodic `estimate_fee_per_vbyte` call: [1](#0-0) 

**ckBTC burned before transaction is built:**

After passing the stale minimum check, `burn_ckbtcs` is called unconditionally and the request is enqueued: [2](#0-1) 

**Fresh fee estimate in `submit_pending_requests`:**

`submit_pending_requests` calls `estimate_fee_per_vbyte` again, obtaining a potentially much higher fee: [3](#0-2) 

**`AmountTooLow` path silently drops the request with no reimbursement:**

When `build_unsigned_transaction` returns `BuildTxError::AmountTooLow`, each request in the batch is passed to `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow` — no reimbursement is scheduled: [4](#0-3) 

`remove_retrieve_btc_request` only records an event and pushes to finalized requests; it does not enqueue any reimbursement: [5](#0-4) 

**Contrast with `InvalidTransaction` path which does reimburse:**

The `InvalidTransaction` arm calls `reimburse_canceled_requests`, demonstrating the asymmetry is not intentional design: [6](#0-5) 

**Reimbursement is structurally impossible for this path:**

`WithdrawalReimbursementReason` has no `AmountTooLow` variant, so the reimbursement infrastructure cannot be used even if the call site were fixed: [7](#0-6) 

The `reimbursement_account` field is populated on every `RetrieveBtcRequest` but is never read in the `AmountTooLow` code path. [8](#0-7) 

The same no-reimbursement pattern also applies to `BuildTxError::DustOutput`: [9](#0-8) 

## Impact Explanation

Any user who calls `retrieve_btc` at or near `fee_based_retrieve_btc_min_amount` during a period of low fees will have their ckBTC burned. If Bitcoin network fees spike before `submit_pending_requests` processes the batch, the request is finalized as `AmountTooLow` and the burned ckBTC is permanently lost with no reimbursement. This is a concrete, permanent loss of in-scope chain-key ledger assets (ckBTC) triggered by an unprivileged user action under realistic network conditions. This matches the **High** impact class: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation

- The minimum fee floor is 1,500 millisat/byte (1.5 sat/vbyte), yielding `fee_based_retrieve_btc_min_amount` = 100,000 sats at low fees. [10](#0-9) 
- Bitcoin fees of 500–1,000+ sat/vbyte have occurred during peak congestion (2023 Ordinals/BRC-20 events); at ~710 sat/vbyte a single-input P2WPKH transaction (~141 vbytes) consumes the entire 100,000 sat withdrawal.
- The window between `retrieve_btc` accepting the request and `submit_pending_requests` running spans multiple IC rounds, during which the Bitcoin mempool can change significantly.
- No special privilege is required — any user calling `retrieve_btc` at the advertised minimum is at risk.
- Multiple users can be affected simultaneously in a single batch during a fee spike.

## Recommendation

In `submit_pending_requests`, when `BuildTxError::AmountTooLow` or `BuildTxError::DustOutput` is encountered, call `reimburse_canceled_requests` instead of `remove_retrieve_btc_request`. This requires:

1. Adding an `AmountTooLow` variant to `WithdrawalReimbursementReason` in `rs/bitcoin/ckbtc/minter/src/reimbursement/mod.rs`.
2. In the `AmountTooLow` arm of `submit_pending_requests`, computing a `reimbursement_fee` (bounded by `retrieve_btc_min_amount`, consistent with the existing assertion in `reimburse_canceled_requests`) and calling `reimburse_canceled_requests(s, batch, reason, reimbursement_fee, runtime)`.
3. Applying the same fix to the `DustOutput` arm for the affected request.

## Proof of Concept

State-machine test outline:

1. Initialize minter with `retrieve_btc_min_amount = 100_000`.
2. Set fee percentiles to `[FeeRate::from_millis_per_byte(1_500); 100]` and call `refresh_fee_percentiles` → `fee_based_retrieve_btc_min_amount` = 100,000.
3. Call `retrieve_btc(amount = 100_000, address = <valid>)` → succeeds, ckBTC burned, request enqueued.
4. Set fee percentiles to `[FeeRate::from_millis_per_byte(1_000_000); 100]` (1,000 sat/vbyte) **before** advancing the timer.
5. Advance time past `max_time_in_queue_nanos` to trigger `submit_pending_requests`.
6. Assert: `retrieve_btc_status(block_index) == AmountTooLow`.
7. Assert: user's ckBTC balance is **not** restored (no reimbursement mint occurred).
8. Assert: `retrieve_btc_status_v2(block_index)` is **not** `WillReimburse` or `Reimbursed`.

pub fn remove_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    status: FinalizedStatus,
    runtime: &R,
) {
    record_event(
        EventType::RemovedRetrieveBtcRequest {
            block_index: request.block_index,
        },
        runtime,
    );

    state.push_finalized_request(FinalizedBtcRequest {
        request: request.into(),
        state: status,
    });
}
