### Title
Missing Field Element and On-Curve Point Validation in Contract-Mode BN128 Precompiles — (File: `engine-sdk/src/bn128.rs`)

### Summary
The production WASM (`#[cfg(feature = "contract")]`) implementations of `alt_bn128_g1_sum`, `alt_bn128_g1_scalar_multiple`, and `alt_bn128_pairing` perform no field element, on-curve, or subgroup validation before forwarding raw bytes to NEAR host functions. The standalone (`#[cfg(not(feature = "contract"))]`) counterparts use `ark_bn254` and fully validate all inputs. This behavioral divergence means that in production, invalid curve points are silently forwarded to the NEAR runtime, whose behavior with malformed inputs is the only guard — and that guard is not guaranteed to match Ethereum EIP-196/197 semantics.

### Finding Description

The standalone path validates every input:

- `read_fq` calls `Fq::deserialize_uncompressed`, rejecting non-canonical field elements with `Bn254Error::FieldPointNotAMember`. [1](#0-0) 
- `new_g1_point` / `new_g2_point` call `is_on_curve()` and `is_in_correct_subgroup_assuming_on_curve()`, rejecting off-curve or wrong-subgroup points with `Bn254Error::AffineGFailedToCreate`. [2](#0-1) 
- `pairing_check` (standalone) calls `read_g1_point` and `read_g2_point` for every pair, propagating errors. [3](#0-2) 

The contract path performs **none** of these checks:

`alt_bn128_g1_sum` (contract): reverses byte order and immediately calls `exports::alt_bn128_g1_sum`, always returning `Ok(output)` with no validation. [4](#0-3) 

`alt_bn128_g1_scalar_multiple` (contract): only normalizes the scalar modulo the group order; the G1 point is never checked for field membership, on-curve, or subgroup membership. [5](#0-4) 

`alt_bn128_pairing` (contract): validates only that `input_bytes.len()` is a multiple of `PAIR_ELEMENT_LEN`, then reverses byte order and calls `exports::alt_bn128_pairing_check`, returning `Ok(result == 1)` with no field, on-curve, or subgroup checks on any G1 or G2 point. [6](#0-5) 

These three functions are the sole backend for the EVM precompiles `Bn256Add` (0x06), `Bn256Mul` (0x07), and `Bn256Pair` (0x08): [7](#0-6) [8](#0-7) [9](#0-8) 

### Impact Explanation

The most critical case is `Bn256Pair` (precompile 0x08). ZK-proof verifier contracts deployed on Aurora (e.g., token bridges, privacy protocols, rollup exits) use `ecPairing` to verify proofs. EIP-197 requires that G2 points be validated for subgroup membership; without this check, an attacker can supply a G2 point that lies on the BN254 curve but outside the prime-order subgroup. If the NEAR host function `alt_bn128_pairing_check` does not enforce subgroup membership (a known omission in several BN128 implementations), the pairing result is mathematically undefined and can be manipulated to return `1` (true) for a forged proof. A successful forgery allows the attacker to drain funds locked behind the verifier.

For `Bn256Add` and `Bn256Mul`, off-curve inputs that the standalone path rejects with `Err` are silently forwarded in the contract path, returning `Ok(output)`. Solidity contracts that rely on a revert from these precompiles to detect invalid inputs will instead receive a (potentially garbage) success result, breaking any guard logic built on that assumption.

### Likelihood Explanation

Any unprivileged EVM user can call a deployed ZK verifier contract with crafted calldata containing invalid curve points. The entry path is fully attacker-controlled: craft calldata → call verifier → verifier calls `ecPairing` (0x08) → Aurora routes to `Bn256Pair::run_inner` → `alt_bn128_pairing` (contract) → NEAR host function with unvalidated points. No admin access or special privilege is required. The likelihood is high wherever a ZK-proof-based application holding user funds is deployed on Aurora.

### Recommendation

Add explicit field element, on-curve, and subgroup validation to all three `#[cfg(feature = "contract")]` functions in `engine-sdk/src/bn128.rs` before calling any NEAR host function. The standalone implementations already contain the correct validation logic (`read_fq`, `new_g1_point`, `new_g2_point`); this logic should be extracted into a shared, feature-agnostic validation layer and called unconditionally in both paths. If the NEAR host function panics on invalid input, the error must be caught and converted to a proper `Err(Bn254Error::...)` rather than allowing an uncontrolled WASM trap.

### Proof of Concept

1. Deploy a Solidity ZK verifier on Aurora that calls `ecPairing` (0x08) and reverts if the result is `false`.
2. Construct a pairing input where the G2 point has coordinates satisfying the curve equation `y² = x³ + b/ξ` (on the twist) but is **not** in the prime-order subgroup (a low-order or cofactor point).
3. Call the verifier with this input.
4. In the standalone path, `new_g2_point` → `is_in_correct_subgroup_assuming_on_curve()` returns `false` → `Err(AffineGFailedToCreate)` → precompile reverts → verifier reverts. [10](#0-9) 
5. In the contract (production) path, no such check exists; the raw bytes are forwarded to `exports::alt_bn128_pairing_check`. If the NEAR runtime does not enforce subgroup membership, the pairing returns an attacker-controlled result, and the verifier accepts a forged proof. [11](#0-10)

    pub fn pairing_check<'a>(
        mut pairs: impl ExactSizeIterator<Item = (&'a [u8], &'a [u8])>,
    ) -> Result<bool, Bn254Error> {
        let pairs_count = pairs.len();
        let (g1_points, g2_points) = pairs.try_fold(
            (
                Vec::with_capacity(pairs_count),
                Vec::with_capacity(pairs_count),
            ),
            |(mut g1_acc, mut g2_acc), (g1_bytes, g2_bytes)| {
                let g1 = read_g1_point(g1_bytes)?;
                let g2 = read_g2_point(g2_bytes)?;

                // Skip pairs where either point is at infinity
                if !g1.is_zero() && !g2.is_zero() {
                    g1_acc.push(g1);
                    g2_acc.push(g2);
                }

                Ok((g1_acc, g2_acc))
            },
        )?;

        if g1_points.is_empty() {
            return Ok(true);
        }

        let pairing_result = Bn254::multi_pairing(&g1_points, &g2_points);
        Ok(pairing_result.0.is_one())
    }

#[cfg(feature = "contract")]
pub fn alt_bn128_g1_sum(input_bytes: &[u8]) -> Result<[u8; 64], Bn254Error> {
    // Buffer is: [0, P1(G1_LEN), 0, P2(G1_LEN)]
    const BUFFER_LEN: usize = 2 + G1_LEN * 2;
    // Register ID to store the result
    const REGISTER_ID: u64 = 1;

    let mut bytes = [0u8; BUFFER_LEN];

    // --- Process P1 (First 64 bytes of input) ---
    // P1.X: 1..1 + FQ_LEN
    write_reversed_chunk(&mut bytes[1..=FQ_LEN], input_bytes, 0);
    // P1.Y: 1 + FQ_LEN..1 + G1_LEN
    write_reversed_chunk(&mut bytes[1 + FQ_LEN..=G1_LEN], input_bytes, FQ_LEN);

    // --- Process P2 (Next 64 bytes of input) ---
    // P2.X
    write_reversed_chunk(
        &mut bytes[2 + G1_LEN..2 + G1_LEN + FQ_LEN],
        input_bytes,
        G1_LEN,
    );
    // P2.Y
    write_reversed_chunk(
        &mut bytes[2 + G1_LEN + FQ_LEN..],
        input_bytes,
        G1_LEN + FQ_LEN,
    );

    let value_ptr = bytes.as_ptr() as u64;
    let value_len = bytes.len() as u64;
    // Prepare output buffer
    let mut output = [0u8; G1_LEN];
    // Call the NEAR host function
    unsafe {
        exports::alt_bn128_g1_sum(value_len, value_ptr, REGISTER_ID);
        exports::read_register(REGISTER_ID, output.as_ptr() as u64);
    }

    // X = LE -> BE
    output[0..FQ_LEN].reverse();
    // Y = LE -> BE
    output[FQ_LEN..G1_LEN].reverse();
    Ok(output)
}

pub fn alt_bn128_g1_scalar_multiple(input_bytes: &[u8]) -> Result<[u8; 64], Bn254Error> {
    use aurora_engine_types::U256;
    // Buffer is: [P1(G1_LEN), Scalar(SCALAR_LEN)] -> Total 96 bytes
    const BUFFER_LEN: usize = G1_LEN + SCALAR_LEN;
    // Register ID to store the result
    const REGISTER_ID: u64 = 1;

    const BN128_SCALAR_ORDER: U256 = U256([
        0x43e1f593f0000001,
        0x2833e84879b97091,
        0xb85045b68181585d,
        0x30644e72e131a029,
    ]);
    let mut bytes = [0u8; BUFFER_LEN];

    // 1. Process Point G1 (First 64 bytes)
    // X coordinate (Input 0..32)
    write_reversed_chunk(&mut bytes[0..FQ_LEN], input_bytes, 0);
    // Y coordinate (Input 32..64)
    write_reversed_chunk(&mut bytes[FQ_LEN..G1_LEN], input_bytes, FQ_LEN);

    // 2. Process Scalar (Next 32 bytes)
    // Scalar (Input 64..96)
    write_reversed_chunk(&mut bytes[G1_LEN..], input_bytes, G1_LEN);

    let scalar_slice = &mut bytes[G1_LEN..];
    let scalar = U256::from_little_endian(scalar_slice);
    if scalar >= BN128_SCALAR_ORDER {
        let normalized = scalar % BN128_SCALAR_ORDER;
        scalar_slice.copy_from_slice(&normalized.to_little_endian());
    }

    let value_ptr = bytes.as_ptr() as u64;
    let value_len = bytes.len() as u64;
    // Prepare output buffer
    let mut output = [0u8; G1_LEN];
    // Call the NEAR host function
    unsafe {
        exports::alt_bn128_g1_multiexp(value_len, value_ptr, REGISTER_ID);
        exports::read_register(REGISTER_ID, output.as_ptr() as u64);
    }

    // 3. Output Point G1 (Little -> Big)
    // X coordinate
    output[0..FQ_LEN].reverse();
    // Y coordinate
    output[FQ_LEN..G1_LEN].reverse();

    Ok(output)
}

#[cfg(feature = "contract")]
pub fn alt_bn128_pairing(input_bytes: &[u8]) -> Result<bool, Bn254Error> {
    use aurora_engine_types::Vec;

    // Empty input implies the product of an empty set, which is the multiplicative identity (1).
    // Therefore, the check passes.
    if input_bytes.is_empty() {
        return Ok(true);
    }

    // Validate input length
    if !input_bytes.len().is_multiple_of(PAIR_ELEMENT_LEN) {
        return Err(Bn254Error::InvalidPairLength);
    }

    let mut bytes = {
        let mut v = Vec::with_capacity(input_bytes.len());
        v.extend_from_slice(input_bytes);
        v
    };

    for pair_chunk in bytes.chunks_exact_mut(PAIR_ELEMENT_LEN) {
        // --- Process G1 (2 * 32 bytes) ---
        // P1.X
        pair_chunk[0..FQ_LEN].reverse();
        // P1.Y
        pair_chunk[FQ_LEN..G1_LEN].reverse();

        // --- Process G2 (2 * 64 bytes) ---
        // P2.X (64 bytes)
        pair_chunk[G1_LEN..G2_LEN].reverse();

        // P2.Y (64 bytes)
        pair_chunk[G2_LEN..PAIR_ELEMENT_LEN].reverse();
    }

    let value_ptr = bytes.as_ptr() as u64;
    let value_len = bytes.len() as u64;

    // Call Host Function.
    let result = unsafe { exports::alt_bn128_pairing_check(value_len, value_ptr) };
    Ok(result == 1)
