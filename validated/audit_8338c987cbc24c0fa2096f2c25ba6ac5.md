Audit Report

## Title
VIP Callers Blocked by Non-VIP Slot Exhaustion in `try_borrow_slot` — (`rs/nervous_system/clients/src/management_canister_client.rs`)

## Summary
`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` unconditionally rejects all callers when `available_slot_count == 0`, including VIP callers (NNS canisters) that are designed to consume zero slots. An unprivileged attacker can exhaust all 167 shared slots via the public `canister_status` endpoint on NNS Root, blocking governance-triggered `take_canister_snapshot` and `load_canister_snapshot` operations for the duration of the attack.

## Finding Description
In `try_borrow_slot` (lines 264–287 of `rs/nervous_system/clients/src/management_canister_client.rs`), the function correctly sets `used_slot_count = 0` for VIP callers, and the `SlotLoan` Drop implementation (lines 365–372) correctly adds back `used_slot_count` on completion — meaning VIP calls are designed to be slot-neutral. However, the exhaustion guard at line 269 fires unconditionally before the subtraction:

```rust
if *available_slot_count == 0 {   // blocks ALL callers, including VIP
    return Err((code, message));
}
*available_slot_count = available_slot_count.saturating_sub(used_slot_count); // never reached for VIP when count==0
```

A VIP caller with `used_slot_count = 0` would not change the counter at all, but it never reaches that line when the pool is at zero.

The exploit path:
1. `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` is fixed at 167 (canister.rs line 50).
2. `canister_status` on NNS Root is an unrestricted `#[update]` (canister.rs lines 88–98) — no access control.
3. An attacker sends 167 concurrent `canister_status` calls to NNS Root. Each call invokes `try_borrow_slot`, decrements the counter by 1, and suspends at the inner management canister `await`. All 167 slots are occupied simultaneously.
4. NNS Governance (a VIP caller, `is_caller_vip = true`) calls `take_canister_snapshot` or `load_canister_snapshot` (canister.rs lines 245–268). Both call `new_management_canister_client()` (lines 53–67), which sets `is_caller_vip = true`, then call `try_borrow_slot`.
5. `try_borrow_slot` sees `available_slot_count == 0` and returns `Err(SysTransient, "Unavailable. Maybe, try again later?")` — the governance call fails.

The existing unit test (tests.rs line 134) uses capacity=2 and starts VIP calls before non-VIP calls fill the pool; it does not test the `available_slot_count == 0` edge case where the guard fires before the VIP path is reached.

## Impact Explanation
This is a sustained, low-cost DoS against NNS governance disaster-recovery operations (`take_canister_snapshot`, `load_canister_snapshot`). If a time-sensitive rollback is required (e.g., corrupted ledger or governance canister state), the attacker can continuously replenish 167 in-flight `canister_status` calls to sustain the block indefinitely. This matches the allowed High impact: **Application/platform-level DoS with concrete NNS governance harm** — blocking certified snapshot/restore operations that are the primary recovery mechanism for NNS canisters.

## Likelihood Explanation
No privileged access is required. The `canister_status` endpoint is explicitly documented as public. The attacker needs only cycles to send 167 concurrent inter-canister calls and sustain them. The IC execution model guarantees all 167 calls can be in-flight simultaneously (each suspends at the `await` point). The attack is repeatable and cheap to sustain.

## Recommendation
Skip the exhaustion guard entirely for VIP callers in `try_borrow_slot`, since they consume zero slots and cannot deplete the pool:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if !self.is_caller_vip && *available_slot_count == 0 {
                let code = RejectCode::SysTransient as i32;
                return Err((code, "Unavailable. Maybe, try again later?".to_string()));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

A corresponding unit test should be added: with capacity=2, fill both slots with non-VIP calls, then assert a VIP call succeeds.

## Proof of Concept
State-machine or PocketIC test:
1. Initialize NNS Root with a mock management canister client that stalls indefinitely before responding.
2. Send 167 concurrent `canister_status` calls from an unprivileged principal to NNS Root; assert `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT == 0`.
3. Send a `load_canister_snapshot` call from the NNS Governance canister principal (VIP).
4. Assert the response is `Err` with reject code `SysTransient` — confirming VIP is blocked despite `is_caller_vip = true` and `used_slot_count = 0`.

The minimal unit-level reproduction requires only the existing `LimitedOutstandingCallsManagementCanisterClient` test harness (tests.rs line 134) with capacity=2: fill both slots with non-VIP calls, then attempt a VIP call and observe it erroneously fails.
