### Title
Unsigned `max_gas_price` in `SubmitArgs` Allows Relayer to Steal User Gas Fees — (File: `engine-types/src/parameters/engine.rs`, `engine/src/contract_methods/evm_transactions.rs`)

---

### Summary

The `submit_with_args` entrypoint accepts a `SubmitArgs` struct containing a signed Ethereum transaction (`tx_data`) alongside two unsigned, relayer-controlled fields: `max_gas_price` and `gas_token_address`. Because these fields are not part of the signed transaction, any relayer (or NEAR block producer) can strip or replace `max_gas_price` before submission, causing the user to pay the full gas price from their signed transaction rather than the capped price they intended. The excess gas fee flows directly to the relayer's EVM address as a priority-fee reward.

---

### Finding Description

`SubmitArgs` is defined as:

```rust
pub struct SubmitArgs {
    pub tx_data: Vec<u8>,                    // signed Ethereum transaction
    pub max_gas_price: Option<u128>,         // NOT signed — relayer-controlled
    pub gas_token_address: Option<Address>,  // NOT signed — relayer-controlled
}
``` [1](#0-0) 

The `submit_with_args` handler reads the entire `SubmitArgs` from NEAR input and passes it directly to `engine::submit` without any verification that `max_gas_price` or `gas_token_address` were authorized by the Ethereum transaction signer:

```rust
pub fn submit_with_args<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    ...
    let args: SubmitArgs = io.read_input_borsh()?;
    ...
    let result = engine::submit(io, env, &args, state, current_account_id, relayer_address, handler)?;
``` [2](#0-1) 

Inside `engine::submit`, `max_gas_price` is used in `charge_gas` as a cap on the effective priority fee:

```rust
let priority_fee_per_gas = max_gas_price.map_or(priority_fee_per_gas, |price| {
    price.min(priority_fee_per_gas)
});
let effective_gas_price = priority_fee_per_gas + block_base_fee_per_gas;
``` [3](#0-2) 

The relayer then receives `priority_fee_per_gas * gas_used` as a direct balance credit:

```rust
if !relayer_reward.is_zero() {
    add_balance(io, relayer, relayer_reward)?;
}
``` [4](#0-3) 

The relayer's address is derived from `predecessor_account_id`, i.e., the NEAR account that calls `submit_with_args`. [5](#0-4) 

---

### Impact Explanation

A user who signs an Ethereum transaction with `gas_price = P` and wraps it in `SubmitArgs { max_gas_price: Some(C) }` (where `C < P`) intends to pay at most `C * gas_used` in ETH gas fees. A malicious relayer can repackage the same signed `tx_data` with `max_gas_price: None`, causing the user to pay `P * gas_used` instead. The difference `(P - C) * gas_used` is credited to the relayer's EVM address. This is a direct, in-motion theft of user ETH funds. The `gas_token_address` field is similarly unsigned and relayer-controlled, though its consumption path in the production engine was not fully traceable in the available index.

**Impact class: Critical — direct theft of user funds in motion.**

---

### Likelihood Explanation

The attack requires no special privilege. Any NEAR account can call `submit_with_args`. In the relayer model (where users send signed Ethereum transactions to a relayer service for NEAR submission), the relayer is the natural attacker. Even without a dedicated relayer, a NEAR block producer observing the mempool can front-run a user's `submit_with_args` call by resubmitting the same `tx_data` with `max_gas_price` stripped. The signed Ethereum transaction is fully public once broadcast, and the Ethereum nonce ensures only one execution succeeds — whichever NEAR transaction is included first wins, consuming the nonce and leaving the user's original call to fail.

**Likelihood: High** — the relayer is a trusted intermediary by design, and the protocol provides no mechanism to enforce the user's intended `max_gas_price`.

---

### Recommendation

Include `max_gas_price` and `gas_token_address` inside the signed Ethereum transaction payload, or require a separate EIP-712-style signature from the Ethereum sender over the full `SubmitArgs` struct (including `max_gas_price` and `gas_token_address`). Alternatively, derive `max_gas_price` solely from the transaction's own `max_fee_per_gas` / `max_priority_fee_per_gas` fields, which are already covered by the Ethereum signature, and remove the unsigned override entirely.

---

### Proof of Concept

1. User signs an Ethereum legacy transaction with `gas_price = 100` and `gas_limit = 30_000`.
2. User intends to submit via `submit_with_args` with `SubmitArgs { tx_data: signed_tx, max_gas_price: Some(5), gas_token_address: None }`, expecting to pay at most `5 * gas_used` ETH.
3. Relayer (or front-runner) intercepts the signed `tx_data` and calls `submit_with_args` with `SubmitArgs { tx_data: signed_tx, max_gas_price: None, gas_token_address: None }`.
4. `charge_gas` computes `priority_fee_per_gas = 100` (uncapped), `effective_gas_price = 100`.
5. User's ETH balance is debited `100 * gas_used` instead of `5 * gas_used`.
6. Relayer's EVM address receives `100 * gas_used` as `relayer_reward`.
7. User's original NEAR transaction (if it reaches the chain) fails with an incorrect-nonce error because the Ethereum nonce was already consumed.

The test utility `submit_transaction_with_args` in `engine-tests/src/utils/mod.rs` demonstrates that `max_gas_price` and `gas_token_address` are assembled by the caller after signing, confirming they are never covered by the Ethereum signature. [6](#0-5)

pub fn submit_with_args<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let args: SubmitArgs = io.read_input_borsh()?;
        let current_account_id = env.current_account_id();
        let relayer_address = predecessor_address(&env.predecessor_account_id());
        let result = engine::submit(
            io,
            env,
            &args,
            state,
            current_account_id,
            relayer_address,
            handler,
        )?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);

        Ok(result)
    })
}
