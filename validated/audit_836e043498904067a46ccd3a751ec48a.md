Audit Report

## Title
Synchronous Unbounded BFS in SNS `cast_vote_and_cascade_follow` Lacks Instruction-Limit Protection, Enabling Governance Canister DoS — (`rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance `cast_vote_and_cascade_follow` is a plain synchronous function with no instruction-limit guard. Its BFS loop iterates over all follower neurons in a single update-call message with no soft or hard instruction checks. The NNS governance has an explicit, well-tested mitigation for this exact scenario (`SOFT_VOTING_INSTRUCTIONS_LIMIT`, `HARD_VOTING_INSTRUCTIONS_LIMIT`, `noop_self_call_if_over_instructions`, and a `VotingStateMachine` timer fallback); the SNS governance has none of these. An attacker who stakes enough tokens to create a large star-topology following graph can trigger instruction exhaustion, causing the canister to trap on every vote attempt for any open proposal.

## Finding Description
`cast_vote_and_cascade_follow` in the SNS is a plain `fn` (not `async`): [1](#0-0) 

The BFS loop runs to completion in a single call with no instruction checks, no soft-limit break, and no timer fallback: [2](#0-1) 

It is called synchronously from `register_vote` with no async boundary: [3](#0-2) 

A grep for `INSTRUCTION`, `instruction_limit`, `over_soft`, or `noop_self_call` in `rs/sns/governance/src/governance.rs` returns zero matches, confirming the complete absence of any instruction guard.

By contrast, the NNS governance defines explicit soft and hard limits: [4](#0-3) 

Its `cast_vote_and_cascade_follow` is `async` and breaks work across messages via `noop_self_call_if_over_instructions`, deferring remaining work to a timer: [5](#0-4) 

The SNS has none of this infrastructure.

## Impact Explanation
When the BFS processes N follower neurons in a single update-call message, each iteration performs `BTreeMap` lookups, `vote_from_ballots_following` evaluation, and `BTreeSet` insertions. The IC per-message instruction budget for update calls is finite (~5–40 billion instructions depending on subnet configuration). The NNS comment states its HARD limit of 750 billion "leaves room for 750 thousand neurons with complex following," implying roughly ~1 million instructions per neuron per BFS step. At the IC's update-call limit, exhaustion is reachable with tens of thousands of followers in a star topology — a number achievable in SNS instances with low token prices or high configured `max_number_of_neurons`. A trap rolls back state but leaves the canister alive; however, repeated triggering on every open proposal renders the SNS governance canister unable to process votes. This matches the allowed impact: **High — Application/platform-level DoS of SNS governance, not based on raw volumetric DDoS.**

## Likelihood Explanation
The attacker must stake governance tokens to create N follower neurons and configure them to follow a hub neuron — both standard, permissionless operations. The economic cost of staking is the primary barrier, but it is not unrealistic: SNS instances with low token prices or high `max_number_of_neurons` limits make the attack practical. The attack is repeatable on every open proposal and does not require any privileged access. The NNS team already recognized and mitigated this exact risk, which confirms the threat model is realistic.

## Recommendation
Port the NNS mitigation to the SNS `cast_vote_and_cascade_follow`:
1. Make the function `async`.
2. Add `SOFT_VOTING_INSTRUCTIONS_LIMIT` and `HARD_VOTING_INSTRUCTIONS_LIMIT` constants mirroring those in `rs/nns/governance/src/voting.rs`.
3. Check `over_soft_message_limit()` inside the BFS loop and call `noop_self_call_if_over_instructions` to break work across messages.
4. Persist in-progress BFS state (e.g., via a `VotingStateMachine`) and drain it in a timer job, mirroring the NNS pattern in `rs/nns/governance/src/voting.rs`.

## Proof of Concept
1. Attacker stakes tokens and creates N neurons in the target SNS, all configured to follow hub neuron H via `manage_neuron { command: Follow { function_id, followees: [H] } }`.
2. A proposal is submitted; all N+1 neurons receive ballots.
3. Attacker calls `manage_neuron { command: RegisterVote { proposal_id, vote: Yes } }` for neuron H.
4. `register_vote` → `cast_vote_and_cascade_follow` → synchronous BFS over N followers in one message.
5. At sufficient N, the canister traps on instruction exhaustion; the trap is reproducible on every subsequent vote attempt for any open proposal.

A deterministic benchmark can be written mirroring the NNS bench pattern in `rs/nns/governance/src/governance/benches.rs` (lines 331–351): construct a star topology with increasing N, call `cast_vote_and_cascade_follow`, and assert `instruction_counter() < IC_UPDATE_CALL_LIMIT`. [6](#0-5)

    pub async fn cast_vote_and_cascade_follow(
        &mut self,
        proposal_id: ProposalId,
        voting_neuron_id: NeuronId,
        vote_of_neuron: Vote,
        topic: Topic,
    ) {
        let voting_started = self.env.now();

        if !self.heap_data.proposals.contains_key(&proposal_id.id) {
            // This is a critical error, but there is nothing that can be done about it
            // at this place.  We somehow have a vote for a proposal that doesn't exist.
            eprintln!(
                "error in cast_vote_and_cascade_follow: Proposal not found: {}",
                proposal_id.id
            );
            return;
        }

        // First we cast the ballot.
        self.record_neuron_vote(proposal_id, voting_neuron_id, vote_of_neuron, topic);

        // We process until voting is finished, and then do any other work that fits into the soft
        // limit of the current message.  Votes are guaranteed to be recorded before the function
        // returns, but recent_ballots for neurons might be recorded later in a timer job.  This
        // ensures we return to the caller in a reasonable amount of time.
        let mut is_voting_finished = false;

        while !is_voting_finished {
            // Now we process until we are done or we are over a limit and need to
            // make a self-call.
            with_voting_state_machines_mut(|voting_state_machines| {
                voting_state_machines.with_machine(proposal_id, topic, |machine| {
                    self.process_machine_until_soft_limit(machine, over_soft_message_limit);
                    is_voting_finished = machine.is_voting_finished();
                });
            });

            // This returns an error if we hit the hard limit, which should basically never happen
            // in production, but we need a way out of this loop in the worst case to prevent
            // the canister from being unable to upgrade.
            if let Err(e) = noop_self_call_if_over_instructions(
                SOFT_VOTING_INSTRUCTIONS_LIMIT,
                Some(HARD_VOTING_INSTRUCTIONS_LIMIT),
            )
            .await
            {
                println!(
                    "Error in cast_vote_and_cascade_follow, \
                        voting will be processed in timers: {}",
                    e
                );
                break;
            }
        }
