### Title
Zero-Address Recipient in `receive_base_tokens` Causes Permanent ETH Freeze — (File: engine/src/engine.rs)

---

### Summary

The `receive_base_tokens` function in `engine/src/engine.rs` accepts a zero Ethereum address as the recipient when minting bridged ETH. Because the function succeeds silently (returning `Ok(None)`), the `ft_on_transfer` handler signals a full refund of zero tokens to the NEP-141 connector, permanently crediting the bridged ETH to `address(0)` with no recovery path.

---

### Finding Description

`FtTransferMessageData::try_from` in `engine-types/src/parameters/connector.rs` parses the 40-character hex `msg` field of an `ft_on_transfer` call into an `Address` with no zero-address guard:

```
if message.len() == 40 {
    let mut address_bytes = [0; 20];
    hex::decode_to_slice(message, &mut address_bytes)...;
    Address::from_array(address_bytes)   // zero address accepted silently
}
```

`receive_base_tokens` then uses that address verbatim:

```rust
let receipient = message_data.recipient;          // may be Address::zero()
let balance    = get_balance(&self.io, &receipient);
let new_balance = balance.checked_add(amount)...;
set_balance(&mut self.io, &receipient, &new_balance);  // credits zero address
Ok(None)                                          // success — no refund triggered
```

Because the function returns `Ok(None)`, the `ft_on_transfer` dispatcher in `engine/src/contract_methods/connector.rs` takes the `else` branch and sets `amount_to_return = 0`:

```rust
let amount_to_return = if let Err(_err) = &result {
    args.amount.as_u128()   // refund path — NOT taken
} else {
    0                       // taken: connector keeps the tokens
};
```

The NEP-141 ETH connector therefore retains the full bridged amount while the EVM balance is credited only to `address(0)`, from which no private key can ever spend.

**Contrast with `receive_erc20_tokens`:** when the recipient is `address(0)` in the ERC-20 path, OpenZeppelin's `_mint` reverts inside the EVM, the error propagates through `and_then(submit_result_or_err)?`, and `ft_on_transfer` correctly returns the full amount to the sender. The base-ETH path has no equivalent guard.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any ETH bridged via `ft_transfer_call` with `msg = "0000000000000000000000000000000000000000"` is irrecoverably credited to `address(0)` inside the Aurora EVM. The NEP-141 connector balance is debited, the EVM zero-address balance is incremented, and no refund is issued. The funds are permanently inaccessible.

---

### Likelihood Explanation

**Medium.**

The `msg` field is fully user-controlled. A user who mistakenly omits or zero-fills the recipient field (e.g., a buggy integration, a frontend that defaults to an empty address, or a developer testing the bridge) will silently lose all bridged ETH. No privileged access is required; any NEAR account that can call `ft_transfer_call` on the ETH connector can trigger this path.

---

### Recommendation

Reject a zero recipient address at the earliest possible point. Either:

1. In `FtTransferMessageData::try_from` (`engine-types/src/parameters/connector.rs`), after constructing the `Address`, return `Err(ParseOnTransferMessageError::InvalidHexData)` (or a new dedicated variant) if `address_bytes == [0u8; 20]`.
2. Or, in `receive_base_tokens` (`engine/src/engine.rs`), add an explicit guard before `set_balance`:
   ```rust
   if receipient == Address::zero() {
       return Err(errors::ERR_ZERO_ADDRESS_RECIPIENT.into());
   }
   ```

Either fix causes `ft_on_transfer` to take the error branch and return the full amount to the sender, matching the behaviour already present in the ERC-20 path.

---

### Proof of Concept

1. A user holds NEP-141 ETH on the ETH connector account.
2. The user calls `ft_transfer_call` on the connector:
   ```json
   {
     "receiver_id": "aurora",
     "amount": "1000000000000000000",
     "msg": "0000000000000000000000000000000000000000"
   }
   ```
3. The connector calls `ft_on_transfer` on the Aurora engine with `msg = "0000000000000000000000000000000000000000"`.
4. `FtTransferMessageData::try_from` succeeds, returning `recipient = Address::zero()`. [1](#0-0) 
5. `receive_base_tokens` calls `set_balance(&mut self.io, &Address::zero(), &new_balance)` and returns `Ok(None)`. [2](#0-1) 
6. `ft_on_transfer` sets `amount_to_return = 0` and returns without refunding. [3](#0-2) 
7. The user's NEP-141 ETH balance is permanently reduced; the ETH exists only at `address(0)` in the EVM state and is unspendable.
