Audit Report

## Title
SNS Extension Operations Permanently Blocked When WASM Removed from ALLOWED_EXTENSIONS — (File: `rs/sns/governance/src/extensions.rs`)

## Summary
The `ALLOWED_EXTENSIONS` whitelist in `rs/sns/governance/src/extensions.rs` is applied symmetrically to both registering and executing operations on an already-registered extension. When KongSwap's WASM hash was removed from this map on April 6, 2026 (leaving it intentionally empty), any SNS that had previously registered and funded a KongSwap TreasuryManager extension lost all ability to execute `withdraw`, upgrade, or deregister that extension through governance. No `DeregisterExtension` proposal action exists, leaving deposited treasury funds permanently inaccessible through the normal SNS governance flow.

## Finding Description
`ALLOWED_EXTENSIONS` is a `thread_local` `BTreeMap<[u8; 32], ExtensionSpec>` that is now intentionally empty: [1](#0-0) 

`validate_extension_wasm` rejects any hash not present in this map: [2](#0-1) 

This check is applied in three places:

**Registration** (`validate_register_extension`): [3](#0-2) 

**Upgrade** (`validate_upgrade_extension`): [4](#0-3) 

**Operation execution** — `validate_execute_extension_operation` calls `get_extension_spec_and_update_cache`, which fetches the installed WASM hash from the extension canister and calls `validate_extension_wasm` on it: [5](#0-4) [6](#0-5) 

`perform_execute_extension_operation` in `governance.rs` calls `validate_execute_extension_operation` with no bypass for withdrawal operations: [7](#0-6) 

The SNS `Action` enum contains `RegisterExtension` (Id=17), `ExecuteExtensionOperation` (Id=18), and `UpgradeExtension` (Id=19), but no `DeregisterExtension`: [8](#0-7) 

There is no alternative governance path (e.g., `ExecuteGenericNervousSystemFunction`) that can directly invoke the extension canister's `withdraw` method, because extension canisters are not registered as generic nervous system function targets and are controlled exclusively by Root and Governance.

## Impact Explanation
Any SNS that deposited treasury funds into a KongSwap TreasuryManager extension before April 6, 2026 now has those funds permanently inaccessible through the SNS governance flow. The extension canister remains registered with Root and controlled by Root and Governance, but every governance path to recover funds (`ExecuteExtensionOperation{withdraw}`, `UpgradeExtension`, and any hypothetical `DeregisterExtension`) is blocked. This constitutes permanent loss of SNS governance-controlled canister funds — matching the allowed High impact: *"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."*

## Likelihood Explanation
This is not theoretical. The KongSwap WASM hash was explicitly removed from `ALLOWED_EXTENSIONS` on April 6, 2026, as documented in the source comment. Any SNS that followed the standard governance flow (submit `RegisterExtension` proposal, then `ExecuteExtensionOperation{deposit}`) before that date is now in the locked state. No privileged access was required to enter this state — any neuron holder could submit the relevant proposals.

## Recommendation
1. **Add a `DeregisterExtension` proposal action** that removes an extension from the SNS Root registry without requiring the extension's WASM to be in `ALLOWED_EXTENSIONS`, analogous to how removal is handled elsewhere.
2. **Exempt `withdraw` operations from the `ALLOWED_EXTENSIONS` check** in `validate_execute_extension_operation`. Withdrawal is a risk-reducing operation and should not be gated by the same check that prevents new deposits. The check at line 1443 should be skipped or softened for operations classified as withdrawals.
3. **Establish a migration window** before removing any entry from `ALLOWED_EXTENSIONS`: SNSes with registered instances of that extension should be notified and given a governance window to withdraw funds before the hash is removed.

## Proof of Concept
```
1. SNS submits RegisterExtension proposal for KongSwap (WASM hash in ALLOWED_EXTENSIONS).
   → validate_register_extension (L1141) succeeds; extension registered with Root.

2. SNS submits ExecuteExtensionOperation{operation_name: "deposit"}.
   → validate_execute_extension_operation (L1502-1507) succeeds; funds transferred.

3. KongSwap WASM hash removed from ALLOWED_EXTENSIONS (April 6, 2026).
   → ALLOWED_EXTENSIONS is now empty (L48-54).

4. SNS submits ExecuteExtensionOperation{operation_name: "withdraw"}.
   → get_extension_spec_and_update_cache (L1443) calls validate_extension_wasm.
   → validate_extension_wasm_with_allowed (L952-959) returns Err: hash not in map.
   → Proposal fails. Funds remain locked.

5. SNS submits UpgradeExtension with any new WASM.
   → validate_upgrade_extension (L1277) calls validate_extension_wasm.
   → Fails: new hash also not in ALLOWED_EXTENSIONS.

6. No DeregisterExtension action exists (L741-753). Funds permanently inaccessible.
```

A deterministic integration test can reproduce this by: (a) populating `ALLOWED_EXTENSIONS` with a test hash via the `test` feature flag, (b) registering and funding an extension, (c) clearing `ALLOWED_EXTENSIONS`, and (d) asserting that `ExecuteExtensionOperation{withdraw}` returns `InvalidProposal`.
