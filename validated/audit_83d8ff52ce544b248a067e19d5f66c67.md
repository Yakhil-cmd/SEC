### Title
ERC20 Approval Race Condition (TOCTOU) in CW20ERC20Pointer Enables Double-Spend — (`contracts/src/CW20ERC20Pointer.sol`)

---

### Summary

The `approve` function in `CW20ERC20Pointer` uses a read-then-write pattern: it reads the current CW20 allowance and then issues a relative `increase_allowance` or `decrease_allowance` call. A malicious spender can front-run the owner's `approve` transaction between the read and the write, causing the final allowance to be set to the new target value while the spender has already consumed the old allowance — spending more tokens than the owner ever intended.

---

### Finding Description

`CW20ERC20Pointer.approve` at lines 59–77 of `contracts/src/CW20ERC20Pointer.sol`:

```solidity
uint256 currentAllowance = allowance(msg.sender, spender);   // (1) read
if (currentAllowance > amount) {
    // decrease_allowance by (currentAllowance - amount)      // (2a) relative write
} else if (currentAllowance < amount) {
    // increase_allowance by (amount - currentAllowance)      // (2b) relative write
}
``` [1](#0-0) 

Because the CW20 backend stores allowances as absolute values but this wrapper adjusts them relatively, the stale read at step (1) can be invalidated by a `transferFrom` that executes between blocks (or via mempool ordering). The delta computed at step (2) is then applied to the already-drained allowance, restoring it to exactly `amount` — as if the front-run never happened.

This is strictly worse than a standard ERC20 `approve` (which unconditionally overwrites the allowance). A standard `approve(newAmount)` after a front-run still lands at `newAmount`. Here, the same sequence also lands at `newAmount`, but the spender has already consumed the old allowance on top of it.

---

### Impact Explanation

**Attacker-controlled path (no privileged access required):**

1. Owner previously called `approve(spender, 100)` → CW20 allowance = 100.
2. Owner decides to increase to 150 and broadcasts `approve(spender, 150)`.
3. Spender observes the pending tx in the mempool and front-runs it with `transferFrom(owner, attacker, 100)` → CW20 allowance = 0.
4. Owner's `approve(spender, 150)` executes: reads `currentAllowance = 0`, computes delta = 150 − 0 = 150, calls `increase_allowance(150)` → CW20 allowance = 150.
5. Spender calls `transferFrom(owner, attacker, 150)` → CW20 allowance = 0.

**Net result:** spender transferred 250 tokens; owner intended to authorize at most 150. The excess 100 tokens are stolen.

The same race applies when the owner tries to *decrease* the allowance: if the spender partially consumes the allowance before the `approve(lowerAmount)` lands, the `decrease_allowance` delta is computed against a stale (higher) value, and the resulting allowance may be higher than the owner intended.

Impact classification: **Critical** — unauthorized token transfer; fund loss scales with token value and allowance size.

---

### Likelihood Explanation

- Sei is an EVM-compatible chain with a public mempool; transaction ordering is observable.
- The attack requires only a standard `transferFrom` call — no privileged role, no governance, no validator key.
- Any spender who has been granted an allowance can execute this unilaterally.
- The window exists whenever an owner attempts to change an existing non-zero allowance.

---

### Recommendation

Replace the read-then-relative-write pattern with a direct absolute-set approach. CW20 contracts expose a `send` / `transfer_from` path but not a direct `set_allowance`; the correct mitigation is to:

1. First call `decrease_allowance` to zero out the existing allowance (or verify it is already zero before proceeding), **and require that step to succeed atomically before issuing the new approval**, or
2. Expose a `set_allowance` message in the underlying CW20 contract and call that instead of the relative helpers, or
3. Require callers to pass the expected current allowance (like OpenZeppelin's `increaseAllowance`/`decreaseAllowance` pattern) and revert if it has changed.

The two-step "approve(0) then approve(N)" pattern mentioned in the external report does **not** fix this because the two transactions are not atomic — the same front-run window exists between them.

    function approve(address spender, uint256 amount) public override returns (bool) {
        // if amount is larger uint128 then set amount to uint128 max
        if (amount > type(uint128).max) {
            amount = type(uint128).max;
        }
        uint256 currentAllowance = allowance(msg.sender, spender);
        if (currentAllowance > amount) {
            string memory spenderAddr = _formatPayload("spender", _doubleQuotes(AddrPrecompile.getSeiAddr(spender)));
            string memory amt = _formatPayload("amount", _doubleQuotes(Strings.toString(currentAllowance - amount)));
            string memory req = _curlyBrace(_formatPayload("decrease_allowance", _curlyBrace(_join(spenderAddr, amt, ","))));
            _execute(bytes(req));
        } else if (currentAllowance < amount) {
            string memory spenderAddr = _formatPayload("spender", _doubleQuotes(AddrPrecompile.getSeiAddr(spender)));
            string memory amt = _formatPayload("amount", _doubleQuotes(Strings.toString(amount - currentAllowance)));
            string memory req = _curlyBrace(_formatPayload("increase_allowance", _curlyBrace(_join(spenderAddr, amt, ","))));
            _execute(bytes(req));
        }
        return true;
    }
