### Title
`native_per_pubdata` Truncates to Zero via Integer Division, Enabling Free Pubdata Generation — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In `validation_impl.rs`, the per-pubdata-byte native resource cost is computed with integer (floor) division. When `pubdata_price < native_price`, the result truncates to `0`. Every downstream pubdata-cost check then multiplies by `0`, making all pubdata completely free in native resources. This is the direct analog of the LendingPool rounding bug: a division that silently produces zero, bypassing the intended resource charge.

---

### Finding Description

**Root cause — line 142:**

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is integer (floor) division. When `pubdata_price < native_price` (e.g., `pubdata_price = 1`, `native_price = 2`), the result is `0`. There is no subsequent guard requiring `native_per_pubdata > 0`.

**Propagation — `get_resources_to_charge_for_pubdata`:**

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // 0 × anything = 0
    .ok_or(out_of_native_resources!())?;
```

With `native_per_pubdata = 0`, the native cost for any quantity of pubdata is `0`. [1](#0-0) 

**Downstream check always passes:**

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs
let enough = resources.has_enough(&resources_for_pubdata); // resources_for_pubdata = 0
```

`check_enough_resources_for_pubdata` always returns `enough = true`, so the post-execution pubdata gate never fires. [2](#0-1) 

**Intrinsic pubdata overhead also zeroed:**

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata); // = 0
```

The native limit is not reduced for intrinsic pubdata either, giving the transaction the full native budget. [3](#0-2) 

The same zero is stored in `TxContextForPreAndPostProcessing::native_per_pubdata` and carried through the entire refund path. [4](#0-3) 

The identical truncation is present in the public API helper `validate_l2_tx_intrinsic_native_resources`: [5](#0-4) 

---

### Impact Explanation

When `native_per_pubdata = 0`:

1. A transaction can write to as many storage slots as its EVM gas limit allows (each SSTORE costs 20 000 gas and emits 32 bytes of pubdata).
2. The native resource charge for all that pubdata is `0`, so the transaction never reverts due to pubdata cost.
3. The operator must still publish every byte to L1 and bears that cost entirely uncompensated through the native resource mechanism.
4. An attacker pays only EVM gas for SSTORE opcodes — the pubdata surcharge that is supposed to cover L1 data costs is completely bypassed.

This matches the LendingPool pattern: a division that rounds to zero lets a user extract a resource (here: L1 data bandwidth) without paying the corresponding fee.

---

### Likelihood Explanation

`pubdata_price` and `native_price` are block-level metadata set by the sequencer/operator. The condition `pubdata_price < native_price` can arise:

- During periods of low L1 gas prices relative to proving costs (the two prices track different cost components).
- Through operator misconfiguration.

Once the condition holds, **any** unprivileged L2 transaction sender can exploit it — no special access is required. The attacker only needs to observe the public block context values and submit storage-writing transactions.

---

### Recommendation

1. **Require `native_per_pubdata > 0`** after computing it, or validate `pubdata_price >= native_price` at block-context ingestion time.
2. **Use ceiling division** (`div_ceil`) instead of floor division for `native_per_pubdata`, consistent with how `native_per_gas` is already computed:
   ```rust
   // native_per_gas already uses div_ceil — apply the same to native_per_pubdata
   let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
       .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
   ```
   This mirrors the fix recommended in the LendingPool report: round up the resource cost so that no non-zero consumption is ever priced at zero.

---

### Proof of Concept

```
Block context: native_price = 2, pubdata_price = 1, eip1559_basefee = 1

Step 1: native_per_pubdata = pubdata_price.wrapping_div(native_price) = 1 / 2 = 0

Step 2: Attacker submits a transaction with gas_limit = 3_000_000 that executes
        150 SSTORE operations (150 × 20_000 = 3_000_000 gas).
        Each SSTORE emits 32 bytes of pubdata → 4 800 bytes total.

Step 3: get_resources_to_charge_for_pubdata returns:
          native = 4_800 × 0 = 0

Step 4: check_enough_resources_for_pubdata → enough = true (0 ≤ any remaining native)

Step 5: Transaction succeeds. Attacker paid EVM gas only.
        Operator publishes 4 800 bytes to L1 at their own expense.

Repeat across many transactions to continuously drain operator L1 budget.
```
