Audit Report

## Title
Hot Keys Not Cleared When NNS Neuron Is Spawned to a New Controller - (File: rs/nns/governance/src/governance.rs)

## Summary
When `spawn_neuron` is called with a `new_controller` different from the parent neuron's controller, the parent's hot keys are unconditionally cloned onto the child neuron. This grants the original hot key holders persistent, unauthorized governance authority over a neuron they do not control and whose new controller never consented to their presence.

## Finding Description
In `spawn_neuron` (governance.rs L2613), the caller must be the controller of the parent neuron (L2631–2633). The `child_controller` is resolved from `spawn.new_controller` if provided, otherwise falls back to the parent's controller (L2651–2655). No branch exists to clear or filter hot keys when `child_controller != parent_neuron.controller()`.

At L2704–2718, the child neuron is built with `.with_hot_keys(parent_neuron.hot_keys.clone())` unconditionally:

```rust
let child_neuron = NeuronBuilder::new(
    child_nid, to_subaccount, child_controller, ...
)
.with_hot_keys(parent_neuron.hot_keys.clone())  // ← always copied
.with_followees(parent_neuron.followees.clone())
...
.build();
```

`is_authorized_to_vote` (types.rs L243–255) delegates to `is_hotkey_or_controller`, which returns `true` for any principal in `hot_keys`. This means every hot key holder on the parent neuron can immediately exercise `RegisterVote`, `Follow`, `JoinCommunityFund`, `LeaveCommunityFund`, and `ChangeAutoStakeMaturity` on the child neuron whose controller is a completely different principal.

## Impact Explanation
This constitutes unauthorized access to a neuron's governance capabilities. The new controller (User B) receives a neuron they believe they fully control, but the spawner's hot key holders retain the ability to vote on NNS proposals on User B's behalf, redirect followees, and commit User B's future maturity to the Neurons' Fund via `JoinCommunityFund` — a financially consequential action. This matches the High bounty impact: *Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints* ($2,000–$10,000).

## Likelihood Explanation
Spawning to a different controller is the standard on-chain mechanism for transferring maturity-derived ICP as a reward or gift. Controllers routinely add hot keys for convenience (e.g., mobile keys, delegation keys). Any such controller who spawns to another party inadvertently transfers those authorizations. The recipient has no in-protocol notification of inherited hot keys. Deliberate exploitation requires the spawner to have pre-added a hot key they intend to retain; unintentional leakage requires only the common pattern of having any hot key at all. Likelihood of unintentional leakage is medium-to-high; deliberate exploitation is low-to-medium.

## Recommendation
When `spawn_neuron` is called with a `new_controller` that differs from the parent neuron's controller, the child neuron should be initialized with an empty hot key list. Concretely, replace the unconditional `.with_hot_keys(parent_neuron.hot_keys.clone())` with a conditional:

```rust
.with_hot_keys(
    if child_controller == parent_neuron.controller() {
        parent_neuron.hot_keys.clone()
    } else {
        vec![]
    }
)
```

If hot key inheritance across controller boundaries is ever desired, it should be an explicit opt-in field in the `Spawn` message, not the default behavior.

## Proof of Concept
1. User A (controller of neuron N) adds hot key `H1` to neuron N via `AddHotKey`.
2. User A calls `manage_neuron` → `Spawn { new_controller: Some(UserB), nonce: None, percentage_to_spawn: Some(100) }`.
3. Governance executes L2704–2718: child neuron C is created with `controller = UserB` and `hot_keys = [H1]`.
4. User A (via `H1`) calls `manage_neuron` on neuron C with `RegisterVote { proposal_id, vote: Yes }`.
5. `is_authorized_to_vote` (types.rs L243–255) returns `true` for `H1` → vote accepted.
6. User A (via `H1`) calls `manage_neuron` on neuron C with `JoinCommunityFund {}` → User B's maturity is committed to the Neurons' Fund without User B's knowledge or consent.
7. This is reproducible as a deterministic PocketIC integration test: create parent neuron, add hot key, spawn to new controller, assert hot key is present on child, assert `RegisterVote` and `JoinCommunityFund` succeed from the hot key principal.
