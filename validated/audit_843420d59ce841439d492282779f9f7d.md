### Title
Unchecked Precompile Call Return Value Burns ERC-20 Tokens Without Releasing Funds on NEAR/Ethereum - (File: etc/eth-contracts/contracts/EvmErc20V2.sol)

---

### Summary

In `EvmErc20V2.sol` (and `EvmErc20.sol`), both `withdrawToNear()` and `withdrawToEthereum()` burn the caller's ERC-20 tokens via `_burn()` **before** invoking the exit precompile via inline assembly. The return value of the assembly `call` opcode is captured in `res` but **never checked**. If the precompile call fails for any reason, the tokens are permanently destroyed on Aurora with no corresponding release of funds on NEAR or Ethereum.

---

### Finding Description

In `EvmErc20V2.sol`, both exit functions follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    _burn(sender, amount);                          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    uint input_size = 1 + 20 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — silent failure is possible
    }
}

The same pattern exists in the older `EvmErc20.sol`: [3](#0-2) 

The exit precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` (`ExitToNear`) has multiple documented failure paths in `engine-precompiles/src/native.rs`, including:
- `ERR_INVALID_RECIPIENT_ADDRESS` — if the NEAR recipient bytes are malformed
- `get_nep141_from_erc20` failure — if the ERC-20 is not registered in the bridge mapping
- `ERR_ETH_ATTACHED_FOR_ERC20_EXIT` — if ETH value is non-zero [4](#0-3) 

In standard EVM semantics, a failed sub-`call` sets `res = 0` but does **not** revert the calling context. Since neither `withdrawToNear` nor `withdrawToEthereum` checks `res` or reverts on failure, the `_burn()` is final and the user's tokens are gone.

---

### Impact Explanation

**Permanent freezing of funds (Critical).**

A user who calls `withdrawToNear` with a malformed or invalid NEAR recipient address (e.g., an account ID that fails validation inside the precompile) will have their ERC-20 tokens burned on Aurora with no corresponding `ft_transfer` issued on the NEAR side. The tokens are unrecoverable: they no longer exist on Aurora and were never released on NEAR. The same applies to `withdrawToEthereum` if the `ExitToEth` precompile call fails.

---

### Likelihood Explanation

**Medium.** Any unprivileged EVM user can trigger this by calling `withdrawToNear` with a recipient byte string that passes Solidity-level encoding but fails the NEAR account ID validation inside the precompile. No special privileges, governance access, or external oracle compromise is required. The entry path is a standard ERC-20 bridge exit call available to every token holder.

---

### Recommendation

Reorder operations so that the precompile call is made **before** burning tokens, and revert if the call fails:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    uint input_size = 1 + 20 + 32 + recipient.length;

    bool success;
    assembly {
        success := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    }
    require(success, "ExitToNear precompile call failed");

    _burn(sender, amount);   // only burn after confirmed success
}
```

Apply the same fix to `withdrawToEthereum` and to the equivalent functions in `EvmErc20.sol`.

---

### Proof of Concept

1. Deploy or use an existing `EvmErc20V2` mirror token on Aurora (e.g., a bridged NEP-141).
2. Acquire a balance of that token.
3. Call `withdrawToNear(invalidRecipientBytes, amount)` where `invalidRecipientBytes` is a byte sequence that fails NEAR account ID validation inside the `ExitToNear` precompile (e.g., an empty byte array, or bytes containing characters illegal in NEAR account IDs).
4. The `_burn(sender, amount)` executes and succeeds — the ERC-20 balance is reduced to zero.
5. The assembly `call` to the precompile returns `res = 0` (failure) because the recipient is invalid.
6. Since `res` is not checked, the function returns without reverting.
7. The user's tokens are permanently destroyed on Aurora; no `ft_transfer` is issued on NEAR. Funds are irrecoverably lost. [5](#0-4) [6](#0-5)
