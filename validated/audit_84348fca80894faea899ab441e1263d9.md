### Title
Silo Mode Fee Bypass via `max_fee_per_gas = 0` Collapses `effective_gas_price` to Zero, Enabling Free Execution — (`engine/src/engine.rs`)

---

### Summary

When Aurora is deployed in silo mode with `fixed_gas` configured, an attacker can submit a transaction with `max_fee_per_gas = 0` while `block_base_fee_per_gas = 0` (the Aurora default). The early-return guard in `charge_gas` is not triggered because `fixed_gas.is_some()`, but the subsequent fee computation yields `effective_gas_price = 0`, making `prepaid_amount = fixed_gas * 0 = 0`. The sender pays nothing. `refund_unused_gas` also returns early because `effective_gas_price.is_zero()`. The silo operator's fee model is completely bypassed.

---

### Finding Description

**Entry point:** `submit_with_args` → `charge_gas` in `engine/src/engine.rs`.

**Step 1 — `fixed_gas` is loaded from silo state:** [1](#0-0) 

**Step 2 — Early-return guard in `charge_gas`:** [2](#0-1) 

The guard requires **all three** conditions simultaneously:
- `max_fee_per_gas.is_zero()` → **true** (attacker sets 0)
- `fixed_gas.is_none()` → **false** (silo has `fixed_gas = Some(x)`)
- `block_base_fee_per_gas.is_zero()` → **true** (Aurora default)

Because `fixed_gas.is_none()` is false, the guard does **not** fire. Execution continues.

**Step 3 — `max_fee_per_gas < block_base_fee_per_gas` check:** [3](#0-2) 

`0 < 0` is false → no rejection.

**Step 4 — `effective_gas_price` computation:** [4](#0-3) 

With `max_priority_fee_per_gas = 0` (forced by the line-1083 check that `max_priority_fee_per_gas ≤ max_fee_per_gas = 0`):
- `priority_fee_per_gas = min(0, 0 − 0) = 0`
- `effective_gas_price = 0 + 0 = 0`

**Step 5 — `prepaid_amount` computation:** [5](#0-4) 

`fixed_gas.as_u256()` (e.g. `1_000_000`) is selected, but then multiplied by `effective_gas_price = 0`, yielding `prepaid_amount = Wei(0)`. Nothing is deducted from the sender.

**Step 6 — `refund_unused_gas` early return:** [6](#0-5) 

`effective_gas_price.is_zero()` → returns immediately. No relayer reward, no accounting correction.

**No existing guard prevents this.** The only `fixed_gas`-related check is: [7](#0-6) 

This only rejects `fixed_gas > gas_limit`; it does not enforce `max_fee_per_gas > 0` when `fixed_gas` is set.

---

### Impact Explanation

The silo operator configures `fixed_gas` precisely to guarantee a minimum fee per transaction, covering the NEAR gas costs they pay on behalf of users. With this bypass, any attacker can execute arbitrarily many transactions at zero ETH cost. The silo operator continues paying NEAR gas for each transaction while receiving no ETH fee revenue. This directly causes **insolvency** of the silo: the operator's NEAR balance is drained with no offsetting income.

---

### Likelihood Explanation

The preconditions are all realistic and require no privilege:
- `block_base_fee_per_gas = 0` is the Aurora default (Aurora does not implement EIP-1559 base-fee burning).
- `fixed_gas` being set is a normal silo operator configuration.
- Submitting a transaction with `max_fee_per_gas = 0` is trivially achievable by any user.

No admin compromise, leaked keys, or external oracle failure is required. The attack is fully self-contained and repeatable.

---

### Recommendation

Add an explicit rejection when `fixed_gas` is set but `effective_gas_price` resolves to zero, or enforce that `max_fee_per_gas > 0` is required whenever `fixed_gas` is configured. For example, after computing `effective_gas_price`, insert:

```rust
if fixed_gas.is_some() && effective_gas_price.is_zero() {
    return Err(GasPaymentError::MaxFeePerGasLessThanBaseFee);
}
```

Alternatively, restructure the early-return guard to also short-circuit when `fixed_gas` is set but `effective_gas_price` would be zero, treating it the same as the all-zero case only if the operator explicitly intends free transactions.

---

### Proof of Concept

Trace through `charge_gas` with:
- `fixed_gas = Some(EthGas::new(1_000_000))`
- `transaction.max_fee_per_gas = U256::zero()`
- `transaction.max_priority_fee_per_gas = U256::zero()`
- `block_base_fee_per_gas = U256::zero()` (Aurora default)

```
guard condition: max_fee_per_gas.is_zero() [T] && fixed_gas.is_none() [F] && base_fee.is_zero() [T]
→ guard NOT triggered

max_fee_per_gas < base_fee: 0 < 0 → false, no error

priority_fee_per_gas = min(0, 0 - 0) = 0
effective_gas_price  = 0 + 0 = 0

prepaid_amount = 1_000_000 * 0 = 0   ← sender pays nothing

refund_unused_gas: effective_gas_price.is_zero() → early return
```

Assert: `prepaid_amount == Wei::zero()` and sender balance is unchanged, confirming complete fee bypass despite `fixed_gas` being set.
