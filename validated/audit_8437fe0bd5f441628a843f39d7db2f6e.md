### Title
ERC-20 Tokens Permanently Lost When `ExitToNear` NEAR Promise Fails Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`)

### Summary
The `ExitToNear` precompile burns ERC-20 tokens in the EVM before the corresponding NEAR `ft_transfer` promise is executed. When compiled without the `error_refund` feature, if the NEAR-side promise fails (e.g., recipient not registered with the NEP-141 contract), the burned tokens are permanently lost with no refund mechanism. This is directly analogous to the external report: a call to a non-existent/failing target "succeeds" from the EVM's perspective, leading to irreversible fund loss.

### Finding Description

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` handles ERC-20 token withdrawals to NEAR. The flow is:

1. An ERC-20 contract burns tokens and calls the `ExitToNear` precompile.
2. The precompile resolves the NEP-141 account ID from the ERC-20 address via `get_nep141_from_erc20`.
3. A NEAR promise is created to call `ft_transfer` on the NEP-141 contract.
4. The EVM transaction returns `Ok(PrecompileOutput {...})` — **success** — with tokens already burned.
5. The NEAR promise executes asynchronously.

The critical issue is the conditional refund mechanism gated behind the `error_refund` feature flag: [1](#0-0) 

When `error_refund` is **not** enabled, `refund` is `None`. For a standard ERC-20 exit (no wNEAR unwrap), `transfer_near_args` is also `None`, making `callback_args` equal to `ExitToNearPrecompileCallbackArgs::default()`. This causes the promise to be created **without any callback**: [2](#0-1) 

No callback means no failure detection. If the NEAR `ft_transfer` promise fails, the engine never learns of the failure and issues no refund.

The callback handler `exit_to_near_precompile_callback` confirms: when `args.refund` is `None`, it does nothing on failure: [3](#0-2) 

The `get_nep141_from_erc20` lookup only validates that the ERC-20→NEP-141 mapping exists in storage — it does **not** validate that the NEP-141 account actually exists on NEAR or that the recipient is registered: [4](#0-3) 

The ERC-20 token burn happens inside the ERC-20 contract before calling the precompile. Once the EVM call to the precompile returns `Ok`, the burn is committed and irreversible.

The test suite explicitly documents this behavior: [5](#0-4) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

ERC-20 tokens are burned in the EVM (irreversible state change) before the NEAR-side `ft_transfer` promise is confirmed. If the NEAR promise fails for any reason — recipient not registered with the NEP-141 contract, NEP-141 contract paused, or NEP-141 account deleted — the tokens are permanently destroyed with no recovery path when `error_refund` is not enabled.

### Likelihood Explanation

**Medium.** NEP-141 contracts (NEAR's fungible token standard) require recipients to explicitly register (via `storage_deposit`) before they can receive tokens. A user who exits ERC-20 tokens to an unregistered NEAR account will trigger a failed `ft_transfer` promise. This is a realistic, user-triggerable scenario requiring no special privileges. The attacker-controlled entry path is: call the ERC-20 `withdraw` function targeting an unregistered NEAR recipient → tokens burned → NEAR promise fails → permanent loss.

### Recommendation

1. **Enable `error_refund` unconditionally** in production builds, or remove the feature gate and make the refund callback mandatory for all ERC-20 exits.
2. Alternatively, validate at the EVM level (before burning) that the NEAR promise is likely to succeed — though this is difficult given NEAR's asynchronous execution model.
3. At minimum, document clearly which production deployments have `error_refund` disabled and warn users of the risk.

### Proof of Concept

1. User holds ERC-20 tokens on Aurora backed by a NEP-141 contract (e.g., `token.near`).
2. User calls the ERC-20 `withdraw` function targeting a NEAR account `victim.near` that is **not registered** with `token.near`.
3. The ERC-20 contract burns the tokens and calls the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`.
4. `get_nep141_from_erc20` succeeds (the ERC-20→NEP-141 mapping exists in storage).
5. The precompile returns `Ok(PrecompileOutput {...})` — EVM call succeeds, tokens burned.
6. NEAR schedules `ft_transfer` on `token.near` targeting `victim.near`.
7. `token.near` rejects the transfer: `victim.near` is not registered.
8. Without `error_refund`, no callback fires, no refund is issued.
9. Tokens are permanently destroyed. [6](#0-5) [7](#0-6)

fn get_nep141_from_erc20<I: IO>(erc20_token: &[u8], io: &I) -> Result<AccountId, ExitError> {
    AccountId::try_from(
        io.read_storage(bytes_to_key(KeyPrefix::Erc20Nep141Map, erc20_token).as_slice())
            .map(|s| s.to_vec())
            .ok_or(ExitError::Other(Cow::Borrowed(ERR_TARGET_TOKEN_NOT_FOUND)))?,
    )
    .map_err(|_| ExitError::Other(Cow::Borrowed("ERR_INVALID_NEP141_ACCOUNT")))
}
