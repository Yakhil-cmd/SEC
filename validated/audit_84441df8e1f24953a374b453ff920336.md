Audit Report

## Title
Any Deployed SNS Swap Canister Can Settle Another SNS's Neurons' Fund Participation - (File: rs/nns/governance/src/governance.rs)

## Summary

`Governance::settle_neurons_fund_participation` authorizes the caller by checking only that it is *any* valid SNS swap canister in SNS-W's global list, without verifying that the caller is the specific swap canister associated with the `nns_proposal_id` in the request. The code itself acknowledges this with the comment `// Check authorization. Note that a Swap could settle each other's participation.` A malicious SNS swap canister can supply an attacker-controlled `sns_governance_canister_id` in the `Committed` variant, causing NNS Governance to mint ICP to the attacker's address and permanently freeze the victim SNS's Neurons' Fund settlement via the idempotency guard.

## Finding Description

**Entry point** — `rs/nns/governance/canister/canister.rs` lines 529–539: the public `#[update]` endpoint passes `caller()` and the raw request directly to `Governance::settle_neurons_fund_participation` with no pre-filtering. [1](#0-0) 

**Authorization check** — `rs/nns/governance/src/governance.rs` lines 7018–7038: the only authorization performed is a call to `is_canister_id_valid_swap_canister_id`, which queries SNS-W's `list_deployed_snses` and returns `Ok` if the caller appears as *any* swap canister in the global list. There is no binding between the caller and the specific `nns_proposal_id` supplied in the request. [2](#0-1) 

**`is_canister_id_valid_swap_canister_id`** — lines 8191–8225: iterates over all deployed SNS instances and returns `Ok` on any match, with no proposal-scoped check. [3](#0-2) 

**Swap canister ID not stored in `ProposalData`** — `execute_create_service_nervous_system_proposal` (lines 4569–4578) explicitly notes that `canisters` fields from `deploy_new_sns_response` are not stored on `ProposalData`, making a per-proposal binding check impossible with the current data model. [4](#0-3) 

**Attacker-controlled mint destination** — lines 7249–7273: `mint_to_sns_governance` is called with `sns_governance_canister_id` taken directly from the caller-supplied `Committed` struct, not from on-chain proposal state. [5](#0-4) 

**`Committed.sns_governance_canister_id` is a free field** — `rs/nns/governance/api/src/types.rs` lines 3547–3555: no derivation from on-chain state. [6](#0-5) 

**Idempotency guard** — lines 7166–7174: once `final_neurons_fund_participation` is set for a proposal, all subsequent calls return the previously stored result without re-executing, permanently locking the victim SNS's settlement to the attacker's forged snapshot. [7](#0-6) 

## Impact Explanation

A successful exploit produces three simultaneous harms:

1. **ICP theft**: NNS Governance mints ICP (sourced from burned NNS neuron maturity) to the attacker-supplied `sns_governance_canister_id` instead of the legitimate SNS treasury.
2. **Permanent settlement freeze**: The idempotency guard stores the attacker's forged `final_neurons_fund_participation` snapshot. The victim SNS's legitimate swap canister can never re-settle; every subsequent call returns the attacker's result.
3. **Maturity burn without neuron creation**: NNS neuron maturity is permanently burned for Neurons' Fund participants, but no corresponding SNS neurons are created for them in the victim SNS.

This matches the **High** allowed impact: *"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."* If the Neurons' Fund participation for the targeted SNS exceeds $1M in ICP value, the impact escalates to **Critical**: *"Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant ICP/Cycles or in-scope chain-key/ledger assets, especially over $1M."*

## Likelihood Explanation

**Required preconditions:**
1. The attacker must control a valid SNS swap canister. This requires submitting a `CreateServiceNervousSystem` NNS proposal and having it adopted by NNS governance. This is a meaningful but realistic barrier — multiple SNS launches have passed NNS governance on mainnet. Any entity that already controls a deployed SNS swap canister can exploit this immediately with no additional setup.
2. A victim SNS with Neurons' Fund participation must have an unsettled proposal (lifecycle = `Open`). This is routinely true on mainnet whenever multiple SNS launches are in progress concurrently.

The attack is a single canister call with no race condition requirement. It is repeatable against any unsettled SNS proposal as long as the attacker's swap canister remains in SNS-W's list.

## Recommendation

1. **Bind authorization to the specific proposal**: Store the swap canister ID on `ProposalData` when `execute_create_service_nervous_system_proposal` runs (the `deploy_new_sns_response.canisters.swap` field is already available at line 4578 but explicitly discarded). In `settle_neurons_fund_participation`, replace the global membership check with `caller == proposal_data.swap_canister_id`.

2. **Derive `sns_governance_canister_id` from on-chain state**: Store the SNS governance canister ID on `ProposalData` at deployment time and use that stored value in `mint_to_sns_governance` instead of the caller-supplied field from the `Committed` request.

## Proof of Concept

**Minimal deterministic integration test plan (PocketIC or state-machine test):**

1. Deploy NNS with Neurons' Fund neurons holding maturity.
2. Submit and execute `CreateServiceNervousSystem` proposal for SNS A → obtain `swap_A` canister ID (registered in SNS-W).
3. Submit and execute `CreateServiceNervousSystem` proposal for SNS B with Neurons' Fund participation → obtain proposal ID `P_B`; SNS B's swap lifecycle is `Open`.
4. From `swap_A`, call `settle_neurons_fund_participation` with:
   ```
   SettleNeuronsFundParticipationRequest {
       nns_proposal_id: Some(P_B),
       result: Some(Committed {
           sns_governance_canister_id: Some(<attacker_wallet>),
           total_direct_participation_icp_e8s: Some(<max_direct_participation>),
           total_neurons_fund_participation_icp_e8s: Some(0),
       }),
   }
   ```
5. Assert: `is_canister_id_valid_swap_canister_id(swap_A)` returns `Ok` (swap_A is in SNS-W list).
6. Assert: ICP balance of `<attacker_wallet>` increased by the Neurons' Fund matched amount.
7. Assert: `proposal_data(P_B).sns_token_swap_lifecycle == Committed`.
8. From `swap_B`, call `settle_neurons_fund_participation(P_B, Committed { sns_governance_canister_id: <legitimate_sns_b_governance>, ... })`.
9. Assert: call returns the attacker's forged snapshot (idempotency guard fires); SNS B's governance treasury balance remains 0; NNS neuron maturity is burned.
