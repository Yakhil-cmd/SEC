### Title
Silent Precompile Failure in `withdrawToNear` / `withdrawToEthereum` Causes Permanent ERC-20 Token Loss — (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` burns a user's ERC-20 tokens **before** calling the Aurora exit precompile, and the assembly `call` return value is never checked. If the precompile rejects the call (invalid NEAR account ID, amount > `u128::MAX`, or unregistered ERC-20), the burn is committed but no bridge transfer is initiated. The user's tokens are permanently destroyed with no refund path.

---

### Finding Description

`EvmErc20.sol` implements two bridge-out functions:

```solidity
// etc/eth-contracts/contracts/EvmErc20.sol lines 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked — silent failure
    }
}

The `ExitToNear` precompile rejects calls under several conditions that are entirely user-controlled:

1. **Invalid NEAR account ID** — `parse_recipient` returns `ERR_INVALID_RECEIVER_ACCOUNT_ID` if the recipient bytes are not a valid NEAR account ID (e.g., uppercase letters, leading/trailing hyphens, or invalid UTF-8). [3](#0-2) 

2. **Amount exceeds `u128::MAX`** — `parse_amount` returns `ERR_INVALID_AMOUNT` if the Solidity `uint256` amount overflows `u128`. [4](#0-3) 

3. **ERC-20 not registered** — `get_nep141_from_erc20` returns `ERR_TARGET_TOKEN_NOT_FOUND` if the ERC-20 address has no NEP-141 mapping. [5](#0-4) 

In all three cases the precompile returns an `ExitError`, the EVM `call` opcode sets `res = 0`, but because the Solidity assembly never checks `res`, the outer function does not revert. The `_burn` executed before the precompile call is already committed to EVM state. The tokens are gone.

A secondary, compounding gap exists in the Rust layer: when the `error_refund` feature is **not** compiled in, the `refund` field in `ExitToNearPrecompileCallbackArgs` is hardcoded to `None`, so even a valid precompile call whose downstream NEAR `ft_transfer` promise fails produces no on-chain refund. [6](#0-5) [7](#0-6) 

The test suite explicitly documents this loss:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [8](#0-7) 

---

### Impact Explanation

**Critical — Permanent freezing / destruction of funds.**

Any holder of a bridged ERC-20 token on Aurora who calls `withdrawToNear` or `withdrawToEthereum` with a recipient that the precompile rejects will have their tokens permanently burned. The EVM state change (`_burn`) is irreversible; there is no refund path in the Solidity contract, and without `error_refund` there is no refund path in the Rust callback either. The NEP-141 balance on the NEAR side is unaffected, so the total supply accounting diverges: ERC-20 supply decreases while NEP-141 supply held by Aurora does not.

---

### Likelihood Explanation

**Medium.** The trigger conditions are reachable by any unprivileged token holder:

- A user supplying a NEAR account ID with uppercase letters (common mistake for users coming from Ethereum) silently loses all tokens passed to `withdrawToNear`.
- A user holding a balance exceeding `u128::MAX` (theoretically possible for low-decimal tokens with large supplies) triggers `ERR_INVALID_AMOUNT`.
- Any integrating contract that does not validate the recipient before calling `withdrawToNear` propagates the silent failure to its users.

No special privileges, governance access, or external oracle is required.

---

### Recommendation

1. **Check the precompile return value and revert on failure** in both `withdrawToNear` and `withdrawToEthereum`:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

2. **Invert the operation order**: validate the recipient (e.g., call the precompile in a view-like manner or perform off-chain validation) before burning, or burn only after confirming the precompile accepted the call.

3. **Always compile with `error_refund`** enabled in production, or make the refund path unconditional in `exit_to_near_precompile_callback` so that a failed NEAR-side promise always triggers an EVM-side refund.

---

### Proof of Concept

1. Alice holds 1000 units of a bridged ERC-20 token on Aurora.
2. Alice calls `withdrawToNear("ALICE.NEAR", 1000)` — note the uppercase letters, which are invalid in NEAR account IDs.
3. `_burn(Alice, 1000)` executes; Alice's ERC-20 balance drops to 0.
4. The precompile call is made with `recipient = b"ALICE.NEAR"`.
5. `parse_recipient` calls `"ALICE.NEAR".parse::<AccountId>()`, which fails because NEAR account IDs must be lowercase.
6. The precompile returns `ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID")`.
7. The EVM `call` opcode returns `res = 0`.
8. The assembly block does not check `res`; `withdrawToNear` returns normally.
9. Alice's 1000 ERC-20 tokens are permanently destroyed. The NEP-141 balance held by Aurora is unchanged. Alice receives nothing on NEAR.

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
