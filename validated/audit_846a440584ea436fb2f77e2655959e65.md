### Title
PayAsYouGo HTTP Outcall Cycles Overcharge: Full Payment Taken Upfront but Refund Mechanism Unimplemented - (`rs/https_outcalls/pricing/src/lib.rs`)

---

### Summary

When a canister calls `FlexibleHttpRequest`, the execution environment takes the **entire payment upfront** under `PayAsYouGo` pricing and stores `refundable_cycles = payment − base_fee`. However, `PricingFactory::new_tracker` unconditionally returns a `LegacyTracker` regardless of the pricing version (a known TODO), and the client immediately rejects every `PayAsYouGo` request with `SysFatal`. The response is delivered back to the canister with `refund: request.payment = 0`, so the canister loses `payment − base_fee` cycles that should have been returned. The `refund_status.refundable_cycles` field is populated but never applied to the canister's balance.

---

### Finding Description

**Step 1 – `FlexibleHttpRequest` always uses `PayAsYouGo`.**

Every context created by `generate_from_flexible_args` hard-codes `PricingVersion::PayAsYouGo`: [1](#0-0) 

**Step 2 – Execution environment takes the full payment upfront.**

In `try_add_http_context_to_replicated_state`, for `PayAsYouGo` the entire payment is consumed (`take()` sets it to zero), while `refundable_cycles = payment − base_fee` is stored in the context: [2](#0-1) 

**Step 3 – `PricingFactory::new_tracker` ignores the pricing version.**

The factory always returns a `LegacyTracker`, with an explicit TODO acknowledging the gap: [3](#0-2) 

**Step 4 – The client rejects every `PayAsYouGo` request with `SysFatal` and a zero-refund receipt.** [4](#0-3) 

`LegacyTracker::create_payment_receipt` returns `CanisterHttpPaymentReceipt { refund: Cycles::new(0) }`: [5](#0-4) 

**Step 5 – Response is delivered to the canister with `refund: request.payment = 0`.**

When the `SubnetMessage::Response` arrives for a `CanisterHttpRequest` context, the refund field is taken directly from `request.payment`, which is already zero: [6](#0-5) 

**Step 6 – `refund_status.refundable_cycles` is never applied.**

The `RefundStatus` struct stores `refundable_cycles` and `per_replica_allowance`, but no code path in the response delivery applies these values to the canister's balance. The `check_refund_allowance` call in the payload builder only validates that the receipt's refund does not *exceed* the allowance; it does not credit the canister: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

Any canister on a subnet where `FlexibleHttpRequest` is enabled that calls the endpoint with payment `P` will:
- Have `P` cycles deducted upfront.
- Receive a `SysFatal` reject with 0 cycles refunded.
- Permanently lose `P − base_fee` cycles that should have been returned.

The `base_fee` for a 13-node subnet is on the order of tens of millions of cycles; the `legacy_fee` (which includes a per-byte response term) can be orders of magnitude larger. A canister that sends a large payment loses the entire surplus. Repeated calls drain the canister's balance.

**Impact: Medium** — cycles loss is bounded to the calling canister's own payment; no protocol-wide treasury is drained.

---

### Likelihood Explanation

The `FlexibleHttpRequest` feature is gated behind a subnet feature flag that is disabled by default: [9](#0-8) 

If the flag is enabled on any application subnet (e.g., for testing or staged rollout), every canister on that subnet is exposed. The `ALLOWED_HTTP_OUTCALLS_PRICING_VERSIONS` constant confirms that `PayAsYouGo` is not yet an accepted version for the legacy `HttpRequest` endpoint, so only `FlexibleHttpRequest` callers are affected: [10](#0-9) 

**Likelihood: Medium** — requires the feature flag to be enabled; no other preconditions.

---

### Recommendation

1. **In `PricingFactory::new_tracker`**: implement the `PayAsYouGo` tracker branch (resolve TODO IC-1937) before enabling `FlexibleHttpRequest` on any production subnet. Until then, reject `PayAsYouGo` requests at the execution-environment level (before taking the payment) rather than at the client level after the payment has already been consumed.

2. **In the response delivery path** (`execute_subnet_message` / `SubnetMessage::Response` for `CanisterHttpRequest` contexts): apply `refund_status.refundable_cycles` to the response's `refund` field when the pricing version is `PayAsYouGo`, so that the canister recovers the unused portion of its payment regardless of whether the outcall succeeded or was rejected.

3. **Guard**: add an assertion or early-return in `try_add_http_context_to_replicated_state` that prevents `PayAsYouGo` payment deduction when the client-side tracker does not yet support it.

---

### Proof of Concept

1. Enable `FlexibleHttpRequest` on an application subnet.
2. Deploy a canister with a large cycles balance.
3. Call `FlexibleHttpRequest` with payment `P >> base_fee`.
4. Observe: the canister receives a `SysFatal` reject and 0 cycles are refunded.
5. Repeat: each call drains `P − base_fee` cycles from the canister.

The execution-environment unit test `execute_flexible_canister_http_request` confirms that `request.payment` is set to zero upfront for `PayAsYouGo` and that `refundable_cycles = payment − base_fee` is stored but never returned: [11](#0-10)

pub(crate) fn check_refund_allowance(
    receipt: &CanisterHttpPaymentReceipt,
    per_replica_allowance: Cycles,
) -> Result<(), InvalidCanisterHttpPayloadReason> {
    if receipt.refund > per_replica_allowance {
        return Err(InvalidCanisterHttpPayloadReason::RefundExceedsAllowance {
            refund: receipt.refund,
            per_replica_allowance,
        });
    }
    Ok(())
}
