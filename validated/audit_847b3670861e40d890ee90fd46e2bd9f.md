### Title
Unchecked Exit Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Loss - (File: etc/eth-contracts/contracts/EvmErc20.sol)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` burn the caller's ERC-20 tokens **before** invoking the exit precompile via a low-level assembly `call`. The return value of that `call` is captured into `res` but never inspected. If the exit precompile rejects the call for any reason, the burn is not rolled back, the tokens are permanently destroyed, and no NEAR/Ethereum transfer is ever initiated.

---

### Finding Description

In `etc/eth-contracts/contracts/EvmErc20.sol`, `withdrawToNear` executes:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no revert on failure
    }
}
```

The identical pattern appears in `withdrawToEthereum` (same file, line 65–76) and in `EvmErc20V2.sol`.

The callee is the `ExitToNear` precompile (`engine-precompiles/src/native.rs`). That precompile enforces a hard `MAX_INPUT_SIZE = 1_024` bytes:

```rust
const MAX_INPUT_SIZE: usize = 1_024;
```

validated inside `parse_input` → `validate_input_size`:

```rust
fn parse_input(input: &[u8]) -> Result<&[u8], ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    Ok(&input[1..])
}
```

The `withdrawToNear` input layout is `0x01 (1 byte) + amount (32 bytes) + recipient (variable)`. Any `recipient` longer than **991 bytes** (= 1024 − 1 − 32) causes `validate_input_size` to return `Err(ExitError::Other("ERR_INVALID_INPUT"))`, which the EVM surfaces as a failed sub-call (`res = 0`). Because `res` is never checked, the Solidity function returns normally — with the tokens already burned and no NEAR transfer scheduled.

Additional precompile failure paths that produce the same outcome:
- `recipient` contains bytes that are not a valid NEAR account ID (`parse_recipient` → `AccountId::try_from` fails).
- `amount` exceeds `u128::MAX` (`parse_amount` returns `Err`).
- The precompile is invoked in static context (`ERR_INVALID_IN_STATIC`).

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the exit precompile call fails silently:
1. The user's ERC-20 tokens are irreversibly burned (supply reduced).
2. No NEP-141 `ft_transfer` promise is ever created on the NEAR side.
3. The user receives nothing; the tokens are gone with no recovery path.

This is a direct, permanent loss of user funds triggered by a reachable code path in the production bridge contract.

---

### Likelihood Explanation

**Medium.** The `recipient` parameter is fully user-controlled and unbounded in the Solidity interface. A user who passes a NEAR account ID longer than 991 bytes (e.g., a deeply nested sub-account path, or any bytes blob) will silently lose their entire `amount`. The same outcome occurs for any non-UTF-8 or structurally invalid NEAR account ID. No privileged role is required; any token holder can trigger this against themselves, and any integrating contract that forwards an untrusted `recipient` can trigger it against its users.

---

### Recommendation

Check `res` after every low-level `call` to the exit precompile and revert on failure:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum` in `EvmErc20.sol` and both withdrawal functions in `EvmErc20V2.sol`. Alternatively, validate `recipient` length and content on the Solidity side before burning, so the burn never executes with input that the precompile will reject.

---

### Proof of Concept

1. Deploy `EvmErc20` with the Aurora engine as admin.
2. Mint 1000 tokens to `victim`.
3. `victim` calls `withdrawToNear(recipient, 1000)` where `recipient` is a 992-byte byte string (any content).
4. `_burn(victim, 1000)` executes — victim's balance drops to 0, total supply decreases by 1000.
5. The assembly `call` to the `ExitToNear` precompile is made with `input_size = 1 + 32 + 992 = 1025 > 1024`.
6. Inside `engine-precompiles/src/native.rs`, `validate_input_size` returns `Err` → precompile returns `ExitError` → EVM sets `res = 0`.
7. `res` is not checked; `withdrawToNear` returns without reverting.
8. Transaction is mined successfully. Victim has 0 ERC-20 tokens and 0 NEAR tokens received. Funds are permanently lost.
