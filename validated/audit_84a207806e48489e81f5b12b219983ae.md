Audit Report

## Title
RSA Keys Accepted as Delegation Trust Anchors Despite Being Rejected as Direct Signers — (`rs/validator/src/ingress_validation.rs`)

## Summary
`validate_delegation` accepts bare `RsaSha256PublicKeyDer` keys in its `basic_sig` match arm (line 813), while `validate_signature` explicitly rejects the same key type with an `AlgorithmNotSupported` error (lines 694–700). Because `validate_delegations` updates `pubkey` to the delegated key before returning, the RSA rejection branch in `validate_signature` is never evaluated when RSA is used as the trust anchor in a delegation chain. An attacker holding an RSA keypair can authenticate as the corresponding self-authenticating principal against any canister that grants it access, fully bypassing the stated policy that "RSA signatures are not allowed except in webauthn context."

## Finding Description
The asymmetry is confirmed at the code level:

**`validate_signature`** (lines 694–701): `RsaSha256PublicKeyDer` is an explicit error arm: [1](#0-0) 

**`validate_delegation`** (lines 810–818): `RsaSha256PublicKeyDer` is silently grouped with Ed25519/ECDSA and accepted: [2](#0-1) 

The exploit call chain:

1. `validate_signature` calls `validate_delegations` with `signature.signer_pubkey` (the RSA key) as the initial trust anchor: [3](#0-2) 

2. Inside `validate_delegations`, `validate_delegation` is called with the RSA key — this succeeds via the `basic_sig` arm: [4](#0-3) 

3. `pubkey` is updated to the delegated key (e.g., Ed25519) at line 749, so `validate_delegations` returns the Ed25519 key.

4. Back in `validate_signature`, `public_key_from_bytes` is called on the **final** (Ed25519) key at line 657, and the match at line 659 takes the Ed25519 branch (line 671) — the RSA rejection at lines 694–700 is never reached: [5](#0-4) 

5. `validate_user_id` binds the `sender` principal to `signer_pubkey` (the RSA key), so the request is authenticated as the RSA-derived self-authenticating principal: [6](#0-5) 

## Impact Explanation
Any canister that grants access to a principal derived from an RSA public key (`PrincipalId::new_self_authenticating(rsa_der)`) can be reached by an attacker holding the corresponding RSA private key, despite the explicit protocol-level policy rejecting bare RSA for ingress. This constitutes unauthorized access to canister-controlled resources (funds, governance votes, identity operations) for any principal whose access control was designed under the assumption that RSA-derived principals are permanently unreachable via ingress. This maps to **High ($2,000–$10,000)**: unauthorized access to wallets, identities, or canister-controlled funds where exploitation is constrained to targets with RSA-derived principal access control.

## Likelihood Explanation
The attack requires only a standard RSA-2048+ keypair and the ability to craft an HTTP ingress message with a `sender_delegation` field — both are trivially available to any unprivileged external user. No privileged access, no threshold corruption, no social engineering is required. The path is entirely local and deterministically reproducible.

## Recommendation
`validate_delegation` must apply the same algorithm policy as `validate_signature`. Bare `RsaSha256PublicKeyDer` must be rejected in `validate_delegation` with the same `AlgorithmNotSupported` error used at lines 694–700. The cleanest fix is to extract a shared `is_allowed_signing_algorithm(pk_type: KeyBytesContentType) -> bool` predicate (or a `check_signing_algorithm` function returning `Result`) called by both `validate_signature` and `validate_delegation`, so the two match arms cannot diverge again.

## Proof of Concept
```
1. Generate RSA-2048 keypair (rsa_sk, rsa_pk_der).
2. Generate Ed25519 keypair (ed_sk, ed_pk_der).
3. Construct Delegation { pubkey: ed_pk_der, expiration: far_future, targets: None }.
4. Sign delegation bytes with rsa_sk → rsa_delegation_sig.
5. Build UserSignature {
       signer_pubkey: rsa_pk_der,
       sender_delegation: Some([SignedDelegation { delegation, signature: rsa_delegation_sig }]),
       signature: ed25519_sign(message_id, ed_sk),
   }.
6. Set sender = PrincipalId::new_self_authenticating(rsa_pk_der).
7. Call validate_request(...) → expect Ok(CanisterIdSet::all()).
```
The RSA-signed delegation passes `validate_delegation` at line 813; the final key is Ed25519; `validate_signature` takes the Ed25519 branch at line 671; the RSA rejection at lines 694–700 is never evaluated.
