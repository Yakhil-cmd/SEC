### Title
Missing Storage Collision Check in CREATE/CREATE2 Allows Deployment Over Attacker-Poisoned Storage — (`evm_interpreter/src/ee_trait_impl.rs`)

---

### Summary

ZKsync OS's EVM interpreter explicitly skips the EIP-161/Yellow Paper requirement that a CREATE/CREATE2 deployment must fail if the target address has **non-empty storage**. The collision guard only checks `code_len != 0 || nonce != 0`. An attacker can pre-populate storage at a deterministic CREATE2 address (via deploy → SSTORE → SELFDESTRUCT), leaving the address with zero code and zero nonce but non-zero storage. ZKsync OS then allows a subsequent CREATE2 deployment to that address to succeed — inheriting the attacker-controlled storage — whereas Ethereum would revert with a collision. This is a reachable EVM semantic mismatch with a concrete fund-loss path.

---

### Finding Description

In `before_executing_frame` in `evm_interpreter/src/ee_trait_impl.rs`, the collision check for constructor frames reads:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    // ... burn gas, emit CreateCollision, return false
}
```

The comment explicitly acknowledges the missing check. The Ethereum specification (EIP-161, Yellow Paper §7) requires that a CREATE/CREATE2 collision is triggered when the target account has **non-zero nonce OR non-empty code OR non-empty storage**. ZKsync OS only checks the first two conditions.

The test index files confirm this is a known, unresolved deviation:

- `create2collisionStorageParis.json` — `enabled: false`, comment: `"We do not check storage for collision"`
- `RevertInCreateInInitCreate2Paris.json` — `enabled: false`, comment: `"We do not check for storage collisions"`
- `dynamicAccountOverwriteEmpty_Paris.json` — `enabled: false`, comment: `"Expects revert due to storage collision"`

**Attack path to pre-poison storage:**

1. Attacker computes the deterministic CREATE2 address for a victim's upcoming deployment (salt, deployer address, and initcode are often public or predictable — e.g., factory contracts with known salts).
2. Attacker deploys a contract at that address using the same CREATE2 parameters.
3. Attacker's contract writes malicious storage values (e.g., `balances[attacker] = type(uint256).max` in a token contract's storage layout).
4. Attacker's contract calls `SELFDESTRUCT`, which clears code and resets nonce to 0 but **leaves storage intact**.
5. The target address now has: `code_len == 0`, `nonce == 0`, but **non-empty storage**.
6. Victim deploys their contract via CREATE2 to the same address. In Ethereum this reverts (`CreateCollision`). In ZKsync OS, the check at line 429 passes (`deployee_code_len == 0 && deployee_nonce == 0`), and the constructor executes successfully.
7. The victim's contract is now live with attacker-controlled storage slots, subverting its invariants.

---

### Impact Explanation

**Impact: High**

A victim contract deployed at a storage-poisoned address inherits attacker-controlled storage. For any contract that assumes clean storage at construction time (token balances, ownership mappings, access control flags), the attacker can:

- Pre-set `balances[attacker]` to `MAX_UINT256` in a token contract, enabling theft of all deposited funds.
- Pre-set `owner = attacker` in an ownable contract, granting full administrative control.
- Pre-set any storage slot that the constructor would otherwise initialize to a safe value.

This constitutes a direct, externally-reachable path to loss of user funds in deployed contracts on ZKsync OS.

Additionally, this is a **forward/proving divergence**: the forward system (sequencer) and any Ethereum-equivalent reference implementation would produce different state roots for the same transaction sequence, breaking ZK proof soundness for EVM-equivalent execution.

---

### Likelihood Explanation

**Likelihood: Medium**

- CREATE2 addresses are fully deterministic and computable by anyone given the deployer address, salt, and initcode — all of which are typically public (factory contracts, protocol deployments, etc.).
- The SELFDESTRUCT + storage-poisoning pattern is a known Ethereum attack vector (the `create2collisionStorageParis` test case exists precisely for this scenario).
- The attacker must front-run the victim's deployment transaction, which is feasible on ZKsync (sequencer ordering is not trustless from the user's perspective).
- The attack requires two transactions (deploy + selfdestruct) before the victim's deployment, which is straightforward.

---

### Recommendation

Implement the storage non-emptiness check as part of the CREATE/CREATE2 collision guard in `before_executing_frame`. Before allowing a constructor frame to execute, verify that the target address has no non-zero storage slots. If the flat storage model makes a full storage scan prohibitive, a practical mitigation is to track a per-account "has ever had storage written" flag (a single bit in `AccountProperties`) that is set on any SSTORE and never cleared, enabling an O(1) collision check.

Alternatively, align with the post-Cancun behavior where SELFDESTRUCT no longer clears code/nonce (EIP-6780), which eliminates the primary vector for creating zero-code/zero-nonce accounts with non-empty storage.

```
//   code_len = 0  ✓ (passes ZKsync OS check)
//   nonce    = 0  ✓ (passes ZKsync OS check)
//   storage[0] = MAX_UINT256  ← NOT checked by ZKsync OS

// Step 2: Victim deploys their token contract at the same address
// ZKsync OS: deployment SUCCEEDS (collision check passes)
// Ethereum:  deployment FAILS  (storage collision)

// Result: victim's token contract has balances[attacker] = MAX_UINT256
```

The collision guard at `evm_interpreter/src/ee_trait_impl.rs:429` passes because `deployee_code_len == 0` and `deployee_nonce == 0`, despite the address having attacker-controlled storage. The constructor runs, the contract is deployed, and the attacker's pre-set storage values are now live in the victim contract's state.
