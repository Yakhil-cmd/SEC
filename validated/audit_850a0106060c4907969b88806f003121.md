### Title
Partial Refund from `ft_transfer_call` in Omni Exit Path Permanently Freezes NEP-141 Tokens in Aurora's Account - (File: `engine/src/contract_methods/connector.rs`, `engine-precompiles/src/native.rs`)

---

### Summary

When a user exits ERC-20 tokens to NEAR using the Omni message path (which triggers `ft_transfer_call` on the NEP-141 contract), the ERC-20 tokens are burned upfront inside the EVM. If the receiver's `ft_on_transfer` returns a non-zero refund amount (a partial refund), the NEP-141 contract's `ft_resolve_transfer` returns those tokens to Aurora's NEP-141 balance. However, `exit_to_near_precompile_callback` only checks whether the promise succeeded or failed — it never inspects the actual transferred amount and never re-mints the refunded ERC-20 tokens. The refunded NEP-141 tokens accumulate in Aurora's account with no corresponding ERC-20 tokens, making them permanently inaccessible to any user.

---

### Finding Description

The `ExitToNear` precompile supports an "Omni" message path that calls `ft_transfer_call` on the NEP-141 contract instead of `ft_transfer`. [1](#0-0) 

For this path, `transfer_near_args` is set to `None`, meaning the callback args contain only the `refund` field (populated only when the `error_refund` feature is compiled in) and `transfer_near: None`. [2](#0-1) 

The promise structure attaches `exit_to_near_precompile_callback` as the callback after `ft_transfer_call` completes. [3](#0-2) 

Inside the callback, the logic is binary — success or failure: [4](#0-3) 

The `PromiseResult::Successful(_)` arm ignores the actual value `_`. For `ft_transfer_call`, a successful result carries the net amount transferred (i.e., `amount - refunded`). If the receiver's `ft_on_transfer` returns `Y > 0` tokens as a refund, the NEP-141 contract's `ft_resolve_transfer` credits those `Y` tokens back to Aurora's NEP-141 balance. The callback sees `PromiseResult::Successful(...)` and takes no action. The `Y` NEP-141 tokens now sit in Aurora's account with no corresponding ERC-20 tokens — they cannot be claimed by any user because the ERC-20 tokens that would be needed to burn and trigger an exit were already destroyed.

The existing test explicitly acknowledges that without `error_refund`, a completely failed `ft_transfer` also causes permanent loss: [5](#0-4) 

But the partial-refund case is unaddressed even when `error_refund` is enabled, because the callback never reads the promise result value.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

`Y` NEP-141 tokens are permanently locked in Aurora's account. No ERC-20 tokens exist to burn and claim them. There is no admin rescue function in the engine that allows arbitrary NEP-141 transfers out of Aurora's account. The invariant `Aurora's NEP-141 balance == ERC-20 total supply` is broken, and the surplus tokens can never be recovered by any user.

---

### Likelihood Explanation

**Medium.** The Omni path is a legitimate, documented feature for cross-contract DeFi interactions. Any receiver contract that implements `ft_on_transfer` and returns a non-zero refund (e.g., a DEX that cannot fill the full order, a lending protocol with a cap, or any contract with conditional acceptance logic) will trigger this freeze. A user exiting to such a protocol in good faith will silently lose the refunded portion of their tokens. The entry path is fully unprivileged — any ERC-20 holder can call the burn function with an Omni-formatted message.

---

### Recommendation

In `exit_to_near_precompile_callback`, when the promise result is `PromiseResult::Successful(bytes)`, decode the returned value (the net amount actually transferred by `ft_transfer_call`) and compare it to the original exit amount stored in the callback args. If the net transferred amount is less than the original amount, re-mint the difference as ERC-20 tokens to the refund address (analogous to what `refund_on_error` does for complete failures). This closes the gap between complete-failure refunds (already handled by `error_refund`) and partial-refund scenarios (currently unhandled in all configurations).

---

### Proof of Concept

1. User holds `X` ERC-20 tokens on Aurora (backed by `X` NEP-141 tokens in Aurora's account).
2. User calls the ERC-20 burn function with an Omni-formatted message targeting a receiver contract `R` on NEAR.
3. The `ExitToNear` precompile burns `X` ERC-20 tokens and schedules `ft_transfer_call(receiver=R, amount=X, msg=...)` on the NEP-141 contract.
4. `R`'s `ft_on_transfer` is called; `R` returns `Y` (where `0 < Y ≤ X`) as the refund amount.
5. The NEP-141 contract's `ft_resolve_transfer` refunds `Y` tokens to Aurora's NEP-141 balance.
6. `exit_to_near_precompile_callback` is invoked with `PromiseResult::Successful(X - Y encoded)`.
7. The callback enters the success branch, ignores the value, and returns `None` — no ERC-20 tokens are re-minted.
8. Final state: user has `0` ERC-20 tokens, received `X - Y` NEP-141 tokens at destination, and `Y` NEP-141 tokens are permanently locked in Aurora's account with no claimant.
