Audit Report

## Title
SNS Swap Permanent DoS via Single-Participant ICP Target Saturation - (File: rs/sns/swap/src/swap.rs)

## Summary
When an SNS swap is configured with `max_participant_icp_e8s == max_direct_participation_icp_e8s` (an explicitly documented valid configuration), a single attacker can contribute the full ICP target, triggering `can_abort` to return `true` before `min_participants` is reached. The heartbeat's `run_periodic_tasks` then calls `try_abort`, permanently transitioning the swap to `LIFECYCLE_ABORTED`. The attacker recovers all ICP via the permissionless `finalize_swap`, paying only two ICP ledger transfer fees.

## Finding Description
**Root cause:** `can_abort` evaluates `icp_target_progress().is_reached_or_exceeded() && !sufficient_participation()` without any time-delay guard after the ICP target is first reached.

`can_abort` in `rs/sns/swap/src/swap.rs` (L2906–2914):
```rust
pub fn can_abort(&self, now_seconds: u64) -> bool {
    if self.lifecycle() != Lifecycle::Open { return false; }
    (self.swap_due(now_seconds) || self.icp_target_progress().is_reached_or_exceeded())
        && !self.sufficient_participation()
}
```

`sufficient_participation` (L2796–2797) requires both `min_participation_reached()` (`buyers.len() >= min_participants`) AND `min_direct_participation_icp_e8s_reached()`. If a single attacker is the only buyer, `buyers.len() == 1 < min_participants` makes this false.

In `refresh_buyer_token_e8s` (L1177, L1224, L1237), the accepted increment is first capped at `available_direct_participation_e8s()` (L522–535), then further capped at `max_participant_icp_e8s` (L1237). When `max_participant_icp_e8s == max_direct_participation_icp_e8s` — a configuration explicitly documented as valid ("Can effectively be disabled by setting it to `max_icp_e8s`", proto L378–380) — a single participant can consume the entire remaining ICP capacity in one call.

The heartbeat's `run_periodic_tasks` (L1048–1054) calls `try_abort` unconditionally when `can_abort` returns true, with no minimum window after target saturation. `finalize_swap` in `canister.rs` (L150–159) has no caller restriction.

**Exploit flow:**
1. Attacker transfers `max_participant_icp_e8s` (== `max_direct_participation_icp_e8s`) ICP to their swap subaccount.
2. Attacker calls `refresh_buyer_tokens` → `current_direct_participation_e8s = max_direct_participation_e8s`, `buyers.len() = 1`.
3. `icp_target_progress().is_reached_or_exceeded()` → `true`; `sufficient_participation()` → `false` (1 < `min_participants`).
4. Next heartbeat: `run_periodic_tasks` → `try_abort` → `lifecycle = ABORTED` (terminal).
5. Attacker calls `finalize_swap` → `sweep_icp` refunds all ICP minus one transfer fee.
6. Net cost: ~0.0002 ICP.

**Existing guards that fail:** The `max_participant_icp_e8s` cap (L1237) does not prevent this when it equals `max_direct_participation_icp_e8s`. The `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` check (L1187–1197) only blocks new participants when the neuron basket limit is hit, not the ICP target. There is no guard preventing abort-on-target-reached from firing immediately.

## Impact Explanation
The SNS swap is a one-shot mechanism. Once `LIFECYCLE_ABORTED` is set (L998), it is a terminal state with no re-open path. The SNS project fails to decentralize, dapp control reverts to fallback controllers, and all legitimate participants lose the opportunity to participate. This is a concrete, permanent, application-level DoS against the SNS framework — matching **High ($2,000–$10,000): Significant SNS or infrastructure security impact with concrete user or protocol harm**.

## Likelihood Explanation
The attack requires holding `max_participant_icp_e8s` ICP transiently (fully recovered minus ~0.0002 ICP). It is only exploitable when `max_participant_icp_e8s == max_direct_participation_icp_e8s`, which is a valid and documented configuration. No privileged access, governance majority, or cryptographic capability is required. The entry path is two standard unprivileged ingress `update` calls (`refresh_buyer_tokens`, `finalize_swap`). For swaps with lower caps (e.g., 50,000–500,000 ICP), this is feasible for a well-funded attacker or a competitor motivated to prevent a rival SNS from launching.

## Recommendation
1. **Enforce a minimum participation window after ICP target is reached:** Do not allow `icp_target_reached && !sufficient_participation` to trigger an immediate abort. Require the swap to remain open for at least a configurable `min_participant_window_seconds` after the ICP target is first reached, giving legitimate participants time to join.
2. **Cap single-participant contribution relative to `min_participants`:** Enforce that no single buyer can contribute more than `max_direct_participation_icp_e8s / min_participants` during the open phase, preventing any single actor from saturating the ICP target alone.
3. **Decouple ICP-target-reached abort from the immediate heartbeat path:** Require a secondary confirmation (e.g., from NNS governance or SNS root) before aborting on target-reached-without-sufficient-participation, analogous to permissioning emergency settlement functions.

## Proof of Concept
```
Preconditions:
  - SNS swap is OPEN
  - min_participants = 100
  - max_direct_participation_icp_e8s = 500_000 * E8
  - max_participant_icp_e8s = 500_000 * E8  (valid: == max_direct_participation_icp_e8s)
  - current_direct_participation_e8s = 0
  - buyers.len() = 0

Step 2: Attacker calls refresh_buyer_tokens({ buyer: attacker_principal })
  → available_direct_participation_e8s() = 500_000 * E8
  → actual_increment_e8s = min(500_000*E8, 500_000*E8) = 500_000*E8
  → new_balance_e8s = min(500_000*E8, max_participant_icp_e8s=500_000*E8) = 500_000*E8

  → current_direct_participation_e8s = 500_000*E8 = max_direct_participation_e8s
  → icp_target_progress() = Reached(500_000*E8)

Step 3: Heartbeat fires → run_periodic_tasks → can_abort(now):
  → icp_target_progress().is_reached_or_exceeded() = true
