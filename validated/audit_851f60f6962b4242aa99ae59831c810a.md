### Title
Unchecked Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Burn Without Bridge Transfer - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first burning the caller's ERC-20 tokens via `_burn`, then invoking the Aurora exit precompile via inline assembly. The return value `res` of the `call` opcode is captured but **never checked**. If the precompile call fails for any reason (invalid recipient account ID, out of gas, unregistered token, etc.), the burn is not reverted, the transaction succeeds from the EVM's perspective, and the user's tokens are permanently destroyed with no corresponding NEP-141 transfer on the NEAR side.

---

### Finding Description

In `EvmErc20.sol`, `withdrawToNear`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no `if iszero(res) { revert(0,0) }`
    }
}
``` [1](#0-0) 

The identical pattern exists in `withdrawToEthereum` (same file, lines 65–76) and in both functions of `EvmErc20V2.sol`: [2](#0-1) [3](#0-2) [4](#0-3) 

The exit precompile (`ExitToNear::run` in `engine-precompiles/src/native.rs`) returns `Err(ExitError::...)` for multiple reachable conditions:

- Invalid or too-long NEAR recipient account ID (parsed from the raw `bytes memory recipient` argument with no prior validation in the Solidity contract)
- ERC-20 address not registered in the NEP-141 mapping (`get_nep141_from_erc20` fails)
- Malformed input (missing flag byte, wrong length)
- ETH value attached to an ERC-20 exit call [5](#0-4) 

When the precompile returns an `ExitError`, the EVM sets the `call` opcode result to `0`. Because `res` is never tested, execution falls through and the function returns successfully. The `_burn` that already executed is not rolled back.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The user's ERC-20 mirror tokens are irreversibly burned on the Aurora side. No `ft_transfer` or `ft_transfer_call` promise is ever scheduled on the NEAR side. The underlying NEP-141 tokens remain locked in the Aurora engine contract with no mechanism to recover them. Total supply on Aurora decreases while the NEAR-side balance of the Aurora contract does not decrease, creating a permanent accounting discrepancy and a permanent loss of the user's bridged assets.

---

### Likelihood Explanation

**Medium–High.** The `recipient` parameter in `withdrawToNear` is raw `bytes memory` with no length or character validation in Solidity before the precompile call. A user who passes a recipient string that is too long (NEAR account IDs are capped at 64 bytes), contains invalid characters, or is otherwise malformed will silently lose their tokens. This is a realistic user error path, not a contrived attack. Additionally, any future condition that causes the precompile to return an error (e.g., a paused state, a missing NEP-141 registration) would trigger the same silent loss.

---

### Recommendation

After the `call` opcode, check `res` and revert if it is zero:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply this fix to all four assembly blocks across `EvmErc20.sol` and `EvmErc20V2.sol` (both `withdrawToNear` and `withdrawToEthereum` in each file). This ensures that if the precompile rejects the call for any reason, the entire transaction reverts, the `_burn` is rolled back, and the user retains their tokens.

---

### Proof of Concept

1. Deploy `EvmErc20` (or `EvmErc20V2`) as a bridged NEP-141 token on Aurora.
2. Mint tokens to `alice` (EVM address).
3. `alice` calls `withdrawToNear(recipient, amount)` where `recipient` is a 65-byte string (one byte over the NEAR account ID limit).
4. `_burn(alice, amount)` executes — `alice`'s ERC-20 balance drops to zero.
5. The precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` rejects the call (invalid account ID), returning `ExitError`. The `call` opcode returns `res = 0`.
6. `res` is never checked. The function returns without reverting.
7. The EVM transaction succeeds. `alice`'s ERC-20 tokens are gone. No `ft_transfer` promise is scheduled. The NEP-141 balance of the Aurora contract is unchanged. `alice`'s funds are permanently lost.
