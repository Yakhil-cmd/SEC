After thorough investigation of the ZKsync OS codebase, I found a valid analog to the reported vulnerability class.

---

### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Causes Systematic User Gas Overcharge — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

In `compute_gas_refund`, the dual-resource reconciliation step adjusts `gas_used` upward when native consumption exceeds gas consumption (`delta_gas > 0`), but **never adjusts it downward** when native consumption is lower (`delta_gas < 0`). Users whose transactions consume less native resource than their EVM gas would imply are systematically overcharged, with the excess fee going to the operator rather than being refunded. A `// TODO: return delta_gas to gas_used?` comment in the code explicitly acknowledges this asymmetry.

### Finding Description

ZKsync OS implements a dual-resource model: EVM gas (ergs) and a native resource (RISC-V proving cost). After execution, `compute_gas_refund` reconciles the two via a `delta_gas` value:

```
delta_gas = (native_used / native_per_gas) - gas_used
``` [1](#0-0) 

The code only handles the positive branch:

```rust
if delta_gas > 0 {
    // native consumption > gas consumption → charge extra gas
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

When `delta_gas < 0` (native consumption is **lower** than gas consumption), `gas_used` is left unchanged. The user is then refunded `(gas_limit - gas_used) * gas_price`, but the correct refund should be `(gas_limit - (gas_used + delta_gas)) * gas_price`. Since `delta_gas < 0`, the user is underrefunded by `|delta_gas| * gas_price`.

The design documentation in `docs/double_resource_accounting.md` only describes the positive case:

> "If deltaGas > 0, we add it to gasUsed and charge it from ergs." [2](#0-1) 

The negative case is entirely absent from the spec, and the in-code TODO confirms it is an unresolved gap.

The `ResourcesForTx` struct splits native resources into `main_resources` (capped at `MAX_NATIVE_COMPUTATIONAL`) and `withheld` (excess, for pubdata only). The withheld resources are correctly reclaimed before refund calculation. The asymmetry is therefore isolated to the `delta_gas` step. [3](#0-2) [4](#0-3) 

### Impact Explanation

Any L2 transaction (ZK flow) where native resource consumption is lower than EVM gas consumption results in the user being overcharged. The overcharge amount is `|delta_gas| * gas_price`. This is a direct, permanent financial loss to the user — the excess fee is credited to the operator via the normal `token_to_pay_operator = gas_used * gas_price_for_operator` path. [5](#0-4) 

This matches the analog vulnerability class: an intermediate resource value is capped/floored asymmetrically in a multi-step accounting flow, and the excess is not returned to the rightful owner.

### Likelihood Explanation

- Requires non-zero gas price (standard for all real L2 transactions).
- Occurs whenever a transaction performs significant EVM computation (high gas) but relatively few storage writes or precompile calls (low native). This is a common pattern for compute-heavy contracts (e.g., AMM math, signature verification loops, complex state machines).
- No special attacker setup is needed — any user submitting a standard L2 transaction can trigger this.
- The `#[cfg(not(feature = "unlimited_native"))]` guard confirms this path is active in production builds. [1](#0-0) 

### Recommendation

When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (subject to the `minimal_gas_used` floor) to return the native-resource savings to the user as a gas refund:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else {
    // delta_gas < 0: native consumption is lower than gas consumption.
    // Return the savings to the user, but do not go below minimal_gas_used.
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

### Proof of Concept

**Setup**: L2 transaction, `gas_limit = 100_000`, `gas_price = 1_000 wei`, `native_price = 100`, so `native_per_gas = 10`.

**Execution**: Transaction uses 50,000 EVM gas (`gas_used = 50,000`) but only 100,000 native units (`native_used = 100,000`).

**Current behavior**:
- `delta_gas = (100_000 / 10) - 50_000 = 10_000 - 50_000 = -40_000` → no adjustment
- `gas_used` stays at `50_000`
- Refund = `(100_000 - 50_000) * 1_000 = 50_000_000 wei`

**Correct behavior**:
- `gas_used` should be reduced to `10_000`
- Refund = `(100_000 - 10_000) * 1_000 = 90_000_000 wei`

**User loss**: `40_000 * 1_000 = 40_000_000 wei` per transaction, permanently transferred to the operator. [6](#0-5) [7](#0-6)
