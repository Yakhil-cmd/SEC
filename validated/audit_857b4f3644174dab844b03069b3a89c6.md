### Title
Unbounded Iteration Over All Block Events in EIP-6110 Deposit Parser Causes State-Transition Divergence and Unbounded Proving Work - (`File: basic_bootloader/src/bootloader/block_flow/ethereum/eip_6110_deposit_events_parser/mod.rs`)

### Summary

`eip6110_events_parser` iterates over every event emitted in the entire block without any cap on the number of deposit events processed. The EIP-6110 specification mandates `MAX_DEPOSIT_REQUESTS_PER_PAYLOAD = 8192`. ZKsync OS enforces no such limit, unlike its own EIP-7002 implementation which correctly caps at `MAX_WITHDRAWAL_REQUESTS_PER_BLOCK = 16`. This creates both a state-transition divergence from the Ethereum spec and unbounded post-block proving work that is not charged to any transaction.

### Finding Description

`eip6110_events_parser` is called during block finalization (post-op) to collect EIP-6110 deposit requests and hash them into the block's `requests_hash`. The function unconditionally iterates over the full block event list:

```rust
for event in system.io.events_iterator() {
    if event.address != &DEPOSIT_CONTRACT_ADDRESS {
        continue;
    }
    if event.topics.len() > 0 && event.topics[0] == DEPOSIT_EVENT_SIGNATURE_HASH {
        // process deposit — no counter, no cap
        validate_and_write_event_data(event, requests_hasher, &mut logger)...
    }
}
``` [1](#0-0) 

There is no `MAX_DEPOSIT_REQUESTS_PER_PAYLOAD` constant and no `core::cmp::min(...)` guard anywhere in this function. Compare this directly with the EIP-7002 implementation in the same codebase, which explicitly caps the number of items dequeued per block:

```rust
const MAX_WITHDRAWAL_REQUESTS_PER_BLOCK: usize = 16;
...
let num_dequeued = core::cmp::min(
    u256_to_usize_saturated(&num_in_queue),
    MAX_WITHDRAWAL_REQUESTS_PER_BLOCK,
);
``` [2](#0-1) [3](#0-2) 

The EIP-6110 parser is invoked in both the proving path (`post_op_io_touching_impl`) and the sequencing path (`post_op`): [4](#0-3) [5](#0-4) 

The iteration is over `system.io.events_iterator()`, which returns **all events from all transactions in the block** — not just deposit events. Every non-deposit event is also visited (and skipped), making the work O(total block events), not O(deposit events). [6](#0-5) 

This post-op work runs outside any per-transaction gas or native resource budget. The resources used in `eip6110_events_parser` are not charged to any transaction.

### Impact Explanation

**State-transition divergence (primary)**: A block containing more than 8192 deposit events from the deposit contract would have all of them hashed into `requests_hash` by ZKsync OS, while a spec-compliant Ethereum execution client would stop at 8192. The resulting `requests_hash` field in the block header would differ, making the block invalid from the Ethereum consensus perspective. This is a direct EVM semantic mismatch / state-transition bug.

**Resource accounting bug (secondary)**: The post-op phase runs with `with_infinite_ergs` — no native resource is charged for the iteration. A block maximally filled with deposit-emitting transactions (each individually valid within the block gas limit) forces the prover to perform O(N) unbounded work in the post-op phase with no corresponding resource charge. This can cause valid-execution unprovability if the RISC-V prover has a fixed cycle budget for block finalization.

### Likelihood Explanation

The deposit contract (`0x00000000219ab540356cBB839Cbe05303d7705Fa`) is a real Ethereum predeployed contract. Any unprivileged user can trigger a deposit event by sending ETH to it with the correct calldata. A block proposer can fill a block with such transactions — each individually valid and within the block gas limit — to maximize the number of deposit events. With a 30M gas block and a deposit transaction costing roughly 50,000–100,000 gas each, hundreds to thousands of deposit events per block are achievable, well above the 8192 spec limit in extreme cases. No privileged access is required.

### Recommendation

1. Add a `MAX_DEPOSIT_REQUESTS_PER_PAYLOAD` constant (value `8192` per the EIP-6110 spec) and enforce it with a counter inside the loop, breaking or returning once the limit is reached — mirroring the pattern already used in `eip7002_system_part`.
2. Consider tracking the deposit event count separately from the full event scan to avoid O(total_events) work; alternatively, enforce the cap early to bound the iteration.

### Proof of Concept

1. Deploy a contract that calls the EIP-6110 deposit contract with valid deposit calldata in a loop.
2. Submit a transaction that emits, say, 9000 deposit events from `DEPOSIT_CONTRACT_ADDRESS` with the correct `DEPOSIT_EVENT_SIGNATURE_HASH` topic.
3. ZKsync OS's `eip6110_events_parser` will hash all 9000 deposits into `requests_hash`.
4. A reference Ethereum client (geth/reth) will only hash the first 8192 deposits.
5. The resulting `requests_hash` values diverge, demonstrating the state-transition bug.

pub fn eip6110_events_parser<S: EthereumLikeTypes>(
    system: &System<S>,
    requests_hasher: &mut impl crypto::sha256::Digest,
) -> Result<bool, SystemError>
where
    S::IO: IOSubsystemExt + IOTeardown<S::IOTypes>,
{
    // we can not easily get the number from one scan, so we will accumulate into hasher directly

    let mut event_encountered = false;
    let mut logger = system.get_logger();
    for event in system.io.events_iterator() {
        if event.address != &DEPOSIT_CONTRACT_ADDRESS {
            continue;
        }
        if event.topics.len() > 0 && event.topics[0] == DEPOSIT_EVENT_SIGNATURE_HASH {
            if event_encountered == false {
                event_encountered = true;
                requests_hasher.update(&[DEPOSIT_REQUEST_EIP_7685_TYPE]);
            }
            let Ok(_) = validate_and_write_event_data(event, requests_hasher, &mut logger) else {
                panic!("invalid deposit event structure");
            };
        }
    }

    Ok(event_encountered)
}
