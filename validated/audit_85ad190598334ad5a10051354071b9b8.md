### Title
Silo Address Whitelist Not Enforced for ETH Base-Token Bridge Deposits — (`engine/src/engine.rs`)

---

### Summary

In Silo mode with the `Address` whitelist enabled, `receive_base_tokens()` mints ETH to any recipient address without checking the whitelist, while `receive_erc20_tokens()` correctly enforces the same whitelist by redirecting tokens to the configured `erc20_fallback_address`. Any token holder can bridge ETH to a non-whitelisted EVM address, bypassing the Silo access-control policy and permanently depositing ETH outside the operator's control.

---

### Finding Description

Aurora Engine's Silo mode exposes a `WhitelistKind::Address` list that is intended to restrict which EVM addresses may participate in the silo. The function `is_allow_receive_erc20_tokens` enforces this list for ERC-20 bridge deposits:

```rust
// engine/src/engine.rs  receive_erc20_tokens()
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;   // redirect to fallback
}
``` [1](#0-0) 

`is_allow_receive_erc20_tokens` delegates to `is_address_allowed`, which returns `false` for any address absent from the enabled `Address` whitelist:

```rust
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}
fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
    !list.is_enabled() || list.is_exist(address)
}
``` [2](#0-1) 

However, the sibling function `receive_base_tokens()`, which is invoked when the eth-connector bridges ETH (i.e., when `predecessor_account_id == get_connector_account_id`), contains **no whitelist check whatsoever**:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let receipient = message_data.recipient;
    let balance = get_balance(&self.io, &receipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &receipient, &new_balance);
    ...
}
``` [3](#0-2) 

The dispatch between the two paths occurs in `ft_on_transfer`:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)   // ← no whitelist check
} else {
    engine.receive_erc20_tokens(...)    // ← whitelist enforced
};
``` [4](#0-3)

---

### Impact Explanation

In a Silo deployment where the `Address` whitelist is active (e.g., for KYC/AML or permissioned-chain compliance), the operator's invariant is that **only whitelisted EVM addresses may hold or receive tokens**. ERC-20 tokens respect this invariant: a non-whitelisted recipient causes the tokens to be redirected to the `erc20_fallback_address`, keeping them under the operator's control. ETH deposits violate this invariant: they are minted directly to the caller-specified recipient with no redirect, permanently placing ETH in a non-whitelisted address that the Silo operator cannot reclaim. This is a **whitelist bypass** leading to **permanent loss of operator control over bridged ETH** — equivalent in class to the external report's finding where a blacklisted entity can still receive and hold assets that the protocol intended to block.

---

### Likelihood Explanation

Any NEAR account that holds ETH on the eth-connector side can trigger this by calling `ft_transfer_call` on the connector contract with `msg` set to the hex-encoded address of any non-whitelisted EVM address. No special privilege is required. The operation is a standard bridge deposit flow available to all token holders.

---

### Recommendation

Apply the same whitelist-and-fallback logic to `receive_base_tokens` that already exists in `receive_erc20_tokens`:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let mut receipient = message_data.recipient;

+   // Enforce Silo Address whitelist for ETH deposits, mirroring ERC-20 behaviour.
+   if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
+       && !silo::is_allow_receive_erc20_tokens(&self.io, &receipient)
+   {
+       receipient = fallback_address;
+   }

    let balance = get_balance(&self.io, &receipient);
    ...
}

### Proof of Concept

1. Deploy Aurora Engine in Silo mode: enable `WhitelistKind::Address` whitelist and set `erc20_fallback_address` to `fallback_addr`.
2. Do **not** add EVM address `victim_addr` to the Address whitelist.
3. Verify ERC-20 behaviour: call `ft_transfer_call` on a NEP-141 token with `msg = hex(victim_addr)`. Observe tokens land at `fallback_addr`, not `victim_addr`. ✓ whitelist enforced.
4. Now call `ft_transfer_call` on the **eth-connector** with `msg` encoding `victim_addr` as the ETH recipient.
5. Observe that `ft_on_transfer` dispatches to `receive_base_tokens`, which mints ETH directly to `victim_addr` with no whitelist check and no redirect to `fallback_addr`. ✗ whitelist bypassed.
6. `victim_addr` now holds ETH inside the Silo despite never being whitelisted; the Silo operator has no mechanism to recover it.
