### Title
Node Operator Can Permanently Block Their Own Removal from NNS Registry - (File: rs/registry/canister/src/mutations/do_remove_node_operators.rs)

### Summary
The `do_remove_node_operators` function in the NNS registry canister silently skips any node operator that has at least one associated node record. Because a registered node operator can call `add_node` directly on the registry canister at any time (including during the multi-day NNS proposal voting period), a targeted node operator can permanently prevent their own removal by ensuring they always maintain at least one node record.

### Finding Description
`do_remove_node_operators` is the execution handler for the `NnsFunction::RemoveNodeOperators` governance proposal type. It first filters out operators that have no `NodeOperatorRecord`, then calls `filter_out_node_operators_that_have_nodes`, which silently drops from the removal list any operator that owns at least one `NodeRecord` in the registry. [1](#0-0) 

The function explicitly documents this behavior: *"this retains only 'empty' node operators, i.e. those that have ZERO nodes."* The result is that the governance proposal executes successfully (no error, no revert), but the targeted operator is simply not removed. [2](#0-1) 

The `add_node` endpoint on the registry canister is callable directly by any principal that already has a `NodeOperatorRecord`. It uses `dfn_core::api::caller()` to identify the caller and checks only that the caller is a registered node operator with remaining quota. [3](#0-2) 

NNS governance proposals have a voting period of multiple days before execution. During this window, the targeted node operator can submit an `add_node` call with valid cryptographic keys, creating a `NodeRecord` under their principal. When the proposal finally executes, `filter_out_node_operators_that_have_nodes` silently excludes them, and the proposal completes with `Executed` status while the operator remains in the registry.

This is confirmed by the integration test that explicitly documents the behavior: [4](#0-3) 

### Impact Explanation
A malicious or non-compliant node operator can permanently evade removal from the NNS registry. The NNS governance — the only entity authorized to call `remove_node_operators` — loses the ability to enforce operator deregistration. The operator retains their `NodeOperatorRecord`, continues to be able to add nodes to the IC, and continues to receive node provider rewards. This undermines the NNS's authority over the node operator set, which is a core governance authorization guarantee of the IC.

### Likelihood Explanation
The attack requires no special privileges beyond being a registered node operator (which is the precondition for being targeted by a `RemoveNodeOperators` proposal in the first place). The attacker has days to observe the pending proposal on the NNS dashboard and submit a single `add_node` call. The `add_node` endpoint is a standard, publicly documented registry update method. No cryptographic secrets, subnet-majority corruption, or social engineering is required. The only technical barrier is generating valid node public keys, which is a routine operation for any node operator.

### Recommendation
**Short term:** Remove the silent-skip behavior from `do_remove_node_operators`. Instead, either:
1. Allow removal of a node operator even if they have nodes, and atomically delete or reassign the associated `NodeRecord` entries in the same mutation batch; or
2. Reject the proposal with an explicit error if the targeted operator has nodes, so governance can first pass a `RemoveNodes` proposal to clear the operator's nodes before retrying removal.

**Long term:** Audit all NNS governance execution handlers for similar patterns where the target of a privileged administrative action can manipulate their own state to silently nullify the action. The `filter_out_node_operators_that_have_nodes` pattern — silently skipping targets rather than failing — is particularly dangerous because it gives no feedback to governance that the intended action did not occur.

### Proof of Concept
1. NNS governance submits a `RemoveNodeOperators` proposal targeting node operator principal `Eve`.
2. During the voting period, `Eve` calls `add_node` on the registry canister with a valid `AddNodePayload` (valid cryptographic keys, valid endpoint). This succeeds because `Eve` has a `NodeOperatorRecord` with remaining `max_rewardable_nodes` quota.
3. The proposal passes and `do_remove_node_operators` executes.
4. `filter_out_node_operators_that_have_nodes` scans all `NodeRecord` entries and finds `Eve`'s node; `Eve` is removed from `valid_node_operator_ids_to_remove`.
5. The mutation batch contains no deletion for `Eve`'s `NodeOperatorRecord`. The proposal status is `Executed`.
6. `Eve` remains a registered node operator. She can repeat step 2 for every future removal proposal targeting her. [5](#0-4) [6](#0-5)

    pub fn do_remove_node_operators(&mut self, payload: RemoveNodeOperatorsPayload) {
        println!("{LOG_PREFIX}do_remove_node_operators: {payload:?}");

        let mut mutations = vec![];

        // Filter Node Operator IDs that have a NodeOperatorRecord in the Registry
        let mut valid_node_operator_ids_to_remove: Vec<PrincipalId> = payload
            .principal_ids_to_remove()
            .into_iter()
            .filter(|node_operator_id| {
                let node_operator_record_key =
                    make_node_operator_record_key(*node_operator_id).into_bytes();
                self.get(&node_operator_record_key, self.latest_version())
                    .is_some()
            })
            .collect();

        self.filter_out_node_operators_that_have_nodes(&mut valid_node_operator_ids_to_remove);

        for node_operator_id in valid_node_operator_ids_to_remove {
            let node_operator_record_key =
                make_node_operator_record_key(node_operator_id).into_bytes();
            mutations.push(RegistryMutation {
                mutation_type: registry_mutation::Type::Delete as i32,
                key: node_operator_record_key,
                value: vec![],
            });
        }

        self.maybe_apply_mutation_internal(mutations);
    }

    /// Takes a set of node operators and removes the node operators for which
    /// there exist at least one node record that is managed by the node
    /// operator.
    ///
    /// In other words, this retains only "empty" node operators, i.e. those
    /// that have ZERO nodes.
    fn filter_out_node_operators_that_have_nodes(&self, node_operators: &mut Vec<PrincipalId>) {
        if node_operators.is_empty() {
            return;
        }

        // This implementation is inefficient, because it does a full scan of all nodes.
        for (_key, node_record) in get_key_family_iter::<NodeRecord>(self, NODE_RECORD_KEY_PREFIX) {
            // Throw out node operators that operate the node (that this loop is currently considering).
            node_operators
                .retain(|node_operator| node_operator.to_vec() != node_record.node_operator_id);
        }
    }
