Audit Report

## Title
ckERC20 Withdrawal Permanently Stuck With No Reimbursement Path When Gas Fees Spike After Request Acceptance — (`rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary

When a user calls `withdraw_erc20`, the minter burns ckETH (gas fee) and ckERC20 tokens immediately at request-acceptance time, locking `max_transaction_fee` based on the gas estimate at that instant. If Ethereum gas fees spike before the minter's timer processes the request, `create_transaction` returns `InsufficientTransactionFee` and `create_transactions_batch` only reschedules the request with no reimbursement triggered. Because `maybe_reimburse` is only populated inside `record_created_transaction` — which is only reached on success — a request that never passes `create_transaction` has no protocol-level escape hatch: no cancel endpoint, no timeout-based reimbursement, and no way to top up `max_transaction_fee`.

## Finding Description

**Fee locked at request time.** In `main.rs`, `estimate_erc20_transaction_fee()` is called and the result is immediately burned from the user's ckETH account, then stored as the immutable `Erc20WithdrawalRequest.max_transaction_fee`. [1](#0-0) [2](#0-1) 

**Gas fee re-checked at execution time.** When the timer fires, `lazy_refresh_gas_fee_estimate()` fetches the current fee and passes it to `create_transactions_batch`. Inside `create_transaction`, for `WithdrawalRequest::CkErc20`, the code computes `actual_min_max_fee_per_gas` from the fresh estimate and compares it against `request_max_fee_per_gas` derived from the locked `max_transaction_fee`. [3](#0-2) 

**Mismatch causes indefinite rescheduling with no reimbursement.** When `InsufficientTransactionFee` is returned, `create_transactions_batch` calls only `reschedule_withdrawal_request`, moving the request to the back of the queue. No reimbursement is scheduled. [4](#0-3) 

**Reimbursement pipeline is unreachable for stuck pending requests.** The `maybe_reimburse` set — which drives the entire reimbursement flow — is only populated inside `record_created_transaction`, which is only reached when `create_transaction` succeeds. A request that never passes the fee check never enters the reimbursement pipeline. [5](#0-4) 

**No cancel endpoint exists.** A grep across all minter source files confirms there is no `cancel_withdrawal` or equivalent endpoint. The `max_transaction_fee` field in `Erc20WithdrawalRequest` is set once at burn time and is structurally immutable for the lifetime of the request. [6](#0-5) 

**Post-creation resubmission path also only logs on fee excess.** If a sent ckERC20 transaction cannot be resubmitted because the new fee exceeds `allowed_max_transaction_fee`, `resubmit_transactions_batch` only logs the error. [7](#0-6) 

## Impact Explanation

A user whose `withdraw_erc20` request is accepted during a low-fee window but not processed before a gas fee spike will have both their ckETH (gas fee) and ckERC20 tokens burned with no reimbursement and no cancellation path. The request cycles indefinitely in `pending_withdrawal_requests`. If gas fees remain elevated for an extended period, the user's funds are effectively frozen with no protocol-level recourse. This constitutes concrete, demonstrable user-fund harm in an in-scope ck-token / Chain Fusion component, matching the **High** impact tier: *"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."*

## Likelihood Explanation

Ethereum gas fees are historically volatile; 5–20× spikes within minutes are well-documented during network congestion events. The minter's gas fee cache has a 60-second maximum age (`MAX_AGE_NS = 60_000_000_000`), and the processing queue introduces a natural delay between request acceptance and transaction creation, widening the divergence window. [8](#0-7) 

The condition is reachable by any unprivileged user via the public `withdraw_erc20` endpoint with no special privileges required. [9](#0-8) 

## Recommendation

1. **Add a reimbursement path for indefinitely stuck pending requests.** In `create_transactions_batch`, when `InsufficientTransactionFee` is returned for a `CkErc20` request, track a consecutive-failure counter or a `created_at`-relative timeout per request. After N failures or after a configurable deadline, trigger the existing reimbursement flow (populating `reimbursement_requests` directly, bypassing `maybe_reimburse`) to return both the burned ckETH and ckERC20 to the user.

2. **Add a user-callable cancellation endpoint for pending ckERC20 withdrawals.** Allow the original requester to cancel a withdrawal that has not yet had a transaction created, triggering reimbursement of both burned amounts minus a small penalty fee, mirroring the existing ckERC20 burn-failure reimbursement logic already present in `main.rs`. [10](#0-9) 

3. **Defer the ckETH burn until transaction creation time**, so the burned amount reflects the actual execution-time fee rather than the request-time estimate, eliminating the mismatch entirely.

## Proof of Concept

1. Ethereum gas fees are at 10 gwei. User calls `withdraw_erc20` for 100 ckUSDC. Minter calls `estimate_erc20_transaction_fee()`, burns ~0.0014 ckETH and 100 ckUSDC from the user. `max_transaction_fee` = ~0.0014 ETH is locked in the `Erc20WithdrawalRequest`.

2. Before the minter's next processing timer fires, Ethereum gas fees spike to 200 gwei (20× increase).

3. `process_retrieve_eth_requests` fires. `lazy_refresh_gas_fee_estimate` returns 200 gwei. `create_transaction` computes `actual_min_max_fee_per_gas = 200 + priority_fee gwei`. `request_max_fee_per_gas = 0.0014 ETH / 65_000 ≈ 21.5 gwei`. Since `200+ > 21.5`, `InsufficientTransactionFee` is returned. [11](#0-10) 

4. `create_transactions_batch` calls `reschedule_withdrawal_request`. No reimbursement is scheduled. `maybe_reimburse` is never populated. [4](#0-3) 

5. The request cycles indefinitely. The user's 0.0014 ckETH and 100 ckUSDC are burned. `retrieve_eth_status` returns `Pending` indefinitely. No cancel endpoint exists. The user has no recourse within the protocol.

A deterministic integration test can reproduce this by: (a) submitting a `withdraw_erc20` call with a low gas estimate, (b) injecting a high gas fee estimate into `last_transaction_price_estimate` before the timer fires, (c) asserting that after N timer invocations the request remains in `pending_withdrawal_requests` with no entry in `reimbursement_requests` and no entry in `maybe_reimburse`.
