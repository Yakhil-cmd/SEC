Audit Report

## Title
VIP Callers Blocked by Non-VIP Slot Exhaustion in `try_borrow_slot` — (`rs/nervous_system/clients/src/management_canister_client.rs`)

## Summary
`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` contains an early-exit guard (`if *available_slot_count == 0`) that fires before the VIP distinction is applied, causing VIP callers (NNS Governance) to receive `Err(SysTransient)` even though VIP callers are designed to consume zero slots. An unprivileged attacker can exhaust all 167 slots via NNS Root's public `canister_status` endpoint, then block Governance-triggered `load_canister_snapshot` and `take_canister_snapshot` operations for the duration of the flood.

## Finding Description

**Root cause — `try_borrow_slot`** (`rs/nervous_system/clients/src/management_canister_client.rs`, lines 264–287):

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if *available_slot_count == 0 {   // ← blocks VIP too
                let code = RejectCode::SysTransient as i32;
                let message = "Unavailable. Maybe, try again later?".to_string();
                return Err((code, message));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
``` [1](#0-0) 

The `used_slot_count = 0` branch for VIP callers is computed but never reached when `available_slot_count == 0`, because the guard returns `Err` unconditionally before the subtraction.

**Slot pool** (`rs/nns/handlers/root/impl/canister/canister.rs`, line 50): initialized to 167, shared across all callers of NNS Root. [2](#0-1) 

**VIP determination** (`rs/nns/handlers/root/impl/canister/canister.rs`, lines 57–66): Governance is a VIP because it is in `ALL_NNS_CANISTER_IDS`, so `is_caller_vip = true` when Governance calls `load_canister_snapshot` or `take_canister_snapshot`. [3](#0-2) 

**Public attack surface** (`rs/nns/handlers/root/impl/canister/canister.rs`, lines 88–98): `canister_status` has no access control. Each call borrows a slot and holds it for the duration of the in-flight management canister call. [4](#0-3) 

**Blocked VIP paths**: Both `load_canister_snapshot` and `take_canister_snapshot` call `new_management_canister_client()`, which routes through `try_borrow_slot`. [5](#0-4) 

**Error propagation via `perform_offline_canister_maintenance`**: For non-Governance target canisters, the flow is: acquire lock → stop target canister (via `CdkRuntime` directly, no slot consumed) → call `management_canister_client.load_canister_snapshot(args)` → `try_borrow_slot` returns `Err(SysTransient)` → restart target canister → return `LoadCanisterSnapshotResponse::Err(...)` to Governance. [6](#0-5) 

The stop/start calls in `perform_offline_canister_maintenance` use `CdkRuntime` directly and do not consume slots, so the target canister is stopped and restarted with no snapshot taken or loaded. [7](#0-6) 

## Impact Explanation

Governance-approved `TakeCanisterSnapshot` and `LoadCanisterSnapshot` proposals fail with an error response from NNS Root. For non-Governance target canisters (ledger, SNS canisters, etc.), the target canister is stopped and restarted with no snapshot taken or loaded — the proposal execution is wasted. For the Governance canister itself, the background task silently fails. This constitutes an application/platform-level DoS on NNS governance disaster recovery operations, matching the **High ($2,000–$10,000)** impact class: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation

The `canister_status` endpoint is publicly callable with no rate limiting in the canister code. 167 concurrent in-flight calls is well within IC queue limits (500 per queue). Management canister `canister_status` calls complete in milliseconds, so the attacker must continuously re-flood, but this is straightforward to automate. No privileged access, key material, or social engineering is required. The attack is repeatable and can be sustained indefinitely.

## Recommendation

In `try_borrow_slot`, skip the `== 0` guard entirely for VIP callers, since they consume zero slots and should never be blocked:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if !self.is_caller_vip && *available_slot_count == 0 {
                let code = RejectCode::SysTransient as i32;
                let message = "Unavailable. Maybe, try again later?".to_string();
                return Err((code, message));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

State-machine test outline mirroring the existing test structure in `rs/nervous_system/clients/src/management_canister_client/tests.rs`: [8](#0-7) 

1. Initialize NNS Root with `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT = 167`.
2. Submit 167 concurrent ingress `canister_status` calls from an unprivileged principal, each targeting a valid NNS canister ID. Configure the mock management canister to hold responses (simulate slow responses).
3. Assert `available_slot_count == 0`.
4. Submit a `load_canister_snapshot` call from the Governance canister principal (`is_caller_vip = true`).
5. Assert the response is `LoadCanisterSnapshotResponse::Err { code: Some(SysTransient), ... }` instead of `Ok`.
