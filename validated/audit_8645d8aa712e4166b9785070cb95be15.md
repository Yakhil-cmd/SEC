Audit Report

## Title
Unchecked Signed-to-Unsigned Cast in `ledgeramount_from_amount` Produces Phantom Token Amount - (File: rs/rosetta-api/icp/src/models/amount.rs)

## Summary
`from_amount` intentionally accepts negative `i128` values (to represent debits) but performs no sign check before returning. `ledgeramount_from_amount` then casts the result directly to `u64` via `as`, which in Rust is defined two's-complement truncation — a negative input such as `-1000` produces `18_446_744_073_709_550_616`. An additional instance of the same pattern exists in `convert.rs` for the `Fee` operation type. The ICP ledger will ultimately reject any submitted transaction exceeding the sender's balance, but the Rosetta API layer silently constructs and propagates the malformed `Tokens` object, producing incorrect data for exchange integrations that rely on the Rosetta API as a trusted intermediary.

## Finding Description

**Root cause — `from_amount` (amount.rs L23-40):**
`from_amount` parses the user-supplied string into `i128`, validates only that `val.abs()` fits in `u64`, and returns `val` unchanged. A negative value such as `"-1000"` passes this check because `1000_i128` fits in `u64`. [1](#0-0) 

**Unsafe cast — `ledgeramount_from_amount` (amount.rs L42-44):**
The returned signed value is cast to `u64` with `as` — no sign check, no `try_from`. In Rust, `(-1000_i128) as u64 == 18_446_744_073_709_550_616`. [2](#0-1) 

**Call site — `Disburse` branch (convert.rs L258-266):**
`ledgeramount_from_amount` is called directly for `Disburse` operations, passing the phantom large `Tokens` value into `state.disburse`. [3](#0-2) 

**Second instance — `Fee` branch (convert.rs L188):**
For `Fee` operations, `(-amount) as u64` is applied inline. A positive fee amount (attacker-supplied) produces a negative intermediate, which wraps on cast. [4](#0-3) 

**Third instance — `State::transaction` debit branch (state.rs L123-126):**
`(-amount) as u64` is used for the debit side of a `Transaction` operation. For `i128::MIN`, `-amount` overflows `i128`; however, `from_amount`'s `val.abs()` check causes a panic (debug) or wraps to `i128::MIN` (release), which then fails `u64::try_from`, so `i128::MIN` is rejected before reaching this point. For all other negative values the cast is arithmetically correct (negation of a valid negative fits in u64), so this branch is less impactful than the `ledgeramount_from_amount` path. [5](#0-4) 

**Existing checks are insufficient:**
`from_amount`'s only guard is `u64::try_from(val.abs())`, which explicitly preserves the sign of the return value. There is no downstream sign check in `ledgeramount_from_amount` or in the `Fee` inline cast. [6](#0-5) 

## Impact Explanation

An unprivileged caller can cause the ICP Rosetta API to construct `Tokens` objects with phantom astronomically large e8s values. These malformed objects are propagated through construction endpoints (`/construction/parse`, `/construction/payloads`) and returned to callers. Exchange integrations that treat the Rosetta API as a trusted intermediary will receive incorrect operation amounts, leading to incorrect internal accounting, misleading construction responses, and potential operational failures. This matches **High ($2,000–$10,000): Significant Rosetta API security impact with concrete user or protocol harm** — the Rosetta API is an in-scope financial integration component and the malformed data is directly observable by any caller without authentication.

## Likelihood Explanation

The Rosetta API is a publicly reachable HTTP service requiring no authentication. The attacker supplies a JSON body with a negative `value` string (e.g., `"-1000"`) in the `Amount` field of a `Disburse` or `Fee` operation. `from_amount` is explicitly designed to accept negative values, so the input passes validation and reaches the unsafe cast on every invocation. No special privileges, victim interaction, or race conditions are required. The attack is trivially repeatable.

## Recommendation

In `ledgeramount_from_amount`, add an explicit non-negativity check before the cast:

```rust
pub fn ledgeramount_from_amount(amount: &Amount, token_name: &str) -> Result<Tokens, String> {
    let inner = from_amount(amount, token_name)?;
    if inner < 0 {
        return Err(format!("Amount must be non-negative, got {inner}"));
    }
    Ok(Tokens::from_e8s(inner as u64))
}
```

In `convert.rs` for the `Fee` branch, replace the inline `(-amount) as u64` with a checked negation and `u64::try_from`:

```rust
let abs_amount = u64::try_from((-amount)).map_err(|_| ...)?;
state.fee(account, Tokens::from_e8s(abs_amount))?;
```

Alternatively, `from_amount` itself could be split into a signed variant (for `State::transaction`) and an unsigned variant (for all other callers), enforcing the sign constraint at the type level.

## Proof of Concept

1. Start the ICP Rosetta API pointed at any ICP node.
2. POST to `/construction/parse` (unsigned) with a `Disburse` operation whose `amount.value` is `"-1000"` and `currency` is `{"symbol":"ICP","decimals":8}`.
3. Observe: `from_amount` returns `-1000_i128`; `ledgeramount_from_amount` computes `(-1000_i128) as u64 == 18_446_744_073_709_550_616`; `Tokens::from_e8s(18_446_744_073_709_550_616)` is constructed and returned in the parsed operations.
4. A unit test can confirm this without a running node:
```rust
#[test]
fn test_negative_amount_wraps() {
    let amount = Amount { value: "-1000".to_string(), currency: Currency::new("ICP".into(), 8), metadata: None };
    let tokens = ledgeramount_from_amount(&amount, "ICP").unwrap();
    // Without the fix, this assertion holds — demonstrating the phantom value:
    assert_eq!(tokens.get_e8s(), (-1000_i128) as u64);
}
```
