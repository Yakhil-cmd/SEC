### Title
Unchecked Precompile Call Return Value in `withdrawToNear` / `withdrawToEthereum` Causes Permanent Token Burn Without Bridge Credit - (`File: etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first burning the caller's ERC-20 tokens and then invoking the Aurora exit precompile via a low-level `call` in inline assembly. The return value of that `call` (`res`) is captured but **never checked**. If the precompile call fails and returns `0`, the Solidity function returns successfully anyway — the tokens are permanently destroyed on the EVM side while the corresponding NEP-141 tokens on NEAR are never released.

---

### Finding Description

In `EvmErc20.sol`, `withdrawToNear`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — silent failure
    }
}
``` [1](#0-0) 

The identical pattern appears in `withdrawToEthereum` (calling `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`): [2](#0-1) 

And in both functions of `EvmErc20V2.sol`: [3](#0-2) [4](#0-3) 

The two hardcoded precompile addresses are the Aurora-specific `ExitToNear` and `ExitToEthereum` precompiles: [5](#0-4) [6](#0-5) 

These precompiles can fail and return `0` from the EVM `call` opcode in several reachable conditions:

1. **Precompile is paused** — the engine supports pausing `EXIT_TO_NEAR` and `EXIT_TO_ETHEREUM` individually via `PrecompileFlags`: [7](#0-6) 
When paused, the precompile set returns `PrecompileFailure::Fatal`, which the EVM translates to a failed sub-call (return value `0`). [8](#0-7) 

2. **ERC-20 not registered in NEP-141 map** — `get_nep141_from_erc20` returns `ERR_TARGET_TOKEN_NOT_FOUND`, causing the precompile to return `ExitError`, which again yields `call` return value `0`: [9](#0-8) 

3. **ETH connector account not set in storage** — `get_eth_connector_contract_account` returns `ERR_KEY_NOT_FOUND`: [10](#0-9) 

In all cases, because `res` is never tested with `if iszero(res) { revert(0, 0) }`, the `_burn` is not rolled back and the user's tokens are gone.

---

### Impact Explanation

**Critical — Permanent freezing/destruction of funds.**

When the precompile call fails silently:
- The user's ERC-20 balance is reduced to zero (burned).
- No NEAR-side `ft_transfer` or `withdraw` promise is ever scheduled.
- The NEP-141 tokens that back the ERC-20 remain locked in the Aurora contract with no mechanism to reclaim them.
- There is no recovery path: the burn is irreversible and no refund logic exists in the Solidity contract.

---

### Likelihood Explanation

**Medium.** The most realistic trigger is the precompile being paused for maintenance while users continue to call `withdrawToNear`/`withdrawToEthereum`. Pausing is a documented, legitimate operational action (`PrecompileFlags::EXIT_TO_NEAR`, `EXIT_TO_ETHEREUM`). Any user who submits a withdrawal transaction during a pause window permanently loses their tokens. A secondary trigger is a misconfigured or unregistered ERC-20/NEP-141 mapping, which can occur during token deployment.

---

### Recommendation

After the `call` in each assembly block, check `res` and revert if it is zero. This ensures the `_burn` is rolled back on precompile failure:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to all four assembly blocks across `EvmErc20.sol` and `EvmErc20V2.sol` (both `withdrawToNear` and `withdrawToEthereum` in each file).

---

### Proof of Concept

1. Deploy `EvmErc20` for a NEP-141 token that is registered in the Aurora bridge.
2. Mint `1000` tokens to Alice via `mint(alice, 1000)`.
3. Admin pauses the `ExitToNear` precompile (legitimate maintenance).
4. Alice calls `withdrawToNear("alice.near", 1000)`.
5. `_burn(alice, 1000)` executes — Alice's balance is now `0`.
6. The assembly `call` to `0xe9217bc7...` returns `0` (precompile is paused → `PrecompileFailure::Fatal`).
7. `res` is never checked; the function returns without reverting.
8. Transaction succeeds. Alice has `0` ERC-20 tokens and receives `0` NEP-141 tokens on NEAR.
9. The `1000` units of value are permanently destroyed.

fn get_nep141_from_erc20<I: IO>(erc20_token: &[u8], io: &I) -> Result<AccountId, ExitError> {
    AccountId::try_from(
        io.read_storage(bytes_to_key(KeyPrefix::Erc20Nep141Map, erc20_token).as_slice())
            .map(|s| s.to_vec())
            .ok_or(ExitError::Other(Cow::Borrowed(ERR_TARGET_TOKEN_NOT_FOUND)))?,
    )
    .map_err(|_| ExitError::Other(Cow::Borrowed("ERR_INVALID_NEP141_ACCOUNT")))
}
