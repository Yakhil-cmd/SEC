Audit Report

## Title
Hot Keys Not Cleared When NNS Neuron Is Spawned to a New Controller - (File: rs/nns/governance/src/governance.rs)

## Summary
When `spawn_neuron` is called with a `new_controller` that differs from the parent neuron's controller, the child neuron unconditionally inherits the parent's full `hot_keys` list. Because hot key authorization is checked solely by membership in `hot_keys`, the original hot key holders retain the ability to vote, change followees, and join/leave the Neurons' Fund on the new controller's neuron without the new controller's knowledge or consent.

## Finding Description
In `rs/nns/governance/src/governance.rs`, `spawn_neuron` resolves `child_controller` from the optional `new_controller` field of the `Spawn` message (lines 2651ŌĆō2655). If `new_controller` is `Some(p)`, `child_controller` is set to a completely different principal from the parent's controller. The child neuron is then built at lines 2704ŌĆō2718:

```rust
let child_neuron = NeuronBuilder::new(
    child_nid, to_subaccount, child_controller, ...
)
.with_hot_keys(parent_neuron.hot_keys.clone())  // ŌåÉ unconditional copy
.with_followees(parent_neuron.followees.clone())
...
.build();
``` [1](#0-0) 

There is no conditional branch that clears or filters `hot_keys` when `child_controller != parent_neuron.controller()`. The child neuron is born with a controller chosen by the spawner **and** with all of the spawner's hot keys still attached.

Authorization for non-privileged operations is checked in `is_hotkey_or_controller`:

```rust
fn is_hotkey_or_controller(&self, principal: &PrincipalId) -> bool {
    self.is_controlled_by(principal) || self.hot_keys.contains(principal)
}
``` [2](#0-1) 

`is_authorized_to_vote` delegates directly to this function: [3](#0-2) 

`register_vote` accepts any caller for whom `is_authorized_to_vote` returns `true`: [4](#0-3) 

`JoinCommunityFund` and `LeaveCommunityFund` are explicitly permitted for hot keys in `is_authorized_to_configure_or_err`: [5](#0-4) 

No existing guard in `spawn_neuron` checks whether the new controller differs from the parent's controller before copying hot keys.

## Impact Explanation
This is an unauthorized access to governance assets impact. After a spawn-to-new-controller, any principal listed in the parent's `hot_keys` can, without the new controller's consent:

1. **Cast NNS votes** on behalf of the new controller's neuron, directly influencing governance outcomes.
2. **Change followees** for most governance topics on the new controller's neuron.
3. **Join the Neurons' Fund** on behalf of the new controller's neuron, committing the new controller's future maturity to the fund ŌĆö a financially consequential action that is not automatically reversible.

This matches the **High ($2,000ŌĆō$10,000)** impact class: *Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints.* The constraint is that the spawner must have previously added hot keys to the parent neuron and must spawn to a different controller, which is a realistic and common scenario (e.g., distributing maturity-derived ICP as a reward).

## Likelihood Explanation
`spawn_neuron` with a distinct `new_controller` is the primary on-chain mechanism for transferring maturity-derived ICP to another party. A spawner who has added hot keys for convenience (e.g., a mobile key or a team key) will inadvertently transfer those authorizations to the recipient's neuron. The recipient has no protocol-level notification of inherited hot keys and may never discover them. Deliberate exploitation requires only that the spawner retain a hot key on the parent neuron before spawning ŌĆö a zero-cost precondition. Likelihood of unintentional authorization leakage is **medium-to-high**; likelihood of deliberate exploitation is **low-to-medium**.

## Recommendation
In `spawn_neuron`, when `child_controller` differs from `parent_neuron.controller()`, create the child neuron with an empty hot key list instead of cloning the parent's:

```rust
let child_hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};
// ...
.with_hot_keys(child_hot_keys)
```

If hot key inheritance across controller boundaries is ever desired, it should be an explicit opt-in field in the `Spawn` message, not the default behavior.

## Proof of Concept
1. User A (controller of neuron N) adds hot key `H1` to neuron N via `manage_neuron ŌåÆ Configure ŌåÆ AddHotKey`.
2. User A calls `manage_neuron ŌåÆ Spawn { new_controller: Some(UserB), nonce: None, percentage_to_spawn: None }`.
3. Governance executes `spawn_neuron`, creating child neuron C with `controller = UserB` and `hot_keys = [H1]` (confirmed at lines 2707, 2714).
4. User A (via `H1`) calls `manage_neuron` on neuron C with `RegisterVote { proposal_id, vote: Yes }`. Governance accepts because `is_authorized_to_vote` returns `true` for `H1` (line 5594ŌĆō5595).
5. User A (via `H1`) calls `manage_neuron` on neuron C with `Configure ŌåÆ JoinCommunityFund {}`. Governance accepts because `is_authorized_to_configure_or_err` permits hot keys for `JoinCommunityFund` (lines 780ŌĆō790).
6. User B's neuron is now enrolled in the Neurons' Fund and has voted on a proposal ŌĆö both without User B's consent.
7. A deterministic integration test using `PocketIC` or the existing `state_machine_builder_for_nns_tests` framework (as used in `rs/nns/integration_tests/src/governance_neurons.rs`) can reproduce this in a local replica without touching mainnet.
