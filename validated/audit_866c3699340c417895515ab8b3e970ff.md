### Title
Unprivileged Caller Can Force Arbitrary Principal's ICP Participation in SNS Swap, Bypassing Confirmation Text Consent - (`rs/sns/swap/canister/canister.rs`)

### Summary

The SNS Swap canister's `refresh_buyer_tokens` endpoint accepts a `buyer` field in its request payload and uses it directly as the participating principal without verifying that the caller is that principal. Any unprivileged caller can specify any arbitrary principal as the `buyer`, forcing that principal's ICP (already deposited in the swap subaccount) to be registered as swap participation — including bypassing the SNS-configured confirmation text consent requirement. This is a direct analog to the STON.fi `cross_provide_lp` vulnerability: in both cases, a user-controlled address field is used as the "initiator/owner" of a multi-step financial operation without caller verification.

---

### Finding Description

**Root Cause**

In `rs/sns/swap/canister/canister.rs`, the `refresh_buyer_tokens` update method resolves the participating principal from the caller-supplied `arg.buyer` string field with no authorization check:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()  // ← attacker-controlled
    };
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
``` [1](#0-0) 

The proto definition explicitly documents this as intentional for the empty-string case ("If not specified, the caller is used"), but imposes no restriction when a non-empty `buyer` is provided: [2](#0-1) 

**What `refresh_buyer_token_e8s` does with the buyer principal**

The function queries the ICP ledger balance of `subaccount(swap_canister, buyer)` and records it as the buyer's participation:

```rust
let account = Account {
    owner: this_canister.get().0,
    subaccount: Some(principal_to_subaccount(&buyer)),  // ← victim's subaccount
};
icp_ledger.account_balance(account).await ...
``` [3](#0-2) 

Then it writes the balance into the swap's buyer state under the victim's principal:

```rust
self.buyers
    .entry(buyer.to_string())
    .or_insert_with(|| BuyerState::new(0))
    .set_amount_icp_e8s(new_balance_e8s);
``` [4](#0-3) 

**Confirmation text bypass**

The `confirmation_text` field is also passed through from the attacker's request. If an SNS configures a confirmation text (a legal/consent mechanism requiring the buyer to explicitly agree to swap terms), the attacker supplies the correct text (which is public, set at SNS initialization) on behalf of the victim: [5](#0-4) 

This means Alice's ICP is locked into the swap with the confirmation text "accepted" — without Alice ever having read or agreed to the terms.

---

### Impact Explanation

1. **Forced participation / ICP lock-in**: Alice deposits ICP into `subaccount(swap_canister, alice)` as a preparatory step but has not yet decided to commit. Eve calls `refresh_buyer_tokens { buyer: alice, confirmation_text: <correct_text> }`. Alice's ICP is now registered as swap participation and locked until finalization. If the swap commits, Alice receives SNS tokens she did not choose to acquire. If it aborts, she gets ICP back minus transfer fees — but her funds were locked without consent.

2. **Confirmation text consent bypass**: The confirmation text is a legal/consent gate. Any attacker who knows the text (it is public) can satisfy it on behalf of any buyer, rendering the mechanism ineffective for its intended purpose.

3. **Swap outcome manipulation**: An attacker can call `refresh_buyer_tokens` for multiple principals that have ICP in their subaccounts, forcing them to participate. This can:
   - Push the swap past the minimum-participants threshold, causing it to commit when it otherwise would abort
   - Exhaust the `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` limit, blocking legitimate new participants [6](#0-5) 

---

### Likelihood Explanation

- **No privilege required**: Any unprivileged ingress sender can call `refresh_buyer_tokens` with an arbitrary `buyer` field.
- **Precondition**: The victim must have already transferred ICP to `subaccount(swap_canister, victim_principal)` on the ICP ledger. This is a normal, observable on-chain action.
- **Confirmation text**: Publicly visible in the SNS initialization parameters; no secret knowledge required.
- **Effort**: A single ingress update call. Trivially automatable to target many victims simultaneously.

---

### Recommendation

**Short term**: In `refresh_buyer_tokens`, when `arg.buyer` is non-empty, assert that `PrincipalId::from_str(&arg.buyer).unwrap() == caller_principal_id()`. Only the buyer themselves (or an explicitly whitelisted delegate) should be permitted to register their own participation.

**Long term**: Treat `refresh_buyer_tokens` as a caller-authenticated operation. Remove the ability to specify an arbitrary `buyer` principal entirely, or introduce an explicit delegation/allowance mechanism (analogous to ICRC-2 `approve`/`transfer_from`) if third-party notification is a desired feature.

---

### Proof of Concept

**Setup**:
- SNS swap is in `Open` lifecycle with confirmation text `"I agree to participate"`.
- Alice (principal `alice`) transfers 10 ICP to `Account { owner: swap_canister, subaccount: principal_to_subaccount(alice) }` on the ICP ledger.
- Alice has not yet called `refresh_buyer_tokens`.

**Attack** (Eve, any unprivileged principal):
```
refresh_buyer_tokens({
    buyer: alice.to_string(),
    confirmation_text: Some("I agree to participate"),
})
```

**Result**:
- The swap canister queries the ICP ledger balance of Alice's subaccount, finds 10 ICP.
- Alice's participation is recorded: `swap.buyers["alice"] = BuyerState { amount_icp_e8s: 10_00_000_000 }`.
- Alice's ICP is locked in the swap.
- The confirmation text is marked as accepted for Alice — without Alice ever having called the endpoint.
- If the swap commits, Alice receives SNS tokens she did not choose to acquire at this time. [7](#0-6) [8](#0-7)

    pub async fn refresh_buyer_token_e8s(
        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

        // Look for the token balance of the specified principal's subaccount on 'this' canister.
        let e8s = {
            let account = Account {
                owner: this_canister.get().0,
                subaccount: Some(principal_to_subaccount(&buyer)),
            };
            icp_ledger
                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
        };
