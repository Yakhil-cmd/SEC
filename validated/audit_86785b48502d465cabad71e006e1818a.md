Audit Report

## Title
Unbounded Growth of Permanent Allowances in `ALLOWANCES_MEMORY` Stable Storage - (File: `rs/ledger_suite/common/ledger_core/src/approvals.rs`, `rs/ledger_suite/icp/ledger/src/lib.rs`, `rs/ledger_suite/icrc1/ledger/src/lib.rs`)

## Summary

The `AllowanceTable::approve()` function stores allowances with `expires_at: None` permanently in `ALLOWANCES_MEMORY` without inserting any entry into the expiration queue. The `prune()` function operates exclusively on the expiration queue and returns immediately when it is empty, making it structurally incapable of reclaiming permanent allowances. Because neither the ICP nor the ICRC-1 ledger enforces a per-account or global cap on stored allowances, an unprivileged caller can inflate `ALLOWANCES_MEMORY` without bound, degrading ledger performance and eventually exhausting canister stable memory.

## Finding Description

In `rs/ledger_suite/common/ledger_core/src/approvals.rs`, the `approve()` function at lines 265–275 only calls `insert_expiry` when `expires_at` is `Some`; when it is `None` the allowance is written to stable storage with no corresponding expiry record:

```rust
if let Some(expires_at) = expires_at {
    table.allowances_data.insert_expiry(expires_at, key.clone());
}
table.allowances_data.set_allowance(key, Allowance { amount, expires_at, arrived_at: now });
```

The `prune()` function at lines 372–399 iterates only the expiration queue via `first_expiry()`. When the queue is empty (all allowances are permanent) it returns `0` immediately at line 384 (`None => { return pruned; }`), so it can never remove a single permanent allowance regardless of how many exist.

Both ledgers back the allowance table with an unbounded `StableBTreeMap`:

- ICP ledger (`rs/ledger_suite/icp/ledger/src/lib.rs`, lines 136–137): `StableBTreeMap<(AccountIdentifier, AccountIdentifier), StorableAllowance, VirtualMemory<DefaultMemoryImpl>>` with no size cap.
- ICRC-1 ledger (`rs/ledger_suite/icrc1/ledger/src/lib.rs`, lines 526–527): `StableBTreeMap<AccountSpender, StorableAllowance, VirtualMemory<DefaultMemoryImpl>>` with no size cap.

The only allowance-count constant in the codebase, `MAX_TAKE_ALLOWANCES = 500` (`rs/ledger_suite/icrc1/ledger/src/lib.rs`, line 74), limits query response size only; it imposes no write-time constraint. No per-account or global insertion cap exists anywhere in the ledger suite.

Exploit path:
1. Attacker acquires a small token balance across many subaccounts of a single principal (2^256 subaccounts are available).
2. For each subaccount, attacker calls `icrc2_approve` with `expires_at: None` and a distinct spender.
3. Each call succeeds, inserting a permanent entry into `ALLOWANCES_MEMORY` with no expiry record.
4. Attacker transfers the token balance back out, leaving zero-balance accounts with permanent allowances.
5. `prune()` is called on every transaction but returns `0` immediately because the expiration queue is empty.
6. `ALLOWANCES_MEMORY` grows monotonically with no automatic reclamation path.

## Impact Explanation

This matches the allowed ICP bounty impact: **Application/platform-level DoS not based on raw volumetric DDoS** (High, $2,000–$10,000). The ICP and ICRC-1 ledgers are critical financial infrastructure explicitly listed in scope. As `ALLOWANCES_MEMORY` grows, every stable-memory operation (transfers, approvals, burns) incurs increasing BTree traversal cost, degrading throughput for all users. At sufficient scale, stable memory exhaustion halts the ledger entirely, preventing all transactions. This is a protocol-level resource accounting failure, not a volumetric network attack.

## Likelihood Explanation

Medium. Each allowance entry costs one approval fee (10,000 e8s for ICP). The attacker can reclaim the token principal by transferring it out after approval, so the net cost is fees only. Creating 10 million entries costs approximately 1,000 ICP (~$5,000 at current prices) and consumes several gigabytes of stable memory, causing measurable performance degradation. Full stable-memory exhaustion (400 GiB) would require hundreds of millions of ICP and is economically prohibitive, but significant degradation is achievable at realistic cost. The attack is silent, gradual, and requires no special privileges — only the ability to call `icrc2_approve`.

## Recommendation

1. **Enforce a per-account cap**: Reject `icrc2_approve` if the approver already has N active allowances (e.g., N = 100–1,000). This directly bounds the attack surface.
2. **Require or incentivize expiration**: Charge a higher fee for permanent allowances, or require `expires_at` to be set, making the attack economically prohibitive.
3. **Add a global allowance count limit**: Introduce a configurable `MAX_ALLOWANCES` constant analogous to `MAX_TRANSACTIONS_IN_WINDOW`, rejecting new allowances when the global count is reached.
4. **Implement balance-based cleanup**: Add a callable function that removes allowances whose approver account has a zero balance, since a zero-balance account cannot pay fees and is unlikely to need active allowances.

## Proof of Concept

A deterministic PocketIC integration test can prove this:

1. Deploy the ICP or ICRC-1 ledger canister locally.
2. Create principal `P` with subaccounts `sub_0 … sub_N`; mint the approval fee to each.
3. For each `sub_i`, call `icrc2_approve({ spender: S_i, amount: 1, expires_at: None })`.
4. Assert each call returns `Ok`.
5. Call `prune(now, 1_000_000)` and assert it returns `0`.
6. Query `ALLOWANCES_MEMORY` length and assert it equals `N`.
7. Transfer the token balance out of each subaccount.
8. Call `prune(now, 1_000_000)` again and assert it still returns `0`.
9. Assert `ALLOWANCES_MEMORY` length is still `N` — confirming permanent, irremovable growth.

This test requires no mainnet interaction and is fully reproducible in a local replica or PocketIC environment.
