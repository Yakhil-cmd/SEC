### Title
Permanent ERC-20 Token Loss When `ft_transfer` Promise Fails Without `error_refund` Feature — (File: `engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is not enabled, the `ExitToNear` precompile creates `ft_transfer` NEAR promises without a refund callback. If the `ft_transfer` promise fails for any reason — including attacker-induced failure — the user's ERC-20 tokens are permanently burned on Aurora with no corresponding NEP-141 tokens received on NEAR and no recovery path.

---

### Finding Description

The `ExitToNear::run` function in `engine-precompiles/src/native.rs` implements the bridge exit flow:

1. ERC-20 tokens are burned on Aurora (irreversible EVM state change committed before the promise is dispatched).
2. A NEAR promise is created to call `ft_transfer` on the NEP-141 contract.
3. **Only when `error_refund` is compiled in** is a callback (`exit_to_near_precompile_callback`) attached to handle failures and re-mint burned tokens.

When `error_refund` is **not** enabled, the `refund` field is hardcoded to `None`: [1](#0-0) 

Because `callback_args` then equals `ExitToNearPrecompileCallbackArgs::default()`, the branch at line 470 selects `PromiseArgs::Create(transfer_promise)` — a bare promise with **no callback**: [2](#0-1)

When `error_refund` is absent, `args.refund` is always `None`, so `refund_on_error` is never called and the burned tokens are never re-minted.

The `refund_on_error` function itself (which would re-mint ERC-20 tokens or return ETH) is only reachable through the callback path: [4](#0-3)

### Impact Explanation

**Critical — Permanent freezing of funds.**

The user's ERC-20 tokens are burned on Aurora (EVM state is committed) but no NEP-141 tokens arrive on NEAR. There is no recovery mechanism. The tokens are permanently destroyed. This matches the "Permanent freezing of funds" impact class.

---

### Likelihood Explanation

The `ft_transfer` promise can fail through multiple realistic paths:

1. **User error**: Recipient NEAR account not registered with the NEP-141 contract — the most common failure mode, confirmed by the existing test `test_exit_to_near_refund`.
2. **Attacker-induced**: An attacker monitoring the mempool can frontrun a user's `exit_to_near` transaction and drain the Aurora contract's NEP-141 balance (by transferring their own bridged tokens out), causing the user's `ft_transfer` to fail due to insufficient balance.
3. **NEP-141 contract state**: Contract paused, storage exhausted, or any other condition causing `ft_transfer` to revert.

The existing test suite explicitly documents the permanent loss: [6](#0-5) 

The same pattern is confirmed for ETH exits: [7](#0-6) 

The `error_refund` feature is opt-in (Rust features are disabled by default), meaning any production deployment that does not explicitly enable it is vulnerable.

---

### Recommendation

1. Make `error_refund` a **default feature** in `engine/Cargo.toml` and `engine-precompiles/Cargo.toml` so it is always active unless explicitly opted out.
2. Alternatively, remove the feature flag entirely and always attach the refund callback whenever ERC-20 tokens are burned — the callback is a no-op on success, so there is no correctness cost.
3. Add a compile-time or runtime assertion that prevents deployment without the refund path.

---

### Proof of Concept

**Setup**: Deploy Aurora Engine compiled **without** the `error_refund` feature.

**Steps**:
1. Bridge NEP-141 tokens into Aurora via `ft_on_transfer` → receive ERC-20 tokens.
2. Call `exit_to_near` (via the ERC-20 contract's `withdrawToNear`) specifying a recipient NEAR account that is **not registered** with the NEP-141 contract.
3. The EVM transaction succeeds: ERC-20 tokens are burned, a bare `ft_transfer` promise is dispatched (no callback).
4. The `ft_transfer` NEAR promise fails: recipient not registered.
5. No `exit_to_near_precompile_callback` is scheduled (because `callback_args == default()`).
6. **Result**: ERC-20 tokens are permanently burned; user receives nothing; no refund is possible.

**Attacker variant**: Replace step 2 with a valid registered recipient. Before the user's `ft_transfer` promise executes, the attacker (who holds bridged tokens in the same NEP-141) calls `ft_transfer` to drain the Aurora contract's NEP-141 balance below the user's exit amount. The user's `ft_transfer` then fails due to insufficient balance, with the same permanent loss outcome.

The root cause is the conditional compilation at: [1](#0-0) 

and the absence of a callback when `callback_args` is default: [8](#0-7)
