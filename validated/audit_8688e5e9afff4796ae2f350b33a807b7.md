Audit Report

## Title
Node Provider Monthly Reward Fully Rejected Instead of Capped When Exceeding `maximum_node_provider_rewards_e8s` - (File: rs/nns/governance/src/governance.rs)

## Summary
In the NNS Governance canister, `reward_node_provider_helper` returns a hard `Err(PreconditionFailed)` when a node provider's computed monthly ICP reward exceeds `maximum_node_provider_rewards_e8s`, causing the node provider to receive zero ICP for that month instead of the capped maximum. The reward is permanently lost with no retry or rollover mechanism, and the failure is only logged internally.

## Finding Description
The automated monthly reward flow has two inconsistent stages:

**Stage 1 — Calculation (no cap):** `get_node_provider_reward` at `rs/nns/governance/src/governance.rs:8254–8255` computes `amount_e8s` purely from XDR metrics and the ICP/XDR conversion rate, with no reference to `maximum_node_provider_rewards_e8s`.

**Stage 2 — Distribution (hard reject):** `reward_node_provider_helper` at `rs/nns/governance/src/governance.rs:3953–3960` checks `reward.amount_e8s > maximum_node_provider_rewards_e8s` and returns `Err(GovernanceError::new_with_message(ErrorType::PreconditionFailed, ...))`. No ICP is minted.

**Automated path:** `reward_node_providers` at `rs/nns/governance/src/governance.rs:3993–4002` iterates over all computed rewards, calls `reward_node_provider_helper` for each, logs any error via `println!`, and continues. The affected node provider receives nothing. `reward_node_providers_from_proposal` at `rs/nns/governance/src/governance.rs:4014–4015` invokes `mint_monthly_node_provider_rewards` for registry-derived rewards, which feeds into this same path.

The mainnet value of `maximum_node_provider_rewards_e8s` is 100,000 ICP (`rs/nns/governance/api/src/lib.rs:105`). The field is documented as a cap ("maximum rewards to be distributed to NodeProviders in a single distribution event"), yet the implementation treats it as a rejection threshold rather than a ceiling.

## Impact Explanation
A node provider whose calculated monthly reward exceeds the cap receives **zero ICP** instead of the capped 100,000 ICP. The reward is permanently lost — there is no retry, rollover, or notification to the node provider. This constitutes a permanent, silent loss of ICP that should have been minted and distributed, fitting the "Significant NNS security impact with concrete user or protocol harm" category at **High** severity. At current ICP prices, 100,000 ICP represents substantial financial harm per affected provider per month.

## Likelihood Explanation
The mainnet cap is 100,000 ICP. The trigger condition requires a node provider whose XDR reward, converted at the effective ICP/XDR rate (floored at `minimum_icp_xdr_rate`), exceeds 100,000 ICP. At the minimum rate floor (1 XDR ≈ 1 ICP), a provider operating ~5,001+ nodes at ~20 XDR/node/month crosses the threshold. While no single provider currently operates at this scale, the condition becomes reachable if: (a) the network grows and node provider concentration increases, (b) ICP price falls to the minimum floor, or (c) NNS governance reduces `maximum_node_provider_rewards_e8s`. The bug is triggered automatically by the monthly reward process — no attacker action is required; the affected node provider need only exist and have a sufficiently large node fleet.

## Recommendation
In `reward_node_provider_helper` (`rs/nns/governance/src/governance.rs`), replace the hard rejection with a cap:

```rust
let amount_e8s = reward.amount_e8s.min(maximum_node_provider_rewards_e8s);
// construct a capped RewardNodeProvider and proceed with mint_reward_to_neuron_or_account
```

Alternatively, apply the cap inside `get_node_provider_reward` so rewards are bounded before reaching the distribution stage, ensuring consistency between the calculation and distribution phases.

## Proof of Concept
1. Deploy NNS governance with `maximum_node_provider_rewards_e8s = 100_000 * E8` (mainnet value).
2. Register a node provider with enough nodes such that their XDR reward computes to `> 100_000 * E8` ICP (e.g., 5,001 nodes × 20 XDR/month = 100,020 XDR at the minimum ICP/XDR floor).
3. Trigger the monthly reward event (via a `RewardNodeProviders` proposal with `use_registry_derived_rewards = true`, or wait for the automated timer).
4. `get_node_provider_reward` returns `amount_e8s = 100_020 * E8`.
5. `reward_node_provider_helper` evaluates `100_020 * E8 > 100_000 * E8 → true` → returns `Err(PreconditionFailed)`.
6. `reward_node_providers` logs the error and continues; the node provider's ICP balance is unchanged (receives 0 ICP).
7. Verify via ledger query that no mint transaction was issued for the node provider's account.
8. A deterministic integration test using PocketIC can reproduce this by setting `minimum_icp_xdr_rate` to its floor, registering a large node provider, and asserting the ledger balance remains zero after the reward event.
