### Title
Unprotected Legacy Transaction Signature Reuse Across Chains Enables Cross-Chain Replay Attack - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

### Summary

ZKsync OS accepts pre-EIP-155 "unprotected" legacy transactions (where `v == 27` or `v == 28`) whose signing hash contains **no chain ID**. Because the signing hash is chain-agnostic, a signature produced for Ethereum mainnet (or any other EVM chain) is cryptographically identical to a valid signature on ZKsync OS. An unprivileged attacker who observes a victim's unprotected legacy transaction on any EVM chain can replay it verbatim on ZKsync OS, executing the same call under the victim's address and draining their ZKsync OS funds.

### Finding Description

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, the code branches on whether the signature is EIP-155-protected:

```rust
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    // Unprotected legacy — hash covers only [nonce, gasPrice, gasLimit, to, value, data]
    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);
    hasher.finalize_reset().into()
} else {
    // EIP-155 protected — chain_id is mixed into the hash
    ...
}
``` [1](#0-0) 

`is_eip155()` returns `false` exactly when `v == 27 || v == 28`: [2](#0-1) 

When the branch is taken, the signing hash is `keccak256(rlp([nonce, gasPrice, gasLimit, to, value, data]))` — identical to the hash that Ethereum mainnet (or any other EVM chain) would verify for the same transaction. The parser then stores the result as `RlpEncodedTxInner::Legacy`, which is fully accepted and executed: [3](#0-2) 

The `chain_id()` accessor confirms that `Legacy` transactions carry no chain binding: [4](#0-3) 

Signature validation in both the Ethereum-like and ZK transaction flows then runs `secp256k1_ec_recover` against this chain-agnostic hash and accepts the transaction if the recovered address matches `from`: [5](#0-4) [6](#0-5) 

### Impact Explanation

An attacker who observes a victim's unprotected legacy transaction on Ethereum mainnet (or any EVM chain) can submit the identical raw bytes to ZKsync OS. If the victim's ZKsync OS account nonce matches the nonce in the replayed transaction, the transaction executes under the victim's address. This can:

- Transfer ETH/tokens out of the victim's ZKsync OS account.
- Execute arbitrary calldata (e.g., approve a spender, call a DeFi contract) on behalf of the victim.

The impact is **direct, irreversible loss of user funds** on ZKsync OS.

### Likelihood Explanation

- Pre-EIP-155 transactions are still generated by some hardware wallets, older tooling, and certain DeFi protocols.
- A victim whose ZKsync OS account nonce is 0 (a new account) is immediately vulnerable to replay of any unprotected nonce-0 transaction they have ever signed on any EVM chain.
- The attacker needs only to monitor public mempools or on-chain history for unprotected legacy transactions — no privileged access is required.
- Likelihood is **medium**: the nonce alignment requirement limits the attack surface, but new ZKsync OS users are directly exposed.

### Recommendation

Reject unprotected legacy transactions at parse time. In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, return `Err(InvalidTransaction::InvalidChainId)` (or a new `MissingChainId` variant) when `legacy_signature.is_eip155() == false`. Only EIP-155-protected legacy transactions (and typed EIP-2718 transactions, which already carry a mandatory `chain_id` field) should be accepted. [7](#0-6) 

### Proof of Concept

1. On Ethereum mainnet, Alice signs and broadcasts an unprotected legacy transaction (v=27 or v=28) with nonce=0, sending 1 ETH to Bob.
2. Alice later creates a ZKsync OS account (nonce starts at 0) and deposits funds.
3. Attacker copies the raw RLP bytes of Alice's Ethereum transaction verbatim.
4. Attacker submits those bytes to ZKsync OS as a new transaction.
5. ZKsync OS parses the transaction, computes `keccak256(rlp([0, gasPrice, gasLimit, Bob, 1 ETH, ""]))` — the same hash Ethereum computed — and recovers Alice's address.
6. Nonce check passes (Alice's ZKsync OS nonce is 0).
7. The transaction executes: 1 ETH is transferred from Alice's ZKsync OS balance to Bob on ZKsync OS.

The root cause is the absence of chain ID in the signing hash for `RlpEncodedTxInner::Legacy` transactions, directly analogous to the Connext finding where the same signature bytes were valid in two distinct operational contexts.
