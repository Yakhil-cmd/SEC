### Title
DoS via `update_balance` Guard Slot Exhaustion in ckBTC Minter - (File: `rs/bitcoin/ckbtc/minter/src/guard.rs`)

### Summary
The ckBTC minter enforces a hard cap of `MAX_CONCURRENT = 100` concurrent `update_balance` calls via a per-account `Guard<PendingBalanceUpdates>`. An unprivileged attacker can fill all 100 slots by submitting concurrent `update_balance` calls from 100 distinct accounts (e.g., different subaccounts of the same principal). While those calls are in-flight awaiting the Bitcoin canister's `get_utxos` response, every new `update_balance` call from a legitimate user is rejected with `TooManyConcurrentRequests`. The attacker can sustain the blockade by re-submitting calls as old ones complete, at the cost of cycles only — no BTC or ckBTC is required.

### Finding Description
`Guard<PendingBalanceUpdates>::new` inserts the caller's `Account` into `state.update_balance_accounts` and rejects any new call once the set reaches `MAX_CONCURRENT = 100`:

```rust
const MAX_CONCURRENT: usize = 100;
// ...
if accounts.len() >= MAX_CONCURRENT {
    return Err(GuardError::TooManyConcurrentRequests);
}
accounts.insert(account);
``` [1](#0-0) 

The guard is released only when the `Guard` object is dropped (i.e., when `update_balance` returns). Between the guard acquisition and the return, the function awaits `get_utxos` from the Bitcoin canister:

```rust
let _guard = balance_update_guard(caller_account)?;
// ...
let utxos = get_utxos(btc_network, &address, min_confirmations, CallSource::Client, runtime).await?;
``` [2](#0-1) 

The `update_balance` endpoint accepts an optional `args.owner` field, allowing the caller to specify any principal as the account owner. This means a single attacker principal can trivially generate 100 distinct `Account` values (e.g., by varying `subaccount`) and submit 100 concurrent calls, each holding a guard slot for the duration of the Bitcoin canister round-trip. [3](#0-2) 

The `Drop` implementation removes the account from the set on normal completion, but the guard is held for the entire async span. There is no timeout or expiry on guard slots. [4](#0-3) 

### Impact Explanation
`update_balance` is the sole mechanism by which users mint ckBTC after depositing BTC on-chain. Blocking it prevents any new ckBTC from being minted for the duration of the attack. Legitimate depositors receive `TemporarilyUnavailable` / `TooManyConcurrentRequests` errors and cannot advance their UTXO state. The attacker can sustain the blockade indefinitely by cycling 100 concurrent calls, paying only IC cycles — no BTC or ckBTC capital is required.

### Likelihood Explanation
The attack is reachable by any unprivileged ingress sender. The attacker needs only:
1. One principal with 100 distinct subaccounts (or 100 different principals).
2. Enough cycles to sustain 100 concurrent canister calls.
3. No BTC, no ckBTC, no governance power.

The Bitcoin canister's `get_utxos` round-trip takes several seconds, giving the attacker a comfortable window to re-fill slots before they drain. The attack is cheap to sustain and requires no special access.

### Recommendation
- **Short term**: Add a per-principal rate limit or a minimum deposit threshold check before acquiring the guard, so that accounts with no UTXOs are rejected cheaply without holding a slot.
- **Long term**: Introduce a guard-slot timeout so that slots held for longer than a configurable threshold are automatically released, preventing indefinite slot exhaustion. Alternatively, increase `MAX_CONCURRENT` and add per-subnet-principal quotas to raise the cost of exhaustion.

### Proof of Concept
1. Attacker generates 100 accounts: `Account { owner: attacker_principal, subaccount: Some([i; 32]) }` for `i` in `0..100`.
2. Attacker sends 100 concurrent `update_balance` calls, one per account, each with `args.owner = Some(attacker_principal)` and `args.subaccount = Some([i; 32])`.
3. Each call acquires a guard slot and then suspends at `get_utxos(...).await`.
4. `update_balance_accounts.len()` reaches 100.
5. Any new `update_balance` call from a legitimate user hits `accounts.len() >= MAX_CONCURRENT` and returns `GuardError::TooManyConcurrentRequests` → `UpdateBalanceError::TemporarilyUnavailable`.
6. As old calls complete and slots free up, the attacker immediately re-submits to refill them, sustaining the blockade at the cost of cycles only. [5](#0-4) [6](#0-5)

const MAX_CONCURRENT: usize = 100;

#[derive(Eq, PartialEq, Debug)]
pub enum GuardError {
    AlreadyProcessing,
    TooManyConcurrentRequests,
}

pub trait PendingRequests {
    fn pending_requests(state: &mut CkBtcMinterState) -> &mut BTreeSet<Account>;
}

pub struct PendingBalanceUpdates;

impl PendingRequests for PendingBalanceUpdates {
    fn pending_requests(state: &mut CkBtcMinterState) -> &mut BTreeSet<Account> {
        &mut state.update_balance_accounts
    }
}
pub struct RetrieveBtcUpdates;

impl PendingRequests for RetrieveBtcUpdates {
    fn pending_requests(state: &mut CkBtcMinterState) -> &mut BTreeSet<Account> {
        &mut state.retrieve_btc_accounts
    }
}

/// Guards a block from executing twice when called by the same user and from being
/// executed [MAX_CONCURRENT] or more times in parallel.
#[must_use]
pub struct Guard<PR: PendingRequests> {
    account: Account,
    _marker: PhantomData<PR>,
}

impl<PR: PendingRequests> Guard<PR> {
    /// Attempts to create a new guard for the current block. Fails if there is
    /// already a pending request for the specified [principal] or if there
    /// are at least [MAX_CONCURRENT] pending requests.
    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }
}

impl<PR: PendingRequests> Drop for Guard<PR> {
    fn drop(&mut self) {
        mutate_state(|s| PR::pending_requests(s).remove(&self.account));
    }
}

pub async fn update_balance<R: CanisterRuntime>(
    args: UpdateBalanceArgs,
    runtime: &R,
) -> Result<Vec<UtxoStatus>, UpdateBalanceError> {
    let caller = runtime.caller();
    if args.owner.unwrap_or(caller) == runtime.id() {
        ic_cdk::trap("cannot update minter's balance");
    }

    // Record start time of method execution for metrics
    let start_time = runtime.time();

    // When the minter is in the mode using a whitelist we only want a certain
    // set of principal to be able to mint. But we also want those principals
    // to mint at any desired address. Therefore, the check below is on "caller".
    state::read_state(|s| s.mode.is_deposit_available_for(&caller))
        .map_err(UpdateBalanceError::TemporarilyUnavailable)?;

    init_ecdsa_public_key().await;

    let caller_account = Account {
        owner: args.owner.unwrap_or(caller),
        subaccount: args.subaccount,
    };
    let _guard = balance_update_guard(caller_account)?;

    let address = state::read_state(|s| runtime.derive_user_address(s, &caller_account));

    let (btc_network, min_confirmations) =
        state::read_state(|s| (s.btc_network, s.min_confirmations));

    let utxos = get_utxos(
        btc_network,
        &address,
        min_confirmations,
        CallSource::Client,
        runtime,
    )
    .await?
    .utxos;
