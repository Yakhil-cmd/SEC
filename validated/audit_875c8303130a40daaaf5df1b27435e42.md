### Title
No Minimum Quorum on `upgrade_delay_blocks` Allows Immediate Time-Lock Bypass - (File: engine/src/contract_methods/admin.rs)

### Summary
`set_upgrade_delay_blocks` accepts any `u64` value — including zero — with no lower-bound validation. Because `upgrade_delay_blocks` is the sole time-lock protecting the two-step contract upgrade process, setting it to `0` collapses the delay to zero blocks, allowing the owner to stage and deploy an arbitrary new WASM contract in the same block, eliminating the intended user-protection window.

### Finding Description
`set_upgrade_delay_blocks` reads the caller-supplied value and writes it directly into `EngineState` without any minimum-value check:

```rust
// engine/src/contract_methods/admin.rs L134-145
pub fn set_upgrade_delay_blocks<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        let args: SetUpgradeDelayBlocksArgs = io.read_input_borsh()?;
        state.upgrade_delay_blocks = args.upgrade_delay_blocks;   // ← no minimum check
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
```

`stage_upgrade` then computes the earliest deploy block as:

```rust
// engine/src/contract_methods/admin.rs L158
let delay_block_height = env.block_height() + state.upgrade_delay_blocks;
```

When `upgrade_delay_blocks == 0`, `delay_block_height == env.block_height()`, so `deploy_upgrade` can be called in the very same block as `stage_upgrade`, making the time-lock completely ineffective.

`upgrade_delay_blocks` is stored in `EngineState` and persists across all subsequent upgrades:

```rust
// engine/src/state.rs L26
pub upgrade_delay_blocks: u64,
```

### Impact Explanation
The `upgrade_delay_blocks` time-lock is the only mechanism that gives users and observers a window to detect and react to a malicious or erroneous contract upgrade. With the delay set to `0`, the owner can atomically stage and deploy a replacement WASM binary that rewrites balance accounting, mints arbitrary ETH, or redirects bridge withdrawals — resulting in direct theft of all user funds held in the Aurora Engine (ETH balances, ERC-20 mirror balances, in-flight bridge deposits). This maps to **Critical: direct theft of user funds**.

### Likelihood Explanation
The call path is a single privileged transaction (`set_upgrade_delay_blocks` with `upgrade_delay_blocks: 0`) followed immediately by `stage_upgrade` + `deploy_upgrade`. No external dependency, no oracle, no multi-step coordination is required beyond owner key access. The absence of any on-chain guard means the misconfiguration can be introduced silently and exploited in the same block. Likelihood is **medium** (mirrors the external report's score of 3): the owner key must be involved, but no additional preconditions exist once the key is available.

### Recommendation
Add a minimum-value guard inside `set_upgrade_delay_blocks` (and enforce the same minimum during `new` initialization). For example:

```rust
const MIN_UPGRADE_DELAY_BLOCKS: u64 = 1; // or a protocol-defined minimum

if args.upgrade_delay_blocks < MIN_UPGRADE_DELAY_BLOCKS {
    return Err(errors::ERR_INVALID_UPGRADE_DELAY.into());
}
state.upgrade_delay_blocks = args.upgrade_delay_blocks;
```

The same check should be applied in `NewCallArgs` deserialization so the invariant holds from contract initialization.

### Proof of Concept
1. Owner calls `set_upgrade_delay_blocks` with `upgrade_delay_blocks: 0`. No error is returned.
2. In the same block, owner calls `stage_upgrade` with malicious WASM. `delay_block_height = block_height + 0 = block_height` is stored.
3. Owner immediately calls `deploy_upgrade`. The block-height check passes (`block_height >= block_height`), and the malicious contract is deployed atomically.
4. The malicious contract's `state_migration` can rewrite all EVM balances or redirect the ETH connector, draining user funds with no advance warning.

pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let delay_block_height = env.block_height() + state.upgrade_delay_blocks;
        require_owner_only(&state, &env.predecessor_account_id())?;
        io.read_input_and_store(&storage::bytes_to_key(KeyPrefix::Config, CODE_KEY));

            &delay_block_height.to_le_bytes(),
        );
        Ok(())
    })
}
