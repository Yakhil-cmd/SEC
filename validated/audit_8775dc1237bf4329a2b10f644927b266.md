Audit Report

## Title
Permanent Slot Exhaustion via Unauthenticated `canister_status` on Non-Controlled Canister — (`rs/nns/handlers/root/impl/canister/canister.rs`)

## Summary

The public `canister_status` endpoint on NNS Root has no caller authentication and calls `.unwrap()` on the management canister response, causing a Wasm trap on any rejection. Because Wasm traps do not unwind the Rust stack, the `SlotLoan` RAII guard that restores `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` is never dropped. The slot decrement is committed to canister state during the pre-await phase and is not rolled back when the callback traps. An unprivileged attacker can permanently exhaust all 167 slots with 167 sequential update calls, blocking all management-canister operations routed through NNS Root — including those from NNS Governance.

## Finding Description

**Entry point:** `canister_status` at `rs/nns/handlers/root/impl/canister/canister.rs` lines 88–98 is a `#[update]` endpoint with no caller authentication check. [1](#0-0) 

**Slot accounting:** `new_management_canister_client()` wraps the real client in `LimitedOutstandingCallsManagementCanisterClient`, which calls `try_borrow_slot()` before forwarding the call. `try_borrow_slot()` decrements `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` (initialized to 167) and returns a `SlotLoan` RAII guard. [2](#0-1) [3](#0-2) 

**Slot decrement before await:** In `LimitedOutstandingCallsManagementCanisterClient::canister_status`, `_loan` is created (decrement committed) before the `.await` suspends the future. The IC commits canister state when an inter-canister call is sent, so the decrement survives the await boundary. [4](#0-3) 

**Trap on error:** Back in the top-level `canister_status`, `.unwrap()` is called unconditionally on the result. When the management canister rejects (e.g., NNS Root is not a controller of the target canister), this panics, producing a Wasm trap. [5](#0-4) 

**RAII guard never dropped:** Wasm traps are abrupt termination — Rust destructors are not called. `SlotLoan::drop()`, which would restore the slot count, is never invoked. [6](#0-5) 

**IC rollback does not help:** The IC rolls back state changes made *during the callback* execution, but the slot decrement occurred in the pre-await phase (already committed). It is not part of the callback's state delta and is not rolled back.

**VIP callers also blocked:** When `available_slot_count == 0`, `try_borrow_slot()` returns `Err` for all callers unconditionally — the VIP path only sets `used_slot_count = 0` (affecting the subtraction), but the `== 0` guard fires before any subtraction, blocking NNS Governance equally. [7](#0-6) 

**Acknowledged design:** The codebase explicitly documents that NNS Root's `canister_status` uses trapping as its error-propagation mechanism, confirming this is the actual production behavior. [8](#0-7) 

## Impact Explanation

Once all 167 slots are exhausted, every call to `try_borrow_slot()` returns `Err((SysTransient, "Unavailable. Maybe, try again later?"))`. This blocks all management-canister operations routed through NNS Root: canister upgrades, settings changes, stop/start, snapshot operations, etc. NNS Governance, which relies on NNS Root for these operations, is fully blocked. The slot count cannot self-recover because the only recovery path (`SlotLoan::drop()`) requires a non-trapping callback, which is impossible when the count is 0. Recovery requires an NNS Root canister upgrade (via an NNS proposal), introducing a governance-latency window during which NNS-managed canister operations are unavailable.

This matches the **High ($2,000–$10,000)** impact class: *Application/platform-level DoS... or subnet availability impact not based on raw volumetric DDoS*, specifically targeting NNS governance infrastructure.

## Likelihood Explanation

- No authentication, no special privileges required — any principal including anonymous can call `canister_status`.
- Only 167 sequential update calls are needed; no concurrency or coordination required.
- The management canister deterministically rejects `canister_status` for any canister NNS Root does not control, making the trap reliable and repeatable.
- Normal update-call fees are the only cost.

## Recommendation

Change `canister_status` to return `Result<CanisterStatusResult, String>` and propagate errors without trapping:

```rust
#[update]
async fn canister_status(canister_id_record: CanisterIdRecord) -> Result<CanisterStatusResult, String> {
    let client = new_management_canister_client();
    client
        .canister_status(canister_id_record)
        .await
        .map(CanisterStatusResult::from)
        .map_err(|(_, msg)| msg)
}
```

This ensures `SlotLoan` is always dropped normally (restoring the slot count) regardless of whether the management canister accepts or rejects the call. Callers currently relying on the trap-as-error behavior (as documented in `client.rs`) would need to be updated to handle the `Result` return type.

## Proof of Concept

State-machine or PocketIC integration test:

1. Deploy NNS with the Root canister.
2. Select any canister ID not controlled by Root (e.g., `CanisterId::from_u64(9999)`).
3. From the anonymous principal, call `canister_status` on Root 167 times with that canister ID. Each call should return a replica-level error (trap/reject).
4. Verify `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT == 0` (observable via the `/metrics` HTTP endpoint or by attempting a 168th call and observing the `SysTransient / "Unavailable"` rejection).
5. Have NNS Governance attempt any management-canister operation through Root (e.g., `change_nns_canister`). Assert it fails with `SysTransient / "Unavailable. Maybe, try again later?"`.

    fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
        let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

        self.available_slot_count
            .with_borrow_mut(|available_slot_count| {
                if *available_slot_count == 0 {
                    // This is somewhat of a lie, but is the best fit.
                    let code = RejectCode::SysTransient as i32;

                    let message = "Unavailable. Maybe, try again later?".to_string();

                    return Err((code, message));
                }

                *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
                Ok(())
            })?;

        let available_slot_count = self.available_slot_count;
        Ok(SlotLoan {
            available_slot_count,
            used_slot_count,
        })
    }
