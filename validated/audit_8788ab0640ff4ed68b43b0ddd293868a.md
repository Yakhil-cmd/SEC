### Title
Improper ETH Connector Migration in `set_eth_connector_contract_account` Leaves User Funds Frozen - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

`set_eth_connector_contract_account` updates the engine's stored ETH connector account ID without verifying or migrating the ETH balances held by the old connector. After the swap, every connector-facing operation (`withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_deposit`, etc.) is routed to the new connector via `return_promise`, which reads the freshly-written account ID. Because the new connector holds no ETH, all user withdrawals and NEP-141 transfers fail until the owner manually intervenes.

---

### Finding Description

`set_eth_connector_contract_account` is the Aurora Engine's analog of `setGauge()` in the reference report. It is the single function that controls which external NEAR contract the engine treats as the authoritative ETH connector. [1](#0-0) 

The function accepts a `SetEthConnectorContractAccountArgs` struct, writes the new account ID and serialization type to storage, and returns. There is no check on the old connector's outstanding balance, no cross-contract call to drain or migrate funds, and no revert if the old connector still holds ETH.

Every connector-facing user operation ultimately calls the private helper `return_promise`, which resolves the target account dynamically at call time: [2](#0-1) 

`get_connector_account_id` reads the storage key written by `set_eth_connector_contract_account`. The moment the key is overwritten, **all** of the following methods silently redirect to the new, empty connector:

- `withdraw` → `engine_withdraw`
- `ft_transfer` → `engine_ft_transfer`
- `ft_transfer_call` → `engine_ft_transfer_call`
- `storage_deposit` / `storage_withdraw` / `storage_unregister`
- `ft_metadata` [3](#0-2) 

Additionally, `ft_on_transfer` uses the stored connector account ID to decide whether an incoming NEP-141 transfer represents a base-ETH deposit or an ERC-20 deposit: [4](#0-3) 

After the swap, a deposit arriving from the **old** connector is no longer recognized as a base-token deposit; it is treated as an ERC-20 transfer and will fail or be misrouted.

The public WASM entrypoint is exposed unconditionally: [5](#0-4) 

---

### Impact Explanation

**High — Temporary (potentially permanent) freezing of funds.**

After `set_eth_connector_contract_account` is called with a new connector:

1. The old connector still holds all bridged ETH (its NEP-141 supply is unchanged).
2. The new connector holds zero ETH.
3. Every call to `withdraw` is routed to the new connector, which has no ETH to release; the promise fails and the user's Aurora ETH is locked.
4. `ft_transfer` and `ft_transfer_call` similarly fail, preventing users from moving their ETH-on-NEAR balance.
5. Deposits from the old connector are misclassified, breaking the inbound bridge path.

The freeze persists until the owner either reverts to the old connector account or manually migrates the ETH balance out-of-band. If the owner does not act, the freeze is permanent.

---

### Likelihood Explanation

The owner legitimately calls `set_eth_connector_contract_account` when upgrading the connector contract (e.g., to patch a bug or add a feature). The function gives no warning about outstanding balances and performs no balance check, so the owner may not realize that the ETH held by the old connector is stranded. The scenario is directly analogous to the reference report's gauge-upgrade scenario and requires no malicious intent.

---

### Recommendation

Before overwriting the connector account ID, `set_eth_connector_contract_account` should:

1. **Read the old connector's total ETH supply** (via a cross-contract view call to `ft_total_supply` on the old connector).
2. **Reject the update** (or require an explicit override flag) if the old connector still holds a non-zero balance.
3. Alternatively, **emit a cross-contract call** to transfer the full NEP-141 balance from the old connector to the new one before updating the stored account ID, mirroring the fix applied in the reference report.

---

### Proof of Concept

```
1. Users deposit ETH into Aurora; the old connector (old_connector.near) holds
   N ETH as NEP-141 tokens.

2. Owner calls set_eth_connector_contract_account(new_connector.near, Borsh).
   Storage key EthConnectorStorageId::EthConnectorAccount is overwritten.
   No balance check, no migration.

3. User calls withdraw(recipient_eth_address, amount).
   return_promise reads get_connector_account_id() → new_connector.near.
   Promise targets new_connector.near::engine_withdraw.
   new_connector.near has zero ETH → promise fails.
   User's Aurora ETH balance is unchanged but unwithdrawable.

4. old_connector.near still holds N ETH with no path to release it,
   because the engine no longer routes any calls to it.

5. Owner must either:
   (a) call set_eth_connector_contract_account(old_connector.near, ...) to revert, or
   (b) manually orchestrate an out-of-band migration — the same cumbersome
       multi-step process described in the reference report.
```

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}
