### Title
`EvmErc20.sol` / `EvmErc20V2.sol` Burn Tokens Before Unchecked Precompile Call, Causing Permanent Fund Loss on Failure - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` irreversibly burn a user's ERC-20 tokens **before** calling the `ExitToNear` / `ExitToEthereum` precompile, and never check the return value of that call. If the precompile call fails for any reason, the tokens are permanently destroyed with no recovery path.

---

### Finding Description

In `withdrawToNear` (and `withdrawToEthereum`) the pattern is:

```solidity
// EvmErc20.sol lines 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here, irreversibly

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — silent failure
    }
}
``` [1](#0-0) 

The identical pattern appears in `EvmErc20V2.sol`: [2](#0-1)

**Concrete failure path — input format mismatch with `error_refund` feature:**

The `ExitToNear` precompile has two compile-time input formats controlled by the `error_refund` Cargo feature:

```rust
// engine-precompiles/src/native.rs
#[cfg(not(feature = "error_refund"))]
const MIN_INPUT_SIZE: usize = 3;
#[cfg(feature = "error_refund")]
const MIN_INPUT_SIZE: usize = 21;
``` [4](#0-3) 

When `error_refund` is **enabled**, `parse_input` consumes bytes `1..21` as a `refund_address`:

```rust
#[cfg(feature = "error_refund")]
fn parse_input(input: &[u8]) -> Result<(Address, &[u8]), ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    let mut buffer = [0; 20];
    buffer.copy_from_slice(&input[1..21]);
    let refund_address = Address::from_array(buffer);
    Ok((refund_address, &input[21..]))
}
``` [5](#0-4) 

`EvmErc20.sol` (V1) sends: `flag(1) + amount_b(32) + recipient(variable)` — **no refund address**.

When the engine is compiled with `error_refund`, the precompile misparses this as:
- `refund_address` = first 20 bytes of `amount_b`
- `amount` = last 12 bytes of `amount_b` + first 20 bytes of `recipient` → almost certainly exceeds `u128::MAX`

```rust
// engine-precompiles/src/native.rs line 759
let amount = parse_amount(&input[..32])?;
// parse_amount returns ERR_INVALID_AMOUNT if amount > u128::MAX
``` [6](#0-5) 

The precompile returns `ERR_INVALID_AMOUNT`. The EVM `call` returns `res = 0`. Because `res` is never checked, the Solidity function returns normally. The tokens are already burned. No NEAR promise is ever created. No `error_refund` callback fires (it only fires on promise failure, not precompile failure).

`EvmErc20V2.sol` correctly includes the sender as refund address for the `error_refund` path, but its `withdrawToEthereum` still does not check `res`, leaving it vulnerable to any other precompile failure.

**Additional failure triggers (no feature flag required):**
- User supplies an invalid NEAR account ID as `recipient` → `ERR_INVALID_RECEIVER_ACCOUNT_ID` from `parse_recipient`
- Amount encoding produces a value > `u128::MAX` → `ERR_INVALID_AMOUNT`

In all cases: tokens burned, `res` unchecked, no revert, permanent loss.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any user calling `withdrawToNear` or `withdrawToEthereum` on a V1 `EvmErc20` contract deployed against an `error_refund`-enabled engine, or supplying an invalid recipient on either contract version, will have their ERC-20 tokens permanently destroyed. The NEP-141 tokens on the NEAR side are never released. There is no admin recovery path.

---

### Likelihood Explanation

**Medium.**

- `EvmErc20.sol` (V1) and `EvmErc20V2.sol` (V2) coexist in the repository; V1 contracts deployed before the `error_refund` feature was introduced remain live on-chain.
- Any user who mistypes a NEAR account ID triggers the bug on either contract version, regardless of the feature flag.
- The silent failure (no revert, no event) means users receive no feedback and cannot distinguish a successful withdrawal from a failed one.

---

### Recommendation

1. Check the return value of the precompile `call` and revert on failure:
   ```solidity
   assembly {
       let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
       if iszero(res) { revert(0, 0) }
   }
   ```
   This ensures `_burn` is rolled back (via revert) if the precompile fails.

2. Migrate all deployed V1 `EvmErc20` contracts to V2 (or a corrected version) when the engine is compiled with `error_refund`, to eliminate the input-format mismatch.

---

### Proof of Concept

1. Deploy `EvmErc20.sol` on an Aurora Engine compiled with the `error_refund` feature.
2. Mint tokens to `alice`.
3. `alice` calls `withdrawToNear("alice.near", 1000)`.
4. `_burn(alice, 1000)` executes — alice's balance drops to 0.
5. Input to precompile: `\x01` + `bytes32(1000)` + `"alice.near"`.
6. Precompile (with `error_refund`) reads `refund_address` = first 20 bytes of `bytes32(1000)` = `0x00...03e8`.
7. Remaining input = last 12 bytes of `bytes32(1000)` + `"alice.near"` = 22 bytes.
8. `parse_amount(&input[..32])` — only 22 bytes available, but `input[..32]` panics or reads garbage; more precisely `parse_amount` receives the 22-byte slice padded, producing a value >> `u128::MAX` → returns `Err(ERR_INVALID_AMOUNT)`.
9. Precompile returns failure; EVM `call` returns `res = 0`.
10. `res` is never checked; `withdrawToNear` returns normally.
11. Alice's 1000 tokens are gone. No NEAR transfer occurs. Funds permanently frozen. [1](#0-0) [7](#0-6) [8](#0-7)

impl<'a> TryFrom<&'a [u8]> for ExitToNearParams<'a> {
    type Error = ExitError;

    fn try_from(input: &'a [u8]) -> Result<Self, Self::Error> {
        // The first byte of the input is a flag, selecting the behavior to be triggered:
        // 0x00 -> Eth(base) token withdrawal
        // 0x01 -> ERC-20 token withdrawal
        let flag = input
            .first()
            .copied()
            .ok_or_else(|| ExitError::Other(Cow::from("ERR_MISSING_FLAG")))?;

        #[cfg(feature = "error_refund")]
        let (refund_address, input) = parse_input(input)?;
        #[cfg(not(feature = "error_refund"))]
        let input = parse_input(input)?;

        match flag {
            0x0 => {
                let Recipient {
                    receiver_account_id,
                    message,
                } = parse_recipient(input)?;

                Ok(Self::BaseToken(BaseTokenParams {
                    #[cfg(feature = "error_refund")]
                    refund_address,
                    receiver_account_id,
                    message,
                }))
            }
            0x1 => {
                let amount = parse_amount(&input[..32])?;
                let Recipient {
                    receiver_account_id,
                    message,
                } = parse_recipient(&input[32..])?;

                Ok(Self::Erc20TokenParams(Erc20TokenParams {
                    #[cfg(feature = "error_refund")]
                    refund_address,
                    receiver_account_id,
                    amount,
                    message,
                }))
            }
            _ => Err(ExitError::Other(Cow::from("ERR_INVALID_FLAG"))),
        }
    }
