Audit Report

## Title
Missing Lower Bound on `max_dissolve_delay_seconds` Disables SNS Dissolve-Delay Governance Incentive — (`File: rs/sns/governance/src/types.rs`, `rs/sns/init/src/lib.rs`)

## Summary
Both the `SnsInitPayload` validation path and the runtime `NervousSystemParameters::validate()` path accept `max_dissolve_delay_seconds = 0` because they only check for presence, not for a meaningful lower bound. When combined with `neuron_minimum_dissolve_delay_to_vote_seconds = 0`, this passes every validation gate and causes an SNS to launch with its dissolve-delay governance incentive completely disabled and with fully dissolved neurons eligible to vote on equal footing with long-term stakers.

## Finding Description
`SnsInitPayload::validate_max_dissolve_delay_seconds` at `rs/sns/init/src/lib.rs` L1214–1219 only unwraps the `Option` and returns `Ok(())` with no value check: [1](#0-0) 

`NervousSystemParameters::validate_max_dissolve_delay_seconds` at `rs/sns/governance/src/types.rs` L792–796 is identical — it only calls `.ok_or_else(...)` with no lower-bound assertion: [2](#0-1) 

`SnsInitPayload::validate_neuron_minimum_dissolve_delay_to_vote_seconds` at `rs/sns/init/src/lib.rs` L1064–1085 compares the submitted `neuron_minimum_dissolve_delay_to_vote_seconds` against the **default** `max_dissolve_delay_seconds` (pulled from `NervousSystemParameters::with_default_values()`), not against `self.max_dissolve_delay_seconds`. So submitting `(max=0, min_vote=0)` passes because `0 ≤ default_max`: [3](#0-2) 

The runtime cross-check at `rs/sns/governance/src/types.rs` L764–771 only enforces `neuron_minimum_dissolve_delay_to_vote_seconds ≤ max_dissolve_delay_seconds`, so `0 ≤ 0` passes: [4](#0-3) 

At runtime, `compute_ballots_for_new_proposal` in `rs/sns/governance/src/governance.rs` L5258–5261 skips neurons whose `dissolve_delay_seconds < min_dissolve_delay_for_vote`. When `min_dissolve_delay_for_vote = 0`, this condition is never satisfied (dissolve delay is always ≥ 0), so every neuron — including fully dissolved ones — receives a ballot: [5](#0-4) 

The `voting_power` function in `rs/sns/governance/src/neuron.rs` L213–219 guards against divide-by-zero with `if max_dissolve_delay_seconds > 0`, but the else branch silently returns `0` for the bonus, meaning the entire dissolve-delay incentive is zeroed out for every neuron: [6](#0-5) 

## Impact Explanation
This is a **High** severity SNS governance security impact. The dissolve-delay bonus is the core mechanism that gives long-term token holders proportionally more voting power than short-term or dissolved holders. Setting `max_dissolve_delay_seconds = 0` collapses all voting power to raw stake, allows dissolved neurons to vote, and eliminates the economic incentive for long-term commitment. SNS token holders who participate in the swap are directly harmed: the governance model they were presented with does not function as designed. This matches the allowed impact: *"Significant SNS security impact with concrete user or protocol harm."*

## Likelihood Explanation
The entry point is a `CreateServiceNervousSystem` NNS proposal, which any NNS neuron holder meeting the minimum dissolve delay threshold can submit. The attacker does not need to control an NNS governance majority — they only need to submit a proposal that passes automated validation. The validation failure means NNS voters reviewing the proposal have no automated signal that the parameters are degenerate. The attack is repeatable for any SNS launch and requires no special privileges beyond holding an NNS neuron. [7](#0-6) 

## Recommendation
- **Short term:** Add a non-zero lower bound check in both `SnsInitPayload::validate_max_dissolve_delay_seconds` and `NervousSystemParameters::validate_max_dissolve_delay_seconds`. A sensible minimum (e.g., one week = 604800 seconds) should be enforced. Apply the same fix to `validate_max_neuron_age_for_age_bonus` / `validate_max_neuron_age_seconds_for_age_bonus`.
- **Long term:** Fix `SnsInitPayload::validate_neuron_minimum_dissolve_delay_to_vote_seconds` to compare against `self.max_dissolve_delay_seconds` (the submitted value) rather than the default. Enforce a non-zero minimum for `neuron_minimum_dissolve_delay_to_vote_seconds` to prevent dissolved neurons from voting.

## Proof of Concept
Submit a `CreateServiceNervousSystem` NNS proposal with:
```
max_dissolve_delay_seconds = 0
neuron_minimum_dissolve_delay_to_vote_seconds = 0
```

Validation trace:
1. `SnsInitPayload::validate_max_dissolve_delay_seconds()` → `Some(0)` present → **passes** (L1214–1219)
2. `SnsInitPayload::validate_neuron_minimum_dissolve_delay_to_vote_seconds()` → compares `0` against default max (e.g., 252460800) → `0 ≤ 252460800` → **passes** (L1064–1085)
3. `NervousSystemParameters::validate_max_dissolve_delay_seconds()` → `Some(0)` present → **passes** (L792–796)
4. `NervousSystemParameters::validate_neuron_minimum_dissolve_delay_to_vote_seconds()` → `0 ≤ 0` → **passes** (L764–771)

Runtime effect:
- `compute_ballots_for_new_proposal`: `min_dissolve_delay_for_vote = 0` → every neuron including dissolved ones receives a ballot (L5258–5261)
- `voting_power(...)`: `max_dissolve_delay_seconds = 0` → dissolve delay bonus branch returns `0` → all neurons receive only base stake as voting power (L213–219)

A deterministic unit test can be written against `NervousSystemParameters::validate()` and `SnsInitPayload::validate_pre_execution()` with `max_dissolve_delay_seconds = Some(0)` and `neuron_minimum_dissolve_delay_to_vote_seconds = Some(0)` to confirm both return `Ok(...)`, and against `Neuron::voting_power(...)` with `max_dissolve_delay_seconds = 0` to confirm the bonus is zero regardless of the neuron's dissolve delay.
