Audit Report

## Title
`ic0.cycles_burn128` During DTS Execution Allows Ingress Induction Fees to Be Silently Dropped - (`rs/replicated_state/src/canister_state/system_state.rs`)

## Summary
When a canister has a paused DTS execution, ingress induction costs are deferred into `ingress_induction_cycles_debit` rather than immediately deducted from `cycles_balance`. The `cycles_burn` function, invoked by `ic0.cycles_burn128` during Wasm execution, only enforces the freeze threshold against the raw `cycles_balance` without accounting for the pending debit. A canister can therefore burn its balance below the outstanding debit. When `apply_ingress_induction_cycles_debit` is called at execution completion, the excess debit is explicitly acknowledged as a bug and silently dropped, causing ingress induction fees to go uncollected — a cycles conservation violation.

## Finding Description

**Step 1 — Ingress induction cost deferred as pending debit.**

`charge_ingress_induction_cost` detects a paused execution via `has_paused_execution_or_install_code()` and records the cost using `add_postponed_charge_to_ingress_induction_cycles_debit`. The affordability check uses `debited_balance()` (i.e., `cycles_balance - ingress_induction_cycles_debit`), so the debit accumulates without touching `cycles_balance`. [1](#0-0) 

**Step 2 — `cycles_burn` ignores the pending debit.**

`cycles_burn` computes the maximum burnable amount as `*cycles_balance - threshold`, with no reference to `ingress_induction_cycles_debit`. A canister can burn down to the freeze threshold, reducing `cycles_balance` below the accumulated debit. [2](#0-1) 

`ic0.cycles_burn128` is available in `Update`, `ReplyCallback`, `RejectCallback`, `Cleanup`, `Init`, `PreUpgrade`, `ReplicatedQuery`, and `SystemTask` contexts — all of which can occur during or around DTS execution. [3](#0-2) 

**Step 3 — Excess debit is silently dropped at execution completion.**

`apply_ingress_induction_cycles_debit` computes `remaining_debit = ingress_induction_cycles_debit - cycles_balance` (saturating). When this is nonzero, the code logs an error, increments a counter, and explicitly drops the remaining debit. The subsequent `consume_cycles` call uses the full `ingress_induction_cycles_debit` value, but because `Cycles` arithmetic is saturating, the balance clamps to zero and the excess is never collected. [4](#0-3) 

This function is called at the end of both update and response callback execution: [5](#0-4) [6](#0-5) 

The `debited_balance()` helper confirms the two-field design: `cycles_balance - ingress_induction_cycles_debit`, which `cycles_burn` bypasses entirely. [7](#0-6) 

## Impact Explanation

This is a cycles conservation violation: ingress induction fees that the protocol requires to be burned are silently dropped. Cycles that should be destroyed are effectively preserved, constituting illegal minting. The attack is repeatable across arbitrarily many canisters and rounds. While the per-message induction fee is small, the attack is fully automatable and scalable with no per-iteration cost beyond deploying canisters and submitting messages. This maps to the **High** impact tier: significant protocol-level accounting integrity loss with concrete, repeatable economic harm to the cycles economy.

## Likelihood Explanation

Both preconditions are fully attacker-controlled: (1) the canister Wasm code (to trigger DTS and call `ic0.cycles_burn128`) and (2) the ingress messages sent while the canister is paused (which the attacker can send themselves). No privileged access, governance majority, subnet corruption, or victim interaction is required. The `ic0.cycles_burn128` system call is broadly available across execution contexts. The attack is deterministic and reproducible.

## Recommendation

`cycles_burn` should use `debited_balance()` instead of `cycles_balance` when computing the maximum burnable amount. The cap should be `debited_balance() - threshold` rather than `*cycles_balance - threshold`. This requires passing `ingress_induction_cycles_debit` into `cycles_burn`, or restructuring the call site to pass `debited_balance()` as the effective balance. Alternatively, the `sandbox_safe_system_state.cycles_burn128` path should enforce the same `debited_balance()` constraint that `charge_ingress_induction_cost` uses.

## Proof of Concept

```
1. Deploy canister C funded with B cycles, with an update method that:
   a. Executes enough instructions to span multiple DTS slices.
   b. Calls ic0.cycles_burn128(B - freeze_threshold) to drain to freeze threshold.

2. Submit the update call to C.
   → Execution begins; C is paused after the first DTS slice.
   → cycles_balance = B (burn has not yet executed).

3. While C is paused, submit N ingress messages to C.
   → charge_ingress_induction_cost detects paused execution.
   → Each message's induction cost is added to ingress_induction_cycles_debit.
   → debited_balance() check passes (balance still appears sufficient).

4. C's execution resumes and the burn executes:
   → cycles_balance drops to freeze_threshold.
   → ingress_induction_cycles_debit >> cycles_balance.

5. Execution finishes; apply_ingress_induction_cycles_debit is called:
   → remaining_debit = ingress_induction_cycles_debit - cycles_balance > 0.
   → Error counter incremented; remaining debit DROPPED.
   → N ingress messages inducted without paying their fees.

Verification: assert that the sum of cycles burned from the subnet's perspective
is less than (B - freeze_threshold) + N * per_message_induction_fee.
A deterministic integration test using PocketIC can confirm the discrepancy
by comparing pre- and post-execution subnet cycle totals.
```
