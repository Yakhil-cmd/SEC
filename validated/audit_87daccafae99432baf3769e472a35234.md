Audit Report

## Title
Permissionless `refresh_buyer_tokens` Bypasses Confirmation-Text Consent and Forces Participation on Behalf of Any Buyer — (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` endpoint accepts an arbitrary `buyer` principal from the caller without verifying it matches the authenticated caller identity. Any unprivileged ingress sender can invoke the function on behalf of any other buyer, committing that buyer's ICP to the swap and deleting their open ticket. When an SNS configures a `confirmation_text` — a mechanism explicitly designed to require affirmative buyer consent — a third party can supply the publicly readable text verbatim, bypassing the consent guard entirely and locking the victim's ICP into the swap without the victim ever having agreed to the terms.

## Finding Description
In `rs/sns/swap/canister/canister.rs` (L130–134), the canister resolves the target buyer from the caller-supplied `arg.buyer` string with no authentication check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()   // any principal accepted
};
```

The `RefreshBuyerTokensRequest.buyer` field carries no authentication binding (`rs/sns/swap/src/gen/ic_sns_swap.pb.v1.rs` L1117–1126).

Inside `refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs`):

1. **Confirmation text validation** (L1149–1150) checks only that the caller-supplied string matches the SNS-stored string. It does not verify that the caller is the buyer. The stored text is publicly readable via `get_init`.
2. **Ledger balance query** (L1153–1163) reads the *victim's* subaccount balance.
3. **Ticket deletion** (L1268–1271): if `amount_ticket <= requested_increment_e8s`, the victim's open ticket is deleted and participation is committed.
4. **`error_refund_icp` is blocked** (L1950–1959) while the buyer has ICP in escrow, so the victim cannot reclaim their ICP until the swap finalizes.

**Exploit flow:**

| Step | Actor | Action |
|------|-------|--------|
| 1 | Victim | Calls `new_sale_ticket` for amount X ICP |
| 2 | Victim | Transfers X ICP to their swap subaccount |
| 3 | Griefer | Calls `refresh_buyer_tokens { buyer: "<victim>", confirmation_text: "<public_text>" }` |
| 4 | Swap | Reads victim's balance, validates text (passes), deletes ticket, commits participation |
| 5 | Victim | ICP is locked in escrow; `error_refund_icp` returns precondition error until swap finalizes |

Note: the claim's assertion that the victim's subsequent `refresh_buyer_tokens` call "fails with an error" is inaccurate — the code at L1215–1221 returns `Ok` when `old_amount_icp_e8s >= e8s`. The core attack (confirmation bypass + forced commitment) is nonetheless valid.

Existing guards are insufficient: `validate_confirmation_text` (`rs/sns/swap/src/swap.rs` L363–384) performs only a string equality check against the stored text, with no binding to the caller's identity.

## Impact Explanation
This is a **Medium** SNS security impact. The `confirmation_text` feature is explicitly designed to require affirmative buyer consent before participation is committed (documented in `rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto` L847–850 and `rs/sns/init/proto/ic_sns_init/pb/v1/sns_init.proto` L134–137). A griefer can render this consent mechanism entirely ineffective for any buyer who has transferred ICP to their swap subaccount, committing their funds to the swap without the buyer ever having agreed to the terms. The victim's ICP is not stolen but is locked in escrow until swap finalization, and if the swap commits, the victim receives SNS tokens without having consented. This constitutes a significant SNS security impact with concrete user harm: unauthorized commitment of user funds and bypass of a protocol-level consent guard.

## Likelihood Explanation
No special privilege is required. Any ingress sender with a non-anonymous principal can call `refresh_buyer_tokens`. The victim's principal is observable from public swap state (`list_direct_participants`, `get_buyer_state`). The confirmation text is readable from `get_init`. The attack costs only ledger query cycles and is trivially scriptable. The only precondition is that the victim has transferred ICP to their swap subaccount, which is the normal participation flow.

## Recommendation
1. **Bind the `buyer` field to the caller**: if `arg.buyer` is non-empty, assert `PrincipalId::from_str(&arg.buyer) == caller_principal_id()` and reject mismatches. This is the minimal fix.
2. **Alternatively**, remove the `buyer` field entirely and always derive the target from `caller_principal_id()`, as `new_sale_ticket` already does.
3. If backward compatibility with the existing "notify on behalf of" use-case (used in integration tests) is required, restrict third-party calls to the case where no `confirmation_text` is configured for the swap, and require the caller to be the buyer whenever `confirmation_text` is set.

## Proof of Concept
Minimal integration test using `StateMachine` / PocketIC:

```rust
// Setup: SNS swap open with confirmation_text = "I agree to the terms"
// Victim creates ticket and transfers ICP
new_sale_ticket(env, swap_id, victim, min_icp_e8s, None);
icp_ledger_transfer(env, victim, swap_subaccount(victim), min_icp_e8s);

// Griefer (different principal) commits victim's participation
let args = Encode!(&RefreshBuyerTokensRequest {
    buyer: victim.to_string(),                          // victim's principal
    confirmation_text: Some("I agree to the terms".to_string()), // public text
}).unwrap();
env.execute_ingress_as(griefer, swap_id, "refresh_buyer_tokens", args).unwrap();

// Assert: victim's participation is committed without victim ever calling refresh_buyer_tokens
let state = get_buyer_state(env, swap_id, &victim);
assert!(state.buyer_state.unwrap().icp.unwrap().amount_e8s >= min_icp_e8s);

// Assert: victim's ticket is deleted
assert!(get_open_ticket(env, swap_id, victim).ticket().is_none());

// Assert: victim cannot reclaim ICP (locked in escrow)
let refund = error_refund_icp(env, swap_id, victim);
assert!(refund.result.unwrap().is_err()); // precondition error: "in escrow"
```
