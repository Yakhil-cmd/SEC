### Title
f64 Precision Loss in `convert_tokens_to_string_representation` Causes Consent Message to Display Incorrect Token Amount — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

---

### Summary

`convert_tokens_to_string_representation` converts a `Nat` token amount to `f64` via `to_f64()` before formatting it for display in an ICRC-21 `GenericDisplayMessage`. Because `f64` has only 53 bits of mantissa, any integer value above 2^53 is rounded to the nearest representable float. The consent message therefore displays a value that differs from the actual transfer amount the user is authorizing, violating the core ICRC-21 invariant that the displayed amount must be exact.

---

### Finding Description

The vulnerable function is:

```rust
// packages/icrc-ledger-types/src/icrc21/responses.rs, lines 318–327
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let tokens = tokens.0.to_f64().ok_or(Icrc21Error::GenericError { ... })?;
    Ok(format!("{}", tokens / 10_f64.pow(decimals)))
}
``` [1](#0-0) 

This function is called from `add_amount`, `add_fee`, `add_allowance`, and `add_existing_allowance` — but **only** for the `ConsentMessage::GenericDisplayMessage` branch. The `FieldsDisplayMessage` branch uses `nat_to_u64`, which preserves exact integer values. [2](#0-1) 

Any caller can request `GenericDisplay` by setting `device_spec: Some(DisplayMessageType::GenericDisplay)` in the `ConsentMessageRequest`. This is an unprivileged, publicly accessible ingress path: [3](#0-2) 

The builder wires the caller-supplied `device_spec` directly into the message variant selection: [4](#0-3)

### Impact Explanation

`f64` has 52 explicit mantissa bits (53 with the implicit leading 1). For any integer N > 2^53, `N as f64` rounds to the nearest representable value. The rounding error is at most `2^(floor(log2(N)) - 52)` units.

For amounts within the `u64` range (up to 2^64 − 1), the maximum absolute rounding error is `2^(64−52) = 4096` token units. For ICP (8 decimals) this is 0.00004096 ICP — negligible in value. However:

- The ICRC-21 standard's security guarantee is **exact** display of the authorized amount. Any discrepancy, however small, breaks this invariant.
- A malicious dapp can craft an amount `A` such that `A.to_f64()` rounds to `B < A`. The consent message shows `B`; the actual transfer executes with `A`. The user authorizes more than they believe they are sending.
- For tokens with 0 decimals and high per-unit value, the 4096-unit discrepancy could represent meaningful value.
- For `Nat` values above `u64::MAX` (which `to_f64()` accepts, unlike `nat_to_u64`), the rounding error grows unboundedly — though in practice most ledger implementations would reject such transfers at execution time.

---

### Likelihood Explanation

The attack requires a malicious dapp that:
1. Intercepts or crafts the transfer amount.
2. Requests `GenericDisplay` consent (the default when `device_spec` is `None`).
3. Submits the unrounded amount after the user approves the rounded display.

This is a realistic threat model for ICRC-21, which exists precisely to protect users from malicious dapps. The entrypoint is fully unprivileged and reachable via normal ingress. The precision loss is deterministic and locally testable.

---

### Recommendation

Replace the `f64` conversion with exact integer arithmetic. The correct approach is to perform integer division and modulo to split the amount into whole and fractional parts, then format them as a decimal string — the same pattern used in `format_tokens` in `rs/nervous_system/humanize/src/lib.rs`: [5](#0-4) 

Concretely, for `convert_tokens_to_string_representation`:
- Compute `whole = tokens / 10^decimals` (exact `BigUint` division).
- Compute `frac = tokens % 10^decimals` (exact remainder).
- Format as `"{whole}.{frac:0>decimals$}"` with trailing-zero trimming.

This eliminates the `f64` conversion entirely and preserves exact representation for all `Nat` values.

---

### Proof of Concept

```rust
// Demonstrates the precision loss
use candid::Nat;
use num_bigint::BigUint;
use num_traits::ToPrimitive;

let amount = BigUint::from(2u64.pow(53) + 1); // 9007199254740993
let as_f64 = amount.to_f64().unwrap();         // 9007199254740992.0  ← rounds DOWN
let displayed = format!("{}", as_f64 / 1.0);   // "9007199254740992"

assert_ne!(displayed, "9007199254740993");      // FAILS: displayed != actual
// User sees 9007199254740992 but authorizes 9007199254740993
```

A malicious dapp targeting a 0-decimal token near `u64::MAX` can achieve a discrepancy of up to 4096 units between the displayed and executed amount, with the user having no way to detect the difference from the consent message alone.
