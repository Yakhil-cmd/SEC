### Title
`ExitToNear` Precompile Burns ERC-20 Tokens Without Refund When NEP-141 Transfer Fails (Without `error_refund` Feature) - (`engine-precompiles/src/native.rs`)

---

### Summary

When a user calls `withdrawToNear` on an `EvmErc20` contract, the ERC-20 tokens are burned first, then a NEAR `ft_transfer` promise is scheduled asynchronously. If that promise fails and the `error_refund` feature is not compiled in, the `ExitToNear` precompile sets `refund: None` in the callback args, so `exit_to_near_precompile_callback` performs no refund. The user's ERC-20 tokens are permanently destroyed with no NEP-141 tokens received.

---

### Finding Description

The `ExitToNear` precompile's `run` method constructs `ExitToNearPrecompileCallbackArgs` with a conditional refund field:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is absent (it is **not** in the `default` feature set of `engine/Cargo.toml`), `refund` is always `None`. [2](#0-1) 

The callback `exit_to_near_precompile_callback` only triggers a refund when `args.refund` is `Some(...)`:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None
};
``` [3](#0-2) 

Without `error_refund`, the `else` branch is always taken on failure — no refund, no re-mint.

The Solidity entry point `EvmErc20.withdrawToNear` burns tokens **before** calling the precompile:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    ...
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, ...)
    }
}
``` [4](#0-3) 

`EvmErc20V2.sol` has the identical pattern: [5](#0-4) 

The burn is committed inside the EVM transaction. The `ft_transfer` executes asynchronously as a NEAR promise. If it fails (e.g., recipient not registered in the NEP-141 contract, or the NEP-141 contract has a blocklist and the recipient is on it), the callback fires with `refund = None` and the tokens are gone.

The codebase's own test explicitly documents this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When `ft_transfer` fails and `error_refund` is not compiled in, the user's ERC-20 tokens are permanently burned with no NEP-141 tokens received. The funds are irrecoverably lost from the user's perspective; there is no admin recovery path for the burned ERC-20 side.

---

### Likelihood Explanation

The `ft_transfer` call can fail for multiple realistic reasons:

- The recipient NEAR account is not registered (has no storage deposit) in the NEP-141 contract — a common operational mistake.
- The NEP-141 contract implements a blocklist (analogous to USDC/USDT on Ethereum) and the recipient is on it.
- The NEP-141 contract is paused at the time the async promise executes.

Any of these conditions, combined with the `error_refund` feature being absent from the default build, produces a permanent loss. The `error_refund` feature is not in `default` in `engine/Cargo.toml`, making this the code path that executes unless explicitly opted into at compile time.

---

### Recommendation

1. **Enable `error_refund` by default** — add it to the `default` feature set in `engine/Cargo.toml` so the refund path is always compiled in.
2. **Alternatively, restructure the flow** — do not burn ERC-20 tokens until the `ft_transfer` promise has succeeded (i.e., burn only inside the success branch of the callback). This eliminates the need for a refund path entirely.
3. **At minimum**, document clearly in the contract ABI and user-facing interfaces that `withdrawToNear` is irreversible if the NEP-141 transfer fails, and validate the recipient is registered before burning.

---

### Proof of Concept

1. Deploy a NEP-141 token with a blocklist; blocklist address `alice`.
2. Bridge the NEP-141 to Aurora, minting ERC-20 tokens for `alice`.
3. `alice` calls `EvmErc20.withdrawToNear(alice_near_account, amount)`.
4. Inside the EVM transaction: `_burn(alice, amount)` executes — ERC-20 balance goes to zero.
5. The `ExitToNear` precompile schedules `ft_transfer` to the NEP-141 contract with `refund: None` (because `error_refund` is not enabled).
6. The NEP-141 `ft_transfer` promise fails because `alice` is blocklisted.
7. `exit_to_near_precompile_callback` fires; `args.refund` is `None`; the `else` branch returns `None` with no re-mint.
8. `alice` has zero ERC-20 tokens and zero NEP-141 tokens — funds permanently frozen.

The existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` (lines 623–665) already demonstrates the token loss when `error_refund` is absent, confirming the root cause is present in the production code path. [7](#0-6)
