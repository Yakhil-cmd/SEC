Audit Report

## Title
Bitcoin Checker Canister Provider Rotation Has No Exhaustion Detection, Causing Permanent ckBTC Deposit Liveness Failure When All Providers Are Down - (File: rs/bitcoin/checker/src/fetch.rs)

## Summary
The `try_fetch_tx` function in the Bitcoin Checker canister advances to the next provider via `err.provider.next()` on every `FetchTxStatus::Error` match arm, with no mechanism to detect that all providers in the ring have already been tried for a given `txid`. When all three Mainnet providers (Btcscan, Blockstream, MempoolSpace) are simultaneously unreachable, the canister cycles through them indefinitely, always returning `CheckTransactionRetriable::TransientInternalError`. The ckBTC minter treats every `Retriable` response as `UpdateBalanceError::TemporarilyUnavailable`, permanently blocking `update_balance` for any UTXO whose transaction cannot be fetched, with no on-chain recovery path for the depositing user.

## Finding Description
In `rs/bitcoin/checker/src/fetch.rs` lines 85–90, the `FetchTxStatus::Error` arm of `try_fetch_tx` unconditionally advances to the next provider:

```rust
Some(FetchTxStatus::Error(err)) => (
    err.provider.next(),
    err.max_response_bytes,
),
```

`Provider::next()` in `rs/bitcoin/checker/src/providers.rs` lines 71–74 implements a closed ring:

```
Btcscan → Blockstream → MempoolSpace → Btcscan → …
```

`FetchTxStatusError` in `rs/bitcoin/checker/src/state.rs` lines 54–59 stores only the single last-failing provider — no set of exhausted providers, no retry counter. Each failed fetch overwrites the stored error with the new failing provider, so the state machine has no memory of how many distinct providers have been tried.

When all three providers fail in sequence, the stored status cycles: `Error{A}` → `Error{B}` → `Error{C}` → `Error{A}` → … indefinitely. Every iteration returns `FetchResult::Error(err)`, which `check_transaction_inputs` (main.rs line 552) converts via `err.into_response(txid)`. `into_response` (fetch.rs lines 25–31) maps all non-`ResponseTooLarge` errors to `CheckTransactionRetriable::TransientInternalError`.

The ckBTC minter's `check_utxo` in `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs` lines 432–440 treats any `Retriable` response as an immediate early-return `UpdateBalanceError::TemporarilyUnavailable` — it does not retry across calls. The `FetchTxStatus::Error` entry persists in the LRU heap cache (up to 10,000 entries, `state.rs` lines 121–132) until evicted by pressure or a canister upgrade. There is no terminal `PermanentlyFailed` / `AllProvidersExhausted` state that would allow the minter to quarantine the UTXO and inform the user definitively.

The same cycling logic applies inside `check_fetched` (fetch.rs lines 196–226) when fetching input transactions of an already-fetched main transaction, compounding the issue for multi-input Bitcoin transactions.

## Impact Explanation
This is a **High** impact finding matching the allowed class: *"Significant Chain Fusion, ck-token, ledger… security impact with concrete user or protocol harm."*

Any user who sends BTC to the ckBTC minter's deposit address while all three Bitcoin API providers are unreachable will receive `UpdateBalanceError::TemporarilyUnavailable` on every `update_balance` call, indefinitely. Because the Bitcoin Checker canister never transitions to a terminal error state, the ckBTC minter has no signal to mint ckBTC or to quarantine the UTXO. The deposited BTC remains locked in the minter-controlled threshold-ECDSA Bitcoin address with no unilateral on-chain recovery path for the user. The only remediation is an NNS governance proposal to upgrade the Bitcoin Checker canister, introducing substantial delay. Withdrawals (`retrieve_btc`) are unaffected because `check_address` is a pure in-memory SDN list lookup.

## Likelihood Explanation
The trigger requires all three providers (Btcscan, Blockstream, MempoolSpace) to be simultaneously unreachable from the IC subnet at the time each respective provider is tried. Plausible scenarios include: all three providers independently blocking IC egress IP ranges (each has done so individually in the past for ckETH), coordinated legal/regulatory takedown, or a subnet-level HTTPS outcall failure. No privileged access, governance action, or attacker capability beyond external provider availability is required — the condition is triggered purely by infrastructure state outside the protocol's control. The scenario is unlikely under normal operations but is a credible escalation of the already-observed single-provider failure mode.

## Recommendation
Add a `providers_tried: BTreeSet<ProviderId>` (or an equivalent exhaustion counter) to `FetchTxStatusError`. In `try_fetch_tx`, after computing the next provider via `err.provider.next()`, check whether that provider is already in `providers_tried`. If all providers in the ring have been exhausted, transition the status to a new terminal variant (e.g., `FetchTxStatus::AllProvidersExhausted`) and return a `CheckTransactionIrrecoverableError` (new variant `AllProvidersExhausted`). The ckBTC minter should handle this terminal error by quarantining the UTXO and returning a definitive `GenericError` to the user rather than looping on `TemporarilyUnavailable`.

## Proof of Concept
A deterministic unit test using the existing `FetchEnv` mock infrastructure in `rs/bitcoin/checker/src/fetch/tests` can reproduce this without any network access:

1. Instantiate a mock `FetchEnv` whose `http_get_tx` always returns `Err(HttpGetTxError::Rejected { code: 500, message: "down".into() })` for all providers.
2. Call `try_fetch_tx(txid)` three times in sequence (once per provider in the ring).
3. Assert that after three calls the stored `FetchTxStatus` is still `Error{…}` (not a terminal state) and that the fourth call selects the same provider as the first call, confirming the cycle.
4. Assert that every call returns `TryFetchResult::ToFetch` whose resolved value is `FetchResult::Error(…)`, which `into_response` maps to `CheckTransactionRetriable::TransientInternalError`.
5. Confirm no terminal / irrecoverable status is ever stored, demonstrating the infinite cycle.
