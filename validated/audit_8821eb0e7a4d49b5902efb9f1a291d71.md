## Title
Cycles Drained Without Work via Out-of-Bounds `read_canister_snapshot_data` on Public-Visibility Snapshots â€” (File: `rs/execution_environment/src/canister_manager.rs`)

## Summary
`read_snapshot_data` charges cycles upfront based on the requested `size` and records them in `consumed_cycles` before performing the memory bounds check. When `get_memory_chunk` returns `InvalidSubslice`, the error path in `execute_mgmt_operation_on_canister` rolls back canister state to `saved_canister` (restoring the original cycle balance) but then immediately re-applies `consumed_cycles`, permanently consuming cycles proportional to the requested size despite no data being read. Any unprivileged user can exploit this against canisters with `SnapshotVisibility::Public` at zero cost to themselves.

## Finding Description
In `read_snapshot_data` (`rs/execution_environment/src/canister_manager.rs`):

1. `validate_snapshot_visibility` at line 2572 passes for any caller when `SnapshotVisibility::Public` is set. The ingress filter at lines 247â€“261 also admits the message via the same check. [1](#0-0) 

2. `get_response_size` at lines 3027â€“3041 validates only that `size â‰¤ MAX_SLICE_SIZE_BYTES` (2,000,000 bytes). It does **not** validate `offset + size â‰¤ actual_memory_size`. [2](#0-1) 

3. Cycles for `baseline_instructions + size` are immediately consumed from the canister and recorded in `consumed_cycles` at lines 2575â€“2593, before any bounds check occurs. [3](#0-2) 

4. `get_memory_chunk` at lines 466â€“468 performs the real bounds check (`offset.saturating_add(size) > page_map_size_bytes`) and returns `Err(CanisterSnapshotError::InvalidSubslice)`. [4](#0-3) 

5. The error propagates via `chunk?` at line 2629 back to `execute_mgmt_operation_on_canister`. [5](#0-4) 

6. In the error branch of `execute_mgmt_operation_on_canister` (lines 650â€“659), `*canister = saved_canister` restores the pre-call state (including the original cycle balance), then `consumed_cycles.apply(canister, ...)` immediately re-charges the full size-proportional cost onto the restored canister. The net result is a permanent cycle deduction with no corresponding work. [6](#0-5) 

The `saved_canister` snapshot is taken at line 640, before `op` is called, so it captures the pre-charge balance. The rollback + re-apply sequence is designed for operations that do partial work before failing, but here zero work is done â€” the bounds check fails before any memory is accessed. [7](#0-6) 

## Impact Explanation
An attacker can loop calls with `offset = memory_size - 1, size = 2_000_000` against any canister with `SnapshotVisibility::Public`. Each call drains approximately 2,000,000 instruction-cycles from the target canister while returning an error. The ingress induction fee is charged to the target canister, not the sender, so the attacker's cost is zero. Repeated calls drain the target canister's cycle balance to zero, permanently halting its execution. This constitutes an **application/platform-level DoS** matching the High ($2,000â€“$10,000) impact tier.

## Likelihood Explanation
`SnapshotVisibility::Public` is a documented, supported setting (not the default, but explicitly available). The attack requires no special privileges, no tooling beyond standard ingress calls, and is rate-limited only by ingress throughput â€” not by any per-call cost to the attacker. The exploit is deterministic and locally reproducible. [8](#0-7) 

## Recommendation
Move the bounds check **before** the cycle charge in `read_snapshot_data`. Specifically, validate `offset + size â‰¤ actual_memory_size` immediately after `get_response_size` and before calling `consume_cycles_for_management_canister_instructions`. Alternatively, treat `InvalidSubslice` as a validation error (analogous to `SliceTooLarge`) and do not record it in `consumed_cycles` â€” i.e., return early before line 2581 when the requested range is provably out of bounds.
