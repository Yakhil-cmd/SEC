Audit Report

## Title
Unprivileged Caller Can Force-Register Another User's ICP Participation in SNS Swap - (`rs/sns/swap/canister/canister.rs`)

## Summary

The `refresh_buyer_tokens` endpoint in the SNS Swap canister accepts an arbitrary `buyer` principal in the request body with no check that the caller matches the specified buyer. Any unprivileged caller can supply a victim's principal after the victim has transferred ICP to their swap subaccount, causing the swap canister to register the victim as a committed participant. Once registered and the swap commits, the victim's ICP is swept to SNS governance and the victim receives SNS tokens — a forced financial outcome they cannot reverse.

## Finding Description

In `rs/sns/swap/canister/canister.rs` at lines 128–143, the `refresh_buyer_tokens` update method resolves the buyer principal directly from the caller-supplied `arg.buyer` field with no authorization check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // ← attacker-controlled
};
```

The resolved `p` is passed directly to `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs`. That function reads the ICP balance of `swap_canister_subaccount[p]` on the ICP ledger (lines 1153–1163) and, if it meets the minimum, writes `p` into `self.buyers` (lines 1285–1288), permanently registering the victim as a committed participant.

The only other check is `validate_confirmation_text` (lines 1149–1150, implemented at lines 363–384 of `swap.rs`). This check compares the supplied text against the SNS init payload's `confirmation_text` field. That field is optional (many SNS swaps omit it entirely), and when set, its value is public on-chain — it provides no meaningful access control.

The integration test at `rs/tests/nns/sns/lib/src/sns_deployment.rs` lines 919–926 explicitly demonstrates the attack path: `default_sns_agent` (a different identity) calls `refresh_buyer_tokens(Some(wealthy_user_identity.principal_id), None)` after the wealthy user has transferred ICP, and the call succeeds. The same test at lines 807–818 confirms that the call fails before the transfer (balance is zero, below minimum), establishing that the precondition is the ICP transfer.

The `error_refund_icp` function (`swap.rs` lines 1925–1932) is gated on the swap being in ABORTED or COMMITTED state — it cannot be called while the swap is OPEN. Once the victim is registered in `self.buyers` and the swap commits, `sweep_icp` transfers the victim's ICP to SNS governance. After that, the victim's subaccount is empty and `error_refund_icp` returns a transfer error (confirmed by the test at `swap.rs` lines 1927–1943). The victim has no recourse.

## Impact Explanation

This matches the High bounty impact: **"Unauthorized access to... ledger... or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."**

A victim who transferred ICP to the swap subaccount but changed their mind before calling `refresh_buyer_tokens` is forced into the swap by a third party. If the swap commits, the victim's ICP is permanently transferred to SNS governance and replaced with SNS tokens. The victim cannot recover their ICP. The SNS tokens received may be worth less than the ICP contributed (e.g., if the SNS token price declines post-swap), and the victim is denied the ability to exit before commitment. This is a concrete, irreversible financial harm caused by an unprivileged caller with no special access.

## Likelihood Explanation

The precondition — victim has transferred ICP to the swap subaccount but not yet called `refresh_buyer_tokens` — exists in normal usage. The ICP ledger is a public, queryable canister; an attacker can monitor it for transfers to swap subaccounts (the subaccount derivation from a principal is deterministic and public). The attack requires no key material, no privileged access, and no on-chain majority. It is repeatable across any number of victims during any open SNS swap. The confirmation text provides no barrier when absent (the majority of SNS swaps), and is trivially bypassed when present since the text is readable from the swap canister's public state.

## Recommendation

Add a caller authorization check in `refresh_buyer_tokens` in `rs/sns/swap/canister/canister.rs`. When `arg.buyer` is non-empty, verify that the caller matches the specified buyer before proceeding:

```rust
#[update]
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let specified = PrincipalId::from_str(&arg.buyer).unwrap();
        if specified != caller {
            panic!("Caller is not authorized to refresh tokens for a different buyer");
        }
        specified
    };
    // ... rest unchanged
}
```

This preserves the existing behavior for callers who omit the `buyer` field while preventing third parties from registering participation on behalf of others.

## Proof of Concept

The existing integration test at `rs/tests/nns/sns/lib/src/sns_deployment.rs` lines 919–926 already constitutes a proof of concept. It calls `refresh_buyer_tokens(Some(wealthy_user_identity.principal_id), None)` from `default_sns_agent` (a different identity) after the wealthy user has transferred ICP, and asserts the call succeeds (`unwrap()`). To demonstrate the full attack impact:

1. Deploy a local SNS swap (no `confirmation_text` set).
2. Have Alice transfer `min_participant_icp_e8s` ICP to her swap subaccount.
3. Before Alice calls `refresh_buyer_tokens`, have Bob call `refresh_buyer_tokens({ buyer: alice_principal, confirmation_text: null })`.
4. Verify Alice appears in `self.buyers` via `get_buyer_state`.
5. Advance time past the swap deadline with sufficient participation to commit.
6. Call `finalize_swap`; observe `sweep_icp` transfers Alice's ICP to SNS governance.
7. Attempt `error_refund_icp` for Alice; observe it fails with a transfer error (subaccount empty).
8. Verify Alice received SNS tokens she did not consent to receive.
