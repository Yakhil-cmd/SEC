Audit Report

## Title
Unbounded Neuron Iteration in `compute_ballots_for_new_proposal` Can Exhaust Per-Message Instruction Limit, Permanently Blocking SNS Proposal Submission - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance canister's `compute_ballots_for_new_proposal` performs a synchronous, unbounded `for (k, v) in self.proto.neurons.iter()` loop over the entire heap-resident neuron map on every `make_proposal` call. No instruction-limit guard exists. Once the neuron population grows large enough that this loop consumes the 40-billion-instruction per-message cap, every subsequent `make_proposal` call traps, permanently preventing new governance proposals from being submitted to the SNS DAO.

## Finding Description
The unbounded loop is at [1](#0-0)  — it iterates every entry in `self.proto.neurons`, computes voting power, and inserts a `Ballot` into a `BTreeMap`. There is no early exit, no instruction counter check, and no batching. The function returns `Err` only for voting-power overflow or an empty electoral roll; it has no path that gracefully aborts when approaching the instruction limit.

This function is called synchronously inside the externally reachable `make_proposal` update handler: [2](#0-1) 

The hard per-message instruction cap on application subnets is 40 billion: [3](#0-2)  Deterministic Time Slicing (DTS) allows execution to span multiple 2 B-instruction slices, but the 40 B total message cap is still enforced; DTS does not raise the ceiling.

The NNS governance already identified and fixed this exact pattern by replacing the equivalent loop with a pre-computed voting-power snapshot: [4](#0-3)  The SNS governance has not received the same fix and still uses raw heap `BTreeMap` iteration. No `noop_self_call_if_over_instructions` guard or equivalent exists anywhere in the SNS `make_proposal` path.

## Impact Explanation
Once the instruction limit is exceeded, **every** `make_proposal` call traps. No new governance proposals — parameter changes, treasury disbursements, canister upgrades — can be submitted. The SNS DAO is effectively frozen. Recovery requires an out-of-band canister upgrade (e.g., NNS root intervention), which may be unavailable for fully decentralized SNS instances. This matches the allowed High impact: *Application/platform-level DoS with concrete user and protocol harm*.

## Likelihood Explanation
Any unprivileged principal holding a valid SNS neuron can call `make_proposal`. Neuron count grows organically as users stake tokens; no adversarial action is required. The NNS benchmark projects that even the optimized stable-memory path approaches 25 billion instructions at scale: [5](#0-4)  The SNS path uses the slower heap `BTreeMap`, making it more instruction-intensive per neuron. Popular SNS projects accumulating tens of thousands of neurons are a realistic scenario, and the condition is permanent once triggered.

## Recommendation
Replace the unbounded loop in `compute_ballots_for_new_proposal` with the same voting-power snapshot mechanism already used by NNS governance (`compute_voting_power_snapshot_for_standard_proposal`), migrating SNS neurons to stable memory as NNS did. As a shorter-term mitigation, add an instruction-counter guard (analogous to `noop_self_call_if_over_instructions`) so that if the limit is approached the call fails gracefully with a descriptive error rather than trapping, and enforce a hard cap on `self.proto.neurons.len()` inside the neuron-creation path (`claim_swap_neurons` and related entry points).

## Proof of Concept
1. Deploy an SNS with default parameters on a local replica or PocketIC instance.
2. Programmatically stake tokens and claim neurons for a large number of principals until `self.proto.neurons.len()` is large enough that iterating all entries in one message exceeds 40 billion instructions (extrapolate from the NNS benchmark: ~25 B instructions at NNS max with the faster stable-memory path; the SNS heap path will hit the ceiling at a lower neuron count).
3. Call `make_proposal` from any valid neuron holder.
4. Observe the call rejected with `CanisterInstructionLimitExceeded`.
5. Repeat — every subsequent `make_proposal` call fails identically, confirming the governance canister is permanently unable to accept new proposals.
A deterministic integration test can instrument instruction counts via the replica's `performance_counter` system API or use PocketIC's cycle-counting hooks to confirm the loop crosses the 40 B threshold at a specific neuron count.
