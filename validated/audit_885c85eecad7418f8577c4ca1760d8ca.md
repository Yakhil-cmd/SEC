Audit Report

## Title
Unprivileged Guard Exhaustion DoS in `update_balance` — (`rs/bitcoin/ckbtc/minter/src/guard.rs`, `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary
The `update_balance` endpoint allows callers to specify an arbitrary `args.owner` principal with no requirement that it match the ingress caller. The global `PendingBalanceUpdates` guard set is capped at `MAX_CONCURRENT = 100` and keyed by `Account` (owner + subaccount). A single non-anonymous attacker can exhaust this shared pool by submitting 100 concurrent calls with 100 distinct `args.owner` values, each holding a guard slot while suspended at the `get_utxos` inter-canister call, causing all subsequent `update_balance` calls from any principal to receive `UpdateBalanceError::TemporarilyUnavailable` indefinitely.

## Finding Description
In `guard.rs`, `MAX_CONCURRENT` is set to 100 and the guard set is a flat `BTreeSet<Account>` with no per-caller partitioning: [1](#0-0) [2](#0-1) 

In `update_balance.rs`, the guarded account is constructed directly from the caller-supplied `args.owner` field, with no enforcement that it equals the ingress caller: [3](#0-2) 

The only caller validation is `check_anonymous_caller()`, which only rejects the anonymous principal: [4](#0-3) 

The guard is acquired synchronously and then held across the `get_utxos` inter-canister `await`: [5](#0-4) 

On the IC, a canister processes other queued ingress messages while suspended at an `await` point. An attacker submits 100 calls each with a distinct `args.owner`; each acquires a unique guard slot and suspends at `get_utxos`. Once all 100 slots are occupied, `accounts.len() >= MAX_CONCURRENT` is true for every new call, returning `GuardError::TooManyConcurrentRequests`, mapped to: [6](#0-5) 

The attacker sustains the DoS by re-submitting calls as guards are released, keeping the set perpetually full.

## Impact Explanation
This is an application-level DoS on the ckBTC minter's deposit path — a financial integration explicitly in scope. Legitimate depositors cannot notify the minter of their Bitcoin deposits and receive `TemporarilyUnavailable` for the duration of the attack. No funds are directly stolen, but deposit finality is blocked for all users. This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS not based on raw volumetric DDoS**, and **Significant ck-token security impact with concrete user or protocol harm**.

## Likelihood Explanation
The attack requires only a single non-anonymous principal and exactly 100 ingress messages — well within the IC's per-canister ingress queue capacity. No privileged access, governance majority, or key material is needed. The attacker pays `get_utxos` cycle costs, which are modest. The attack is fully repeatable and can be sustained indefinitely by cycling through distinct accounts as guards are released.

## Recommendation
Bind the guarded account to the ingress caller: require that `args.owner`, if specified, equals `caller` (or a principal the caller is explicitly authorized to act on behalf of). Alternatively, partition the concurrent guard limit by caller principal — enforce a per-caller cap (e.g., 1–5 concurrent calls per principal) in addition to the global cap, so no single principal can exhaust the shared pool.

## Proof of Concept
```rust
// State-machine test sketch
for i in 0..100u64 {
    env.submit_ingress_as(
        attacker_principal,
        minter_id,
        "update_balance",
        Encode!(&UpdateBalanceArgs {
            owner: Some(Principal::from_slice(&i.to_le_bytes())), // 100 distinct owners
            subaccount: None,
        }).unwrap(),
    );
}
// All 100 calls are now in-flight, each holding a PendingBalanceUpdates guard
// while awaiting get_utxos from the Bitcoin canister.

// Victim's call:
let res = env.execute_ingress_as(
    victim_principal,
    minter_id,
    "update_balance",
    Encode!(&UpdateBalanceArgs { owner: None, subaccount: None }).unwrap(),
).unwrap();
let decoded = Decode!(&res.bytes(), Result<Vec<UtxoStatus>, UpdateBalanceError>).unwrap();
assert!(matches!(decoded, Err(UpdateBalanceError::TemporarilyUnavailable(_))));
```
The existing unit test in `guard.rs` at lines 141–162 already demonstrates that `MAX_CONCURRENT` distinct accounts exhaust the guard set and the 101st call returns `TooManyConcurrentRequests`, confirming the mechanism is exactly as described. [7](#0-6)

    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }
