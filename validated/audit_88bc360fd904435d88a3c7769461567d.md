### Title
SNS Swap Neuron ID Pre-Registration Can Block Finalization for Direct Participants - (File: `rs/sns/governance/src/governance.rs`, `rs/sns/swap/src/swap.rs`)

### Summary
SNS neuron IDs for direct swap participants are computed deterministically from `(buyer_principal, memo)` using `compute_neuron_staking_subaccount_bytes`. The swap reserves memos in `[NEURON_BASKET_MEMO_RANGE_START, SALE_NEURON_MEMO_RANGE_END]` = `[1_000_000, 10_000_000]` for swap-issued neurons. While developer neurons are validated against this range at SNS init time, the SNS governance `claim_or_refresh_neuron_by_memo_and_controller` endpoint contains no such check. A direct swap participant can pre-register a neuron using their own principal and a memo in the reserved range, causing `claim_swap_neurons` to fail for their basket during finalization.

### Finding Description

SNS neuron IDs are computed deterministically:

```rust
// rs/nervous_system/common/src/ledger.rs:6-7
pub fn compute_neuron_staking_subaccount_bytes(controller: PrincipalId, nonce: u64) -> [u8; 32] {
    compute_neuron_domain_subaccount_bytes(controller, b"neuron-stake", nonce)
}
``` [1](#0-0) 

During swap finalization, `create_sns_neuron_basket_for_direct_participant` assigns each direct participant a basket of neurons with memos starting at `NEURON_BASKET_MEMO_RANGE_START = 1_000_000`:

```rust
// rs/sns/swap/src/swap.rs:89-90
pub const NEURON_BASKET_MEMO_RANGE_START: u64 = 1_000_000;
pub const SALE_NEURON_MEMO_RANGE_END: u64 = 10_000_000;
``` [2](#0-1) 

The basket for a direct participant uses memos `[memo_offset, memo_offset + basket_count - 1]` where `memo_offset = NEURON_BASKET_MEMO_RANGE_START`: [3](#0-2) 

The protection against memo collisions is only enforced for **developer neurons** at SNS initialization:

```rust
// rs/sns/init/src/distributions.rs:220-227
if NEURON_BASKET_MEMO_RANGE_START <= *memo && *memo <= SALE_NEURON_MEMO_RANGE_END {
    return Err(format!(
        "Error: Developer neuron with controller {} cannot have a memo in the range {} to {}",
        ...
    ));
}
``` [4](#0-3) 

However, the SNS governance `claim_or_refresh_neuron_by_memo_and_controller` — callable by any user via `manage_neuron` — performs **no such range check**:

```rust
// rs/sns/governance/src/governance.rs:4210-4227
async fn claim_or_refresh_neuron_by_memo_and_controller(
    &mut self,
    caller: &PrincipalId,
    memo_and_controller: &MemoAndController,
) -> Result<(), GovernanceError> {
    let controller = memo_and_controller.controller.unwrap_or(*caller);
    let memo = memo_and_controller.memo;
    let nid = NeuronId::from(ledger::compute_neuron_staking_subaccount_bytes(controller, memo));
    match self.get_neuron_result(&nid) {
        Ok(neuron) => { ... self.refresh_neuron(&nid).await }
        Err(_) => self.claim_neuron(nid, &controller).await,
    }
}
``` [5](#0-4) 

Similarly, `new_neuron_id` in SNS governance only checks for existence, not for reserved memo ranges: [6](#0-5) 

### Impact Explanation

A direct swap participant who pre-registers a neuron with their own principal and memo `= NEURON_BASKET_MEMO_RANGE_START` causes the swap's `claim_swap_neurons` call to SNS governance to return `AlreadyExists` for that neuron. The swap canister counts this as a `failure` in the `SweepResult`. The participant's SNS tokens, already transferred to the SNS governance subaccount, become stuck — the participant loses their swap allocation permanently since the neuron cannot be re-claimed and the swap has no recovery path for this case.

### Likelihood Explanation

The attack requires the participant to call `manage_neuron` (`ClaimOrRefreshNeuron`) on the SNS governance canister with a memo in `[1_000_000, 10_000_000]` before swap finalization. The exploitability depends on whether the SNS governance `PreInitializationSwap` mode permits `ClaimOrRefreshNeuron` calls — which is plausible since users need to claim developer/airdrop neurons during this phase. The attacker is any direct swap participant (unprivileged ingress sender), and the target neuron ID is fully predictable from public information (`buyer_principal` and `NEURON_BASKET_MEMO_RANGE_START`). The attacker can only grief their own swap allocation, limiting the blast radius to self-inflicted loss, but this could be used to grief a specific targeted principal if the attacker can predict their participation.

### Recommendation

Add a memo range guard in SNS governance's `claim_or_refresh_neuron_by_memo_and_controller` (and any other user-facing neuron-claiming path) that rejects memos in `[NEURON_BASKET_MEMO_RANGE_START, SALE_NEURON_MEMO_RANGE_END]`, mirroring the check already present in `rs/sns/init/src/distributions.rs`. Alternatively, the swap's `claim_swap_neurons` handler in SNS governance should treat `AlreadyExists` as a recoverable condition (e.g., by verifying the existing neuron belongs to the correct swap and merging the stake) rather than a hard failure.

### Proof of Concept

1. Attacker participates in an SNS swap as a direct buyer with principal `P`, contributing ICP.
2. Before swap finalization, attacker calls `manage_neuron` on the SNS governance canister with `ClaimOrRefreshNeuron { controller: P, memo: 1_000_000 }` and first transfers the minimum stake to `subaccount = compute_neuron_staking_subaccount_bytes(P, 1_000_000)` of SNS governance.
3. SNS governance creates a neuron with `NeuronId = hash(P, 1_000_000)`.
4. Swap finalizes; `create_sns_neuron_basket_for_direct_participant` generates a recipe for `P` with `memo = NEURON_BASKET_MEMO_RANGE_START = 1_000_000`, producing the same `NeuronId`.
5. The swap calls `claim_swap_neurons` on SNS governance; governance returns `AlreadyExists` for that neuron ID.
6. The swap records a `failure` for that recipe. The SNS tokens transferred to `subaccount(P, 1_000_000)` of SNS governance are now inaccessible — the attacker's swap allocation is permanently lost. [7](#0-6) [8](#0-7)
