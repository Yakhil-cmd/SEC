Audit Report

## Title
Unbounded `ingress_expiries` Loop Allows Unauthenticated OOM DoS — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary

The ICP Rosetta server's `construction_payloads` handler accepts client-controlled `ingress_start` and `ingress_end` values from the JSON request body and enters a `while` loop that pushes one `u64` per step with no prior bounds validation. Supplying `ingress_start=0` and `ingress_end=u64::MAX` causes approximately 153 billion iterations, exhausting process memory and crashing the server. The ICRC1 Rosetta counterpart contains explicit guards that the ICP handler is missing.

## Finding Description

**Root cause:** In `construction_payloads.rs` lines 99–107, the loop iterates from `ingress_start` to `ingress_end` in steps of `interval` (120 s = 120,000,000,000 ns), pushing one `u64` per iteration into `ingress_expiries` with no pre-loop size check:

```rust
let mut ingress_expiries = vec![];
let mut now = ingress_start;
while now < ingress_end {
    let ingress_expiry = (now + ic_limits::MAX_INGRESS_TTL.saturating_sub(ic_limits::PERMITTED_DRIFT))
        .as_nanos_since_unix_epoch();
    ingress_expiries.push(ingress_expiry);
    now += interval;
}
``` [1](#0-0) 

`interval` is computed as `MAX_INGRESS_TTL(300s) − PERMITTED_DRIFT(60s) − 120s = 120s`: [2](#0-1) 

Both `ingress_start` and `ingress_end` are `Option<u64>` fields deserialized directly from the JSON request body with no range validation: [3](#0-2) 

They are mapped directly to `ic_types::time::Time` values and used as loop bounds without any window-size check: [4](#0-3) 

**Exploit path:** An attacker sends a single unauthenticated HTTP POST to `/construction/payloads` with `metadata.ingress_start=0` and `metadata.ingress_end=18446744073709551615`. The loop runs ≈153,722,867,280 iterations, each allocating 8 bytes, totalling ≈1.23 TB of heap demand. The OS OOM killer terminates the process after exhausting available RAM (typically a few GB), well before the loop completes.

**Missing guard:** The ICRC1 Rosetta implementation contains explicit pre-loop guards that the ICP handler lacks entirely: [5](#0-4) 

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS, crash... or subnet availability impact not based on raw volumetric DDoS"* and *"Significant... Rosetta... infrastructure security impact with concrete user or protocol harm."* The ICP Rosetta server is a financial integration component used by exchanges and infrastructure operators for ICP transfers and neuron management. A crash takes down the entire server, blocking all dependent operations until manual restart.

## Likelihood Explanation

The attack requires a single unauthenticated HTTP POST with two integer fields set to extreme values. No credentials, keys, or special access are needed. The endpoint is publicly exposed by exchanges and infrastructure operators. The exploit is trivially reproducible and repeatable — the server can be crashed again immediately after restart.

## Recommendation

Add a maximum ingress window size check before the loop, mirroring the ICRC1 implementation. For example:

```rust
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 60 * 60);
if ingress_end > ingress_start + MAX_INGRESS_WINDOW {
    return Err(ApiError::invalid_request(
        "ingress_end exceeds maximum allowed window from ingress_start",
    ));
}
if ingress_start >= ingress_end {
    return Err(ApiError::invalid_request(
        "ingress_start must be before ingress_end",
    ));
}
```

This caps the maximum number of loop iterations to `86400s / 120s = 720`, bounding `ingress_expiries` to at most 720 elements (~5.76 KB).

## Proof of Concept

**Local manual steps:**

1. Start the ICP Rosetta server locally.
2. Send the following request:

```http
POST /construction/payloads HTTP/1.1
Host: localhost:8080
Content-Type: application/json

{
  "network_identifier": { "blockchain": "Internet Computer", "network": "<network_id>" },
  "operations": [{ "operation_identifier": {"index": 0}, "type": "TRANSACTION", ... }],
  "public_keys": [{ "hex_bytes": "<valid_pubkey>", "curve_type": "edwards25519" }],
  "metadata": {
    "ingress_start": 0,
    "ingress_end": 18446744073709551615
  }
}
```

3. Observe the server process is killed by the OOM killer within seconds and returns no response.

**Unit test plan:** Add a test in `construction_payloads.rs` that calls `construction_payloads` with `ingress_start=0` and `ingress_end=u64::MAX` and asserts the function returns an `Err(ApiError)` rather than running the loop. Without the fix, this test will hang or OOM; with the fix, it returns immediately with an error.
