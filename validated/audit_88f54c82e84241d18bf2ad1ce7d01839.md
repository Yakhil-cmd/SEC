Audit Report

## Title
Silent `wasm_memory_threshold` Omission in `validate_and_render_manage_dapp_canister_settings` Enables Hidden Dapp Canister Mutations — (File: `rs/sns/governance/src/proposal.rs`)

## Summary

`validate_and_render_manage_dapp_canister_settings` omits `wasm_memory_threshold` from both the `no_change` guard and the human-readable proposal rendering, while the field is fully present in the governance protobuf struct, preserved through all conversions, and forwarded to the SNS Root canister at execution time. An SNS neuron holder can craft a `ManageDappCanisterSettings` proposal that visibly proposes one setting change while silently applying an arbitrary `wasm_memory_threshold` value to every listed dapp canister — a mutation voters are never shown.

## Finding Description

In `rs/sns/governance/src/proposal.rs`, `validate_and_render_manage_dapp_canister_settings` (lines 1801–1873) iterates over every optional field of `ManageDappCanisterSettings` to build the voter-facing rendered text and to determine whether at least one field is set. The fields handled are: `compute_allocation`, `memory_allocation`, `freezing_threshold`, `reserved_cycles_limit`, `log_visibility`, `snapshot_visibility`, and `wasm_memory_limit`. The field `wasm_memory_threshold` is absent from both checks.

The `ManageDappCanisterSettings` protobuf struct in `rs/sns/governance/src/gen/ic_sns_governance.pb.v1.rs` (lines 876–898) defines `wasm_memory_threshold` as a first-class `Option<u64>` field at tag 8, on equal footing with all other fields.

The `From<pb::ManageDappCanisterSettings> for pb_api::ManageDappCanisterSettings` conversion in `rs/sns/governance/src/pb/conversions.rs` (lines 798–826) explicitly copies `wasm_memory_threshold` through the conversion chain. `perform_manage_dapp_canister_settings` in `rs/sns/governance/src/governance.rs` (lines 3214–3218) then converts the governance struct into a `ManageDappCanisterSettingsRequest` via `ManageDappCanisterSettingsRequest::from(manage_dapp_canister_settings)` and dispatches it to the SNS Root canister. `ManageDappCanisterSettingsRequest` in `rs/sns/root/src/gen/ic_sns_root.pb.v1.rs` (lines 344–363) also carries `wasm_memory_threshold` at tag 8.

Exploit flow:
1. Attacker submits a `ManageDappCanisterSettings` proposal with `compute_allocation: Some(50)` and `wasm_memory_threshold: Some(u64::MAX)`.
2. `validate_and_render_manage_dapp_canister_settings` sees `compute_allocation` is set, sets `no_change = false`, renders only `"# Set compute allocation to: 50%"`, and returns `Ok`.
3. Voters see only the compute allocation change and approve.
4. On adoption, `perform_manage_dapp_canister_settings` forwards the full struct — including `wasm_memory_threshold = u64::MAX` — to the root canister, which applies it to every listed dapp canister.

A secondary consequence: a proposal with **only** `wasm_memory_threshold` set is incorrectly rejected with `"must change at least one value, all values are None"`, silently discarding a legitimate governance action.

## Impact Explanation

This is a concrete SNS governance authorization bypass: the on-chain vote is cast on an incomplete description of the state change being authorized. Voters have no protocol-level mechanism to detect the hidden field from the proposal text alone. Setting `wasm_memory_threshold` to `u64::MAX` on a dapp canister causes it to permanently signal a low-memory condition, which can trigger `on_low_wasm_memory` hooks, disrupt memory-aware logic, or cause unintended operational behavior — all without any voter awareness. This constitutes significant SNS governance security impact with concrete harm to dapp canisters governed by the SNS, fitting the **High** tier: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation

The attacker must hold an SNS neuron with sufficient stake and dissolve delay to submit proposals, and the proposal must pass a majority vote. However, because the hidden field is entirely absent from the rendered proposal text, voters have no mechanism to detect the deception from the proposal alone — the attack exploits a protocol-level information gap, not social engineering. Any SNS neuron holder meeting the submission threshold can construct this proposal. Likelihood is **medium**.

## Recommendation

Add `wasm_memory_threshold` to both the `no_change` tracking block and the rendered proposal text in `validate_and_render_manage_dapp_canister_settings`, consistent with how every other optional field is handled:

```rust
if let Some(wasm_memory_threshold) = &manage_dapp_canister_settings.wasm_memory_threshold {
    render += &format!("# Set Wasm memory threshold to: {wasm_memory_threshold}\n");
    no_change = false;
}
```

Add a unit test analogous to `validate_and_render_manage_dapp_canister_settings_no_changes` (line 5168) that verifies a proposal with only `wasm_memory_threshold` set is accepted and that the rendered text includes the threshold value.

## Proof of Concept

Submit the following `ManageDappCanisterSettings` proposal via `manage_neuron` on any SNS governance canister:

```
ManageDappCanisterSettings {
    canister_ids: [<registered_dapp_canister_id>],
    compute_allocation: Some(50),        // visible in rendered text
    wasm_memory_threshold: Some(u64::MAX), // NOT in rendered text
    // all other fields: None
}
```

**Expected (correct)**: Rendered text includes both the compute allocation and the wasm memory threshold changes.

**Observed (buggy)**: `validate_and_render_manage_dapp_canister_settings` returns `Ok` with rendered text showing only `"# Set compute allocation to: 50%"`. On adoption, `perform_manage_dapp_canister_settings` forwards both fields to the root canister, which applies `wasm_memory_threshold = u64::MAX` to the dapp canister — a mutation no voter was shown.

A unit test can be added directly alongside the existing `validate_and_render_manage_dapp_canister_settings_no_changes` test at line 5168 of `rs/sns/governance/src/proposal.rs` to confirm the current broken behavior and verify the fix. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
