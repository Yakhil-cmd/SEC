Audit Report

## Title
Sandbox Process Can Crash Replica via `ftruncate` on Shared Backing fd â€” (`rs/replicated_state/src/page_map/page_allocator/mmap.rs`)

## Summary
The replica unconditionally passes the raw backing file descriptor of the `MmapBasedPageAllocatorCore` to each canister sandbox process via `SCM_RIGHTS`. The deployed SELinux policy grants `ic_canister_sandbox_t` the `write` permission on `ic_canister_mem_t` files, which under Linux is sufficient for `ftruncate(2)` on an open fd. A malicious canister's sandbox can call `ftruncate` to alter the file size, causing either a panic via `assert_eq!(file_len, self.file_len)` in `new_allocation_area` or a `SIGBUS` on the next replica access to a truncated mapped page â€” both crashing the replica process. The project's own SELinux documentation explicitly acknowledges this as an unmitigated security goal violation.

## Finding Description

**Step 1 â€” fd delivery to sandbox**

`MmapBasedPageAllocator::serialize` at `rs/replicated_state/src/page_map/page_allocator/mmap.rs:237-248` packages `core.file_descriptor` into a `PageAllocatorSerialization`. The `EnumerateInnerFileDescriptors` impl at `rs/canister_sandbox/src/protocol/sbxsvc.rs:141-145` pushes this fd into the SCM_RIGHTS vector, delivering it to the sandbox process over the Unix-domain socket transport.

**Step 2 â€” SELinux permits `ftruncate`**

`ic-os/components/guestos/selinux/ic-node/ic-node.te:316` contains:
```
allow ic_canister_sandbox_t ic_canister_mem_t : file { map read write getattr };
```
Under Linux SELinux, `ftruncate(2)` on an already-open fd requires only the `write` file permission â€” there is no separate `truncate` class for the `file` object class. No `F_ADD_SEALS` / `F_SEAL_SHRINK` sealing is applied to the memfd before it is handed off; the grep for `F_ADD_SEALS`, `F_SEAL`, and `memfd_create` in `rs/replicated_state/` confirms no sealing is present.

**Step 3 â€” Panic path via assert**

`rs/replicated_state/src/page_map/page_allocator/mmap.rs:583-593`:
```rust
fn new_allocation_area(&mut self) -> AllocationArea {
    ...
    let file_len = unsafe { get_file_length(self.file_descriptor) };
    // Allocation is the only operation that modifies the file size.
    // Ensure that the file size did not change since the last allocation.
    assert_eq!(file_len, self.file_len);
```
If the sandbox calls `ftruncate` to any size other than the current `self.file_len` before the next allocation slow-path, `get_file_length` returns the modified size, the assert fires, and the replica process panics. No race window is required â€” the modification persists until the next allocation cycle.

**Step 4 â€” SIGBUS path**

If the sandbox truncates the file below an already-mapped region, the next replica read or write to any page in that range delivers `SIGBUS`, terminating the replica. `ic-os/guestos/docs/SELinux-Policy.adoc:263-265` states verbatim: *"this allows calling ftruncate on the state files. If replica has these files mmapped concurrently, then any access to a page that has been truncated will result in SIGBUS. This allows crashing the replica through sandbox."*

**Step 5 â€” Acknowledged, unmitigated**

`ic-os/guestos/docs/SELinux-Policy.adoc:323-336` lists this under "Security goal violations" with no fix deployed and candidate remedies (memfd sealing, architecture change) explicitly noted as not yet implemented.

## Impact Explanation

Crashing the replica process on a node removes that node from subnet consensus participation. Because the same canister executes on every replica in the subnet, a single deployed canister can trigger this crash on all replicas simultaneously, causing subnet-wide unavailability until each node restarts and catches up. This matches the allowed ICP bounty impact: **High â€” Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS** ($2,000â€“$10,000). The impact could escalate toward Critical if the simultaneous crash of all subnet replicas is demonstrated to prevent recovery within a consensus round.

## Likelihood Explanation

- Any unprivileged canister deployer can trigger this; no special privileges are required.
- The fd is unconditionally passed to every sandbox process for every canister.
- The SELinux policy as deployed allows `write` (and therefore `ftruncate`) on the shared files.
- The assert path requires no timing precision â€” a single `ftruncate` call before the next allocation slow-path suffices.
- The SIGBUS path requires only truncation below a mapped offset, trivially achievable.
- The project's own documentation confirms this is a known, unmitigated violation with no deployed fix.

## Recommendation

1. **Immediate**: After creating the memfd and before passing it to the sandbox, apply `fcntl(fd, F_ADD_SEALS, F_SEAL_SHRINK | F_SEAL_GROW)`. This prevents any `ftruncate` call from succeeding on the fd regardless of SELinux policy. The documentation already identifies this as a candidate remedy; the constraint noted (that expanding memory requires IPC) can be addressed by sealing only after the final size is established, or by using a separate unsealable fd in the replica for growth while passing a sealed duplicate to the sandbox.
2. **Medium-term**: Apply `fsetfilecon` to label the shared memfd with a dedicated type that explicitly denies `write` to `ic_canister_sandbox_t` while permitting `map`, `read`, and `getattr`. The documentation confirms `fsetfilecon` has been demonstrated to work for memfd files.
3. **Long-term**: Restructure so the replica does not keep the backing file mmapped while the sandbox holds a writable fd, as noted in the documentation's architectural remedy options.

## Proof of Concept

```rust
// Inside malicious canister native code (sandbox process context).
// The sandbox receives the page allocator fd via SCM_RIGHTS as `backing_fd`.

// Single-shot variant (no race needed):
// ftruncate once to 0; the assert fires on the very next allocation slow-path in the replica.
unsafe { libc::ftruncate(backing_fd, 0); }
// Result: next call to new_allocation_area in replica:
//   get_file_length() returns 0
//   assert_eq!(0, self.file_len) fires â†’ replica panics â†’ node down

// SIGBUS variant:
// ftruncate to a size smaller than the currently-mapped region.
// The next replica access to any page in the truncated range delivers SIGBUS â†’ replica killed.
unsafe { libc::ftruncate(backing_fd, 0); }
// No further action needed; SIGBUS is delivered on next page access in replica.
```

A local integration test can be written by spawning a thread that calls `ftruncate(fd, 0)` on the allocator's backing fd, then triggering a new allocation cycle in the replica-side allocator and observing the panic or signal.
