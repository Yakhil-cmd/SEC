Audit Report

## Title
Unpermissioned `refresh_buyer_tokens` Allows Anyone to Force SNS Swap Participation for Another Buyer — (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` update handler in the SNS Swap canister accepts an arbitrary `buyer` principal with no check that the caller equals the specified buyer. Any unprivileged ingress sender can call this endpoint with a victim's principal, forcing the victim's ICP (already sitting in the swap subaccount) to be registered as committed participation. Because `error_refund_icp` is only available after the swap closes, the victim's ICP is locked for the duration of the swap and, if the swap commits, the victim receives SNS tokens they did not consent to purchase.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 127–143, the handler resolves the buyer from the request argument with no authorization check:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()  // no caller == buyer check
    };
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
``` [1](#0-0) 

Inside `refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs` lines 1152–1163), the function reads the ICP balance from the victim's swap subaccount (which is controlled by the swap canister, not the victim) and registers it as committed participation: [2](#0-1) 

The ticket system at lines 1248–1272 does **not** prevent this attack. The code explicitly states: *"If there exists no ticket for the buyer, the payment flow will simply ignore the ticket."* Tickets are optional; their absence does not block `refresh_buyer_token_e8s` from proceeding. [3](#0-2) 

The `confirmation_text` check (`rs/sns/swap/src/swap.rs` lines 363–384) is not a security control. It compares caller-supplied text against the SNS-configured text stored in the init payload, which is publicly readable via `get_sale_parameters`. Any attacker can read and replay it verbatim. [4](#0-3) 

Critically, the victim cannot reclaim ICP from the swap subaccount directly — the subaccount is `Account { owner: swap_canister_id, subaccount: principal_to_subaccount(victim) }`, controlled by the swap canister. The only reclaim path is `error_refund_icp`, which is only available after the swap closes (committed or aborted), as documented in the function comment at line 1130: [5](#0-4) 

The capping logic at lines 1223–1225 means a victim who transferred more ICP than the remaining swap capacity will have only a portion accepted as participation, with the remainder locked in the subaccount until close: [6](#0-5) 

## Impact Explanation
This is a **High** severity finding. An unprivileged attacker can force any user who has transferred ICP to the swap subaccount into committed participation without their consent. The victim's ICP is locked for the duration of the swap (potentially days). If the swap commits, the victim receives SNS tokens at a clearing price they did not agree to. If the swap is near its ICP ceiling, the victim's participation is capped and excess ICP is locked until close. This constitutes unauthorized access to and forced disposition of user funds, matching the allowed impact: *"Significant SNS security impact with concrete user or protocol harm"* and *"Unauthorized access to governance assets or canister-controlled funds."*

## Likelihood Explanation
The attack requires zero privileged access. The attacker needs only the victim's principal ID (public, derivable from any on-chain interaction), the swap canister ID (public), and the `confirmation_text` if set (public, readable via `get_sale_parameters`). Many SNS swaps do not set a `confirmation_text`, making the attack even simpler. The attack is most impactful when the swap is near its `max_direct_participation_e8s` ceiling — an observable on-chain condition. The attack is repeatable against any user who has transferred ICP to the swap subaccount but has not yet called `refresh_buyer_tokens` themselves.

## Recommendation
Add a caller-equality check in the canister handler before dispatching to `refresh_buyer_token_e8s`:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let buyer = PrincipalId::from_str(&arg.buyer).unwrap();
        if buyer != caller {
            panic!("caller is not authorized to refresh tokens for another buyer");
        }
        buyer
    };
    ...
}
```

If third-party notification by protocol bots is desired for UX, restrict it to a whitelist of trusted canister callers (not arbitrary ingress senders), mirroring the pattern used in `authorize_caller_to_call_notify_create_canister_on_behalf_of_creator` in `rs/nns/cmc/src/main.rs`. [1](#0-0) 

## Proof of Concept
1. Victim transfers 100 ICP to `Account { owner: swap_canister_id, subaccount: principal_to_subaccount(victim_principal) }` on the ICP ledger.
2. Victim decides to reclaim their ICP (swap terms are unfavorable; swap is near its ceiling with only 50 ICP of capacity remaining).
3. Victim cannot directly transfer ICP back — the subaccount is controlled by the swap canister.
4. Attacker queries `get_sale_parameters` to read the public `confirmation_text` (or notes it is absent).
5. Attacker submits an ingress update call to `refresh_buyer_tokens` with `buyer = victim_principal.to_text()` and the correct `confirmation_text`.
6. The swap canister reads the victim's subaccount balance (100 ICP), caps accepted participation at 50 ICP (remaining capacity), and registers 50 ICP as the victim's committed participation.
7. Victim's 100 ICP is now locked in the swap subaccount until close; only 50 ICP is accepted as participation; the other 50 ICP requires `error_refund_icp` after close.
8. If the swap commits, victim receives SNS tokens for 50 ICP at the clearing price — participation they did not consent to.

A deterministic unit test can be written against `refresh_buyer_token_e8s` directly: construct a swap in OPEN state with a mock ledger returning a non-zero balance for a victim principal, call `refresh_buyer_token_e8s` with a different caller principal, and assert that the victim's buyer entry is created and their ICP is registered as committed participation.
