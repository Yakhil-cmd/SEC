### Title
SNS Governance Misconfigured `NervousSystemParameters` Can Permanently Lock Out All Proposal Submission and Voting - (`rs/sns/governance/src/governance.rs`, `rs/sns/governance/src/types.rs`)

### Summary

The SNS governance canister allows any neuron holder with `SubmitProposal` permission to pass a `ManageNervousSystemParameters` proposal that sets `neuron_minimum_dissolve_delay_to_vote_seconds` equal to `max_dissolve_delay_seconds`. Because the only validation is that the former does not *exceed* the latter (strict `>`), setting them equal is accepted. If all existing neurons have a dissolve delay strictly less than the new minimum (which is now the maximum possible delay), no neuron can vote or submit proposals, permanently bricking the SNS governance canister. There is no recovery path: no proposal can be submitted, no vote can be cast, and no parameter can be changed. Tokens locked in the SNS treasury and neurons are permanently inaccessible through governance.

### Finding Description

`NervousSystemParameters::validate_neuron_minimum_dissolve_delay_to_vote_seconds` only rejects the case where `neuron_minimum_dissolve_delay_to_vote_seconds > max_dissolve_delay_seconds`:

```rust
if neuron_minimum_dissolve_delay_to_vote_seconds > max_dissolve_delay_seconds {
    Err(...)
} else {
    Ok(())
}
```

Setting `neuron_minimum_dissolve_delay_to_vote_seconds == max_dissolve_delay_seconds` passes validation. Since `max_dissolve_delay_seconds` is the ceiling on how long any neuron's dissolve delay can be set, no existing neuron can have a dissolve delay equal to `max_dissolve_delay_seconds` unless it was explicitly set to that exact value before the parameter change. In practice, all neurons with dissolve delays below the new minimum are excluded from the electoral roll in `compute_ballots_for_new_proposal`:

```rust
if v.dissolve_delay_seconds(now_seconds) < min_dissolve_delay_for_vote {
    continue;
}
```

If the resulting `electoral_roll` is empty, proposal submission fails with `"No eligible voters."`. Since no proposal can be submitted, the parameter cannot be corrected via governance. The SNS instance becomes permanently unusable.

Additionally, the same attack surface exists via `reject_cost_e8s`: a proposal can set `reject_cost_e8s` higher than any neuron's stake, preventing any neuron from affording to submit a proposal (as shown in the existing integration test `test_neuron_configuration_needed_for_proposals`). The `ManageNervousSystemParameters` proposal validation does not check whether the new parameters would leave the current neuron population able to govern.

### Impact Explanation

An SNS governance canister can be permanently bricked — no proposals can be submitted, no votes can be cast, and no parameters can be changed. All SNS treasury funds (ICP, SNS tokens) and all staked neurons become permanently inaccessible through governance. The SNS root canister, which is controlled by SNS governance, cannot be upgraded or instructed to return dapp canisters to their fallback controllers. This is a governance authorization bug with permanent, irreversible impact on all token holders and dapp users of the affected SNS instance.

### Likelihood Explanation

The entry path is a legitimate `ManageNervousSystemParameters` proposal submitted by any neuron holder with `SubmitProposal` permission. This is a standard, unprivileged ingress call to the SNS governance canister. The scenario can occur through:
1. A malicious neuron holder who passes a proposal to set `neuron_minimum_dissolve_delay_to_vote_seconds = max_dissolve_delay_seconds`.
2. An accidental misconfiguration by a well-intentioned SNS community.

The proposal passes the existing on-chain validation because the boundary check is `>` rather than `>=` (relative to `max_dissolve_delay_seconds`). No special access is required beyond holding a neuron with sufficient voting power to pass a `ManageNervousSystemParameters` proposal (normal criticality: 3% of total voting power, 50% of exercised).

### Recommendation

1. **Tighten the validation bound**: In `validate_neuron_minimum_dissolve_delay_to_vote_seconds`, change the condition to `>=` so that `neuron_minimum_dissolve_delay_to_vote_seconds == max_dissolve_delay_seconds` is also rejected:

```rust
if neuron_minimum_dissolve_delay_to_vote_seconds >= max_dissolve_delay_seconds {
    Err(...)
}
```

2. **Add a live-neuron-population check**: Before accepting a `ManageNervousSystemParameters` proposal, verify that at least one existing neuron would remain eligible to vote under the proposed parameters. This mirrors the LlamaCore recommendation to check that remaining permissions support ongoing operation.

3. **Classify `ManageNervousSystemParameters` as a critical proposal**: Raise the voting threshold for this action to the critical level (20% of total, 67% of exercised), making it harder for a minority to pass a governance-bricking parameter change.

4. **Add a floor on `neuron_minimum_dissolve_delay_to_vote_seconds`**: Enforce a maximum value (e.g., no more than 50% of `max_dissolve_delay_seconds`) to prevent the parameter from being set so high that no neuron can qualify.

### Proof of Concept

**Attacker-controlled entry path**: An unprivileged ingress sender holding an SNS neuron with `SubmitProposal` and `Vote` permissions submits the following `ManageNeuron` → `MakeProposal` call to the SNS governance canister:

```
ManageNeuron {
  subaccount: <attacker_neuron_subaccount>,
  command: MakeProposal(Proposal {
    title: "Update parameters",
    action: ManageNervousSystemParameters(NervousSystemParameters {
      neuron_minimum_dissolve_delay_to_vote_seconds: Some(max_dissolve_delay_seconds), // e.g. 8 * ONE_YEAR_SECONDS
      ..Default::default()
    })
  })
}
```

**Why it passes validation**: `validate_neuron_minimum_dissolve_delay_to_vote_seconds` only rejects `neuron_minimum_dissolve_delay_to_vote_seconds > max_dissolve_delay_seconds`. Setting them equal passes. [1](#0-0) 

**Why it bricks governance**: After execution via `perform_manage_nervous_system_parameters`, the new parameters are stored. On the next proposal submission attempt, `compute_ballots_for_new_proposal` filters out all neurons whose dissolve delay is less than `min_dissolve_delay_for_vote` (now equal to `max_dissolve_delay_seconds`). Since no neuron can have a dissolve delay exceeding `max_dissolve_delay_seconds`, the electoral roll is empty and the call returns `"No eligible voters."`. [2](#0-1) [3](#0-2) 

**No recovery**: Since no proposal can be submitted (empty electoral roll), the parameter cannot be corrected. The SNS is permanently bricked. The code explicitly acknowledges this danger in the `RemoveNeuronPermissions` documentation but provides no analogous guard for parameter changes that eliminate all eligible voters. [4](#0-3)

        &mut self,
        proposed_params: NervousSystemParameters,
    ) -> Result<(), GovernanceError> {
        // Only set `self.proto.parameters` if "applying" the proposed params to the
        // current params results in valid params
        let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());

        log!(
            INFO,
            "Setting Governance nervous system params to: {:?}",
            &new_params
        );

        match new_params.validate() {
            Ok(()) => {
                self.proto.parameters = Some(new_params);
                Ok(())
            }

            // Even though proposals are validated when they are first made, this is still
            // possible, because the inner value of a ManageNervousSystemParameters
            // proposal is only valid with respect to the current
            // nervous_system_parameters() at the time when the proposal was first
            // made. If nervous_system_parameters() changed (by another proposal) since
            // the current proposal was first made, the current proposal might have become
            // invalid. Basically, this might occur if there are conflicting (concurrent)
            // proposals, but we expect this to be highly unusual in practice.
            Err(msg) => Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!(
                    "Failed to perform ManageNervousSystemParameters action, proposed \
                        parameters would lead to invalid NervousSystemParameters: {msg}"
                ),
            )),
        }
