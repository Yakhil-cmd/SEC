Audit Report

## Title
Third-Party Caller Can Bypass `confirmation_text` Consent Gate in SNS Swap Participation - (`rs/sns/swap/canister/canister.rs`)

## Summary
The `refresh_buyer_tokens` endpoint accepts a caller-supplied `buyer` principal without verifying that the caller is that principal. When an SNS configures a `confirmation_text` as an explicit per-user consent gate, any unprivileged attacker can supply the publicly visible confirmation text on behalf of a victim, committing the victim's already-deposited ICP to the swap without the victim's own acknowledgment. The code comment for `refresh_buyer_token_e8s` explicitly states "the caller of this function must accept this confirmation," but the implementation does not enforce caller-equals-buyer when `confirmation_text` is required.

## Finding Description

**No caller-vs-buyer identity check in the canister handler**

`rs/sns/swap/canister/canister.rs` L130–134 resolves the buyer from the request argument with no assertion that `caller_principal_id() == p`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // any caller, any buyer
};
``` [1](#0-0) 

The resolved principal and the caller-supplied `confirmation_text` are passed directly to `refresh_buyer_token_e8s`: [2](#0-1) 

**`validate_confirmation_text` checks only string equality, not caller identity**

`rs/sns/swap/src/swap.rs` L363–384 validates only that the supplied string matches the SNS-init-configured string. It does not verify that the caller is the buyer: [3](#0-2) 

Because `confirmation_text` is set in the public SNS init payload, it is readable by anyone.

**The doc comment contradicts the implementation**

The doc comment for `refresh_buyer_token_e8s` at L1123–1126 states: *"the caller of this function must accept this confirmation by sending the exact same text as an argument."* The code does not enforce this — the caller and the buyer can be different principals: [4](#0-3) 

**Participation is committed against the victim's subaccount balance**

`refresh_buyer_token_e8s` reads the ICP balance of the buyer's subaccount and records it as accepted participation: [5](#0-4) 

**Third-party callability confirmed in integration tests**

The integration test at `rs/tests/nns/sns/lib/src/sns_deployment.rs` L919–926 explicitly demonstrates that `default_sns_agent` (a different identity) successfully calls `refresh_buyer_tokens` for `wealthy_user_identity.principal_id`. This test passes `None` for `confirmation_text`, confirming the third-party design is intentional for the no-confirmation-text case, but it does not test or guard the confirmation-text bypass: [6](#0-5) 

The proto field comment at L844 ("If not specified, the caller is used") confirms third-party calling is a deliberate design choice, but the proto comment at L847–849 ("a participant should send the confirmation text") implies the participant themselves must supply it: [7](#0-6) 

## Impact Explanation

The `confirmation_text` feature is the only explicit per-user consent gate in the SNS swap participation flow. An attacker who knows a victim's principal (derivable from any on-chain interaction) and the confirmation text (readable from the public SNS init payload) can commit the victim's already-deposited ICP to the swap without the victim ever calling `refresh_buyer_tokens` themselves. Once committed, the ICP is locked until finalization; if the swap succeeds, the victim receives SNS tokens instead of ICP against their will. This constitutes a **significant SNS security impact with concrete user harm** — specifically, forced commitment of user funds bypassing the protocol's own consent mechanism.

This matches: **High ($2,000–$10,000) — Significant SNS security impact with concrete user or protocol harm.**

## Likelihood Explanation

The attack requires no privileged access, no leaked secrets, and no governance majority. The attacker needs only: (1) the victim's principal, which is public on-chain; (2) the `confirmation_text`, which is in the public SNS init payload; and (3) the victim to have deposited ICP into their swap subaccount, which is a normal step in the participation flow. Any user who deposits ICP and pauses before confirming is exposed for the entire window between their ICP transfer and their own call to `refresh_buyer_tokens`. The attack is trivially repeatable against any such user.

## Recommendation

Add a caller-identity check before accepting a non-empty `buyer` field in `rs/sns/swap/canister/canister.rs`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let requested = PrincipalId::from_str(&arg.buyer).unwrap();
    if requested != caller_principal_id() {
        panic!("caller is not the specified buyer");
    }
    requested
};
```

Alternatively, when `confirmation_text` is required by the SNS init config, reject any call where the resolved buyer differs from the authenticated caller, preserving the third-party-notification use case only for swaps without a confirmation text.

## Proof of Concept

1. Deploy an SNS swap in `Open` state with `confirmation_text = "I agree to the terms"` (public in init payload).
2. Victim transfers 5 ICP to `Account { owner: swap_canister_id, subaccount: principal_to_subaccount(victim_principal) }` on the ICP ledger, intending to review the terms before confirming.
3. Attacker (any principal) sends an ingress update:
   ```
   refresh_buyer_tokens({
     buyer: victim_principal.to_string(),
     confirmation_text: Some("I agree to the terms")
   })
   ```
4. The swap canister resolves `p = victim_principal`, calls `validate_confirmation_text` (passes — strings match), reads the 5 ICP balance from the victim's subaccount, and records the victim as a committed participant.
5. Victim's 5 ICP is now locked in the swap. The victim never called `refresh_buyer_tokens` themselves and never explicitly accepted the confirmation text through their own authenticated call.

A deterministic reproduction can be written as a unit test in `rs/sns/swap/tests/swap.rs` mirroring the existing `test_swap_participation_confirmation` test, but calling `refresh_buyer_token_e8s` with a `buyer` principal that differs from the simulated caller, supplying the correct `confirmation_text`, and asserting the participation is committed.
