Looking at the event hook dispatch path, I need to trace how `system_context_event_hook` and `interop_root_reporter_event_hook` are triggered and whether they enforce caller authorization.

### Title
Missing Caller Authorization in Event Hooks Enables Unprivileged Settlement Layer Chain ID Corruption and Fake Interop Root Injection - (File: `system_hooks/src/event_hooks/system_context.rs`, `system_hooks/src/event_hooks/interop_root_reporter.rs`)

---

### Summary

The event hooks `system_context_event_hook` and `interop_root_reporter_event_hook` perform privileged system state mutations (`update_settlement_layer_chain_id`, `add_interop_root`) without any caller authorization check. Unlike every call hook in the codebase — which explicitly validates `caller != AUTHORIZED_ADDRESS` before acting — the event hooks fire purely on event signature match, with no check on who caused the emitting contract to emit the event. Any unprivileged transaction sender who can cause the `SystemContext` contract (`0x800b`) or `L2InteropRootStorage` contract (`0x10008`) to emit the matching event signature triggers the privileged hook unconditionally.

---

### Finding Description

**Call hooks enforce caller identity; event hooks do not.**

Every call hook checks the `caller` field of `ExternalCallRequest` before performing any privileged action:

- `l1_messenger_hook`: `if caller != L1_MESSENGER_ADDRESS { return empty; }`
- `mint_base_token_hook`: `if caller != L2_BASE_TOKEN_ADDRESS { return empty; }`
- `set_bytecode_on_address_hook`: `if caller != CONTRACT_DEPLOYER_ADDRESS && caller != COMPLEX_UPGRADER_ADDRESS { return empty; }`
- `contract_deployer_temp_hook`: `if caller != COMPLEX_UPGRADER_ADDRESS { return empty; }` [1](#0-0) [2](#0-1) [3](#0-2) 

The event hooks have **no equivalent check**. `system_context_event_hook` receives a `caller_ee: u8` parameter that is immediately shadowed as `_caller_ee` and never used. The hook fires on any event emitted from `SYSTEM_CONTEXT_ADDRESS` (`0x800b`) that matches the `SettlementLayerChainIdUpdated` signature: [4](#0-3) 

Similarly, `interop_root_reporter_event_hook` fires on any `InteropRootAdded` event from `L2_INTEROP_ROOT_STORAGE_ADDRESS` (`0x10008`) with no emitter authorization: [5](#0-4) 

**How the hook is dispatched.** `System::emit_event` is called from the EVM `LOG` instruction with `address = self.address` (the currently executing contract). If `address.try_into_low()` matches a registered event hook key, `try_intercept_event` fires the hook unconditionally — no caller identity is passed or checked: [6](#0-5) [7](#0-6) 

**Attacker-controlled entry path.** The `SystemContext` contract at `0x800b` is a predeployed EVM contract. The `setSettlementLayerChainId` function is intended to be called only via service transactions (enforced by the bootloader's `SERVICE_DESTINATION_WHITELIST`): [8](#0-7) 

However, this whitelist only restricts *service transaction* routing. A regular L2 transaction can call any contract address directly. If the `SystemContext` Solidity contract does not enforce `msg.sender == BOOTLOADER_FORMAL_ADDRESS` (or equivalent) on `setSettlementLayerChainId`, an unprivileged L2 transaction sender can:

1. Call `SystemContext.setSettlementLayerChainId(attacker_chain_id)` directly
2. `SystemContext` emits `SettlementLayerChainIdUpdated(attacker_chain_id)` from address `0x800b`
3. `system_context_event_hook` fires unconditionally and calls `update_settlement_layer_chain_id(attacker_chain_id)` [9](#0-8) 

The same path applies to `L2InteropRootStorage` at `0x10008` for fake interop root injection.

**The hooks are registered without any authorization metadata:** [10](#0-9) [11](#0-10) 

---

### Impact Explanation

**`system_context_event_hook`:** Calls `update_settlement_layer_chain_id`, which writes to `NewSettlementLayerChainIdStorage`. This value is committed into `BatchOutput.settlement_layer_chain_id` and verified on the settlement layer. An attacker who triggers this first in a block:
- Blocks the legitimate service transaction from updating the chain ID (the storage enforces at-most-one update per block, returning an `internal_error` on a second call)
- Injects an arbitrary `settlement_layer_chain_id` into the batch output, causing proof verification failure or cross-chain misbehavior [12](#0-11) [13](#0-12) 

**`interop_root_reporter_event_hook`:** Calls `add_interop_root`, injecting attacker-controlled `(root, chain_id, block_or_batch_number)` tuples into the interop root rolling hash committed in `BatchOutput.interop_roots_rolling_hash`. Fake interop roots corrupt cross-chain message verification.

---

### Likelihood Explanation

The exploitability depends on whether the `SystemContext` and `L2InteropRootStorage` Solidity contracts enforce caller restrictions at the contract level. The ZKsync OS Rust code (the hooks themselves) provides **no such enforcement**. The call hooks demonstrate the correct pattern; the event hooks omit it entirely. If the system contracts rely solely on the bootloader's service-transaction routing for protection — without an on-contract `msg.sender` check — any L2 transaction sender can trigger the hooks.

---

### Recommendation

Add an emitter/caller authorization check to both event hooks, mirroring the pattern used in all call hooks. The hook receives `caller_ee` and the system has access to the emitting address via the hook registration key. At minimum, verify that the event was emitted in the context of a service transaction (e.g., by checking a flag set by the bootloader before executing service transactions), or add an explicit `msg.sender` guard to the `SystemContext` and `L2InteropRootStorage` Solidity contracts analogous to the `ignoreNonBootloader` modifier pattern already present in `DefaultAccountEra.sol`. [14](#0-13) 

---

### Proof of Concept

```
1. Deploy attacker EOA on the ZKsync OS chain.
2. Craft a regular L2 EIP-1559 transaction:
     to:   0x000000000000000000000000000000000000800b  (SystemContext)
     data: 0x040203e6 ++ abi.encode(uint256(999))     (setSettlementLayerChainId(999))
3. Submit transaction. If SystemContext lacks msg.sender guard:
   - SystemContext executes setSettlementLayerChainId(999)
   - Emits SettlementLayerChainIdUpdated(999) from 0x800b
   - system_context_event_hook fires (system_hooks/src/event_hooks/system_context.rs:61)
   - update_settlement_layer_chain_id(999) is called
4. Any subsequent legitimate service transaction calling setSettlementLayerChainId
   hits the "more than once" guard and returns internal_error, blocking the update.
5. BatchOutput.settlement_layer_chain_id = 999 (attacker-controlled),
   causing settlement-layer proof verification to fail or accept a wrong chain ID.
```

pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // First, ensure we're capturing the InteropRootAdded event
    if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
        return Ok(());
    }
    // Internal error if the data supplied doesn't match the expected value
    if data.len() != 96 {
        return Err(internal_error!("Interop root reporter event hook received bad data").into());
    }

    // Parse data
    let offset: u32 = match U256::from_be_slice(&data[..32]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad offset").into(),
            );
        }
    };
    // This event is part of the system, but we check it anyways
    if offset != 32 {
        return Err(internal_error!("Interop root reporter event hook received bad offset").into());
    }

    let len: u32 = match U256::from_be_slice(&data[32..64]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad length").into(),
            );
        }
    };
    // It should have exactly one side
    if len != 1 {
        return Err(internal_error!("Interop root reporter event hook received bad length").into());
    }
    // Validate topics length
    if topics.len() != 3 {
        return Err(internal_error!("Interop root reporter event hook received bad topics").into());
    }

    let root = Bytes32::from_array(data[64..96].try_into().unwrap());
    let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
    system.io.add_interop_root(
        ExecutionEnvironmentType::NoEE,
        resources,
        InteropRoot {
            root,
            block_or_batch_number,
            chain_id,
        },
    )?;

    Ok(())

    /// Emit an event, potentially capturing some using an event hook.
    pub fn emit_event(
        &mut self,
        hooks: &mut HooksStorage<S, S::Allocator>,
        ee_type: ExecutionEnvironmentType,
        resources: &mut S::Resources,
        address: &<S::IOTypes as SystemIOTypesConfig>::Address,
        topics: &ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
        data: &[u8],
    ) -> Result<(), SystemError> {
        // First, emit the event using io subsystem
        self.io
            .emit_event(ee_type, resources, address, topics, data)?;

        // If successful, intercept event hook, if any
        if let Some(address_low) = address.try_into_low() {
            let _ = hooks.try_intercept_event(
                address_low,
                topics,
                data,
                ee_type as u8,
                self,
                resources,
            )?;
        }
        Ok(())
    }

    /// Intercepts events emitted from low addresses (< 2^32) and executes hooks
    /// stored under that address. If no hook is stored there, return `Ok(None)`.
    ///
    pub fn try_intercept_event(
        &mut self,
        address_low: u32,
        topics: &arrayvec::ArrayVec<
            <S::IOTypes as SystemIOTypesConfig>::EventKey,
            MAX_EVENT_TOPICS,
        >,
        data: &[u8],
        caller_ee: u8,
        system: &mut System<S>,
        resources: &mut S::Resources,
    ) -> Result<Option<()>, SystemError> {
        let Some(hook) = self.event_hooks.get(&address_low) else {
            return Ok(None);
        };
        hook.0(topics, data, caller_ee, system, resources)?;

        Ok(Some(()))
    }
