### Title
Pre-EIP-155 Legacy Transaction Cross-Chain Replay Bypasses Chain-ID Binding — (`engine/src/engine.rs`, `engine-transactions/src/legacy.rs`)

---

### Summary

Aurora Engine's `submit()` entrypoint accepts pre-EIP-155 legacy Ethereum transactions (those with `v = 27` or `v = 28`) and explicitly skips chain-ID validation for them. A transaction signed by a user on Ethereum mainnet (or any other EVM chain) without EIP-155 protection carries no chain binding in its signature. An unprivileged attacker who observes such a transaction on-chain can replay it verbatim against Aurora Engine and drain the victim's Aurora ETH balance, provided the victim's Aurora nonce matches the replayed transaction's nonce.

---

### Finding Description

**Root cause — two cooperating code paths:**

**Path 1 — `chain_id` is silently dropped for pre-EIP-155 transactions.**

In `engine-transactions/src/legacy.rs`, `LegacyEthSignedTransaction::chain_id()` returns `None` whenever `v ∈ {27, 28}`: [1](#0-0) 

When `NormalizedEthTransaction` is constructed from a `Legacy` variant, it propagates this `None` directly into the `chain_id` field: [2](#0-1) 

**Path 2 — the chain-ID guard in `submit_with_alt_modexp` is conditional on `Some`.**

In `engine/src/engine.rs`, the only chain-ID check is:

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [3](#0-2) 

When `chain_id` is `None` the entire guard is skipped. The transaction proceeds to nonce-check, gas-charge, and EVM execution with no binding to Aurora's chain ID.

**Historical context:** Version 2.4.0 of the engine explicitly rejected chain-ID-less transactions as a security improvement, but version 2.6.0 re-enabled them to support EIP-1820 deployment: [4](#0-3) [5](#0-4) 

The re-enablement removed the chain-ID binding for *all* pre-EIP-155 transactions, not just the EIP-1820 deployer address.

---

### Impact Explanation

**Critical — direct theft of user funds.**

An attacker who replays a victim's pre-EIP-155 Ethereum mainnet transaction on Aurora can cause arbitrary EVM execution (ETH transfer, contract call) to occur from the victim's Aurora account. If the victim holds ETH on Aurora, those funds can be transferred to an attacker-controlled address. The victim signed only for Ethereum mainnet; Aurora Engine executes the transaction as if the victim authorized it on Aurora.

---

### Likelihood Explanation

**Medium.**

Prerequisites:
1. The victim must hold ETH on Aurora at the same address they use on Ethereum mainnet (common — Aurora uses the same address space).
2. The victim must have broadcast a pre-EIP-155 legacy transaction on Ethereum mainnet (v=27 or v=28). Such transactions are still produced by older tooling, hardware wallets in legacy mode, and EIP-1820 deployment flows.
3. The victim's Aurora nonce must equal the nonce in the replayed transaction. This is most easily satisfied when the victim has never used Aurora (nonce = 0) and the replayed transaction also has nonce = 0 — a common first-transaction scenario.

The attacker's only cost is monitoring Ethereum mainnet for pre-EIP-155 transactions and submitting them to Aurora's `submit()` NEAR method, which is publicly callable by any NEAR account.

---

### Recommendation

Reject pre-EIP-155 transactions (those with `chain_id == None`) at the `submit_with_alt_modexp` validation stage, or restrict acceptance to a hard-coded allowlist of known chain-agnostic deployer addresses (e.g., the EIP-1820 deployer `0xa990077c3205cbDf861e17Fa532eeB069cE9fF96`). The guard at lines 1054–1059 of `engine/src/engine.rs` should be changed from an `if let Some(chain_id)` conditional to an unconditional rejection when `chain_id` is `None`, with a narrow exception for the EIP-1820 use case if required. [3](#0-2) 

---

### Proof of Concept

1. Alice holds 10 ETH on Aurora at address `0xALICE`. Her Aurora nonce is 0.
2. Alice previously sent a pre-EIP-155 legacy transaction on Ethereum mainnet: `{nonce:0, to:0xBOB, value:1 ETH, v:27, r:..., s:...}`. This transaction is publicly visible on Etherscan.
3. Attacker Bob encodes the identical RLP bytes and calls Aurora Engine's `submit()` NEAR method with them.
4. `LegacyEthSignedTransaction::chain_id()` returns `None` for `v=27`.
5. `NormalizedEthTransaction.chain_id` is `None`; the chain-ID guard at line 1055 is skipped entirely.
6. `check_nonce` passes (Alice's Aurora nonce is 0, matching the transaction).
7. The EVM executes: 1 ETH is transferred from Alice's Aurora account to `0xBOB` on Aurora.
8. Alice loses funds she never intended to move on Aurora. [6](#0-5) [7](#0-6) [8](#0-7)
