### Title
`set_eth_connector_contract_account` Can Be Called After User Deposits, Permanently Routing All Withdrawals to the Wrong Connector — (`engine/src/contract_methods/connector.rs`)

---

### Summary

The `set_eth_connector_contract_account` function allows the owner (or a private call) to replace the stored ETH connector account ID at any time, with no migration mechanism and no guard against re-setting after user funds have been deposited. Every user-facing withdrawal and token-transfer operation in the engine resolves its cross-contract call target by reading this stored account ID at call time. Replacing it after users have bridged ETH into Aurora causes all subsequent withdrawal attempts to be routed to the new connector, which holds none of the users' NEP-141 balances, temporarily freezing all bridged funds.

---

### Finding Description

`set_eth_connector_contract_account` in `engine/src/contract_methods/connector.rs` unconditionally overwrites the stored connector account ID with no check for whether one is already set and no migration step:

```rust
// engine/src/contract_methods/connector.rs  lines 418-438
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);                          // ← unconditional overwrite
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}
```

Every user-facing connector operation — `withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_unregister`, `storage_withdraw`, `storage_balance_of`, `ft_total_eth_supply_on_near`, `ft_balance_of`, and `ft_metadata` — resolves its cross-contract call target through the shared `return_promise` helper, which reads the stored account ID at call time:

```rust
// engine/src/contract_methods/connector.rs  lines 598-616

    mut io: I,
    env: &E,
    method: &str,
    args: Vec<u8>,
    deposit: Yocto,

    let promise_args = PromiseCreateArgs {
        target_account_id: get_connector_account_id(&io)?,   // ← read at call time
        ...
    };
    ...

```

The `withdraw` function itself is a direct example:

```rust

    io: I,

    ...

```

Additionally, `ft_on_transfer` uses the same stored account ID to distinguish base-token (ETH) deposits from ERC-20 transfers:

```rust
// engine/src/contract_methods/connector.rs  line 81

    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(...)
};
```

Replacing the connector account after users have deposited means:
1. All `withdraw` / `ft_transfer` / `storage_*` calls are dispatched to the new connector, which holds no user balances. The calls fail or silently no-op on the new contract.
2. Callbacks from the old connector via `ft_on_transfer` are no longer recognized as base-token deposits; they are misrouted as ERC-20 transfers, corrupting accounting for any in-flight deposits.

---

### Impact Explanation

**High — Temporary freezing of funds.**

All bridged ETH balances held in the old connector become inaccessible to users. No withdrawal, transfer, or storage operation can succeed because every one of them is dispatched to the wrong NEAR account. The freeze persists until the owner restores the original connector account ID. If the owner is unavailable or the key is lost, the freeze becomes permanent.

---

### Likelihood Explanation

The owner is the only required actor. A legitimate upgrade scenario — deploying a new version of the ETH connector contract and pointing the engine at it — is a realistic operational action. Without a migration mechanism, any such upgrade silently freezes all existing user balances. No attacker compromise is required; the owner's own routine administrative action is sufficient to trigger the impact.

---

### Recommendation

1. **Allow setting only once**: Check whether a connector account ID is already stored and revert if so, mirroring the pattern recommended in the original report.
2. **Migration mechanism**: Before switching the connector account, atomically transfer all NEP-141 balances from the old connector to the new one.
3. **Time-lock**: Introduce a delay between announcing a connector change and it taking effect, giving users time to withdraw before the switch.

---

### Proof of Concept

1. Users bridge ETH to Aurora. Their NEP-141 balances are recorded in `connector_v1.near`.
2. The engine's stored connector account ID is `connector_v1.near`.
3. Owner deploys `connector_v2.near` (empty, no user balances) and calls:
   ```
   set_eth_connector_contract_account({ account: "connector_v2.near", withdraw_serialize_type: Borsh })
   ```
   This executes `set_connector_account_id(io, &args.account)`, overwriting the stored key.
4. User calls `withdraw` on the engine. `return_promise` reads `get_connector_account_id` → `connector_v2.near` and dispatches `engine_withdraw` there.
5. `connector_v2.near` has no record of the user's balance; the call fails or silently succeeds without burning any tokens or initiating an ETH transfer.
6. User's bridged ETH is permanently stuck in `connector_v1.near` with no engine-level path to reach it.
