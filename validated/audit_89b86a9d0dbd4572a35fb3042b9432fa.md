### Title
Unbounded L1→L2 Priority Transaction Calldata Can Permanently Block the Priority Queue via `BlockNativeLimitReached` - (`basic_bootloader/src/bootloader/block_flow/zk/mod.rs`)

### Summary

An attacker can submit an L1→L2 priority transaction with a large `data` (calldata) field. When the sequencer attempts to include it in a block, the intrinsic native resource cost — which scales linearly with calldata length — can exceed `MAX_NATIVE_COMPUTATIONAL`. The `check_for_block_limits` function then reverts the transaction and does **not** add it to the `enforced_transaction_hashes_accumulator`. Because the priority queue requires strict sequential processing, this permanently blocks all subsequent L1→L2 priority transactions.

### Finding Description

**Root cause 1 — `MAX_TX_LEN_BYTES` is defined but never enforced.**

`MAX_TX_LEN_BYTES = 1 << 23` (8 MB) is declared in constants but is referenced only within that file and is never checked during transaction ingestion or processing. [1](#0-0) 

The `try_begin_next_tx` path only guards against a u32 overflow, not against the declared application-level limit: [2](#0-1) 

**Root cause 2 — Intrinsic native cost scales with calldata length.**

`calculate_l1_tx_intrinsic_computational_native_resources` charges `L1_TX_INTRINSIC_COMPUTATIONAL_NATIVE_PER_CALLDATA_BYTE = COPY_BYTE_NATIVE_COST` per byte of calldata: [3](#0-2) 

This cost is always included in `computational_native_used` returned by `process_l1_transaction`, even when the transaction body itself does nothing: [4](#0-3) 

**Root cause 3 — `check_for_block_limits` treats priority transactions identically to regular transactions.**

After `process_l1_transaction` returns successfully, the tx loop checks cumulative block-level native usage: [5](#0-4) 

If the limit is exceeded, the transaction is reverted and reported as a validation error — **without** being added to `enforced_transaction_hashes_accumulator`: [6](#0-5) 

The accumulator is only updated on the success path: [7](#0-6) 

**Root cause 4 — `prepare_and_check_resources` explicitly acknowledges the priority queue risk but only protects against transaction-level resource exhaustion, not block-level limits.** [8](#0-7) 

The saturating arithmetic in `prepare_and_check_resources` ensures the transaction is never *invalidated* at the transaction level, but it provides no protection against the block-level `MAX_NATIVE_COMPUTATIONAL` check that happens afterward.

**Attack path:**

1. Attacker submits an L1→L2 priority transaction on L1 with a calldata field large enough that `L1_TX_INTRINSIC_NATIVE_COST + calldata_len × COPY_BYTE_NATIVE_COST > MAX_NATIVE_COMPUTATIONAL`.
2. The L1 contract accepts it (no matching size limit enforced on-chain that accounts for L2 native costs).
3. The sequencer must include it (priority queue ordering).
4. `process_l1_transaction` completes without error (saturating arithmetic).
5. `check_for_block_limits` fires `BlockNativeLimitReached`; the transaction is reverted and excluded from `enforced_transaction_hashes_accumulator`.
6. The L1 contract rejects any batch that does not include this transaction's hash in the enforced set.
7. The sequencer cannot advance past this transaction; all subsequent L1→L2 priority transactions are permanently blocked.

### Impact Explanation

All L1→L2 priority transactions (deposits, forced transactions) are permanently blocked. Users cannot bridge assets from L1 to L2. Any funds already locked in the L1 bridge contract become inaccessible until the protocol is upgraded to handle or skip the malicious transaction — a process requiring validator/governance coordination. During periods of market volatility this can cause direct financial loss to users who need to move collateral.

### Likelihood Explanation

The attacker only needs to submit a single L1 transaction with a large `data` field and pay the corresponding L1 gas. No privileged access, validator key, or governance majority is required. The L1 contract does not enforce a calldata size limit calibrated to the L2 block's `MAX_NATIVE_COMPUTATIONAL` budget. The `MAX_TX_LEN_BYTES` constant that could serve as such a guard is defined but never enforced anywhere in the processing pipeline.

### Recommendation

1. **Enforce `MAX_TX_LEN_BYTES` at ingestion.** Add a check in `try_begin_next_tx` (or in `AbiEncodedTransaction::try_from_buffer`) that rejects any transaction whose encoded length exceeds `MAX_TX_LEN_BYTES`.

2. **Add a pre-execution native-cost guard for priority transactions.** Before executing an L1 priority transaction, verify that its intrinsic computational native cost (`L1_TX_INTRINSIC_NATIVE_COST + calldata_len × COPY_BYTE_NATIVE_COST`) does not exceed `MAX_NATIVE_COMPUTATIONAL`. If it does, the transaction should be treated as a no-op (failed execution, funds refunded) rather than causing the block to be rejected — consistent with the existing philosophy in `prepare_and_check_resources`.

3. **Alternatively, exempt priority transactions from `check_for_block_limits`.** Priority transactions that would exceed block limits should be processed in isolation (dedicated block) or have their native cost capped, rather than being silently excluded from the accumulator.

### Proof of Concept

```
// Attacker crafts an L1→L2 priority transaction with calldata of size N where:
//   L1_TX_INTRINSIC_NATIVE_COST + N * COPY_BYTE_NATIVE_COST > MAX_NATIVE_COMPUTATIONAL
//
// Step 1: Submit on L1 (no size limit prevents this).
// Step 2: Sequencer includes it in a block (priority queue forces inclusion).
// Step 3: process_l1_transaction() returns Ok(ZkTxResult { computational_native_used: X })
//         where X > MAX_NATIVE_COMPUTATIONAL.
// Step 4: check_for_block_limits() returns Err(BlockNativeLimitReached).
// Step 5: Transaction is reverted; enforced_transaction_hashes_accumulator is NOT updated.
// Step 6: L1 contract rejects the batch (missing enforced tx hash).
// Step 7: All subsequent priority transactions are permanently blocked.
//
// Relevant constants (basic_bootloader/src/bootloader/constants.rs):
//   MAX_TX_LEN_BYTES = 1 << 23  // 8 MB — defined but NEVER enforced
//   L1_TX_INTRINSIC_COMPUTATIONAL_NATIVE_PER_CALLDATA_BYTE = COPY_BYTE_NATIVE_COST
//
// The attacker bisects calldata sizes (analogous to the OPinit PoC) to find
// the threshold where computational_native_used crosses MAX_NATIVE_COMPUTATIONAL.
```
