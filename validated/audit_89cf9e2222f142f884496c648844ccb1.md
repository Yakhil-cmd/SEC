### Title
Missing Deadline and Slippage Protection in SNS Treasury Manager Deposit/Withdraw API — (File: `rs/sns/treasury_manager/src/lib.rs`)

---

### Summary

The `DepositRequest` and `WithdrawRequest` structs in the SNS Treasury Manager API contain no `deadline` or slippage/price-limit fields. The `execute_treasury_manager_deposit` function in SNS governance dispatches `deposit` calls to the treasury manager canister with no time-bound or price-protection constraints. Because SNS governance proposals have a multi-day voting window before execution, a front-runner can observe a pending deposit proposal, manipulate the target DEX price, and cause the SNS treasury to receive fewer LP tokens than expected.

---

### Finding Description

**Root cause — `DepositRequest` and `WithdrawRequest` lack deadline and slippage fields:**

`rs/sns/treasury_manager/src/lib.rs` defines the canonical types that every compliant Treasury Manager implementation must accept:

```rust
pub struct DepositRequest {
    pub allowances: Vec<Allowance>,   // no deadline, no min_price_ratio
}

pub struct WithdrawRequest {
    pub withdraw_accounts: Option<BTreeMap<Principal, Account>>,  // no deadline
}
``` [1](#0-0) 

Because these types are the protocol-level contract, no compliant Treasury Manager implementation can receive or enforce deadline or slippage parameters from the governance layer — the fields simply do not exist in the API.

**Execution path — `execute_treasury_manager_deposit` dispatches with no time or price guard:**

`rs/sns/governance/src/extensions.rs` constructs the deposit payload and calls `deposit` on the treasury manager canister:

```rust
async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    ...
    governance.env.call_canister(extension_canister_id, "deposit", arg_blob).await
    ...
}
``` [2](#0-1) 

No deadline is derived from the proposal's creation or voting timestamps, and no slippage bound is constructed or passed. The same applies to `execute_treasury_manager_withdraw`. [3](#0-2) 

**The codebase itself acknowledges the risk but provides no mitigation:**

`rs/sns/treasury_manager/treasury_manager.did` documents it as a "Known Security Risk":

> Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved. [4](#0-3) 

`rs/sns/governance/src/proposal.rs` repeats the warning in the proposal rendering for `RegisterExtension`:

> Some Decentralized Exchanges lack slippage protection during deposits. Consequently, deposited asset ratios may deviate from those specified in the proposal. This can expose liquidity pool adaptors to mispricing, making them vulnerable to front-running or sandwich attacks. [5](#0-4) 

Despite both acknowledgements, neither a `deadline` field nor a `min_price_ratio`/`slippage_tolerance` field is added to `DepositRequest` or `WithdrawRequest`, and the governance execution layer enforces nothing.

**Validation gap — deposit is capped at 50% of balance but not price-protected:**

`validate_deposit_operation_impl` checks that the requested amount does not exceed 50% of the current treasury balance, but performs no price-ratio or timing check: [6](#0-5) 

---

### Impact Explanation

An SNS treasury deposit proposal transfers real ICP and SNS tokens to a DEX liquidity pool. If the DEX price is manipulated between proposal adoption and execution, the treasury receives fewer LP tokens than the ratio implied at proposal time. The attacker profits from the price difference. Because the `DepositRequest` type has no slippage field, no compliant Treasury Manager implementation can reject an execution that occurs at an unfavorable price — the protocol-level API structurally prevents it. The loss is permanent and proportional to the manipulated price deviation and the deposit size (up to 50% of the treasury balance per proposal).

---

### Likelihood Explanation

SNS governance proposals are public and have a voting period of multiple days before execution. Any actor who can observe IC state (which is fully public) and submit transactions to the target DEX can prepare a sandwich attack. The attacker does not need any privileged role: they only need to watch for `ExecuteExtensionOperation` proposals with `operation_name = "deposit"` and have sufficient liquidity to move the DEX price. The window between proposal adoption and execution is deterministic and observable on-chain, making timing straightforward.

---

### Recommendation

1. **Add a `deadline_timestamp_ns: Option<u64>` field to `DepositRequest` and `WithdrawRequest`** in `rs/sns/treasury_manager/src/lib.rs` and `rs/sns/treasury_manager/treasury_manager.did`. Treasury Manager implementations must reject calls where `ic_cdk::api::time() > deadline_timestamp_ns`.

2. **Add a `min_price_ratio` or `slippage_tolerance_bps` field to `DepositRequest`** so that the governance proposal can encode the price ratio observed at proposal creation time, and the Treasury Manager implementation can abort the deposit if the DEX price has moved beyond the tolerance.

3. **In `execute_treasury_manager_deposit`**, derive a deadline from the proposal's adoption timestamp (e.g., adoption time + a short execution window) and include it in the constructed `DepositRequest` payload.

4. **In `validate_deposit_operation_impl`**, optionally validate that the current time is within a reasonable window of the proposal's creation, rejecting stale proposals before funds are approved.

---

### Proof of Concept

1. An SNS governance proposal is submitted: `ExecuteExtensionOperation { operation_name: "deposit", operation_arg: { treasury_allocation_icp_e8s: X, treasury_allocation_sns_e8s: Y } }`.
2. The proposal enters the voting period (publicly visible on-chain). The attacker reads the deposit amounts and the target DEX canister ID from the proposal.
3. The proposal is adopted. The attacker waits for the execution heartbeat.
4. Just before execution, the attacker submits a large swap on the target DEX, moving the ICP/SNS price significantly in their favor.
5. `execute_treasury_manager_deposit` fires: it calls `approve_treasury_manager` (granting the treasury manager an ICRC-2 allowance), then calls `deposit` on the treasury manager canister with a `DepositRequest` containing only `allowances` — no deadline, no price bound.
6. The treasury manager deposits at the manipulated price, receiving fewer LP tokens than the ratio at proposal creation time.
7. The attacker reverses their swap, extracting the price difference as profit. The SNS treasury has permanently lost value. [7](#0-6) [8](#0-7)

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

/// Execute a treasury manager deposit operation
async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())
}
