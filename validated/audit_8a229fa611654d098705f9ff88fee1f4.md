### Title
Scheduled XCC Promise Permanently Lost on Execution Failure, Freezing Attached NEAR — (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The `execute_scheduled` function in the XCC router contract removes a scheduled promise from persistent storage **before** the cross-contract call is dispatched. If the dispatched call fails, NEAR state changes are not reverted (only panics revert state in NEAR), so the promise entry is permanently deleted with no retry path. Any NEAR attached to the failed call is returned to the router sub-account, where it is irrecoverable because the router exposes no general-purpose withdrawal function.

---

### Finding Description

In `etc/xcc-router/src/lib.rs`, `execute_scheduled` follows this sequence:

```
1. self.scheduled_promises.remove(&nonce.0)   // state committed on function return
2. Self::promise_create(promise)               // dispatches the cross-contract call
3. env::promise_return(promise_id)             // returns the promise to the runtime
``` [1](#0-0) 

In NEAR, state mutations are committed when the function **returns without panicking**. A subsequently failing promise does **not** roll back the state of the function that created it. Therefore:

- The `remove` at step 1 is committed permanently.
- If the cross-contract call at step 2 fails (target panics, out of gas, account does not exist, etc.), the NEAR attached via `attached_balance` is refunded to the router sub-account.
- The promise entry is gone from `scheduled_promises`; there is no nonce-based re-insertion, no callback, and no event emitted.

The router exposes no general-purpose NEAR withdrawal. The only egress is `send_refund`, which unconditionally transfers the hard-coded constant `REFUND_AMOUNT` (2 NEAR) to the parent engine — it is not a recovery mechanism for arbitrary stuck balances. [2](#0-1) 

The `schedule` entry point (callable only by the parent engine, which is triggered by the EVM XCC precompile) stores the promise and increments the nonce. Once `execute_scheduled` consumes that entry, the nonce slot is gone. [3](#0-2) 

This is structurally identical to the Teleportation.sol bug: a sequential identifier (`totalDisbursements` / `nonce`) advances past a failed operation, making the failed operation permanently unretriable.

---

### Impact Explanation

**Permanent freezing of funds (Critical).**

Any NEAR that was unwrapped from wNEAR and forwarded to the router sub-account to fund the XCC call is returned to the router on failure. Because the router has no function to return an arbitrary NEAR balance to the originating EVM address or to the engine, those funds are permanently locked in the router sub-account. The user's wNEAR was already burned on the engine side during the `withdraw_wnear_to_router` step, so the loss is final from the user's perspective. [4](#0-3) 

---

### Likelihood Explanation

**Medium.** The scenario requires:

1. A user invoking the XCC precompile with a `schedule` call and non-zero `required_near` (attached NEAR for the target call).
2. The target contract being temporarily unavailable, paused, or reverting at the time `execute_scheduled` is called.

`execute_scheduled` is intentionally open to any caller (no access control), so any party can trigger execution at any time, including before the target contract is ready. [5](#0-4) 

Temporary contract pauses, storage registration failures, and out-of-gas conditions on the target are realistic production events on NEAR mainnet.

---

### Recommendation

1. **Attach a callback** to the promise created in `execute_scheduled`. In the callback, check `env::promise_result`. If the result is `Failed`, re-insert the original `PromiseArgs` back into `scheduled_promises` under the same nonce so the call can be retried.
2. **Emit a log** on failure (analogous to the Teleportation.sol recommendation) so off-chain monitoring can detect stuck promises.
3. **Add a recovery withdrawal** function callable only by the parent engine to drain excess NEAR from the router back to the engine, mitigating the fund-freeze consequence even if a retry is not possible.

---

### Proof of Concept

```
1. Alice holds wNEAR on Aurora and calls the XCC precompile targeting
   some_contract.near, attaching 1 NEAR for the call.

2. handle_precompile_promise withdraws 1 NEAR worth of wNEAR from Alice's
   ERC-20 balance, sends it to Alice's router sub-account, and calls
   router.schedule(promise_with_1_NEAR_attached) at nonce=0.

3. Bob (anyone) calls router.execute_scheduled(0).
   - scheduled_promises.remove(0) → state committed, entry gone.
   - promise_create dispatches the call to some_contract.near with 1 NEAR.

4. some_contract.near is temporarily paused and panics.
   - 1 NEAR is refunded to the router sub-account.
   - execute_scheduled already returned successfully; the remove is permanent.

5. Alice's wNEAR is burned. The router holds 1 NEAR with no withdrawal path.
   Nonce 0 no longer exists in scheduled_promises; retry is impossible.
   Funds are permanently frozen.
``` [1](#0-0) [6](#0-5)

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")

    pub fn send_refund(&self) -> Promise {
        let parent = self.get_parent().unwrap_or_else(env_panic);

        require_caller(&parent)
            .and_then(|_| require_no_failed_promises())
            .unwrap_or_else(env_panic);

pub fn withdraw_wnear_to_router<I: IO + Copy, E: Env, M: ModExpAlgorithm, H: PromiseHandler>(

    engine: &mut Engine<I, E, M>,
