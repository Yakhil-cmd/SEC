### Title
Untracked Failed `ft_transfer` Promise in `ExitToNear` Precompile Causes Permanent ETH Burn Without Refund - (File: engine-precompiles/src/native.rs)

---

### Summary
When the `error_refund` compile-time feature is not enabled, the `ExitToNear` precompile burns ETH from the user's EVM balance and schedules a NEAR `ft_transfer` promise with **no attached callback**. If that promise fails for any reason, the burned ETH is permanently unrecoverable — the EVM supply decreases while the eth-connector's NEP-141 balance is unchanged, creating a permanent fund freeze for the affected user.

---

### Finding Description

`ExitToNear::run` in `engine-precompiles/src/native.rs` handles bridging ETH (base token) or ERC-20 tokens from Aurora to NEAR. The flow for a base-token (ETH) exit is:

1. The EVM deducts `context.apparent_value` (the ETH sent to the precompile) from the caller's balance — this is an irreversible EVM state change committed before any NEAR promise executes.
2. The precompile constructs a `PromiseCreateArgs` targeting the eth-connector's `ft_transfer` method.
3. It then decides whether to wrap that promise in a callback:

```rust
// engine-precompiles/src/native.rs  lines 449-483
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,                          // ← always None without the feature
    transfer_near: transfer_near_args,     // ← None for base-token exit
};

let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)  // ← NO callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { … })
};
```

For a base-token exit **without** the `error_refund` feature:
- `callback_args.refund` = `None`
- `callback_args.transfer_near` = `None`
- `callback_args == default()` → `PromiseArgs::Create(transfer_promise)` — **no callback**

The `exit_to_near_precompile_callback` function, which is the only place `engine::refund_on_error` is called, is therefore **never scheduled**:

```rust
// engine/src/contract_methods/connector.rs  lines 231-239
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    …
} else {
    None   // ← taken when refund is None; no recovery
};
```

If the `ft_transfer` NEAR promise fails (e.g., the eth-connector is paused, the recipient account does not exist on NEAR, or the recipient has not registered storage with the eth-connector), the ETH is already burned from the EVM with no path to restoration.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The user's ETH is irreversibly removed from their EVM balance. The eth-connector retains the corresponding NEP-141 tokens (it never transferred them), so the system is over-collateralised from the connector's perspective, but the individual user has suffered a total, unrecoverable loss of the exited amount. There is no on-chain mechanism to detect or compensate for this discrepancy without the `error_refund` feature.

---

**Medium.**
