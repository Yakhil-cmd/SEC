### Title
Unbounded `move_to_block` Loop in Hashchain Exhausts NEAR Gas, Permanently Freezing the Protocol - (File: `engine-hashchain/src/hashchain.rs`)

---

### Summary

The Aurora Engine hashchain integrity system contains an unbounded `while` loop in `Hashchain::move_to_block` that iterates once per NEAR block elapsed since the last Aurora transaction. This function is invoked unconditionally at the start of every state-altering contract method via `with_hashchain` / `with_logs_hashchain`. If the hashchain is enabled and a sufficiently long period of inactivity occurs (no Aurora transactions submitted), the loop will exhaust the NEAR 300 TGas per-transaction gas limit, making every subsequent call to `submit`, `call`, `deploy_code`, and all other state-altering methods permanently revert. There is no partial-advance escape hatch, so the protocol cannot recover without an admin upgrade.

---

**The unbounded loop in `move_to_block`:**

```rust
pub fn move_to_block(
    &mut self,
    next_block_height: u64,
) -> Result<(), BlockchainHashchainError> {
    if next_block_height <= self.current_block_height {
        return Err(BlockchainHashchainError::BlockHeightIncorrect);
    }

    while self.current_block_height < next_block_height {   // iterates (gap) times
        self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
            &self.chain_id,
            self.contract_account_id.as_bytes(),
            self.current_block_height,
            self.previous_block_hashchain,
        );
        self.block_hashchain_computer.clear_txs();
        self.current_block_height += 1;
    }

    Ok(())
}
``` [6](#0-5) 

Each iteration calls `compute_block_hashchain`, which concatenates ~370 bytes of data and invokes the NEAR host `keccak256` function:

```rust
pub fn compute_block_hashchain(...) -> RawH256 {
    let txs_hash = self.txs_merkle_tree.compute_hash();
    let data = [
        chain_id,                      // 32 bytes
        contract_account_id,           // ~10-20 bytes
        &current_block_height.to_be_bytes(), // 8 bytes
        &previous_block_hashchain,     // 32 bytes
        &txs_hash,                     // 32 bytes
        self.txs_logs_bloom.as_bytes(), // 256 bytes
    ].concat();
    keccak(&data).0
}
``` [7](#0-6) 

The number of iterations equals `current_block_height_at_call_time − hashchain.current_block_height`, i.e., the number of NEAR blocks elapsed since the last Aurora transaction. NEAR produces approximately one block per second. There is no cap, no chunking, and no partial-advance function.

---

### Impact Explanation

**Impact: Critical — Permanent freezing of funds.**

Once the loop's gas cost exceeds NEAR's 300 TGas per-transaction limit, every call to `submit`, `call`, `deploy_code`, and all other state-altering methods will run out of gas and revert before any user logic executes. All ETH and ERC-20 tokens held inside the Aurora EVM become permanently inaccessible because:

1. No EVM transaction can be submitted (all paths go through `with_logs_hashchain`).
2. There is no partial-advance function analogous to the fix applied in SmartYield PR#37.
3. The only recovery path is an admin contract upgrade, which is a privileged action outside the scope of normal protocol operation. [8](#0-7) 

---

### Likelihood Explanation

**Likelihood: Medium.**

The hashchain must be explicitly initialized to be active. Once active, the trigger condition is simply a sufficiently long period of inactivity on the Aurora network. NEAR produces ~1 block/second. The exact block-gap threshold that exhausts 300 TGas depends on the NEAR host gas cost for `keccak256` plus the `concat` allocation per iteration, but even at a conservative estimate of 1 GGas per iteration, only ~300,000 iterations (≈3.5 days of zero Aurora activity) are needed to hit the limit. Longer inactivity periods (e.g., during a network incident, a market downturn, or a coordinated griefing campaign that discourages relayers) make this reachable without any privileged access.

---

### Recommendation

1. **Add a public partial-advance function** (analogous to SmartYield PR#37) that advances the hashchain by at most `N` blocks per call, where `N` is a safe constant derived from the NEAR gas budget.
2. **Cap the loop** inside `move_to_block` to a maximum number of iterations per invocation, returning a distinct error or storing intermediate state when the cap is hit, so subsequent calls can continue from where the previous one left off.
3. **Add a NEAR gas guard** at the start of the loop body using `env::used_gas()` to break early and persist partial progress.

---

### Proof of Concept

**Attacker-controlled entry path (unprivileged):**

1. Aurora Engine is deployed with the hashchain enabled (initialized at block height `B0`).
2. No Aurora transactions are submitted for `G` NEAR blocks (e.g., `G = 500,000`, achievable in ~5.8 days).
3. Any user (or the attacker themselves) calls `aurora.submit(rlp_encoded_tx)`.
4. Execution enters `with_logs_hashchain` → `load_hashchain` → `move_to_block(B0 + G)`.
5. The `while` loop runs `G` iterations, each computing a keccak hash over ~370 bytes.
6. NEAR gas is exhausted before the loop completes; the transaction reverts with `GasExceeded`.
7. All subsequent calls to any state-altering method on Aurora also revert at the same point.
8. Funds locked in the Aurora EVM are permanently frozen.

pub fn with_hashchain<I, E, T, F>(
    mut io: I,
    env: &E,
    function_name: &str,
    f: F,
) -> Result<T, ContractError>
where
    I: IO + Copy,
    E: Env,
    F: for<'a> FnOnce(CachedIO<'a, I>) -> Result<T, ContractError>,
{
    let block_height = env.block_height();
    let maybe_hashchain = load_hashchain(&io, block_height)?;

    let cache = RefCell::new(IOCache::default());
    let hashchain_io = CachedIO::new(io, &cache);
    let result = f(hashchain_io)?;

    if let Some(mut hashchain) = maybe_hashchain {
        let cache_ref = cache.borrow();
        hashchain.add_block_tx(
            block_height,
            function_name,
            &cache_ref.input,
            &cache_ref.output,
            &Bloom::default(),
        )?;
        save_hashchain(&mut io, &hashchain)?;
    }

    Ok(result)
}
