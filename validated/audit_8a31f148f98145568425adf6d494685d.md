Audit Report

## Title
Missing Coinbase Input Guard Causes Permanent `TransientInternalError` for Coinbase UTXOs — (`rs/bitcoin/checker/src/fetch.rs`, `rs/bitcoin/checker/src/state.rs`)

## Summary

`TransactionCheckData::from_transaction` maps every `TxIn` to a `PreviousOutput` without skipping the coinbase sentinel (`txid=[0;32]`, `vout=0xFFFFFFFF`). When the checker subsequently tries to resolve the input address for that sentinel, it issues an HTTP outcall for a transaction that does not exist; every provider returns a non-200 response, which is permanently classified as `FetchTxStatus::Error` and surfaced as `CheckTransactionRetriable::TransientInternalError`. Because the error is retriable rather than irrecoverable, the minter retries indefinitely and the coinbase UTXO can never be minted as ckBTC.

## Finding Description

**Step 1 — No coinbase guard in input extraction.**

`TransactionCheckData::from_transaction` iterates every `TxIn` unconditionally:

```rust
let inputs = tx.input.iter()
    .map(|input| PreviousOutput {
        txid: Txid::from(*(input.previous_output.txid.as_ref() as &[u8; 32])),
        vout: input.previous_output.vout,
    })
    .collect();
``` [1](#0-0) 

There is no `if tx.is_coinbase() { … }` or `if input.previous_output == OutPoint::null() { continue }` guard. A grep across the entire `rs/bitcoin/checker/` tree returns zero matches for `coinbase`, `is_coinbase`, `OutPoint::null`, or `all_zeros`.

**Step 2 — `check_fetched` attempts to fetch `txid=[0;32]`.**

For every input whose `input_addresses[index]` is still `None`, `check_fetched` calls `try_fetch_tx(input.txid)`. For a coinbase transaction that txid is `[0;32]`: [2](#0-1) 

**Step 3 — HTTP provider returns non-200; classified as transient.**

In `http_get_tx`, any non-200 HTTP status becomes `HttpGetTxError::Rejected { code: 2 /*SYS_TRANSIENT*/ }`: [3](#0-2) 

**Step 4 — `into_response` maps `Rejected` to `TransientInternalError`.**

`FetchResult::Error(err)` at line 260 calls `err.into_response(input_txid)`, and `Rejected` always maps to `TransientInternalError`: [4](#0-3) 

**Step 5 — Retry logic cycles providers indefinitely.**

On every subsequent call, `try_fetch_tx` sees `FetchTxStatus::Error` and advances to the next provider — but all providers return 404 for `txid=[0;32]`: [5](#0-4) 

**Step 6 — `check_for_blocked_input_addresses` always returns `MissingInputAddresses`.**

Because `input_addresses[0]` is never set to `Some(…)`, the check never resolves: [6](#0-5) 

The coinbase txid remains `Fetched` with a permanently-`None` input address, so every call re-enters the fetch loop, issues a new outcall to the next provider, receives 404, stores `FetchTxStatus::Error`, and returns `TransientInternalError`. No existing guard, timeout, or irrecoverable-error path breaks this cycle.

## Impact Explanation

Any coinbase UTXO deposited to a ckBTC minter deposit address can never be minted as ckBTC. The checker returns `TransientInternalError` (→ minter surfaces `TemporarilyUnavailable`) on every call, indefinitely. Because the error is retriable rather than irrecoverable, the minter has no signal to quarantine or reject the UTXO; the deposit is permanently frozen. This constitutes a concrete, permanent user-funds harm within the ckBTC / Chain Fusion scope, matching the **High** impact class: *"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."*

## Likelihood Explanation

Exploitation requires a miner to direct one or more coinbase outputs to a ckBTC minter deposit address. This requires no privileged access, no governance vote, no key material, and no victim mistake — it is entirely valid on-chain behavior. After Bitcoin's mandatory 100-block coinbase maturity, the UTXO becomes spendable and the minter's UTXO scanner surfaces it, automatically triggering the check. The attack is repeatable across any number of deposit addresses and requires only block-production capability (or cooperation with a miner).

## Recommendation

In `TransactionCheckData::from_transaction` (or at the entry of `check_fetched`), detect and skip coinbase inputs before they are added to the `inputs` vector:

```rust
// Bitcoin coinbase sentinel: txid = [0;32], vout = 0xFFFF_FFFF
let inputs = tx.input.iter()
    .filter(|input| {
        !(input.previous_output.txid == bitcoin::Txid::all_zeros()
          && input.previous_output.vout == 0xFFFF_FFFF)
    })
    .map(|input| PreviousOutput {
        txid: Txid::from(*(input.previous_output.txid.as_ref() as &[u8; 32])),
        vout: input.previous_output.vout,
    })
    .collect();
```

Alternatively, call `tx.is_coinbase()` before constructing `TransactionCheckData` and short-circuit to `Passed` (coinbase outputs have no taint history to check).

## Proof of Concept

1. Implement a mock `FetchEnv` whose `http_get_tx` returns `Err(HttpGetTxError::Rejected { code: 2, message: "404".into() })` for `txid = [0u8;32]` and returns a valid coinbase transaction for any other txid.
2. Call `check_transaction_inputs(coinbase_txid)`.
3. Assert the response is `CheckTransactionRetriable::TransientInternalError`.
4. Call `check_transaction_inputs(coinbase_txid)` again (simulating minter retry).
5. Assert the response is again `TransientInternalError` — never `Passed` or `Failed` — confirming the invariant is permanently violated regardless of retry count.
