### Title
Unbounded Per-Connection Aggregate Memory Allocation via Concurrent QUIC Streams — (`rs/p2p/quic_transport/src/request_handler.rs`)

---

### Summary

A Byzantine subnet node with an established TLS-authenticated connection can open up to `MAX_CONCURRENT_BIDI_STREAMS = 1000` bidirectional QUIC streams simultaneously and send `MAX_MESSAGE_SIZE_BYTES = 128 MiB` on each. Because `read_request` calls `recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)` with no aggregate per-connection memory cap, and because Quinn extends its QUIC flow-control windows as `read_to_end` drains the internal receive buffer, the replica can accumulate up to **1000 × 128 MiB ≈ 128 GiB** of heap allocations before any stream completes, causing an OOM crash.

---

### Finding Description

**Constants involved:**

- `MAX_MESSAGE_SIZE_BYTES = 128 * 1024 * 1024` (128 MiB) [1](#0-0) 

- `MAX_CONCURRENT_BIDI_STREAMS = VarInt::from_u32(1_000)` [2](#0-1) 

- `STREAM_RECEIVE_WINDOW = VarInt::from_u32(4_000_000)` (4 MiB per stream)
- `RECEIVE_WINDOW = VarInt::from_u32(200_000_000)` (200 MiB connection-level) [3](#0-2) 

**Vulnerable call site:**

`read_request` calls `recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)`. The `max_size` argument is only a hard cap — it does **not** pre-allocate memory, but it does allow the internal `Vec` to grow to 128 MiB per stream. [4](#0-3) 

**Why flow-control windows do not bound total heap usage:**

Quinn's `read_to_end` works by repeatedly calling `read_chunk`, which moves data from Quinn's internal receive buffer into the application-owned `Vec`. Each such move frees Quinn's internal buffer and triggers a `MAX_STREAM_DATA` / `MAX_DATA` window-extension frame back to the sender. This means the QUIC flow-control windows (`STREAM_RECEIVE_WINDOW = 4 MiB`, `RECEIVE_WINDOW = 200 MiB`) only throttle the *rate* of data arrival — they do not cap the *total* heap held by all concurrent `read_to_end` calls. As each stream's `Vec` grows, the sender is continuously granted new credits to send more.

**No aggregate inflight limit in the acceptor loop:**

`start_stream_acceptor` spawns a new Tokio task for every accepted bidirectional stream into an unbounded `JoinSet`, with no cap on the number of concurrently live tasks or their combined memory footprint. [5](#0-4) 

The comment in the code even acknowledges this: *"The extreme result of a slow handler is that the stream limit will be reached, hence having buffered up to the stream limit number of messages/requests."* [6](#0-5) 

**Attack arithmetic:**

| Parameter | Value |
|---|---|
| Streams per connection | 1,000 |
| Max bytes per stream | 128 MiB |
| Worst-case heap | **~128 GiB** |

A typical IC replica node has 32–512 GiB RAM. 128 GiB of heap from a single peer connection is sufficient to trigger the Linux OOM killer on most configurations.

---

### Impact Explanation

OOM crash of a single replica. The Linux OOM killer terminates the replica process, causing it to drop out of consensus participation. A single Byzantine node can target any one peer it is connected to. This matches the stated scope: *non-volumetric platform-level crash of a single replica*.

---

### Likelihood Explanation

- **Precondition:** The attacker must hold a valid subnet node TLS certificate (i.e., be a Byzantine subnet member). This is within the single-node fault model, below the consensus fault threshold.
- **No rate-limiting or authentication bypass required.** The connection is already established; the attacker simply opens streams and sends data.
- **Reproducible locally** with a simulated QUIC peer sending 1000 streams of 128 MiB each.

---

### Recommendation

1. **Add a per-connection aggregate in-flight byte counter.** Before spawning a task for a new stream, check that `current_inflight_bytes + MAX_MESSAGE_SIZE_BYTES` does not exceed a safe per-connection budget (e.g., 256 MiB total). Reject or back-pressure new streams that would exceed the budget.
2. **Reduce `MAX_CONCURRENT_BIDI_STREAMS`** to a value consistent with the actual application-level concurrency needs (e.g., 10–50), rather than 1,000.
3. **Use a semaphore-guarded task pool** in `start_stream_acceptor` so that at most `N` streams are reading concurrently, where `N * MAX_MESSAGE_SIZE_BYTES` is within a safe memory envelope.
4. **Consider streaming deserialization** instead of `read_to_end` to avoid materializing the full message body in one allocation.

---

### Proof of Concept

```
State-machine test outline:
1. Establish a TLS-authenticated QUIC connection to the victim replica.
2. Open 1000 bidirectional streams simultaneously.
3. On each stream, send exactly MAX_MESSAGE_SIZE_BYTES (128 MiB) of valid
   protobuf-encoded HttpRequest bytes, but do NOT send FIN (keep streams open).
4. Monitor victim replica RSS / heap via /proc/<pid>/status or metrics.
5. Assert: replica heap exceeds 64 GiB before any stream completes.
6. Expected result: OOM kill or heap exhaustion within seconds.
```
