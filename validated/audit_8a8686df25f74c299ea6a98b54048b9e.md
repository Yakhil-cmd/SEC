Audit Report

## Title
`allowed_downloads` integer underflow via stale-task/peer-re-add race in `handle_downloaded_chunk_result` — (`rs/p2p/state_sync_manager/src/ongoing.rs`)

## Summary
`OngoingStateSync` tracks download concurrency with `active_downloads: HashMap<NodeId, u64>` and `allowed_downloads: usize`. When a task completes with `NoContent`, the run loop applies `saturating_sub(1)` to the per-peer counter, then calls `handle_downloaded_chunk_result`, which removes the peer and subtracts `PARALLEL_CHUNK_DOWNLOADS` from `allowed_downloads` using plain `-`. If a stale in-flight task fires after the peer has been removed and re-added, `active_downloads.remove` succeeds (returning `Some(0)`), and the plain subtraction underflows `allowed_downloads` from 0 to `usize::MAX`. In debug builds this panics; in release builds it silently wraps and halts state sync for the current cycle, repeatable by the attacker on every restart.

## Finding Description
`allowed_downloads` is incremented by `PARALLEL_CHUNK_DOWNLOADS` (= 10) each time a peer is added via `Entry::Vacant` (line 155–156). It is decremented by the same amount in `handle_downloaded_chunk_result` on `NoContent` or `RequestError` (lines 195–197, 206–208) using plain subtraction, guarded only by `active_downloads.remove(&peer_id).is_some()`.

The run loop comment at lines 124–129 explicitly acknowledges the stale-task scenario and applies `saturating_sub` to the per-peer counter (line 132), but does **not** extend the same protection to `allowed_downloads`.

Exploit path:
1. Peer P added → `active_downloads = {P: 0}`, `allowed_downloads = 10`.
2. 10 chunks dispatched → `active_downloads = {P: 10}`.
3. First task completes `NoContent`: `saturating_sub(1)` → `{P: 9}`; `remove(&P)` returns `Some(9)` → `allowed_downloads -= 10` → `0`; `active_downloads = {}`.
4. 9 stale tasks remain in `downloading_chunks`.
5. P re-advertises: `Entry::Vacant` succeeds → `active_downloads = {P: 0}`, `allowed_downloads = 10`.
6. Stale task completes `NoContent`: `saturating_sub(1)` on P → `{P: 0}` (no underflow here); `remove(&P)` returns `Some(0)` → `.is_some()` is `true` → `allowed_downloads -= 10` → **underflow** (`0usize - 10`).

After step 6, `active_downloads` is empty and `allowed_downloads` is `usize::MAX - 9`. The `debug_assert` at line 162 fires in debug builds. In release builds, `active_downloads.is_empty()` at line 173 is `true` and the sync loop breaks, halting state sync for the current cycle.

## Impact Explanation
**Debug/test builds**: arithmetic overflow panic crashes the replica process.  
**Release builds**: `allowed_downloads` wraps to `usize::MAX - 9`; `active_downloads` is empty so the loop breaks immediately (line 173–175), halting state sync. The attacker can repeat this on every state sync restart (every 5–10 minutes per the code comment), permanently preventing a lagging node from completing state sync and rejoining the subnet. This constitutes a sustained, targeted **application/platform-level DoS on state sync** — a High-severity impact under the ICP bounty scope: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation
Requires a single Byzantine replica peer (below the consensus fault threshold) that: (1) advertises a state, (2) responds `NoContent` to one chunk request, (3) re-advertises the same state before remaining in-flight tasks complete, (4) responds `NoContent` on a stale task. All four steps are normal protocol-level peer behaviors requiring no privileged access, no key compromise, and no subnet-majority corruption. The race window (steps 3–5) is wide because `CHUNK_DOWNLOAD_TIMEOUT` is 10 seconds and 9 stale tasks remain outstanding. Repeatable indefinitely.

## Recommendation
Replace the plain subtraction in `handle_downloaded_chunk_result` with `saturating_sub`, mirroring the fix already applied to the per-peer counter at line 132:

```rust
// lines 196 and 207
self.allowed_downloads = self.allowed_downloads.saturating_sub(PARALLEL_CHUNK_DOWNLOADS);
```

Alternatively, add a generation counter to `active_downloads` entries (e.g., `HashMap<NodeId, (u64, u64)>` where the second field is a session ID) and only decrement `allowed_downloads` when the removed entry belongs to the current session.
