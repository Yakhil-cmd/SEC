### Title
Funds Permanently Lost When `gas_limit > u64::MAX` Passes `charge_gas` But Fails `try_into::<u64>()` — (`engine/src/engine.rs`)

### Summary

In `submit_with_alt_modexp`, `charge_gas` deducts the sender's balance before `transaction.gas_limit` is validated to fit in a `u64`. If `gas_limit > u64::MAX` and `effective_gas_price > 0`, `charge_gas` succeeds and deducts `prepaid_amount`, but the subsequent `gas_limit.try_into()` returns an error and causes an early return — bypassing `refund_unused_gas` entirely. The deducted ETH is permanently lost.

### Finding Description

`Transaction1559.gas_limit` is decoded as a raw `U256` with no upper-bound cap: [1](#0-0) 

`NormalizedEthTransaction.gas_limit` is also `U256`: [2](#0-1) 

In `submit_with_alt_modexp`, the pre-execution validation only checks that `gas_limit >= intrinsic_gas` (a `u64`), which a `gas_limit > u64::MAX` trivially satisfies: [3](#0-2) 

`charge_gas` then computes `prepaid_amount = gas_limit.checked_mul(effective_gas_price)`. For any `gas_limit` in `(u64::MAX, U256::MAX / effective_gas_price]`, this multiplication does not overflow, so `charge_gas` succeeds and **immediately writes the reduced balance to storage**: [4](#0-3) 

After `charge_gas` returns `Ok`, the code attempts to narrow `gas_limit` to `u64`: [5](#0-4) 

The `?` operator causes an early return with `EngineErrorKind::GasOverflow`. The `refund_unused_gas` call at lines 1148–1159 is **never reached**: [6](#0-5) 

The deducted `prepaid_amount` is not credited to any account — it is permanently burned from the sender's balance.

### Impact Explanation

Any ETH pre-paid via `charge_gas` is permanently destroyed. The sender's balance is reduced by `gas_limit * effective_gas_price` with no refund path. This constitutes **permanent freezing/loss of user funds** (Critical).

### Likelihood Explanation

The attacker must sign and submit an EIP-1559 transaction with `gas_limit > u64::MAX`. While standard wallets cap `gas_limit` at `u64::MAX`, the Aurora engine accepts raw RLP-encoded bytes and performs no such cap during parsing. A custom client or a buggy wallet can trivially encode a `U256` gas_limit exceeding `u64::MAX`. The `effective_gas_price` only needs to be nonzero (any nonzero base fee or priority fee suffices). The sender must hold enough balance to cover `prepaid_amount`, which they would in any realistic scenario.

### Recommendation

Move the `gas_limit.try_into::<u64>()` conversion **before** the `charge_gas` call in `submit_with_alt_modexp`. This ensures the transaction is fully validated before any balance is deducted:

```rust
// Validate gas_limit fits in u64 BEFORE deducting balance
let gas_limit: u64 = transaction
    .gas_limit
    .try_into()
    .map_err(|_| EngineErrorKind::GasOverflow)?;

// Now safe to charge gas
let prepaid_amount = match engine.charge_gas(...) { ... };
```
