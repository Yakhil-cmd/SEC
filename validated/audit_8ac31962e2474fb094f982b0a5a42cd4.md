### Title
Stale Self-Granted Neuron Permissions Persist After `ManagePrincipals` Revocation in SNS Governance - (`rs/sns/governance/src/governance.rs`)

### Summary

In the SNS governance canister, a principal holding `ManagePrincipals` permission can use `AddNeuronPermissions` to grant themselves additional high-privilege permissions (e.g., `Disburse`, `Split`, `DisburseMaturity`). When the neuron owner later revokes only the `ManagePrincipals` permission via `RemoveNeuronPermissions`, the self-granted permissions persist. The revoked principal retains the ability to perform privileged neuron operations — including disbursing the neuron's staked tokens — even though the owner intended to fully revoke their access.

### Finding Description

The SNS governance system has two distinct permission-granting mechanisms:

1. **`ManagePrincipals`** — a broad operator-level permission that allows a principal to call `AddNeuronPermissions` and `RemoveNeuronPermissions` on behalf of a neuron (analogous to `setApprovalForAll`).
2. **Individual permissions** such as `Disburse`, `Split`, `DisburseMaturity` — granted per-principal via `AddNeuronPermissions` (analogous to token-specific `approve`).

The `add_neuron_permissions` function in `rs/sns/governance/src/governance.rs` checks only that the caller holds `ManagePrincipals` (or `ManageVotingPermission` for voting-related permissions), and that the requested permissions are within `neuron_grantable_permissions`. It places **no restriction on the caller granting permissions to themselves**: [1](#0-0) 

The authorization check in `check_principal_authorized_to_change_permissions` confirms this — it only verifies the caller's permission type, not whether `caller == principal_id` (the target): [2](#0-1) 

The `remove_neuron_permissions` function removes only the explicitly listed permissions from the explicitly named principal. There is no cascade that revokes permissions a principal previously self-granted when their `ManagePrincipals` is removed: [3](#0-2) 

The existing test `test_manage_principals_allows_adding_voting_permissions_to_self` explicitly confirms that a principal with `ManagePrincipals` can add `NeuronPermissionList::all()` to themselves, including `Disburse`: [4](#0-3) 

This vulnerability is gated by `neuron_grantable_permissions`. The default value is an empty list, but SNS deployments that configure it to include sensitive permissions (e.g., `Disburse`) are fully exposed: [5](#0-4) [6](#0-5) 

### Impact Explanation

On an SNS with `neuron_grantable_permissions` including `Disburse` (a common configuration in production SNS deployments), a malicious principal Bob who was granted `ManagePrincipals` can:

1. Self-grant `Disburse` (and `Split`, `DisburseMaturity`, etc.) before Alice revokes his access.
2. After Alice calls `RemoveNeuronPermissions` targeting only `ManagePrincipals`, Bob retains `Disburse`.
3. Bob calls `manage_neuron` with the `Disburse` command, transferring the neuron's entire staked governance token balance to an arbitrary account.

This constitutes a **governance authorization bug** with direct financial impact: the neuron's staked tokens (potentially large amounts of SNS governance tokens) can be stolen by a previously-trusted-but-now-revoked principal.

### Likelihood Explanation

The scenario requires:
- An SNS configured with `neuron_grantable_permissions` including `Disburse` (common in production SNS deployments, as shown in integration tests).
- Alice granting Bob `ManagePrincipals` (a normal delegation pattern for neuron management).
- Bob acting maliciously by self-granting `Disburse` before Alice revokes him.
- Alice revoking only `ManagePrincipals` without explicitly also revoking `Disburse`.

This is a realistic multi-step scenario. The neuron owner has no obvious indication that revoking `ManagePrincipals` is insufficient to fully revoke access, making the residual permission easy to miss.

### Recommendation

`remove_neuron_permissions` should cascade: when `ManagePrincipals` is removed from a principal, all permissions that principal holds on the neuron should be revoked simultaneously, or at minimum the system should warn that other permissions remain. Alternatively, `add_neuron_permissions` should prohibit a caller from granting permissions to themselves beyond what they already hold, preventing the self-escalation step entirely.

### Proof of Concept

```
// SNS configured with neuron_grantable_permissions = NeuronPermissionList::all()

// Step 1: Alice (neuron owner) grants Bob ManagePrincipals
alice -> manage_neuron(AddNeuronPermissions {
    principal_id: bob,
    permissions_to_add: [ManagePrincipals],
})

// Step 2: Bob self-grants Disburse using his ManagePrincipals
bob -> manage_neuron(AddNeuronPermissions {
    principal_id: bob,          // self-grant, not blocked
    permissions_to_add: [Disburse, Split, DisburseMaturity],
})
// Succeeds: check_principal_authorized_to_change_permissions passes (bob has ManagePrincipals)
// check_permissions_are_grantable passes (Disburse is in neuron_grantable_permissions)

// Step 3: Alice revokes Bob's ManagePrincipals
alice -> manage_neuron(RemoveNeuronPermissions {
    principal_id: bob,
    permissions_to_remove: [ManagePrincipals],
})
// Bob now has: [Disburse, Split, DisburseMaturity] — NOT revoked

// Step 4: Bob disburses the neuron's staked tokens to his own account
bob -> manage_neuron(Disburse {
    to_account: bob_account,
    amount: full_stake,
})
// Succeeds: bob still has Disburse permission
```

    fn remove_neuron_permissions(
        &mut self,
        neuron_id: &NeuronId,
        caller: &PrincipalId,
        remove_neuron_permissions: &RemoveNeuronPermissions,
    ) -> Result<(), GovernanceError> {
        let neuron = self.get_neuron_result(neuron_id)?;

        let permissions_to_remove = remove_neuron_permissions
            .permissions_to_remove
            .as_ref()
            .ok_or_else(|| {
                GovernanceError::new_with_message(
                    ErrorType::InvalidCommand,
                    "RemoveNeuronPermissions command must provide permissions to remove",
                )
            })?;

        // A simple check to prevent DoS attack with large number of permission changes.
        if permissions_to_remove.permissions.len() > NeuronPermissionType::all().len() {
            return Err(GovernanceError::new_with_message(
                ErrorType::InvalidCommand,
                "RemoveNeuronPermissions command provided more permissions than exist in the system",
            ));
        }

        let principal_id = remove_neuron_permissions
            .principal_id
            .ok_or_else(|| {
                GovernanceError::new_with_message(
                    ErrorType::InvalidCommand,
                    "RemoveNeuronPermissions command must provide a PrincipalId to remove permissions from",
                )
            })?;

        neuron.check_principal_authorized_to_change_permissions(
            caller,
            permissions_to_remove.clone(),
        )?;

        // Re-borrow the neuron mutably to update now that the preconditions have been met
        let principal_id_was_removed = self
            .get_neuron_result_mut(neuron_id)?
            .remove_permissions_for_principal(
                principal_id,
                permissions_to_remove.permissions.clone(),
            )?;

        if principal_id_was_removed == RemovePermissionsStatus::AllPermissionTypesRemoved {
            GovernanceProto::remove_neuron_from_principal_in_principal_to_neuron_ids_index(
                &mut self.principal_to_neuron_ids_index,
                neuron_id,
                &principal_id,
            )
        }

        Ok(())
    }

    pub(crate) fn check_principal_authorized_to_change_permissions(
        &self,
        caller: &PrincipalId,
        permissions_to_change: NeuronPermissionList,
    ) -> Result<(), GovernanceError> {
        // If the permissions to change are exclusively voting related,
        // ManagePrincipals or ManageVotingPermission is sufficient.
        // Otherwise, only ManagePrincipals is sufficient.
        let sufficient_permissions = if permissions_to_change.is_exclusively_voting_related() {
            vec![
                NeuronPermissionType::ManagePrincipals,
                NeuronPermissionType::ManageVotingPermission,
            ]
        } else {
            vec![NeuronPermissionType::ManagePrincipals]
        };

        // The caller is authorized if they have any of the sufficient permissions
        let caller_authorized = sufficient_permissions
            .iter()
            .any(|sufficient_permission| self.is_authorized(caller, *sufficient_permission));

    pub fn check_permissions_are_grantable(
        &self,
        neuron_permission_list: &NeuronPermissionList,
    ) -> Result<(), GovernanceError> {
        let mut illegal_permissions = HashSet::new();

        let grantable_permissions: HashSet<&i32> = self
            .neuron_grantable_permissions
            .as_ref()
            .expect("NervousSystemParameters.neuron_grantable_permissions must be present")
            .permissions
            .iter()
            .collect();

        for permission in &neuron_permission_list.permissions {
            if !grantable_permissions.contains(&permission) {
                illegal_permissions.insert(NeuronPermissionType::try_from(*permission).ok());

                ErrorType::AccessControlList,

                ),
            ));
