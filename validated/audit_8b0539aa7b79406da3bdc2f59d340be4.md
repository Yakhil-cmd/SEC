Audit Report

## Title
Silent ICP Burn Failure After Cycles Minting Breaks ICP/Cycles Conservation Invariant — (File: rs/nns/cmc/src/main.rs)

## Summary
The `burn_and_log` function in the Cycles Minting Canister unconditionally swallows ICP ledger burn errors after cycles have already been irreversibly deposited or minted. Because the `blocks_notified` entry is set to a terminal processed state immediately after `burn_and_log` returns — regardless of whether the burn succeeded — the notification cannot be retried, the ICP is never burned, and cycles exist without the corresponding ICP destruction required by the protocol's conservation invariant.

## Finding Description
In `rs/nns/cmc/src/main.rs`, the three notify paths follow an identical pattern:

1. Cycles are deposited or minted first (irreversible).
2. `burn_and_log` is called and its `()` return value is discarded.
3. The `blocks_notified` entry is set to a terminal `NotifiedTopUp`/`NotifiedMint`/`NotifiedCreateCanister` status.

`burn_and_log` (L2014–2049) is explicitly designed to never propagate errors:

```rust
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    ...
    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
``` [1](#0-0) 

The three callers:
- `process_top_up` L2001: `burn_and_log(sub, amount).await;` then `Ok(cycles)` [2](#0-1) 
- `process_create_canister` L1945: `burn_and_log(sub, amount).await;` then `Ok(canister_id)` [3](#0-2) 
- `process_mint_cycles` L1968: `burn_and_log(sub, amount).await;` then `Ok(NotifyMintCyclesSuccess {...})` [4](#0-3) 

The `is_transient_error` guard at L1219–1221 and L1310–1312 only removes the `blocks_notified` entry when the overall `process_*` result is itself a transient error. Since `burn_and_log` returns `()` and the `process_*` functions return `Ok(...)`, `is_transient_error` evaluates to `false` and the terminal entry is committed. [5](#0-4) 

There is no heartbeat, timer, or stable-state queue that retries failed burns. The ICP sits permanently unburned in the CMC's subaccount with no automated recovery path.

## Impact Explanation
When `burn_and_log` fails: cycles are irreversibly credited to the caller; the ICP is not burned from the CMC subaccount; the notification is permanently finalized; and the ICP total supply is not reduced as the protocol requires. This constitutes illegal cycles minting — cycles exist without the corresponding ICP destruction — directly violating the ICP/cycles conservation invariant. Repeated occurrences accumulate unburned ICP in CMC subaccounts, inflating the effective ICP supply relative to cycles issued. This maps to the High impact category: significant NNS/ledger/financial-integration security impact with concrete protocol harm to the ICP/cycles accounting invariant.

## Likelihood Explanation
An unprivileged user submitting a normal `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` ingress message during any window of ICP ledger unavailability (subnet upgrade, replica restart, transient inter-canister rejection) will trigger the silent failure path. No special privileges or attacker-controlled conditions are required beyond opportunistic timing. Subnet upgrades are routine and recurrent, making the window realistic and repeatable. The attacker does not need to control the timing — they only need to submit a valid notification during a naturally occurring unavailability window.

## Recommendation
Separate the double-minting prevention concern from the burn-completion concern:

1. **Retry via stable queue**: On burn failure, persist the pending burn (subaccount + amount) in stable state and retry it from a heartbeat or timer, analogous to how the CMC already retries other async operations. The `Processing` guard already prevents double-minting, so retrying the burn is safe.
2. **Do not finalize on burn failure**: Have `burn_and_log` return a `Result`, and only insert the terminal `blocks_notified` entry when the burn succeeds. On burn failure, remove the `Processing` entry so the user can re-notify. This is safe because the `Processing` guard prevents concurrent double-execution.

The current comment's rationale ("we don't want to reject the transaction notification because then it could be retried") is valid for preventing double-minting but does not justify permanently abandoning the burn — these two concerns can be cleanly separated.

## Proof of Concept
1. User transfers ICP to the CMC subaccount derived from their principal on the ICP ledger.
2. User calls `notify_top_up` with the block index during a subnet upgrade window when the ICP ledger is temporarily unavailable.
3. CMC fetches the transaction, sets `blocks_notified[block_index] = Processing`.
4. CMC calls `deposit_cycles` — succeeds; cycles are credited to the target canister.
5. CMC calls `burn_and_log(sub, amount).await` — `call_protobuf` returns `Err(SysTransient, ...)`.
6. `burn_and_log` logs the error and returns `()`.
7. CMC sets `blocks_notified[block_index] = NotifiedTopUp(Ok(cycles))` — notification finalized.
8. **Result**: cycles minted, ICP not burned, notification permanently closed, ICP orphaned in CMC subaccount.

A deterministic integration test can reproduce this by mocking the ICP ledger's `send_pb` endpoint to return a `SysTransient` rejection on the first call after `deposit_cycles` succeeds, then asserting that (a) the target canister's cycle balance increased, (b) the CMC subaccount still holds the ICP, and (c) a second `notify_top_up` call with the same block index returns `AlreadyProcessed`.

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
