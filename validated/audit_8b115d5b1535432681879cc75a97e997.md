### Title
Permanent Loss of ERC-20 Tokens When `ft_transfer` Fails Without `error_refund` Feature - (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is not enabled, the `ExitToNear` precompile does not schedule a refund callback after the downstream `ft_transfer` promise. If that promise fails (e.g., the recipient NEAR account is not registered with the NEP-141 contract), the ERC-20 tokens that were already burned are permanently lost. There is no mechanism to restore them.

---

### Finding Description

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` handles ERC-20 bridge withdrawals. The flow is:

1. The ERC-20 contract (`EvmErc20V2.sol`) calls `_burn(sender, amount)` first, then calls the `ExitToNear` precompile.
2. The precompile schedules an `ft_transfer` (or `ft_transfer_call`) promise to the NEP-141 contract.
3. Whether a refund callback is attached depends entirely on the `error_refund` compile-time feature flag.

When `error_refund` is **not** enabled:

```rust
// engine-precompiles/src/native.rs
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,                          // ← no refund info
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `callback_args` equals `default()` (i.e., `refund: None` and `transfer_near: None`), no callback promise is attached at all:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback, no refund on failure
} else {
    PromiseArgs::Callback(...)
};
``` [2](#0-1) 

The ERC-20 contract burns tokens before the precompile call:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    _burn(sender, amount);
    // ... calls ExitToNear precompile
}
``` [3](#0-2) 

If `ft_transfer` fails (e.g., recipient not registered with the NEP-141 contract), the burned ERC-20 tokens are permanently gone. The `exit_to_near_precompile_callback` only refunds when `args.refund` is `Some`:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
``` [4](#0-3) 

The existing test suite explicitly documents this behavior:

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

---

### Impact Explanation

When `error_refund` is not compiled in, any EVM user who calls `withdrawToNear` with a recipient NEAR account that is not registered with the NEP-141 contract will have their ERC-20 tokens permanently burned with zero NEP-141 tokens received. This is a **permanent, irreversible loss of user funds** — the ERC-20 supply decreases but the NEP-141 side never transfers, breaking the 1:1 bridge invariant and destroying value.

**Impact: Critical — Permanent freezing/loss of funds.**

---

### Likelihood Explanation

The trigger condition — calling `withdrawToNear` with an unregistered recipient — is a realistic user mistake (e.g., a typo in the NEAR account ID, or a newly created account that has not yet called `storage_deposit` on the NEP-141 contract). The `error_refund` feature is a compile-time opt-in, meaning any deployment built without it is permanently vulnerable. The attacker-controlled entry path requires no special privileges: any EVM account holding ERC-20 tokens can trigger this.

---

### Recommendation

1. **Enable `error_refund` by default** in production builds, or make it a required feature rather than an opt-in.
2. **Restructure the burn-then-exit flow**: consider locking tokens (not burning) until the `ft_transfer` promise resolves successfully, then burning on success and unlocking on failure.
3. **Always attach a refund callback** regardless of the feature flag, so that a failed `ft_transfer` always triggers `exit_to_near_precompile_callback` with the ability to re-mint burned tokens.

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Bridge a NEP-141 token to Aurora, receiving ERC-20 tokens at address `victim`.
3. Call `withdrawToNear("unregistered.near", amount)` on the ERC-20 contract from `victim`.
4. The ERC-20 contract calls `_burn(victim, amount)` — tokens are gone from EVM state.
5. The `ExitToNear` precompile schedules `ft_transfer` to the NEP-141 contract with no callback.
6. The NEP-141 `ft_transfer` fails because `"unregistered.near"` has no storage deposit.
7. No callback fires. `victim`'s ERC-20 balance is permanently zero; no NEP-141 tokens were received.

This is confirmed by `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` at lines 623–665, which explicitly asserts that without `error_refund`, the exit amount is permanently deducted from the user's balance even when the transfer fails. [6](#0-5)
