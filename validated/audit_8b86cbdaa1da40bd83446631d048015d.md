### Title
Silo Address Whitelist Not Enforced in `receive_base_tokens`, Allowing Non-Whitelisted EVM Addresses to Receive ETH — (`engine/src/engine.rs`)

---

### Summary

In Aurora Engine's Silo mode, the `Address` whitelist (`WhitelistKind::Address`) is correctly enforced when minting ERC-20 tokens via `receive_erc20_tokens`, but is **entirely absent** when minting ETH (base tokens) via `receive_base_tokens`. This allows any caller to bridge ETH to a non-whitelisted EVM address, bypassing the Silo's access control model. The minted ETH is then permanently frozen because the non-whitelisted address cannot submit EVM transactions.

---

### Finding Description

Aurora Engine's Silo mode provides four whitelists to restrict who can interact with the EVM:

- `WhitelistKind::Account` — NEAR account IDs allowed to relay transactions
- `WhitelistKind::Address` — EVM addresses allowed to submit/receive
- `WhitelistKind::Admin` / `WhitelistKind::EvmAdmin` — deployer restrictions [1](#0-0) 

When a NEP-141 ERC-20 token is bridged into Aurora via `ft_on_transfer`, `receive_erc20_tokens` correctly checks the recipient address against the `Address` whitelist and redirects to a fallback address if the recipient is not whitelisted: [2](#0-1) 

However, when ETH (base tokens) is bridged via the same `ft_on_transfer` path (when `predecessor_account_id == get_connector_account_id`), `receive_base_tokens` is called instead: [3](#0-2) 

`receive_base_tokens` parses the recipient address from `args.msg` and directly mints ETH to it **with no whitelist check whatsoever**: [4](#0-3) 

The `args.msg` field is fully attacker-controlled — it is the `msg` parameter of the NEP-141 `ft_transfer_call` call, which any user can set to any 40-character hex EVM address.

For comparison, `receive_erc20_tokens` calls `silo::is_allow_receive_erc20_tokens` before minting: [5](#0-4) 

which internally checks the `Address` whitelist: [6](#0-5) 

The `receive_base_tokens` path has no equivalent guard.

---

### Impact Explanation

When the `Address` whitelist is enabled in Silo mode, the intent is that only whitelisted EVM addresses may participate. However:

1. Any user can call `ft_transfer_call` on the ETH connector NEP-141 contract, specifying any EVM address as the recipient in `msg`.
2. `receive_base_tokens` mints ETH to that address without checking the `Address` whitelist.
3. The non-whitelisted address now holds ETH balance inside the Silo.
4. Because `assert_access` blocks `submit` for non-whitelisted addresses, the ETH cannot be spent or withdrawn by the recipient. [7](#0-6) 

The ETH is permanently frozen in the non-whitelisted address with no recovery path. This constitutes **permanent freezing of funds**. Additionally, a Silo operator who bans an address (removes it from the whitelist) cannot prevent new ETH from being deposited to that address, undermining the ban entirely — a direct analog to the original report's whitelist-on-wrong-entity bug.

---

### Likelihood Explanation

- The `ft_on_transfer` entry point is a standard NEP-141 callback, callable by any NEAR account that holds ETH connector tokens.
- No special privileges are required; any token holder can trigger this path.
- The attacker only needs to specify a non-whitelisted EVM address in the `msg` field of `ft_transfer_call`.
- The inconsistency between `receive_base_tokens` (no check) and `receive_erc20_tokens` (has check) makes this easy to exploit once Silo mode with the `Address` whitelist is active.

---

### Recommendation

Add a whitelist check in `receive_base_tokens` analogous to the one in `receive_erc20_tokens`. If a fallback address is configured and the recipient is not whitelisted, redirect the ETH to the fallback address:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let mut receipient = message_data.recipient;

    // Add the same guard as receive_erc20_tokens
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
        && !silo::is_allow_receive_erc20_tokens(&self.io, &receipient)
    {
        receipient = fallback_address;
    }

    let balance = get_balance(&self.io, &receipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &receipient, &new_balance);
    Ok(None)
}
``` [4](#0-3) 

---

### Proof of Concept

**Setup**: Silo mode active, `Address` whitelist enabled, fallback address configured. EVM address `0xDEAD...` is NOT in the whitelist.

**Attack**:
1. Attacker holds ETH connector NEP-141 tokens on NEAR.
2. Attacker calls `ft_transfer_call` on the ETH connector contract with:
   - `receiver_id`: Aurora engine account
   - `amount`: any ETH amount
   - `msg`: `"dead...dead"` (hex encoding of `0xDEAD...`)
3. The ETH connector calls `ft_on_transfer` on Aurora.
4. `ft_on_transfer` detects `predecessor_account_id == connector_account_id` and calls `receive_base_tokens`. [8](#0-7) 

5. `receive_base_tokens` parses `0xDEAD...` from `msg` and mints ETH to it — **no whitelist check**.
6. `0xDEAD...` now has ETH balance but cannot submit any EVM transactions (blocked by `assert_access`).
7. ETH is permanently frozen. The Silo operator's ban on `0xDEAD...` is bypassed for ETH deposits. [2](#0-1)

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
