### Title
Fee-on-Transfer NEP-141 Token Causes ERC-20 Mirror Insolvency and Permanent Fund Freeze - (File: engine/src/engine.rs)

### Summary

The `receive_erc20_tokens` function in `engine/src/engine.rs` mints ERC-20 mirror tokens using the `args.amount` value supplied by the NEP-141 contract's `ft_on_transfer` callback, without verifying the actual NEP-141 balance change in Aurora's account. For fee-on-transfer NEP-141 tokens, Aurora's actual NEP-141 balance will be less than the total ERC-20 supply minted, creating an insolvent bridge state. When users later call `withdrawToNear` to exit, the `ft_transfer` call on the NEP-141 contract will fail for the last users because Aurora does not hold enough NEP-141 tokens to cover the full ERC-20 supply.

### Finding Description

**Entry path:**

1. Any NEP-141 token can be registered on Aurora via `deploy_erc20_token`, creating an ERC-20 mirror.
2. A user calls `ft_transfer_call` on a fee-on-transfer NEP-141 token, specifying amount `X`.
3. The NEP-141 contract deducts a fee, crediting Aurora's account with only `X - fee` tokens, but calls `ft_on_transfer` on Aurora with `amount = X` (the pre-fee amount).
4. Aurora's `ft_on_transfer` handler dispatches to `receive_erc20_tokens`: [1](#0-0) 

5. Inside `receive_erc20_tokens`, the minted amount is taken directly from `args.amount` without any balance check: [2](#0-1) 

Aurora mints `X` ERC-20 tokens but only holds `X - fee` NEP-141 tokens. This discrepancy accumulates with every deposit.

**Exit path (where the freeze manifests):**

When a user calls `withdrawToNear` on the ERC-20 contract: [3](#0-2) 

The ERC-20 is burned and the `ExitToNear` precompile schedules an `ft_transfer` on the NEP-141 contract for the full burned amount: [4](#0-3) 

Once Aurora's actual NEP-141 balance is exhausted (because total ERC-20 supply > NEP-141 held), the `ft_transfer` call reverts. The ERC-20 tokens are already burned at this point; without the `error_refund` feature enabled, the user's funds are permanently lost.

### Impact Explanation

**Insolvency (Critical):** The ERC-20 mirror's total supply exceeds Aurora's actual NEP-141 balance. The bridge is structurally insolvent for any fee-on-transfer NEP-141 token.

**Permanent fund freeze (Critical):** Users who attempt to exit after Aurora's NEP-141 balance is depleted have their ERC-20 tokens burned with no corresponding NEP-141 transfer. Without the `error_refund` feature, there is no on-chain refund path, making the loss permanent.

### Likelihood Explanation

Any NEP-141 token with fee-on-transfer behavior can be registered on Aurora via the permissionless `deploy_erc20_token` entrypoint. The discrepancy grows with every deposit. A single attacker can deliberately amplify the insolvency by repeatedly depositing small amounts, each time causing Aurora to over-mint ERC-20 tokens, until the bridge is insolvent enough to freeze other users' funds.

### Recommendation

In `receive_erc20_tokens`, record Aurora's NEP-141 balance before and after the `ft_transfer_call` completes, and use the actual delta (not `args.amount`) as the mint amount. Since `ft_on_transfer` is a synchronous callback on NEAR, the balance at callback time already reflects the credited amount. Alternatively, return `args.amount` (refund all) if the token is known to be fee-on-transfer, or enforce that only non-fee-on-transfer NEP-141 tokens can be registered.

### Proof of Concept

1. Deploy a NEP-141 token that charges a 1% fee on every transfer (credits receiver `amount * 0.99`), but calls `ft_on_transfer` with the original `amount`.
2. Call `deploy_erc20_token` on Aurora to register the ERC-20 mirror.
3. Call `ft_transfer_call(receiver_id: aurora, amount: 1_000_000, msg: <evm_address>)`.
4. Aurora mints `1_000_000` ERC-20 tokens but holds only `990_000` NEP-141 tokens.
5. Repeat step 3 ten times. Aurora now holds `9_900_000` NEP-141 but has minted `10_000_000` ERC-20.
6. The first `9_900_000` ERC-20 worth of `withdrawToNear` calls succeed.
7. The remaining `100_000` ERC-20 holders call `withdrawToNear`; their tokens are burned but `ft_transfer` on the NEP-141 contract reverts. Funds are permanently frozen.
