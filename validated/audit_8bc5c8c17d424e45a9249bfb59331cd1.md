Audit Report

## Title
Unbounded Nested Loop in SNS `distribute_rewards` Exhausts Instruction Limit, Permanently Halting Voting Reward Distribution - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS governance canister's `distribute_rewards` function executes three unbounded loops over all unsettled proposals and their ballot maps in a single synchronous canister message, with no instruction-limit guard. An unprivileged user can stake SNS tokens to inflate the neuron and ballot count. When the total work (proposals Ă— neurons) exceeds the IC instruction limit, the periodic task traps and voting rewards are permanently halted until a governance upgrade is deployed.

## Finding Description
`distribute_rewards` (line 5763) is called synchronously from `run_periodic_tasks` at line 5513 with no instruction-limit check before or after the call.

Inside `distribute_rewards`, three unbounded loops execute atomically:

**Loop 1 (lines 5894â€“5931):** Nested iteration over every `considered_proposals` entry (up to `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS = 700`, confirmed at `rs/sns/governance/src/proposal.rs:79`) and every ballot in each proposal's `ballots` map. Each iteration performs `HashMap` insertion and `Decimal` arithmetic.

**Loop 2 (lines 5954â€“5997):** Iteration over every entry in `neuron_id_to_reward_shares` to mutate neuron maturity via `get_neuron_result_mut`.

**Loop 3 (lines 6013â€“6081):** Iteration over all `considered_proposals` again to settle them and clear ballots, including a `process_proposal` call per proposal.

A grep search for `is_message_over_threshold`, `is_over_instructions_limit`, and equivalent patterns across all of `rs/sns/governance/**/*.rs` returns **no matches**, confirming the complete absence of any instruction-limit guard in the SNS governance codebase.

By contrast, NNS governance uses a `RewardsDistributionInProgress` state machine in `rs/nns/governance/src/reward/distribution.rs` with an explicit `is_over_instructions_limit` check at line 184 inside `continue_processing`, allowing it to resume across multiple message executions.

The growth vector is attacker-controlled: any principal can stake SNS tokens to create neurons (permissionless). With N neurons, every proposal carries N ballot entries. At 700 proposals Ă— N neurons, Loop 1 alone processes 700Ă—N entries with `HashMap` insertions and `Decimal` arithmetic per entry.

## Impact Explanation
When `distribute_rewards` exceeds the IC instruction limit (~40 billion instructions per message), the canister traps mid-execution. Because `run_periodic_tasks` is invoked on a timer and the state is not checkpointed mid-function, every subsequent invocation re-enters the same code path and traps again. Voting rewards are permanently halted for all SNS neuron holders. Recovery requires a governance upgrade proposal, which itself depends on functioning governance â€” creating a potential deadlock. All reward rounds that elapsed during the outage are lost, causing permanent financial harm to neuron holders.

This matches the allowed impact: **High ($2,000â€“$10,000) â€” Significant SNS security impact with concrete user or protocol harm**, specifically a platform-level DoS on reward distribution that is not based on raw volumetric DDoS.

## Likelihood Explanation
The `max_number_of_neurons` parameter in `NervousSystemParameters` is SNS-configurable and can be set to large values. Even at moderate neuron counts (e.g., 10,000â€“50,000 neurons), 700 proposals Ă— N neurons produces tens of millions of ballot entries with `HashMap` insertions and `Decimal` arithmetic per entry, well within reach of the ~40 billion instruction limit given the per-iteration overhead. The attack requires only the ability to stake SNS tokens â€” an unprivileged, permissionless action available to any token holder. Any SNS with active governance and a large neuron count is at risk without any deliberate attack.

## Recommendation
Apply the same batched-distribution pattern already used in NNS governance:

1. Introduce a persistent `RewardsDistributionInProgress` state in SNS governance (analogous to the NNS `RewardsDistributionStateMachine`).
2. In `distribute_rewards`, compute `neuron_id_to_reward_shares` and store it in this persistent state rather than processing it immediately.
3. Process the maturity-increment loop (Loop 2) in a separate periodic timer task that checks `is_message_over_threshold` after each neuron and resumes in the next message if the limit is reached.
4. Add an instruction-limit check inside Loop 1 (ballot accumulation) as well, since it also iterates over proposals Ă— neurons.

## Proof of Concept
1. Deploy an SNS with `max_number_of_neurons` set to a large value (e.g., 50,000).
2. Create ~50,000 neurons by staking SNS tokens from many principals (permissionless).
3. Submit and let expire ~700 proposals so that `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS` unsettled proposals accumulate, each with ~50,000 ballot entries.
4. Wait for the next `run_periodic_tasks` invocation to call `distribute_rewards`.
5. Loop 1 (lines 5894â€“5931) processes 700 Ă— 50,000 = 35,000,000 ballot entries with `HashMap` insertions and `Decimal` arithmetic per entry, followed by 50,000 neuron store mutations in Loop 2 â€” exceeding the IC instruction limit.
6. The canister traps; every subsequent periodic task invocation traps at the same point; voting rewards are permanently halted.

A deterministic integration test using PocketIC can reproduce this by instantiating an SNS canister with a large neuron set, advancing time past the reward round boundary, and asserting that `run_periodic_tasks` traps or that `latest_reward_event` is never updated after the neuron count crosses the threshold.
