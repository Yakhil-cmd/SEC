### Title
Precision Loss in `native_per_pubdata` Floor Division Undercharges Users for Pubdata Native Resources - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary

ZKsync OS computes `native_per_pubdata` using floor (truncating) integer division while computing `native_per_gas` using ceiling division. This asymmetry causes systematic undercharging of native resources for pubdata consumption, allowing users to generate more pubdata than the operator-set pricing intends to permit.

### Finding Description

In `validation_impl.rs`, the two key resource-pricing ratios are computed with different rounding modes:

```rust
// native_per_gas: uses CEILING division — conservative (charges more)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...

// native_per_pubdata: uses FLOOR division — lenient (charges less)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...
``` [1](#0-0) 

The same floor division is replicated in the public API helper: [2](#0-1) 

`native_per_pubdata` is then used to charge native resources for every byte of pubdata a transaction generates: [3](#0-2) 

**Concrete precision loss example:**
- `native_price = 1000`, `pubdata_price = 1999`
- `native_per_pubdata = floor(1999 / 1000) = 1` (actual cost is 1.999 ≈ 2 with ceiling)
- A user with a native budget of 1,000,000 can generate **1,000,000 bytes** of pubdata instead of the intended **~500,000 bytes** — a 2× excess

**Extreme case — `native_per_pubdata` collapses to zero:**
When `pubdata_price < native_price` (e.g., `pubdata_price = 9`, `native_price = 10`):
- `native_per_pubdata = floor(9/10) = 0`
- Pubdata costs **zero** native resources regardless of how much is generated
- The user is bounded only by EVM gas, not by the native resource budget intended to limit proving costs

A secondary precision loss also exists in `refund_calculation.rs` where `delta_gas` is computed as `floor(native_used / native_per_gas)`, meaning the extra gas charged to cover native overconsumption is also rounded down: [4](#0-3) 

The documentation explicitly states the intended formula as `deltaGas := (nativeUsed / nativePerGas) - gasUsed`, but does not specify rounding direction, leaving the floor truncation as an unintended precision loss. [5](#0-4) 

### Impact Explanation

- **Resource accounting bug**: Users are systematically undercharged for pubdata in native resources. The undercharge scales with `(native_price - 1)` native units per pubdata byte — up to `native_price - 1` units per byte.
- **Pubdata budget bypass**: When `pubdata_price < native_price`, `native_per_pubdata = 0` and pubdata generation consumes no native resources at all, allowing a user to fill a block's pubdata budget using only EVM gas.
- **Operator revenue loss**: The operator sets `pubdata_price` to reflect the cost of publishing data to L1. Floor division means the operator is systematically underpaid for pubdata in native resource terms.
- **Proving cost underestimation**: Native resources model RISC-V proving cycles. Undercharging pubdata in native resources means the prover may receive transactions that cost more to prove than the fee collected.

### Likelihood Explanation

- The operator sets `pubdata_price` and `native_price` based on market conditions. These values will rarely be exact multiples of each other, so the floor division precision loss is present in virtually every block.
- Any unprivileged user submitting a transaction with storage writes (which generate pubdata) benefits from this undercharging automatically — no special setup required.
- The `native_per_pubdata = 0` case is reachable whenever the operator sets `pubdata_price < native_price`, which is a plausible configuration (e.g., during low L1 gas price periods).

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata` computation, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — undercharges pubdata):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...

// After (ceiling division — consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))...
```

Apply the same fix in `api/src/helpers.rs` line 427. Additionally, consider using ceiling division in the `delta_gas` calculation in `refund_calculation.rs` to avoid undercharging users for native overconsumption.

### Proof of Concept

1. Operator sets block context: `native_price = 1000`, `pubdata_price = 1500`.
2. User submits a transaction with `gas_price = 2000`, `gas_limit = 500_000`.
3. Bootloader computes:
   - `native_per_gas = ceil(2000 / 1000) = 2`
   - `native_per_pubdata = floor(1500 / 1000) = 1` ← should be `ceil = 2`
   - `native_limit = 500_000 * 2 = 1_000_000`
4. User generates 900,000 bytes of pubdata during execution.
5. Native charged for pubdata = `900_000 * 1 = 900_000` (fits within limit).
6. **Correct charge** would be `900_000 * 2 = 1_800_000` > `native_limit`, causing a revert.
7. The user successfully publishes 900,000 bytes of pubdata that should have been rejected, paying half the intended native cost.
