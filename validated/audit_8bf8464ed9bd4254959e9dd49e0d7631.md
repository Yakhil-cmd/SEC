Audit Report

## Title
Unbounded `environment_variables` in `CanisterSettingsArgs` Enables Pre-Validation Resource Exhaustion via `create_canister`/`update_settings` - (File: rs/execution_environment/src/canister_settings.rs)

## Summary
`CanisterSettingsArgs.environment_variables` is typed as `Option<Vec<EnvironmentVariable>>` — a plain, unbounded `Vec` — while the analogous `controllers` and `allowed_viewers` fields use `BoundedVec` wrappers that enforce their limits at Candid deserialization time. An unprivileged caller can submit a 2 MB ingress message containing ~200,000 minimal `EnvironmentVariable` records; the replica will fully deserialize the list and perform an O(n log n) `BTreeMap` insertion pass in native Rust before the count limit of 32 is checked and the call is rejected.

## Finding Description
`CanisterSettingsArgs` is defined in `rs/types/management_canister_types/src/lib.rs` at line 2370. The `controllers` field is `Option<BoundedControllers>` (line 2372), where `BoundedControllers = BoundedVec<10, UNBOUNDED, UNBOUNDED, PrincipalId>` (lines 2331–2332). `BoundedVec`'s `Deserialize` implementation rejects sequences exceeding `MAX_ALLOWED_LEN` element-by-element during deserialization (lines 108–113 of `rs/types/management_canister_types/src/bounded_vec.rs`). `BoundedAllowedViewers` (lines 1194–1195) applies the same pattern for log/snapshot visibility viewers.

By contrast, `environment_variables` is `Option<Vec<EnvironmentVariable>>` (line 2382) — no `BoundedVec` wrapper. In `TryFrom<CanisterSettingsArgs> for CanisterSettings` (`rs/execution_environment/src/canister_settings.rs`, lines 177–190), the code calls `.len()` and `.into_iter()` on the raw `Vec`, then collects every `(name, value)` pair into a `BTreeMap<String, String>` before any count check occurs. The count limit (`MAX_ENVIRONMENT_VARIABLES = 32`, `rs/config/src/execution_environment.rs` line 216) is only enforced later inside `validate_environment_variables()` (`rs/execution_environment/src/canister_manager.rs`, lines 290–299), which is called from `validate_and_update_canister_settings()`. The name/value length limits (128 bytes each, lines 219 and 223 of `rs/config/src/execution_environment.rs`) are also checked only inside `validate_environment_variables()`, after the BTreeMap is already built.

Exploit flow:
1. Attacker crafts a `create_canister` or `update_settings` ingress message with `environment_variables` set to ~200,000 `{name: ""; value: ""}` records (~10 bytes each in Candid, fitting within the 2 MB app-subnet ingress limit at `rs/limits/src/lib.rs` line 85).
2. The replica deserializes the full `Vec<EnvironmentVariable>` into heap memory.
3. `TryFrom` iterates all 200,000 entries and inserts them into a `BTreeMap` — O(n log n) native Rust work, outside Wasm instruction metering.
4. Only after the BTreeMap is built does the count check fire and reject the call.
5. Repeated at the block rate, with up to `MAX_INGRESS_MESSAGES_PER_BLOCK = 1000` messages per block (`rs/limits/src/lib.rs` line 78), bounded by `MAX_INGRESS_BYTES_PER_BLOCK = 4 MB` (line 77), the attacker can force sustained unmetered CPU work on every replica in the subnet.

## Impact Explanation
This is a **High** severity application/platform-level DoS. The unmetered O(n log n) BTreeMap construction per malicious message, repeated across every block, can degrade execution throughput on the subnet, delaying or starving legitimate canister calls. The attack does not require cycles, canister deployment, or any privileged access. It is not raw volumetric DDoS — it exploits a specific missing deserialization-time bound in production execution environment code.

## Likelihood Explanation
`create_canister` and `update_settings` are publicly callable management canister methods reachable by any principal. The malicious payload is trivially constructable with any Candid encoder. No special credentials, on-chain state, or prior canister ownership is required. The attack is fully automatable and repeatable at the ingress submission rate.

## Recommendation
Apply the same `BoundedVec` pattern already used for `controllers` and `allowed_viewers`. Define a bounded type in `rs/types/management_canister_types/src/lib.rs`:

```rust
const MAX_ALLOWED_ENV_VARS_COUNT: usize = 32;
pub type BoundedEnvironmentVariables =
    BoundedVec<MAX_ALLOWED_ENV_VARS_COUNT, UNBOUNDED, UNBOUNDED, EnvironmentVariable>;
```

Replace `Option<Vec<EnvironmentVariable>>` with `Option<BoundedEnvironmentVariables>` in `CanisterSettingsArgs`. This enforces the count limit at Candid deserialization time, before any heap allocation or BTreeMap construction, consistent with how `controllers` and `allowed_viewers` are already protected. Additionally, consider applying per-element data-size bounds via `BoundedVec`'s `MAX_ALLOWED_ELEMENT_DATA_SIZE` parameter to bound name/value lengths at deserialization as well.

## Proof of Concept
A deterministic integration test using `PocketIC` or `StateMachine`:

```rust
// Construct ~200,000 minimal EnvironmentVariable records
let env_vars: Vec<EnvironmentVariable> = (0..200_000)
    .map(|_| EnvironmentVariable { name: "".to_string(), value: "".to_string() })
    .collect();
let settings = CanisterSettingsArgs {
    environment_variables: Some(env_vars),
    ..Default::default()
};
// Encode and submit as update_settings to the management canister.
// Observe: call is rejected with "Too many environment variables",
// but only after full deserialization and BTreeMap construction.
// Measure wall-clock time for the TryFrom conversion to confirm
// O(n log n) work occurs before rejection.
```

Alternatively, a unit test directly calling `CanisterSettings::try_from(args)` with 200,000 entries and timing the result confirms the unmetered work occurs before the error is returned.
