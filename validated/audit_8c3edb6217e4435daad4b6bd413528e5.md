### Title
`EvmErc20V2.withdrawToNear` Silently Burns User Tokens Without Completing NEP-141 Transfer Due to Input Format Mismatch - (File: `etc/eth-contracts/contracts/EvmErc20V2.sol`)

### Summary

`EvmErc20V2.withdrawToNear` encodes a `sender` address into the precompile calldata that the `ExitToNear` precompile does not consume when the `error_refund` feature is disabled. The precompile misparses the sender bytes as the token amount, causing `parse_amount` to reject the call. Because the Solidity assembly block does not check the return value, the function silently succeeds after burning the caller's ERC-20 tokens — permanently destroying them with no corresponding NEP-141 transfer.

### Finding Description

`EvmErc20V2.withdrawToNear` constructs its precompile input as:

```
\x01 | sender (20 bytes) | amount_b (32 bytes) | recipient (N bytes)
``` [1](#0-0) 

This layout is designed for the `error_refund` compilation feature, where `parse_input` strips the flag byte **and** the 20-byte refund address before handing the remainder to the flag dispatcher:

```rust
#[cfg(feature = "error_refund")]
fn parse_input(input: &[u8]) -> Result<(Address, &[u8]), ExitError> {
    ...
    Ok((refund_address, &input[21..]))   // strips flag + 20-byte address
}
``` [2](#0-1) 

Without `error_refund`, `parse_input` strips **only** the flag byte:

```rust
#[cfg(not(feature = "error_refund"))]
fn parse_input(input: &[u8]) -> Result<&[u8], ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    Ok(&input[1..])   // strips flag only
}
``` [3](#0-2) 

The flag-0x1 branch then reads `input[..32]` as the amount:

```rust
0x1 => {
    let amount = parse_amount(&input[..32])?;
``` [4](#0-3) 

Without `error_refund`, `input[..32]` is `sender_address (20 bytes) || amount_b[0..12]`. Any non-zero sender address places bits above bit 128, so `parse_amount` rejects it:

```rust
fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);
    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
    }
    Ok(amount)
}
``` [5](#0-4) 

The precompile call returns 0 (failure). The Solidity assembly block assigns the result to `res` but never checks it, so `withdrawToNear` returns normally:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
}
``` [6](#0-5) 

The `_burn` on line 55 has already executed and is not rolled back. The ERC-20 tokens are gone; the NEP-141 tokens are never transferred. [7](#0-6) 

By contrast, `EvmErc20` (V1) does **not** include `sender` in the input and works correctly regardless of the `error_refund` feature:

```solidity
bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
``` [8](#0-7) 

### Impact Explanation

Every call to `EvmErc20V2.withdrawToNear` from a non-zero address when `error_refund` is disabled results in **permanent, unrecoverable destruction** of the caller's ERC-20 tokens. The 1:1 accounting invariant between ERC-20 supply on Aurora and NEP-141 supply on NEAR is broken: ERC-20 tokens are burned but the corresponding NEP-141 tokens remain locked in the Aurora engine account forever. This is **permanent freezing of user funds** — Critical severity.

### Likelihood Explanation

Any user who holds `EvmErc20V2`-backed tokens and calls `withdrawToNear` triggers the loss. No special privilege is required. The only precondition is that the Aurora engine binary was compiled without the `error_refund` feature. The code explicitly supports both compilation modes (`#[cfg(feature = "error_refund")]` / `#[cfg(not(feature = "error_refund"))]`), making this a realistic deployment scenario.

### Recommendation

1. **In `EvmErc20V2.sol`**: Check the return value of the precompile `call` and revert if it fails, so the `_burn` is rolled back on precompile failure.
2. **In `EvmErc20V2.sol`**: Gate the `sender`-inclusive input format behind a compile-time or runtime flag that matches the engine's `error_refund` feature, or align the input format with `EvmErc20` (V1) when refund support is not needed.
3. **In `native.rs`**: Add an integration-level assertion or test that verifies `EvmErc20V2.withdrawToNear` succeeds end-to-end for both feature configurations.

### Proof of Concept

1. Deploy Aurora engine **without** the `error_refund` feature.
2. Register a NEP-141 token and deploy its `EvmErc20V2` mirror on Aurora.
3. Mint 1000 `EvmErc20V2` tokens to `alice` (EVM address `0xABCD...1234`).
4. Alice calls `EvmErc20V2.withdrawToNear("alice.near", 1000)`.
5. `_burn(alice, 1000)` executes — Alice's ERC-20 balance drops to 0.
6. Precompile receives input: `\x01 | 0xABCD...1234 (20 bytes) | 0x...03E8 (32 bytes) | "alice.near"`.
7. Without `error_refund`, `parse_input` strips only the flag; `parse_amount` reads `0xABCD...1234????...` — a value far above `u128::MAX` — and returns `ERR_INVALID_AMOUNT`.
8. The assembly `call` returns 0; `withdrawToNear` does not revert.
9. Alice's 1000 ERC-20 tokens are permanently destroyed. Her NEP-141 balance on NEAR remains 0. The NEP-141 tokens corresponding to her burned ERC-20 tokens are permanently locked in the Aurora engine account with no recovery path.
