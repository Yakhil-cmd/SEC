### Title
Deposit Blocklist Only Checks Ethereum Sender (`from_address`), Not Beneficiary IC Account — (File: `rs/ethereum/cketh/minter/src/deposit.rs`)

---

### Summary

The ckETH/ckERC20 minter enforces the OFAC Ethereum address blocklist only on the **sender** (`from_address`) of a deposit event. It does not check whether the **beneficiary** IC account (the `principal` + `subaccount` encoded in the Ethereum deposit transaction) is controlled by a sanctioned entity. A blocked Ethereum address can bypass this restriction entirely by using a proxy (non-blocked) Ethereum address to deposit ETH or ERC-20 tokens, while specifying their own IC principal as the beneficiary. The minter will mint ckETH/ckERC20 to the sanctioned entity's IC account without any blocklist rejection.

---

### Finding Description

In `register_deposit_events` in `rs/ethereum/cketh/minter/src/deposit.rs`, the only blocklist check performed during deposit processing is:

```rust
if crate::blocklist::is_blocked(&event.from_address()) {
    // mark as InvalidDeposit
} else {
    mutate_state(|s| process_event(s, event.into_deposit()));
}
``` [1](#0-0) 

The `event.beneficiary()` — the IC `Account { owner: principal, subaccount }` that will receive the minted tokens — is never checked against the blocklist. [2](#0-1) 

The `principal` and `subaccount` fields are **user-controlled**: they are encoded as `bytes32` arguments in the Ethereum `depositEth` or `depositErc20` call to the helper smart contract, and are emitted verbatim in the `ReceivedEthOrErc20` log event that the minter scrapes. [3](#0-2) 

The blocklist comment itself states: *"ETH is not accepted from nor sent to addresses on this list."* The "sent to" enforcement is only applied on the **withdrawal** path via `validate_address_as_destination`, which checks the Ethereum destination address. [4](#0-3) [5](#0-4) 

The deposit path has no symmetric check on the beneficiary.

---

### Impact Explanation

A sanctioned Ethereum address (present in `ETH_ADDRESS_BLOCKLIST`) can receive ckETH or ckERC20 tokens on the Internet Computer by:

1. Controlling an IC principal (trivially obtained — any IC identity).
2. Using a non-blocked Ethereum address (a proxy wallet) to call `depositEth(principal, subaccount)` or `depositErc20(token, amount, principal, subaccount)` on the helper smart contract.
3. The minter scrapes the log, sees a clean `from_address`, and mints ckETH/ckERC20 to the sanctioned entity's IC account without any rejection.

The sanctioned entity now holds chain-key tokens on the IC, fully bypassing the OFAC sanctions enforcement that the blocklist is designed to provide. This is a **chain-fusion mint bypass** — the minter mints tokens to a beneficiary it should have refused.

**Impact: Medium** — Sanctions enforcement is incomplete; sanctioned entities can receive and hold ckETH/ckERC20 on the IC.

---

### Likelihood Explanation

The attack requires no privileged access, no key compromise, and no governance majority. Any sanctioned entity that:
- Has or creates an IC principal (zero barrier),
- Has access to any non-blocked Ethereum address (trivially obtained via a new wallet),

can execute this bypass. The Ethereum helper contract imposes no restrictions on the `principal` argument. [6](#0-5) 

**Likelihood: Medium** — Technically trivial to execute; requires only a proxy Ethereum wallet and an IC identity.

---

### Recommendation

The minter should maintain a separate IC-principal-level denylist (analogous to the Ethereum address blocklist) and check `event.beneficiary().owner` against it before accepting a deposit event. Alternatively, the protocol documentation should explicitly acknowledge that the blocklist only prevents direct deposits from sanctioned Ethereum addresses and does not prevent sanctioned entities from receiving ckETH/ckERC20 via proxy deposits.

---

### Proof of Concept

1. Sanctioned address `0x0330070FD38Ec3bB94F58FA55D40368271E9e54A` (first entry in `ETH_ADDRESS_BLOCKLIST`) controls IC principal `P_sanctioned`.
2. Attacker creates a fresh Ethereum wallet `0xABCD...` (not on blocklist).
3. Attacker calls `depositEth(encode(P_sanctioned), 0x00...00)` from `0xABCD...` with 1 ETH.
4. Minter scrapes the `ReceivedEthOrErc20` log: `from_address = 0xABCD...`, `principal = P_sanctioned`.
5. `is_blocked(&0xABCD...)` returns `false` → deposit is accepted.
6. `mint()` calls `icrc1_transfer` with `to = Account { owner: P_sanctioned, subaccount: None }`.
7. ckETH is minted to `P_sanctioned` — the sanctioned entity now holds ckETH on the IC. [7](#0-6) [8](#0-7)
