### Title
Unimplemented `add_batch_actions` Stub in XCC Router Causes Reachable Panic and wNEAR Fund Freeze - (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router contract contains a production function `add_batch_actions` that is defined but not implemented when the `all-promise-actions` feature is absent (which is the default). An unprivileged EVM user can craft XCC calldata containing a `NearPromise::Simple(SimpleNearPromise::Batch(...))` or `NearPromise::Then { callback: SimpleNearPromise::Batch(...) }` payload. This triggers the `unimplemented!()` panic inside the router's `execute` call, while wNEAR already transferred in the EVM context is left frozen in the engine's implicit address with no refund path.

---

### Finding Description

In `etc/xcc-router/src/lib.rs`, the function `add_batch_actions` has two conditional implementations: [1](#0-0) 

When the `all-promise-actions` feature is **not** enabled (the default, per `Cargo.toml`): [2](#0-1) 

the function body is:

```rust
#[cfg(not(feature = "all-promise-actions"))]
fn add_batch_actions(_id: PromiseIndex, _actions: &[PromiseAction]) {
    unimplemented!("NEAR batch transactions are not supported...")
}
```

This stub is called from `recursive_promise_create` whenever a `NearPromise::Simple(SimpleNearPromise::Batch(...))` or a `NearPromise::Then { callback: SimpleNearPromise::Batch(...) }` variant is encountered: [3](#0-2) 

`recursive_promise_create` is called by `promise_create`, which is called by the public `execute` and `schedule` methods: [4](#0-3) 

**Attack path through the XCC precompile:**

The XCC precompile (`engine-precompiles/src/xcc.rs`) deserializes `CrossContractCallArgs` directly from EVM calldata: [5](#0-4) 

For an `Eager` call, the inner `PromiseArgs` (which can be `Recursive(NearPromise::Simple(SimpleNearPromise::Batch(...)))`) is Borsh-serialized and forwarded verbatim as the argument to the router's `execute` method: [6](#0-5) 

Before the NEAR promise is dispatched, the precompile performs an EVM-level `transferFrom` of wNEAR from the caller to the engine's implicit address when NEAR payment is required (e.g., first-time router deployment costs `STORAGE_AMOUNT` = 2 NEAR): [7](#0-6) 

This EVM state change is committed as part of the successful EVM execution. The subsequent NEAR promise calling `execute` on the router then panics due to `unimplemented!()`. The wNEAR already transferred to the engine's implicit EVM address has no automatic refund path — the `send_refund` function only refunds the NEAR storage amount to the parent, not the wNEAR ERC-20 balance: [8](#0-7) 

---

### Impact Explanation

**High — Temporary/Permanent Freezing of Funds.**

wNEAR ERC-20 tokens transferred from the user to the engine's implicit address during the XCC precompile execution are irrecoverable when the router's `execute` panics. The engine's implicit address holds the wNEAR balance in EVM state, but no EVM approval exists for the user to reclaim it, and no NEAR-level callback refunds the ERC-20 balance. For a first-time router deployment, this is at minimum 2 NEAR worth of wNEAR per attack invocation, plus any `attached_near` embedded in the crafted promise payload.

---

### Likelihood Explanation

**Medium.** Any EVM user with wNEAR balance can call the XCC precompile address (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`) with a Borsh-encoded `CrossContractCallArgs::Eager(PromiseArgs::Recursive(NearPromise::Simple(SimpleNearPromise::Batch(...))))` payload. No special privileges are required. The `all-promise-actions` feature is absent from the default build, so the stub is active in production deployments built with default features.

---

### Recommendation

Either:
1. Enable the `all-promise-actions` feature in the production build of the XCC router, or
2. Validate the `PromiseArgs` in the XCC precompile before dispatching, rejecting any `Recursive` variant that contains a `SimpleNearPromise::Batch` sub-promise when batch actions are unsupported, or
3. Replace the `unimplemented!()` stub with a graceful `env::panic_str(...)` and add a NEAR-level callback on the engine side that refunds the wNEAR to the caller on promise failure.

---

### Proof of Concept

1. Craft Borsh-encoded calldata for `CrossContractCallArgs::Eager(PromiseArgs::Recursive(NearPromise::Simple(SimpleNearPromise::Batch({ target_account_id: "any.near", actions: [PromiseAction::CreateAccount] }))))`.
2. Call the XCC precompile at `0x516cded1d16af10cad47d6d49128e2eb7d27b372` from an EVM account that holds wNEAR and has approved the XCC precompile.
3. The precompile executes `transferFrom` of wNEAR (≥ 2 NEAR for first-time router deployment) from the caller to the engine's implicit address — this EVM state change is committed.
4. The engine dispatches a NEAR promise calling `execute` on the router with the serialized `PromiseArgs`.
5. The router's `execute` → `promise_create` → `recursive_promise_create` → `add_batch_actions` hits `unimplemented!()` and panics.
6. The NEAR promise fails; the wNEAR remains frozen in the engine's implicit EVM address with no refund mechanism.

    fn recursive_promise_create(promise: &NearPromise) -> PromiseIndex {
        match promise {
            NearPromise::Simple(x) => match x {
                SimpleNearPromise::Create(call) => Self::base_promise_create(call),
                SimpleNearPromise::Batch(batch) => {
                    let target = batch.target_account_id.as_ref().parse().unwrap();
                    let id = env::promise_batch_create(&target);
                    Self::add_batch_actions(id, &batch.actions);
                    id
                }
            },
            NearPromise::Then { base, callback } => {
                let base_index = Self::recursive_promise_create(base);
                match callback {
                    SimpleNearPromise::Create(call) => env::promise_then(
                        base_index,
                        call.target_account_id.as_ref().parse().unwrap(),
                        call.method.as_str(),
                        &call.args,
                        NearToken::from_yoctonear(call.attached_balance.as_u128()),
                        Gas::from_gas(call.attached_gas.as_u64()),
                    ),
                    SimpleNearPromise::Batch(batch) => {
                        let id = env::promise_batch_then(
                            base_index,
                            &batch.target_account_id.as_ref().parse().unwrap(),
                        );
                        Self::add_batch_actions(id, &batch.actions);
                        id
                    }
                }
            }
            NearPromise::And(promises) => {
                let indices: Vec<PromiseIndex> = promises
                    .iter()
                    .map(Self::recursive_promise_create)
                    .collect();
                env::promise_and(&indices)
            }
        }
    }
