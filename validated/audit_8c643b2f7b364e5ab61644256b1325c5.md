### Title
Unchecked Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

In `EvmErc20V2.sol`, both `withdrawToNear` and `withdrawToEthereum` burn ERC-20 tokens via `_burn` before calling the Aurora exit precompile via inline assembly. The return value of the precompile `call` is stored in `res` but **never checked**. If the precompile call fails (e.g., due to an invalid NEAR recipient account ID), the ERC-20 tokens are permanently destroyed with no corresponding NEP-141 transfer on the NEAR side, resulting in permanent loss of user funds.

---

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) can return a non-fatal `ExitError` (mapped to `PrecompileFailure::Error`) in several conditions:

- Invalid NEAR recipient account ID (e.g., contains illegal characters)
- NEP-141 mapping not found for the calling ERC-20 address (`ERR_TARGET_TOKEN_NOT_FOUND`)
- Malformed input (e.g., missing flag byte, invalid amount encoding) [3](#0-2) 

In the EVM, a `PrecompileFailure::Error` causes the inner `call` to return `success = 0` without reverting the calling frame. Since `res` is never inspected, the `withdrawToNear`/`withdrawToEthereum` function returns successfully even though the NEAR-side transfer was never scheduled. The `_burn` that already executed is **not rolled back**.

The precompile dispatch in `engine-precompiles/src/lib.rs` confirms that only `PrecompileFailure::Fatal` (used for the paused-precompile case) propagates upward to revert the whole transaction; ordinary `ExitError` results do not: [4](#0-3) 

---

### Impact Explanation

**Critical â€” Permanent freezing of funds.**

When the precompile call fails silently:
- The user's ERC-20 mirror token balance is reduced to zero (burned).
- No NEP-141 `ft_transfer` or `ft_transfer_call` promise is ever created on the NEAR side.
- The NEP-141 tokens remain locked in the Aurora engine contract with no mechanism to recover them.
- The user has permanently lost their bridged assets with no recourse.

---

### Likelihood Explanation

**Medium.** The trigger is entirely user-controlled: any caller who supplies an invalid NEAR account ID string as `recipient` (e.e., one containing illegal characters like `!`, `@`, or exceeding 64 characters) will hit this path. No special privileges are required. The `_burn` check (OpenZeppelin) only validates that the caller has sufficient ERC-20 balance; it does not validate the NEAR recipient. A user making a typo in the recipient field, or a contract integrating `withdrawToNear` without pre-validating the account ID, will silently lose funds.

---

### Recommendation

Check the return value of the precompile `call` in both functions and revert if it indicates failure:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Applying the same fix to `withdrawToEthereum`:

```solidity
assembly {
    let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the exit precompile rejects the call for any reason, the entire transaction reverts (including the `_burn`), preserving the user's ERC-20 balance.

### Proof of Concept

1. User holds 1000 units of a bridged NEP-141 token as ERC-20 on Aurora (minted via `ft_transfer_call` into Aurora).
2. User calls `withdrawToNear("invalid!!account.near", 1000)` â€” the recipient string contains `!` which is illegal in a NEAR account ID.
3. `_burn(sender, 1000)` executes successfully; user's ERC-20 balance drops to 0.
4. The assembly `call` invokes the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`.
5. Inside `exit_erc20_token_to_near`, `parse_recipient` (or `AccountId` validation) rejects `"invalid!!account.near"` and returns `ExitError`.
6. The precompile dispatch wraps this as `PrecompileFailure::Error`; the EVM `call` returns `res = 0`.
7. The assembly block does not check `res`; `withdrawToNear` returns normally.
8. No NEAR promise is scheduled; the NEP-141 tokens remain locked in the Aurora contract forever.
9. User has 0 ERC-20 tokens and 0 NEP-141 tokens â€” 1000 units permanently lost. [1](#0-0) [3](#0-2) [5](#0-4)

fn process_precompile(
    p: &dyn Precompile,
    handle: &impl PrecompileHandle,
) -> Result<PrecompileOutput, PrecompileFailure> {
    let input = handle.input();
    let gas_limit = handle.gas_limit();
    let context = handle.context();
    let is_static = handle.is_static();

    p.run(input, gas_limit.map(EthGas::new), context, is_static)
        .map_err(|exit_status| PrecompileFailure::Error { exit_status })
}
