Audit Report

## Title
Unbounded ingress-expiry loop in `construction_payloads` allows OOM via attacker-controlled `ingress_start`/`ingress_end` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The ICP Rosetta `construction_payloads` handler reads `ingress_start` and `ingress_end` directly from the unauthenticated HTTP request body and iterates over the window with a fixed ~120-second step, pushing one `u64` per iteration into `ingress_expiries`. No server-side bound on the window size exists. An unauthenticated attacker can set `ingress_start=0` and `ingress_end=2^64-1`, causing ~153 billion loop iterations and exhausting process memory, crashing the Rosetta node.

## Finding Description
`interval` is computed as `MAX_INGRESS_TTL - PERMITTED_DRIFT - 120s ≈ 120,000,000,000 ns`. [1](#0-0) 

Both `ingress_start` and `ingress_end` are taken verbatim from `ConstructionPayloadsRequestMetadata` with no range validation: [2](#0-1) 

The loop at lines 99–107 runs `while now < ingress_end` with no iteration cap or window-size guard: [3](#0-2) 

After the loop, `add_payloads` is called for each transaction type and pushes **two** heap-allocated `SigningPayload` structs (containing hex-encoded strings) per entry in `ingress_expiries`: [4](#0-3) 

With `ingress_start=0` and `ingress_end=u64::MAX (18446744073709551615)`, the number of iterations is `u64::MAX / 120_000_000_000 ≈ 1.53 × 10^11`. The process OOMs before completing. The ICRC1 sibling at least validates `ingress_start < ingress_end`, which the ICP Rosetta does not: [5](#0-4) 

## Impact Explanation
An unauthenticated attacker can crash the ICP Rosetta process with a single HTTP POST, taking the node offline and preventing all ICP ledger operations (transfers, neuron management, staking) that depend on it. This matches the allowed High impact: **Application/platform-level DoS not based on raw volumetric DDoS**, with concrete user and protocol harm to the Rosetta financial integration component.

## Likelihood Explanation
The `/construction/payloads` endpoint is unauthenticated and publicly reachable. The exploit requires a single HTTP request with two integer fields set to extreme values. No credentials, special knowledge, or volume is needed. The attack is trivially repeatable and can keep the node permanently offline.

## Recommendation
Add a server-side cap on the ingress window before the loop:

```rust
let max_window = interval * 100; // allow at most 100 intervals (~3.3 hours)
if ingress_end.saturating_sub(ingress_start.as_nanos_since_unix_epoch())
    > max_window.as_nanos() as u64
{
    return Err(ApiError::invalid_request("ingress window too large"));
}
```

Alternatively, mirror the ICRC1 pattern by also adding `ingress_start >= ingress_end` rejection, and cap `ingress_expiries` to a fixed maximum (e.g., 100 entries) with an early error return.

## Proof of Concept
```
POST /construction/payloads
{
  "network_identifier": { ... },
  "operations": [{ "type": "REMOVE_HOTKEY", ... }],
  "public_keys": [{ ... }],
  "metadata": {
    "ingress_start": 0,
    "ingress_end": 18446744073709551615
  }
}
```
The Rosetta process enters the `while now < ingress_end` loop and allocates memory until OOM. A unit test asserting `ingress_expiries.len() <= 100` after calling `construction_payloads` with this input would fail on the current code.
