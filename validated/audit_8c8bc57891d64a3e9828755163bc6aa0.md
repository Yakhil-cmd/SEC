Audit Report

## Title
Unbounded `ingress_expiries` Vec allocation via attacker-controlled `ingress_start`/`ingress_end` in ICP Rosetta `/construction/payloads` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The ICP Rosetta `construction_payloads` handler accepts arbitrary `ingress_start` and `ingress_end` nanosecond timestamps from any unauthenticated HTTP client and feeds them directly into an unbounded `while` loop that allocates one `u64` per 120-second step. There is no cap on the window size and no `ingress_start < ingress_end` guard before allocation begins. A single crafted request with `ingress_start=0, ingress_end=u64::MAX` causes ~153 billion loop iterations and ~1.2 TB of attempted heap allocation, crashing the Rosetta server process via OOM.

## Finding Description
In `construction_payloads` at lines 59–107 of `rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`:

`interval` is fixed at `MAX_INGRESS_TTL(300s) - PERMITTED_DRIFT(60s) - 120s = 120s = 120,000,000,000 ns`. [1](#0-0) 

Both `ingress_start` and `ingress_end` are deserialized from caller-supplied JSON metadata as raw `u64` nanosecond values with no validation: [2](#0-1) 

The loop has no guard on `(ingress_end - ingress_start)`: [3](#0-2) 

With `ingress_start=0` and `ingress_end=u64::MAX (18_446_744_073_709_551_615)`, the iteration count is `18_446_744_073_709_551_615 / 120_000_000_000 ≈ 153_722_867_280` iterations, each pushing 8 bytes, totalling ~1.2 TB. The process OOMs before completing.

`ConstructionPayloadsRequestMetadata.ingress_start` and `.ingress_end` are plain `Option<u64>` with no range constraint at the type level: [4](#0-3) 

The ICRC1 Rosetta sibling function at `rs/rosetta-api/icrc1/src/construction_api/services.rs` has explicit guards rejecting `ingress_start >= ingress_end` and `ingress_end` too close to `now` before the loop: [5](#0-4) 

The ICP Rosetta handler has no equivalent guards whatsoever.

## Impact Explanation
An unauthenticated attacker can crash the ICP Rosetta server process (OOM kill) with a single HTTP POST to `/construction/payloads`. This denies service to all users of the Rosetta API (exchanges, wallets, integrators) until the process is restarted. Repeated requests prevent recovery. This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS** and **Significant Rosetta/financial-integration security impact with concrete user and protocol harm**.

## Likelihood Explanation
The `/construction/payloads` endpoint requires no authentication, no API key, and no prior state. The malicious payload is a small, valid JSON object. The attack is trivially reproducible locally and requires no special knowledge beyond the Rosetta API spec. Any external party can execute it.

## Recommendation
Add a window-size cap before the loop, mirroring the ICRC1 implementation:

```rust
if ingress_start >= ingress_end {
    return Err(ApiError::invalid_request(
        "ingress_start must be before ingress_end"
    ));
}
// Reject if window exceeds a reasonable maximum (e.g., 24 hours)
const MAX_INGRESS_WINDOW_NS: u64 = 24 * 60 * 60 * 1_000_000_000;
if ingress_end - ingress_start > MAX_INGRESS_WINDOW_NS {
    return Err(ApiError::invalid_request(
        "ingress_end - ingress_start exceeds maximum allowed window"
    ));
}
```

Alternatively, pre-compute the iteration count and reject if it exceeds a hard cap (e.g., 720 iterations for a 24-hour window at 120s intervals).

## Proof of Concept
```bash
curl -s -X POST http://<rosetta-host>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
    "operations": [{"operation_identifier":{"index":0},"type":"TRANSACTION",
      "account":{"address":"<valid_address>"},"amount":{"value":"-1","currency":{"symbol":"ICP","decimals":8}}}],
    "public_keys": [{"hex_bytes":"<valid_pubkey>","curve_type":"edwards25519"}],
    "metadata": {
      "ingress_start": 0,
      "ingress_end": 18446744073709551615
    }
  }'
# Expected: server RSS grows unboundedly; process is OOM-killed.
# Assert: server should return 4xx before allocating.
```

A unit test can also confirm this directly by calling `RosettaRequestHandler::construction_payloads` with `ingress_start=0` and `ingress_end=u64::MAX` and asserting it returns an `Err` before the loop executes.
