### Title
Unsanitized Memo Injection into Markdown `GenericDisplayMessage` Breaks Consent Message Integrity — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

### Summary
`add_memo` embeds attacker-controlled memo bytes directly into a backtick-delimited markdown code span with no escaping. A memo containing a backtick character terminates the code span early, allowing arbitrary markdown to be injected into the `GenericDisplayMessage` string. A malicious dApp can use this to display fake `**Amount:**`, `**To:**`, and `**Fees:**` fields that are visually indistinguishable from the ledger-generated ones.

### Finding Description

In `add_memo`, when the memo bytes are valid UTF-8, the raw string is interpolated directly into the markdown template:

```rust
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
``` [1](#0-0) 

No escaping of backticks, newlines, or any other markdown-special characters is performed. The only gate is `std::str::from_utf8`, which accepts any valid UTF-8 — including backticks and newlines. [2](#0-1) 

The `GenericDisplayMessage` variant is a raw markdown string, as evidenced by the ledger itself using `#` headers, `**bold**`, and backtick code spans throughout `add_intent`, `add_account`, `add_amount`, and `add_fee`: [3](#0-2) [4](#0-3) [5](#0-4) 

The `TransferArg.memo` field is decoded directly from the caller-supplied `arg` blob with no restriction on content: [6](#0-5) 

**Concrete injection:** A memo of `` `\n\n**Amount:** `1 ICP`\n\n**To:** `legitimate_address`\n\n**Ignore:** `` produces:

```
**Memo:**
``

**Amount:** `1 ICP`

**To:** `legitimate_address`

**Ignore:** `
```

The injected `**Amount:**` and `**To:**` lines are rendered identically to the real ledger-generated fields that appear earlier in the message.

### Impact Explanation

A malicious dApp constructs a `TransferArg` with a large real amount to the attacker's address, but a crafted memo that injects fake fields showing a small amount to a legitimate address. The wallet renders the `GenericDisplayMessage` as markdown and the user sees the injected (fake) fields. The user approves, believing they authorized a small payment to a known service; the actual on-chain transfer sends the large amount to the attacker. The `FieldsDisplayMessage` variant is **not** affected — the memo is stored as a structured `Value::Text` field there. [7](#0-6) 

### Likelihood Explanation

- Requires no privilege: any principal can call `icrc21_canister_call_consent_message` with an arbitrary `arg` blob.
- The memo field accepts up to 32 bytes (ICRC-1 spec), which is sufficient to inject a backtick plus a newline and a fake field label.
- Wallets that implement ICRC-21 and render `GenericDisplayMessage` as markdown (the intended rendering, given the format used) are directly affected.
- The attack is local-testable with a unit test.

### Recommendation

Escape backticks and newlines in `memo_str` before embedding it in the markdown template. The minimal fix is to replace all backtick characters with their escaped form and strip or replace newlines:

```rust
let safe_memo = memo_str.replace('`', "\\`").replace('\n', " ").replace('\r', "");
message.push_str(&format!("\n\n**Memo:**\n`{safe_memo}`"));
```

Alternatively, use a fenced code block (triple backtick) and escape any triple-backtick sequences in the memo, or switch to `FieldsDisplayMessage` for memo display.

### Proof of Concept

```rust
// memo = backtick + injected markdown
let memo_bytes = b"`\n\n**Amount:** `1 ICP`\n\n**To:** `attacker_address`\n\n**Ignore:**".to_vec();
let transfer_arg = TransferArg {
    memo: Some(Memo(memo_bytes)),
    amount: Nat::from(100_000_000_000u64), // 1000 ICP real amount
    to: attacker_account,
    ..Default::default()
};
// Call icrc21_canister_call_consent_message with Encode!(&transfer_arg)
// Assert: rendered GenericDisplayMessage contains injected "**Amount:**" outside the memo section
```

The resulting `GenericDisplayMessage` will contain two `**Amount:**` lines and two `**To:**` lines — the real ones (1000 ICP / attacker) and the injected ones (1 ICP / attacker_address) — with the injected ones appearing as normal markdown fields indistinguishable from ledger-generated content.
