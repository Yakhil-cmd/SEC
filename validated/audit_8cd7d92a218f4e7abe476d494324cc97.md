Audit Report

## Title
Hardcoded `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` Causes Permanent ckETH Loss and Withdrawal Blockage for High-Gas ERC-20 Tokens - (File: rs/ethereum/cketh/minter/src/withdraw.rs)

## Summary
The ckETH minter applies a single compile-time constant gas limit of `65_000` to every ckERC20 withdrawal transaction with no per-token override. Any ckERC20 token whose underlying ERC-20 `transfer()` execution requires more than 65,000 gas will have every Ethereum withdrawal transaction revert out-of-gas. On each such failure the minter reimburses only the ckERC20 token amount; the ckETH gas fee is permanently burned. Every subsequent withdrawal attempt by any user repeats the same failure, permanently blocking withdrawals for that token and draining ckETH from every user who tries.

## Finding Description
`CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` is set to `65_000` at compile time in `rs/ethereum/cketh/minter/src/withdraw.rs` lines 43–44. The function `estimate_gas_limit` at lines 296–301 of the same file returns this constant unconditionally for every `WithdrawalRequest::CkErc20` variant, with no per-token lookup or override path. `create_transactions_batch` (lines 249–294) calls `estimate_gas_limit` and passes the result directly into `create_transaction`, which embeds it as the EIP-1559 `gas_limit` field sent to Ethereum. The same constant is used in `estimate_erc20_transaction_fee` (main.rs lines 545–553) to compute how much ckETH is burned upfront, and in `eip_1559_transaction_price` (main.rs lines 173–188) for the public price-query endpoint.

The `CkErc20Token` struct (`rs/ethereum/cketh/minter/src/erc20.rs` lines 18–28) stores only `erc20_ethereum_network`, `erc20_contract_address`, `ckerc20_token_symbol`, and `ckerc20_ledger_id`; there is no `gas_limit` field, so no per-token configuration is possible.

When the Ethereum transaction reverts out-of-gas, `record_finalized_transaction` creates a `ReimbursementIndex::CkErc20` entry whose `reimbursed_amount` is the ckERC20 withdrawal amount only. The `process_reimbursement` function (withdraw.rs lines 46–142) dispatches this reimbursement to the ckERC20 ledger (`ledger_id`), not the ckETH ledger. No ckETH reimbursement entry is created. This is confirmed by the unit test `should_reimburse_tokens_when_ckerc20_withdrawal_fails` (state/transactions/tests.rs lines 1642–1687), which asserts that only a `ReimbursementIndex::CkErc20` request exists after a failed transaction, with `reimbursed_amount` equal to the ckERC20 withdrawal amount and no corresponding ckETH mint. The project documentation at `rs/ethereum/cketh/docs/ckerc20.adoc` line 275 explicitly states: "Overcharged transaction fees are not reimbursed."

Existing guards do not mitigate this: the `InsufficientTransactionFee` error path in `create_transactions_batch` only fires when the pre-burned ckETH fee is insufficient to cover the gas price estimate, not when the Ethereum transaction itself runs out of gas at execution time.

## Impact Explanation
Once a ckERC20 token whose underlying ERC-20 `transfer()` requires more than 65,000 gas is registered (a normal NNS governance operation), every unprivileged user who calls `withdraw_erc20` for that token suffers permanent, irreversible loss of ckETH gas fees. The withdrawal is permanently blocked for that token across all users. This constitutes a significant ck-token financial impact with concrete, repeatable user harm, matching the High bounty tier: "Significant Chain Fusion, ck-token, ledger… security impact with concrete user or protocol harm."

## Likelihood Explanation
The NNS governance process that adds ckERC20 tokens enforces no gas-usage check against the 65,000 limit. Many production ERC-20 tokens exceed this threshold: tokens with transfer hooks (ERC-777 adapters), fee-on-transfer mechanics, rebasing logic, or complex allowance/balance storage patterns routinely consume 70,000–120,000 gas per transfer. USDT on Ethereum mainnet has historically been near or above this boundary depending on storage slot initialization state. Once a single such token is added, exploitation requires no special privileges—any holder of that ckERC20 token can trigger the loss simply by calling `withdraw_erc20`. The failure is deterministic and repeatable on every attempt.

## Recommendation
1. Add an optional `erc20_gas_limit: Option<GasAmount>` field to `CkErc20Token` in `rs/ethereum/cketh/minter/src/erc20.rs`.
2. Update `estimate_gas_limit` in `rs/ethereum/cketh/minter/src/withdraw.rs` to use the per-token value when present, falling back to `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT`.
3. Update `eip_1559_transaction_price` and `estimate_erc20_transaction_fee` in `rs/ethereum/cketh/minter/src/main.rs` to use the per-token gas limit when querying for a specific token.
4. Update `add_ckerc20_token` and the `AddCkErc20Token` endpoint to accept and persist this optional field.
5. Consider reimbursing the ckETH gas fee (minus a small penalty) when the Ethereum transaction fails due to out-of-gas, to avoid penalizing users for a protocol-level misconfiguration.

## Proof of Concept
1. Submit an NNS governance proposal to add a ckERC20 token whose underlying ERC-20 `transfer(address,uint256)` requires, e.g., 80,000 gas (any fee-on-transfer or hook-bearing token).
2. As an unprivileged user, call `withdraw_erc20` on the ckETH minter with that token's ledger ID, a valid amount, and a valid Ethereum destination.
3. Observe: the minter burns `erc20_tx_fee` ckETH (computed via `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT = 65_000`) and burns the ckERC20 withdrawal amount.
4. The minter constructs an EIP-1559 transaction with `gas_limit = 65_000` via `create_transactions_batch` → `estimate_gas_limit` → `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT`.
5. The Ethereum transaction reverts out-of-gas.
6. The minter creates a `ReimbursementIndex::CkErc20` entry and mints back only the ckERC20 amount; no ckETH is returned.
7. Repeat step 2: identical outcome, additional ckETH permanently lost.

A deterministic integration test can be written using the existing `StateMachine`/PocketIC harness (see `rs/ethereum/cketh/minter/tests/ckerc20.rs`) by registering a mock ERC-20 contract that consumes >65,000 gas in its `transfer()` function, executing a withdrawal, and asserting that the ckETH balance decreases by the gas fee with no corresponding reimbursement mint on the ckETH ledger.
