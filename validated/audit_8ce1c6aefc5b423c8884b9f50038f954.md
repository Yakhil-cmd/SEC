### Title
`set_owner` Performs a One-Step Irreversible Owner Transfer, Permanently Locking All Owner-Gated Functions on Mistake - (File: `engine/src/contract_methods/admin.rs`)

---

### Summary

The `set_owner` function in the Aurora Engine immediately overwrites `state.owner_id` with the caller-supplied `new_owner` in a single atomic step. There is no pending-owner staging, no acceptance step from the new address, and no timelock. If an incorrect or inaccessible NEAR account ID is supplied, all owner-gated administrative functions — including `resume_contract`, `upgrade`, `stage_upgrade`, `set_upgrade_delay_blocks`, `resume_precompiles`, `set_key_manager`, and `attach_full_access_key` — become permanently unreachable, with no on-chain recovery path.

---

### Finding Description

`set_owner` reads the new owner from calldata and writes it directly to persistent state in one transaction:

```rust
// engine/src/contract_methods/admin.rs, lines 103-121
pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;

        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }

        state.owner_id = args.new_owner;   // ← immediate, unconditional overwrite
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
``` [1](#0-0) 

The only guard is `require_owner_only`, which checks `state.owner_id == predecessor_account_id`:

```rust
// engine/src/contract_methods/mod.rs, lines 79-87
pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}
``` [2](#0-1) 

Once `state.owner_id` is overwritten with an inaccessible account, the same `require_owner_only` guard blocks every subsequent owner-gated call — including another call to `set_owner` to correct the mistake. There is no secondary recovery mechanism in the contract.

The full set of functions gated exclusively by `require_owner_only` that become permanently inaccessible:

| Function | Consequence if locked |
|---|---|
| `set_owner` | Cannot correct the mistake |
| `resume_contract` | Contract cannot be unpaused → permanent fund freeze |
| `upgrade` / `stage_upgrade` | Contract cannot be patched |
| `set_upgrade_delay_blocks` | Upgrade delay cannot be changed |
| `resume_precompiles` | Paused precompiles cannot be re-enabled |
| `set_key_manager` | Relayer key manager cannot be rotated |
| `attach_full_access_key` | No new full-access key can be added | [3](#0-2) [4](#0-3) [5](#0-4)

### Impact Explanation

**Critical — Permanent freezing of funds.**

The direct path to a permanent fund freeze is:

1. The current owner calls `set_owner` with a typo'd or non-existent NEAR account ID (e.g., `aurora-dao.near` instead of `aurora-dao.sputnik-dao.near`). The call succeeds immediately.
2. `state.owner_id` now points to an account for which no private key exists.
3. The contract is subsequently paused — either by the old owner in the same block before noticing the mistake, or by any future governance action that triggers `pause_contract` through the (now inaccessible) new owner.
4. `resume_contract` requires `require_owner_only` and will permanently revert with `ERR_NOT_ALLOWED`.
5. All ETH and ERC-20 token balances held in the Aurora Engine are permanently frozen with no on-chain recovery path.

Even without a pause event, the permanent loss of `upgrade` capability means any subsequently discovered vulnerability in the engine cannot be patched, creating a latent path to fund theft. [6](#0-5) 

---

### Likelihood Explanation

**Medium.** The Aurora Engine owner is a governance-controlled NEAR account (e.g., a DAO or multisig). Owner changes require a governance vote and a signed transaction. A single character typo in a NEAR account ID (which are human-readable strings, not checksummed hex addresses) is a realistic operational error. NEAR account IDs have no checksum mechanism analogous to EIP-55, so a mistyped ID that happens to be a valid-format string (e.g., passes `AccountId` parsing) will be silently accepted. The `ERR_SAME_OWNER` guard only prevents setting the same owner, not an invalid one. [7](#0-6) [8](#0-7) 

---

### Recommendation

Replace the single-step transfer with a two-step commit-accept pattern:

1. **Propose** (`set_owner`): The current owner writes a `pending_owner: Option<AccountId>` field to state. The existing `owner_id` is unchanged.
2. **Accept** (`accept_owner`): Only the account matching `pending_owner` can call this function, which atomically moves `pending_owner` into `owner_id` and clears the pending field.

A mistake in step 1 is corrected by calling `set_owner` again with the correct address before the new account accepts. Optionally, a block-height timelock on acceptance provides an additional safety window.

---

### Proof of Concept

```
1. Current owner_id = "aurora-dao.sputnik-dao.near"
2. Owner submits: set_owner { new_owner: "aurora-dao.near" }  // typo — valid format, wrong account
3. Transaction succeeds; state.owner_id = "aurora-dao.near"
4. "aurora-dao.sputnik-dao.near" calls set_owner again to fix → ERR_NOT_ALLOWED
5. "aurora-dao.sputnik-dao.near" calls resume_contract (if paused) → ERR_NOT_ALLOWED
6. "aurora-dao.sputnik-dao.near" calls upgrade → ERR_NOT_ALLOWED
7. No on-chain recovery path exists. All owner-gated functions are permanently locked.
``` [9](#0-8) [2](#0-1) [10](#0-9)

        public_key,
        nonce: 0, // not actually used - depends on block height

        target_account_id: current_account_id,

    let promise_id = handler.promise_create_batch(&promise);

    Ok(())
