### Title
wNEAR Permanently Stranded in Engine Implicit Address on Router Deploy Failure — (`engine/src/contract_methods/xcc.rs`, `engine-precompiles/src/xcc.rs`)

### Summary

When a user invokes the XCC precompile for the first time (or when a router upgrade is needed), the precompile immediately transfers wNEAR from the user's EVM balance to the engine's implicit EVM address within the same EVM receipt. The subsequent NEAR promise chain — deploy router → `factory_update_address_version` → `withdraw_wnear_to_router` → user call — is scheduled across separate NEAR receipts. If the deploy batch fails, `withdraw_wnear_to_router` is invoked as a callback with `promise_result_check()` returning `Some(false)`, causing an early return before the EVM call that would burn/forward the wNEAR. Because the original EVM receipt already committed the `transferFrom`, the wNEAR is left stranded in the engine's implicit address with no automated refund path.

### Finding Description

**Step 1 — wNEAR deducted in the original EVM receipt**

Inside `CrossContractCall::run_with_handle`, when `required_near != ZERO_YOCTO`, the precompile executes an inline EVM sub-call:

```
transferFrom(sender, engine_implicit_address, required_near)
```

on the wNEAR ERC-20 contract. This state change is committed as part of the user's EVM transaction receipt. [1](#0-0) 

**Step 2 — Promise chain scheduled across separate NEAR receipts**

`handle_precompile_promise` in `engine/src/xcc.rs` builds the following chain when a new router sub-account must be created:

```
[deploy batch] → factory_update_address_version (callback)
              → withdraw_wnear_to_router        (callback)
              → user's target call              (callback)
```

`withdraw_wnear_to_router` is attached as a callback to `setup_id`, which is the `factory_update_address_version` promise. [2](#0-1) 

**Step 3 — Deploy failure propagates to `withdraw_wnear_to_router`**

If the deploy batch fails (e.g., `initialize` panics, insufficient gas, or the engine's NEAR balance is too low to cover `STORAGE_AMOUNT`), `factory_update_address_version` is called with `Some(false)` and panics with `ERR_ROUTER_DEPLOY_FAILED`. [3](#0-2) 

In NEAR, a callback is always invoked regardless of whether its parent promise succeeded or failed. So `withdraw_wnear_to_router` is then called as a callback to the failed `factory_update_address_version` receipt.

**Step 4 — Early return without executing the EVM withdrawal**

`withdraw_wnear_to_router` checks `promise_result_check()` and returns `Err(b"ERR_CALLBACK_OF_FAILED_PROMISE")` before reaching the EVM call that would burn the wNEAR and forward NEAR to the router sub-account: [4](#0-3) 

The EVM call at lines 51–57 is never reached: [5](#0-4) 

**Step 5 — No refund path**

There is no code anywhere in the callback chain that refunds the wNEAR from the engine's implicit address back to the user. The `sdk_unwrap()` panic in `lib.rs` causes the NEAR receipt to fail and reverts only the state changes of that receipt — it does not revert the original EVM receipt where the `transferFrom` was committed. [6](#0-5) 

### Impact Explanation

The user's wNEAR is debited from their EVM balance in Receipt 1 and credited to the engine's implicit EVM address. When the downstream NEAR receipt for `withdraw_wnear_to_router` fails, those tokens remain in the engine's implicit address indefinitely. The engine's implicit address is not controlled by any user key; only an admin-level `call` transaction could move those tokens. There is no automated recovery. This constitutes a **High — Temporary freezing of funds**.

### Likelihood Explanation

The trigger condition is a failure of the router sub-account deploy batch. Realistic causes include:

- The engine's NEAR balance falls below `STORAGE_AMOUNT` (2 NEAR) at the time of the deploy.
- The router `initialize` function panics (e.g., the wNEAR NEP-141 account is not registered or the router WASM has a bug).
- Insufficient gas allocated to the `initialize` call (`INITIALIZE_GAS = 15 TGas`) for a given router version.

Any user performing their first XCC call (or after a router upgrade) is exposed. The condition is not attacker-controlled but is a realistic operational failure mode.

### Recommendation

Add a refund callback after `withdraw_wnear_to_router` that, on failure, transfers the stranded wNEAR from the engine's implicit address back to the original sender. Alternatively, restructure the promise chain so that the wNEAR `transferFrom` is only executed inside `withdraw_wnear_to_router` itself (i.e., move the deduction from the precompile into the callback), so that a deploy failure causes the entire chain to revert atomically without any pre-committed EVM state change.

### Proof of Concept

1. Deploy Aurora Engine locally with a wNEAR ERC-20 bridge and XCC router code loaded.
2. Fund a test account with wNEAR in the EVM; approve the XCC precompile.
3. Patch the router's `initialize` function to unconditionally panic (simulating a deploy failure).
4. Submit an EVM transaction calling the XCC precompile with `CrossContractCallArgs::Eager` targeting any NEAR account. `required_near` will include `STORAGE_AMOUNT` because no router exists yet.
5. Observe that the `transferFrom` in the precompile succeeds and the user's wNEAR balance is reduced.
6. Observe that the deploy batch receipt fails, `factory_update_address_version` panics, and `withdraw_wnear_to_router` is called with `promise_result_check() == Some(false)`, returning `ERR_CALLBACK_OF_FAILED_PROMISE`.
7. Confirm that the engine's implicit EVM address now holds the user's wNEAR with no mechanism to return it.
