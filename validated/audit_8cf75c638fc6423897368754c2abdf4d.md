### Title
Unchecked `transferFrom` Return Value in XCC Precompile Allows Bypassing wNEAR Payment - (File: `engine-precompiles/src/xcc.rs`)

### Summary
The Cross-Contract Call (XCC) precompile calls `transferFrom` on the wNEAR ERC-20 contract to collect payment from the user before executing a NEAR cross-contract call. The EVM-level exit reason is checked, but the ABI-encoded boolean return value of `transferFrom` is never decoded or validated. If the wNEAR contract returns `false` instead of reverting on a failed transfer, the precompile silently proceeds, emitting the promise log and triggering the XCC without the user having paid.

### Finding Description
In `engine-precompiles/src/xcc.rs`, when `required_near != ZERO_YOCTO`, the precompile constructs a `transferFrom(sender, engine_implicit_address, amount)` call against the wNEAR ERC-20 contract and dispatches it via `handle.call(...)`. [1](#0-0) 

The result is destructured as `(exit_reason, return_value)`. The `match` on `exit_reason` correctly handles `Revert`, `Error`, and `Fatal` outcomes, but when the outcome is `ExitReason::Succeed(_)`, execution falls through with no further check. The variable `return_value` — which holds the ABI-encoded `bool` returned by `transferFrom` — is never decoded or inspected. [2](#0-1) 

Per the ERC-20 specification, `transferFrom` is permitted to return `false` on failure rather than reverting. If the wNEAR contract behaves this way (e.g., due to insufficient allowance or balance), the EVM call exits with `Succeed` and a return value of `abi.encode(false)`. The precompile interprets this as a successful payment and continues to emit the promise log. [3](#0-2) 

The promise log is what the Aurora Engine reads to schedule the actual NEAR cross-contract call, funded from the engine's own NEAR balance. [4](#0-3) 

The `transfer_from_args` helper correctly encodes the `transferFrom` selector and arguments, confirming this is a real ERC-20 call whose boolean return value is expected but ignored. [5](#0-4) 

### Impact Explanation
An attacker who can cause the wNEAR `transferFrom` to return `false` (e.g., by having zero allowance set for the XCC precompile address, or by interacting with a wNEAR deployment that follows the return-false path) can trigger XCC calls — including first-time router deployment costing `STORAGE_AMOUNT = 2 NEAR` — without paying. Repeated invocations drain the Aurora Engine's NEAR balance, constituting direct theft of protocol funds at rest.

**Impact: Critical — direct theft of engine-held NEAR funds.**

### Likelihood Explanation
The wNEAR address is read from mutable engine storage (`state::get_wnear_address`). [6](#0-5) [7](#0-6) 

The currently deployed wNEAR contract is likely OpenZeppelin-based and reverts on failure, making exploitation unlikely under normal conditions. However, the code is structurally unsound: any wNEAR contract that is ERC-20 compliant but returns `false` on failure (which is explicitly permitted by the standard) would make this immediately exploitable by any EVM user who calls the XCC precompile without a sufficient wNEAR allowance or balance.

**Likelihood: Medium** — depends on wNEAR implementation behavior, but the root cause is a code-level invariant violation that requires no special privilege to trigger.

### Recommendation
After the `ExitReason::Succeed(_)` branch, decode `return_value` and revert if the boolean is `false`:

```rust
aurora_evm::ExitReason::Succeed(_) => {
    // ERC-20 transferFrom returns abi.encode(bool)
    let success = return_value.last().copied().unwrap_or(0) != 0;
    if !success {
        return Err(revert_with_message("ERR_WNEAR_TRANSFER_FAILED"));
    }
}
```

Alternatively, use a `SafeERC20`-style wrapper or assert that the return data decodes to `true` before proceeding.

### Proof of Concept

1. Deploy Aurora Engine with a wNEAR ERC-20 that returns `false` on `transferFrom` failure (valid per ERC-20 spec) instead of reverting.
2. As an unprivileged EVM user, call the XCC precompile (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`) with a valid `CrossContractCallArgs::Eager` payload and zero wNEAR allowance granted to the precompile.
3. The `handle.call(wnear_address, ...)` returns `(ExitReason::Succeed(_), abi.encode(false))`.
4. The `match` arm `Succeed(_) => ()` fires; `return_value` is discarded.
5. The promise log is emitted; the Aurora Engine schedules the NEAR XCC and pays `STORAGE_AMOUNT` (2 NEAR) from its own balance.
6. The attacker's wNEAR balance is unchanged; the engine's NEAR balance is reduced by 2 NEAR per call.
7. Repeat to drain the engine's NEAR balance. [8](#0-7)
