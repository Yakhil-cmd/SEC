Audit Report

## Title
In-Batch Dedup Skipped on `HandleInvalid` Enables O(K) `crypto_verify_complaint()` Calls Per Cycle — (`rs/consensus/idkg/src/complaints.rs`)

## Summary

`validate_complaints` inserts a `ComplaintKey` into `validated_complaints` only on `MoveToValidated`, never on `HandleInvalid`. A Byzantine subnet node can flood the unvalidated pool with K complaints sharing the same `(transcript_id, dealer_id, complainer_id)` but distinct `internal_complaint_raw` bytes; each carries a valid BLS signature over invalid content. In a single `on_state_change` cycle every one of the K entries passes both dedup checks and triggers a full `crypto_verify_complaint()` call, enabling O(K) expensive threshold-crypto operations per cycle and stalling the complaint handler.

## Finding Description

In `validate_complaints`, the in-batch dedup set is only updated on success:

```rust
let action = self.crypto_verify_complaint(id, transcript, signed_complaint);
if let Some(IDkgChangeAction::MoveToValidated(_)) = action {
    validated_complaints.insert(key);   // never reached on HandleInvalid
}
``` [1](#0-0) 

When `crypto_verify_complaint` returns `HandleInvalid` (permanent error path at lines 394 and 423), the key is not inserted, so the next complaint with the same `ComplaintKey` but different bytes passes the `validated_complaints.contains(&key)` check at line 125 and proceeds to another full crypto call. [2](#0-1) 

The cross-batch guard `has_complainer_issued_complaint` queries only the **validated** pool, so it also provides no protection against unvalidated duplicates: [3](#0-2) 

The unvalidated pool is keyed by `IDkgArtifactId::from(&message)`, which is content-derived, so K complaints with distinct `internal_complaint_raw` bytes produce K distinct IDs and are all admitted without collision: [4](#0-3) 

`crypto_verify_complaint` performs two expensive operations per call: a BLS signature verification and a full `verify_complaint` over the transcript: [5](#0-4) 

## Impact Explanation

O(K) threshold-crypto operations per `on_state_change` cycle stalls the complaint handler, blocking transcript loading and threshold-signature availability for the affected subnet. This matches the **High** bounty impact: *"Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

A single Byzantine subnet node (within the f < n/3 fault budget) suffices. The attacker signs K complaints with valid BLS signatures over crafted `internal_complaint_raw` bytes, gossips them via normal P2P, and the attack repeats every `on_state_change` cycle. No threshold corruption, no key leakage, no external dependency is required. The unvalidated pool has no observed per-sender admission cap.

## Recommendation

Insert the key into `validated_complaints` on **any terminal outcome**, not only on `MoveToValidated`:

```rust
let action = self.crypto_verify_complaint(id, transcript, signed_complaint);
match &action {
    Some(IDkgChangeAction::MoveToValidated(_))
    | Some(IDkgChangeAction::HandleInvalid(_, _)) => {
        validated_complaints.insert(key);
    }
    _ => {}
}
```

This bounds crypto calls to O(1) per `(complainer, transcript, dealer)` per cycle regardless of unvalidated pool size. [6](#0-5) 

## Proof of Concept

Inject 1000 `SignedIDkgComplaint` messages into the unvalidated pool, all sharing the same `(transcript_id, dealer_id, complainer_id)` but with distinct `internal_complaint_raw` bytes and valid BLS signatures over invalid complaint content. Call `validate_complaints` once. Assert that the crypto mock's `verify_complaint` invocation counter equals exactly 1000. Apply the fix; assert the counter equals 1. [7](#0-6)

    fn validate_complaints(
        &self,
        idkg_pool: &dyn IDkgPool,
        block_reader: &dyn IDkgBlockReader,
        active_transcripts: &BTreeMap<IDkgTranscriptId, &IDkgTranscript>,
    ) -> IDkgChangeSet {
        let requested_transcripts = self.requested_transcripts(block_reader);

        // Collection of validated complaints <complainer Id, transcript Id, dealer Id>
        let mut validated_complaints = BTreeSet::new();

        let mut ret = Vec::new();
        for (id, signed_complaint) in idkg_pool.unvalidated().complaints() {
            let complaint = signed_complaint.get();
            // Remove the duplicate entries
            let key = ComplaintKey::from(&signed_complaint);
            if validated_complaints.contains(&key) {
                self.metrics
                    .complaint_errors_inc("duplicate_complaints_in_batch");
                ret.push(IDkgChangeAction::RemoveUnvalidated(id));
                continue;
            }

            match Action::action(
                block_reader,
                active_transcripts,
                &requested_transcripts,
                complaint.idkg_complaint.transcript_id.source_height(),
                &complaint.idkg_complaint.transcript_id,
            ) {
                Action::Process(transcript) => {
                    if self.has_complainer_issued_complaint(
                        idkg_pool,
                        &complaint.idkg_complaint,
                        &signed_complaint.signature.signer,
                    ) {
                        self.metrics.complaint_errors_inc("duplicate_complaint");
                        ret.push(IDkgChangeAction::RemoveUnvalidated(id));
                    } else {
                        let action = self.crypto_verify_complaint(id, transcript, signed_complaint);
                        if let Some(IDkgChangeAction::MoveToValidated(_)) = action {
                            validated_complaints.insert(key);
                        }
                        ret.append(&mut action.into_iter().collect());
                    }
                }
                Action::Drop => ret.push(IDkgChangeAction::RemoveUnvalidated(id)),
                Action::Defer => {}
            }
        }

        ret
    }
