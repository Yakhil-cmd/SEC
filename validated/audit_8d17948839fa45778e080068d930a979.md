Audit Report

## Title
VetKD Share Verification Accepts Cryptographically Invalid Content — (`rs/crypto/src/vetkd/mod.rs`)

## Summary

`verify_encrypted_key_share_internal` verifies only the Ed25519 node signature over the raw bytes of `encrypted_key_share` without ever calling `EncryptedKeyShare::deserialize` to validate the content. A single Byzantine subnet node can sign empty or malformed bytes with its legitimate key, pass share verification, enter the validated pool, and then cause every subsequent `combine_encrypted_key_shares_internal` call to abort at the deserialization step — permanently blocking VetKD key derivation for any targeted request.

## Finding Description

**Verification gap.** `verify_encrypted_key_share_internal` performs exactly two operations: it reads the registry version from the threshold-sig data store, then calls `BasicSigVerifierInternal::verify_basic_sig` over `key_share.encrypted_key_share`. There is no call to `EncryptedKeyShare::deserialize` and no check that the bytes represent a valid BLS12-381 group element triple. If the Ed25519 signature is valid over empty bytes, the function returns `Ok(())`. [1](#0-0) 

**Deserialization enforced only at combination time.** `combine_encrypted_key_shares_internal` iterates over every share in the validated pool and calls `EncryptedKeyShare::deserialize` on each one. The `.collect::<Result<_, _>>()?` on line 366 means a single deserialization failure aborts the entire combination, mapping to `VetKdKeyShareCombinationError::InvalidArgumentEncryptedKeyShare`. [2](#0-1) 

`EncryptedKeyShare::deserialize` immediately rejects any byte slice whose length differs from `Self::BYTES` (= `2 * G1Affine::BYTES + G2Affine::BYTES`). [3](#0-2) 

**Fallback path is unreachable for malformed shares.** The fallback to `EncryptedKey::combine_valid_shares` (which filters per-share validity) is only triggered when `combine_all` returns `EncryptedKeyCombinationError::InvalidShares`. The deserialization failure occurs at line 366, before `combine_all` is ever called, so the fallback is structurally unreachable for malformed shares. [4](#0-3) 

**Validated pool accepts the malformed share.** In `signer.rs`, `validate_signature_share` calls `VetKdProtocol::verify_encrypted_key_share`; on `Ok(())` it returns `Ok(IDkgMessage::VetKdKeyShare(share))`, which the outer handler promotes to `IDkgChangeAction::MoveToValidated(share)`. [5](#0-4) [6](#0-5) 

**Combiner uses all validated shares.** `combine_shares` in the chain-key payload builder passes the full validated share map directly to `combine_encrypted_key_shares`. [7](#0-6) 

## Impact Explanation

A single Byzantine subnet node below the fault threshold can permanently block VetKD key derivation for any targeted canister request. The malformed share, once in the validated pool, is included in every combination attempt. Because deserialization fails before any filtering occurs, no amount of honest shares can rescue the combination. This is a targeted, persistent application/platform-level DoS on VetKD key derivation — matching the **High ($2,000–$10,000)** impact class: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* [8](#0-7) 

## Likelihood Explanation

Any single compromised or malicious subnet node can execute this attack. The node requires only its own registered Ed25519 signing key (which it legitimately holds) and the ability to craft a `VetKdKeyShare` protobuf with an empty or wrong-length `encrypted_key_share` field. The protobuf deserialization at the P2P layer accepts arbitrary byte slices for this field. No threshold corruption, no special privileges, and no external dependencies are required. The attack is repeatable and can target any active VetKD derivation request. [9](#0-8) 

## Recommendation

Add `EncryptedKeyShare::deserialize` validation inside `verify_encrypted_key_share_internal`, immediately after the signature check. If deserialization fails, return a `VetKdKeyShareVerificationError` — either a new `InvalidEncryptedKeyShareContent` variant or reuse `VerificationError` with an appropriate `CryptoError::InvalidArgument`. This ensures any share that cannot be deserialized is rejected at admission time and never enters the validated pool. [10](#0-9) 

## Proof of Concept

```rust
// Unit test sketch
let byzantine_node_id = /* valid subnet node with registered Ed25519 key */;
let empty_content = VetKdEncryptedKeyShareContent(vec![]);
// Sign the empty bytes with the node's legitimate Ed25519 key
let sig = ed25519_sign(&empty_content.0, &byzantine_node_sk);
let malformed_share = VetKdEncryptedKeyShare {
    encrypted_key_share: empty_content,
    node_signature: sig,
};
// verify_encrypted_key_share returns Ok(()) — signature is valid over empty bytes
assert_eq!(
    crypto.verify_encrypted_key_share(byzantine_node_id, &malformed_share, &args),
    Ok(())  // BUG: should be Err(...)
);
// combine_encrypted_key_shares fails when this share is included,
// even with threshold-many honest shares present
let mut shares = honest_shares; // threshold-many valid shares
shares.insert(byzantine_node_id, malformed_share);
assert!(matches!(
    crypto.combine_encrypted_key_shares(&shares, &args),
    Err(VetKdKeyShareCombinationError::InvalidArgumentEncryptedKeyShare)
));
```

A deterministic integration test can be written using `PocketIC` or the existing crypto test harness: register a node, have it submit a `VetKdKeyShare` with a zero-length `encrypted_key_share` signed with its real key, confirm `verify_encrypted_key_share` returns `Ok(())`, add the share to the pool alongside threshold-many honest shares, and confirm `combine_encrypted_key_shares` returns `Err(InvalidArgumentEncryptedKeyShare)`.
