### Title
ERC-20 Tokens Permanently Burned When NEP-141 `ft_transfer` Fails Without `error_refund` Feature - (`engine-precompiles/src/native.rs`)

### Summary
The `ExitToNear` precompile burns a user's ERC-20 tokens during EVM execution and then schedules a NEAR cross-contract `ft_transfer` call on the corresponding NEP-141 token. When the `error_refund` compile-time feature is disabled, no callback is registered to handle a failed `ft_transfer`. If the underlying NEP-141 token is paused (a standard capability in many NEP-141 implementations), the `ft_transfer` fails silently and the burned ERC-20 tokens are permanently lost with no refund path.

### Finding Description
In `engine-precompiles/src/native.rs`, the `ExitToNear::run()` function constructs `callback_args` as follows:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None without the feature
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For a standard ERC-20 exit (not wNEAR unwrap), `transfer_near_args` is also `None`, so `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`. The promise dispatch logic then schedules **only** the `ft_transfer` promise with no callback:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback registered
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

The ERC-20 burn happens atomically inside the EVM execution that triggers the precompile. The `ft_transfer` on the NEP-141 is a separate asynchronous NEAR promise. If that promise fails (e.g., because the NEP-141 contract is paused), there is no registered callback to re-mint the burned tokens.

The callback handler `exit_to_near_precompile_callback` in `engine/src/contract_methods/connector.rs` confirms the refund path only executes when `args.refund` is `Some`:

```rust
} else if let Some(args) = args.refund {
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← no refund, tokens are gone
}
``` [3](#0-2) 

The engine test suite explicitly documents this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

### Impact Explanation
**Critical — Permanent loss of funds.** When `error_refund` is not compiled in, any failure of the NEP-141 `ft_transfer` (including a paused token) results in the user's ERC-20 tokens being permanently burned with no recovery path. The NEP-141 balance held by the Aurora contract is unaffected, creating an accounting discrepancy: Aurora holds NEP-141 tokens that no ERC-20 holder can ever claim.

### Likelihood Explanation
**Medium.** Many NEP-141 tokens deployed on NEAR include a pause/stop mechanism (analogous to the stETH pattern described in the report). Any such token bridged into Aurora as an ERC-20 is affected. A user who attempts to exit during a pause window loses their tokens permanently. The `error_refund` feature being a compile-time flag means the exposure depends on the production build configuration, which could not be confirmed from the available files.

### Recommendation
1. **Enable `error_refund` unconditionally** in the production build, or promote it to always-on behavior rather than a feature flag.
2. Alternatively, before burning ERC-20 tokens, verify that the NEP-141 contract is not paused by querying its state (analogous to the `require(!stETH.isStopped())` recommendation in the original report).
3. At minimum, always register the callback promise so that a failed `ft_transfer` can trigger a re-mint of the burned ERC-20 tokens, regardless of the feature flag state.

### Proof of Concept
1. Deploy a NEP-141 token with a pause mechanism and bridge it to Aurora (creating an ERC-20 mirror).
2. Pause the NEP-141 token.
3. As an ERC-20 holder, call `withdrawToNear` on the ERC-20 contract, triggering the `ExitToNear` precompile.
4. The EVM execution burns the ERC-20 tokens and schedules `ft_transfer` on the paused NEP-141.
5. The `ft_transfer` promise fails because the NEP-141 is paused.
6. With `error_refund` disabled: no callback is registered; the burned ERC-20 tokens are gone. The user's balance is zero on Aurora, and the NEP-141 tokens remain locked in the Aurora contract with no claimant.
7. Confirmed by the test at `engine-tests/src/tests/erc20_connector.rs:623-665` which shows `balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT)` (i.e., tokens are lost) when `error_refund` is not enabled. [5](#0-4)
