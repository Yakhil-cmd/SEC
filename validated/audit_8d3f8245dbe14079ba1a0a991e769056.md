### Title
Stale ICP/XDR Conversion Rate Used for Cycles Minting Without Staleness Validation - (File: rs/nns/cmc/src/main.rs)

### Summary
The Cycles Minting Canister (CMC) converts ICP to cycles using `state.icp_xdr_conversion_rate` without ever checking the age of that rate's `timestamp_seconds`. If the Exchange Rate Canister (XRC) is persistently unavailable, the stored rate becomes arbitrarily stale. Any unprivileged user can then call `notify_top_up` or `notify_mint_cycles` and receive cycles computed from the stale (potentially inflated) rate, over-minting cycles relative to ICP's current market value.

### Finding Description

The function `tokens_to_cycles` in `rs/nns/cmc/src/main.rs` is the sole conversion path for all three public minting endpoints (`notify_top_up`, `notify_create_canister`, `notify_mint_cycles`):

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);   // ← only reads the rate value
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles {
                xdr_permyriad_per_icp,
                cycles_per_xdr: state.cycles_per_xdr,
            }
            .to_cycles(amount)),
            None => { /* error */ }
        }
    })
}
```

The only guard is a `None` check — there is **no check on `timestamp_seconds`**. The stored `IcpXdrConversionRate` carries a `timestamp_seconds` field that records when the rate was sampled, but `tokens_to_cycles` ignores it entirely. [1](#0-0) 

The CMC refreshes the rate every 5 minutes via heartbeat by calling the XRC. On XRC failure it retries every minute: [2](#0-1) [3](#0-2) 

There is no upper bound on how stale the rate can become before `tokens_to_cycles` refuses to use it. The `IcpXdrConversionRate` struct stores `timestamp_seconds` but it is never consulted at conversion time: [4](#0-3) 

Additionally, a second staleness path exists via the `DivergedRate` mechanism: when `set_update_exchange_rate_state` receives `DivergedRate`, it sets `UpdateExchangeRateState::Disabled`, permanently halting automatic XRC polling until an `EnableAutomaticExchangeRateUpdates` proposal arrives. During this window the CMC continues to mint cycles from the frozen rate with no expiry: [5](#0-4) 

The conversion arithmetic itself confirms that a higher `xdr_permyriad_per_icp` directly yields more cycles per ICP token: [6](#0-5) 

### Impact Explanation

Cycles are intended to be pegged to XDR value (1 XDR = 1T cycles). If the stored rate reflects 10 XDR/ICP but ICP has since fallen to 7 XDR/ICP, every `notify_top_up` call mints ~43% more cycles than the current ICP value justifies. This is a **cycles/resource accounting bug**: the protocol over-mints cycles, effectively subsidising compute resources from protocol reserves and diluting the value of cycles for all existing holders. The effect compounds with volume — any user who notices the stale rate can drain the discrepancy at scale by repeatedly topping up canisters.

The three affected public endpoints are:
- `notify_top_up` — tops up a canister with cycles
- `notify_create_canister` — creates a canister funded by cycles
- `notify_mint_cycles` — mints cycles to the cycles ledger [7](#0-6) 

### Likelihood Explanation

The XRC is a separate IC canister. Persistent XRC errors (e.g., `StablecoinRateTooFewRates`, network partition to the XRC subnet) are explicitly handled in the CMC's integration tests, confirming the scenario is considered realistic: [8](#0-7) 

During any such outage, the rate silently ages. A sophisticated observer watching the CMC's `cmc_icp_xdr_conversion_rate_timestamp_seconds` metric can detect the divergence and time `notify_top_up` calls to maximise the discount. No privileged access is required.

### Recommendation

Add a maximum-age guard inside `tokens_to_cycles` (or in a shared helper called by all three notify paths). If `now - icp_xdr_conversion_rate.timestamp_seconds` exceeds a defined threshold (e.g., 1 hour), reject the conversion with a retriable error rather than proceeding with a stale rate. This mirrors the pattern already used in `should_refresh_xdr_rate` in the Governance canister, which refuses to use a rate older than one day: [9](#0-8) 

### Proof of Concept

1. The XRC subnet experiences a prolonged outage; the CMC heartbeat logs `FailedToRetrieveRate` and schedules retries every minute.
2. ICP market price falls 25% (e.g., from 10 XDR/ICP to 7.5 XDR/ICP) during the outage.
3. `state.icp_xdr_conversion_rate` still holds the pre-outage value (`xdr_permyriad_per_icp = 100_000`, i.e., 10 XDR/ICP).
4. An unprivileged user transfers 100 ICP to the CMC subaccount for their canister and calls `notify_top_up`.
5. `tokens_to_cycles` computes: `100 ICP × 100_000 permyriad × 1T cycles/XDR / 10_000 = 100T cycles`.
6. At the current market rate the correct amount would be `75T cycles`.
7. The user receives **33% more cycles** than the current ICP value justifies, with no privileged access required.
8. The user can repeat this for any number of ICP blocks until the XRC recovers and the rate is refreshed.
