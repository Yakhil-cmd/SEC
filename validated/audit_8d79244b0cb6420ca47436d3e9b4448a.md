### Title
NEAR Permanently Frozen When wNEAR Unwrap Transfer Fails in `exit_to_near_precompile_callback` - (File: engine/src/contract_methods/connector.rs)

---

### Summary

When a user unwraps wNEAR ERC-20 tokens to NEAR via the `exit_to_near` precompile using the `UnwrapWnear` message, the ERC-20 tokens are burned and `near_withdraw` is called on the wNEAR NEP-141 contract. If `near_withdraw` succeeds but the subsequent NEAR transfer to the target account fails (e.g., target account does not exist), the NEAR is permanently stuck in the engine account with no recovery path, while the user's wNEAR ERC-20 tokens are already burned.

---

### Finding Description

The `exit_to_near` precompile in `engine-precompiles/src/native.rs` handles the `UnwrapWnear` message by setting `transfer_near_args` to carry the target account and amount: [1](#0-0) 

This produces a two-step async promise chain:
1. Call `near_withdraw` on the wNEAR NEP-141 contract (burns wNEAR NEP-141 tokens, sends NEAR to the engine account).
2. Callback `exit_to_near_precompile_callback` â€” if step 1 succeeded, create a NEAR `Transfer` promise to the user-specified target account.

The callback is implemented in `engine/src/contract_methods/connector.rs`: [2](#0-1) 

The critical flaw: the NEAR `Transfer` promise is created and returned via `handler.promise_return(promise_id)`, but **no callback is attached to it**. If the transfer fails (e.g., target account does not exist on NEAR), the NEAR is returned to the engine account by the NEAR runtime, but:

- The user's wNEAR ERC-20 tokens were already burned during EVM execution (before any NEAR promises ran).
- The `else if let Some(args) = args.refund` refund branch is only triggered when `near_withdraw` itself fails â€” it is never reached when `near_withdraw` succeeds but the subsequent transfer fails.
- There is no engine-level mechanism to track or redistribute NEAR that accumulates in the engine account from failed transfers. [3](#0-2) 

---

### Impact Explanation

**Critical â€” Permanent freezing of user funds.**

The user's wNEAR ERC-20 tokens are irreversibly burned. The equivalent NEAR lands in the engine account's general balance with no accounting entry linking it to the user. There is no admin method, no user-callable method, and no callback path that can recover or redistribute this NEAR to the affected user.

---

### Likelihood Explanation

**Medium.**

The `UnwrapWnear` path is a production-facing feature for bridging wNEAR back to native NEAR. The failure condition is triggered whenever the user-supplied target NEAR account ID does not exist (e.g., typo, account not yet created, account deleted). This is a realistic user-error scenario that the protocol must handle gracefully but does not.

---

### Recommendation

Attach a failure-handling callback to the NEAR `Transfer` promise created inside `exit_to_near_precompile_callback`. If the transfer fails, the callback should re-mint the burned wNEAR ERC-20 tokens to the original sender (mirroring the existing `refund_on_error` logic used for the `near_withdraw` failure case).

---

### Proof of Concept

1. User holds wNEAR ERC-20 tokens on Aurora and calls `exit_to_near` with `Message::UnwrapWnear` targeting a non-existent NEAR account (e.g., `"typo.near"`).
2. EVM execution burns the user's wNEAR ERC-20 balance and schedules the `near_withdraw` + callback promise chain.
3. `near_withdraw` on the wNEAR NEP-141 contract succeeds; NEAR is credited to the engine account.
4. `exit_to_near_precompile_callback` fires, sees `PromiseResult::Successful`, and creates a `Transfer` promise to `"typo.near"`.
5. The `Transfer` promise fails because `"typo.near"` does not exist; NEAR is returned to the engine account by the NEAR runtime.
6. No further callback exists; the refund branch is never reached.
7. **Result**: user's wNEAR ERC-20 tokens are permanently burned; the equivalent NEAR is absorbed into the engine's undifferentiated NEAR balance with no recovery path. [4](#0-3) [5](#0-4)

#[named]
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
