Audit Report

## Title
Attacker-Induced Single-Block Log Overflow Causes Permanent Deposit Loss in ckETH/ckERC20 Minter — (`rs/ethereum/cketh/minter/src/deposit.rs`)

## Summary

In `scrape_block_range`, when a single Ethereum block's `get_logs` response exceeds the 2 MB HTTP-outcall ceiling after all `DoubleCycles` retries, the minter permanently records the block in `skipped_blocks` and advances `last_scraped_block_number` past it. There is no recovery path: the `assert!` in `record_skipped_block_for_contract` would panic the canister on any re-skip attempt, and no governance or admin endpoint exists to reset the scraping cursor. Every ETH and ERC-20 deposit event in the targeted block is permanently unprocessed.

## Finding Description

`scrape_block_range` issues `get_logs` with an initial response-size estimate of `ETH_GET_LOGS_INITIAL_RESPONSE_SIZE_ESTIMATE + HEADER_SIZE_LIMIT = 100 + 2048 = 2148 bytes`. [1](#0-0) 

The `DoubleCycles` retry strategy doubles the estimate on each retry up to the 2 MB HTTP-outcall ceiling. When the response for a single block still exceeds 2 MB after all retries, the code reaches the terminal branch at `deposit.rs` L273–283: [2](#0-1) 

`process_event` calls `record_skipped_block_for_contract`, which inserts the block into `skipped_blocks` behind an `assert!` that panics on any re-insertion attempt, making the skip permanent: [3](#0-2) 

`update_last_scraped_block_number` then advances the scraping cursor past the block, so subsequent scraping cycles never revisit it. The only active scraping path is `LogScrapingId::EthOrErc20DepositWithSubaccount`; the other two IDs (`EthDepositWithoutSubaccount`, `Erc20DepositWithoutSubaccount`) are `Deprecated`: [4](#0-3) 

This single active scraping path covers both ETH and ERC-20 deposits through one helper contract address, so flooding either token type in one block triggers the skip for all deposits in that block.

## Impact Explanation

Every deposit event emitted by the helper contract in the skipped block is permanently ignored. Depositors' ETH or ERC-20 tokens are locked in the Ethereum helper contract with no corresponding ckETH/ckERC20 minted on the IC. There is no admin endpoint, governance proposal path, or state-machine mechanism to force re-scraping of a block already recorded in `skipped_blocks`. This constitutes a **permanent, irrecoverable loss of user funds** for all depositors whose transactions landed in the targeted block — a concrete High-severity Chain Fusion / ck-token security impact with direct user harm.

## Likelihood Explanation

Each log entry is approximately 600 bytes; exceeding 2 MB requires roughly 3,334 matching log entries in a single Ethereum block. [5](#0-4) 

An Ethereum block's gas limit (~30 M gas) bounds independent transactions, but a single transaction can emit many events via a batch-deposit contract or a looping contract that calls the helper repeatedly. An attacker willing to spend the gas can concentrate thousands of events in one block. The attacker's own deposits are also lost, making this a griefing/vandalism attack rather than a profitable one — this lowers likelihood. However, the permanent and irrecoverable nature of the impact, the absence of any on-chain or off-chain recovery path, and the high value of assets bridged through ckETH/ckERC20 make this a meaningful risk.

## Recommendation

1. **Remove the permanent-skip branch.** Instead of advancing `last_scraped_block_number` past an oversized single block, return an error and let the timer retry. Introduce a per-block retry counter; only skip after N consecutive failures, and emit an alert.
2. **Add a governance-controlled re-scrape endpoint** that allows the NNS or minter controller to reset `last_scraped_block_number` to a prior value so a previously skipped block can be re-attempted after the root cause is resolved.
3. **Remove or soften the `assert!` in `record_skipped_block_for_contract`** (replace with a no-op or log) so that a re-scrape attempt does not panic the canister. [6](#0-5) 
4. **Raise the 2 MB ceiling or make it configurable** via upgrade argument, so that a temporary spike in log volume does not immediately trigger the skip path.

## Proof of Concept

The existing test `should_skip_single_block_containing_too_many_events` already demonstrates the exact mechanism: 3,500 log entries (~2.1 MB) in a single block cause the minter to emit `SkippedBlock` and advance past the block, with no further attempt to process those deposits. [7](#0-6) 

An attacker replicates this on mainnet by submitting transactions within one Ethereum block that call the `ReceivedEthOrErc20WithSubaccount` helper contract enough times to produce ≥ 3,334 log entries. The next scraping cycle of the ckETH minter will:

1. Attempt `get_logs` for the block range containing the flooded block.
2. Receive a `LimitExceeded` HTTP-outcall error at every doubling of the response-size estimate up to 2 MB.
3. Bisect down to the single flooded block (`from_block == to_block`).
4. Emit `SkippedBlockForContract` and advance `last_scraped_block_number` past it — permanently discarding all deposits in that block. [8](#0-7)

#[test]
fn should_skip_single_block_containing_too_many_events() {
    let cketh = CkEthSetup::default();
    let deposit = DepositParams::default().to_log_entry();
    // around 600 bytes per log
    // we need at least 3334 logs to reach the 2MB limit
    let large_amount_of_logs = multi_logs_for_single_transaction(deposit.clone(), 3_500);
    assert!(serde_json::to_vec(&large_amount_of_logs).unwrap().len() > 2_000_000);

    cketh.env.advance_time(SCRAPING_ETH_LOGS_INTERVAL);
    MockJsonRpcProviders::when(JsonRpcMethod::EthGetBlockByNumber)
        .respond_for_all_with(block_response(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 3))
        .build()
        .expect_rpc_calls(&cketh);

    for max_response_bytes in cketh.all_eth_get_logs_response_size_estimates() {
        MockJsonRpcProviders::when(JsonRpcMethod::EthGetLogs)
            .with_request_params(json!([{
                "fromBlock": BlockNumber::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 1),
                "toBlock": BlockNumber::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 3),
                "address": [ETH_HELPER_CONTRACT_ADDRESS],
                "topics": [cketh.received_eth_event_topic()]
            }]))
            .with_max_response_bytes(max_response_bytes)
            .respond_for_all_with(large_amount_of_logs.clone())
            .build()
            .expect_rpc_calls(&cketh);
    }

    for max_response_bytes in cketh.all_eth_get_logs_response_size_estimates() {
        MockJsonRpcProviders::when(JsonRpcMethod::EthGetLogs)
            .with_request_params(json!([{
                "fromBlock": BlockNumber::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 1),
                "toBlock": BlockNumber::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 2),
                "address": [ETH_HELPER_CONTRACT_ADDRESS],
                "topics": [cketh.received_eth_event_topic()]
            }]))
            .with_max_response_bytes(max_response_bytes)
            .respond_for_all_with(large_amount_of_logs.clone())
            .build()
            .expect_rpc_calls(&cketh);
    }

    for max_response_bytes in cketh.all_eth_get_logs_response_size_estimates() {
        MockJsonRpcProviders::when(JsonRpcMethod::EthGetLogs)
            .with_request_params(json!([{
                "fromBlock": BlockNumber::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 1),
                "toBlock": BlockNumber::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 1),
                "address": [ETH_HELPER_CONTRACT_ADDRESS],
                "topics": [cketh.received_eth_event_topic()]
            }]))
            .with_max_response_bytes(max_response_bytes)
            .respond_for_all_with(large_amount_of_logs.clone())
            .build()
            .expect_rpc_calls(&cketh);
    }

    MockJsonRpcProviders::when(JsonRpcMethod::EthGetLogs)
        .with_request_params(json!([{
            "fromBlock": BlockNumber::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 2),
            "toBlock": BlockNumber::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 3),
            "address": [ETH_HELPER_CONTRACT_ADDRESS],
            "topics": [cketh.received_eth_event_topic()]
        }]))
        .with_max_response_bytes(cketh.all_eth_get_logs_response_size_estimates()[0])
        .respond_for_all_with(empty_logs())
        .build()
        .expect_rpc_calls(&cketh);

        .check_audit_logs_and_upgrade(Default::default())
