Audit Report

## Title
Third-Party Caller Can Bypass Confirmation-Text Consent Gate and Force-Register Participants in SNS Swap — (`rs/sns/swap/canister/canister.rs`)

## Summary
The `refresh_buyer_tokens` update method in the SNS Swap canister accepts a caller-supplied `buyer` principal without verifying it matches `ic_cdk::caller()`. Because the swap's `confirmation_text` is a publicly readable initialization parameter and any ingress sender can supply an arbitrary `buyer`, a third party can register any principal who has already deposited ICP into the swap subaccount as a consenting participant — without that principal ever calling the method. If enough such principals are force-registered, the swap's accepted ICP total can be pushed past `max_direct_participation_icp_e8s`, triggering an early commit and locking all participants' ICP until finalization.

## Finding Description
In `rs/sns/swap/canister/canister.rs` (L130–134), the effective buyer is resolved as:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()   // attacker-controlled
};
```

`p` is passed directly to `refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs`, L1134). Inside that function, `validate_confirmation_text` (L363–384) performs only a string-equality check against the swap's publicly visible init parameter — it does not verify that the caller is `p`. The code comment at L1123–1126 states "the caller of this function must accept this confirmation," but the implementation does not enforce this invariant.

After confirmation validation, the function queries the ICP ledger for the balance of `p`'s subaccount (L1153–1163), then inserts or updates `p`'s `BuyerState` in `self.buyers` (L1285–1288) and calls `self.update_total_participation_amounts()` (L1291). The swap's `can_commit` logic (L2884–2897) commits the swap when `icp_target_progress().is_reached_or_exceeded()` AND `sufficient_participation()` — both of which can be satisfied by force-registering pending participants.

The `RefreshBuyerTokensRequest` proto documents the open-caller design ("If not specified, the caller is used"), but this design is incompatible with the confirmation text serving as a per-participant consent gate.

## Impact Explanation
This is a **High** severity finding matching the impact class: *"Significant SNS security impact with concrete user or protocol harm."*

Two concrete harms:

1. **Confirmation-text consent bypass**: The confirmation text is the only mechanism by which SNS swap creators can require explicit per-participant acknowledgement before ICP is accepted. Any unprivileged attacker can trivially circumvent it for any principal who has deposited ICP, rendering the consent gate meaningless.

2. **Forced early swap commitment with fund lockup**: By force-registering multiple pending participants, an attacker can push the swap past `max_direct_participation_icp_e8s`. On the next periodic task, `try_commit` fires (L1040), transitioning the swap to `Committed`. Participants cannot reclaim their ICP via `error_refund_icp` until finalization completes — a potentially extended period during which their funds are locked without their consent.

## Likelihood Explanation
The attack requires no privileges. Any ingress sender can submit an update call to `refresh_buyer_tokens` with an arbitrary `buyer` string. The confirmation text is set at SNS initialization and is readable from the canister's public state. The only prerequisite — that the target principal has already transferred ICP to the swap subaccount — is publicly observable on the ICP ledger. The attack is repeatable across all pending participants in a single block window.

## Recommendation
Enforce that the effective buyer equals the authenticated caller when a non-empty `buyer` field is supplied:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let requested = PrincipalId::from_str(&arg.buyer).unwrap();
    if requested != caller_principal_id() {
        panic!("buyer field must match caller");
    }
    requested
};
```

If third-party notification is a required use-case (e.g., a relayer), the confirmation text check must be separated from the balance-refresh step so that only the principal themselves can supply the confirmation, or the consent must be expressed via a caller-signed attestation rather than a publicly known string.

## Proof of Concept
1. Alice transfers 10 ICP to the swap canister's subaccount derived from her principal, intending to review the confirmation text before committing.
2. Attacker reads the swap's `confirmation_text` from the canister's public init parameters (e.g., via `get_state`).
3. Attacker submits:
   ```
   dfx canister call <swap_id> refresh_buyer_tokens \
     '(record { buyer = "<alice_principal>"; confirmation_text = opt "<public_text>" })'
   ```
4. The swap canister resolves `p = alice_principal` (L130–134), validates the attacker-supplied confirmation text against the public init value (L1150), queries the ICP ledger and finds Alice's 10 ICP (L1153–1163), and inserts Alice into `self.buyers` as a consenting participant (L1285–1288) — without Alice ever calling the method.
5. If this registration (or a batch of such registrations across multiple pending participants) tips `icp_target_progress()` to `Reached`, the next invocation of `run_periodic_tasks` calls `try_commit` (L1040), transitioning the swap to `Committed` and locking all participants' ICP until finalization.

A deterministic unit test can reproduce step 4 by calling `swap.refresh_buyer_token_e8s(alice_principal, Some(public_text), ...)` from a different caller context and asserting that Alice appears in `swap.buyers` with a non-zero balance.
