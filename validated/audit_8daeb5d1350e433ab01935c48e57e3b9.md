### Title
Unauthenticated `ft_on_transfer` Combined with Unrestricted `deploy_erc20_token` Allows Minting of Unbacked ERC-20 Tokens — (`engine/src/contract_methods/connector.rs`)

---

### Summary

The `ft_on_transfer` NEAR entrypoint in Aurora Engine carries no caller-identity restriction. Any NEAR account — including a purpose-built attacker contract — can invoke it directly. Because `deploy_erc20_token` is equally unrestricted, an attacker can first register a fake NEP-141 contract with Aurora, then call `ft_on_transfer` from that contract to mint an arbitrary quantity of the corresponding ERC-20 token on Aurora without ever depositing real NEP-141 tokens. The minted tokens are fully unbacked, enabling direct theft of real assets held by other users in Aurora DeFi protocols.

---

### Finding Description

**Root cause — `ft_on_transfer` has no caller guard**

`ft_on_transfer` is exposed as a public WASM export with no `assert_private_call()`, no whitelist check, and no verification that the call arrived as a genuine NEP-141 `ft_transfer_call` callback:

```rust
// engine/src/contract_methods/connector.rs  lines 62-109
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let predecessor_account_id = env.predecessor_account_id();
        ...
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)          // mints ETH
        } else {
            engine.receive_erc20_tokens(               // mints ERC-20
                &predecessor_account_id, &args, ...)
        };
        ...
    })
}
```

The only branching logic is whether the predecessor equals the ETH-connector account. For every other predecessor the engine calls `receive_erc20_tokens`, which looks up the ERC-20 contract registered for that predecessor as a NEP-141 and mints tokens to the address encoded in `args.msg`. No check is made that real NEP-141 tokens were actually transferred.

**Enabling condition — `deploy_erc20_token` is equally unrestricted**

```rust
// engine/src/contract_methods/connector.rs  lines 112-159
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        // ← no require_owner_only, no whitelist, no assert_private_call
        let args = DeployErc20TokenArgs::deserialize(&bytes)...;
        ...
    })
}
```

Any NEAR account can register any NEP-141 account ID with Aurora and obtain a live ERC-20 contract for it.

**Complete attacker-controlled entry path**

1. Attacker deploys `fake_nep141.near` — a NEAR contract whose sole purpose is to cross-contract-call `ft_on_transfer` on Aurora with attacker-chosen `sender_id`, `amount`, and `msg` (attacker's EVM address).
2. Attacker calls `deploy_erc20_token` on Aurora, passing `fake_nep141.near`. Aurora deploys a real ERC-20 contract and records the `fake_nep141.near → ERC-20 address` mapping.
3. Attacker triggers `fake_nep141.near` to call `ft_on_transfer` on Aurora. The NEAR predecessor is `fake_nep141.near`; `args.amount` is set to any desired value (e.g., `u128::MAX`).
4. Aurora's `receive_erc20_tokens` finds the registered ERC-20, mints `amount` tokens to the attacker's EVM address, and returns `0` (no refund). No real NEP-141 tokens were ever transferred.
5. Attacker holds unbacked ERC-20 tokens indistinguishable from legitimately bridged ones and swaps them for real assets in any Aurora DeFi pool.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

The minted ERC-20 tokens are accepted by every Aurora DeFi protocol as genuine bridged assets. The attacker can drain liquidity pools, borrow against the fake collateral, or exit to NEAR via the standard `exit_to_near` precompile, receiving real NEP-141 tokens that belong to other depositors. The total supply of the ERC-20 on Aurora becomes permanently unbacked, causing insolvency for all legitimate holders.

---

### Likelihood Explanation

**High.** The attack requires only:
- Deploying one NEAR contract (permissionless, costs a few NEAR for storage).
- Two cross-contract calls (`deploy_erc20_token`, then `ft_on_transfer` via the fake NEP-141).
- No privileged access, no leaked keys, no governance capture.

The attack is fully self-contained and can be executed by any NEAR account holder.

---

### Recommendation

1. **Restrict `deploy_erc20_token` to the contract owner** (add `require_owner_only`) so that only vetted NEP-141 tokens can be registered. This is the minimal fix that closes the enabling condition.
2. **Alternatively, add a caller whitelist to `ft_on_transfer`**: verify that `predecessor_account_id` is in an admin-maintained set of approved NEP-141 contracts before calling `receive_erc20_tokens`.
3. **Defense-in-depth**: track a per-NEP-141 "expected balance" (tokens actually received via `ft_transfer_call`) and assert it matches the minted supply inside `receive_erc20_tokens`.

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })
}
