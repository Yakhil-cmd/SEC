### Title
Dequeued EIP-7002 and EIP-7251 Queue Storage Slots Not Zeroed After Processing — (`basic_bootloader/src/bootloader/block_flow/ethereum/eip_7002_withdrawal_contract/mod.rs`, `basic_bootloader/src/bootloader/block_flow/ethereum/eip_7251_consolidation_contract/mod.rs`)

---

### Summary

The native ZKsync OS implementations of the EIP-7002 (withdrawal requests) and EIP-7251 (consolidation requests) system contracts read and process queue entries at block finalization but never zero out the storage slots that held the dequeued data. The reference EVM bytecode for both predeploys does zero those slots. This produces a persistent state-root divergence: the storage of both predeploy contracts will contain stale non-zero validator addresses and BLS public keys in slots that the reference implementation would have cleared.

---

### Finding Description

`eip7002_system_part` in `eip_7002_withdrawal_contract/mod.rs` implements the EIP-7002 system-level dequeue logic natively (bypassing the EVM bytecode). For each dequeued entry it reads three storage slots — `slot_0` (address), `slot_1` (pubkey part 0), `slot_2` (pubkey part 1 + amount) — and feeds them into the requests hasher. After the loop it only updates the queue head pointer (or resets head/tail to zero when the queue is fully drained). It never writes `Bytes32::ZERO` back to the three data slots it just consumed. [1](#0-0) 

The same pattern exists in `eip7251_system_part`: four slots per entry (address, source pubkey, target pubkey) are read and hashed but never zeroed. [2](#0-1) 

After the queue is fully drained, both head and tail are reset to zero: [3](#0-2) 

This means the next batch of requests will be written starting at `OFFSET + 0 * SLOTS_PER_REQUEST` again, overwriting the old data — but only when new requests arrive. Between the drain and the next enqueue, the old validator addresses and BLS public keys remain in persistent storage at slots `[4, 5, 6, 7, 8, 9, ...]` of `WITHDRAWAL_REQUEST_PREDEPLOY_ADDRESS`.

The reference EVM bytecode (commented out in the file) zeros each dequeued entry. The native implementation skips this step entirely. [4](#0-3) 

---

### Impact Explanation

The storage of `WITHDRAWAL_REQUEST_PREDEPLOY_ADDRESS` and `CONSOLIDATION_REQUEST_PREDEPLOY_ADDRESS` will contain non-zero values in slots that the reference EIP-7002/EIP-7251 bytecode would have zeroed. Because the Ethereum state root commits to all storage slots of all accounts, the state root produced by ZKsync OS will diverge from the state root that a correct Ethereum execution client would produce for the same block. This is a **state-transition correctness bug**: the committed post-state is wrong, which can invalidate ZK proofs or cause the sequencer and prover to disagree on the canonical state.

Additionally, any EVM contract that reads those storage slots via `SLOAD` after the queue is drained will observe stale validator addresses and BLS public keys instead of zero, which is semantically incorrect and observable on-chain.

---

### Likelihood Explanation

This is triggered every block in which at least one EIP-7002 withdrawal request or EIP-7251 consolidation request is submitted and processed. Any unprivileged user can submit such a request by calling the predeploy contract with the required ETH fee. The bug fires deterministically on every such block — likelihood is high once the Pectra fork is active.

---

### Recommendation

After the dequeue loop, zero out each consumed storage slot before updating the queue head pointer. For EIP-7002 (3 slots per entry):

```rust
for i in 0..num_dequeued {
    let base = WITHDRAWAL_REQUEST_QUEUE_STORAGE_OFFSET
        + ((queue_head_index + U256::from(i as u64)) * SLOTS_PER_REQUEST);
    for j in 0u64..3 {
        let slot = Bytes32::from_array((base + U256::from(j)).to_be_bytes::<32>());
        resources.with_infinite_ergs(|r| {
            system.io.storage_write::<false>(
                ExecutionEnvironmentType::NoEE, r,
                &WITHDRAWAL_REQUEST_PREDEPLOY_ADDRESS, &slot, &Bytes32::ZERO,
            )
        })?;
    }
}
```

Apply the same fix for EIP-7251 (4 slots per entry). This matches the behaviour of the reference EVM bytecode.

---

### Proof of Concept

1. Deploy ZKsync OS with Pectra fork active (EIP-7002 and EIP-7251 predeploys present).
2. Submit 3 withdrawal requests in block N (queue: head=0, tail=3; slots 4–12 filled with validator data).
3. Block N is finalized: `eip7002_system_part` dequeues all 3 entries, resets head=0, tail=0. Slots 4–12 still contain non-zero validator addresses and BLS public keys.
4. No new requests are submitted in block N+1.
5. Read `SLOAD(WITHDRAWAL_REQUEST_PREDEPLOY_ADDRESS, 4)` — returns the stale address from the first dequeued request instead of zero.
6. Compare the state root produced by ZKsync OS against a reference Ethereum client for the same block — they will differ because the reference client zeroes those slots. [5](#0-4) [6](#0-5)
