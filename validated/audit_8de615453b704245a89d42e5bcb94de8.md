Audit Report

## Title
Rate-Limit Bypass in `ensure_balance` via CMC Pre-Funding - (File: `rs/nns/cmc/src/main.rs`)

## Summary
`ensure_balance` computes `cycles_to_mint = cycles - current_balance` using saturating subtraction and charges only that delta to the `base_limiter`, not the full ICP-backed `cycles` amount. Because any canister can call the management canister's `deposit_cycles` to increase the CMC's balance before a minting call, an attacker can pre-fund the CMC to shrink `cycles_to_mint` to an arbitrarily small value (including zero), causing the limiter to record far less than the true ICP-backed value being converted. This allows sustained ICP-to-cycles conversion well above the protocol's `base_cycles_limit`.

## Finding Description
`ensure_balance` in `rs/nns/cmc/src/main.rs` (lines 2306–2325) reads the CMC's live balance at call time and computes the incremental mint:

```rust
let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
let cycles_to_mint = cycles - current_balance;   // saturating_sub
```

The limiter is then charged `cycles_to_mint`, not `cycles`:

```rust
limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;
state.total_cycles_minted += cycles_to_mint;
```

`Cycles::sub` is implemented as `saturating_sub` (`rs/types/cycles/src/cycles.rs` lines 133–138), so when `current_balance >= cycles`, `cycles_to_mint` saturates to zero — the limiter records nothing while the full ICP amount is still burned. `check_and_add_cycles` in `rs/nns/cmc/src/limiter.rs` (lines 34–56) enforces `count + cycles_to_mint <= limit`; with `cycles_to_mint = 0` this check trivially passes.

This path is reachable from all three public minting endpoints:
- `notify_top_up` → `process_top_up` → `deposit_cycles` → `ensure_balance` (line 2117)
- `notify_mint_cycles` → `process_mint_cycles` → `do_mint_cycles` → `ensure_balance` (line 2151)
- `notify_create_canister` → `process_create_canister` → `do_create_canister` → `ensure_balance` (line 2249)

**Note on the secondary wrapping claim**: The submitted report incorrectly states that underflow wraps to `u128::MAX` in release mode. `Cycles` uses `saturating_sub` throughout, so underflow saturates to zero. This does not weaken the primary finding; it makes the extreme pre-funding case (`current_balance >= cycles`) more severe, not less.

## Impact Explanation
The `base_cycles_limit` (150 × 10¹⁵ cycles/hour) is the protocol's primary rate control on ICP burning. By pre-funding the CMC with `X` cycles before submitting a `notify_top_up` for `Z` cycles (where `Z > base_cycles_limit`), the attacker causes the limiter to record only `Z − X` minted cycles. If `Z − X ≤ base_cycles_limit`, the call succeeds. In the extreme case where `X ≥ Z`, the limiter records zero while the full `Z` ICP-backed value is converted. Repeating this allows sustained above-limit ICP burning, distorting the ICP/cycles exchange rate and undermining the tokenomic controls the limit is designed to enforce. This constitutes illegal minting beyond protocol-sanctioned limits and falls under the High–Critical impact class for NNS/CMC financial integrity.

## Likelihood Explanation
No privileged role is required. Any canister on any subnet can call the management canister's `deposit_cycles` targeting the CMC principal. The attacker needs only cycles (obtainable by minting at the normal rate over prior hours) and ICP. The attack is fully deterministic and synchronous — `ensure_balance` reads the balance and mints in the same execution slice, so pre-funding before the ingress message is sufficient. No timing precision, governance vote, or threshold key is needed.

## Recommendation
Charge the limiter based on the full `cycles` parameter (the ICP-backed value being converted), not on the balance delta. Add an explicit guard for the already-funded case to avoid charging the limiter for zero:

```rust
fn ensure_balance(cycles: Cycles, limiter_to_use: CyclesMintingLimiterSelector) -> Result<(), String> {
    let now = now_system_time();
    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    if current_balance >= cycles {
        return Ok(());
    }
    let cycles_to_mint = cycles - current_balance;

    with_state_mut(|state| {
        // Charge the limiter for the full ICP-backed amount
        limiter_to_use.check_and_add_cycles(state, now, cycles)?;
        state.total_cycles_minted += cycles_to_mint;
        Ok::<_, String>(())
    })?;

    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())
}
```

## Proof of Concept
Assume `base_cycles_limit` = 150 × 10¹⁵ cycles/hour.

1. **Accumulate cycles**: Over prior hours, attacker mints up to the limit, accumulating 150 × 10¹⁵ cycles.
2. **Pre-fund CMC**: Attacker calls management canister `deposit_cycles` targeting the CMC, depositing 150 × 10¹⁵ cycles. CMC `current_balance` is now ≥ 150 × 10¹⁵.
3. **Submit large top-up**: Attacker transfers 3,000 ICP (≈ 300 × 10¹⁵ cycles at typical rates) to the CMC subaccount and calls `notify_top_up`.
4. **`ensure_balance` executes**: `cycles_to_mint = 300×10¹⁵ − 150×10¹⁵ = 150×10¹⁵` (saturating). Limiter check: `0 + 150×10¹⁵ ≤ 150×10¹⁵` → passes.
5. **Result**: CMC mints 150 × 10¹⁵ new cycles, combines with the 150 × 10¹⁵ pre-funded cycles, deposits 300 × 10¹⁵ cycles to the target. Limiter records only 150 × 10¹⁵ this hour — 2× the intended limit.
6. **Extreme case**: Pre-fund with ≥ 300 × 10¹⁵ cycles; `cycles_to_mint` saturates to 0; limiter records nothing; unlimited ICP conversion in one call.

A deterministic integration test using PocketIC can reproduce this by: (a) minting cycles to the attacker canister over simulated hours, (b) calling `deposit_cycles` on the management canister targeting the CMC, (c) calling `notify_top_up` with a large ICP transfer, and (d) asserting that `base_limiter.get_count()` is less than the actual ICP-backed cycles converted.     pub fn check_and_add_cycles(
```rust
        &mut self,
        now: SystemTime,
        cycles_to_mint: Cycles,
        limit: Cycles,
    ) -> Result<(), String> {
        self.purge_old(now);
        let count = self.get_count();

        if count + cycles_to_mint > limit {
            LIMITER_REJECT_COUNT.with(|count| {
                count.set(count.get().saturating_add(1));
            });

            return Err(format!(
                "More than {} cycles have been minted in the last {} seconds, please try again later.",
                limit,
                self.get_max_age().as_secs(),
            ));
        }
        self.add(now, cycles_to_mint);
        Ok(())
    }
```
