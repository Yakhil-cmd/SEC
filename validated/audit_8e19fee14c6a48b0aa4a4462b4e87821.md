### Title
Unchecked `transferFrom` Boolean Return Value in XCC Precompile Allows NEAR Payment Bypass - (File: engine-precompiles/src/xcc.rs)

### Summary
The Cross-Contract Call (XCC) precompile calls `transferFrom` on the wNEAR ERC-20 contract to collect the required NEAR payment from the caller, but only checks the EVM exit reason (revert/error/fatal) and never decodes or validates the ABI-encoded `bool` return value. If the wNEAR ERC-20 contract returns `false` without reverting — a valid behavior for non-reverting ERC-20 implementations — the precompile silently proceeds to schedule the cross-contract call without the payment having been collected.

### Finding Description
In `engine-precompiles/src/xcc.rs`, when a caller invokes the XCC precompile and `required_near != ZERO_YOCTO`, the precompile issues an internal EVM call to the wNEAR ERC-20 contract's `transferFrom` selector to move wNEAR from the caller to the engine's implicit address:

```rust
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
match exit_reason {
    // Transfer successful, nothing to do
    aurora_evm::ExitReason::Succeed(_) => (),
    aurora_evm::ExitReason::Revert(r) => { ... }
    aurora_evm::ExitReason::Error(e) => { ... }
    aurora_evm::ExitReason::Fatal(f) => { ... }
}
``` [1](#0-0) 

The variable `return_value` — which holds the raw ABI-encoded `bool` returned by `transferFrom` — is used only in the `Revert` branch (as the revert output bytes) and is **never decoded or checked** in the `Succeed` branch. The comment `// Transfer successful, nothing to do` assumes that `ExitReason::Succeed` implies a successful transfer, but ERC-20's `transferFrom` is specified to return `false` on failure without necessarily reverting. A non-reverting ERC-20 implementation would produce `ExitReason::Succeed` with `return_value = abi.encode(false)`, which this code treats identically to a successful transfer.

The `transferFrom` call is constructed with the ERC-20 selector `0x23b872dd`: [2](#0-1) 

and dispatched against the wNEAR address obtained from storage: [3](#0-2) 

After the unchecked call, the precompile unconditionally emits the `promise_log` that schedules the cross-contract call: [4](#0-3) 

### Impact Explanation
If the wNEAR ERC-20 contract returns `false` without reverting (e.g., due to insufficient allowance or balance), the XCC precompile proceeds to schedule the NEAR cross-contract call without having collected the required wNEAR payment. The caller's wNEAR balance is not reduced, and the engine's implicit address does not receive the tokens. The attacker effectively obtains free cross-contract call scheduling, bypassing the economic accounting that prevents unbounded NEAR consumption. This constitutes a **High — theft of unclaimed yield / temporary freezing of funds** impact because the engine's NEAR balance is consumed for calls that were not paid for, and legitimate users' cross-contract calls may be starved of gas.

### Likelihood Explanation
The wNEAR ERC-20 contract address is set by the admin via `factory_set_wnear_address`. The deployed wNEAR contract on Aurora mainnet is expected to be a standard reverting ERC-20. However, the code makes no defensive assumption about this: it does not decode the return bytes, does not assert `return_value == abi.encode(true)`, and does not revert on a `false` result. Any future wNEAR contract upgrade or alternative deployment that follows the non-reverting ERC-20 pattern (as used by USDT and other major tokens) would immediately expose this path to exploitation by any EVM user who calls the XCC precompile with insufficient wNEAR allowance.

### Recommendation
After the `ExitReason::Succeed` arm, decode the `return_value` as an ABI-encoded `bool` and revert if it is `false`:

```rust
aurora_evm::ExitReason::Succeed(_) => {
    // Decode the bool return value of transferFrom
    let success = return_value
        .get(31)
        .copied()
        .unwrap_or(0) == 1;
    if !success {
        return Err(PrecompileFailure::Revert {
            exit_status: aurora_evm::ExitRevert::Reverted,
            output: b"ERR_WNEAR_TRANSFER_FAILED".to_vec(),
        });
    }
}
```

This mirrors the fix applied in the Compound analog: revert when the return value indicates failure rather than silently continuing.

### Proof of Concept
1. Deploy a wNEAR ERC-20 contract whose `transferFrom` returns `false` instead of reverting when the caller has insufficient allowance (valid ERC-20 behavior per EIP-20).
2. Admin sets this contract as the wNEAR address via `factory_set_wnear_address`.
3. Attacker calls the XCC precompile from an EVM contract with `CrossContractCallArgs::Eager(...)` targeting any NEAR contract, with `required_near > 0` but zero wNEAR allowance granted to the XCC precompile address.
4. The internal `handle.call(transferFrom)` returns `ExitReason::Succeed` with `return_value = abi.encode(false)`.
5. The precompile matches `Succeed(_) => ()` and emits the `promise_log` scheduling the cross-contract call.
6. The attacker's wNEAR balance is unchanged; the engine's NEAR is consumed for the call. [5](#0-4)
