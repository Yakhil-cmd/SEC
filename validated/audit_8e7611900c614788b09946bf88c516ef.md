Audit Report

## Title
Unbounded `ingress_expiries` Loop in `construction_payloads` Enables Single-Request OOM DoS — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary
The `construction_payloads` function in the ICRC-1 Rosetta node accepts attacker-controlled `ingress_start` and `ingress_end` values from HTTP request metadata and drives a `while` loop with no upper bound on the number of iterations. An unprivileged attacker can set `ingress_end = u64::MAX`, which passes both existing guards, causing the loop to execute ~139 million iterations and exhaust process heap memory, crashing the Rosetta node.

## Finding Description
In `construction_payloads` (`rs/rosetta-api/icrc1/src/construction_api/services.rs`), `ingress_start` and `ingress_end` are read directly from attacker-supplied request metadata with no range cap: [1](#0-0) 

Two guards precede the loop:

1. `ingress_start >= ingress_end` → error [2](#0-1) 
2. `ingress_end < now + ingress_interval` → error [3](#0-2) 

Neither guard bounds `ingress_end` from above. The unbounded loop follows immediately: [4](#0-3) 

The step size per iteration is `ingress_interval - INGRESS_INTERVAL_OVERLAP`. With `MAX_INGRESS_TTL = 300s`, `PERMITTED_DRIFT = 60s`, and `INGRESS_INTERVAL_OVERLAP = 120s` (confirmed in `constants.rs`): [5](#0-4) 

- `ingress_interval = 240,000,000,000 ns`
- step = `240,000,000,000 - 120,000,000,000 = 120,000,000,000 ns`
- iterations with `ingress_end = u64::MAX`: `(1.8e19 - ~1.7e18) / 1.2e11 ≈ 139,000,000`

The resulting `ingress_expiries` Vec is passed to `handle_construction_payloads`, which allocates one `EnvelopeContent::Call` struct and one `SigningPayload` per entry: [6](#0-5) 

The entire `envelope_contents` Vec is then CBOR-serialized and hex-encoded into the `unsigned_transaction` string, compounding memory usage further. [7](#0-6) 

## Impact Explanation
The Rosetta node process exhausts heap memory and is OOM-killed or panics. This takes down the entire ICRC-1 Rosetta node, making transaction construction, signing, and submission unavailable. This matches the allowed High impact: **"Application/platform-level DoS, crash... or subnet availability impact not based on raw volumetric DDoS"** and **"Significant... Rosetta... security impact with concrete user or protocol harm."**

## Likelihood Explanation
The ICRC-1 Rosetta construction API is a public HTTP endpoint requiring no authentication, no keys, and no prior state. A single ~100-byte JSON request suffices. The crafted `ingress_end = u64::MAX` trivially passes both guards. The attack is deterministic, repeatable, and requires no special privileges or volumetric traffic.

## Recommendation
Add a hard cap on the number of ingress expiries before entering the loop, or validate the window size upfront:

```rust
// Option 1: cap on window size (e.g., 24 hours)
const MAX_INGRESS_WINDOW_NS: u64 = 86_400_000_000_000; // 24h in ns
if ingress_end.saturating_sub(ingress_start) > MAX_INGRESS_WINDOW_NS {
    return Err(Error::processing_construction_failed(
        &"ingress_end exceeds maximum allowed window of 24 hours"
    ));
}

// Option 2: cap on loop iterations
const MAX_INGRESS_EXPIRIES: usize = 360; // ~24h / 4min
if ingress_expiries.len() >= MAX_INGRESS_EXPIRIES {
    return Err(Error::processing_construction_failed(
        &"ingress_end exceeds maximum allowed window"
    ));
}
```

Either check should be placed before or inside the loop at `services.rs` line 163.

## Proof of Concept
```bash
curl -X POST http://<rosetta-node>:8081/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
    "operations": [<valid_transfer_operation>],
    "public_keys": [<valid_public_key>],
    "metadata": {
      "ingress_start": 1700000000000000000,
      "ingress_end": 18446744073709551615
    }
  }'
```

`ingress_end = 18446744073709551615` (`u64::MAX`) satisfies both guards (`> ingress_start` and `> now + ingress_interval`), triggering ~139 million loop iterations. A deterministic unit test can confirm the OOM by calling `construction_payloads` directly with these values and observing unbounded allocation growth before the function returns.
