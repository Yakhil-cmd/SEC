Audit Report

## Title
TOCTOU Blocklist Bypass: Pending Withdrawal Requests Not Re-Checked After Canister Upgrade — (`rs/ethereum/cketh/minter/src/withdraw.rs`, `rs/ethereum/cketh/minter/src/state/audit.rs`)

## Summary

The ckETH minter enforces its sanctions blocklist exclusively at withdrawal submission time via `validate_address_as_destination`. Once an `AcceptedEthWithdrawalRequest` event is persisted, the destination address is never re-validated. Because the blocklist is a compile-time constant that changes only via canister upgrade, and because `post_upgrade` fully restores pending withdrawal requests through `replay_events()` without any blocklist re-check, a withdrawal submitted before an upgrade will be executed after the upgrade even if the destination address is now on the expanded blocklist. This violates the stated invariant and constitutes a sanctions-compliance failure with direct, irreversible on-chain impact.

## Finding Description

**Blocklist is compile-time only.**
`ETH_ADDRESS_BLOCKLIST` is a `const &[Address]` baked into the binary. [1](#0-0) 
`is_blocked` performs a binary search over this static array. [2](#0-1) 

**Blocklist check is ingress-only.**
`validate_address_as_destination` calls `is_blocked` and is the sole enforcement point. [3](#0-2) 
It is called once at the start of `withdraw_eth`: [4](#0-3) 
And once at the start of `withdraw_erc20`: [5](#0-4) 
After the check passes, the burn executes and the event is recorded with no further blocklist reference: [6](#0-5) 

**No blocklist check in the async processing pipeline.**
A `grep` for `is_blocked` across the entire `rs/ethereum/cketh/minter/` tree returns hits only in `blocklist.rs`, `address.rs`, `deposit.rs`, and `main.rs` — never in `withdraw.rs`. `create_transactions_batch` dequeues pending requests and creates transactions with no blocklist re-check: [7](#0-6) 
`sign_transactions_batch` and `send_transactions_batch` similarly contain no `is_blocked` call: [8](#0-7) 
All three are called from `process_retrieve_eth_requests`, which runs on a timer: [9](#0-8) 

**Pending withdrawal requests survive canister upgrades without re-validation.**
`post_upgrade` calls `replay_events()`: [10](#0-9) 
`replay_events_internal` iterates every stored event and calls `apply_state_transition`: [11](#0-10) 
The `AcceptedEthWithdrawalRequest` arm of `apply_state_transition` calls `record_withdrawal_request` with no blocklist check: [12](#0-11) 
Timers are re-registered immediately after replay: [13](#0-12) 

**Complete exploit flow:**
1. Attacker holds ckETH and calls `withdraw_eth` with destination address X (not yet on the blocklist). The burn succeeds; `AcceptedEthWithdrawalRequest` is appended to the event log.
2. An NNS governance proposal to add X to the blocklist passes (all proposals are public; the voting window is typically days, giving ample time to front-run).
3. The minter is upgraded with the new WASM containing the expanded `ETH_ADDRESS_BLOCKLIST`.
4. `post_upgrade` → `replay_events()` → `apply_state_transition(AcceptedEthWithdrawalRequest)` → `record_withdrawal_request` restores the pending request for X with no blocklist re-check.
5. `setup_timers()` re-registers the processing timer.
6. Timer fires → `process_retrieve_eth_requests` → `create_transactions_batch` → `sign_transactions_batch` → `send_transactions_batch` → ETH is sent on-chain to the now-sanctioned address X.

## Impact Explanation

The stated invariant — *"ETH is not accepted from nor sent to addresses on this list"* — is directly violated. The ckETH minter is a production Chain Fusion canister holding real ETH; sending funds to a newly-sanctioned address is an irreversible on-chain action constituting a concrete sanctions-compliance failure. This matches the allowed High impact: **Significant Chain Fusion / ck-token security impact with concrete protocol harm** ($2,000–$10,000). The transaction cannot be reversed once broadcast to Ethereum.

## Likelihood Explanation

The attack requires no privileged access and no victim mistake. The attacker only needs: (a) a ckETH balance sufficient to meet the minimum withdrawal amount, and (b) the ability to observe a pending NNS upgrade proposal (all proposals are public on the NNS dashboard). The NNS voting period is typically four days, providing a wide window to submit the withdrawal before the upgrade executes. The attack is repeatable across multiple addresses and multiple upgrade cycles.

## Recommendation

Add a blocklist re-check inside `create_transactions_batch` before creating a transaction. If the destination is now blocked, cancel the withdrawal request and reimburse the user's ckETH (mirroring the existing `InsufficientTransactionFee` reimbursement path via `EventType::FailedErc20WithdrawalRequest` / a new `CancelledWithdrawalRequest` event). The same guard should be applied inside `apply_state_transition` for `AcceptedEthWithdrawalRequest` during `replay_events_internal`, or alternatively at the point where requests are dequeued for processing, so that any request whose destination is blocked in the current binary is immediately moved to the reimbursement queue rather than the transaction creation queue.

## Proof of Concept

A state-machine / PocketIC integration test can confirm this without mainnet access:

1. Deploy the minter WASM with an `ETH_ADDRESS_BLOCKLIST` that does **not** contain `0xAAAA...AAAA`.
2. Call `withdraw_eth` with `recipient = 0xAAAA...AAAA`; assert `AcceptedEthWithdrawalRequest` is recorded and `retrieve_eth_status` returns `Pending`.
3. Upgrade the minter with a new WASM whose `ETH_ADDRESS_BLOCKLIST` includes `0xAAAA...AAAA`.
4. Advance the canister clock past `PROCESS_ETH_RETRIEVE_TRANSACTIONS_INTERVAL`.
5. Assert that the mock EVM RPC receives an `eth_sendRawTransaction` call whose decoded `to` field equals `0xAAAA...AAAA` — confirming the transaction was sent to the now-blocked address.

The absence of any `is_blocked` call in `create_transactions_batch` and the unconditional `record_withdrawal_request` in `apply_state_transition` make this locally reproducible without any privileged access or mainnet interaction.

    let address =

async fn sign_transactions_batch() {
    let transactions_batch: Vec<_> = read_state(|s| {
        s.eth_transactions
            .transactions_to_sign_batch(TRANSACTIONS_TO_SIGN_BATCH_SIZE)
    });
    log!(DEBUG, "Signing transactions {transactions_batch:?}");
    let results = join_all(
        transactions_batch
            .into_iter()
            .map(|(withdrawal_id, tx)| async move { (withdrawal_id, tx.sign().await) }),
    )
    .await;
    let mut errors = Vec::new();
    for (withdrawal_id, result) in results {
        match result {
            Ok(transaction) => mutate_state(|s| {
                process_event(
                    s,
                    EventType::SignedTransaction {
                        withdrawal_id,
                        transaction,
                    },
                )
            }),
            Err(e) => errors.push(e),
        }
    }
    if !errors.is_empty() {
        // At this point there might be a gap in transaction nonces between signed transactions, e.g.,
        // transactions 1,2,4,5 were signed, but 3 was not due to some unexpected error.
        // This means that transactions 4 and 5 are currently stuck until transaction 3 is signed.
        // However, we still proceed with transactions 4 and 5 since that way they might be mined faster
        // once transaction 3 is sent on the next iteration. Otherwise, we would need to re-sign transactions 4 and 5
        // and send them (together with transaction 3) on the next iteration.
        log!(INFO, "Errors encountered during signing: {errors:?}");
    }
}
async fn send_transactions_batch(latest_transaction_count: Option<TransactionCount>) {
    let latest_transaction_count = match latest_transaction_count {
        Some(latest_transaction_count) => latest_transaction_count,
        None => {
            return;
        }
    };
    let transactions_to_send: Vec<_> = read_state(|s| {
        s.eth_transactions
            .transactions_to_send_batch(latest_transaction_count, TRANSACTIONS_TO_SEND_BATCH_SIZE)
    });

    let rpc_client = read_state(rpc_client);
    let results = join_all(transactions_to_send.iter().map(async |tx| {
        rpc_client
            .send_raw_transaction(tx.raw_transaction_hex())
            .with_cycles(MIN_ATTACHED_CYCLES)
            .try_send()
            .await
            .reduce_with_strategy(AnyOf)
    }))
    .await;

    for (signed_tx, result) in zip(transactions_to_send, results) {
        log!(DEBUG, "Sent transaction {signed_tx:?}: {result:?}");
        match result {
            Ok(SendRawTransactionStatus::Ok(_)) | Ok(SendRawTransactionStatus::NonceTooLow) => {
                // In case of resubmission we may hit the case of SendRawTransactionStatus::NonceTooLow
                // if the stuck transaction was mined in the meantime.
                // It will be cleaned-up once the transaction is finalized.
            }
            Ok(SendRawTransactionStatus::InsufficientFunds)
            | Ok(SendRawTransactionStatus::NonceTooHigh) => log!(
                INFO,
                "Failed to send transaction {signed_tx:?}: {result:?}. Will retry later.",
            ),
            Err(e) => {
                log!(
                    INFO,
                    "Failed to send transaction {signed_tx:?}: {e:?}. Will retry later."
                )
            }
        };
    }
}
