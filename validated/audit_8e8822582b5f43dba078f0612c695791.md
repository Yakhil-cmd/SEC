### Title
Silent ICP Burn Failure in CMC `burn_and_log` Leads to ICP Conservation Violation - (File: `rs/nns/cmc/src/main.rs`)

---

### Summary

The Cycles Minting Canister (CMC) contains a `burn_and_log` function that is explicitly designed to swallow ICP ledger burn failures. When the ICP ledger is temporarily unavailable at the moment of the burn call, cycles (or a new canister) are already created while the corresponding ICP is never burned. This produces a permanent ICP supply conservation violation: the ICP remains stranded in the CMC's subaccount, the block is marked as processed (preventing any retry), and no recovery path exists.

---

### Finding Description

After successfully depositing cycles or creating a canister, the CMC calls `burn_and_log` to destroy the ICP that funded the operation. The function issues a `send_pb` call to the ICP ledger but unconditionally discards the result: [1](#0-0) 

The comment at line 2014 explicitly states the design intent:

> "Burning doesn't return errors — we don't want to reject the transaction notification because then it could be retried."

`burn_and_log` is invoked in three critical paths, all of which have already committed an irreversible side-effect before the call:

- `process_create_canister` — canister already created: [2](#0-1) 

- `process_top_up` — cycles already deposited: [3](#0-2) 

- `process_mint_cycles` — cycles already minted to the cycles ledger: [4](#0-3) 

The same silent-failure pattern also appears inside `refund_icp`, where the fee portion is supposed to be burned after the user refund is sent: [5](#0-4) 

Because the block is recorded as `NotifiedTopUp` / `NotifiedCreateCanister` / `NotifiedMint` before `burn_and_log` is called, the user cannot retry the notification. The ICP is permanently stranded in the CMC's per-principal/per-canister subaccount with no on-chain recovery path.

---

### Impact Explanation

**Ledger conservation bug.** Every failed `burn_and_log` call leaves ICP in the CMC's subaccount that should have been destroyed. The ICP ledger's total supply is therefore permanently higher than the protocol intends. Although the stranded ICP cannot be directly withdrawn by the user (the CMC exposes no withdrawal endpoint for these subaccounts), the invariant that "every cycle minted corresponds to burned ICP" is broken. Over time, repeated transient ledger outages accumulate unburned ICP, silently inflating the circulating supply.

---

### Likelihood Explanation

The ICP ledger can return a transient error (e.g., `TemporarilyUnavailable`, reject code 5) during routine archiving, upgrades, or subnet congestion. Any user who calls `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` during such a window triggers the condition. No special privilege is required; any unprivileged ingress sender can reach this path.

---

### Recommendation

Introduce a persistent retry queue for failed burns. When `burn_and_log` receives an error, record the `(subaccount, amount)` pair in stable state and schedule a background task (similar to the existing `ProcessLogic` task pattern used in the ckBTC minter) to retry the burn. This eliminates the conservation violation without re-opening the double-spend window, because the block is already marked as processed before the burn is attempted.

---

### Proof of Concept

1. User sends `N` ICP to `AccountIdentifier(CMC, Subaccount::from(&canister_id))`.
2. User calls `notify_top_up { block_index, canister_id }` while the ICP ledger is returning `TemporarilyUnavailable`.
3. CMC calls `deposit_cycles` → succeeds; cycles are credited to `canister_id`.
4. CMC calls `burn_and_log(sub, N)` → `send_pb` returns `Err((5, "TemporarilyUnavailable"))`.
5. `burn_and_log` logs the error and returns `()`.
6. CMC records `blocks_notified[block_index] = NotifiedTopUp(Ok(cycles))`.
7. User retries `notify_top_up` → CMC returns the cached `Ok(cycles)` without re-burning.
8. `N` ICP remains in the CMC's subaccount; ICP total supply is `N` higher than expected; no recovery path exists on-chain.

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
