### Title
Permanent Freezing of ERC-20 Tokens When `ft_transfer` Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

---

### Summary

In the production build of Aurora Engine (compiled with `--features=contract`, without `error_refund`), the `ExitToNear` precompile's `run` function uses `PromiseArgs::Create` (no callback) for all non-wNEAR ERC-20 exits. If the resulting `ft_transfer` NEAR-side call fails — for example, because the receiver account is not registered for storage with the NEP-141 contract — the already-burned ERC-20 tokens have no recovery path and are permanently frozen.

---

### Finding Description

**Step 1 — `callback_args` construction**

In `ExitToNear::run`, `callback_args` is built as:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is absent, `refund` is unconditionally `None`.

**Step 2 — `transfer_near` is `None` for non-wNEAR ERC-20 exits**

The legacy ERC-20 path (non-wNEAR, no Omni message) returns `transfer_near_args = None`:

```rust
_ => {
    (nep141_account_id, format!(...), "ft_transfer", None, ...)
}
``` [2](#0-1) 

**Step 3 — `callback_args == default()` triggers `PromiseArgs::Create`**

`ExitToNearPrecompileCallbackArgs` derives `Default`, which is `{ refund: None, transfer_near: None }`: [3](#0-2) 

The branch selection:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback registered
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [4](#0-3) 

With both fields `None`, the condition is always true for non-wNEAR ERC-20 exits in the production build, so **no `exit_to_near_precompile_callback` is ever scheduled**.

**Step 4 — Production build does not include `error_refund`**

`Makefile.toml` sets:

```
CARGO_FEATURES_BUILD = "contract"
``` [5](#0-4) 

The `error_refund` feature is a separate opt-in: [6](#0-5) 

It is never included in the standard production build task (`build` → `CARGO_FEATURES_BUILD`). [7](#0-6) 

**Step 5 — ERC-20 burn is committed before the NEAR promise executes**

The EVM transaction (including the ERC-20 `burn`) is committed atomically. The `ft_transfer` promise is dispatched asynchronously afterward. If `ft_transfer` panics on the NEAR side (e.g., `ERR_ACCOUNT_NOT_REGISTERED` because the receiver has not called `storage_deposit` on the NEP-141 contract), the EVM state is already final. The `exit_to_near_precompile_callback` that would invoke `refund_on_error` is never registered, so there is no re-mint path. [8](#0-7) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user who calls `ERC20.withdraw(amount, receiver)` where `receiver` has not registered storage with the underlying NEP-141 contract will have their ERC-20 tokens burned with no possibility of recovery. The NEP-141 tokens remain locked in Aurora's account. There is no admin escape hatch or retry mechanism for this case in the production binary.

---

### Likelihood Explanation

**Medium-High.** Storage registration is a mandatory prerequisite for receiving NEP-141 tokens on NEAR. Many NEAR accounts that are valid account IDs are not pre-registered with every NEP-141 contract. A user who is unfamiliar with this requirement — or who specifies a freshly created NEAR account — will trigger this path. The condition requires no attacker; a regular user mistake is sufficient to cause permanent loss.

---

### Recommendation

1. **Always register a callback** for the `ft_transfer` / `ft_transfer_call` promise in `ExitToNear::run`, regardless of whether `error_refund` is enabled. The callback should at minimum detect a failed promise result and re-mint (refund) the burned ERC-20 tokens to the original sender.
2. Alternatively, **enable `error_refund` in the production build** so that `refund_call_args` always populates `callback_args.refund`, ensuring `callback_args != default()` and the `PromiseArgs::Callback` branch is always taken.
3. Consider pre-validating receiver storage registration before burning, though this is not possible atomically within the EVM execution context.

---

### Proof of Concept

```
1. Deploy a NEP-141 token T; bridge it to Aurora as ERC-20 E.
2. User holds 100 E on Aurora.
3. User calls E.withdraw(100, "fresh-account.near")
   where "fresh-account.near" exists but has NOT called
   T.storage_deposit() (not registered).
4. Aurora EVM executes:
   - E burns 100 tokens from user (EVM state committed).
   - ExitToNear precompile builds callback_args = {refund: None, transfer_near: None}.
   - callback_args == default() → PromiseArgs::Create(ft_transfer).
   - No exit_to_near_precompile_callback scheduled.
5. NEAR runtime executes ft_transfer on T:
   - Panics with ERR_ACCOUNT_NOT_REGISTERED.
   - Promise fails; no callback to handle failure.
6. Result:
   - User's 100 E are burned (gone from EVM).
   - T's balance for Aurora unchanged (tokens not transferred).
   - No refund, no re-mint. Funds permanently frozen.
```
