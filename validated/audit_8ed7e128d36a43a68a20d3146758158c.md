### Title
Stale Maturity Modulation Applied to Pending Disbursements When `maturity_modulation` Is Updated Mid-Batch — (File: `rs/nns/governance/src/governance/disburse_maturity.rs`)

### Summary

The NNS Governance canister's maturity disbursement finalization reads the **current** `maturity_modulation.current_value_permyriad` at the moment each batch of disbursements is processed. Because the modulation value is updated daily by a separate timer task (`UpdateIcpXdrRateRelatedData`) and disbursements are finalized in a separate periodic task, the modulation factor applied to a disbursement can differ from the one that was in effect when the disbursement was initiated. This is by design for the 7-day delay window, but the analogous issue is that the **same batch** of pending disbursements can be split across multiple timer firings, and the modulation value can change between those firings — meaning different neurons in the same "cohort" of disbursements receive different ICP amounts for the same nominal maturity, with no record of which modulation was applied to which neuron.

### Finding Description

When a neuron owner calls `DisburseMaturity`, the governance canister records the maturity amount to disburse and a `finalize_disbursement_timestamp_seconds` 7 days in the future. The actual minting happens in `finalize_maturity_disbursement`, which is called periodically. [1](#0-0) 

At finalization time, `next_maturity_disbursement_to_finalize` reads `maturity_modulation_basis_points` from the **live** governance state — specifically `heap_data.maturity_modulation.current_value_permyriad`. [2](#0-1) 

This value is updated daily by `UpdateIcpXdrRateRelatedData`: [3](#0-2) 

The modulation can move by up to `MATURITY_MODULATION_DAILY_SPEED_LIMIT_PERMYRIAD` = 30 permyriad per day, and is bounded between −1000 and +200 permyriad (Mission 70 bounds): [4](#0-3) 

The finalization task processes one disbursement per timer firing (every ~60 seconds). If the daily modulation update fires between two consecutive disbursement finalizations, neurons whose disbursements are finalized before the update receive a different ICP amount than those finalized after — even though all disbursements were initiated at the same time and for the same nominal maturity amount.

The same pattern applies to `maybe_spawn_neurons`, which reads `maturity_modulation.current_value_permyriad` once per batch invocation but processes multiple neurons across multiple async `await` points: [5](#0-4) 

The `spawn_neuron` pre-check uses a hardcoded worst-case of 5% (`1_f64 - 0.05`) rather than the actual Mission 70 lower bound of −10%: [6](#0-5) 

This means a neuron that passes the spawn eligibility check (assuming −5% worst case) can actually receive 0 ICP if the modulation is at −10% at spawn time, since the actual minting uses the live `maturity_modulation` value, not the −5% assumption used at validation.

### Impact Explanation

**Ledger conservation bug / governance authorization bug.** Any neuron owner who initiates a `DisburseMaturity` or `SpawnNeuron` operation receives a different ICP amount than what the worst-case check at initiation time guaranteed. Specifically:

- The `spawn_neuron` eligibility check assumes worst-case modulation of −5% (`MIN_MATURITY_MODULATION_PERMYRIAD = -500` permyriad from the SNS/shared library), but the NNS `maybe_spawn_neurons` now enforces the Mission 70 range of −10% to +2%. A neuron with maturity just above the −5% threshold can be accepted for spawning but then minted with −10% modulation, producing a stake below `neuron_minimum_stake_e8s`.
- Neurons in the same disbursement cohort receive inconsistent ICP amounts depending on which timer firing finalizes them. [7](#0-6) [8](#0-7) 

### Likelihood Explanation

**Medium.** The Mission 70 maturity modulation range was recently expanded to [−1000, +200] permyriad (−10% to +2%), while the spawn eligibility check still uses the old [−500, +500] permyriad library constant as its worst-case assumption. Any neuron owner with maturity between the −5% and −10% thresholds who calls `spawn_neuron` will pass validation but receive less ICP than the minimum stake at actual spawn time. This is reachable by any unprivileged neuron controller via the standard `ManageNeuron::Spawn` ingress message. [6](#0-5) 

### Recommendation

1. **Align the spawn eligibility check with the actual Mission 70 lower bound**: replace the hardcoded `(1_f64 - 0.05)` with `(1_f64 + MATURITY_MODULATION_MIN_PERMYRIAD_MISSION_70 as f64 / 10_000_f64)` so the pre-check uses the same bound as the actual minting.
2. **Snapshot the modulation at disbursement initiation**: store the modulation value (or a conservative bound) in `MaturityDisbursement` so that finalization applies a consistent, pre-committed factor rather than the live value at finalization time.
3. **Validate the modulation range** in `finalize_maturity_disbursement` the same way `maybe_spawn_neurons` does, and skip/defer if the live value is out of the expected range.

### Proof of Concept

1. Neuron owner calls `ManageNeuron::Spawn` with maturity = `M` where `M * 0.95 >= neuron_minimum_stake_e8s` (passes the −5% check at line 2666).
2. The child neuron enters spawning state with `spawn_at_timestamp_seconds` set.
3. Before `maybe_spawn_neurons` fires, the daily `UpdateIcpXdrRateRelatedData` timer updates `maturity_modulation.current_value_permyriad` to −1000 (−10%, the Mission 70 minimum).
4. `maybe_spawn_neurons` reads the live −1000 permyriad value and calls `apply_maturity_modulation(M, -1000)`, minting `M * 0.90` ICP — which may be below `neuron_minimum_stake_e8s`.
5. The ledger transfer succeeds (the ledger has no minimum), so the spawned neuron ends up with a stake below the governance minimum, violating the invariant the pre-check was meant to enforce. [9](#0-8)

fn next_maturity_disbursement_to_finalize(
    neuron_store: &NeuronStore,
    in_flight_commands: &HashMap<u64, NeuronInFlightCommand>,
    maturity_modulation_basis_points: Option<i32>,
    now_seconds: u64,
) -> Result<Option<MaturityDisbursementFinalization>, FinalizeMaturityDisbursementError> {
    let maturity_modulation_basis_points = maturity_modulation_basis_points
        .ok_or(FinalizeMaturityDisbursementError::NoMaturityModulation)?;

    // Try to find the first neuron eligible for finalizing maturity disbursement, that is not
    // locked.
    let Some(neuron_id) = neuron_store
        .get_neuron_ids_ready_to_finalize_maturity_disbursement(now_seconds)
        .into_iter()
        .find(|neuron_id| !in_flight_commands.contains_key(&neuron_id.id))
    else {
        // If all neurons are locked, we don't need to finalize anything.
        return Ok(None);
    };
    // Either of the errors below indicates a bug in the maturity disbursement index.
    let maturity_disbursement_in_progress = neuron_store
        .with_neuron(&neuron_id, |neuron| {
            neuron.maturity_disbursements_in_progress().first().cloned()
        })
        .map_err(|_| FinalizeMaturityDisbursementError::NeuronNotFound(neuron_id))?
        .ok_or(FinalizeMaturityDisbursementError::NoMaturityDisbursement(
            neuron_id,
        ))?;

    let MaturityDisbursement {
        amount_e8s: original_maturity_e8s_equivalent,
        destination,
        finalize_disbursement_timestamp_seconds,
        timestamp_of_disbursement_seconds: _,
    } = maturity_disbursement_in_progress;

    // Make sure the disbursement is ready to be finalized. Failure at this step probably means the
    // maturity disbursement index is wrong.
    if now_seconds < finalize_disbursement_timestamp_seconds {
        return Err(
            FinalizeMaturityDisbursementError::NotTimeToFinalizeMaturityDisbursement {
                neuron_id,
                finalize_disbursement_timestamp_seconds,
                now_seconds,
            },
        );
    }

    // Apply the maturity modulation to the disbursement amount. This should not fail unless
    // something else in the system is wrong, such as an insanely large amount of maturity or an
    // incorrect maturity modulation basis points.
    let maturity_to_disburse_after_modulation_e8s = apply_maturity_modulation(
        original_maturity_e8s_equivalent,
        maturity_modulation_basis_points,
    )
    .map_err(
        |reason| FinalizeMaturityDisbursementError::MaturityModulationFailure {
            maturity_before_modulation_e8s: original_maturity_e8s_equivalent,
            maturity_modulation_basis_points,
            reason,
        },
    )?;

fn update_maturity_modulation(
    icp_price_history: &IcpPriceHistory,
    maturity_modulation: &mut MaturityModulation,
    current_day: u64,
) {
    if maturity_modulation.updated_at_days_since_epoch == Some(current_day) {
        return;
    }

    let previous = match (
        maturity_modulation.current_value_permyriad,
        maturity_modulation.updated_at_days_since_epoch,
    ) {
        (Some(p), Some(d)) => Some((p as i64, d)),
        _ => None,
    };

    match compute_maturity_modulation_permyriad(
        &icp_price_history.icp_xdr_rates,
        current_day,
        previous,
    ) {
        Ok(new_permyriad) => {
            maturity_modulation.current_value_permyriad = Some(new_permyriad as i32);
            maturity_modulation.updated_at_days_since_epoch = Some(current_day);
        }
        Err(reason) => {
            // Reaches this branch only when the buffer has no rate at or before any day in the
            // recent window (e.g., a fresh canister where every backfill fetch has failed so far,
            // or every fetched rate was zero). Log and leave the prior modulation untouched —
            // subsequent rounds will retry the missing days.
            println!(
                "{}update_maturity_modulation: skipping update: {}; leaving prior modulation \
                 unchanged",
                LOG_PREFIX, reason
            );
        }
    }
