### Title
Unbounded CPU Cost in Delegation Chain Validation via COSE RSA-8192 Keys — (`rs/validator/src/ingress_validation.rs`, `rs/crypto/internal/crypto_lib/basic_sig/rsa_pkcs1/src/lib.rs`)

---

### Summary

An unprivileged ingress sender can construct a delegation chain of up to 20 entries where every signing key is a COSE-wrapped RSA-8192 public key (`RsaSha256PublicKeyDerWrappedCose`). Each delegation hop forces the replica to perform a full RSA-8192 PKCS#1 v1.5 modular exponentiation during `verify_pkcs1_sha256`. No per-chain CPU budget is enforced. The only guard — `MAXIMUM_NUMBER_OF_DELEGATIONS = 20` — limits chain length but not per-hop computational cost. The total message fits comfortably within the 2 MB ingress size limit (~44 KB for 20 RSA-8192 keys + signatures), so the size guard provides no protection.

### Impact Explanation

RSA-8192 public-key verification (modular exponentiation with a 1024-byte modulus) is roughly 16× more expensive than RSA-2048 and orders of magnitude more expensive than Ed25519. With 20 delegations per request and no CPU budget, an attacker can force the replica to spend ~20× RSA-8192 verification time per ingress message. Pre-computing valid RSA-8192 signatures is done offline by the attacker; the replica bears the full verification cost on every submission. Flooding with many such requests concurrently saturates the `spawn_blocking` thread pool, starving legitimate ingress validation and degrading subnet throughput.

---

### Likelihood Explanation

- No privileged access required; any principal can submit an ingress message with delegations.
- RSA-8192 key generation and signing is expensive but done once offline; the resulting request is replayable (within TTL) or re-submitted with fresh expiry.
- The attack payload is ~44 KB (20 × ~1.5 KB COSE RSA-8192 keys + 20 × 1 KB signatures), well within the 2 MB limit.
- No rate-limiting on per-request cryptographic cost exists in the validation path.

---

### Recommendation

1. **Enforce a per-chain CPU budget**: Track estimated verification cost (e.g., assign weights: Ed25519 = 1, ECDSA-P256 = 2, RSA-2048 = 50, RSA-8192 = 200) and reject chains exceeding a threshold.
2. **Restrict RSA key sizes in delegations**: Cap delegation keys at RSA-2048 or RSA-4096, or disallow RSA COSE keys in delegation chains entirely (they are a WebAuthn-specific feature primarily needed for the final signing key, not intermediate delegation keys).
3. **Add concurrency limiting** on the `spawn_blocking` ingress validation pool with back-pressure to prevent CPU saturation.

// Expected: RSA-8192 chain is >100x slower per request

fn build_rsa8192_delegation_chain(n: usize) -> Vec<SignedDelegation> {
    let mut keys: Vec<RsaKeyPair8192> = (0..=n).map(|_| RsaKeyPair8192::generate()).collect();
    let mut delegations = Vec::new();
    for i in 0..n {
        let delegation = Delegation { pubkey: keys[i+1].cose_der_wrapped_pubkey(), expiry: far_future() };
        let sig = keys[i].sign_delegation(&delegation); // RSA-8192 PKCS1v15-SHA256
        delegations.push(SignedDelegation { delegation, signature: sig });
    }
    delegations
}

// Submit ingress with sender_pubkey = keys[0].cose_der_wrapped_pubkey()
// and sender_delegation = build_rsa8192_delegation_chain(20)
// Each validate_delegation call invokes verify_pkcs1_sha256 with an 8192-bit modulus.
// Ratio of wall-clock time vs Ed25519 chain should be asserted within acceptable bound.

    pub fn verify_pkcs1_sha256(&self, message: &[u8], signature: &[u8]) -> CryptoResult<()> {
        let digest = Sha256::hash(message);

        match &self
            .key
            .verify(Pkcs1v15Sign::new::<sha2::Sha256>(), &digest, signature)
        {
            Ok(_) => Ok(()),
            Err(e) => Err(CryptoError::SignatureVerification {
                algorithm: AlgorithmId::RsaSha256,
                public_key_bytes: self.as_der().to_vec(),
                sig_bytes: signature.to_vec(),
                internal_error: format!("{e:?}"),
            }),
        }
    }
