### Title
Missing `pausable` Guard on Exit Functions Allows Withdrawals During Emergency Pause — (`etc/eth-contracts/contracts/EvmErc20.sol`, `EvmErc20V2.sol`)

---

### Summary

`withdrawToNear` and `withdrawToEthereum` in both `EvmErc20` and `EvmErc20V2` carry no `pausable` modifier. After `adminPause(type(uint).max)` sets every pause bit, any non-admin user can still invoke these functions, burning their tokens and dispatching a precompile call whose return value is never checked.

---

### Finding Description

`AdminControlled` defines the `pausable` modifier: [1](#0-0) 

When `adminPause(type(uint).max)` is called, `paused` becomes `2²⁵⁶ − 1`, so `(paused & flag) == 0` is false for every non-zero flag, blocking any non-admin caller on any function decorated with `pausable`. [2](#0-1) 

However, a grep across all `.sol` files confirms `pausable` is **only defined** in `AdminControlled.sol` and is **never applied** to any function in `EvmErc20` or `EvmErc20V2`. Both exit functions are bare `external override`: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

Compounding the issue, the low-level `call` to the exit precompile stores its return value in `res` but never inspects it. A failed precompile call (returning 0) does not revert the transaction, so `_burn` completes and the user's tokens are permanently destroyed with no corresponding asset release. [7](#0-6) 

---

### Impact Explanation

Two concrete harm paths exist:

1. **Permanent fund loss (Critical):** Precompile call fails silently during the paused/inconsistent engine state. Tokens are burned on the EVM side; no NEAR or Ethereum assets are released. The user's balance is gone with no recourse.

2. **Supply/backing desync (Insolvency):** If the admin paused because the engine is in an inconsistent state (e.g., mid-upgrade, known accounting bug), continued exits processed by a misbehaving precompile can release backing assets that should be frozen, leaving remaining token holders with claims against insufficient reserves.

---

### Likelihood Explanation

The precondition — admin calling `adminPause(type(uint).max)` — is the intended emergency mechanism. It is not an attacker action. Any token holder can then immediately call `withdrawToNear` or `withdrawToEthereum` with no special privilege. The path is fully permissionless and requires zero additional conditions beyond the admin having used the pause feature.

---

### Recommendation

Apply the `pausable` modifier to both exit functions in `EvmErc20` and `EvmErc20V2`, using a dedicated pause bit (e.g., bit 2 for withdrawals):

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount)
    external
    override
    pausable(PAUSE_WITHDRAW)   // add this
{
    ...
}

function withdrawToEthereum(address recipient, uint256 amount)
    external
    override
    pausable(PAUSE_WITHDRAW)   // add this
{
    ...
}
```

Additionally, check the precompile call return value and revert on failure to prevent silent token burns:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

### Proof of Concept

```solidity
// 1. Deploy EvmErc20 with attacker as non-admin holder
// 2. Admin calls adminPause(type(uint).max) — all bits set
// 3. Non-admin holder calls withdrawToNear(recipientBytes, amount)
// 4. Assert: call does NOT revert (no pausable guard)
// 5. Assert: holder's ERC-20 balance decreased by `amount` (_burn executed)
// 6. Assert: precompile call was dispatched (even if it fails silently)
// Result: tokens destroyed during a fully-paused emergency state,
//         with no guarantee the exit was processed on the NEAR side.
```

The proof requires no privileged access beyond the admin's own legitimate use of `adminPause`. The non-admin exit succeeds unconditionally because neither `withdrawToNear` nor `withdrawToEthereum` carries the `pausable` modifier. [8](#0-7) [9](#0-8)
