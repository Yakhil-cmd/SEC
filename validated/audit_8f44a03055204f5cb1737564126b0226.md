Audit Report

## Title
Unbounded Growth of `retrieve_btc_account_to_block_indices` Vec Causes Permanent DoS of `retrieve_btc_status_v2_by_account` - (File: rs/bitcoin/ckbtc/minter/src/state.rs)

## Summary
The `CkBtcMinterState` field `retrieve_btc_account_to_block_indices: BTreeMap<Account, Vec<u64>>` accumulates every accepted `retrieve_btc` block index per account and is never pruned. The query `retrieve_btc_status_v2_by_account` iterates the full Vec for a given account, invoking the multi-lookup `retrieve_btc_status_v2` for each entry. An unprivileged ckBTC holder can grow this Vec without bound across multiple Bitcoin confirmation cycles, eventually causing every call to `retrieve_btc_status_v2_by_account` for that account to exceed the IC instruction limit and permanently trap.

## Finding Description
**Root cause — non-shrinking Vec:**

`accept_retrieve_btc_request` in `rs/bitcoin/ckbtc/minter/src/state/audit.rs` (L27–33) unconditionally appends to the per-account Vec:
```rust
state
    .retrieve_btc_account_to_block_indices
    .entry(account)
    .and_modify(|entry| entry.push(request.block_index))
    .or_insert(vec![request.block_index]);
```
A search across all files touching `retrieve_btc_account_to_block_indices` (state.rs ×7, eventlog.rs ×2, audit.rs ×1) reveals **no removal, retain, truncate, drain, or clear operation** on the inner `Vec<u64>`. Finalized requests are moved out of `pending_retrieve_btc_requests` and into `finalized_requests`, but their block indices remain in this map permanently.

**Unbounded loop — `retrieve_btc_status_v2_by_account`:**

`state.rs` L807–821 clones the full Vec and maps every entry through `retrieve_btc_status_v2` (L824–868), which performs up to five sequential BTreeMap lookups (`pending_reimbursements`, `pending_withdrawal_reimbursements`, `reimbursed_transactions`, `reimbursed_withdrawals`, and a full `retrieve_btc_status` call). The per-entry instruction cost is non-trivial; with enough accumulated entries the aggregate cost exceeds the IC per-message instruction limit.

**Guard is insufficient:**

`MAX_CONCURRENT_PENDING_REQUESTS` (retrieve_btc.rs L174) only throttles the number of *incomplete* requests at any moment. Once a batch is confirmed on Bitcoin, those entries leave the incomplete count but their block indices remain in the Vec. An attacker can cycle: submit up to the limit → wait for Bitcoin confirmations → repeat, accumulating indices indefinitely.

## Impact Explanation
Once the Vec for a target account grows large enough, every invocation of `retrieve_btc_status_v2_by_account` for that account traps with an instruction-limit-exceeded error. The DoS is permanent because the Vec never shrinks. The affected account permanently loses the ability to query the status of any of its withdrawal requests via this endpoint. This matches the allowed High impact: **Application/platform-level DoS with concrete user harm, not based on raw volumetric DDoS** — the ckBTC withdrawal status query is a core financial-integration endpoint.

## Likelihood Explanation
The `retrieve_btc` endpoint is publicly callable by any ckBTC holder with no special privileges. The only throttle (`MAX_CONCURRENT_PENDING_REQUESTS`) is bypassed by cycling through Bitcoin confirmation rounds. The economic cost is real (minimum retrieve amount per block index, plus Bitcoin fees), but finite and predictable. A well-funded attacker can self-target or target another account (by specifying the `target` parameter in `retrieve_btc_status_v2_by_account`) after accumulating enough indices under their own account. The attack is repeatable and deterministic.

## Recommendation
1. **Prune terminal entries**: When a request reaches a terminal state (confirmed, reimbursed, quarantined), remove its block index from `retrieve_btc_account_to_block_indices`.
2. **Cap Vec length**: Enforce a maximum number of entries per account (e.g., keep only the most recent N block indices) in `accept_retrieve_btc_request`.
3. **Paginate the query**: Add `start` and `length` parameters to `retrieve_btc_status_v2_by_account` so it never iterates the full Vec in a single call.

## Proof of Concept
1. Attacker holds ckBTC and calls `retrieve_btc` (or `retrieve_btc_with_approval`) up to `MAX_CONCURRENT_PENDING_REQUESTS` times with the minimum allowed amount.
2. Each accepted request appends a new `block_index` to `retrieve_btc_account_to_block_indices[attacker_account]` via `accept_retrieve_btc_request` (audit.rs L31).
3. The minter processes and finalizes the batch on Bitcoin. The block indices remain in the Vec.
4. Attacker repeats steps 1–3 across multiple Bitcoin confirmation cycles.
5. After enough cycles, calling `retrieve_btc_status_v2_by_account` for the attacker's account iterates the full Vec (state.rs L813–819), invoking `retrieve_btc_status_v2` (state.rs L824–868) for each entry. The accumulated instruction cost exceeds the IC per-message limit and the query permanently traps.
6. A deterministic integration test using PocketIC can reproduce this by replaying `AcceptedRetrieveBtcRequest` events in a loop, populating the Vec to a threshold count, then asserting that `retrieve_btc_status_v2_by_account` traps.
