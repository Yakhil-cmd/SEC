Audit Report

## Title
Global Withdrawal DoS via 1000-Nonce Saturation in `withdrawal_requests_batch` - (File: `rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

## Summary

`withdrawal_requests_batch` enforces a hard ceiling of `MAX_NUM_PENDING_TRANSACTION_NONCES = 1000` across `created_tx` and `sent_tx`. Once that ceiling is reached the function returns an empty batch, and the minter's timer loop stops converting any pending request into an Ethereum transaction. An attacker who can afford the gas cost of ~1 000 Ethereum transactions can hold the entire withdrawal pipeline frozen for as long as those transactions remain unconfirmed on Ethereum. The claim's assertion that `pending_withdrawal_requests` is unbounded is incorrect — `retrieve_withdraw_guard` enforces `MAX_PENDING = 100` — but this guard does not prevent the nonce-saturation attack; it only means the attacker must cycle through the 100-item queue ~10 times rather than filling it in one pass.

## Finding Description

**Root cause — the 1000-nonce ceiling:**

`withdrawal_requests_batch` in `rs/ethereum/cketh/minter/src/state/transactions/mod.rs` (L906–923) computes:

```rust
const MAX_NUM_PENDING_TRANSACTION_NONCES: usize = 1000;
let unique_pending_transaction_nonces: BTreeSet<_> =
    self.created_tx.keys().chain(self.sent_tx.keys()).collect();
let actual_batch_size = min(
    MAX_NUM_PENDING_TRANSACTION_NONCES
        .saturating_sub(unique_pending_transaction_nonces.len()),
    requested_batch_size,
);
```

When `created_tx ∪ sent_tx` reaches 1 000 unique nonces, `saturating_sub` yields 0, `actual_batch_size` is 0, and the function returns an empty `Vec`. The caller `create_transactions_batch` in `rs/ethereum/cketh/minter/src/withdraw.rs` (L249–293) iterates over this empty result and does nothing. Every request already in `pending_withdrawal_requests` remains stuck there indefinitely.

**Why the `MAX_PENDING = 100` guard does not prevent the attack:**

`retrieve_withdraw_guard` in `rs/ethereum/cketh/minter/src/guard/mod.rs` (L50–68) checks `withdrawal_requests_len()` (the length of `pending_withdrawal_requests`) against `MAX_PENDING = 100` at the moment a new request is accepted. This caps the pending queue at 100 items at any instant. However, `created_tx` and `sent_tx` are entirely separate maps that are not subject to this cap. Once a request is promoted from `pending_withdrawal_requests` into `created_tx` by the timer, it no longer counts toward the 100-item guard. The attacker therefore cycles:

1. Fill `pending_withdrawal_requests` to 100 (using up to 100 distinct principals, since the guard also blocks the same principal from having two concurrent in-flight requests).
2. Wait for the timer (batch size 5, `WITHDRAWAL_REQUESTS_BATCH_SIZE = 5` at L39 of `withdraw.rs`) to drain the queue into `created_tx`.
3. Refill the queue to 100 again.
4. Repeat ~10 times until `created_tx ∪ sent_tx` holds 1 000 nonces.

At that point `withdrawal_requests_batch` returns 0 on every timer tick. All legitimate users' requests in `pending_withdrawal_requests` are never promoted. The freeze persists until the attacker's 1 000 Ethereum transactions finalize.

**No per-principal nonce quota exists.** There is no mechanism that limits how many nonces a single actor can occupy in `created_tx` or `sent_tx`.

## Impact Explanation

Once the 1 000-nonce budget is exhausted, `withdrawal_requests_batch` returns an empty slice on every timer tick, and `create_transactions_batch` processes zero requests. Every legitimate user whose withdrawal is already queued in `pending_withdrawal_requests` is frozen indefinitely. No new Ethereum transactions are created for any user until the attacker's 1 000 transactions finalize on-chain. This is a global, protocol-level denial of service on all ckETH and ckERC20 withdrawals — a concrete, non-volumetric application-level DoS on an in-scope Chain Fusion / ck-token financial integration with direct user harm (funds locked in the bridge, unable to be withdrawn). This matches the allowed High impact: *"Application/platform-level DoS… or subnet availability impact not based on raw volumetric DDoS"* and *"Significant Chain Fusion, ck-token, ledger… security impact with concrete user or protocol harm."*

## Likelihood Explanation

The attacker's ckETH principal is not destroyed — it is converted to ETH at the destination address. The net cost is only Ethereum gas fees for ~1 000 transactions (~21 000 gas each for ETH transfers, ~65 000 for ERC-20). During low-fee periods this is economically feasible. The attack requires no privileged role, no governance vote, and no subnet-majority corruption. The `MAX_PENDING = 100` guard adds friction (the attacker must cycle through the queue ~10 times) but does not raise the economic bar meaningfully. During Ethereum congestion the attacker can keep resubmitting with low gas to delay finalization, extending the freeze window at no additional cost.

## Recommendation

**Short term:** Add a per-principal nonce quota inside `withdrawal_requests_batch` or at the point where a nonce is assigned in `create_transactions_batch`. Reject or defer requests from any principal that already occupies more than a configurable fraction (e.g., 10–20) of the 1 000-nonce budget. This prevents a single actor from monopolising the budget.

**Long term:** Replace the hard global ceiling with a sliding window or a per-principal fair-share scheduler so that a single actor cannot starve all other users. Consider also capping the total number of unconfirmed nonces per originating principal in `created_tx` and `sent_tx` directly, mirroring the existing `MAX_PENDING` guard on the pending queue.

## Proof of Concept

1. Attacker acquires ≥ 1 000 × `cketh_minimum_withdrawal_amount` of ckETH, distributed across ≥ 100 principals (or sequentially from fewer principals between timer ticks).
2. Attacker calls `withdraw_eth` from 100 distinct principals. Each call burns ckETH and appends to `pending_withdrawal_requests` via `record_withdrawal_request`. The guard allows up to 100 concurrent entries.
3. The minter timer runs `create_transactions_batch` with `WITHDRAWAL_REQUESTS_BATCH_SIZE = 5`, processing 5 requests per tick and moving them into `created_tx`.
4. After signing, they move to `sent_tx` and are broadcast to Ethereum with a deliberately low `max_priority_fee_per_gas` so they sit unconfirmed.
5. As `pending_withdrawal_requests` drains below 100, the attacker refills it with another 100 requests.
6. After ~10 cycles (~200 timer ticks total), `created_tx.keys().chain(sent_tx.keys())` reaches 1 000 unique nonces.
7. `withdrawal_requests_batch` now returns `[]` on every tick: `actual_batch_size = min(1000 − 1000, 5) = 0`.
8. All legitimate users' requests remain in `pending_withdrawal_requests` and are never promoted. The ckETH/ckERC20 withdrawal bridge is frozen for all users until the attacker's transactions confirm on Ethereum.

A deterministic integration test can reproduce this by: (a) pre-populating `created_tx` and `sent_tx` with 1 000 mock nonces in a `PocketIC` or unit-test harness, (b) calling `withdrawal_requests_batch(5)`, and (c) asserting the returned `Vec` is empty.
