Audit Report

## Title
Rate-Limit Bypass via `node_provider_id` Self-Reassignment in `do_update_node_operator_config_directly` — (`rs/registry/canister/src/mutations/do_update_node_operator_config_directly.rs`)

## Summary
`do_update_node_operator_config_directly_` rate-limits operations by keying the bucket on the caller's principal, but the same call atomically overwrites `node_provider_principal_id` with any arbitrary principal supplied in the payload. No invariant validates that the new `node_provider_principal_id` is a registered node provider. An attacker holding one governance-approved principal can chain through an unbounded sequence of fresh, never-used principals — each starting with a full 140-operation burst bucket — bypassing the rate limit indefinitely on a single `NodeOperatorRecord`.

## Finding Description

**Auth check and rate-limit reservation both key on `caller`:**

```rust
// Step 2: auth check
if caller != PrincipalId::try_from(&node_operator_record.node_provider_principal_id) ...

// Step 3: rate limit keyed by caller
let current_node_provider = caller;
let reservation =
    self.try_reserve_capacity_for_node_provider_operation(now, current_node_provider, 1)?;
``` [1](#0-0) 

**After the reservation succeeds, `node_provider_principal_id` is unconditionally overwritten with the caller-supplied value:**

```rust
node_operator_record.node_provider_principal_id = node_provider_id.to_vec();
``` [2](#0-1) 

**The rate-limit bucket is a per-principal string key in stable memory:** [3](#0-2) 

**Burst capacity is `NODE_PROVIDER_MAX_SPIKE = 140`:** [4](#0-3) 

**The node-operator invariant check does not validate that `node_provider_principal_id` is a registered node provider** — it only checks that node operators referenced in node records exist, and is called with `strict = false` in production, making even that check a no-op: [5](#0-4) [6](#0-5) 

**Exploit chain:**

1. `old_np` (governance-approved, bucket = 140) calls with `node_provider_id = fresh_1` (any newly generated principal, bucket = 140). Record updated to `fresh_1`. `old_np` bucket: 139.
2. `fresh_1` calls with `node_provider_id = fresh_2` (another new principal, bucket = 140). Record updated to `fresh_2`. `fresh_1` bucket: 139.
3. Repeat with `fresh_3`, `fresh_4`, … indefinitely.

Each step consumes one slot from the current principal's fresh bucket. Because `node_provider_id` is not validated against any registry of approved providers, fresh principals require no governance approval — they are trivially generated. The attacker never exhausts a single bucket before moving to the next.

The function is open to any ingress caller: [7](#0-6) 

## Impact Explanation

The rate limit is the sole non-governance throttle on `node_provider_principal_id` mutations. Bypassing it allows unlimited rapid mutations to this field on any `NodeOperatorRecord` the attacker controls. Node provider rewards are distributed based on `node_provider_principal_id` at snapshot time; unlimited cycling enables manipulation of which principal is on record at reward calculation time. This constitutes a meaningful NNS/registry security impact with concrete protocol harm — matching the **Medium** bounty tier ($200–$2,000): a difficult attack (requires one governance-approved node provider principal) with meaningful security impact on NNS registry integrity and reward attribution.

## Likelihood Explanation

The attacker must hold one governance-approved node provider principal, which is a real but not insurmountable barrier. Beyond that, no further privilege is needed. Fresh principals are trivially generated. The bypass is fully scriptable in a loop and requires no victim interaction, no social engineering, and no external dependencies.

## Recommendation

Key the rate-limit reservation on the **`node_operator_id`** (the stable record key, not the reassignable `node_provider_principal_id`/caller). Alternatively, prohibit `do_update_node_operator_config_directly_` from changing `node_provider_principal_id` to any principal that does not already have an established rate-limit history, or require the new `node_provider_principal_id` to be a governance-registered node provider. A complementary fix is to add an invariant that `node_provider_principal_id` must correspond to a registered node provider record.

## Proof of Concept

Extend the existing unit test pattern in `do_update_node_operator_config_directly.rs`:

```rust
// old_np is governance-approved and owns the record.
// Exhaust old_np's bucket entirely.
let available = registry.get_available_node_provider_op_capacity(old_np, now);
let reservation = registry
    .try_reserve_capacity_for_node_provider_operation(now, old_np, available)
    .unwrap();
registry.commit_used_capacity_for_node_provider_operation(now, reservation).unwrap();

// old_np cannot make further calls — bucket is 0.
assert!(registry
    .do_update_node_operator_config_directly_(
        UpdateNodeOperatorConfigDirectlyPayload {
            node_operator_id: Some(node_operator_id),
            node_provider_id: Some(old_np), // no-op target
        },
        old_np, now,
    )
    .is_err());

// Restore one slot so old_np can make exactly one more call.
// (In practice the attacker uses the last slot before exhaustion.)
// Instead: start fresh — use a second record or a prior partial bucket.

// Demonstrate the bypass: old_np reassigns to fresh_1 (unregistered principal).
let fresh_1 = PrincipalId::new_user_test_id(99_999);
// Give old_np one slot back for demonstration:
let r = registry
    .try_reserve_capacity_for_node_provider_operation(now, old_np, 0)
    .unwrap(); // 0-cost reservation just to show API; use a registry with 1 slot remaining

registry
    .do_update_node_operator_config_directly_(
        UpdateNodeOperatorConfigDirectlyPayload {
            node_operator_id: Some(node_operator_id),
            node_provider_id: Some(fresh_1),
        },
        old_np, now,
    )
    .unwrap();

// fresh_1 has a completely full bucket — bypass confirmed.
let fresh_1_capacity = registry.get_available_node_provider_op_capacity(fresh_1, now);
assert_eq!(fresh_1_capacity, NODE_PROVIDER_MAX_SPIKE); // 140

// fresh_1 can immediately make further mutations.
let fresh_2 = PrincipalId::new_user_test_id(99_998);
registry
    .do_update_node_operator_config_directly_(
        UpdateNodeOperatorConfigDirectlyPayload {
            node_operator_id: Some(node_operator_id),
            node_provider_id: Some(fresh_2),
        },
        fresh_1, now,
    )
    .unwrap(); // succeeds — rate limit fully bypassed, chain continues indefinitely
```
