### Title
Ceiling Rounding in `native_per_gas` Computation Allows Users to Consume More Native Resources Than Paid For - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_gas` is computed using `div_ceil` (ceiling division), while the inverse conversion `native_used / native_per_gas` in the `delta_gas` refund path uses floor division. Both round in the user's favor. An unprivileged transaction sender can choose a `gas_price` value that maximizes the rounding benefit, obtaining up to ~2× the native resource budget they paid for, while the `delta_gas` correction underestimates the extra gas to charge. The operator/protocol bears the uncompensated proving cost.

---

### Finding Description

**Step 1 — `native_per_gas` is rounded UP (ceiling):**

In `validation_impl.rs` line 135, `native_per_gas` is computed as:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [1](#0-0) 

The same ceiling division is used for L1 transactions in `process_l1_transaction.rs`:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

**Step 2 — Native resource budget is inflated:**

The native limit is then set as `native_prepaid_from_gas = native_per_gas * gas_limit`: [3](#0-2) 

Because `native_per_gas` is rounded up, the user receives a native budget of `gas_limit * ceil(gas_price / native_price)`, which is strictly larger than the fair value `gas_limit * gas_price / native_price`.

**Step 3 — `delta_gas` correction uses floor division with the inflated denominator:**

In `refund_calculation.rs` line 72, the gas equivalent of native consumption is computed as:

```rust
(native_used / native_per_gas) as i64 - (gas_used as i64)
``` [4](#0-3) 

Rust integer division is floor division. With `native_per_gas` inflated by ceiling rounding, `floor(native_used / ceil(gas_price/native_price))` is strictly smaller than the fair value `floor(native_used * native_price / gas_price)`. This means `delta_gas` is underestimated, so less extra gas is charged for native consumption.

**The asymmetry:** Both rounding operations favor the user — the budget is rounded up, and the gas-equivalent of consumption is rounded down.

The documentation confirms the intended formula is `nativePerGas := gasPrice/nativePrice` (exact division), but the implementation uses ceiling: [5](#0-4) 

---

### Impact Explanation

**Concrete example:**

Let `native_price = 1000`, `gas_price = 1001` (just above `native_price`), `gas_limit = 100,000`.

- Actual: `native_per_gas = ceil(1001/1000) = 2`, native budget = `200,000`
- Fair: `native_per_gas_fair = 1.001`, native budget = `100,100`

The user receives **~2× the native resources** they paid for.

After execution with `native_used = 150,000`, `gas_used = 50,000`:
- Actual `delta_gas = floor(150,000 / 2) - 50,000 = 25,000`; total gas charged = `75,000`; tokens paid = `1001 × 75,000 = 75,075,000`
- Fair `delta_gas = floor(150,000 / 1.001) - 50,000 ≈ 99,850`; total gas charged = `149,850`; tokens paid ≈ `149,999,850`

The user pays **~50% of the fair cost** for native-dominated computation. The operator/protocol must prove the extra computation without compensation, constituting a direct economic loss to the operator and a resource exhaustion vector against the proving system.

---

### Likelihood Explanation

Any unprivileged L2 or L1→L2 transaction sender can trigger this by setting `gas_price = k * native_price + 1` for the smallest `k` satisfying `gas_price >= base_fee`. No special access, governance role, or external oracle manipulation is required. The attacker controls `gas_price` directly in the transaction fields. The rounding benefit is maximized when `gas_price mod native_price` is close to `native_price - 1`, which is a freely choosable parameter. The attack is repeatable across every transaction.

---

### Recommendation

Replace `div_ceil` with floor division (`wrapping_div`) for `native_per_gas` to match the documented formula `nativePerGas := gasPrice / nativePrice`:

```rust
// In validation_impl.rs and process_l1_transaction.rs
let native_per_gas = u256_try_to_u64(&gas_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::NativeResourcesAreTooExpensive))?;
```

This is consistent with how `native_per_pubdata` is already computed using `wrapping_div` (floor division): [6](#0-5) 

Alternatively, if ceiling is intentional for `native_per_gas` (to ensure users always have enough native budget), the `delta_gas` correction must use the same ceiling convention: `native_used.div_ceil(native_per_gas)` instead of floor division, so that the two rounding directions cancel rather than compound.

---

### Proof of Concept

1. Operator sets `native_price = 1000`, `base_fee = 1000`.
2. Attacker submits an L2 transaction with `gas_price = 1001`, `gas_limit = 30_000_000` (block gas limit).
3. Bootloader computes `native_per_gas = ceil(1001/1000) = 2`.
4. Native budget = `30_000_000 × 2 = 60_000_000` (fair: `30_030_000`).
5. Attacker's contract performs computation consuming `50_000_000` native units and `1_000_000` EVM gas.
6. `delta_gas = floor(50_000_000 / 2) - 1_000_000 = 24_000_000`.
7. Total gas charged = `25_000_000`; tokens paid = `1001 × 25_000_000 = 25,025,000,000`.
8. Fair tokens = `1001 × (1_000_000 + floor(50_000_000 / 1.001)) ≈ 1001 × 50_950_049 ≈ 51,000,999,049`.
9. Attacker pays ~49% of the fair cost, consuming ~67% more native resources than paid for. The operator proves the excess computation at a loss. [1](#0-0) [7](#0-6)
