### Title
Missing Minimum LP-Token Output Parameter in `TreasuryManager` Deposit/Withdraw API Enables Pre-Proposal-Execution Pool Manipulation Against SNS Treasury - (File: `rs/sns/treasury_manager/treasury_manager.did`, `rs/sns/governance/src/extensions.rs`)

---

### Summary

The `TreasuryManager` Candid interface and its Rust trait define `DepositRequest` and `WithdrawRequest` with no minimum-output (slippage) parameter. The SNS governance canister executes these operations without enforcing any minimum LP-token or token-out guarantee. Because SNS proposals are public and have multi-day voting windows, an unprivileged DEX liquidity provider can observe a pending `TreasuryManagerDeposit` or `TreasuryManagerWithdraw` proposal, manipulate the DEX pool state before execution, and extract value from the SNS treasury.

---

### Finding Description

`DepositRequest` in the `TreasuryManager` API contains only `allowances` (the input token amounts):

```candid
type DepositRequest = record {
  allowances : vec Allowance;
};
```

There is no `min_lp_tokens_out` field. [1](#0-0) 

`WithdrawRequest` similarly contains only `withdraw_accounts` with no `min_tokens_out` field. [2](#0-1) 

The Rust `TreasuryManager` trait mirrors this design exactly, with `deposit` and `withdraw` accepting only the request structs and returning `TreasuryManagerResult` (a balance snapshot), with no minimum-output enforcement at the trait level. [3](#0-2) 

On the governance side, `validate_deposit_operation_impl` only checks that the requested amount does not exceed 50% of the current treasury balance. It performs no check on the minimum LP tokens the treasury should receive in return. [4](#0-3) 

`execute_treasury_manager_deposit` approves the allowance, calls `deposit`, decodes the returned `Balances`, and logs them — but never validates the returned balances against any minimum. [5](#0-4) 

`execute_treasury_manager_withdraw` has the same pattern: it calls `withdraw` and logs the result without asserting any minimum token-out. [6](#0-5) 

The codebase itself acknowledges this gap. The `treasury_manager.did` file labels it a "Known Security Risk":

> Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved. [7](#0-6) 

The proposal-rendering function `validate_and_render_register_extension` also emits a WARNING to voters about front-running and sandwich attacks, but this is only informational — no enforcement exists in the execution path. [8](#0-7) 

---

### Impact Explanation

An attacker who controls DEX liquidity can manipulate the pool ratio between proposal adoption and execution, causing the SNS treasury to receive fewer LP tokens (on deposit) or fewer underlying tokens (on withdrawal) than the ratio implied at proposal time. The attacker then removes their liquidity and captures the difference. The SNS treasury suffers a direct, permanent loss of funds proportional to the pool manipulation. Because the governance canister never validates the returned balances against a minimum, the operation succeeds and the loss is silently accepted.

---

### Likelihood Explanation

SNS proposals have a voting period measured in days and are fully public on-chain. Any DEX liquidity provider — an unprivileged canister caller — can observe a pending `TreasuryManagerDeposit` or `TreasuryManagerWithdraw` proposal, compute the expected execution parameters, and manipulate the pool state before the proposal executes. No privileged access, key compromise, or consensus-level attack is required. The attack window is large (days), and the required capital is bounded by the treasury deposit size. Likelihood is **medium**: it requires meaningful DEX liquidity but is straightforward to execute given the long, public proposal lifecycle.

---

### Recommendation

1. Add a `min_lp_tokens_out : opt nat` field to `DepositRequest` and a `min_tokens_out : opt vec record { principal; nat }` field to `WithdrawRequest` in `treasury_manager.did` and the corresponding Rust structs in `rs/sns/treasury_manager/src/lib.rs`.
2. Extend `validate_deposit_operation_impl` in `rs/sns/governance/src/extensions.rs` to require and validate these minimum-output fields.
3. In `execute_treasury_manager_deposit` and `execute_treasury_manager_withdraw`, assert that the returned `Balances` satisfy the minimum-output constraints before returning `Ok(())`. If the constraint is violated, revert the allowance approval and return an error.
4. Consider encoding the minimum-output values directly in the proposal payload so that voters explicitly approve the slippage tolerance, making the trade-off transparent and auditable.

---

### Proof of Concept

**Setup:** An SNS has a treasury with 100 SNS tokens and 100 ICP. A `TreasuryManagerDeposit` proposal is submitted to deposit 50 SNS + 50 ICP into a DEX liquidity pool at the current 1:1 ratio.

**Attack steps (all callable by an unprivileged DEX user):**

1. **Observe** the pending proposal on-chain (public, days before execution).
2. **Manipulate** the DEX pool: add a large amount of ICP (e.g., 10,000 ICP) to the pool, skewing the ratio to ~200:1 (ICP:SNS).
3. **Wait** for the proposal to pass and `execute_treasury_manager_deposit` to fire. The treasury manager calls `deposit` with 50 SNS + 50 ICP. Because the pool ratio is now skewed, the treasury receives far fewer LP tokens than it would at the original 1:1 ratio. `execute_treasury_manager_deposit` logs the returned balances and returns `Ok(())` — no minimum check exists.
4. **Withdraw** the 10,000 ICP from the pool. The attacker now holds more value than they deposited, having extracted the difference from the SNS treasury's LP position.

The SNS treasury's LP tokens are worth less than the deposited assets, and the loss is permanent. The governance canister has no mechanism to detect or revert this outcome.

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

/// Execute a treasury manager deposit operation
async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())
}
