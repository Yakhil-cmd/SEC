Audit Report

## Title
Unprivileged Client-Reachable `todo!()` Panic in Rosetta Block Hash Lookup — (`rs/rosetta-api/icp/src/request_handler.rs`)

## Summary
When `RosettaBlocksMode::Enabled` is active, any unauthenticated HTTP client can send a `POST /block` request with only a `hash` field (no `index`) and trigger an unconditional `todo!()` panic inside `get_block()`. The `todo!()` macro expands to `panic!()` and is reached whenever `get_verified_block_by_hash` returns `Err(ApiError::InvalidBlockId(_, _))`, which occurs for any hash that is not a verified ICP block hash — including any random 32-byte hex string.

## Finding Description
The public `/block` handler in `rosetta_server.rs` (lines 58–68) accepts unauthenticated `POST` requests and delegates directly to `RosettaRequestHandler::block()`. [1](#0-0) 

`block()` calls `get_block()` with the client-supplied `block_identifier`: [2](#0-1) 

Inside `get_block()`, the branch matching `PartialBlockIdentifier { index: None, hash: Some(hash) }` checks `is_rosetta_blocks_mode_enabled()` (the only gate, satisfied by deployment config) and then calls `get_verified_block_by_hash`. When that returns `Err(ApiError::InvalidBlockId(_, _))` — which it does for any hash not in the verified ICP block store — the error arm unconditionally executes `todo!()`: [3](#0-2) 

No authentication, authorization, or rate-limiting guard exists between the HTTP ingress and this branch. The `HttpServer` is constructed without any `catch_unwind` middleware or panic boundary wrapping the `/block` handler: [4](#0-3) 

## Impact Explanation
This matches the allowed ICP bounty impact: **"Significant Rosetta/boundary/API security impact with concrete user or protocol harm" (High, $2,000–$10,000)**. The `todo!()` panic aborts the handler task on every such request. Actix-web's internal `catch_unwind` may prevent full process termination, but the panicking task is torn down, the client receives a connection-level error rather than a structured API error, and the behavior is repeatable and unauthenticated. Any deployment of the ICP Rosetta node with `RosettaBlocksMode::Enabled` is affected.

## Likelihood Explanation
- Requires only a valid HTTP client and knowledge of the public Rosetta `/block` endpoint (documented in the Rosetta API spec).
- `RosettaBlocksMode::Enabled` is the intended production configuration for ICP Rosetta nodes supporting the Rosetta block format.
- The attacker does not need to know any specific hash; any 32-byte hex string that is not a verified ICP block hash (including a valid Rosetta block hash) triggers the panic.
- The attack is trivially repeatable with no rate-limiting or authentication barrier.

## Recommendation
Replace the `todo!()` with a structured error return:

```rust
Err(ApiError::InvalidBlockId(_, _)) => {
    Err(ApiError::invalid_block_id(
        "Fetching Rosetta Blocks by hash is not yet supported; provide an index."
    ))
}
``` [5](#0-4)
