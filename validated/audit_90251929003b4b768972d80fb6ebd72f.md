### Title
`get_minter_info` Query Returns Inaccurate `retrieve_btc_min_amount` That Does Not Reflect Minter Mode Restrictions - (File: rs/bitcoin/ckbtc/minter/src/main.rs)

### Summary
The ckBTC minter's `get_minter_info` query endpoint returns a `retrieve_btc_min_amount` value that does not account for the minter's operational `Mode`. When the minter is in `ReadOnly` or `RestrictedTo` mode, all withdrawal calls will be rejected for the caller, yet `get_minter_info` continues to report a non-zero `retrieve_btc_min_amount` as if withdrawals are available. This is a direct analog to the `sUSX` `maxWithdraw`/`maxRedeem` not accounting for `mintCap`: a publicly-callable query function returns a value that misrepresents the actual withdrawal capability, leading users to construct and submit transactions that will fail.

### Finding Description
The `get_minter_info` query function in the ckBTC minter returns a `MinterInfo` struct containing `retrieve_btc_min_amount` — the minimum amount a user can withdraw. However, the function reads only fee-based and configuration state, and never checks `s.mode`:

```rust
// rs/bitcoin/ckbtc/minter/src/main.rs
#[query]
fn get_minter_info() -> MinterInfo {
    read_state(|s| MinterInfo {
        check_fee: s.check_fee,
        min_confirmations: s.min_confirmations,
        retrieve_btc_min_amount: s.fee_based_retrieve_btc_min_amount,  // ← no mode check
        deposit_btc_min_amount: Some(s.effective_deposit_min_btc_amount()),
    })
}
```

Meanwhile, the actual withdrawal functions `retrieve_btc` and `retrieve_btc_with_approval` both check `s.mode.is_withdrawal_available_for(&caller)` as their first gate:

```rust
// rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs
state::read_state(|s| s.mode.is_withdrawal_available_for(&caller))
    .map_err(RetrieveBtcError::TemporarilyUnavailable)?;
```

The `Mode` enum has four variants — `ReadOnly`, `RestrictedTo(Vec<Principal>)`, `DepositsRestrictedTo(Vec<Principal>)`, and `GeneralAvailability`. In `ReadOnly` mode, **all** withdrawals are rejected. In `RestrictedTo` mode, withdrawals are rejected for any principal not on the allow-list. In both cases, `get_minter_info` still returns a positive `retrieve_btc_min_amount`, implying withdrawals are possible when they are not.

The `MinterInfo` type returned by `get_minter_info` does not include a `mode` field, so callers have no way to infer the restriction from this endpoint alone.

### Impact Explanation
Any user or wallet/dApp that calls `get_minter_info` to determine whether a withdrawal is feasible will receive a misleading answer. They will:
1. Observe a valid `retrieve_btc_min_amount` (e.g., 100,000 satoshi).
2. Prepare a `retrieve_btc` or `retrieve_btc_with_approval` call with a sufficient amount.
3. Have the call rejected with `TemporarilyUnavailable` because the minter is in `ReadOnly` or `RestrictedTo` mode.

This causes failed transactions and wasted cycles/fees. For `retrieve_btc` (the non-approval flow), the user must first transfer ckBTC to the withdrawal account before calling `retrieve_btc`; a failed call means their funds are temporarily locked in the withdrawal account until they retry or transfer back. The discrepancy between the query result and actual behavior is a chain-fusion ledger conservation concern: users may burn ckBTC on the ledger (in the `retrieve_btc_with_approval` flow) only to have the minter reject the withdrawal, triggering a reimbursement flow with associated fees.

### Likelihood Explanation
The `ReadOnly` and `RestrictedTo` modes are explicitly designed for use during minter upgrades and migrations, which are routine operational events on the IC. Any user querying `get_minter_info` during such a window — a realistic scenario for any active ckBTC user or integrating dApp — will receive inaccurate information. The entry path requires no special privileges: `get_minter_info` is a public query callable by any anonymous or authenticated principal.

### Recommendation
The `get_minter_info` function should either:
1. Include the current `mode` field in the returned `MinterInfo` struct so callers can determine availability themselves, or
2. Return a sentinel value (e.g., `retrieve_btc_min_amount = u64::MAX`) when the minter mode prohibits withdrawals for the general public (`ReadOnly` or `RestrictedTo`).

The same pattern applies to `deposit_btc_min_amount` when the mode is `ReadOnly` or `DepositsRestrictedTo`.

pub enum Mode {
    /// Minter's state is read-only.
    ReadOnly,
    /// Only the specified principals can modify the minter's state.
    RestrictedTo(Vec<Principal>),
    /// Only the specified principals can deposit BTC.
    DepositsRestrictedTo(Vec<Principal>),
    #[default]
    /// No restrictions on the minter interactions.
    GeneralAvailability,
}

impl Mode {
    /// Returns Ok if the specified principal can convert BTC to ckBTC.
    pub fn is_deposit_available_for(&self, p: &Principal) -> Result<(), String> {
        match self {
            Self::GeneralAvailability => Ok(()),
            Self::ReadOnly => Err("the minter is in read-only mode".to_string()),
            Self::RestrictedTo(allow_list) => {
                if !allow_list.contains(p) {
                    return Err("access to the minter is temporarily restricted".to_string());
                }
                Ok(())
            }
            Self::DepositsRestrictedTo(allow_list) => {
                if !allow_list.contains(p) {
                    return Err("BTC deposits are temporarily restricted".to_string());
                }
                Ok(())
            }
        }
    }

    /// Returns Ok if the specified principal can convert ckBTC to BTC.
    pub fn is_withdrawal_available_for(&self, p: &Principal) -> Result<(), String> {
        match self {
            Self::GeneralAvailability | Self::DepositsRestrictedTo(_) => Ok(()),
            Self::ReadOnly => Err("the minter is in read-only mode".to_string()),
            Self::RestrictedTo(allow_list) => {
                if !allow_list.contains(p) {
                    return Err("BTC withdrawals are temporarily restricted".to_string());
                }
                Ok(())
            }
        }
    }

#[test]
fn test_upgrade_read_only() {
    let env = new_state_machine();
    let ledger_id = install_ledger(&env);
    let minter_id = install_minter(&env, ledger_id);

    let authorized_principal =
        Principal::from_str("k2t6j-2nvnp-4zjm3-25dtz-6xhaa-c7boj-5gayf-oj3xs-i43lp-teztq-6ae")
            .unwrap();

    // upgrade
    let upgrade_args = UpgradeArgs {
        mode: Some(Mode::ReadOnly),
        ..Default::default()
    };
    let minter_arg = MinterArg::Upgrade(Some(upgrade_args));
    env.upgrade_canister(minter_id, minter_wasm(), Encode!(&minter_arg).unwrap())
        .expect("Failed to upgrade the minter canister");

    // when the mode is ReadOnly then the minter should reject all update calls.

    // 1. update_balance
    let update_balance_args = UpdateBalanceArgs {
        owner: None,
        subaccount: None,
    };
    let res = env
        .execute_ingress_as(
            authorized_principal.into(),
            minter_id,
            "update_balance",
            Encode!(&update_balance_args).unwrap(),
        )
        .expect("Failed to call update_balance");
    let res = Decode!(&res.bytes(), Result<Vec<UtxoStatus>, UpdateBalanceError>).unwrap();
    assert!(
        matches!(res, Err(UpdateBalanceError::TemporarilyUnavailable(_))),
        "unexpected result: {res:?}"
    );

    // 2. retrieve_btc
    let retrieve_btc_args = RetrieveBtcArgs {
        amount: 10,
        address: "".into(),
    };
    let res = env
        .execute_ingress_as(
            authorized_principal.into(),
            minter_id,
            "retrieve_btc",
            Encode!(&retrieve_btc_args).unwrap(),
        )
        .expect("Failed to call retrieve_btc");
    let res = Decode!(&res.bytes(), Result<RetrieveBtcOk, RetrieveBtcError>).unwrap();
    assert!(
        matches!(res, Err(RetrieveBtcError::TemporarilyUnavailable(_))),
        "unexpected result: {res:?}"
    );
}
