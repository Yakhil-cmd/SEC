### Title
Unbacked ERC-20 Minting via Unchecked `ft_on_transfer` Amount — (`engine/src/engine.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

`ft_on_transfer` in Aurora Engine unconditionally trusts the caller-supplied `amount` field when minting ERC-20 mirror tokens. Because `deploy_erc20_token` is permissionless and `ft_on_transfer` has no caller authentication for the ERC-20 path, any NEAR account can register itself as a NEP-141, then call `ft_on_transfer` directly to mint an arbitrary quantity of its corresponding ERC-20 on Aurora without depositing a single token. The same root cause also means that a fee-on-transfer NEP-141 token will cause Aurora to mint more ERC-20 than NEP-141 tokens actually received, leading to bridge insolvency.

---

### Finding Description

**Root cause — `receive_erc20_tokens` trusts `args.amount` without balance verification**

In `engine/src/engine.rs`, `receive_erc20_tokens` reads the amount directly from the callback argument and passes it straight to the ERC-20 `mint` call: [1](#0-0) 

No before/after NEP-141 balance snapshot is taken. The engine never checks whether Aurora's actual NEP-141 balance increased by `amount`.

The same pattern exists in `receive_base_tokens` for the ETH path, though that path is gated by a connector-account check: [2](#0-1) 

**Entry point — `ft_on_transfer` is a public, unauthenticated NEAR method**

`ft_on_transfer` is exported with no access control: [3](#0-2) 

Inside the handler, the only branching is whether `predecessor_account_id` equals the ETH connector. For every other caller, `receive_erc20_tokens` is invoked with `token = predecessor_account_id`: [4](#0-3) 

**Permissionless ERC-20 registration amplifies the attack**

`deploy_erc20_token` requires only `require_running` — no owner or whitelist check: [5](#0-4) 

Any NEAR account can register itself as a NEP-141 and obtain a corresponding ERC-20 address on Aurora.

**Combined attack path**

1. Attacker creates NEAR account `attacker.near`.
2. Attacker calls `deploy_erc20_token` on Aurora with `nep141 = "attacker.near"` → ERC-20 deployed and registered.
3. Attacker calls `ft_on_transfer` on Aurora **directly** (not via `ft_transfer_call`) with `{"sender_id":"attacker.near","amount":"1000000000","msg":"<attacker_evm_address>"}`.
4. `predecessor_account_id = "attacker.near"` → `receive_erc20_tokens` is called → `get_erc20_from_nep141` succeeds → `mint(attacker_evm_address, 1_000_000_000)` is executed.
5. Attacker holds 1 billion ERC-20 tokens backed by zero NEP-141 tokens.

**Fee-on-transfer variant (no direct call needed)**

A non-standard NEP-141 that charges a transfer fee calls `ft_on_transfer` with the pre-fee `amount` while only crediting Aurora with `amount - fee` tokens. Aurora mints the full `amount`, creating an unbacked surplus. Repeated deposits inflate the ERC-20 supply beyond the NEP-141 reserve, making the bridge insolvent for the last withdrawers. [6](#0-5) 

---

### Impact Explanation

- **Direct attack**: Attacker mints arbitrary ERC-20 tokens on Aurora with zero NEP-141 collateral. If those tokens are accepted by any DeFi protocol on Aurora (AMM, lending, etc.), real user funds can be drained.
- **Fee-on-transfer attack**: Bridge becomes insolvent — ERC-20 total supply exceeds NEP-141 reserve. The last users to exit cannot redeem their tokens. This is a permanent partial freeze of funds proportional to accumulated fees.

Both paths fall within the allowed impact scope: **insolvency** and **permanent freezing of funds**.

---

### Likelihood Explanation

- The direct minting attack requires only two permissionless transactions (`deploy_erc20_token` + direct `ft_on_transfer` call). No privileged access, no leaked keys, no governance capture.
- The fee-on-transfer attack requires only that a non-standard NEP-141 token be bridged to Aurora. `deploy_erc20_token` is permissionless, so any actor can bridge any NEP-141.
- Both attack paths are reachable by an unprivileged EVM user or NEAR account holder.

---

### Recommendation

Add a before/after NEP-141 balance check in `receive_erc20_tokens` (and `receive_base_tokens`), analogous to the router path in the original LSSVM report:

```rust
// Before minting, snapshot Aurora's NEP-141 balance via a cross-contract view call
// or require the caller to be the registered NEP-141 contract itself.
let before_balance = nep141_balance_of(aurora_account_id, token);
// ... transfer already happened via ft_transfer_call ...
let after_balance = nep141_balance_of(aurora_account_id, token);
let actual_received = after_balance - before_balance;
require!(actual_received == args.amount, "NEP-141 not transferred in");
```

Alternatively, restrict `ft_on_transfer` so that it can only be triggered as a callback from a registered NEP-141 contract (i.e., verify `predecessor_account_id` is a known, registered NEP-141 before minting), preventing direct calls from arbitrary accounts.

---

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
