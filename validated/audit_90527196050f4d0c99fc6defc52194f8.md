### Title
Missing Rollback of `new_settlement_layer_chain_id_storage` in `finish_io_frame` Causes Block-Finalization Panic — (`basic_system/src/system_implementation/system/io_subsystem.rs`)

---

### Summary

`FullIOStateSnapshot` omits a snapshot slot for `new_settlement_layer_chain_id_storage`, so `finish_io_frame` never rolls it back when a frame reverts. If any transaction causes the `SettlementLayerChainIdUpdated` event to be emitted and is subsequently reverted, the storage slot in `SystemContext` is rolled back while `new_settlement_layer_chain_id_storage` retains the stale new value. At block finalization, `read_batch_context_inputs` compares the two values with a bare `assert_eq!`, which panics, aborting the entire block.

---

### Finding Description

`FullIO` holds six sub-storages. Five of them are snapshotted and rolled back correctly; `new_settlement_layer_chain_id_storage` is not.

**`FullIOStateSnapshot` has no field for it:** [1](#0-0) 

**`finish_io_frame` rolls back five storages but silently skips `new_settlement_layer_chain_id_storage`:** [2](#0-1) 

`new_settlement_layer_chain_id_storage` does expose `start_frame` / `finish_frame` methods: [3](#0-2) 

…but they are never wired into the IO frame lifecycle.

The value is written by the `system_context_event_hook` whenever the SystemContext contract (0x800b) emits `SettlementLayerChainIdUpdated`: [4](#0-3) 

At block finalization, `read_batch_context_inputs` reads the chain ID from persistent storage **and** from `new_settlement_layer_chain_id_storage`, then asserts they are equal: [5](#0-4) 

If a transaction emits the event (writing `new_settlement_layer_chain_id_storage`) and is then reverted:

- `events_storage` is rolled back (event removed). [6](#0-5) 
- `storage` (the persistent slot in SystemContext) is rolled back. [7](#0-6) 
- `new_settlement_layer_chain_id_storage` is **not** rolled back — it retains the new chain ID.

The `assert_eq!` at finalization then compares the old on-chain value against the stale new value and panics.

---

### Impact Explanation

A panic inside `read_batch_context_inputs` propagates up through the block-finalization path, aborting the entire block. In forward (sequencer) mode this crashes the sequencer process; in proving mode it makes the block unprovable. Either outcome is a complete denial-of-service for the chain for that block. Because `FullIO` is re-created per block, the corruption does not persist across blocks, but the attacker can repeat the attack every block.

---

### Likelihood Explanation

The trigger requires a transaction that:
1. Calls the SystemContext contract (0x800b) in a way that causes it to emit `SettlementLayerChainIdUpdated`.
2. Is subsequently reverted (e.g., the outer call frame reverts after the inner SystemContext call succeeds).

If the `setSettlementLayerChainId` function in the SystemContext EVM bytecode is access-controlled to governance/service transactions only, an unprivileged attacker cannot directly trigger it. However:

- A legitimate service transaction that updates the chain ID and is then reverted (e.g., out-of-gas in a subsequent step) would trigger the same panic.
- The `new_settlement_layer_chain_id_storage.update` function also returns an `InternalError` if called a second time within the same block, which would surface as a fatal error rather than a clean revert, compounding the issue.

The likelihood is **medium**: the bug is deterministically triggered by any reverted frame that emits the event, regardless of whether the caller is privileged.

---

### Recommendation

1. Add a `new_settlement_layer_chain_id: NewSettlementLayerChainIdSnapshotId` field to `FullIOStateSnapshot`.
2. Call `self.new_settlement_layer_chain_id_storage.start_frame()` inside `start_io_frame` and store the result.
3. Call `self.new_settlement_layer_chain_id_storage.finish_frame(rollback_handle.map(|x| x.new_settlement_layer_chain_id))` inside `finish_io_frame`.
4. Replace the bare `assert_eq!` in `read_batch_context_inputs` with a proper `Result`-returning check so that a mismatch produces a recoverable `InternalError` rather than a panic.

---

### Proof of Concept

```
Block containing two transactions:

Tx 1 (attacker-controlled):
  CALL SystemContext(0x800b) → setSettlementLayerChainId(new_id)
    → SystemContext emits SettlementLayerChainIdUpdated(new_id)
    → event_hook fires → new_settlement_layer_chain_id_storage = new_id
    → SystemContext storage slot 0 = new_id
  Outer call REVERTS
    → events_storage rolled back (event removed)
    → SystemContext storage slot 0 rolled back to old_id
    → new_settlement_layer_chain_id_storage NOT rolled back → still = new_id

Block finalization:
  read_batch_context_inputs():
    settlement_layer_chain_id = storage_read(SystemContext, slot 0) = old_id
    new_settlement_layer_chain_id_storage.value() = Some(new_id)
    assert_eq!(new_id, old_id)  ← PANIC → block aborted
