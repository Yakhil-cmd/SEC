### Title
Missing Reward Settlement Prior to `VotingRewardsParameters` Update Leads to Incorrect Reward Computation in SNS Governance - (File: rs/sns/governance/src/governance.rs)

### Summary

When an SNS governance proposal changes `VotingRewardsParameters` (specifically `initial_reward_rate_basis_points`, `final_reward_rate_basis_points`, or `reward_rate_transition_duration_seconds`) via `ManageNervousSystemParameters`, the pending reward rounds that elapsed under the old parameters are not settled first. The next call to `distribute_rewards` retroactively applies the new rate to all unsettled rounds, causing neuron holders to receive incorrect maturity for the period before the parameter change.

### Finding Description

`perform_manage_nervous_system_parameters` directly overwrites `self.proto.parameters` with the new parameters without first triggering a reward distribution:

```rust
// rs/sns/governance/src/governance.rs
fn perform_manage_nervous_system_parameters(
    &mut self,
    proposed_params: NervousSystemParameters,
) -> Result<(), GovernanceError> {
    let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());
    match new_params.validate() {
        Ok(()) => {
            self.proto.parameters = Some(new_params); // ← rate overwritten, no prior settlement
            Ok(())
        }
        ...
    }
}

The subsequent `distribute_rewards` call reads the **current** (post-update) `voting_rewards_parameters` and uses them to compute rewards for **all** pending rounds, including those that elapsed before the change:

### Impact Explanation

**Scenario**: An SNS is 6 months old. `initial_reward_rate_basis_points = 200` (2%), `final_reward_rate_basis_points = 100` (1%). No reward distribution has occurred for 5 days (e.g., no settled proposals). A governance proposal passes that sets `initial_reward_rate_basis_points = 100` (matching the final rate, effectively disabling the initial bonus). The next `distribute_rewards` call computes all 5 pending days at the new flat 1% rate instead of the correct ~1.8% rate that was in effect during those 5 days. Neuron holders receive approximately 10% less maturity than they earned. The reverse is also true: a rate increase retroactively inflates rewards for the unsettled period, minting excess maturity.

The `round_duration_seconds` sub-case is also affected: if the round duration increases (e.g., from 1 day to 7 days) while 3 days of rewards are pending, `new_rounds_count = 3 / 7 = 0`, causing `distribute_rewards` to return early and those 3 days' rewards to be computed later under the new (longer) round duration, distorting the per-round reward amount. [5](#0-4) 

### Likelihood Explanation

Triggering this requires a `ManageNervousSystemParameters` proposal to pass, which requires an SNS governance majority. This is a normal governance operation — not a "malicious" majority — and is explicitly supported by the `Action::ManageNervousSystemParameters` proposal type. The proto documentation even notes that `voting_rewards_parameters` "probably should not be changed, because the results would probably be pretty confusing," implicitly acknowledging the transition problem without fixing it. SNS DAOs that adjust reward schedules over time (a common lifecycle event) will trigger this silently. [6](#0-5) 

### Recommendation

Before applying new `NervousSystemParameters` that include a changed `voting_rewards_parameters`, call `distribute_rewards` (or an equivalent settlement step) to settle all pending reward rounds under the old parameters. This ensures the old rate is used for the period it was in effect, and the new rate only applies to rounds that begin after the parameter change takes effect.

### Proof of Concept

1. Deploy an SNS with `initial_reward_rate_basis_points = 200`, `final_reward_rate_basis_points = 100`, `round_duration_seconds = 86400` (1 day).
2. Advance time by 5 days without triggering `distribute_rewards` (e.g., no settled proposals).
3. Submit and pass a `ManageNervousSystemParameters` proposal setting `initial_reward_rate_basis_points = 100` (equal to final, disabling the initial bonus).
4. Trigger `distribute_rewards` (advance time by 1 more day to ensure `new_rounds_count > 0`).
5. Observe that the 5 pending days are computed at 1% instead of ~1.8%, and neuron maturity is lower than it should be for those days.

The execution path is: `perform_action` → `perform_manage_nervous_system_parameters` (overwrites params) → later `run_periodic_tasks` → `distribute_rewards` (uses new params for old rounds). [7](#0-6) [8](#0-7)

    pub fn reward_rate_at(&self, now: Instant) -> RewardRate {
        let reward_rate_transition_duration_seconds = self
            .reward_rate_transition_duration_seconds
            .expect("reward_rate_transition_duration_seconds unset");

        let time_since_genesis = {
            let result = now - *GENESIS;
            // For the purposes of determining reward rate, treat times before
            // genesis the same as at genesis. This is not expected to occur in
            // practice. This code is just being extra defensive.
            if result.as_secs() < i2d(0) {
                Duration { days: i2d(0) }
            } else {
                result
            }
        };
        if reward_rate_transition_duration_seconds == 0
            || time_since_genesis.as_secs() >= i2d(reward_rate_transition_duration_seconds)
        {
            return self.final_reward_rate();
        }

        // s linearly varies from 1 -> 0 as seconds_since_genesis varies from 0
        // to reward_rate_transition_duration_seconds.
        let transition = LinearMap::new(
            dec!(0)..i2d(reward_rate_transition_duration_seconds),
            dec!(1)..dec!(0),
        );
        let s = transition.apply(time_since_genesis.as_secs());
        // s2 varies quadratically from 1 -> 0 (again, as seconds_since_genesis
        // varies from 0 to reward_rate_transition_duration_seconds), and
        // flattens out as seconds_since_genesis approaches
        // reward_rate_transition_duration_seconds.
        let s2 = s * s;

        // This looks backwards, but we think of variable rate as being added to
        // final growth rate, not initial, and the amount to add is up to
        // initial - final (where initial is thought of as being greater than
        // final).
        let dr = self.initial_reward_rate() - self.final_reward_rate();
        // variable_reward_rate varies from dr to 0 as round varies from
        // 1 to transition_round_count.
        let variable_reward_rate = s2 * dr;

        self.final_reward_rate() + variable_reward_rate

#[derive(Default, candid::CandidType, candid::Deserialize, Debug, Clone, Copy, PartialEq)]
pub struct VotingRewardsParameters {
    /// The amount of time between reward events.
    ///
    /// Must be > 0.
    ///
    /// During such periods, proposals enter the ReadyToSettle state. Once the round is over, voting
    /// for those proposals entitle voters to voting rewards. Such rewards are calculated in
    /// the governance canister's run_periodic_tasks function.
    ///
    /// This is a nominal amount. That is, the actual time between reward
    /// calculations and distribution cannot be guaranteed to be perfectly
    /// periodic, but actual inter-reward periods are generally expected to be
    /// within a few seconds of this.
    ///
    /// This supersedes super.reward_distribution_period_seconds.
    pub round_duration_seconds: Option<u64>,
    /// The amount of time that the growth rate changes (presumably, decreases)
    /// from the initial growth rate to the final growth rate. (See the two
    /// *_reward_rate_basis_points fields bellow.) The transition is quadratic, and
    /// levels out at the end of the growth rate transition period.
    pub reward_rate_transition_duration_seconds: Option<u64>,
    /// The amount of rewards is proportional to token_supply * current_rate. In
    /// turn, current_rate is somewhere between `initial_reward_rate_basis_points`
    /// and `final_reward_rate_basis_points`. In the first reward period, it is the
    /// initial growth rate, and after the growth rate transition period has elapsed,
    /// the growth rate becomes the final growth rate, and remains at that value for
    /// the rest of time. The transition between the initial and final growth rates is
    /// quadratic, and levels out at the end of the growth rate transition period.
    ///
    /// (A basis point is one in ten thousand.)
    pub initial_reward_rate_basis_points: Option<u64>,
    pub final_reward_rate_basis_points: Option<u64>,
}
