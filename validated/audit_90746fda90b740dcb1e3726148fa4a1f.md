Audit Report

## Title
Unbounded `ingress_expiries` Vec Allocation via Attacker-Controlled `ingress_end` in `construction_payloads` — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary
The `construction_payloads` function in both the ICRC1 and ICP Rosetta servers accepts attacker-controlled `ingress_start` and `ingress_end` values from the HTTP request body and enters an unbounded `while` loop that pushes one `u64` per iteration into a `Vec`. No guard caps the window size. Supplying `ingress_start=0` and `ingress_end=u64::MAX` causes the server to attempt allocating hundreds of gigabytes of memory, triggering OOM termination or, due to u64 wrapping in release mode, a permanent infinite loop that hangs the handler task and denies all service.

## Finding Description
**Entry point (ICRC1):** `POST /construction/payloads` → `endpoints::construction_payloads` (L90–107) passes raw metadata fields directly to `services::construction_payloads` with no pre-validation of window size. [1](#0-0) 

Inside `services::construction_payloads`, two guards exist but neither bounds the window: [2](#0-1) 

- Guard 1 (`ingress_start >= ingress_end`): With `ingress_start=0, ingress_end=u64::MAX`, `0 >= u64::MAX` is false — passes.
- Guard 2 (`ingress_end < now + ingress_interval`): With `now ≈ 1.75×10¹⁸ ns` (current epoch time) and `ingress_interval = 240_000_000_000 ns`, `u64::MAX (≈1.844×10¹⁹) < ~1.75×10¹⁸` is false — passes.

Execution then falls into the unbounded loop: [3](#0-2) 

**Step calculation:** `ingress_interval = (MAX_INGRESS_TTL − PERMITTED_DRIFT).as_nanos() = (300s − 60s) = 240_000_000_000 ns`. `INGRESS_INTERVAL_OVERLAP = 120_000_000_000 ns`. Effective step = `240_000_000_000 − 120_000_000_000 = 120_000_000_000 ns`.

**Iteration count:** `u64::MAX / 120_000_000_000 ≈ 153,722,867,280` iterations × 8 bytes = ~1.2 TB. The server OOMs long before completion. Additionally, because `ingress_start += step` uses wrapping arithmetic in Rust release mode, the wrapped value falls below `u64::MAX` again, converting the OOM scenario into a **permanent infinite loop** on the handler task if the allocator does not kill the process first.

The identical pattern exists in the ICP Rosetta server with no guards whatsoever: [4](#0-3) 

## Impact Explanation
A single unauthenticated HTTP POST to `/construction/payloads` causes the Rosetta server process to exhaust all available heap memory and be killed by the OS OOM killer, or to spin indefinitely on the handler task. All concurrent and subsequent requests are denied service. This constitutes an **application/platform-level DoS** of the Rosetta API — a financial integration component — with concrete harm to exchange integrations relying on the Rosetta construction API for ICP and ICRC1 token transfer signing. This matches the **High ($2,000–$10,000)** impact class: "Application/platform-level DoS… or subnet availability impact not based on raw volumetric DDoS" and "Significant… Rosetta… infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
The endpoint is publicly reachable with no authentication. The payload is a standard JSON body. The attack requires a single HTTP request and is trivially reproducible by any external party. No privileged access, key material, or protocol-level knowledge is required beyond knowing the network identifier. The attack is repeatable: if the process is restarted, a single follow-up request re-triggers the condition.

## Recommendation
Add an explicit upper bound on the ingress window before entering the loop in both:
- `rs/rosetta-api/icrc1/src/construction_api/services.rs` (after line 158)
- `rs/rosetta-api/icp/src/request_handler/construction_payloads.rs` (before line 99)

Example for the ICRC1 server:
```rust
const MAX_INGRESS_WINDOW_ENTRIES: usize = 100; // ~200 minutes at 120s step
let step = ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
if step == 0 || ingress_end.saturating_sub(ingress_start) / step > MAX_INGRESS_WINDOW_ENTRIES as u64 {
    return Err(Error::processing_construction_failed(
        "Ingress window exceeds maximum allowed size"
    ));
}
```
Apply equivalent logic to the ICP Rosetta server using its `interval` variable.

## Proof of Concept
```bash
curl -X POST http://<rosetta-host>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"<network_id>"},
    "operations": [<valid_transfer_op>],
    "public_keys": [<valid_pubkey>],
    "metadata": {
      "ingress_start": 0,
      "ingress_end": 18446744073709551615
    }
  }'
```
Expected result: server process is killed by OOM or the handler task spins indefinitely; all subsequent requests to the Rosetta server time out or are refused. A deterministic unit test can confirm the behavior by calling `services::construction_payloads` directly with `ingress_start=0, ingress_end=u64::MAX` and asserting it returns an error rather than hanging.
