### Title
`gas_per_pubdata_limit` User-Specified Cap Silently Ignored in L2 ZK Transaction Validation — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

ZKsync OS's ZK-type L2 transaction format includes a user-signed `gasPerPubdataByteLimit` field that is intended to cap the pubdata price a user consents to pay. However, the bootloader's L2 transaction validation path reads `pubdata_price` and `native_price` exclusively from the operator-controlled block context and never validates them against the user's `gas_per_pubdata_limit`. The field is explicitly annotated `#[allow(dead_code)]` in the parsed transaction struct, confirming it is never consumed. This is a direct analog to the Teller M-7 finding: a user-specified protection parameter is bypassed because execution reads from a mutable external source (the block context oracle) instead of enforcing the user's signed bound.

---

### Finding Description

The `AbiEncodedTransaction` struct parses `gas_per_pubdata_limit` from the ZK transaction wire format but marks it dead:

```rust
/// The maximum amount of gas the user is willing to pay for a byte of pubdata.
#[allow(dead_code)]
pub gas_per_pubdata_limit: ParsedValue<u32>,
``` [1](#0-0) 

The `Transaction` wrapper exposes a `gas_per_pubdata_limit()` accessor, but it is never called during L2 ZK transaction validation: [2](#0-1) 

Inside `validate_and_compute_fee_for_transaction`, the effective pubdata cost is derived entirely from the operator-supplied block context:

```rust
let pubdata_price = system.get_pubdata_price();   // operator-controlled
let native_price  = system.get_native_price();    // operator-controlled
...
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [3](#0-2) 

The `native_per_pubdata` derived here is then used to charge the user for all pubdata generated during execution: [4](#0-3) 

By contrast, L1→L2 transactions correctly read `gas_per_pubdata` from the transaction itself, not from the block context:

```rust
// For L1->L2 transactions we always use the pubdata price provided by the transaction.
let gas_per_pubdata = transaction.gas_per_pubdata_limit.read();
``` [5](#0-4) 

The block context fields `pubdata_price` and `native_price` are explicitly documented as sequencer-controlled: [6](#0-5) 

---

### Impact Explanation

An operator who sets a high `pubdata_price` in the block context causes every L2 ZK transaction to be charged pubdata at that elevated rate, regardless of the `gasPerPubdataByteLimit` the user signed. Two concrete outcomes:

1. **Post-execution revert with full gas burn**: If the pubdata generated during execution exceeds what the user's gas budget can cover at the inflated rate, the transaction reverts after execution and the full `gas_limit` is consumed. The user loses all prepaid gas with no state change. This is confirmed by the existing test `test_l2_tx_not_enough_native_for_pubdata_uses_full_gas_limit`.

2. **Silent overcharge**: If the user's gas budget is large enough to absorb the inflated pubdata cost, the transaction succeeds but the user is charged at a pubdata rate they never consented to — potentially many multiples of their `gasPerPubdataByteLimit`.

In both cases the user suffers a direct financial loss (gas/ETH) that they explicitly tried to bound with a signed field that the protocol ignores.

---

### Likelihood Explanation

The operator controls `pubdata_price` and `native_price` in every block context. No on-chain constraint caps these values. The missing check is a single absent comparison in the validation path. Any operator who sets `pubdata_price` above the level implied by users' `gasPerPubdataByteLimit` values triggers the impact. The `#[allow(dead_code)]` annotation confirms this is not an accidental omission in a feature branch — the field is structurally present but deliberately unused in the current validation logic.

---

### Recommendation

In `validate_and_compute_fee_for_transaction`, after computing the effective `native_per_pubdata` from the block context, compare it against the user's signed `gas_per_pubdata_limit`. Reject the transaction (validation error) if the block's effective pubdata price exceeds the user's stated maximum, mirroring the protection already in place for `max_fee_per_gas` vs. `base_fee`:

```rust
// After computing native_per_pubdata from block context:
let user_gas_per_pubdata = transaction.gas_per_pubdata_limit();
if !user_gas_per_pubdata.is_zero() {
    // Convert block-level native_per_pubdata back to gas units and compare
    let effective_gas_per_pubdata = /* native_per_pubdata * native_price / gas_price */;
    require!(
        effective_gas_per_pubdata <= user_gas_per_pubdata,
        InvalidTransaction::PubdataPriceTooHigh,
        system
    )?;
}
```

This is exactly the pattern used for L1→L2 transactions and for `max_fee_per_gas` enforcement.

---

### Proof of Concept

1. User signs a ZK L2 transaction with `gasPerPubdataByteLimit = 800` (standard value) and `max_fee_per_gas = 1000`.
2. Operator sets `pubdata_price = 1_000_000` and `native_price = 1` in the block context, making `native_per_pubdata = 1_000_000`.
3. Bootloader calls `validate_and_compute_fee_for_transaction`:
   - Line 106: `pubdata_price = 1_000_000` (from block context)
   - Line 142: `native_per_pubdata = 1_000_000`
   - User's `gas_per_pubdata_limit = 800` is **never read**
4. Transaction executes; any storage write generates pubdata charged at `1_000_000` native/byte.
5. Post-execution pubdata check (`check_enough_resources_for_pubdata`) exhausts the user's resources; transaction reverts and full `gas_limit` is burned.
6. User loses all prepaid gas despite having specified `gasPerPubdataByteLimit = 800` to prevent exactly this outcome.

The `#[allow(dead_code)]` annotation at `basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs:50` and the absence of any call to `transaction.gas_per_pubdata_limit()` in `validate_and_compute_fee_for_transaction` are the definitive code evidence.

pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
