Audit Report

## Title
SNS Governance Reward Inflation via `end_timestamp_seconds` Defaulting to Unix Epoch - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance canister's `distribute_rewards` and `should_distribute_rewards` functions use `.unwrap_or_default()` on the `optional` protobuf field `end_timestamp_seconds` of the stored `latest_reward_event`. Any SNS canister whose persisted `RewardEvent` predates the introduction of this field will have `end_timestamp_seconds = None`, causing `unwrap_or_default()` to silently return `0` (Unix epoch). This makes `distribute_rewards` calculate and distribute rewards for approximately 19,675 rounds in a single invocation, massively inflating neuron maturity and causing illegal token minting when neurons disburse.

## Finding Description

**Root cause — `unwrap_or_default()` on an optional field that can legitimately be `None`:**

In `should_distribute_rewards` (lines 5735–5738), the period boundary is read as:

```rust
let seconds_since_last_reward_event = now.saturating_sub(
    self.latest_reward_event()
        .end_timestamp_seconds
        .unwrap_or_default(),   // returns 0 if None
);
``` [1](#0-0) 

In `distribute_rewards` (lines 5808–5814), the same pattern determines how many rounds to accumulate:

```rust
let reward_start_timestamp_seconds = self
    .latest_reward_event()
    .end_timestamp_seconds
    .unwrap_or_default();       // returns 0 if None
let new_rounds_count = now
    .saturating_sub(reward_start_timestamp_seconds)
    .saturating_div(round_duration_seconds);
``` [2](#0-1) 

**Why `end_timestamp_seconds` can be `None`:**

The field is declared `optional` in the protobuf schema:

```proto
optional uint64 end_timestamp_seconds = 5;
``` [3](#0-2) 

The predecessor field `round` is explicitly deprecated in favor of it, confirming `end_timestamp_seconds` was added in a later version:

```proto
// DEPRECATED: Use end_timestamp_seconds instead.
uint64 round = 1;
``` [4](#0-3) 

**The initialization guard does not cover the upgrade scenario:**

`Governance::new` only populates `end_timestamp_seconds` when `latest_reward_event` is entirely absent:

```rust
if proto.latest_reward_event.is_none() {
    proto.latest_reward_event = Some(RewardEvent {
        end_timestamp_seconds: Some(now),
        ...
    })
}
``` [5](#0-4) 

If a canister already has a stored `RewardEvent` from a pre-`end_timestamp_seconds` version, this branch is skipped entirely. There is no post-upgrade migration to backfill the field from the deprecated `round` value.

**The reward loop iterates over all synthetic rounds:**

With `reward_start_timestamp_seconds = 0`, the loop at lines 5861–5872 iterates `new_rounds_count = now / round_duration_seconds` times — approximately 19,675 iterations for a 1-day round at current Unix time — accumulating a full reward purse for each:

```rust
for i in 1..=new_rounds_count {
    ...
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
``` [6](#0-5) 

After this single inflated distribution, `latest_reward_event` is updated with `end_timestamp_seconds: Some(reward_event_end_timestamp_seconds)`, so subsequent rounds are correct — the inflation is a one-time event per affected canister, but it is irreversible. [7](#0-6) 

## Impact Explanation

When neurons call `disburse_maturity` or `spawn_neuron` after receiving the inflated maturity, the SNS ledger mints tokens proportional to the ~19,675× inflated amount. For any SNS with a non-trivial token supply and market value, this constitutes **illegal minting** causing severe and irreversible token supply inflation and protocol insolvency for the affected SNS. This matches the High impact category: *"Significant SNS security impact with concrete user or protocol harm"* and potentially Critical if the affected SNS token supply exceeds $1M in value.

## Likelihood Explanation

The trigger is fully automatic — no attacker action is required beyond holding a neuron in the affected SNS. The canister's periodic timer calls `run_periodic_tasks` → `should_distribute_rewards` → `distribute_rewards` without any user interaction. The only precondition is that the SNS was deployed before `end_timestamp_seconds` was introduced and has since been upgraded. The deprecation of `round` in the proto confirms the field was added after initial SNS deployment, making this a realistic upgrade scenario for early SNS instances.

## Recommendation

1. Replace `unwrap_or_default()` with a fallback to `genesis_timestamp_seconds` (or reconstruct from `round * round_duration_seconds`) when `end_timestamp_seconds` is `None`, so the period boundary is never treated as Unix epoch.
2. Add a post-upgrade migration in `Governance::new` that backfills `end_timestamp_seconds` from the deprecated `round` field and `round_duration_seconds` for any stored `RewardEvent` where `end_timestamp_seconds` is `None`.
3. Add an explicit upper bound on `new_rounds_count` (e.g., cap at a few hundred rounds) with a high-severity log if the cap is hit, to limit blast radius from any future similar issue.

## Proof of Concept

1. Deploy an SNS governance canister using a WASM version that predates `end_timestamp_seconds` in `RewardEvent`. Allow several normal reward rounds to complete; `latest_reward_event.end_timestamp_seconds` is `None` in stable state.
2. Upgrade the canister to the current WASM. `Governance::new` skips the initialization branch at line 726 because `latest_reward_event` is already `Some(...)`.
3. The periodic timer fires and calls `should_distribute_rewards`. With `end_timestamp_seconds = None`, `unwrap_or_default()` returns `0`, so `seconds_since_last_reward_event = now ≈ 1.7e9 s`, which far exceeds any `round_duration_seconds`. Returns `true`.
4. `distribute_rewards` is called. `reward_start_timestamp_seconds = 0`. `new_rounds_count = now / round_duration_seconds ≈ 19,675` for a 1-day round.
5. The loop accumulates rewards for 19,675 rounds. For a 1% annual rate on a 1 billion token supply (1e17 e8s), the purse is ≈ 5.4 × 10¹⁶ e8s instead of the intended ≈ 2.7 × 10¹² e8s — a ~19,675× inflation.
6. All voting neurons receive proportionally inflated maturity. Calling `disburse_maturity` or `spawn_neuron` mints the excess tokens from the SNS ledger.
7. A deterministic integration test using PocketIC can reproduce this by: (a) initializing governance state with a `RewardEvent` where `end_timestamp_seconds = None`, (b) advancing the clock, and (c) asserting that `distribute_rewards` produces `new_rounds_count >> 1` and that neuron maturity is inflated by the corresponding factor.
