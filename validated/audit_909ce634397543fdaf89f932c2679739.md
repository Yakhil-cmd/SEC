### Title
Unrestricted `submit` Caller Allows Frontrunning to Steal EIP-1559 Relayer Rewards — (File: `engine/src/contract_methods/evm_transactions.rs`)

---

### Summary

The `submit` and `submit_with_args` entry points on the Aurora Engine derive the relayer's EVM address directly from `predecessor_account_id()` and pay the EIP-1559 priority-fee tip to that address. Because neither function enforces any restriction on who may call them, any NEAR account can observe a pending `submit` call in the NEAR mempool, rebroadcast the identical Ethereum transaction bytes with a higher NEAR gas price, and claim the relayer reward that was intended for the legitimate relayer.

---

### Finding Description

`submit` and `submit_with_args` are public WASM exports callable by any NEAR account: [1](#0-0) 

Inside `submit`, the relayer address is set unconditionally to whoever is the current NEAR predecessor: [2](#0-1) 

The same pattern applies to `submit_with_args`: [3](#0-2) 

After EVM execution, `refund_unused_gas` credits the EIP-1559 priority fee (tip) to that relayer address: [4](#0-3) 

Specifically, `relayer_reward = gas_used × priority_fee_per_gas` is added to the relayer's on-chain ETH balance. For legacy (pre-EIP-1559) transactions the full `gas_price` is the priority fee, making the reward potentially large.

The only caller restriction that exists is the silo-mode whitelist check inside `assert_access`: [5](#0-4) 

This check is entirely absent in the default (non-silo) deployment, leaving `submit` open to any caller.

---

### Impact Explanation

An attacker who observes a pending NEAR transaction `relayer.near → aurora::submit(eth_tx_bytes)` can immediately rebroadcast `attacker.near → aurora::submit(eth_tx_bytes)` with a marginally higher NEAR gas price. Because NEAR validators order transactions by gas price, the attacker's call is included first. The attacker receives the full EIP-1559 priority-fee tip (or the full gas price for legacy transactions) that was intended for the legitimate relayer. The legitimate relayer's subsequent NEAR transaction reverts because the Ethereum nonce is already consumed, so the relayer also wastes NEAR gas. This constitutes **theft of unclaimed yield** (the relayer fee) from the legitimate relayer.

**Impact: High — Theft of unclaimed yield.**

---

### Likelihood Explanation

The NEAR mempool is observable by validators and by any node connected to the NEAR network. The Ethereum transaction bytes are fully public once submitted. The attack requires no special privilege — only the ability to submit a NEAR transaction. Profitability holds whenever `gas_used × priority_fee_per_gas > NEAR gas cost of the frontrunning call`. For any EIP-1559 transaction with a non-trivial tip, or any legacy transaction with a meaningful gas price, this condition is easily satisfied. Automated bots can monitor the NEAR mempool and frontrun every eligible `submit` call, making this a systemic and repeatable attack.

**Likelihood: High.**

---

### Recommendation

Add an access-control check at the top of `submit` and `submit_with_args` that restricts callers to a set of approved relayer accounts (analogous to the `key_manager` / `add_relayer_key` system already present for function-call access keys). Alternatively, bind the relayer reward to the Ethereum transaction's `from` address rather than to the NEAR predecessor, so that frontrunning the NEAR call yields no financial benefit to the attacker.

---

### Proof of Concept

1. Alice (a legitimate relayer) signs and broadcasts the NEAR transaction:
   `alice.near → aurora.near::submit(<eth_tx_bytes with priority_fee=X>)`

2. Attacker observes this pending NEAR transaction in the mempool and immediately broadcasts:
   `attacker.near → aurora.near::submit(<same eth_tx_bytes>)` with NEAR gas price + 1.

3. Attacker's NEAR transaction is ordered first. Inside `submit`:
   ```
   relayer_address = predecessor_address("attacker.near")  // attacker's EVM address
   ```
   `refund_unused_gas` executes:
   ```
   add_balance(io, &attacker_evm_address, relayer_reward)  // attacker receives tip
   ```

4. Alice's NEAR transaction is processed next. The Ethereum nonce is already consumed; the call reverts. Alice loses NEAR gas and receives no relayer reward.

5. Net result: attacker gains `gas_used × priority_fee_per_gas` ETH on Aurora; Alice loses the same amount plus NEAR gas costs. [6](#0-5) [7](#0-6)

pub fn submit<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let tx_data = io.read_input().to_vec();
        let current_account_id = env.current_account_id();
        let relayer_address = predecessor_address(&env.predecessor_account_id());
        let args = SubmitArgs {
            tx_data,
            ..Default::default()
        };
        let result = engine::submit(
            io,
            env,
            &args,
            state,
            current_account_id,
            relayer_address,
            handler,
        )?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);

        Ok(result)
    })
}

pub fn refund_unused_gas<I: IO>(
    io: &mut I,
    sender: &Address,
    gas_used: u64,
    gas_result: &GasPaymentResult,
    relayer: &Address,
    fixed_gas: Option<EthGas>,
) -> Result<(), GasPaymentError> {
    if gas_result.effective_gas_price.is_zero() {
        return Ok(());
    }

    let (refund, relayer_reward) = {
        let gas_to_wei = |price: U256| {
            fixed_gas
                .map_or_else(|| gas_used.into(), EthGas::as_u256)
                .checked_mul(price)
                .map(Wei::new)
                .ok_or(GasPaymentError::EthAmountOverflow)
        };

        let spent_amount = gas_to_wei(gas_result.effective_gas_price)?;
        let reward_amount = gas_to_wei(gas_result.priority_fee_per_gas)?;

        let refund = gas_result
            .prepaid_amount
            .checked_sub(spent_amount)
            .ok_or(GasPaymentError::EthAmountOverflow)?;

        (refund, reward_amount)
    };

    if !refund.is_zero() {
        add_balance(io, sender, refund)?;
    }

    if !relayer_reward.is_zero() {
        add_balance(io, relayer, relayer_reward)?;
    }

    Ok(())
}

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
