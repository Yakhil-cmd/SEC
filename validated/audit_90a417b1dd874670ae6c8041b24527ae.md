### Title
Router Self-Deletion via Unguarded `DeleteAccount` Batch Action — (`etc/xcc-router/src/lib.rs`)

---

### Summary

When the `all-promise-actions` feature is compiled in, the XCC router's `add_batch_actions` function unconditionally executes every `PromiseAction` variant — including `DeleteAccount` — without checking whether the batch target is the router's own account. An attacker with a valid EVM address and a funded router can schedule a `PromiseBatchAction` targeting the router itself with a `DeleteAccount` action, causing the router to delete its own NEAR account and transfer its entire balance to an attacker-controlled beneficiary.

---

### Finding Description

**Entry point — XCC precompile (`engine-precompiles/src/xcc.rs`)**

The precompile parses `CrossContractCallArgs` from raw EVM input and, for the `Delayed` variant, serialises the inner `PromiseArgs` verbatim and forwards it to the router's `schedule` method. No inspection of the inner promise structure is performed. [1](#0-0) 

**Router `schedule` — only checks caller identity**

`schedule` calls `assert_preconditions()`, which verifies only that `predecessor_account_id == self.parent` (the Aurora engine). Because the engine unconditionally forwards whatever `PromiseArgs` the precompile produces, any structurally valid `PromiseArgs` — including one containing `DeleteAccount` — passes this gate. [2](#0-1) [3](#0-2) 

**`execute_scheduled` — open to any caller**

The comment at line 146 explicitly states this function is intentionally callable by anyone. It removes the stored promise and calls `Self::promise_create`, which dispatches to `recursive_promise_create`. [4](#0-3) 

**`recursive_promise_create` — no target validation**

For `NearPromise::Simple(SimpleNearPromise::Batch(batch))`, the function creates a batch promise targeting `batch.target_account_id` without any check that the target is not `env::current_account_id()`. [5](#0-4) 

**`add_batch_actions` — no action filtering**

Under `#[cfg(feature = "all-promise-actions")]`, every `PromiseAction` variant is dispatched without restriction. The `DeleteAccount` arm calls `env::promise_batch_action_delete_account` directly, with the attacker-supplied `beneficiary_id`. [6](#0-5) 

**Missing invariant:** Neither the precompile, the engine's `handle_precompile_promise`, nor the router itself ever checks:
- whether a batch's `target_account_id` equals the router's own `current_account_id`, or
- whether the action list contains `DeleteAccount` (or `AddFullAccessKey`, which is equally dangerous). [7](#0-6) 

---

### Impact Explanation

1. **Theft of funds**: The router account holds at minimum 2 NEAR for storage staking (`REFUND_AMOUNT`), plus any unwrapped wNEAR the user deposited for XCC calls. `DeleteAccount` transfers the entire account balance to the attacker-controlled `beneficiary_id`.
2. **Permanent freezing**: Once the router is deleted, all future XCC calls from the associated EVM address fail until a new router is deployed. The attacker can repeat the attack each time the router is redeployed, continuously draining the user's NEAR and disrupting XCC functionality. [8](#0-7) 

---

### Likelihood Explanation

- Requires `all-promise-actions` feature to be compiled in — this is a stated precondition and a legitimate production feature enabling NEAR batch transactions.
- Requires the attacker to have a valid EVM address (no special privilege — any Aurora user qualifies).
- The router account must exist and be funded (standard precondition for any XCC use).
- `execute_scheduled` is callable by anyone with no restrictions, so the attacker does not need to be the original scheduler.
- No cryptographic secrets, admin keys, or governance capture required.

---

### Recommendation

In `add_batch_actions`, before dispatching `DeleteAccount` (and `AddFullAccessKey`), assert that the batch target is not the router's own account:

```rust
PromiseAction::DeleteAccount { beneficiary_id } => {
    let beneficiary_id = beneficiary_id.as_ref().parse().unwrap();
    // Guard: never allow self-deletion
    assert_ne!(
        id_target, // the target resolved at batch creation
        env::current_account_id(),
        "ERR_CANNOT_DELETE_ROUTER"
    );
    env::promise_batch_action_delete_account(id, &beneficiary_id)
}
```

More robustly, pass `current_account_id` into `add_batch_actions` and reject any batch whose `target_account_id` equals the router's own account before even creating the batch promise in `recursive_promise_create`. [9](#0-8) 

---

### Proof of Concept

```rust
// In etc/xcc-router/src/tests.rs (unit test, no mainnet required)
#[test]
#[cfg(feature = "all-promise-actions")]
fn test_delete_account_self_rejected() {
    use aurora_engine_types::parameters::{
        NearPromise, PromiseAction, PromiseBatchAction, SimpleNearPromise,
    };

    let (parent, mut contract) = create_contract();
    let router_id = near_sdk::env::current_account_id();

    // Schedule a DeleteAccount batch targeting the router itself
    let malicious_promise = PromiseArgs::Recursive(NearPromise::Simple(
        SimpleNearPromise::Batch(PromiseBatchAction {
            target_account_id: router_id.as_str().parse().unwrap(),
            actions: vec![PromiseAction::DeleteAccount {
                beneficiary_id: "attacker.near".parse().unwrap(),
            }],
        }),
    ));

    contract.schedule(malicious_promise);

    // execute_scheduled is open to anyone
    testing_env!(VMContextBuilder::new()
        .predecessor_account_id(near_sdk::test_utils::test_env::bob())
        .build());

    // This should panic/reject — currently it does NOT, proving the bug
    contract.execute_scheduled(0.into());

    // Assert router account still exists (will fail on unpatched code)
    // In practice the DeleteAccount receipt is emitted and the account is gone
}
```

On unpatched code with `all-promise-actions` enabled, `execute_scheduled` succeeds and emits a `DeleteAccount` receipt targeting the router, draining its balance to `attacker.near`. [10](#0-9) [11](#0-10)

    fn recursive_promise_create(promise: &NearPromise) -> PromiseIndex {
        match promise {
            NearPromise::Simple(x) => match x {
                SimpleNearPromise::Create(call) => Self::base_promise_create(call),
                SimpleNearPromise::Batch(batch) => {
                    let target = batch.target_account_id.as_ref().parse().unwrap();
                    let id = env::promise_batch_create(&target);
                    Self::add_batch_actions(id, &batch.actions);
                    id
                }
            },
            NearPromise::Then { base, callback } => {
                let base_index = Self::recursive_promise_create(base);
                match callback {
                    SimpleNearPromise::Create(call) => env::promise_then(
                        base_index,
                        call.target_account_id.as_ref().parse().unwrap(),
                        call.method.as_str(),
                        &call.args,
                        NearToken::from_yoctonear(call.attached_balance.as_u128()),
                        Gas::from_gas(call.attached_gas.as_u64()),
                    ),
                    SimpleNearPromise::Batch(batch) => {
                        let id = env::promise_batch_then(
                            base_index,
                            &batch.target_account_id.as_ref().parse().unwrap(),
                        );
                        Self::add_batch_actions(id, &batch.actions);
                        id
                    }
                }
            }
            NearPromise::And(promises) => {
                let indices: Vec<PromiseIndex> = promises
                    .iter()
                    .map(Self::recursive_promise_create)
                    .collect();
                env::promise_and(&indices)
            }
        }
    }

    #[cfg(feature = "all-promise-actions")]
    fn add_batch_actions(id: PromiseIndex, actions: &[PromiseAction]) {
        for action in actions.iter() {
            match action {
                PromiseAction::CreateAccount => env::promise_batch_action_create_account(id),
                PromiseAction::Transfer { amount } => env::promise_batch_action_transfer(
                    id,
                    NearToken::from_yoctonear(amount.as_u128()),
                ),
                PromiseAction::DeployContract { code } => {
                    env::promise_batch_action_deploy_contract(id, code)
                }
                PromiseAction::FunctionCall {
                    name,
                    args,
                    attached_yocto,
                    gas,
                } => env::promise_batch_action_function_call(
                    id,
                    name,
                    args,
                    NearToken::from_yoctonear(attached_yocto.as_u128()),
                    Gas::from_gas(gas.as_u64()),
                ),
                PromiseAction::Stake { amount, public_key } => env::promise_batch_action_stake(
                    id,
                    NearToken::from_yoctonear(amount.as_u128()),
                    &to_sdk_pk(public_key),
                ),
                PromiseAction::AddFullAccessKey { public_key, nonce } => {
                    env::promise_batch_action_add_key_with_full_access(
                        id,
                        &to_sdk_pk(public_key),
                        *nonce,
                    )
                }
                PromiseAction::AddFunctionCallKey {
                    public_key,

                    allowance,
                    receiver_id,
                    function_names,

                    env::promise_batch_action_add_key_allowance_with_function_call(

                        &receiver_id,
                        function_names,

                    env::promise_batch_action_delete_key(id, &to_sdk_pk(public_key))

                    let beneficiary_id = beneficiary_id.as_ref().parse().unwrap();
                    env::promise_batch_action_delete_account(id, &beneficiary_id)
