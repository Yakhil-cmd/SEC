### Title
XCC Precompile Burns User wNEAR Before NEAR Promise Executes With No Target Account Existence Check, Causing Fund Freeze on Promise Failure - (File: `engine-precompiles/src/xcc.rs`, `etc/xcc-router/src/lib.rs`)

---

### Summary

The Aurora XCC precompile irrevocably burns a user's wNEAR from their EVM balance during EVM execution, before the corresponding NEAR cross-contract promise is dispatched. Neither the precompile nor the XCC router validates that the target NEAR account exists. If the target account is absent or deleted between EVM execution and asynchronous promise dispatch, the promise fails silently, the NEAR is refunded only to the router sub-account, and the user's wNEAR is permanently unrecoverable without admin intervention.

---

### Finding Description

**Step 1 – wNEAR burned unconditionally during EVM execution.**

In `engine-precompiles/src/xcc.rs`, the precompile computes `required_near` and immediately executes an EVM-level `transferFrom` call that moves the user's wNEAR to the engine's implicit address. This happens synchronously inside the EVM transaction, before any NEAR promise is created. [1](#0-0) 

There is no check at this point that the user-supplied `target_account_id` inside the `PromiseArgs` corresponds to an existing NEAR account.

**Step 2 – No existence check on the user's target account anywhere in the precompile.**

The precompile parses the user's `CrossContractCallArgs`, extracts the inner `PromiseArgs` (which carries the user-chosen `target_account_id`), serialises it as the argument to the router's `execute` method, and emits a log. The user's target is never validated. [2](#0-1) 

**Step 3 – Router dispatches the promise with no existence check.**

In `etc/xcc-router/src/lib.rs`, the router's `execute` method calls `Self::promise_create(promise)`, which routes to `base_promise_create`. That function calls `env::promise_create` directly with the user-supplied `target_account_id`. NEAR's runtime does not reject `promise_create` for a non-existent account at scheduling time; the promise is queued and only fails at execution time. [3](#0-2) [4](#0-3) 

**Step 4 – No failure callback is attached to the user's promise.**

In `engine/src/xcc.rs`, `handle_precompile_promise` attaches the user's promise as the final step in the chain. No failure-handling callback is attached after it. If the user's promise fails, there is no on-chain mechanism to reverse the wNEAR burn or return the refunded NEAR from the router to the user. [5](#0-4) 

**Step 5 – `require_no_failed_promises` does not protect against this.**

The router's `assert_preconditions` check (which calls `require_no_failed_promises`) only guards against failures in the *setup/withdraw* promises that precede `execute`. It does not guard against the failure of the user's outgoing promise, which is created *after* the precondition check. [6](#0-5) 

---

### Impact Explanation

The user's wNEAR is burned from their EVM balance during EVM execution (irreversible at the EVM layer). When the NEAR promise to the non-existent or deleted target account fails, NEAR's runtime refunds the attached deposit to the predecessor — the router sub-account. The router has no `withdraw` or user-facing refund function. The NEAR accumulates in the router sub-account and is inaccessible to the user without privileged admin intervention via the Aurora Engine owner. This constitutes **temporary (potentially permanent) freezing of user funds** — High severity per the allowed impact scope.

---

### Likelihood Explanation

Any EVM user can call the XCC precompile and specify an arbitrary NEAR `target_account_id` with attached NEAR. Two realistic triggering conditions exist:

1. **Race condition (analogous to self-destruct):** A user submits an XCC call to a live NEAR contract. Between EVM execution and asynchronous promise dispatch (which can span multiple NEAR blocks), the target contract's owner calls `DeleteAccount`. The promise fails; the user's wNEAR is lost.
2. **Typo / stale reference:** A user targets a NEAR account that has already been deleted or was never created. The EVM transaction succeeds, wNEAR is burned, and the promise fails.

Both paths are reachable by any unprivileged EVM user interacting with the XCC precompile at address `0x516cded1d16af10cad47d6d49128e2eb7d27b372`. [7](#0-6) 

---

### Recommendation

**Short term:**
1. In `engine-precompiles/src/xcc.rs`, before burning wNEAR, verify that the user's target NEAR account exists using `env::is_valid_account_id` or an equivalent host function check.
2. In `engine/src/xcc.rs` (`handle_precompile_promise`), attach a failure callback after the user's promise that re-mints the equivalent wNEAR back to the user's EVM address if the promise fails.

**Long term:**
Monitor NEAR account lifecycle events for accounts that are targets of scheduled XCC calls, and consider adding a `Delayed` XCC path that validates account existence at execution time rather than at scheduling time.

---

### Proof of Concept

1. Deploy a NEAR contract `victim.near`.
2. From an EVM address, call the XCC precompile (`0x516cded1...`) with `CrossContractCallArgs::Eager(PromiseArgs::Create(PromiseCreateArgs { target_account_id: "victim.near", attached_balance: 1_000_000_000_000_000_000_000_000 (1 NEAR), ... }))`.
3. The EVM transaction executes: the user's wNEAR is burned via `transferFrom`.
4. Before the NEAR promise executes (next block or later), the owner of `victim.near` calls `DeleteAccount`.
5. The NEAR promise dispatched by the router to `victim.near` fails.
6. NEAR runtime refunds 1 NEAR to the router sub-account (`{user_hex}.aurora`).
7. The user's EVM wNEAR balance is permanently reduced; the 1 NEAR sits in the router with no user-accessible recovery path. [8](#0-7) [3](#0-2) [5](#0-4)
