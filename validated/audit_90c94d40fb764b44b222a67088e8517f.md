### Title
`SenderInfoContent` Canister Signature Does Not Bind to Sender Principal, Enabling Cross-Sender Replay - (File: `rs/types/types/src/messages/http.rs`)

### Summary

The `SenderInfoContent` type, which defines the signed bytes for the `sender_info` canister attestation feature, includes only the raw `info` blob in its signed content. The sender's principal is not bound into the signature. Any attacker who observes a valid `sender_info` in a public ingress message can replay the same `{info, signer, sig}` triple in their own request under a different principal. A canister relying on `ic0.msg_caller_info_data()` for authorization decisions will receive the replayed attestation as if it were freshly issued for the attacker's principal.

### Finding Description

`SenderInfoContent` is defined as a newtype wrapper over a raw byte slice, and its `SignedBytesWithoutDomainSeparator` implementation appends only those raw bytes to the signed buffer:

```
rs/types/types/src/messages/http.rs:342-348
```

```rust
pub struct SenderInfoContent<'a>(pub &'a [u8]);

impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.0);   // only the info blob, no sender principal
    }
}
```

The full signed bytes are therefore `"\x0Eic-sender-info" || info_bytes` — the sender's principal (`UserId`) and the target canister ID are absent.

The validator in `verify_sender_info_canister_sig` verifies only three things:

```
rs/validator/src/ingress_validation.rs:494-544
```

1. `sender_pubkey` is a valid canister-signature public key.
2. The canister ID embedded in `sender_pubkey` matches `sender_info.signer`.
3. The canister signature `σ` is valid over `SenderInfoContent(&sender_info.info)`.

None of these checks bind `σ` to the principal of the request sender. The sender's principal appears in the outer message hash (and is covered by the user's own envelope signature), but the inner canister attestation `σ` is completely decoupled from it.

**Exploit path:**

1. User A (principal `P_A`) submits an ingress call to canister C with `sender_info = {info: bytes, signer: II, sig: σ}`, where `σ = II.sign("\x0Eic-sender-info" || bytes)`. The ingress message is publicly observable.
2. Attacker B (principal `P_B`) constructs a new ingress call to canister C (or any other canister) with the identical `sender_info = {info: bytes, signer: II, sig: σ}`.
3. B signs the new message ID (which commits to `P_B` as sender and to the replayed `sender_info` triple) with B's own key. This envelope signature is valid.
4. `validate_sender_info` → `verify_sender_info_canister_sig` accepts the request: `σ` is a valid canister signature over `SenderInfoContent(bytes)`, and the canister-ID check passes.
5. Canister C executes with `ic0.msg_caller()` = `P_B` and `ic0.msg_caller_info_data()` = `bytes` — the same attestation blob that II originally issued for user A.

### Impact Explanation

Any canister that uses `ic0.msg_caller_info_data()` to make security-sensitive decisions (e.g., Internet Identity attesting KYC status, age verification, or user-specific credentials) can be deceived. An unprivileged attacker with no keys other than their own can impersonate another user's attested attributes to any canister, potentially bypassing access controls, gaining unauthorized privileges, or corrupting per-user state. Impact is medium because exploitation requires a canister that (a) trusts `sender_info` for authorization and (b) does not embed the sender principal inside the `info` blob itself; however, the feature is explicitly designed for exactly this use case (Internet Identity attesting caller attributes), making the risk realistic.

### Likelihood Explanation

Ingress messages are public on the IC. Any observer can extract `{info, signer, sig}` from any accepted ingress message and replay it immediately in a fresh request. No privileged access, no key material, and no cryptographic break are required. The only constraint is finding a target canister that acts on `msg_caller_info_data()` without re-checking the sender principal inside the blob — which is the intended usage pattern for the feature.

### Recommendation

Include the sender's principal in the signed content of `SenderInfoContent`. The simplest fix is to change `SenderInfoContent` to carry both the sender principal bytes and the info blob, and to update `write_signed_bytes_without_domain_separator` to append both:

```rust
pub struct SenderInfoContent<'a> {
    pub sender: &'a [u8],   // sender principal bytes
    pub info: &'a [u8],
}

impl SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.sender);
        bytes.extend_from_slice(self.info);
    }
}
```

`validate_sender_info` must then pass `request.sender().get().as_slice()` when constructing `SenderInfoContent` for verification, ensuring the canister signature is bound to the specific principal that submitted the request.

### Proof of Concept

**Setup:** Internet Identity canister `II` issues `σ = II.sign("\x0Eic-sender-info" || b"age_verified=true")` for user A (principal `P_A`). User A submits an ingress call; the `sender_info = {info: b"age_verified=true", signer: II, sig: σ}` is visible in the public ingress pool.

**Replay:** Attacker B (principal `P_B`) constructs:
```
HttpCanisterUpdate {
    sender: P_B,
    sender_info: Some(RawSignedSenderInfo {
        info: b"age_verified=true",
        signer: II,
        sig: σ,          // copied verbatim from A's request
    }),
    ...
}
```
B signs the resulting `message_id` with B's own key. The request passes all validation checks in `validate_sender_info` / `verify_sender_info_canister_sig` because `σ` is a valid canister signature over `SenderInfoContent(b"age_verified=true")` regardless of who the sender is. The target canister receives `msg_caller_info_data() = b"age_verified=true"` while `msg_caller() = P_B`, granting B the same attested attributes as A.
