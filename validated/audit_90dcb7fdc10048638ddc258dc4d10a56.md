Audit Report

## Title
Missing Anonymous Principal Validation in `error_refund_icp` Allows Theft of ICP from Swap Canister's Anonymous Subaccount - (File: rs/sns/swap/src/swap.rs)

## Summary
The `error_refund_icp` function in the SNS Swap canister accepts a caller-controlled `source_principal_id` without validating it is not the anonymous principal. Any unprivileged caller can trigger a transfer of ICP from the swap canister's anonymous subaccount to the anonymous principal's ICP ledger account, from which any anonymous ingress caller can subsequently drain the funds via `icrc1_transfer`.

## Finding Description
In `rs/sns/swap/src/swap.rs`, the `error_refund_icp` function (L1925–2031) unpacks `source_principal_id` from the request at L1938–1948 and immediately proceeds to compute `source_subaccount = principal_to_subaccount(source_principal_id)` at L1971, query the balance, and transfer the full balance minus fee to `Account { owner: source_principal_id.0, subaccount: None }` at L1992–2004. No check is performed that `source_principal_id` is not the anonymous principal. [1](#0-0) [2](#0-1) 

The canister-level handler in `rs/sns/swap/canister/canister.rs` (L161–167) passes the request directly to the core logic with no caller validation: [3](#0-2) 

By contrast, `new_sale_ticket` in the same file explicitly rejects anonymous callers at L2537–2539, demonstrating the pattern is known and intentionally applied elsewhere: [4](#0-3) 

The ICP ledger's `icrc1_transfer` endpoint (L807–843) gates on `can_send`, and the `can_send` implementation in `rs/ledger_suite/icp/ledger/src/lib.rs` contains no reference to the anonymous principal, meaning the anonymous principal is not blocked from sending tokens it holds: [5](#0-4) 

**Exploit path:**
1. ICP arrives in `swap_canister[principal_to_subaccount(anonymous)]` — either via accidental user transfer or deliberate attacker deposit.
2. The swap reaches `Lifecycle::Aborted` or `Lifecycle::Committed` (normal terminal state for every SNS swap).
3. Attacker calls `error_refund_icp({ source_principal_id: Some(anonymous) })` from any ingress identity.
4. The function transfers the balance to `Account { owner: anonymous, subaccount: None }` on the ICP ledger.
5. Attacker calls `icrc1_transfer` on the ICP ledger as the anonymous principal, moving funds to their own account.

The comment at L1920–1924 asserts the function is "secure because it only transfers tokens from a principal's subaccount to the principal's own account," but this invariant breaks for the anonymous principal because the anonymous principal's account is accessible to any anonymous ingress caller. [6](#0-5) 

## Impact Explanation
This is an unauthorized access to canister-controlled ICP funds. The funds at risk are bounded to whatever ICP resides in the swap canister's anonymous subaccount at the time of exploitation. Accidental deposits to the anonymous subaccount are a realistic precondition (users mistyping or misconfiguring their principal). The theft is complete and irreversible once the `icrc1_transfer` step executes. This matches the **High** impact class: unauthorized access to canister-controlled funds with constraints (requires ICP in the anonymous subaccount and a terminal swap lifecycle).

## Likelihood Explanation
The swap reaching a terminal state is a certainty for every deployed SNS swap. The `error_refund_icp` endpoint is publicly callable with no authentication. The only non-trivial precondition is ICP being present in the anonymous subaccount, which can occur via accidental user transfer — a realistic event given the ICP ledger accepts transfers to any account identifier without validation. The attacker can also self-fund the subaccount and race a concurrent depositor, though this is less likely to be profitable.

## Recommendation
Add an anonymous principal check immediately after unpacking `source_principal_id`, consistent with the guard already used in `new_sale_ticket`:

```rust
if source_principal_id.is_anonymous() {
    return ErrorRefundIcpResponse::new_invalid_request_error(
        "source_principal_id must not be the anonymous principal.",
    );
}
```

This should be inserted after L1948 in `rs/sns/swap/src/swap.rs`. [7](#0-6) 

## Proof of Concept
1. Deploy an SNS swap canister and advance it to `Committed` or `Aborted` state using a PocketIC or local replica integration test.
2. Transfer any amount of ICP (e.g., 1 ICP) to `AccountIdentifier::new(swap_canister_id, Some(Subaccount::from(&PrincipalId::new_anonymous())))` on the ICP ledger.
3. Call `error_refund_icp({ source_principal_id: Some(PrincipalId::new_anonymous()) })` from any ingress identity.
4. Query `icrc1_balance_of({ owner: anonymous, subaccount: None })` on the ICP ledger — balance equals the deposited amount minus one transfer fee.
5. Call `icrc1_transfer` on the ICP ledger as the anonymous principal with the attacker's account as destination — funds are moved, completing the theft.
