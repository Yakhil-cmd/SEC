### Title
Malfunctioning Archive Canister Permanently Blocks All Subsequent Archive Upgrades With No Removal Mechanism - (`rs/ethereum/ledger-suite-orchestrator/src/scheduler/mod.rs`)

### Summary

The `UpgradeLedgerSuiteSubtask::UpgradeArchives` handler in the Ledger Suite Orchestrator (LSO) iterates over all managed archive canisters for a given token sequentially, using `?` to propagate the first error. If any single archive canister permanently fails to upgrade (e.g., its `pre_upgrade` hook panics, or the canister was deleted), the loop aborts and all subsequent archives in the list are never upgraded. There is no mechanism in the LSO to remove, skip, or disable a specific archive canister. Because `UpgradeCanisterError` and `StopCanisterError` are unconditionally marked as recoverable, the task retries indefinitely on every timer tick, always failing at the same archive and never making progress on the remaining ones.

### Finding Description

In `UpgradeLedgerSuiteSubtask::UpgradeArchives`, the execution path is:

```rust
for canister_id in archives {
    upgrade_canister::<Archive, _>(canister_id, compressed_wasm_hash, runtime)
        .await?;
}
``` [1](#0-0) 

The `?` operator immediately propagates any error from `upgrade_canister`, which itself performs three sequential management-canister calls:

1. `stop_canister` → `UpgradeLedgerSuiteError::StopCanisterError`
2. `install_code` (upgrade) → `UpgradeLedgerSuiteError::UpgradeCanisterError`
3. `start_canister` → `UpgradeLedgerSuiteError::StartCanisterError` [2](#0-1) 

All three error variants are unconditionally marked `is_recoverable() = true`, regardless of the underlying `CallError` reason:

```rust
UpgradeLedgerSuiteError::StopCanisterError(_) => true,
UpgradeLedgerSuiteError::StartCanisterError(_) => true,
UpgradeLedgerSuiteError::UpgradeCanisterError(_) => true,
``` [3](#0-2) 

This means even a `Reason::Rejected("canister not found")` (deleted archive) or a `Reason::CanisterError("canister trapped: ...")` (panicking `pre_upgrade`) causes the task to be rescheduled and retried on the next timer, forever.

The `UpgradeArg` Candid interface exposes no endpoint to remove or disable a specific archive canister from the managed list: [4](#0-3) 

The only way to add archives is through `manage_ledger_suites`, which adds suites but cannot remove individual archive principals from an existing suite.

### Impact Explanation

Given a token with archives `[A, B, C]` where `B` is permanently broken:

- Every `UpgradeArchives` execution: `A` is upgraded (wastefully re-upgraded on each retry), `B` fails, `C` is never reached.
- The `UpgradeLedgerSuite` task is rescheduled on every timer tick due to `is_recoverable = true`.
- Archive `C` (and any subsequent archives) can never be upgraded through the LSO.
- If the archive wasm contains a critical security vulnerability, it cannot be patched via the LSO for that token's suite. The NNS would need to bypass the LSO and manually call `install_code` on each affected archive directly.
- The LSO's upgrade task queue is permanently occupied with a non-progressing retry loop for the affected token.

### Likelihood Explanation

Two realistic failure modes exist:

1. **Archive wasm `pre_upgrade` panic**: If the archive wasm has a bug causing `pre_upgrade` to trap (e.g., instruction limit exceeded due to large state, or a deserialization bug), `install_code` returns `CanisterError("canister trapped: ...")`. This is classified as non-recoverable by `is_recoverable(CallError)` but overridden to recoverable by `UpgradeCanisterError => true`, causing infinite retries.

2. **Deleted archive canister**: If an archive canister is deleted (e.g., cycles exhausted before the LSO tops it up), `stop_canister` returns `Rejected("canister not found")`. `StopCanisterError` is unconditionally recoverable, causing infinite retries.

The LSO manages production ckERC20 tokens (ckUSDC, ckUSDT, ckETH) with real user funds. Archive canisters accumulate over time as ledger transaction volume grows. [5](#0-4) 

### Recommendation

1. **Change `UpgradeArchives` to continue on error**: Replace `await?` with explicit error handling that logs the failure and continues to the next archive:
   ```rust
   for canister_id in archives {
       if let Err(e) = upgrade_canister::<Archive, _>(canister_id, compressed_wasm_hash, runtime).await {
           log!(ERROR, "Failed to upgrade archive {canister_id}: {e:?}. Skipping.");
       }
   }
   ```

2. **Add a `remove_archive` or `skip_archive` field to `UpgradeArg`**: Allow the NNS to explicitly exclude a specific archive principal from the managed list or from upgrade tasks, analogous to the "enable/disable derivative" mitigation applied in the SafEth H-03 fix.

3. **Fix `is_recoverable` for `UpgradeCanisterError`**: Delegate to the inner `CallError::is_recoverable()` instead of unconditionally returning `true`, so that permanent failures (e.g., `Rejected`) are not retried indefinitely.

### Proof of Concept

1. The LSO manages ckUSDC with two archive canisters: `archive_1` and `archive_2`.
2. A bug in the archive wasm causes `archive_1`'s `pre_upgrade` hook to trap.
3. An NNS proposal triggers an archive upgrade via `UpgradeArg { archive_compressed_wasm_hash: Some(...), ... }`.
4. The LSO schedules `Task::UpgradeLedgerSuite(UpgradeLedgerSuite { subtasks: [UpgradeArchives { token_id: ckUSDC, ... }], ... })`.
5. `UpgradeArchives` executes: `stop_canister(archive_1)` succeeds, `install_code(archive_1, new_wasm)` fails with `CanisterError("canister trapped: ...")`.
6. The `?` propagates `UpgradeCanisterError`, aborting the loop. `archive_2` is never upgraded.
7. `UpgradeCanisterError::is_recoverable() == true` → task is rescheduled.
8. On every subsequent timer tick, the same failure repeats. `archive_2` remains on the old wasm indefinitely.
9. There is no `UpgradeArg` field to remove `archive_1` from the managed list or skip it. [6](#0-5) [3](#0-2) [7](#0-6)

fn is_recoverable(e: &CallError) -> bool {
    match &e.reason {
        Reason::OutOfCycles => true,
        Reason::CanisterError(msg) => msg.ends_with("is stopped") || msg.ends_with("is stopping"),

        Reason::TransientInternalError(_) => true,
        Reason::InternalError(_) => false,

async fn upgrade_canister<T: StorableWasm, R: CanisterRuntime>(
    canister_id: Principal,
    wasm_hash: &WasmHash,
    runtime: &R,
) -> Result<(), UpgradeLedgerSuiteError> {
    let wasm = match read_wasm_store(|s| wasm_store_try_get::<T>(s, wasm_hash)) {
        Ok(Some(wasm)) => Ok(wasm),
        Ok(None) => Err(UpgradeLedgerSuiteError::WasmHashNotFound(wasm_hash.clone())),
        Err(e) => Err(UpgradeLedgerSuiteError::WasmStoreError(e)),
    }?;

    log!(DEBUG, "Stopping canister {}", canister_id);
    runtime
        .stop_canister(canister_id)
        .await
        .map_err(UpgradeLedgerSuiteError::StopCanisterError)?;

    log!(
        DEBUG,
        "Upgrading wasm module of canister {} to {}",
        canister_id,
        wasm_hash
    );
    runtime
        .upgrade_canister(canister_id, wasm.to_bytes())
        .await
        .map_err(UpgradeLedgerSuiteError::UpgradeCanisterError)?;

    log!(DEBUG, "Starting canister {}", canister_id);
    runtime
        .start_canister(canister_id)
        .await
        .map_err(UpgradeLedgerSuiteError::StartCanisterError)?;

type UpgradeArg = record {
    // Hexadecimal encoding of the SHA-1 git commit hash used for this upgrade, e.g.,

    // [IC repository](https://github.com/dfinity/ic).
    // If this field is present, the orchestrator will register all embedded wasms (ledger, index, and archive) in its stable memory,
    // so that those exact wasms can be used to upgrade managed canisters as specified below.
    git_commit_hash: opt text;

   // Hexadecimal encoding of the SHA2-256 ledger compressed wasm hash, e.g.,
   // "3148f7a9f1b0ee39262c8abe3b08813480cf78551eee5a60ab1cf38433b5d9b0".
   // This exact version will be used for upgrading all existing ledger canisters managed by the orchestrator.
   // Leaving this field empty will not upgrade the ledger canisters.
   ledger_compressed_wasm_hash: opt text;

   archive_compressed_wasm_hash: opt text;

= Ledger Suite Orchestrator

The ledger suite orchestrator manages ledger suites of all ckERC20 tokens.
Each ledger suite, one for each ckERC20 token, consists of multiple canisters:
one ledger; one index; and zero, one or multiple archives.

The ledger suite orchestrator has the following 3 main responsibilities:

. Create a new ledger suite for a new ckERC20 token with a single NNS proposal.
. Upgrade managed canisters with a single NNS proposal.
. Top up the managed canisters with cycles.
