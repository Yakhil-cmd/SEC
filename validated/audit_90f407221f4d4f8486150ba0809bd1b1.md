Audit Report

## Title
Unchecked Subtraction in `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` Permanently Blocks Governance-Approved SNS Treasury Transfers - (File: rs/sns/governance/src/proposal.rs)

## Summary
In `rs/sns/governance/src/proposal.rs`, the execution-time guard `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` performs a plain subtraction `allowance_tokens - spent_tokens` at line 2631, where both operands are `rust_decimal::Decimal` (a signed type). Because `allowance_tokens` is derived from the treasury valuation stored at proposal submission time while `spent_tokens` is the live 7-day rolling total at execution time, price volatility can cause `spent_tokens > allowance_tokens`, yielding a negative `remainder_tokens`. Any positive `transfer_amount_tokens > remainder_tokens` check then always returns `PreconditionFailed`, permanently failing a governance-approved proposal.

## Finding Description
The function `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` ( [1](#0-0) ) computes `allowance_tokens` from the submission-time `Valuation` stored in `ActionAuxiliary::TransferSnsTreasuryFunds` ( [2](#0-1) ) and `spent_tokens` as the live 7-day rolling total at execution time ( [3](#0-2) ). The code itself acknowledges the mismatch in a comment: *"The total calculated here _could_ be different from what was calculated at proposal submission/creation time."*

The plain subtraction at line 2631 is the root cause:
```rust
let remainder_tokens = allowance_tokens - spent_tokens;
``` [4](#0-3) 

Both operands are `rust_decimal::Decimal` (signed), confirmed by the import at line 58 and the return type of `transfer_sns_treasury_funds_7_day_total_upper_bound_tokens` in `proposals_amount_total_limit/src/lib.rs`: [5](#0-4) 

When `spent_tokens > allowance_tokens`, `remainder_tokens` is negative. The subsequent check:
```rust
if transfer_amount_tokens > remainder_tokens { return Err(...PreconditionFailed...); }
``` [6](#0-5) 

is trivially true for any positive `transfer_amount_tokens`, so the proposal always fails.

The submission-time path in `treasury_valuation_if_proposal_amount_is_small_enough_or_err` correctly uses `checked_sub`:
```rust
let allowance_remainder_tokens = max_tokens.checked_sub(spent_tokens).ok_or_else(|| { ... })?;
``` [7](#0-6) 

The execution-time caller in `perform_transfer_sns_treasury_funds` passes the submission-time valuation directly: [8](#0-7) 

**Cross-regime trigger scenario:**
1. Treasury is large (>1.2M XDR). Proposal B (140 tokens, limit ~150 tokens) is submitted and executed. `spent_tokens` = 140.
2. Token price drops 10×. Treasury is now medium (<1.2M XDR). Proposal A (1 token) is submitted; `allowance_tokens_A` = 25% × 860 = **215 tokens**.
3. Price recovers. Proposal C (140 tokens, large regime) is submitted and executed. `spent_tokens` = **280**.
4. Proposal A executes: `allowance_tokens_A` = 215, `spent_tokens` = 280 → **215 − 280 = −65** → `transfer_amount_tokens (1) > −65` → `PreconditionFailed`. Proposal A is permanently failed.

The `ProposalDecisionStatus::Failed` state is final and non-retryable: [9](#0-8) 

## Impact Explanation
A governance-approved `TransferSnsTreasuryFunds` proposal is permanently marked `Failed` with `ErrorType::PreconditionFailed` due to incorrect arithmetic, not due to any actual spending limit violation. The SNS treasury transfer mechanism is effectively DoS'd for any proposal submitted during a lower-value regime that overlaps with a higher-value spending window within the same 7-day rolling window. This constitutes a significant SNS governance impact with concrete harm: legitimately voted-on treasury operations cannot be executed, and the community must wait for the 7-day window to roll past the high-value proposals before resubmitting. This matches the **High ($2,000–$10,000)** impact category: *"Significant SNS... security impact with concrete user or protocol harm."*

## Likelihood Explanation
No privileged access is required. Any SNS governance participant can submit a `TransferSnsTreasuryFunds` proposal via the standard ingress path. The trigger requires only: (a) token price volatility causing a regime transition between proposal submission and execution (common for SNS tokens), and (b) other proposals executed in the same 7-day window under a higher-value regime (normal governance activity). Both conditions are realistic under normal market conditions. The bug is deterministic and repeatable whenever the cross-regime condition is met.

## Recommendation
Replace the plain subtraction at line 2631 with a checked subtraction, mirroring the submission-time check at line 801:

```rust
// Before (line 2631):
let remainder_tokens = allowance_tokens - spent_tokens;

// After:
let remainder_tokens = allowance_tokens.checked_sub(spent_tokens).ok_or_else(|| {
    GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        format!(
            "The 7-day spending total ({spent_tokens} tokens) already exceeds the \
             allowance ({allowance_tokens} tokens) computed from the valuation at \
             proposal submission time. Try again after the 7-day window resets.",
        ),
    )
})?;
```
