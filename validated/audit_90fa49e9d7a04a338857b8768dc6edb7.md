Audit Report

## Title
Unauthenticated Caller Can Force SNS Swap Participation on Behalf of Any Principal, Bypassing Confirmation Text Consent — (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` update endpoint accepts a caller-supplied `buyer` principal with no check that the caller matches the buyer. Any unprivileged ingress sender can invoke this function specifying an arbitrary victim principal and the publicly visible confirmation text, causing the swap canister to commit the victim's ICP to the swap without the victim ever consenting. The confirmation text mechanism — designed to obtain explicit in-band consent — is rendered ineffective.

## Finding Description
In `rs/sns/swap/canister/canister.rs` at lines 127–143, the `#[update]` endpoint `refresh_buyer_tokens` resolves the buyer principal from `arg.buyer` with no authorization check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no caller == buyer assertion
};
```

There is no `inspect_message` guard and no post-resolution `assert!(caller == p)`.

The downstream `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs` at lines 363–384 validates the confirmation text purely by string equality against the stored `init` value — it does not verify that the caller is the buyer:

```rust
pub fn validate_confirmation_text(&self, confirmation_text: Option<String>) -> Result<(), String> {
    match (self.init_or_panic().confirmation_text.as_ref(), confirmation_text) {
        (Some(expected_text), Some(text)) => {
            if &text != expected_text { Err(...) } else { Ok(()) }
        }
        ...
    }
}
```

The confirmation text is stored in the swap's public `init` state, readable by anyone. After passing this check, the function reads the ICP ledger balance of the victim's subaccount (lines 1153–1163) and, if sufficient, commits the victim's ICP to the swap (lines 1285–1288). The entire exploit path requires no privileged access.

## Impact Explanation
This matches the allowed High impact: **"Significant SNS security impact with concrete user or protocol harm."** A victim who deposited ICP into their swap subaccount but chose not to call `refresh_buyer_tokens` — for example, because they did not wish to agree to the confirmation text — can be forcibly enrolled by any attacker. Once enrolled, the victim's ICP is locked in the swap until it reaches a terminal state (committed or aborted). The confirmation text consent mechanism, which exists specifically to record explicit participant agreement, is completely bypassed because the attacker — not the victim — supplies the text.

## Likelihood Explanation
The attack requires only: (1) the victim's principal ID, which is publicly derivable from any prior on-chain interaction; (2) the confirmation text, which is stored in the swap's public `init` state and readable by any caller; and (3) the victim to have ICP already sitting in their swap subaccount. No privileged role, key material, or subnet-majority corruption is required. The endpoint is a standard `#[update]` method reachable by any ingress sender with no `inspect_message` guard. The attack is repeatable against any victim who has funded their subaccount.

## Recommendation
When `arg.buyer` is non-empty, assert `caller_principal_id() == p` before proceeding:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let p = PrincipalId::from_str(&arg.buyer).unwrap();
    assert_eq!(caller_principal_id(), p, "caller must match buyer");
    p
};
```

This preserves the stated design intent (the empty-buyer path defaults to the caller) while preventing third-party forced enrollment.

## Proof of Concept
1. Deploy an SNS swap with `confirmation_text = "I agree to the SNS terms"`.
2. Alice sends 5 ICP to her swap subaccount (`subaccount = principal_to_subaccount(Alice)`), intending to evaluate before committing.
3. Alice decides not to participate and does not call `refresh_buyer_tokens`.
4. Bob (attacker) submits an ingress call:
   ```
   refresh_buyer_tokens({
     buyer: "<Alice's principal text>",
     confirmation_text: opt "I agree to the SNS terms"
   })
   ```
5. The swap canister resolves `p = Alice`, calls `validate_confirmation_text` (passes, text matches), reads Alice's subaccount balance (5 ICP ≥ `min_participant_icp_e8s`), and commits Alice's 5 ICP to the swap.
6. Alice's ICP is now locked until the swap reaches a terminal lifecycle state — all without Alice ever calling the endpoint or agreeing to the terms.

A deterministic integration test or PocketIC test can reproduce this by: constructing a swap in `Open` state with a `confirmation_text`, crediting a victim subaccount on a mock ICP ledger, calling `refresh_buyer_token_e8s` with a different `buyer` principal than the simulated caller, and asserting the victim appears in `swap.buyers` with a non-zero balance.
