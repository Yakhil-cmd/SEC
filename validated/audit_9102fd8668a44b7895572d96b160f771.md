Audit Report

## Title
Single Failed `get_transaction_receipt` Call Aborts All Pending ckETH/ckERC20 Withdrawal Finalizations — (File: `rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary

In `finalize_transactions_batch`, all `get_transaction_receipt` calls are fired in parallel via `join_all`, but the result-processing loop contains a bare `return` on any `Err` result. This exits the entire async function, silently discarding all successfully fetched receipts for other withdrawals in the same batch. Because `NoReduction` fails on any provider disagreement — a realistic, non-adversarial condition — every pending ckETH/ckERC20 withdrawal whose nonce is below the on-chain finalized count is blocked from completing in that timer tick. A secondary `assert_eq!` in the same function can panic the canister if all transactions for a given nonce return `Ok(None)`.

## Finding Description

`finalize_transactions_batch` (line 386) is an async function. After collecting all transaction hashes via `sent_transactions_to_finalize` and firing all `get_transaction_receipt` calls in parallel, the result loop at lines 410–444 processes each `(hash, withdrawal_id, result)` triple. On `Err(e)`, line 442 executes a bare `return`, which exits `finalize_transactions_batch` entirely — not just the current loop iteration. All receipts already inserted into the `receipts` map are discarded; no `mutate_state` / `process_event` call is made for any withdrawal.

The reduction strategy applied to each receipt call is `NoReduction` (lines 400–406). As confirmed at lines 177–182 of `eth_rpc_client/mod.rs`, `NoReduction::reduce` returns `Err(MultiCallError::InconsistentResults(...))` whenever any single provider returns a different value from the others — not only when all providers fail. This is the strictest possible strategy and is applied on top of the already-aggregated `EvmMultiRpcResult`.

By contrast, `send_transactions_batch` (line 340) uses `AnyOf` (line 359), which succeeds if at least one provider agrees, and its result loop (lines 363–383) never `return`s on error — it logs and continues.

The secondary risk: after the loop, lines 447–450 execute `assert_eq!(expected_finalized_withdrawal_ids, actual_finalized_withdrawal_ids, ...)`. `expected_finalized_withdrawal_ids` is the set of all unique `LedgerBurnIndex` values from `txs_to_finalize`. If all resubmitted transaction hashes for a given nonce return `Ok(None)` (e.g., the EVM RPC reports a finalized count ahead of what providers can serve receipts for), the corresponding `LedgerBurnIndex` is absent from `receipts`, the assert fires, and the canister traps. On the IC, a trap causes a full message rollback, preventing any state update until the next successful invocation.

## Impact Explanation

Every ckETH and ckERC20 withdrawal in the `sent_tx` state with a nonce below the on-chain finalized count is blocked from completing in the affected timer tick. This is a concrete, application-level DoS on ckETH/ckERC20 withdrawal finalization — an explicitly in-scope Chain Fusion / ck-token component. If the provider inconsistency persists across ticks (e.g., a provider is consistently lagging), the delay extends across multiple timer cycles, causing prolonged user-visible withdrawal delays. The assert-panic path additionally risks a canister trap and full rollback, which is a more severe availability impact. This matches the High severity class: "Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."

## Likelihood Explanation

No adversarial action is required. The `NoReduction` strategy fails whenever any one of the 4 mainnet EVM RPC providers returns a different receipt value from the others. `eth_getTransactionReceipt` responses are particularly susceptible to transient provider disagreement because different nodes may have slightly different chain views at any moment (e.g., one provider is one block behind). This is a routine, non-adversarial network condition. The minter's `Threshold { total: Some(4), min: 3 }` consensus strategy at the `EvmRpcClient` level does not prevent `NoReduction` from failing on the already-aggregated result. The failure is repeatable and self-sustaining as long as any provider disagreement persists.

## Recommendation

1. **Per-item error handling**: Replace the `return` at line 442 with `continue`, so a single failed receipt fetch does not abort processing of all other withdrawals. Log the failure and allow the failed item to be retried on the next timer tick.
2. **Relax the reduction strategy**: Consider replacing `NoReduction` with a majority-based strategy (e.g., `StrictMajorityByKey`) for `get_transaction_receipt`, consistent with how `send_raw_transaction` uses `AnyOf`.
3. **Guard the assert**: Replace the `assert_eq!` at lines 447–450 with a graceful log and early `return` to prevent a canister trap in the unexpected-state scenario.

## Proof of Concept

1. Two users queue ckETH withdrawals: burn index 1 (nonce 5), burn index 2 (nonce 6). Both nonces advance below the on-chain finalized transaction count.
2. `finalize_transactions_batch` is invoked by the timer.
3. `sent_transactions_to_finalize` returns hashes for both nonces (including any resubmissions).
4. `join_all` fires all `get_transaction_receipt` calls in parallel.
5. The receipt for one of nonce 5's hashes returns `Err(InconsistentResults)` (two providers disagree).
6. The loop reaches line 442 and `return`s, exiting `finalize_transactions_batch`.
7. The receipt for nonce 6 (already successfully fetched) is discarded.
8. Neither withdrawal is finalized. Both users wait for the next timer tick.
9. If the provider inconsistency persists, the DoS extends across multiple timer cycles.

A deterministic integration test can reproduce this by mocking one EVM RPC provider to return a different receipt value for a specific transaction hash, then asserting that no `FinalizedTransaction` event is emitted for any withdrawal in the batch — including those whose receipts were consistent across all providers.
