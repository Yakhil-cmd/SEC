Audit Report

## Title
Non-Disclosed Rate-Limit Rule Metadata Exposed to Anonymous Callers via Unguarded Non-Replicated Query Methods — (`rs/boundary_node/rate_limits/canister/canister.rs`, `getter.rs`, `confidentiality_formatting.rs`)

## Summary

The rate-limits canister relies solely on `inspect_message` to enforce access control on `get_config`, `get_rule_by_id`, and `get_rules_by_incident_id`. On the Internet Computer, `inspect_message` is only invoked for ingress update messages; non-replicated query calls bypass it unconditionally. All three methods are annotated `#[query]`, so any anonymous caller can invoke them as non-replicated queries, bypassing the only access-control gate. The `RuleConfidentialityFormatter` and `ConfigConfidentialityFormatter` only redact `rule_raw` and `description` for non-disclosed rules, leaving `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` intact in every response. An anonymous caller can therefore enumerate the full set of non-disclosed rules and incidents, their canister-version lifecycle, and the incident-to-rule mapping — all before public disclosure.

## Finding Description

**Root cause — `inspect_message` does not fire for non-replicated query calls.**

`inspect_message` is registered as the ingress filter hook:

```rust
// canister.rs L30-31
const UPDATE_METHODS: [&str; 2] = ["add_config", "disclose_rules"];
const REPLICATED_QUERY_METHOD: &str = "get_config";
``` [1](#0-0) 

The hook's `else` branch traps "all other calls": [2](#0-1) 

Per the IC protocol specification, `inspect_message` is invoked **only for ingress update messages**. Non-replicated query calls skip it entirely. The trap in the `else` branch therefore never fires for query-type invocations.

**`get_rule_by_id` and `get_rules_by_incident_id` are `#[query]` methods absent from every guard.** [3](#0-2) 

Neither method appears in `UPDATE_METHODS` nor equals `REPLICATED_QUERY_METHOD`. They are fully reachable as non-replicated queries by any caller.

**`get_config` is also a `#[query]` method, bypassing `inspect_message` when called as a non-replicated query.** [4](#0-3) 

The `inspect_message` check for `get_config` requires `has_full_access || has_full_read_access`, but this check is never reached for non-replicated query calls.

**`ConfigConfidentialityFormatter` leaves `rule_id` and `incident_id` unredacted for non-disclosed rules.** [5](#0-4) 

Only `description` and `rule_raw` are cleared. `rule_id` and `incident_id` are always serialized into the response via `From<OutputRule> for api::OutputRule`: [6](#0-5) 

**`RuleConfidentialityFormatter` leaves `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` unredacted.** [7](#0-6) 

The full `api::OutputRuleMetadata` conversion always includes all four identifying/versioning fields: [8](#0-7) 

**`AccessLevelResolver` assigns `RestrictedRead` to any caller that is not the authorized principal or an API boundary node**, including the anonymous principal: [9](#0-8) 

**`RuleGetter.get()` confirms the leak path for `RestrictedRead` callers.** [10](#0-9) 

The formatter is applied but the resulting struct still carries all metadata fields. The existing unit test at `getter.rs` lines 433–446 explicitly documents that `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` are present in the `RestrictedRead` response for a non-disclosed rule: [11](#0-10) 

**Complete exploit chain:**
1. Call `get_config` as a non-replicated query → `inspect_message` skipped → `RestrictedRead` path → `ConfigConfidentialityFormatter` applied → response contains `rule_id` and `incident_id` for **all** rules, including non-disclosed ones.
2. For each harvested `rule_id`, call `get_rule_by_id` as a non-replicated query → `inspect_message` skipped → `RuleConfidentialityFormatter` applied → response contains `rule_id`, `incident_id`, `added_in_version`, `removed_in_version`.
3. For each harvested `incident_id`, call `get_rules_by_incident_id` as a non-replicated query → same result for every rule in the incident.

## Impact Explanation

An anonymous caller obtains: the existence of every non-disclosed rate-limit rule and incident, the incident-to-rule mapping, and the exact canister versions at which each rule was added and removed. This reveals the structure and timing of active incident response before public disclosure. An attacker can detect when new rate-limit rules are being deployed and adjust evasion behavior accordingly, directly undermining the confidentiality guarantee of the boundary node rate-limiting system. This constitutes a **significant boundary/API security impact with concrete protocol harm** — matching the High ($2,000–$10,000) impact tier for boundary node infrastructure.

## Likelihood Explanation

The exploit requires only standard IC query calls with no special tooling, no privileged access, and no brute-forcing. UUID enumeration is fully solved by calling `get_config` as a query first. The path is entirely local-testable with `dfx` or PocketIC and is unconditionally repeatable on every canister deployment.

## Recommendation

1. **Add in-handler authorization checks** inside `get_rule_by_id`, `get_rules_by_incident_id`, and `get_config` that reject `RestrictedRead` callers at the application level, independent of `inspect_message`. Do not rely on `inspect_message` as a security boundary for query calls.
2. **Redact `rule_id` and `incident_id`** in `RuleConfidentialityFormatter` and `ConfigConfidentialityFormatter` for non-disclosed rules, so that even if the methods remain callable, the existence of a non-disclosed rule is not confirmed.
3. Optionally convert `get_rule_by_id` and `get_rules_by_incident_id` to `#[update]` methods and add them to `UPDATE_METHODS` if replicated execution and `inspect_message` enforcement is the intended access-control model.

## Proof of Concept

```bash
# Step 1: Harvest all rule_id and incident_id values (including non-disclosed)
dfx canister call rate_limits_canister get_config '(null)' --query
# Response: rule_id and incident_id present for ALL rules; rule_raw/description null for non-disclosed.

# Step 2: For each non-disclosed rule_id from step 1:
dfx canister call rate_limits_canister get_rule_by_id '("<non-disclosed-uuid>")' --query
# Response: rule_id, incident_id, added_in_version, removed_in_version present; rule_raw/description null.

# Step 3: For each incident_id from step 1:
dfx canister call rate_limits_canister get_rules_by_incident_id '("<incident-uuid>")' --query
# Response: full metadata for all rules in the incident, including non-disclosed ones.
```

The existing unit test `test_get_rule_success` in `getter.rs` (lines 433–446) already serves as a deterministic proof: it asserts that `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` are non-null in the `RestrictedRead` response for a non-disclosed rule, confirming the leak at the application layer. A PocketIC integration test invoking the same methods as the anonymous principal over the query endpoint would confirm the `inspect_message` bypass at the protocol layer.

impl ConfidentialityFormatting for ConfigConfidentialityFormatter {
    type Input = OutputConfig;

    fn format(&self, config: OutputConfig) -> OutputConfig {
        let mut config = config;
        config.is_redacted = true;
        // Redact (hide) fields of non-disclosed rules
        config.rules.iter_mut().for_each(|rule| {
            if rule.disclosed_at.is_none() {
                rule.description = None;
                rule.rule_raw = None;
            }
        });
        config
    }
}
