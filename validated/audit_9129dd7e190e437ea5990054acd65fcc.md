Audit Report

## Title
ckERC20 Withdrawal with Insufficient `max_transaction_fee` Permanently Blocks All Subsequent Withdrawal Finalizations - (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

## Summary

When a ckERC20 withdrawal transaction's `max_transaction_fee` cap is too low to cover a bumped gas price during resubmission, `create_resubmit_transactions` pushes an `InsufficientTransactionFee` error and immediately returns, leaving the stuck nonce in `sent_tx` indefinitely. Because `sent_transactions_to_finalize` only processes nonces strictly below the on-chain finalized transaction count, and Ethereum enforces sequential nonce ordering, the stuck nonce permanently prevents finalization of every subsequent withdrawal (nonces N+1, N+2, …) until gas prices fall or a canister upgrade is performed. Any unprivileged user can trigger this by submitting a ckERC20 withdrawal with a minimal `max_transaction_fee`.

## Finding Description

**Root cause — `create_resubmit_transactions` early-returns on first fee-cap breach:** [1](#0-0) 

The function iterates over all pending sent transactions in nonce order. When `SignedTransactionRequest::resubmit` returns `InsufficientTransactionFee` for nonce N (because `new_tx_price.max_transaction_fee() > self.resubmission.allowed_max_transaction_fee()`), the function pushes the error and immediately `return`s, leaving nonces N+1, N+2, … unprocessed. [2](#0-1) 

**ckERC20 uses `GuaranteeEthAmount` with user-supplied fee cap:** [3](#0-2) 

The `allowed_max_transaction_fee` is exactly `ckerc20.max_transaction_fee` — the value the user supplied at withdrawal time. A user can set this to the minimum value that passes the initial `create_transaction` check at current gas prices.

**The caller silently logs and discards the error:** [4](#0-3) 

`resubmit_transactions_batch` logs the error and does nothing else. The stuck transaction remains in `sent_tx` with its original low fee cap on every subsequent timer tick.

**Finalization is gated on the Ethereum finalized transaction count:** [5](#0-4) 

`sent_transactions_to_finalize` only considers nonces strictly below `finalized_transaction_count`. Because Ethereum will not mine nonce N+1 before nonce N is mined, the finalized count never advances past N, so no subsequent withdrawal can ever be finalized.

**Sequential nonce assignment:** [6](#0-5) 

Each withdrawal request receives `next_nonce` which is then incremented, creating a strict sequential dependency.

**The code itself acknowledges the blocking:** [7](#0-6) 

The comment states: *"if a transaction with nonce n could not be resubmitted… then the next transactions with nonces n+1, n+2, … are blocked anyway."*

**No cancel/quarantine/skip mechanism exists.** There is no admin endpoint, no `cancel_withdrawal` function, and no operator path to remove a stuck nonce from `sent_tx` without a canister upgrade. The `reschedule_withdrawal_request` mechanism at the creation stage (line 290 in `withdraw.rs`) only applies before a nonce is assigned and does not help once a transaction is in `sent_tx`. [8](#0-7) 

## Impact Explanation

A single ckERC20 withdrawal with an insufficient `max_transaction_fee` permanently halts finalization of all subsequent ckETH and ckERC20 withdrawals. Users behind the stuck nonce cannot receive their funds. This constitutes a significant Chain Fusion / ck-token platform-level DoS with concrete user harm — matching the **High ($2,000–$10,000)** impact class: *"Significant Chain Fusion, ck-token, ledger… security impact with concrete user or protocol harm."*

## Likelihood Explanation

Any unprivileged user calling `withdraw_erc20` can trigger this. The attacker sets `max_transaction_fee` to the minimum value that passes the `create_transaction` check at current gas prices, then waits for a natural gas-price spike (5–10× spikes within hours are historically common on Ethereum mainnet). No privileged access, no threshold corruption, and no external oracle manipulation is required. An adversary can also time the submission to coincide with a known high-activity event. The condition is repeatable and deterministic once triggered.

## Recommendation

1. **Add a cancel-and-reimburse path for stuck nonces.** When `create_resubmit_transactions` returns `InsufficientTransactionFee` for nonce N, the minter should burn the nonce by sending a zero-value self-transfer at the current gas price (funded from the minter's ETH reserve), remove the entry from `sent_tx`, and reimburse the user's ckERC20 tokens. This unblocks nonces N+1, N+2, ….
2. **Expose an operator-callable `cancel_withdrawal` endpoint** under NNS governance control that performs the same nonce-burn and reimbursement.
3. **Enforce a minimum `max_transaction_fee` multiplier at submission time** (e.g., 2–3× the current gas estimate) to reduce the probability of a fee cap being breached during normal volatility.

## Proof of Concept

1. Ethereum mainnet gas price is 10 gwei. Attacker calls `withdraw_erc20` with `max_transaction_fee` = 650,000 gwei (barely covers 10 gwei × 65,000 gas limit). The minter creates a transaction with nonce N and sends it.
2. Gas price spikes to 15 gwei. The minter's timer calls `resubmit_transactions_batch` → `create_resubmit_transactions(latest_tx_count, new_gas_fee)`. For nonce N, `new_tx_price.max_transaction_fee()` = 975,000 gwei > 650,000 gwei = `allowed_max_transaction_fee`. The function pushes `Err(InsufficientTransactionFee)` and returns immediately.
3. `resubmit_transactions_batch` logs the error and does nothing. Nonce N remains in `sent_tx` with the original 10-gwei fee cap. Ethereum will not mine it at 15 gwei.
4. All subsequent withdrawal requests (nonces N+1, N+2, …) are in `sent_tx` but `sent_transactions_to_finalize` filters them out because `finalized_transaction_count` is still N.
5. Every subsequent timer tick re-runs `resubmit_transactions_batch`, hits the same error for nonce N, and exits. No subsequent withdrawal is ever finalized until gas prices drop below 10 gwei or a canister upgrade intervenes.

A deterministic integration test can reproduce this by: (a) creating a ckERC20 withdrawal with a minimal fee cap, (b) advancing the mock gas fee estimate above the cap, (c) calling `create_resubmit_transactions`, and (d) asserting that `sent_transactions_to_finalize` returns an empty map for all nonces ≥ N even after multiple iterations.
