### Title
Hotkey Bypasses Controller-Only Guard for NeuronManagement Following via `topic: None` Catch-All — (`rs/nns/governance/src/pb/mod.rs`)

### Summary

`SetFollowing::validate_authorized` checks for `NeuronManagement` topic by comparing `followees_for_topic.topic == Some(Topic::NeuronManagement as i32)`. A caller supplying `topic: None` makes this comparison `false`, so only `is_authorized_to_vote` (hotkey-sufficient) is required. However, `into_followees` maps `None` to `0` via `unwrap_or_default()`, storing the entry under key `0` (= `Topic::Unspecified`), which is the catch-all fallback used when no explicit per-topic following exists — including for `NeuronManagement` proposals. A registered hotkey therefore silently installs catch-all following that covers `NeuronManagement`, bypassing the controller-only invariant.

### Finding Description

**Authorization check** (`rs/nns/governance/src/pb/mod.rs`, lines 154–162):

```rust
let any_manage_neuron = self.topic_following.iter().any(|followees_for_topic| {
    followees_for_topic.topic == Some(Topic::NeuronManagement as i32)
});

if any_manage_neuron {
    neuron.is_controlled_by(caller)
} else {
    neuron.is_authorized_to_vote(caller)   // hotkey is sufficient
}
``` [1](#0-0) 

`any_manage_neuron` is `false` when `topic` is `None`, because `None != Some(NeuronManagement as i32)`.

**Execution path** (`rs/nns/governance/src/pb/mod.rs`, lines 82–89):

```rust
pub fn into_followees(self) -> HashMap<i32, Followees> {
    self.topic_following
        .into_iter()
        .map(|FolloweesForTopic { topic, followees }| {
            (topic.unwrap_or_default(), Followees { followees })  // None → 0
        })
        .collect()
}
``` [2](#0-1) 

`topic.unwrap_or_default()` converts `None` to `0i32` = `Topic::Unspecified`, which is the catch-all key. The uniqueness validator also explicitly documents this equivalence:

```rust
// Treat None the same as Some(0). This also occurs during execution.
let topic = followees_for_topic.topic.unwrap_or_default();
``` [3](#0-2) 

The catch-all (key `0`) is the fallback used by the governance voting cascade for any topic that has no explicit per-topic following entry, including `NeuronManagement`. A hotkey can therefore write to key `0` and have that following apply to `NeuronManagement` proposals on the victim neuron.

---

### Impact Explanation

- A registered hotkey of neuron V calls `manage_neuron(SetFollowing { topic_following: [FolloweesForTopic { topic: None, followees: [attacker_neuron_id] }] })`.
- `validate_authorized` passes because `any_manage_neuron = false` and the caller is a valid hotkey.
- `into_followees` stores `{ 0 → attacker_neuron }` in `neuron.followees`.
- When a `NeuronManagement` proposal targeting neuron V is submitted (and neuron V has no explicit `NeuronManagement` following), the governance cascade falls back to key `0` and neuron V follows the attacker's neuron.
- `NeuronManagement` proposals can change neuron V's following, add/remove hotkeys, and perform other privileged operations — effectively giving the attacker control over neuron V's governance participation without ever holding the controller key.

---

### Likelihood Explanation

The attacker only needs to be a registered hotkey of the target neuron, which is a common, unprivileged role. The call is a standard ingress `manage_neuron` message. No threshold corruption, admin key, or social engineering is required. The path is fully local-testable.

---

### Recommendation

In `validate_authorized`, treat `topic: None` (which maps to `0` = catch-all) the same as an explicit `NeuronManagement` entry, since catch-all following covers `NeuronManagement` as a fallback. Concretely, change the `any_manage_neuron` predicate to also return `true` when `topic` is `None` or `Some(0)`:

```rust
let any_manage_neuron = self.topic_following.iter().any(|followees_for_topic| {
    let effective_topic = followees_for_topic.topic.unwrap_or_default();
    effective_topic == Topic::NeuronManagement as i32
        || effective_topic == Topic::Unspecified as i32  // catch-all covers NeuronManagement
});
``` [4](#0-3)

### Proof of Concept

1. Register principal `hotkey` as a hotkey on neuron V (controller is a separate principal).
2. Call `manage_neuron` as `hotkey` with `Command::SetFollowing(SetFollowing { topic_following: [FolloweesForTopic { topic: None, followees: [attacker_neuron_id] }] })`.
3. `validate_authorized`: `any_manage_neuron = false` (None ≠ Some(NeuronManagement)), so `is_authorized_to_vote` passes for the hotkey.
4. `into_followees`: stores `{ 0 → attacker_neuron }` in neuron V's followees map.
5. Submit a `NeuronManagement` proposal targeting neuron V with no explicit `NeuronManagement` following set.
6. Observe that the governance cascade fires for neuron V using key `0` (catch-all), voting as the attacker's neuron directs.

    fn validate_authorized(
        &self,
        caller: &PrincipalId,
        neuron: &Neuron,
    ) -> Result<(), GovernanceError> {
        let ok = {
            let any_manage_neuron = self.topic_following.iter().any(|followees_for_topic| {
                followees_for_topic.topic == Some(Topic::NeuronManagement as i32)
            });

            if any_manage_neuron {
                neuron.is_controlled_by(caller)
            } else {
                neuron.is_authorized_to_vote(caller)
            }
        };

        if !ok {
            return Err(GovernanceError::new_with_message(
                ErrorType::NotAuthorized,
                format!(
                    "Caller ({}) is not authorized to make such changes to the following of neuron {}.",
                    caller,
                    neuron.id().id,
                ),
            ));
        }

        Ok(())
    }
