### Title
Changing ETH Connector Account Without Supply Check Permanently Freezes All User NEP-141 ETH Balances - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

`set_eth_connector_contract_account` overwrites the stored connector account ID with no check that the old connector's NEP-141 total supply is zero. Every subsequent call to `withdraw`, `ft_transfer`, `ft_transfer_call`, `ft_balance_of`, and the `ExitToNear` base-token path is routed to the **new** connector, while all existing user balances remain on the **old** connector with no redemption path. The result is a permanent, protocol-level freeze of every user's bridged ETH.

---

### Finding Description

`return_promise` is the single dispatch helper used by every NEP-141 connector-facing method in the engine:


`get_connector_account_id` reads the account ID from a single storage slot:

```

```

`set_eth_connector_contract_account` unconditionally overwrites that slot:

```
// engine/src/contract_methods/connector.rs:418-438
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();
        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }
        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;
        set_connector_account_id(io, &args.account);          // ← no supply check
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
        Ok(())
    })
}
```

There is no guard that verifies the old connector's NEP-141 total supply is zero before the pointer is moved.

After the pointer is moved:

1. **`withdraw`** (line 58) calls `return_promise(…, "engine_withdraw", …)` → dispatched to the **new** connector, which has no record of any user's balance → every withdrawal fails.
2. **`ft_transfer` / `ft_transfer_call`** (lines 264, 285) are similarly routed to the new connector.
3. **`ft_on_transfer`** (line 81) uses `get_connector_account_id` to decide whether an incoming `ft_transfer_call` is a base-ETH deposit. After the change, a transfer from the **old** connector is no longer recognized as a base-token deposit and is instead treated as an unknown ERC-20, breaking the deposit path as well.
4. **`ExitToNear` precompile** (native.rs line 431) calls `get_eth_connector_contract_account` to build the `ft_transfer` promise for base-ETH exits; it now targets the new connector, which holds no ETH.

All NEP-141 ETH balances that existed on the old connector are permanently inaccessible through any protocol-provided path.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Every user who deposited ETH into Aurora (receiving a NEP-141 balance on the old connector) loses the ability to withdraw or transfer those tokens. The old connector still holds the underlying ETH, but the engine no longer routes any call to it. There is no migration or redemption function in the engine for "previous connector" balances.

---

### Likelihood Explanation

The function is callable by the contract owner (a single key or multisig) without any time-lock or supply guard. A legitimate connector upgrade (e.g., deploying a new eth-connector contract) would naturally trigger this call. Without an explicit protocol requirement to drain the old connector first, the freeze can occur during any routine upgrade. The absence of a supply check makes this a latent risk in every future connector migration.

---

### Recommendation

1. Before overwriting the connector account ID, read the old connector's `ft_total_supply` and reject the call if it is non-zero.
2. Alternatively, implement a two-phase migration: mark the old connector as "draining" (deposits disabled, withdrawals still routed to it), and only allow the pointer swap once supply reaches zero.
3. At minimum, emit an on-chain event with the old account ID and its last-known supply so that off-chain tooling can detect a dangerous transition.

---

### Proof of Concept

**State before:**
- Old connector `old-eth-connector.near` holds 1 000 ETH worth of NEP-141 supply.
- User Alice has 10 ETH balance on `old-eth-connector.near`.

**Attack / upgrade sequence:**
1. Owner calls `set_eth_connector_contract_account` with `account = "new-eth-connector.near"`.
   - `set_connector_account_id` overwrites the storage slot. No supply check is performed.

**State after:**
2. Alice calls `engine.withdraw(recipient_eth_address, 10 ETH)`.
   - `withdraw` → `return_promise(…, "engine_withdraw", …)`.
   - `get_connector_account_id` returns `"new-eth-connector.near"`.
   - `engine_withdraw` is called on `new-eth-connector.near`, which has no record of Alice → call fails.
3. Alice calls `engine.ft_transfer(…)` → same failure.
4. Alice calls `engine.ft_balance_of(alice)` → routed to new connector → returns 0 (her balance is invisible).
5. Alice's 10 ETH is permanently locked in `old-eth-connector.near` with no engine-provided redemption path.
