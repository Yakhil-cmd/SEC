### Title
`set_eth_connector_contract_account` Can Be Called While ETH Tokens Are Still in Circulation, Permanently Freezing User Funds - (`engine/src/contract_methods/connector.rs`)

---

### Summary

The `set_eth_connector_contract_account` function allows the contract owner to overwrite the stored ETH connector account ID at any time, with no check on whether ETH tokens are currently in circulation. The `ExitToEthereum` precompile and the `withdraw` path both resolve the connector account dynamically from storage at call time. If the owner changes the connector account while users hold bridged ETH on Aurora, all subsequent ETH withdrawals will route to the wrong (or non-existent) account and fail permanently, freezing user funds.

---

### Finding Description

`set_eth_connector_contract_account` in `engine/src/contract_methods/connector.rs` unconditionally overwrites the stored connector account ID: [1](#0-0) 

It calls `set_connector_account_id`, which blindly writes the new account to storage: [2](#0-1) 

There is no check on the current total ETH supply or any other circulation guard before overwriting.

The `ExitToEthereum` precompile (flag `0x0`, base ETH transfer) reads this stored account at execution time via `get_eth_connector_contract_account` and routes the withdrawal call to it: [3](#0-2) 

Similarly, the NEAR-level `withdraw` function in `engine/src/lib.rs` delegates to `contract_methods::connector::withdraw`, which also forwards to the connector account via `return_promise`: [4](#0-3) 

The `BijectionMap` backing the NEP-141↔ERC-20 registry (`nep141_erc20_map`) has no `remove` operation, so ERC-20 exit paths are unaffected. The vulnerability is specific to the base ETH (bridged ETH) withdrawal path, which is entirely dependent on the stored connector account ID. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any user who has bridged ETH to Aurora (ETH locked in the eth-connector contract, credited as EVM balance on Aurora) and has not yet withdrawn will be permanently unable to withdraw if the owner changes the connector account to a different or non-existent account. The EVM balance remains in the engine's state, but the withdrawal promise will be dispatched to the wrong NEAR account and fail. There is no recovery path for the user; only the owner can restore the correct connector account, and even then only if they notice the mistake.

---

### Likelihood Explanation

The owner has a legitimate operational reason to call `set_eth_connector_contract_account`: upgrading or migrating the eth-connector contract. The function is explicitly exposed as a public owner-callable method and is used in the Silo deployment flow (e.g., `set_eth_connector_contract_account` is called during silo setup in tests). An owner performing a connector upgrade without first verifying that the ETH supply on Aurora is zero would silently break all pending and future withdrawals. No warning, no guard, and no revert exists to prevent this. [6](#0-5) 

---

### Recommendation

Before overwriting the connector account, check that the total ETH supply currently held on Aurora is zero. Concretely, read the ETH total supply (sum of all EVM balances) and revert if it is non-zero. Alternatively, implement a two-phase migration: first pause deposits and withdrawals, wait for all in-flight exits to settle, verify zero supply, then switch the connector account.

---

### Proof of Concept

1. User calls `ft_transfer_call` on the eth-connector, bridging `N` ETH to Aurora. The eth-connector locks the ETH and calls `ft_on_transfer` on the engine, which credits the user's EVM address with `N` ETH.
2. Owner calls `set_eth_connector_contract_account` with a new account ID (e.g., `new-connector.near`) as part of a connector upgrade, without checking the current ETH supply. [7](#0-6) 
3. User calls `withdraw` (or triggers the `ExitToEthereum` precompile with flag `0x0`) to retrieve their ETH.
4. The engine reads the connector account from storage — now `new-connector.near` — and dispatches the withdrawal promise to it. [8](#0-7) 
5. `new-connector.near` does not implement `engine_withdraw` / `withdraw`, so the promise fails. The user's EVM balance was already debited (burned) before the promise was dispatched, and there is no refund path for the base ETH exit.
6. The user's ETH is permanently frozen: gone from the EVM state, not recoverable from the connector.

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,

    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}
