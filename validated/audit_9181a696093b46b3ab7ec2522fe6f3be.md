Audit Report

## Title
Off-by-One Epoch in SNS `distribute_rewards` Causes Systematic Under-Rewarding of Neuron Holders - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS `distribute_rewards` function computes the reward rate for each round using the **end** of the round rather than the **beginning**. Because `reward_rate_at` is monotonically decreasing during the transition period, every round's contribution to `rewards_purse_e8s` is understated. The shortfall is permanent — the unissued maturity is never recovered — and affects every SNS instance with a non-zero `reward_rate_transition_duration_seconds`.

## Finding Description
In `rs/sns/governance/src/governance.rs` lines 5861–5872, the reward accumulation loop is:

```rust
for i in 1..=new_rounds_count {
    let seconds_since_genesis = round_duration_seconds
        .saturating_mul(i)                    // ← END of round i
        .saturating_add(reward_start_timestamp_seconds)
        .saturating_sub(self.proto.genesis_timestamp_seconds);

    let current_reward_rate = voting_rewards_parameters.reward_rate_at(
        crate::reward::Instant::from_seconds_since_genesis(i2d(seconds_since_genesis)),
    );
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
```

For round `i`, the timestamp fed to `reward_rate_at` is `reward_start + i × round_duration − genesis`, which is the **end** of round `i` (equivalently, the start of round `i+1`). The correct timestamp is `reward_start + (i−1) × round_duration − genesis` (the **beginning** of round `i`).

The NNS parallel implementation in `rs/nns/governance/src/reward/calculation.rs` lines 82–84 explicitly documents: *"we don't integrate the rate here. Instead we multiply the rate at the beginning of that day with the considered duration."* The SNS implementation diverges from this documented design intent.

`reward_rate_at` in `rs/sns/governance/src/reward.rs` lines 219–241 computes a quadratically decreasing rate from `initial_reward_rate` to `final_reward_rate` over `reward_rate_transition_duration_seconds`. The test `reward_is_convex_and_decreasing` (lines 576–590) asserts `previous > current` for every consecutive pair of rounds, confirming strict monotonic decrease. Therefore `rate(end_of_round_i) < rate(start_of_round_i)` for every round in the transition window, and every round's purse contribution is understated.

The under-computed purse is then distributed proportionally to neuron voting shares at lines 5973–5975, so every voting neuron holder receives less maturity than entitled.

## Impact Explanation
Every SNS neuron holder who participates in governance voting receives permanently less maturity than the protocol intends to issue. The total cumulative shortfall across the full transition telescopes to:

```
(initial_rate − final_rate) × round_duration × token_supply
```

For a large SNS (e.g., 1 billion tokens, `initial = 200 bp`, `final = 100 bp`, weekly rounds), this equals approximately one full round's worth of the rate-spread reward — a non-trivial and irrecoverable loss distributed across all voting neuron holders. The lost maturity is simply not minted; no party gains it. This constitutes a **significant SNS security impact with concrete, permanent user harm**, qualifying as **High** severity.

## Likelihood Explanation
This code path executes automatically on every SNS governance canister via `run_periodic_tasks` whenever `new_rounds_count > 0`. No privileged action is required. Any SNS instance with `reward_rate_transition_duration_seconds > 0` and `initial_reward_rate > final_reward_rate` is affected for the entire duration of its transition window. The condition is met by the default/typical SNS configuration.

## Recommendation
Change `saturating_mul(i)` to `saturating_mul(i.saturating_sub(1))` (or equivalently `i - 1` since `i >= 1`) to use the rate at the **beginning** of round `i`:

```rust
for i in 1..=new_rounds_count {
    let seconds_since_genesis = round_duration_seconds
        .saturating_mul(i - 1)   // rate at the START of round i
        .saturating_add(reward_start_timestamp_seconds)
        .saturating_sub(self.proto.genesis_timestamp_seconds);

    let current_reward_rate = voting_rewards_parameters.reward_rate_at(
        crate::reward::Instant::from_seconds_since_genesis(i2d(seconds_since_genesis)),
    );
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
```

This aligns the SNS implementation with the NNS design intent documented in `rs/nns/governance/src/reward/calculation.rs`.

## Proof of Concept
A deterministic unit test suffices. Configure `VotingRewardsParameters` with `initial_reward_rate_basis_points = 200`, `final_reward_rate_basis_points = 100`, `reward_rate_transition_duration_seconds = 42 × 7 × 86400`, `round_duration_seconds = 7 × 86400`. Set `genesis_timestamp_seconds = 0`, `reward_start_timestamp_seconds = 0`, `new_rounds_count = 1`, `supply = 1_000_000_000_00` (1 billion tokens in e8s).

- **Correct** purse (using `i-1=0`): `rate_at(0) × round_duration × supply` = `200bp/yr × (7/365.25) × 1e17`
- **Actual** purse (using `i=1`): `rate_at(604800s) × round_duration × supply` ≈ `195.3bp/yr × (7/365.25) × 1e17`
- **Shortfall**: ≈ `4.7bp/yr × (7/365.25) × 1e17` ≈ 898,000 e8s (~8.98 tokens) per round, accumulating to ~(initial−final) × round_duration × supply over the full 42-round transition.

The existing test `reward_is_convex_and_decreasing` in `rs/sns/governance/src/reward.rs` already proves the rate is strictly decreasing, making the shortfall direction deterministic and reproducible.
