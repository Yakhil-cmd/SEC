Audit Report

## Title
Existing SNS Swap Participant Incorrectly Blocked from Increasing Contribution at Maximum Capacity - (File: rs/sns/swap/src/swap.rs)

## Summary
In `refresh_buyer_token_e8s`, the maximum-participants capacity guard at lines 1179–1198 unconditionally applies a `+1` to `self.buyers.len()` without first checking whether the caller is already a participant. When the swap is exactly at capacity, an existing buyer who transfers additional ICP and calls `refresh_buyer_token_e8s` receives an error that explicitly promises "existing participants may still increase their ICP participation amount" — a guarantee the code itself breaks. The transferred ICP is locked in the subaccount until the swap closes.

## Finding Description
`refresh_buyer_token_e8s` enforces a hard cap on direct participants to ensure enough SNS neurons can be minted:

```rust
// rs/sns/swap/src/swap.rs L1179-1198
let num_direct_participants = self.buyers.len() as u64;
if (num_direct_participants + 1) * num_sns_neurons_per_basket > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS {
    return Err(format!("… existing participants may still increase their ICP participation amount …"));
}
```

`self.buyers.len()` counts all current buyers including the caller if already present. The `+1` is applied unconditionally, treating an existing buyer as a new entrant. The actual membership check only appears ~77 lines later:

```rust
// rs/sns/swap/src/swap.rs L1275-1276
let is_preexisting_buyer = self.buyers.contains_key(&buyer.to_string());
if !is_preexisting_buyer { … }
```

Because the capacity guard returns early before this check, an existing participant's top-up is rejected even though no new participant slot would be consumed. The error message compounds the issue by explicitly promising the opposite behavior.

## Impact Explanation
This matches the allowed High impact: "Significant SNS security impact with concrete user or protocol harm." An existing participant who transfers ICP to their swap subaccount has that ICP locked until the swap closes, losing the opportunity to increase their stake. The error message constitutes a broken protocol guarantee. The ICP is recoverable only via `error_refund_icp` post-swap, not during the active swap period.

## Likelihood Explanation
Any SNS swap that reaches the maximum number of direct participants (a normal, expected condition for popular launches) will trigger this for every existing participant who attempts a top-up. The call path is fully unprivileged — any principal can invoke `refresh_buyer_token_e8s`. No special conditions beyond swap-at-capacity are required.

## Recommendation
Move the `is_preexisting_buyer` check before the capacity guard and skip the guard for existing participants:

```rust
let is_preexisting_buyer = self.buyers.contains_key(&buyer.to_string());
if !is_preexisting_buyer {
    let num_direct_participants = self.buyers.len() as u64;
    let num_sns_neurons_per_basket = params
        .neuron_basket_construction_parameters
        .as_ref()
        .expect("neuron_basket_construction_parameters must be specified")
        .count;
    if (num_direct_participants + 1) * num_sns_neurons_per_basket > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS {
        return Err(format!("The swap has reached the maximum number of direct participants …"));
    }
}
```

The `is_preexisting_buyer` variable computed here can then be reused at line 1275, eliminating the redundant `contains_key` call.

## Proof of Concept
1. Deploy an SNS swap with `neuron_basket_construction_parameters.count = 1` and `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS = N`.
2. Fill the swap with exactly `N` distinct buyers, each contributing `min_participant_icp_e8s`.
3. As buyer #1 (already in `self.buyers`), transfer an additional `min_participant_icp_e8s` to the swap subaccount.
4. Call `refresh_buyer_token_e8s` as buyer #1.
5. **Observed**: returns `Err("The swap has reached the maximum number of direct participants (N) and does not accept new participants; existing participants may still increase their ICP participation amount.")`.
6. **Expected**: call succeeds and buyer #1's balance is increased, since no new participant slot is consumed.

The ICP transferred in step 3 is locked in the subaccount until the swap closes, at which point the buyer must call `error_refund_icp` to recover it.
