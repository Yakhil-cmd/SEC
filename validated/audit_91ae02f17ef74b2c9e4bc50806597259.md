### Title
Incorrect ICRC-106 Standard URL and Unconditional Advertisement Without Index Principal - (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

### Summary
The ICRC-1 ledger's `supported_standards()` function unconditionally advertises ICRC-106 support regardless of whether an index principal is configured, and uses a GitHub pull-request URL (`https://github.com/dfinity/ICRC/pull/106`) instead of a proper specification document URL. This is the direct IC analog of the ERC1155 metadata URI bug: a canister claims to implement a standard but the implementation is incorrect or incomplete, breaking off-chain integrations that rely on the advertisement.

### Finding Description

In `rs/ledger_suite/icrc1/ledger/src/main.rs`, the `supported_standards()` function (exposed as `icrc1_supported_standards` and `icrc10_supported_standards`) unconditionally includes ICRC-106 in the returned list: [1](#0-0) 

The URL provided is `https://github.com/dfinity/ICRC/pull/106` — a GitHub pull-request page, not a specification document. Every other standard in the same list uses a proper document URL: [2](#0-1) 

Meanwhile, the actual `icrc106_get_index_principal` endpoint returns `Err(Icrc106Error::IndexPrincipalNotSet)` whenever no index principal has been configured: [3](#0-2) 

By contrast, ICRC-152 is correctly gated behind a feature flag check before being advertised: [4](#0-3) 

There are two distinct defects:

1. **Wrong URL**: The ICRC-106 entry points to a PR (`/pull/106`) rather than a specification document. Off-chain tooling that follows the URL to understand the standard will reach a PR page, not a spec.
2. **Unconditional advertisement**: ICRC-106 is always listed even when `icrc106_get_index_principal` will return an error, because no index principal was set at init/upgrade time. The `icrc1_supported_standards` response is a public, query-callable signal that wallets, explorers, and aggregators use to discover capabilities.

### Impact Explanation

Any unprivileged caller (wallet, DEX, explorer, aggregator) that:
1. Calls `icrc1_supported_standards` and observes ICRC-106 listed,
2. Follows the URL to read the specification — and reaches a PR page instead of a spec,
3. Calls `icrc106_get_index_principal` to discover the associated index canister,
4. Receives `Err(Icrc106Error::IndexPrincipalNotSet)` instead of a valid principal,

…will fail to integrate correctly. Integrations that treat the supported-standards list as authoritative (as ICRC-10 intends) will either crash, silently skip index-based features, or display incorrect information to end users. The `icrc1_metadata` endpoint also exposes `icrc106:index_principal` as a metadata key, compounding the inconsistency for metadata-reading clients. [5](#0-4) 

### Likelihood Explanation

The `icrc1_supported_standards` endpoint is a standard, publicly queryable, zero-argument query call. Every ICRC-1-aware off-chain integration is expected to call it. Any ledger deployed without an index principal (the common case for standalone deployments) will exhibit both defects on every query. The entry path requires no privileges, no special state, and no timing — it is always reachable.

### Recommendation

1. **Fix the ICRC-106 URL** to point to the actual specification document (e.g., `https://github.com/dfinity/ICRC/blob/main/ICRCs/ICRC-106/ICRC-106.md`) instead of the PR URL.
2. **Gate ICRC-106 advertisement** on whether an index principal is actually configured, mirroring the pattern already used for ICRC-152:
   ```rust
   if Access::with_ledger(|ledger| ledger.index_principal().is_some()) {
       standards.push(StandardRecord {
           name: "ICRC-106".to_string(),
           url: "https://github.com/dfinity/ICRC/blob/main/ICRCs/ICRC-106/ICRC-106.md".to_string(),
       });
   }
   ```

### Proof of Concept

1. Deploy the ICRC-1 ledger (`rs/ledger_suite/icrc1/ledger/`) without setting an index principal in the init args.
2. Call `icrc1_supported_standards` (query, no authentication required) → response includes `{ name: "ICRC-106", url: "https://github.com/dfinity/ICRC/pull/106" }`.
3. Navigate to the URL → GitHub PR page, not a specification document.
4. Call `icrc106_get_index_principal` (query, no authentication required) → returns `Err(IndexPrincipalNotSet)`.
5. An off-chain integration that trusted the supported-standards advertisement now has no valid index principal and cannot use index-based features, despite the canister claiming full ICRC-106 compliance. [6](#0-5) [3](#0-2)
