Audit Report

## Title
Unprivileged Caller Can Force Any Principal With ICP in Swap Subaccount Into SNS Token Swap Participation - (File: rs/sns/swap/canister/canister.rs)

## Summary
`refresh_buyer_tokens` in `rs/sns/swap/canister/canister.rs` accepts a caller-supplied `buyer` principal with no check that the caller matches the specified principal. Any unprivileged ingress sender can invoke this method with a victim's principal ID, causing the victim's ICP already held in the swap's derived subaccount to be irrevocably registered as swap participation — locking the victim's funds until the swap finalizes and potentially exchanging their ICP for SNS tokens without their consent.

## Finding Description
In `rs/sns/swap/canister/canister.rs` at L130–134, the effective buyer is resolved from the caller-supplied `arg.buyer` string with no authorization check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // any principal accepted
};
```

This `p` is passed directly to `refresh_buyer_token_e8s` (L137), which at L1152–1163 reads the ICP balance from `principal_to_subaccount(&buyer)` on the swap canister and, if it meets the minimum threshold, registers `buyer` as a committed participant (L1285–1291).

The ticket system at L1250–1272 is not a mitigation: the entire block is guarded by `if let Some(ticket_sns_sale_canister) = memory::OPEN_TICKETS_MEMORY.with(...)`, and the inline comment explicitly states *"If there exists no ticket for the buyer, the payment flow will simply ignore the ticket."* A victim who transferred ICP directly (without going through the ticket flow) has no ticket, so the check is skipped entirely.

The `confirmation_text` field is not a meaningful mitigation: it is stored in the swap's `Init` payload and is publicly readable via the `get_init` query endpoint. An attacker reads it and includes it verbatim.

Once registered, the victim cannot reclaim their ICP via `error_refund_icp` while the swap is `OPEN` — that endpoint is gated at L1932–1935 on the swap being `ABORTED` or `COMMITTED`.

## Impact Explanation
This is a **High** severity finding matching "Unauthorized access to … ledgers, or canister-controlled funds." A victim's ICP is committed to a financial outcome (SNS token purchase) without their authorization. If the swap commits, the victim receives SNS tokens — potentially of negligible market value — in exchange for their ICP, with no consent. If the swap aborts, the victim recovers their ICP only after the swap closes (potentially days later), minus transfer fees. The victim has no recourse while the swap is `OPEN`.

## Likelihood Explanation
- The attack requires only a standard ingress call from any unprivileged principal — no special role, key, or majority is needed.
- The victim's precondition (ICP sitting in the swap subaccount) is a normal intermediate state every participant passes through; the window is non-trivial.
- The `confirmation_text` (when present) is publicly readable from `get_init`, providing no barrier.
- The attack is cheap (one update call) and repeatable across all open SNS swaps simultaneously.

## Recommendation
In `refresh_buyer_tokens`, reject any request where `arg.buyer` is non-empty and does not match `caller_principal_id()`:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let requested = PrincipalId::from_str(&arg.buyer).unwrap();
        if requested != caller {
            panic!("caller {} is not authorized to refresh tokens on behalf of {}", caller, requested);
        }
        requested
    };
    // ...
}
```

Alternatively, remove the `buyer` field entirely and always use `caller_principal_id()`, since the only legitimate use case is a principal notifying the swap of their own transfer.

## Proof of Concept
**Setup:**
- SNS swap is `OPEN`, no `confirmation_text` (or attacker reads it from `get_init`).
- Victim `V` transfers 10 ICP to the swap canister's subaccount `principal_to_subaccount(V)` on the ICP ledger.
- Victim `V` has no open ticket (transferred directly, bypassing the ticket flow).

**Attack (single ingress call from attacker `A`):**
```
dfx canister call <swap_canister_id> refresh_buyer_tokens '(
  record {
    buyer = "<victim_principal_text_id>";
    confirmation_text = null
  }
)'
```

**Result:**
- `refresh_buyer_token_e8s` reads the balance of `principal_to_subaccount(V)`, finds 10 ICP, skips the ticket check (no ticket exists), and registers `V` as a buyer with `amount_icp_e8s = 10_0000_0000`.
- `V` is now a committed participant. Their ICP is locked until the swap finalizes.
- `V` cannot call `error_refund_icp` while the swap is `OPEN` (blocked at L1932–1935).
- If the swap commits, `V` receives SNS tokens they never agreed to purchase.

A deterministic integration test or PocketIC test can reproduce this by: (1) opening a swap, (2) transferring ICP from principal `V` to the swap subaccount, (3) calling `refresh_buyer_tokens` from a distinct principal `A` with `buyer = V`, and (4) asserting `V` appears in `self.buyers` with the transferred amount.
