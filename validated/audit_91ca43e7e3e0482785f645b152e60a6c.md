### Title
Fee-on-Transfer NEP-141 Deposit Mints Inflated ERC-20 Balance, Causing Insolvency for Last Withdrawers - (File: `engine/src/engine.rs`)

---

### Summary

Aurora Engine's `receive_erc20_tokens` function mints ERC-20 tokens equal to `args.amount` — the amount declared in the `ft_on_transfer` callback — without accounting for any transfer fee that a NEP-141 token may deduct during the transfer. Because `deploy_erc20_token` is permissionless, any user can register a fee-on-transfer (FoT) NEP-141 token as an Aurora ERC-20 mirror. Every deposit then inflates the ERC-20 supply beyond the actual NEP-141 balance held by Aurora Engine, making the last withdrawer(s) permanently unable to exit to NEAR.

---

### Finding Description

The deposit flow for NEP-141 ERC-20 mirrors is:

1. A user calls `ft_transfer_call` on a NEP-141 contract targeting Aurora Engine.
2. The NEP-141 contract credits Aurora Engine's internal balance with `amount - fee` and then calls `ft_on_transfer(sender, amount, msg)` on Aurora Engine.
3. Aurora Engine's `ft_on_transfer` dispatches to `receive_erc20_tokens`.

Inside `receive_erc20_tokens`:

```rust
// engine/src/engine.rs  line 803
let amount = args.amount.as_u128();   // ← declared amount, NOT actual received amount
```

This `amount` is then passed directly to the ERC-20 mint call:

```rust
// engine/src/engine.rs  lines 826-837
let result = self
    .call(
        &erc20_admin_address,
        &erc20_token,
        Wei::zero(),
        setup_receive_erc20_tokens_input(&recipient, amount),  // mints `amount`
        ...
    )
```

For a FoT NEP-141 token with a 1 % fee, a deposit of 1 000 tokens causes:
- Aurora Engine's NEP-141 balance: **+990 tokens**
- ERC-20 tokens minted in the EVM: **+1 000 tokens**

The discrepancy of 10 tokens per deposit accumulates. When users later call `withdrawToNear` / `ExitToNear`, Aurora Engine issues `ft_transfer` on the NEP-141 contract for the full ERC-20 amount burned. The last withdrawer(s) will find that Aurora Engine's NEP-141 balance is insufficient, and their exit will revert — permanently freezing their funds.

The same arithmetic flaw exists in `receive_base_tokens` for the ETH/wNEAR base token path:

```rust
// engine/src/engine.rs  line 778
let amount = Wei::new_u128(args.amount.as_u128());  // ← same pattern
```

Registration of a new ERC-20 mirror via `deploy_erc20_token` carries no access-control guard, so any unprivileged user can register a FoT token and begin depositing.

---

### Impact Explanation

Every deposit of a FoT NEP-141 token over-mints ERC-20 tokens relative to the actual NEP-141 balance held by Aurora Engine. The aggregate shortfall grows with each deposit. When the pool of depositors begins withdrawing, withdrawals succeed in FIFO order until the NEP-141 balance is exhausted; the last withdrawer(s) receive a failed `ft_transfer` and their ERC-20 tokens are burned with no corresponding NEP-141 release — **permanent fund freeze / insolvency** for those users.

Impact classification: **Critical — permanent freezing of funds / insolvency**.

---

### Likelihood Explanation

- `deploy_erc20_token` is callable by any account with no permission check. [1](#0-0) 
- `ft_on_transfer` is the standard NEP-141 callback; any NEP-141 token that charges a transfer fee triggers the bug automatically. [2](#0-1) 
- No special privileges, governance capture, or oracle manipulation are required.
- The attacker only needs to deploy a FoT NEP-141 token (standard NEAR contract), register it, and deposit — all permissionless on-chain actions.

Likelihood: **High**.

---

### Recommendation

After the NEP-141 transfer completes, measure the actual balance change rather than trusting `args.amount`. The standard pattern is:

1. Read Aurora Engine's NEP-141 balance **before** the transfer (or store it in state).
2. After `ft_on_transfer` is invoked, compute `actual_received = balance_after - balance_before`.
3. Use `actual_received` — not `args.amount` — as the quantity to mint.

Alternatively, reject deposits from NEP-141 tokens whose `ft_on_transfer` callback reports a received amount that differs from the declared amount (i.e., return `args.amount` instead of `0` to signal refusal), and document that FoT tokens are unsupported.

---

### Proof of Concept

**Setup**

1. Deploy a NEP-141 token `fot.near` that deducts a 1 % fee on every `ft_transfer` / `ft_transfer_call`.
2. Call `deploy_erc20_token` on Aurora Engine for `fot.near` — succeeds with no access control. [3](#0-2) 

**Deposit (inflate supply)**

3. Call `fot.near::ft_transfer_call({ receiver_id: "aurora", amount: "1000", msg: "<evm_address>" })`.
4. `fot.near` deducts 1 % fee → credits Aurora Engine with **990** tokens, then calls `aurora::ft_on_transfer(sender, "1000", msg)`.
5. Aurora Engine executes `receive_erc20_tokens` with `amount = 1000` and mints **1 000** ERC-20 tokens to `<evm_address>`. [4](#0-3) 

**Withdraw (drain)**

6. Two users each hold 500 ERC-20 tokens. User A calls `withdrawToNear(500)` → Aurora Engine calls `fot.near::ft_transfer(A, 500)` → succeeds (990 − 500 = 490 remaining).
7. User B calls `withdrawToNear(500)` → Aurora Engine calls `fot.near::ft_transfer(B, 500)` → **fails** (only 490 available). User B's 500 ERC-20 tokens are burned but no NEP-141 tokens are released — funds permanently frozen.

The shortfall (10 tokens per 1 000 deposited) accumulates linearly with each deposit, making the insolvency proportional to total deposit volume.

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
