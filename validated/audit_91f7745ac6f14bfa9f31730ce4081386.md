### Title
Boundary Node Query Cache Key Omits `request_type`, Enabling Cross-Version Cache Poisoning — (File: `rs/boundary_node/ic_boundary/src/routes.rs`)

### Summary

The `RequestContext` struct used as the cache lookup key for the boundary node's query response cache deliberately omits the `request_type` field from both its `Hash` and `PartialEq` implementations. Because `QueryV2` and `QueryV3` are both considered cacheable query types, a response cached for one API version is served to callers of the other API version when all other key fields match. The V2 and V3 query endpoints return structurally different NNS delegation certificate formats, so a V3 client that receives a cached V2 response will fail certificate verification or receive a malformed delegation.

### Finding Description

`RequestContext` is the cache key type used by `KeyExtractorContext` in the boundary node's `cache_middleware`. [1](#0-0) 

The `Hash` and `PartialEq` implementations for `RequestContext` hash/compare only `canister_id`, `sender`, `method_name`, `ingress_expiry`, and `arg` (or `http_request`). The `request_type` field is explicitly absent from both: [2](#0-1) 

The `BypasserIC` bypasses caching only for non-query requests, requests carrying a nonce, and (optionally) non-anonymous requests. All three query variants — `QueryV2`, `QueryV3`, and `QuerySubnetV3` — pass the bypasser and are eligible for caching: [3](#0-2) [4](#0-3) 

The two query API versions are served at distinct URL paths: [5](#0-4) 

They produce structurally different responses. The `Version` enum in the replica's query handler controls the NNS delegation format embedded in the signed query response — V2 uses a flat canister-range format and V3 uses a tree format: [6](#0-5) 

Because `request_type` is absent from the cache key, a `QueryV2` response cached for `(canister_id=C, sender=anon, method=M, ingress_expiry=T, arg=A)` is returned verbatim to a subsequent `QueryV3` request with the same tuple, and vice versa.

Additionally, for anonymous senders the ingress validator skips `ingress_expiry` validation entirely: [7](#0-6) 

This means an attacker can freely choose any `ingress_expiry` value (including `0`) for an anonymous `QueryV2` request, making it trivial to match the cache key of a victim's anonymous `QueryV3` request that uses the same fixed or predictable expiry.

### Impact Explanation

A boundary node operator who enables the query cache (via `--cache-size`) exposes all anonymous query callers to cross-version cache poisoning. A V3 client that receives a V2-formatted response will fail to parse or verify the NNS delegation certificate, causing the query to appear to fail. This is a **boundary/API integrity and availability** impact: legitimate V3 callers are denied correct responses for the duration of the cache TTL (default 1 second, configurable). [8](#0-7) 

### Likelihood Explanation

The cache TTL defaults to 1 second. For the poisoning to succeed, the attacker's `QueryV2` request must be cached before the victim's `QueryV3` request arrives, and both must share the same `(canister_id, sender, method_name, ingress_expiry, arg)` tuple. For anonymous requests, `ingress_expiry` is not validated, so the attacker can set it to `0` or any fixed value. Popular canisters queried with fixed arguments (e.g., token balance queries with a fixed `arg`) are the most exposed. No privileged access is required; any unprivileged boundary node user can trigger this.

### Recommendation

Include `request_type` in both the `Hash` and `PartialEq` implementations of `RequestContext` in `rs/boundary_node/ic_boundary/src/routes.rs`:

```rust
impl Hash for RequestContext {
    fn hash<H: Hasher>(&self, state: &mut H) {
        self.request_type.hash(state);  // add this line
        self.canister_id.hash(state);
        self.sender.hash(state);
        self.method_name.hash(state);
        self.ingress_expiry.hash(state);
        if self.http_request.is_some() {
            self.http_request.hash(state);
        } else {
            self.arg.hash(state);
        }
    }
}

impl PartialEq for RequestContext {
    fn eq(&self, other: &Self) -> bool {
        let r = self.request_type == other.request_type  // add this field
            && self.canister_id == other.canister_id
            && self.sender == other.sender
            && self.method_name == other.method_name
            && self.ingress_expiry == other.ingress_expiry;
        if self.http_request.is_some() {
            r && self.http_request == other.http_request
        } else {
            r && self.arg == other.arg
        }
    }
}
```

### Proof of Concept

1. Enable the boundary node query cache: `--cache-size=100MB`.
2. Send an anonymous `QueryV2` POST to `/api/v2/canister/<C>/query` with body `{canister_id: C, method_name: "balance", arg: [], sender: anon, ingress_expiry: 0, nonce: null}`. The response (V2 flat-delegation format) is stored in the cache.
3. Within the cache TTL window (≤1 s), send an anonymous `QueryV3` POST to `/api/v3/canister/<C>/query` with the identical CBOR body.
4. The boundary node returns the cached V2 response. The V3 client's certificate-delegation parser receives a flat-format delegation instead of the expected tree-format delegation, causing verification failure. [9](#0-8) [4](#0-3)
