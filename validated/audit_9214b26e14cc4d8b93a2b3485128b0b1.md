### Title
`set_eth_connector_contract_account` Overwrites Connector Reference Without Draining Previous Connector, Permanently Freezing Bridged ETH - (File: engine/src/contract_methods/connector.rs)

---

### Summary

`set_eth_connector_contract_account` unconditionally overwrites the stored ETH connector account ID. Every subsequent user-facing withdrawal and transfer operation is then routed exclusively to the new connector. Any bridged ETH still held in the old connector becomes permanently inaccessible through the Aurora Engine interface, freezing those user funds with no recovery path inside the protocol.

---

### Finding Description

The ETH connector is an external NEAR contract that custodies all bridged ETH (as NEP-141 tokens). The Aurora Engine stores a single connector account ID in its own storage under the key `EthConnectorStorageId::EthConnectorAccount`. Every user-facing operation that touches the connector — `withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_unregister`, `storage_withdraw`, `ft_metadata`, `ft_balance_of`, and `ft_total_eth_supply_on_near` — is dispatched through the shared helper `return_promise`, which reads this single stored account ID at call time: [1](#0-0) 

The setter `set_eth_connector_contract_account` (callable by the owner or via a private call) replaces this stored ID atomically with no precondition check: [2](#0-1) 

The underlying write is a plain storage overwrite: [3](#0-2) 

After the overwrite, `get_connector_account_id` returns the new account for every subsequent call: [4](#0-3) 

Additionally, `ft_on_transfer` uses the same stored ID to decide whether an incoming NEP-141 transfer is a base-token deposit from the connector: [5](#0-4) 

Once the ID is replaced, the old connector can no longer trigger `receive_base_tokens`, and no engine method can route a call back to it. There is no alternative code path, no "previous connector" field, and no migration helper.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

All bridged ETH held in the old connector at the moment of the setter call becomes permanently inaccessible through the Aurora Engine. Users cannot `withdraw`, `ft_transfer`, or perform any other operation against the old connector because every such call is unconditionally forwarded to the new connector account. The old connector contract still exists on NEAR and holds the tokens, but the engine has no mechanism to reach it. Recovery would require an out-of-band contract upgrade to the engine itself.

---

### Likelihood Explanation

The owner account (`require_owner_only`) or any account that can make a private call to the engine can invoke this setter. The most realistic trigger is a routine connector upgrade: the operator deploys a new connector contract and calls `set_eth_connector_contract_account` to point the engine at it, without first verifying that the old connector's token supply is zero. Because the engine exposes no view of the old connector's balance and no drain-first guard, this mistake is easy to make silently. The setter is exposed as a named NEAR entrypoint (`set_eth_connector_contract_account` in `lib.rs`): [6](#0-5) 

---

### Recommendation

Before overwriting the connector account ID, query the old connector's `ft_total_supply` (or equivalent) and revert if it is non-zero. Alternatively, implement a two-step migration: keep the old connector ID in a separate "legacy" storage slot and allow `withdraw` / `ft_transfer` to drain it while restricting new deposits to the new connector only. This mirrors the recommendation in the reference report: check that the previous pool has no remaining liquidity, or allow withdrawals from any previously registered connector while restricting new deposits to the current one.

---

### Proof of Concept

1. Users bridge ETH through the current connector (`old_connector.near`). The engine stores `old_connector.near` under `EthConnectorStorageId::EthConnectorAccount`.
2. The owner deploys a new connector (`new_connector.near`) and calls:
   ```
   set_eth_connector_contract_account({ account: "new_connector.near", withdraw_serialize_type: Borsh })
   ```
3. `set_connector_account_id` overwrites the stored key with `new_connector.near` — no balance check, no revert.
4. A user calls `withdraw` on the engine. `return_promise` reads `get_connector_account_id` → `new_connector.near` and dispatches `engine_withdraw` there. The user's balance on `old_connector.near` is never touched.
5. The user's bridged ETH remains locked in `old_connector.near` indefinitely. No engine entrypoint can route a call back to it. [7](#0-6) [8](#0-7)

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}

fn get_connector_account_id<I: IO>(io: &I) -> Result<AccountId, ContractError> {
    io.read_storage(&construct_contract_key(
        EthConnectorStorageId::EthConnectorAccount,
    ))
    .ok_or(errors::ERR_CONNECTOR_STORAGE_KEY_NOT_FOUND)
    .and_then(|x| {
        x.to_value()
            .map_err(|_| errors::ERR_BORSH_DESERIALIZE.as_bytes())
    })
    .map_err(Into::into)
}
