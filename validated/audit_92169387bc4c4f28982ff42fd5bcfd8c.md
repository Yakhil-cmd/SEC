Audit Report

## Title
Unbounded `inbound_connecting` JoinSet Enables Pre-Authentication Resource Exhaustion via QUIC Connection Flood — (`rs/p2p/quic_transport/src/connection_manager.rs`)

## Summary
The QUIC transport connection manager spawns an unbounded Tokio task into `inbound_connecting` for every accepted `Incoming` QUIC connection before any TLS or subnet-membership authentication occurs. An off-subnet attacker who can send UDP packets to the replica's transport port and complete QUIC address validation (Retry token exchange) can accumulate an unbounded number of tasks, each holding heap-allocated QUIC connection state for up to 10 seconds, exhausting heap memory or saturating the Tokio scheduler and blocking legitimate subnet peer connections.

## Finding Description
`inbound_connecting` is declared as a plain, unbounded `JoinSet` with no capacity limit at line 121. The event loop's `select!` branch at lines 354–362 accepts every `Incoming` from `endpoint.accept()` unconditionally and calls `handle_inbound_conn_attemp`. Inside `handle_inbound_conn_attemp` (lines 538–588), a task is unconditionally spawned into `inbound_connecting` at line 587 with no prior size check. All authentication — TLS handshake completion (`incoming.await`), certificate extraction, `node_id_from_certificate_der`, and the `peer_id > node_id` dialer check — executes inside the spawned task (lines 542–578), after the task is already in the JoinSet. The TLS server config is updated on topology change (lines 403–411) to restrict to subnet nodes, but this restriction is enforced during the TLS handshake inside the spawned task, not before spawning. The only call to `inbound_connecting.len()` in the entire module is at line 375, used exclusively for metrics collection, not as a guard. No `concurrent_connections`, `max_incoming`, or any other connection-count limit is set anywhere in the `rs/p2p/` tree. Quinn's `endpoint.accept()` yields an `Incoming` after QUIC address validation (Retry token exchange) but before the TLS handshake; an attacker with a real IP address can complete address validation without any subnet TLS certificate, causing `Incoming` objects to be yielded and tasks to be spawned. Each spawned task holds pending QUIC connection state and lives for up to `CONNECT_TIMEOUT = 10s` (line 82) before timing out.

## Impact Explanation
An attacker maintaining a sustained flood of QUIC Initial packets (each completing address validation) accumulates tasks in `inbound_connecting` faster than they drain. Each task holds heap-allocated QUIC connection state for up to 10 seconds. At sufficient rate this exhausts heap memory or saturates the Tokio task scheduler, preventing the connection manager's `select!` loop from scheduling legitimate inbound connection tasks from subnet peers. This directly stalls consensus peer connectivity on the affected replica, constituting an application/platform-level DoS and subnet availability impact not based on raw volumetric DDoS — matching the High ($2,000–$10,000) allowed impact: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
The replica's QUIC transport port is publicly reachable. The attacker requires no cryptographic material, no subnet membership, and no privileged access — only the ability to send UDP packets and respond to Quinn's Retry packets (standard QUIC address validation). This is a low-cost, repeatable attack: new tasks are spawned faster than they time out at 10 seconds, so a modest sustained packet rate maintains a large and growing JoinSet indefinitely. No victim mistake, social engineering, key compromise, or governance majority is required.

## Recommendation
1. **Cap `inbound_connecting` before spawning**: In `handle_inbound_conn_attemp`, check `self.inbound_connecting.len()` against a hard limit (e.g., `2 × subnet_size` or a fixed constant such as 1,000) before calling `self.inbound_connecting.spawn(...)`. If the limit is exceeded, call `incoming.refuse()` or drop the `Incoming` to reject the connection at zero task cost.
2. **Use `Incoming::refuse()`**: Quinn's `Incoming` type exposes a `refuse()` method that sends a QUIC `CONNECTION_CLOSE` frame without completing the handshake, cleanly rejecting excess connections.
3. **Per-IP rate limiting**: Track connection attempts per source IP in the event loop before spawning and drop `Incoming` objects from IPs exceeding a threshold.

## Proof of Concept
1. Identify the replica's QUIC transport port.
2. Open N UDP sockets to that port.
3. For each socket, send a valid QUIC Initial packet (any or no TLS certificate).
4. Respond to Quinn's Retry packet with the correct token to pass address validation.
5. `endpoint.accept()` yields `Incoming` → `handle_inbound_conn_attemp` spawns a task into `inbound_connecting`.
6. Repeat at rate R > (tasks completing per second ≈ N / 10s).
7. `inbound_connecting.len()` grows without bound.
8. At ~10,000 tasks: heap pressure causes OOM or Tokio scheduler starvation; legitimate subnet peer `Incoming` objects are accepted but their tasks are never scheduled → `CONNECT_TIMEOUT` fires → peers retry → consensus stalls.

A deterministic integration test can reproduce this locally: spawn a Quinn endpoint, flood it with address-validated `Incoming` connections from a non-subnet certificate, and assert that `inbound_connecting.len()` grows unboundedly while a legitimate subnet peer connection times out.
