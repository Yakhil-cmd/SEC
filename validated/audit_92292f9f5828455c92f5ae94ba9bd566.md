### Title
Single Node Operator Controlling ≥ N−f NNS Nodes Can Unilaterally Execute a Root Governance Upgrade — (`rs/nns/handlers/root/impl/src/root_proposals.rs`)

---

### Summary

The root proposal voting system for upgrading the NNS governance canister counts **ballot slots per node** rather than **per unique operator**. A single node operator who controls k nodes on the NNS subnet holds k ballot slots. Because `is_byzantine_majority_yes()` simply sums Yes-marked slots against the total slot count, one operator controlling ≥ N−f nodes can reach the Byzantine threshold alone — either at proposal submission time (auto-Yes) or with a single `vote_on_root_proposal_to_upgrade_governance_canister` call — and immediately replace the governance canister with arbitrary code.

---

### Finding Description

**Ballot population — one slot per node, not per operator**

In `submit_root_proposal_to_upgrade_governance_canister`, the ballot list is built by iterating over every node in the NNS subnet membership and looking up its operator: [1](#0-0) 

If operator O controls k nodes, the resulting `node_operator_ballots` contains k entries `(O, Yes)` (for the proposer) or `(O, Undecided)` (for others). There is no deduplication.

**Voting updates all slots for the caller**

In `vote_on_root_proposal_to_upgrade_governance_canister`, the vote loop iterates over every entry and flips every slot whose principal matches the caller: [2](#0-1) 

One `vote_on_root_proposal_to_upgrade_governance_canister` call from operator O sets all k of O's slots to the chosen ballot.

**Threshold check counts raw Yes slots, not unique operators**

`is_byzantine_majority_yes()` computes `num_nodes = node_operator_ballots.len()` (total slots = total nodes N) and counts all Yes-marked slots: [3](#0-2) 

The threshold is `N − f` where `f = (N−1)/3`. There is no check that those Yes votes come from distinct operators.

**Immediate execution on submission**

At proposal submission, the proposer's k nodes are pre-marked Yes: [4](#0-3) 

If k ≥ N−f, `is_byzantine_majority_yes()` is already true at submission time. The proposal executes before any other party can vote.

**Stated design intent is violated**

The code comment explicitly states the intended security model: [5](#0-4) 

The intent is that the security level equals the subnet itself — meaning N−f *independent* parties must agree. The implementation provides no such independence guarantee.

---

### Impact Explanation

A successful exploit allows a single node operator to install arbitrary WASM into the NNS governance canister via `change_canister`. This gives the attacker full control over all NNS governance decisions, neuron balances, treasury, and all downstream SNS/chain-fusion governance. The impact is complete compromise of the Internet Computer's governance layer.

---

### Likelihood Explanation

The attacker must be a registered NNS subnet node operator controlling ≥ N−f nodes. On a 7-node subnet (as in the test fixture), that is 5 nodes. On the production NNS subnet with ~40 nodes, the threshold is ~27. While controlling 27 NNS nodes is a high bar, the vulnerability is structurally present and exploitable at any subnet size where one operator holds a sufficient fraction. The exploit requires only a single ingress call to `submit_root_proposal_to_upgrade_governance_canister` — no coordination, no timing attack, no additional steps.

---

### Recommendation

Deduplicate ballots by operator principal before computing the threshold. The ballot map should be keyed by `PrincipalId` (operator), not by node. Each operator gets exactly one vote regardless of how many nodes they operate. The threshold N−f should be computed over the number of distinct operators, not the number of nodes. Alternatively, store a `BTreeMap<PrincipalId, RootProposalBallot>` instead of `Vec<(PrincipalId, RootProposalBallot)>` and count map entries for the threshold.

### Proof of Concept

State-machine test sketch (7-node NNS subnet, one operator controls nodes 1–5):

1. Configure registry so that `TEST_USER1_PRINCIPAL` is the node operator for nodes 1, 2, 3, 4, and 5 (out of 7 total NNS nodes).
2. Call `submit_root_proposal_to_upgrade_governance_canister` from `TEST_USER1_PRINCIPAL` with a malicious WASM.
3. At submission, the ballot list has 5 entries `(TEST_USER1_PRINCIPAL, Yes)` and 2 entries `(other_operator, Undecided)`.
4. `is_byzantine_majority_yes()`: `num_nodes=7`, `max_faults=2`, threshold=5, `votes_yes=5` → `5 >= 5` → **true**.
5. The proposal executes immediately inside `submit_root_proposal_to_upgrade_governance_canister` — no other vote is needed.
6. Assert `get_pending_root_proposals_to_upgrade_governance_canister()` returns an empty list and the governance canister module hash has changed to the malicious WASM hash.

The existing test at `rs/nns/integration_tests/src/root_proposals.rs` already demonstrates the per-node ballot structure and confirms that 5 Yes votes on a 7-node subnet pass the threshold: [6](#0-5) 

The only difference in the exploit is that all 5 Yes votes come from a single operator rather than 5 distinct ones — the threshold logic does not distinguish between the two cases.

    fn is_byzantine_majority_yes(&self) -> bool {
        let num_nodes = self.node_operator_ballots.len();
        let max_faults = (num_nodes - 1) / 3;
        let votes_yes: usize = self
            .node_operator_ballots
            .iter()
            .map(|(_, b)| match b {
                RootProposalBallot::Yes => 1,
                _ => 0,
            })
            .sum();
        votes_yes >= (num_nodes - max_faults)
    }
