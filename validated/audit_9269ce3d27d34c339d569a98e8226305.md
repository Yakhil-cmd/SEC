### Title
Owner Can Bypass `upgrade_delay_blocks` Timelock via Direct `upgrade()` Function - (File: `engine/src/contract_methods/admin.rs`)

---

### Summary

The Aurora Engine implements a staged upgrade mechanism with `upgrade_delay_blocks` as a timelock safeguard, giving users time to exit before a code change takes effect. However, a separate `upgrade()` entrypoint exists that allows the owner to immediately deploy arbitrary new WASM code with **no delay check whatsoever**, completely bypassing the intended timelock. This renders the `upgrade_delay_blocks` safeguard illusory and enables immediate replacement of the entire contract — including all fund-handling logic — without any warning period.

---

### Finding Description

The `EngineState` struct stores `upgrade_delay_blocks` as an explicit security parameter: [1](#0-0) 

The staged upgrade path (`stage_upgrade` + `deploy_upgrade`) correctly enforces this delay. In `stage_upgrade`, the unlock block height is computed as `block_height + upgrade_delay_blocks` and stored: [2](#0-1) 

And `deploy_upgrade` enforces the delay before allowing deployment: [3](#0-2) 

However, a **separate** `upgrade()` function exists that takes WASM code directly and immediately schedules a `DeployContract` promise with **no delay check**: [4](#0-3) 

The only guard on `upgrade()` is `require_owner_only`. There is no reference to `upgrade_delay_blocks`, no staged storage, and no block-height comparison. The owner can call `upgrade()` at any time and the new WASM is deployed in the same transaction batch, atomically.

This is exposed as a public `extern "C"` entrypoint: [5](#0-4) 

Additionally, the owner can call `set_upgrade_delay_blocks` to reduce the delay to zero before using even the staged path: [6](#0-5) 

---

### Impact Explanation

**Critical — Direct theft of any user funds.**

The Aurora Engine holds all bridged ETH and ERC-20 token balances on behalf of users. The `upgrade()` function allows the owner to immediately replace the entire WASM binary with arbitrary code. A malicious or compromised owner can deploy a new contract that:

- Redirects all `ft_transfer` / `withdraw` calls to an attacker-controlled address.
- Mints arbitrary EVM balances.
- Drains the NEP-141 token supply held by the connector.

Because `upgrade()` bypasses `upgrade_delay_blocks` entirely, users have **zero blocks** of warning to exit their positions. The `upgrade_delay_blocks` field — the only stated safeguard against this — is rendered meaningless by the existence of the direct `upgrade()` path.

---

### Likelihood Explanation

The `upgrade()` function is callable by the single `owner_id` account with no additional multisig, timelock, or governance requirement enforced at the contract level. The owner account is a single point of failure. Any compromise of the owner key (phishing, infrastructure breach, insider threat) results in immediate, irreversible fund loss for all Aurora users. The existence of `upgrade_delay_blocks` in the state creates a false expectation of a safety window that does not apply to the `upgrade()` path.

---

### Recommendation

1. **Remove or gate the direct `upgrade()` function.** Either delete it entirely and require all upgrades to go through `stage_upgrade` + `deploy_upgrade`, or add the same `upgrade_delay_blocks` enforcement to `upgrade()`.
2. **Enforce a minimum non-zero `upgrade_delay_blocks`.** Add a lower-bound check in `set_upgrade_delay_blocks` so the delay cannot be set to zero.
3. **Consider requiring multisig or DAO governance** for the owner role, so no single key can trigger an immediate upgrade.

---

### Proof of Concept

**Attack path:**

1. Owner account calls the `upgrade` entrypoint with a crafted WASM binary that, in its `ft_transfer` handler, redirects all token transfers to an attacker address.
2. `upgrade()` in `admin.rs` passes `require_owner_only`, reads the raw WASM bytes, and schedules `PromiseAction::DeployContract { code }` followed by `state_migration` — all in a single promise batch with no delay.
3. The new WASM is live in the next block. No `upgrade_delay_blocks` check is performed.
4. All subsequent user withdrawals and bridge exits are redirected to the attacker.
5. All bridged ETH and NEP-141 tokens are drained.

```rust
pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())
}
```
