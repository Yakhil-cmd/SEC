### Title
Unclosed IO Frame on NoEE (Zero-Code Account) Call Path — (`basic_bootloader/src/bootloader/runner.rs`)

### Summary
`execute_call` opens a global IO frame via `start_global_frame` (line 276) and passes the `rollback_handle` by value to `call_execute_callee_frame`. When the callee resolves to `ExecutionEnvironmentType::NoEE`, `call_execute_callee_frame` returns early (lines 435–442) without ever calling `finish_global_frame`. `SystemFrameSnapshot` has no `Drop` implementation, so dropping it is a no-op on the underlying frame stacks. The five sub-storage frame stacks (`storage`, `transient_storage`, `logs_storage`, `events_storage`, `interop_root_storage`) each have one extra open frame for the remainder of the transaction.

### Finding Description
The call chain is:

```
execute_call
  → start_global_frame()          // opens frame N on all 5 sub-stacks
  → perform_transfer_if_required  // writes transfer into frame N
  → call_execute_callee_frame(…, rollback_handle)
      if next_ee_type == NoEE:
          return Ok(…)            // rollback_handle dropped, frame N NEVER closed
```

Every other exit from `call_execute_callee_frame` (the `End` preemption point, line 507) and every exit from `call_to_special_address_execute_callee_frame` (lines 547, 596, 616) correctly calls `finish_global_frame`. The NoEE path is the sole missing call.

After the early return, the frame stack depth is off by one for the rest of the transaction. The next `finish_global_frame` call issued by the transaction-level code (e.g., `create_frame_and_execute_transaction_payload` in `zk/mod.rs` line 343 or `ethereum/mod.rs` line 428) closes frame N instead of the transaction-level frame it intended to close. The transaction-level frame then remains open, and the pattern repeats for every subsequent frame operation in that block.

### Impact Explanation
- **Transfer state leaks across frame boundaries.** The ETH/token transfer written inside frame N is committed or rolled back at the wrong nesting level. In the success path the transfer is committed when the transaction-level frame is closed (one level too late), meaning a revert of the outer frame would not undo it.
- **Forward/proving divergence.** The circuit enforces that every `start_io_frame` is matched by a `finish_io_frame`. An execution trace with a mismatched frame count is either unprovable (causing a liveness failure) or, depending on prover behavior, provable with incorrect state (invalid state transition accepted).
- **Cascading frame corruption.** Because the frame stack is off by one for the entire remainder of the block, all subsequent transactions in the same block inherit the corrupted frame depth, amplifying the impact.

### Likelihood Explanation
The trigger is a plain EVM `CALL` (or top-level transaction) to any address that has no deployed code and is not in the special-address space, with a non-zero `msg.value`. This is a completely normal, unprivileged operation (e.g., sending ETH to a fresh EOA). No special role, leaked key, or oracle manipulation is required.

### Recommendation
Add `self.system.finish_global_frame(None)?;` immediately before the `return Ok(…)` in the `NoEE` branch of `call_execute_callee_frame`:

```rust
if next_ee_type == ExecutionEnvironmentType::NoEE {
    // ... existing checks ...
    self.system
        .finish_global_frame(None)
        .map_err(|_| internal_error!("must finish execution frame"))?;
    return Ok((
        external_call_launch_params.external_call.available_resources,
        CallResult::Successful { return_values: ReturnValues::empty() },
    ));
}
```

This mirrors the pattern used in `call_to_special_address_execute_callee_frame` for its non-intercepted path (lines 615–617).

### Proof of Concept
1. Construct a transaction from address `A` (with sufficient balance) calling a fresh zero-code address `B` with `value = 1`.
2. Record the IO frame depth before and after the call returns.
3. Assert `depth_after == depth_before`. With the bug, `depth_after == depth_before + 1`.
4. Differential test: run in forward mode and proving mode; assert the frame depth at block end is identical. With the bug, proving mode will either panic on the mismatched frame count or produce a proof for an invalid state transition where the transfer to `B` is not correctly scoped to the transaction's rollback boundary.
