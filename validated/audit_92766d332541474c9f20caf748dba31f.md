### Title
ckERC20 Minter `Erc20Balances` Tracking Is Broken for Rebasing ERC-20 Tokens, Causing Permanent Fund Loss or Minter Panic - (`rs/ethereum/cketh/minter/src/state.rs`)

---

### Summary

The ckETH minter canister maintains an internal `Erc20Balances` ledger that tracks how many ERC-20 tokens the minter's Ethereum address holds. This ledger is computed **exclusively from audit events** (deposit logs and finalized withdrawal receipts) and is never reconciled against the actual on-chain balance. For rebasing ERC-20 tokens — whose balances change outside of `Transfer` events — this tracked balance permanently diverges from the real balance, causing either irreversible fund loss (positive rebase) or a minter canister panic/trap that halts all withdrawals (negative rebase followed by a successful partial withdrawal).

---

### Finding Description

The `Erc20Balances` struct in `rs/ethereum/cketh/minter/src/state.rs` is the sole source of truth for how much of each ERC-20 token the minter holds on Ethereum:

```rust
/// Current balance of ERC-20 tokens held by the minter.
/// Computed based on audit events.
pub erc20_balances: Erc20Balances,
``` [1](#0-0) 

The balance is updated in exactly two places:

**On deposit** — `erc20_add` is called with the `value` field from the `ReceivedErc20Event` log entry:

```rust
ReceivedEvent::Erc20(event) => self
    .erc20_balances
    .erc20_add(event.erc20_contract_address, event.value),
``` [2](#0-1) 

**On successful withdrawal** — `erc20_sub` is called with the `value` decoded from the finalized Ethereum transaction's call data:

```rust
if receipt.status == TransactionStatus::Success && !tx.transaction_data().is_empty() {
    ...
    self.erc20_balances.erc20_sub(*tx.destination(), value);
}
``` [3](#0-2) 

`erc20_sub` panics unconditionally on underflow:

```rust
let new_value = previous_value
    .checked_sub(withdrawal_amount)
    .unwrap_or_else(|| {
        panic!("BUG: underflow when subtracting {withdrawal_amount} from {previous_value}")
    });
``` [4](#0-3) 

For a rebasing ERC-20 token (e.g., stETH), the minter's actual Ethereum balance changes outside of `Transfer` events. The minter never queries the actual on-chain balance; it only observes `ReceivedEthOrErc20` log events emitted by the helper contract:

```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);
``` [5](#0-4) 

The `value` field in `ReceivedErc20Event` is parsed directly from the log data:

```rust
value: Erc20Value::from_be_bytes(value_bytes),
``` [6](#0-5) 

This means the minter's tracked balance only reflects the amount at the time of the `transferFrom` call. Any subsequent rebase is invisible to the minter.

---

### Impact Explanation

**Positive rebase (e.g., stETH yield accrual):**

1. User deposits 100 stETH → minter mints 100 ckstETH, `erc20_balances` = 100.
2. Rebase occurs: minter's actual Ethereum balance becomes 110 stETH.
3. User withdraws 100 ckstETH → Ethereum tx succeeds, `erc20_sub(100)` → `erc20_balances` = 0.
4. The extra 10 stETH remain in the minter's Ethereum address permanently. No ckstETH was ever minted for them, and the tracked balance is 0, so no withdrawal can ever claim them. **These funds are permanently locked.**

**Negative rebase:**

1. User deposits 100 stETH → minter mints 100 ckstETH, `erc20_balances` = 100.
2. Negative rebase: minter's actual Ethereum balance becomes 90 stETH.
3. User A withdraws 90 ckstETH → Ethereum tx succeeds, `erc20_sub(90)` → `erc20_balances` = 10.
4. User B withdraws 10 ckstETH → Ethereum tx fails (0 stETH left). `erc20_sub` is NOT called (only on `Success`). `erc20_balances` stays at 10.
5. The tracked balance is now permanently wrong (10 vs actual 0). Every future withdrawal attempt for this token will fail on Ethereum, burning users' ckERC20 tokens without delivering ERC-20 tokens.

**Minter canister panic (trap):**

If a scenario arises where `erc20_sub` is called with an amount exceeding the tracked balance — for example, due to a race between a rebase and a successful partial withdrawal — the minter canister will `panic!`, which on the IC translates to a canister trap. This halts the minter's update call, and if it occurs inside a timer task, it can permanently break the minter's withdrawal processing loop for that token.

---

### Likelihood Explanation

The currently supported ckERC20 tokens (USDC, USDT, LINK, PEPE, SHIB, UNI, WBTC, wstETH, XAUt, EURC) are all non-rebasing standard ERC-20 tokens. `wstETH` (the wrapped, non-rebasing version of stETH) is explicitly chosen over `stETH` for this reason.

However, the minter is designed to be extensible: any ERC-20 token can be added via an NNS governance proposal targeting the ledger suite orchestrator. There is no on-chain enforcement preventing a rebasing token from being added. If governance adds a rebasing token (e.g., stETH, aTokens from Aave, or similar yield-bearing tokens), the vulnerability is immediately exploitable by any depositor. The likelihood is therefore **low but non-zero**, contingent on governance adding a rebasing token.

---

### Recommendation

1. **Document the restriction explicitly** in the `add_ckerc20_token` endpoint and the NNS proposal template: rebasing tokens (tokens whose balances change outside of `Transfer` events) are not supported and must not be added.
2. **Add a balance reconciliation mechanism**: periodically query the actual ERC-20 balance of the minter's Ethereum address via `eth_call` (using the existing EVM RPC canister) and compare it against `erc20_balances`. Alert or halt minting/withdrawals if a divergence is detected.
3. **Replace the `panic!` in `erc20_sub` with a graceful error**: instead of trapping the canister, log the inconsistency and skip the balance update (or quarantine the token), so the minter remains operational.

---

### Proof of Concept

```
1. Governance adds a rebasing ERC-20 token T (e.g., stETH) as a supported ckERC20 token.

2. Alice deposits 1000 T via the helper contract:
   - Helper emits ReceivedEthOrErc20(T, Alice, 1000, alice_principal, 0x00)
   - Minter scrapes log, calls erc20_add(T, 1000) → erc20_balances[T] = 1000
   - Minter mints 1000 ckT to Alice on the IC ledger.

3. Positive rebase occurs: minter's actual T balance on Ethereum becomes 1100.
   - erc20_balances[T] remains 1000 (no event was emitted for the rebase).

4. Alice withdraws 1000 ckT:
   - Minter burns 1000 ckT from Alice's IC ledger account.
   - Minter sends Ethereum tx: transfer(Alice_eth, 1000 T) → SUCCESS.
   - update_balance_upon_withdrawal → erc20_sub(T, 1000) → erc20_balances[T] = 0.

5. Result: 100 T permanently stuck in minter's Ethereum address.
   - erc20_balances[T] = 0, so no withdrawal can ever claim them.
   - No ckT was minted for the rebased 100 T.
   - Funds are permanently lost from users' perspective.
```

    pub fn erc20_add(&mut self, erc20_contract: Address, deposit: Erc20Value) {
        match self.balance_by_erc20_contract.get(&erc20_contract) {
            Some(previous_value) => {
                let new_value = previous_value.checked_add(deposit).unwrap_or_else(|| {
                    panic!("BUG: overflow when adding {deposit} to {previous_value}")
                });
                self.balance_by_erc20_contract
                    .insert(erc20_contract, new_value);
            }
            None => {
                self.balance_by_erc20_contract
                    .insert(erc20_contract, deposit);
            }
        }
    }

    pub fn erc20_sub(&mut self, erc20_contract: Address, withdrawal_amount: Erc20Value) {
        let previous_value = self
            .balance_by_erc20_contract
            .get(&erc20_contract)
            .expect("BUG: Cannot subtract from a missing ERC-20 balance");
        let new_value = previous_value
            .checked_sub(withdrawal_amount)
            .unwrap_or_else(|| {
                panic!("BUG: underflow when subtracting {withdrawal_amount} from {previous_value}")
            });
        self.balance_by_erc20_contract
            .insert(erc20_contract, new_value);
