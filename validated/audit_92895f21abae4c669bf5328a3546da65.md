Audit Report

## Title
Unbounded Memory Allocation in CUP Response Body Collection — (`rs/orchestrator/src/catch_up_package_provider.rs`)

## Summary
The `fetch_catch_up_package` function in the orchestrator collects the full HTTP response body from a peer node into heap memory with no byte-size cap. A single Byzantine subnet node can stream an arbitrarily large response body within the `self.backoff` window, exhausting the orchestrator process's memory and causing an OOM crash. Because the backoff doubles on each timeout, repeated attacks on restart give the attacker progressively larger windows, making the node permanently unable to catch up or upgrade without operator intervention.

## Finding Description
At line 349 of `rs/orchestrator/src/catch_up_package_provider.rs`, the response body is collected with no size bound:

```rust
let body_req = timeout(self.backoff, res.into_body().collect());
``` [1](#0-0) 

`self.backoff` is initialized to 30 seconds: [2](#0-1) 

On timeout, the backoff doubles unconditionally: [3](#0-2) 

The raw `collect()` call accumulates every byte chunk into a single `Bytes` buffer on the heap. There is no `http_body_util::Limited` wrapper, no `Content-Length` check, and no per-response byte counter anywhere in the orchestrator's CUP path — confirmed by a full grep of `rs/orchestrator/` returning zero matches for `Limited::new`, `body_size_limit`, or `max_response_size`.

By contrast, the HTTPS outcalls adapter correctly wraps the body before collecting: [4](#0-3) 

The orchestrator's CUP fetcher has no equivalent guard. The TLS handshake authenticates the peer's node identity but does not bound the response body size. Any registered subnet node can serve an arbitrarily large body to the orchestrator of any other node on the same subnet.

## Impact Explanation
An OOM crash of the orchestrator process means the targeted node can no longer detect that a newer CUP exists, download and verify a new CUP, or trigger a replica upgrade. The node stalls at its current height indefinitely. Because the backoff doubles on each timeout/crash cycle (30 s → 60 s → 120 s → …), the attacker's window grows with each restart, making the attack progressively easier to sustain. The node is permanently excluded from subnet participation until an operator manually intervenes. This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS**, specifically a persistent, targeted crash of a node's orchestrator that prevents it from ever rejoining the subnet.

## Likelihood Explanation
The attacker must control one registered subnet node with valid TLS credentials — a single Byzantine node, which is explicitly within the IC's fault-tolerance model (below the consensus fault threshold). The TLS handshake authenticates the peer's node identity but does not bound the response body size. At 100 MB/s sustained throughput, 3 GB can be pushed in 30 seconds before the initial timeout fires. The backoff doubles on each timeout, so subsequent attempts give the attacker an ever-larger window (60 s, 120 s, …), making the attack progressively easier over time. The attack is repeatable on every orchestrator restart with no additional cost to the attacker.

## Recommendation
Replace the bare `.collect()` with a size-bounded variant, mirroring the pattern already used in the HTTPS outcalls adapter:

```rust
use http_body_util::Limited;

const MAX_CUP_RESPONSE_BYTES: usize = 10 * 1024 * 1024; // e.g. 10 MiB

let body_req = timeout(
    self.backoff,
    Limited::new(res.into_body(), MAX_CUP_RESPONSE_BYTES).collect(),
);
```

If the limit is exceeded, `collect()` returns a `LengthLimitError`, which should be mapped to an `Err` and logged without doubling the backoff (to avoid the attacker extending the window on each oversized response).

## Proof of Concept
1. Register a node on the target subnet (Byzantine node scenario).
2. Serve a custom HTTPS endpoint at `/_/catch_up_package` that returns HTTP 200 with `Transfer-Encoding: chunked` and streams 4 GB of zero bytes at maximum bandwidth.
3. Ensure the target orchestrator selects this node as a peer (it will, since it is a registered subnet member).
4. Observe the orchestrator process's RSS grow unboundedly during the 30-second `self.backoff` window.
5. The orchestrator OOM-crashes; the node stops catching up and upgrading.
6. On restart the backoff has doubled to 60 seconds, giving the attacker a larger window for the next attempt.
7. A deterministic integration test can be written using the existing `start_server` / `TestService` harness already present in the test module, adding a `TestService::StreamLargeBody` variant that yields chunks indefinitely, and asserting that `fetch_catch_up_package` returns an error before memory is exhausted.
