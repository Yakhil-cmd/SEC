### Title
EIP-2537 BLS12-381 Precompiles Charge Zero Native (Proving) Cost Despite Significant RISC-V Cycle Consumption - (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/`)

---

### Summary

All seven EIP-2537 BLS12-381 precompiles (G1/G2 addition, G1/G2 MSM, pairing check, and field-to-curve mappings) hard-code `cost_native = 0`, charging no native resource for operations that consume substantial RISC-V proving cycles. Comparable BN254 precompiles charge millions of native units for equivalent operations. An unprivileged caller can invoke these precompiles repeatedly within a single transaction, exhausting the prover's computational budget without triggering the native resource limit, causing the protocol to subsidize proving work that was never paid for.

---

### Finding Description

ZKsync OS implements a **double resource accounting** model: every operation charges both *ergs* (EVM gas equivalent) and *native* resource (a proxy for RISC-V proving cycles). The native limit per transaction is derived from `gasLimit × (gasPrice / nativePrice)`, and a block-level cap of `MAX_NATIVE_COMPUTATIONAL = 2^35` is enforced.

Every BLS12-381 precompile sets `cost_native = 0`:

**G1 Addition** (`addition.rs` line 22):
```rust

let cost_native = 0;
```

**G2 Addition** (`addition.rs` line 63):


**G1 MSM** (`msm.rs` line 198):


**G2 MSM** (`msm.rs` line 266):

**G1/G2 Mappings** (`mappings.rs` lines 26, 76):


Contrast this with the BN254 precompiles in `basic_system/src/cost_constants.rs`, which charge:

| Operation | Native Cost |
|---|---|
| BN254 EC Add | `native_with_delegations!(46_000, 1650, 0)` ≈ 52,600 |
| BN254 EC Mul | `native_with_delegations!(600_000, 41_000, 0)` ≈ 764,000 |
| BN254 Pairing (base) | `native_with_delegations!(13_000_000, 500_000, 0)` ≈ 15,000,000 |
| ECRECOVER | `native_with_delegations!(350_000, 43_000, 0)` ≈ 522,000 |
| P256 Verify | `native_with_delegations!(500_000, 71_000, 0)` ≈ 784,000 |

BLS12-381 operates over a 381-bit prime field (vs BN254's 254-bit field), making its operations **more** expensive in RISC-V cycles than BN254, yet they are charged **zero** native cost. The G2 MSM and pairing operations in particular involve multi-scalar multiplication over a degree-2 extension field and a full Miller loop + final exponentiation, which are among the most expensive operations in the entire system.

The `deltaGas` compensation mechanism in `refund_calculation.rs` (line 72) only adds gas back when `native_used > gas_used * native_per_gas`. Since BLS12-381 operations contribute zero to `native_used`, this correction never fires for them, and the prover absorbs the full cost.

---

### Impact Explanation

An attacker crafts a transaction that calls the BLS12-381 pairing precompile (address `0x0f`) or G2 MSM precompile (address `0x0e`) in a tight loop. Each call:
- Consumes real RISC-V cycles (proving cost) proportional to the number of pairs/points
- Charges only ergs (gas), not native resource
- Does not increment `computational_native_used` toward `MAX_NATIVE_COMPUTATIONAL`

The block-level native limit check in `basic_bootloader/src/bootloader/block_flow/zk/mod.rs` (line 69) is never triggered by these calls. The prover must prove the full computation but the transaction fee only covers the EVM gas cost, not the proving overhead. This is a direct protocol value leak: the sequencer/prover subsidizes proving work that users did not pay for.

---

### Likelihood Explanation

The BLS12-381 precompiles are unconditionally registered when the `eip-2537` feature is enabled (see `mod.rs` `initialize_eip_2537`). Any unprivileged EOA can call them. The gas cost per call is bounded (e.g., a single BLS12-381 pairing with 1 pair costs 37,700 + 32,600 = 70,300 gas), so within a 30M gas block an attacker can invoke ~427 pairing operations, each consuming zero native resource but substantial RISC-V cycles. The attack requires no special privileges, no leaked keys, and no oracle manipulation.

---

### Recommendation

Benchmark the actual RISC-V cycle consumption of each BLS12-381 operation (as was done for BN254 in `basic_system/src/cost_constants.rs`) and assign non-zero `cost_native` values proportional to their proving complexity. At minimum, BLS12-381 G1 addition should be comparable to BN254 EC add (~52,600 native), G2 MSM should be comparable to or exceed BN254 EC mul (~764,000 native per point), and pairing should be comparable to or exceed BN254 pairing (~15,000,000 native base). The same audit should be applied to the EIP-152 BLAKE2F precompile, which also sets `cost_native = 0`.

---

### Proof of Concept

1. Deploy a contract that calls `address(0x0f)` (BLS12-381 pairing) in a loop with valid 1-pair input (384 bytes).
2. Submit a transaction with `gasLimit = 30_000_000` (block gas limit).
3. Each pairing call costs 70,300 gas → ~427 calls per block.
4. Observe: `computational_native_used` reported for the block is near zero despite 427 pairing operations being proven.
5. Compare: a single BN254 pairing with 2 pairs charges `2 × 15,000,000 = 30,000,000` native, which would exhaust `MAX_NATIVE_COMPUTATIONAL = 2^35 ≈ 34,359,738,368` after ~1,145 calls — but BLS12-381 pairing never contributes to this counter regardless of call count.
