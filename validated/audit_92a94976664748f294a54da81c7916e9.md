Audit Report

## Title
Node Provider Can Reassign Node Operator Record to Any Unregistered Principal Without Governance Approval - (`rs/registry/canister/src/mutations/do_update_node_operator_config_directly.rs`)

## Summary
The `do_update_node_operator_config_directly_` function in the Registry canister verifies that the caller is the current `node_provider_principal_id` of a Node Operator record, but then unconditionally writes any caller-supplied `PrincipalId` as the new `node_provider_principal_id` without validating that the target principal is a registered Node Provider in NNS governance. This allows any active Node Provider to silently transfer ownership of their Node Operator record — and all associated nodes — to an arbitrary, unregistered principal, undermining the NNS governance accountability model for Node Providers.

## Finding Description
The canister entry point `update_node_operator_config_directly` carries no governance gate and is explicitly open to all callers:

```rust
// This method can be called by anyone
```

The inner function `do_update_node_operator_config_directly_` performs two checks before writing: (1) the caller must equal the current `node_provider_principal_id` stored in the record, and (2) a rate-limit reservation must succeed. After both pass, the function writes the caller-supplied `node_provider_id` directly into the record with no further validation:

```rust
node_operator_record.node_provider_principal_id = node_provider_id.to_vec();
```

The `UpdateNodeOperatorConfigDirectlyPayload` struct accepts any `PrincipalId` for `node_provider_id` with no constraint that it must correspond to a principal registered as a Node Provider through NNS governance. The registry's `check_node_operator_invariants` function, called via `maybe_apply_mutation_internal`, only checks that node operators referenced in node records exist as registry keys — it does not validate that `node_provider_principal_id` corresponds to a registered Node Provider, and is additionally called with `strict = false` in `check_global_state_invariants`, meaning even the node-operator-existence check is skipped at runtime.

By contrast, the governance-gated `validate_assign_noid_payload` function, which runs when a new Node Operator is created via NNS proposal, explicitly checks:

```rust
let is_registered = node_providers
    .iter()
    .any(|np| np.id.unwrap() == decoded_payload.node_provider_principal_id.unwrap());
if !is_registered {
    return Err("The node provider specified in the payload is not registered".to_string());
}
```

This validation is entirely absent from the direct-call update path. The unit test confirms the gap: `new_np_id = PrincipalId::new_user_test_id(10_001)` — a principal never registered as a Node Provider — is accepted and written without error.

## Impact Explanation
A Node Provider anticipating a governance removal proposal can call `update_node_operator_config_directly` to transfer their Node Operator record's `node_provider_principal_id` to a fresh, unregistered principal they control. The fresh principal immediately inherits control of the Node Operator record and all associated nodes. Because the fresh principal is not a registered Node Provider in NNS governance, governance tooling that targets Node Provider principals cannot directly attribute or sanction the entity through the Node Provider removal path. Node rewards, which are distributed based on `node_provider_principal_id` in the registry, flow to the fresh principal. The transfer can be chained to additional fresh principals within the rate-limit window (up to 140 operations per 7-day spike window). This constitutes a significant NNS governance security impact: the accountability mechanism for Node Providers — the ability to remove a Node Provider and have that removal affect their associated Node Operator records — is bypassed without any NNS proposal.

## Likelihood Explanation
The attack requires only a valid ingress message signed by the current Node Provider's identity. No privileged access, threshold corruption, or social engineering is needed. The rate limiter (`NODE_PROVIDER_MAX_SPIKE = 140` operations per 7-day window) throttles frequency but does not block the transfer. Any Node Provider who monitors NNS governance for proposals targeting their principal has a clear, low-effort window to execute the transfer before the proposal passes.

## Recommendation
`do_update_node_operator_config_directly_` should validate that the supplied `node_provider_id` corresponds to a principal already registered as a Node Provider in the NNS registry before writing the mutation, consistent with the check already present in `validate_assign_noid_payload`. Alternatively, the ability to change `node_provider_principal_id` should be removed from this direct-call path entirely and reserved for the governance-gated `update_node_operator_config` proposal flow, which enforces `check_caller_is_governance_and_log`.

## Proof of Concept
1. Node Provider NP-A controls Node Operator record NO-1 (`node_provider_principal_id = NP-A`).
2. NNS governance initiates a proposal to remove NP-A's Node Provider record.
3. Before the proposal passes, NP-A calls `update_node_operator_config_directly` with `node_operator_id = NO-1` and `node_provider_id = NP-B` (a fresh, never-registered principal controlled by NP-A).
4. The registry writes `node_provider_principal_id = NP-B` for NO-1 — confirmed by the existing unit test `test_update_node_operator_config_directly_happy_path` which uses `PrincipalId::new_user_test_id(10_001)` (unregistered) and asserts success.
5. NP-A's original principal is now disconnected from NO-1; NP-B controls it and receives node rewards.
6. The governance proposal to remove NP-A has no effect on NO-1 or its nodes.
7. NP-B can repeat step 3 with NP-C if NNS governance targets NP-B, creating an indefinite chain within the rate-limit window.
