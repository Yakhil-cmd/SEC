Audit Report

## Title
SNS Governance `remove_neuron_permissions` Allows Permanently Locking Staked Tokens by Removing All Permissions Without Checking Neuron Stake - (File: rs/sns/governance/src/governance.rs)

## Summary

The `remove_neuron_permissions` function in the SNS governance canister performs no check on a neuron's `cached_neuron_stake_e8s` before allowing all `NeuronPermission` entries to be stripped. Once all permissions are removed, no principal can satisfy the `NeuronPermissionType::Disburse` check in `disburse_neuron`, permanently locking the staked SNS tokens in the neuron's ledger subaccount with no on-chain recovery path.

## Finding Description

`remove_neuron_permissions` (rs/sns/governance/src/governance.rs, L4659‚Äì4716) validates only that the caller holds `ManagePrincipals` (or `ManageVotingPermission` for voting-related permissions) and that the target principal actually holds the permissions being removed. It performs no check on the neuron's stake and no check on whether the resulting permission set would leave the neuron ownerless:

```rust
// Re-borrow the neuron mutably to update now that the preconditions have been met
let principal_id_was_removed = self
    .get_neuron_result_mut(neuron_id)?
    .remove_permissions_for_principal(
        principal_id,
        permissions_to_remove.permissions.clone(),
    )?;
```

The underlying `remove_permissions_for_principal` (rs/sns/governance/src/neuron.rs, L782‚Äì793) simply drops the `NeuronPermission` entry when `remaining_permission_types` is empty, with no stake guard:

```rust
if remaining_permission_types.is_empty() {
    self.permissions.swap_remove(existing_permission_position);
    return Ok(RemovePermissionsStatus::AllPermissionTypesRemoved);
}
```

`disburse_neuron` (rs/sns/governance/src/governance.rs, L1125‚Äì1127) unconditionally requires `NeuronPermissionType::Disburse` on the caller:

```rust
let neuron = self.get_neuron_result(id)?;
neuron.check_authorized(caller, NeuronPermissionType::Disburse)?;
```

With `permissions: []`, `check_authorized` will fail for every principal, making the staked tokens permanently inaccessible. The function's own docstring acknowledges this: *"This is a dangerous operation as it is possible to remove all permissions for a neuron and no longer be able to modify its state, i.e. disbursing the neuron back into the governance token."* No guard was added to enforce the invariant.

The integration test `test_neuron_remove_all_permissions_of_self` (rs/sns/integration_tests/src/neuron.rs, L2212‚Äì2265) confirms the behavior: it stakes tokens, removes all permissions, and asserts `neuron.permissions.len() == 0` ‚Äî with no error and no stake check.

**Exploit path (malicious co-controller):**
1. User A stakes SNS tokens and claims a neuron, receiving all default permissions including `ManagePrincipals` and `Disburse`.
2. User A grants User B `ManagePrincipals` via `AddNeuronPermissions`.
3. User B calls `manage_neuron { RemoveNeuronPermissions }` targeting User A, removing all of User A's permissions.
4. User B calls `manage_neuron { RemoveNeuronPermissions }` targeting themselves, removing all of their own permissions.
5. The neuron now has `permissions: []` but `cached_neuron_stake_e8s > 0`.
6. Every subsequent `disburse_neuron` call by any principal fails with `NotAuthorized`.
7. Tokens are permanently locked; the neuron record persists in governance state indefinitely.

No existing check prevents step 3 or 4. There is no governance proposal path or admin override to recover a permissionless neuron.

## Impact Explanation

Staked SNS tokens held in the neuron's governance ledger subaccount become permanently inaccessible. The neuron record persists in governance state but no principal can disburse, split, merge, or otherwise recover the tokens. This constitutes a permanent, irreversible loss of user funds in an in-scope SNS governance canister, matching: **High ($2,000‚Äì$10,000) ‚Äî Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints.**

## Likelihood Explanation

The attack requires a malicious co-controller who has been granted `ManagePrincipals` by the neuron owner. This is a meaningful per-target constraint: User A must first grant User B `ManagePrincipals`. However, `ManagePrincipals` is a standard neuron permission routinely granted to co-controllers, DAOs, or automation scripts. Once granted, the attack requires only two standard `manage_neuron` ingress calls ‚Äî no privileged system access, no governance majority, no subnet compromise. The attack is irreversible and repeatable across any neuron whose owner has granted `ManagePrincipals` to an untrusted party.

## Recommendation

**Short term**: In `remove_neuron_permissions`, after computing the would-be resulting permission set across all principals, reject the operation if it would leave the neuron with zero total permissions and a non-zero `cached_neuron_stake_e8s` (or non-zero `maturity_e8s_equivalent`). At minimum, verify that at least one principal retains `NeuronPermissionType::Disburse` when the neuron's stake is non-zero.

**Long term**: Enforce the SNS neuron state machine invariant that a neuron with non-zero stake must always have at least one principal holding `Disburse` permission. Add invariant-based fuzz tests verifying this property is preserved across all `manage_neuron` commands.

## Proof of Concept

The existing integration test `test_neuron_remove_all_permissions_of_self` (rs/sns/integration_tests/src/neuron.rs, L2212‚Äì2265) already demonstrates the self-removal path. To reproduce the malicious co-controller path:

1. Extend the test to stake tokens with User A and grant User B `ManagePrincipals`.
2. Have User B call `remove_neuron_permissions_or_panic` targeting User A with `NeuronPermissionType::all()`.
3. Have User B call `remove_neuron_permissions_or_panic` targeting themselves with `NeuronPermissionType::all()`.
4. Assert `neuron.permissions.len() == 0` and `neuron.cached_neuron_stake_e8s > 0`.
5. Have any principal attempt `disburse_neuron` and assert the call returns `NotAuthorized`.

All steps succeed without error up to step 5, confirming permanent token lock.
