Audit Report

## Title
Unbounded Neuron Iteration in SNS `compute_ballots_for_new_proposal` Can Exhaust Instruction Limit, Permanently Blocking Proposal Submission - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS Governance canister's `compute_ballots_for_new_proposal` function iterates synchronously over every neuron in `self.proto.neurons` with no instruction-limit guard. When the neuron count is large enough, any call to `make_proposal` will trap with `CanisterInstructionLimitExceeded`, permanently preventing new governance proposals from being submitted to that SNS. No instruction-counter check, soft-limit escape hatch, or pre-computed snapshot exists in this code path, in contrast to the NNS Governance canister which explicitly handles this with `noop_self_call_if_over_instructions` and a voting-power snapshot.

## Finding Description
`make_proposal` in `rs/sns/governance/src/governance.rs` calls `compute_ballots_for_new_proposal` unconditionally before inserting any proposal: [1](#0-0) 

`compute_ballots_for_new_proposal` contains a plain `for` loop over the entire neuron map with no guard: [2](#0-1) 

Per iteration, the function reads the neuron's dissolve state, computes a multi-factor voting-power bonus, and inserts a `Ballot` into a `BTreeMap`. A grep over the entire SNS governance source confirms there is no call to `ic_cdk::api::instruction_counter()`, no `over_soft_message_limit()` check, and no `noop_self_call_if_over_instructions` escape hatch anywhere in this code path.

By contrast, the NNS Governance canister defines explicit hard and soft instruction limits: [3](#0-2) 

And uses `noop_self_call_if_over_instructions` to yield before the limit is hit: [4](#0-3) 

The NNS comment explicitly states "This leaves room for 750 thousand neurons with complex following" against a 750 billion instruction hard limit. The SNS runs on an application subnet with a 5 billion instruction limit — 150× smaller — yet has no analogous protection. Once the neuron count crosses the threshold where per-neuron work (dissolve-delay arithmetic, age-bonus computation, `BTreeMap::insert`) × N > 5 × 10⁹, every `make_proposal` call traps. Because the neuron map size is unchanged between calls, the canister is permanently blocked.

## Impact Explanation
This is a concrete application/platform-level DoS on SNS governance proposal submission. No new governance proposals can be submitted to the affected SNS. Ongoing proposals are unaffected, but the SNS governance is permanently frozen for new actions — including any remediation proposal that would itself require `make_proposal` to succeed. The only escape hatch is an NNS-level upgrade of the SNS governance canister, which requires NNS governance action and significant delay. This matches the allowed High impact: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS" and "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
Any SNS that achieves organic growth in its neuron population will naturally approach the threshold over time. An adversary holding sufficient stake can accelerate this by repeatedly splitting neurons down to the minimum stake, filling the neuron map. The `make_proposal` entry point requires only a neuron with `SubmitProposal` permission and sufficient stake to cover `reject_cost_e8s` — a low bar for any legitimate participant. No privileged access, no threshold corruption, and no external oracle is required. The attack is repeatable and irreversible without an NNS upgrade.

## Recommendation
1. **Adopt the NNS pattern**: replace the live neuron iteration with a pre-computed, periodically refreshed voting-power snapshot updated in a timer job where instruction overruns can be handled incrementally.
2. **Add an instruction-limit guard**: if the live iteration must be kept, add an `ic_cdk::api::instruction_counter()` check inside the loop and return a graceful error (not a trap) when the soft limit is approached, keeping the canister responsive.
3. **Enforce a tighter neuron cap**: lower `max_number_of_neurons` in `NervousSystemParameters` to a value provably safe given the per-neuron instruction cost on a 5 × 10⁹ instruction budget, and add a benchmark test that enforces this bound.

## Proof of Concept
1. Deploy an SNS with default `NervousSystemParameters`.
2. Create N neurons (by staking and claiming, or by splitting existing neurons) where N × (per-neuron instruction cost for dissolve-delay read + voting-power computation + `BTreeMap::insert`) > 5 × 10⁹.
3. Call `manage_neuron` → `MakeProposal` from any neuron with `SubmitProposal` permission.
4. Observe the call trap with `CanisterInstructionLimitExceeded`.
5. Confirm that no subsequent `make_proposal` call succeeds regardless of the proposer, because the neuron map size is unchanged.
6. A deterministic integration test using PocketIC can instrument the instruction counter before and after `compute_ballots_for_new_proposal` with a large synthetic neuron population to establish the exact threshold.
