Audit Report

## Title
TOCTOU Race in ckBTC Minter Allows `MAX_CONCURRENT_PENDING_REQUESTS` Limit to Be Exceeded - (File: `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary
The `retrieve_btc` and `retrieve_btc_with_approval` functions check `count_incomplete_retrieve_btc_requests()` against `MAX_CONCURRENT_PENDING_REQUESTS` (5 000) before multiple inter-canister `await` points, but perform no re-check inside `mutate_state` before calling `accept_retrieve_btc_request`. Because the IC yields execution at every `await`, up to `MAX_CONCURRENT` (100) concurrent callers from distinct accounts can all observe the same stale count, all pass the guard, and all commit their requests, pushing the pending queue beyond the intended limit. The per-account `retrieve_btc_guard` prevents the same account from racing with itself but does not prevent distinct accounts from racing against the global count.

## Finding Description
In `retrieve_btc` (lines 174–179), the count check is a plain `read_state` call with no lock held across the subsequent awaits:

```rust
// retrieve_btc.rs L174
if read_state(|s| s.count_incomplete_retrieve_btc_requests() >= MAX_CONCURRENT_PENDING_REQUESTS) {
    return Err(...);
}
let balance = balance_of(caller).await?;          // yields
let status  = check_address(...).await?;           // yields
let block_index = burn_ckbtcs(...).await?;         // yields
mutate_state(|s| state::audit::accept_retrieve_btc_request(s, request, ...));
```

The identical pattern exists in `retrieve_btc_with_approval` at lines 274–279. The `retrieve_btc_guard` (guard.rs L41–60) atomically checks-and-inserts into `retrieve_btc_accounts` inside a single `mutate_state`, correctly preventing the same account from racing. However, it only caps total concurrent in-flight calls at `MAX_CONCURRENT = 100` (guard.rs L6); it does not protect the global `count_incomplete_retrieve_btc_requests` threshold. `accept_retrieve_btc_request` (audit.rs L26) unconditionally pushes to `pending_retrieve_btc_requests` with no second count check. `count_incomplete_retrieve_btc_requests` (state.rs L962–970) sums pending, in-flight, and submitted requests.

Exploit flow:
1. Bring the minter to `count == 4 999`.
2. Submit up to 100 `retrieve_btc` calls from 100 distinct funded accounts in the same IC round (bounded by `MAX_CONCURRENT = 100`).
3. All 100 callers read `count == 4 999 < 5 000` and pass the guard.
4. Each caller's `balance_of` → `check_address` → `burn_ckbtcs` chain completes; ckBTC is burned for each.
5. Each caller's `accept_retrieve_btc_request` executes, pushing 100 new entries into `pending_retrieve_btc_requests`.
6. Final queue depth: `4 999 + 100 = 5 099`, exceeding the intended maximum with no error returned to any caller.

The claim's PoC overstates the magnitude (5 001 concurrent callers) because the `retrieve_btc_guard` caps concurrent in-flight calls at 100, bounding the overshoot per wave to 100 entries. The TOCTOU itself is confirmed.

## Impact Explanation
This is a concrete bypass of a safety limit in the ckBTC minter, a production chain-key financial integration. The pending request queue can be pushed beyond `MAX_CONCURRENT_PENDING_REQUESTS` by up to 100 entries per attack wave. While the overshoot is bounded (not unbounded as claimed), it represents a real violation of the minter's accounting invariant: ckBTC is burned and BTC withdrawal requests are accepted beyond the intended cap. Repeated waves are blocked once the count exceeds 5 000, but the first wave succeeds silently. This fits the allowed impact class: **Significant ck-token security impact with concrete user or protocol harm** — High ($2,000–$10,000).

## Likelihood Explanation
Any unprivileged principal holding ckBTC can call `retrieve_btc`. No special role or governance action is required. The attacker needs at most 100 distinct funded accounts and the ability to submit 100 messages in the same IC round, which is straightforwardly achievable with the IC agent library. The ckBTC burned is recovered as BTC; the cost is only transaction fees.

## Recommendation
Re-check `count_incomplete_retrieve_btc_requests()` inside `mutate_state`, immediately before calling `accept_retrieve_btc_request`, so the check and the state mutation are atomic:

```rust
mutate_state(|s| {
    if s.count_incomplete_retrieve_btc_requests() >= MAX_CONCURRENT_PENDING_REQUESTS {
        return Err(RetrieveBtcError::TemporarilyUnavailable(
            "too many pending retrieve_btc requests".to_string(),
        ));
    }
    state::audit::accept_retrieve_btc_request(s, request, &IC_CANISTER_RUNTIME);
    Ok(())
})?;
```

Apply the same fix to `retrieve_btc_with_approval`. This mirrors the correct atomic check-and-insert pattern already used by `Guard::new` (guard.rs L45–60).

## Proof of Concept
1. Initialize a PocketIC instance with the ckBTC minter and ledger canisters.
2. Pre-populate the minter state so `count_incomplete_retrieve_btc_requests() == 4 999` (e.g., by inserting synthetic pending requests directly into state or by running 4 999 successful `retrieve_btc` calls).
3. Create 100 distinct test principals, each holding sufficient ckBTC in their withdrawal subaccount.
4. Submit all 100 `retrieve_btc` calls in the same PocketIC round (before advancing the IC clock or processing callbacks).
5. Advance the IC to process all callbacks.
6. Assert `read_state(|s| s.count_incomplete_retrieve_btc_requests()) > 5 000` — the invariant is violated.
7. Assert all 100 calls returned `Ok(RetrieveBtcOk { block_index })` with no `TemporarilyUnavailable` error.
