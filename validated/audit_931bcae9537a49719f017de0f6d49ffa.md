### Title
`get_blockhash` Returns `InternalError` Instead of Zero for Unavailable Historical Hashes - (File: `zk_ee/src/system/mod.rs`)

---

### Summary

The `get_blockhash` function in `zk_ee/src/system/mod.rs` implements the EVM `BLOCKHASH` opcode. The EVM specification requires `BLOCKHASH` to return `0` when the requested block hash is unavailable (e.g., block does not exist or is out of range). However, when a block number falls within the 256-block window but the corresponding hash is absent from the oracle-supplied metadata, `get_blockhash` returns an `InternalError` instead of `Ok(Bytes32::ZERO)`. This `InternalError` propagates through the EVM interpreter via the `?` operator and can escalate to a fatal block-processing failure — triggered by any unprivileged transaction sender on a young chain.

---

### Finding Description

In `zk_ee/src/system/mod.rs` lines 127–142, `get_blockhash` correctly returns `Ok(Bytes32::ZERO)` for block numbers outside the 256-block window:

```rust
pub fn get_blockhash(&self, block_number: u64) -> Result<Bytes32, InternalError> {
    let current_block_number = self.metadata.block_number();
    if block_number >= current_block_number
        || block_number < current_block_number.saturating_sub(256)
    {
        Ok(Bytes32::ZERO)   // ← correct: out-of-range → zero
    } else {
        let depth = current_block_number - block_number;
        self.metadata
            .block_historical_hash(depth)
            .ok_or(internal_error!(          // ← BUG: missing hash → InternalError
                "historical hash of limited depth must be provided"
            ))
    }
}
``` [1](#0-0) 

The range guard uses `saturating_sub(256)`. When `current_block_number < 256` (a young chain), `saturating_sub(256)` returns `0`, so **every** block number from `0` to `current_block_number - 1` is considered "in range." For example, at block 10, block numbers 0–9 are all in range. If the oracle-supplied metadata does not carry a hash for depth 10 (i.e., the genesis block), `block_historical_hash(10)` returns `None`, and the function returns `Err(InternalError)` instead of `Ok(Bytes32::ZERO)`.

The `BLOCKHASH` opcode implementation in `evm_interpreter/src/instructions/environment.rs` propagates this error unconditionally:

```rust
let block_hash = U256::from_be_bytes(system.get_blockhash(block_number)?.as_u8_array());
``` [2](#0-1) 

The `?` converts the `InternalError` into a `SubsystemError::LeafDefect`, which is a fatal, non-recoverable error in ZKsync OS — distinct from a normal EVM revert. The EVM specification (Yellow Paper) mandates that `BLOCKHASH` returns `0` for any block whose hash is unavailable, including non-existent blocks on a young chain.

The interface contract is defined in `zk_ee/src/system/io.rs` and `zk_ee/src/system/mod.rs`. The `IOSubsystem` trait and `System` wrapper expose `get_blockhash` as a fallible function returning `Result<Bytes32, InternalError>`, but callers in the EVM interpreter treat any `Err` as a fatal defect, not a graceful "hash unavailable" signal. [3](#0-2) 

---

### Impact Explanation

An `InternalError` (a `SubsystemError::LeafDefect`) in ZKsync OS is not caught at the transaction level — it propagates to the bootloader and causes the entire block to fail processing. This means:

- Any contract that calls `BLOCKHASH` with a block number that is within the 256-block window but for which no hash is stored in the oracle metadata will cause the block to abort.
- On a young chain (block numbers 1–255), the genesis block (block 0) is always within the 256-block window but may have no hash in the metadata, making this trivially triggerable.
- The impact is a **liveness failure**: the sequencer cannot finalize the block, halting the chain.

---

### Likelihood Explanation

- **Reachable by any unprivileged transaction sender**: any EVM contract can execute `BLOCKHASH(n)` for arbitrary `n`.
- **Triggered on young chains**: every ZKsync OS deployment starts at block 0. During the first 256 blocks, at least one block number is always in range but potentially missing from the metadata.
- **No special privileges required**: the attacker only needs to submit a transaction to a contract that calls `BLOCKHASH` with a block number in the vulnerable range.
- **No external oracle manipulation required**: the root cause is in the ZKsync OS code itself — the missing graceful fallback to `Bytes32::ZERO`.

---

### Recommendation

Replace the `ok_or(internal_error!(...))` with a graceful fallback to `Bytes32::ZERO`, matching the EVM specification:

```rust
pub fn get_blockhash(&self, block_number: u64) -> Result<Bytes32, InternalError> {
    let current_block_number = self.metadata.block_number();
    if block_number >= current_block_number
        || block_number < current_block_number.saturating_sub(256)
    {
        Ok(Bytes32::ZERO)
    } else {
        let depth = current_block_number - block_number;
        // Return zero if the hash is not available (e.g., young chain, genesis block)
        Ok(self.metadata.block_historical_hash(depth).unwrap_or(Bytes32::ZERO))
    }
}
```

This aligns with the EVM Yellow Paper requirement that `BLOCKHASH` returns `0` for any block whose hash is unavailable, and eliminates the fatal error path reachable by unprivileged callers.

---

### Proof of Concept

1. Deploy a ZKsync OS chain starting at block 0.
2. At block number `N` where `1 ≤ N ≤ 255`, deploy a contract with bytecode that executes `BLOCKHASH(0)` (block 0 is within the 256-block window but may have no hash in the metadata).
3. Submit a transaction calling that contract.
4. The `get_blockhash(0)` call computes `depth = N - 0 = N`, calls `block_historical_hash(N)`, which returns `None` if the metadata does not carry a hash for that depth.
5. `get_blockhash` returns `Err(InternalError("historical hash of limited depth must be provided"))`.
6. The `?` in `evm_interpreter/src/instructions/environment.rs:89` propagates this as a `LeafDefect`.
7. Block processing aborts with a fatal internal error instead of the contract receiving `0` from `BLOCKHASH`. [1](#0-0) [4](#0-3)
