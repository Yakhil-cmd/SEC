### Title
Permanent Fund Loss in `ExitToNear` Precompile When `ft_transfer` Fails Without `error_refund` Feature - (`File: engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` compile-time feature is not enabled, the `ExitToNear` precompile burns ERC-20 tokens (or deducts ETH) in the EVM and schedules an async `ft_transfer` promise to the NEP-141 contract, but attaches **no callback** to handle promise failure. If the `ft_transfer` promise fails for any reason, the burned tokens are permanently lost with no recovery path.

### Finding Description

The `ExitToNear` precompile's `run` function constructs a `callback_args` struct whose `refund` field is unconditionally `None` when the `error_refund` feature is absent:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
```

For a standard ERC-20 exit (no wNEAR unwrap), `transfer_near` is also `None`, making `callback_args` equal to the default value. The branch then selects a bare `PromiseArgs::Create` with no callback:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no failure handler
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
```

The ERC-20 burn (or ETH deduction) has already been committed to EVM state before this promise is scheduled. If the `ft_transfer` promise fails on the NEAR side (e.g., recipient account not registered with the NEP-141 contract, NEP-141 contract paused, or any other NEAR-side rejection), there is no `exit_to_near_precompile_callback` invocation, no re-mint, and no ETH refund. The user's funds are permanently destroyed.

This is explicitly confirmed by the test suite, which documents the divergent behavior:

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

The same pattern applies to ETH exits:

```rust
#[cfg(feature = "error_refund")]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE);
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
```

### Impact Explanation

**Critical — Permanent freezing/loss of funds.**

When `error_refund` is not compiled in and the downstream `ft_transfer` promise fails, the user's ERC-20 tokens are permanently burned in the EVM (or ETH is permanently deducted) with no on-chain recovery mechanism. The NEP-141 tokens never arrive at the recipient. There is no retry, no re-mint, and no refund. The funds are irrecoverably lost.

### Likelihood Explanation

**Medium.** The failure condition — a rejected `ft_transfer` — is reachable by any EVM user without special privileges. Common triggers include:
- Specifying a recipient NEAR account that is not storage-registered with the NEP-141 contract (a common user mistake).
- The NEP-141 contract being paused or having its own internal balance checks fail.
- Sending to a NEAR account ID that does not exist.

The `error_refund` feature being absent is a valid production build configuration (the codebase explicitly supports and tests both modes, and ships two different ERC-20 bytecodes: `EvmErc20.bin` vs `EvmErc20V2.bin`).

### Recommendation

1. **Always attach a failure-handling callback** in `ExitToNear::run`, regardless of the `error_refund` feature flag. The callback should at minimum re-mint burned ERC-20 tokens or refund ETH when the `ft_transfer` promise fails.
2. If the `error_refund` feature is intentionally omitted for gas savings, document clearly that exit calls to unregistered recipients will result in permanent fund loss, and add a pre-flight check (e.g., verify recipient registration) inside the precompile before burning tokens.
3. Consider making `error_refund` a mandatory feature for any production deployment that handles real user funds.

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature flag.
2. Bridge a NEP-141 token to Aurora, receiving ERC-20 tokens.
3. Call the ERC-20 `withdraw` function targeting a NEAR account that is **not registered** with the NEP-141 contract (e.g., a freshly created account that has never called `storage_deposit`).
4. The ERC-20 tokens are burned in the EVM (state committed).
5. The `ft_transfer` promise to the NEP-141 contract fails because the recipient is not registered.
6. No callback fires. No re-mint occurs.
7. Observe: ERC-20 balance is reduced, NEP-141 balance of recipient is 0, NEP-141 balance of Aurora contract is unchanged. Funds are permanently lost.

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);

            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
