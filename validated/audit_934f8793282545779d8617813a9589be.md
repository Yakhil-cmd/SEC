### Title
Truncating Division in `native_per_pubdata` Calculation Underestimates Pubdata Cost — (`File: basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary

In the ZK transaction validation path, `native_per_pubdata` is computed using truncating (floor) division via `wrapping_div`, while the analogous `native_per_gas` uses ceiling division (`div_ceil`). This inconsistency causes the native resource cost per pubdata byte to be systematically underestimated whenever `pubdata_price % native_price != 0`, allowing every L2 ZK transaction sender to underpay for pubdata.

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the two resource-rate calculations are:

```rust
// Line 135 — uses ceiling division (correct, conservative)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...

// Line 142 — uses truncating division (incorrect, too lenient)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...
```

`wrapping_div` on unsigned integers is identical to floor/truncating division. When `pubdata_price` is not an exact multiple of `native_price`, the result is `⌊pubdata_price / native_price⌋`, which is one unit less than `⌈pubdata_price / native_price⌉`.

`native_per_pubdata` is then used in three downstream places:

1. **`create_resources_for_tx`** (gas_helpers.rs line 352): `intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata)` — the native budget deducted for intrinsic pubdata is too small, giving the user a slightly larger effective native budget.

2. **`check_enough_resources_for_pubdata`** (gas_helpers.rs line 451) and **`get_resources_to_charge_for_pubdata`** (line 430): `native = current_pubdata_spent.checked_mul(native_per_pubdata)` — the native cost charged for each pubdata byte is underestimated by up to `(native_price − 1)` native units per byte.

3. **`compute_gas_refund`** (refund_calculation.rs line 72): `delta_gas = (native_used / native_per_gas) − gas_used` — because `native_used` is smaller than it should be, the gas adjustment is smaller, and the user is charged less gas for pubdata.

### Impact Explanation

Every L2 ZK transaction that generates pubdata when `pubdata_price % native_price ≠ 0` underpays for pubdata. The underpayment per transaction is up to `pubdata_bytes_used × (native_price − 1)` native units, which translates to approximately `pubdata_bytes_used × (native_price − 1) / native_per_gas` gas units. This is a resource accounting bug: the operator/protocol receives less compensation for L1 data availability costs than it should, and the pubdata budget check (`check_enough_resources_for_pubdata`) is more lenient than intended, potentially allowing transactions to publish more pubdata than they have paid for.

**Impact: Medium** — systematic underpayment for pubdata across all affected L2 ZK transactions; the protocol absorbs the difference.

### Likelihood Explanation

**Likelihood: High** — `wrapping_div` is hardcoded. The condition `pubdata_price % native_price != 0` is the common case in production (both values are set independently by the operator via block metadata). No special transaction crafting is required; any ordinary L2 ZK transaction triggers this path.

### Recommendation

Replace `wrapping_div` with `div_ceil` to match the rounding convention used for `native_per_gas`:

```diff
- let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
+ let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
      .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
