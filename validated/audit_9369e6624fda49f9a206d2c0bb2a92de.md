### Title
`exit_to_near_precompile_callback` Blocked by `require_running` Causes Permanent Fund Loss When Engine Is Paused - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `exit_to_near_precompile_callback` function in `engine/src/contract_methods/connector.rs` enforces `require_running` at the top of its body. This callback is the settlement receipt that either (a) transfers unwrapped wNEAR to the user on success, or (b) refunds burned EVM tokens on failure of the underlying NEP-141 transfer. Because NEAR receipts are executed independently and state changes from the original EVM execution (token burn) are not automatically reverted when a callback receipt fails, pausing the engine between the original `ExitToNear` precompile call and the callback receipt causes the refund path to be permanently skipped, resulting in irreversible loss of user funds.

---

### Finding Description

The `ExitToNear` precompile burns the user's EVM tokens (ETH or ERC-20) and schedules a NEP-141 transfer promise with `exit_to_near_precompile_callback` attached as the settlement callback. The callback is responsible for two critical paths:

1. **Success path**: If the NEP-141 transfer succeeded and `transfer_near` is set, transfer wNEAR to the user.
2. **Refund path**: If the NEP-141 transfer failed and `refund` args are present, call `engine::refund_on_error` to re-mint the burned EVM tokens.

`require_running` is defined as:

        return Err(errors::ERR_PAUSED.into());

The engine owner can pause the contract at any time via `pause_contract`, which sets `state.is_paused = true`: [3](#0-2) 

In NEAR Protocol, each receipt (including callbacks) is an independent execution unit. The EVM token burn committed in the original `submit` receipt is **not** rolled back if a subsequent callback receipt fails. Therefore, if the engine is paused between the original call and the callback receipt, `require_running` causes the callback to return `Err(ERR_PAUSED)`, the refund path (`engine::refund_on_error`) is never reached, and the user's burned tokens are permanently lost. [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing/loss of user funds.**

When the refund path is blocked:
- The user's EVM tokens (ETH or ERC-20) were already burned in the original EVM execution.
- The NEP-141 transfer on NEAR failed (e.g., no storage deposit, account does not exist).
- The callback that would re-mint the burned tokens cannot execute.
- The tokens are permanently destroyed with no recovery path.

This matches the "Permanent freezing of funds" impact category.

---

### Likelihood Explanation

The engine owner (a privileged but not necessarily malicious role) can call `pause_contract` at any time. In NEAR, the callback receipt is typically scheduled in the same block or the next block after the original call. A window exists between the original `ExitToNear` precompile execution and the callback receipt execution. A malicious or negligent owner who pauses the engine during this window (even for legitimate reasons such as responding to an incident) will inadvertently or deliberately cause permanent fund loss for any in-flight `ExitToNear` operations whose underlying NEP-141 transfer fails.

The scenario does not require a compromised key — it only requires the owner to exercise their normal `pause_contract` privilege at an inopportune moment.

---

### Recommendation

Remove `require_running` from `exit_to_near_precompile_callback`. Settlement/refund callbacks must be executable regardless of the engine's pause state, because the funds at stake were already committed in a prior receipt. The pause guard is appropriate for user-initiated entry points (like `submit`, `ft_on_transfer`, `withdraw`) but must not gate callbacks that finalize already-in-flight fund movements.

Analogously, any other callback in `engine/src/contract_methods/xcc.rs` that calls `require_running` and handles refund or settlement logic should be audited for the same pattern. [5](#0-4) 

---

### Proof of Concept

1. User calls the `ExitToNear` precompile from an EVM transaction (via `submit`). The precompile burns the user's EVM tokens and schedules a NEP-141 `ft_transfer` promise with `exit_to_near_precompile_callback` attached.

2. The NEP-141 transfer promise is scheduled as a NEAR receipt. Before it executes (or after it executes but before the callback receipt executes), the engine owner calls `pause_contract`. [6](#0-5) 

3. The NEP-141 transfer fails (e.g., the target NEAR account has no storage deposit for the token).

4. The callback receipt `exit_to_near_precompile_callback` fires. It immediately hits `require_running`, which returns `Err(ERR_PAUSED)`. [7](#0-6) 

5. The refund branch — `engine::refund_on_error` — is never reached. [8](#0-7) 

6. The user's EVM tokens remain burned. No refund is issued. The owner resumes the engine via `resume_contract`, but the callback receipt has already been finalized as failed; it will never re-execute. The user's funds are permanently lost.

#[named]
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}

#[named]
pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
