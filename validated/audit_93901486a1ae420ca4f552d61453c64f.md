### Title
wNEAR Attached to XCC Calls Is Locked in Engine's Implicit EVM Address When Engine Is Paused — (`engine-precompiles/src/xcc.rs`, `engine/src/contract_methods/xcc.rs`)

---

### Summary

When a user invokes the XCC precompile, the required wNEAR (including `STORAGE_AMOUNT` on first use) is immediately transferred from the user's EVM balance to the engine's implicit EVM address as an EVM state change. A subsequent NEAR promise then calls `withdraw_wnear_to_router` to forward those funds to the router sub-account. If the engine is paused between these two steps, `withdraw_wnear_to_router` fails its `require_running` guard, and the wNEAR is permanently stranded in the engine's implicit EVM address with no recovery path available to the user.

---

### Finding Description

**Step 1 — wNEAR is irrevocably debited from the user in the EVM.**

In `engine-precompiles/src/xcc.rs`, the precompile computes `required_near` and immediately executes a wNEAR `transferFrom` inside the EVM, moving funds from the caller to the engine's implicit address: [1](#0-0) 

This EVM state change is committed as part of the current NEAR transaction. There is no rollback if the downstream promise fails.

**Step 2 — A NEAR promise is scheduled to forward those funds.**

After the EVM execution, `handle_precompile_promise` in `engine/src/xcc.rs` schedules a call to `withdraw_wnear_to_router` on the engine itself: [2](#0-1) 

This promise executes in a **subsequent NEAR block**, creating a window between the committed EVM debit and the actual forwarding of funds.

**Step 3 — `withdraw_wnear_to_router` is gated by `require_running`.**

In `engine/src/contract_methods/xcc.rs`, the callback that actually moves the wNEAR to the router checks the engine's pause state first: [3](#0-2) 

If the engine is paused (`is_paused = true`) between Block N (EVM execution) and Block N+1 (promise execution), this call panics and the wNEAR remains in the engine's implicit EVM address.

**Step 4 — No recovery path exists while paused.**

All EVM entry points (`submit`, `call`, `submit_with_args`) also enforce `require_running`: [4](#0-3) [5](#0-4) 

There is no special-case escape hatch to retrieve wNEAR stranded in the engine's implicit address. The `pause_contract` / `resume_contract` mechanism is owner-controlled: [6](#0-5) 

If the engine is paused indefinitely (e.g., during a deprecation or emergency shutdown analogous to `exodusMode`), the stranded wNEAR is permanently frozen.

---

### Impact Explanation

Every user who had an in-flight XCC call at the moment of pause loses the wNEAR they attached to that call. For first-time XCC users this includes `STORAGE_AMOUNT` = 2 NEAR: [7](#0-6) [8](#0-7) 

The funds are held in an EVM address that is inaccessible while the engine is paused, constituting a **temporary (potentially permanent) freeze of user funds**. This maps to **High — Temporary freezing of funds** in the allowed impact scope.

---

### Likelihood Explanation

The engine can be paused by the owner at any time via `pause_contract`. The vulnerable window is the one-block gap between the EVM transaction that debits wNEAR and the NEAR promise that forwards it. Any pause issued in that window triggers the loss. Because XCC is a production feature used regularly, and because emergency pauses are a realistic operational event, the likelihood is **Medium**.

---

### Recommendation

1. **Atomic recovery on promise failure:** Attach a failure callback to the `withdraw_wnear_to_router` promise that credits the wNEAR back to the original sender's EVM address if the promise fails for any reason (including `ERR_PAUSED`).

2. **Pause-exempt recovery method:** Add a dedicated, pause-exempt NEAR-level method (callable only by the affected user or the engine owner) that returns stranded wNEAR from the engine's implicit EVM address to the rightful owner.

3. **Alternatively, perform the wNEAR debit inside the promise:** Defer the EVM `transferFrom` to occur inside `withdraw_wnear_to_router` rather than in the precompile, so that a failed promise never leaves the user's balance debited.

---

### Proof of Concept

1. User approves the XCC precompile to spend their wNEAR ERC-20 balance.
2. User submits an EVM transaction calling the XCC precompile with `attached_near > 0` (first call: `attached_near + STORAGE_AMOUNT` is debited).
3. The EVM commits the wNEAR `transferFrom` from the user to the engine's implicit address.
4. Before the next NEAR block executes the `withdraw_wnear_to_router` promise, the engine owner calls `pause_contract`.
5. `withdraw_wnear_to_router` executes, hits `require_running`, and panics.
6. The user's wNEAR (up to `attached_near + 2 NEAR`) remains in the engine's implicit EVM address.
7. The user attempts `submit` to transfer the wNEAR back — it fails with `ERR_PAUSED`.
8. No other recovery path exists; funds are frozen for the duration of the pause.

pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
