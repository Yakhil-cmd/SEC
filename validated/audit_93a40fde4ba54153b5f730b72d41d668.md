Audit Report

## Title
Unbounded `CANISTER_DISPLAY_NAMES` HashMap Growth via Attacker-Controlled `network_identifier.network` Causes OOM DoS — (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`)

## Summary
The `RosettaMetricsMiddleware` extracts the raw `network_identifier.network` string from every incoming POST body without any length or format validation and inserts it into the process-global `CANISTER_DISPLAY_NAMES: Mutex<HashMap<String,String>>` via `RosettaMetrics::new`. Because the map has no size cap and no eviction policy, and because the ICRC-1 Axum server applies this middleware to all routes with no authentication and no body-size limit (`usize::MAX`), an unauthenticated attacker can exhaust the Rosetta process heap by sending requests with distinct strings in that field, causing a complete denial of service.

## Finding Description
**Root cause — unbounded global map:**
`CANISTER_DISPLAY_NAMES` is declared as a process-global `lazy_static` with no capacity bound: [1](#0-0) 

`RosettaMetrics::new` unconditionally inserts the caller-supplied `canister_id` and `token_display_name` into this map on every invocation: [2](#0-1) 

**Exploit path through the middleware:**
`RosettaMetricsMiddleware::call` calls `extract_canister_id` on the raw request body, then — when a `canister_id` is found — calls `RosettaMetrics::get_display_name_from_canister_id`, which returns the ID string itself when it is not already registered, and immediately passes it to `RosettaMetrics::new`, inserting the attacker-controlled string into the map: [3](#0-2) 

`extract_canister_id` performs no length or character validation — it accepts any JSON string value at `network_identifier.network`: [4](#0-3) 

The body is buffered with `usize::MAX` as the size limit, meaning there is no effective cap on how large each individual request body can be: [5](#0-4) 

**Secondary compounding vector — Prometheus label proliferation:**
Every unique `token_display_name` passed to `inc_api_status_count` and `observe_request_duration` creates a new, never-evicted time-series in the default Prometheus registry: [6](#0-5) [7](#0-6) 

**Attack surface — all routes, no authentication, `0.0.0.0`:**
The metrics layer is applied to every route in the ICRC-1 Rosetta server, which binds to `0.0.0.0:8080` with no authentication: [8](#0-7) 

**Existing guards are insufficient:** `get_display_name_from_canister_id` only reads the map; it does not prevent insertion. There is no allowlist, no principal-format check, no entry count limit, and no body-size limit anywhere in the middleware path.

## Impact Explanation
Each unique string in `network_identifier.network` permanently occupies heap memory in two locations: the `CANISTER_DISPLAY_NAMES` HashMap and the Prometheus registry. With arbitrarily long strings (no body-size limit) and arbitrarily many requests, the Rosetta process heap grows without bound until the OS OOM-killer terminates it, causing a complete denial of service of the ICRC-1 Rosetta node. This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS of a Rosetta API node**, which is an explicitly in-scope financial integration component. This is not a raw volumetric DDoS; it exploits a specific unbounded-allocation code flaw.

## Likelihood Explanation
The attack requires only unauthenticated HTTP POST access to any endpoint (e.g., `/network/status`). No credentials, no privileged role, and no protocol knowledge beyond the Rosetta JSON schema are needed. The default bind address is `0.0.0.0`, making the port reachable from any network-adjacent host. The attack is trivially scriptable with a loop sending requests with distinct `network` strings. Each request permanently increases memory consumption; the attacker does not need to maintain connections or sustain high throughput.

## Recommendation
1. **Cap `CANISTER_DISPLAY_NAMES`**: enforce a maximum entry count (e.g., 1,000) and silently ignore insertions beyond that limit.
2. **Validate the extracted canister ID**: accept only strings matching the canonical IC principal text format (e.g., `[a-z0-9]{5}(-[a-z0-9]{5}){4}`) before inserting into the map or using as a Prometheus label.
3. **Do not construct new `RosettaMetrics` instances per request**: the middleware should look up an existing, pre-registered `RosettaMetrics` by canister ID and fall back to the default rather than calling `RosettaMetrics::new` (which inserts into the map) for every unrecognised ID.
4. **Apply a request body size limit** in the ICRC-1 Axum server (e.g., 4 MB, consistent with the ICP Actix server) by replacing `usize::MAX` with a concrete bound in `extract_canister_id`.

## Proof of Concept
```python
import requests, random, string

TARGET = "http://<rosetta-host>:8080/network/status"

for i in range(100_000):
    unique_id = ''.join(random.choices(string.ascii_lowercase, k=64)) + str(i)
    requests.post(TARGET, json={
        "network_identifier": {"blockchain": "Internet Computer", "network": unique_id},
        "metadata": {}
    })
    if i % 1000 == 0:
        print(f"Sent {i} requests")
# Monitor RSS of the rosetta process: grows proportionally to i
```

A deterministic Rust integration test can assert the invariant directly:
```rust
for i in 0..N {
    // simulate middleware call with unique network string
    let id = format!("fake-canister-id-{i}");
    let _display = RosettaMetrics::get_display_name_from_canister_id(&id);
    let _m = RosettaMetrics::new(_display, id);
}
let map = CANISTER_DISPLAY_NAMES.lock().unwrap();
assert_eq!(map.len(), N); // holds for arbitrary N, demonstrating unbounded growth
```
