Audit Report

## Title
Missing Minimum-Return Check on SNS Treasury Manager Withdraw Operation — (File: rs/sns/governance/src/extensions.rs)

## Summary
The SNS Governance canister's `ExecuteExtensionOperation` proposal action supports a `withdraw` operation that calls a treasury manager extension to pull liquidity from an external DEX. The `validate_withdraw_operation` function performs no amount or minimum-return validation, and `execute_treasury_manager_withdraw` only logs the returned balances without checking them against any threshold. Because `WithdrawRequest` in the treasury manager interface carries no `min_amounts` field, there is no on-chain mechanism for the SNS community to specify acceptable slippage bounds, and a sandwich attack between proposal passage and execution can cause the SNS treasury to receive far less than the fair-market value of its LP position.

## Finding Description

**Root cause — `validate_withdraw_operation`** accepts any `ExtensionOperationArg` and only rejects non-empty arguments; it performs no balance check and no minimum-return check:

```rust
// rs/sns/governance/src/extensions.rs L396-407
fn validate_withdraw_operation(
    _governance: &Governance,
    arg: ExtensionOperationArg,
) -> BoxFuture<'_, Result<ValidatedOperationArg, String>> {
    Box::pin(async move {
        let ExtensionOperationArg { value } = arg;
        ValidatedWithdrawOperationArg::try_from(value)
            .map(ValidatedOperationArg::TreasuryManagerWithdraw)
    })
}
```

`ValidatedWithdrawOperationArg::try_from` (L1733-1747) only checks that `original.value.is_none()` — it stores no minimum-amount fields.

**Contrast with deposit** — `validate_deposit_operation_impl` (L276-321) at least checks that the requested amounts do not exceed 50% of the current treasury balance. No analogous guard exists for withdrawals.

**Execution — `execute_treasury_manager_withdraw`** (L1612-1661) calls the treasury manager's `withdraw` endpoint, decodes the `TreasuryManagerResult`, and then only logs the returned `Balances` struct. It returns `Ok(())` unconditionally on success, regardless of how little was returned:

```rust
log!(INFO, "TreasuryManager.withdraw succeeded with response: {:?}", balances);
Ok(())
```

**Interface — `WithdrawRequest`** in `treasury_manager.did` (L88-93) carries only `withdraw_accounts`; there is no `min_amounts` or slippage field, so the SNS community cannot encode acceptable bounds in the proposal itself.

**Exploit path:**
1. A legitimate SNS `ExecuteExtensionOperation` withdraw proposal passes the voting threshold.
2. An attacker observes the pending execution and submits a large swap against the DEX pool (e.g., selling SNS tokens for ICP), moving the pool price adversely.
3. The governance timer fires; `perform_execute_extension_operation` → `validate_execute_extension_operation` → `execute_treasury_manager_withdraw` → `call_canister("withdraw", ...)`.
4. The treasury manager removes liquidity at the manipulated price; the SNS treasury receives a fraction of the expected ICP/SNS.
5. `execute_treasury_manager_withdraw` logs the result and returns `Ok(())` — no error is raised, no revert occurs.
6. The attacker reverses their swap, profiting from the price impact at the SNS treasury's expense.

**Why existing checks are insufficient:** `perform_execute_extension_operation` re-runs `validate_execute_extension_operation` at execution time (L2570-2571), but this only re-validates the deposit amount against the treasury balance for deposit operations. For withdraw operations, `validate_withdraw_operation` is called, which only checks that the argument is empty — it adds no minimum-return guard.

## Impact Explanation

The attack results in permanent, irreversible loss of SNS treasury funds (ICP and/or SNS tokens). The treasury manager holds LP positions on behalf of the SNS DAO; a successful sandwich attack can drain a significant fraction of the withdrawn position's value. This matches the allowed impact: **"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds"** and potentially **"Theft, permanent loss … involving exorbitant ICP/Cycles or in-scope chain-key/ledger assets"** depending on treasury size. Severity: **High ($2,000–$10,000)**, constrained by the requirement that a legitimate withdraw proposal must first pass governance and that the attacker must have sufficient capital to move the pool price.

## Likelihood Explanation

1. Governance proposal execution timing is publicly observable; proposals execute after passing the voting threshold via the governance canister's timer.
2. The DEX canister is permissionless — any principal can submit swap messages to it.
3. The IC's async inter-canister call model means other messages (including the attacker's swap) can be processed between the governance canister's calls to the treasury manager and the treasury manager's calls to the DEX.
4. No `min_amounts` field exists in `WithdrawRequest`, so the SNS community cannot specify slippage bounds even if they wanted to.
5. The attack is repeatable for every withdraw proposal.

## Recommendation

1. Add a `min_amounts : opt vec record { principal; nat }` field to `WithdrawRequest` in `treasury_manager.did` specifying the minimum token amounts the SNS treasury must receive.
2. Add a corresponding `min_amounts` field to `ValidatedWithdrawOperationArg` and validate its presence and structural correctness in `validate_withdraw_operation`.
3. In `execute_treasury_manager_withdraw`, after decoding the `TreasuryManagerResult`, compare the actual returned balances against the minimum amounts and return a `GovernanceError` if any threshold is not met.
4. For the deposit path, similarly add a `min_lp_tokens` or `min_amounts_deposited` field to `DepositRequest` so that the DEX deposit can be rejected if the price has moved unfavorably.

## Proof of Concept

**Deterministic integration test plan (PocketIC or local replica):**

1. Deploy an SNS governance canister with a treasury manager extension registered against a mock DEX canister.
2. Submit and pass an `ExecuteExtensionOperation` proposal with `operation_name = "withdraw"`.
3. Before the governance timer fires, submit a large adversarial swap to the mock DEX canister that moves the pool price by, e.g., 80%.
4. Trigger the governance timer to execute the proposal.
5. Assert that `execute_treasury_manager_withdraw` returns `Ok(())` (it will).
6. Assert that the SNS treasury's post-withdrawal ICP balance is significantly less than the pre-manipulation fair-market value of the LP position.
7. Assert that no `GovernanceError` was raised and no proposal failure was recorded.

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

/// Execute a treasury manager withdraw operation
async fn execute_treasury_manager_withdraw(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedWithdrawOperationArg,
) -> Result<(), GovernanceError> {
    let arg_blob = construct_treasury_manager_withdraw_payload(arg.original).map_err(|err| {
        GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!("Failed to construct treasury manager withdraw payload: {err}"),
        )
    })?;

    let balances = governance
        .env
        .call_canister(extension_canister_id, "withdraw", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.withdraw failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!(
                        "Error decoding TreasuryManager.withdraw response: {err:?}"
                    ),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.withdraw failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.withdraw succeeded with response: {:?}",
        balances
    );

    Ok(())
}

impl TryFrom<Option<Precise>> for ValidatedWithdrawOperationArg {
    type Error = String;

    fn try_from(value: Option<Precise>) -> Result<Self, Self::Error> {
        let original = value.unwrap_or_default();

        // For now, only allow empty arguments
        // This ensures withdraw operations don't accept parameters yet
        if original.value.is_some() {
            return Err("Withdraw operation does not accept arguments at this time".to_string());
        }

        Ok(Self { original })
    }
}
