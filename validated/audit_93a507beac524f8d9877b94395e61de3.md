### Title
Zero `native_per_pubdata` Due to Integer Truncation Enables Unpaid Pubdata Generation — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When the operator-set `pubdata_price` is less than `native_price`, the integer floor-division used to compute `native_per_pubdata` truncates to zero. With `native_per_pubdata == 0`, every pubdata resource check passes trivially, allowing any unprivileged user to generate the maximum block pubdata at zero cost. The operator must still publish that data to L1 without receiving any compensation.

---

### Finding Description

In `validation_impl.rs` line 142, `native_per_pubdata` is derived with plain floor division:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

Contrast this with `native_per_gas`, which uses **ceiling** division to guarantee a non-zero result whenever `gas_price > 0`:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

Whenever `pubdata_price < native_price` (e.g., `pubdata_price = 5`, `native_price = 10`), the floor division yields `native_per_pubdata = 0`. No minimum-threshold guard exists anywhere in the call chain.

With `native_per_pubdata == 0`, the two downstream enforcement points both become no-ops:

**1. `get_resources_to_charge_for_pubdata`** — multiplies pubdata bytes by zero, returning a zero-cost resource object regardless of how many bytes were written:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // 0 * anything = 0
    .ok_or(out_of_native_resources!())?;
``` [3](#0-2) 

**2. `check_enough_resources_for_pubdata`** — the `has_enough` comparison against a zero-cost resource always succeeds, so the post-execution pubdata gate never fires:

```rust
let enough = resources.has_enough(&resources_for_pubdata); // always true
``` [4](#0-3) 

The same truncation is replicated in the off-chain pre-validation helper `validate_l2_tx_intrinsic_native_resources` in `api/src/helpers.rs`, so the sequencer's admission check also fails to catch the condition:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [5](#0-4) 

The intrinsic pubdata overhead charged at resource-creation time is equally zeroed out:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata); // 0
``` [6](#0-5) 

---

### Impact Explanation

When `native_per_pubdata == 0`:

- Any unprivileged sender can submit transactions that write to storage (or otherwise generate pubdata) up to the block's `pubdata_limit` without paying a single unit of native resource for that data.
- The operator must still publish every byte to L1 (calldata or blobs), bearing the full L1 data-availability cost with zero on-chain compensation.
- Across many blocks, an attacker can continuously fill the pubdata budget for free, crowding out legitimate transactions and imposing unbounded L1 costs on the operator — a direct analogue to the "unliquidatable dust debt" accumulating protocol losses in the reference report.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is realistic:

- During periods of low L1 gas prices the operator may legitimately lower `pubdata_price` to a value smaller than `native_price`.
- The truncation is non-obvious: an operator setting `pubdata_price = 5` and `native_price = 10` intends to charge 0.5 native per pubdata byte but actually charges 0.
- No warning, assertion, or minimum-threshold guard exists in the production code path to alert the operator or reject the configuration.
- Once the condition holds, exploitation requires only a standard EVM transaction (e.g., repeated `SSTORE` calls); no special privilege is needed.

---

### Recommendation

Replace the floor division with ceiling division for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (mirrors native_per_gas; result is at least 1 when pubdata_price > 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` `validate_l2_tx_intrinsic_native_resources`. Optionally, add an explicit guard that rejects block contexts where `native_per_pubdata == 0` and `pubdata_price > 0`.

---

### Proof of Concept

1. Operator configures block with `pubdata_price = 1`, `native_price = 10` (a plausible low-L1-gas setting).
2. `native_per_pubdata = 1 / 10 = 0` (floor division).
3. Attacker submits a transaction that writes 20 distinct storage slots (each slot write produces ~65 bytes of pubdata).
4. `get_resources_to_charge_for_pubdata` returns `(~1300, Resources::from_native(0))`.
5. `check_enough_resources_for_pubdata` returns `true` regardless of remaining native budget.
6. Transaction succeeds; attacker pays only EVM gas for the `SSTORE` opcodes, pays **zero** for ~1 300 bytes of pubdata.
7. Operator publishes those bytes to L1 at their own expense.
8. Repeated across every block, the attacker fills the pubdata budget for free indefinitely.
