### Title
Integer Division Truncation in `native_per_pubdata` Computation Silently Zeroes Pubdata Native Charge - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When the block-level `pubdata_price` is less than `native_price`, the integer division `pubdata_price.wrapping_div(native_price)` truncates to `0`. This makes `native_per_pubdata = 0`, which silently removes all native-resource charges for pubdata from every transaction in that block. Any unprivileged sender can then generate the maximum allowed pubdata per transaction at zero native-resource cost, bypassing the double-resource-accounting protection that is supposed to bound prover work.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (the L2 ZK transaction validation path), `native_per_pubdata` is computed as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

This is **truncating (floor) integer division**. When `pubdata_price < native_price` (e.g., `pubdata_price = 99`, `native_price = 100`), the result is `0`.

By contrast, `native_per_gas` uses **ceiling division** to guarantee at least 1 native unit per gas:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:135
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

The same truncating pattern is replicated in the public API helper:

```rust
// api/src/helpers.rs:427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

When `native_per_pubdata = 0`, the following downstream effects occur:

1. **Intrinsic pubdata overhead is zeroed** in `create_resources_for_tx`:
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   // = 0 * intrinsic_pubdata = 0
   ``` [4](#0-3) 

2. **Post-execution pubdata check always passes** in `get_resources_to_charge_for_pubdata`:
   ```rust
   let native = current_pubdata_spent.checked_mul(native_per_pubdata)
   // = pubdata_bytes * 0 = 0 native required
   ``` [5](#0-4) 

3. **Gas refund from pubdata is zeroed** — `compute_gas_refund` derives gas-from-pubdata using `native_per_pubdata`, so no gas is charged for pubdata either.

---

### Impact Explanation

When `pubdata_price < native_price`, every L2 ZK transaction in the block can generate the maximum allowed pubdata (bounded only by the block-level `pubdata_limit`) at zero native-resource cost. The prover must still prove all state diffs, but the transaction sender pays nothing in native resources for them. This breaks the double-resource-accounting invariant: the native resource is supposed to model prover complexity, and pubdata (state diffs) is a dominant component of that cost. Undercharging for pubdata allows a sender to force the prover to do unbounded work relative to what was paid, which can be used to grief the sequencer/prover or to extract disproportionate state writes.

---

### Likelihood Explanation

`pubdata_price` and `native_price` are independent operator-set block parameters. The condition `pubdata_price < native_price` is realistic — for example, if the operator sets `native_price` to reflect high proving costs while `pubdata_price` is set low (e.g., during low L1 data-availability cost periods), the truncation silently activates. No privileged access is required by the attacker; any unprivileged transaction sender benefits automatically whenever this block-level condition holds.

---

### Recommendation

Replace the truncating division with ceiling division, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, guarantees at least 1 when pubdata_price > 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. [1](#0-0) [3](#0-2) 

---

### Proof of Concept

**Setup**: Deploy a contract that writes to 100 storage slots (generating ~3,200 bytes of pubdata). Set block context with `native_price = 1000`, `pubdata_price = 999` (so `999 / 1000 = 0` by integer division).

**Expected behavior**: The transaction should be charged `3200 * native_per_pubdata` native resources for pubdata. With correct ceiling division, `native_per_pubdata = 1`, so 3200 native units are charged.

**Actual behavior**: `native_per_pubdata = 0`. The post-execution check in `check_enough_resources_for_pubdata` computes `3200 * 0 = 0` native required, so it always passes. The transaction succeeds with zero native cost for all pubdata, regardless of how many storage slots are written.

**Contrast**: Setting `pubdata_price = 1000` (equal to `native_price`) gives `native_per_pubdata = 1` and correctly charges 3200 native. Setting `pubdata_price = 999` drops it to 0 — a 1-unit difference in operator parameters causes a complete bypass of pubdata native charging. [6](#0-5) [7](#0-6)

pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
```
