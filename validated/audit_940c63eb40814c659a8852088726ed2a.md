### Title
Silo Address Whitelist Not Enforced in `receive_base_tokens`, Bypassing Access Control for ETH Deposits - (File: engine/src/engine.rs)

### Summary

In Silo mode, the `receive_erc20_tokens` path enforces the `Address` whitelist by redirecting tokens to a configured fallback address when the recipient is not whitelisted. However, the `receive_base_tokens` path — triggered when the ETH connector calls `ft_on_transfer` — performs **no whitelist check at all**, allowing any arbitrary EVM address to receive ETH even when the Silo's Address whitelist is enabled and enforced everywhere else.

### Finding Description

The Silo mode whitelist system in Aurora Engine uses four whitelist kinds: `Admin`, `EvmAdmin`, `Account`, and `Address`. The `Address` whitelist is specifically intended to control which EVM addresses are permitted to receive tokens and submit transactions within the Silo.

In `engine/src/engine.rs`, the `receive_erc20_tokens` function correctly checks the whitelist before minting ERC-20 tokens:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

However, the `receive_base_tokens` function, which handles ETH deposits via the bridge, contains **no equivalent whitelist check**:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let receipient = message_data.recipient;
    ...
    set_balance(&mut self.io, &receipient, &new_balance);
``` [2](#0-1) 

The routing between these two paths occurs in `ft_on_transfer` in `engine/src/contract_methods/connector.rs`: if the caller is the ETH connector account, `receive_base_tokens` is called; otherwise `receive_erc20_tokens` is called. [3](#0-2) 

The `is_allow_receive_erc20_tokens` function delegates to `is_address_allowed`, which checks the `Address` whitelist: [4](#0-3) 

The whitelist is disabled by default (returns `false` when the storage key is absent), meaning the bypass only matters when the Silo operator has explicitly enabled the `Address` whitelist: [5](#0-4) 

### Impact Explanation

In a Silo deployment where the operator has enabled the `Address` whitelist to restrict which EVM addresses may receive tokens:

- **ERC-20 tokens**: correctly redirected to the fallback address when the recipient is not whitelisted.
- **ETH (base tokens)**: minted directly to any EVM address specified in the bridge message, with no whitelist enforcement.

This creates two concrete impacts:

1. **Whitelist bypass / unauthorized fund receipt**: A non-whitelisted EVM address receives ETH in the Silo, violating the operator's access control policy. If the `Account` whitelist is not enabled (or the NEAR caller is whitelisted), the non-whitelisted EVM address can then submit transactions and interact with the Silo freely.

2. **Temporary fund freeze**: If the `Address` whitelist also blocks `submit` for non-whitelisted addresses (via `assert_access` → `is_allow_submit`), the ETH deposited to the non-whitelisted address becomes inaccessible — the address holds ETH but cannot transact to move it, resulting in a temporary freeze of those funds. [6](#0-5) 

### Likelihood Explanation

Any NEAR user can trigger this path by calling `ft_transfer_call` on the ETH connector contract with an arbitrary EVM recipient address in the message. The ETH connector then calls `ft_on_transfer` on Aurora, which routes to `receive_base_tokens`. No special privilege is required. The issue is only exploitable when the Silo operator has enabled the `Address` whitelist, which is the intended production configuration for Silo deployments.

### Recommendation

Apply the same whitelist check in `receive_base_tokens` that is applied in `receive_erc20_tokens`. If the `Address` whitelist is enabled and the recipient is not whitelisted, either reject the deposit (returning the full amount to the sender) or redirect to the configured `erc20_fallback_address`, consistent with the ERC-20 behavior:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &receipient)
{
    receipient = fallback_address;
}
```

This ensures consistent whitelist enforcement across both ETH and ERC-20 deposit paths.

### Proof of Concept

1. Deploy Aurora in Silo mode. Enable the `Address` whitelist via `set_whitelist_status` with `WhitelistKind::Address`. Do **not** add address `0xDEAD...` to the whitelist.
2. Configure a fallback address via `set_erc20_fallback_address`.
3. Call `ft_transfer_call` on the ETH connector, specifying `0xDEAD...` as the recipient in the message field.
4. Aurora's `ft_on_transfer` is called by the ETH connector. Since `predecessor_account_id == get_connector_account_id`, `receive_base_tokens` is invoked.
5. `receive_base_tokens` mints ETH directly to `0xDEAD...` with no whitelist check.
6. Observe that `0xDEAD...` now holds ETH in the Silo despite not being whitelisted, while an equivalent ERC-20 transfer would have been redirected to the fallback address. [7](#0-6) [8](#0-7)

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
