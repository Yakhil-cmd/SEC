### Title
Unordered Execution of Scheduled XCC Promises Allows Permanent Consumption of Dependent Operations, Causing Temporary Fund Freeze - (File: `etc/xcc-router/src/lib.rs`)

### Summary
The XCC router's `execute_scheduled` function is intentionally open to any caller and accepts a caller-controlled `nonce` parameter. When a user schedules multiple dependent cross-contract promises (e.g., deposit then withdraw), any external actor can execute them in arbitrary order. Because the promise is removed from storage before execution, a promise executed out of order that fails is permanently consumed — the user's funds can be left locked in an external protocol with the unlock/claim promise gone.

### Finding Description
The `schedule` function in `etc/xcc-router/src/lib.rs` stores promises at sequential nonces (0, 1, 2, …) and is restricted to the Aurora Engine parent: [1](#0-0) 

The `execute_scheduled` function, however, is explicitly open to **any caller** and accepts any nonce: [2](#0-1) 

The critical detail is that `self.scheduled_promises.remove(&nonce.0)` commits the removal from storage before the cross-contract call is dispatched. In NEAR, if `execute_scheduled` itself does not panic, the state change (promise deletion) is finalized regardless of whether the dispatched cross-contract call succeeds or fails. A promise executed out of order that fails at the target contract is still permanently consumed.

The `CrossContractCallArgs::Delayed` variant in the XCC precompile is the user-facing entry point that populates the `scheduled_promises` map: [3](#0-2) 

The precompile routes `Delayed` calls to the router's `schedule` method: [4](#0-3) 

A user who schedules two separate promises — nonce 0: deposit funds into a DeFi protocol; nonce 1: withdraw or claim from that protocol — intends sequential execution. An attacker who calls `execute_scheduled(nonce=1)` first causes the withdrawal to fail at the target contract (no deposit yet), permanently consuming the promise. When nonce 0 is later executed, the deposit succeeds but the withdrawal promise no longer exists in storage.

### Impact Explanation
**High — Temporary (potentially extended) freezing of funds.**

The user's funds become locked inside an external NEAR protocol (e.g., a lending contract, a time-locked vault, or a fungible-token contract requiring `storage_deposit` before withdrawal). The unlock/claim promise has been permanently consumed. The user must manually re-schedule the operation via a new EVM transaction through the XCC precompile. If the external protocol has a time-sensitive claim window, a one-time redemption, or requires a specific predecessor for the withdrawal call, the funds may be frozen for an extended period or permanently inaccessible.

### Likelihood Explanation
**Medium.** Any NEAR account can call `execute_scheduled` on any router sub-account (of the form `{evm_address}.aurora`). Router sub-accounts and their scheduled nonces are publicly visible on-chain. An attacker only needs to monitor NEAR for `schedule` calls and immediately call `execute_scheduled` with a later nonce before the user does. No special privileges, leaked keys, or governance capture are required.

### Recommendation
Enforce sequential execution order in `execute_scheduled`: require that all nonces below the requested nonce have already been executed (i.e., are absent from `scheduled_promises`). Alternatively, restrict `execute_scheduled` to the parent Aurora Engine account (same as `schedule`), or add a minimum time-lock before a promise can be executed by a non-parent caller, giving the user priority to execute their own promises in order.

### Proof of Concept

1. User's EVM contract calls the XCC precompile with `CrossContractCallArgs::Delayed(promise_A)` — e.g., `ft_transfer_call` depositing 100 wNEAR into a lending protocol. This calls `router.schedule(promise_A)` → stored at nonce 0. [1](#0-0) 

2. User's EVM contract calls the XCC precompile again with `CrossContractCallArgs::Delayed(promise_B)` — e.g., `withdraw(100)` from the lending protocol. Stored at nonce 1.

3. Attacker observes both `schedule` calls on-chain and immediately calls:
   ```
   router.execute_scheduled({"nonce": "1"})
   ```
   The lending protocol's `withdraw` call fails (no deposit yet). The promise at nonce 1 is **permanently removed** from `scheduled_promises`. [5](#0-4) 

4. User (or attacker) calls `execute_scheduled({"nonce": "0"})`. The deposit succeeds — 100 wNEAR is now locked in the lending protocol.

5. Nonce 1 no longer exists in storage. The user's 100 wNEAR is frozen in the lending protocol. The user must submit a new EVM transaction through the XCC precompile to re-schedule the withdrawal, paying gas again and potentially missing a time-sensitive window.

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

    /// There is no security risk to allowing this function to be open because it can only
    /// act on promises that were created via `schedule`.
    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);

    }
