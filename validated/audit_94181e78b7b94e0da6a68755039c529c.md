Audit Report

## Title
Per-IP and Per-Subnet Rate Limiting Not Applied to `PATH_SUBNET_CALL_V4` â€” (File: `rs/boundary_node/ic_boundary/src/core.rs`)

## Summary
In `setup_router()`, the per-IP and per-subnet rate-limiting layers (`add_ip_rate_limiting`, `add_subnet_rate_limiting`) are applied only to `canister_call_routes` covering `PATH_CALL_V2/V3/V4`. The subnet-addressed update call endpoint `PATH_SUBNET_CALL_V4` (`/api/v4/subnet/{subnet_id}/call`) is assembled into a separate `subnet_routes` group that receives no equivalent rate-limiting layers, allowing any unprivileged external sender to bypass both configured call rate limits entirely by using the subnet-addressed call path.

## Finding Description
In `setup_router()` at `rs/boundary_node/ic_boundary/src/core.rs` lines 860â€“879, the per-IP and per-subnet rate limiters are conditionally applied to `canister_call_routes`:

```rust
let canister_call_routes = {
    let mut route = Router::new()
        .route(PATH_CALL_V2, canister_handler.clone())
        .route(PATH_CALL_V3, canister_handler.clone())
        .route(PATH_CALL_V4, canister_handler.clone());

    if let Some(rl) = cli.rate_limiting.rate_limit_per_second_per_ip {
        route = RateLimit::try_from(rl).unwrap().add_ip_rate_limiting(route);
    }
    if let Some(rl) = cli.rate_limiting.rate_limit_per_second_per_subnet {
        route = RateLimit::try_from(rl).unwrap().add_subnet_rate_limiting(route)
    }
    route
};
```

At lines 1051â€“1056, `subnet_routes` is assembled with `PATH_SUBNET_CALL_V4` and only `subnet_layers` applied â€” which contains `middleware_generic_limiter` (optional, rule-based, requires explicit YAML/canister configuration) but no `add_ip_rate_limiting` or `add_subnet_rate_limiting`:

```rust
let subnet_routes = Router::new()
    .route(PATH_SUBNET_READ_STATE_V2, subnet_handler.clone())
    .route(PATH_SUBNET_READ_STATE_V3, subnet_handler.clone())
    .route(PATH_SUBNET_QUERY_V3, subnet_handler.clone())
    .route(PATH_SUBNET_CALL_V4, subnet_handler.clone())  // no rate limiting
    .layer(subnet_layers);
```

`PATH_SUBNET_CALL_V4` is confirmed as a call/update endpoint: `RequestType::is_call()` explicitly includes `CallSubnetV4`. The CLI documentation for `rate_limit_per_second_per_ip` and `rate_limit_per_second_per_subnet` states these apply to "update calls," which `PATH_SUBNET_CALL_V4` is. The `middleware_generic_limiter` present in `subnet_layers` is not a substitute â€” it is disabled by default and requires explicit operator-supplied YAML rules to function, whereas the per-IP/per-subnet limiters are always-on when configured.

**Exploit path:**
1. Operator configures `rate_limit_per_second_per_ip` and/or `rate_limit_per_second_per_subnet` to throttle update call traffic.
2. Attacker sends all update calls to `/api/v4/subnet/{subnet_id}/call` instead of `/api/v{2,3,4}/canister/{canister_id}/call`.
3. The `GovernorLayer` rate limiters are never reached for this path; requests pass through `subnet_layers` without any per-IP or per-subnet token-bucket check.
4. The attacker submits unlimited ingress messages to the targeted subnet, exhausting the ingress pool.

## Impact Explanation
This is an application/platform-level DoS impact at the boundary node and replica ingress layer, not a raw volumetric DDoS â€” it exploits a specific structural code gap (missing rate-limiting layers on a valid update call path) to bypass a configured security control. When `rate_limit_per_second_per_ip` or `rate_limit_per_second_per_subnet` is configured, the operator's intent to throttle update call traffic is silently ineffective for the subnet call path. A single IP can flood a specific subnet's ingress pool without being throttled, causing `SERVICE_UNAVAILABLE` for legitimate users on that subnet. This matches the **High** impact category: application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS.

## Likelihood Explanation
`PATH_SUBNET_CALL_V4` is part of the publicly documented IC HTTP API spec and is reachable by any unauthenticated external sender. The bypass requires only knowledge of the alternative URL path â€” no special privileges, credentials, or protocol knowledge beyond the public API spec. Any attacker performing a targeted DoS would naturally probe all available call endpoints. The rate-limiting flags are optional, so the vulnerability is only active when an operator has configured them, but when configured, the bypass is trivially repeatable and persistent.

## Recommendation
Apply the same per-IP and per-subnet rate-limiting layers to `PATH_SUBNET_CALL_V4` as are applied to `canister_call_routes`. The simplest fix is to extract the rate-limiting application logic into a shared helper function and call it for both `canister_call_routes` and a new `subnet_call_route` sub-router before merging into `subnet_routes`, or to build a unified `call_routes` group that includes all update call endpoints (`PATH_CALL_V2`, `PATH_CALL_V3`, `PATH_CALL_V4`, `PATH_SUBNET_CALL_V4`) and applies the rate-limiting layers once to the combined group.
