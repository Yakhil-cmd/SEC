### Title
Incomplete `delta_gas` Adjustment in `compute_gas_refund` Silently Overcharges Users When Native Resource Consumption Is Below EVM Gas Consumption - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` implements a one-directional `delta_gas` adjustment: it only increases `gas_used` when native resource consumption exceeds EVM gas consumption, but never decreases it when the opposite is true. The incomplete branch is explicitly flagged with `// TODO: return delta_gas to gas_used?` at line 80. As a result, any transaction whose EVM gas consumption exceeds its native resource equivalent is silently overcharged — the user receives a smaller refund than the double-resource accounting model intends.

---

### Finding Description

In `compute_gas_refund`, after computing `delta_gas`:

```
delta_gas := (native_used / native_per_gas) - gas_used
```

only the positive branch is handled: [1](#0-0) 

```rust
#[cfg(not(feature = "unlimited_native"))]
{
    let delta_gas = if native_per_gas == 0 {
        0
    } else {
        (native_used / native_per_gas) as i64 - (gas_used as i64)
    };

    if delta_gas > 0 {
        // native > gas: charge extra gas to cover native costs
        gas_used += delta_gas as u64;
    }
    // TODO: return delta_gas to gas_used?   ← negative case unimplemented
}
```

The design documentation explicitly describes both directions of the adjustment: [2](#0-1) 

> `deltaGas := (nativeUsed / nativePerGas) - gasUsed`
> If `deltaGas > 0`, we add it to `gasUsed`…
> Finally, any remaining gas left is refunded as usual.

The doc only specifies the positive case, but the TODO comment in the code signals that the negative case was deliberately left open and unresolved. The downstream consequence is at line 83: [3](#0-2) 

```rust
let total_gas_refund = gas_limit - gas_used;
```

When `delta_gas < 0`, `gas_used` is not reduced, so `total_gas_refund` is smaller than it should be, and the user receives less ETH back.

This `RefundInfo` is consumed by both the ZK L2 transaction path: [4](#0-3) 

and the L1→L2 transaction path: [5](#0-4) 

---

### Impact Explanation

**Vulnerability class:** Resource accounting bug — asymmetric `delta_gas` adjustment causes user overpayment.

**Concrete scenario:**

A transaction performs heavy EVM computation (e.g., many `SLOAD`s, arithmetic opcodes) but writes little pubdata. Suppose:
- `gas_used` (EVM) = 100,000
- `native_used / native_per_gas` = 40,000

Then `delta_gas = -60,000`. Under the current code, `gas_used` stays at 100,000. The user is refunded for `gas_limit - 100,000` gas instead of `gas_limit - 40,000` gas. The user overpays by 60,000 gas × `gas_price` in ETH. The operator retains the difference.

The double-resource accounting model is designed so that users pay for whichever resource they actually consumed. The missing negative branch breaks this symmetry: users always pay at least the full EVM gas amount, even when native resource consumption is far lower.

---

### Likelihood Explanation

**High.** Any unprivileged L2 or L1→L2 transaction sender whose transaction is computation-heavy but pubdata-light will trigger this path. No special privileges, oracle manipulation, or governance access is required. The condition `delta_gas < 0` is a normal operating condition for many transaction types (e.g., pure computation, read-heavy contracts, ERC-20 transfers with warm storage).

---

### Recommendation

Resolve the TODO by implementing the negative branch. When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|`, subject to a floor of `minimal_gas_used` to prevent undercharging below the intrinsic minimum:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native resource consumption was lower than EVM gas consumption.
    // Return the difference to the user, but never go below minimal_gas_used.
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This ensures the user pays `max(minimal_gas_used, max(gas_used_evm, native_used / native_per_gas))` — symmetric with the positive case and consistent with the double-resource accounting model.

---

### Proof of Concept

1. Deploy any computation-heavy contract (e.g., iterative arithmetic loop) that does not write storage.
2. Submit a ZK L2 transaction calling it with a gas limit of 200,000 and a `gas_price` such that `native_per_gas` is non-zero.
3. After execution, observe `gas_used` returned in `RefundInfo`.
4. Compute `native_used / native_per_gas` independently.
5. Confirm `gas_used > native_used / native_per_gas` (i.e., `delta_gas < 0`).
6. Confirm the user's ETH refund is `(gas_limit - gas_used) * gas_price` rather than the larger `(gas_limit - native_used/native_per_gas) * gas_price`.

The existing test `test_delta_gas` in `tests/instances/transactions/src/native_charging.rs` only covers the positive `delta_gas` case (native > gas). No test covers the negative case, consistent with the unimplemented branch. [6](#0-5)
