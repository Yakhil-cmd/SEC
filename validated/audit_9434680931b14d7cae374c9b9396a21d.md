### Title
TOCTOU Race Condition in XCC Precompile Router Deployment Causes Permanent wNEAR Loss - (File: engine-precompiles/src/xcc.rs)

### Summary
The XCC precompile checks whether a router contract is deployed for a sender's EVM address at EVM execution time, but the version record update happens asynchronously via a NEAR promise callback. If a user submits two XCC transactions before the first callback executes — or calls the XCC precompile twice within a single EVM transaction — both executions see `sender_code_version = None`, both charge the user `STORAGE_AMOUNT` (2 NEAR) in wNEAR via an EVM-level `transferFrom`, but only the first promise chain succeeds. The second chain fails when `CreateAccount` panics on an already-existing account, leaving the user's wNEAR permanently stranded in the engine's implicit EVM address with no automatic refund path.

---

### Finding Description

**Step 1 — Stale read at EVM execution time**

In `engine-precompiles/src/xcc.rs`, the precompile reads the on-chain version record to decide whether to charge `STORAGE_AMOUNT`:

```rust
let required_near =
    match state::get_code_version_of_address(&self.io, &Address::new(sender)) {
        None => attached_near + state::STORAGE_AMOUNT,
        Some(_) => attached_near,
    };
``` [1](#0-0) 

If `required_near != ZERO_YOCTO`, the precompile immediately executes an EVM-level `transferFrom` that moves the user's wNEAR to the engine's implicit EVM address:

```rust
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
``` [2](#0-1) 

This EVM-level transfer is **final and irreversible** within the EVM execution context.

**Step 2 — Version record is only updated asynchronously**

The version record (`get_code_version_of_address`) is written only inside the `factory_update_address_version` callback, which is a NEAR promise that executes in a **subsequent block**:

```rust
xcc::set_code_version_of_address(&mut io, &args.address, args.version);
``` [3](#0-2) 

This callback is attached to the router deployment batch in `handle_precompile_promise`:

```rust
let callback = PromiseCreateArgs {
    target_account_id: current_account_id.clone(),
    method: "factory_update_address_version".into(),
    ...
};
Some(handler.promise_attach_callback(promise_id, &callback))
``` [4](#0-3) 

**Step 3 — Two concurrent executions both see `create_needed: true`**

Because the version record is not updated until the callback fires, any second XCC execution (whether from a second NEAR transaction in the same block, or from a second XCC precompile call within the same EVM transaction) reads the same stale `None` and enters the `DeployNeeded { create_needed: true }` branch:

```rust
let deploy_needed = AddressVersionStatus::new(io, latest_code_version, sender_code_version);
``` [5](#0-4) 

```rust
match target_code_version {
    None => Self::DeployNeeded { create_needed: true },
    ...
}
``` [6](#0-5) 

Both executions schedule a batch containing `CreateAccount + Transfer(STORAGE_AMOUNT) + DeployContract + FunctionCall(initialize)`.

**Step 4 — Second promise chain fails, wNEAR is stranded**

When the second batch executes, `CreateAccount` panics because the sub-account already exists. The entire promise chain — including `factory_update_address_version`, `withdraw_wnear_to_router`, `send_refund`, and the final `execute` — is aborted. The `withdraw_wnear_to_router` callback explicitly checks for a failed predecessor:

```rust
if matches!(handler.promise_result_check(), Some(false)) {
    return Err(b"ERR_CALLBACK_OF_FAILED_PROMISE".into());
}
``` [7](#0-6) 

The wNEAR that was already transferred from the user to the engine's implicit EVM address in Step 1 is **never unwrapped and never returned**. There is no refund branch anywhere in the promise chain for this failure mode.

---

### Impact Explanation

**Impact: Critical — Permanent freezing of user funds.**

The user's wNEAR (`STORAGE_AMOUNT` = 2 NEAR, plus any `attached_near` for the XCC call itself) is transferred to the engine's implicit EVM address at EVM execution time. When the second promise chain fails, no code path returns this wNEAR to the user. The engine's implicit EVM address is not directly accessible to users, and no public contract method exists to recover stranded wNEAR for a specific user. The funds are permanently inaccessible without manual owner intervention.

---

### Likelihood Explanation

**Likelihood: Medium.**

Two realistic trigger paths exist:

1. **Single EVM transaction**: A Solidity contract (or a user's EVM transaction) calls the XCC precompile twice in the same transaction. Both calls see the same stale storage, both charge `STORAGE_AMOUNT`, and both produce XCC logs. The engine processes both logs and creates two conflicting deployment chains.

2. **Two NEAR transactions in the same block**: A user (or automated system) submits two XCC transactions before the first callback fires. NEAR callbacks execute in subsequent blocks, so any two transactions landing in the same block trigger the condition. This is routine for bots, DeFi protocols, or any user who retries quickly.

No special privileges are required. Any EVM address using the XCC precompile for the first time is vulnerable.

---

### Recommendation

1. **Add a wNEAR refund branch**: In the `handle_precompile_promise` promise chain, attach an error-handling callback that detects a failed `CreateAccount` batch and returns the stranded wNEAR from the engine's implicit EVM address back to the sender's EVM address.

2. **Guard against duplicate deployment**: Before scheduling the `CreateAccount` batch, verify at the NEAR host level (not just engine storage) that the sub-account does not already exist, and skip the deployment if it does.

3. **Atomic version reservation**: Write a provisional "deployment in progress" marker to the version storage at EVM execution time (before the promise is created) so that subsequent concurrent executions see a non-`None` version and do not charge `STORAGE_AMOUNT` again.

### Proof of Concept

```
// Attacker-controlled Solidity contract
contract XCCRaceExploit {
    address constant XCC_PRECOMPILE = 0x516cded1d16af10cad47d6d49128e2eb7d27b372;

    // Victim calls this once; contract calls XCC precompile twice in the same tx.
    // Both calls see sender_code_version = None (no router deployed yet).
    // Both charge STORAGE_AMOUNT (2 NEAR) in wNEAR from the caller.
    // First promise chain succeeds; second fails silently.
    // Caller loses 2 NEAR worth of wNEAR with no refund.
    function triggerRace(bytes calldata xccArgs) external {
        (bool ok1,) = XCC_PRECOMPILE.call(xccArgs);
        require(ok1);
        (bool ok2,) = XCC_PRECOMPILE.call(xccArgs); // second call, same block state
        require(ok2);
        // At this point the caller has paid 2×STORAGE_AMOUNT in wNEAR.
        // The second NEAR promise chain will fail in the next block.
        // No refund is issued.
    }
}
```

**Step-by-step**:
1. Victim (or victim's contract) calls `triggerRace` with valid XCC calldata.
2. EVM executes: XCC precompile called twice; both see `sender_code_version = None`; both transfer `STORAGE_AMOUNT` wNEAR from victim to engine implicit address.
3. Engine creates two promise chains, both with `create_needed: true`.
4. Next block: first chain succeeds — router deployed, version updated.
5. Next block: second chain fails — `CreateAccount` panics on existing account; `withdraw_wnear_to_router` aborts; no refund issued.
6. Victim has lost 2 NEAR worth of wNEAR permanently.
