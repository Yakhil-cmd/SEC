Audit Report

## Title
Unbounded Iteration and Async Processing Loop in `maybe_finalize_disburse_maturity` Can Permanently Freeze SNS Maturity Disbursements - (File: rs/sns/governance/src/governance.rs)

## Summary
`maybe_finalize_disburse_maturity` in the SNS governance canister performs an unbounded scan of all neurons and then executes an unbounded loop of async cross-canister ledger transfers — one per ready disbursement — in a single periodic task execution. If the function traps at any point mid-loop (instruction exhaustion on a large neuron set, or any other panic), `is_finalizing_disburse_maturity` is never reset to `None`, permanently blocking all future finalization attempts and leaving users' maturity disbursements stuck in the canister.

## Finding Description
`maybe_finalize_disburse_maturity` (lines 4920–5083) first sets the guard flag at line 4935, then performs a full scan of `self.proto.neurons.values()` (lines 4938–4975) with no upper bound on the number of neurons examined. It then iterates over every collected entry (line 4976), performing an `.await`-ed `transfer_funds` cross-canister call for each one. The flag is only cleared at line 5082, after the loop completes normally.

There is no cap on neurons scanned or disbursements processed per invocation. The IC enforces per-message instruction limits (~20B with DTS), and while DTS can slice synchronous work across rounds, the async structure (multiple `.await` points) means each inter-canister call is a separate message execution; the synchronous scan phase itself can exhaust instructions on a sufficiently large neuron map. If the function traps for any reason — instruction exhaustion, a panic in the loop body, or a failed inter-canister call that propagates — `is_finalizing_disburse_maturity` remains `Some(true)`. `can_finalize_disburse_maturity` then returns `false` on every subsequent invocation, permanently blocking all future disbursement finalization.

The NNS governance canister already fixed this exact pattern: `finalize_maturity_disbursement` processes exactly one disbursement per timer invocation using an indexed lookup (`get_neuron_ids_ready_to_finalize_maturity_disbursement`), and `unstake_maturity_of_dissolved_neurons` caps work at `MAX_NEURONS_TO_UNSTAKE = 100`. The SNS governance has no equivalent safeguard.

`maybe_move_staked_maturity` (lines 5087–5099) also iterates all neurons without bound, though its synchronous-only nature makes instruction exhaustion less immediately severe.

## Impact Explanation
If the function traps mid-execution, all pending `DisburseMaturityInProgress` entries across all SNS neurons become permanently unprocessable via the normal periodic task path. Users who called `disburse_maturity` cannot receive their maturity tokens. This constitutes a direct loss of access to user funds held by the SNS governance canister — matching the High impact class: "Significant SNS security impact with concrete user or protocol harm" and "Application/platform-level DoS... with concrete user or protocol harm." Severity: **High ($2,000–$10,000)**.

## Likelihood Explanation
`disburse_maturity` is an unprivileged action callable by any neuron holder. An SNS with organic growth to thousands of neurons, or a coordinated burst of `disburse_maturity` calls timed to mature simultaneously, is realistic on a popular SNS. No special privileges, governance majority, or key compromise is required. The SNS governance code confirms there is no per-SNS aggregate cap on `DisburseMaturityInProgress` entries across neurons (unlike NNS's `MAX_NUM_DISBURSEMENTS`). Once triggered, the stuck flag survives until a canister upgrade resets it — but the next run under the same load will trap again.

## Recommendation
1. Cap the number of disbursements processed per invocation (e.g., `MAX_DISBURSEMENTS_PER_TICK = 50`) analogous to NNS's `MAX_NEURONS_TO_UNSTAKE = 100`.
2. Introduce a maturity disbursement index in SNS governance (as NNS does via `get_neuron_ids_ready_to_finalize_maturity_disbursement`) to avoid a full neuron scan on every tick.
3. Use a RAII guard or `defer`-style pattern to ensure `is_finalizing_disburse_maturity` is always reset to `None` even if the function panics or traps mid-execution.
4. Apply a similar bound to `maybe_move_staked_maturity`.

## Proof of Concept
1. Deploy an SNS canister (local replica or PocketIC).
2. Create N neurons (e.g., 10,000) and have each controller call `disburse_maturity`, setting `finalize_disbursement_timestamp_seconds` to the same future time T.
3. Advance the replica clock past T.
4. Trigger `run_periodic_tasks` → `maybe_finalize_disburse_maturity`.
5. Observe the synchronous scan phase iterating all N neurons, then the loop attempting N async ledger transfers.
6. At sufficient N, the instruction counter exceeds the IC limit; the message traps.
7. Query `is_finalizing_disburse_maturity`: it remains `Some(true)`.
8. All subsequent periodic task invocations return early at `can_finalize_disburse_maturity()`, confirming permanent blockage.
9. All pending maturity disbursements are stuck until a canister upgrade — after which the next run under the same load reproduces the trap.
