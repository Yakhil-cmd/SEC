Audit Report

## Title
Unbounded Memory Exhaustion via `SLOT_TABLE_NO_LIMIT` in Non-Ingress P2P Channels — (`rs/replica/setup_ic_network/src/lib.rs`, `rs/p2p/consensus_manager/src/receiver.rs`)

## Summary

Five of six artifact broadcast channels (consensus, certifier, dkg, idkg, https_outcalls) are configured with `SLOT_TABLE_NO_LIMIT = usize::MAX` as their per-peer slot table limit. A single authenticated subnet peer below the BFT fault threshold can send an unbounded stream of slot updates with distinct slot numbers and artifact IDs, causing the `slot_table`, `active_assembles`, and `artifact_processor_tasks` data structures to grow without bound, leading to OOM crash or severe performance degradation on every honest replica that receives the peer's messages, halting subnet consensus.

## Finding Description

In `rs/replica/setup_ic_network/src/lib.rs`, two constants are defined: [1](#0-0) 

Only the ingress channel is passed `SLOT_TABLE_LIMIT_INGRESS`: [2](#0-1) 

All five remaining channels — consensus (both code paths), certifier, dkg, idkg, and https_outcalls — are passed `SLOT_TABLE_NO_LIMIT`: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

In `ConsensusManagerReceiver::handle_slot_update_receive`, the slot limit guard only fires for `Entry::Vacant` when `peer_slot_table_len < self.slot_limit`. With `slot_limit = usize::MAX`, this condition is always true: [9](#0-8) 

When a new slot entry is accepted and the artifact ID is not already in `active_assembles`, a new entry is unconditionally inserted into `active_assembles` and a new Tokio task is spawned into `artifact_processor_tasks`: [10](#0-9) 

The three affected data structures are all unbounded for these channels: [11](#0-10) 

Each spawned `process_slot_update` task holds a `watch::Receiver<PeerCounter>`, a cloned `Assembler`, and a `CancellationToken`, and blocks waiting for artifact assembly or peer-counter drain. A malicious peer producing new slot updates faster than tasks complete causes monotonically increasing heap and task-stack consumption.

## Impact Explanation

A single malicious subnet node can crash every honest replica on the subnet by exhausting their memory, halting subnet consensus entirely. This matches the **High ($2,000–$10,000)** impact category: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* The attack is not volumetric DDoS — it exploits a missing resource bound in the authenticated P2P protocol state machine, requiring only a tight loop of valid protobuf-encoded slot update messages over the existing TLS-authenticated QUIC connection.

## Likelihood Explanation

The attack requires control of exactly one subnet node, which is explicitly within the IC threat model (Byzantine peer below the consensus fault threshold). The QUIC transport authenticates peers via TLS, so only subnet members can send slot updates — but a single compromised or malicious node suffices. No governance action, key compromise, or majority collusion is required. The attack is trivially scriptable: send a tight loop of `SlotUpdate` messages with incrementing `slot_id` and unique `id` fields. The five vulnerable channels (consensus, certifier, dkg, idkg, https_outcalls) are all reachable this way.

## Recommendation

Apply a finite per-peer slot limit to all five non-ingress channels, analogous to `SLOT_TABLE_LIMIT_INGRESS`. The appropriate bound for each artifact type should reflect the maximum number of in-flight artifacts a legitimate peer would ever advertise simultaneously (e.g., a small multiple of the expected consensus round depth for consensus messages, the committee size for certifications, the DKG transcript count for dkg/idkg). Additionally, consider bounding `active_assembles` independently of the slot table, since the same artifact ID can be re-inserted after a task completes (see `handle_artifact_processor_joined`). [12](#0-11) 

## Proof of Concept

```rust
// Pseudocode: malicious subnet node sends to honest peer on the consensus channel
for slot_id in 0u64.. {
    let msg = pb::SlotUpdate {
        commit_id: slot_id,
        slot_id,
        update: Some(pb::slot_update::Update::Id(
            unique_artifact_id(slot_id).encode_to_vec()
        )),
    };
    transport.send(honest_peer, "/consensus/update", msg.encode_to_vec()).await;
    // Each iteration: +1 slot_table entry, +1 active_assembles entry, +1 spawned task
}
// honest peer's heap grows without bound → OOM
```

A deterministic integration test can be written using `ConsensusManagerReceiver` directly: construct a receiver with `slot_limit = usize::MAX`, feed it N slot updates with distinct `slot_number` and `id` values from a single `peer_id`, and assert that `active_assembles.len() == N` and `artifact_processor_tasks.len() == N` after processing, confirming unbounded growth. The same test with a finite limit should cap growth at the limit and increment `slot_table_limit_exceeded_total`.
