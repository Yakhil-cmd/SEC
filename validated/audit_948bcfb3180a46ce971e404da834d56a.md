Audit Report

## Title
Cross-SNS Neurons' Fund Settlement with Attacker-Controlled Mint Destination — (`rs/nns/governance/src/governance.rs`)

## Summary

`Governance::settle_neurons_fund_participation` authorizes any legitimately deployed SNS swap canister to settle the Neurons' Fund participation for **any** NNS proposal, not just its own. Because the caller-supplied `sns_governance_canister_id` field in the `Committed` variant is used as the ICP mint destination without being validated against the proposal's stored data, a malicious SNS swap canister can redirect Neurons' Fund ICP to an attacker-controlled address and permanently lock the victim SNS's settlement state via the idempotency guard.

## Finding Description

**Root cause — authorization check is proposal-agnostic:**

`settle_neurons_fund_participation` performs two authorization steps. First, it verifies the caller is a valid `CanisterId`. Second, it calls `is_canister_id_valid_swap_canister_id`, which queries SNS-W's `list_deployed_snses` and checks only that the caller appears as *any* registered swap canister:

```rust
let is_swap = list_deployed_snses_response
    .instances
    .iter()
    .any(|sns| sns.swap_canister_id == Some(target_canister_id.into()));
```

The comment at line 7018 explicitly acknowledges the gap: *"Note that a Swap could settle each other's participation."* There is no check that the caller's canister ID matches the swap canister recorded for the specific `nns_proposal_id` supplied in the request.

**Root cause — mint destination taken from attacker-controlled request field:**

After passing the authorization check, the function extracts `sns_governance_canister_id` directly from the caller-supplied `SwapResult::Committed` variant and passes it to `mint_to_sns_governance`:

```rust
} else if let SwapResult::Committed {
    sns_governance_canister_id,
    ...
} = request.swap_result
{
    let mint_icp_result = self
        .mint_to_sns_governance(
            &request.nns_proposal_id,
            sns_governance_canister_id,   // ← attacker-controlled
            ...
        )
        .await;
```

`mint_to_sns_governance` constructs the ledger destination directly from this value:

```rust
let destination =
    AccountIdentifier::new(sns_governance_canister_id, /* subaccount = */ None);
```

There is no cross-check against the SNS governance canister ID stored in SNS-W's `DeployedSns` record for the proposal's associated SNS instance, nor against any field in the `ProposalData` itself.

**Permanent state corruption via idempotency guard:**

After a successful mint, the function sets the proposal lifecycle to `Committed` and records `final_neurons_fund_participation`. The idempotency guard at line 7166 then returns the attacker's fraudulent data to any subsequent caller, permanently preventing the legitimate swap canister from re-settling.

**`ValidatedSettleNeuronsFundParticipationRequest::try_from` does not close the gap:**

The validation at lines 1189–1241 only checks that `nns_proposal_id`, `sns_governance_canister_id`, `total_direct_participation_icp_e8s`, and `total_neurons_fund_participation_icp_e8s` are present and non-`None`. It performs no binding check between the caller identity and the proposal.

## Impact Explanation

This is a **High** severity finding matching: *"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."*

A successful attack causes:
1. **Theft of Neurons' Fund ICP**: The full matched-funding amount for the victim SNS (potentially millions of ICP e8s) is minted and transferred to an attacker-controlled address.
2. **Permanent corruption of the victim SNS**: The proposal's lifecycle is set to `Committed` and `final_neurons_fund_participation` is recorded with fraudulent data. The legitimate swap canister can never re-settle; the idempotency guard returns the attacker's data.
3. **Permanent maturity loss for NNS neuron holders**: Neurons' Fund neurons have their reserved maturity consumed without receiving corresponding SNS tokens.

The potential stolen amount could reach Critical thresholds (>$1M) depending on the size of the victim SNS's Neurons' Fund participation, but the barrier of deploying a legitimate SNS places this at High rather than Critical.

## Likelihood Explanation

Deploying a legitimate SNS requires submitting a `CreateServiceNervousSystem` NNS proposal that passes governance voting, which requires meaningful ICP stake. This is a real but reachable barrier for a motivated attacker targeting a large Neurons' Fund allocation. Once the attacker's SNS is deployed and its swap canister is registered in SNS-W, the attack requires a single crafted cross-canister call to NNS Governance. The attack window is any time between a victim SNS swap opening and its legitimate settlement call. No privileged keys, threshold corruption, or social engineering beyond the normal SNS deployment process are required.

## Recommendation

1. **Bind the caller to the proposal**: After the `is_canister_id_valid_swap_canister_id` check, additionally verify that the caller's canister ID matches the swap canister ID in the SNS-W `DeployedSns` record for the `nns_proposal_id`'s associated SNS instance. The `list_deployed_snses` response already contains both `swap_canister_id` and `governance_canister_id` per instance.

2. **Derive `sns_governance_canister_id` from authoritative data**: Do not trust the caller-supplied `sns_governance_canister_id`. Instead, look up the SNS governance canister ID from the SNS-W deployment record for the proposal's SNS instance and use that as the mint destination, ignoring the request field entirely.

3. **Remove the cross-swap settlement allowance**: The comment "a Swap could settle each other's participation" describes the current broken behavior, not an intended feature. The authorization check should enforce `caller == swap_canister_id_for_this_proposal`.

## Proof of Concept

**Setup:**
- Attacker deploys SNS-A (legitimate, registered in SNS-W). Attacker controls `swap_A`.
- Victim SNS-B exists with `nns_proposal_id = P_B`, Neurons' Fund participation reserved (`initial_neurons_fund_participation` set), swap not yet settled (`final_neurons_fund_participation = None`).

**Attack call (from `swap_A`):**
```rust
ic_cdk::call(
    NNS_GOVERNANCE_CANISTER_ID,
    "settle_neurons_fund_participation",
    SettleNeuronsFundParticipationRequest {
        nns_proposal_id: Some(P_B),          // victim's proposal
        result: Some(Committed {
            sns_governance_canister_id: Some(ATTACKER_PRINCIPAL),  // attacker's address
            total_direct_participation_icp_e8s: Some(u64::MAX),    // maximize NF participation
            total_neurons_fund_participation_icp_e8s: Some(0),
        }),
    }
)
```

**Execution path in NNS Governance:**
1. `is_canister_id_valid_swap_canister_id(swap_A)` → `Ok` (`swap_A` is a real swap canister).
2. Looks up proposal `P_B` → finds NF participation data.
3. Computes `final_neurons_fund_participation` for `P_B` using attacker-supplied `total_direct_participation_icp_e8s`.
4. Calls `mint_to_sns_governance(P_B, ATTACKER_PRINCIPAL, ...)` → ICP Ledger mints NF ICP to `ATTACKER_PRINCIPAL`.
5. Sets `P_B` lifecycle = `Committed`, records `final_neurons_fund_participation`.
6. Returns success to `swap_A`.

**Result:** Attacker receives Neurons' Fund ICP meant for SNS-B. SNS-B's proposal is permanently marked `Committed` with fraudulent data. SNS-B's legitimate swap can never re-settle (idempotency guard returns attacker's data).

**Reproducible test plan:** Write a PocketIC or `#[test]` integration test that (1) creates a proposal with `CreateServiceNervousSystem` and sets `initial_neurons_fund_participation`, (2) registers two SNS instances in a mock SNS-W, (3) calls `settle_neurons_fund_participation` from the second SNS's swap canister with the first SNS's `nns_proposal_id` and an attacker-controlled `sns_governance_canister_id`, and (4) asserts that the ICP ledger mint call targets the attacker-controlled address and that the subsequent call from the legitimate swap canister hits the idempotency guard.     fn try_from(request: SettleNeuronsFundParticipationRequest) -> Result<Self, Self::Error> {
        // Validate request.nns_proposal_id
        let validated_proposal_id = {
            if let Some(id) = request.nns_proposal_id {
                Ok(ProposalId { id })
            } else {
                Err(vec!["Request.nns_proposal_id is unspecified.".to_string()])
            }
        };
        let request_str = format!("{:#?}", &request);
        // Validate request.result
        let swap_result = if let Some(result) = request.result {
            SwapResult::try_from(result).map_err(|err| vec![err])
        } else {
            Err(vec![
                "Request.result is unspecified (must be either Committed or Aborted).".to_string(),
            ])
        };
        // Compose the validated results
        match (validated_proposal_id, swap_result) {
            (Ok(nns_proposal_id), Ok(swap_result)) => Ok(Self {
                request_str,
                nns_proposal_id,
                swap_result,
            }),
            (Ok(_), Err(proposal_type_defects)) => Err(proposal_type_defects),
            (Err(proposal_id_defects), Ok(_)) => Err(proposal_id_defects),
            (Err(proposal_id_defects), Err(proposal_type_defects)) => {
                let defects = proposal_id_defects
                    .into_iter()
                    .chain(proposal_type_defects)
                    .collect();
                Err(defects)
            }

                GovernanceError::new_with_message(

                    format!(
                        "SettleNeuronsFundParticipation is invalid for the following reason(s):\n  - {}",
                        defects.join("\n  - "),

            })
