### Title
ETH Exit-to-NEAR Permanently Burns Funds When `ft_transfer` Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

### Summary

When the Aurora Engine is compiled without the `error_refund` feature flag, an ETH (base token) exit via the `ExitToNear` precompile schedules only a bare `PromiseArgs::Create(ft_transfer)` with no callback. If the NEAR `ft_transfer` promise fails, the ETH has already been deducted from the sender's EVM balance and no on-chain mechanism exists to re-credit it. The funds are permanently lost.

---

### Finding Description

**Step 1 — Precompile builds `callback_args` with `refund: None`**

In `ExitToNear::run()`, the callback arguments are constructed conditionally on the `error_refund` feature:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

**Step 2 — For ETH exits, `transfer_near_args` is always `None`**

`exit_base_token_to_near` returns `None` for `transfer_near_args` in both the `None` (legacy `ft_transfer`) and `Omni` (`ft_transfer_call`) message branches: [2](#0-1) 

So without `error_refund`, `callback_args = { refund: None, transfer_near: None }` which equals `ExitToNearPrecompileCallbackArgs::default()`.

**Step 3 — Default `callback_args` causes no callback to be scheduled**

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← bare promise, no callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [3](#0-2) 

The promise log emitted contains only `PromiseArgs::Create(ft_transfer)`. `filter_promises_from_logs` in `engine.rs` faithfully schedules exactly what is in the log — no refund callback is appended.

**Step 4 — `refund_on_error` in `engine.rs` does not help**

`refund_on_error` handles EVM-level reverts (the EVM transaction itself failing). It does not fire when the EVM transaction succeeds but a subsequent NEAR cross-contract promise fails. Once the EVM state is committed, there is no rollback path.

**Step 5 — `MIN_INPUT_SIZE` confirms the feature changes the wire format**

```rust
#[cfg(not(feature = "error_refund"))]
const MIN_INPUT_SIZE: usize = 3;
#[cfg(feature = "error_refund")]
const MIN_INPUT_SIZE: usize = 21;
``` [4](#0-3) 

The 20-byte `refund_address` field is entirely absent from the input format when the feature is off, confirming this is a compile-time binary choice, not a runtime toggle.

---

### Impact Explanation

When `error_refund` is disabled:

1. User sends ETH to the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`).
2. EVM deducts ETH from sender's balance; EVM state is committed.
3. A bare `ft_transfer` NEAR promise is scheduled with no callback.
4. If `ft_transfer` fails (invalid recipient, connector rejection, gas exhaustion, etc.), no callback fires.
5. ETH is permanently unrecoverable — it is neither at the sender's EVM address nor transferred on NEAR.

The actual impact is **permanent loss of principal ETH**, which maps to **Critical — Permanent Freezing of Funds** (more severe than the "theft of unclaimed yield" framing in the question). Any ETH being exited — not just yield — is at risk.

---

### Likelihood Explanation

- The `error_refund` feature is a compile-time flag. If the production binary is built without it (which the explicit `#[cfg(not(feature = "error_refund"))]` branches indicate is a supported configuration), every ETH exit where the NEAR `ft_transfer` fails triggers the loss.
- `ft_transfer` can fail for mundane reasons: recipient account does not exist on NEAR, connector contract paused, or insufficient attached gas.
- No attacker action is required beyond initiating a normal ETH exit to a NEAR account that cannot receive the transfer.

---

### Recommendation

1. **Enable `error_refund` unconditionally** in the production build, or promote it from a feature flag to always-on behavior.
2. Alternatively, always wrap the `ft_transfer` promise in a `PromiseWithCallbackArgs` pointing to `exit_to_near_precompile_callback`, regardless of the feature flag, so a failed transfer always triggers a re-credit.
3. Add an integration test (local sandbox) that: (a) initiates an ETH exit to a non-existent NEAR account, (b) asserts the NEAR promise fails, and (c) asserts the sender's EVM balance is restored to the pre-exit value.

---

### Proof of Concept

```
1. Compile Aurora Engine WITHOUT the `error_refund` feature.
2. Deploy locally (sandbox).
3. Fund EVM address A with 1 ETH.
4. Call ExitToNear precompile (flag=0x00) from address A,
   targeting a NEAR account that does not exist (e.g., "nonexistent.near").
   Input: [0x00] ++ b"nonexistent.near"
5. The EVM transaction succeeds; ETH is deducted from A.
6. The scheduled ft_transfer NEAR promise fails (account not found).
7. No callback fires (callback_args == default → PromiseArgs::Create only).
8. Assert: EVM balance of A == pre-exit balance - 1 ETH  (funds lost)
   Expected (correct): EVM balance of A == pre-exit balance  (refunded)
```

The assertion at step 8 will fail, proving the ETH is permanently burned. [5](#0-4)
