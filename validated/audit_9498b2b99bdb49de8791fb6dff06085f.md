### Title
Unchecked Exit-Precompile Return Value After ERC-20 Burn Causes Permanent Fund Loss — (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

In both `EvmErc20.sol` and `EvmErc20V2.sol`, the `withdrawToNear` and `withdrawToEthereum` bridge-exit functions first burn the caller's ERC-20 tokens via `_burn()`, then invoke the Aurora exit precompile through an inline `assembly { call(...) }` block. The return value of that assembly call is captured in a local variable `res` but is **never checked**. If the precompile call fails for any reason, the ERC-20 tokens are already irreversibly burned while no corresponding NEP-141 transfer is ever scheduled, resulting in permanent loss of the user's bridged funds.

---

### Finding Description

Both contracts share the same vulnerable pattern in `withdrawToNear`:

**`EvmErc20.sol`** [1](#0-0) 

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← state change committed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — silent failure path exists
    }
}
```

**`EvmErc20V2.sol`** exhibits the identical pattern for both `withdrawToNear` and `withdrawToEthereum`: [2](#0-1) 

The EVM `call()` opcode returns `0` on failure without reverting the caller. Because `_burn()` executes *before* the assembly block, its state change (reducing the caller's ERC-20 balance to zero) is already committed to EVM storage. If the precompile call returns `0`, the Solidity function returns normally with no revert, no error event, and no refund. The ERC-20 tokens are gone; the NEAR-side promise is never created.

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) can legitimately return an error in several reachable conditions: [3](#0-2) 

- The call is made in a static or delegate context (`ERR_INVALID_IN_STATIC`, `ERR_INVALID_IN_DELEGATE`).
- The `recipient` bytes do not parse as a valid NEAR `AccountId` (e.g., too long, illegal characters, empty).
- The NEP-141 account lookup for the token fails internally.

In all these cases, `res` is `0`, the assembly block exits silently, and the function returns `Ok` to the EVM — with the user's tokens already burned.

The same flaw exists in `withdrawToEthereum` in both contracts, where the `ExitToEthereum` precompile at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` is called with the same unchecked pattern. [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user who calls `withdrawToNear` or `withdrawToEthereum` with a recipient that causes the precompile to fail will have their ERC-20 tokens burned with no recovery path:

- The ERC-20 balance is reduced to zero (burn is final).
- No NEAR promise is created, so no NEP-141 tokens are ever transferred.
- The `error_refund` callback mechanism only handles failures of the *NEAR-side promise* after it has been scheduled; it cannot help when the precompile call itself never schedules a promise.
- There is no on-chain mechanism to detect or reverse this state.

The lost tokens represent real bridged value (NEP-141 tokens locked in the Aurora engine contract on NEAR), making this a direct, permanent loss of user funds.

---

### Likelihood Explanation

**Medium.** The failure condition is reachable by any unprivileged EVM user without any special privilege:

1. A user supplies a `recipient` bytes argument that is not a valid NEAR account ID (e.g., an account name exceeding 64 characters, containing illegal characters, or being empty). The ERC-20 contract performs no validation before burning.
2. A contract calling `withdrawToNear` programmatically may construct a malformed recipient from user-supplied data.
3. The `ExitToNear` precompile's input parsing is strict; any deviation from the expected format causes a hard error that propagates as a failed `call()`.

No admin access, governance capture, or external oracle is required. Any EVM address holding a balance of the bridged ERC-20 token can trigger this path.

---

### Recommendation

Check the return value of every assembly `call()` to the exit precompiles and revert if the call fails, so that the `_burn()` is also reverted:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. This ensures that if the precompile cannot schedule the NEAR-side transfer, the entire EVM transaction reverts, preserving the user's ERC-20 balance.

---

### Proof of Concept

```
1. Alice holds 1000 units of a bridged ERC-20 token on Aurora.

2. Alice (or a contract acting on her behalf) calls:
       erc20.withdrawToNear(bytes("a".repeat(65)), 1000)
   — a recipient that exceeds NEAR's 64-character account-ID limit.

3. Inside withdrawToNear:
   a. _burn(Alice, 1000) executes → Alice's ERC-20 balance becomes 0.
   b. The ExitToNear precompile is called via assembly.
   c. The precompile fails to parse the recipient → returns error.
   d. The assembly call() returns res = 0.
   e. res is never checked; the function returns normally.

4. Result:
   - Alice's ERC-20 balance: 0  (burned, irreversible)
   - NEP-141 tokens transferred to Alice: 0  (promise never created)
   - Alice has permanently lost 1000 units of bridged value.
``` [1](#0-0) [5](#0-4) [3](#0-2)
