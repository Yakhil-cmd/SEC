### Title
Pre-EIP-155 Legacy Transactions Accepted Without Chain ID Binding Enables Cross-Chain Replay — (`engine/src/engine.rs`)

---

### Summary

Aurora Engine deliberately accepts pre-EIP-155 legacy Ethereum transactions (those with `v = 27` or `v = 28`, which carry no chain ID in their signature). The chain ID validation inside `submit_with_alt_modexp` is a conditional guard that is silently skipped when `transaction.chain_id` is `None`. Any such transaction can therefore be replayed verbatim on any other Aurora Engine instance (e.g., a different Silo deployment, a fork, or testnet) where the sender holds a balance and the nonce matches, enabling direct theft of funds.

---

### Finding Description

In `engine/src/engine.rs`, the sole replay-protection check reads:

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [1](#0-0) 

The guard fires only when `chain_id` is `Some`. For legacy (pre-EIP-155) transactions the field is `None`:

```rust
pub const fn chain_id(&self) -> Option<u64> {
    match self.v {
        0..=34 => None,          // v=27 or v=28 → no chain ID
        _ => Some((self.v - 35) / 2),
    }
}
``` [2](#0-1) 

When `EthTransactionKind::Legacy` is normalized, `chain_id` is taken directly from `tx.chain_id()`, which propagates `None` into `NormalizedEthTransaction`: [3](#0-2) 

Acceptance of pre-EIP-155 transactions is intentional — it was re-enabled in v2.6.0 to support EIP-1820 — but it was done without any restriction to the specific EIP-1820 deployer transaction:

> "Original ETH transactions which do not contain a Chain ID are allowed again to allow for use of EIP-1820" [4](#0-3) 

The `submit` and `submit_with_args` entrypoints both funnel into `submit_with_alt_modexp`, so both are affected: [5](#0-4) 

---

### Impact Explanation

**Critical — Direct theft of user funds.**

An attacker who observes a pre-EIP-155 transaction on one Aurora Engine instance (e.g., Aurora mainnet, chain ID 1313161554) can submit the identical signed bytes to any other Aurora Engine instance (e.g., a Silo deployment with a different chain ID). If the victim holds a balance on the second instance and the nonce matches, the transaction executes identically, transferring the victim's funds to the same recipient address — which the attacker controls or has arranged.

---

### Likelihood Explanation

**Low-to-Medium.**

Pre-EIP-155 transactions are uncommon with modern tooling, but the engine explicitly accepts them for all callers, not only for the EIP-1820 deployer. Realistic triggering conditions:

- Users of older hardware wallets or custom signing scripts that omit chain ID.
- Developers testing with raw RLP-encoded transactions.
- Any user whose first transaction on a new Aurora Silo instance has nonce 0, matching a pre-EIP-155 transaction already broadcast on another instance.

The Aurora ecosystem includes multiple live Engine instances (mainnet, testnet, various Silo deployments) sharing overlapping user bases, making cross-instance replay a realistic threat surface.

---

### Recommendation

Reject pre-EIP-155 transactions whose `chain_id` is `None` unless the transaction matches the exact known EIP-1820 deployer bytes. Concretely, change the guard in `submit_with_alt_modexp` from an optional check to a mandatory one:

```rust
// Require chain ID to be present and match:
let chain_id = transaction.chain_id
    .ok_or(EngineErrorKind::InvalidChainId)?;
if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

If EIP-1820 support must be preserved, add an explicit allowlist check for the known EIP-1820 deployer transaction hash before the chain ID gate.

---

### Proof of Concept

1. Alice holds 1 ETH on Aurora Silo A (chain ID 0xAA) and 1 ETH on Aurora Silo B (chain ID 0xBB). Her nonce is 0 on both.
2. Alice signs a pre-EIP-155 legacy transaction (`v = 27`) sending 1 ETH to address `0xDEAD` on Silo A. The signed bytes contain no chain ID.
3. Attacker observes the transaction on Silo A and submits the identical RLP bytes to Silo B via `submit`.
4. In `submit_with_alt_modexp`, `transaction.chain_id` is `None`; the `if let Some(chain_id)` guard at line 1055 is skipped entirely.
5. The nonce check passes (nonce 0 matches on Silo B).
6. Alice's 1 ETH on Silo B is transferred to `0xDEAD`, which the attacker controls. [6](#0-5)

pub fn submit_with_alt_modexp<
    I: IO + Copy,
    E: Env,
    P: PromiseHandler,
    M: ModExpAlgorithm + 'static,
>(
    mut io: I,
    env: &E,
    args: &SubmitArgs,
    state: EngineState,
    current_account_id: AccountId,
    relayer_address: Address,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    #[cfg(feature = "contract")]
    let transaction = NormalizedEthTransaction::try_from(
        EthTransactionKind::try_from(args.tx_data.as_slice())
            .map_err(EngineErrorKind::FailedTransactionParse)?,
    )
    .map_err(|_e| EngineErrorKind::InvalidSignature)?;

    #[cfg(not(feature = "contract"))]
    // The standalone engine must use the backwards compatible parser to reproduce the NEAR state,
    // but the contract itself does not need to make such checks because it never executes historical
    // transactions.
    let transaction: NormalizedEthTransaction = {
        let adapter =
            aurora_engine_transactions::backwards_compatibility::EthTransactionKindAdapter::new(
                ZERO_ADDRESS_FIX_HEIGHT,
            );
        let block_height = env.block_height();
        let tx: EthTransactionKind = adapter
            .try_parse_bytes(args.tx_data.as_slice(), block_height)
            .map_err(EngineErrorKind::FailedTransactionParse)?;
        tx.try_into()
            .map_err(|_e| EngineErrorKind::InvalidSignature)?
    };
    // Retrieve the signer of the transaction:
    let sender = transaction.address;

    let fixed_gas = silo::get_fixed_gas(&io);

    // Check if the sender has rights to submit transactions or deploy code.
    assert_access(&io, env, &transaction)?;

    // Validate the chain ID, if provided inside the signature:
