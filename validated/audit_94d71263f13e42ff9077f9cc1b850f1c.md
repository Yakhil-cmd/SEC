Audit Report

## Title
Hot Keys Unconditionally Inherited by Child Neuron on `spawn_neuron` with New Controller - (`rs/nns/governance/src/governance.rs`)

## Summary

When `spawn_neuron` is called with a `new_controller` that differs from the parent's controller, the child neuron is created with the parent's full `hot_keys` list copied verbatim. The previous owner's hot keys retain the ability to vote on governance proposals, change followees, and join/leave the Neuron Fund on behalf of the new controller's neuron — all without the new controller's knowledge or consent.

## Finding Description

In `spawn_neuron`, the `child_controller` is correctly set to `spawn.new_controller` when provided:

```rust
// rs/nns/governance/src/governance.rs L2651-2655
let child_controller = if let Some(child_controller) = &spawn.new_controller {
    *child_controller
} else {
    parent_neuron.controller()
};
```

However, the child neuron is then built with the parent's hot keys copied unconditionally, regardless of whether `child_controller` differs from the parent's controller:

```rust
// rs/nns/governance/src/governance.rs L2704-2718
let child_neuron = NeuronBuilder::new(
    child_nid,
    to_subaccount,
    child_controller,
    ...
)
.with_hot_keys(parent_neuron.hot_keys.clone())  // ← unconditional copy
...
```

The authorization functions confirm that hot keys are treated as trusted principals for non-trivial operations:

- `is_authorized_to_vote` returns `true` for hot keys, allowing them to vote and set followees on the child neuron.
- `is_authorized_to_configure_or_err` allows hot keys to call `JoinCommunityFund` / `LeaveCommunityFund`.

There is no post-spawn mechanism that notifies the new controller of inherited hot keys, and no automatic cleanup. The new controller must explicitly audit and remove all inherited hot keys to regain exclusive control.

Note: The `split_neuron` function always uses `*caller` as the child controller and does not accept a `new_controller` parameter, so the split path cited in the report is not currently exploitable — it is a speculative future concern only.

## Impact Explanation

This is unauthorized access to governance assets. After a legitimate controller transfer via `spawn_neuron`, the previous owner's hot keys retain the ability to:

1. **Vote on NNS governance proposals** on behalf of the new owner's neuron, influencing governance outcomes without consent.
2. **Redirect voting power** by changing the child neuron's followees.
3. **Commit the new owner's maturity to the Neuron Fund** via `JoinCommunityFund`, a financially consequential action the new controller did not authorize.

This maps to the High ($2,000–$10,000) impact class: *Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints.* The constraint here is that the attacker must have previously added hot keys to their own neuron before spawning — a common, expected workflow.

## Likelihood Explanation

The exploit path is fully unprivileged. Any neuron controller can call `manage_neuron` with `Command::Spawn { new_controller: Some(other_principal), ... }`. The `new_controller` field is a documented, supported feature. Any user who has ever added a hot key (e.g., a dapp frontend key) and then spawns maturity to a different principal — a common use case for selling or gifting maturity — triggers this condition. No special access, no victim mistake, and no social engineering is required beyond the normal spawn workflow.

## Recommendation

When `spawn_neuron` is called with a `new_controller` that differs from the parent's controller, the child neuron should be initialized with an empty hot keys list:

```rust
// rs/nns/governance/src/governance.rs, in spawn_neuron, before NeuronBuilder
let child_hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};
// Then: .with_hot_keys(child_hot_keys)
```

The new controller can then add their own hot keys. The same guard should be applied defensively to `split_neuron` if a `new_controller` parameter is ever added.

## Proof of Concept

1. Alice controls neuron N and has added hot key H (e.g., a web frontend key).
2. Alice calls `manage_neuron(N, Command::Spawn { new_controller: Some(Bob), percentage_to_spawn: 100, nonce: None })`.
3. Child neuron C is created: `controller = Bob`, `hot_keys = [H]`.
4. Alice (via H) calls `manage_neuron(C, Command::Follow(...))` — succeeds because `is_authorized_to_vote` returns `true` for hot keys.
5. Alice (via H) calls `manage_neuron(C, Command::Configure(JoinCommunityFund))` — succeeds because `is_authorized_to_configure_or_err` permits hot keys for `JoinCommunityFund`.
6. Bob has no notification of inherited hot keys and retains no exclusive control until he explicitly removes H.

A deterministic integration test can be written using the existing NNS governance test harness: create a neuron with a hot key, spawn to a new controller, then assert that the hot key principal can successfully call `Follow` and `JoinCommunityFund` on the child neuron.
