### Title
Unchecked Return Values from `eip7002_system_part` and `eip7251_system_part` Allow Silent Mid-Execution Failure, Producing an Incorrect `requests_hash` and Inconsistent Queue State — (`basic_bootloader/src/bootloader/block_flow/ethereum/post_tx_op_sequencing.rs`)

---

### Summary

In `post_tx_op_sequencing.rs`, the return values of `eip6110_events_parser`, `eip7002_system_part`, and `eip7251_system_part` are all silently discarded with `let _ =`. Both `eip7002_system_part` and `eip7251_system_part` return `Result<bool, SystemError>` and can fail **after** they have already partially mutated the shared `requests_hasher`. When they do, the error is swallowed, the hasher is left in a corrupt intermediate state, and the `requests_hash` computed from it is wrong. Additionally, the storage-level queue-head-pointer and excess-count updates that are supposed to dequeue processed requests are never applied, leaving the on-chain queue in an inconsistent state.

---

### Finding Description

In `post_tx_op_sequencing.rs` the post-block hook computes the EIP-7685 `requests_hash` by feeding three sub-routines into a shared SHA-256 hasher:

```rust
// Environment may have no such contracts predeployed for tests or sequencing purposes
let _ = eip6110_events_parser(&system, &mut requests_hasher);
let _ = eip7002_system_part(&mut system, &mut requests_hasher);
let _ = eip7251_system_part(&mut system, &mut requests_hasher);

let requests_hash = Bytes32::from_array(requests_hasher.finalize().into());
``` [1](#0-0) 

The comment acknowledges that the contracts may not be deployed, and the `let _ =` is intended to tolerate that case. However, both `eip7002_system_part` and `eip7251_system_part` can also fail **after** they have already written data into the hasher.

**Partial-write path in `eip7002_system_part`:**

1. The function first checks that the withdrawal contract is deployed (lines 56–76). If not deployed, it returns `Err` before touching the hasher — this is the intended tolerated case.
2. If the contract **is** deployed and the queue is non-empty, the function writes the EIP-7685 type byte `0x01` into the hasher unconditionally at line 177 **before** entering the per-element loop.
3. Inside the loop (lines 181–236), each element requires three `storage_read` calls. Each uses `?`, so any oracle-level failure returns `Err` immediately — **after** the type byte has already been fed to the hasher but before the element data is written.
4. The caller discards this `Err` with `let _ =`.
5. `eip7251_system_part` then runs and appends its own type byte `0x02` and element data on top of the already-corrupted hasher state. [2](#0-1) 

The same partial-write pattern exists in `eip7251_system_part`: [3](#0-2) 

**Storage-state inconsistency:** If either function fails after the loop but before the queue-head-pointer write (lines 238–272 in `eip7002_system_part`, lines 185–219 in `eip7251_consolidation_contract`), the queue head is never advanced. The same withdrawal/consolidation requests will be re-processed in the next block, violating the dequeue invariant. [4](#0-3) 

---

### Impact Explanation

1. **Incorrect `requests_hash`:** The `requests_hash` is computed from a hasher that received a type-prefix byte but not the corresponding element payloads. The resulting hash is cryptographically distinct from the correct one. This hash is part of the sealed Ethereum block header committed by `result_keeper.record_sealed_block`, causing a state-transition divergence between what the sequencer produces and what the prover or consensus layer expects.

2. **Queue state corruption:** Because the queue-head-pointer storage write is never reached when the function returns early, the same requests survive into the next block. This breaks the EIP-7002/7251 dequeue invariant and can cause double-processing of validator withdrawal or consolidation requests.

---

### Likelihood Explanation

The trigger requires that (a) the EIP-7002 or EIP-7251 predeploy contract exists in the environment (nonce == 1, bytecode present), (b) the queue is non-empty, and (c) a `storage_read` inside the per-element loop returns an error. Condition (c) is reachable by any actor who can influence oracle-provided storage data — specifically a prover/forward-execution input that omits or corrupts the storage slot values for the withdrawal/consolidation predeploy addresses. This is within the stated ZKsync OS Immunefi attack surface ("oracle-data-influencing caller," "prover/forward execution input").

---

### Recommendation

Replace the three `let _ =` discards with proper error propagation. Since the stated intent is to tolerate "contract not deployed" environments, distinguish that specific error variant and ignore only it; propagate all other errors:

```rust
match eip7002_system_part(&mut system, &mut requests_hasher) {
    Ok(_) => {}
    Err(SystemError::LeafDefect(_)) => { /* contract not deployed — tolerated */ }
    Err(e) => return Err(e.into()),
}
match eip7251_system_part(&mut system, &mut requests_hasher) {
    Ok(_) => {}
    Err(SystemError::LeafDefect(_)) => {}
    Err(e) => return Err(e.into()),
}
```

Alternatively, refactor `eip7002_system_part` and `eip7251_system_part` to write to the hasher only after all storage reads have succeeded (collect data first, hash second), so that a mid-loop failure leaves the hasher unmodified.

---

### Proof of Concept

1. Deploy the EIP-7002 withdrawal predeploy at `WITHDRAWAL_REQUEST_PREDEPLOY_ADDRESS` with nonce 1 and non-zero bytecode.
2. Enqueue at least one withdrawal request (queue tail > queue head).
3. Craft oracle data such that the first `storage_read` inside the per-element loop (slot_0 for element 0) returns a `SystemError` instead of the stored value.
4. Execute a block. `eip7002_system_part` writes `0x01` to `requests_hasher`, then returns `Err`. The error is discarded.
5. `eip7251_system_part` runs normally and appends `0x02` + its element data to the hasher.
6. `requests_hash` is finalized from `[0x01, <eip7251 data>]` instead of the correct `[0x01, <eip7002 element data>, 0x02, <eip7251 element data>]`.
7. The sealed block header contains the wrong `requests_hash`, and the withdrawal queue head pointer remains at its original value, causing the same request to be re-processed in the next block. [5](#0-4) [6](#0-5) [7](#0-6)
