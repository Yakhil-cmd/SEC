### Title
Base Token Permanently Locked at System Hook Addresses via Unauthorized Value Transfer - (File: `system_hooks/src/call_hooks/l1_messenger.rs`, `system_hooks/src/call_hooks/mint_base_token.rs`, `system_hooks/src/call_hooks/set_bytecode_on_address.rs`, `system_hooks/src/call_hooks/contract_deployer_temp.rs`)

---

### Summary

System hook addresses (`0x7001`, `0x7002`, `0x7100`, `0x8006`) return a **success** result when called by unauthorized callers ("pretend to be an empty account"). The bootloader's `perform_transfer_if_required` commits the ETH value transfer to the hook address's balance **before** the hook executes. Because the hook returns success, the transfer is never rolled back. The ETH is permanently locked at the hook address with no rescue or withdrawal mechanism.

---

### Finding Description

The bootloader runner's `perform_transfer_if_required` unconditionally transfers `nominal_token_value` from the caller to the callee **before** dispatching to the hook: [1](#0-0) 

This transfer is a committed state change. The hook is then invoked. For unauthorized callers, all four system hooks return `CallResult::Successful` via `make_return_state_from_returndata_region`: [2](#0-1) 

Contrast this with the error path, which returns `CallResult::Failed`: [3](#0-2) 

The four affected hooks and their unauthorized-caller paths:

**L1 messenger hook (`0x7001`)** — any caller that is not `0x8008` gets a success return: [4](#0-3) 

**Set-bytecode hook (`0x7002`)** — any caller that is not `0x8006` or `0x800f` gets a success return: [5](#0-4) 

**Mint-base-token hook (`0x7100`)** — any caller that is not `0x800a` gets a success return: [6](#0-5) 

**Contract-deployer hook (`0x8006`)** — any caller that is not `0x800f` gets a success return: [7](#0-6) 

None of these hook addresses implement a withdrawal or rescue function. The hook addresses are defined as: [8](#0-7) 

The design intent is documented as EVM-compatibility ("indistinguishable from a call to an empty account"): [9](#0-8) 

This EVM-compatibility design choice is the root cause: an empty-account call with value succeeds and commits the transfer in EVM semantics, so the hook correctly mimics that — but the hook address has no code to spend or forward the received ETH, making it a permanent sink.

---

### Impact Explanation

Any user who sends a transaction with non-zero `value` to one of the four system hook addresses permanently loses that ETH. The base token balance is credited to the hook address's account in the state tree and cannot be recovered: there is no `rescue`, `sweep`, or `withdraw` function on any hook address, and no bootloader-level mechanism to drain them. This is a direct, irreversible loss of user funds.

---

### Likelihood Explanation

Likelihood is **low but non-zero**. The hook addresses (`0x7001`, `0x7002`, `0x7100`, `0x8006`) are not advertised as user-facing endpoints, but:
- Developers integrating with ZKsync OS may probe these addresses with value to test behavior.
- A contract that iterates over a range of system addresses and forwards ETH (e.g., a batch-payment contract) could accidentally hit them.
- The `0x8006` address is the `CONTRACT_DEPLOYER_ADDRESS`, a well-known ZKsync address that developers may call directly with value expecting deployer behavior.

The impact per incident is bounded by the value in a single transaction, but there is no upper bound on how many times this can occur.

---

### Recommendation

Unauthorized callers that supply non-zero `nominal_token_value` should receive `make_error_return_state` (i.e., `CallResult::Failed`) rather than `make_return_state_from_returndata_region`. A failed call causes the EVM runner to roll back the value transfer. The fix is a one-line guard in each hook's unauthorized-caller branch:

```rust
// Before returning "pretend to be empty account", reject value transfers
if nominal_token_value != U256::ZERO {
    return Ok((make_error_return_state(available_resources), return_memory));
}
// Pretend to be an empty account
return Ok((make_return_state_from_returndata_region(available_resources, &[]), return_memory));
```

This preserves EVM compatibility for zero-value calls (empty-account semantics) while preventing ETH from being silently absorbed.

---

### Proof of Concept

1. User constructs a transaction: `from = <any EOA>`, `to = 0x7001` (L1 messenger hook), `value = 1 ETH`, `calldata = <anything>`.
2. Bootloader calls `perform_transfer_if_required`: transfers 1 ETH from the EOA to `0x7001` via `transfer_nominal_token_value`. Transfer succeeds; state is updated.
3. Bootloader dispatches to `l1_messenger_hook`. The caller is not `0x8008`, so the hook immediately returns `Ok((make_return_state_from_returndata_region(..., &[]), ...))` — a `CallResult::Successful`.
4. Because the call succeeded, the runner does not roll back the value transfer.
5. `0x7001`'s balance is now 1 ETH. No function exists to withdraw it. The ETH is permanently lost.

The same sequence applies identically to `0x7002`, `0x7100`, and `0x8006`.

fn make_return_state_from_returndata_region<S: SystemTypes>(
    remaining_resources: S::Resources,
    returndata: &[u8],
) -> CompletedExecution<'_, S> {
    let return_values = ReturnValues {
        returndata,
        return_scratch_space: None,
    };
    CompletedExecution {
        resources_returned: remaining_resources,
        result: CallResult::Successful { return_values },
    }
}
