Let me check a few more details about the bouncer and transport limits.

### Title
Unbounded `ingress_messages` Count in `TryFrom<pb::StrippedBlockProposal>` Enables Single Byzantine Peer to Exhaust Replica Memory and Tokio Task Scheduler - (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs`)

---

### Summary

A single Byzantine P2P peer (below the consensus fault threshold) can serve a malformed `pb::StrippedBlockProposal` containing an arbitrarily large number of fake `ingress_messages` entries under a valid, bouncer-accepted `StrippedConsensusMessageId`. Because `TryFrom<pb::StrippedBlockProposal>` performs no count bound check, the deserialization allocates an unbounded `Vec`, `BlockProposalAssembler::new` mirrors it into a second `Vec`, and `assemble_message` then spawns one tokio task per entry via `JoinSet::spawn` — each task looping indefinitely in `download_stripped_message`. This violates the invariant that a single sub-threshold peer must not exhaust replica memory or the task scheduler, and can cause OOM → replica crash → subnet finalization halt.

---

### Finding Description

**Attacker-controlled entrypoint:** A Byzantine P2P peer advertises a valid `StrippedConsensusMessageId` (wrapping a real `ConsensusMessageId` for a block proposal the victim is actively trying to download). The bouncer accepts this ID. The peer then serves a `pb::StrippedBlockProposal` with N=millions of fake `pb::StrippedIngressMessage` entries.

**Step 1 — Unbounded deserialization in `TryFrom<pb::StrippedBlockProposal>`:** [1](#0-0) 

The `ingress_messages` field is collected into a `Vec<SignedIngressId>` with no upper-bound check. The protocol limit `MAX_INGRESS_MESSAGES_PER_BLOCK = 1000` exists at the consensus validation layer but is never consulted here. [2](#0-1) 

**Step 2 — `BlockProposalAssembler::new` mirrors the Vec:** [3](#0-2) 

A second `Vec<(SignedIngressId, Option<SignedIngress>)>` of N entries is allocated. The developer comment at line 440 even acknowledges the expected bound ("at most 1000 elements") but enforces nothing: [4](#0-3) 

**Step 3 — N tokio tasks spawned with no cap:** [5](#0-4) 

One `get_or_fetch` task is spawned per entry. Each task, on a cache miss, enters `download_stripped_message`, which loops indefinitely with exponential backoff trying to fetch the (non-existent) ingress message from peers: [6](#0-5) 

**Step 4 — ID integrity check is post-hoc, not a pre-spawn guard:**

`try_assemble()` verifies the assembled block ID matches the claimed ID only after all tasks complete. By then, N tasks are already running and memory is already allocated. [7](#0-6) 

**Step 5 — No transport-level size limit:**

The `DefaultBodyLimit::disable()` on the artifact server side confirms no size cap is enforced on incoming artifact bodies: [8](#0-7) 

**The bouncer does NOT validate artifact content.** It only checks whether the `StrippedConsensusMessageId` (the block proposal hash) is wanted — not the number of ingress entries inside the artifact body. [9](#0-8) 

---

### Impact Explanation

With N = 1,000,000 fake entries:
- Each `SignedIngressId` ≈ 72 bytes → ~72 MB for the deserialized Vec
- `BlockProposalAssembler::new` allocates a second Vec → ~144 MB total
- 1,000,000 tokio tasks, each with stack space and looping network I/O → severe memory pressure and CPU exhaustion
- Tasks run until the bouncer marks the artifact `Unwanted` (when the height advances), but the attacker can repeat the attack for every new block proposal

**Scoped impact:** Replica OOM crash → subnet availability loss → consensus finalization halted. A single Byzantine node (no threshold corruption required) can sustain this attack continuously.

---

### Likelihood Explanation

- Requires only one Byzantine P2P peer, which is explicitly within the threat model (below fault threshold)
- The attacker only needs to know a valid block proposal hash being advertised on the subnet (publicly observable via P2P gossip)
- No cryptographic material, admin keys, or privileged access required
- The attack is repeatable every consensus round

---

### Recommendation

Add a count bound check immediately in `TryFrom<pb::StrippedBlockProposal>` before collecting `ingress_messages` and `stripped_idkg_dealings`, rejecting any artifact exceeding `MAX_INGRESS_MESSAGES_PER_BLOCK` (1000) and the subnet's `dkg_dealings_per_block` respectively. This mirrors the existing consensus-layer validation and closes the gap at the P2P deserialization boundary. [1](#0-0)
