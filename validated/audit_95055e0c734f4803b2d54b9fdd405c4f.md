### Title
Missing Refund Callback for Failed `ExitToNear` Promises Causes Permanent Token Freeze — (File: `engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is disabled (the default production build), the `ExitToNear` precompile creates NEAR-side transfer promises **without any failure callback**. If the promise fails, ERC-20 tokens already burned from the EVM or ETH already deducted are never restored. The refund mechanism exists in the codebase but is gated behind a non-default feature flag, making it absent in the standard contract build. Any unprivileged EVM user who triggers an exit to an unregistered or otherwise unreachable NEAR account suffers a permanent, irrecoverable loss of funds.

---

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) handles bridging of ETH and ERC-20 tokens from the Aurora EVM to NEAR. The flow is:

1. ERC-20 tokens are burned (or ETH is deducted) from the EVM state — this is irreversible within the same transaction.
2. A NEAR-side promise (`ft_transfer` or `ft_transfer_call`) is scheduled to credit the recipient on the NEP-141 side.
3. Optionally, a callback (`exit_to_near_precompile_callback`) is attached to handle promise failure and re-mint/refund the burned tokens.

The critical flaw is in how the callback is conditionally attached:

```rust
// engine-precompiles/src/native.rs lines 449–453
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← hardcoded None in default build
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For regular ERC-20 exits and ETH exits (non-wNEAR), `transfer_near_args` is also `None`, so `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`. The next branch then creates the promise **without any callback**:

```rust
// engine-precompiles/src/native.rs lines 470–483
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

If the `ft_transfer` promise fails (e.g., recipient account not registered with the NEP-141 contract, contract paused, gas exhaustion), there is no callback to invoke `refund_on_error`, so the burned tokens are permanently lost.

The `error_refund` feature is **not** in the default feature set of either `aurora-engine-precompiles` or `aurora-engine`:

```toml
# engine-precompiles/Cargo.toml line 35
default = ["std"]
error_refund = []   # ← not included in default
``` [3](#0-2) 

```toml
# engine/Cargo.toml line 43
default = ["std"]
error_refund = ["aurora-engine-precompiles/error_refund"]  # ← not included in default
``` [4](#0-3) 

The Makefile confirms the standard contract build (`clippy-contract`) uses only `--features contract`, while `error_refund` is a separate opt-in target (`clippy-contract-refund`): [5](#0-4) 

The callback handler in `engine/src/contract_methods/connector.rs` confirms that without `refund: Some(...)`, the failure path silently returns `None` — no refund, no error:

```rust
// engine/src/contract_methods/connector.rs lines 231–242
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent no-op when refund is None and promise failed
};
``` [6](#0-5) 

The test suite explicitly acknowledges this behavior as a known outcome when the feature is absent:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [7](#0-6) 

---

### Impact Explanation

**Impact: Critical — Permanent freezing of funds.**

When the `ft_transfer` or `ft_transfer_call` promise fails during an `ExitToNear` operation, the ERC-20 tokens that were burned from the EVM (or ETH deducted) are never restored. The NEP-141 tokens remain held by the Aurora contract on the NEAR side (the `ft_transfer` never completed), while the EVM-side balance is already zero. The user loses the full exit amount with no recovery path. This constitutes a permanent freeze of user funds.

---

### Likelihood Explanation

**Likelihood: High.**

Promise failures during `ft_transfer` are a realistic and common occurrence:
- The recipient NEAR account may not be registered with the NEP-141 contract (storage deposit not paid).
- The NEP-141 contract may be paused.
- Gas exhaustion on the NEAR side can cause promise failure.
- Any EVM user can trigger this path by calling the `ExitToNear` precompile — no special privileges are required.

The standard ERC-20 bridge flow (burn on Aurora, transfer on NEAR) is a core user-facing operation. The failure mode is not exotic.

---

### Recommendation

Enable the `error_refund` feature in the production contract build, or restructure the refund logic so it is unconditionally active at runtime rather than gated behind a compile-time flag. The `refund_on_error` function in `engine/src/engine.rs` already implements the correct recovery logic (re-minting burned ERC-20 tokens or returning ETH from the precompile address); it simply needs to be reachable on promise failure in all build configurations. [8](#0-7) 

---

### Proof of Concept

1. User holds ERC-20 tokens on Aurora (bridged from a NEP-141 contract).
2. User calls the `ExitToNear` precompile (flag `0x01`) specifying a NEAR recipient account that is **not registered** with the NEP-141 contract.
3. The precompile burns the ERC-20 tokens from the EVM state (irreversible).
4. Because `error_refund` is not enabled, `callback_args.refund = None` and `callback_args.transfer_near = None`, so `callback_args == default()`.
5. The promise is created as `PromiseArgs::Create(transfer_promise)` — no callback.
6. The `ft_transfer` promise fails because the recipient is unregistered.
7. No callback fires. No `refund_on_error` is called.
8. The user's ERC-20 balance on Aurora is zero; the NEP-141 balance on NEAR is unchanged (transfer never completed).
9. Funds are permanently frozen.

This is confirmed by the test `test_exit_to_near_refund` which explicitly shows that without `error_refund`, the ERC-20 balance after a failed exit is `FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT` (i.e., the exit amount is gone with no refund). [9](#0-8)

```rust
pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
}
