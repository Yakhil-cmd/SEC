### Title
Unbounded `ecrecover` Loop in EIP-7702 Authorization List Processing Exhausts NEAR Gas Before EVM Accounting - (File: engine-transactions/src/eip_7702.rs)

### Summary
`SignedTransaction7702::authorization_list()` iterates over every entry in the attacker-controlled `authorization_list` field and performs a full `ecrecover` (secp256k1 signature recovery) for each valid tuple. There is no cap on the list length. This loop executes during transaction normalization, **before** any EVM gas is charged or checked. An attacker can craft an EIP-7702 transaction with a large number of authorization tuples to exhaust the NEAR transaction gas budget (300 TGas), causing the NEAR call to fail before EVM execution begins. Because EVM gas is never charged on a failed NEAR transaction, the attacker bears no EVM cost and can repeat the attack indefinitely, draining relayer NEAR balances and temporarily preventing legitimate users from submitting transactions.

### Finding Description

**Root cause — unbounded loop with expensive cryptographic work:**

`SignedTransaction7702::authorization_list()` in `engine-transactions/src/eip_7702.rs` iterates over every `AuthorizationTuple` in `self.transaction.authorization_list`: [1](#0-0) 

For each tuple that passes the cheap pre-checks (`s ≤ SECP256K1N_HALF`, `chain_id` match, `parity ≤ 1`), the function:
1. RLP-encodes `[chain_id, address, nonce]`
2. Computes `keccak256` of the encoded bytes
3. Calls `ecrecover` — a full secp256k1 public-key recovery [2](#0-1) 

There is no size limit enforced on `authorization_list` anywhere in the struct definition or the decoder: [3](#0-2) 

**Where the loop is called — before EVM gas accounting:**

`authorization_list()` is called inside `TryFrom<EthTransactionKind> for NormalizedEthTransaction`: [4](#0-3) 

This normalization step runs at the very beginning of `submit_with_alt_modexp`, before intrinsic gas is checked and before EVM execution begins: [5](#0-4) 

The EVM `gas_limit` field in the transaction only constrains EVM execution. The NEAR gas consumed by the ecrecover loop is entirely outside EVM gas accounting.

**Attacker-controlled entry path:**

Any unprivileged user can submit an EIP-7702 transaction (type byte `0x04`) to the Aurora `submit` NEAR function. The `authorization_list` field is a plain `Vec<AuthorizationTuple>` decoded from attacker-supplied RLP bytes with no length validation. NEAR allows up to 4 MB of function-call arguments; at ~200 bytes per authorization tuple, an attacker can embed ~20,000 tuples in a single transaction.

**Why EVM gas does not protect against this:**

The intrinsic gas formula does account for authorization list length (`gas_per_auth_base_cost × len`): [6](#0-5) 

However, this check runs **after** `authorization_list()` has already completed all ecrecover calls. The NEAR gas is already spent by the time the EVM gas check could reject the transaction.

### Impact Explanation

**High — Temporary freezing of funds.**

In Aurora, relayers pay NEAR gas to submit user Ethereum transactions. If the NEAR transaction fails before EVM execution (due to NEAR gas exhaustion in the ecrecover loop), the relayer loses NEAR gas fees and receives no EVM gas compensation. The attacker's Ethereum transaction is never executed, so no EVM gas is charged to the attacker.

An attacker with no ETH balance can repeatedly submit EIP-7702 transactions with large authorization lists at essentially zero cost. Sustained flooding drains relayer NEAR balances, causing relayers to stop processing transactions. When relayers stop, legitimate users cannot submit transactions and cannot move their funds — a temporary freeze.

### Likelihood Explanation

**Medium.** The attack requires no privileged access, no ETH balance, and no on-chain state. The attacker only needs to craft valid-looking RLP-encoded EIP-7702 transactions with many authorization tuples (signatures can be arbitrary; invalid signatures still pass the cheap pre-checks and enter the ecrecover branch if `s ≤ SECP256K1N_HALF` and `parity ≤ 1`). The attack is cheap to sustain and difficult for relayers to filter without inspecting transaction internals.

### Recommendation

Enforce a maximum authorization list length before entering the ecrecover loop. EIP-7702 itself does not mandate a specific cap, but a practical bound (e.g., 255 entries, matching Ethereum client implementations) should be validated immediately after RLP decoding:

```rust
pub fn authorization_list(&self) -> Result<Vec<Authorization>, Error> {
    const MAX_AUTHORIZATION_LIST_LEN: usize = 255;
    if self.transaction.authorization_list.is_empty() {
        return Err(Error::EmptyAuthorizationList);
    }
    if self.transaction.authorization_list.len() > MAX_AUTHORIZATION_LIST_LEN {
        return Err(Error::AuthorizationListTooLong);
    }
    // ... existing loop
}
```

This bounds the NEAR gas consumed by the ecrecover loop to a fixed maximum regardless of attacker input.

### Proof of Concept

1. Craft a `Transaction7702` with `authorization_list` containing N tuples (e.g., N = 500), each with `s ≤ SECP256K1N_HALF`, `parity = 0`, and `chain_id` matching the Aurora chain ID. Signatures can be arbitrary (invalid signatures still enter the ecrecover branch).
2. RLP-encode and prepend type byte `0x04`.
3. Submit to the Aurora `submit` NEAR function via any relayer.
4. The NEAR transaction enters `submit_with_alt_modexp` → `NormalizedEthTransaction::try_from` → `authorization_list()`, which calls `ecrecover` N times before any EVM gas check.
5. For sufficiently large N, NEAR gas is exhausted; the NEAR transaction fails; the relayer loses NEAR gas fees; the attacker's EVM gas is not charged.
6. Repeat from step 1 to continuously drain relayer NEAR balances.

    pub fn authorization_list(&self) -> Result<Vec<Authorization>, Error> {
        if self.transaction.authorization_list.is_empty() {
            return Err(Error::EmptyAuthorizationList);
        }
        let current_tx_chain_id = U256::from(self.transaction.chain_id);
        let mut authorization_list = Vec::with_capacity(self.transaction.authorization_list.len());
        let mut rlp_stream = RlpStream::new();
        let mut message_bytes = vec![MAGIC; 1];
        // According to EIP-7702, we should validate each authorization. We shouldn't skip any of them.
        // And just put the `is_valid` flag to `false` if any of them is invalid. It's related to
        // gas calculation, as each `authorization_list` must be charged, even if it's invalid.
        for auth in &self.transaction.authorization_list {
            // According to EIP-7702 step 1. validation, we should verify it as
            // `chain_id = 0 || current_chain_id`.
            // AS `current_chain_id` we used `transaction.chain_id` as we will validate `chain_id` in
            // Engine `submit_transaction` method.

            // Step 2 - validation logic inside EVM itself.
            // Step 3. Checking: authority = ecrecover(keccak(MAGIC || rlp([chain_id, address, nonce])), y_parity, r, s])
            // Validate the signature, as in tests it is possible to have invalid signature values.
            // Value `v` shouldn't be greater than 1
            let mut is_valid = if auth.s > SECP256K1N_HALF {
                false
            } else {
                (auth.chain_id.is_zero() || auth.chain_id == current_tx_chain_id)
                    && auth.parity <= U256::one()
            };

            let auth_address = if is_valid {
                rlp_stream.begin_list(3);
                rlp_stream.append(&auth.chain_id);
                rlp_stream.append(&auth.address);
                rlp_stream.append(&auth.nonce);

                message_bytes.extend_from_slice(rlp_stream.as_raw());

                let signature_hash = aurora_engine_sdk::keccak(&message_bytes);
                // U256::as_u32() is safe because here we're sure that the parity <= 1.
                let v = u8::try_from(auth.parity.as_u32()).unwrap_or(u8::MAX);
                let auth_address = ecrecover(signature_hash, &super::vrs_to_arr(v, auth.r, auth.s))
                    .unwrap_or_else(|_| {
                        is_valid = false;
                        Address::default()
                    });

                message_bytes.truncate(1);
                rlp_stream.clear();

                auth_address
            } else {
                Address::default()
            };

            // Validations steps 2,4-9 0f EIP-7702 provided by EVM itself.
            authorization_list.push(Authorization {
                authority: auth_address.raw(),
                address: auth.address,
                nonce: auth.nonce,
                is_valid,
            });
        }
        Ok(authorization_list)
    }
