### Title
EVM Deviation: CREATE/CREATE2 Succeeds on Address with Non-Empty Storage Due to Missing EIP-161 Storage Check — (`evm_interpreter/src/ee_trait_impl.rs`)

### Summary

ZKsync OS's `before_executing_frame` function explicitly omits the EIP-161 storage-emptiness check for CREATE/CREATE2 collision detection. The code only checks `code_len == 0 && nonce == 0`, allowing deployment over an address that has pre-existing storage slots.

### Finding Description

The collision guard in `before_executing_frame` reads:

```rust
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    // burn gas, return CreateCollision
    return Ok(false);
}
``` [1](#0-0) 

The EVM specification (EIP-161) requires CREATE to fail if the target address has **any** of: non-zero nonce, non-zero code length, or non-empty storage. ZKsync OS checks only the first two conditions. The third — non-empty storage — is explicitly skipped with the comment "we cannot perform such a check for now." [2](#0-1) 

### Impact Explanation

An unprivileged attacker can:
1. Pre-populate storage at a deterministic CREATE2 target address (e.g., by deploying a contract there, writing storage, then self-destructing it to clear code/nonce while leaving storage intact).
2. Trigger CREATE2 to that address — ZKsync OS allows it; mainnet EVM would reject it with a collision.
3. The newly deployed contract's constructor and runtime code now observe pre-existing storage values it did not write, violating the invariant that a freshly deployed contract starts with empty storage.

This enables griefing: an attacker front-runs a victim's CREATE2 deployment by pre-seeding storage at the deterministic address, causing the victim's contract to behave incorrectly from initialization.

### Likelihood Explanation

- Requires no privileged access — any EOA can write storage to a predicted CREATE2 address.
- CREATE2 addresses are fully deterministic and publicly computable.
- The deviation is confirmed by the code comment and the missing branch in the guard.
- Practical exploitation requires the victim's contract to rely on storage being zero at startup (common pattern for initializers, proxy implementations, etc.).

### Recommendation

Implement a storage-emptiness check before allowing CREATE/CREATE2 to proceed. This requires the IO subsystem to expose a "has any non-zero storage slot" query for a given address. If a full enumeration is too expensive, a storage root / commitment check (comparing against the empty trie hash) would suffice. Until this is feasible, the deviation should be documented as a known EVM incompatibility in the protocol specification.

### Proof of Concept

1. Deploy contract `A` at address `X` via CREATE2 (salt `S`, deployer `D`).
2. In `A`'s constructor, write `storage[0] = 1`.
3. Call `A.selfdestruct()` — clears code and nonce, leaves storage slot `0 = 1` at `X`.
4. From `D`, issue CREATE2 with the same salt `S` and a new init code.
5. On ZKsync OS: deployment succeeds; the new contract at `X` sees `storage[0] == 1`.
6. On mainnet EVM: step 4 reverts with a collision error.
7. Differential test against a reference EVM (e.g., revm) confirms the gap. [3](#0-2)
