### Title
Integer Division Truncation in `native_per_pubdata` Causes Pubdata to Be Undercharged in Native Resources - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The computation of `native_per_pubdata` in the L2 transaction validation path uses truncating integer division (`wrapping_div`) instead of ceiling division (`div_ceil`). When `pubdata_price < native_price`, this truncates to `0`, causing all pubdata generated by a transaction to be charged **zero native resources**. This is a direct analog to the Unlock Protocol M-11 bug: a division that silently truncates to zero, causing a resource accounting value to be computed incorrectly and leading to under-charging.

---

### Finding Description

In `validate_and_compute_fee_for_transaction`, the bootloader computes two key resource ratios:

**`native_per_gas`** — uses ceiling division:
```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

**`native_per_pubdata`** — uses truncating (floor) division:
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

When `pubdata_price < native_price` (e.g., `pubdata_price = 2`, `native_price = 100`), `wrapping_div` returns `0`. This zero propagates into two critical downstream calculations:

1. **Intrinsic pubdata overhead** — charged upfront from the native budget:
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   // = 0 * intrinsic_pubdata = 0
   ``` [2](#0-1) 

2. **Post-execution pubdata charge** — native charged for all pubdata generated:
   ```rust
   let native = current_pubdata_spent.checked_mul(native_per_pubdata)
   // = current_pubdata_spent * 0 = 0
   ``` [3](#0-2) 

The same truncating division is replicated in the public API helper:
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [4](#0-3) 

The asymmetry is clear: `native_per_gas` uses `div_ceil` to ensure at least 1 native unit per gas, but `native_per_pubdata` uses `wrapping_div`, silently truncating to 0 when `pubdata_price < native_price`.

The documentation confirms the intended formula is `nativePerGas := gasPrice/nativePrice` (a ratio), and the same ratio logic applies to pubdata: [5](#0-4) 

---

### Impact Explanation

When `native_per_pubdata = 0`:

- The transaction's native budget is **not reduced** by any pubdata cost, regardless of how many storage writes the transaction performs.
- The `check_enough_resources_for_pubdata` check always passes (0 native required for any amount of pubdata).
- The `delta_gas` adjustment in `compute_gas_refund` that converts native consumption back to gas units also uses `native_per_gas` (not `native_per_pubdata`), so the pubdata cost is not recovered through the gas path either. [6](#0-5) 

The result: a transaction sender can generate an unbounded amount of pubdata (up to the block pubdata limit) while paying **zero native resources** for it. The operator/protocol bears the full L1 data publication cost without compensation from the transaction fee.

**Vulnerability type**: Resource accounting bug — integer division truncation causes pubdata to be free in native terms.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is reachable in normal operation. The test suite itself exercises this exact scenario:

```rust
let pubdata_price = U256::from(2);
let native_price = U256::from(100);
// native_per_pubdata = 2 / 100 = 0 by integer division
``` [7](#0-6) 

During periods of low L1 gas prices, `pubdata_price` (which tracks L1 calldata/blob cost) can fall below `native_price` (which tracks proving cost). Any unprivileged L2 transaction sender can exploit this by submitting transactions with many SSTORE operations when this condition holds.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (consistent with native_per_gas, charges at least 1 native unit per pubdata byte):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. [8](#0-7) [9](#0-8) 

---

### Proof of Concept

1. Operator sets block context: `native_price = 100`, `pubdata_price = 2` (valid during low L1 gas).
2. Attacker submits an L2 EIP-1559 transaction with `max_fee_per_gas = 100`, `gas_limit = 1_000_000`, and calldata that triggers 100 SSTORE operations (generating ~6,600 bytes of pubdata).
3. In `validate_and_compute_fee_for_transaction`:
   - `native_per_gas = ceil(100 / 100) = 1`
   - `native_per_pubdata = floor(2 / 100) = 0` ← truncation
4. `intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0` — no upfront native deducted for pubdata.
5. After execution, `get_resources_to_charge_for_pubdata` computes `6600 * 0 = 0` native for pubdata.
6. Transaction succeeds; attacker pays only computational native (for EVM execution), **zero native for 6,600 bytes of pubdata**.
7. The operator must publish this pubdata to L1 at their own cost. [10](#0-9) [11](#0-10)

pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}

First we define the ratio between EVM gas and native resource as:
  `nativePerGas := gasPrice/nativePrice`
Note: for call simulation we use a constant for it, as gasPrice might be set to 0.

Next we define the limit for the native resource as:
  `nativeLimit := gasLimit * nativePerGas`

Then we process the transaction, charging both Ergs for EE execution and native resource for any kind of computation (EE, bootloader or system work).

If execution doesn't run out of native resources, we first charge for pubdata from native resource.
Then we compute the difference between the implicit gas used derived from native resource consumption and the gas used by EEs from the ergs used. We call this value `deltaGas`.
  `deltaGas := (nativeUsed / nativePerGas) - gasUsed`
