### Title
Fee Encoding Endianness Mismatch in Legacy `ft_on_transfer` Message Parsing Silently Fails Bridge Deposits - (File: `engine-types/src/parameters/connector.rs`)

---

### Summary

The `FtTransferMessageData::try_from` parser reads the relayer fee field from the legacy `ft_on_transfer` message using `U256::from_little_endian`, but every other part of the codebase that constructs or encodes this fee uses big-endian byte order. Any non-zero fee value encoded in big-endian (the only documented and tested encoding) is silently misinterpreted as a number far exceeding `u128::MAX`, causing an `OverflowNumber` parse error. This error propagates up through `receive_base_tokens` and causes `ft_on_transfer` to refund the full bridged amount to the sender without minting any ETH on Aurora.

---

### Finding Description

In `engine-types/src/parameters/connector.rs`, the `TryFrom<&str>` implementation for `FtTransferMessageData` handles the legacy bridge message format `"{relayer_id}:{fee_hex}{address_hex}"`. The 32-byte fee field is decoded at line 88 using:

```rust
let fee_u128: u128 = U256::from_little_endian(&data[..32])
    .try_into()
    .map_err(|_| errors::ParseOnTransferMessageError::OverflowNumber)?;
``` [1](#0-0) 

The only test that exercises this code path, `test_relayer_charge_fee` in `engine-tests/src/tests/erc20_connector.rs`, encodes the fee using **big-endian**:

```rust
let fee_encoded = U256::from(fee).to_big_endian();
// ...
&format!("{}{}", recipient.encode(), hex::encode(fee_encoded))
``` [2](#0-1) 

The standalone engine test in `engine-tests/src/tests/standalone/sync.rs` also constructs the legacy message using `fee.to_bytes()` (which is big-endian for `Wei`): [3](#0-2) 

The endianness mismatch is concrete: for `fee = 51`, `U256::from(51).to_big_endian()` produces `[0x00, …, 0x00, 0x33]` (51 in the last byte). When `U256::from_little_endian` reads this, it treats the last byte as the most significant, yielding `0x33 × 256^31`, which is astronomically larger than `u128::MAX`. The `try_into::<u128>()` call then returns `Err`, triggering `OverflowNumber`.

The test covering this exact scenario is silently suppressed with `#[ignore]`: [4](#0-3) 

The error propagates through `receive_base_tokens`: [5](#0-4) 

And is caught in `ft_on_transfer`, which returns the full token amount to the sender (no ETH minted): [6](#0-5) 

---

### Impact Explanation

When a relayer submits a bridge deposit using the legacy `ft_on_transfer` message format with any non-zero fee, the Aurora engine fails to mint ETH for the recipient. The NEP-141 tokens are returned to the sender by the FT contract (since `ft_on_transfer` returns the full amount), but the user's intended bridge deposit is silently dropped. The user must retry without a relayer fee or use the new 40-character message format. During the window between submission and refund, the user's funds are in transit and unavailable. This is a **connector/bridge accounting bug** with **High — Temporary freezing of funds** impact.

---

### Likelihood Explanation

The `FtTransferFee` struct is marked `#[deprecated]` but the code path remains fully active for backward compatibility. Any external relayer or bridge client that still constructs the legacy message format with a non-zero fee (which is the entire purpose of the relayer fee mechanism) will trigger this bug on every bridge deposit. The suppressed test (`#[ignore]`) means this has never been caught by CI.

---

### Recommendation

Change line 88 of `engine-types/src/parameters/connector.rs` from `U256::from_little_endian` to `U256::from_big_endian` to match the encoding used everywhere else in the codebase:

```rust
let fee_u128: u128 = U256::from_big_endian(&data[..32])
    .try_into()
    .map_err(|_| errors::ParseOnTransferMessageError::OverflowNumber)?;
```

Additionally, remove the `#[ignore]` attribute from `test_relayer_charge_fee` once the standalone runner supports `ft_on_transfer`, and add an explicit unit test for the `TryFrom<&str>` implementation with a non-zero fee value to prevent regression.

---

### Proof of Concept

1. A relayer constructs a legacy bridge message: `"relayer.near:{fee_big_endian_hex}{recipient_hex}"` where `fee = 51` → `fee_encoded = U256::from(51).to_big_endian()` = `[0x00 × 31, 0x33]`.
2. The ETH connector calls `ft_on_transfer` on the Aurora engine with `amount = 10` and this message.
3. `receive_base_tokens` calls `FtTransferMessageData::try_from(msg)`.
4. `U256::from_little_endian([0x00 × 31, 0x33])` = `0x33 × 256^31` >> `u128::MAX`.
5. `.try_into::<u128>()` returns `Err` → `OverflowNumber` error.
6. `receive_base_tokens` returns `Err(ContractError)`.
7. `ft_on_transfer` sets `amount_to_return = 10` (full refund), returns `Ok(None)`.
8. The FT contract refunds 10 tokens to the sender. Zero ETH is minted on Aurora. The bridge deposit is silently lost. [7](#0-6) [8](#0-7)
