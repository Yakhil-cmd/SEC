### Title
Permanent ERC-20 Token Loss in `ExitToNear` Precompile When `ft_transfer` Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

### Summary

The `ExitToNear` precompile burns ERC-20 tokens from the user's EVM balance (state update) before the corresponding NEAR `ft_transfer` cross-contract call completes. When the `error_refund` compile-time feature is not enabled, no callback is attached to handle `ft_transfer` failure. If the outbound `ft_transfer` fails for any reason, the burned ERC-20 tokens are permanently destroyed with no refund, constituting a permanent freeze of user funds. This is the Aurora-Engine analog of the NFTPair checks-effects-interactions violation: state is mutated before an external call, and there is no recovery path when the external call fails.

### Finding Description

**Root cause — `engine-precompiles/src/native.rs`, `ExitToNear::run`:**

When the `ExitToNear` precompile executes, it constructs `ExitToNearPrecompileCallbackArgs` with the `refund` field conditionally compiled:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← no refund args when feature is absent
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For a standard ERC-20 `ft_transfer` exit (the most common case), `transfer_near_args` is also `None`. When both fields are `None`, `callback_args` equals the default value, so the promise is created **without any callback**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no failure callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

The ERC-20 burn already happened inside the EVM execution that called this precompile (the ERC-20 contract burns tokens and then calls the precompile). By the time the NEAR `ft_transfer` promise is dispatched, the EVM state has already been committed with the reduced balance. If `ft_transfer` fails (e.g., recipient not registered with the NEP-141 contract, insufficient gas, NEP-141 contract paused), there is no `exit_to_near_precompile_callback` to invoke `refund_on_error`, so the burned tokens are never re-minted.

**Refund path — `engine/src/engine.rs`, `refund_on_error`:**

The refund function re-mints ERC-20 tokens to the original sender, but it is only reachable through the callback, which is only attached when `error_refund` is compiled in: [3](#0-2) 

**Callback handler — `engine/src/contract_methods/connector.rs`, `exit_to_near_precompile_callback`:**

The callback only executes the refund branch when `args.refund` is `Some(...)`, which requires `error_refund` to have been compiled in: [4](#0-3) 

**Test confirmation — `engine-tests/src/tests/erc20_connector.rs`:**

The test suite explicitly documents and accepts this behavior:

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

The same pattern applies to the ETH (base token) exit path: [6](#0-5) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

When `error_refund` is not compiled into the production binary:

- Any user who calls `ExitToNear` for an ERC-20 token and whose outbound `ft_transfer` fails will have their ERC-20 tokens permanently destroyed. The NEP-141 tokens remain in the Aurora engine account (not transferred to the recipient), and the ERC-20 tokens are gone from the EVM. There is no administrative recovery path — the EVM state has been committed and no re-mint is scheduled.
- The same applies to ETH exits: ETH is transferred to the `ExitToNear` precompile address inside the EVM, and if the outbound `ft_transfer` fails without `error_refund`, the ETH is permanently locked at that precompile address.

### Likelihood Explanation

The `ft_transfer` can fail for reasons entirely outside the user's control or through predictable conditions:

1. The NEAR recipient account is not registered with the NEP-141 contract (storage deposit not paid) — a common user mistake.
2. The NEP-141 contract is paused or has been upgraded with breaking changes.
3. Insufficient gas is forwarded to the `ft_transfer` call (`FT_TRANSFER_GAS = 10 TGas`, which may be insufficient for some NEP-141 implementations).
4. The NEP-141 contract account does not exist.

Any of these conditions, which are reachable without privileged access, trigger the permanent loss. The `ExitToNear` precompile is a primary user-facing bridge exit path, making this a high-frequency risk surface.

### Recommendation

1. **Enable `error_refund` unconditionally** in all production builds, or promote it to a default Cargo feature so it cannot be accidentally omitted.
2. **Always attach the failure callback** regardless of the feature flag. The callback cost is minimal (10 TGas) and the protection is essential for bridge correctness.
3. **Alternatively**, follow the checks-effects-interactions pattern strictly: do not commit the EVM burn until the NEAR `ft_transfer` has succeeded. This would require restructuring the flow so the burn happens inside the callback on success, not before the promise is dispatched.

### Proof of Concept

1. Deploy a NEP-141 token. Bridge it to Aurora as an ERC-20.
2. Acquire ERC-20 tokens on Aurora (via `ft_transfer_call` into Aurora).
3. Call the ERC-20's `withdrawToNear` (or equivalent burn+exit function) targeting a NEAR account that has **not** paid the NEP-141 storage deposit.
4. The EVM executes: ERC-20 tokens are burned; `ExitToNear` precompile fires; a bare `ft_transfer` promise is dispatched with no callback.
5. The `ft_transfer` fails on the NEP-141 contract (unregistered recipient).
6. No callback fires. No re-mint occurs.
7. Observe: ERC-20 balance is zero; NEP-141 balance of recipient is zero; NEP-141 balance of Aurora engine is unchanged (tokens remain locked in the engine account). Funds are permanently frozen.

This is confirmed by the existing test `test_exit_to_near_refund` which explicitly asserts the reduced ERC-20 balance when `error_refund` is absent: [7](#0-6)
