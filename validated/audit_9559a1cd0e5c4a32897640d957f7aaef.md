Audit Report

## Title
Unbounded `canisters` HashMap Growth in `QuerySchedulerCore` via Arbitrary Canister IDs — (`rs/execution_environment/src/query_handler/query_scheduler/internal.rs`)

## Summary
`QuerySchedulerCore::canisters` permanently retains one `CanisterData` entry per unique `CanisterId` ever seen and never evicts them. Because canister-existence validation occurs inside the deferred worker-thread closure — after the HashMap entry is already created — an unprivileged caller can drive unbounded HashMap growth by issuing HTTP query calls with distinct, non-existent `receiver` fields, eventually exhausting replica memory.

## Finding Description
In `QuerySchedulerCore::push`, every new `canister_id` unconditionally allocates a `CanisterData` entry:

```rust
let canister = self
    .canisters
    .entry(canister_id)
    .or_insert_with(CanisterData::new);
``` [1](#0-0) 

Neither `pop` nor `notify_finished_execution` ever calls `self.canisters.remove(...)`. After a query for a non-existent canister completes (`active_threads` returns to 0, both `incoming` and `leftover` are empty), the `CanisterData` entry remains in the map indefinitely. [2](#0-1) 

The insertion point in `HttpQueryHandler::call` uses the raw `query.receiver` field and calls `query_scheduler.push` before any canister-existence check. The `CanisterNotFound` error only fires inside `internal.query(...)`, which runs later on a worker thread inside the closure: [3](#0-2) [4](#0-3) 

`QueryScheduler::push` has no size cap and no rejection path: [5](#0-4) 

## Impact Explanation
Each `CanisterData` entry holds two empty `VecDeque<Query>` allocations, a `Duration`, a `usize`, and a `bool`, plus the `CanisterId` key and HashMap overhead — approximately 100–200 bytes per entry. Accumulation is permanent: entries are never freed even after execution completes. At sustained query rates with rotating canister IDs, the replica heap grows without bound, eventually triggering OOM and crashing the query-handler process. This matches the allowed impact: **High — Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS**.

## Likelihood Explanation
The attack requires only standard unauthenticated HTTP query calls with varying `receiver` fields — no privileged access, no key material, no consensus participation. Boundary-node rate limits are typically per-IP or per-user, not per distinct canister ID, and can be circumvented by rotating source IPs. Critically, the memory leak is permanent: entries never expire even after the attack stops, so even a low sustained rate accumulates unbounded state over time.

## Recommendation
Evict `CanisterData` entries when both queues are empty and `active_threads == 0`. The natural eviction points are the end of `pop` (after draining) and `notify_finished_execution`:

```rust
if !canister.has_queries() && canister.active_threads == 0 {
    self.canisters.remove(&canister_id);
}
```

This guard should be added at the end of both `pop` and `notify_finished_execution` in `QuerySchedulerCore`. Alternatively, cap the total number of entries in `canisters` and return an error from `push` when the cap is reached. [6](#0-5) [2](#0-1) 

## Proof of Concept
```rust
// In internal.rs test module:
let scheduler = QuerySchedulerInternal::new(1, Duration::from_millis(100), &MetricsRegistry::new());
for i in 0..1_000_000u64 {
    let cid = CanisterId::from(i);
    scheduler.push(cid, Query(Box::new(|| Duration::ZERO)));
    if let Some((cid, _queries)) = scheduler.try_pop() {
        scheduler.notify_finished_execution(cid, Duration::ZERO, vec![]);
    }
}
let core = scheduler.core.lock().unwrap();
// Currently FAILS: len == 1_000_000 instead of 0
assert_eq!(core.canisters.len(), 0);
```

    fn pop(&mut self) -> Option<(CanisterId, Vec<Query>)> {
        let canister_id = self.scheduled.pop_front()?;
        // It is safe to unwrap here because of the invariants in
        // `validate_invariants()`: each canister in the round-robin list must
        // be present in the canister table.
        let canister = self.canisters.get_mut(&canister_id).unwrap();
        debug_assert!(canister.has_been_scheduled);
        canister.has_been_scheduled = false;

        let total = canister.queries_per_time_slice(self.time_slice_per_canister);

        // Collect queries from the `leftover` queue first.
        let from_leftover = total.min(canister.leftover.len());
        let mut result: Vec<_> = canister.leftover.drain(0..from_leftover).collect();

        // Get the remaining queries from the `incoming` queue.
        let from_incoming = (total - from_leftover).min(canister.incoming.len());
        result.extend(canister.incoming.drain(0..from_incoming));

        // Follows from the main invariant.
        debug_assert!(!result.is_empty());

        canister.active_threads += 1;

        // We removed the canister from the schedule at the beginning of this
        // method and cleared `has_been_scheduled`.
        debug_assert!(!canister.has_been_scheduled);
        if canister.should_be_scheduled(self.max_threads_per_canister) {
            canister.has_been_scheduled = true;
            self.scheduled.push_back(canister_id);
        }

        #[cfg(debug_assertions)]
        self.verify_invariants();

        Some((canister_id, result))
    }
