Audit Report

## Title
Silent ICP Burn Failure in `burn_and_log` Causes Permanent ICP Supply Inflation - (File: rs/nns/cmc/src/main.rs)

## Summary
The `burn_and_log` function in the Cycles Minting Canister unconditionally swallows all errors from the ICP ledger burn call. When the ICP ledger is temporarily unavailable (e.g., during a canister upgrade), cycles are already minted and deposited but the corresponding ICP is never burned, permanently inflating the ICP total supply. The block is marked as fully processed, so neither the user nor the CMC can retry the burn.

## Finding Description
After a successful cycles operation, `process_top_up` (L1999–2002), `process_mint_cycles` (L1966–1968), and `process_create_canister` (L1943–1946) each call `burn_and_log`. The function is explicitly designed to never propagate errors:

```rust
// rs/nns/cmc/src/main.rs L2014-2016
/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
```

The `call_protobuf` result at L2040 is matched only for logging; errors are silently discarded (L2044–2047). The function always returns `()`.

Because `burn_and_log` always returns `Ok`-equivalent, the callers always return `Ok(cycles)`/`Ok(canister_id)`/`Ok(NotifyMintCyclesSuccess{…})`. The notify handlers then unconditionally write a terminal `NotifiedTopUp(Ok(…))`/`NotifiedMint(Ok(…))`/`NotifiedCreateCanister(Ok(…))` status into `blocks_notified` (L1214–1222, L1305–1313, L1411–1419). The `is_transient_error` guard at L1219/L1310/L1416 only removes the entry on transient errors returned from the process function — a successful cycles deposit with a failed burn is not a transient error, so the entry is never removed. The block is permanently sealed. No retry path exists for the burn.

The exploit path:
1. User sends N ICP to `CMC_SUBACCOUNT(canister_id)` with `MEMO_TOP_UP_CANISTER`.
2. User calls `notify_top_up { block_index, canister_id }`.
3. CMC sets block to `Processing`, calls `process_top_up`.
4. `deposit_cycles` succeeds — cycles are in circulation.
5. `burn_and_log` calls `send_pb` on the ICP ledger — ledger is mid-upgrade, call rejected.
6. Error is logged and discarded; `burn_and_log` returns `()`.
7. CMC writes `NotifiedTopUp(Ok(cycles))` — block permanently sealed.
8. N ICP remain in `CMC_SUBACCOUNT(canister_id)`, unburned. ICP total supply is N ICP higher than protocol intends. User cannot retry.

## Impact Explanation
This is a concrete ICP ledger accounting violation. Each successful exploit permanently inflates the ICP total supply by the amount of the failed burn. The ICP is not recoverable: it is stranded in the CMC's subaccount, inaccessible to the user and not retried by the CMC. Repeated occurrences (e.g., across multiple ledger upgrade windows or coordinated timing) compound the inflation. This constitutes a significant NNS/ICP ledger security impact with concrete protocol harm, matching the **High ($2,000–$10,000)** impact class: "Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
NNS proposals for ICP ledger upgrades are publicly visible on-chain before execution. An unprivileged user can monitor the NNS proposal queue, send ICP to the CMC, and time a `notify_top_up`/`notify_mint_cycles`/`notify_create_canister` call to reach the `burn_and_log` phase during the ledger's brief upgrade window. No special privileges, leaked keys, or social engineering are required. The condition also arises accidentally for any user whose notification happens to race a ledger upgrade. The attack is repeatable across every ledger upgrade cycle.

## Recommendation
Replace the fire-and-forget pattern with one of:

1. **Persist and retry**: Record failed burns in CMC stable state and retry them in the heartbeat, similar to how ckBTC handles pending operations. The `blocks_notified` deduplication map already prevents double-spending on the cycles side.
2. **Pre-burn before minting**: Perform the ICP burn before minting cycles. A burn failure then prevents cycles from being issued rather than leaving ICP unburned after the fact.
3. **Propagate with selective retry**: Return a transient error to the caller on ledger unavailability, and use the existing `is_transient_error` + `blocks_notified.remove` path (already used for other transient errors at L1219–1221) to allow the user to retry — but only clear `Processing` on transient ledger errors, not on permanent ones, to preserve the double-spend protection.

## Proof of Concept
A deterministic PocketIC or StateMachine integration test can reproduce this:

1. Initialize NNS with ICP ledger and CMC.
2. Fund a test user account with N ICP.
3. Transfer N ICP from the test user to `CMC_SUBACCOUNT(target_canister)` with `MEMO_TOP_UP_CANISTER`; record `block_index`.
4. Stop the ICP ledger canister (simulating an upgrade window).
5. Call `notify_top_up { block_index, canister_id: target_canister }` from the test user.
6. Assert the call returns `Ok(cycles)` — cycles are deposited.
7. Assert `blocks_notified[block_index]` is `NotifiedTopUp(Ok(cycles))` — block is sealed.
8. Restart the ICP ledger.
9. Assert the ICP ledger has no burn record for the N ICP (no `send_pb` block from CMC to minting account).
10. Assert the CMC subaccount for `target_canister` still holds N ICP.
11. Assert the ICP total supply equals the initial supply (N ICP was not burned — supply is inflated by N ICP).
