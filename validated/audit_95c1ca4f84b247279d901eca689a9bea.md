### Title
Unprotected Pre-EIP-155 Legacy Transactions Accepted Without Chain ID Validation Enables Cross-Chain Replay Attacks — (`basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

ZKsync OS accepts legacy transactions with `v = 27` or `v = 28` (pre-EIP-155, "unprotected") without any chain ID binding. Because the signing hash for these transactions omits the chain ID entirely, a valid signature on one chain is equally valid on every other EVM-compatible chain. An attacker who observes an unprotected legacy transaction on Ethereum mainnet (or any other chain) can replay it verbatim on ZKsync OS, draining the sender's funds without their authorization, provided the sender's nonce and balance match on ZKsync OS.

---

### Finding Description

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, the signing hash is computed over only the 6-field payload `[nonce, gasPrice, gasLimit, to, value, data]` when `legacy_signature.is_eip155() == false`:

```rust
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    // Unprotected legacy
    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);
    hasher.finalize_reset().into()
} else {
    // EIP-155 protected: chain_id is mixed into the hash
    ...
}
``` [1](#0-0) 

The chain ID is never mixed into the hash for the unprotected path. The caller (`parse_and_compute_signed_hash`) then accepts the result as `Self::Legacy` with no chain ID check:

```rust
let tx = if sig_data.is_eip155() {
    Self::LegacyWithEIP155(tx, sig_data)
} else {
    Self::Legacy(tx, sig_data)
};
Ok((tx, sig_hash))
``` [2](#0-1) 

This is in direct contrast to every typed transaction variant (EIP-1559, EIP-2930, EIP-4844, EIP-7702), all of which explicitly enforce `tx.chain_id == expected_chain_id` before returning:

```rust
if tx.chain_id != expected_chain_id {
    return Err(InvalidTransaction::InvalidChainId.into());
}
``` [3](#0-2) 

The `chain_id()` accessor confirms the asymmetry — it returns `None` only for the unprotected `Legacy` variant, meaning no downstream code can distinguish "no chain ID" from "chain ID not yet checked":

```rust
pub fn chain_id(&self) -> Option<u64> {
    match &self.inner {
        RlpEncodedTxInner::Legacy(_, _) => None,
        _ => Some(self.chain_id),
    }
}
``` [4](#0-3) 

The validation path in `validate_and_compute_fee_for_transaction` performs signature recovery against `suggested_signed_hash` and checks nonce/balance, but never rejects a transaction whose `chain_id()` is `None`: [5](#0-4) 

---

### Impact Explanation

An attacker can take any unprotected legacy transaction broadcast on Ethereum mainnet (or any other EVM chain) and submit it to ZKsync OS. If the original sender:

- has the same address on ZKsync OS (always true — addresses are deterministic),
- has a matching nonce on ZKsync OS (common for new accounts, nonce = 0), and
- holds sufficient balance on ZKsync OS,

then the transaction executes successfully, transferring the sender's ZKsync OS funds to the original `to` address without the sender's knowledge or consent. This is a direct, unauthorized loss of user funds — the highest-severity impact class.

---

### Likelihood Explanation

Pre-EIP-155 transactions are rare in modern wallets but are not extinct. They appear in:

- Transactions created by older tooling or hardware wallets that predate EIP-155,
- Deliberate "chain-agnostic" deployments (e.g., deterministic deployer contracts like `0x4e59b44847b379578588920cA78FbF26c0B4956C`),
- Any user who manually constructs a raw transaction without a chain ID.

The nonce-matching requirement limits the attack to cases where the victim's ZKsync OS nonce equals the nonce in the replayed transaction. For new ZKsync OS accounts (nonce = 0), this is trivially satisfied by any unprotected legacy transaction that was the sender's first transaction on the source chain. An attacker only needs to monitor public mempools/block explorers for such transactions and check whether the sender has a ZKsync OS balance.

---

### Recommendation

Reject unprotected legacy transactions at the parsing layer. In `parse_and_compute_signed_hash`, after the legacy path, add:

```rust
if !sig_data.is_eip155() {
    return Err(InvalidTransaction::InvalidChainId.into());
}
```

Alternatively, if backward compatibility with pre-EIP-155 tooling is required, gate acceptance behind an explicit configuration flag (`Config::ALLOW_UNPROTECTED_LEGACY`) that is `false` in production. The `docs/not-a-bug.md` file should document whichever policy is chosen.

---

### Proof of Concept

1. Alice holds 10 ETH on ZKsync OS at address `0xABC`, nonce = 0.
2. Alice previously sent an unprotected legacy transaction on Ethereum mainnet (nonce = 0, `v = 27`): `transfer 5 ETH to 0xEVIL`.
3. Attacker copies the raw RLP bytes of Alice's Ethereum transaction.
4. Attacker submits those bytes to ZKsync OS.
5. ZKsync OS parses the transaction as `RlpEncodedTxInner::Legacy`, computes the signing hash over the 6-field payload (no chain ID), recovers `from = 0xABC`, checks nonce (0 == 0 ✓), checks balance (10 ETH ≥ 5 ETH ✓), and executes the transfer.
6. Alice loses 5 ETH on ZKsync OS without having authorized any ZKsync OS transaction. [6](#0-5) [7](#0-6)
