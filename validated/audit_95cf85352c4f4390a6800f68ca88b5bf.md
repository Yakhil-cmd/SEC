Audit Report

## Title
Malicious Connected Peer Can Permanently Poison Block Cache With Empty-txdata Block, Causing ckBTC to Miss Deposits — (`rs/bitcoin/adapter/src/blockchainmanager.rs`, `rs/bitcoin/adapter/src/blockchainstate.rs`)

## Summary

`received_block_message` removes a block hash from `getdata_request_info` on first receipt without verifying the responding peer matches the one the `getdata` was addressed to. Simultaneously, `add_block` skips merkle root validation when `txdata` is empty because `compute_merkle_root()` returns `None` for an empty transaction list. A connected malicious Bitcoin peer can respond to any outstanding `getdata` with a crafted block carrying the real header (valid hash) but `txdata=[]`, permanently occupying the cache slot and causing the ckBTC canister to process a block with zero transactions, silently dropping all Bitcoin deposits in that block.

## Finding Description

**Root cause 1 — No per-peer matching in `received_block_message`.** [1](#0-0) 

The only peer check is `peer_info.contains_key(addr)` — any connected peer passes. The removed `request` contains a `socket` field recording which peer the `getdata` was sent to, but `addr` is never compared against `request.socket`. Any connected peer can claim any outstanding `getdata` response.

**Root cause 2 — Merkle root check is bypassed for empty `txdata`.** [2](#0-1) 

The guard is `block.compute_merkle_root().is_some() && !block.check_merkle_root()`. For `txdata = []`, `compute_merkle_root()` returns `None`, making the entire condition `false`. The block is serialized and inserted into the cache unconditionally. An attacker can take the real block header (public on the Bitcoin network, containing the correct merkle root for the real transactions) and pair it with `txdata = []`. The block hash matches (derived from the header), and the merkle root check is skipped.

**Root cause 3 — No re-fetch after cache hit.** [3](#0-2) 

`enqueue_new_blocks_to_download` only enqueues a hash when absent from both `getdata_request_info` and the block cache. Once the poisoned block occupies the cache slot, the legitimate block is never requested again.

**Codebase confirmation — empty-txdata blocks are accepted in production code.** [4](#0-3) 

The existing test adds `Block { header: h3, txdata: Vec::new() }` and calls `.unwrap()`, confirming the path succeeds.

**Exploit flow:**
1. Attacker connects as a Bitcoin peer to the adapter (passes `peer_info.contains_key` check).
2. Adapter sends `GetData([Block(hash_B)])` to peer_A (legitimate).
3. Attacker (peer_B) immediately sends `NetworkMessage::Block(Block { header: <real header for hash_B>, txdata: [] })`.
4. `received_block_message`: peer_B passes the peer check; `getdata_request_info.remove(&hash_B)` succeeds; `add_block` is called; `compute_merkle_root()` returns `None` → merkle check skipped; block inserted into cache.
5. peer_A's legitimate response arrives: `getdata_request_info.remove(&hash_B)` returns `None` → `ReceivedBlockMessageError::UnknownBlock`; legitimate block discarded.
6. `block_sync_queue` never re-enqueues `hash_B` (it is in `block_cache`).
7. ckBTC canister receives the block with `txdata=[]` — all deposits in that block are missed permanently.

## Impact Explanation

The ckBTC canister consumes blocks from the adapter's block cache. A block delivered with `txdata=[]` contains no transactions. Every Bitcoin UTXO in the real block is invisible to the canister. Affected deposits are permanently lost from the ckBTC perspective — the canister relies entirely on the adapter's block stream with no independent re-scan mechanism. The attacker can repeat this for every block the adapter requests, effectively halting all ckBTC minting. This constitutes permanent loss of in-scope chain-key/ledger assets (ckBTC), matching the High–Critical impact tier for significant ck-token security impact with concrete user and protocol harm.

## Likelihood Explanation

- **Attacker entry point**: connecting as a Bitcoin peer to the adapter via the standard Bitcoin P2P protocol — no credentials required.
- **Knowledge of target hashes**: Bitcoin block hashes and headers are public; the adapter's sync position is inferrable from accepted headers.
- **Race condition**: the attacker only needs to respond before the legitimate peer; since the attacker controls their own response latency and the adapter processes the first valid response, the window is wide.
- **No retry after cache hit**: the poisoned cache entry is permanent until the adapter is restarted or the block is explicitly pruned via `prune_blocks`.
- **Repeatability**: the attack can be applied to every block the adapter requests, with no per-block cost beyond sending a crafted message.

## Recommendation

1. **Enforce per-peer matching**: in `received_block_message`, after removing the entry from `getdata_request_info`, verify `addr == request.socket`. If not, re-insert the entry and return an error, so the legitimate peer's response can still be accepted.

2. **Reject blocks with empty `txdata` or treat `None` merkle root as invalid**: in `add_block`, change the guard to also reject when `compute_merkle_root()` returns `None` for a block whose header's `merkle_root` is non-zero, or add an explicit check that `txdata` is non-empty before the merkle root guard.

## Proof of Concept

```rust
// Minimal integration test sketch
let peer_a = SocketAddr::from_str("127.0.0.1:8333").unwrap();
let peer_b = SocketAddr::from_str("127.0.0.1:9999").unwrap(); // attacker

// Both peers are added (both pass peer_info check)
blockchain_manager.add_peer(&mut channel, &peer_a);
blockchain_manager.add_peer(&mut channel, &peer_b);

// Adapter sends getdata to peer_a for block_hash
blockchain_manager.block_sync_queue.insert(target_block_hash);
blockchain_manager.sync_blocks(&mut channel);
// getdata_request_info now has: target_block_hash -> { socket: peer_a, ... }

// Attacker (peer_b) responds first with real header but empty txdata
let poisoned_block = Block { header: real_header, txdata: vec![] };
let result = blockchain_manager.received_block_message(&peer_b, &poisoned_block).await;
assert!(result.is_ok()); // succeeds — peer_b passes peer check, merkle check skipped

// Legitimate peer_a response is now rejected
let result = blockchain_manager.received_block_message(&peer_a, &real_block).await;
assert!(matches!(result, Err(ReceivedBlockMessageError::UnknownBlock)));

// Block is in cache with txdata=[] — never re-fetched
assert!(blockchain_manager.blockchain.get_block(&target_block_hash).is_some());
blockchain_manager.enqueue_new_blocks_to_download(vec![real_header.into()]);
assert!(!blockchain_manager.block_sync_queue.contains(&target_block_hash));
```
