Audit Report

## Title
Missing Execution-Time 7-Day Amount Check for `MintSnsTokens` Proposals — (`rs/sns/governance/src/governance.rs`)

## Summary

`perform_transfer_sns_treasury_funds` enforces an execution-time re-tally guard (`transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`) that prevents the 7-day minting/transfer cap from being exceeded when multiple proposals execute in sequence. The `perform_mint_sns_tokens` path receives no `Valuation` and no `proposal_id`, making it structurally impossible to perform the same execution-time check, even though both action types share the same 7-day upper-bound infrastructure. Multiple `MintSnsTokens` proposals can therefore be submitted, each passing the per-proposal submission-time check, and then both execute — cumulatively exceeding the protocol-intended cap with no guard firing.

## Finding Description

In `perform_action` (governance.rs L2203–2212), `TransferSnsTreasuryFunds` retrieves a stored `Valuation` from `action_auxiliary` and forwards it to `perform_transfer_sns_treasury_funds`, which immediately calls `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` (governance.rs L3000–3005). That function re-tallies all executed `TransferSnsTreasuryFunds` proposals from the past 7 days and rejects execution if the running total would exceed the valuation-based cap.

`MintSnsTokens` is dispatched as `self.perform_mint_sns_tokens(mint).await` — no `proposal_id`, no `Valuation`, no `action_auxiliary` lookup (governance.rs L2212). The function signature makes an execution-time re-check structurally impossible. The only guard is the submission-time check inside `validate_and_render_action`, which runs once at proposal creation and cannot account for other proposals that are adopted but not yet executed.

Both action types share the same underlying bound function (`ProposalsAmountTotalUpperBound::in_tokens` in `proposals_amount_total_limit/src/lib.rs` L14–18), confirming the design intent that both should be subject to the same 7-day cap. The asymmetry is a missing implementation, not a design choice.

**Exploit flow:**
1. Proposal A: `MintSnsTokens(amount = 0.6L)` — submission-time check passes (past-7-day total = 0).
2. Proposal B: `MintSnsTokens(amount = 0.6L)` — submission-time check passes (A not yet executed, past-7-day total still = 0).
3. Both proposals are adopted by governance vote.
4. Proposal A executes: total minted = 0.6L. No execution-time check.
5. Proposal B executes: total minted = 1.2L > L. No execution-time check fires.

The `TransferSnsTreasuryFunds` path would have rejected step 5 via the execution-time guard.

## Impact Explanation

This constitutes illegal minting of SNS tokens beyond the protocol-intended 7-day cap, directly diluting all existing SNS token holders. For large SNS treasuries, the cap is 300,000 XDR worth of tokens (lib.rs L41); excess minting can reach up to 2× the cap in the two-proposal scenario and higher with more concurrent proposals. This matches the allowed impact: **"Significant SNS security impact with concrete user or protocol harm"** — specifically unauthorized token inflation undermining the economic safety guarantees the 7-day limit is designed to provide. Severity: **High**.

## Likelihood Explanation

No single malicious actor is required. The scenario arises naturally when multiple neuron holders independently submit `MintSnsTokens` proposals within the same 7-day window, each passing the per-proposal submission check, and the proposals are later executed in sequence. The gap between proposal adoption and execution (hours to days) makes this a realistic race condition in any active SNS. No special privileges beyond normal neuron ownership are required to submit proposals.

## Recommendation

Mirror the `TransferSnsTreasuryFunds` pattern:

1. Add a `MintSnsTokens`-specific `Valuation` variant to `ActionAuxiliary`, stored at proposal-submission time (analogous to `ActionAuxiliary::TransferSnsTreasuryFunds`).
2. In `perform_action`, retrieve that auxiliary valuation via `get_action_auxiliary` and pass it (along with `proposal_id`) to `perform_mint_sns_tokens`.
3. Inside `perform_mint_sns_tokens`, implement and call a `mint_sns_tokens_amount_is_small_enough_at_execution_time_or_err` function that re-tallies all executed `MintSnsTokens` proposals from the past 7 days using `mint_sns_tokens_7_day_total_upper_bound_tokens` and rejects execution if the running total would exceed the cap.

## Proof of Concept

A deterministic integration test or PocketIC test can reproduce this:

1. Initialize an SNS with a treasury large enough that the 7-day cap `L` is well-defined.
2. Submit two `MintSnsTokens` proposals each with `amount = 0.6L`; verify both pass submission-time validation.
3. Adopt both proposals via governance vote.
4. Execute Proposal A; assert total minted = 0.6L.
5. Execute Proposal B; assert it succeeds and total minted = 1.2L — demonstrating the cap is exceeded with no error.
6. Contrast with two analogous `TransferSnsTreasuryFunds` proposals: step 5 would return `PreconditionFailed`. [1](#0-0) [2](#0-1) [3](#0-2)

```rust
    async fn perform_transfer_sns_treasury_funds(
        &mut self,
        proposal_id: u64, // This is just to control concurrency.
        valuation: Result<Valuation, GovernanceError>,
        transfer: &TransferSnsTreasuryFunds,
    ) -> Result<(), GovernanceError> {
        // Only execute one proposal of this type at a time.
        thread_local! {
            static IN_PROGRESS_PROPOSAL_ID: RefCell<Option<u64>> = const { RefCell::new(None) };
        }
        let release_on_drop = acquire(&IN_PROGRESS_PROPOSAL_ID, proposal_id);
        if let Err(already_in_progress_proposal_id) = release_on_drop {
            return Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!(
                    "Another TransferSnsTreasuryFunds proposal (ID = {already_in_progress_proposal_id}) is already in progress.",
                ),
            ));
        }

        transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err(
            transfer,
            valuation?,
            self.proto.proposals.values(),
            self.env.now(),
        )?;
```
