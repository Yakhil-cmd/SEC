### Title
SNS Governance Reward Settlement Uses `now` Instead of Reward Round End Timestamp, Causing Proposals to Be Settled in the Wrong Reward Event - (`File: rs/sns/governance/src/governance.rs`)

---

### Summary

The SNS governance `distribute_rewards` function settles proposals using `self.env.now()` as the cutoff for `ReadyToSettle` status, while the reward event's purse only covers complete reward rounds ending at `reward_event_end_timestamp_seconds`. Proposals whose voting deadlines fall in the gap between `reward_event_end_timestamp_seconds` and `now` are incorrectly settled in the current reward event instead of the next one. The NNS governance explicitly avoids this by using `most_recent_fully_elapsed_reward_round_end_timestamp_seconds()` as the cutoff.

---

### Finding Description

In `rs/sns/governance/src/governance.rs`, `distribute_rewards` calls `ready_to_be_settled_proposal_ids()` at line 5822:

```rust
let considered_proposals: Vec<ProposalId> =
    self.ready_to_be_settled_proposal_ids().collect();
``` [1](#0-0) 

That helper function, defined at line 1927, uses `self.env.now()` as the timestamp:

```rust
fn ready_to_be_settled_proposal_ids(&self) -> impl Iterator<Item = ProposalId> + '_ {
    let now = self.env.now();
    self.proto
        .proposals
        .iter()
        .filter(move |(_, data)| data.reward_status(now) == ProposalRewardStatus::ReadyToSettle)
        .map(|(k, _)| ProposalId { id: *k })
}
``` [2](#0-1) 

However, the reward event's purse covers only **complete** reward rounds. The actual end of the covered period is computed **after** the proposal list is collected:

```rust
let reward_event_end_timestamp_seconds = new_rounds_count
    .saturating_mul(round_duration_seconds)
    .saturating_add(reward_start_timestamp_seconds);
``` [3](#0-2) 

Because `reward_event_end_timestamp_seconds ≤ now` (the gap is `(now - reward_start) mod round_duration`, up to nearly one full round duration), any proposal whose voting deadline falls in that gap is included in the current event even though its voting period extended past the end of the covered reward round.

The NNS governance explicitly avoids this by computing the correct cutoff first and passing it to `ready_to_be_settled_proposal_ids`:

```rust
let as_of_timestamp_seconds =
    self.most_recent_fully_elapsed_reward_round_end_timestamp_seconds();
let considered_proposals: Vec<ProposalId> = self
    .ready_to_be_settled_proposal_ids(as_of_timestamp_seconds)
    .collect();
``` [4](#0-3) 

The NNS `most_recent_fully_elapsed_reward_round_end_timestamp_seconds` truncates `now` down to the nearest complete reward-round boundary: [5](#0-4) 

The NNS `ready_to_be_settled_proposal_ids` accepts an explicit `as_of_timestamp_seconds` parameter and uses it as the time reference: [6](#0-5) 

The SNS `ProposalData::reward_status` uses `now_seconds` to decide `AcceptVotes` vs `ReadyToSettle` via `accepts_vote`: [7](#0-6) 

---

### Impact Explanation

Proposals whose voting deadlines fall between `reward_event_end_timestamp_seconds` and `now` are settled in the current reward event. Their voters receive maturity from a reward purse that was calculated only over complete rounds ending at `reward_event_end_timestamp_seconds`, even though those proposals' voting periods extended past that boundary. The next reward event will have fewer proposals to settle, potentially causing unnecessary rollover of the reward purse. While total rewards distributed over time remain the same, the per-event accounting is incorrect: voters on "gap" proposals receive rewards from the wrong period, and the reward purse for the next event is not shared with those proposals as intended.

---

### Likelihood Explanation

This occurs in every SNS deployment on every call to `distribute_rewards` whenever any proposal's deadline falls in the window `(reward_event_end_timestamp_seconds, now)`. Since `distribute_rewards` is triggered by `run_periodic_tasks` and the gap can be up to nearly one full `round_duration_seconds` wide, this is a routine occurrence in normal SNS operation, not an edge case. Any SNS token holder can submit a proposal and influence when its deadline falls relative to reward round boundaries.

---

### Recommendation

Replace the parameterless `ready_to_be_settled_proposal_ids()` call in `distribute_rewards` with a version that accepts an explicit cutoff timestamp, mirroring the NNS pattern. Compute `reward_event_end_timestamp_seconds` first, then pass it as the cutoff:

```rust
let reward_event_end_timestamp_seconds = new_rounds_count
    .saturating_mul(round_duration_seconds)
    .saturating_add(reward_start_timestamp_seconds);

let considered_proposals: Vec<ProposalId> =
    self.ready_to_be_settled_proposal_ids(reward_event_end_timestamp_seconds).collect();
```

Update `ready_to_be_settled_proposal_ids` to accept an `as_of_timestamp_seconds: u64` parameter and pass it to `reward_status`, exactly as the NNS governance does.

---

### Proof of Concept

1. An SNS is configured with `round_duration_seconds = 604800` (1 week).
2. The last reward event ended at `T = reward_start + 604800`.
3. `distribute_rewards` is called at `now = T + 300000` (≈3.5 days into the next round).
4. A proposal was created with a deadline of `T + 100` (100 seconds after the round ended).
5. `ready_to_be_settled_proposal_ids()` uses `now = T + 300000`, so `accepts_vote(now)` returns `false` for this proposal (deadline is in the past), and it is classified `ReadyToSettle`.
6. The proposal is included in `considered_proposals` and settled with `reward_event_end_timestamp_seconds = T`.
7. Its voters receive maturity from the current event's purse, even though the proposal's voting period extended 100 seconds past the reward round boundary.
8. The next reward event (covering `T` to `T + 604800`) will not include this proposal, even though it was active during that period.

The NNS governance would have excluded this proposal from the current event by using `as_of_timestamp_seconds = T`, deferring it to the next event. [8](#0-7) [3](#0-2)
