Audit Report

## Title
Missing Caller Ownership Check in `refresh_buyer_tokens` Allows Unauthorized Swap Participation Registration — (File: `rs/sns/swap/canister/canister.rs`)

## Summary
The SNS swap canister's `refresh_buyer_tokens` endpoint resolves the buyer principal from the caller-supplied `arg.buyer` field without verifying that the caller is that buyer. Any unprivileged ingress sender can register another principal's ICP participation in an SNS swap, including supplying the mandatory `confirmation_text` (terms-of-service consent gate) on the victim's behalf, as long as the victim has already transferred ICP to the swap canister's subaccount.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 130–134, when `arg.buyer` is non-empty, the buyer principal is parsed directly from the caller-supplied string with no check that `caller == buyer`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no ownership check
};
```

This `p` is then passed to `refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs`, lines 1134–1163), which:
1. Calls `validate_confirmation_text` (lines 363–384) — this only checks that the supplied text matches the SNS-configured text, not that the entity supplying it is the buyer. The `confirmation_text` is publicly readable via `get_init`.
2. Reads the ICP balance of `buyer`'s subaccount on the swap canister (lines 1153–1163).
3. Writes the buyer's participation amount into `self.buyers` (lines 1285–1291).

There is no check anywhere in this path that `caller == buyer`. The proto comment at `rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto` lines 843–851 confirms the `buyer` field is optional and defaults to the caller only when omitted, and that `confirmation_text` is a consent gate that "a participant should confirm before they may participate." The validation does not enforce that the confirming entity is the buyer.

The existing test at `rs/tests/nns/sns/lib/src/sns_deployment.rs` lines 919–926 explicitly demonstrates the attack pattern — a `default_sns_agent` calling `refresh_buyer_tokens` with `wealthy_user_identity.principal_id` — confirming this path is reachable and exercised.

## Impact Explanation
This is a **High** severity finding. The `confirmation_text` mechanism exists precisely to record explicit user consent to swap terms before committing their funds. An attacker can bypass this consent gate entirely by supplying the publicly known confirmation text on behalf of any victim who has transferred ICP to the swap subaccount. The victim's ICP is then irrevocably committed to the swap (until it closes), and if the swap commits, the victim's ICP is swept to the SNS treasury and they receive SNS tokens — a financial outcome they never explicitly consented to. This constitutes a significant SNS security impact with concrete user harm: unauthorized commitment of user funds and bypass of the protocol's explicit consent mechanism.

## Likelihood Explanation
- No privileged access is required; any unprivileged ingress identity can trigger this.
- The victim's principal is public on-chain information.
- The victim's ICP subaccount balance on the swap canister is queryable via ledger queries.
- The `confirmation_text` is set at SNS initialization and is publicly readable from the swap canister's `get_init` query endpoint.
- The window of vulnerability (victim has transferred ICP but not yet called `refresh_buyer_tokens`) is realistic: multi-step UI flows, accidental transfers, or deliberate delays pending review of terms are common.
- The attack is repeatable across any SNS swap with a `confirmation_text` requirement.

## Recommendation
Add a caller ownership check in `refresh_buyer_tokens` (`rs/sns/swap/canister/canister.rs`) when a non-empty `buyer` is specified:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let specified = PrincipalId::from_str(&arg.buyer).unwrap();
    if specified != caller_principal_id() {
        ic_cdk::trap("caller must match the specified buyer");
    }
    specified
};
```

Alternatively, enforce the check inside `refresh_buyer_token_e8s` by passing the caller separately and asserting `caller == buyer` when `confirmation_text` validation is required.

## Proof of Concept
1. Victim transfers 100 ICP to `swap_canister_subaccount(victim_principal)` on the ICP ledger.
2. Victim has not yet called `refresh_buyer_tokens` (e.g., reviewing SNS terms).
3. Attacker reads the SNS `confirmation_text` via `get_init` (public query).
4. Attacker (any identity) calls:
   ```
   refresh_buyer_tokens({
     buyer: "<victim_principal>",
     confirmation_text: opt "<correct_text>"
   })
   ```
5. Swap canister reads victim's 100 ICP balance, validates the attacker-supplied consent text, and writes victim's participation into `self.buyers`.
6. If the swap commits, victim's ICP is swept to the SNS treasury; victim receives SNS tokens without having agreed to the terms themselves.

A deterministic integration test can be written using `PocketIC` or `StateMachine`: create a swap with a `confirmation_text`, transfer ICP from `user_A` to the swap subaccount, then call `refresh_buyer_tokens` from `user_B` with `buyer = user_A` and the correct `confirmation_text`, and assert that `user_A`'s buyer state is populated — confirming the unauthorized registration.
