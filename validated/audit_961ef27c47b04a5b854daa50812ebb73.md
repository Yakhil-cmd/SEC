### Title
EIP-7702 Authority Addresses Bypass Silo Address Whitelist Check — (`File: engine/src/engine.rs`)

### Summary

Aurora Engine's Silo mode enforces an address whitelist (`WhitelistKind::Address`) that restricts which EVM addresses may submit transactions. The `assert_access` function checks only the **transaction sender's** address against this whitelist. However, EIP-7702 transactions carry an `authorization_list` whose authority addresses receive code delegation and have their nonces incremented — effectively acting as participants in the transaction — without any whitelist check on those authority addresses. An unprivileged attacker who controls a whitelisted sender address can use an EIP-7702 transaction to install delegation code on any non-whitelisted authority address, causing that address to gain EVM code and have its nonce mutated, bypassing the Silo address whitelist entirely.

### Finding Description

**Root cause — `assert_access` only checks the transaction sender, not EIP-7702 authority addresses:**

In `engine/src/engine.rs`, `assert_access` is called before EVM execution:

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };
    ...
}
```

`transaction.address` is the **sender** of the EIP-7702 transaction. The `authorization_list` field — which contains one or more authority addresses that will have their code overwritten and nonces incremented — is never checked against the `WhitelistKind::Address` whitelist.

**Code path:**

1. `submit_with_alt_modexp` in `engine/src/engine.rs` calls `assert_access` at line 1052, checking only `transaction.address` (the sender).
2. The `authorization_list` is parsed in `engine-transactions/src/eip_7702.rs` via `SignedTransaction7702::authorization_list()`, which validates chain ID, parity, `s`-value, and ecrecovers the authority address — but performs no whitelist check.
3. The validated `authorization_list` is passed through `NormalizedEthTransaction` (line 156 of `engine-transactions/src/lib.rs`) and forwarded to `engine.call(...)` at line 1124 of `engine/src/engine.rs`.
4. The EVM executor processes each `Authorization` entry, incrementing the authority's nonce and writing delegation code (`EF0100 || target_address`) to the authority's account — all without any Silo whitelist gate.

**Analogy to the KintoWallet bug:** Just as KintoWallet only checked `owner[0]`'s KYC while `owner[1]` and `owner[2]` could be non-KYC'd accounts that still controlled the wallet, Aurora's Silo whitelist only checks the EIP-7702 transaction sender while the authority addresses in the `authorization_list` — which receive code and nonce mutations — are never checked.

### Impact Explanation

In Silo mode with the `WhitelistKind::Address` whitelist enabled, the operator's intent is that only explicitly approved EVM addresses may interact with the EVM. An attacker who controls a single whitelisted address can:

1. Craft an EIP-7702 transaction with an `authorization_list` containing any number of non-whitelisted authority addresses.
2. Submit it through the whitelisted sender.
3. Each authority address receives `EF0100 || target_contract` delegation code and has its nonce incremented — bypassing the whitelist entirely.
4. The authority address now has EVM code installed and can subsequently send transactions as a delegated account (EIP-3607 is bypassed for delegated addresses per line 1097 of `engine/src/engine.rs`).

This constitutes a **whitelist bypass** allowing non-whitelisted addresses to gain EVM state mutations and subsequently transact. In Silo deployments where the whitelist is the primary access control boundary (e.g., permissioned chains), this undermines the entire access control model. The impact is **temporary freezing** (if the authority address is a legitimate account whose state is corrupted) or **unauthorized state changes** enabling further exploitation.

### Likelihood Explanation

- Requires only a single whitelisted EVM address — a realistic attacker precondition in any Silo deployment where the attacker has legitimately obtained whitelist access.
- EIP-7702 is a standard transaction type already supported and tested in this codebase.
- No privileged keys, admin access, or social engineering required.
- The attack is deterministic and reproducible with a single crafted transaction.

### Recommendation

In `assert_access` (or immediately after `authorization_list()` is resolved in `submit_with_alt_modexp`), iterate over all authority addresses in `transaction.authorization_list` and verify each against the `WhitelistKind::Address` whitelist before proceeding with EVM execution:

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError { kind: EngineErrorKind::NotAllowed, gas_used: 0 });
    }

    // NEW: check each EIP-7702 authority address against the address whitelist
    for auth in &transaction.authorization_list {
        let authority = Address::new(auth.authority);
        if !silo::is_address_allowed(io, &authority) {
            return Err(EngineError { kind: EngineErrorKind::NotAllowed, gas_used: 0 });
        }
    }

    Ok(())
}
```

### Proof of Concept

**Setup:** Silo mode enabled, `WhitelistKind::Address` whitelist active. Attacker controls whitelisted address `A`. Target non-whitelisted address `B` (any EOA).

**Steps:**

1. Attacker signs an `AuthorizationTuple` from address `B`'s key: `sign_eip7702_authorization(chain_id, target_contract, nonce_of_B, B_secret_key)`.
2. Attacker constructs a `Transaction7702` with `authorization_list: [auth_for_B]`, signed by whitelisted address `A`.
3. Attacker submits via `submit` NEAR method with `predecessor_account_id` in the Account whitelist.
4. `assert_access` passes because `transaction.address == A` (whitelisted).
5. EVM executor processes the authorization: `B`'s nonce is incremented, `B`'s code is set to `EF0100 || target_contract`.
6. `B` now has delegation code installed and can send transactions as a delegated account (bypassing EIP-3607 per line 1097 of `engine/src/engine.rs`), despite never being in the address whitelist.

**Expected (correct) behavior:** The engine should reject the transaction with `NotAllowed` because `B` is not in the address whitelist.

**Observed behavior:** The transaction succeeds; `B` receives delegation code and a nonce increment, bypassing the Silo address whitelist. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)

    pub fn authorization_list(&self) -> Result<Vec<Authorization>, Error> {
        if self.transaction.authorization_list.is_empty() {
            return Err(Error::EmptyAuthorizationList);
        }
        let current_tx_chain_id = U256::from(self.transaction.chain_id);
        let mut authorization_list = Vec::with_capacity(self.transaction.authorization_list.len());
        let mut rlp_stream = RlpStream::new();
        let mut message_bytes = vec![MAGIC; 1];
        // According to EIP-7702, we should validate each authorization. We shouldn't skip any of them.
        // And just put the `is_valid` flag to `false` if any of them is invalid. It's related to
        // gas calculation, as each `authorization_list` must be charged, even if it's invalid.
        for auth in &self.transaction.authorization_list {
            // According to EIP-7702 step 1. validation, we should verify it as
            // `chain_id = 0 || current_chain_id`.
            // AS `current_chain_id` we used `transaction.chain_id` as we will validate `chain_id` in
            // Engine `submit_transaction` method.

            // Step 2 - validation logic inside EVM itself.
            // Step 3. Checking: authority = ecrecover(keccak(MAGIC || rlp([chain_id, address, nonce])), y_parity, r, s])
            // Validate the signature, as in tests it is possible to have invalid signature values.
            // Value `v` shouldn't be greater than 1
            let mut is_valid = if auth.s > SECP256K1N_HALF {
                false
            } else {
                (auth.chain_id.is_zero() || auth.chain_id == current_tx_chain_id)
                    && auth.parity <= U256::one()
            };

            let auth_address = if is_valid {
                rlp_stream.begin_list(3);
                rlp_stream.append(&auth.chain_id);
                rlp_stream.append(&auth.address);
                rlp_stream.append(&auth.nonce);

                message_bytes.extend_from_slice(rlp_stream.as_raw());

                let signature_hash = aurora_engine_sdk::keccak(&message_bytes);
                // U256::as_u32() is safe because here we're sure that the parity <= 1.
                let v = u8::try_from(auth.parity.as_u32()).unwrap_or(u8::MAX);
                let auth_address = ecrecover(signature_hash, &super::vrs_to_arr(v, auth.r, auth.s))
                    .unwrap_or_else(|_| {
                        is_valid = false;
                        Address::default()
                    });

                message_bytes.truncate(1);
                rlp_stream.clear();

                auth_address
            } else {
                Address::default()
            };

            // Validations steps 2,4-9 0f EIP-7702 provided by EVM itself.
            authorization_list.push(Authorization {
                authority: auth_address.raw(),
                address: auth.address,
                nonce: auth.nonce,
                is_valid,
            });
        }
        Ok(authorization_list)
    }
