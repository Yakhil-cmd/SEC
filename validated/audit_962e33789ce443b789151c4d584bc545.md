### Title
`set_owner` Accepts Any Syntactically Valid NEAR Account Without Existence Validation, Enabling Permanent Loss of Engine Admin Control - (File: engine/src/contract_methods/admin.rs)

### Summary
`set_owner` in `admin.rs` writes the caller-supplied `new_owner` directly into `state.owner_id` without verifying that the account actually exists on NEAR. If the current owner mistakenly supplies a syntactically valid but non-existent NEAR account ID, the engine permanently loses its owner with no on-chain recovery path.

### Finding Description
The `set_owner` function at lines 104–121 of `engine/src/contract_methods/admin.rs` performs exactly three checks before committing the new owner to state:

1. `require_running` – engine must not be paused
2. `require_owner_only` – caller must be the current `state.owner_id`
3. `ERR_SAME_OWNER` – `args.new_owner` must differ from the current owner

```rust
// engine/src/contract_methods/admin.rs  lines 111-117
let args: SetOwnerArgs = io.read_input_borsh()?;
if state.owner_id == args.new_owner {
    return Err(errors::ERR_SAME_OWNER.into());
}
state.owner_id = args.new_owner;   // ← written unconditionally
state::set_state(&mut io, &state)?;
```

NEAR's `AccountId` type enforces only *syntactic* validity (non-empty string, valid character set, length bounds). It does **not** verify on-chain existence. A string such as `"typo-account.near"` passes Borsh deserialization and is written directly to `EngineState::owner_id` in storage. There is no two-step transfer, no confirmation, and no cross-contract existence check.

The `EngineState` struct stores `owner_id: AccountId` as a required (non-`Option`) field, so it cannot be cleared; once set to a non-existent account it is permanently locked.

### Impact Explanation
If `owner_id` is set to a non-existent NEAR account, every owner-gated function becomes permanently inaccessible:

| Function | Consequence |
|---|---|
| `set_owner` | Cannot correct the mistake |
| `stage_upgrade` / `upgrade` | No contract upgrades ever again |
| `pause_contract` | No emergency pause |
| `set_eth_connector_contract_account` | Connector address permanently frozen |
| `set_key_manager` / `attach_full_access_key` | Relayer key management lost |
| `resume_precompiles` | Paused precompiles cannot be re-enabled |

The engine continues executing EVM transactions, but is permanently unmanageable. If a critical bug is subsequently discovered in the bridge connector or EVM execution path, the contract cannot be paused or upgraded. Bridged ETH and ERC-20 tokens held in the connector become permanently frozen. This satisfies the **Critical** impact criterion: permanent freezing of funds.

### Likelihood Explanation
NEAR account IDs are human-readable strings (e.g., `aurora.near`, `relay.aurora.near`). A single character typo produces a syntactically valid but non-existent account. The existing `ERR_SAME_OWNER` guard demonstrates the developers considered input validation at this call site but did not add an existence check. There is no staging period, no confirmation step, and no off-chain tooling enforced by the contract. The mistake requires only one erroneous call by the owner.

### Recommendation
Implement a two-step ownership transfer pattern:
1. The current owner calls a `propose_owner(new_owner)` function that stores the candidate in a separate storage slot.
2. The proposed new owner calls `accept_owner()` to finalize the transfer.

This guarantees the new owner account exists and is controlled by a party who can sign NEAR transactions, eliminating the permanent-lockout risk.

### Proof of Concept
1. Current owner calls `set_owner` with `new_owner = "typo-account.near"` (syntactically valid, non-existent on NEAR).
2. `io.read_input_borsh::<SetOwnerArgs>()` succeeds — `AccountId` format validation passes.
3. `state.owner_id == args.new_owner` is `false` — `ERR_SAME_OWNER` is not triggered.
4. `state.owner_id` is written as `"typo-account.near"` and persisted via `state::set_state`.
5. All subsequent calls to any `require_owner_only`-gated function fail permanently.
6. `set_eth_connector_contract_account` can no longer be called; all bridge withdrawal and transfer operations that route through `get_connector_account_id` continue to use whatever connector was last set, and if that connector is also misconfigured, bridged funds are permanently frozen with no recovery path.
