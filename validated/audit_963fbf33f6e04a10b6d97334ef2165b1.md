### Title
Owner Can Change ETH Connector Account After User Commits Bridge Funds, Disrupting In-Flight Deposits - (File: engine/src/contract_methods/connector.rs)

### Summary

The `ft_on_transfer` function determines whether incoming tokens are ETH (base tokens) or ERC-20 tokens by comparing the NEAR `predecessor_account_id` against the connector account ID read live from storage at callback execution time. The owner can call `set_eth_connector_contract_account` at any point — including between the user's `ft_transfer_call` receipt and the `ft_on_transfer` callback receipt — to change that stored value. This allows the owner to cause in-flight ETH bridge deposits to be misclassified and fail, temporarily freezing the user's committed funds.

### Finding Description

When a user bridges ETH to Aurora, they call `ft_transfer_call` on the ETH connector contract. The connector then calls `ft_on_transfer` on the Aurora engine in a subsequent NEAR receipt. Inside `ft_on_transfer`, the engine decides how to credit the tokens:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(...)
};
``` [1](#0-0) 

The `get_connector_account_id` call reads the connector account from Aurora's storage at the moment `ft_on_transfer` executes, not at the moment the user committed their funds: [2](#0-1) 

The owner can overwrite this stored value at any time via `set_eth_connector_contract_account`, which requires only `require_owner_only` and `require_running`: [3](#0-2) 

This is exposed as a public NEAR entry point: [4](#0-3) 

Because `ft_transfer_call` (receipt N) and `ft_on_transfer` (receipt N+1) are separate NEAR receipts, the owner can insert a `set_eth_connector_contract_account` call between them. When `ft_on_transfer` then executes, `predecessor_account_id` (the old connector) no longer matches `get_connector_account_id` (the new value). The check fails, the tokens are routed through `receive_erc20_tokens` with the old connector as the NEP-141 account, no ERC-20 mapping exists for it, the call returns an error, and the full amount is refunded back to the connector contract — not to the user. The user's ETH is temporarily frozen until they retry.

The owner can also perform the analog of the gold-card "selective pick": by monitoring the mempool for specific users' deposits and timing the connector swap to coincide only with those receipts, the owner can selectively block individual deposits while leaving others unaffected.

### Impact Explanation

**High — Temporary freezing of funds.** A user who calls `ft_transfer_call` on the ETH connector has committed their ETH. If the owner changes the connector account ID before `ft_on_transfer` executes, the deposit fails and the ETH is returned to the connector contract rather than credited to the user on Aurora. The user cannot complete the bridge operation until the owner restores the correct connector account. The owner can repeat this selectively against targeted users.

### Likelihood Explanation

**Low-to-Medium.** The attack requires the owner to act between two NEAR receipts in the same block. While NEAR receipt ordering within a block is deterministic and observable, inserting a transaction between two receipts of the same cross-contract call requires the owner to submit their transaction in the same block with a receipt ordering that lands between the two. This is operationally non-trivial but not impossible, especially for a sophisticated or colluding block producer. The owner can also simply change the connector account preemptively to affect all subsequent deposits without needing precise timing.

### Recommendation

- **Snapshot the connector account ID at commitment time**: Record the connector account ID in the `ft_transfer_call` arguments or in a per-deposit storage slot so `ft_on_transfer` uses the value that was current when the user committed.
- **Explicit trust model**: Document that the connector account is owner-controlled and can affect in-flight deposits; require a time-locked multisig (analogous to the client's own mitigation in the referenced report) before `set_eth_connector_contract_account` takes effect.
- **Cancel-on-change**: Emit an event or revert pending deposits when the connector account changes, so users can observe and react.

### Proof of Concept

1. User calls `ft_transfer_call` on the ETH connector contract, locking their ETH. This schedules `ft_on_transfer` on Aurora as receipt N+1.
2. Owner calls `set_eth_connector_contract_account` with a different account ID. This executes as receipt N+0.5 (inserted between N and N+1 in the same block).
3. `ft_on_transfer` executes. `predecessor_account_id` is the old connector; `get_connector_account_id(&io)?` returns the new account. The check at line 81 fails.
4. `receive_erc20_tokens` is called with the old connector as the NEP-141 account. No ERC-20 mapping exists; the function returns an error.
5. `ft_on_transfer` returns the full refund amount; the NEP-141 contract refunds the tokens to the connector, not to the user.
6. The user's ETH is temporarily frozen. The owner can restore the original connector account and repeat the attack selectively against any future deposit. [5](#0-4) [3](#0-2) [6](#0-5)

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}

fn get_connector_account_id<I: IO>(io: &I) -> Result<AccountId, ContractError> {
    io.read_storage(&construct_contract_key(
        EthConnectorStorageId::EthConnectorAccount,
    ))
    .ok_or(errors::ERR_CONNECTOR_STORAGE_KEY_NOT_FOUND)
    .and_then(|x| {
        x.to_value()
            .map_err(|_| errors::ERR_BORSH_DESERIALIZE.as_bytes())
    })
    .map_err(Into::into)
}
