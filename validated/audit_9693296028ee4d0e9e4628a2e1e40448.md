Audit Report

## Title
Unbounded Neuron Iteration in SNS `compute_ballots_for_new_proposal` Can Exhaust Instruction Limit, Permanently Blocking Proposal Submission - (File: rs/sns/governance/src/governance.rs)

## Summary

`make_proposal` in the SNS Governance canister unconditionally calls `compute_ballots_for_new_proposal`, which performs a synchronous, unbounded iteration over every neuron in `self.proto.neurons` with no instruction-limit guard. If the neuron population grows large enough, every subsequent call to `make_proposal` will trap with `CanisterInstructionLimitExceeded`, permanently preventing new governance proposals from being submitted to that SNS.

## Finding Description

`make_proposal` calls `compute_ballots_for_new_proposal` unconditionally before inserting the proposal: [1](#0-0) 

`compute_ballots_for_new_proposal` iterates over the entire neuron map in a plain `for` loop: [2](#0-1) 

Per iteration, the function calls `v.dissolve_delay_seconds(now_seconds)`, then `v.voting_power(...)` (multi-factor arithmetic), and performs a `BTreeMap::insert`. There is no call to `ic_cdk::api::instruction_counter()`, no `over_soft_message_limit()` check, and no `noop_self_call_if_over_instructions` escape hatch anywhere in this code path — confirmed by a full-text search of `rs/sns/governance/src/governance.rs` returning zero matches for all three patterns.

By contrast, the NNS Governance canister defines explicit hard and soft instruction limits: [3](#0-2) 

And uses `noop_self_call_if_over_instructions` to yield before the limit is hit: [4](#0-3) 

The NNS comment explicitly states this design "leaves room for 750 thousand neurons with complex following," acknowledging that unbounded iteration is a real instruction-limit risk. SNS has no equivalent protection. SNS canisters run on application subnets where the update-call instruction limit is 5×10⁹; the per-neuron work in `compute_ballots_for_new_proposal` will exhaust this budget at a neuron count well within the documented `MAX_NUMBER_OF_NEURONS` ceiling.

## Impact Explanation

Once the neuron count crosses the per-message instruction threshold, every call to `make_proposal` traps. No new governance proposals can be submitted to the SNS — including any remediation proposal that would itself require `make_proposal` to succeed. This is a complete, self-reinforcing denial-of-service on SNS governance proposal submission, matching the allowed High impact: **Application/platform-level DoS with concrete user and protocol harm** (SNS governance permanently frozen for new actions).

## Likelihood Explanation

Any SNS with organic growth in its neuron population approaches the threshold over time. An adversary holding SNS tokens can accelerate this by repeatedly splitting neurons or staking small amounts to create new neurons. The `make_proposal` entry point requires only a neuron with `SubmitProposal` permission and sufficient stake to cover `reject_cost_e8s` — a low bar for any legitimate participant. No privileged access, no governance majority, and no external oracle is required. The attack is repeatable and irreversible without a canister upgrade.

## Recommendation

1. **Adopt the NNS pattern**: replace the live neuron iteration with a pre-computed, periodically refreshed voting-power snapshot updated in a timer job, analogous to `compute_voting_power_snapshot_for_standard_proposal` in the NNS neuron store.
2. **Add an instruction-limit guard**: if live iteration must be kept, add an `ic_cdk::api::instruction_counter()` check inside the loop and return a graceful error (not a trap) when the soft limit is approached.
3. **Enforce a tighter neuron cap**: lower `MAX_NUMBER_OF_NEURONS` to a value provably safe given the per-neuron instruction cost, and add an integration test that verifies `make_proposal` succeeds at the cap.

## Proof of Concept

1. Deploy an SNS with default `NervousSystemParameters`.
2. Create N neurons (by staking and claiming) where N × (per-neuron instruction cost for `dissolve_delay_seconds` + `voting_power` + `BTreeMap::insert`) > 5×10⁹.
3. Call `manage_neuron` → `MakeProposal` from any neuron with `SubmitProposal` permission.
4. Observe the call trap with `CanisterInstructionLimitExceeded`.
5. Confirm no subsequent `make_proposal` call succeeds, because the neuron map size is unchanged.

A deterministic integration test using PocketIC can reproduce this by creating neurons in a loop until `make_proposal` traps, verifying the threshold is reachable within `MAX_NUMBER_OF_NEURONS`.
