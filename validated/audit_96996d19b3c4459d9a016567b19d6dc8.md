Audit Report

## Title
Fee-on-Transfer ERC20 Tokens Cause ckERC20 Undercollateralization via Inflated Deposit Event Amount — (`rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`, `rs/ethereum/cketh/minter/ERC20DepositHelper.sol`)

## Summary

Both ckERC20 deposit helper contracts emit a deposit event carrying the caller-supplied `amount` parameter rather than the actual tokens received by the minter address. The IC ckETH minter canister scrapes these events and mints exactly `event.value()` ckERC20 tokens to the user. If the underlying ERC20 token charges a fee on transfer, the minter receives fewer tokens than the event reports but mints the full reported amount, permanently undercollateralizing the ckERC20 token.

## Finding Description

In `CkDeposit.depositErc20` (`DepositHelperWithSubaccount.sol`), the contract calls `safeTransferFrom` with the caller-supplied `amount` and then emits `ReceivedEthOrErc20` with that same `amount` — not the balance delta actually received by `minterAddress`: [1](#0-0) 

In `CkErc20Deposit.deposit` (`ERC20DepositHelper.sol`), the same pattern applies — `safeTransferFrom` is called and the event is emitted with the original `amount`: [2](#0-1) 

The IC minter canister's `ReceivedErc20LogParser` and `ReceivedEthOrErc20LogParser` in `parser.rs` parse the `value` field directly from the event log data bytes with no balance-delta verification: [3](#0-2) [4](#0-3) 

The `mint()` function in `deposit.rs` then calls `icrc1_transfer` with `amount: event.value()` — unconditionally trusting the event-reported value: [5](#0-4) 

There is no balance-delta check anywhere in the minting pipeline. For a fee-on-transfer ERC20, the minter receives `amount - fee` but mints `amount` ckERC20 tokens, creating a deficit that compounds with every deposit.

## Impact Explanation

Every deposit of a fee-on-transfer ERC20 token mints more ckERC20 than the minter holds in ERC20 backing. The ckERC20 token supply grows faster than the minter's ERC20 reserve. When users attempt to withdraw ckERC20 back to ERC20, the minter will eventually be unable to fulfill withdrawals — a direct, permanent loss of funds for ckERC20 holders. This constitutes illegal minting and protocol insolvency for the affected ckERC20 token, matching the High impact class: "Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."

## Likelihood Explanation

The minter enforces a whitelist of supported ERC20 tokens, and currently supported tokens (e.g., USDC) do not charge transfer fees. However, two realistic paths exist: (1) USDT, which has an upgradeable contract with latent fee-on-transfer logic already present in its code, is a prominent candidate for future addition; (2) an already-supported upgradeable token (such as USDC) could enable fees via a contract upgrade by its issuer, triggering the vulnerability without any IC governance action. Either path requires no privileged IC access — any unprivileged depositor would immediately exploit the discrepancy.

## Recommendation

Replace the emitted `amount` with the actual balance delta received by the minter address. In both `depositErc20` and `deposit`, record the minter's ERC20 balance before and after the `safeTransferFrom` call and emit the difference:

```solidity
uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 received = erc20Token.balanceOf(minterAddress) - balanceBefore;
emit ReceivedEthOrErc20(erc20Address, msg.sender, received, principal, subaccount);
```

This ensures the IC minter only mints ckERC20 tokens equal to the ERC20 tokens actually received, regardless of any fee-on-transfer behavior.

## Proof of Concept

1. A fee-on-transfer ERC20 token (e.g., USDT after enabling its fee, or a newly whitelisted token) is present in the minter's supported token list.
2. An unprivileged user calls `depositErc20(tokenAddress, 1000, principal, subaccount)` on the `CkDeposit` helper contract (`DepositHelperWithSubaccount.sol` line 511).
3. The token's `transferFrom` deducts a 1% fee: the minter receives 990 tokens, but `ReceivedEthOrErc20` records `amount = 1000` (line 525–531).
4. The IC minter's `ReceivedEthOrErc20LogParser::parse_log` reads `value = 1000` from the event data (`parser.rs` lines 149–160).
5. `mint()` calls `icrc1_transfer` with `amount: event.value()` = 1000 (`deposit.rs` lines 73–81), minting 1000 ckERC20 to the user.
6. The minter's ERC20 reserve is 990, but 1000 ckERC20 are in circulation — a 10-token deficit per deposit that widens with every subsequent deposit until the minter cannot honor withdrawals.
