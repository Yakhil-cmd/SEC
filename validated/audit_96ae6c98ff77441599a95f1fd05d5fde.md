### Title
Tendermint WebSocket RPC Has No Message Size Limit, Enabling Memory-Exhaustion DoS — (`sei-tendermint/rpc/jsonrpc/server/ws_handler.go`)

---

### Summary

The Sei chain's Tendermint RPC WebSocket handler initializes `readLimit` to `0` (the Go zero value), which gorilla/websocket interprets as **no limit**. An unauthenticated attacker can connect to the public Tendermint RPC WebSocket endpoint and send arbitrarily large messages, exhausting node memory and crashing the RPC server. This is a direct analog of the reported ZetaChain vulnerability — and is actually worse, since ZetaChain at least had a 32 MB cap.

---

### Finding Description

In `sei-tendermint/rpc/jsonrpc/server/ws_handler.go`, the `wsConnection` struct holds a `readLimit int64` field: [1](#0-0) 

Its zero value is `0`. In `newWSConnection`, this value is applied directly to the underlying gorilla/websocket connection: [2](#0-1) 

Per the gorilla/websocket documentation, `SetReadLimit(0)` means **no limit** — the library will buffer the entire incoming message into memory regardless of size. The `ReadLimit` option function exists to override this: [3](#0-2) 

However, no call site passes this option when constructing the `WebsocketManager`. The `wsConnOptions` slice is populated only with whatever the caller provides, and no default read limit is enforced: [4](#0-3) 

There is also no rate-limiting mechanism anywhere in the `evmrpc/` WebSocket path:



**Contrast with the EVM WebSocket server**, which correctly sets a 10 MB cap: [5](#0-4) [6](#0-5) 

The Tendermint RPC WebSocket has no such protection.

---

### Impact Explanation

An attacker can open one or more WebSocket connections to the Tendermint RPC port (default 26657) and stream arbitrarily large messages. Each message is fully buffered in memory before being processed. With enough parallel connections or sufficiently large messages, this exhausts node memory and crashes the RPC server. Because the Tendermint RPC is used by validators, relayers, and monitoring infrastructure, its crash constitutes an **unauthorized RPC crash** — Medium severity per the Sei bounty scope.

---

### Likelihood Explanation

The Tendermint RPC WebSocket endpoint is publicly accessible by default. No authentication, no API key, and no rate limiting is required. The attack requires only a standard WebSocket client and a network connection to the node. It is trivially reproducible with off-the-shelf tooling.

---

### Recommendation

1. **Set a default read limit** in `newWSConnection` — e.g., `wsc.readLimit = 10 * 1024 * 1024` (10 MB) — so that `SetReadLimit` is never called with `0`.
2. **Pass a `ReadLimit` option** from every `NewWebsocketManager` call site to enforce the limit explicitly.
3. **Add per-connection rate limiting** (message count per second) to both the Tendermint and EVM WebSocket handlers to prevent flood attacks even within the size limit.

---

### Proof of Concept

```
# Connect to Tendermint RPC WebSocket (port 26657)
# Send a single JSON-RPC message with a multi-GB "params" field
# The server buffers the entire payload into memory before responding
# Repeat with parallel workers to exhaust available RAM

wscat -c ws://<node-ip>:26657/websocket
> {"jsonrpc":"2.0","method":"subscribe","params":["<14MB random string>"],"id":1}
```

With 30 parallel workers each sending messages of several hundred MB, node memory is exhausted and the RPC process is killed by the OOM killer, crashing the Tendermint RPC server.

func newWSConnection(baseConn *websocket.Conn, funcMap map[string]*RPCFunc, options ...func(*wsConnection)) *wsConnection {
	wsc := &wsConnection{
		remoteAddr:      baseConn.RemoteAddr().String(),
		baseConn:        baseConn,
		funcMap:         funcMap,
		readWait:        defaultWSReadWait,
		pingPeriod:      defaultWSPingPeriod,
		readRoutineQuit: make(chan struct{}),
	}
	for _, option := range options {
		option(wsc)
	}
	wsc.baseConn.SetReadLimit(wsc.readLimit)
	return wsc
}
