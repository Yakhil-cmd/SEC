### Title
Missing Intrinsic Gas Floor in ZK Transaction Refund Calculation Allows Underpayment of Intrinsic Gas - (`File: basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`compute_calldata_tokens` returns `minimal_gas_used = 0` for ZK-style transactions when the `eip_7623` feature is disabled, while the Ethereum-style transaction path hard-codes `minimal_gas_used = TX_INTRINSIC_GAS (21 000)`. Because `minimal_gas_used` is the floor passed to `compute_gas_refund`, EVM SSTORE-clear refunds can reduce the final `gas_used` below the intrinsic gas cost for any ZK-format transaction, letting the sender underpay the operator.

---

### Finding Description

**Root cause — divergent `minimal_gas_used` between transaction types**

`compute_calldata_tokens` (shared by ZK L2 and L1 transaction paths) returns `0` as the gas floor when EIP-7623 is off:

```rust
// zk/validation_impl.rs  (also called from process_l1_transaction.rs)
#[cfg(not(feature = "eip_7623"))]
{
    (num_tokens, 0)   // ← minimal_gas_used = 0
}
``` [1](#0-0) 

The Ethereum-style path inlines the same block but uses `TX_INTRINSIC_GAS`:

```rust
// ethereum/validation_impl.rs
#[cfg(not(feature = "eip_7623"))]
{
    (num_tokens, TX_INTRINSIC_GAS)   // ← minimal_gas_used = 21 000
}
``` [2](#0-1) 

The ZK path stores this as `minimal_ergs_to_charge`:

```rust
minimal_ergs_to_charge: Ergs(minimal_gas_used.saturating_mul(ERGS_PER_GAS)),
// = Ergs(0) for ZK txs
``` [3](#0-2) 

In `before_refund` this becomes the `min_gas_used` argument to `compute_gas_refund`:

```rust
let min_gas_used = context.minimal_ergs_to_charge.0 / ERGS_PER_GAS;  // = 0
let refund_info = compute_gas_refund(system, ..., min_gas_used, ...)?;
``` [4](#0-3) 

Inside `compute_gas_refund`, the floor is applied as:

```rust
let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);
// For ZK txs: max(gas_used, 0) = gas_used  → no floor
// For ETH txs: max(gas_used, 21000)         → intrinsic gas protected
``` [5](#0-4) 

**How the EVM refund counter interacts**

EVM SSTORE-clear operations accumulate a refund (4 800 gas per cleared slot) in the refund counter, capped at `gas_used / 5` per EIP-3529:

```rust
let evm_refund = {
    let full_refund_gas = full_refund_ergs.0.div_floor(ERGS_PER_GAS);
    let max_refund = gas_used / 5;
    core::cmp::min(full_refund_gas, max_refund)
};
gas_used -= evm_refund;
``` [6](#0-5) 

The refund counter is populated by `refund_for_storage_write` when a slot is zeroed:

```rust
if !value_at_tx_start.is_zero() && new_value.is_zero() {
    refund_counter.add_ergs(Ergs(4800 * ERGS_PER_GAS));
}
``` [7](#0-6) 

**Concrete arithmetic**

For a ZK transaction whose body uses 0 extra gas (e.g., a simple call that clears one storage slot):

| Step | ETH tx | ZK tx |
|---|---|---|
| `gas_used` after body | 21 000 | 21 000 |
| `evm_refund` (1 slot cleared) | min(4 800, 4 200) = 4 200 | min(4 800, 4 200) = 4 200 |
| after refund | 16 800 | 16 800 |
| `max(gas_used, floor)` | max(16 800, **21 000**) = **21 000** | max(16 800, **0**) = **16 800** |
| fee paid to operator | 21 000 × gas_price | **16 800 × gas_price** |

The ZK sender pays **~20 % less** than the intrinsic gas cost.

---

### Impact Explanation

An unprivileged sender of any ZK-format transaction (L2 or L1→L2) that triggers at least one SSTORE-clear during execution can reduce the gas charged to them below the intrinsic gas floor. The operator receives less fee than the protocol intends. The maximum per-transaction savings is `intrinsic_gas / 5`; for calldata-heavy transactions (large access lists, large calldata) the intrinsic gas can be hundreds of thousands of gas, making the savings proportionally larger. This is a direct resource-accounting loss for the operator/protocol.

---

### Likelihood Explanation

The attack requires only a ZK-format transaction that calls any contract which clears at least one storage slot (sets a previously non-zero slot to zero). This is a common pattern (e.g., ERC-20 `approve(spender, 0)`, any cleanup function). No privileged access, oracle manipulation, or special setup is needed. The attacker simply submits a normal ZK transaction.

---

### Recommendation

In `compute_calldata_tokens` (or at the call-site in `validate_and_compute_fee_for_transaction` for ZK transactions), return `TX_INTRINSIC_GAS` as the floor when EIP-7623 is disabled, mirroring the Ethereum transaction path:

```rust
#[cfg(not(feature = "eip_7623"))]
{
-   (num_tokens, 0)
+   (num_tokens, TX_INTRINSIC_GAS)
}
``` [1](#0-0) 

This ensures `compute_gas_refund` cannot reduce `gas_used` below the intrinsic gas cost for ZK transactions, consistent with the Ethereum transaction path. [5](#0-4) 

---

### Proof of Concept

1. Deploy a contract with a non-zero storage slot (e.g., slot 0 = 1).
2. Add a function that sets slot 0 to 0 (SSTORE clear → +4 800 gas refund).
3. Send a ZK-format transaction calling that function with `gas_limit = 30 000`.

**Expected (ETH tx):** `gas_used = max(21 000 − 4 200, 21 000) = 21 000`
**Actual (ZK tx):** `gas_used = max(21 000 − 4 200, 0) = 16 800`

The operator is paid for 16 800 gas instead of 21 000 gas — a 4 200 gas shortfall per transaction. Scaling to high-calldata transactions (intrinsic gas >> 21 000) amplifies the savings proportionally.
