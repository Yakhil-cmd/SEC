### Title
`native_per_pubdata` Truncates to Zero via Floor Division, Allowing Free Pubdata Generation — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio — which controls how many native resource units a transaction must pay per byte of pubdata (L1 data availability) — is computed using `wrapping_div` (truncating integer division). When `pubdata_price < native_price`, this truncates to zero. With `native_per_pubdata == 0`, all pubdata cost checks pass trivially (0 × bytes = 0), allowing a transaction to generate the maximum allowed pubdata without paying any native resource cost for L1 DA. The operator bears the L1 DA cost without compensation.

---

### Finding Description

In `validate_and_compute_fee_for_transaction`, line 142:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is Rust's wrapping (floor/truncating) integer division. When `pubdata_price < native_price` — for example `pubdata_price = 500 wei/byte`, `native_price = 1000 wei/native_unit` — the result is `500 / 1000 = 0`. There is no guard for the case `native_per_pubdata == 0 && pubdata_price > 0`.

The zero propagates through every downstream pubdata cost site:

1. **Intrinsic pubdata overhead** (`create_resources_for_tx`, line 352):
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   // = 0 * intrinsic_pubdata = 0 → nothing deducted from native limit
   ```

2. **Runtime pubdata charge** (`get_resources_to_charge_for_pubdata`, line 430–432):
   ```rust
   let native = current_pubdata_spent
       .checked_mul(native_per_pubdata)   // = pubdata_bytes * 0 = 0
       .ok_or(out_of_native_resources!())?;
   ```

3. **Post-execution pubdata check** (`check_enough_resources_for_pubdata`): always passes because the charge is 0.

The same truncation appears in the off-chain helper `validate_l2_tx_intrinsic_native_resources` (`api/src/helpers.rs`, line 427):
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

**Contrast with `native_per_gas`**, which uses ceiling division to prevent undercharging:
```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

The inconsistency — `div_ceil` for gas, `wrapping_div` for pubdata — is the root cause. There is also no `native_per_pubdata == 0` guard analogous to the `native_per_gas == 0` guard that sets `free_native = true` and grants unlimited native.

---

### Impact Explanation

When `native_per_pubdata == 0`:

- A transaction can write up to `pubdata_limit` bytes of state diffs (via SSTORE, contract deployments, etc.) without consuming any native resources for L1 DA.
- The operator must still publish those bytes to L1 and pay the corresponding DA cost, but receives zero compensation from the transaction fee.
- The attacker pays only EVM gas (ergs) for the EVM opcodes themselves, not for the L1 DA cost of the resulting state diffs.
- At `pubdata_limit = 100,000 bytes` and `pubdata_price = 999 wei/byte`, the operator absorbs ~99,900,000 wei of uncompensated L1 DA cost per transaction.

This is a **resource accounting bug / operator funds-loss path**.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is realistic:

- `native_price` reflects the cost of ZK proving (RISC-V cycles), which can be high.
- `pubdata_price` reflects L1 DA cost, which fluctuates independently.
- An operator setting `native_price = 1000` and `pubdata_price = 999` intends to charge ~1 native unit per pubdata byte but actually charges 0 due to truncation.
- The operator has no in-protocol warning that truncation has occurred; the transaction validates and executes normally.
- The default test `BlockContext` already uses `pubdata_price: U256::from(0)` and `native_price: U256::from(10)`, confirming the zero case is reachable and untested for the non-zero-but-less-than-native-price scenario.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with `native_per_gas`:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, ensuring pubdata_price > 0 always yields >= 1):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427.

Alternatively, add an explicit guard:
```rust
if native_per_pubdata == 0 && !pubdata_price.is_zero() {
    return Err(TxError::Validation(InvalidTransaction::PubdataPriceTooLow));
}
```

### Proof of Concept

**Setup:**
- `native_price = 1000` (e.g., 1000 wei per native unit)
- `pubdata_price = 999` (e.g., 999 wei per pubdata byte — operator intends to charge ~1 native unit/byte)

**Computation:**
```
native_per_pubdata = 999.wrapping_div(1000) = 0
```

**Attack:**
1. Attacker submits an EIP-1559 transaction with `max_fee_per_gas` satisfying the base fee check.
2. Transaction body executes 1,000 `SSTORE` operations writing to new slots → ~32,000 bytes of pubdata.
3. `get_resources_to_charge_for_pubdata` computes `32000 * 0 = 0` native cost.
4. `check_enough_resources_for_pubdata` passes; transaction succeeds.
5. Operator publishes 32,000 bytes to L1 at 999 wei/byte = **31,968,000 wei uncompensated**.
