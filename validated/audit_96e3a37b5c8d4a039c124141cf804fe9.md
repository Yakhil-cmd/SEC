Audit Report

## Title
Unescaped Memo Bytes Enable Markdown Injection into ICRC-21 GenericDisplayMessage — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
The `add_memo` function in `responses.rs` interpolates attacker-controlled UTF-8 memo bytes directly into a single-backtick Markdown code span without escaping backticks or newlines. Any unprivileged caller can submit a `ConsentMessageRequest` with a crafted memo that breaks out of the code span and injects arbitrary Markdown — including fake bold fields like `**Amount:**` or `**To:**` — into the `GenericDisplayMessage` returned by the ICRC-21 consent endpoint. This breaks the core security invariant of ICRC-21: that the consent message faithfully and exclusively represents the transaction the user is authorizing.

## Finding Description
In `add_memo` at line 294 of `packages/icrc-ledger-types/src/icrc21/responses.rs`, when the memo bytes are valid UTF-8, the decoded string is placed verbatim inside a single-backtick code span:

```rust
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
```

No escaping of backticks or newlines is performed on `memo_str`. In CommonMark, a single-backtick code span ends at the next single backtick. If `memo_str` contains a backtick, the code span is terminated early and the remainder of `memo_str` is parsed as raw Markdown.

**Concrete injection example:**

Memo bytes (valid UTF-8, ≤32 bytes): `` `\n\n**Amount:** `999999 ICP ``

Resulting `GenericDisplayMessage` fragment:
```
**Memo:**
``

**Amount:** `999999 ICP`
```

A wallet rendering this as Markdown (as the ICRC-21 spec requires) displays a bold `**Amount:**` field followed by a code-span `999999 ICP` — visually indistinguishable from the real `**Amount:**` field generated by `add_amount` at line 114.

The call path is direct and requires no privileges:
1. Caller submits `icrc21_canister_call_consent_message` with a `TransferArg` whose `memo` field contains the crafted bytes.
2. `build_icrc21_consent_info` decodes the arg and passes `GenericMemo::Icrc1Memo(memo.0)` to the builder (lib.rs lines 392–394).
3. `ConsentMessageBuilder::build` calls `message.add_memo(memo)` unconditionally if a memo is present (lib.rs lines 290–291).
4. `add_memo` decodes the bytes as UTF-8 and interpolates without escaping (responses.rs lines 288–294).

Non-UTF-8 bytes are hex-encoded (line 290) and are not injectable, but the UTF-8 path has no such protection.

## Impact Explanation
The `GenericDisplayMessage` is the primary human-readable consent surface for ICRC-21. The ICRC-21 standard specifies that wallets should render it as Markdown. A malicious dApp can inject fake `**Amount:**`, `**To:**`, or `**Fees:**` fields that are visually identical to the real ones generated by the ledger. Because the memo section is appended last, the injected fields appear after the real ones. A wallet UI that paginates, truncates, or scrolls the consent message may present the injected content more prominently than the real fields, enabling a user to be deceived into approving a transaction with parameters different from what they believe they are authorizing. This constitutes a significant ledger security impact with concrete user harm — potential unauthorized fund transfers — matching the High impact category for ICP ledger security.

## Likelihood Explanation
Any unprivileged caller can submit a `ConsentMessageRequest` with a crafted memo via the public `icrc21_canister_call_consent_message` update endpoint. The ICRC-1 memo field accepts up to 32 bytes, which is sufficient to inject a fake field. Wallets implementing ICRC-21 (Plug, NFID, hardware wallets) are expected to render `GenericDisplayMessage` as Markdown per the spec. The attack requires no special privileges, no key material, and no governance majority. The only constraint is that the victim wallet renders Markdown and the user does not read the full consent message from top to bottom — both realistic conditions in production wallet UIs.

## Recommendation
Escape all Markdown control characters in `memo_str` before interpolation. At minimum, replace backticks and newlines:

```rust
let safe_memo = memo_str
    .replace('`', "\\`")
    .replace('\n', " ")
    .replace('\r', " ");
message.push_str(&format!("\n\n**Memo:**\n`{safe_memo}`"));
```

Alternatively, hex-encode all memo bytes unconditionally for `GenericDisplayMessage` (as is already done for non-UTF-8 bytes at line 290), since memo bytes have no semantic meaning that requires human-readable display.

## Proof of Concept
```rust
let memo_bytes = b"`\n\n**Amount:** `999999 ICP".to_vec();
let memo = GenericMemo::Icrc1Memo(memo_bytes);
let mut msg = ConsentMessage::GenericDisplayMessage(String::new());
msg.add_intent(Icrc21Function::Transfer, Some("ICP".to_string()));
msg.add_amount(Some(Nat::from(1u64)), 8, &"ICP".to_string()).unwrap();
msg.add_memo(memo);

if let ConsentMessage::GenericDisplayMessage(text) = &msg {
    assert!(
        !text.contains("\n\n**Amount:** `999999 ICP`"),
        "FAIL: injected fake Amount field present in consent message:\n{text}"
    );
}
```

This unit test can be added directly to `packages/icrc-ledger-types/src/icrc21/responses.rs` and will fail against the current unpatched code, demonstrating the injection.
