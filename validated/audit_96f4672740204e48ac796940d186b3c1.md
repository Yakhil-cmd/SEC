### Title
Pre-EIP-155 Legacy Transactions Bypass Chain ID Validation, Enabling Cross-Deployment Replay Attacks - (File: `engine/src/engine.rs`)

---

### Summary

The `submit_with_alt_modexp` function in `engine/src/engine.rs` only validates the chain ID when it is present in the transaction signature. Pre-EIP-155 legacy transactions (signed with `v = 27` or `v = 28`) carry no chain ID, so `transaction.chain_id` is `None` and the guard is silently skipped. The same signed transaction bytes are therefore accepted by every Aurora deployment (Mainnet chain ID `1313161554`, Testnet `1313161555`, Local `1313161556`), enabling a cross-deployment replay attack that can drain a victim's funds on a chain they never intended to interact with.

---

### Finding Description

**Root cause — conditional chain ID guard in `submit_with_alt_modexp`:**

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

The `if let Some(chain_id)` pattern means the entire check is a no-op when `chain_id` is `None`. [1](#0-0) 

**How `chain_id` becomes `None` — `LegacyEthSignedTransaction::chain_id()`:**

```rust
pub const fn chain_id(&self) -> Option<u64> {
    match self.v {
        0..=34 => None,          // v = 27 or 28 → pre-EIP-155, no chain ID
        _ => Some((self.v - 35) / 2),
    }
}
```

A legacy transaction signed with `v = 27` or `v = 28` (the pre-EIP-155 convention) returns `None`. The parser explicitly accepts these values rather than rejecting them. [2](#0-1) [3](#0-2) 

**Propagation through `NormalizedEthTransaction`:**

```rust
Legacy(tx) => Self {
    address: tx.sender()?,
    chain_id: tx.chain_id(),   // None for pre-EIP-155
    ...
}
```

The `None` value flows unchanged into the normalized transaction that `submit_with_alt_modexp` receives, so the guard at line 1055 is never triggered. [4](#0-3) 

**Attacker-controlled entry path:**

Any unprivileged user can call the public `submit` NEAR entry point with a pre-EIP-155 RLP-encoded transaction. No special privileges are required. [5](#0-4) 

---

### Impact Explanation

**Critical — direct theft of user funds.**

A victim signs a pre-EIP-155 legacy transaction on Aurora Testnet (e.g., to transfer ETH to a friend). An attacker observes the raw transaction bytes and submits them verbatim to Aurora Mainnet via the `submit` entry point. If the victim's nonce on Mainnet matches the replayed transaction's nonce (common for accounts that have not yet been used on Mainnet, or for accounts whose nonce happens to align), the Mainnet engine accepts and executes the transaction, transferring real ETH from the victim's Mainnet balance to the same destination address — an outcome the victim never authorized on Mainnet.

---

### Likelihood Explanation

**Medium.**

- Pre-EIP-155 transactions (`v = 27/28`) are still producible by older wallets, hardware signers in compatibility mode, raw RLP tooling, and some programmatic signers. They are not exotic.
- The attacker only needs to observe a single pre-EIP-155 transaction on any Aurora deployment (e.g., by watching the NEAR blockchain) and replay it on another deployment where the sender's nonce matches.
- Nonce alignment is most likely for accounts that are active on one deployment but have never transacted on another — a common situation for testnet-only or mainnet-only users.
- No admin access, key compromise, or governance capture is required.

---

### Recommendation

Reject pre-EIP-155 legacy transactions outright in `submit_with_alt_modexp`. Change the conditional guard to also error when `chain_id` is absent:

```rust
match transaction.chain_id {
    None => return Err(EngineErrorKind::InvalidChainId.into()),
    Some(chain_id) if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) => {
        return Err(EngineErrorKind::InvalidChainId.into());
    }
    _ => {}
}
```

This mirrors the fix applied in the referenced Eco report (`route.destination != block.chainid` → hard revert) and closes the replay surface entirely. [1](#0-0) 

---

### Proof of Concept

1. Alice holds 10 ETH on Aurora Mainnet (chain ID `1313161554`) and 0.01 ETH on Aurora Testnet (chain ID `1313161555`).
2. Alice uses an older tool to sign a pre-EIP-155 legacy transaction on Testnet: send 0.01 ETH to Bob, nonce = 0, `v = 27`. The transaction is broadcast and included on Testnet.
3. Alice has never transacted on Mainnet, so her Mainnet nonce is also 0.
4. Attacker Eve copies the raw RLP bytes of Alice's Testnet transaction and calls `submit` on Aurora Mainnet with those bytes.
5. `submit_with_alt_modexp` parses the transaction: `chain_id = None` (because `v = 27`). The guard at line 1055 evaluates `if let Some(chain_id) = None` → false → skipped.
6. `check_nonce` passes (Alice's Mainnet nonce is 0, matching the transaction).
7. The EVM executes the transfer: 0.01 ETH leaves Alice's Mainnet balance and arrives at Bob's address on Mainnet.
8. Eve can repeat this for any pre-EIP-155 transaction Alice signs on Testnet, draining Alice's Mainnet balance one transaction at a time. [6](#0-5) [7](#0-6)
