### Title
ETH Permanently Frozen in `exit_to_near::ADDRESS` When `ft_transfer` Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` compile-time feature is absent (which is the case in the production build), the `ExitToNear` precompile schedules the NEAR `ft_transfer` promise with **no failure callback** for native ETH exits. If the `ft_transfer` promise fails, there is no mechanism to return the ETH to the user. The ETH is permanently locked at the `exit_to_near::ADDRESS` precompile address with no recovery path.

### Finding Description

The `ExitToNear` precompile's `run()` function constructs `callback_args` conditionally on the `error_refund` feature flag: [1](#0-0) 

When `error_refund` is **not** enabled, `refund` is hardcoded to `None`. For a base-ETH exit (flag `0x0`), `exit_base_token_to_near` also returns `transfer_near_args = None`: [2](#0-1) 

This means `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()` (both fields `None`). The promise-creation branch then fires: [3](#0-2) 

A bare `PromiseArgs::Create` is scheduled — **no callback at all**. If the `ft_transfer` to the eth-connector fails (e.g., recipient account not registered with the NEP-141 token), NEAR simply discards the failure silently. The ETH that was transferred into `exit_to_near::ADDRESS` during the EVM call is never returned.

The refund logic in `refund_on_error` correctly handles the ETH case via a native EVM value transfer: [4](#0-3) 

But this code is **never reached** because the callback is never scheduled when `error_refund` is absent.

The production build feature set in `Makefile.toml` is `contract` only — `error_refund` is not included: [5](#0-4) 

The `error_refund` feature is a separate opt-in: [6](#0-5) 

The integration test explicitly acknowledges the fund loss when the feature is absent: [7](#0-6) 

### Impact Explanation

**Critical — Permanent freezing of funds.** Any ETH sent through the `ExitToNear` precompile (base-token path) whose downstream `ft_transfer` promise fails is irrecoverably locked at `exit_to_near::ADDRESS` (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`). There is no admin recovery function, no sweep mechanism, and no way for the user to reclaim the funds.

### Likelihood Explanation

The `ft_transfer` promise can fail for ordinary, user-reachable reasons: the recipient NEAR account is not registered with the NEP-141 eth-connector, the account does not exist, or the connector is paused. Any EVM user who calls `ExitToNear` with ETH value targeting such an account triggers the loss. No special privilege is required.

### Recommendation

Enable the `error_refund` feature in the production build (`CARGO_FEATURES_BUILD = "contract,error_refund"`), or unconditionally schedule the failure callback for the base-ETH exit path regardless of the feature flag, so that `exit_to_near_precompile_callback` is always invoked and can execute `refund_on_error` to return ETH to the user on promise failure. [8](#0-7) 

### Proof of Concept

1. User calls an EVM contract that invokes the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` with flag `0x00` (base ETH), sending `N` ETH and targeting `unregistered.near` as the NEAR recipient.
2. The EVM deducts `N` ETH from the caller and credits `exit_to_near::ADDRESS`.
3. The precompile (with `error_refund` absent) builds `callback_args = { refund: None, transfer_near: None }` which equals `default()`, so it emits `PromiseArgs::Create(ft_transfer_promise)` — no callback.
4. The NEAR runtime executes `ft_transfer` on the eth-connector; it fails because `unregistered.near` has no storage deposit.
5. No callback fires. `exit_to_near::ADDRESS` retains `N` ETH permanently.
6. The user's ETH is gone with no recovery path.
