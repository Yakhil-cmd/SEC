### Title
Unchecked Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Burn Without Exit - (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first burning the caller's ERC-20 tokens and then invoking the Aurora exit precompile via inline assembly. The return value of the low-level `call` opcode (`res`) is captured but **never checked**. If the precompile call fails with `ExitError` (not `Fatal`), the EVM sub-call returns 0, the parent function continues normally, and the tokens are permanently destroyed with no corresponding NEP-141 release on the NEAR side.

---

### Finding Description

In `EvmErc20.sol` and `EvmErc20V2.sol`, both exit functions follow the same pattern:

```solidity
// EvmErc20.sol lines 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no require(res == 1)
    }
}
```

The same unchecked pattern appears in `withdrawToEthereum` (lines 65–76 of `EvmErc20.sol` and lines 66–77 of `EvmErc20V2.sol`).

The precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` is the `exit_to_near` precompile. In `engine-precompiles/src/lib.rs`, when a precompile is **paused**, it returns `PrecompileFailure::Fatal`, which propagates as a transaction-level abort and would revert `_burn`. However, the precompile's `run` function also returns `ExitError` variants in several code paths:

```rust
// engine-precompiles/src/native.rs lines 406-416
if let Some(target_gas) = target_gas && required_gas > target_gas {
    return Err(ExitError::OutOfGas);
}
if is_static {
    return Err(ExitError::Other(Cow::from("ERR_INVALID_IN_STATIC")));
} else if context.address != exit_to_near::ADDRESS.raw() {
    return Err(ExitError::Other(Cow::from("ERR_INVALID_IN_DELEGATE")));
}
let exit_to_near_params = ExitToNearParams::try_from(input)?;  // can return ExitError
```

When the precompile returns `ExitError` (not `Fatal`), the EVM `call` opcode returns 0 and execution of the parent frame continues. The `_burn` is **not** reverted. The tokens are gone.

**Concrete trigger**: A user calling `withdrawToNear` with an invalid or empty `recipient` (e.g., zero-length bytes, a recipient string that fails NEAR account ID validation inside the precompile) causes `ExitToNearParams::try_from(input)` to return `ExitError`. The precompile call returns 0, `res` is silently discarded, and the function returns successfully — with the user's tokens permanently burned.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The ERC-20 tokens are burned on the Aurora side (removed from total supply), but the corresponding NEP-141 tokens on the NEAR side are never released. The user's funds are permanently destroyed with no recovery path. This affects every holder of any NEP-141-backed ERC-20 token deployed via `EvmErc20` or `EvmErc20V2`.

---

### Likelihood Explanation

**High.** Any user who supplies an invalid `recipient` argument to `withdrawToNear` (e.g., an empty byte string, a string exceeding NEAR account ID length limits, or a malformed account ID) will trigger the silent failure. This is a user-reachable code path with no access control — any token holder can call it. The failure is silent: the transaction succeeds (no revert), the user sees no error, and their tokens are gone.

---

### Recommendation

After the assembly `call`, check the return value and revert if the precompile call failed:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    bool success;
    assembly {
        success := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    }
    require(success, "ERR_EXIT_TO_NEAR_FAILED");
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. Alternatively, restructure to call the precompile **before** burning, and only burn if the precompile call succeeds.

---

### Proof of Concept

1. Deploy an `EvmErc20` token (or use an existing one on Aurora).
2. Mint tokens to `alice`.
3. `alice` calls `withdrawToNear("", amount)` — an empty recipient byte string.
4. `_burn(alice, amount)` executes: alice's balance is reduced to 0, total supply decreases.
5. The precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` receives input `"\x01" + amount_b + ""`. Parsing fails inside `ExitToNearParams::try_from` with `ExitError`.
6. The assembly `call` returns 0. `res` is discarded. The function returns normally.
7. The transaction is marked successful. Alice's ERC-20 tokens are gone. No NEP-141 tokens are released on NEAR.
8. Alice has permanently lost her funds.
