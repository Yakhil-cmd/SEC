Audit Report

## Title
Unbounded Prometheus Label Cardinality via Attacker-Controlled URI Path and `network_identifier.network` Causes Memory Exhaustion — (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`)

## Summary
`RosettaMetricsMiddleware::call` uses the raw HTTP request URI path and the `network_identifier.network` JSON body field as Prometheus label values without sanitization, allowlisting, or cardinality caps. An unauthenticated HTTP client can exhaust the Rosetta process heap by sending requests with unboundedly many unique paths and/or unique `network_identifier.network` values, each permanently allocating a new `Histogram` time series (with its full bucket array) in the global Prometheus registry. The `CANISTER_DISPLAY_NAMES` `HashMap` also grows without bound as a secondary allocation sink.

## Finding Description
**Vector 1 — `endpoint` label (raw URI path):**
At line 326, `let path = req.uri().path().to_owned()` captures the raw URI path with no normalization. At line 369, this is passed to `metrics.observe_request_duration(&path, ...)`, which at lines 167–171 calls `ENDPOINTS_METRICS.request_duration.with_label_values([token, path, method, status]).observe(duration)`. Every distinct path string creates a new `HistogramVec` time series (15+ buckets each) permanently in the global Prometheus registry.

**Vector 2 — `token_display_name` label (`network_identifier.network`):**
At lines 397–402, `extract_canister_id` reads `network_identifier.network` from the JSON body and returns it as `canister_id`. At lines 342–348, `RosettaMetrics::get_display_name_from_canister_id` is called; at lines 253–258 it falls back to returning the raw attacker-supplied string when no mapping exists. `RosettaMetrics::new` is then called with this raw string, which at lines 119–120 inserts it into `CANISTER_DISPLAY_NAMES`, growing the `HashMap` without bound. The raw string is also used as the `token_display_name` Prometheus label, creating unbounded cardinality in that dimension as well.

**Middleware scope:**
The `metrics_layer` is applied to the entire Axum `Router` at line 383 of `main.rs`, and the server binds to `0.0.0.0` at line 393. Every HTTP request — including those to unregistered paths returning 404 — passes through the metrics recording path. No authentication is required.

**No existing guards:** There is no path normalization, no allowlist check on `network_identifier.network`, no cardinality cap, and no rate limiting on label creation.

## Impact Explanation
Each unique `(token_display_name, endpoint, method, status)` tuple permanently allocates a new `Histogram` with its full bucket array in the global Prometheus registry. Sending N unique paths creates O(N × buckets) live allocations that are never freed. At 10,000 unique paths with the default 11 histogram buckets, this creates ~110,000 live histogram objects. Combined with unbounded `CANISTER_DISPLAY_NAMES` `HashMap` growth, this exhausts process heap memory, causing the Rosetta process to crash (OOM kill or allocation panic), making the Rosetta API unavailable to all legitimate clients.

This matches: **High ($2,000–$10,000) — Application/platform-level DoS, crash, or availability impact not based on raw volumetric DDoS.** The Rosetta API is explicitly in-scope as a financial integration component.

## Likelihood Explanation
The Rosetta HTTP port is publicly accessible by design (`0.0.0.0:8080`, line 393 of `main.rs`). No authentication or credentials are required. The attack requires only the ability to send HTTP POST requests with arbitrary paths and JSON bodies. It is trivially scriptable, requires no special knowledge, and is persistent — each unique request permanently grows the registry. The attack is qualitatively distinct from raw volumetric DDoS: a modest number of requests (10,000) causes unbounded permanent memory growth rather than transient load.

## Recommendation
- **Normalize the `endpoint` label** to the matched Axum route template using `axum::extract::MatchedPath` (returns `/block` rather than the raw URI). For unmatched routes, use a fixed label value such as `"unknown"`.
- **Validate `network_identifier.network`** against a pre-registered allowlist of known canister IDs before using it as a label value. If not in the allowlist, use the configured default `token_display_name`.
- **Remove the `CANISTER_DISPLAY_NAMES` insertion** from the per-request hot path (`RosettaMetrics::new` lines 119–120); populate it only at startup from trusted configuration.

## Proof of Concept
```python
import requests, threading

TARGET = "http://<rosetta-host>:8080"

def flood(n):
    for i in range(n):
        requests.post(
            f"{TARGET}/unique_path_{i}",
            json={"network_identifier": {"network": f"fake-canister-{i}"}},
            timeout=2
        )

threads = [threading.Thread(target=flood, args=(2500,)) for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()
# Monitor Rosetta RSS: watch -n1 'ps aux | grep rosetta'
# Expect unbounded memory growth and eventual OOM crash.
```
