### Title
Reimbursement Address Hardcoded to Caller Principal in ckBTC and ckETH Minters Can Lock Tokens if Caller Canister Is Deleted — (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`, `rs/ethereum/cketh/minter/src/main.rs`)

---

### Summary

The ckBTC minter's `retrieve_btc_with_approval` and the ckETH minter's `withdraw_eth` both hardcode the reimbursement destination to the `caller` principal at the time of the withdrawal request, with no option to specify a separate recovery address. If the caller is a canister that is subsequently deleted (or otherwise rendered unable to spend ledger tokens), any ckBTC or ckETH reimbursed after a failed cross-chain transaction is minted to an inaccessible ledger account, permanently locking the funds.

---

### Finding Description

**ckBTC minter — `retrieve_btc_with_approval`**

In `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`, the `reimbursement_account` field of every `RetrieveBtcRequest` is unconditionally set to the IC caller's principal and subaccount:

```rust
let request = RetrieveBtcRequest {
    ...
    reimbursement_account: Some(Account {
        owner: caller,
        subaccount: args.from_subaccount,
    }),
};
``` [1](#0-0) 

The `RetrieveBtcWithApprovalArgs` struct accepted by this endpoint contains no field for a user-specified reimbursement address. When a BTC withdrawal fails (e.g., due to a tainted address check, fee estimation failure, or Bitcoin network rejection), the minter calls `reimburse_withdrawals`, which mints ckBTC back to `reimbursement_account`: [2](#0-1) 

**ckETH minter — `withdraw_eth`**

In `rs/ethereum/cketh/minter/src/main.rs`, the `EthWithdrawalRequest` stores `from: caller` and `from_subaccount` with no separate reimbursement address field:

```rust
let withdrawal_request = EthWithdrawalRequest {
    withdrawal_amount: amount,
    destination,
    ledger_burn_index,
    from: caller,
    from_subaccount: from_subaccount.and_then(LedgerSubaccount::from_bytes),
    created_at: Some(now),
};
``` [3](#0-2) 

The `WithdrawalArg` Candid type accepted by `withdraw_eth` has no reimbursement address field: [4](#0-3) 

When the Ethereum transaction fails, `process_reimbursement` mints ckETH back to `reimbursement_request.to` / `reimbursement_request.to_subaccount`, which are derived directly from `withdrawal_request.from` / `withdrawal_request.from_subaccount`: [5](#0-4) 

The `ReimbursementRequest` struct confirms the reimbursement destination is always the original caller: [6](#0-5) 

---

### Impact Explanation

On the Internet Computer, ICRC-1 ledger balances are keyed by `Account { owner: Principal, subaccount }`. If a canister principal is deleted, its ledger balance persists but becomes permanently inaccessible — no entity can sign transactions on behalf of a deleted canister. Any ckBTC or ckETH minted to a deleted canister's principal is therefore irretrievably locked.

A canister-based DeFi protocol (e.g., a liquidity aggregator or automated vault) that calls `retrieve_btc_with_approval` or `withdraw_eth` on behalf of its users is the realistic caller. If that canister is deleted (e.g., during a protocol migration, upgrade to a new canister ID, or emergency shutdown) between the time the withdrawal is submitted and the time the reimbursement is processed, the reimbursed tokens are locked. The minter has no mechanism to redirect the reimbursement to a different address after the fact.

---

### Likelihood Explanation

The likelihood is **low** but non-zero. Canister deletion is an explicit operator action, but it is a normal part of the IC canister lifecycle (e.g., protocol migrations, emergency stops, or replacement with a new canister ID). The window between a withdrawal submission and its reimbursement can span multiple minutes to hours (Bitcoin finality, Ethereum finality, minter processing cycles). Any canister-based protocol that automates ckBTC/ckETH withdrawals is exposed during this window. The scenario requires no privileged access, no governance majority, and no threshold corruption — only the normal deletion of the calling canister.

---

### Recommendation

Add an optional `reimbursement_account` field to `RetrieveBtcWithApprovalArgs` and `WithdrawalArg`. If provided, the minter should use it as the reimbursement destination instead of the caller's principal. If absent, fall back to the current behavior (reimburse to caller). This mirrors the fix applied to AmarokFacet (taking `recovery` as an argument rather than hardcoding `msg.sender`).

For ckBTC:
- Add `reimbursement_account: Option<Account>` to `RetrieveBtcWithApprovalArgs`
- In `retrieve_btc_with_approval`, use `args.reimbursement_account.unwrap_or(Account { owner: caller, subaccount: args.from_subaccount })` when constructing `RetrieveBtcRequest`

For ckETH:
- Add `reimbursement_account: Option<Account>` to `WithdrawalArg`
- In `withdraw_eth`, store the reimbursement account separately from `from`/`from_subaccount` in `EthWithdrawalRequest` (or a new field), and use it in `ReimbursementRequest`

---

### Proof of Concept

1. Deploy a canister `C` on the IC.
2. From `C`, call `retrieve_btc_with_approval` on the ckBTC minter with a valid BTC address and amount. The minter records `reimbursement_account = { owner: C, subaccount: None }`.
3. Before the BTC transaction is confirmed (or if it fails), delete canister `C` via `ic00::delete_canister`.
4. The minter's `reimburse_withdrawals` timer fires and calls `mint_ckbtc` with `to = { owner: C, subaccount: None }`.
5. The ledger mint succeeds (the ledger does not check canister liveness), recording a balance for principal `C`.
6. No entity can ever spend this balance — `C` is deleted and its principal has no controller. [7](#0-6) [8](#0-7)

pub async fn reimburse_withdrawals<R: CanisterRuntime>(runtime: &R) {
    if state::read_state(|s| s.pending_withdrawal_reimbursements.is_empty()) {
        return;
    }
    let pending_reimbursements = state::read_state(|s| s.pending_withdrawal_reimbursements.clone());
    let mut error_count = 0;
    for (burn_index, reimbursement) in pending_reimbursements {
        // Ensure that even if we were to panic in the callback, after having contacted the ledger to mint the tokens,
        // this reimbursement request will not be processed again.
        let prevent_double_minting_guard = scopeguard::guard(burn_index, |index| {
            state::mutate_state(|s| {
                state::audit::quarantine_withdrawal_reimbursement(s, index, runtime)
            });
        });
        let memo = MintMemo::ReimburseWithdrawal {
            withdrawal_id: burn_index,
        };
        match runtime
            .mint_ckbtc(
                reimbursement.amount,
                reimbursement.account,
                Memo::from(crate::memo::encode(&memo)),
            )
            .await
        {
            Ok(mint_index) => {
                log!(
                    Priority::Debug,
                    "[reimburse_withdrawals]: Successfully reimbursed {:?} at mint block index {}",
                    reimbursement,
                    mint_index
                );
                state::mutate_state(|s| {
                    state::audit::reimburse_withdrawal_completed(s, burn_index, mint_index, runtime)
                });
            }
            Err(err) => {
                log!(
                    Priority::Info,
                    "[reimburse_withdrawals]: Failed to reimburse {:?}: {:?}. Will retry later",
                    reimbursement,
                    err
                );
                error_count += 1;
            }
        }
        // Defuse the guard. Note that in case of a panic in the callback (either before or after this point)
        // the defuse will not be effective (due to state rollback), and the guard that was
        // setup before the `mint_ckbtc` async call will be invoked.
        scopeguard::ScopeGuard::into_inner(prevent_double_minting_guard);
    }
