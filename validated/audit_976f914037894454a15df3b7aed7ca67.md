### Title
ckERC20 Minter Over-Mints Tokens for Fee-on-Transfer ERC20 Assets Due to Event Amount vs. Actual Received Amount Discrepancy - (File: `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`)

---

### Summary

The ckETH/ckERC20 deposit helper contracts emit the user-specified `amount` parameter in the `ReceivedEthOrErc20` / `ReceivedErc20` event rather than the actual amount received by the minter's Ethereum address after `safeTransferFrom`. For fee-on-transfer or rebasing ERC20 tokens (e.g., USDT with fees enabled), the minter's address receives `amount - fee`, but the IC minter canister mints the full `amount` of ckERC20 and records `amount` in its internal balance. Over time, the total ckERC20 supply exceeds the actual ERC20 backing held by the minter, making the system insolvent and causing a denial of service for the last withdrawers.

---

### Finding Description

**Root Cause — Helper Contracts:**

Both deployed helper contracts emit the caller-supplied `amount` directly in the deposit event, without checking the actual balance change at the minter address:

`DepositHelperWithSubaccount.sol` (`CkDeposit.depositErc20`):
```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);
//                                                  ^^^^^^ user-supplied, not actual received
``` [1](#0-0) 

`ERC20DepositHelper.sol` (`CkErc20Deposit.deposit`):
```solidity
erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
emit ReceivedErc20(erc20_address, msg.sender, amount, principal);
//                                             ^^^^^^ user-supplied, not actual received
``` [2](#0-1) 

**Root Cause — IC Minter Canister:**

The IC minter scrapes these event logs and mints exactly `event.value()` — the amount from the event — to the user, with no cross-check against the actual on-chain balance:

```rust
amount: event.value(),  // taken directly from the log event
``` [3](#0-2) 

The minter's internal `erc20_balances` is also incremented by the event-reported value, not the actual received value:

```rust
fn update_balance_upon_deposit(&mut self, event: &ReceivedEvent) {
    match event {
        ReceivedEvent::Erc20(event) => self
            .erc20_balances
            .erc20_add(event.erc20_contract_address, event.value),
    };
}
``` [4](#0-3) 

For a fee-on-transfer token with a 1% fee, a deposit of `1_000_000` units causes:
- Minter receives: `990_000` units of ERC20
- Minter mints: `1_000_000` units of ckERC20
- Minter's internal balance records: `1_000_000`

After N such deposits, the minter holds `N × 990_000` ERC20 but has issued `N × 1_000_000` ckERC20. The shortfall grows with every deposit.

---

### Impact Explanation

**Vulnerability type:** Chain-fusion mint/burn conservation bug (ledger conservation bug).

The 1:1 backing invariant between ckERC20 supply and actual ERC20 held by the minter is broken. When users attempt to withdraw ckERC20 → ERC20, the minter issues Ethereum transactions for the full withdrawal amount. Once the cumulative shortfall exceeds the minter's actual ERC20 balance, the minter's Ethereum transactions will fail (insufficient balance), and the last withdrawers cannot redeem their ckERC20 tokens. No funds are immediately lost, but users are permanently unable to access their assets through the normal withdrawal path.

The minter's `erc20_sub` will also eventually panic with an underflow if the internal accounting diverges enough from reality: [5](#0-4) 

---

### Likelihood Explanation

**Medium-low.** Currently supported ckERC20 tokens (USDC, USDT) do not have active transfer fees. However:

1. **USDT** has a built-in fee mechanism (`basisPointsRate`, `maximumFee`) that is currently set to zero but can be enabled unilaterally by Tether at any time without any IC governance action.
2. Future NNS proposals could add ERC20 tokens with fee-on-transfer or rebasing behavior.
3. The minter has no validation that prevents fee-on-transfer tokens from being added as supported ckERC20 tokens.
4. The entry path is fully unprivileged: any Ethereum user can call `depositErc20` on the helper contract.

---

### Recommendation

In both helper contracts, measure the actual received amount by checking the minter's balance before and after `safeTransferFrom`, and emit the delta rather than the user-supplied `amount`:

```solidity
function depositErc20(address erc20Address, uint256 amount, bytes32 principal, bytes32 subaccount) public {
    IERC20 erc20Token = IERC20(erc20Address);
    uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
    emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
}
```

The IC minter canister would then mint and record only the actual received amount, preserving the 1:1 backing invariant.

---

### Proof of Concept

1. USDT's fee mechanism is enabled (e.g., `basisPointsRate = 10`, i.e., 0.1% fee).
2. User calls `depositErc20(USDT_ADDRESS, 1_000_000, principal, subaccount)` on `CkDeposit`.
3. `safeTransferFrom` transfers `1_000_000` USDT from user; minter receives `999_000` (after 0.1% fee).
4. Event emits `amount = 1_000_000`.
5. IC minter scrapes the log, calls `mint(1_000_000 ckUSDT)` on the ckUSDT ledger.
6. IC minter's `erc20_balances` records `+1_000_000`.
7. After 1000 such deposits: minter holds `999_000_000` USDT but has issued `1_000_000_000` ckUSDT.
8. The 1000th withdrawer who tries to redeem `1_000_000` ckUSDT finds the minter's Ethereum transaction fails — the minter's actual USDT balance is exhausted. [6](#0-5) [2](#0-1) [3](#0-2) [4](#0-3)
