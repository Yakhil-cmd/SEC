Audit Report

## Title
Subaccount Collision in ICRC-1 Rosetta Storage Causes Aggregated Balance Undercounting - (`rs/rosetta-api/icrc1/src/common/storage/storage_operations/mod.rs`)

## Summary
The `update_account_balances` function maintains an in-memory cache keyed by `Account` (where Rust equality treats `None ≠ Some([0u8;32])`), but flushes both representations to SQLite using `account.effective_subaccount().as_slice()`, which maps both to the same 32-byte zero array. This produces two DB rows for the same `(principal, subaccount=[0u8;32])` at different `block_idx` values. The `get_aggregated_balance_for_principal_at_block_idx` query then selects only the row with the highest `block_idx` for that subaccount, silently discarding the earlier balance. Any ICRC-1 ledger that legitimately uses both `None` and `Some([0u8;32])` to represent the default subaccount will cause the Rosetta API to undercount the aggregated balance for that principal.

## Finding Description

**Cache flush — both representations written to the same DB key:**

In `update_account_balances`, the cache is `HashMap<Account, BTreeMap<u64, Nat>>`. When block 0 mints to `Account { subaccount: None }` and block 1 mints to `Account { subaccount: Some([0u8;32]) }`, the cache holds two distinct entries. During the flush loop, both are inserted with `account.effective_subaccount().as_slice()` as the subaccount column value: [1](#0-0) 

This produces two rows in `account_balances`:
- `(block_idx=0, principal=P, subaccount=[0;32], amount=6000000)`
- `(block_idx=1, principal=P, subaccount=[0;32], amount=1000000)`

**Aggregation query — returns only the latest row per subaccount, discarding the earlier balance:**

`get_aggregated_balance_for_principal_at_block_idx` uses a correlated subquery to select the row with `MAX(block_idx)` for each distinct `(principal, subaccount)` pair: [2](#0-1) 

For `subaccount=[0;32]`, `MAX(block_idx)=1` is selected, returning `amount=1000000`. The 6,000,000 balance recorded at `block_idx=0` is permanently shadowed. The total returned is 2,000,000 instead of 8,000,000.

**Individual balance query — same normalization, so `get_account_balance_at_block_idx` also conflates both representations:** [3](#0-2) 

**Exposed via the `aggregate_all_subaccounts` metadata flag on the standard `/account/balance` endpoint:** [4](#0-3) 

**Note on the submitted test:** `test_debug_aggregated_balance_sql` contains no `assert!` statements — it only prints. It always passes regardless of the bug's presence. The researcher's specific claim of "7,000,000 actual" is also incorrect; the actual result under the described scenario is 2,000,000 (the `None` balance at `block_idx=0` is shadowed by the `Some([0;32])` balance at `block_idx=1`, and only the non-zero subaccount's 1,000,000 plus the shadowed 1,000,000 are summed). The "developers confirmed" framing is inaccurate — the test was not an asserting regression test. [5](#0-4) 

## Impact Explanation

This is a **Medium** severity finding. The Rosetta API is explicitly in scope under "Financial integrations: ICP/ICRC ledgers, Rosetta API." The concrete harm is that any caller using `aggregate_all_subaccounts: true` on `/account/balance` receives a materially understated balance when the underlying ledger has used both `None` and `Some([0u8;32])` for the same principal's default subaccount. Exchanges, wallets, or DeFi protocols relying on this aggregated figure for withdrawal eligibility or accounting could incorrectly reject valid operations or misreport user holdings. Actual token balances on the ledger are unaffected; only the Rosetta indexer's reported view is wrong.

## Likelihood Explanation

The ICRC-1 standard (section on `Account`) explicitly defines `None` and `Some([0u8;32])` as equivalent representations of the default subaccount. Different client libraries (e.g., agent-rs vs. motoko stdlib) may emit one or the other. A ledger that has processed transactions from multiple clients could legitimately have both representations in its block log. The trigger is an unprivileged, standard API call to `/account/balance` with the `aggregate_all_subaccounts` metadata flag. No special privileges are required. Likelihood is **Medium** — the scenario requires a ledger that has used both representations, which is not universal but is a valid and realistic ICRC-1 deployment.

## Recommendation

Normalize the `Account` key in the in-memory cache to use `effective_subaccount()` before inserting into the `HashMap`, so the cache and DB agree on identity:

```rust
// Before inserting into cache, normalize the account key:
let canonical_account = Account {
    owner: account.owner,
    subaccount: Some(*account.effective_subaccount()),
};
account_balances_cache.entry(canonical_account).or_default().insert(index, new_balance);
```

This ensures that `None` and `Some([0u8;32])` are treated as the same account throughout the entire pipeline (cache lookup, cache write, and DB write), eliminating the split-entry problem. Add an asserting regression test that mints to both representations and asserts `get_aggregated_balance_for_principal_at_block_idx` equals the sum of all minted amounts.

## Proof of Concept

1. Create an in-memory `StorageClient`.
2. Store block 0: `Mint { to: Account { owner: P, subaccount: None }, amount: 6_000_000 }`.
3. Store block 1: `Mint { to: Account { owner: P, subaccount: Some([0u8;32]) }, amount: 1_000_000 }`.
4. Store block 2: `Mint { to: Account { owner: P, subaccount: Some([0,…,1]) }, amount: 1_000_000 }`.
5. Call `update_account_balances()`.
6. Call `get_aggregated_balance_for_principal_at_block_idx(P, 1000)`.
7. **Assert** result equals `Nat::from(8_000_000u64)`.

The assert will fail: the actual result is `Nat::from(2_000_000u64)` because the `None` balance (6,000,000 at `block_idx=0`) is shadowed in the DB by the `Some([0;32])` balance (1,000,000 at `block_idx=1`) under the same `(principal, subaccount=[0;32])` key, and the aggregation query returns only the MAX-`block_idx` row per subaccount. [6](#0-5)
