Audit Report

## Title
Single Canister Can Monopolize the Subnet-Wide Threshold Signature Queue, Denying Service to All Other Canisters - (File: `rs/execution_environment/src/execution_environment.rs`)

## Summary
The `sign_with_threshold` function enforces only a subnet-wide queue-size limit with no per-sender cap. A single canister can fill all available slots for a given threshold key, blocking every other canister on the subnet from obtaining threshold signatures until the attacker's requests are processed. The existing test `test_sign_with_threshold_key_queue_fills_up` directly confirms this behavior.

## Finding Description
In `sign_with_threshold()` at `rs/execution_environment/src/execution_environment.rs` lines 3844–3858, the sole admission control is a comparison of the total in-flight request count against `dynamic_queue_size`:

```rust
if state
    .metadata
    .subnet_call_context_manager
    .sign_with_threshold_contexts_count(&threshold_key)
    >= dynamic_queue_size
{ return Err(...); }
```

`sign_with_threshold_contexts_count` (lines 436–453 of `rs/replicated_state/src/metadata_state/subnet_call_context_manager.rs`) iterates over all pending contexts and filters only by key ID — it does not filter by `context.request.sender`. There is no per-canister quota anywhere in the admission path.

`get_dynamic_signature_queue_size` (lines 5060–5079 of `execution_environment.rs`) computes the limit as `stash_size.clamp(max_queue_size, MAX_PAIRED_PRE_SIGNATURES)`. `MAX_PAIRED_PRE_SIGNATURES` is 100 (`rs/limits/src/lib.rs` line 102); the registry default is 20. A single canister submitting 20–100 requests in rapid succession fills every slot.

The fee-exemption path (lines 3783–3818) skips charging entirely when `source_subnet == nns_subnet_id`. Any canister on the NNS subnet can therefore flood the queue at zero cost. For application-subnet canisters the cost is `dynamic_queue_size × signature_fee` cycles per DoS cycle — a small, repeatable expenditure.

No existing guard (key-enabled check, payment check, or queue-full check) limits how many slots a single sender may occupy.

## Impact Explanation
This is an application/platform-level DoS of the threshold signing service — not raw volumetric DDoS. While the queue is in-flight, every other canister on the subnet that calls `sign_with_ecdsa`, `sign_with_schnorr`, or `vetkd_derive_key` for the targeted key receives a hard rejection. Threshold signing is a critical primitive for chain-key integrations (ckBTC, ckETH, cross-chain bridges). Sustained blocking of this service constitutes a concrete subnet availability impact matching the High bounty tier: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
Any unprivileged canister on an application subnet can execute this attack by paying `dynamic_queue_size × signature_fee` cycles per round (e.g., 20 × 10 B = 200 B cycles for ECDSA at default settings — a trivially affordable amount). The attack is fully deterministic, requires no special privileges, no leaked keys, and no social engineering. The attacker simply re-submits as soon as their batch is processed, maintaining a near-continuous DoS. The NNS-subnet zero-cost variant requires governance approval to deploy a canister there, so it is a constrained but real escalation path.

## Recommendation
Introduce a per-sender cap inside `sign_with_threshold` before enqueuing. Count how many in-flight requests `request.sender` already has for the given key and reject if it exceeds a configured per-canister limit (e.g., a fixed small constant or `max_queue_size / N`). This mirrors the two-tier callback quota already present in the execution environment (`canister_guaranteed_callback_quota` / `subnet_callback_soft_limit` in `rs/config/src/execution_environment.rs` lines 266–276). A corresponding `sign_with_threshold_contexts_count_for_sender(key_id, sender)` helper in `SubnetCallContextManager` would be the minimal surgical change.

## Proof of Concept
The existing integration test `test_sign_with_threshold_key_queue_fills_up` in `rs/execution_environment/tests/threshold_signatures.rs` lines 848–905 already proves the primitive: a single canister fills all 20 slots and the 21st request from the same canister is rejected. To demonstrate cross-canister DoS, extend the test:

1. Build the `StateMachine` with signing disabled (as the test already does).
2. Create canister `A`; submit `max_queue_size` sign requests from `A` — all accepted.
3. Create canister `B`; submit one sign request from `B` — assert it is rejected with `"request queue for key … is full."`.
4. Tick until `A`'s requests are processed; submit from `B` again — assert it now succeeds.
5. Repeat steps 2–4 in a loop to confirm the attacker can maintain the DoS indefinitely.

No mainnet interaction is required; the test runs entirely on a local `StateMachine`.
