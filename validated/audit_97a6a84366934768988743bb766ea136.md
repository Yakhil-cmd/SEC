### Title
`SenderInfoContent` Signed Structure Has No Expiry Field, Enabling Indefinite Reuse of Canister-Attested Authorization Data — (File: `rs/validator/src/ingress_validation.rs`)

---

### Summary

The `SenderInfoContent` structure, which a canister (e.g., Internet Identity) signs to attest user attributes in HTTP requests, contains no expiry or deadline field. The `verify_sender_info_canister_sig` validation function checks only cryptographic correctness — not any time bound. Because canister signatures embed a historical certificate that cannot be invalidated after issuance, a user who obtains a `sender_info` signature can reuse it in arbitrarily many future requests, indefinitely, even if the signing canister intended the authorization to be time-limited or has since changed the user's attributes.

---

### Finding Description

**Root cause — no expiry in the signed structure:**

`SenderInfoContent` is defined as a bare byte slice with no timestamp or deadline field:

```rust
pub struct SenderInfoContent<'a>(pub &'a [u8]);

impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.0);  // raw bytes only, no expiry
    }
}
``` [1](#0-0) 

The decoded `SignedSenderInfo` stored in the request also carries no expiry:

```rust
pub struct SignedSenderInfo {
    pub info: Vec<u8>,   // arbitrary bytes — no deadline
    pub signer: CanisterId,
    pub sig: Vec<u8>,
}
``` [2](#0-1) 

**Root cause — validation never checks time:**

`verify_sender_info_canister_sig` performs three checks: (1) parse the canister-sig public key, (2) match the canister ID, (3) verify the BLS-backed canister signature. There is no step that compares any timestamp in the `info` bytes or the embedded certificate against `current_time`:

```rust
fn verify_sender_info_canister_sig<R: RootOfTrustProvider>(
    sender_info: &SignedSenderInfo,
    sender_pubkey_bytes: &[u8],
    validator: &dyn IngressSigVerifier,
    root_of_trust_provider: &R,
) -> Result<(), RequestValidationError> {
    // 1. parse pubkey
    // 2. match canister ID
    // 3. verify canister sig
    // ← no expiry check anywhere
    Ok(())
}
``` [3](#0-2) 

Compare this with how the same validation layer handles ingress expiry and delegation expiry — both have explicit time-range checks: [4](#0-3) [5](#0-4) 

**Root cause — canister signatures are permanently valid once issued:**

A canister signature embeds a historical BLS certificate at creation time. The `verify` function in `rs/crypto/iccsa/src/api.rs` verifies the embedded certificate — it does not query the canister's current certified data:

```rust
pub fn verify(msg: &[u8], sig: SignatureBytes, pk: PublicKeyBytes,
              root_pubkey: &ThresholdSigPublicKey) -> CryptoResult<()> {
    let (canister_id, seed) = parse_pubkey_bytes(&pk)?;
    let parsed_sig = parse_signature_bytes(&sig)?;
    verify_certified_data(parsed_sig.certificate.as_ref(), ...)?;  // historical cert
    lookup_path_in_tree(&seed, msg, &canister_sig_tree(parsed_sig, &sig)?, ...)?;
    Ok(())
}
``` [6](#0-5) 

Because the certificate is embedded in the signature bytes, the signing canister cannot revoke an already-issued `sender_info` signature. Removing the sig path from the canister's current certified data has no effect on previously issued signatures.

**End-to-end exploit path:**

1. A signing canister (e.g., Internet Identity) issues a `sender_info` canister signature for user U with `info = <credit_limit: 1000 ICP>` at time T.
2. User U attaches this `sender_info` to requests targeting a lending canister; the lending canister reads the `info` via `msg_caller_info_data()` and grants credit.
3. At time T+N the signing canister revokes U's credit limit (e.g., due to default).
4. User U still holds the old `sender_info` bytes and signature. Because the embedded BLS certificate is permanently valid and `verify_sender_info_canister_sig` performs no expiry check, the replica accepts the old signature in every new request.
5. User U constructs fresh requests (different `ingress_expiry`, different nonce) each carrying the same stale `sender_info`. Each request passes full validation.
6. The lending canister receives the stale `info` and continues granting credit.

The user's own envelope signature changes per request (it covers the `MessageId`, which includes `sender_info`), so there is no ingress-replay protection against this reuse. [7](#0-6) 

---

### Impact Explanation

Any canister that relies on `sender_info` to make authorization or financial decisions (credit limits, KYC status, role grants) is permanently exposed to stale attestations. The signing canister has no protocol-level mechanism to invalidate an already-issued `sender_info` signature. A user whose attributes were revoked can continue presenting the old signature indefinitely, bypassing the signing canister's intent. This is a **boundary/API validation bypass** with direct financial impact in any lending, credit, or access-control canister that consumes `sender_info`.

---

### Likelihood Explanation

The attack requires only that a user previously received a valid `sender_info` signature — a normal, unprivileged operation. No key compromise, no governance majority, no threshold attack is needed. The user simply retains the bytes they already received and reuses them. The likelihood is **high** for any deployment that uses `sender_info` for time-sensitive authorization.

---

### Recommendation

1. **Add an `expiry_ns: u64` field to `SenderInfoContent`** (nanoseconds since Unix epoch), making it part of the signed bytes.
2. **Enforce the expiry in `verify_sender_info_canister_sig`**: reject any `sender_info` whose `expiry_ns < current_time`.
3. Optionally, also enforce a maximum future window (analogous to `MAX_INGRESS_TTL`) to prevent pre-issued far-future attestations.

---

### Proof of Concept

```
// Step 1: signing canister issues sender_info at time T
let info_bytes = encode({ credit_limit: 1000, /* no expiry field */ });
let sig = canister.sign(SenderInfoContent(&info_bytes));  // permanent canister sig

// Step 2: at time T+N (after revocation), user reuses the same sig
let request = HttpCanisterUpdate {
    ingress_expiry: current_time + 5_minutes,  // fresh, passes ingress expiry check
    nonce: Some(fresh_nonce),                  // fresh, unique MessageId
    sender_info: Some(RawSignedSenderInfo {
        info: Blob(info_bytes),   // stale — but no expiry enforced
        signer: Blob(canister_id),
        sig: Blob(old_sig),       // permanent canister sig — still verifies
    }),
    ..
};
// verify_sender_info_canister_sig passes: pubkey valid, canister ID matches,
// BLS cert in old_sig still verifies against root key.
// Lending canister reads info_bytes and grants 1000 ICP credit.
```

The replica's `verify_sender_info_canister_sig` at [3](#0-2)  will accept this request because it performs no expiry check. The `SenderInfoContent` at [1](#0-0)  contains no deadline field for the protocol to enforce.

```rust
fn verify_sender_info_canister_sig<R: RootOfTrustProvider>(
    sender_info: &SignedSenderInfo,
    sender_pubkey_bytes: &[u8],
    validator: &dyn IngressSigVerifier,
    root_of_trust_provider: &R,
) -> Result<(), RequestValidationError>
where
    R::Error: std::error::Error,
{
    // Parse the envelope-level sender_pubkey DER to extract the raw
    // public key bytes and verify it's a valid canister signature public key.
    let pk_bytes = public_key_bytes_from_der(sender_pubkey_bytes).map_err(|e| {
        InvalidSenderInfo(format!(
            "sender_pubkey is not a valid canister signature public key: {e}"
        ))
    })?;

    // Extract the canister ID from the parsed public key and verify
    // it matches the declared signer.
    let parsed_pk = ic_crypto_iccsa::types::PublicKey::try_from(&pk_bytes)
        .map_err(|e| InvalidSenderInfo(format!("invalid canister sig public key: {e:?}")))?;
    let pubkey_canister_id = parsed_pk.signing_canister_id();
    if pubkey_canister_id != sender_info.signer {
        return Err(InvalidSenderInfo(format!(
            "signer {} does not match canister ID {} in sender_pubkey",
            sender_info.signer, pubkey_canister_id
        )));
    }

    // Construct the UserPublicKey for verification.
    let public_key = UserPublicKey {
        key: pk_bytes.0,
        algorithm_id: AlgorithmId::IcCanisterSignature,
    };

    // Construct the signable content (domain = "ic-sender-info")
    let sender_info_content = SenderInfoContent(&sender_info.info);
    let canister_sig = CanisterSigOf::from(CanisterSig(sender_info.sig.clone()));

    verify_canister_sig_with_fallback!(
        validator,
        &canister_sig,
        &sender_info_content,
        &public_key,
        root_of_trust_provider,
        |e| InvalidSenderInfo(format!("signature verification failed: {e}")),
        |e: <R as RootOfTrustProvider>::Error| InvalidSenderInfo(format!(
            "failed to get root of trust: {e}"
        ))
    );
    Ok(())
}

fn validate_ingress_expiry<C: HttpRequestContent>(
    request: &HttpRequest<C>,
    current_time: Time,
) -> Result<(), RequestValidationError> {
    let ingress_expiry = request.ingress_expiry();
    let provided_expiry = Time::from_nanos_since_unix_epoch(ingress_expiry);
    let min_allowed_expiry = current_time;
    // We need to account for time drift and be more forgiving at rejecting ingress
    // messages due to their expiry being too far in the future.
    // If this logic changes, then the migration canister in `//rs/migration_canister`
    // must be updated, too.
    let max_expiry_diff = MAX_INGRESS_TTL
        .checked_add(PERMITTED_DRIFT_AT_VALIDATOR)
        .ok_or_else(|| {
            InvalidRequestExpiry(format!(
                "Addition of MAX_INGRESS_TTL {MAX_INGRESS_TTL:?} with \
                PERMITTED_DRIFT_AT_VALIDATOR {PERMITTED_DRIFT_AT_VALIDATOR:?} overflows",
            ))
        })?;
    let max_allowed_expiry = min_allowed_expiry
        .checked_add(max_expiry_diff)
        .ok_or_else(|| {
            InvalidRequestExpiry(format!(
                "Addition of min_allowed_expiry {min_allowed_expiry:?} \
                with max_expiry_diff {max_expiry_diff:?} overflows",
            ))
        })?;
    if !(min_allowed_expiry <= provided_expiry && provided_expiry <= max_allowed_expiry) {
        let msg = format!(
            "Specified ingress_expiry not within expected range: \
             Minimum allowed expiry: {min_allowed_expiry}, \
             Maximum allowed expiry: {max_allowed_expiry}, \
             Provided expiry:        {provided_expiry}"
        );
        return Err(InvalidRequestExpiry(msg));
    }
    Ok(())
}

pub fn verify(
    msg: &[u8],
    sig: SignatureBytes,
    pk: PublicKeyBytes,
    root_pubkey: &ThresholdSigPublicKey,
) -> CryptoResult<()> {
    let (canister_id, seed) = parse_pubkey_bytes(&pk)?;
    let parsed_sig = parse_signature_bytes(&sig)?;
    verify_certified_data(
        parsed_sig.certificate.as_ref(),
        &canister_id,
        root_pubkey,
        &parsed_sig.tree.digest(),
        &sig,
        &pk,
    )?;
    let canister_sig_tree = canister_sig_tree(parsed_sig, &sig)?;
    lookup_path_in_tree(&seed, msg, &canister_sig_tree, &pk, &sig)?;
    Ok(())
}
