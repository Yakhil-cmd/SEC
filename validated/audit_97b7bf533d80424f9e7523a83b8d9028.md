### Title
Asymmetric `delta_gas` Adjustment Causes User Overpayment When Native Resource Consumption Is Less Than EVM Gas Consumption - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the `delta_gas` adjustment is only applied when `delta_gas > 0` (native resource consumption exceeds EVM gas consumption). When `delta_gas < 0` (EVM gas consumption exceeds native resource consumption), `gas_used` is **not** reduced, and the user is charged for more gas than the actual cost of the transaction warrants. This is a resource accounting bug analogous to the reported M-06 pattern: a conditional branch that should update an accounting variable is missing, causing the variable to retain an incorrect value that is then used unconditionally in downstream fee/refund calculations.

---

### Finding Description

In `compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`, the double-resource accounting adjustment reads:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // In this case, the native resource consumption is more than the
    // gas consumption accounted for. Consume extra gas.
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

The `TODO` comment on line 80 explicitly acknowledges the missing symmetric case. When `delta_gas < 0`, meaning the EVM gas consumed is *greater* than what the native resource consumption implies, `gas_used` is not reduced. The user is therefore charged for more gas than the actual proving cost of the transaction requires.

The `gas_used` value computed here flows directly into:
1. `refund_and_commit_fee` — the user's token refund is `gas_price * (tx_gas_limit - gas_used)`, so a higher `gas_used` means a smaller refund.
2. `token_to_pay_operator` — the operator receives `gas_used * gas_price_for_operator`, so a higher `gas_used` means the operator receives more than the actual cost.

The design intent, documented in `docs/double_resource_accounting.md`, is:

> `deltaGas := (nativeUsed / nativePerGas) - gasUsed`
> If `deltaGas > 0`, we add it to `gasUsed`...

The documentation only describes the positive case. The negative case — where EVM gas consumption exceeds native resource consumption — is silently dropped, leaving `gas_used` inflated relative to the actual proving cost.

---

### Impact Explanation

**Resource accounting bug / user funds loss path.**

A user who submits a transaction where EVM gas consumption is high but native resource consumption is low (e.g., a transaction that does many cheap EVM operations that are inexpensive to prove) will be charged more gas than the actual cost of proving the transaction. The excess fee flows to the operator. The user receives a smaller refund than they are entitled to. This is a direct, quantifiable loss of user funds proportional to `|delta_gas| * gas_price`.

The magnitude depends on the ratio `native_per_gas` and the specific workload. For transactions where EVM gas consumption significantly exceeds `native_used / native_per_gas`, the user overpays by `|delta_gas| * gas_price` tokens per transaction.

---

### Likelihood Explanation

This is reachable by any unprivileged L2 transaction sender. The condition `delta_gas < 0` occurs whenever a transaction's EVM gas consumption (ergs-based) exceeds the native resource consumption scaled by `native_per_gas`. This is a normal operating condition — for example, transactions with large EVM gas refunds (SSTORE clearing) reduce `gas_used` after the EVM refund cap, but native resource consumption is not similarly reduced. The `native_per_gas` ratio is set by the operator and is a normal runtime parameter. No special access or privileged role is required to trigger this path.

---

### Recommendation

Apply the symmetric adjustment: when `delta_gas < 0`, reduce `gas_used` by `|delta_gas|`, subject to the `minimal_gas_used` floor. The existing `TODO` comment at line 80 already acknowledges this gap. The fix should be:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native resource consumption is less than EVM gas consumption.
    // Return the difference to gas_used (reduce gas charged).
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction);
    // Re-apply the minimal_gas_used floor
    gas_used = core::cmp::max(gas_used, minimal_gas_used);
}
```

### Proof of Concept

**Setup:**
- `gas_limit = 100_000`
- `native_per_gas = 50` (i.e., `gasPrice / nativePrice = 50`)
- After execution, `gas_used` (from ergs) = `80_000`
- `evm_refund` (SSTORE clearing) = `10_000` → `gas_used = 70_000`
- `minimal_gas_used = 21_000` → `gas_used = max(70_000, 21_000) = 70_000`
- `native_used = 2_000_000` (native units consumed)
- `native_used / native_per_gas = 2_000_000 / 50 = 40_000`
- `delta_gas = 40_000 - 70_000 = -30_000`

**Current behavior:** `delta_gas < 0`, so the `if` branch is not taken. `gas_used` remains `70_000`. The user is charged for `70_000` gas and refunded `30_000 * gas_price` tokens.

**Correct behavior:** `gas_used` should be reduced to `max(40_000, 21_000) = 40_000`. The user should be charged for `40_000` gas and refunded `60_000 * gas_price` tokens.

**User loss per transaction:** `30_000 * gas_price` tokens, paid to the operator in excess of the actual proving cost.
