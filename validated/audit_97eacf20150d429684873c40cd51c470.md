### Title
Unchecked Precompile Call Return Value in `EvmErc20.withdrawToNear()` / `withdrawToEthereum()` Causes Permanent Token Loss When `error_refund` Feature Is Disabled — (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` burn user ERC-20 tokens **before** calling the exit precompile, and the inline-assembly `call()` return value is never checked. When the `error_refund` compile-time feature is disabled (the default production build), any failure of the downstream NEAR-side `ft_transfer` promise leaves the burned tokens permanently unrecoverable — a direct analog to the `transfer()` 2300-gas revert pattern where a failed fund-transfer silently destroys user assets.

---

### Finding Description

In both `EvmErc20.sol` and `EvmErc20V2.sol`, the `withdrawToNear()` and `withdrawToEthereum()` functions follow this pattern:

```solidity
// EvmErc20.sol – withdrawToNear (lines 53-63)
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here, irreversibly

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — a failed precompile call is silently swallowed
    }
}
``` [1](#0-0) [2](#0-1) 

The same pattern appears in `withdrawToEthereum()` for both contracts. [3](#0-2) [4](#0-3) 

**Two compounding failure modes exist:**

**1. Unchecked precompile call return value (EVM layer)**

In standard EVM semantics, a `call()` that fails returns `0` in `res` but does **not** revert the caller. The `_burn()` has already executed and is not rolled back. If the precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` (the `ExitToNear` precompile) returns an error for any reason, the tokens are burned with no corresponding NEAR-side transfer.

**2. No refund path when `error_refund` feature is disabled (NEAR promise layer)**

Even when the precompile call itself succeeds, it schedules a NEAR cross-contract call (`ft_transfer` or `ft_transfer_call`). If that promise fails on the NEAR side, the refund path is gated behind the `error_refund` compile-time feature:

```rust
// engine-precompiles/src/native.rs – lines 449-453
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,   // ← no refund args encoded when feature is off
    transfer_near: transfer_near_args,
};
``` [5](#0-4) 

When `refund` is `None`, the `ExitToNearPrecompileCallbackArgs` struct equals its `Default`, so no callback is scheduled at all:

```rust
// lines 470-471
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback, no refund possible
``` [6](#0-5) 

The `error_refund` feature is **not** enabled by default in either `engine/Cargo.toml` or `engine-precompiles/Cargo.toml`: [7](#0-6) [8](#0-7) 

The codebase itself acknowledges this loss in the test suite:

```rust
// engine-tests/src/tests/erc20_connector.rs – lines 658-660
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [9](#0-8) 

---

### Impact Explanation

When a user calls `withdrawToNear()` or `withdrawToEthereum()` on an `EvmErc20` contract and the downstream NEAR promise fails (or the precompile call itself fails silently), the user's ERC-20 tokens are permanently burned with no corresponding NEP-141 tokens delivered. The tokens are irrecoverably destroyed — **permanent freezing/loss of user funds**.

This matches the Critical impact tier: **Permanent freezing of funds**.

---

### Likelihood Explanation

The NEAR-side `ft_transfer` promise fails in realistic, user-reachable scenarios:

- The recipient NEAR account is not registered with the NEP-141 contract (storage deposit not paid). This is a common operational condition — the test `test_exit_to_near_refund` explicitly demonstrates this failure mode using `"unregistered.near"` as the recipient.
- The Aurora engine account has insufficient NEP-141 balance (e.g., due to accounting drift).
- Any transient NEAR runtime error during promise execution.

Any EVM user who calls `withdrawToNear()` with an unregistered NEAR recipient triggers this path. No special privileges are required.

---

### Recommendation

1. **Check the precompile call return value** in `EvmErc20.withdrawToNear()` and `withdrawToEthereum()`: if `res == 0`, revert the transaction so the `_burn()` is rolled back.

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

2. **Enable `error_refund` by default** in the production build, or make the refund callback unconditional so that a failed NEAR promise always triggers an EVM-side refund mint.

---

### Proof of Concept

1. Deploy `EvmErc20` (without `error_refund` feature) and bridge NEP-141 tokens to it.
2. Call `withdrawToNear("unregistered.near", amount)` from an EVM address holding tokens.
3. The `_burn()` executes, reducing the caller's ERC-20 balance by `amount`.
4. The precompile schedules `ft_transfer` to `"unregistered.near"` on the NEP-141 contract.
5. The NEAR promise fails because `"unregistered.near"` has no storage deposit.
6. No callback is scheduled (since `error_refund` is off, `refund: None`).
7. The caller's ERC-20 balance is permanently reduced; no NEP-141 tokens are received.

The test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` (lines 623–665) confirms this exact outcome: without `error_refund`, `balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into()` — the exited tokens are gone. [10](#0-9)
