### Title
Unchecked `transferFrom` Boolean Return Value in XCC Precompile Allows Silent Payment Bypass - (File: engine-precompiles/src/xcc.rs)

### Summary
The Cross-Contract Call (XCC) precompile calls `transferFrom` on the wNEAR ERC-20 token to collect payment from the caller before scheduling a NEAR cross-contract call. The EVM sub-call's exit reason is checked, but the ABI-encoded boolean return value is silently discarded. A wNEAR token that returns `false` without reverting would allow the precompile to proceed as if payment succeeded, draining the engine's NEAR balance.

### Finding Description
In `engine-precompiles/src/xcc.rs`, `run_with_handle` issues an internal EVM call to the wNEAR ERC-20 contract's `transferFrom` selector to move `required_near` wNEAR from the caller to the engine's implicit address:

```rust
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
match exit_reason {
    // Transfer successful, nothing to do
    aurora_evm::ExitReason::Succeed(_) => (),
    ...
}
``` [1](#0-0) 

When `exit_reason` is `Succeed`, the code does nothing — `return_value` (which carries the ABI-encoded `bool` mandated by ERC-20) is never decoded or validated. The ERC-20 standard explicitly permits a token to signal failure by returning `false` without reverting. The `return_value` variable is bound but never inspected in the success branch. [2](#0-1) 

The `transfer_from_args` helper correctly encodes the call with the `transferFrom` selector and expects a `bool` output (as confirmed by the unit test), yet the caller never reads that output: [3](#0-2) 

### Impact Explanation
If the wNEAR ERC-20 token returns `false` (without reverting) on a failed `transferFrom`, the XCC precompile continues execution, emits the promise log, and the Aurora engine schedules the NEAR cross-contract call — attaching real NEAR from the engine's own balance — without having received the corresponding wNEAR payment from the caller. Repeated exploitation drains the engine's NEAR reserve, leading to insolvency or inability to fund future cross-contract calls for all users.

**Impact class:** Critical — insolvency / direct theft of engine-held NEAR funds.

### Likelihood Explanation
The wNEAR token address is read from Aurora's own storage (`state::get_wnear_address`) and is the canonical bridged wNEAR deployed by the Aurora team. The current OpenZeppelin-based ERC-20 implementation reverts on insufficient allowance/balance rather than returning `false`, so the bug is not immediately exploitable against the standard deployment. However:

- The ERC-20 standard does not prohibit returning `false`, and any future upgrade or alternative wNEAR deployment that follows the non-reverting pattern would make this immediately exploitable.
- Any unprivileged EVM user who calls the XCC precompile is the attacker-controlled entry point — no special role is required.

**Likelihood:** Medium (not exploitable today with the standard wNEAR, but one token-implementation change away from critical exploitation).

### Recommendation
After the `Succeed` branch, decode `return_value` as a `bool` and treat `false` as a revert:

```rust
aurora_evm::ExitReason::Succeed(_) => {
    // ERC-20 transferFrom returns a bool; false means failure
    let transfer_ok = return_value
        .last()
        .map_or(false, |&b| b != 0);
    if !transfer_ok {
        return Err(revert_with_message("ERR_WNEAR_TRANSFER_FAILED"));
    }
}
```

This mirrors the defensive pattern already used in the `Revert`/`Error`/`Fatal` arms and ensures the precompile cannot proceed without a confirmed token transfer.

### Proof of Concept

1. Deploy a wNEAR-compatible ERC-20 contract whose `transferFrom` always returns `false` (without reverting).
2. Register this contract as the wNEAR address in Aurora's XCC storage.
3. Call the XCC precompile (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`) with a valid `CrossContractCallArgs::Eager` payload and any `attached_near > 0`.
4. Observe: `exit_reason` is `Succeed`, `return_value` is `abi.encode(false)`, but the precompile emits the promise log and the engine schedules the NEAR cross-contract call — no wNEAR was actually transferred from the caller.
5. The engine's NEAR balance is consumed without compensation, repeatable until the engine is insolvent. [4](#0-3)
