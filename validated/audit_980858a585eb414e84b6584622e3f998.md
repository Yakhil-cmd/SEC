### Title
Permanent ETH Freeze on Failed `ExitToNear` Promise Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile immediately deducts ETH from the caller's EVM balance (transferring it to the precompile address) before the NEAR-side `ft_transfer` promise resolves. When the `error_refund` compile-time feature is **not** enabled, a failed promise leaves the ETH permanently frozen at the precompile address with no recovery path. This is a connector/bridge accounting bug leading to permanent fund freeze.

---

### Finding Description

When a user calls the `ExitToNear` precompile with ETH (base token, flag `0x00`), the EVM immediately moves `context.apparent_value` from the caller's balance to `exit_to_near::ADDRESS`. The precompile then schedules a NEAR cross-contract call (`ft_transfer` or `ft_transfer_call`) on the eth-connector.

The refund callback is only populated when the `error_refund` feature is compiled in:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,   // ← no refund args when feature is absent
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When the promise is dispatched without a callback (because `callback_args == default()`), or when the callback fires but `args.refund` is `None`, the failure branch in `exit_to_near_precompile_callback` is never reached:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent no-op on failure; ETH stays at precompile address
};
``` [2](#0-1) 

The `refund_on_error` function for ETH exits transfers the frozen ETH back from `exit_to_near::ADDRESS` to the user — but this is only reachable when `refund` is `Some`: [3](#0-2) 

The `error_refund` feature is **not** a default feature in the engine crate; it must be explicitly opted into at build time: [4](#0-3) 

The test suite explicitly acknowledges the fund loss when the feature is absent:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [5](#0-4) 

---

### Impact Explanation

Any ETH sent through `ExitToNear` (base token path) where the NEAR-side `ft_transfer` promise fails is **permanently frozen** at `exit_to_near::ADDRESS` (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`). There is no admin escape hatch, no secondary withdrawal path, and no on-chain mechanism to recover the ETH once it is stranded there. This constitutes **Critical: Permanent freezing of funds**.

---

### Likelihood Explanation

The `ft_transfer` promise can fail for several reasons reachable by an ordinary user:

- Sending to a NEAR account that is not registered with the NEP-141 eth-connector (no storage deposit).
- Sending to a non-existent NEAR account ID.
- Insufficient NEAR gas attached to the promise (misconfigured `costs::FT_TRANSFER_GAS`).

The `ExitToNear` precompile is a core, publicly reachable interface callable by any EVM user via a standard ETH-value transaction to `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`. [6](#0-5) 

---

### Recommendation

1. **Enable `error_refund` unconditionally** in the production build, or promote it from an optional feature to a default feature in `engine/Cargo.toml` and `engine-precompiles/Cargo.toml`.
2. Alternatively, always attach the callback with a non-`None` refund argument regardless of the feature flag, so that a failed promise always triggers `refund_on_error`.
3. For `ExitToEthereum`, audit whether the `withdraw` promise can similarly fail without a refund path, as that precompile has no callback mechanism at all. [7](#0-6) 

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Fund EVM address `Alice` with 1 ETH.
3. Alice calls the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with `value = 1 ETH` and recipient `"unregistered.near"` (an account with no NEP-141 storage deposit).
4. The EVM immediately deducts 1 ETH from Alice and credits `exit_to_near::ADDRESS`.
5. The NEAR `ft_transfer` promise fires and fails (unregistered recipient).
6. The callback fires with `args.refund == None`; the `else { None }` branch executes — no refund.
7. Alice's EVM balance is permanently 0; `exit_to_near::ADDRESS` holds 1 ETH with no recovery path.
