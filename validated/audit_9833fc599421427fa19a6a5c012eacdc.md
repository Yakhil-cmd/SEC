Audit Report

## Title
Missing Context Validation in `push_response` `SendTransactionResponse`/`SendTransactionReject` Arms Allows Malicious Block Proposer to Prematurely Consume Pending Bitcoin Transaction Contexts — (`rs/replicated_state/src/bitcoin.rs`)

## Summary
The `SendTransactionResponse` and `SendTransactionReject` arms of `push_response` unconditionally enqueue a `ConsensusResponse` without verifying the `callback_id` exists in `bitcoin_send_transaction_internal_contexts`. Combined with a `validate_self_validating_payload_impl` that performs no callback_id-to-state cross-check, a single malicious block proposer can finalize a block containing a fabricated response targeting a live pending context, causing the bitcoin canister to receive a premature acknowledgment and permanently orphaning the real adapter response when it arrives.

## Finding Description

**Root cause — `push_response` in `rs/replicated_state/src/bitcoin.rs`:**

The `GetSuccessorsResponse` arm validates the `callback_id` against `bitcoin_get_successors_contexts` and returns `StateError::BitcoinNonMatchingResponse` on mismatch: [1](#0-0) 

The `SendTransactionResponse` arm performs no such lookup and unconditionally pushes to `consensus_queue`: [2](#0-1) 

The `SendTransactionReject` arm has the same omission: [3](#0-2) 

**Validation bypass — `validate_self_validating_payload_impl`:**

The function used by all honest nodes when notarizing a block proposal does not check `callback_id` values against replicated state at all. It only checks whether the payload is empty or returns the byte size: [4](#0-3) 

This is called by both `validate_self_validating_payload` and `validate_payload`: [5](#0-4) [6](#0-5) 

**Exploit flow:**

1. Malicious block proposer observes `next_callback_id` from certified state and identifies a live entry in `bitcoin_send_transaction_internal_contexts`.
2. Proposer builds a `SelfValidatingPayload` containing a `BitcoinAdapterResponse { response: SendTransactionResponse, callback_id: <target> }`.
3. Honest nodes call `validate_self_validating_payload_impl` — it passes (non-empty, within size limit).
4. Block is notarized and finalized. In `demux.rs`, `push_response_bitcoin` is called: [7](#0-6) 
5. `push_response` returns `Ok(())` and the fabricated `ConsensusResponse` enters `consensus_queue`.
6. The scheduler drains `consensus_queue` and calls `execute_subnet_message` with the fabricated response: [8](#0-7) 
7. `execute_subnet_message` calls `retrieve_context(callback_id)`, which removes the live context from `bitcoin_send_transaction_internal_contexts` and delivers the fake `EmptyBlob` reply to the bitcoin canister: [9](#0-8) 
8. The bitcoin canister treats the `send_transaction` as acknowledged and advances its state machine.
9. When the legitimate adapter response arrives, `push_response` again returns `Ok(())` (no validation), but `retrieve_context` now returns `None` (context already consumed), so the response is silently discarded as `ExecuteSubnetMessageResultType::Finished`: [10](#0-9) 
10. The `fatal!` assertion is satisfied because `consensus_queue` is fully drained: [11](#0-10) 

**Why existing checks fail:**

`bitcoin_requests_iter` (used during payload building) correctly iterates only over known contexts: [12](#0-11) 

But this constraint is only enforced on the honest proposer's build path, not during validation. A malicious proposer bypasses it entirely by constructing the payload directly.

## Impact Explanation

This is a **High** severity finding. A malicious block proposer on the Bitcoin subnet can prematurely consume a live `bitcoin_send_transaction_internal_contexts` entry by injecting a fabricated `SendTransactionResponse` with a targeted `callback_id`. The bitcoin canister receives a fake acknowledgment and considers the transaction submitted. The real adapter response is permanently orphaned. If the Bitcoin transaction was not yet broadcast to the Bitcoin network at the time of the attack, the corresponding ckBTC withdrawal may be permanently stuck or lost, constituting concrete user and protocol harm to Chain Fusion / ckBTC. This matches the allowed impact: **"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."**

## Likelihood Explanation

- Any node on the Bitcoin subnet can become block proposer through normal round-robin/rank-based selection — no collusion required.
- A single Byzantine node below the fault threshold suffices.
- The `next_callback_id` counter is part of the certified state and is observable, making targeted attacks against live contexts feasible.
- The fabricated payload passes `validate_self_validating_payload_impl` on all honest nodes because it performs no callback_id-to-state cross-check.
- The attack is deterministic and reproducible: all honest replicas process the fabricated payload identically, producing the same state hash.

## Recommendation

Add the same context lookup guard to the `SendTransactionResponse` and `SendTransactionReject` arms as exists in `GetSuccessorsResponse`:

```rust
BitcoinAdapterResponseWrapper::SendTransactionResponse(_) => {
    let callback_id = CallbackId::from(response.callback_id);
    state
        .metadata
        .subnet_call_context_manager
        .bitcoin_send_transaction_internal_contexts
        .get(&callback_id)
        .ok_or_else(|| StateError::BitcoinNonMatchingResponse {
            callback_id: callback_id.get(),
        })?;
    let payload = Payload::Data(EmptyBlob.encode());
    state.consensus_queue.push(ConsensusResponse::new(callback_id, payload));
    Ok(())
}
```

Apply the same fix to `SendTransactionReject`. Additionally, `validate_self_validating_payload_impl` should validate that every `callback_id` in the payload corresponds to a pending context in the state at `validation_context.certified_height`, mirroring the constraint already enforced in `bitcoin_requests_iter` during payload building.
