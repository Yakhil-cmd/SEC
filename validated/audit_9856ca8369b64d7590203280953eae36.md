### Title
Missing BLS12-381 Precompile Hook Registration in ZK Path — (`basic_bootloader/src/bootloader/block_flow/zk/post_init_op.rs`)

### Summary
When the `eip-2537` feature is enabled, BLS12-381 precompile addresses (`0x0b`–`0x11`) are included in `PRECOMPILE_ADDRESSES_LOWS` — the source of truth used by `is_precompile()` for gas accounting — but are never registered as call hooks in the ZK (EraVM) execution path. Any EVM call to these addresses in the ZK path silently returns empty data instead of executing the precompile, while still being charged warm-access gas costs.

### Finding Description

`PRECOMPILE_ADDRESSES_LOWS` in `evm_interpreter/src/precompile_addresses.rs` conditionally includes all seven BLS12-381 addresses when `eip-2537` is enabled: [1](#0-0) 

The `is_precompile()` helper consults this array: [2](#0-1) 

The account cache uses `is_precompile()` to skip cold-access charges: [3](#0-2) 

The **Ethereum** post-init path registers all seven BLS12-381 hooks: [4](#0-3) 

The **ZK** post-init path only calls `add_precompiles` (which covers `0x01`–`0x08` and optional `0x09`/`0x0a`/`0x0100`) and never calls `initialize_eip_2537`: [5](#0-4) 

`HooksStorage::try_intercept` returns `Ok(None)` when no hook is registered for an address, causing the caller to treat the call as a call to an empty account: [6](#0-5) 

The `add_precompile` sanity check enforces that only addresses in `PRECOMPILE_ADDRESSES_LOWS` can be registered, but the inverse is not enforced — an address can be in `PRECOMPILE_ADDRESSES_LOWS` with no hook: [7](#0-6) 

### Impact Explanation

When `eip-2537` is enabled and an EraVM transaction triggers EVM execution that calls any BLS12-381 precompile address (`0x0b`–`0x11`):

1. **Wrong return data**: The call returns empty bytes instead of the precompile result. Any contract relying on BLS12-381 for signature verification or pairing checks will receive a silent wrong answer.
2. **Gas accounting mismatch**: `is_precompile()` returns `true`, so warm-access cost (100 gas) is charged instead of cold (2600 gas), creating a divergence between the ZK path and the Ethereum path.
3. **Forward/proving divergence**: The Ethereum path executes the precompile correctly; the ZK path does not. This is a state-transition inconsistency between the two execution modes.

**Impact: Medium**

### Likelihood Explanation

- The `eip-2537` feature must be enabled at compile time. The full implementation exists in the codebase (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/`), indicating it is intended for production use.
- The ZK path registers standard EVM precompiles (`0x01`–`0x08`) via `add_precompiles`, confirming that EVM precompile calls are expected to work in EraVM transactions.
- Any EraVM contract that calls an EVM contract using BLS12-381 (e.g., for ZK proof verification, BLS signature aggregation) would trigger this path.

**Likelihood: Medium**

### Recommendation

Add `initialize_eip_2537` (and `initialize_eip_152`) to the ZK post-init path, mirroring the Ethereum path:

```rust
// basic_bootloader/src/bootloader/block_flow/zk/post_init_op.rs
fn post_init_op<Config: BasicBootloaderExecutionConfig>(
    _system: &mut System<S>,
    system_functions: &mut HooksStorage<S, <S as SystemTypes>::Allocator>,
) -> Result<(), InternalError> {
    system_hooks::add_precompiles(system_functions)?;
+   hooks::eip_2537::initialize_eip_2537(system_functions)?;
+   hooks::eip_152::initialize_eip_152(system_functions)?;
    // ... existing system hooks ...
    Ok(())
}
```

Alternatively, enforce at compile time that every address in `PRECOMPILE_ADDRESSES_LOWS` has a corresponding registered hook in every execution path.

### Proof of Concept

1. Enable the `eip-2537` Cargo feature.
2. Submit an EraVM (ZK-path) transaction that calls an EVM contract.
3. The EVM contract calls `0x000000000000000000000000000000000000000b` (BLS12_G1ADD) with valid G1 addition input.
4. **Expected**: The precompile executes and returns the G1 addition result (128 bytes).
5. **Actual**: `try_intercept` finds no hook for `0x0b` in the ZK path, returns `Ok(None)`, the call is treated as an empty account, and the contract receives empty return data.
6. Any downstream logic that checks the return value (e.g., BLS signature verification) will silently produce an incorrect result.

/// Helper to check if an address is an ethereum precompile
#[inline(always)]
pub fn is_precompile(address: &B160) -> bool {
    let limbs = address.as_limbs();
    if limbs[1] != 0u64 || limbs[2] != 0u64 {
        return false;
    }
    let Ok(low) = limbs[0].try_into() else {
        return false;
    };
    precompile_addresses::PRECOMPILE_ADDRESSES_LOWS.contains(&low)
}

    pub fn try_intercept<'a>(
        &mut self,
        address_low: u16,
        request: ExternalCallRequest<S>,
        caller_ee: u8,
        system: &mut System<S>,
        return_memory: &'a mut [MaybeUninit<u8>],
    ) -> Result<(Option<CompletedExecution<'a, S>>, &'a mut [MaybeUninit<u8>]), SystemError> {
        let Some(hook) = self.call_hooks.get(&address_low) else {
            return Ok((None, return_memory));
        };
        let (res, remaining_memory) = hook.0(request, caller_ee, system, return_memory)?;

        Ok((Some(res), remaining_memory))
    }
