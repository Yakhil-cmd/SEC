### Title
Incomplete MPT Insertion Logic Causes Runtime Panic via Unimplemented `PathDiverged`/`EmptyBranchTaken` Arms - (File: basic_system/src/system_implementation/ethereum_storage_model/mpt/updates/mod.rs)

---

### Summary

The `find_insertion_strategy` function in the Ethereum MPT implementation contains two `todo!()` macros for the `AppendPath::PathDiverged` and `AppendPath::EmptyBranchTaken` match arms. In Rust, `todo!()` unconditionally panics at runtime with "not yet implemented". These two arms represent legitimate, reachable MPT tree-traversal outcomes during proof-assisted insertion. Any Ethereum-STF transaction that writes to a new storage slot whose insertion path traverses unreferenced (oracle-provided) proof nodes and encounters either a diverging path or an empty branch will cause the system to panic, halting block processing and proof generation.

---

### Finding Description

`find_insertion_strategy` is the core routing function for MPT leaf insertion. It runs two sequential loops:

1. **First loop** — descends through nodes already resident in the in-memory cache via `descend_through_existing_nodes`. If it encounters an `UnreferencedPathEncountered` result (a node whose subtree has not yet been loaded), it breaks into the second loop.

2. **Second loop** — descends through oracle-provided proof nodes via `descend_through_proof`. This loop has five match arms. Three are implemented (`BranchTaken`, `LeafReached`, `BranchReached`, `Follow`). Two are not:

```rust
AppendPath::PathDiverged { allocated_node } => {
    self.link_if_needed(current_node, parent_branch_index, allocated_node)?;
    todo!();   // ← panics
}
AppendPath::EmptyBranchTaken { allocated_node, .. } => {
    self.link_if_needed(current_node, parent_branch_index, allocated_node)?;
    todo!();   // ← panics
}
```

`PathDiverged` is returned when the key being inserted shares a common prefix with an existing key in the proof subtree but diverges before reaching a leaf — the standard case for inserting a new key that collides on a prefix with an existing one. `EmptyBranchTaken` is returned when the proof traversal reaches a branch node whose slot at the relevant nibble index is empty — the standard case for inserting a new key into a branch that has a free slot. Both are routine MPT insertion scenarios.

The call chain that reaches this code is:


`persist_changes` is invoked during Ethereum-STF block finalization whenever storage diffs must be committed to the Ethereum MPT state root.

---

### Impact Explanation

When either incomplete arm is reached, the Rust runtime panics with `"not yet implemented"`. In the ZKsync OS execution context this means:

- **Forward execution halts**: the block cannot be finalized; no state root is produced.
- **Proof generation fails**: the prover never receives a valid post-state to prove.
- **Liveness loss**: any block whose storage diff triggers this path is permanently unprocessable until the code is patched.

Because the panic occurs inside the state-persistence step (after transaction execution has already mutated in-memory state), the failure is not gracefully recoverable — it is not a transaction revert but a hard crash of the block-processing loop.

---

### Likelihood Explanation

`PathDiverged` is triggered whenever a new storage key is inserted whose hex-nibble path shares a prefix with an existing key in the proof subtree but diverges before the leaf. This is the overwhelmingly common case for any contract that writes to more than one storage slot, since distinct slot keys will almost always share some nibble prefix in the keccak-hashed MPT path. `EmptyBranchTaken` is triggered when the insertion path reaches a branch node with a free nibble slot — equally common for sparse storage tries. An unprivileged user needs only to submit an Ethereum transaction that performs a `SSTORE` to a new slot in a contract whose storage trie already has at least one entry, which is trivially achievable.

---

### Recommendation

Implement the missing insertion logic for both arms in the second loop of `find_insertion_strategy`. For `PathDiverged`, the correct behavior is to split the existing extension/leaf at the divergence point and create a new branch node, analogous to the `ValueInsertionStrategy::Split` path already handled in the first loop. For `EmptyBranchTaken`, the correct behavior is to attach a new leaf to the empty branch slot, analogous to `ValueInsertionStrategy::MakeLeafAttachedToBranch`. Until these are implemented, the Ethereum-STF storage model cannot safely handle insertions into non-trivial tries.

---

### Proof of Concept

1. Configure ZKsync OS to run the Ethereum STF.
2. Deploy a contract `C` and execute a transaction `T1` that writes `SSTORE(slot_A, val_1)` — this populates the storage trie with one leaf.
3. Submit a second transaction `T2` that writes `SSTORE(slot_B, val_2)` where `keccak256(slot_B)` shares a nibble prefix with `keccak256(slot_A)` but diverges before the leaf (trivially satisfied by any two distinct slots in a real contract).
4. During `persist_changes` for the block containing `T2`, `find_insertion_strategy` enters the second loop, `descend_through_proof` returns `AppendPath::PathDiverged`, and the process panics with `"not yet implemented"`.
5. Block finalization fails; no state root or proof is produced. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
