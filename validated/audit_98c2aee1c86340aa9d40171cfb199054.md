### Title
Missing Slippage Protection in SNS Treasury Manager Deposit Interface Enables Front-Running of DAO Treasury Funds - (File: `rs/sns/treasury_manager/treasury_manager.did`, `rs/sns/governance/src/extensions.rs`)

---

### Summary
The SNS Treasury Manager `DepositRequest` interface, as defined in IC production code, contains no `min_return_amount` or slippage-tolerance parameter. The `execute_treasury_manager_deposit` function in SNS governance approves and transfers SNS treasury assets to a DEX-backed treasury manager without enforcing any minimum LP token return. This is an exact analog of the reported vulnerability: a buyer cannot set a minimum `returnAmount`, so price manipulation between proposal approval and execution can silently drain SNS DAO treasury value.

---

### Finding Description
The `DepositRequest` type in `rs/sns/treasury_manager/treasury_manager.did` is defined as:

```candid
type DepositRequest = record {
  allowances : vec Allowance;
};
```

It carries only the input token allowances — there is no field for a minimum expected return (LP tokens or otherwise). [1](#0-0) 

The IC code itself acknowledges this gap as a "Known Security Risk":

> Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved. [2](#0-1) 

The `execute_treasury_manager_deposit` function in `rs/sns/governance/src/extensions.rs` approves treasury funds and calls `deposit` on the treasury manager canister with no post-call check on the minimum value received:

```rust
// 1. Transfer funds from treasury to treasury manager
governance.approve_treasury_manager(
    extension_canister_id,
    treasury_allocation_sns_e8s,
    treasury_allocation_icp_e8s,
).await?;

// 2. Call deposit on treasury manager
let balances = governance.env.call_canister(
    extension_canister_id, "deposit", arg_blob
).await ...?;
``` [3](#0-2) 

The `validate_and_render_register_extension` function in `rs/sns/governance/src/proposal.rs` also explicitly warns about this in the proposal rendering text, confirming the risk is known but unmitigated at the protocol level:

> Some Decentralized Exchanges lack slippage protection during deposits. Consequently, deposited asset ratios may deviate from those specified in the proposal. This can expose liquidity pool adaptors to mispricing, making them vulnerable to front-running or sandwich attacks. [4](#0-3) 

The `TreasuryManager` trait in `rs/sns/treasury_manager/src/lib.rs` defines `deposit` with only a `DepositRequest` argument and no slippage parameter, making it structurally impossible for any conforming implementation to enforce a minimum return: [5](#0-4) 

---

### Impact Explanation
An unprivileged IC user who can interact with the DEX liquidity pool referenced by a treasury manager can execute a sandwich attack:

1. Observe the pending SNS governance proposal to deposit treasury funds (proposals are public).
2. Before the proposal executes, submit a large swap on the DEX to move the pool price adversarially.
3. The SNS governance canister executes `execute_treasury_manager_deposit`, transferring SNS and ICP treasury tokens to the treasury manager, which deposits them into the DEX at the manipulated price — receiving far fewer LP tokens than the fair-market rate.
4. The attacker reverses their position, extracting the price difference.

The SNS DAO treasury (holding ICP and SNS tokens) suffers a direct, quantifiable loss of value. The `DepositRequest` interface provides no mechanism to specify a minimum acceptable return, so the IC code cannot prevent this even if a treasury manager implementer wanted to enforce it.

---

### Likelihood Explanation
- SNS governance proposals are public and have a voting delay, giving attackers a predictable execution window.
- Any IC user with ICP or SNS tokens can interact with a DEX liquidity pool — no privileged access is required.
- The `TreasuryManager` API is explicitly designed for integration with external DEXes, making this the primary operational use case.
- The IC code itself documents this as a "Known Security Risk," confirming the attack surface is real and reachable.

---

### Recommendation
Add a `min_return_amount_decimals` (or equivalent slippage tolerance) field to `DepositRequest` in `rs/sns/treasury_manager/treasury_manager.did`. The `execute_treasury_manager_deposit` function in `rs/sns/governance/src/extensions.rs` should pass a caller-specified minimum return derived from the governance proposal parameters, and the treasury manager implementation must enforce it before completing the DEX deposit. Without this, the `DepositRequest` interface structurally cannot provide slippage protection regardless of the DEX used.

---

### Proof of Concept

1. An SNS DAO submits a governance proposal to deposit 100,000 SNS tokens and 10,000 ICP into a DEX liquidity pool via a registered treasury manager.
2. The proposal passes and is queued for execution (publicly visible on-chain).
3. An attacker calls the DEX directly to buy a large amount of SNS tokens, moving the pool price significantly against the DAO.
4. `execute_treasury_manager_deposit` executes: it calls `approve_treasury_manager` then `deposit` with a `DepositRequest { allowances: [...] }` — no minimum LP return is specified or checked. [6](#0-5) 
5. The treasury manager deposits at the manipulated price, receiving e.g. 30% fewer LP tokens than fair value.
6. The attacker sells their SNS tokens back, restoring the price and pocketing the spread.
7. The SNS treasury has permanently lost value with no protocol-level recourse, because `DepositRequest` carries no `min_return_amount` field and `execute_treasury_manager_deposit` performs no post-deposit value check. [1](#0-0)
