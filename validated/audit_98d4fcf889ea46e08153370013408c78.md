### Title
Unrestricted `deploy_erc20_token` Allows Any Caller to Pre-Register Any NEP-141 Token, Permanently Blocking Legitimate Bridge Registration - (File: engine/src/contract_methods/connector.rs)

---

### Summary

The `deploy_erc20_token` function in the Aurora Engine has no caller access control. Any NEAR account can call it with an arbitrary NEP-141 account ID, deploying a spurious ERC-20 and permanently writing the NEP-141→ERC-20 mapping into contract storage. Because `register_token` rejects any subsequent registration attempt for an already-mapped NEP-141 with `TokenAlreadyRegistered`, the legitimate bridge can never overwrite the attacker-planted mapping. All future `ft_on_transfer` calls from the real NEP-141 contract will mint into the attacker-planted ERC-20, and the correct ERC-20 can never be registered, permanently freezing the bridge for that token.

---

### Finding Description

`deploy_erc20_token` in `engine/src/contract_methods/connector.rs` performs only a liveness check (`require_running`) before proceeding: [1](#0-0) 

There is no `require_owner_only`, no `assert_private_call`, and no whitelist check. Any NEAR account can supply any `nep141: AccountId` in the `DeployErc20TokenArgs::Legacy` variant and the engine will:

1. Deploy a fresh ERC-20 contract at a deterministic address.
2. Call `engine.register_token(address, nep141)`, which writes the permanent NEP-141→ERC-20 mapping. [2](#0-1) 

`register_token` enforces a one-time-write invariant — once a NEP-141 is mapped, any subsequent attempt returns `TokenAlreadyRegistered` and the mapping cannot be changed: [3](#0-2) 

Contrast this with `mirror_erc20_token`, which correctly gates the analogous operation behind `require_owner_only`: [4](#0-3) 

---

### Impact Explanation

Once an attacker pre-registers `target.nep141`, every subsequent `ft_on_transfer` call originating from the real `target.nep141` contract routes into `receive_erc20_tokens`, which resolves the ERC-20 address via `get_erc20_from_nep141` and mints into the attacker-planted ERC-20: [5](#0-4) [6](#0-5) 

Users who bridge tokens receive ERC-20 balances in a contract they did not expect and cannot exit through the normal bridge flow. The legitimate NEP-141 can never be re-registered. This constitutes **permanent freezing of bridged funds** for the targeted token.

---

### Likelihood Explanation

The attack requires only a standard NEAR account and a single transaction. An attacker can monitor the NEAR mempool or governance channels for upcoming bridge listings and front-run the legitimate `deploy_erc20_token` call. No special privilege, leaked key, or oracle dependency is needed. The cost is negligible NEAR gas.

---

### Recommendation

Add a caller restriction to `deploy_erc20_token` analogous to the one already present on `mirror_erc20_token`. At minimum, gate the function behind `require_owner_only` or a dedicated admin whitelist. If permissionless deployment is desired for some use cases, introduce a two-step flow: an owner-approved allowlist of NEP-141 accounts that may be registered, checked before `register_token` is called.

---

### Proof of Concept

1. Attacker observes that `usdc.near` is about to be officially bridged to Aurora.
2. Attacker calls `deploy_erc20_token` on the Aurora Engine contract with `DeployErc20TokenArgs::Legacy("usdc.near")` from any NEAR account.
3. The engine deploys a spurious ERC-20 at address `A` and writes `usdc.near → A` into storage.
4. The legitimate bridge operator calls `deploy_erc20_token("usdc.near")` — `register_token` returns `TokenAlreadyRegistered` and the call fails.
5. Any user who calls `ft_transfer_call` on `usdc.near` targeting Aurora will trigger `ft_on_transfer`; the engine resolves `usdc.near → A` and mints into the spurious ERC-20. The user's NEP-141 tokens are locked in Aurora's custody with no valid exit path through the expected ERC-20. [7](#0-6) [8](#0-7)

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)

            recipient = fallback_address;

        let erc20_admin_address = current_address(current_account_id);

            .call(

                u64::MAX,

                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

pub fn deploy_erc20_token<I: IO + Copy, E: Env, P: PromiseHandler>(
    nep141: AccountId,
    metadata: Option<Erc20Metadata>,
    io: I,
    env: &E,
    handler: &mut P,
) -> Result<Address, DeployErc20Error> {
    let current_account_id = env.current_account_id();
    let input = setup_deploy_erc20_input(&current_account_id, metadata);
    let mut engine: Engine<_, _> = Engine::new(
        aurora_engine_sdk::types::near_account_to_evm_address(
            env.predecessor_account_id().as_bytes(),
        ),
        current_account_id,
        io,
        env,
    )
    .map_err(DeployErc20Error::State)?;

    let address = match engine.deploy_code_with_input(input, None, handler) {
        Ok(result) => match result.status {
            TransactionStatus::Succeed(ret) => {
                Address::new(H160(ret.as_slice().try_into().unwrap()))
            }
            other => return Err(DeployErc20Error::Failed(other)),
        },
        Err(e) => return Err(DeployErc20Error::Engine(e)),
    };

    sdk::log!("Deployed ERC-20 in Aurora at: {:#?}", address);
    engine
        .register_token(address, nep141)
        .map_err(DeployErc20Error::Register)?;

    Ok(address)
}
