Audit Report

## Title
Single-Principal Pending Queue Exhaustion via `retrieve_doge_with_approval` — (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary

`retrieve_btc_with_approval` enforces a global ceiling of `MAX_CONCURRENT_PENDING_REQUESTS = 5000` on accepted-but-unconfirmed withdrawal requests with no per-principal sub-limit. A single unprivileged actor with sufficient ckDOGE capital can sequentially fill all 5000 slots, causing every subsequent legitimate withdrawal from any principal to receive `TemporarilyUnavailable("too many pending retrieve_btc requests")` until the minter processes all attacker requests on-chain.

## Finding Description

`retrieve_doge_with_approval` in `rs/dogecoin/ckdoge/minter/src/main.rs` (L130–143) delegates directly to `retrieve_btc_with_approval` in `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`.

Two guards exist, neither of which prevents a single principal from filling the queue:

**Guard 1 — `retrieve_btc_guard` (`MAX_CONCURRENT = 100`)** (`rs/bitcoin/ckbtc/minter/src/guard.rs`, L6, L41–60): Tracks accounts **currently in-flight** in `retrieve_btc_accounts`, a `BTreeSet<Account>`. It is released via `Drop` the moment the async function returns (L63–67). Once a request is accepted into the pending queue and the function returns, the guard is dropped. The same account can immediately submit another request sequentially.

**Guard 2 — `MAX_CONCURRENT_PENDING_REQUESTS = 5000` (global, no per-principal sub-limit)** (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`, L22, L274–279): This is a global ceiling on all accepted-but-unconfirmed requests. No per-principal or per-account sub-limit exists anywhere in the minter source.

**Attack sequence:**
1. Attacker acquires 5000 × `min_retrieve_amount` of ckDOGE and sets ICRC-2 approvals.
2. Attacker calls `retrieve_doge_with_approval` sequentially with a valid address and amount ≥ `min_retrieve_amount`. Each call: acquires the guard, passes the global count check, burns ckDOGE, adds a `RetrieveBtcRequest` to the pending queue, drops the guard.
3. After 5000 accepted requests, `count_incomplete_retrieve_btc_requests() >= MAX_CONCURRENT_PENDING_REQUESTS` is true.
4. Every subsequent call from any principal returns `TemporarilyUnavailable("too many pending retrieve_btc requests")`.

The attacker does not need multiple subaccounts — sequential calls with a single account suffice because the in-flight guard is dropped after each completed call.

## Impact Explanation

The ckDOGE withdrawal path is completely blocked for all users until the minter processes all 5000 attacker requests and confirms them on the Dogecoin network. The attacker can refill the queue as slots are freed, sustaining the DoS indefinitely. This matches the allowed ICP bounty impact: **High — Application/platform-level DoS with concrete user harm, not based on raw volumetric DDoS.**

## Likelihood Explanation

The attacker does not permanently lose funds — ckDOGE is burned and DOGE is returned to their address. The only cost is opportunity cost of capital and Dogecoin network fees. The attack requires no privileged access, no key compromise, and no governance manipulation — only standard ingress calls available to any principal. It is repeatable as long as the attacker has capital.

## Recommendation

Add a per-principal (or per-`Account`) cap on accepted pending requests, checked atomically within the guard window and before the burn:

```rust
const MAX_PENDING_REQUESTS_PER_PRINCIPAL: usize = 50;

if read_state(|s| s.count_pending_requests_for(caller) >= MAX_PENDING_REQUESTS_PER_PRINCIPAL) {
    return Err(RetrieveBtcWithApprovalError::TemporarilyUnavailable(
        "too many pending requests for this principal".into(),
    ));
}
```

This check must occur **after** `retrieve_btc_guard` is acquired and **before** the burn, so it is enforced within the async execution window.

```rust
impl<PR: PendingRequests> Guard<PR> {
    /// Attempts to create a new guard for the current block. Fails if there is
    /// already a pending request for the specified [principal] or if there
    /// are at least [MAX_CONCURRENT] pending requests.
    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }
```
