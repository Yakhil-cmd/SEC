### Title
ERC-20 Tokens Burned During `exit_to_near` Are Permanently Lost When `ft_transfer` Promise Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

### Summary

When a user bridges ERC-20 tokens from Aurora to NEAR via the `exit_to_near` precompile, the tokens are burned from the user's EVM balance before the NEAR-side `ft_transfer` promise executes. If that promise fails (e.g., recipient account not registered with the NEP-141 contract), the callback `exit_to_near_precompile_callback` is invoked. However, the refund path inside that callback is **entirely gated behind the `error_refund` compile-time feature flag**, which is **not included in the default feature set**. Without it, `ExitToNearPrecompileCallbackArgs.refund` is always `None`, the re-mint branch is never taken, and the burned tokens are permanently lost.

### Finding Description

**Step 1 — Tokens are burned before the promise resolves.**

In `engine-precompiles/src/native.rs`, the `ExitToNear` precompile burns the caller's ERC-20 tokens (via the ERC-20 contract's burn/withdraw path) and schedules an `ft_transfer` promise on the NEP-141 contract. The burn is irreversible at the EVM level before the promise outcome is known.

**Step 2 — The refund address is only parsed under `error_refund`.** [1](#0-0) 

The `refund_address` field in both `BaseTokenParams` and `Erc20TokenParams` is `#[cfg(feature = "error_refund")]`-gated. Without the feature, no refund address is stored in the params struct.

**Step 3 — `refund_call_args` is entirely absent without the feature.** [2](#0-1) 

The function `refund_call_args` that constructs the `RefundCallArgs` (recipient address + ERC-20 address + amount) is compiled only under `#[cfg(feature = "error_refund")]`. Without it, `ExitToNearPrecompileCallbackArgs { refund: None, ... }` is always serialized into the callback args.

**Step 4 — The callback silently skips the refund.** [3](#0-2) 

In `exit_to_near_precompile_callback`, the refund branch `else if let Some(args) = args.refund` is never entered because `args.refund` is always `None` when the feature is absent. No re-mint of ERC-20 tokens and no ETH credit is issued.

**Step 5 — `error_refund` is not a default feature.** [4](#0-3) 

`default = ["std"]` — `error_refund` is absent from the default feature set. The production WASM contract compiled with `contract` feature does not include `error_refund`, confirming the refund path is dead code in production.

**Step 6 — Tests confirm the loss.** [5](#0-4) 

The test explicitly documents: *"If the refund feature is not enabled then there is no refund in the EVM"* — the user's ERC-20 balance is permanently reduced by `FT_EXIT_AMOUNT` with no recovery path.

### Impact Explanation

**Permanent freezing/loss of user funds (Critical).** Any EVM user who calls `exit_to_near` (directly or via an ERC-20 `withdraw` function) and whose `ft_transfer` promise fails will have their tokens permanently burned with no recourse. The NEP-141 tokens remain in the Aurora engine account (not transferred to the recipient), while the ERC-20 tokens are gone from the user's EVM balance. There is no admin recovery path — the ERC-20 supply is deflated and the user's funds are unrecoverable.

### Likelihood Explanation

**Medium-High.** The `ft_transfer` promise can fail for multiple realistic reasons reachable by any user:
- The recipient NEAR account is not registered with the NEP-141 contract (storage deposit not paid).
- The NEP-141 contract is paused.
- The recipient account ID is valid but the contract rejects the transfer for any reason.

These are ordinary user-facing conditions, not exotic attack scenarios. Any user who makes a typo in the recipient account or sends to an unregistered account triggers the loss.

### Recommendation

Enable the `error_refund` feature in the production build, or unconditionally include the refund address in `ExitToNearParams` and always populate `ExitToNearPrecompileCallbackArgs.refund` so that `exit_to_near_precompile_callback` can re-mint burned ERC-20 tokens (or credit ETH) whenever the downstream `ft_transfer` promise fails. The refund logic in `engine::refund_on_error` already exists and is correct — it only needs to be reachable.

### Proof of Concept

1. User holds 100 units of ERC-20 token `T` on Aurora (backed by NEP-141 `t.near`).
2. User calls `T.withdraw(amount=100, recipient="unregistered.near")`, which triggers the `exit_to_near` precompile.
3. The precompile burns 100 units from the user's EVM balance and schedules `t.near::ft_transfer({"receiver_id":"unregistered.near","amount":"100"})`.
4. `ft_transfer` fails because `unregistered.near` has no storage deposit with `t.near`.
5. `exit_to_near_precompile_callback` is invoked. `args.refund` is `None` (no `error_refund` feature). The `else if` branch is skipped. No re-mint occurs.
6. User's ERC-20 balance: 0. NEP-141 balance of `unregistered.near`: 0. NEP-141 balance of Aurora engine: unchanged (tokens not transferred). ERC-20 supply: permanently reduced by 100. Funds are lost.

#[cfg(feature = "error_refund")]
#[allow(clippy::unnecessary_wraps)]
fn refund_call_args(
    params: &ExitToNearParams,
    event: &events::ExitToNear,
) -> Option<RefundCallArgs> {
    Some(RefundCallArgs {
        recipient_address: match params {
            ExitToNearParams::BaseToken(params) => params.refund_address,
            ExitToNearParams::Erc20TokenParams(params) => params.refund_address,
        },
        erc20_address: match params {
            ExitToNearParams::BaseToken(_) => None,
            ExitToNearParams::Erc20TokenParams(_) => {
                let erc20_address = match event {
                    events::ExitToNear::Legacy(legacy) => legacy.erc20_address,
                    events::ExitToNear::Omni(omni) => omni.erc20_address,
                };
                Some(erc20_address)
            }
        },
        amount: types::u256_to_arr(&match event {
            events::ExitToNear::Legacy(legacy) => legacy.amount,
            events::ExitToNear::Omni(omni) => omni.amount,
        }),
    })
}
