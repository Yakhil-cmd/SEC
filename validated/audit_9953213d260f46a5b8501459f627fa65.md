Audit Report

## Title
Missing Freshness Check on `icp_xdr_conversion_rate` in `tokens_to_cycles()` Enables Stale-Rate Arbitrage During XRC Outage - (File: `rs/nns/cmc/src/main.rs`)

## Summary
The CMC function `tokens_to_cycles()` reads only `xdr_permyriad_per_icp` from the stored `icp_xdr_conversion_rate` and performs no comparison of `timestamp_seconds` against the current time. When the Exchange Rate Canister (XRC) is unavailable, the CMC retries at one-minute intervals but never invalidates or age-gates the stored rate. Any unprivileged user can exploit the resulting stale rate by calling `notify_top_up`, `notify_mint_cycles`, or `notify_create_canister` while ICP's market price has diverged from the stale stored rate, receiving excess cycles per ICP burned.

## Finding Description
`tokens_to_cycles()` at `rs/nns/cmc/src/main.rs` lines 1900–1923 extracts only `rate.xdr_permyriad_per_icp` and discards `rate.timestamp_seconds` entirely:

```rust
let xdr_permyriad_per_icp = state
    .icp_xdr_conversion_rate
    .as_ref()
    .map(|rate| rate.xdr_permyriad_per_icp);
```

This function is called unconditionally by all three public ICP-to-cycles conversion paths: `process_create_canister` (line 1932), `process_mint_cycles` (line 1965), and `process_top_up` (line 1991).

The heartbeat-driven `update_exchange_rate` in `rs/nns/cmc/src/exchange_rate_canister.rs` refreshes the rate every 5 minutes (`REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS`, line 16). On XRC failure, `schedule_next_attempt` (lines 149–163) reschedules a retry at the next minute but leaves `state.icp_xdr_conversion_rate` untouched. `do_set_icp_xdr_conversion_rate` (lines 1022–1030 of `main.rs`) enforces only monotonicity (new timestamp > current timestamp) and imposes no maximum age on the stored rate. There is no point in any of the three notify call paths where `rate.timestamp_seconds` is compared against `ic_cdk::api::time()` before the conversion executes.

The CMC's default state initializes `icp_xdr_conversion_rate` to a hardcoded May 10, 2021 timestamp (`DEFAULT_ICP_XDR_CONVERSION_RATE_TIMESTAMP_SECONDS`), confirming the system is designed to operate with an arbitrarily stale rate at startup and during XRC outages.

## Impact Explanation
**Severity: High** — Significant XRC/NNS infrastructure security impact with concrete protocol harm.

During any XRC outage (upgrade, transient error, persistent failure), the CMC continues minting cycles at the last stored rate. If ICP's market price drops while the stale rate is higher, an attacker purchases ICP cheaply on the open market and converts it to cycles at the inflated stale rate, receiving more cycles per ICP than the protocol should issue. This constitutes illegal minting: cycles are issued in excess of their XDR-backed value, diluting the cycle economy.

The `base_cycles_limit` of 150P cycles/hour (`DEFAULT_CYCLES_LIMIT = 150e15`) bounds the per-hour damage but does not eliminate it. Over an extended XRC outage with a significant rate discrepancy (e.g., 2.5×), the attacker can extract substantial excess cycles within the hourly window, and repeat across multiple hours until the XRC recovers. The impact is concrete and measurable: excess cycles minted per ICP burned, directly harming the protocol's economic integrity.

## Likelihood Explanation
The XRC is a live NNS canister subject to upgrades and transient failures. The CMC's own error-handling code explicitly anticipates XRC unavailability by scheduling 1-minute retries. The exploit requires only: (1) a period of XRC unavailability, (2) a meaningful divergence between the stale stored rate and the current market price of ICP, and (3) a standard permissionless ICP transfer to the CMC's subaccount followed by a notify call. No special privileges, governance access, or key material are required. The attacker can repeat the exploit each hour (resetting the rate limiter window) for as long as the XRC remains unavailable.

## Recommendation
In `tokens_to_cycles()`, retrieve the current time via `ic_cdk::api::time() / 1_000_000_000` and compare it against `rate.timestamp_seconds`. If the rate age exceeds a defined maximum staleness threshold (e.g., `REFRESH_RATE_INTERVAL_SECONDS * N` for some small N, such as 3–6), return a `NotifyError::Other` with a descriptive message indicating the rate is stale, allowing callers to retry once the rate is refreshed. This check should be added before the `TokensToCycles` computation and should not affect the `None` branch (which already returns an error).

## Proof of Concept
1. Deploy a local replica with the NNS. Confirm the CMC's `icp_xdr_conversion_rate` is set to rate `R` at time `T`.
2. Disable or mock the XRC to return persistent errors (e.g., `ExchangeRateError::StablecoinRateTooFewRates` as used in the existing integration test at `rs/nns/integration_tests/src/cycles_minting_canister_with_exchange_rate_canister.rs` line 169).
3. Advance the state machine clock by 1+ hours. Confirm via `get_icp_xdr_conversion_rate` that the stored rate timestamp has not advanced (the CMC retries but fails, leaving the old rate in place).
4. Call `notify_mint_cycles` with a valid ICP transfer at the current (lower) market price.
5. Observe that cycles are minted at rate `R` (the stale rate), not at the current market rate. The minted cycle count will exceed what the current XDR value of the burned ICP should produce.
6. Confirm no rejection occurs due to rate staleness — the only rejection path in `tokens_to_cycles()` is `None` (no rate stored at all), not a stale rate.
