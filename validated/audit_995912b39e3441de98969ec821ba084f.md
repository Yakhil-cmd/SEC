Audit Report

## Title
Stale ICP/XDR Exchange Rate Used Without Age Check in Cycles Minting Canister - (File: rs/nns/cmc/src/main.rs)

## Summary
The `tokens_to_cycles` function in the Cycles Minting Canister reads the cached `icp_xdr_conversion_rate` and uses it for all ICP-to-cycles conversions without checking whether the rate's `timestamp_seconds` is within an acceptable age window. When the Exchange Rate Canister (XRC) is temporarily unavailable, the CMC heartbeat silently fails to refresh the rate, and any user can continue calling `notify_top_up` or `notify_create_canister` to mint cycles at the arbitrarily stale rate, causing direct economic harm to the protocol or to users depending on the direction of price movement.

## Finding Description
`tokens_to_cycles` at `rs/nns/cmc/src/main.rs` L1900–1923 extracts only `xdr_permyriad_per_icp` from the cached rate and performs a `None` check — `rate.timestamp_seconds` is never compared against the current canister time:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles { ... }.to_cycles(amount)),
            None => Err(...),
        }
    })
}
```

`validate_exchange_rate` in `rs/nervous_system/clients/src/exchange_rate_canister_client.rs` L111–129 only validates source counts (`base_asset_num_received_rates`, `quote_asset_num_received_rates`); it contains no timestamp/age check. `do_set_icp_xdr_conversion_rate` at `rs/nns/cmc/src/main.rs` L1022–1030 only enforces that a new rate's timestamp is strictly greater than the stored one — it imposes no upper bound on how old the stored rate may become. `REFRESH_RATE_INTERVAL_SECONDS` (5 minutes) in `rs/nns/cmc/src/exchange_rate_canister.rs` L15–16 governs only when to fetch a new rate, not whether to reject a stale cached value at use time. `update_exchange_rate` at `rs/nns/cmc/src/exchange_rate_canister.rs` L259–274 returns `UpdateExchangeRateError::FailedToRetrieveRate` on XRC failure without invalidating or expiring the cached rate.

The exploit path is direct: `notify_top_up` (L1139–1227) calls `process_top_up` (L1985–2011), which calls `tokens_to_cycles` (L1991). `notify_create_canister` (L1345–1410) calls `process_create_canister` (L1925–1955), which calls `tokens_to_cycles` (L1932). Both endpoints are `#[update]` with no privilege requirement for normal callers.

## Impact Explanation
This matches the High bounty impact: "Significant XRC, NNS, or infrastructure security impact with concrete user or protocol harm." If the ICP market price falls while the cached rate is stale (e.g., 20% drop over a 30-minute XRC outage), every `notify_top_up` call mints cycles at the inflated stale rate — a direct, permanent economic loss for the protocol (cycles are minted below cost and ICP is burned at the wrong ratio). If the price rises, users receive fewer cycles than they paid for. The CMC is the sole on-chain mechanism for minting cycles; the impact affects every ICP-to-cycles conversion during the stale window. The magnitude is bounded by outage duration, price movement, and call volume, making a single-event loss in the range of thousands to hundreds of thousands of dollars realistic but not reliably over $1M, placing this at High rather than Critical.

## Likelihood Explanation
No attacker capability beyond submitting a normal ingress message is required. The XRC is a system canister on a dedicated subnet; rolling upgrades, transient network partitions, and XRC canister upgrades are routine operational events that can cause multi-minute to multi-hour unavailability. During such a window the CMC continues accepting `notify_top_up` calls. An opportunistic user needs only to observe that the XRC is down (visible from CMC logs or by querying the cached rate's timestamp) and submit a valid ICP transfer followed by a `notify_top_up` call. The attack is repeatable for every new ICP transfer submitted during the outage window.

## Recommendation
In `tokens_to_cycles`, compare `rate.timestamp_seconds` against the current canister time and reject conversions if the rate exceeds a defined maximum age (e.g., `MAX_RATE_AGE_SECONDS = 2 * REFRESH_RATE_INTERVAL_SECONDS = 600`):

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let now = now_seconds();
        match state.icp_xdr_conversion_rate.as_ref() {
            Some(rate) if now.saturating_sub(rate.timestamp_seconds) <= MAX_RATE_AGE_SECONDS => {
                Ok(TokensToCycles {
                    xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
                    cycles_per_xdr: state.cycles_per_xdr,
                }.to_cycles(amount))
            }
            Some(_) => Err(NotifyError::Other {
                error_code: NotifyErrorCode::Internal as u64,
                error_message: "ICP/XDR conversion rate is stale; retry later".to_string(),
            }),
            None => Err(...),
        }
    })
}
```

Additionally, add a `ValidateExchangeRateError::RateTooOld` variant to `validate_exchange_rate` so that freshly fetched rates whose timestamp is far in the past are rejected before being stored.

## Proof of Concept
1. Deploy CMC and XRC on a local PocketIC or StateMachine test environment.
2. Set an initial ICP/XDR rate via `update_exchange_rate` so `icp_xdr_conversion_rate` is populated.
3. Disable or mock the XRC client to return errors (simulating an outage).
4. Advance the test clock by more than `REFRESH_RATE_INTERVAL_SECONDS` (5 minutes) — the CMC heartbeat will attempt and fail to refresh the rate, leaving the old rate in place.
5. Record the current `icp_xdr_conversion_rate.timestamp_seconds` and confirm it is now stale relative to the simulated clock.
6. Submit a valid ICP transfer to the CMC subaccount for a target canister.
7. Call `notify_top_up` with the resulting `block_index`.
8. Observe that `tokens_to_cycles` succeeds and mints cycles at the stale rate — no staleness error is returned.
9. Vary the mocked stale rate value to demonstrate that the minted cycle count reflects the old rate, not the current market rate.
