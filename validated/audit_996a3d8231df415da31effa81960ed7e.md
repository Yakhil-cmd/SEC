Audit Report

## Title
`get_running_version()` Fails Entirely When Any Archive Canister Status Is Unavailable, Permanently Corrupting SNS `deployed_version` — (File: `rs/sns/governance/src/sns_upgrade.rs`)

## Summary
`get_running_version()` uses a `collect::<Result<Vec<_>, _>>()?` over all archive canister summaries, causing the entire function to return `Err` if any single archive canister's status is `None`. Because `get_owned_canister_summary` silently returns `CanisterSummary { status: None }` on any management canister call failure, a single frozen or stopped archive canister during the 5-minute upgrade verification window causes `check_upgrade_status()` to permanently mark the upgrade as failed and clear `pending_version` without updating `deployed_version`, leaving the SNS with a stale recorded version that diverges from the actual running code.

## Finding Description
In `rs/sns/governance/src/sns_upgrade.rs`, `get_running_version()` collects archive hashes via:

```rust
let archive_wasm_hash = archives
    .into_iter()
    .map(|canister_summary| get_hash(canister_summary, "Ledger Archive"))
    .collect::<Result<Vec<_>, _>>()?
``` [1](#0-0) 

The `get_hash` closure returns `Err("{label} had no status")` when `canister_summary.status` is `None`, and the `?` propagates this error out of `get_running_version()` entirely. [2](#0-1) 

In `rs/sns/root/src/lib.rs`, `get_owned_canister_summary()` silently swallows management canister call failures and returns `CanisterSummary { status: None }`: [3](#0-2) 

All archive canisters are queried via `join_all` with no special error handling: [4](#0-3) 

In `check_upgrade_status()`, when `get_running_version()` returns `Err` and `self.env.now() > mark_failed_at`, `complete_sns_upgrade_to_next_version()` is called with `deployed_version: None`: [5](#0-4) 

`complete_sns_upgrade_to_next_version()` only updates `deployed_version` when the argument is `Some`; with `None`, it clears `pending_version` without recording the new version: [6](#0-5) 

The `mark_failed_at_seconds` deadline is only 5 minutes from upgrade initiation: [7](#0-6) 

## Impact Explanation
After the timeout, the SNS's `proto.deployed_version` permanently records the pre-upgrade version while the actual canister runs the new code. All subsequent `UpgradeSnsToNextVersion` proposals compute upgrade paths from the stale `deployed_version`, potentially re-upgrading already-upgraded canisters or failing to find a valid upgrade path. The automatic advancement mechanism (`initiate_upgrade_if_sns_behind_target_version`) is similarly confused. This constitutes a significant, concrete SNS governance security impact — permanently disrupting the upgrade mechanism for the affected SNS instance — matching the **High ($2,000–$10,000)** impact class: "Significant SNS or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
Any SNS instance with active trading volume will accumulate archive canisters over time. Archive canisters are not automatically topped up by SNS governance. A naturally cycle-depleted archive canister during any upgrade window is sufficient to trigger the bug. An unprivileged attacker can accelerate archive canister creation by submitting many small ICRC-1 transfers (the `icrc1_transfer` endpoint is publicly accessible), increasing the probability that an archive canister is frozen during the 5-minute window. No special privileges are required; the attacker only needs to hold a small amount of the SNS token.

## Recommendation
1. **Tolerate missing archive statuses**: In `get_running_version()`, replace the `collect::<Result<Vec<_>, _>>()?` with a filter that skips archives with `None` status (or treats them as unknown), consistent with the existing `unwrap_or_default()` for the zero-archive case.
2. **Decouple archive verification from core canister verification**: The upgrade verification for root, governance, ledger, index, and swap should not be blocked by archive availability.
3. **Extend `mark_failed_at_seconds`**: The 5-minute timeout is too short for a system dependent on inter-canister calls that may be transiently unavailable. A 30-minute timeout would significantly reduce false failure marking.

## Proof of Concept
1. Deploy an SNS with an active ICRC-1 ledger that has at least one archive canister.
2. Deplete the archive canister's cycles below its freezing threshold (naturally or by generating transaction volume via `icrc1_transfer`).
3. Submit and execute an `UpgradeSnsToNextVersion` proposal; `pending_version` is set with `mark_failed_at_seconds = now + 300`.
4. During the 5-minute window, `run_periodic_tasks()` calls `check_upgrade_status()` → `get_running_version()` → `get_sns_canisters_summary` with `update_canister_list: true`.
5. SNS Root calls `get_owned_canister_summary` for the frozen archive; management canister returns an error; `status` is set to `None`.
6. `get_hash(archive_summary, "Ledger Archive")` returns `Err("Ledger Archive had no status")`; `?` propagates; `get_running_version()` returns `Err`.
7. After 5 minutes, `complete_sns_upgrade_to_next_version(proposal_id, Timeout, message, None)` is called; `pending_version` is cleared; `deployed_version` is **not** updated.
8. Verify: `proto.deployed_version` still holds the pre-upgrade version hash while the actual canister runs the new wasm. Subsequent upgrade proposals fail or re-upgrade the already-upgraded canister.

A deterministic integration test using PocketIC can reproduce this by: deploying a full SNS, registering a mock archive canister that returns an error on `canister_status`, triggering an upgrade proposal, advancing the IC time by 6 minutes, and asserting that `deployed_version` was not updated despite the upgrade succeeding.
