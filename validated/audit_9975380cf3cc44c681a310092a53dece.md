Audit Report

## Title
Missing Response-Type/Callback-ID Cross-Check in `push_response` Allows Byzantine Node to Permanently Discard a `BitcoinGetSuccessors` Context — (`rs/replicated_state/src/bitcoin.rs`, `rs/bitcoin/consensus/src/payload_builder.rs`)

## Summary

The `SendTransactionResponse` branch of `push_response` in `rs/replicated_state/src/bitcoin.rs` performs no lookup against `bitcoin_send_transaction_internal_contexts` before enqueuing an `EmptyBlob` response. A single Byzantine subnet node can propose a block whose `SelfValidatingPayload` contains a `BitcoinAdapterResponse { response: SendTransactionResponse, callback_id: X }` where X is a live `BitcoinGetSuccessors` context. Validation passes because `validate_self_validating_payload_impl` only checks payload size. After finalization, `push_response` enqueues `EmptyBlob` for callback_id X, and `retrieve_context` permanently removes the `BitcoinGetSuccessors` context from state, stalling the bitcoin canister's UTXO chain-tip synchronization for that request slot.

## Finding Description

**Root cause — `validate_self_validating_payload_impl` performs no semantic validation:**

The function at `rs/bitcoin/consensus/src/payload_builder.rs` lines 234–251 only checks whether the payload is empty and returns its byte size. It never verifies that each `BitcoinAdapterResponse.callback_id` exists in state, nor that the response variant matches the request type stored for that callback_id. [1](#0-0) 

`validate_payload` (the `BatchPayloadBuilder` implementation) delegates entirely to `validate_self_validating_payload_impl` after a decode and size check, adding no callback_id or type validation. [2](#0-1) 

**Root cause — `SendTransactionResponse` branch of `push_response` has no guard:**

The `SendTransactionResponse` arm at lines 64–76 of `rs/replicated_state/src/bitcoin.rs` extracts the `callback_id` from the response, constructs `EmptyBlob`, and unconditionally pushes to `consensus_queue`. It never looks up `callback_id` in `bitcoin_send_transaction_internal_contexts`. [3](#0-2) 

**Contrast — `GetSuccessorsResponse` branch does validate:**

The `GetSuccessorsResponse` arm at lines 28–39 calls `.get_mut(&callback_id).ok_or_else(|| StateError::BitcoinNonMatchingResponse { ... })?` against `bitcoin_get_successors_contexts`, returning an error if the callback_id is absent. The `SendTransactionResponse` branch has no equivalent guard. [4](#0-3) 

**Permanent context removal — `retrieve_context`:**

When execution processes the `consensus_queue` entry for callback_id X, `retrieve_context` is called. It iterates through all context maps and calls `.remove(&callback_id)` on each, including `bitcoin_get_successors_contexts`. Once removed, no honest proposer can re-deliver the real `GetSuccessorsResponse` for that slot, because `bitcoin_requests_iter` will no longer yield callback_id X. [5](#0-4) 

**Exploit flow:**

1. State contains `BitcoinGetSuccessorsContext` at `callback_id = X` in `bitcoin_get_successors_contexts`.
2. Byzantine node proposes a block with `SelfValidatingPayload` containing `BitcoinAdapterResponse { response: SendTransactionResponse(()), callback_id: X }`.
3. All honest validators call `validate_payload` → `validate_self_validating_payload_impl` → size check only → `Ok(())`. Block is finalized.
4. Message routing calls `push_response` with the crafted response. `SendTransactionResponse` branch: no lookup, enqueues `ConsensusResponse { callback_id: X, payload: EmptyBlob }` to `consensus_queue`.
5. Execution round drains `consensus_queue`, calls `retrieve_context(X)`, which removes the `BitcoinGetSuccessors` context from `bitcoin_get_successors_contexts`.
6. The bitcoin canister's `GetSuccessors` callback receives `EmptyBlob` (Candid-encoded) instead of a `BitcoinGetSuccessorsResponse`. Candid decoding fails; the canister traps. The canister's heap is rolled back, but the subnet-level removal of the context from `bitcoin_get_successors_contexts` persists.
7. `bitcoin_requests_iter` no longer yields callback_id X; no honest proposer will ever re-deliver the real response. The `GetSuccessors` request slot is permanently lost.

## Impact Explanation

This matches the allowed High impact: **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact"** and **"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."**

The bitcoin canister's UTXO chain-tip synchronization stalls permanently for the affected request slot. Since ckBTC minting and redemption depend on the bitcoin canister maintaining an up-to-date UTXO set, permanent disruption of `GetSuccessors` processing constitutes concrete harm to the ckBTC Chain Fusion integration. The false-minting claim is correctly not supported (canister traps and heap rolls back), but the permanent loss of the `GetSuccessors` context is a deterministic, irreversible outcome.

## Likelihood Explanation

Requires a single Byzantine subnet node below the consensus fault threshold — explicitly in-scope per the ICP attacker model. The attack is deterministic: one crafted block proposal suffices. No victim interaction, no phishing, no key leakage, and no governance majority is required. The attack is repeatable: a Byzantine node can target each new `GetSuccessors` context as it appears in state.

## Recommendation

1. In `validate_self_validating_payload_impl`, load the certified state and for each `BitcoinAdapterResponse` in the payload, verify that `callback_id` exists in the correct context map: `bitcoin_get_successors_contexts` for `GetSuccessorsResponse`/`GetSuccessorsReject`, and `bitcoin_send_transaction_internal_contexts` for `SendTransactionResponse`/`SendTransactionReject`. Return `InvalidArtifact` on any mismatch or missing callback_id.

2. In the `SendTransactionResponse` branch of `push_response`, add the same guard present in `GetSuccessorsResponse`: look up `callback_id` in `bitcoin_send_transaction_internal_contexts` and return `StateError::BitcoinNonMatchingResponse` if not found. Apply the same fix to `GetSuccessorsReject` and `SendTransactionReject` branches, which also lack callback_id validation.

## Proof of Concept

```rust
// Setup: state with a BitcoinGetSuccessors context at callback_id=0
let callback_id = state.metadata.subnet_call_context_manager.push_context(
    SubnetCallContext::BitcoinGetSuccessors(BitcoinGetSuccessorsContext { ... }),
);
assert_eq!(callback_id, CallbackId::from(0));

// Byzantine payload: SendTransactionResponse targeting the GetSuccessors callback_id
let result = push_response(
    &mut state,
    BitcoinAdapterResponse {
        response: BitcoinAdapterResponseWrapper::SendTransactionResponse(
            SendTransactionResponse {}
        ),
        callback_id: 0,
    },
);

// Passes with Ok(()) — no error, no validation
assert!(result.is_ok());

// consensus_queue now contains EmptyBlob for a GetSuccessors callback_id
assert_eq!(state.consensus_queue.len(), 1);
assert_eq!(
    state.consensus_queue[0].payload,
    Payload::Data(EmptyBlob.encode())
);

// GetSuccessors context is still present (removed later by retrieve_context during execution)
assert!(state.metadata.subnet_call_context_manager
    .bitcoin_get_successors_contexts.contains_key(&CallbackId::from(0)));

// After execution processes consensus_queue: retrieve_context(0) removes the
// BitcoinGetSuccessors context permanently. bitcoin_requests_iter no longer
// yields callback_id=0, so no honest proposer can re-deliver the real response.
```

A unit test in `rs/replicated_state/src/bitcoin.rs` can directly call `push_response` with a mismatched response type and assert that it currently returns `Ok(())` and enqueues `EmptyBlob`, confirming the missing guard.
