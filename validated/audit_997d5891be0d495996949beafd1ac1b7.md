### Title
`native_per_pubdata` Truncates to Zero via Floor Division When `pubdata_price < native_price`, Allowing Pubdata Fee Bypass - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When the operator-set `pubdata_price` is less than `native_price`, the integer floor division used to compute `native_per_pubdata` truncates to zero. This causes the entire pubdata charging subsystem to charge zero native resources for any amount of pubdata generated by a transaction, allowing an unprivileged sender to generate unbounded pubdata at no native cost.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (ZK transaction flow), the per-pubdata-byte native cost is computed as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

This uses **floor (truncating) integer division**. Whenever `pubdata_price < native_price`, the result is `0`. For example, with `native_price = 100` and `pubdata_price = 2` (a realistic configuration used in the test suite), `native_per_pubdata = floor(2/100) = 0`.

Contrast this with `native_per_gas`, which correctly uses **ceiling division** to avoid rounding to zero:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 135
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

The same floor-division pattern is replicated in `api/src/helpers.rs`:

```rust
// api/src/helpers.rs, line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

When `native_per_pubdata = 0`, the downstream charging function `get_resources_to_charge_for_pubdata` computes:

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs, lines 430-432
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // = 0 for any pubdata_spent
    .ok_or(out_of_native_resources!())?;
``` [4](#0-3) 

This returns `Resources::from_native(0)` regardless of how many pubdata bytes the transaction generated. Consequently, `check_enough_resources_for_pubdata` always returns `enough = true`, the post-execution pubdata revert path is never triggered, and `compute_gas_refund` charges zero native for pubdata. [5](#0-4) 

The intrinsic pubdata overhead in `create_resources_for_tx` is also zeroed out:

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs, line 352
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata); // = 0
``` [6](#0-5) 

The double-resource-accounting model is documented to charge pubdata from native resources separately from EVM gas: [7](#0-6) 

When `native_per_pubdata = 0`, this entire pubdata-native charging path is silently bypassed.

---

### Impact Explanation

An unprivileged transaction sender operating in a block where `pubdata_price < native_price` can submit transactions that write to many storage slots (generating large state diffs / pubdata) and pay only EVM gas for computation, with zero native resource cost for pubdata. The operator must still publish this pubdata to the settlement layer and bear the L1 data availability cost. This is a resource accounting bug: the user's fee does not cover the actual cost imposed on the protocol. At scale, this enables a sustained, cheap DoS against the operator's L1 publishing budget.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is realistic in normal operation. The test suite itself uses `native_price = 100, pubdata_price = 2` as a standard test configuration, which produces `native_per_pubdata = 0`. [8](#0-7) 

Any block where the operator sets `pubdata_price` to a value smaller than `native_price` (e.g., during periods of low L1 data costs relative to proving costs) creates the exploitable condition for all transactions in that block.

---

### Recommendation

Replace the floor division for `native_per_pubdata` with ceiling division, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates to zero when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...

// After (rounds up, ensuring at least 1 native unit per pubdata byte when pubdata_price > 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))...
```

Apply the same fix in `api/src/helpers.rs`. Additionally, consider adding an explicit guard: if `pubdata_price > 0` but `native_per_pubdata` would be 0 after division, either reject the transaction or enforce a minimum of 1.

---

### Proof of Concept

1. Operator sets block context: `native_price = 1000`, `pubdata_price = 999`.
2. `native_per_pubdata = floor(999 / 1000) = 0`.
3. Attacker submits a transaction calling a contract that writes to 1000 distinct storage slots (generating ~32,000 bytes of pubdata).
4. `get_resources_to_charge_for_pubdata` returns `(32000, Resources::from_native(0))`.
5. `check_enough_resources_for_pubdata` returns `enough = true`.
6. Transaction succeeds; attacker pays only EVM gas (~5M gas for 1000 SSTOREs), zero native for pubdata.
7. Operator must publish 32,000 bytes to L1 at their own cost.
8. Repeat across many transactions to drain operator's L1 budget.
