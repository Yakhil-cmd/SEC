### Title
Missing Subgroup Check in G1ADD and G2ADD Precompiles Causes EIP-2537 Deviation — (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/addition.rs`)

---

### Summary

`Bls12381G1AdditionPrecompile` and `Bls12381G2AdditionPrecompile` call `parse_g1`/`parse_g2` (which only verify the point is on the curve) instead of the already-implemented `parse_g1_with_subgroup_check`/`parse_g2_with_subgroup_check`. An unprivileged caller can supply a point that is on the BLS12-381 curve but outside the prime-order subgroup; the precompile will succeed and return a point also outside the subgroup. EIP-2537 mandates rejection of such inputs for G1ADD and G2ADD.

---

### Finding Description

`parse_g1_bytes` in `crypto/src/bls12_381/eip2537.rs` constructs a point with `G1Affine::new_unchecked(x, y)` and only calls `is_on_curve()`: [1](#0-0) 

The wrapper `parse_g1` in `mod.rs` forwards this result without any subgroup check: [2](#0-1) 

The correct helper `parse_g1_with_subgroup_check` exists in the same file and calls `is_in_correct_subgroup_assuming_on_curve()`: [3](#0-2) 

However, `Bls12381G1AdditionPrecompile::execute` calls the unchecked `parse_g1` for both inputs: [4](#0-3) 

And `Bls12381G2AdditionPrecompile::execute` does the same with `parse_g2`: [5](#0-4) 

By contrast, both `Bls12381G1MSMPrecompile` and `Bls12381PairingCheckPrecompile` correctly use `parse_g1_with_subgroup_check`: [6](#0-5) [7](#0-6) 

The inconsistency is clear: the subgroup-check infrastructure is present and used everywhere except the addition precompiles.

---

### Impact Explanation

- **EVM behavioral deviation**: ZKsync OS accepts non-subgroup G1/G2 points in G1ADD/G2ADD and returns a result, while Ethereum mainnet (revm, geth) rejects them with a precompile error. Any smart contract that relies on the precompile reverting for out-of-subgroup inputs will behave differently on ZKsync OS.
- **Cryptographic protocol breakage**: A BLS signature aggregation or ZK-proof verification contract that uses G1ADD to combine public keys can be fed a rogue non-subgroup key. The addition succeeds and produces a point outside the subgroup, silently corrupting the aggregated key and potentially allowing signature forgery or proof acceptance for invalid statements.
- **Cross-chain inconsistency**: Contracts deployed identically on Ethereum and ZKsync OS will diverge in behavior for this input class, breaking the EVM-equivalence guarantee.

---

### Likelihood Explanation

The attack requires only knowledge of a BLS12-381 curve point outside the r-order subgroup (trivially constructed as `cofactor * arbitrary_curve_point`) and a standard EVM call to the G1ADD precompile address. No privileged access, leaked keys, or external oracle cooperation is needed. The precompile is live once the Prague/Osaka hardfork activates.

---

### Recommendation

In `addition.rs`, replace `parse_g1` with `parse_g1_with_subgroup_check` and `parse_g2` with `parse_g2_with_subgroup_check`:

```rust
// addition.rs line 34
let p0 = parse_g1_with_subgroup_check(input[0..G1_SERIALIZATION_LEN].try_into().unwrap())?;
let p1 = parse_g1_with_subgroup_check(input[G1_SERIALIZATION_LEN..(2 * G1_SERIALIZATION_LEN)].try_into().unwrap())?;
```

And equivalently for G2. This matches the pattern already used in `msm.rs` and `pairing.rs`.

---

### Proof of Concept

1. Compute a non-subgroup G1 point: pick any curve point `Q` and compute `P = cofactor * Q` where `cofactor = 0x396c8c005555e1568c00aaab0000aaab`. `P` is on the curve but not in the r-order subgroup.
2. Encode `P` in EIP-2537 uncompressed format (128 bytes: 64-byte x, 64-byte y, each zero-padded to 64 bytes).
3. Encode the standard G1 generator `G` in the same format.
4. Concatenate: `input = encode(P) || encode(G)` (256 bytes total).
5. Call the G1ADD precompile at address `BLS12_G1ADD` with this input.
6. **Expected (EIP-2537 / mainnet)**: precompile reverts with `PointNotInSubgroup`.
7. **Actual (ZKsync OS)**: precompile returns `Ok(())` with a 128-byte output representing `P + G`, a point outside the subgroup.
8. Compare with revm's output to confirm the deviation.

```rust
fn parse_g1_with_subgroup_check(
    input: &[u8; G1_SERIALIZATION_LEN],
) -> Result<G1Affine, Bls12PrecompileSubsystemError> {
    let point = parse_g1(input)?;
    if point.is_zero() || point.is_in_correct_subgroup_assuming_on_curve() {
        Ok(point)
    } else {
        Err(interface_error!(
            Bls12PrecompileInterfaceError::PointNotInSubgroup
        ))
    }
}
```
