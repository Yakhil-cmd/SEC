### Title
`VotingPowerEconomics.clear_following_after_seconds` Can Be Set to Zero via `ManageNetworkEconomics` Proposal, Causing a Panic in `LinearMap::new()` That Breaks NNS Governance Voting Power Calculations - (File: `rs/nns/governance/src/network_economics.rs`)

---

### Summary

`VotingPowerEconomics::validate()` explicitly permits `clear_following_after_seconds = 0`, but the runtime function `deciding_voting_power_adjustment_factor_function()` constructs a `LinearMap` whose `from` range degenerates to `begin..begin` when `clear_following_after_seconds` is zero. `LinearMap::new()` contains a hard `assert!` that panics on this degenerate range. A `ManageNetworkEconomics` governance proposal setting this field to zero passes all validation gates, is committed to state, and then causes every subsequent NNS governance message that computes deciding voting power to trap — effectively freezing the NNS governance canister.

---

### Finding Description

**Root cause — missing lower-bound validation in `VotingPowerEconomics::validate()`:** [1](#0-0) 

The comment on line 350 explicitly states *"They are allowed to be set to 0 though."* No check enforces `clear_following_after_seconds > 0`.

**Runtime panic path — `deciding_voting_power_adjustment_factor_function()`:** [2](#0-1) 

When `clear_following_after_seconds = 0`:
- `begin = get_start_reducing_voting_power_after_seconds()` (any value, e.g. the default 6 months)
- `end = begin.saturating_add(0) = begin`
- `from_range = begin..begin` — a zero-length range

**`LinearMap::new()` panics on a zero-length `from` range:** [3](#0-2) 

The `assert!(from.end != from.start, "{from:#?}")` fires unconditionally.

**Validation is called at both proposal creation and execution time, but passes for zero:** [4](#0-3) 

`apply_changes_and_validate` calls `validate()`, which accepts `clear_following_after_seconds = Some(0)` without error, so the proposal is committed to state.

**The panicking function is on the hot path for every neuron's voting power:** [5](#0-4) 

Every update call that creates a proposal, registers a vote, or reads neuron info invokes `deciding_voting_power_adjustment_factor`, which calls `deciding_voting_power_adjustment_factor_function`, which panics.

---

### Impact Explanation

Once a `ManageNetworkEconomics` proposal setting `clear_following_after_seconds = 0` is executed and committed to the NNS governance canister's stable state, every subsequent ingress message or inter-canister call that computes deciding voting power traps. This includes:

- Proposal creation (`make_proposal`)
- Vote registration (`register_vote`)
- Neuron info queries that return `deciding_voting_power`

The NNS governance canister becomes unable to process governance actions. Because the bad value is persisted in state and the panic occurs inside the message handler (causing a rollback of the *current* message but not the earlier state-commit), the canister is stuck in a broken state until an upgrade replaces the canister code or the state is repaired via a hotfix proposal — which itself cannot be submitted because proposal creation also panics.

---

### Likelihood Explanation

The attacker path requires a `ManageNetworkEconomics` governance proposal to pass NNS voting. This is a high bar, but:

1. The proposal looks syntactically valid and passes all validation checks — there is no error message to alert voters.
2. The field `clear_following_after_seconds` is a relatively obscure parameter; a proposal that sets it to zero alongside other legitimate changes could go unnoticed.
3. The vulnerability class (missing bounds check on a critical parameter) is exactly the same as the external report: the validation function explicitly documents that zero is "allowed," creating a false sense of safety for proposal authors and reviewers.

The analog to the original report is direct: just as `CollateralizationRatio = 0` passes the owner's setter without rejection and silently breaks liquidation, `clear_following_after_seconds = 0` passes `validate()` without rejection and silently breaks voting power computation — with the difference that the IC failure mode is a hard panic rather than a silent invariant violation.

---

### Recommendation

Add a lower-bound check in `VotingPowerEconomics::validate()`:

```rust
if let Some(0) = self.clear_following_after_seconds {
    defects.push(
        "clear_following_after_seconds must be greater than zero.".to_string()
    );
}
```

Alternatively, guard `deciding_voting_power_adjustment_factor_function()` against the degenerate case by returning a constant function (always 1 or always 0) when `clear_following_after_seconds == 0`, and remove the `assert!` from `LinearMap::new()` in favour of a saturating/clamped fallback.

---

### Proof of Concept

1. Submit a `ManageNetworkEconomics` proposal with:
   ```
   voting_power_economics = VotingPowerEconomics {
       start_reducing_voting_power_after_seconds: Some(15_778_800), // 6 months (default)
       clear_following_after_seconds: Some(0),                      // ← zero
       neuron_minimum_dissolve_delay_to_vote_seconds: Some(15_778_800),
   }
   ```
2. `validate_manage_network_economics` calls `apply_changes_and_validate` → `validate()` → returns `Ok(())` because `clear_following_after_seconds` is `Some(0)`, which is not `None`.
3. The proposal is executed; `heap_data.economics.voting_power_economics.clear_following_after_seconds` is now `Some(0)`.
4. Any subsequent call to `potential_and_deciding_voting_power` invokes `deciding_voting_power_adjustment_factor_function()`:
   - `begin = 15_778_800`, `end = 15_778_800.saturating_add(0) = 15_778_800`
   - `LinearMap::new(15_778_800..15_778_800, 1..0)` → `assert!(15_778_800 != 15_778_800)` → **PANIC / TRAP**
5. All governance update calls that touch voting power (proposal creation, vote registration) now trap, freezing the NNS. [6](#0-5) [7](#0-6) [2](#0-1)
