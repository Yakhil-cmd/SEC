### Title
Unbacked ERC-20 Token Minting via Unauthenticated `ft_on_transfer` Call - (File: engine/src/contract_methods/connector.rs)

### Summary

`ft_on_transfer` — the NEP-141 token-receipt callback — performs no check that the caller has actually transferred tokens to Aurora before minting ERC-20 tokens. Because `deploy_erc20_token` is also open to any caller, an unprivileged NEAR account can register its own NEP-141 token and then directly invoke `ft_on_transfer` to mint an arbitrary quantity of the corresponding ERC-20 tokens without depositing a single NEP-141 token. This breaks the 1:1 backing invariant and enables direct theft of funds held by Aurora for legitimate depositors of that token.

---

### Finding Description

**Vulnerability class:** ERC-20 mirror accounting bug / auth bypass

**Step 1 — Anyone can register a NEP-141 token.**

`deploy_erc20_token` contains only a liveness check (`require_running`); there is no owner or whitelist guard: [1](#0-0) 

Any NEAR account can call this method and obtain a registered ERC-20 address for an arbitrary NEP-141 account ID it controls.

**Step 2 — `ft_on_transfer` routes to the minting handler without verifying the caller transferred tokens.**

The function reads `predecessor_account_id` from the NEAR environment and immediately dispatches to either `receive_base_tokens` (if the predecessor is the configured connector) or `receive_erc20_tokens` (for every other account). There is no `assert_private_call`, no promise-result check, and no whitelist guard: [2](#0-1) 

The NEP-141 standard requires that `ft_on_transfer` is called only as a callback from `ft_transfer_call`, after the token contract has already transferred tokens to the receiver. Aurora enforces none of this.

**Step 3 — `receive_erc20_tokens` mints tokens using only a registry lookup.**

The only guard inside the minting path is `get_erc20_from_nep141`, which succeeds as soon as the predecessor is a registered NEP-141 account — a condition the attacker already satisfied in Step 1: [3](#0-2) 

The ERC-20 `mint` call is then executed with the Aurora engine as the admin, so it always succeeds.

---

### Impact Explanation

**Critical — direct theft of user funds / insolvency.**

Aurora's ERC-20 tokens are supposed to be backed 1:1 by NEP-141 tokens held by the Aurora contract. Once an attacker mints unbacked ERC-20 tokens:

- If legitimate users have deposited the attacker's NEP-141 token to Aurora (e.g., the attacker marketed it as a useful token), the attacker can call `ft_on_transfer` to mint additional ERC-20 tokens equal to the deposited amount and then use the exit precompile to withdraw the legitimate users' NEP-141 deposits — a direct theft.
- Even without prior deposits, the unbacked ERC-20 tokens can be used as collateral in DeFi protocols deployed on Aurora, draining liquidity from those protocols.
- The backing invariant is permanently broken for the affected token, causing insolvency for all holders.

---

### Likelihood Explanation

**High.** The attack requires no special privileges:

1. Create a NEAR account.
2. Call `deploy_erc20_token` (no authorization required).
3. Call `ft_on_transfer` directly from that account with an arbitrary `amount` and a target EVM address.

All three steps are executable by any unprivileged NEAR account in a single block sequence. No leaked keys, governance capture, or oracle manipulation is needed.

---

### Recommendation

Add a check inside `ft_on_transfer` that enforces the call is arriving as a genuine NEP-141 callback (i.e., it is a promise callback with at least one successful promise result), analogous to the `assert_private_call` pattern used in other sensitive callbacks such as `exit_to_near_precompile_callback` and `withdraw_wnear_to_router`: [4](#0-3) [5](#0-4) 

Concretely, require `handler.promise_results_count() >= 1` and that the single promise result is `Successful`, which is only possible when the function is invoked as a NEAR callback — not as a direct top-level call. Additionally, consider restricting `deploy_erc20_token` to the owner or a whitelist to prevent arbitrary token registration.

---

### Proof of Concept

1. Attacker creates NEAR account `attacker-token.near` (a minimal NEP-141 implementation is not required for the exploit itself).
2. Attacker calls `deploy_erc20_token` on the Aurora engine with `nep141 = "attacker-token.near"`. No authorization check is performed; an ERC-20 address is returned and the mapping is stored. [6](#0-5) 
3. Attacker calls `ft_on_transfer` on Aurora **directly** from `attacker-token.near` with `{ sender_id: "attacker.near", amount: "1000000000", msg: "<attacker_evm_address>" }`.
4. `ft_on_transfer` reads `predecessor_account_id = "attacker-token.near"`, which is not the connector account, so it calls `receive_erc20_tokens("attacker-token.near", args, ...)`. [7](#0-6) 
5. `receive_erc20_tokens` resolves the ERC-20 address via `get_erc20_from_nep141` (succeeds because of Step 2) and calls the ERC-20 `mint` selector with `amount = 1_000_000_000` for the attacker's EVM address. [8](#0-7) 
6. The attacker now holds 1 000 000 000 ERC-20 tokens backed by zero NEP-141 tokens. Any legitimate NEP-141 deposits from other users can be drained via the exit precompile.

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;
