### Title
Missing Storage Non-Empty Collision Check in CREATE/CREATE2 Allows Deployment at Addresses with Pre-Existing Attacker-Controlled Storage - (File: `evm_interpreter/src/ee_trait_impl.rs`)

---

### Summary

ZKsync OS's EVM interpreter omits the storage-non-empty component of the EIP-161 CREATE/CREATE2 collision check. The code explicitly acknowledges this gap with a comment. An attacker can pre-populate storage at a deterministic CREATE2 address (by deploying a contract there, writing storage, then self-destructing it), and when a victim later redeploys at the same address, the constructor executes against attacker-controlled storage values — a scenario Ethereum would have rejected outright.

---

### Finding Description

EIP-161 / Yellow Paper specifies that a CREATE or CREATE2 must fail with a collision if the target address satisfies **any** of:
- `code_len != 0`
- `nonce != 0`
- **`storage != empty`** ← **this check is absent in ZKsync OS**

In `evm_interpreter/src/ee_trait_impl.rs`, the `before_executing_frame` function performs the collision check:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    ...
    tracer.evm_tracer().on_call_error(&EvmError::CreateCollision);
    return Ok(false);
}
``` [1](#0-0) 

The comment `"NB: EVM also specifies that the address should have empty storage, but we cannot perform such a check for now"` is the root cause admission. The condition `deployee_code_len != 0 || deployee_nonce != 0` is **incomplete** — it never checks whether the target address has non-zero storage slots.

This is corroborated by multiple disabled test entries across the test suite:

- `create2collisionStorageParis.json` — `comment: We do not check storage for collision`
- `RevertInCreateInInitCreate2Paris.json` — `comment: We do not check for storage collisions`
- `dynamicAccountOverwriteEmpty_Paris.json` — `comment: Expects revert due to storage collision`
- `test_init_collision_create_opcode.json` — `comment: We do not check the storage`
- `test_init_collision_create_tx.json` — `comment: We do not check the storage` [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

### Impact Explanation

**Attack scenario (Cancun semantics, where EIP-6780 preserves storage after SELFDESTRUCT in a later tx):**

1. Attacker computes the CREATE2 address `X` that a victim protocol will use for a future deployment (salt `S`, deployer `D`, initcode `I`).
2. Attacker deploys a contract at `X` (using the same `S`, `D`, `I` or any other means that lands code at `X`).
3. Attacker's contract writes attacker-controlled values to storage slots (e.g., slot 0 = `attacker_value`).
4. Attacker's contract self-destructs in a **later transaction** — under Cancun, `SELFDESTRUCT` no longer clears storage (EIP-6780). Address `X` now has: `nonce=0`, `code_len=0`, `storage={0: attacker_value}`.
5. Victim protocol deploys at `X` via CREATE2. ZKsync OS checks only `code_len` and `nonce` — both are 0 — so the collision check **passes** and the constructor executes.
6. The victim contract's constructor reads storage slot 0 and receives `attacker_value` instead of the expected zero.
7. On Ethereum mainnet, step 5 would have been **rejected** with a collision revert.

**Consequence:** The newly deployed contract is initialized with attacker-controlled state. Depending on the contract logic, this can lead to:
- Bypassed access controls (e.g., an `owner` slot pre-set to the attacker's address)
- Incorrect accounting (e.g., a balance or supply slot pre-set to a large value)
- Broken invariants that the constructor assumes start from zero

This is a **state-transition bug / EVM semantic mismatch** with a direct, unprivileged attacker-controlled entry path.

---

### Likelihood Explanation

The attack requires:
1. Predicting a future CREATE2 address — straightforward for any deterministic deployment pattern.
2. Deploying a contract at that address first — requires only ETH for gas; no privileged access.
3. Self-destructing it in a later transaction — standard operation.
4. Waiting for the victim to redeploy — applicable to any protocol with upgrade/redeploy patterns.

This is realistic for protocols that use CREATE2 for deterministic addresses and have any redeployment or upgrade path. The attacker needs no special permissions.

---

### Recommendation

Implement the storage non-empty check as part of the CREATE/CREATE2 collision guard. Since ZKsync OS uses a flat storage model, the check requires determining whether any storage slot under the target address is non-zero. The `before_executing_frame` guard should be extended to query the storage subsystem for the target address and reject the deployment if any slot is non-zero, consistent with EIP-161 and EIP-7610 (`create2collisionStorageParis` test vectors).

If a full storage scan is architecturally infeasible, a nonce-bump-on-first-storage-write approach (as used in some L2s) can serve as a proxy: any address that has ever had storage written will have a non-zero nonce, making the existing nonce check sufficient.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

// Step 1: Deploy this at the target CREATE2 address, write storage, then self-destruct.
contract Poisoner {
    constructor() {
        // Write attacker-controlled value to slot 0
        assembly { sstore(0, 0xdeadbeef) }
        selfdestruct(payable(msg.sender)); // Must be called in a LATER tx for storage to persist (EIP-6780)
    }
    function destroy() external {
        selfdestruct(payable(msg.sender));
    }
}

// Step 2: Victim deploys this at the same address via CREATE2.
// On ZKsync OS: deployment succeeds, slot 0 reads 0xdeadbeef.
// On Ethereum:  deployment reverts with CreateCollision (storage non-empty).
contract Victim {
    uint256 public owner; // slot 0
    constructor() {
        // Expects owner == 0, but gets 0xdeadbeef on ZKsync OS
        require(owner == 0, "storage poisoned");
    }
}
