### Title
Single-Step Owner Transfer Enables Permanent Loss of Engine Control - (File: `engine/src/contract_methods/admin.rs`)

### Summary
The `set_owner` function in the Aurora Engine immediately and irrevocably replaces `state.owner_id` with the caller-supplied `new_owner` in a single transaction, with no confirmation step from the new owner. If the wrong NEAR account ID is provided, all owner-gated engine functions become permanently inaccessible, permanently freezing the engine and all user funds it holds.

### Finding Description

`set_owner` in `engine/src/contract_methods/admin.rs` performs a single-step ownership transfer:

```rust
pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }
        state.owner_id = args.new_owner;   // ← immediate, irreversible write
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
``` [1](#0-0) 

The only validation performed is that the new owner differs from the current owner (`ERR_SAME_OWNER`). There is no pending-owner storage slot, no acceptance call, and no time-lock. The new `owner_id` is written to persistent state atomically and takes effect immediately. [2](#0-1) 

The public NEAR contract entrypoint is exposed unconditionally:

```rust
#[unsafe(no_mangle)]
pub extern "C" fn set_owner() {
    let io = Runtime;
    let env = Runtime;
    contract_methods::admin::set_owner(io, &env)
        .map_err(ContractError::msg)
        .sdk_unwrap();
}
``` [3](#0-2) 

The `owner_id` field in `EngineState` is the sole authorization gate for the following critical functions, all of which check `require_owner_only`:

| Function | Effect |
|---|---|
| `upgrade` | Deploys arbitrary new WASM bytecode to the engine account |
| `stage_upgrade` | Stages new WASM code for deployment |
| `attach_full_access_key` | Adds a full-access key to the engine NEAR account |
| `pause_contract` / `resume_contract` | Pauses or unpauses all EVM execution |
| `set_upgrade_delay_blocks` | Controls the upgrade time-lock |
| `set_key_manager` | Controls the relayer key manager role | [4](#0-3) [5](#0-4) [6](#0-5) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

If `set_owner` is called with a NEAR account ID that does not exist, is a typo, or is otherwise inaccessible (e.g., a burned account), the engine permanently loses its owner. Every owner-gated function becomes unreachable forever. Specifically:

- `resume_contract` can never be called. If the engine is ever paused (by the current owner before the mistake, or by any future mechanism), it can never be unpaused. All EVM execution halts permanently.
- `upgrade` can never be called. Critical security bugs in the engine cannot be patched.
- `attach_full_access_key` can never be called. The engine account loses its last administrative escape hatch.

All ETH, ERC-20 tokens, and NEP-141 tokens held by or bridged through the Aurora Engine contract are permanently frozen with no recovery path.

**Critical — Direct theft of user funds (secondary scenario).**

If the wrong account ID happens to be controlled by an attacker (e.g., via a front-running or social-engineering vector that causes the owner to supply an attacker-chosen account ID), the attacker immediately gains the ability to call `upgrade` and deploy malicious WASM code, or call `attach_full_access_key` to add a full-access key to the engine account. Either action enables complete draining of all user funds.

### Likelihood Explanation

**Medium.** NEAR account IDs are human-readable UTF-8 strings (e.g., `aurora-dao.sputnik-dao.near`, `aurora.near`), not fixed-length hex addresses. They are significantly more prone to typographical errors than Ethereum addresses. The Aurora Engine owner is a multisig or governance contract; governance proposals that encode the new owner account ID as a string argument are a realistic source of transcription errors. A single mistaken character in the account ID string produces a valid `AccountId` that passes Borsh deserialization and the `ERR_SAME_OWNER` check, but refers to a non-existent or wrong account. [7](#0-6) 

### Recommendation

Implement a two-step ownership transfer:

1. **Propose step:** The current owner calls `propose_new_owner(new_owner: AccountId)`, which writes a `pending_owner_id: Option<AccountId>` field to `EngineState` but does **not** change `owner_id`.
2. **Accept step:** The proposed new owner calls `accept_ownership()`, which checks `env.predecessor_account_id() == state.pending_owner_id`, then atomically sets `state.owner_id = pending_owner_id` and clears `pending_owner_id`.

This ensures the new owner account demonstrably exists and is accessible before the transfer is finalized. Additionally, add a `cancel_ownership_proposal()` callable by the current owner to abort a pending transfer.

### Proof of Concept

```
1. Current owner of Aurora Engine is `aurora-dao.sputnik-dao.near`.
2. A governance vote passes to transfer ownership to `aurora-new-dao.sputnik-dao.near`.
3. The proposal encodes the new owner as `aurora-new-dao.sputnikdao.near` (missing hyphen before "dao").
4. The owner calls:
       set_owner({ new_owner: "aurora-new-dao.sputnikdao.near" })
5. `set_owner` passes all checks:
   - `require_owner_only`: predecessor is current owner ✓
   - `ERR_SAME_OWNER`: new != old ✓
   - `state.owner_id` is immediately written to `"aurora-new-dao.sputnikdao.near"` and persisted.
6. `"aurora-new-dao.sputnikdao.near"` does not exist on NEAR mainnet.
7. All subsequent calls to `upgrade`, `resume_contract`, `attach_full_access_key`, etc. fail with `ERR_NOT_ALLOWED`.
8. If the engine is subsequently paused (e.g., by a precompile pauser), it can never be unpaused.
9. All user funds are permanently frozen with no on-chain recovery path.
``` [8](#0-7)
