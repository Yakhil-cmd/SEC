### Title
Cross-Chain Replay of Unprotected Legacy Transactions via Missing Chain-ID Binding in Signing Hash — (`basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

ZKsync OS accepts pre-EIP-155 ("unprotected") legacy transactions whose signing hash is computed **without** the chain ID. A transaction signed on any other EVM-compatible chain (e.g., Ethereum mainnet) with `v = 27` or `v = 28` can be submitted verbatim to ZKsync OS and will pass signature verification, because the hash committed to by the signer contains no chain-specific data. This is the direct analog of the SGX replay-attack finding: just as `releaseHash` omits environment data and therefore allows the same attestation to be accepted on a different deployment, the unprotected-legacy signing hash omits the chain ID and therefore allows the same signed transaction to be accepted on a different chain.

---

### Finding Description

`LegacyPayloadParser::try_parse_and_hash_for_signature_verification` in `legacy_tx.rs` branches on whether the signature uses EIP-155 (`v ≥ 35`) or the pre-EIP-155 convention (`v = 27 / 28`):

```rust
// legacy_tx.rs L89-94
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    // Unprotected legacy
    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);          // only [nonce, gasPrice, gasLimit, to, value, data]
    hasher.finalize_reset().into()
} else {
    // EIP-155 protected: chain_id is mixed in
    ...
}
``` [1](#0-0) 

`is_eip155()` returns `false` exactly when `v == 27 || v == 28`:

```rust
pub fn is_eip155(&self) -> bool {
    self.v != 27 && self.v != 28
}
``` [2](#0-1) 

The resulting `sig_hash` (containing no chain ID) is then passed directly to `ecrecover` during validation:

```rust
ecrecover_input[0..32].copy_from_slice(suggested_signed_hash.as_u8_array_ref());
``` [3](#0-2) 

Because the hash is chain-agnostic, a valid signature produced on Ethereum mainnet (or any other EVM chain) for an unprotected legacy transaction is also a valid signature on ZKsync OS for the identical byte-for-byte transaction.

The `RlpEncodedTxInner` parser explicitly preserves both variants (`Legacy` and `LegacyWithEIP155`) and routes them through the same downstream validation path without any additional chain-binding check for the unprotected case: [4](#0-3) 

---

### Impact Explanation

**Vulnerability class**: Cross-chain replay / missing domain binding in signing hash (state-transition bug / public funds-loss path).

An attacker who observes an unprotected legacy transaction on Ethereum mainnet (or any other EVM chain) can submit the identical raw bytes to ZKsync OS. If the victim's account on ZKsync OS happens to have the same nonce as the replayed transaction, the bootloader will:

1. Compute the same chain-ID-free signing hash.
2. Recover the same `from` address via `ecrecover`.
3. Accept the transaction as valid.
4. Execute the transfer/call, draining the victim's ZKsync OS balance.

The victim never signed anything for ZKsync OS; their private key is never exposed. The attacker only needs the raw transaction bytes from the source chain.

---

### Likelihood Explanation

- Pre-EIP-155 transactions are still produced by older hardware wallets, certain DeFi protocols, and any wallet that explicitly sets `v = 27/28`.
- A victim's nonce on ZKsync OS must match the nonce in the replayed transaction. For accounts that are new to ZKsync OS (nonce = 0) and whose first Ethereum transaction was unprotected, the attack is immediately applicable.
- The attacker needs no privileged access: they only need to monitor the public mempool or historical chain data of any EVM chain.

---

### Recommendation

Reject unprotected legacy transactions outright at parse time. ZKsync OS is a new chain with no pre-EIP-155 history; there is no backward-compatibility reason to accept `v = 27/28`. Add an explicit check in `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`:

```rust
if !legacy_signature.is_eip155() {
    return Err(InvalidTransaction::InvalidChainId.into());
}
```

This mirrors the behavior already enforced for all typed transactions (EIP-2930, EIP-1559, EIP-7702), each of which rejects a chain-ID mismatch: [5](#0-4) 

---

### Proof of Concept

1. On Ethereum mainnet, broadcast an unprotected legacy ETH transfer from address `A` to address `B` with nonce `N`, `v = 27`. Record the raw RLP bytes `TX_RAW`.
2. Ensure address `A` holds funds on ZKsync OS and has nonce `N` there.
3. Submit `TX_RAW` unchanged to ZKsync OS via the standard transaction submission path.
4. ZKsync OS computes `keccak256(rlp([nonce, gasPrice, gasLimit, to, value, data]))` — identical to the Ethereum signing hash — recovers `A` via `ecrecover`, passes all validation checks, and executes the transfer, moving `A`'s ZKsync OS funds to `B`.

The victim (`A`) never authorized any ZKsync OS transaction; the attacker replayed a signature that was never intended for this chain.
