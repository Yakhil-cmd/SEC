Audit Report

## Title
SNS Neuron Maturity Drained to Old Owner via Pending Disbursement Before Permission Transfer - (`rs/sns/governance/src/governance.rs`)

## Summary
When an SNS neuron owner calls `disburse_maturity`, the destination account is irrevocably fixed and maturity is immediately deducted from `neuron.maturity_e8s_equivalent`, while the actual token minting is deferred by `MATURITY_DISBURSEMENT_DELAY_SECONDS`. Neither `add_neuron_permissions` nor `remove_neuron_permissions` checks for pending `disburse_maturity_in_progress` entries, allowing a malicious seller to drain all maturity to their own account and then transfer full neuron control to a buyer who receives a neuron with zero maturity and no recourse.

## Finding Description
In `governance.rs`, `disburse_maturity` (L1681–1698) immediately deducts maturity from `neuron.maturity_e8s_equivalent` and appends a `DisburseMaturityInProgress` record with a fixed `account_to_disburse_to` pointing to the caller's account. The periodic governance timer (L4969–5046) later mints tokens to that fixed account once `finalize_disbursement_timestamp_seconds` is reached.

`add_neuron_permissions` (L4570–4643) performs only three checks: caller authorization, grantable-permissions validation, and principal-count limit. It does not inspect `neuron.disburse_maturity_in_progress`. `remove_neuron_permissions` (L4659–4715) similarly checks only caller authorization and permission existence. Neither function has any guard against pending disbursements.

No `cancel_disburse_maturity` endpoint exists anywhere in the codebase (confirmed by exhaustive grep). The attack sequence is:
1. Seller calls `disburse_maturity { percentage: 100, to_account: seller }` — maturity drops to 0, pending record created pointing to seller.
2. Seller calls `add_neuron_permissions` to grant buyer all permissions.
3. Seller calls `remove_neuron_permissions` to remove their own permissions.
4. After 7 days, governance timer mints full maturity to seller's account.

The buyer cannot prevent step 1 from being inserted between their last neuron query and their acceptance of the transfer, because the IC has no atomic "read-then-accept" primitive for OTC neuron trades. The seller controls the entire sequence unilaterally.

## Impact Explanation
This matches **High ($2,000–$10,000): Significant SNS security impact with concrete user or protocol harm**. The new neuron owner suffers a direct, irreversible loss of SNS governance tokens proportional to the neuron's maturity. The maturity is already removed from the neuron's visible state at attack time, and the pending disbursement cannot be cancelled or redirected by anyone. The loss is bounded only by the neuron's accumulated maturity, which for large or long-staked neurons can be substantial.

## Likelihood Explanation
Any SNS neuron holder with `DisburseMaturity` and `ManagePrincipals` permissions — the default set — can execute this attack. OTC neuron trading is a documented use case for the `add/remove_neuron_permissions` mechanism. The attack requires no privileged access, no external dependencies, and no victim mistake beyond trusting the seller's stated neuron state. The seller can race `disburse_maturity` against the buyer's final pre-transfer query in consecutive IC messages, making detection by the buyer practically impossible without an atomic escrow mechanism.

## Recommendation
In `remove_neuron_permissions`, before completing removal of the last permission for a principal, check whether `neuron.disburse_maturity_in_progress` is non-empty and return an error (e.g., `PreconditionFailed: "Cannot remove all permissions while maturity disbursements are pending"`). Alternatively, add a `cancel_disburse_maturity` endpoint callable by any principal holding `ManagePrincipals` permission, which removes pending entries and restores `maturity_e8s_equivalent`. The former is the minimal fix; the latter also protects buyers who acquire a neuron that already has in-flight disbursements.

## Proof of Concept
Using PocketIC or a local SNS integration test:

1. Create an SNS neuron for principal `A` with `maturity_e8s_equivalent = 10_000_000_000`.
2. Call `manage_neuron(DisburseMaturity { percentage_to_disburse: 100, to_account: A })` as `A`. Assert `neuron.maturity_e8s_equivalent == 0` and `neuron.disburse_maturity_in_progress.len() == 1` with `account_to_disburse_to == A`.
3. Call `manage_neuron(AddNeuronPermissions { principal_id: B, permissions_to_add: [all] })` as `A`. Assert success.
4. Call `manage_neuron(RemoveNeuronPermissions { principal_id: A, permissions_to_remove: [all] })` as `A`. Assert success — confirming no check on pending disbursements.
5. Advance the test clock by `MATURITY_DISBURSEMENT_DELAY_SECONDS + 1`.
6. Trigger the governance periodic task (`run_periodic_tasks`).
7. Assert that `A`'s ledger account received `~10_000_000_000` SNS tokens and `B`'s account received nothing. Assert `neuron.disburse_maturity_in_progress` is now empty and `neuron.maturity_e8s_equivalent == 0`.
