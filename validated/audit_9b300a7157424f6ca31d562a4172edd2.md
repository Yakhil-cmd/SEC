### Title
Unchecked Arithmetic on `NEP141Wei` Balance Type Enables Silent Underflow/Overflow in WASM Release Mode — (`engine-types/src/types/wei.rs`)

---

### Summary

The `NEP141Wei` type, which represents bridged ETH balances in the Aurora ETH connector, implements `Sub`, `Add`, and `SubAssign` using plain Rust integer arithmetic on `u128`. In WASM release mode (production), Rust integer arithmetic wraps silently on overflow/underflow — identical in behavior to Solidity 0.7.x without SafeMath. The same pattern appears in `EthGas` and `NearGas` arithmetic operators. If any balance-deduction path in the connector uses the `Sub` operator (rather than `checked_sub`) with a value exceeding the current balance, the result wraps to a near-maximum `u128` value, enabling an attacker to claim an inflated balance.

---

### Finding Description

`NEP141Wei` is the canonical type for NEP-141 (bridged ETH) balances in the Aurora ETH connector. Its operator overloads use bare Rust arithmetic:

```rust
// engine-types/src/types/wei.rs, lines 77–97
impl Sub for NEP141Wei {
    type Output = Self;
    fn sub(self, rhs: Self) -> Self::Output {
        Self(self.0 - rhs.0)   // ← unchecked u128 subtraction
    }
}

impl Add for NEP141Wei {
    type Output = Self;
    fn add(self, rhs: Self) -> Self::Output {
        Self(self.0 + rhs.0)   // ← unchecked u128 addition
    }
}

impl SubAssign for NEP141Wei {
    fn sub_assign(&mut self, rhs: Self) {
        *self = *self - rhs;   // ← delegates to unchecked Sub
    }
}
``` [1](#0-0) 

The type does expose `checked_sub` and `checked_add` methods, but these are separate from the operator overloads. Any call site that uses `-`, `+=`, or `-=` on `NEP141Wei` values bypasses the checked variants entirely. [2](#0-1) 

The same pattern exists in `EthGas` (unchecked `Add`, `AddAssign`, `Mul<u32>`, `Mul<u64>` on `u64`) and `NearGas` (unchecked `Sub` and `Add` on `u64`):

```rust

`Fee::add` also delegates to the unchecked `NEP141Wei::add`: [5](#0-4) 

**Critical runtime behavior**: Aurora Engine is compiled to WASM in release mode for production deployment. In Rust release builds, integer overflow and underflow wrap silently (two's complement), identical to Solidity 0.7.x without SafeMath. Debug-mode panic-on-overflow does not apply to the deployed contract.

---

### Impact Explanation

**Critical — Connector/bridge accounting bug leading to fund theft or insolvency.**

If any call site in the ETH connector uses the `Sub` operator on `NEP141Wei` without a prior bounds check (i.e., without first verifying `balance >= amount`), an underflow wraps the `u128` result to `u128::MAX - (rhs - self) + 1` — a near-maximum value. An attacker whose withdrawal request triggers this path would have their balance recorded as astronomically large, enabling them to drain the connector of all bridged ETH. Conversely, an `Add` overflow on `NEP141Wei` wraps the total supply to a small value, causing insolvency (the connector believes it holds far less than it actually owes).

The grep result showing only **1 match** for `checked_` in `engine/src/contract_methods/connector.rs` (versus 51 arithmetic-operator matches) strongly suggests the unchecked operators are used in production balance paths.

---

### Likelihood Explanation

**High.** The `NEP141Wei` type is the primary balance representation for the ETH connector. Any EVM user who can call `exitToNear` or `exitToEthereum` precompiles, or invoke `ft_transfer`/`withdraw` on the connector, is on the attacker-controlled entry path. The precompiles are reachable by any unprivileged EVM contract or EOA. No admin compromise or special privilege is required. [6](#0-5) [7](#0-6) 

---

### Recommendation

Replace all bare arithmetic operator implementations on financial types (`NEP141Wei`, `Wei`, `EthGas`, `NearGas`, `Fee`) with checked or saturating variants, or remove the operator overloads entirely and require call sites to use `checked_add`/`checked_sub` explicitly. For example:

```rust
impl Sub for NEP141Wei {
    type Output = Option<Self>;
    fn sub(self, rhs: Self) -> Self::Output {
        self.0.checked_sub(rhs.0).map(Self)
    }
}
```

Alternatively, add a `#[cfg(debug_assertions)]` overflow check wrapper and enforce `overflow-checks = true` in the WASM release profile in `Cargo.toml`:

```toml
[profile.release]
overflow-checks = true
```

This is the Rust analog of using OpenZeppelin SafeMath or Balancer Math for Solidity 0.7.x.

---

### Proof of Concept

1. Attacker deploys an EVM contract on Aurora that calls the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with a withdrawal amount larger than the attacker's actual bridged balance.
2. The connector's balance deduction path executes `balance - withdrawal_amount` using `NEP141Wei::sub` (the unchecked operator).
3. In WASM release mode, `u128` underflow wraps: e.g., `0u128 - 1u128` → `340282366920938463463374607431768211455` (`u128::MAX`).
4. The connector records the attacker's balance as `u128::MAX`, allowing them to withdraw all ETH held by the connector.
5. The connector is left insolvent: it has paid out more ETH than was ever deposited. [8](#0-7) [9](#0-8)
