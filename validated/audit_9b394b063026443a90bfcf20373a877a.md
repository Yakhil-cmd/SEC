### Title
Incorrect Decimal Scaling Constant in `wei_from_milli_ether` Overrepresents Wei Amount by 1000x - (File: `rs/ethereum/cketh/minter/src/numeric/mod.rs`)

---

### Summary

In `rs/ethereum/cketh/minter/src/numeric/mod.rs`, the public function `wei_from_milli_ether` uses a constant `MILLI_ETHER = 1_000_000_000_000_000_000` (10^18) when the correct value for one milli-ether in wei is `1_000_000_000_000_000` (10^15). The constant is 1000x too large. The inline comment in the same function even states the intended exponent is 10^15, directly contradicting the actual constant. This is the same class of bug as the reported issue: a miswritten numeric literal that silently inflates a scaling factor.

---

### Finding Description

In `rs/ethereum/cketh/minter/src/numeric/mod.rs` at lines 25–30:

```rust
pub fn wei_from_milli_ether(value: u128) -> Wei {
    const MILLI_ETHER: u64 = 1_000_000_000_000_000_000;  // 10^18 — WRONG
    Wei::new(value)
        .checked_mul(MILLI_ETHER)
        .expect("any u128 multiplied by 10^15 always fits in a u256")  // comment says 10^15
}
```

The Ethereum denomination hierarchy is:
- 1 ETH = 10^18 wei
- 1 milli-ETH = 0.001 ETH = **10^15 wei**

The constant `MILLI_ETHER` is set to `1_000_000_000_000_000_000` = 10^18, which is the wei value of **1 full ETH**, not 1 milli-ETH. The correct value is `1_000_000_000_000_000` = 10^15.

The `.expect()` message on line 29 reads `"any u128 multiplied by 10^15 always fits in a u256"`, which is the developer's own documentation of the intended exponent — directly contradicting the actual constant used.

The sole test for this function (`should_not_overflow_when_converting_from_milli_ether`) only verifies that the result does not overflow for `u128::MAX`; it does not assert a semantically correct output value, so the test passes despite the wrong constant. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

Any caller of `wei_from_milli_ether` receives a `Wei` value that is **1000x larger than intended**. In the ckETH minter context, this function is part of the public API of the `numeric` module and is `pub`-exported. If it is invoked from any production code path (e.g., to compute minimum withdrawal thresholds, fee estimates, or deposit amounts denominated in milli-ETH), the resulting Wei amount would be inflated by 1000x. This could cause:

- **Ledger conservation bug**: Minting or burning 1000x the correct amount of ckETH tokens relative to the actual ETH deposited/withdrawn.
- **Chain-fusion mint/burn/replay bug**: Incorrect ETH↔ckETH conversion amounts passed to the Ethereum transaction layer, causing transactions to carry 1000x the intended value.
- **Denial of service**: Withdrawal requests computed using this function would require 1000x more ETH than the user holds, causing all such requests to fail. [1](#0-0) 

---

### Likelihood Explanation

The function is `pub` and resides in the production module `rs/ethereum/cketh/minter/src/numeric/mod.rs`. Based on a full-repo grep, it is currently only called from the test file `rs/ethereum/cketh/minter/src/numeric/tests.rs`. However, the function is part of the public surface of the `numeric` module and is available for use by any future or existing caller within the minter crate. The bug is latent: it is triggered the moment any production code path calls `wei_from_milli_ether`. The incorrect constant is not caught by the existing test, which only checks for non-overflow, not for value correctness. [2](#0-1) 

---

### Recommendation

Replace the incorrect constant with the correct milli-ether value:

```diff
pub fn wei_from_milli_ether(value: u128) -> Wei {
-    const MILLI_ETHER: u64 = 1_000_000_000_000_000_000;
+    const MILLI_ETHER: u64 = 1_000_000_000_000_000;
    Wei::new(value)
        .checked_mul(MILLI_ETHER)
        .expect("any u128 multiplied by 10^15 always fits in a u256")
}
```

Additionally, add a correctness assertion to the test:

```rust
assert_eq!(wei_from_milli_ether(1), Wei::from(1_000_000_000_000_000_u64)); // 10^15
assert_eq!(wei_from_milli_ether(1000), Wei::from(1_000_000_000_000_000_000_u64)); // 1 ETH
``` [1](#0-0)

### Proof of Concept

**Denomination math:**
- 1 ETH = 1,000,000,000,000,000,000 wei (10^18)
- 1 milli-ETH = 0.001 ETH = 1,000,000,000,000,000 wei (10^15)

**Current behavior:**
```
wei_from_milli_ether(1) = 1 * 1_000_000_000_000_000_000 = 10^18 wei = 1 ETH  ← WRONG
```

**Correct behavior:**
```
wei_from_milli_ether(1) = 1 * 1_000_000_000_000_000     = 10^15 wei = 0.001 ETH  ← CORRECT
```

The developer's own `.expect()` comment on line 29 — `"any u128 multiplied by 10^15 always fits in a u256"` — is the smoking gun: it documents the intended exponent as 10^15, while the actual constant is 10^18. [3](#0-2)
