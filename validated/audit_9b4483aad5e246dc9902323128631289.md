### Title
Asymmetric `delta_gas` Accounting Silently Overcharges Users When Native Resource Consumption Is Below EVM Gas Equivalent — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` in the ZKsync OS bootloader implements a one-sided adjustment for the dual-resource (EVM gas / native proving resource) accounting model. When the native resource consumed is *greater* than the EVM gas equivalent (`delta_gas > 0`), `gas_used` is increased and the user pays more. When the native resource consumed is *less* than the EVM gas equivalent (`delta_gas < 0`), `gas_used` is **not** decreased. The "unused" native resource equivalent is silently discarded rather than returned to the user as a gas refund. A developer TODO comment at the exact decision point acknowledges the gap. The result is that users are systematically overcharged on every ZK L2 transaction (and L1 priority transaction) where EVM gas consumption exceeds the native proving cost.

---

### Finding Description

`compute_gas_refund` is the single function that determines how much gas a user is charged and how much is refunded after execution. It implements the ZKsync OS "double resource accounting" model described in `docs/double_resource_accounting.md`:

> `deltaGas := (nativeUsed / nativePerGas) - gasUsed`
> If `deltaGas > 0`, we add it to `gasUsed` … We expect the base fee to be enough to cover most transactions without the need of additional gas.
> Finally, any remaining gas left is refunded as usual.

The documentation only describes the `delta_gas > 0` branch. The code implements only that branch:

```rust
// basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs  lines 66-81
#[cfg(not(feature = "unlimited_native"))]
{
    let delta_gas = if native_per_gas == 0 {
        0
    } else {
        (native_used / native_per_gas) as i64 - (gas_used as i64)
    };

    if delta_gas > 0 {
        // native > EVM gas: charge extra
        gas_used += delta_gas as u64;
    }
    // TODO: return delta_gas to gas_used?   ← acknowledged gap
}
```

When `delta_gas < 0` (native resource used is less than the EVM gas equivalent), `gas_used` is left at the full EVM gas value. The refund is then computed as:

```rust
let total_gas_refund = gas_limit - gas_used;   // line 83
```

Because `gas_used` was not reduced by `|delta_gas|`, the user receives a smaller refund than the native resource consumption would justify. The difference `gas_price * |delta_gas|` flows to the operator instead of back to the user.

This is called for every ZK L2 transaction via `ZkTransactionFlowOnlyEOA::before_refund` and for every L1 priority transaction via `process_l1_transaction`, both of which pass a non-zero `native_per_gas`:

- L2 ZK path: `context.native_per_gas` (derived from `gas_price / native_price`)
- L1 priority path: `native_per_gas` derived from `gas_price / L1_TX_NATIVE_PRICE`

The Ethereum transaction flow is unaffected because it always passes `native_per_gas = 0`, which forces `delta_gas = 0`.

---

### Impact Explanation

Every ZK L2 or L1 priority transaction where the actual native proving cost is lower than the EVM gas cost results in the user being overcharged. The overcharge per transaction is:

```
overcharge = gas_price * |delta_gas|
           = gas_price * (gas_used - native_used / native_per_gas)
```

The excess is credited to the operator (`coinbase`) rather than refunded to the sender. This is a direct, quantifiable loss of user funds on every affected transaction. Because native resource costs are not constant per EVM gas unit (they depend on the specific opcodes executed), the condition `delta_gas < 0` is routinely reachable for transactions that are EVM-gas-heavy but proving-cheap (e.g., many SSTORE operations, large calldata processing, or arithmetic-heavy contracts).

---

### Likelihood Explanation

The condition `native_used / native_per_gas < gas_used` is structurally common:

1. `native_per_gas` is a global ratio (`gas_price / native_price`), not a per-opcode ratio. Individual opcodes have varying native costs that do not track EVM gas 1:1.
2. Any transaction that spends EVM gas on operations with below-average native cost (e.g., SSTORE at 20 000 gas but relatively cheap to prove, or large memory operations) will produce `delta_gas < 0`.
3. The feature gate `#[cfg(not(feature = "unlimited_native"))]` means this path is active in all production builds (the `unlimited_native` feature is a testing convenience).
4. The TODO comment at line 80 is direct evidence that the asymmetry was noticed and left unresolved.

No special privileges, oracle manipulation, or unusual transaction structure is required. Any ordinary user submitting a ZK transaction triggers this path.

---

### Recommendation

Apply the symmetric correction: when `delta_gas < 0`, decrease `gas_used` by `|delta_gas|` (subject to the `minimal_gas_used` floor), so the user is refunded for the unused native resource equivalent:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else {
    // Return unused native resource equivalent to the user
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This mirrors the GeVault mitigation pattern: account for unused resources in the balance/refund calculation rather than silently discarding them.

### Proof of Concept

**Setup (ZK L2 transaction):**
- `gas_limit = 100_000`, `gas_price = 1_000`, `native_price = 10`
- `native_per_gas = 1_000 / 10 = 100`
- `native_limit = 100_000 * 100 = 10_000_000`

**Execution:**
- EVM gas consumed: `gas_used = 50_000` (ergs remaining → 50 000 gas left → 50 000 used)
- Native resource consumed: `native_used = 2_000_000` (transaction is proving-cheap)

**`compute_gas_refund` path:**
1. `gas_used = 50_000` (from ergs)
2. `delta_gas = (2_000_000 / 100) - 50_000 = 20_000 - 50_000 = -30_000`
3. `delta_gas < 0` → branch not taken, `gas_used` stays at `50_000`
4. `total_gas_refund = 100_000 - 50_000 = 50_000`
5. User refunded: `1_000 * 50_000 = 50_000_000`

**Correct behavior:**
- `gas_used` should be reduced to `max(20_000, minimal_gas_used)`
- User should be refunded: `1_000 * (100_000 - 20_000) = 80_000_000`
- **User loss per transaction: `30_000_000` units (30 000 × gas_price)**

The operator receives `1_000 * 50_000 = 50_000_000` instead of the correct `1_000 * 20_000 = 20_000_000`, capturing `30_000_000` units that belong to the user.
