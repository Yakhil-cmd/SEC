Audit Report

## Title
Missing Neuron Lock in SNS Governance `disburse_neuron` Enables Concurrent Stake Over-Drainage - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS governance `disburse_neuron` function performs two sequential async inter-canister ledger calls without first acquiring a per-neuron `NeuronInFlightCommand` lock. The IC scheduler can interleave a second concurrent `disburse_neuron` call for the same neuron at each `await` suspension point. Both calls read the same `cached_neuron_stake_e8s`, compute the same transfer amount, and both ledger transfers can succeed, draining more tokens from the neuron's subaccount than any single call was authorized to disburse.

## Finding Description
The `lock_neuron_for_command` helper exists in SNS governance and is used by other async operations, but is entirely absent from `disburse_neuron`.

`disburse_neuron` at [1](#0-0)  performs authorization and dissolved-state checks synchronously, then immediately issues two async ledger calls with no neuron-level guard:

- Transfer 1 (fee burn) at [2](#0-1) 
- Transfer 2 (stake disburse) at [3](#0-2) 

The `lock_neuron_for_command` function at [4](#0-3)  checks `in_flight_commands` and inserts a lock entry before returning a `LedgerUpdateLock` RAII guard. This is the exact mechanism designed to prevent concurrent interleaving, as documented in the proto definition. [5](#0-4) 

By contrast, NNS governance `disburse_neuron` acquires this lock before any async call: [6](#0-5) 

The exploit flow:
1. Two concurrent `manage_neuron { Disburse { amount: S/2 } }` ingress messages both pass `check_authorized` and `NeuronState::Dissolved` checks synchronously.
2. Both read the same `cached_neuron_stake_e8s = S` and compute `disburse_amount_e8s = S/2 - tx_fee`. [7](#0-6) 
3. Both suspend at Transfer 1's `await`; both fee-burn transfers complete independently.
4. Both proceed to Transfer 2; both succeed because the ledger subaccount balance is sufficient for each individual transfer.
5. Call A updates `cached_neuron_stake_e8s` to `S/2`; Call B then subtracts the same amount, reaching `0`. [8](#0-7) 

Result: the caller receives `≈ S` tokens while only one disburse of `S/2` was authorized, and `cached_neuron_stake_e8s` diverges from the actual ledger balance.

## Impact Explanation
This is a **High** severity finding. Any principal holding `NeuronPermissionType::Disburse` on a dissolved SNS neuron can drain the full neuron stake by submitting two concurrent half-stake disburse requests. This constitutes unauthorized drainage of canister-controlled SNS token funds with concrete, direct financial loss to the neuron owner or the SNS treasury. The impact matches: *"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds."* The impact is bounded to individual neuron stakes rather than global protocol insolvency, placing it in the High ($2,000–$10,000) range unless a neuron holds assets exceeding the Critical threshold.

## Likelihood Explanation
Any principal with `Disburse` permission on a dissolved SNS neuron can trigger this — no privileged role, key compromise, or subnet-majority is required. The attacker submits two `manage_neuron` ingress messages in rapid succession using any standard IC agent library. The IC's asynchronous execution model guarantees the interleaving window exists at every `await` point. The attack is straightforward to automate and repeatable across any SNS deployment.

## Recommendation
Acquire a `NeuronInFlightCommand` lock immediately after precondition checks and before the first async ledger call, mirroring the NNS governance pattern:

```rust
// After precondition checks, before any .await
let _neuron_lock = self.lock_neuron_for_command(
    id,
    NeuronInFlightCommand {
        timestamp: self.env.now(),
        command: Some(neuron_in_flight_command::Command::Disburse(disburse.clone())),
    },
)?;
```

This ensures a second concurrent call for the same neuron returns `NeuronLocked` immediately, preventing any interleaved ledger transfers. [4](#0-3) 

## Proof of Concept
**Setup**: A dissolved SNS neuron with stake `S` where the caller holds `NeuronPermissionType::Disburse`.

**Steps**:
1. Using an IC agent, concurrently submit two `manage_neuron { Disburse { amount: Some(S/2) } }` calls targeting the same neuron ID before the first message's `await` completes.
2. Observe both calls pass authorization and dissolved-state checks.
3. Observe both Transfer 1 (fee burn) and Transfer 2 (stake disburse) execute for both calls.
4. Verify the caller's ledger account received `≈ S` tokens (two successful transfers of `S/2 - tx_fee`).
5. Verify `cached_neuron_stake_e8s` is `0` and the neuron subaccount ledger balance is `≈ 2 × tx_fee`, confirming divergence.

A deterministic integration test using PocketIC can reproduce this by submitting two concurrent ingress messages and asserting the final ledger balance of the destination account exceeds `S/2 - tx_fee`. [9](#0-8)

    fn lock_neuron_for_command(
        &mut self,
        nid: &NeuronId,
        command: NeuronInFlightCommand,
    ) -> Result<LedgerUpdateLock, GovernanceError> {
        let nid = nid.to_string();
        if self.proto.in_flight_commands.contains_key(&nid) {
            return Err(GovernanceError::new_with_message(
                ErrorType::NeuronLocked,
                "Neuron has an ongoing operation.",
            ));
        }

        self.proto.in_flight_commands.insert(nid.clone(), command);

        Ok(LedgerUpdateLock { nid, gov: self })
