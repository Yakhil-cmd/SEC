### Title
Blocklist Bypass via Proxy/Intermediary Contract Allows Sanctioned Addresses to Receive ckETH/ckERC20 — (`rs/ethereum/cketh/minter/src/deposit.rs`)

### Summary
The `register_deposit_events()` function checks only `event.from_address()` (the Ethereum `msg.sender` of the helper contract call) against the blocklist. A sanctioned address can route its deposit through any non-blocked intermediary contract, causing `from_address` to be the intermediary rather than the sanctioned address, bypassing the compliance check entirely.

### Finding Description
In `register_deposit_events()`:

```rust
if crate::blocklist::is_blocked(&event.from_address()) {
``` [1](#0-0) 

`from_address` is populated directly from the Ethereum event log topic, which records `msg.sender` of the helper contract call:

```solidity
emit ReceivedEthOrErc20(ZERO_ADDRESS, msg.sender, msg.value, principal, subaccount);
``` [2](#0-1) 

The log parser extracts this as `from_address`:

```rust
let from_address = parse_address(&entry.topics[2], event_source)?;
``` [3](#0-2) 

`is_blocked()` performs a binary search only on the address passed to it:

```rust
pub fn is_blocked(address: &Address) -> bool {
    ETH_ADDRESS_BLOCKLIST.binary_search(address).is_ok()
}
``` [4](#0-3) 

The blocklist comment states the invariant: *"ETH is not accepted from nor sent to addresses on this list."* [5](#0-4) 

This invariant is broken because the check only covers the direct caller of the helper contract, not the ultimate source of funds.

### Impact Explanation
A sanctioned address (e.g., a Tornado Cash-associated address on `ETH_ADDRESS_BLOCKLIST`) can receive ckETH or ckERC20 tokens on the Internet Computer by routing through a proxy contract. This constitutes a compliance bypass in the chain-fusion asset bridge, allowing sanctioned entities to convert ETH/ERC20 into IC-native wrapped assets.

### Likelihood Explanation
Trivially executable by any Ethereum user. Deploying a proxy contract costs a few dollars in gas. No privileged access, no key compromise, no governance manipulation required. The attacker only needs to be an unprivileged Ethereum account holder.

### Recommendation
The blocklist check is fundamentally limited to on-chain observable data. Mitigations include:
- Documenting this as a known limitation (the blocklist is best-effort, not enforcement-grade).
- If stricter enforcement is desired, consider rejecting deposits where the transaction `origin` (not `msg.sender`) is a blocked address — though this requires EVM RPC support for `tx.origin` in log context, which is not available in log events.
- Alternatively, accept that proxy-based bypasses are an inherent limitation of `msg.sender`-based blocklisting on Ethereum and scope the blocklist accordingly.

### Proof of Concept
Construct a state-machine test:
1. Create a `ReceivedEthEvent` with `from_address = proxy_address` (not on blocklist) and `principal = any_ic_principal`.
2. Call `register_deposit_events()` with this event.
3. Assert the deposit is accepted (not marked `InvalidDeposit`) and proceeds to minting.
4. Confirm that if `from_address` were set to a known blocked address (e.g., `SAMPLE_BLOCKED_ADDRESS`), the deposit would be rejected. [6](#0-5) 

This directly demonstrates that the blocklist check is bypassable by any one-hop indirection on Ethereum.
