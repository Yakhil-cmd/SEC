### Title
Incorrect Unaligned Offset Calculation in `read_memory_as_u8` Causes Integer Underflow and Panic - (File: `callable_oracles/src/utils/evaluate.rs`)

### Summary
The helper function `read_memory_as_u8` in the oracle evaluation layer contains an incorrect formula for computing the number of bytes remaining in the current 4-byte word when the read offset is unaligned. The expression `4 - offset` is used instead of the correct `4 - (offset % 4)`. For any unaligned offset `>= 4`, this produces a `u32` integer underflow, yielding a near-maximal value. This causes a subsequent slice index to exceed the 4-byte word boundary, triggering a panic in the prover. An attacker who can influence the memory layout of the RISC-V execution environment (e.g., via a blob transaction or hash-to-prime oracle path) can trigger this panic, halting the proving pipeline.

### Finding Description

In `callable_oracles/src/utils/evaluate.rs`, the function `read_memory_as_u8` handles unaligned memory reads by first consuming the partial bytes at the start of the first 4-byte word:

```rust
if !offset.is_multiple_of(4) {
    let max_take_bytes = 4 - offset;          // BUG: should be 4 - (offset % 4)
    let take_bytes = std::cmp::min(max_take_bytes, len);
    let aligned = (offset >> 2) << 2;
    let value = memory.get(aligned as u64, AccessType::MemLoad, &mut trap);
    ...
    let value = value.to_le_bytes();
    result.extend_from_slice(&value[offset as usize % 4..][..take_bytes as usize]);
    offset += max_take_bytes;
    len -= take_bytes;
}
``` [1](#0-0) 

The correct formula for the number of bytes remaining in the current aligned word is `4 - (offset % 4)`. The code uses `4 - offset` instead, which is only correct when `offset < 4`.

**Concrete underflow scenario:**

| `offset` | `offset % 4` | Correct `max_take_bytes` | Actual `max_take_bytes` (`4 - offset`) |
|---|---|---|---|
| 1 | 1 | 3 | 3 ✓ |
| 3 | 3 | 1 | 1 ✓ |
| 5 | 1 | 3 | `u32::MAX` ✗ |
| 6 | 2 | 2 | `u32::MAX - 1` ✗ |
| 9 | 1 | 3 | `u32::MAX - 4` ✗ |

When `offset = 5`:
- `max_take_bytes = 4u32 - 5u32` → wraps to `u32::MAX` (release) or panics (debug)
- `take_bytes = min(u32::MAX, len) = len`
- `value[offset % 4..][..take_bytes]` = `value[1..][..len]`
- `value` is only 4 bytes; `value[1..]` is 3 bytes; if `len > 3`, this panics with an out-of-bounds slice index

The function is explicitly designed to handle unaligned offsets (the `if !offset.is_multiple_of(4)` guard exists for this purpose), and its callers in the blob KZG commitment and hash-to-prime oracle paths pass real RISC-V memory addresses that are routinely `>= 4`. [2](#0-1) [3](#0-2) 

### Impact Explanation

The `callable_oracles` crate is part of the proving pipeline. When the prover processes a block containing a transaction that exercises the blob KZG commitment or hash-to-prime oracle path with data stored at an unaligned RISC-V memory address `>= 4`, `read_memory_as_u8` is called with that address. The underflow causes either:

1. **Panic (debug/test builds):** immediate abort of the prover process.
2. **Out-of-bounds slice panic (release builds):** `value[1..][..len]` panics when `len > 3`, which is the common case for any non-trivial read.

Either outcome halts the proving pipeline for the affected block. Because the prover cannot produce a proof for the block, the block cannot be finalized on L1. This is a **valid-execution unprovability** / **denial-of-service** impact: a transaction that executes correctly in forward execution cannot be proven.

### Likelihood Explanation

Any unaligned RISC-V memory address `>= 4` triggers the bug. Real RISC-V programs routinely place data at addresses such as 5, 6, 9, etc. (e.g., after a 4-byte-aligned struct with a 1-byte field). An attacker who can submit a blob transaction or trigger the hash-to-prime oracle path with data whose RISC-V pointer is unaligned and `>= 4` can reliably trigger the panic. The entry path is an unprivileged EIP-4844 blob transaction or any transaction that exercises the affected oracle.

### Recommendation

Replace the incorrect formula with the modulo-based one:

```rust
// Before (incorrect):
let max_take_bytes = 4 - offset;

// After (correct):
let max_take_bytes = 4 - (offset % 4);
```

This ensures the number of bytes remaining in the current aligned word is computed correctly for any offset value.

### Proof of Concept

```
offset = 5, len = 10

max_take_bytes = 4u32 - 5u32 = 4294967295  (u32 wrapping underflow)
take_bytes     = min(4294967295, 10) = 10
aligned        = (5 >> 2) << 2 = 4
value          = memory.get(4)  → [b0, b1, b2, b3]  (4 bytes)
value[offset % 4 ..][..take_bytes]
  = value[1..][..10]
  = slice of length 3 indexed with length 10
  → PANIC: index out of bounds: the len is 3 but the index is 10
``` [1](#0-0)
