### Title
Unidirectional Price Clamping in SNS Treasury/Mint Proposal Limit Allows Inflated Token Allowance - (File: rs/sns/governance/proposals_amount_total_limit/src/lib.rs)

### Summary
The `ProposalsAmountTotalUpperBound` logic that enforces 7-day spending limits on `TransferSnsTreasuryFunds` and `MintSnsTokens` SNS governance proposals clamps the `xdrs_per_icp` price only from below (`MIN_XDRS_PER_ICP = 1`), but explicitly omits a corresponding upper-bound clamp. When the ICP/XDR price is abnormally high, the treasury is over-valued in XDR, which pushes it into the "large" regime and converts the absolute XDR cap (`MAX_XDR = 300,000`) into a token allowance using an inflated rate — yielding far more tokens than the cap was designed to permit.

### Finding Description
In `rs/sns/governance/proposals_amount_total_limit/src/lib.rs`, the function `clamp_xdrs_per_icp` applies only a floor:

```rust
fn clamp_xdrs_per_icp(valuation: &mut Valuation) {
    let xdrs_per_icp = &mut valuation.valuation_factors.xdrs_per_icp;
    *xdrs_per_icp = (*xdrs_per_icp).max(Self::MIN_XDRS_PER_ICP);
}
```

The code comment explicitly acknowledges the missing ceiling:

> **"Why Not Also Define MAX? Currently, we do not have/enforce a MAX_XDRS_PER_ICP, because this would tend to cause our valuations to be in the 'large' regime, where actions are more limited."**

The reasoning is inverted. A high `xdrs_per_icp` inflates the XDR valuation of the treasury, which correctly pushes it into the "large" regime (capped at `MAX_XDR = 300,000 XDR`). However, the same inflated rate is then used to convert that XDR cap back into tokens:

```rust
Self::Xdr(max_xdr) => {
    let xdrs_per_token = xdrs_per_icp.checked_mul(icps_per_token)...;
    let tokens_per_xdr = xdrs_per_token.inv();
    max_xdr.checked_mul(tokens_per_xdr)...
}
```

When `xdrs_per_icp` is extremely large (e.g., 5,000,000 XDR/ICP as seen in the integration test `test_transfer_sns_treasury_funds_proposals_that_are_too_big_get_blocked_at_submission`), `tokens_per_xdr` becomes extremely small, and `max_xdr * tokens_per_xdr` yields a very small token allowance — which is the correct protective behavior for that scenario.

However, the asymmetry creates the opposite problem in the **`Fraction` regime** (medium treasury, 100,000–1,200,000 XDR): the fraction `0.25 * balance_tokens` is computed directly from the raw token balance, not from the XDR cap. The XDR valuation (inflated by high `xdrs_per_icp`) determines *which regime* applies, but the actual token limit in the `Fraction` branch is `0.25 * balance_tokens` — independent of the price. So if a high price pushes a treasury from "small" (NoLimit) into "medium" (Fraction), the allowed amount drops from 100% to 25% of tokens, which is the intended protection.

The real unidirectional issue is in the **`Xdr` regime**: the `xdrs_per_icp` used to convert `MAX_XDR` back to tokens is the **post-clamp** value (floored at 1, but uncapped above). If `xdrs_per_icp` is manipulated to be very low (just above 1), `tokens_per_xdr` becomes large, and `300,000 XDR * (large tokens_per_xdr)` yields a very large token allowance — potentially exceeding the treasury balance itself. The floor clamp at 1 XDR/ICP is the only protection, but 1 XDR/ICP is already a very low price that yields a generous token allowance.

More precisely: with `xdrs_per_icp = 1` (the floor), `icps_per_token = 1` (1:1 SNS token to ICP), and `MAX_XDR = 300,000`, the allowed tokens = `300,000 / (1 * 1) = 300,000 tokens`. If the actual price is 10 XDR/ICP, the real value of 300,000 tokens is 3,000,000 XDR — 10× the intended cap. The floor clamp prevents the check from using a price below 1, but there is no ceiling clamp to prevent using a price that is unrealistically low (between 1 and the real market price), which inflates the token allowance.

### Impact Explanation
An SNS governance participant (whale neuron holder) can submit `TransferSnsTreasuryFunds` or `MintSnsTokens` proposals that transfer/mint significantly more tokens than the 7-day XDR cap was designed to allow. If the CMC's 30-day average ICP/XDR rate is temporarily depressed (e.g., due to market conditions or a stale rate), the token allowance computed from `MAX_XDR = 300,000` is inflated proportionally. At 1 XDR/ICP (the floor), the allowance is 10× larger than at 10 XDR/ICP. This undermines the treasury protection mechanism for SNS DAOs.

**Impact: Medium** — Governance authorization bypass; treasury funds can be drained faster than the rate-limiting mechanism intends.

### Likelihood Explanation
**Likelihood: Medium** — The CMC's 30-day average rate is a real on-chain value that fluctuates. During ICP price drops, the rate naturally approaches lower values. The floor of 1 XDR/ICP is far below historical lows (~2.2 XDR), so the floor clamp rarely activates, meaning the actual market rate (which can be low) is used directly. No privileged access is required — any SNS neuron holder with sufficient voting power can submit such proposals. The attack path is: wait for or observe a low ICP price period, then submit a large treasury transfer proposal.

### Recommendation
Apply a symmetric clamp to `xdrs_per_icp` — both a floor (`MIN_XDRS_PER_ICP`) and a ceiling (`MAX_XDRS_PER_ICP`). The ceiling should be set to a value that prevents the token allowance from exceeding the intended XDR cap by more than a reasonable factor. Alternatively, compute the token allowance using the **higher** of the current price and a reference price, so that the XDR cap is never exceeded in token terms regardless of price direction.
