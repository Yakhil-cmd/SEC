Audit Report

## Title
Unprivileged Caller Can Prematurely Fail SNS Upgrade Proposals via `fail_stuck_upgrade_in_progress` - (File: rs/sns/governance/canister/canister.rs)

## Summary
The `fail_stuck_upgrade_in_progress` update method on the SNS governance canister exposes no caller access control, allowing any unprivileged ingress sender to invoke it once `mark_failed_at_seconds` has elapsed. If an upgrade completes successfully but the periodic `check_upgrade_status` task has not yet committed the confirmation (i.e., the upgrade completed within the last periodic-task interval before the deadline), an attacker calling this method will mark the proposal as failed and leave `deployed_version` stale, creating a persistent governance state divergence from the actual running canister.

## Finding Description
The canister-level handler at `rs/sns/governance/canister/canister.rs` L526‚Äì535 has no caller check:

```rust
#[update]
fn fail_stuck_upgrade_in_progress(
    request: FailStuckUpgradeInProgressRequest,
) -> FailStuckUpgradeInProgressResponse {
    log!(INFO, "fail_stuck_upgrade_in_progress");
    FailStuckUpgradeInProgressResponse::from(governance_mut().fail_stuck_upgrade_in_progress(
        sns_gov_pb::FailStuckUpgradeInProgressRequest::from(request),
    ))
}
```

The internal implementation at `rs/sns/governance/src/governance.rs` L6342 acts unconditionally once the deadline passes:

```rust
if now > pending_version.mark_failed_at_seconds {
    ...
    self.complete_sns_upgrade_to_next_version(
        pending_version.proposal_id,
        status,
        message,
        None,   // deployed_version NOT updated
    );
}
```

`complete_sns_upgrade_to_next_version` at L6309‚Äì6313 clears `pending_version` and, because `deployed_version` is `None`, does not advance `proto.deployed_version`:

```rust
self.proto.pending_version = None;
if let Some(deployed_version) = deployed_version {
    self.proto.deployed_version.replace(deployed_version);
}
```

The deadline is set to only 5 minutes (`mark_failed_at_seconds: self.env.now() + 5 * 60` at L6638), while the periodic task that confirms upgrade completion runs every 10 seconds (`RUN_PERIODIC_TASKS_INTERVAL = Duration::from_secs(10)` at L76).

The race window arises because `check_upgrade_status` is async: it sets `checking_upgrade_lock`, issues an inter-canister call to root's `get_sns_canisters_summary`, and only commits the success/failure result when the response arrives. On the IC, while the canister is awaiting that response, it can process other ingress messages. If `mark_failed_at_seconds` elapses during this await window, an attacker's `fail_stuck_upgrade_in_progress` call will clear `pending_version` and mark the proposal failed. When the root response arrives and the `check_upgrade_status` continuation attempts `self.proto.pending_version.as_mut().unwrap()` (L6177‚Äì6181), it will trap (unwrap on None), rolling back only the continuation's state ‚Äî but `fail_stuck_upgrade_in_progress`'s committed changes remain. The code itself acknowledges the missing guard with a TODO comment at L6337‚Äì6338:

```rust
// Maybe, we should look at the checking_upgrade_lock field and only
// proceed if it is false, or the request has force set to true.
```

## Impact Explanation
This matches the allowed impact: **"Significant SNS security impact with concrete user or protocol harm."** Concretely: (1) a successfully executed upgrade proposal is permanently recorded as failed with no recourse; (2) `proto.deployed_version` remains at the pre-upgrade value while the canister on-chain runs the new WASM; (3) `initiate_upgrade_if_sns_behind_target_version` uses the stale `deployed_version` to compute upgrade paths, potentially triggering a redundant re-upgrade of the already-upgraded canister; (4) the upgrade journal records an incorrect `ExternalFailure` outcome. This is a persistent, irrecoverable governance state corruption for the targeted SNS instance. Severity: **High** (lower bound), or **Medium** given the narrow timing constraint.

## Likelihood Explanation
The attacker requires no special privileges ‚Äî any principal including anonymous can send an ingress update call. The `pending_version.mark_failed_at_seconds` is publicly readable via the `get_running_sns_version` query. The attack window is the overlap of: (a) `now > mark_failed_at_seconds` (after 5 minutes), and (b) `check_upgrade_status` is mid-await on root's response. This window is narrow (seconds), requires the upgrade to complete close to the 5-minute deadline, and demands precise timing. Likelihood: **Medium-Low** ‚Äî no privilege required, but strict timing conditions must be met.

## Recommendation
Add a caller access control check at the canister entry point in `rs/sns/governance/canister/canister.rs` before delegating to the internal implementation. At minimum, restrict to canister controllers:

```rust
#[update]
fn fail_stuck_upgrade_in_progress(
    request: FailStuckUpgradeInProgressRequest,
) -> FailStuckUpgradeInProgressResponse {
    let caller = ic_cdk::api::caller();
    assert!(
        ic_cdk::api::is_controller(&caller),
        "Caller is not a controller"
    );
    ...
}
```

Additionally, implement the acknowledged TODO at L6337‚Äì6338 in `rs/sns/governance/src/governance.rs`: check `checking_upgrade_lock != 0` and refuse to proceed (or require a `force` flag) to prevent racing with an in-flight `check_upgrade_status` async call.

## Proof of Concept
1. Submit an `UpgradeSnsToNextVersion` proposal to an SNS and vote it through; observe `pending_version.mark_failed_at_seconds` via `get_running_sns_version`.
2. Wait until `now > mark_failed_at_seconds` (5 minutes after upgrade initiation).
3. Arrange for `check_upgrade_status` to be mid-await (e.g., by timing the call during the inter-canister call window to root, or by using a PocketIC integration test that advances time to just past the deadline while a `get_sns_canisters_summary` call is in flight).
4. From any principal (including anonymous), send: `fail_stuck_upgrade_in_progress({})`.
5. Observe: `pending_version` is cleared, the proposal is marked failed (`ExternalFailure`), and `deployed_version` remains at the pre-upgrade value despite the canister running the new WASM.
6. A deterministic integration test can reproduce this by: (a) constructing a `Governance` instance with `pending_version.mark_failed_at_seconds = now - 1` and a mock root that delays its `get_sns_canisters_summary` response, (b) calling `check_upgrade_status` to initiate the async call, (c) interleaving `fail_stuck_upgrade_in_progress`, and (d) asserting the proposal is failed and `deployed_version` is stale.
