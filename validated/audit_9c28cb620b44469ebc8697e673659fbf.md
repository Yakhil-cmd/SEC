### Title
Wrong Field Used in `max_neurons_fund_participation_icp_e8s` Assignment Causes Incorrect Neurons' Fund Participation Cap - (File: rs/sns/swap/src/neurons_fund.rs)

### Summary
In `rs/sns/swap/src/neurons_fund.rs`, the `From<ValidatedNeuronsFundParticipationConstraints<F>> for NeuronsFundParticipationConstraintsPb` conversion implementation assigns `value.min_direct_participation_threshold_icp_e8s` to the `max_neurons_fund_participation_icp_e8s` field instead of `value.max_neurons_fund_participation_icp_e8s`. This is a copy-paste error where one field is correctly set but the other related field is silently set to the wrong source value — the same class of incomplete/incorrect state update as the BloomPool M-02 finding.

### Finding Description

In `rs/sns/swap/src/neurons_fund.rs` lines 502–527:

```rust
impl<F> From<ValidatedNeuronsFundParticipationConstraints<F>>
    for NeuronsFundParticipationConstraintsPb
where
    F: IdealMatchingFunction,
{
    fn from(value: ValidatedNeuronsFundParticipationConstraints<F>) -> Self {
        Self {
            min_direct_participation_threshold_icp_e8s: Some(
                value.min_direct_participation_threshold_icp_e8s,   // ✓ correct
            ),
            max_neurons_fund_participation_icp_e8s: Some(
                value.min_direct_participation_threshold_icp_e8s,   // ✗ BUG: should be
                                                                     //   value.max_neurons_fund_participation_icp_e8s
            ),
            ...
        }
    }
}
``` [1](#0-0) 

The two fields being set are semantically distinct:
- `min_direct_participation_threshold_icp_e8s`: the minimum direct participation amount below which the Neurons' Fund does not participate at all (e.g., 36,000 ICP e8s).
- `max_neurons_fund_participation_icp_e8s`: the maximum amount the Neurons' Fund will contribute to the swap (e.g., 100,000 ICP e8s).

Both are set from `value.min_direct_participation_threshold_icp_e8s`, so `max_neurons_fund_participation_icp_e8s` ends up holding the minimum threshold value instead of the actual maximum participation cap.

### Impact Explanation

When this `From` conversion is invoked, the resulting `NeuronsFundParticipationConstraintsPb` carries a `max_neurons_fund_participation_icp_e8s` that equals `min_direct_participation_threshold_icp_e8s` — a value that is typically orders of magnitude smaller than the true maximum.

This incorrect cap is consumed in `rs/sns/swap/src/swap.rs` inside `update_total_participation_amounts`:

```rust
neurons_fund_participation_icp_e8s.min(
    constraints
        .max_neurons_fund_participation_icp_e8s
        .unwrap_or(u64::MAX),
)
``` [2](#0-1) 

With the wrong cap, the Neurons' Fund participation amount reported by the swap canister is artificially clamped to the minimum threshold value. Consequences:
1. **Under-counting of Neurons' Fund participation**: `neurons_fund_participation_icp_e8s` stored in swap state is too low, causing `current_total_participation_e8s()` to under-report total participation.
2. **Swap may fail to commit**: If the under-counted total falls below `min_direct_participation_icp_e8s`, the swap will abort even though the true combined participation (direct + Neurons' Fund) would have been sufficient to commit.
3. **Incorrect `available_direct_participation_e8s`**: Downstream calculations that depend on the total participation amount are skewed, potentially allowing over-subscription of direct participation slots. [3](#0-2) 

### Likelihood Explanation

Any SNS swap that uses the Neurons' Fund matched-funding scheme and triggers this `From` conversion path is affected. The conversion is reachable whenever `ValidatedNeuronsFundParticipationConstraints` is serialized back to protobuf (e.g., during canister upgrades, state export, or any code path that calls `.into()` on the validated type). Any unprivileged user calling `refresh_buyer_tokens` triggers `update_total_participation_amounts`, which reads the corrupted cap. No privileged access is required.

### Recommendation

Change line 513 from:
```rust
max_neurons_fund_participation_icp_e8s: Some(
    value.min_direct_participation_threshold_icp_e8s,
),
```
to:
```rust
max_neurons_fund_participation_icp_e8s: Some(
    value.max_neurons_fund_participation_icp_e8s,
),
```

Add a unit test asserting that a round-trip conversion `ValidatedNeuronsFundParticipationConstraints → NeuronsFundParticipationConstraintsPb → ValidatedNeuronsFundParticipationConstraints` preserves both fields independently.

### Proof of Concept

1. Create a `ValidatedNeuronsFundParticipationConstraints` with:
   - `min_direct_participation_threshold_icp_e8s = 36_000 * E8`
   - `max_neurons_fund_participation_icp_e8s = 100_000 * E8`
2. Call `NeuronsFundParticipationConstraintsPb::from(constraints)`.
3. Observe that the resulting protobuf has `max_neurons_fund_participation_icp_e8s = 36_000 * E8` (the minimum threshold) instead of `100_000 * E8`.
4. Feed this protobuf into an SNS Swap's `Init` and call `refresh_buyer_tokens` with direct participation of `50_000 * E8`.
5. Observe that `neurons_fund_participation_icp_e8s` is capped at `36_000 * E8` instead of the correct matched amount, causing `current_total_participation_e8s()` to under-report and potentially preventing the swap from committing. [4](#0-3) [5](#0-4)
