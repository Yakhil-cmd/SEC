### Title
Multiple XCC Precompile Calls in One EVM Transaction Permanently Freeze User wNEAR via Stale Router-Version Check - (File: engine-precompiles/src/xcc.rs)

### Summary

The XCC precompile reads `get_code_version_of_address` from NEAR storage during EVM execution to decide whether to charge `STORAGE_AMOUNT` (2 NEAR) for a new router deployment. Because the storage key is only written back asynchronously via the `factory_update_address_version` NEAR promise callback, every XCC precompile invocation within the same EVM transaction sees the same stale `None` value and charges `STORAGE_AMOUNT` again. The excess wNEAR is transferred from the user to the engine's implicit EVM address during EVM execution; when the second promise chain's `CreateAccount` action fails (the router already exists from the first chain), the downstream `withdraw_wnear_to_router` callback aborts, and the wNEAR is permanently frozen in the engine's implicit address with no recovery path.

### Finding Description

In `engine-precompiles/src/xcc.rs`, the `run_with_handle` method of `CrossContractCall` computes `required_near` by checking whether a router contract has already been deployed for the calling EVM address:

```rust
let required_near =
    match state::get_code_version_of_address(&self.io, &Address::new(sender)) {
        None => attached_near + state::STORAGE_AMOUNT,
        Some(_) => attached_near,
    };
``` [1](#0-0) 

If `required_near != ZERO_YOCTO`, the precompile immediately executes a synchronous EVM sub-call to the wNEAR ERC-20 contract to `transferFrom` the user to the engine's implicit address:

```rust
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
``` [2](#0-1) 

The storage key that records the deployed version is only written by `factory_update_address_version`, which runs as an asynchronous NEAR promise callback **after** the EVM transaction completes:

```rust
xcc::set_code_version_of_address(&mut io, &args.address, args.version);
``` [3](#0-2) 

Therefore, if a Solidity contract calls the XCC precompile twice within one EVM transaction while the caller has no router deployed, both calls observe `None` from `get_code_version_of_address` and each deducts `STORAGE_AMOUNT` (2 NEAR) from the user's wNEAR balance synchronously.

After EVM execution, `filter_promises_from_logs` processes the two XCC logs and chains their promise trees sequentially via `previous_promise`: [4](#0-3) 

The second promise tree's deployment batch includes `CreateAccount`, which fails because the router account was already created by the first chain. `withdraw_wnear_to_router` is a callback of that deployment and guards itself:

```rust
if matches!(handler.promise_result_check(), Some(false)) {
    return Err(b"ERR_CALLBACK_OF_FAILED_PROMISE".into());
}
``` [5](#0-4) 

Because `withdraw_wnear_to_router` aborts, the second `STORAGE_AMOUNT` worth of wNEAR that was already transferred to the engine's implicit EVM address during EVM execution is never forwarded to the router and is never refunded to the user. The engine's implicit address is derived deterministically from the NEAR account ID and has no accessible private key, so the wNEAR is permanently frozen. [6](#0-5) 

### Impact Explanation

**Critical — Permanent freezing of user funds.**

For every extra XCC precompile call made within one EVM transaction when the caller has no router deployed, the user permanently loses `STORAGE_AMOUNT` = 2 NEAR worth of wNEAR. The wNEAR is transferred synchronously during EVM execution and cannot be recovered because the downstream promise chain that would have forwarded it aborts, and the engine's implicit EVM address has no accessible key. [7](#0-6) 

### Likelihood Explanation

**Medium.** Making multiple XCC calls in one EVM transaction is a natural and documented use case (e.g., a Solidity contract that fans out to two different NEAR contracts). The bug only triggers on the first-ever XCC call for a given EVM address (before the router is deployed). Any user who writes a multi-XCC Solidity contract and calls it before their router exists will silently lose 2 NEAR per extra call. No special privileges or adversarial intent are required. [8](#0-7) 

### Recommendation

Track a **pending deployment flag** in EVM-execution-time state (e.g., a transient in-memory set keyed by sender address within the current `run_with_handle` call chain, or by writing a provisional storage key that is readable within the same NEAR function call). Before charging `STORAGE_AMOUNT`, check both the persisted `get_code_version_of_address` **and** whether a deployment has already been scheduled for this sender in the current EVM transaction. Charge `STORAGE_AMOUNT` at most once per sender per EVM transaction.

Alternatively, after the first XCC call schedules a deployment, write a provisional "pending" version to storage that subsequent calls within the same EVM transaction can observe, preventing the duplicate charge. [9](#0-8) 

### Proof of Concept

1. Deploy a Solidity contract `MultiXCC` on Aurora that calls the XCC precompile (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`) twice with two distinct `CrossContractCallArgs::Eager` payloads.
2. Ensure the caller EVM address has never used XCC before (no router deployed; `get_code_version_of_address` returns `None`).
3. Approve the XCC precompile to spend at least `2 × STORAGE_AMOUNT` = 4 NEAR worth of wNEAR from the caller.
4. Call `MultiXCC` in one EVM transaction.
5. Observe: 4 NEAR worth of wNEAR is deducted from the caller during EVM execution.
6. After NEAR promise resolution: the first promise chain succeeds (router deployed, 2 NEAR refunded to engine). The second promise chain's `CreateAccount` fails; `withdraw_wnear_to_router` aborts; the second 2 NEAR worth of wNEAR remains in the engine's implicit EVM address permanently.
7. The caller's net loss is `STORAGE_AMOUNT` = 2 NEAR worth of wNEAR with no recovery mechanism. [1](#0-0) [10](#0-9) [11](#0-10)
