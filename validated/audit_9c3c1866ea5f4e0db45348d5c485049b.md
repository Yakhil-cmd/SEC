### Title
Silo Mode Fixed-Gas Fee Fully Bypassed via Zero `max_fee_per_gas` - (`engine/src/engine.rs`)

---

### Summary

In Silo mode, the Aurora Engine charges a fixed gas amount (`fixed_gas`) per transaction. However, the actual ETH fee collected is `fixed_gas × effective_gas_price`, where `effective_gas_price` is derived entirely from the user-supplied `max_fee_per_gas`. Because `block_base_fee_per_gas` is hardcoded to `0` and no minimum gas price is enforced in Silo mode, any whitelisted user can submit a transaction with `max_fee_per_gas = 0`, making `effective_gas_price = 0` and `prepaid_amount = 0`. The user executes transactions for free, and the relayer collects no fee reward.

---

### Finding Description

`block_base_fee_per_gas` is permanently hardcoded to `U256::zero()` in the Aurora Engine: [1](#0-0) 

In `charge_gas`, the early-return guard that skips fee collection requires `fixed_gas.is_none()`: [2](#0-1) 

When Silo mode is active (`fixed_gas = Some(...)`), this guard is bypassed and execution continues. The `effective_gas_price` is then computed as:

```
priority_fee_per_gas = min(max_priority_fee_per_gas, max_fee_per_gas - 0)
effective_gas_price  = priority_fee_per_gas + 0
``` [3](#0-2) 

If the user sets `max_fee_per_gas = 0` (and `max_priority_fee_per_gas = 0`), then `effective_gas_price = 0`. The prepaid amount becomes:

```
prepaid_amount = fixed_gas * 0 = 0
``` [4](#0-3) 

The sender's balance is not reduced. The only guard against zero fees in the non-Silo path is the early return at line 476, which is explicitly skipped when `fixed_gas` is set. There is no check that `max_fee_per_gas > 0` when Silo mode is active.

The `refund_unused_gas` function also short-circuits when `effective_gas_price` is zero, so the relayer receives no reward either: [5](#0-4) 

The full submit path reads `fixed_gas` from storage and passes it to `charge_gas` without any minimum price enforcement: [6](#0-5) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

The Silo operator configures `fixed_gas` so that the relayer collects `fixed_gas × gas_price` ETH per transaction as a fee reward. Tests confirm this design intent: [7](#0-6) 

By setting `max_fee_per_gas = 0`, a whitelisted user executes transactions at zero cost. The relayer receives nothing. The Silo operator's entire fee revenue stream is bypassed for every transaction submitted with a zero gas price.

---

### Likelihood Explanation

**High.** Any user who is already on the Silo whitelist (a prerequisite to submit transactions at all) can trivially craft a transaction with `max_fee_per_gas = 0`. This requires no special privileges beyond whitelist membership, no external oracle, and no admin compromise. The EVM transaction format allows `max_fee_per_gas = 0` by design, and the engine imposes no floor check in Silo mode. The only existing guard (`max_fee_per_gas < block_base_fee_per_gas`) never triggers because `block_base_fee_per_gas` is always 0. [8](#0-7) 

---

### Recommendation

In `charge_gas`, when `fixed_gas` is `Some(...)`, enforce that `effective_gas_price > 0` (or equivalently that `max_fee_per_gas > 0`) before proceeding. Return a `GasPaymentError` (e.g., a new `ZeroGasPriceInSiloMode` variant) if the condition is violated. This mirrors the recommendation in the original report: prohibit the operation when the fee condition cannot be satisfied rather than silently falling back to zero cost.

---

### Proof of Concept

1. Deploy Aurora Engine with Silo mode enabled: `set_silo_params(Some(SiloParamsArgs { fixed_gas: EthGas::new(1_000_000), erc20_fallback_address: ... }))`.
2. Add a NEAR account and EVM address to the whitelists.
3. Submit a transaction signed by that address with `max_fee_per_gas = 0` and `max_priority_fee_per_gas = 0` (e.g., a simple ETH transfer).
4. Observe that `charge_gas` computes `effective_gas_price = 0`, `prepaid_amount = 0`, and the sender's balance is unchanged by the fee. The relayer's balance also remains unchanged.
5. Compare with a transaction using `max_fee_per_gas = 1`: the sender's balance decreases by `fixed_gas × 1 = 1_000_000 wei` and the relayer receives that amount.

The existing test `test_address_transfer_success` demonstrates the normal fee path: [9](#0-8) 

The same test with `gas_price = 0` instead of `TWO_GAS_PRICE` would show zero fee deduction, confirming the bypass.
