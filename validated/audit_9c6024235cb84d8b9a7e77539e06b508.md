### Title
Unchecked Exit Precompile `call` Return Value After Irreversible Token Burn Causes Permanent Fund Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first burning the caller's ERC-20 tokens via `_burn`, then invoking the Aurora exit precompile through inline assembly. The assembly `call` opcode return value (`res`) is captured but **never checked**. If the precompile call fails for any reachable reason, the tokens are permanently destroyed with no corresponding NEAR or Ethereum withdrawal, resulting in irreversible fund loss.

---

### Finding Description

In `EvmErc20.sol` and `EvmErc20V2.sol`, both `withdrawToNear` and `withdrawToEthereum` follow the same unsafe pattern:

**Step 1 — Irreversible burn (state change committed):**
```solidity
_burn(_msgSender(), amount);
```

**Step 2 — Precompile call with unchecked return value:**
```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    // res is NEVER checked
}
```

The EVM `call` opcode returns `0` on failure and `1` on success. Since `res` is never inspected, any failure of the exit precompile causes the function to return successfully while the user's tokens have already been permanently destroyed.

This pattern appears in four locations:

- `EvmErc20.sol` `withdrawToNear` (line 61) — calls `exitToNear` precompile at `0xe9217bc7...`
- `EvmErc20.sol` `withdrawToEthereum` (line 74) — calls `exitToEthereum` precompile at `0xb0bd02f6...`
- `EvmErc20V2.sol` `withdrawToNear` (line 62) — same precompile
- `EvmErc20V2.sol` `withdrawToEthereum` (line 75) — same precompile

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) can return a non-success exit reason under several reachable conditions, including:

1. **Insufficient gas forwarded**: The precompile checks `required_gas > target_gas` and returns `ExitError::OutOfGas` if the gas forwarded via `call(gas(), ...)` is less than `EXIT_TO_NEAR_GAS`. When a calling contract forwards a capped gas amount (e.g., via `call{gas: N}(...)`), this is reachable without admin involvement.
2. **Precompile paused**: `PrecompileFlags::EXIT_TO_NEAR` or `EXIT_TO_ETHEREUM` can be set, causing the precompile dispatcher to return `ExitFatal::Other("ERR_PAUSED")`, which maps to `call` returning `0`.
3. **Token not registered**: `get_nep141_from_erc20` returns `ERR_TARGET_TOKEN_NOT_FOUND` if the ERC-20 address has no NEP-141 mapping, causing the precompile to error.

In all cases, the `_burn` has already executed and the ERC-20 supply is permanently reduced.

---

### Impact Explanation

**Critical — Permanent freezing/destruction of user funds.**

When the exit precompile call fails silently:
- The user's ERC-20 token balance is reduced to zero (burned).
- No NEAR-side `ft_transfer` or Ethereum-side `withdraw` promise is created.
- There is no refund path: the `error_refund` callback mechanism (`exit_to_near_precompile_callback`) is only triggered when the NEAR-side promise fails *after* the precompile succeeds. It is not triggered when the precompile itself fails silently at the EVM level.
- The tokens are permanently lost with no recovery mechanism.

---

### Likelihood Explanation

**Medium.** The most user-reachable trigger is the gas-forwarding scenario. Any contract that wraps `withdrawToNear` and forwards a capped gas amount (a common pattern in DeFi integrations) can cause the precompile to fail with `OutOfGas` while the burn has already committed. A user who calls `withdrawToNear` directly with a manually set low gas limit also hits this path. The paused-precompile trigger requires an admin action but is not the only trigger. The unregistered-token trigger is possible during any deployment race condition or misconfiguration.

---

### Recommendation

Check the `call` return value in assembly and revert if the precompile call fails. This ensures the `_burn` is atomically rolled back on precompile failure:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        if iszero(res) { revert(0, 0) }
    }
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. Alternatively, restructure the functions to call the precompile *before* burning, so a failed precompile call reverts the entire transaction without any state change.

---

### Proof of Concept

1. Deploy `EvmErc20` (or `EvmErc20V2`) as a bridged NEP-141 mirror token on Aurora.
2. Mint `1000` tokens to `victim`.
3. Deploy a wrapper contract `Caller` that calls `erc20.withdrawToNear{gas: N}(recipient, 1000)` where `N` is enough gas to execute `_burn` but less than `EXIT_TO_NEAR_GAS` (the fixed precompile cost checked at `engine-precompiles/src/native.rs` line 404–410).
4. `victim` calls `Caller.doWithdraw()`.
5. Inside `EvmErc20.withdrawToNear`: `_burn(victim, 1000)` executes — victim's balance drops to 0.
6. The assembly `call` to `0xe9217bc7...` is attempted; the precompile returns `ExitError::OutOfGas` because `target_gas < EXIT_TO_NEAR_GAS`; the `call` opcode returns `res = 0`.
7. `res` is never checked; `withdrawToNear` returns normally.
8. Result: victim's 1000 ERC-20 tokens are permanently burned. No NEAR-side `ft_transfer` promise was created. Funds are irrecoverably lost.
