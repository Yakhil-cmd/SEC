### Title
Inherited Hot Keys on `spawn_neuron` with New Controller Allow Unauthorized Governance Actions - (File: `rs/nns/governance/src/governance.rs`)

---

### Summary

When `spawn_neuron` is called with a `new_controller` that differs from the parent neuron's controller, the child neuron unconditionally inherits the parent's `hot_keys` list. The new controller never authorized these hot keys, yet they retain the ability to vote on governance proposals, set followees, and join/leave the Neurons' Fund on behalf of the new controller's neuron.

---

### Finding Description

`spawn_neuron` in `rs/nns/governance/src/governance.rs` allows a parent neuron's controller to spawn a child neuron and assign it to a completely different `new_controller`. The child neuron is built with `NeuronBuilder` and explicitly copies the parent's `hot_keys` and `followees` (including the `NeuronManagement`-topic followees) without clearing them:

```rust
let child_neuron = NeuronBuilder::new(
    child_nid,
    to_subaccount,
    child_controller,          // ← new, different controller
    ...
)
.with_spawn_at_timestamp_seconds(...)
.with_hot_keys(parent_neuron.hot_keys.clone())   // ← parent's hot keys copied verbatim
.with_followees(parent_neuron.followees.clone()) // ← parent's followees copied verbatim
.with_kyc_verified(parent_neuron.kyc_verified)
.with_maturity_e8s_equivalent(maturity_to_spawn)
.build();
``` [1](#0-0) 

The NNS neuron authorization model has two distinct authorization types relevant here:

1. **Controller** (`controller` field) — the sole principal that can perform privileged operations (add/remove hot keys, disburse, split, change dissolve delay).
2. **Hot keys** (`hot_keys` vec) — principals authorized to vote, follow, and join/leave the Neurons' Fund on behalf of the neuron. [2](#0-1) 

When `new_controller` is set, the controller is correctly updated to the new principal. However, the hot keys — which were set by the *old* controller — are never cleared. The new controller has no knowledge of these inherited hot keys and no mechanism to discover them proactively.

The same pattern exists in `split_neuron`, though there the controller does not change so the inherited hot keys are at least set by the same principal: [3](#0-2) 

The `spawn_neuron` case is the critical one: the parent controller can pre-load hot keys before spawning to a new controller.

---

### Impact Explanation

An attacker who controls a parent neuron can:

1. Add attacker-controlled principals as hot keys on the parent neuron.
2. Spawn a child neuron to a victim's principal (`new_controller = victim`).
3. The child neuron is created with the victim as controller but with the attacker's principals still listed as hot keys.
4. The attacker's hot-key principals can now:
   - **Vote on NNS proposals** using the victim's neuron's voting power, directly influencing governance outcomes.
   - **Set followees** on the victim's neuron (on any topic except `NeuronManagement`) to follow the attacker's neuron, effectively amplifying the attacker's voting power through liquid democracy.
   - **Join the Neurons' Fund** on behalf of the victim's neuron, committing the victim's ICP maturity to SNS swaps without the victim's consent.

The victim can remove the hot keys after the fact, but only if they are aware of their existence. There is no notification mechanism.

---

### Likelihood Explanation

The entry path is a standard unprivileged ingress call to `manage_neuron` with a `Spawn` command. Any NNS neuron controller can trigger this. The scenario is realistic in any context where neurons are transferred or sold (e.g., a marketplace or OTC trade where the seller spawns a child neuron to the buyer's principal). The attacker needs only to own a neuron with sufficient maturity — a common condition on the NNS.

---

### Recommendation

When `spawn_neuron` is called with a `new_controller` that differs from the parent neuron's controller, the child neuron should be created with an **empty hot keys list** (and optionally empty `NeuronManagement` followees). The new controller can add their own hot keys after taking ownership.

A targeted fix:

```rust
let hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]  // do not inherit hot keys when transferring to a new controller
};

let child_neuron = NeuronBuilder::new(...)
    .with_hot_keys(hot_keys)
    ...
```

The same fix should be applied to the `NeuronManagement`-topic followees in the `with_followees` call when `child_controller != parent_neuron.controller()`.

---

### Proof of Concept

1. Attacker A controls NNS neuron N with sufficient maturity. A also controls a second principal A'.
2. A calls `manage_neuron` → `Configure` → `AddHotKey(A')` on neuron N. Now `hot_keys = [A']`.
3. A calls `manage_neuron` → `Spawn { new_controller: Some(victim_principal), ... }`.
4. Governance creates child neuron C with `controller = victim_principal` and `hot_keys = [A']`. [4](#0-3) 

5. A' calls `manage_neuron` on neuron C with `RegisterVote` for any open proposal. The `is_authorized_to_vote` check passes because A' is in `hot_keys`: [5](#0-4) 

6. A' successfully votes on behalf of the victim's neuron C, without the victim's knowledge or consent.
7. A' also calls `manage_neuron` → `Follow` on neuron C, setting it to follow A's neuron on all non-`NeuronManagement` topics, permanently amplifying A's voting power through liquid democracy until the victim notices and removes the hot key.
