### Title
ICRC-1 Ledger `icrc2_approve` Deduplication Bypass via `fee=None` vs `fee=Some(F)` Produces Different Transaction Hashes — (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

### Summary
In the ICRC-1 ledger, `icrc2_approve_not_async` preserves the caller's `fee: Option<Nat>` as `None` or `Some(...)` in the internal `Operation::Approve`. Because the transaction hash is computed over the CBOR-serialized `FlattenedTransaction` where `fee` is skipped when `None`, two semantically identical approve calls with `fee=None` and `fee=Some(F)` produce distinct hashes and bypass the deduplication window, both succeeding and both debiting the ledger fee.

### Finding Description
`icrc2_approve_not_async` constructs the internal `Transaction` with `fee: arg.fee.map(|_| expected_fee_tokens)`, preserving the `None`/`Some` distinction from the caller's argument. The `FlattenedTransaction` serializer uses `#[serde(skip_serializing_if = "Option::is_none")]` on the `fee` field, so `fee=None` and `fee=Some(F)` produce different CBOR encodings and thus different SHA-256 hashes. The deduplication check in `apply_transaction` compares hashes, so the second call is not recognized as a duplicate. Both calls then execute `burn(from, fee.unwrap_or(effective_fee))`, which resolves to the same token amount in both cases, debiting the fee twice.

### Impact Explanation
An unprivileged caller can drain 2×F from their own account per `created_at_time` value by submitting two approve calls with identical parameters except `fee=None` vs `fee=Some(F)`. The allowance is set twice (idempotent on final state), but the fee is charged twice. Across many `created_at_time` values within the deduplication window, the attacker can multiply the fee drain. This affects all ICRC-1 ledger deployments (ckBTC, ckETH, etc.).

### Likelihood Explanation
The exploit requires only two sequential ingress calls to the public `icrc2_approve` endpoint with the same `created_at_time`. No privileged access, no key material, no governance majority is needed. It is trivially reproducible in a local replica.

### Recommendation
Normalize the fee field before constructing the internal `Transaction`. Either always set `fee: Some(expected_fee_tokens)` regardless of whether the caller passed `None` or `Some(F)`, or always set `fee: None` and rely on `effective_fee`. This ensures both call variants produce the same CBOR encoding and thus the same hash, making the second call correctly detected as a duplicate.

```rust
// Change from:
fee: arg.fee.map(|_| expected_fee_tokens),
// To:
fee: Some(expected_fee_tokens),
```

### Proof of Concept
1. Fund account A with balance ≥ 2×F.
2. Call `icrc2_approve` with `{spender: B, amount: X, created_at_time: T, fee: None}` → succeeds, balance becomes balance − F.
3. Call `icrc2_approve` with `{spender: B, amount: X, created_at_time: T, fee: Some(F)}` → succeeds (not detected as duplicate), balance becomes balance − 2×F.
4. Assert both calls returned `Ok(block_idx)` and that balance decreased by 2×F. [7](#0-6) [3](#0-2) [2](#0-1) [8](#0-7) [9](#0-8)
