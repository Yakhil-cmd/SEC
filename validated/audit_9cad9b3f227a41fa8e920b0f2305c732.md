Audit Report

## Title
Malfunctioning Archive Canister Permanently Blocks All Subsequent Archive Upgrades With No Removal Mechanism - (File: rs/ethereum/ledger-suite-orchestrator/src/scheduler/mod.rs)

## Summary
In `UpgradeLedgerSuiteSubtask::UpgradeArchives`, the archive upgrade loop uses `await?` to propagate the first error, aborting iteration over remaining archives. All three error variants that can arise from `upgrade_canister` (`StopCanisterError`, `UpgradeCanisterError`, `StartCanisterError`) unconditionally return `is_recoverable() = true`, causing the task to be rescheduled every 5 seconds indefinitely. There is no interface mechanism to remove or skip a broken archive canister, so archives after the broken one can never be upgraded through the LSO.

## Finding Description
At lines 415–417 of `rs/ethereum/ledger-suite-orchestrator/src/scheduler/mod.rs`, the loop:
```rust
for canister_id in archives {
    upgrade_canister::<Archive, _>(canister_id, compressed_wasm_hash, runtime)
        .await?;
}
```
aborts on the first `Err`, leaving all subsequent archive canisters unupgraded.

`upgrade_canister` (lines 1271–1303) performs three sequential management-canister calls, each mapped to an error variant: `StopCanisterError`, `UpgradeCanisterError`, `StartCanisterError`. All three are unconditionally `is_recoverable() = true` (lines 674–676), regardless of the underlying `CallError::Reason`. This contrasts with the standalone `is_recoverable(e: &CallError)` function (lines 688–695), which correctly returns `false` for `Reason::Rejected` and `Reason::InternalError`.

The `run_task` function (lines 185–200) installs a `scopeguard` that reschedules the task after `RETRY_FREQUENCY` (5 seconds) on any recoverable error. Because all archive upgrade errors are recoverable, the task retries forever, always failing at the same broken archive and never reaching subsequent ones.

The `UpgradeArg` Candid interface (lines 19–52 of `ledger_suite_orchestrator.did`) exposes no field to remove or skip a specific archive principal. `manage_ledger_suites` can only add suites, not remove individual archive entries from an existing suite.

## Impact Explanation
This is a High-severity impact under "Significant Chain Fusion, ck-token, ledger infrastructure security impact with concrete user or protocol harm." The LSO manages production ckERC20 tokens (ckUSDC, ckUSDT, ckETH) with real user funds. If any archive canister in a token's suite becomes permanently unupgradeable (e.g., due to a `pre_upgrade` trap or cycles exhaustion), all archives after it in the list are also permanently blocked from receiving upgrades through the LSO. If the archive wasm contains a critical security vulnerability, it cannot be patched via the LSO for the affected token's suite. The NNS retains the ability to bypass the LSO and call `install_code` directly, but this requires additional governance proposals and manual intervention, and the LSO's upgrade task queue remains permanently occupied with a non-progressing retry loop.

## Likelihood Explanation
Two realistic failure modes exist without any attacker action: (1) a bug in the archive wasm causing `pre_upgrade` to trap on instruction limit or deserialization failure — `CanisterError("canister trapped: ...")` is classified as non-recoverable by the inner `is_recoverable(CallError)` but overridden to recoverable by `UpgradeCanisterError => true`; (2) an archive canister running out of cycles before the LSO tops it up, causing `stop_canister` to return `Rejected("canister not found")`, which is also overridden to recoverable. Both are plausible in production as archive canisters accumulate over time with growing transaction volume.

## Recommendation
1. Replace `await?` with explicit per-archive error handling that logs and continues:
   ```rust
   for canister_id in archives {
       if let Err(e) = upgrade_canister::<Archive, _>(canister_id, compressed_wasm_hash, runtime).await {
           log!(ERROR, "Failed to upgrade archive {canister_id}: {e:?}. Skipping.");
       }
   }
   ```
2. Fix `is_recoverable` for `StopCanisterError`, `StartCanisterError`, and `UpgradeCanisterError` to delegate to the inner `is_recoverable(CallError)` instead of unconditionally returning `true`, so permanent failures are not retried indefinitely.
3. Add a field to `UpgradeArg` (e.g., `skip_archive_ids: opt vec principal`) to allow the NNS to explicitly exclude specific archive principals from upgrade tasks without requiring a full LSO code upgrade.

## Proof of Concept
1. Deploy the LSO managing a token with two archive canisters `[archive_1, archive_2]`.
2. Introduce a `pre_upgrade` trap in `archive_1` (e.g., via a direct `install_code` with a wasm that panics in `pre_upgrade`).
3. Submit an NNS proposal with `UpgradeArg { archive_compressed_wasm_hash: Some(new_hash), ... }`.
4. Observe: `stop_canister(archive_1)` succeeds, `install_code(archive_1, new_wasm)` returns `CanisterError("canister trapped: ...")`, `?` propagates `UpgradeCanisterError`, loop aborts, `archive_2` is never upgraded.
5. Observe: `UpgradeCanisterError::is_recoverable() == true` → task rescheduled after 5 seconds.
6. Observe: on every subsequent timer tick, the same failure repeats; `archive_2` remains on the old wasm indefinitely.
7. Confirm: no field in `UpgradeArg` allows skipping `archive_1`; the only remediation is a direct NNS `install_code` call bypassing the LSO.
