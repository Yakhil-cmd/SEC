Audit Report

## Title
`sender_info.sig` Included in `MessageId` Hash Enables Deduplication Bypass via Non-Deterministic Canister Signatures — (`rs/types/types/src/messages/http.rs`)

## Summary

`representation_independent_hash_call_or_query` includes the full `sender_info` map — including the `sig` field — in the hash used to compute the `MessageId`. Because canister signatures are non-deterministic Merkle witnesses (the same `info` blob produces structurally different but cryptographically valid `sig` bytes at different certified-state heights), an attacker who controls a signing canister can produce two distinct valid `sender_info.sig` values for the same `info` blob, yielding two distinct `MessageId`s for what is semantically the same request. Both pass `validate_sender_info`, both pass ingress deduplication, and both are inducted and executed — violating the at-most-once execution invariant.

## Finding Description

**Root cause — `MessageId` includes `sender_info.sig`:**

In `rs/types/types/src/messages/http.rs`, `representation_independent_hash_call_or_query` inserts the entire `sender_info` struct — including `sig` — into the hash map used to compute the `MessageId`:

```rust
if let Some(RawSignedSenderInfoSlices { info, signer, sig }) = sender_info {
    map.insert(
        "sender_info",
        Map(btreemap! {
            "info" => Bytes(info),
            "signer" => Bytes(signer),
            "sig" => Bytes(sig),   // non-deterministic field included
        }),
    );
}
```

This is propagated to `SignedIngressContent::id()` in `rs/types/types/src/messages/ingress_messages.rs` (L113–129), which passes `sig: &sender_info.sig` into the hash function.

**Deduplication is keyed solely on `MessageId`:**

`ValidSetRuleImpl::is_duplicate` in `rs/messaging/src/scheduling/valid_set_rule.rs` (L206–208) checks `state.get_ingress_status(&msg.content().id()) != &IngressStatus::Unknown`. `IngressHistorySet::contains` in `rs/ingress_manager/src/ingress_selector.rs` (L698–700) checks `(self.get_status)(&msg_id.into()) != IngressStatus::Unknown`. Both use `MessageId` as the sole deduplication key. Two requests with identical logical content but different `sender_info.sig` bytes have different `MessageId`s and are treated as distinct messages.

**Canister signatures are non-deterministic:**

A canister signature is a CBOR struct containing a `certificate` (a BLS-signed state tree snapshot) and a `tree` (a Merkle witness). The same `info` blob signed at two different certified-state heights produces two structurally different but both-valid `sig` byte strings. `check_sig_path` in `packages/ic-signature-verification/src/canister_sig.rs` (L56–71) only verifies that the path `sig/<seed_hash>/<msg_hash>` exists in the witness tree — it does not require the witness to be canonical. `verify_canister_sig` (L16–29) verifies the BLS certificate independently of the Merkle witness structure, so both `sig1` and `sig2` pass verification.

**Attack sequence:**

1. Attacker controls canister C.
2. At certified state S1: obtain `sig1 = canister_sign(info_blob)`. Compute `MessageId1` (includes `sig1`). Obtain outer `sender_sig1 = canister_sign(MessageId1)`.
3. At certified state S2 ≠ S1: obtain `sig2 = canister_sign(info_blob)` (different Merkle witness, same logical message). Compute `MessageId2` (includes `sig2`, so `MessageId2 ≠ MessageId1`). Obtain outer `sender_sig2 = canister_sign(MessageId2)`.
4. Submit R1 and R2 with identical `{sender, canister_id, method, arg, expiry, nonce, info, signer}` but `sig1 ≠ sig2`.
5. Both pass `validate_sender_info` (both have valid canister signatures at their respective state heights).
6. Both pass `is_duplicate` / `IngressHistorySet::contains` (different `MessageId`s).
7. Both are inducted and executed — the same logical call runs twice.

**Existing checks are insufficient:**

`validate_sender_info` in `rs/validator/src/ingress_validation.rs` (L459–488) calls `verify_sender_info_canister_sig`, which verifies the canister signature on the `info` blob. It correctly rejects invalid signatures but cannot distinguish between two different valid canister signatures for the same `info` blob — both are legitimate. The deduplication check never fires because the `MessageId` differs.

## Impact Explanation

Any canister with side effects targeted by a `sender_info`-bearing request is vulnerable to double execution. The most severe case is ICP/Cycles transfers: an attacker controlling a signing canister can cause the same transfer to execute twice, draining the victim's balance. This matches the **High** impact category: significant ledger/financial security impact with concrete user or protocol harm, where exploitation requires controlling a signing canister (meaningful per-target work).

## Likelihood Explanation

The `sender_info` feature is implemented and tested in production code. Any canister controller can deploy a signing canister and exploit this. The attack requires making two calls to the signing canister at different certified-state heights — a low-effort operation. No privileged access beyond canister controller status is required.

## Recommendation

Exclude `sender_info.sig` from the `MessageId` hash. Only `sender_info.info` and `sender_info.signer` are semantically meaningful for deduplication — they identify the logical claim being made. The `sig` is a proof of that claim and is non-deterministic. The corrected hash in `rs/types/types/src/messages/http.rs` should be:

```rust
if let Some(RawSignedSenderInfoSlices { info, signer, .. }) = sender_info {
    map.insert(
        "sender_info",
        Map(btreemap! {
            "info" => Bytes(info),
            "signer" => Bytes(signer),
            // sig intentionally excluded from MessageId
        }),
    );
}
```

The `sig` must still be validated by `validate_sender_info` — it just must not contribute to the `MessageId`.

## Proof of Concept

A state-machine test would:
1. Create a mock canister signer that returns a different Merkle witness for the same `info` blob on each call (simulating two different certified-state heights).
2. Construct R1 and R2 with identical `{sender, canister_id, method, arg, expiry, nonce, info, signer}` but `sig1 ≠ sig2`.
3. Assert `MessageId1 ≠ MessageId2` (trivially true given the hash includes `sig`; the existing `message_id_changes_when_sender_info_is_present` test in `rs/types/types/src/messages/message_id.rs` already demonstrates this).
4. Submit both to a state-machine test replica with a side-effecting target canister (e.g., stable memory counter).
5. Assert the counter increments twice — confirming double execution.
