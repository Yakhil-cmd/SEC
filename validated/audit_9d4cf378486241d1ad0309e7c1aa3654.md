### Title
Missing Size Bound on `VetKdEncryptedKeyShareContent` Allows Byzantine Node to Cause Per-Artifact Memory Exhaustion тАФ (`rs/types/types/src/consensus/idkg.rs`, `rs/crypto/src/vetkd/mod.rs`)

---

### Summary

A Byzantine subnet node can craft a `pb::VetKdKeyShare` protobuf artifact with an arbitrarily large `encrypted_key_share` blob. The `TryFrom<pb::VetKdKeyShare>` conversion performs no length check, the P2P transport layer explicitly disables body-size limits, and the subsequent `verify_encrypted_key_share_internal` path materializes a full copy of the blob via `as_signed_bytes()` before the signature check fails. The `EncryptedKeyShare::deserialize` length guard that does exist is only invoked during key combination, not during per-share verification.

---

### Finding Description

**Step 1 тАФ No size check in `TryFrom`.** [1](#0-0) 

`value.encrypted_key_share` (a raw `Vec<u8>`) is wrapped directly into `VetKdEncryptedKeyShareContent` with no length validation. The legitimate size is `EncryptedKeyShare::BYTES` = `2 * G1Affine::BYTES + G2Affine::BYTES` (тЙИ 144 bytes for BLS12-381). [2](#0-1) 

**Step 2 тАФ P2P transport has no body-size limit.** [3](#0-2) 

The axum router for all consensus artifacts explicitly calls `.layer(DefaultBodyLimit::disable())`. A Byzantine peer can push a 100 MB `SlotUpdate` containing the crafted artifact with no transport-level rejection.

**Step 3 тАФ Verification allocates a full copy of the blob.**

`verify_encrypted_key_share_internal` calls `BasicSigVerifierInternal::verify_basic_sig` with `&key_share.encrypted_key_share` as the `Signable` message: [4](#0-3) 

`verify_basic_sig` calls `message.as_signed_bytes()`: [5](#0-4) 

`as_signed_bytes()` is implemented generically as: [6](#0-5) 

For `VetKdEncryptedKeyShareContent`, `write_signed_bytes_without_domain_separator` does: [7](#0-6) 

This `extend_from_slice` allocates a new `Vec` containing the entire blob тАФ 100 MB for a 100 MB artifact тАФ before the Ed25519 signature check fails.

**Step 4 тАФ The existing length guard does NOT protect here.**

`EncryptedKeyShare::deserialize` checks `val.len() != Self::BYTES` and returns early: [8](#0-7) 

But this function is only called inside `combine_encrypted_key_shares_internal`: [9](#0-8) 

It is **never called** during `verify_encrypted_key_share_internal`. The verification path has no early-exit length check.

**Step 5 тАФ Parallel validation amplifies the impact.**

`validate_signature_shares` collects all unvalidated shares and processes them via a thread pool with `into_par_iter()`: [10](#0-9) 

A Byzantine node can submit one oversized artifact per active VetKd request. With N active requests and a thread pool of width W, up to W ├Ч 100 MB can be allocated simultaneously.

---

### Impact Explanation

A single Byzantine subnet node (below the consensus fault threshold) can cause OOM on one honest replica by sending crafted `VetKdKeyShare` artifacts with oversized `encrypted_key_share` blobs. The replica process is killed by the OS, temporarily removing it from the subnet. The subnet continues operating (safety is not broken), but availability of the targeted replica is disrupted.

---

### Likelihood Explanation

The attack requires being a registered subnet node (or having compromised one). Within the IC's Byzantine fault model, a single malicious node is an explicitly in-scope attacker. The exploit path is deterministic, requires no timing, and is locally testable. The VetKd feature is active on subnets processing vetKD requests.

---

### Recommendation

Add a length check in `TryFrom<pb::VetKdKeyShare>` before constructing `VetKdEncryptedKeyShareContent`:

```rust
impl TryFrom<pb::VetKdKeyShare> for VetKdKeyShare {
    type Error = ProxyDecodeError;
    fn try_from(value: pb::VetKdKeyShare) -> Result<Self, Self::Error> {
        use ic_crypto_internal_bls12_381_vetkd::EncryptedKeyShare;
        if value.encrypted_key_share.len() != EncryptedKeyShare::BYTES {
            return Err(ProxyDecodeError::Other(format!(
                "VetKdKeyShare::encrypted_key_share has invalid length: {} (expected {})",
                value.encrypted_key_share.len(),
                EncryptedKeyShare::BYTES,
            )));
        }
        Ok(Self { ... })
    }
}
```

Alternatively, add the same check at the top of `verify_encrypted_key_share_internal` before calling `BasicSigVerifierInternal::verify_basic_sig`.

### Proof of Concept

```rust
// Construct an oversized artifact
let large_share = pb::VetKdKeyShare {
    signer_id: Some(valid_node_id_proto),
    request_id: Some(valid_request_id_proto),
    encrypted_key_share: vec![0u8; 10_000_000], // 10 MB
    node_signature: vec![0u8; 64],
};

// TryFrom succeeds тАФ no length check
let share = VetKdKeyShare::try_from(large_share).unwrap();

// verify_encrypted_key_share_internal allocates ~10 MB in as_signed_bytes()
// before returning VerificationError (invalid signature)
let result = crypto.verify_encrypted_key_share(signer_id, &share.share, &args);
assert!(matches!(result, Err(VetKdKeyShareVerificationError::VerificationError(_))));
// Memory spike of ~10 MB occurred before the error was returned
```
