Audit Report

## Title
Quarantined Reimbursement State Has No Automated Recovery Path, Permanently Locking User Funds - (`rs/ethereum/cketh/minter/src/withdraw.rs`, `rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

## Summary

The ckETH/ckERC20 minter (and ckBTC minter) implement a `QuarantinedReimbursement` terminal state that is explicitly documented as requiring "manual intervention" to resolve. When a panic occurs in the reimbursement callback after a successful ledger mint call, the IC CDK's cleanup mechanism fires the `scopeguard`, recording `EventType::QuarantinedReimbursement` and permanently removing the request from the retry queue. No automated retry, user-callable recovery endpoint, or governance-accessible unquarantine mechanism exists in the production canister interface. User funds are permanently locked until a governance-approved canister upgrade manually resolves the quarantine.

## Finding Description

**Scopeguard fires on IC callback panic (confirmed by DFINITY's own code):**

The IC CDK implements a cleanup callback mechanism (`call_on_cleanup`) that runs Rust drop handlers when a canister callback panics. This is confirmed by DFINITY's own comments in `rs/bitcoin/ckbtc/minter/src/reimbursement/mod.rs` (lines 104–107):

> "Note that in case of a panic in the callback (either before or after this point) the defuse will not be effective (due to state rollback), and the guard that was setup before the `mint_ckbtc` async call will be invoked."

The same pattern and comment appear in `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs` (lines 375–378). This confirms the scopeguard mechanism is intentional and does fire on panic.

**Trigger path in ckETH/ckERC20:**

1. User calls `withdraw_erc20` (unprivileged). ckETH and ckERC20 are burned from the ledger.
2. The Ethereum transaction finalizes with `TransactionStatus::Failure`. A `ReimbursementRequest` is enqueued.
3. The timer-driven `process_reimbursement()` fires. The scopeguard is armed at line 70:

```rust
// rs/ethereum/cketh/minter/src/withdraw.rs:70-72
let prevent_double_minting_guard = scopeguard::guard(index.clone(), |index| {
    mutate_state(|s| process_event(s, EventType::QuarantinedReimbursement { index }));
});
```

4. `client.transfer(args).await` succeeds (ledger mint completes, tokens minted to user).
5. A panic occurs in `mutate_state(|s| process_event(s, event))` at line 138 (e.g., deterministic integer overflow, `unwrap()` on `None`, or any other bug in the state transition).
6. The IC rolls back heap changes from the callback. The scopeguard fires in the cleanup callback, recording `QuarantinedReimbursement`.
7. `record_quarantined_reimbursement` removes the request from `reimbursement_requests` and inserts `Err(ReimbursedError::Quarantined)` into `reimbursed`:

```rust
// rs/ethereum/cketh/minter/src/state/transactions/mod.rs:775-779
pub fn record_quarantined_reimbursement(&mut self, index: ReimbursementIndex) {
    self.reimbursement_requests.remove(&index);
    self.reimbursed
        .insert(index, Err(ReimbursedError::Quarantined));
}
```

8. `ReimbursedError::Quarantined` is explicitly documented as terminal:

```rust
// rs/ethereum/cketh/minter/src/state/transactions/mod.rs:271-277
pub enum ReimbursedError {
    /// Whether reimbursement was minted or not is unknown,
    /// most likely because there was an unexpected panic in the callback.
    /// The reimbursement request is quarantined to avoid any double minting and
    /// will not be further processed without manual intervention.
    Quarantined,
}
```

**No recovery endpoint exists:**

The full `cketh_minter.did` Candid interface (lines 696–750) exposes no `unquarantine_reimbursement`, `retry_quarantined`, or equivalent admin/governance call. The `TxFinalizedStatus` type has no `Quarantined` variant, so the user sees `PendingReimbursement` indefinitely. The quarantined state is only surfaced as a dashboard display entry with no action attached.

**Same pattern confirmed in ckBTC:**

`rs/bitcoin/ckbtc/minter/src/reimbursement/mod.rs` (lines 64–108) uses an identical scopeguard pattern. `rs/bitcoin/ckbtc/minter/src/state.rs` (lines 1773–1777) implements `quarantine_withdrawal_reimbursement` with the same terminal semantics. `rs/bitcoin/ckbtc/minter/src/state.rs` (line 860) maps `ReimbursedError::Quarantined` to `RetrieveBtcStatusV2::Unknown`, hiding the quarantine from users entirely.

## Impact Explanation

**Concrete impact:** When a user's ckETH or ckERC20 withdrawal fails on Ethereum and the subsequent reimbursement mint panics mid-callback, the user's burned tokens are permanently locked. The ledger mint may or may not have succeeded (state is unknown), but the minter records `Quarantined` and will never retry. The user loses access to their funds with no self-service recovery path. For ckBTC/ckDOGE, the same applies to canceled withdrawals. This is a significant ck-token security impact with concrete, per-user fund loss.

**Severity: High** — Matches "Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm." Funds are permanently locked for affected users until a governance-approved canister upgrade manually resolves the quarantine. The minter continues operating normally for other users, limiting global impact.

## Likelihood Explanation

**Medium-High.** The quarantine guard fires on any unexpected panic in the reimbursement callback after a successful ledger mint. The ckBTC mainnet has already experienced a deterministic panic in a closely related async callback path (the resubmission path), documented in `rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md` (lines 31–33): "There is a deterministic panic occurring in the minter when it tries to resubmit those transactions, which explains why those transactions are currently stuck." The ckETH `process_reimbursement` loop calls `mutate_state` after an `await` at line 138 — any deterministic bug in `record_finalized_reimbursement` (e.g., an `unwrap()` on a missing key, integer overflow, or assertion failure) would trigger this. The condition is not attacker-controlled but is reachable through normal user withdrawal activity combined with any minter-side bug, and real-world evidence confirms such bugs occur.

## Recommendation

1. **Add a governance-callable or timer-driven unquarantine endpoint** that, given a `ReimbursementIndex`, queries the ledger for a mint transaction matching the expected memo (`MintMemo::ReimburseWithdrawal { withdrawal_id }`) and, if found, transitions the state to `Reimbursed`; if not found, re-enqueues the request for retry.
2. **Alternatively**, on canister upgrade or timer, scan `reimbursed` entries with `Err(Quarantined)`, query the ledger for the corresponding mint block, and resolve the ambiguity before permanently discarding the reimbursement.
3. **For ckBTC/ckDOGE**, apply the same fix to `QuarantinedWithdrawalReimbursement` entries in `reimbursed_withdrawals`.
4. **Expose the quarantined state** in `TxFinalizedStatus` / `RetrieveBtcStatusV2` so users can identify affected withdrawals.

## Proof of Concept

**Minimal reproducible path (PocketIC integration test):**

1. Deploy ckETH minter + ckETH ledger in PocketIC.
2. Call `withdraw_erc20(amount=X, ...)` — burns ckETH and ckERC20.
3. Inject a finalized transaction receipt with `TransactionStatus::Failure` to enqueue a `ReimbursementRequest`.
4. Patch `record_finalized_reimbursement` (or any function called by `mutate_state` at line 138 of `withdraw.rs`) to panic deterministically (e.g., add `panic!("test")` after the ledger mint succeeds).
5. Trigger the timer to run `process_reimbursement()`.
6. Observe: the scopeguard fires, `QuarantinedReimbursement` is recorded in the event log, `reimbursement_requests` is empty, `reimbursed` contains `Err(Quarantined)`.
7. Verify: calling `retrieve_eth_status(block_index)` returns `TxFinalized(PendingReimbursement(...))` indefinitely. No endpoint exists to recover the funds.
