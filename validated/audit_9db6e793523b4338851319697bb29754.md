Audit Report

## Title
Missing Identity-Point Guard in `TransportPublicKey::deserialize` Allows Trivial vetKD Key Recovery - (`rs/crypto/internal/crypto_lib/bls12_381/vetkd/src/lib.rs`)

## Summary

`TransportPublicKey::deserialize` accepts the BLS12-381 G1 identity point (encoded as `[0xC0, 0x00, …, 0x00]`) because `G1Affine::from_compressed` treats the identity as a valid prime-order subgroup member. When the identity is used as the transport public key in `EncryptedKeyShare::create`, the blinding term vanishes and `c3` becomes the raw BLS signature `msg * dsk` — the derived key material — written in plaintext inside the returned ciphertext. An unprivileged requester who submits this 48-byte constant as their transport public key receives the vetKD derived key directly, with no transport secret key required.

## Finding Description

**Root cause — no identity check in `TransportPublicKey::deserialize`:**

`TransportPublicKey::deserialize` delegates entirely to `G1Affine::deserialize`, which calls `ic_bls12_381::G1Affine::from_compressed`. That function performs a prime-order subgroup check, but the identity element is the neutral element of every subgroup and therefore passes unconditionally. No subsequent guard rejects the identity. [1](#0-0) [2](#0-1) 

**Encryption degenerates with identity transport key:**

In `EncryptedKeyShare::create`, `c3` is computed as:

```
c3 = transport_pk.point() * r + msg * dsk
```

When `transport_pk` is the identity, `identity * r = identity` (additive identity), so:

```
c3 = identity + msg * dsk = msg * dsk
```

`msg * dsk` is the BLS signature of the derivation input under the derived secret key — the vetKD key material itself — stored unencrypted in `c3`. [3](#0-2) 

**Vault propagates the degenerate key without additional checks:**

After `TransportPublicKey::deserialize` succeeds, `create_encrypted_vetkd_key_share_internal` proceeds directly to `EncryptedKeyShare::create` with no further validation of the transport key. [4](#0-3) [5](#0-4) 

**Existing test does not cover the identity point:**

The only test for invalid transport keys XORs `0x40` onto a non-identity point, producing an inconsistent encoding that `from_compressed` rejects. The properly-encoded identity `[0xC0, 0x00, …]` is never tested and passes deserialization. [6](#0-5) 

**Combined key recovery:**

Each node's share has `c3_i = msg * dsk_i`. Lagrange interpolation over the shares yields the combined `c3 = msg * dsk` (the full threshold derived key). The attacker reads `c3` from bytes `[48..144]` of the returned `VetKdEncryptedKeyShareContent` or the combined `VetKdEncryptedKey` — no transport secret key is needed.

## Impact Explanation

This constitutes compromise of vetKD threshold key material: the derived secret key (as a BLS group element) is returned in plaintext to any requester who supplies the identity point. Any data encrypted to the corresponding derived public key is immediately decryptable. This matches the Critical bounty impact: **Disclosure of subnet key shares or compromise of threshold signing/key material**.

## Likelihood Explanation

The attack requires no privileges, no threshold corruption, and no social engineering. The attacker is any canister or user who can submit a vetKD request. The malicious transport key is a well-known 48-byte constant (`0xC0` followed by 47 zero bytes). The attack is deterministic, locally reproducible, and requires a single API call.

## Recommendation

Add an explicit identity check in `TransportPublicKey::deserialize` immediately after deserialization:

```rust
pub fn deserialize(bytes: &[u8]) -> Result<Self, TransportPublicKeyDeserializationError> {
    let public_key = G1Affine::deserialize(&bytes)
        .map_err(|_| TransportPublicKeyDeserializationError::InvalidPublicKey)?;
    if bool::from(public_key.is_identity()) {
        return Err(TransportPublicKeyDeserializationError::InvalidPublicKey);
    }
    Ok(Self { public_key })
}
```

This single fix covers all callers: `create_encrypted_vetkd_key_share_internal`, `combine_encrypted_key_shares_internal`, and `verify_encrypted_key_internal`, all of which go through `TransportPublicKey::deserialize`. [1](#0-0) 

## Proof of Concept

```rust
#[test]
fn identity_transport_key_leaks_derived_key() {
    // BLS12-381 G1 identity encoding: compression bit | infinity bit, rest zeros
    let identity_tpk: Vec<u8> = { let mut v = vec![0u8; 48]; v[0] = 0xC0; v };

    // Step 1: deserialization succeeds (this is the bug)
    let tpk = TransportPublicKey::deserialize(&identity_tpk)
        .expect("identity point must be rejected but is accepted");

    // Step 2: create a share with a known node secret key
    let mut rng = reproducible_rng();
    let node_sk = Scalar::random(&mut rng);
    let master_pk = G2Affine::from(G2Affine::generator() * &Scalar::random(&mut rng));
    let context = DerivationContext::new(b"canister", b"ctx");
    let input = b"test-input";

    let share = EncryptedKeyShare::create(&mut rng, &master_pk, &node_sk, &tpk, &context, input);

    // Step 3: c3 == msg * dsk (the raw derived key), no transport secret needed
    let (dpk, delta) = context.derive_key_internal(&master_pk); // delta + node_sk = dsk
    let dsk = delta + node_sk;
    let msg = G1Affine::augmented_hash(&dpk, input);
    let expected_c3 = G1Affine::from(msg * &dsk);

    assert_eq!(share.c3, expected_c3, "c3 is the plaintext derived key");
}
```
