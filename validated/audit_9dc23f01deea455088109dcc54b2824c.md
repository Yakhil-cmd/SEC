Audit Report

## Title
Predictable ChaCha20Rng Seed Combined with Unauthenticated `get_rule_by_id` Query Enables Confidential Rule UUID Enumeration and `incident_id` Disclosure — (`rs/boundary_node/rate_limits/canister/random.rs`, `canister.rs`, `getter.rs`, `confidentiality_formatting.rs`)

## Summary

The rate-limits canister seeds its `ChaCha20Rng` with only 8 bytes of publicly observable IC time and 24 hardcoded bytes (`0x2A`), making the full RNG state reconstructible by any observer. Because `get_rule_by_id` is a `#[query]` method, it is freely callable by any anonymous principal and is never subject to `inspect_message`. An attacker who reconstructs the seed can predict all generated rule UUIDs, use `get_rule_by_id` as an existence oracle, and extract the `incident_id` of confidential (non-disclosed) rules — leaking active security incident identifiers before public disclosure.

## Finding Description

**Root cause 1 — Weak RNG seed** (`random.rs` lines 8–13):

```rust
thread_local! {
  static RNG: RefCell<ChaCha20Rng> = {
    let mut seed = [42; 32];
    seed[..8].copy_from_slice(&time().to_le_bytes());
    RefCell::new(ChaCha20Rng::from_seed(seed))
  };
}
```

The 32-byte seed has only 8 bytes of entropy: the IC nanosecond timestamp at first RNG access (the first `add_config` call). The remaining 24 bytes are hardcoded to `0x2A`. IC block timestamps are deterministic and publicly readable from the IC's replicated state. This makes the full seed reconstructible with zero brute-force.

**Root cause 2 — `get_rule_by_id` bypasses all access control** (`canister.rs` lines 30–67, 123–133):

`inspect_message` is an IC hook that only fires for ingress update messages. It explicitly handles `get_config` (as a replicated query) and `add_config`/`disclose_rules` (as updates). The `#[query]` annotation on `get_rule_by_id` means it is served as a non-replicated query call, which never triggers `inspect_message`. Any anonymous caller can invoke it freely. The `else` branch in `inspect_message` that traps "all other calls" only applies to update-path ingress messages, not query calls.

**Root cause 3 — Incomplete redaction creates existence oracle and `incident_id` leak** (`getter.rs` lines 219–246, `confidentiality_formatting.rs` lines 31–42):

For unauthorized callers, `RuleConfidentialityFormatter` only nulls out `rule_raw` and `description`. It returns `Ok(OutputRuleMetadata { rule_id, incident_id, added_in_version, removed_in_version, ... })` for any existing rule, disclosed or not. For non-existent rule IDs it returns `Err(NotFound)`. This binary distinction is a direct existence oracle. The `incident_id` field in the `Ok` response leaks the security incident identifier for confidential rules. This is confirmed by the unit test at `getter.rs` lines 433–446, which asserts `incident_id` is present in the response for `RestrictedRead` callers on non-disclosed rules.

**Attack chain:**

1. Observe the IC block timestamp of the first `add_config` call (public replicated state).
2. Reconstruct the exact `ChaCha20Rng` seed: `seed[..8] = init_time_ns.to_le_bytes(); seed[8..] = [42; 24]`.
3. Count total rules ever generated (observable from `get_config` across all versions — rule counts per version are public).
4. Advance a local RNG by that count to predict the next UUID(s).
5. Call `get_rule_by_id(predicted_uuid)` as an anonymous query — no authentication, no `inspect_message`.
6. `Ok(...)` response confirms rule existence and reveals `incident_id`; `Err(NotFound)` confirms absence.

## Impact Explanation

An unprivileged anonymous attacker can enumerate the UUIDs of confidential rate-limit rules and extract their `incident_id` values before public disclosure. The rate-limits canister is designed to keep confidential rules (those with `disclosed_at: None`) hidden from unauthorized callers. This design invariant is broken: the attacker learns which security incidents have active, non-disclosed rate-limit rules, defeating the confidentiality model of the boundary node rate-limits system. This constitutes a significant boundary/API infrastructure security impact with concrete harm — premature disclosure of active security incident identifiers to any anonymous internet user.

**Severity: High ($2,000–$10,000)** — Significant boundary/API infrastructure security impact with concrete user or protocol harm.

## Likelihood Explanation

The attack requires no privileges, no victim interaction, and no special network position. All inputs (IC block timestamp, rule counts from `get_config`) are publicly available. Query calls are free and unauthenticated on the IC. The seed search space is effectively zero once the block timestamp is known (it is deterministic per block). The attack is fully reproducible locally with a PocketIC or local replica test. It can be repeated for every new `add_config` operation.

## Recommendation

1. **Fix the RNG seed**: Replace `time()`-based seeding with `ic_cdk::api::management_canister::main::raw_rand()` (the IC's certified randomness beacon) called asynchronously during `init`/`post_upgrade`. Store the resulting 32 bytes as the RNG seed in stable memory.
2. **Eliminate the existence oracle**: For unauthorized callers querying non-disclosed rules, return `Err(NotFound)` (or a generic opaque error) instead of a redacted `Ok` response. This removes the binary signal that distinguishes existing from non-existing confidential rules.
3. **Redact `incident_id`** in `RuleConfidentialityFormatter` for non-disclosed rules, consistent with the intent to keep confidential rule metadata hidden from unauthorized callers.
