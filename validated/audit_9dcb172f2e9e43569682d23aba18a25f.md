Audit Report

## Title
Cache Key Collision Between QueryV2 and QueryV3 Allows Wrong-Version Response Serving — (`rs/boundary_node/ic_boundary/src/routes.rs`)

## Summary

The custom `Hash` and `PartialEq` implementations on `RequestContext` — the type used as the boundary-node query cache key — omit the `request_type` field. Because `is_query()` returns `true` for both `QueryV2` and `QueryV3`, both request types pass the cache bypasser and are stored/looked up under identical keys when all other fields match. A `QueryV3` caller can therefore receive a cached `QueryV2` response body (which carries no certificate), silently stripping the certification guarantee that `QueryV3` is designed to provide.

## Finding Description

`KeyExtractorContext` extracts `Arc<RequestContext>` directly as the cache key: [1](#0-0) 

The `Hash` implementation on `RequestContext` hashes only `canister_id`, `sender`, `method_name`, `ingress_expiry`, and `arg`/`http_request` — `request_type` is never fed into the hasher: [2](#0-1) 

The `PartialEq` implementation compares the same subset and also omits `request_type`: [3](#0-2) 

The cache bypasser only skips caching when `!ctx.request_type.is_query()`: [4](#0-3) 

And `is_query()` returns `true` for both `QueryV2` and `QueryV3`: [5](#0-4) 

Exploit flow: (1) Attacker sends a `QueryV2` POST to `/api/v2/canister/{id}/query` with parameters `{canister_id, sender, method_name, ingress_expiry, arg}`. The response is stored in the cache under key `K`. (2) A `QueryV3` client sends the same logical query to `/api/v3/canister/{id}/query` with identical parameters. `KeyExtractorContext` extracts a `RequestContext` that hashes and compares equal to `K` (because `request_type` is excluded). The cache returns the `QueryV2` body as a hit. The `QueryV3` client receives a response with no certificate.

## Impact Explanation

`QueryV3` exists specifically to return a `CertifiedQueryResponse` containing a certificate that clients use for cryptographic verification of the result. Serving a `QueryV2` (uncertified) body to a `QueryV3` caller strips this guarantee: clients enforcing certificate verification will receive a malformed/absent certificate and fail; clients that do not strictly validate will silently accept an uncertified response as if it were certified, undermining the integrity property `QueryV3` is designed to provide. This is a concrete boundary/API security impact with direct harm to users relying on response certification — matching the High impact category for significant boundary/API security impact.

## Likelihood Explanation

The only precondition is `cache_size > 0`, which is the default production configuration. No privilege is required. A single client making the same logical query over both API versions (or two independent clients) triggers the collision deterministically and repeatably. The path is fully local to the boundary node and requires no network-level attack.

## Recommendation

Include `request_type` in both the `Hash` and `PartialEq` implementations of `RequestContext`:

```rust
impl Hash for RequestContext {
    fn hash<H: Hasher>(&self, state: &mut H) {
        self.request_type.hash(state); // ADD
        self.canister_id.hash(state);
        self.sender.hash(state);
        self.method_name.hash(state);
        self.ingress_expiry.hash(state);
        if self.http_request.is_some() {
            self.http_request.hash(state);
        } else {
            self.arg.hash(state);
        }
    }
}

impl PartialEq for RequestContext {
    fn eq(&self, other: &Self) -> bool {
        self.request_type == other.request_type // ADD
            && self.canister_id == other.canister_id
            && self.sender == other.sender
            && self.method_name == other.method_name
            && self.ingress_expiry == other.ingress_expiry
            && if self.http_request.is_some() {
                self.http_request == other.http_request
            } else {
                self.arg == other.arg
            }
    }
}
```

The comment at lines 91–93 states "They should both work on the same fields" — `request_type` must be one of those fields. [6](#0-5) 

## Proof of Concept

Unit test (no network required):

```rust
use std::collections::hash_map::DefaultHasher;
use std::hash::{Hash, Hasher};

fn hash_of(ctx: &RequestContext) -> u64 {
    let mut h = DefaultHasher::new();
    ctx.hash(&mut h);
    h.finish()
}

let a = RequestContext {
    request_type: RequestType::QueryV2,
    canister_id: Some(p), sender: Some(s),
    method_name: Some("m".into()), ingress_expiry: Some(0),
    arg: Some(vec![1, 2, 3]), ..Default::default()
};
let b = RequestContext { request_type: RequestType::QueryV3, ..a.clone() };

assert_eq!(hash_of(&a), hash_of(&b)); // collision confirmed
assert_eq!(a, b);                      // equality confirmed
```

Integration test: send a `QueryV2` POST to populate the cache, then send a `QueryV3` POST with identical parameters and observe `X-Cache: hit` with a response body lacking a certificate field in the CBOR payload.
