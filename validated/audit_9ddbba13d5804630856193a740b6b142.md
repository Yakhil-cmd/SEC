Audit Report

## Title
FD Leak in Replica IPC Transport via Excess SCM_RIGHTS from Compromised Sandbox — (`rs/canister_sandbox/src/transport.rs`)

## Summary

`install_file_descriptors` fails to close or drain extra file descriptors when a received IPC message carries more FDs in SCM_RIGHTS ancillary data than its `enumerate_fds` slots declare. Because the `fds` accumulator vector is shared across all messages in the `socket_read_messages` loop, these extras persist indefinitely in the replica process without ever being closed. Since every `ctlsvc::Request` variant declares zero FD slots, a compromised sandbox can exhaust the replica's open-FD limit by sending a burst of normal-looking messages each carrying up to 16 extra FDs.

## Finding Description

**Root cause — `install_file_descriptors` (`transport.rs:501-504`):**

```rust
if fd_locs.len() < fds.len() {
    fds.drain(0..fd_locs.len());   // only removes fd_locs.len() items
    // extras at [fd_locs.len()..fds.len()] remain — never closed
} else {
    fds.clear();
}
```

When `fd_locs.len() == 0` (the case for every `ctlsvc::Request` variant), `fds.drain(0..0)` is a no-op. All FDs received in that call remain in the vector.

**The `fds` vector is allocated once and reused across every message in the loop (`transport.rs:444, 448`):**

```rust
let mut fds = Vec::<RawFd>::new();
...
install_file_descriptors(&mut frame, &mut fds);
```

Because subsequent messages also have zero FD slots, the extras are never consumed by a later iteration either.

**The receive path blindly accepts all SCM_RIGHTS FDs with no count validation (`transport.rs:662-682`):** every FD in the ancillary data is pushed into `fds` unconditionally.

**`ctlsvc::Request` declares zero FD slots (`ctlsvc.rs:43-45`):**

```rust
impl EnumerateInnerFileDescriptors for Request {
    fn enumerate_fds<'a>(&'a mut self, _fds: &mut Vec<&'a mut std::os::unix::io::RawFd>) {}
}
```

This covers all three sandbox→controller message types: `ExecutionFinished`, `ExecutionPaused`, and `LogViaReplica`.

**The replica's IPC reader for the sandbox channel is a long-running dedicated thread (`launch_as_process.rs:115-131`)** that calls `socket_read_messages` in a loop with no FD-count guard, so the leak accumulates for the lifetime of the sandbox connection.

## Impact Explanation

A compromised sandbox can send a rapid stream of `ExecutionFinished` (or any zero-FD-slot) messages, each carrying up to `MAX_NUM_FD_PER_MESSAGE = 16` extra FDs in SCM_RIGHTS. Each `recvmsg` call duplicates those FDs into the replica process's file descriptor table. They are never closed. Once the replica's open-FD limit (`/proc/sys/fs/nr_open`, typically 1,048,576 per process) is exhausted, every subsequent `socket()`, `open()`, `spawn_socketed_process()`, or `memfd_create()` call fails with `EMFILE`, breaking canister execution, state sync, and inter-process communication for the entire replica node. This constitutes a **subnet availability impact** matching the High bounty tier: *Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS.*

## Likelihood Explanation

**Prerequisite**: the sandbox process must already be compromised (e.g., via a Wasm sandbox escape). This is a non-trivial prerequisite. However, the IC's security model explicitly treats the sandbox as an untrusted peer — the IPC transport is one of the permitted interaction channels and must be hardened against a malicious sender. Given that the replica controller reads from the sandbox socket in a dedicated thread with no FD-count validation, a compromised sandbox can trigger this leak with a single burst of messages. The attack is repeatable, requires no special timing, and can be executed entirely through the normal IPC channel without any additional privileges beyond the sandbox escape itself.

## Recommendation

In `install_file_descriptors`, explicitly close any FDs that exceed the message's declared slot count before draining them:

```rust
if fd_locs.len() < fds.len() {
    for &fd in &fds[fd_locs.len()..] {
        // SAFETY: fd was received via SCM_RIGHTS and is owned by this process.
        unsafe { libc::close(fd); }
    }
    fds.clear();
} else {
    fds.clear();
}
```

Additionally, add a defensive check in `receive_message` or `socket_read_messages` that rejects (closes and drops) any message carrying more FDs than the maximum expected for the message type, logging the anomaly as a security event.
