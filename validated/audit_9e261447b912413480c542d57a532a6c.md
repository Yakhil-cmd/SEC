Audit Report

## Title
Byzantine Peer Inflates `PeerCounter` via Multi-Slot Same-ID Advertisement, Permanently Blocking Unvalidated Pool Cleanup — (`rs/p2p/consensus_manager/src/receiver.rs`)

## Summary

`PeerCounter` is a `HashMap<NodeId, u32>` reference count. A Byzantine subnet peer can advertise the same artifact ID across N distinct slot numbers, inflating its counter to N. Because `PeerCounter::remove` decrements by exactly 1 per call, withholding even one retraction leaves the counter permanently above zero. `process_slot_update` then blocks indefinitely at `peer_rx.wait_for(|p| p.is_empty())`, the `UnvalidatedArtifactMutation::Remove` message is never sent, and the artifact is permanently retained in the unvalidated pool with a leaked Tokio task.

## Finding Description

**Root cause — `PeerCounter` uses reference counting, not set semantics.**

`PeerCounter::insert` for an already-present `NodeId` increments the `u32` counter and returns `false`; it does not enforce the invariant that one peer maps to at most one count unit per artifact ID. [1](#0-0) 

**Mutation is applied unconditionally by `send_if_modified`.**

In `handle_slot_update_receive`, when `to_add = true` and an assemble task already exists for the artifact ID, the code calls:

```rust
sender.send_if_modified(|h| h.insert(peer_id));
```

Tokio's `send_if_modified` modifies the watched value **unconditionally**; it only skips watcher notification when the closure returns `false`. So even when `insert` returns `false` (peer already present, counter incremented), the counter is durably incremented in the shared watch value. [2](#0-1) 

**Retraction decrements by exactly 1 per slot overwrite.**

When a slot is overwritten, `sender.send_if_modified(|h| h.remove(peer_id))` is called once. `PeerCounter::remove` decrements by 1. If the peer advertised on 3 slots and only retracts 2, the counter stays at 1 permanently. [3](#0-2) 

**`process_slot_update` blocks indefinitely waiting for the counter to reach zero.**

After assembly completes and `Insert` is sent, the task waits unconditionally:

```rust
let _ = peer_rx.wait_for(|p| p.is_empty()).await;
```

With the counter stuck at 1, this future never resolves, `Remove` is never sent, and the task leaks. [4](#0-3) 

**Topology-update path does not compensate.**

`handle_topology_update` calls `set.remove(*n)` exactly once per departing peer (the `any()` short-circuit stops after the first `true`, and `nodes_leaving_topology` is a `HashSet` with at most one entry per `NodeId`). A counter inflated to N is only decremented to N-1, not to zero. [5](#0-4) 

**`handle_artifact_processor_joined` re-spawns the task when the counter is non-zero**, so the leak is perpetual, not one-shot. [6](#0-5) 

## Impact Explanation

A Byzantine peer can repeat this attack for many distinct artifact IDs (up to `slot_limit` slots per artifact, across arbitrarily many artifacts). Each attack instance permanently retains one artifact in the unvalidated pool and leaks one Tokio task plus its associated `watch` channel. The cumulative effect is unbounded memory growth and task accumulation on every honest receiving node in the subnet, constituting a targeted application-level DoS against the P2P receive path.

This matches the **High** bounty impact: *Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS.*

## Likelihood Explanation

A single authenticated Byzantine subnet node operating below the consensus fault threshold can execute this attack. No threshold corruption, key compromise, privileged access, or external dependency is required. Slot update messages are accepted from any peer in the current subnet topology. The attack is repeatable and low-cost.

## Recommendation

`PeerCounter` should enforce set semantics: a `NodeId` should contribute at most 1 to the count regardless of how many slots it uses to advertise the same artifact ID. The simplest fix is to replace `HashMap<NodeId, u32>` with `HashSet<NodeId>`, so `insert` is idempotent and `remove` always brings the count to zero for that peer. Alternatively, before calling `insert` in `handle_slot_update_receive`, check whether the peer already has a nonzero count for this artifact ID and skip the increment. The invariant to enforce is: the counter for a `NodeId` is 1 if and only if at least one slot in that peer's slot table maps to this artifact ID, and 0 otherwise.

## Proof of Concept

The existing test harness in `rs/p2p/consensus_manager/src/receiver.rs` supports a direct unit test. The existing `duplicate_ids_on_different_slots` test (lines 1298–1386) covers the happy path where all slots are retracted; the following variant demonstrates the stuck state:

```rust
#[tokio::test]
async fn byzantine_peer_inflates_counter_blocks_remove() {
    // assembler immediately returns Done
    fn make_assembler() -> MockArtifactAssembler { /* same as duplicate_ids_on_different_slots */ }

    let (mut mgr, mut channels) = ReceiverManagerBuilder::new()
        .with_artifact_assembler_maker(make_assembler)
        .build();
    let cancel = CancellationToken::new();

    // Byzantine peer advertises artifact ID 0 on 3 slots → counter = 3
    for slot in 1u64..=3 {
        mgr.handle_slot_update_receive(
            SlotUpdate { slot_number: SlotNumber::from(slot), commit_id: CommitId::from(slot), update: Update::Id(0) },
            NODE_1, ConnId::from(1), cancel.clone(),
        );
    }

    // Retract only 2 of the 3 slots → counter = 1
    for slot in 1u64..=2 {
        mgr.handle_slot_update_receive(
            SlotUpdate { slot_number: SlotNumber::from(slot), commit_id: CommitId::from(slot + 10), update: Update::Id(99) },
            NODE_1, ConnId::from(1), cancel.clone(),
        );
    }

    // Insert arrives (assembly completed), but Remove never arrives
    assert_eq!(
        channels.unvalidated_artifact_receiver.recv().await.unwrap(),
        UnvalidatedArtifactMutation::Insert((U64Artifact::id_to_msg(0, 100), NODE_1))
    );
    tokio::time::timeout(
        Duration::from_secs(2),
        channels.unvalidated_artifact_receiver.recv(),
    )
    .await
    .expect_err("Remove should never arrive — artifact is permanently stuck");
}
```
