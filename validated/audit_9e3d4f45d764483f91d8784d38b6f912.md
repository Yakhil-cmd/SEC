### Title
Unprivileged `execute_scheduled` Allows Any Caller to Trigger a User's Scheduled XCC Promise Prematurely, Causing Temporary Fund Freeze - (File: etc/xcc-router/src/lib.rs)

### Summary
The `execute_scheduled` function in the XCC router contract carries no caller restriction and is explicitly open to any NEAR account. An attacker can call it with a victim's scheduled-promise nonce at an unfavorable moment — before the target contract is in a state that will accept the call. When the promise fails, the NEAR that was already burned from the victim's wNEAR EVM balance is returned to the router (not to the EVM), leaving the victim's EVM wNEAR balance permanently reduced while the underlying NEAR is stranded in the router sub-account and inaccessible as EVM-side wNEAR without a further XCC round-trip.

### Finding Description

**Root cause — `execute_scheduled` has no access control:**

```rust
/// It is intentional that this function can be called by anyone (not just the parent).
/// There is no security risk to allowing this function to be open because it can only
/// act on promises that were created via `schedule`.
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
``` [1](#0-0) 

The team's stated justification — "it can only act on promises that were created via `schedule`" — is incomplete. It ignores the timing dimension: the user may have scheduled a promise that is only safe to execute after a specific on-chain condition is satisfied (e.g., a DEX is unpaused, a price window is open, a prior settlement has completed). Any third party can race to execute it before that condition holds.

**How the NEAR is pre-funded at schedule time:**

When a user calls the XCC precompile with `CrossContractCallArgs::Delayed`, the engine computes `attached_near = call.total_near()` and, via `handle_precompile_promise`, withdraws that exact amount of wNEAR from the user's EVM balance and transfers it to the router sub-account *before* the `schedule` call is made. [2](#0-1) [3](#0-2) 

By the time the promise sits in `scheduled_promises`, the user's EVM wNEAR balance is already debited. The NEAR is in the router.

**What happens on a premature, attacker-triggered execution that fails:**

NEAR's runtime refunds `attached_balance` to the calling account (the router) when a cross-contract call fails. The router receives the NEAR back, but the user's EVM wNEAR balance remains at the reduced value. The user cannot directly convert that stranded NEAR back to EVM-side wNEAR without scheduling a new XCC call — which itself requires wNEAR in the EVM. If the user's EVM wNEAR balance is now zero (all of it was committed to the failed promise), they are in a deadlock: no EVM wNEAR → cannot pay for a new XCC call → cannot recover the stranded NEAR as wNEAR.

**Nonce is trivially predictable:**

The nonce is a monotonically incrementing `u64` starting at 0. [4](#0-3) 

An attacker watching the NEAR chain for `schedule` events on a target router immediately knows the nonce of the newly stored promise.

### Impact Explanation

**High — Temporary (potentially extended) freezing of funds.**

The victim's EVM wNEAR balance is irreversibly reduced at schedule time. If the attacker forces a failed execution, the underlying NEAR is stranded in the router sub-account. Recovery requires the victim to already hold additional EVM wNEAR to fund a new XCC call, creating a deadlock for users who committed their full wNEAR balance to the scheduled promise. The funds are not permanently destroyed (they remain in the router), but they are inaccessible as EVM-side wNEAR for an indefinite period, satisfying the "temporary freezing of funds" criterion.

### Likelihood Explanation

**Medium.** The attacker must:
1. Identify a target router sub-account (trivial — derived deterministically from the EVM address).
2. Observe a `schedule` call on-chain (public NEAR transaction log).
3. Know or guess when the target contract will be in a rejecting state (e.g., paused, price out of range, registration missing).
4. Submit `execute_scheduled` before the user does.

Steps 1–2 require no privilege. Step 3 is realistic for any attacker monitoring a specific DeFi protocol. Step 4 is a standard NEAR transaction race.

### Recommendation

1. **Restrict `execute_scheduled` to the parent (Aurora engine) or the owning EVM address.** The comment's reasoning that "it can only act on promises created via `schedule`" does not account for timing sensitivity. The simplest fix is:

```rust
pub fn execute_scheduled(&mut self, nonce: U64) {
    let parent = self.get_parent().unwrap_or_else(env_panic);
    require_caller(&parent).unwrap_or_else(env_panic);
    // ... rest unchanged
}
```

2. **Alternatively, add a minimum execution delay** (e.g., a block-height or timestamp threshold stored alongside the promise) so that even if anyone can call it, they cannot do so before the user's intended window.

3. **Add a `cancel_scheduled` function** restricted to the parent, so users can abort a scheduled promise before it is executed.

### Proof of Concept

1. Alice holds 100 wNEAR in the EVM. She calls the XCC precompile with `CrossContractCallArgs::Delayed` targeting a DEX `swap` method with `attached_balance = 100 NEAR`. The engine burns 100 wNEAR from Alice's EVM balance and transfers 100 NEAR to Alice's router. The promise is stored at nonce `0`. [4](#0-3) 

2. Bob (attacker) observes the `schedule` transaction on-chain. He knows Alice's router account ID (it is `{hex(alice_address)}.aurora`) and the nonce is `0`.

3. Bob calls `execute_scheduled({"nonce": "0"})` on Alice's router at a moment when the DEX is paused or Alice's slippage tolerance would be violated.

4. The DEX rejects the call. NEAR runtime refunds 100 NEAR to Alice's router. Alice's EVM wNEAR balance remains at 0. [5](#0-4) 

5. Alice now has 0 EVM wNEAR and cannot schedule a new XCC call to recover the 100 NEAR from the router as wNEAR. Her funds are frozen until she acquires EVM wNEAR through another path (e.g., bridging from NEAR mainnet), which may not be immediately possible.
