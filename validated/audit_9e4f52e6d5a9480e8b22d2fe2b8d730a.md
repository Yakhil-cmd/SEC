### Title
`native_per_pubdata` Truncates to Zero via Floor Division When `pubdata_price < native_price`, Enabling Free Pubdata Generation — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In `validate_and_compute_fee_for_transaction`, the per-pubdata-byte native cost is computed with integer (floor) division. When `pubdata_price < native_price`, the result truncates to zero. With `native_per_pubdata = 0`, every downstream pubdata cost check trivially passes, allowing any unprivileged transaction sender to generate arbitrary amounts of pubdata (storage writes) without paying for it, while the operator must still publish that data to L1.

---

### Finding Description

The root cause is at line 142 of `validation_impl.rs`:

```rust
// We checked native_price != 0 above
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is integer floor division. When `pubdata_price < native_price` (e.g., `pubdata_price = 50`, `native_price = 100`), the result is `0`.

This zero propagates through the entire pubdata accounting chain:

**1. Intrinsic pubdata overhead is zeroed out** in `create_resources_for_tx`:
```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// = 0 * intrinsic_pubdata = 0  → no native deducted for intrinsic pubdata
```

**2. Execution pubdata cost is zeroed out** in `get_resources_to_charge_for_pubdata`:
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)  // = pubdata_bytes * 0 = 0
    .ok_or(out_of_native_resources!())?;
```

**3. The post-execution pubdata sufficiency check always passes** in `check_enough_resources_for_pubdata`:
```rust
let enough = resources.has_enough(&resources_for_pubdata);
// resources_for_pubdata = 0 native → always enough
```

The asymmetry is visible by contrast: `native_per_gas` uses **ceiling** division (`div_ceil`), ensuring it is never zero when `gas_price > 0`. `native_per_pubdata` uses **floor** division, so it silently becomes zero when `pubdata_price` is a non-zero value smaller than `native_price`.

The same floor-division pattern is replicated in the public API helper:
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

### Impact Explanation

When `native_per_pubdata = 0`, a transaction can write to an unbounded number of storage slots and the post-execution pubdata check (`check_enough_resources_for_pubdata`) will always return `enough = true`. The operator must still publish all generated pubdata to L1 (paying L1 data-availability costs), but receives zero compensation for it through the transaction fee mechanism. This is a direct financial loss to the operator proportional to the amount of pubdata generated.

The `native_per_gas` path correctly uses `div_ceil` to prevent a symmetric zero-result, confirming the floor-division choice for pubdata is an oversight rather than intentional design. [6](#0-5) [7](#0-6) 

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a realistic operator configuration. `pubdata_price` reflects the cost of L1 data publication and `native_price` reflects the cost of a single RISC-V proving cycle. During periods of low L1 gas prices, `pubdata_price` can legitimately be set below `native_price`. Any unprivileged L2 transaction sender can exploit this condition by submitting transactions that write to many storage slots. No privileged access, oracle manipulation, or key compromise is required. [8](#0-7) 

---

### Recommendation

Replace floor division with ceiling division for `native_per_pubdata`, mirroring the treatment of `native_per_gas`:

```rust
// Before (floor division — can produce 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Alternatively, add an explicit guard analogous to the original report's recommendation:

```rust
if native_per_pubdata == 0 && !pubdata_price.is_zero() {
    return Err(TxError::Validation(InvalidTransaction::PubdataPriceTooLow));
}
```

Apply the same fix to the `api/src/helpers.rs` helper function. [1](#0-0) [9](#0-8)

### Proof of Concept

1. Operator sets `pubdata_price = 50`, `native_price = 100` (a realistic low-L1-cost configuration).
2. `native_per_pubdata = 50 / 100 = 0` (floor division truncates).
3. Attacker submits an L2 transaction with `max_fee_per_gas` sufficient to pass the gas fee check, but the transaction body writes to 1000 storage slots.
4. `create_resources_for_tx` charges `0 * intrinsic_pubdata = 0` native for intrinsic pubdata.
5. After execution, `get_resources_to_charge_for_pubdata` returns `(pubdata_bytes, 0_native)`.
6. `check_enough_resources_for_pubdata` returns `enough = true` regardless of pubdata generated.
7. Transaction succeeds; operator publishes ~32,000 bytes of pubdata to L1 and receives zero compensation for it.
8. Attacker repeats indefinitely, draining operator L1 funds. [10](#0-9) [11](#0-10)

pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
