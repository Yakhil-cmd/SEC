Audit Report

## Title
Unbounded Synchronous BFS in SNS `cast_vote_and_cascade_follow` Enables Griefing via Follower Flooding - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS governance canister's `cast_vote_and_cascade_follow` function performs a fully synchronous, unbounded BFS over all follower neurons in a single message execution with no instruction-limit guard. Because any unprivileged user can create neurons and set following relationships on a target neuron without the target's consent, an attacker can flood a victim neuron's follower set up to `MAX_NUMBER_OF_NEURONS_CEILING = 200,000`. When the victim subsequently calls `register_vote`, the synchronous BFS exhausts the IC per-message instruction budget, causing the update call to trap and the vote to be permanently unrecorded.

## Finding Description
`cast_vote_and_cascade_follow` is a plain synchronous function with no `async`/`await` and no instruction-limit check: [1](#0-0) 

The BFS loop iterates over all follower neurons without any guard: [2](#0-1) 

It is called synchronously from `register_vote`: [3](#0-2) 

There is no `is_over_instructions_limit` check, no `noop_self_call_if_over_instructions` await point, and no state-machine checkpointing anywhere in the SNS governance codebase. By contrast, NNS governance uses an async state machine with explicit soft/hard instruction-limit guards: [4](#0-3) 

The `follow` function only caps how many neurons a single neuron may *follow* (`max_followees_per_function`, ceiling 15), but imposes no cap on how many neurons may *follow* a given neuron: [5](#0-4) 

`MAX_NUMBER_OF_NEURONS_CEILING` is 200,000: [6](#0-5) 

**Exploit path:**
1. Attacker stakes `neuron_minimum_stake_e8s` tokens per neuron and creates up to `max_number_of_neurons` (≤ 200,000) neurons.
2. Each attacker neuron calls `follow` targeting the victim neuron on any `function_id`. The victim cannot prevent or remove these following relationships.
3. A proposal is created; ballots are allocated for all neurons including all attacker neurons.
4. Victim calls `register_vote`. The synchronous BFS in `cast_vote_and_cascade_follow` iterates over all 200,000 follower neurons, performing O(log N) `BTreeMap` lookups per follower. The IC per-message instruction limit is exhausted; the update call traps; the vote is not recorded.

## Impact Explanation
The victim neuron's `register_vote` update call traps due to instruction exhaustion. The vote is permanently unrecorded for that message execution. In an SNS with a small number of legitimate neurons, silencing a single large-stake neuron can materially affect governance outcomes — proposals may pass or fail contrary to the victim's intent. This constitutes a significant SNS governance security impact with concrete user and protocol harm, matching the **High** bounty impact category: *"Significant SNS security impact with concrete user or protocol harm."*

## Likelihood Explanation
The attack requires staking tokens across many neurons, but `neuron_minimum_stake_e8s` and `max_number_of_neurons` are SNS-configurable and can be set very low. The following relationship is set unilaterally by the follower — the victim has no ability to block or remove followers. The attack is persistent: attacker neurons remain in place until dissolved (subject to dissolve delay), meaning the victim is silenced for the entire duration of any proposal created while the flood is active. No special privileges are required; any unprivileged user with sufficient tokens can execute this.

## Recommendation
1. Make `register_vote` async and introduce a `ProposalVotingStateMachine` equivalent for SNS, mirroring the NNS pattern in `rs/nns/governance/src/voting.rs`.
2. Add `is_over_instructions_limit` checks inside the BFS loop with state checkpointing, and resume processing in a timer job analogous to `process_voting_state_machines` in NNS.
3. Alternatively (or additionally), enforce a per-neuron cap on the number of neurons that may follow it (a "max followers per neuron" limit), bounding the BFS fan-out independently of `max_followees_per_function`.

## Proof of Concept
```
// 1. Attacker creates N neurons (N = max_number_of_neurons, up to 200,000)
for i in 0..N:
    stake(neuron_minimum_stake_e8s)
    claim_neuron(i)
    follow(attacker_neuron[i], target_neuron_id, function_id=<any>)

// 2. Someone creates a proposal (ballots allocated for all neurons)
proposal_id = make_proposal(...)

// 3. Victim tries to vote
register_vote(target_neuron_id, proposal_id, Vote::Yes)
// -> cast_vote_and_cascade_follow iterates over N followers synchronously
// -> BTreeMap lookups O(log N) per follower, vote_from_ballots_following per follower
// -> instruction limit exceeded -> message traps -> vote not recorded
```

A deterministic integration test using PocketIC can reproduce this by: (a) initializing an SNS with `max_number_of_neurons = 200_000` and `neuron_minimum_stake_e8s = 1`, (b) creating N attacker neurons all following a target neuron, (c) creating a proposal, and (d) asserting that `register_vote` from the target neuron returns a trap/error and the ballot remains `Unspecified`.
