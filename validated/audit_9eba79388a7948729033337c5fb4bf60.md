### Title
Zero Native Resource Charge on All EIP-2537 BLS12-381 Precompiles Enables Prover Resource Exhaustion - (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/`)

---

### Summary

All seven EIP-2537 BLS12-381 precompiles hardcode `cost_native = 0`, charging no native (proving) resource despite performing computationally intensive elliptic-curve operations. Because the block-level native limit check relies entirely on the accumulated `computational_native_used` counter, an unprivileged attacker can submit transactions that consume far more actual RISC-V proving cycles than the system accounts for, causing the prover to exceed `MAX_NATIVE_COMPUTATIONAL` on a block the sequencer accepted as valid.

---

### Finding Description

ZKsync OS implements a **double resource accounting** model: every operation charges both EVM gas (ergs) and a **native** resource that models the RISC-V proving cost. The native resource is the primary guard against prover overload; `MAX_NATIVE_COMPUTATIONAL = 1 << 35` (~34 billion cycles) is enforced at the block level.

All seven BLS12-381 precompiles introduced by EIP-2537 set `cost_native = 0`:

**G1/G2 Addition** (`addition.rs`):
```rust
let cost_native = 0;
resources.charge(&R::from_ergs_and_native(
    cost_ergs,
    <R::Native as zk_ee::system::Computational>::from_computational(cost_native),
))?;
``` [1](#0-0) [2](#0-1) 

**G1/G2 MSM** (`msm.rs`):
```rust
let cost_native = 0;
resources.charge(&R::from_ergs_and_native(cost_ergs, ...cost_native...))?;
``` [3](#0-2) [4](#0-3) 

**Pairing Check** (`pairing.rs`):
```rust
let cost_native = 0;
resources.charge(&R::from_ergs_and_native(cost_ergs, ...cost_native...))?;
``` [5](#0-4) 

**Field-to-Curve Mappings** (`mappings.rs`): [6](#0-5) [7](#0-6) 

Contrast this with every other cryptographic precompile, which charges substantial native costs:

```rust
pub const BN254_ECADD_NATIVE_COST: u64 = native_with_delegations!(46_000, 1650, 0);   // ~52,600
pub const BN254_ECMUL_NATIVE_COST: u64 = native_with_delegations!(600_000, 41_000, 0); // ~764,000
pub const BN254_PAIRING_BASE_NATIVE_COST: u64 = native_with_delegations!(13_000_000, 500_000, 0); // ~15,000,000
pub const ECRECOVER_NATIVE_COST: u64 = native_with_delegations!(350_000, 43_000, 0);  // ~522,000
``` [8](#0-7) 

BLS12-381 operations are **more expensive** than their BN254 counterparts due to larger field sizes (381-bit vs 254-bit). A BLS12-381 pairing check with `n` pairs costs `37700 + 32600*n` EVM gas but 0 native. With a block gas limit of ~30M gas, an attacker can include ~900 pairs in a single transaction, consuming an estimated 50–100× more RISC-V cycles per pair than BN254 pairing (which already costs ~15M native per pair). This yields an actual proving cost of tens of billions of cycles — well above `MAX_NATIVE_COMPUTATIONAL`.

The block-level guard that should prevent this:

```rust
} else if !cfg!(feature = "resources_for_tester")
    && computational_native_used > MAX_NATIVE_COMPUTATIONAL
{
    Err(InvalidTransaction::BlockNativeLimitReached)
``` [9](#0-8) 

…is bypassed entirely because `computational_native_used` never accumulates BLS12-381 costs. The constant:

```rust
pub const MAX_NATIVE_COMPUTATIONAL: u64 = 1 << 35;
``` [10](#0-9) 

…is never triggered for BLS12-381 work.

---

### Impact Explanation

**Valid-execution unprovability / resource accounting bug.** The sequencer (forward system) accepts a block containing BLS12-381-heavy transactions because EVM gas is within limits and native cost is reported as 0. The prover (RISC-V / Airbender) must then execute the same block and actually perform all BLS12-381 arithmetic, consuming RISC-V cycles that were never budgeted. The prover either:

1. Exceeds its segment/cycle budget and fails to generate a valid proof, halting the chain.
2. Produces a proof that diverges from the forward execution result (forward/proving divergence).

Either outcome is a critical liveness failure for the ZK rollup.

---

### Likelihood Explanation

**High.** Any unprivileged user can submit a standard EVM transaction calling precompile addresses `0x0b`–`0x11` (BLS12-381 precompiles) with maximum gas and crafted input. No privileged role, leaked key, or governance action is required. The attack is cheap: the attacker pays only EVM gas (which is refunded on OOG, or simply the cost of a high-gas transaction), while the prover bears unbounded computational cost. [11](#0-10) 

---

### Recommendation

Assign non-zero native costs to all BLS12-381 precompiles, calibrated against actual RISC-V cycle benchmarks (as done for BN254 and other precompiles). At minimum, BLS12-381 pairing should carry a per-pair native cost at least as large as `BN254_PAIRING_PER_PAIR_NATIVE_COST` (~15M), and G2 MSM/mapping operations should carry proportionally scaled costs. The existing `cost_constants.rs` pattern should be extended with BLS12-381 entries. [12](#0-11) 

---

### Proof of Concept

1. Deploy or use an existing contract that calls `address(0x0f)` (BLS12-381 pairing check) in a loop.
2. Submit a transaction with gas limit near the block gas limit (~30M gas).
3. Craft input with the maximum number of valid G1/G2 pairs (~900 pairs at 32,600 gas/pair + 37,700 fixed).
4. The sequencer accepts the transaction: EVM gas is within limits, native cost = 0.
5. The prover attempts to re-execute the block and must perform ~900 BLS12-381 pairings.
6. Actual RISC-V cycle consumption far exceeds `MAX_NATIVE_COMPUTATIONAL = 2^35`, causing the prover to fail or the block to be unprovable.

The same attack applies to G2 MSM (`0x0e`) with 128+ pairs, and to repeated G2 mapping calls (`0x11`), all of which charge 0 native. [13](#0-12) [14](#0-13)
