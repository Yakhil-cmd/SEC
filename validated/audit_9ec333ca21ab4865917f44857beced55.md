The code evidence is conclusive. Let me confirm the exact promise construction and the absence of any equivalent `error_refund` path for `ExitToEthereum`.

### Title
Missing Error-Callback on `ExitToEthereum` ETH Withdrawal Permanently Freezes Caller's EVM ETH Balance - (`engine-precompiles/src/native.rs`)

---

### Summary

`ExitToEthereum::run` for `flag=0x0` debits the caller's EVM ETH balance via `context.apparent_value`, then schedules a bare `PromiseArgs::Create(withdraw_promise)` with no error callback. If the NEAR-side `withdraw` call fails for any reason, the ETH is permanently frozen at the keyless precompile address `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` with no recovery path. The analogous `ExitToNear` precompile has an `error_refund` feature that attaches an `exit_to_near_precompile_callback` to handle failures and refund the EVM balance; `ExitToEthereum` has no equivalent, and the grep confirms zero callback/refund wiring for it anywhere in the codebase.

---

### Finding Description

In `ExitToEthereum::run`, the ETH path (`flag=0x0`) builds a `withdraw` promise and serialises it unconditionally as `PromiseArgs::Create`:

```rust
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [1](#0-0) 

There is no `PromiseArgs::Callback` wrapper, no `exit_to_ethereum_precompile_callback`, and no `RefundCallArgs` construction anywhere in the `ExitToEthereum` implementation.

Contrast with `ExitToNear`, which conditionally wraps the transfer promise in a `PromiseArgs::Callback` pointing at `exit_to_near_precompile_callback` whenever `callback_args != default()`: [2](#0-1) 

That callback invokes `engine::refund_on_error`, which transfers ETH back from the precompile address to the original caller on failure: [3](#0-2) 

The `error_refund` feature is defined in `engine-precompiles/Cargo.toml` and is only wired into `ExitToNear`; `ExitToEthereum` has no `#[cfg(feature = "error_refund")]` blocks at all: [4](#0-3) 

The `withdraw` call can fail due to a serialization mismatch: `get_withdraw_serialize_type` reads `WithdrawSerializeType` from storage and defaults to `Borsh` if the key is absent, but the connector contract may expect `Json` (or vice versa) after an upgrade: [5](#0-4) [6](#0-5) 

When the promise fails, the ETH transferred to the precompile address during the EVM execution is irrecoverable because the address has no private key and no on-chain refund path.

---

### Impact Explanation

The caller's EVM ETH balance is debited at EVM execution time (the value transfer from caller to precompile address is committed when the NEAR transaction succeeds). If the asynchronous `withdraw` promise then fails, the ETH sits permanently at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` with no mechanism to recover it. This is **permanent freezing of funds** (Critical), which is within the allowed impact scope.

---

### Likelihood Explanation

The most realistic non-admin trigger is a `WithdrawSerializeType` mismatch between what Aurora's storage says and what the deployed connector contract expects. This can occur during connector upgrades or migrations where the storage key is not atomically updated. It does not require any admin compromise — the mismatch can arise from a normal upgrade sequence. Any user who calls `ExitToEthereum` with ETH during such a window loses their funds permanently.

---

### Recommendation

Mirror the `ExitToNear` pattern for `ExitToEthereum`:

1. Add a `refund_address` field to the `flag=0x0` input (analogous to `ExitToNear`'s `error_refund` path).
2. Wrap the `withdraw_promise` in a `PromiseArgs::Callback` pointing at a new `exit_to_ethereum_precompile_callback` method on the engine contract.
3. In that callback, check `handler.promise_result(0)`: on failure, call `engine::refund_on_error` with `erc20_address: None` to transfer the ETH back from the precompile address to the caller.
