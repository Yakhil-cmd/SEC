Audit Report

## Title
Rebasing ERC-20 Token Rebase Rewards Permanently Stuck in ckETH Minter's Ethereum Address — (`rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`, `rs/ethereum/cketh/minter/src/state.rs`)

## Summary

The ckETH minter's `erc20_balances` is driven exclusively by audit events (deposit log values and withdrawal amounts) and is never reconciled against the minter's actual on-chain Ethereum balance. If a rebasing ERC-20 token such as stETH is added as a supported ckERC20 token via a legitimate NNS governance proposal, the minter's Ethereum-side balance grows with every Lido oracle rebase while the IC-side accounting never reflects this growth. The accumulated rebase rewards are permanently locked in the minter's Ethereum address with no sweep, rescue, or reconciliation mechanism.

## Finding Description

**Deposit path — amount recorded from event log, not actual received balance**

`depositErc20` in `DepositHelperWithSubaccount.sol` (L511–532) calls `safeTransferFrom(msg.sender, minterAddress, amount)` and then emits `ReceivedEthOrErc20(..., amount, ...)` where `amount` is the caller-supplied parameter. The IC minter scrapes this log and calls `update_balance_upon_deposit`, which adds `event.value` (the `amount` field from the log) to `erc20_balances` (state.rs L332–339). The field is explicitly documented as "Computed based on audit events" (state.rs L74–76) and is never reconciled against the minter's actual on-chain Ethereum balance.

**Withdrawal path — sends and subtracts exactly the user-requested amount**

When a user calls `withdraw_erc20`, the minter constructs an ERC-20 `transfer` call encoding `request.withdrawal_amount` (transactions/mod.rs L1175–1183). After finalization, `update_balance_upon_withdrawal` subtracts only that withdrawal amount from `erc20_balances` (state.rs L377–383).

**The accounting gap**

For a rebasing token like stETH, the minter's Ethereum address balance increases with every Lido oracle rebase. The minter's `erc20_balances` never increases to reflect this. The delta — accumulated rebase rewards — sits in the minter's Ethereum address indefinitely. A grep across the entire minter codebase confirms there is no `sweep`, `rescue`, or balance-reconciliation endpoint. The DID file (cketh_minter.did L217–226) even acknowledges "This might be less that the actual amount available on the `minter_address()`" for both `eth_balance` and `erc20_balances`, confirming the design does not guarantee parity — but provides no recovery path for the divergence.

**No validation at token-addition time**

`record_add_ckerc20_token` (state.rs L398–424) only checks for duplicate symbols and addresses. There is no check that the ERC-20 contract is non-rebasing, no allowlist of token types, and no documentation warning operators against adding rebasing tokens.

## Impact Explanation

Once a rebasing token is added as a supported ckERC20 token, every Lido oracle rebase cycle causes the minter's actual stETH balance on Ethereum to exceed the sum of all outstanding ckstETH. The excess is permanently unclaimable by any user or operator. ckstETH holders cannot claim their staking rewards (the rebase portion), and the `erc20_balances` metric reported by `get_minter_info` diverges from reality, misleading operators and auditors. This constitutes a **significant Chain Fusion / ck-token security impact with concrete and growing user and protocol harm**, matching the **High ($2,000–$10,000)** impact class: "Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation

Adding stETH as a supported ckERC20 token requires a standard NNS upgrade proposal targeting the ledger suite orchestrator — no malicious governance is required. A legitimate, well-intentioned proposal to support stETH (a top-5 DeFi token by TVL) would satisfy the precondition. Once the token is added, the vulnerability is triggered automatically by any unprivileged user making a normal deposit; no special attacker action is needed. Lido's stETH rebase is daily and automatic, so the divergence grows continuously from the moment of the first deposit. The precondition is realistic and does not require governance to act maliciously — it is the expected, legitimate use of the token-addition mechanism.

## Recommendation

1. **Reject rebasing tokens at the minter level**: In the `add_ckerc20_token` handler, validate that the ERC-20 contract does not implement rebasing semantics, or document explicitly that only non-rebasing tokens are supported and enforce this check.
2. **Use share-based accounting**: For tokens like stETH, record and transfer shares (via `transferShares`/`sharesOf`) rather than balance amounts.
3. **Prefer wrapped non-rebasing variants**: Only support `wstETH` (the non-rebasing wrapper of stETH) rather than stETH directly.
4. **Add a balance-reconciliation query**: Expose a mechanism to compare `erc20_balances` against the actual on-chain balance so that any divergence (from rebasing or fee-on-transfer tokens) is observable and actionable.

## Proof of Concept

1. An NNS proposal adds stETH (`0xae7ab96520DE3A18E5e111B5EaAb095312D7fE84`) as a supported ckERC20 token via the ledger suite orchestrator.
2. Alice calls `depositErc20(stETH_address, 10 ether, principal, subaccount)` on the deposit helper. The helper transfers 10 stETH to the minter's Ethereum address and emits `ReceivedEthOrErc20(..., 10 ether, ...)`.
3. The IC minter scrapes the log, mints 10 ckstETH to Alice, and sets `erc20_balances[stETH] = 10 ether`.
4. Lido's daily oracle rebase runs. The minter's stETH balance becomes `10.03 ether` (≈3% APR / 365 days accumulated over a year). `erc20_balances[stETH]` remains `10 ether`.
5. Alice calls `withdraw_erc20(10 ether, ckstETH_ledger, eth_address)`. The minter burns 10 ckstETH, sends 10 stETH to Alice, and sets `erc20_balances[stETH] = 0`.
6. The minter's Ethereum address still holds `~0.03 ether` worth of stETH. `erc20_balances[stETH] = 0` but actual on-chain balance > 0. These tokens are permanently stuck with no recovery path.

A deterministic integration test can reproduce this by: (a) deploying a mock rebasing ERC-20 that increases `balanceOf` without a transfer event, (b) adding it as a supported ckERC20 token, (c) simulating a deposit, (d) triggering a rebase by calling the mock's rebase function, (e) simulating a full withdrawal, and (f) asserting that the mock token's `balanceOf(minterAddress) > 0` while `erc20_balances[token] == 0`.
