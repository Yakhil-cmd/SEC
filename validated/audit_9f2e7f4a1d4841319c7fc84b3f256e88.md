### Title
No-Refund on Failed `exitToNear` Promise Permanently Freezes User ETH/ERC-20 Tokens - (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is not enabled, the `ExitToNear` precompile creates a fire-and-forget NEAR promise with no error-handling callback. If the downstream `ft_transfer` or `ft_transfer_call` promise fails (e.g., recipient account not registered with the NEP-141 contract), the user's EVM-side ETH or ERC-20 tokens have already been burned/deducted, but no refund is issued. The tokens are permanently frozen.

---

### Finding Description

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` handles two exit paths: ETH (base token) and ERC-20. In both cases, the EVM-side balance is reduced first (ETH deducted from the caller's balance, or ERC-20 tokens burned by the calling contract), and then a NEAR cross-contract promise is scheduled to transfer the corresponding NEP-141 tokens to the recipient.

The refund logic is gated behind the `error_refund` Cargo feature: [1](#0-0) 

When `error_refund` is **not** enabled, `refund` is set to `None`. The promise dispatch logic then checks whether the callback args equal the default (both `refund: None` and `transfer_near: None`): [2](#0-1) 

For a standard ETH or ERC-20 exit (no wNEAR unwrap, no OMNI message), `transfer_near` is also `None`, so `callback_args == default()` is `true`, and the promise is dispatched as a bare `PromiseArgs::Create` with **no callback**. There is no mechanism to detect failure or issue a refund.

The `exit_to_near_precompile_callback` function, which contains the refund logic, is only attached when `callback_args != default()`: [3](#0-2) 

The `refund_on_error` function that would re-mint ERC-20 tokens or transfer ETH back from the precompile address exists but is unreachable in this path: [4](#0-3) 

The test suite explicitly documents this behavior: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

When the `error_refund` feature is absent from the production build:

- A user calling `exitToNear` for ETH has their EVM ETH balance permanently reduced with no recovery path if the NEP-141 `ft_transfer` fails.
- A user calling `exitToNear` for an ERC-20 token has their ERC-20 tokens permanently burned with no re-mint if the NEP-141 `ft_transfer` fails.

The NEP-141 tokens remain in Aurora's account (or the transfer is simply rejected), while the user's EVM-side assets are gone. This is a **permanent freeze of user funds** — the same class as the original report's "ante stuck in the borrower contract."

---

### Likelihood Explanation

The failure condition is realistic and unprivileged:

- Any user can trigger it by calling `exitToNear` with a recipient NEAR account that is not registered with the NEP-141 contract (storage deposit not paid).
- The NEAR NEP-141 standard requires storage registration; unregistered recipients cause `ft_transfer` to fail.
- No admin action, oracle error, or governance capture is required.
- The attacker-controlled entry path is: submit an EVM transaction calling the `exitToNear` precompile address (`0x...exitToNear`) with a recipient that has no storage deposit on the target NEP-141 contract. [7](#0-6) 

---

### Recommendation

Ensure the `error_refund` feature is always enabled in production builds, or unconditionally attach the `exit_to_near_precompile_callback` as a callback on every exit promise so that failures are always detected and refunds are always issued. The refund logic already exists in `engine::refund_on_error`; the gap is only in whether the callback is scheduled. [8](#0-7) 

---

### Proof of Concept

The existing test `test_exit_to_near_refund` (ERC-20) and `test_exit_to_near_eth_refund` (ETH) already demonstrate the issue. When compiled **without** `error_refund`:

1. User has `FT_TRANSFER_AMOUNT` ERC-20 tokens on Aurora.
2. User calls `exitToNear` with recipient `"unregistered.near"` (not registered with the NEP-141).
3. ERC-20 tokens are burned from the user's EVM balance.
4. The `ft_transfer` NEAR promise fails (unregistered account).
5. No callback fires; no re-mint occurs.
6. User's ERC-20 balance is `FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT` — permanently reduced. [9](#0-8)
