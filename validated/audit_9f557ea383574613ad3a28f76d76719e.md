Audit Report

## Title
Uncached `encode_manifest` Re-Serialization on Every `ManifestChunk` Request Enables CPU Exhaustion by Byzantine Peer — (`rs/state_manager/src/state_sync/types.rs`)

## Summary
Every `ManifestChunk` request served by `StateSyncMessage::get_chunk` unconditionally calls `encode_manifest(&self.manifest)`, performing a full protobuf serialization of the manifest with no caching. The P2P chunk handler in `rs/p2p/state_sync_manager/src/routes/chunk.rs` dispatches each request to a `spawn_blocking` task with no per-peer rate limiting, no concurrency cap, and no deduplication. A Byzantine subnet peer can flood the serving replica with repeated manifest-chunk requests, causing sustained O(manifest_size) CPU work per request and potential exhaustion of the Tokio blocking thread pool.

## Finding Description
In `StateSyncMessage::get_chunk`, the `ManifestChunk(index)` branch unconditionally calls `encode_manifest(&self.manifest)` on every invocation: [1](#0-0) 

`encode_manifest` performs a full protobuf serialization, allocating a new `Vec<u8>` each time: [2](#0-1) 

`StateSyncMessage` holds only the raw `Manifest` with no cached encoded-bytes field: [3](#0-2) 

The chunk handler in `state_sync_chunk_handler` dispatches every incoming request directly to `state_sync.chunk()` via `tokio::task::spawn_blocking` with no per-peer rate limiting, no deduplication, and no concurrency cap: [4](#0-3) 

The router is assembled with no middleware layers for rate limiting or concurrency control: [5](#0-4) 

A search across all of `rs/p2p/state_sync_manager/src/` for `rate_limit`, `concurrency_limit`, `tower.*limit`, and `GlobalConcurrencyLimitLayer` returns zero matches, confirming the absence of any admission control on the serving side. The only `TOO_MANY_REQUESTS` reference in `chunk.rs` is in the *client-side* response parser (`parse_chunk_handler_response`), not in the server handler. [6](#0-5) 

## Impact Explanation
A Byzantine subnet peer (a single node below the consensus fault threshold) can send a continuous stream of `ManifestChunk` requests to the `/state-sync/chunk` endpoint. Each request causes a full protobuf serialization of the manifest in a `spawn_blocking` task. Saturating the Tokio blocking thread pool (default cap: 512 threads) with concurrent serializations will:
1. Consume sustained high CPU on the serving replica.
2. Delay or starve other blocking I/O operations (checkpoint reads, file chunk serving for legitimate state-sync peers).
3. Degrade the replica's consensus participation and legitimate state-sync serving capacity.

This matches the **High** bounty impact: *Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS.*

## Likelihood Explanation
The attacker must be a valid subnet peer — a single Byzantine node below the consensus fault threshold suffices. No admin key, governance majority, or threshold corruption is required. The exploit is mechanically trivial: send repeated POST requests to `/state-sync/chunk` with a valid `chunk_id` in the `ManifestChunk` range (`MANIFEST_CHUNK_ID_OFFSET` through `MANIFEST_CHUNK_ID_OFFSET + sub_manifest_hashes.len() - 1`). The attack is repeatable and requires no victim interaction.

## Recommendation
Cache the encoded manifest bytes inside `StateSyncMessage` so `encode_manifest` is called at most once per checkpoint regardless of how many `ManifestChunk` requests arrive — for example, using a `once_cell::sync::OnceCell<Vec<u8>>` or pre-computing an `Arc<Vec<u8>>` at construction time. Additionally, add per-peer request rate limiting or a global concurrency semaphore in `state_sync_chunk_handler` to bound the number of concurrent blocking serialization tasks.

```rust
pub(crate) async fn state_sync_chunk_handler<T: 'static>(
    State(state): State<Arc<StateSyncChunkHandler<T>>>,
    payload: Bytes,
) -> Result<Bytes, StatusCode> {
    // Parse payload
    let pb::StateSyncChunkRequest { id, chunk_id } =
        pb::StateSyncChunkRequest::decode(payload).map_err(|_| StatusCode::BAD_REQUEST)?;
    let artifact_id: StateSyncArtifactId = id.map(From::from).ok_or(StatusCode::BAD_REQUEST)?;
    let chunk_id = ChunkId::from(chunk_id);

    let jh =
        tokio::task::spawn_blocking(
            move || match state.state_sync.chunk(&artifact_id, chunk_id) {
                Some(data) => {
                    let pb_chunk = pb::StateSyncChunkResponse { data: data.take() };
                    let mut raw = BytesMut::with_capacity(pb_chunk.encoded_len());
                    pb_chunk.encode(&mut raw).expect("Allocated enough memory");
                    let raw = raw.freeze();

                    let compressed = zstd::bulk::compress(&raw, zstd::DEFAULT_COMPRESSION_LEVEL)
                        .expect("Compression failed");
                    state
                        .metrics
                        .compression_ratio
                        .observe(raw.len() as f64 / compressed.len() as f64);
                    Ok(compressed)
                }
                None => Err(StatusCode::NO_CONTENT),
            },
        );
    let data = jh.await.map_err(|_| StatusCode::INTERNAL_SERVER_ERROR)??;

    Ok(data.into())
}
```
