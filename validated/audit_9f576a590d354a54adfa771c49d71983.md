### Title
Whitelist Removal Race Condition Allows Silo Users to Bypass Access Revocation - (File: `engine/src/contract_methods/silo/mod.rs`)

---

### Summary

In Silo mode, a user whose Ethereum address is being removed from the `Address` whitelist can observe the pending `remove_entry_from_whitelist` NEAR transaction in the mempool and race it with a `submit` call to transfer their EVM tokens to another whitelisted address before the removal is finalized. The whitelist check and the removal are two independent, non-atomic NEAR transactions, creating a window of exploitation identical in structure to the reported Solidity race condition.

---

### Finding Description

Aurora Engine's Silo mode enforces access control through four whitelists (`Admin`, `EvmAdmin`, `Account`, `Address`). Before any EVM transaction is executed, `assert_access` is called inside `engine::submit`: [1](#0-0) 

This calls `silo::is_allow_submit`, which reads the current whitelist state from storage at the time of the `submit` call: [2](#0-1) 

The owner removes a user by calling the `remove_entry_from_whitelist` entrypoint: [3](#0-2) 

Which delegates to: [4](#0-3) 

These are two separate, independent NEAR transactions. NEAR's mempool is publicly observable via RPC. A user monitoring for a `remove_entry_from_whitelist` call targeting their address can immediately submit a `submit` transaction (an EVM `transfer` to another whitelisted address they control) before the removal transaction is included in a block. Because NEAR does not enforce ordering between transactions from different accounts within the same block, the user's `submit` can be processed first, and the whitelist check at that moment still returns `true`.

There is no atomic "check-and-remove" primitive, no nonce-based sequencing between admin and user transactions, and no mandatory pause step before removal.

---

### Impact Explanation

A user being evicted from the Silo whitelist can front-run the removal to transfer all their EVM-native ETH balance (or call ERC-20 `transfer` within the EVM) to a second address they control that remains whitelisted. The admin's intent to freeze the user's funds is completely defeated. The user retains full control of their assets under a different address, and the removal has zero practical effect on their token holdings.

This maps to: **High — Temporary freezing of funds** (the freeze never takes effect).

---

### Likelihood Explanation

- NEAR RPC nodes expose pending transactions; the mempool is observable without any special access.
- The attacker only needs to watch for `remove_entry_from_whitelist` calls targeting their own address — a trivial monitoring task.
- No elevated privilege, leaked key, or external dependency is required; the attacker is the token holder themselves.
- The attack requires submitting a single signed EVM transaction via `submit` before the removal lands, which is a standard operation for any Silo participant.
- Likelihood is **High** for any Silo deployment where users have a financial incentive to retain their token balances.

---

### Recommendation

The recommended mitigation mirrors the original report's suggestion: the owner should call `pause_contract` in a confirmed transaction before calling `remove_entry_from_whitelist`. The `pause_contract` entrypoint sets `state.is_paused = true`: [5](#0-4) 

And `require_running` inside `submit` will then reject all user transactions: [6](#0-5) 

However, this is not enforced by the protocol — it is an undocumented operational requirement. A stronger fix would be to add an atomic "pause-then-remove" entrypoint, or to enforce that `remove_entry_from_whitelist` can only be called while the contract is paused, making the safe sequence mandatory rather than advisory.

---

### Proof of Concept

1. Silo mode is active; both `WhitelistKind::Account` and `WhitelistKind::Address` whitelists are enabled.
2. Bob's NEAR account (`bob.near`) and EVM address (`0xBob`) are whitelisted. Bob holds 100 ETH in the Aurora EVM. A second address `0xBob2` (also whitelisted) is controlled by Bob.
3. Alice (the owner) submits `remove_entry_from_whitelist` targeting `0xBob` to the NEAR network. This transaction is visible in the mempool.
4. Bob observes the pending removal and immediately submits a `submit` call containing a signed EVM transaction: `transfer(0xBob2, 100 ETH)`. Bob's NEAR account (`bob.near`) is still whitelisted at this moment.
5. Bob's `submit` is included in the same block before Alice's removal (NEAR does not guarantee cross-account ordering).
6. `assert_access` reads the whitelist — `0xBob` is still present — and returns `Ok`. The transfer executes successfully. [7](#0-6) 

7. Alice's `remove_entry_from_whitelist` is then processed. `0xBob` is removed, but Bob's 100 ETH now sits at `0xBob2`, which remains whitelisted. The freeze is ineffective.

The existing test `test_submit_with_removing_entries` in `engine-tests/src/tests/silo.rs` demonstrates the sequential case (remove then submit fails), but does not test the race condition where submit precedes remove: [8](#0-7)

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}

pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })

#[test]
fn test_submit_with_removing_entries() {
    let (mut runner, signer, receiver) = initialize_transfer();
    let sender = utils::address_from_secret_key(&signer.secret_key);
    let caller: AccountId = CALLER_ACCOUNT_ID.parse().unwrap();
    let transaction = utils::transfer_with_price(
        receiver,
        TRANSFER_AMOUNT,
        INITIAL_NONCE.into(),
        ONE_GAS_PRICE.raw(),
    );

    set_silo_params(&mut runner, Some(SILO_PARAMS_ARGS));
    enable_all_whitelists(&mut runner);

    // Allow submitting transactions.
    add_account_to_whitelist(&mut runner, caller.clone());
    add_address_to_whitelist(&mut runner, sender);

    validate_address_balance_and_nonce(&runner, sender, INITIAL_BALANCE, INITIAL_NONCE.into())
        .unwrap();
    validate_address_balance_and_nonce(&runner, receiver, ZERO_BALANCE, INITIAL_NONCE.into())
        .unwrap();

    // perform transfer
    let result = runner
        .submit_transaction(&signer.secret_key, transaction.clone())
        .unwrap();
    assert!(matches!(result.status, TransactionStatus::Succeed(_)));

    // validate post-state
    validate_address_balance_and_nonce(
        &runner,
        sender,
        INITIAL_BALANCE - TRANSFER_AMOUNT - FIXED_GAS * ONE_GAS_PRICE,
        (INITIAL_NONCE + 1).into(),
    )
    .unwrap();
    validate_address_balance_and_nonce(&runner, receiver, TRANSFER_AMOUNT, INITIAL_NONCE.into())
        .unwrap();

    // Remove account id and address from whitelists.
    remove_account_from_whitelist(&mut runner, caller);
    remove_address_from_whitelist(&mut runner, sender);

    // perform transfer
    let err = runner
        .submit_transaction(&signer.secret_key, transaction)
        .unwrap_err();
    assert_eq!(err.kind, EngineErrorKind::NotAllowed);

    // validate post-state
    validate_address_balance_and_nonce(
        &runner,
        sender,
        INITIAL_BALANCE - TRANSFER_AMOUNT - FIXED_GAS * ONE_GAS_PRICE,
        (INITIAL_NONCE + 1).into(),
    )
    .unwrap();
    validate_address_balance_and_nonce(&runner, receiver, TRANSFER_AMOUNT, INITIAL_NONCE.into())
        .unwrap();
}
