Audit Report

## Title
Single Principal Can Exhaust Global `MAX_CONCURRENT_PENDING_REQUESTS` Queue, DoSing All Withdrawals — (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary

`retrieve_btc_with_approval` enforces a single global cap of 5,000 pending withdrawal requests with no per-principal sub-limit. The `retrieve_btc_guard` only prevents duplicate *concurrent in-flight* calls for the same `Account`; it is dropped as soon as each call returns `Ok`, allowing the same principal to sequentially accumulate unlimited entries in the pending queue. A single funded attacker can fill the entire queue, causing every subsequent legitimate caller to receive `TemporarilyUnavailable` until the minter drains all 5,000 requests.

## Finding Description

**Code path:**

`retrieve_doge_with_approval` in `rs/dogecoin/ckdoge/minter/src/main.rs` (L131–143) delegates directly to `retrieve_btc_with_approval` in `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`.

Inside `retrieve_btc_with_approval`, the guard is acquired at L263:

```rust
let _guard = retrieve_btc_guard(caller_account)?;
```

`retrieve_btc_guard` is implemented in `rs/bitcoin/ckbtc/minter/src/guard.rs`. It inserts `caller_account` into `state.retrieve_btc_accounts: BTreeSet<Account>` and returns `AlreadyProcessing` if the same `Account` is already present, or `TooManyConcurrentRequests` if the set already has `MAX_CONCURRENT = 100` entries. Critically, the guard is a RAII struct — it is **dropped when `retrieve_btc_with_approval` returns**, removing the account from the set. [1](#0-0) [2](#0-1) [3](#0-2) 

After the guard is dropped, the same `Account` can immediately submit another request. Each successful call appends a `RetrieveBtcRequest` to `state.pending_retrieve_btc_requests` via `accept_retrieve_btc_request`. [4](#0-3) 

The global throttle at L274–279 checks only the aggregate count across all principals:

```rust
if read_state(|s| s.count_incomplete_retrieve_btc_requests() >= MAX_CONCURRENT_PENDING_REQUESTS)
``` [5](#0-4) 

`count_incomplete_retrieve_btc_requests` sums `pending_retrieve_btc_requests.len()` + `requests_in_flight.len()` + submitted transaction requests — with no per-principal breakdown. [6](#0-5) 

**Attack sequence:**

1. Attacker acquires `5000 × min_amount` ckDOGE.
2. Attacker calls `icrc2_approve` + `retrieve_doge_with_approval` sequentially (or in batches of up to 100 using different subaccounts). Each call succeeds, drops the guard, and enqueues a `RetrieveBtcRequest`.
3. After 5,000 successful calls the global counter reaches `MAX_CONCURRENT_PENDING_REQUESTS`.
4. Every subsequent call from any principal returns `TemporarilyUnavailable("too many pending retrieve_btc requests")`.
5. The attacker's ckDOGE is burned but they receive DOGE back once the minter processes the queue. Net cost: 5,000 × `LEDGER_TRANSFER_FEE`.

**Why existing guards fail:**

- `retrieve_btc_guard` prevents only *concurrent* duplicate calls for the same `Account` (max 100 in-flight simultaneously). It does not limit how many *completed* requests a principal has in the pending queue.
- There is no per-principal counter in `CkBtcMinterState` and no check against `retrieve_btc_account_to_block_indices` before accepting a new request. [7](#0-6) 

## Impact Explanation

This is a complete withdrawal DoS on the ckDOGE (and ckBTC) chain-fusion bridge. All users are blocked from converting ck-tokens back to the native asset for the entire duration of queue drainage. On the Dogecoin network, with its characteristically slow confirmation times, this window can be extended significantly. This matches the allowed impact: **Application/platform-level DoS with concrete user and protocol harm** — **High ($2,000–$10,000)**.

## Likelihood Explanation

The attack requires only a sufficient ckDOGE balance (obtainable by depositing DOGE through the normal bridge flow) and the ability to call a public update endpoint. No privileged access, governance majority, or cryptographic material is needed. The economic cost is limited to ledger transfer fees (5,000 × `LEDGER_TRANSFER_FEE`), which is negligible relative to the disruption. The attack is repeatable: once the queue drains, the attacker can refill it.

## Recommendation

Introduce a **per-principal pending request limit** (e.g., 10–50 requests per principal) enforced inside `retrieve_btc_with_approval` before the global check. Track per-principal pending counts using the existing `retrieve_btc_account_to_block_indices` map or a dedicated counter in `CkBtcMinterState`. This prevents any single actor from monopolising the global queue while preserving the global cap as a secondary safety valve.

## Proof of Concept

State-machine test sketch (safe, local, deterministic):

```rust
// Fill the queue from a single attacker principal using sequential calls
for i in 0..MAX_CONCURRENT_PENDING_REQUESTS {
    let subaccount = Some([i as u8; 32]); // distinct subaccounts bypass concurrent guard
    icrc2_approve(attacker, min_amount, minter_id, subaccount);
    assert!(
        minter.retrieve_doge_with_approval(attacker, min_amount, addr, subaccount).is_ok()
    );
}

// Legitimate user is now blocked
let result = minter.retrieve_doge_with_approval(victim, min_amount, addr, None);
assert_eq!(
    result,
    Err(RetrieveDogeWithApprovalError::TemporarilyUnavailable(
        "too many pending retrieve_btc requests".to_string()
    ))
);
```

The global counter check at `retrieve_btc.rs` L274 fires for the victim while the attacker's 5,000 requests remain unprocessed. The test is reproducible in a PocketIC or state-machine test environment without touching mainnet. [8](#0-7) [9](#0-8)

        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {

            })
        })
