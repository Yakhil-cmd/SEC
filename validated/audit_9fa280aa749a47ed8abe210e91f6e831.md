Audit Report

## Title
Unprivileged Caller Can DoS Arbitrary Victim Accounts via Attacker-Controlled `owner` in `balance_update_guard` — (`rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary
`update_balance` constructs the guard key from the caller-supplied `args.owner` field with no authorization check, allowing any unprivileged principal to acquire guard slots for arbitrary victim accounts. By submitting 100 concurrent calls each specifying a distinct victim principal as `owner`, an attacker can fill `update_balance_accounts` to `MAX_CONCURRENT=100`, causing legitimate owners of those accounts to receive `AlreadyProcessing` or `TemporarilyUnavailable` errors for the duration of the attacker's in-flight calls. The attacker can re-submit continuously to maintain a persistent DoS against ckBTC minting.

## Finding Description
In `update_balance` (`rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`, lines 164–168), the account passed to `balance_update_guard` is derived from the ingress-supplied `args.owner`:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),  // attacker-controlled
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(caller_account)?;
```

`args.owner` is an `Option<Principal>` in `UpdateBalanceArgs` with no authorization check — any principal can set it to any value. The only pre-guard checks are: (1) line 149 traps if the resolved owner equals the minter's own ID, and (2) line 159 checks `caller` (not `args.owner`) against the deposit whitelist mode. Neither prevents a caller from specifying an arbitrary victim principal.

The guard is then held across the `get_utxos` inter-canister call at lines 175–183, which has non-trivial latency (Bitcoin canister round-trips, routinely several seconds). Because the IC allows multiple update calls to be in-flight concurrently at `await` points, an attacker can submit 100 calls in rapid succession — each with a distinct victim `owner` — before any complete. Each call acquires a guard slot for the victim's account.

`Guard::new` (guard.rs lines 45–60) enforces:
- `AlreadyProcessing` — if the exact account is already in the set
- `TooManyConcurrentRequests` — if `accounts.len() >= MAX_CONCURRENT` (100)

Both conditions are triggered against legitimate victims once the attacker's calls are in-flight.

## Impact Explanation
This is a High-severity application-level DoS against ckBTC, an in-scope Chain Fusion financial integration. Any of the 100 targeted victim principals calling `update_balance` for their own account receives `AlreadyProcessing` (their specific account is locked) or `TemporarilyUnavailable` (set is full). This blocks ckBTC minting for those accounts for the entire duration of the attacker's calls. The attacker can cycle through victim sets continuously to maintain persistent DoS. No funds are permanently lost or stolen, placing this in the High tier: "Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."

## Likelihood Explanation
No privileged access is required — any IC principal can call `update_balance` with arbitrary `owner`. Bitcoin canister `get_utxos` latency is routinely several seconds, easily long enough for 100 sequential ingress submissions to all be in-flight simultaneously at the await point. The attacker can cycle through victim sets continuously to maintain persistent DoS. Cost to attacker is bounded by cycles for 100 inter-canister calls per rotation, which is economically feasible.

## Recommendation
Restrict the guard key to the authenticated caller, not the caller-supplied `owner`. A minimal fix:

```rust
// Always use authenticated caller for the guard key
let guard_account = Account {
    owner: caller,
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(guard_account)?;
// caller_account (with args.owner) is still used for address derivation and minting
```

If third-party calls (caller ≠ owner) must be supported, add a per-caller rate limit separate from the per-account guard, so the guard's DoS-prevention property is tied to the authenticated caller rather than the attacker-supplied argument.

## Proof of Concept
State-machine test outline:
1. Initialize minter with a mock Bitcoin canister that delays `get_utxos` responses by several seconds.
2. From a single attacker principal, submit 100 `update_balance` calls each with `owner = Some(victim_N)` for N in 0..100.
3. Assert all 100 calls are in-flight (guards held, `update_balance_accounts.len() == 100`).
4. From `victim_0`'s own principal, call `update_balance(owner=None)` (which resolves to `victim_0`).
5. Assert the result is `Err(UpdateBalanceError::AlreadyProcessing)` or `Err(TemporarilyUnavailable("too many concurrent requests"))`.

This directly matches the existing test structure in `guard.rs` lines 141–162, which already demonstrates that filling the set to 100 blocks further entries — the missing piece is that the set can be filled by a single attacker using distinct victim `owner` values.
