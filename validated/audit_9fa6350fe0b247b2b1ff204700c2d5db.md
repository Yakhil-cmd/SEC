### Title
Permanent Fund Loss When `ft_transfer` Fails in `exit_to_near` Precompile Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

---

### Summary

The production Aurora Engine build compiles without the `error_refund` feature flag. As a result, when a user calls `withdrawToNear` on an ERC-20 mirror contract, the ERC-20 tokens are burned immediately, but if the subsequent `ft_transfer` cross-contract call to the NEP-141 contract fails (e.g., unregistered recipient, or a NEP-141 with access controls that block the recipient), no refund callback is scheduled. The burned ERC-20 tokens are permanently destroyed and the corresponding NEP-141 tokens remain locked inside the Aurora contract forever.

---

### Finding Description

**Step 1 — ERC-20 tokens are burned before the cross-contract call.**

In `EvmErc20.sol`, `withdrawToNear` unconditionally burns the caller's tokens before invoking the precompile:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // tokens destroyed here
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    assembly { let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, ...) }
}
``` [1](#0-0) 

**Step 2 — The precompile builds `callback_args` with `refund: None` in the production build.**

In `engine-precompiles/src/native.rs`, the `ExitToNearPrecompileCallbackArgs` struct is populated. The `refund` field is gated behind `#[cfg(feature = "error_refund")]`. When that feature is absent, `refund` is always `None`:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [2](#0-1) 

For the legacy ERC-20 exit path, `transfer_near` is also `None` (it is only `Some` for the wNEAR unwrap case), so `callback_args == ExitToNearPrecompileCallbackArgs::default()`.

**Step 3 — No callback is scheduled when `callback_args` is default.**

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // fire-and-forget, no callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [3](#0-2) 

The `ft_transfer` call is dispatched as a fire-and-forget promise. If it fails, there is no `exit_to_near_precompile_callback` invocation, so `refund_on_error` is never called and no ERC-20 re-mint occurs. [4](#0-3) 

**Step 4 — The production build does not include `error_refund`.**

`Makefile.toml` defines the production feature set as `contract` only:

```toml
CARGO_FEATURES_BUILD = "contract"
``` [5](#0-4) 

The `error_refund` feature is defined but not part of `default` or `contract`: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

When `ft_transfer` to the NEP-141 contract fails:
- The user's ERC-20 mirror tokens are **permanently burned** (destroyed on the EVM side).
- The NEP-141 tokens remain **permanently locked** inside the Aurora contract with no recovery path.
- There is no admin rescue function, no retry mechanism, and no refund callback.

This constitutes **permanent freezing/destruction of user funds** — a Critical impact.

---

### Likelihood Explanation

The failure condition is realistic and reachable by any unprivileged EVM user:

1. **Unregistered recipient**: NEP-141 contracts following the NEP-141 standard require storage registration. If a user specifies a NEAR account that has not called `storage_deposit` on the NEP-141 contract, `ft_transfer` will revert. This is a common operational mistake.
2. **NEP-141 with access controls**: Any NEP-141 token that implements a blocklist or allowlist (analogous to the SnailBrook Token in the original report) will cause `ft_transfer` to fail for blocked recipients.
3. **No warning or guard**: The ERC-20 `withdrawToNear` function provides no pre-check; the burn is irreversible before the cross-contract call result is known.

The test suite explicitly documents this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [8](#0-7) 

---

### Recommendation

1. **Enable `error_refund` in the production build** by adding it to `CARGO_FEATURES_BUILD` in `Makefile.toml`:
   ```toml
   CARGO_FEATURES_BUILD = "contract,error_refund"
   ```
   This activates the existing `refund_on_error` path in `engine/src/engine.rs` which re-mints burned ERC-20 tokens when `ft_transfer` fails. [9](#0-8) 

2. **Ensure ERC-20 contracts pass the refund address**: `EvmErc20V2.sol` already encodes `sender` in the precompile input (required by the `error_refund` input format), while the legacy `EvmErc20.sol` does not. Deployments should use `EvmErc20V2.sol`. [10](#0-9) 

3. **Add a pre-flight check** in `withdrawToNear` to verify the recipient is registered with the NEP-141 contract before burning, or restructure the flow to burn only after a successful transfer confirmation.

---

### Proof of Concept

1. Deploy a NEP-141 token and bridge it to Aurora (creating an ERC-20 mirror).
2. As a user, acquire ERC-20 mirror tokens on Aurora.
3. Call `withdrawToNear(recipient_bytes, amount)` where `recipient_bytes` encodes a NEAR account that has **not** called `storage_deposit` on the NEP-141 contract (or is on a blocklist).
4. Observe: ERC-20 tokens are burned (balance decreases to zero), the `ft_transfer` cross-contract call fails silently (no callback), and the NEP-141 tokens remain in the Aurora contract.
5. The user has no ERC-20 tokens and cannot recover the NEP-141 tokens. Funds are permanently lost.

This is confirmed by the existing test `test_exit_to_near_refund` which demonstrates that without `error_refund`, the ERC-20 balance after a failed `ft_transfer` is `FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT` (i.e., the burned amount is not refunded): [11](#0-10)
