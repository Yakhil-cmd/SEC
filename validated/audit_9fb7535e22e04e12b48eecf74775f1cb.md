Audit Report

## Title
Voting Power Snapshot Task Permanently Freezes After First Spike, Disabling Anti-Frontrunning Defense — (`rs/nns/governance/src/timer_tasks/snapshot_voting_power.rs`)

## Summary

`SnapshotVotingPowerTask::execute` guards against recording spike snapshots by checking whether the **latest stored** snapshot is already a spike. Once a spike snapshot is written, every subsequent daily execution sees the same stored spike and returns early, permanently halting new snapshot collection. After `MAXIMUM_STALENESS_SECONDS` (3 months), all stored snapshots are filtered out as stale, silently disabling spike detection. An attacker who can cause a single spike snapshot to be recorded can then submit NNS proposals with inflated voting power that bypasses the anti-frontrunning mechanism entirely.

## Finding Description

`SnapshotVotingPowerTask::execute` calls `is_latest_snapshot_a_spike` before deciding whether to record a new snapshot: [1](#0-0) 

`is_latest_snapshot_a_spike` reads the **last stored entry** from `voting_power_totals` and checks whether that stored value exceeds 1.5× the minimum across all stored snapshots: [2](#0-1) 

Because the task never computes the **current** voting power before deciding to skip, and because the spike snapshot remains the latest stored entry, every subsequent daily execution sees the same spike and returns early — indefinitely. The unit test explicitly documents and asserts this freeze: [3](#0-2) 

`MAXIMUM_STALENESS_SECONDS` is 3 months: [4](#0-3) 

The staleness filter inside `totals_entry_with_minimum_total_potential_voting_power_if_voting_power_spiked` silently drops all entries older than this threshold: [5](#0-4) 

Once all entries are dropped, `min_by_key` returns `None` via `?`, so `totals_entry_with_minimum_total_potential_voting_power_if_voting_power_spiked` returns `None`, and `previous_ballots_if_voting_power_spike_detected` returns `None` for any input. The test file explicitly documents this outcome: [6](#0-5) 

`compute_ballots_for_standard_proposal` then falls through to the "normal case" branch, using the live (potentially inflated) voting power: [7](#0-6) 

No production code path calls `VotingPowerSnapshots::clear()`. The only call to `clear()` is in a test marked for removal: [8](#0-7) 

## Impact Explanation

The voting power snapshot mechanism is the NNS governance canister's primary defense against flash-loan-style attacks on proposal ballot creation. Once the snapshots are stale, `compute_ballots_for_standard_proposal` uses the live neuron store directly for every new proposal. An attacker holding inflated voting power at that moment has ballots created from that inflated power, potentially allowing them to tip or control the outcome of NNS proposals — including proposals governing subnet upgrades, canister installations, and economic parameters of the Internet Computer. This matches the **High** impact class: unauthorized influence over governance assets and NNS proposal outcomes with meaningful constraints (capital requirement).

## Likelihood Explanation

The attacker must acquire enough ICP to push total potential voting power above 1.5× the historical minimum, stake it in neurons with sufficient dissolve delay, wait up to 24 hours for the daily task to record the spike snapshot, and then hold the position for 3 months until all prior snapshots expire. The capital requirement is substantial (roughly 50% of the entire currently-staked ICP supply), making this a low-to-medium likelihood attack. However, it is fully permissionless — no privileged role is required — and the attack is deterministic once the spike snapshot is recorded.

## Recommendation

Replace the pre-skip guard with a check on the **current** computed voting power rather than the latest stored snapshot. Specifically, `execute` should:

1. Compute the current `VotingPowerSnapshot`.
2. Check whether that current snapshot would be a spike relative to the stored history.
3. Skip recording only if the **current** power is a spike (to avoid polluting history with spike data).
4. If the current power is normal, record it — even if the latest stored snapshot was a spike — so the history can recover.

Additionally, introduce a bounded recovery window: if no snapshot has been recorded for longer than 2× `VOTING_POWER_SNAPSHOT_INTERVAL`, force a new snapshot regardless of spike status, to prevent permanent freeze.

## Proof of Concept

```
T=0:   Attacker acquires large ICP stake; neurons created with sufficient dissolve delay.
T=24h: SnapshotVotingPowerTask runs.
       is_latest_snapshot_a_spike() → false (no prior spike).
       Snapshot recorded: total_potential_voting_power = SPIKE_VALUE (>1.5× historical min).
T=48h: Task runs again.
       is_latest_snapshot_a_spike() → true (latest stored = SPIKE_VALUE).
       Task returns early. No new snapshot recorded.
T=48h … T=90d: Task runs daily, always returns early. Snapshots frozen.
T=90d+1: All stored snapshots age past MAXIMUM_STALENESS_SECONDS.
          totals_entry_with_minimum_total_potential_voting_power_if_voting_power_spiked()
          → None (all entries filtered by staleness).
          previous_ballots_if_voting_power_spike_detected() → None for any input.
T=90d+1: Attacker submits NNS proposal.
          compute_ballots_for_standard_proposal() takes the "normal case" branch.
          Ballots created from live neuron store with SPIKE_VALUE voting power.
          Attacker's inflated voting power is unchecked.
```

A deterministic unit test can reproduce this by: (1) filling 7 snapshots with normal values, (2) recording one spike snapshot, (3) advancing time by 1 day and asserting the task skips (already asserted in the existing test), (4) advancing time by `MAXIMUM_STALENESS_SECONDS + 1` and asserting `previous_ballots_if_voting_power_spike_detected` returns `None` for `u64::MAX` (already asserted in `voting_power_snapshots_tests.rs` line 111–114), and (5) asserting `compute_ballots_for_standard_proposal` uses the live snapshot.

        // If there are no snapshots, then there is no spike.

            return false;
