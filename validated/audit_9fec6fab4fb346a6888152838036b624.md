### Title
Underestimated `EXIT_TO_NEAR_CALLBACK_GAS` Causes Permanent ERC-20 Token Loss When `ft_transfer` Fails - (`engine-precompiles/src/native.rs`)

### Summary

The `exit_to_near` precompile schedules `exit_to_near_precompile_callback` with only `EXIT_TO_NEAR_CALLBACK_GAS = 10 TGas`. When the base `ft_transfer` promise fails and the `error_refund` feature is enabled, the callback must execute `refund_on_error`, which runs a full EVM execution (re-minting burned ERC-20 tokens). This EVM execution requires significantly more than 10 TGas due to cold NEAR storage reads and EVM bytecode execution overhead. If the callback runs out of gas, the ERC-20 tokens are permanently burned with no refund issued.

### Finding Description

The `exit_to_near` precompile in `engine-precompiles/src/native.rs` burns ERC-20 tokens in the EVM and schedules a two-step NEAR promise: a base `ft_transfer` call followed by a callback `exit_to_near_precompile_callback`. [1](#0-0) 

The callback gas constant is set to 10 TGas, with a comment indicating it was "determined experimentally based on tests" — not under cold-storage conditions. [2](#0-1) 

When the `error_refund` feature is enabled, the callback is always created with a `RefundCallArgs` payload. If the base `ft_transfer` fails (e.g., recipient not registered with the NEP-141 contract), the callback takes the refund branch: [3](#0-2) 

`refund_on_error` runs a full EVM execution — loading ERC-20 contract code from cold NEAR storage, executing the `mint` function, and writing storage: [4](#0-3) 

A single cold NEAR storage read costs ~5.5 TGas. The callback must perform: `with_hashchain` (reads/writes hashchain state), `state::get_state` (reads engine state), `io.read_input_borsh` (reads input), `handler.promise_result` (reads promise result), and then the full EVM execution in `refund_on_error` (loads ERC-20 bytecode, executes `mint`, writes ERC-20 storage). This chain of cold reads and EVM execution far exceeds 10 TGas.

For comparison, the XCC router's `ROUTER_EXEC_BASE` for a simple promise dispatch is already 7 TGas, and `WITHDRAW_GAS` for a wNEAR withdrawal is 40 TGas: [5](#0-4) 

When the callback runs OOG, NEAR rolls back only the callback receipt's state changes. The ERC-20 burn (in the original `submit` receipt) and the failed `ft_transfer` (in its own receipt) are not rolled back. The user's tokens are permanently destroyed.

The `EXIT_TO_NEAR_GAS` cost for calling the precompile is also set to 0 with an unresolved TODO, confirming the gas accounting for this precompile has never been fully analyzed: [6](#0-5) 

### Impact Explanation

**Critical — Permanent freezing of funds.** A user who calls `exit_to_near` to bridge ERC-20 tokens to NEAR, where the recipient NEAR account is not registered with the NEP-141 contract, will have their ERC-20 tokens permanently burned with no refund. The `error_refund` feature exists precisely to prevent this, but the insufficient callback gas defeats it. The loss is irreversible because NEAR receipt atomicity means only the failed callback's changes are rolled back, not the prior burn.

### Likelihood Explanation

**Medium.** The `error_refund` feature must be enabled in the production build (it is present in production code paths, not test-only). The triggering condition — `ft_transfer` failing due to an unregistered recipient — is a realistic user error. The gas underestimation is structural: the constant was measured under warm-storage test conditions, not cold-storage production conditions, mirroring exactly the methodology flaw in the referenced report.

### Recommendation

Benchmark `exit_to_near_precompile_callback` under cold-storage conditions (isolated transaction, no prior warm-up) for the refund path. Increase `EXIT_TO_NEAR_CALLBACK_GAS` to cover the worst-case cost of `refund_on_error`, including cold reads of engine state, ERC-20 contract code, and ERC-20 storage slots, plus EVM execution overhead. A value in the range of 50–100 TGas (similar to `WITHDRAWAL_GAS`) should be validated empirically.

### Proof of Concept

1. Enable the `error_refund` feature in the Aurora Engine build.
2. Bridge a NEP-141 token to Aurora as an ERC-20.
3. Call the `exit_to_near` precompile from the ERC-20 contract, targeting a NEAR account that is **not registered** with the NEP-141 contract.
4. The ERC-20 tokens are burned in the EVM (state committed in the `submit` receipt).
5. The `ft_transfer` promise fails (recipient not registered).
6. `exit_to_near_precompile_callback` is dispatched with 10 TGas.
7. The callback enters the `refund_on_error` branch, which calls `engine.call()` to re-mint the burned tokens — a full EVM execution requiring cold reads of engine state and ERC-20 contract code.
8. The callback exhausts its 10 TGas budget and fails with OOG.
9. No refund is issued. The ERC-20 tokens are permanently destroyed. [7](#0-6) [8](#0-7) [9](#0-8)

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
