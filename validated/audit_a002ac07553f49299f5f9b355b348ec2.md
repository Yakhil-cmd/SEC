Audit Report

## Title
Unbounded Synchronous Neuron Iteration in `compute_ballots_for_new_proposal` Enables Instruction-Limit DoS on SNS Proposal Submission - (File: rs/sns/governance/src/governance.rs)

## Summary

The SNS Governance canister's `compute_ballots_for_new_proposal` function iterates synchronously over the entire `self.proto.neurons` map within a single execution slice on every proposal submission. Any neuron holder with `SubmitProposal` permission can trigger this path. If the neuron count grows large enough to exhaust the 40-billion-instruction per-message limit, every subsequent call to `make_proposal` traps, permanently preventing new proposals and freezing SNS governance. The NNS governance canister solved this identical problem via a pre-computed voting power snapshot; SNS governance has no equivalent mechanism and no benchmark validating instruction safety at scale.

## Finding Description

**Root cause — unbounded synchronous loop:**

`compute_ballots_for_new_proposal` at `rs/sns/governance/src/governance.rs` lines 5255–5280 iterates over every entry in `self.proto.neurons` with no batching, pagination, or pre-computed snapshot:

```rust
for (k, v) in self.proto.neurons.iter() {
    if v.dissolve_delay_seconds(now_seconds) < min_dissolve_delay_for_vote {
        continue;
    }
    let voting_power = v.voting_power(...);
    total_power += voting_power as u128;
    electoral_roll.insert(k.clone(), Ballot { ... });
}
```

The cost is O(N) in the number of neurons, executed entirely within one update-call execution slice.

**Reachable call path:**

`make_proposal` at lines 3557–3559 calls `compute_ballots_for_new_proposal` unconditionally after standard unprivileged checks (valid neuron, `SubmitProposal` permission, sufficient dissolve delay, stake ≥ `reject_cost_e8s`). No privileged role is required.

**No instruction-limit guard and no SNS benchmark:**

The SNS governance benchmarks file (`rs/sns/governance/src/governance/benches.rs`) contains only a trivial placeholder `bench_example` function — there is no benchmark for `compute_ballots_for_new_proposal` at any neuron count. By contrast, the NNS governance benchmark `compute_ballots_for_new_proposal_with_stable_neurons` explicitly projects instruction usage at `MAX_NUMBER_OF_NEURONS` (200,000) and validates it stays under 25 billion instructions — a guard that SNS entirely lacks.

**NNS fix that SNS lacks:**

NNS governance avoids this problem by using `VOTING_POWER_SNAPSHOTS` and `compute_voting_power_snapshot_for_standard_proposal`, which pre-compute voting power outside the proposal submission path. SNS governance has no equivalent snapshot mechanism.

**Growing neuron map:**

`self.proto.neurons` is a `BTreeMap<String, Neuron>` in heap memory bounded by the configurable `max_number_of_neurons` parameter in `NervousSystemParameters`. Large SNS token swaps directly populate this map at scale via `claim_swap_neurons`.

## Impact Explanation

When `self.proto.neurons.len()` reaches a value where iterating over all neurons exceeds the 40-billion-instruction per-message limit, every call to `make_proposal` traps. Since `make_proposal` is the sole path for submitting governance proposals, the SNS governance canister becomes permanently unable to accept new proposals without NNS-level intervention. This matches the allowed High impact: **Application/platform-level DoS** — specifically, permanent loss of SNS governance availability, blocking all upgrade proposals, parameter changes, and treasury disbursements.

## Likelihood Explanation

The likelihood is medium-to-high for mature SNS deployments. A large token swap with tens of thousands of participants creates a proportional number of neurons via `claim_swap_neurons`, organically pushing the neuron count toward `max_number_of_neurons`. An attacker with sufficient SNS tokens can also deliberately create many neurons (one per principal) to accelerate this. The NNS benchmark projects ~25 billion instructions at 200,000 neurons for comparable per-neuron work; the SNS loop performs equivalent operations (dissolve delay check, voting power computation, BTreeMap insert), placing the instruction-limit crossing well within the configurable neuron count range.

## Recommendation

1. Adopt the same snapshot-based approach used by NNS governance: maintain a periodically-updated `VotingPowerSnapshot` in SNS governance and use it in `compute_ballots_for_new_proposal` instead of iterating over `self.proto.neurons` inline.
2. Alternatively, enforce a hard cap on `max_number_of_neurons` that is validated against the instruction budget, backed by a benchmark analogous to `compute_ballots_for_new_proposal_with_stable_neurons` in NNS governance.
3. Add a `compute_ballots_for_new_proposal` benchmark to `rs/sns/governance/src/governance/benches.rs` that projects instruction usage at the maximum allowed neuron count and fails CI if the projection exceeds a safe threshold.

## Proof of Concept

**Preconditions:**
- An SNS with `max_number_of_neurons` set to a large value (e.g., 100,000+).
- Neuron count approaching that limit, achievable via a large swap or deliberate neuron creation by an attacker.

**Trigger steps:**
1. Organic growth or attacker causes `self.proto.neurons.len()` to reach value N where iterating over N neurons exceeds 40B instructions.
2. Any principal with a valid neuron calls `manage_neuron` with `MakeProposal` command (e.g., a `Motion` proposal).
3. Execution enters `make_proposal` → `compute_ballots_for_new_proposal` → `for (k, v) in self.proto.neurons.iter()` exhausts the instruction budget.
4. The update call traps with an instruction-limit-exceeded error.
5. All subsequent `make_proposal` calls also trap — governance is permanently frozen.

**Reproducible test plan:** Write a PocketIC integration test that creates an SNS governance canister, populates `self.proto.neurons` to a large count via `claim_swap_neurons`, then calls `make_proposal` and asserts the call traps with an instruction limit error. Alternatively, extend the existing SNS canbench setup to benchmark `compute_ballots_for_new_proposal` at 10,000, 50,000, and 100,000 neurons and observe the instruction count trend.
