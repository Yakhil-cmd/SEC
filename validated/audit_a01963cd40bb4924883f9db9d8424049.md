### Title
wNEAR Permanently Locked in Engine Implicit Address When XCC Router `deploy_upgrade` Fails During Upgrade Path — (`engine/src/xcc.rs`, `engine/src/contract_methods/xcc.rs`)

---

### Summary

When a user's XCC router exists but is outdated (`DeployNeeded { create_needed: false }`) and the user attaches NEAR to their XCC call (`required_near > 0`), the XCC precompile synchronously debits wNEAR from the user via `transferFrom` into the engine's implicit EVM address. A promise chain is then scheduled to unwrap that wNEAR. However, if `deploy_upgrade` fails (e.g., gas exhaustion within the fixed `UPGRADE_GAS + INITIALIZE_GAS` budget), `factory_update_address_version` returns `ERR_ROUTER_DEPLOY_FAILED`, causing `withdraw_wnear_to_router` — chained as its callback — to see `Some(false)` and return `ERR_CALLBACK_OF_FAILED_PROMISE` without ever unwrapping the wNEAR. No refund path exists for this case. The wNEAR is permanently locked.

---

### Finding Description

**Step 1 — Synchronous wNEAR debit (EVM execution, committed to state)**

In `engine-precompiles/src/xcc.rs`, when `required_near != ZERO_YOCTO`, the precompile calls `transferFrom` on the wNEAR ERC-20 contract, moving `required_near` wNEAR from the user's EVM address to the engine's implicit EVM address: [1](#0-0) 

This EVM state change is committed as part of the original transaction. It is not rolled back regardless of what happens in subsequent NEAR promise callbacks.

**Step 2 — Promise chain construction in `handle_precompile_promise`**

When `DeployNeeded { create_needed: false }` and `required_near > 0`, the following chain is built:

```
deploy_upgrade (35 TGas on router)
  → factory_update_address_version (5 TGas, callback on engine)
    → withdraw_wnear_to_router (40 TGas, callback on engine)
      → user_call (callback on router)
``` [2](#0-1) [3](#0-2) [4](#0-3) 

**Step 3 — `factory_update_address_version` fails when `deploy_upgrade` fails** [5](#0-4) 

If `deploy_upgrade` panics or runs out of gas, NEAR delivers `Some(false)` to this callback. The function returns `Err(b"ERR_ROUTER_DEPLOY_FAILED")`, causing this callback to panic and fail.

**Step 4 — `withdraw_wnear_to_router` short-circuits without unwrapping** [6](#0-5) 

Because `factory_update_address_version` failed, NEAR delivers `Some(false)` to `withdraw_wnear_to_router`. The function returns immediately with `ERR_CALLBACK_OF_FAILED_PROMISE` — the wNEAR `transferFrom` from Step 1 is never reversed, and the unwrap to actual NEAR never happens.

**Step 5 — No refund path for the upgrade case**

The `refund_needed` flag is only set when `create_needed = true`: [7](#0-6) 

When `create_needed = false` (the upgrade path), `refund_needed = false`, so no `send_refund` call is ever scheduled. There is no mechanism to return the wNEAR to the user.

### Impact Explanation

The user's wNEAR — debited synchronously during EVM execution — is permanently locked in the engine's implicit EVM address. It cannot be recovered by the user. The engine's EVM-side wNEAR balance grows by `required_near` with no corresponding NEAR liability, constituting insolvency: the engine holds wNEAR it cannot account for or return. Impact: **Critical — Permanent freezing of funds / Insolvency**.

---

### Likelihood Explanation

The trigger conditions are:
1. The engine operator deploys a new router version (routine maintenance), making all existing routers outdated.
2. Any user with an outdated router makes an XCC call with `attached_near > 0` (i.e., their cross-contract call attaches NEAR to the target).
3. `deploy_upgrade` fails within its `UPGRADE_GAS + INITIALIZE_GAS = 35 TGas` budget. [8](#0-7) 

The gas constants are described as "estimated from simulation tests," not formally proven lower bounds. If the actual router upgrade requires more than 35 TGas (e.g., due to a larger code blob, storage migration in `initialize`, or NEAR runtime gas schedule changes), every affected user who attaches NEAR to their XCC call loses their wNEAR. This is a systemic, not user-specific, failure mode. Likelihood: **Medium-High** (depends on whether the gas budget is tight; the upgrade path is exercised every time the engine operator updates router code).

---

### Recommendation

1. **Add a wNEAR refund path for the upgrade failure case.** In `factory_update_address_version`, when `Some(false)` is detected, schedule a callback that returns the wNEAR to the original sender rather than simply erroring. Alternatively, restructure the chain so `withdraw_wnear_to_router` runs unconditionally (not as a callback of `factory_update_address_version`) and the version update is a fire-and-forget side effect.

2. **Decouple the upgrade step from the wNEAR withdrawal step.** The upgrade (`deploy_upgrade` + `factory_update_address_version`) and the wNEAR unwrap (`withdraw_wnear_to_router`) should not be sequentially dependent. If the upgrade fails, the wNEAR withdrawal should still proceed (or the wNEAR should be returned to the user).

3. **Increase and formally bound `UPGRADE_GAS`.** Measure actual gas consumption of `deploy_upgrade` under worst-case router code size and add a safety margin. Consider making the gas budget configurable by the operator.

### Proof of Concept

**Setup:**
- Deploy Aurora engine locally with a router version `v1`.
- Register a user EVM address; have them make one XCC call to deploy their router (version `v1`).
- Operator calls `factory_update` to deploy router version `v2`, incrementing `latest_code_version`.

**Attack:**
- User makes an XCC call with `attached_near = 1 NEAR` (so `required_near = 1 NEAR`).
- The precompile executes `transferFrom(user_evm_addr, engine_implicit_evm_addr, 1 NEAR)` — wNEAR leaves the user's EVM balance.
- The promise chain is scheduled: `deploy_upgrade → factory_update_address_version → withdraw_wnear_to_router → user_call`.
- Simulate `deploy_upgrade` failure by setting `UPGRADE_GAS` to a value below the actual gas needed (e.g., 1 TGas in a test environment), or by having the router's `deploy_upgrade` method panic.

**Assertion:**
- After all callbacks settle, assert:
  - `engine_implicit_evm_addr` wNEAR balance = 1 NEAR (nonzero — locked).
  - User's EVM wNEAR balance decreased by 1 NEAR.
  - No NEAR was delivered to the router.
  - The user's XCC call was never executed.

This directly demonstrates the invariant violation: wNEAR debited from the user is permanently locked in the engine's implicit address with no recovery path.
