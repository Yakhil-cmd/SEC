Audit Report

## Title
Unbounded Allowance Table Growth via Short-Lived Approvals Outpacing Fixed Prune Limit — (`rs/ledger_suite/common/ledger_core/src/approvals.rs`, `rs/ledger_suite/common/ledger_canister_core/src/ledger.rs`)

## Summary
`apply_transaction` calls `ledger.approvals_mut().prune(now, APPROVE_PRUNE_LIMIT)` with a hardcoded limit of 100 per transaction. `AllowanceTable::approve` inserts into `allowances` and `expiration_queue` with no guard on total count. An unprivileged attacker who pays approval fees can create N >> 100 short-lived allowances with distinct spenders, causing the allowance table and expiration queue to grow to O(N) and remain bloated for O(N/100) subsequent transactions, exhausting canister memory and degrading ledger performance.

## Finding Description
`APPROVE_PRUNE_LIMIT = 100` is hardcoded at [1](#0-0)  and `prune()` is called once per `apply_transaction` invocation. The `prune()` function iterates at most `limit` times and exits early if the queue is empty or the next expiry is in the future. [2](#0-1) 

The `approve` path for a new allowance (the `None` branch) inserts unconditionally into `allowances_data` and `expiration_queue` with no check on the current total count. [3](#0-2) 

No `MAX_ALLOWANCES` constant or per-account cap exists anywhere in the codebase. The underlying storage structures are unbounded `BTreeMap` and `BTreeSet`. [4](#0-3) 

**Exploit flow:**
1. Attacker submits N `icrc2_approve` calls, each to a unique spender, each with `expires_at = now + small_delta`. At submission time none are expired, so `prune()` removes 0 per transaction. Net: +1 allowance per transaction, 0 pruned.
2. After the expiry window passes, N stale entries remain in both `allowances` and `expiration_queue`.
3. Each subsequent ledger transaction removes at most 100. Clearing N entries requires ⌈N/100⌉ additional transactions. With N = 1,000,000 and a normal rate of ~10 tx/s, the backlog persists for ~1,000 seconds and occupies ~100–200 MB of memory.

Note: the throttle bypass claim in the submission is slightly inaccurate. The `num_pruned` variable at line 223–227 comes from `purge_old_transactions` (transaction deduplication window cleanup), not from the allowance prune. [5](#0-4)  This does not affect the validity of the core vulnerability.

## Impact Explanation
This is a **High** severity application/platform-level DoS. For ICP/ICRC ledgers backed by stable memory, the `allowances` BTreeMap and `expiration_queue` BTreeSet grow without bound, exhausting canister stable or heap memory. For heap-backed ICRC1 ledgers, the canister can trap on memory allocation. Additionally, BTreeMap operations are O(log N), so a table with millions of entries degrades every `approve`, `transfer_from`, and `allowance` query for all users of the ledger. This matches the allowed impact: "Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS" against in-scope financial integration targets (ICP/ICRC ledgers, ckBTC, ckETH, ckERC20).

## Likelihood Explanation
The attack requires only a token balance sufficient to pay N approval fees — a one-time economic cost with no privileged access, no key compromise, and no consensus-level attack. The entry point is the standard `icrc2_approve` ingress endpoint available to any principal. At ~0.0001 ICP per approval, creating 1,000,000 expired allowances costs ~100 ICP. The attack is repeatable and requires no victim interaction.

## Recommendation
1. **Enforce a hard cap** on the total number of allowances (globally or per-account) inside `AllowanceTable::approve`, returning an error when the cap is reached.
2. **Increase or make adaptive the prune limit** — e.g., prune proportionally to the backlog size, or prune a larger batch when the queue exceeds a threshold.
3. **Charge a higher fee for approvals with short expiries**, or require a minimum `expires_at` distance from `now`, to raise the economic cost of this attack.

## Proof of Concept
```rust
// State-machine test sketch
let attacker_balance = 1_000_000 * FEE;
for i in 0..1_000 {
    icrc2_approve(spender = unique_account(i), amount = 1, expires_at = now() + 1);
}
advance_time(2); // all 1000 approvals are now expired
assert_eq!(allowance_table.len(), 1000); // still 1000 entries
for _ in 0..10 {
    icrc1_transfer(...); // 10 normal transactions, each prunes 100
}
// After 10 transactions pruning 100 each: 1000 - 1000 = 0 cleared
// With N=100_000 and only 10 transactions: 100_000 - 1_000 = 99_000 stale entries remain
assert!(allowance_table.len() > 0);
```
A deterministic integration test using PocketIC or the state-machine test framework can confirm the invariant violation: after all approvals expire and only 10 transactions occur, the table still holds up to 99,000 stale entries.

    pub fn prune(&mut self, now: TimeStamp, limit: usize) -> usize {
        self.with_postconditions_check(|table| {
            let mut pruned = 0;
            for _ in 0..limit {
                match table.allowances_data.first_expiry() {
                    Some((ts, _key)) => {
                        if ts > now {
                            return pruned;
                        }
                    }
                    None => {
                        return pruned;
                    }
                }
                if let Some((_, (account, spender))) = table.allowances_data.pop_first_expiry() {
                    let key = (account, spender);
                    if let Some(allowance) = table.allowances_data.get_allowance(&key)
                        && allowance.expires_at.unwrap_or_else(remote_future) <= now
                    {
                        table.allowances_data.remove_allowance(&key);
                        pruned += 1;
                    }
                }
            }
            pruned
        })
    }
