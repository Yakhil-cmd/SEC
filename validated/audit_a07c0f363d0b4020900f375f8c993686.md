Audit Report

## Title
Unchecked Integer Arithmetic in SNS Governance Neuron Maturity Accumulation During Reward Distribution - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS governance `distribute_rewards` function accumulates per-neuron voting rewards into `maturity_e8s_equivalent` and `staked_maturity_e8s_equivalent` using plain Rust `+`/`+=` operators with no overflow protection. The analogous NNS governance path uses `saturating_add` for the same operation. On IC canister release builds — which enable overflow checks — an overflow traps the canister mid-distribution, silently destroying the entire reward round for all neurons.

## Finding Description
In `rs/sns/governance/src/governance.rs` at lines 5989–5996, the reward accumulation is:

```rust
if neuron.auto_stake_maturity.unwrap_or(false) {
    neuron.staked_maturity_e8s_equivalent = Some(
        neuron.staked_maturity_e8s_equivalent.unwrap_or(0) + neuron_reward_e8s,  // line 5991
    );
} else {
    neuron.maturity_e8s_equivalent += neuron_reward_e8s;  // line 5994
}
distributed_e8s_equivalent += neuron_reward_e8s;  // line 5996
``` [1](#0-0) 

All three additions are unchecked `u64` arithmetic. The `rewards_purse_e8s` is correctly guarded against overflow before the loop (lines 5878–5890), but this only bounds the per-round reward amount — it does not bound the cumulative `maturity_e8s_equivalent` field, which accumulates across all reward rounds. A neuron that has been voting for many rounds can have a `maturity_e8s_equivalent` close to `u64::MAX`; the next reward addition then overflows. [2](#0-1) 

The only overflow check present is a `debug_assert!` at lines 6004–6006, which is compiled out in production (release) builds and checks the wrong invariant (purse vs. distributed, not field overflow): [3](#0-2) 

By direct contrast, the NNS governance reward distribution at `rs/nns/governance/src/reward/distribution.rs` lines 162–171 uses `saturating_add` on both maturity fields: [4](#0-3) 

The SNS `add_maturity` handler (line 6508) also uses `saturating_add`, confirming the intended pattern: [5](#0-4) 

## Impact Explanation
IC canister Wasm binaries are compiled with overflow checks enabled in release builds. When `maturity_e8s_equivalent + neuron_reward_e8s` overflows `u64`, the Wasm execution traps. This trap occurs inside the periodic `distribute_rewards` call, causing the entire reward round to be rolled back — no neuron receives maturity for that round. The lost rewards are permanently unrecoverable. This constitutes a **High** impact: significant SNS governance security impact with concrete, permanent financial harm to SNS participants (loss of earned voting rewards, a financial asset).

## Likelihood Explanation
No attacker action is required. The overflow is triggered by the normal periodic reward distribution timer. Any SNS with a large token supply and long-running neurons that have accumulated maturity near `u64::MAX` (~1.84 × 10¹⁹ e8s) will hit this condition through ordinary operation. The condition is more likely in high-supply SNS deployments with neurons that have `auto_stake_maturity = false` and have been voting for many reward rounds. No special privileges are needed; the trigger is the governance canister's own scheduler.

## Recommendation
Replace the plain `+`/`+=` operators in `distribute_rewards` with `saturating_add`, matching the NNS governance implementation:

```rust
// staked maturity path
neuron.staked_maturity_e8s_equivalent = Some(
    neuron.staked_maturity_e8s_equivalent.unwrap_or(0)
        .saturating_add(neuron_reward_e8s),
);

// unstaked maturity path
neuron.maturity_e8s_equivalent =
    neuron.maturity_e8s_equivalent.saturating_add(neuron_reward_e8s);

// accumulator
distributed_e8s_equivalent =
    distributed_e8s_equivalent.saturating_add(neuron_reward_e8s);
```

Additionally, promote the `debug_assert!` at line 6004 to a hard `assert!` or a logged error with graceful handling so overflow conditions are observable in production.

## Proof of Concept
1. Deploy an SNS with a token supply of `u64::MAX / 2` e8s and a high initial reward rate.
2. Create a single neuron holding 100% of the voting power with `auto_stake_maturity = false`.
3. Vote on proposals across enough reward rounds so that cumulative `neuron_reward_e8s` additions cause `maturity_e8s_equivalent` to approach `u64::MAX`.
4. On the overflow round, `distribute_rewards` traps (overflow checks enabled in IC release builds), rolling back the entire reward event.
5. Observe that `latest_reward_event().distributed_e8s_equivalent` is not updated and the neuron's `maturity_e8s_equivalent` is unchanged — the round's rewards are permanently lost.

This can be reproduced deterministically in a PocketIC integration test by pre-seeding a neuron's `maturity_e8s_equivalent` to `u64::MAX - 1` and triggering one reward round.
