Audit Report

## Title
Provider Inconsistency (2-of-4 Split) Halts ckETH/ckERC20 Deposit Scraping via `NoReduction` Strategy - (`File: rs/ethereum/cketh/minter/src/deposit.rs`)

## Summary

The ckETH minter's `scrape_block_range` applies `NoReduction` after querying the EVM RPC canister, which converts any `EvmMultiRpcResult::Inconsistent` response into an immediate `Err`. When `scrape_until_block` receives that error it returns early, abandoning all remaining block ranges and leaving the last-scraped-block pointer unchanged. A 2-of-4 provider split — where two providers return one set of logs and two return another — satisfies this condition and permanently stalls deposit scraping until a governance upgrade replaces a provider. Three documented mainnet emergency upgrades confirm this is exploitable in production.

**Correction to the submitted title:** the current code uses `ConsensusStrategy::Threshold { min: 3 }` on the EVM RPC client, so a *single* misbehaving provider does **not** cause a halt; the existing test `should_mint_when_1_error_with_3_out_of_4_strategy` confirms minting succeeds when 3-of-4 providers agree. The residual vulnerability requires exactly a 2-2 split (or any configuration that prevents the threshold from being met).

## Finding Description

`scrape_logs()` runs three sequential passes: [1](#0-0) 

Each pass calls `scrape_until_block`, which iterates over block-range chunks and calls `scrape_block_range`. On any non-size error, `scrape_until_block` immediately returns without advancing the scraped-block pointer: [2](#0-1) 

Inside `scrape_block_range`, the EVM RPC result is reduced with `NoReduction`: [3](#0-2) 

`NoReduction.reduce` passes any `EvmMultiRpcResult::Inconsistent` directly to the closure that returns `Err(MultiCallError::InconsistentResults(...))`: [4](#0-3) 

The underlying `consistent_result_or_reduce` only short-circuits to `Ok` when the EVM RPC canister returns `EvmMultiRpcResult::Consistent`: [5](#0-4) 

The EVM RPC client is configured with `ConsensusStrategy::Threshold { total: Some(4), min: 3 }` for mainnet: [6](#0-5) 

This threshold means the EVM RPC canister returns `Consistent` when ≥3 providers agree, so a single rogue provider is tolerated. However, when exactly 2 providers return one result and 2 return another, the threshold is not met, the canister returns `Inconsistent`, `NoReduction` converts it to `Err`, `scrape_until_block` returns early, and no block-pointer advancement occurs. The minter retries the same range on the next timer tick and stalls indefinitely.

The non-size-error branch in `scrape_block_range` propagates the error upward rather than skipping the range: [7](#0-6) 

## Impact Explanation

When scraping stalls, no `ReceivedEth` or `ReceivedErc20` events are detected, no ckETH or ckERC20 is minted, and user ETH/ERC20 already sent to the helper contract is inaccessible on the IC side until a governance upgrade replaces a provider. This is a concrete, sustained application-level DoS of a live financial integration component. It maps to **High ($2,000–$10,000): Application/platform-level DoS with concrete user or protocol harm** — specifically significant Chain Fusion / ck-token security impact with concrete user funds impact.

## Likelihood Explanation

A 2-2 provider split has already occurred in production. The 2024-09-12 incident documents LlamaNodes being down simultaneously with Pocket Network returning divergent responses across replicas, causing exactly this halt: [8](#0-7) 

The 2024-09-11 notes explicitly acknowledge the structural gap and name the long-term fix: [9](#0-8) 

The 2024-03-18 incident (Cloudflare returning wrong logs post-Dencun) is a third real-world trigger: [10](#0-9) 

Each incident required an emergency governance upgrade to restore service. The minter integrates with 4 independent external providers, each an independent failure point; simultaneous degradation of two is a realistic and observed condition.

## Recommendation

1. Replace `NoReduction` in `scrape_block_range` with `StrictMajorityByKey` (already implemented in `rs/ethereum/cketh/minter/src/eth_rpc_client/mod.rs` at L221–242) so that a result agreed upon by ≥3 providers is accepted even when the fourth disagrees.
2. For the 2-2 split case where no strict majority exists, log the inconsistency and advance the scraped-block pointer by a configurable amount rather than retrying the same range indefinitely.
3. Consider per-provider circuit-breaking: track providers that repeatedly produce inconsistent results and temporarily exclude them, continuing with the remaining providers.

## Proof of Concept

The existing integration test directly demonstrates the 2-2 halt: [11](#0-10) 

Provider1+3 return one log set, Provider2+4 return another; `NoReduction` returns `InconsistentResults`; `scrape_until_block` returns early; `expect_no_mint()` passes. The complementary test confirms the 3-of-4 threshold does handle the single-provider case (minting succeeds when 3 agree): [12](#0-11) 

The `should_retry_from_same_block_when_scrapping_fails` test further confirms that on a non-size error the `SyncedToBlock` event remains at `LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL`, proving the block pointer does not advance: [13](#0-12)

```rust
fn consistent_result_or_reduce<T, F>(
    result: EvmMultiRpcResult<T>,
    reduce: F,
) -> Result<T, MultiCallError<T>>
where
    F: Fn(MultiCallResults<T>) -> Result<T, MultiCallError<T>>,
{
    match result {
        EvmMultiRpcResult::Consistent(result) => {
            result.map_err(|e| MultiCallError::ConsistentError(e.into()))
        }
        EvmMultiRpcResult::Inconsistent(results) => {
            reduce(MultiCallResults::from_non_empty_iter(results))
        }
    }
```

#[test]
fn should_not_mint_when_logs_too_inconsistent() {
    let deposit_params = DepositCkEthParams::default();
    let (block_pi_logs, public_node_logs) = {
        let block_pi_log_entry = deposit_params.to_log_entry();
        let llama_nodes_log_entry = DepositCkEthParams {
            amount: deposit_params.amount + 1,
            ..deposit_params.clone()
        }
        .to_log_entry();
        (vec![block_pi_log_entry], vec![llama_nodes_log_entry])
    };
    assert_ne!(block_pi_logs, public_node_logs);

    CkEthSetup::default()
        .deposit(deposit_params)
        .with_mock_eth_get_logs(move |mock| {
            mock.respond_with(JsonRpcProvider::Provider1, block_pi_logs.clone())
                .respond_with(JsonRpcProvider::Provider2, public_node_logs.clone())
                .respond_with(JsonRpcProvider::Provider3, block_pi_logs.clone())
                .respond_with(JsonRpcProvider::Provider4, public_node_logs.clone())
        })
        .expect_no_mint();
}
