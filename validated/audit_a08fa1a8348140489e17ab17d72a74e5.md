### Title
`exit_to_near_precompile_callback` Unconditionally Enforces `require_running` Blocking the Refund Path When Contract Is Paused - (File: `engine/src/contract_methods/connector.rs`)

### Summary

`exit_to_near_precompile_callback` unconditionally calls `require_running` before branching into either the success path (transfer wNEAR) or the failure/refund path (restore burned ERC-20 tokens). If the Aurora contract is paused between the moment a user initiates `exit_to_near` and the moment the callback executes, and the underlying `ft_transfer` promise fails, the refund branch is permanently blocked. The user's ERC-20 tokens were already burned in the EVM and can never be recovered.

### Finding Description

In `engine/src/contract_methods/connector.rs`, the callback function `exit_to_near_precompile_callback` is structured as follows:

```rust
pub fn exit_to_near_precompile_callback<...>(...) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;          // <-- unconditional, blocks ALL paths
        env.assert_private_call()?;
        ...
        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            // success: optionally transfer wNEAR to recipient
            ...
        } else if let Some(args) = args.refund {
            // failure: MUST refund burned ERC-20 tokens to user
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
            ...
        } else {
            None
        };
        Ok(maybe_result)
    })
}
```

The `require_running` guard at line 203 is applied unconditionally before the function determines whether it is handling a successful exit or a failed exit requiring a refund. The two branches serve fundamentally different purposes:

- **Success branch**: The `ft_transfer` to the NEAR recipient already succeeded; the wNEAR transfer is a secondary action. Blocking it when paused is acceptable.
- **Refund branch**: The `ft_transfer` failed; the user's ERC-20 tokens were already burned in the EVM. The refund is the only mechanism to restore them. Blocking it when paused causes permanent, irrecoverable token loss.

The Aurora project itself recognized this exact class of bug for `ft_resolve_transfer` and fixed it in v3.6.2 (CHANGES.md line 209: *"The `ft_resolve_transfer` callback doesn't require running the contract to finish the `ft_transfer_call` correctly"*). The same fix was not applied to `exit_to_near_precompile_callback`.

### Impact Explanation

**Critical — Permanent freezing/theft of user funds.**

When the contract is paused while a user's `exit_to_near` callback is pending and the `ft_transfer` promise has failed:

1. The user's ERC-20 tokens are already burned in the Aurora EVM (irreversible).
2. The NEAR-side `ft_transfer` failed (e.g., recipient account not registered with the NEP-141 token).
3. `exit_to_near_precompile_callback` is invoked as a NEAR callback.
4. `require_running` returns `ERR_PAUSED` and the entire callback aborts.
5. `engine::refund_on_error` never executes.
6. The user permanently loses their ERC-20 tokens with no recovery path.

### Likelihood Explanation

The contract owner can pause the Aurora contract at any time via `pause_contract`. A pause during routine maintenance, an emergency response to another incident, or a targeted griefing action by a malicious owner is sufficient. The `ft_transfer` failure condition (unregistered recipient) is a normal user error that occurs in practice. The combination of these two independently plausible events creates a realistic attack/accident path.

### Recommendation

Move `require_running` inside the conditional branches so it only applies to the success path, mirroring the fix applied to `ft_resolve_transfer` in PR #906:

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    require_running(&state)?;   // only enforce when not a refund
    if let Some(args) = args.transfer_near {
        ...
    }
    None
} else if let Some(args) = args.refund {
    // Refund MUST proceed regardless of pause state
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None
};
```

### Proof of Concept

1. User holds ERC-20 tokens on Aurora (bridged from a NEP-141).
2. User calls `exit_to_near` targeting an unregistered NEAR account. The ERC-20 tokens are burned in the EVM; a `ft_transfer` promise is scheduled.
3. Before the callback executes, the Aurora owner calls `pause_contract` (e.g., for an upgrade or emergency).
4. The NEAR runtime invokes `exit_to_near_precompile_callback`. The `ft_transfer` promise result is `Failed`.
5. `require_running` at line 203 returns `Err(ERR_PAUSED)`. The callback aborts.
6. `engine::refund_on_error` is never called. The user's ERC-20 tokens are permanently lost.
