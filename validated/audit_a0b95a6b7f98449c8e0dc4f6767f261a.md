Audit Report

## Title
ckERC20 Minter Over-Mints Tokens for Fee-on-Transfer ERC-20 Deposits Due to Missing Balance Verification - (File: `rs/ethereum/cketh/minter/src/deposit.rs`)

## Summary
Both Solidity helper contracts (`DepositHelperWithSubaccount.sol` and `ERC20DepositHelper.sol`) emit the caller-supplied `amount` parameter in their deposit events rather than the actual balance delta received by the minter address. The IC minter canister scrapes these logs and mints exactly `event.value()` ckERC20 tokens with no balance reconciliation, meaning any fee-on-transfer ERC-20 token will cause the minter to mint more ckERC20 than the ERC-20 tokens it actually holds, breaking the 1:1 backing invariant. USDT (`0xdAC17F958D2ee523a2206206994597C13D831ec7`) is already a supported ckERC20 token and contains a dormant fee mechanism that Tether can enable at any time.

## Finding Description

**Solidity root cause — `DepositHelperWithSubaccount.sol` L511–532:**
`depositErc20` calls `erc20Token.safeTransferFrom(msg.sender, minterAddress, amount)` and then emits `ReceivedEthOrErc20(..., amount, ...)` using the input parameter `amount`, not the post-transfer balance change at `minterAddress`. For a fee-on-transfer token, `safeTransferFrom` delivers `amount - fee` to `minterAddress`, but the event records `amount`.

**Solidity root cause — `ERC20DepositHelper.sol` L498–503:**
Identical pattern: `safeTransferFrom` followed by `emit ReceivedErc20(..., amount, ...)` using the input `amount`.

**IC minter root cause — `rs/ethereum/cketh/minter/src/deposit.rs` L73–81:**
The `mint()` function passes `amount: event.value()` directly to the ICRC1 ledger `transfer` call. There is no pre- or post-deposit balance check against the minter's actual Ethereum ERC-20 holdings.

**Event value source — `rs/ethereum/cketh/minter/src/eth_logs/mod.rs` L57–75 and `src/eth_logs/parser.rs` L92–103:**
`ReceivedErc20Event.value` is populated by `Erc20Value::from_be_bytes(value_bytes)` decoded directly from the Ethereum log data field, which contains the input `amount` from the Solidity event emission, not a balance delta.

**Internal accounting also corrupted — `rs/ethereum/cketh/minter/src/state.rs` L332–338:**
`update_balance_upon_deposit` credits `event.value` to `erc20_balances`, so the minter's internal ERC-20 balance tracker is also inflated by the fee amount on every deposit of a fee-on-transfer token.

**No existing guard:** The `add_ckerc20_token` endpoint (`src/main.rs` L563–574) and `record_add_ckerc20_token` (`src/state.rs` L398–424) check only for duplicate symbols, ledger IDs, and contract addresses. There is no check that the candidate ERC-20 token does not implement fee-on-transfer behavior.

## Impact Explanation
This is an illegal minting / ledger conservation failure affecting the ckERC20 system, which is an explicitly in-scope Chain Fusion financial integration. For every deposit of a fee-on-transfer ERC-20 token, the minter mints `fee` unbacked ckERC20 tokens. Over repeated deposits, the minter's ERC-20 reserves become insufficient to back the outstanding ckERC20 supply. When users attempt to withdraw ckERC20 back to ERC-20, the minter cannot fulfill all requests, resulting in direct, permanent loss for ckERC20 holders. This matches the allowed impact: **"Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant ICP/Cycles or in-scope chain-key/ledger assets"** and **"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."**

## Likelihood Explanation
USDT (`0xdAC17F958D2ee523a2206206994597C13D831ec7`) is already a supported ckERC20 token (added via NNS proposal, confirmed in `rs/ethereum/cketh/mainnet/orchestrator_upgrade_2024_07_22_ckusdt.md`). The USDT contract contains a documented, non-speculative fee mechanism (the `basisPointsRate` and `maximumFee` storage variables) that is currently set to zero but can be enabled unilaterally by Tether at any time without any IC governance action. Once enabled, any unprivileged Ethereum address can call `depositErc20` on the helper contract to trigger the over-minting. No IC privileges, no social engineering, and no third-party compromise are required — only Tether exercising a feature that already exists in their deployed contract. Additionally, any future NNS proposal to add a fee-on-transfer ERC-20 token would make this immediately exploitable. Likelihood is **medium-to-high** given the live USDT exposure.

## Recommendation
1. **Solidity helper contracts:** Record the actual balance delta instead of the input `amount`. Before and after `safeTransferFrom`, read `erc20Token.balanceOf(minterAddress)` and emit the difference:
```solidity
uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
```
2. **IC minter canister:** Add a governance-level check in `add_ckerc20_token` / `record_add_ckerc20_token` that rejects ERC-20 tokens known to implement fee-on-transfer behavior, or document this as an explicit invariant that NNS voters must verify before approving any new ckERC20 token.
3. **Internal accounting:** Ensure `update_balance_upon_deposit` is driven by the same verified amount used for minting, not the raw log value.

## Proof of Concept
1. Tether enables USDT transfer fees (e.g., sets `basisPointsRate = 1`, `maximumFee = 1e18`).
2. Attacker calls `depositErc20(USDT_ADDRESS, 1_000_000_000, principal, subaccount)` on the deployed `DepositHelperWithSubaccount` helper contract.
3. `safeTransferFrom` transfers `999_999_900` USDT to the minter (after 0.00001% fee, capped at `maximumFee`).
4. The helper emits `ReceivedEthOrErc20(..., 1_000_000_000, ...)`.
5. The IC minter scrapes the log via `scrape_logs()` → `scrape_block_range::<ReceivedEthOrErc20LogScraping>()` → `ReceivedEthOrErc20LogParser::parse_log()`, reads `value = 1_000_000_000`, stores it in `ReceivedErc20Event.value`.
6. `mint()` calls `client.transfer(TransferArg { amount: event.value(), ... })` and mints `1_000_000_000` ckUSDT.
7. The minter holds `999_999_900` USDT but has minted `1_000_000_000` ckUSDT — `100` unbacked ckUSDT per deposit.
8. Repeated at scale, the minter's USDT reserves are drained relative to outstanding ckUSDT supply, eventually making some withdrawals impossible.

A deterministic integration test can be written using the existing `CkErc20Setup` state-machine test infrastructure (`rs/ethereum/cketh/minter/tests/ckerc20.rs`) with a mock ERC-20 token that deducts a fee in `transferFrom` but emits the pre-fee `amount` in the helper contract event, then asserting that the ckERC20 ledger balance exceeds the mock ERC-20 balance held by the minter.
