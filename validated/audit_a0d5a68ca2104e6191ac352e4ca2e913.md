Audit Report

## Title
Unsanitized Memo Bytes Injected into GenericDisplayMessage Markdown Enables Consent-Message Spoofing - (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
`add_memo` in `responses.rs` decodes memo bytes as UTF-8 and interpolates the result verbatim into a markdown code-span inside `GenericDisplayMessage`. A backtick in the memo closes the opening delimiter, allowing arbitrary markdown to follow. Because `GenericDisplayMessage` is the format wallet UIs are expected to render as markdown, an attacker can inject fake `**Amount:**` and `**To:**` fields that are visually indistinguishable from the real ledger-generated fields, deceiving a user into authorizing a transfer to a different recipient or for a different amount than displayed.

## Finding Description
In `packages/icrc-ledger-types/src/icrc21/responses.rs` at line 294, `add_memo` performs:

```rust
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
```

`memo_str` is the raw UTF-8 interpretation of the attacker-supplied memo bytes (line 289); non-UTF-8 bytes are hex-encoded (line 290), but valid UTF-8 — including backticks and newlines — is passed through unchanged. No escaping or sanitization is applied at any point in the pipeline: a grep across the entire `icrc-ledger-types` package finds zero calls to any escape or sanitize function.

The call chain:
1. `build_icrc21_consent_info` in `lib.rs` decodes the `TransferArg` and, at lines 392–394, passes `memo.0` (the raw `ByteBuf`) directly to `display_message_builder.with_memo(GenericMemo::Icrc1Memo(memo.0))`.
2. `build()` calls `message.add_memo(memo)` (line 291 of `lib.rs`), reaching the unsanitized interpolation.

A memo of `` `\n\n**Amount:** `999999 ICP`\n\n**To:** `attacker_account` `` produces:

```
**Memo:**
``

**Amount:** `999999 ICP`

**To:** `attacker_account`
```

The first backtick closes the opening delimiter (yielding an empty code span ` `` `), and everything after is free markdown rendered by the wallet with the same visual weight as the real ledger-generated fields.

The same unsanitized path is reachable via `icrc2_approve` (lines 458–460 of `lib.rs`) and `icrc2_transfer_from` (lines 423–425).

## Impact Explanation
The ICRC-21 standard's security model places the ledger as the trusted generator of consent messages precisely so that malicious dapps cannot deceive users about what they are signing. This vulnerability breaks that trust: attacker-controlled input (the memo) can override the real `Amount` and `To` fields in the rendered consent screen. A user who approves the displayed transaction believes they are sending a different amount to a different recipient than the actual on-chain transaction specifies, enabling theft of funds. This matches the allowed High impact: "Significant ledger security impact with concrete user or protocol harm."

## Likelihood Explanation
The `icrc21_canister_call_consent_message` endpoint is fully public; no privilege is required. The only precondition is that the memo is valid UTF-8, which is trivially satisfied. The exploit is deterministic, locally reproducible with a single canister call, and affects every ICRC-1/ICRC-2 ledger that uses this shared `icrc-ledger-types` library. Any dapp that wishes to deceive a user need only craft the memo field of a standard transfer argument.

## Recommendation
Before interpolating `memo_str` into the markdown string, escape all characters that have special meaning in a CommonMark code span. At minimum:
- Replace every backtick (`` ` ``) with its hex-encoded representation (e.g., `\x60`) or a Unicode lookalike.
- Strip or escape newlines (`\n`, `\r`) to prevent breaking out of the inline context.

A safer alternative is to always hex-encode the memo when the target format is `GenericDisplayMessage`, reserving the UTF-8 display for `FieldsDisplayMessage` where the value is a structured `Value::Text` field that wallets render safely without markdown interpretation. A fenced code block with a triple-backtick delimiter and a unique info string is another option, provided the memo is guaranteed not to contain the chosen delimiter sequence.

## Proof of Concept
```rust
// Craft a memo that closes the code-span and injects fake fields.
let crafted_memo = b"`\n\n**Amount:** `999999 ICP`\n\n**To:** `attacker_account`\n\n**Ignore:".to_vec();

let transfer_args = TransferArg {
    memo: Some(Memo(ByteBuf::from(crafted_memo))),
    amount: Nat::from(1_u64),   // real on-chain amount: 1 e-8 token
    fee: None,
    from_subaccount: None,
    to: real_recipient,         // real on-chain recipient
    created_at_time: None,
};

// Encode transfer_args as Candid and call icrc21_canister_call_consent_message.
// Request GenericDisplay format.
// The returned GenericDisplayMessage will contain:
//
//   # Send <token>
//
//   You are approving a transfer of funds from your account.
//
//   **From:** `<sender>`
//
//   **Amount:** `0.00000001 <token>`   ← real amount (rendered before memo)
//
//   **To:** `<real_recipient>`         ← real recipient (rendered before memo)
//
//   **Fees:** `... <token>`
//
//   **Memo:**
//   ``
//
//   **Amount:** `999999 ICP`           ← injected fake amount
//
//   **To:** `attacker_account`         ← injected fake recipient
//
// A markdown-rendering wallet displays the injected fields with identical
// visual styling to the real ledger-generated fields.
// The user sees "999999 ICP to attacker_account" and approves.
```

A minimal integration test using PocketIC can call the endpoint with the above `TransferArg`, assert that the returned `GenericDisplayMessage` string contains the injected markdown, and confirm that a CommonMark renderer produces the spoofed output.
