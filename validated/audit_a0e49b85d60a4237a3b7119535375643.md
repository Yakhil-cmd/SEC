### Title
Unchecked Exit Precompile Return Value After Token Burn Causes Permanent Fund Loss — (`File: etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` burn a user's tokens **before** calling the Aurora exit precompile, and then **silently ignore the return value** of that low-level `call`. If the precompile call fails for any reason — including an ABI mismatch between the V1/V2 contract format and the precompile's compile-time `error_refund` feature flag — the tokens are permanently destroyed with no cross-chain transfer ever occurring. This is a direct structural analog to the Curve pool `lp_token()` / `token()` ABI mismatch: a call that can silently fail is never checked, and the resulting stuck state causes permanent fund loss.

---

### Finding Description

**Root cause — unchecked low-level call after irreversible burn:**

In `EvmErc20.sol`, `withdrawToNear`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // tokens destroyed first
    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}
```

`res` is assigned but never inspected. A failed precompile call (returning `0`) is silently swallowed. The same pattern appears in `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

**ABI mismatch between V1 and V2 — the direct Curve analog:**

`EvmErc20.sol` (V1) encodes the precompile input as:
```
[flag(1)] [amount(32)] [recipient(N)]
```

`EvmErc20V2.sol` (V2) encodes it as:
```
[flag(1)] [sender(20)] [amount(32)] [recipient(N)]
```

The precompile's parsing is controlled by a **compile-time** `error_refund` feature flag:

- **Without** `error_refund`: expects `[flag(1)][amount(32)][recipient(N)]` — compatible with V1 only.
- **With** `error_refund`: expects `[flag(1)][refund_address(20)][amount(32)][recipient(N)]` — compatible with V2 only.

If `EvmErc20V2.sol` is deployed against a precompile compiled **without** `error_refund`, the precompile receives `[0x01, sender(20), amount(32), recipient(N)]` and parses the amount from bytes `[1..33]` — i.e., the 20-byte sender address concatenated with the first 12 bytes of the actual amount. This 256-bit value will almost always exceed `u128::MAX`, causing `parse_amount` to return `Err`, the precompile call to fail, and `res = 0`. Since `res` is never checked, the transaction succeeds on Aurora: tokens are burned, but no NEAR transfer is scheduled.

The symmetric case — `EvmErc20.sol` (V1) against a precompile compiled **with** `error_refund` — causes the first 20 bytes of the `amount` field to be misread as the `refund_address`, and the parsed amount becomes the last 12 bytes of the real amount concatenated with the first 20 bytes of the recipient string, producing a wildly incorrect transfer amount or a parse failure.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the precompile call fails silently:
1. `_burn` has already executed and is irreversible.
2. No NEAR-side `ft_transfer` or `withdraw` promise is ever created.
3. The user's tokens are gone from Aurora with no corresponding receipt on NEAR or Ethereum.
4. There is no recovery path: the burned supply is unrecoverable, and no refund mechanism is triggered because the call failure is never observed.

---

### Likelihood Explanation

**High.** The two ERC-20 contract versions (`EvmErc20` and `EvmErc20V2`) exist precisely because the precompile input format changed when the `error_refund` feature was introduced. Any Aurora deployment or Silo instance where the deployed ERC-20 contract version does not match the precompile's compile-time feature flag will exhibit this bug for every user who calls `withdrawToNear` or `withdrawToEthereum`. The mismatch is invisible to the user and requires no attacker — any ordinary token holder triggering a withdrawal is sufficient.

---

### Recommendation

1. **Check the return value** of every low-level `call` to the exit precompile. Revert if `res == 0`:
   ```solidity
   assembly {
       let res := call(gas(), EXIT_PRECOMPILE, 0, add(input, 32), input_size, 0, 32)
       if iszero(res) { revert(0, 0) }
   }
   ```
   This ensures the burn is rolled back if the precompile call fails, preventing permanent fund loss.

2. **Perform the burn after confirming the precompile call succeeds**, or use a two-step commit/reveal pattern where the burn is conditional on the cross-chain transfer being scheduled.

3. **Version-gate the input encoding** in the Solidity contracts to match the deployed precompile's feature set, or make the precompile accept both formats by inspecting the input length.

---

### Proof of Concept

**Step 1.** Deploy an Aurora Silo instance with the precompile compiled **without** the `error_refund` feature.

**Step 2.** Deploy `EvmErc20V2.sol` (which encodes `sender` in the precompile input) and register it as a bridged token.

**Step 3.** As an ordinary user, call `withdrawToNear(recipient, 1000)`:
- `_burn(msg.sender, 1000)` executes — 1000 tokens destroyed.
- Precompile input: `[0x01, sender_addr(20 bytes), 0x...03e8(32 bytes), recipient(N bytes)]`
- Precompile (non-`error_refund`) reads amount from bytes `[1..33]` = `sender_addr ++ amount_first_12_bytes` → a 256-bit value with the sender address in the high bits → `> u128::MAX` → `parse_amount` returns `Err`.
- `call` returns `0`. `res` is never checked. Transaction succeeds.
