### Title
Immutable NEP-141 ↔ ERC-20 `BijectionMap` Causes Permanent Fund Freeze on NEP-141 Token Migration — (File: `engine/src/engine.rs`)

---

### Summary

The Aurora Engine maintains a permanent, write-once bijection between NEAR NEP-141 token account IDs and their EVM ERC-20 mirror addresses. There is no mechanism to update this mapping. If a NEP-141 token migrates to a new NEAR account ID (a common lifecycle event for major tokens), all ERC-20 mirror tokens already minted in Aurora become permanently unredeemable, freezing user funds.

---

### Finding Description

The bridge mapping is stored in a `BijectionMap<NEP141Account, ERC20Address, I>` backed by persistent NEAR storage: [1](#0-0) 

The mapping is written once at token registration time via `register_token`: [2](#0-1) 

Critically, line 735 explicitly rejects any attempt to re-register an already-mapped NEP-141 token: [3](#0-2) 

This means the mapping `old-nep141.near ↔ 0xERC20` is **immutable for the lifetime of the contract**.

When a user exits ERC-20 tokens back to NEAR, the engine looks up the NEP-141 account ID from the ERC-20 address via the reverse side of the `BijectionMap` and calls `ft_transfer` on that stored account ID. If the NEP-141 token has migrated to a new account (`new-nep141.near`), the engine still calls `ft_transfer` on the old, deprecated `old-nep141.near` contract. If that contract is non-functional or has had its supply migrated away, the cross-contract call fails. The `exit_to_near_precompile_callback` re-mints the ERC-20 tokens on failure: [4](#0-3) 

This creates a permanent loop: exit fails → tokens re-minted → exit fails again. Users cannot recover their value.

On the deposit side, `receive_erc20_tokens` uses `get_erc20_from_nep141` keyed on `predecessor_account_id`: [5](#0-4) 

Transfers from the new NEP-141 contract (`new-nep141.near`) will return `Nep141NotFound` and refund the sender, so new deposits are blocked too. The ERC-20 mirror token is permanently orphaned.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any user who bridged a NEP-141 token into Aurora before a token migration holds ERC-20 mirror tokens that:
1. Cannot be redeemed back to NEAR (exit calls the defunct old contract, fails, re-mints).
2. Cannot receive new deposits from the migrated contract (new account ID not in mapping).
3. Have no on-chain recovery path because the mapping cannot be updated.

The bridged principal is permanently locked inside Aurora with no escape route.

---

### Likelihood Explanation

**Medium.** NEP-141 token migrations are a documented lifecycle event on NEAR (contract upgrades, rebranding, security patches). The Aurora bridge is designed to support arbitrary NEP-141 tokens. Any token that undergoes a contract account migration after being registered in Aurora triggers this condition. No attacker action is required — the token's own governance team performing a standard migration is sufficient.

---

### Recommendation

- **Short-term:** Document that the NEP-141 ↔ ERC-20 mapping is immutable and that token migrations will permanently break the bridge for that token pair.
- **Long-term:** Introduce an owner-gated `update_nep141_mapping(old_nep141, new_nep141)` function that atomically replaces the old NEP-141 key in the `BijectionMap` while preserving the ERC-20 address, so existing mirror token balances remain valid and exits route to the new contract. Ensure the update is protected by the same access control as `set_eth_connector_contract_account`.

---

### Proof of Concept

1. Owner calls `deploy_erc20_token` for `token-v1.near` → ERC-20 `0xTOKEN` is deployed; `BijectionMap` stores `token-v1.near ↔ 0xTOKEN`.
2. Alice calls `ft_transfer_call` on `token-v1.near` with Aurora as receiver and her EVM address in `msg`. `ft_on_transfer` fires, `receive_erc20_tokens` mints 1000 `0xTOKEN` to Alice.
3. The token team migrates to `token-v2.near`. `token-v1.near` is deprecated (supply zeroed or contract deleted).
4. Alice calls the `ExitToNear` precompile to burn her 1000 `0xTOKEN` and receive `token-v1.near` tokens on NEAR. The engine looks up `token-v1.near` from the `BijectionMap` reverse side and schedules `ft_transfer` on `token-v1.near`.
5. The cross-contract call to `token-v1.near` fails (contract deprecated). `exit_to_near_precompile_callback` detects failure and calls `refund_on_error`, re-minting 1000 `0xTOKEN` to Alice. [4](#0-3) 

6. Alice's 1000 `0xTOKEN` are permanently frozen. No admin function exists to remap `token-v1.near` to `token-v2.near` because `register_token` rejects re-registration: [6](#0-5)
