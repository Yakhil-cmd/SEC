### Title
`new_settlement_layer_chain_id_storage` Excluded from IO Frame Snapshot Prevents Rollback on Transaction Revert — (`basic_system/src/system_implementation/system/io_subsystem.rs`)

---

### Summary

`FullIOStateSnapshot` and the `start_io_frame` / `finish_io_frame` pair in `FullIO` snapshot and roll back five of the six IO sub-storages, but silently omit `new_settlement_layer_chain_id_storage`. If any transaction that calls `update_settlement_layer_chain_id` subsequently reverts, the chain-ID write is never undone. The block is then finalized with a settlement-layer chain ID that was set by a reverted transaction, corrupting the block's public output and permanently blocking any legitimate chain-ID update for the rest of the block.

---

### Finding Description

`FullIO` owns six independent state components:

| Component | In `FullIOStateSnapshot`? | Rolled back in `finish_io_frame`? |
|---|---|---|
| `storage` (M) | ✅ | ✅ |
| `transient_storage` | ✅ | ✅ |
| `logs_storage` | ✅ | ✅ |
| `events_storage` | ✅ | ✅ |
| `interop_root_storage` | ✅ | ✅ |
| **`new_settlement_layer_chain_id_storage`** | **❌** | **❌** |

`FullIOStateSnapshot` is defined as:

    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,

The storage also enforces a once-per-block invariant:

```rust
pub fn update(&mut self, new_sl_chain_id: U256) -> Result<(), SystemError> {
    if self.value().is_some() {
        return Err(internal_error!(
            "Tried to update settlement layer chain id more than once in a block"
        ).into());
    }
    self.history.update(new_sl_chain_id);
    Ok(())
}
``` [5](#0-4) 

Because the update is never rolled back on revert, a reverted transaction that called `update_settlement_layer_chain_id` leaves the storage permanently set. Any subsequent legitimate call to `update_settlement_layer_chain_id` in the same block then hits the "more than once" guard and returns an `InternalError`, which propagates as a fatal error through the bootloader.

The `update_settlement_layer_chain_id` entry point is exposed through the `IOSubsystem` trait and is charged native resources before writing:

```rust
fn update_settlement_layer_chain_id(
    &mut self, _ee_type: ExecutionEnvironmentType,
    resources: &mut Self::Resources, new_sl_chain_id: U256,
) -> Result<(), SystemError> {
    ...
    self.new_settlement_layer_chain_id_storage.update(new_sl_chain_id)
}
``` [6](#0-5) 

The transaction processing pipeline opens a global frame before execution and rolls it back on revert, but because `new_settlement_layer_chain_id_storage` is absent from the snapshot, the rollback is incomplete:

```rust
let main_body_rollback_handle = system.start_global_frame()?;
// ... execute ...
ExecutionResult::Revert { .. } => {
    system.finish_global_frame(Some(&main_body_rollback_handle))?;
    // storage, transient, logs, events, interop_roots are rolled back
    // new_settlement_layer_chain_id_storage is NOT rolled back
}
``` [7](#0-6) 

---

### Impact Explanation

**State-transition correctness violation.** The settlement layer chain ID is part of the block's public output committed to L1. A reverted transaction can:

1. **Corrupt the block's public output** — the chain ID written by the reverted transaction is included in the finalized block, causing the L1 verifier to accept a block with a chain ID that was never legitimately set.
2. **Permanently DoS legitimate chain-ID updates** — the once-per-block guard fires for any subsequent valid call in the same block, returning an `InternalError` that aborts block processing.
3. **Forward/proving divergence** — the forward (sequencer) and proving (RISC-V) paths both use the same `FullIO` implementation, so both will produce the same incorrect output; the proof will be valid for the wrong state.

---

### Likelihood Explanation

The `update_settlement_layer_chain_id` hook is reachable from EVM execution via the system-hook dispatch layer. Any transaction that can invoke the relevant system contract address and then revert (e.g., by running out of gas after the hook call, or by reverting in a subsequent sub-call) triggers the bug. The attacker does not need any privileged role — only the ability to submit a transaction that reaches the hook and then reverts. This is a realistic scenario for any block that processes a settlement-layer chain-ID upgrade transaction.

---

### Recommendation

Add `new_settlement_layer_chain_id_storage` to `FullIOStateSnapshot` and include it in both `start_io_frame` and `finish_io_frame`, mirroring the pattern already used for `interop_root_storage`:

```rust
pub struct FullIOStateSnapshot<M: StorageModel> {
    io: M::StateSnapshot,
    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,
    new_sl_chain_id: NewSettlementLayerChainIdSnapshotId, // ADD
}

fn start_io_frame(&mut self) -> Result<Self::StateSnapshot, InternalError> {
    ...
    let new_sl_chain_id = self.new_settlement_layer_chain_id_storage.start_frame(); // ADD
    Ok(FullIOStateSnapshot { ..., new_sl_chain_id })
}

fn finish_io_frame(&mut self, rollback_handle: Option<&Self::StateSnapshot>) -> Result<(), InternalError> {
    ...
    self.new_settlement_layer_chain_id_storage  // ADD
        .finish_frame(rollback_handle.map(|x| x.new_sl_chain_id));
    Ok(())
}
```

### Proof of Concept

1. Attacker submits a transaction that calls the system contract responsible for `update_settlement_layer_chain_id` with an arbitrary `new_sl_chain_id` value (e.g., `U256::from(9999)`).
2. The call succeeds and `new_settlement_layer_chain_id_storage` is set to `9999`.
3. The transaction then reverts (e.g., the outer call reverts, or the transaction runs out of gas in a subsequent step).
4. `finish_global_frame(Some(&rollback_handle))` is called, rolling back `storage`, `transient_storage`, `logs_storage`, `events_storage`, and `interop_root_storage` — but **not** `new_settlement_layer_chain_id_storage`.
5. `new_settlement_layer_chain_id_storage.value()` still returns `Some(9999)`.
6. Any subsequent legitimate transaction in the same block that calls `update_settlement_layer_chain_id` hits the guard and returns `InternalError("Tried to update settlement layer chain id more than once in a block")`.
7. The block is finalized with chain ID `9999` in its public output, which was set by a reverted transaction.
