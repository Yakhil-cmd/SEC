### Title
Quarantined ETH/ERC20 Deposits Have No Refund or Retry Mechanism - (`File: rs/ethereum/cketh/minter/src/deposit.rs`)

### Summary
When the ckETH/ckERC20 minter panics during the minting callback after observing a valid Ethereum deposit, the deposit is permanently quarantined via `EventType::QuarantinedDeposit`. The user's ETH or ERC20 tokens are already held in the minter's Ethereum address and cannot be returned. No automatic refund path exists; recovery requires manual canister upgrade intervention.

### Finding Description

The `mint()` function in `rs/ethereum/cketh/minter/src/deposit.rs` processes observed Ethereum deposit events and mints the corresponding ckETH/ckERC20 tokens on the IC ledger. To prevent double-minting across async message boundaries, a `scopeguard` is armed before each ledger call: [1](#0-0) 

If the minter panics at any point after the ledger `transfer` call is issued but before the guard is defused, the guard fires and records `EventType::QuarantinedDeposit` for that deposit source: [2](#0-1) 

The `QuarantinedDeposit` event type is explicitly documented as requiring "manual intervention" and will not be retried automatically: [2](#0-1) 

Once quarantined, the deposit source is excluded from `events_to_mint()` permanently. The user's ETH or ERC20 tokens remain in the minter's Ethereum address with no on-chain mechanism to return them. There is no `refund_deposit` endpoint, no scheduled reimbursement task, and no equivalent of the ckBTC `schedule_deposit_reimbursement` flow for this case.

By contrast, the ckBTC minter and ckETH withdrawal path both implement explicit reimbursement queues: [3](#0-2) 

No analogous mechanism exists for quarantined ckETH/ckERC20 deposits.

The `QuarantinedReimbursement` event type exists for the reimbursement path but not for the deposit path: [4](#0-3) 

### Impact Explanation

**Impact: High.** A user who deposits ETH or any supported ERC20 token via the helper smart contract has their funds irrevocably transferred to the minter's Ethereum address at the Ethereum layer. If the IC minting callback panics (e.g., due to an unexpected ledger response, a canister memory trap, or a replica-level error during the async call), the deposit is quarantined. The user receives no ckETH/ckERC20 and has no mechanism to recover their deposited funds without a governance-approved canister upgrade that manually processes the quarantined event. This constitutes a permanent, unrecoverable loss of user funds under the described failure condition.

### Likelihood Explanation

**Likelihood: Low-Medium.** A panic in the minting callback can be triggered by:
- An unexpected response format from the ckETH ledger canister (e.g., after a ledger upgrade with breaking changes)
- A replica-level trap or out-of-cycles condition during the inter-canister call
- A bug in the minter's callback deserialization logic

The minter's own documentation acknowledges this risk by arming the quarantine guard. The condition is not user-controllable but is reachable through realistic operational scenarios (ledger upgrades, transient replica faults). Historical precedent exists: the ckETH minter was stuck due to an Ethereum RPC provider returning wrong results, demonstrating that external-dependency failures do occur in production. [5](#0-4) 

### Recommendation

1. Implement a `schedule_deposit_reimbursement`-style queue for quarantined deposits, analogous to the ckBTC deposit reimbursement flow. When a deposit is quarantined, record the depositor's IC principal and the deposit amount so that a future reimbursement mint can be issued.
2. Expose a canister endpoint (callable by the minter itself or NNS governance) to manually trigger reimbursement for quarantined deposits, similar to how ckBTC handles `pending_reimbursements`.
3. Consider adding a `QuarantinedDeposit` → `ReimbursedDeposit` state transition with appropriate double-minting guards, mirroring the `quarantine_withdrawal_reimbursement` / `reimburse_withdrawal_completed` pattern already used for withdrawals. [6](#0-5) 

### Proof of Concept

**Attacker-controlled entry path:** Any unprivileged user who calls `depositEth` or `depositErc20` on the Ethereum helper smart contract triggers the deposit flow. No special privileges are required.

**Vulnerable execution path:**

1. User calls `depositEth(amount, principal, subaccount)` on the Ethereum helper contract. ETH is transferred to the minter's Ethereum address and a `ReceivedEth` log event is emitted.
2. The ckETH minter scrapes the log via `scrape_logs()` and records `EventType::AcceptedDeposit`.
3. On the next timer tick, `mint()` is called. The `prevent_double_minting_guard` is armed for the deposit event.
4. The minter calls `client.transfer(...)` to the ckETH ledger. The call crosses an async message boundary.
5. A panic occurs in the callback (e.g., the ledger returns an unexpected Candid encoding after an upgrade, causing a deserialization trap).
6. The `scopeguard` fires, recording `EventType::QuarantinedDeposit { event_source }`. [7](#0-6) 

7. The deposit source is now permanently excluded from `events_to_mint()`. The user's ETH remains in the minter's Ethereum address. No ckETH is minted. No refund is issued. The user has no recourse without a canister upgrade.

The `QuarantinedDeposit` event is recorded in the minter's append-only event log and is observable via `get_events`, confirming the stuck state: [8](#0-7)

pub async fn reimburse_withdrawals<R: CanisterRuntime>(runtime: &R) {
    if state::read_state(|s| s.pending_withdrawal_reimbursements.is_empty()) {
        return;
    }
    let pending_reimbursements = state::read_state(|s| s.pending_withdrawal_reimbursements.clone());
    let mut error_count = 0;
    for (burn_index, reimbursement) in pending_reimbursements {
        // Ensure that even if we were to panic in the callback, after having contacted the ledger to mint the tokens,
        // this reimbursement request will not be processed again.
        let prevent_double_minting_guard = scopeguard::guard(burn_index, |index| {
            state::mutate_state(|s| {
                state::audit::quarantine_withdrawal_reimbursement(s, index, runtime)
            });
        });
        let memo = MintMemo::ReimburseWithdrawal {
            withdrawal_id: burn_index,
        };
        match runtime
            .mint_ckbtc(
                reimbursement.amount,
                reimbursement.account,
                Memo::from(crate::memo::encode(&memo)),
            )
            .await
        {
            Ok(mint_index) => {
                log!(
                    Priority::Debug,
                    "[reimburse_withdrawals]: Successfully reimbursed {:?} at mint block index {}",
                    reimbursement,
                    mint_index
                );
                state::mutate_state(|s| {
                    state::audit::reimburse_withdrawal_completed(s, burn_index, mint_index, runtime)
                });
            }
            Err(err) => {
                log!(
                    Priority::Info,
                    "[reimburse_withdrawals]: Failed to reimburse {:?}: {:?}. Will retry later",
                    reimbursement,
                    err
                );
                error_count += 1;
            }
        }
        // Defuse the guard. Note that in case of a panic in the callback (either before or after this point)
        // the defuse will not be effective (due to state rollback), and the guard that was
        // setup before the `mint_ckbtc` async call will be invoked.
        scopeguard::ScopeGuard::into_inner(prevent_double_minting_guard);
    }

    if error_count > 0 {
        log!(
            Priority::Info,
            "[reimburse_withdrawals] Failed to reimburse {error_count} withdrawal requests, retrying later."
        );
    }
}
