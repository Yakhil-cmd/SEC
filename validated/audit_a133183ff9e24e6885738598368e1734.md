Audit Report

## Title
`check_utxo` Retry Loop Returns Error on `Retriable` Response Instead of Retrying - (File: `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary

The `check_utxo` function contains a retry loop (`for i in 0..MAX_CHECK_TRANSACTION_RETRY`) that correctly uses `continue` for `NotEnoughCycles` but immediately returns `UpdateBalanceError::TemporarilyUnavailable` for `CheckTransactionStatus::Retriable`, despite the Bitcoin checker canister's interface explicitly documenting `Retriable` as "the result is not available, but calls **can be retried**." This causes every `update_balance` call to fail immediately when the checker is in a `Pending` or `HighLoad` state, preventing ckBTC minting for all depositors during those windows.

## Finding Description

In `check_utxo` (lines 400–450 of `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`), the retry loop handles `NotEnoughCycles` with `continue` at line 430, but the `Retriable` arm at lines 432–440 executes `return Err(UpdateBalanceError::TemporarilyUnavailable(...))`, exiting the loop entirely after the very first call: [1](#0-0) 

The `CheckTransactionStatus` type in `rs/bitcoin/checker/lib/types.rs` explicitly documents `Retriable` as retriable (line 82), covering three transient sub-cases: `Pending` (checker is mid-flight fetching Bitcoin tx data via HTTP outcall), `HighLoad` (fetch guard exhausted), and `TransientInternalError` (transient HTTP error): [2](#0-1) 

The `fetch.rs` code confirms `Pending` is returned when `FetchTxStatus::PendingOutcall` is set (line 84) and `HighLoad` when the fetch guard fails (line 95) — both are normal operational states: [3](#0-2) 

The existing test `should_call_check_transaction_again_when_cycles_not_enough` (lines 42–82 of `rs/bitcoin/ckbtc/minter/src/updates/tests.rs`) demonstrates that `NotEnoughCycles` correctly retries, but no equivalent test exists for `Retriable`, confirming the gap: [4](#0-3) 

## Impact Explanation

This is a **High** impact issue matching "Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm." During any window where the Bitcoin checker returns `Retriable` (which occurs in normal operation during HTTP outcalls for transaction data fetching), every `update_balance` call fails immediately with `TemporarilyUnavailable`. No ckBTC is minted for any depositor during those windows. While funds are not permanently lost (users can retry later), the ckBTC minting process is rendered unreliable for all depositors during normal checker load, constituting concrete protocol harm to the ckBTC Chain Fusion integration.

## Likelihood Explanation

`Retriable::Pending` is returned whenever the checker is mid-flight fetching Bitcoin transaction input data via HTTP outcalls — a multi-round process that is a normal part of every new transaction check. `Retriable::HighLoad` is returned when the fetch guard is exhausted under concurrent load. Both conditions occur during normal ckBTC deposit activity. No special privileges, adversarial setup, or unusual conditions are required. Any user who calls `update_balance` while the checker is in one of these states (which is routine) will trigger the bug.

## Recommendation

Change the `Retriable` arm at line 432 to use `continue` instead of `return Err(...)`, matching the `NotEnoughCycles` arm. The loop is already bounded by `MAX_CHECK_TRANSACTION_RETRY = 10` to prevent infinite looping:

```rust
CheckTransactionResponse::Unknown(CheckTransactionStatus::Retriable(status)) => {
    log!(
        Priority::Debug,
        "The Bitcoin checker canister is temporarily unavailable: {:?}, retrying...",
        status
    );
    continue; // was: return Err(UpdateBalanceError::TemporarilyUnavailable(...))
}
``` [5](#0-4) 

## Proof of Concept

1. User deposits BTC and calls `update_balance`.
2. The minter calls `check_utxo`, which calls `runtime.check_transaction(...)`.
3. The Bitcoin checker is mid-flight fetching transaction input data and returns `CheckTransactionResponse::Unknown(CheckTransactionStatus::Retriable(CheckTransactionRetriable::Pending))`.
4. The `Retriable` arm at line 432 executes `return Err(UpdateBalanceError::TemporarilyUnavailable(...))`.
5. The retry loop exits after the first call, with 9 iterations unused.
6. The user receives `TemporarilyUnavailable`; no ckBTC is minted.

A minimal unit test mirroring `should_call_check_transaction_again_when_cycles_not_enough` but substituting `NotEnoughCycles` responses with `Retriable(Pending)` responses would reproduce the failure: the test would expect a successful mint after retries but instead observe an error return after the first call. [6](#0-5)

#[derive(CandidType, Debug, Clone, Deserialize, Serialize)]
pub enum CheckTransactionStatus {
    /// Caller should call with a minimum of `CHECK_TRANSACTION_CYCLES_REQUIRED` cycles.
    NotEnoughCycles,
    /// The result is not available, but calls can be retried.
    Retriable(CheckTransactionRetriable),
    /// The result is unknown due to an irrecoverable error.
    Error(CheckTransactionIrrecoverableError),
}

#[derive(CandidType, Debug, Clone, Deserialize, Serialize)]
pub enum CheckTransactionRetriable {
    /// Work is already in progress, and the result is pending.
    Pending,
    /// The service is experience high load.
    HighLoad,
    /// There was a transient error fetching data.
    TransientInternalError(String),
}
