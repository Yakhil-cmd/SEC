### Title
Deprecated 64-bit Cycle Transfer APIs Cause Traps and Silent Truncation in SNS-WASM Canister — (File: `rs/nns/sns-wasm/canister/canister.rs`)

### Summary
The SNS-WASM canister's `CanisterApiImpl` uses multiple deprecated 64-bit cycle system APIs (`ic0_call_cycles_add`, `ic0_msg_cycles_available`, `ic0_msg_cycles_accept`, `ic0_canister_cycle_balance`) instead of their 128-bit successors. This is the direct IC analog to the Solidity `.transfer()`/`.send()` deprecation: the 64-bit APIs trap when cycle values exceed `u64::MAX`, and one call site uses `.try_into().unwrap()` which panics unconditionally on overflow. The file suppresses all warnings with `#![allow(deprecated)]`.

### Finding Description

In `rs/nns/sns-wasm/canister/canister.rs`, the `CanisterApiImpl` struct implements the `CanisterApi` trait using four deprecated 64-bit cycle APIs:

**1. `create_canister` — hard panic on overflow** [1](#0-0) 

`cycles.get()` returns a `u128`. The `.try_into().unwrap()` call panics (traps the canister) if the `Cycles` value exceeds `u64::MAX` (~18.4T cycles). `call_with_payment` is the deprecated 64-bit variant; the correct replacement is `call_with_payment128`.

**2. `this_canister_has_enough_cycles` — traps if canister balance > u64::MAX** [2](#0-1) 

`ic_cdk::api::canister_balance()` wraps the deprecated `ic0_canister_cycle_balance` syscall, which traps if the canister's balance cannot fit in 64 bits. The 128-bit replacement is `ic_cdk::api::canister_balance128()`.

**3. `message_has_enough_cycles` / `accept_message_cycles` — traps if message cycles > u64::MAX** [3](#0-2) 

`msg_cycles_available()` wraps the deprecated `ic0_msg_cycles_available` syscall (traps if available > u64::MAX). `msg_cycles_accept()` wraps the deprecated `ic0_msg_cycles_accept` syscall (silently caps at u64::MAX). The 128-bit replacements are `msg_cycles_available128()` and `msg_cycles_accept128()`.

**4. `send_cycles_to_canister` — 64-bit ceiling on cycle deposits** [4](#0-3) 

`call_with_payment` caps the cycle attachment at `u64::MAX`. The replacement is `call_with_payment128`.

The IC interface spec explicitly marks these as deprecated: [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

The same pattern appears in `rs/nns/handlers/root/impl/src/canister_management.rs` (also suppressed with `#![allow(deprecated)]`): [9](#0-8) [10](#0-9) 

### Impact Explanation

**Trap on canister balance overflow:** If the SNS-WASM canister's cycle balance ever exceeds `u64::MAX` (~18.4T cycles ≈ 18.4 ICP), every call to `this_canister_has_enough_cycles` traps. This permanently blocks all SNS deployments until the canister is upgraded — a governance-level outage.

**Trap on incoming message cycles overflow:** If any caller (including a governance-approved principal on the allowed list) attaches more than `u64::MAX` cycles to a `deploy_new_sns` call, `msg_cycles_available()` traps, aborting the deployment. The cycles are refunded, but the deployment fails.

**Hard panic in `create_canister`:** If the SNS deployment configuration specifies more than `u64::MAX` cycles for canister creation (e.g., via a governance proposal that sets a high initial cycle endowment), the `.try_into().unwrap()` panics, trapping the SNS-WASM canister mid-deployment and leaving partially-created SNS canisters in an inconsistent state.

**Silent truncation in `send_cycles_to_canister`:** The `u64` parameter ceiling means SNS sub-canisters can never receive more than `u64::MAX` cycles in a single deposit call, even if the SNS-WASM canister holds far more.

### Likelihood Explanation

The `u64::MAX` threshold (~18.4T cycles) is reachable for a long-lived system canister. The SNS-WASM canister accumulates cycles from every SNS deployment fee. The `#![allow(deprecated)]` suppression indicates the issue is known but unresolved. The `try_into().unwrap()` panic path is reachable if any governance proposal sets an SNS canister's initial cycle endowment above `u64::MAX`. The `msg_cycles_available()` trap path is reachable by any caller who can invoke `deploy_new_sns` and attaches an oversized cycle payment.

### Recommendation

Replace all deprecated 64-bit cycle APIs with their 128-bit counterparts throughout `rs/nns/sns-wasm/canister/canister.rs` and `rs/nns/handlers/root/impl/src/canister_management.rs`:

| Deprecated (64-bit) | Replacement (128-bit) |
|---|---|
| `call_with_payment(…, cycles_u64)` | `call_with_payment128(…, cycles_u128)` |
| `canister_balance()` | `canister_balance128()` |
| `msg_cycles_available()` | `msg_cycles_available128()` |
| `msg_cycles_accept(u64)` | `msg_cycles_accept128(u128)` |

Remove the `#![allow(deprecated)]` suppression after migration. Replace `.try_into().unwrap()` on `Cycles` with direct `u128` passing to `call_with_payment128`. Remove the `u64` type constraint from `send_cycles_to_canister`'s parameter.

### Proof of Concept

1. Deploy the SNS-WASM canister on a test subnet.
2. Top up the SNS-WASM canister's cycle balance to just above `u64::MAX` (e.g., via the CMC's `deposit_cycles` using `call_with_payment128`).
3. Submit a governance proposal to deploy a new SNS.
4. Observe that `this_canister_has_enough_cycles` traps because `ic_cdk::api::canister_balance()` (wrapping `ic0_canister_cycle_balance`) traps when the balance exceeds `u64::MAX`.
5. All subsequent SNS deployments are blocked until the canister is upgraded.

Alternatively, for the `msg_cycles_available()` path: submit a `deploy_new_sns` call from an allowed principal with `u64::MAX + 1` cycles attached. The canister traps at line 168 before any SNS canister is created.
