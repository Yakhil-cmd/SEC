Audit Report

## Title
Unauthenticated Third-Party `refresh_buyer_tokens` Allows Swap Capacity Exhaustion and Forced Participation - (File: rs/sns/swap/canister/canister.rs, rs/sns/swap/src/swap.rs)

## Summary

The `refresh_buyer_tokens` public update endpoint accepts an arbitrary `buyer` principal with no caller-identity check. Because the ticketing system is explicitly optional (skipped when no ticket exists for the target principal), any unprivileged attacker can transfer ICP to a victim's deterministically-derived subaccount of the swap canister and then call `refresh_buyer_tokens` on behalf of that victim, crediting them with participation they never authorized. Repeating this across many sybil principals exhausts `available_direct_participation_e8s`, blocks legitimate buyers, and can force premature swap commitment.

## Finding Description

**Unauthenticated buyer field:**
In `rs/sns/swap/canister/canister.rs` at lines 130–134, when `arg.buyer` is non-empty the endpoint uses the supplied principal directly without verifying it matches `ic_cdk::api::msg_caller()`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no caller check
};
```

**External balance used directly:**
`refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs` at lines 1152–1163 queries the raw on-chain ICP ledger balance of the victim's subaccount (deterministically derived via `principal_to_subaccount(&buyer)`) and uses it as the participation amount. Any ICP deposited to that subaccount by anyone is counted.

**Ticket check is optional and bypassable:**
The ticket guard at lines 1248–1272 is inside an `if let Some(...)` block with the explicit comment: *"If there exists no ticket for the buyer, the payment flow will simply ignore the ticket."* This is also confirmed by the integration test comment at `rs/nervous_system/integration_tests/tests/sns_lifecycle.rs:635–636`: *"The ticketing system is optional in the current implementation."* An attacker targeting a victim who has no open ticket faces zero ticket-based resistance.

**Exploit path:**
1. Attacker derives `victim_subaccount = principal_to_subaccount(&victim_principal)` (public, deterministic).
2. Attacker transfers `max_participant_icp_e8s` ICP to `Account { owner: swap_canister_id, subaccount: Some(victim_subaccount) }` on the ICP ledger.
3. Attacker calls `refresh_buyer_tokens { buyer: victim_principal.to_string(), confirmation_text: None }` from any identity.
4. No ticket exists for victim → ticket check skipped; ledger balance is read and credited.
5. Victim's `BuyerState` is set to `max_participant_icp_e8s`; `available_direct_participation_e8s` decreases by the same amount.
6. Repeat with N sybil victim principals to fill the swap to capacity.

## Impact Explanation

This is a **High** severity finding matching: *"Significant SNS infrastructure security impact with concrete user or protocol harm"* and *"Application/platform-level DoS … not based on raw volumetric DDoS."*

Concrete harms:
- **Swap capacity exhaustion**: Once `available_direct_participation_e8s` reaches zero, all new legitimate buyer calls are rejected. The cost per slot consumed is bounded by `max_participant_icp_e8s` (attacker loses this ICP to the SNS on commit or it is refunded to the victim on abort, but the disruption is real and repeatable).
- **Premature swap commitment**: Filling `current_direct_participation_e8s` to `max_direct_participation_icp_e8s` satisfies `can_commit`, allowing the swap to be committed before the intended participant set has joined.
- **Forced participation of unwilling users**: Victims are credited with ICP participation they never authorized; their ICP (sent by the attacker) is locked until finalization.

## Likelihood Explanation

- `refresh_buyer_tokens` is a standard `#[update]` endpoint reachable by any ingress sender with no access control.
- The `buyer` field is explicitly designed to accept third-party principals (bots/agents), making the bypass trivial.
- ICP transfers to arbitrary subaccounts of the swap canister are a normal part of the participation flow and are accepted without restriction by the ICP ledger.
- The ticketing system is documented as optional; no ticket is required for the attack to succeed.
- No special privilege, key, or majority is required. The attack is repeatable across as many sybil principals as the attacker is willing to fund.

## Recommendation

1. **Restrict `refresh_buyer_tokens` to self-refresh only**: Require `arg.buyer` to be empty or equal to `ic_cdk::api::msg_caller()`. If third-party refresh is genuinely needed (e.g., for bots), require a signed authorization from the buyer or restrict to an explicit allowlist.
2. **Make the ticket mandatory for new participants**: Require an open ticket to exist before crediting any new participation. This ensures only principals who explicitly initiated the flow (via `new_sale_ticket`, which enforces caller identity) can be credited.
3. **Use internal accounting**: Track the last-accepted balance per buyer internally and only accept increments that correspond to a verified transfer (e.g., by block index/memo), rather than trusting the raw ledger balance which can be inflated by any third party.

## Proof of Concept

Minimal state-machine integration test (extending existing patterns in `rs/sns/integration_tests/src/payment_flow.rs`):

```rust
// Setup: open swap with max_direct_participation = N * max_participant_icp_e8s
// Attacker identity: any principal (e.g., PrincipalId::new_user_test_id(999))
// Victim identities: sybil principals user_0..user_N (never interact with swap themselves)

for i in 0..N {
    let victim = PrincipalId::new_user_test_id(i);
    let victim_subaccount = principal_to_subaccount(&victim);
    let victim_swap_account = Account {
        owner: swap_canister_id.get().0,
        subaccount: Some(victim_subaccount),
    };

    // Step 1: Attacker transfers ICP to victim's subaccount
    icp_ledger_transfer(attacker, victim_swap_account, max_participant_icp_e8s);

    // Step 2: Attacker calls refresh_buyer_tokens for victim (no ticket exists)
    let result = state_machine.execute_ingress_as(
        attacker,
        swap_canister_id,
        "refresh_buyer_tokens",
        Encode!(&RefreshBuyerTokensRequest {
            buyer: victim.to_string(),
            confirmation_text: None,
        }).unwrap(),
    );
    assert!(result.is_ok()); // succeeds with no ticket check

    // Assert victim is now credited
    assert_eq!(get_buyer_state(victim).icp.amount_e8s, max_participant_icp_e8s);
}

// Assert swap capacity is exhausted
assert_eq!(swap.available_direct_participation_e8s(), 0);

// Assert legitimate buyer is now rejected
let legitimate_result = refresh_buyer_tokens(legitimate_buyer, max_participant_icp_e8s);
assert!(legitimate_result.is_err()); // "ICP target" error
```

This mirrors the existing `test_refresh_buyer_tokens_token_limit` test in `rs/sns/swap/tests/swap.rs`, which already demonstrates that exhausting `available_direct_participation_e8s` blocks further participation.
