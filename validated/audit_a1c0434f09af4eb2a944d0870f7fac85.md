Audit Report

## Title
Unbounded `ingress_expiries` Loop via Attacker-Controlled `ingress_start`/`ingress_end` Causes OOM Crash — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary

The ICP Rosetta server's unauthenticated `POST /construction/payloads` endpoint accepts attacker-controlled `ingress_start` and `ingress_end` values with no window-size guard. The loop that builds the `ingress_expiries` vector is unbounded: with `ingress_start=0` and `ingress_end=u64::MAX`, the loop executes billions of iterations pushing 8-byte values until the process is killed by OOM. The `Time::add_assign` implementation additionally wraps on overflow via a truncating `as u64` cast, making the loop infinite if OOM does not terminate the process first.

## Finding Description

**Root cause — unbounded loop with no window guard:**

The loop at `construction_payloads.rs` lines 99–107 iterates from `ingress_start` to `ingress_end` in steps of `interval = MAX_INGRESS_TTL - PERMITTED_DRIFT - 120s = 300s - 60s - 120s = 120s = 120_000_000_000 ns`: [1](#0-0) 

`ingress_start` and `ingress_end` are plain `Option<u64>` fields deserialized directly from the JSON request body with no validation or bounds check: [2](#0-1) 

They are passed directly into `Time::from_nanos_since_unix_epoch` before the loop with no window-size check: [3](#0-2) 

**Overflow behavior makes the loop infinite after OOM:**

`Time::add_assign` delegates to `from_duration`, which performs a truncating `as u64` cast on the `u128` result of `Duration::as_nanos()`: [4](#0-3) [5](#0-4) 

When `now` approaches `u64::MAX`, `Duration::from_nanos(self.0) + other` produces a `Duration` whose `as_nanos()` exceeds `u64::MAX`; the `as u64` cast silently truncates it to a small value still less than `ingress_end = u64::MAX`, so the loop condition `now < ingress_end` remains true indefinitely.

**No authentication on the endpoint:** [6](#0-5) 

**ICRC1 Rosetta has explicit guards; ICP Rosetta does not:**

The ICRC1 implementation rejects invalid windows before the loop: [7](#0-6) 

The ICP Rosetta `construction_payloads` handler has no equivalent guard.

**Interval constants confirmed:** [8](#0-7) [9](#0-8) 

## Impact Explanation

With `ingress_start=0, ingress_end=u64::MAX`, the loop pushes one `u64` (8 bytes) per 120-second interval. At 8 bytes per entry, a process with 8 GB of RAM exhausts memory after ~1 billion iterations. The process is killed by the OS OOM killer before the wrap point is reached. Even if it were not, the wrap-on-overflow behavior restarts `now` from a small value, making the loop infinite. The Rosetta server process crashes, taking down the ICP Rosetta API service used by exchanges and wallets. This matches the allowed impact: **Significant Rosetta security impact with concrete user or protocol harm — High ($2,000–$10,000)**.

## Likelihood Explanation

- No authentication is required; the endpoint is a plain unauthenticated HTTP POST.
- The exploit requires a single HTTP request with two integer fields set to extreme values.
- No rate limiting is present in the server setup.
- The attack is trivially repeatable: the server can be crashed again immediately after restart.
- Any unprivileged network-reachable attacker can trigger this.

## Recommendation

Add a maximum ingress window size check before the loop, mirroring the ICRC1 Rosetta guard:

```rust
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 3600);
if ingress_end
    .checked_duration_since(ingress_start)
    .map_or(true, |w| w > MAX_INGRESS_WINDOW)
{
    return Err(ApiError::invalid_request("ingress window too large"));
}
```

This caps the vector at `24h / 120s = 720` entries maximum. Additionally, `Time::add_assign` should use `checked_add` or saturating arithmetic to avoid silent truncation.

## Proof of Concept

```bash
curl -X POST http://<rosetta-host>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
    "operations": [{"operation_identifier":{"index":0},"type":"TRANSACTION","account":{"address":"<valid_addr>"},"amount":{"value":"-1","currency":{"symbol":"ICP","decimals":8}}}],
    "public_keys": [{"hex_bytes":"<valid_pubkey>","curve_type":"edwards25519"}],
    "metadata": {
      "ingress_start": 0,
      "ingress_end": 18446744073709551615
    }
  }'
```

The server process exhausts memory and is killed by OOM before returning a response. A unit test can reproduce this by calling `construction_payloads` directly with `ingress_start = Time::from_nanos_since_unix_epoch(0)` and `ingress_end = Time::from_nanos_since_unix_epoch(u64::MAX)` and asserting it returns an error rather than looping.
