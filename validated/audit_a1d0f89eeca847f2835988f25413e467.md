### Title
Silo Whitelist Removal Permanently Freezes User ERC-20 Token Balances - (File: engine/src/engine.rs)

---

### Summary

In Aurora Engine's Silo mode, when the `Address` whitelist is enabled, removing a user's EVM address from the whitelist via `remove_entry_from_whitelist` permanently blocks all EVM transactions from that address. Because the only path to exit ERC-20 tokens from the Aurora EVM requires submitting an EVM transaction (calling `withdrawToNear` or `withdrawToEthereum` on the ERC-20 contract), a de-whitelisted user's ERC-20 token balances are permanently frozen with no on-chain recovery mechanism.

---

### Finding Description

Aurora Engine's Silo mode enforces an `Address` whitelist on every EVM transaction submission. The gate is applied inside `submit_with_alt_modexp` in `engine/src/engine.rs`:

```rust
// engine/src/engine.rs:1052
assert_access(&io, env, &transaction)?;
```

`assert_access` delegates to `silo::is_allow_submit`, which checks the `Address` whitelist:

```rust
// engine/src/engine.rs:1756-1774
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }
    Ok(())
}
```

`is_allow_submit` requires the sender's EVM address to be present in the `Address` whitelist:

```rust
// engine/src/contract_methods/silo/mod.rs:136-138
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}
```

```rust
// engine/src/contract_methods/silo/mod.rs:155-158
fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
    !list.is_enabled() || list.is_exist(address)
}
```

The Silo owner can remove any address from the whitelist at any time:

```rust
// engine/src/contract_methods/silo/mod.rs:92-95
pub fn remove_entry_from_whitelist<I: IO + Copy>(io: &I, args: &WhitelistArgs) {
    let (kind, entry) = get_kind_and_entry(args);
    Whitelist::init(io, kind).remove(entry);
}
```

Once an address is removed, every EVM transaction from it is rejected with `EngineErrorKind::NotAllowed`. This blocks the user from calling `withdrawToNear` or `withdrawToEthereum` on any ERC-20 contract they hold tokens in.

The ERC-20 fallback mechanism (`is_allow_receive_erc20_tokens` / `erc20_fallback_address`) only applies to **incoming** token transfers via `ft_on_transfer` — it redirects tokens destined for a non-whitelisted address to a fallback address at mint time:

```rust
// engine/src/engine.rs:818-822
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
```

This mechanism provides **no relief** for tokens already held by a de-whitelisted address. There is no admin-callable "seize" or "force-exit" function that can move ERC-20 tokens out of a frozen address.

Note: the NEAR-level `withdraw` function in `engine/src/contract_methods/connector.rs` (for the ETH base token) does not check the whitelist, so native ETH balances are recoverable. Only ERC-20 token balances are permanently frozen.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any ERC-20 tokens held by a de-whitelisted EVM address in a Silo deployment are permanently inaccessible. The user cannot call `withdrawToNear`, `withdrawToEthereum`, or any other EVM function. The Silo operator also has no on-chain mechanism to recover or redistribute those tokens. The tokens remain locked in the ERC-20 contract's storage indefinitely.

---

### Likelihood Explanation

Silo mode is an explicitly supported production deployment mode for enterprise/private Aurora instances. The `remove_entry_from_whitelist` operation is a routine administrative action (e.g., for regulatory compliance, banning bad actors). Any Silo deployment that (a) enables the `Address` whitelist and (b) ever removes a user who holds ERC-20 token balances will trigger this freeze. The scenario is directly analogous to the passport-burning case in the original report and is realistic in any compliance-oriented Silo deployment.

---

### Recommendation

1. **Seize/force-exit function**: Add an owner-only function that can transfer ERC-20 tokens from a frozen address to a designated recovery address, bypassing the whitelist check (similar to the original report's suggestion of a "seize funds" method).
2. **Whitelist-exempt withdrawal path**: Allow a de-whitelisted address to submit a restricted transaction type (exit-only) that is exempt from the `assert_access` check, so users can always recover their own tokens.
3. **Pre-removal drain window**: Before removing an address from the whitelist, require a time-lock during which the user can exit their funds.
4. **Documentation**: At minimum, explicitly document that removing an address from the whitelist permanently freezes any ERC-20 balances held by that address, so Silo operators are aware of the consequence before acting.

---

### Proof of Concept

1. Deploy Aurora Engine in Silo mode with the `Address` whitelist enabled.
2. Whitelist address `0xUSER` and have the user deposit NEP-141 tokens via `ft_transfer_call`, receiving ERC-20 tokens at `0xUSER` inside the EVM.
3. The Silo owner calls `remove_entry_from_whitelist` with `WhitelistKind::Address` for `0xUSER`.
4. `0xUSER` attempts to call `withdrawToNear` on the ERC-20 contract by submitting an EVM transaction.
5. `submit_with_alt_modexp` calls `assert_access` → `is_allow_submit` → `is_address_allowed` → returns `false` → transaction rejected with `EngineErrorKind::NotAllowed`.
6. The ERC-20 balance at `0xUSER` is permanently frozen. No admin function exists to recover it. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)
