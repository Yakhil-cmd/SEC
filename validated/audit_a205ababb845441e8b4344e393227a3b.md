### Title
Zero Gas Price Bypasses Silo Fixed-Gas Fee Accounting - (`engine/src/engine.rs`)

---

### Summary

In Silo mode, the fixed-gas fee charged per transaction is computed as `fixed_gas * effective_gas_price`. Because `effective_gas_price` is derived entirely from the user-controlled `max_fee_per_gas` field, a user who sets `max_fee_per_gas = 0` (valid when `block_base_fee_per_gas = 0`, which is the Aurora default) causes `effective_gas_price = 0` and therefore `prepaid_amount = 0`. The Silo operator's fixed-gas fee is completely bypassed.

---

### Finding Description

In `submit_with_alt_modexp`, the Silo fixed gas is retrieved unconditionally: [1](#0-0) 

It is then passed to `charge_gas`. Inside `charge_gas`, the early-return guard that skips gas charging reads: [2](#0-1) 

The `fixed_gas.is_none()` clause was added so that the zero-fee fast path is not taken when a fixed gas is configured. However, this guard only prevents the *early return*; it does not prevent `effective_gas_price` from being zero. When `max_fee_per_gas = 0` and `block_base_fee_per_gas = 0`:

```
priority_fee_per_gas = min(max_priority_fee_per_gas, 0 - 0) = 0
effective_gas_price  = 0 + 0 = 0
prepaid_amount       = fixed_gas * 0 = 0
``` [3](#0-2) 

The sender's balance is debited by zero. Then in `refund_unused_gas`, the first guard short-circuits immediately: [4](#0-3) 

So the relayer receives nothing either. The entire fixed-gas fee mechanism is silently skipped.

The `SiloParamsArgs` struct shows that `fixed_gas` is the operator's intended per-transaction charge: [5](#0-4) 

The only existing guard against misuse of `fixed_gas` checks that `gas_limit >= fixed_gas`, not that `gas_price > 0`: [6](#0-5) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

The Silo operator deploys `fixed_gas` specifically to collect a predictable fee from every transaction. Any user can submit a valid signed Ethereum transaction with `max_fee_per_gas = 0`. Because Aurora's default `block_base_fee_per_gas` is zero (as confirmed by the default `EngineState`), this is always a valid transaction that passes the `max_fee_per_gas < block_base_fee_per_gas` check. The operator collects zero fees from such transactions, losing all expected yield indefinitely for every user who exploits this.

---

### Likelihood Explanation

**High.** Setting `max_fee_per_gas = 0` is a trivial, one-field change in any standard Ethereum transaction. No special privileges, no contract deployment, and no governance interaction are required. Any unprivileged EVM user interacting with a Silo-mode Aurora deployment can exploit this on every transaction. The attack is free (costs the attacker nothing) and repeatable.

---

### Recommendation

Decouple the fixed-gas fee from the user-supplied gas price. When `fixed_gas` is `Some`, the engine should enforce a minimum effective gas price greater than zero, or charge the fixed-gas fee as a flat Wei deduction independent of `effective_gas_price`. Concretely, add a check such as:

```rust
if fixed_gas.is_some() && effective_gas_price.is_zero() {
    return Err(GasPaymentError::MaxFeePerGasLessThanBaseFee);
}
```

or compute the fixed-gas cost using a Silo-configured minimum gas price rather than the transaction-supplied one.

---

### Proof of Concept

1. Silo operator calls `set_silo_params` with `fixed_gas = EthGas::new(1_000_000)` and any `erc20_fallback_address`. Aurora's `block_base_fee_per_gas` is 0 (default).
2. Attacker constructs a standard EIP-1559 transaction with `max_fee_per_gas = 0`, `max_priority_fee_per_gas = 0`, and `gas_limit >= 1_000_000` (to pass the `FixedGasOverflow` check at line 1066).
3. Attacker submits via `submit`.
4. Inside `charge_gas`: `effective_gas_price = 0`, `prepaid_amount = 1_000_000 * 0 = 0`. Sender balance unchanged.
5. Inside `refund_unused_gas`: `effective_gas_price.is_zero()` → early return. Relayer balance unchanged.
6. Transaction executes successfully. Operator receives **0 Wei** instead of the expected `1_000_000 * gas_price` fee.

The existing test `test_relayer_balance_after_transfer` only exercises `ONE_GAS_PRICE = 1` and never tests the `gas_price = 0` case, confirming the gap. [7](#0-6)
