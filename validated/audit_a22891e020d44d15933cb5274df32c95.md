Audit Report

## Title
ckETH Minter Permanently Skips Blocks With Excessive Deposit Logs, Causing Irrecoverable ETH/ERC-20 Deposit Loss - (File: `rs/ethereum/cketh/minter/src/deposit.rs`)

## Summary
In `scrape_block_range`, when a single-block `eth_getLogs` response exceeds the configured size limit, the minter emits `EventType::SkippedBlockForContract` and immediately advances `last_scraped_block_number` to that block. Because all future scraping starts from `last_scraped_block_number + 1`, the skipped block is never re-examined and all ETH or ERC-20 deposits it contained are permanently unprocessed, leaving user funds locked in the Ethereum helper contract with no ckETH/ckERC-20 ever minted.

## Finding Description
In `rs/ethereum/cketh/minter/src/deposit.rs` lines 273–283, when `from_block == to_block` and `is_response_too_large` is true, the code unconditionally calls `S::update_last_scraped_block_number(s, to_block)` after recording the skip event:

```rust
if from_block == to_block {
    mutate_state(|s| {
        process_event(s, EventType::SkippedBlockForContract {
            contract_address,
            block_number: to_block,
        });
    });
    mutate_state(|s| S::update_last_scraped_block_number(s, to_block));
```

`scrape_until_block` computes its starting block as `last_scraped_block_number.checked_increment()` (lines 200–204), so the skipped block is permanently excluded from all future scraping iterations. The `SkippedBlockForContract` event is handled in `apply_state_transition` (audit.rs lines 120–125) solely by calling `record_skipped_block_for_contract`, which only inserts the block number into a `BTreeSet` inside `skipped_blocks` (state.rs lines 386–396). No code path reads `skipped_blocks` to re-attempt processing; grep across the entire minter source confirms zero matches for any re-scrape or recovery logic. All three scraping pipelines (`ReceivedEthLogScraping`, `ReceivedErc20LogScraping`, `ReceivedEthOrErc20LogScraping`) share the same `scrape_block_range` function (deposit.rs lines 156–158) and are equally affected.

## Impact Explanation
Any user whose `depositEth` or `depositErc20` transaction was included in a skipped block permanently loses their deposited assets. The ETH or ERC-20 tokens remain locked in the Ethereum helper contract; no ckETH or ckERC-20 is ever minted; no refund or error is surfaced to the user. This constitutes permanent, silent loss of in-scope chain-key/ledger assets (ckETH, ckERC-20), matching the **Critical** impact class: *Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant ICP/Cycles or in-scope chain-key/ledger assets*. The scale of loss is unbounded — a single skipped block can contain arbitrarily many deposits from many distinct users.

## Likelihood Explanation
An unprivileged Ethereum user can deliberately trigger this by submitting enough `depositEth` calls within a single block so that the `eth_getLogs` response for that block exceeds `ETH_GET_LOGS_INITIAL_RESPONSE_SIZE_ESTIMATE + HEADER_SIZE_LIMIT`. The binary-split retry loop will eventually converge to `from_block == to_block` still exceeding the limit, triggering the skip. No special privileges, governance access, or node control are required. The condition can also arise organically during periods of high protocol activity, making it a latent risk even without a deliberate attacker.

## Recommendation
1. **Do not advance the cursor on skip**: Remove the `S::update_last_scraped_block_number(s, to_block)` call in the `from_block == to_block` skip branch. Record the skip separately without moving the scraping cursor, so the block remains eligible for future retry.
2. **Add a privileged re-scrape endpoint**: Expose a canister update callable via NNS governance proposal that re-scrapes a specific block number for a given contract address, bypassing the normal cursor logic.
3. **Raise a critical alert**: Increment a critical error counter (analogous to `critical_error_induct_response_failed` in message routing) when `SkippedBlockForContract` is emitted, so operators are immediately notified.

## Proof of Concept
1. Attacker submits N `depositEth(amount, principal, subaccount)` calls to the Ethereum helper contract within a single block, where N is large enough that `eth_getLogs` for that block returns a response exceeding `ETH_GET_LOGS_INITIAL_RESPONSE_SIZE_ESTIMATE + HEADER_SIZE_LIMIT`.
2. The ckETH minter's periodic `scrape_logs` timer fires and calls `scrape_until_block::<ReceivedEthLogScraping>`.
3. `scrape_block_range` binary-splits the range until `from_block == to_block` equals the target block.
4. The single-block `eth_getLogs` response is still too large; `is_response_too_large` returns `true`.
5. `EventType::SkippedBlockForContract { contract_address, block_number }` is emitted and `last_scraped_block_number` is advanced to the target block.
6. All subsequent scraping starts from `target_block + 1`. The target block is never processed again.
7. All users who deposited in that block never receive ckETH; their ETH is permanently locked in the helper contract.

A deterministic integration test can reproduce this by mocking the EVM RPC client to return an `HttpOutcallError` with `is_response_too_large = true` for a single-block range, then asserting that `last_scraped_block_number` advances past the block while `skipped_blocks` contains it and no `AcceptedDeposit` events are recorded for transactions in that block.
