Audit Report

## Title
Unprivileged Caller Can Trigger `panic!()` in `manage_neuron_internal` via `command: None` — (File: rs/nns/governance/src/governance.rs)

## Summary
The `manage_neuron_internal` function contains a bare `None => panic!()` arm at line 6203 that is reachable by any unprivileged caller who submits a `ManageNeuron` message with any `NeuronIdOrSubaccount::NeuronId(...)` value and `command: None`. The public wrapper `manage_neuron` uses `.unwrap_or_else(ManageNeuronResponse::error)` which only catches `Result::Err` — it does not catch Rust panics. Each such call causes a governance canister trap and state rollback for that message.

## Finding Description
The public entry point at lines 6081–6089 wraps `manage_neuron_internal` with `.unwrap_or_else(ManageNeuronResponse::error)`, which handles only `Err` variants, not panics.

The reachable path to `None => panic!()` at line 6203 with `command: None`:

1. **Lines 6096–6103**: `None.unwrap_or_default()` evaluates to `false`, so `check_heap_can_grow()` runs and succeeds under normal conditions.
2. **Line 6106**: The `if let Some(Command::ClaimOrRefresh(...))` guard does not match `None` — skipped.
3. **Lines 6150 / 6207–6229**: `neuron_id_from_manage_neuron(mgmt)?` is called. Critically, when `neuron_id_or_subaccount` is `Some(NeuronIdOrSubaccount::NeuronId(id))`, the function returns `Ok(id)` **without checking whether the neuron exists** (line 6212: `Some(NeuronIdOrSubaccount::NeuronId(id)) => Ok(id)`). Any arbitrary `NeuronId` value suffices — no valid/existing neuron is required.
4. **Line 6203**: `match &mgmt.command` is entered with `None`, hitting `None => panic!()`.

The `ClaimOrRefresh` arm at line 6191–6193 also contains `panic!("This should have already returned")`, but that arm requires `command` to be `Some(Command::ClaimOrRefresh(...))` and is not reachable via the `None` path.

## Impact Explanation
On the Internet Computer, a Rust `panic!()` inside a canister message handler causes a canister trap. The governance canister remains alive and processes subsequent messages, but each malicious call produces a trap and state rollback for that invocation. This constitutes an application-level DoS on the NNS governance canister's `manage_neuron` endpoint: any caller can force a trap on demand at negligible cost (only the cycles for the ingress message), violating the invariant that `manage_neuron` must never trap on any well-formed Candid input. This matches the allowed impact: **High — Application/platform-level DoS on NNS governance with concrete protocol harm**.

## Likelihood Explanation
- Requires zero privileges, no key material, no social engineering
- Any `NeuronId` value works — no existing neuron required
- The path is deterministic and locally testable
- The `ManageNeuron` Candid interface is public; `command: None` is a valid encoding
- Repeatable at will by any external caller

## Recommendation
Replace the bare `panic!()` at line 6203 with a proper `Err` return:

```rust
None => Err(GovernanceError::new_with_message(
    ErrorType::InvalidCommand,
    "No command specified in ManageNeuron request.",
)),
```

Additionally, the `Some(Command::ClaimOrRefresh(_)) => panic!(...)` arm at line 6191–6193 should similarly be replaced with an `Err` return, as it is also unreachable by design but would trap if somehow reached.

## Proof of Concept

```rust
// Unit test sketch — no existing neuron required
let response = governance.manage_neuron(
    &caller_principal,
    &ManageNeuron {
        neuron_id_or_subaccount: Some(NeuronIdOrSubaccount::NeuronId(NeuronId { id: 12345 })),
        command: None,
        id: None,
    },
).await;
// Expected: ManageNeuronResponse::error(InvalidCommand)
// Actual:   canister trap at governance.rs:6203 — `None => panic!()`
```

The `neuron_id_from_manage_neuron` function returns `Ok(NeuronId { id: 12345 })` for any supplied `NeuronId` without a store lookup, so no setup of neuron state is needed.
