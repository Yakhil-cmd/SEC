### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Silently Overcharges Users — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` applies a `delta_gas` correction only in one direction: when native resource consumption exceeds EVM gas consumption (`delta_gas > 0`), `gas_used` is increased and the user pays more. When native resource consumption is *less* than EVM gas consumption (`delta_gas < 0`), `gas_used` is **not** reduced. The user is therefore overcharged by `gas_price × |delta_gas|` tokens that are never returned. A developer-acknowledged `TODO` comment marks the gap.

---

### Finding Description

In `compute_gas_refund`, after computing `gas_used` from remaining ergs and `native_used` from remaining native resources, the code computes:

```rust
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);

if delta_gas > 0 {
    // native resource consumption is more than gas consumption — charge extra
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [1](#0-0) 

The `delta_gas < 0` branch is entirely absent. The `TODO` comment on line 80 explicitly acknowledges this gap.

The resulting `gas_used` is then used in two places:

1. **User refund** (ZK flow): `gas_price × (tx_gas_limit − gas_used)` — the user receives a smaller refund than deserved. [2](#0-1) 

2. **Operator payment** (ZK flow): `gas_price × gas_used` — the operator receives more than it should. [3](#0-2) 

The same `gas_used` flows through the Ethereum-compatible path as well. [4](#0-3) 

The ZKsync OS documentation confirms the intended symmetric design:

> `deltaGas := (nativeUsed / nativePerGas) − gasUsed`. If `deltaGas > 0`, we add it to `gasUsed`… [5](#0-4) 

The document describes only the positive case; the negative case is unspecified and unimplemented.

---

### Impact Explanation

When `delta_gas < 0` (EVM ergs are the binding constraint, native resources are underused):

- The user is overcharged by `gas_price × |delta_gas|` tokens.
- Without `burn_base_fee`: the full excess is transferred to the operator.
- With `burn_base_fee`: `base_fee × |delta_gas|` is permanently burned (irrecoverable), and `(gas_price − base_fee) × |delta_gas|` goes to the operator.

In the burn case, user funds are permanently destroyed with no recourse. This is a direct, quantifiable loss of user funds triggered by a normal transaction submission.

---

### Likelihood Explanation

`delta_gas < 0` occurs whenever:

```
remaining_ergs / ERGS_PER_GAS  <  remaining_native / native_per_gas
```

i.e., the transaction exhausts its EVM gas budget before exhausting its native resource budget. This is the **common case** for transactions that:

- Set a high `gas_price` (high priority fee), which inflates `native_per_gas = gas_price / native_price` and therefore the native limit;
- Perform EVM-heavy computation (many opcodes) that consumes ergs quickly but is relatively cheap to prove natively.

Any unprivileged user submitting a standard EIP-1559 transaction with a non-trivial priority fee can trigger this path. No special access or privileged role is required.

---

### Recommendation

Apply the negative delta symmetrically:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else {
    // delta_gas < 0: native resources were underused; reduce gas_used
    // so the user is refunded for the unused native budget.
    gas_used = gas_used.saturating_sub((-delta_gas) as u64);
    // Ensure gas_used never falls below minimal_gas_used (already applied above).
    gas_used = gas_used.max(minimal_gas_used);
}
```

This mirrors the existing upward adjustment and eliminates the asymmetric overcharge.

### Proof of Concept

**Setup:**
- `native_price = 500`, `gas_price = 1000` → `native_per_gas = 2`
- `gas_limit = 100_000`
- Transaction uses `80_000` EVM gas (ergs-based) but only `100_000` native units.

**Computation:**
- `gas_used` (ergs-based) = `80_000`
- `native_used` = `100_000 * 2 − remaining_native` ≈ `100_000` (if native barely used)
- `delta_gas = (100_000 / 2) − 80_000 = 50_000 − 80_000 = −30_000`

**Current behavior:**
- `delta_gas < 0` → branch not taken → `gas_used` stays at `80_000`
- User refund = `gas_price × (100_000 − 80_000)` = `1000 × 20_000` = `20_000_000 wei`
- Operator receives `1000 × 80_000` = `80_000_000 wei`

**Correct behavior:**
- `gas_used` should be reduced to `80_000 − 30_000 = 50_000`
- User refund = `1000 × 50_000` = `50_000_000 wei`
- Operator receives `1000 × 50_000` = `50_000_000 wei`

**User loss per transaction:** `30_000_000 wei` (30,000 gas × gas_price), silently transferred to the operator (or burned under `burn_base_fee`). [6](#0-5)
