### Title
Unrestricted `execute_scheduled` Allows Any Caller to Front-Run and Prematurely Execute a User's Scheduled XCC Promise, Temporarily Freezing Funds - (File: etc/xcc-router/src/lib.rs)

---

### Summary

The `execute_scheduled` function in the XCC router contract has no access control and is intentionally callable by any NEAR account. An attacker who observes a scheduled promise (nonce is publicly logged on-chain) can call `execute_scheduled(nonce)` before the legitimate user, removing the promise from storage and executing it prematurely. If the promise execution fails at that premature moment (e.g., the target contract is not yet ready, or required conditions are unmet), the promise is permanently gone from storage while the user's wNEAR remains stranded in the router sub-account. The user's own subsequent `execute_scheduled` call fails with `ERR_PROMISE_NOT_FOUND`.

---

### Finding Description

The XCC router (`etc/xcc-router/src/lib.rs`) stores user-scheduled NEAR cross-contract promises in a `LookupMap<u64, PromiseArgs>` keyed by a monotonically incrementing nonce. The `schedule` function (callable only by the parent Aurora engine) inserts a promise and emits a log: `"Promise scheduled at nonce {nonce}"`. This log is publicly visible on-chain.

The `execute_scheduled` function:

```rust
/// It is intentional that this function can be called by anyone (not just the parent).
/// There is no security risk to allowing this function to be open because it can only
/// act on promises that were created via `schedule`.
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

The promise is **removed from storage before execution**. If the resulting NEAR receipt fails (e.g., the target contract rejects the call due to unmet preconditions), the promise is gone and cannot be retried via the same mechanism. The wNEAR that was already transferred to the router by the engine's `withdraw_wnear_to_router` callback remains stranded in the router sub-account.

The attack path:
1. User calls the XCC precompile with `CrossContractCallArgs::Delayed(...)`, which causes the engine to call `schedule` on the router and transfer wNEAR to it.
2. The nonce is emitted in a public log.
3. Attacker observes the log and calls `execute_scheduled(nonce)` before the user, at a moment when the target contract is not ready (e.g., not storage-registered, paused, or lacking balance).
4. The promise is removed from storage and executed; the execution fails.
5. The user's `execute_scheduled(nonce)` call fails with `ERR_PROMISE_NOT_FOUND`.
6. The wNEAR that was sent to the router is now stranded there; the user must schedule a new promise to recover it.

---

### Impact Explanation

**High — Temporary freezing of funds.**

The user's wNEAR (transferred to the router sub-account as part of the XCC flow) is stranded in the router contract after the attacker-triggered promise execution fails. The user cannot re-execute the original promise (it was removed from storage). They must issue a new EVM transaction to re-schedule, paying gas again, while the attacker can repeat the front-run indefinitely. During this window, the user's wNEAR is inaccessible for its intended purpose and effectively frozen in the router sub-account.

---

### Likelihood Explanation

**Medium.** NEAR does not have a traditional gas-price mempool, but NEAR transactions are publicly observable in real time via RPC and indexers. The nonce is emitted as a plaintext log. Any NEAR account can call `execute_scheduled` with zero cost beyond gas. An attacker monitoring the chain for XCC schedule events can reliably front-run by submitting their `execute_scheduled` call in the same or next block, particularly when the target contract is known to be in a state that will reject the call (e.g., during a known maintenance window or before a required storage deposit is made).

---

### Recommendation

Restrict `execute_scheduled` so that only the user whose EVM address corresponds to this router sub-account (or the parent Aurora engine on their behalf) can trigger execution. The router sub-account name encodes the user's EVM address as a hex prefix (`{address}.{aurora}`), so the caller can be validated against `env::current_account_id()`. Alternatively, require that `execute_scheduled` can only be called by `env::predecessor_account_id() == parent` or by the EVM address owner verified through a signed message, eliminating the open-to-anyone attack surface.

---

### Proof of Concept

**Setup:**
- Aurora Engine deployed; XCC router code uploaded via `factory_update`.
- User (EVM address `0xABCD...`) calls the XCC precompile with `CrossContractCallArgs::Delayed(ft_transfer_promise)` where `ft_transfer_promise` targets a NEP-141 token contract that requires a storage deposit the router has not yet made.
- Engine schedules the promise in the router at nonce `0` and transfers wNEAR to the router.
- Log emitted: `"Promise scheduled at nonce 0"`.

**Attack:**
- Attacker observes the log and immediately calls `execute_scheduled({"nonce": "0"})` on the router sub-account (`abcd....aurora`).
- The router removes the promise from `scheduled_promises` and fires the `ft_transfer` receipt.
- The NEP-141 contract rejects the call (`ERR_ACCOUNT_NOT_REGISTERED`) because the router is not storage-registered.
- The promise is gone from storage; wNEAR remains in the router.

**Result:**
- User calls `execute_scheduled({"nonce": "0"})` → panics with `ERR_PROMISE_NOT_FOUND`.
- User's wNEAR is stranded in the router sub-account.

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
