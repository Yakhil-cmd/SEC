### Title
Missing Zero-Address Validation in `receive_base_tokens` and `receive_erc20_tokens` Allows Permanent Fund Freeze — (File: `engine/src/engine.rs`)

---

### Summary

The `receive_base_tokens` and `receive_erc20_tokens` functions in `engine/src/engine.rs` parse a recipient EVM address from the user-controlled `msg` field of `ft_on_transfer` without validating that the parsed address is not the zero address. A user who bridges ETH or ERC-20 tokens to Aurora with `msg = "0000000000000000000000000000000000000000"` will have their tokens permanently minted to the zero address, from which they can never be recovered.

---

### Finding Description

The `ft_on_transfer` entrypoint in `engine/src/contract_methods/connector.rs` dispatches to one of two minting paths depending on the caller: [1](#0-0) 

**Path 1 — ETH (base token) bridging via `receive_base_tokens`:**

The function parses the recipient from `args.msg` using `FtTransferMessageData::try_from`, then directly credits the balance: [2](#0-1) 

The parser in `engine-types/src/parameters/connector.rs` accepts any 40-character hex string — including 40 zeros — as a valid address, with no zero-address guard: [3](#0-2) 

**Path 2 — ERC-20 token bridging via `receive_erc20_tokens`:**

The function parses the recipient from the first 40 hex characters of `args.msg` with no zero-address check, then calls the ERC-20 `mint` selector with that address: [4](#0-3) 

The calldata is assembled by `setup_receive_erc20_tokens_input`, which encodes the recipient directly into the ERC-20 `mint(address,uint256)` call: [5](#0-4) 

In both paths, if the parsed recipient is `0x0000000000000000000000000000000000000000`, the tokens are credited to the zero address in Aurora's EVM state. No private key controls the zero address; the funds are permanently inaccessible.

---

### Impact Explanation

**Permanent fund freeze (High).** Bridged ETH or ERC-20 tokens minted to the zero address are irrecoverable. The NEP-141 tokens have already been transferred to the Aurora contract on the NEAR side, so the bridge accounting is complete — but the EVM-side credit goes to an address no one controls. There is no admin recovery path in the engine for funds sitting at the zero address.

---

### Likelihood Explanation

**Medium.** The `msg` field is entirely user-controlled and is passed verbatim through the NEP-141 `ft_transfer_call` → `ft_on_transfer` flow. A user who programmatically constructs the bridge call and makes a mistake (e.g., passes an uninitialized buffer, a default-zeroed struct, or an explicit burn intent) will silently lose their funds. The protocol provides no warning and no rejection.

---

### Recommendation

Add an explicit zero-address guard in both `receive_base_tokens` and `receive_erc20_tokens` immediately after the recipient is parsed, before any balance mutation or ERC-20 mint call is issued. Return an appropriate `ContractError` so that the `ft_on_transfer` handler returns the full token amount to the sender (the existing error-return path at line 96 already handles this correctly).

```rust
if recipient == Address::zero() {
    return Err(errors::ERR_ZERO_ADDRESS_RECIPIENT);
}
```

Apply the same guard inside `FtTransferMessageData::try_from` in `engine-types/src/parameters/connector.rs` so that the zero address is rejected at the parsing layer for all callers. [3](#0-2) 

---

### Proof of Concept

**ETH freeze:**
1. User calls `ft_transfer_call` on the eth-connector NEP-141 contract, specifying Aurora as the receiver and `msg = "0000000000000000000000000000000000000000"`.
2. The eth-connector calls `ft_on_transfer` on Aurora with `predecessor = connector_account_id`.
3. `receive_base_tokens` parses `recipient = Address::zero()` from the msg.
4. `set_balance` credits the zero address with the bridged ETH amount.
5. The user's NEP-141 ETH balance is debited; the EVM-side ETH is permanently frozen at `0x0000…0000`.

**ERC-20 freeze:**
1. User calls `ft_transfer_call` on any registered NEP-141 token, specifying Aurora as the receiver and `msg = "0000000000000000000000000000000000000000"`.
2. The NEP-141 token calls `ft_on_transfer` on Aurora.
3. `receive_erc20_tokens` parses `recipient = Address::zero()`.
4. The ERC-20 `mint(address(0), amount)` call is executed inside the EVM.
5. Tokens are minted to the zero address and permanently frozen. [2](#0-1) [6](#0-5)

    pub fn receive_base_tokens(
        &mut self,
        args: &FtOnTransferArgs,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
        let amount = Wei::new_u128(args.amount.as_u128());
        let receipient = message_data.recipient;
        let balance = get_balance(&self.io, &receipient);
        let new_balance = balance
            .checked_add(amount)
            .ok_or(errors::ERR_BALANCE_OVERFLOW)?;

        set_balance(&mut self.io, &receipient, &new_balance);

        sdk::log!("Mint {amount} base tokens for: {}", receipient.encode());

        Ok(None)
    }
