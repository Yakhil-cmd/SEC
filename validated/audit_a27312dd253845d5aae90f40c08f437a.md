### Title
Silent Precompile Failure on Empty Recipient Burns ERC-20 Tokens Permanently - (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20.withdrawToNear` and `EvmErc20V2.withdrawToNear` burn the caller's tokens before invoking the `ExitToNear` precompile, but never check the return value of the assembly `call`. When an empty `recipient` is supplied, the precompile rejects the input (empty string is not a valid NEAR account ID), returns 0, and the Solidity code silently continues. The burn is committed, no exit promise is created, and the tokens are permanently destroyed.

---

### Finding Description

**Step 1 – Entry point (no access control)**

`withdrawToNear` in both `EvmErc20` and `EvmErc20V2` is `external` with no caller restriction. Any EVM address that holds a nonzero balance can call it. [1](#0-0) [2](#0-1) 

**Step 2 – Burn before precompile call, no recipient validation**

Both contracts call `_burn` unconditionally before the precompile invocation. There is no `require(recipient.length > 0)` guard.

`EvmErc20`:
```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no revert on failure
    }
}
``` [1](#0-0) 

**Step 3 – Precompile rejects empty recipient**

When `recipient.length == 0`, the input to the precompile is exactly 33 bytes: `0x01 ++ amount_b`. After stripping the flag byte, `parse_input` passes the remaining 32 bytes to the `0x1` branch, which calls `parse_recipient(&input[32..])` — an empty slice.

`parse_recipient` on an empty slice:
- `str::from_utf8(&[])` → `Ok("")`
- `"".parse::<AccountId>()` → `Err(...)` (empty string is not a valid NEAR account ID)
- Returns `Err(ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID"))` [3](#0-2) [4](#0-3) 

**Step 4 – Return value ignored, transaction succeeds**

In standard EVM semantics, a failed inner `call` returns 0 to the caller; it does not revert the outer frame. The assembly captures `res` but never uses it to revert. The outer transaction commits successfully with the burn applied and no promise log emitted. [5](#0-4) 

---

### Impact Explanation

The ERC-20 total supply decreases (tokens are burned), but no `ExitToNear` promise log is produced. The corresponding NEP-141 tokens on the NEAR side are never released. The burned amount is permanently unrecoverable — **permanent freezing/destruction of user funds**.

---

### Likelihood Explanation

- No privilege required: any token holder can trigger this.
- The call is trivially constructable: `withdrawToNear("", amount)`.
- No on-chain guard prevents it; the only protection would be an off-chain UI that rejects empty strings, which is not a security boundary.

---

### Recommendation

Add an explicit recipient length check before burning:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    require(recipient.length > 0, "ERR_EMPTY_RECIPIENT");
    _burn(_msgSender(), amount);
    // ...
    assembly {
        let res := call(...)
        if iszero(res) { revert(0, 0) }
    }
}
```

Both the recipient guard and the assembly return-value check are necessary. The return-value check alone is insufficient because the burn has already occurred; the recipient guard prevents the burn from happening at all when the input is invalid. [1](#0-0) [2](#0-1) 

---

### Proof of Concept

```solidity
// Deploy EvmErc20 with a registered NEP-141 mapping.
// Mint 1000 tokens to attacker address.
// Attacker calls:
erc20.withdrawToNear("", 1000);
// Expected (correct): revert
// Actual: transaction succeeds, totalSupply decreases by 1000,
//         no ExitToNear promise log in engine output,
//         NEP-141 balance on NEAR side unchanged.
// Result: 1000 tokens permanently destroyed.
```

Integration test assertion:
1. Record `totalSupply` before call.
2. Call `withdrawToNear("", amount)` — assert `tx.status == Success`.
3. Assert `totalSupply` decreased by `amount`.
4. Assert engine output contains **zero** promise logs with `ExitToNear` data.
5. Assert NEP-141 balance of any NEAR account is unchanged. [1](#0-0) [3](#0-2)

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
