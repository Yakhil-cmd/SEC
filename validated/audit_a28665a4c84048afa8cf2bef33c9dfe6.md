### Title
Excessive Per-Stream Message Buffer Limit in QUIC Transport Enables Memory Exhaustion by a Malicious Subnet Peer — (File: rs/p2p/quic_transport/src/lib.rs)

---

### Summary

The IC's QUIC-based P2P transport layer sets a per-stream read buffer ceiling of **128 MiB** (`MAX_MESSAGE_SIZE_BYTES`). With up to **1,000 concurrent bidirectional streams** permitted per connection (`MAX_CONCURRENT_BIDI_STREAMS`), a single authenticated subnet peer can force a victim replica to allocate up to **128 GiB** of heap memory, crashing or severely degrading the node and disrupting subnet consensus.

---

### Finding Description

In `rs/p2p/quic_transport/src/lib.rs`, the constant `MAX_MESSAGE_SIZE_BYTES` is set to 128 MiB:

```rust

/// E.g. summary blocks generated by the consensus protocol for 40 node subnet can be bigger than 5MB.
pub(crate) const MAX_MESSAGE_SIZE_BYTES: usize = 128 * 1024 * 1024;

The stream acceptor loop in `start_stream_acceptor` spawns an unbounded `JoinSet` task for every accepted stream, up to the QUIC-level concurrent stream limit:

```rust
const MAX_CONCURRENT_BIDI_STREAMS: VarInt = VarInt::from_u32(1_000);
``` [5](#0-4) 

The acceptor loop itself has no application-level cap on in-flight tasks:

```rust
let mut inflight_requests: JoinSet<Result<(), P2PError>> = tokio::task::JoinSet::new();
// ...
inflight_requests.spawn(... handle_bi_stream(...) ...);
``` [6](#0-5) 

`read_to_end` in Quinn allocates a `Vec<u8>` that grows as data arrives and automatically extends the QUIC stream receive window as the application consumes bytes. The `STREAM_RECEIVE_WINDOW` of 4 MiB is the *initial* flow-control window; it is extended by Quinn as the application reads, so the effective per-stream memory ceiling is the application-level `MAX_MESSAGE_SIZE_BYTES` (128 MiB), not the QUIC window. [7](#0-6) 

---

### Impact Explanation

A single malicious subnet node can:

1. Open 1,000 concurrent bidirectional QUIC streams to a victim replica (the QUIC-enforced maximum).
2. On each stream, slowly trickle data (respecting flow control to avoid idle timeout) up to 128 MiB.
3. Each `read_request` task allocates a `Vec<u8>` that grows to 128 MiB.
4. Total heap pressure: **1,000 × 128 MiB = 128 GiB**.

This causes OOM on the victim replica, crashing it or making it unresponsive. A crashed replica drops out of consensus participation, degrading subnet fault tolerance. If the attacker controls `f` nodes in an `f`-fault-tolerant subnet, it can simultaneously attack all honest nodes, halting the subnet entirely.

---

### Likelihood Explanation

- **Attacker role**: A single subnet node (below the consensus fault threshold). No majority corruption is required.
- **Authentication barrier**: The QUIC endpoint enforces mutual TLS authentication against the registry-verified subnet topology, so only legitimate subnet peers can connect. However, a single compromised or malicious node is sufficient.
- **Operational justification**: The comment explicitly acknowledges the large limit is intentional ("On purpose the value is big"), mirroring the exact pattern in the external report.
- **No rate limiting**: There is no per-peer stream rate limit or per-peer memory budget at the application layer.
- **Realistic**: A node operator who is malicious or whose node key is compromised can execute this without any additional privileges.

---

### Recommendation

1. **Short term**: Enforce a per-peer application-level cap on the number of concurrently buffered streams (e.g., reject or back-pressure new streams when the in-flight task count exceeds a threshold per peer). Additionally, reduce `MAX_MESSAGE_SIZE_BYTES` to the smallest value that still accommodates the largest legitimate message type (consensus summary blocks are cited as >5 MiB; a limit of 10–20 MiB would be far safer than 128 MiB).

2. **Long term**: Introduce per-peer memory accounting at the transport layer. Track total bytes buffered across all in-flight streams for a given peer and close the connection if the aggregate exceeds a configurable budget. Align `MAX_MESSAGE_SIZE_BYTES` with the actual maximum payload size enforced by the consensus and state-sync protocols that use this transport.

---

### Proof of Concept

A malicious subnet node (with a valid registry TLS certificate) executes the following:

```
for i in 1..=1000:
    open bidirectional QUIC stream to victim node
    slowly write 128 MiB of arbitrary bytes on stream i
    (do not close the stream — keep it open to hold the allocation)
```

Each open stream causes the victim's `read_request` → `recv_stream.read_to_end(134_217_728)` to allocate and grow a `Vec<u8>` up to 128 MiB. With 1,000 concurrent streams, the victim's heap grows by up to 128 GiB, triggering OOM and crashing the replica process.

The relevant code path is:

pub async fn start_stream_acceptor(
    log: ReplicaLogger,
    peer_id: NodeId,
    conn_handle: ConnectionHandle,
    metrics: QuicTransportMetrics,
    router: Router,
) {
    let mut inflight_requests: JoinSet<Result<(), P2PError>> = tokio::task::JoinSet::new();
    let mut quic_metrics_scrape = tokio::time::interval(QUIC_METRIC_SCRAPE_INTERVAL);
    // The extreme result of a slow handler is that the stream limit will be reach, hence
    // having buffered up to the stream limit number of messages/requests.
    // A better approach will be to use a router implemented as a tower service and accept
    // streams iff the router is ready. Then the actual number of buffered messages is determined
    // by the handlers instead by the underlying implementation.
    loop {
        tokio::select! {
            Some(completed_request) = inflight_requests.join_next() => {
                match completed_request {
                    Ok(res) => {
                        let _ = res.inspect_err(|err| info!(every_n_seconds => 60, log, "{:?}", err));
                    }
                    Err(err) => {
                        // Cancelling tasks is ok. Panicking tasks are not.
                        if err.is_panic() {
                            std::panic::resume_unwind(err.into_panic());
                        }
                    }
                }
            },
            bi = conn_handle.conn().accept_bi() => {
                match bi {
                    Ok((bi_tx, bi_rx)) => {
                        let send_stream = ResetStreamOnDrop::new(
                            bi_tx,
                            metrics.connection_handle_incoming_streams_total.clone()
                        );
                        inflight_requests.spawn(
                            metrics.request_task_monitor.instrument(
                                handle_bi_stream(
                                    peer_id,
                                    conn_handle.conn_id(),
                                    metrics.clone(),
                                    router.clone(),
                                    send_stream,
                                    bi_rx
                                )
                            )
                        );
