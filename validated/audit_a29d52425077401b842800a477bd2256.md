### Title
Unauthenticated `ft_on_transfer` Allows Minting ERC-20 Tokens Without Depositing NEP-141 Tokens — (`engine/src/contract_methods/connector.rs`)

### Summary

The `ft_on_transfer` entry point on the Aurora Engine is callable by any NEAR account without any validation that the call originates from a legitimate NEP-141 token contract executing the `ft_transfer_call` flow. An attacker can call it directly, supplying an arbitrary `amount` and a target EVM address, and cause the engine to mint ERC-20 mirror tokens without any corresponding NEP-141 token deposit ever occurring.

### Finding Description

`ft_on_transfer` is the NEP-141 receiver callback. In the legitimate flow a NEP-141 token contract calls `ft_transfer_call`, which atomically transfers tokens to Aurora and then calls `ft_on_transfer` on Aurora. Aurora trusts the `predecessor_account_id` to be the NEP-141 token contract and mints the corresponding ERC-20 tokens.

The implementation contains only one caller check:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
``` [1](#0-0) 

This check only distinguishes between the ETH connector (base-token path) and every other account (ERC-20 path). There is **no check** that the call is a callback from a legitimate `ft_transfer_call` invocation, and no check that the predecessor is a registered, trusted NEP-141 contract. Any NEAR account can call `ft_on_transfer` directly.

The function is exported as a public WASM entry point with no additional guard:

```rust
pub extern "C" fn ft_on_transfer() {
    let io = Runtime;
    let env = Runtime;
    let mut handler = Runtime;
    contract_methods::connector::ft_on_transfer(io, &env, &mut handler)
        ...
}
``` [2](#0-1) 

Compare this with other sensitive callbacks in the same codebase that correctly enforce `env.assert_private_call()`:

- `deploy_erc20_token_callback` — line 169: `env.assert_private_call()?;`
- `exit_to_near_precompile_callback` — line 204: `env.assert_private_call()?;`
- `withdraw_wnear_to_router` — line 32: `env.assert_private_call()?;`
- `factory_update_address_version` — line 89: `env.assert_private_call()?;` [3](#0-2) [4](#0-3) [5](#0-4) 

`ft_on_transfer` is the only token-minting callback that has no such guard.

Additionally, `deploy_erc20_token` has no caller restriction either, so any NEAR account can register itself as a NEP-141 token and obtain a deployed ERC-20 mirror on Aurora: [6](#0-5) 

The ERC-20 `mint` function is gated by `onlyAdmin` where admin is the Aurora engine address, so the engine can mint freely once `receive_erc20_tokens` is invoked: [7](#0-6) 

### Impact Explanation

**Insolvency / Direct theft of user funds.**

Attack path:

1. Attacker creates NEAR account `attacker.near`.
2. Attacker calls `deploy_erc20_token` on Aurora to register `attacker.near` as a NEP-141 token and deploy an ERC-20 mirror (no access control on this call).
3. Attacker calls `ft_on_transfer` directly on Aurora from `attacker.near`, supplying `amount = N` (any value) and `msg = <attacker_evm_address>`.
4. Aurora's `receive_erc20_tokens` mints `N` ERC-20 tokens to the attacker's EVM address — **no NEP-141 tokens were ever transferred to Aurora**.
5. If any legitimate user has deposited real `attacker.near` NEP-141 tokens into Aurora (e.g., after the attacker markets the token), the attacker can now burn the unbacked ERC-20 tokens and withdraw those real deposits, draining them.
6. Even without step 5, the attacker can use the unbacked ERC-20 tokens as collateral in any DeFi protocol deployed on Aurora to borrow real assets.

The ERC-20 supply is permanently inflated beyond the actual NEP-141 tokens held by Aurora, making the bridge insolvent for that token.

### Likelihood Explanation

The entry point is a standard public WASM export callable by any NEAR account with no deposit requirement beyond what `require_running` checks. The attacker needs only a NEAR account and gas. The `deploy_erc20_token` step is also unrestricted. No privileged access is required at any step.

### Recommendation

Add a caller-origin guard to `ft_on_transfer`. The simplest approach consistent with the rest of the codebase is to verify that the call arrives as a callback (i.e., `predecessor_account_id == current_account_id`, using `env.assert_private_call()`) **or** maintain an explicit allowlist of registered NEP-141 token accounts and reject calls from accounts not on that list.

A stricter fix is to require that `ft_on_transfer` is only reachable as a NEAR callback (promise result), not as a top-level call, mirroring the pattern already used for `exit_to_near_precompile_callback` and `deploy_erc20_token_callback`.

### Proof of Concept

```
# Step 1 – register attacker account as a NEP-141 token on Aurora
near call aurora deploy_erc20_token \
  '{"nep141": "attacker.near"}' \
  --accountId attacker.near

# Step 2 – call ft_on_transfer directly, minting 1,000,000 tokens
#           with no actual NEP-141 deposit
near call aurora ft_on_transfer \
  '{"sender_id":"attacker.near","amount":"1000000","msg":"<attacker_evm_address_hex>"}' \
  --accountId attacker.near

# Result: 1,000,000 ERC-20 mirror tokens appear in attacker's EVM wallet
# with zero NEP-141 tokens ever transferred to Aurora.
```

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
