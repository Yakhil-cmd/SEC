Audit Report

## Title
Any Registered SNS Swap Canister Can Redirect Neurons' Fund ICP Minting to Arbitrary Account via Unvalidated `sns_governance_canister_id` — (`rs/nns/governance/src/governance.rs`)

## Summary
The `settle_neurons_fund_participation` function in NNS Governance authorizes any SNS-W-registered swap canister to settle any proposal's Neurons' Fund participation, not just the swap canister associated with that specific proposal. The `sns_governance_canister_id` supplied in the `Committed` variant is used directly as the ICP ledger mint destination without being cross-checked against the SNS governance canister recorded for the target proposal. An attacker controlling a legitimately deployed SNS swap canister can redirect Neurons' Fund ICP minting to an attacker-controlled account.

## Finding Description
**Authorization check (lines 7018–7038):** `is_canister_id_valid_swap_canister_id` queries SNS-W's `list_deployed_snses` and checks only that the caller's canister ID matches *any* swap canister in the list. The code comment at line 7018 explicitly acknowledges: *"Note that a Swap could settle each other's participation."* There is no check that the caller is the swap canister associated with `request.nns_proposal_id`.

```rust
// rs/nns/governance/src/governance.rs:8215-8218
let is_swap = list_deployed_snses_response
    .instances
    .iter()
    .any(|sns| sns.swap_canister_id == Some(target_canister_id.into()));
```

**No proposal-level cross-check for `sns_governance_canister_id`:** The `ProposalData` struct does not store the deployed SNS canister IDs. The comment at line 4575–4577 in `execute_create_service_nervous_system_proposal` even acknowledges: *"subnet_id and canisters fields in deploy_new_sns_response are not used. Would probably make sense to stick them on the ProposalData."* SNS-W does expose `get_deployed_sns_by_proposal_id` which could be used to cross-check, but `settle_neurons_fund_participation` never calls it.

**Mint destination is caller-supplied (lines 7249–7273 and 7495–7496):** The `sns_governance_canister_id` extracted from the `Committed` variant is passed directly to `mint_to_sns_governance`, which uses it as `AccountIdentifier::new(sns_governance_canister_id, None)` — the ICP ledger mint destination — with no validation against any on-chain record of the correct SNS governance canister for that proposal.

**Exploit flow:**
1. Attacker passes an NNS `CreateServiceNervousSystem` proposal, obtaining a legitimate SNS swap canister ID recognized by SNS-W.
2. Before the victim SNS's legitimate swap canister calls `settle_neurons_fund_participation`, the attacker's swap canister calls it first, targeting the victim's `nns_proposal_id` with `Committed { sns_governance_canister_id = attacker_principal, total_direct_participation_icp_e8s = max_direct, ... }`.
3. Governance accepts (caller is a valid swap canister per SNS-W), looks up the victim proposal's `initial_neurons_fund_participation`, computes the NF contribution, and calls `mint_to_sns_governance(attacker_principal, amount_icp_e8s)`.
4. ICP ledger mints to `AccountIdentifier::new(attacker_principal, None)`.
5. The idempotency check at line 7166 prevents the legitimate swap canister from re-settling, locking in the theft.

## Impact Explanation
This constitutes illegal minting of ICP to an attacker-controlled account. The Neurons' Fund can contribute up to 10% of total NF maturity to a single SNS swap; at current ICP valuations this can exceed $1M. This matches the Critical/High allowed impact: *"Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant ICP/Cycles or in-scope chain-key/ledger assets"* and *"Unauthorized access to governance assets or canister-controlled funds where exploitation requires meaningful per-target work."*

## Likelihood Explanation
The precondition — obtaining a legitimately deployed SNS swap canister ID — requires passing an NNS governance vote, which is non-trivial but achievable (it is the standard SNS launch process). The attack must be timed to front-run the legitimate swap canister's `finalize_swap` → `settle_neurons_fund_participation` call. On the IC, inter-canister call ordering is not guaranteed, making front-running feasible. The attack is one-shot per proposal due to the idempotency lock.

## Recommendation
1. In `settle_neurons_fund_participation`, after the SNS-W authorization check, call `get_deployed_sns_by_proposal_id(nns_proposal_id)` on SNS-W to retrieve the canonical SNS canister IDs for that proposal, and verify that (a) the caller matches the recorded swap canister ID, and (b) the `sns_governance_canister_id` from the request matches the recorded governance canister ID.
2. Alternatively, store the deployed SNS canister IDs on `ProposalData` at execution time (as the comment at line 4575 already suggests) and use them for validation in `settle_neurons_fund_participation`.

## Proof of Concept
```rust
// Attacker's swap canister (legitimately registered with SNS-W) sends:
settle_neurons_fund_participation(SettleNeuronsFundParticipationRequest {
    nns_proposal_id: Some(victim_proposal_id), // any open CSNS proposal
    result: Some(Result::Committed(Committed {
        sns_governance_canister_id: Some(attacker_principal), // arbitrary
        total_direct_participation_icp_e8s: Some(max_direct_participation),
        total_neurons_fund_participation_icp_e8s: Some(0),
    })),
})
// Governance:
// 1. Verifies caller is any SNS-W swap canister — PASSES (attacker has legitimate SNS)
// 2. Looks up victim proposal's initial_neurons_fund_participation — found
// 3. Computes NF amount from victim proposal data
// 4. Calls mint_to_sns_governance(attacker_principal, amount_icp_e8s)
// 5. ICP ledger mints to AccountIdentifier::new(attacker_principal, None)
// 6. Idempotency lock set — legitimate swap canister cannot re-settle
```

A deterministic integration test using PocketIC can reproduce this by: deploying two SNSes (attacker and victim), opening both swaps, committing the victim swap with direct participants, then having the attacker's swap canister call `settle_neurons_fund_participation` with the victim's proposal ID and an attacker-controlled `sns_governance_canister_id` before the victim's swap canister finalizes.
