### Title
Unchecked Return Data in `notify_l2_asset_tracker` Allows Silent Accounting Bypass on Empty Address - (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The `notify_l2_asset_tracker` function calls `L2_ASSET_TRACKER_ADDRESS` via `run_single_interaction` and only checks whether the call reverted (`asset_tracker_result.failed()`), but never inspects the return data. By EVM semantics, a call to an address with no deployed code always returns `CallResult::Successful` with empty returndata. The code itself explicitly acknowledges this gap. Any L1→L2 priority transaction submitted while `L2_ASSET_TRACKER_ADDRESS` has no deployed contract will silently skip the asset-tracker notification and proceed to mint tokens, leaving the tracker's accounting permanently inconsistent.

---

### Finding Description

`notify_l2_asset_tracker` is responsible for notifying the L2 asset tracker about every base-token movement (value mint, operator fee, refund) that originates from an L1→L2 transaction. It does so by calling `handleFinalizeBaseTokenBridgingOnL2(uint256,uint256)` on `L2_ASSET_TRACKER_ADDRESS` through `run_single_interaction`. [1](#0-0) 

After the call, the function evaluates only `asset_tracker_result.failed()`:

```rust
let failed = resources.with_infinite_ergs(|inf_ergs| {
    let CompletedExecution { resources_returned, result: asset_tracker_result } =
        BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
            ...
            true, // should_make_frame
            ...
        )?;
    *inf_ergs = resources_returned;
    Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
})?;

if failed {
    return Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted").into());
}
```

The code comment directly acknowledges the gap:

> *"If no contract is deployed at L2AssetTracker, the call succeeds silently (a call to an empty address returns success with no returndata in EVM). However, we are certain that L2AssetTracker is available after the upgrade."* [2](#0-1) 

This is structurally identical to the external report's root cause: the system assumes that a non-reverting call means the operation succeeded, without verifying the return data. In EVM, a call to an empty account always returns `CallResult::Successful` with zero returndata — indistinguishable from a successful call to a live contract.

`run_single_interaction` itself is correct: it faithfully returns the `CompletedExecution` result. [3](#0-2) 

The caller (`notify_l2_asset_tracker`) is where the return data goes unchecked.

`mint_base_token` calls `notify_l2_asset_tracker` before `transfer_from_treasury`, meaning tokens are transferred from the treasury even when the tracker notification silently did nothing: [4](#0-3) 

`mint_base_token` is invoked for every token movement in `process_l1_transaction`: [5](#0-4) 

---

### Impact Explanation

If `L2_ASSET_TRACKER_ADDRESS` has no deployed contract at the time an L1→L2 transaction is processed:

1. `notify_l2_asset_tracker` calls the address; EVM returns `CallResult::Successful` with empty returndata.
2. `asset_tracker_result.failed()` is `false`; no error is raised.
3. `transfer_from_treasury` executes and mints tokens to the recipient.
4. The L2 asset tracker's internal accounting is never updated.

The result is a permanent divergence between the actual token supply on L2 and the asset tracker's recorded state. This breaks the invariant that the tracker's `totalSupply()` accurately reflects bridged tokens — an invariant the tracker itself relies on (the comment references `_needToForceSetAssetMigrationOnL2` using `totalSupply() == 0`). Repeated occurrences compound the discrepancy.

---

### Likelihood Explanation

The vulnerable window is any period during which `L2_ASSET_TRACKER_ADDRESS` has no deployed contract — most concretely, the interval between chain genesis and the upgrade that deploys the tracker. L1→L2 priority transactions can be submitted by any unprivileged user at any time, including during this window. The attacker does not need to manipulate the contract; they only need to submit a standard L1→L2 transaction while the system is in this state.

---

### Recommendation

After calling `L2_ASSET_TRACKER_ADDRESS`, verify that the return data is non-empty (or contains the expected ABI-encoded success value). A minimal fix:

```rust
// After run_single_interaction returns:
if asset_tracker_result.failed() || asset_tracker_result.return_values().returndata.is_empty() {
    return Err(internal_error!("L2AssetTracker call failed or returned no data").into());
}
```

Alternatively, add a pre-call check that `L2_ASSET_TRACKER_ADDRESS` has deployed code before invoking `run_single_interaction`, and treat the absence of code as a fatal error rather than a silent success.

---

### Proof of Concept

1. Deploy a ZKsync OS chain **without** deploying a contract at `L2_ASSET_TRACKER_ADDRESS` (`0x...` as configured in `system_hooks::addresses_constants::L2_ASSET_TRACKER_ADDRESS`).
2. Submit an L1→L2 priority transaction with `to_mint > 0` from any unprivileged address.
3. `process_l1_transaction` calls `mint_base_token` → `notify_l2_asset_tracker`.
4. `run_single_interaction` targets the empty address; EVM returns `CallResult::Successful { return_values: ReturnValues::empty() }`.
5. `asset_tracker_result.failed()` evaluates to `false`; `notify_l2_asset_tracker` returns `Ok(())`.
6. `transfer_from_treasury` executes: tokens are minted to the recipient.
7. The L2 asset tracker has no record of this mint. Its `totalSupply()` and per-chain accounting are permanently incorrect.
