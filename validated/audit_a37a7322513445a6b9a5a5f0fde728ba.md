Audit Report

## Title
Unsolicited `Pong` Unconditionally Disconnects Any Post-Handshake Bitcoin Adapter Connection — (`rs/bitcoin/adapter/src/connectionmanager.rs`)

## Summary
`process_pong_message` sets `valid_pong = false` when `conn.ping_state()` is `PingState::Idle` and unconditionally calls `conn.disconnect()`. Because `PingState::Idle` is both the initial state and the normal resting state after every successful ping/pong exchange, any Bitcoin peer that has completed the version handshake can tear down its own connection — and, by reconnecting repeatedly, hold the Bitcoin adapter at zero active connections. This is a peer-triggerable availability attack against the IC's Bitcoin integration (ckBTC / Chain Fusion).

## Finding Description
In `connectionmanager.rs`, `process_pong_message` (L517–531) matches on `conn.ping_state()`:

```rust
let valid_pong = match conn.ping_state() {
    PingState::ExpectingPong { ping_sent_at: _, nonce: ping_nonce } => *ping_nonce == nonce,
    PingState::Idle { last_pong_at: _ } => false,   // ← always false
};
if valid_pong {
    conn.idle();
} else {
    conn.disconnect();   // ← unconditional disconnect
};
``` [1](#0-0) 

`conn.disconnect()` sets `ConnectionState::NodeDisconnected` and aborts the async task: [2](#0-1) 

`reap_disconnected()` then removes the entry from `self.connections` and calls `address_book.remove_from_active()`: [3](#0-2) 

The message dispatcher routes `Pong` directly to `process_pong_message` with no prior connection-state guard: [4](#0-3) 

`PingState::Idle` is the state set at construction and after every valid pong — it is the normal resting state of every live post-handshake connection: [5](#0-4) [6](#0-5) 

There is no guard in `process_pong_message` or `get_connection` that checks `ConnectionState` before acting on the ping state. Any nonce value triggers the disconnect because the `Idle` arm returns `false` unconditionally regardless of the nonce.

## Impact Explanation
This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

An attacker who can operate or compromise any Bitcoin node reachable by the adapter can: (1) complete the public, unauthenticated Bitcoin version handshake, (2) immediately send one `Pong` message, (3) have the adapter disconnect and delist the peer. By repeating this across all peers the adapter attempts to connect to, the attacker can hold the Bitcoin adapter at zero active connections, preventing the IC's Bitcoin integration from relaying Bitcoin data to the subnet — a concrete, sustained availability impact on ckBTC and Chain Fusion.

## Likelihood Explanation
The precondition is low-barrier: the Bitcoin version handshake is a public, unauthenticated protocol. Any party operating a Bitcoin node (or performing a man-in-the-middle at the TCP layer) can complete it. No privileged access, key material, or subnet-majority corruption is required. The attack is repeatable: after each disconnect the adapter will attempt to reconnect, and the attacker can trigger the same path again immediately.

## Recommendation
In `process_pong_message`, treat an unsolicited `Pong` (i.e., `PingState::Idle`) as a no-op rather than a disconnect trigger:

```rust
PingState::Idle { last_pong_at: _ } => {
    // Unsolicited pong — ignore silently.
    return Ok(());
}
```

If penalizing misbehaving peers is desired, use `conn.discard()` (which blacklists the address) rather than `conn.disconnect()` (which merely removes it from active and allows reconnection). The primary fix is to not disconnect on an unsolicited `Pong`. [7](#0-6) 

## Proof of Concept
State-machine unit test (no network required):

1. Construct a `ConnectionManager` and inject a connection in `ConnectionState::HandshakeComplete` with `PingState::Idle`.
2. Call `process_bitcoin_network_message(addr, &NetworkMessage::Pong(0u64))` (any nonce).
3. Assert `conn.state()` is `ConnectionState::NodeDisconnected`.
4. Call `reap_disconnected()`.
5. Assert `self.connections` is empty and `address_book` no longer lists the peer as active.

The single root-cause line is `PingState::Idle { last_pong_at: _ } => false` at L522 of `connectionmanager.rs`. [8](#0-7)

    fn reap_disconnected(&mut self) {
        let mut disconnects = vec![];
        for (addr, conn) in self.connections.iter() {
            match conn.state() {
                ConnectionState::AdapterDiscarded => {
                    self.address_book.discard(conn.address_entry());
                }
                ConnectionState::NodeDisconnected => {
                    self.address_book.remove_from_active(conn.address_entry());
                }
                _ => {}
            }

            if conn.is_disconnected() {
                disconnects.push(*addr);
            }
        }

        for addr in disconnects {
            self.connections.remove(&addr);
        }
