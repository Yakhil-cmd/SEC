Audit Report

## Title
Unbounded Unvalidated Pool Accumulation via Deferred Remote DKG Dealings — (File: `rs/consensus/dkg/src/lib.rs`)

## Summary
In `validate_dealings_for_dealer`, remote DKG dealings whose `NiDkgId` has no matching config are silently deferred by returning `Mutations::new()` — leaving them in the unvalidated pool indefinitely until the next DKG interval purge. Because the DKG artifact channel is registered with `SLOT_TABLE_NO_LIMIT` and `DkgPoolImpl::insert()` has no size guard, a single Byzantine subnet member can flood the unvalidated pool with arbitrarily many distinct remote DKG dealings within one DKG interval, causing unbounded memory growth on honest replicas and potential OOM-induced subnet liveness failure.

## Finding Description

**Deferred path — no removal:**
In `validate_dealings_for_dealer` at `rs/consensus/dkg/src/lib.rs` lines 207–211, when `configs.get(message_dkg_id)` returns `None` and `message_dkg_id.target_subnet.is_remote()` is true, the function returns `Mutations::new()` immediately — no `HandleInvalid`, no `RemoveFromUnvalidated`. The dealing stays in the unvalidated pool. [1](#0-0) 

Crucially, this early return occurs **before** `crypto_validate_dealing` is ever called (line 240), so no signature or dealing verification is performed on the deferred path. The attacker does not need valid BLS12-381 crypto material. [2](#0-1) 

**Purge only fires at DKG interval boundary:**
The only removal mechanism is the purge at lines 302–304, which fires only when `start_height > dkg_pool.get_current_start_height()` — i.e., at the next DKG summary block, roughly every ~500 blocks. [3](#0-2) 

**No size guard in `DkgPoolImpl::insert()`:**
`DkgPoolImpl::insert()` unconditionally inserts into a `PoolSection` (`BTreeMap`) with no capacity check. [4](#0-3) 

**`DkgBouncer` only filters by height:**
The bouncer accepts any dealing at the current `start_height` regardless of `target_subnet` type, so all crafted messages pass. [5](#0-4) 

**`SLOT_TABLE_NO_LIMIT` for DKG:**
The DKG artifact channel is registered with `SLOT_TABLE_NO_LIMIT = usize::MAX`, meaning there is no per-peer cap on how many distinct DKG artifact IDs a peer can advertise via the P2P slot table. [6](#0-5) [7](#0-6) 

The slot table enforcement in `handle_slot_update_receive` only drops new entries when `peer_slot_table_len >= self.slot_limit`; with `usize::MAX` this check never triggers. [8](#0-7) 

**Each distinct `NiDkgTargetId` produces a unique pool entry:**
`DkgMessageId` is keyed by `crypto_hash(message)`, so each message with a distinct `NiDkgTargetId` (a 32-byte field) produces a unique key and a unique pool entry. [9](#0-8) 

`NiDkgTargetId` is a plain 32-byte array with no structural constraints. [10](#0-9) 

**Test explicitly confirms the deferred accumulation behavior:**
The existing test `test_remote_dealing_validation_is_deferred_until_context_exists` confirms that deferred remote dealings remain in the unvalidated pool (`assert_eq!(dkg_pool.get_unvalidated().count(), 2)`) and are only cleared at the next DKG interval purge. [11](#0-10) 

**Attack flow:**
1. Byzantine node crafts N messages with `target_subnet = NiDkgTargetSubnet::Remote(distinct_id_i)` for i=1..N, each with `start_block_height` = current DKG start height and arbitrary (non-verified) dealing bytes.
2. Advertises all N artifacts via the P2P slot table (no per-peer limit for DKG).
3. Each message passes the `DkgBouncer` height check and is inserted into the unvalidated pool.
4. `validate_dealings_for_dealer` returns `Mutations::new()` for each — no removal occurs.
5. Pool grows without bound until the next DKG interval purge (~500 blocks).

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

Each `NiDkgDealing` contains BLS12-381 crypto material (public coefficients, ciphertexts, ZK proofs) amounting to several KB per message for a typical subnet size. Injecting millions of such messages within one DKG interval can exhaust heap memory on honest replicas, causing OOM kills and liveness failure of the DKG pipeline and subnet stall. The attack is not raw volumetric DDoS — it exploits a specific protocol logic gap (deferred validation without removal) that bypasses all existing guards.

## Likelihood Explanation

- **Attacker prerequisite**: Must be a registered subnet member (Byzantine node below fault threshold). IC P2P uses mutual TLS with node certificates, so only registered subnet members can connect. A single compromised or malicious node suffices.
- **Ease of exploit**: Trivial — vary the 32-byte `NiDkgTargetId` field. No valid crypto material is required since the deferred path returns before `crypto_validate_dealing`.
- **No rate limiting**: `SLOT_TABLE_NO_LIMIT` for DKG means no per-peer cap exists at the P2P layer.
- **Bounded window**: The pool is purged at each DKG interval (~500 blocks, ~500 seconds at 1s block time). This is a partial mitigation but does not prevent within-interval memory exhaustion.

## Recommendation

1. **Add a pool size cap** in `DkgPoolImpl::insert()` or in `on_state_change` before processing unvalidated dealings, rejecting new entries once a configurable limit is reached.
2. **Apply a slot table limit for DKG** analogous to `SLOT_TABLE_LIMIT_INGRESS` (50,000) rather than `SLOT_TABLE_NO_LIMIT`.
3. **Cap deferred remote dealings per `(dealer_id, start_height)` tuple** — track the number of distinct unknown remote target IDs and reject beyond a small bound (e.g., the maximum number of plausible concurrent `SetupInitialDKG` / `ReshareChainKey` contexts).
4. **Consider issuing `HandleInvalid`** for remote dealings whose `target_subnet` remote ID is not plausibly associated with any pending context visible in the certified state, rather than deferring unconditionally.

## Proof of Concept

The existing test `test_remote_dealing_validation_is_deferred_until_context_exists` already demonstrates the core behavior. A minimal extension to prove unbounded accumulation:

```rust
// In a test using DkgPoolImpl and DkgImpl with CryptoReturningOk:
let start_height = dkg_pool.get_current_start_height();
for i in 0u64..10_000 {
    let mut id_bytes = [0u8; 32];
    id_bytes[..8].copy_from_slice(&i.to_le_bytes());
    let target_id = NiDkgTargetId::new(id_bytes); // distinct IDs
    let dkg_id = NiDkgId {
        start_block_height: start_height,
        dealer_subnet: subnet_id,
        dkg_tag: NiDkgTag::LowThreshold,
        target_subnet: NiDkgTargetSubnet::Remote(target_id), // no matching config
    };
    let msg = create_dealing(i as u8, dkg_id);
    dkg_pool.insert(UnvalidatedArtifact {
        message: msg,
        peer_id: attacker_node_id,
        timestamp: UNIX_EPOCH,
    });
}
// Each on_state_change call returns empty mutations — nothing is removed
let changeset = dkg_impl.on_state_change(&dkg_pool);
assert!(changeset.iter().all(|a| !matches!(a, ChangeAction::HandleInvalid(..)
    | ChangeAction::RemoveFromUnvalidated(..))));
assert_eq!(dkg_pool.get_unvalidated().count(), 10_000); // unbounded accumulation confirmed
```

This test can be run locally without any network access, using the existing `CryptoReturningOk` mock and `create_dealing` test utility already present in the codebase. [12](#0-11)
