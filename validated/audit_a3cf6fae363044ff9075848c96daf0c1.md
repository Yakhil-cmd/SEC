### Title
L1→L2 Transaction `gas_per_pubdata_limit` Can Be Set to Zero, Bypassing Pubdata Cost Accounting — (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

### Summary
The ZKsync OS bootloader processes L1→L2 priority transactions using a user-supplied `gas_per_pubdata_limit` field without validating it is non-zero. When set to zero, the derived `native_per_pubdata` becomes zero, causing all post-execution pubdata resource checks to pass trivially. The transaction can then generate pubdata (up to the block limit) without paying for it, shifting the L1 publication cost entirely to the operator.

### Finding Description
In `process_l1_transaction.rs`, the `gas_per_pubdata` value is read directly from the transaction struct with no minimum bound check:

```rust
let gas_per_pubdata = transaction.gas_per_pubdata_limit.read();
```

This value is then used in `prepare_and_check_resources` to derive `native_per_pubdata`:

```rust
let native_per_pubdata = (gas_per_pubdata as u64)
    .checked_mul(native_per_gas)
    .unwrap_or_else(|| { ... u64::MAX });
```

If `gas_per_pubdata = 0`, then `native_per_pubdata = 0` regardless of `native_per_gas`. This zero value propagates into `get_resources_to_charge_for_pubdata` in `gas_helpers.rs`:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)  // 0 * anything = 0
    .ok_or(out_of_native_resources!())?;
```

The resulting `resources_for_pubdata` is zero, so `check_enough_resources_for_pubdata` always returns `enough = true`. The post-execution pubdata check at line 713–722 of `process_l1_transaction.rs` never reverts the transaction regardless of how much pubdata it generates.

The code comment acknowledges the design reliance on L1 validation:
> *"Note that the 'validation errors' are practically unreachable, as gas_limit, gas_price and gas_per_pubdata are either checked or set by the L1 contracts."*

However, unlike the balance invariant (which has an explicit double-check comment), there is no corresponding defensive check for `gas_per_pubdata > 0` inside the ZKsync OS bootloader itself.

### Impact Explanation
An L1→L2 transaction with `gas_per_pubdata_limit = 0` executes with zero pubdata cost. Because L1→L2 transactions are priority-queue transactions that the sequencer **cannot reject**, the operator is forced to publish the resulting state-diff pubdata to L1 at their own expense with no compensation. The attacker's cost is only the L1 gas required to enqueue the priority transaction. If the pubdata generated per transaction exceeds the L1 gas cost to submit it, the attack is economically profitable and can be repeated to drain operator funds over time. The impact is classified as a **resource accounting bug** leading to operator financial loss.

### Likelihood Explanation
Exploitability depends on whether the L1 bridge contracts enforce a minimum `gas_per_pubdata_limit > 0` before enqueuing the priority transaction. If L1 validation is present and correct, the attack is blocked before reaching ZKsync OS. However, the ZKsync OS bootloader provides no independent defense-in-depth check, meaning any gap or future regression in L1 validation directly exposes this path. The attacker entry point — submitting an L1→L2 transaction — requires no privileged access.

### Recommendation
Add an explicit minimum validation for `gas_per_pubdata_limit` inside `prepare_and_check_resources` in `process_l1_transaction.rs`, analogous to the existing `native_price != 0` guard in the L2 ZK validation path (`validation_impl.rs` line 122–123). A zero `gas_per_pubdata_limit` should either be rejected (if L1 contracts can be updated to enforce this) or mapped to a protocol-defined minimum floor value, preventing `native_per_pubdata` from collapsing to zero.

### Proof of Concept
1. Construct an L1→L2 (`0x7f`) ABI-encoded transaction with `gas_per_pubdata_limit = 0`, a non-zero `gas_limit`, and calldata that writes to multiple storage slots (generating pubdata).
2. Submit the transaction to the L1 priority queue (assuming L1 contracts do not reject `gas_per_pubdata_limit = 0`).
3. The ZKsync OS bootloader reads `gas_per_pubdata = 0` at [1](#0-0) , computes `native_per_pubdata = 0` at [2](#0-1) , and the pubdata resource check at [3](#0-2)  returns zero cost.
4. The transaction succeeds, all storage writes are committed, and the operator publishes the resulting pubdata to L1 without receiving any compensation from the transaction sender.
