Audit Report

## Title
Missing Cryptographic Validity Check in `verify_encrypted_key_share_internal` Allows Byzantine Node to Force O(n·pairing) vetKD Fallback — (`rs/crypto/src/vetkd/mod.rs`)

## Summary

`verify_encrypted_key_share_internal` admits encrypted key shares that pass only a basic node signature check, with no pairing-equation validation. A single Byzantine subnet node below the reconstruction threshold can submit a structurally valid but cryptographically invalid share that passes pool admission, causing `combine_all` to fail and triggering the `combine_valid_shares` fallback on every vetKD request. The system remains correct but vetKD throughput is persistently degraded.

## Finding Description

**Step 1 — Share admission is signature-only.**

`verify_encrypted_key_share_internal` at `rs/crypto/src/vetkd/mod.rs` lines 292–301 performs exactly one check:

```rust
BasicSigVerifierInternal::verify_basic_sig(
    csp_signer, registry, &signature,
    &key_share.encrypted_key_share,
    signer, registry_version_from_store,
)
``` [1](#0-0) 

There is no call to `EncryptedKeyShare::is_valid`, no pairing check, and no verification that the G1/G2 points satisfy the vetKD relation. A Byzantine node can sign arbitrary valid-curve-point bytes with its legitimate node key and the share is admitted to the validated pool.

**Step 2 — VetKD threshold-stop optimization is unconditionally disabled.**

`inputs_already_have_enough_shares` in `rs/consensus/idkg/src/signer.rs` returns `false` for all VetKD inputs:

```rust
ThresholdSigInputs::VetKd(_inputs) => return false,
``` [2](#0-1) 

This means all n shares from all nodes — including the Byzantine node's crafted share — are admitted to the validated pool and passed to `combine_encrypted_key_shares`.

**Step 3 — `combine_all` fails on one bad share; fallback is O(n·pairing).**

`combine_all` interpolates all shares and checks the combined result with a single `is_valid` call. One cryptographically invalid share corrupts the interpolation, so `is_valid` fails and `InvalidShares` is returned: [3](#0-2) 

The caller in `combine_encrypted_key_shares_internal` then falls back to `combine_valid_shares`: [4](#0-3) 

`combine_valid_shares` calls `EncryptedKeyShare::is_valid` on each share individually. Each call performs two `Gt::multipairing` operations (the first checks `e(c1,g2)==e(g1,c2)`, the second checks `e(c3,g2)==e(tpk,c2)*e(msg,dpki)`): [5](#0-4) 

`combine_valid_shares` does stop early once `reconstruction_threshold` valid shares are found, so the worst-case is checking all n shares (bad share last in BTreeMap order). Even in the average case, per-share pairing checks are substantially more expensive than the 3-pairing happy path of `combine_all`.

**Step 4 — Confirmed by existing integration test.**

The test `should_succeed_if_reconstruction_threshold_many_shares_are_valid` in `rs/crypto/tests/vetkd.rs` explicitly demonstrates that swapping c1/c3 bytes (structurally valid, cryptographically invalid) causes `combine_all` to fail and the fallback to be triggered: [6](#0-5) 

## Impact Explanation

This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

A single Byzantine node persistently forces the expensive `combine_valid_shares` fallback on every vetKD key derivation request for the lifetime of that node's participation. BLS12-381 pairings are among the most expensive operations in the crypto stack. For a 34-node subnet, the worst-case overhead is approximately 23× the expected pairing cost per combine operation. Sustained triggering degrades vetKD throughput, causing queued requests to back up and response latency to grow. The attack is protocol-level (not volumetric), persistent, and requires no special tooling.

## Likelihood Explanation

Any single compromised subnet node below the reconstruction threshold can execute this attack. The attacker needs only to: generate three random valid BLS12-381 curve points, serialize them as an `EncryptedKeyShare`, sign the bytes with the node's legitimate signing key, and broadcast. The share passes every admission check. No special privileges, no victim mistakes, no external dependencies. The attack is persistent across all future vetKD requests until the node is removed via governance.

## Recommendation

Add a cryptographic validity check inside `verify_encrypted_key_share_internal`. After the basic signature passes, deserialize the share and call `EncryptedKeyShare::is_valid` against the node's individual public key (derivable from the transcript data already in the store via `lazily_calculated_public_key_from_store`). Shares that fail this check should be rejected with a new `VetKdKeyShareVerificationError::InvalidShareContent` variant and removed from the unvalidated pool, preventing them from ever reaching `combine_encrypted_key_shares`. This moves the O(pairing) cost to the per-share validation step (which is already parallelized across nodes) and eliminates the O(n·pairing) fallback trigger entirely.

## Proof of Concept

The existing integration test at `rs/crypto/tests/vetkd.rs:370–403` already constitutes a reproducible proof. To demonstrate the attack vector specifically:

1. Byzantine node B is a legitimate subnet member (below threshold).
2. B generates three random valid G1/G2 points: `c1 ∈ G1`, `c2 ∈ G2`, `c3 ∈ G1`.
3. B serializes them as an `EncryptedKeyShare` (structurally valid, passes `EncryptedKeyShare::deserialize`).
4. B signs the raw bytes with its node signing key → `node_signature` is valid.
5. B broadcasts `VetKdKeyShare { signer_id: B, share: { encrypted_key_share, node_signature } }`.
6. Receiving replicas call `verify_encrypted_key_share` → basic sig check passes → share admitted to validated pool.
7. `combine_encrypted_key_shares_internal` is called with n shares including B's.
8. `combine_all` interpolates all shares → combined result fails `is_valid` → returns `InvalidShares`.
9. Fallback: `combine_valid_shares` iterates shares, calls `is_valid` per share (2 multi-pairings each).
10. Result: correct key is produced, but at significantly higher pairing cost.
11. Repeat for every vetKD request → sustained throughput degradation.

To reproduce as a unit test: extend `should_succeed_if_reconstruction_threshold_many_shares_are_valid` to corrupt exactly one share (not `to_corrupt` shares) and verify the fallback log message still appears, confirming a single bad share is sufficient to trigger the expensive path.

fn check_validity(
    c1: &G1Affine,
    c2: &G2Affine,
    c3: &G1Affine,
    tpk: &TransportPublicKey,
    verification_pk: &G2Affine,
    msg: &G1Affine,
) -> bool {
    let neg_g2_g = G2Prepared::neg_generator();
    let c2_prepared = G2Prepared::from(c2);

    // check e(c1,g2) == e(g1, c2)
    let c1_c2 = Gt::multipairing(&[(c1, neg_g2_g), (G1Affine::generator(), &c2_prepared)]);
    if !c1_c2.is_identity() {
        return false;
    }

    let verification_key_prepared = G2Prepared::from(verification_pk);

    // check e(c3, g2) == e(tpk, c2) * e(msg, dpki)
    let c3_c2_msg = Gt::multipairing(&[
        (c3, neg_g2_g),
        (tpk.point(), &c2_prepared),
        (msg, &verification_key_prepared),
    ]);

    if !c3_c2_msg.is_identity() {
        return false;
    }

    true
}
