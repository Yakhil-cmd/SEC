### Title
Unbounded `while` Loop in `Hashchain::move_to_block` Causes Permanent Service Blockage - (File: engine-hashchain/src/hashchain.rs)

### Summary
`Hashchain::move_to_block` contains an unbounded `while` loop that iterates once per skipped NEAR block. This function is called on every Aurora transaction via `load_hashchain`. If the gap between the stored hashchain block height and the current NEAR block height is large enough, the loop exhausts the 300 Tgas NEAR transaction gas limit. Because the hashchain state is only persisted on success, the stored height never advances, causing every subsequent transaction to fail identically — permanently blocking the service and freezing all user funds.

### Finding Description

`Hashchain::move_to_block` in `engine-hashchain/src/hashchain.rs` iterates once per block height between the stored `current_block_height` and the target `next_block_height`, performing a keccak hash computation per iteration: [1](#0-0) 

This function is called unconditionally from `load_hashchain` in `engine/src/hashchain.rs` whenever the current NEAR block height exceeds the stored hashchain height: [2](#0-1) 

`load_hashchain` is invoked at the start of every call wrapped by `with_hashchain` or `with_logs_hashchain`: [3](#0-2) [4](#0-3) 

These wrappers cover virtually every production contract method (e.g., `submit`, `deploy_erc20_token`, `register_relayer`, etc.).

The hashchain state is only written to storage **after** the loop completes and the main function succeeds: [5](#0-4) 

If the loop exhausts gas before completing, no state is written. The stored `current_block_height` remains at its old value. Every subsequent transaction will re-enter the same loop with the same gap and also fail.

### Impact Explanation

If Aurora experiences a period of inactivity (no transactions submitted for N NEAR blocks), the first transaction after that period triggers a loop of N iterations. NEAR enforces a hard 300 Tgas limit per transaction. Once the gap is large enough to exhaust this budget, the transaction fails. Because the hashchain state is never updated on failure, all subsequent transactions also fail. Users cannot withdraw funds, transfer tokens, or interact with any Aurora contract — a permanent fund freeze.

### Likelihood Explanation

NEAR produces approximately one block per second. A gap of several hundred thousand blocks (a few days of inactivity) is sufficient to exhaust the gas budget given the per-iteration keccak cost. This can occur naturally during:
- Planned maintenance windows where the contract is paused
- Periods of low network activity with no relayer submissions
- Any scenario where the relayer goes offline

No privileged access is required to trigger the failure — any unprivileged user submitting the first transaction after a long gap will hit the gas limit and trigger the permanent blockage.

### Recommendation

Replace the per-block iteration with a direct computation. Since empty blocks (no transactions) all produce the same hash structure, the hashchain for N consecutive empty blocks can be computed in O(1) using a closed-form or cached approach, rather than iterating N times. Alternatively, cap the number of iterations per call and defer the remainder to a dedicated catch-up method, or store the hashchain height lazily without iterating over skipped blocks.

### Proof of Concept

1. Deploy Aurora Engine with hashchain enabled via `start_hashchain` at block height H.
2. Submit no transactions for N NEAR blocks (N large enough to exhaust 300 Tgas in the loop).
3. Submit any valid Aurora transaction (e.g., a simple ETH transfer via `submit`).
4. `with_logs_hashchain` calls `load_hashchain(H + N)`.
5. `load_hashchain` calls `hashchain.move_to_block(H + N)`.
6. The `while` loop runs N iterations, each calling `compute_block_hashchain` (keccak).
7. Gas is exhausted; the transaction fails; no state is written.
8. All subsequent transactions repeat steps 4–7 and also fail.
9. All user funds in Aurora are permanently frozen. [6](#0-5) [7](#0-6) [8](#0-7)

    pub fn move_to_block(
        &mut self,
        next_block_height: u64,
    ) -> Result<(), BlockchainHashchainError> {
        if next_block_height <= self.current_block_height {
            return Err(BlockchainHashchainError::BlockHeightIncorrect);
        }

        while self.current_block_height < next_block_height {
            self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
                &self.chain_id,
                self.contract_account_id.as_bytes(),
                self.current_block_height,
                self.previous_block_hashchain,
            );

            self.block_hashchain_computer.clear_txs();
            self.current_block_height += 1;
        }

        Ok(())
    }

fn load_hashchain<I: IO>(io: &I, block_height: u64) -> Result<Option<Hashchain>, ContractError> {
    let mut maybe_hashchain = read_current_hashchain(io)?;
    if let Some(hashchain) = maybe_hashchain.as_mut()
        && block_height > hashchain.get_current_block_height()
    {
        hashchain.move_to_block(block_height)?;
    }
    Ok(maybe_hashchain)
}

pub fn start_hashchain<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    let mut state = state::get_state(&io)?;
    require_paused(&state)?;
    require_key_manager_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let args = StartHashchainArgs::try_from_slice(&input).map_err(|_| errors::ERR_SERIALIZE)?;
    let block_height = env.block_height();

    // Starting hashchain must be for an earlier block
    if block_height < args.block_height {
        return Err(errors::ERR_ARGS.into());
    }

    let mut hashchain = Hashchain::new(
        state.chain_id,
        env.current_account_id(),
        args.block_height + 1,
        args.block_hashchain,
    );

    if hashchain.get_current_block_height() < block_height {
        hashchain.move_to_block(block_height)?;
    }
