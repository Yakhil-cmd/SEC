Audit Report

## Title
Unbounded `JoinSet::spawn` via unchecked `ingress_messages` count in `TryFrom<pb::StrippedBlockProposal>` — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs`)

## Summary

`TryFrom<pb::StrippedBlockProposal>` collects `ingress_messages` into a `Vec<SignedIngressId>` with no upper-bound check. A Byzantine subnet peer can craft a response carrying up to ~1.28 million fake entries within the 128 MB transport limit. `assemble_message` then spawns one indefinitely-looping tokio task per entry via `JoinSet::spawn`, exhausting replica memory and the tokio scheduler before any cancellation or ID-mismatch check fires.

## Finding Description

**Root cause — no count bound in `TryFrom<pb::StrippedBlockProposal>`:** [1](#0-0) 

`ingress_messages` is iterated and collected with no length guard. The developer comment at line 440 ("at most 1000 elements") is an assumption, not an enforced invariant. [2](#0-1) 

**Task spawning — one task per entry, unconditionally:** [3](#0-2) 

All N tasks are spawned before the bouncer select-loop is entered. The bouncer check at line 237 only fires during subsequent loop iterations, after all tasks are already live. [4](#0-3) 

**Each spawned task loops indefinitely:**
`download_stripped_message` uses `with_max_elapsed_time(None)` and an unconditional `loop {}`, so each task retries forever until the bouncer fires. [5](#0-4) [6](#0-5) 

**ID check does not cover the ingress list:**
`MaybeStrippedConsensusMessage::id()` derives the artifact ID solely from `unstripped_consensus_message_id`, not from the ingress list. A crafted artifact with a valid block-proposal hash and N fake ingress entries passes the `message.id() == id` check in `FetchArtifact`. [7](#0-6) 

**`try_assemble()` ID mismatch check is too late:** [8](#0-7) 

This check only runs after all N tasks have already been spawned and have completed (or the bouncer fires). It does not prevent the task explosion.

**Transport size limit — 128 MB, not a meaningful constraint:** [9](#0-8) 

At ~100 bytes per `StrippedIngressMessage` proto entry, a single 128 MB response encodes ~1.28 million fake entries — far exceeding any reasonable task budget.

**Bouncer refresh period — 3 seconds:** [10](#0-9) 

The window between task spawning and bouncer invalidation is sufficient to exhaust memory. If the Byzantine peer is the sole advertiser, tasks persist until the real block arrives.

## Impact Explanation

Each spawned tokio task allocates stack and heap state. N=1M tasks × ~8 KB stack = ~8 GB minimum stack allocation, plus heap for `Vec` entries and per-task RPC state. This causes replica OOM crash or severe tokio scheduler degradation, dropping the node out of consensus and halting subnet finalization until restart. This matches the allowed High impact: **Application/platform-level DoS, crash, consensus blocking, or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000).

## Likelihood Explanation

The attacker must be a legitimate subnet peer (below the Byzantine fault threshold — no key material, no majority required). They observe a valid `StrippedConsensusMessageId` from public P2P advertisements, then serve a crafted `StrippedBlockProposal` response when selected as the random download peer. The probability of selection approaches 1 if the Byzantine peer is the sole advertiser of the artifact. The attack is repeatable every consensus round.

## Recommendation

Add a count bound check in `TryFrom<pb::StrippedBlockProposal>` before collecting `ingress_messages` and `stripped_idkg_dealings`:

```rust
// In stripped.rs, before line 100
const MAX_INGRESS_MESSAGES_PER_BLOCK: usize = 1_000;
if value.ingress_messages.len() > MAX_INGRESS_MESSAGES_PER_BLOCK {
    return Err(ProxyDecodeError::Other(format!(
        "Too many ingress messages: {} > {}",
        value.ingress_messages.len(),
        MAX_INGRESS_MESSAGES_PER_BLOCK,
    )));
}
```

Apply the same bound to `stripped_idkg_dealings`. Ideally derive the constant from the subnet's `max_block_payload_size` / `max_ingress_bytes_per_message` registry parameters rather than hardcoding it.

## Proof of Concept

```rust
#[test]
fn unbounded_task_spawn_via_fake_ingress_ids() {
    let n = 500_000usize;
    let fake_ingress_messages: Vec<pb::StrippedIngressMessage> = (0..n)
        .map(|i| pb::StrippedIngressMessage {
            stripped: Some(/* fake IngressMessageId proto */),
            ingress_bytes_hash: vec![i as u8 % 256; 32],
        })
        .collect();

    let proto = pb::StrippedBlockProposal {
        pruned_block_proposal: Some(/* valid pruned proto with empty ingress_payload */),
        unstripped_consensus_message_id: Some(/* valid BlockProposal hash */),
        ingress_messages: fake_ingress_messages,
        stripped_idkg_dealings: vec![],
    };

    // TryFrom succeeds — no count check
    let stripped = StrippedBlockProposal::try_from(proto).unwrap();
    assert_eq!(stripped.stripped_ingress_payload.ingress_messages.len(), n);

    // BlockProposalAssembler::new allocates Vec of n entries
    let assembler = BlockProposalAssembler::new(stripped);
    let missing = assembler.missing_stripped_messages();
    assert_eq!(missing.len(), n);
    // In assemble_message, join_set.spawn() is called n times → OOM
}
```

The existing test `assemble_aborts_when_bouncer_becomes_unwanted` (assembler.rs line 1108) demonstrates the task-spawning path and can be extended with a large fake ingress list to confirm the unbounded spawn behavior.
