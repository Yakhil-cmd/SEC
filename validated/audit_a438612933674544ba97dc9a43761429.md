Audit Report

## Title
Cache Key Collision Between QueryV2 and QueryV3 Allows Cross-Version Response Poisoning — (`rs/boundary_node/ic_boundary/src/routes.rs`)

## Summary
`RequestContext::Hash` and `RequestContext::PartialEq` omit `request_type` from their implementations, causing QueryV2 and QueryV3 requests with identical `(canister_id, sender, method_name, ingress_expiry, arg)` tuples to collide on the same cache key at the boundary node. Because both request types pass the `BypasserIC` check and share the same `CacheState` instance, a cached QueryV2 response will be served to a QueryV3 client (or vice versa), delivering a structurally mismatched CBOR-encoded certificate delegation that conformant clients will reject.

## Finding Description
**Missing `request_type` in cache key:** `Hash` at `routes.rs:94–108` feeds only `canister_id`, `sender`, `method_name`, `ingress_expiry`, and `arg`/`http_request` into the hasher. `PartialEq` at `routes.rs:111–125` compares the same five fields. `request_type` is absent from both, so two `RequestContext` values that differ only in `request_type` are considered equal and produce the same hash.

**Both QueryV2 and QueryV3 pass the bypasser:** `is_query()` at `http/mod.rs:62–63` returns `true` for `QueryV2`, `QueryV3`, and `QuerySubnetV3`. The `BypasserIC::bypass` check at `cache.rs:74–76` only bypasses when `!ctx.request_type.is_query()`, so both versions are admitted to the cache.

**Shared `CacheState`:** `core.rs:856–858` registers both `PATH_QUERY_V2` and `PATH_QUERY_V3` into `canister_query_routes`. `core.rs:1016–1049` applies a single `canister_layers` stack — containing the same `cache_state` instance — to the merged route set. There is no per-version cache partition.

**`KeyExtractorContext` uses `RequestContext` directly as the key:** `cache.rs:33–44` extracts `Arc<RequestContext>` as the cache key. The `Hash` and `PartialEq` impls on `RequestContext` are therefore the sole determinants of cache key equality, and they do not distinguish V2 from V3.

**Structurally different responses per version:** `nns_delegation_test.rs:479–524` confirms that QueryV2 responses embed NNS delegations in `CertificateDelegationFormat::Flat` and QueryV3 responses embed them in `CertificateDelegationFormat::Tree`. The `interlaced_v2_and_v3_query_requests` test at lines 527–532 guards against this at the replica level only; it does not exercise the boundary node cache.

**Exploit flow:**
1. Attacker sends a QueryV2 POST to `/api/v2/canister/{id}/query` with a chosen `(canister_id, sender, method_name, ingress_expiry, arg)` tuple → cache miss, V2 response body stored.
2. Within the TTL window (default 1 s), attacker or victim sends a QueryV3 POST to `/api/v3/canister/{id}/query` with the identical tuple → cache hit, V2 response body returned to V3 client.
3. The V3 client receives a flat-format delegation instead of the expected tree-format delegation and rejects it.

## Impact Explanation
Any unprivileged client can deterministically poison the boundary node cache for a given `(canister_id, sender, method_name, ingress_expiry, arg)` tuple, causing all QueryV3 requests for that tuple within the TTL window to receive a structurally invalid (for V3) response. Conformant QueryV3 SDKs and clients that validate the delegation format will reject the response, producing a client-visible availability failure. This matches the allowed High impact: **Application/platform-level DoS or boundary/API security impact with concrete user harm** ($2,000–$10,000). There is no confidentiality or integrity violation; the node signature over the response content remains valid.

## Likelihood Explanation
No special privileges, keys, or network position are required. The collision is deterministic and locally reproducible. Any client can trigger it by sending a QueryV2 request followed by a QueryV3 request with the same parameters within the cache TTL. The attack is repeatable on every new TTL window.

## Recommendation
Include `request_type` in both `Hash` and `PartialEq` for `RequestContext` in `rs/boundary_node/ic_boundary/src/routes.rs`:

```rust
// In Hash::hash()
self.request_type.hash(state);

// In PartialEq::eq()
&& self.request_type == other.request_type
```

The comment at `routes.rs:91–93` already states the invariant that both impls must operate on the same fields — `request_type` simply needs to be added to both.

## Proof of Concept
```rust
let ctx_v2 = RequestContext {
    request_type: RequestType::QueryV2,
    canister_id: Some(principal),
    sender: Some(ANONYMOUS_PRINCIPAL),
    method_name: Some("foo".into()),
    ingress_expiry: Some(0),
    arg: Some(vec![1, 2, 3]),
    ..Default::default()
};
let ctx_v3 = RequestContext { request_type: RequestType::QueryV3, ..ctx_v2.clone() };

// Both hold today due to missing request_type in Hash/PartialEq:
assert_eq!(ctx_v2, ctx_v3);
assert_eq!(hash_of(&ctx_v2), hash_of(&ctx_v3));

// Drive ctx_v2 through cache_middleware → CacheStatus::Miss, V2 body cached
// Drive ctx_v3 through cache_middleware → CacheStatus::Hit, V2 body returned ← wrong version
```
