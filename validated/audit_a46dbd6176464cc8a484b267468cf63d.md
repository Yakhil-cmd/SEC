### Title
Floor Division in `native_per_pubdata` Computation Causes Systematic Pubdata Undercharging — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` — the native resource cost charged per byte of pubdata — is computed using truncating (floor) integer division. Because `native_per_gas` is computed with ceiling division (`div_ceil`) while `native_per_pubdata` uses `wrapping_div` (floor), every transaction that produces pubdata is systematically undercharged for its pubdata proving cost. The operator is undercompensated, and any unprivileged sender can exploit this to produce pubdata at a lower native cost than the protocol intends.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (ZK transaction flow), the two key per-unit native costs are computed as follows:

**`native_per_gas` — ceiling division (correct):** [1](#0-0) 

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
```

**`native_per_pubdata` — floor division (bug):** [2](#0-1) 

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` on `U256` is identical to truncating integer division (floor). When `pubdata_price` is not exactly divisible by `native_price`, the result is rounded down. This means:

```
native_per_pubdata * native_price  <  pubdata_price
```

So for every pubdata byte produced, the native resource charged is less than the actual cost. The same truncated value is propagated into:

1. `create_resources_for_tx` — intrinsic pubdata overhead is undercharged: [3](#0-2) 

2. `get_resources_to_charge_for_pubdata` — post-execution pubdata charge is undercharged: [4](#0-3) 

3. The public API helper `validate_l2_tx_intrinsic_native_resources` mirrors the same truncation: [5](#0-4) 

The asymmetry is intentional for `native_per_gas` (ceiling protects the operator) but is an oversight for `native_per_pubdata` (floor harms the operator).

**Concrete example:**
- `pubdata_price = 11`, `native_price = 10`
- `native_per_pubdata = 11 / 10 = 1` (floor), should be `2` (ceiling)
- For a transaction producing 100,000 pubdata bytes: charged = 100,000 native, actual cost = 110,000 native
- Operator loss per transaction: up to `(native_price - 1) * pubdata_bytes` native units

The double-resource accounting model is documented to use `nativePerGas := gasPrice/nativePrice` as the conversion ratio: [6](#0-5) 

The same rounding discipline should apply to `native_per_pubdata`.

---

### Impact Explanation

The operator is systematically undercompensated for pubdata proving costs. Every L2 transaction that produces pubdata pays fewer native resources than the protocol intends. An attacker can maximize this by crafting transactions that produce the maximum allowed pubdata per block, extracting the rounding discount on every byte. Over many transactions, this constitutes a resource accounting loss for the operator/sequencer. The `check_enough_resources_for_pubdata` gate is also too lenient, allowing transactions to produce more pubdata than their native budget should permit.

---

### Likelihood Explanation

This is triggered by any L2 transaction whenever `pubdata_price % native_price != 0`. In production, `pubdata_price` and `native_price` are independently set oracle values with no divisibility guarantee. The condition is therefore met in virtually every real block. Any unprivileged transaction sender triggers this path through normal transaction submission.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — undercharges pubdata):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — correct):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix to the mirrored computation in `api/src/helpers.rs` line 427. [5](#0-4) 

---

### Proof of Concept

1. Set block context: `pubdata_price = 11`, `native_price = 10`, `eip1559_basefee = 10`.
2. Submit an L2 ZKsync transaction with `max_fee_per_gas = 10`, `gas_limit = 1_000_000`.
3. Observe: `native_per_gas = ceil(10/10) = 1`; `native_per_pubdata = floor(11/10) = 1`.
4. The transaction's native budget is `1 * 1_000_000 = 1_000_000` native units.
5. Each pubdata byte costs only `1` native unit, but the true cost is `1.1` native units.
6. The transaction can produce up to `1_000_000` pubdata bytes before exhausting native resources, but the operator's actual proving cost for that pubdata is `1_100_000` native units — a 10% shortfall.
7. With `wrapping_div` replaced by `div_ceil`, `native_per_pubdata = 2`, and the transaction can only produce `500_000` pubdata bytes within the same native budget, correctly reflecting the true cost.
