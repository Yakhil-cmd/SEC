### Title
Third-Party Permissionless Stake Refresh Permanently Dilutes Neuron Age Bonus - (`rs/nns/governance/src/governance.rs`)

### Summary

The NNS (and SNS) governance `ClaimOrRefresh` command with the `NeuronIdOrSubaccount` variant performs no authorization check on the caller. Any unprivileged ingress sender can trigger a stake refresh for any neuron. When a third party sends ICP to a victim's neuron subaccount on the ledger and then calls `manage_neuron` with `ClaimOrRefresh { by: NeuronIdOrSubaccount }`, the governance canister calls `update_stake_adjust_age`, which permanently dilutes the neuron's `aging_since_timestamp_seconds`. This reduces the neuron's age bonus, voting power, and voting rewards — without the neuron owner's knowledge or consent. This is the direct IC analog of the StakedEXA bug: a third party deposits to another account and resets the time-weighted average (`avgStart` / `aging_since_timestamp_seconds`) without first settling the accrued benefit for the affected account.

---

### Finding Description

**Root cause in `refresh_neuron_by_id_or_subaccount` / `refresh_neuron`**

`refresh_neuron_by_id_or_subaccount` (NNS governance) performs no caller authorization check: [1](#0-0) 

It delegates to `refresh_neuron`, which also has no authorization check: [2](#0-1) 

When the ledger balance exceeds the cached stake (`Ordering::Less` branch), it calls `update_stake_adjust_age`: [3](#0-2) 

**`update_stake_adjust_age` permanently dilutes the age**

`update_stake_adjust_age` computes a weighted-average age: `new_age = (old_age * old_stake) / new_stake`. Adding new stake with age=0 always reduces the neuron's effective age: [4](#0-3) 

**Age bonus directly affects voting power and rewards**

The age bonus multiplier is applied to compute both potential and deciding voting power: [5](#0-4) 

**The same issue exists in SNS governance**

`claim_or_refresh_neuron_by_memo_and_controller` and `refresh_neuron` in SNS governance also have no authorization check on the caller: [6](#0-5) [7](#0-6) 

The SNS `update_stake` function performs the same age dilution: [8](#0-7) 

**The permissionless nature is confirmed by tests**

The test suite explicitly documents that any caller can refresh a neuron by subaccount: [9](#0-8) 

---

### Impact Explanation

A neuron that has been non-dissolving for years accumulates an age bonus of up to 25% on top of its base voting power. This bonus is a core economic incentive for long-term staking. When an attacker dilutes the age:

1. The neuron's `aging_since_timestamp_seconds` is permanently advanced toward `now`, reducing `age_seconds`.
2. The age bonus multiplier decreases, reducing both `potential_voting_power` and `deciding_voting_power`.
3. Voting rewards (which are proportional to voting power) are permanently reduced.
4. The lost age cannot be recovered — the neuron must re-age from the diluted baseline.

For a neuron with 4 years of age (maximum age bonus = 25%) and 10 ICP staked, an attacker who sends 10 ICP to the subaccount and triggers a refresh halves the age to 2 years, reducing the age bonus from 25% to ~12.5%. This is a permanent, irreversible loss of governance influence and reward income.

---

### Likelihood Explanation

The attack requires the attacker to send ICP to the victim's neuron subaccount (the ICP is not lost — it increases the victim's stake). The attacker's cost is the opportunity cost of locking ICP in the victim's neuron. The neuron subaccount is deterministic and publicly computable from the controller principal and memo. Any ingress sender can call `manage_neuron` with `ClaimOrRefresh { by: NeuronIdOrSubaccount }` targeting any neuron ID or subaccount. No privileged access is required. The attack is repeatable: the attacker can send small amounts repeatedly to continuously dilute the age as it re-accumulates.

---

### Recommendation

Add an authorization check to `refresh_neuron_by_id_or_subaccount` and `refresh_neuron` in both NNS and SNS governance. Only the neuron's controller, hot keys, or the neuron owner should be permitted to trigger a stake refresh that modifies `aging_since_timestamp_seconds`. Alternatively, decouple the age-dilution side-effect from the permissionless balance-sync operation: allow anyone to sync the cached stake upward, but only allow the neuron owner to accept the age dilution that comes with a new deposit.

---

### Proof of Concept

**Attack path (NNS governance):**

1. Victim has neuron N with `cached_neuron_stake_e8s = S`, `aging_since_timestamp_seconds = T_old` (e.g., 4 years of age).
2. Attacker computes the neuron's subaccount from the public neuron ID.
3. Attacker sends `S` ICP to the neuron's subaccount on the ICP ledger (ledger balance becomes `2S`).
4. Attacker calls `manage_neuron` (no authentication required beyond a valid ingress) with:
   ```
   ManageNeuron {
     neuron_id_or_subaccount: NeuronId(N),
     command: ClaimOrRefresh { by: NeuronIdOrSubaccount }
   }
   ```
5. `refresh_neuron_by_id_or_subaccount` → `refresh_neuron` → `update_stake_adjust_age(2S, now)` is called.
6. New age = `(old_age * S) / (2S)` = `old_age / 2`. The neuron's age is halved.
7. Victim's age bonus drops from 25% to ~12.5%, permanently reducing voting power and rewards.

The same attack applies to SNS governance neurons using `manage_neuron` with `ClaimOrRefresh { by: MemoAndController { controller: victim_principal, memo: victim_memo } }`.

    pub fn update_stake_adjust_age(&mut self, updated_stake_e8s: u64, now: u64) {
        // If the updated stake is less than the original stake, preserve the
        // age and distribute it over the new amount. This should not happen
        // in practice, so this code exists merely as a defensive fallback.
        //
        // TODO(NNS1-954) Consider whether update_stake_adjust_age (and other
        // similar methods) should use a neurons effective stake rather than
        // the cached stake.
        if updated_stake_e8s < self.cached_neuron_stake_e8s {
            println!(
                "{}Reducing neuron {:?} stake via update_stake_adjust_age: {} -> {}",
                LOG_PREFIX,
                self.id(),
                self.cached_neuron_stake_e8s,
                updated_stake_e8s
            );
            self.cached_neuron_stake_e8s = updated_stake_e8s;
        } else {
            // If one looks at "stake * age" as describing an area, the goal
            // at this point is to increase the stake while keeping the area
            // constant. This means decreasing the age in proportion to the
            // additional stake, which is the purpose of combine_aged_stakes.
            let (new_stake_e8s, new_age_seconds) = combine_aged_stakes(
                self.cached_neuron_stake_e8s,
                self.age_seconds(now),
                updated_stake_e8s.saturating_sub(self.cached_neuron_stake_e8s),
                0,
            );
            // A consequence of the math above is that the 'new_stake_e8s' is
            // always the same as the 'updated_stake_e8s'. We use
            // 'combine_aged_stakes' here to make sure the age is
            // appropriately pro-rated to accommodate the new stake.
            assert!(new_stake_e8s == updated_stake_e8s);
            self.cached_neuron_stake_e8s = new_stake_e8s;

            let new_aging_since_timestamp_seconds = now.saturating_sub(new_age_seconds);
            let new_disolved_dissolve_state_and_age = self
                .dissolve_state_and_age()
                .adjust_age(new_aging_since_timestamp_seconds);
            self.set_dissolve_state_and_age(new_disolved_dissolve_state_and_age);
        }

        let account = self.neuron_account_id(subaccount);

    pub fn update_stake(&mut self, new_stake_e8s: u64, now: u64) {
        // If this neuron has an age and its stake is being increased, adjust the
        // neuron's age
        if self.aging_since_timestamp_seconds < now && self.cached_neuron_stake_e8s <= new_stake_e8s
        {
            let old_stake = self.cached_neuron_stake_e8s as u128;
            let old_age = now.saturating_sub(self.aging_since_timestamp_seconds) as u128;
            let new_age = (old_age * old_stake) / (new_stake_e8s as u128);

            // new_age * new_stake = old_age * old_stake -
            // (old_stake * old_age) % new_stake. That is, age is
            // adjusted in proportion to the stake, but due to the
            // discrete nature of u64 numbers, some resolution is
            // lost due to the division above. This means the age
            // bonus is derived from a constant times age times
            // stake, minus up to new_stake - 1 each time the
            // neuron is refreshed. Only if old_age * old_stake is
            // a multiple of new_stake does the age remain
            // constant after the refresh operation. However, in
            // the end, the most that can be lost due to rounding
            // from the actual age, is always less 1 second, so
            // this is not a problem.
            self.aging_since_timestamp_seconds = now.saturating_sub(new_age as u64);
            // Note that if new_stake == old_stake, then
            // new_age == old_age, and
            // now - new_age =
            // now-(now-neuron.aging_since_timestamp_seconds)
            // = neuron.aging_since_timestamp_seconds.
        }

        self.cached_neuron_stake_e8s = new_stake_e8s;
    }
