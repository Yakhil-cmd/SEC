### Title
Scheduled XCC Promises Have No Expiration and No Cancellation — (`File: etc/xcc-router/src/lib.rs`)

### Summary

The XCC router contract stores scheduled NEAR promises indefinitely with no expiration timestamp and provides no cancellation mechanism. The `execute_scheduled` function is intentionally callable by any unprivileged account. This means NEAR tokens committed to a scheduled promise are permanently locked in the router sub-account until the promise is executed, and any third party can force-execute the promise at any time the user did not intend, with no recourse for the user to recover their funds.

### Finding Description

When an EVM contract on Aurora uses the `CrossContractCallArgs::Delayed` variant of the XCC precompile, the Aurora Engine calls `schedule` on the user's router sub-account. The `schedule` function stores the `PromiseArgs` in a `LookupMap` keyed by a monotonically increasing nonce, with no block-height deadline or timestamp recorded alongside it: [1](#0-0) 

The stored promise persists in contract storage indefinitely. There is no `cancel_scheduled` function anywhere in the router contract. The only way to remove a scheduled promise from storage is to execute it via `execute_scheduled`: [2](#0-1) 

The comment explicitly acknowledges that `execute_scheduled` is open to any caller. The rationale given is that the promise was already authorized by the parent engine. However, this reasoning ignores two compounding problems:

1. **No expiration**: A promise scheduled at block N can be executed at block N+1,000,000 with no check. The user has no way to set a deadline after which the promise becomes void.
2. **No cancellation**: The user (or the parent engine on their behalf) has no function to remove a scheduled promise and return the committed NEAR to the router's balance for reuse or withdrawal.

The `CrossContractCallArgs` type documents the `Delayed` variant's purpose as allowing a fresh 300 Tgas budget in a separate transaction: [3](#0-2) 

When a `Delayed` call includes `attached_balance > 0`, the engine first unwraps wNEAR from the user's EVM balance and transfers real NEAR to the router sub-account before calling `schedule`. That NEAR is now held by the router. Because there is no expiration and no cancellation path, those funds are locked in the router until `execute_scheduled` is called — by anyone, at any time.

The full router interface contains only: `initialize`, `get_version`, `execute`, `schedule`, `execute_scheduled`, `deploy_upgrade`, and `send_refund`. None of these allow a user or the parent engine to cancel a pending scheduled promise and reclaim the attached NEAR. [4](#0-3) 

### Impact Explanation

**High — Temporary freezing of funds.**

NEAR tokens unwrapped from a user's wNEAR balance and committed to a scheduled promise are locked in the router sub-account with no recovery path. The user cannot cancel the promise. The NEAR cannot be returned to the user's EVM balance or to any other destination. The funds remain frozen until `execute_scheduled` is called. Because any third party can call `execute_scheduled`, a malicious actor can also force-execute the promise at a moment the user did not intend (e.g., after a target contract has been upgraded to a malicious version, or during adverse market conditions for a DeFi call), causing the user's NEAR to be irrecoverably sent to the target specified at scheduling time.

### Likelihood Explanation

**Medium.** The `Delayed` XCC path with `attached_balance > 0` is a documented, production-supported flow. Any EVM contract or user that schedules a cross-contract call with attached NEAR and subsequently needs to abort (due to changed conditions, a compromised target, or a protocol upgrade) is exposed. The force-execution attack requires only that an adversary know the router sub-account address (deterministically derived from the user's EVM address and the engine account ID) and the nonce of the scheduled promise (logged on-chain at scheduling time). [5](#0-4) 

### Recommendation

1. Record a `deadline_block` alongside each scheduled promise at `schedule` time (e.g., current block + a configurable maximum window).
2. In `execute_scheduled`, reject execution if `env::block_height() > deadline_block`.
3. Add a `cancel_scheduled(nonce)` function callable only by the parent engine that removes the promise and transfers the router's NEAR balance back to the engine (or credits the user's wNEAR).

### Proof of Concept

1. User's EVM contract calls the XCC precompile with `CrossContractCallArgs::Delayed(PromiseArgs::Create(PromiseCreateArgs { target_account_id: "target.near", attached_balance: Yocto::new(1_000_000_000_000_000_000_000_000), ... }))`.
2. The engine unwraps 1 NEAR from the user's wNEAR EVM balance, sends it to `<user_hex>.aurora`, and calls `schedule` on that router. The nonce `0` is logged: `"Promise scheduled at nonce 0"`.
3. The user's target contract is later compromised. The user has no function to call to cancel nonce `0` and recover the 1 NEAR.
4. An adversary observes the log, calls `<user_hex>.aurora/execute_scheduled({"nonce": "0"})` with max gas. The promise executes, sending 1 NEAR to the compromised target. The user's funds are lost with no recourse. [6](#0-5)

#[near]
impl Router {
    #[init(ignore_state)]
    #[must_use]
    pub fn initialize(wnear_account: AccountId, must_register: bool) -> Self {
        // The first time this function is called there is no state and the parent is set to be
        // the predecessor account id. In subsequent calls, only the original parent is allowed to
        // call this function. The idea is that the Create, Deploy and Initialize actions are done in a single
        // NEAR batch when a new router is deployed by the engine, so the caller will be the Aurora
        // engine instance that the user's address belongs to. If we update this contract and deploy
        // a new version of it, again the Deploy and Initialize actions will be done in a single batch
        // by the engine.
        let caller = env::predecessor_account_id();
        let mut parent = LazyOption::new(StorageKey::Parent, None);
        match parent.get() {
            None => {
                parent.set(&caller);
            }
            Some(parent) => {
                // Allow self-calls to `initialize` also.
                // This happens during the upgrade flow.
                if (caller != parent) && (caller != env::current_account_id()) {
                    env::panic_str(ERR_ILLEGAL_CALLER);
                }
            }
        }

        if must_register {
            env::promise_create(
                wnear_account.clone(),
                "storage_deposit",
                b"{}",
                WNEAR_REGISTER_AMOUNT,
                WNEAR_REGISTER_GAS,
            );
        }

        let mut version = LazyOption::new(StorageKey::Version, None);
        if version.get().unwrap_or_default() != CURRENT_VERSION {
            // Future migrations would go here

            version.set(&CURRENT_VERSION);
        }

        let nonce = LazyOption::new(StorageKey::Nonce, None);
        let scheduled_promises = LookupMap::new(StorageKey::Map);
        Self {
            parent,
            version,
            nonce,
            scheduled_promises,
            wnear_account,
        }
    }

    pub fn get_version(&self) -> u32 {
        self.version.get().unwrap_or_default()
    }

    /// This function can only be called by the parent account (i.e. Aurora engine) to ensure that
    /// no one can create calls on behalf of the user this router contract is deployed for.
    /// The engine only calls this function when the special precompile in the EVM for NEAR cross
    /// contract calls is used by the address associated with the sub-account this router contract
    /// is deployed at.
    pub fn execute(&self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }

    /// Similar security considerations here as for `execute`.
    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

    /// It is intentional that this function can be called by anyone (not just the parent).
    /// There is no security risk to allowing this function to be open because it can only
    /// act on promises that were created via `schedule`.
    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }

    /// Allows the parent contract to trigger an update to the logic of this contract
    /// (by deploying a new contract to this account);
    #[payable]
    pub fn deploy_upgrade(&mut self, #[serializer(borsh)] args: DeployUpgradeParams) {
        self.assert_preconditions();

        let promise_id = env::promise_batch_create(&env::current_account_id());
        env::promise_batch_action_deploy_contract(promise_id, &args.code);
        env::promise_batch_action_function_call(
            promise_id,
            INITIALIZE,
            &args.initialize_args,
            NearToken::default(),
            INITIALIZE_GAS,
        );
        env::promise_return(promise_id);
    }

    pub fn send_refund(&self) -> Promise {
        let parent = self.get_parent().unwrap_or_else(env_panic);

        require_caller(&parent)
            .and_then(|_| require_no_failed_promises())
            .unwrap_or_else(env_panic);

        Promise::new(parent).transfer(REFUND_AMOUNT)
    }
}
