Audit Report

## Title
Ed25519 Torsion-Point Public Key Bypasses Signature Verification in Ingress Path — (`rs/crypto/internal/crypto_lib/basic_sig/ed25519/src/api.rs`, `packages/ic-ed25519/src/lib.rs`)

## Summary

The `verify()` function and the `verify_basic_sig_by_public_key` function used in the ingress validation path call `deserialize_raw` followed immediately by `verify_signature` without any torsion-freeness check. The `verify_signature` implementation uses the ZIP215 cofactor equation `8*(s·B − k·A − R) = 0`, which is trivially satisfied for any small-order public key A (where `8·A = 0`) by choosing R = s·B for any scalar s. The codebase's own test suite confirms that all 18 canonical torsion points pass `deserialize_raw().unwrap()`. An attacker can derive a self-authenticating principal from any such torsion key and forge a valid signature for any message without possessing any secret scalar.

## Finding Description

**Root cause — `verify()` skips the torsion check that `verify_public_key()` performs**

`verify_public_key` at `rs/crypto/internal/crypto_lib/basic_sig/ed25519/src/api.rs` L97–103 explicitly calls `pk.is_torsion_free()` and is the only function that enforces subgroup membership. `verify()` at the same file L70–91 calls `deserialize_raw` then `verify_signature` with no intervening torsion check. These are two separate code paths and the ingress path never reaches `verify_public_key`. [1](#0-0) [2](#0-1) 

**`deserialize_raw` explicitly accepts torsion points**

The library documents and tests that `deserialize_raw` does not check subgroup membership. The test `public_key_accepts_but_can_detect_keys_with_torsion_component` confirms all 18 canonical torsion points pass `deserialize_raw().unwrap()` while failing `is_torsion_free()`. [3](#0-2) [4](#0-3) 

**`verify_signature` uses the ZIP215 cofactor equation**

The check is `(s·B − k·A − R).mul_by_cofactor().is_identity()`, i.e., `8·(s·B − k·A − R) = 0`. For any torsion point A with `ord(A) | 8`, `k·(8·A) = 0`, so the equation collapses to `8·(s·B − R) = 0`. Choosing R = s·B satisfies this for every message and every challenge scalar k. [5](#0-4) 

**Ingress path never calls `verify_public_key`**

The full ingress call chain is:

```
validate_user_id_and_signature          [ingress_validation.rs L842]
  → validate_signature                  [ingress_validation.rs L635]
      → public_key_from_bytes           [sign_utils.rs L50 — deserialize_rfc8410_der, no torsion check]
      → validate_signature_plain        [ingress_validation.rs L705]
          → verify_basic_sig_by_public_key  [standalone-sig-verifier/src/lib.rs L20]
              → deserialize_raw             [no torsion check]
              → pk.verify_signature         [ZIP215, no torsion check]
``` [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8) 

`user_public_key_from_bytes` calls `deserialize_rfc8410_der`, which is also documented as performing no torsion check. [10](#0-9) 

## Impact Explanation

An attacker can pick any of the 18 confirmed canonical torsion points, DER-encode it, and derive a self-authenticating principal P = SHA224(DER(A)) ‖ 0x02. For any ingress message M, they choose any scalar s, compute R = s·B, and submit the 64-byte signature (R ‖ s). `verify_signature` returns `Ok(())` for every message because `8·(s·B − k·A − s·B) = 8·(−k·A) = −k·(8·A) = 0`. The attacker can submit arbitrary authenticated ingress calls attributed to P without possessing any secret key material. Any canister that P controls or has been granted permissions on is fully accessible. This constitutes unauthorized access to canister-controlled assets and identities, matching the High bounty impact tier: unauthorized access to wallets, identities, or canister-controlled funds where the attacker must first establish P as a privileged principal.

## Likelihood Explanation

The attack requires no privileged access and is computationally trivial (one scalar multiplication to compute R = s·B). The principal P is deterministic and publicly computable from any of the 18 known torsion-point encodings. The forged signature is valid for every message, so the attack is repeatable without limit. The only constraint is that P must hold some on-chain permission, which the attacker can arrange by deploying a canister with P as controller or by participating in any protocol that accepts self-authenticating principals.

## Recommendation

Add a torsion-freeness check inside `verify()` in `rs/crypto/internal/crypto_lib/basic_sig/ed25519/src/api.rs` before calling `verify_signature`:

```rust
pub fn verify(sig: &types::SignatureBytes, msg: &[u8], pk: &types::PublicKeyBytes) -> CryptoResult<()> {
    let public_key = ic_ed25519::PublicKey::deserialize_raw(&pk.0).map_err(|e| {
        CryptoError::MalformedPublicKey { ... }
    })?;
    if !public_key.is_torsion_free() {
        return Err(CryptoError::MalformedPublicKey {
            algorithm: AlgorithmId::Ed25519,
            key_bytes: Some(pk.0.to_vec()),
            internal_error: "Public key is not in the prime-order subgroup".to_string(),
        });
    }
    public_key.verify_signature(msg, &sig.0).map_err(...)
}
```

The same guard should be added to `verify_basic_sig_by_public_key` in `rs/crypto/standalone-sig-verifier/src/lib.rs` L20–44. Alternatively, enforce the check in `user_public_key_from_bytes` in `rs/crypto/standalone-sig-verifier/src/sign_utils.rs` so that any torsion-component key is rejected at the deserialization boundary before reaching any verification function. ## Proof of Concept

```rust
use hex_literal::hex;
use ic_ed25519::PublicKey;
use ic_types::PrincipalId;

// 1. Any of the 18 confirmed canonical torsion points
let torsion_raw = hex!("c7176a703d4dd84fba3c0b760d10670f2a2053fa2c39ccc64ec7fd7792ac03fa");

// Confirm it passes deserialize_raw and fails is_torsion_free
let pk = PublicKey::deserialize_raw(&torsion_raw).unwrap();
assert!(!pk.is_torsion_free()); // confirmed torsion point

// 2. DER-encode and derive self-authenticating principal
let pk_der = PublicKey::convert_raw32_to_der(torsion_raw);
let principal = PrincipalId::new_self_authenticating(&pk_der);

// 3. Forge signature: choose s=1, R = 1*B (base point compressed)
// ED25519_BASEPOINT_COMPRESSED = 5866666...
let r_bytes = hex!("5866666666666666666666666666666666666666666666666666666666666666");
let s_bytes: [u8; 32] = { let mut b = [0u8; 32]; b[0] = 1; b }; // scalar 1
let mut forged_sig = [0u8; 64];
forged_sig[..32].copy_from_slice(&r_bytes);
forged_sig[32..].copy_from_slice(&s_bytes);

// 4. verify_signature returns Ok(()) for any message because:
//    8*(1*B - k*A - 1*B) = 8*(-k*A) = -k*(8*A) = -k*0 = 0
let result = pk.verify_signature(b"any message at all", &forged_sig);
assert!(result.is_ok()); // forgery succeeds

// 5. Submit ingress with sender=principal, signer_pubkey=pk_der, signature=forged_sig
// validate_user_id_and_signature → validate_signature → verify_basic_sig_by_public_key
// returns Ok(()) — ingress accepted
```
