Fee-on-Transfer NEP-141 Token Causes ERC-20 Mirror Over-Minting and Bridge Insolvency - (`engine/src/engine.rs`)

---

### Summary

`receive_erc20_tokens` and `receive_base_tokens` in `engine/src/engine.rs` unconditionally trust `args.amount` from the `ft_on_transfer` callback to determine how many ERC-20 (or base ETH) tokens to mint on Aurora. For a fee-on-transfer NEP-141 token, the actual amount credited to Aurora's account on the NEP-141 contract is `args.amount - fee`, but Aurora mints the full `args.amount`. This permanently inflates the ERC-20 mirror supply beyond its NEP-141 backing, making the bridge insolvent for that token.

---

### Finding Description

The `ft_on_transfer` entrypoint in `engine/src/contract_methods/connector.rs` is the NEAR callback invoked by any NEP-141 contract after a `ft_transfer_call`. It dispatches to either `receive_base_tokens` or `receive_erc20_tokens` in `engine/src/engine.rs`.

In `receive_erc20_tokens`:

```rust
// engine/src/engine.rs:803
let amount = args.amount.as_u128();
// ...
setup_receive_erc20_tokens_input(&recipient, amount)  // line 831 — mints `amount` ERC-20 tokens
```

In `receive_base_tokens`:

```rust
// engine/src/engine.rs:778
let amount = Wei::new_u128(args.amount.as_u128());
// ...
set_balance(&mut self.io, &receipient, &new_balance);  // line 785 — credits `amount` ETH
```

Neither function queries the NEP-141 contract's balance of Aurora before and after the transfer to determine the real amount received. `args.amount` is a caller-supplied field from the NEP-141 contract's `ft_on_transfer` invocation. For a fee-on-transfer NEP-141 token, the NEP-141 contract deducts `amount` from the sender, applies a fee (e.g., burns or redirects a percentage), credits only `amount - fee` to Aurora's NEP-141 balance, and then calls `ft_on_transfer` with the original `amount`. Aurora mints `amount` ERC-20 tokens despite holding only `amount - fee` NEP-141 tokens.

`setup_receive_erc20_tokens_input` encodes a `mint(recipient, amount)` call to the `EvmErc20` contract:

```rust
// engine/src/engine.rs:1306-1313
pub fn setup_receive_erc20_tokens_input(recipient: &Address, amount: u128) -> Vec<u8> {
    let selector = ERC20_MINT_SELECTOR;
    let tail = ethabi::encode(&[
        ethabi::Token::Address(recipient.raw().0.into()),
        ethabi::Token::Uint(amount.into()),
    ]);
    [selector, tail.as_slice()].concat()
}
```

The `EvmErc20.mint` function in `etc/eth-contracts/contracts/EvmErc20.sol:49-51` mints unconditionally:

```solidity
function mint(address account, uint256 amount) public onlyAdmin {
    _mint(account, amount);
}
```

There is no reconciliation step anywhere in the call chain.

---

### Impact Explanation

Every deposit of a fee-on-transfer NEP-141 token inflates the ERC-20 mirror supply by `fee` tokens beyond the NEP-141 backing held by Aurora. Over time, or in a single large deposit, the cumulative over-minted ERC-20 supply exceeds Aurora's NEP-141 balance. When users attempt to exit (burn ERC-20 via `withdrawToNear` → `exitToNear` precompile → `ft_transfer` on the NEP-141 contract), the last `total_fee` worth of ERC-20 tokens cannot be redeemed because Aurora does not hold the corresponding NEP-141 tokens. Those ERC-20 holders suffer a permanent, irrecoverable loss of funds — a bridge insolvency and permanent fund freeze.

**Impact: Critical — Insolvency / Permanent Freezing of Funds.**

---

### Likelihood Explanation

The NEP-141 standard does not prohibit fee-on-transfer implementations. Any token deployer can publish a NEP-141 contract that applies a transfer fee and then call `deploy_erc20_token` on Aurora to register it. No privileged role is required to register a new NEP-141 token — `deploy_erc20_token` is a public entrypoint. Once registered, any user performing a `ft_transfer_call` into Aurora triggers the vulnerable path. The attacker does not need to compromise any key or governance mechanism; they only need to deploy a compliant-looking NEP-141 token with a hidden fee.

**Likelihood: Medium** — requires deploying a custom NEP-141 token, but no privileged access is needed and the registration path is fully open.

---

### Recommendation

In `receive_erc20_tokens` and `receive_base_tokens`, record the NEP-141 balance of the Aurora contract before the transfer is accepted and compute the actual received amount as the balance increase. Mint only the verified received amount. Because `ft_on_transfer` is a callback (the transfer has already occurred), the balance check can be performed synchronously via a cross-contract read or, more practically, by returning `args.amount` (rejecting the deposit) whenever the token is known to be fee-on-transfer, or by minting only `min(args.amount, verified_balance_increase)`.

A simpler conservative fix: after the NEP-141 `ft_transfer_call` settles, use a callback that reads Aurora's actual NEP-141 balance and mints only the delta, rather than trusting `args.amount` directly.

---

### Proof of Concept

1. Deploy a NEP-141 token contract `fee_token.near` that implements a 1% burn fee on every transfer: when `ft_transfer_call(aurora, 1000, msg)` is called, it credits only 990 tokens to Aurora and burns 10, then calls `ft_on_transfer(sender, 1000, msg)` on Aurora.

2. Call `deploy_erc20_token` on Aurora to register `fee_token.near` → Aurora deploys a corresponding `EvmErc20` mirror contract at some EVM address.

3. Call `ft_transfer_call` on `fee_token.near` with `receiver_id = aurora`, `amount = 1000`, `msg = <recipient_evm_address>`.

4. Aurora's `ft_on_transfer` fires:
   - `args.amount = 1000`
   - `receive_erc20_tokens` executes `setup_receive_erc20_tokens_input(recipient, 1000)` → mints **1000** ERC-20 tokens.
   - Aurora's actual NEP-141 balance of `fee_token.near` increased by only **990**.

5. Repeat N times. After N deposits of 1000, Aurora holds `990 * N` NEP-141 tokens but has `1000 * N` ERC-20 tokens in circulation.

6. The first `990 * N / 1000` ERC-20 holders can exit successfully. The remaining `10 * N` ERC-20 tokens can never be redeemed — those holders' funds are permanently frozen, and the bridge is insolvent for this token.
