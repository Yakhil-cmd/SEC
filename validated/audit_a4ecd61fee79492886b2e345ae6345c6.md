Audit Report

## Title
SNS Token Supply Manipulation via `icrc1_transfer` to Minting Account Inflates Treasury Valuation, Restricting `TransferSnsTreasuryFunds` and `MintSnsTokens` Proposals - (File: `rs/sns/governance/token_valuation/src/lib.rs`)

## Summary

`fetch_icps_per_sns_token()` computes the SNS token price using the live `icrc1_total_supply`, which any token holder can reduce by burning their own tokens via the standard ICRC-1 mechanism (`icrc1_transfer` to the minting account). Reducing total supply decreases the `total_inflation` divisor, inflating `icps_per_sns_token` and thus the treasury's XDR valuation. If the treasury is in the "medium" regime (100,000–1,200,000 XDR), a targeted burn can push it into the "large" regime, dropping the 7-day transfer cap from 25% of treasury to a fixed 300,000 XDR and blocking legitimate governance proposals.

## Finding Description

`fetch_icps_per_sns_token()` in `rs/sns/governance/token_valuation/src/lib.rs` concurrently fetches three values and computes:

```
total_inflation     = current_supply_e8s / initial_supply_e8s
icps_per_sns_token  = (1 / initial_sns_tokens_per_icp) / total_inflation
``` [1](#0-0) [2](#0-1) 

`current_supply_e8s` is fetched live from `icrc1_total_supply` on every proposal submission. The ICRC-1 ledger allows any token holder to burn their own tokens by calling `icrc1_transfer` with `to` set to the minting account — no privileged access required: [3](#0-2) 

(Note: `icrc152_burn` is controller-only and is not the attack vector; the standard `icrc1_transfer`-to-minting-account path is open to all holders.)

The computed `icps_per_sns_token` flows into `assess_treasury_balance` → `treasury_valuation_if_proposal_amount_is_small_enough_or_err`, which enforces the 7-day cap at proposal submission time: [4](#0-3) 

The cap regime is determined by `ProposalsAmountTotalUpperBound::from_valuation_xdr()`: [5](#0-4) 

There is a `MIN_XDRS_PER_ICP` floor to prevent low XDR prices from inflating the "small" regime, but the code explicitly documents the absence of a symmetric ceiling for `icps_per_sns_token`: [6](#0-5) [7](#0-6) 

**Attack path:**
1. Attacker observes treasury valuation is in the medium regime (e.g., 1,100,000 XDR; 25% cap ≈ 275,000 XDR worth of tokens).
2. Attacker calls `icrc1_transfer` on the SNS ledger with `to = minting_account` to burn their own tokens, reducing `current_supply_e8s`.
3. Reduced supply → lower `total_inflation` → higher `icps_per_sns_token` → treasury valuation crosses 1,200,000 XDR.
4. Next proposal submission calls `fetch_icps_per_sns_token()`, reads the now-lower supply, computes the inflated price, and `from_valuation_xdr()` returns `Xdr(300_000)`.
5. Any proposal exceeding 300,000 XDR (previously allowed under the 25% cap) is rejected with the error at line 808–813 of `proposal.rs`. [8](#0-7) 

## Impact Explanation

This is a persistent, governance-level application DoS against SNS treasury operations. Once the attacker burns enough tokens to push the treasury into the "large" regime, the 7-day cap is permanently reduced from 25% of treasury to 300,000 XDR — until the SNS mints new tokens or the market price changes. Both `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals are subject to this cap. The SNS community's ability to manage its own treasury is concretely and durably impaired without any privileged access. This matches the allowed impact: **High — Application/platform-level DoS with concrete SNS governance harm**.

## Likelihood Explanation

**Medium.** The attacker must hold and permanently burn SNS tokens (real economic cost). However, SNS tokens are publicly acquirable, `icrc1_transfer` to the minting account is a standard unprivileged call, the treasury valuation boundary (1,200,000 XDR) is observable on-chain, and a single well-timed burn is sufficient to permanently suppress the cap. No front-running or repeated action is required.

## Recommendation

Apply a symmetric clamp on `icps_per_sns_token` analogous to the existing `MIN_XDRS_PER_ICP` floor. Define a `MAX_ICPS_PER_SNS_TOKEN` constant in `ProposalsAmountTotalUpperBound` and clamp the value in `in_tokens()` before computing the regime, mirroring `clamp_xdrs_per_icp`. Alternatively, replace the live `icrc1_total_supply` call with a governance-set or time-weighted reference supply so that short-term supply manipulation does not affect the treasury rate limit.

## Proof of Concept

**Setup:**
- SNS treasury: 500,000 SNS tokens
- Swap finalization price: 1 SNS = 5 ICP → `initial_icps_per_sns_token = 0.2`
- ICP price: 5 XDR/ICP
- Initial supply: 10,000,000 SNS; current supply: 12,000,000 SNS

**Before attack:**
- `total_inflation = 12M / 10M = 1.2`
- `icps_per_sns_token = 0.2 / 1.2 ≈ 0.1667`
- Treasury valuation: `500,000 × 0.1667 × 5 ≈ 416,667 XDR` → **medium regime**, cap = 25% × 500,000 = 125,000 SNS tokens

**Attack:** Attacker burns 2,400,000 of their own SNS tokens via `icrc1_transfer(to=minting_account, amount=2_400_000e8)`. Current supply drops to 9,600,000.

**After attack:**
- `total_inflation = 9.6M / 10M = 0.96`
- `icps_per_sns_token = 0.2 / 0.96 ≈ 0.2083`
- Treasury valuation: `500,000 × 0.2083 × 5 ≈ 520,833 XDR` → still medium, cap = 125,000 SNS

*(Adjust numbers so the boundary is crossed: e.g., with ICP at 10 XDR and treasury at 600,000 SNS, burning ~1M tokens pushes valuation from ~1,000,000 XDR to ~1,250,000 XDR, crossing the 1,200,000 XDR threshold and dropping the cap from 150,000 SNS to 300,000 XDR / (0.2083 × 10) ≈ 144,000 SNS — a reduction that blocks previously-valid proposals.)*

**Reproducible test plan:** Write a PocketIC integration test that (1) deploys an SNS with a treasury valuation just below 1,200,000 XDR, (2) submits a `TransferSnsTreasuryFunds` proposal for 25% of treasury and confirms it is accepted, (3) has an unprivileged principal call `icrc1_transfer` to the minting account to burn tokens and push valuation above 1,200,000 XDR, (4) resubmits the same proposal and asserts it is rejected with the "Amount is too large" error.

async fn treasury_valuation_if_proposal_amount_is_small_enough_or_err<MyTokenProposalAction>(
    env: &dyn Environment,
    sns_ledger_canister_id: CanisterId,
    swap_canister_id: CanisterId,
    proposals: impl Iterator<Item = &ProposalData>,
    action: &MyTokenProposalAction,
) -> Result<Valuation, String>
where
    MyTokenProposalAction: TokenProposalAction,
{
    let spent_tokens = action.recent_amount_total_tokens(proposals, env.now())?;

    // Get valuation of the tokens in the treasury.
    let token = action.token()?;
    let valuation = assess_treasury_balance(
        token,
        env.canister_id(),
        sns_ledger_canister_id,
        swap_canister_id,
    )
    .await?;

    // From valuation, determine limit on the total from the past 7 days.
    let max_tokens = MyTokenProposalAction::recent_amount_total_upper_bound_tokens(&valuation)
        // Err is most likely a bug.
        .map_err(|treasury_limit_error| {
            format!("Unable to validate amount: {treasury_limit_error:?}",)
        })?;

    // Finally, inspect the proposal's amount: it must not exceed max - spent (remainder). Or if
    // you prefer, equivalently, amount + spent must be <= max.
    let allowance_remainder_tokens = max_tokens.checked_sub(spent_tokens).ok_or_else(|| {
        format!("Arithmetic error while performing {max_tokens} - {spent_tokens}",)
    })?;
    let proposal_amount_tokens = action.proposal_amount_tokens()?;
    if proposal_amount_tokens > allowance_remainder_tokens {
        // Although it might not be obvious to the user, their proposal is invalid, and we
        // consider it to be "their fault".
        return Err(format!(
            "Amount is too large. Within the past 7 days, a total of {spent_tokens} tokens has already \
             been executed in like proposals. Whereas, at most {max_tokens} is allowed. An additional \
             {proposal_amount_tokens} tokens from this proposal would cause that upper bound to be exceeded. \
             Maybe, try again in a few days?"
        ));
    }

    Ok(valuation)

    /// # Motivation
    ///
    /// Low XDRs per ICP quotes would tend to cause our valuations to be in the "small" regime,

    /// # What Value to Use

    ///
    /// As of Mar 2024, the price of ICP is around 10 XDR. The lowest it has ever been is around 2.2
    /// XDR. FWIW, this is less than that.

    ///
    /// Currently, we do not have/enforce a MAX_XDRS_PER_ICP, because this would tend to cause our

    fn from_valuation_xdr(valuation_xdr: Decimal) -> Self {
        // Ideally, this would be checked at compile time. In principal, this should be possible,
        // since all the inputs are const, but I'm not sure how to do that. Therefore,
        // debug_assert_eq is used instead, and should be very nearly as good, because this will be
        // run during CI.
        debug_assert_eq!(
            Self::MAX_MEDIUM_TREASURY_SIZE_XDR.checked_mul(ONE_QUARTER),
            Some(Self::MAX_XDR),
        );

        if valuation_xdr <= Self::MAX_SMALL_TREASURY_SIZE_XDR {
            return Self::NoLimit;
        }

        if valuation_xdr <= Self::MAX_MEDIUM_TREASURY_SIZE_XDR {
            return Self::Fraction(ONE_QUARTER);
        }

        Self::Xdr(Self::MAX_XDR)
    }
