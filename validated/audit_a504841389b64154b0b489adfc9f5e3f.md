### Title
ERC-20 Token Permanent Loss When `ft_transfer` Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` Cargo feature is not enabled (it is **not** in the crate's `default` features), the `ExitToNear` precompile burns a user's ERC-20 tokens before the NEAR-side `ft_transfer` is confirmed. If that transfer fails, no refund callback is scheduled and the burned tokens are permanently unrecoverable.

---

### Finding Description

The `ExitToNear` precompile handles bridging ERC-20 tokens from Aurora back to NEAR as NEP-141 tokens. The flow is:

1. The ERC-20 contract calls the precompile, which **burns** the tokens from the user's Aurora balance.
2. The precompile schedules an `ft_transfer` (or `ft_transfer_call`) promise to the NEP-141 contract on NEAR.
3. Optionally, a callback `exit_to_near_precompile_callback` is attached to handle the result.

The callback is only attached when `callback_args != ExitToNearPrecompileCallbackArgs::default()`: [1](#0-0) 

The `refund` field of `callback_args` is populated **only** when the `error_refund` feature is compiled in: [2](#0-1) 

`error_refund` is **not** listed in the `default` features of either the `engine` or `engine-precompiles` crates: [3](#0-2) [4](#0-3) 

**Consequence for regular ERC-20 exits (non-wNEAR):** When `error_refund` is disabled and `transfer_near` is also `None`, `callback_args` equals the default, so **no callback is scheduled at all** — `PromiseArgs::Create(transfer_promise)` is used. If `ft_transfer` fails (e.g., recipient not registered with the NEP-141 contract), there is no recovery path.

**Consequence for wNEAR exits:** A callback is scheduled (because `transfer_near` is `Some`), but in the failure branch of `exit_to_near_precompile_callback`, `args.refund` is `None`, so the `else { None }` branch is taken — again, no refund: [5](#0-4) 

The test suite explicitly acknowledges this behavior: [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user's ERC-20 tokens are burned from their Aurora balance before the NEAR-side transfer is confirmed. If the `ft_transfer` fails (a common scenario in NEAR's storage-deposit model when the recipient is not registered), the tokens are permanently destroyed with no recovery path when `error_refund` is not enabled.

---

### Likelihood Explanation

**High.** The `ft_transfer` failure scenario is realistic and common: NEAR's NEP-141 standard requires recipients to have a storage deposit before they can receive tokens. Any user who exits ERC-20 tokens to a NEAR account that has not registered with the NEP-141 contract will trigger this failure. The `error_refund` feature is not in the default feature set, meaning any deployment that does not explicitly opt into it is vulnerable.

---

### Recommendation

1. Add `error_refund` to the `default` features of both `engine` and `engine-precompiles` crates so refund logic is always compiled in.
2. Alternatively, restructure the exit flow to use a check-effects-interactions pattern: do not burn ERC-20 tokens until the NEAR-side transfer is confirmed successful in the callback.
3. At minimum, document clearly that deployments without `error_refund` expose users to permanent token loss on failed exits.

---

### Proof of Concept

1. User holds 1,000 `TOKEN` ERC-20 on Aurora (backed by a NEP-141 on NEAR).
2. User calls the `ExitToNear` precompile to bridge 1,000 `TOKEN` to `unregistered.near` (an account with no storage deposit on the NEP-141 contract).
3. The precompile burns 1,000 `TOKEN` from the user's ERC-20 balance.
4. Because `error_refund` is not enabled, `callback_args` is the default, so only `PromiseArgs::Create(ft_transfer_promise)` is scheduled — no callback.
5. The `ft_transfer` promise fails because `unregistered.near` has no storage deposit.
6. No callback fires. The 1,000 `TOKEN` ERC-20 balance is gone. The user receives nothing on NEAR.
7. The tokens are permanently lost.

The test at `engine-tests/src/tests/erc20_connector.rs:623` (`test_exit_to_near_refund`) demonstrates exactly this scenario, confirming the behavior is reachable and reproducible. [7](#0-6)
