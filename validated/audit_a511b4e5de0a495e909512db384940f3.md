### Title
ERC-20 Mirror Over-Minting via Fee-on-Transfer NEP-141 Tokens in `receive_erc20_tokens` - (File: `engine/src/engine.rs`)

### Summary

The `receive_erc20_tokens` function mints ERC-20 tokens on Aurora based on the `amount` field supplied in the `FtOnTransferArgs` callback argument, without verifying the actual NEP-141 balance change received by the engine. If a fee-on-transfer NEP-141 token is registered with Aurora, the engine will mint more ERC-20 tokens than the NEP-141 tokens it actually holds, creating an insolvency gap that grows with every bridge-in operation.

### Finding Description

When a NEP-141 token is bridged into Aurora via `ft_transfer_call`, the NEP-141 contract calls `ft_on_transfer` on the Aurora Engine. Aurora's handler dispatches to `receive_erc20_tokens`, which reads the amount to mint directly from the callback argument:

```rust
// engine/src/engine.rs L803
let amount = args.amount.as_u128();
```

This `amount` is then used verbatim to call the ERC-20 `mint` function:

```rust
// engine/src/engine.rs L826-837
let result = self
    .call(
        &erc20_admin_address,
        &erc20_token,
        Wei::zero(),
        setup_receive_erc20_tokens_input(&recipient, amount),
        ...
    )
    .and_then(submit_result_or_err)?;
```

The engine never reads its own NEP-141 balance before and after the transfer to confirm how many tokens it actually received. For a fee-on-transfer NEP-141 token, the token contract transfers `amount - fee` to Aurora but reports `amount` in the `ft_on_transfer` callback. Aurora mints the full `amount` of ERC-20 tokens, creating `fee` units of unbacked ERC-20 supply per bridge-in call.

The `ft_on_transfer` dispatcher in `engine/src/contract_methods/connector.rs` performs no balance verification either; it simply passes `args` through:

```rust
// engine/src/contract_methods/connector.rs L80-90
let args: FtOnTransferArgs = read_json_args(&io)?;
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
```

### Impact Explanation

**Critical — Insolvency.** Each bridge-in of a fee-on-transfer NEP-141 token mints more ERC-20 tokens than the NEP-141 backing held by the engine. Over repeated calls the cumulative deficit grows. When users attempt to exit (burn ERC-20 and receive NEP-141 via `withdrawToNear`), the last withdrawers will find the engine holds insufficient NEP-141 tokens to honour their burns. The engine becomes insolvent for that token.

### Likelihood Explanation

**Medium.** The attack requires a fee-on-transfer NEP-141 token to be registered with Aurora (i.e., have an ERC-20 mirror deployed via `deploy_erc20_token`). Once registered, any user can trigger the over-minting by calling `ft_transfer_call` on the NEP-141 contract targeting Aurora. No privileged access is required after registration. Fee-on-transfer tokens exist in the NEAR ecosystem, and the registration flow contains no guard against them.

### Recommendation

In `receive_erc20_tokens`, record the ERC-20 token's total supply (or the engine's NEP-141 balance) before and after the mint call and use the actual delta as the canonical minted amount, rather than trusting `args.amount`. Alternatively, record the NEP-141 balance of the engine account before and after `ft_transfer_call` completes (analogous to the fix in the referenced report) and use that delta to drive minting.

```rust
// Pseudocode fix
let balance_before = nep141_balance_of(engine_account, token);
// ... ft_transfer_call executes ...
let balance_after = nep141_balance_of(engine_account, token);
let actual_amount = balance_after - balance_before;
setup_receive_erc20_tokens_input(&recipient, actual_amount)
```

### Proof of Concept

1. Deploy a NEP-141 token that deducts a 10% fee on every transfer (transfers `amount * 0.9` to the recipient but reports `amount` in `ft_on_transfer`).
2. Register it with Aurora via `deploy_erc20_token`, creating an ERC-20 mirror.
3. Call `ft_transfer_call` on the NEP-141 contract with `amount = 1000`, `receiver_id = aurora`.
4. The NEP-141 contract transfers 900 tokens to Aurora and calls `ft_on_transfer` with `amount = 1000`.
5. Aurora calls `receive_erc20_tokens` and mints 1000 ERC-20 tokens to the caller.
6. Aurora holds 900 NEP-141 tokens but has issued 1000 ERC-20 tokens — 100 units of unbacked supply.
7. Repeat to grow the deficit until the engine cannot honour all exit requests. [1](#0-0) [2](#0-1) [3](#0-2)

```rust
    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
```
