Audit Report

## Title
Unbounded `ingress_expiries` Loop via Client-Controlled `ingress_start`/`ingress_end` Causes OOM Crash of ICP Rosetta Server — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
`RosettaRequestHandler::construction_payloads` accepts client-supplied `ingress_start` and `ingress_end` values from request metadata with no range validation. These values directly control a `while` loop that pushes entries into a `Vec<u64>`. With `ingress_start = 0` and `ingress_end = u64::MAX`, the loop executes ~153 million iterations before `Time::add_assign`'s truncating `as u64` cast wraps `now` back to a small value, after which the loop becomes infinite, exhausting heap memory and crashing the Rosetta server process.

## Finding Description
The vulnerable loop is at `construction_payloads.rs` lines 99–107:

```rust
let mut ingress_expiries = vec![];
let mut now = ingress_start;
while now < ingress_end {
    let ingress_expiry = (now + ic_limits::MAX_INGRESS_TTL.saturating_sub(ic_limits::PERMITTED_DRIFT))
        .as_nanos_since_unix_epoch();
    ingress_expiries.push(ingress_expiry);
    now += interval;  // interval = 120_000_000_000 ns
}
``` [1](#0-0) 

`ingress_start` and `ingress_end` are taken directly from client-supplied metadata with no bounds or range check: [2](#0-1) 

`interval` is computed as `MAX_INGRESS_TTL (300s) - PERMITTED_DRIFT (60s) - 120s = 120s = 120,000,000,000 ns`: [3](#0-2) [4](#0-3) 

`Time::add_assign` calls `from_duration`, which uses a truncating `as u64` cast on the `u128` result of `Duration::as_nanos()`: [5](#0-4) [6](#0-5) 

When `now` approaches `u64::MAX`, adding `interval` (120,000,000,000 ns) causes `Duration::from_nanos(self.0) + other` to produce a `u128` value exceeding `u64::MAX`. The `as u64` truncation wraps `now` back to a small value (~120 seconds). Since that wrapped value is still `< u64::MAX` (`ingress_end`), the loop condition remains true and the loop continues indefinitely.

The endpoint is unauthenticated: [7](#0-6) 

The ICRC1 Rosetta counterpart has explicit guards that the ICP version lacks entirely: [8](#0-7) 

## Impact Explanation
With `ingress_start = 0` and `ingress_end = u64::MAX`: before the wrap, ~153,722,867 iterations each push an 8-byte `u64`, allocating ~1.2 GB in `ingress_expiries` alone. After the wrap, the loop becomes infinite and the process is killed by the OS OOM killer. This crashes the ICP Rosetta server, blocking all ICP exchange operations (transfers, neuron management, staking) routed through that instance until the process is restarted. This matches the allowed impact: **High ($2,000–$10,000) — Significant Rosetta/financial-integration security impact with concrete user or protocol harm.**

## Likelihood Explanation
The `/construction/payloads` endpoint requires no authentication or privileged role. The malicious payload is a standard JSON POST body containing two integer fields; no special tooling is required. The attack is deterministic, reproducible locally, and repeatable — a single request is sufficient to crash the server, and the attack can be re-issued immediately after each restart. The ICRC1 Rosetta already carries the fix, confirming developer awareness of the pattern.

## Recommendation
Add range validation before the loop in `RosettaRequestHandler::construction_payloads`, mirroring the ICRC1 implementation in `rs/rosetta-api/icrc1/src/construction_api/services.rs` lines 148–158:

1. Reject if `ingress_start >= ingress_end`.
2. Reject if `ingress_end` exceeds `ingress_start` by more than a reasonable maximum (e.g., 24 hours).
3. Add an explicit `MAX_INGRESS_EXPIRIES` cap and return an `ApiError` if the computed count would exceed it.

## Proof of Concept
```bash
curl -s -X POST http://<rosetta-host>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
    "operations": [...],
    "public_keys": [{"hex_bytes":"<valid_pubkey>","curve_type":"edwards25519"}],
    "metadata": {
      "ingress_start": 0,
      "ingress_end": 18446744073709551615
    }
  }'
```

The server process exhausts available memory and is killed before returning a response. A unit-level reproduction can be written directly against `RosettaRequestHandler::construction_payloads` with a mock ledger, passing `ingress_start = Time::from_nanos_since_unix_epoch(0)` and `ingress_end = Time::from_nanos_since_unix_epoch(u64::MAX)`, and asserting the function returns an `ApiError` rather than looping.
