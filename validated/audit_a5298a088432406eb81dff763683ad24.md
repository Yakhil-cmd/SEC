Audit Report

## Title
Silent Integer Wrap in `add_expiration` FieldsDisplay Branch Produces Incorrect Consent-Message Expiration Timestamp — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
The `add_expiration` function in `responses.rs` contains two divergent code paths for `GenericDisplay` and `FieldsDisplay`. The `FieldsDisplay` branch performs two successive unsafe casts (`u64 → i64`, then `i64 → u64`) when converting nanosecond timestamps to seconds, silently wrapping for any `expires_at > i64::MAX`. A malicious dApp can submit `expires_at = u64::MAX` to cause a `FieldsDisplay`-capable wallet to render the approval expiration as Unix epoch (Jan 1 1970) while the on-chain approval is stored with `expires_at = u64::MAX` (~584 years from now), undermining the ICRC-21 consent message integrity guarantee.

## Finding Description

**Root cause — `FieldsDisplay` branch** (lines 264–271):

```rust
Some(expires_at) => {
    let seconds = (expires_at as i64) / 10_i64.pow(9);   // u64 → i64 wraps for expires_at > i64::MAX
    fields_display.fields.push((
        "Approval expiration".to_string(),
        Value::TimestampSeconds {
            amount: seconds as u64,   // negative i64 wraps again silently
        },
    ))
}
```

Confirmed at [1](#0-0) 

**Concrete arithmetic for `expires_at = u64::MAX`:**
- `u64::MAX as i64` → `-1` (two's-complement wrap)
- `-1_i64 / 1_000_000_000` → `0` (Rust truncates toward zero)
- `0_i64 as u64` → `0`
- Result: `TimestampSeconds { amount: 0 }` = Unix epoch, Jan 1 1970

**Concrete arithmetic for `expires_at = i64::MAX + 1`:**
- `as i64` → `i64::MIN = -9_223_372_036_854_775_808`
- `/ 1_000_000_000` → `-9_223_372_036`
- `as u64` → `18_446_744_064_486_227_580` (year ~584 billion)

**`GenericDisplay` branch** (lines 232–243) correctly handles the same inputs by calling `time::OffsetDateTime::from_unix_timestamp(seconds)` and `replace_nanosecond(nanos)`, returning error strings for out-of-range values: [2](#0-1) 

**Ledger accepts any `u64` for `expires_at` without range validation.** The only guard in the approvals layer is: [3](#0-2) 

`u64::MAX` is far in the future, so `expires_at <= now` is false and the approval is accepted and stored on-chain verbatim: [4](#0-3) 

## Impact Explanation

The ICRC-21 consent message system exists specifically to protect users from malicious dApps by having the trusted ledger canister generate the consent message. This bug breaks that guarantee for `FieldsDisplay` wallets: a malicious dApp sets `expires_at = u64::MAX`, the ledger generates `TimestampSeconds { amount: 0 }`, the wallet renders the expiration as Jan 1 1970 (appearing already expired or trivially short-lived), the user signs believing the approval is harmless, and the on-chain approval persists for ~584 years. This is a concrete consent-message integrity failure in the ICRC ledger with direct potential for user fund loss, matching: **High ($2,000–$10,000) — Significant ledger security impact with concrete user or protocol harm.**

## Likelihood Explanation

The attack requires a malicious dApp that deliberately sets `expires_at > i64::MAX` and a wallet that uses `FieldsDisplay`. `FieldsDisplay` is an explicitly supported variant in the ICRC-21 standard and present in production `.did` files. The entrypoint is fully public, requires no privilege, and the arithmetic is deterministic. The malicious dApp does not need to phish or socially engineer the user beyond normal dApp interaction — the deception occurs in the ledger-generated consent message itself, which the user is supposed to trust.

## Recommendation

Replace the unsafe cast chain in the `FieldsDisplay` branch with unsigned division:

```rust
Some(expires_at) => {
    let seconds = expires_at / 1_000_000_000_u64;  // no cast needed
    fields_display.fields.push((
        "Approval expiration".to_string(),
        Value::TimestampSeconds { amount: seconds },
    ))
}
```

Optionally, mirror the `GenericDisplay` validity check by verifying `time::OffsetDateTime::from_unix_timestamp(seconds as i64).is_ok()` before pushing, to keep both branches consistent.

## Proof of Concept

```rust
#[test]
fn test_add_expiration_fields_display_wrap() {
    let mut msg = ConsentMessage::FieldsDisplayMessage(FieldsDisplay::default());
    msg.add_expiration(Some(u64::MAX), None);
    if let ConsentMessage::FieldsDisplayMessage(ref fd) = msg {
        let (_, val) = fd.fields.last().unwrap();
        // Current buggy code produces TimestampSeconds { amount: 0 } (Unix epoch)
        // Correct result should be u64::MAX / 1_000_000_000 = 18_446_744_073
        assert_eq!(
            *val,
            Value::TimestampSeconds { amount: u64::MAX / 1_000_000_000 },
            "FieldsDisplay wraps u64::MAX to epoch instead of correct seconds"
        );
    }
}
```

This test fails against the current code. A differential test comparing `GenericDisplay` and `FieldsDisplay` output for `expires_at = u64::MAX` will show the two paths disagree: `GenericDisplay` returns `"Invalid nanosecond: 4294967295"`, while `FieldsDisplay` returns `TimestampSeconds { amount: 0 }`.
