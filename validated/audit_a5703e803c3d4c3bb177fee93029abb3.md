Audit Report

## Title
Hardcoded Default ICP/XDR Conversion Rate Enables Cycles Over-Minting During CMC Initialization Window - (File: rs/nns/cmc/src/main.rs, rs/nns/cmc/src/lib.rs)

## Summary
`State::default()` unconditionally sets `icp_xdr_conversion_rate` to a hardcoded value of `1_000_000` permyriad (100 XDR/ICP), which is 20–33x above the real market rate. The `init()` function calls `State::default()` and does not accept or apply a real rate from `CyclesCanisterInitPayload`. Any unprivileged user who calls `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` before governance submits the first `set_icp_xdr_conversion_rate` proposal will receive cycles computed at the inflated rate.

## Finding Description
In `rs/nns/cmc/src/lib.rs` L33–34, two constants define the bootstrap rate: [1](#0-0) 

`State::default()` in `rs/nns/cmc/src/main.rs` L360–373 constructs `initial_icp_xdr_conversion_rate` from these constants and assigns it as `Some(...)` to both `icp_xdr_conversion_rate` and `average_icp_xdr_conversion_rate`: [2](#0-1) 

The `init()` function at L466 calls `State::default()` and then only populates `ledger_canister_id`, `governance_canister_id`, `minting_account_id`, `last_purged_notification`, `exchange_rate_canister_id`, and `cycles_ledger_canister_id` from the payload — `icp_xdr_conversion_rate` is never overridden: [3](#0-2) 

`CyclesCanisterInitPayload` has no field for an initial conversion rate, so there is no mechanism to supply a real rate at install time: [4](#0-3) 

`tokens_to_cycles()` reads `state.icp_xdr_conversion_rate` directly. Because `State::default()` sets it to `Some(...)`, the `None` guard at L1912–1920 is never reached after `init()`, and the hardcoded rate is used for all cycle computations: [5](#0-4) 

All three public minting paths call `tokens_to_cycles` unconditionally: [6](#0-5) [7](#0-6) [8](#0-7) 

The `do_set_icp_xdr_conversion_rate` guard only rejects a proposed rate whose timestamp is ≤ the current timestamp (`1_620_633_600`). Any governance proposal with a real current timestamp passes, confirming the guard does not close the initialization window: [9](#0-8) 

## Impact Explanation
The hardcoded default of `1_000_000` permyriad (100 XDR/ICP) is 20–33x above the real market rate (~30,000–50,000 permyriad). During the window between `init()` and the first governance rate update, an unprivileged user can mint cycles at this inflated rate. For 1 ICP: `1e8 × 1_000_000 × 1e12 / (1e8 × 10_000) = 100T cycles` versus the correct ~4T cycles at 40,000 permyriad. This constitutes illegal minting of cycles — a concrete protocol accounting harm — matching the High impact class: **Significant NNS/CMC security impact with concrete protocol harm** ($2,000–$10,000). It does not reach Critical because exploitation requires the precondition of a fresh CMC installation or reinstall, limiting the realistic attack surface.

## Likelihood Explanation
The CMC is a system canister on the NNS subnet. A fresh install or governance-approved reinstall is the required precondition. Once that precondition is met, the subnet is accessible to external users, and the window between the reinstall proposal execution and the first `set_icp_xdr_conversion_rate` proposal execution is observable on-chain. Any user monitoring the chain can detect the fresh initialization and race to send ICP before the rate is corrected. No special privileges are required to call `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles`. The attack is repeatable for the full duration of the window.

## Recommendation
Add an `initial_icp_xdr_conversion_rate: Option<IcpXdrConversionRate>` field to `CyclesCanisterInitPayload`. In `init()`, if this field is `Some`, apply it to `state.icp_xdr_conversion_rate` instead of the hardcoded default. Governance should always supply the current market rate as part of any CMC installation proposal. As a defense-in-depth measure, the CMC should treat `icp_xdr_conversion_rate = None` as the initial state and return an explicit error from `tokens_to_cycles` until at least one governance-supplied rate has been set, rather than falling back to the hardcoded constant.

## Proof of Concept
1. A governance proposal reinstalls the CMC. `init()` calls `State::default()`, setting `icp_xdr_conversion_rate = Some({ timestamp_seconds: 1_620_633_600, xdr_permyriad_per_icp: 1_000_000 })`.
2. Before governance submits a `set_icp_xdr_conversion_rate` proposal with the real rate, an attacker sends 1 ICP to the CMC's top-up subaccount for their canister.
3. The attacker calls `notify_top_up`. `tokens_to_cycles(1 ICP)` computes: `1e8 × 1_000_000 × 1e12 / (1e8 × 10_000) = 100T cycles`.
4. At the real rate of ~40,000 permyriad, the correct amount is ~4T cycles.
5. The attacker receives ~25x more cycles than they should.
6. A deterministic integration test can reproduce this by: (a) installing the CMC with a standard `CyclesCanisterInitPayload` (no rate field), (b) immediately calling `notify_top_up` with a known ICP amount, and (c) asserting the returned cycles equal `amount × 1_000_000 × DEFAULT_CYCLES_PER_XDR / (TOKEN_SUBDIVIDABLE_BY × 10_000)` — confirming the hardcoded rate is active.

fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles {
                xdr_permyriad_per_icp,
                cycles_per_xdr: state.cycles_per_xdr,
            }
            .to_cycles(amount)),

                let error_message =

                print(&error_message);
                Err(NotifyError::Other {
                    error_code: NotifyErrorCode::Internal as u64,
                    error_message,
                })
            }
        }
    })
