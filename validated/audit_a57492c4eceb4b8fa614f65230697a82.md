Audit Report

## Title
Unchecked Application-Level Return Value in SNS Generic Nervous System Function Execution Marks Failed Proposals as Executed - (File: rs/sns/governance/src/canister_control.rs)

## Summary

`perform_execute_generic_nervous_system_function_call` in `rs/sns/governance/src/canister_control.rs` discards the reply bytes from the target canister call and unconditionally returns `Ok(())`, treating any transport-level reply as a successful execution. A target canister method that returns an encoded `Err(...)` value (rather than trapping/rejecting) causes the SNS proposal to be permanently marked as executed even though the intended action had no effect. The proposal is consumed and cannot be re-executed.

## Finding Description

In `rs/sns/governance/src/canister_control.rs`, `perform_execute_generic_nervous_system_function_call` dispatches a canister call and matches on the result. The `Ok(_reply)` arm at line 302 discards the reply bytes entirely and returns `Ok(())`, with a `TODO` comment explicitly acknowledging that application-level errors in the reply are not detected:

```rust
Ok(_reply) => {
    // TODO: Do something with reply. E.g. store it in the proposal,
    // and/or deserialize it so that we can detect whether there was an
    // application-level error, as opposed to a communication error.
    // For now, any reply is considered a success.
    Ok(())
}
```

The validator path `perform_execute_generic_nervous_system_function_validate_and_render_call` (lines 262–274) does decode the reply as `Result<String, String>` and propagates errors. The execution path does not apply the same discipline.

This `Ok(())` propagates through `perform_execute_generic_nervous_system_function` (lines 2531–2555 of `governance.rs`) and `perform_action` (line 2172–2175), and is passed to `set_proposal_execution_status`, which permanently sets `proposal.executed_timestamp_seconds = self.env.now()` (line 1725) and clears any failure state.

The call chain is fully confirmed:
- `perform_action` → `perform_execute_generic_nervous_system_function` → `perform_execute_generic_nervous_system_function_call` → returns `Ok(())` → `set_proposal_execution_status` marks proposal as executed.

Existing guards are insufficient: the only guard is the transport-level `Err(err)` arm (line 297), which catches IC-level rejections/traps. It does not catch application-level `Err(...)` values encoded in the reply bytes.

## Impact Explanation

Any `ExecuteGenericNervousSystemFunction` SNS proposal targeting a canister method that returns `Result<_, _>` and returns `Err(...)` under failure conditions will be permanently marked as `executed_timestamp_seconds > 0`. The intended action (e.g., a parameter change, treasury operation, or dapp configuration update) does not take effect. The proposal is consumed and cannot be re-executed. SNS governance participants and monitoring tools will observe the proposal as successfully executed, creating a false belief that the system state was updated. This constitutes a significant SNS governance security impact with concrete user and protocol harm, matching the **High** impact category.

## Likelihood Explanation

Any SNS neuron holder can submit an `ExecuteGenericNervousSystemFunction` proposal once a `NervousSystemFunction` is registered. Many real-world canister methods return `Result<_, _>` and may return `Err(...)` due to precondition failures, misconfiguration, or transient state. The scenario is realistic and reachable without any privileged access beyond normal SNS neuron holding. The TODO comment in the production code confirms the developers are aware of this gap.

## Recommendation

**Short term:** Document that any reply from the target canister — including encoded application-level errors — is treated as a successful execution. SNS operators should ensure target canister methods reject (trap/panic) rather than return `Err(...)` to signal failure.

**Long term:** Decode the reply bytes and check for application-level errors before marking the proposal as executed. The code already contains a `TODO` with the exact approach:

```rust
Ok(reply) => {
    match candid::Decode!(&reply, Result<String, String>) {
        Ok(Err(e)) => Err(GovernanceError::new_with_message(ErrorType::External, e)),
        _ => Ok(()),
    }
}
```

A convention (e.g., requiring target methods to return `Result<_, String>`) should be established and enforced at proposal registration time.

## Proof of Concept

1. Register a `NervousSystemFunction` whose `target_method` returns `Result<(), String>` and returns `Err("precondition failed".to_string())` under certain conditions.
2. Submit and adopt an `ExecuteGenericNervousSystemFunction` proposal targeting that method when the error condition is active.
3. Observe that `perform_execute_generic_nervous_system_function_call` receives `Ok(encoded_err_bytes)` from `env.call_canister`.
4. The `Ok(_reply)` arm at line 302 of `canister_control.rs` discards the bytes and returns `Ok(())`.
5. `set_proposal_execution_status` sets `proposal.executed_timestamp_seconds = self.env.now()` at line 1725 of `governance.rs`.
6. The proposal is permanently marked as executed; the intended action had no effect; no retry is possible.

This can be demonstrated with a PocketIC integration test: deploy a mock target canister that returns `candid::Encode!(&Err::<(), String>("precondition failed".to_string()))`, register it as a `NervousSystemFunction`, submit and adopt a proposal, and assert that `executed_timestamp_seconds > 0` while the intended state change did not occur.
