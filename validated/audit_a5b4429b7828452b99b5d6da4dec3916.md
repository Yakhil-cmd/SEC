### Title
Permanent Fund Freeze via Unhandled Transfer Failure in wNEAR Unwrap Exit Flow — (`engine/src/contract_methods/connector.rs`)

---

### Summary

When an EVM user exits wNEAR ERC-20 tokens back to NEAR using the `:unwrap` suffix via the `exitToNear` precompile, the engine burns the ERC-20 tokens, calls `near_withdraw` on the wNEAR contract, and then — in `exit_to_near_precompile_callback` — issues a bare `PromiseAction::Transfer` to the user-supplied `target_account_id` **with no failure callback**. If that Transfer fails (because the target NEAR account does not exist or has been deleted), the NEAR is refunded to Aurora's own balance with no mechanism to credit it back to the user. The ERC-20 tokens are already burned and irrecoverable, resulting in a permanent fund freeze.

---

### Finding Description

The wNEAR unwrap exit flow is a two-step async process:

**Step 1 — `ExitToNear::run` in `engine-precompiles/src/native.rs`:**

When the exit flag is `Message::UnwrapWnear` and the caller is the wNEAR ERC-20 contract, the precompile constructs a `PromiseWithCallbackArgs` whose base is `near_withdraw` on the wNEAR NEP-141 account, and whose callback is `exit_to_near_precompile_callback` on the Aurora engine. The user-supplied `receiver_account_id` is stored verbatim in `TransferNearArgs::target_account_id` inside the callback args. [1](#0-0) 

The callback args are assembled with `transfer_near: transfer_near_args` (the user-controlled account) and passed to the callback promise: [2](#0-1) 

**Step 2 — `exit_to_near_precompile_callback` in `engine/src/contract_methods/connector.rs`:**

If `promise_result(0)` (the `near_withdraw` result) is `Successful`, the callback creates a `PromiseBatchAction` with a single `PromiseAction::Transfer` to `args.target_account_id` and calls `handler.promise_return(promise_id)`. **There is no further callback attached to this Transfer promise.** [3](#0-2) 

The existing refund branch only fires when `promise_result(0)` is **not** `Successful` (i.e., when `near_withdraw` itself fails): [4](#0-3) 

There is no code path that handles the case where `near_withdraw` succeeds but the subsequent `Transfer` fails. In NEAR Protocol, a `Transfer` action to a non-existent named account fails and the attached NEAR is refunded to the predecessor (Aurora). Aurora has no callback to detect this and re-credit the user.

The `ExitToNearPrecompileCallbackArgs` struct confirms that `transfer_near` carries only the target account and amount — no refund address for the Transfer leg: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The ERC-20 wNEAR tokens are burned at the EVM level before any NEAR-side promise executes. Once `near_withdraw` succeeds, the NEAR is held by Aurora. If the subsequent `Transfer` to `target_account_id` fails, the NEAR is returned to Aurora's account balance but there is no callback, no storage record, and no recovery function that allows the user to reclaim it. The funds are permanently frozen inside the Aurora contract.

---

### Likelihood Explanation

**Medium.**

There are two realistic trigger paths:

1. **User error (self-inflicted, no coordination needed):** A user specifies a non-existent NEAR named account (e.g., a typo, a future account that has not been created yet, or an account that was deleted). The `receiver_account_id` is validated only as a syntactically valid NEAR account ID — not for on-chain existence — so any well-formed but non-existent account ID passes through.

2. **Adversarial account deletion (direct analog to the reported ATA ownership change):** A user exits wNEAR to a counterparty's NEAR account (e.g., a DeFi protocol or a known recipient). Between the submission of the exit transaction and the execution of the Transfer receipt (which can be in a subsequent block), the target account deletes itself. The Transfer fails, and the user's NEAR is permanently frozen. This mirrors the original report's scenario where the maker changes ATA ownership to block the taker's collateral claim.

---

### Recommendation

Attach a failure-handling callback to the `Transfer` promise created inside `exit_to_near_precompile_callback`. If the Transfer fails, the callback should either:

- Re-mint the equivalent amount of wNEAR ERC-20 tokens to the original sender's EVM address (analogous to the `refund_on_error` path used when `near_withdraw` fails), or
- Store the NEAR amount in Aurora's state keyed to the sender so it can be claimed later.

Concretely, replace the bare `promise_create_batch` + `promise_return` with a `promise_attach_batch_callback` that calls a new `exit_to_near_transfer_callback` method on the Aurora engine, which checks the Transfer result and triggers a refund if it failed.

---

### Proof of Concept

1. User holds `N` wNEAR ERC-20 tokens on Aurora (address `0xUser`).
2. User calls the `exitToNear` precompile with flag `0x01` (ERC-20), amount `N`, and `receiver_account_id = "deleted-account.near"` (a syntactically valid but non-existent NEAR account).
3. Aurora burns `N` wNEAR ERC-20 tokens from `0xUser` and schedules:
   - Base: `near_withdraw({"amount": N})` on the wNEAR NEP-141 contract.
   - Callback: `exit_to_near_precompile_callback` on Aurora with `transfer_near = {target_account_id: "deleted-account.near", amount: N}`.
4. `near_withdraw` executes successfully; Aurora's NEAR balance increases by `N` yoctoNEAR.
5. `exit_to_near_precompile_callback` fires, sees `PromiseResult::Successful`, and issues `Transfer { amount: N }` to `"deleted-account.near"` with no callback.
6. The Transfer fails (account does not exist on NEAR). NEAR Protocol refunds `N` yoctoNEAR to Aurora's account.
7. No callback fires. Aurora's balance is silently increased. `0xUser` has zero wNEAR ERC-20 tokens and zero NEAR. Funds are permanently frozen. [3](#0-2) [6](#0-5)
