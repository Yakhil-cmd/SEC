Audit Report

## Title
Zero-Cost Ingress Induction for `UpdateSettings` and `CreateCanister` Enables Ingress Queue Flooding - (File: `rs/cycles_account_manager/src/cycles_account_manager.rs`)

## Summary
Two management canister ingress paths — `UpdateSettings` with payload ≤ 338 bytes and `CreateCanister` — are inducted into replicated state with `IngressInductionCost::Free` on application subnets. Because the only backstop is a global, non-per-sender `ingress_history_max_messages` cap, any unprivileged user can monopolize the ingress history and cause a complete denial of service of the subnet's ingress path.

## Finding Description

**Path 1 — `UpdateSettings` with small payload:**

In `ingress_induction_cost` (`rs/cycles_account_manager/src/cycles_account_manager.rs`, lines 570–578), when the method is `UpdateSettings` and `is_delayed_ingress_induction_cost` returns true (payload ≤ `MAX_DELAYED_INGRESS_COST_PAYLOAD_SIZE` = 338 bytes, line 45), `paying_canister` is set to `None`, which falls through to `IngressInductionCost::Free` at line 595. The authorization check (whether the sender is a controller) occurs only during execution, not at induction time, so any principal can trigger this path.

The deferred charge applied during execution at lines 1043–1052 of `rs/execution_environment/src/execution_environment.rs` explicitly discards the error (`let _ignore_error = ...`), meaning even a canister with zero cycles processes the message for free end-to-end.

**Path 2 — `CreateCanister` (and `ProvisionalCreateCanisterWithCycles`, `ProvisionalTopUpCanister`):**

`extract_effective_canister_id` in `rs/types/types/src/messages/ingress_messages.rs` (lines 585–587) returns `Ok(None)` for these methods. Back in `ingress_induction_cost`, since the method is not `UpdateSettings`, the code takes the `else` branch at line 580 and sets `paying_canister = effective_canister_id = None`, again yielding `IngressInductionCost::Free` at line 595.

**No cycle gate at induction:**

In `rs/messaging/src/scheduling/valid_set_rule.rs` (lines 286–291), `IngressInductionCost::Free` messages are pushed directly into the ingress queue with no cycle balance check. The only guard is the global `ingress_history_max_messages` cap (lines 250–255), which is not per-sender. The ingress selector in `rs/ingress_manager/src/ingress_selector.rs` (lines 592–594) similarly does nothing for free messages. No per-sender rate limit exists anywhere in the induction path.

## Impact Explanation

An unprivileged attacker can continuously submit `UpdateSettings` (≤ 338-byte payload) or `CreateCanister` ingress messages at zero cycles cost. These fill the subnet's `ingress_history_max_messages` global cap. Once the cap is reached, all subsequent legitimate ingress messages are rejected with `IngressHistoryFull`, causing a complete denial of service of the application subnet's ingress path. This matches the **High ($2,000–$10,000)** impact: *"Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

The attack requires only a valid IC identity (anonymous or self-authenticating principal) and knowledge of any existing canister ID on the target subnet. No privileged access, governance majority, or threshold corruption is needed. The `UpdateSettings` payload is trivially constructable (Candid-encoded struct with a canister ID and any settings field, well under 338 bytes). The attack is fully reachable from the public `/api/v3/canister/.../call` endpoint and is repeatable indefinitely.

## Recommendation

1. **Remove the unconditional free-induction exception for `UpdateSettings`.** Charge the induction cost upfront and refund it if the canister is found to be frozen and the sender is a verified controller. Alternatively, gate the free-induction path on a pre-check that the sender is a controller of the target canister at induction time.
2. **Assign a non-zero induction cost to `CreateCanister` ingress messages.** Introduce a sender-funded deposit or require the sender to have a registered cycle balance, since no canister currently pays for these messages.
3. **Add a per-sender rate limit** in the ingress pool or ingress selector to prevent any single principal from monopolizing the ingress history.

## Proof of Concept

```
# Attacker constructs a minimal UpdateSettings payload (≤ 338 bytes):
# Candid: record { canister_id = <any_canister_id>; settings = record {} }

# Send in a tight loop to the target application subnet:
for i in 1..N:
    POST /api/v3/canister/aaaaa-aa/call
    {
      request_type: "call",
      canister_id: "aaaaa-aa",   # ic:00
      method_name: "update_settings",
      arg: <candid-encoded UpdateSettingsArgs with target canister_id>,
      sender: <attacker_principal>,
      ingress_expiry: <now + 5min>,
    }

# Each message is inducted with IngressInductionCost::Free (no cycles deducted).
# After ingress_history_max_messages messages, all legitimate ingress to the
# subnet is rejected with IngressHistoryFull.
```

A deterministic integration test can be written using PocketIC: create an application subnet, submit `UpdateSettings` messages in a loop from an anonymous principal targeting any canister, and assert that after `ingress_history_max_messages` submissions, a legitimate ingress message from a different principal is rejected with `IngressHistoryFull`.
