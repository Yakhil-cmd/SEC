### Title
Negative `delta_gas` Silently Discarded in `compute_gas_refund` Causes User Overcharge — (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, a signed `i64` value `delta_gas` is computed to reconcile EVM gas consumption with native resource consumption. Only the positive branch is acted upon; the negative branch is silently discarded. This is structurally identical to the external report's root cause: a signed quantity is coerced to an unsigned context by ignoring its sign, causing callers to receive an incorrect (too-high) `gas_used` value and therefore an incorrect (too-small) refund.

---

### Finding Description

`compute_gas_refund` computes:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [1](#0-0) 

`delta_gas` is the difference between the gas implied by native resource consumption and the gas already charged from ergs. It is a signed quantity:

- **`delta_gas > 0`**: native consumption implies more gas than was charged → `gas_used` is increased (user pays more). This branch is handled.
- **`delta_gas < 0`**: native consumption implies less gas than was charged → `gas_used` should be decreased (user gets a larger refund). This branch is **silently discarded**. The `// TODO: return delta_gas to gas_used?` comment explicitly acknowledges the omission.

The result is that `gas_used` is never reduced below the ergs-derived value, even when native resource consumption is lower than what the gas consumption would imply. The final refund is computed as:

```rust
let total_gas_refund = gas_limit - gas_used;
``` [2](#0-1) 

Because `gas_used` is inflated (negative `delta_gas` not applied), `total_gas_refund` is deflated, and the user is overcharged.

The `RefundInfo` struct carrying `gas_used` is then used directly to compute the token refund and operator payment in both the L2 flow: [3](#0-2) 

and the L1 flow: [4](#0-3) 

The sign information of `delta_gas` is discarded in exactly the same way the external report describes `getExcessCollateral` discarding the `isExcess` boolean: the magnitude is used but the direction is lost.

---

### Impact Explanation

When `delta_gas < 0` — i.e., when EVM gas consumption exceeds what native resource consumption would imply — the user is charged more gas than they should be. The refund paid back to the sender is smaller than correct, and the operator receives excess fees. This constitutes a direct loss of funds for the transaction sender. The magnitude of overcharge is `|delta_gas| * gas_price` per affected transaction.

The condition `delta_gas < 0` arises when `gas_used > native_used / native_per_gas`. Since `native_per_gas = gas_price / native_price`, a high gas price (e.g., high priority fee) lowers `native_per_gas`, making `native_used / native_per_gas` larger and `delta_gas` less likely to be negative. Conversely, at low gas prices or for transactions that are EVM-gas-heavy but proving-cheap (e.g., many arithmetic opcodes, few storage writes), `delta_gas < 0` is reachable by any unprivileged sender.

---

### Likelihood Explanation

Any L2 or L1 transaction processed under the `#[cfg(not(feature = "unlimited_native"))]` path can trigger this. The condition requires `gas_used > native_used / native_per_gas`, which is reachable without any privileged access. A sender crafting a transaction with high EVM gas consumption but low proving complexity (e.g., tight arithmetic loops, few storage writes) can reliably land in the negative-`delta_gas` regime. The TODO comment confirms the developers are aware the negative branch is unhandled.

---

### Recommendation

Apply the negative `delta_gas` correction symmetrically. When `delta_gas < 0`, reduce `gas_used` by `(-delta_gas) as u64`, clamped to not go below `minimal_gas_used`:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This mirrors the documented intent in `docs/double_resource_accounting.md`:

> If `deltaGas > 0`, we add it to `gasUsed`… [5](#0-4) 

The symmetric negative case should reduce `gas_used` and return the difference to the user as a larger refund.

---

### Proof of Concept

1. Deploy a contract that executes a tight arithmetic loop (e.g., 10 000 iterations of `ADD`/`MUL`) — high EVM gas, low proving native cost.
2. Submit an L2 transaction calling this contract with `gas_limit` set to exactly cover the loop.
3. After execution, observe `gas_used` in the `ZkTxResult`. It equals the ergs-derived value.
4. Compute `expected_gas_used = native_used / native_per_gas`. If `expected_gas_used < gas_used`, the user was overcharged `(gas_used - expected_gas_used) * gas_price` tokens.
5. Confirm the operator balance increased by more than `expected_gas_used * gas_price` and the sender's refund is smaller than `(gas_limit - expected_gas_used) * gas_price`.
