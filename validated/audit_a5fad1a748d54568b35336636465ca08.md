### Title
ERC-20 Tokens Permanently Burned Without Re-minting When `ft_transfer_call` Partially Refunds in the Omni Exit Path - (`engine-precompiles/src/native.rs`)

---

### Summary

When a user calls `withdrawToNear` on an `EvmErc20` contract using the Omni message path, the ERC-20 tokens are burned atomically on Aurora before the cross-contract `ft_transfer_call` is dispatched to the NEP-141 contract. If the NEAR-side receiver's `ft_on_transfer` returns a non-zero unused amount, the NEP-141 standard refunds those tokens back to Aurora's NEP-141 balance. However, no callback exists to re-mint the corresponding ERC-20 tokens for the returned amount. The burned ERC-20 tokens are permanently lost, and the returned NEP-141 tokens become orphaned in Aurora's account.

---

### Finding Description

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` handles three exit sub-paths for ERC-20 tokens: legacy `ft_transfer`, wNEAR unwrap, and Omni (`ft_transfer_call`). The Omni path is triggered when the recipient string contains a `:` separator followed by a JSON message (e.g., `some_protocol.near:{"recipient":"eth:...","fee":"0"}`).

**Step 1 – ERC-20 burn (irreversible, on-chain):**

In `EvmErc20.sol`, `withdrawToNear` burns the full `amount` from the caller before calling the precompile:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // <-- tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, ...)
    }
}
``` [1](#0-0) 

**Step 2 – Precompile dispatches `ft_transfer_call` with no partial-refund callback:**

In `exit_erc20_token_to_near`, the Omni branch sets `transfer_near_args = None` and uses `ft_transfer_call` as the method:

```rust
Some(Message::Omni(msg)) => (
    nep141_account_id,
    ft_transfer_call_args(&exit_params.receiver_account_id, exit_params.amount, msg)?,
    "ft_transfer_call",
    None,   // <-- transfer_near_args is None
    ...
),
``` [2](#0-1) 

Back in the precompile's `run` method, the promise is constructed. Because `transfer_near_args` is `None` and (without the `error_refund` feature) `refund` is also `None`, `callback_args` equals the default value, so **no callback is attached**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // No callback!
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [3](#0-2) 

**Step 3 – NEP-141 standard returns unused tokens to Aurora:**

Per the NEP-141 `ft_transfer_call` standard, if the receiver's `ft_on_transfer` returns a non-zero unused amount, `ft_resolve_transfer` credits those tokens back to the sender (Aurora's account). Aurora's NEP-141 balance increases, but no ERC-20 tokens are re-minted.

**Step 4 – Even with `error_refund` feature, partial refunds are not handled:**

When `error_refund` is compiled in, a callback (`exit_to_near_precompile_callback`) is attached. However, this callback only re-mints ERC-20 tokens if the promise **failed entirely**. If `ft_transfer_call` succeeds (even partially), the callback takes the success branch and does nothing for the returned amount:

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    if let Some(args) = args.transfer_near {
        // Only transfers NEAR for wNEAR unwrap path
    }
    None   // <-- no re-mint for partial refund
} else if let Some(args) = args.refund {
    // Only reached on full failure
    engine::refund_on_error(io, env, state, &args, handler)?;
    ...
};
``` [4](#0-3) 

The `refund_on_error` function re-mints ERC-20 tokens only when called, which only happens on full promise failure: [5](#0-4) 

---

### Impact Explanation

**Impact: High – Permanent freezing/loss of user funds.**

- The user's ERC-20 tokens are burned atomically and irreversibly before the NEAR-side promise executes.
- If the NEAR receiver returns any amount as unused (a standard NEP-141 behavior), those NEP-141 tokens accumulate in Aurora's account with no corresponding ERC-20 tokens.
- The user permanently loses the returned portion of their ERC-20 tokens. There is no recovery path: the ERC-20 tokens are gone, and the orphaned NEP-141 tokens in Aurora's account cannot be claimed by the user.
- This inflates Aurora's NEP-141 balance relative to its ERC-20 supply, creating a persistent accounting mismatch in the bridge.

---

### Likelihood Explanation

**Likelihood: Medium.**

The Omni path is the production path for bridging ERC-20 tokens to NEAR DeFi protocols (e.g., Omni Bridge, lending protocols, DEXes). Any NEAR protocol that:
- Applies a fee and returns the remainder,
- Has a deposit cap and rejects the excess,
- Partially fills an order,

...will trigger this bug. This is standard, documented NEP-141 behavior. No attacker action is required; a normal user interacting with a legitimate NEAR DeFi protocol via the Omni path is sufficient to trigger the loss.

---

### Recommendation

For the Omni `ft_transfer_call` path, attach a callback that reads the promise result's returned unused amount and re-mints the corresponding ERC-20 tokens to the original caller. Concretely:

1. Extend `ExitToNearPrecompileCallbackArgs` with an optional `partial_refund` field carrying the ERC-20 address and the original caller's address.
2. In the Omni branch of `exit_erc20_token_to_near`, always attach a callback (similar to the wNEAR unwrap path).
3. In `exit_to_near_precompile_callback`, on success, parse the promise result to extract the unused amount returned by `ft_on_transfer`, and call `refund_on_error` (or an equivalent re-mint) for that amount.

This mirrors the fix applied in Seaport PR #839: ensuring the actual consumed amount is reconciled against what was committed.

---

### Proof of Concept

1. User holds 1,000 ERC-20 (e.g., USDT mirror) tokens on Aurora.
2. User calls `erc20.withdrawToNear("omni_protocol.near:{\"recipient\":\"...\",\"fee\":\"0\"}", 1000)`.
3. `EvmErc20.withdrawToNear` burns 1,000 ERC-20 tokens from the user. [6](#0-5) 
4. `ExitToNear` precompile schedules `ft_transfer_call(receiver=omni_protocol.near, amount=1000, msg=...)` with no callback. [7](#0-6) 
5. `omni_protocol.near`'s `ft_on_transfer` processes 800 tokens and returns `200` as unused.
6. NEP-141's `ft_resolve_transfer` credits 200 NEP-141 tokens back to Aurora's account.
7. No callback fires to re-mint 200 ERC-20 tokens. [8](#0-7) 
8. **Result**: User has lost 200 ERC-20 tokens permanently. Aurora's NEP-141 balance has 200 orphaned tokens with no corresponding ERC-20 supply. The bridge is now insolvent by 200 units for this token.
