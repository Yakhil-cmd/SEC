Audit Report

## Title
`can_auto_finalize` Permanently Blocks SNS Swap Auto-Finalization When `already_tried_to_auto_finalize` Is `None` - (File: rs/sns/swap/src/swap.rs)

## Summary
In `rs/sns/swap/src/swap.rs`, the `can_auto_finalize` function uses `self.already_tried_to_auto_finalize.unwrap_or(true)` to guard against repeated auto-finalization. When the optional protobuf field is `None` — which occurs for any swap canister whose persisted state predates the introduction of this field — the guard evaluates to `true` and permanently blocks auto-finalization, even though no attempt has ever been made. The caller in `run_periodic_tasks` silently discards the error on every heartbeat, leaving the swap permanently stuck in `COMMITTED` or `ABORTED` until a manual `finalize` call is made.

## Finding Description
`can_auto_finalize` is defined at `rs/sns/swap/src/swap.rs` lines 2916–2944. The critical guard is at line 2936:

```rust
if self.already_tried_to_auto_finalize.unwrap_or(true) {
    return Err(format!(
        "self.already_tried_to_auto_finalize is {:?}, indicating that an attempt \
         has already been made to auto-finalize. ...",
        self.already_tried_to_auto_finalize
    ));
}
```

The field `already_tried_to_auto_finalize` is declared as `Option<bool>` in the generated protobuf struct (`rs/sns/swap/src/gen/ic_sns_swap.pb.v1.rs`, lines 227–228). For any swap canister whose stable state was serialized before this field was introduced, the field deserializes as `None`. `unwrap_or(true)` then returns `true`, causing the function to return an error claiming auto-finalization was already attempted — even though it never was.

The caller in `run_periodic_tasks` (line 1059) silently discards this error:

```rust
else if self.can_auto_finalize().is_ok() {
    // ... attempt auto-finalization
}
```

The comment at lines 1056–1058 explicitly states errors are discarded to avoid log spam. As a result, on every heartbeat, the swap silently skips auto-finalization with no observable signal to operators or participants.

The field is initialized to `Some(false)` only when a swap is opened (confirmed by the test assertion at `rs/sns/swap/tests/swap.rs` line 1132). Any swap that was already in a terminal lifecycle state (`COMMITTED` or `ABORTED`) at the time of canister upgrade will retain `None` for this field, triggering the bug on every subsequent heartbeat.

## Impact Explanation
Any SNS swap canister upgraded from a version that did not include `already_tried_to_auto_finalize` will never auto-finalize after reaching `COMMITTED` or `ABORTED`. SNS token buyers cannot receive their SNS neurons automatically, and ICP refunds are not issued automatically. The swap is permanently stuck until an external caller manually invokes `finalize`. This constitutes a concrete governance/lifecycle denial-of-service for the SNS auto-finalization path, matching the allowed impact: **High ($2,000–$10,000) — Significant SNS security impact with concrete user or protocol harm.**

## Likelihood Explanation
No attacker action is required. The condition is triggered automatically on every heartbeat by `run_periodic_tasks` for any swap canister that was live before the `already_tried_to_auto_finalize` field was introduced and subsequently upgraded. IC canister upgrades preserve existing stable state, and protobuf optional fields absent from the serialized state deserialize as `None`. The entry path requires no special privilege; the periodic task fires unconditionally.

## Recommendation
Change the default from `true` to `false` at `rs/sns/swap/src/swap.rs` line 2936:

```diff
- if self.already_tried_to_auto_finalize.unwrap_or(true) {
+ if self.already_tried_to_auto_finalize.unwrap_or(false) {
```

`None` semantically means the field was never written, which is equivalent to "not yet attempted." The conservative default must be `false` so that swaps with absent state are permitted to auto-finalize rather than being permanently blocked.

## Proof of Concept
1. Deploy an SNS swap canister from a version that does not include the `already_tried_to_auto_finalize` field.
2. Allow the swap to reach `COMMITTED` or `ABORTED` state.
3. Upgrade the canister to the current version. The field remains `None` in persisted state.
4. Observe that `run_periodic_tasks` fires on every heartbeat but `can_auto_finalize()` always returns `Err("self.already_tried_to_auto_finalize is None, indicating that an attempt has already been made...")`.
5. Confirm the swap never auto-finalizes; participants must wait for a manual `finalize` call.

A targeted unit test can reproduce this by constructing a `Swap` struct with `already_tried_to_auto_finalize: None`, setting lifecycle to `Committed`, and asserting that `can_auto_finalize()` returns `Err` — which it does, confirming the bug. The existing test suite at `rs/sns/swap/tests/swap.rs` lines 1131–1139 only asserts the `Some(false)` case and provides no coverage for the `None` path.
