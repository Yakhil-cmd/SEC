### Title
Silo Mode `Address` Whitelist Applied to Exit/Withdrawal Transactions Causes Permanent Fund Freeze — (File: `engine/src/contract_methods/silo/mod.rs`)

---

### Summary

In Silo mode, the `is_allow_submit` whitelist check is applied uniformly to **all** EVM transactions submitted via the `submit` entrypoint — including those that call the `ExitToNear` and `ExitToEthereum` precompiles to withdraw user funds. If a user's EVM address is removed from the `Address` whitelist after they have deposited ETH or ERC-20 tokens into Aurora, they have no path to withdraw their funds, resulting in a permanent fund freeze.

---

### Finding Description

`is_allow_submit` in `engine/src/contract_methods/silo/mod.rs` gates every EVM transaction on the `Address` whitelist: [1](#0-0) 

The inner `is_address_allowed` function returns `false` for any address not present in the enabled `Address` whitelist: [2](#0-1) 

This check is applied at the `submit` entrypoint level, blocking **all** EVM transactions from non-whitelisted addresses — including calls to the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) and `ExitToEthereum` precompile (`0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`): [3](#0-2) 

The `receive_erc20_tokens` function does have a fallback address mechanism for **incoming** tokens to non-whitelisted addresses: [4](#0-3) 

However, there is **no analogous fallback or exception** for **outgoing** exit/withdrawal transactions. A user whose address is removed from the `Address` whitelist after depositing funds has no alternative path to recover them.

The whitelist is mutable at any time by the owner: [5](#0-4) 

The existing test `test_submit_with_removing_entries` explicitly demonstrates that removing an address from the whitelist causes subsequent `submit` calls to fail with `NotAllowed`, confirming the blocking behavior: [6](#0-5) 

---

### Impact Explanation

Any ETH (base token) or ERC-20 token balance held by an EVM address that is subsequently removed from the `Address` whitelist becomes permanently unrecoverable. The user cannot call `ExitToNear` or `ExitToEthereum` because the `submit` entrypoint rejects their transaction before the EVM even executes. There is no admin-callable rescue function analogous to the `revokePayout` fix in the referenced report. Impact: **Critical — permanent freezing of funds**.

---

### Likelihood Explanation

Silo mode with an active `Address` whitelist is a documented, production-intended configuration. Whitelist membership changes are routine operational events (compliance removal, key rotation, account migration). A user who deposited funds while whitelisted and is later removed — even for legitimate reasons — loses access to their funds permanently. No malicious intent is required. Likelihood: **Medium**.

---

### Recommendation

1. Introduce an exception in the `submit` whitelist check: if the EVM transaction's `to` address is `ExitToNear::ADDRESS` or `ExitToEthereum::ADDRESS`, bypass the `Address` whitelist check so users can always withdraw their own funds.
2. Alternatively, add an admin-callable rescue function (analogous to `revokePayout`) that can transfer a frozen address's EVM balance to a designated recovery address.
3. Document clearly that enabling the `Address` whitelist and removing addresses will freeze any existing balances held by those addresses.

---

### Proof of Concept

1. Deploy Aurora Engine in Silo mode; enable the `Address` whitelist via `set_whitelist_status`.
2. Add `user_address` to the `Address` whitelist via `add_entry_to_whitelist`.
3. `user_address` deposits ETH into Aurora (e.g., via the ETH connector bridge), acquiring a non-zero EVM balance.
4. Owner calls `remove_entry_from_whitelist` for `user_address` (legitimate compliance action).
5. `user_address` submits an EVM transaction calling `ExitToNear` (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) to withdraw their ETH.
6. `is_allow_submit` → `is_address_allowed` returns `false` → transaction rejected with `NotAllowed`.
7. `user_address`'s ETH balance remains in Aurora with no withdrawal path. Funds are permanently frozen. [7](#0-6) [8](#0-7)
