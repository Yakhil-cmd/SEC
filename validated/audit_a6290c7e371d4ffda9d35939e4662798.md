## Title
Panic in `burn_from` Callback After ckETH Burn Leaves User Funds Permanently Lost Without Reimbursement in `withdraw_erc20` Multi-Canister Flow - (File: `rs/ethereum/cketh/minter/src/ledger_client.rs`)

## Summary

The `burn_from` helper in `ledger_client.rs` unconditionally `panic!`s on four `TransferFromError` variants (`BadFee`, `TooOld`, `CreatedInFuture`, `Duplicate`). In the `withdraw_erc20` flow, the ckETH gas-fee burn (Step 1) is finalized before the ckERC20 burn (Step 2) is attempted. If the ckERC20 ledger returns any of these variants during Step 2, the minter's reply callback traps, rolling back all in-callback state mutations but leaving the already-committed ckETH burn irreversible. The `EventType::FailedErc20WithdrawalRequest` entry is never written, so `process_reimbursement` has no record to act on and the user's ckETH is permanently destroyed.

## Impact Explanation

This is a **chain-fusion ck-token accounting conservation bug**: ckETH supply decreases (burn recorded on the ckETH ledger) with no corresponding Ethereum transaction sent and no reimbursement issued. The user's ckETH is permanently destroyed. This matches the allowed impact: *"Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm"* â€” **High ($2,000â€“$10,000)**.

## Likelihood Explanation

The four panicking variants are suppressed under a correct ICRC-2 implementation because `fee: None` and `created_at_time: None` are passed. However, the precondition is realistic:

1. A ckERC20 ledger upgrade that introduces a regression in fee validation could return `BadFee`.
2. A community-deployed ckERC20 token added via NNS governance with a non-standard ICRC-2 implementation could return any of these variants.
3. Once such a ledger is registered, any unprivileged user calling `withdraw_erc20` with that token triggers the bug â€” no special attacker capability is required beyond a normal withdrawal call.

The trigger is constrained (requires a non-conforming registered ledger) but is not purely theoretical given the open ckERC20 token registration process.

## Recommendation

Replace all four `panic!("BUG: ...")` branches in `burn_from` (`ledger_client.rs` lines 100â€“130) with recoverable `LedgerBurnError::TemporarilyUnavailable` variants. This ensures unexpected ledger responses propagate as `Err(LedgerBurnError::...)` to `withdraw_erc20`, which then records `EventType::FailedErc20WithdrawalRequest` and schedules reimbursement. As a defense-in-depth measure, adopt the `scopeguard` pattern already used in `withdraw.rs` lines 70â€“72: install a guard immediately after the ckETH burn succeeds that records `FailedErc20WithdrawalRequest` if the subsequent callback traps before reaching the normal error-handling branch.
