Audit Report

## Title
XNet Peer Selection Permanently Biased via RTT EMA Manipulation Through Header-Only Responses - (File: rs/xnet/payload_builder/src/proximity.rs)

## Summary
`XNetClientImpl::query()` records the RTT observation immediately after receiving HTTP response headers, before reading the response body. A malicious node operator on a remote subnet can send headers instantly (~1 µs) while stalling or dropping the body, driving its EMA to the minimum clamp (1,000 ns) and its selection weight to the maximum (1,000,000,000). This creates up to a 100,000× selection bias over honest nodes, allowing the malicious node to monopolize XNet pulls and stall cross-subnet message delivery indefinitely.

## Finding Description

In `XNetClientImpl::query()`, `observe_roundtrip_time` is called at [1](#0-0)  immediately after `http_client.get()` resolves (i.e., after response headers are received), but before the body is consumed at [2](#0-1) .

The RTT is clamped to `[1_000 ns, 1_000_000_000 ns]` in `observe_roundtrip_time`: [3](#0-2) 

The EMA is updated as `(*ema * 9 + duration_nanos) / 10`: [4](#0-3) 

The weight formula is `1_000 * NANOS_PER_SEC / ema_nanos`: [5](#0-4) 

A malicious node that sends headers in ~1 µs and then stalls the body will have its RTT clamped to 1,000 ns → weight = 1,000,000,000. An honest node with 100 ms RTT has weight = 10,000. The 100,000× disparity is applied on every call to `pick_node()` at [6](#0-5) .

The 5-second `tokio::time::timeout` wrapping the entire async block does not prevent the attack: the RTT observation is committed before the body stall triggers the timeout. [7](#0-6) 

The existing `mean_weight` fallback for unobserved nodes does not help — the malicious node has an explicit, extremely high weight that dominates the weighted random selection in `pick_node()`. [8](#0-7) 

The test `pick_node_extreme_roundtrip_times` explicitly validates that a 1 µs RTT operator is selected 1,000,000× more often than a 1 s RTT operator, confirming the extreme weight disparity is by design and fully reachable: [9](#0-8) 

## Impact Explanation

Once the malicious node's EMA is poisoned, it is selected for nearly all XNet pulls from the victim subnet. It can then serve `204 No Content` on every pull, leaving the victim subnet's XNet slice pool for the remote subnet empty. No XNet messages from the remote subnet are included in the victim subnet's blocks, stalling cross-subnet message delivery indefinitely. This is a persistent, self-reinforcing subnet availability impact on XNet communication — matching **High ($2,000–$10,000): Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS**.

## Likelihood Explanation

The attacker requires control of exactly one registered node on the target remote subnet. Node operators are permissioned via NNS governance, but the IC has hundreds of node operators across many subnets. A single compromised or malicious node operator is a realistic threat explicitly within the "protocol peer below the consensus fault threshold" threat model. No threshold corruption, DNS hijack, privileged key, or user interaction is required. The attack is fully automated and repeatable.

## Recommendation

1. **Move `observe_roundtrip_time` after successful body read** — only record RTT when a complete, valid response body is received, so header-only fast-path manipulation does not affect the EMA.
2. **Record a penalty RTT on body failure or timeout** — treat `BodyReadError` and `Timeout` as a high-RTT observation (e.g., 1 s) to penalize nodes that respond with headers but fail to deliver content.
3. **Cap the maximum weight ratio** — enforce a floor on node weights (e.g., no node may have weight less than `max_weight / 1000`) to bound the maximum selection bias regardless of RTT spread.
4. **Apply a minimum selection probability** — guarantee every registered node on a subnet has at least a small baseline probability of being selected, preventing complete monopolization.

## Proof of Concept

**Setup**: Attacker controls node `N` (operator `OP_A`) on subnet B. Subnet A's XNet payload builder pulls streams from subnet B.

**Step 1 — Poison the EMA**: Node `N` serves XNet HTTP requests by sending response headers with status 200 in ~1 µs, then immediately closes the connection without a body. `XNetClientImpl::query()` records RTT ≈ 1 µs (clamped to 1,000 ns) at [1](#0-0) , then fails with `BodyReadError` at [10](#0-9) .

**Step 2 — EMA converges**: After ~10 such interactions, `OP_A`'s EMA ≈ 1,000 ns → weight = `1_000 * 1_000_000_000 / 1_000 = 1_000_000_000`. Honest nodes on subnet B with 100 ms RTT have weight = `10,000`.

**Step 3 — Monopolize selection**: With one malicious node (weight 1,000,000,000) and two honest nodes (weight 10,000 each), the malicious node is selected with probability ≈ 98% on every call to `pick_node()`. [11](#0-10) 

**Step 4 — Stall XNet**: Node `N` serves `204 No Content` on every pull. Subnet A's slice pool for subnet B remains empty. Cross-subnet message delivery stalls indefinitely.

**Verification**: The existing unit test `pick_node_extreme_roundtrip_times` at [12](#0-11)  directly confirms the 1,000,000× selection bias between minimum and maximum EMA nodes, validating the extreme weight disparity is reachable and deterministic.
