Audit Report

## Title
Unguarded `wait_for` After `assemble_artifact` Resolves Causes Permanent Task and Pool Leak — (`rs/p2p/consensus_manager/src/receiver.rs`)

## Summary

In `process_slot_update`, after the outer `select!` commits to the `assemble_artifact` branch, both the `AssembleResult::Done` and `AssembleResult::Unwanted` arms call `peer_rx.wait_for(|p| p.is_empty()).await` with no cancellation guard. A Byzantine peer below the consensus fault threshold that advertises an artifact and never retracts it causes this `wait_for` to block permanently, preventing the `Remove` mutation from being sent, leaking the assemble task, and leaving the `active_assembles` entry permanently allocated.

## Finding Description

`process_slot_update` constructs an outer `select!` with three branches: `cancellation_token.cancelled()`, `assemble_artifact`, and `all_peers_deleted_artifact`. [1](#0-0) 

Once `assemble_artifact` wins the race, execution is committed to its match arm. The cancellation token and `all_peers_deleted_artifact` futures are dropped. Inside the `Done` arm, `Insert` is sent to the unvalidated pool, then: [2](#0-1) 

This `wait_for` is **not** wrapped in a `select!` with the cancellation token. The identical pattern exists in the `Unwanted` arm: [3](#0-2) 

Both are annotated `// TODO: NET-1774`, confirming developer awareness. The only ways `wait_for` can resolve are:

1. The advertising peer overwrites its slot with a different artifact ID, triggering `sender.send_if_modified(|h| h.remove(peer_id))` in `handle_slot_update_receive` — a Byzantine peer will not do this.
2. The peer leaves the subnet topology, handled in `handle_topology_update` — a peer below the fault threshold remains in the topology indefinitely.
3. The `watch::Sender` in `active_assembles` is dropped — this only occurs in `handle_artifact_processor_joined` at `self.active_assembles.remove(&id)`, which only runs after the task joins, which never happens because the task is blocked. [4](#0-3) [5](#0-4) 

The result is a permanent deadlock: the task cannot join because it is blocked on `wait_for`; `active_assembles.remove` cannot fire because the task has not joined.

## Impact Explanation

This matches the High bounty impact: **Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS.**

- **Unvalidated pool leak**: Every artifact assembled from a Byzantine advertisement that is never retracted stays in the unvalidated pool. Consensus repeatedly re-evaluates it, wasting CPU each round.
- **Assemble task leak**: Each blocked task permanently occupies a `JoinSet` slot, a `watch::Receiver`, and a `Sender` channel reference.
- **`active_assembles` map leak**: The `watch::Sender` entry is never removed, so the map grows without bound relative to Byzantine peers × `slot_limit`.

The impact scales linearly with the number of Byzantine peers (up to the fault threshold `f`) and `slot_limit`, making it a viable sustained resource-exhaustion vector against subnet replicas. The `debug_assert` invariant at line 270–275 (`active_assembles.len() == artifact_processor_tasks.len()`) will also permanently diverge from the expected state in release builds. [6](#0-5) 

## Likelihood Explanation

The attacker is a subnet peer operating below the consensus fault threshold — the standard Byzantine adversary model. The exploit requires only two steps: (1) send a slot update advertising an artifact, and (2) never send another slot update for that slot. No cryptographic break, no majority corruption, no privileged access, and no special network conditions are required. The attack is trivially repeatable and can be sustained indefinitely for each slot up to `slot_limit`.

## Recommendation

Wrap both `wait_for` calls in a `select!` that also listens to the cancellation token:

```rust
select! {
    _ = cancellation_token.cancelled() => {}
    _ = peer_rx.wait_for(|p| p.is_empty()) => {}
}
```

`cancellation_token` must be threaded into the `process_slot_update` function (it is already a parameter). This ensures that on node shutdown or when the artifact becomes irrelevant, the task unblocks and — in the `Done` case — sends `Remove` before exiting. The NET-1774 ticket should be treated as a security-relevant fix.

## Proof of Concept

Minimal state-machine test (no network required, uses existing test infrastructure):

1. Build a `ConsensusManagerReceiver` with a `MockArtifactAssembler` that immediately returns `AssembleResult::Done`.
2. Call `handle_slot_update_receive` for `NODE_1`, slot 1, artifact ID 0.
3. Receive `Insert(artifact_0)` from `channels.unvalidated_artifact_receiver`.
4. Do **not** call `handle_slot_update_receive` again for `NODE_1` slot 1 (Byzantine peer never retracts).
5. Assert via `tokio::time::timeout(Duration::from_millis(500), channels.unvalidated_artifact_receiver.recv())` that `Remove(0)` is **never** received.
6. Assert `mgr.artifact_processor_tasks.len() == 1` (task permanently blocked).
7. Assert `mgr.active_assembles.len() == 1` (entry never cleaned up).

This directly demonstrates that the `Remove` mutation is withheld indefinitely and both the task and `active_assembles` entry leak, violating the pool purge invariant.
