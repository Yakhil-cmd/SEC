### Title
Unverified Oracle Advice in Modexp Delegation Enables Forward/Proving Divergence - (File: `basic_system/src/system_functions/modexp/delegation/bigint.rs`)

### Summary
In proving mode (RISC-V target), the `OracleAdvisor::get_reduction_op_advice` function accepts oracle-provided quotient and remainder values for big integer division without verifying their mathematical correctness. A malicious prover can supply wrong values within the accepted length bounds, causing the modexp precompile to return an incorrect result in proving mode while the forward execution returns the correct result — a forward/proving divergence.

### Finding Description
The modexp precompile, when compiled for the RISC-V proving target, delegates each big-integer division step to the oracle via `OracleAdvisor::get_reduction_op_advice`. [1](#0-0) 

The oracle returns two values: `q_len` (quotient word count) and `r_len` (remainder word count), followed by the actual quotient and remainder digits. The code performs only an upper-bound length check: [2](#0-1) 

The comment at line 877 reads: *"We assume that oracle's response is well-formed lengths-wise, and we will check value-wise separately."* However, no value-wise check exists anywhere in the function or in the calling code. The `write_bigint` helper blindly writes the oracle-supplied digits into `quotient_dst` and `remainder_dst` without verifying the invariant `quotient × modulus + remainder == dividend`. [3](#0-2) 

The oracle interface documentation explicitly states that oracle responses are untrusted and **must** be validated by the calling code: [4](#0-3) 

The `OracleAdvisor` is gated to the proving target only: [5](#0-4) 

In forward mode the native `::modexp::modexp` library is used instead, so the forward path is always correct: [6](#0-5) 

### Impact Explanation
A malicious prover controls the oracle responses fed to the RISC-V execution. By supplying quotient = 0 and remainder = 0 (both within the accepted length bounds of 0 words) for every division step, the prover causes every modular reduction to return 0, making the modexp precompile return 0 for any input. More generally, the prover can return any value within the length bounds. The Airbender prover then generates a valid ZK proof of this incorrect execution. The verifier accepts the proof because the RISC-V execution trace is internally consistent with the supplied oracle hints. The forward execution (sequencer) and the proving execution (prover) diverge on the modexp result, allowing the prover to commit an incorrect state root that the verifier cannot distinguish from a correct one. Any smart contract relying on modexp for cryptographic verification (RSA, Diffie-Hellman, signature schemes) is affected.

### Likelihood Explanation
Exploitation requires a malicious prover who controls the oracle responses in proving mode. The Immunefi scope for ZKsync OS explicitly lists **prover/forward execution input** as a valid attacker profile. The prover role is not a leaked key or governance majority — it is a defined system participant whose inputs flow directly into the proving pipeline. Likelihood: **2/5** (requires a compromised or malicious prover, but no additional preconditions once that role is held).

### Recommendation
After receiving `q_len`, `r_len`, and the digit data from the oracle, verify the division identity before writing to the destination buffers:

```
assert_eq!(quotient * modulus + remainder, dividend,
    "Oracle-provided division advice is incorrect");
```

This check must be performed using the same big-integer arithmetic already available in the codebase, ensuring that a malicious prover cannot produce a valid proof of incorrect modexp output.

### Proof of Concept
1. A transaction calls the modexp precompile (address `0x05`) with inputs `base`, `exp`, `modulus`.
2. In proving mode, `modexp_as_system_function_inner` calls `delegation::modexp`, which calls `modexp_inner` with an `OracleAdvisor`.
3. For each division step `a / m`, `get_reduction_op_advice` queries the oracle.
4. The malicious prover returns `q_len = 0`, `r_len = 0` (both pass the length-bound assertions at lines 905–906 since `0 <= max_quotient_digits` and `0 <= max_remainder_digits`).
5. `write_bigint` writes zero digits into `quotient_dst` and `remainder_dst`, setting both to 0.
6. The modexp algorithm proceeds with `quotient = 0`, `remainder = 0` for every reduction, producing an incorrect final result (e.g., always 0 or 1).
7. The Airbender prover generates a valid proof of this execution.
8. The verifier accepts the proof; the incorrect modexp output is committed to the state root. [7](#0-6) [8](#0-7)

fn write_bigint(
    it: &mut impl ExactSizeIterator<Item = usize>,
    mut to_consume: usize,
    dst: &mut BigintRepr<impl Allocator + Clone>,
) {
    const {
        assert!(core::mem::size_of::<usize>() == core::mem::size_of::<u32>());
    }
    // NOTE: even if oracle overstates the number of digits (so - iterator length), it is not important
    // as long as caller checks that number of digits is within bounds of soundness
    unsafe {
        let num_digits = to_consume.next_multiple_of(8) / 8;
        let dst_capacity = dst.clear_as_capacity_mut();
        for dst in dst_capacity[..num_digits].iter_mut() {
            let dst: *mut u32 = dst.as_mut_ptr().cast::<[u32; 8]>().cast();
            for i in 0..8 {
                if to_consume > 0 {
                    to_consume -= 1;
                    let digit = it.next().unwrap();
                    dst.add(i).write(digit as u32);
                } else {
                    dst.add(i).write(0);
                }
            }
        }
        assert_eq!(to_consume, 0);
        dst.set_num_digits(num_digits);
    }

#[cfg(any(all(target_arch = "riscv32", feature = "proving"), test))]
impl<'a, O: IOOracle> ModexpAdvisor for OracleAdvisor<'a, O> {
    fn get_reduction_op_advice<A: Allocator + Clone>(
        &mut self,
        a: &BigintRepr<A>,
        m: &BigintRepr<A>,
        quotient_dst: &mut BigintRepr<A>,
        remainder_dst: &mut BigintRepr<A>,
    ) {
        let arg: ModExpAdviceParams = {
            let a_len = a.digits;
            let a_ptr = a.backing.as_ptr();

            let modulus_len = m.digits;
            let modulus_ptr = m.backing.as_ptr();

            assert!(modulus_len > 0);

            ModExpAdviceParams {
                op: 0,
                a_ptr: a_ptr.addr() as u32,
                a_len: a_len as u32,
                b_ptr: 0,
                b_len: 0,
                modulus_ptr: modulus_ptr.addr() as u32,
                modulus_len: modulus_len as u32,
            }
        };

        // We assume that oracle's response is well-formed lengths-wise, and we will check value-wise separately
        let mut it = self
            .inner
            .raw_query(
                MODEXP_ADVICE_QUERY_ID,
                &((&arg as *const ModExpAdviceParams).addr() as u32),
            )
            .unwrap();

        let q_len = it.next().expect("quotient length");
        let r_len = it.next().expect("remainder length");

        let max_quotient_digits = if a.digits < m.digits {
            0
        } else if a.digits == m.digits {
            1
        } else {
            a.digits + 1 - m.digits
        };

        let max_remainder_digits = m.digits;

        const {
            assert!(core::mem::size_of::<usize>() == core::mem::size_of::<u32>());
        }

        // check that hint is "sane" in upper bound

        assert!(q_len.next_multiple_of(8) / 8 <= max_quotient_digits);
        assert!(r_len.next_multiple_of(8) / 8 <= max_remainder_digits);

        write_bigint(&mut it, q_len, quotient_dst);
        write_bigint(&mut it, r_len, remainder_dst);

        assert!(it.next().is_none());
    }
