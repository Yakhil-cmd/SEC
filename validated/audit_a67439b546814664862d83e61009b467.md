### Title
Unclaimed Initialization: `new` Lacks Caller Restriction, Enabling Front-Run Takeover of `owner_id` - (File: `engine/src/contract_methods/admin.rs`)

---

### Summary

The Aurora Engine's `new` initialization function enforces a one-time-only guard (`ERR_ALREADY_INITIALIZED`) but performs **no check on who is calling it**. On NEAR Protocol, WASM deployment and contract initialization are always separate transactions. Any unprivileged NEAR account that observes the deployment block can race to call `new` first, supplying an attacker-controlled `owner_id`. Because `owner_id` gates every privileged action in the engine — including contract upgrades and full-access-key attachment — a successful front-run yields complete, irrecoverable control of the Aurora Engine account and all funds it holds.

### Finding Description

The exported `new` entry point in `engine/src/lib.rs` delegates to `contract_methods::admin::new`: [1](#0-0) 

The implementation is: [2](#0-1) 

The only guard is the state-existence check on line 57. There is no call to `env.predecessor_account_id()` and no comparison against `env.current_account_id()`. Any NEAR account can call `new` with arbitrary `NewCallArgs`, including an attacker-chosen `owner_id`, as long as the state key is absent.

`NewCallArgs` carries the fields that seed `EngineState`: [3](#0-2) 

`EngineState.owner_id` is the sole authorization token for every privileged method: [4](#0-3) 

On NEAR Protocol, deploying a WASM binary and calling an initialization function are **always two separate transactions**. The deployment transaction uploads the binary; the initialization transaction must follow in a subsequent block or even the same block but as a separate action. This window is observable by any participant watching the chain.

---

### Impact Explanation

An attacker who wins the race to call `new` becomes `owner_id`. With that role they can immediately:

1. **Deploy arbitrary replacement code** via `stage_upgrade` + `upgrade`, which issues a `DeployContract` promise on the engine account itself: [5](#0-4) 

2. **Attach a full access key** to the engine account via `attach_full_access_key`, giving the attacker unrestricted NEAR account control: [6](#0-5) 

3. **Redirect the ETH connector** to a malicious contract via `set_eth_connector_contract_account`, intercepting all bridged ETH deposits and withdrawals.

Any of these three paths leads to **direct theft of all user funds** held by or routed through the Aurora Engine (bridged ETH, NEP-141 tokens, NEAR staked for storage). The engine is the `aurora` mainnet account; the total value at risk is the entire TVL of the Aurora bridge.

---

### Likelihood Explanation

NEAR Protocol makes this window structural, not accidental:

- Every deployment of the Aurora Engine WASM — initial deployment or any future redeployment — leaves the contract uninitialized until `new` is called in a subsequent transaction.
- NEAR transactions are publicly visible in real time. An attacker running a NEAR node or watching the RPC can detect the deployment block and submit a competing `new` call in the same or next block.
- No special privilege, leaked key, or social engineering is required. The attacker only needs a funded NEAR account and knowledge of the `NewCallArgs` Borsh schema, which is public.
- The `upgrade` flow (owner calls `stage_upgrade` then `upgrade`) also re-deploys the WASM via a promise, but that path goes through `require_owner_only` first, so it is not itself vulnerable — the risk is confined to the initial `new` call window.

---

### Recommendation

Restrict `new` so it can only be called by the contract account itself (i.e., `env.predecessor_account_id() == env.current_account_id()`). On NEAR, a contract can call itself via a batch action that combines `DeployContract` and `FunctionCall` in a single atomic transaction, making deployment and initialization inseparable. The fix is:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    // Ensure only the contract itself (via a deploy+init batch) can initialize.
    if env.predecessor_account_id() != env.current_account_id() {
        return Err(b"ERR_NOT_ALLOWED".into());
    }
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // ... rest unchanged
}
```

The deployment script should be updated to use a NEAR batch transaction that atomically combines `DeployContract` and `FunctionCall { method: "new", ... }`, so no separate initialization transaction is ever needed.

---

### Proof of Concept

**Step 1 — Attacker observes deployment.**
The legitimate deployer submits a transaction that uploads the Aurora Engine WASM to the `aurora` account. This transaction is visible on-chain before `new` has been called; `state::get_state` returns `Err(NotFound)`.

**Step 2 — Attacker front-runs `new`.**
The attacker submits a NEAR transaction calling `aurora.new` with Borsh-encoded `NewCallArgsV2`:
```
chain_id:              [0u8; 32]          // any valid chain id
owner_id:              "attacker.near"    // attacker's account
upgrade_delay_blocks:  0                  // no delay on upgrades
```
Because there is no caller check in `new`, this succeeds and writes `EngineState { owner_id: "attacker.near", ... }` to storage. [7](#0-6) 

**Step 3 — Legitimate `new` call is rejected.**
When the deployer's own `new` transaction lands, `state::get_state(&io).is_ok()` is now `true`, and the call returns `ERR_ALREADY_INITIALIZED`. The deployer cannot recover ownership through `new`.

**Step 4 — Attacker escalates to full account takeover.**
With `owner_id = "attacker.near"`, the attacker calls `attach_full_access_key` with their own public key. `require_owner_only` passes, and a `AddFullAccessKey` promise is issued on the `aurora` account: [8](#0-7) 

The attacker now holds a full access key on the `aurora` account and can transfer all NEAR, drain bridged ETH, and replace the contract with arbitrary code. All user funds are lost.

#[named]
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())
}

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
}
