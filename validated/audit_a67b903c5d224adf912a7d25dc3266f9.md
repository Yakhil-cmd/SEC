Audit Report

## Title
Silent Burn Failure in `burn_and_log` Permanently Skips ICP Destruction After Successful Cycle Minting - (File: rs/nns/cmc/src/main.rs)

## Summary
The `burn_and_log` function in the Cycles Minting Canister is intentionally designed to swallow inter-canister call errors to the ICP ledger. When the ledger is transiently unavailable (e.g., during an NNS-governed upgrade), the burn silently fails while the primary action (cycles deposited, canister created, cycles minted) has already succeeded. Because the notification block is subsequently marked with a terminal `NotifiedCreateCanister`/`NotifiedTopUp`/`NotifiedMint` status, the user cannot retry, and the unburned ICP remains permanently stranded in the CMC's subaccount with no automated recovery path.

## Finding Description
The three public notification flows each follow the same pattern: perform the primary action, then call `burn_and_log` to destroy the backing ICP.

In `process_create_canister` (line 1943–1946), `process_mint_cycles` (line 1966–1969), and `process_top_up` (line 1999–2002), `burn_and_log` is awaited inside the process function. The process function returns `Ok(...)` regardless of whether the burn succeeded. The caller then unconditionally writes a terminal notification status:

```rust
// lines 1411-1419
with_state_mut(|state| {
    state.blocks_notified.insert(
        block_index,
        NotificationStatus::NotifiedCreateCanister(result.clone()),
    );
    if is_transient_error(&result) {
        state.blocks_notified.remove(&block_index);
    }
});
```

The `is_transient_error` guard only clears the status when `result` itself is an error. Because `process_create_canister` returns `Ok(canister_id)` even when `burn_and_log` fails, the block is permanently sealed as `NotifiedCreateCanister(Ok(...))`. Any subsequent call to `notify_create_canister` for the same block returns the cached success result without re-attempting the burn.

`burn_and_log` (lines 2017–2049) explicitly discards the ledger error:

```rust
Err((code, err)) => {
    let code = code as i32;
    print(format!("{msg} failed with code {code}: {err:?}"))
}
```

The comment at line 2014–2016 confirms this is intentional: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."* There is no persistent retry queue, no heartbeat-driven retry, and no administrative sweep endpoint in the CMC.

## Impact Explanation
Cycles are minted against ICP that is never destroyed. This breaks the ICP↔cycles economic invariant: the ICP total supply is not reduced as the protocol requires, while the corresponding cycles enter circulation. The unburned ICP is stranded in a CMC subaccount (keyed by controller or canister principal) with no automated recovery. This constitutes a concrete, repeatable NNS/ledger protocol harm — illegal minting of cycles without the corresponding ICP burn — matching the High impact class: *"Significant NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

## Likelihood Explanation
The ICP ledger is upgraded via NNS governance proposals on a regular cadence. During the upgrade window (typically seconds to a few minutes), any canister call to the ledger receives a `SysTransient` reject. Any unprivileged user who submits `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` during this window will trigger the condition. No special privilege, no victim mistake, and no coordination is required. The condition is repeatable across every ledger upgrade cycle.

## Recommendation
Replace the fire-and-forget pattern with a persistent retry mechanism. On ledger call failure in `burn_and_log`, store the `(subaccount, amount)` pair in stable state. Drive retries from the CMC's existing timer or heartbeat, or expose a permissionless `retry_burn` endpoint. This is the standard IC pattern for eventual consistency of cross-canister financial operations. At minimum, failed burn entries must be persisted so that a governance proposal can drain them, restoring the ICP supply invariant.

## Proof of Concept
1. User transfers ICP to the CMC's top-up subaccount for canister `C` and calls `notify_top_up` during a ledger upgrade window.
2. CMC calls `deposit_cycles(C, ...)` — succeeds; cycles are credited to `C`.
3. CMC calls `burn_and_log`; ledger returns `SysTransient`. `burn_and_log` logs the failure and returns `()`.
4. `process_top_up` returns `Ok(cycles)`.
5. Caller writes `NotificationStatus::NotifiedTopUp(Ok(cycles))` to `blocks_notified`; `is_transient_error` is false, so the entry is kept.
6. User retries `notify_top_up` — CMC returns the cached `Ok(cycles)` without re-attempting the burn.
7. Cycles are in canister `C`. ICP remains in the CMC's subaccount. ICP supply is not reduced.

A deterministic integration test can reproduce this by installing a mock ledger that rejects `send_pb` with a `SysTransient` code, calling `notify_top_up`, and asserting that (a) the returned result is `Ok`, (b) the block status is terminal, and (c) the CMC subaccount balance is unchanged.
