Audit Report

## Title
VIP Slot Bypass Broken in `try_borrow_slot`: Non-Privileged Callers Can Block Governance `update_canister_settings` — (`rs/nervous_system/clients/src/management_canister_client.rs`)

## Summary
`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` sets `used_slot_count = 0` for VIP callers (NNS canisters) to avoid consuming a slot, but the `available_slot_count == 0` guard at line 269 is unconditional and fires for VIP callers too. An unprivileged external caller can exhaust all 167 slots via concurrent `canister_status` calls, causing a subsequent governance-initiated `update_canister_settings` to return `Err(SysTransient)` even though governance is a VIP.

## Finding Description
In `try_borrow_slot` ( [1](#0-0) ), the VIP distinction only sets `used_slot_count = 0` at line 265, but the early-return guard at line 269 (`if *available_slot_count == 0`) is unconditional. A VIP caller with `used_slot_count = 0` still hits this guard and returns `Err((SysTransient, ...))` when the pool is empty.

The `canister_status` endpoint in the root canister has no access control ( [2](#0-1) ), so any external principal can call it. The shared pool is 167 slots ( [3](#0-2) ). An attacker sends 167 concurrent `canister_status` calls; each holds a slot for the full management canister round-trip. While all 167 slots are occupied, governance calls `update_canister_settings` ( [4](#0-3) ), which calls `new_management_canister_client()` with `is_caller_vip = true` ( [5](#0-4) ), but `try_borrow_slot` still returns `Err` because the pool is at 0.

The existing test (`test_limit_outstanding_calls`) does not catch this because each invocation of the `canister_status` helper creates its own isolated `thread_local` SLOTS counter ( [6](#0-5) ), so VIP and non-VIP calls never compete for the same pool. The test is structurally incapable of detecting the shared-pool bug.

`SlotLoan::drop` adds `used_slot_count` back to the pool ( [7](#0-6) ); for VIP callers `used_slot_count = 0`, so no slot is ever returned — confirming VIP callers are intended to be zero-cost, but the guard prevents them from proceeding at all when the pool is empty.

## Impact Explanation
A governance proposal executing `UpdateCanisterSettings` (or `change_canister_controllers`, `take_canister_snapshot`, `load_canister_snapshot`) will fail transiently while the slot pool is exhausted. This delays or prevents time-sensitive NNS configuration changes on NNS-controlled canisters. This matches the **High** bounty impact: "Application/platform-level DoS... or NNS... security impact with concrete user or protocol harm."

## Likelihood Explanation
The `canister_status` endpoint is intentionally public. The IC per-canister ingress queue supports ~500 concurrent messages; 167 is well within reach. Each call holds a slot for the management canister round-trip duration. An attacker can repeatedly send 167 concurrent calls to keep the pool continuously exhausted, making the attack window repeatable and sustainable with modest effort.

## Recommendation
Add a VIP bypass to the guard in `try_borrow_slot`:

```rust
if !self.is_caller_vip && *available_slot_count == 0 {
    let code = RejectCode::SysTransient as i32;
    return Err((code, "Unavailable. Maybe, try again later?".to_string()));
}
```

Additionally, fix the unit test to use a single shared `thread_local` SLOTS across all concurrent calls so the VIP-bypass behavior is actually validated under shared-pool exhaustion.

## Proof of Concept
A minimal unit test using a single shared `thread_local` SLOTS (capacity 2):
1. Spawn 2 non-VIP `canister_status` tasks that hold their slots (inner never resolves during the test window).
2. With both slots occupied, construct a `LimitedOutstandingCallsManagementCanisterClient` with `is_caller_vip = true` sharing the same SLOTS.
3. Call `update_settings` on the VIP client.
4. Assert the result is `Err((SysTransient, _))` — demonstrating VIP is incorrectly blocked.

This is a deterministic unit test requiring no network access, directly exercising the shared `available_slot_count` path in `try_borrow_slot`.

    fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
        let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

        self.available_slot_count
            .with_borrow_mut(|available_slot_count| {
                if *available_slot_count == 0 {
                    // This is somewhat of a lie, but is the best fit.
                    let code = RejectCode::SysTransient as i32;

                    let message = "Unavailable. Maybe, try again later?".to_string();

                    return Err((code, message));
                }

                *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
                Ok(())
            })?;

        let available_slot_count = self.available_slot_count;
        Ok(SlotLoan {
            available_slot_count,
            used_slot_count,
        })
    }
