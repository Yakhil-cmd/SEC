Audit Report

## Title
Missing Minimum Fee Rate Floor in `DogecoinFeeEstimator::estimate_nth_fee` Allows Near-Zero Fee Dogecoin Transactions - (File: `rs/dogecoin/ckdoge/minter/src/fees/mod.rs`)

## Summary

`DogecoinFeeEstimator::estimate_nth_fee` returns raw fee percentile values from the Dogecoin canister with no minimum floor, unlike `BitcoinFeeEstimator::estimate_nth_fee` which applies an explicit `minimum_fee_per_vbyte()` guard added after an anomalously-low fee event on 2025-06-21. During any period of anomalously low Dogecoin network fee percentiles, the ckDOGE minter will build, sign, and submit Dogecoin withdrawal transactions with near-zero fees that the Dogecoin network will not relay or confirm, resulting in users' ckDOGE being permanently burned on the IC ledger with no corresponding DOGE received.

## Finding Description

In `rs/bitcoin/ckbtc/minter/src/fees/mod.rs`, `BitcoinFeeEstimator::estimate_nth_fee` applies a hardcoded minimum floor at line 112:

```rust
median_fee.map(|f| f.max(self.minimum_fee_per_vbyte()))
```

where `minimum_fee_per_vbyte()` (lines 80–87) returns 1,500 millisat/vbyte on Mainnet, with an explicit code comment documenting the 2025-06-21 anomalous fee event as the motivation.

In `rs/dogecoin/ckdoge/minter/src/fees/mod.rs`, `DogecoinFeeEstimator::estimate_nth_fee` (lines 53–66) has no equivalent guard:

```rust
Network::Mainnet => {
    if fee_percentiles.len() < 100 || nth >= 100 {
        return None;
    }
    Some(fee_percentiles[nth])  // no minimum floor
}
```

The returned value flows through the shared `estimate_fee_per_vbyte` in `rs/bitcoin/ckbtc/minter/src/lib.rs` (lines 245–249), which stores it directly as `s.last_median_fee_per_vbyte` and computes `s.fee_based_retrieve_btc_min_amount` from it. Both values are then used by:

1. `estimate_withdrawal_fee` (main.rs lines 98–128), which reads `s.last_median_fee_per_vbyte` and passes it to `estimate_retrieve_doge_fee`, returning a misleadingly low fee estimate to users.
2. `retrieve_doge_with_approval` (main.rs lines 130–143), callable by any non-anonymous user, which triggers `retrieve_btc_with_approval` using the stored near-zero fee rate to build, sign (via threshold ECDSA), and submit Dogecoin transactions.

`evaluate_transaction_fee` (fees/mod.rs lines 118–121) computes `fee_rate.fee_ceil(tx_size)`, which evaluates to 0 when `fee_rate` is zero, producing a zero-fee transaction.

## Impact Explanation

When the Dogecoin canister returns anomalously low fee percentiles, any unprivileged user calling `retrieve_doge_with_approval` will have their ckDOGE burned on the IC ledger while the corresponding Dogecoin transaction is submitted with a near-zero fee. The Dogecoin network requires at least 0.001 DOGE/kB (1,000 koinu/byte) to relay transactions; zero-fee transactions are silently dropped. The user's ckDOGE is already burned and unrecoverable through normal flow. This constitutes a concrete, permanent loss of ck-token assets for affected users, matching the allowed impact: **High ($2,000–$10,000) — Significant Chain Fusion / ck-token security impact with concrete user fund harm.**

## Likelihood Explanation

The ckBTC minter already experienced this exact failure mode on 2025-06-21 (documented in the source code at `rs/bitcoin/ckbtc/minter/src/fees/mod.rs` lines 77–79), which is precisely why `minimum_fee_per_vbyte()` was added to `BitcoinFeeEstimator`. The ckDOGE minter shares the same fee-refresh infrastructure (`estimate_fee_per_vbyte` in `rs/bitcoin/ckbtc/minter/src/lib.rs`) and the same Dogecoin canister integration, making it equally susceptible to the same class of anomaly. No special privileges are required; any non-anonymous user can call `retrieve_doge_with_approval` during such a window.

## Recommendation

Add a `minimum_fee_per_byte()` method to `DogecoinFeeEstimator` analogous to `BitcoinFeeEstimator::minimum_fee_per_vbyte()`, and apply it in `estimate_nth_fee`:

```rust
const fn minimum_fee_per_byte(&self) -> FeeRate {
    let rate = match &self.network {
        Network::Mainnet => 1_000_000, // 1 koinu/byte = 1_000_000 millikoinu/byte
        Network::Regtest => 0,
    };
    FeeRate::from_millis_per_byte(rate)
}
```

Then in `estimate_nth_fee` (line 62):
```rust
Some(fee_percentiles[nth].max(self.minimum_fee_per_byte()))
```

The minimum should be at least the Dogecoin relay fee threshold (1 koinu/byte = 1,000,000 millikoinu/byte).

## Proof of Concept

1. The Dogecoin canister returns 100 fee percentiles all set to `FeeRate::from_millis_per_byte(0)` (mirroring the ckBTC event of 2025-06-21).
2. The ckDOGE minter's timer fires `estimate_fee_per_vbyte`, which calls `fee_estimator.estimate_median_fee(&fees)` → `estimate_nth_fee(&fees, 50)`.
3. `DogecoinFeeEstimator::estimate_nth_fee` returns `Some(FeeRate::from_millis_per_byte(0))` with no floor applied (line 62 of `rs/dogecoin/ckdoge/minter/src/fees/mod.rs`).
4. `last_median_fee_per_vbyte` is set to `FeeRate(0)` and `fee_based_retrieve_btc_min_amount` is set to the base minimum (line 248 of `rs/bitcoin/ckbtc/minter/src/lib.rs`).
5. A user calls `retrieve_doge_with_approval` with the minimum amount. The minter burns their ckDOGE and queues the withdrawal.
6. The minter builds a Dogecoin transaction; `evaluate_transaction_fee(tx, FeeRate(0))` = 0 koinu fee (line 120 of `rs/dogecoin/ckdoge/minter/src/fees/mod.rs`).
7. The transaction is signed via threshold ECDSA and submitted to the Dogecoin network.
8. The Dogecoin network rejects/ignores the zero-fee transaction. The user's ckDOGE is burned but DOGE is never received.

A unit test can reproduce this by constructing a `DogecoinFeeEstimator` with `Network::Mainnet`, passing a 100-element `fee_percentiles` slice of all-zero `FeeRate` values to `estimate_nth_fee`, and asserting the returned value is `Some(FeeRate(0))` — confirming the absence of any floor.
