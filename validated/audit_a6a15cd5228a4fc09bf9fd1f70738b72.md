Audit Report

## Title
SNS Token Treasury Valuation Uses Manipulable Spot Supply to Compute `icps_per_token`, Enabling Governance Rate-Limit Bypass - (File: `rs/sns/governance/token_valuation/src/lib.rs`)

## Summary

`fetch_icps_per_sns_token` derives the SNS token price by dividing the initial ICP-per-token ratio by `current_supply / initial_supply`, where `current_supply` is read live from `icrc1_total_supply` at proposal-submission time. Because `MintSnsTokens` validation unconditionally returns `Decimal::MAX` as its upper bound (TODO NNS1-2982), an attacker with SNS governance majority can first mint an arbitrarily large supply, then submit a `TransferSnsTreasuryFunds` proposal whose valuation is computed against the inflated supply, driving the treasury's XDR value below the 100,000 XDR `NoLimit` threshold and bypassing the 7-day rate limit entirely.

## Finding Description

**Pricing path:**

`assess_treasury_balance` → `try_get_sns_token_balance_valuation` → `IcpsPerSnsTokenClient::fetch_icps_per_sns_token`

Inside `fetch_icps_per_sns_token` (L397–414, `rs/sns/governance/token_valuation/src/lib.rs`), the price is:

```
total_inflation     = current_supply_e8s / initial_supply_e8s   // live icrc1_total_supply
icps_per_sns_token  = initial_icps_per_sns_token / total_inflation
```

There is no floor on `icps_per_token`; only `xdrs_per_icp` is clamped via `clamp_xdrs_per_icp` (`rs/sns/governance/proposals_amount_total_limit/src/lib.rs`, L137–140).

**`MintSnsTokens` upper bound is uncapped:**

`MintSnsTokens::recent_amount_total_upper_bound_tokens` (`rs/sns/governance/src/proposal.rs`, L1035–1041) returns `Decimal::MAX` unconditionally. The real implementation is commented out pending TODO NNS1-2982.

**Validation flow for both proposal types** (`treasury_valuation_if_proposal_amount_is_small_enough_or_err`, L770–817):

1. Calls `assess_treasury_balance` → computes `icps_per_token` from live supply.
2. Calls `recent_amount_total_upper_bound_tokens` → for `MintSnsTokens`, always `Decimal::MAX`.
3. Checks `proposal_amount ≤ max - spent`; with `max = Decimal::MAX`, any mint amount passes.

**Treasury classification** (`from_valuation_xdr`, L126–134):

- `valuation_xdr ≤ 100,000` → `NoLimit` → `balance_tokens` returned as allowance (L76–77).
- `valuation_xdr ≤ 1,200,000` → `Fraction(0.25)`.
- Otherwise → `Xdr(300,000)`.

**Valuation snapshot stored at submission time** (`ActionAuxiliary::TransferSnsTreasuryFunds(valuation)`, L591–593) and reused at execution time (`transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`, L2600–2617), so the manipulated valuation persists through execution.

**Attack chain:**

1. Attacker holds SNS governance majority.
2. Submits `MintSnsTokens` for e.g. 10¹⁵ SNS tokens. Passes because `recent_amount_total_upper_bound_tokens` = `Decimal::MAX`.
3. After execution, `icrc1_total_supply` is inflated by factor ~10⁶.
4. Submits `TransferSnsTreasuryFunds` for the full treasury balance. `fetch_icps_per_sns_token` reads inflated supply → `icps_per_token ≈ 0` → `valuation_xdr ≪ 100,000` → `NoLimit` → full balance is the allowance.
5. Proposal passes; entire treasury is drained in one 7-day window.

## Impact Explanation

The SNS treasury 7-day rate-limiting system exists specifically to constrain what even a governance-majority actor can extract in a single window. This bug allows that control to be fully neutralized, enabling complete treasury drainage (ICP or SNS tokens) in a single proposal cycle. This matches: **High ($2,000–$10,000) — Significant SNS security impact with concrete user or protocol harm** (unauthorized drain of canister-controlled funds via bypass of an explicit governance rate-limiting guard).

## Likelihood Explanation

- Requires SNS governance majority — a meaningful constraint, but the rate-limiting system is specifically designed to protect against exactly this class of attacker.
- The `MintSnsTokens` unlimited minting gap is explicitly acknowledged in production code (TODO NNS1-2982), confirming it is currently live and reachable.
- The attack requires only two sequential governance proposals and no off-chain infrastructure.
- No analogous floor exists for `icps_per_token`, so the valuation can be driven arbitrarily close to zero regardless of ICP price.

## Recommendation

1. **Enforce the `MintSnsTokens` upper bound immediately** by uncommenting the `TODO(NNS1-2982)` block in `MintSnsTokens::recent_amount_total_upper_bound_tokens` (`rs/sns/governance/src/proposal.rs`, L1025–1032).
2. **Add a floor on `icps_per_token`** in `ProposalsAmountTotalUpperBound::in_tokens` analogous to the existing `clamp_xdrs_per_icp` (`rs/sns/governance/proposals_amount_total_limit/src/lib.rs`, L137–140), preventing supply inflation from driving the computed SNS token price to near-zero.
3. **Consider using a time-weighted or snapshot-anchored supply** rather than live `icrc1_total_supply` at proposal submission time.
