Audit Report

## Title
Unbounded `ingress_expiries` Loop Allows Unauthenticated OOM DoS of ICP Rosetta — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary

The ICP Rosetta `construction_payloads` handler accepts attacker-controlled `ingress_start` and `ingress_end` values from request metadata with no window-size validation. It then enters an unbounded `while now < ingress_end` loop, pushing one `u64` per ~120-second interval into a heap-allocated `Vec`. A single unauthenticated HTTP POST supplying `ingress_start=0` and `ingress_end=u64::MAX` forces the process to attempt allocating ~1.2 TB of memory, triggering OOM termination and taking the Rosetta service offline.

## Finding Description

**Interval computation** (`construction_payloads.rs` L59–60):

```rust
let interval =
    ic_limits::MAX_INGRESS_TTL - ic_limits::PERMITTED_DRIFT - Duration::from_secs(120);
```

`MAX_INGRESS_TTL` is 5 minutes and `PERMITTED_DRIFT` is 30 seconds, so `interval ≈ 120 seconds` (2 minutes). [1](#0-0) 

**Unvalidated caller-supplied window** (L74–84): `ingress_start` and `ingress_end` are deserialized directly from the JSON metadata object with no range or maximum-window-size check applied before use. [2](#0-1) 

**Unbounded loop** (L99–107): The loop runs for `(ingress_end - ingress_start) / interval` iterations, pushing one `u64` per iteration into `ingress_expiries`. With `ingress_start=0` and `ingress_end=u64::MAX`, this is `u64::MAX / 120e9 ≈ 153,722,867,280` iterations, requiring ~1.2 TB of allocation before the process is killed. [3](#0-2) 

**No semantic validation in deserialization** (`models.rs` L240–248): `ConstructionPayloadsRequestMetadata::try_from` is a pure `serde_json` deserialization with no semantic checks on the ingress window. [4](#0-3) 

**Contrast with ICRC1 handler** (`icrc1/src/construction_api/services.rs` L148–158): The ICRC1 path has explicit pre-loop guards that reject `ingress_start >= ingress_end` and windows that are too stale. The ICP handler has no equivalent guards. [5](#0-4) 

**Exploit path:**
1. Attacker sends a single unauthenticated HTTP POST to `/construction/payloads` with `"ingress_start": 0, "ingress_end": 18446744073709551615` in the metadata.
2. The handler deserializes the values without validation and enters the `while now < ingress_end` loop.
3. The loop executes ~153 billion iterations, each pushing 8 bytes onto the heap.
4. The OS OOM killer terminates the Rosetta process before completion.
5. The Rosetta API is offline until manually restarted.

No authentication, no prior knowledge beyond the public API spec, and no rate limiting or request-size guard is present in the handler.

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* A single request takes the ICP Rosetta service — the primary programmatic interface for exchanges and custodians integrating ICP — completely offline. The IC consensus layer is unaffected, but all Rosetta-dependent operations (balance queries, transaction construction, submission) are unavailable until the process is manually restarted. The attack is repeatable: the attacker can re-crash the process immediately after each restart.

## Likelihood Explanation

The endpoint requires no authentication. The attack payload is a standard JSON POST body. The attacker needs no prior knowledge beyond the public Rosetta API specification. The attack is a single request, is trivially reproducible, and can be repeated indefinitely to prevent recovery. No rate limiting or ingress window size guard is visible in the handler or its call chain.

## Recommendation

Add a maximum ingress window size check immediately after computing `ingress_start` and `ingress_end`, mirroring the ICRC1 pattern:

```rust
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 60 * 60); // 24 hours

if ingress_start >= ingress_end {
    return Err(ApiError::invalid_request(
        "ingress_start must be before ingress_end",
    ));
}
if ingress_end > ingress_start + MAX_INGRESS_WINDOW {
    return Err(ApiError::invalid_request(
        "ingress_end exceeds maximum allowed ingress window of 24 hours",
    ));
}
```

This caps the loop at `24h / 120s = 720` iterations maximum, bounding memory use to a few kilobytes regardless of attacker input.

## Proof of Concept

```bash
curl -X POST http://<rosetta-host>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
    "operations": [{"operation_identifier":{"index":0},"type":"TRANSACTION","account":{"address":"<valid_account>"},"amount":{"value":"-100000000","currency":{"symbol":"ICP","decimals":8}}}],
    "public_keys": [{"hex_bytes":"<valid_pubkey>","curve_type":"edwards25519"}],
    "metadata": {
      "ingress_start": 0,
      "ingress_end": 18446744073709551615
    }
  }'
```

Expected: Rosetta process RSS grows unboundedly and is killed by the OOM killer. Iteration count: `18446744073709551615 / 120000000000 ≈ 153,722,867,280`; each iteration allocates 8 bytes → ~1.2 TB total allocation before termination.

A deterministic unit test can confirm the bug without triggering OOM by asserting that a call with `ingress_end = ingress_start + (MAX_INGRESS_WINDOW + 1ns)` returns an `Err` before the loop is entered.
