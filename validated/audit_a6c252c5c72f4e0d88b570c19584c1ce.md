Audit Report

## Title
`SenderInfoContent` Canister Signature Not Bound to Ingress Request Enables Cross-Request Replay — (File: `rs/validator/src/ingress_validation.rs`)

## Summary

The `SenderInfoContent` signable type used to verify the `sender_info` field of an ingress message signs only the raw `info` bytes under the `"ic-sender-info"` domain separator, with no binding to the `MessageId`, target `canister_id`, `ingress_expiry`, or any other request-specific field. This means a valid `(info, sig)` pair obtained from any single interaction with a signing canister (e.g., Internet Identity) can be replayed verbatim in arbitrarily many different ingress requests to different canisters, methods, and argument payloads, for the full lifetime of the canister signature's certified-data proof. Any canister that makes authorization decisions based on `msg_caller_info_data()` is vulnerable to this replay.

## Finding Description

`SenderInfoContent` is defined in `rs/types/types/src/messages/http.rs` (lines 344–348):

```rust
impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.0);  // only raw info bytes
    }
}
```

The domain separator is set in `rs/types/types/src/crypto/sign.rs` (lines 161–165):

```rust
impl<'a> SignatureDomain for SenderInfoContent<'a> {
    fn domain(&self) -> Vec<u8> {
        domain_with_prepended_length("ic-sender-info")
    }
}
```

The full signed byte representation is therefore: `len("ic-sender-info") || "ic-sender-info" || info_bytes` — nothing else.

In `rs/validator/src/ingress_validation.rs`, `verify_sender_info_canister_sig` constructs the signable content at line 530:

```rust
let sender_info_content = SenderInfoContent(&sender_info.info);
```

The `validate_sender_info` function (lines 459–488) receives the full `HttpRequest<C>` — which exposes `request.id()` (the `MessageId` committing to `canister_id`, `method_name`, `arg`, `sender`, `ingress_expiry`, `nonce`) — but passes only `sender_info.info` into the signed content. None of the request-specific fields are included.

Because the canister signature scheme certifies `hash(seed) || hash(domain || info_bytes)` in the canister's certified data tree, and `msg` here is only `domain || info_bytes`, the same certified-data proof is valid for every ingress message carrying the same `(sender_pubkey, info, sig)` triple, regardless of target canister, method, arguments, or expiry.

The `msg_caller_info_data()` system API is confirmed as a real, non-feature-flagged production API (present in `rs/embedders/src/wasmtime_embedder/system_api.rs`, `rs/interfaces/src/execution_environment.rs`, and tested in `rs/execution_environment/tests/hypervisor.rs`), meaning canisters in production can and do consume this attested data for authorization decisions.

## Impact Explanation

This is a **High** severity finding. Any canister that reads `msg_caller_info_data()` and makes authorization or access-control decisions based on the attested bytes is vulnerable to cross-request replay. A concrete scenario: Internet Identity issues a `sender_info` attesting `info = "role:admin"` for a user's session on canister A. The user replays the identical `(info, sig)` in a new request to canister B (different canister, different method, different arguments, different `ingress_expiry`). Canister B's `validate_sender_info` check passes because `SenderInfoContent(info_bytes)` is unchanged. Canister B grants the user access it was never intended to have. This matches the allowed impact: **unauthorized access to canister-controlled resources/identities where exploitation requires obtaining a valid `sender_info` first** (a low constraint — any normal interaction with the signing canister suffices).

## Likelihood Explanation

The attack requires only a valid `sender_info` obtained through any normal interaction with the signing canister. No privileged access, no key compromise, and no threshold corruption is needed. The attacker is the legitimate sender — they simply reuse the same `(info, sig)` bytes in a different request. The canister signature's certified-data proof remains valid until the signing canister explicitly rotates its certified data tree. This is a zero-privilege, low-effort replay that any user of the `sender_info` feature can perform.

## Recommendation

Bind the `SenderInfoContent` signed payload to the specific ingress request by including the `MessageId` in the signed bytes:

```rust
pub struct SenderInfoContent<'a> {
    pub info: &'a [u8],
    pub message_id: &'a MessageId,
}

impl SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.message_id.as_bytes());
        bytes.extend_from_slice(self.info);
    }
}
```

`validate_sender_info` already has access to the full `HttpRequest<C>` and can supply `request.id()` to the content constructor. This makes each `sender_info` signature valid for exactly one ingress message.

## Proof of Concept

```rust
// Step 1: obtain a valid sender_info for request R1 to canister_A
let info_bytes = b"role:admin".to_vec();
// signing canister certifies hash(seed) || hash("ic-sender-info" || info_bytes)
let sig = signing_canister.sign_sender_info(&info_bytes).await;
let valid_sender_info = RawSignedSenderInfo {
    info: Blob(info_bytes.clone()),
    signer: Blob(signing_canister_id.get().into_vec()),
    sig: Blob(sig),
};

// Step 2: send R1 to canister_A — accepted (expected)
send_update(canister_A, "privileged_method", args_A, valid_sender_info.clone()).await;

// Step 3: replay IDENTICAL sender_info to canister_B with different args/method
// validate_sender_info passes: SenderInfoContent(info_bytes) is unchanged
send_update(canister_B, "other_privileged_method", args_B, valid_sender_info).await;
// canister_B sees msg_caller_info_data() == b"role:admin" — unintended grant
```

A deterministic integration test using PocketIC (which already has `msg_caller_info_data` test coverage in `packages/pocket-ic/tests/unix.rs`) can confirm this by: (1) obtaining a canister signature over `SenderInfoContent(info_bytes)`, (2) submitting it with request R1, (3) resubmitting the identical `sender_info` with a structurally different request R2 to a different canister, and (4) asserting that `msg_caller_info_data()` returns the attested bytes in both cases.
