### Title
Unchecked Exit Precompile Return Value in Bridge Withdrawal Burns Tokens Without Releasing Funds — (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.withdrawToNear`, `EvmErc20.withdrawToEthereum`, `EvmErc20V2.withdrawToNear`, and `EvmErc20V2.withdrawToEthereum` burn the caller's ERC-20 tokens first, then invoke the Aurora exit precompile via inline assembly. The return value of the assembly `call` opcode is captured in `res` but is never checked. If the precompile call fails with an `ExitError` (sub-call failure), the `call` opcode returns `0`, execution continues, and the function returns successfully — with the tokens permanently burned but no corresponding NEAR-side or Ethereum-side release ever scheduled.

---

### Finding Description

In `EvmErc20.sol` and `EvmErc20V2.sol`, both withdrawal functions follow the same pattern:

1. `_burn(_msgSender(), amount)` — irreversibly destroys the caller's ERC-20 tokens.
2. An inline assembly block calls the exit precompile at the hardcoded address.
3. The return value `res` of the `call` opcode is stored but **never read or checked**.

```solidity
// EvmErc20.sol withdrawToNear (line 53–62)
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    ...
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked
    }
}
``` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

The exit precompile (`ExitToNear` at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`, `ExitToEthereum` at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`) returns `ExitError` in multiple reachable conditions:

- `ERR_INVALID_RECEIVER_ACCOUNT_ID` — if the `recipient` bytes are not valid UTF-8 or not a valid NEAR account ID.
- `ERR_TARGET_TOKEN_NOT_FOUND` — if the ERC-20 contract address is not registered in the NEP-141 map.
- `ERR_INVALID_AMOUNT` — if the amount exceeds `u128::MAX`.
- `ERR_KEY_NOT_FOUND` — if the ETH connector account is not set in storage. [5](#0-4) [6](#0-5) 

In the EVM (SputnikVM), when a precompile returns `PrecompileFailure::Error`, the sub-call fails and the `call` opcode returns `0` to the caller. The caller's own state changes — including the `_burn` — are **not** reverted. The calling function continues and returns successfully. The tokens are gone with no corresponding release on the NEAR or Ethereum side.

Note: `PrecompileFailure::Fatal` (e.g., from a paused precompile) propagates up and aborts the entire transaction, which would revert the burn. The dangerous case is specifically `ExitError`, which only fails the sub-call. [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent freezing/loss of funds.**

When the precompile call fails silently, the user's bridged ERC-20 tokens are burned on the Aurora EVM side. No NEAR-side `ft_transfer` or `withdraw` promise is ever created. The corresponding NEP-141 tokens remain locked in the ETH connector contract with no mechanism to recover them. The user's funds are permanently destroyed on the EVM side and permanently frozen on the NEAR side.

---

### Likelihood Explanation

**Medium.** The `recipient` parameter in `withdrawToNear` is a raw `bytes` value supplied by the caller. Any user who passes bytes that are not a valid NEAR account ID (e.g., bytes containing non-UTF-8 sequences, an empty byte string, or a string exceeding NEAR's account ID length limit) will trigger `ERR_INVALID_RECEIVER_ACCOUNT_ID` in the precompile. This can happen accidentally (e.g., a frontend bug encoding the recipient incorrectly) or deliberately (a user testing with an invalid address). The `withdrawToEthereum` path is less likely to fail silently since the Ethereum address is a fixed 20-byte value, but `ERR_TARGET_TOKEN_NOT_FOUND` and `ERR_KEY_NOT_FOUND` remain reachable if storage is in an unexpected state. [6](#0-5) 

---

### Recommendation

After the assembly `call`, check `res` and revert if it is `0`. This ensures the `_burn` is also reverted when the precompile call fails, preserving the user's tokens:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply this fix to all four assembly blocks across `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. Deploy `EvmErc20` with a registered NEP-141 mapping.
2. Mint tokens to `alice`.
3. `alice` calls `withdrawToNear(bytes("\xff\xfe"), amount)` — `\xff\xfe` is not valid UTF-8.
4. `_burn(alice, amount)` executes; alice's ERC-20 balance drops to zero.
5. The precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` parses the recipient, fails at `str::from_utf8(recipient)` with `ERR_INVALID_RECEIVER_ACCOUNT_ID`, and returns `ExitError`.
6. The `call` opcode returns `0`; `res` is never checked; `withdrawToNear` returns successfully.
7. No promise log is emitted; no NEP-141 `ft_transfer` is ever scheduled.
8. Alice's tokens are permanently destroyed on the EVM side; the NEP-141 balance remains locked in the connector with no recovery path. [8](#0-7) [1](#0-0)

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
