### Title
Floor Division Truncation in `delta_gas` Calculation Causes Systematic Under-Charging of Gas — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

The `compute_gas_refund` function computes `delta_gas` — the extra gas to charge when native resource consumption exceeds what EVM gas alone accounts for — using floor (truncating) integer division. Because `native_used / native_per_gas` always truncates downward, up to `native_per_gas − 1` native units of actual consumption are silently dropped from the gas-used accounting every transaction. The user is systematically over-refunded by up to 1 gas unit per transaction, and the operator is correspondingly underpaid.

---

### Finding Description

In `compute_gas_refund`, the `deltaGas` adjustment is computed as:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
``` [1](#0-0) 

The documentation specifies the intended formula as:

> `deltaGas := (nativeUsed / nativePerGas) - gasUsed` [2](#0-1) 

The division `native_used / native_per_gas` is Rust integer division, which truncates toward zero (floor). If `native_used = native_per_gas * k + r` with `0 < r < native_per_gas`, then:

- **Actual result**: `delta_gas = k − gas_used`
- **Correct result**: `delta_gas = (k + 1) − gas_used` (ceiling division)

The residual `r` native units — representing real proving/computation cost — are never reflected in `gas_used`. The user is therefore refunded 1 extra gas unit, and the operator receives 1 fewer gas unit of payment, on every transaction where `native_used % native_per_gas ≠ 0`.

`native_per_gas` itself is computed with ceiling division (`div_ceil`) to ensure the user's native budget is sufficient:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [3](#0-2) 

This asymmetry — ceiling when setting the rate, floor when recovering gas from native — is the root of the precision error. The same floor truncation also appears in the `native_per_pubdata` derivation:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
``` [4](#0-3) 

`wrapping_div` is also floor division, meaning the native cost per pubdata byte is systematically underestimated by up to `native_price − 1` units, compounding the under-charging when pubdata is written.

---

### Impact Explanation

Every transaction where `native_used % native_per_gas ≠ 0` (which is virtually every transaction) results in:

1. **User over-refund**: The user receives a refund for 1 extra gas unit they did not actually have available in native resources.
2. **Operator underpayment**: The operator is paid `gas_price` wei less than the actual native cost warrants.
3. **Cumulative loss**: Across high-throughput blocks, the aggregate underpayment to the operator grows proportionally with transaction count.

The per-transaction financial impact is bounded by `1 gas × gas_price`. At a gas price of 1 Gwei and 1,000 transactions per block, this is ~1,000 Gwei per block of systematic operator underpayment.

---

### Likelihood Explanation

**High.** The condition `native_used % native_per_gas ≠ 0` holds for virtually every real transaction. `native_used` is the sum of many heterogeneous native charges (arithmetic opcodes, memory expansion, pubdata, proving overhead), making exact divisibility by `native_per_gas` essentially impossible in practice. No special transaction construction is required; any normal EVM transaction triggers this path. [5](#0-4) 

---

### Recommendation

Replace the floor division with ceiling division in the `delta_gas` calculation:

```rust
// Before (floor):
(native_used / native_per_gas) as i64 - (gas_used as i64)

// After (ceiling):
native_used.div_ceil(native_per_gas) as i64 - (gas_used as i64)
```

Similarly, replace `wrapping_div` with `div_ceil` for `native_per_pubdata` in `validation_impl.rs` to ensure pubdata native costs are not systematically underestimated.

---

### Proof of Concept

Consider a transaction with:
- `gas_limit = 100_000`
- `native_per_gas = 7` (i.e., `gas_price = 7 * native_price`)
- `native_used = 699_999` (i.e., `gas_limit * native_per_gas - 1`)
- `gas_used = 99_999` (from ergs)

Then:
- `floor(699_999 / 7) = 99_999` → `delta_gas = 99_999 − 99_999 = 0`
- `ceil(699_999 / 7) = 100_000` → `delta_gas = 100_000 − 99_999 = 1`

With floor division, `gas_used` stays at `99_999` and the user is refunded 1 gas unit. With ceiling division, `gas_used` becomes `100_000` and no refund is issued. The user consumed `699_999` native units — equivalent to `100_000` gas — but is only charged for `99_999` gas. The operator is underpaid by `gas_price` wei.

This is directly analogous to the Basin LP token sqrt precision error: a mathematical operation (sqrt/floor-div) with larger-than-expected truncation error causes excess value to remain unaccounted for, systematically benefiting one party (LP depositor/tx sender) at the expense of another (protocol/operator).
