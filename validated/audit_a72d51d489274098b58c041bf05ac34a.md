### Title
Partial `ft_transfer_call` Refunds Not Re-minted to EVM User in `ExitToNear` Omni Path — (File: `engine-precompiles/src/native.rs`)

---

### Summary

When a user exits ERC-20 (or base ETH) tokens to NEAR using the **Omni message format** of the `ExitToNear` precompile, the engine burns the user's EVM tokens and dispatches a `ft_transfer_call` to the NEP-141 contract. The NEP-141 standard permits the receiver to **partially accept** the tokens and return the remainder to the sender (Aurora engine). The `exit_to_near_precompile_callback` only handles complete failure (triggering a full refund) or complete success (doing nothing). It never inspects the `ft_transfer_call` return value for a **partial refund**. The refunded NEP-141 tokens accumulate silently in Aurora's NEP-141 balance while the user's ERC-20 tokens remain permanently burned.

---

### Finding Description

**Root cause — `engine-precompiles/src/native.rs`, Omni branch of `exit_erc20_token_to_near`:**

When the caller supplies a colon-delimited Omni message (e.g. `receiver.near:{"some":"json"}`), `parse_recipient` sets `message = Some(Message::Omni(msg))`. The function `exit_erc20_token_to_near` then selects the `ft_transfer_call` method: [1](#0-0) 

The ERC-20 tokens are burned from the caller's EVM balance before this call is dispatched. The NEP-141 `ft_transfer_call` standard (NEP-141 §5) requires the receiver to implement `ft_on_transfer` and return the number of tokens it does **not** want to keep. The NEP-141 contract's `ft_resolve_transfer` then credits that refund amount back to the sender — in this case, the Aurora engine contract's own NEP-141 balance — without calling Aurora's `ft_on_transfer`.

**Root cause — `engine/src/contract_methods/connector.rs`, `exit_to_near_precompile_callback`:**

The callback that fires after the `ft_transfer_call` promise resolves only branches on complete success vs. complete failure: [2](#0-1) 

`PromiseResult::Successful(_)` is matched without inspecting the inner bytes, which for `ft_transfer_call` contain the **amount refunded** (the amount the receiver declined). A partial refund still produces `Successful`, so the callback takes the "no action" path and the refunded NEP-141 tokens are never re-minted as ERC-20 tokens to the user.

**Callback attachment — `engine-precompiles/src/native.rs`:**

The callback is only attached when the `error_refund` compile-time feature is enabled: [3](#0-2) 

Without `error_refund`, no callback is attached at all for the Omni path, making the vulnerability unconditional. With `error_refund`, the callback exists but still does not handle partial refunds.

**The same structural flaw applies to the base-token (ETH) Omni path** in `exit_base_token_to_near`: [4](#0-3) 

---

### Impact Explanation

A user who exits ERC-20 tokens via the Omni path loses the portion of tokens that the receiver declines. The ERC-20 burn is irreversible; the refunded NEP-141 tokens sit in Aurora's NEP-141 balance with no on-chain mechanism to recover them. This constitutes **permanent freezing / direct theft of user funds** — matching the Critical/High tier of the allowed impact scope.

---

### Likelihood Explanation

Any NEAR contract that legitimately or maliciously returns a non-zero value from `ft_on_transfer` (e.g., a DEX with insufficient liquidity, a contract with deposit caps, or a purpose-built griefing contract) triggers the loss. The Omni path is the intended integration point for Aurora ↔ NEAR DeFi interactions, so real-world exposure is high. No privileged access is required; any EVM address can call the `ExitToNear` precompile.

---

### Recommendation

In `exit_to_near_precompile_callback` (`engine/src/contract_methods/connector.rs`), deserialize the bytes inside `PromiseResult::Successful(bytes)` when the base promise was a `ft_transfer_call`. The return value is the refunded amount (a JSON-encoded `U128`). If it is non-zero, call `engine::refund_on_error` (or an equivalent mint path) to re-credit that amount of ERC-20 tokens to the original EVM sender. The `RefundCallArgs` already carries the sender address and ERC-20 contract address needed for this re-mint.

Alternatively, expose a `min_amount_accepted` parameter in the Omni message format and revert the entire exit if the receiver accepts fewer tokens than the minimum.

---

### Proof of Concept

1. User holds 1000 units of ERC-20 token `T` on Aurora EVM.
2. User calls the `ExitToNear` precompile (flag `0x1`) with Omni message targeting `dex.near:{"action":"swap"}` and `amount = 1000`.
3. Aurora burns 1000 `T` from the user's EVM balance.
4. Aurora dispatches `ft_transfer_call("dex.near", 1000, '{"action":"swap"}')` on the NEP-141 contract for `T`.
5. `dex.near`'s `ft_on_transfer` has only 1 unit of liquidity available; it returns `999` (refund).
6. The NEP-141 contract's `ft_resolve_transfer` credits 999 tokens back to Aurora's NEP-141 balance. The promise result is `Successful("999")`.
7. `exit_to_near_precompile_callback` matches `PromiseResult::Successful(_)`, ignores the inner value, and returns `None` — no re-mint occurs.
8. User received 1 unit of output on `dex.near` and permanently lost 999 units of `T` with no recourse.
