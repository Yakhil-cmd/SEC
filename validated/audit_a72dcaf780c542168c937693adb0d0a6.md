Audit Report

## Title
Unbounded Per-Connection Stream Task Spawning Enables Memory Exhaustion via Byzantine Subnet Peer — (`rs/p2p/quic_transport/src/request_handler.rs`)

## Summary
The `start_stream_acceptor` function in `rs/p2p/quic_transport/src/request_handler.rs` spawns an unbounded Tokio task into an uncapped `JoinSet` for every accepted bidirectional QUIC stream. Each task calls `read_to_end(MAX_MESSAGE_SIZE_BYTES)` which accumulates up to 128 MiB per stream into a heap-allocated `Vec<u8>`. A Byzantine subnet peer with an established authenticated connection can open up to `MAX_CONCURRENT_BIDI_STREAMS` (1,000) streams and slowly fill each stream's buffer, exhausting replica heap memory and crashing the process.

## Finding Description

**Confirmed constants:**
- `MAX_CONCURRENT_BIDI_STREAMS = VarInt::from_u32(1_000)` at `rs/p2p/quic_transport/src/connection_manager.rs:74`
- `MAX_MESSAGE_SIZE_BYTES: usize = 128 * 1024 * 1024` at `rs/p2p/quic_transport/src/lib.rs:74`
- `STREAM_RECEIVE_WINDOW = VarInt::from_u32(4_000_000)` and `RECEIVE_WINDOW = VarInt::from_u32(200_000_000)` at `rs/p2p/quic_transport/src/connection_manager.rs:71-73`

**Unbounded task spawning:**

`start_stream_acceptor` holds an unbounded `JoinSet<Result<(), P2PError>>` (`inflight_requests`). Every accepted bidi stream unconditionally spawns a new task with no concurrency cap, no semaphore, and no per-connection memory accounting:

```rust
// rs/p2p/quic_transport/src/request_handler.rs:50
let mut inflight_requests: JoinSet<Result<(), P2PError>> = tokio::task::JoinSet::new();
```

```rust
// rs/p2p/quic_transport/src/request_handler.rs:79-90
inflight_requests.spawn(
    metrics.request_task_monitor.instrument(
        handle_bi_stream(peer_id, conn_handle.conn_id(), metrics.clone(), router.clone(), send_stream, bi_rx)
    )
);
```

The code itself acknowledges this design gap at lines 52–56:
> "The extreme result of a slow handler is that the stream limit will be reached, hence having buffered up to the stream limit number of messages/requests."

**Per-task heap allocation:**

Each spawned task calls `read_to_end(MAX_MESSAGE_SIZE_BYTES)` in `read_request`:

```rust
// rs/p2p/quic_transport/src/request_handler.rs:173-175
let request_bytes = recv_stream
    .read_to_end(MAX_MESSAGE_SIZE_BYTES)
    .await
```

`read_to_end` allocates a `Vec<u8>` that grows as data arrives, up to 128 MiB. Critically, as Quinn reads incoming bytes into this Vec, it advances the QUIC flow control window (sends `MAX_DATA` frames), allowing the sender to push more data. The `STREAM_RECEIVE_WINDOW` (4 MB) and `RECEIVE_WINDOW` (200 MB) are sliding windows that throttle throughput but do not cap total accumulated memory across 1,000 long-lived tasks. The attacker simply sends data slowly on each stream, and the receiver accumulates it indefinitely until the stream closes.

**Authentication gate:**

The TLS server config at `rs/p2p/quic_transport/src/connection_manager.rs:208-215` restricts connections to authenticated subnet peers. The attacker must be a legitimate (but Byzantine) subnet node — a single compromised node, which is below the consensus fault threshold and within scope.

## Impact Explanation

A single Byzantine subnet peer can exhaust the heap of a targeted honest replica (up to ~128 GiB theoretical maximum across 1,000 streams × 128 MiB/stream), causing an OOM crash. This halts consensus participation for the affected node. By selectively targeting specific replicas, a Byzantine peer could reduce the honest-node count toward the fault threshold, weakening the subnet's Byzantine fault tolerance guarantees.

This matches the allowed High impact: **Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000).

## Likelihood Explanation

- Requires only one compromised subnet node (below fault threshold, within scope).
- The attack is straightforward: open 1,000 streams, send large bodies slowly while respecting the 4 MB stream window.
- No rate limiting, no per-connection memory cap, no backpressure, and no concurrency limit exists in the stream acceptor.
- The code comment at lines 52–56 explicitly acknowledges the buffering risk.
- Replica nodes typically run many concurrent workloads; 128 GiB of additional heap pressure is a realistic OOM trigger.

## Recommendation

1. **Cap concurrent inflight tasks per connection**: Replace the unbounded `JoinSet` with a semaphore-guarded or bounded channel so that `accept_bi` blocks when the in-flight task count reaches a safe limit (e.g., 32–64).
2. **Apply a per-connection memory budget**: Track total bytes being read across all active tasks for a connection and reject new streams when the budget is exceeded.
3. **Reduce `MAX_CONCURRENT_BIDI_STREAMS`**: Lower from 1,000 to a value consistent with actual P2P protocol needs.
4. **Implement streaming request parsing**: Instead of `read_to_end` (full buffering), use incremental reads so memory is bounded by what the handler has consumed, not what the peer has sent.

## Proof of Concept

```
1. Attacker controls a legitimate subnet node (Byzantine, below fault threshold).
2. Attacker's node has an established QUIC connection to the victim replica
   (present in peer_map after TLS handshake).
3. Attacker opens 1,000 bidirectional QUIC streams simultaneously.
4. On each stream, attacker slowly sends data (respecting the 4 MB stream window),
   advancing the flow control window as the receiver's read_to_end consumes bytes.
5. start_stream_acceptor spawns 1,000 tasks, each blocked in read_to_end(128 MiB).
6. Each task accumulates up to 128 MiB in its Vec<u8> buffer.
7. Total heap: ~128 GiB → OOM → replica process crash.
8. Consensus participation halts for the victim node.

Reproducible test plan:
- Write an integration test using PocketIC or a local multi-node setup.
- Spawn a mock Byzantine peer that opens MAX_CONCURRENT_BIDI_STREAMS streams
  and slowly writes data on each stream without closing them.
- Assert that the victim replica's memory usage grows unboundedly and eventually
  triggers an OOM or memory limit error.
