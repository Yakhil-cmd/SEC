### Title
SNS Treasury Manager Deposit Enforces No Slippage Bounds, Enabling Front-Running of Governance-Approved DEX Deposits - (File: `rs/sns/treasury_manager/src/lib.rs`, `rs/sns/governance/src/extensions.rs`)

---

### Summary

The SNS Treasury Manager framework allows SNS DAOs to deposit treasury funds into DEX liquidity pools via a governance-approved `ExecuteExtensionOperation` proposal. The `DepositRequest` type carries no slippage parameter, and neither the proposal validation path nor the execution path in SNS Governance enforces any price-impact bound before funds are committed to the DEX. The codebase itself acknowledges this as a known risk in two separate places but takes no mitigating action at the protocol level. An unprivileged on-chain actor who observes the pending proposal can front-run the deposit, causing the SNS treasury to receive significantly fewer LP tokens than expected.

---

### Finding Description

The `DepositRequest` struct, which is the sole argument passed to the treasury manager's `deposit` canister method, contains only an `allowances` field:

```rust
pub struct DepositRequest {
    pub allowances: Vec<Allowance>,
}
```

There is no `min_lp_tokens_out`, `max_price_impact`, or any other slippage-bounding field. [1](#0-0) 

The Candid interface definition mirrors this exactly, and the file's own header explicitly flags the gap:

```
// Known Security Risks:
// Some liquidity pools do not implement slippage protection
// for deposits. In other words, the price ratio at the time
// of execution may differ from the ratio at the time the proposal was
// approved.
``` [2](#0-1) 

The governance execution function `execute_treasury_manager_deposit` approves the allowance and then calls `deposit` on the treasury manager canister with the raw `DepositRequest` blob. No slippage check is inserted between the two steps:

```rust
governance.approve_treasury_manager(extension_canister_id, ...).await?;
governance.env.call_canister(extension_canister_id, "deposit", arg_blob).await ...
``` [3](#0-2) 

The proposal validation function `validate_deposit_operation_impl` only checks that the requested amounts do not exceed 50% of the current treasury balance. It performs no price-impact or slippage validation: [4](#0-3) 

The proposal rendering function `validate_and_render_register_extension` does include a human-readable `## WARNING` about slippage, but this is purely cosmetic text in the proposal description — it is not enforced by any on-chain check: [5](#0-4) 

---

### Impact Explanation

When an SNS governance proposal to deposit treasury funds into a DEX liquidity pool is approved and executed, the actual execution happens at a later block than the proposal approval. Because no minimum LP token output or maximum price deviation is enforced in the `DepositRequest`, the deposit proceeds unconditionally regardless of how much the pool price has moved. A front-running actor can sandwich the deposit: push the pool price unfavorably before the deposit lands, let the governance canister deposit at the manipulated price, then reverse the trade — extracting value directly from the SNS treasury. The SNS treasury (and by extension all SNS token holders) permanently loses the difference between the expected and actual LP token value received. The `Allowance.amount_decimals` field sets only an upper bound on how much the treasury manager may spend; it does not bound the price at which those funds are exchanged. [6](#0-5) 

---

### Likelihood Explanation

The attack is realistic and requires no privileged access. The governance proposal is public and visible on-chain before execution. Any canister or boundary-node-connected actor can observe the pending proposal, compute the expected deposit size, and submit a front-running transaction to the same DEX before the governance canister's `deposit` call lands. IC consensus ordering means the attacker's transaction can be placed in the same or an immediately preceding block. The SNS Treasury Manager framework is explicitly designed to interact with DEXes such as KongSwap (referenced in `extensions.rs`), which are live on mainnet, making this a concrete and reachable scenario. [7](#0-6) 

---

### Recommendation

1. **Add slippage parameters to `DepositRequest`**: Extend the struct with fields such as `min_lp_tokens_out: Option<Nat>` or `max_price_impact_bps: Option<u64>`. The treasury manager implementation must pass these bounds to the underlying DEX call.

2. **Enforce slippage bounds in `validate_deposit_operation_impl`**: Require that any `ExecuteExtensionOperation` proposal for a deposit includes a non-zero, governance-approved slippage bound. Reject proposals that omit it.

3. **Enforce slippage bounds at execution time**: In `execute_treasury_manager_deposit`, verify that the returned `Balances` reflect an outcome within the approved slippage tolerance; revert (or flag for manual recovery) if the DEX returned fewer LP tokens than the minimum.

4. **Remove the informational-only warning**: The `## WARNING` text in `validate_and_render_register_extension` gives token voters a false sense of awareness without any protocol-level protection. Replace it with a hard validation requirement.

---

### Proof of Concept

1. An SNS DAO submits and passes a `ExecuteExtensionOperation` proposal to deposit `X` SNS tokens and `Y` ICP into a KongSwap liquidity pool via the treasury manager extension.
2. The proposal is approved by token holders. The execution is scheduled.
3. An unprivileged on-chain actor observes the approved proposal (public state) and submits a large swap on the same KongSwap pool, moving the SNS/ICP price significantly against the treasury's deposit direction.
4. The IC governance canister executes `execute_treasury_manager_deposit`, which calls `approve_treasury_manager` then `deposit` on the treasury manager canister.
5. The treasury manager calls the KongSwap DEX `deposit` with the full approved allowance. Because `DepositRequest` carries no `min_lp_tokens_out`, the DEX accepts the deposit at the manipulated price and issues far fewer LP tokens than the pre-proposal price would have implied.
6. The front-runner reverses their swap, profiting from the spread. The SNS treasury has permanently lost value with no on-chain recourse.

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}
