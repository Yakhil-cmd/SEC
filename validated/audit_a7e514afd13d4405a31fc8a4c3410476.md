### Title
Pre-EIP-155 Legacy Transaction Replay Bypasses Chain-ID Guard, Enabling Cross-Chain Replay Attacks — (`engine/src/engine.rs`, `engine-transactions/src/legacy.rs`)

---

### Summary

`submit_with_alt_modexp` validates the chain ID only when `transaction.chain_id` is `Some`. For pre-EIP-155 legacy transactions (`v = 27` or `v = 28`), `chain_id()` returns `None`, so the guard is silently skipped. Any such transaction broadcast on another EVM chain can be replayed verbatim on Aurora whenever the sender's Aurora nonce matches the replayed nonce and the sender holds ETH on Aurora.

---

### Finding Description

**Step 1 — `chain_id()` returns `None` for pre-EIP-155 transactions.**

`LegacyEthSignedTransaction::chain_id()` maps `v ∈ [0, 34]` (which includes 27 and 28) to `None`: [1](#0-0) 

**Step 2 — `NormalizedEthTransaction` propagates `None` unchanged.**

The `TryFrom<EthTransactionKind>` implementation for the `Legacy` variant copies `chain_id: tx.chain_id()` directly into the normalized struct: [2](#0-1) 

**Step 3 — The chain-ID guard is an `if let Some(...)` — it is a no-op when `chain_id` is `None`.**

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

When `chain_id` is `None` the entire block is skipped. Execution falls through unconditionally to `check_nonce`, `charge_gas`, and EVM execution: [3](#0-2) 

**Step 4 — `assert_access` (silo whitelist) does not block this.**

`assert_access` checks the silo whitelist, but each whitelist is disabled by default (`!list.is_enabled()` returns `true` when not configured), so on a standard Aurora deployment every address passes: [4](#0-3) [5](#0-4) 

**Step 5 — EVM executes the replayed calldata.**

After the skipped guard, the engine charges gas from the sender's balance and executes the transaction payload (call or contract deployment) without any further chain-identity check: [6](#0-5) 

The comment on the `submit` export explicitly states the intent: *"Must match `CHAIN_ID` to make sure it's signed for given chain vs replayed from another chain."* [7](#0-6) 

The implementation contradicts this intent for the pre-EIP-155 case.

---

### Impact Explanation

An attacker who locates a victim's pre-EIP-155 transaction on any other EVM chain (all are publicly visible on-chain) can submit it to Aurora's `submit` endpoint. If the victim's Aurora nonce equals the replayed transaction's nonce and the victim holds ETH on Aurora:

- **Theft**: the replayed transaction's `to` field is an address the attacker controls on Aurora → victim's ETH is transferred to the attacker.
- **Permanent freezing**: the replayed transaction sends ETH to a contract address that does not exist on Aurora (e.g., a mainnet contract), or deploys code that locks funds → victim's ETH is irrecoverably lost.

Both outcomes match the allowed Critical impact scope.

---

### Likelihood Explanation

Pre-EIP-155 transactions (`v = 27/28`) are rare but exist in large numbers on Ethereum mainnet history (all transactions before block ~2,675,000 and many wallets/tools thereafter). The binding constraint is nonce alignment: the victim's Aurora nonce must equal the nonce in the replayed transaction. This is a real constraint but not an impossible one, particularly for accounts that have used Aurora since early in their Ethereum history, or for accounts with low nonce values. The attacker needs no special privilege — only the ability to call the public `submit` endpoint with a valid RLP-encoded transaction.

Likelihood is **low-to-medium**: the nonce alignment requirement limits the attack surface, but the attack is fully permissionless and requires no leaked keys or admin access.

---

### Recommendation

Reject pre-EIP-155 transactions outright at the submission boundary. The simplest fix is to change the chain-ID guard from an `if let Some` to an explicit rejection when `chain_id` is `None`:

```rust
let chain_id = transaction.chain_id
    .ok_or(EngineErrorKind::InvalidChainId)?;
if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

Alternatively, add an explicit early-return for `chain_id == None` before the existing guard. Either approach closes the replay window entirely.

### Proof of Concept

1. On a local Aurora sandbox, fund address `A` with ETH and set its nonce to `N`.
2. Construct a `TransactionLegacy` with `nonce = N`, `to = attacker_address`, `value = victim_balance`, signed with `v = 27` (pre-EIP-155, no chain ID).
3. RLP-encode the signed transaction and submit it to the `submit` endpoint.
4. Assert: `check_nonce` passes (nonce matches), `charge_gas` deducts from `A`, EVM transfers ETH to `attacker_address`, `A`'s balance is zero.

impl LegacyEthSignedTransaction {
    /// Returns sender of given signed transaction by doing ecrecover on the signature.
    pub fn sender(&self) -> Result<Address, Error> {
        let mut rlp_stream = RlpStream::new();
        // See details of CHAIN_ID computation here - https://github.com/ethereum/EIPs/blob/master/EIPS/eip-155.md#specification
        let (chain_id, rec_id) = match self.v {
            0..=26 | 29..=34 => return Err(Error::InvalidV),
            27..=28 => (
                None,
                u8::try_from(self.v - 27).map_err(|_e| Error::InvalidV)?,
            ),
            _ => (
                Some((self.v - 35) / 2),
                u8::try_from((self.v - 35) % 2).map_err(|_e| Error::InvalidV)?,
            ),
        };
        self.transaction
            .rlp_append_unsigned(&mut rlp_stream, chain_id);
        let message_hash = sdk::keccak(rlp_stream.as_raw());
        sdk::ecrecover(message_hash, &super::vrs_to_arr(rec_id, self.r, self.s))
            .map_err(|_| Error::EcRecover)
    }

    /// Returns chain id encoded in `v` parameter of the signature if that was done, otherwise None.
    #[must_use]
    pub const fn chain_id(&self) -> Option<u64> {
        match self.v {
            0..=34 => None,
            _ => Some((self.v - 35) / 2),
        }
    }

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
