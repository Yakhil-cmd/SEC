All code claims are verified. The `ConsolidateUtxos` task type exists in the shared `ic_ckbtc_minter` library and is used by ckBTC's `setup_tasks()`, but ckDOGE's `setup_tasks()` never enqueues it, and the self-rescheduling guard only fires after the first execution — which never happens.

Audit Report

## Title
Missing `TaskType::ConsolidateUtxos` Scheduling in ckDOGE Minter Causes Permanent UTXO Fragmentation and Withdrawal DoS — (`rs/dogecoin/ckdoge/minter/src/main.rs`)

## Summary
ckDOGE's `setup_tasks()` omits `schedule_now(TaskType::ConsolidateUtxos, ...)`, which ckBTC explicitly includes. Because the `ConsolidateUtxos` task handler self-reschedules only after its first execution via a `scopeguard`, and it is never initially enqueued, `consolidate_utxos` is permanently dead code in ckDOGE. Over time — and accelerated by a low-cost attacker — UTXO fragmentation causes all withdrawals above a threshold to fail permanently with `BuildTxError::InvalidTransaction(TooManyInputs)`, freezing ckDOGE redemptions until a code upgrade is deployed.

## Finding Description
ckBTC's `setup_tasks()` schedules three tasks on `init` and `post_upgrade`: [1](#0-0) 

ckDOGE's `setup_tasks()` schedules only two, omitting `ConsolidateUtxos`: [2](#0-1) 

Both `init` and `post_upgrade` call `setup_tasks()`, so no canister restart can recover the missing task: [3](#0-2) [4](#0-3) 

The `ConsolidateUtxos` task handler in the shared `ic_ckbtc_minter` library re-enqueues itself only via a `scopeguard` that fires at the end of its first execution: [5](#0-4) 

Since the task is never initially enqueued in ckDOGE, the guard never fires, and `consolidate_utxos` never runs. The `available_utxos` pool grows monotonically. The hard cap is 500 inputs per transaction: [6](#0-5) 

Once the 500 largest UTXOs in the pool are insufficient to cover a requested withdrawal amount, `estimate_withdrawal_fee` returns `EstimateWithdrawalFeeError::AmountTooHigh` and `retrieve_doge_with_approval` fails: [7](#0-6) 

## Impact Explanation
This is a **High** severity issue matching the allowed impact: *"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."* ckDOGE is an in-scope Chain Fusion ck-token. The result is a permanent, irreversible withdrawal DoS — ckDOGE holders cannot redeem their tokens for DOGE — until a governance-approved code upgrade is deployed. User funds are effectively frozen in the ckDOGE ledger.

## Likelihood Explanation
- `update_balance` is open to any non-anonymous caller; no privilege is required.
- DOGE has 1-minute block times and fees of ~0.01 DOGE per transaction, making 501 small deposits trivially cheap (under $1 USD at current prices).
- Fragmentation also occurs organically from change outputs during normal withdrawals, so the condition is reached even without a deliberate attacker — it is a matter of time.
- The bug is present in both `init` and `post_upgrade`, so it cannot be remediated by a canister restart alone.

## Recommendation
Add `schedule_now(TaskType::ConsolidateUtxos, &DOGECOIN_CANISTER_RUNTIME)` to ckDOGE's `setup_tasks()`, mirroring ckBTC:

```rust
// rs/dogecoin/ckdoge/minter/src/main.rs
fn setup_tasks() {
    schedule_now(TaskType::ProcessLogic, &DOGECOIN_CANISTER_RUNTIME);
    schedule_now(TaskType::RefreshFeePercentiles, &DOGECOIN_CANISTER_RUNTIME);
    schedule_now(TaskType::ConsolidateUtxos, &DOGECOIN_CANISTER_RUNTIME); // add this
}
```

## Proof of Concept
1. Deploy a ckDOGE minter canister; `init` fires `setup_tasks()` without `ConsolidateUtxos`.
2. As an unprivileged principal, send 501 separate DOGE deposits of `retrieve_doge_min_amount` each to the minter-derived address.
3. Call `update_balance` after each deposit confirms; verify each call adds one entry to `available_utxos`.
4. Observe that no `ConsolidateUtxos`-related event ever appears in `get_events` output.
5. Attempt a withdrawal of `501 × retrieve_doge_min_amount` DOGE via `retrieve_doge_with_approval`.
6. Assert the call returns an error mapping to `BuildTxError::InvalidTransaction(TooManyInputs)`, surfaced as `EstimateWithdrawalFeeError::AmountTooHigh`.
7. Confirm the user's ckDOGE balance is unchanged and unwithdrawable.
