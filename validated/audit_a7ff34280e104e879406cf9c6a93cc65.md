### Title
CMC Minting Rate Limiter Bypass via Direct Cycle Transfer to the Canister - (File: rs/nns/cmc/src/main.rs)

### Summary

The `ensure_balance` function in the Cycles Minting Canister (CMC) computes the number of cycles to newly mint as `cycles_to_mint = cycles - ic_cdk::api::canister_balance128()` (saturating subtraction). The minting rate limiter is checked against `cycles_to_mint` — the delta to mint — rather than against `cycles` — the total amount to deliver. Because any canister on the IC can send cycles directly to the CMC via inter-canister calls, an attacker can inflate the CMC's cycle balance before a conversion, causing the limiter to record fewer (or zero) newly minted cycles, effectively bypassing the hourly/monthly minting rate cap.

---

### Finding Description

`ensure_balance` in `rs/nns/cmc/src/main.rs` is the single gate that enforces the minting rate limit for all ICP→cycles conversions (`notify_top_up`, `notify_create_canister`, `notify_mint_cycles`):

```rust
fn ensure_balance(cycles: Cycles, limiter_to_use: CyclesMintingLimiterSelector) -> Result<(), String> {
    let now = now_system_time();
    let current_balance = Cycles::from(ic_cdk::api::canister_balance128()); // reads live CMC balance
    let cycles_to_mint = cycles - current_balance;                           // saturating subtraction

    with_state_mut(|state| {
        limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;   // limiter sees delta, not total
        state.total_cycles_minted += cycles_to_mint;                         // accounting also uses delta
        Ok::<_, String>(())
    })?;

    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);
    Ok(())
}
``` [1](#0-0) 

The limiter's `check_and_add_cycles` enforces a cap of `base_cycles_limit` (150 × 10¹⁵ cycles/hour) or `subnet_rental_cycles_limit` (500 × 10¹⁵ cycles/month): [2](#0-1) 

The limiter is checked with `cycles_to_mint`, not with `cycles`. If an attacker pre-loads the CMC with X cycles by sending them via an inter-canister call, then for any subsequent conversion requesting Y cycles:

- `cycles_to_mint = Y - X` (saturating, so 0 if X ≥ Y)
- The limiter records `Y - X` (or 0) against the cap
- The CMC still delivers Y cycles to the target (using the pre-loaded balance)
- The ICP is still burned

This allows an attacker to make the limiter believe far fewer cycles were minted than were actually delivered, enabling repeated conversions that collectively exceed the rate cap within a single window.

The `total_cycles_minted` counter is also undercounted by the same delta, corrupting the on-chain accounting metric exposed via `total_cycles_minted()` and the `/metrics` endpoint. [3](#0-2) 

The three public entry points that all funnel through `ensure_balance` are:

- `notify_top_up` → `process_top_up` → `deposit_cycles` → `ensure_balance`
- `notify_create_canister` → `do_create_canister` → `ensure_balance`
- `notify_mint_cycles` → `process_mint_cycles` → `do_mint_cycles` → `ensure_balance` [4](#0-3) [5](#0-4) 

---

### Impact Explanation

An unprivileged attacker canister can:

1. Send a large amount of cycles to the CMC (any canister can do this via `call_with_payment128`).
2. Trigger or wait for legitimate ICP→cycles conversions.
3. The limiter records 0 (or a reduced amount) against the hourly cap.
4. Repeat: the attacker can keep the CMC's balance inflated across multiple conversion calls, allowing the total ICP burned per hour to far exceed the 150 × 10¹⁵ cycles/hour cap.

The rate limit exists to bound the rate at which ICP is burned and new cycles enter the economy. Bypassing it allows an attacker to burn arbitrarily large amounts of ICP per hour, potentially manipulating ICP supply and the cycles economy. The `total_cycles_minted` metric is also permanently undercounted, breaking auditability. [6](#0-5) 

---

### Likelihood Explanation

The attack requires the attacker to possess cycles (to pre-load the CMC). Cycles can be obtained by converting ICP through the CMC itself (within the rate limit), or purchased on secondary markets. The cost is the cycles spent, but those cycles are not destroyed — they are transferred to the target canister via the CMC's normal delivery path. The attacker effectively "recycles" existing cycles to suppress the limiter's counter, making the attack economically viable for any party with a meaningful cycle balance. No privileged access, governance majority, or subnet compromise is required.

---

### Recommendation

The limiter should be checked against `cycles` (the total amount to deliver), not against `cycles_to_mint` (the delta to newly mint). The CMC's pre-existing balance is irrelevant to the rate-limiting policy, which is intended to cap the total ICP burned per time window:

```rust
fn ensure_balance(cycles: Cycles, limiter_to_use: CyclesMintingLimiterSelector) -> Result<(), String> {
    let now = now_system_time();
    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    let cycles_to_mint = cycles - current_balance;

    with_state_mut(|state| {
        // Check against `cycles` (total delivered), not `cycles_to_mint` (delta minted)
        limiter_to_use.check_and_add_cycles(state, now, cycles)?;
        state.total_cycles_minted += cycles_to_mint;
        Ok::<_, String>(())
    })?;

    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);
    Ok(())
}
```

Alternatively, the CMC should not accept unsolicited incoming cycles (e.g., by burning any cycles received outside of its own minting flow), so that its balance cannot be externally inflated.

```
// Attacker canister pseudocode (Motoko/Rust canister on any subnet)

// Step 1: Pre-load the CMC with enough cycles to cover the desired conversion.
// The rate limit is 150e15 cycles/hour. We want to bypass it entirely.
let target_cycles: u128 = 150_000_000_000_000_000; // 150e15 cycles

// Send cycles to the CMC via inter-canister call (any canister can do this).
// The CMC has no guard against receiving cycles.
ic_cdk::api::call::call_with_payment128(
    CMC_CANISTER_ID,
    "notify_top_up",  // or any update method — cycles are attached to the call
    (dummy_arg,),
    target_cycles,    // attach cycles to the call; CMC accepts them
).await;

// Step 2: Now trigger ICP→cycles conversions.
// Each call to notify_top_up / notify_mint_cycles will see:
//   current_balance >= cycles  =>  cycles_to_mint = 0
//   limiter records 0 against the 150e15/hour cap
// So the cap is never hit, regardless of how many ICP are burned.

// Before attack:
// CMC balance: B
// Limiter count: L (near cap)
// ensure_balance(Y): cycles_to_mint = Y - B, limiter rejects if L + (Y-B) > cap

// After pre-loading CMC with X >= Y:
// CMC balance: B + X
// ensure_balance(Y): cycles_to_mint = Y - (B+X) = 0 (saturating)
// limiter.check_and_add_cycles(0) => always passes
// ICP is burned, Y cycles delivered, limiter untouched

async fn do_mint_cycles(
    account: Account,
    cycles: Cycles,
    deposit_memo: Option<Vec<u8>>,
) -> Result<CyclesLedgerDepositResult, String> {
    let Some(cycles_ledger_canister_id) = with_state(|state| state.cycles_ledger_canister_id)
    else {
        return Err("No cycles ledger canister id configured.".to_string());
    };
    // Always use base cycles limit for minting cycles, since the Subnet Rental Canister
    // doesn't call endpoints using this function.
    ensure_balance(cycles, CyclesMintingLimiterSelector::BaseLimit)?;

    let arg = CyclesLedgerDepositArgs {
        to: account,
        memo: deposit_memo,
    };

    let result: CallResult<(CyclesLedgerDepositResult,)> = ic_cdk::api::call::call_with_payment128(
        cycles_ledger_canister_id.get().0,
        "deposit",
        (arg,),
        u128::from(cycles),
    )
    .await;

    result.map(|r| r.0).map_err(|(code, msg)| {
        format!(
            "Cycles ledger rejected deposit call with code {}: {:?}",
            code as i32, msg
        )
    })
}

    pub fn check_and_add_cycles(
        &mut self,
        now: SystemTime,
        cycles_to_mint: Cycles,
        limit: Cycles,
    ) -> Result<(), String> {
        self.purge_old(now);
        let count = self.get_count();

        if count + cycles_to_mint > limit {
            LIMITER_REJECT_COUNT.with(|count| {
                count.set(count.get().saturating_add(1));
            });

            return Err(format!(
                "More than {} cycles have been minted in the last {} seconds, please try again later.",
                limit,
                self.get_max_age().as_secs(),
            ));
        }
        self.add(now, cycles_to_mint);
        Ok(())
    }
