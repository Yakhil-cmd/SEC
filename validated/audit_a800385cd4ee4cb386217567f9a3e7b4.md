### Title
Unchecked Exit-Precompile Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Loss When Precompile Is Paused — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20` and `EvmErc20V2` burn the caller's tokens **before** calling the exit precompile via inline assembly. The assembly `call` return value is captured in `res` but is **never checked**. When the Aurora Engine pauses `EXIT_TO_NEAR` or `EXIT_TO_ETHEREUM` (a legitimate operational action), the precompile call fails silently while the burn is already committed, permanently destroying user funds with no recovery path.

---

### Finding Description

In both `EvmErc20.sol` and `EvmErc20V2.sol`, `withdrawToNear` and `withdrawToEthereum` follow this pattern:

1. **Burn first**: `_burn(_msgSender(), amount)` — irreversibly removes tokens from the EVM side.
2. **Call precompile via assembly** — the EVM `call` opcode stores its success flag in `res`.
3. **`res` is never validated** — no `require`, no `if iszero(res) { revert(...) }`.

```solidity
// EvmErc20.sol lines 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}
``` [1](#0-0) 

The same pattern appears in `withdrawToEthereum` and is replicated verbatim in `EvmErc20V2.sol`: [2](#0-1) [3](#0-2) 

The Aurora Engine exposes a first-class pause mechanism for exactly these two precompiles via `PrecompileFlags` in `engine/src/pausables.rs`:

```rust
pub struct PrecompileFlags: u32 {
    const EXIT_TO_NEAR        = 0b01;
    const EXIT_TO_ETHEREUM    = 0b10;
}
``` [4](#0-3) 

When a precompile is paused, the engine rejects any EVM `call` to its address, returning failure (`res = 0`). Because `EvmErc20` never inspects `res`, the outer transaction succeeds: tokens are burned, no NEAR-side transfer is scheduled, and the user's funds are gone.

The precompile addresses hardcoded in the Solidity contracts match the constants defined in the engine:

- `EXIT_TO_NEAR`: `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`
- `EXIT_TO_ETHEREUM`: `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing/loss of funds.**

Any user who calls `withdrawToNear` or `withdrawToEthereum` while the corresponding exit precompile is paused will have their ERC-20 tokens irreversibly burned on the Aurora (EVM) side with no corresponding NEP-141 credit on NEAR. There is no refund mechanism and no way to recover the burned tokens.

---

### Likelihood Explanation

**Medium.** Pausing exit precompiles is a documented, intentional operational capability (e.g., used during security incidents or bridge maintenance). Users are not notified of the paused state at the contract level and may attempt withdrawals at any time. The `EvmErc20` contract itself provides no guard against this scenario. Any user interacting with a bridged ERC-20 token during a pause window is at risk.

---

### Recommendation

Check the return value of every assembly `call` to an exit precompile and revert if it fails, so the burn is also reverted:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum` and to `EvmErc20V2`. Alternatively, restructure the functions to call the precompile **before** burning tokens (check-effects-interactions order), so a precompile failure reverts the entire call without any state change.

---

### Proof of Concept

1. The Aurora Engine operator calls `pause_precompiles(PrecompileFlags::EXIT_TO_NEAR)` — a routine security action.
2. A user, unaware of the pause, calls `withdrawToNear(recipient, 1000)` on an `EvmErc20` token.
3. `_burn(_msgSender(), 1000)` executes and commits — 1000 tokens are destroyed.
4. The assembly `call` to `0xe9217bc7...` returns `res = 0` because the engine rejects the paused precompile.
5. No `revert` or `require` checks `res`; the Solidity function returns normally.
6. The user has lost 1000 tokens permanently: burned on Aurora, never minted on NEAR. [7](#0-6) [1](#0-0)
