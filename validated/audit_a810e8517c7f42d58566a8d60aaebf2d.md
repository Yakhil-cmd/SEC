### Title
Contract-Level Pause Blocks All User Fund Exits, Causing Temporary Fund Freeze - (File: `engine/src/contract_methods/evm_transactions.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

When the Aurora Engine contract is paused via `pause_contract`, the `require_running` guard blocks `submit`, `submit_with_args`, `call`, and `withdraw`. Because `EXIT_TO_NEAR` and `EXIT_TO_ETHEREUM` are Aurora-specific precompiles invoked exclusively through EVM transactions submitted via `submit`/`submit_with_args`, pausing the contract simultaneously freezes every user's ability to exit funds to NEAR or Ethereum. This is a direct analog to the reported finding: a pause mechanism that is intended to halt new activity also inadvertently blocks the one class of operations (fund exits) that users need most during a maintenance or security event.

---

### Finding Description

Aurora Engine has two independent pause mechanisms:

**1. Contract-level pause** (`state.is_paused`):
`pause_contract` sets `state.is_paused = true`. Every mutative entry point then calls `require_running`, which returns `ERR_PAUSED` if the flag is set. [1](#0-0) 

The affected entry points include:

- `submit` / `submit_with_args` — the only way to execute EVM transactions on Aurora
- `call` — the privileged EVM call path
- `deploy_code`
- `withdraw` — the direct ETH-to-NEAR withdrawal path
- `ft_on_transfer` — the NEP-141 token receipt path [2](#0-1) [3](#0-2) 

**2. Precompile-level pause** (`PrecompileFlags`):
Independently, `pause_precompiles` can set `EXIT_TO_NEAR` and/or `EXIT_TO_ETHEREUM` bits. When set, the precompile dispatcher returns a fatal `ERR_PAUSED` error for those addresses. [4](#0-3) [5](#0-4) 

**The critical gap**: `EXIT_TO_NEAR` and `EXIT_TO_ETHEREUM` are the only mechanisms for EVM users to move their funds out of Aurora. Both are invoked as precompile calls inside EVM transactions, which means they can only be reached through `submit` or `submit_with_args`. When the contract-level pause is active, `submit` is blocked unconditionally, so no exit is possible regardless of the precompile-level pause state. [6](#0-5) 

---

### Impact Explanation

**High — Temporary freezing of funds.**

During any period in which the owner invokes `pause_contract` (e.g., for maintenance, a security investigation, or an upgrade), every Aurora user loses the ability to:

- Exit ETH or ERC-20 tokens to NEAR via the `EXIT_TO_NEAR` precompile.
- Exit ETH to Ethereum via the `EXIT_TO_ETHEREUM` precompile.
- Withdraw ETH directly via the `withdraw` entry point.

All user balances remain locked inside the Aurora EVM state for the entire duration of the pause. The funds are not stolen, but they are inaccessible to their owners, satisfying the "temporary freezing of funds" impact class.

---

### Likelihood Explanation

**Medium.** The owner legitimately pauses the contract during maintenance windows or security incidents. The Aurora mainnet contract (`aurora`) has been paused in the past. The longer the pause, the more severe the impact. No attacker action is required; the freeze is an automatic consequence of the design whenever the owner exercises a routine administrative function.

---

### Recommendation

Two mitigations mirror those in the referenced report:

1. **Preferred**: Exempt exit operations from the contract-level pause. Specifically, `submit`/`submit_with_args` should be allowed to proceed when the transaction's target is the `EXIT_TO_NEAR` or `EXIT_TO_ETHEREUM` precompile address, and `withdraw` should not call `require_running`. New deposits and general EVM execution can remain blocked.

2. **Alternative**: Introduce a grace period after `resume_contract` during which users can exit before any new state-changing operations are accepted, giving users time to react.

---

### Proof of Concept

1. Owner calls `pause_contract` on the Aurora Engine contract. [7](#0-6) 

2. User constructs a signed Ethereum transaction calling the `EXIT_TO_NEAR` precompile address (`exit_to_near::ADDRESS`) and submits it via `submit`.

3. `submit` calls `require_running(&state)`: [8](#0-7) 

4. `require_running` returns `Err(ERR_PAUSED)` because `state.is_paused == true`: [1](#0-0) 

5. The transaction is rejected. The user's funds remain locked in Aurora EVM state with no exit path available. The same outcome occurs for `withdraw`: [9](#0-8) 

6. The freeze persists until the owner calls `resume_contract`. All users are affected simultaneously for the entire pause duration.

            relayer_address,
            handler,
        )?;

        io.return_output(&result_bytes);

        Ok(result)
    })

#[named]
pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
