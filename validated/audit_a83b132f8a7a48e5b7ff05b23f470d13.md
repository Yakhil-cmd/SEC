Audit Report

## Title
VIP Caller Starvation via Slot Exhaustion in `try_borrow_slot` — (`rs/nervous_system/clients/src/management_canister_client.rs`)

## Summary
`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` rejects all callers — including VIPs — when `available_slot_count == 0`, before the VIP's `used_slot_count = 0` branch has any effect. An unprivileged attacker who floods NNS Root's public `canister_status` endpoint with 167 concurrent calls exhausts all slots, causing NNS Governance's subsequent calls (`update_canister_settings`, `stop_canister`, etc.) to receive `SysTransient` errors and fail.

## Finding Description
In `rs/nervous_system/clients/src/management_canister_client.rs` at lines 264–287, `try_borrow_slot` sets `used_slot_count = 0` for VIP callers (line 265), intending that VIPs never consume slots. However, the `if *available_slot_count == 0` guard at line 269 fires unconditionally for all callers before the subtraction at line 278 is reached. When the counter is exactly 0, VIPs are rejected identically to non-VIPs.

The attack surface is `canister_status` in `rs/nns/handlers/root/impl/canister/canister.rs` at lines 88–98: it is a public `#[update]` with no access control. Any caller can invoke it. Each call goes through `new_management_canister_client()` (lines 53–67), which creates a `LimitedOutstandingCallsManagementCanisterClient` with `is_caller_vip = false` for non-NNS callers, consuming one slot per outstanding call.

`AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` is initialized to 167 (line 50). An attacker sending 167 concurrent `canister_status` calls exhausts the pool. Governance-only operations — `update_canister_settings` (line 221), `take_canister_snapshot` (line 245), `load_canister_snapshot` (line 259) in `canister.rs`, and `stop_canister` / `delete_canister` (lines 326–340 of `management_canister_client.rs`) — all call `try_borrow_slot()` against the same thread-local and are rejected with `Err((SysTransient, "Unavailable. Maybe, try again later?"))`.

`SlotLoan::drop` (lines 365–372 of `management_canister_client.rs`) restores `used_slot_count` slots on completion. Non-VIP slots are restored after the management canister responds (~1–2 rounds), so the attacker must repeat the flood each round to maintain exhaustion.

## Impact Explanation
This is an application/platform-level DoS against NNS governance operations. An unprivileged attacker can indefinitely delay or stall `update_canister_settings`, `stop_canister`, `take_canister_snapshot`, `load_canister_snapshot`, and `delete_canister` calls from NNS Governance to NNS Root. These are the exact calls used to execute NNS proposals for canister upgrades, controller changes, and emergency stops — including security patches and recovery actions. This matches the **High** impact category: "Application/platform-level DoS… or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
The attack requires only a canister on the IC (accessible to anyone with cycles) and the ability to send 167 concurrent inter-canister calls to NNS Root's `canister_status` per round. The IC's per-canister outgoing call queue limit is 500, so 167 concurrent calls is well within reach. The attack window per batch is ~1–2 seconds (one management canister round-trip), but it is repeatable and fully automatable. The cycle cost is non-trivial but not prohibitive for a motivated attacker. No special privileges, social engineering, or external compromise is required.

## Recommendation
In `try_borrow_slot`, skip the `== 0` rejection for VIP callers:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if *available_slot_count == 0 && !self.is_caller_vip {
                let code = RejectCode::SysTransient as i32;
                let message = "Unavailable. Maybe, try again later?".to_string();
                return Err((code, message));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

This ensures VIP callers always pass through regardless of slot counter state, while non-VIP callers remain rate-limited.

## Proof of Concept
State-machine / PocketIC integration test:
1. Deploy NNS Root with `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT = 167`.
2. Deploy an attacker canister with a non-NNS principal.
3. From the attacker canister, send 167 concurrent inter-canister calls to NNS Root's `canister_status` targeting any NNS-controlled canister; hold the calls open (e.g., by targeting a canister that delays its management canister response).
4. Before any response returns, send an `update_canister_settings` call from NNS Governance (VIP principal).
5. Assert the Governance call returns `Err((SysTransient, "Unavailable. Maybe, try again later?"))`.
6. Allow the 167 attacker calls to complete; assert the slot counter recovers to 167 and a subsequent Governance call succeeds.
