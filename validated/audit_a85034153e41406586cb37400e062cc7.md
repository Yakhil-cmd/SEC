Audit Report

## Title
Silent ICP Burn Failure Breaks ICP/Cycles Conservation Invariant After Successful Cycles Minting - (File: rs/nns/cmc/src/main.rs)

## Summary
The `burn_and_log` function in the Cycles Minting Canister unconditionally swallows all errors from the ICP ledger burn call. When called after a successful cycles mint or deposit, a failed burn results in cycles existing on-chain without a corresponding ICP destruction, permanently breaking the ICP/cycles conservation invariant. The notification is then marked complete, closing any retry path.

## Finding Description
Three flows call `burn_and_log` only after the primary operation has already irreversibly succeeded:

- `process_top_up` (L2001): after `deposit_cycles` returns `Ok(())`
- `process_create_canister` (L1945): after `do_create_canister` returns `Ok(canister_id)`
- `process_mint_cycles` (L1968): after `do_mint_cycles` returns `Ok(deposit_result)`

`burn_and_log` (L2017–2049) calls `call_protobuf(ledger_canister_id, "send_pb", send_args).await` and on any `Err` variant only prints a log line — no error is returned, no state is recorded, no retry is scheduled. The design rationale is explicit in the comment at L2015: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."*

After `process_top_up` / `process_create_canister` / `process_mint_cycles` return, the caller (e.g., `notify_top_up` at L1214–1222) unconditionally inserts `NotificationStatus::NotifiedTopUp(result)` into `blocks_notified`. The `is_transient_error` guard only removes the entry for errors propagated from the process function itself — a `burn_and_log` failure is invisible to this guard because `burn_and_log` returns `()` regardless. The notification is therefore permanently closed.

No grep match exists for any `retry_burn`, `failed_burns`, or `pending_burn` state in the CMC codebase, confirming there is no deferred recovery path.

## Impact Explanation
When `burn_and_log` fails: cycles have been irreversibly deposited to a canister or minted to the cycles ledger; the corresponding ICP remains in the CMC's subaccount (keyed by canister ID or controller principal) and is permanently inaccessible to any party; the notification is closed so no user-initiated retry is possible; and no on-chain record of the discrepancy exists beyond a canister log print. This constitutes illegal cycles minting (cycles created without ICP destruction) and permanent loss of ICP locked in the CMC, directly harming the ICP/cycles conservation invariant that underpins the economic integrity of the NNS and the cycles economy. This matches the **High** impact class: *"Significant NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

## Likelihood Explanation
The ICP ledger undergoes routine upgrades via NNS governance proposals. During an upgrade the ledger canister is temporarily unavailable and rejects inter-canister calls. The window between a successful `deposit_cycles`/`do_create_canister`/`do_mint_cycles` and the subsequent `burn_and_log` call crosses an `await` point, meaning the ledger could be mid-upgrade at exactly that moment. Any unprivileged ICP holder calling `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` during such a window triggers the condition. No special privileges, front-running, or coordination is required. The condition is repeatable across every ledger upgrade cycle and accumulates silently over time.

## Recommendation
Record failed burn attempts in stable CMC state (subaccount + amount) before returning from `burn_and_log`. Implement a periodic timer or heartbeat task that retries pending burns against the ledger. Alternatively, expose a privileged `retry_burn` endpoint callable by NNS governance to drain the pending-burn queue. At minimum, emit a certified metric or stable-memory counter of failed burns so that the conservation invariant can be audited and corrected via governance proposal. The existing double-mint concern is already addressed by the `blocks_notified` deduplication map, so propagating or recording the burn failure does not reintroduce that risk.

## Proof of Concept
1. Deploy a local replica with the NNS (CMC + ICP ledger).
2. Send ICP to the CMC subaccount for a target canister and call `notify_top_up`.
3. Intercept execution between `deposit_cycles` returning `Ok` and `burn_and_log` completing by stopping/upgrading the ICP ledger canister (simulated via PocketIC's `stop_canister` on the ledger before the await resumes, or by injecting a test hook that pauses the ledger).
4. Observe: `deposit_cycles` credited cycles to the target canister; `burn_and_log` logs a failure; `notify_top_up` returns `Ok(cycles)`; `blocks_notified` contains `NotifiedTopUp(Ok(cycles))`; the CMC subaccount for the canister ID still holds the original ICP amount; no retry is ever attempted.
5. Confirm the invariant violation: query the target canister's cycle balance (increased) and the CMC subaccount balance (non-zero, permanently locked).
