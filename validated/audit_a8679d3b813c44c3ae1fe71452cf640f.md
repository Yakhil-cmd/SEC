### Title
`receive_erc20_tokens` Trusts Callback-Supplied Amount Without Before/After Balance Verification, Enabling Insolvency via Fee-on-Transfer NEP-141 Tokens - (File: `engine/src/engine.rs`)

---

### Summary

`receive_erc20_tokens` in `engine/src/engine.rs` mints ERC-20 tokens equal to `args.amount` as supplied by the NEP-141 `ft_on_transfer` callback, without verifying the actual NEP-141 balance change in Aurora's account. Because `deploy_erc20_token` has no access control, any user can register a fee-on-transfer NEP-141 token. When such a token is used, Aurora mints more ERC-20 than the NEP-141 it actually holds, causing insolvency: the ERC-20 total supply permanently exceeds the backing NEP-141 reserve.

---

### Finding Description

The `ft_on_transfer` entry point dispatches to `receive_erc20_tokens` for any registered NEP-141 token that is not the base ETH connector: [1](#0-0) 

Inside `receive_erc20_tokens`, the amount to mint is taken directly from the callback argument with no balance snapshot before or after: [2](#0-1) 

The minted amount is passed verbatim to the ERC-20 `mint` selector: [3](#0-2) 

The `deploy_erc20_token` function that registers a NEP-141 ↔ ERC-20 mapping has no owner or access-control guard — only a liveness check: [4](#0-3) 

Any NEAR account can therefore register an arbitrary NEP-141 token, including one with a transfer fee.

---

### Impact Explanation

**Impact: Critical — Insolvency / Permanent freezing of funds.**

When a fee-on-transfer NEP-141 token is used:

- Aurora's NEP-141 balance increases by `amount − fee`.
- Aurora mints `amount` ERC-20 tokens.
- The ERC-20 total supply permanently exceeds the NEP-141 reserve.
- The last holders of the ERC-20 token cannot exit: the `ft_transfer` call in the `ExitToNear` precompile will fail because Aurora's NEP-141 balance is insufficient, permanently freezing their funds.

---

### Likelihood Explanation

**Likelihood: Medium.**

`deploy_erc20_token` is callable by any unprivileged NEAR account with no restriction beyond the contract being in a running state. An attacker can:

1. Deploy a custom fee-on-transfer NEP-141 token on NEAR.
2. Call `deploy_erc20_token` on Aurora to register it.
3. Repeatedly call `ft_transfer_call` to inflate the ERC-20 supply beyond the NEP-141 reserve.

No privileged access, leaked keys, or governance capture is required.

---

### Recommendation

In `receive_erc20_tokens`, record Aurora's NEP-141 balance before the `ft_transfer_call` completes and use the actual balance delta — not `args.amount` — as the mint amount. Because `ft_on_transfer` is a synchronous callback on NEAR, the balance has already changed by the time the function executes; a cross-contract `ft_balance_of` query (or a stored pre-transfer snapshot) can be used to compute the true received amount and mint only that.

---

### Proof of Concept

1. Attacker deploys `FeeToken`, a NEP-141 contract that silently deducts 10 % on every transfer (receiver gets 90 % of the transferred amount).
2. Attacker calls `deploy_erc20_token` on Aurora with `FeeToken`'s account ID — succeeds with no access check.
3. Attacker calls `FeeToken.ft_transfer_call(receiver_id=aurora, amount=1000, msg=<attacker_evm_address>)`.
   - `FeeToken` transfers 1 000 from attacker, credits Aurora with 900, then calls `aurora.ft_on_transfer(sender=attacker, amount=1000, msg=...)`.
4. `receive_erc20_tokens` reads `args.amount = 1000` and mints 1 000 ERC-20 tokens to the attacker's EVM address.
5. Aurora holds 900 `FeeToken` NEP-141 but has issued 1 000 ERC-20.
6. Attacker calls `withdrawToNear` (ExitToNear precompile) for 1 000 ERC-20 — Aurora attempts `FeeToken.ft_transfer(amount=1000)` but only holds 900, so the last 100 units of ERC-20 held by any user are permanently frozen. [5](#0-4) [6](#0-5)

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
