Audit Report

## Title
Self-Transfer Balance Accounting Bug in ICP Rosetta Balance Book - (File: rs/rosetta-api/icp/ledger_canister_blocks_synchronizer/src/blocks.rs)

## Summary
In `update_balance_book_execution`, the `None` branch of the `Operation::Transfer` self-transfer path records `amount` as the new balance without deducting the transaction fee. When a Rosetta node syncs from a non-zero block index and an account's first appearance in the synced range is a self-transfer, the balance book stores an inflated value that is then served directly by the `/account/balance` endpoint to exchanges and financial integrations.

## Finding Description
At lines 449–463 of `rs/rosetta-api/icp/ledger_canister_blocks_synchronizer/src/blocks.rs`, the `Operation::Transfer` branch queries the `to` account's prior balance and detects self-transfers:

```rust
let account_balance_opt = extract_latest_balance(to)?;
let self_transfer = from.to_hex() == to.to_hex();
match account_balance_opt {
    Some(mut balance) => {
        balance.1 += amount.get_e8s();
        if self_transfer {
            balance.1 -= amount.get_e8s();
            balance.1 -= fee.get_e8s();
        }
        new_balances.push(balance);
    }
    None => {
        new_balances.push((to.to_hex(), amount.get_e8s()));  // BUG: no fee deduction, no self-transfer check
    }
}
```

The `Some` branch correctly handles self-transfers: it adds then subtracts `amount` (net zero) and deducts `fee`. The `None` branch — reached when the account has no prior entry in the SQLite DB — does not check `self_transfer` at all. It unconditionally records `amount` as the new balance. Additionally, the `if !self_transfer` debit block at line 464 is skipped, so no debit is applied to the `from` side either.

A self-transfer semantically requires a pre-existing balance (the on-chain ledger enforces this). The correct net effect is `old_balance - fee`. The `None` branch instead writes `amount`, which is an arbitrary value chosen by the sender, as the account's balance.

The `/account/balance` endpoint in `rs/rosetta-api/icp/src/request_handler.rs` at line 190 reads directly from this balance book via `blocks.get_account_balance`, which executes `SELECT tokens FROM account_balances WHERE block_idx<=?1 AND account=?2 ORDER BY block_idx DESC LIMIT 1`. There is no secondary validation against the on-chain ledger state.

## Impact Explanation
This matches the allowed High impact: "Significant Rosetta security impact with concrete user or protocol harm." The Rosetta `/account/balance` endpoint is the primary interface used by exchanges to query ICP balances. An exchange querying this endpoint after the bug is triggered receives `amount` (attacker-controlled) as the account balance instead of the correct `old_balance - fee`. The attacker can set `amount` to any value up to their actual on-chain balance. An exchange that credits or permits withdrawal based on this inflated Rosetta balance suffers direct financial loss. The balance book is the sole source of truth for the Rosetta node's balance responses.

## Likelihood Explanation
The `None` branch is reached when `extract_latest_balance(to)` finds no row for the account at or before the current block index. This occurs when a Rosetta node is configured to start syncing from a non-zero block index (a documented operational mode), so accounts with balances established before the sync start point have no DB entries. Any unprivileged ICP user can submit a self-transfer; the only precondition is that the Rosetta node's DB lacks a prior entry for their account, which is a realistic operational state after partial sync or database reset. The `prune_account_balances` function (lines 540–577) does not create this condition on its own (it retains the latest entry per account), but a fresh sync from a non-genesis block does.

## Recommendation
The `None` branch must explicitly reject self-transfers, since a self-transfer requires a pre-existing balance and the absence of a DB entry for a self-transfer indicates a sync gap that cannot be safely resolved by recording `amount`:

```rust
None => {
    if self_transfer {
        return Err(BlockStoreError::Other(format!(
            "Self-transfer from account {} with no prior balance at block {}",
            to, hb.index
        )));
    }
    new_balances.push((to.to_hex(), amount.get_e8s()));
}
```

## Proof of Concept
1. Configure a Rosetta node to start syncing from block N > 0 (so no balance entries exist for accounts active only before block N).
2. Account A holds a real ICP balance established before block N (no entry in the Rosetta DB).
3. Account A submits a self-transfer (`from=A`, `to=A`, `amount=X`, `fee=F`) at block N+1.
4. The Rosetta node syncs block N+1 and calls `update_balance_book_execution`.
5. `extract_latest_balance(A)` returns `None` (no prior DB entry).
6. The `None` branch executes: `new_balances.push((A.to_hex(), X))` — fee `F` is never deducted, and `X` is attacker-chosen.
7. The Rosetta DB now records A's balance as `X`.
8. A call to `/account/balance` for account A returns `X` instead of the correct `old_balance - F`.
9. An exchange crediting or permitting withdrawal based on this value suffers loss proportional to `X - (old_balance - F)`.

A unit test can reproduce this by constructing a `HashedBlock` containing a self-transfer `Operation::Transfer` with `from == to`, calling `update_balance_book_execution` against an empty DB (simulating a non-genesis sync start), and asserting that the function returns an error rather than inserting `amount` as the balance.
