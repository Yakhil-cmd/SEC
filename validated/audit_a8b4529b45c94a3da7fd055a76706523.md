Audit Report

## Title
Global Pending-Withdrawal Queue Saturation Enables Sustained DoS on All ckETH/ckERC20 Withdrawals - (File: `rs/ethereum/cketh/minter/src/guard/mod.rs`)

## Summary
The ckETH minter enforces a single global cap of `MAX_PENDING = 100` on pending withdrawal requests with no per-principal sub-limit. An attacker controlling 100 distinct IC principals can fill `pending_withdrawal_requests` entirely, causing every subsequent `withdraw_eth` or `withdraw_erc20` call from any user to `ic_cdk::trap` (hard canister error) until the attacker's Ethereum transactions confirm. Because the attacker recovers their ETH after each cycle, the attack can be repeated indefinitely at the cost of Ethereum gas fees alone.

## Finding Description

**Root cause — `rs/ethereum/cketh/minter/src/guard/mod.rs` (L9-10, L50-54)**

`MAX_PENDING` is set to 100 and the gate in `Guard::new` checks only the global queue length:

```rust
pub const MAX_PENDING: usize = 100;

fn new(principal: Principal) -> Result<Self, GuardError> {
    mutate_state(|s| {
        if PR::pending_requests_count(s) >= MAX_PENDING {   // global gate, no per-user split
            return Err(GuardError::TooManyPendingRequests);
        }
```

`PendingWithdrawalRequests::pending_requests_count` delegates to `state.eth_transactions.withdrawal_requests_len()`, which returns `self.pending_withdrawal_requests.len()` — a count of every entry in the shared `VecDeque<WithdrawalRequest>` regardless of submitter.

**Shared queue covers both token types — `rs/ethereum/cketh/minter/src/state/transactions/mod.rs` (L35-39, L929-931)**

`WithdrawalRequest` is a single enum holding both `CkEth` and `CkErc20` variants in one `VecDeque`. Flooding with one token type blocks the other.

**Trap on guard failure — `rs/ethereum/cketh/minter/src/main.rs` (L274-278, L401-405)**

Both public endpoints call `ic_cdk::trap` (not `return Err(...)`) when the guard is denied:

```rust
let _guard = retrieve_withdraw_guard(caller).unwrap_or_else(|e| {
    ic_cdk::trap(format!(
        "Failed retrieving guard for principal {caller}: {e:?}"
    ))
});
```

This produces a hard `CANISTER_ERROR` rejection. The ckETH burn occurs *after* the guard, so user funds are not lost, but the withdrawal is completely blocked.

**Slow queue drain — `rs/ethereum/cketh/minter/src/withdraw.rs` (L39)**

The minter processes only `WITHDRAWAL_REQUESTS_BATCH_SIZE = 5` requests per timer tick. A request moves out of `pending_withdrawal_requests` only when a transaction is created for it. The attacker can re-submit to refill the queue as slots open.

**Exploit flow:**
1. Attacker generates 100 IC principals `P_1…P_100`, each funded with ≥ `cketh_minimum_withdrawal_amount` ckETH.
2. Each principal calls `withdraw_eth` once. The `AlreadyProcessing` check does not fire (different principals); the `MAX_CONCURRENT` check does not fire (100 concurrent in-flight calls is the limit, and these can be submitted sequentially). Each call burns ckETH and enqueues a `WithdrawalRequest`.
3. After all 100 calls complete, `withdrawal_requests_len() == 100`.
4. Any subsequent `withdraw_eth` or `withdraw_erc20` call from any principal hits `pending_requests_count(s) >= MAX_PENDING` and traps.
5. The minter drains 5 requests per tick; the attacker re-submits to keep the queue saturated.
6. After Ethereum confirmation, the attacker's ETH is returned. Repeat indefinitely.

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

All ckETH and ckERC20 withdrawals are completely blocked for the duration of each attack cycle. Legitimate users receive `CANISTER_ERROR` on every withdrawal attempt. The ckETH minter is an in-scope financial integration; blocking all withdrawals constitutes a concrete, sustained platform-level DoS with direct user harm (inability to exit ck-token positions).

## Likelihood Explanation

- **No privileged access required.** Any IC user can generate 100 key pairs trivially.
- **Capital requirement is modest and fully recovered.** 100 × minimum withdrawal amount (≈ 0.03 ETH each) ≈ 3 ETH (~$10,000), returned after each Ethereum confirmation cycle.
- **Sustained cost is gas fees only** (~$500/cycle at current prices), making indefinite DoS economically feasible.
- **Repeatable.** As the minter drains 5 requests per tick, the attacker re-submits to maintain saturation.
- **No on-chain complexity.** Only standard IC ingress calls to `withdraw_eth` are required.

## Recommendation

1. **Add a per-principal pending-request cap** inside `Guard::new`: count how many entries in `pending_withdrawal_requests` belong to the calling principal and reject if it exceeds a small limit (e.g., 3).
2. **Return a graceful error instead of trapping** when `TooManyPendingRequests` is hit, so callers receive a retryable `TemporarilyUnavailable` response rather than a hard `CANISTER_ERROR`.
3. **Raise `MAX_PENDING`** to a value that makes queue saturation economically prohibitive, or tie it to a per-principal sub-quota.
4. **Separate the ckETH and ckERC20 pending counters** so that flooding one token type cannot block the other.

## Proof of Concept

**Deterministic integration test plan (safe, local replica / PocketIC):**

1. Initialize a local minter canister with `MAX_PENDING = 100`.
2. Create 100 test principals, each with sufficient ckETH balance.
3. Submit `withdraw_eth` from each principal sequentially; assert all 100 succeed and `withdrawal_requests_len() == 100`.
4. Submit one additional `withdraw_eth` from a fresh principal; assert the call traps with a message containing `TooManyPendingRequests`.
5. Advance the timer by one tick (processes 5 requests); assert `withdrawal_requests_len() == 95`.
6. Submit 5 new requests from attacker principals; assert `withdrawal_requests_len() == 100` again, confirming re-saturation.
7. Assert that legitimate user calls continue to trap throughout steps 4–6.
