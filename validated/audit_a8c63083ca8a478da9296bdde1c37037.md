Audit Report

## Title
Unbounded Synchronous Loop in `create_sns_neuron_recipes` Traps After Lock Commit, Permanently Blocking SNS Swap Finalization - (File: rs/sns/swap/src/swap.rs)

## Summary

`create_sns_neuron_recipes` iterates over all direct buyers and all Neurons' Fund participants in a single synchronous execution with no instruction-limit guard. It is invoked from `finalize_inner` after at least two `await` boundaries (`sweep_icp`, `settle_neurons_fund_participation`), meaning the `finalize_swap_in_progress` lock has already been committed to stable canister state in a prior message execution. If the synchronous scan exhausts the IC instruction budget and traps, the IC rolls back only the current message's state changes — the lock, committed earlier, remains set. Every subsequent call to `finalize_swap` returns immediately with a lock-held error, leaving the swap permanently stuck in `Lifecycle::Committed` until a canister upgrade clears the lock.

## Finding Description

**Lock acquisition and commitment path:**

`finalize` acquires the lock before any async work: [1](#0-0) 

The lock is committed to canister state when the first inter-canister call (`sweep_icp`) is dispatched at the first `await` boundary: [2](#0-1) 

A second `await` follows for `settle_neurons_fund_participation`: [3](#0-2) 

After both awaits, `create_sns_neuron_recipes` is called **synchronously** — no `await`, no message boundary: [4](#0-3) 

**Unbounded loops with no instruction guard:**

Loop 1 iterates over every direct buyer, calling `create_sns_neuron_basket_for_direct_participant` (hashing, struct allocation, `Vec::extend`) per participant: [5](#0-4) 

Loop 2 iterates over every Neurons' Fund participant and each of their `cf_neurons`: [6](#0-5) 

A grep for `is_over_instructions_limit`, `performance_counter`, or any instruction-budget check in `rs/sns/swap/src/swap.rs` returns zero matches — confirmed by search above.

**Second unbounded scan in `claim_swap_neurons`:**

Before its first `await`, `claim_swap_neurons` builds a full `claimable_neurons_index` over all `neuron_recipes` synchronously: [7](#0-6) 

This runs after three prior `await` boundaries, so the same lock-commit problem applies.

**Code acknowledges the panic/trap risk but provides no mitigation:** [8](#0-7) 

**Scale of the problem:**

The participant cap `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` is enforced at participation time: [9](#0-8) 

The integration test confirms the combined neuron count is bounded only by a memory ceiling, not an instruction ceiling: [10](#0-9) 

`MAX_NEURONS_FUND_PARTICIPANTS = 5,000` is defined in NNS governance: [11](#0-10) 

## Impact Explanation

If the trap occurs:
1. The IC rolls back only the current message execution's state (partial `neuron_recipes` writes, counter increments).
2. The `finalize_swap_in_progress = true` lock, committed in the first message execution, is **not** rolled back.
3. Every subsequent `finalize_swap` call hits the lock check and returns `"Finalize swap is already in progress"`.
4. The swap is permanently stuck in `Lifecycle::Committed`; direct participants cannot receive SNS neurons; ICP already swept to SNS governance cannot be reclaimed by participants.
5. Recovery requires a canister upgrade — a governance action that takes time and may not be immediately available.

This matches the allowed High impact: **Application/platform-level DoS with concrete user-fund harm** and **Significant SNS security impact with concrete user or protocol harm**.

## Likelihood Explanation

- `finalize_swap` is a public update call with no access control — any principal can invoke it.
- The trigger condition (swap near `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` direct participants plus large NF participation) is realistic for high-profile SNS launches, which are the exact use case the SNS framework is designed for.
- The participant cap is a memory bound, not an instruction bound; no benchmarks exist to confirm the synchronous scan stays within the IC's per-message instruction limit at maximum scale.
- The attack requires no special privilege: participate in the swap (or simply call `finalize_swap` as a third party after the swap commits).

## Recommendation

1. Add an `is_over_instructions_limit()` (or `ic_cdk::api::instruction_counter()`) check inside both loops in `create_sns_neuron_recipes`. Use the existing `has_created_neuron_recipes` idempotency flag to persist partial progress and return a partial `SweepResult` so the caller can retry across multiple messages.
2. Apply the same chunked-processing pattern to the pre-loop in `claim_swap_neurons` (lines 1675–1697), splitting index construction across messages if needed.
3. Implement a post-upgrade hook that unconditionally clears `finalize_swap_in_progress` — the comment at line 1528 acknowledges this need but it is not implemented.
4. Add an instruction-count benchmark for `create_sns_neuron_recipes` at maximum participant counts, analogous to the existing memory-bound test in `constraints_dependencies.rs`.

## Proof of Concept

1. Deploy an SNS swap with `min_participant_icp_e8s` at minimum and `neuron_basket_construction_parameters.count` set to a large value (e.g., 10).
2. Fill the swap to near `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` direct participants and ensure maximum NF participation (`MAX_NEURONS_FUND_PARTICIPANTS = 5,000`).
3. Allow the swap to reach `Lifecycle::Committed`.
4. Call `finalize_swap` from any principal.
5. `sweep_icp` completes (first `await`) → lock is committed to canister state.
6. `settle_neurons_fund_participation` completes (second `await`).
7. `create_sns_neuron_recipes` executes synchronously over the full participant set → instruction limit exhausted → IC traps → current message state rolled back.
8. Observe: `finalize_swap_in_progress = true` persists in canister state; all subsequent `finalize_swap` calls return the lock-held error.
9. Swap is permanently stuck; participant funds cannot be distributed without a canister upgrade.

A deterministic integration test using PocketIC can reproduce this by setting participant counts to the maximum values and asserting that `finalize_swap_in_progress` remains `true` after a simulated instruction-limit trap in the `create_sns_neuron_recipes` execution round.
