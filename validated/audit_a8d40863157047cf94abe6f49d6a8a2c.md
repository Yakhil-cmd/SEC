### Title
Hardcoded `L1_TX_NATIVE_PRICE` Constant Used Instead of On-Chain `native_price` for L1→L2 Transaction Resource Accounting — (`File: basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

In `prepare_and_check_resources`, the native resource limit for every L1→L2 (priority) transaction is derived using the hardcoded compile-time constant `L1_TX_NATIVE_PRICE = 10` instead of the actual `native_price` read from the block-metadata oracle. For L2 transactions the code correctly calls `system.get_native_price()`. The discrepancy means that whenever the operator-set `native_price` diverges from 10, L1→L2 transactions receive a native resource budget that does not reflect the real proving cost, enabling either over-consumption of prover resources (DoS on the prover) or under-allocation that causes legitimate L1→L2 transactions to fail (DoS on the bridge).

---

### Finding Description

**Vulnerable code path** — `prepare_and_check_resources` in `process_l1_transaction.rs`:

```rust
// For L1->L2 txs, we use a constant native price to avoid censorship.
let native_price = L1_TX_NATIVE_PRICE;   // ← hardcoded = 10
let native_per_gas = ...
    u256_try_to_u64(&gas_price.div_ceil(native_price))...;
let native_prepaid_from_gas = native_per_gas.checked_mul(gas_limit)...;
``` [1](#0-0) 

The constant is defined with an open TODO:

```rust
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
``` [2](#0-1) 

**Correct path for L2 transactions** — `validation_impl.rs` reads the live oracle value:

```rust
let native_price = system.get_native_price();   // ← reads oracle
...
u256_try_to_u64(&gas_price.div_ceil(native_price))...
``` [3](#0-2) 

`system.get_native_price()` delegates to `ZkSpecificPricingMetadata::native_price()`, which returns the `native_price` field of `BlockMetadataFromOracle` — a value supplied by the operator and committed to as a public input for the prover. [4](#0-3) [5](#0-4) 

The `native_per_gas` ratio drives the entire native resource budget:

```
native_limit = gas_limit × (gas_price / native_price)
``` [6](#0-5) 

Because `L1_TX_NATIVE_PRICE` is fixed at 10 regardless of the operator-set value, the budget is wrong whenever the two values differ.

---

### Impact Explanation

**Case 1 — actual `native_price` > 10 (production scenario where proving is expensive):**

`native_per_gas_L1 = gas_price / 10` vs. `native_per_gas_correct = gas_price / actual_native_price`.

The ratio of over-allocation is `actual_native_price / 10`. If the operator sets `native_price = 1000` (a plausible production value), every L1→L2 transaction receives **100× more native resource budget** than it paid for. An attacker can submit L1→L2 transactions with the maximum gas limit and computationally heavy calldata, consuming native resources (RISC-V proving cycles) at 100× the rate they paid for. Because L1→L2 transactions cannot be rejected by the sequencer (they come from the priority queue), the prover is forced to process them. Sustained submission of such transactions constitutes a **DoS on the prover / block-proving pipeline**.

**Case 2 — actual `native_price` < 10 (cheap proving environment):**

`native_per_gas_L1 = gas_price / 10` is lower than the correct value. Legitimate L1→L2 transactions that would succeed under the correct budget run out of native resources and revert, causing a **DoS on the L1→L2 bridge** without any attacker action beyond normal usage.

Both directions arise from the same root cause: a hardcoded constant substituted for the live oracle value.

---

### Likelihood Explanation

The `native_price` is an operator-controlled parameter that is expected to track real proving costs. The codebase's own documentation states: *"nativePrice [is] a constant set by the operator, reflecting the 'cost of processing a single cycle'. Note: for L1→L2 transactions we use a code constant instead of one provided by operator."* [7](#0-6) 

The open TODO in the constant's definition confirms the team has not yet determined a correct value. Any production deployment where the operator tunes `native_price` away from exactly 10 (which is the default test value, not a production-calibrated figure) triggers the discrepancy. The attacker entry path requires only the ability to submit L1→L2 transactions — an unprivileged, permissionless action on L1.

---

### Recommendation

Replace the hardcoded constant with the live oracle value, mirroring the L2 transaction path:

```rust
// Before (vulnerable):
let native_price = L1_TX_NATIVE_PRICE;

// After (correct):
let native_price = system.get_native_price();
```

If censorship-resistance requires a floor, apply a `max(system.get_native_price(), CENSORSHIP_FLOOR)` rather than ignoring the oracle entirely. The TODO at `EVM-1157` should be resolved before production deployment.

---

### Proof of Concept

1. Operator deploys ZKsync OS with `native_price = 1000` (reflecting real proving costs).
2. Attacker submits an L1→L2 priority transaction with:
   - `gas_price = 1000`
   - `gas_limit = MAX_TX_GAS_LIMIT` (e.g., ~28M)
   - Calldata that maximises EVM computation (e.g., tight loop of expensive opcodes)
3. `prepare_and_check_resources` computes:
   - `native_per_gas = 1000 / 10 = 100` (hardcoded)
   - `native_limit = 28_000_000 × 100 = 2_800_000_000`
4. Correct computation would give:
   - `native_per_gas = 1000 / 1000 = 1`
   - `native_limit = 28_000_000 × 1 = 28_000_000`
5. The transaction is admitted to the priority queue on L1 and **cannot be rejected** by the sequencer. The prover must prove 100× more native resource consumption than the attacker paid for.
6. Repeating this attack with many L1→L2 transactions exhausts prover capacity, halting block production. [1](#0-0) [2](#0-1)
