### Title
Missing Native Resource Charge in All EIP-2537 BLS12-381 Precompiles Allows Prover Resource Exhaustion - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/`)

### Summary

All seven EIP-2537 BLS12-381 precompile implementations hardcode `cost_native = 0`, meaning they charge zero native resource (the metric for RISC-V proving cost) despite performing computationally intensive elliptic-curve operations. An unprivileged transaction sender can call these precompiles repeatedly within a single block, paying only EVM gas while consuming unbounded prover cycles, causing the block to become unprovable despite being sequencer-valid.

### Finding Description

ZKsync OS implements a double resource accounting model: **Ergs** (EVM gas) and **Native** (RISC-V proving cycles). The native resource is the primary DoS protection for the prover — if a transaction exhausts it, the transaction reverts; if a block exhausts it, the block cannot be proven.

All seven BLS12-381 precompiles introduced by EIP-2537 hardcode `cost_native = 0`:

**`Bls12381G1AdditionPrecompile`** and **`Bls12381G2AdditionPrecompile`**: [1](#0-0) [2](#0-1) 

**`Bls12381G1MSMPrecompile`** and **`Bls12381G2MSMPrecompile`**: [3](#0-2) [4](#0-3) 

**`Bls12381PairingCheckPrecompile`**: [5](#0-4) 

**`Bls12381G1MappingPrecompile`** and **`Bls12381G2MappingPrecompile`**: [6](#0-5) [7](#0-6) 

This is in direct contrast to every other computationally expensive precompile, which charges non-zero native costs. For example, BN254 pairing charges `BN254_PAIRING_BASE_NATIVE_COST = native_with_delegations!(13_000_000, 500_000, 0)` per pair: [8](#0-7) 

The native resource limit per transaction is derived from `gasLimit * nativePerGas`, and the system enforces a `MAX_NATIVE_COMPUTATIONAL` cap per block: [9](#0-8) 

Because BLS12-381 precompiles charge `cost_native = 0`, the native resource counter is never decremented by these calls. The sequencer accepts the block (EVM gas is correctly charged), but the actual RISC-V cycle cost of proving the BLS12-381 operations is not bounded by the native resource budget.

The attacker entry path is direct: any EVM transaction can call precompile addresses `0x0b`–`0x11`: [10](#0-9) 

### Impact Explanation

An attacker submits a block-filling transaction (or many transactions) that repeatedly call `BLS12_PAIRING_CHECK` (address `0x0f`) or `BLS12_G2MSM` (address `0x0e`) with maximum valid input. Each call pays EVM gas (e.g., 37,700 + 32,600 × N for pairing) but zero native resource. The sequencer processes the block successfully. When the prover attempts to generate a ZK proof, the actual RISC-V cycle count far exceeds the block's native resource budget, making the block **unprovable**. This constitutes **valid-execution unprovability** — a critical integrity failure in the ZK rollup's trust model. The sequencer and prover diverge: the sequencer committed to a block that cannot be proven, stalling the chain.

### Likelihood Explanation

The EIP-2537 precompiles are enabled via the `eip-2537` feature flag and are reachable by any unprivileged transaction sender. No special role, key, or governance access is required. The attacker only needs to submit a standard EVM transaction calling address `0x0f` (BLS12-381 pairing) with a large valid input. The cost to the attacker is only EVM gas; the cost to the prover is unbounded RISC-V cycles. This is a straightforward, low-cost griefing attack.

### Recommendation

Assign non-zero native resource costs to all EIP-2537 BLS12-381 precompiles, calibrated against actual RISC-V cycle benchmarks (analogous to how `BN254_PAIRING_BASE_NATIVE_COST` and `BN254_PAIRING_PER_PAIR_NATIVE_COST` are defined). At minimum, the pairing check and MSM operations must have per-pair native costs proportional to their proving complexity. The `cost_native` constants should be added to `basic_system/src/cost_constants.rs` alongside the existing BN254 native cost constants. [11](#0-10) 

### Proof of Concept

1. Deploy a contract that calls `address(0x0f).call{gas: gasleft()}(input)` in a loop, where `input` is a valid BLS12-381 pairing input with the maximum number of pairs that fit within the EVM gas limit.
2. Submit the transaction with a gas limit of, e.g., 30,000,000 (block gas limit).
3. The sequencer accepts the block: EVM gas is consumed correctly, native resource counter remains near its initial value (since `cost_native = 0` for each pairing call).
4. The prover attempts to prove the block. The actual RISC-V cycle count for the BLS12-381 multi-pairing operations vastly exceeds the block's native resource budget.
5. The prover fails to generate a valid proof, stalling the rollup.
