Audit Report

## Title
Wasm Trap in `canister_status` Callback Rolls Back `SlotLoan::drop`, Permanently Leaking Slots — (`rs/nns/handlers/root/impl/canister/canister.rs`)

## Summary

The NNS root canister's public `canister_status` update endpoint calls `.unwrap()` on the management canister result after the `SlotLoan` RAII guard has already been dropped within the same callback execution. Because the IC execution environment rolls back all heap mutations when a response callback traps, the slot restoration performed by `SlotLoan::drop` is undone. An unprivileged caller can exhaust all 167 slots by repeatedly triggering this trap, permanently disabling the `canister_status` endpoint for all non-NNS principals until the canister is upgraded.

## Finding Description

**Public entry point with no access control:**

The `canister_status` endpoint is a plain `#[update]` with no `check_caller_is_*` guard. [1](#0-0) 

Any principal can invoke it with an arbitrary `CanisterIdRecord`.

**Slot borrow in Message 1:**

`new_management_canister_client()` wraps the inner client in a `LimitedOutstandingCallsManagementCanisterClient` with `is_caller_vip = false` for non-NNS callers. [2](#0-1) 

`try_borrow_slot` decrements `available_slot_count` by 1 and returns a `SlotLoan`. This decrement is committed to the heap at the end of Message 1 (before the first `await`). [3](#0-2) 

**`SlotLoan::drop` restores the slot — but in Message 2:**

`_loan` is a local variable in `LimitedOutstandingCallsManagementCanisterClient::canister_status`. In Rust, it is dropped at the end of that function's scope, which is after `self.inner.canister_status(...).await` returns but before the result is observed by the caller. Both the drop and the subsequent `.unwrap()` panic occur within the same IC callback execution (Message 2). [4](#0-3) [5](#0-4) 

**The trap site:**

When the management canister rejects the call (NNS root is not the controller of the target canister), `client.canister_status(...).await` returns `Err(...)`. After `_loan` is dropped (slot incremented), `.unwrap()` panics, producing a Wasm trap. [6](#0-5) 

**IC rollback of Message 2:**

The IC execution environment confirms this behavior: when a response callback traps, the pre-callback `clean_canister` state is used and all heap mutations from that callback are discarded. [7](#0-6) 

**Execution sequence in Message 2:**
```
[heap at start of callback: available_slot_count = N-1]
  1. inner.canister_status returns Err(...)
  2. _loan is dropped → available_slot_count incremented to N   ← heap mutation
  3. Err propagates to canister.rs::canister_status
  4. .map(...) → still Err
  5. .unwrap() panics → Wasm TRAP
  6. IC rolls back Message 2 heap → available_slot_count reverts to N-1
```

**Slot exhaustion:**

The pool starts at 167 slots. After 167 trapped callbacks, `available_slot_count` reaches 0. [8](#0-7) 

Once exhausted, `try_borrow_slot` returns `Err(SysTransient, "Unavailable. Maybe, try again later?")` for all non-VIP callers. [9](#0-8) 

VIP callers (NNS canisters) use `used_slot_count = 0` and are unaffected. [10](#0-9) 

## Impact Explanation

After 167 trapped callbacks, every non-VIP call to `canister_status` on the NNS root canister returns a `SysTransient` error immediately without making any inter-canister call. The public `canister_status` endpoint — which is documented as intentionally public ("the status of NNS canisters should be public information: anyone can get the status of any NNS canister") — becomes permanently non-functional for all non-NNS principals until the canister is upgraded or restarted. This matches the allowed impact: **High ($2,000–$10,000) — Significant NNS infrastructure security impact with concrete user or protocol harm**, specifically application-level DoS of a public NNS endpoint triggered without any privileged access.

## Likelihood Explanation

- The `canister_status` endpoint is public with no access control.
- Triggering a management canister rejection is trivial: pass any canister ID that NNS root does not control (e.g., any user-owned canister or a freshly created canister).
- Only 167 update calls are required. Each costs only standard ingress fees.
- No privileged access, key material, subnet-majority corruption, or social engineering is required.
- The attack is fully deterministic and repeatable.

## Recommendation

Replace `.unwrap()` with proper error propagation so the endpoint returns a rejection to the caller instead of trapping:

```rust
// Instead of:
canister_status_response.unwrap()

// Use (CDK trap encodes a reject, not a Wasm trap that rolls back state):
canister_status_response.map_err(|(_, msg)| ic_cdk::api::trap(&msg))
// or propagate as a Result<CanisterStatusResult, String> return type
```

The root cause is that `.unwrap()` must never be called on a `Result` that can be `Err` inside a response callback when the `Err` path has already performed heap mutations (via `SlotLoan::drop`) that must be preserved. The correct fix is to ensure the function never traps after the slot is borrowed — either by propagating the error or by using `ic_cdk::api::trap` (which encodes a canister reject rather than a Wasm execution trap that triggers rollback).

## Proof of Concept

1. Obtain any canister ID that NNS root does not control (e.g., a freshly created user canister `X`).
2. Send 167 update calls to NNS root's `canister_status` endpoint with `CanisterIdRecord { canister_id: X }`.
3. For each call: management canister rejects (NNS root is not controller of `X`) → `Err` returned → `_loan` dropped (slot incremented) → `.unwrap()` panics → Wasm TRAP → IC rolls back Message 2 → slot reverts to N-1.
4. On the 168th call from any non-VIP principal, observe `Err(SysTransient, "Unavailable. Maybe, try again later?")` returned immediately without any inter-canister call being made.

A deterministic integration test using `StateMachine` or PocketIC can reproduce this by: (a) creating a canister not controlled by NNS root, (b) sending 167 `canister_status` calls to NNS root with that canister ID, (c) asserting that the 168th call returns the `SysTransient` unavailability error.
