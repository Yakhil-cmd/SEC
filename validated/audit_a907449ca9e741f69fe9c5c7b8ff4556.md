Audit Report

## Title
SNS Swap Capacity Permanently Locked When Remaining Participation Falls Below Per-Participant Minimum - (File: rs/sns/swap/src/swap.rs)

## Summary

In `refresh_buyer_token_e8s`, the available capacity (`max_increment_e8s`) is silently capped before the post-cap minimum check, but the per-participant ceiling is applied after the cap. When `available_direct_participation_e8s()` drops below `min_participant_icp_e8s` and all existing participants are already at `max_participant_icp_e8s`, the remaining capacity becomes permanently inaccessible: new participants are rejected by the post-cap minimum check, and existing participants at the ceiling produce no net change. A sophisticated actor can deliberately trigger this condition to sabotage an SNS swap.

## Finding Description

The sequential logic in `refresh_buyer_token_e8s` is:

1. **Line 1177**: `max_increment_e8s = available_direct_participation_e8s()` — captures remaining global capacity.
2. **Line 1202**: Pre-cap check: rejects if `e8s < min_participant_icp_e8s`. A new participant depositing exactly `min_participant_icp_e8s` passes this.
3. **Line 1224**: `actual_increment_e8s = min(max_increment_e8s, requested_increment_e8s)` — silently reduces the increment to the remaining capacity.
4. **Line 1225**: `new_balance_e8s = old_amount_icp_e8s + actual_increment_e8s`.
5. **Line 1237**: `new_balance_e8s = min(new_balance_e8s, max_participant_icp_e8s)` — clamps to per-participant ceiling.
6. **Line 1241**: Post-cap check: rejects if `new_balance_e8s < min_participant_icp_e8s`.

When `available_direct_participation_e8s() < min_participant_icp_e8s`:
- A **new participant** (`old_amount_icp_e8s = 0`) who deposits `min_participant_icp_e8s` passes step 2, but step 3 caps their increment to the smaller remaining amount, so `new_balance_e8s < min_participant_icp_e8s` → rejected at step 6.
- An **existing participant at `max_participant_icp_e8s`** who tries to top up: step 5 clamps `new_balance_e8s` back to `max_participant_icp_e8s`, producing zero net change.

The `compute_participation_increment` function (line 3240) already implements the correct guard — `if user_participation.saturating_add(max_available_increment) < min_user_participation { return Err((0, 0)); }` — but `refresh_buyer_token_e8s` does not apply equivalent logic.

The existing test `test_swap_cannot_finalize_via_new_participation_if_remaining_lt_minimal_participation_amount` sets `user_1_first_participation_amount_icp_e8s = 400_000` against `max_participant_icp_e8s = 500_000`, so user 1 is below the per-participant ceiling and can still fill the gap. The test does not cover the case where all participants are already at `max_participant_icp_e8s`.

## Impact Explanation

This matches **High ($2,000–$10,000): Significant SNS security impact with concrete user or protocol harm.**

- The SNS swap can never reach `max_direct_participation_icp_e8s`, preventing early finalization.
- If the locked remainder is the difference between current participation and `min_direct_participation_icp_e8s`, the swap aborts at the deadline despite near-sufficient participation, causing the entire SNS launch to fail.
- ICP transferred to swap subaccounts by would-be participants is inaccessible until the swap closes and `error_refund_icp` is called.
- The final SNS token price is distorted by the sub-target participation.

## Likelihood Explanation

The condition is triggered whenever `max_direct_participation_icp_e8s mod max_participant_icp_e8s < min_participant_icp_e8s`. These parameters are set at SNS initialization via NNS governance proposal and are immutable after the swap opens. Any unprivileged actor who monitors the public on-chain swap state can deliberately be the last participant to bring the total to exactly `max_direct - r` (where `r < min_participant`), intentionally locking the remaining capacity. No special privileges are required beyond the ability to participate in the swap.

## Recommendation

After computing `actual_increment_e8s` and before the post-cap minimum check at line 1241, add a guard analogous to `compute_participation_increment`:

- If `old_amount_icp_e8s == 0` (new participant) and `max_increment_e8s < min_participant_icp_e8s`, return a descriptive error immediately rather than silently capping and then rejecting.
- If `old_amount_icp_e8s > 0` (existing participant), allow the top-up as long as `new_balance_e8s >= min_participant_icp_e8s`, even if the increment itself is below `min_participant_icp_e8s`.

This mirrors the guard already present in `compute_participation_increment` at line 3240:
```rust
if user_participation.saturating_add(max_available_increment) < min_user_participation {
    return Err((0, 0));
}
```

## Proof of Concept

```
Swap parameters:
  max_direct_participation_icp_e8s = 1_050_000 * E8
  max_participant_icp_e8s          =   500_000 * E8
  min_participant_icp_e8s          =   100_000 * E8

Step 1: User A calls refresh_buyer_tokens with ledger balance 500_000 * E8.
        → accepted; buyer state = 500_000 * E8.

Step 2: User B calls refresh_buyer_tokens with ledger balance 500_000 * E8.
        → accepted; buyer state = 500_000 * E8.
        → available_direct_participation_e8s() = 50_000 * E8.

Step 3: User C (new) transfers 100_000 * E8 to their swap subaccount,
        then calls refresh_buyer_tokens.
        → e8s = 100_000 * E8 >= min_participant_icp_e8s ✓ (pre-cap check passes)
        → actual_increment = min(50_000, 100_000) = 50_000 * E8
        → new_balance = 0 + 50_000 = 50_000 * E8 < 100_000 * E8
        → ERROR: "Rejecting participation of effective amount 50000e8s;
                  minimum required to participate: 100000e8s"

Step 4: User A transfers 50_000 * E8 more, calls refresh_buyer_tokens.
        → new_balance = min(500_000 + 50_000, 500_000) = 500_000 * E8
        → no change in buyer state; 50_000 * E8 permanently locked.

Result: 50_000 * E8 of swap capacity is permanently inaccessible.
        The swap cannot reach max_direct_participation_icp_e8s = 1_050_000 * E8.
```

A unit test extending `test_swap_cannot_finalize_via_new_participation_if_remaining_lt_minimal_participation_amount` with `user_1_first_participation_amount_icp_e8s = max_participant_icp_e8s` and a second participant also at `max_participant_icp_e8s` would reproduce the deadlock deterministically.
