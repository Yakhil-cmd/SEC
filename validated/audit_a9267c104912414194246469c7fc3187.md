Audit Report

## Title
Delegated `Disburse`/`DisburseMaturity` Permission Defaults Destination to Caller's Account, Enabling Fund Redirection - (File: rs/sns/governance/src/governance.rs)

## Summary
In SNS governance, `disburse_neuron` and `disburse_maturity` resolve the default destination account using `caller.0` rather than the neuron owner's principal. Because `NeuronPermissionType::Disburse` and `NeuronPermissionType::DisburseMaturity` are independently grantable to arbitrary principals via `AddNeuronPermissions`, a delegated principal can omit `to_account` and receive the neuron's entire stake or maturity into their own wallet — not the neuron owner's.

## Finding Description
In `disburse_neuron` at lines 1142–1147 of `rs/sns/governance/src/governance.rs`, after passing the authorization check at line 1127 (`neuron.check_authorized(caller, NeuronPermissionType::Disburse)`), the destination account is resolved as:

```rust
// If no account was provided, transfer to the caller's (default) account.
let to_account = match disburse.to_account.as_ref() {
    None => Account {
        owner: caller.0,   // ← caller, not neuron owner
        subaccount: None,
    },
    ...
};
```

The identical pattern exists in `disburse_maturity` at lines 1618–1623. The authorization check only verifies that the caller holds the `Disburse` or `DisburseMaturity` permission on the neuron — it does not require the caller to be the neuron owner. Because `AddNeuronPermissions` (lines 4570–4640) allows any principal holding `ManagePrincipals` to grant these permissions to an arbitrary third party, the caller and the neuron owner can be entirely different principals.

By contrast, NNS governance's `disburse_neuron` gates the entire function behind `is_controlled_by(caller)` (lines 1970–1978 of `rs/nns/governance/src/governance.rs`), making the caller-as-default-destination safe there. SNS deliberately separates `Disburse` into a standalone grantable permission, breaking that invariant.

Exploit path:
1. Alice owns a dissolved SNS neuron with 100 SNS tokens.
2. Alice grants `NeuronPermissionType::Disburse` to Bob via `AddNeuronPermissions`.
3. Bob calls `manage_neuron` with `Command::Disburse { amount: None, to_account: None }`.
4. `check_authorized` passes (Bob holds `Disburse`).
5. `to_account` resolves to `Account { owner: Bob, subaccount: None }`.
6. The ledger transfer sends ~100 SNS tokens to Bob's account.
7. Alice's neuron stake is zero; the transfer is irreversible.

The same path applies to `DisburseMaturity` for accrued maturity.

## Impact Explanation
This constitutes unauthorized access to neuron-controlled SNS ledger funds. A delegated principal can irreversibly drain a neuron's entire stake or maturity to their own account without the neuron owner's consent. This matches the allowed High impact: *"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."* The constraint is that the attacker must hold a granted `Disburse` or `DisburseMaturity` permission, which is a realistic precondition given the SNS documentation and tooling actively encourage permission delegation to dapp frontends, co-signers, and automation bots.

## Likelihood Explanation
`Disburse` and `DisburseMaturity` are explicitly listed as grantable permission types in the SNS proto definition and are included in `neuron_grantable_permissions`. The SNS permission model is designed and documented to encourage delegation. A neuron owner who grants `Disburse` to a dapp canister or automation bot for operational convenience would not expect that omitting `to_account` redirects funds to the delegate. The attack requires no privileged access beyond a granted permission, no key compromise, and no consensus manipulation — only a standard `manage_neuron` ingress call.

## Recommendation
When `to_account` is `None`, resolve the destination to the neuron owner's account rather than the caller's. The neuron owner can be identified as the principal holding `ManagePrincipals` (or the original claimer). Alternatively, require `to_account` to be explicitly specified whenever the caller does not hold `ManagePrincipals`:

```rust
// disburse_neuron — suggested fix
let to_account = match disburse.to_account.as_ref() {
    None => {
        // Default to the neuron owner, not the caller
        let owner = neuron.owner_principal()?;
        Account { owner: owner.0, subaccount: None }
    }
    Some(ai_pb) => Account::try_from(ai_pb.clone())...
};
```

Apply the same fix to `disburse_maturity`.

## Proof of Concept
A deterministic unit test in `rs/sns/governance/tests/governance.rs` can reproduce this:
1. Create a governance fixture with a dissolved neuron owned by principal `Alice`.
2. Call `add_neuron_permissions` to grant `NeuronPermissionType::Disburse` to principal `Bob`.
3. Invoke `disburse_neuron` with `caller = Bob`, `disburse = Disburse { amount: None, to_account: None }`.
4. Assert that the ledger transfer destination is `Account { owner: Bob, ... }` (demonstrating the bug) rather than `Account { owner: Alice, ... }` (the expected behavior).
5. Confirm Alice's neuron stake is now zero and Bob's ledger balance increased by the neuron's stake minus fees.
