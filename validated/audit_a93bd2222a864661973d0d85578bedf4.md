### Title
Missing Content Validation of `neuron_basket_construction_parameters` in SNS Swap `Init::validate()` — (File: `rs/sns/swap/src/types.rs`)

### Summary
The `Init::validate()` function in the SNS swap canister checks that `neuron_basket_construction_parameters` is present but does **not** validate its content (`count >= 2`, `dissolve_delay_interval_seconds > 0`). That content validation lives exclusively in `Params::validate()` and `SnsInitPayload::validate_neuron_basket_construction_params()`, which are called in separate, external code paths. This is the direct IC analog of the Atlendis pattern: validation logic is separated from the initializer and relies on an assumption that an upstream caller has already validated the data.

### Finding Description

`Init::validate()` in `rs/sns/swap/src/types.rs` performs the following checks: [1](#0-0) 

It calls `validate_swap_init_for_one_proposal_flow()`, which in turn calls `check_mandatory_swap_fields_are_set()`: [2](#0-1) 

This only asserts that `neuron_basket_construction_parameters` is **not `None`**. It does not inspect the values of `count` or `dissolve_delay_interval_seconds`.

The actual content constraints — `count >= 2` and `dissolve_delay_interval_seconds > 0` — are enforced only in `Params::validate()`: [3](#0-2) 

And in `SnsInitPayload::validate_neuron_basket_construction_params()`: [4](#0-3) 

In the one-proposal flow, `Swap::new(init)` is the constructor. It calls `init.validate()` (which panics on failure) but never calls `Params::validate()`: [5](#0-4) 

`Params::try_from(&init)` is called inside `Swap::new()` to populate `res.params`, but this conversion also performs no content validation of `neuron_basket_construction_parameters`: [6](#0-5) 

The downstream consumer of these parameters, `generate_vesting_schedule()`, does check `count == 0` at call time: [7](#0-6) 

But by that point the swap is already in the `Adopted` or `Open` state with participant funds locked in.

The pattern is duplicated: `SnsInitPayload::validate_neuron_basket_construction_params()` in the SNS init library is the upstream guard, but the swap canister's own `Init::validate()` does not replicate these checks. The swap canister's correctness depends on the assumption that the deployer (NNS governance) has already validated the content.

### Impact Explanation

If a swap canister is initialized with `neuron_basket_construction_parameters.count = 0` or `dissolve_delay_interval_seconds = 0`, `Swap::new()` succeeds and the swap enters the `Adopted`/`Open` lifecycle. When `create_sns_neuron_recipes()` is later called during finalization, `generate_vesting_schedule()` returns an error for `count == 0`, permanently blocking finalization. ICP contributed by participants and SNS tokens held in escrow become locked in the canister with no recovery path. For `dissolve_delay_interval_seconds = 0`, all neurons in every basket receive `dissolve_delay_seconds = 0`, defeating the vesting schedule entirely and making all SNS tokens immediately liquid — undermining the economic design of the decentralization swap.

### Likelihood Explanation

The current NNS governance deployment path does validate these fields via `SnsInitPayload::validate_neuron_basket_construction_params()` before the swap canister is installed. However, the pattern is error-prone for the same reason identified in the Atlendis report: a developer who adds a new deployment path, a canister upgrade path, or a test harness that constructs `Swap` directly from an `Init` (as already done in integration tests with `count: 1`) will find that `Init::validate()` passes without complaint. The integration test at `rs/sns/integration_tests/src/sns_treasury.rs` already demonstrates this: [8](#0-7) 

`count: 1` violates the `count >= 2` invariant enforced by `Params::validate()`, yet the swap canister accepts it because `Init::validate()` does not check it.

### Recommendation

- **Short term:** Add content validation of `neuron_basket_construction_parameters` directly inside `Init::validate()` in `rs/sns/swap/src/types.rs`, mirroring the checks already present in `Params::validate()` (lines 427–453) and `SnsInitPayload::validate_neuron_basket_construction_params()` (lines 1466–1475 of `rs/sns/init/src/lib.rs`).
- **Long term:** Follow the principle stated in the Atlendis report: keep validation logic as close as possible to where variables are declared and used. The swap canister's `Init::validate()` should be the authoritative, self-contained validator for all fields in `Init`, not a partial check that relies on upstream callers having already validated content.

### Proof of Concept

1. Construct an `Init` with `neuron_basket_construction_parameters: Some(NeuronBasketConstructionParameters { count: 0, dissolve_delay_interval_seconds: 0 })` and all other required fields valid.
2. Call `Swap::new(init)`. `init.validate()` passes because `check_mandatory_swap_fields_are_set()` only checks `is_none()`.
3. The swap enters `Adopted` state. `Params::try_from(&init)` stores the invalid basket parameters in `res.params` without error.
4. Participants contribute ICP; the swap reaches `Committed`.
5. `create_sns_neuron_recipes()` is called. It reads `params.neuron_basket_construction_parameters` and calls `generate_vesting_schedule(amount)`. Because `count == 0`, `generate_vesting_schedule` returns `Err("NeuronBasketConstructionParameters.count must be greater than zero")`.
6. `create_sns_neuron_recipes()` increments `sweep_result.failure` for every buyer. Finalization cannot complete. All ICP and SNS tokens remain locked in the swap canister permanently. [9](#0-8) [1](#0-0) [10](#0-9)

    fn generate_vesting_schedule(
        &self,
        total_amount_e8s: u64,
    ) -> Result<Vec<ScheduledVestingEvent>, String> {
        if self.count == 0 {
            return Err(
                "NeuronBasketConstructionParameters.count must be greater than zero".to_string(),
            );
        }

        let dissolve_delay_seconds_list = (0..(self.count))
            .map(|i| i * self.dissolve_delay_interval_seconds)
            .collect::<Vec<u64>>();

        let chunks_e8s = apportion_approximately_equally(total_amount_e8s, self.count)?;
        Ok(dissolve_delay_seconds_list
            .into_iter()
            .zip(chunks_e8s)
            .map(
                |(dissolve_delay_seconds, amount_e8s)| ScheduledVestingEvent {
                    dissolve_delay_seconds,
                    amount_e8s,
                },
            )
            .collect())
    }
