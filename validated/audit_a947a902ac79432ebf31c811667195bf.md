### Title
Stale wNEAR ERC-20 Approval After `factory_set_wnear_address` Update Breaks All XCC Calls Requiring NEAR Payment - (File: engine-precompiles/src/xcc.rs)

### Summary

The XCC precompile requires each user to pre-approve the XCC precompile address (`cross_contract_call::ADDRESS`) to spend their wNEAR ERC-20 tokens via `transferFrom`. These approvals are recorded on a specific ERC-20 contract instance. The owner can update the wNEAR ERC-20 address at any time via `factory_set_wnear_address`. After such an update, the XCC precompile reads the new wNEAR address from storage and calls `transferFrom` on the new contract — but no user has approved the XCC precompile on the new contract. Every XCC call that requires NEAR payment (all first-time calls needing storage staking, and any call with attached NEAR) will revert for every existing user until they individually re-approve the new wNEAR contract.

### Finding Description

The `CrossContractCall::run_with_handle` function in `engine-precompiles/src/xcc.rs` dynamically reads the wNEAR ERC-20 address from storage at call time:

```rust
let wnear_address = state::get_wnear_address(&self.io);  // line 193
```

It then issues an EVM sub-call to that address invoking `transferFrom(sender, engine_implicit_address, required_near)`, with `cross_contract_call::ADDRESS` as the EVM `caller` (i.e., `msg.sender` from the ERC-20's perspective):

```rust
let context = aurora_evm::Context {
    address: wnear_address.raw(),
    caller: cross_contract_call::ADDRESS.raw(),  // line 196
    ...
};
handle.call(wnear_address.raw(), None, tx_data, None, false, &context);  // line 200
```

For `transferFrom` to succeed, the user must have previously called `approve(cross_contract_call::ADDRESS, amount)` on the wNEAR ERC-20 contract. This approval is stored in the ERC-20 contract's own state and is specific to that contract's address.

The owner can replace the wNEAR address at any time via `factory_set_wnear_address` in `engine/src/contract_methods/xcc.rs`:

```rust
xcc::set_wnear_address(&mut io, &Address::from_array(address));  // line 112
```

After this update, the precompile targets the new wNEAR contract. The new contract has zero allowances for the XCC precompile from any user. Every subsequent XCC call where `required_near != ZERO_YOCTO` will revert at the `transferFrom` step.

### Impact Explanation

**High — Temporary freezing of funds.**

After `factory_set_wnear_address` is called with a new wNEAR ERC-20 address (a legitimate protocol upgrade action, e.g., redeploying the wNEAR bridge contract), every existing Aurora user's XCC calls that require NEAR payment will revert. This covers:

- All first-time XCC callers (who need storage staking: `required_near = attached_near + STORAGE_AMOUNT`)
- Any XCC call with non-zero attached NEAR

The XCC precompile is a core Aurora production interface. Users' wNEAR tokens are effectively locked out of the XCC system — they cannot execute cross-contract calls to NEAR — until each user individually re-approves the new wNEAR contract. There is no on-chain mechanism to notify users or batch-reset approvals.

### Likelihood Explanation

**Medium.** The `factory_set_wnear_address` function is a legitimate, documented owner operation. It would be called whenever the wNEAR bridge contract is upgraded or redeployed — a realistic protocol maintenance event. The Aurora Engine has already undergone multiple upgrades (evidenced by the `test_xcc_upgrade` test and the `xcc_router_v1.wasm` artifact). Each such upgrade cycle is a trigger for this issue. No attacker action is required; the disruption occurs automatically upon a routine admin call.

### Recommendation

Before updating the stored wNEAR address, or immediately after, the engine should not silently invalidate all existing user approvals. Two mitigations:

1. **Emit an on-chain event** when `factory_set_wnear_address` is called so that off-chain tooling can alert users to re-approve.
2. **Prefer a migration path** where the old wNEAR address remains valid for a transition period, or the engine documents that all users must re-approve after a wNEAR address change.

A deeper fix would be to avoid relying on per-user ERC-20 allowances for the XCC payment mechanism entirely, instead using a pull-payment or escrow pattern that is not address-sensitive.

### Proof of Concept

1. User Alice calls `approve(cross_contract_call::ADDRESS, 500e24)` on wNEAR contract at address `0xOLD`.
2. Owner calls `factory_set_wnear_address(0xNEW)` — a new wNEAR ERC-20 is deployed and registered.
3. Alice calls the XCC precompile with `CrossContractCallArgs::Eager(...)` and non-zero attached NEAR.
4. Inside `run_with_handle`, `state::get_wnear_address` returns `0xNEW`.
5. The precompile calls `transferFrom(alice, engine_implicit, required_near)` on `0xNEW`.
6. `0xNEW.allowance(alice, cross_contract_call::ADDRESS) == 0` → `transferFrom` reverts.
7. The XCC precompile returns `PrecompileFailure::Revert`, and Alice's transaction fails.
8. Alice (and every other existing user) cannot use XCC until they re-approve `0xNEW`.

#[named]
pub fn factory_set_wnear_address<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        let address = io.read_input_arr20()?;
        xcc::set_wnear_address(&mut io, &Address::from_array(address));
        Ok(())
    })
}
