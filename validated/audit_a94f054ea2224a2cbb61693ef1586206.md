Audit Report

## Title
Missing TLS Accept Timeout on XNet HTTP Endpoint Enables Resource Exhaustion by Malicious Protocol Peer - (File: rs/http_endpoints/xnet/src/lib.rs)

## Summary
The XNet inter-subnet HTTP endpoint (`start_server`) accepts TCP connections and spawns an unbounded number of Tokio tasks, one per connection, with no timeout on the TLS handshake. A registered IC node acting maliciously can open many TCP connections and stall indefinitely at the TLS handshake phase, exhausting Tokio task and OS file-descriptor resources on the target replica, disrupting cross-subnet message routing.

## Finding Description
In `rs/http_endpoints/xnet/src/lib.rs`, `start_server` loops on `listener.accept()` and unconditionally spawns a `tokio::spawn` task per accepted connection:

```
// L230-293
tokio::spawn(async move {
    ...
    match tls_acceptor.accept(stream).await {   // ← NO TIMEOUT
        Ok(tls_stream) => {
            let conn = http.serve_connection(io, hyper_service);
            if let Err(err) = conn.await { ... } // ← NO TIMEOUT
        }
        ...
    };
});
```

`tls_acceptor.accept(stream).await` (L277) blocks until the peer completes the TLS handshake. There is no `tokio::time::timeout` wrapping this call and no `tokio_io_timeout::TimeoutStream` applied to the raw TCP stream. A peer that opens a TCP connection and sends no bytes will hold the spawned task open indefinitely.

The `XNET_ENDPOINT_MAX_CONCURRENT_REQUESTS` semaphore (L54, value 4) is acquired inside `handle_xnet_request` (L156), which is the HTTP application handler — it is only reached after TLS completes. It does not bound the number of concurrently accepted but not-yet-TLS-completed connections.

By contrast, `rs/http_endpoints/public/src/lib.rs` (L451, L473–494) reads a `connection_read_timeout_seconds` config value, applies `timeout(read_timeout, stream.peek(&mut b))` before TLS, and wraps the stream with `tokio_io_timeout::TimeoutStream` afterward. The XNet endpoint has no equivalent protection.

## Impact Explanation
A single registered IC node (Byzantine protocol peer below the consensus fault threshold) can open thousands of TCP connections to the target replica's XNet port and send no bytes. Each accepted connection spawns a Tokio task blocked on `tls_acceptor.accept`. This causes:

1. Unbounded Tokio task accumulation consuming heap memory.
2. OS file-descriptor exhaustion (each open TCP socket holds an FD).
3. Once FDs are exhausted, the XNet endpoint can no longer accept new legitimate connections from other subnets, halting cross-subnet stream slice delivery and stalling cross-subnet message routing.

This matches the allowed High impact: **Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000). The disruption is protocol-level resource exhaustion, not a bandwidth flood.

## Likelihood Explanation
The XNet endpoint is protected by nftables rules restricting inbound connections to IPs of nodes registered in the IC registry. The attacker must therefore be a registered IC node acting maliciously — a Byzantine protocol peer below the consensus fault threshold, which is explicitly within scope. Once network access is established, the attack is trivial: open many TCP connections and send no data. No cryptographic material, no special tooling, and no victim interaction is required. The attack is repeatable and can be sustained indefinitely.

## Recommendation
Apply a read timeout to the raw TCP stream before the TLS handshake in `start_server`, mirroring the pattern in the public HTTP endpoint:

```rust
let mut stream = tokio_io_timeout::TimeoutStream::new(stream);
stream.set_read_timeout(Some(connection_read_timeout));
let stream = Box::pin(stream);
match tls_acceptor.accept(stream).await { ... }
```

Alternatively, wrap the TLS accept with `tokio::time::timeout(connection_read_timeout, tls_acceptor.accept(stream))`. A `connection_read_timeout_seconds` field should be added to the XNet config (analogous to `rs/config/src/http_handler.rs`) and passed into `start_server`.

## Proof of Concept
1. Control a registered IC node (or use an IP that passes the nftables registry check).
2. From that node, open a large number of TCP connections to the target replica's XNet port (configured via `xnet_port`).
3. For each connection, complete the TCP three-way handshake but send no further bytes.
4. Each connection causes `start_server` to spawn a Tokio task blocked at `tls_acceptor.accept(stream).await` (L277) with no timeout.
5. After exhausting OS file descriptors or Tokio task memory, the target replica's XNet endpoint stops accepting new connections, halting cross-subnet stream slice delivery.

A deterministic integration test can reproduce this locally: spawn the `XNetEndpoint` (non-test cfg or a patched build), open N raw TCP connections without sending a TLS ClientHello, and assert that after N connections the endpoint fails to accept a legitimate connection within a bounded time.
