Audit Report

## Title
Fee Double-Deduction in NNS `disburse_neuron` When Caller Specifies Explicit Amount - (File: `rs/nns/governance/src/governance.rs`)

## Summary
When a neuron controller calls `disburse_neuron` with an explicit `amount = Some(X)`, the NNS governance canister subtracts `neuron_fees_e8s` from `X` before computing the disbursement transfer, causing the user to receive `X - fees - tx_fee` instead of the intended `X - tx_fee`. The fees are also burned separately in a distinct ledger transfer, meaning the fee deduction is applied twice: once to reduce what the user receives, and once as an actual on-chain burn. This is confirmed to be inconsistent with the SNS governance implementation of the same operation, which does not subtract fees from the user-specified amount.

## Finding Description
In `rs/nns/governance/src/governance.rs` at lines 2016–2021, the disbursement amount is computed as:

```rust
let mut disburse_amount_e8s = disburse
    .amount
    .as_ref()
    .map_or(neuron_minted_stake_e8s, |a| {
        a.e8s.saturating_sub(fees_amount_e8s)   // fees subtracted from user-specified amount
    });
```

Then at lines 2025–2027, `transaction_fee_e8s` is also subtracted from `disburse_amount_e8s`.

Separately, at lines 2046–2064, `fees_amount_e8s` is burned from the neuron's subaccount via a distinct ledger transfer. At lines 2091–2100, `disburse_amount_e8s` (already reduced by fees) is transferred to the user.

**Concrete accounting for `amount = Some(X)` with `fees > tx_fee`:**
- Burn transfer: `fees` taken from neuron, sent to minting account
- Disburse transfer: `(X - fees - tx_fee)` sent to user, `tx_fee` taken from neuron
- Total taken from neuron: `fees + (X - fees - tx_fee) + tx_fee = X` ✓ (neuron accounting is correct)
- User receives: `X - fees - tx_fee` ✗ (user expected `X - tx_fee`)

The comment at lines 2012–2015 claims "there is symmetry here," but this is false. The `None` path uses `neuron_minted_stake_e8s = cached_stake - fees`, which already accounts for fees — no further fee subtraction occurs. The `Some(X)` path subtracts fees a second time from the user-specified value. If a user passes `X = minted_stake = cached_stake - fees`, the `None` path yields `cached_stake - fees - tx_fee` to the user, while the `Some(minted_stake)` path yields `cached_stake - 2·fees - tx_fee` — `fees` fewer tokens for the same logical request.

By contrast, `rs/sns/governance/src/governance.rs` at lines 1160–1166 correctly uses `a.e8s` directly without fee subtraction, then caps at `neuron.stake_e8s()`.

The existing guard at line 2025 (`if disburse_amount_e8s > transaction_fee_e8s`) only prevents underflow; it does not prevent the fee double-deduction.

## Impact Explanation
Any NNS neuron controller who has accumulated `neuron_fees_e8s > 0` (from rejected governance proposals) and calls `disburse_neuron` with an explicit `amount` suffers a direct, unannounced ICP loss equal to `neuron_fees_e8s`. The lost tokens are burned rather than stolen, but the user receives materially fewer ICP than they specified. This constitutes concrete user financial harm in NNS governance — an explicitly in-scope system — and maps to: **High ($2,000–$10,000): Significant NNS security impact with concrete user or protocol harm.**

## Likelihood Explanation
The preconditions are: (1) the neuron has `neuron_fees_e8s > 0`, which occurs after any rejected governance proposal that charged a fee; (2) the controller calls `disburse_neuron` with `amount = Some(X)` rather than `None`. Both conditions are routine and require no special privileges beyond being the neuron's controller. The bug is deterministic and repeatable for every affected call.

## Recommendation
Remove the fee subtraction from the `Some` branch and align with the SNS implementation. Change lines 2016–2021 from:

```rust
.map_or(neuron_minted_stake_e8s, |a| {
    a.e8s.saturating_sub(fees_amount_e8s)
});
```

to:

```rust
.map_or(neuron_minted_stake_e8s, |a| a.e8s);
```

Then add a cap analogous to the SNS guard (`disburse_amount_e8s = disburse_amount_e8s.min(neuron_minted_stake_e8s)`) to prevent disbursing more than the minted stake. Update or remove the misleading "symmetry" comment at lines 2012–2015.

## Proof of Concept
A deterministic unit test suffices:

1. Create a neuron with `cached_neuron_stake_e8s = 1_000_000_000` and `neuron_fees_e8s = 10_000_000` (10M e8s in fees from a rejected proposal). `minted_stake = 990_000_000`.
2. Call `disburse_neuron` with `amount = Some(Tokens { e8s: 990_000_000 })` (the full minted stake) and a target account.
3. Observe that the user receives `990_000_000 - 10_000_000 - tx_fee = 979_900_000` (assuming `tx_fee = 10_000`).
4. Call `disburse_neuron` with `amount = None` on an equivalent neuron.
5. Observe that the user receives `990_000_000 - tx_fee = 989_990_000`.
6. The `Some(minted_stake)` path yields `10_000_000` fewer tokens than the `None` path for the same logical disbursement — confirming the double-deduction. This test can be written against the existing mock ledger infrastructure in `rs/nns/governance/src/governance/tests/`.
