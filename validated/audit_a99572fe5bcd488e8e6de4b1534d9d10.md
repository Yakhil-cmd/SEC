### Title
Single-Step Ownership Transfer Enables Permanent Fund Freeze - (File: engine/src/contract_methods/admin.rs)

### Summary
The `set_owner` function in the Aurora Engine immediately overwrites `state.owner_id` with the caller-supplied `new_owner` in a single atomic step, with no pending-owner confirmation. If the owner mistakenly supplies an incorrect or inaccessible NEAR account ID, all owner-gated functions — including `resume_contract` — become permanently unreachable. If the contract is paused at the time of the mistake (or is paused later by any path that still works), all user funds held in the Aurora EVM are permanently frozen.

### Finding Description
`set_owner` in `engine/src/contract_methods/admin.rs` performs the ownership change atomically:

```rust
// engine/src/contract_methods/admin.rs  lines 103-121
pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;

        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }

        state.owner_id = args.new_owner;   // ← immediate, irreversible
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
```

The only guard is `ERR_SAME_OWNER`, which rejects a no-op but does not prevent setting an arbitrary, potentially inaccessible account. `SetOwnerArgs` carries a plain `AccountId` string; NEAR account IDs are human-readable strings (e.g. `"aurora"`, `"my-dao.near"`), making a typo or a reference to a not-yet-created sub-account a realistic mistake.

`require_owner_only` is a strict equality check:

```rust
// engine/src/contract_methods/mod.rs  lines 79-87
pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}
```

Every critical administrative function gates on this check, including `resume_contract`:

```rust
// engine/src/contract_methods/admin.rs  lines 262-272
pub fn resume_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_paused(&state)?;
        state.is_paused = false;
        ...
    })
}
```

`upgrade` / `stage_upgrade`, `set_upgrade_delay_blocks`, `attach_full_access_key`, `set_key_manager`, `resume_precompiles`, `factory_update`, and all silo whitelist functions share the same gate. There is no recovery path once `owner_id` is set to an inaccessible account.

### Impact Explanation
**Critical — Permanent freezing of funds.**

The Aurora Engine holds all user ETH and ERC-20 balances in its NEAR contract storage. The `is_paused` flag in `EngineState` gates every `submit` / `call` path via `require_running`. If the contract is paused (e.g., during a routine upgrade window) and `set_owner` is called with a wrong account ID in the same window, `resume_contract` becomes permanently unreachable. Every user balance — ETH, bridged ERC-20 tokens, wNEAR — is frozen with no on-chain recovery mechanism.

Even if the contract is not currently paused, the loss of `upgrade` access means a subsequently discovered critical bug cannot be patched, and `attach_full_access_key` (which also requires owner) cannot be used to recover control.

### Likelihood Explanation
NEAR account IDs are arbitrary UTF-8 strings. A one-character typo (e.g., `"auroa"` instead of `"aurora"`, or a DAO sub-account that has not yet been created) produces a syntactically valid `AccountId` that passes Borsh deserialization and the `ERR_SAME_OWNER` check, but refers to an account nobody controls. Ownership transfers are infrequent, high-stakes operations performed manually, exactly the conditions under which typos occur. The River Protocol audit found the same class of bug exploitable under identical conditions.

### Recommendation
Implement a two-step ownership transfer:

1. **Propose**: The current owner calls `propose_owner(new_owner)`, which stores `new_owner` as a `pending_owner` field in `EngineState` without changing `owner_id`.
2. **Accept**: Only the account stored in `pending_owner` can call `accept_ownership()`, which then overwrites `owner_id`.

This guarantees the new owner account exists and is accessible before the transfer is finalized, eliminating the risk of an irreversible mistake.

### Proof of Concept

**Entry path** (callable by the current owner — a privileged but non-compromised actor making a human error):

1. Aurora Engine is paused for a scheduled upgrade (`pause_contract` succeeds, `state.is_paused = true`).
2. Owner calls `set_owner` with `new_owner = "auroa"` (typo for `"aurora"`).
   - `require_owner_only` passes (caller is still the current owner).
   - `ERR_SAME_OWNER` does not trigger (`"auroa" != "aurora"`).
   - `state.owner_id` is atomically set to `"auroa"` and persisted.
3. Owner attempts `resume_contract` from the real `"aurora"` account.
   - `require_owner_only` fails: `"aurora" != "auroa"` → `ERR_NOT_ALLOWED`.
4. No other account can call `resume_contract` (no fallback, no governance override in the contract).
5. All user ETH and ERC-20 balances in the Aurora EVM are permanently frozen.
