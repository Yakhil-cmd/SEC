Audit Report

## Title
Store-before-validate ordering in `sync_blocks_interval` enables permanent forged-block injection into Rosetta SQLite — (`rs/rosetta-api/icrc1/src/ledger_blocks_synchronization/blocks_synchronizer.rs`)

## Summary

`sync_blocks_interval` commits each fetched batch to SQLite via `store_blocks` (line 444) before the trailing parent-hash check that anchors the batch to the already-stored chain (line 463). Because `store_blocks` uses `INSERT OR IGNORE`, a malicious ICRC-1 ledger canister operator can permanently inject a hash-chain-disconnected batch of forged blocks into Rosetta's database. Subsequent correct syncs silently drop the real blocks at those indices, and `update_account_balances` — called unconditionally even after a sync failure — processes the forged blocks and writes fabricated balances to the `account_balances` table.

## Finding Description

**Step 1 — Attacker-controlled certified tip.**
`get_certified_chain_tip()` in `packages/icrc-ledger-agent/src/lib.rs` (lines 342–343) correctly verifies the IC BLS certificate over the canister's `certified_data`. However, the IC subnet signs whatever value the canister sets via `ic0.certified_data_set()`. A canister the attacker controls can set `certified_data` to a hash-tree containing any `tip_hash` / `last_block_index`, so `get_certified_chain_tip()` returns an attacker-chosen hash as the certified tip.

**Step 2 — `sync_from_the_tip` anchors the sync range to the forged tip.**
`sync_from_the_tip` (lines 285–299) builds `SyncRange` with `leading_block_hash = ByteBuf::from(tip_block_hash)` (attacker-controlled) and `trailing_parent_hash = Some(block.get_block_hash())` (the hash of the last legitimately stored block, read from the local DB).

**Step 3 — `sync_blocks_interval` stores before the trailing-hash check.**
Inside the loop:
1. Fetch batch (line 403).
2. `is_valid_blockchain` (line 419) — validates internal chain consistency and that the highest block's hash equals `leading_block_hash`. This passes because the attacker crafted the forged chain to end at their certified hash.
3. **`store_blocks` (line 444)** — forged blocks are committed to SQLite here.
4. `leading_block_hash` updated to the parent hash of the lowest fetched block (line 440).
5. On the final iteration only: `leading_block_hash == sync_range.trailing_parent_hash` (line 463) — fires `bail!` because the forged chain does not connect to the real stored chain. But the forged blocks are **already committed**.

**Step 4 — `INSERT OR IGNORE` makes the injection permanent.**
`store_blocks` uses `INSERT OR IGNORE INTO blocks` (line 687 of `storage_operations/mod.rs`). On every subsequent sync cycle, `get_block_with_highest_block_idx()` returns the forged tip, so `trailing_parent_hash` is now the forged block's hash. Real ledger blocks at those indices are silently dropped.

**Step 5 — `update_account_balances` is called unconditionally.**
In `start_synching_blocks` (line 200), `update_account_balances` is called regardless of whether `sync_failed` is true. The gap check inside `update_account_balances` (line 429 of `storage_client.rs`) only detects index gaps, not hash-chain discontinuities. Since the forged blocks occupy consecutive indices (N..N+M-1) immediately following the real blocks (0..N-1), no index gap exists, the gap check passes, and fabricated balances are written to `account_balances`.

## Impact Explanation

After injection, `update_account_balances` processes the forged blocks and writes phantom token balances to the `account_balances` table. Any exchange or custodian querying Rosetta for balances receives fabricated amounts derived from the forged chain. The attacker can mint arbitrary token amounts in the forged blocks with no on-chain cap enforced by Rosetta's local DB. This constitutes a significant Rosetta security impact with concrete financial harm to downstream users — matching the **High ($2,000–$10,000)** impact class: "Significant Rosetta security impact with concrete user or protocol harm."

## Likelihood Explanation

The attacker must control the ICRC-1 ledger canister that a Rosetta instance is configured to sync from. Deploying a canister on the IC is permissionless. The realistic threat model is a malicious token issuer whose token an exchange has listed and whose Rosetta instance is pointed at the issuer's canister. No special privileges beyond canister controller access are required. The attack is repeatable and deterministic.

## Recommendation

Move the trailing parent-hash check **before** calling `store_blocks`. After computing `leading_block_hash = fetched_blocks[0].get_parent_hash()` (line 440), when `*next_index_interval.start() == *sync_range.index_range.start()`, verify `leading_block_hash == sync_range.trailing_parent_hash` before the `store_blocks` call at line 444. Only store the batch if the check passes.

Alternatively, wrap the entire `sync_blocks_interval` loop body in a SQLite transaction that is rolled back on any validation failure, including the trailing-hash check.

## Proof of Concept

```
1. Sync N real blocks into Rosetta (indices 0..N-1).
2. Replace the Icrc1Agent with a mock that:
   a. Returns a certified tip at index N+M-1 with hash H_fake
      (canister sets certified_data = hash_tree{tip_hash: H_fake}).
   b. Returns M forged blocks [N..N+M-1] forming a valid internal chain
      ending at H_fake, but with block[N].parent_hash ≠ hash(real block N-1).
3. Call sync_from_the_tip (or start_synching_blocks with RecurrencyMode::OneShot).
4. is_valid_blockchain passes (internal consistency + leading hash == H_fake).
5. store_blocks commits forged blocks N..N+M-1 to SQLite.
6. Trailing-hash check fires bail! — but DB already contains forged blocks.
7. update_account_balances is called unconditionally; gap check passes (no index gap);
   fabricated balances are written to account_balances.
8. Assert fabricated balances appear in account_balances table.
9. Re-run sync with the real ledger; assert real blocks at N..N+M-1
   are silently dropped by INSERT OR IGNORE.
```
