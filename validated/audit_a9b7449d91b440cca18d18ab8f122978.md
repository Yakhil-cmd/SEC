### Title
Permanent Loss of ERC-20/ETH Funds When `ft_transfer` Fails in `ExitToNear` Precompile Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

When the `error_refund` compile-time feature is disabled (the default), the `ExitToNear` precompile burns ERC-20 tokens (or deducts ETH) from the user's EVM balance and schedules a NEAR `ft_transfer` promise. If that promise fails, the `exit_to_near_precompile_callback` silently does nothing — no refund is issued. The burned tokens are permanently lost. This is the direct Aurora analog of the Cairo ERC20Handler bug: instead of a panic/assert leaving a transaction in Pending state, Aurora's callback silently swallows the failure and leaves the user's funds in a permanently unrecoverable state.

---

### Finding Description

**Step 1 — Tokens are burned before the cross-chain promise is confirmed.**

In `engine-precompiles/src/native.rs`, the `ExitToNear` precompile runs inside the EVM execution. For ERC-20 exits, the ERC-20 contract calls the precompile from its `burn` function — the tokens are already destroyed in EVM state before any NEAR promise is dispatched. For ETH exits, the ETH is transferred out of the caller's EVM balance. The precompile then emits a log that schedules a NEAR `ft_transfer` (or `ft_transfer_call`) promise.

**Step 2 — The callback args carry `refund: None` when `error_refund` is disabled.** [1](#0-0) 

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
```

When `error_refund` is not enabled, `refund` is unconditionally `None`. The `error_refund` feature is **not** in the `default` feature set: [2](#0-1) 

**Step 3 — The callback silently does nothing on promise failure.**

In `engine/src/contract_methods/connector.rs`, `exit_to_near_precompile_callback` checks the promise result: [3](#0-2) 

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // ... handle wNEAR unwrap transfer
    None
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← reached when refund is None AND ft_transfer failed
};
```

When `ft_transfer` fails and `args.refund` is `None` (the default build), execution falls into the final `else { None }` branch. No refund is issued, no error is returned, and the function succeeds — leaving the user's EVM tokens permanently destroyed with nothing received on NEAR.

**Step 4 — The test explicitly confirms the fund loss.** [4](#0-3) 

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

The test acknowledges that without `error_refund`, the exited tokens are gone.

---

### Impact Explanation

**Critical — Permanent freezing/loss of user funds.**

Any EVM user who calls the `ExitToNear` precompile (directly or via an ERC-20 contract's bridge function) and whose `ft_transfer` promise fails will permanently lose their tokens. The EVM state records the burn/deduction as final (the EVM transaction succeeded), but the NEAR-side transfer never completes and no refund is minted back. The funds are irrecoverable.

This matches the analog: the Cairo bug leaves a cross-chain transaction in Pending (funds frozen); Aurora's bug leaves the EVM state with tokens destroyed and no NEAR-side credit (funds permanently lost).

---

### Likelihood Explanation

The `ft_transfer` promise can fail for multiple realistic, user-triggerable reasons:
- The recipient NEAR account is not registered with the NEP-141 contract (storage deposit not paid).
- The NEP-141 contract is paused or has been upgraded to reject transfers.
- The Aurora engine account has insufficient NEP-141 balance (e.g., drained by a prior operation).

All of these are reachable by an ordinary EVM user with no special privileges. The `error_refund` feature is off by default, meaning any standard build of Aurora Engine is affected.

---

### Recommendation

1. Enable `error_refund` by default in the production build, or move the refund logic out from behind a feature flag so it is always active.
2. Alternatively, restructure the exit flow so that ERC-20 tokens are not burned (or ETH not deducted) until the NEAR-side `ft_transfer` promise is confirmed successful — i.e., perform the burn inside the callback rather than before the promise is dispatched.
3. At minimum, document clearly in the precompile interface that callers will permanently lose funds if the recipient is not registered, so users can verify registration before calling.

---

### Proof of Concept

1. Deploy an ERC-20 token on Aurora that is bridged from a NEP-141.
2. As an EVM user, call the ERC-20's `transfer`/`burn` path that invokes the `ExitToNear` precompile, specifying a NEAR recipient account that has **not** paid the NEP-141 storage deposit.
3. The EVM transaction succeeds: the ERC-20 tokens are burned from the caller's balance.
4. The NEAR `ft_transfer` promise fails (unregistered account).
5. `exit_to_near_precompile_callback` is called with `refund: None`; it returns `Ok(None)` without minting back any tokens.
6. The caller's ERC-20 balance is permanently zero; the NEAR recipient received nothing. Funds are lost.

The existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` (lines 623–666) demonstrates this exact outcome when compiled without `--features error_refund`. [5](#0-4) [6](#0-5) [7](#0-6) [2](#0-1)
