Audit Report

## Title
VIP Slot Bypass Fails When Pool Is Exhausted, Allowing Unprivileged DoS of NNS Governance Proposals via NNS Root — (`rs/nervous_system/clients/src/management_canister_client.rs`, `rs/nns/handlers/root/impl/canister/canister.rs`)

## Summary
`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` applies the `available_slot_count == 0` rejection guard unconditionally before the VIP-aware subtraction, so VIP callers (NNS Governance, SNS-W) are rejected identically to non-VIP callers when the 167-slot pool is exhausted. An unprivileged attacker can drain all slots via the public `canister_status` endpoint, causing NNS Governance's calls to `update_canister_settings`, `take_canister_snapshot`, and `load_canister_snapshot` on NNS Root to fail with `SysTransient` for the duration of the attack.

## Finding Description
In `rs/nervous_system/clients/src/management_canister_client.rs` at L264–287, `try_borrow_slot` computes `used_slot_count = 0` for VIP callers (L265) but then enters a `with_borrow_mut` closure that unconditionally checks `if *available_slot_count == 0` (L269) and returns `Err((SysTransient, ...))` before reaching the subtraction at L278. The VIP path (`used_slot_count = 0`, `SlotLoan::drop` is a no-op) is never reached when the pool is at zero.

The slot pool is initialized to 167 in `rs/nns/handlers/root/impl/canister/canister.rs` at L50. The `canister_status` endpoint (L88–98) has no access control and calls `new_management_canister_client()`, consuming one slot per in-flight call. An attacker deploys a canister and issues 167 concurrent calls to this endpoint, each awaiting a management canister round-trip, holding all slots. During this window, `new_management_canister_client()` in `update_canister_settings` (L221–230), `take_canister_snapshot` (L245–254), and `load_canister_snapshot` (L259–268) — all called by NNS Governance — will hit the `== 0` guard and return `SysTransient`. `change_nns_canister` (L136–165) is unaffected because it calls `change_canister()` directly without going through `new_management_canister_client()`.

## Impact Explanation
This is a High-severity application/platform-level DoS. NNS Governance proposals executing `update_canister_settings`, `take_canister_snapshot`, or `load_canister_snapshot` via NNS Root are blocked for the duration of the attack. The attacker requires no privileged access; the attack is sustained by continuously recycling 167 concurrent inter-canister calls to a public endpoint. This matches the allowed impact: "Application/platform-level DoS… or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
Any principal can deploy a canister on the IC and issue inter-canister calls. Sustaining 167 concurrent in-flight calls to `canister_status` requires ongoing cycles expenditure but no special privileges or insider knowledge. The attack is repeatable and can be maintained indefinitely at attacker discretion.

## Recommendation
Condition the early-exit guard on the caller not being a VIP:

```rust
if *available_slot_count == 0 && !self.is_caller_vip {
    let code = RejectCode::SysTransient as i32;
    let message = "Unavailable. Maybe, try again later?".to_string();
    return Err((code, message));
}
```

This preserves the intended invariant that VIP callers are never rejected regardless of non-VIP load, while non-VIP callers are still rate-limited.

## Proof of Concept
State-machine or PocketIC integration test:
1. Initialize NNS Root canister with `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT = 167`.
2. Deploy an attacker canister that issues 167 concurrent calls to NNS Root `canister_status`, each targeting a slow-responding canister to keep calls in-flight.
3. Assert `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT == 0`.
4. Issue `update_canister_settings` from the NNS Governance principal.
5. Assert the call returns `Err((SysTransient, "Unavailable. Maybe, try again later?"))`.
6. Apply the fix (`&& !self.is_caller_vip`) and re-run step 4; assert it now succeeds.
