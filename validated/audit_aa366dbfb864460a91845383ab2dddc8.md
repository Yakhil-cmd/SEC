### Title
Per-Transaction Precompile Warm-Up Loop Incurs Uncharged Native Resource Cost, Growing with EIP Additions — (`basic_bootloader/src/bootloader/block_flow/ethereum/loop_op.rs`)

### Summary

In `generic_loop_op`, for every transaction processed in a block, the bootloader iterates over the entire `PRECOMPILE_ADDRESSES_LOWS` array and calls `touch_account` for each precompile using a local `FORMAL_INFINITE` resource variable. The native resource cost consumed by this loop is never charged to any transaction's native budget. As more EIPs add precompiles (e.g., `eip-2537` adds 7 BLS12 entries, growing the list to 15), the per-transaction uncharged overhead grows, creating a resource accounting mismatch where the actual RISC-V proving cost of a block exceeds the sum of all charged transaction native resources.

---

### Finding Description

In `generic_loop_op`, before executing each transaction, the bootloader unconditionally iterates over every entry in `PRECOMPILE_ADDRESSES_LOWS` and calls `touch_account` with a freshly created, locally scoped `FORMAL_INFINITE` resource:

```rust
// all EVM precompiles must be formally warm
{
    for &address_low in PRECOMPILE_ADDRESSES_LOWS {
        let address = B160::from_limbs([address_low as u64, 0, 0]);
        let mut inf_resources = S::Resources::FORMAL_INFINITE;
        system
            .io
            .touch_account(
                ExecutionEnvironmentType::NoEE,
                &mut inf_resources,
                &address,
            )
            .expect("must warm up precompile");
    }
}
``` [1](#0-0) 

`FORMAL_INFINITE` is defined as `DecreasingNative(u64::MAX)` — a local variable that is discarded at the end of each iteration of the outer transaction loop. Any native resource consumed inside `touch_account` (oracle queries on first access, cache lookups on subsequent accesses) is charged against this throwaway variable and is never reflected in the transaction's actual native budget. [2](#0-1) 

`PRECOMPILE_ADDRESSES_LOWS` is a static slice whose length grows with enabled EIP features. With `eip-2537` enabled it contains 15 entries (8 standard EVM precompiles + 7 BLS12-381 precompiles): [3](#0-2) 

For a block containing **T** transactions:

- **Transaction 1**: 15 cold `touch_account` calls → each triggers `materialize_element` → oracle queries for account data (expensive RISC-V cycles).
- **Transactions 2…T**: 15 warm `touch_account` calls each → cache hash-map lookups (cheaper, but still real RISC-V cycles, and the loop runs unconditionally).

Total uncharged overhead: `15 × oracle_query_cost + 15 × (T−1) × cache_lookup_cost`.

None of this cost appears in any transaction's native resource charge. The coinbase warm-up immediately above the precompile loop has the same pattern: [4](#0-3) 

The ZK transaction loop (`zk/tx_loop.rs`) does **not** contain this precompile warm-up loop, so the issue is specific to the Ethereum execution path. [5](#0-4) 

The double-resource accounting model explicitly states that native resource tracks "how many RISC-V cycles it takes to prove a given computation": [6](#0-5) 

Because the warm-up cycles are real but uncharged, the block's actual proving cost is systematically higher than the sum of all transaction native charges.

---

### Impact Explanation

The actual RISC-V cycle cost of a block exceeds the sum of charged transaction native resources by an amount proportional to `N_precompiles × T_transactions`. This creates two concrete harms:

1. **Operator undercompensation**: The operator is paid based on charged native resources. The uncharged warm-up overhead is a subsidy extracted from the operator on every block.
2. **Prover cycle budget overrun**: If the proving system enforces a per-block cycle budget derived from the sum of transaction native charges, a block filled with many cheap transactions (minimal native cost per tx) can cause the actual cycle count to exceed the budget, potentially stalling or failing the prover — a resource-exhaustion DoS on the proving pipeline.

The severity grows monotonically as new EIPs add precompiles (the list already grew from 8 to 15 with `eip-2537`), directly analogous to the Hubble finding where adding markets increased the gas cost of `settleFunding`.

---

### Likelihood Explanation

Every block that contains more than one Ethereum-path transaction triggers this issue. No special attacker capability is required beyond submitting standard EVM transactions. An attacker wishing to maximize the uncharged overhead simply submits a block's worth of minimal-gas transactions (e.g., simple ETH transfers), each of which pays a small native charge while contributing 15 uncharged `touch_account` calls to the block's proving cost.

---

### Recommendation

1. **Warm precompiles once per block, not per transaction.** Move the precompile warm-up to the pre-transaction block setup phase and charge the cost to a block-level overhead budget rather than repeating it for every transaction.
2. **Include warm-up cost in the intrinsic native charge.** If per-transaction warm-up is required for correctness, add the cost of `N_precompiles × touch_account_native_cost` to the transaction's intrinsic native charge (analogous to how `PER_ADDRESS_ACCESS_LIST_NATIVE_COMPUTATIONAL_OVERHEAD` is charged in `parse_and_warm_up_access_list`). [7](#0-6) 

3. **Skip the loop for transactions after the first.** Since precompiles are already in the warm cache after the first transaction, the loop body is a no-op for correctness on subsequent transactions. Guarding the loop with a `tx_counter == 0` check (already available in scope) would eliminate the O(N × T) overhead for all but the first transaction.

---

### Proof of Concept

Submit a block containing the maximum number of simple ETH-transfer transactions (21 000 gas each, minimal native cost). Each transaction triggers 15 uncharged `touch_account` calls in `generic_loop_op`. With `eip-2537` enabled and, say, 500 transactions in a block:

- Charged native: `500 × native_cost(ETH_transfer)`
- Actual RISC-V cycles: `500 × native_cost(ETH_transfer) + 15 × oracle_query_cost + 15 × 499 × cache_lookup_cost`

The gap between charged and actual grows linearly with transaction count and with the number of precompiles, directly mirroring the Hubble `settleFunding` gas exhaustion pattern. [8](#0-7) [3](#0-2)

## Native resource

The native resource models the offchain cost of processing a transaction. Currently, this is dominated by proving and publishing data. A good intuition for it is "how many RISC-V cycles it takes to prove a given computation".

If a transaction execution runs out of native resources, the entire transaction is reverted. If the same happens during transaction validation, the transaction is considered invalid.
