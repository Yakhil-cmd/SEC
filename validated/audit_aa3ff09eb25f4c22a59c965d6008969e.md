### Title
Fee-on-Transfer NEP-141 Token Breaks ERC-20 Mirror Accounting, Causing Insolvency - (File: `engine/src/engine.rs`)

### Summary

`receive_erc20_tokens` blindly trusts `args.amount` from the `ft_on_transfer` callback to mint ERC-20 tokens on Aurora, without verifying the actual NEP-141 balance change. If a registered NEP-141 token charges a fee on transfer, Aurora mints more ERC-20 tokens than the NEP-141 tokens it actually holds, creating an insolvent bridge.

### Finding Description

When a NEP-141 token is bridged into Aurora via `ft_transfer_call`, the NEP-141 contract calls `ft_on_transfer` on Aurora. The handler dispatches to `receive_erc20_tokens`: [1](#0-0) 

Inside `receive_erc20_tokens`, the amount used to mint ERC-20 tokens is taken directly from the callback argument: [2](#0-1) 

The `amount` variable is `args.amount.as_u128()` — the gross amount the sender declared, not the net amount Aurora actually received. The ERC-20 mint call encodes this unchecked amount: [3](#0-2) 

The `EvmErc20.mint` function mints exactly that amount with no further validation: [4](#0-3) 

No balance-before/balance-after check is performed anywhere in this path. The same pattern exists in `receive_base_tokens` for the ETH connector path, though that path is gated to a single trusted connector account: [5](#0-4) 

### Impact Explanation

If a fee-on-transfer NEP-141 token is registered with Aurora (via `deploy_erc20_token`), the following accounting divergence occurs on every deposit:

- Sender calls `ft_transfer_call` with `amount = X`
- NEP-141 token deducts `X` from sender but credits Aurora only `X − fee`
- NEP-141 calls `ft_on_transfer` on Aurora with `amount = X`
- Aurora mints `X` ERC-20 tokens for the recipient
- Aurora's actual NEP-141 balance is only `X − fee`

After repeated deposits, the total ERC-20 supply exceeds Aurora's NEP-141 reserves. When users burn ERC-20 tokens to exit (via `withdrawToNear` → `exit_to_near` precompile → `ft_transfer` on the NEP-141), the last users to exit cannot be served. This is **insolvency**: Aurora cannot honor all outstanding ERC-20 redemptions. [6](#0-5) 

### Likelihood Explanation

`deploy_erc20_token` does not restrict which NEP-141 tokens can be registered — any NEP-141 account can be bridged. The NEP-141 standard does not prohibit fee-on-transfer behavior. A token that charges a transfer fee (even a small one, e.g., 0.1%) would silently trigger this accounting divergence on every bridge deposit. The divergence accumulates monotonically and is irreversible without an admin intervention that is not currently coded.

### Recommendation

Use the balance-before/balance-after pattern. Before calling `ft_transfer_call` on the NEP-141 token, record Aurora's NEP-141 balance. In the `ft_on_transfer` callback, use the actual balance increase (i.e., `balance_after − balance_before`) as the amount to mint, rather than `args.amount`. Alternatively, explicitly reject registration of fee-on-transfer NEP-141 tokens and document this assumption as a hard invariant.

### Proof of Concept

1. Deploy a NEP-141 token that charges a 10% fee on every transfer (credits receiver `amount * 0.9`).
2. Register it with Aurora via `deploy_erc20_token`.
3. Call `ft_transfer_call(aurora, 1000, recipient_evm_address)` on the NEP-141.
4. NEP-141 credits Aurora with 900 tokens, then calls `ft_on_transfer(sender, 1000, msg)`.
5. Aurora's `receive_erc20_tokens` mints 1000 ERC-20 tokens for the recipient.
6. Repeat 10 times: Aurora holds 9000 NEP-141 but has minted 10000 ERC-20.
7. The 10th user to exit burns their ERC-20 and calls `ft_transfer` for 1000 NEP-141 — Aurora's balance is 0, the transfer fails, and those ERC-20 tokens are permanently frozen. [7](#0-6) [8](#0-7)

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
