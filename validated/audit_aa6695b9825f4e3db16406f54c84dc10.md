### Title
Missing static-context value check for CALLCODE allows ETH transfer inside STATICCALL frame — (`evm_interpreter/src/instructions/host.rs`)

### Summary
`call_impl` guards `CallScheme::Call` against non-zero value in a static context but omits the identical guard for `CallScheme::CallCode`, allowing CALLCODE with `value > 0` to proceed inside a STATICCALL frame without reverting.

### Finding Description
In `call_impl` (host.rs:403-485), the static-context value check is:

```rust
CallScheme::Call => {
    let value = self.stack.pop_1()?;
    if self.is_static && *value != U256::ZERO {          // ← guarded
        return Err(EvmError::CallNotAllowedInsideStatic.into());
    }
    *value
}
CallScheme::CallCode => {
    let value = self.stack.pop_1()?;
    // ← NO static check here
    *value
}
```

The `is_static` flag is correctly propagated to `EVMCallcodeStatic` at line 455, but that modifier is never used to reject the call. `calculate_resources_passed_in_external_call` (ee_trait_impl.rs:302-308) even charges the positive-value gas cost for CALLCODE, confirming the non-zero value is processed rather than rejected.

### Impact Explanation
CALLCODE transfers ETH from the caller's balance to the callee's address — a balance state change. EIP-214 mandates that **any** state-modifying operation inside a STATICCALL frame must revert. Allowing CALLCODE with `value > 0` inside a static frame violates this invariant, enabling ETH transfers from within a read-only context. Contracts that rely on STATICCALL's no-state-change guarantee (e.g., view functions, reentrancy guards using STATICCALL) can be broken.

### Likelihood Explanation
CALLCODE is a valid, reachable EVM opcode. Any deployed contract can issue it. An attacker deploys a contract whose fallback/function uses `CALLCODE` with a non-zero value, then triggers it via a STATICCALL from another contract or off-chain simulation. No privileged access is required.

### Recommendation
Add the same guard for `CallScheme::CallCode` as exists for `CallScheme::Call`:

```rust
CallScheme::CallCode => {
    let value = self.stack.pop_1()?;
    if self.is_static && *value != U256::ZERO {
        return Err(EvmError::CallNotAllowedInsideStatic.into());
    }
    *value
}
```

### Proof of Concept
1. Deploy contract A: `fallback() { assembly { callcode(gas(), addr, 1, 0, 0, 0, 0) } }`
2. Deploy contract B: `test() { A.staticcall(...) }`
3. Call `B.test()` — under correct EVM semantics this must revert with `StateChangeDuringStaticCall`; under ZKsync OS it proceeds without reverting.
4. Differential test against revm: revm reverts, ZKsync OS does not → confirmed deviation.
