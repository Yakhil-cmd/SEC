### Title
Single-Step `set_owner` Lacks Confirmation Safeguard, Enabling Irrecoverable Loss of Admin Control — (`engine/src/contract_methods/admin.rs`)

---

### Summary

The `set_owner` function in the Aurora Engine immediately and irrevocably transfers `owner_id` to any supplied NEAR account ID in a single step, with no pending-owner confirmation. If the current owner supplies a mistyped, non-existent, or inoperable account ID, ownership is permanently lost with no recovery path. Because every critical emergency-response capability (`pause_contract`, `upgrade`, `stage_upgrade`, `resume_precompiles`, `attach_full_access_key`) is gated exclusively behind `require_owner_only`, permanent loss of ownership permanently removes the ability to pause or upgrade the engine in response to a critical bug, which can lead to permanent freezing of user funds.

---

### Finding Description

`set_owner` in `engine/src/contract_methods/admin.rs` performs a one-shot ownership transfer:

```rust
// engine/src/contract_methods/admin.rs  lines 103-121
pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;

        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }

        state.owner_id = args.new_owner;   // ← immediate, irrevocable
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
``` [1](#0-0) 

The only guard is `ERR_SAME_OWNER` — there is no pending-owner slot, no acceptance step, and no cancellation window. NEAR account IDs are arbitrary UTF-8 strings; a single-character typo (e.g., `"auroa"` instead of `"aurora"`) produces a syntactically valid `AccountId` that passes all checks and is immediately written to state.

Every privileged function is gated by `require_owner_only`:

```rust
// engine/src/contract_methods/mod.rs  lines 79-87
pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}
``` [2](#0-1) 

Functions exclusively gated by this check include:

| Function | Consequence if owner is lost |
|---|---|
| `pause_contract` | Cannot halt the engine during an active exploit |
| `resume_contract` | Cannot unpause a stuck engine |
| `upgrade` / `stage_upgrade` | Cannot deploy security patches |
| `resume_precompiles` | Cannot re-enable paused precompiles |
| `attach_full_access_key` | Cannot add a recovery key |
| `set_owner` itself | Cannot correct the mistake | [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Permanent freezing of funds (Critical).**

Aurora Engine is a live EVM bridge holding user ETH and ERC-20 balances. If ownership is accidentally transferred to an inoperable account:

1. `pause_contract` becomes permanently uncallable — an ongoing exploit (e.g., a bridge accounting bug) cannot be stopped.
2. `upgrade` / `stage_upgrade` become permanently uncallable — no security patch can be deployed.
3. `resume_precompiles` becomes permanently uncallable — if a precompile was paused before the ownership loss, it stays paused forever.

The combination of (1) and (2) means any subsequently discovered critical vulnerability in the engine or connector cannot be mitigated, resulting in permanent freezing or theft of all user funds held in the bridge.

---

### Likelihood Explanation

**Low-Medium.** The owner is a trusted operator, but:

- NEAR account IDs are plain strings; a single-character typo produces a valid `AccountId` that passes `BorshDeserialize` and the `ERR_SAME_OWNER` check without error.
- Multisig or DAO governance transitions (common for production contracts) involve constructing the target account ID manually, increasing typo risk.
- There is no on-chain existence check for the target account before the transfer is committed.
- The `SetOwnerArgs` struct contains only `new_owner: AccountId` with no confirmation field. [7](#0-6) 

---

### Recommendation

Replace the single-step transfer with a two-step propose-and-accept pattern:

1. **`propose_new_owner(new_owner: AccountId)`** — callable only by the current owner; stores `pending_owner` in state.
2. **`accept_ownership()`** — callable only by `pending_owner`; atomically moves `pending_owner` into `owner_id`.

This ensures the new owner account is demonstrably live and controllable before the transfer is finalised, matching the safeguard the external `Owner` contract provides in the Solidity ecosystem.

---

### Proof of Concept

1. Current owner calls `set_owner` with `new_owner = "auroar.near"` (one-character typo of `"aurora.near"`).
2. `require_owner_only` passes (caller is current owner). `ERR_SAME_OWNER` does not trigger (different string). `state.owner_id` is immediately overwritten.
3. `"auroar.near"` does not exist on NEAR mainnet; no account can call `require_owner_only`-gated functions.
4. `pause_contract`, `upgrade`, `stage_upgrade`, `resume_precompiles`, and `set_owner` are all permanently inaccessible.
5. Any subsequently discovered critical bug in the engine or bridge connector cannot be patched or paused, leading to permanent freezing or theft of all bridged user funds. [8](#0-7)

pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())
}

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
}
