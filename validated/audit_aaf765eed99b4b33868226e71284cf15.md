### Title
EIP-2537 G1MSM and G2MSM Precompiles Incorrectly Reject Empty Input, Causing EVM Semantic Mismatch - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/msm.rs`)

---

### Summary

Both `Bls12381G1MSMPrecompile` and `Bls12381G2MSMPrecompile` unconditionally return an error when called with empty input. EIP-2537 specifies that empty input to G1MSM/G2MSM must succeed and return the identity point (128 or 256 zero bytes respectively). ZKsync OS instead burns all caller gas and returns a failed call result — a direct EVM semantic mismatch with Ethereum mainnet behavior.

---

### Finding Description

In `basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/msm.rs`, both MSM precompile implementations open with an unconditional empty-input guard:

```rust
if input.len() == 0 {
    return Err(interface_error!(
        Bls12PrecompileInterfaceError::InvalidInputSize
    ));
}
``` [1](#0-0) [2](#0-1) 

This error is classified as `RootCause::Usage(_)` by the subsystem error macro. The precompile dispatch layer in `system_hooks/src/call_hooks/precompiles.rs` handles this class of error by exhausting all remaining ergs and returning a failed call result:

```rust
RootCause::Runtime(RuntimeError::OutOfErgs(_)) | RootCause::Usage(_) => {
    resources.exhaust_ergs();
    Ok((make_error_return_state(resources), rest))
}
``` [3](#0-2) 

The internal `compute_cost` function already handles the zero-pair case correctly by returning `0`:

```rust
let num_pairs = input_len / pair_len;
if num_pairs == 0 {
    return 0;
}
``` [4](#0-3) 

And the internal `msm()` function, when called with `size == 0`, iterates zero times over the base loop and returns `G::ZERO` (the identity point):

```rust
let size = bases.len();
if size < 4 {
    let mut acc = G::ZERO;
    for bit in (0..NUM_BITS).rev() {
        for i in 0..size { ... }  // never executes when size == 0
        if bit > 0 { acc.double_in_place(); }
    }
    return acc;
}
``` [5](#0-4) 

So the downstream logic is already correct for empty input — only the early-exit guard is wrong.

The same pattern also affects `Bls12381PairingCheckPrecompile` in `pairing.rs`, which rejects empty input with the same error, while EIP-2537 requires it to return `true` (32-byte `0x01`) for empty input. [6](#0-5) 

---

### Impact Explanation

On Ethereum mainnet (go-ethereum reference implementation), calling G1MSM with empty input returns 128 zero bytes (the G1 identity point) and costs 0 gas. On ZKsync OS, the same call burns all forwarded gas and returns a failed sub-call result. Any contract that:

- calls G1MSM/G2MSM with empty input as a no-op or identity-point shortcut, or
- conditionally passes empty input based on runtime state,

will behave differently on ZKsync OS versus Ethereum. This constitutes a **state-transition divergence**: the same transaction produces a different execution outcome (success vs. failure, different gas consumption, different return data) depending on which chain processes it. This breaks EVM equivalence guarantees and can cause incorrect proof generation or block re-execution failures.

---

### Likelihood Explanation

Any unprivileged transaction sender can trigger this by calling a contract that forwards empty calldata to the G1MSM precompile address (`0x000...000c` per EIP-2537). The call path is fully permissionless. While empty-input MSM calls are an edge case in typical DeFi usage, they are a valid and specified behavior that BLS-heavy protocols (e.g., aggregated signature verifiers, ZK proof systems) may rely on for conditional or batched operations. The likelihood is **medium**: the code path is reachable with a single transaction, but requires a contract that specifically exercises this boundary.

---

### Recommendation

Remove the `if input.len() == 0 { return Err(...) }` guard from both `Bls12381G1MSMPrecompile::execute` and `Bls12381G2MSMPrecompile::execute`. With the guard removed, `compute_cost` returns 0 (no gas charged), the parsing loop does not execute, `msm()` returns the identity point, and `write_g1`/`write_g2` serializes it to the output — matching the EIP-2537 specified behavior. Apply the same fix to `Bls12381PairingCheckPrecompile::execute`, which should return `true` (32-byte `0x01`) for empty input.

---

### Proof of Concept

1. Deploy a contract on ZKsync OS that executes:
   ```solidity
   (bool ok, bytes memory ret) = address(0x000...000c).call{gas: 100000}("");
   require(ok, "G1MSM empty input failed");
   ```
2. On Ethereum mainnet, `ok == true` and `ret` is 128 zero bytes (identity point).
3. On ZKsync OS, `ok == false` and all 100,000 gas is consumed, causing `require` to revert.
