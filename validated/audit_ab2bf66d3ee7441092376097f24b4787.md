### Title
Unauthorized Caller Can Commit Another Buyer's ICP Into SNS Swap by Supplying Arbitrary `buyer` Principal - (`rs/sns/swap/canister/canister.rs`)

---

### Summary

The `refresh_buyer_tokens` update endpoint of the SNS Swap canister accepts an arbitrary `buyer` principal string. Any unprivileged ingress sender can call this function on behalf of any other principal, bypassing the confirmation-text consent mechanism and irrevocably committing that principal's ICP into the swap without their explicit agreement.

---

### Finding Description

In `rs/sns/swap/canister/canister.rs`, the `refresh_buyer_tokens` handler resolves the buyer principal from the caller-supplied `arg.buyer` field rather than from `ic_cdk::caller()`:

```rust
#[update]
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()   // ← arbitrary principal
    };
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
``` [1](#0-0) 

The downstream `refresh_buyer_token_e8s` function then:
1. Validates the caller-supplied `confirmation_text` against the SNS-configured consent string.
2. Reads the ICP balance from the **victim's** subaccount (`principal_to_subaccount(&buyer)`) of the swap canister.
3. Permanently records that balance as the victim's committed participation in `self.buyers`. [2](#0-1) 

The `confirmation_text` is a public, on-chain value set at SNS initialization time and readable by anyone via `get_init`. The `validate_confirmation_text` function only checks that the caller-supplied string matches this public value — it does **not** verify that the caller is the buyer:

```rust
pub fn validate_confirmation_text(&self, confirmation_text: Option<String>) -> Result<(), String> {
    match (self.init_or_panic().confirmation_text.as_ref(), confirmation_text) {
        (Some(expected_text), Some(text)) => {
            if &text != expected_text { Err(...) } else { Ok(()) }
        }
        ...
    }
}
``` [3](#0-2) 

The `RefreshBuyerTokensRequest` proto explicitly documents this open design:

> "If not specified, the caller is used." [4](#0-3) 

---

### Impact Explanation

**Consent bypass and forced fund commitment.** The confirmation text is the sole mechanism by which a buyer signals informed consent to the swap's terms. An attacker who knows a victim's principal (public information) and the swap's confirmation text (also public) can:

1. Call `refresh_buyer_tokens { buyer: "<victim>", confirmation_text: "<public_text>" }`.
2. The canister reads the ICP balance already sitting in the victim's subaccount (transferred there by the victim, perhaps while still deliberating).
3. That balance is recorded as committed participation in `self.buyers` under the victim's principal. [5](#0-4) 

Once committed, the victim's ICP is swept into the SNS treasury upon swap finalization. The victim receives SNS tokens instead of their ICP — an irreversible outcome they never explicitly authorized. The only recourse is `error_refund_icp`, which is only available after the swap closes (committed or aborted), and only if the swap aborts.

---

### Likelihood Explanation

- **No special privileges required.** Any ingress sender can call `refresh_buyer_tokens` with an arbitrary `buyer` field.
- **All inputs are public.** The victim's principal is observable on-chain; the confirmation text is stored in the swap's `init` state and readable via query.
- **Realistic scenario.** A buyer who transfers ICP to the swap subaccount but has not yet called `refresh_buyer_tokens` (e.g., still reading the confirmation text, or waiting for a better moment) is immediately vulnerable. A griefing attacker or a competing participant wishing to force the swap to commit early can exploit this.
- **Swap auto-finalization amplifies impact.** If `should_auto_finalize` is `true` (the default), forcing enough ICP participation via this vector can trigger immediate swap commitment, locking all participants' funds. [6](#0-5) 

---

### Recommendation

Validate that the resolved buyer principal equals `ic_cdk::caller()` when a non-empty `buyer` field is supplied, or remove the ability to specify an arbitrary buyer entirely and always use `caller()`:

```rust
#[update]
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let requested = PrincipalId::from_str(&arg.buyer).unwrap();
        assert_eq!(caller, requested, "Caller must match the specified buyer");
        requested
    };
    ...
}
```

If relayer/third-party notification is intentionally desired (analogous to Connext's acknowledged design), the confirmation text must be supplied by the buyer out-of-band and the relayer must not be able to forge it — but since the text is public, this is not achievable without a separate per-buyer signed authorization.

---

### Proof of Concept

**Setup:** SNS swap is `Open`. Confirmation text is `"I agree to the terms"` (readable from `get_init`). Victim `Alice` has transferred 10 ICP to her swap subaccount but has not yet called `refresh_buyer_tokens`.

**Attack:**
```
Attacker → SNS Swap canister:
  refresh_buyer_tokens({
    buyer: "<Alice's principal>",
    confirmation_text: "I agree to the terms"
  })
```

**Result:**
- `refresh_buyer_token_e8s` reads Alice's subaccount balance (10 ICP).
- Alice's entry is created in `self.buyers` with `amount_icp_e8s = 10_0000_0000`.
- If the swap subsequently commits (possibly triggered by this very call reaching the ICP target), Alice's 10 ICP is swept to the SNS treasury and she receives SNS tokens — without ever having called `refresh_buyer_tokens` herself or explicitly accepted the confirmation text. [1](#0-0) [7](#0-6)

    pub async fn refresh_buyer_token_e8s(
        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

        // Look for the token balance of the specified principal's subaccount on 'this' canister.
        let e8s = {
            let account = Account {
                owner: this_canister.get().0,
                subaccount: Some(principal_to_subaccount(&buyer)),
            };
            icp_ledger
                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
        };
