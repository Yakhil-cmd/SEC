### Title
Pre-EIP-155 Legacy Transactions Accepted Without Chain-ID Binding, Enabling Cross-Chain Replay - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

The ZKsync OS bootloader accepts pre-EIP-155 legacy transactions (those with `v = 27` or `v = 28`). For these transactions, the signing hash is computed over only the six-field payload `[nonce, gasPrice, gasLimit, to, value, data]` with no `chainId` included. This means a valid pre-EIP-155 transaction signed for Ethereum mainnet (or any other EVM chain) can be submitted to ZKsync OS and will pass signature verification, constituting a cross-chain replay attack.

---

### Finding Description

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, the code branches on whether the signature is EIP-155 protected:

```rust
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    // Unprotected legacy — hash does NOT include chainId
    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);
    hasher.finalize_reset().into()
} else {
    // EIP-155 protected — hash includes chainId
    ...
    apply_u64_encoding_to_hash(chain_id, &mut hasher);
    ...
};
```

`is_eip155()` returns `false` when `v == 27 || v == 28`. In that branch, the signing hash covers only the six transaction fields — no `chainId`, no nonce-in-hash, no domain separator. The transaction is then stored as the `Self::Legacy` variant and proceeds through full validation and execution. [1](#0-0)

The dispatch that stores the unprotected transaction as a valid `Legacy` variant: [3](#0-2) 

All typed transactions (EIP-2930, EIP-1559, EIP-4844, EIP-7702) embed `chainId` in their payload and are validated against `expected_chain_id`. Only the pre-EIP-155 legacy path lacks this protection. [4](#0-3) 

---

### Impact Explanation

An attacker who observes a pre-EIP-155 legacy transaction broadcast on Ethereum mainnet (or any other EVM chain) can submit the identical raw bytes to ZKsync OS. Because the signing hash contains no `chainId`, the `ecrecover` call in the bootloader's validation path will recover the same `from` address and the transaction will be accepted as valid. If the victim's ZKsync nonce matches the replayed transaction's nonce, the transaction executes — potentially transferring ETH/tokens out of the victim's ZKsync account, calling arbitrary contracts on their behalf, or deploying contracts from their address. [5](#0-4) 

---

### Likelihood Explanation

Pre-EIP-155 transactions are uncommon with modern wallets, but they are still generated by older tooling, hardware wallets in legacy mode, and some programmatic signers. The nonce must match the victim's current ZKsync nonce, which is a meaningful constraint. However, for users who are new to ZKsync (nonce = 0) and have previously broadcast a pre-EIP-155 transaction with nonce = 0 on Ethereum, the replay requires no additional conditions. The attacker needs only to monitor public mempools or historical chain data for qualifying transactions.

---

### Recommendation

Reject pre-EIP-155 legacy transactions entirely by returning an `InvalidChainId` error when `legacy_signature.is_eip155()` is `false`. ZKsync OS is a new chain with no legacy pre-EIP-155 history to preserve compatibility with, so there is no backward-compatibility reason to accept them. The fix is a one-line guard:

```rust
if legacy_signature.is_eip155() == false {
    return Err(InvalidTransaction::InvalidChainId.into());
}
```

This mirrors the behavior already enforced for all typed transaction formats. [6](#0-5) 

---

### Proof of Concept

1. On Ethereum mainnet, broadcast a pre-EIP-155 legacy transaction from address `A` with nonce `N`, sending ETH to address `B`. The raw RLP bytes are public.
2. Ensure address `A` has ETH on ZKsync OS and its ZKsync nonce is also `N`.
3. Submit the identical raw RLP bytes to ZKsync OS as a new transaction.
4. The bootloader parses the transaction, detects `v ∈ {27, 28}`, computes the chain-ID-free signing hash, calls `ecrecover`, recovers address `A`, and accepts the transaction.
5. The transaction executes on ZKsync OS, transferring ETH from `A`'s ZKsync balance to `B`.

The attacker spends only the ZKsync gas fee; no private key material is needed. [7](#0-6) [8](#0-7)

impl LegacyPayloadParser {
    pub(crate) fn try_parse_and_hash_for_signature_verification<'a>(
        src: &'a [u8],
        expected_chain_id: u64,
    ) -> Result<(LegacyTXInner<'a>, LegacySignatureData<'a>, Bytes32), TxError> {
        // Legacy path: input must be a single list with 9 elements total.
        let mut outer = Rlp::new(src);

        // Strip the list encoding
        let mut inner = outer.list()?;

        // Outer list must be fully consumed
        if !outer.is_empty() {
            return Err(InvalidTransaction::InvalidStructure.into());
        }

        // Capture the concatenation bytes of the first 6 fields for hashing.
        let mark = inner.mark();
        let legacy_inner: LegacyTXInner<'a> = LegacyTXInner::decode_list_body(&mut inner)?;
        let inner_slice = inner.consumed_since(mark);

        let legacy_signature = LegacySignatureData::decode_list_body(&mut inner)?;
        if !inner.is_empty() {
            return Err(InvalidTransaction::InvalidStructure.into());
        }

        let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
            // Unprotected legacy
            let mut hasher = crypto::sha3::Keccak256::new();
            apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
            hasher.update(inner_slice);
            hasher.finalize_reset().into()
        } else {
            // EIP-155 protected legacy: v must match 35 + 2*chainId (+ {0,1})
            let min_v = U256::from(35) + U256::from(expected_chain_id) * U256::from(2);
            if !(legacy_signature.v == min_v || legacy_signature.v == min_v + U256::ONE) {
                return Err(InvalidTransaction::InvalidEncoding.into());
            }

            // Compute signing hash over the 6-field payload plus chainId and two empty strings.
            let chain_id = expected_chain_id;
            let chain_id_encoding_len = u64_encoding_len(chain_id);

            let mut hasher = crypto::sha3::Keccak256::new();
            apply_list_concatenation_encoding_to_hash(
                (inner_slice.len() + chain_id_encoding_len + 2) as u32, // 0x80, 0x80 for r/s
                &mut hasher,
            );
            hasher.update(inner_slice);
            apply_u64_encoding_to_hash(chain_id, &mut hasher);
            hasher.update(&[0x80, 0x80]);
            hasher.finalize_reset().into()
        };

        Ok((legacy_inner, legacy_signature, sig_hash))
    }
