### Title
Scheduled XCC Promises Never Expire and Are Executable by Anyone — (`File: etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router's `schedule` function stores `PromiseArgs` indefinitely with no expiration timestamp. The `execute_scheduled` function is intentionally open to any caller. Together, these two properties allow a malicious actor to execute a stale scheduled promise long after it was created, at a moment of their choosing, potentially causing the user's cross-contract call to execute against a compromised or unintended target.

---

### Finding Description

In `etc/xcc-router/src/lib.rs`, the `schedule` method inserts a `PromiseArgs` value into `scheduled_promises` keyed by a monotonically increasing nonce:

```rust
// etc/xcc-router/src/lib.rs lines 136–144
pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
    self.assert_preconditions();
    let nonce = self.nonce.get().unwrap_or_default();
    self.scheduled_promises.insert(nonce, promise);
    self.nonce.set(&(nonce + 1));
    near_sdk::log!("Promise scheduled at nonce {}", nonce);
}
```

No creation timestamp, block height, or deadline is stored alongside the promise. There is also no `cancel_scheduled` or equivalent function — the only way to remove an entry from `scheduled_promises` is to execute it.

The `execute_scheduled` function is explicitly open to any caller:

```rust
// etc/xcc-router/src/lib.rs lines 146–156
/// It is intentional that this function can be called by anyone (not just the parent).
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

No expiration check is performed before dispatching the stored promise. The comment justifies the open access by claiming "there is no security risk," but this ignores the time-dimension risk: a promise that was safe to execute at scheduling time may be harmful to execute months later.

Scheduled promises originate from the `CrossContractCallArgs::Delayed` path in the XCC precompile (`engine-precompiles/src/xcc.rs` lines 159–172), where a user's EVM transaction causes the Aurora Engine to call `schedule` on the user's router sub-account. The stored `PromiseArgs` encodes the full cross-contract call — target account, method, args, attached balance, and gas — exactly as the user specified at scheduling time.

---

### Impact Explanation

A scheduled promise encodes a specific cross-contract call (e.g., a NEP-141 `ft_transfer`, a DeFi deposit, or a NEAR transfer) that was valid at the time of scheduling. Because the promise never expires and cannot be cancelled:

1. If the target contract is later exploited or replaced by a malicious actor, executing the stale promise sends funds to the compromised contract.
2. If the user's intent changes (e.g., they no longer want the transfer), they have no mechanism to cancel it.
3. A malicious actor who monitors the NEAR chain can call `execute_scheduled` at the most damaging moment — for example, after the target account has been taken over, or after the user has re-funded the router with wNEAR.

The result is that user funds (NEAR or NEP-141 tokens encoded in the promise's `attached_balance` or method arguments) are transferred to an unintended or attacker-controlled destination. This constitutes **theft of user funds** or **permanent loss** depending on the promise content.

---

### Likelihood Explanation

The `execute_scheduled` entry point is a public NEAR contract method callable by any account with no authentication. The nonce is a sequential integer logged on-chain (`near_sdk::log!("Promise scheduled at nonce {}", nonce)`), so an attacker can trivially enumerate all pending scheduled promises. The only prerequisite is that the router contract has sufficient NEAR balance to cover `attached_balance` in the stored promise — a condition that is met whenever the user has pre-funded the router (which is the normal operating condition for XCC). This makes the attack realistic for any user who uses the `Delayed` XCC path and does not execute their own scheduled promise promptly.

---

### Recommendation

1. **Add an expiration timestamp** to each scheduled promise entry. Store `(PromiseArgs, creation_block_height)` and reject execution in `execute_scheduled` if the promise is older than a configurable maximum (e.g., 7 days in block heights).
2. **Add a `cancel_scheduled` function** restricted to the parent (Aurora Engine), callable on behalf of the originating EVM address, so users can cancel stale promises.
3. Alternatively, restrict `execute_scheduled` to the parent account only, removing the open-caller design entirely, since the security justification in the comment does not account for time-based staleness.

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Alice uses the XCC precompile with `CrossContractCallArgs::Delayed` to schedule a promise that calls `ft_transfer` on a NEP-141 token contract, sending 1000 tokens to `defi.near`. The Aurora Engine calls `schedule` on Alice's router sub-account (`<alice_hex>.aurora`). The promise is stored at nonce `N`.
2. Six months pass. `defi.near` is exploited; its account is now controlled by an attacker.
3. The attacker calls `execute_scheduled(N)` on Alice's router contract. No authentication is required.
4. The router dispatches `ft_transfer` to `defi.near` (now attacker-controlled) with Alice's 1000 tokens.
5. Alice's tokens are permanently lost.
