Audit Report

## Title
Stale ICP/XDR Spot Rate in `tokens_to_cycles` Enables Cycles Arbitrage During Rapid Price Drops - (`rs/nns/cmc/src/main.rs`, `rs/nns/cmc/src/exchange_rate_canister.rs`)

## Summary
The Cycles Minting Canister refreshes its ICP/XDR conversion rate at most once every 5 minutes via the Exchange Rate Canister. The `tokens_to_cycles` function reads the cached spot rate directly from state with no staleness check. During a rapid ICP price decline, any unprivileged user can buy ICP cheaply on the open market and convert it to cycles at the stale, pre-crash rate, extracting value from the protocol for up to 5 minutes per window.

## Finding Description
`REFRESH_RATE_INTERVAL_SECONDS` is set to `5 * ONE_MINUTE_SECONDS` (300 seconds) in `exchange_rate_canister.rs`. The heartbeat calls `update_exchange_rate`, which on success schedules the next attempt at the next 5-minute boundary; on failure it retries after only 1 minute, but the stale rate remains in use throughout.

The `tokens_to_cycles` function in `main.rs` reads `state.icp_xdr_conversion_rate` and checks only whether it is `Some` or `None` — there is no check on `timestamp_seconds` to reject a rate that is too old:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);
        // No staleness guard here
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles { ... }.to_cycles(amount)),
            None => Err(...)
        }
    })
}
```

All three public minting entry points (`notify_top_up`, `notify_create_canister`, `notify_mint_cycles`) call `tokens_to_cycles` and are reachable by any unprivileged principal.

The state already maintains `average_icp_xdr_conversion_rate` (a 30-day TWAP computed in `update_recent_icp_xdr_rates`), but `tokens_to_cycles` uses only the spot `icp_xdr_conversion_rate`, never the average. The `get_icp_xdr_conversion_rate` query is a certified endpoint, so an attacker can monitor the CMC's published rate in real time at zero cost and compare it to external exchange prices.

The `do_set_icp_xdr_conversion_rate` function enforces only that the new rate has a strictly greater timestamp than the current one and that `xdr_permyriad_per_icp > 0`; it does not bound how far the rate can lag behind real time.

## Impact Explanation
This matches the **High** bounty impact: "Significant XRC, NNS, or infrastructure security impact with concrete user or protocol harm." An attacker who observes a rapid ICP price drop (e.g., 15% within a 5-minute window) can buy ICP at the new lower market price and immediately call `notify_top_up` or `notify_mint_cycles`, receiving cycles computed at the old higher rate. The ICP is burned, but the cycles minted exceed the current market value of the ICP burned, constituting value extraction from the protocol. The `DEFAULT_CYCLES_LIMIT` of 150e15 cycles/hour per account bounds but does not eliminate the damage: at 1 trillion cycles/XDR and 5 XDR/ICP, each account can convert ~30,000 ICP-equivalent per hour. A 15% discrepancy over a 5-minute window yields ~375 ICP-equivalent of risk-free profit per account per window. Multiple independent accounts can each hit the limit simultaneously, scaling the total extraction.

## Likelihood Explanation
ICP price volatility of 10–20% within a 5-minute window is historically plausible during market stress events. The attack requires no privileged access: any principal can transfer ICP to the CMC subaccount and call `notify_top_up` or `notify_mint_cycles`. The attacker needs only to monitor `get_icp_xdr_conversion_rate()` (a free certified query) against external exchange prices. The steps are fully automatable with a bot. The window is short but repeatable across multiple accounts.

## Recommendation
1. **Add a staleness guard in `tokens_to_cycles`**: Reject minting requests if `icp_xdr_conversion_rate.timestamp_seconds` is older than a configurable threshold (e.g., 10 minutes), returning a retriable `NotifyError` so users can retry once the rate refreshes.
2. **Use `average_icp_xdr_conversion_rate` for minting**: The CMC already maintains a 30-day TWAP in `average_icp_xdr_conversion_rate`. Using this instead of the spot rate in `tokens_to_cycles` would make the conversion rate far more resistant to short-term price swings.
3. **Reduce `REFRESH_RATE_INTERVAL_SECONDS`**: Shortening the refresh interval narrows the exploitable window.

## Proof of Concept
1. Deploy an attacker canister on mainnet.
2. Poll `get_icp_xdr_conversion_rate()` on the CMC continuously (free certified query).
3. Monitor ICP spot price on external exchanges.
4. When ICP drops ≥10% while the CMC rate has not yet updated (within the 5-minute window):
   a. Buy N ICP at the new lower market price.
   b. Transfer N ICP to `AccountIdentifier::new(CYCLES_MINTING_CANISTER_ID, Some(Subaccount::from(&attacker_canister)))` with memo `MEMO_TOP_UP_CANISTER`.
   c. Call `notify_top_up { block_index, canister_id: attacker_canister }`.
5. CMC calls `tokens_to_cycles(N_ICP)` using the stale higher `xdr_permyriad_per_icp`, minting ~10–15% more cycles than the ICP is worth at current market price.
6. Repeat across multiple attacker canisters up to the per-account per-hour rate limit.
7. A deterministic integration test can reproduce this by: (a) setting `icp_xdr_conversion_rate` to a high value, (b) advancing the mock clock by less than 300 seconds without triggering a heartbeat update, (c) calling `notify_top_up` with a known ICP amount, and (d) asserting that cycles minted exceed the amount justified by the current market rate.
