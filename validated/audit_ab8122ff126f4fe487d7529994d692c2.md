### Title
Stale ICP/XDR Rate in Cycles Minting Canister Enables Cycles Arbitrage During Rapid Price Drops - (`rs/nns/cmc/src/main.rs`, `rs/nns/cmc/src/exchange_rate_canister.rs`)

---

### Summary

The Cycles Minting Canister (CMC) converts ICP to cycles using a single oracle source (the Exchange Rate Canister, XRC) whose rate is cached and refreshed only every **5 minutes**. The `tokens_to_cycles` function reads the spot `icp_xdr_conversion_rate` directly from state without any staleness guard. During a rapid ICP price drop, the CMC continues to mint cycles at the old, higher ICP/XDR rate for up to 5 minutes. Any unprivileged user can exploit this window by buying ICP cheaply on the open market and converting it to cycles at the stale favorable rate, extracting value from the protocol.

---

### Finding Description

The CMC heartbeat calls `update_exchange_rate` at most once every `REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS` (300 seconds). [1](#0-0) 

Inside the heartbeat, the XRC is queried and the result is stored as `state.icp_xdr_conversion_rate`: [2](#0-1) 

All three public cycle-minting entry points (`notify_top_up`, `notify_create_canister`, `notify_mint_cycles`) call `tokens_to_cycles`, which reads this cached spot rate directly: [3](#0-2) 

The conversion formula is: [4](#0-3) 

There is no check on the age of `icp_xdr_conversion_rate` before using it. If the heartbeat fails (e.g., XRC returns too few sources, or the call errors), the next attempt is deferred by only 1 minute, but the stale rate remains in use for all minting operations in the interim. [5](#0-4) 

The XRC itself is a single canister source. While it aggregates a minimum of 4 ICP and 4 CXDR sources internally, the CMC trusts its output as the sole oracle: [6](#0-5) 

---

### Impact Explanation

During a rapid ICP price decline (e.g., ICP drops 15% in under 5 minutes), the CMC continues to mint cycles at the pre-crash rate. An attacker who buys ICP at the new lower market price and immediately calls `notify_top_up` or `notify_mint_cycles` receives cycles valued at the old higher rate, pocketing the difference. The hourly base minting limit (`DEFAULT_CYCLES_LIMIT = 150e15 cycles/hour`) caps but does not eliminate the damage: at 1 trillion cycles/XDR and 5 XDR/ICP, the limit allows ~30,000 ICP-equivalent of cycles per hour. A 15% price discrepancy over a 5-minute window yields ~375 ICP-equivalent of risk-free profit per attacker per window. Multiple accounts can each hit the limit independently. [7](#0-6) 

---

### Likelihood Explanation

ICP price volatility of 10–20% within a 5-minute window is historically plausible during market stress events. The attack requires no privileged access: any principal can transfer ICP to the CMC subaccount and call `notify_top_up` or `notify_mint_cycles`. The attacker needs only to monitor the ICP spot price on external exchanges and compare it to the CMC's published `icp_xdr_conversion_rate` (a certified query). The window is short (up to 5 minutes) but the steps are fully automatable. [8](#0-7) 

---

### Recommendation

1. **Add a staleness guard in `tokens_to_cycles`**: Reject minting requests if `icp_xdr_conversion_rate.timestamp_seconds` is older than a configurable threshold (e.g., 10 minutes), returning a retriable error so users can retry once the rate refreshes.

2. **Use the 30-day average rate for minting**: The CMC already maintains `average_icp_xdr_conversion_rate` (a 30-day TWAP). Using this instead of the spot rate for cycle conversion would make the rate far more resistant to short-term price swings, directly mirroring the mitigation recommended in the original Unlock Protocol report.

3. **Reduce the refresh interval or add a secondary oracle**: Shortening `REFRESH_RATE_INTERVAL_SECONDS` or cross-checking the XRC rate against a second independent source would narrow the exploitable window.

---

### Proof of Concept

1. Attacker monitors `get_icp_xdr_conversion_rate()` on the CMC (certified query, no cost).
2. ICP price drops 15% on external exchanges; CMC rate has not yet updated (within the 5-minute window).
3. Attacker buys N ICP at the new lower market price.
4. Attacker transfers N ICP to `AccountIdentifier::new(CYCLES_MINTING_CANISTER_ID, Some(Subaccount::from(&attacker_canister)))` with memo `MEMO_TOP_UP_CANISTER`.
5. Attacker calls `notify_top_up { block_index, canister_id: attacker_canister }`.
6. CMC calls `tokens_to_cycles(N_ICP)` using the stale higher `xdr_permyriad_per_icp`, minting ~15% more cycles than the ICP is worth at current market price.
7. Attacker sells or uses the excess cycles. Repeat across multiple accounts up to the per-hour rate limit. [9](#0-8) [10](#0-9)

fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles {
                xdr_permyriad_per_icp,
                cycles_per_xdr: state.cycles_per_xdr,
            }
            .to_cycles(amount)),
            None => {
                let error_message =
                    "No conversion rate found in CMC, notification aborted".to_string();
                print(&error_message);
                Err(NotifyError::Other {
                    error_code: NotifyErrorCode::Internal as u64,
                    error_message,
                })
            }
        }
    })
}

pub fn validate_exchange_rate(
    exchange_rate: &ExchangeRate,
) -> Result<(), ValidateExchangeRateError> {
    if exchange_rate.metadata.base_asset_num_received_rates < MINIMUM_ICP_SOURCES {
        return Err(ValidateExchangeRateError::NotEnoughIcpSources {
            received: exchange_rate.metadata.base_asset_num_received_rates,
            queried: exchange_rate.metadata.base_asset_num_queried_sources,
        });
    }

    if exchange_rate.metadata.quote_asset_num_received_rates < MINIMUM_CXDR_SOURCES {
        return Err(ValidateExchangeRateError::NotEnoughCxdrSources {
            received: exchange_rate.metadata.quote_asset_num_received_rates,
            queried: exchange_rate.metadata.quote_asset_num_queried_sources,
        });
    }

    Ok(())
}
