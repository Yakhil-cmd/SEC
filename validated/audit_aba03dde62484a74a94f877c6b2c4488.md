Audit Report

## Title
Unbounded Tokio Task Spawning via Oversized `StrippedBlockProposal` in P2P Artifact Assembly — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/assembler.rs`)

## Summary

When `HASHES_IN_BLOCKS_ENABLED = true` (a compile-time constant set to `true` in production), a single Byzantine subnet peer can push a crafted `StrippedBlockProposal` proto containing an arbitrarily large `ingress_messages` repeated field. The receiver's `assemble_message` spawns one unbounded Tokio task per entry with no count guard, exhausting memory and Tokio scheduler capacity before the bouncer fires. The final ID-mismatch check in `try_assemble` does not prevent the resource exhaustion because it executes only after all tasks have already been spawned and run.

## Finding Description

**Feature is active in production:**

`HASHES_IN_BLOCKS_ENABLED = true` is a compile-time constant. [1](#0-0) 

This causes `FetchStrippedConsensusArtifact` to be used as the consensus assembler. [2](#0-1) 

**No count limit in deserialization:**

`TryFrom<pb::StrippedBlockProposal>` iterates over `value.ingress_messages` and `value.stripped_idkg_dealings` with zero count validation. The proto fields are unbounded `repeated` fields. [3](#0-2) 

`MAX_INGRESS_MESSAGES_PER_BLOCK = 1000` exists but is enforced only at consensus validation time (after assembly), not at P2P receipt time. [4](#0-3) 

**Unbounded task spawning:**

After the inner bouncer check passes on the `StrippedConsensusMessageId` (not the content), `assemble_message` calls `missing_stripped_messages()` which returns all N entries, then spawns one `get_or_fetch` task per entry with no cap: [5](#0-4) 

The comment "We can have at most 1000 elements in the vector" at line 440 is aspirational, not a runtime check: [6](#0-5) 

**Bouncer fires after tasks are already spawned:**

The `JoinSet` is created at line 203, all N tasks are spawned at lines 214–226, and the bouncer watcher is only obtained at line 233 — after all tasks are already live. The `select!` loop at line 237 can only abort tasks that are already running and consuming memory: [7](#0-6) 

The bouncer refresh period is 3 seconds, meaning up to 3 seconds of task accumulation before abort: [8](#0-7) 

**Each spawned task is long-lived:**

`download_stripped_message` is configured with `with_max_elapsed_time(None)` and loops forever with exponential backoff (5 s to 120 s), never returning until a valid response arrives: [9](#0-8) [10](#0-9) 

**No wire-size limit on pushed artifacts:**

The axum router for the consensus manager explicitly disables the default body size limit: [11](#0-10) 

**The final ID check does not help:**

`try_assemble` at line 640 verifies the assembled block ID matches the claimed ID, but this check only runs after all N tasks have completed — by which point the memory and scheduler damage has already occurred. [12](#0-11) 

## Impact Explanation

This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

A Byzantine peer crafts a `StrippedBlockProposal` with N = 10^5 `StrippedIngressMessage` entries. The receiver spawns 10^5 Tokio tasks synchronously. At ~8 KB Tokio task overhead, this is ~800 MB memory spike per attack cycle, repeated every ~3 seconds. This causes sustained replica memory pressure, Tokio scheduler saturation, and consensus stall on the receiving node. If applied to f+1 nodes simultaneously, it can halt the subnet.

## Likelihood Explanation

- The feature is enabled in production (`HASHES_IN_BLOCKS_ENABLED = true`).
- The attacker only needs a valid P2P connection to the target node (any subnet peer qualifies).
- No cryptographic material, governance access, or threshold corruption is required.
- The crafted proto is trivially constructable — `ingress_messages` is an unbounded `repeated` field.
- The attack is repeatable with no rate limiting at the P2P layer for pushed artifacts.
- The `StrippedConsensusMessageId` used for the bouncer check is just a height + hash, trivially forgeable within the consensus window.

## Recommendation

1. **In `TryFrom<pb::StrippedBlockProposal>`** (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs`): Reject deserialization if `value.ingress_messages.len() > MAX_INGRESS_MESSAGES_PER_BLOCK` or `value.stripped_idkg_dealings.len() > DKG_DEALINGS_PER_BLOCK` before iterating. [13](#0-12) 

2. **In `assemble_message`** (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/assembler.rs`): Add a guard before the `for` loop that returns `AssembleResult::Unwanted` if `stripped_message_ids.len()` exceeds the protocol maximum. [5](#0-4) 

## Proof of Concept

```rust
// Construct a malicious StrippedBlockProposal proto
let mut malicious_proto = pb::StrippedBlockProposal::default();
malicious_proto.pruned_block_proposal = Some(pb::BlockProposal::default());
malicious_proto.unstripped_consensus_message_id = Some(valid_block_proposal_id.into());

// Inject 100_000 fake ingress IDs (each ~50 bytes on wire)
for i in 0..100_000u64 {
    malicious_proto.ingress_messages.push(pb::StrippedIngressMessage {
        stripped: Some(pb::IngressMessageId {
            expiry: i,
            message_id: vec![i as u8; 32],
        }),
        ingress_bytes_hash: vec![0u8; 32],
    });
}
// Push to victim via P2P as MaybeStrippedConsensusMessage::StrippedBlockProposal.
// assemble_message will spawn 100_000 get_or_fetch tasks before bouncer fires.
```

A deterministic integration test can verify this by:
1. Constructing the above proto and delivering it to a `FetchStrippedConsensusArtifact` instance via `assemble_message` with a `Wants` bouncer.
2. Using `SignallingNoPeers` (already present in the test suite at line 1099) to synchronize on the tasks being spawned.
3. Asserting `join_set` contains 100,000 live tasks and observing memory growth before the 3-second bouncer fires.
