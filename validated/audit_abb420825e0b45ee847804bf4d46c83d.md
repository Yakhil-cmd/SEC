### Title
Race Condition in XCC Router's `scheduled_promises` Mapping Allows Out-of-Order Permissionless Execution, Causing User Fund Freeze — (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router stores multiple scheduled promises in a `LookupMap<u64, PromiseArgs>` and exposes a fully permissionless `execute_scheduled(nonce)` function. This is structurally identical to the M-06 pattern: a mapping allows multiple valid entries to coexist simultaneously, and a permissionless function can act on any entry in any order. When a user schedules two interdependent promises (e.g., approve-then-transfer), an adversary can execute them in reverse order, causing the dependent promise to fail. Because the user's wNEAR ERC-20 balance is burned at scheduling time, the NEAR returned to the router on failure is permanently inaccessible to the user through the EVM, resulting in a fund freeze.

---

### Finding Description

**Root cause — mapping instead of single-slot state:**

The `Router` struct stores all pending promises in a `LookupMap<u64, PromiseArgs>`: [1](#0-0) 

The `schedule` function (callable only by the Aurora Engine parent) inserts a new promise at the current nonce and increments the nonce counter, allowing an unbounded number of promises to be live simultaneously: [2](#0-1) 

**Permissionless execution with no ordering constraint:**

`execute_scheduled` is explicitly open to any caller and accepts any nonce: [3](#0-2) 

The design comment acknowledges the permissionless nature but only addresses the "can only act on scheduled promises" safety property — it does not address the ordering problem. There is no check that promise `N-1` has been executed before promise `N`.

**Race condition exploit path:**

1. A user sends two EVM transactions through the XCC precompile using `CrossContractCallArgs::Delayed`:
   - **Nonce 0**: Call NEAR contract A to approve a token spend (setup step).
   - **Nonce 1**: Call NEAR contract B to execute a swap that requires the approval from step 0.
2. Both promises are stored in `scheduled_promises` at nonces 0 and 1. The user's wNEAR ERC-20 balance is burned at scheduling time (the wNEAR is transferred to the router's NEAR account).
3. An adversary calls `execute_scheduled(1)` before `execute_scheduled(0)`.
4. Promise 1 executes against contract B without the approval being in place; the call fails. The promise is removed from the map (`scheduled_promises.remove(&nonce.0)`) and is gone permanently.
5. The NEAR attached to promise 1 is returned by the NEAR runtime to the router's account — not to the user's EVM address.
6. The user's ERC-20 wNEAR balance was already burned. There is no EVM-accessible path to recover the NEAR from the router.

**Why the fix from M-06 applies here:**

M-06 recommended replacing the mapping with a single-slot variable so only one entry can be valid at a time. The analogous fix here is to enforce sequential execution: `execute_scheduled(N)` should only succeed if all nonces `< N` have already been consumed, or the system should only allow one scheduled promise to be live at a time per router.

---

### Impact Explanation

**High — Temporary (potentially permanent) freezing of user funds.**

The user's wNEAR is converted to NEAR and held in the router at scheduling time. If the dependent promise fails due to out-of-order execution, the NEAR is returned to the router's account balance. The user has no EVM-side mechanism to reclaim it (their ERC-20 balance is already zero). Recovery requires privileged intervention by the Aurora Engine operator (e.g., deploying a new router version or using `send_refund`, which only transfers a fixed `REFUND_AMOUNT` of 2 NEAR and is not designed for arbitrary recovery). [4](#0-3) 

---

### Likelihood Explanation

**Low-Medium.** The scenario requires a user to schedule two interdependent promises as separate `Delayed` XCC calls rather than using a single chained `PromiseArgs::Callback` or `PromiseArgs::Recursive` call. Users who understand the XCC API would typically chain dependent calls. However, the system provides no guardrail or documentation warning against this pattern, and the permissionless `execute_scheduled` is reachable by any NEAR account with no preconditions. A griefing adversary monitoring the router's state can trivially trigger this by calling `execute_scheduled` with the higher nonce first.

---

### Recommendation

Enforce sequential nonce execution in `execute_scheduled`: require that the supplied nonce equals the lowest unexecuted nonce (i.e., track a `min_executable_nonce` cursor). Alternatively, document clearly that scheduled promises must be fully independent and add a runtime assertion that rejects scheduling when more than one promise is already pending, mirroring the M-06 fix of replacing the mapping with a single-slot variable.

---

### Proof of Concept

```
1. User EVM address 0xABCD sends TX1:
     XCC precompile call → CrossContractCallArgs::Delayed(PromiseArgs::Create {
         target: "defi.near", method: "approve", args: {...}, attached_balance: 0
     })
   → Engine calls router.schedule(promise_0)
   → scheduled_promises[0] = promise_0; nonce = 1
   → User's wNEAR ERC-20 balance burned; NEAR held in router

2. User EVM address 0xABCD sends TX2:
     XCC precompile call → CrossContractCallArgs::Delayed(PromiseArgs::Create {
         target: "defi.near", method: "swap", args: {...}, attached_balance: 1_000_000 yoctoNEAR
     })
   → Engine calls router.schedule(promise_1)
   → scheduled_promises[1] = promise_1; nonce = 2
   → User's wNEAR ERC-20 balance burned; NEAR held in router

3. Adversary (any NEAR account) calls:
     router_for_0xABCD.execute_scheduled(nonce: 1)
   → promise_1 removed from map, executed against "defi.near"
   → "defi.near" rejects swap (no approval exists) → promise fails
   → NEAR returned to router account; user has no EVM path to recover

4. User calls execute_scheduled(0) — succeeds, but promise_1 is already gone.
   User's wNEAR for promise_1 is permanently inaccessible.
``` [5](#0-4)

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

    /// It is intentional that this function can be called by anyone (not just the parent).
    /// There is no security risk to allowing this function to be open because it can only
    /// act on promises that were created via `schedule`.
    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }
