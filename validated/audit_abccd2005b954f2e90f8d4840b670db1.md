Audit Report

## Title
ICP Permanently Locked in CMC Subaccount When Refund Fails and Block Notification Is Purged - (File: rs/nns/cmc/src/main.rs)

## Summary

The Cycles Minting Canister implements a compensation pattern in `process_create_canister`, `process_top_up`, and `process_mint_cycles`: if the primary action fails, `refund_icp` is called. If `refund_icp` itself fails (e.g., ledger temporarily unavailable during an upgrade), the error propagates as `NotifyError::Other { error_code: RefundFailed }`, which `is_transient_error` classifies as retriable, causing the block to be removed from `blocks_notified`. If the user does not retry before `purge_old_notifications` advances `last_purged_notification` past the block index, the user permanently receives `TransactionTooOld` and their ICP remains locked in the CMC subaccount with no recovery path.

## Finding Description

**Step 1 — Refund failure produces a retriable error.**

In `process_top_up`, `process_create_canister`, and `process_mint_cycles`, when the primary action fails, `refund_icp` is called with the `?` operator:

```rust
let refund_block = refund_icp(sub, from, amount, TOP_UP_CANISTER_REFUND_FEE).await?;
``` [1](#0-0) 

If the ledger call fails, `refund_icp` maps the error to `NotifyError::Other { error_code: RefundFailed }` and propagates it via `?`: [2](#0-1) 

**Step 2 — `is_transient_error` classifies `NotifyError::Other` as retriable.**

`is_transient_error` delegates to `is_retriable()`: [3](#0-2) 

`is_retriable()` returns `true` for everything except `Refunded`: [4](#0-3) 

`NotifyError::Other { RefundFailed }` is not `Refunded`, so `is_retriable()` returns `true`.

**Step 3 — Block is removed from `blocks_notified`, enabling retry but also enabling purge.**

In all three notify flows, when `is_transient_error` is true, the block entry is deleted: [5](#0-4) [6](#0-5) 

**Step 4 — `purge_old_notifications` advances `last_purged_notification` past the user's block.**

`purge_old_notifications` removes the oldest entries from `blocks_notified` when it exceeds `MAX_NOTIFY_HISTORY`, and monotonically advances `last_purged_notification`: [7](#0-6) 

Because the user's block was already removed from `blocks_notified` in Step 3, it does not protect the user's slot. As other notifications arrive and are purged, `last_purged_notification` can advance past the user's block index.

**Step 5 — Retry returns `TransactionTooOld`.**

On retry, the check `block_index <= state.last_purged_notification` fires: [8](#0-7) 

`TransactionTooOld` is non-retriable per the Candid interface. The ICP remains in the CMC subaccount (`Subaccount::from(&canister_id)` for top-up, `Subaccount::from(&controller)` for create canister) with no user-accessible recovery endpoint.

## Impact Explanation

This is a permanent, irrecoverable loss of user ICP held by an in-scope NNS canister (the CMC). ICP enters the CMC subaccount via a legitimate ledger transfer and cannot exit once the notification window expires. There is no `sweep`, `recover`, or `error_refund_icp`-style endpoint on the CMC. This matches the High bounty impact: **significant NNS/ledger security impact with concrete user funds harm** — permanent loss of user ICP with no protocol-level recovery path.

## Likelihood Explanation

The ICP ledger is a high-availability NNS canister, but it undergoes periodic upgrades that cause brief unavailability windows. The attack window is: (1) user's primary action fails (e.g., topping up a non-existent canister), (2) the ledger is mid-upgrade when `refund_icp` is called, (3) the user does not retry before `MAX_NOTIFY_HISTORY` new notifications push `last_purged_notification` past the user's block index. Under high CMC notification volume, the purge window can close faster than expected. The condition is reachable by any unprivileged user without special privileges. Likelihood is **low** but non-zero; impact is **high** (permanent ICP loss per affected user).

## Recommendation

1. **Add a recovery endpoint**: Expose a function callable by the original sender that transfers any remaining balance from a CMC subaccount back to the owner, callable regardless of block notification status (analogous to `error_refund_icp` in the SNS swap canister).

2. **Decouple refund retry from notification expiry**: Store failed refund tasks in a persistent queue (analogous to `pending_reimbursements` in the ckBTC minter) and process them independently of the notification history window.

3. **Do not purge blocks in a failed-refund state**: If a block's refund failed and the block was removed from `blocks_notified`, track it separately so that `last_purged_notification` cannot advance past it until the refund succeeds.

## Proof of Concept

1. Alice sends 10 ICP to `AccountIdentifier(CMC, Subaccount::from(&canister_id))`. Block index = `B`.
2. Alice calls `notify_top_up(block_index: B, canister_id: X)` where `X` does not exist.
3. CMC calls `deposit_cycles(X, ...)` → fails (canister not found).
4. CMC calls `refund_icp(...)` → calls `send_pb` on the ICP ledger. Ledger is mid-upgrade → call fails with transport error.
5. `refund_icp` returns `Err(NotifyError::Other { error_code: RefundFailed })` via `?`.
6. `is_transient_error` returns `true`; block `B` is removed from `blocks_notified`. Alice can retry.
7. Alice does not retry promptly. Many other notifications arrive, advancing `last_purged_notification` past `B`.
8. Alice calls `notify_top_up(block_index: B, canister_id: X)` again.
9. CMC returns `NotifyError::TransactionTooOld` — non-retriable.
10. Alice's 10 ICP remains in `Subaccount::from(&canister_id)` permanently with no recovery path.

**Reproducible test plan**: Using PocketIC or `StateMachine`, pause the mock ledger between the `deposit_cycles` failure and the `refund_icp` call, advance `last_purged_notification` past `B` by inserting `MAX_NOTIFY_HISTORY + 1` dummy notifications, then assert that `notify_top_up` returns `TransactionTooOld` and the CMC subaccount balance remains non-zero.

    fn purge_old_notifications(&mut self, max_history: usize) {
        let mut last_purged = 0;
        let mut cnt = 0;
        // Remove elements from the beginning of self.blocks_notified until either
        // it is small enough, or MAX_NOTIFY_PURGE entries have been removed.
        while self.blocks_notified.len() > max_history && cnt < MAX_NOTIFY_PURGE {
            // pop_first is nightly only
            let block_height = *self.blocks_notified.iter().next().unwrap().0;
            self.blocks_notified.remove(&block_height);
            last_purged = block_height;
            cnt += 1;
        }
        // make sure this grows monotonically (a delayed callback might have added older status)
        self.last_purged_notification = last_purged.max(self.last_purged_notification);
    }
