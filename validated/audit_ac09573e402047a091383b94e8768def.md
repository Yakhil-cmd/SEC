Audit Report

## Title
Voting Power Decay Restriction Bypass via Neuron Split — (File: `rs/nns/governance/src/governance.rs`)

## Summary
When a neuron is split via `split_neuron`, the child neuron is constructed with `voting_power_refreshed_timestamp_seconds` set to `now()`, unconditionally, regardless of the parent's stale value. This allows any neuron holder whose deciding voting power has been reduced or zeroed by the decay mechanism to recover full deciding voting power over their split stake with a single unprivileged `manage_neuron` call, bypassing the NNS's active-participation incentive mechanism.

## Finding Description
In `rs/nns/governance/src/governance.rs` at L2207 and L2241–2257, `split_neuron` sets `created_timestamp_seconds = self.env.now()` and passes it as the final argument to `NeuronBuilder::new`. The builder initialises `voting_power_refreshed_timestamp_seconds` from `created_timestamp_seconds`, so the child always receives a fresh timestamp equal to the current time. The parent's `voting_power_refreshed_timestamp_seconds` is never propagated to the child and is never updated by the split operation.

The deciding voting power computation in `rs/nns/governance/src/neuron/types.rs` at L390–399 computes:
```
adjustment_factor = deciding_voting_power_adjustment_factor(now - voting_power_refreshed_timestamp_seconds)
deciding_voting_power = adjustment_factor × potential_voting_power
```
When `voting_power_refreshed_timestamp_seconds` is > 6 months stale, `adjustment_factor` decreases linearly to 0 over the following month. The child neuron, having `voting_power_refreshed_timestamp_seconds = now()`, always has `adjustment_factor = 1.0`.

The test at `rs/nns/governance/tests/governance.rs` L5602–5616 explicitly asserts this behaviour — that the child's timestamp equals `driver.now()` and is strictly greater than the parent's — confirming it is present in the current codebase and not a test artifact.

The `with_voting_power_refreshed_timestamp_seconds` setter on `NeuronBuilder` is gated `#[cfg(test)]` only (`rs/nns/governance/src/neuron/types.rs` L1923–1930), so there is no production-code path to override this default in `split_neuron`.

No existing guard in `split_neuron` checks or propagates the parent's `voting_power_refreshed_timestamp_seconds`. The only documented refresh actions are direct voting and setting followees (`rs/nns/governance/src/neuron/types.rs` L145–151); splitting is not one of them, yet it silently resets the timestamp for the child.

## Impact Explanation
Any NNS neuron holder whose deciding voting power has been reduced or eliminated due to governance inactivity can unilaterally restore full deciding voting power over any portion of their stake by splitting. In the extreme case (α = 0, all stake split), the attacker recovers full deciding voting power from zero. This concretely undermines the NNS governance mechanism designed to reduce the influence of inactive stakeholders, constituting a significant NNS governance security impact with concrete protocol harm. This matches the allowed High impact tier: "Significant… NNS… security impact with concrete user or protocol harm."

## Likelihood Explanation
Any unprivileged neuron holder with a stale `voting_power_refreshed_timestamp_seconds` can execute this bypass with a single `manage_neuron(Split { amount_e8s: X })` call. No special privileges, coordination, timing, or elevated cost beyond the standard ICP transaction fee are required. The condition (stale timestamp > 6 months) is reachable by any neuron holder who has not voted or set followees for 6+ months, which is a realistic and common state on the NNS. The attack is repeatable and deterministic.

## Recommendation
In `split_neuron`, propagate the parent's `voting_power_refreshed_timestamp_seconds` to the child neuron. The child should inherit the minimum of `now()` and the parent's timestamp, ensuring the child never receives a fresher timestamp than the parent:

```rust
// In NeuronBuilder construction for split_neuron:
.with_voting_power_refreshed_timestamp_seconds(
    parent_neuron.voting_power_refreshed_timestamp_seconds
        .min(created_timestamp_seconds)
)
```

The `with_voting_power_refreshed_timestamp_seconds` setter must first be promoted out of `#[cfg(test)]` to be usable in production code. The same fix should be evaluated for `disburse_to_neuron` and `spawn_neuron`, which similarly default to `now()` via `NeuronBuilder::new`.

## Proof of Concept
The existing test at `rs/nns/governance/tests/governance.rs` L5602–5616 already constitutes a proof of concept: it asserts `child_neuron.voting_power_refreshed_timestamp_seconds == Some(driver.now())` and that it is strictly greater than the parent's value. A targeted PoC test would:

1. Create neuron N with 1,000,000,000 e8s stake and `voting_power_refreshed_timestamp_seconds` set to `now()`.
2. Advance the mock clock by 7+ months (beyond the 6-month decay window plus 1-month linear decay period).
3. Assert `N.deciding_voting_power ≈ 0`.
4. Call `split_neuron(N, Split { amount_e8s: 500_000_000 })` to produce child C.
5. Assert `C.voting_power_refreshed_timestamp_seconds == now()` (fresh).
6. Assert `C.deciding_voting_power == C.potential_voting_power` (full power, adjustment_factor = 1.0).
7. Assert `N.voting_power_refreshed_timestamp_seconds` is unchanged (still stale).

This test can be written as a unit test in `rs/nns/governance/tests/governance.rs` using the existing `FakeDriver` infrastructure, mirroring the split test already present at that file around L5560–5616.
