Audit Report

## Title
Stale ICP/XDR Rate Used for Cycle Minting Without Staleness Check — (File: rs/nns/cmc/src/main.rs)

## Summary
`tokens_to_cycles` in `rs/nns/cmc/src/main.rs` reads `state.icp_xdr_conversion_rate` and uses it directly with no check on the rate's `timestamp_seconds` relative to current time. When the Exchange Rate Canister (XRC) is temporarily unavailable — a realistic condition during subnet upgrades — the CMC continues minting cycles at the last stored rate indefinitely. An unprivileged caller who monitors the publicly readable rate timestamp can time `notify_top_up` calls to exploit a price discrepancy, receiving more cycles per ICP than the current market rate warrants.

## Finding Description
`tokens_to_cycles` (line 1900) extracts only `xdr_permyriad_per_icp` from the stored rate, with no age validation:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);  // timestamp never checked
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles { ... }.to_cycles(amount)),
            None => Err(...)
        }
    })
}
```

This function is the sole conversion path called by `notify_top_up`, `notify_create_canister`, and `notify_mint_cycles`.

The rate is refreshed every five minutes via `canister_heartbeat` → `update_exchange_rate`. When the XRC call fails (e.g., during a subnet upgrade), `UpdateExchangeRateError::FailedToRetrieveRate` is returned and the next attempt is scheduled one minute later — the stored rate is never invalidated or cleared.

`do_set_icp_xdr_conversion_rate` (line 1022) only enforces monotonically increasing timestamps (new > current), not that the rate is recent relative to wall-clock time. `validate_exchange_rate` (line 111 in `rs/nervous_system/clients/src/exchange_rate_canister_client.rs`) only checks the number of data sources, with no timestamp age check.

One inaccuracy in the submitted claim: `UpdateExchangeRateState::Disabled` is not triggered automatically by divergence detection — it is set only via a governance proposal with `UpdateIcpXdrConversionRatePayloadReason::DivergedRate` (line 311 in `rs/nns/cmc/src/exchange_rate_canister.rs`). This weakens the "indefinite staleness" scenario, but the XRC unavailability path (subnet upgrades) remains fully realistic and unprivileged-exploitable.

## Impact Explanation
Over-minting of cycles is a concrete protocol economic harm: ICP is burned at an incorrect (stale, pre-drop) conversion rate, permanently inflating the cycle supply relative to the ICP burned. This maps to the High impact class: "Significant XRC, NNS, or infrastructure security impact with concrete user or protocol harm." The magnitude per event is bounded by the price movement during the outage window and the attacker's ICP balance, making Critical ($1M+) unlikely for a typical upgrade window but meaningful at scale or during extended outages.

## Likelihood Explanation
NNS subnet upgrades occur regularly and can render the XRC unavailable for 10–30 minutes. The CMC's certified rate timestamp is publicly readable via `get_icp_xdr_conversion_rate`, so any observer can detect when the rate has stopped advancing. No special privileges are required to call `notify_top_up`. The attacker only needs ICP and the ability to monitor the public endpoint. Likelihood is low-medium: requires a coincident price movement during an observable outage window, but the entry path is fully unprivileged and the staleness is publicly detectable.

## Recommendation
Add a maximum-age guard inside `tokens_to_cycles` that rejects conversions when the stored rate's `timestamp_seconds` is older than a configurable threshold (e.g., 3600 seconds):

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let rate = state.icp_xdr_conversion_rate.as_ref().ok_or_else(|| ...)?;
        let age = env.now_timestamp_seconds().saturating_sub(rate.timestamp_seconds);
        if age > MAX_RATE_AGE_SECONDS {
            return Err(NotifyError::Other {
                error_code: NotifyErrorCode::Internal as u64,
                error_message: format!("ICP/XDR rate is stale ({age}s old)"),
            });
        }
        Ok(TokensToCycles { xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp, ... }.to_cycles(amount))
    })
}
```

Additionally, `validate_exchange_rate` should verify that the returned rate's timestamp is within an acceptable window of the current time before it is accepted into state.

## Proof of Concept
1. Monitor `get_icp_xdr_conversion_rate` and record `timestamp_seconds` of the stored rate.
2. During a planned NNS subnet or XRC canister upgrade, observe that `timestamp_seconds` stops advancing (heartbeat failures logged as `FailedToRetrieveRate`).
3. If ICP market price drops during the outage window, call `notify_top_up` with ICP.
4. `tokens_to_cycles` uses the pre-drop rate with no age check, minting more cycles than the current market rate warrants.
5. Verify by comparing cycles received against `current_market_rate * icp_burned` — the excess is the over-minted amount.

A deterministic integration test can reproduce this by: initializing CMC state with a rate timestamped at `T`, advancing the mock clock to `T + 7200`, calling `notify_top_up`, and asserting that cycles minted exceed what the current (lower) market rate would produce — confirming no staleness rejection occurs.
