### Title
Pre-EIP-155 Transaction Replay Across Aurora Engine Deployments — (`engine/src/engine.rs`)

### Summary
The `submit` and `submit_with_args` entrypoints in Aurora Engine accept pre-EIP-155 (legacy) Ethereum transactions that carry no `chain_id` in their signature. The chain-ID guard is gated on `if let Some(chain_id) = transaction.chain_id`, so when `chain_id` is `None` the check is silently skipped. A raw transaction broadcast on one Aurora deployment (e.g., testnet, chain ID 1313161555) can be replayed verbatim on any other Aurora deployment (e.g., mainnet, chain ID 1313161554) by any unprivileged NEAR caller, draining the victim's funds on the second chain.

### Finding Description

**Root cause — conditional chain-ID guard**

In `engine/src/engine.rs`, `submit_with_alt_modexp` (the shared implementation for both `submit` and `submit_with_args`) performs the following check:

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [1](#0-0) 

The guard only fires when `transaction.chain_id` is `Some(x)` **and** `x` mismatches the stored chain ID. When `transaction.chain_id` is `None` — the case for every pre-EIP-155 (type-0) transaction signed without replay protection — the entire check is bypassed and the transaction is accepted unconditionally.

**Entry points**

Both public NEAR contract methods delegate to this shared path:

- `submit` — reads raw RLP bytes and constructs `SubmitArgs { tx_data, ..Default::default() }` with no chain-ID enforcement before calling `engine::submit`.
- `submit_with_args` — reads a `SubmitArgs` struct from Borsh input and calls the same `engine::submit`. [2](#0-1) [3](#0-2) 

Both are exposed as public WASM exports callable by any NEAR account: [4](#0-3) 

**Transaction parsing**

`NormalizedEthTransaction` carries `chain_id: Option<u64>`. For a legacy (type-0) transaction signed without EIP-155, the RLP encoding omits the chain-ID fields, so `chain_id` deserialises as `None`. [5](#0-4) 

Aurora Engine is deployed across three distinct chain IDs (mainnet 1313161554, testnet 1313161555, local 1313161556): [6](#0-5) 

### Impact Explanation

**Impact: Critical — direct theft of user funds.**

A victim who signs a pre-EIP-155 ETH transfer (e.g., sending ETH to a counterparty on Aurora testnet) produces a raw signed transaction that is valid on every Aurora deployment where the victim's nonce matches. An attacker who observes that transaction (from NEAR RPC logs, a relayer, or any public source) can submit the identical bytes to Aurora mainnet via `submit`. If the victim holds funds on mainnet and the nonce is the same (trivially true for new accounts, or after the attacker waits for nonce alignment), the transfer executes on mainnet, draining the victim's mainnet balance to the same destination address.

### Likelihood Explanation

**Likelihood: Medium.**

Pre-EIP-155 transactions remain valid in the Ethereum ecosystem and are still produced by some wallets, hardware signers, and scripts that omit `chain_id`. Aurora mainnet and testnet share the same address space (same private key → same address), so a user active on both networks will frequently have matching nonces, especially at account creation (nonce 0). All NEAR transactions are publicly observable on-chain, giving any attacker a reliable source of replayable raw transaction bytes. No privileged access is required; the attacker only needs to call the public `submit` method on the target Aurora deployment.

### Recommendation

Reject any transaction whose `chain_id` is `None` (i.e., pre-EIP-155 transactions) at the `submit_with_alt_modexp` validation boundary. Change the guard from an `if let Some` conditional to an unconditional assertion:

```rust
// Require chain_id to be present and match the deployment chain:
let chain_id = transaction.chain_id
    .ok_or(EngineErrorKind::InvalidChainId)?;
if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

This mirrors the fix applied in the referenced `PaymentProcessor.sol` report: the domain (here, chain ID) must be asserted against the deployment state unconditionally, not only when the caller chooses to include it.

### Proof of Concept

1. On Aurora testnet (chain ID 1313161555), victim signs a pre-EIP-155 legacy transaction:
   - `nonce = 0`, `to = attacker_address`, `value = 1 ETH`, **no `chain_id` field** (v = 27 or 28).
2. Victim submits the transaction to Aurora testnet via `submit`. It succeeds; nonce advances to 1 on testnet.
3. Attacker captures the raw RLP bytes of the signed transaction from NEAR RPC.
4. Victim also holds 1 ETH on Aurora mainnet (chain ID 1313161554) with nonce = 0 (fresh account).
5. Attacker calls `submit` on Aurora mainnet with the identical raw bytes.
6. `submit_with_alt_modexp` parses the transaction; `transaction.chain_id` is `None`; the guard at line 1055 is skipped; nonce check passes (nonce 0 == 0); the transfer executes.
7. Victim loses 1 ETH on mainnet to the attacker's address. [1](#0-0) [7](#0-6)

pub fn submit<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let tx_data = io.read_input().to_vec();
        let current_account_id = env.current_account_id();
        let relayer_address = predecessor_address(&env.predecessor_account_id());
        let args = SubmitArgs {
            tx_data,
            ..Default::default()
        };
        let result = engine::submit(
            io,
            env,
            &args,
            state,
            current_account_id,
            relayer_address,
            handler,
        )?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);

        Ok(result)
    })
}
