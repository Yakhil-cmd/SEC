### Title
Unverified NEP-141 Registration in `deploy_erc20_token` Enables Unbacked ERC-20 Minting — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

`deploy_erc20_token` has no access control. Any unprivileged NEAR account can register an arbitrary NEP-141 account ID as a bridged token in Aurora's official registry. A malicious actor can register a fake NEP-141 contract, then directly call `ft_on_transfer` on Aurora to mint ERC-20 tokens without depositing any real tokens. The resulting ERC-20 tokens are deployed via the official Aurora Engine and appear legitimate to users and DeFi protocols.

---

### Finding Description

**Root cause 1 — No access control on `deploy_erc20_token`:**

`deploy_erc20_token` (connector.rs:112–159) only checks `require_running`. There is no `require_owner_only` or allowlist check in either the `Legacy` or `WithMetadata` branch:

```rust

    io: I, env: &E, handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;   // ← only check
        ...
        DeployErc20TokenArgs::Legacy(nep141) => {
            let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;
``` [1](#0-0) 

Any NEAR account can call this with any `nep141` account ID, causing Aurora to deploy an official ERC-20 token (hardcoded `EvmErc20.bin`) and write the mapping `nep141 → ERC-20 address` into storage.

Notably, the `WithMetadata` branch contains a comment that contradicts the missing guard:

```rust
// Safe because these promises are read-only calls to the main engine contract
// and this transaction could be executed by the owner of the contract only.
``` [2](#0-1) 

The comment documents the intended restriction, but no code enforces it.

**Root cause 2 — `ft_on_transfer` mints based solely on caller identity and `amount` field:**

`ft_on_transfer` (connector.rs:62–109) routes any call whose `predecessor_account_id` is not the connector account into `receive_erc20_tokens`:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
``` [3](#0-2) 

`receive_erc20_tokens` (engine.rs:796–844) looks up the ERC-20 address for the caller in the registry and mints `args.amount` tokens with no verification that any real tokens were actually transferred:

```rust
let erc20_token = get_erc20_from_nep141(&self.io, token)?;
let erc20_admin_address = current_address(current_account_id);
let result = self.call(
    &erc20_admin_address,
    &erc20_token,
    Wei::zero(),
    setup_receive_erc20_tokens_input(&recipient, amount),
    ...
``` [4](#0-3) 

In NEAR, any contract can call any other contract's methods directly. A malicious NEP-141 contract can call `ft_on_transfer` on Aurora without going through `ft_transfer_call`, supplying an arbitrary `amount`.

---

### Impact Explanation

**Vulnerability class:** ERC-20 mirror accounting bug / direct fund theft.

1. The attacker mints ERC-20 tokens that are not backed by any real NEP-141 tokens — the bridge accounting is broken.
2. The ERC-20 token is deployed by the official Aurora Engine (`EvmErc20.bin`), so it appears legitimate to users and DeFi protocols.
3. The attacker can use the unbacked ERC-20 tokens to drain DEX liquidity pools, lending protocols, or any other DeFi contract on Aurora that accepts the token.
4. Victims who provided real assets to those protocols lose their funds.

**Impact rating: Critical** — direct theft of user funds at-rest in DeFi protocols on Aurora.

---

### Likelihood Explanation

- Requires only deploying a NEAR contract and calling two public, permissionless functions (`deploy_erc20_token` and `ft_on_transfer`).
- No special privileges, leaked keys, or governance capture needed.
- The attack is fully self-contained and repeatable.

**Likelihood: High.**

---

### Recommendation

Add `require_owner_only` (or a dedicated allowlist check) to `deploy_erc20_token`, consistent with the existing comment that states the function "could be executed by the owner of the contract only." This mirrors the fix applied in TRST-M-2: instead of accepting an arbitrary address/account, derive the trusted mapping from an authoritative source controlled by the protocol owner.

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
+       require_owner_only(&state, &env.predecessor_account_id())?;
        ...
    })
}
``` [5](#0-4)

### Proof of Concept

1. **Attacker deploys** a malicious NEAR contract at `malicious.near`. The contract implements a `ft_on_transfer`-like call initiator but does not actually hold or transfer any tokens.

2. **Attacker calls** `deploy_erc20_token` on the official Aurora Engine with `nep141 = "malicious.near"`. No access control blocks this call. [6](#0-5) 

3. Aurora deploys an official `EvmErc20` contract and writes `malicious.near → <erc20_address>` into the `Nep141Erc20Map` storage. [7](#0-6) 

4. **Attacker's contract calls** `ft_on_transfer` on Aurora directly (not via `ft_transfer_call`) with:
   ```json
   { "sender_id": "attacker.near", "amount": "1000000000000000000000", "msg": "<attacker_evm_address>" }
   ```
   The `predecessor_account_id` is `malicious.near`.

5. Aurora's `ft_on_transfer` routes to `receive_erc20_tokens`, finds `malicious.near` in the registry, and mints `1000000000000000000000` ERC-20 tokens to the attacker's EVM address — with zero real tokens deposited. [8](#0-7) 

6. Attacker swaps the unbacked ERC-20 tokens on a DEX for real bridged assets (e.g., USDC, wETH), draining liquidity providers' funds. Users trust the ERC-20 because it was deployed by the official Aurora Engine.
