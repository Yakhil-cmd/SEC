### Title
Unbounded `move_to_block` Loop in Hashchain Causes Gas Exhaustion and Temporary Bridge Freeze — (`engine-hashchain/src/hashchain.rs`)

---

### Summary

The `move_to_block` function in `engine-hashchain/src/hashchain.rs` contains an unbounded `while` loop that iterates once per skipped NEAR block, performing a keccak hash computation each iteration. This function is invoked on **every** Aurora Engine contract call via `load_hashchain` → `with_hashchain` / `with_logs_hashchain`. If the hashchain's stored block height falls sufficiently far behind the current NEAR block height, the loop exhausts the NEAR 300 TGas transaction gas limit, causing the call to fail at the runtime level. Because the hashchain state is only persisted on success, the stored height never advances, and every subsequent call fails identically — freezing the bridge until privileged admin intervention.

---

### Finding Description

**Root cause — unbounded loop in `move_to_block`:** [1](#0-0) 

```rust
while self.current_block_height < next_block_height {
    self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
        &self.chain_id,
        self.contract_account_id.as_bytes(),
        self.current_block_height,
        self.previous_block_hashchain,
    );
    self.block_hashchain_computer.clear_txs();
    self.current_block_height += 1;
}
```

Each iteration calls `compute_block_hashchain`, which internally calls `keccak` twice (once for the Merkle tree hash, once for the block hashchain). Both calls go through `aurora_engine_sdk::keccak`, which maps to the NEAR host function `keccak256`. The NEAR gas cost for `keccak256` is approximately 1 TGas base + ~0.01 TGas/byte. For a ~370-byte input (chain_id[32] + account_id + block_height[8] + previous_hashchain[32] + txs_hash[32] + bloom[256]), each keccak call costs roughly 4–5 TGas, so each loop iteration costs ~9–10 TGas. With the 300 TGas NEAR transaction limit, the loop can complete at most ~30–32 iterations before gas is exhausted.

**Call path — triggered on every contract method:** [2](#0-1) 

```rust
fn load_hashchain<I: IO>(io: &I, block_height: u64) -> Result<Option<Hashchain>, ContractError> {
    let mut maybe_hashchain = read_current_hashchain(io)?;
    if let Some(hashchain) = maybe_hashchain.as_mut()
        && block_height > hashchain.get_current_block_height()
    {
        hashchain.move_to_block(block_height)?;
    }
    Ok(maybe_hashchain)
}
```

`load_hashchain` is called at the top of both `with_hashchain` and `with_logs_hashchain`: [3](#0-2) 

These wrappers are used by every production contract method, including `submit`, `call`, `register_relayer`, `ft_on_transfer`, and all admin methods.

**Hashchain state is only persisted on success:** [4](#0-3) 

The `save_hashchain` call is inside the `if let Some(mut hashchain) = maybe_hashchain` block that executes only after `f` succeeds. When gas is exhausted inside `load_hashchain` (before `f` is even called), the NEAR runtime terminates execution and rolls back all state changes. The stored `current_block_height` is never updated. Every subsequent call faces the same or larger gap and fails identically.

**Scenario — period of inactivity after hashchain activation:**

Once the key manager calls `start_hashchain` to activate the hashchain: [5](#0-4) 

…the hashchain's `current_block_height` is set to approximately the current NEAR block height. If the Aurora Engine then experiences a gap of ≥ ~32 consecutive NEAR blocks (~32 seconds at 1 block/second) with no successful transactions, the next call will exhaust gas in `move_to_block`. Because the failure prevents the hashchain from advancing, all subsequent calls also fail, freezing the bridge.

---

### Impact Explanation

**Temporary freezing of funds (High).**

All Aurora Engine entry points — `submit`, `call`, `ft_on_transfer`, `withdraw`, and others — route through `with_hashchain` or `with_logs_hashchain`. Once the block gap exceeds the gas-exhaustion threshold, every call fails before any business logic executes. ETH and ERC-20 tokens locked in the bridge cannot be moved, minted, or withdrawn. The freeze persists until the privileged key manager pauses the contract and calls `start_hashchain` with a fresh block height, which is a non-trivial operational recovery requiring coordinated admin action.

---

### Likelihood Explanation

**Medium.**

The hashchain must be active (requires prior key manager action). Once active, a gap of only ~32 NEAR blocks (~32 seconds of zero successful transactions) is sufficient to trigger the freeze. This threshold can be reached:

- Naturally during low-traffic periods or scheduled maintenance windows.
- During a contract pause (the contract is paused → no transactions succeed → hashchain falls behind → unpausing immediately re-freezes the contract on the first call).
- By an attacker who front-runs or otherwise suppresses transactions for a brief window, then submits a call to lock the hashchain at the stale height.

Any unprivileged EVM user or NEAR account can submit the triggering call — no special role is required.

---

### Recommendation

1. **Bound the loop**: In `move_to_block`, add a maximum iteration count (e.g., 20 blocks per call). Return a new error variant (e.g., `BlockGapTooLarge`) if the gap exceeds the bound, and let callers handle partial advancement.
2. **Persist partial progress**: If a partial advance is performed, save the intermediate hashchain state so subsequent calls continue from where the previous one left off rather than restarting from the stale height.
3. **Advance hashchain on failure path**: Consider saving the advanced hashchain even when `f` returns an error, so the stored height always moves forward regardless of business-logic outcomes.

---

### Proof of Concept

1. Deploy Aurora Engine on NEAR with hashchain enabled (key manager calls `start_hashchain` at block height N).
2. Allow 33+ NEAR blocks to pass with no successful Aurora Engine transactions (natural inactivity or attacker-induced suppression).
3. Submit any valid Aurora Engine call (e.g., a simple ETH transfer via `submit`).
4. Observe: `load_hashchain` calls `move_to_block(N + 33)`, which iterates 33 times × ~9 TGas = ~297 TGas, exhausting the 300 TGas limit. The NEAR runtime aborts execution.
5. The hashchain stored height remains at N. Every subsequent call repeats step 4 and fails identically.
6. All bridge operations (deposits, withdrawals, EVM calls) are frozen until the key manager executes a privileged `start_hashchain` reset.

    pub fn move_to_block(
        &mut self,
        next_block_height: u64,
    ) -> Result<(), BlockchainHashchainError> {
        if next_block_height <= self.current_block_height {
            return Err(BlockchainHashchainError::BlockHeightIncorrect);
        }

        while self.current_block_height < next_block_height {
            self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
                &self.chain_id,
                self.contract_account_id.as_bytes(),
                self.current_block_height,
                self.previous_block_hashchain,
            );

            self.block_hashchain_computer.clear_txs();
            self.current_block_height += 1;
        }

        Ok(())
    }

pub fn with_hashchain<I, E, T, F>(
    mut io: I,
    env: &E,
    function_name: &str,
    f: F,
) -> Result<T, ContractError>
where
    I: IO + Copy,
    E: Env,
    F: for<'a> FnOnce(CachedIO<'a, I>) -> Result<T, ContractError>,
{
    let block_height = env.block_height();
    let maybe_hashchain = load_hashchain(&io, block_height)?;

    let cache = RefCell::new(IOCache::default());
    let hashchain_io = CachedIO::new(io, &cache);
    let result = f(hashchain_io)?;

    if let Some(mut hashchain) = maybe_hashchain {
        let cache_ref = cache.borrow();
        hashchain.add_block_tx(
            block_height,
            function_name,
            &cache_ref.input,
            &cache_ref.output,
            &Bloom::default(),
        )?;
        save_hashchain(&mut io, &hashchain)?;
    }

    Ok(result)
}
