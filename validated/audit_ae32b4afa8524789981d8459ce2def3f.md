Audit Report

## Title
Unbounded Neuron Scan in `maybe_finalize_disburse_maturity` Can Exhaust Instruction Limit and Permanently Latch `is_finalizing_disburse_maturity` - (File: rs/sns/governance/src/governance.rs)

## Summary
`maybe_finalize_disburse_maturity` in the SNS governance canister performs a full, unbounded scan of `self.proto.neurons.values()` before any `await` point, collecting every neuron with a ready disbursement, then processes all of them in a single periodic task execution. The `is_finalizing_disburse_maturity` guard is set to `Some(true)` before the scan and only reset at the very end of the function; if the synchronous scan segment traps due to instruction exhaustion, the flag is permanently latched, blocking all future finalization attempts and leaving users' maturity disbursements stuck. The NNS governance canister has already addressed this exact pattern with a bounded, indexed approach.

## Finding Description
In `rs/sns/governance/src/governance.rs`, `maybe_finalize_disburse_maturity` (line 4921) sets `self.proto.is_finalizing_disburse_maturity = Some(true)` at line 4935, then immediately performs a synchronous full-table scan:

```rust
let neuron_id_and_disbursements: Vec<(NeuronId, DisburseMaturityInProgress)> = self
    .proto
    .neurons
    .values()
    .filter_map(|neuron| { ... })
    .collect();   // lines 4938–4975, no bound, no early exit
```

This entire `.collect()` executes in a single synchronous execution segment (before the first `await` at line 5046). The IC enforces a per-message instruction limit; if the neuron map is large enough, this segment traps. Because the flag was set at line 4935 and the reset at line 5082 is never reached, `can_finalize_disburse_maturity()` will return `false` on every subsequent periodic task invocation, permanently blocking all maturity disbursements for the SNS.

The processing loop (lines 4976–5081) also processes every collected disbursement, each with an `await`-ed cross-canister ledger call. While each `await` splits execution into separate segments (limiting per-segment instruction cost), the synchronous setup work per iteration still accumulates before the first `await` in the initial call frame.

`maybe_move_staked_maturity` (lines 5087–5099) has the same unbounded full-scan pattern, though it is synchronous and has no flag-latch risk.

By contrast, NNS governance uses `get_neuron_ids_ready_to_finalize_maturity_disbursement` (an indexed lookup) and processes exactly **one** disbursement per timer invocation (`rs/nns/governance/src/governance/disburse_maturity.rs`, lines 451–469), and caps `unstake_maturity_of_dissolved_neurons` at `MAX_NEURONS_TO_UNSTAKE = 100` (`rs/nns/governance/src/governance.rs`, line 6394). SNS governance has no equivalent safeguard.

## Impact Explanation
This matches the **High** impact class: "Application/platform-level DoS... or subnet availability impact" and "Significant SNS... security impact with concrete user or protocol harm." If the periodic task traps with the flag latched, every SNS user who called `disburse_maturity` has their pending disbursement permanently frozen in the canister. Recovery requires a governance upgrade proposal to clear the flag — an out-of-band privileged action that may not be available quickly on all SNS deployments. The maturity tokens are not lost but are inaccessible until the flag is cleared and the root cause addressed.

## Likelihood Explanation
`disburse_maturity` is an unprivileged operation callable by any neuron holder. An SNS with thousands of active neurons (realistic for a popular project) where many disbursements mature simultaneously — either organically or via a coordinated burst — can trigger the condition. The per-message instruction limit for timer callbacks on the IC is finite; the scan cost scales linearly with neuron count and disbursement queue depth. No special privileges, social engineering, or external compromise are required.

## Recommendation
1. Replace the full-scan + process-all loop in `maybe_finalize_disburse_maturity` with a bounded approach: process at most N disbursements per periodic task invocation (e.g., `MAX_DISBURSEMENTS_PER_TICK = 100`), analogous to `MAX_NEURONS_TO_UNSTAKE` in NNS governance.
2. Introduce a maturity disbursement index in SNS governance (as NNS does via `get_neuron_ids_ready_to_finalize_maturity_disbursement`) to avoid scanning all neurons on every tick.
3. Protect `is_finalizing_disburse_maturity` with a RAII guard or `defer`-style reset so it is cleared even if the function traps mid-execution.
4. Apply a similar bound to `maybe_move_staked_maturity` to prevent instruction exhaustion as neuron counts grow.

## Proof of Concept
1. Deploy an SNS canister with a large neuron map (e.g., 50,000+ neurons, achievable via repeated `claim_neuron` calls from many principals).
2. Have a significant fraction of neuron holders call `disburse_maturity`, accumulating `DisburseMaturityInProgress` entries with the same `finalize_disbursement_timestamp_seconds`.
3. Wait for the finalization timestamp to pass.
4. Observe the next `run_periodic_tasks` invocation calls `maybe_finalize_disburse_maturity`.
5. The synchronous `.collect()` scan at lines 4938–4975 iterates all neurons in a single execution segment; with sufficient neuron count, the IC instruction counter is exhausted and the message traps.
6. Verify that `self.proto.is_finalizing_disburse_maturity` remains `Some(true)` (line 4935 was executed, line 5082 was not).
7. Confirm that all subsequent periodic task invocations return early at `can_finalize_disburse_maturity()`, leaving all pending disbursements permanently stuck.
8. A deterministic integration test using PocketIC can reproduce this by pre-populating the neuron store and mocking the instruction counter or using a sufficiently large synthetic neuron set.
