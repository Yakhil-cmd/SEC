### Title
Hardcoded `is_deployment = false` in L1 Transaction Intrinsic Gas Calculation Undercharges Deployment Transactions - (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

### Summary

`calculate_tx_intrinsic_gas` is called with a hardcoded `false` for `is_deployment` when processing all L1 transactions, regardless of whether the actual transaction is a deployment. If an L1 deployment transaction (with `to = None`) is submitted, the intrinsic gas is undercharged by `DEPLOYMENT_TX_EXTRA_INTRINSIC_GAS` (32,000 gas) plus the initcode word cost, while the transaction is still executed as a deployment. This is a resource accounting bug directly analogous to the GMX wrong-parameter class: a context-dependent flag is hardcoded to the wrong value, causing a critical calculation to produce an incorrect result.

### Finding Description

In `prepare_and_check_resources` (called from `process_l1_transaction`), the intrinsic gas is computed as:

```rust
// L1 transactions never carry an access list or authorization list, so
// the corresponding counts are 0 and the intrinsic-gas helper collapses
// to TX_INTRINSIC_GAS + calldata-token cost (deployment is also false).
let intrinsic_gas = calculate_tx_intrinsic_gas(
    transaction.calldata().len() as u64,
    calldata_tokens,
    false, // is_deployment   <-- hardcoded
    0,
    0,
    0,
);
``` [1](#0-0) 

`calculate_tx_intrinsic_gas` branches on `is_deployment` to add `DEPLOYMENT_TX_EXTRA_INTRINSIC_GAS` (32,000) and the per-word initcode cost:

```rust
if is_deployment {
    intrinsic_gas = intrinsic_gas.saturating_add(DEPLOYMENT_TX_EXTRA_INTRINSIC_GAS);
    let initcode_gas_cost =
        evm_interpreter::gas_constants::INITCODE_WORD_COST * calldata_len.div_ceil(32);
    intrinsic_gas = intrinsic_gas.saturating_add(initcode_gas_cost);
}
``` [2](#0-1) 

However, the execution path that follows does **not** enforce the same assumption. `execute_or_deploy_inner` derives `to_ee_type` from the actual transaction:

```rust
let to_ee_type = transaction.is_deployment();
let TxExecutionResult { ... } = match to_ee_type {
    Some(to_ee_type) => Self::perform_deployment(...),
    None => Self::execute_call(...),
};
``` [3](#0-2) 

There is no explicit guard in `process_l1_transaction` that rejects an L1 transaction whose `to` field is absent. The documentation notes "deployment transactions from L1 transactions are not supported," but this is not enforced by a validation error — it is only assumed by the hardcoded `false`. [4](#0-3) 

For comparison, the L2 (ZK) transaction path correctly derives `is_deployment` from the transaction:

```rust
let is_deployment = transaction.is_deployment().is_some();
...
let intrinsic_gas = calculate_tx_intrinsic_gas(
    calldata.len() as u64,
    calldata_tokens,
    is_deployment,   // <-- context-derived
    ...
);
``` [5](#0-4) 

### Impact Explanation

An L1 deployment transaction is charged only `TX_INTRINSIC_GAS` (21,000) instead of `TX_INTRINSIC_GAS + DEPLOYMENT_TX_EXTRA_INTRINSIC_GAS + initcode_word_cost`. The difference is at minimum 32,000 gas, plus 2 gas per 32-byte word of initcode. This extra gas is available to the constructor execution frame. The attacker can therefore:

1. Deploy contracts whose constructors would otherwise exhaust gas under correct accounting.
2. Obtain more execution gas than the L1 deposit covers, breaking the invariant that `total_deposited >= gas_price * gas_limit` enforced at line 133–137 of the same file. [6](#0-5) 

### Likelihood Explanation

L1 transactions are submitted by the L1 bridge. Whether the bridge permits `to = None` (deployment) depends on the L1 contract, which is outside ZKsync OS scope. However, the ZKsync OS bootloader itself imposes no guard, so any L1 deployment transaction that reaches the bootloader is processed with undercharged intrinsic gas. The vulnerability is reachable by any party that can submit an L1 transaction with `to = None` to the bridge.

### Recommendation

Either:

1. **Derive `is_deployment` from the actual transaction** in `prepare_and_check_resources`, mirroring the L2 path:
   ```rust
   let is_deployment = transaction.is_deployment().is_some();
   let intrinsic_gas = calculate_tx_intrinsic_gas(
       transaction.calldata().len() as u64,
       calldata_tokens,
       is_deployment, // derived, not hardcoded
       0, 0, 0,
   );
   ```

2. **Or explicitly reject L1 deployment transactions** before resource calculation:
   ```rust
   if transaction.is_deployment().is_some() {
       return Err(internal_error!("L1 deployment transactions are not supported").into());
   }
   ```

### Proof of Concept

1. Craft an L1 transaction with `to = None` and large initcode (e.g., 10,000 bytes → 313 words × 2 = 626 extra gas, plus 32,000 base = 32,626 gas undercharged).
2. Set `gas_limit` such that `gas_limit - 21_000` is just enough for the constructor, but `gas_limit - 53_626` would be insufficient.
3. Submit via the L1 bridge.
4. `prepare_and_check_resources` charges only 21,000 intrinsic gas (hardcoded `false`).
5. `execute_or_deploy_inner` detects `to = None`, calls `perform_deployment`.
6. The constructor runs with 32,626 more gas than it paid for, succeeding where it should have reverted. [7](#0-6) [8](#0-7)

```rust
pub fn calculate_tx_intrinsic_gas(
    calldata_len: u64,
    calldata_tokens: u64,
    is_deployment: bool,
    access_list_accounts: u64,
    access_list_storage_keys: u64,
    authorization_list_num: u64,
) -> u64 {
    let mut intrinsic_gas = TX_INTRINSIC_GAS;

    if is_deployment {
        intrinsic_gas = intrinsic_gas.saturating_add(DEPLOYMENT_TX_EXTRA_INTRINSIC_GAS);
        let initcode_gas_cost =
            evm_interpreter::gas_constants::INITCODE_WORD_COST * calldata_len.div_ceil(32);
        intrinsic_gas = intrinsic_gas.saturating_add(initcode_gas_cost);
    }
    intrinsic_gas =
        intrinsic_gas.saturating_add(calldata_tokens.saturating_mul(CALLDATA_TOKEN_GAS_COST));

    // EIP-2930 access list: per-address + per-storage-key.
    intrinsic_gas = intrinsic_gas.saturating_add(
        access_list_accounts.saturating_mul(evm_interpreter::gas_constants::ACCESS_LIST_ADDRESS),
    );
    intrinsic_gas = intrinsic_gas.saturating_add(
        access_list_storage_keys
            .saturating_mul(evm_interpreter::gas_constants::ACCESS_LIST_STORAGE_KEY),
    );

    // EIP-7702 authorization list: per-authorization. We precharge the
    // empty-account cost; when the authority turns out to be non-empty the
    // delta (NEWACCOUNT - PER_AUTH_BASE_COST) is added back as a gas refund
    // inside `validate_and_apply_delegation`.
    intrinsic_gas = intrinsic_gas.saturating_add(
        authorization_list_num.saturating_mul(evm_interpreter::gas_constants::NEWACCOUNT),
    );

    intrinsic_gas
}
```
