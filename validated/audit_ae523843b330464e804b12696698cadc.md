### Title
Pre-EIP-155 Legacy Transaction Cross-Chain Replay Attack - (File: `engine/src/engine.rs`, `engine-transactions/src/legacy.rs`)

---

### Summary

The Aurora Engine deliberately accepts pre-EIP-155 legacy transactions (those with `v = 27` or `v = 28`, carrying no chain ID) to support EIP-1820 registry deployment. Because the chain-ID validation in `submit_with_alt_modexp` is guarded by an `if let Some(chain_id)` pattern, it is silently skipped whenever the transaction carries no chain ID. A signed pre-EIP-155 transaction observed on one Aurora deployment (e.g., testnet, chain ID `1313161555`) can be replayed verbatim on any other Aurora deployment (e.g., mainnet, chain ID `1313161554`) by any unprivileged actor, draining the victim's funds on the second chain.

---

### Finding Description

**Root cause — conditional chain-ID check in `engine/src/engine.rs`:**

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [1](#0-0) 

When `transaction.chain_id` is `None` the entire guard is bypassed; no chain-ID check is performed and execution continues normally.

**How `chain_id` becomes `None` — `engine-transactions/src/legacy.rs`:**

`LegacyEthSignedTransaction::sender()` extracts the chain ID from the `v` field. For `v ∈ {27, 28}` (pre-EIP-155) it returns `chain_id = None`:

```rust
let (chain_id, rec_id) = match self.v {
    0..=26 | 29..=34 => return Err(Error::InvalidV),
    27..=28 => (None, ...),   // ← pre-EIP-155: no chain ID
    _ => (Some((self.v - 35) / 2), ...),
};
``` [2](#0-1) 

`NormalizedEthTransaction` propagates this `None` directly:

```rust
Legacy(tx) => Self {
    address: tx.sender()?,
    chain_id: tx.chain_id(),   // None for pre-EIP-155
    ...
}
``` [3](#0-2) 

**Why pre-EIP-155 transactions are accepted:**

The changelog records that pre-EIP-155 support was explicitly re-enabled in v2.6.0 to allow EIP-1820 registry deployment:

> "Original ETH transactions which do not contain a Chain ID are allowed again to allow for use of EIP-1820" [4](#0-3) 

This means the engine's `submit` / `submit_with_args` entry points accept any pre-EIP-155 transaction from any caller without any chain-binding check. [5](#0-4) 

---

### Impact Explanation

**Impact: High — Theft of user funds (direct)**

Aurora is deployed on three distinct chains sharing the same contract address pattern:

| Network | Chain ID |
|---|---|
| Mainnet | `1313161554` |
| Testnet | `1313161555` |
| Local | `1313161556` |

A pre-EIP-155 signed transaction is chain-agnostic. Any attacker who observes such a transaction on one Aurora network can submit the identical raw bytes to any other Aurora network. Because nonces are per-address and independent across chains, and because the signature is valid on all chains, the engine will execute the transaction on the second chain and debit the victim's balance there.

Concrete theft scenario:
1. Victim signs a pre-EIP-155 ETH transfer (or ERC-20 `approve`/`transfer`) on Aurora testnet.
2. Attacker copies the raw transaction bytes from the testnet mempool or block.
3. Attacker submits the same bytes to Aurora mainnet via `submit`.
4. The engine passes signature verification (ecrecover succeeds), skips the chain-ID check (chain_id is `None`), passes the nonce check (nonce matches on mainnet), and executes the transfer — draining the victim's mainnet ETH or ERC-20 balance.

The victim's mainnet funds are stolen without any key compromise.

---

### Likelihood Explanation

**Likelihood: Medium**

- Pre-EIP-155 transactions are rare but explicitly supported and documented as intentional.
- EIP-1820 deployment is the primary use case; any user or protocol deploying EIP-1820 on Aurora testnet produces a replayable transaction.
- Users commonly control the same address on both mainnet and testnet (same private key, same derived address).
- All Aurora transactions are publicly visible on-chain; no privileged access is needed to observe and copy them.
- The attacker's only requirement is to monitor one Aurora network and submit to another — a trivially automated operation.

---

### Recommendation

1. **Reject pre-EIP-155 transactions outright** (revert to the v2.4.0 policy from PR #432). EIP-1820 can be deployed via a chain-ID-aware wrapper or a privileged admin call instead.
2. **If pre-EIP-155 support must be retained**, add an explicit allowlist of known pre-EIP-155 transaction hashes (e.g., the canonical EIP-1820 deployer transaction) and reject all other chain-ID-less transactions.
3. **Alternatively**, treat `chain_id = None` as equivalent to "matches any chain" only for a hardcoded set of well-known deployment transactions, and reject all others.

---

### Proof of Concept

```
1. On Aurora testnet, sign and submit a legacy ETH transfer with chain_id = None (v = 27 or 28):
   tx = { nonce: N, gas_price: P, gas_limit: G, to: VICTIM_CONTROLLED_ADDR, value: V, data: [] }
   signed_bytes = rlp_encode(tx, v=27, r, s)   // no chain ID in signing hash

2. Observe signed_bytes in the testnet block/mempool.

3. Submit signed_bytes unchanged to Aurora mainnet via the `submit` NEAR method.

4. Engine flow on mainnet:
   a. EthTransactionKind::try_from(signed_bytes) → Legacy(tx) with v=27
   b. NormalizedEthTransaction::try_from → chain_id = None
   c. submit_with_alt_modexp:
      - if let Some(chain_id) = None  →  guard NOT entered, no InvalidChainId error
      - check_nonce(&io, &sender, &tx.nonce) → passes if nonce matches on mainnet
      - engine executes transfer, debiting sender's mainnet balance

5. Victim's mainnet ETH is transferred to the `to` address specified in the original testnet transaction.
```
