### Title
Unhandled Partial Fill in `ft_transfer_call` Omni Exit Path Causes Permanent Loss of User Funds - (File: `engine-precompiles/src/native.rs`)

### Summary
When a user exits ERC-20 (or base ETH) tokens to NEAR using the Omni message path in the `ExitToNear` precompile, the protocol calls `ft_transfer_call` on the target NEP-141 contract. Per the NEP-141 standard, the receiver's `ft_on_transfer` callback may return a portion of the tokens as "unused" (partial fill), which the NEP-141 contract refunds back to Aurora via `ft_resolve_transfer`. The Aurora Engine attaches no callback capable of re-minting the returned ERC-20 tokens for the user. The ERC-20 burn is irreversible, but the returned NEP-141 tokens accumulate silently in Aurora's NEP-141 balance with no recovery path, causing a permanent loss of user principal.

### Finding Description

In `exit_erc20_token_to_near`, the Omni branch sets `transfer_near_args` to `None`: [1](#0-0) 

The same pattern applies to the base-token Omni branch in `exit_base_token_to_near`: [2](#0-1) 

Back in `ExitToNear::run`, the callback is only attached when `callback_args != ExitToNearPrecompileCallbackArgs::default()`. For the Omni path, `transfer_near` is `None` and, without the `error_refund` feature, `refund` is also `None`, so `callback_args` equals the default and **no callback is attached at all**: [3](#0-2) 

Even when the `error_refund` feature is compiled in (so `refund` is `Some`), the `exit_to_near_precompile_callback` only branches on complete promise failure to re-mint ERC-20 tokens. On a **successful** `ft_transfer_call` result — which is what NEAR reports even when the receiver returns tokens as unused — the callback does nothing for the Omni path because `transfer_near` is `None`: [4](#0-3) 

The NEP-141 `ft_resolve_transfer` mechanism silently credits the returned tokens back to Aurora's NEP-141 balance. There is no subsequent EVM-side step to re-mint the corresponding ERC-20 tokens for the user.

### Impact Explanation

A user who calls `ExitToNear` with an Omni message burns X ERC-20 tokens irreversibly. If the NEAR-side receiver returns Y tokens as unused (a normal outcome for any DeFi protocol that partially fills an order), the user receives only X−Y NEP-141 tokens at the destination while Y NEP-141 tokens accumulate in Aurora's account. No on-chain path exists to reclaim or re-mint those Y tokens. The loss is permanent and proportional to the unused amount returned by the receiver.

**Impact class: Critical — Permanent freezing of funds.**

### Likelihood Explanation

The Omni exit path is the intended mechanism for Aurora EVM users to interact with arbitrary NEAR DeFi protocols (DEXes, lending markets, bridges) that implement `ft_on_transfer`. Partial fills — where a receiver accepts less than the full amount and returns the remainder — are standard, expected behavior in these protocols. Any user routing tokens through such a protocol via the Omni exit path is exposed. No special privilege or attacker capability is required beyond calling the `ExitToNear` precompile, which is permissionless.

**Likelihood: Medium** — triggered whenever a legitimate NEAR DeFi receiver partially fills an order, which is routine for DEX-style contracts.

### Recommendation

Attach a callback to the `ft_transfer_call` promise in the Omni exit path. The callback should:

1. Read the net-transferred amount from the `ft_transfer_call` promise result (the return value of `ft_resolve_transfer`, a serialized `U128`).
2. Compute the unused amount: `unused = original_amount − net_transferred`.
3. If `unused > 0`, re-mint `unused` ERC-20 tokens to the original sender's EVM address (mirroring the existing `error_refund` re-mint logic in `refund_on_error`).

This requires extending `ExitToNearPrecompileCallbackArgs` with an Omni-specific refund field and updating `exit_to_near_precompile_callback` to handle the partial-success case by reading the promise return bytes.

### Proof of Concept

1. User holds X wUSDC ERC-20 tokens on Aurora (backed by X USDC NEP-141 held by Aurora).
2. User calls the `ExitToNear` precompile with flag `0x1` (ERC-20), amount X, and an Omni message targeting a NEAR DEX's `ft_on_transfer`-compatible contract.
3. The ERC-20 contract burns X tokens from the user's EVM balance — irreversible.
4. Aurora schedules `ft_transfer_call(dex_account, X, msg)` on the NEP-141 USDC contract with no EVM-side callback (or only a failure-only callback with `error_refund`).
5. The DEX's `ft_on_transfer` partially fills the order and returns Y tokens as unused.
6. NEP-141's `ft_resolve_transfer` refunds Y tokens to Aurora's NEP-141 balance. The `ft_transfer_call` promise resolves as `Successful`.
7. Aurora's EVM callback (if any) sees `Successful` and, since `transfer_near` is `None`, does nothing.
8. Y NEP-141 tokens sit permanently in Aurora's account. The user's X ERC-20 tokens are gone. The user has lost Y tokens with no recourse. [5](#0-4) [6](#0-5)
