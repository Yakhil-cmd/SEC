Audit Report

## Title
ckERC20 Minter Mints Full Event-Emitted Amount Without Verifying Actual Received Balance for Fee-on-Transfer ERC20 Tokens - (File: rs/ethereum/cketh/minter/ERC20DepositHelper.sol, rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol, rs/ethereum/cketh/minter/src/deposit.rs)

## Summary

Both ckERC20 deposit helper contracts emit the caller-supplied `amount` parameter in their deposit events rather than the actual balance delta received by the minter address. The IC ckETH minter canister scrapes these events and unconditionally mints ckERC20 tokens equal to the event-logged `amount` with no on-chain balance reconciliation. For any fee-on-transfer ERC20 token, this inflates the ckERC20 supply beyond actual collateral held, breaking the ledger conservation invariant and causing the last withdrawers to permanently lose funds.

## Finding Description

**Helper contract side — Ethereum:**

`ERC20DepositHelper.sol` `deposit()` at L498–503 calls `safeTransferFrom(msg.sender, cketh_minter_main_address, amount)` and immediately emits `ReceivedErc20(erc20_address, msg.sender, amount, principal)` using the input parameter `amount`, not the actual balance delta at `cketh_minter_main_address`. [1](#0-0) 

`DepositHelperWithSubaccount.sol` `depositErc20()` at L511–532 does the same: `safeTransferFrom(msg.sender, minterAddress, amount)` followed by `emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, ...)` using the input `amount`. [2](#0-1) 

Neither contract measures `balanceOf(minterAddress)` before and after the transfer to compute the actual received amount.

**IC minter side — Rust canister:**

`parser.rs` L86–97 reads the event's data field directly as the deposit value: `value: Erc20Value::from_be_bytes(value_bytes)`. There is no cross-check against any on-chain balance query. [3](#0-2) 

`deposit.rs` L73–81 mints exactly `event.value()` ckERC20 tokens to the beneficiary via ICRC-1 `transfer`, with no balance-delta verification step. [4](#0-3) 

`register_deposit_events` at L311–360 processes all scraped events and schedules minting without any reconciliation against actual Ethereum balances. [5](#0-4) 

`state.rs` `erc20_add` at L742–756 adds the full event-reported deposit value to internal accounting, compounding the divergence from actual on-chain holdings. [6](#0-5) 

A search across all minter source files confirms no `balanceOf` reconciliation exists in the deposit flow path.

## Impact Explanation

For every deposit of a fee-on-transfer ERC20 token, the ckERC20 ledger total supply grows by `amount` while the minter's Ethereum address only holds `amount - fee`. The deficit compounds with each deposit. When users withdraw, the minter constructs Ethereum transactions for the full withdrawal amount. Once cumulative fees exceed the minter's surplus, withdrawal transactions fail on-chain due to insufficient ERC20 balance, permanently locking ckERC20 tokens for the last depositors. This constitutes **illegal minting and protocol insolvency for an in-scope chain-key/ck-token asset**, matching the High impact category ("Significant Chain Fusion, ck-token, ledger security impact with concrete user or protocol harm") and potentially Critical if the affected ck-token holds assets exceeding $1M.

## Likelihood Explanation

The exploit requires a supported ERC20 token to have an active transfer fee. Two realistic paths exist: (1) USDT has a built-in fee mechanism currently set to 0 but activatable by Tether at any time without IC governance action — if ckUSDT is supported, Tether activating its fee immediately triggers the vulnerability for all subsequent deposits; (2) any future NNS proposal adding a fee-on-transfer token (intentionally or by oversight) enables the issue. Once a token is supported, the deposit flow is fully permissionless — any unprivileged Ethereum user can trigger deposits. Likelihood is **Medium**: not currently active for USDC/USDT at zero fee, but a realistic and documented future condition.

## Recommendation

In both helper contracts, measure the actual balance delta instead of emitting the input `amount`:

```solidity
uint256 balanceBefore = IERC20(erc20Address).balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 received = IERC20(erc20Address).balanceOf(minterAddress) - balanceBefore;
emit ReceivedEthOrErc20(erc20Address, msg.sender, received, principal, subaccount);
```

This ensures the event-logged value always equals actual collateral received. As a defense-in-depth measure, the minter canister could also periodically reconcile its internal `erc20_balances` against on-chain `eth_call` balance queries and halt minting when a deficit is detected.

## Proof of Concept

1. A fee-on-transfer ERC20 token `FEE_TOKEN` (1% fee per transfer) is added as a supported ckERC20 token via NNS proposal, or USDT activates its built-in fee mechanism.
2. Alice calls `depositErc20(FEE_TOKEN, 1_000_000, alice_principal, subaccount)` on `DepositHelperWithSubaccount`.
3. `safeTransferFrom(Alice, minterAddress, 1_000_000)` executes; due to the 1% fee, `minterAddress` receives `990_000`. The contract emits `ReceivedEthOrErc20(FEE_TOKEN, Alice, 1_000_000, ...)`.
4. The IC minter scrapes the log, reads `value = 1_000_000` from `parser.rs` L97, and mints `1_000_000` ckFEE_TOKEN to Alice via `deposit.rs` L80.
5. After 100 such deposits: minter holds `99_000_000` FEE_TOKEN on Ethereum; ckFEE_TOKEN total supply is `100_000_000`.
6. The 100th user's withdrawal transaction fails on Ethereum (minter holds 0 FEE_TOKEN), permanently locking `1_000_000` ckFEE_TOKEN.
7. A deterministic integration test or PocketIC fork test can reproduce this by deploying a mock ERC20 with a configurable transfer fee, performing N deposits through the helper contract, and asserting that the minter's Ethereum balance is less than the ckERC20 total supply after minting.
