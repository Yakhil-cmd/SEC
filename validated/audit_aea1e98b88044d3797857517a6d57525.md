### Title
Insufficient Gas Check in `return_promise` Causes Permanent Freezing of User NEAR Deposits — (`File: engine/src/contract_methods/connector.rs`)

### Summary
`calculate_attached_gas` in `engine/src/contract_methods/connector.rs` silently returns `NearGas::new(0)` when the caller's prepaid gas is exhausted, instead of returning an error. This 0-gas value is forwarded to `return_promise`, which creates a cross-contract promise to the ETH connector (e.g., `engine_storage_deposit`) with 0 gas attached. The promise fails silently. Because Aurora Engine created the promise, NEAR's runtime refunds the attached deposit back to the Aurora Engine contract — not to the original user. There is no failure callback to detect this and return the funds. The user's NEAR is permanently frozen inside the Aurora Engine contract.

### Finding Description

`calculate_attached_gas` computes how much gas to forward to the downstream promise: [1](#0-0) 

```rust
// TODO: Return `Result` with an error about lacking of gas instead.
fn calculate_attached_gas<E: Env>(env: &E) -> NearGas {
    let required_gas = env.used_gas().saturating_add(GAS_FOR_PROMISE_CREATION);
    if required_gas >= env.prepaid_gas() {
        NearGas::new(0)   // ← silently returns 0 instead of erroring
    } else {
        env.prepaid_gas() - required_gas
    }
}
```

The developer-written TODO comment explicitly acknowledges this is wrong: *"Return `Result` with an error about lacking of gas instead."*

This value is consumed unconditionally by `return_promise`: [2](#0-1) 

`return_promise` is called by `storage_deposit` with the full user-attached NEAR deposit: [3](#0-2) 

```rust
pub fn storage_deposit<I: IO + Copy + PromiseHandler, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    ...
    return_promise(
        io, env,
        "engine_storage_deposit",
        args,
        Yocto::new(env.attached_deposit()),  // ← full user deposit forwarded
    )
}
```

When `calculate_attached_gas` returns 0, the promise to `engine_storage_deposit` on the ETH connector is created with 0 gas. NEAR's runtime fails the promise immediately. Per NEAR's promise refund semantics, the attached deposit is refunded to the **promise creator** (the Aurora Engine contract), not to the original user who called `storage_deposit`. No failure callback exists in `return_promise` to detect this and relay the refund to the user.

The two-step async structure is:
1. **Step 1**: User calls `storage_deposit` on Aurora Engine — NEAR deposit is committed and deducted from the user.
2. **Step 2**: Promise to `engine_storage_deposit` on the ETH connector must execute — but with 0 gas it fails silently.

There is no guard at Step 1 that prevents the call from proceeding when there is insufficient gas to complete Step 2.

### Impact Explanation

The user's NEAR deposit (which can be arbitrarily large, as `storage_deposit` accepts any `attached_deposit`) is permanently frozen inside the Aurora Engine contract. The user receives no refund and no error. The Aurora Engine contract has no on-chain mechanism to return these funds to the original caller. This constitutes **permanent freezing of user funds** (Critical).

### Likelihood Explanation

Any user or dApp that calls `storage_deposit` on the Aurora Engine NEP-141 facade with a prepaid gas value at or below `GAS_FOR_PROMISE_CREATION` (2 TGas) will trigger this path. While standard wallets default to 300 TGas, programmatic callers, gas-optimizing relayers, or users following incorrect documentation can easily supply insufficient gas. The silent `Ok(())` return from `return_promise` gives no indication of failure, making this hard to detect without monitoring the ETH connector's execution.

### Recommendation

Replace the silent 0-gas fallback with an explicit error, as the TODO comment already prescribes:

```rust
fn calculate_attached_gas<E: Env>(env: &E) -> Result<NearGas, ContractError> {
    let required_gas = env.used_gas().saturating_add(GAS_FOR_PROMISE_CREATION);
    if required_gas >= env.prepaid_gas() {
        Err(/* ERR_INSUFFICIENT_GAS */)
    } else {
        Ok(env.prepaid_gas() - required_gas)
    }
}
```

Propagate this `Result` through `return_promise` and all callers (`storage_deposit`, `withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_unregister`, `storage_withdraw`) so the transaction reverts before any NEAR deposit is committed to a doomed promise.
