Audit Report

## Title
Unbounded O(N) Candid Decoding Loop in `handle_construction_hash` Enables Application-Level DoS — (`rs/rosetta-api/icrc1/src/construction_api/utils.rs`)

## Summary
`handle_construction_hash` iterates every envelope in a caller-supplied `SignedTransaction` and performs full Candid decoding plus hash computation for each one, with no envelope-count guard and no HTTP body-size cap on the axum server. An unauthenticated attacker can submit a single crafted request containing arbitrarily many envelopes and drive the Rosetta process to CPU exhaustion, blocking legitimate ICRC-1 transaction submission.

## Finding Description
The loop in `handle_construction_hash` (L331–335 of `utils.rs`) iterates every envelope unconditionally:

```rust
let mut tx_hashes = HashSet::new();
for envelope in signed_transaction.envelopes {
    let transaction_hash = build_transaction_hash_from_envelope_content(&envelope.content)?;
    tx_hashes.insert(transaction_hash);
}
```

Each iteration calls `build_transaction_hash_from_envelope_content` (L194–216), which performs: (1) method-name parsing via `CanisterMethodName::new_from_envelope_content`, (2) full Candid decoding of the `arg` bytes (`TransferArg`, `ApproveArgs`, or `TransferFromArgs`), and (3) ICRC-1 transaction hash computation. The loop does not short-circuit when a second distinct hash is detected; the `tx_hashes.len() > 1` check (L338–342) fires only after the entire envelope vector is exhausted.

`SignedTransaction` carries an unbounded `Vec<Envelope>` (types.rs L45–48). The axum router in `main.rs` (L361–391) applies only a metrics layer, a tracing span layer, and a request-ID layer — no `DefaultBodyLimit` middleware is present, confirmed by grep returning no matches for `DefaultBodyLimit` anywhere in the icrc1 codebase. The `construction_hash` endpoint (endpoints.rs L67–76) performs no authentication check before deserializing and processing the request body.

## Impact Explanation
This is an application/platform-level DoS exploiting an algorithmic complexity vulnerability, not raw volumetric DDoS. A single HTTP request containing N envelopes forces N sequential Candid decodes before any response is returned. At N = 100,000 (easily fitting in a multi-MB body with no server-side cap), the Rosetta process saturates CPU on a single request. Concurrent such requests multiply the effect. The Rosetta API is the sole binary handling all ICRC-1 token operations for an exchange or integrator; sustained CPU saturation prevents legitimate `/construction/submit` calls from being processed. This matches the High ($2,000–$10,000) impact class: "Application/platform-level DoS... not based on raw volumetric DDoS" against a Rosetta financial integration component.

## Likelihood Explanation
The endpoint is unauthenticated and bound on `0.0.0.0:8080` by default. No API key, rate limiting, or body-size cap exists anywhere in the codebase. The `SignedTransaction` wire format (hex-encoded CBOR) is fully documented and trivially reproducible from the public `FromStr`/`Display` implementations. Any unprivileged network-reachable attacker can craft the payload with no special knowledge beyond the public ICRC-1 Rosetta documentation. The attack is repeatable and requires no victim interaction.

## Recommendation
1. Add a `DefaultBodyLimit` layer to the axum router (e.g., 1–2 MB) to cap total request body size.
2. Add an envelope-count guard at the top of `handle_construction_hash` — reject if `envelopes.len() > MAX_ENVELOPES`, where `MAX_ENVELOPES` reflects the legitimate 24 h / ingress-interval ceiling (≈ 288 for a 5-minute ingress interval, as noted in the types.rs comment at L43–44).
3. Exit the loop early once `tx_hashes.len() > 1` is detected, avoiding processing of remaining envelopes.

## Proof of Concept
```rust
// Build N envelopes with distinct created_at_time → distinct tx hash per envelope
let mut envelopes = vec![];
for i in 0..100_000u64 {
    let arg = Encode!(&TransferArg {
        from_subaccount: None,
        to: Account { owner: Principal::anonymous(), subaccount: None },
        fee: None,
        created_at_time: Some(i),
        memo: None,
        amount: 1u64.into(),
    }).unwrap();
    envelopes.push(Envelope {
        content: Cow::Owned(EnvelopeContent::Call {
            nonce: None, ingress_expiry: u64::MAX,
            sender: Principal::anonymous(),
            canister_id: Principal::anonymous(),
            method_name: "icrc1_transfer".to_string(),
            arg,
        }),
        sender_pubkey: None, sender_sig: None, sender_delegation: None,
    });
}
let tx = SignedTransaction { envelopes };
// POST /construction/hash with signed_transaction = tx.to_string()
// Server performs 100,000 Candid decodes before returning any response
```

Start a local ICRC-1 Rosetta instance, POST the above payload to `/construction/hash`, and observe CPU pinned at 100% for the duration of the request. Repeat concurrently to confirm denial of service to legitimate callers.
