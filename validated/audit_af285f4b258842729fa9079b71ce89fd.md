### Title
`is_empty_storage` Cache Inconsistency Causes Incorrect EIP-7610 Rejection of `CREATE`/`CREATE2` Deployments - (File: `engine/src/engine.rs`)

---

### Summary

The `is_empty_storage` function in `engine/src/engine.rs` uses `contract_storage_cache.contains_key` to short-circuit the EIP-7610 non-empty storage check. However, the `storage` function populates `contract_storage_cache` with `H256::zero()` (the default empty value) when a slot is absent. Once a zero value is cached for slot 0 of an address, `is_empty_storage` incorrectly concludes that storage is non-empty and rejects any subsequent `CREATE`/`CREATE2` to that address within the same transaction, even though the slot is actually empty.

---

### Finding Description

The `Engine` struct maintains a `contract_storage_cache` of type `RefCell<FullCache<(Address, H256), H256>>`. [1](#0-0) 

The `storage` backend method (called on every `SLOAD` opcode) populates this cache via `get_or_insert_with`. When the underlying storage slot is absent, `get_storage` returns `H256::zero()` via `unwrap_or_default()`, and that zero value is stored in the cache: [2](#0-1) [3](#0-2) 

The `is_empty_storage` backend method (called during `CREATE`/`CREATE2` opcode processing for EIP-7610) checks only whether the key **exists** in the cache, not what the cached **value** is: [4](#0-3) 

If `(address, H256::zero())` is present in the cache with value `H256::zero()` (meaning the slot is empty), `is_empty_storage` returns `false` — asserting that storage is **non-empty** — which is incorrect. The correct behavior is to return `true` (storage is empty) when the cached value is `H256::zero()`.

This is the direct analog of the reported mvkv iterator stale-value bug: two different read paths (`storage` via cache, `is_empty_storage` via `storage_has_key`) observe inconsistent views of the same key within a single transaction, and the validation path (`is_empty_storage`) acts on the wrong signal.

---

### Impact Explanation

EIP-7610 is enforced by SputnikVM calling `is_empty_storage` before allowing a `CREATE`/`CREATE2` to proceed. If `is_empty_storage` incorrectly returns `false`, the opcode is rejected with `CreateCollision`. Any EVM transaction that:

1. Executes `SLOAD(target_addr, 0)` where slot 0 is empty (caching `H256::zero()`), and
2. Subsequently executes `CREATE2` targeting `target_addr` in the same transaction

will have the deployment silently rejected. If ETH or ERC-20 tokens are pre-funded to `target_addr` (a counterfactual CREATE2 wallet address), those funds become inaccessible until a separate transaction deploys the contract without the preceding `SLOAD`. This constitutes **temporary freezing of funds**.

---

### Likelihood Explanation

Medium-low. The trigger requires `SLOAD(addr, 0)` followed by `CREATE2(addr)` in the same transaction where slot 0 is empty. This pattern arises naturally in:

- Factory contracts that inspect a target address's state before deploying to it.
- Counterfactual wallet deployment flows where the deployer contract reads storage of the target before creating it.
- Any protocol using CREATE2 with pre-funding where the deployment logic includes a storage read of the target address.

An unprivileged EVM user can trigger this by submitting a crafted transaction through the standard `submit` entrypoint.

---

### Recommendation

Change `is_empty_storage` to check the **value** in the cache, not merely the key's presence. If the cached value is `H256::zero()`, the slot is empty and the function should return `true`:

```rust
fn is_empty_storage(&self, address: H160) -> bool {
    let index = H256::zero();
    let address = Address::new(address);
    // Check cache value, not just key presence
    if let Some(cached) = self.contract_storage_cache.borrow().get(&(address, index)) {
        return cached == &H256::zero();
    }
    let generation = *self
        .generation_cache
        .borrow_mut()
        .entry(address)
        .or_insert_with(|| get_generation(&self.io, &address));
    !storage_has_key(&self.io, &address, &index, generation)
}
```

This ensures that a cached zero value is correctly interpreted as empty storage, consistent with how `storage_has_key` would answer the same question.

---

### Proof of Concept

Deploy the following Solidity contract on Aurora and call `trigger(target_salt)` where `target_addr` has no storage but has been pre-funded with ETH:

```solidity
contract PoC {
    bytes32 constant INIT = hex"00"; // minimal bytecode

    function trigger(bytes32 salt) external {
        address target = address(uint160(uint256(keccak256(
            abi.encodePacked(bytes1(0xff), address(this), salt, keccak256(INIT))
        ))));

        // Step 1: SLOAD slot 0 of target — returns 0, caches H256::zero()
        assembly { pop(sload(target)) }

        // Step 2: CREATE2 to target — is_empty_storage(target) incorrectly
        // returns false due to cached zero, causing CreateCollision
        assembly {
            let addr := create2(0, add(INIT, 0x20), mload(INIT), salt)
            if iszero(addr) { revert(0, 0) }
        }
    }
}
```

1. Pre-fund `target_addr` with ETH.
2. Call `trigger(salt)` — the `SLOAD` caches `H256::zero()` for `(target_addr, 0)`.
3. The `CREATE2` calls `is_empty_storage(target_addr)` → finds key in cache → returns `false` → deployment rejected.
4. ETH in `target_addr` is inaccessible until a separate transaction deploys without the preceding `SLOAD`. [5](#0-4) [6](#0-5)
