### Title
Unsafe Narrowing Cast of `gas_per_pubdata` to `u64` Silently Truncates Attacker-Controlled L1 Transaction Field — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

In the L1 transaction resource-setup path, the `gas_per_pubdata` field read from the transaction is cast directly to `u64` with `as u64` — a truncating, non-panicking Rust cast — before being used to compute `native_per_pubdata`. If `gas_per_pubdata` is a `u128` (as is standard for ZKsync transaction fields derived from Solidity `uint256`), any value above `u64::MAX` is silently truncated to its lower 64 bits. The subsequent `checked_mul` only guards against multiplication overflow on the already-truncated value; it does not detect the prior truncation. This is the direct analog of the Nibbl `uint32` cast vulnerability: a user-controlled input is narrowed without a bounds check, producing a silently incorrect intermediate value that propagates through resource accounting.

---

### Finding Description

In `process_l1_transaction.rs`, the function that computes resource and fee information for L1→L2 transactions contains:

```rust
let native_per_pubdata = (gas_per_pubdata as u64)   // ← truncating cast, no overflow check
    .checked_mul(native_per_gas)
    .unwrap_or_else(|| {
        system_log!(system,
            "Native per pubdata calculation for L1 tx overflows, using saturated arithmetic instead");
        u64::MAX
    });
``` [1](#0-0) 

The `as u64` cast in Rust is defined to truncate: for a `u128` value `v`, `v as u64` yields `v & 0xFFFF_FFFF_FFFF_FFFF`. No panic, no error, no flag. The `checked_mul` that follows only detects overflow in the multiplication of the already-truncated value.

By contrast, the L2 transaction path uses a proper checked conversion:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [2](#0-1) 

The L1 path has no equivalent guard. The `gas_per_pubdata` field originates from the ABI-encoded ZKsync transaction, where `gasPerPubdataByteLimit` is a Solidity `uint256`, stored in Rust as `u128` (confirmed by the explicit `as u64` cast — a no-op cast would not be written for a `u64` field). The field is read from the transaction and passed directly to the resource calculation without any upper-bound validation. [3](#0-2) 

---

### Impact Explanation

`native_per_pubdata` is the per-byte native resource cost for pubdata. It is used in `get_resources_to_charge_for_pubdata` to compute the total native charge for all pubdata the transaction writes:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [4](#0-3) 

If `native_per_pubdata` is artificially small due to truncation, the transaction is charged far less native resource per pubdata byte than it should be. Within the same native resource budget, the transaction can write proportionally more pubdata to L1. For example, if the intended `gas_per_pubdata` is `2^64 + 1` but truncates to `1`, the per-byte cost drops from `(2^64 + 1) × native_per_gas` to `1 × native_per_gas` — a factor of `2^64` reduction. The attacker effectively publishes unbounded pubdata at near-zero cost, forcing the operator to bear the L1 data-availability cost for data the attacker did not pay for. This is a **resource accounting bug / public funds-loss path** for the operator.

---

### Likelihood Explanation

L1→L2 transactions are submitted permissionlessly from L1. Any externally-owned account can craft a transaction with an arbitrary `gas_per_pubdata` value. No privileged role, governance action, or oracle manipulation is required. The attacker only needs to set `gas_per_pubdata` to any value of the form `k × 2^64 + x` where `x` is small, and the truncation silently produces `x`. The `L1ResourcesPolicy` saturates rather than rejects on downstream overflow, so the transaction is processed rather than dropped. [5](#0-4) 

---

### Recommendation

Replace the truncating `as u64` cast with a checked conversion that rejects or saturates explicitly:

```rust
// Option A: reject if out of range (mirrors L2 path)
let gas_per_pubdata_u64 = u64::try_from(gas_per_pubdata)
    .unwrap_or(u64::MAX); // or log + saturate, consistent with L1 policy

// Option B: use the same helper used elsewhere
let gas_per_pubdata_u64 = gas_per_pubdata.min(u64::MAX as u128) as u64;
```

The L2 path's `u256_try_to_u64` pattern should be adopted here for consistency.

---

### Proof of Concept

1. Attacker submits an L1→L2 transaction with `gas_per_pubdata = 2^64 + 1` (a valid `u128` value, well within the Solidity `uint256` range).
2. In `process_l1_transaction.rs` line 481, `gas_per_pubdata as u64` evaluates to `1` (the upper 64 bits are silently discarded).
3. `native_per_pubdata = 1 * native_per_gas` — a value `2^64` times smaller than intended.
4. The transaction executes and writes large amounts of pubdata (e.g., many storage writes). Each pubdata byte costs only `1 * native_per_gas` native resources instead of `(2^64 + 1) * native_per_gas`.
5. The operator pays the L1 data-availability cost for all published pubdata, while the attacker paid only a negligible fraction of the correct amount. [1](#0-0)
