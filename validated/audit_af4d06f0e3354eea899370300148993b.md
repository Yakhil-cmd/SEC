### Title
`BatchOutput::hash()` Implementation Includes Undocumented Fields, Causing Batch Public Input Hash Mismatch with Settlement Layer Specification — (`File: basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/public_input.rs`)

---

### Summary

`BatchOutput::hash()` hashes three fields — `number_of_layer_2_txs`, `interop_roots_rolling_hash`, and `settlement_layer_chain_id` — that are entirely absent from the documented `batch_output_hash` specification in `docs/l1_integration.md`. Any settlement layer contract implementing the hash according to the public specification will compute a different digest than the prover, causing all batch proof verifications to fail.

---

### Finding Description

The `BatchOutput` struct in `public_input.rs` has 12 fields. Its `hash()` method serializes all 12 of them into a keccak256 digest:

```
chain_id (32 B) | first_block_timestamp (8 B) | last_block_timestamp (8 B)
| da_commitment_scheme (32 B padded) | pubdata_commitment (32 B)
| number_of_layer_1_txs (32 B) | number_of_layer_2_txs (32 B)   ← undocumented
| priority_operations_hash (32 B) | l2_logs_tree_root (32 B)
| upgrade_tx_hash (32 B) | interop_roots_rolling_hash (32 B)     ← undocumented
| settlement_layer_chain_id (32 B)                               ← undocumented
``` [1](#0-0) 

The public specification in `docs/l1_integration.md` defines `batch_output_hash` as keccak256 of only nine fields:

```
used_chain_id | first_block_timestamp | last_block_timestamp
| da_commitment_scheme | da_commitment | number_of_layer_1_txs
| rolling_l1_txs_keccak256 | l2_logs_tree_root | upgrade_tx_hash
``` [2](#0-1) 

Three fields present in the implementation are completely absent from the specification:

| Field | Present in code | Present in docs |
|---|---|---|
| `number_of_layer_2_txs` | ✓ | ✗ |
| `interop_roots_rolling_hash` | ✓ | ✗ |
| `settlement_layer_chain_id` | ✓ | ✗ |

Additionally, `first_block_timestamp` and `last_block_timestamp` are encoded as 8-byte `u64` values in the implementation, not as 32-byte words as implied by the documentation, introducing a further encoding inconsistency. [3](#0-2) 

The `batch_output` hash feeds directly into `BatchPublicInput::hash()`, which is the final public input committed to by the ZK proof. [4](#0-3) 

This public input hash is what the settlement layer contract must recompute and verify against the proof. Both the single-block and multi-block batch proving paths produce this hash: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

The settlement layer contract (a Solidity contract on L1) must independently recompute `batch_output_hash` to verify the ZK proof's public input. It will implement this hash according to the documented specification. Because the implementation includes three extra fields and uses non-uniform field widths not described in the spec, the prover-computed hash and the settlement-layer-computed hash will always diverge. Every batch proof submitted to the settlement layer will fail verification. No batch can ever be finalized, permanently halting L2→L1 state transitions and message delivery. This is a **valid-execution unprovability** / **forward-proving divergence** issue with critical impact on the entire rollup liveness.

---

### Likelihood Explanation

The mismatch is structural and deterministic — it affects every single batch proof regardless of transaction content. Any settlement layer contract faithfully implementing the documented spec will reject all proofs. The root cause is in production proving code, not in a test or configuration path.

---

### Recommendation

Synchronize the implementation and the specification. Either:

1. **Update `docs/l1_integration.md`** to include `number_of_layer_2_txs`, `interop_roots_rolling_hash`, and `settlement_layer_chain_id` in the `batch_output_hash` field list, and clarify that `first_block_timestamp`/`last_block_timestamp` are encoded as 8-byte big-endian values; **or**

2. **Remove the three undocumented fields from `BatchOutput::hash()`** if they were added without updating the spec and the settlement layer contract does not include them.

Additionally, standardize all timestamp fields to 32-byte encoding to match the uniform word-size encoding implied by the documentation and used by all other fields.

---

### Proof of Concept

The divergence is directly observable by comparing the two field lists. A Solidity settlement layer contract implementing the documented spec would compute:

```solidity
// Per docs/l1_integration.md
bytes32 batchOutputHash = keccak256(abi.encodePacked(
    chainId,           // 32 bytes
    firstTimestamp,    // 32 bytes (docs imply word-aligned)
    lastTimestamp,     // 32 bytes
    daScheme,          // 32 bytes
    daCommitment,      // 32 bytes
    numL1Txs,          // 32 bytes
    priorityOpsHash,   // 32 bytes
    l2LogsRoot,        // 32 bytes
    upgradeTxHash      // 32 bytes
));
```

The Rust prover computes (per `BatchOutput::hash()`):

```rust
// 32 + 8 + 8 + 32 + 32 + 32 + 32 + 32 + 32 + 32 + 32 + 32 = 356 bytes total
chain_id(32) | first_ts(8) | last_ts(8) | da_scheme(32) | pubdata(32)
| num_l1_txs(32) | num_l2_txs(32)  // extra
| priority_hash(32) | l2_logs_root(32) | upgrade_hash(32)
| interop_rolling_hash(32)           // extra
| settlement_layer_chain_id(32)      // extra
```

The two byte strings are structurally different in both length and content, so `keccak256` will always produce different digests, and proof verification will always fail.
