### Title
Uninitialized Engine State Allows Attacker to Seize Ownership and Permanently Freeze Bridged Funds - (File: `engine/src/contract_methods/admin.rs`)

### Summary
The Aurora Engine's `new()` initialization function performs no caller authentication. If the contract is deployed without atomically calling `new()` in the same batch transaction, any unprivileged NEAR account can front-run the initialization, set themselves as `owner_id`, then use owner-gated upgrade functions to replace the engine with arbitrary WASM, permanently freezing all bridged user funds.

### Finding Description
The `new()` function in `engine/src/contract_methods/admin.rs` guards only against re-initialization (checking whether state already exists), but performs **no check on who the caller is**:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // ... no predecessor_account_id check ...
    let state: EngineState = args.into();   // owner_id comes from attacker-supplied args
    state::set_state(&mut io, &state)?;
    Ok(())
}
``` [1](#0-0) 

The deployment flow in the workspace builder issues `deploy` and `new` as **separate async transactions**, not as a single atomic NEAR batch:

```rust
let contract = account.deploy(&self.code...).await?;
// ...
engine.new(self.chain_id, self.owner_id, self.upgrade_delay_blocks).transact().await...
``` [2](#0-1) 

Between these two transactions there is a window in which the contract exists on-chain with no state. Any NEAR account can call `new()` during this window, supplying an attacker-controlled `owner_id`.

Once the attacker holds `owner_id`, they can call `stage_upgrade()` (owner-only) to stage arbitrary WASM: [3](#0-2) 

After `upgrade_delay_blocks` blocks, `deploy_upgrade()` — which has **no caller check** and is intentionally callable by anyone — deploys the staged code: [4](#0-3) 

The attacker-supplied WASM replaces the Aurora Engine contract entirely, destroying all EVM state and making all bridged assets permanently inaccessible.

### Impact Explanation
All ETH and ERC-20 tokens bridged into Aurora are held as accounting state within the Aurora Engine NEAR contract. Replacing the engine with arbitrary (e.g., empty or destructive) WASM permanently destroys the EVM state trie and the NEP-141 connector accounting. Users cannot withdraw their bridged funds. This constitutes **permanent freezing of funds** — a Critical impact.

### Likelihood Explanation
The attack requires only a single-block window between the `deploy` transaction and the `new()` transaction. NEAR block times are ~1 second. Any NEAR account (no special privileges needed) can monitor the mempool or chain for a freshly deployed Aurora Engine contract with no state and immediately call `new()`. The attack is straightforward, requires no capital, and is fully reachable by an unprivileged external actor. Likelihood is **High** if deployment is not done atomically.

### Recommendation
1. **Atomic deployment**: Issue `DeployContract` and the `new()` `FunctionCall` as a single NEAR batch action so no intermediate state exists on-chain.
2. **Caller guard in `new()`**: Add a check that `env.predecessor_account_id() == env.current_account_id()` (i.e., only a self-call or the deploying account can initialize), analogous to the EasyTrack recommendation of adding a constructor that pre-initializes storage.
3. **Deployment scripts**: Audit all deployment scripts to ensure they use batch transactions for deploy + init, never separate transactions.

### Proof of Concept

1. Attacker monitors NEAR for a transaction deploying the Aurora Engine WASM to account `aurora`.
2. In the next block (before the legitimate `new()` call), attacker submits:
   ```
   aurora.new({chain_id: ..., owner_id: "attacker.near", upgrade_delay_blocks: 1})
   ```
   This succeeds because `get_state(&io).is_ok()` returns `false` — no state exists yet. [5](#0-4) 
3. Attacker calls `aurora.stage_upgrade(malicious_wasm_bytes)` — succeeds because `state.owner_id == "attacker.near"`. [6](#0-5) 
4. After 1 block, attacker (or anyone) calls `aurora.deploy_upgrade()` — no caller check, deploys malicious WASM. [7](#0-6) 
5. Aurora Engine is replaced with attacker-controlled code. All bridged user funds are permanently frozen.

pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

#[named]
pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let delay_block_height = env.block_height() + state.upgrade_delay_blocks;
        require_owner_only(&state, &env.predecessor_account_id())?;
        io.read_input_and_store(&storage::bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.write_storage(
            &storage::bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY),
            &delay_block_height.to_le_bytes(),
        );
        Ok(())
    })
}

    pub extern "C" fn deploy_upgrade() {
        // This function is intentionally not implemented in `contract_methods`
        // because it only makes sense in the context of the NEAR runtime.
        let mut io = Runtime;
        let state = state::get_state(&io).sdk_unwrap();
        require_running(&state)
            .map_err(ContractError::msg)
            .sdk_unwrap();
        let index = internal_get_upgrade_index();
        if io.block_height() <= index {
            sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
        }
        Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
    }
