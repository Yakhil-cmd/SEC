### Title
Unvalidated `refund_recipient` in L1→L2 Transactions Allows Permanent Token Burn from Treasury - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

L1→L2 priority transactions carry a user-controlled `refund_recipient` field (`transaction.reserved[1]`). The ZKsync OS bootloader reads this field without any validation and mints the unused-gas refund directly to whatever address is specified. A user can set this to `address(0xdead)` or any other permanently inaccessible address, causing the refund portion of the deposit to be minted from the treasury to a burn address, permanently removing those tokens from circulation.

---

### Finding Description

In `process_l1_transaction`, after computing the unused-gas refund amount (`to_refund_recipient`), the bootloader reads the refund recipient address directly from the transaction's `reserved[1]` field with no validation:

```rust
let refund_recipient = u256_to_b160_checked(transaction.reserved[1].read());
mint_base_token::<S, Config>(
    system, system_functions, memories.reborrow(),
    &to_refund_recipient,
    &refund_recipient,   // ← user-controlled, never validated
    ...
)
``` [1](#0-0) 

The `validate_structure` function in `AbiEncodedTransaction` explicitly leaves this field unvalidated with a `// TODO: validate address?` comment:

```rust
// reserved[1] = refund recipient for l1 to l2 and upgrade txs
match tx_type {
    Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
        // TODO: validate address?
    }
    ...
}
``` [2](#0-1) 

The `refund_recipient` field is a first-class user-supplied field in `ZKsyncL1Tx`: [3](#0-2) 

The refund amount is computed as `prepaid_fee - pay_to_operator` (unused gas × gas_price) on success, or `total_deposited - pay_to_operator` on revert: [4](#0-3) 

`mint_base_token` transfers tokens from the treasury (`BASE_TOKEN_HOLDER_ADDRESS`) to the specified address: [5](#0-4) 

---

### Impact Explanation

When a user sets `refund_recipient` to `address(0xdead)` or any other inaccessible address:

1. The treasury (`BASE_TOKEN_HOLDER_ADDRESS`) is debited by `to_refund_recipient` tokens.
2. Those tokens are credited to the burn address and are permanently inaccessible.
3. The treasury balance is permanently reduced with no recovery path.

A user can maximize the burned amount by setting a very high `gas_limit` relative to actual execution cost. On a reverted L1→L2 transaction, the refund is `total_deposited - pay_to_operator`, which can be nearly the entire deposit. This is a direct, permanent loss of base tokens from the protocol treasury.

---

### Likelihood Explanation

Any user submitting an L1→L2 priority transaction can exploit this. The `refund_recipient` field is set on L1 before the transaction is submitted to the priority queue. The ZKsync OS bootloader cannot reject L1 transactions (doing so would halt the chain), so there is no on-chain defense once the transaction is in the queue. The attack requires no special privileges, no leaked keys, and no external oracle manipulation. The user bears the cost of their own refund being burned, but the treasury suffers the same loss.

---

### Recommendation

1. In `validate_structure` (or in `process_l1_transaction` before minting), reject or redirect the refund if `refund_recipient` is `B160::ZERO`, `0xdead...`, or any other known burn address.
2. Alternatively, if `refund_recipient` is zero, fall back to `transaction.from()` as the refund destination.
3. Resolve the `// TODO: validate address?` comment in `abi_encoded/mod.rs` with an explicit address check.

---

### Proof of Concept

1. Submit an L1→L2 priority transaction with:
   - `gas_limit = 1_000_000` (large)
   - `gas_price = 1_000`
   - `to_mint = gas_limit * gas_price` (= 1,000,000,000)
   - `refund_recipient = address(0x000000000000000000000000000000000000dEaD)`
   - Simple calldata that uses only ~21,000 gas

2. The bootloader computes:
   - `pay_to_operator = 21_000 * 1_000 = 21_000_000`
   - `to_refund_recipient = 1_000_000_000 - 21_000_000 = 979_000_000`

3. `mint_base_token` is called with `to = 0xdead`, transferring 979,000,000 tokens from the treasury to the burn address.

4. The treasury loses 979,000,000 tokens permanently. The user also loses their refund, but the treasury loss is the protocol-level impact. [6](#0-5)
