Audit Report

## Title
Unguarded `Vec<EnvelopePair>` index access causes panic in `construction_parse` and `construction_hash` — (`rs/rosetta-api/icp/src/request_handler/construction_parse.rs`, `rs/rosetta-api/icp/src/request_handler/construction_hash.rs`)

## Summary
The ICP Rosetta API deserializes attacker-supplied CBOR into a `SignedTransaction` without validating that each `Vec<EnvelopePair>` is non-empty. Both `/construction/parse` and `/construction/hash` then unconditionally index into that vector (`updates[0]`, `envelope_pairs[0]`), causing an index-out-of-bounds Rust panic on any request carrying an empty envelope list. An unauthenticated attacker can trigger this with a single crafted HTTP POST, causing the Actix-web worker to panic and the request to fail with an unhandled 500; repeated requests constitute a persistent application-level DoS against the Rosetta service.

## Finding Description
`pub type Request = (RequestType, Vec<EnvelopePair>)` carries no minimum-length invariant on the `Vec`. `ParsedTransaction::try_from` in `models.rs` (lines 180–197) performs only CBOR deserialization — it does not check that any `Vec<EnvelopePair>` is non-empty before returning `ParsedTransaction::Signed`. In `construction_parse.rs` line 42, the code immediately maps over `signed_transaction.requests` and accesses `updates[0]` inside the closure without an emptiness guard. A `SignedTransaction { requests: vec![(RequestType::Send, vec![])] }` is structurally valid CBOR and passes deserialization, but the subsequent `updates[0]` panics. The identical pattern exists in `construction_hash.rs` line 27 (`&envelope_pairs[0].update`), where `.find(|(rt, _)| rt.is_transfer())` checks only the `RequestType`, not whether the paired `Vec<EnvelopePair>` is non-empty. No existing guard in either code path prevents the panic.

## Impact Explanation
The panic propagates through the Actix-web handler. While Actix-web 4.x catches panics per-worker and returns a 500, the worker is torn down and restarted on each panic, making the endpoint non-functional for the duration of the restart cycle. Repeated unauthenticated requests sustain this condition indefinitely, constituting an application-level DoS against the Rosetta API — a financial integration component explicitly in scope. This matches: **High ($2,000–$10,000) — Significant Rosetta/boundary-API security impact with concrete user or protocol harm.**

## Likelihood Explanation
The `/construction/parse` and `/construction/hash` endpoints are public and unauthenticated. The malicious payload requires only serializing a `SignedTransaction` with one `(RequestType::Send, vec![])` entry to CBOR and hex-encoding it — no key material, privileged access, or network-level capability is needed. The attack is trivially repeatable and non-volumetric.

## Recommendation
Add an explicit non-empty guard before the index access in `construction_parse.rs`:

```rust
.map(|(request_type, updates)| {
    if updates.is_empty() {
        return Err(ApiError::invalid_request(
            "EnvelopePair list must not be empty",
        ));
    }
    match updates[0].update.content.clone() {
        HttpCallContent::Call { update } => Ok((request_type.clone(), update)),
    }
})
.collect::<Result<Vec<_>, _>>()?
```

Apply the same guard in `construction_hash.rs` before accessing `envelope_pairs[0]`:

```rust
if envelope_pairs.is_empty() {
    return Err(ApiError::invalid_request("EnvelopePair list must not be empty"));
}
TransactionIdentifier::try_from_envelope(request_type.clone(), &envelope_pairs[0].update)
```

Optionally enforce the invariant at the type level or in `ParsedTransaction::try_from` to prevent the same pattern from recurring elsewhere.

## Proof of Concept
```rust
use serde_cbor;
use hex;

// Construct the malicious payload
let malicious = SignedTransaction {
    requests: vec![(RequestType::Send, vec![])],
};
let cbor_bytes = serde_cbor::to_vec(&malicious).unwrap();
let hex_payload = hex::encode(cbor_bytes);

// POST /construction/parse
// Body: { "network_identifier": {...}, "signed": true, "transaction": "<hex_payload>" }
// -> panics at updates[0] in construction_parse.rs:42, Actix worker crashes and restarts

// POST /construction/hash
// Body: { "network_identifier": {...}, "signed_transaction": "<hex_payload>" }
// -> panics at envelope_pairs[0] in construction_hash.rs:27
```

A deterministic unit test can reproduce this by calling `RosettaRequestHandler::construction_parse` directly with a `ConstructionParseRequest` whose `transaction` field encodes `SignedTransaction { requests: vec![(RequestType::Send, vec![])] }` and asserting that the call panics (or, after the fix, returns an `Err`).
