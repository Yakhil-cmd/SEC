### Title
Truncating Integer Division in `native_per_pubdata` Computation Causes Systematic Pubdata Underpayment — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio is computed with truncating (floor) integer division while the analogous `native_per_gas` ratio uses ceiling division. The truncation silently underestimates the native cost per pubdata byte, causing the post-execution pubdata sufficiency check to pass when it should fail, and allowing users to publish pubdata without paying the operator's full intended price.

---

### Finding Description

In `validation_impl.rs` the two key ratios are computed as follows:

```rust
// native_per_gas — uses CEILING division (rounds UP)
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [1](#0-0) 

```rust
// native_per_pubdata — uses FLOOR division (rounds DOWN, wrapping_div == truncating div)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [2](#0-1) 

The same asymmetry is present in the public API helper that mirrors bootloader behaviour:

```rust
// native_per_gas = ceil(gas_price / native_price)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(())?;
// native_per_pubdata = pubdata_price / native_price  ← floor, no comment on rounding
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

`native_per_pubdata` is then used in two places that gate whether a transaction is allowed to proceed:

1. **Intrinsic pubdata overhead** charged upfront in `create_resources_for_tx`:
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   ``` [4](#0-3) 

2. **Post-execution pubdata check** in `get_resources_to_charge_for_pubdata` / `check_enough_resources_for_pubdata`:
   ```rust
   let native = current_pubdata_spent
       .checked_mul(native_per_pubdata)
       .ok_or(out_of_native_resources!())?;
   ``` [5](#0-4) 

Because `native_per_pubdata` is floored, the native cost charged for every pubdata byte is strictly less than `pubdata_price / native_price` whenever `pubdata_price % native_price ≠ 0`. The check `check_enough_resources_for_pubdata` therefore passes with fewer native resources than the operator's pricing model requires. [6](#0-5) 

---

### Impact Explanation

**Resource accounting bug / valid-execution underpayment path.**

A user who crafts a transaction that generates the maximum allowed pubdata pays `floor(pubdata_price / native_price)` native per byte instead of the exact ratio. The per-byte shortfall is up to `(native_price − 1) / native_price`, which is less than 1 native unit per byte. Over a large pubdata-heavy transaction (e.g. 100 000 pubdata bytes with `native_price = 3`, `pubdata_price = 10`):

- Intended charge: `⌈10/3⌉ × 100 000 = 400 000` native  
- Actual charge: `⌊10/3⌋ × 100 000 = 300 000` native  
- Shortfall: **100 000 native (25 %)**

The shortfall scales linearly with pubdata volume and with how far `pubdata_price` is from the nearest multiple of `native_price`. An attacker who maximises pubdata output per transaction (e.g. by writing many distinct storage slots) can systematically underpay the operator for data-availability costs.

---

### Likelihood Explanation

The condition `pubdata_price % native_price ≠ 0` holds for almost any real-world operator configuration where `native_price` and `pubdata_price` are independently set market prices. Every L2 transaction that writes storage or emits events is affected. No special privilege is required; any externally-owned account submitting a normal EIP-1559 transaction triggers the path.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, matching the rounding direction already used for `native_per_gas`:

```rust
// Before (truncates, undercharges pubdata)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling, consistent with native_per_gas)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427 to keep the off-chain validation helper consistent with the bootloader. [7](#0-6) 

---

### Proof of Concept

**Concrete numbers** (`pubdata_price = 10`, `native_price = 3`):

| Step | Value |
|---|---|
| `native_per_pubdata` (floor) | `10 / 3 = 3` |
| `native_per_pubdata` (correct ceiling) | `⌈10/3⌉ = 4` |
| Pubdata bytes in tx | 90 000 |
| Native charged | `3 × 90 000 = 270 000` |
| Native that should be charged | `4 × 90 000 = 360 000` |
| Underpayment | **90 000 native (25 %)** |

The user submits a transaction that writes 90 000 pubdata bytes (achievable by writing ~2 800 distinct storage slots). The post-execution check `check_enough_resources_for_pubdata` passes because the user holds exactly 270 000 native units — enough at the truncated rate but 90 000 short of the operator's intended price. The transaction is accepted and the operator absorbs the shortfall.
