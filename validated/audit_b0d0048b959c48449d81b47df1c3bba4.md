### Title
`ExitToEthereum` Precompile Ignores `WithdrawSerializeType` for ERC-20 Withdrawals, Causing Permanent Token Loss When Borsh Serialization Is Configured — (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile correctly reads the configured `WithdrawSerializeType` (JSON or Borsh) when serializing ETH base-token withdrawal arguments, but unconditionally uses JSON serialization for ERC-20 token withdrawals. When the connector is configured to use Borsh serialization — which is the **default** when the key is absent from storage — every ERC-20 withdrawal via `ExitToEthereum` sends JSON-formatted arguments to a connector that expects Borsh, causing the cross-contract `withdraw` call to fail. Because ERC-20 tokens are burned inside the EVM state before the promise is dispatched, and a failed promise does not roll back EVM state on Aurora, the burned tokens are permanently lost.

---

### Finding Description

In `ExitToEthereum::run`, the flag byte selects between two withdrawal paths:

**Flag `0x0` — ETH base token (correct):**
The code reads `WithdrawSerializeType` from storage and dispatches to either `json_args` or `borsh_args` accordingly.

```rust
// engine-precompiles/src/native.rs  lines 897-907
let serialize_fn = match get_withdraw_serialize_type(&self.io)? {
    WithdrawSerializeType::Json => json_args,
    WithdrawSerializeType::Borsh => borsh_args,
};
...
serialize_fn(recipient_address, context.apparent_value)?
``` [1](#0-0) 

**Flag `0x1` — ERC-20 token (broken):**
The code ignores `WithdrawSerializeType` entirely and always produces a JSON string.

```rust
// engine-precompiles/src/native.rs  lines 949-958
format!(
    r#"{{"amount": "{}", "recipient": "{}"}}"#,
    amount.as_u128(),
    recipient_in_hex
)
.into_bytes()
``` [2](#0-1) 

The helper `get_withdraw_serialize_type` defaults to **Borsh** when the storage key is absent:

```rust
// engine-precompiles/src/native.rs  lines 322-331
fn get_withdraw_serialize_type<I: IO>(io: &I) -> Result<WithdrawSerializeType, ExitError> {
    io.read_storage(...)
      .map_or(Ok(WithdrawSerializeType::Borsh), |value| { ... })
}
``` [3](#0-2) 

This is structurally identical to the reported ZetaChain bug: one branch (`ExitToEthereum` ETH path / `AddToOutTxTracker`) correctly handles both variants, while the sibling branch (`ExitToEthereum` ERC-20 path / `AddToInTxTracker`) silently hard-codes only one variant, causing the other to always fail.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The ERC-20 `withdraw` flow burns tokens inside the EVM execution before the precompile returns. The precompile then emits a NEAR promise log (a `PromiseArgs::Create` targeting the NEP-141 contract's `withdraw` method). If the NEP-141 contract is configured to expect Borsh-encoded arguments but receives JSON, it will fail to deserialize and the cross-contract call will revert. Because NEAR promise failures do not roll back the already-committed EVM state, the ERC-20 tokens remain burned on Aurora while no corresponding unlock occurs on the Ethereum side — a permanent, irrecoverable loss for the user. [4](#0-3) 

---

### Likelihood Explanation

- The default `WithdrawSerializeType` is **Borsh** (returned by `get_withdraw_serialize_type` when the storage key is unset), so the vulnerable code path is active in any deployment that has not explicitly overridden the serialization type to JSON.
- The entry point is the standard ERC-20 `withdraw` function callable by **any unprivileged user** holding a registered Aurora ERC-20 token.
- No special permissions, governance capture, or admin compromise are required; a normal token holder triggers the bug simply by attempting a bridge withdrawal.

---

### Recommendation

In the ERC-20 branch (flag `0x1`) of `ExitToEthereum::run`, read `WithdrawSerializeType` from storage and dispatch to the appropriate serialization function, mirroring the ETH base-token branch:

```rust
// flag 0x1 — ERC-20 path (fixed)
let serialize_fn = match get_withdraw_serialize_type(&self.io)? {
    WithdrawSerializeType::Json => json_args,
    WithdrawSerializeType::Borsh => borsh_args,
};
let serialized_args = serialize_fn(recipient_address, amount)?;
```

---

### Proof of Concept

1. Deploy Aurora Engine with the default configuration (no explicit `WithdrawSerializeType` set → defaults to Borsh).
2. Register a NEP-141 token and deploy its ERC-20 mirror on Aurora.
3. As an unprivileged user, call the ERC-20 contract's `withdraw(amount, eth_recipient)` function.
4. The ERC-20 contract burns `amount` tokens from the caller's balance and calls the `ExitToEthereum` precompile with flag `0x1`.
5. The precompile serializes the withdrawal arguments as JSON (`{"amount": "...", "recipient": "..."}`).
6. A NEAR promise is dispatched to the NEP-141 contract's `withdraw` method with those JSON bytes.
7. The NEP-141 contract attempts to Borsh-deserialize the JSON payload → deserialization fails → promise reverts.
8. The EVM state (with burned tokens) is already committed; no rollback occurs.
9. The user's ERC-20 tokens are permanently destroyed with no corresponding Ethereum-side unlock. [5](#0-4)
