### Title
ETH Sent Directly to the ckETH Minter's Ethereum Address Is Permanently Lost With No Recovery Mechanism - (File: rs/ethereum/cketh/minter/src/state.rs)

### Summary
The ckETH minter canister exposes its Ethereum address via the public `minter_address()` endpoint. Any ETH sent directly to that address — bypassing the required helper smart contract — is received by the minter's EOA but is never minted as ckETH and cannot be recovered. The minter's own code acknowledges this gap but provides no on-chain remedy.

### Finding Description
The ckETH minter controls an Ethereum EOA via threshold ECDSA. The intended deposit flow requires the user to call `depositEth` on the helper smart contract (`EthDepositHelper.sol` / `CkDeposit`), which forwards the ETH to the minter address **and** emits a `ReceivedEth` / `ReceivedEthOrErc20` event. The minter scrapes those events on a timer and mints ckETH to the corresponding IC principal.

The minter's Ethereum address is publicly queryable by any caller via the `minter_address()` update call:

```
// IMPORTANT: Do NOT send ETH to this address directly. Use the helper smart contract instead so that the minter
// knows to which IC principal the funds should be deposited.
minter_address : () -> (text);
``` [1](#0-0) 

Because the minter address is a plain EOA, it can receive ETH from any Ethereum transaction without any event being emitted. The minter's internal `EthBalance` tracking explicitly acknowledges this gap:

```rust
/// Note that invalid deposits are not accounted for and so this value
/// might be less than what is displayed by Etherscan
/// or retrieved by the JSON-RPC call `eth_getBalance`.
/// Also, some transactions may have gone directly to the minter's address
/// without going via the helper smart contract.
eth_balance: Wei,
``` [2](#0-1) 

The minter canister exposes no endpoint to:
- detect that ETH arrived outside the helper-contract flow,
- mint ckETH retroactively for such a deposit, or
- refund the ETH to the original sender.

The `withdraw_eth` endpoint only burns ckETH from the caller's ledger balance and then sends ETH out; it cannot be used to recover ETH that was never credited as ckETH. [3](#0-2) 

The helper contracts themselves have no `receive()` fallback and forward ETH directly to the minter address:

```solidity
function depositEth(bytes32 principal, bytes32 subaccount) public payable {
    emit ReceivedEthOrErc20(ZERO_ADDRESS, msg.sender, msg.value, principal, subaccount);
    minterAddress.transfer(msg.value);
}
``` [4](#0-3) 

So the ETH arrives at the minter's EOA, the minter's log-scraping loop sees no matching event, no ckETH is minted, and the ETH silently accumulates in the minter's Ethereum balance with no attribution and no recovery path.

### Impact Explanation
Any ETH sent directly to the minter's Ethereum address is permanently lost from the depositor's perspective:
- No ckETH is minted on the IC ledger.
- The minter's `eth_balance` field (used for accounting and displayed on the dashboard) under-counts the actual on-chain balance.
- The surplus ETH is silently absorbed into the minter's pool and may eventually be consumed as gas fees for other users' withdrawals, effectively redistributing the victim's funds to other withdrawers.
- There is no recovery endpoint, no refund path, and no governance proposal mechanism to credit the affected user.

This is a **chain-fusion ledger conservation bug**: real ETH enters the system but no corresponding ckETH is ever issued, breaking the 1:1 backing invariant for the affected amount.

### Likelihood Explanation
The minter address is publicly queryable by any unprivileged caller via `minter_address()`. Users who are unfamiliar with the helper-contract requirement — or who copy the minter address from the dashboard or `get_minter_info` — can trivially trigger this loss. The DID file itself warns against it, confirming the path is reachable and the risk is known. Likelihood is **Medium**: the warning exists, but it is not enforced at the protocol level, and the minter address is prominently surfaced.

### Recommendation
1. **Add a recovery endpoint** (callable by the affected user or by governance) that, given an Ethereum transaction hash of a direct transfer, verifies the transfer on-chain via the EVM-RPC canister and mints the corresponding ckETH to a specified IC account.
2. **Alternatively**, deploy a smart-contract proxy at the minter's Ethereum address that rejects plain ETH transfers (i.e., a `receive()` that reverts), forcing all deposits through the helper contract. This requires migrating the minter to a new tECDSA-controlled address backed by a smart contract wallet.
3. At minimum, update `get_minter_info` to compare the on-chain `eth_getBalance` against the internal `eth_balance` and surface any discrepancy as a metric or alert.

### Proof of Concept
1. Call `minter_address()` on the ckETH minter canister (`sv3dd-oaaaa-aaaar-qacoa-cai`) to obtain the minter's Ethereum address.
2. Send any amount of ETH directly to that address via a plain Ethereum transfer (no calldata, no helper contract).
3. Observe that the minter's log-scraping timer fires but finds no `ReceivedEth` / `ReceivedEthOrErc20` event for this transaction.
4. Observe that `get_minter_info().eth_balance` does not increase, while `eth_getBalance(<minter_address>)` on Ethereum does.
5. Observe that no ckETH is minted to any IC account, and no refund is issued.
6. Confirm there is no canister endpoint to recover the ETH.

The root cause is confirmed at:
- [5](#0-4)  — explicit acknowledgment that direct transfers are unaccounted.
- [6](#0-5)  — warning that direct sends are unsupported, with no enforcement.
- [3](#0-2)  — `withdraw_eth` requires a prior ckETH ledger balance; it cannot recover unattributed ETH.

#[update]
async fn withdraw_eth(
    WithdrawalArg {
        amount,
        recipient,
        from_subaccount,
    }: WithdrawalArg,
) -> Result<RetrieveEthRequest, WithdrawalError> {
    let caller = validate_caller_not_anonymous();
    let _guard = retrieve_withdraw_guard(caller).unwrap_or_else(|e| {
        ic_cdk::trap(format!(
            "Failed retrieving guard for principal {caller}: {e:?}"
        ))
    });

    let destination = validate_address_as_destination(&recipient).map_err(|e| match e {
        AddressValidationError::Invalid { .. } | AddressValidationError::NotSupported(_) => {
            ic_cdk::trap(e.to_string())
        }
        AddressValidationError::Blocked(address) => WithdrawalError::RecipientAddressBlocked {
            address: address.to_string(),
        },
    })?;

    let amount = Wei::try_from(amount).expect("failed to convert Nat to u256");

    let minimum_withdrawal_amount = read_state(|s| s.cketh_minimum_withdrawal_amount);
    if amount < minimum_withdrawal_amount {
        return Err(WithdrawalError::AmountTooLow {
            min_withdrawal_amount: minimum_withdrawal_amount.into(),
        });
    }

    let client = read_state(LedgerClient::cketh_ledger_from_state);
    let now = ic_cdk::api::time();
    log!(INFO, "[withdraw]: burning {:?}", amount);
    match client
        .burn_from(
            Account {
                owner: caller,
                subaccount: from_subaccount,
            },
            amount,
            BurnMemo::Convert {
                to_address: destination,
            },
        )
        .await
    {
        Ok(ledger_burn_index) => {
            let withdrawal_request = EthWithdrawalRequest {
                withdrawal_amount: amount,
                destination,
                ledger_burn_index,
                from: caller,
                from_subaccount: from_subaccount.and_then(LedgerSubaccount::from_bytes),
                created_at: Some(now),
            };

            log!(
                INFO,
                "[withdraw]: queuing withdrawal request {:?}",
                withdrawal_request,
            );

            mutate_state(|s| {
                process_event(
                    s,
                    EventType::AcceptedEthWithdrawalRequest(withdrawal_request.clone()),
                );
            });
            Ok(RetrieveEthRequest::from(withdrawal_request))
        }
        Err(e) => Err(WithdrawalError::from(e)),
    }
}
