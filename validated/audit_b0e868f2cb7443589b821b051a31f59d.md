Audit Report

## Title
Missing Execution-Time Capacity Check in `perform_add_generic_nervous_system_function` Allows Exceeding `MAX_NUMBER_OF_GENERIC_NERVOUS_SYSTEM_FUNCTIONS` - (File: rs/sns/governance/src/governance.rs)

## Summary
`perform_add_generic_nervous_system_function` in `rs/sns/governance/src/governance.rs` lacks a re-check of the registry size against `MAX_NUMBER_OF_GENERIC_NERVOUS_SYSTEM_FUNCTIONS` at execution time. The capacity guard exists only in `validate_and_render_add_generic_nervous_system_function` in `rs/sns/governance/src/proposal.rs`, which runs against a snapshot of state at proposal-submission time. Two adopted `AddGenericNervousSystemFunction` proposals with distinct IDs can both execute successfully when the registry is at `MAX - 1`, pushing it to `MAX + 1` and permanently violating the intended invariant.

## Finding Description
**Submission-time check** (`rs/sns/governance/src/proposal.rs`, lines 1395–1397):
```rust
if existing_functions.len() >= MAX_NUMBER_OF_GENERIC_NERVOUS_SYSTEM_FUNCTIONS {
    return Err("Reached maximum number of allowed GenericNervousSystemFunctions".to_string());
}
```
This check is evaluated against `id_to_nervous_system_functions` at the moment the proposal is submitted, not at execution time.

**Execution function** (`rs/sns/governance/src/governance.rs`, lines 2247–2299): `perform_add_generic_nervous_system_function` is a synchronous (non-`async`) function. It performs three guards: (1) native-function rejection, (2) duplicate-ID rejection via `is_registered_function_id`, and (3) reserved-canister-target rejection. There is no guard of the form `id_to_nervous_system_functions.len() >= MAX_NUMBER_OF_GENERIC_NERVOUS_SYSTEM_FUNCTIONS`. The function then unconditionally inserts the new entry at line 2295–2297.

The duplicate-ID guard at line 2261 only prevents the same ID from being inserted twice; it does not prevent two proposals with *different* IDs from both succeeding when the registry is already at capacity.

By contrast, `perform_manage_nervous_system_parameters` (lines 2595–2616) explicitly re-validates the merged parameters at execution time and carries a code comment acknowledging the concurrent-proposal risk: *"Even though proposals are validated when they are first made, this is still possible, because the inner value of a ManageNervousSystemParameters proposal is only valid with respect to the current nervous_system_parameters() at the time when the proposal was first made."* No equivalent guard exists in `perform_add_generic_nervous_system_function`.

**Execution flow**: `start_proposal_execution` (line 2118) spawns `perform_action` as a background future via `spawn_in_canister_env`. Since `perform_add_generic_nervous_system_function` is synchronous, each invocation runs atomically to completion. When two adopted proposals are processed sequentially, the first inserts its function (registry reaches `MAX`), and the second — seeing only that its ID is not a duplicate — inserts its function (registry reaches `MAX + 1`) with no error.

## Impact Explanation
The SNS governance canister's `id_to_nervous_system_functions` map ends up with more entries than `MAX_NUMBER_OF_GENERIC_NERVOUS_SYSTEM_FUNCTIONS` permits. This is a persistent state invariant violation in an in-scope SNS core canister. Downstream logic that relies on the cap being respected — including memory-budget calculations, `list_nervous_system_functions` responses, and any future enforcement logic — can behave incorrectly. The corrupted state persists until a corrective governance proposal is executed, meaning the SNS governance canister operates in an invalid state for an extended period. This matches the allowed impact: **High — Significant SNS governance security impact with concrete protocol harm**, as it corrupts the governance canister's authoritative registry of callable functions in a way that cannot be self-corrected without further governance action.

## Likelihood Explanation
No special privilege, key compromise, or malicious majority is required. Any SNS neuron holder with sufficient stake to submit proposals can trigger the precondition. Two neuron holders independently submit `AddGenericNervousSystemFunction` proposals with different IDs when the registry is at `MAX - 1` — a realistic operational state for active SNSes. Both proposals pass submission-time validation against the same state snapshot. Both gather enough votes through normal governance participation. Both execute sequentially (not even truly concurrently), with the second executing after the first has already filled the cap. No coordination between the two proposers is needed; the race condition arises from the structural gap between validation time and execution time.

## Recommendation
Add the capacity check inside `perform_add_generic_nervous_system_function`, mirroring the pattern in `perform_manage_nervous_system_parameters`:

```rust
fn perform_add_generic_nervous_system_function(
    &mut self,
    nervous_system_function: NervousSystemFunction,
) -> Result<(), GovernanceError> {
    // ... existing native/duplicate/reserved-canister checks ...

    // Re-check capacity at execution time; state may have changed since proposal submission.
    let current_count = self.proto.id_to_nervous_system_functions
        .values()
        .filter(|f| !f.is_native())  // exclude deletion markers / native entries
        .count();
    if current_count >= MAX_NUMBER_OF_GENERIC_NERVOUS_SYSTEM_FUNCTIONS {
        return Err(GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            "Reached maximum number of allowed GenericNervousSystemFunctions",
        ));
    }

    self.proto.id_to_nervous_system_functions.insert(id, nervous_system_function);
    Ok(())
}
```

The filter must match the logic used in `validate_and_render_add_generic_nervous_system_function` (i.e., exclude deletion markers) to ensure the two checks are consistent.

## Proof of Concept
1. Initialize an SNS governance canister with exactly `MAX_NUMBER_OF_GENERIC_NERVOUS_SYSTEM_FUNCTIONS - 1` entries in `id_to_nervous_system_functions`.
2. Call `perform_add_generic_nervous_system_function` with a valid `NervousSystemFunction` having `id = 1000`. Assert `Ok(())` is returned. Registry now has `MAX` entries.
3. Call `perform_add_generic_nervous_system_function` again with a valid `NervousSystemFunction` having `id = 1001` (different ID, same or different canister targets). Assert `Ok(())` is returned. Registry now has `MAX + 1` entries.
4. Assert `id_to_nervous_system_functions.len() > MAX_NUMBER_OF_GENERIC_NERVOUS_SYSTEM_FUNCTIONS`.

This is directly exercisable as a unit test against `Governance::perform_add_generic_nervous_system_function` in `rs/sns/governance/src/governance/assorted_governance_tests.rs`, following the pattern of the existing `test_add_generic_nervous_system_function_succeeds` test at line 3537. No async machinery or inter-canister calls are required since the function is synchronous.
