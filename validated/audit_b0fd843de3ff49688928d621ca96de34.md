### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Overcharges Users When Native Resource Is Cheaper Than EVM Gas — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

`compute_gas_refund` applies `delta_gas` only in the positive direction (native cost > EVM gas cost), but silently discards the negative case (EVM gas cost > native cost). This creates an asymmetric floor identical in structure to the StakingVault `min(_initialAssets, previewedAssets)` pattern: users always pay `max(ergs_gas_used, native_gas_equivalent)`, absorbing 100% of the upside cost when native is expensive but receiving zero benefit when native is cheap. The surplus fee is misdirected to the operator rather than refunded to the sender. A `// TODO: return delta_gas to gas_used?` comment on the dead branch explicitly acknowledges the gap.

### Finding Description

In `compute_gas_refund`:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // native cost > gas cost: charge extra gas
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?   ← negative case unhandled
``` [1](#0-0) 

Two compounding issues mirror the StakingVault report exactly:

**(a) Asymmetric floor.** `gas_used` is adjusted upward when `delta_gas > 0` but is never adjusted downward when `delta_gas < 0`. The effective charge is always `max(ergs_gas_used, native_used / native_per_gas)`. A user whose transaction consumes, say, 50 000 EVM gas but only 20 000 native-gas-equivalent pays for 50 000 gas; a user whose transaction consumes 50 000 EVM gas but 80 000 native-gas-equivalent pays for 80 000 gas. The asymmetry is one-directional.

**(b) Surplus misdirection.** When `delta_gas < 0`, the difference `(gas_used − native_used/native_per_gas) × gas_price` is collected by the operator via `refund_and_commit_fee` as part of `token_to_pay_operator = gas_used × gas_price_for_operator`. The sender is refunded only `(gas_limit − gas_used) × gas_price`, never the native-cost savings. The surplus is not burned (unlike the StakingVault case) but is misdirected to the operator beyond their actual proving cost. [2](#0-1) 

The `before_refund` hook in both the ZK and Ethereum transaction flows feeds directly into this function, so the asymmetry applies to every L2 and L1→L2 transaction processed by the bootloader. [3](#0-2) [4](#0-3) 

The documentation for double resource accounting describes only the positive `deltaGas` case and is silent on the negative case, consistent with the TODO comment being an unresolved design gap rather than intentional behaviour. [5](#0-4) 

### Impact Explanation

Every transaction where `native_used / native_per_gas < gas_used` (i.e., the proving cost is proportionally cheaper than the EVM computation cost) results in the sender paying more than the actual combined resource cost. The overpayment equals `(gas_used − native_used/native_per_gas) × gas_price` and flows to the operator as unearned fee. For a transaction with `gas_limit = 200 000`, `gas_price = 1 000`, and a native-gas-equivalent of 50 000, the sender overpays by 150 000 × 1 000 = 150 000 000 wei per transaction. This is a repeating, non-recoverable financial loss to users on every such transaction. The operator is correspondingly overpaid relative to their actual proving cost.

### Likelihood Explanation

Any unprivileged user submitting a transaction triggers this path. Transactions dominated by EVM-expensive but proving-cheap operations (e.g., many `SLOAD`/`SSTORE` with warm slots, large calldata that is cheap to prove but expensive in EVM gas) will consistently produce a negative `delta_gas`. The condition is reachable without any special privileges, malicious intent, or oracle manipulation — it fires on ordinary user transactions whenever the native resource consumption is proportionally lower than EVM gas consumption.

### Recommendation

Apply `delta_gas` symmetrically. When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (clamped to zero), so the sender is refunded the native-cost savings:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else {
    // Native was cheaper than EVM gas; return the savings to the sender.
    gas_used = gas_used.saturating_sub((-delta_gas) as u64);
    // Clamp to minimal_gas_used to preserve the validation floor.
    gas_used = gas_used.max(minimal_gas_used);
}
```

This mirrors the fix described in the StakingVault report (Option B: lock the exchange rate at the time of commitment) — here, the commitment is the ergs consumed, and the native cost should only increase the charge, never silently pocket the difference.

### Proof of Concept

**Scenario:** `gas_limit = 100 000`, `native_per_gas = 10`, `gas_price = 1 000`.

1. Transaction executes and consumes 60 000 EVM gas (ergs_remaining → 40 000 gas left).
2. Native resource consumed = 200 000 units → `native_gas_equivalent = 200 000 / 10 = 20 000`.
3. `delta_gas = 20 000 − 60 000 = −40 000` (negative).
4. Current code: `delta_gas < 0` branch is skipped; `gas_used = 60 000`.
5. Sender refund = `(100 000 − 60 000) × 1 000 = 40 000 000 wei`.
6. Operator receives `60 000 × 1 000 = 60 000 000 wei`.

**Correct behaviour:** `gas_used` should be reduced to `max(20 000, minimal_gas_used)`.
- Sender refund = `(100 000 − 20 000) × 1 000 = 80 000 000 wei`.
- Operator receives `20 000 × 1 000 = 20 000 000 wei` (actual proving cost).
- Overpayment per transaction: `40 000 000 wei` — permanently misdirected to the operator.

The `// TODO: return delta_gas to gas_used?` comment at line 80 of `refund_calculation.rs` is the in-code acknowledgement of this exact gap. [6](#0-5)
