### Title
`set_eth_connector_contract_account` Does Not Migrate Token Balances, Causing Permanent Fund Freeze — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

When the ETH connector contract account is changed via `set_eth_connector_contract_account`, all future `withdraw` calls are silently rerouted to the new connector. However, existing user NEP-141 balances remain in the old connector. The new connector has no record of those balances, so every subsequent `withdraw` call fails — permanently freezing all ETH that was deposited before the connector change.

---

### Finding Description

`set_eth_connector_contract_account` (lines 418–438 of `engine/src/contract_methods/connector.rs`) updates only two storage keys:

```rust
set_connector_account_id(io, &args.account);
set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
``` [1](#0-0) 

It performs no balance migration, no state transfer, and no guard against an existing non-zero total supply. After the call, `get_connector_account_id` returns the new account for every subsequent operation.

Every user-facing connector operation — `withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_unregister`, `storage_withdraw` — is dispatched through `return_promise`, which resolves the target account dynamically at call time:

```rust
fn return_promise<I: IO + PromiseHandler, E: Env>(...) -> Result<(), ContractError> {
    let promise_args = PromiseCreateArgs {
        target_account_id: get_connector_account_id(&io)?,   // ← always the NEW connector
        ...
    };
``` [2](#0-1) 

Simultaneously, `ft_on_transfer` uses the same stored account ID to decide whether an incoming NEP-141 transfer is a base-token (ETH) deposit or an ERC-20 deposit:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(...)
};
``` [3](#0-2) 

After the connector account is changed:

| Operation | Effect |
|---|---|
| `withdraw` | Calls `engine_withdraw` on the **new** connector; new connector has no user balance → fails |
| `ft_transfer` / `ft_transfer_call` | Routed to new connector; new connector has no balance → fails |
| Old connector calling `ft_on_transfer` | Treated as an unregistered ERC-20, not base token → rejected |

All ETH deposited through the old connector is permanently inaccessible.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Every user who deposited ETH before the connector change loses the ability to withdraw. The old connector still holds the NEP-141 balances; the new connector has none. There is no recovery path without a separate, manual migration transaction that the protocol does not provide.

---

### Likelihood Explanation

**Medium.** `set_eth_connector_contract_account` is a documented, intentional admin function used during connector upgrades (e.g., the test harness calls it during setup). A protocol upgrade that deploys a new ETH connector and points the engine at it — without first migrating all NEP-141 balances — triggers the freeze. This is a realistic operational scenario, not a theoretical one. The function carries no warning, no guard on existing supply, and no migration step. [4](#0-3) 

---

### Recommendation

1. **Guard on existing supply**: Before accepting a new connector account, read the current total supply from the old connector. If it is non-zero, revert with an explicit error requiring migration first.
2. **Atomic migration**: Provide a companion function that transfers all NEP-141 balances from the old connector to the new one atomically before updating the pointer.
3. **Documentation**: At minimum, add a prominent warning that calling this function while users hold balances will permanently freeze those funds.

---

### Proof of Concept

1. Users deposit ETH; the old connector (`old-connector.near`) records their NEP-141 balances.
2. Owner calls `set_eth_connector_contract_account` pointing to `new-connector.near` (a freshly deployed, empty connector).
3. User calls `withdraw` on the Aurora Engine.
4. `return_promise` resolves `get_connector_account_id` → `new-connector.near`.
5. A cross-contract call to `new-connector.near::engine_withdraw` is scheduled.
6. `new-connector.near` has no record of the user's balance → the call fails.
7. The user's ETH is permanently locked in `old-connector.near` with no withdrawal path. [5](#0-4) [6](#0-5) [2](#0-1)

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}
