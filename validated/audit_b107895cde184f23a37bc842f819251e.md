### Title
Silent Precompile Failure in `withdrawToNear` Burns ERC-20 Tokens Without Scheduling NEAR Transfer — (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` by first burning the caller's ERC-20 tokens via `_burn`, then calling the `ExitToNear` precompile via inline assembly. The return value of the assembly `call` is never checked. If the precompile rejects the input (e.g., because the user-supplied `recipient` bytes are not a valid NEAR account ID), the tokens are permanently burned with no corresponding NEP-141 `ft_transfer` scheduled on NEAR. The NEP-141 tokens remain locked in Aurora's custody forever, with no ERC-20 tokens left to reclaim them.

---

### Finding Description

`EvmErc20.sol` implements `withdrawToNear` as:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is never checked
    }
}
``` [1](#0-0) 

`EvmErc20V2.sol` has the same unchecked pattern: [2](#0-1) 

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) parses the `recipient` bytes and returns `ExitError` for any of the following conditions:

- `recipient` is not valid UTF-8 → `ERR_INVALID_RECEIVER_ACCOUNT_ID`
- The parsed string is not a valid NEAR account ID → `ERR_INVALID_RECEIVER_ACCOUNT_ID`
- `amount > u128::MAX` → `ERR_INVALID_AMOUNT`
- The ERC-20 address is not registered in the NEP-141 map → `ERR_TARGET_TOKEN_NOT_FOUND` [3](#0-2) [4](#0-3) 

When the precompile returns `ExitError`, the EVM `call` opcode returns `0` (failure). In EVM semantics, a low-level `call` failure does **not** revert the caller's own state changes. The `_burn` executed before the assembly block is not rolled back. The function returns normally from the ERC-20 contract's perspective.

The contract selection at deploy time is:

```rust
#[cfg(feature = "error_refund")]
let erc20_contract = include_bytes!("../../etc/eth-contracts/res/EvmErc20V2.bin");
#[cfg(not(feature = "error_refund"))]
let erc20_contract = include_bytes!("../../etc/eth-contracts/res/EvmErc20.bin");
``` [5](#0-4) 

The `error_refund` callback mechanism (`exit_to_near_precompile_callback`) only handles NEAR-side `ft_transfer` failures — it is never scheduled when the precompile itself fails before producing a promise. So even with `error_refund` enabled and `EvmErc20V2` deployed, a precompile-level parse failure leaves no refund path.

---

### Impact Explanation

When `withdrawToNear` is called with a recipient that fails precompile validation:

1. The caller's ERC-20 tokens are burned (destroyed on Aurora).
2. No `ft_transfer` promise is scheduled on NEAR.
3. The NEP-141 tokens remain locked in Aurora's account on NEAR.
4. There are no ERC-20 tokens left to initiate a new withdrawal.

The NEP-141 tokens are permanently frozen in Aurora's custody. This is a **permanent freezing of funds** matching the Critical impact tier.

---

### Likelihood Explanation

The `recipient` parameter is a raw `bytes` argument, fully controlled by the caller. Any of the following inputs trigger a silent precompile failure:

- Non-UTF-8 byte sequences (e.g., `\xff\xfe`)
- Strings that are not valid NEAR account IDs (e.g., containing uppercase letters, spaces, or exceeding 64 characters)
- An empty byte array

A user who mistypes or programmatically constructs an invalid NEAR account ID will silently lose their tokens. No on-chain validation or revert occurs to warn them. The likelihood is **Medium** — accidental misuse is realistic for any user interacting directly with the contract or via a buggy integration.

---

### Recommendation

Check the return value of the precompile `call` and revert if it fails:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    bool success;
    assembly {
        success := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
    }
    require(success, "ExitToNear precompile call failed");
}
```

This ensures that if the precompile rejects the input, the entire transaction reverts (including the `_burn`), preserving the user's tokens.

---

### Proof of Concept

1. Deploy an `EvmErc20` token for a registered NEP-141 (e.g., `wrap.near`).
2. Mint 100 tokens to address `Alice`.
3. Alice calls `withdrawToNear("\xff\xfe", 100)` — a non-UTF-8 recipient.
4. `_burn(Alice, 100)` executes; Alice's ERC-20 balance drops to 0.
5. The precompile receives `\x01 + bytes32(100) + "\xff\xfe"`.
6. `parse_recipient` fails with `ERR_INVALID_RECEIVER_ACCOUNT_ID` (non-UTF-8). [6](#0-5) 

7. The assembly `call` returns `0`; `res` is discarded.
8. `withdrawToNear` returns normally with no revert.
9. Alice's 100 ERC-20 tokens are gone. The 100 NEP-141 tokens remain locked in Aurora's NEAR account with no ERC-20 tokens to reclaim them — permanent freeze.

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))

                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
