### Title
Silo Account/Address Whitelist Not Enforced in `call` Entrypoint — (File: `engine/src/contract_methods/evm_transactions.rs`)

### Summary
Aurora Engine's Silo mode exposes a whitelist system (`Account` and `Address` kinds) to restrict which NEAR accounts and EVM addresses may interact with the private EVM. The `submit` and `submit_with_args` entrypoints correctly enforce these restrictions via `assert_access`. The `call` entrypoint does not perform any whitelist check at all, allowing any NEAR account to execute arbitrary EVM calls inside a Silo, bypassing the intended access control entirely.

### Finding Description

The Silo whitelist system defines four categories of access control. The two relevant to transaction submission are:

- `WhitelistKind::Account` — restricts which NEAR `predecessor_account_id` may submit transactions.
- `WhitelistKind::Address` — restricts which EVM address (derived from the transaction signer) may be `msg.sender`. [1](#0-0) 

These are enforced in `engine/src/engine.rs` via `assert_access`, which is called unconditionally inside `submit_with_alt_modexp`: [2](#0-1) [3](#0-2) 

`assert_access` delegates to `silo::is_allow_submit` (for calls) or `silo::is_allow_deploy` (for contract creation), both of which read the enabled whitelists from storage: [4](#0-3) 

However, the `call` entrypoint in `evm_transactions.rs` performs **no whitelist check whatsoever**. It only calls `require_running` and then directly executes the EVM call: [5](#0-4) 

The `call` entrypoint is publicly exposed as a NEAR contract method: [6](#0-5) 

`CallArgs` lets the caller specify an arbitrary EVM contract address, ETH value, and calldata. The EVM `msg.sender` is derived deterministically from the NEAR `predecessor_account_id`, so any NEAR account can craft a `call` invocation that executes inside the Silo's EVM with a valid-looking sender address, completely bypassing the `Account` and `Address` whitelist restrictions that the Silo operator configured.

### Impact Explanation

A Silo operator enables the `Account` and `Address` whitelists specifically to create a private, permissioned EVM environment. Any NEAR account that is **not** on the whitelist is supposed to be blocked from interacting with the Silo's EVM contracts. Because `call` skips the whitelist check, any unprivileged NEAR account can:

1. Call any EVM contract deployed inside the Silo.
2. Transfer ETH or ERC-20 tokens held by those contracts if the contracts themselves do not implement independent access control (which is a reasonable assumption in a Silo that relies on the engine-level whitelist as its perimeter).
3. Trigger arbitrary state changes in the Silo's EVM state.

This constitutes a direct bypass of the Silo's access control perimeter, with potential for direct theft of user funds held in Silo EVM contracts. **Impact: Critical (direct theft of user funds / auth bypass).**

### Likelihood Explanation

The `call` entrypoint is a standard, publicly callable NEAR contract method requiring no special keys or privileges. Any NEAR account can invoke it. The bypass requires no exploit tooling — the attacker simply uses `call` instead of `submit`. Likelihood is **High**.

### Recommendation

Add a silo whitelist check inside the `call` function, mirroring the logic in `assert_access`. At minimum, verify that `env.predecessor_account_id()` is permitted by the `Account` whitelist and that the derived EVM address (`predecessor_address(&predecessor_account_id)`) is permitted by the `Address` whitelist before proceeding to `engine.call_with_args`.

### Proof of Concept

1. Deploy an Aurora Engine in Silo mode.
2. Enable the `Account` and `Address` whitelists via `set_whitelist_status`.
3. Do **not** add the attacker's NEAR account to either whitelist.
4. Attacker calls `submit` with a signed Ethereum transaction → rejected with `NotAllowed` (whitelist enforced).
5. Attacker calls `call` with `CallArgs { contract: <target_evm_address>, value: 0, input: <calldata> }` → **succeeds**, executing inside the Silo's EVM with no whitelist check, demonstrating the bypass.

The root cause is structurally identical to the reference report: the whitelist data structures and management functions exist and work correctly, but the enforcement check is simply absent from one of the execution entry paths. [5](#0-4) [7](#0-6)

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
