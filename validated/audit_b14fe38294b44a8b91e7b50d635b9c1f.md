Audit Report

## Title
Unbounded Loop Over Proposals × Ballots in SNS `distribute_rewards` Causes Permanent Instruction-Limit DoS — (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance canister's `distribute_rewards` function performs five sequential unbounded loops over all `ReadyToSettle` proposals and their per-neuron ballots in a single synchronous message, with no instruction-limit guard anywhere in the function. If the combined work exceeds the IC per-message instruction limit, the message traps, state is rolled back, and every subsequent periodic timer invocation encounters the same or larger workload — permanently preventing reward distribution and proposal settlement without a canister upgrade.

## Finding Description
`distribute_rewards` at `rs/sns/governance/src/governance.rs:5763` contains five sequential unbounded loops:

1. `for i in 1..=new_rounds_count` (L5861) — iterates over every missed reward round.
2. `for proposal_id in &considered_proposals` (L5894) — iterates over every `ReadyToSettle` proposal.
3. `for (voter, ballot) in &proposal.ballots` (L5896) — for each proposal, iterates over every ballot (one per eligible neuron at proposal creation time).
4. `for (neuron_id, neuron_reward_shares) in neuron_id_to_reward_shares` (L5954) — iterates over every neuron that voted across all settled proposals.
5. `for pid in &considered_proposals` (L6013) — iterates over all settled proposals again to clear ballots and call `process_proposal`.

A grep search across the entire `rs/sns/governance/` directory confirms there are zero calls to `is_message_over_threshold`, `is_over_instructions_limit`, `performance_counter`, or any instruction-counting primitive. The function is called from the SNS periodic-task timer with no batching mechanism.

By contrast, the NNS governance canister (`rs/nns/governance/src/reward/distribution.rs:8,17,43,184`) imports `is_message_over_threshold` from `ic_nervous_system_long_message`, sets a `DISTRIBUTION_MESSAGE_LIMIT` of 1 billion instructions, and uses a `RewardsDistributionStateMachine` that persists intermediate progress across multiple timer invocations. The SNS governance canister received no equivalent fix.

When the instruction limit is exceeded, the IC traps the message and rolls back all state changes. Because `latest_reward_event` is only updated at L6084 — after all five loops complete — a trap at any point leaves `latest_reward_event` unchanged. The next timer invocation calls `distribute_rewards` again with the same (or larger) set of proposals and ballots, trapping again. The system is permanently stuck.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

Concrete consequences:
- Voting rewards are never distributed to SNS neurons.
- Proposals in `ReadyToSettle` state are never settled; ballots are never cleared, causing unbounded state growth.
- The accumulation of unsettled proposals and their ballots grows over time, making the workload progressively worse.
- Recovery requires a canister upgrade — an SNS governance action that itself requires a functioning governance system.

## Likelihood Explanation
**Low-to-Medium.** Any SNS token holder can stake to create neurons (no privileged access required) and any neuron with sufficient stake can submit proposals. The total work is proportional to `P × N` where `P` = proposals in `ReadyToSettle` and `N` = neurons eligible per proposal. For a popular SNS with thousands of neurons and dozens of accumulated proposals, this threshold can be reached organically. A motivated attacker holding SNS tokens can accelerate this by submitting many proposals and creating many neurons. No admin key, governance majority, or threshold attack is required.

## Recommendation
Apply the same batching pattern already implemented in NNS governance:

1. Introduce a `RewardsDistributionStateMachine` equivalent for SNS that persists intermediate reward-distribution progress in stable storage.
2. Add `is_message_over_threshold` checks inside each loop body and break when the soft limit is reached, re-scheduling the timer to continue in the next message.
3. Update `latest_reward_event` only after all loops have fully completed across potentially multiple messages.
4. Enforce a maximum number of proposals that can accumulate in `ReadyToSettle` state simultaneously to bound worst-case work per invocation.

## Proof of Concept
**Setup:**
- Deploy an SNS with a short `round_duration_seconds`.
- Create N neurons (e.g., 5,000) by having token holders stake.
- Submit P proposals (e.g., 50–100) and allow them to reach `ReadyToSettle` state.

**Trigger:**
- Wait for the periodic task timer to fire and invoke `distribute_rewards`.
- The inner ballot loop at L5896 processes N × P entries. At ~2,000 instructions per ballot entry, 50,000 neurons × 100 proposals = 5,000,000 entries × ~2,000 instructions = ~10 billion instructions, approaching or exceeding the IC per-message limit for timer calls.

**Verification:**
- Observe that `latest_reward_event` (L6084) is not updated after the trap.
- Observe that the next timer invocation encounters the same workload and traps again.
- Write a deterministic integration test using PocketIC: populate the SNS governance state with a large number of neurons and `ReadyToSettle` proposals, then call `run_periodic_tasks` and assert that either (a) the call traps, or (b) `latest_reward_event` is not updated, confirming the permanent DoS condition.
