### Title
Permanent Loss of User Funds When Exit-to-NEAR Transfer Fails Without `error_refund` Feature - (`engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

The `ExitToNear` precompile irrevocably burns ERC-20 tokens or deducts ETH from the user's EVM balance before scheduling the NEAR-side transfer promise. When the compile-time `error_refund` feature is not enabled, no refund callback is wired for failure cases. If the NEAR-side `ft_transfer`, `ft_transfer_call`, or `near_withdraw` promise fails, the user's funds are permanently lost with no recovery path.

---

### Finding Description

**Step 1 — Tokens are committed before the NEAR promise executes.**

In `engine-precompiles/src/native.rs`, the `ExitToNear::run` precompile constructs `ExitToNearPrecompileCallbackArgs`. When the `error_refund` feature is absent, `refund` is hardcoded to `None`:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

**Step 2 — No callback is scheduled for regular ERC-20 / ETH exits.**

The callback promise is only scheduled when `callback_args != default()`. For regular ERC-20 exits and ETH exits (non-wNEAR), both `refund` and `transfer_near` are `None`, so `callback_args == default()` and only a bare `PromiseArgs::Create` is emitted — no failure handler:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // no callback, no refund on failure
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

**Step 3 — wNEAR unwrap path: callback is scheduled but the failure branch is a no-op.**

For the wNEAR `:unwrap` path, `transfer_near` is `Some(...)`, so a callback *is* scheduled. However, inside `exit_to_near_precompile_callback`, when the base `near_withdraw` promise fails, the code checks `args.refund`:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // <-- reached when error_refund is disabled; burned wNEAR is gone
};
``` [3](#0-2) 

Because `args.refund` is `None` (feature disabled), execution falls into the `else { None }` branch. The wNEAR ERC-20 tokens that were already burned in the EVM are permanently unrecoverable.

**The `ExitToNearPrecompileCallbackArgs` struct confirms both fields are optional:** [4](#0-3) 

---

### Impact Explanation

Three distinct fund-loss scenarios exist when `error_refund` is not compiled in:

| Exit path | Tokens committed | NEAR-side failure | Refund? |
|---|---|---|---|
| Regular ERC-20 → NEAR | ERC-20 burned | `ft_transfer` fails | **No** |
| ETH → NEAR | ETH deducted from EVM | `ft_transfer` fails | **No** |
| wNEAR ERC-20 → unwrap | wNEAR ERC-20 burned | `near_withdraw` fails | **No** |

In all three cases the EVM-side state change is committed atomically with the EVM transaction, while the NEAR-side promise executes asynchronously. A failure of the NEAR promise leaves the user with neither their original tokens nor the bridged output. The tests explicitly confirm this:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

Impact: **Critical — permanent freezing/loss of user funds.**

---

### Likelihood Explanation

Any unprivileged EVM user who calls the exit precompile can trigger this. Realistic failure conditions for the NEAR-side promise include:

- Recipient account not registered with the NEP-141 contract (storage deposit missing).
- Aurora's NEP-141 balance insufficient to cover the `ft_transfer` (e.g., drained by a concurrent operation).
- `near_withdraw` on the wNEAR contract failing due to insufficient balance or contract pause.

These are not exotic edge cases; the first condition (unregistered recipient) is a routine user error. [6](#0-5) 

---

### Recommendation

1. **Enable `error_refund` unconditionally** in the production build, or remove the feature gate and always populate `refund` in `ExitToNearPrecompileCallbackArgs`.
2. For the no-callback path (regular ERC-20/ETH exits without `error_refund`), always attach a callback that re-mints/refunds on promise failure, mirroring the `refund_on_error` logic already present in `engine/src/engine.rs`. [7](#0-6) 

---

### Proof of Concept

1. Deploy an ERC-20 token bridged from a NEP-141 on Aurora (without `error_refund` compiled in).
2. Call the ERC-20's `burn`/exit function targeting an unregistered NEAR account as recipient.
3. The ERC-20 tokens are burned in the EVM (EVM state committed).
4. The `ft_transfer` NEAR promise fails (recipient not registered).
5. No callback is scheduled → no `refund_on_error` is called.
6. User's ERC-20 balance is zero; the NEP-141 balance on Aurora is unchanged; the user receives nothing.

The same sequence applies to ETH exits via `withdrawEthToNear` when the connector's `ft_transfer` fails. [8](#0-7) [9](#0-8)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
