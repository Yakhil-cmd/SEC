### Title
Changing wNEAR Address During In-Flight XCC Operation Causes Permanent Fund Freeze — (File: `engine/src/contract_methods/xcc.rs`)

---

### Summary

The `factory_set_wnear_address` admin function can be called by the owner while an XCC promise chain is in-flight. The `withdraw_wnear_to_router` callback reads the wNEAR ERC-20 address from storage **at execution time**, not at promise-creation time. If the owner changes the wNEAR address between when the XCC precompile fires and when `withdraw_wnear_to_router` executes as a NEAR receipt callback, the function targets the wrong ERC-20 contract, the EVM call fails, and any wNEAR already transferred from the user to the engine's EVM address is permanently frozen.

---

### Finding Description

The XCC flow involves a multi-step NEAR promise chain:

1. **Block N** — User submits an EVM transaction that triggers the XCC precompile. The precompile transfers wNEAR from the user's EVM balance to the engine's EVM address (via `transferFrom`) and emits a log. `filter_promises_from_logs` in `engine.rs` calls `handle_precompile_promise`, which schedules a `withdraw_wnear_to_router` self-call as a NEAR receipt.

2. **Block N+1** — `withdraw_wnear_to_router` executes as a NEAR receipt callback. At line 43 of `contract_methods/xcc.rs`, it reads the wNEAR address **fresh from storage**:

```rust
let wnear_address = aurora_engine_precompiles::xcc::state::get_wnear_address(&io);
```

It then constructs an EVM call targeting that address to invoke `withdrawToNear`, burning the engine's wNEAR and delivering NEAR to the router sub-account.

The owner can call `factory_set_wnear_address` at any time between Block N and Block N+1:

```rust
pub fn factory_set_wnear_address<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        ...
        require_owner_only(&state, &env.predecessor_account_id())?;
        let address = io.read_input_arr20()?;
        xcc::set_wnear_address(&mut io, &Address::from_array(address));
        Ok(())
    })
}
```

There is no guard preventing this call while XCC receipts are pending. When `withdraw_wnear_to_router` executes, it reads the **new** address, constructs an EVM call to a contract that does not implement `withdrawToNear` (or implements it differently), the EVM call fails, and the function returns `ERR_WITHDRAW_FAILED`. The entire XCC promise chain aborts.

The wNEAR that was transferred from the user to the engine's EVM address in Block N is now stranded in the engine's implicit EVM address with no user-accessible recovery path.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The user's wNEAR tokens are transferred to the engine's EVM address during the EVM execution in Block N. If `withdraw_wnear_to_router` fails in Block N+1 due to a stale wNEAR address, those tokens remain in the engine's EVM address indefinitely. The user has no contract method to reclaim them. The engine owner could theoretically recover them via an admin EVM call, but the user's funds are frozen from their perspective.

---

### Likelihood Explanation

**Low-to-medium.** The owner has a legitimate reason to call `factory_set_wnear_address` during a wNEAR contract migration. NEAR receipts from a single transaction are processed across multiple blocks, creating a window of at least one block between XCC precompile execution and `withdraw_wnear_to_router` execution. During high-throughput periods, many XCC operations may be in-flight simultaneously. A single `factory_set_wnear_address` call during migration would affect all in-flight XCC operations that require NEAR withdrawal, freezing all their wNEAR.

---

### Recommendation

Snapshot the wNEAR address at promise-creation time and pass it as an argument to `withdraw_wnear_to_router` via `WithdrawWnearToRouterArgs`, rather than re-reading it from storage at callback execution time. This ensures the callback always uses the address that was valid when the user's XCC operation was initiated.

Alternatively, add a check in `factory_set_wnear_address` that rejects the call if any `withdraw_wnear_to_router` receipts are known to be pending (though this is harder to implement in a stateless NEAR contract).

---

**Step 2.** Before the `withdraw_wnear_to_router` receipt executes (i.e., before Block N+1), the owner calls `factory_set_wnear_address` with a new address (e.g., a zero address or a newly deployed contract that does not implement `withdrawToNear`).

        io.read_storage(&key).map_or_else(
