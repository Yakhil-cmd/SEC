Audit Report

## Title
Single Malicious Callee Can Exhaust Composite Query Instruction Budget, Causing Permanent DoS - (File: rs/execution_environment/src/query_handler/query_call_graph.rs)

## Summary
The composite query execution framework initializes `round_limits.instructions` from `max_query_call_graph_instructions` (5 B) and caps each sub-call at `min(max_instructions_per_query, remaining_budget)`. Because both limits are identically 5 B, the very first sub-call in a composite query is permitted to consume the entire shared budget. On the next DFS loop iteration, `instruction_limit_reached()` fires and the whole composite query is terminated with `QueryCallGraphTotalInstructionLimitExceeded`. Any canister that fans out composite queries to a registry it does not fully control is permanently blockable by a single malicious callee.

## Finding Description
`QueryContext::new` initializes the shared budget directly from `max_query_call_graph_instructions`:

```rust
instructions: as_round_instructions(max_query_call_graph_instructions),
``` [1](#0-0) 

The per-sub-call instruction limit is computed as:

```rust
let instruction_limit = self.max_instructions_per_query.min(NumInstructions::new(
    self.round_limits.instructions.get().max(0) as u64,
));
``` [2](#0-1) 

Both `max_instructions_per_query` (`MAX_INSTRUCTIONS_PER_QUERY_MESSAGE = 5 B`) and the graph budget (`MAX_INSTRUCTIONS_PER_COMPOSITE_QUERY_CALL = 5_000_000_000`) are identical: [3](#0-2) [4](#0-3) 

Therefore the first sub-call receives `min(5B, 5B) = 5B` instructions. If it consumes all of them, `round_limits.instructions` reaches zero (then goes negative after the 50 M overhead deduction applied *after* execution): [5](#0-4) 

On the very next iteration of `evaluate_query_call_graph`, the guard at the top of the loop fires:

```rust
if query_context.instruction_limit_reached() {
    let error = UserError::new(
        ErrorCode::QueryCallGraphTotalInstructionLimitExceeded,
        "Composite query calls exceeded the instruction limit.",
    );
    return QueryResponse::UserError(error);
}
``` [6](#0-5) 

All remaining callees in the graph are never reached. The existing test `composite_query_callgraph_max_instructions_is_enforced` explicitly confirms that exceeding the graph budget causes the entire query to fail with this error code: [7](#0-6) 

## Impact Explanation
Any canister that builds its composite query call graph from an open or user-extensible registry (DEX aggregators, portfolio trackers, open index canisters) can have its composite query endpoint permanently disabled. The attacker deploys one canister that spins a tight Wasm loop until its instruction allowance is consumed. Once that canister appears anywhere in the call graph, the entire composite query fails and no subsequent callees are reached. This is a concrete, persistent application-level DoS matching the allowed impact: **High — Application/platform-level DoS not based on raw volumetric DDoS**.

## Likelihood Explanation
The attack requires only an unprivileged canister deployment (one-time cost) and registration in the target's open registry. The malicious canister's instructions are charged to the query caller, so the attacker bears zero per-query cost. The attack is persistent: the malicious canister remains deployed indefinitely and every subsequent composite query that includes it fails identically. No privileged access, social engineering, or network-level compromise is required.

## Recommendation
Decouple the per-sub-call instruction limit from the total graph budget. Cap each sub-call at `max_query_call_graph_instructions / MAX_QUERY_CALL_DEPTH` (≈ 833 M with current defaults of 5 B / 6), so no single callee can exhaust the shared pool. Alternatively, introduce a dedicated `max_instructions_per_composite_query_sub_call` configuration parameter distinct from `max_instructions_per_query`, and set it to a fraction of the total graph budget. The overhead comment ("This would allow 100 calls") already signals the designers' intent that the budget should support many calls, not just one. [8](#0-7) 

## Proof of Concept
1. Deploy canister **M** whose query method executes a tight Wasm loop consuming all available instructions before returning.
2. Deploy canister **A** with a composite query that sequentially calls `[M, B, C]` (M is first in the list).
3. Send a composite query to **A**.
4. **A** calls **M**; **M** receives `min(5B, 5B) = 5B` instructions and consumes them all.
5. `round_limits.instructions` drops to 0, then to −50 M after the overhead deduction.
6. On the next DFS iteration, `instruction_limit_reached()` returns `true`; `evaluate_query_call_graph` returns `QueryCallGraphTotalInstructionLimitExceeded`.
7. **B** and **C** are never called; the user receives an error.
8. Every subsequent composite query to **A** that includes **M** fails identically at zero cost to the attacker.

A minimal reproducible test can be written by adapting `composite_query_callgraph_max_instructions_is_enforced` in `rs/execution_environment/src/query_handler/tests.rs`: set `max_query_call_graph_instructions` to the production value (5 B), configure one callee to execute a tight loop consuming the full 5 B allowance, and assert that the composite query fails with `QueryCallGraphTotalInstructionLimitExceeded` even though subsequent callees were never attempted. [9](#0-8)

#[test]
fn composite_query_callgraph_max_instructions_is_enforced() {
    const NUM_CANISTERS: u64 = 20;
    const NUM_SUCCESSFUL_QUERIES: u64 = 5; // Number of calls expected to succeed

    let mut test = ExecutionTestBuilder::new()
        .with_max_query_call_graph_instructions(NumInstructions::from(
            NUM_SUCCESSFUL_QUERIES * INSTRUCTION_OVERHEAD_PER_QUERY_CALL,
        ))
        .build();

    let mut canisters = vec![];
    for _ in 0..NUM_CANISTERS {
        canisters.push(test.universal_canister_with_cycles(CYCLES_BALANCE).unwrap());
    }

    // Generate call tree of depth 1.
    // Canister 0 will call into each canister 1..num_canisters exactly once in a sequential manner.
    // This will therefore *not* hit the call graph depth limit, but should hit a limit
    // on the maximum number of instructions in a call graph.
    fn generate_call_to(
        canisters: &[ic_types::CanisterId],
        canister_idx: usize,
    ) -> ic_universal_canister::PayloadBuilder {
        assert_lt!(canister_idx, canisters.len());

        let reply = if canister_idx <= 1 {
            wasm().stable_size().reply_int()
        } else {
            generate_call_to(canisters, canister_idx - 1)
        };

        wasm().stable_grow(10).composite_query(
            canisters[canister_idx],
            call_args()
                .other_side(wasm().reply_data(b"ignore".as_ref()))
                .on_reply(reply),
        )
    }

    // Those should succeed
    for num_calls in 1..NUM_SUCCESSFUL_QUERIES {
        let test = test.non_replicated_query(
            canisters[0],
            "composite_query",
            generate_call_to(&canisters, num_calls as usize).build(),
        );
        match &test {
            Ok(_) => {}
            Err(err) => panic!(
                "Query with {num_calls} calls failed, when it should have succeeded: {err:?}"
            ),
        }
    }
    for num_calls in NUM_SUCCESSFUL_QUERIES..NUM_CANISTERS {
        let test = test.non_replicated_query(
            canisters[0],
            "composite_query",
            generate_call_to(&canisters, num_calls as usize).build(),
        );
        match &test {
            Ok(_) => panic!("Query with {num_calls} calls should have failed!"),
            Err(err) => assert_eq!(
                err.code(),
                ErrorCode::QueryCallGraphTotalInstructionLimitExceeded
            ),
        }
    }
