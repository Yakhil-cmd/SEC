### Title
Single-Step Ownership Transfer Allows Irrecoverable Loss of Engine Control - (File: engine/src/contract_methods/admin.rs)

### Summary
The `set_owner` function in the Aurora Engine performs ownership transfer in a single atomic step with no confirmation from the new owner. If the current owner mistypes the new account ID or specifies a non-existent/inaccessible NEAR account, ownership is immediately and irrevocably transferred. Because all critical administrative functions — including `resume_contract`, `upgrade`, and `stage_upgrade` — are gated exclusively behind `require_owner_only`, a lost owner permanently locks the engine's administrative surface. If the contract is paused at the time of the erroneous transfer (or is paused afterward by the new, inaccessible owner), all user funds bridged into Aurora are permanently frozen with no recovery path.

### Finding Description
`set_owner` in `engine/src/contract_methods/admin.rs` reads a `SetOwnerArgs` struct containing `new_owner: AccountId` and immediately overwrites `state.owner_id` in a single step:

```rust
// engine/src/contract_methods/admin.rs, lines 103-121
pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;

        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }

        state.owner_id = args.new_owner;   // ← immediate, unconditional overwrite
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
```

The only guard is that the new owner must differ from the current owner (`ERR_SAME_OWNER`). There is no pending-owner stage, no acceptance call, and no time-lock. The new `AccountId` is never verified to correspond to an existing, accessible NEAR account.

Every critical administrative function checks `require_owner_only` against `state.owner_id`:

| Function | Effect if owner is lost |
|---|---|
| `resume_contract` | Contract stays paused forever |
| `pause_contract` | Cannot pause for emergency response |
| `upgrade` / `stage_upgrade` | Contract cannot be patched |
| `set_upgrade_delay_blocks` | Upgrade delay cannot be adjusted |
| `resume_precompiles` | Paused precompiles stay paused |
| `set_key_manager` | Relayer key management frozen | [1](#0-0) [2](#0-1) 

### Impact Explanation
**Critical — Permanent freezing of funds.**

The Aurora Engine holds bridged ETH and ERC-20 tokens on behalf of users. The `pause_contract` / `resume_contract` pair is the primary emergency-response mechanism. If ownership is transferred to an inaccessible account while the contract is paused (or if the contract is paused before the error is noticed), `resume_contract` can never be called again because it unconditionally requires `require_owner_only`. All user funds locked in the engine become permanently inaccessible. Additionally, `upgrade` is owner-only, so no code fix can be deployed to recover the situation. [3](#0-2) [4](#0-3) 

### Likelihood Explanation
**Medium.** NEAR account IDs are human-readable strings (e.g., `aurora.near`, `aurora-dao.near`). A single character typo — transposing letters, using `.` vs `-`, or specifying a sub-account that has not yet been created — produces a syntactically valid `AccountId` that passes Borsh deserialization and the `ERR_SAME_OWNER` check, but corresponds to an account no one controls. The `AccountId` type only validates format, not existence on-chain. [5](#0-4) 

### Recommendation
Implement a two-step ownership transfer:

1. **Step 1 — `propose_owner(new_owner: AccountId)`**: Store `new_owner` as a `pending_owner` field in `EngineState`. Only callable by the current owner.
2. **Step 2 — `accept_ownership()`**: Callable only by the account whose ID matches `pending_owner`. On success, set `state.owner_id = pending_owner` and clear `pending_owner`.

This ensures the new owner demonstrably controls the target account before the transfer is finalized, eliminating the risk of transferring to a non-existent or inaccessible account. [6](#0-5) 

### Proof of Concept

1. Current owner (`aurora.near`) calls `set_owner` with `new_owner = "auroa.near"` (one-character typo).
2. `set_owner` passes the `require_owner_only` check (caller is `aurora.near` = current owner).
3. `"auroa.near" != "aurora.near"` so `ERR_SAME_OWNER` is not triggered.
4. `state.owner_id` is immediately set to `"auroa.near"` and persisted via `state::set_state`.
5. `aurora.near` is no longer the owner and cannot call any owner-gated function.
6. `"auroa.near"` does not exist; no one can call `resume_contract`, `upgrade`, or any other owner-only method.
7. If the contract is currently paused or is paused by any means before the error is caught, all bridged user funds are permanently frozen with no on-chain recovery path. [7](#0-6) [8](#0-7)
