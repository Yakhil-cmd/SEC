### Title
Operator Fee Lost When `refund_and_commit_fee` Fails After Successful Execution ŌĆö (`basic_bootloader/src/bootloader/transaction_flow/process_transaction.rs`)

---

### Summary

In ZKsync OS, the operator (coinbase) fee for an L2 transaction is not deducted from the sender at validation time and credited to the operator atomically. Instead, the full fee is pre-charged from the sender at validation, and the actual operator payment is deferred to a post-execution step (`refund_and_commit_fee`). If that post-execution step fails and is rolled back, the operator receives no fee despite the transaction having been fully executed and the sender's pre-charged funds being consumed.

---

### Finding Description

The L2 transaction processing pipeline in `process_l2_transaction` follows this sequence:

1. **Validation + precharge**: `precharge_fee` deducts `gas_price * gas_limit` from the sender's balance.
2. **Execution**: `create_frame_and_execute_transaction_payload` runs the transaction body. A revert here is handled gracefully ŌĆö the sender's execution-frame state is rolled back, but the fee pre-charge is **not** reversed.
3. **Refund + operator payment**: `refund_and_commit_fee` (a) refunds unused gas to the sender and (b) credits `gas_price * gas_used` to the coinbase (operator).

The critical issue is in step 3. `refund_and_commit_fee` is wrapped in its own storage frame (`refund_rollback_handle`):

```rust
let refund_rollback_handle = system.start_global_frame()?;

match F::refund_and_commit_fee::<Config>(system, &transaction, &mut tx_context, tracer) {
    Ok(_) => {
        system.finish_global_frame(None)?;
    }
    Err(e) => {
        system_log!(system, "Error on refund {:?}\n", &e);
        system.finish_global_frame(Some(&refund_rollback_handle))?; // <-- rolls back refund AND operator payment
        return Err(e.into());
    }
}
```

Inside `refund_and_commit_fee` for `ZkTransactionFlowOnlyEOA`, the refund to the sender is performed **first**, and the operator payment is performed **second**:

```rust
// Step 1: refund sender
system.io.update_account_nominal_token_balance(..., &refund_recipient, &token_to_refund, false, ...)?;

// Step 2: pay operator
let token_to_pay_operator = U256::from(context.gas_used).checked_mul(gas_price_for_operator)
    .ok_or(internal_error!("gu*gpfo"))?;  // <-- can fail here

system.io.update_account_nominal_token_balance(..., &coinbase, &token_to_pay_operator, false, ...)?;
```

If the `checked_mul` at line 514ŌĆō516 returns `None` (overflow), or if the balance update for the coinbase returns `CantPayOperatorOverflow`, the entire `refund_and_commit_fee` call returns `Err`, and the `refund_rollback_handle` frame is rolled back. This means:

- The sender's refund (step 1) is also rolled back.
- The operator payment (step 2) was never committed.
- The sender's original pre-charge (from `precharge_fee`) **remains consumed** ŌĆö it was committed in a parent frame that is not rolled back.

The net result: the sender loses `gas_price * gas_limit` tokens, the operator receives zero, and the "burned" tokens are permanently lost from circulation.

The same structural issue exists in `EthereumTransactionFlow::refund_and_commit_fee` in `basic_bootloader/src/bootloader/transaction_flow/ethereum/mod.rs` at lines 490ŌĆō654.

The overflow condition `gas_used * gas_price_for_operator` overflowing `U256` is the most realistic trigger. `gas_used` is a `u64` (max ~1.8├Ś10┬╣Ōü╣), and `gas_price_for_operator` is a `U256`. A transaction sender can set `max_fee_per_gas` to a very large value (up to `U256::MAX`). After the `burn_base_fee` subtraction (`gas_price.saturating_sub(base_fee)`), if `gas_price_for_operator` is large enough that `U256::from(gas_used) * gas_price_for_operator` overflows `U256`, the `checked_mul` returns `None` and the function returns an internal error.

---

### Impact Explanation

- The operator (coinbase) receives **zero fee** for a fully executed transaction.
- The sender's pre-charged fee (`gas_price * gas_limit`) is permanently destroyed ŌĆö it was deducted in a committed frame and is never returned.
- This is a **resource accounting bug** and a **public funds-loss path**: tokens are destroyed without being credited to any party.
- The sender suffers a loss of `gas_price * gas_limit` tokens with no corresponding benefit.
- The operator suffers a loss of the fee they were entitled to for executing the transaction.

---

### Likelihood Explanation

The overflow path requires `U256::from(gas_used).checked_mul(gas_price_for_operator)` to overflow. Since `gas_used` is at most `u64::MAX` (~1.8├Ś10┬╣Ōü╣) and `U256` can hold values up to ~1.16├Ś10ŌüĘŌüĘ, the product overflows only when `gas_price_for_operator > U256::MAX / u64::MAX Ōēł 6.4├Ś10ŌüĄŌüĘ`. A sender can craft a transaction with `max_fee_per_gas` set to a value near `U256::MAX`. The validation step checks `gas_price * gas_limit` fits in `U256` (for the pre-charge), but does not check that `gas_used * gas_price_for_operator` fits ŌĆö and `gas_used Ōēż gas_limit`, so the pre-charge check does not bound the operator payment multiplication. This is an attacker-controlled path: any unprivileged transaction sender can set `max_fee_per_gas` to a sufficiently large value.

---

### Recommendation

1. **Validate at transaction admission** that `gas_limit * effective_gas_price` does not overflow `U256`. Since `gas_used Ōēż gas_limit`, this guarantees `gas_used * gas_price_for_operator` also cannot overflow.
2. Alternatively, use `saturating_mul` instead of `checked_mul` for the operator payment computation, treating overflow as a protocol invariant violation rather than a recoverable error.
3. Structurally, consider making the operator payment non-rollbackable (committed in the same frame as the pre-charge), so that a failure in the refund step cannot silently destroy the pre-charged funds.

---

### Proof of Concept

1. Attacker submits an L2 transaction with `max_fee_per_gas = U256::MAX` (or any value where `gas_used * (max_fee_per_gas - base_fee)` overflows `U256`).
2. Validation passes: `precharge_fee` deducts `gas_price * gas_limit` from the sender. This is committed.
3. Transaction executes (may succeed or revert ŌĆö irrelevant).
4. `before_refund` computes `gas_used` and `gas_price`.
5. `refund_and_commit_fee` is called inside a new frame (`refund_rollback_handle`).
6. The sender refund (step 1 inside `refund_and_commit_fee`) succeeds.
7. `U256::from(context.gas_used).checked_mul(gas_price_for_operator)` returns `None` ŌåÆ `internal_error!("gu*gpfo")` is returned.
8. The error path at `process_transaction.rs:178ŌĆō182` rolls back `refund_rollback_handle`, undoing the sender refund.
9. The operator payment was never committed.
10. The sender's pre-charge remains consumed. Tokens are destroyed.
