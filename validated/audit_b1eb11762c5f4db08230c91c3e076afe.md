Audit Report

## Title
Persistent `DownloadChunkError::Overloaded` (HTTP 429) Never Evicts Peer, Enabling Indefinite State Sync Stall — (`rs/p2p/state_sync_manager/src/ongoing.rs`)

## Summary
In `OngoingStateSync::handle_downloaded_chunk_result`, the `Overloaded | Timeout | Cancelled` error arm re-queues the failed chunk via `download_failed` but never calls `active_downloads.remove(&peer_id)`. Because the event loop's only self-termination condition is `active_downloads.is_empty()`, a Byzantine subnet peer that persistently returns HTTP 429 can prevent a lagging replica from ever completing state sync, permanently blocking its participation in consensus.

## Finding Description
**HTTP 429 → `Overloaded` mapping** is explicit in `parse_chunk_handler_response`: [1](#0-0) 

**`Overloaded` handling** re-queues the chunk but does not remove the peer: [2](#0-1) 

**Contrast with `NoContent` and `RequestError`**, which both call `active_downloads.remove(&peer_id)` and decrement `allowed_downloads`: [3](#0-2) 

**`download_failed` is a bare `push`** — no retry counter, no cooldown, no deduplication: [4](#0-3) 

**The loop's only self-exit path** is `active_downloads.is_empty()`: [5](#0-4) 

**`CHUNK_DOWNLOAD_TIMEOUT`** is 10 s, but `Timeout` is handled in the same no-eviction arm as `Overloaded`: [6](#0-5) 

There is no wall-clock deadline anywhere in `ongoing.rs` for the overall sync session. The existing test `test_cancel_if_running` already uses a `MockTransport` that always returns 429 and confirms the loop does not self-terminate — it only exits when `shutdown.shutdown()` is called externally: [7](#0-6) 

## Impact Explanation
A lagging replica whose sole entry in `active_downloads` is a Byzantine peer returning HTTP 429 will spin indefinitely: one 10-second timeout per chunk, 10 parallel slots, re-dispatching the same chunk to the same peer with no progress. The replica cannot advance its certified state, cannot participate in consensus, and is effectively removed from the subnet's active node set for as long as the condition persists. This constitutes a **High** impact: application/platform-level certified-state disruption and subnet availability impact not based on raw volumetric DDoS.

## Likelihood Explanation
Any subnet peer below the consensus fault threshold can trigger this by controlling its own HTTP response codes — no special privileges required. The precondition is that the Byzantine peer is the sole entry in `active_downloads`. This occurs when the Byzantine peer is the first to advertise the target state (triggering `maybe_start_state_sync`) and honest peers are slow to connect or the network is degraded. `ADVERT_BROADCAST_INTERVAL` is 5 seconds: [8](#0-7) 

In normal operation honest peers will advertise within seconds, but the window is real and exploitable in degraded-network or targeted-isolation scenarios. Once the Byzantine peer is the sole peer, the condition is self-sustaining indefinitely.

## Recommendation
1. **Evict on repeated `Overloaded`/`Timeout`**: Track a consecutive-failure counter per peer in `active_downloads`; after a configurable threshold (e.g., 5), call `active_downloads.remove(&peer_id)` and decrement `allowed_downloads`.
2. **Exponential backoff**: Place failed chunks in a time-delayed retry queue instead of immediately re-queuing via `download_failed`, preventing CPU spin.
3. **Overall sync timeout**: Add a wall-clock deadline inside `OngoingStateSync::run` (e.g., 10 minutes) after which the sync is cancelled and the `Chunkable` object is dropped, allowing a fresh attempt with a new peer set.

## Proof of Concept
The existing `test_cancel_if_running` test already demonstrates the behavior. A minimal extension to prove the liveness violation without external shutdown:

```rust
#[test]
fn test_overloaded_peer_never_evicted() {
    with_test_replica_logger(|log| {
        let mut t = MockTransport::default();
        // Byzantine peer always returns 429
        t.expect_rpc().returning(|_, _| {
            Ok(Response::builder()
                .status(StatusCode::TOO_MANY_REQUESTS)
                .body(compress_empty_bytes())
                .unwrap())
        });
        let mut c = MockChunkable::<TestMessage>::default();
        c.expect_chunks_to_download()
            .returning(|| Box::new(std::iter::once(ChunkId::from(1))));

        let rt = Runtime::new().unwrap();
        let ongoing = start_ongoing_state_sync(
            log,
            rt.handle(),
            OngoingStateSyncMetrics::new(&MetricsRegistry::default()),
            Arc::new(Mutex::new(Box::new(c))),
            StateSyncArtifactId { height: Height::from(1), hash: CryptoHash(vec![]) },
            Arc::new(t),
        );

        rt.block_on(async move {
            ongoing.sender.send(NODE_1).await.unwrap();
            // No shutdown call — assert task is still running after 60 seconds
            tokio::time::sleep(Duration::from_secs(60)).await;
            assert!(!ongoing.shutdown.completed(),
                "State sync should NOT have self-terminated");
        });
    });
}
```

Without `shutdown.shutdown()`, the task never exits, confirming the liveness violation. Metrics will show the same chunk attempted hundreds of times against the same peer.
