Audit Report

## Title
Unbounded DKG Unvalidated Pool Growth via Byzantine P2P Peer — (`rs/artifact_pool/src/dkg_pool.rs`)

## Summary
`DkgPoolImpl::insert` performs an unconditional `BTreeMap` insertion with no capacity bound. The DKG P2P channel is configured with `SLOT_TABLE_NO_LIMIT = usize::MAX`, disabling the per-peer slot table guard that protects other artifact types. The `DkgBouncer` accepts any message whose `id.height` equals the current DKG interval start height without checking dealer membership or signature validity. A single Byzantine subnet node can flood the unvalidated pool with an unbounded number of syntactically valid, height-correct DKG messages, exhausting replica heap memory before the next purge cycle.

## Finding Description

**Root cause 1 — No capacity check in `DkgPoolImpl::insert`.**
`DkgPoolImpl::insert` delegates directly to `PoolSection::insert`, which wraps an unbounded `BTreeMap`. There is no per-peer quota, no total pool size limit, and no rejection path. [1](#0-0) [2](#0-1) 

**Root cause 2 — DKG slot table limit is `usize::MAX`.**
`SLOT_TABLE_NO_LIMIT` is set to `usize::MAX` and passed to the DKG broadcast channel. The per-peer slot table guard at `receiver.rs:373` (`peer_slot_table_len < self.slot_limit`) is therefore never triggered for DKG. Ingress, by contrast, uses `SLOT_TABLE_LIMIT_INGRESS = 50_000`. [3](#0-2) [4](#0-3) [5](#0-4) 

**Root cause 3 — `DkgBouncer` only checks block height.**
The bouncer returns `BouncerValue::Wants` for any `DkgMessageId` whose `height` equals `dkg_pool.get_current_start_height()`. It does not check dealer membership, signature validity, or dealing well-formedness. A Byzantine node trivially satisfies this check by using the correct `start_block_height`. [6](#0-5) 

**Root cause 4 — `DkgMessageId` is a crypto hash of the full message.**
Each distinct `NiDkgDealing` payload produces a unique `DkgMessageId`. A Byzantine node can generate an unbounded stream of unique IDs by varying the dealing bytes, each occupying a distinct slot number and a distinct pool entry. [7](#0-6) 

**Root cause 5 — Validator processes only one message per `(dealer, DKG ID)` group per cycle, and remote-DKG messages with unknown configs are never removed.**
`on_state_change` groups unvalidated messages by `(signer, dkg_id)` and calls `validate_dealings_for_dealer` with only `messages.first()`. For messages referencing a remote DKG ID not yet in the state, the validator returns `Mutations::new()` — the message is silently deferred and never removed until the next purge. This means a Byzantine node can craft messages with `target_subnet = Remote(arbitrary_id)` at the current interval height, and they accumulate indefinitely. [8](#0-7) [9](#0-8) 

**Root cause 6 — Purge only removes messages from previous intervals.**
`purge` removes entries with `id.height < height` (strictly less than). Messages at the current interval height accumulate until the next summary block advances the interval. [10](#0-9) 

**Exploit flow:**
1. Byzantine node (registered subnet member, valid TLS cert) connects to honest replicas.
2. For each `i` in `0..N`, it sends a slot update on slot number `i` carrying a unique `DkgMessageId` derived from a distinct `NiDkgDealing` payload, with `height == current_start_height`.
3. `DkgBouncer` returns `Wants` for each (height matches).
4. `SLOT_TABLE_NO_LIMIT` means all N slot entries are accepted into the per-peer slot table.
5. Each triggers artifact download and `DkgPoolImpl::insert` — no capacity check, pool grows by N entries.
6. For remote-DKG-targeted messages, the validator defers them indefinitely; for local-DKG messages, the validator removes only one per group per cycle, so cleanup lags far behind insertion rate.
7. Replica heap is exhausted; OOM kill halts consensus participation.

## Impact Explanation

A single Byzantine subnet node within the IC f-fault model (f out of 3f+1) can exhaust the heap of every honest replica it is connected to. `NiDkgDealing` messages contain large cryptographic blobs (ciphertext arrays sized to the number of receivers). Sending 10^4–10^5 unique messages at the current DKG interval height will grow the unvalidated pool to gigabytes before the next summary block. The replica process is killed by the OS OOM killer, halting its participation in consensus. If enough replicas are affected simultaneously, the subnet stalls.

This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS.**

## Likelihood Explanation

The attacker must be a registered subnet node with a valid TLS certificate — not a completely external party. However, a single compromised node is within the IC Byzantine fault model. The attack requires no special preconditions beyond being a P2P peer at the current DKG interval. The `DkgBouncer` height check is trivially satisfied by using the correct `start_block_height`. The slot table imposes no limit. The attack is mechanically straightforward and locally testable without touching mainnet.

## Recommendation

1. **Add a per-peer slot limit for DKG** analogous to `SLOT_TABLE_LIMIT_INGRESS`. Replace `SLOT_TABLE_NO_LIMIT` with a registry-derived constant (e.g., `num_dealers * num_dkg_configs * small_multiplier`). [11](#0-10) 

2. **Add a capacity check in `DkgPoolImpl::insert`** that rejects insertions beyond a computed bound (e.g., `num_subnet_nodes * num_active_dkg_configs * MAX_DEALINGS_PER_DEALER`). [1](#0-0) 

3. **Extend `DkgBouncer` to check dealer membership** against the current DKG summary before returning `BouncerValue::Wants`, so messages from non-dealers are dropped at the P2P layer before pool insertion. [6](#0-5) 

4. **Remove deferred remote-DKG messages** after a bounded number of `on_state_change` cycles rather than deferring them indefinitely. [9](#0-8) 

## Proof of Concept

A deterministic integration test can be written against `DkgPoolImpl` and `ConsensusManagerReceiver` directly:

```rust
// 1. Instantiate DkgPoolImpl with current_start_height = H.
// 2. Instantiate ConsensusManagerReceiver with slot_limit = usize::MAX (SLOT_TABLE_NO_LIMIT).
// 3. For i in 0..100_000:
//    a. Craft a DKG message with start_block_height = H and dealing bytes = [i as u8; N].
//    b. Compute DkgMessageId = crypto_hash(msg) — unique per i.
//    c. Send SlotUpdate { slot_number: i, commit_id: i, update: Update::Id(id) } from a single peer.
//    d. Verify slot table grows without bound (no limit exceeded metric fires).
//    e. Call DkgPoolImpl::insert for each downloaded artifact.
// 4. Assert pool.get_unvalidated().count() == 100_000 — no rejection occurred.
// 5. Assert no ChangeAction removes more than 1 message per on_state_change call
//    when all messages share the same (signer, dkg_id) group.
```

This test requires no mainnet interaction and no cryptographic keys — `dummy_dealing(i)` from `ic_crypto_test_utils_ni_dkg` produces distinct dealing bytes per index, and `BasicSignature::fake(node_id)` is sufficient to populate the pool (signature verification only occurs in `on_state_change`, not in `insert`).     messages: BTreeMap<K, V>,

impl<K: Ord, V: HasLabel> PoolSection<K, V> {
    pub(crate) fn new(metrics_registry: MetricsRegistry, pool: &str, pool_type: &str) -> Self {
        Self {
            messages: Default::default(),
            metrics: PoolMetrics::new(metrics_registry, pool, pool_type),
        }
    }

    pub(crate) fn insert(&mut self, key: K, value: V) -> Option<V> {
        self.metrics
            .observe_insert(MESSAGE_SIZE_BYTES, value.label());
        let replaced = self.messages.insert(key, value);
        if let Some(replaced) = &replaced {
            self.metrics
                .observe_duplicate(MESSAGE_SIZE_BYTES, replaced.label());
        }
        replaced
    }

impl<Pool: DkgPool> BouncerFactory<DkgMessageId, Pool> for DkgBouncer {
    fn new_bouncer(&self, dkg_pool: &Pool) -> Bouncer<DkgMessageId> {
        let _timer = self.metrics.update_duration.start_timer();

        let start_height = dkg_pool.get_current_start_height();
        Box::new(move |id| {
            use std::cmp::Ordering;
            match id.height.cmp(&start_height) {
                Ordering::Equal => BouncerValue::Wants,
                Ordering::Greater => BouncerValue::MaybeWantsLater,
                Ordering::Less => BouncerValue::Unwanted,
            }
        })
    }

    fn refresh_period(&self) -> std::time::Duration {
        std::time::Duration::from_secs(3)
    }
}
