Audit Report

## Title
Unbounded `ingress_expiries` Allocation via Attacker-Controlled `ingress_end` in `construction_payloads` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The ICP Rosetta API's `construction_payloads` handler accepts attacker-controlled `ingress_start` and `ingress_end` nanosecond timestamps from request metadata and feeds them directly into an unbounded `while` loop that pushes entries into a heap-allocated `Vec`. No validation caps the window size before allocation begins. A single small HTTP POST can force the Rosetta process to exhaust heap memory and be OOM-killed, denying service to all users of that Rosetta instance.

## Finding Description
The interval is computed at [1](#0-0)  as `MAX_INGRESS_TTL(300s) - PERMITTED_DRIFT(60s) - 120s = 120s = 1.2×10¹¹ ns` (the claim states 180s but the actual subtraction of 120s yields 120s, making the attack *worse* than described).

`ingress_start` and `ingress_end` are taken verbatim from attacker-supplied JSON metadata with no range or delta validation: [2](#0-1) 

The loop then runs with no upper-bound guard: [3](#0-2) 

After `ingress_expiries` is built, `add_payloads` iterates over it and allocates two `SigningPayload` heap objects per expiry per operation: [4](#0-3) 

The server's only body-size guard is a 4 MB JSON limit on the *request*: [5](#0-4)  This is irrelevant — the malicious payload is two small integers, well under 100 bytes.

The ICRC1 variant has partial validation (checks `ingress_start >= ingress_end` and a minimum window): [6](#0-5)  but still does not cap the window size. The ICP Rosetta variant has **no validation at all**.

`MAX_INGRESS_TTL` and `PERMITTED_DRIFT` are fixed constants: [7](#0-6) 

## Impact Explanation
With `ingress_end = ingress_start + u64::MAX/2` (≈ 9.2×10¹⁸ ns):
- Loop iterations: `9.2×10¹⁸ / 1.2×10¹¹ ≈ 76.7 million` (more than the claim's estimate due to the correct 120s interval)
- `ingress_expiries` vec alone: ~614 MB of `u64` entries
- `add_payloads` then allocates 2 `SigningPayload` structs per expiry (each containing hex-encoded message IDs): tens of GB

This matches the allowed ICP bounty impact: **High ($2,000–$10,000) — Application/platform-level DoS, crash** of the Rosetta node. The Rosetta API is explicitly in-scope under "Financial integrations: ICP/ICRC ledgers, Rosetta API."

## Likelihood Explanation
The `/construction/payloads` endpoint is unauthenticated and publicly reachable. The attack requires a single HTTP POST with a ~200-byte JSON body. No key, token, or privileged role is needed. The attack is repeatable — if the process is restarted, a new request immediately re-triggers it. Any external user can execute this.

## Recommendation
Add a window-size guard immediately after parsing `ingress_start`/`ingress_end`, before the loop:

```rust
let max_window = ic_limits::MAX_INGRESS_TTL * 100; // cap at 100 intervals
if ingress_end.saturating_sub(ingress_start.as_nanos_since_unix_epoch())
    > max_window.as_nanos() as u64
{
    return Err(ApiError::invalid_request("ingress window too large"));
}
```

Alternatively, break out of the loop after a fixed maximum number of iterations (e.g., 100), consistent with the ICRC1 variant's intent and practical retry windows.

## Proof of Concept
```http
POST /construction/payloads HTTP/1.1
Content-Type: application/json

{
  "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
  "operations": [{"operation_identifier":{"index":0},"type":"TRANSACTION","account":{"address":"<valid_addr>"},"amount":{"value":"-1","currency":{"symbol":"ICP","decimals":8}}}],
  "public_keys": [{"hex_bytes":"<valid_pubkey>","curve_type":"edwards25519"}],
  "metadata": {
    "ingress_start": 1000000000000000000,
    "ingress_end":   9223372036854775807
  }
}
```
The server enters the `while now < ingress_end` loop and allocates ~76.7 million `u64` entries into `ingress_expiries`, then attempts to build `SigningPayload` objects for each, exhausting heap memory and triggering OOM termination of the Rosetta process.
