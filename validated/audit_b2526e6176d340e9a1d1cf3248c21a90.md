### Title
Permissionless `deploy_erc20_token` + Unvalidated `ft_on_transfer` Caller Enables Unbacked ERC-20 Minting - (File: `engine/src/contract_methods/connector.rs`, `engine/src/lib.rs`)

---

### Summary

The Aurora Engine's `deploy_erc20_token` entrypoint has no access control, allowing any NEAR account to register an arbitrary NEP-141 account ID. Combined with `ft_on_transfer` not verifying that actual NEP-141 tokens were transferred before minting ERC-20 tokens, an attacker can deploy a fake NEP-141 contract, register it permissionlessly, and then call `ft_on_transfer` directly on Aurora to mint unlimited ERC-20 tokens without depositing any real tokens.

---

### Finding Description

**Root cause 1 — Permissionless `deploy_erc20_token`:**

`engine/src/lib.rs` exposes `deploy_erc20_token` with no caller restriction:

```rust
pub extern "C" fn deploy_erc20_token() {
    let io = Runtime;
    let env = Runtime;
    let mut handler = Runtime;
    contract_methods::connector::deploy_erc20_token(io, &env, &mut handler)
        .map_err(ContractError::msg)
        .sdk_unwrap();
}
```

The inner function in `engine/src/contract_methods/connector.rs` only calls `require_running` — no `require_owner_only` or any other caller check:

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(...) {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;   // only check: not paused
        ...
        DeployErc20TokenArgs::Legacy(nep141) => {
            let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;
```

Any NEAR account can call this and register any `AccountId` as a NEP-141 token, deploying a corresponding ERC-20 in Aurora's EVM state and writing the mapping into `nep141_erc20_map`.

**Root cause 2 — `ft_on_transfer` trusts the registry without verifying actual token receipt:**

`engine/src/contract_methods/connector.rs` dispatches on `predecessor_account_id`:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
```

`receive_erc20_tokens` in `engine/src/engine.rs` only checks that the predecessor is in the registry:

```rust
let erc20_token = get_erc20_from_nep141(&self.io, token)?;
```

If the lookup succeeds, it immediately mints ERC-20 tokens to the recipient via an EVM `mint` call — with no verification that Aurora actually received any NEP-141 tokens from the predecessor.

---

### Impact Explanation

An attacker who controls `fake_token.near` and has registered it via `deploy_erc20_token` can call `ft_on_transfer` on Aurora directly (with `predecessor = fake_token.near`, arbitrary `amount`, and their EVM address as `msg`). Aurora will mint that arbitrary amount of ERC-20 tokens to the attacker's EVM address with zero NEP-141 backing. The ERC-20 supply for that token on Aurora becomes permanently unbacked, constituting **insolvency** for that token's bridge representation. If the minted ERC-20 is accepted by any DeFi protocol deployed on Aurora (AMMs, lending markets), the attacker can drain real value from those protocols.

**Impact class: Critical — Insolvency / ERC-20 mirror accounting bug.**

---

### Likelihood Explanation

- `deploy_erc20_token` is callable by any NEAR account with no permission check.
- `ft_on_transfer` is a public NEAR contract method callable by any account.
- The attacker only needs to deploy a minimal NEAR contract and make two calls.
- No admin compromise, oracle error, or social engineering is required.

**Likelihood: High.**

---

### Recommendation

1. **Add access control to `deploy_erc20_token`**: Restrict it to the contract owner or a governance-approved whitelist, analogous to how `set_eth_connector_contract_account` uses `require_owner_only`.

2. **Validate the caller of `ft_on_transfer`**: Maintain a whitelist of approved NEP-141 account IDs (those registered through the controlled `deploy_erc20_token` path). In `ft_on_transfer`, reject calls from any `predecessor_account_id` not on that whitelist before minting.

3. **Checks-effects-interactions ordering**: Ensure that any state change (minting) only occurs after confirming the token transfer actually happened (i.e., the predecessor legitimately called `ft_transfer_call` on its own contract, not a direct `ft_on_transfer` invocation).

---

### Proof of Concept

1. Attacker deploys `fake_token.near` — a minimal NEAR contract that can call arbitrary methods on other contracts.

2. Attacker calls `deploy_erc20_token` on Aurora Engine with `fake_token.near` as the NEP-141 account ID. No access control blocks this. [1](#0-0) [2](#0-1) 

3. Aurora registers `fake_token.near → <erc20_address>` in `nep141_erc20_map` and deploys the ERC-20 contract. [3](#0-2) 

4. From `fake_token.near`, attacker calls `ft_on_transfer` on Aurora with:
   - `sender_id`: any account
   - `amount`: `1_000_000_000_000_000_000` (arbitrary large value)
   - `msg`: attacker's EVM address (hex-encoded)

5. Aurora's `ft_on_transfer` sees `predecessor_account_id = fake_token.near`, which is not the connector account, so it calls `receive_erc20_tokens`. [4](#0-3) 

6. `receive_erc20_tokens` calls `get_erc20_from_nep141` — which succeeds because the token is registered — and immediately mints the full `amount` of ERC-20 tokens to the attacker's EVM address. [5](#0-4) 

7. Attacker now holds `1_000_000_000_000_000_000` ERC-20 tokens on Aurora with zero NEP-141 backing. The bridge is insolvent for this token.
