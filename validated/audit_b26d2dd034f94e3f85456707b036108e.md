### Title
Missing Caller Restriction on SNS Treasury Manager `deposit`/`withdraw` Enables Proactive Sandwich Attacks - (File: rs/sns/treasury_manager/treasury_manager.did)

### Summary
The SNS Treasury Manager's `deposit` and `withdraw` functions are publicly callable with no caller restriction. Any unprivileged principal can trigger DEX interactions at arbitrary times, enabling proactive sandwich attacks on SNS treasury assets. The DID itself explicitly acknowledges the slippage risk this creates.

### Finding Description
The treasury manager DID defines `deposit` and `withdraw` as unrestricted public update functions:

```candid
service : (TreasuryManagerArg) -> {
  deposit : (DepositRequest) -> (Result);
  withdraw : (WithdrawRequest) -> (Result);
  ...
}
```

The DID explicitly acknowledges the resulting risk at lines 35–40:

> "Known Security Risks: Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved."

The governance canister executes a two-step flow in `execute_treasury_manager_deposit` (`rs/sns/governance/src/extensions.rs`):
1. `approve_treasury_manager` — sets ICRC-2 allowances on the ledger for the treasury manager canister.
2. `call_canister(extension_canister_id, "deposit", arg_blob)` — calls `deposit` on the treasury manager.

Because these are two separate async inter-canister calls, there is a window between step 1 and step 2 during which the ICRC-2 allowances exist on the ledger but `deposit` has not yet been called by governance. Any external principal can observe the allowance (via a ledger query) and call `deposit` first.

The `TreasuryManager` trait (`rs/sns/treasury_manager/src/lib.rs`) also imposes no caller restriction, so every conforming implementation inherits this exposure.

The `WithdrawRequest` additionally accepts an optional `withdraw_accounts` field mapping ledger canister IDs to arbitrary `Account` values, meaning an attacker who calls `withdraw` can potentially redirect funds to an account of their choosing, depending on the implementation.

### Impact Explanation
**Governance authorization bug / cycles-resource accounting bug.** An attacker can:
- Front-run governance's `deposit` call, consuming the ICRC-2 allowance and triggering the DEX deposit at a time of the attacker's choosing (e.g., after manipulating the DEX liquidity pool).
- Cause the SNS treasury to receive fewer LP tokens or assets than expected, permanently reducing treasury value.
- Exhaust the allowance so governance's own `deposit` call fails, requiring a new governance proposal to retry.
- Call `withdraw` with attacker-controlled `withdraw_accounts` to redirect treasury assets.

### Likelihood Explanation
Realistic. On the IC, DEXs are canisters. A malicious canister can atomically: (a) manipulate the DEX pool, (b) call `deposit` on the treasury manager (consuming the governance-approved ICRC-2 allowance), and (c) restore the pool — all within a single execution round or across consecutive rounds. The attacker only needs to poll the ledger's `icrc2_allowance` query endpoint to detect when governance has set up the allowance, then act before governance's own `deposit` call is processed.

### Recommendation
Add an explicit caller restriction to both `deposit` and `withdraw` in the treasury manager DID and in the `TreasuryManager` trait. Only the SNS governance canister that registered the treasury manager extension should be permitted to invoke these functions. For example, the canister implementation should check `ic_cdk::api::msg_caller() == sns_governance_canister_id` at the top of each handler and trap otherwise.

### Proof of Concept
1. An SNS governance proposal executes `execute_treasury_manager_deposit` in `rs/sns/governance/src/extensions.rs` (line 1546).
2. Governance calls `approve_treasury_manager`, which sets ICRC-2 allowances on the SNS and ICP ledgers for the treasury manager canister (line 1567–1573).
3. Attacker polls `icrc2_allowance` on the ledger and detects the new allowance.
4. Attacker's malicious canister calls the DEX to skew the pool price.
5. Attacker calls `deposit(DepositRequest { allowances: [...] })` directly on the treasury manager canister — no privileged access required.
6. Treasury manager uses the governance-approved ICRC-2 allowance to pull funds from governance and deposit them into the DEX at the manipulated price.
7. Attacker restores the DEX pool and profits; governance's subsequent `deposit` call (line 1576–1578) fails because the allowance is exhausted, requiring a new governance proposal. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

/// Execute a treasury manager deposit operation
async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())
}
