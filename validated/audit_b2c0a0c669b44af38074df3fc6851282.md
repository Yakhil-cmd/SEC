### Title
Unchecked Exit-Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Burn Without Bridging - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` irreversibly burn user tokens **before** calling the Aurora exit precompile, and never inspect the low-level `call` return value. If the precompile call fails for any reason (paused precompile, missing connector configuration, missing NEP-141 mapping, etc.), the EVM tokens are permanently destroyed while no NEAR or Ethereum-side assets are ever released. The transaction completes with `TransactionStatus::Succeed`, giving the caller no indication that the bridge leg failed.

---

### Finding Description

In `EvmErc20.sol`, both withdrawal functions follow the same unsafe pattern:

```solidity
// withdrawToNear  (lines 53-63)
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here, irreversibly

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — failure is silently swallowed
    }
}

`EvmErc20V2.sol` contains the identical unchecked pattern in both functions. [3](#0-2) 

The `ExitToNear` precompile (`0xe9217bc7…`) and `ExitToEthereum` precompile (`0xb0bd02f6…`) are registered in the `Precompiles` set and can return failure (`ExitError`) for multiple reasons:

- **Precompile paused**: `Precompiles::execute` returns `ERR_PAUSED` when the address is in `paused_precompiles`. [4](#0-3) 
- **Missing eth-connector account**: `get_eth_connector_contract_account` returns `ERR_KEY_NOT_FOUND` if the key is absent from storage. [5](#0-4) 
- **Missing NEP-141 mapping**: `get_nep141_from_erc20` returns `ERR_TARGET_TOKEN_NOT_FOUND` if no mapping exists for the calling ERC-20 address. [6](#0-5) 
- **Static/delegate context guard**: the precompile reverts if called in a delegatecall context. [7](#0-6) 

In every failure case the EVM `call` opcode returns `0` in `res`. Because `res` is never tested, the Solidity function returns normally. The `_burn` that already executed is not rolled back (it is not inside a `try/catch` or preceded by a `require`), so the tokens are gone.

---

### Impact Explanation

**Critical — Permanent freezing / destruction of user funds.**

A user who calls `withdrawToNear` or `withdrawToEthereum` while the precompile is in a failing state loses their bridged ERC-20 tokens permanently. The tokens are burned on the Aurora EVM side, but the corresponding NEP-141 `ft_transfer` (or Ethereum-side withdrawal) promise is never scheduled. There is no refund path: the `error_refund` callback mechanism lives inside the precompile itself and is never reached when the precompile call fails at the EVM level.

---

### Likelihood Explanation

**Medium.** The precompile fails silently under at least three independently reachable conditions:

1. An admin legitimately pauses the `exitToNear` or `exitToEthereum` precompile (a documented operational capability via `PrecompileFlags`). Any user who calls `withdrawToNear` during the pause window loses tokens. [8](#0-7) 
2. An `EvmErc20` token is deployed (e.g., via `adminDelegatecall` or a factory) without a corresponding NEP-141 entry in the `Erc20Nep141Map` storage. Any withdrawal attempt immediately hits `ERR_TARGET_TOKEN_NOT_FOUND` and silently burns tokens.
3. The engine is deployed or migrated in a state where `EthConnectorStorageId::EthConnectorAccount` is absent, causing `ERR_KEY_NOT_FOUND` on every withdrawal.

All three conditions are reachable without private-key compromise.

---

### Recommendation

Check the return value of the low-level `call` and revert on failure, so the `_burn` is rolled back:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    bool success;
    assembly {
        success := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
    }
    require(success, "exitToNear precompile call failed");
}
```

Apply the same fix to `withdrawToEthereum` and to both functions in `EvmErc20V2.sol`.

---

### Proof of Concept

1. Deploy an `EvmErc20` token on Aurora (standard bridge flow).
2. Pause the `exitToNear` precompile via the engine's admin interface (sets the address in `paused_precompiles`).
3. As an ordinary token holder, call `withdrawToNear(recipientBytes, amount)`.
4. **Observed**: the transaction succeeds; the caller's ERC-20 balance decreases by `amount`; no NEP-141 `ft_transfer` promise is ever created; the tokens are permanently gone.
5. **Expected**: the transaction should revert, preserving the caller's balance.

The same scenario applies to `withdrawToEthereum` when `exitToEthereum` is paused, and to both functions when the NEP-141 mapping or connector account is absent from storage.

bitflags! {
    /// Wraps unsigned integer where each bit identifies a different precompile.
    #[derive(BorshSerialize, BorshDeserialize, Default)]
    #[borsh(crate = "aurora_engine_types::borsh")]
    pub struct PrecompileFlags: u32 {
        const EXIT_TO_NEAR        = 0b01;
        const EXIT_TO_ETHEREUM    = 0b10;
    }
}

impl PrecompileFlags {
    #[must_use]
    pub fn from_address(address: &Address) -> Option<Self> {
        Some(if address == &exit_to_ethereum::ADDRESS {
            Self::EXIT_TO_ETHEREUM
        } else if address == &exit_to_near::ADDRESS {
            Self::EXIT_TO_NEAR
        } else {
            return None;
        })
    }

    /// Checks if the precompile belonging to the `address` is marked as paused.
    #[must_use]
    pub fn is_paused_by_address(&self, address: &Address) -> bool {
        Self::from_address(address).is_some_and(|precompile_flag| self.contains(precompile_flag))
    }
