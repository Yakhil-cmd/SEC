Audit Report

## Title
Unbounded `ingress_end` in `construction_payloads` Enables Single-Request OOM/CPU DoS — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary
The `construction_payloads` function accepts an attacker-controlled `ingress_end` value from the unauthenticated POST `/construction/payloads` endpoint with no upper-bound validation. Setting `ingress_end` to `u64::MAX` causes the loop that builds `ingress_expiries` to execute approximately 139 million iterations, allocating ~1.1 GB of heap memory and blocking the Axum worker thread, crashing or hanging the Rosetta process.

## Finding Description
In `construction_payloads` at [1](#0-0) , the loop iterates while `ingress_start < ingress_end`, advancing by `ingress_interval - INGRESS_INTERVAL_OVERLAP` per step. With `ingress_interval = (MAX_INGRESS_TTL - PERMITTED_DRIFT).as_nanos()` ≈ 240,000,000,000 ns [2](#0-1)  and `INGRESS_INTERVAL_OVERLAP = Duration::from_secs(120)` = 120,000,000,000 ns [3](#0-2) , the effective step is 120 seconds in nanoseconds.

The only pre-loop guards check that `ingress_start < ingress_end` and `ingress_end >= now + ingress_interval` [4](#0-3) . Both are trivially satisfied by supplying `ingress_start = now` and `ingress_end = u64::MAX`. There is no cap on the number of iterations or the size of the resulting `ingress_expiries` vector.

With `now` ≈ 1.7 × 10¹⁸ ns, the range `u64::MAX - now` ≈ 1.67 × 10¹⁹ ns divided by a step of 1.2 × 10¹¹ ns yields ~139 million iterations. Each iteration pushes a `u64` (8 bytes), totaling ~1.1 GB of heap allocation. The function is synchronous (not `async`), so it blocks the Axum worker thread for the entire duration.

The endpoint is registered with no authentication middleware [5](#0-4) .

## Impact Explanation
A single unauthenticated HTTP request crashes or hangs the ICRC-1 Rosetta process via OOM or CPU exhaustion. This halts block synchronization and transaction submission for all ICRC-1 tokens served by that Rosetta instance. This matches the allowed High impact: **Application/platform-level DoS** affecting a Rosetta financial integration component with concrete user and protocol harm.

## Likelihood Explanation
The endpoint is public, requires no credentials, and the malicious field is a plain JSON integer. The attack requires a single HTTP request from any network-reachable client. No special privileges, victim interaction, or preconditions are needed. The attack is trivially repeatable.

## Recommendation
Add an explicit cap on the number of generated expiries before entering the loop:

```rust
const MAX_INGRESS_EXPIRIES: usize = 100;
let step = ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
if step > 0 && ingress_end.saturating_sub(ingress_start) / step > MAX_INGRESS_EXPIRIES as u64 {
    return Err(Error::processing_construction_failed(&"ingress window too large"));
}
```

Alternatively, enforce `ingress_end <= now + MAX_ALLOWED_WINDOW` before the loop, where `MAX_ALLOWED_WINDOW` is a reasonable constant (e.g., a few hours).
