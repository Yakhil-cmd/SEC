### Title
Permanently Stuck ckERC20 Withdrawal Requests When Gas Prices Spike After Token Burns - (File: `rs/ethereum/cketh/minter/src/withdraw.rs`)

### Summary

The `withdraw_erc20` endpoint in the ckETH minter burns both ckETH (for gas fees) and ckERC20 tokens before queuing the withdrawal. The gas fee is estimated at call time. If Ethereum gas prices spike significantly after the withdrawal is accepted, `create_transaction` fails with `InsufficientTransactionFee` and the request is silently rescheduled to the back of the queue indefinitely. No reimbursement path exists for this case, leaving the user's already-burned ckETH and ckERC20 tokens permanently stuck with no cancellation mechanism.

### Finding Description

In `rs/ethereum/cketh/minter/src/main.rs`, the `withdraw_erc20` function:

1. Estimates the current ERC-20 transaction fee via `estimate_erc20_transaction_fee()` [1](#0-0) 
2. Burns `erc20_tx_fee` ckETH from the user's account [2](#0-1) 
3. Burns `ckerc20_withdrawal_amount` ckERC20 from the user's account [3](#0-2) 
4. Records the `Erc20WithdrawalRequest` with `max_transaction_fee: erc20_tx_fee` [4](#0-3) 

Later, in `create_transactions_batch` in `rs/ethereum/cketh/minter/src/withdraw.rs`, when the minter tries to create the Ethereum transaction, `create_transaction` checks whether the stored `max_transaction_fee` covers the current gas price:

```rust
if actual_min_max_fee_per_gas > request_max_fee_per_gas {
    return Err(CreateTransactionError::InsufficientTransactionFee { ... });
}
``` [5](#0-4) 

When this error occurs, the handler in `create_transactions_batch` only reschedules the request — it does **not** reimburse the user:

```rust
Err(CreateTransactionError::InsufficientTransactionFee { ... }) => {
    log!(...);
    mutate_state(|s| s.eth_transactions.reschedule_withdrawal_request(request));
}
``` [6](#0-5) 

The `maybe_reimburse` set — which triggers the reimbursement flow — is only populated when a transaction is **successfully created** via `record_created_transaction`, not when it is rescheduled: [7](#0-6) 

This means a ckERC20 withdrawal request that repeatedly fails `create_transaction` due to a gas price spike will cycle in the queue indefinitely. The user's ckETH and ckERC20 are already burned, and there is no timeout, cancellation, or reimbursement mechanism for this state.

By contrast, `withdraw_eth` has an upfront `minimum_withdrawal_amount` guard that prevents accepting a withdrawal that cannot cover fees: [8](#0-7) 

No equivalent guard exists in `withdraw_erc20` for the case where the fee estimate becomes stale.

### Impact Explanation

- **Chain-fusion burn/stuck bug**: Both ckETH (gas fee) and ckERC20 tokens are burned at acceptance time. If gas prices spike beyond the safety margin, the Ethereum transaction can never be created.
- The withdrawal request loops in `pending_withdrawal_requests` indefinitely with no user-visible error and no recovery path.
- The user has no mechanism to cancel the request or reclaim their burned tokens.
- The `eip1559_transaction_price` query endpoint exists for users to estimate fees, but the minter internally controls the fee used — the user cannot influence it.

### Likelihood Explanation

- Moderate. The fee estimate includes a safety margin for resubmissions (~10% per bump), but extreme gas price spikes (e.g., during Ethereum network congestion events) can exceed this margin.
- Historical precedent exists: the ckBTC minter had real stuck transactions due to an extremely low fee per vbyte being chosen, requiring an emergency upgrade (see `rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md`). [9](#0-8) 
- Any unprivileged user calling `withdraw_erc20` during a period of low gas prices that is followed by a spike is affected.

### Recommendation

1. **Add a reimbursement path for rescheduled ckERC20 requests**: After N consecutive failed `create_transaction` attempts (or after a configurable timeout), automatically reimburse the user's burned ckETH and ckERC20 tokens via the existing reimbursement infrastructure.
2. **Add a maximum queue age**: Track the timestamp of each `Erc20WithdrawalRequest` and trigger reimbursement if the request has been pending longer than a threshold (e.g., 24 hours).
3. **Expose a user-facing cancellation endpoint**: Allow the original caller to cancel a pending ckERC20 withdrawal request and receive a reimbursement, analogous to the ckBTC `cancel_retrieve_btc` pattern.

### Proof of Concept

1. User calls `withdraw_erc20` during a period of low Ethereum gas prices (e.g., 10 gwei base fee).
2. Minter estimates `erc20_tx_fee` at ~10 gwei × 65,000 gas = 650,000 gwei and burns ckETH and ckERC20 from the user.
3. Ethereum gas prices spike to 200 gwei (e.g., during a major NFT mint or network event).
4. `create_transactions_batch` calls `create_transaction` for the queued request.
5. `actual_min_max_fee_per_gas > request_max_fee_per_gas` → `InsufficientTransactionFee` is returned.
6. The request is rescheduled to the back of the queue. [6](#0-5) 
7. Steps 4–6 repeat on every timer tick. The user's ckETH and ckERC20 remain burned. No reimbursement is ever triggered because `maybe_reimburse` is never populated for this request. [7](#0-6) 
8. The user has no way to recover their tokens unless gas prices drop back below the original estimate.
