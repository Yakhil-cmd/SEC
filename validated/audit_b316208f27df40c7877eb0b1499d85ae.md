Audit Report

## Title
SNS Swap Validation Uses Direct-Only ICP Cap While Distribution Divides by Total (Direct + Neurons' Fund) ICP — (`rs/sns/init/src/lib.rs`, `rs/sns/swap/src/swap.rs`)

## Summary
The SNS initialization validation at `rs/sns/init/src/lib.rs:1636–1638` computes the minimum SNS tokens a participant receives using `max_direct_participation_icp_e8s` as the denominator, but the actual token distribution in `create_sns_neuron_recipes` divides by `total_participant_icp_e8s`, which includes Neurons' Fund (NF) participation. When NF participation is non-zero, every direct participant contributing exactly `min_participant_icp_e8s` receives fewer SNS tokens than the validation guaranteed. If the shortfall pushes per-neuron allocation below `neuron_minimum_stake_e8s`, the SNS governance canister rejects the neuron claim after tokens have already been transferred to neuron subaccounts, permanently stranding those tokens with no recovery path.

## Finding Description

**Validation site (`rs/sns/init/src/lib.rs:1636–1642`):** [1](#0-0) 

The denominator is `max_direct_participation_icp_e8s` — the direct-only cap. NF participation is not accounted for.

**Actual distribution (`rs/sns/swap/src/swap.rs:848–852`):** [2](#0-1) 

The denominator is `total_participant_icp_e8s`, computed by `current_total_participation_e8s()`: [3](#0-2) 

When NF participates, `total_participant_icp_e8s > max_direct_participation_icp_e8s`, so `Swap::scale` returns a strictly smaller value than the validation assumed.

**`generate_vesting_schedule` does not enforce minimum stake.** The function simply divides `amount_sns_token_e8s` evenly across `count` neurons: [4](#0-3) 

It succeeds even when per-neuron amounts are below `neuron_minimum_stake_e8s`. This means `create_sns_neuron_recipes` succeeds and recipes are created with sub-threshold amounts.

**`sweep_sns` then transfers those sub-threshold amounts to neuron subaccounts** (subaccounts of the SNS governance canister): [5](#0-4) 

**`claim_swap_neurons` subsequently fails** because SNS governance rejects neurons whose stake is below `neuron_minimum_stake_e8s`. The tokens are already out of the swap canister and into governance subaccounts. There is no refund path in the swap canister for tokens already swept.

The finalization sequence confirms the ordering — `sweep_sns` precedes `claim_swap_neurons`: [6](#0-5) 

A second validation site in `rs/sns/swap/src/types.rs:346–348` uses `self.max_icp_e8s` (the legacy total-max field) rather than the actual `total_participant_icp_e8s` at finalization time, presenting the same class of mismatch: [7](#0-6) 

## Impact Explanation

A direct participant who contributes exactly `min_participant_icp_e8s` ICP has their ICP committed to SNS governance via `sweep_icp`, their SNS tokens transferred to neuron subaccounts via `sweep_sns`, and then their neuron claim rejected by SNS governance because each neuron's allocation is below `neuron_minimum_stake_e8s`. The SNS tokens are permanently stranded in the neuron subaccounts (subaccounts of the SNS governance canister keyed by principal + memo). The swap canister has no mechanism to recover tokens already transferred out. The participant loses both their ICP (committed) and their SNS tokens (stranded). This is a **High** impact: significant SNS security impact with concrete, irreversible user fund loss.

## Likelihood Explanation

NF participation via matched funding is a standard, non-privileged feature of SNS swaps. The NF participation amount is computed dynamically from the matched-funding polynomial and can be a substantial fraction of direct participation. Any SNS whose `min_participant_sns_e8s` is configured close to the `neuron_basket_count × (neuron_minimum_stake_e8s + transaction_fee_e8s)` threshold — a common configuration to maximize participation breadth — will silently violate the guarantee for minimum-contributing participants whenever NF participation is active. No attacker action is required; the condition arises from normal swap operation with standard parameters.

## Recommendation

Replace the denominator in both validation sites with the worst-case total participation, i.e., `max_direct_participation_icp_e8s + max_neurons_fund_participation_icp_e8s`, to match the maximum possible `total_participant_icp_e8s` at distribution time. Additionally, use ceiling division in the validation so the check is conservative in the correct direction:

```rust
// rs/sns/init/src/lib.rs
let max_total_icp = max_direct_participation_icp_e8s as u128
    + max_neurons_fund_participation_icp_e8s as u128;
let min_participant_sns_e8s = (min_participant_icp_e8s as u128
    * initial_swap_amount_e8s as u128
    + max_total_icp - 1)
    / max_total_icp;
```

Apply the same fix to `rs/sns/swap/src/types.rs`, using `max_direct_participation_icp_e8s + max_neurons_fund_participation_icp_e8s` instead of `max_icp_e8s`.

## Proof of Concept

**Parameters (all in e8s):**
- `min_participant_icp_e8s` = 2 × 10⁸
- `sns_token_e8s` = 1,000,000 × 10⁸
- `max_direct_participation_icp_e8s` = 100 × 10⁸
- `neuron_basket_count` = 3
- `neuron_minimum_stake_e8s` = 3,000
- `sns_transaction_fee_e8s` = 1,000
- NF contributes 100 × 10⁸ (matched 1:1)

**Validation check (passes):**
```
min_participant_sns_e8s = floor(2e8 × 1e14 / 100e8) = 20,000
threshold = 3 × (3,000 + 1,000) = 12,000
20,000 ≥ 12,000 → PASSES
```

**Actual distribution at finalization:**
```
total_participant_icp_e8s = 100e8 + 100e8 = 200e8
amount_sns_e8s = floor(2e8 × 1e14 / 200e8) = 10,000
per_neuron = floor(10,000 / 3) = 3,333
after_fee = 3,333 - 1,000 = 2,333 < 3,000 = neuron_minimum_stake_e8s
→ SNS governance rejects neuron claim
→ 10,000 SNS e8s stranded in neuron subaccounts, participant's 2 ICP committed, no neurons created
```

A deterministic unit test can reproduce this by constructing a `Swap` with the above parameters, setting `neurons_fund_participation_icp_e8s` to 100 × 10⁸, calling `create_sns_neuron_recipes()` (which succeeds), then calling `sweep_sns` (which transfers tokens), then mocking SNS governance to return `ClaimedSwapNeuronStatus::Failed` for the sub-threshold neurons in `claim_swap_neurons`, and asserting that `claim_neuron_result.failure > 0` while the neuron subaccount balances are non-zero.
