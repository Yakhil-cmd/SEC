### Title
Permanent ERC-20 Token Loss When `ft_transfer` Promise Fails Without Error-Refund Feature — (`engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` compile-time feature is disabled, the `ExitToNear` precompile schedules a bare NEAR promise (`PromiseArgs::Create`) with **no error-handling callback** for the legacy `ft_transfer` and Omni `ft_transfer_call` ERC-20 exit paths. Because the ERC-20 tokens are burned before the promise is dispatched, any failure of the downstream `ft_transfer` call results in permanent, unrecoverable loss of the user's tokens.

### Finding Description

The `ExitToNear` precompile's `run()` function in `engine-precompiles/src/native.rs` decides whether to attach an `exit_to_near_precompile_callback` based on whether `callback_args` equals the default value:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None without the feature
    transfer_near: transfer_near_args,
};
// ...
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback, no error handling
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [1](#0-0) 

For the two most common ERC-20 exit paths — the **legacy `ft_transfer`** path and the **Omni `ft_transfer_call`** path — `transfer_near_args` is `None`:

```rust
// Legacy flow
_ => (nep141_account_id, format!(...), "ft_transfer", None, ...)
// Omni flow
Some(Message::Omni(msg)) => (nep141_account_id, ft_transfer_call_args(...)?, "ft_transfer_call", None, ...)
``` [2](#0-1) 

Without `error_refund`, `callback_args` is `{ refund: None, transfer_near: None }` which equals `ExitToNearPrecompileCallbackArgs::default()`, so the promise is dispatched as a bare `PromiseArgs::Create` — no callback is attached.

The ERC-20 burn happens **before** the promise is dispatched, inside `EvmErc20V2.withdrawToNear()`:

```solidity
_burn(sender, amount);
// ... then calls exit precompile
``` [3](#0-2) 

If the NEP-141 `ft_transfer` call fails (e.g., the recipient account is not registered with the NEP-141 token), the ERC-20 tokens are already destroyed and there is no mechanism to re-mint them. The NEP-141 tokens remain in Aurora's account, inaccessible to the user.

The test suite explicitly acknowledges this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

The `exit_to_near_precompile_callback` function, which is the only place where `refund_on_error` (ERC-20 re-mint) is invoked, is never scheduled in this path: [5](#0-4) 

### Impact Explanation

**Permanent freezing of funds.** A user who calls `withdrawToNear()` on a bridged ERC-20 token with a recipient that is not registered with the underlying NEP-141 contract will have their ERC-20 tokens burned with no refund. The NEP-141 tokens remain locked in Aurora's account. There is no recovery path for the user.

### Likelihood Explanation

**Medium.** The trigger condition — `ft_transfer` failing because the recipient is not registered — is a normal, reachable NEP-141 failure mode. Any ERC-20 token holder can trigger this by specifying an unregistered NEAR account as the recipient. The vulnerability is gated on the `error_refund` feature being absent from the production build; if the feature is compiled in, the refund path is active and the issue does not manifest.

### Recommendation

Ensure the `error_refund` feature is always enabled in production builds. Alternatively, unconditionally attach the `exit_to_near_precompile_callback` for all ERC-20 exit paths (not just when `callback_args != default()`), so that a failed `ft_transfer` always triggers `refund_on_error` to re-mint the burned tokens to the original sender.

### Proof of Concept

1. User holds 100 units of a bridged ERC-20 token on Aurora.
2. User calls `erc20.withdrawToNear("unregistered.near", 100)`.
3. `EvmErc20V2._burn(sender, 100)` executes — tokens are destroyed.
4. The `ExitToNear` precompile schedules `PromiseArgs::Create(ft_transfer)` with no callback (because `error_refund` is not enabled and `transfer_near_args` is `None`).
5. The NEAR runtime executes `ft_transfer` on the NEP-141 contract; it fails because `"unregistered.near"` has no storage deposit.
6. No callback fires. No re-mint occurs.
7. User's 100 ERC-20 tokens are permanently gone; the NEP-141 balance remains in Aurora's account.

The test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` (lines 623–665) directly demonstrates this outcome: without `error_refund`, the user's ERC-20 balance decreases by `FT_EXIT_AMOUNT` with no refund. [6](#0-5)
