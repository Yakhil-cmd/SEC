### Title
Scheduled XCC Promises Cannot Be Cancelled and Are Executable by Any Caller — (`etc/xcc-router/src/lib.rs`)

---

### Summary
The XCC router's `execute_scheduled` function is callable by any external actor with no access control. There is no corresponding cancellation function. Once a user schedules a cross-contract call via the `CrossContractCallArgs::Delayed` path, the NEAR tokens funding that call are locked in the router sub-account and the scheduled promise cannot be invalidated. Any actor who knows the nonce can force execution at an arbitrary, potentially unfavorable time.

---

### Finding Description

When a user calls the XCC precompile with `CrossContractCallArgs::Delayed`, the engine calls `schedule` on the user's router sub-account. The `schedule` function is correctly gated to the parent (Aurora Engine): [1](#0-0) 

The sequential nonce is incremented and the promise is stored in `scheduled_promises`. The nonce value is also emitted on-chain via `near_sdk::log!`, making it publicly observable.

The `execute_scheduled` function, however, is explicitly open to any caller: [2](#0-1) 

There is no `cancel_scheduled`, no deadline field on `PromiseArgs`, and no mechanism for the parent engine or the originating EVM address to remove a scheduled entry without executing it. The entire `Router` struct exposes no invalidation path: [3](#0-2) 

The NEAR tokens that fund the scheduled call are transferred to the router sub-account balance at scheduling time (via `withdraw_wnear_to_router`). Once there, they are inaccessible to the user until the promise executes. [4](#0-3) 

---

### Impact Explanation

**High — Temporary freezing of funds.**

A user who schedules a cross-contract call (e.g., a token swap, a governance vote, a deposit into a protocol) cannot cancel it if conditions change. Their NEAR tokens remain locked in the router sub-account until the promise is executed. Because `execute_scheduled` is open to anyone, a malicious actor can:

1. Observe the scheduled nonce on-chain.
2. Call `execute_scheduled(nonce)` at the worst possible moment for the user (e.g., when the target contract is in a degraded state, or when a sandwich attack is profitable).

The user has no recourse: they cannot cancel the promise, cannot retrieve the locked NEAR, and cannot prevent forced execution. In the worst case (e.g., a DEX swap forced at a manipulated price), this escalates toward direct loss of user funds.

---

### Likelihood Explanation

**Medium.**

- The scheduled nonce is logged on-chain and is trivially discoverable.
- `execute_scheduled` requires no special permissions, no deposit beyond what the caller chooses to attach, and no knowledge beyond the nonce.
- Any NEAR account — including a bot — can call it at any block.
- The attack is most profitable in MEV-rich contexts (e.g., forcing a swap at a manipulated price), which are realistic on any public chain.

---

### Recommendation

1. **Add a `cancel_scheduled` function** restricted to the parent (Aurora Engine) that removes a promise from `scheduled_promises` and returns the associated NEAR balance to the originating address.
2. **Expose a cancellation path through the EVM** (e.g., a new selector on the XCC precompile) so that the originating EVM address can trigger cancellation via a normal EVM transaction.
3. **Add a deadline field** to `PromiseArgs` so that stale scheduled promises expire automatically and cannot be force-executed after the user's intended window.

---

### Proof of Concept

1. Alice calls the XCC precompile with `CrossContractCallArgs::Delayed` to schedule a token swap on a NEAR DEX. Her wNEAR is transferred to the router sub-account. The router logs `"Promise scheduled at nonce 7"`.
2. Before Alice executes the swap, the DEX liquidity is drained by a separate attack, making the swap highly unfavorable.
3. Alice attempts to cancel — there is no cancellation function. Her NEAR tokens remain locked in the router.
4. Bob (attacker) observes nonce 7 on-chain and calls `router.execute_scheduled({nonce: 7})` immediately, forcing Alice's swap at the worst possible price.
5. Alice's NEAR tokens are consumed by the forced cross-contract call. Bob profits via a sandwich around the forced swap.

The root cause is the combination of:
- No cancellation path in `etc/xcc-router/src/lib.rs` [5](#0-4) 
- Unrestricted public access to `execute_scheduled` [6](#0-5)

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);

pub fn withdraw_wnear_to_router<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;
        if matches!(handler.promise_result_check(), Some(false)) {
            return Err(b"ERR_CALLBACK_OF_FAILED_PROMISE".into());
        }
        let args: WithdrawWnearToRouterArgs = io.read_input_borsh()?;
        let current_account_id = env.current_account_id();
        let recipient = AccountId::try_from(format!(
            "{}.{}",
            args.target.encode(),
            current_account_id.as_ref()
        ))?;
        let wnear_address = aurora_engine_precompiles::xcc::state::get_wnear_address(&io);
        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&current_account_id),
            current_account_id,
            io,
            env,
        );
        let (result, ids) = xcc::withdraw_wnear_to_router(
            &recipient,
            args.amount,
            wnear_address,
            &mut engine,
            handler,
        )?;
        if !result.status.is_ok() {
            return Err(b"ERR_WITHDRAW_FAILED".into());
        }
        let id = ids.last().ok_or(b"ERR_NO_PROMISE_CREATED")?;
        handler.promise_return(*id);
        Ok(result)
    })
