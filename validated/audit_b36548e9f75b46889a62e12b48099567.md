Audit Report

## Title
Markdown Injection via Unsanitized UTF-8 Memo in ICRC-21 GenericDisplay Consent Messages - (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
`ConsentMessage::add_memo` interpolates caller-supplied ICRC-1 memo bytes verbatim into a Markdown inline code span in the `GenericDisplayMessage` without escaping backtick or newline characters. An attacker can craft a memo that terminates the code span early and injects arbitrary Markdown block elements — including spoofed `**Amount:**` and `**To:**` fields — into the consent message that ICRC-21-compliant wallets (including hardware wallets) display to users before they authorize a transaction. This breaks the core security guarantee of ICRC-21: that the human-readable consent message faithfully represents the transaction being signed.

## Finding Description
In `packages/icrc-ledger-types/src/icrc21/responses.rs` at line 294, the `GenericDisplayMessage` branch of `add_memo` formats the memo as:

```rust
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
```

`memo_str` is the raw UTF-8 interpretation of the memo bytes (lines 288–291); no escaping of `` ` `` or `\n` is performed. A single backtick in `memo_str` closes the inline code span; any content following it is rendered as ordinary Markdown. A newline allows new block-level elements (headings, bold labels) to begin.

The builder is populated with the caller-supplied `TransferArg.memo` at `packages/icrc-ledger-types/src/icrc21/lib.rs` lines 392–395, and `build()` calls `add_memo` unconditionally when a memo is present (lines 290–292). The same shared library is invoked by the ICP ledger's `icrc21_canister_call_consent_message` endpoint (`rs/ledger_suite/icp/ledger/src/main.rs` lines 1531–1540) for all ICRC-1/ICRC-2 method paths.

**Exploit flow:**
1. Attacker constructs a `TransferArg` with `amount = 1000 ICP`, `to = attacker_account`, and `memo = b"\`\n\n**Amount:** \`0.00001 ICP\`\n\n**To:**\n\`legitimate-recipient\`"`.
2. Attacker (or a malicious dapp) calls `icrc21_canister_call_consent_message` with this arg and `device_spec = GenericDisplay`.
3. The returned `GenericDisplayMessage` contains the real fields first (`**Amount:** 1000 ICP`, `**To:** attacker_account`) followed by the injected fake fields (`**Amount:** 0.00001 ICP`, `**To:** legitimate-recipient`) rendered as visually identical bold-label Markdown sections.
4. On a paginated hardware-wallet display, the injected page may be the only one the user reads before approving. On GUI wallets that render Markdown, the injected section is visually indistinguishable from the legitimate fields.
5. The user approves, believing they are sending 0.00001 ICP to a legitimate recipient; the actual on-chain effect is a transfer of 1000 ICP to the attacker.

No existing guard prevents this: the only sanitization applied is the UTF-8 validity check (line 288), which is precisely the condition that enables the injection. The ckBTC minter explicitly documents and defends against this class of injection for addresses (`rs/bitcoin/ckbtc/minter/src/updates/icrc21.rs` lines 195–209) via structural validation, but no equivalent protection exists for the memo field.

## Impact Explanation
The `GenericDisplayMessage` is the sole human-readable artifact that ICRC-21-compliant wallets — including Ledger hardware wallets — present to users for transaction authorization. Injecting false `**Amount:**` and `**To:**` fields into this message can cause a user to authorize a large transfer to an attacker's account while believing they are approving a trivial transfer to a legitimate recipient. This constitutes a concrete path to theft of user funds via ledger transactions, matching the High impact class: "Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
The ICRC-1 memo field accepts arbitrary bytes up to 32 bytes. Any caller of `icrc1_transfer`, `icrc2_approve`, or `icrc2_transfer_from` on any ledger using the shared `build_icrc21_consent_info` library can supply a crafted memo. No privilege, stake, or governance role is required. The attack requires a single unauthenticated call to `icrc21_canister_call_consent_message` with a crafted `TransferArg`. It is deterministic, requires no brute-force or timing, and is repeatable against any user of any ICRC-21-compliant wallet interacting with the affected ledgers.

## Recommendation
1. **Escape backticks and newlines in `memo_str` before interpolation** in the `GenericDisplayMessage` branch: replace every `` ` `` with `` \` `` (or use a double-backtick delimiter) and replace `\n`/`\r` with a space or strip them entirely.
2. **Alternatively, hex-encode all memo bytes unconditionally** in the `GenericDisplayMessage` path, reserving UTF-8 display for the `FieldsDisplayMessage` path where the value is a structured `Value::Text` field and Markdown rendering does not apply.
3. **Add a regression test** verifying that a memo containing `` ` ``, `\n`, and `**bold**` does not produce injected Markdown sections in the `GenericDisplayMessage` output, analogous to the existing ckBTC/ckDOGE address-injection tests.

## Proof of Concept
Call `icrc21_canister_call_consent_message` on the ICP ledger (or any ICRC-1 ledger using the shared library) with:

```
method = "icrc1_transfer"
arg = Candid-encode(TransferArg {
    from_subaccount: None,
    to: <legitimate_recipient>,
    fee: None,
    amount: 100_000_000_000,   // 1000 ICP
    created_at_time: None,
    memo: Some(Memo(ByteBuf::from(
        b"`\n\n**Amount:** `0.00001 ICP`\n\n**To:**\n`legitimate-recipient`"
    ))),
})
user_preferences.device_spec = Some(GenericDisplay)
```

The returned `GenericDisplayMessage` will contain the real fields (`**Amount:** 1000 ICP`, `**To:** attacker_account`) followed by the injected Markdown sections. A unit test in `packages/icrc-ledger-types/src/icrc21/` can assert that the output string does not contain the injected bold headers, confirming both the vulnerability and the fix.
