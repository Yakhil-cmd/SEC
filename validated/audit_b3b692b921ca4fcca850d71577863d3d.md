### Title
Unchecked Precompile Call Return Value After Token Burn Enables Permanent Fund Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` burn the caller's ERC-20 tokens **before** calling the exit precompile, and then silently ignore the return value of that precompile call. If the precompile call fails for any reason (e.g., out-of-gas, missing storage key, paused state), the tokens are permanently destroyed with no corresponding NEAR or Ethereum transfer and no revert.

---

### Finding Description

In `withdrawToNear()` and `withdrawToEthereum()` of both `EvmErc20` and `EvmErc20V2`, the execution order is:

1. `_burn(sender, amount)` — ERC-20 tokens are irreversibly destroyed.
2. An inline assembly `call()` to the exit precompile is made.
3. The return value `res` is captured in a local assembly variable but **never inspected**. No `if iszero(res) { revert(0, 0) }` guard exists.

`EvmErc20.sol` `withdrawToNear` (lines 53–63):
```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked
    }
}
```

The identical pattern appears in `EvmErc20V2.sol` `withdrawToNear` (lines 53–64) and `withdrawToEthereum` in both contracts.

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`, `run()`) can return `ExitError::OutOfGas` when the forwarded gas is below `costs::EXIT_TO_NEAR_GAS`, or `ExitError::Other(...)` for missing storage keys (`ERR_KEY_NOT_FOUND`, `ERR_TARGET_TOKEN_NOT_FOUND`). When the precompile returns an error, the EVM `call()` instruction returns `0`. Because `res` is never checked, the outer function does not revert, and the burn is finalized with no promise scheduled.

The `error_refund` feature only handles the case where the NEAR-side promise fails **after** a successful precompile invocation. It provides no protection when the precompile itself fails at the EVM level.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the precompile call fails silently:
- The user's ERC-20 token balance is reduced to zero (burned).
- No NEAR promise is scheduled, so no NEP-141 tokens are transferred to the recipient.
- No Ethereum withdrawal is initiated.
- There is no refund path: the `error_refund` callback is only registered when the precompile succeeds and returns a `PromiseWithCallbackArgs`.
- The tokens are permanently destroyed with no corresponding credit anywhere.

---

### Likelihood Explanation

**Medium.** The most direct unprivileged trigger is gas manipulation: a caller (or an intermediate contract) can supply a transaction gas limit that is sufficient to execute `_burn` but leaves fewer than `EXIT_TO_NEAR_GAS` units for the precompile call. Because `call(gas(), ...)` forwards all *remaining* gas at the point of the call instruction, a carefully chosen total gas limit causes the precompile to return `ExitError::OutOfGas`, the `call()` returns `0`, and the function completes without reverting. Any token holder of a bridged NEP-141 asset can trigger this against their own balance; a malicious contract wrapping the call can trigger it against any user who delegates the withdrawal to it.

---

### Recommendation

Check the return value of every precompile `call()` and revert on failure. Apply this to all four affected assembly blocks in both contracts:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Alternatively, restructure the functions to call the precompile **before** burning, and only burn if the call succeeds.

---

### Proof of Concept

1. Deploy `EvmErc20` (or `EvmErc20V2`) with a valid NEP-141 mapping.
2. Mint `1000` tokens to `attacker`.
3. `attacker` calls `withdrawToNear(recipient, 1000)` with a transaction gas limit chosen so that after `_burn` executes, fewer than `EXIT_TO_NEAR_GAS` units remain.
4. Inside the assembly block, `call(gas(), 0xe9217bc7..., ...)` is executed. The precompile's gas check fires:
   ```rust
   // engine-precompiles/src/native.rs
   if let Some(target_gas) = target_gas && required_gas > target_gas {
       return Err(ExitError::OutOfGas);
   }
   ```
   The precompile returns `ExitError::OutOfGas`; the EVM `call` returns `0`.
5. `res` is never read. The assembly block exits normally. `withdrawToNear` returns without reverting.
6. `attacker`'s ERC-20 balance is `0`. No NEP-141 transfer was scheduled. Tokens are permanently lost. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
