Audit Report

## Title
`disable_api` Pause Flag Not Enforced in Timer-Driven Processing — (File: rs/migration_canister/src/processing.rs)

## Summary
The migration canister's `DISABLED` stable-memory flag, set by the privileged `disable_api()` endpoint (callable by NNS governance), is checked only in the ingress-facing `migrate_canister()` update method. The entire timer-driven state machine — nine 1-second interval callbacks dispatched through `process_all_by_predicate` — never reads `migrations_disabled()`. Consequently, any migration already in-flight at the moment `disable_api` is called continues executing irreversible on-chain operations, including `delete_canister` and registry routing-table mutations, contrary to the governance-mandated pause intent.

## Finding Description
The `DISABLED` flag is stored in stable memory and exposed via `migrations_disabled()`: [1](#0-0) [2](#0-1) 

`disable_api()` in `privileged.rs` sets this flag; it is callable by controllers and the NNS governance canister (`rrkah-fqaaa-aaaaa-aaaaq-cai`): [3](#0-2) 

The only production read of `migrations_disabled()` is the ingress guard in `migrate_canister()`: [4](#0-3) 

`start_timers()` schedules nine independent 1-second callbacks, each calling `process_all_by_predicate` for a different pipeline stage: [5](#0-4) 

`process_all_by_predicate` — the shared dispatcher for all eight stage processors — contains no call to `migrations_disabled()`: [6](#0-5) 

A grep across all `rs/migration_canister/src/*.rs` files confirms `migrations_disabled` / `DISABLED` / `disable_api` appear only in `canister_state.rs`, `migration_canister.rs`, and `privileged.rs` — entirely absent from `processing.rs`. The `MethodGuard` present in `process_all_by_predicate` only prevents concurrent re-entry of the same stage; it does not consult the disabled flag.

The stages that continue executing after `disable_api` is called include:
- `process_accepted` → strips original controllers via `update_settings`
- `process_controllers_changed` → stops both canisters
- `process_renamed` → renames the replaced canister
- `process_updated` / `process_routing_table` → mutates the registry routing table and calls `delete_canister`
- `process_migrated_canister_deleted` → restores controllers after the mandatory 6-minute wait

## Impact Explanation
When NNS governance passes a `PauseCanisterMigrations` proposal (mapped to `disable_api` on the migration canister), the intent is to halt all migration activity. Because the timer callbacks ignore the `DISABLED` flag, every in-flight migration continues through all remaining stages. This includes the irreversible `delete_canister` call and the registry routing-table mutation (`migrate_canisters`), which proceed contrary to what governance intended when it voted to pause. This constitutes a significant NNS/infrastructure safety-mechanism failure with concrete user and protocol harm — matching the **Medium** allowed impact class (meaningful security impact under constrained conditions: the pause must already have been triggered).

## Likelihood Explanation
The migration state machine includes a mandatory ~6-minute wait (`MAX_INGRESS_TTL + PERMITTED_DRIFT_AT_VALIDATOR + 30s`) before `process_migrated_canister_deleted` completes. Because the NNS governance process itself takes time (proposal submission → voting → execution), it is virtually certain that migrations will be in-flight at the moment `disable_api` executes in response to an incident. Any migration that has progressed past `Accepted` at that moment will run to completion regardless of the flag.

## Recommendation
Add a `migrations_disabled()` guard at the top of `process_all_by_predicate` in `rs/migration_canister/src/processing.rs`, before the `MethodGuard` acquisition:

```rust
pub async fn process_all_by_predicate<F>(
    tag: &str,
    predicate: impl Fn(&RequestState) -> bool,
    processor: impl Fn(RequestState) -> F,
) where
    F: Future<Output = ProcessingResult<RequestState, RequestState>>,
{
+   if crate::canister_state::migrations_disabled() {
+       return;
+   }
    let Ok(_guard) = MethodGuard::new(tag) else { return; };
    // ...
}
```

The same guard should be added to `process_all_failed` and `process_all_succeeded` for completeness. Alternatively, if the intended semantics are that `disable_api` only gates new requests, rename the endpoint (e.g., `disable_new_requests`) and update governance documentation to prevent operators from relying on it as a full-stop mechanism.

## Proof of Concept
A PocketIC integration test can reproduce this deterministically:

1. Deploy the migration canister with a test allowlist.
2. Call `migrate_canister(A, B)` from an allowed principal → request enters `RequestState::Accepted`.
3. Advance the PocketIC clock by 1 second → `process_accepted` fires, stripping controllers of A and B.
4. Call `disable_api()` from the governance canister principal → `DISABLED = true`.
5. Advance the clock by 1 second → assert that `process_controllers_changed` still fires (stops both canisters) despite `DISABLED = true`.
6. Continue advancing the clock through all stages → assert `delete_canister` is called and the migration reaches `RestoredControllers` / `Succeeded` while `DISABLED` remains `true`.
7. Assert that `migration_status` returns `Succeeded` — confirming the pause had zero effect on the in-flight migration.

pub async fn process_all_by_predicate<F>(
    tag: &str,
    predicate: impl Fn(&RequestState) -> bool,
    processor: impl Fn(RequestState) -> F,
) where
    F: Future<Output = ProcessingResult<RequestState, RequestState>>,
{
    // Ensures this method runs only once at any given time.
    let Ok(_guard) = MethodGuard::new(tag) else {
        return;
    };
    let mut tasks = vec![];
    let requests = list_by(predicate);
    if requests.is_empty() {
        return;
    }
    println!(
        "Entering `{}` with {} pending requests",
        tag,
        requests.len()
    );
    for request in requests.iter() {
        tasks.push(processor(request.clone()));
    }
    let results = join_all(tasks).await;
    let mut success_counter = 0;
    for (req, res) in zip(requests, results) {
        if res.is_success() {
            success_counter += 1;
        }
        res.transition(req);
    }
    println!(
        "Exiting `{}` with {} successful transitions.",
        tag, success_counter
    );
}
