## Title
Silently Ignored Burn Result in `burn_and_log` Breaks ICP/Cycles Conservation Invariant — (File: `rs/nns/cmc/src/main.rs`)

## Summary
In the Cycles Minting Canister, `process_create_canister` calls `burn_and_log` after successfully creating a canister and minting cycles, but the `()` return value is silently discarded. If the ICP ledger call inside `burn_and_log` fails during a ledger upgrade, cycles are minted and the canister is created without the corresponding ICP being burned. The notification block index is permanently recorded as processed, preventing any retry of the burn.

## Finding Description
In `rs/nns/cmc/src/main.rs`, `process_create_canister` (lines 1925–1956) follows this sequence:

1. `tokens_to_cycles(amount)` converts ICP to cycles.
2. `do_create_canister(controller, cycles, ...)` mints cycles and creates the canister.
3. On success, `burn_and_log(sub, amount).await` is called — its `()` return value is unconditionally discarded at line 1945.

`burn_and_log` (lines 2014–2049) is explicitly designed to swallow errors. The comment at lines 2014–2016 reads: *"Burning doesn't return errors — we don't want to reject the transaction notification because then it could be retried."* When `call_protobuf(ledger_canister_id, "send_pb", send_args)` at line 2040 returns `Err((code, err))`, the error is only printed; the function returns `()` regardless.

In `notify_create_canister` (lines 1405–1423), after `process_create_canister` returns `Ok(canister_id)`, the block index is written to `blocks_notified` as `NotificationStatus::NotifiedCreateCanister(Ok(canister_id))` at line 1412–1414. The `is_transient_error` guard at line 1416 only removes the entry for transient `Err` results — since `Ok(canister_id)` is returned even when the burn fails, the entry is never removed. Any subsequent call with the same block index returns the cached `Ok(canister_id)` immediately, permanently preventing a retry of the burn.

## Impact Explanation
This is a **High** severity finding. The CMC is a core NNS canister responsible for the ICP/cycles conservation invariant. When `burn_and_log` fails silently, cycles are minted without the corresponding ICP being destroyed. The ICP remains permanently stranded in CMC's subaccount — not burned, not refunded. Over repeated failures across ledger upgrade windows, the cycles supply grows unbacked by burned ICP, breaking the fundamental economic invariant of the IC protocol. This constitutes concrete, measurable protocol harm to NNS/CMC financial integrity, matching the "Significant NNS security impact with concrete user or protocol harm" High impact category.

## Likelihood Explanation
The ICP ledger undergoes periodic upgrades on the NNS subnet. During an upgrade window, calls to the ledger canister are rejected with a transient error. Any `notify_create_canister` call that reaches the `burn_and_log` step during this window will silently skip the burn. Ledger upgrades are publicly observable events visible on the NNS dashboard. An unprivileged user can trigger this by timing a `notify_create_canister` call to coincide with a ledger upgrade. No privileged access, key material, or majority corruption is required. The exploit is repeatable across every ledger upgrade cycle.

## Recommendation
`burn_and_log` should return a `Result` and `process_create_canister` should either propagate the failure or record it in a pending-burns queue for later retry. The current design conflates two separate concerns — preventing notification replay and ensuring the burn completes — and resolves the tension by silently sacrificing conservation correctness. A pending-burns queue would allow the burn to be retried independently without enabling notification replay.

## Proof of Concept
1. User sends 10 ICP to CMC's subaccount for their principal.
2. User calls `notify_create_canister` with the corresponding block index, timed to coincide with a ledger upgrade.
3. CMC calls `do_create_canister` — succeeds, canister created, cycles minted.
4. CMC calls `burn_and_log` — the ICP ledger is mid-upgrade, `call_protobuf` at line 2040 returns `Err((code, msg))`.
5. `burn_and_log` prints the error and returns `()`.
6. `process_create_canister` returns `Ok(canister_id)`.
7. `notify_create_canister` writes `NotificationStatus::NotifiedCreateCanister(Ok(canister_id))` to `blocks_notified` at lines 1412–1414 — permanently processed; `is_transient_error` at line 1416 does not remove it.
8. Result: canister exists with full cycles allocation; 10 ICP sits unburned in CMC's subaccount; cycles supply is inflated relative to burned ICP.
