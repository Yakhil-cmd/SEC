### Title
Silent U256→u128 Truncation in `ExitToEthereum` ERC-20 Withdrawal Amount — (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile's ERC-20 branch reads a 32-byte (U256) amount from calldata and calls `.as_u128()` on it without first checking that the value fits in 128 bits. The upper 128 bits are silently discarded. The ERC-20 tokens are burned for the full U256 amount, but the withdrawal JSON sent to the eth-connector records only the truncated u128 value. The difference is permanently unrecoverable.

---

### Finding Description

In `ExitToEthereum::run()`, the ERC-20 branch (flag `0x1`) parses a 32-byte amount from calldata:

```rust
let amount = parse_amount(&input[..32])?;
```

`parse_amount` accepts a 32-byte slice (a full U256 word) and returns a `U256`. The result is then used directly as:

```rust
format!(
    r#"{{"amount": "{}", "recipient": "{}"}}"#,
    amount.as_u128(),   // ← silent truncation: upper 128 bits dropped
    recipient_in_hex
)
```

`U256::as_u128()` returns only the low 128 bits of the value. If the caller supplies an amount where any of the upper 128 bits are set (i.e., `amount > u128::MAX`), the serialized withdrawal amount will be `amount mod 2^128` — a completely different, smaller number — while the ERC-20 burn on the Aurora side consumes the full U256 amount.

Contrast this with the `ExitToNear` path, which routes through `ft_transfer_call_args` and contains an explicit guard:

```rust
if amount > U256::from(u128::MAX) {
    return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
}
```

This guard is entirely absent in the `ExitToEthereum` ERC-20 branch. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

A user who holds an ERC-20 token balance exceeding `u128::MAX` on Aurora calls the token's `withdraw` function, which triggers the `ExitToEthereum` precompile. The ERC-20 contract burns the full U256 amount from the user's balance. The precompile then serializes `amount.as_u128()` — the low 128 bits — into the withdrawal JSON forwarded to the eth-connector. The eth-connector releases only the truncated amount on the Ethereum side. The difference (`amount - (amount mod 2^128)`) is permanently destroyed with no recourse. This constitutes **permanent freezing / direct theft of user funds**.

---

### Likelihood Explanation

ERC-20 balances in the EVM are natively U256. A token with a sufficiently large total supply (e.g., a token with 18 decimals and a supply of `~3.4 × 10^20` tokens) can produce individual balances exceeding `u128::MAX ≈ 3.4 × 10^38` raw units. Any such token deployed on Aurora is affected. An attacker does not need to craft the overflow; a legitimate user with a large balance triggers it inadvertently. The missing check is a single missing guard that the developers already applied in the parallel `ExitToNear` path, confirming awareness of the issue class.

---

### Recommendation

Add the same overflow guard used in `ft_transfer_call_args` immediately after parsing the amount in the `ExitToEthereum` ERC-20 branch:

```rust
let amount = parse_amount(&input[..32])?;
if amount > U256::from(u128::MAX) {
    return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
}
``` [3](#0-2) 

---

### Proof of Concept

1. Deploy an ERC-20 token on Aurora with a total supply of `2^128 + 1` raw units (achievable with 18 decimals and a supply of ~`3.4 × 10^20` tokens).
2. Transfer the entire supply to a single address so its balance exceeds `u128::MAX`.
3. Call the token's `withdraw(amount, eth_recipient)` with `amount = 2^128 + 1`.
4. The ERC-20 contract calls `ExitToEthereum` precompile; the full `2^128 + 1` tokens are burned.
5. The precompile serializes `(2^128 + 1).as_u128() == 1` into the withdrawal JSON.
6. The eth-connector releases only `1` token unit on Ethereum.
7. `2^128` token units are permanently destroyed. [4](#0-3)
