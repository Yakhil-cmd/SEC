Audit Report

## Title
Unbounded Loop via Integer Overflow in `construction_payloads` Causes OOM DoS of ICP Rosetta Node — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The ICP Rosetta `construction_payloads` handler accepts caller-controlled `ingress_start` and `ingress_end` values with no bounds validation. When `ingress_start` is near `u64::MAX`, the `now += interval` step in the ingress-expiry loop silently wraps via a truncating `as u64` cast in `Time::from_duration`, resetting `now` to ~120 seconds since epoch and causing the loop to execute ~153 billion additional iterations. Each iteration pushes a `u64` onto a `Vec`, exhausting process memory (~1.23 TB allocation) and crashing the Rosetta node.

## Finding Description

**Loop (confirmed at L99–107):** [1](#0-0) 

`interval` is computed as `MAX_INGRESS_TTL - PERMITTED_DRIFT - 120s = 300s - 60s - 120s = 120_000_000_000 ns`. [2](#0-1) [3](#0-2) 

**Truncating cast (confirmed at L55–58 and L103–105):**
`now += interval` calls `Time::add_assign`, which calls `Time::from_duration`, which does `t.as_nanos() as u64` — a silent truncating cast with no overflow check. [4](#0-3) [5](#0-4) 

**No input validation (confirmed at L74–84 and L200–223):**
`ingress_start` and `ingress_end` are raw `Option<u64>` fields deserialized directly from the request with no bounds checking before the loop. [6](#0-5) [7](#0-6) 

**Overflow trace:**
- `ingress_start = u64::MAX - 1`, `ingress_end = u64::MAX`
- Iteration 1: condition `now < ingress_end` is true; push entry; then `now += 120_000_000_000`
- `Duration::from_nanos(u64::MAX - 1) + Duration::from_nanos(120_000_000_000)` → `as_nanos()` returns `u128 = 18_446_744_193_709_551_614`
- `as u64` truncates to `119_999_999_998` (~120 seconds since epoch)
- Iteration 2+: `119_999_999_998 < u64::MAX` → true → loop runs ~153 billion more iterations
- Each iteration pushes a `u64` (8 bytes) → ~1.23 TB of allocation → OOM crash

**Contrast with ICRC1 handler (confirmed at L148–158):**
The ICRC1 handler explicitly rejects `ingress_start >= ingress_end` and validates the window before its loop. The ICP handler has no equivalent guard. [8](#0-7) 

A safe checked-add API already exists but is not used in `AddAssign`: [9](#0-8) 

## Impact Explanation
A single unauthenticated HTTP POST to `/construction/payloads` crashes the ICP Rosetta node process via OOM. The Rosetta API is an explicitly in-scope financial integration component. This matches the allowed High impact: **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"** and **"Significant … Rosetta … security impact with concrete user or protocol harm."** Exchange operators and integrators relying on the Rosetta node for ICP transfers and neuron management lose availability until the process is restarted.

## Likelihood Explanation
The Rosetta API is a public HTTP endpoint with no authentication. The exploit requires a single crafted JSON request with two specific `u64` field values. No privileged access, key material, or special network position is needed. The attack is trivially repeatable — the node can be crashed again immediately after restart.

## Recommendation

1. **Add pre-loop validation** in `construction_payloads` mirroring the ICRC1 handler:
   ```rust
   if ingress_end <= ingress_start {
       return Err(ApiError::invalid_request("ingress_end must be after ingress_start"));
   }
   let max_window = ic_limits::MAX_INGRESS_TTL * 2; // or a fixed 24h cap
   if ingress_end.as_nanos_since_unix_epoch()
       .saturating_sub(ingress_start.as_nanos_since_unix_epoch())
       > max_window.as_nanos() as u64
   {
       return Err(ApiError::invalid_request("ingress window too large"));
   }
   ```
2. **Replace the truncating cast** in `Time::add_assign` with `Time::checked_add`, making overflow explicit rather than silent:
   ```rust
   impl std::ops::AddAssign<Duration> for Time {
       fn add_assign(&mut self, other: Duration) {
           *self = self.checked_add(other).expect("Time overflow in AddAssign");
       }
   }
   ```
3. **Mirror the ICRC1 validation** pattern consistently across all Rosetta handlers that accept caller-controlled time windows.
