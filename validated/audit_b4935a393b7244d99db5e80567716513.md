### Title
Stale Proposal-Submission-Time Valuation Used as Treasury Transfer Limit at Execution Time - (`rs/sns/governance/src/proposal.rs`)

### Summary

The SNS Governance canister stores a treasury `Valuation` (token balance + ICP/XDR price) captured at **proposal submission time** inside `ProposalData::action_auxiliary`. At execution time, `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` reuses this stale valuation to compute `allowance_tokens` — the 7-day upper bound on how many tokens may be transferred from the SNS treasury. Because the treasury balance captured in the valuation can be materially lower than the actual current balance (e.g., the treasury received large deposits after the proposal was submitted), the computed `allowance_tokens` is artificially low, causing legitimately-sized transfers to be blocked. Conversely, if the treasury was drained after submission, the stale valuation allows a transfer that exceeds the intended fraction of the current treasury.

### Finding Description

**Vulnerability class:** Governance authorization bug — stale cached value used in a critical limit calculation.

**Root cause:**

At proposal submission, `validate_and_render_transfer_sns_treasury_funds` calls `treasury_valuation_if_proposal_amount_is_small_enough_or_err`, which calls `assess_treasury_balance` to fetch the live treasury balance and ICP/XDR price, producing a `Valuation`. This `Valuation` is stored in `ProposalData.action_auxiliary` as `TransferSnsTreasuryFundsActionAuxiliary { valuation }`. [1](#0-0) 

At execution time (which can be days later, after voting completes), `perform_transfer_sns_treasury_funds` retrieves this stored valuation and passes it to `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`: [2](#0-1) 

Inside that function, `allowance_tokens` is computed from the **stale** valuation: [3](#0-2) 

The code itself acknowledges this is the submission-time value: [4](#0-3) 

The `Valuation` struct captures `tokens` (treasury balance), `icps_per_token`, and `xdrs_per_icp` at a single point in time: [5](#0-4) 

The limit formula in `ProposalsAmountTotalUpperBound::in_tokens` uses `balance_tokens` from the valuation directly as the `NoLimit` or `Fraction` case: [6](#0-5) 

**The mismatch:** The `spent_tokens` counter is recalculated from live proposal history at execution time, but `allowance_tokens` is frozen at submission time. If the treasury balance grew (e.g., from swap proceeds or token minting) between submission and execution, `allowance_tokens` is too small relative to the current treasury, causing the execution-time check to reject a transfer that is actually within the intended fraction of the current treasury. If the treasury shrank (e.g., another proposal drained it), `allowance_tokens` is too large, allowing a transfer that exceeds the intended fraction of the current treasury.

### Impact Explanation

**Over-restriction (DoS of legitimate governance):** An SNS treasury that receives a large deposit after a `TransferSnsTreasuryFunds` proposal is submitted will have its execution blocked even though the transfer amount is well within the intended 7-day limit relative to the current balance. The proposal fails with `PreconditionFailed`, and the SNS community must re-submit and re-vote, wasting governance cycles and delaying treasury operations.

**Under-restriction (excess treasury drain):** If the treasury balance drops significantly after proposal submission (e.g., a concurrent proposal executes and drains a large portion), the stale valuation still permits the original transfer amount. The execution-time check uses the old (higher) `allowance_tokens`, so the transfer may proceed even though it now represents a much larger fraction of the actual remaining treasury than the protocol intends to allow. For a "medium" treasury (25% limit), this could allow draining a disproportionate share of the remaining balance.

The impact is a **governance authorization bug**: the 7-day treasury transfer limit — a core SNS safety mechanism — is computed from stale data, causing it to be either too permissive or too restrictive depending on treasury balance changes during the voting period.

### Likelihood Explanation

SNS voting periods are typically several days. Treasury balances change routinely (swap proceeds, token minting, other proposals). Any SNS with active treasury activity will encounter this mismatch. The attacker-controlled entry path is: submit a `TransferSnsTreasuryFunds` proposal when the treasury is large, wait for the treasury to shrink (or cause it to shrink via a concurrent proposal), then execute the now-adopted proposal using the stale high-balance valuation to bypass the current-balance limit. This is reachable by any SNS neuron holder with sufficient voting power to pass a proposal — an unprivileged canister caller.

### Recommendation

At execution time, re-fetch the current treasury balance and recompute `allowance_tokens` from the live valuation rather than the stale submission-time valuation. The `perform_transfer_sns_treasury_funds` function should call `assess_treasury_balance` again before invoking `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`, replacing the stored `valuation` with a fresh one. The stored valuation in `action_auxiliary` can be retained for audit/display purposes but should not be used for the execution-time limit check.

### Proof of Concept

1. SNS treasury holds 10,000 ICP. ICP price = 42 XDR. Treasury XDR value = 420,000 XDR → "medium" regime → 25% limit = 2,500 ICP.
2. Attacker (neuron holder) submits `TransferSnsTreasuryFunds` for 2,400 ICP. Valuation stored: `tokens=10000, icps_per_token=1, xdrs_per_icp=42`. `allowance_tokens` = 2,500 ICP.
3. During the voting period (e.g., 4 days), a second proposal executes and transfers 2,000 ICP out of the treasury. Treasury now holds 8,000 ICP. The 25% limit on 8,000 ICP = 2,000 ICP.
4. The first proposal is adopted and executed. `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` is called with the **stale** valuation (`tokens=10000`), computing `allowance_tokens = 2,500 ICP`. `spent_tokens` = 2,000 ICP (from the already-executed proposal). `remainder_tokens = 500 ICP`. The 2,400 ICP transfer is blocked — **over-restriction**.
5. Alternatively: if the second proposal had not yet executed at check time but executes concurrently, `spent_tokens = 0`, `remainder_tokens = 2,500 ICP`, and the 2,400 ICP transfer proceeds — but the actual treasury is now 8,000 ICP, so 2,400/8,000 = 30% > 25% limit — **under-restriction**.

    async fn perform_transfer_sns_treasury_funds(
        &mut self,
        proposal_id: u64, // This is just to control concurrency.
        valuation: Result<Valuation, GovernanceError>,
        transfer: &TransferSnsTreasuryFunds,
    ) -> Result<(), GovernanceError> {
        // Only execute one proposal of this type at a time.
        thread_local! {
            static IN_PROGRESS_PROPOSAL_ID: RefCell<Option<u64>> = const { RefCell::new(None) };
        }
        let release_on_drop = acquire(&IN_PROGRESS_PROPOSAL_ID, proposal_id);
        if let Err(already_in_progress_proposal_id) = release_on_drop {
            return Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!(
                    "Another TransferSnsTreasuryFunds proposal (ID = {already_in_progress_proposal_id}) is already in progress.",
                ),
            ));
        }

        transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err(
            transfer,
            valuation?,
            self.proto.proposals.values(),
            self.env.now(),
        )?;
