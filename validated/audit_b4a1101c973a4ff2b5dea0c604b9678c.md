### Title
Unchecked Precompile Call Return Value in `withdrawToNear` and `withdrawToEthereum` Allows Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` burn the caller's ERC-20 tokens before invoking the exit precompile via inline assembly. The `call` opcode's return value (`res`) is captured but never validated. If the precompile call fails, the burn is irreversible and the user's tokens are permanently destroyed with no corresponding NEP-141 release on NEAR.

### Finding Description

In both `EvmErc20.sol` and `EvmErc20V2.sol`, the `withdrawToNear` and `withdrawToEthereum` functions follow this pattern:

```solidity
// EvmErc20.sol - withdrawToNear (lines 53-63)
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // <-- tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no revert if call returns 0
    }
}
```

The same pattern appears in:
- `EvmErc20.sol` `withdrawToEthereum` (lines 65–76), calling `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`
- `EvmErc20V2.sol` `withdrawToNear` (lines 53–64)
- `EvmErc20V2.sol` `withdrawToEthereum` (lines 66–77)

The EVM `call` opcode returns `0` on failure and `1` on success. In Solidity, a failed inner `call` does **not** revert the outer transaction — only the inner call's state changes are rolled back. Since `_burn` is executed in the outer context before the assembly block, it is not rolled back when the precompile call fails. The `res` variable is assigned but the assembly block ends with no conditional revert, so the function returns normally regardless of whether the precompile succeeded.

The ExitToNear precompile (`engine-precompiles/src/native.rs`) has multiple failure paths: the engine can be paused, the NEP-141 mapping for the ERC-20 may not exist, the input may fail validation, or the precompile may return `ExitError` for other reasons. Any of these causes the inner `call` to return `0`, which is silently ignored.

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the precompile call fails silently:
1. The user's ERC-20 tokens are burned (supply reduced, balance zeroed).
2. No NEP-141 `ft_transfer` or `withdraw` is scheduled on NEAR.
3. The user has no recourse: the ERC-20 tokens no longer exist and no NEAR-side tokens were released.

Funds are permanently destroyed with no recovery path.

### Likelihood Explanation

**Medium.** The precompile call can fail in realistic, non-admin-controlled scenarios:

- The Aurora Engine exposes a pause mechanism (`pausables.rs`). If the engine or the exit precompile is paused at the moment of the call, the precompile returns an error.
- If the NEP-141 ↔ ERC-20 mapping is not registered (e.g., a token deployed via a non-standard path), `get_nep141_from_erc20` returns an error.
- Any future precompile-level validation added to the exit path could introduce new silent failure modes.

A user calling `withdrawToNear` during a brief pause window, or on a token whose mapping is misconfigured, will lose their tokens permanently with no on-chain indication of failure (the transaction succeeds with status `1`).

### Recommendation

Check the return value of the precompile `call` and revert if it is zero:

```solidity
// EvmErc20.sol / EvmErc20V2.sol — withdrawToNear
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    _burn(sender, amount);

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    uint input_size = 1 + 20 + 32 + recipient.length;

    bool success;
    assembly {
        success := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    }
    require(success, "ExitToNear precompile call failed");
}
```

Apply the same fix to `withdrawToEthereum` in both contracts. Alternatively, restructure the functions to call the precompile first and only burn on success, though this is harder to implement atomically in EVM.

### Proof of Concept

1. Deploy `EvmErc20` or `EvmErc20V2` with a valid NEP-141 mapping.
2. Pause the Aurora Engine (or arrange for the ExitToNear precompile to return an error — e.g., by calling during a pause window).
3. Call `withdrawToNear(recipient, amount)` as a token holder.
4. Observe: the transaction succeeds (EVM status = 1), the caller's ERC-20 balance is reduced by `amount`, but no NEP-141 transfer occurs on NEAR.
5. The tokens are permanently lost — the ERC-20 supply is reduced but no NEAR-side balance is credited.
