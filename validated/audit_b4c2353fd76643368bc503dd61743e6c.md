### Title
ICRC-1 Ledger `upgrade()` Allows Setting `fee_collector` to Anonymous Principal Without Validation — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

---

### Summary

The ICRC-1 ledger's `Ledger::upgrade()` and `Ledger::from_init_args()` functions validate that the `fee_collector` account is not the same as the `minting_account`, but perform **no check** against the anonymous principal (`Principal::anonymous()`). A controller who upgrades the ledger with `ChangeFeeCollector::SetTo(Account { owner: Principal::anonymous(), subaccount: None })` will silently route all future transfer fees into a permanently inaccessible account, effectively burning them. This is the IC analog of the Algebra vault being set to `address(0)`.

---

### Finding Description

The `Ledger::upgrade()` function processes `change_fee_collector` as follows:

```rust
// rs/ledger_suite/icrc1/ledger/src/lib.rs:943-950
if let Some(change_fee_collector) = args.change_fee_collector {
    self.fee_collector = change_fee_collector.into();
    if self.fee_collector.as_ref().map(|fc| fc.fee_collector) == Some(self.minting_account)
    {
        ic_cdk::trap(
            "The fee collector account cannot be the same account as the minting account",
        );
    }
}
``` [1](#0-0) 

The only guard is against the fee collector equalling the minting account. There is no guard against `Principal::anonymous()`. The identical gap exists at initialization:

```rust
// rs/ledger_suite/icrc1/ledger/src/lib.rs:726-729
if ledger.fee_collector.as_ref().map(|fc| fc.fee_collector) == Some(ledger.minting_account)
{
    ic_cdk::trap("The fee collector account cannot be the same as the minting account");
}
``` [2](#0-1) 

The `ChangeFeeCollector` type accepts any `Account` without further restriction:

```rust
// rs/ledger_suite/icrc1/ledger/src/lib.rs:281-283
pub enum ChangeFeeCollector {
    Unset,
    SetTo(Account),
}
``` [3](#0-2) 

The `UpgradeArgs` struct exposes `change_fee_collector` as a public field reachable via the canister's upgrade endpoint: [4](#0-3) 

The Candid interface confirms this is a publicly documented upgrade parameter: [5](#0-4) 

---

### Impact Explanation

Once `fee_collector` is set to `Account { owner: Principal::anonymous(), subaccount: None }`:

1. Every subsequent `icrc1_transfer`, `icrc2_transfer_from`, and `icrc2_approve` call credits the transaction fee to the anonymous principal's account.
2. The anonymous principal cannot sign messages on the IC, so those tokens are permanently inaccessible — a ledger conservation bug where fees are silently burned into an uncontrolled sink.
3. Transfers themselves do **not** revert (unlike the Algebra case), so the failure is silent and may go undetected for an extended period.
4. The total supply accounting is distorted: fees that should reduce supply (when no collector is set) or accrue to the collector are instead locked forever.

The existing test `test_setting_fee_collector_to_minting_account` demonstrates that the developers explicitly intended to guard against invalid fee-collector destinations, confirming this gap is unintentional: [6](#0-5) 

---

### Likelihood Explanation

Triggering this requires a canister controller to perform an upgrade with the malformed argument — the same privilege level as the Algebra admin who calls `setVaultAddress`. The external report was accepted at Medium severity on exactly this basis. On the IC, SNS-governed ledgers require a governance proposal to upgrade, meaning a malicious or mistaken proposal could affect all token holders of that SNS. For ledgers controlled by a single developer key, the risk is a simple operator mistake with no on-chain safeguard.

---

### Recommendation

Add an anonymous-principal guard in both `from_init_args` and `upgrade`, mirroring the existing minting-account guard:

```rust
if let Some(fc) = self.fee_collector.as_ref() {
    if fc.fee_collector.owner == Principal::anonymous() {
        ic_cdk::trap("The fee collector account cannot be the anonymous principal");
    }
    if fc.fee_collector == self.minting_account {
        ic_cdk::trap("The fee collector account cannot be the same account as the minting account");
    }
}
```

Alternatively, reject any `ChangeFeeCollector::SetTo` whose `owner` is `Principal::anonymous()` at the point of deserialization or in the upgrade handler.

---

### Proof of Concept

1. Deploy an ICRC-1 ledger with a normal fee collector.
2. Upgrade the ledger with:
   ```
   change_fee_collector = opt variant { SetTo = record {
       owner = principal "2vxsx-fae";   // anonymous principal
       subaccount = null
   }}
   ```
3. Execute any `icrc1_transfer` with a non-zero fee.
4. Observe that the fee is credited to `2vxsx-fae`'s account — permanently inaccessible — while the transfer itself succeeds and no error is raised.
5. The intended fee collector receives nothing; the token supply accounting diverges silently. [1](#0-0) [2](#0-1)

        .expect("should successfully install ledger");
