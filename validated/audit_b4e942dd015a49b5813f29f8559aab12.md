### Title
Unchecked Precompile Call Return Value Burns ERC-20 Tokens Without Releasing NEP-141 Funds — (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20` and `EvmErc20V2` implement `withdrawToNear` and `withdrawToEthereum` by first calling `_burn` to destroy the caller's ERC-20 tokens, then invoking the exit precompile via inline assembly. The return value of the assembly `call` is captured into a local variable `res` but is **never checked**. If the precompile call fails for any reachable reason (invalid recipient encoding, input too large, precompile paused, NEP-141 mapping absent), the ERC-20 tokens are permanently destroyed while the corresponding NEP-141 tokens on NEAR are never released. The transaction succeeds silently, and the user's funds are gone with no recourse.

---

### Finding Description

In both `EvmErc20.sol` and `EvmErc20V2.sol`, the withdrawal functions follow this pattern:

```solidity
// EvmErc20.sol – withdrawToNear (line 53-63)
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked — no revert on failure
    }
}
``` [1](#0-0) 

The same pattern appears in `withdrawToEthereum` and in `EvmErc20V2`: [2](#0-1) [3](#0-2) [4](#0-3) 

The `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` has multiple reachable failure paths that return `ExitError` and cause the EVM `call` to return `0`:

- `ERR_INVALID_INPUT` — triggered when `input.len() < MIN_INPUT_SIZE` or `input.len() > MAX_INPUT_SIZE` (1024 bytes). A recipient string longer than ~991 bytes causes this. [5](#0-4) 

- `ERR_INVALID_RECEIVER_ACCOUNT_ID` — triggered when the recipient bytes are not valid UTF-8 or not a valid NEAR account ID. [6](#0-5) 

- `ERR_TARGET_TOKEN_NOT_FOUND` — triggered when the ERC-20 address has no NEP-141 mapping in storage. [7](#0-6) 

In all these cases, the EVM sub-call returns `0` (failure). Because the assembly block contains no `if iszero(res) { revert(0,0) }` guard, execution continues normally. The `_burn` that already executed is **not rolled back** — it is a state change in the parent call frame, not inside the failed sub-call. The transaction commits with the user's ERC-20 balance reduced and total supply decreased, but no NEAR-side promise is ever scheduled.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The ERC-20 tokens are destroyed from the Aurora EVM state. The NEP-141 tokens that back them remain locked in the connector contract on NEAR with no mechanism to recover them. The user has no way to re-attempt the withdrawal because their ERC-20 balance is already zero. The backing NEP-141 tokens become permanently inaccessible, constituting a permanent freeze of real user funds.

---

### Likelihood Explanation

**Medium-High.** The trigger conditions are reachable by any unprivileged EVM user:

1. A user or a contract acting on their behalf calls `withdrawToNear` with a `recipient` byte array that is not a valid NEAR account ID (e.g., contains non-ASCII bytes, exceeds 64 characters, or is longer than the 1024-byte input cap). This is a realistic mistake for any integrating contract that constructs the recipient programmatically.
2. A user calls `withdrawToNear` for an ERC-20 token whose NEP-141 mapping has not yet been registered, causing `ERR_TARGET_TOKEN_NOT_FOUND`.

No privileged access, key compromise, or governance action is required.

---

### Recommendation

Add an explicit revert on precompile call failure in the assembly block:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the precompile rejects the call for any reason, the entire transaction reverts, rolling back the `_burn` and preserving the user's balance. Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. Deploy an `EvmErc20` token that is properly registered with a NEP-141 mapping.
2. Mint tokens to address `Alice`.
3. Alice calls `withdrawToNear(recipient, amount)` where `recipient` is a byte string of length > 1024 bytes (e.g., `bytes(new string(1025))`).
4. Inside `withdrawToNear`:
   - `_burn(Alice, amount)` executes successfully; Alice's balance drops to zero.
   - The assembly `call` to the precompile fires with `input_size = 1 + 32 + 1025 = 1058` bytes.
   - The precompile's `validate_input_size` check fails (`1058 > 1024`), returning `ExitError`.
   - The EVM `call` returns `res = 0`.
   - No `revert` is issued; the assembly block exits silently.
5. The transaction is mined successfully. Alice's ERC-20 balance is zero. No NEAR-side `ft_transfer` promise was created. Alice's funds are permanently lost. [8](#0-7) [1](#0-0)
