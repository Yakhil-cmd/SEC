Audit Report

## Title
Stale ICP/XDR Conversion Rate Used Without Freshness Check Enables Excess Cycles Minting - (`rs/nns/cmc/src/main.rs`)

## Summary

The CMC's `tokens_to_cycles` function reads `state.icp_xdr_conversion_rate` and checks only for `Some`/`None`, never comparing `rate.timestamp_seconds` against the current wall-clock time. When the Exchange Rate Canister (XRC) is temporarily unavailable, the CMC silently continues minting cycles at the last stored (potentially arbitrarily stale) rate. Any unprivileged user who sends ICP to the CMC during a period when the stored rate is higher than the true market rate receives more cycles than their ICP is economically worth, constituting illegal minting.

## Finding Description

`tokens_to_cycles` at `rs/nns/cmc/src/main.rs:1900–1922` extracts only `rate.xdr_permyriad_per_icp` from the stored `IcpXdrConversionRate`, discarding the `timestamp_seconds` field entirely:

```rust
let xdr_permyriad_per_icp = state
    .icp_xdr_conversion_rate
    .as_ref()
    .map(|rate| rate.xdr_permyriad_per_icp);
```

The `IcpXdrConversionRate` struct carries `timestamp_seconds` (`rs/nns/cmc/src/main.rs:218`) but `tokens_to_cycles` never reads it. No `MAX_RATE_AGE_SECONDS` guard exists anywhere in the minting path.

The rate is refreshed every `REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS` (`rs/nns/cmc/src/exchange_rate_canister.rs:15–16`) via `canister_heartbeat` → `update_exchange_rate` (`rs/nns/cmc/src/main.rs:2397–2401`). When the XRC call fails, `schedule_next_attempt` reschedules for the next minute (`rs/nns/cmc/src/exchange_rate_canister.rs:153–161`) but **does not clear or invalidate the stored rate**. The error is only logged (`rs/nns/cmc/src/main.rs:2417–2428`).

`do_set_icp_xdr_conversion_rate` (`rs/nns/cmc/src/main.rs:1022–1030`) enforces only monotonicity (new timestamp > stored timestamp), not freshness relative to `now`. `validate_exchange_rate` (`rs/nervous_system/clients/src/exchange_rate_canister_client.rs:111–129`) checks only source counts, not age.

All three public minting endpoints (`notify_top_up`, `notify_create_canister`, `notify_mint_cycles`) call `tokens_to_cycles` and are therefore affected. The exploit path is:

1. XRC becomes temporarily unavailable (upgrade, `FailedToRetrieveRate` storm, `StablecoinRateTooFewRates`). CMC retries but keeps the last stored rate.
2. ICP market price falls below the stored `xdr_permyriad_per_icp`.
3. Attacker buys ICP at the depressed market price.
4. Attacker sends ICP to their CMC subaccount and calls `notify_top_up`.
5. CMC calls `tokens_to_cycles` → reads stale (inflated) rate → mints cycles at the old higher rate.
6. Attacker receives more cycles than their ICP is worth at the true market rate.

The `base_cycles_limit` rate limiter (`CyclesMintingLimiterSelector::BaseLimit`) bounds per-hour damage but does not prevent the attack; it only caps the rate at which excess cycles can be extracted.

## Impact Explanation

This is illegal minting of cycles: the CMC issues cycles in excess of the economic value of the ICP burned, violating the cycles conservation invariant. The impact class is **illegal minting / protocol insolvency involving ICP/Cycles assets**. Severity is **High** because the `base_cycles_limit` rate limiter constrains the per-hour extraction rate, and the attack requires the external condition of XRC unavailability coinciding with an ICP price drop — conditions that bound the realistic damage below the Critical ($1M+) threshold under normal XRC availability. However, during a sustained XRC outage (hours to days) combined with a significant ICP price decline, cumulative excess minting across multiple attackers could be substantial and constitutes concrete, measurable protocol harm.

## Likelihood Explanation

The CMC's own error-handling code explicitly anticipates `FailedToRetrieveRate`, `FailedToSetRate`, and `InvalidRate` outcomes (`rs/nns/cmc/src/main.rs:2419–2422`; `rs/nns/cmc/src/exchange_rate_canister.rs:153–161`), confirming this is a known operational scenario. XRC upgrades on the NNS subnet, transient inter-canister call failures, and `StablecoinRateTooFewRates` errors are all realistic triggers. The attack requires no special privileges — any principal can call `notify_top_up`. The attacker does not need to cause the XRC outage; they only need to observe it and act opportunistically. ICP price volatility during outages is a realistic market condition.

## Recommendation

In `tokens_to_cycles`, read `rate.timestamp_seconds` and reject conversions when the rate age exceeds a defined threshold (e.g., `MAX_RATE_AGE_SECONDS = 2 * REFRESH_RATE_INTERVAL_SECONDS = 600`):

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let rate = state.icp_xdr_conversion_rate.as_ref();
        match rate {
            Some(rate) => {
                let now = now_seconds();
                let age = now.saturating_sub(rate.timestamp_seconds);
                if age > MAX_RATE_AGE_SECONDS {
                    let error_message = format!(
                        "ICP/XDR conversion rate is stale (age: {}s, max: {}s)",
                        age, MAX_RATE_AGE_SECONDS
                    );
                    print(&error_message);
                    return Err(NotifyError::Other {
                        error_code: NotifyErrorCode::Internal as u64,
                        error_message,
                    });
                }
                Ok(TokensToCycles {
                    xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
                    cycles_per_xdr: state.cycles_per_xdr,
                }.to_cycles(amount))
            }
            None => Err(/* existing None error */),
        }
    })
}
```

This ensures that if the XRC is unavailable long enough for the rate to become economically unreliable, minting is suspended rather than proceeding at a stale rate.

## Proof of Concept

**Deterministic integration test plan (PocketIC or local replica):**

1. Initialize CMC with `exchange_rate_canister_id` set and store an initial rate: `xdr_permyriad_per_icp = 50_000`, `timestamp_seconds = T`.
2. Disable the mock XRC (return `FailedToRetrieveRate` on all calls). Advance time by `> REFRESH_RATE_INTERVAL_SECONDS` (5 min). Confirm CMC heartbeat logs the error but `state.icp_xdr_conversion_rate.timestamp_seconds` remains `T`.
3. Advance time by an additional 10 minutes (total > 10 min stale). Confirm the stored rate is unchanged.
4. Mint a ledger block sending 1 ICP to the CMC subaccount for a test canister.
5. Call `notify_top_up` from an unprivileged principal.
6. Assert the returned `Cycles` value equals `TokensToCycles { xdr_permyriad_per_icp: 50_000, cycles_per_xdr: ... }.to_cycles(1 ICP)` — i.e., computed at the stale rate, not at the "true" rate of 25_000 that would reflect a 50% price drop.
7. The test passes (demonstrating the bug): cycles minted = 2× the economically correct amount.

**Expected fix behavior:** after applying the freshness check, step 5 returns `Err(NotifyError::Other { ... "stale" ... })` instead of minting excess cycles.
