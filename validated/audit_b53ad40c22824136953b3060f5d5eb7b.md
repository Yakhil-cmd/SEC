### Title
ERC-20 Tokens Permanently Burned Without Refund on Failed `ExitToEthereum` NEAR-Side Withdrawal - (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.withdrawToEthereum()` and `EvmErc20V2.withdrawToEthereum()` burn the caller's ERC-20 tokens before calling the `ExitToEthereum` precompile. The precompile schedules a `withdraw` promise on the eth-connector NEP-141 contract with **no callback** to handle failure. If the NEAR-side `withdraw` call fails, the ERC-20 tokens are permanently destroyed with no refund path, and the corresponding NEP-141 tokens remain permanently frozen inside the Aurora engine account.

---

### Finding Description

The withdrawal flow in `EvmErc20.withdrawToEthereum()` is:

1. `_burn(_msgSender(), amount)` — ERC-20 tokens are destroyed from the caller's EVM balance.
2. An assembly `call` is made to the `ExitToEthereum` precompile (`0xb0bd02f6...`). The return value `res` is captured but **never checked** — if the precompile itself fails, the burn is already committed and the function returns successfully.
3. If the precompile succeeds, it schedules a `PromiseArgs::Create(withdraw_promise)` targeting the eth-connector's `withdraw` method.

The critical issue is in step 3: the promise is created with `PromiseArgs::Create` — there is **no callback** attached. If the eth-connector's `withdraw` call fails (e.g., the connector is paused, the Aurora account's NEP-141 balance is insufficient, or any other runtime error), the failure is silently ignored. The ERC-20 tokens are already burned and cannot be recovered.

This contrasts with `ExitToNear`, which has a conditional `error_refund` feature that attaches a `exit_to_near_precompile_callback` to re-mint burned tokens on failure. `ExitToEthereum` has **no equivalent mechanism** at all.

**Root cause chain:**

- `EvmErc20.withdrawToEthereum()` burns tokens then calls precompile without checking `res`: [1](#0-0) 

- `EvmErc20V2.withdrawToEthereum()` has the identical pattern: [2](#0-1) 

- `ExitToEthereum::run()` creates only a bare `PromiseArgs::Create` with no failure callback: [3](#0-2) 

- Compare with `ExitToNear`, which conditionally attaches a callback for refund (only when `error_refund` feature is enabled): [4](#0-3) 

- The `exit_to_near_precompile_callback` re-mints ERC-20 tokens on failure — no equivalent exists for `ExitToEthereum`: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the eth-connector's `withdraw` call fails after `withdrawToEthereum` is invoked:
- The user's ERC-20 tokens are permanently burned (destroyed in the EVM).
- The corresponding NEP-141 tokens remain locked inside the Aurora engine account with no mechanism to release or recover them.
- There is no admin recovery path: the ERC-20 supply is reduced but the NEP-141 backing is not, creating a permanent accounting discrepancy and frozen funds.

The same issue applies to the assembly `call` return value: if the precompile itself fails (e.g., eth connector account not configured, input parsing error), `res = 0` is silently ignored, the burn is committed, and no promise is ever scheduled. [6](#0-5) 

---

### Likelihood Explanation

**Medium.** The eth-connector's `withdraw` can fail under realistic conditions:
- The eth-connector contract is paused (a supported operational state, as shown by the `pa_pause_feature` mechanism in tests).
- The Aurora engine account's NEP-141 balance is insufficient due to accounting drift.
- The eth-connector contract is upgraded or migrated and temporarily unavailable.

Any user calling `withdrawToEthereum` during such a window loses their tokens permanently. The `ExitToNear` path has a partial mitigation (`error_refund` feature), but `ExitToEthereum` has none. [7](#0-6) 

---

### Recommendation

1. **Add a failure callback to `ExitToEthereum`** analogous to `exit_to_near_precompile_callback`. On failure, re-mint the burned ERC-20 tokens to the original sender.
2. **Check the `res` return value** in `EvmErc20.withdrawToEthereum()` and `EvmErc20V2.withdrawToEthereum()` and revert if the precompile call fails, so the burn is not committed on precompile failure.

```diff
// EvmErc20.sol
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    bytes32 amount_b = bytes32(amount);
    bytes20 recipient_b = bytes20(recipient);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient_b);
    uint input_size = 1 + 32 + 20;
    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
+       if iszero(res) { revert(0, 0) }
    }
}
```

For the NEAR-side failure, attach a callback promise that re-mints the burned tokens if the `withdraw` call fails, mirroring the `error_refund` pattern used in `ExitToNear`.

---

### Proof of Concept

1. User holds 100 units of an ERC-20 token on Aurora (backed by 100 NEP-141 tokens held by the Aurora engine account).
2. The eth-connector is paused by its admin (a supported operational action).
3. User calls `EvmErc20.withdrawToEthereum(recipient, 100)`.
4. `_burn(user, 100)` executes — ERC-20 balance goes to 0.
5. `ExitToEthereum` precompile schedules `withdraw` on the eth-connector.
6. The eth-connector's `withdraw` fails because it is paused.
7. No callback fires. The ERC-20 tokens are gone. The 100 NEP-141 tokens remain locked in the Aurora engine account forever.
8. The user has lost 100 tokens with no recovery path. [1](#0-0) [3](#0-2)
