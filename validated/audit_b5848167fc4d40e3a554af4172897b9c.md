### Title
Missing Owner Check in `deploy_upgrade` Allows Anyone to Force Deployment of Staged Upgrades - (File: engine/src/lib.rs)

### Summary

The `deploy_upgrade` entrypoint in the Aurora Engine NEAR contract is missing a `require_owner_only` access control check. While only the owner can stage new WASM code via `stage_upgrade`, any unprivileged NEAR account can call `deploy_upgrade` to force the deployment of that staged code once the delay has elapsed. This was explicitly documented as a security fix in v2.4.0 (`CHANGES.md`: "Security improvement: only Engine contract owner can use the `deploy_upgrade` method") but the check is absent in the current production code.

### Finding Description

The two-step upgrade lifecycle is:

1. `stage_upgrade` — owner-only, uploads new WASM and records `delay_block_height`
2. `deploy_upgrade` — **no owner check**, callable by anyone after the delay

`stage_upgrade` correctly enforces `require_owner_only`: [1](#0-0) 

`deploy_upgrade` only checks `require_running` and the block-height delay, with no caller restriction: [2](#0-1) 

The `require_owner_only` helper exists and is used consistently everywhere else: [3](#0-2) 

The CHANGES.md entry for v2.4.0 explicitly records this check as a deliberate security improvement that was later regressed: [4](#0-3) 

### Impact Explanation

The critical scenario is:

1. The owner calls `stage_upgrade` with new WASM code (only the owner can do this).
2. After staging, the owner discovers a critical vulnerability in the new code (e.g., incorrect accounting, reentrancy, or broken access control in the new binary).
3. The owner wants to abort the upgrade. There is no `cancel_upgrade` function — the only recourse is to re-stage with different code, which resets the delay timer.
4. Once the original delay elapses, **any NEAR account** can call `deploy_upgrade` and force the deployment of the vulnerable code before the owner can re-stage a fix.
5. The attacker then exploits the vulnerability in the newly deployed engine to steal user funds or permanently freeze them.

Even without a bug in the staged code, forced deployment at an attacker-chosen moment (e.g., during high congestion, before the owner is ready) removes the owner's ability to control the rollout window, which is the entire purpose of `upgrade_delay_blocks`. [5](#0-4) 

### Likelihood Explanation

- The Aurora Engine is actively upgraded; `stage_upgrade` is a routine operation.
- Any NEAR account (zero privilege required) can call `deploy_upgrade` after the delay.
- The attacker only needs to monitor on-chain state for a staged upgrade and wait for `block_height > CODE_STAGE_KEY`.
- The scenario where an owner discovers a post-staging bug and needs to abort is realistic for any non-trivial upgrade.

### Recommendation

Add `require_owner_only` to `deploy_upgrade`, restoring the security property that was intentionally introduced in v2.4.0:

```rust
pub extern "C" fn deploy_upgrade() {
    let mut io = Runtime;
    let state = state::get_state(&io).sdk_unwrap();
    require_running(&state)
        .map_err(ContractError::msg)
        .sdk_unwrap();
    // Restore the v2.4.0 security fix:
    require_owner_only(&state, &io.predecessor_account_id())
        .map_err(ContractError::msg)
        .sdk_unwrap();
    let index = internal_get_upgrade_index();
    if io.block_height() <= index {
        sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
    }
    Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
    io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
}
```

Additionally, consider adding a `cancel_upgrade` function (owner-only) that clears `CODE_KEY` and `CODE_STAGE_KEY` so the owner can abort a staged upgrade before the delay expires.

### Proof of Concept

1. Owner calls `stage_upgrade(new_wasm_bytes)` — succeeds, stores code and sets `CODE_STAGE_KEY = current_block + upgrade_delay_blocks`.
2. Owner discovers a critical bug in `new_wasm_bytes`.
3. Attacker monitors NEAR state, observes `block_height > CODE_STAGE_KEY`.
4. Attacker calls `deploy_upgrade()` from any NEAR account — **succeeds** because there is no `require_owner_only` check.
5. `Runtime::self_deploy` replaces the engine contract with the vulnerable WASM.
6. Attacker exploits the vulnerability to drain user ETH balances or freeze the engine. [2](#0-1) [1](#0-0)

pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {

pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}
