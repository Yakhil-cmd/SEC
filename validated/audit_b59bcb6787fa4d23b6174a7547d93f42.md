### Title
Step-Function Pubdata Pricing Due to Floor Division in `native_per_pubdata` — (`File: basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The ZKsync OS bootloader computes `native_per_pubdata` (the native resource cost per pubdata byte) using floor/truncating integer division (`wrapping_div`), while `native_per_gas` is computed using ceiling division (`div_ceil`). This asymmetry causes the effective pubdata cost to be a step function of `pubdata_price` rather than the intended linear function, systematically undercharging users for pubdata and enabling pubdata spam when `pubdata_price < native_price`.

---

### Finding Description

In `validation_impl.rs`, the two key pricing ratios are computed as:

```rust
// native_per_gas: ceiling division — rounds UP
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [1](#0-0) 

```rust
// native_per_pubdata: floor division — rounds DOWN
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [2](#0-1) 

The same floor division appears in the public API helper:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

`native_per_pubdata` is then used in `get_resources_to_charge_for_pubdata` to compute the actual native resource charge for pubdata:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [4](#0-3) 

And in `create_resources_for_tx` to deduct intrinsic pubdata overhead from the native budget:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
let native_limit = match native_limit.checked_sub(intrinsic_pubdata_overhead) { ... };
``` [5](#0-4) 

The `native_used` from pubdata feeds into the `delta_gas` calculation in `compute_gas_refund`:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
``` [6](#0-5) 

**The step-function behavior**: For any `pubdata_price` in the range `[k × native_price, (k+1) × native_price − 1]`, `native_per_pubdata` is always `k`. The effective native cost per pubdata byte is a staircase function of `pubdata_price`, not a smooth linear function. Critically, when `pubdata_price < native_price`, `native_per_pubdata = 0`, meaning users pay **zero** native resources for all pubdata they generate.

---

### Impact Explanation

**Systematic pubdata undercharging**: The operator sets `pubdata_price` to recover the L1 data availability cost. Due to floor division, the effective native charge per pubdata byte is always `floor(pubdata_price / native_price)` instead of the intended `pubdata_price / native_price`. The undercharge per byte is up to `(native_price − 1) / native_price` native units.

**Zero-cost pubdata spam**: When `pubdata_price < native_price` (a valid operator configuration), `native_per_pubdata = 0`. Users can generate pubdata up to the block's pubdata limit at zero native resource cost. Since `native_used` is lower, `delta_gas` is also lower, so the user pays less gas (and thus less ETH/token) than the operator intended.

**Concrete example**: With `native_price = 100` and `pubdata_price = 199` (operator intends ~2 native/byte), `native_per_pubdata = 1`. A user generating 10,000 bytes of pubdata pays 10,000 native instead of ~19,900 native. With `native_per_gas = 2`, this reduces `delta_gas` by ~4,950 gas units, saving the user `4,950 × gas_price` in fees — roughly half the intended pubdata fee.

---

### Likelihood Explanation

Any unprivileged L2 transaction sender can trigger this. The attacker simply submits transactions that generate pubdata (e.g., storage writes). No special access is required. The condition `pubdata_price < native_price` is a valid operator configuration (the operator may set a low pubdata price during low L1 congestion). Even when `pubdata_price ≥ native_price`, the floor division always undercharges by up to `(native_price − 1)` native units per pubdata byte. This is continuously exploitable on every transaction that generates pubdata.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata` to match the rounding direction used for `native_per_gas`:

```rust
// Before (floor division — undercharges pubdata):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. This ensures the effective pubdata cost is a linear (not step-function) relationship with `pubdata_price`, and eliminates the zero-cost pubdata case when `pubdata_price < native_price`.

---

### Proof of Concept

1. Operator sets `native_price = 100`, `pubdata_price = 50` (valid: pubdata is cheap during low L1 congestion).
2. `native_per_pubdata = floor(50 / 100) = 0`.
3. Attacker submits a transaction that writes to 100 new storage slots (generating ~3,200 bytes of pubdata).
4. `get_resources_to_charge_for_pubdata` charges `3200 × 0 = 0` native for pubdata.
5. `native_used` reflects only computational native, not pubdata native.
6. `delta_gas` is lower than intended → `gas_used` is lower → attacker pays less fee.
7. The operator bears the full L1 data availability cost for the pubdata, but collected insufficient fees to cover it.
8. Repeat across many transactions to drain operator margin on pubdata costs.

The same attack works at any `pubdata_price` that is not an exact multiple of `native_price` — the undercharge is proportional to `pubdata_price mod native_price`.
