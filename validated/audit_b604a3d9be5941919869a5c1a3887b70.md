### Title
Unvalidated `attached_deposit` in EVM Transaction Entry Points Causes Permanent NEAR Fund Loss - (File: `engine/src/contract_methods/evm_transactions.rs`)

---

### Summary

The `submit`, `submit_with_args`, `call`, and `deploy_code` functions in `engine/src/contract_methods/evm_transactions.rs` do not validate that `attached_deposit` is zero. Any NEAR attached by a caller to these functions is silently accepted and permanently locked inside the Aurora Engine contract, with no refund path.

---

### Finding Description

The external report's vulnerability class is: **a payment-receiving function accepts more value than it should, causing the excess to be silently consumed**. In CosmWasm the vector is multiple coin types in `info.funds`; in Aurora Engine (NEAR), the analog is a non-zero `attached_deposit` on functions that require no NEAR payment.

The four public EVM-execution entry points exported from `engine/src/lib.rs` are:

- `submit` → `contract_methods::evm_transactions::submit`
- `submit_with_args` → `contract_methods::evm_transactions::submit_with_args`
- `call` → `contract_methods::evm_transactions::call`
- `deploy_code` → `contract_methods::evm_transactions::deploy_code`

None of these functions check `env.attached_deposit()`. The `Env` trait exposes `attached_deposit() -> u128` and the SDK provides `assert_one_yocto()` for the exact-one-yocto pattern, but no analogous guard (e.g., `assert_zero_deposit`) is applied here. [1](#0-0) 

Compare with functions that **do** guard their deposit:

- `withdraw` calls `env.assert_one_yocto()` before proceeding.
- `storage_unregister` and `storage_withdraw` both call `env.assert_one_yocto()`. [2](#0-1) 

The `assert_one_yocto` helper is defined in the `Env` trait and checks `attached_deposit() == 1`: [3](#0-2) 

No equivalent zero-deposit guard exists for the EVM transaction functions. The core `submit_with_alt_modexp` function reads `env` for chain ID, block height, etc., but never reads or validates `env.attached_deposit()`: [4](#0-3) 

In NEAR Protocol, when a function call succeeds, any `attached_deposit` that was not explicitly refunded via a promise stays in the callee contract's balance. There is no automatic refund on success. The Aurora Engine contract has no mechanism to return excess NEAR to callers of these functions.

---

### Impact Explanation

Any NEAR attached to a `submit`, `submit_with_args`, `call`, or `deploy_code` invocation is permanently transferred into the Aurora Engine contract's balance. The caller has no recourse: the EVM execution succeeds, the NEAR is gone, and no refund promise is created. This constitutes **permanent freezing of user funds** (from the user's perspective, the NEAR is irrecoverable without admin intervention).

---

### Likelihood Explanation

The likelihood is **low-to-moderate**:

- A user scripting NEAR calls (e.g., via `near-cli` or a custom dApp) could accidentally include a `deposit` field on a `submit` call.
- A dApp that wraps both NEAR-native and EVM calls in a single UI could misconstruct the call.
- A user confused about the Aurora interface (expecting to pay NEAR for gas) might intentionally attach NEAR.
- The NEAR protocol requires explicit deposit attachment, so this is not automatic, but the absence of a guard means any such mistake is unrecoverable.

---

### Recommendation

Add a zero-deposit assertion at the top of each affected function, analogous to the existing `assert_one_yocto` pattern:

```rust
// In submit, submit_with_args, call, deploy_code:
if env.attached_deposit() != 0 {
    return Err(errors::ERR_ATTACHED_DEPOSIT_NOT_ZERO.into());
}
```

This mirrors the fix applied in the referenced Kryptonite report: reject the call entirely if unexpected value is attached, rather than silently accepting it.

---

### Proof of Concept

1. User holds 10 NEAR and wants to submit an EVM transaction via `submit`.
2. User (or a buggy dApp) constructs a NEAR function call to `aurora` → method `submit` with `deposit: 1_000_000_000_000_000_000_000_000` (1 NEAR) and a valid RLP-encoded EVM transaction as input.
3. The call executes: `submit` → `engine::submit` → `submit_with_alt_modexp`. No deposit check occurs.
4. The EVM transaction executes successfully (e.g., a simple ETH transfer).
5. The 1 NEAR `attached_deposit` is now part of the Aurora Engine contract's NEAR balance.
6. The user's NEAR wallet shows 1 NEAR missing. No refund promise was created. The user cannot recover the NEAR. [5](#0-4) [1](#0-0)

            handler,
        )?;

        Ok(result)
    })
}
