Audit Report

## Title
Incomplete `functions_disallowed_in_pre_initialization_swap` Allows Developer Neurons to Unilaterally Pass High-Impact Proposals During Swap Window - (File: `rs/sns/governance/src/types.rs`)

## Summary
The `PreInitializationSwap` mode was designed to protect swap participants from developer unilateralism, but `functions_disallowed_in_pre_initialization_swap` omits several high-impact proposal actions. Because swap-participant neurons do not exist until `claim_swap_neurons` is called after finalization, developer neurons hold 100% of realized voting power during the open swap window and can unilaterally adopt `UpgradeSnsToNextVersion`, `ManageLedgerParameters`, `AddGenericNervousSystemFunction`, and `ManageDappCanisterSettings` proposals before any swap participant has a neuron or any ability to oppose.

## Finding Description

**Developer neuron voting power at genesis.**
`FractionalDeveloperVotingPower::get_initial_neurons` in `rs/sns/init/src/distributions.rs` (L61–66) computes:
```rust
let developer_voting_power_percentage_multiplier =
    ((swap.initial_swap_amount_e8s as u128) * 100)
    .checked_div(swap.total_e8s as u128) ... as u64;
```
When `initial_swap_amount_e8s == total_e8s` (a valid and documented configuration), the multiplier is 100, giving developer neurons full, unreduced voting power from genesis.

**`MakeProposal` is explicitly permitted in `PreInitializationSwap` mode.**
`Mode::manage_neuron_command_is_allowed_in_pre_initialization_swap_or_err` in `rs/sns/governance/src/types.rs` (L187–192) explicitly returns `true` for `C::MakeProposal(_)` and `C::RegisterVote(_)`.

**The disallowed-proposal list is incomplete.**
`functions_disallowed_in_pre_initialization_swap` in `rs/sns/governance/src/types.rs` (L253–262) blocks only six actions: `manage_nervous_system_parameters`, `transfer_sns_treasury_funds`, `mint_sns_tokens`, `upgrade_sns_controlled_canister`, `register_dapp_canisters`, `deregister_dapp_canisters`. The following high-impact actions are absent from the blocklist and confirmed in `NativeAction` (L88–131) and `From<Action> for NervousSystemFunction` (L1361–1412):
- `UpgradeSnsToNextVersion` (id=7)
- `ManageLedgerParameters` (id=13)
- `AddGenericNervousSystemFunction` (id=4)
- `ManageDappCanisterSettings` (id=14)
- `AdvanceSnsTargetVersion` (id=15)

**Electoral roll contains only developer neurons during the swap.**
`compute_ballots_for_new_proposal` in `rs/sns/governance/src/governance.rs` (L5255–5280) iterates `self.proto.neurons` with no guard for governance mode. `claim_swap_neurons` (L4431–4440) is callable only by the swap canister and only after the swap commits. Until that call, no swap-participant neuron exists in governance state. Developer neurons therefore hold 100% of realized voting power regardless of the configured multiplier.

**Exploit path:**
1. Developer deploys SNS with `initial_swap_amount_e8s == total_e8s`; developer neurons receive `voting_power_percentage_multiplier = 100`.
2. Swap opens (`Lifecycle::Open`). Swap participants transfer ICP; no participant neurons exist in SNS governance.
3. Developer calls `manage_neuron` with `MakeProposal { action: UpgradeSnsToNextVersion {} }`. Mode check passes (MakeProposal allowed). Action check passes (UpgradeSnsToNextVersion not in disallowed list).
4. `compute_ballots_for_new_proposal` builds an electoral roll containing only developer neurons. The proposal is adopted immediately with 100% of voting power.
5. The SNS governance canister is upgraded (to the next NNS-blessed version) before the swap finalizes and before any swap participant has a neuron or any ability to vote against the change.

The same path applies to `ManageLedgerParameters` (altering transaction fees mid-swap), `AddGenericNervousSystemFunction` (expanding the callable attack surface), and `ManageDappCanisterSettings` (altering dapp canister settings that swap participants believe they are investing in).

## Impact Explanation

This is a **High** severity finding. The SNS `PreInitializationSwap` mode is the primary protocol-level protection guaranteeing that swap participants cannot be harmed by developer unilateralism during the open swap window. The incomplete disallowed list breaks this guarantee. Concrete harms include: governance canister upgrade before any participant neuron exists, ledger fee changes affecting swap economics, and dapp canister setting changes affecting the assets under investment. This matches the allowed impact class: *"Significant SNS security impact with concrete user or protocol harm."*

## Likelihood Explanation

The attack requires a developer who intentionally deploys an SNS and exploits the window between swap open and swap finalization. The developer role is semi-trusted, but the entire value proposition of the decentralization swap is protection from developer unilateralism. The configuration `initial_swap_amount_e8s == total_e8s` is the default example in SNS documentation and is common. The attack window spans the full swap duration (days to weeks), providing ample time. No external dependencies, victim mistakes, or social engineering are required. Likelihood is **medium**.

## Recommendation

1. Expand `functions_disallowed_in_pre_initialization_swap` in `rs/sns/governance/src/types.rs` to include `NervousSystemFunction::upgrade_sns_to_next_version()`, `NervousSystemFunction::advance_sns_target_version()`, `NervousSystemFunction::manage_ledger_parameters()`, `NervousSystemFunction::add_generic_nervous_system_function()`, and `NervousSystemFunction::manage_dapp_canister_settings()`.
2. Alternatively, disallow `MakeProposal` entirely in `PreInitializationSwap` mode and defer all governance activity until after the swap finalizes and swap-participant neurons exist.
3. Add a test asserting that every native action except `Motion` and `ManageNervousSystemParameters` (or a defined safe set) is blocked in `PreInitializationSwap` mode, to prevent future regressions.

## Proof of Concept

**Minimal unit test plan (safe, local, no mainnet interaction):**

```rust
// In rs/sns/governance/src/governance/assorted_governance_tests.rs or similar
#[test]
fn test_upgrade_sns_to_next_version_allowed_in_pre_initialization_swap() {
    // 1. Build a GovernanceProto in PreInitializationSwap mode with one developer neuron
    //    having voting_power_percentage_multiplier = 100 and sufficient dissolve delay.
    // 2. Call make_proposal with Action::UpgradeSnsToNextVersion.
    // 3. Assert the proposal is NOT rejected by allows_proposal_action_or_err.
    // 4. Assert the proposal is adopted immediately (developer neuron holds 100% voting power).
    // This test is expected to PASS currently, demonstrating the gap.
}
```

Alternatively, use the existing `Mode::allows_proposal_action_or_err` unit test infrastructure in `rs/sns/governance/src/types.rs` to assert that `Action::UpgradeSnsToNextVersion`, `Action::ManageLedgerParameters`, `Action::AddGenericNervousSystemFunction`, and `Action::ManageDappCanisterSettings` return `Err` in `PreInitializationSwap` mode — these assertions will currently fail, confirming the vulnerability.
