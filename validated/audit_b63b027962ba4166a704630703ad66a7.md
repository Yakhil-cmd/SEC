### Title
Permanent ETH Freeze via Failed Push-Refund in `exit_to_near_precompile_callback` — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

When a contract without a `receive()` function calls the `ExitToNear` precompile to bridge ETH to NEAR, and the subsequent NEAR `ft_transfer` fails, the engine's push-refund attempt also fails. The callback panics with `ERR_REFUND_FAILURE`, but the original EVM state change (ETH moved to the exit precompile address) is already committed in a prior NEAR receipt. The ETH is permanently frozen at `exit_to_near::ADDRESS` with no user-accessible recovery path.

---

### Finding Description

**Step 1 — Token burn and promise scheduling.**
When a user's EVM contract calls `EvmErc20V2.withdrawToNear(recipient, amount)`, the contract burns the caller's ERC-20 tokens and encodes the caller's address as `refund_address` in the precompile input: [1](#0-0) 

The `ExitToNear` precompile schedules a NEAR `ft_transfer` promise and attaches `exit_to_near_precompile_callback` as the callback, with `refund_address` baked into the callback args: [2](#0-1) 

**Step 2 — NEAR receipt commits EVM state.**
The original NEAR receipt executes the EVM, moves ETH from the caller to `exit_to_near::ADDRESS`, and commits this state. The `ft_transfer` promise is then dispatched as a separate receipt.

**Step 3 — `ft_transfer` fails, callback enters refund path.**
If `ft_transfer` fails (e.g., recipient not registered with the NEP-141 contract), `exit_to_near_precompile_callback` enters the refund branch and calls `engine::refund_on_error`: [3](#0-2) 

**Step 4 — Push-refund fails for contracts without `receive()`.**
`refund_on_error` for the ETH case executes an EVM call that pushes ETH from `exit_to_near::ADDRESS` to `refund_address`: [4](#0-3) 

If `refund_address` is a contract without a `receive()` or `fallback()` function, the EVM call reverts. `refund_result.status` is not `ok()`, so `ERR_REFUND_FAILURE` is returned and the callback receipt panics: [5](#0-4) 

**Step 5 — Permanent freeze.**
In NEAR's execution model, each receipt's state changes are committed independently. The callback's panic reverts only its own state changes. The ETH committed in Step 2 (at `exit_to_near::ADDRESS`) is never recovered. No admin or user function exists to drain `exit_to_near::ADDRESS`.

The `refund_address` is set to the EVM caller's address by `EvmErc20V2.withdrawToNear` when the `error_refund` feature is enabled: [6](#0-5) 

The `mint` function in `EvmErc20` / `EvmErc20V2` is `onlyAdmin`, so the ERC-20 re-mint path succeeds. The vulnerable path is exclusively the ETH (base token) push-refund: [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**
Any ETH sent through the `ExitToNear` precompile from a contract without a `receive()` function is permanently frozen at `exit_to_near::ADDRESS` if the NEAR `ft_transfer` fails. There is no user-accessible or admin recovery mechanism in the contract.

---

### Likelihood Explanation

**Medium.** Two independent conditions must hold:

1. The EVM caller of `withdrawToNear` is a contract without a `receive()` or `fallback()` function. This is common in DeFi protocols that only handle ERC-20 tokens and never expect to receive raw ETH.
2. The NEAR `ft_transfer` fails. This occurs whenever the recipient NEAR account is not registered with the NEP-141 contract, does not exist, or the NEP-141 contract is paused.

Both conditions can arise naturally without malicious intent. A user integrating Aurora bridging into a DeFi contract that lacks a `receive()` function and specifies an unregistered NEAR recipient will permanently lose their ETH.

---

### Recommendation

Replace the push-refund pattern with a pull-refund pattern. When `refund_on_error` returns a non-ok status for the ETH case, instead of panicking:

1. Write `(refund_address, amount)` into a persistent storage mapping (e.g., keyed by `exit_to_near::ADDRESS` storage).
2. Expose a public `claim_eth_refund()` function that allows the original caller to withdraw their frozen ETH.

This mirrors the fix recommended in the external report and eliminates the permanent freeze.

---

### Proof of Concept

```solidity
// Deploy on Aurora
contract NoReceive {
    // No receive() or fallback() — cannot accept ETH

    function exitEth(address erc20, bytes memory nearRecipient, uint256 amount) external {
        // Burns ERC-20 and calls ExitToNear precompile;
        // refund_address = address(this)
        IExit(erc20).withdrawToNear(nearRecipient, amount);
    }
}
```

1. Deploy `NoReceive` on Aurora.
2. Fund `NoReceive` with bridged ETH (ERC-20 balance).
3. Call `NoReceive.exitEth(erc20Addr, "unregistered.near", amount)`.
4. EVM burns ERC-20 tokens; ETH moves to `exit_to_near::ADDRESS` (committed).
5. NEAR `ft_transfer` to `"unregistered.near"` fails (not registered).
6. `exit_to_near_precompile_callback` calls `refund_on_error`, which tries to push ETH to `address(NoReceive)`.
7. EVM call reverts — `NoReceive` has no `receive()`.
8. `ERR_REFUND_FAILURE` is returned; callback receipt panics.
9. ETH remains permanently at `exit_to_near::ADDRESS`. [8](#0-7) [9](#0-8) [10](#0-9)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;

            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
