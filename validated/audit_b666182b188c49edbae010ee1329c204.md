### Title
Race Condition in Per-IP WebSocket Subscriber Limit Allows Limit Bypass — (`rs/boundary_node/ic_boundary/src/http/handlers.rs`)

---

### Summary

A TOCTOU race condition between `logs_canister` and `logs_canister_ws` allows an unprivileged client to establish more than `max_subscribers_per_ip_per_topic` concurrent WebSocket connections from a single IP to a single canister's log stream. The root cause is that `ip_cache.get_with(...)` and `counter.lock()` are not atomic with respect to `ip_cache.invalidate(...)`, creating a window where two concurrent connection attempts can each observe a counter value of `0` on different `Arc<Mutex<u16>>` instances.

---

### Finding Description

`LogsState` tracks per-`(IpAddr, Principal)` connection counts using a Moka cache mapping to `Arc<Mutex<u16>>`: [1](#0-0) 

In `logs_canister`, the check-and-increment is: [2](#0-1) 

In `logs_canister_ws`, the teardown is: [3](#0-2) 

**Exact race (with `max = 1`):**

1. One connection is open. Cache: `(ip, cid) → Arc_A<Mutex<1>>`.
2. That connection closes. `logs_canister_ws` acquires the lock on `Arc_A`, decrements to `0`, and is about to call `invalidate` — **lock is held, counter = 0, cache entry still present**.
3. Concurrently, new connection **A** calls `ip_cache.get_with(...)` — cache still has `Arc_A` — returns `Arc_A`. Then tries `counter.lock()` — **blocks** because `logs_canister_ws` holds it.
4. `logs_canister_ws` calls `ip_cache.invalidate(...)`, removing `Arc_A` from the cache, then releases the lock.
5. Concurrently, new connection **B** calls `ip_cache.get_with(...)` — cache is now **empty** — creates a fresh `Arc_B<Mutex<0>>`, inserts it. Acquires lock, sees `0 < 1`, increments to `1`. **Connection B is accepted.**
6. Connection **A** unblocks, acquires lock on `Arc_A`, sees `0 < 1` (it was decremented in step 2), increments to `1`. **Connection A is also accepted.**

Both A and B are now open. The limit of `1` is violated. `Arc_A` is now orphaned (not in the cache), so when connection A eventually closes, it calls `ip_cache.invalidate` on the key that now maps to `Arc_B` — evicting the live entry and resetting the accounting entirely, compounding the bypass.

---

### Impact Explanation

An attacker can open arbitrarily many WebSocket connections from a single IP to a single canister's log stream, bypassing the per-IP rate limit. The only remaining backstop is the global `max_subscribers_per_topic` broker limit (default 1000). By monopolizing all broker subscriber slots from one IP, the attacker can deny log access to all other clients and exhaust boundary node file descriptors / memory proportional to the number of open WebSocket connections.

---

### Likelihood Explanation

The race window is small but reliably triggerable under concurrent load. The attacker controls both the timing of connection teardown and the timing of new connection attempts. A simple concurrent stress test (e.g., a tight loop of close-then-connect from multiple goroutines/threads) reliably hits this window. The endpoint is publicly reachable with no authentication required when `obs_log_websocket` is enabled. [4](#0-3) 

---

### Recommendation

The `get_with` call and the counter check/increment must be atomic with respect to `invalidate`. One correct approach: replace the two-level structure (`Cache<key, Arc<Mutex<u16>>>`) with a single `DashMap` or `Mutex<HashMap>` so that lookup, check, increment, and invalidation all occur under one lock. Alternatively, never call `ip_cache.invalidate` from `logs_canister_ws`; instead, let Moka's TTI/TTL evict stale zero-count entries, and treat a counter value of `0` in the cache as equivalent to absent.

---

### Proof of Concept

```rust
// Pseudocode stress test
// max_subscribers_per_ip_per_topic = 1
// 1. Open connection C0 (counter = 1)
// 2. Spawn two tasks simultaneously:
//    Task A: close C0 (triggers decrement → 0 → invalidate)
//    Task B: open C1 (calls get_with → gets Arc_A while lock held → blocks)
// 3. Spawn Task C immediately after Task A releases lock:
//    Task C: open C2 (calls get_with → cache empty → new Arc_B → counter=1 → accepted)
// 4. Task B unblocks: Arc_A counter=0 → increments to 1 → accepted
// Assert: C1 and C2 are both open simultaneously → limit violated
```

The existing test at line 347–364 already demonstrates the sequential case works; a concurrent variant racing close and open at the exact boundary reproduces the bypass. [5](#0-4)
