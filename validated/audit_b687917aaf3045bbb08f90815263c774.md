Audit Report

## Title
`SenderInfoContent` Lacks Request-Context Binding, Enabling Cross-Canister Attestation Replay — (`rs/types/types/src/messages/http.rs`, `rs/validator/src/ingress_validation.rs`)

## Summary
`SenderInfoContent` signs only the raw `info` blob, omitting `canister_id`, `sender`, and `ingress_expiry`. Because the inner `sender_info.sig` canister signature is not bound to any request-context field, a user who legitimately obtains an attestation for one canister can replay it verbatim against any other canister. Additionally, `validate_sender_info` performs no certificate-age check, so a revoked attestation remains cryptographically valid indefinitely. Both defects are confirmed in production code.

## Finding Description
`SenderInfoContent` is defined at `rs/types/types/src/messages/http.rs` L342–348:

```rust
pub struct SenderInfoContent<'a>(pub &'a [u8]);

impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.0);   // only info bytes
    }
}
```

The full signed payload is therefore `\x0Eic-sender-info` ‖ `info_bytes` — with no `canister_id`, `sender`, or `ingress_expiry`.

`verify_sender_info_canister_sig` at `rs/validator/src/ingress_validation.rs` L530–543 constructs the signable content as `SenderInfoContent(&sender_info.info)` and verifies only that (a) `sender_pubkey` is a valid canister-sig key, (b) the canister ID in that key matches `sender_info.signer`, and (c) the BLS-backed certificate verifies. No request-context field is checked.

`validate_sender_info` at L459–488 calls `verify_sender_info_canister_sig` and returns immediately on success; it never extracts the certificate `time` field and never compares it against `current_time`.

The `sender_info` struct (including `sig`) is hashed into the `MessageId` at `rs/types/types/src/messages/ingress_messages.rs` L113–129, so the envelope-level `sender_sig` does bind the specific `sender_info.sig` bytes to the specific request. However, this only prevents a third party from swapping the `sig` field; it does not prevent the legitimate holder of the key from reusing the same `sender_info.sig` in a fresh request targeting a different canister, because the inner canister signature itself carries no canister-ID or sender binding.

**Cross-canister replay path:**
1. User A obtains `sender_info.sig` over `info_bytes = b"role:admin"` from Internet Identity for a request to canister X.
2. User A constructs a new `HttpCanisterUpdate` targeting canister Y, embedding the identical `sender_info.sig`.
3. User A signs the new `MessageId` (which now includes `canister_Y` and a fresh `ingress_expiry`) with their own canister-sig key — this is valid because the envelope signature is fresh.
4. `validate_sender_info` passes: `sender_pubkey` encodes Internet Identity's canister ID, `pubkey_canister_id == sender_info.signer`, and `SenderInfoContent(b"role:admin")` verifies against the reused certificate.
5. Canister Y's `msg_caller_info_data()` returns `b"role:admin"` and grants elevated access.

**Stale-attestation path:**
1. Internet Identity certifies `info_bytes` for User A at time T.
2. Internet Identity later revokes the attestation by updating its certified data.
3. User A retains the old certificate. Because `verify_canister_sig` checks only the BLS signature on the certificate root and the Merkle proof, and `validate_sender_info` never compares the certificate timestamp to `current_time`, the old certificate continues to pass verification indefinitely.

## Impact Explanation
Any canister that calls `msg_caller_info_data()` and makes access-control decisions on the returned blob is affected. The `msg_caller_info_data` system API is implemented in production embedder code (`rs/embedders/src/wasmtime_embedder/system_api.rs`) and exposed to all canisters. An unprivileged user who has legitimately obtained one `sender_info.sig` can replay it against arbitrary target canisters or continue using it after revocation, causing those canisters to act on false or revoked attestation data. This matches the medium-severity impact class: *forged or stale certified response accepted under constrained conditions, with moderate security impact*.

## Likelihood Explanation
The attacker must be the legitimate holder of the canister-sig key pair (i.e., the same principal who originally obtained the attestation) and must target a canister that makes security decisions based on `msg_caller_info_data()`. No special privileges beyond a valid prior attestation are required. The `sender_info` feature is new and actively used (integration tests exist in `rs/tests/crypto/ingress_verification_test.rs`), and the system API is available to all canisters on all subnets. Likelihood is medium.

## Recommendation
1. **Bind `SenderInfoContent` to the request context.** Extend the struct to carry `canister_id` and `sender`, and include them in the signed bytes before `info`:
```rust
pub struct SenderInfoContent<'a> {
    pub canister_id: &'a [u8],
    pub sender: &'a [u8],
    pub info: &'a [u8],
}
impl SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.canister_id);
        bytes.extend_from_slice(self.sender);
        bytes.extend_from_slice(self.info);
    }
}
```
Pass `canister_id` and `sender` from the request into `validate_sender_info` and `verify_sender_info_canister_sig`.

2. **Enforce a certificate-age deadline.** In `validate_sender_info`, extract the `time` field from the canister-signature certificate and reject it if `current_time - cert_time > MAX_INGRESS_TTL`.

## Proof of Concept
```
1. Call the signing canister (e.g. Internet Identity) to obtain a valid
   sender_info.sig over info_bytes = b"role:admin" for canister X.

2. Construct HttpCanisterUpdate {
       canister_id: canister_Y,          // different target
       method_name: "privileged_action",
       sender: user_A,
       ingress_expiry: <fresh>,
       sender_info: Some(RawSignedSenderInfo {
           info: b"role:admin",
           signer: II_canister_id,
           sig: <reused sig from step 1>,
       }),
   }

3. Sign the new MessageId with user_A's canister-sig key (envelope sig is fresh).

4. Submit to the replica. validate_sender_info passes because:
   - sender_pubkey encodes II_canister_id ✓
   - pubkey_canister_id == sender_info.signer ✓
   - SenderInfoContent(b"role:admin") verifies against the reused certificate ✓
   - No canister_id or expiry check is performed ✓

5. Canister Y's msg_caller_info_data() returns b"role:admin".

Deterministic integration test: use PocketIC to issue a sender_info.sig
targeting canister X, then submit an otherwise-valid update call to canister Y
with the same sig; assert msg_caller_info_data() returns the attested bytes
and that the validator accepts the request.
```
