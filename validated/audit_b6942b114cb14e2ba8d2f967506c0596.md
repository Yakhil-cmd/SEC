Audit Report

## Title
`estimate_withdrawal_fee` Permanently Drains `available_utxos` When Called as a Replicated Update — (`rs/dogecoin/ckdoge/minter/src/main.rs`)

## Summary
`estimate_withdrawal_fee` is annotated `#[query]` but internally calls `mutate_state` with a mutable borrow of `available_utxos`. The inline comment asserting that state changes are discarded in replicated mode is factually incorrect: on the Internet Computer, a `#[query]`-annotated method invoked as an update call (replicated query) **does** commit state changes. Because the UTXO selection algorithm removes UTXOs from `available_utxos` without reinserting them on success, any unprivileged caller can silently drain the minter's spendable UTXO pool, causing all subsequent legitimate withdrawals to fail with `NotEnoughFunds`.

## Finding Description

**Root cause — `main.rs` lines 98–128:**
`estimate_withdrawal_fee` is `#[query]` but calls `mutate_state`, passing `&mut s.available_utxos` to `estimate_retrieve_doge_fee`. [1](#0-0) 

**Mutation path — `fees/mod.rs` lines 130–151:**
`estimate_retrieve_doge_fee` accepts `available_utxos: &mut UtxoSet` and delegates directly to `ic_ckbtc_minter::queries::estimate_withdrawal_fee` with the same mutable reference. [2](#0-1) 

**UTXO removal — `queries.rs` line 57:**
`estimate_withdrawal_fee` calls `utxos_selection(withdrawal_amount, available_utxos, 1)`, which calls `greedy`, which calls `available_utxos.remove(&utxo)` for each selected UTXO. [3](#0-2) 

**No reinsertion on success — `lib.rs` lines 1070–1099:**
`greedy` only reinserts UTXOs into `available_utxos` on the failure path (not enough funds). On success, the selected UTXOs are returned in `solution` and subsequently dropped after fee computation — they are never reinserted. [4](#0-3) 

**Proof the comment is wrong — `main.rs` lines 248–257:**
`http_request` in the same file explicitly guards against replicated execution with `ic_cdk::api::in_replicated_execution()`. This guard is present for `http_request` but absent from `estimate_withdrawal_fee`, confirming the developers are aware of the behavior but failed to apply the guard consistently. [5](#0-4) 

**Why the existing test does not catch this — `tests.rs` lines 609–615:**
The test calls `estimate_withdrawal_fee` via `update_call` (replicated path) and asserts that `get_known_utxos` is unchanged. However, `get_known_utxos` reads from `utxos_state_addresses` (per-account UTXO map), not from `available_utxos`. Draining `available_utxos` leaves `utxos_state_addresses` untouched, so the assertion passes even when the minter's spendable UTXO pool has been silently emptied. [6](#0-5) 

**Test utility confirms the replicated call path is accepted — `minter.rs` lines 147–163:**
The test helper already calls `estimate_withdrawal_fee` via `env.update_call` (not `query_call`), confirming the runtime accepts the call on the update path. [7](#0-6) 

## Impact Explanation

`available_utxos` is the pool from which the minter selects inputs when building withdrawal transactions. After the attack, `available_utxos` is empty or reduced below the required threshold. Every subsequent call to `retrieve_doge_with_approval` fails with `NotEnoughFunds` / `AmountTooHigh`, blocking all user withdrawals from the ckDOGE minter. No funds are transferred to the attacker; the impact is a complete denial-of-service on the withdrawal path and corruption of UTXO accounting. Recovery requires a canister upgrade that re-derives `available_utxos` from the event log.

This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS with concrete user and protocol harm** on a Chain Fusion / ck-token financial integration component.

## Likelihood Explanation

- Requires no privilege, no key, no governance vote — any principal (including anonymous) can send an ingress update call to a `#[query]` endpoint.
- The IC protocol explicitly supports calling query methods as update calls (replicated queries); this is standard, documented behavior in the IC interface specification.
- The attack is a single ingress message with a valid `amount` argument.
- The test utility already exercises this exact call path (`update_call` to `estimate_withdrawal_fee`), confirming the runtime accepts it.
- The attack is repeatable: each call drains additional UTXOs until the pool is empty.

## Recommendation

Apply the same guard already present in `http_request`:

```rust
#[query]
fn estimate_withdrawal_fee(
    arg: EstimateFeeArg,
) -> Result<WithdrawalFee, EstimateWithdrawalFeeError> {
    if ic_cdk::api::in_replicated_execution() {
        ic_cdk::trap("update call rejected");
    }
    // ...
}
```

Alternatively, replace `mutate_state` with `read_state` and rewrite `estimate_retrieve_doge_fee` to operate on a cloned snapshot of `available_utxos`, so no mutation of live state occurs regardless of call mode.

The integration test should also be updated to assert that `available_utxos` (not just `get_known_utxos`) is unchanged after calling `estimate_withdrawal_fee` via the update path.

//    with a valid withdrawal amount.
// 4. Assert available_utxos count is UNCHANGED — this assertion FAILS,
//    proving permanent state mutation.

    minter.id,
    Principal::anonymous(),
    "estimate_withdrawal_fee",
    Encode!(&EstimateFeeArg { amount: Some(RETRIEVE_DOGE_MIN_AMOUNT) }).unwrap(),
