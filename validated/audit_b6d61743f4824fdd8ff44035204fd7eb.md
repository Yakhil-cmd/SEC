Audit Report

## Title
VIP Callers Blocked by Non-VIP Slot Exhaustion in `try_borrow_slot` — (`rs/nervous_system/clients/src/management_canister_client.rs`)

## Summary
`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` unconditionally rejects all callers when `available_slot_count == 0`, including VIP callers (NNS canisters) that are designed to consume zero slots. An unprivileged attacker can exhaust all 167 shared slots via the public `canister_status` endpoint on NNS Root, permanently blocking governance-triggered `take_canister_snapshot` and `load_canister_snapshot` for the duration of the attack.

## Finding Description

**Root cause — unconditional exhaustion guard in `try_borrow_slot`:**

In `rs/nervous_system/clients/src/management_canister_client.rs` at line 265, VIP callers are assigned `used_slot_count = 0`, meaning they should not consume a slot. However, the exhaustion guard at line 269 fires before the subtraction and returns `Err` for all callers unconditionally:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };  // VIP = 0

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if *available_slot_count == 0 {          // ← fires for VIP too
                let code = RejectCode::SysTransient as i32;
                return Err((code, message));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
``` [1](#0-0) 

A VIP caller with `used_slot_count = 0` would leave the counter unchanged if it reached the subtraction — but it never does when the counter is at 0.

**Attacker-controlled entrypoint — public `canister_status`:**

NNS Root exposes `canister_status` as an unrestricted `#[update]` endpoint with no caller check: [2](#0-1) 

Each call borrows a slot for the full round-trip duration of the inner management canister call.

**Fixed slot pool of 167:** [3](#0-2) 

**VIP determination is correct but insufficient:**

Governance calling `take_canister_snapshot` or `load_canister_snapshot` correctly receives `is_caller_vip = true`: [4](#0-3) 

But `try_borrow_slot` still rejects it when `available_slot_count == 0`.

**Governance-only endpoints that are blocked:** [5](#0-4) 

**Existing test gap:**

The unit test in `tests.rs` uses `SLOTS = 2` and starts VIP calls at `pre_flight_pause_duration = 0ms` (before non-VIP calls at 50ms), so the pool never reaches 0 when a VIP call arrives. The `available_slot_count == 0` edge case with a VIP caller is never exercised. [6](#0-5) 

## Impact Explanation

An attacker can sustain a DoS against NNS Root's governance-triggered disaster recovery operations (`take_canister_snapshot`, `load_canister_snapshot`) indefinitely at the cost of cycles alone. These operations are the primary mechanism for rolling back corrupted NNS canisters (e.g., ledger, governance). Blocking them during a time-sensitive incident constitutes a significant NNS security impact with concrete protocol harm. This matches the **High** bounty impact: *"Application/platform-level DoS... or subnet availability impact not based on raw volumetric DDoS"* and *"Significant NNS... security impact with concrete user or protocol harm."*

## Likelihood Explanation

No special privileges are required. Any canister on the IC can call NNS Root's public `canister_status` endpoint. The IC execution model allows 167 concurrent in-flight inter-canister calls from a single canister. An attacker sends 167 `canister_status` messages in rapid succession; NNS Root suspends each at the `await` point, leaving all 167 management canister calls in-flight simultaneously. The attacker replenishes calls as they complete, sustaining `available_slot_count == 0` continuously. The only cost is cycles.

## Recommendation

Skip the exhaustion guard entirely for VIP callers in `try_borrow_slot`, since they consume zero slots and cannot deplete the pool:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if !self.is_caller_vip && *available_slot_count == 0 {
                let code = RejectCode::SysTransient as i32;
                return Err((code, "Unavailable. Maybe, try again later?".to_string()));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

Additionally, add a unit test that sets `SLOTS = 0` (or exhausts to 0 with non-VIP calls first) and then asserts a VIP call succeeds.

## Proof of Concept

State-machine test outline:

1. Initialize `LimitedOutstandingCallsManagementCanisterClient` with `capacity = 167` and `is_caller_vip = false` for 167 concurrent non-VIP calls; drive `available_slot_count` to 0 by holding all 167 loans open (do not drop `SlotLoan`).
2. Assert `available_slot_count == 0`.
3. Construct a new client with `is_caller_vip = true` (simulating Governance) and call `try_borrow_slot()`.
4. Assert the result is `Err` with reject code `SysTransient` — confirming VIP is blocked despite `used_slot_count = 0`.

For an integration test: deploy an attacker canister on a local PocketIC NNS subnet, send 167 concurrent `canister_status` calls to NNS Root with a stalling mock inner client, then call `load_canister_snapshot` from the Governance canister principal and assert `SysTransient` is returned.
