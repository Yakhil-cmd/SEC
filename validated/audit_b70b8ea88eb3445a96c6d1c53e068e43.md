### Title
`native_per_pubdata` Truncates to Zero via Integer Division When `pubdata_price < native_price`, Enabling Free Pubdata Generation — (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS's L2 transaction validation, `native_per_pubdata` is computed using truncating integer division (`wrapping_div`). When the operator-set `pubdata_price` is less than `native_price`, the result truncates to zero. A zero `native_per_pubdata` means every byte of pubdata a transaction generates costs zero native resources, allowing any unprivileged user to generate large amounts of pubdata (storage writes) without paying the native resource cost the protocol must absorb to publish that data to L1.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` at line 142:

```rust
// We checked native_price != 0 above
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is Rust's standard integer (floor) division. When `pubdata_price < native_price`, the quotient is `0`. There is no guard, minimum-of-1 clamp, or `div_ceil` applied here, unlike `native_per_gas` which uses `div_ceil`:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(...)
```

The zero `native_per_pubdata` propagates through the entire transaction lifecycle:

**1. Intrinsic pubdata overhead becomes zero** (`gas_helpers.rs` line 352):
```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// 0 * intrinsic_pubdata = 0 → no native deducted for intrinsic pubdata
```

**2. Execution pubdata cost becomes zero** (`gas_helpers.rs` lines 430–431):
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)  // any_amount * 0 = 0
    .ok_or(out_of_native_resources!())?;
```

**3. The `delta_gas` adjustment in refund calculation ignores pubdata** (`refund_calculation.rs` lines 69–73):
```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
```
Because pubdata consumed zero native resources, `native_used` is lower, `delta_gas` is lower, and the user receives a larger gas refund — further reducing what they pay.

The same truncation exists in the `api/src/helpers.rs` mirror function `validate_l2_tx_intrinsic_native_resources` (line 427):
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

---

### Impact Explanation

When `native_per_pubdata = 0`, a user can write to as many storage slots as their EVM gas limit allows, generating large amounts of pubdata, without any native resource cost for that pubdata. The protocol must still publish all state diffs to L1 (the actual cost of pubdata), but collects nothing from the user to cover it. This is a direct financial loss to the operator/protocol proportional to the amount of pubdata generated.

The native resource system is specifically designed to ensure users pay for the off-chain proving and L1 publication costs of their transactions. Truncating `native_per_pubdata` to zero completely defeats this mechanism for pubdata, which is the dominant cost component.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a valid and realistic operator configuration. The default test metadata in `zk_ee/src/system/metadata/zk_metadata.rs` already demonstrates this:

```rust
pubdata_price: U256::from(0u64),
native_price: U256::from(10),
```

Any block where `pubdata_price` is set below `native_price` (e.g., `native_price = 100`, `pubdata_price = 50`) silently enables this. An attacker monitoring block context parameters can detect this condition and immediately submit pubdata-heavy transactions (many `SSTORE` operations) to exploit it. No privileged access is required.

---

### Recommendation

Replace the truncating `wrapping_div` with `div_ceil` (rounding up), consistent with how `native_per_gas` is computed, and enforce a minimum value of 1 when `pubdata_price > 0`:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, consistent with native_per_gas):
let native_per_pubdata = if pubdata_price.is_zero() {
    0u64
} else {
    u256_try_to_u64(&pubdata_price.div_ceil(native_price))
        .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?
        .max(1)
};
```

Apply the same fix to `api/src/helpers.rs` line 427.

---

### Proof of Concept

**Setup:** Operator sets block context with `native_price = 1000`, `pubdata_price = 999` (a realistic near-parity configuration).

**Computation:**
```
native_per_pubdata = 999 / 1000 = 0  (integer division truncates)
```

**Attacker transaction:** Submit an EIP-1559 transaction calling a contract that performs 100 `SSTORE` operations (each writing a new storage slot). Each `SSTORE` to a new slot generates 32 bytes of pubdata → 3200 bytes total pubdata.

**Expected charge:** `3200 * native_per_pubdata_correct = 3200 * 1` (with `div_ceil`) = 3200 native units.

**Actual charge:** `3200 * 0 = 0` native units.

**Result:** The attacker pays only EVM gas for the `SSTORE` opcodes (which is bounded by their gas limit), but pays zero native resource cost for the 3200 bytes of pubdata the protocol must publish to L1.
