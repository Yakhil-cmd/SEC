### Title
Missing Deadline on Scheduled XCC Promises Allows Stale Execution by Anyone — (`File: etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router's `execute_scheduled` function carries no deadline or expiry check on stored promises. Any actor can trigger execution of a user's scheduled cross-contract call at an arbitrarily late time, after market or protocol conditions have changed, causing the user to suffer worse-than-intended outcomes on time-sensitive NEAR-native DeFi interactions. There is also no cancellation path, so the user cannot withdraw from a stale scheduled promise.

---

### Finding Description

When an EVM contract on Aurora uses the XCC precompile with `CrossContractCallArgs::Delayed`, the engine calls `schedule` on the user's router sub-account, which stores the `PromiseArgs` in `scheduled_promises` keyed by a monotonically increasing nonce. [1](#0-0) 

The stored `PromiseArgs` contains fixed calldata, a fixed attached NEAR balance, and fixed gas — all set at scheduling time. There is no timestamp, no deadline field, and no expiry stored alongside the promise.

`execute_scheduled` is explicitly open to any caller: [2](#0-1) 

The function removes the promise from storage and executes it unconditionally, with no check against any time bound: [3](#0-2) 

The `PromiseArgs` type that is stored carries no deadline field at all: [4](#0-3) 

Furthermore, the router exposes no `cancel_scheduled` or equivalent function. Once a promise is scheduled, the only way to remove it from storage is to execute it. [5](#0-4) 

---

### Impact Explanation

A user schedules a promise to interact with a NEAR-native DeFi protocol (e.g., a token swap, liquidity deposit, or lending action) with amounts and calldata fixed at scheduling time. If the promise is executed significantly later — either by a malicious actor who monitors the chain and waits for unfavorable conditions, or simply due to delayed execution — the fixed calldata is applied against a changed protocol state. The user receives fewer tokens than expected, or the attached NEAR is consumed at a rate that was only acceptable at the original scheduling time. Because there is no cancellation path, the user cannot prevent this outcome once the promise is scheduled.

This maps to **High: Theft of unclaimed yield / Temporary freezing of funds** — the user's NEAR locked in the router is irrevocably committed to execution under stale conditions.

---

### Likelihood Explanation

The `execute_scheduled` function is callable by any NEAR account with no restriction. A griefing or MEV-style actor can observe scheduled promises on-chain (NEAR state is public), wait for market conditions to move against the user, and then call `execute_scheduled` at the worst moment. This requires no special privilege, no leaked key, and no admin compromise — only the ability to submit a NEAR transaction.

---

### Recommendation

1. Add an optional `deadline` (NEAR block timestamp in nanoseconds) to the `schedule` call and store it alongside the `PromiseArgs`.
2. In `execute_scheduled`, check `env::block_timestamp() <= deadline` and panic if the deadline has passed.
3. Add a `cancel_scheduled(nonce)` function restricted to the parent (Aurora engine), callable when the originating EVM transaction sender requests cancellation, so users can reclaim their locked NEAR if they no longer want the promise executed.

---

### Proof of Concept

1. Alice's EVM contract calls the XCC precompile with `CrossContractCallArgs::Delayed(swap_promise)`, where `swap_promise` encodes a call to a NEAR-native DEX to swap 100 wNEAR for token X at a rate acceptable today.
2. The engine calls `router.schedule(swap_promise)` — the promise is stored at nonce `0` with no expiry.
3. The DEX price of token X drops 40% over the next week. Alice's slippage tolerance (encoded in the fixed calldata) is 5%, so the swap would still execute but at a rate 40% worse than intended.
4. Bob (any NEAR account) calls `router.execute_scheduled({"nonce": "0"})`.
5. The swap executes: Alice receives token X at the stale, unfavorable rate. Alice has no recourse — she could not cancel the promise, and the deadline check that would have protected her does not exist. [6](#0-5) [1](#0-0)

#[near]
impl Router {
    #[init(ignore_state)]
    #[must_use]
    pub fn initialize(wnear_account: AccountId, must_register: bool) -> Self {
        // The first time this function is called there is no state and the parent is set to be
        // the predecessor account id. In subsequent calls, only the original parent is allowed to
        // call this function. The idea is that the Create, Deploy and Initialize actions are done in a single
        // NEAR batch when a new router is deployed by the engine, so the caller will be the Aurora
        // engine instance that the user's address belongs to. If we update this contract and deploy
        // a new version of it, again the Deploy and Initialize actions will be done in a single batch
        // by the engine.
        let caller = env::predecessor_account_id();
        let mut parent = LazyOption::new(StorageKey::Parent, None);
        match parent.get() {
            None => {
                parent.set(&caller);
            }
            Some(parent) => {
                // Allow self-calls to `initialize` also.
                // This happens during the upgrade flow.
                if (caller != parent) && (caller != env::current_account_id()) {
                    env::panic_str(ERR_ILLEGAL_CALLER);
                }
            }
        }

        if must_register {
            env::promise_create(
                wnear_account.clone(),
                "storage_deposit",
                b"{}",
                WNEAR_REGISTER_AMOUNT,
                WNEAR_REGISTER_GAS,
            );
        }

        let mut version = LazyOption::new(StorageKey::Version, None);
        if version.get().unwrap_or_default() != CURRENT_VERSION {
            // Future migrations would go here

            version.set(&CURRENT_VERSION);
        }

        let nonce = LazyOption::new(StorageKey::Nonce, None);
        let scheduled_promises = LookupMap::new(StorageKey::Map);
        Self {
            parent,
            version,
            nonce,
            scheduled_promises,
            wnear_account,
        }
    }

    pub fn get_version(&self) -> u32 {
        self.version.get().unwrap_or_default()
    }

    /// This function can only be called by the parent account (i.e. Aurora engine) to ensure that
    /// no one can create calls on behalf of the user this router contract is deployed for.
    /// The engine only calls this function when the special precompile in the EVM for NEAR cross
    /// contract calls is used by the address associated with the sub-account this router contract
    /// is deployed at.
    pub fn execute(&self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }

    /// Similar security considerations here as for `execute`.
    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

    /// It is intentional that this function can be called by anyone (not just the parent).
    /// There is no security risk to allowing this function to be open because it can only
    /// act on promises that were created via `schedule`.
    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }

    /// Allows the parent contract to trigger an update to the logic of this contract
    /// (by deploying a new contract to this account);
    #[payable]
    pub fn deploy_upgrade(&mut self, #[serializer(borsh)] args: DeployUpgradeParams) {
        self.assert_preconditions();

        let promise_id = env::promise_batch_create(&env::current_account_id());
        env::promise_batch_action_deploy_contract(promise_id, &args.code);
        env::promise_batch_action_function_call(
            promise_id,
            INITIALIZE,
            &args.initialize_args,
            NearToken::default(),
            INITIALIZE_GAS,
        );
        env::promise_return(promise_id);
    }

    pub fn send_refund(&self) -> Promise {
        let parent = self.get_parent().unwrap_or_else(env_panic);

        require_caller(&parent)
            .and_then(|_| require_no_failed_promises())
            .unwrap_or_else(env_panic);

        Promise::new(parent).transfer(REFUND_AMOUNT)
    }
}

pub enum CrossContractCallArgs {
    /// The promise is to be executed immediately (as part of the same NEAR transaction as the EVM call).
    Eager(PromiseArgs),
    /// The promise is to be stored in the router contract, and can be executed in a future transaction.
    /// The purpose of this is to expand how much NEAR gas can be made available to a cross contract call.
    /// For example, if an expensive EVM call ends with a NEAR cross contract call, then there may not be
    /// much gas left to perform it. In this case, the promise could be `Delayed` (stored in the router)
    /// and executed in a separate transaction with a fresh 300 Tgas available for it.
    Delayed(PromiseArgs),
}
