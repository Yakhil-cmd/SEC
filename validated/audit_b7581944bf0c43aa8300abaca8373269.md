### Title
Silo Whitelist Bypass via `call` and `deploy_code` Entry Points — (File: `engine/src/contract_methods/evm_transactions.rs`)

---

### Summary

Aurora Engine's Silo mode enforces a whitelist on both the NEAR account (`WhitelistKind::Account`) and the EVM address (`WhitelistKind::Address`) when processing transactions via `submit` / `submit_with_args`. However, the `call` and `deploy_code` entry points execute EVM code without performing any whitelist check, allowing a non-whitelisted NEAR account to bypass the silo access restrictions entirely.

---

### Finding Description

The `submit` and `submit_with_args` functions route through `engine::submit` → `submit_with_alt_modexp`, which calls `assert_access`:

```rust
// engine/src/engine.rs
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };
    if !allowed {
        return Err(EngineError { kind: EngineErrorKind::NotAllowed, gas_used: 0 });
    }
    Ok(())
}
```

`is_allow_submit` requires **both** the NEAR account and the EVM address to be whitelisted:

```rust
// engine/src/contract_methods/silo/mod.rs
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}
```

In contrast, `call` and `deploy_code` in `evm_transactions.rs` only call `require_running` and then directly invoke the EVM — **no whitelist check is performed**:

```rust
// engine/src/contract_methods/evm_transactions.rs
pub fn call<I: IO + Copy, E: Env, H: PromiseHandler>(...) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;          // ← only liveness check
        // ... no assert_access / whitelist check
        let result = engine.call_with_args(args, handler)?;
        ...
    })
}

pub fn deploy_code<I: IO + Copy, E: Env, H: PromiseHandler>(...) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;          // ← only liveness check
        // ... no assert_access / whitelist check
        let result = engine.deploy_code_with_input(input, None, handler)?;
        ...
    })
}
```

This is a direct structural analog to the reported issue: one set of entry points enforces the access control, while another set of entry points for equivalent operations does not.

---

### Impact Explanation

**Impact: High — Temporary (and potentially permanent) freezing bypass / unauthorized fund movement.**

A Silo operator enables whitelists to restrict which NEAR accounts and EVM addresses may interact with the EVM. Any non-whitelisted NEAR account can:

1. Call `call` to execute arbitrary EVM transactions (e.g., transfer ETH or ERC-20 tokens held in the EVM, interact with deployed contracts).
2. Call `deploy_code` to deploy new EVM contracts.

This completely nullifies the silo whitelist's purpose. Funds held in the EVM by a non-whitelisted address can be moved, and the restricted execution environment can be used freely, defeating the operator's access control policy.

---

### Likelihood Explanation

**Likelihood: Medium.**

The `call` and `deploy_code` methods are standard, documented NEAR contract entry points on the Aurora Engine. Any NEAR account that knows these methods exist can invoke them directly. No special privilege, leaked key, or governance capture is required — only knowledge of the alternative entry points. Silo deployments that rely on whitelists for access control are directly affected.

---

### Recommendation

Apply the same `assert_access` (or equivalent whitelist enforcement) inside `call` and `deploy_code` that is already applied in `submit_with_alt_modexp`. Specifically:

- In `call`: resolve the caller's EVM address from `predecessor_account_id` and verify it against `silo::is_allow_submit`.
- In `deploy_code`: verify the caller against `silo::is_allow_deploy`.

This mirrors the fix pattern from the reference report: ensure every execution entry point applies the same access control checks, not just the primary one.

---

### Proof of Concept

1. Operator deploys Aurora Engine in Silo mode, enables `WhitelistKind::Account` and `WhitelistKind::Address` whitelists, and does **not** add attacker's NEAR account or EVM address.
2. Attacker calls `submit` with a signed Ethereum transaction → rejected with `EngineErrorKind::NotAllowed` (whitelist enforced via `assert_access`).
3. Attacker calls `call` with `CallArgs` targeting any EVM contract → **succeeds**, whitelist is never consulted.
4. Attacker calls `deploy_code` with arbitrary bytecode → **succeeds**, whitelist is never consulted.
