Audit Report

## Title
Unbounded Neuron Iteration in `compute_ballots_for_new_proposal` Without Instruction-Limit Guard or Neuron Cap Can DoS SNS Proposal Submission - (File: `rs/sns/governance/src/governance.rs`)

## Summary

`compute_ballots_for_new_proposal` iterates over every entry in `self.proto.neurons` in a single synchronous loop with no instruction-limit guard and no hard cap on the number of neurons. It is called unconditionally from `make_proposal` before any state is mutated. If the neuron population grows large enough, every `make_proposal` call will exceed the IC per-message instruction budget, permanently preventing new proposals until the canister is upgraded via NNS intervention.

## Finding Description

`make_proposal` calls `compute_ballots_for_new_proposal` at the very start of proposal processing: [1](#0-0) 

The function at line 5226 iterates over the entire `self.proto.neurons` map, computing `dissolve_delay_seconds` and `voting_power` per neuron, with no instruction-limit check, no chunking, and no ability to pause and resume: [2](#0-1) 

No calls to `is_message_over_threshold`, `noop_self_call_if_over_instructions`, or any equivalent guard exist anywhere in the SNS governance codebase (confirmed by exhaustive search). There is also no `MAX_NUMBER_OF_NEURONS`-style hard cap on SNS neurons — unlike NNS governance, which enforces such a cap and whose bench test projects up to 25 billion instructions for ballot computation at that cap: [3](#0-2) 

The NNS cap is what keeps NNS ballot computation within the ~40 billion instruction DTS budget. SNS has no equivalent cap, so the instruction cost is unbounded. Because the failure occurs before any state mutation, the canister state is never corrupted, but every subsequent `make_proposal` call fails identically. Recovery requires a canister upgrade, which itself requires a proposal — creating a deadlock resolvable only by NNS-level intervention.

## Impact Explanation

This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS... or subnet availability impact not based on raw volumetric DDoS"* and *"Significant SNS... security impact with concrete user or protocol harm."* The entire SNS governance system is frozen: no upgrades, no parameter changes, no treasury actions can be proposed. The SNS is effectively bricked until the NNS intervenes to upgrade the governance canister.

## Likelihood Explanation

Creating SNS neurons is a permissionless action — any principal can stake SNS tokens and call the neuron creation endpoint. The SNS token supply is finite, which bounds the maximum neuron count, but there is no protocol-enforced cap. A targeted attacker who acquires sufficient SNS tokens can create enough neurons to push ballot computation over the instruction limit. Organic growth of a popular SNS can reach the same threshold without any deliberate attack. The NNS bench data confirms the computation is instruction-intensive at scale; SNS performs identical per-neuron work with no chunking.

## Recommendation

1. Enforce a hard `MAX_NUMBER_OF_NEURONS` constant in SNS governance (analogous to NNS governance) and reject `claim_or_refresh_neuron` once the cap is reached. This is the simplest fix and mirrors the NNS design.
2. Alternatively, pre-compute and cache the electoral roll on a timer so `make_proposal` reads from a snapshot rather than iterating all neurons inline.
3. As a defense-in-depth measure, add an `is_message_over_threshold` guard inside the loop and return an error if the limit is approached, preventing silent instruction exhaustion.

## Proof of Concept

1. Deploy an SNS with default parameters and a low minimum stake.
2. Have many principals each stake SNS tokens with a dissolve delay above `neuron_minimum_dissolve_delay_to_vote_seconds`, creating a large number of eligible neurons.
3. Call `make_proposal` from any neuron with sufficient proposing stake.
4. `make_proposal` calls `compute_ballots_for_new_proposal` (line 3557), which enters the unbounded loop (line 5255) over all neurons.
5. Once the neuron count is sufficient to push instruction consumption past the IC DTS budget (~40 billion instructions), the call is rejected. All subsequent `make_proposal` calls fail identically.
6. A deterministic integration test or canbench test can reproduce this by populating `self.proto.neurons` with a large number of eligible neurons and measuring instruction consumption via `check_projected_instructions`, following the pattern already established in `rs/nns/governance/src/governance/benches.rs`.
