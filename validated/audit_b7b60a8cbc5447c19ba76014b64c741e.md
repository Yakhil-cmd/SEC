### Title
ERC-20 Token Permanent Loss When `ft_transfer` Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns ERC-20 tokens before the cross-contract `ft_transfer` to the NEP-141 contract succeeds. If the NEAR-side transfer fails (e.g., recipient not registered in the NEP-141 contract), the refund path that re-mints ERC-20 tokens is gated behind the `error_refund` Cargo feature. That feature is **not** in the `default` feature set. When it is absent, no callback is scheduled and no refund occurs: the ERC-20 tokens are permanently destroyed while the NEP-141 tokens remain locked in Aurora's account.

---

### Finding Description

**Step 1 — Tokens are burned before the transfer.**

`EvmErc20.withdrawToNear` calls `_burn` on the caller's balance, then invokes the `ExitToNear` precompile. The burn is unconditional and happens inside the same EVM transaction, before any NEAR promise is resolved.

**Step 2 — The refund field is conditionally compiled out.**

Inside `ExitToNear::run`, the `ExitToNearPrecompileCallbackArgs` struct is built with:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None when feature is absent
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

**Step 3 — No callback is scheduled for a plain ERC-20 exit.**

For a regular ERC-20 exit (not wNEAR unwrap), `transfer_near_args` is also `None`. When both fields are `None`, `callback_args` equals `default()`, so the code takes the simpler branch and schedules only the bare `ft_transfer` promise — **no callback at all**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

**Step 4 — `error_refund` is not a default feature.**

```toml
[features]
default = ["std"]
...
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [3](#0-2) 

Any production build that does not explicitly opt into `error_refund` is affected.

**Step 5 — The callback, when it does exist, only refunds if `args.refund` is `Some`.**

```rust
} else if let Some(args) = args.refund {
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent no-op; no refund
}
``` [4](#0-3) 

Without the feature, even if a callback were scheduled, `args.refund` would be `None` and the else branch silently returns `None`.

---

### Impact Explanation

When `ft_transfer` fails (e.g., recipient NEAR account not registered in the NEP-141 contract):

- The user's ERC-20 tokens are **permanently burned** — they no longer exist on Aurora.
- The corresponding NEP-141 tokens remain in Aurora's contract account with no on-chain mechanism to recover them.
- This is a **permanent freeze of user funds** and an effective **permanent loss** from the user's perspective.

The test suite explicitly documents this outcome:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

---

### Likelihood Explanation

NEP-141 tokens on NEAR require explicit storage registration before a `ft_transfer` to an account can succeed. A user who calls `withdrawToNear` targeting a NEAR account that has not called `storage_deposit` on the NEP-141 contract will trigger this failure path. This is a realistic, common mistake — especially for new recipients or accounts that have never interacted with the specific NEP-141 token. No privileged access or attacker coordination is required; the user's own transaction is sufficient to trigger the loss.

---

### Recommendation

1. **Enable `error_refund` by default** or fold it unconditionally into the production build so the refund path is always compiled in.
2. Alternatively, restructure the exit flow to follow a true pull pattern: record the user's entitlement in storage when the ERC-20 is burned, and only release the NEP-141 tokens (or re-mint the ERC-20) after the NEAR-side promise result is confirmed — analogous to the pull-pattern fix described in the reference report.
3. At minimum, document clearly that deployments without `error_refund` expose users to permanent fund loss on failed exits.

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Bridge a NEP-141 token to an ERC-20 on Aurora; fund address `victim`.
3. `victim` calls `EvmErc20.withdrawToNear(unregistered_near_account, amount)`.
   - `_burn(victim, amount)` executes — ERC-20 balance is destroyed.
   - `ExitToNear` precompile schedules `ft_transfer` to `unregistered_near_account` with no callback.
4. The NEAR runtime executes `ft_transfer`; it panics because `unregistered_near_account` has no storage deposit.
5. There is no callback to handle the failure; no re-mint occurs.
6. Result: `victim`'s ERC-20 balance is 0, NEP-141 balance of `unregistered_near_account` is 0, and the NEP-141 tokens remain in Aurora's account — permanently inaccessible to `victim`. [6](#0-5) [7](#0-6) [3](#0-2)

```rust
#[named]
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
