Audit Report

## Title
Silent Permanent Fund Loss When `reimbursement_account` Is Absent in Canceled ckBTC Withdrawal Requests — (File: rs/bitcoin/ckbtc/minter/src/lib.rs)

## Summary
The `reimburse_canceled_requests()` function in the ckBTC minter silently skips reimbursement and permanently loses user funds when a `RetrieveBtcRequest` carries `reimbursement_account: None`. Legacy requests deserialized from the event log have this field as `None` due to the `#[serde(skip_serializing_if = "Option::is_none")]` annotation. If any such request is canceled via the `TooManyInputs` path, the ckBTC burn is already committed on the ledger but no reimbursement mint is ever scheduled, resulting in permanent, irrecoverable loss of user funds.

## Finding Description
`RetrieveBtcRequest` declares `reimbursement_account` as `Option<Account>` with `skip_serializing_if = "Option::is_none"`, meaning legacy event log entries omit the field entirely and deserialize with `None`. [1](#0-0) 

During event log replay, `AcceptedRetrieveBtcRequest` events are pushed back into `pending_retrieve_btc_requests` without any backfill of the missing field. [2](#0-1) 

In `submit_pending_requests`, when `build_unsigned_transaction` returns `BuildTxError::InvalidTransaction(TooManyInputs{…})`, `reimburse_canceled_requests` is called directly with the batch. [3](#0-2) 

Inside `reimburse_canceled_requests`, the `None` branch does nothing except emit a log line — no `ScheduleWithdrawalReimbursement` event is recorded, no `mint_ckbtc` call is made, and the request is silently dropped. [4](#0-3) 

The ckBTC burn is committed to the ledger *before* `accept_retrieve_btc_request` is called, so the burn is final and unrecoverable if no reimbursement is scheduled. [5](#0-4) 

The same silent-drop path is also reachable via `resubmit_transactions` when a stuck transaction is replaced with a cancellation transaction and subsequently confirmed. [6](#0-5) 

The mainnet state, as confirmed by the replay test, has pending requests at block indices 3459007, 3459009, 3459013, 3489347, and 3489353 — all of which are legacy stuck-transaction requests that transited back to pending and may carry `reimbursement_account: None`. [7](#0-6) 

## Impact Explanation
Any legacy `RetrieveBtcRequest` in the pending queue with `reimbursement_account: None` that is subsequently canceled results in permanent, irrecoverable loss of the user's ckBTC. The ledger records a burn with no corresponding reimbursement mint. This is a concrete financial integrity failure in the ckBTC system — a Chain Fusion / ck-token asset with real BTC backing — qualifying as **High ($2,000–$10,000): Significant Chain Fusion / ck-token security impact with concrete user or protocol harm**.

## Likelihood Explanation
The ckBTC minter has been running on mainnet since before `reimbursement_account` was introduced. The `skip_serializing_if = "Option::is_none"` annotation confirms that serialized legacy events omit the field, so any request accepted before the field was added deserializes with `None`. The mainnet state currently has 7,519 available UTXOs, making the `TooManyInputs` threshold of 1,000 inputs reachable for large-value withdrawals requiring many small UTXOs. The `TooManyInputs` cancellation path was itself introduced to handle a real production stuck-transaction scenario (upgrade proposal `minter_upgrade_2025_08_13.md`). The combination of legacy requests in state and a `TooManyInputs` cancellation trigger is a realistic, non-theoretical scenario requiring no special privileges — it is triggered automatically by the minter's own timer-driven `process_logic`. [8](#0-7) 

## Recommendation
1. **Harden the cancellation path**: In `reimburse_canceled_requests`, replace the silent `else` branch with a hard `panic!` or, at minimum, emit a dedicated on-chain `ScheduleWithdrawalReimbursement`-equivalent event so the loss is auditable and can be remediated via governance upgrade.
2. **Backfill legacy state on upgrade**: On the next minter upgrade, scan `pending_retrieve_btc_requests` for entries with `reimbursement_account: None` and populate the field from the ledger burn record's `from` account before any cancellation logic can run.
3. **Enforce presence at acceptance time**: Change `reimbursement_account` from `Option<Account>` to `Account` for all new requests; both current call sites (`retrieve_btc` and `retrieve_btc_with_approval`) already always supply a value.

## Proof of Concept
1. Identify a `RetrieveBtcRequest` in `pending_retrieve_btc_requests` with `reimbursement_account: None` (present in the mainnet event log for requests accepted before the field was introduced).
2. Ensure the minter has >1,000 available UTXOs (mainnet currently has 7,519).
3. The minter's timer fires `process_logic` → `submit_pending_requests`. The pending request is batched. `build_unsigned_transaction` selects >1,000 UTXOs to cover the withdrawal amount, returning `BuildTxError::InvalidTransaction(TooManyInputs{…})`.
4. `reimburse_canceled_requests` is called with the batch. For the request with `reimbursement_account: None`, the `else` branch at `lib.rs:321–327` logs and returns.
5. No `ScheduleWithdrawalReimbursement` event is recorded. No `mint_ckbtc` call is made. The user's ckBTC burn (step 1) is final and the funds are permanently lost.
6. A deterministic integration test can reproduce this by: (a) replaying mainnet events to obtain a state with legacy pending requests, (b) injecting >1,000 small UTXOs, (c) calling `submit_pending_requests` with a mock runtime, and (d) asserting that `pending_withdrawal_reimbursements` is empty while the burn block index has no corresponding reimbursement event. [9](#0-8)

fn reimburse_canceled_requests<R: CanisterRuntime>(
    state: &mut state::CkBtcMinterState,
    requests: BTreeSet<state::RetrieveBtcRequest>,
    reason: WithdrawalReimbursementReason,
    total_fee: u64,
    runtime: &R,
) {
    assert!(!requests.is_empty());
    let fees = distribute(total_fee, requests.len() as u64);
    // This assertion makes sure the fee is smaller than each request amount
    assert!(
        fees[0] <= state.retrieve_btc_min_amount,
        "BUG: fees {fees:?} for {} withdrawal requests are larger than `retrieve_btc_min_amount` {}",
        requests.len(),
        state.retrieve_btc_min_amount
    );
    for (request, fee) in requests.into_iter().zip(fees.into_iter()) {
        if let Some(account) = request.reimbursement_account {
            let amount = request.amount.saturating_sub(fee);
            if amount > 0 {
                state::audit::reimburse_withdrawal(
                    state,
                    request.block_index,
                    amount,
                    account,
                    reason.clone(),
                    runtime,
                );
            }
        } else {
            log!(
                Priority::Info,
                "[reimburse_canceled_requests]: account is not found for retrieve_btc request ({:?})",
                request
            );
        }
    }
}
