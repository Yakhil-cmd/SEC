### Title
SNS Treasury Manager Deposit Interface Lacks Slippage Protection, Enabling Price-Manipulation Attacks on DAO Treasury Deposits - (File: rs/sns/treasury_manager/treasury_manager.did, rs/sns/governance/src/extensions.rs)

### Summary

The SNS Treasury Manager deposit interface (`DepositRequest`) only accepts maximum token allowances with no minimum liquidity amount or slippage tolerance parameter. The governance execution path in `rs/sns/governance/src/extensions.rs` additionally enforces that the deposit proposal map contains **exactly 2 fields** (`treasury_allocation_sns_e8s` and `treasury_allocation_icp_e8s`), making it structurally impossible to pass slippage bounds through the protocol. An attacker who can interact with the underlying DEX can manipulate the pool price in the window between proposal approval and execution, causing the SNS treasury to receive far less liquidity than intended.

### Finding Description

The `DepositRequest` type defined in the Treasury Manager interface contains only token allowances:

```
type DepositRequest = record {
  allowances : vec Allowance;
};
```

`Allowance` carries only `asset`, `amount_decimals` (a maximum), and `owner_account` — no minimum liquidity amount. [1](#0-0) 

The governance-side `ValidatedDepositOperationArg` struct mirrors this: it holds only `treasury_allocation_sns_e8s`, `treasury_allocation_icp_e8s`, and the raw `original` Precise value. [2](#0-1) 

When the proposal is executed, `construct_deposit_allowances` is called with the `original` Precise map and enforces that it contains **exactly 2 entries**:

```rust
if map.len() != 2 {
    return Err(format!(
        "{PREFIX}Top-level type must be PreciseMap with exactly 2 entries."
    ));
}
``` [3](#0-2) 

This check fires at execution time inside `execute_treasury_manager_deposit`, which calls `construct_treasury_manager_deposit_payload` → `construct_treasury_manager_deposit_allowances` → `construct_deposit_allowances`. [4](#0-3) 

Because `ValidatedDepositOperationArg::try_from` does **not** enforce the 2-field limit during proposal validation, a proposer could submit a proposal with a third field (e.g., `min_liquidity_amount`), it would pass the voting phase, but the execution would revert with "Top-level type must be PreciseMap with exactly 2 entries." [5](#0-4) 

The result is that no slippage bound can ever reach the treasury manager canister through the governance path. The `DepositRequest` forwarded to the treasury manager canister is constructed solely from the two token allowances: [6](#0-5) 

The codebase itself acknowledges this gap in two places:

1. `treasury_manager.did` under "Known Security Risks": *"Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved."* [7](#0-6) 

2. `validate_and_render_register_extension` emits a WARNING in the proposal rendering: *"Some Decentralized Exchanges lack slippage protection during deposits … making them vulnerable to front-running or sandwich attacks."* [8](#0-7) 

The acknowledgment confirms the risk is known but unfixed at the protocol level.

### Impact Explanation

An attacker who can trade on the DEX that a Treasury Manager extension targets can:

1. Monitor the SNS governance canister for a pending `ExecuteExtensionOperation` deposit proposal (all proposals are public).
2. Before the proposal executes, buy a large amount of one token in the pool, skewing the price ratio.
3. The governance canister calls `approve_treasury_manager` (ICRC-2 approve for both SNS and ICP tokens) and then calls `deposit` on the treasury manager with only the token allowances and no minimum liquidity bound. [9](#0-8) 
4. The treasury manager deposits at the manipulated price, receiving significantly less LP liquidity than the SNS DAO voted to obtain.
5. The attacker sells back the tokens, profiting from the price impact while the SNS treasury permanently holds an under-valued LP position.

The financial loss is borne by the SNS treasury (i.e., all token holders of that SNS). The attacker's profit is bounded only by the size of the deposit and the depth of the pool.

### Likelihood Explanation

- SNS governance proposals have a public, predictable execution timeline (voting period + wait-for-quiet extension). The attacker has hours to days of advance notice.
- On the Internet Computer, there is no traditional mempool, but the attacker does not need sub-block front-running: manipulating the DEX price any time before the proposal round executes is sufficient.
- The attack requires capital to move the DEX price, but for a large SNS treasury deposit this is economically rational.
- The `approve_treasury_manager` call grants a 1-hour ICRC-2 allowance, creating a concrete execution window. [10](#0-9) 

Likelihood: **Medium**.

### Recommendation

1. **Add a `min_liquidity_amount` field to `DepositRequest`** in `rs/sns/treasury_manager/treasury_manager.did` and `rs/sns/treasury_manager/src/lib.rs`, analogous to Orca's `increase_liquidity` interface.
2. **Add `min_liquidity_amount_e8s` to the governance deposit proposal map** and parse it in `ValidatedDepositOperationArg::try_from`. Remove or relax the hard `map.len() != 2` check in `construct_deposit_allowances` to allow this additional field.
3. **Forward the minimum liquidity bound** through `construct_treasury_manager_deposit_payload` into the `DepositRequest` so treasury manager implementations can enforce it when calling the DEX.
4. Treasury manager implementations should reject the deposit and return tokens to the SNS treasury if the DEX would provide less than `min_liquidity_amount` liquidity.

### Proof of Concept

**Step 1 – Attacker observes a pending deposit proposal** targeting a DEX pool (SNS/ICP). The proposal specifies `treasury_allocation_sns_e8s = 1_000_000_000` and `treasury_allocation_icp_e8s = 500_000_000`.

**Step 2 – Before the proposal executes**, the attacker buys a large amount of ICP from the pool, driving the ICP price up (SNS/ICP ratio skewed).

**Step 3 – Proposal executes**. `execute_treasury_manager_deposit` calls `approve_treasury_manager` then `deposit`. The `DepositRequest` forwarded to the treasury manager contains only the two allowances — no minimum liquidity bound. [9](#0-8) 

**Step 4** – The treasury manager deposits at the manipulated price. Because the pool is skewed, the SNS treasury receives, say, 40% of the expected LP tokens.

**Step 5** – The attacker sells ICP back into the pool, restoring the price and pocketing the arbitrage profit. The SNS treasury is left with a permanently under-valued LP position.

**Why no slippage check can be added today**: any attempt to include a third field in the proposal map (e.g., `min_liquidity_amount_e8s`) passes proposal validation but causes `construct_deposit_allowances` to revert at execution time with "Top-level type must be PreciseMap with exactly 2 entries." [3](#0-2)

async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())

impl TryFrom<Option<Precise>> for ValidatedDepositOperationArg {
    type Error = String;

    fn try_from(value: Option<Precise>) -> Result<Self, Self::Error> {
        let Some(original) = value else {
            return Err("Deposit operation arguments must be provided".to_string());
        };

        let map = match &original.value {
            Some(precise::Value::Map(PreciseMap { map })) => map,
            _ => return Err("Deposit operation arguments must be a PreciseMap".to_string()),
        };

        let treasury_allocation_sns_e8s = map
            .get("treasury_allocation_sns_e8s")
            .and_then(|p| match &p.value {
                Some(precise::Value::Nat(n)) => Some(*n),
                _ => None,
            })
            .ok_or_else(|| "treasury_allocation_sns_e8s must be a Nat value".to_string())?;

        let treasury_allocation_icp_e8s = map
            .get("treasury_allocation_icp_e8s")
            .and_then(|p| match &p.value {
                Some(precise::Value::Nat(n)) => Some(*n),
                _ => None,
            })
            .ok_or_else(|| "treasury_allocation_icp_e8s must be a Nat value".to_string())?;

        Ok(Self {
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
            original,
        })
    }
