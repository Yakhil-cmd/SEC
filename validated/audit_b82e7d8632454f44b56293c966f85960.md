Audit Report

## Title
Unbounded Ingress-Range Loop in ICP Rosetta `construction_payloads` Enables OOM DoS — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary

The ICP Rosetta `construction_payloads` handler accepts attacker-controlled `ingress_start` and `ingress_end` values from request metadata and feeds them directly into an unbounded `while now < ingress_end` loop with no range validation. Supplying `ingress_start=0` and `ingress_end=u64::MAX` causes the loop to iterate approximately 153 billion times, pushing 8-byte entries into a `Vec` until the process exhausts all available RAM and is terminated by the OOM killer.

## Finding Description

**Attacker-controlled input path:**

`ConstructionPayloadsRequestMetadata` deserializes `ingress_start` and `ingress_end` as plain `Option<u64>` with no validation. [1](#0-0) 

These values are passed directly into `ic_types::time::Time::from_nanos_since_unix_epoch` with no bounds check: [2](#0-1) 

**The unbounded loop:**

`interval` is computed as `MAX_INGRESS_TTL - PERMITTED_DRIFT - 120s = 300s - 60s - 120s = 120s = 120,000,000,000 ns`: [3](#0-2) [4](#0-3) 

The loop then runs with no upper-bound guard: [5](#0-4) 

With `ingress_start=0` and `ingress_end=u64::MAX (18,446,744,073,709,551,615)`, the loop runs approximately `u64::MAX / 120_000_000_000 ≈ 153,722,867,281` iterations. Each iteration pushes a `u64` (8 bytes) into `ingress_expiries`. At 8 bytes/entry, the process exhausts all available RAM (e.g., 64 GB at ~8 billion entries) long before the loop terminates. The final iteration would cause a `Duration` addition overflow panic, but OOM occurs billions of iterations earlier.

**Existing checks are absent:** There is no guard before the loop in the ICP handler — no check for `ingress_start >= ingress_end`, no check for maximum range size, and no cap on the number of expiry entries.

**Note on ICRC1 comparison:** The ICRC1 handler does have a `ingress_start >= ingress_end` guard and a staleness check: [6](#0-5) 

However, neither of those guards bounds the range size from above — the ICRC1 handler also lacks an explicit maximum-range guard. The ICP handler is more exposed because it lacks even the basic `ingress_start >= ingress_end` check, but the correct fix for both handlers is an explicit cap on `ingress_end - ingress_start`.

## Impact Explanation

The ICP Rosetta node is a financial integration component used by exchanges and custodians for ICP ledger transaction submission. The `/construction/payloads` endpoint is unauthenticated. A single malicious HTTP POST with `metadata.ingress_start=0` and `metadata.ingress_end=18446744073709551615` causes the Rosetta process to allocate memory until the OOM killer terminates it, halting all ICP ledger transaction submissions and balance queries for any operator running the node. This matches the **High** impact category: *"Application/platform-level DoS, crash"* and *"Significant Rosetta … security impact with concrete user or protocol harm."*

## Likelihood Explanation

The endpoint is unauthenticated and publicly reachable. The exploit requires a single HTTP POST with two integer fields set to boundary values (`0` and `u64::MAX`). No special knowledge, credentials, or infrastructure is needed. The attack is repeatable — each restart of the Rosetta node can be immediately crashed again with the same request.

## Recommendation

Add range validation before the `while` loop in `construction_payloads`:

1. Reject if `ingress_start >= ingress_end`.
2. Reject if `ingress_end - ingress_start` exceeds a reasonable maximum (e.g., 24 hours = `86_400_000_000_000 ns`), or equivalently cap the number of expiry entries (e.g., `<= 720` entries for a 24-hour window at 2-minute intervals).

Apply the same upper-bound cap to the ICRC1 handler's loop as well, since its existing guards do not bound the range size from above.

// Unit test — no network required
#[test]
fn test_construction_payloads_oom_dos() {
    use ic_rosetta_api::models::ConstructionPayloadsRequestMetadata;
    // ingress_start = 0, ingress_end = u64::MAX
    let meta = ConstructionPayloadsRequestMetadata {
        ingress_start: Some(0),
        ingress_end: Some(u64::MAX),
        memo: None,
        created_at_time: None,
    };
    // Without the guard, this call will exhaust RAM and crash the process.
    // With the fix, it must return Err before allocating unbounded memory.
    let result = handler.construction_payloads(build_request(meta));
    assert!(result.is_err(), "Must reject unbounded ingress range");
}
