### Title
SNS Governance Proposal URL Lacks Scheme and Domain Validation, Enabling Phishing Link Injection - (File: rs/sns/governance/src/proposal.rs)

### Summary

The SNS Governance canister's `validate_and_render_proposal` function validates the proposal `url` field using only a character-count check (`validate_chars_count`), with no scheme enforcement, no domain allowlist, and no content sanitization. Any SNS neuron holder can submit a proposal containing an arbitrary URL (e.g., `http://phishing.com`, `javascript:alert(1)`, `data:text/html,...`) that will be stored on-chain and rendered in official SNS/NNS frontends, enabling phishing and potential XSS attacks against token holders who vote on proposals.

### Finding Description

In `rs/sns/governance/src/proposal.rs`, `validate_and_render_proposal` validates the three proposal string fields as follows:

```rust
defects_push(validate_len("title",   &proposal.title,   NO_MIN, PROPOSAL_TITLE_BYTES_MAX));
defects_push(validate_len("summary", &proposal.summary, NO_MIN, PROPOSAL_SUMMARY_BYTES_MAX));
defects_push(validate_chars_count("url", &proposal.url, NO_MIN, PROPOSAL_URL_CHAR_MAX));
```

`validate_chars_count` (defined in `rs/sns/governance/src/lib.rs`) performs only a Unicode character count check — it does not inspect the URL scheme, domain, or content in any way. [1](#0-0) [2](#0-1) 

This is in direct contrast to NNS Governance, which calls `ic_nervous_system_common_validation::validate_url` with an explicit `allowed_domains: Some(vec!["forum.dfinity.org"])` allowlist, rejecting any URL not on that domain: [3](#0-2) 

The shared `validate_url` function in `rs/nervous_system/common/validation/src/lib.rs` enforces `https://` scheme, rejects `@` credentials, and optionally enforces an allowlist — none of which are applied to SNS proposal URLs: [4](#0-3) 

The `title` and `summary` fields also receive only byte-length checks with no character-class or content sanitization, making them vectors for injecting HTML/script payloads if a frontend renders them without encoding. [5](#0-4) 

The SNS `Proposal` struct stores `url`, `title`, and `summary` as plain `String` fields with no further transformation after validation: [6](#0-5) 

### Impact Explanation

An unprivileged SNS neuron holder (any principal with a staked neuron meeting the proposal submission threshold) can submit a governance proposal whose `url` field contains:

- An `http://` or arbitrary-scheme URL pointing to a phishing site
- A `javascript:` URI that executes in browsers that render it as a link
- A `data:text/html,...` URI embedding inline HTML/script

The proposal is stored immutably on-chain and displayed in official SNS/NNS dapp frontends (e.g., nns.ic0.app). Token holders reviewing or voting on the proposal will see the attacker-controlled URL presented as the official proposal reference link. Clicking it can redirect them to a phishing site designed to steal ICP, SNS tokens, or seed phrases. The `title` and `summary` fields (up to 256 bytes and 30,000 bytes respectively) can carry HTML/script payloads that trigger XSS if any frontend renders them without HTML-encoding.

**Impact: 3/5** — Phishing and potential XSS against token holders via official governance frontends; no direct on-chain fund loss from the smart contract itself, but high social-engineering potential given the trusted context.

### Likelihood Explanation

Any principal holding an SNS neuron with sufficient stake to submit a proposal can exploit this. The SNS framework is permissionless — any project can deploy an SNS, and any neuron holder in that SNS can submit proposals. The attacker needs no special privilege, no key compromise, and no majority. The barrier is only the proposal rejection cost (a small token amount).

**Likelihood: 3/5** — Low technical barrier; requires only a valid SNS neuron. The attack is economically cheap relative to the potential phishing yield.

### Recommendation

Replace the bare `validate_chars_count` call for the SNS proposal `url` field in `validate_and_render_proposal` with a call to `ic_nervous_system_common_validation::validate_url`, mirroring the NNS governance pattern. For SNS proposals, the `allowed_domains` parameter should be set to `None` (to permit any HTTPS domain, since SNS projects are diverse) but the `https://` scheme requirement and `@`-credential rejection must be enforced. Additionally, consider adding a character-class allowlist or explicit rejection of control characters and HTML metacharacters (`<`, `>`, `"`, `'`, `&`) in the `title` and `summary` fields to eliminate XSS vectors independent of frontend encoding.

### Proof of Concept

1. Obtain an SNS neuron with sufficient stake to submit a proposal (the `proposal_reject_cost_e8s` threshold).
2. Call the SNS Governance canister's `manage_neuron` endpoint with a `MakeProposal` command containing:
   ```
   Proposal {
     title: "<img src=x onerror=alert(document.cookie)>",
     summary: "Legitimate-looking proposal text...",
     url: "http://steal-your-icp.phishing.example.com/fake-nns",
     action: Motion { motion_text: "..." }
   }
   ```
3. The canister accepts the proposal: `validate_chars_count` passes (title ≤ 256 bytes, url ≤ 2048 chars), and no scheme or content check fires. [1](#0-0) 
4. The proposal is stored on-chain and rendered in the SNS dapp frontend. The `url` appears as the official proposal reference link. Token holders who click it are redirected to the phishing site. If the frontend renders `title` as raw HTML, the `<img onerror>` payload executes.
5. Contrast: submitting the same `http://` URL to NNS Governance is rejected at `validate_proposal_url` because `validate_url` enforces `https://` and the `forum.dfinity.org` allowlist. [3](#0-2)

fn validate_chars_count(
    field_name: &str,
    field_value: &str,
    min: usize,
    max: usize,
) -> Result<(), String> {
    let len = field_value.chars().count();

    if len < min {
        let defect = &format!("too short (min = {min} vs. observed = {len})");

        return field_err(field_name, field_value.to_string(), defect);
    }

    if len > max {
        let defect = &format!("too long (max = {max} vs. observed = {len})");
        let bounded_field_value = field_value.chars().take(max).collect::<String>();

        return field_err(field_name, bounded_field_value, defect);
    }

    Ok(())
}

/// Verifies that the url is within the allowed length, and begins with `https://`. In addition, it
/// will return an error in case of a possibly "dangerous" condition, such as the url containing a
/// username or password, or having a port, or not having a domain name.
pub fn validate_url(
    url: &str,
    min_length: usize,
    max_length: usize,
    field_name: &str,
    allowed_domains: Option<Vec<&str>>,
) -> Result<(), String> {
    // // Check that the URL is a sensible length
    if url.len() > max_length {
        return Err(format!(
            "{field_name} must be less than {max_length} characters long, but it is {} characters long. (Field was set to `{url}`.)",
            url.len(),
        ));
    }
    if url.len() < min_length {
        return Err(format!(
            "{field_name} must be greater or equal to than {min_length} characters long, but it is {} characters long. (Field was set to `{url}`.)",
            url.len(),
        ));
    }

    //

    if !url.starts_with("https://") {
        return Err(format!(
            "{field_name} must begin with https://. (Field was set to `{url}`.)",
        ));
    }

    let parts_url: Vec<&str> = url.split("://").collect();
    if parts_url.len() > 2 {
        return Err(format!(
            "{field_name} contains an invalid sequence of characters"
        ));
    }

    if parts_url.len() < 2 {
        return Err(format!("{field_name} is missing content after protocol."));
    }

    if url.contains('@') {
        return Err(format!(
            "{field_name} cannot contain authentication information"
        ));
    }

    let parts_past_protocol = parts_url[1].split_once('/');

    let (domain, _path) = match parts_past_protocol {
        Some((domain, path)) => (domain, Some(path)),
        None => (parts_url[1], None),
    };

    match allowed_domains {
        Some(allowed) => match allowed.contains(&domain) {
            true => Ok(()),
            false => Err(format!(
                "{field_name} was not in the list of allowed domains: {allowed:?}"
            )),
        },
        None => Ok(()),
    }
}
