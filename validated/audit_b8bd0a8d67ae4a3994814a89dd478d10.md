Audit Report

## Title
Floating-Point Precision Loss and u64 Overflow in ICRC-21 Consent Message Token Amount Display - (File: `packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
`convert_tokens_to_string_representation` converts a `Nat` token amount to `f64` before performing decimal division, silently losing precision for any amount exceeding 2^53 ≈ 9×10^15. For ckETH (18 decimals), this means every transfer above ~0.009 ETH displays a rounded amount in the ICRC-21 consent message. Separately, the `FieldsDisplayMessage` path calls `nat_to_u64`, which returns an `Icrc21Error` for any ckETH amount exceeding `u64::MAX` ≈ 18.4 ETH, causing the consent message endpoint to fail entirely for those transfers.

## Finding Description

**Root cause 1 — `convert_tokens_to_string_representation` (GenericDisplayMessage path):**

`packages/icrc-ledger-types/src/icrc21/responses.rs` lines 318–327:
```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let tokens = tokens.0.to_f64().ok_or(Icrc21Error::GenericError { ... })?;
    Ok(format!("{}", tokens / 10_f64.pow(decimals)))
}
```
`Nat` wraps `num_bigint::BigUint`. The `ToPrimitive::to_f64()` implementation for `BigUint` **always** returns `Some(f64)` — it never returns `None`, even for values exceeding `f64::MAX` (it returns `Some(f64::INFINITY)`). Consequently:
- The `ok_or` guard is dead code and never fires.
- Any `Nat` exceeding 2^53 is silently rounded to the nearest representable double before the decimal shift.
- Astronomically large values produce `"inf"` in the output string.

This function is called from `add_amount`, `add_fee`, `add_allowance`, and `add_existing_allowance` whenever the message type is `GenericDisplayMessage`.

**Root cause 2 — `nat_to_u64` (FieldsDisplayMessage path):**

Lines 329–334:
```rust
fn nat_to_u64(tokens: Nat) -> Result<u64, Icrc21Error> {
    tokens.0.to_u64().ok_or(Icrc21Error::GenericError {
        error_code: Nat::from(500_u64),
        description: "Failed to convert tokens to u64".to_owned(),
    })
}
```
`BigUint::to_u64()` returns `None` for values exceeding `u64::MAX` = 18,446,744,073,709,551,615. For ckETH (18 decimals), `u64::MAX` ≈ 18.4 ETH. Any `FieldsDisplay` consent message request for a transfer above this threshold propagates the error up through `add_amount`/`add_fee`/`add_allowance`, causing `build_icrc21_consent_info` to return `Err`, and the `icrc21_canister_call_consent_message` update call to fail entirely.

**Exploit path:**
1. Caller invokes `icrc21_canister_call_consent_message` on the ICRC-1 ledger (or ICP ledger) with a transfer/approve argument containing a large `amount`.
2. For `GenericDisplay` (default): `convert_tokens_to_string_representation` silently rounds the amount via `f64`, producing an incorrect string in the consent message.
3. For `FieldsDisplay`: `nat_to_u64` returns `Err` for amounts > `u64::MAX`, and the entire endpoint returns an error — the user cannot obtain a consent message.

No special privileges are required; `icrc21_canister_call_consent_message` is a publicly callable update method.

## Impact Explanation

The ICRC-21 consent message is the sole user-facing security display that hardware wallets and browser extensions present before a user signs a transaction. Displaying an incorrect token amount directly undermines the purpose of the standard. For ckETH:
- **Precision loss**: Every transfer above ~0.009 ETH (9×10^15 wei) displays a rounded amount. The rounding error grows with magnitude; at 1 ETH the error is up to 256 wei, at 100 ETH up to ~32,768 wei.
- **DoS**: Any `FieldsDisplay` consent message request for a ckETH transfer > 18.4 ETH fails entirely, preventing the user from obtaining a consent message via wallets that request `FieldsDisplay`.

This constitutes a **High** impact: significant ck-token/ledger security impact with concrete user harm — incorrect amount shown in the security-critical signing display, and denial of consent message generation for large transfers.

## Likelihood Explanation

**High.** The endpoint is publicly callable with no authentication. For ckETH, essentially all real-world transfers (> 0.009 ETH) trigger the precision loss in `GenericDisplay`. The `FieldsDisplay` DoS is triggered by any transfer > 18.4 ETH when a wallet requests `FieldsDisplay`. No attacker-controlled state or special setup is required beyond submitting a standard `icrc1_transfer` argument with a large `amount`.

## Recommendation

Replace the `f64`-based conversion in `convert_tokens_to_string_representation` with exact integer arithmetic on the `BigUint` directly:
```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let divisor = num_bigint::BigUint::from(10u64).pow(decimals as u32);
    let whole = &tokens.0 / &divisor;
    let frac = &tokens.0 % &divisor;
    if frac.is_zero() {
        Ok(format!("{whole}"))
    } else {
        let frac_str = format!("{frac:0>width$}", width = decimals as usize);
        Ok(format!("{whole}.{}", frac_str.trim_end_matches('0')))
    }
}
```
This mirrors the correct pattern already used in `rs/bitcoin/ckbtc/minter/src/updates/icrc21.rs` (`format_amount`). For `nat_to_u64`, replace the `u64` representation in `Value::TokenAmount` with a `String` or `BigUint`-based representation, or at minimum use `u128` to extend the safe range, and handle overflow gracefully rather than returning an error.

## Proof of Concept

For a ckETH ledger (`decimals = 18`), call `icrc21_canister_call_consent_message` with `icrc1_transfer` and `amount = 1_000_000_000_000_000_001` (1 ETH + 1 wei):

**GenericDisplay path:**
1. `tokens.0.to_f64()` → `Some(1_000_000_000_000_000_000.0)` (trailing 1 lost; 10^18+1 > 2^53).
2. `format!("{}", 1e18_f64 / 1e18_f64)` → `"1"`.
3. Consent message displays `"1 ckETH"` instead of `"1.000000000000000001 ckETH"`.

**FieldsDisplay DoS path:**
Call with `amount = 20_000_000_000_000_000_000` (20 ETH > `u64::MAX` ≈ 18.4 ETH) and `device_spec = FieldsDisplay`:
1. `nat_to_u64` → `Err(Icrc21Error::GenericError { ... })`.
2. `icrc21_canister_call_consent_message` returns `Err` — no consent message is produced.
