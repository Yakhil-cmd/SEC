### Title
Multiblock Batch Prover Panic via `ArrayVec` Overflow in `apply_to_array_vec` - (File: `zk_ee/src/common_structs/logs_storage.rs`)

### Summary

In the multiblock batch proving path, `LogsStorage::apply_to_array_vec` accumulates L2→L1 log hashes from each block into a shared `ArrayVec<Bytes32, 16384>` held in `ZKBatchDataKeeper`. The per-block log limit (`MAX_NUMBER_OF_LOGS = 16_384`) is enforced only per-block, not per-batch. When a batch spans multiple blocks each carrying up to 16,384 logs, the total pushed into the fixed-capacity `ArrayVec` exceeds 16,384, causing `ArrayVec::push` to panic and making the batch unprovable.

### Finding Description

`MAX_NUMBER_OF_LOGS` is defined as 16,384 — the capacity of the L2→L1 log Merkle tree: [1](#0-0) 

The ZK transaction loop enforces this limit **per block** via `check_for_block_limits`: [2](#0-1) 

The check uses `>` (not `>=`), so a single block may contain exactly 16,384 logs.

During multiblock batch finalization, `post_tx_op_proving_multiblock_batch.rs` calls `apply_to_array_vec` to accumulate each block's log hashes into `batch_data.logs_storage`: [3](#0-2) 

`batch_data.logs_storage` is declared as `ArrayVec<Bytes32, 16384>` — the same fixed capacity as the per-block limit: [4](#0-3) 

`apply_to_array_vec` uses the panicking `ArrayVec::push` with no capacity check: [5](#0-4) 

`arrayvec::ArrayVec::push` panics when the array is full. If a batch contains two or more blocks each with 16,384 logs, the second block's `apply_to_array_vec` call overflows the accumulator and panics, aborting the prover.

### Impact Explanation

The prover panics during batch finalization, making the entire multiblock batch unprovable. This is a **valid-execution unprovability** / **liveness failure**: each individual block is valid (it respects the per-block log limit), but the batch cannot be proven. The chain cannot advance until the batch is somehow discarded or the operator avoids the condition, constituting a denial-of-service against the proving pipeline.

### Likelihood Explanation

Any unprivileged user can emit L2→L1 logs by calling the L1 Messenger system hook (`sendToL1(bytes)`). An attacker can fill two consecutive blocks with 16,384 logs each (within the per-block gas and pubdata limits), then wait for the operator to include them in a multiblock batch. No special access is required.

### Recommendation

The batch-level `logs_storage` accumulator must be sized to hold the maximum total logs across all blocks in a batch (i.e., `max_blocks_per_batch × 16_384`), or the per-batch log total must be enforced as a separate limit before `apply_to_array_vec` is called. Alternatively, replace the panicking `push` with `try_push` and propagate an error if the batch log capacity is exceeded.

### Proof of Concept

1. Attacker submits transactions calling the L1 Messenger system hook in Block A, filling it to exactly 16,384 L2→L1 logs (within the per-block limit enforced at `check_for_block_limits`).
2. Attacker submits at least one more L2→L1 log in Block B of the same multiblock batch.
3. The operator runs the prover in multiblock batch mode (`ZKHeaderStructurePostTxOpProvingMultiblockBatch`).
4. After Block A is processed, `batch_data.logs_storage` holds 16,384 entries (full capacity).
5. After Block B is processed, `apply_to_array_vec` calls `ArrayVec::push` on a full array → **panic**.
6. The prover aborts; the batch is unprovable.
