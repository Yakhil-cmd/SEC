### Title
Unrestricted `deploy_erc20_token` Allows Any Caller to Permanently Hijack NEP-141 Bridge Registration - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `deploy_erc20_token` public entrypoint imposes no caller restriction. Any NEAR account can invoke it with an arbitrary NEP-141 account ID. Because `register_token` enforces a one-time-only registration (returning `TokenAlreadyRegistered` on any subsequent attempt) and no `unregister_token` path exists anywhere in the codebase, a single attacker transaction permanently occupies the NEP-141 → ERC-20 slot for any token, blocking the legitimate bridge from ever registering it.

---

---

### Impact Explanation

An attacker who calls `deploy_erc20_token("usdc.near")` (or any high-value NEP-141) before the legitimate bridge operator does will:

1. Deploy a spurious ERC-20 contract inside Aurora's EVM state.
2. Permanently write the `usdc.near → <spurious ERC-20 address>` mapping into Aurora's storage.
3. Cause every subsequent legitimate call to `deploy_erc20_token("usdc.near")` to fail with `ERR_NEP141_TOKEN_ALREADY_REGISTERED`.

From that point forward, all real USDC tokens transferred to Aurora via `ft_on_transfer` are minted into the spurious ERC-20 contract. The spurious ERC-20's admin is set to the Aurora contract address (not the attacker), so the attacker cannot directly steal minted balances. However:

- The legitimate ERC-20 address for that token can never be established, breaking all integrations, DEXes, and wallets that rely on the canonical address.
- The bridge registration for that NEP-141 is permanently corrupted with no on-chain recovery path.
- Users who bridge tokens receive balances in an unexpected ERC-20 contract, which may be incompatible with downstream DeFi protocols, effectively freezing their bridged value from a usability and composability standpoint.

**Impact: High — Permanent freezing of bridged funds / permanent DoS of the NEP-141 bridge for any targeted token.** [6](#0-5) 

---

### Likelihood Explanation

The attack requires only a standard NEAR account and a single function call to `deploy_erc20_token` on the Aurora contract. No privileged keys, no special role, no ETH/NEAR balance beyond gas fees. The attacker can front-run any legitimate bridge registration by monitoring the NEAR mempool or simply acting before any NEP-141 is registered. The cost is one NEAR transaction. The effect is permanent. [7](#0-6) 

---

### Recommendation

Add a caller restriction to `deploy_erc20_token` matching the pattern already used by `mirror_erc20_token`:

```rust
let state = state::get_state(&io)?;
require_running(&state)?;
require_owner_only(&state, &env.predecessor_account_id())?;
```

Alternatively, implement an allowlist of accounts permitted to register NEP-141 tokens (as the `TODO` comment in `mirror_erc20_token` already anticipates). Additionally, consider adding an admin-only `unregister_token` escape hatch to recover from erroneous or malicious registrations. [8](#0-7) 

---

### Proof of Concept

1. Attacker observes that `wrap.near` is not yet registered on Aurora.
2. Attacker submits a NEAR transaction calling `aurora.deploy_erc20_token` with Borsh-encoded `DeployErc20TokenArgs::Legacy("wrap.near")`.
3. `deploy_erc20_token` passes `require_running`, deploys a spurious ERC-20, and writes `wrap.near → <spurious_address>` into Aurora's `Nep141Erc20Map` storage.
4. The legitimate bridge operator later calls `deploy_erc20_token("wrap.near")` — it fails with `ERR_NEP141_TOKEN_ALREADY_REGISTERED`.
5. All subsequent `ft_on_transfer` calls from `wrap.near` mint wNEAR into the spurious ERC-20 at `<spurious_address>`, which is unknown to any DEX, wallet, or integration. Bridged wNEAR is permanently inaccessible through the canonical bridge address. [9](#0-8) [10](#0-9)
