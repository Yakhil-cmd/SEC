Audit Report

## Title
Unbounded `StateSyncId.hash` Allocation in `state_sync_chunk_handler` Enables Per-Request Memory Amplification — (`rs/p2p/state_sync_manager/src/routes/chunk.rs`)

## Summary
The `state_sync_chunk_handler` decodes a `StateSyncChunkRequest` protobuf and passes `id.hash` — an unconstrained `bytes` field — directly into `CryptoHash(id.hash)` with no length validation. A Byzantine subnet peer can craft a request with a hash field up to ~128 MiB, causing the serving replica to heap-allocate the full payload before the request is discarded with `NO_CONTENT`. With 1,000 concurrent QUIC streams permitted per peer connection, this enables sustained heap pressure on a single replica.

## Finding Description
In `state_sync_chunk_handler` (chunk.rs L46-48), the raw payload is decoded and the `id` field is immediately converted to `StateSyncArtifactId` via `From::from`:

```rust
let pb::StateSyncChunkRequest { id, chunk_id } =
    pb::StateSyncChunkRequest::decode(payload).map_err(|_| StatusCode::BAD_REQUEST)?;
let artifact_id: StateSyncArtifactId = id.map(From::from).ok_or(StatusCode::BAD_REQUEST)?;
```

The `From<p2p_pb::StateSyncId> for StateSyncArtifactId` impl (state_sync.rs L23-29) wraps `id.hash` directly:

```rust
hash: CryptoHash(id.hash),   // id.hash is an unbounded Vec<u8>
```

No length check exists before this allocation. The proto definition (`state_sync_manager.proto` L9-12) declares `bytes hash = 2;` with no size constraint. The only upstream guard is `read_to_end(MAX_MESSAGE_SIZE_BYTES)` in `request_handler.rs` L173-175, where `MAX_MESSAGE_SIZE_BYTES = 128 * 1024 * 1024` (lib.rs L74) — intentionally large for consensus summary blocks. The router setup in `lib.rs` L68-75 applies no `RequestBodyLimit` or equivalent middleware to the `/state-sync/chunk` route. The `MAX_CHUNK_SIZE` constant (chunk.rs L19) is only used for response decompression (L107), not for limiting incoming request size. After the allocation, `state_sync.chunk()` returns `None` for any unknown hash, and the handler responds `NO_CONTENT` — having allocated and immediately discarded up to 128 MiB per request.

The QUIC transport permits `MAX_CONCURRENT_BIDI_STREAMS = 1_000` per peer (connection_manager.rs L74), with a 200 MB connection receive window (L71) and 4 MB per-stream window (L73). As `read_to_end` reads and issues window updates, the sender can push the full payload through the 4 MB stream window incrementally, growing the heap buffer to the full payload size before the handler runs.

## Impact Explanation
A Byzantine subnet peer can sustain heap pressure on a single replica by cycling through 1,000 concurrent streams, each carrying a near-maximum-size `hash` field. A legitimate `StateSyncChunkRequest` is ~40 bytes; a malicious one can be ~128 MiB — a ~3-million-fold amplification per request. Sustained allocation can exhaust the replica's heap, causing an OOM crash or severe degradation of state sync and other replica functions. This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS**.

## Likelihood Explanation
The attacker must be a TLS-authenticated subnet peer — a realistic Byzantine node scenario below the consensus fault threshold. No governance majority, key compromise, or external infrastructure attack is required. The exploit is mechanically trivial: craft a protobuf with a large `bytes` field and send it repeatedly on many concurrent streams. The attack is repeatable and requires no victim interaction.

## Recommendation
Add an explicit length check on `id.hash` immediately after decode, before `CryptoHash` construction. A legitimate `CryptoHash` is always 32 bytes (SHA-256). Reject any request where `id.hash.len() != 32` with `StatusCode::BAD_REQUEST` in the `From` impl or inline in the handler. Alternatively, apply a `RequestBodyLimitLayer` scoped to the `/state-sync/chunk` route with a tight bound (e.g., 256 bytes), consistent with the pattern used in other routes in the codebase.

## Proof of Concept
```rust
// Attacker-side: craft a StateSyncChunkRequest with a 64 MiB hash
let malicious_request = pb::StateSyncChunkRequest {
    id: Some(pb::StateSyncId {
        height: 1,
        hash: vec![0u8; 64 * 1024 * 1024],  // 64 MiB
    }),
    chunk_id: 0,
};
let mut buf = BytesMut::with_capacity(malicious_request.encoded_len());
malicious_request.encode(&mut buf).unwrap();
// Send buf as the body of POST /state-sync/chunk via QUIC transport
// Expected (buggy) behavior: handler allocates 64 MiB, returns NO_CONTENT
// Expected (fixed) behavior: handler returns BAD_REQUEST before allocation
```

A unit test can verify this by calling `state_sync_chunk_handler` directly with a crafted payload and asserting that the response is `BAD_REQUEST` when `id.hash.len() != 32`, and that no `spawn_blocking` task is spawned. An integration test can confirm memory usage does not spike when a malicious peer sends oversized requests on multiple concurrent streams.
