Audit Report

## Title
Global Base Limiter Exhaustion via Single Large Minting Call - (`rs/nns/cmc/src/limiter.rs`, `rs/nns/cmc/src/main.rs`)

## Summary
The CMC's `base_limiter` is a single shared global rolling-window counter with no per-principal isolation. A single unprivileged caller holding sufficient ICP can consume the entire hourly minting budget (`DEFAULT_CYCLES_LIMIT = 150e15`) in one `notify_top_up` call. All subsequent minting calls from any user that route through `CyclesMintingLimiterSelector::BaseLimit` will be rejected with a rate-limit error until the one-hour window expires.

## Finding Description
`check_and_add_cycles` in `limiter.rs` performs a single global check:

```rust
if count + cycles_to_mint > limit {
    return Err("...please try again later.");
}
self.add(now, cycles_to_mint);
```

The condition uses strict greater-than (`>`), so a call with `cycles_to_mint == limit` when `count == 0` passes (`0 + limit > limit` is `false`) and records the full budget. Every subsequent call then evaluates `limit + any_positive > limit` as `true` and is rejected.

The `base_limiter` is initialized with `max_age = Duration::from_secs(60 * 60)` (one hour) and `base_cycles_limit = Cycles::from(DEFAULT_CYCLES_LIMIT)` (150T cycles). All callers except the Subnet Rental Canister topping up itself are routed to `CyclesMintingLimiterSelector::BaseLimit` in `notify_top_up`, and the same limiter is used by `notify_create_canister`, `notify_mint_cycles`, and `create_canister` via `ensure_balance`. There is no per-principal sub-limiter, no per-call maximum, and no guard preventing a single call from consuming the entire budget.

## Impact Explanation
This is a platform-level DoS of the NNS cycles minting service. All users are blocked from minting cycles for up to one hour per attack cycle. The CMC is a core NNS canister; its unavailability prevents canister creation and top-ups across the entire platform. This matches the allowed impact: **High — Application/platform-level DoS not based on raw volumetric DDoS**.

## Likelihood Explanation
At 100 XDR/ICP and 1T cycles/XDR, minting 150T cycles requires 1,500 ICP. The attacker receives cycles in return, so the net economic cost is near zero (only opportunity cost of the ICP). The attack is executable with a single ledger transfer followed by a single `notify_top_up` call, requires no privileged access, and is repeatable every hour. The ICP amount is non-trivial but realistic for a motivated attacker.

## Recommendation
1. Add a **per-principal sub-limiter** alongside the global limiter, capping how many cycles any single caller can mint per hour (e.g., a fraction of the global limit).
2. Alternatively, enforce a **per-call maximum** on `cycles_to_mint` independent of the global limit, so no single call can consume more than a defined fraction of the hourly budget.
3. Consider a tiered structure where mints above a threshold require a separate governance-approved high-value path.

## Proof of Concept
1. Attacker transfers exactly `base_limit_cost_icp` ICP (1,500 ICP at 100 XDR/ICP) to the CMC subaccount for their canister.
2. Attacker calls `notify_top_up` — `check_and_add_cycles` evaluates `0 + 150T > 150T` → `false`, passes, and records 150T in the global `base_limiter`.
3. Any subsequent call from any user to `notify_top_up`, `notify_create_canister`, `notify_mint_cycles`, or `create_canister` evaluates `150T + N > 150T` → `true` for any positive `N`, returning `NotifyError::Refunded { reason: "...please try again later." }`.
4. All minting is blocked until `purge_old` expires the window (~1 hour).
5. Reproducible via the existing state-machine test harness: adapt `cmc_notify_top_up_rate_limited` to use `Tokens::new(1500, 0)` in the first call and verify a second call from a different principal is immediately rejected.
