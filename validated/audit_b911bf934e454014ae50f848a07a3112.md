### Title
Ingress Induction Fee Burned Before Canister Status Check, Causing Irrecoverable Cycle Loss on Stopping/Stopped Canisters - (File: `rs/messaging/src/scheduling/valid_set_rule.rs`)

### Summary

In `rs/messaging/src/scheduling/valid_set_rule.rs`, the `enqueue` function permanently burns a canister's ingress induction fee cycles **before** checking whether the target canister is in `Running` state. When the canister is `Stopping` or `Stopped`, the function returns an error without refunding the already-burned cycles, causing permanent, irrecoverable cycle loss.

### Finding Description

The `enqueue` function in `ValidSetRuleImpl` follows this ordering:

1. **Charge cycles** (line 305–315): calls `charge_ingress_induction_cost`, which is documented as consuming cycles "i.e. burnt" — permanently reducing the canister's balance with no refund path.
2. **Check canister status** (lines 317–331): only *after* burning cycles does the code check whether the canister is `Running`. If it is `Stopping` or `Stopped`, it returns `IngressInductionError::CanisterStopping` / `IngressInductionError::CanisterStopped` with no cycle refund. [1](#0-0) 

The `charge_ingress_induction_cost` function explicitly documents that cycles are consumed (burned), not reserved: [2](#0-1) 

When the canister does not have a paused execution, `consume_with_threshold` is called, which calls `system_state.consume_cycles()` — a permanent, non-refundable deduction: [3](#0-2) 

The existing tests `ingress_to_stopping_canister_is_rejected` and `ingress_to_stopped_canister_is_rejected` confirm the rejection path exists but do **not** assert that cycles are preserved after rejection, confirming the missing refund: [4](#0-3) 

### Impact Explanation

When a canister transitions to `Stopping` or `Stopped` between block proposal and block execution, ingress messages targeting that canister can be included in a block (because the ingress selector uses the certified state, which may lag behind the execution state). During induction, `enqueue` burns the ingress fee and then rejects the message. The burned cycles are permanently lost — there is no refund path in the error handling of `induct_message`: [5](#0-4) 

**Impact**: Cycles/resource accounting bug. A canister loses its ingress induction fee for every message inducted against it while it is in `Stopping`/`Stopped` state, with no recovery mechanism. The canister pays for a service (message induction) it never receives.

### Likelihood Explanation

The IC protocol's certified state lags behind the actual execution state. The ingress selector (`rs/ingress_manager/src/ingress_selector.rs`) validates messages against the certified state at block proposal time: [6](#0-5) 

If a `stop_canister` call is executed in block N-1, the canister transitions to `Stopping` in the execution state. However, block N may already have been proposed using the certified state from block N-2 (which still shows the canister as `Running`), so messages to the now-`Stopping` canister can be included in block N. When block N is executed, `enqueue` burns the fee and then rejects the message. This race condition is inherent to the IC's pipelined block proposal/execution model and requires no privileged access — any user sending ingress messages to a canister that is concurrently being stopped can trigger it.

### Recommendation

Move the canister status check **before** the `charge_ingress_induction_cost` call in the `enqueue` function. The corrected ordering should be:

1. Check canister status — return error immediately if `Stopping` or `Stopped` (no cycles charged).
2. Only if `Running`, charge the ingress induction fee.
3. Enqueue the message.

This mirrors the correct pattern used in `should_accept_ingress_message` in `rs/execution_environment/src/execution_environment.rs`, which checks status before any resource accounting. [7](#0-6) 

### Proof of Concept

1. Canister `C` is `Running` with sufficient cycles.
2. Block N-1 is executed: a `stop_canister` call transitions `C` to `Stopping`. The certified state still reflects `C` as `Running` (lag).
3. Block N is proposed: the ingress selector uses the certified state, sees `C` as `Running`, and includes an ingress message `M` targeting `C`.
4. Block N is executed: `enqueue` is called for `M`.
   - `charge_ingress_induction_cost` is called → ingress fee cycles are **permanently burned** from `C`'s balance.
   - `canister.status()` returns `Stopping` → `IngressInductionError::CanisterStopping` is returned.
   - No refund is issued.
5. `C` has lost the ingress fee with no corresponding message processed. Repeating across multiple rounds drains `C`'s cycles proportional to the number of in-flight ingress messages during the transition window.

    /// Charges the canister for ingress induction cost.
    ///
    /// Note that this method reports the cycles withdrawn as consumed (i.e.
    /// burnt).
    ///
    /// # Errors
    ///
    /// Returns a `CanisterOutOfCyclesError` if the
    /// requested amount is greater than the currently available.
    pub fn charge_ingress_induction_cost(
        &self,
        canister: &mut CanisterState,
        canister_current_memory_usage: NumBytes,
        canister_current_message_memory_usage: MessageMemoryUsage,
        canister_compute_allocation: ComputeAllocation,
        cycles: Cycles,
        subnet_cycles_config: CyclesAccountManagerSubnetConfig,
        reveal_top_up: bool,
    ) -> Result<(), CanisterOutOfCyclesError> {
        let threshold = self.freeze_threshold_cycles(
            canister.system_state.freeze_threshold,
            canister.system_state.memory_allocation,
            canister_current_memory_usage,
            canister_current_message_memory_usage,
            canister_compute_allocation,
            subnet_cycles_config,
            canister.system_state.reserved_balance(),
        );
        if canister.has_paused_execution_or_install_code() {
            if canister.system_state.debited_balance() < cycles + threshold {
                return Err(CanisterOutOfCyclesError {
                    canister_id: canister.canister_id(),
                    available: canister.system_state.debited_balance(),
                    requested: cycles,
                    threshold,
                    reveal_top_up,
                });
            }
            canister
                .system_state
                .add_postponed_charge_to_ingress_induction_cycles_debit(cycles);
            Ok(())
        } else {
            self.consume_with_threshold::<IngressInduction>(
                &mut canister.system_state,
                CompoundCycles::new(cycles, subnet_cycles_config.cost_schedule),
                threshold,
                reveal_top_up,
            )
        }
    }

    pub fn consume_cycles<T: CyclesUseCaseKind>(&mut self, requested_amount: CompoundCycles<T>) {
        let requested_real = requested_amount.real();
        let use_case = T::cycles_use_case();
        let remaining_amount = match use_case {
            CyclesUseCase::Memory | CyclesUseCase::ComputeAllocation | CyclesUseCase::Uninstall => {
                let covered_by_reserved_balance = requested_real.min(self.reserved_balance);
                self.reserved_balance -= covered_by_reserved_balance;
                requested_real - covered_by_reserved_balance
            }
            CyclesUseCase::IngressInduction
            | CyclesUseCase::Instructions
            | CyclesUseCase::RequestAndResponseTransmission
            | CyclesUseCase::CanisterCreation
            | CyclesUseCase::ECDSAOutcalls

            | CyclesUseCase::VetKd
            | CyclesUseCase::HTTPOutcalls
            | CyclesUseCase::DeletedCanisters
            | CyclesUseCase::BurnedCycles
            | CyclesUseCase::DroppedMessages => requested_real,
        };
