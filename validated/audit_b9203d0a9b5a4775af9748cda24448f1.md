Audit Report

## Title
Cross-Subaccount Allowance Enumeration via Incomplete Break Condition in `get_allowances` — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

## Summary
The `get_allowances` function in the ICRC-1 ledger terminates its BTreeMap range scan only when the iterated entry's `owner` differs from the queried `from.owner`. Because `Account::cmp` sorts by `(owner, effective_subaccount())`, a scan starting at `{owner:X, subaccount:S}` continues through all entries with the same owner but different subaccounts, returning their allowances to the caller. Any unprivileged principal can call `icrc103_get_allowances` with an arbitrary `from_account` and receive allowance records (spender identities, approved amounts, expiration timestamps) belonging to subaccounts of the queried owner that were never requested.

## Finding Description

`Account::cmp` sorts first by `owner`, then by `effective_subaccount()`: [1](#0-0) 

`AccountSpender::cmp` delegates to `Account::cmp` for the account field: [2](#0-1) 

The `ALLOWANCES_MEMORY` `StableBTreeMap` is therefore ordered as:
```
{owner:X, sub:[0;32], spender:S1}
{owner:X, sub:[0;32], spender:S2}
{owner:X, sub:[1;32], spender:S3}   ← different subaccount, same owner
{owner:X, sub:[2;32], spender:S4}   ← different subaccount, same owner
{owner:Y, ...}
```

The range scan starts at `{account: from, spender: empty_principal}`: [3](#0-2) 

The loop break guard checks **only `owner`**, not the full account: [4](#0-3) 

When a caller queries `from_account = {owner:X, subaccount:Some([0u8;32])}`, the scan starts at that entry and advances through `sub:[1;32]`, `sub:[2;32]`, etc. The guard `account_spender.account.owner != from.owner` evaluates to `false` for all of them (owner is still X), so every entry is appended to `result` and returned. The correct termination condition must compare the full `Account` (which normalizes `None` and `Some([0u8;32])` via `effective_subaccount()` in `PartialEq`): [5](#0-4) 

`icrc103_get_allowances` is a public `#[query]` with no caller authentication or authorization: [6](#0-5) 

## Impact Explanation
An unprivileged attacker can enumerate spender identities, approved amounts, and expiration timestamps for every subaccount of a target principal that is lexicographically greater than the queried subaccount. This constitutes unauthorized disclosure of sensitive ledger state — financial metadata (who has been approved, for how much, until when) — across subaccount boundaries of the same principal. While `icrc2_allowance` is already a public query, it requires knowing the exact `(account, spender)` pair; this bug enables full enumeration without that prior knowledge, leaking the existence and details of allowances the caller has no right to discover. This is a significant ICRC ledger security impact with concrete user harm, qualifying as **High** under the "Significant... ledger... security impact with concrete user or protocol harm" category.

## Likelihood Explanation
The endpoint is exposed as a public `#[query]` with no access control. Any principal can call it with an arbitrary `from_account` in a single query call at zero cost. The precondition — that a principal has approved from more than one subaccount — is a standard pattern for wallets and DeFi protocols using subaccount-based account separation. No special privileges, victim interaction, or social engineering is required.

## Recommendation
Replace the owner-only break condition with a full-account comparison:

```rust
// Before (line 1204):
if account_spender.account.owner != from.owner {
    break;
}

// After:
if account_spender.account != from {
    break;
}
```

`Account::PartialEq` already normalizes `None` and `Some([0u8;32])` via `effective_subaccount()`, so this correctly handles the default-subaccount equivalence case. [4](#0-3) 

## Proof of Concept

State-machine test using the existing `list_allowances` helper: [7](#0-6) 

```rust
// Setup: owner X approves from two distinct subaccounts
icrc2_approve(X, subaccount=Some([1u8;32]), spender=S1, amount=100);
icrc2_approve(X, subaccount=Some([2u8;32]), spender=S2, amount=200);

// Attacker queries subaccount [1u8;32] only
let result = list_allowances(&env, ledger_id, attacker, GetAllowancesArgs {
    from_account: Some(Account { owner: X, subaccount: Some([1u8;32]) }),
    prev_spender: None,
    take: None,
});

// BUG: result contains BOTH allowances (sub:[1;32]→S1 AND sub:[2;32]→S2)
// EXPECTED: result contains only the allowance for sub:[1;32]→S1
assert!(result.unwrap().iter().all(|a| a.from_account.subaccount == Some([1u8;32])));
// This assertion FAILS under the current code
```

```rust
#[query]
fn icrc103_get_allowances(arg: GetAllowancesArgs) -> Result<Allowances, GetAllowancesError> {
    let from_account = arg.from_account.unwrap_or_else(|| Account {
        owner: ic_cdk::api::msg_caller(),
        subaccount: None,
    });
    let max_take_allowances = Access::with_ledger(|ledger| ledger.max_take_allowances());
    let max_results = arg
        .take
        .map(|take| take.0.to_u64().unwrap_or(max_take_allowances))
        .map(|take| std::cmp::min(take, max_take_allowances))
        .unwrap_or(max_take_allowances);
    Ok(get_allowances(
        from_account,
        arg.prev_spender,
        max_results,
        ic_cdk::api::time(),
    ))
```
