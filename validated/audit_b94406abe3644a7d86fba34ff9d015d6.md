Audit Report

## Title
Missing Upper Bound on RSA Public Exponent Enables Unmetered CPU Exhaustion in WebAuthn Ingress Validation — (`rs/crypto/internal/crypto_lib/basic_sig/rsa_pkcs1/src/lib.rs`)

## Summary

`RsaPublicKey::from_der_spki` enforces oddness of `e`, oddness of `n`, and a size range on `n` (2048–8192 bits), but imposes no upper bound on the bit-length of `e`. An attacker can register a WebAuthn identity whose COSE RSA key carries a 2047-bit odd exponent. Every ingress message signed with that identity forces the replica to compute `s^e mod n` with a ~120× larger exponent than the standard `e = 65537`, in the unmetered ingress-validation path, degrading throughput for all users on the subnet.

## Finding Description

In `from_der_spki` (`rs/crypto/internal/crypto_lib/basic_sig/rsa_pkcs1/src/lib.rs`, lines 87–140), three guards are applied after parsing the DER key via the `rsa` crate's `pkcs8::DecodePublicKey::from_public_key_der`:

- Line 100: `e` must be odd
- Line 109: `n` must be odd
- Lines 120–134: `n.bits()` must be in `[MINIMUM_RSA_KEY_SIZE, MAXIMUM_RSA_KEY_SIZE]` (2048–8192)

There is no check on `e.bits()`. The `rsa` crate 0.9.x (confirmed in `bazel/rust.MODULE.bazel` as `^0.9.10`) does not impose an upper bound on `e` either — the IC code explicitly adds the oddness and modulus-size checks precisely because the crate omits them, as the inline comments state. A 2047-bit odd value for `e` passes all three guards.

The COSE parsing path feeds arbitrary attacker-controlled bytes directly into this function:

`parse_cose_public_key` → `CosePublicKey::from_cbor` → `parse_rsa_pkcs1_sha256` (cose/src/lib.rs, lines 219–225) → `RsaPublicKey::from_components` → `spki_from_components` (encodes arbitrary-length `e` into ASN.1) → `from_der_spki`

Once stored, every call to `verify_pkcs1_sha256` (lib.rs, lines 152–167) invokes `self.key.verify(Pkcs1v15Sign::new::<sha2::Sha256>(), ...)`, which performs the full `s^e mod n` modular exponentiation via `num-bigint-dig`. The cost is O(log e) multiplications mod n; with `e` at 2047 bits vs. 17 bits (65537), this is ~120× more work per verification call.

The ingress validation path (`rs/validator/src/ingress_validation.rs`, lines 659–669) handles `RsaSha256PublicKeyDerWrappedCose` by calling `validate_webauthn_sig` → `verify_basic_sig_by_public_key` → `verify_pkcs1_sha256`. This path is outside the Wasm instruction-metered execution environment; no instruction counter is decremented.

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

A single attacker submitting ingress messages at the permitted rate, signed with a 2047-bit-exponent WebAuthn key, consumes ~120× more CPU per message in the unmetered ingress-validation path compared to a standard RSA key. This degrades throughput for all legitimate users sharing the same replica, without requiring any elevated privileges, governance vote, or subnet-majority corruption.

## Likelihood Explanation

The attack is fully self-service and requires no special access:
1. Generate a valid RSA key pair where `e` is a 2047-bit odd prime and `n` is a valid 2048-bit modulus (mathematically valid; RSA requires only that `gcd(e, φ(n)) = 1`).
2. Encode the key as a COSE RSA key (CBOR map with `kty=3, alg=-257`), DER-wrap it.
3. Derive a self-authenticating principal and submit signed ingress calls.
4. Each call forces the replica to compute `s^e mod n` with a 2047-bit exponent.

The attacker can produce valid signatures (using the corresponding private key), ensuring the verification completes the full exponentiation before returning. Even with an invalid signature, the `rsa` crate performs the exponentiation before checking the result. The attack is repeatable and persistent.

## Recommendation

Add an upper-bound check on `e.bits()` immediately after the existing oddness check in `from_der_spki` (`rs/crypto/internal/crypto_lib/basic_sig/rsa_pkcs1/src/lib.rs`, after line 106). Per NIST SP 800-56B Rev. 2 §6.2, the safe range is `3 ≤ e ≤ 2^32 − 1`. A concrete fix:

```rust
const MAXIMUM_RSA_EXPONENT_SIZE: u64 = 33; // bits; covers e=65537 (17 bits) with margin
if parsed.e().bits() > MAXIMUM_RSA_EXPONENT_SIZE {
    return Err(CryptoError::MalformedPublicKey {
        algorithm: AlgorithmId::RsaSha256,
        key_bytes: Some(bytes.to_vec()),
        internal_error: "RSA public exponent is too large".to_string(),
    });
}
```
