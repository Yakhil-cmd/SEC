Audit Report

## Title
Stale ICP/XDR Exchange Rate Used in Cycles Minting Without Age Validation - (File: `rs/nns/cmc/src/main.rs`)

## Summary
The `tokens_to_cycles()` function in `rs/nns/cmc/src/main.rs` converts ICP to cycles using the stored `icp_xdr_conversion_rate` with no check on the rate's age relative to the current canister time. When the Exchange Rate Canister (XRC) is unavailable, the CMC retries every minute but never invalidates or bounds the staleness of the stored rate. Any unprivileged caller can invoke `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` during such an outage and receive cycles computed at an arbitrarily stale rate, enabling overminting if the ICP market price has fallen since the last successful rate update.

## Finding Description
`tokens_to_cycles()` (lines 1900–1923 of `rs/nns/cmc/src/main.rs`) reads `state.icp_xdr_conversion_rate` and extracts `xdr_permyriad_per_icp` with only a `None` guard:

```rust
let xdr_permyriad_per_icp = state
    .icp_xdr_conversion_rate
    .as_ref()
    .map(|rate| rate.xdr_permyriad_per_icp);
```

There is no comparison of `rate.timestamp_seconds` against the current canister time. The rate's age is entirely unbounded at the point of use.

The rate is refreshed by `update_exchange_rate()` in `rs/nns/cmc/src/exchange_rate_canister.rs` (lines 236–280), which calls the XRC every 5 minutes on success and retries every minute on failure (lines 153–160 of the same file, `schedule_next_attempt`). On failure, the stored rate is never cleared or marked stale.

When a new rate is received, `validate_exchange_rate()` in `rs/nervous_system/clients/src/exchange_rate_canister_client.rs` (lines 111–129) only checks the number of data sources (`base_asset_num_received_rates`, `quote_asset_num_received_rates`); it performs no check that the returned rate's `timestamp` is within a tolerable window of the current time.

`do_set_icp_xdr_conversion_rate()` (lines 1022–1030 of `rs/nns/cmc/src/main.rs`) enforces only monotonicity — the new timestamp must exceed the current one — not freshness relative to wall-clock time. There is no upper bound on how old the stored rate may be when it is used for minting.

All three public minting entry points — `notify_top_up` (line 1139), `notify_mint_cycles` (line 1239), and `notify_create_canister` — call `tokens_to_cycles()` and thus consume the potentially stale rate. `notify_top_up` and `notify_mint_cycles` are `#[update]` methods with no access-control checks; any ingress sender can call them.

## Impact Explanation
When the XRC is unavailable and the ICP market price has dropped since the last successful rate update, any unprivileged user can call `notify_top_up` (or the other minting endpoints) and receive cycles computed at the stale, higher rate. This constitutes illegal minting: cycles are created in excess of what the current ICP/XDR peg justifies, diluting the cycles economy and representing a direct loss to the IC network's economic integrity. The XRC is an explicitly in-scope system canister, and the impact — overminting of cycles at a stale rate — constitutes a significant XRC/NNS security impact with concrete protocol harm, matching the **High ($2,000–$10,000)** impact class for "Significant XRC, NNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
XRC temporary unavailability is a realistic and explicitly anticipated operational scenario: the CMC already contains retry logic for it, and the integration test in `rs/nns/integration_tests/src/cycles_minting_canister_with_exchange_rate_canister.rs` (lines 162–185) explicitly exercises the case where the XRC returns errors across multiple consecutive 5-minute windows. During such a window, the stored rate silently ages with no upper bound. An attacker who observes that the XRC is down (observable via the public `get_icp_xdr_conversion_rate` query, which exposes the rate's `timestamp_seconds`) and that the stored rate is favorable (ICP price has since dropped) can immediately and repeatedly call `notify_top_up` to extract excess cycles. No special privileges are required; the attack is repeatable for the entire duration of the XRC outage.

## Recommendation
1. **Add a staleness check in `tokens_to_cycles()`**: Compare `rate.timestamp_seconds` against `now_seconds()`. If the rate exceeds a configurable maximum age (e.g., 30–60 minutes), return a `NotifyError` rather than minting at a stale rate.
2. **Add a timestamp freshness check in `validate_exchange_rate()`**: After receiving a rate from the XRC, verify that `exchange_rate.timestamp` is within a reasonable window of the current canister time before accepting it into state.
3. Consider exposing the rate age in monitoring/metrics so operators can detect and respond to prolonged XRC outages before the stored rate becomes dangerously stale.

## Proof of Concept
1. The CMC stores a rate at time T: `xdr_permyriad_per_icp = 100_000` (1 ICP = 10 XDR).
2. The XRC becomes unavailable (e.g., `StablecoinRateTooFewRates` error). The CMC retries every minute but all calls fail. The stored rate is never invalidated.
3. The real ICP market price drops to 5 XDR. The CMC's stored rate remains `100_000`.
4. An attacker calls `notify_top_up` with 1 ICP. `tokens_to_cycles()` reads the stale rate and mints cycles equivalent to 10 XDR — double the current market rate.
5. The attacker repeats until the XRC recovers.

A deterministic integration test can reproduce this by: (a) setting an initial rate via the mock XRC, (b) switching the mock XRC to return `StablecoinRateTooFewRates` for multiple 5-minute windows (as already done in `cycles_minting_canister_with_exchange_rate_canister.rs` lines 162–185), (c) advancing canister time by several hours, and (d) asserting that `notify_top_up` still succeeds and mints at the original stale rate rather than returning an error.
