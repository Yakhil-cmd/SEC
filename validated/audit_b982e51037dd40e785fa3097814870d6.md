### Title
Unnecessary `#[payable]` on `execute_scheduled` Permanently Locks Caller's NEAR Funds - (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The `execute_scheduled` function in the XCC router contract is marked `#[payable]`, allowing any caller to attach a non-zero NEAR deposit. The function body never reads, forwards, or refunds `env::attached_deposit()`. Because the router sub-account has no general-purpose NEAR withdrawal mechanism, any NEAR attached to this call is permanently locked.

---

### Finding Description

In `etc/xcc-router/src/lib.rs`, two functions carry the `#[payable]` attribute:

```rust
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
``` [1](#0-0) 

The function is intentionally open to any caller (the comment on line 146 explicitly states this). However, the attached NEAR deposit is silently accepted and never consumed. The NEAR runtime transfers the deposit into the router sub-account's balance, where it becomes inaccessible.

The only NEAR egress path on the router is `send_refund`, which transfers a hardcoded constant `REFUND_AMOUNT = NearToken::from_near(2)` back to the parent (Aurora Engine), and only when called by the parent after a successful promise:

```rust
pub fn send_refund(&self) -> Promise {
    ...
    Promise::new(parent).transfer(REFUND_AMOUNT)
}
``` [2](#0-1) 

There is no function to withdraw an arbitrary amount of NEAR from the router, so any deposit sent to `execute_scheduled` is permanently frozen.

The second `#[payable]` function, `deploy_upgrade`, is restricted to the parent (Aurora Engine) via `assert_preconditions()`, and the engine intentionally sends `fund_amount` to it for storage staking purposes — that case is by design and not a vulnerability. [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any NEAR attached by a caller to `execute_scheduled` is deposited into the router sub-account and can never be recovered. The router has no withdrawal function for arbitrary balances. The only egress (`send_refund`) transfers a fixed 2 NEAR to the parent engine, not to the original depositor, and is gated by the parent caller check.

---

### Likelihood Explanation

**Medium.** `execute_scheduled` is intentionally callable by any unprivileged NEAR account. A user who attaches NEAR (e.g., believing it is required to fund the cross-contract call being executed, or due to a wallet/tooling default) will permanently lose those funds. The open-call design makes this a realistic user-facing surface. [4](#0-3) 

---

### Recommendation

Remove the `#[payable]` attribute from `execute_scheduled`. The function does not require any NEAR deposit to operate — the scheduled promise's `attached_balance` is encoded in the `PromiseArgs` stored at scheduling time and is drawn from the router's existing balance, not from the caller's deposit.

If `#[payable]` must be retained for any reason (e.g., callback compatibility), add an explicit refund of the attached deposit:

```rust
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let deposit = env::attached_deposit();
    // ... existing logic ...
    if deposit.as_yoctonear() > 0 {
        Promise::new(env::predecessor_account_id()).transfer(deposit);
    }
}
```

---

### Proof of Concept

1. Alice's EVM address uses the XCC precompile with a `Delayed` promise, causing the Aurora Engine to call `schedule` on her router sub-account (e.g., `aabbcc...ff.aurora`).
2. Bob (any NEAR account) calls `execute_scheduled` on `aabbcc...ff.aurora` with `nonce: 0` and attaches `10 NEAR`.
3. The NEAR runtime transfers 10 NEAR into the router sub-account's balance before executing the function body.
4. `execute_scheduled` removes the stored promise and dispatches it — it never reads `env::attached_deposit()`.
5. Bob's 10 NEAR is now in the router sub-account. `send_refund` can only be called by the parent engine and only transfers the fixed `REFUND_AMOUNT` (2 NEAR) to the engine itself — not to Bob. There is no other withdrawal path.
6. Bob's 10 NEAR is permanently frozen. [1](#0-0) [5](#0-4)
