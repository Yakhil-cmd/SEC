### Title
Forward/Proving Divergence via Skipped EOA Signature Validation in `BasicBootloaderForwardSimulationConfig` - (File: `basic_bootloader/src/bootloader/config.rs`)

---

### Summary

`BasicBootloaderForwardSimulationConfig` sets `VALIDATE_EOA_SIGNATURE = false`, causing the sequencer's forward run to skip EOA signature verification entirely. The proving run uses `BasicBootloaderProvingExecutionConfig` with `VALIDATE_EOA_SIGNATURE = true` and enforces the check. Any transaction with an invalid or forged signature is accepted by the forward run but rejected by the proving run, producing a committed state that cannot be proven — a liveness-breaking forward/proving divergence.

---

### Finding Description

`BasicBootloaderExecutionConfig` exposes two compile-time flags: `VALIDATE_EOA_SIGNATURE` and `SIMULATION`. Four concrete implementations exist: [1](#0-0) 

The sequencer's forward run uses `BasicBootloaderForwardSimulationConfig` (`VALIDATE_EOA_SIGNATURE = false`, `SIMULATION = false`). The prover uses `BasicBootloaderProvingExecutionConfig` (`VALIDATE_EOA_SIGNATURE = true`, `SIMULATION = false`).

Both the ZK and Ethereum transaction validation paths share the same guard:

```rust
if !Config::VALIDATE_EOA_SIGNATURE | Config::SIMULATION {
    // skip — only charge native cost
} else {
    // ecrecover + from-address equality check
}
``` [2](#0-1) [3](#0-2) 

When `BasicBootloaderForwardSimulationConfig` is active, `!false | false` evaluates to `true`, so the entire `ecrecover` call and `recovered_from != from` check are bypassed. The transaction is accepted regardless of whether the signature is valid or whether `from` matches the signing key.

The proving run evaluates the same guard with `!true | false = false`, so it enters the `else` branch and enforces the full signature check.

---

### Impact Explanation

A transaction with `from = victim_address` and a random/invalid signature is accepted by the forward run (state mutated: nonce incremented, balance deducted) but rejected by the proving run (`InvalidTransaction::IncorrectFrom`). The sequencer has committed to a state root that the prover cannot reproduce. The block is unprovable, causing a **liveness failure** — the chain halts until the offending block is discarded or the sequencer is replaced.

Additionally, within the forward run itself, the victim's account state (nonce, balance) is modified without authorization, constituting an unauthorized state mutation even before the proving failure is considered.

---

### Likelihood Explanation

The sequencer's forward run relies on `BasicBootloaderForwardSimulationConfig` as documented and used in `forward_system/src/run/mod.rs`. Any transaction submitted to the mempool with a structurally valid but cryptographically invalid signature will pass forward validation. No special privilege is required — any unprivileged sender can submit such a transaction. The sequencer has no in-bootloader mechanism to detect the divergence before committing.

---

### Recommendation

1. **Enforce signature validation in the forward run** by switching the sequencer to `BasicBootloaderForwardETHLikeConfig` (which has `VALIDATE_EOA_SIGNATURE = true`, `SIMULATION = false`) or by removing `BasicBootloaderForwardSimulationConfig` entirely.
2. If the optimization of skipping ecrecover in the forward run is intentional, add an **explicit pre-bootloader mempool filter** that rejects transactions with invalid signatures before they reach the bootloader, and document this as a required invariant.
3. Add a **divergence test** that runs the same transaction set through both `BasicBootloaderForwardSimulationConfig` and `BasicBootloaderProvingExecutionConfig` and asserts identical accept/reject outcomes.

---

### Proof of Concept

1. Construct a valid RLP-encoded transaction with `from = 0xVICTIM`, `nonce = current_nonce`, and `(v, r, s)` set to arbitrary non-zero bytes that do not correspond to a valid secp256k1 signature over the transaction hash.
2. Submit to the sequencer. The forward run calls `validate_and_compute_fee_for_transaction<BasicBootloaderForwardSimulationConfig>`. The guard `!false | false = true` is taken; ecrecover is never called; the transaction is accepted.
3. The sequencer commits the resulting state root (victim nonce incremented, fee deducted).
4. The prover calls `validate_and_compute_fee_for_transaction<BasicBootloaderProvingExecutionConfig>`. The guard `!true | false = false` is taken; ecrecover runs; `recovered_from != 0xVICTIM`; the function returns `Err(InvalidTransaction::IncorrectFrom)`.
5. The prover cannot reproduce the committed state root → block is unprovable → chain liveness failure.

        // Even if we don't validate a signature, we still need to charge for ecrecover for equivalent behavior
        // Note that gas is charged already in intrinsic cost, so now
        // we only need to charge native resources.
        if !Config::VALIDATE_EOA_SIGNATURE | Config::SIMULATION {
            intrinsic_resources.charge(&Resources::from_native(
                <<S as SystemTypes>::Resources as Resources>::Native::from_computational(
                    ECRECOVER_NATIVE_COST,
                ),
            ))?;
        } else {
            if U256::from_be_slice(s) > U256::from_be_bytes(SECP256K1N_HALF) {
                return Err(InvalidTransaction::MalleableSignature.into());
            }

            let mut ecrecover_input = [0u8; 128];
            ecrecover_input[0..32].copy_from_slice(suggested_signed_hash.as_u8_array_ref());
            ecrecover_input[63] = (parity as u8) + 27;
            ecrecover_input[64..96][(32 - r.len())..].copy_from_slice(r);
            ecrecover_input[96..128][(32 - s.len())..].copy_from_slice(s);

            let mut ecrecover_output = ArrayBuilder::default();
            // We already charged gas for ecrecover in intrinsic cost, so we only need to charge native resources here.
            intrinsic_resources.with_infinite_ergs(|resources| {
                S::SystemFunctions::secp256k1_ec_recover(
                    ecrecover_input.as_slice(),
                    &mut ecrecover_output,
                    resources,
                    system.get_allocator(),
                )
                .map_err(SystemError::from)
            })?;

            if ecrecover_output.is_empty() {
                return Err(InvalidTransaction::IncorrectFrom {
                    recovered: B160::ZERO,
                    tx: from,
                }
                .into());
            }

            let recovered_from = B160::try_from_be_slice(&ecrecover_output.build()[12..])
                .ok_or(internal_error!("Invalid ecrecover return value"))?;

            if recovered_from != from {
                return Err(InvalidTransaction::IncorrectFrom {
                    recovered: recovered_from,
                    tx: from,
                }
                .into());
            }
        }
