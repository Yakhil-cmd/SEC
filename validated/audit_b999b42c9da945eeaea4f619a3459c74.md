### Title
Unverified `args.amount` in `ft_on_transfer` Enables Unbacked ERC-20 Minting via Permissionless NEP-141 Registration — (`engine/src/contract_methods/connector.rs`, `engine/src/engine.rs`)

---

### Summary

The Aurora Engine's `ft_on_transfer` entry point trusts the caller-supplied `args.amount` field without verifying that an actual NEP-141 token transfer occurred. Because `deploy_erc20_token` has no access control, any NEAR account can register itself as a NEP-141 token, then call `ft_on_transfer` directly with a fabricated amount to mint arbitrary ERC-20 tokens on Aurora with no real backing.

---

### Finding Description

**Step 1 — Permissionless NEP-141 registration via `deploy_erc20_token`**

The public entry point `deploy_erc20_token` in `engine/src/contract_methods/connector.rs` performs only a liveness check (`require_running`) and no caller-identity check:

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;   // ← only check
        ...
        DeployErc20TokenArgs::Legacy(nep141) => {
            let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;
```

Any NEAR account can call this with an arbitrary `nep141` account ID (including their own account) and have a corresponding ERC-20 token deployed and the bidirectional mapping `nep141 ↔ ERC-20 address` stored in Aurora's state.

**Step 2 — `ft_on_transfer` trusts caller-supplied `amount` without transfer verification**

`ft_on_transfer` dispatches to `receive_erc20_tokens` for any predecessor that is a registered NEP-141:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
```

Inside `receive_erc20_tokens`, the engine looks up the ERC-20 address for the predecessor and mints exactly `args.amount` tokens to the specified recipient:

```rust
let amount = args.amount.as_u128();
...
let erc20_token = get_erc20_from_nep141(&self.io, token)?;
let erc20_admin_address = current_address(current_account_id);
let result = self.call(
    &erc20_admin_address,
    &erc20_token,
    Wei::zero(),
    setup_receive_erc20_tokens_input(&recipient, amount),
    u64::MAX, ...
```

There is no check that the predecessor actually executed an `ft_transfer_call` that moved real NEP-141 tokens into Aurora before this call. The engine unconditionally mints `args.amount` ERC-20 tokens.

**The analog to `VestingEscrowSimple`**

| `VestingEscrowSimple` | Aurora Engine |
|---|---|
| `initialize(amount)` stores `amount` without checking `token.balanceOf(self) >= amount` | `ft_on_transfer(args)` mints `args.amount` ERC-20 tokens without verifying an actual NEP-141 transfer occurred |
| Bypassed by calling `initialize` directly instead of through the factory | Bypassed by calling `ft_on_transfer` directly instead of through a legitimate `ft_transfer_call` flow |

---

### Impact Explanation

An attacker can mint an unbounded quantity of any registered ERC-20 token on Aurora with zero real NEP-141 backing. These tokens are indistinguishable from legitimately bridged tokens and can be:

- Traded or used as collateral in DeFi protocols on Aurora, draining real user funds.
- Exited back to NEAR via the `exit_to_near` precompile, extracting real NEP-141 tokens from the bridge pool.

This constitutes **direct theft of user funds** and **bridge insolvency**. Impact: **Critical**.

---

### Likelihood Explanation

The attack requires no special privileges:
1. Create a NEAR account (trivial, costs ~0.1 NEAR).
2. Call `deploy_erc20_token` with that account ID (permissionless, no owner check).
3. Call `ft_on_transfer` directly on Aurora from that account with any `amount` and a recipient address.

All three steps are executable by any unprivileged NEAR user. Likelihood: **High**.

---

### Recommendation

1. **Add access control to `deploy_erc20_token`**: Restrict NEP-141 registration to the contract owner or a governed allowlist, consistent with the comment already present in the `WithMetadata` path (`"this transaction could be executed by the owner of the contract only"`).

2. **Verify actual transfer in `ft_on_transfer`**: The engine should not trust `args.amount` alone. One approach is to query the NEP-141 contract's balance of the Aurora account before and after the call (or use a cross-contract call receipt pattern) to confirm the real transfer amount, storing only the verified delta.

3. **Alternatively, enforce the NEP-141 `ft_transfer_call` invariant**: Require that `ft_on_transfer` is only reachable as a callback from a legitimate `ft_transfer_call` by checking that the call is part of a promise chain originating from the registered NEP-141 contract.

---

### Proof of Concept

```
// Step 1: Attacker registers their own account as a NEP-141 (no access control)
aurora.deploy_erc20_token(
    DeployErc20TokenArgs::Legacy("attacker.near".parse().unwrap())
)
// → ERC-20 deployed at address 0xABCD..., mapping stored: attacker.near ↔ 0xABCD...

// Step 2: Attacker calls ft_on_transfer directly from attacker.near
// predecessor_account_id = "attacker.near" (registered NEP-141)
// No actual ft_transfer_call was ever made; no real tokens moved.
aurora.ft_on_transfer(
    sender_id: "attacker.near",
    amount: "1000000000000000000000",   // fabricated, no backing
    msg: "<attacker_evm_address_hex>"
)
// → receive_erc20_tokens mints 1e21 ERC-20 tokens to attacker's EVM address
// → attacker now holds unbacked ERC-20 tokens indistinguishable from bridged ones
```

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
