Audit Report

## Title
SNS-Specific Upgrade Path Entry Cannot Be Updated After Initial Set — (`rs/nns/sns-wasm/src/sns_wasm.rs`)

## Summary

`insert_sns_specific_upgrade_path_entry` in the `UpgradePath` impl silently discards all update attempts on an already-occupied entry. The `Entry::Occupied` branch logs an "is being overwritten" message but never calls `occupied.insert(to)`, so the first value written is permanently frozen. Every subsequent NNS Governance proposal to correct or update that SNS-specific upgrade path returns success while leaving state unchanged.

## Finding Description

In `insert_sns_specific_upgrade_path_entry` (lines 1927–1951), the `Entry::Occupied` branch only prints a log message and exits without mutating the map: [1](#0-0) 

The sibling function `insert_upgrade_path_entry` (lines 1954–1970) correctly calls `occupied.insert(to)` in the same branch: [2](#0-1) 

The caller `insert_upgrade_path_entries` iterates over submitted upgrade steps and delegates to `insert_sns_specific_upgrade_path_entry` for every step when an `sns_governance_canister_id` is present: [3](#0-2) 

The function unconditionally returns `InsertUpgradePathEntriesResponse { error: None }` regardless of whether any state was actually mutated: [4](#0-3) 

The misleading log message ("is being overwritten") actively conceals the failure from operators monitoring canister logs.

## Impact Explanation

The SNS-specific upgrade path is the designated on-chain escape hatch for SNSes that cannot follow the standard upgrade path due to a bug, as documented in the proto definition: [5](#0-4) 

Once a `(governance_canister_id, current_version)` entry is set — even incorrectly — it can never be corrected via any on-chain mechanism. A mistaken first proposal permanently locks the wrong `next_version`. The affected SNS is left permanently unable to upgrade its canisters through the intended recovery mechanism, with no on-chain remedy. This constitutes a **significant SNS governance impact with concrete user and protocol harm**, matching the High ($2,000–$10,000) bounty tier: "Significant... NNS, SNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation

Triggering the bug requires two legitimate NNS Governance proposals of type `InsertUpgradePathEntries` targeting the same `(governance_canister_id, current_version)` pair — a routine operational sequence when correcting a previously submitted entry. No privilege escalation, key compromise, or governance corruption is required. The misleading log output ("is being overwritten") causes operators to believe the correction succeeded, delaying detection indefinitely.

## Recommendation

Add `occupied.insert(to)` in the `Entry::Occupied` branch of `insert_sns_specific_upgrade_path_entry`, mirroring the existing correct behavior in `insert_upgrade_path_entry`:

```rust
Entry::Occupied(mut occupied) => {
    println!(
        "Special Entry for {}  from {:?} to {:?} is being overwritten with new value {:?}",
        sns_governance_canister_id,
        occupied.key(),
        occupied.get(),
        to
    );
    occupied.insert(to);  // ← add this line
}
```
