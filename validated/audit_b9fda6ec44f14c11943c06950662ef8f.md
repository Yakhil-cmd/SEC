### Title
wNEAR Permanently Frozen in Engine Implicit Address on Setup Batch Failure — (`engine-precompiles/src/xcc.rs`, `engine/src/xcc.rs`, `engine/src/contract_methods/xcc.rs`)

---

### Summary

When a user invokes the XCC precompile for the first time (no router deployed), the precompile synchronously debits wNEAR from the user's EVM balance into the engine's implicit EVM address during EVM execution. The subsequent NEAR promise chain is structured so that `withdraw_wnear_to_router` only runs if the setup batch (CreateAccount + Transfer + DeployContract + FunctionCall) succeeds. If the setup batch fails — which an attacker can force by pre-creating the router sub-account — `withdraw_wnear_to_router` aborts early via `ERR_CALLBACK_OF_FAILED_PROMISE` without executing the EVM transfer, and no refund path exists. The wNEAR is permanently frozen in the engine's implicit address.

---

### Finding Description

**Step 1 — Synchronous wNEAR debit during EVM execution**

Inside `run_with_handle` in `engine-precompiles/src/xcc.rs`, when `required_near != ZERO_YOCTO`, the precompile immediately calls `transferFrom` on the wNEAR ERC-20 contract to move tokens from the caller (`sender`) to the engine's implicit address: [1](#0-0) 

This EVM state change is committed as part of the EVM transaction. No NEAR promise has been scheduled yet.

**Step 2 — Promise chain construction in `handle_precompile_promise`**

When `create_needed = true` and `required_near > 0`, `handle_precompile_promise` builds the following chain:

```
setup_batch (CreateAccount + Transfer + DeployContract + FunctionCall)
  └─► factory_update_address_version  [setup_id]
        └─► withdraw_wnear_to_router
              └─► send_refund
                    └─► user's actual call
``` [2](#0-1) 

`withdraw_wnear_to_router` is attached as a callback to `factory_update_address_version`, not directly to the setup batch. [3](#0-2) 

**Step 3 — `factory_update_address_version` propagates failure**

`factory_update_address_version` checks `promise_result_check()` and returns `ERR_ROUTER_DEPLOY_FAILED` if the setup batch failed, causing it to fail itself: [4](#0-3) 

**Step 4 — `withdraw_wnear_to_router` aborts without refunding**

`withdraw_wnear_to_router` checks `promise_result_check()` against its immediate predecessor (`factory_update_address_version`). Since that failed, it returns `ERR_CALLBACK_OF_FAILED_PROMISE` **without executing the EVM call** that would move wNEAR from the engine's implicit address to the router: [5](#0-4) 

The EVM `transferFrom` that would have moved wNEAR onward never runs. No subsequent callback refunds the user. The wNEAR remains at the engine's implicit EVM address indefinitely.

**Step 5 — Engine's implicit address has no private key**

The engine's implicit address is derived deterministically as `near_account_to_evm_address(engine_account_id.as_bytes())`: [6](#0-5) 

No party holds the private key for this address. The wNEAR ERC-20 balance at this address cannot be spent by any EVM transaction, and no engine admin method exists to recover arbitrary ERC-20 balances from it. The funds are permanently frozen.

---

### Impact Explanation

Every user whose first XCC call is sabotaged loses `STORAGE_AMOUNT` (2 NEAR) plus any NEAR they attached to the call. The wNEAR is irrecoverable without a contract upgrade. This meets **Critical — Permanent freezing of funds** (and at minimum **High — Temporary freezing of funds** pending an upgrade).

---

### Likelihood Explanation

The attack is cheap and permissionless. NEAR allows any account to create a sub-account of any existing account. An attacker can pre-create `{victim_address_hex}.{engine_account_id}` for any target user before that user's first XCC call. The cost is only the NEAR storage deposit for the new account (~0.1 NEAR). The attacker does not need any special privileges. The victim's wNEAR loss is `STORAGE_AMOUNT + attached_near` per attack. [7](#0-6) 

---

### Recommendation

The fix must ensure that if the setup batch fails, the wNEAR already debited from the user is returned. Two complementary approaches:

1. **Add a failure-path refund callback directly on the setup batch.** Attach a `refund_wnear_to_user` callback using `promise_attach_callback` that fires when the setup batch fails (NEAR's `then` semantics always fire callbacks; the callback checks `promise_result_check()` for `Some(false)` and executes the EVM refund).

2. **Defer the wNEAR debit.** Instead of debiting wNEAR synchronously in the EVM precompile, record the obligation and only debit inside `withdraw_wnear_to_router` after the setup batch has succeeded. This eliminates the window entirely.

---

### Proof of Concept

```
1. Attacker calls NEAR's `create_account` to create
   `{victim_evm_address_hex}.{engine_account_id}` with minimal balance.

2. Victim submits an EVM transaction calling the XCC precompile
   (CrossContractCallArgs::Eager) with any attached NEAR.

3. XCC precompile (run_with_handle):
   - get_code_version_of_address → None (engine storage has no record)
   - required_near = attached_near + STORAGE_AMOUNT (2 NEAR)
   - EVM transferFrom: victim_evm_addr → engine_implicit_addr  ✓ committed

4. handle_precompile_promise schedules:
   setup_batch → factory_update_address_version → withdraw_wnear_to_router
   → send_refund → user_call

5. setup_batch executes: CreateAccount fails (account already exists).
   Entire batch fails.

6. factory_update_address_version callback fires:
   promise_result_check() → Some(false) → ERR_ROUTER_DEPLOY_FAILED → fails.

7. withdraw_wnear_to_router callback fires:
   promise_result_check() → Some(false) → ERR_CALLBACK_OF_FAILED_PROMISE
   → returns early, no EVM call executed.

8. send_refund and user_call never execute.

9. Victim's wNEAR (STORAGE_AMOUNT + attached_near) is permanently frozen
   at engine_implicit_addr with no recovery path.
```
