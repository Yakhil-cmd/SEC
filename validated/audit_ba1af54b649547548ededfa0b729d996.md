### Title
L1→L2 Deposit Processing Permanently Halts If `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` Reverts — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

Every L1→L2 deposit transaction with a non-zero `total_deposited` amount calls `notify_l2_asset_tracker`, which invokes `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` as an EVM sub-call. If that sub-call reverts for any reason, the bootloader treats it as a fatal internal error and halts block processing entirely. Because L1→L2 transactions sit in the priority queue and cannot be skipped, a single permanently-reverting `L2AssetTracker` call bricks the chain.

---

### Finding Description

`notify_l2_asset_tracker` is called from `mint_base_token` for every L1→L2 deposit (operator fee, value mint, and refund legs). The function explicitly documents that a revert is fatal:

```rust
if failed {
    // A revert here means the chain's token accounting would be inconsistent.
    // Treated as a fatal system error — block processing cannot continue.
    return Err(internal_error!(
        "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
    ).into());
}
``` [1](#0-0) 

This error propagates as a `BootloaderSubsystemError::Internal`, which the ZK transaction loop treats as a non-recoverable block-level failure: [2](#0-1) 

The call is made unconditionally for any `amount > U256::ZERO`, which is always true for a deposit transaction: [3](#0-2) 

`mint_base_token` calls `notify_l2_asset_tracker` before `transfer_from_treasury`, meaning the critical treasury transfer is gated behind the external contract call: [4](#0-3) 

The `l1_chain_id` passed to the call is read from `L2AssetTracker` storage slot 154 at runtime: [5](#0-4) 

`L2AssetTracker` is an upgradeable proxy contract (`Initializable + OwnableUpgradeable + Ownable2StepUpgradeable`). Its `handleFinalizeBaseTokenBridgingOnL2` function can revert due to:
- A bug in the contract triggered by a specific `(fromChainId, amount)` combination
- State corruption caused by a prior L2 transaction interacting with the contract
- An upgrade that introduces a regression
- Any internal guard/modifier that reverts under unexpected conditions

The comment in the code acknowledges the risk but dismisses it with "we are certain that L2AssetTracker is available after the upgrade": [6](#0-5) 

This is the exact same structural flaw as the external report: a critical, non-skippable operation (`blacklistProtocol` / L1→L2 deposit processing) is coupled with a potentially-reverting sub-operation (withdrawal / `handleFinalizeBaseTokenBridgingOnL2`), and the revert of the sub-operation blocks the critical operation entirely.

---

### Impact Explanation

If `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` reverts for any L1→L2 deposit transaction, block processing halts with a fatal internal error. Because L1→L2 transactions are drawn from the priority queue and **cannot be invalidated or skipped** (as explicitly documented in the codebase): [7](#0-6) 

...the chain is permanently bricked. No further blocks can be produced. All user funds on L2 become inaccessible. This is a **chain-halting, funds-loss-path** impact.

---

### Likelihood Explanation

`L2AssetTracker` is an upgradeable EVM contract. Any of the following realistic scenarios triggers the halt:

1. **Attacker-crafted L2 transaction**: An attacker submits an L2 transaction that calls a public function on `L2AssetTracker` and corrupts its internal state (e.g., `interopInfo` mapping, `isAssetRegistered`, or `L1_CHAIN_ID` slot) such that subsequent `handleFinalizeBaseTokenBridgingOnL2` calls revert.
2. **Crafted L1→L2 deposit**: An attacker submits an L1→L2 deposit with parameters that trigger a revert path in `handleFinalizeBaseTokenBridgingOnL2` (e.g., an amount that causes an overflow in the contract's accounting).
3. **Contract upgrade regression**: A governance upgrade to `L2AssetTracker` introduces a bug that causes the function to revert.
4. **Initialization race**: If `L2AssetTracker` is not yet initialized when the first deposit arrives, the function may revert.

The attacker entry path for scenario 1 and 2 requires only an unprivileged L2 or L1 transaction — no privileged access is needed.

---

### Recommendation

Decouple the critical state change (treasury transfer + fee accounting) from the `L2AssetTracker` notification. Two options:

1. **Graceful degradation**: If `handleFinalizeBaseTokenBridgingOnL2` reverts, log the failure and continue processing the L1 transaction. The asset tracker accounting becomes temporarily inconsistent but the chain does not halt. A separate recovery mechanism can reconcile the accounting later.

2. **Separate the notification**: Analogous to the external report's recommendation, add a `needToNotify` parameter or a retry queue. Process the L1 deposit (treasury transfer, fee payment, refund) unconditionally, and enqueue failed asset-tracker notifications for later retry once the contract is fixed.

The key invariant to preserve: **L1→L2 deposit processing must never halt block execution**, regardless of the state of any L2 contract.

---

### Proof of Concept

1. Attacker deploys or interacts with `L2AssetTracker` via an L2 transaction that sets its internal state such that `handleFinalizeBaseTokenBridgingOnL2(l1_chain_id, amount)` reverts (e.g., by corrupting the `isAssetRegistered` mapping or triggering an overflow in `totalSuccessfulDepositsFromL1`).
2. Any user submits a legitimate L1→L2 deposit transaction (e.g., bridging ETH from L1).
3. The bootloader calls `mint_base_token` → `notify_l2_asset_tracker` → `run_single_interaction` targeting `L2AssetTracker`.
4. `handleFinalizeBaseTokenBridgingOnL2` reverts; `asset_tracker_result.failed()` returns `true`.
5. `notify_l2_asset_tracker` returns `Err(internal_error!(...))`.
6. This propagates as `BootloaderSubsystemError::Internal` through `mint_base_token` → `process_l1_transaction`.
7. The ZK tx loop receives `Err(TxError::Internal(err))` and returns the error, halting block processing.
8. No further blocks can be produced. The chain is bricked. [8](#0-7)

fn notify_l2_asset_tracker<'a, S: EthereumLikeTypes + 'a, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    system_functions: &mut HooksStorage<S, S::Allocator>,
    memories: RunnerMemoryBuffers<'a>,
    amount: U256,
    l1_chain_id: U256,
    resources: &mut S::Resources,
    tracer: &mut impl Tracer<S>,
    validator: &mut impl TxValidator<S>,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
    S::Metadata: ZkSpecificPricingMetadata
        + BasicMetadata<S::IOTypes, TransactionMetadata = TxLevelMetadata<S::IOTypes>>,
{
    if amount > U256::ZERO || Config::SIMULATION {
        // Encode calldata for handleFinalizeBaseTokenBridgingOnL2(uint256,uint256):
        // selector 0x03117c8c + abi-encoded (fromChainId, amount)
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]);
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution {
                resources_returned,
                result: asset_tracker_result,
            } = BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system,
                system_functions,
                memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,
                inf_ergs.clone(),
                &U256::ZERO,
                true, // should_make_frame - isolate state changes
                tracer,
                validator,
            )?;
            // Overwrite resources inside the closure so that
            // with_infinite_ergs correctly restores ergs afterwards.
            *inf_ergs = resources_returned;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
        })?;

        if failed {
            system_log!(
                system,
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 failed for amount {amount:?}\n"
            );
            // A revert here means the chain's token accounting would be inconsistent.
            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
            )
            .into());
        }
    }
    Ok(())

fn read_l1_chain_id<S: EthereumLikeTypes>(system: &mut System<S>) -> U256
where
    S::IO: IOSubsystemExt,
{
    // L2AssetTracker storage layout (verified via `forge inspect`):
    //   slots 0-100:   Initializable + OwnableUpgradeable + Ownable2StepUpgradeable
    //   slots 101-150: Ownable2Step __gap
    //   slot 151:      mapping chainBalance
    //   slot 152:      mapping assetMigrationNumber
    //   slot 153:      mapping isAssetRegistered
    //   slot 154:      uint256 L1_CHAIN_ID
    let l1_chain_id_slot = Bytes32::from_u256_be(&U256::from(154));
    let mut inf_resources = S::Resources::FORMAL_INFINITE;
    let chain_id = system
        .io
        .storage_read::<false>(
            ExecutionEnvironmentType::NoEE,
            &mut inf_resources,
            &L2_ASSET_TRACKER_ADDRESS,
            &l1_chain_id_slot,
        )
        .expect("must read L2AssetTracker L1_CHAIN_ID");
    U256::from_be_bytes(chain_id.as_u8_array())
