### Title
EOA Signature Validation Disabled in Production Forward Execution Path Causes Forward/Proving Divergence DoS - (`forward_system/src/run/mod.rs`)

### Summary

The production `run_block` function — the public sequencer-facing API — is hardcoded to use `BasicBootloaderForwardSimulationConfig`, which sets `VALIDATE_EOA_SIGNATURE: false`, completely disabling EOA signature verification during forward execution. The proving run uses `BasicBootloaderProvingExecutionConfig` with `VALIDATE_EOA_SIGNATURE: true`. An attacker who submits a transaction bearing an invalid signature that is not caught by the sequencer's external mempool validator can cause the forward run to accept and apply state changes that the proving run will subsequently reject, producing an unprovable block and a sustained DoS against the proving pipeline.

### Finding Description

**Root cause — config definition**

`basic_bootloader/src/bootloader/config.rs` defines four execution configs. The one used by the production forward path has signature validation explicitly off:

```rust
// config.rs lines 18-23
pub struct BasicBootloaderForwardSimulationConfig;

impl BasicBootloaderExecutionConfig for BasicBootloaderForwardSimulationConfig {
    const VALIDATE_EOA_SIGNATURE: bool = false;   // ← disabled
    const SIMULATION: bool = false;
}
```

The proving config, by contrast, enables it:

```rust
// config.rs lines 12-15
impl BasicBootloaderExecutionConfig for BasicBootloaderProvingExecutionConfig {
    const VALIDATE_EOA_SIGNATURE: bool = true;    // ← enforced
    const SIMULATION: bool = false;
}
```

**Root cause — guard condition in both Ethereum and ZK validation paths**

Both `basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs` (line 201) and `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` (line 253) share the same guard:

```rust
if !Config::VALIDATE_EOA_SIGNATURE | Config::SIMULATION {
    // skip ecrecover entirely
} else {
    // perform malleable-s check + ecrecover + from-address comparison
}
```

For `BasicBootloaderForwardSimulationConfig`: `!false | false = true` → the entire ecrecover block is skipped. The `from` address embedded in the transaction is accepted without any cryptographic verification.

**Root cause — production `run_block` wires in the unsafe config**

`forward_system/src/run/mod.rs` line 96:

```rust
pub fn run_block<...>(...) -> Result<BlockOutput, ForwardSubsystemError> {
    ...
    run_forward::<BasicBootloaderForwardSimulationConfig>(   // ← no sig validation
        oracle, &mut result_keeper, tracer, validator,
    );
    Ok(result_keeper.into())
}
```

This function is the sole implementation behind the `RunBlock` trait exposed via `forward_system/src/run/interface_impl.rs` (`RunBlockForward`), which is the interface the sequencer calls for every block.

**Divergence path**

The prover re-executes the same transactions using `BasicBootloaderProvingExecutionConfig`. Any transaction whose `from` field does not match the ecrecover result will be rejected by the proving run with `InvalidTransaction::IncorrectFrom`, while the forward run already committed its state changes. The two runs produce irreconcilable state roots; no valid ZK proof can be generated for that block.

### Impact Explanation

An attacker who can place a transaction with a forged or invalid signature into the sequencer's block (bypassing or absent external mempool signature validation) causes:

1. **Unprovable block**: The forward run applies state changes; the proving run rejects the transaction; the state roots diverge; no ZK proof can be produced.
2. **Proving pipeline DoS**: The prover cannot finalize the block on L1. The sequencer must detect the divergence, discard the block, and re-sequence — wasting prover resources and delaying L1 finality.
3. **Repeated attack**: Because `run_block` never validates signatures, the attacker can repeat this for every block as long as the external validator remains absent or bypassable.

### Likelihood Explanation

The `run_block` / `RunBlockForward` API accepts a `TxValidator` parameter that callers are expected to populate with signature-checking logic. If the sequencer operator deploys with a no-op validator (as is the case in all test harnesses using `NopTxValidator`), or if the sequencer's mempool validation has a gap, the attack is trivially executable by any unprivileged user who can submit a raw transaction. The ZKsync OS library itself provides no fallback: the bootloader unconditionally skips ecrecover when `BasicBootloaderForwardSimulationConfig` is active.

### Recommendation

**Short term**: Replace `BasicBootloaderForwardSimulationConfig` with `BasicBootloaderForwardETHLikeConfig` (which sets `VALIDATE_EOA_SIGNATURE: true`) in the production `run_block` function, or introduce a new config that enables signature validation while keeping other forward-run optimizations. The ecrecover cost is already charged as intrinsic gas, so the proving-equivalence accounting is unaffected.

**Long term**: Add a compile-time or runtime assertion that the config passed to `run_block` (the sequencer-facing API) always has `VALIDATE_EOA_SIGNATURE: true`, preventing future regressions. Separate the "sequencer optimization" config from the "call simulation" config so that the optimization cannot accidentally be applied to block production.

### Proof of Concept

1. Craft a valid EIP-1559 transaction structure with a legitimate `from` address but with `r`, `s`, `v` set to arbitrary non-zero bytes that do not correspond to a valid secp256k1 signature over the transaction hash.
2. Submit this raw transaction to the sequencer's RPC endpoint (`eth_sendRawTransaction`).
3. If the sequencer's external `TxValidator` is a no-op (or absent), the transaction enters the tx source fed to `run_block`.
4. `run_forward::<BasicBootloaderForwardSimulationConfig>` executes: the guard `!false | false = true` causes the ecrecover block to be skipped entirely; the transaction is accepted and state changes (nonce increment, value transfer, contract execution) are committed.
5. The prover re-executes the block with `BasicBootloaderProvingExecutionConfig`: `!true | false = false` → ecrecover runs → recovered address ≠ `from` → `InvalidTransaction::IncorrectFrom` → transaction rejected.
6. Forward-run state root ≠ proving-run state root → no valid ZK proof can be generated → block is permanently unprovable → DoS.
