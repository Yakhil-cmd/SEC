Audit Report

## Title
Ingress Payload Validation Uses Raw `balance()` Instead of `debited_balance()` During DTS Execution, Enabling Block-Space Waste - (File: `rs/ingress_manager/src/ingress_selector.rs`)

## Summary
When a canister has a paused Deterministic Time Slicing (DTS) execution, the ingress payload validator and builder check cycle affordability against `system_state.balance()` (raw `cycles_balance`), while the actual induction path checks against `system_state.debited_balance()` (`cycles_balance − ingress_induction_cycles_debit`). Messages that pass validation against the inflated raw balance are included in a block but immediately rejected during induction, consuming block space without delivering any work and at no cost to the submitter.

## Finding Description

**Validation path** (`rs/ingress_manager/src/ingress_selector.rs` L566–584):

```rust
let cumulative_ingress_cost =
    cycles_needed.entry(payer).or_insert_with(Cycles::zero);
if let Err(err) = self
    .cycles_account_manager
    .can_withdraw_cycles_with_threshold(
        &canister.system_state,   // ← passes SystemState, not CanisterState
        *cumulative_ingress_cost + ingress_cost,
        ...
    )
```

`can_withdraw_cycles_with_threshold` (`rs/cycles_account_manager/src/cycles_account_manager.rs` L903) evaluates:

```rust
if threshold + requested > system_state.balance() {  // ← raw cycles_balance
```

It has no knowledge of `ingress_induction_cycles_debit` and no branch for `has_paused_execution_or_install_code()`.

**Induction path** (`rs/cycles_account_manager/src/cycles_account_manager.rs` L335–348):

```rust
if canister.has_paused_execution_or_install_code() {
    if canister.system_state.debited_balance() < cycles + threshold {
        return Err(CanisterOutOfCyclesError { ... });
    }
    canister.system_state
        .add_postponed_charge_to_ingress_induction_cycles_debit(cycles);
    Ok(())
}
```

The induction path correctly uses `debited_balance()` = `cycles_balance − ingress_induction_cycles_debit` (`rs/replicated_state/src/canister_state/system_state.rs` L947–950).

**Confirmed gap**: A grep for `debited_balance` and `ingress_induction_cycles_debit` in `rs/ingress_manager/` returns zero matches. The correct value is present in certified state but never consulted during validation.

**Exploit flow**:
1. Attacker deploys canister C and triggers a long DTS execution (large `install_code` or compute-heavy update), causing `ingress_induction_cycles_debit = D` to accumulate, so `debited_balance() = balance() − D`.
2. While execution is paused, attacker submits many ingress messages to C with individual cost `c` where `c ≤ balance()` but `c > debited_balance()`.
3. Ingress selector validates each message against `balance()` — all pass.
4. During induction, `charge_ingress_induction_cost` checks `debited_balance()` — all fail with `CanisterOutOfCycles`.
5. Block space is consumed; no message executes; no induction fee is charged to anyone (the error is returned before any debit is applied).

The `cycles_needed` accumulator correctly sums costs across messages in the same block, but it is compared against `balance()`, not `debited_balance()`, so the accumulated debit from prior inductions in the same DTS window is invisible to the validator.

## Impact Explanation

This is a sustained, unprivileged application/platform-level DoS against a subnet. An attacker can fill blocks with ingress messages that are guaranteed to fail induction, degrading effective subnet throughput for the duration of any DTS execution on any canister. The per-canister round-robin quota limits the fraction of a single block that can be wasted per canister, but the attack can be run against multiple canisters simultaneously and sustained across all blocks while DTS executions are active — a normal, production condition. This matches the **High** impact class: "Application/platform-level DoS… or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation

- DTS executions are triggered by any sufficiently large `install_code` or compute-heavy update — a routine production event.
- No privileged role, key material, or subnet-majority corruption is required.
- Any unprivileged ingress sender can submit messages to any canister.
- The certified state already contains the correct `ingress_induction_cycles_debit` value; it is simply not read by the validator.
- The attack is repeatable for the full duration of each DTS execution and can be applied to multiple canisters concurrently.
- Likelihood: **Medium** — requires deliberate setup (triggering a DTS execution) but is fully reachable by an unprivileged actor with no special knowledge.

## Recommendation

In `validate_ingress` (`rs/ingress_manager/src/ingress_selector.rs`), mirror the branch logic of `charge_ingress_induction_cost`: when `canister.has_paused_execution_or_install_code()` is true, check the cumulative cost against `canister.system_state.debited_balance()` rather than delegating to `can_withdraw_cycles_with_threshold` (which only reads `balance()`). The cleanest approach is to expose a dedicated helper in `CyclesAccountManager` that replicates the exact branching of `charge_ingress_induction_cost` in read-only form, so validation and induction are structurally guaranteed to stay in sync.

## Proof of Concept

```
1. Deploy canister C on an application subnet.
   Fund C with exactly `threshold + cost` cycles, where `cost` is one
   ingress induction cost unit.

2. Send C a large update message that will run for multiple DTS rounds.
   After the first slice, `ingress_induction_cycles_debit = cost`,
   so `debited_balance() = threshold` (effectively 0 spendable above threshold).

3. While C's execution is paused, submit two ingress messages M1, M2
   each with induction cost `cost/2` from any unprivileged user principal.

4. Observe both M1 and M2 appear in the next finalized block:
   - Validation: `balance() = threshold + cost`; cumulative checks
     `cost/2 ≤ threshold + cost` and `cost ≤ threshold + cost` both pass.

5. Observe both M1 and M2 are rejected during induction with
   `CanisterOutOfCycles`:
   - Inductor: `debited_balance() = threshold < threshold + cost/2`.

6. Block space consumed; neither message executes; no fee charged.

A deterministic integration test using PocketIC can reproduce this by:
- Creating a canister with a fixed cycle balance.
- Installing a Wasm that triggers a multi-round DTS execution.
- Submitting ingress messages between DTS slices and asserting they appear
  in the block payload but are rejected at induction.
```
