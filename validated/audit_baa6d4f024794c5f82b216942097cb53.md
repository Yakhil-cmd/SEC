### Title
ckETH/ckERC20 Minter Lacks ICRC-21 Consent Message Support, Preventing User Verification of Withdrawal Parameters Before Signing - (File: rs/ethereum/cketh/minter/src/main.rs, rs/ethereum/cketh/minter/cketh_minter.did)

---

### Summary

The ckETH minter canister does not implement the ICRC-21 (`icrc21_canister_call_consent_message`) standard for its `withdraw_eth` and `withdraw_erc20` endpoints. This means ICRC-21-compatible wallets cannot display any transaction parameters — recipient address, withdrawal amount, gas limit, `maxFeePerGas`, `maxPriorityFeePerGas`, or the ERC-20 `data` payload — to the user before they approve the canister call. A malicious dapp can present these calls with attacker-chosen parameters and the user has no wallet-level mechanism to inspect them.

---

### Finding Description

The IC's ICRC-21 standard is the protocol-level equivalent of MetaMask's transaction approval screen: before a user's wallet signs an update call, it queries `icrc21_canister_call_consent_message` on the target canister to obtain a human-readable description of what the call will do. Both the ckBTC minter and the ckDOGE minter implement this standard for their withdrawal endpoints.

The ckBTC minter exposes `icrc21_canister_call_consent_message` for `retrieve_btc_with_approval`, displaying Amount and BTC destination address: [1](#0-0) 

The ckDOGE minter does the same for `retrieve_doge_with_approval`: [2](#0-1) 

The ckETH minter's entire public service interface contains **no** ICRC-21 or ICRC-10 endpoints: [3](#0-2) 

The `withdraw_eth` and `withdraw_erc20` update functions in `main.rs` have no corresponding consent message handler: [4](#0-3) [5](#0-4) 

For `withdraw_erc20` specifically, the minter constructs an EIP-1559 Ethereum transaction whose `data` field encodes an ERC-20 `transfer(to, value)` call: [6](#0-5) 

The full set of parameters that would be visible in a MetaMask-equivalent display — `nonce`, `gas_limit`, `max_fee_per_gas`, `max_priority_fee_per_gas`, `data`, `destination` — are all present in `Eip1559TransactionRequest` but are never surfaced to the user through any consent interface: [7](#0-6) 

The `UnsignedTransaction` type (with all critical fields) is defined and used internally but is only accessible post-hoc via the `get_events` debug endpoint, not before the user approves the call: [8](#0-7) 

---

### Impact Explanation

An unprivileged user interacting with the ckETH minter through an ICRC-21-compatible wallet (hardware wallet, Oisy, or any wallet implementing ICRC-21) receives no human-readable consent message. The wallet either blocks the call entirely or presents an opaque raw-bytes warning. In the latter case, the user cannot verify:

- The exact ETH recipient address
- The withdrawal amount (which is reduced by a dynamically estimated gas fee)
- For `withdraw_erc20`: the ERC-20 contract address being called, the `data` payload encoding the token transfer, and the gas parameters

A malicious dapp can craft a `withdraw_eth` or `withdraw_erc20` call with an attacker-controlled `recipient` address. Since the user has pre-approved the minter's allowance on the ledger (a prerequisite for withdrawal), the only remaining protection is the user manually inspecting the raw Candid arguments — which no standard wallet UI presents in a readable form.

For `withdraw_erc20`, the impact is higher: the Ethereum transaction's `data` field contains an ERC-20 `transfer` call to the `erc20_contract_address` (not the user's `recipient`), with the user's `recipient` encoded inside the calldata. Without ICRC-21, the user cannot distinguish a legitimate ERC-20 withdrawal from one targeting a malicious contract.

---

### Likelihood Explanation

The ckETH/ckERC20 withdrawal flow is the highest-value operation in the chain-fusion stack. The ICRC-21 gap is reachable by any unprivileged caller who has been directed to a malicious dapp. The attacker needs only to construct a valid `WithdrawalArg` or `WithdrawErc20Arg` with a substituted `recipient` field and present it to the user's wallet. No privileged access, no key compromise, and no subnet-majority attack is required. The prerequisite (ledger allowance) is routinely set by users following standard withdrawal instructions.

---

### Recommendation

**Short term:** Implement `icrc21_canister_call_consent_message` and `icrc10_supported_standards` in the ckETH minter for `withdraw_eth` and `withdraw_erc20`, following the pattern already established in the ckBTC and ckDOGE minters. The consent message for `withdraw_eth` should display at minimum: withdrawal amount, recipient address, estimated gas limit, and estimated max fee. The consent message for `withdraw_erc20` should additionally display the ERC-20 token symbol, the ERC-20 contract address, and the `data` field summary.

**Long term:** Add validation warnings in the consent message for unusual values (e.g., gas limit significantly above `CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT = 21_000` or `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT = 65_000`) to help users identify potentially anomalous withdrawal requests.

---

### Proof of Concept

1. A malicious dapp obtains a user's IC identity session (e.g., via a phishing site mimicking a legitimate ckETH frontend).
2. The dapp constructs a `withdraw_erc20` call with `recipient = attacker_eth_address`, `amount = user_full_balance`, `ckerc20_ledger_id = ckUSDC_ledger`.
3. The dapp submits this call through the user's ICRC-21-compatible wallet.
4. The wallet queries `icrc21_canister_call_consent_message` on the ckETH minter — the endpoint does not exist, so the wallet either rejects with `UnsupportedCanisterCall` or falls back to a raw-bytes display.
5. If the wallet falls back (permissive mode), the user sees no human-readable description of the recipient address or the ERC-20 `data` payload and approves the call.
6. The minter burns the user's ckETH (for gas) and ckUSDC, then constructs and signs an EIP-1559 transaction with `destination = USDC_contract`, `data = transfer(attacker_eth_address, user_full_balance)`, and submits it to Ethereum.
7. The user's USDC is transferred to the attacker's Ethereum address.

The ckBTC minter's ICRC-21 implementation confirms the pattern that should exist: [9](#0-8) 

The absence of any analogous code in `rs/ethereum/cketh/minter/src/` is the root cause. [10](#0-9)

type Eip1559TransactionPrice = record {
    // Maximum amount of gas transaction is authorized to consume.
    gas_limit : nat;

    // Maximum amount of Wei per gas unit that the transaction is willing to pay in total.
    // This covers the base fee determined by the network and the `max_priority_fee_per_gas`.
    max_fee_per_gas : nat;

    // Maximum amount of Wei per gas unit that the transaction gives to miners
    // to incentivize them to include their transaction (priority fee).
    max_priority_fee_per_gas : nat;

    // Maximum amount of Wei that can be charged for the transaction,
    // computed as `max_fee_per_gas * gas_limit`
    max_transaction_fee : nat;

    // Timestamp of when the price was estimated.
    // Nanoseconds since the UNIX epoch.
    timestamp : opt nat64;
};

/// Estimate price of EIP-1559 transaction based on the
/// `base_fee_per_gas` included in the last finalized block.
#[query]
async fn eip_1559_transaction_price(
    token: Option<Eip1559TransactionPriceArg>,
) -> Eip1559TransactionPrice {
    let gas_limit = match token {
        None => CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT,
        Some(Eip1559TransactionPriceArg { ckerc20_ledger_id }) => {
            match read_state(|s| s.find_ck_erc20_token_by_ledger_id(&ckerc20_ledger_id)) {
                Some(_) => CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT,
                None => {
                    if ckerc20_ledger_id == read_state(|s| s.cketh_ledger_id) {
                        CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT
                    } else {
                        ic_cdk::trap(format!(
                            "ERROR: Unsupported ckERC20 token ledger {ckerc20_ledger_id}"
                        ))
                    }
                }
            }
        }
    };
    match read_state(|s| s.last_transaction_price_estimate.clone()) {
        Some((ts, estimate)) => {
            let mut result = Eip1559TransactionPrice::from(estimate.to_price(gas_limit));
            result.timestamp = Some(ts);
            result
        }
        None => ic_cdk::trap("ERROR: last transaction price estimate is not available"),
    }
}

#[update]
async fn withdraw_eth(
    WithdrawalArg {
        amount,
        recipient,
        from_subaccount,
    }: WithdrawalArg,
) -> Result<RetrieveEthRequest, WithdrawalError> {
    let caller = validate_caller_not_anonymous();
    let _guard = retrieve_withdraw_guard(caller).unwrap_or_else(|e| {
        ic_cdk::trap(format!(
            "Failed retrieving guard for principal {caller}: {e:?}"
        ))
    });

    let destination = validate_address_as_destination(&recipient).map_err(|e| match e {
        AddressValidationError::Invalid { .. } | AddressValidationError::NotSupported(_) => {
            ic_cdk::trap(e.to_string())
        }
        AddressValidationError::Blocked(address) => WithdrawalError::RecipientAddressBlocked {
            address: address.to_string(),
        },
    })?;

    let amount = Wei::try_from(amount).expect("failed to convert Nat to u256");

    let minimum_withdrawal_amount = read_state(|s| s.cketh_minimum_withdrawal_amount);
    if amount < minimum_withdrawal_amount {
        return Err(WithdrawalError::AmountTooLow {
            min_withdrawal_amount: minimum_withdrawal_amount.into(),
        });
    }

    let client = read_state(LedgerClient::cketh_ledger_from_state);
    let now = ic_cdk::api::time();
    log!(INFO, "[withdraw]: burning {:?}", amount);
    match client
        .burn_from(
            Account {
                owner: caller,
                subaccount: from_subaccount,
            },
            amount,
            BurnMemo::Convert {
                to_address: destination,
            },
        )
        .await
    {
        Ok(ledger_burn_index) => {
            let withdrawal_request = EthWithdrawalRequest {
                withdrawal_amount: amount,
                destination,
                ledger_burn_index,
                from: caller,
                from_subaccount: from_subaccount.and_then(LedgerSubaccount::from_bytes),
                created_at: Some(now),
            };

            log!(
                INFO,
                "[withdraw]: queuing withdrawal request {:?}",
                withdrawal_request,
            );

            mutate_state(|s| {
                process_event(
                    s,
                    EventType::AcceptedEthWithdrawalRequest(withdrawal_request.clone()),
                );
            });
            Ok(RetrieveEthRequest::from(withdrawal_request))
        }
        Err(e) => Err(WithdrawalError::from(e)),
    }
}
