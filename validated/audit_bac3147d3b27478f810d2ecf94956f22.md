Audit Report

## Title
Self-Withdrawal Guard Bypass via Bech32 Case Normalization in `retrieve_btc` and `retrieve_btc_with_approval` — (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary
The guards in `retrieve_btc` and `retrieve_btc_with_approval` that prevent withdrawals to the minter's own address use a plain string equality check against a lowercase bech32 string. Because `BitcoinAddress::parse` accepts bech32 addresses in any case, an attacker can supply the minter's address in uppercase or mixed case, bypass the string check, and cause ckBTC to be permanently burned while BTC is sent to the minter's own address — resulting in irreversible loss of user funds.

## Finding Description
`derive_minter_address_str` calls `BitcoinAddress::display(network)` which internally calls `encode_bech32` → `bech32::encode(hrp, data, Variant::Bech32)`. The `bech32` crate's `encode` always produces all-lowercase output (e.g., `bc1q...`). The guard `if args.address == main_address_str` at lines 158 and 256 is a byte-for-byte string comparison against this lowercase string.

However, `BitcoinAddress::parse` routes both `'b'` and `'B'` (and `'t'`/`'T'`) to `parse_bip173_address`, which calls `bech32::decode(address)` (case-insensitive per BIP-173) and compares HRPs using `.to_lowercase()`. An attacker who supplies `BC1QABCDEF...` (uppercase of the minter's address) passes the string guard (since `"BC1Q..." != "bc1q..."`), but `BitcoinAddress::parse` decodes it to the identical `P2wpkhV0(minter_pkhash)`. The `RetrieveBtcRequest` is then enqueued with `address: P2wpkhV0(minter_pkhash)`, ckBTC is burned from the caller's account, and the minter's `ProcessLogic` task sends BTC to its own address.

## Impact Explanation
This matches **High ($2,000–$10,000): Significant Chain Fusion / ck-token security impact with concrete user or protocol harm.** The caller's ckBTC is permanently burned (irreversible ledger operation) while BTC is sent to the minter's own address rather than a user-controlled address. The user receives no BTC in exchange for their burned ckBTC. Additionally, the minter receives unexpected UTXOs at its main address, which may interfere with change-output tracking and reimbursement logic. The ckBTC supply is reduced without a corresponding reduction in BTC reserves, creating an accounting imbalance.

## Likelihood Explanation
The attack requires no privileges. The minter's main address is publicly queryable via `get_btc_address`. The uppercase transformation is trivial. Both `retrieve_btc` and `retrieve_btc_with_approval` are affected. The only prerequisite is holding a nonzero ckBTC balance. The attack is fully deterministic and repeatable.

## Recommendation
Replace the string comparison with a structural comparison after parsing both addresses:

```rust
let main_address = state::read_state(|s| runtime.derive_minter_address(s));
let parsed_address = BitcoinAddress::parse(&args.address, btc_network)?;
if parsed_address == main_address {
    ic_cdk::trap("illegal retrieve_btc target");
}
```

`BitcoinAddress` implements `Eq` on the decoded byte representation, making the check encoding- and case-agnostic. Apply the same fix in both `retrieve_btc` and `retrieve_btc_with_approval`.

## Proof of Concept
1. Call `get_btc_address` (public query) → receive `bc1qabcdef...` (the minter's main address).
2. Convert to uppercase: `BC1QABCDEF...`.
3. Call `retrieve_btc { address: "BC1QABCDEF...", amount: <min_amount> }`.
4. Line 158: `"BC1QABCDEF..." == "bc1qabcdef..."` → `false` → guard does **not** trap.
5. Line 173: `BitcoinAddress::parse("BC1QABCDEF...", network)` → `Ok(P2wpkhV0(minter_pkhash))` → succeeds.
6. ckBTC burn executes; `RetrieveBtcRequest { address: P2wpkhV0(minter_pkhash), ... }` is enqueued.
7. Minter's `ProcessLogic` task sends BTC to its own address; caller's ckBTC is permanently destroyed with no BTC redemption.

A deterministic integration test can be written using PocketIC: initialize the minter, fund a test account with ckBTC, call `retrieve_btc` with the uppercased minter address, and assert that the call succeeds (no trap) and the ckBTC balance decreases while no BTC is sent to the test account.
