### Title
`MintSnsTokens` Proposal Action Bypasses 7-Day Minting Limit, Allowing Unlimited SNS Token Inflation - (File: `rs/sns/governance/src/proposal.rs`)

---

### Summary

The `MintSnsTokens` governance proposal action in SNS Governance does not enforce the 7-day minting upper bound that was designed to protect against excessive token inflation. The correct limit check is commented out and replaced with a stub returning `Decimal::MAX` (effectively no limit). Additionally, unlike `TransferSnsTreasuryFunds`, there is no secondary execution-time limit check for `MintSnsTokens`. Any SNS neuron holder with sufficient voting power can pass a `MintSnsTokens` proposal minting an arbitrary number of tokens.

---

### Finding Description

The `TokenProposalAction` trait defines `recent_amount_total_upper_bound_tokens()` to cap the 7-day rolling total of token-moving proposals. For `TransferSnsTreasuryFunds`, this is properly implemented:

```rust
fn recent_amount_total_upper_bound_tokens(valuation: &Valuation) -> Result<Decimal, String> {
    transfer_sns_treasury_funds_7_day_total_upper_bound_tokens(*valuation)...
}
```

For `MintSnsTokens`, the correct implementation is **commented out** and replaced with a stub that returns `Decimal::MAX`:

```rust
/* TODO(NNS1-2982): Uncomment.
fn recent_amount_total_upper_bound_tokens(valuation: &Valuation) -> Result<Decimal, String> {
    mint_sns_tokens_7_day_total_upper_bound_tokens(*valuation)...
}
*/

// TODO(NNS1-2982): Delete.
fn recent_amount_total_upper_bound_tokens(_valuation: &Valuation) -> Result<Decimal, String> {
    Ok(Decimal::MAX)
}
``` [1](#0-0) 

This means `treasury_valuation_if_proposal_amount_is_small_enough_or_err()` — which computes `max_tokens - spent_tokens` and rejects proposals exceeding the remainder — will always compute `Decimal::MAX - spent_tokens`, which is always positive, so the check never rejects any `MintSnsTokens` proposal regardless of amount. [2](#0-1) 

Furthermore, `TransferSnsTreasuryFunds` has a **second, execution-time** limit check (`transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`) called inside `perform_transfer_sns_treasury_funds`: [3](#0-2) 

But `perform_mint_sns_tokens` has **no such check** — it directly mints the requested amount with no limit validation: [4](#0-3) 

The execution dispatch confirms `MintSnsTokens` goes directly to `perform_mint_sns_tokens` without retrieving or applying any stored valuation: [5](#0-4) 

Compare this to `TransferSnsTreasuryFunds`, which retrieves the stored valuation and passes it to the execution-time check: [6](#0-5) 

The limit function `mint_sns_tokens_7_day_total_upper_bound_tokens` exists and is correct — it is simply never called in production: [7](#0-6) 

---

### Impact Explanation

An SNS neuron holder with sufficient voting power can submit a `MintSnsTokens` proposal with `amount_e8s = u64::MAX` (or any arbitrarily large value). The proposal passes validation because the upper bound check returns `Decimal::MAX`. Upon execution, the SNS ledger mints the full requested amount to any target account. This causes unbounded SNS token inflation, diluting all existing token holders and potentially collapsing the token's value. The 7-day rolling limit (designed to cap minting at 25% of treasury value for medium-sized treasuries, or 300,000 XDR for large ones) is entirely bypassed.

**Impact: High** — unlimited token minting, complete ledger conservation failure for SNS tokens.

---

### Likelihood Explanation

Any SNS neuron holder who can pass a proposal (i.e., commands a voting majority, which in early-stage SNS deployments is often a single whale neuron) can exploit this. The entry path is the standard `make_proposal` ingress call — no privileged access, no key compromise, no subnet-majority attack required. The attacker only needs the same voting power required to pass any other SNS proposal.

**Likelihood: Medium** — requires governance majority, but SNS deployments frequently have concentrated voting power, and the limit is supposed to protect against even governance-approved over-minting.

---

### Recommendation

1. Uncomment the correct `recent_amount_total_upper_bound_tokens` implementation for `MintSnsTokens` (remove the `/* TODO(NNS1-2982): Uncomment. */` block and delete the stub).
2. Add an execution-time check analogous to `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` inside `perform_mint_sns_tokens`, using the `ActionAuxiliary::MintSnsTokens(valuation)` stored at proposal submission time.
3. Retrieve the stored valuation in the `Action::MintSnsTokens` dispatch arm (mirroring the `TransferSnsTreasuryFunds` arm) and pass it to the execution-time check.

---

### Proof of Concept

1. Deploy an SNS where the attacker controls a neuron with majority voting power.
2. Submit a `MintSnsTokens` proposal:
   ```
   MintSnsTokens {
       amount_e8s: Some(u64::MAX),  // ~184 billion tokens
       to_principal: Some(attacker_principal),
       to_subaccount: None,
       memo: None,
   }
   ```
3. The proposal passes `validate_and_render_mint_sns_tokens` because `recent_amount_total_upper_bound_tokens` returns `Decimal::MAX`, so `proposal_amount_tokens > allowance_remainder_tokens` is always `false`.
4. The proposal is adopted and `perform_mint_sns_tokens` executes, calling `self.ledger.transfer_funds(u64::MAX, 0, None, attacker_account, memo)` with no limit check.
5. The attacker receives `u64::MAX` e8s of SNS tokens, inflating total supply by ~184 billion tokens. [8](#0-7) [9](#0-8)

    async fn perform_mint_sns_tokens(
        &mut self,
        mint: MintSnsTokens,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: mint
                .to_principal
                .ok_or(GovernanceError::new_with_message(
                    ErrorType::InvalidProposal,
                    "Expected mint to have a target principal",
                ))?
                .0,
            subaccount: mint
                .to_subaccount
                .as_ref()
                .map(|s| bytes_to_subaccount(&s.subaccount[..]))
                .transpose()?,
        };
        let amount_e8s = mint.amount_e8s.ok_or(GovernanceError::new_with_message(
            ErrorType::InvalidProposal,
            "Expected MintSnsTokens to have an an amount_e8s",
        ))?;
        self.ledger
            .transfer_funds(amount_e8s, 0, None, to, mint.memo())
            .await?;
        Ok(())
    }
