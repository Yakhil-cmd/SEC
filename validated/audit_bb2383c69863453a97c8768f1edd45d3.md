Audit Report

## Title
Missing Staleness Check on ICP/XDR Conversion Rate Allows Excess Cycles Minting - (File: rs/nns/cmc/src/main.rs)

## Summary

The `tokens_to_cycles` function in the Cycles Minting Canister reads only the numeric value of the cached `icp_xdr_conversion_rate` and never validates the rate's `timestamp_seconds` against the current time. When the Exchange Rate Canister (XRC) is unavailable or polling is halted, the stale rate remains in use indefinitely, allowing any unprivileged caller of `notify_top_up`, `notify_mint_cycles`, or `notify_create_canister` to mint cycles at a rate that no longer reflects the live ICP market price, constituting illegal over-issuance of cycles backed by burned ICP.

## Finding Description

`tokens_to_cycles` at `rs/nns/cmc/src/main.rs` lines 1900–1922 extracts only `rate.xdr_permyriad_per_icp` via `.map(|rate| rate.xdr_permyriad_per_icp)`, silently discarding `rate.timestamp_seconds`. The only guard is a `None` check; a rate of any age is accepted unconditionally:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp); // timestamp_seconds dropped
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles { ... }.to_cycles(amount)),
            None => Err(...),
        }
    })
}
```

A grep across all of `rs/nns/cmc/src/` for `stale`, `staleness`, `MAX_RATE_AGE`, or `rate_age` returns zero results, confirming no staleness guard exists anywhere in the minting path.

Two independent staleness windows exist:

1. **XRC failure path** (`rs/nns/cmc/src/exchange_rate_canister.rs` lines 149–161): on `FailedToRetrieveRate`, `FailedToSetRate`, or `InvalidRate`, the next attempt is scheduled one minute later via `get_rate_at_next_minute`. If the XRC remains unavailable, the CMC retries every minute but never clears or invalidates the cached rate. The rate can grow arbitrarily stale while minting continues.

2. **`Disabled` state path** (`rs/nns/cmc/src/exchange_rate_canister.rs` lines 133–138 and 291–310): when a governance proposal sets reason `DivergedRate`, `update_exchange_rate_canister_state` is set to `UpdateExchangeRateState::Disabled`. The `schedule_next_attempt` guard returns immediately without rescheduling, halting all XRC polling until a subsequent governance proposal re-enables it. During this window — which can span hours — the frozen rate is still used by `tokens_to_cycles` without any age check.

The `IcpXdrConversionRate` struct (referenced at `rs/nns/cmc/src/main.rs` line 217–218) carries `timestamp_seconds` precisely to enable freshness checks, but that field is never read in the minting path.

## Impact Explanation

Cycles are backed by burned ICP; over-issuing cycles against a stale (inflated) rate is illegal minting — the protocol issues more computational resource than the deposited ICP justifies at current market prices. The ICP is still burned (`burn_and_log` is called after successful minting), so the attacker receives excess cycles at no additional ICP cost. This matches the allowed High impact: *"Significant XRC… security impact with concrete user or protocol harm"* and approaches the Critical threshold of *"illegal minting… of exorbitant ICP/Cycles"* depending on the staleness window and price movement magnitude.

## Likelihood Explanation

An unprivileged user needs only two conditions: (a) the CMC's cached rate is stale (XRC unavailable for an extended period, or `Disabled` state active), and (b) the ICP market price has fallen since the rate was last set. Condition (a) is reachable without any special privileges — XRC outages occur during canister upgrades or repeated retrieval errors, and the `Disabled` state is a documented operational mode. Condition (b) is a normal market event. The attack requires no victim interaction, no phishing, and no privileged access; the attacker simply transfers ICP to their canister's subaccount and calls `notify_top_up`. The attack is repeatable for the entire duration of the staleness window.

## Recommendation

In `tokens_to_cycles`, compare `rate.timestamp_seconds` against `now_seconds()` and return a retriable error when the age exceeds a defined threshold (e.g., 30 minutes, well above the normal 5-minute refresh interval):

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        match state.icp_xdr_conversion_rate.as_ref() {
            Some(rate) => {
                let age = now_seconds().saturating_sub(rate.timestamp_seconds);
                if age > MAX_RATE_AGE_SECONDS {
                    return Err(NotifyError::Other {
                        error_code: NotifyErrorCode::Internal as u64,
                        error_message: format!("ICP/XDR rate is stale ({age}s old)"),
                    });
                }
                Ok(TokensToCycles {
                    xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
                    cycles_per_xdr: state.cycles_per_xdr,
                }.to_cycles(amount))
            }
            None => Err(...),
        }
    })
}
```

`MAX_RATE_AGE_SECONDS` should be set to a value larger than `REFRESH_RATE_INTERVAL_SECONDS` (300 s) but small enough to bound worst-case price divergence (e.g., 1800 s). Additionally, consider emitting a metric or log when the rate age approaches the threshold so operators can detect XRC unavailability proactively.

## Proof of Concept

1. Deploy a local replica with the CMC and a mock XRC that returns errors on every call.
2. Pre-populate the CMC state with `icp_xdr_conversion_rate = IcpXdrConversionRate { xdr_permyriad_per_icp: 100_000, timestamp_seconds: T }` (10 XDR/ICP).
3. Advance the replica clock by 2 hours (simulating XRC unavailability); confirm via `get_icp_xdr_conversion_rate` that the cached rate is unchanged.
4. Transfer 1 ICP to the CMC subaccount for a test canister and call `notify_top_up`.
5. Observe that `tokens_to_cycles` returns cycles computed at 10 XDR/ICP regardless of elapsed time.
6. Repeat with a mock that simulates a live rate of 8 XDR/ICP and confirm the minted cycles exceed what the live rate would produce by ~25%.
