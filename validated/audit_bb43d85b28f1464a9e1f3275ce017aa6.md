Audit Report

## Title
Panic (DoS) via crafted BitString `unused_bits` in `public_key_from_der` — (`rs/crypto/utils/threshold_sig_der/src/lib.rs`)

## Summary
`public_key_from_der` validates the BLS public key length by checking the `bit_count` field of the parsed `ASN1Block::BitString`, but this check is not equivalent to checking the actual byte-vector length. When `simple_asn1` parses a BitString with `unused_bits=8` and 97 data bytes, it produces `bit_count=768` (passing the guard) but `data_vec.len()=97`. The subsequent `copy_from_slice` into a 96-byte array panics, crashing the replica process.

## Finding Description
In `public_key_from_der`, the only length guard is:

```rust
if *len != PUBLIC_KEY_SIZE * 8 {
    return Err(format!("unexpected key length: {len} bits"));
}
``` [1](#0-0) 

`simple_asn1` computes `bit_count = (content_bytes.len() - 1) * 8 - unused_bits` and sets `data_vec = content_bytes[1..]`. It does not validate that `unused_bits ∈ [0, 7]`. With `unused_bits=8` and 97 data bytes: `97 * 8 - 8 = 768 = PUBLIC_KEY_SIZE * 8`, so the guard at line 52 passes. However, `data_vec.len() = 97 ≠ 96 = PUBLIC_KEY_SIZE`. The call at line 58:

```rust
key_bytes.copy_from_slice(key.as_slice());
``` [2](#0-1) 

panics unconditionally because Rust's `copy_from_slice` requires equal lengths. The sibling library `der_utils` already recognized this exact gap and added the correct cross-check:

```rust
if *bits_count != key_bytes.len() * 8 {
    return Err(Self::parsing_error("Inconsistent key length"));
}
``` [3](#0-2) 

`threshold_sig_der` lacks this guard entirely. [4](#0-3) 

## Impact Explanation
A Rust panic unwinds the thread or aborts the process. If `public_key_from_der` is invoked during certificate verification, state sync, or any replica path that processes externally-supplied DER blobs outside of the canister sandbox, the replica process crashes. This matches the allowed High impact: **Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000). Because the crafted DER is trivial to construct and can be replayed, a single attacker can repeatedly crash targeted replicas.

## Likelihood Explanation
The crafted DER requires only a one-byte change to a valid BLS public key DER blob (setting the `unused_bits` byte to `0x08` and appending one extra zero byte). No cryptographic material or privileged access is needed to construct the payload. Reachability depends on whether callers of `public_key_from_der` accept network-supplied bytes; BLS public keys are parsed from IC certificates and subnet configurations, which can be observed and in some protocol paths submitted by unprivileged or Byzantine peers below the fault threshold.

## Recommendation
Add an explicit byte-length check on `key` before `copy_from_slice`, mirroring the pattern already used in `der_utils`:

```rust
if key.len() != PUBLIC_KEY_SIZE {
    return Err(format!("unexpected key data length: {} bytes", key.len()));
}
key_bytes.copy_from_slice(key.as_slice());
```

This check must appear after the `bit_count` guard and before `copy_from_slice`. [5](#0-4) 

## Proof of Concept
Construct a DER blob that is structurally identical to a valid BLS SubjectPublicKeyInfo but with the BitString content set to `0x08` (unused_bits=8) followed by 97 zero bytes. Concretely, replace the BitString content bytes `[0x00, <96 bytes>]` with `[0x08, <97 bytes>]` and update the length fields accordingly. Feed this to `public_key_from_der`. `simple_asn1::from_der` returns `BitString(_, 768, vec![0u8; 97])`. The `*len != 768` check passes. `copy_from_slice(&[0u8; 97])` into `[0u8; 96]` panics with `source slice length (97) does not match destination slice length (96)`. A unit test in `rs/crypto/utils/threshold_sig_der/tests/tests.rs` can reproduce this deterministically with no network access. [6](#0-5)
