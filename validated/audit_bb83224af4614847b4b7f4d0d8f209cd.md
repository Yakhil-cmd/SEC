### Title
Stale ETH Backing After `set_eth_connector_contract_account` Migration Causes Permanent Fund Freeze - (File: engine/src/contract_methods/connector.rs)

### Summary
`set_eth_connector_contract_account` updates only the stored connector account ID and serialization type. It performs no migration of the ETH backing held by the old connector contract. After the switch, the engine's `withdraw` path routes `engine_withdraw` calls to the new connector, which holds zero ETH backing, permanently freezing all user funds that were deposited through the old connector.

### Finding Description
`set_eth_connector_contract_account` in `engine/src/contract_methods/connector.rs` writes exactly two storage keys:

```rust
set_connector_account_id(io, &args.account);
set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
``` [1](#0-0) 

No other state is touched. In particular:

1. **The ETH backing stays in the old connector.** All NEP-141 tokens representing bridged ETH are held by the old connector contract. The new connector starts with zero balance.

2. **`withdraw` immediately routes to the new connector.** The `withdraw` entrypoint calls `return_promise(io, env, "engine_withdraw", args, ONE_YOCTO)`, which dispatches `engine_withdraw` to whichever account `get_connector_account_id` returns at call time — now the new, empty connector. [2](#0-1) 

3. **`ft_on_transfer` from the old connector is misrouted.** The base-token deposit path checks `predecessor_account_id == get_connector_account_id(&io)?`. After the switch the old connector no longer matches, so any `ft_on_transfer` it sends is treated as an ERC-20 transfer. Because the old connector's NEP-141 account is not in `nep141_erc20_map`, the call fails and tokens are returned — deposits from the old connector are frozen. [3](#0-2) 

4. **`nep141_erc20_map` is unchanged.** The bidirectional map (`KeyPrefix::Nep141Erc20Map` / `KeyPrefix::Erc20Nep141Map`) persists across the migration, so any ERC-20 tokens whose NEP-141 backing was held by the old connector remain mapped to stale ERC-20 addresses. [4](#0-3) 

### Impact Explanation
**Critical — Permanent freezing of funds.**

All ETH that users deposited through the old connector is permanently inaccessible:
- `withdraw` calls `engine_withdraw` on the new connector, which has no backing → call fails or reverts.
- The old connector's NEP-141 balance is stranded with no engine path to reach it.
- Users' EVM balances in Aurora cannot be redeemed for real ETH.

### Likelihood Explanation
The owner of the Aurora Engine contract (`require_owner_only`) can call `set_eth_connector_contract_account` at any time, including as part of a legitimate connector migration or upgrade. The function is exposed as a public NEAR entrypoint (`set_eth_connector_contract_account` in `engine/src/lib.rs`). [5](#0-4) 

The function is also reachable as a private callback (`env.assert_private_call()` branch), meaning it can be triggered by the engine itself during cross-contract flows. Any legitimate migration attempt silently produces the freeze with no on-chain warning. [6](#0-5) 

### Recommendation
- Before switching the connector account, atomically transfer all NEP-141 backing from the old connector to the new one, or reject the call if the old connector holds a non-zero balance.
- Alternatively, remove the migration functionality entirely and require a full redeployment with a fresh state, as suggested in the analogous external report.
- At minimum, emit a clear error if `set_eth_connector_contract_account` is called while the old connector holds user funds.

### Proof of Concept
1. Users deposit ETH into Aurora; the old connector (`old_connector.near`) accumulates 1 000 ETH in NEP-141 tokens.
2. Owner calls `set_eth_connector_contract_account` with `account = new_connector.near`. Only the two storage keys are updated; `new_connector.near` holds 0 ETH.
3. User calls `withdraw(recipient, 1 ETH)` on the engine. The engine dispatches `engine_withdraw` to `new_connector.near`.
4. `new_connector.near` has no balance → the call fails (or succeeds but sends 0 ETH).
5. The 1 000 ETH sitting in `old_connector.near` is permanently unreachable through any engine entrypoint, because `ft_on_transfer` from `old_connector.near` is now misrouted as an ERC-20 transfer and rejected. [7](#0-6) [8](#0-7)

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}

fn get_connector_account_id<I: IO>(io: &I) -> Result<AccountId, ContractError> {
    io.read_storage(&construct_contract_key(
        EthConnectorStorageId::EthConnectorAccount,
    ))
    .ok_or(errors::ERR_CONNECTOR_STORAGE_KEY_NOT_FOUND)
    .and_then(|x| {
        x.to_value()
            .map_err(|_| errors::ERR_BORSH_DESERIALIZE.as_bytes())
    })
    .map_err(Into::into)
}

pub fn set_connector_account_id<I: IO + Copy>(mut io: I, account_id: &AccountId) {
    io.write_borsh(
        &construct_contract_key(EthConnectorStorageId::EthConnectorAccount),
        account_id,
    );
