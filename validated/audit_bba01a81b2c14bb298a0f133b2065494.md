### Title
Unrestricted Allowance Enumeration via Missing Caller Authorization in `icrc103_get_allowances` - (File: `rs/ledger_suite/icrc1/ledger/src/main.rs`)

### Summary
The `icrc103_get_allowances` query endpoint on the ICRC-1 ledger canister accepts an arbitrary `from_account` argument without verifying that the caller is the owner of that account. Any unprivileged ingress sender can enumerate the complete allowance list — including spender identities and approved amounts — for any account on the ledger.

### Finding Description
In `rs/ledger_suite/icrc1/ledger/src/main.rs`, the `icrc103_get_allowances` handler resolves the target account from the caller-supplied argument with no authorization check:

```rust
#[query]
fn icrc103_get_allowances(arg: GetAllowancesArgs) -> Result<Allowances, GetAllowancesError> {
    let from_account = arg.from_account.unwrap_or_else(|| Account {
        owner: ic_cdk::api::msg_caller(),
        subaccount: None,
    });
    // No check: is msg_caller() == from_account.owner ?
    Ok(get_allowances(
        from_account,
        arg.prev_spender,
        max_results,
        ic_cdk::api::time(),
    ))
}
``` [1](#0-0) 

When `arg.from_account` is `Some(victim_account)`, the caller's identity is completely ignored and the full allowance list of `victim_account` is returned. The `from_account` defaults to the caller only when the field is `None` — but any caller can supply any other principal's account explicitly.

The `GetAllowancesError` type already defines an `AccessDenied` variant, which is never returned by this implementation, strongly indicating the restriction was intended but never enforced: [2](#0-1) 

The underlying `get_allowances` function in `rs/ledger_suite/icrc1/ledger/src/lib.rs` iterates the stable `ALLOWANCES_MEMORY` B-tree and returns all non-expired allowances for the given account, up to `max_take_allowances`: [3](#0-2) 

The analog to the reported Solidity bug is direct: the Solidity `burn(account, amount)` accepted an arbitrary `account` without checking `msg.sender == account`; here `icrc103_get_allowances(from_account)` accepts an arbitrary `from_account` without checking `msg_caller() == from_account.owner`.

### Impact Explanation
An attacker can call `icrc103_get_allowances` with any victim's `Account` as `from_account` and receive a paginated list of every (spender, amount, expiry) tuple the victim has ever approved. This reveals:

- The complete set of spenders the victim has authorized
- The exact token amounts approved to each spender
- Expiration timestamps of each approval

This constitutes a **ledger conservation / governance authorization bug**: sensitive financial relationships that users expect to be private (only queryable by the account owner or a specific spender via `icrc2_allowance`) are fully exposed to any unauthenticated ingress caller. In practice this enables targeted attacks — an adversary who learns that account A has approved spender B for a large amount can attempt to compromise spender B's key or craft phishing attacks with precise knowledge of the victim's financial state.

### Likelihood Explanation
Exploitation requires only a standard ingress query call to the ICRC-1 ledger canister with a crafted `GetAllowancesArgs`. No privileged access, no key material, and no threshold corruption is needed. The endpoint is publicly listed in the canister's Candid interface: [4](#0-3) 

Any boundary node user or canister can issue this query at zero cost. Likelihood is **high**.

### Recommendation
Add a caller authorization check before dispatching to `get_allowances`. The `AccessDenied` error variant already exists for this purpose:

```rust
#[query]
fn icrc103_get_allowances(arg: GetAllowancesArgs) -> Result<Allowances, GetAllowancesError> {
    let caller = ic_cdk::api::msg_caller();
    let from_account = arg.from_account.unwrap_or_else(|| Account {
        owner: caller,
        subaccount: None,
    });
    // Enforce: caller must own the queried account
    if from_account.owner != caller {
        return Err(GetAllowancesError::AccessDenied {
            reason: "caller is not the owner of the queried account".to_string(),
        });
    }
    // ... rest unchanged
}
```

This mirrors the fix described in the external report: restrict the operation to the initiating principal's own data.

### Proof of Concept

1. Alice (`principal_A`) calls `icrc2_approve` to grant spender `principal_S` an allowance of 1,000,000 tokens.
2. Attacker (`principal_X`, completely unrelated) sends an ingress query:
   ```
   icrc103_get_allowances({
     from_account: Some({ owner: principal_A, subaccount: None }),
     prev_spender: None,
     take: None
   })
   ```
3. The ledger resolves `from_account = principal_A`'s account (line 1216–1219), skips any caller check, and calls `get_allowances(principal_A_account, ...)`.
4. The response contains `[{ from_account: principal_A, to_spender: principal_S, allowance: 1_000_000, expires_at: None }]`.
5. Attacker now knows Alice's complete approval graph without Alice's consent and without holding any tokens or permissions on the ledger. [5](#0-4)

fn icrc103_get_allowances(arg: GetAllowancesArgs) -> Result<Allowances, GetAllowancesError> {
    let from_account = arg.from_account.unwrap_or_else(|| Account {
        owner: ic_cdk::api::msg_caller(),
        subaccount: None,
    });
    let max_take_allowances = Access::with_ledger(|ledger| ledger.max_take_allowances());
    let max_results = arg
        .take
        .map(|take| take.0.to_u64().unwrap_or(max_take_allowances))
        .map(|take| std::cmp::min(take, max_take_allowances))
        .unwrap_or(max_take_allowances);

        from_account,
        arg.prev_spender,

pub fn get_allowances(
    from: Account,
    spender: Option<Account>,
    max_results: u64,
    now: u64,
) -> Allowances {
    let mut result = vec![];
    let start_account_spender = match spender {
        Some(spender) => AccountSpender {
            account: from,
            spender,
        },
        None => AccountSpender {
            account: from,
            spender: Account {
                owner: Principal::from_slice(&[0_u8; 0]),
                subaccount: None,
            },
        },
    };
    ALLOWANCES_MEMORY.with_borrow(|allowances| {
        for (account_spender, storable_allowance) in
            allowances.range(start_account_spender.clone()..)
        {
            if spender.is_some() && account_spender == start_account_spender {
                continue;
            }
            if result.len() >= max_results as usize {
                break;
            }
            if account_spender.account.owner != from.owner {
                break;
            }
            if let Some(expires_at) = storable_allowance.expires_at
                && expires_at.as_nanos_since_unix_epoch() <= now
            {
                continue;
            }
            result.push(Allowance103 {
                from_account: account_spender.account,
                to_spender: account_spender.spender,
                allowance: Nat::from(storable_allowance.amount),
                expires_at: storable_allowance
                    .expires_at
                    .map(|t| t.as_nanos_since_unix_epoch()),
            });
        }
    });
    result
