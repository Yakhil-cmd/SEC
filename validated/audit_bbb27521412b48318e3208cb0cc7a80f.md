### Title
Unchecked NEAR Transfer Promise Result in wNEAR Unwrap Flow Causes Permanent Fund Loss - (`engine/src/contract_methods/connector.rs`)

### Summary

In `exit_to_near_precompile_callback`, when a user unwraps wNEAR ERC-20 tokens to native NEAR (the `:unwrap` flow), the final NEAR `Transfer` promise is dispatched without any callback to verify its success. If the NEAR transfer fails (e.g., the recipient named account does not exist), the user's wNEAR ERC-20 tokens are already permanently burned from the EVM state, and the unwrapped NEAR is silently stranded in the Aurora engine account with no recovery path.

### Finding Description

The wNEAR unwrap exit flow proceeds as follows:

1. The user calls the `exit_to_near` precompile with the `:unwrap` suffix (e.g., `recipient.near:unwrap`) from an ERC-20 wNEAR contract on Aurora EVM.
2. In `exit_erc20_token_to_near`, the `Message::UnwrapWnear` branch sets `transfer_near_args` to `Some(TransferNearArgs { target_account_id, amount })` and schedules a `near_withdraw` call on the wNEAR NEP-141 contract. [1](#0-0) 

3. A callback `exit_to_near_precompile_callback` is attached to the `near_withdraw` promise.
4. Inside the callback, when `near_withdraw` succeeds (line 214), a `PromiseAction::Transfer` is created and dispatched via `handler.promise_create_batch` and `handler.promise_return`: [2](#0-1) 

**No callback is attached to this NEAR transfer promise.** If the transfer fails, execution silently continues and returns `None` (line 230). The `error_refund` branch on line 231 only handles the case where `near_withdraw` itself failed — it does not handle failure of the subsequent NEAR transfer. [3](#0-2) 

The wNEAR ERC-20 tokens are burned from the user's EVM balance at the point the precompile runs, before any NEAR-side promise executes. There is no mechanism to re-mint them if the downstream NEAR transfer fails.

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the NEAR `Transfer` action fails (e.g., the recipient is a named NEAR account that does not exist), the unwrapped NEAR remains in the Aurora engine account indefinitely. The user's wNEAR ERC-20 tokens have already been burned from the EVM state. There is no callback, no refund path, and no administrative recovery function for this specific failure mode. The funds are permanently inaccessible to the user.

### Likelihood Explanation

**Medium.** Any user who specifies a named NEAR account (e.g., `alice.near`) that does not yet exist as the `:unwrap` recipient will trigger this. Named NEAR accounts must be explicitly created before they can receive a `Transfer` action; unlike 64-character implicit accounts, they are not auto-created on receipt. A user making a typo, specifying a future account, or whose target account is deleted between submission and execution will lose funds. This is reachable by any unprivileged EVM user holding wNEAR ERC-20 tokens on Aurora.

### Recommendation

Attach a second callback to the NEAR `Transfer` promise that checks whether the transfer succeeded. If the transfer failed, the callback should re-mint the equivalent amount of wNEAR ERC-20 tokens to the original sender (analogous to the existing `refund_on_error` path used when `near_withdraw` fails). Concretely, the `PromiseBatchAction` for the NEAR transfer should be followed by a `promise_attach_callback` targeting a new engine method (e.g., `transfer_near_callback`) that inspects `handler.promise_result(0)` and, on failure, calls `engine::refund_on_error` with the wNEAR ERC-20 address and the original amount.

### Proof of Concept

**Step 1.** User holds wNEAR ERC-20 tokens on Aurora EVM.

**Step 2.** User submits an EVM transaction calling the wNEAR ERC-20 contract's burn/exit function, which internally calls the `exit_to_near` precompile with input encoding `flag=1 | refund_address | amount | "nonexistent.near:unwrap"`. The account `nonexistent.near` does not exist on NEAR.

**Step 3.** The precompile burns the user's wNEAR ERC-20 balance and schedules:
- Promise A: `near_withdraw({"amount": "<N>"})` on the wNEAR NEP-141 contract
- Callback B: `exit_to_near_precompile_callback` on the Aurora engine [4](#0-3) 

**Step 4.** Promise A succeeds. The Aurora engine account receives N yoctoNEAR.

**Step 5.** Callback B executes. `handler.promise_result(0)` is `Successful`. The code creates Promise C: `Transfer { amount: N }` to `nonexistent.near` and calls `handler.promise_return(promise_id)`. [5](#0-4) 

**Step 6.** Promise C fails because `nonexistent.near` does not exist. No callback is attached to Promise C, so the failure is never observed by the engine. The N yoctoNEAR remains in the Aurora engine account.

**Step 7.** The user's wNEAR ERC-20 balance is zero (burned in Step 2). The user has permanently lost their funds. The `error_refund` path is never triggered because `near_withdraw` (Promise A) succeeded. [6](#0-5)
