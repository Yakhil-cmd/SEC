### Title
Single-Step Owner Transfer Allows Outgoing Owner to Retain Permanent Backdoor via `attach_full_access_key` - (File: engine/src/contract_methods/admin.rs)

### Summary

The `set_owner` function performs an immediate, single-step ownership transfer with no acceptance step from the new owner. The outgoing owner can atomically (or sequentially, since they control the timing) call `attach_full_access_key` before calling `set_owner`, permanently retaining a Full Access Key (FAK) on the Aurora Engine NEAR account. After the transfer, the new owner believes they have exclusive control, but the old owner's FAK allows them to deploy arbitrary contract code, drain NEAR balances, or delete the account — bypassing all `require_owner_only` guards entirely.

### Finding Description

`set_owner` in `engine/src/contract_methods/admin.rs` immediately overwrites `state.owner_id` with the caller-supplied `new_owner` in a single transaction:

```rust
pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }
        state.owner_id = args.new_owner;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
``` [1](#0-0) 

There is no two-step commit-accept pattern and no time-lock. The new owner has no opportunity to inspect the engine's state before ownership is finalized.

`attach_full_access_key`, also gated by `require_owner_only`, adds a NEAR Full Access Key directly to the Aurora Engine account:

```rust
pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;
    ...
    let action = PromiseAction::AddFullAccessKey { public_key, nonce: 0 };
    ...
}
``` [2](#0-1) 

A NEAR Full Access Key is not subject to any contract-level access control. It allows the holder to sign arbitrary NEAR transactions on behalf of the engine account — including deploying new contract code, transferring NEAR tokens, and deleting the account — completely bypassing `require_owner_only`.

The attack sequence:

1. Old owner calls `attach_full_access_key` with their own public key → FAK is added to the Aurora Engine NEAR account.
2. Old owner calls `set_owner(new_owner)` → `state.owner_id` is updated; new owner believes they have exclusive control.
3. Old owner uses their retained FAK to deploy a malicious contract to the Aurora Engine account, or to drain NEAR balances, or to delete the account.

Because the old owner controls the timing of both calls and NEAR does not have a public mempool that would allow the new owner to interpose, this is not a probabilistic race — it is a guaranteed, deterministic attack.

A secondary attack vector exists via `set_eth_connector_contract_account`, which the outgoing owner can call before `set_owner` to redirect all ETH connector operations (deposits, withdrawals) to a malicious contract:

```rust
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I, env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;

            require_owner_only(&state, &env.predecessor_account_id())?;
        }
        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;
        set_connector_account_id(io, &args.account);
        ...
    })
}
``` [3](#0-2) 

### Impact Explanation

**Critical.** The `attach_full_access_key` vector gives the outgoing owner permanent, irrevocable control over the Aurora Engine NEAR account after the ownership transfer. Using the FAK, the attacker can:

- Deploy a malicious WASM contract to the engine account, stealing all ETH and ERC-20 token balances held by the engine.
- Transfer NEAR tokens from the engine account.
- Delete the engine account, permanently freezing all user funds.

The `set_eth_connector_contract_account` vector allows the outgoing owner to redirect all user deposits and withdrawals to an attacker-controlled contract, constituting direct theft of user funds in motion.

Both impacts fall within the allowed scope: **direct theft of user funds** and **permanent freezing of funds**.

### Likelihood Explanation

The likelihood is **medium**. The scenario requires the outgoing owner to act maliciously specifically during an ownership handoff. Ownership transfers are infrequent but high-stakes events. The attack requires no special technical capability beyond what the outgoing owner already possesses, and it is deterministic (not probabilistic). The new owner has no on-chain mechanism to detect or prevent it before the transfer is finalized.

### Recommendation

1. **Implement a two-step ownership transfer**: introduce a `propose_owner` function that records a pending new owner, and a separate `accept_ownership` function that the new owner must call. Ownership only changes upon acceptance.
2. **Alternatively, implement a time-lock**: after `set_owner` is called, enforce a delay (e.g., `upgrade_delay_blocks`) before the new owner takes effect, giving the new owner time to inspect the engine state and abort if harmful actions were taken.
3. **Audit state before accepting ownership**: document clearly that the new owner must verify `get_paused_precompiles`, `get_eth_connector_contract_account`, and the set of access keys on the engine account before accepting ownership.
4. **Consider restricting `attach_full_access_key`**: given its extreme power, consider removing this function or requiring a multi-sig / governance process.

### Proof of Concept

```
// Step 1: Outgoing owner adds their own FAK to the engine account
aurora_engine.attach_full_access_key(
    { public_key: ATTACKER_PUBLIC_KEY },
    caller = current_owner  // passes require_owner_only
)

// Step 2: Outgoing owner transfers ownership (appears legitimate)
aurora_engine.set_owner(
    { new_owner: "new_owner.near" },
    caller = current_owner
)

// Step 3: new_owner.near now believes they control the engine.
// Attacker signs a DeployContract action using their FAK:
sign_and_send_near_tx(
    signer_account = "aurora",
    signer_key = ATTACKER_PRIVATE_KEY,  // FAK — bypasses all contract guards
    actions = [DeployContract(malicious_wasm)]
)

// Result: malicious contract is live on the aurora engine account.
// All user ETH and ERC-20 balances are now under attacker control.
```

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
}
