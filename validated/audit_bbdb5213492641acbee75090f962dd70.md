### Title
Silent Precompile Failure Allows ERC-20 Burn Without NEP-141 Transfer, Causing Permanent Fund Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

### Summary

In `EvmErc20.sol` and `EvmErc20V2.sol`, the `withdrawToNear` (and `withdrawToEthereum`) functions burn ERC-20 tokens first, then call the exit precompile via inline assembly. The return value of the precompile `call` is captured in a local variable `res` but **never checked**. If the precompile call fails for any reason (e.g., invalid NEAR recipient account ID), the ERC-20 tokens are permanently burned while no NEP-141 tokens are ever transferred to the user. This is the direct analog to the Sake inconsistency: the design intent documented in the precompile source ("expected to be called from the ERC-20 burn function") implies atomicity between burn and transfer, but the Solidity contracts do not enforce it.

### Finding Description

In `EvmErc20.sol`, `withdrawToNear` executes as follows:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ERC-20 tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}
``` [1](#0-0) 

The identical pattern exists in `EvmErc20V2.sol`: [2](#0-1) 

The `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` validates the recipient as a NEAR `AccountId` and returns `Err(ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID"))` for any malformed input: [3](#0-2) 

When the precompile returns an error, the EVM `call` opcode returns `0` (failure) but does **not** revert the caller's prior state changes. The `_burn` that already executed is permanent. The precompile comment itself acknowledges the expected coupling: [4](#0-3) 

The same unchecked-return pattern applies to `withdrawToEthereum` in both contracts, calling the `ExitToEthereum` precompile at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`: [5](#0-4) 

### Impact Explanation

A user whose `withdrawToNear` call triggers a precompile failure loses their ERC-20 tokens permanently:

- The `_burn` is irreversible on the EVM side.
- No NEP-141 `ft_transfer` promise is ever scheduled on the NEAR side.
- The user receives no tokens on either chain.

This constitutes **permanent freezing of funds** (Critical per the allowed impact scope).

### Likelihood Explanation

The failure mode is directly user-reachable without any privileged access:

1. A user calls `withdrawToNear` with a NEAR account ID that fails `AccountId::parse()` — e.g., an account ID containing uppercase letters, special characters, or exceeding 64 bytes. NEAR account ID rules are non-obvious to Ethereum-native users.
2. The precompile's `parse_recipient` returns `Err`, causing the `call` opcode to return `0`.
3. The Solidity code ignores `res` and returns normally.
4. The user's ERC-20 balance is zero; no NEP-141 tokens arrive.

No admin access, governance capture, or external oracle is required. Any token holder interacting with the bridge can trigger this by supplying a malformed recipient string.

### Recommendation

Add a `require` check on the precompile call return value in both `withdrawToNear` and `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures the entire transaction reverts if the precompile fails, preserving the user's ERC-20 balance and maintaining the atomicity that the design intends.

### Proof of Concept

1. Deploy `EvmErc20` (or use an existing bridged token).
2. Mint tokens to `attacker` address.
3. Call `withdrawToNear("INVALID ACCOUNT!!!", amount)` from `attacker`.
4. The precompile's `parse_recipient` fails on `AccountId::parse()` for the invalid string.
5. The `call` returns `0`; `res` is ignored; the transaction succeeds.
6. `attacker`'s ERC-20 balance is now `0`; no NEP-141 tokens are received on NEAR.
7. Funds are permanently lost.

    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
