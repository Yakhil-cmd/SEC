Audit Report

## Title
Any Caller Can Register SNS Swap Participation for Any Buyer, Bypassing Confirmation Text Requirement - (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` endpoint accepts a caller-supplied `buyer` principal with no authorization check, allowing any unprivileged ingress sender to register swap participation on behalf of any other principal. Combined with the publicly readable `confirmation_text` init parameter, an attacker can bypass the SNS-configured informed-consent requirement and lock a victim's ICP into the swap without the victim ever sending the confirmation text themselves.

## Finding Description
In `rs/sns/swap/canister/canister.rs` (L130–134), when `arg.buyer` is non-empty the canister resolves the target buyer from the request payload rather than from the authenticated caller, with no check that `caller == buyer`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no authorization check
};
``` [1](#0-0) 

This `p` is passed directly to `refresh_buyer_token_e8s`, which calls `validate_confirmation_text` against the SNS-configured text: [2](#0-1) 

`validate_confirmation_text` simply compares the caller-supplied string against `self.init_or_panic().confirmation_text`: [3](#0-2) 

The configured `confirmation_text` is a public init parameter readable by anyone querying the swap canister's state. The `RefreshBuyerTokensRequest.buyer` field is documented as "If not specified, the caller is used," but when it is specified there is no enforcement that the caller equals the buyer: [4](#0-3) 

Once `refresh_buyer_token_e8s` records the buyer's participation, `error_refund_icp` is unavailable until the swap reaches ABORTED or COMMITTED state, and even then it is blocked while the buyer has ICP in escrow: [5](#0-4) 

**Exploit path:**
1. SNS swap is OPEN; `init.confirmation_text = Some("I agree to the terms")`.
2. Victim transfers ICP to `swap_canister_subaccount(victim_principal)` on the ICP ledger, intending to review the terms before confirming.
3. Attacker (any principal) sends an ingress update:
   ```
   RefreshBuyerTokensRequest {
       buyer: "<victim_principal>",
       confirmation_text: Some("I agree to the terms"),
   }
   ```
4. `refresh_buyer_token_e8s` reads the victim's subaccount balance, validates the confirmation text (passes, because the text is correct and public), and records the victim as a participant.
5. Victim's ICP is now locked in escrow. `error_refund_icp` returns a precondition error while the swap is OPEN.

## Impact Explanation
This is a **High** severity finding under "Significant SNS security impact with concrete user or protocol harm." The confirmation text mechanism is the sole informed-consent gate for SNS swap participation. An attacker who bypasses it can commit a victim's ICP to a swap the victim never explicitly agreed to join. If the swap commits, the victim's ICP is swept to SNS governance and the victim receives SNS tokens in exchange — tokens they did not consent to purchase. If those tokens trade below the ICP value at participation time, the victim suffers a direct financial loss. The victim has no recourse during the OPEN lifecycle.

## Likelihood Explanation
Medium-low. The precondition is that the victim has transferred ICP to the swap subaccount but has not yet called `refresh_buyer_tokens` — a realistic window for any user who transfers first and intends to review the confirmation text before confirming. The confirmation text is a public init parameter; no privileged knowledge is required. The victim's principal is observable on-chain. The attack requires a single ingress call and is repeatable across any victim who is in this pending state.

## Recommendation
- Enforce `caller == buyer` inside `refresh_buyer_tokens` when a non-empty `buyer` is supplied, or remove the `buyer` field entirely and always derive the buyer from the authenticated caller.
- If third-party notification is a desired feature (e.g., to allow a frontend canister to notify on behalf of a user), restrict it to an explicit allowlist of trusted callers rather than permitting any principal.

## Proof of Concept
A deterministic unit test against `Swap::refresh_buyer_token_e8s` suffices:

1. Build a `Swap` in `Open` state with `confirmation_text = Some("I agree")`.
2. Mock the ICP ledger to return a non-zero balance for `victim_subaccount`.
3. Call `swap.refresh_buyer_token_e8s(victim_principal, Some("I agree"), swap_canister_id, &ledger)` from a *different* principal (attacker).
4. Assert `Ok(...)` is returned and `swap.buyers` now contains `victim_principal` with the registered amount.
5. Assert that calling `swap.error_refund_icp(...)` while still in `Open` state returns a `TYPE_PRECONDITION` error, confirming the victim's ICP is locked.

This follows the same pattern as the existing `test_swap_participation_confirmation` test in `rs/sns/swap/tests/swap.rs` (L5637–5700), extended to pass a different caller principal than the `buyer` argument. [6](#0-5)

/// Test that the `refresh_buyer_token_e8s` function handles confirmations correctly.
#[test]
fn test_swap_participation_confirmation() {
    let confirmation_text = "Please confirm that 2+2=4".to_string();
    let another_text = "Please confirm that 2+2=5".to_string();
    let user = PrincipalId::new_user_test_id(1);
    let amount = 101 * E8;

    let buy_token = |swap: &mut Swap, confirmation_text: Option<String>| {
        swap.refresh_buyer_token_e8s(
            user,
            confirmation_text,
            SWAP_CANISTER_ID,
            &mock_stub(vec![LedgerExpect::AccountBalance(
                Account {
                    owner: SWAP_CANISTER_ID.get().into(),
                    subaccount: Some(principal_to_subaccount(&user)),
                },
                Ok(Tokens::from_e8s(amount)),
            )]),
        )
        .now_or_never()
        .unwrap()
    };

    // A. SNS specifies confirmation text & client sends confirmation text
    {
        let mut swap = SwapBuilder::new()
            .with_lifecycle(Open)
            .with_confirmation_text(confirmation_text.clone())
            .build();
        // A.1. The texts match
        assert_is_ok!(buy_token(&mut swap, Some(confirmation_text.clone())));
        // A.2. The texts do not match
        assert_is_err!(buy_token(&mut swap, Some(another_text)));
    }

    // B. SNS specifies confirmation text & client does not send a confirmation text
    {
        let mut swap = SwapBuilder::new()
            .with_lifecycle(Open)
            .with_confirmation_text(confirmation_text.clone())
            .build();
        assert_is_err!(buy_token(&mut swap, None));
    }

    // C. SNS does not specify confirmation text & client sends a confirmation text
    {
        let mut swap = SwapBuilder::new()
            .with_lifecycle(Open)
            .without_confirmation_text()
            .build();
        assert_is_err!(buy_token(&mut swap, Some(confirmation_text)));
    }

    // D. SNS does not specify confirmation text & client does not send a confirmation text
    {
        let mut swap = SwapBuilder::new()
            .with_lifecycle(Open)
            .without_confirmation_text()
            .build();
        assert_is_ok!(buy_token(&mut swap, None));
    }
}
