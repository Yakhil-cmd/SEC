### Title
Permanent Loss of ERC-20/ETH Funds When `ExitToNear` Promise Fails Due to Missing Refund Mechanism in Production Builds — (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is not enabled (the default in production), a failed NEP-141 `ft_transfer` or `ft_transfer_call` during an `ExitToNear` precompile call results in the permanent, unrecoverable loss of the user's ERC-20 tokens or ETH. The risk of the external NEP-141 transfer failing is silently transferred onto the exiting user with no on-chain recourse.

---

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) handles bridging of ERC-20 tokens and ETH from Aurora back to NEAR. The flow is:

1. ERC-20 tokens are burned (or ETH is deducted from the user's balance).
2. A NEP-141 `ft_transfer` (or `ft_transfer_call`) promise is scheduled to deliver tokens to the NEAR recipient.
3. Optionally, a callback (`exit_to_near_precompile_callback`) is attached to handle failures.

The critical issue is in how the callback is conditionally constructed:

```rust
// engine-precompiles/src/native.rs, lines 449-455
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is not enabled, `refund` is always `None`. For all exit paths except wNEAR unwrap, `transfer_near` is also `None`. This means `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`, and the promise degrades to a bare `PromiseArgs::Create` with **no callback at all**:

```rust
// engine-precompiles/src/native.rs, lines 470-483
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)  // No callback — no refund on failure
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

The `error_refund` feature is **not** in the default feature set of either `aurora-engine-precompiles` or `aurora-engine`:

```toml
# engine-precompiles/Cargo.toml, line 35-39
[features]
default = ["std"]
...
error_refund = []
``` [3](#0-2) 

```toml
# engine/Cargo.toml, line 42-48
[features]
default = ["std"]
...
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [4](#0-3) 

The `contract` feature (used for production WASM builds) does not enable `error_refund`. Therefore, in all production deployments, the refund path is compiled out entirely.

The callback handler in `engine/src/contract_methods/connector.rs` confirms: when `args.refund` is `None` and the promise fails, the function returns `Ok(None)` — silently discarding the failure with no remediation:

```rust
// engine/src/contract_methods/connector.rs, lines 214-242
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    ...
    None
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None  // <-- No refund, no error, silent loss
};
``` [5](#0-4)

---

### Impact Explanation

**Critical — Permanent loss of user funds.**

When the NEP-141 `ft_transfer` promise fails (for any reason: unregistered recipient account, paused NEP-141 contract, insufficient storage deposit, etc.), the user's ERC-20 tokens have already been burned and their ETH already deducted. Without the `error_refund` feature, there is no on-chain path to recover these assets. The funds are permanently destroyed. This matches the "permanent freezing of funds" impact class.

---

### Likelihood Explanation

**Medium.** Any user who:
- Exits to a NEAR account not registered with the NEP-141 contract (a common mistake),
- Exits to a NEAR account with insufficient storage balance,
- Exits when the NEP-141 contract is paused or rejects the transfer for any reason,

will permanently lose their tokens. This is a realistic, user-triggerable scenario requiring no special privileges. The `ExitToNear` precompile is a core, publicly accessible bridge function. [8](#0-7) 

---

### Recommendation

Enable the `error_refund` feature in production WASM builds by adding it to the `contract` feature in `engine/Cargo.toml`:

```toml
contract = ["log", "error_refund", "aurora-engine-sdk/contract", "aurora-engine-precompiles/contract"]
```

Alternatively, remove the compile-time gate entirely and always include the refund callback. The `refund_call_args` function and `refund_on_error` logic already exist and are correct — the only issue is that they are compiled out in production. [9](#0-8) 

---

### Proof of Concept

1. User holds ERC-20 tokens on Aurora (bridged from a NEP-141 contract).
2. User calls the ERC-20 `withdraw` function specifying an unregistered NEAR account as recipient. This triggers the `ExitToNear` precompile.
3. The precompile burns the ERC-20 tokens and schedules a NEP-141 `ft_transfer` promise to the unregistered account.
4. Because `error_refund` is not enabled in production, `callback_args == default()`, so no callback is attached to the promise.
5. The NEP-141 `ft_transfer` fails (unregistered account).
6. No callback fires. No refund is issued. The ERC-20 tokens are permanently gone.
7. The user has lost their funds with no recourse.

The same scenario applies to ETH exits via `withdrawEthToNear`, as confirmed by `test_exit_to_near_eth_refund`. [10](#0-9) [11](#0-10)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
