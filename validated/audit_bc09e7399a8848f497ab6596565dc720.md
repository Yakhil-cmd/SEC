Audit Report

## Title
Shared `QueryScheduler` Allows Query Flood to Starve Ingress Filter Checks Per Canister — (`rs/execution_environment/src/ingress_filter.rs`)

## Summary
A single `QueryScheduler` instance is shared between the query handler and the ingress filter service. Both push closures into the same per-canister FIFO queue with no capacity bound. An attacker who fills canister A's per-canister queue with 400 max-instruction queries can delay the ingress filter check for canister A beyond the 300-second HTTP request timeout, causing `504 Gateway Timeout` on `/api/v2/call` for that canister and preventing ingress message submission.

## Finding Description

**Shared scheduler confirmed:**

`setup_execution_helper` creates one `QueryScheduler` and clones it to both `IngressFilterServiceImpl` and the query handler. [1](#0-0) 

Both `IngressFilterServiceImpl::call` and `HttpQueryHandler::call` call `self.query_scheduler.push(canister_id, ...)` targeting the same per-canister `incoming: VecDeque<Query>`. [2](#0-1) [3](#0-2) 

**Unbounded per-canister queue:**

`QuerySchedulerCore::push` appends unconditionally to `canister.incoming` with no size limit or back-pressure. [4](#0-3) 

**FIFO drain — ingress filter check queued behind user queries:**

`pop()` drains `leftover` then `incoming` in arrival order. An ingress filter closure pushed after 400 user queries sits behind all of them. [5](#0-4) 

**Concurrency and timing constants:**

- `QUERY_EXECUTION_THREADS_PER_CANISTER = 2` — at most 2 threads drain canister A's queue simultaneously. [6](#0-5) 
- `QUERY_EXECUTION_THREADS_TOTAL = 4`. [7](#0-6) 
- `max_query_concurrent_requests = QUERY_EXECUTION_THREADS_TOTAL * 100 = 400` — the HTTP layer admits up to 400 concurrent query requests, each of which pushes a closure into the scheduler before the HTTP response is sent. [8](#0-7) 
- `request_timeout_seconds = 300` — the global axum timeout layer returns `504` after 300 s, dropping the receiver `rx` for the ingress filter future. [9](#0-8) [10](#0-9) 

**Attack path:**

1. Attacker opens one HTTP/2 connection (within the per-IP nftables limit) and sends 400 concurrent queries targeting canister A's max-instruction query method. Each accepted HTTP request pushes a closure into canister A's `incoming` queue before the HTTP response is returned.
2. Victim submits an ingress message to canister A via `/api/v2/call`. The ingress filter closure is pushed to the back of canister A's `incoming` queue, behind the 400 attacker closures.
3. With `max_threads_per_canister = 2` and ~5 s per max-instruction query, drain time ≈ 400 / 2 × 5 s = **1000 s**.
4. After 300 s the axum timeout fires, drops `rx`, and returns `504 Gateway Timeout` to the victim. When the ingress filter closure is eventually dequeued, `tx.is_closed()` is true and the result is discarded. [11](#0-10) 

No existing guard prevents this: `poll_ready` on `IngressFilterServiceImpl` always returns `Poll::Ready(Ok(()))`, and the `max_call_concurrent_requests = 50` limit only throttles HTTP-layer concurrency for the call endpoint — it does not give ingress filter closures any priority or separate queue inside the scheduler. [12](#0-11) 

## Impact Explanation
An unprivileged attacker can prevent all users from submitting ingress messages to a targeted canister for the duration of the attack. The impact is canister-scoped (other canisters are unaffected) and sustained. Victims receive `504 Gateway Timeout` on `/api/v2/call`. This matches the allowed High impact: **Application/platform-level DoS not based on raw volumetric DDoS** — the attack exploits a specific scheduler design flaw (shared FIFO queue, no priority, no depth bound) rather than bandwidth saturation.

## Likelihood Explanation
The attack requires only a single HTTP/2 connection multiplexing 400 streams (within `http_max_concurrent_streams = 1000`) and a canister with any query method that approaches the instruction limit — a realistic precondition on mainnet. The nftables new-connection rate limit is irrelevant because all 400 streams share one TCP connection. The attacker does not need to sustain new queries after the initial burst; the 400 already-queued closures are sufficient to exceed the 300 s timeout.

## Recommendation
1. **Separate the ingress filter from the query scheduler.** Give `IngressFilterServiceImpl` its own dedicated thread pool so ingress filter checks are never blocked by user queries.
2. **Alternatively, add a priority lane** inside `QuerySchedulerCore` for ingress filter tasks, processed before `incoming` user queries.
3. **Or bound the per-canister queue depth** in `QuerySchedulerCore::push` and reject (or back-pressure) new queries when the queue exceeds a configurable limit, preventing unbounded accumulation.

## Proof of Concept
```
1. Deploy canister A with a query method that loops to the instruction limit.
2. Open one HTTP/2 connection to a replica node.
3. Send 400 concurrent query calls to canister A's slow method (all 400 closures
   are pushed into canister A's incoming queue before any HTTP response is returned).
4. Concurrently submit an ingress message to canister A via /api/v2/call.
5. Observe: /api/v2/call returns 504 Gateway Timeout after 300 s.
6. Confirm via scheduler metrics (execution_query_scheduler_queue_length) that
   the ingress filter closure was enqueued behind the 400 query closures.
```
A deterministic unit test can reproduce this without mainnet: instantiate `QuerySchedulerInternal` with `max_threads_per_canister = 2`, push 400 slow `Query` closures for canister A, push one ingress-filter closure for canister A, run the scheduler on a thread pool, and assert the ingress-filter closure is not executed within 300 s.

    fn push(&mut self, canister_id: CanisterId, query: Query) {
        let canister = self
            .canisters
            .entry(canister_id)
            .or_insert_with(CanisterData::new);
        canister.incoming.push_back(query);

        self.metrics
            .queue_length
            .observe((canister.incoming.len() + canister.leftover.len()) as f64);

        if !canister.has_been_scheduled
            && canister.should_be_scheduled(self.max_threads_per_canister)
        {
            canister.has_been_scheduled = true;
            self.scheduled.push_back(canister_id);
        }

        #[cfg(debug_assertions)]
        self.verify_invariants();
    }

    /// Returns a batch of queries to execute if there are any.
    fn pop(&mut self) -> Option<(CanisterId, Vec<Query>)> {
        let canister_id = self.scheduled.pop_front()?;
        // It is safe to unwrap here because of the invariants in
        // `validate_invariants()`: each canister in the round-robin list must
        // be present in the canister table.
        let canister = self.canisters.get_mut(&canister_id).unwrap();
        debug_assert!(canister.has_been_scheduled);
        canister.has_been_scheduled = false;

        let total = canister.queries_per_time_slice(self.time_slice_per_canister);

        // Collect queries from the `leftover` queue first.
        let from_leftover = total.min(canister.leftover.len());
        let mut result: Vec<_> = canister.leftover.drain(0..from_leftover).collect();

        // Get the remaining queries from the `incoming` queue.
        let from_incoming = (total - from_leftover).min(canister.incoming.len());
        result.extend(canister.incoming.drain(0..from_incoming));

        // Follows from the main invariant.
        debug_assert!(!result.is_empty());

        canister.active_threads += 1;

        // We removed the canister from the schedule at the beginning of this
        // method and cleared `has_been_scheduled`.
        debug_assert!(!canister.has_been_scheduled);
        if canister.should_be_scheduled(self.max_threads_per_canister) {
            canister.has_been_scheduled = true;
            self.scheduled.push_back(canister_id);
        }

        #[cfg(debug_assertions)]
        self.verify_invariants();

        Some((canister_id, result))
    }
