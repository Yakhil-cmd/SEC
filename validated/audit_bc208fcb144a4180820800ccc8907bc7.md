Audit Report

## Title
SNS Governance Voting Rewards Permanently Lost When `total_reward_shares` Is Zero During Proposal Settlement - (File: `rs/sns/governance/src/types.rs`, `rs/sns/governance/src/governance.rs`)

## Summary

In the SNS governance canister, `rewards_rolled_over()` uses `settled_proposals.is_empty()` as a proxy for "were rewards distributed?", but these two conditions are not equivalent. When `total_reward_shares == 0`, proposals are settled but no rewards are distributed, causing the entire `rewards_purse_e8s` — including any accumulated rollover maturity from prior rounds — to be silently and permanently discarded rather than carried forward.

## Finding Description

**Root cause — incorrect rollover predicate:**

`rewards_rolled_over()` in `rs/sns/governance/src/types.rs` (line 2065–2067) checks only whether `settled_proposals` is empty:

```rust
pub(crate) fn rewards_rolled_over(&self) -> bool {
    self.settled_proposals.is_empty()
}
``` [1](#0-0) 

`e8s_equivalent_to_be_rolled_over()` returns the purse only when `rewards_rolled_over()` is true, otherwise returns 0: [2](#0-1) 

**Distribution skip when `total_reward_shares == 0`:**

In `distribute_rewards` (`rs/sns/governance/src/governance.rs`), when all neurons have zero voting power, the distribution loop is bypassed entirely and `distributed_e8s_equivalent` remains 0: [3](#0-2) 

**RewardEvent written with non-empty `settled_proposals`:**

Despite no rewards being distributed, the final `RewardEvent` is written with `settled_proposals: considered_proposals` (non-empty) and `total_available_e8s_equivalent` set to the non-zero purse: [4](#0-3) 

**Exploit chain:**
1. `rewards_purse_e8s` is built from prior rollover + current round supply reward (non-zero).
2. `total_reward_shares == 0` → distribution loop skipped → `distributed_e8s_equivalent = 0`.
3. `RewardEvent` written: `settled_proposals = [P1, ...]` (non-empty), `total_available_e8s_equivalent = R'` (non-zero).
4. Next round: `rewards_rolled_over()` → `settled_proposals.is_empty()` → `false` → `e8s_equivalent_to_be_rolled_over()` → `0`.
5. The entire purse `R'` is gone. No neuron ever receives it; no recovery path exists.

The NNS governance has the identical structural flaw: [5](#0-4) 

## Impact Explanation

Any SNS governance token maturity accumulated in `rewards_purse_e8s` — whether from the current round's supply-proportional reward or from multiple prior rollover rounds — is permanently destroyed the moment a proposal is settled in a round where every neuron has zero voting power. The maturity is never credited to any neuron and is never carried forward. This constitutes a concrete, irreversible loss of SNS governance rewards for token holders, matching the **High** impact class: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation

The trigger condition `total_reward_shares == 0` is reachable without any privileged access:

- Neuron fees accumulate automatically on rejected proposals; any neuron where `neuron_fees_e8s >= cached_neuron_stake_e8s` has zero voting power.
- Any user holding even a depleted neuron can submit a proposal; once the voting period expires the proposal enters `ReadyToSettle`.
- `run_periodic_tasks` calls `distribute_rewards` automatically — no privileged call is needed. [6](#0-5) 

The existing test `zero_total_reward_shares` in `rs/sns/integration_tests/src/neuron.rs` explicitly constructs this exact state (neuron with `cached_neuron_stake_e8s == neuron_fees_e8s == 1_000_000_000`, confirmed `voting_power == 0`): [7](#0-6) 

The scenario is most likely in early-stage SNS deployments where all founding neurons have been fee-depleted, or in any SNS where repeated rejected proposals drain neuron stakes before a reward round closes.

## Recommendation

`rewards_rolled_over()` must reflect whether rewards were **actually distributed**, not merely whether proposals were settled. Two equivalent fixes:

1. **Track distribution explicitly**: Add a boolean field `rewards_were_distributed` to `RewardEvent`; set it to `true` only when `distributed_e8s_equivalent > 0`. Update `rewards_rolled_over()` to use this flag instead of `settled_proposals.is_empty()`.

2. **Force rollover on zero distribution**: In `distribute_rewards`, when `total_reward_shares == 0` and `considered_proposals` is non-empty, write `settled_proposals: vec![]` in the `RewardEvent` (or a dedicated rollover marker) so the existing rollover path is taken in the next round.

The NNS governance `rewards_rolled_over()` in `rs/nns/governance/src/reward/calculation.rs` has the same structural issue and must be reviewed and fixed in parallel. [8](#0-7) 

## Proof of Concept

Extend the existing `zero_total_reward_shares` test in `rs/sns/integration_tests/src/neuron.rs`:

1. Set up a depleted neuron (`cached_neuron_stake_e8s == neuron_fees_e8s`), confirming `voting_power == 0`.
2. Pre-populate `latest_reward_event` with `settled_proposals: vec![]` and `total_available_e8s_equivalent: R > 0` (simulating prior rollover rounds).
3. Add a `ReadyToSettle` proposal to `considered_proposals`.
4. Call `distribute_rewards(supply)`.
5. Assert `latest_reward_event.distributed_e8s_equivalent == 0` (no rewards distributed).
6. Assert `latest_reward_event.settled_proposals` is non-empty.
7. Call `latest_reward_event.e8s_equivalent_to_be_rolled_over()` and assert it returns `0` — proving the purse `R` is lost.
8. Call `distribute_rewards` again for the next round and assert `rewards_purse_e8s` does not include `R`. [9](#0-8)
