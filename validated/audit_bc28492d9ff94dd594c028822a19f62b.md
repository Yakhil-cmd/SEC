### Title
Unchecked Exit-Precompile Return Value in `EvmErc20` Burn-Then-Call Pattern Causes Permanent Fund Freeze — (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first calling `_burn` on the caller's ERC-20 balance and then invoking the Aurora exit precompile via inline assembly. The return value of the assembly `call` opcode is captured in a local variable `res` that is **never checked**. If the precompile call fails (returns 0), the ERC-20 tokens are permanently destroyed while the corresponding NEP-141 tokens remain locked inside Aurora with no redemption path, causing a permanent freeze of the user's bridged funds.

---

### Finding Description

In `etc/eth-contracts/contracts/EvmErc20.sol`, both exit functions follow the same pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — execution continues regardless
    }
}
``` [1](#0-0) 

The same unchecked pattern appears in `withdrawToEthereum`: [2](#0-1) 

And identically in `EvmErc20V2.sol`: [3](#0-2) 

The exit precompile (`ExitToNear`) can legitimately return failure (EVM `call` returns 0) in several conditions enforced inside `engine-precompiles/src/native.rs`:

- `is_static == true` (static-call context)
- `context.address != exit_to_near::ADDRESS.raw()` (delegate-call context)
- `ExitToNearParams::try_from(input)` fails — e.g., the `recipient` bytes do not form a valid NEAR account ID, or the input length is outside the accepted range [4](#0-3) 

Because `_burn` executes **before** the precompile call and the EVM does not revert on a failed sub-`call`, the token destruction is committed while the NEAR-side transfer is never scheduled. The `exit_to_near_precompile_callback` refund path — which re-mints tokens when the downstream NEAR promise fails — is only reachable when the precompile **succeeds** in scheduling a promise: [5](#0-4) 

If the precompile never schedules the promise, no callback fires and no re-mint occurs.

The `receive_erc20_tokens` minting path (the bridge's deposit side) correctly guards against unregistered callers via `get_erc20_from_nep141`, which returns an error for any unregistered NEP-141: [6](#0-5) [7](#0-6) 

But the withdrawal side has no symmetric guard: the burn is unconditional and the precompile result is silently discarded.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the precompile call fails silently:

1. The caller's ERC-20 balance is reduced to zero (tokens destroyed).
2. The NEP-141 tokens that backed those ERC-20 tokens remain held by the Aurora engine contract with no corresponding ERC-20 supply to redeem them.
3. There is no on-chain recovery path: no callback is scheduled, no re-mint occurs, and the NEP-141 tokens are permanently inaccessible to the user.

The accounting invariant `ERC-20 total supply == NEP-141 balance held by Aurora` is broken in the deflationary direction, permanently locking the user's bridged assets.

---

### Likelihood Explanation

**Medium.**

The precompile fails on any input that does not parse as a valid `ExitToNearParams`. The `recipient` argument to `withdrawToNear` is a raw `bytes memory` parameter with no on-chain validation before the burn. A caller supplying:

- An empty byte array
- Bytes that exceed `MAX_INPUT_SIZE`
- Bytes that do not encode a valid NEAR account ID

will silently lose their tokens. This is reachable by any EVM user who holds bridged ERC-20 tokens and calls `withdrawToNear` or `withdrawToEthereum` with malformed input — no special privilege is required.

---

### Recommendation

1. **Check the assembly `call` return value and revert on failure** in both `withdrawToNear` and `withdrawToEthereum` in `EvmErc20.sol` and `EvmErc20V2.sol`:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

2. **Validate `recipient` before burning**: add an on-chain length/character check for the NEAR account ID before calling `_burn`, so invalid inputs are rejected early without any state change.

---

### Proof of Concept

**Entry path (no privilege required):**

1. Attacker holds any amount of a bridged ERC-20 token (e.g., obtained via the normal `ft_transfer_call` → `ft_on_transfer` bridge flow).
2. Attacker calls `withdrawToNear(invalidRecipient, amount)` where `invalidRecipient` is, for example, an empty `bytes` array or a byte sequence that the precompile rejects.
3. `_burn(_msgSender(), amount)` executes — ERC-20 balance decremented, tokens destroyed.
4. The assembly `call` to the `exitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) returns `0` (failure) because `ExitToNearParams::try_from(input)` fails on the malformed recipient.
5. `res = 0` is never inspected; the function returns normally.
6. No NEAR promise is scheduled; `exit_to_near_precompile_callback` is never invoked; no re-mint occurs.
7. The attacker's ERC-20 tokens are permanently destroyed and the backing NEP-141 tokens remain locked in Aurora — permanent fund freeze. [8](#0-7) [9](#0-8) [10](#0-9) [11](#0-10)

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })

pub fn get_erc20_from_nep141<I: IO>(
    io: &I,
    nep141_account_id: &AccountId,
) -> Result<Address, GetErc20FromNep141Error> {
    let key = bytes_to_key(KeyPrefix::Nep141Erc20Map, nep141_account_id.as_bytes());
    io.read_storage(&key)
        .map(|v| {
            let mut buf = [0u8; 20];
            v.copy_to_slice(&mut buf);
            Address::from_array(buf)
        })
        .ok_or(GetErc20FromNep141Error::Nep141NotFound)
}
