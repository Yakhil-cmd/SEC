Audit Report

## Title
Missing Minimum Fee Rate Floor in `DogecoinFeeEstimator::estimate_nth_fee` Can Cause ckDOGE Withdrawal Transactions to Become Permanently Stuck - (File: `rs/dogecoin/ckdoge/minter/src/fees/mod.rs`)

## Summary

`DogecoinFeeEstimator::estimate_nth_fee()` returns the raw fee percentile from the Dogecoin canister with no minimum floor, unlike `BitcoinFeeEstimator::estimate_nth_fee()` which applies `f.max(self.minimum_fee_per_vbyte())` after a documented production incident on 2025-06-21. If the Dogecoin canister returns anomalously low fee percentiles, the ckDOGE minter will build and broadcast a withdrawal transaction with a near-zero fee that is never mined, permanently freezing the user's funds since the ckDOGE burn is irreversible.

## Finding Description

`DogecoinFeeEstimator::estimate_nth_fee()` at line 62 of `rs/dogecoin/ckdoge/minter/src/fees/mod.rs` returns `Some(fee_percentiles[nth])` with no floor:

```rust
Some(fee_percentiles[nth])  // no minimum applied
```

In contrast, `BitcoinFeeEstimator::estimate_nth_fee()` at line 112 of `rs/bitcoin/ckbtc/minter/src/fees/mod.rs` applies:

```rust
median_fee.map(|f| f.max(self.minimum_fee_per_vbyte()))
```

where `minimum_fee_per_vbyte()` returns 1,500 millis/byte for Mainnet. This floor was added in PR #5742 as a stop-gap after the 2025-06-21 incident, documented in `rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md`.

The ckDOGE minter delegates entirely to the shared `ic_ckbtc_minter` infrastructure: `rs/dogecoin/ckdoge/minter/src/main.rs` line 60 calls `ic_ckbtc_minter::timer(DOGECOIN_CANISTER_RUNTIME)`, and `retrieve_doge_with_approval` at line 135 calls `ic_ckbtc_minter::updates::retrieve_btc::retrieve_btc_with_approval`. The `DogeCanisterRuntime` supplies `DogecoinFeeEstimator` as the `Estimator` associated type (line 58 of `rs/dogecoin/ckdoge/minter/src/lib.rs`). The shared timer's `ProcessLogic` task calls `resubmit_transactions` using this estimator, which reads `fee_rate` derived from `estimate_median_fee` → `estimate_nth_fee`. With no floor, a near-zero fee from the Dogecoin canister propagates directly into the signed transaction.

The resubmission path at `rs/bitcoin/ckbtc/minter/src/lib.rs` line 870 computes `fee_rate.max(prev_fee_rate + Fee::MIN_RELAY_FEE_RATE_INCREASE)`. The panic fix from PR #5713 is already present in the shared library, so the secondary panic risk is mitigated. However, the primary impact — a stuck transaction with a near-zero fee — remains fully exploitable independently of the panic.

## Impact Explanation

This is a **High** severity finding matching the allowed impact: *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

A user calling `retrieve_doge_with_approval` has their ckDOGE burned (irreversible ledger operation) and a Dogecoin transaction submitted with a near-zero fee. The transaction is never mined. The user's DOGE is not delivered. Funds remain frozen until a governance-approved canister upgrade is deployed. This is the identical dual impact (user fund loss + withdrawal functionality disruption) as the documented ckBTC incident.

## Likelihood Explanation

The exact same failure mode already occurred in production with ckBTC on 2025-06-21. The root cause — anomalously low fee percentiles returned by the canister — is not Bitcoin-specific and can equally affect the Dogecoin canister integration. Any user calling `retrieve_doge_with_approval` during such a period triggers the stuck-transaction scenario. No special privileges are required; the endpoint is public and accessible to any non-anonymous principal.

## Recommendation

Add a `minimum_fee_per_vbyte()` method to `DogecoinFeeEstimator` in `rs/dogecoin/ckdoge/minter/src/fees/mod.rs` and apply it in `estimate_nth_fee()`, analogous to the fix in `BitcoinFeeEstimator`:

```rust
impl DogecoinFeeEstimator {
    const fn minimum_fee_per_vbyte(&self) -> FeeRate {
        let rate = match &self.network {
            Network::Mainnet => 100_000, // calibrate to Dogecoin fee market
            Network::Regtest => 0,
        };
        FeeRate::from_millis_per_byte(rate)
    }
}

// In estimate_nth_fee, replace:
Some(fee_percentiles[nth])
// with:
Some(fee_percentiles[nth].max(self.minimum_fee_per_vbyte()))
```

The exact floor value should be calibrated to Dogecoin's typical fee market, but any meaningful non-zero floor prevents the near-zero fee scenario.

## Proof of Concept

1. Deploy ckDOGE minter on a local replica with `DogecoinFeeEstimator` wired to a mock Dogecoin canister.
2. Configure the mock to return `vec![0u64; 100]` (all-zero fee percentiles) from `dogecoin_get_current_fee_percentiles`.
3. Call `retrieve_doge_with_approval` with a valid amount and recipient address from an unprivileged principal.
4. Observe: the minter burns ckDOGE from the caller's ledger account, builds a transaction with fee rate 0 millikoinus/byte, and broadcasts it.
5. Observe: the transaction is never mined (zero-fee transactions are rejected by relay nodes).
6. Observe: subsequent timer ticks attempt resubmission but the fee remains at or near `MIN_RELAY_FEE_RATE_INCREASE` (100,000 millikoinus/byte), which may still be below the network relay minimum.
7. Contrast: repeating with `BitcoinFeeEstimator` under the same mock conditions produces a fee of at least `minimum_fee_per_vbyte()` due to the floor applied at line 112 of `rs/bitcoin/ckbtc/minter/src/fees/mod.rs`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)

    // The default dustRelayFee is 3 sat/vB,
    // which translates to a dust threshold of 546 satoshi for P2PKH outputs.
    // The threshold for other types is lower,
    // so we simply use 546 satoshi as the minimum amount per output.
    const DUST_LIMIT: u64 = 546;

    const MIN_RELAY_FEE_RATE_INCREASE: FeeRate = FeeRate::from_millis_per_byte(1_000);

    fn estimate_nth_fee(&self, fee_percentiles: &[FeeRate], nth: usize) -> Option<FeeRate> {
        /// The default fee we use on regtest networks.
