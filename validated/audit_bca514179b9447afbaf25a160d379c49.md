Audit Report

## Title
Unauthenticated Unbounded Correlated SQL Scan via `aggregate_all_subaccounts=true` Causes Rosetta Node DoS — (`rs/rosetta-api/icrc1/src/common/storage/storage_operations/mod.rs`)

## Summary
Any unauthenticated HTTP client can POST to `/account/balance` with `metadata.aggregate_all_subaccounts=true` for a principal owning many subaccounts. This triggers an unbounded correlated SQLite subquery with no LIMIT, no timeout, and no rate-limiting guard. Because `StorageClient` wraps a single `tokio_rusqlite::Connection` background thread, the long-running query serializes and blocks all other DB operations — including block synchronization writes — for its entire duration, rendering the Rosetta node unresponsive.

## Finding Description
The `/account/balance` route is registered with no authentication or rate-limiting middleware: [1](#0-0) 

The handler passes caller-supplied metadata directly to `services::account_balance_with_metadata` with no guards: [2](#0-1) 

The service reads `aggregate_all_subaccounts` from attacker-controlled metadata with no subaccount-count guard, then dispatches to `get_aggregated_balance_for_principal_at_block_idx`: [3](#0-2) 

That function executes a correlated subquery — for every `(principal, subaccount)` pair it fires an inner `MAX(block_idx)` lookup — with no LIMIT and no query timeout: [4](#0-3) 

The schema defines the PK as `(principal, subaccount, block_idx)` and adds only a `block_idx` secondary index. There is no covering index on `(principal)` or `(principal, subaccount)` to make the per-principal aggregation efficient: [5](#0-4) [6](#0-5) 

`StorageClient` uses `tokio_rusqlite::Connection`, which serializes all closures through a single background thread. No `progress_handler`, `busy_timeout`, or interrupt mechanism exists anywhere in the codebase: [7](#0-6) 

## Impact Explanation
While the long-running correlated scan executes on the single `tokio_rusqlite` background thread, every other `call(...)` invocation queues behind it. Block synchronization (`update_account_balances`, `store_blocks`), transaction submission via `/construction/submit`, and all other balance/block queries stall for the full duration of the scan. Repeated requests keep the node permanently degraded. This is a concrete application/platform-level DoS against the ICRC-1 Rosetta API — an in-scope financial integration component — matching the **High ($2,000–$10,000)** impact class: *"Application/platform-level DoS… or subnet availability impact not based on raw volumetric DDoS"* and *"Significant… Rosetta… security impact with concrete user or protocol harm."*

## Likelihood Explanation
The endpoint requires zero authentication. The precondition — a principal with a large number of subaccounts — can be manufactured by an attacker who sends many small transfers to distinct subaccounts of a chosen principal (a one-time setup cost). Once established, a single HTTP POST with `"aggregate_all_subaccounts": true` triggers the full scan. The attack is repeatable indefinitely at negligible per-request cost.

## Recommendation
1. **Add a subaccount-count guard**: before executing the aggregation, count distinct subaccounts for the principal and reject or cap requests above a configurable threshold.
2. **Add a SQLite query timeout** via `rusqlite`'s `progress_handler` to abort runaway queries after a bounded wall-clock duration.
3. **Add per-IP or global rate-limiting** on `/account/balance`.
4. **Rewrite the correlated subquery** as a `GROUP BY` with a `MAX` aggregate and add a composite index on `(principal, subaccount, block_idx DESC)` to make the per-principal scan efficient.

## Proof of Concept
```bash
# 1. Send N transfers to distinct subaccounts of principal P on the ledger.
# 2. Wait for the Rosetta node to sync all blocks.
# 3. Trigger the unbounded scan:
curl -X POST http://<rosetta-host>:8080/account/balance \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"ICRC-1","network":"<ledger-id>"},
    "account_identifier": {"address":"<P-address>"},
    "metadata": {"aggregate_all_subaccounts": true}
  }'
# 4. Observe: response hangs; concurrent /account/balance and
#    /construction/submit requests also hang; block sync stalls.
```

pub fn get_aggregated_balance_for_principal_at_block_idx(
    connection: &Connection,
    principal: &PrincipalId,
    block_idx: u64,
) -> anyhow::Result<Nat> {
    // Query to get the latest balance for each subaccount of the principal at or before the given block index
    let mut stmt = connection.prepare_cached(
        "SELECT a1.subaccount, a1.amount
         FROM account_balances a1
         WHERE a1.principal = :principal
         AND a1.block_idx = (
             SELECT MAX(a2.block_idx)
             FROM account_balances a2
             WHERE a2.principal = a1.principal
             AND a2.subaccount = a1.subaccount
             AND a2.block_idx <= :block_idx
         )",
    )?;

    let rows = stmt.query_map(
        named_params! {
            ":principal": principal.as_slice(),
            ":block_idx": block_idx
        },
        |row| {
            let amount_str: String = row.get(1)?;
            Nat::from_str(&amount_str).map_err(|_| {
                rusqlite::Error::InvalidColumnType(
                    1,
                    "amount".to_string(),
                    rusqlite::types::Type::Text,
                )
            })
        },
    )?;

    let mut total_balance = Nat(BigUint::zero());
    for balance_result in rows {
        let balance = balance_result?;
        total_balance = Nat(total_balance
            .0
            .checked_add(&balance.0)
            .with_context(|| "Overflow while aggregating balances")?);
    }

    Ok(total_balance)
}
