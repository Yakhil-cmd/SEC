### Title
Unbounded Growth of `retrieve_btc_account_to_block_indices` Per Account Enables Cycles-Exhaustion DoS on `retrieve_btc_status_v2_by_account` - (File: rs/bitcoin/ckbtc/minter/src/state.rs)

### Summary

The ckBTC minter canister stores a per-account `Vec<u64>` of all historical `retrieve_btc` block indices in `retrieve_btc_account_to_block_indices`. This list grows without bound as a user submits more withdrawal requests. The query endpoint `retrieve_btc_status_v2_by_account` iterates over the entire list for the requested account, calling `retrieve_btc_status_v2` for each entry. An unprivileged user can accumulate thousands of entries in their own account's list (bounded only by the global `MAX_CONCURRENT_PENDING_REQUESTS = 5000` across all users), and any caller querying that account's status will exhaust the query instruction limit, permanently bricking that query for the targeted account.

### Finding Description

**Root cause — unbounded append with no per-account cap:**

In `rs/bitcoin/ckbtc/minter/src/state/audit.rs`, every accepted `retrieve_btc` request unconditionally appends the new `block_index` to the account's `Vec<u64>`:

```rust
state
    .retrieve_btc_account_to_block_indices
    .entry(account)
    .and_modify(|entry| entry.push(request.block_index))
    .or_insert(vec![request.block_index]);
``` [1](#0-0) 

Entries are **never removed** from this map. The field is declared as `BTreeMap<Account, Vec<u64>>` with no size limit:

```rust
pub retrieve_btc_account_to_block_indices: BTreeMap<Account, Vec<u64>>,
``` [2](#0-1) 

**Vulnerable consumer — full linear scan per query:**

`retrieve_btc_status_v2_by_account` fetches the entire `Vec<u64>` for the target account and maps each entry through `retrieve_btc_status_v2`, which itself performs multiple `BTreeMap` lookups per entry:

```rust
let block_indices: Vec<u64> = self
    .retrieve_btc_account_to_block_indices
    .get(&target_account)
    .unwrap_or(&vec![])
    .to_vec();

let result: Vec<BtcRetrievalStatusV2> = block_indices
    .iter()
    .map(|&block_index| BtcRetrievalStatusV2 {
        block_index,
        status_v2: Some(self.retrieve_btc_status_v2(block_index)),
    })
    .collect();
``` [3](#0-2) 

`retrieve_btc_status_v2` itself performs up to five sequential `BTreeMap::get` lookups per block index: [4](#0-3) 

**Public endpoint — any caller can query any account:**

The Candid interface exposes this as a `query` call accepting an optional `Account` argument, meaning any unprivileged ingress sender can trigger the full iteration over any account's list:

```
retrieve_btc_status_v2_by_account : (opt Account) -> (vec record { ... }) query;
``` [5](#0-4) 

The canister entry point passes the argument directly to the state function with no pagination or size guard:

```rust
#[query]
fn retrieve_btc_status_v2_by_account(target: Option<Account>) -> Vec<BtcRetrievalStatusV2> {
    read_state(|s| s.retrieve_btc_status_v2_by_account(target))
}
``` [6](#0-5) 

**Attacker-controlled entry path:**

1. Attacker controls account `A` and repeatedly calls `retrieve_btc` (or `retrieve_btc_with_approval`) with the minimum withdrawal amount. Each successful call appends one entry to `retrieve_btc_account_to_block_indices[A]`.
2. The global cap `MAX_CONCURRENT_PENDING_REQUESTS = 5000` limits total *pending* requests across all users, but finalized/reimbursed requests remain in the map permanently. The attacker can cycle: submit → wait for finalization → submit again, accumulating entries indefinitely over time.
3. Once the list for account `A` is large enough, any call to `retrieve_btc_status_v2_by_account(Some(A))` — including by the victim themselves — will exceed the IC query instruction limit and be rejected with `CanisterInstructionLimitExceeded`.

### Impact Explanation

- **Permanent DoS of `retrieve_btc_status_v2_by_account` for the targeted account.** Once the list is large enough, no query can ever succeed for that account. The victim loses the ability to programmatically check the status of all their withdrawal requests in a single call.
- **Cycles/resource accounting bug class:** The IC instruction limit for query calls is fixed (~5 billion instructions). With thousands of entries each requiring multiple `BTreeMap` lookups, the limit is reachable.
- The attacker pays only the ckBTC withdrawal fee per request (minimum ~0.00001 BTC per cycle), making this economically feasible over time.
- The minter canister's own state grows unboundedly, consuming stable memory proportional to the number of historical requests per account.

### Likelihood Explanation

- Any unprivileged user with a ckBTC balance can trigger this by repeatedly calling `retrieve_btc` or `retrieve_btc_with_approval` with the minimum amount.
- No admin key, governance majority, or privileged access is required.
- The attack is gradual and can be executed over days/weeks to avoid detection.
- The `MAX_CONCURRENT_PENDING_REQUESTS = 5000` global limit only throttles the rate, not the total accumulation over time. [7](#0-6) 

### Recommendation

1. **Cap the per-account list size.** Enforce a maximum number of entries in `retrieve_btc_account_to_block_indices` per account (e.g., 500). Reject new `retrieve_btc` requests if the account's list would exceed this cap, or evict the oldest finalized entries.
2. **Prune finalized entries.** When a request reaches a terminal state (`Confirmed`, `Reimbursed`, `AmountTooLow`), remove its `block_index` from `retrieve_btc_account_to_block_indices`. This is the direct analog of the Biconomy fix: remove entries once they are no longer needed.
3. **Paginate the query.** Add `start` and `length` parameters to `retrieve_btc_status_v2_by_account` so callers cannot trigger unbounded iteration in a single query call.

### Proof of Concept

```
// Attacker account A with sufficient ckBTC balance:
for i in 0..N:
    retrieve_btc_with_approval(min_amount, attacker_btc_address)
    // wait for finalization (entry stays in retrieve_btc_account_to_block_indices)

// After N iterations, retrieve_btc_account_to_block_indices[A].len() == N
// Now any caller executes:
retrieve_btc_status_v2_by_account(Some(A))
// -> iterates N entries, each calling retrieve_btc_status_v2 with 5 BTreeMap lookups
// -> exceeds query instruction limit -> CanisterInstructionLimitExceeded
```

    pub fn retrieve_btc_status_v2(&self, block_index: u64) -> RetrieveBtcStatusV2 {
        // Hack to avoid a Candid breaking change in `ReimbursementReason`
        // which is in the return type of `retrieve_btc_status_v2`
        fn map_reimbursement_reason(reason: &WithdrawalReimbursementReason) -> ReimbursementReason {
            match reason {
                WithdrawalReimbursementReason::InvalidTransaction(
                    InvalidTransactionError::TooManyInputs { .. },
                ) => ReimbursementReason::CallFailed,
            }
        }

        if let Some(reimbursement) = self.pending_reimbursements.get(&block_index) {
            return RetrieveBtcStatusV2::WillReimburse(reimbursement.clone());
        }

        if let Some(reimbursement) = self.pending_withdrawal_reimbursements.get(&block_index) {
            return RetrieveBtcStatusV2::WillReimburse(ReimburseDepositTask {
                account: reimbursement.account,
                amount: reimbursement.amount,
                reason: map_reimbursement_reason(&reimbursement.reason),
            });
        }

        if let Some(reimbursement) = self.reimbursed_transactions.get(&block_index) {
            return RetrieveBtcStatusV2::Reimbursed(reimbursement.clone());
        }

        if let Some(maybe_reimbursed) = self.reimbursed_withdrawals.get(&block_index) {
            return match maybe_reimbursed {
                Ok(reimbursement) => RetrieveBtcStatusV2::Reimbursed(ReimbursedDeposit {
                    account: reimbursement.account,
                    amount: reimbursement.amount,
                    reason: map_reimbursement_reason(&reimbursement.reason),
                    mint_block_index: reimbursement.mint_block_index,
                }),
                Err(err) => match err {
                    ReimbursedError::Quarantined => RetrieveBtcStatusV2::Unknown,
                },
            };
        }

        let status_v2: RetrieveBtcStatusV2 = self.retrieve_btc_status(block_index).into();

        status_v2
    }
