### Title
Unguarded `new` Initializer Allows Frontrunning to Seize Engine Ownership - (File: `engine/src/contract_methods/admin.rs`)

### Summary

The Aurora Engine's `new` initialization function accepts `owner_id` entirely from caller-supplied input arguments and performs no check on `predecessor_account_id`. Any account that calls `new` before the legitimate deployer can set itself as the engine owner, gaining full administrative control over the contract.

### Finding Description

`admin::new` (lines 56–88) is the sole initialization entry point for the Aurora Engine. Its only guard is a check that state does not already exist:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input)...;
    let state: EngineState = args.into();   // owner_id comes from args
    state::set_state(&mut io, &state)?;
    Ok(())
}
``` [1](#0-0) 

`owner_id` is a field in every version of `NewCallArgs` (`LegacyNewCallArgs`, `NewCallArgsV2`, `NewCallArgsV3`, `NewCallArgsV4`) and is copied verbatim into `EngineState` without any verification against `predecessor_account_id`: [2](#0-1) [3](#0-2) 

The `env` parameter is used only for `block_height()` and `current_account_id()` inside the hashchain path — never to authenticate the caller.

On NEAR, a `DeployContract` action and a subsequent function call can be placed in the same batch transaction (making them atomic), but this is not enforced by the contract. If the deployer issues the `DeployContract` action and the `new` call as separate transactions — as demonstrated in the workspace utility: [4](#0-3) 

— a window exists between the two transactions during which any external account can race to call `new` with an attacker-controlled `owner_id`.

### Impact Explanation

The engine owner has unrestricted access to:

- `stage_upgrade` + `upgrade`: deploy arbitrary new contract bytecode to the engine account, enabling complete fund drainage.
- `attach_full_access_key`: add a full-access key to the engine's NEAR account, giving the attacker direct account control.
- `set_owner`, `pause_contract`, `set_upgrade_delay_blocks`: full administrative takeover. [5](#0-4) [6](#0-5) 

An attacker who wins the race becomes the permanent owner and can drain all ETH and ERC-20 balances held by the engine. **Impact: Critical — direct theft of all user funds.**

### Likelihood Explanation

On NEAR, the attack requires the deployer to issue `DeployContract` and `new` as separate transactions. This is not an exotic scenario: the `engine-workspace` utility does exactly this (two separate `.await` calls). Any operator deploying a new Aurora Engine instance (e.g., for a Silo chain) who follows this pattern is vulnerable. The attacker only needs to monitor the NEAR chain for a `DeployContract` action targeting the engine WASM and immediately submit a `new` call with their own `owner_id`. No special privileges are required.

### Recommendation

Enforce that the caller of `new` is the contract account itself (i.e., that deployment and initialization are atomic):

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // Enforce atomic deploy+init: only the contract itself may call new
    if env.predecessor_account_id() != env.current_account_id() {
        return Err(b"ERR_NOT_ALLOWED".into());
    }
    ...
}
```

This mirrors the XCC router's design, where the comment explicitly states "the Create, Deploy and Initialize actions are done in a single NEAR batch": [7](#0-6) 

Alternatively, derive `owner_id` from `predecessor_account_id` rather than from caller-supplied args, analogous to the fix recommended in the reference report.

### Proof of Concept

1. Attacker monitors the NEAR chain for a `DeployContract` action uploading the Aurora Engine WASM to account `aurora.example.near`.
2. In the next transaction (before the legitimate deployer calls `new`), attacker submits:
   ```
   FunctionCall {
     contract_id: "aurora.example.near",
     method_name: "new",
     args: NewCallArgsV2 { chain_id: ..., owner_id: "attacker.near", upgrade_delay_blocks: 1 },
   }
   ```
3. `state::get_state` returns `Err` (state not yet set) → guard passes.
4. `EngineState { owner_id: "attacker.near", ... }` is written to storage.
5. Legitimate deployer's `new` call now fails with `ERR_ALREADY_INITIALIZED`.
6. Attacker calls `stage_upgrade` with malicious bytecode, waits `upgrade_delay_blocks` (set to 1), calls `upgrade` → malicious contract deployed → all funds drained. [1](#0-0) [8](#0-7)

pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())
}

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
}
