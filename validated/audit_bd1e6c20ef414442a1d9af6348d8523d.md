## Title
Boundary Node Generic Rate Limiter `.max(1)` Guard Silently Multiplies Effective Aggregate Limit When `n < scale` - (File: `rs/boundary_node/ic_boundary/src/rate_limiting/generic.rs`)

## Summary
In `GenericLimiter::process_rules`, the per-BN scaled limit is computed as `(n / scale).max(1)`. When the configured count `n` is less than the number of API boundary nodes `scale`, integer division yields `0` and `.max(1)` silently promotes it to `1`. Each of the `scale` independent BNs then enforces a limit of `1`, making the aggregate effective limit `scale` instead of the intended `n`. An unprivileged attacker who distributes requests across BNs — whose IPs are publicly discoverable from the IC registry — can bypass the configured rate limit by a factor of up to `scale / n`.

## Finding Description
In `rs/boundary_node/ic_boundary/src/rate_limiting/generic.rs` at lines 319–322, `process_rules` scales each rule's limit:

```rust
if let Action::Limit(n, d) = rule.limit {
    // Make sure the limit doesn't go below 1
    let limit = (n / scale).max(1);
    rule.limit = Action::Limit(limit, d);
}
```

`n` is a `u32` from the rule (e.g., `3` in `3/1h`). `scale` is computed at lines 449–451 as `v.api_bns.len().max(1) as u32` — the number of API boundary nodes from the registry snapshot. Rust integer division is floor division. When `n < scale` (e.g., `n=3`, `scale=10`), `n / scale = 0`, and `.max(1)` promotes it to `1`. Each BN independently enforces `1/period`, so the aggregate across all BNs is `scale` instead of `n`.

The `Action::Block` variant exists in `rs/boundary_node/rate_limits/api/src/schema_versions/v1.rs` at line 76 and is correctly handled in `Bucket::evaluate` at lines 148–149 (`Action::Block` → `Decision::Block`). The parser at line 64 rejects `count == 0`, so `Action::Limit(0, d)` cannot be constructed — but `Action::Block` is a valid, fully-supported alternative that would correctly enforce a zero-request limit. The `.max(1)` guard was added to avoid the parser-rejected zero case, but it is the root cause of the bypass.

The autoscale feature is enabled in production via `RATE_LIMIT_GENERIC_AUTOSCALE="true"` in `ic-os/components/guestos/share/ic-boundary.env` line 8. The `Limiter::Single` path at line 154 uses one independent `Ratelimiter` per BN per rule, so each BN's counter is fully isolated — there is no cross-BN coordination.

## Impact Explanation
This is a significant boundary/API security impact with concrete user and protocol harm. Rate limit rules configured with low counts (e.g., `1/1h`, `2/1h`, `3/1h`) to tightly restrict access to specific canisters or subnets are completely defeated. For `n=1` with 20 BNs, the effective aggregate limit becomes `20/period` — a 20× bypass. For `n=3` with 10 BNs, the effective limit is `10/period` instead of `3/period`. This maps to **High ($2,000–$10,000): Significant boundary/API security impact with concrete user or protocol harm**, as the rate limiting mechanism — the primary defense against canister-targeted abuse at the boundary layer — is silently rendered ineffective for the most restrictive rules, which are precisely those intended to protect against abuse.

## Likelihood Explanation
The autoscale feature is confirmed enabled in production. Production deployments have approximately 10–20 API boundary nodes. Rules with low counts relative to BN count are a natural operational choice for tightly restricting specific canisters or subnets. BN IP addresses are publicly discoverable from the IC registry with no authentication required. The attack requires only the ability to send standard HTTP requests to different BN endpoints — no privileged access, key material, governance majority, or special tooling is needed. The bypass is repeatable and persistent until the rule is reconfigured or the code is fixed.

## Recommendation
Replace the `.max(1)` guard with a branch that converts to `Action::Block` when the scaled limit would be zero, preserving the operator's intent:

```rust
if let Action::Limit(n, d) = rule.limit {
    let limit = n / scale;
    rule.limit = if limit == 0 {
        Action::Block
    } else {
        Action::Limit(limit, d)
    };
}
```

Alternatively, if ceiling semantics are preferred (aggregate is at least `n`):

```rust
let limit = n.div_ceil(scale); // always >= 1 when n >= 1
rule.limit = Action::Limit(limit, d);
```

The `Action::Block` branch is the correct fix when the documented intent is "the total ratelimit of all API BNs would be `n`," since a configured `n < scale` means each BN should allow zero requests.

## Proof of Concept
**Setup:** Rule configured as `3/1h` for a specific canister. IC registry reports 10 API BNs → `scale = 10`.

**Execution in `process_rules`:**
```
n = 3, scale = 10
n / scale = 0   (Rust integer division)
(0).max(1) = 1
→ each of 10 BNs enforces limit = 1/1h
```

**Attacker action:** Send 1 request per hour to each of the 10 publicly-known BN IP addresses. Each BN independently passes the request (its counter is at 1/1, within its local limit). Total requests processed: **10/hour** instead of the intended **3/hour**.

**Minimal unit test to reproduce:**
```rust

fn test_autoscale_bypass() {

    let scale: u32 = 10;
    if let Action::Limit(n, d) = rule.limit {
        let limit = (n / scale).max(1);
        assert_eq!(limit, 1); // each BN gets 1, aggregate = 10, not 3
    }
}
```
