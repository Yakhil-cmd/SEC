### Title
Unchecked NEAR Promise Result in `ExitToEthereum` Precompile Causes Permanent Fund Freeze â€” (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile schedules a NEAR cross-contract `withdraw` promise but **never attaches a callback** to verify whether that promise succeeded. If the `withdraw` call fails for any reason, the user's EVM-side tokens are already burned while the NEAR-side withdrawal silently does nothing â€” permanently freezing the funds.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToEthereum::run()` function handles both ETH base-token and ERC-20 exits to Ethereum. After computing the arguments, it unconditionally wraps the outbound call as `PromiseArgs::Create`: [1](#0-0) 

```rust
let withdraw_promise = PromiseCreateArgs { ... method: "withdraw" ... };
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
```

`PromiseArgs::Create` schedules a fire-and-forget cross-contract call. There is no `PromiseArgs::Callback` variant here â€” unlike the `ExitToNear` precompile, which conditionally attaches an `exit_to_near_precompile_callback` when refund or wNEAR-unwrap logic is needed: [2](#0-1) 

The `ExitToEthereum` path has **no equivalent callback path at all**. The EVM execution that burns the user's tokens completes and is committed to state before the NEAR promise is dispatched. If the `withdraw` promise on the ETH connector contract fails (panics, runs out of gas, or is rejected), the engine has no mechanism to detect the failure and refund the user.

The same structural gap exists in the `ExitToNear` precompile for the standard `ft_transfer` legacy flow (non-wNEAR, `error_refund` feature disabled): `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`, so the promise is also emitted as `PromiseArgs::Create` with no callback: [3](#0-2) 

---

### Impact Explanation

**Critical â€” Permanent freezing of funds.**

The EVM state transition (token burn / ETH deduction) is committed atomically within the NEAR transaction that executes the EVM call. The outbound `withdraw` NEAR promise is a *separate* asynchronous receipt. If that receipt fails:

- The user's EVM balance is already zero (burned).
- The ETH connector never processes the withdrawal.
- No refund path exists because no callback was registered.
- The tokens are irrecoverably frozen.

This matches the exact impact class of the reference report: funds are deducted from the sender but never delivered, with no recovery.

---

### Likelihood Explanation

The `withdraw` promise can fail under realistic conditions:

1. **Gas exhaustion**: `costs::WITHDRAWAL_GAS` is a fixed constant. If the ETH connector's `withdraw` implementation changes or the NEAR runtime gas pricing shifts, the attached gas may be insufficient.
2. **Connector contract panic**: Any assertion or invariant violation inside the ETH connector's `withdraw` method causes the receipt to fail silently from the engine's perspective.
3. **Paused connector**: If the connector is paused at the moment the async receipt executes (after the EVM call already committed), the withdrawal fails with no refund.

Any unprivileged user who calls `exitToEthereum` is exposed to this risk on every invocation.

---

### Recommendation

Attach a callback promise to the `withdraw` call in `ExitToEthereum::run()`, mirroring the pattern already used in `ExitToNear` for the wNEAR-unwrap and `error_refund` paths:

```rust
// Instead of:
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();

// Use:
let promise = borsh::to_vec(&PromiseArgs::Callback(PromiseWithCallbackArgs {
    base: withdraw_promise,
    callback: PromiseCreateArgs {
        target_account_id: self.current_account_id.clone(),
        method: "exit_to_ethereum_precompile_callback".to_string(),
        args: borsh::to_vec(&refund_args).unwrap(),
        attached_balance: Yocto::new(0),
        attached_gas: costs::EXIT_TO_ETH_CALLBACK_GAS,
    },
})).unwrap();
```

The callback should inspect `handler.promise_result(0)` and, on failure, re-mint the burned tokens to the original caller â€” exactly as `exit_to_near_precompile_callback` does via `engine::refund_on_error`: [4](#0-3) 

The same fix should be applied to the `ExitToNear` `ft_transfer` legacy path when the `error_refund` feature is disabled.

---

### Proof of Concept

1. User calls an EVM contract that invokes the `exitToEthereum` precompile (address `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`) with `flag = 0x00` and their ETH value.
2. The EVM execution burns the user's ETH balance and emits a `PromiseArgs::Create` log targeting the ETH connector's `withdraw` method. [5](#0-4) 

3. The NEAR runtime dispatches the `withdraw` receipt asynchronously. Suppose the connector panics (e.g., it is paused, or gas is insufficient).
4. The receipt fails. Because `PromiseArgs::Create` carries no callback, the engine receives no notification.
5. The user's EVM ETH balance is zero; the ETH connector never credited the withdrawal. Funds are permanently frozen.

The `Tester.sol` test contract already demonstrates calling this precompile path: [6](#0-5) 

The `res` variable from the assembly `call` is captured but never checked â€” a Solidity-layer echo of the same unchecked-result pattern â€” and the underlying NEAR promise has no callback either.
