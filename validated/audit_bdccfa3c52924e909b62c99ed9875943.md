### Title
Unbounded `factory_deps` Array in L1/Upgrade Transactions Causes Unaccounted Proving Resource Consumption - (File: `basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`)

---

### Summary

The `factory_deps` field in ABI-encoded L1 priority and upgrade transactions (types `0x7f` / `0x7e`) has no length limit enforced by ZKsync OS. The intrinsic native resource pre-charge for L1 transactions accounts only for calldata length, not for `factory_deps` length. Because the transaction hash computation processes the entire transaction encoding — including `factory_deps` — a malicious L1 transaction sender can force the prover to perform unbounded keccak work that is not paid for, with the system explicitly falling back to infinite resources when native resources are exhausted.

---

### Finding Description

**Parsing — no length bound on `factory_deps`:**

In `AbiEncodedTransaction::try_from_buffer`, the `factory_deps` field is parsed as an arbitrary-length `bytes32[]` vector:

```rust
let factory_deps = parser.parse_bytes32_vector()?;
``` [1](#0-0) 

The subsequent `validate_structure()` call checks paymaster, signature, reserved fields, and reserved_dynamic, but contains **no check on `factory_deps` length**: [2](#0-1) 

**Intrinsic native cost ignores `factory_deps`:**

`calculate_l1_tx_intrinsic_computational_native_resources` charges only for calldata bytes:

```rust
pub fn calculate_l1_tx_intrinsic_computational_native_resources(calldata_byte_length: u64) -> u64 {
    let mut intrinsic_computational_native_resources = L1_TX_INTRINSIC_NATIVE_COST;
    intrinsic_computational_native_resources = intrinsic_computational_native_resources
        .saturating_add(
            calldata_byte_length
                .saturating_mul(L1_TX_INTRINSIC_COMPUTATIONAL_NATIVE_PER_CALLDATA_BYTE),
        );
    intrinsic_computational_native_resources
}
``` [3](#0-2) 

It is called with only `transaction.calldata().len()`, not the total transaction size including `factory_deps`: [4](#0-3) 

**Hash computation falls back to infinite resources:**

In `process_l1_transaction`, `calculate_hash` is called against the pre-charged `resources`. If native resources are exhausted (which they will be for a large `factory_deps` array, since the pre-charge did not account for it), the code explicitly re-runs the hash with `S::Resources::FORMAL_INFINITE`:

```rust
let (tx_hash, preparation_out_of_resources): (Bytes32, bool) =
    match transaction.calculate_hash(&mut resources) {
        Ok(h) => (h.into(), false),
        Err(e) => {
            // ...
            resources.exhaust_ergs();
            let mut inf_resources = S::Resources::FORMAL_INFINITE;
            (
                transaction
                    .calculate_hash(&mut inf_resources)
                    .expect("must succeed")
                    .into(),
                true,
            )
        }
    };
``` [5](#0-4) 

The prover must prove this keccak computation regardless of whether it was paid for. The keccak native cost scales with input length (`keccak256_native_cost_u64(len)` = `rounds * KECCAK256_ROUND_NATIVE_COST + base`), so a `factory_deps` array of N entries (each 32 bytes) adds O(N) uncharged keccak rounds to the proof. [6](#0-5) 

**Contrast with L2 transactions:** For L2 (ZK) transactions, the keccak cost is pre-charged over the full transaction length before resources are materialized:

```rust
if !transaction.is_service() {
    charge_keccak(transaction.len(), &mut intrinsic_resources)?;
}
``` [7](#0-6) 

No equivalent charge exists for L1 transactions.

---

### Impact Explanation

An attacker submitting an L1→L2 transaction with a `factory_deps` array of, say, 10 000 entries (320 KB) forces the prover to execute and prove hundreds of additional keccak rounds that are not covered by the transaction's native resource budget. Because the fallback path uses `FORMAL_INFINITE` resources, the system never rejects the transaction at parse time — it always computes the hash. The prover must include this work in the proof. At sufficient scale (many such transactions in one block, or a single extremely large array), this can:

1. Make a block economically unprofitable to prove (prover bears uncompensated RISC-V cycle cost).
2. In the extreme case, push the total proving work beyond circuit capacity, making the block unprovable — a liveness/safety impact on the rollup.

---

### Likelihood Explanation

Any user who can submit an L1→L2 priority transaction (i.e., any Ethereum user interacting with the ZKsync L1 bridge) can craft a transaction with an arbitrarily large `factory_deps` array. No privileged role is required. The L1 bridge contract may impose its own size limits, but ZKsync OS itself imposes none, so the vulnerability is present at the OS layer regardless of bridge-side mitigations.

---

### Recommendation

1. **Enforce a maximum `factory_deps` count** in `AbiEncodedTransaction::validate_structure()`. A reasonable bound is the maximum number of distinct bytecodes that could be deployed in a single transaction (e.g., matching the existing `MAX_BLOBS_PER_BLOCK` pattern used for EIP-4844 blobs).

2. **Include `factory_deps` byte length in the intrinsic native cost** for L1 transactions inside `calculate_l1_tx_intrinsic_computational_native_resources`, analogous to how `L1_TX_INTRINSIC_COMPUTATIONAL_NATIVE_PER_CALLDATA_BYTE` is applied to calldata.

---

### Proof of Concept

1. Craft an L1→L2 transaction (type `0x7f`) with a `factory_deps` array containing 50 000 `bytes32` entries (1.6 MB of hash data).
2. Submit it to the L1 bridge; ZKsync OS parses it without error — `validate_structure()` does not check `factory_deps.len()`.
3. `prepare_and_check_resources` computes intrinsic native cost using only `calldata.len()` — `factory_deps` contributes zero to the pre-charge.
4. `calculate_hash` attempts to keccak-hash the full transaction encoding (≈1.6 MB of `factory_deps` alone), exhausting native resources mid-computation.
5. The fallback branch re-runs `calculate_hash` with `FORMAL_INFINITE` resources, completing successfully.
6. The prover must prove ≈11 800 additional keccak rounds (1.6 MB / 136-byte block size) that were never charged to the transaction sender.
7. Repeating this across multiple transactions in a block can make the block unprovable or economically unviable to prove.
