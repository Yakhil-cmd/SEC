Audit Report

## Title
Existing SNS Swap Participant Incorrectly Blocked from Increasing ICP Participation When Neuron Basket Limit Is Reached - (File: rs/sns/swap/src/swap.rs)

## Summary
In `refresh_buyer_token_e8s()`, the guard enforcing `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` unconditionally adds `+1` to the current participant count before checking whether the caller is already a registered buyer. When the neuron basket limit is exactly met, an existing participant who transfers additional ICP and calls `refresh_buyer_token_e8s` is rejected with the same error as a new entrant, stranding their ICP in the swap subaccount until the swap closes — directly contradicting the error message's own guarantee that "existing participants may still increase their ICP participation amount."

## Finding Description
In `rs/sns/swap/src/swap.rs` at lines 1179–1198, the participant ceiling check is:

```rust
let num_direct_participants = self.buyers.len() as u64;
if (num_direct_participants + 1) * num_sns_neurons_per_basket
    > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS
{
    return Err(format!("... existing participants may still increase their ICP participation amount ..."));
}
```

`self.buyers.len()` already includes the existing buyer. Adding `+1` unconditionally inflates the projected count by one extra slot for callers who are already registered. The `is_preexisting_buyer` check at line 1275 (`self.buyers.contains_key(&buyer.to_string())`) only executes if the function has not already returned. When `num_direct_participants * num_sns_neurons_per_basket == MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` exactly, the expression `(num_direct_participants + 1) * num_sns_neurons_per_basket` exceeds the limit and the function returns an error for all callers — including existing buyers — before ever reaching the preexisting-buyer check.

## Impact Explanation
An existing swap participant who has already registered in `self.buyers` and transfers additional ICP to their swap subaccount cannot call `refresh_buyer_token_e8s` to increase their contribution once the neuron basket limit is exactly reached. Their ICP is stranded in the swap subaccount until the swap closes (committed or aborted), at which point they must use `error_refund_icp` to recover it. This constitutes a significant SNS protocol harm with concrete user impact: the protocol's own documented guarantee (stated in the error message) that existing participants may always top up is violated. This fits the High impact class: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
The condition requires `num_direct_participants * num_sns_neurons_per_basket == MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` exactly. This is reachable in any SNS swap configured with a large `neuron_basket_count` (the existing test uses 33,000 to reach the limit with only 3 participants). Any unprivileged principal can call `refresh_buyer_token_e8s` on the publicly exposed SNS Swap canister. No special role or key is required. The scenario is realistic for high-demand SNS launches where the basket limit is hit.

## Recommendation
Check whether the caller is already in `self.buyers` before deciding whether to add `+1` to the participant count:

```rust
let is_preexisting_buyer = self.buyers.contains_key(&buyer.to_string());
let projected_participants = if is_preexisting_buyer {
    num_direct_participants
} else {
    num_direct_participants + 1
};
if projected_participants * num_sns_neurons_per_basket > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS {
    return Err(/* new-participant-only error */);
}
```

This mirrors the correct pattern already used in SNS governance's `add_neuron_permissions` at `rs/sns/governance/src/governance.rs` lines 4619–4630, where the limit check is conditioned on `existing_permissions.is_none()`.

## Proof of Concept
1. Configure an SNS swap with `neuron_basket_count = 33_000` (as in the existing test at `rs/sns/swap/tests/swap.rs:4992`).
2. Have 3 participants join, reaching `3 * 33_000 = 99_000` neurons (below `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS = 100_000`).
3. Participant 2 (already in `self.buyers`) transfers additional ICP to their swap subaccount and calls `refresh_buyer_token_e8s`.
4. The check evaluates `(3 + 1) * 33_000 = 132_000 > 100_000` → returns error "The swap has reached the maximum number of direct participants … existing participants may still increase their ICP participation amount."
5. Participant 2 is blocked despite being an existing buyer; their newly transferred ICP is stranded in the swap subaccount until the swap closes.
