### Title
Missing Zero-Address Validation on Recipient in `ExitToEthereum` Precompile Enables Permanent Fund Loss — (File: engine-precompiles/src/native.rs)

---

### Summary

The `ExitToEthereum` precompile accepts a caller-supplied Ethereum recipient address for both ETH base-token and ERC-20 withdrawals without checking whether that address is the zero address (`0x000…000`). Any EVM user can trigger a withdrawal that routes funds to the permanently inaccessible zero address, causing irreversible loss.

---

### Finding Description

In `ExitToEthereum::run` the input is validated only for byte-length (21–53 bytes) and flag value. For the ETH base-token path (flag `0x0`), the recipient is parsed directly from the raw input bytes: [1](#0-0) 

For the ERC-20 path (flag `0x1`), the recipient is similarly parsed without any zero-address guard: [2](#0-1) 

In both branches the parsed `recipient_address` is forwarded verbatim into the serialized `withdraw` arguments that are dispatched as a NEAR promise to the ETH connector: [3](#0-2) 

There is no `error_refund` callback wired for `ExitToEthereum` (unlike `ExitToNear`), so a failed or misdirected withdrawal has no on-chain recovery path.

The precompile address is fixed at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`: [4](#0-3) 

---

### Impact Explanation

When a caller supplies `0x0000000000000000000000000000000000000000` as the recipient:

- **ETH path**: the EVM deducts the attached value from the caller's balance, the connector burns the corresponding NEP-141 balance, and the Ethereum-side withdrawal is directed to the zero address — permanently inaccessible because no valid secp256k1 private key maps to it.
- **ERC-20 path**: the ERC-20 contract burns the tokens on the EVM side before calling the precompile; the connector then initiates a withdrawal to the zero address. The tokens are gone from both sides.

In either case the outcome is **permanent freezing (destruction) of user funds** with no recovery mechanism.

---

### Likelihood Explanation

Medium-low. The precompile is callable by any EVM account or contract. Realistic triggering scenarios include:

1. A user-facing ERC-20 wrapper contract that passes a caller-supplied address directly to `ExitToEthereum` without its own zero-address guard.
2. An uninitialized or default-value recipient variable in a calling contract.
3. A deliberate griefing contract that tricks users into approving a withdrawal to the zero address.

The precompile is a production interface used for all ETH/ERC-20 bridge-out operations, so the attack surface is continuously exposed.

---

### Recommendation

Add an explicit zero-address rejection immediately after parsing `recipient_address` in both the `0x0` and `0x1` branches:

```rust
if recipient_address == Address::zero() {
    return Err(ExitError::Other(Cow::from("ERR_ZERO_RECIPIENT_ADDRESS")));
}
```

This mirrors the pattern already used elsewhere in the precompile layer (e.g., `ERR_INVALID_RECIPIENT_ADDRESS` for malformed bytes) and is the direct analog of the `require(_addr != address(0))` guards recommended in the external report.

---

### Proof of Concept

**ETH base-token path:**

1. Attacker (or buggy contract) calls `ExitToEthereum` at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` with calldata `0x00` + `0x0000000000000000000000000000000000000000` and attaches `N` wei.
2. EVM deducts `N` wei from the caller's balance; `context.apparent_value = N`.
3. `ExitToEthereum::run` parses `recipient_address = Address::zero()` — no rejection.
4. A `withdraw` promise is dispatched to the ETH connector with recipient = zero address and amount = `N`.
5. The connector burns the NEP-141 balance and finalises the Ethereum withdrawal to `0x000…000`.
6. `N` wei is permanently inaccessible; no refund callback exists for `ExitToEthereum`.

**ERC-20 path:**

1. User calls an ERC-20 `withdrawTo(address(0), amount)` function that internally burns tokens and calls `ExitToEthereum` with flag `0x1`, `amount`, and `address(0)`.
2. Tokens are burned on the EVM side before the precompile runs.
3. Same zero-address dispatch as above; tokens are permanently lost on both sides.
