Audit Report

## Title
Stale ICP/XDR Conversion Rate Used Without Staleness Check in CMC `tokens_to_cycles` — (File: rs/nns/cmc/src/main.rs)

## Summary
The `tokens_to_cycles` function in the Cycles Minting Canister reads `state.icp_xdr_conversion_rate` and uses it unconditionally without checking the rate's `timestamp_seconds` for freshness. When the exchange rate canister returns errors, the CMC silently retains the last-known rate indefinitely. An unprivileged user can exploit a window where the stored rate is stale and ICP market price has dropped to mint cycles at an inflated XDR/ICP ratio, extracting value from the cycles economy up to the `base_cycles_limit` per hour.

## Finding Description
`tokens_to_cycles` at `rs/nns/cmc/src/main.rs:1899–1923` reads `state.icp_xdr_conversion_rate` and only checks for `Some`/`None` — it never reads `timestamp_seconds`:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp); // timestamp_seconds never checked
        ...
    })
}
```

The stored rate at `rs/nns/cmc/src/main.rs:218` carries a `timestamp_seconds` field that is never consulted at consumption time. The rate is refreshed every `REFRESH_RATE_INTERVAL_SECONDS` (5 minutes, `rs/nns/cmc/src/exchange_rate_canister.rs:16`), but when the exchange rate canister returns an error, `rs/nns/cmc/src/exchange_rate_canister.rs:270–274` returns `UpdateExchangeRateError::FailedToRetrieveRate` and the old rate is silently retained. The `do_set_icp_xdr_conversion_rate` function at `rs/nns/cmc/src/main.rs:1018–1039` only enforces that a new rate's timestamp is strictly greater than the current one — it imposes no freshness requirement at the point of minting. The integration test at `rs/nns/integration_tests/src/cycles_minting_canister_with_exchange_rate_canister.rs:162–185` explicitly asserts that after a `StablecoinRateTooFewRates` error, the CMC retains the previous rate unchanged.

Exploit path:
1. Exchange rate canister begins returning errors (e.g., `StablecoinRateTooFewRates`, network issues, canister trap).
2. CMC retains the last-known rate; its `timestamp_seconds` grows stale.
3. ICP market price drops below the stale rate.
4. Attacker calls `notify_top_up` or `notify_mint_cycles` with ICP.
5. `tokens_to_cycles` uses the stale rate, minting cycles at the inflated XDR/ICP ratio.
6. Attacker repeats up to `base_cycles_limit` per hour across multiple hours.

## Impact Explanation
This constitutes illegal minting of cycles: an unprivileged user receives more cycles per ICP than the current market rate warrants, extracting value from the cycles economy. The `base_cycles_limit` rate limiter (`rs/nns/cmc/src/main.rs:232–233`) caps per-hour damage but does not eliminate it; a sustained exchange rate canister outage allows repeated extraction across hours. This matches the High impact class: "Significant XRC, NNS, or infrastructure security impact with concrete user or protocol harm" — specifically, systematic over-issuance of cycles at below-market cost.

## Likelihood Explanation
The exchange rate canister is a system canister that can legitimately return errors (`StablecoinRateTooFewRates`, insufficient data sources, canister traps). The CMC's own integration test suite exercises this exact failure mode and confirms the stale-rate retention behavior. ICP price is volatile; a sustained outage of minutes to hours combined with a sharp price drop is a realistic, monitorable condition. An attacker watching on-chain metrics can detect when the CMC's published rate diverges from market price and exploit the window without any special privileges — only a valid ICP ledger transaction is required.

## Recommendation
Add a staleness guard in `tokens_to_cycles` (or in `process_top_up`/`process_mint_cycles`) that rejects minting if the stored rate's `timestamp_seconds` is older than a configurable threshold (e.g., 30 minutes):

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let rate = state.icp_xdr_conversion_rate.as_ref()
            .ok_or_else(|| NotifyError::Other {
                error_code: NotifyErrorCode::Internal as u64,
                error_message: "No conversion rate found in CMC".to_string(),
            })?;

        let now = ic_cdk::api::time() / 1_000_000_000;
        let age_seconds = now.saturating_sub(rate.timestamp_seconds);
        if age_seconds > MAX_RATE_AGE_SECONDS {
            return Err(NotifyError::Other {
                error_code: NotifyErrorCode::Internal as u64,
                error_message: format!(
                    "ICP/XDR conversion rate is stale ({age_seconds}s old); minting suspended"
                ),
            });
        }

        Ok(TokensToCycles {
            xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
            cycles_per_xdr: state.cycles_per_xdr,
        }.to_cycles(amount))
    })
}
```

`MAX_RATE_AGE_SECONDS` should be set to a value meaningfully larger than `REFRESH_RATE_INTERVAL_SECONDS` (5 min) to tolerate transient failures, but small enough to bound exposure (e.g., 30 minutes).

## Proof of Concept
1. Deploy a local replica with the CMC and a mock exchange rate canister (as done in `rs/nns/integration_tests/src/cycles_minting_canister_with_exchange_rate_canister.rs`).
2. Set the mock exchange rate canister to return `ExchangeRateError::StablecoinRateTooFewRates`.
3. Advance time by more than `REFRESH_RATE_INTERVAL_SECONDS` and trigger the CMC heartbeat; confirm the stored rate's `timestamp_seconds` is now stale.
4. Call `notify_top_up` with a valid ICP ledger block at the stale (inflated) rate.
5. Observe that cycles are minted at the stale rate — more than the current market rate warrants — confirming the absence of a staleness guard in `tokens_to_cycles`.
