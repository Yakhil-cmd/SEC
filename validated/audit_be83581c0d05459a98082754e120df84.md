### Title
Attacker Can Force Arbitrary Principal into SNS Swap, Consuming Their Per-Participant Quota - (`rs/sns/swap/canister/canister.rs`)

### Summary

The SNS Swap canister's `refresh_buyer_tokens` endpoint accepts an arbitrary `buyer` principal in its request body and performs no check that the caller is the buyer. Any unprivileged ingress sender can call `refresh_buyer_tokens` with a victim's principal as the `buyer` field, causing the swap canister to read the victim's ICP subaccount balance and register (or increase) the victim's participation in the swap — without the victim's consent and potentially without the victim having intentionally sent ICP to the swap subaccount.

### Finding Description

The canister entry point `refresh_buyer_tokens` in `rs/sns/swap/canister/canister.rs` resolves the buyer principal from the request body rather than from the caller identity:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()  // ← attacker-controlled
    };
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, ...)
        .await
``` [1](#0-0) 

Inside `refresh_buyer_token_e8s`, the `buyer` argument is used to look up the ICP balance of `principal_to_subaccount(&buyer)` on the swap canister, and if the balance meets `min_participant_icp_e8s`, the buyer's participation record is created or updated:

```rust
let account = Account {
    owner: this_canister.get().0,
    subaccount: Some(principal_to_subaccount(&buyer)),
};
// ...
self.buyers
    .entry(buyer.to_string())
    .or_insert_with(|| BuyerState::new(0))
    .set_amount_icp_e8s(new_balance_e8s);
``` [2](#0-1) [3](#0-2) 

The `confirmation_text` check is validated against the **caller's** supplied text, not the buyer's consent. An attacker who knows the SNS's `confirmation_text` (which is public, stored in `Init`) can supply it on behalf of the victim:

```rust
self.validate_confirmation_text(confirmation_text)?;
``` [4](#0-3) 

The `confirmation_text` is set at SNS initialization and is publicly readable from the swap state: [5](#0-4) 

The per-participant cap `max_participant_icp_e8s` is enforced, but once a victim's participation is registered at the cap, they cannot increase it further: [6](#0-5) 

### Impact Explanation

**Scenario 1 — Forced participation / quota exhaustion:** A victim transfers ICP to the swap subaccount intending to participate at a specific amount. Before the victim calls `refresh_buyer_tokens` themselves, an attacker calls it first with the victim's principal. The victim's participation is registered at the current subaccount balance (capped at `max_participant_icp_e8s`). If the victim later transfers more ICP to increase their participation, the attacker can call `refresh_buyer_tokens` again to consume the remaining quota. The victim's ICP is locked in the swap with no ability to withdraw until the swap closes.

**Scenario 2 — Forced participation without victim's intent:** If a victim accidentally sends ICP to the swap subaccount (e.g., wrong address), an attacker can immediately call `refresh_buyer_tokens` for the victim, locking the ICP into the swap as a committed participation. The victim loses the ability to reclaim via `error_refund_icp` (which only works for ICP that was never registered as participation).

**Scenario 3 — Swap outcome manipulation:** An attacker can force many principals (who have ICP in their swap subaccounts for any reason) into the swap, artificially inflating `direct_participation_icp_e8s` and potentially triggering early swap commitment (`try_commit`) before the intended participants have had a chance to withdraw.

The impact is: **irreversible locking of victim ICP into an SNS swap without the victim's consent**, and **manipulation of swap finalization timing**.

### Likelihood Explanation

The attack requires only:
1. Knowledge of the victim's principal (public on-chain).
2. The victim having ICP in their swap subaccount (observable on the ICP ledger, which is public).
3. Knowing the `confirmation_text` (readable from swap state via `get_state`).

All three conditions are trivially satisfiable by any unprivileged ingress sender. No privileged access, key material, or social engineering is required. The attack is cheap (only IC ingress fees) and can be executed atomically.

### Recommendation

Enforce that the `buyer` field in `RefreshBuyerTokensRequest` must equal the caller's principal, or remove the `buyer` override entirely and always use `caller_principal_id()`. The legitimate use case of a third party notifying the swap on behalf of a buyer (e.g., a relayer) does not require the third party to be able to register a *different* principal's participation — the buyer can always call the endpoint themselves.

```rust
// Enforce caller == buyer
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = caller_principal_id(); // always use caller
    ...
}
```

### Proof of Concept

1. Victim `V` transfers `min_participant_icp_e8s` ICP to `swap_canister_subaccount(V)` on the ICP ledger, intending to call `refresh_buyer_tokens` themselves.
2. Attacker `A` reads `confirmation_text` from `swap.get_state()` (public query).
3. `A` sends an ingress update call to the swap canister:
   ```
   refresh_buyer_tokens({ buyer: V.to_string(), confirmation_text: Some("...") })
   ```
4. The swap canister reads the ICP balance of `principal_to_subaccount(V)`, finds it ≥ `min_participant_icp_e8s`, and registers `V` as a buyer with that amount.
5. `V`'s ICP is now committed to the swap. `V` cannot reclaim it via `error_refund_icp` because their participation is already registered. `V` must wait for the swap to close (commit or abort) to receive SNS tokens or a refund — without ever having consented to participate.
