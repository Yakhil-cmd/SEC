Audit Report

## Title
SNS Governance `disburse_maturity` Defaults Maturity Disbursement to Caller's Account Instead of Neuron Owner's Account When No Destination Is Specified - (File: `rs/sns/governance/src/governance.rs`)

## Summary
In SNS governance, `disburse_maturity` allows any principal holding `NeuronPermissionType::DisburseMaturity` permission to disburse a neuron's maturity. When the caller omits the optional `to_account` field, the function defaults the destination to the caller's own principal account rather than the neuron owner's account. A delegated caller (non-owner) who holds this permission can therefore redirect 100% of a neuron's accumulated maturity to themselves, permanently deducting it from the victim's neuron after the 7-day disbursement delay.

## Finding Description
`disburse_maturity` in `rs/sns/governance/src/governance.rs` first checks that the caller holds `DisburseMaturity` permission:

```rust
neuron.check_authorized(caller, NeuronPermissionType::DisburseMaturity)?;
```

This permission can be granted to any arbitrary principal by the neuron owner via `AddNeuronPermissions`, and the SNS init payload sets `neuron_grantable_permissions` to all permissions by default, making `DisburseMaturity` freely grantable in standard SNS deployments.

After the authorization check, when no destination account is provided, the code defaults to the caller's account:

```rust
// If no account was provided, transfer to the caller's account.
let to_account: Account = match disburse_maturity.to_account.as_ref() {
    None => Account {
        owner: caller.0,   // ŌåÉ caller, not neuron owner
        subaccount: None,
    },
    ...
};
```

The maturity is then immediately deducted from the neuron and queued for disbursement to that account:

```rust
neuron.maturity_e8s_equivalent = neuron
    .maturity_e8s_equivalent
    .saturating_sub(maturity_to_deduct);
neuron
    .disburse_maturity_in_progress
    .push(disbursement_in_progress);
```

There is no check that the caller is the neuron controller, no check that `to_account` is the neuron owner's account when omitted, and no guard preventing a non-owner delegatee from self-directing the disbursement. By contrast, NNS governance's `initiate_maturity_disbursement` enforces `CallerIsNotNeuronController` and rejects any non-controller caller outright, making the caller-equals-owner invariant hold there. The SNS implementation has no equivalent restriction.

## Impact Explanation
A principal granted `DisburseMaturity` permission (but who is not the neuron owner) can call `manage_neuron` with `DisburseMaturity { percentage_to_disburse: 100, to_account: None }`. After the 7-day delay, the full accumulated maturity of the victim's neuron is minted as SNS tokens and transferred to the attacker's account. The neuron owner loses their governance token rewards with no recourse. This constitutes unauthorized access to and theft of governance assets (SNS token rewards held as neuron maturity), matching the allowed impact: **High ŌĆö Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints.**

## Likelihood Explanation
The attack requires the neuron owner to have previously granted `DisburseMaturity` permission to the attacker. This is a realistic precondition: the SNS permission system is explicitly designed for fine-grained delegation, and neuron owners commonly grant permissions to hot keys or automation services for operational convenience (e.g., automated maturity disbursement to a fixed account). The `DisburseMaturity` permission name and description ("permission to disburse the neuron's maturity to a given ledger account") do not make it obvious that a grantee can self-direct the disbursement by omitting the destination. The SNS init payload grants all permissions as grantable by default, making such delegation common across deployed SNS instances. The attack is repeatable against any neuron whose owner has granted this permission.

## Recommendation
When `to_account` is `None`, default to the neuron controller's account (the neuron owner), not the caller's account. Alternatively, require that a non-controller caller always explicitly specify a `to_account` destination, returning an error if `to_account` is `None` and the caller is not the neuron controller. This mirrors the NNS governance behavior and eliminates the self-redirection vector.

## Proof of Concept
1. **Setup**: Victim `V` creates an SNS neuron with accumulated maturity. `V` grants `DisburseMaturity` permission to attacker principal `A` via `AddNeuronPermissions` (e.g., for legitimate automation).
2. **Attack**: Attacker `A` calls `manage_neuron` on the SNS governance canister:
   ```
   ManageNeuron {
     subaccount: <victim_neuron_subaccount>,
     command: DisburseMaturity {
       percentage_to_disburse: 100,
       to_account: None,
     }
   }
   ```
3. **Execution**: `disburse_maturity` is called with `caller = A`. Authorization passes because `A` holds `DisburseMaturity` permission. Since `to_account` is `None`, `to_account = Account { owner: A.0, subaccount: None }`. Maturity is deducted from the neuron immediately.
4. **Result**: After the 7-day disbursement delay, SNS tokens are minted and transferred to `A`'s account. `V`'s neuron maturity is reduced to zero.

A minimal unit test can be written using the existing `prepare_setup_for_disburse_maturity_tests` fixture: add a second principal with only `DisburseMaturity` permission, call `governance.disburse_maturity(&neuron_id, &attacker_principal, &DisburseMaturity { percentage_to_disburse: 100, to_account: None })`, and assert that `disburse_maturity_in_progress[0].account_to_disburse_to` equals the attacker's account rather than the neuron controller's account.
