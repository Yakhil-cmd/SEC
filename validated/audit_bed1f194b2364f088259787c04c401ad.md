### Title
Fragmented UTXO Set Causes `BuildTxError::AmountTooLow` With No Reimbursement When Withdrawal Is At `fee_based_minimum_withdrawal_amount` — (`rs/dogecoin/ckdoge/minter/src/fees/mod.rs`)

---

### Summary

`fee_based_minimum_withdrawal_amount` computes the minimum withdrawal amount assuming a fixed 2-input/2-output transaction. When the minter's UTXO set is fragmented and the actual transaction requires significantly more inputs, the real `fee + minter_fee` can exceed the withdrawal amount, triggering `BuildTxError::AmountTooLow`. This error path finalizes the request with `FinalizedStatus::AmountTooLow` and **no reimbursement**, permanently burning the user's ckDOGE.

---

### Finding Description

**Step 1 — Minimum amount is computed with a 2-input assumption:** [1](#0-0) 

`PER_REQUEST_SIZE_BOUND = 374` bytes and `PER_REQUEST_MINTER_FEE_BOUND = 326_000_000` koinu correspond exactly to a 2-input/2-output P2PKH transaction. The minter fee for 2 inputs is `146M*2 + 4M*2 + 26M = 326M koinu`, matching the constant precisely.

**Step 2 — Actual transaction building uses the real input count:** [2](#0-1) 

`evaluate_minter_fee` is called with the actual `input_utxos.len()`. For N inputs, the minter fee is `146M*N + 34M` koinu. The network fee is computed from the actual serialized transaction size. The check at line 1306–1307 compares the sum against `amount`.

**Step 3 — `AmountTooLow` finalizes with no reimbursement:** [3](#0-2) 

Contrast with `BuildTxError::InvalidTransaction`, which triggers `reimburse_canceled_requests`: [4](#0-3) 

`AmountTooLow` has no reimbursement path. The ckDOGE burned at request acceptance is permanently lost.

**Step 4 — The ckDOGE minter uses the same `submit_pending_requests` via `ic_ckbtc_minter::timer`:** [5](#0-4) 

**Step 5 — Ingress check only validates against the pre-computed 2-input minimum:** [6](#0-5) 

The `retrieve_btc_with_approval` call checks `amount >= s.fee_based_retrieve_btc_min_amount` (the 2-input minimum). It does not simulate the actual transaction. The ckDOGE burn happens here, before transaction building.

---

### Impact Explanation

**Numerical proof of breakage:**

At low fee rates, `fee_based_minimum_withdrawal_amount` returns 50 DOGE = 5,000,000,000 koinu (with default `retrieve_doge_min_amount = 50 DOGE`).

For N = 35 inputs:
- `minter_fee = 146,000,000 * 35 + 4,000,000 * 2 + 26,000,000 = 5,144,000,000 koinu`
- `5,144,000,000 > 5,000,000,000 = amount` → `BuildTxError::AmountTooLow`

For 35 inputs to be needed to cover 50 DOGE, each UTXO must be < ~1.43 DOGE. This is entirely realistic — any deposit of 1 DOGE or less creates such a UTXO.

The user's ckDOGE is burned at `retrieve_doge_with_approval` time (before transaction building), and the `AmountTooLow` finalization path issues no reimbursement. The user permanently loses their ckDOGE.

---

### Likelihood Explanation

- **UTXO fragmentation** occurs naturally as the minter processes many small deposits. An unprivileged attacker can accelerate this by making many small deposits (each creates a UTXO in the minter's available set).
- The consolidation mechanism (`utxo_consolidation_threshold`) is a mitigation but not a guarantee — consolidation is triggered by a threshold and runs asynchronously.
- The vulnerability requires only ~35 small UTXOs to trigger at the default minimum, which is a low bar.
- Any user who queries `get_minter_info()` to obtain `retrieve_doge_min_amount` and submits exactly that amount is at risk when the UTXO set is fragmented.

---

### Recommendation

1. **In `fee_based_minimum_withdrawal_amount`**: Use a conservative upper bound on the number of inputs (e.g., the 99th-percentile UTXO count from recent transactions, or a configurable `max_expected_inputs` parameter) rather than the fixed value of 2.
2. **In `submit_pending_requests`**: When `BuildTxError::AmountTooLow` occurs, reimburse the user (minus a small processing fee), analogous to the `InvalidTransaction` path. The comment "there is no point in retrying" is correct, but the user should still be reimbursed.
3. **Alternatively**: Before finalizing with `AmountTooLow`, attempt to re-select UTXOs with a higher-value subset to reduce input count, or split the batch.

---

### Proof of Concept

```
1. Deposit 40 UTXOs of 1.25 DOGE each into the minter (total: 50 DOGE in minter's UTXO set).
2. Query get_minter_info() → retrieve_doge_min_amount = 50 DOGE (at low fee rate).
3. Call retrieve_doge_with_approval({ amount: 50_DOGE, address: "..." }).
   → Ingress check passes (50 DOGE >= 50 DOGE minimum).
   → ckDOGE burned from user account.
4. Timer fires → submit_pending_requests → build_unsigned_transaction:
   - utxos_selection(50 DOGE, available_utxos) selects all 40 UTXOs.
   - minter_fee = 146M * 40 + 34M = 5,874M koinu > 5,000M koinu = amount.
   - Returns BuildTxError::AmountTooLow.
5. Request finalized with FinalizedStatus::AmountTooLow, no reimbursement.
6. User has lost 50 DOGE worth of ckDOGE.
```

    fn fee_based_minimum_withdrawal_amount(&self, median_fee_rate: FeeRate) -> u64 {
        match self.network {
            Network::Mainnet => {
                //in Koinu
                const PER_REQUEST_RBF_BOUND: u64 = 374_000;
                // in Bytes
                // Size of a typical transaction made by the minter,
                // which is a P2PKH transaction with 2 inputs and 2 outputs
                const PER_REQUEST_SIZE_BOUND: u64 = 374;
                //in Koinu
                const PER_REQUEST_MINTER_FEE_BOUND: u64 = 326_000;

                let min_withdrawal_amount_increment = self.retrieve_doge_min_amount >> 1;
                ((PER_REQUEST_RBF_BOUND
                    + median_fee_rate.fee_ceil(PER_REQUEST_SIZE_BOUND)
                    + PER_REQUEST_MINTER_FEE_BOUND)
                    / min_withdrawal_amount_increment)
                    * min_withdrawal_amount_increment
                    + self.retrieve_doge_min_amount
            }
            Network::Regtest => self.retrieve_doge_min_amount,
        }
    }
