### Title
Hard deserialization error on `Unspecified` `on_low_wasm_memory_hook_status` breaks backward compatibility during rolling upgrades — (`rs/replicated_state/src/canister_state/system_state/task_queue/proto.rs`)

### Summary

During a rolling upgrade, old replicas write checkpoints with `on_low_wasm_memory_hook_status = 0` (proto default for a missing field). New replicas that load those checkpoints via state-sync hit a hard `Err` in `TaskQueue::try_from`, causing checkpoint deserialization to fail. If checkpoint loading failure is fatal (as is standard in IC state manager), affected new replicas crash-loop while old replicas continue, breaking subnet liveness.

### Finding Description

```rust

### Impact Explanation

A new replica that receives a state-synced checkpoint from an old replica (or loads its own old checkpoint after an upgrade) will fail `CanisterStateBits::try_from` for every canister whose `TaskQueue` was written by old code. Checkpoint loading failure in IC state manager is fatal — the replica cannot participate in consensus. If enough replicas upgrade before the old checkpoints are replaced, the subnet loses liveness.

### Likelihood Explanation

This is triggered automatically by the standard rolling upgrade procedure — no external attacker action is required. Any subnet upgrade that introduces `on_low_wasm_memory_hook_status` into `TaskQueue` while old replicas are still writing checkpoints is affected. The window is the duration of the rolling upgrade.

### Recommendation

Treat `Unspecified = 0` as `ConditionNotSatisfied` (the safe default) instead of returning a hard error:

```rust
pb_canister_state_bits::OnLowWasmMemoryHookStatus::Unspecified => {
    Ok(OnLowWasmMemoryHookStatus::ConditionNotSatisfied)
}
``` [5](#0-4) 

This is the standard IC pattern for backward-compatible proto enum migration. The serialization path already never writes `0`, so this change only affects the backward-compatibility window during upgrades.
