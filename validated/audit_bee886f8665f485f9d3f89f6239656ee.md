Audit Report

## Title
Unbounded Subnet Read-State Cache Exhaustion by Unprivileged Client — (`rs/boundary_node/ic_boundary/src/http/middleware/subnet_read_state_cache.rs`)

## Summary
The `subnet_read_state_cache` on boundary nodes uses a `CacheKey` composed of `(SubnetId_from_URL, ReadStatePaths_from_body)`. The path-label validation in `should_cache_paths` only checks structural shape and byte-length (≤ 29 bytes), not whether the second labels correspond to registered subnet IDs. An unprivileged attacker can send requests to a real subnet's read-state endpoint with counter-incremented arbitrary second labels, generating an unbounded number of unique cache keys. Each request reaches the replica (which returns HTTP 200 with certified absent-proofs), and each response is inserted into the shared 10 MB cache with no per-client quota, evicting legitimate routing-table entries and forcing all subsequent legitimate clients to miss the cache.

## Finding Description

**Cache key construction** — `CacheKey` is `(SubnetId, ReadStatePaths)` where `ReadStatePaths` includes the full path labels from the request body. [1](#0-0) 

**Cacheability gate** — `should_cache_paths` admits a request if it has exactly 2 paths, each with exactly 2 labels, the first labels are `canister_ranges`/`subnet` in either order, and the second labels are `≤ Principal::MAX_LENGTH_IN_BYTES` (29 bytes). The second labels are **not** validated against known subnet IDs — any arbitrary byte sequence up to 29 bytes passes. [2](#0-1) 

**Cache insertion** — On a cache miss, if the upstream response is HTTP 200 and the body is within `max_item_size`, the response is inserted unconditionally with no per-client accounting. [3](#0-2) 

**Middleware ordering** — In `subnet_layers`, `middleware_subnet_read_state_cache` wraps `middleware_subnet_lookup`. On a cache miss, the cache middleware calls `next.run(request)`, which passes through `lookup_subnet` and then to the replica. Because the attacker uses a **real, registered** subnet_id in the URL (e.g., the NNS subnet), `lookup_subnet_by_id` succeeds and the request reaches the replica. [4](#0-3) 

**Replica behavior** — The replica processes read_state requests for arbitrary paths and returns HTTP 200 with certified absent-proofs for paths not present in the state tree. These responses are small (well within the 1 MB `max_item_size`) and are cached. [5](#0-4) 

**URL subnet_id validation** — `validate_subnet_request` only checks that the URL subnet_id is a syntactically valid principal; it does not verify it is a registered subnet. The attacker uses a real subnet_id, so this check passes trivially. [6](#0-5) 

**No per-client rate limiting on subnet routes** — `rate_limit_per_second_per_ip` and `rate_limit_per_second_per_subnet` are applied exclusively to `canister_call_routes` (PATH_CALL_V2/V3/V4). They are not applied to subnet routes. [7](#0-6) 

The bouncer and generic limiter are both opt-in and disabled by default. [8](#0-7) 

**Default cache capacity** — 10 MB total, 1 MB per item, 30 s TTL. [9](#0-8) 

**Exploit flow:**
1. Attacker picks a real, registered subnet_id (e.g., NNS subnet — publicly known).
2. Crafts CBOR bodies with paths `[["canister_ranges", <N>], ["subnet", <N>]]` where `<N>` is a counter-incremented byte string ≤ 29 bytes. Each value of N produces a distinct `CacheKey`.
3. Each request passes `validate_subnet_request` (valid principal), passes `should_cache_paths` (correct structure, ≤ 29 bytes), misses the cache, reaches the replica via `lookup_subnet` (real subnet found), and receives HTTP 200 with certified absent-proofs.
4. Each response is inserted into the cache unconditionally.
5. After ~10,000–50,000 requests (depending on response size), the 10 MB cache is full; moka's weighted eviction begins discarding legitimate entries.
6. Legitimate clients querying routing-table data (`canister_ranges`/`subnet` for real subnet IDs) now miss the cache and must wait for replica round-trips on every request for the next 30 s TTL window.

## Impact Explanation
All boundary-node clients that rely on the subnet read-state cache for routing-table lookups experience elevated latency and increased replica load for the duration of the attack. The attacker must sustain ~10 MB/30 s of unique requests to keep the cache exhausted. The service remains functional (replicas still answer), but the caching invariant — that a single client cannot monopolize the shared cache — is violated. This constitutes an application/platform-level DoS on the boundary node cache layer, matching the **High** impact class: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
The attack requires only a valid subnet principal (publicly known), a CBOR library, and a network connection. No authentication, no privileged role, no key material. The attack is fully local-testable. The only mitigations (bouncer, generic rate limiter) are disabled by default in the code; the production env file shows the generic limiter is configured via a canister, but the specific rules deployed are not guaranteed to cover this attack vector, and the code-level default provides no protection.

## Recommendation
- Add a per-client (per-IP or per-sender) insertion quota or a global insertion-rate cap for the subnet read-state cache.
- Restrict cacheable path labels to byte sequences that parse as **known, registered** subnet IDs (validated against the live routing snapshot), reducing the key space to the finite set of real subnets.
- Apply `rate_limit_per_second_per_ip` to subnet routes in addition to canister call routes, or add a hard rate limit on the subnet read-state endpoint independent of the generic limiter.

## Proof of Concept
```python
import cbor2, requests, struct

SUBNET_URL = "https://<boundary_node>/api/v2/subnet/<nns_subnet_id>/read_state"

for n in range(50_000):
    label = struct.pack(">I", n).ljust(4, b'\x00')  # 4-byte unique label, valid principal length
    body = cbor2.dumps({
        "content": {
            "request_type": "read_state",
            "sender": bytes(1),          # anonymous
            "ingress_expiry": 2**63,
            "paths": [
                [b"canister_ranges", label],
                [b"subnet",          label],
            ]
        }
    })
    requests.post(SUBNET_URL, data=body,
                  headers={"Content-Type": "application/cbor"})
# After ~50k requests the cache is full; measure hit rate for legitimate requests → ~0%
```

A deterministic integration test can be written using the existing `setup_app` test harness in `subnet_read_state_cache.rs` with a small cache size, sending requests with incrementing path labels and asserting that the miss counter grows unboundedly while the cache weighted size stays at capacity.
