Audit Report

## Title
SNS Voting Rewards Permanently Lost When All Neurons Abstain in a Reward Round — (File: `rs/sns/governance/src/governance.rs`)

## Summary
In `distribute_rewards`, when proposals are settled but `total_reward_shares == dec!(0)` (all eligible neurons abstained), the function skips maturity distribution but unconditionally records a `RewardEvent` with the settled proposals. The rollover predicate `rewards_rolled_over()` checks only `settled_proposals.is_empty()`, so it returns `false`, causing `e8s_equivalent_to_be_rolled_over()` to return `0`. The entire rewards purse for that round is permanently discarded rather than carried forward.

## Finding Description

**Step 1 — Purse is computed from rollover + token supply:**
`rewards_purse_e8s` is initialized from `e8s_equivalent_to_be_rolled_over()` and augmented by the configured reward rate applied to the token supply. [1](#0-0) 

**Step 2 — Only neurons that actually voted (Yes/No) accumulate reward shares:**
The loop at lines 5892–5934 skips any ballot where `Vote::eligible_for_rewards()` is false (i.e., `Vote::Unspecified`). If all neurons abstained, `neuron_id_to_reward_shares` is empty and `total_reward_shares == dec!(0)`. [2](#0-1) 

**Step 3 — Zero-share path logs a warning and falls through:**
When `total_reward_shares == dec!(0)`, the code logs an error-level warning and skips the maturity-increment loop, but does **not** return early. [3](#0-2) 

**Step 4 — `RewardEvent` is unconditionally written with settled proposals:**
Execution continues to the end of the function, where `latest_reward_event` is set with `settled_proposals: considered_proposals` (non-empty) and `distributed_e8s_equivalent: 0`. [4](#0-3) 

**Step 5 — Rollover predicate only checks `settled_proposals.is_empty()`:**
`rewards_rolled_over()` returns `true` only when `settled_proposals` is empty. Because proposals were just settled, it returns `false`. [5](#0-4) 

**Step 6 — `e8s_equivalent_to_be_rolled_over()` returns 0:**
Since `rewards_rolled_over()` is `false`, `e8s_equivalent_to_be_rolled_over()` returns `0` instead of `total_available_e8s_equivalent`. [6](#0-5) 

**Step 7 — Next round starts with zero rollover:**
In the following reward round, `rewards_purse_e8s` is initialized from `e8s_equivalent_to_be_rolled_over()` which is `0`, so the prior round's purse is permanently gone. [7](#0-6) 

**NNS parallel:** The same structural flaw exists in NNS. When `total_voting_rights < 0.001`, `reward_distribution` is set to `None` but the `RewardEvent` is still emitted with `settled_proposals` populated. [8](#0-7) 

The NNS rollover predicate is identical: [9](#0-8) 

The proposal-creation guard at line 5287–5292 prevents proposals from being created when no eligible voters exist, but it does **not** prevent all eligible voters from abstaining after creation. [10](#0-9) 

## Impact Explanation

For any SNS reward round where proposals exist but all eligible neurons abstain, the entire `rewards_purse_e8s` — a non-zero amount derived from the token supply and the configured reward rate — is silently and permanently discarded. The emitted `RewardEvent` records `total_available_e8s_equivalent > 0` alongside `distributed_e8s_equivalent == 0`, but the difference is never rolled over. Neuron holders permanently lose the maturity they were entitled to for that round. This constitutes a concrete, permanent loss of governance rewards for SNS token holders, matching the **High** impact class: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation

A proposal can only be created when eligible voters exist (the `electoral_roll.is_empty()` guard). However, eligible voters are never forced to cast a vote; a ballot with `Vote::Unspecified` is valid. If every neuron in an SNS abstains on every proposal within a reward round, `total_reward_shares` is exactly `0`. This is reachable without any privileged access: a low-participation SNS, a contentious proposal that all holders choose to ignore, or a coordinated abstention by a majority of stake are all realistic triggers. No special permissions are required — the attacker path is ordinary inaction (or a `manage_neuron` call that does not cast a vote) by any set of neuron holders collectively holding all voting power.

## Recommendation

Change `rewards_rolled_over()` in `rs/sns/governance/src/types.rs` to also treat a round where nothing was distributed as a rollover:

```rust
pub(crate) fn rewards_rolled_over(&self) -> bool {
    self.settled_proposals.is_empty()
        || self.distributed_e8s_equivalent == 0
}
```

Alternatively, `distribute_rewards` should skip settling proposals and skip advancing `end_timestamp_seconds` when `total_reward_shares == dec!(0)`, treating the round identically to a round with no proposals. The same fix should be applied to the NNS `calculate_voting_rewards` path when `total_voting_rights < 0.001`.

## Proof of Concept

1. Deploy an SNS with `voting_rewards_parameters` configured (non-zero reward rate).
2. Create a proposal. At creation time, eligible neurons exist, so the proposal is accepted.
3. Let the entire voting period elapse without any neuron casting a vote (all abstain, `Vote::Unspecified`).
4. Trigger `distribute_rewards` via `run_periodic_tasks`.
5. Observe the emitted `RewardEvent`:
   - `settled_proposals` is non-empty (contains the proposal ID).
   - `total_available_e8s_equivalent > 0` (reward purse was computed from token supply).
   - `distributed_e8s_equivalent == 0` (nothing distributed).
6. Advance time by one more reward round and trigger `distribute_rewards` again.
7. Observe that `rewards_purse_e8s` starts from `0` rollover — the previous purse is gone.
8. No neuron ever receives the maturity that should have been distributed in step 4.

This is reproducible as a deterministic PocketIC integration test by controlling the clock and submitting no `manage_neuron` vote calls during the voting window.
