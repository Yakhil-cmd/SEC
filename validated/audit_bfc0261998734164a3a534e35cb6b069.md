### Title
Permanent Fund Loss in `exit_to_near` Precompile When NEAR-Side Transfer Fails Without `error_refund` Feature - (File: engine-precompiles/src/native.rs)

### Summary
The `exit_to_near` precompile burns EVM-side tokens and schedules an asynchronous NEAR-side `ft_transfer` promise. When the `error_refund` feature is not compiled in, the `refund` field in `ExitToNearPrecompileCallbackArgs` is hardcoded to `None`, and no callback is attached to the promise. If the NEAR-side transfer fails for any reason, the EVM tokens are permanently destroyed with no recovery path — an exact bridge-accounting analog to the FCHAIN "locked-but-not-transferred" state divergence.

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear::run` function constructs the callback args as follows:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None when feature is absent
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `transfer_near` is also `None` (all non-wNEAR ERC-20 and plain ETH exits), `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`, so the branch taken is:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached at all
} else { ... }
``` [2](#0-1) 

No callback means no failure handler. In `exit_to_near_precompile_callback`, the only refund path is:

```rust
} else if let Some(args) = args.refund {
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent no-op when refund is None and promise failed
}
``` [3](#0-2) 

For an **ETH exit**, the `apparent_value` ETH is deducted from the caller's EVM balance by the EVM executor before the precompile returns. For an **ERC-20 exit**, the ERC-20 contract burns the tokens and then calls the precompile. In both cases the EVM-side debit is committed before the asynchronous NEAR promise executes. If that promise fails (recipient not storage-registered for the NEP-141 token, NEP-141 contract paused, insufficient attached yoctoNEAR, etc.) and no callback is present, the tokens are gone from the EVM state while Aurora's NEP-141 balance retains them — a permanent accounting divergence identical in structure to the FCHAIN `tokenLockedBy` bug: internal state says "exited/burned," actual custody says "still held by Aurora."

The test suite explicitly acknowledges this split behavior:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [4](#0-3) 

### Impact Explanation

**Critical — Permanent freezing of funds.** Any user who calls `exit_to_near` when the NEAR-side `ft_transfer` is destined to fail loses their tokens with no recovery path. The EVM balance is debited and the NEP-141 tokens remain locked inside the Aurora contract forever, satisfying the "permanent freezing of funds" criterion.

### Likelihood Explanation

**Medium.** The failure condition is reachable by any unprivileged EVM user without admin access. Common real-world triggers include: recipient NEAR account not registered for the specific NEP-141 token (requires a `storage_deposit` call that many users omit), the NEP-141 contract being paused, or the recipient account not existing. The `exit_to_near` precompile is a primary user-facing bridge exit path, so the exposure surface is large. The vulnerability is conditional on the `error_refund` feature not being compiled into the production binary; the CHANGES.md entry "Improve security of the `exit_to_near` precompile" (v3.9.1) and "Rolled back the backward compatibility of the `exit_to_near` precompile" (v3.10.0) indicate this path has been actively revisited, suggesting the feature flag state in production is non-trivial. [5](#0-4) 

### Recommendation

Remove the `#[cfg(feature = "error_refund")]` conditional compilation guard and make the refund mechanism unconditional. The `refund_call_args` helper already exists and is correct; gating it behind a feature flag creates a silent fund-loss path in any build that omits the feature. Alternatively, enforce at the build level that `error_refund` is always present for any production artifact.

### Proof of Concept

1. User holds 1000 USDC (ERC-20) on Aurora.
2. User calls `exit_to_near` precompile specifying a recipient NEAR account that has never called `storage_deposit` on the USDC NEP-141 contract.
3. The ERC-20 contract burns 1000 USDC from the user's EVM balance and invokes the precompile.
4. The precompile constructs `ExitToNearPrecompileCallbackArgs { refund: None, transfer_near: None }` (without `error_refund`), which equals `default()`, so `PromiseArgs::Create(ft_transfer_promise)` is returned with **no callback**.
5. The NEAR runtime executes `ft_transfer` on the USDC NEP-141 contract; it panics because the recipient has no storage registration.
6. There is no attached callback to observe the failure and re-mint the ERC-20 tokens.
7. The user's 1000 USDC is permanently destroyed: gone from EVM state, unreachable in Aurora's NEP-141 balance.
