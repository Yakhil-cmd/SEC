Audit Report

## Title
Hot Keys Not Cleared When NNS Neuron Is Spawned to a Different Controller - (File: `rs/nns/governance/src/governance.rs`)

## Summary
`spawn_neuron` accepts an optional `new_controller` field that allows spawning a child neuron to an entirely different principal. Despite this, the child neuron is unconditionally built with `.with_hot_keys(parent_neuron.hot_keys.clone())`, meaning the spawning party's hot keys are silently transferred to a neuron now owned by a different controller. The new controller has no notification and must proactively discover and remove the inherited hot keys.

## Finding Description
In `spawn_neuron` (L2613), the `child_controller` is resolved at L2651–2655: if `spawn.new_controller` is `Some(p)`, then `child_controller = p`, which may differ from `parent_neuron.controller()`. The child neuron is then constructed at L2704–2718:

```rust
let child_neuron = NeuronBuilder::new(
    child_nid,
    to_subaccount,
    child_controller,          // ← potentially a completely different principal
    ...
)
.with_hot_keys(parent_neuron.hot_keys.clone())  // ← parent's hot keys copied unconditionally
``` [1](#0-0) 

There is no conditional guard comparing `child_controller` to `parent_neuron.controller()` before copying hot keys. The grep search confirms no such guard exists anywhere in `governance.rs`.

`is_authorized_to_vote` in `neuron/types.rs` (L243–244) grants hot keys the same voting and followee-setting rights as the controller:

```rust
pub(crate) fn is_authorized_to_vote(&self, principal: &PrincipalId) -> bool {
    self.is_hotkey_or_controller(principal)
}
``` [2](#0-1) 

Once the spawned neuron finishes its dissolve/spawn cycle and acquires stake, the inherited hot keys can immediately vote on governance proposals, change followees, and perform all non-privileged neuron operations on behalf of the new controller without their knowledge or consent.

## Impact Explanation
This is **High** severity: "Unauthorized access to neurons, governance assets … where exploitation requires meaningful per-target work or other constraints." The attacker (Alice) can unilaterally redirect the voting power of a neuron she transferred to a victim (Bob), manipulating NNS governance outcomes. Alice can also alter Bob's followees, corrupting his liquid-democracy delegation. Bob has no in-protocol notification; he must proactively call `get_full_neuron` and issue `RemoveHotKey` before Alice acts — a race condition.

## Likelihood Explanation
The attack path requires no special privileges: any principal holding a neuron with accumulated maturity and at least one registered hot key can execute it. The `manage_neuron` → `Spawn` endpoint is a standard, publicly documented NNS operation reachable by any unprivileged ingress sender. No social engineering, key compromise, or governance majority is needed. [3](#0-2) 

## Recommendation
When `spawn.new_controller` resolves to a principal different from `parent_neuron.controller()`, initialize the child neuron with an empty hot keys list:

```rust
let inherited_hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};
// ...
.with_hot_keys(inherited_hot_keys)
```

The same review should be applied to `split_neuron` (L2241–2257), though there the child always shares the parent's controller so the risk is lower. [4](#0-3) 

## Proof of Concept
1. Alice controls neuron `N` with hot key `H` and sufficient maturity.
2. Alice calls `manage_neuron` with `Command::Spawn(Spawn { new_controller: Some(Bob), nonce: Some(42), percentage_to_spawn: Some(100) })`.
3. Governance creates child neuron `C` with `controller = Bob` and `hot_keys = [H]` (confirmed at L2714).
4. After the spawn dissolve delay, `C` acquires stake and voting power.
5. `H` calls `manage_neuron` on `C` with `Command::Follow(...)` or `Command::RegisterVote(...)`, exercising Bob's voting power on any open proposal.
6. Bob, unaware of `H`, cannot prevent this without first querying `get_full_neuron` and issuing `RemoveHotKey` — a race condition if `H` acts first.

A deterministic integration test using PocketIC can verify this by: spawning a neuron to a different controller, asserting `hot_keys` on the child equals the parent's hot keys, then calling `manage_neuron` with the hot key principal and confirming the vote/follow succeeds.
