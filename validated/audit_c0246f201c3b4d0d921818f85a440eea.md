### Title
`new_settlement_layer_chain_id_storage` Excluded from IO Frame Rollback, Persisting Reverted State into Batch Public Input — (`basic_system/src/system_implementation/system/io_subsystem.rs`)

---

### Summary

`FullIO`'s `new_settlement_layer_chain_id_storage` field is never snapshotted in `start_io_frame` and never rolled back in `finish_io_frame`. If a transaction that triggers the `SettlementLayerChainIdUpdated` event hook subsequently reverts (e.g., out-of-gas, explicit revert), the settlement-layer chain-ID update survives the revert and is committed into the batch public input — while the corresponding storage-slot write is correctly rolled back. This creates a provable divergence between on-chain state and the batch PI.

---

### Finding Description

`FullIO` owns six rollbackable sub-storages. Five of them are correctly snapshotted and restored on frame revert: [1](#0-0) 

```
start_io_frame:
  io        ← storage.start_frame()
  transient ← transient_storage.start_frame()
  messages  ← logs_storage.start_frame()
  events    ← events_storage.start_frame()
  interop_roots ← interop_root_storage.start_frame()
  // new_settlement_layer_chain_id_storage  ← MISSING
``` [2](#0-1) 

The sixth sub-storage, `new_settlement_layer_chain_id_storage`, is declared in `FullIO`: [3](#0-2) 

`NewSettlementLayerChainIdStorage` already exposes `start_frame` / `finish_frame` — they simply are never wired into the IO frame lifecycle: [4](#0-3) 

The update path is: EVM contract emits `SettlementLayerChainIdUpdated(uint256)` → `system_context_event_hook` intercepts it → calls `system.io.update_settlement_layer_chain_id(...)`: [5](#0-4) [6](#0-5) 

Because `new_settlement_layer_chain_id_storage` is never snapshotted, a revert of the enclosing frame (triggered by out-of-gas, explicit `REVERT`, or block-limit overflow) leaves the chain-ID update intact. The corresponding storage-slot write to the system-context contract IS rolled back (it lives inside `self.storage`), so the two representations diverge.

The diverged value is later consumed by `ZKBatchDataKeeper` when building the batch public input: [7](#0-6) 

The `settlement_layer_chain_id` field in the batch data keeper is populated from `new_settlement_layer_chain_id_storage` at block finalization, so the PI carries the stale (reverted) chain-ID value.

---

### Impact Explanation

The batch public input encodes the settlement-layer chain ID. If a reverted transaction leaves a stale value in `new_settlement_layer_chain_id_storage`, the PI diverges from the actual committed state. This causes:

- **Proof invalidity / batch rejection**: the prover commits to a chain ID that does not match the state root, making the proof unprovable or causing verifier rejection on L1.
- **Forward/proving divergence**: the forward (sequencer) run and the proving run would produce different batch PIs for the same block, violating the core ZKsync OS invariant.

Impact is **high** — it can halt batch finalization for the affected block.

---

### Likelihood Explanation

The `SettlementLayerChainIdUpdated` event is emitted by the system-context contract, which is currently only called via privileged service transactions. However:

1. The `system_context_event_hook` fires on **any** event emitted from `SYSTEM_CONTEXT_ADDRESS` with the matching topic — there is no additional caller check inside the hook.
2. If the system-context contract's access control is bypassed or if a future upgrade relaxes it, an unprivileged caller could trigger the path.
3. Even under current constraints, a legitimate service transaction that reverts (e.g., due to a gas miscalculation or block-limit overflow mid-execution) silently corrupts the batch PI — no attacker required.

Likelihood is **low-to-medium** given current access controls, but the missing rollback is an unconditional code defect.

---

### Recommendation

Wire `new_settlement_layer_chain_id_storage` into the IO frame lifecycle, mirroring the pattern used for `interop_root_storage`:

```rust
// in FullIOStateSnapshot
pub struct FullIOStateSnapshot<M: StorageModel> {
    io: M::StateSnapshot,
    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,
+   new_sl_chain_id: NewSettlementLayerChainIdSnapshotId,
}

// in start_io_frame
+   let new_sl_chain_id = self.new_settlement_layer_chain_id_storage.start_frame();
    Ok(FullIOStateSnapshot { io, transient, messages, events, interop_roots, new_sl_chain_id })

// in finish_io_frame
+   self.new_settlement_layer_chain_id_storage
+       .finish_frame(rollback_handle.map(|x| Some(x.new_sl_chain_id)));
```

---

### Proof of Concept

1. Deploy a contract at `SYSTEM_CONTEXT_ADDRESS` (or use the existing one) that emits `SettlementLayerChainIdUpdated(newId)` and then reverts.
2. Submit a service transaction calling that contract.
3. Observe that after the block is processed:
   - The storage slot for the chain ID in the system-context contract is **unchanged** (correctly rolled back).
   - `new_settlement_layer_chain_id_storage.value()` returns `Some(newId)` (incorrectly not rolled back).
4. The batch PI generated at block finalization encodes `newId`, diverging from the actual state.

The structural proof is in the code: `FullIOStateSnapshot` has no `new_sl_chain_id` field [8](#0-7) , `start_io_frame` never calls `new_settlement_layer_chain_id_storage.start_frame()` [1](#0-0) , and `finish_io_frame` never calls `new_settlement_layer_chain_id_storage.finish_frame()` [2](#0-1) , while the storage itself has full rollback support that goes unused [4](#0-3) .
