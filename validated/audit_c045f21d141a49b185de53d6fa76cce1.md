### Title
`new_settlement_layer_chain_id_storage` Excluded from Frame Rollback, Stale State Persists After Revert - (`basic_system/src/system_implementation/system/io_subsystem.rs`)

---

### Summary

`FullIO::start_io_frame()` and `finish_io_frame()` snapshot and roll back every stateful sub-component (`storage`, `transient_storage`, `logs_storage`, `events_storage`, `interop_root_storage`) **except** `new_settlement_layer_chain_id_storage`. If a transaction that triggers `update_settlement_layer_chain_id` is later rolled back at the frame level (e.g., due to block-limit overflow), the settlement-layer chain-ID update persists in `new_settlement_layer_chain_id_storage` while the corresponding storage write is reverted. At block finalization, `read_batch_context_inputs` asserts the two values are equal and panics, causing a block-finalization DoS.

---

### Finding Description

**Root cause — missing snapshot/rollback wiring**

`FullIOStateSnapshot` holds rollback handles for every IO sub-component except `new_settlement_layer_chain_id_storage`:

```rust
pub struct FullIOStateSnapshot<M: StorageModel> {
    io: M::StateSnapshot,
    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,
    // ← new_settlement_layer_chain_id is absent
}
``` [1](#0-0) 

`start_io_frame` takes a snapshot for every component except `new_settlement_layer_chain_id_storage`:

```rust
fn start_io_frame(&mut self) -> Result<Self::StateSnapshot, InternalError> {
    let io       = self.storage.start_frame();
    let transient = self.transient_storage.start_frame();
    let messages  = self.logs_storage.start_frame();
    let events    = self.events_storage.start_frame();
    let interop_roots = self.interop_root_storage.start_frame();
    // ← self.new_settlement_layer_chain_id_storage.start_frame() never called
    Ok(FullIOStateSnapshot { io, transient, messages, events, interop_roots })
}
``` [2](#0-1) 

`finish_io_frame` rolls back every component except `new_settlement_layer_chain_id_storage`:

```rust
fn finish_io_frame(&mut self, rollback_handle: Option<&Self::StateSnapshot>) -> Result<(), InternalError> {
    self.storage.finish_frame(rollback_handle.map(|x| &x.io))?;
    self.transient_storage.finish_frame(rollback_handle.map(|x| &x.transient))?;
    self.logs_storage.finish_frame(rollback_handle.map(|x| x.messages));
    self.events_storage.finish_frame(rollback_handle.map(|x| x.events));
    self.interop_root_storage.finish_frame(rollback_handle.map(|x| x.interop_roots));
    // ← self.new_settlement_layer_chain_id_storage.finish_frame() never called
    Ok(())
}
``` [3](#0-2) 

Note that `interop_root_storage` — a structurally identical block-level accumulator — **is** correctly included in the snapshot. `new_settlement_layer_chain_id_storage` is the sole omission. [4](#0-3) 

**How the stale value is written**

The `SettlementLayerChainIdUpdated` event hook calls `update_settlement_layer_chain_id` during transaction execution (inside the frame): [5](#0-4) 

`update_settlement_layer_chain_id` writes to `new_settlement_layer_chain_id_storage`: [6](#0-5) 

`NewSettlementLayerChainIdStorage::update` enforces a once-per-block invariant by checking `self.value().is_some()`: [7](#0-6) 

**Where the panic fires**

At block finalization, `read_batch_context_inputs` asserts the stale in-memory value equals the value actually committed to storage:

```rust
if let Some(new_settlement_layer_chain_id) = io.new_settlement_layer_chain_id_storage.value() {
    assert_eq!(new_settlement_layer_chain_id, &settlement_layer_chain_id);
}
``` [8](#0-7) 

If the transaction was rolled back, `settlement_layer_chain_id` (read from storage) holds the old value while `new_settlement_layer_chain_id_storage` holds the new value → `assert_eq!` panics.

**Exploit path**

1. A service transaction calls `setSettlementLayerChainId(new_id)` on the SystemContext contract.
2. During execution the contract emits `SettlementLayerChainIdUpdated(new_id)`. The event hook fires and writes `new_id` into `new_settlement_layer_chain_id_storage`.
3. The transaction is rolled back at the frame level — e.g., because it exceeds block gas/pubdata limits (handled in the tx loop): [9](#0-8) 
4. `finish_global_frame(Some(&pre_tx_rollback_handle))` rolls back `events_storage` (so the event disappears) and `storage` (so the SystemContext slot reverts), but **not** `new_settlement_layer_chain_id_storage`.
5. Block finalization calls `read_batch_context_inputs`. `settlement_layer_chain_id` from storage = old value; `new_settlement_layer_chain_id_storage.value()` = `Some(new_id)`. The `assert_eq!` panics → block finalization fails in both forward and proving modes.

Additionally, the stale `Some(new_id)` causes `update()` to return an internal error for any subsequent legitimate update attempt in the same block, permanently blocking the SL chain-ID migration path for that block.

---

### Impact Explanation

- **Block finalization DoS**: The `assert_eq!` panic in `read_batch_context_inputs` aborts block finalization in both the sequencer (forward system) and the prover (proof running system), making the block unproducible.
- **Permanent lock on SL chain-ID update**: The stale `Some(value)` causes every subsequent call to `update_settlement_layer_chain_id` within the same block to return an internal error, preventing the legitimate migration from completing.
- **State divergence**: Forward execution and proving execution may diverge if they handle the panic differently, violating the core ZKsync OS invariant.

---

### Likelihood Explanation

The trigger condition — a service transaction that emits `SettlementLayerChainIdUpdated` being rolled back due to block limits — is realistic. Block-limit rollbacks are a normal part of block production (`check_for_block_limits` is called for every transaction). A service transaction that generates unexpected pubdata (e.g., due to cold storage reads in SystemContext) can exceed limits and be rolled back. No privileged key compromise or external oracle manipulation is required; the operator's own normal block-building logic is sufficient to hit this path.

---

### Recommendation

Add `new_settlement_layer_chain_id_storage` to the frame snapshot/rollback mechanism, mirroring the existing treatment of `interop_root_storage`:

```rust
// In FullIOStateSnapshot
pub struct FullIOStateSnapshot<M: StorageModel> {
    io: M::StateSnapshot,
    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,
+   new_sl_chain_id: NewSettlementLayerChainIdSnapshotId,
}

// In start_io_frame
+   let new_sl_chain_id = self.new_settlement_layer_chain_id_storage.start_frame();
    Ok(FullIOStateSnapshot { io, transient, messages, events, interop_roots, new_sl_chain_id })

// In finish_io_frame
+   self.new_settlement_layer_chain_id_storage
+       .finish_frame(rollback_handle.map(|x| Some(x.new_sl_chain_id)));
```

### Proof of Concept

1. Deploy a block containing a single service transaction that calls `setSettlementLayerChainId(42)` on SystemContext.
2. Configure the block so that the transaction's pubdata usage exceeds the block pubdata limit (e.g., set `pubdata_price` high enough that the post-execution pubdata check triggers a rollback, as demonstrated in the existing regression test pattern): [10](#0-9) 
3. Observe that `finish_global_frame(Some(&pre_tx_rollback_handle))` is called, rolling back storage but not `new_settlement_layer_chain_id_storage`.
4. Block finalization calls `read_batch_context_inputs`. `settlement_layer_chain_id` from storage = original value (e.g., 1); `new_settlement_layer_chain_id_storage.value()` = `Some(42)`. The `assert_eq!(42, 1)` panics, aborting block finalization.
