### Title
Permanent NEP-141 → ERC-20 Token Mapping Cannot Be Updated or Removed - (File: `engine/src/engine.rs`)

---

### Summary

The `register_token` function in `engine/src/engine.rs` permanently locks the NEP-141 ↔ ERC-20 address mapping once set. There is no mechanism to update or remove an existing mapping. If the ERC-20 contract at the registered address becomes non-functional or requires replacement, the protocol has no recourse — all subsequent `ft_on_transfer` bridge inflows for that NEP-141 will continue routing to the stale ERC-20 address, permanently freezing bridged user funds.

---

### Finding Description

`Engine::register_token` enforces a one-time-only write guard:

```rust
pub fn register_token(
    &mut self,
    erc20_token: Address,
    nep141_token: AccountId,
) -> Result<(), RegisterTokenError> {
    match get_erc20_from_nep141(&self.io, &nep141_token) {
        Err(GetErc20FromNep141Error::Nep141NotFound) => (),
        ...
        Ok(_) => return Err(RegisterTokenError::TokenAlreadyRegistered),  // ← blocks all updates
    }
    nep141_erc20_map(self.io).insert(&nep141_token, &erc20_token);
    Ok(())
}
``` [1](#0-0) 

Once a NEP-141 account ID is mapped to an ERC-20 address via `nep141_erc20_map`, the mapping is immutable. A grep across the entire repository confirms there is no `unregister_token`, `remove_token`, or `update_token` function anywhere in the codebase. [2](#0-1) 

The public `deploy_erc20_token` contract method — callable by **any** NEAR account, not just the owner — is the sole entry point that invokes `register_token`. It only enforces `require_running`, with no owner-only gate:

### Impact Explanation

If the ERC-20 contract at the registered address becomes non-functional (e.g., due to a bug in `EvmErc20.sol`, a failed upgrade, or a self-destruct), the protocol has no path to redirect the NEP-141 → ERC-20 mapping to a replacement contract. Every subsequent `ft_on_transfer` call for that NEP-141 will attempt to mint on the broken ERC-20, failing silently or reverting. Bridged user funds (NEP-141 tokens locked on the NEAR side) become permanently unredeemable on the Aurora side — a **permanent freeze of bridged funds**.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The `deploy_erc20_token` entry point is permissionless (any NEAR account can call it). The ERC-20 contracts deployed are standard `EvmErc20.bin` bytecode with the Aurora engine as admin. The freeze scenario is triggered if:

1. A bug is discovered in the deployed ERC-20 contract (e.g., in `EvmErc20.sol`'s `mint`, `withdrawToNear`, or admin logic), **or**
2. The Aurora engine needs to migrate ERC-20 contracts to a new implementation.

In either case, the protocol has zero ability to update the mapping. Given that ERC-20 contract upgrades are a routine operational need and the ERC-20 bytecode is embedded at compile time, this scenario is **realistically likely** over the protocol's lifetime.

**Likelihood: Medium.**

---

### Recommendation

Add an owner-gated `unregister_token` (or `update_token`) function that removes or overwrites the entry in `nep141_erc20_map`. Access must be restricted to the contract owner via `require_owner_only`. Emit an event on any mapping change for auditability. Consider also adding owner-only access control to `deploy_erc20_token` itself, consistent with the pattern used by other sensitive admin functions in `admin.rs`.

---

### Proof of Concept

1. Any NEAR account calls `deploy_erc20_token` with `nep141 = "usdt.tether-token.near"`.
2. `engine::deploy_erc20_token` deploys an ERC-20 at address `0xABCD...` and calls `register_token(0xABCD, "usdt.tether-token.near")`.
3. `nep141_erc20_map` now permanently stores `"usdt.tether-token.near" → 0xABCD`.
4. A critical bug is found in `EvmErc20.sol`; the Aurora team deploys a patched ERC-20 at `0xDEAD...`.
5. The team calls `deploy_erc20_token` again for `"usdt.tether-token.near"` → `register_token` returns `Err(RegisterTokenError::TokenAlreadyRegistered)` and reverts.
6. All future `ft_on_transfer` calls for USDT continue minting on the buggy `0xABCD` contract.
7. Users bridging USDT from NEAR receive tokens on a broken ERC-20; their NEP-141 tokens are locked on the NEAR side with no redemption path. Funds are permanently frozen. [7](#0-6)
