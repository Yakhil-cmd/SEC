### Title
Missing BN254 G2 Subgroup Check in `ecPairing` Precompile Allows Invalid Pairing Inputs - (`File: basic_system/src/system_functions/bn254_pairing_check.rs`)

---

### Summary

The BN254 `ecPairing` precompile implementation in ZKsync OS validates G2 points only for curve membership (via `check()`) but never verifies that they belong to the correct r-torsion subgroup. Because BN254 G2 has a large non-trivial cofactor, a point can be on the curve yet outside the subgroup. An unprivileged transaction sender can supply such a crafted G2 point to the `ecPairing` precompile (address `0x08`), causing the pairing to produce a result that diverges from Ethereum mainnet semantics and potentially allowing forged pairing checks.

---

### Finding Description

In `bn254_pairing_check_inner`, each G2 point is constructed and validated as follows:

```rust
let g2_point = if g2_x.is_zero() && g2_y.is_zero() {
    G2Affine::zero()
} else {
    let g2_point = G2Affine::new_unchecked(g2_x, g2_y);
    g2_point.check().map_err(|_| ())?;   // ← only checks is_on_curve()
    g2_point
};
``` [1](#0-0) 

The `check()` method from the `ark_serialize::Valid` trait for `Affine<SWCurveConfig>` only calls `is_on_curve()`. It does **not** call `is_in_correct_subgroup_assuming_on_curve()`.

BN254 G2 has a large non-trivial cofactor:

```
COFACTOR = (36 * X^4) + (36 * X^3) + (30 * X^2) + 6*X + 1
         ≈ 21888242871839275222246405745257275088844257914179612981679871602714643921549
``` [2](#0-1) 

The subgroup check IS implemented for BN254 G2 (checks `[p]P = [6X²]P`):

```rust
fn is_in_correct_subgroup_assuming_on_curve(point: &G2Affine) -> bool {
    let x_times_point = point.mul_bigint(SIX_X_SQUARED);
    let p_times_point = p_power_endomorphism(point);
    x_times_point.eq(&p_times_point)
}
``` [3](#0-2) 

But it is never invoked during input parsing in the pairing precompile. By contrast, BN254 G1 has cofactor 1, so `is_in_correct_subgroup_assuming_on_curve` always returns `true` — the missing check is harmless for G1 but critical for G2. [4](#0-3) 

The BLS12-381 precompiles correctly use `parse_g1_with_subgroup_check` / `parse_g2_with_subgroup_check` for MSM and pairing operations, demonstrating the codebase is aware of the requirement: [5](#0-4) 

---

### Impact Explanation

**EVM semantic mismatch / crypto precompile parsing bug.**

Ethereum mainnet (go-ethereum and all other clients) rejects G2 points not in the r-torsion subgroup for `ecPairing` (EIP-197). ZKsync OS accepts them. This creates two concrete impacts:

1. **Forged pairing checks**: A G2 point outside the subgroup produces a pairing value in a different coset of GT. An attacker can craft a G2 point `Q'` (on the curve, not in subgroup) such that `e(P, Q') = 1` (the GT identity) even when no valid `Q` in the subgroup satisfies this. Any smart contract using `ecPairing` for BLS signature verification, ZK proof verification, or access control can be bypassed.

2. **Forward/proving divergence**: If the prover enforces subgroup membership (as cryptographic soundness requires) but the forward execution does not, a transaction that succeeds in forward execution will fail to generate a valid proof, causing an unprovable valid state transition.

---

### Likelihood Explanation

The `ecPairing` precompile is reachable by any unprivileged transaction sender via a `CALL` to address `0x08`. No special role or privilege is required. Constructing a BN254 G2 point that is on the curve but outside the r-torsion subgroup is straightforward: take any valid G2 point `P` and compute `P' = cofactor_part * P` where `cofactor_part` is a non-trivial divisor of the cofactor. The attack is deterministic and requires only knowledge of BN254 curve parameters.

---

### Recommendation

After constructing the G2 point and calling `check()`, add an explicit subgroup check:

```rust
let g2_point = G2Affine::new_unchecked(g2_x, g2_y);
g2_point.check().map_err(|_| ())?;
if !g2_point.is_zero() && !g2_point.is_in_correct_subgroup_assuming_on_curve() {
    return Err(());
}
```

This mirrors the pattern already used correctly in the BLS12-381 pairing precompile (`parse_g2_with_subgroup_check`). [6](#0-5) 

---

### Proof of Concept

1. Compute a BN254 G2 point `P` in the correct subgroup (e.g., the generator `G2`).
2. Compute `P' = h * P` where `h` is a non-trivial factor of the G2 cofactor (so `P'` is on the curve but `r * P' ≠ O`).
3. Submit a transaction calling address `0x08` with input `(G1_generator, P')`.
4. On Ethereum mainnet: the call reverts (invalid G2 point).
5. On ZKsync OS: the call succeeds and returns a pairing value computed over an invalid G2 point, diverging from mainnet behavior.
6. A contract that checks `ecPairing(A, B, C, D) == 1` can be manipulated by supplying crafted `B` or `D` outside the subgroup to force the pairing product to equal 1. [7](#0-6)
