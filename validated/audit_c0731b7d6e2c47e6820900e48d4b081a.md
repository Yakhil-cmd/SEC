### Title
Unchecked `transferFrom` Boolean Return Value in XCC Precompile Allows Fee-Free Cross-Contract Calls - (File: `engine-precompiles/src/xcc.rs`)

### Summary
The Cross-Contract Call (XCC) precompile calls `transferFrom` on the wNEAR ERC-20 contract to collect the required NEAR payment from the caller. It checks the EVM-level exit reason of the subcall but never decodes or validates the ABI-encoded boolean return value. If the wNEAR contract returns `false` without reverting, the precompile silently proceeds as if the payment succeeded, scheduling the cross-contract call without deducting any wNEAR from the caller.

### Finding Description
In `engine-precompiles/src/xcc.rs`, `run_with_handle` performs an EVM subcall to the wNEAR ERC-20 contract to execute `transferFrom(sender, engine_implicit_address, required_near)`:

```rust
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
match exit_reason {
    // Transfer successful, nothing to do
    aurora_evm::ExitReason::Succeed(_) => (),
    aurora_evm::ExitReason::Revert(r) => { return Err(...) }
    aurora_evm::ExitReason::Error(e) => { return Err(...) }
    aurora_evm::ExitReason::Fatal(f) => { return Err(...) }
}
``` [1](#0-0) 

The `return_value` variable — which holds the ABI-encoded `bool` returned by `transferFrom` — is only forwarded in the `Revert` arm. In the `Succeed(_)` arm it is completely discarded. The EIP-20 specification requires callers to handle a `false` return. Any wNEAR implementation that signals failure by returning `false` (without reverting) would cause the precompile to proceed past the payment gate and emit the promise log, scheduling the cross-contract call without collecting payment.

The `transfer_from_args` helper correctly encodes the `transferFrom` selector and arguments, confirming the intent is a standard ERC-20 `transferFrom` call whose boolean result must be checked. [2](#0-1) 

The wNEAR address is stored in engine state and read unconditionally at call time: [3](#0-2) 

### Impact Explanation
If `transferFrom` returns `false` without reverting, the caller's wNEAR balance is not reduced, the engine's implicit address does not receive the required NEAR, yet the XCC promise log is emitted and the cross-contract call is scheduled. The caller obtains free XCC execution — including free router-contract storage-staking (2 NEAR, `STORAGE_AMOUNT`) — without paying. This constitutes direct theft of protocol funds (the NEAR that should have been collected as payment). [4](#0-3) 

**Impact: High — Theft of unclaimed yield / temporary freezing of funds (router deployed without funded storage staking).**

### Likelihood Explanation
The wNEAR address is set by the owner via `factory_set_wnear_address` and is expected to be a standard OpenZeppelin-derived ERC-20 that reverts on failure. However:

1. The code imposes no enforcement that the stored address points to a revert-on-failure implementation.
2. Any future upgrade or replacement of the wNEAR contract that adopts a return-`false`-on-failure pattern (e.g., USDT-style) would silently break the payment gate.
3. An attacker who can influence which contract is at the wNEAR address (e.g., via a selfdestruct-and-redeploy at the same address) could exploit this immediately.

**Likelihood: Low-Medium** — not exploitable against the current canonical wNEAR deployment, but the missing check is a latent critical defect that becomes exploitable under realistic contract-upgrade or address-reuse scenarios.

### Recommendation
After the `Succeed(_)` arm, decode and assert the boolean return value:

```rust
aurora_evm::ExitReason::Succeed(_) => {
    // Decode the bool returned by transferFrom and revert if false
    let success = return_value
        .get(31)
        .map(|&b| b == 1)
        .unwrap_or(false);
    if !success {
        return Err(revert_with_message("ERR_WNEAR_TRANSFER_FAILED"));
    }
}
```

This mirrors the `safeTransferFrom` pattern recommended in the reference report: treat a `false` return as a revert rather than silently continuing.

### Proof of Concept
1. Deploy a wNEAR-compatible ERC-20 at the registered wNEAR address whose `transferFrom` always returns `false` without reverting (e.g., via selfdestruct + redeploy, or by setting a non-reverting wNEAR in a test environment).
2. Call the XCC precompile (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`) with a valid `CrossContractCallArgs::Eager` payload requiring NEAR payment.
3. Observe: `exit_reason` is `Succeed`, `return_value` is `abi.encode(false)`, the match arm does nothing, the promise log is emitted, and the caller's wNEAR balance is unchanged — the cross-contract call is scheduled without payment. [5](#0-4)
