Audit Report

## Title
Cross-Subaccount Allowance Enumeration via Incomplete Break Condition in `get_allowances` — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

## Summary
The `get_allowances` function in `rs/ledger_suite/icrc1/ledger/src/lib.rs` iterates a `StableBTreeMap<AccountSpender, ...>` ordered by full `Account` (owner + effective subaccount), but its only loop exit guard checks only the `owner` field. After exhausting allowances for the queried subaccount, iteration silently continues into adjacent subaccounts of the same owner. The public `icrc103_get_allowances` query endpoint applies no caller restriction, so any unprivileged principal can enumerate all allowances across every subaccount of a target principal.

## Finding Description
`AccountSpender::Ord` sorts by full `Account` (owner then `effective_subaccount()`), then by spender. [1](#0-0)  `Account::Ord` likewise orders by owner first, then by `effective_subaccount()`, making all subaccounts of the same owner contiguous in the map. [2](#0-1) 

The loop in `get_allowances` starts iteration from `{account: from, spender: <min>}` and its only break guard is:

```rust
if account_spender.account.owner != from.owner {
    break;
}
``` [3](#0-2) 

Once all spenders for `{owner:X, sub:[0;32]}` are yielded, the iterator advances to `{owner:X, sub:[1;32]}`, `{owner:X, sub:[2;32]}`, etc. The guard never fires because `owner` is unchanged across subaccounts. The correct guard must compare the full `Account` using `account_spender.account != from`, which `Account::PartialEq` handles correctly via `effective_subaccount()`. [4](#0-3) 

The `icrc103_get_allowances` endpoint is a `#[query]` with no access control and accepts an arbitrary `from_account` from any caller. [5](#0-4) 

## Impact Explanation
Any unprivileged caller who knows (or guesses) a target principal X can call `icrc103_get_allowances` with `from_account = {owner:X, subaccount:Some([0u8;32])}` and receive allowances belonging to every subaccount of X that has ever called `icrc2_approve`. The response discloses spender identity, approved amount, and expiry for each allowance across all subaccounts — data the subaccount owners have a reasonable expectation of privacy over. This constitutes a significant ICRC ledger security impact with concrete user harm (unauthorized cross-subaccount allowance metadata disclosure), qualifying under **High: Significant ledger security impact with concrete user or protocol harm**.

## Likelihood Explanation
The endpoint is a public query callable by any principal with no authentication or preconditions beyond the target principal having approved from more than one subaccount. The exploit is a single canister call requiring no special privileges, timing, or state manipulation, and is fully repeatable.

## Recommendation
Replace the owner-only comparison with a full `Account` equality check at line 1204 of `rs/ledger_suite/icrc1/ledger/src/lib.rs`:

```rust
// Before:
if account_spender.account.owner != from.owner {
    break;
}

// After:
if account_spender.account != from {
    break;
}
```

`Account::PartialEq` already normalizes `subaccount: None` and `subaccount: Some([0u8;32])` as equivalent via `effective_subaccount()`, so this handles all subaccount representations correctly.

## Proof of Concept
```rust
// Setup: owner X approves from two distinct subaccounts
let sub1 = Some([1u8; 32]);
let sub2 = Some([2u8; 32]);
icrc2_approve(caller=X, from_subaccount=sub1, spender=S1, amount=100);
icrc2_approve(caller=X, from_subaccount=sub2, spender=S2, amount=200);

// Attacker (any principal) queries with the minimum subaccount of X
let result = icrc103_get_allowances(GetAllowancesArgs {
    from_account: Some(Account { owner: X, subaccount: Some([0u8; 32]) }),
    prev_spender: None,
    take: None,
});

// BUG: result contains allowances for sub1 AND sub2, not just [0;32]
assert!(result.iter().all(|a| a.from_account == Account { owner: X, subaccount: Some([0u8;32]) }));
// This assertion FAILS — result includes entries with from_account.subaccount = sub1 and sub2
```

This can be reproduced as a PocketIC state-machine test against the ICRC ledger canister with two `icrc2_approve` calls from distinct subaccounts of the same principal, followed by an `icrc103_get_allowances` query from an unrelated caller.

#[query]
fn icrc103_get_allowances(arg: GetAllowancesArgs) -> Result<Allowances, GetAllowancesError> {
    let from_account = arg.from_account.unwrap_or_else(|| Account {
        owner: ic_cdk::api::msg_caller(),
        subaccount: None,
    });
    let max_take_allowances = Access::with_ledger(|ledger| ledger.max_take_allowances());
    let max_results = arg
        .take
        .map(|take| take.0.to_u64().unwrap_or(max_take_allowances))
        .map(|take| std::cmp::min(take, max_take_allowances))
        .unwrap_or(max_take_allowances);
    Ok(get_allowances(
        from_account,
        arg.prev_spender,
        max_results,
        ic_cdk::api::time(),
    ))
