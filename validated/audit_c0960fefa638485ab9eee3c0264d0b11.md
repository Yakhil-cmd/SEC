### Title
Double-Call to `to_reverted()` Fails to Clear Returndata When Execution Already Reverted — (`basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs`)

---

### Summary

In `ZkTransactionFlowOnlyEOA::execute_or_deploy_inner`, when the post-execution pubdata check fails (`!has_enough`), the code calls `execution_result.to_reverted()` **twice** — once to mutate the local variable (line 887) and once in the return expression (line 892). The `to_reverted()` method only clears returndata when converting a `Success` variant to `Revert { output: &[] }`. When `execution_result` is already a `Revert { output: some_data }`, `to_reverted()` returns it unchanged via the catch-all arm (`a => a`). As a result, when a transaction reverts during execution **and** the post-execution pubdata check also fails, the original revert returndata is preserved in the transaction output instead of being cleared to empty.

---

### Finding Description

`ExecutionResult::to_reverted()` is defined as:

```rust
pub fn to_reverted(self) -> Self {
    match self {
        Self::Success { output: ExecutionOutput::Call(_) }
        | Self::Success { output: ExecutionOutput::Create(_, _) } => Self::Revert { output: &[] },
        a => a,   // ← Revert { output: data } passes through unchanged
    }
}
```

In `execute_or_deploy_inner`, the pubdata-failure branch is:

```rust
if !has_enough {
    execution_result = execution_result.to_reverted();   // line 887 — no-op for Revert
    // ...
    Ok((
        execution_result.to_reverted(),                  // line 892 — still no-op for Revert
        CachedPubdataInfo { pubdata_used, to_charge_for_pubdata },
    ))
}
```

**Case A — `execution_result` is `Success`:**
- Line 887 converts it to `Revert { output: &[] }` ✓
- Line 892 is a no-op on the already-empty revert ✓

**Case B — `execution_result` is `Revert { output: returndata }`:**
- Line 887: `to_reverted()` hits `a => a`, `execution_result` stays `Revert { output: returndata }` ✗
- Line 892: same no-op, returns `Revert { output: returndata }` ✗

The caller (`create_frame_and_execute_transaction_payload`) then sees a `Revert` result and rolls back the outer frame, but the non-empty returndata is already embedded in the returned `ExecutionResult` and propagates into the final `ZkTxResult`. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

The transaction output (`ZkTxResult`) carries non-empty revert returndata when the protocol mandates empty returndata for pubdata-forced reverts. This creates a **state-transition correctness bug**:

1. **Incorrect transaction receipt**: The `returndata` field in the published transaction output is non-empty when it should be `&[]`, diverging from the expected ZKsync OS state-transition semantics.
2. **Forward/proving divergence risk**: If the forward system and the RISC-V proving system evaluate the pubdata check or the returndata clearing differently (e.g., due to resource accounting differences), the two modes produce different transaction outputs, making the block unprovable or producing an invalid proof.
3. **Downstream processing errors**: Any component consuming the `ZkTxResult` (e.g., the block bookkeeper, L1 bridge, or state-diff publisher) that relies on the invariant "pubdata-forced reverts have empty returndata" will observe incorrect data. [3](#0-2) [4](#0-3) 

---

### Likelihood Explanation

The trigger requires two simultaneous conditions:

1. **Transaction reverts during execution** — any EVM `REVERT` opcode, failed `require()`, or out-of-gas during the inner call frame.
2. **Post-execution pubdata check fails** — the remaining native resources are insufficient to cover pubdata generated after validation.

Condition 2 while condition 1 holds is possible when pubdata is generated during execution steps that are not fully rolled back at the system level before `check_enough_resources_for_pubdata` is called (e.g., pubdata from access-list warming, precompile calls, or system-level accounting that occurs outside the inner rollback boundary). An attacker can craft a transaction with a large access list or many storage reads to maximize validation-phase pubdata, then trigger a revert in the execution body, leaving the pubdata counter elevated relative to the available native budget. [5](#0-4) [6](#0-5) 

---

### Recommendation

Replace the double `to_reverted()` pattern with an unconditional empty-revert construction in the `!has_enough` branch:

```rust
if !has_enough {
    system_log!(system, "Not enough gas for pubdata after execution\n");
    context.resources.main_resources.exhaust_ergs();
    return Ok((
        ExecutionResult::Revert { output: &[] },
        CachedPubdataInfo { pubdata_used, to_charge_for_pubdata },
    ));
}
```

This mirrors the fix applied in the analogous L1 transaction path (`process_l1_transaction.rs` line 721: `returndata = Vec::new_in(...)`) and matches the invariant verified by the existing regression tests `test_returndata_cleared_when_reverted_after_execution` and `test_l1_tx_returndata_cleared_on_pubdata_revert`. [7](#0-6) [8](#0-7) 

---

### Proof of Concept

1. Deploy a contract whose bytecode writes to several storage slots (generating pubdata) and then executes `REVERT` with a non-empty reason string.
2. Submit a ZK L2 transaction calling that contract with a `gas_per_pubdata_byte_limit` set high enough that the post-execution pubdata check fails.
3. Observe that `ZkTxResult.result` is `ExecutionResult::Revert { output: <non-empty reason> }` instead of `ExecutionResult::Revert { output: &[] }`.
4. The incorrect returndata propagates into the block output and the published state transition, violating the protocol invariant that pubdata-forced reverts produce empty returndata. [9](#0-8) [10](#0-9)
