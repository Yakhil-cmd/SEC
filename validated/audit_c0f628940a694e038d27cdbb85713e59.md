### Title
Stale Cached Maturity Modulation Applied to ICP Minting Without Freshness Validation — (File: `rs/nns/governance/src/governance.rs`)

### Summary
The NNS Governance canister applies a cached `maturity_modulation.current_value_permyriad` to calculate the ICP amount minted during neuron spawning and maturity disbursement finalization. This cached value is updated at most once per day by the `UpdateIcpXdrRateRelatedData` timer task, but neither `maybe_spawn_neurons` nor `try_finalize_maturity_disbursement` checks the `updated_at_days_since_epoch` field before consuming the value. If the Exchange Rate Canister (XRC) is persistently unavailable, the cached modulation can remain stale for multiple days, causing incorrect ICP to be minted — directly analogous to the external report's pattern of a cached economic value being used without real-time validation.

### Finding Description

`HeapGovernanceData` stores `maturity_modulation: Option<MaturityModulation>`, where `MaturityModulation` carries `current_value_permyriad` (the cached rate) and `updated_at_days_since_epoch` (the day it was last computed). [1](#0-0) 

The `UpdateIcpXdrRateRelatedData` task updates this field daily by fetching ICP/XDR rates from the XRC and computing a 28-day rolling modulation. [2](#0-1) 

In `maybe_spawn_neurons`, the cached value is consumed directly with no staleness guard: [3](#0-2) 

The modulation is then applied to compute the exact ICP stake minted for each spawning neuron: [4](#0-3) 

Identically, `try_finalize_maturity_disbursement` reads the same cached field without checking `updated_at_days_since_epoch`: [5](#0-4) 

The `MaturityModulation` struct carries the staleness timestamp but it is never consulted at the point of use: [6](#0-5) 

The CHANGELOG for Proposal 141771 (2026-05-13) explicitly acknowledges that XRC failures are a real operational concern and introduces LOCF (last-observation-carried-forward) to prevent the update task from stalling. However, LOCF means the cached value can remain at an arbitrarily old reading if the XRC consistently fails: [7](#0-6) 

On canister init, the field is seeded with a neutral `0` permyriad and `updated_at_days_since_epoch: None`, so the first spawning batch always uses a neutral rate. After the first successful update, however, any subsequent XRC outage leaves a non-zero stale value in place indefinitely: [8](#0-7) 

### Impact Explanation

The maturity modulation is bounded to `[-500, +500]` permyriad (±5%), so the worst-case minting error is 10% of the neuron's maturity (e.g., +5% minted when −5% is correct). Because the governance canister mints ICP directly to the ledger, this constitutes a ledger conservation violation: the total ICP supply diverges from what the correct modulation would have produced. Every neuron that spawns or every maturity disbursement that finalizes during a stale-cache window is affected. The error persists until the XRC recovers and the daily task runs successfully.

### Likelihood Explanation

The XRC is a core IC canister but is not immune to outages; the CHANGELOG fix in Proposal 141771 was motivated by real observed XRC failures. The LOCF mechanism prevents the task from stalling but does not bound how long the cached value can be stale. A multi-day XRC outage coinciding with a significant ICP/XDR price move (which is precisely when the modulation matters most) produces the maximum divergence. Likelihood is low-medium given the XRC's general reliability, but the absence of any freshness guard means the window of incorrect minting is unbounded in duration.

### Recommendation

In both `maybe_spawn_neurons` and `try_finalize_maturity_disbursement`, read `updated_at_days_since_epoch` alongside `current_value_permyriad` and compare it to `current_day`. If the cached value is older than a configurable staleness threshold (e.g., 2 days), either block the operation and retry on the next timer tick, or fall back to a neutral 0 permyriad modulation rather than applying a potentially stale non-zero rate. This mirrors the pattern already used by the `UpdateIcpXdrRateRelatedData` task itself, which guards against double-updates by checking `updated_at_days_since_epoch == Some(current_day)`.

### Proof of Concept

1. XRC becomes unavailable for several consecutive days.
2. `UpdateIcpXdrRateRelatedData` runs each midnight, fails to fetch a new rate, and applies LOCF — leaving `current_value_permyriad` at its last successful value (e.g., `+500` permyriad, reflecting a prior high-ICP-price period).
3. ICP/XDR price drops sharply; the correct modulation should now be `−500` permyriad.
4. A batch of neurons whose `spawn_at_timestamp_seconds` has elapsed is processed by `maybe_spawn_neurons`.
5. `apply_maturity_modulation(original_maturity, +500)` is called instead of `apply_maturity_modulation(original_maturity, −500)`, minting ~10% more ICP than the protocol intends.
6. The ledger records the inflated mint; the error persists for every spawning/disbursement batch until the XRC recovers and the daily task successfully recomputes the modulation.
