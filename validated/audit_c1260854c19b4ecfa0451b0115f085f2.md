### Title
Global Cycles-Minting Rate Limit in CMC Can Be Exhausted by a Single Unprivileged User, Denying Service to All Other Users - (File: `rs/nns/cmc/src/main.rs`)

---

### Summary

The Cycles Minting Canister (CMC) enforces a single **global** rate limit of 150 Peta-cycles per hour (`DEFAULT_CYCLES_LIMIT`) shared across all callers. Any unprivileged user who sends enough ICP to mint cycles at or above this threshold in a single call will exhaust the shared bucket, causing every subsequent `notify_top_up` and `notify_mint_cycles` call from any other user to be rejected with a refund for the remainder of the hour. This is a direct analog of the L2→L1 global rate-limit exhaustion described in the external report.

---

### Finding Description

`rs/nns/cmc/src/main.rs` defines a hard-coded global limit:

```rust
const DEFAULT_CYCLES_LIMIT: u128 = 150e15 as u128;   // 150 Peta-cycles / hour
``` [1](#0-0) 

This limit is stored in `State::base_cycles_limit` and enforced by `base_limiter`, a `Limiter` instance with a 1-minute resolution and a 1-hour max-age window: [2](#0-1) 

Every successful cycle-minting path calls `ensure_balance`, which delegates to `Limiter::check_and_add_cycles`:

```rust
pub fn check_and_add_cycles(
    &mut self, now: SystemTime, cycles_to_mint: Cycles, limit: Cycles,
) -> Result<(), String> {
    self.purge_old(now);
    let count = self.get_count();
    if count + cycles_to_mint > limit {
        ...
        return Err(format!(
            "More than {} cycles have been minted in the last {} seconds, please try again later.",
            limit, self.get_max_age().as_secs(),
        ));
    }
    self.add(now, cycles_to_mint);
    Ok(())
}
``` [3](#0-2) 

The check is global — `total_count` accumulates across **all** callers: [4](#0-3) 

Both public entry points feed into this shared bucket:

- `notify_top_up` → `process_top_up` → `deposit_cycles` → `ensure_balance`
- `notify_mint_cycles` → `process_mint_cycles` → `do_mint_cycles` → `ensure_balance` [5](#0-4) [6](#0-5) [7](#0-6) 

---

### Impact Explanation

A single unprivileged user who sends ≥1,500 ICP (at the current ~100 XDR/ICP rate, yielding ≥150 Peta-cycles) in one `notify_top_up` call will atomically consume the entire hourly global budget. For the remainder of that 3,600-second window every other user's `notify_top_up` or `notify_mint_cycles` call is rejected with a refund (minus the transfer fee). Canisters that depend on timely cycle top-ups — including production services, SNS canisters, and subnet-rental operations — are denied service for up to one hour per exhaustion event. The attacker's own ICP is refunded, so the attack is nearly cost-free (only the ICP ledger transfer fee is lost).

**Vulnerability class:** cycles/resource accounting bug — global rate-limit exhaustion causing denial of service.

---

### Likelihood Explanation

The attack requires only a standard ICP ledger transfer followed by a `notify_top_up` call — both are fully permissionless. No privileged role, key, or governance majority is needed. The 150 Peta-cycle threshold corresponds to roughly 1,500 ICP at current prices, an amount well within reach of any moderately funded actor. The attack is repeatable every hour and is essentially free (ICP is refunded). The test suite explicitly documents the 150P/hr limit and confirms it blocks subsequent callers: [8](#0-7) 

---

### Recommendation

Replace the single global `base_limiter` with a **per-caller** (per-`PrincipalId`) rate limiter so that one user exhausting their own quota does not affect others. Alternatively, cap the maximum cycles that can be minted in a single call to a fraction of the global limit, preventing any single transaction from consuming the entire budget. The limit value and window should remain adjustable via NNS governance, but the accounting must be isolated per principal.

---

### Proof of Concept

1. Alice holds 2,000 ICP. She transfers 1,500 ICP to the CMC subaccount for canister `X` and calls `notify_top_up { block_index: B, canister_id: X }`.
2. CMC converts 1,500 ICP → ~150 Peta-cycles. `ensure_balance` calls `check_and_add_cycles(now, 150P, 150P)`. Since `0 + 150P == 150P` (not strictly greater), the call succeeds and `total_count` becomes 150P.
3. Bob immediately calls `notify_top_up` for even 1 ICP (→ 100T cycles). `check_and_add_cycles(now, 100T, 150P)` evaluates `150P + 100T > 150P` → **true** → Bob's call is rejected with `"try again later"` and his ICP is refunded minus the transfer fee.
4. Every other user's `notify_top_up` or `notify_mint_cycles` call is similarly rejected for up to 3,600 seconds.
5. Alice can repeat the attack each hour at near-zero net cost (ICP refunded, only ledger fee lost). [9](#0-8) [10](#0-9)
