Audit Report

## Title
`SenderInfoContent` Canister Signature Lacks Request-Context Binding, Enabling Cross-Request Attestation Replay - (File: `rs/types/types/src/messages/http.rs`)

## Summary

The `SenderInfoContent` signable used in `verify_sender_info_canister_sig` covers only the raw `info` blob under the domain separator `"ic-sender-info"`, with no inclusion of `canister_id`, `sender`, `MessageId`, `method_name`, or `ingress_expiry`. A user who has legitimately obtained a `sender_info` attestation from a signing canister (e.g., Internet Identity) for one request can replay the identical `(info, signer, sig)` triple in any subsequent request to any other canister. Any canister relying on `ic0_msg_caller_info_data_copy` for access-control decisions is therefore vulnerable to receiving a replayed attestation that the signing canister never produced for that specific target.

## Finding Description

`SenderInfoContent` is defined as a thin wrapper over the raw `info` bytes:

```rust
// rs/types/types/src/messages/http.rs, lines 342-348
pub struct SenderInfoContent<'a>(pub &'a [u8]);

impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.0);  // only the info blob
    }
}
```

Its `SignatureDomain` implementation uses only the static string `"ic-sender-info"`:

```rust
// rs/types/types/src/crypto/sign.rs, lines 161-165
impl<'a> SignatureDomain for SenderInfoContent<'a> {
    fn domain(&self) -> Vec<u8> {
        domain_with_prepended_length("ic-sender-info")
    }
}
```

The bytes actually signed are therefore `\x0Eic-sender-info ‖ info_bytes` — with no inclusion of `canister_id`, `sender`, `MessageId`, `method_name`, or `ingress_expiry`.

In `verify_sender_info_canister_sig` (`rs/validator/src/ingress_validation.rs`, lines 490–545), the replica verifies:
1. `sender_pubkey` is a valid canister signature public key.
2. The canister ID embedded in `sender_pubkey` matches `sender_info.signer`.
3. The canister signature over `SenderInfoContent(&sender_info.info)` is valid.

There is no step that checks whether `sender_info.sig` was produced for this specific request. The `(info, signer, sig)` triple is fully portable across any request made by the same sender principal.

The outer envelope signature (via `representation_independent_hash_call_or_query`, lines 43–79 of `http.rs`) does include the `sender_info` fields (`info`, `signer`, `sig` bytes) in the `MessageId` hash. This prevents a **third party** from injecting a different `sender_info` into someone else's request. However, it does **not** prevent the **legitimate sender** from reusing their own previously-obtained `(info, signer, sig)` triple in a different request to a different canister, because the sender controls their own private key and can construct a fresh outer envelope over any `sender_info` they possess.

The IC canister signature scheme embeds the signed message hash in the canister's certified data tree at path `["sig", seed_hash, msg_hash]`. Since `msg_hash` is derived only from `\x0Eic-sender-info ‖ info_bytes`, the same certificate is valid for any request carrying the same `info` bytes, regardless of target canister, method, or timing (within the certificate's validity window).

## Impact Explanation

Any canister that uses `ic0_msg_caller_info_data_copy` / `ic0_msg_caller_info_signer_copy` for access-control decisions is vulnerable. A user who legitimately obtained a `sender_info` attestation of, e.g., `b"role=admin"` from Internet Identity for canister X can replay the identical `(info, signer, sig)` in a request to canister Y. Canister Y's `inspect_message` or update handler will observe `msg_caller_info_data` = `b"role=admin"` as if the signing canister had freshly attested those attributes for that specific call — even though the signing canister never attested anything for canister Y. This constitutes unauthorized access to canister-controlled resources or privileged operations gated by `msg_caller_info_data`, matching the **High** impact category: unauthorized access to canister-controlled assets or identities where exploitation requires meaningful per-target work.

## Likelihood Explanation

Medium. The `sender_info` feature is newly introduced and its primary use case is for identity providers (like Internet Identity) to attest caller attributes. The attacker only needs to have made one legitimate request through the signing canister to obtain a reusable attestation. No privileged access, key compromise, or subnet-majority corruption is required. The attacker must find a target canister that uses `msg_caller_info_data` for access control with the same attestation value, but this is a realistic condition given the feature's intended use for role/attribute-based access control across the IC ecosystem.

## Recommendation

The signed content for `sender_info` must include request-binding context. `SenderInfoContent`'s `write_signed_bytes_without_domain_separator` should serialize `canister_id ‖ sender ‖ info_bytes` (or `message_id ‖ info_bytes`) rather than `info_bytes` alone. The signing canister and the replica validator must agree on this extended format. Concretely, `verify_sender_info_canister_sig` should pass the target `canister_id` and `sender` principal into `SenderInfoContent` construction, and the signing canister must include those fields when producing the signature. This ensures a signature produced for one request cannot be replayed against another.

## Proof of Concept

**Setup:**
- Signing canister C with seed S; user principal P = `self_authenticating(DER(C, S))`.
- Canister X and canister Y both deployed on the IC.
- Canister Y's `inspect_message` grants access only if `msg_caller_info_data` == `b"role=admin"`.

**Steps:**

1. User P makes a legitimate request to canister X with `sender_info`:
   ```
   info   = b"role=admin"
   signer = C
   sig    = canister_sig_C_S( \x0Eic-sender-info || b"role=admin" )
   ```
   The signing canister C certifies this in its certified data tree. The replica accepts it.

2. User P constructs a new request to canister Y with the **identical** `sender_info = {info, signer, sig}` but different `canister_id`, `method_name`, and `arg`. P signs the outer envelope with their own key (valid, since P controls the private key for seed S).

3. The replica calls `verify_sender_info_canister_sig`:
   - Checks `sender_pubkey` encodes canister C ✓
   - Checks `sender_info.signer == C` ✓
   - Verifies `sig` over `SenderInfoContent(b"role=admin")` ✓ (same bytes, same certificate)
   - **No check that this sig was produced for canister Y or this MessageId.**

4. The replica accepts the request. Canister Y's `inspect_message` reads `msg_caller_info_data` = `b"role=admin"` and grants access — even though signing canister C never attested those attributes for canister Y.

**Reproducible test:** A PocketIC integration test can instantiate two canisters, obtain a canister signature over `b"role=admin"` from a mock signing canister, submit it to the first canister to confirm acceptance, then resubmit the identical `sender_info` triple in a request to the second canister and assert that `ic0_msg_caller_info_data_copy` returns `b"role=admin"` — demonstrating the replay without any additional cryptographic work.
