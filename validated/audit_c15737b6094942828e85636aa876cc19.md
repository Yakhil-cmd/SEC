### Title
Missing Self-Delegation Guard in EIP-7702 Authorization List Processing Causes EVM Semantic Mismatch - (File: `basic_bootloader/src/bootloader/transaction/authorization_list.rs`)

---

### Summary

`validate_and_apply_delegation` in ZKsync OS does not check whether the recovered `authority` equals the `delegation_address`. The EIP-7702 specification requires skipping any authorization tuple where `address == authority`. Without this guard, an EOA can self-delegate, writing `0xef0100 || self` as its own bytecode. When that EOA is subsequently called, the runner resolves the delegation, reads the same EOA's bytecode as the delegate, and the EVM attempts to execute the delegation marker bytes — causing an INVALID opcode revert. On Ethereum mainnet the entry is silently skipped and the EOA behaves as code-free. This is a reachable, attacker-controlled EVM semantic mismatch.

---

### Finding Description

`validate_and_apply_delegation` in `basic_bootloader/src/bootloader/transaction/authorization_list.rs` implements the nine EIP-7702 validation steps: [1](#0-0) 

After recovering `authority` from the signature at step 3, the function immediately proceeds to read the authority's account properties and apply the delegation: [2](#0-1) 

There is no check of the form `if authority == B160::from_be_bytes(*delegation_address) { return Ok(false); }` anywhere between the ecrecover call and the `set_delegation` call. The EIP-7702 specification mandates this guard to prevent circular self-delegation.

When `set_delegation` is called with `at_address == delegate`, the account's bytecode is written as `0xef0100 || authority`: [3](#0-2) 

Later, when any caller invokes the self-delegated EOA, `read_callee_account_properties` in the runner resolves exactly one level of delegation: [4](#0-3) 

Because only one level is followed, `delegate_properties` is populated with the same EOA's data. The runner then uses `delegate_properties.bytecode` as the code to execute: [5](#0-4) 

That bytecode is `0xef0100 || authority` — the raw delegation marker. The EVM interprets `0xef` as an INVALID opcode (EIP-3541), causing the call to revert. On Ethereum mainnet the self-delegation tuple is skipped entirely, so the EOA retains no code and calls to it succeed normally.

---

### Impact Explanation

**Vulnerability class**: EVM semantic mismatch / state-transition bug.

Any EOA that self-delegates becomes permanently uncallable by external parties (every incoming call reverts with INVALID opcode) until the EOA clears the delegation with a subsequent EIP-7702 transaction setting `address = 0x0`. This diverges from Ethereum mainnet state-transition semantics, where the same authorization tuple is a no-op. Contracts or protocols that rely on calling a known EOA address (e.g., smart-account wallets, multisig coordinators, paymaster allowlists) would observe unexpected reverts on ZKsync OS that do not occur on mainnet, breaking cross-chain equivalence guarantees.

---

### Likelihood Explanation

Any unprivileged EOA can trigger this by submitting a valid EIP-7702 transaction whose authorization list contains a single entry where `address` equals the signer's own address. No special privilege, governance access, or oracle manipulation is required. The attacker controls the entire entry path: they sign the authorization tuple, submit the transaction, and the bootloader processes it without any guard.

---

### Recommendation

Add a self-delegation guard immediately after the authority is recovered (after step 3, before step 4) in `validate_and_apply_delegation`:

```rust
// After recovering `authority` from ecrecover:
let delegation_addr_b160 = B160::from_be_bytes(*delegation_address);
if authority == delegation_addr_b160 {
    return Ok(false); // EIP-7702: skip self-delegation
}
```

This matches the EIP-7702 specification requirement and aligns ZKsync OS behavior with Ethereum mainnet.

---

### Proof of Concept

1. EOA `A` constructs an EIP-7702 transaction with authorization list entry `[chain_id, A, nonce_A, y_parity, r, s]` (the `address` field equals `A`'s own address).
2. The bootloader calls `validate_and_apply_delegation`. All checks pass; `authority` is recovered as `A`; no self-delegation guard fires.
3. `set_delegation(A, A)` is called. `A`'s bytecode is written as `0xef0100 || A`. [6](#0-5) 
4. Any subsequent call to `A` enters `read_callee_account_properties`. `is_delegated = true`; delegation is parsed; `address = A`; `delegate_properties` is read for `A` again — bytecode is still `0xef0100 || A`. [7](#0-6) 
5. The runner uses `delegate_properties.bytecode` (`0xef0100 || A`) as the code to execute. [8](#0-7) 
6. The EVM executes `0xef` → INVALID opcode → call reverts.
7. On Ethereum mainnet the authorization tuple at step 2 is silently skipped; `A` retains no code; calls to `A` succeed. ZKsync OS produces a different state root for the same input, constituting a state-transition divergence.
