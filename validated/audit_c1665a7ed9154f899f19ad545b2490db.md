### Title
SNS Treasury Valuation Hardcodes 8-Decimal Assumption for Any ICRC-1 Token Balance, Miscalculating Transfer/Mint Limits - (File: `rs/sns/governance/token_valuation/src/lib.rs`)

---

### Summary

`try_get_balance_valuation_factors` in the SNS governance token valuation library unconditionally divides the raw ICRC-1 balance by the hardcoded constant `E8` (10^8) to normalize it into "whole tokens." This is correct only for tokens with exactly 8 decimal places (ICP, most SNS tokens). For any SNS token configured with a different number of decimals (e.g., 6 for USDC-style tokens, or 18 for ETH-style tokens), the computed treasury valuation is wrong by a factor of `10^(8 - actual_decimals)`. Because this valuation directly gates the 7-day transfer and mint limits enforced by `ProposalsAmountTotalUpperBound`, the limits are miscalculated, potentially allowing treasury draining beyond the intended caps.

---

### Finding Description

In `rs/sns/governance/token_valuation/src/lib.rs`, the function `try_get_balance_valuation_factors` fetches the raw ICRC-1 balance (in the token's smallest unit) and normalizes it:

```rust
let tokens = Decimal::from(u128::try_from(balance_of_response.0)...) / Decimal::from(E8);
``` [1](#0-0) 

`E8` is the constant `100_000_000` (10^8), hardcoded as the universal denominator for all tokens. This function is called for both ICP treasury accounts and SNS token treasury accounts. [2](#0-1) 

The ICRC-1 ledger standard allows any `decimals` value to be set at initialization. The SNS ledger suite orchestrator explicitly supports non-8-decimal tokens (e.g., ckUSDC uses 6 decimals): [3](#0-2) 

The resulting `tokens` Decimal is then used in `ValuationFactors::to_xdr()` to compute the XDR value of the treasury:

```rust
tokens * icps_per_token * xdrs_per_icp
``` [4](#0-3) 

This XDR value is fed into `ProposalsAmountTotalUpperBound` to determine the 7-day transfer/mint cap: [5](#0-4) 

The downstream `tokens_to_e8s` function also hardcodes `E8` when converting back: [6](#0-5) 

And `treasury_valuation_amount_e8s` in SNS governance calls this chain: [7](#0-6) 

---

### Impact Explanation

**Impact: High**

For an SNS token with **6 decimals** (e.g., a USDC-backed SNS token):
- Raw balance of 1,000,000 (= 1 token) is divided by 10^8 → computed as 0.01 tokens
- Treasury appears **100× smaller** than its actual value in XDR
- The treasury falls into the "small" regime (`< 100,000 XDR`) even when it is actually large, triggering `ProposalsAmountTotalUpperBound::NoLimit` — **no cap at all** on 7-day transfers/mints
- An SNS governance majority can drain the entire treasury in a single proposal window, bypassing the intended rate-limiting safeguard

For an SNS token with **18 decimals** (ETH-style):
- Treasury appears **10^10× larger** than actual value
- All `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals are blocked as exceeding the limit, even for tiny amounts — **governance DoS**

The `TransferSnsTreasuryFunds` and `MintSnsTokens` proposal types are the primary mechanism for SNS DAOs to manage their treasuries. Incorrect limits directly undermine the economic security model of every affected SNS.

---

### Likelihood Explanation

**Likelihood: Medium**

The ICRC-1 standard and the SNS ledger initialization explicitly support configurable `decimals`. Any SNS launched with a non-8-decimal token (e.g., a stablecoin-pegged SNS token with 6 decimals, or a token mirroring an ERC-20 with 18 decimals) is affected. The bug is triggered automatically at proposal submission time — no special attacker capability is needed beyond being an SNS governance participant with enough voting power to pass a proposal.

---

### Recommendation

Replace the hardcoded `E8` divisor in `try_get_balance_valuation_factors` with the actual token decimals fetched from the ledger via `icrc1_decimals()`. The corrected normalization should be:

```rust
let decimals: u8 = /* fetch via icrc1_decimals() from the ledger */;
let tokens = Decimal::from(raw_balance) / Decimal::from(10_u128.pow(decimals as u32));
```

Similarly, `tokens_to_e8s` should be generalized to accept the token's actual decimal count rather than assuming 8. The `IcpsPerSnsTokenClient` inflation ratio computation (`current_supply_e8s / initial_supply_e8s`) is unit-consistent and does not need to change, but its variable naming should be updated to avoid confusion.

---

### Proof of Concept

Consider an SNS initialized with a token of 6 decimals and a treasury balance of 10,000,000,000 (= 10,000 tokens at 6 decimals, worth ~100,000 XDR at 10 XDR/token):

1. `icrc1_balance_of` returns `Nat(10_000_000_000)`
2. `try_get_balance_valuation_factors` computes: `10_000_000_000 / 100_000_000 = 100` (tokens)
3. Correct value: `10_000_000_000 / 1_000_000 = 10_000` (tokens)
4. XDR valuation: `100 * icps_per_token * xdrs_per_icp` ≈ 1,000 XDR (instead of correct 100,000 XDR)
5. `ProposalsAmountTotalUpperBound::from_valuation_xdr(1_000 XDR)` → `NoLimit` (below 100,000 XDR threshold)
6. A `TransferSnsTreasuryFunds` proposal for the entire treasury (10,000 tokens) passes the limit check with no cap
7. The correct valuation of 100,000 XDR would have placed the treasury in the "medium" regime, capping 7-day transfers at a fraction of the treasury

The root cause is exclusively in the IC production code at `rs/sns/governance/token_valuation/src/lib.rs:181` — the hardcoded `/ Decimal::from(E8)` — which is the direct structural analog to the OmoOracle `getUSDValue` bug that assumed all tokens have 18 decimals. [1](#0-0)

    fn in_tokens(mut valuation: Valuation) -> Result<Decimal, ProposalsAmountTotalLimitError> {
        Self::clamp_xdrs_per_icp(&mut valuation);

        let ValuationFactors {
            tokens: balance_tokens,
            icps_per_token,
            xdrs_per_icp,
        } = valuation.valuation_factors;

        let self_ = Self::from_valuation_xdr(valuation.to_xdr());
        let result_tokens = match self_ {
            Self::NoLimit => balance_tokens,

            Self::Fraction(fraction) => balance_tokens
                .checked_mul(fraction)
                // Overflow should not be possible, since fraction is supposed to be at most 1.0.
                .ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "Unable to perform {balance_tokens} * {fraction}.",
                    ))
                })?,

            Self::Xdr(max_xdr) => {
                let xdrs_per_token = xdrs_per_icp.checked_mul(icps_per_token).ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "XDRs per token could not be calculated from valuation: {valuation:?}"
                    ))
                })?;

                // Calculate the inverse conversion rate.
                if xdrs_per_token == Decimal::from(0) {
                    // This is not reachable, because in this case, valuation.to_xdr() would return
                    // 0, and in that case, we would have taken the NoLimit branch.
                    return Err(ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "It appears that the tokens have zero value in XDR. valuation = {valuation:?}"
                    )));
                }
                let tokens_per_xdr = xdrs_per_token.inv();

                max_xdr.checked_mul(tokens_per_xdr).ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "Max tokens could not be calculated with valuation: {valuation:?}",
                    ))
                })?
            }
        };

        Ok(result_tokens)
