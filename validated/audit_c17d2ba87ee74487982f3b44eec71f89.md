### Title
Missing EIP-7610 Storage Collision Check Allows CREATE/CREATE2 Deployment to Address With Pre-Existing Storage — (`evm_interpreter/src/ee_trait_impl.rs`)

---

### Summary

ZKsync OS's EVM interpreter explicitly skips the EIP-7610 storage-collision check during `CREATE`/`CREATE2`. An unprivileged attacker can pre-plant storage at a deterministic `CREATE2` target address (by deploying a contract there, writing to storage, then self-destructing it in the same transaction under EIP-6780), and then a subsequent `CREATE2` deployment to that address will **succeed** in ZKsync OS while it would **fail** on a standard EVM. The deployed contract inherits the attacker's pre-planted storage, violating the EVM invariant that a freshly deployed contract starts with empty storage.

---

### Finding Description

In `before_executing_frame` in `evm_interpreter/src/ee_trait_impl.rs`, the collision guard for `Constructor` frames reads:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    // burn gas, emit CreateCollision, return Ok(false)
}
```

The comment explicitly acknowledges the gap. Only `code_len` and `nonce` are checked; **storage occupancy is never checked**. This is confirmed by multiple disabled test entries across the test index files:

- `create2collisionStorageParis.json` → `enabled: false, comment: We do not check storage for collision`
- `RevertInCreateInInitCreate2Paris.json` → `enabled: false, comment: We do not check for storage collisions`
- `test_init_collision_create_opcode.json` → `enabled: false, comment: We do not check the storage`
- `test_init_collision_create_tx.json` → `enabled: false, comment: We do not check the storage`

EIP-7610 (retroactively applied to all forks from Paris onward) mandates that `CREATE`/`CREATE2` must fail if the target address has non-empty storage. ZKsync OS does not enforce this.

---

### Impact Explanation

**Vulnerability class:** EVM semantic mismatch / state-transition bug.

An attacker can cause a victim's `CREATE2`-deployed contract to start with attacker-controlled storage values:

1. Attacker computes the victim's future `CREATE2` address deterministically (salt + deployer + initcode hash are all known or predictable).
2. Attacker deploys a contract to that address in a single transaction, writes arbitrary values to storage slots, then calls `SELFDESTRUCT` in the same transaction. Under EIP-6780, same-transaction self-destruct clears the code and resets the nonce to 0, leaving only the storage behind.
3. The address now has `code_len == 0`, `nonce == 0`, but **non-empty storage**.
4. The victim's `CREATE2` deployment proceeds. ZKsync OS's collision check passes (code and nonce are both zero), the constructor runs, and the contract is deployed — but it inherits the attacker's storage.
5. Any contract logic that assumes `SLOAD(slot) == 0` at construction time (e.g., `require(!initialized)`, proxy upgrade guards, ERC-20 balance slots) is now broken.

On a standard EVM (EIP-7610), step 4 would revert with a `CreateCollision` error, preventing the deployment entirely.

---

### Likelihood Explanation

**High.** The attack requires only:
- Knowledge of the victim's future `CREATE2` parameters (salt, deployer, initcode) — often public or predictable.
- A single transaction to pre-plant storage and self-destruct.
- No privileged access, no oracle manipulation, no governance majority.

The pattern is especially dangerous for factory contracts that deploy deterministic proxies or singleton contracts, where the `CREATE2` parameters are fixed and publicly known.

---

### Recommendation

Implement the EIP-7610 storage-occupancy check in `before_executing_frame`. Before allowing a constructor frame to execute, verify that the target address has no non-zero storage slots. If the storage model does not support a cheap "has any storage" query, a conservative alternative is to track whether any storage was ever written to an address (a dirty-bit per address) and use that as the collision signal.

---

### Proof of Concept

```
Block N, Tx 1 (attacker):
  PUSH 0x1234          // value to plant
  PUSH 0               // storage slot 0
  SSTORE               // storage[0] = 0x1234
  PUSH <attacker_addr> // beneficiary
  SELFDESTRUCT         // EIP-6780: same-tx → code cleared, nonce reset, storage REMAINS

  → address X: code_len=0, nonce=0, storage[0]=0x1234

Block N, Tx 2 (victim):
  CREATE2(salt, initcode)  // deterministic address = X

  ZKsync OS check: code_len==0 && nonce==0 → NO collision → deployment SUCCEEDS
  Standard EVM check: storage non-empty → CreateCollision → REVERTS

  Victim contract deployed at X with storage[0] = 0x1234 (attacker-controlled)
```
