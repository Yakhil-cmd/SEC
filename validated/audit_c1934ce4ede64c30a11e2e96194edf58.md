Audit Report

## Title
Unauthenticated `buyer` Field in `refresh_buyer_tokens` Allows Forced Swap Participation and Confirmation-Text Consent Bypass - (File: `rs/sns/swap/canister/canister.rs`)

## Summary
The SNS Swap canister's `refresh_buyer_tokens` endpoint resolves the buyer principal from the request's `buyer` field without verifying that the caller matches that principal. Any unprivileged ingress sender can call this endpoint on behalf of any victim who has already transferred ICP to the swap subaccount (step 1 of the two-step participation flow), forcing their participation and supplying the SNS-configured confirmation text on their behalf, bypassing the explicit consent mechanism entirely.

## Finding Description
In `rs/sns/swap/canister/canister.rs` at lines 128–143, the `refresh_buyer_tokens` handler resolves the buyer from the request without any caller-identity check:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()  // no check that caller == p
    };
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
``` [1](#0-0) 

Inside `refresh_buyer_token_e8s`, the swap checks the ICP balance at the subaccount derived from the supplied `buyer` principal and credits that buyer's participation record:

```rust
let account = Account {
    owner: this_canister.get().0,
    subaccount: Some(principal_to_subaccount(&buyer)),
};
icp_ledger.account_balance(account).await ...
``` [2](#0-1) 

The `confirmation_text` field — the SNS project's explicit consent mechanism — is taken directly from the attacker-controlled request and validated only as a string comparison against the publicly readable SNS init value:

```rust
self.validate_confirmation_text(confirmation_text)?;
``` [3](#0-2) 

The `validate_confirmation_text` function performs a plain string equality check with no caller binding: [4](#0-3) 

The `RefreshBuyerTokensRequest` proto documents the `buyer` field as "If not specified, the caller is used," explicitly permitting third-party specification: [5](#0-4) 

Critically, the existing integration test in `rs/tests/nns/sns/lib/src/sns_deployment.rs` explicitly demonstrates and accepts this behavior: `default_sns_agent` (a different identity) calls `refresh_buyer_tokens(Some(wealthy_user_identity.principal_id), None)` after the wealthy user has transferred ICP, and the call succeeds without any assertion of failure (`res_4`): [6](#0-5) 

Once `refresh_buyer_tokens` registers the buyer, `error_refund_icp` is blocked while the swap is OPEN and also blocked while the swap is COMMITTED/ABORTED but before sweep (the buyer's ICP is "in escrow"): [7](#0-6) 

If the swap commits, the victim's ICP is swept to SNS governance and they receive SNS tokens — they cannot reclaim their ICP.

## Impact Explanation
This is a **High** severity finding. An unprivileged attacker can:

1. **Force participation**: After a victim sends ICP to the swap subaccount (step 1) but before calling `refresh_buyer_tokens` (step 2), the attacker calls `refresh_buyer_tokens(buyer=victim, confirmation_text=<correct text>)`. The victim's ICP is now committed. If the swap commits, the victim receives SNS tokens instead of their ICP — they cannot reclaim their ICP. If the swap aborts, they can reclaim ICP only after finalization and sweep.

2. **Bypass the confirmation-text consent mechanism**: The `confirmation_text` field is the only explicit consent signal in the participation flow. An SNS project may set a legal disclaimer or risk warning. Because the confirmation text is publicly readable from the swap init and any caller can supply it on behalf of any buyer, the consent mechanism is completely undermined — a victim's ICP can be committed to a swap they never explicitly agreed to.

This matches the allowed impact: "Significant SNS or infrastructure security impact with concrete user or protocol harm" (High, $2,000–$10,000). The victim suffers real financial harm (forced conversion of ICP to SNS tokens, or at minimum a multi-day lock of funds) and the SNS-configured consent mechanism is rendered meaningless.

## Likelihood Explanation
The attack window exists for every user who has sent ICP to the swap subaccount but has not yet called `refresh_buyer_tokens`. This is a normal, expected intermediate state in the participation flow. The attacker needs only:
- Knowledge of the victim's principal (public on-chain)
- Knowledge of the confirmation text (publicly queryable from the swap canister's init)
- To submit an ingress message before the victim does

No privileged access, key material, or majority corruption is required. The attack is reachable by any unprivileged ingress sender and is repeatable across all active SNS swaps.

## Recommendation
Validate that the caller is the buyer when a non-empty `buyer` field is supplied in `rs/sns/swap/canister/canister.rs`:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let requested = PrincipalId::from_str(&arg.buyer).unwrap();
        if requested != caller {
            panic!("Caller {} is not authorized to refresh tokens on behalf of {}", caller, requested);
        }
        requested
    };
    ...
}
```

If third-party notification is a desired feature (e.g., for the NNS-dapp backend), an explicit allowlist of trusted callers should be maintained, mirroring the pattern in `rs/nns/cmc/src/main.rs` (`authorize_caller_to_call_notify_create_canister_on_behalf_of_creator`). [8](#0-7) 

The integration test at `rs/tests/nns/sns/lib/src/sns_deployment.rs` lines 919–926 that currently accepts third-party calls would need to be updated to use the participant's own identity.

## Proof of Concept
A deterministic PocketIC or StateMachine integration test can prove this:

1. Initialize an SNS swap with `confirmation_text = "I accept the risk"`.
2. As victim principal `V`, transfer 10 ICP to `swap_canister[subaccount = principal_to_subaccount(V)]`.
3. As attacker principal `A` (any unprivileged user), call:
   ```
   refresh_buyer_tokens({
     buyer: "<V's principal text>",
     confirmation_text: Some("I accept the risk")
   })
   ```
4. Assert the call succeeds and `get_buyer_state(V)` shows V as a committed participant.
5. Assert that `error_refund_icp` for V is blocked with "ICP cannot be refunded as principal V has N ICP (e8s) in escrow" while the swap is open.
6. Commit the swap; assert V receives SNS tokens and cannot reclaim ICP.

The existing test infrastructure in `rs/sns/test_utils/src/state_test_helpers.rs` already demonstrates the pattern (calling `refresh_buyer_tokens` with `buyer: participant_principal_id.to_string()` from a different sender), confirming the exploit path is directly exercisable. [9](#0-8)
