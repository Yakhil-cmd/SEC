Audit Report

## Title
Unconditional ballot overwrite in `vote_on_root_proposal_to_upgrade_governance_canister` allows a single NNS node operator to flip votes and block or force governance-canister upgrades — (`rs/nns/handlers/root/impl/src/root_proposals.rs`)

## Summary
`vote_on_root_proposal_to_upgrade_governance_canister` iterates over `node_operator_ballots` and unconditionally overwrites any matching entry with `*b = ballot.clone()`, with no check on the prior ballot value. Because `RootProposalBallot` has an `Undecided` variant and the mechanism is explicitly designed to tolerate up to f = (N−1)/3 Byzantine node operators, a single malicious-but-registered NNS node operator — well within the BFT budget — can flip their ballot arbitrarily many times within the 7-day proposal window, either permanently blocking a legitimate governance upgrade or forcing one through with fewer genuine Yes votes than the Byzantine threshold requires.

## Finding Description
`GovernanceUpgradeRootProposal` stores ballots as `Vec<(PrincipalId, RootProposalBallot)>`, initialized with one `Yes` entry for the proposer and `Undecided` for all other NNS node operators. [1](#0-0) [2](#0-1) 

The voting function performs four guards before reaching the ballot-write loop: expiry check, registry-version check, wasm-SHA check, and eligibility check (`voted_on == 0`). [3](#0-2) 

None of these guards inspect the *current* value of the ballot. The write at line 367 is unconditional: [4](#0-3) 

The threshold check `is_byzantine_majority_yes` requires `votes_yes >= num_nodes - max_faults`, where `max_faults = (num_nodes - 1) / 3`. Because the tally is recomputed from the mutable ballot vector on every call, and ballots can be overwritten at any time before the threshold is reached, the tally is not monotone — violating the core BFT invariant the mechanism is built on. [5](#0-4) 

**Attack path 1 — persistent blocking (Yes → No):**
1. Attacker (a registered NNS node operator) submits a proposal; their ballot is automatically `Yes`.
2. Other operators vote `Yes` until tally = N − f − 1 (one short of threshold).
3. Attacker calls `vote_on_root_proposal_to_upgrade_governance_canister(ballot=No)`; their `Yes` is overwritten with `No`, dropping tally to N − f − 2.
4. Each time another operator votes `Yes` to restore the tally, the attacker flips again. Repeatable indefinitely within the 7-day window.

**Attack path 2 — forced execution (No → Yes):**
1. A non-proposer node operator votes `No`.
2. Other operators vote `Yes` until tally = N − f − 1.
3. Attacker flips their `No` to `Yes`, pushing tally to N − f and triggering execution — with only N − f − 1 genuine Yes votes.

## Impact Explanation
The governance canister controls all NNS canisters and all ICP/Cycles flows. The root proposal mechanism's own documentation states its security level is intended to equal that of the subnet itself (tolerating up to f Byzantine nodes). [6](#0-5) 

This bug breaks that guarantee: a single node operator — 1 out of N, far below the f-fault budget — can either permanently DoS any governance upgrade for its full 7-day lifetime, or force an upgrade through with fewer genuine approvals than the Byzantine threshold requires. This matches **High ($2,000–$10,000): Application/platform-level DoS or subnet availability impact** and **High ($2,000–$10,000): Significant NNS security impact with concrete user or protocol harm**.

## Likelihood Explanation
Any registered NNS node operator can exploit this. No key compromise, social engineering, or majority collusion is required. The call is a standard ingress message to the NNS root canister. The only precondition is that a proposal is pending and has not yet reached majority — a normal operational state during any governance upgrade cycle. The attack is repeatable for the full 7-day proposal lifetime.

## Recommendation
Add an immutability guard before overwriting the ballot. Only allow a transition from `Undecided`:

```rust
for (p, b) in &mut proposal.node_operator_ballots {
    if p == &caller {
        if !matches!(b, RootProposalBallot::Undecided) {
            return Err(format!(
                "{LOG_PREFIX}Caller: {caller} has already cast a ballot."
            ));
        }
        *b = ballot.clone();
        voted_on += 1;
    }
}
```

This enforces the invariant that each node operator's ballot is immutable once cast, consistent with the BFT model the proposal mechanism is built on. [4](#0-3) 

## Proof of Concept
State-machine test with a 7-node NNS subnet (N=7, f=2, threshold=5):

1. Operator A submits proposal → tally: 1 Yes, 6 Undecided.
2. Operators B, C, D vote Yes → tally: 4 Yes, 3 Undecided (one short of threshold=5).
3. Operator A calls `vote_on_root_proposal_to_upgrade_governance_canister(ballot=No)` → tally: 3 Yes, 1 No, 3 Undecided.
4. Assert proposal is not executed and tally is 3 Yes.
5. Operator E votes Yes → tally: 4 Yes, 1 No, 2 Undecided.
6. Operator A flips back to No → tally: 3 Yes, 1 No, 3 Undecided.
7. Assert proposal is permanently blocked by a single operator.

For the forced-execution variant: after step 2, have operator A (who voted No in a separate scenario) flip to Yes, pushing tally to 5 and triggering `is_byzantine_majority_yes()` with only 4 genuine Yes votes. [7](#0-6)
