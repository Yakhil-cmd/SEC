### Title
Missing Zero-Address Validation for `refund_recipient` in L1→L2 Transaction Processing Causes Permanent Token Burning - (File: basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs)

### Summary
The `validate_structure()` function for ABI-encoded L1/upgrade transactions explicitly skips validation of the `refund_recipient` address field (`reserved[1]`), as acknowledged by a `// TODO: validate address?` comment. When this field is zero, the bootloader unconditionally transfers the refund amount to `address(0)`, permanently burning user funds from the treasury with no recovery path.

### Finding Description
In `validate_structure()`, every other reserved field is either validated or explicitly rejected, but `reserved[1]` (the `refund_recipient`) is left entirely unvalidated:

```rust
// reserved[1] = refund recipient for l1 to l2 and upgrade txs
match tx_type {
    Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
        // TODO: validate address?
    }
    _ => unreachable!(),
}
```

Later, in `process_l1_transaction`, the unvalidated value is read directly and used as the destination for a treasury token transfer:

```rust
let refund_recipient = u256_to_b160_checked(transaction.reserved[1].read());
mint_base_token::<S, Config>(
    system, system_functions, memories.reborrow(),
    &to_refund_recipient,
    &refund_recipient,   // ← zero address accepted without check
    ...
)?;
```

`mint_base_token` calls `transfer_from_treasury`, which subtracts `to_refund_recipient` tokens from the treasury (`BASE_TOKEN_HOLDER_ADDRESS`) and credits them to `refund_recipient`. When `refund_recipient` is `B160::ZERO`, the credit goes to `address(0)` — an address with no known private key — making the tokens permanently inaccessible. The treasury balance is reduced by the full refund amount with no error returned and no rollback triggered.

The existing regression test `test_treasury_based_token_distribution_regression` in `tests/instances/transactions/src/lib.rs` explicitly uses `address(0)` as the refund recipient and asserts the transfer succeeds, confirming this path is live and unguarded.

### Impact Explanation
Any L1→L2 priority transaction that carries a non-zero refund amount (i.e., `gas_limit > gas_used`, which is the common case) and specifies `refund_recipient = address(0)` will cause the bootloader to permanently burn the refund portion of the deposited base tokens from the treasury. The treasury balance is debited, the tokens are credited to `address(0)`, and the state transition is committed. There is no revert, no error, and no recovery mechanism. This constitutes a direct, irreversible loss of base-token funds.

### Likelihood Explanation
L1→L2 transactions are submitted by unprivileged users on L1. The `refund_recipient` field is a standard parameter that users or integrating smart contracts set. A zero value is the natural default for uninitialized or carelessly constructed transactions (e.g., `ZKsyncL1Tx::default()` produces `refund_recipient: Address::ZERO`). Any integrating contract or SDK that omits this field will silently burn the refund on every transaction. The `// TODO: validate address?` comment confirms the developers are aware the check is absent.

### Recommendation
Add a non-zero address check for `reserved[1]` inside `validate_structure()` in `basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`:

```rust
Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
    if u256_try_to_b160(self.reserved[1].read())
        .map_or(true, |addr| addr == B160::ZERO)
    {
        return Err(());
    }
}
```

Alternatively, if a zero `refund_recipient` is intentionally allowed to mean "refund to sender", add explicit fallback logic in `process_l1_transaction` to substitute `transaction.from()` when `reserved[1]` is zero, rather than silently burning the tokens.

### Proof of Concept
1. Submit an L1→L2 priority transaction with `refund_recipient = address(0)` and `gas_limit` larger than the actual execution cost (the common case).
2. The bootloader's `validate_structure()` passes without error (line 269–271 of `basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`).
3. After execution, `to_refund_recipient = gas_price * (gas_limit - gas_used) > 0`.
4. `process_l1_transaction` reads `refund_recipient = u256_to_b160_checked(reserved[1]) = B160::ZERO` (line 337 of `process_l1_transaction.rs`).
5. `mint_base_token` → `transfer_from_treasury` subtracts `to_refund_recipient` from `BASE_TOKEN_HOLDER_ADDRESS` and adds it to `B160::ZERO` (lines 793–831 of `process_l1_transaction.rs`).
6. The state transition commits; the refund tokens are permanently burned.

The existing test `test_treasury_based_token_distribution_regression` (line 1843 of `tests/instances/transactions/src/lib.rs`) already demonstrates this exact scenario passing without error, with `refund_recipient = address(0)` and the full refund amount credited to `address(0)`. ```rust
pub fn transfer_from_treasury<'a, S: EthereumLikeTypes + 'a>(
    system: &mut System<S>,
    nominal_token_value: &U256,
    to: &B160,
    resources: &mut S::Resources,
    fee_payment_in_simulation: bool,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
{
    system_log!(
        system,
        "Transferring {nominal_token_value:?} tokens from treasury to {to:?}\n"
    );

    let treasury_address = &system_hooks::addresses_constants::BASE_TOKEN_HOLDER_ADDRESS;

    let _ = system
        .io
        .update_account_nominal_token_balance(
            zk_ee::execution_environment_type::ExecutionEnvironmentType::EVM,
            resources,
            treasury_address,
            nominal_token_value,
            true, // true = subtract from balance
            fee_payment_in_simulation,
        )
        .map_err(|e| -> BootloaderSubsystemError {
            match e {
                SubsystemError::LeafUsage(balance_error) => {
                    system_log!(system, "Treasury transfer failed: {balance_error:?}");
                    interface_error!(BootloaderInterfaceError::TreasuryTransferFailed)
                }
                _ => wrap_error!(e),
            }
        })?;

    let _ = system
        .io
        .update_account_nominal_token_balance(
            zk_ee::execution_environment_type::ExecutionEnvironmentType::EVM,
            resources,
            to,
            nominal_token_value,
            false, // false = add to balance
            fee_payment_in_simulation,
        )
        .map_err(|e| -> BootloaderSubsystemError {
            match e {
                SubsystemError::LeafUsage(balance_error) => {
                    system_log!(system, "Error while minting: {balance_error:?}");
                    interface_error!(BootloaderInterfaceError::MintingBalanceOverflow)
                }
                _ => wrap_error!(e),
            }
        })?;
