Audit Report

## Title
Hot Keys Unconditionally Inherited by Spawned Neuron When New Controller Differs From Parent Controller - (File: rs/nns/governance/src/governance.rs)

## Summary
When `spawn_neuron` is called with a `new_controller` different from the parent neuron's controller, the child neuron is created with the parent's entire `hot_keys` list copied verbatim. The new controller receives a neuron they believe they fully control, but the spawner's hot key holders retain the ability to vote on NNS proposals, change followees, and join the Neurons' Fund on behalf of the new controller's neuron — without the new controller's knowledge or consent.

## Finding Description
In `rs/nns/governance/src/governance.rs`, `spawn_neuron` resolves `child_controller` at lines 2651–2655: if `spawn.new_controller` is `Some(p)`, `child_controller = p`; otherwise it defaults to `parent_neuron.controller()`. The child neuron is then built at lines 2704–2718 with `.with_hot_keys(parent_neuron.hot_keys.clone())` unconditionally — there is no branch that clears or filters hot keys when `child_controller != parent_neuron.controller()`. [1](#0-0) [2](#0-1) 

Authorization for non-privileged operations (`RegisterVote`, `Follow`, `JoinCommunityFund`, `LeaveCommunityFund`, `ChangeAutoStakeMaturity`) is checked via `is_hotkey_or_controller`, which returns `true` for any principal in `self.hot_keys`: [3](#0-2) 

No existing guard in `spawn_neuron` checks whether the inherited hot keys belong to the new controller or filters them out when the controller changes. The new controller has no in-protocol notification that inherited hot keys exist.

## Impact Explanation
This constitutes unauthorized access to a neuron's governance capabilities. A hot key holder on the parent neuron can, after the spawn:
- **Vote on NNS proposals** on behalf of the new controller's neuron, skewing governance outcomes.
- **Change followees** for governance topics on the new controller's neuron.
- **Join the Neurons' Fund** on behalf of the new controller's neuron, committing the new controller's future maturity — a financially consequential action the new controller did not authorize.

This matches the allowed High impact: *"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."*

## Likelihood Explanation
Any neuron controller who has added hot keys (a common convenience practice) and then spawns to a different principal will inadvertently transfer those authorizations. Deliberate exploitation requires only that the spawner retain a hot key before calling `spawn_neuron` with `new_controller = Some(victim)`. The recipient can remediate only if they are aware of the inherited hot keys; there is no protocol-level warning. Likelihood of unintentional leakage is medium-to-high; deliberate exploitation is low-to-medium.

## Recommendation
When `spawn_neuron` is called with a `new_controller` that differs from the parent's controller, the child neuron should be initialized with an empty hot key list instead of inheriting the parent's. Concretely, replace the unconditional `.with_hot_keys(parent_neuron.hot_keys.clone())` with a conditional:

```rust
let child_hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};
// ...
.with_hot_keys(child_hot_keys)
```

If hot key inheritance across controller boundaries is ever desired, it should be an explicit opt-in field in the `Spawn` message, not the default behavior.

## Proof of Concept
1. User A (controller of neuron N) adds hot key `H1` to neuron N.
2. User A calls `manage_neuron` → `Spawn { new_controller: Some(UserB), nonce: None, percentage_to_spawn: None }`.
3. `spawn_neuron` sets `child_controller = UserB` and builds the child neuron with `.with_hot_keys([H1])`. [4](#0-3) 
4. User A (via `H1`) calls `manage_neuron` on child neuron C with `RegisterVote { proposal_id, vote: Yes }`.
5. `is_hotkey_or_controller` returns `true` for `H1` on neuron C, so the vote is accepted. [5](#0-4) 
6. User A (via `H1`) calls `manage_neuron` on neuron C with `JoinCommunityFund {}`, committing User B's future maturity without consent.
7. A deterministic integration test using PocketIC can confirm this by: creating neuron N under User A, adding hot key H1, spawning to User B, then asserting that a `RegisterVote` call signed by H1 on the child neuron succeeds.
