Audit Report

## Title
`ManageVotingPermission` Self-Propagation to Third Parties via `add_neuron_permissions` — (`rs/sns/governance/src/neuron.rs`, `rs/sns/governance/src/governance.rs`)

## Summary
`ManageVotingPermission` is included in `PERMISSIONS_RELATED_TO_VOTING`, causing `is_exclusively_voting_related()` to return `true` for a list containing only `ManageVotingPermission`. As a result, `check_principal_authorized_to_change_permissions` accepts a caller holding only `ManageVotingPermission` as authorized to grant `ManageVotingPermission` to arbitrary third parties. Those third parties can immediately grant `Vote` and `SubmitProposal` to themselves, casting votes with the victim neuron's stake without the neuron owner's consent.

## Finding Description

`PERMISSIONS_RELATED_TO_VOTING` is defined to include `ManageVotingPermission` itself:

```rust
// rs/sns/governance/src/neuron.rs:61-65
pub const PERMISSIONS_RELATED_TO_VOTING: &'static [NeuronPermissionType] = &[
    NeuronPermissionType::Vote,
    NeuronPermissionType::SubmitProposal,
    NeuronPermissionType::ManageVotingPermission,  // ← self-referential inclusion
];
```

`is_exclusively_voting_related()` returns `true` for any list whose members are all in that set:

```rust
// rs/sns/governance/src/governance.rs:230-238
pub fn is_exclusively_voting_related(&self) -> bool {
    let permissions_related_to_voting = Neuron::PERMISSIONS_RELATED_TO_VOTING
        .iter().map(|p| *p as i32).collect::<Vec<_>>();
    self.permissions.iter().all(|p| permissions_related_to_voting.contains(p))
}
```

`check_principal_authorized_to_change_permissions` then accepts `ManageVotingPermission` as sufficient when the list is exclusively voting-related:

```rust
// rs/sns/governance/src/neuron.rs:152-159
let sufficient_permissions = if permissions_to_change.is_exclusively_voting_related() {
    vec![
        NeuronPermissionType::ManagePrincipals,
        NeuronPermissionType::ManageVotingPermission,
    ]
} else {
    vec![NeuronPermissionType::ManagePrincipals]
};
```

`add_neuron_permissions` calls this check followed by `check_permissions_are_grantable`:

```rust
// rs/sns/governance/src/governance.rs:4596-4600
neuron.check_principal_authorized_to_change_permissions(caller, permissions_to_add.clone())?;
self.nervous_system_parameters_or_panic()
    .check_permissions_are_grantable(permissions_to_add)?;
```

`check_permissions_are_grantable` only verifies membership in `neuron_grantable_permissions`. In production SNS deployments, `SnsInitPayload::get_nervous_system_parameters()` sets `neuron_grantable_permissions` to `NeuronPermissionType::all()`, so `ManageVotingPermission` passes this check unconditionally.

The complete exploit path:
1. Principal A holds only `ManageVotingPermission` on neuron N.
2. A calls `manage_neuron(N, AddNeuronPermissions { principal: B, permissions: [ManageVotingPermission] })`.
3. `[ManageVotingPermission].is_exclusively_voting_related()` → `true`; A has `ManageVotingPermission` → authorized; `ManageVotingPermission ∈ neuron_grantable_permissions` → passes. B now holds `ManageVotingPermission` on N.
4. B calls `manage_neuron(N, AddNeuronPermissions { principal: B, permissions: [Vote] })` — same checks pass. B now holds `Vote` on N.
5. B votes with N's stake.

The existing test `test_manage_voting_permission_allows_adding_voting_permissions_to_self` confirms the code path is live and passes for self-grant of all `PERMISSIONS_RELATED_TO_VOTING` including `ManageVotingPermission`. The same code path applies to third-party grants.

The `max_number_of_principals_per_neuron` guard (default 5) limits fan-out to new principals but does not prevent the attack: step 4 above adds `Vote` to B's existing slot, bypassing the principal-count check entirely.

For neurons whose controller holds only `PERMISSIONS_FOR_NEURONS_FUND_NNS_NEURON_CONTROLLER` (`ManageVotingPermission`, `SubmitProposal`, `Vote` — no `ManagePrincipals`), the original controller cannot use `ManagePrincipals` to revoke rogue grants. Revocation via `ManageVotingPermission` is possible but subject to a race: B can re-grant to C before A revokes from B.

## Impact Explanation

A principal holding only `ManageVotingPermission` — the standard hotkey configuration — can grant voting control over a neuron to colluding principals without the neuron owner's knowledge or consent. Those principals can immediately vote and submit proposals with the neuron's staked tokens, directly affecting SNS governance outcomes. This constitutes unauthorized access to neurons and governance assets with concrete, on-chain harm. This matches the **High** bounty impact: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."

## Likelihood Explanation

Both preconditions are routinely satisfied in production:
1. `ManageVotingPermission` is a standard hotkey permission granted to voting delegates.
2. `SnsInitPayload::get_nervous_system_parameters()` sets `neuron_grantable_permissions` to all permissions, so `ManageVotingPermission` is grantable by default in every SNS launched via the standard init path.

The attack requires only a standard `manage_neuron` ingress call — no privileged access, no key compromise, no social engineering.

## Recommendation

Remove `ManageVotingPermission` from `PERMISSIONS_RELATED_TO_VOTING`:

```rust
pub const PERMISSIONS_RELATED_TO_VOTING: &'static [NeuronPermissionType] = &[
    NeuronPermissionType::Vote,
    NeuronPermissionType::SubmitProposal,
    // ManageVotingPermission removed — granting it requires ManagePrincipals
];
```

This ensures only a `ManagePrincipals` holder can delegate `ManageVotingPermission`, while a `ManageVotingPermission` holder retains the ability to grant and revoke `Vote` and `SubmitProposal` as intended. Update the existing test `test_manage_voting_permission_allows_adding_voting_permissions_to_self` to reflect that granting `ManageVotingPermission` to self now requires `ManagePrincipals`.

## Proof of Concept

Minimal unit test using the existing `GovernanceCanisterFixtureBuilder` harness:

```rust
#[test]
fn test_manage_voting_permission_self_propagation_to_third_party() {
    let caller = *TEST_NEURON_1_OWNER_PRINCIPAL; // holds ManageVotingPermission only
    let attacker = *TEST_NEURON_2_OWNER_PRINCIPAL;

    let (mut governance, neuron) = {
        let permissions: &[(PrincipalId, NeuronPermissionList)] = &[(
            caller,
            vec![NeuronPermissionType::ManageVotingPermission].into(),
        )];
        let neuron_id = neuron_id(PrincipalId::new_user_test_id(0), 0);
        let fixture = GovernanceCanisterFixtureBuilder::new()
            .with_neuron_grantable_permissions(NeuronPermissionList::all())
            .add_neuron_with_permissions(permissions, neuron_id.clone())
            .create();
        (fixture, neuron_id)
    };

    // Step 1: caller grants ManageVotingPermission to attacker — should FAIL but currently succeeds
    governance
        .add_neuron_permissions(
            &neuron,
            attacker,
            vec![NeuronPermissionType::ManageVotingPermission].into(),
            caller,
        )
        .unwrap(); // BUG: this should return Err

    // Step 2: attacker grants Vote to themselves
    governance
        .add_neuron_permissions(
            &neuron,
            attacker,
            vec![NeuronPermissionType::Vote].into(),
            attacker,
        )
        .unwrap();

    // attacker can now vote with the neuron's stake
    governance.assert_principal_has_permissions_for_neuron(
        &neuron,
        attacker,
        vec![NeuronPermissionType::ManageVotingPermission, NeuronPermissionType::Vote].into(),
    );
}
```
