### Title
Missing Zero-Address Recipient Check in `withdrawToEthereum` Enables Permanent Fund Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` expose a `withdrawToEthereum(address recipient, uint256 amount)` bridge-exit function that burns the caller's ERC-20 tokens on Aurora and then calls the `ExitToEthereum` precompile to release the corresponding ETH on Ethereum. Neither the Solidity contract nor the `ExitToEthereum` precompile validates that `recipient != address(0)`. A caller who supplies `address(0)` as the recipient will have their tokens permanently burned on Aurora while the ETH is dispatched to the Ethereum zero address, resulting in irreversible fund destruction.

---

### Finding Description

In `EvmErc20.sol` (line 65–76) and `EvmErc20V2.sol` (line 66–77), the `withdrawToEthereum` function:

1. Calls `_burn(_msgSender(), amount)` — permanently destroying the caller's ERC-20 balance on Aurora.
2. Encodes `bytes20(recipient)` into the precompile calldata and invokes the `ExitToEthereum` precompile at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`. [1](#0-0) 

In the `ExitToEthereum::run()` handler (`engine-precompiles/src/native.rs`, flag `0x1` branch, lines 916–968), the precompile:

1. Parses the 20-byte recipient slice with `Address::try_from_slice(input)` — which succeeds for 20 zero bytes.
2. Hex-encodes it as `"0000000000000000000000000000000000000000"`.
3. Constructs the JSON withdrawal args `{"amount": "X", "recipient": "0000...0000"}` and schedules a `withdraw` promise to the NEP-141 ETH connector. [2](#0-1) 

At no point does either the Solidity contract or the Rust precompile perform a check equivalent to `require(recipient != address(0), "withdraw to the zero address")`. The same gap exists in `EvmErc20V2.sol`: [3](#0-2) 

---

### Impact Explanation

When `withdrawToEthereum(address(0), amount)` is called:

- The caller's ERC-20 tokens are **permanently burned** on Aurora (irreversible, no refund path).
- The `ExitToEthereum` precompile schedules a `withdraw` promise to the ETH connector targeting `address(0)` on Ethereum.
- The ETH connector releases the corresponding ETH to `address(0)` on Ethereum — a provably uncontrolled address from which no one can retrieve funds.

The result is **permanent, irrecoverable destruction of user funds** across both sides of the bridge. This satisfies the **Critical — Permanent freezing of funds** impact class.

---

### Likelihood Explanation

The entry path is directly reachable by any unprivileged EVM user or token holder who holds a balance in a bridged `EvmErc20`/`EvmErc20V2` token. The call requires no special role. The scenario is most likely triggered accidentally (e.g., a UI bug, a misconfigured contract integration, or a user copy-pasting a zero address), but it can also be triggered deliberately as self-harm. Because the `IExit` interface is public and the function accepts any `address` without restriction, any contract that wraps or proxies `withdrawToEthereum` and fails to validate the recipient propagates the same risk to its users. [4](#0-3) 

---

### Recommendation

Add an explicit zero-address guard at the top of `withdrawToEthereum` in both contracts:

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    require(recipient != address(0), "withdraw to the zero address");
    _burn(_msgSender(), amount);
    // ... rest of function
}
```

Additionally, add a corresponding guard in the `ExitToEthereum` precompile (`engine-precompiles/src/native.rs`) after parsing `recipient_address`:

```rust
if recipient_address == Address::zero() {
    return Err(ExitError::Other(Cow::from("ERR_WITHDRAW_TO_ZERO_ADDRESS")));
}
``` [5](#0-4) 

---

### Proof of Concept

1. A bridged ERC-20 token is deployed as `EvmErc20` on Aurora with a legitimate NEP-141 mapping.
2. Alice holds 1000 units of this token.
3. Alice (or a contract acting on her behalf) calls:
   ```solidity
   token.withdrawToEthereum(address(0), 1000);
   ```
4. `_burn(alice, 1000)` executes — Alice's Aurora balance drops to 0.
5. The `ExitToEthereum` precompile receives input `\x01 || amount_b || [0x00 * 20]`.
6. The precompile constructs `{"amount": "1000", "recipient": "0000000000000000000000000000000000000000"}` and schedules a `withdraw` promise.
7. The ETH connector releases 1000 units of ETH to `address(0)` on Ethereum.
8. Alice's 1000 tokens are permanently destroyed with no recovery path on either chain. [6](#0-5) [7](#0-6)
