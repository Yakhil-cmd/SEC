### Title
Missing Zero-Address Check in `AdminControlled` Constructor Permanently Bricks `EvmErc20`/`EvmErc20V2` Token Minting - (File: `etc/eth-contracts/contracts/AdminControlled.sol`)

---

### Summary

The `AdminControlled` constructor sets `admin = _admin` without checking that `_admin != address(0)`. The developer explicitly acknowledged the missing check by suppressing the Slither warning (`// slither-disable-next-line missing-zero-check`) rather than fixing it. Both `EvmErc20` and `EvmErc20V2` — the ERC-20 mirror tokens used in Aurora's NEP-141 bridging — inherit from `AdminControlled` and forward the `admin` parameter without any zero-address guard. If `admin` is set to `address(0)` at deployment, the `mint` function is permanently disabled, causing any subsequently deposited NEP-141 tokens to be permanently frozen.

---

### Finding Description

`AdminControlled.sol` line 10–12 sets the `admin` state variable directly from the constructor argument with no zero-address validation:

```solidity
constructor(address _admin, uint flags) {
    // slither-disable-next-line missing-zero-check
    admin = _admin;
```

The `onlyAdmin` modifier enforces `require(msg.sender == admin)`. Since `msg.sender` can never equal `address(0)` in a live EVM environment, setting `admin = address(0)` permanently locks every `onlyAdmin`-gated function.

Both `EvmErc20` and `EvmErc20V2` pass the `admin` constructor argument directly to `AdminControlled(admin, 0)` with no prior zero check:

```solidity
constructor (string memory metadata_name, string memory metadata_symbol,
             uint8 metadata_decimals, address admin)
    ERC20(metadata_name, metadata_symbol)
    AdminControlled(admin, 0)   // no require(admin != address(0))
```

The `mint` function — the only way to credit ERC-20 tokens to a user after a NEP-141 deposit — is gated by `onlyAdmin`:

```solidity
function mint(address account, uint256 amount) public onlyAdmin {
    _mint(account, amount);
}
```

If `admin == address(0)`, `mint` is permanently unreachable. There is no upgrade path or admin-recovery mechanism in the contract.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a user deposits a NEP-141 token into Aurora, the engine locks the NEP-141 tokens on the NEAR side and calls `mint` on the corresponding `EvmErc20`/`EvmErc20V2` contract to credit the user on the EVM side. If the contract was deployed with `admin = address(0)`, `mint` always reverts. The NEP-141 tokens are locked on NEAR with no corresponding ERC-20 tokens ever minted, and no admin exists to recover the contract. The funds are permanently frozen.

---

### Likelihood Explanation

**Medium.** The `admin` address is supplied at deployment time. In normal operation the Aurora engine sets it to its own EVM-derived address. However:

1. The constructor contains zero defensive validation — the suppressed Slither warning is the only acknowledgment of the risk.
2. A bug in a deployment script, a front-end misconfiguration, or an uninitialized variable (exactly the scenario described in the original report) could supply `address(0)`.
3. The `EvmErc20`/`EvmErc20V2` contracts are deployed programmatically at scale whenever a new NEP-141 token is registered; a single systemic bug in the deployment pipeline would affect every token deployed in that batch.
4. There is no post-deployment recovery path, so even a low-probability event has maximum severity.

---

### Recommendation

Add an explicit zero-address guard in the `AdminControlled` constructor and in each `EvmErc20`/`EvmErc20V2` constructor before forwarding the `admin` argument:

```solidity
constructor(address _admin, uint flags) {
    require(_admin != address(0), "AdminControlled: admin is zero address");
    admin = _admin;
    paused = flags;
}
```

Remove the `// slither-disable-next-line missing-zero-check` suppression comment and integrate Slither into CI so future suppressions are reviewed.

---

### Proof of Concept

1. Deploy `EvmErc20` (or `EvmErc20V2`) with `admin = address(0)`:
   ```solidity
   EvmErc20 token = new EvmErc20("Token", "TKN", 18, address(0));
   ```
2. Attempt to mint tokens (as would happen during a NEP-141 deposit):
   ```solidity
   token.mint(userAddress, 1e18); // reverts: require(msg.sender == admin) → require(msg.sender == address(0))
   ```
3. The call reverts unconditionally. No tokens can ever be minted. Any NEP-141 tokens locked on the NEAR side for this contract are permanently frozen with no recovery path.
