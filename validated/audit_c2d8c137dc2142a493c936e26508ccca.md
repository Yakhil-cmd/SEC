### Title
Zero `native_per_pubdata` Due to Integer Division Truncation Allows Unlimited Free Pubdata Generation - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary
When `pubdata_price < native_price`, the integer floor division used to compute `native_per_pubdata` silently truncates to zero. No guard prevents this zero result from propagating. With `native_per_pubdata == 0`, every pubdata cost check passes trivially, allowing an unprivileged transaction sender to generate arbitrarily large state-diff pubdata without paying any native resource cost for it.

### Finding Description
In `validate_and_compute_fee_for_transaction`, the per-pubdata-byte native cost is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`wrapping_div` is floor (truncating) division. When `pubdata_price < native_price` (e.g., `pubdata_price = 5`, `native_price = 10`), the result is `0`. No subsequent check rejects or adjusts this zero value.

By contrast, `native_per_gas` is computed with **ceiling** division (`div_ceil`), ensuring it is at least 1 whenever `gas_price > 0`:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(...)
``` [2](#0-1) 

The same floor-division pattern is also present in the off-chain API helper:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

When `native_per_pubdata == 0`, the intrinsic pubdata overhead charged during resource creation is:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata); // = 0
``` [4](#0-3) 

And every post-execution pubdata sufficiency check passes trivially:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)  // = 0 for any pubdata_spent
    .ok_or(out_of_native_resources!())?;
``` [5](#0-4) 

The `check_enough_resources_for_pubdata` function will always return `enough = true` because zero native resources are required regardless of how many bytes of pubdata the transaction produces. [6](#0-5) 

### Impact Explanation
An attacker can write to many storage slots in a single transaction, generating large state-diff pubdata. With `native_per_pubdata == 0`, the transaction pays only EVM gas (bounded by `gas_limit`) but zero native resources for pubdata. The operator must still publish all state diffs to L1 at real cost, without being compensated. This is a direct resource accounting bug leading to operator/protocol fund loss proportional to the pubdata generated.

### Likelihood Explanation
The condition `pubdata_price < native_price` is realistic. `native_price` reflects RISC-V proving cycle cost; `pubdata_price` reflects L1 calldata/blob cost. During periods of low L1 gas prices or high proving demand, `pubdata_price` can legitimately be less than `native_price`. The operator may not be aware that this configuration silently disables all pubdata charging. Any unprivileged L2 transaction sender can observe the block context and exploit the window.

### Recommendation
Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — can produce 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — minimum 1 when pubdata_price > 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Alternatively, add an explicit guard: if `pubdata_price > 0` but `native_per_pubdata == 0`, clamp it to 1 or return a validation error. Apply the same fix to the identical pattern in `api/src/helpers.rs`.

### Proof of Concept
1. Operator sets `native_price = 100`, `pubdata_price = 50` (realistic: L1 is cheap relative to proving).
2. `native_per_pubdata = 50.wrapping_div(100) = 0`.
3. Attacker submits an L2 transaction with `gas_limit = 10_000_000` that writes to 300 distinct storage slots (each write ≈ 64 bytes pubdata → ~19 200 bytes total pubdata).
4. `get_resources_to_charge_for_pubdata` returns `(19200, 0_native)`.
5. `check_enough_resources_for_pubdata` returns `enough = true`.
6. Transaction succeeds; attacker pays only EVM gas. Operator publishes ~19 200 bytes to L1 at their own expense.
7. Repeat with many transactions to drain operator funds.
