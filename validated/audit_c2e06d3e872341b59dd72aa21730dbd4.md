Audit Report

## Title
Unbounded Per-Peer Slot Table Growth on Five Non-Ingress P2P Channels Enables OOM DoS — (`rs/p2p/consensus_manager/src/receiver.rs`, `rs/replica/setup_ic_network/src/lib.rs`)

## Summary

Five of six artifact broadcast channels (consensus, certifier, dkg, idkg, https_outcalls) are configured with `SLOT_TABLE_NO_LIMIT = usize::MAX`, bypassing the per-peer slot table guard that protects the ingress channel. A single authenticated subnet peer can send an unbounded stream of slot updates with distinct slot numbers and artifact IDs, causing `slot_table`, `active_assembles`, and `artifact_processor_tasks` to grow without bound, leading to OOM crash or replica stall on every honest node that receives the peer's messages.

## Finding Description

`SLOT_TABLE_NO_LIMIT` is defined as `usize::MAX` and applied to five channels: [1](#0-0) 

Only the ingress channel receives a finite bound; consensus, certifier, dkg, idkg, and https_outcalls all pass `SLOT_TABLE_NO_LIMIT`: [2](#0-1) 

In `handle_slot_update_receive`, the guard `peer_slot_table_len < self.slot_limit` is evaluated for each new (vacant) slot entry. When `slot_limit = usize::MAX`, this condition is always true for any realistic `usize` value, so every new slot number is unconditionally accepted: [3](#0-2) 

When `to_add` is true and the artifact ID has not been seen before, a new entry is inserted into `active_assembles` and a new Tokio task is unconditionally spawned into `artifact_processor_tasks`: [4](#0-3) 

The three affected data structures are all unbounded for the five non-ingress channels: [5](#0-4) 

The inbound channel buffer of 100 (line 43) provides only backpressure, not a security bound — the attacker simply paces sends to drain the buffer continuously.

## Impact Explanation

This matches the allowed High impact: **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."** Each malicious slot update inserts one entry into `slot_table[peer_id]`, one entry into `active_assembles`, and spawns one long-lived async task holding a `watch::Receiver<PeerCounter>`, a cloned `Assembler`, and a `CancellationToken`. Sustained flooding causes monotonically increasing heap and task-stack consumption, resulting in OOM crash or replica stall. If enough honest replicas on a subnet are affected simultaneously, subnet consensus halts.

## Likelihood Explanation

The attack requires control of exactly one subnet node — below the BFT fault threshold — which is explicitly within the IC threat model. QUIC/TLS authentication ensures only subnet members can send slot updates, but a single compromised or malicious node suffices. The attack is trivially scriptable: a tight loop sending protobuf-encoded `SlotUpdate` messages with incrementing `slot_id` and unique `id` fields over the existing authenticated connection. No governance action, key compromise, or majority collusion is required.

## Recommendation

Apply a finite per-peer slot limit to all five non-ingress channels, analogous to `SLOT_TABLE_LIMIT_INGRESS`. The bound for each artifact type should reflect the maximum number of in-flight artifacts a legitimate peer would ever advertise simultaneously (e.g., a small multiple of the expected consensus round depth for consensus messages, committee size for certifications, DKG transcript count for dkg/idkg, etc.). Additionally, consider bounding `active_assembles` independently of the slot table, since the same artifact ID can be re-inserted after a task completes.

## Proof of Concept

```rust
// Pseudocode: malicious subnet node sends to honest peer on any non-ingress channel
for slot_id in 0u64.. {
    let msg = pb::SlotUpdate {
        commit_id: slot_id,
        slot_id,
        update: Some(pb::slot_update::Update::Id(
            unique_artifact_id(slot_id).encode_to_vec()
        )),
    };
    transport.send(honest_peer, "/consensus/update", msg.encode_to_vec()).await;
    // Each iteration: +1 slot_table entry, +1 active_assembles entry, +1 spawned task
    // honest peer's heap grows without bound → OOM
}
```

A deterministic integration test can confirm this by constructing a `ConsensusManagerReceiver` with `slot_limit = usize::MAX`, feeding N slot updates with distinct `slot_number` and `id` values from a single `peer_id`, and asserting that `active_assembles.len() == N` and `artifact_processor_tasks.len() == N` after processing.
