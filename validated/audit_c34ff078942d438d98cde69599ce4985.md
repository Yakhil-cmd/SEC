### Title
Markdown Injection via User-Controlled Memo in ICRC-21 `GenericDisplayMessage` - (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

---

### Summary

The ICRC-21 consent message library embeds user-controlled memo bytes directly into a Markdown-formatted `GenericDisplayMessage` string without sanitizing Markdown control characters. An unprivileged caller can craft a memo containing backticks, newlines, and `**bold**` markers to inject fake fields into the consent message that wallets display to users before transaction signing, potentially misleading users into approving transactions under false pretenses.

---

### Finding Description

In `packages/icrc-ledger-types/src/icrc21/responses.rs`, the `add_memo` function checks whether the raw memo bytes are valid UTF-8 and, if so, interpolates them verbatim into the Markdown string:

```rust
let memo_str = match std::str::from_utf8(memo.as_slice()) {
    Ok(valid_str) => valid_str.to_string(),
    Err(_) => hex::encode(memo.as_slice()),
};
match self {
    ConsentMessage::GenericDisplayMessage(message) => {
        message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
    }
    ...
}
``` [1](#0-0) 

No escaping or stripping of Markdown control characters (`\``, `\n`, `#`, `*`, `_`) is performed on `memo_str` before it is embedded inside the backtick code span. Because the memo is placed inside a single backtick span (`` `{memo_str}` ``), a memo containing a closing backtick immediately breaks out of the code span and allows arbitrary Markdown to follow.

**Concrete injection example.** An attacker crafts a 32-byte (ICRC-1 maximum) UTF-8 memo:

```
`\n\n**Amount:** `0 ICP
```

The resulting `GenericDisplayMessage` fragment becomes:

```
**Memo:**
``
**Amount:** `0 ICP`
```

A Markdown renderer displays this as a bold **Amount:** header followed by a code span `0 ICP`, visually indistinguishable from the real amount field that appeared earlier in the message. The real amount field is still present, but the injected field appears after it and may override the user's perception depending on the wallet's rendering order.

The same `format!` pattern is used in `add_amount`, `add_fee`, `add_allowance`, and `add_existing_allowance` for the `token_symbol` parameter:

```rust
message.push_str(&format!("\n\n**Amount:** `{amount} {token_symbol}`"));
``` [2](#0-1) 

`token_symbol` is not directly user-controlled per-call (it comes from ledger init), but the memo vector is fully attacker-controlled.

The `icrc21_canister_call_consent_message` endpoint is a public `#[update]` method on both the ICP ledger and the ICRC-1 ledger with no access control: [3](#0-2) [4](#0-3) 

The `build_icrc21_consent_info` function in the shared library is called by both ledgers and passes the decoded memo directly to `add_memo` without sanitization: [5](#0-4) 

The developers are demonstrably aware of the Markdown injection risk in this exact context: the ckBTC minter explicitly validates the Bitcoin address before embedding it in the consent message, with a comment stating this "rules out Markdown-injection vectors in the GenericDisplay output (e.g. an 'address' containing newlines or backticks crafted to fake additional fields)": [6](#0-5) 

The same protection was not applied to the memo field in the shared ICRC-21 library used by all ledgers.

---

### Impact Explanation

The ICRC-21 `GenericDisplayMessage` is a Markdown string that wallets (hardware wallets, mobile wallets, browser extensions) render and display to users as the authoritative description of a transaction before the user signs it. The entire security model of ICRC-21 depends on this message faithfully representing the transaction.

A malicious dApp that constructs a `TransferArg` with a crafted memo can cause the wallet to display:
1. A fake **Amount** field showing a different (e.g., smaller) amount than the actual transfer
2. A fake **To** field showing a different recipient
3. Fake approval sections that do not correspond to the actual transaction
4. Any other Markdown structure that misleads the user

Because the actual transaction parameters are encoded in the `arg` bytes (not the consent message), the on-chain effect is whatever the `arg` bytes specify — not what the manipulated consent message shows. A user who approves based on the injected display may unknowingly authorize a transaction with different parameters than displayed.

---

### Likelihood Explanation

The attack path requires no special privileges. Any caller can invoke `icrc21_canister_call_consent_message` with a `TransferArg` containing a crafted memo. The ICRC-1 memo field accepts up to 32 bytes of arbitrary bytes; if those bytes are valid UTF-8 (trivially achievable), they are embedded verbatim. A 20-byte memo is sufficient to inject a complete fake Markdown field. The attack is fully deterministic and requires no timing, no key material, and no governance majority.

---

### Recommendation

Before embedding `memo_str` into the `GenericDisplayMessage`, sanitize it by escaping or stripping Markdown control characters. At minimum, replace backtick (`` ` ``), newline (`\n`), carriage return (`\r`), `#`, `*`, and `_` characters. Alternatively, always hex-encode the memo in `GenericDisplayMessage` (as is already done for non-UTF-8 memos), reserving the human-readable display for `FieldsDisplayMessage` where the value is a structured `Value::Text` field that wallets render in a controlled context.

The ckBTC minter's `validate_address` pattern — rejecting any value that cannot be parsed as a known-safe format before interpolation — is the correct model to follow.

---

### Proof of Concept

Attacker calls `icrc21_canister_call_consent_message` on the ICP ledger or any ICRC-1 ledger with:

```
method = "icrc1_transfer"
arg = Candid-encode(TransferArg {
    from_subaccount: None,
    to: <any valid account>,
    fee: None,
    created_at_time: None,
    amount: 1_000_000,          // 0.01 ICP
    memo: Some(Memo(b"`\n\n**Amount:** `999 ICP")),
})
```

The returned `GenericDisplayMessage` will contain (after the real amount line):

```
**Memo:**
``
**Amount:** `999 ICP`
```

A wallet rendering this Markdown displays a second **Amount:** field showing `999 ICP`, overriding the user's perception of the real `0.01 ICP` amount. The actual on-chain transfer, if signed, moves `0.01 ICP`. [7](#0-6) [5](#0-4)

    pub fn add_memo(&mut self, memo: GenericMemo) {
        match memo {
            GenericMemo::Icrc1Memo(memo) => {
                // Check if the memo is a valid UTF-8 string and display it as such if it is.
                let memo_str = match std::str::from_utf8(memo.as_slice()) {
                    Ok(valid_str) => valid_str.to_string(),
                    Err(_) => hex::encode(memo.as_slice()),
                };
                match self {
                    ConsentMessage::GenericDisplayMessage(message) => {
                        message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
                    }
                    ConsentMessage::FieldsDisplayMessage(fields_display) => fields_display
                        .fields
                        .push(("Memo".to_string(), Value::Text { content: memo_str })),
                }
            }
            GenericMemo::IntMemo(memo) => match self {
                ConsentMessage::GenericDisplayMessage(message) => {
                    message.push_str(&format!("\n\n**Memo:**\n`{memo}`"));
                }
                ConsentMessage::FieldsDisplayMessage(fields_display) => {
                    fields_display.fields.push((
                        "Memo".to_string(),
                        Value::Text {
                            content: memo.to_string(),
                        },
                    ))
                }
            },
        };
    }

#[update]
fn icrc21_canister_call_consent_message(
    consent_msg_request: ConsentMessageRequest,
) -> Result<ConsentInfo, Icrc21Error> {
    let caller_principal = caller();
    let ledger_fee = Nat::from(LEDGER.read().unwrap().transfer_fee.get_e8s());
    let token_symbol = LEDGER.read().unwrap().token_symbol.clone();
    let token_name = LEDGER.read().unwrap().token_name.clone();
    let decimals = ic_ledger_core::tokens::DECIMAL_PLACES as u8;

    if consent_msg_request.method == "transfer" {
        let TransferArgs {
            memo,
            amount,
            fee,
            from_subaccount,
            to,
            created_at_time: _,
        } = Decode!(&consent_msg_request.arg, TransferArgs).map_err(|e| {
            Icrc21Error::UnsupportedCanisterCall(ErrorInfo {
                description: format!("Failed to decode TransferArgs: {e}"),
            })
        })?;
        icrc21_check_fee(&Some(Nat::from(fee)), &ledger_fee)?;
        let from = if caller() == Principal::anonymous() {
            AccountOrId::AccountIdAddress(None)
        } else {
            let account = Account {
                owner: caller(),
                subaccount: from_subaccount.map(|sa| sa.0),
            };
            AccountOrId::AccountIdAddress(Some(AccountIdentifier::from(account).to_hex()))
        };
        let receiver = AccountIdentifier::from_slice(&to).map_err(|e| {
            Icrc21Error::UnsupportedCanisterCall(ErrorInfo {
                description: format!("Failed to parse receiver account id: {e}"),
            })
        })?;
        let args = GenericTransferArgs {
            from,
            receiver: AccountOrId::AccountIdAddress(Some(receiver.to_hex())),
            amount: Nat::from(amount.get_e8s()),
            memo: Some(GenericMemo::IntMemo(memo.0)),
        };
        build_icrc21_consent_info(
            consent_msg_request,
            caller_principal,
            ledger_fee,
            token_symbol,
            token_name,
            decimals,
            Some(args),
        )
    } else {
        build_icrc21_consent_info_for_icrc1_and_icrc2_endpoints(
            consent_msg_request,
            caller_principal,
            ledger_fee,
            token_symbol,
            token_name,
            decimals,
        )
    }
}

/// Verifies that `address` parses as a valid Bitcoin address on the configured
/// network before it gets interpolated into a consent message. This both
/// guarantees the user is shown a meaningful (parseable) destination and rules
/// out Markdown-injection vectors in the GenericDisplay output (e.g. an
/// "address" that contains newlines or backticks crafted to fake additional
/// fields). Uses the same parser as `retrieve_btc_with_approval`, so any
/// address the consent endpoint accepts is also accepted by the actual call.
fn validate_address(address: &str, network: Network) -> Result<(), Icrc21Error> {
    BitcoinAddress::parse(address, network).map_err(|e| {
        Icrc21Error::UnsupportedCanisterCall(ErrorInfo {
            description: format!("Invalid Bitcoin destination address: {e}"),
        })
    })?;
    Ok(())
}
