### Title
Division Before Multiplication in `native_per_pubdata` Causes Systematic Underpayment for Pubdata Native Resources - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS, the `native_per_pubdata` rate is computed by dividing `pubdata_price` by `native_price` using integer (floor) division before it is multiplied by the number of pubdata bytes consumed. This is the same "division before multiplication" precision-loss pattern as the external report. The truncation causes every transaction that generates pubdata to be charged fewer native resources than it should be, allowing users to systematically underpay for pubdata proving costs.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (L2 transaction validation path), the per-pubdata-byte native cost is derived as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

This truncated `native_per_pubdata` is then multiplied by pubdata byte counts in two places:

1. **Intrinsic pubdata overhead** (charged upfront from the native limit):
```rust
// gas_helpers.rs, line 352
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
``` [2](#0-1) 

2. **Execution pubdata charge** (charged at end of execution):
```rust
// gas_helpers.rs, line 430-431
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [3](#0-2) 

The mathematically correct formula is:

```
native_cost_for_pubdata = pubdata_bytes * pubdata_price / native_price
```

But the actual computation is:

```
native_cost_for_pubdata = pubdata_bytes * floor(pubdata_price / native_price)
```

The same truncated division also appears in the off-chain validation helper:

```rust
// api/src/helpers.rs, line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [4](#0-3) 

---

### Impact Explanation

**Vulnerability class:** Resource accounting bug (division before multiplication → precision loss → systematic underpayment).

The truncation error per pubdata byte is `(pubdata_price mod native_price) / native_price`, which is always in `[0, 1)`. Across `N` pubdata bytes the total underpayment is:

```
underpayment = N * (pubdata_price mod native_price) / native_price
```

**Extreme case — `pubdata_price < native_price`:**

When the operator sets `pubdata_price = 999` and `native_price = 1000`:
- `native_per_pubdata = 999 / 1000 = 0` (truncated to zero)
- A transaction publishing 100,000 pubdata bytes is charged **0 native resources** for pubdata
- Correct charge: `100,000 * 999 / 1000 = 99,900` native units

The user publishes pubdata entirely for free in native resources, while the protocol must still prove the corresponding state diffs.

**Typical case — `pubdata_price` slightly above `native_price`:**

With `pubdata_price = 1,000,001` and `native_price = 1,000,000`:
- `native_per_pubdata = 1` (truncated, losing `1/1,000,000` per byte)
- Over 1,000,000 pubdata bytes: underpayment = 1 native unit (negligible)

The severity scales with `pubdata_price mod native_price` relative to `native_price`. The worst case is when `pubdata_price` is just below a multiple of `native_price`.

The impact is that the protocol/operator must prove more computation than users paid for in native resources, undermining the economic security of the native resource model described in `docs/double_resource_accounting.md`. [5](#0-4) 

---

### Likelihood Explanation

- **Entry path:** Any unprivileged user submitting an L2 transaction that generates pubdata (storage writes, events, deployments). This is the common case for virtually all non-trivial transactions.
- **Trigger condition:** Always active whenever `pubdata_price mod native_price != 0`, which is true for almost all operator configurations.
- **No special privileges required:** The attacker only needs to submit a standard EIP-1559 transaction.
- **Amplification:** A user can maximize the underpayment by crafting transactions that generate large amounts of pubdata (e.g., many storage writes), making the aggregate underpayment proportionally larger.

---

### Recommendation

Reorder the arithmetic to multiply before dividing, preserving full precision:

```rust
// In validate_and_compute_fee_for_transaction, do NOT pre-divide.
// Instead, pass pubdata_price and native_price separately and compute:
//   native_cost = pubdata_bytes * pubdata_price / native_price
// at each charge site.
```

Concretely, replace the two-step pattern:

```rust
// BEFORE (division first, precision loss):
let native_per_pubdata = pubdata_price / native_price;          // truncated
let native_cost = pubdata_bytes * native_per_pubdata;           // amplifies error

// AFTER (multiply first, then divide):
let native_cost = pubdata_bytes * pubdata_price / native_price; // exact
```

The same fix must be applied in `api/src/helpers.rs` `validate_l2_tx_intrinsic_native_resources` to keep the off-chain validator consistent with the on-chain bootloader.

---

### Proof of Concept

**Setup:**
```
pubdata_price  = 999
native_price   = 1000
pubdata_bytes  = 100_000
```

**Current (buggy) calculation:**
```
native_per_pubdata = 999 / 1000 = 0          // wrapping_div truncates
native_cost        = 100_000 * 0 = 0         // user pays nothing for pubdata
```

**Correct calculation:**
```
native_cost = 100_000 * 999 / 1000 = 99_900  // user should pay 99,900 native units
```

**Result:** The user publishes 100,000 bytes of pubdata and is charged **0 native resources** instead of 99,900. The protocol must prove the corresponding state diffs at its own cost. A user can exploit this by submitting transactions with maximum pubdata whenever `pubdata_price < native_price`, effectively getting pubdata proving for free.
