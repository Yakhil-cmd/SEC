### Title
ICRC-2 `icrc2_approve` Without `expected_allowance` Enables Allowance Double-Drain Race Condition - (File: rs/ledger_suite/common/ledger_core/src/approvals.rs)

---

### Summary

The ICRC-2 `icrc2_approve` endpoint on both the ICP ledger and ICRC-1 ledger accepts `expected_allowance` as an **optional** field. When omitted (`None`), `AllowanceTable::approve()` skips the current-allowance guard entirely and blindly overwrites the existing allowance. This creates a direct analog to the ERC-20 approve race condition: a spender who monitors the ledger via query calls can drain the old allowance before the owner's second approve is executed, then drain the new allowance afterward, extracting more tokens than the owner ever intended to authorize.

---

### Finding Description

**Vulnerability class**: Ledger conservation bug (allowance double-drain via inter-message race window).

The core logic lives in `AllowanceTable::approve()`:

```rust
// rs/ledger_suite/common/ledger_core/src/approvals.rs  lines 278-292
Some(old_allowance) => {
    if let Some(expected_allowance) = expected_allowance {   // ← guard is SKIPPED when None
        ...
        if expected_allowance != current_allowance {
            return Err(ApproveError::AllowanceChanged { current_allowance });
        }
    }
    // blindly overwrites old_allowance with the new amount
    table.allowances_data.set_allowance(key.clone(), Allowance { amount, ... });
``` [1](#0-0) 

When `expected_allowance` is `None`, the `if let Some(...)` branch is never entered, so the existing allowance is overwritten unconditionally regardless of what the spender may have already consumed or is about to consume.

The `ApproveArgs` struct exposes this as an opt-in field:

```rust
// packages/icrc-ledger-types/src/icrc2/approve.rs  line 18
pub expected_allowance: Option<Nat>,
``` [2](#0-1) 

A concrete production caller that omits this guard is the SNS governance `approve_treasury_manager` function, which passes `None` for `expected_allowance` on both the SNS token ledger and the ICP ledger:

```rust
// rs/sns/governance/src/extensions.rs  lines 791-801
// If expected_allowance is None, the ledger *blindly* overwrites any existing
// allowance (even if non-zero). Therefore, there is no risk of double spending.
self.ledger.icrc2_approve(to, sns_amount_e8s, Some(expiry_time_nsec),
    self.transaction_fee_e8s_or_panic(), self.sns_treasury_subaccount(),
    None,   // ← expected_allowance omitted
).await
``` [3](#0-2) 

The inline comment "Therefore, there is no risk of double spending" is misleading: it only addresses the governance canister's own idempotency, not the race window that the treasury manager canister (the spender) can exploit between two successive `approve_treasury_manager` calls.

---

### Impact Explanation

An unprivileged spender (any canister or principal holding an existing allowance) can:

1. Monitor the ledger via `icrc2_allowance` query calls (no special privilege required).
2. When the account owner submits a second `icrc2_approve` without `expected_allowance`, drain the **old** allowance via `icrc2_transfer_from` before the new approve is executed.
3. After the new approve is committed, drain the **new** allowance via a second `icrc2_transfer_from`.

Net result: the spender extracts `old_allowance + new_allowance` tokens, whereas the owner intended to authorize only `new_allowance`. This is a direct ledger conservation violation — tokens leave the owner's account in excess of their stated intent. [4](#0-3) 

---

### Likelihood Explanation

**Medium.** The race window is the inter-message gap between two successive `icrc2_approve` ingress calls on the same `(account, spender)` pair. Unlike Ethereum, the IC has no public mempool, so the spender cannot front-run a *pending* approve. However:

- The spender can poll `icrc2_allowance` (a query call, free and instant) to detect when the first approve is committed.
- The IC's finality latency is ~1–2 seconds per block; the owner's second approve will not arrive in the same block unless explicitly batched.
- The SNS governance `approve_treasury_manager` path is callable by any SNS proposal execution, and the treasury manager canister is the designated spender — it has both the motive and the on-chain capability to call `icrc2_transfer_from` immediately upon observing the first approval. [5](#0-4) 

---

### Recommendation

1. **Enforce `expected_allowance` in all production callers that update an existing allowance.** The SNS governance `approve_treasury_manager` should pass the current on-chain allowance (fetched immediately before the approve call) as `expected_allowance`, so the ledger rejects the update if the spender has already consumed the old allowance.

2. **In `AllowanceTable::approve()`**, consider whether the `None` path should be restricted or at minimum emit a warning in documentation, since it is the unsafe path that replicates the ERC-20 race condition.

3. **Correct the misleading comment** at `rs/sns/governance/src/extensions.rs` line 791–792. "No risk of double spending" is only true from the governance canister's own perspective; the spender-side race is still present. [6](#0-5) 

---

### Proof of Concept

**Actors**: Alice = SNS governance treasury subaccount (account owner); Bob = treasury manager canister (spender).

1. SNS governance executes `approve_treasury_manager(bob, sns=1000, icp=500)`.  
   → Ledger records allowance `(alice_sns_treasury → bob) = 1000`.

2. Bob calls `icrc2_transfer_from(from=alice_sns_treasury, to=bob_wallet, amount=1000)`.  
   → 1000 SNS tokens transferred. Allowance drops to 0.

3. SNS governance executes `approve_treasury_manager(bob, sns=500, icp=250)` (a revised proposal).  
   → Because `expected_allowance=None`, the ledger **blindly** sets allowance `(alice_sns_treasury → bob) = 500` without checking that Bob already drained the previous 1000.

4. Bob calls `icrc2_transfer_from(from=alice_sns_treasury, to=bob_wallet, amount=500)`.  
   → 500 more SNS tokens transferred.

**Total drained**: 1500 SNS tokens. **Owner's intent**: 500 SNS tokens (the revised amount). **Excess theft**: 1000 SNS tokens.

The same sequence applies to the ICP ledger path in the same function (lines 812–828). [7](#0-6) [8](#0-7)

    /// Returns an error if the allowance would go negative.
    pub fn use_allowance(
        &mut self,
        account: &AD::AccountId,
        spender: &AD::AccountId,
        amount: AD::Tokens,
        now: TimeStamp,
    ) -> Result<AD::Tokens, InsufficientAllowance<AD::Tokens>> {
        self.with_postconditions_check(|table| {
            let key = (account.clone(), spender.clone());

            match table.allowances_data.get_allowance(&key) {
                None => Err(InsufficientAllowance(AD::Tokens::zero())),
                Some(old_allowance) => {
                    if old_allowance.expires_at.unwrap_or_else(remote_future) <= now {
                        Err(InsufficientAllowance(AD::Tokens::zero()))
                    } else {
                        if old_allowance.amount < amount {
                            return Err(InsufficientAllowance(old_allowance.amount));
                        }
                        let mut new_allowance = old_allowance.clone();
                        new_allowance.amount = old_allowance
                            .amount
                            .checked_sub(&amount)
                            .expect("Underflow when using allowance");
                        let rest = new_allowance.amount.clone();
                        if rest.is_zero() {
                            if let Some(expires_at) = old_allowance.expires_at {
                                table.allowances_data.remove_expiry(expires_at, key.clone());
                            }
                            table.allowances_data.remove_allowance(&key);
                        } else {
                            table.allowances_data.set_allowance(key, new_allowance);
                        }
                        Ok(rest)
                    }
                }
            }
        })
