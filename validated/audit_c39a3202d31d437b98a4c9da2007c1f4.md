### Title
Incomplete Failure Handling in `exit_to_near_precompile_callback` Causes Permanent Fund Loss When `error_refund` Feature Is Disabled - (File: `engine/src/contract_methods/connector.rs`, `engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` compile-time feature is disabled, the `ExitToNear` precompile constructs `ExitToNearPrecompileCallbackArgs` with `refund: None`. If the downstream `ft_transfer` or `near_withdraw` NEAR promise subsequently fails, the `exit_to_near_precompile_callback` callback silently returns `None` without refunding the already-burned ERC-20 tokens. For regular ERC-20 exits, no callback is attached at all, so there is no recovery path whatsoever. In both cases, the user's ERC-20 tokens are permanently destroyed with no corresponding NEP-141 transfer completing.

### Finding Description

**Step 1 — Promise construction in the precompile (`engine-precompiles/src/native.rs`, lines 449–483):**

The `ExitToNear` precompile builds the callback args as:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None when feature is off
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

Then the callback is only attached when `callback_args != default()`:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback for regular ERC-20 exit
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

For a **regular ERC-20 exit** (non-wNEAR), `transfer_near` is `None` and `refund` is `None` (feature off), so `callback_args == default()` and **no callback is attached**. If `ft_transfer` fails, there is no recovery path.

For the **wNEAR unwrap exit**, `transfer_near` is `Some(...)`, so a callback is attached. However, `refund` is still `None`.

**Step 2 — Incomplete failure branch in the callback (`engine/src/contract_methods/connector.rs`, lines 214–242):**

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success: optionally transfer NEAR
    None
} else if let Some(args) = args.refund {
    // failure: refund burned tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None  // ← silent no-op when refund is None; burned tokens are unrecoverable
};
``` [3](#0-2) 

When `error_refund` is disabled, `args.refund` is always `None`, so the `else if` branch is never taken. A failed promise causes the callback to return `None` silently — no refund, no error propagation.

**Step 3 — ERC-20 tokens are burned before the promise is created:**

In `EvmErc20.sol` and `EvmErc20V2.sol`, `_burn` is called unconditionally before the precompile is invoked:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // ← irreversible burn
    ...
    assembly {
        let res := call(gas(), 0xe9217bc7..., ...)  // precompile call
    }
}
``` [4](#0-3) [5](#0-4) 

The burn is committed to EVM state before the asynchronous NEAR promise executes. If the promise fails and no refund occurs, the user's ERC-20 tokens are gone with no NEP-141 tokens delivered.

### Impact Explanation

**Critical — Permanent freezing of funds.** Any user who calls `withdrawToNear` on an ERC-20 token and whose downstream `ft_transfer` promise fails will permanently lose their tokens. The ERC-20 tokens are burned in the EVM state, the NEP-141 transfer never completes, and no refund is issued. The tokens are destroyed with no recovery path.

### Likelihood Explanation

**Medium.** The `ft_transfer` promise can fail for several realistic reasons:
- The recipient NEAR account has not registered storage on the NEP-141 contract.
- The NEP-141 contract is paused or has access controls.
- Gas exhaustion in the promise execution.
- The NEP-141 contract itself has a bug or is upgraded to reject transfers.

Any user performing a bridge withdrawal is exposed to this risk whenever the `error_refund` feature is not compiled into the production binary.

### Recommendation

1. Always attach a callback to the `ft_transfer` promise in the `ExitToNear` precompile, regardless of the `error_refund` feature flag.
2. In `exit_to_near_precompile_callback`, treat a failed promise with `args.refund == None` as an error (return `Err(...)`) rather than silently returning `None`, so the NEAR runtime can surface the failure.
3. Alternatively, ensure the `error_refund` feature is always enabled in production builds and enforce this via CI.
4. Consider making the refund path unconditional (not feature-gated) since it is a critical safety mechanism for bridge correctness.

### Proof of Concept

1. User holds ERC-20 tokens on Aurora (bridged from NEP-141 `token.near`).
2. User calls `withdrawToNear(recipient_bytes, amount)` on the ERC-20 contract.
3. `_burn(msg.sender, amount)` executes — ERC-20 tokens are destroyed.
4. `ExitToNear` precompile runs; since `error_refund` is disabled, `refund: None` and `transfer_near: None` → `callback_args == default()` → **no callback attached**.
5. Aurora engine schedules `ft_transfer` on `token.near`.
6. `ft_transfer` fails (e.g., recipient has no storage deposit on `token.near`).
7. No callback fires. No refund is issued.
8. User has lost their ERC-20 tokens permanently with no NEP-141 tokens received. [6](#0-5) [7](#0-6) [4](#0-3)
