### Title
Single-Step Irrecoverable Owner Transfer Permanently Locks All Owner-Gated Engine Controls - (`engine/src/contract_methods/admin.rs`)

---

### Summary

The Aurora Engine exposes a `set_owner` NEAR function that transfers the engine's `owner_id` in a single atomic step with no confirmation from the new owner. If an incorrect or inaccessible NEAR account ID is supplied, every owner-gated function — including `upgrade`, `resume_contract`, `stage_upgrade`, `set_key_manager`, and `attach_full_access_key` — becomes permanently unreachable. The most severe consequence is that a paused engine can never be resumed, permanently freezing all bridged ETH and ERC-20 funds held in Aurora.

---

### Finding Description

`set_owner` in `engine/src/contract_methods/admin.rs` (lines 103–121) reads a `SetOwnerArgs` struct from calldata and immediately overwrites `state.owner_id` with the caller-supplied `new_owner` value:

```rust
// engine/src/contract_methods/admin.rs  lines 103-121
pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;

        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }

        state.owner_id = args.new_owner;   // ← immediate, irrecoverable write
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
```

The only guard is `ERR_SAME_OWNER` — there is no zero-address check, no existence check for the new account, and crucially **no two-step handshake requiring the new owner to accept**. Once `state.owner_id` is written with an inaccessible account, the change is final.

`EngineState.owner_id` is the sole credential checked by `require_owner_only`:

```rust
// engine/src/contract_methods/mod.rs  lines 79-87
pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}
```

Every critical engine function gates on this check:

| Function | Effect if owner is lost |
|---|---|
| `upgrade` | Contract can never be upgraded |
| `stage_upgrade` | New WASM can never be staged |
| `resume_contract` | Paused engine can never be resumed |
| `pause_contract` | Engine can never be paused for emergency |
| `set_key_manager` | Relayer key management frozen |
| `attach_full_access_key` | No new FAK can be added |
| `set_upgrade_delay_blocks` | Upgrade delay permanently fixed |
| `resume_precompiles` | Paused precompiles permanently disabled |

---

### Impact Explanation

**Permanent freezing of funds (Critical).**

The engine can be paused via `pause_contract` (owner-only). If the engine is paused — whether before or after the bad `set_owner` call — and ownership is simultaneously or subsequently transferred to an inaccessible account, `resume_contract` can never be called. All ETH and ERC-20 tokens bridged to Aurora (held in the engine's NEAR account) are permanently frozen because no EVM transactions can execute while `is_paused == true`.

Even outside the paused scenario, the permanent loss of `upgrade` access means any future critical bug in the engine cannot be patched, which can lead to permanent fund loss through an unmitigated exploit.

---

### Likelihood Explanation

**Medium.** The owner is a NEAR account ID (not an EVM address), so format validation prevents obviously malformed inputs. However:
- A valid-format NEAR account ID that does not yet exist on-chain, or whose keys are lost, is accepted without error.
- Governance multisig operations involve multiple parties and off-chain coordination; a typo or stale account ID in a governance proposal is a realistic human-error scenario.
- The error is not immediately apparent — the contract continues to function normally until an owner-only function is needed.

---

### Recommendation

Implement a two-step ownership transfer:

1. **Step 1 — Propose**: The current owner calls `propose_owner(new_owner: AccountId)`, which stores a `pending_owner` in `EngineState` but does not change `owner_id`.
2. **Step 2 — Accept**: The `pending_owner` account calls `accept_owner()`, which verifies `predecessor_account_id == pending_owner` and then writes the new `owner_id`.

This ensures the new owner demonstrably controls the account before the transfer is finalized. Additionally, add an expiry on the pending proposal to prevent stale proposals from being accepted.

---

### Proof of Concept

1. Current owner (e.g., `aurora-dao.near`) calls `set_owner` with `new_owner = "aurora-dao-v2.near"` — a valid NEAR account ID format but one whose private key is unknown or lost.
2. `set_owner` passes all checks (`require_running`, `require_owner_only`, `ERR_SAME_OWNER`) and writes `state.owner_id = "aurora-dao-v2.near"` to storage.
3. The engine continues operating normally.
4. A critical bug is discovered; the team calls `stage_upgrade` → `ERR_NOT_ALLOWED`.
5. The team attempts `pause_contract` to halt further damage → `ERR_NOT_ALLOWED`.
6. The team attempts `resume_contract` if already paused → `ERR_NOT_ALLOWED`.
7. All bridged funds are permanently frozen with no recovery path.
