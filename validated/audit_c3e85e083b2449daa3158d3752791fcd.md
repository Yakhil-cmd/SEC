### Title
Callee Can Exhaust Entire Transaction's Native Resources Without EVM Gas Limit Constraint, Causing Unconditional Top-Level Revert - (`basic_bootloader/src/bootloader/runner.rs`)

---

### Summary

ZKsync OS implements a double resource accounting model: EVM gas (ergs) and a "native" resource representing proving cost (RISC-V cycles). While EVM gas passed to a callee is bounded by the 63/64 rule, **native resources are transferred in full to every callee with no per-call cap**. Because exhausting native resources triggers an unconditional top-level transaction revert with all gas consumed, any malicious callee that receives a non-trivial EVM gas allowance can deliberately exhaust the entire transaction's native budget, reverting the caller's transaction and burning all gas fees.

---

### Finding Description

ZKsync OS tracks two independent resources per transaction:

1. **Ergs** (EVM gas × 256): bounded per sub-call by the 63/64 rule, as in standard EVM.
2. **Native resource**: models RISC-V proving cycles. Explicitly passed in full to every callee.

The design is documented explicitly:

> "The native resources are passed fully from frame to frame, a call cannot set a limit on how much of it the callee can spend." [1](#0-0) 

The code that implements this unconditional transfer is in `read_callee_and_prepare_frame_state`:

```rust
// Give native resource to the callee.
resources_in_caller_frame.give_native_to(&mut callee_resources);
``` [2](#0-1) 

Contrast this with EVM gas, which is correctly bounded:

```rust
let max_passable_ergs =
    gas_utils::apply_63_64_rule(resources_available_in_caller_frame.ergs());
let ergs_to_pass = core::cmp::min(call_request.ergs_to_pass, max_passable_ergs);
``` [3](#0-2) 

When native resources are exhausted anywhere in the call stack, the error propagates as a `FatalRuntimeError` and is caught at the top-level transaction handler, which **unconditionally reverts the entire transaction and exhausts all remaining EVM gas**:

```rust
// Out of native is converted to a top-level revert and gas is exhausted.
Err(e) => match e.root_cause() {
    RootCause::Runtime(e @ RuntimeError::FatalRuntimeError(_)) => {
        context.resources.main_resources.exhaust_ergs();
        system.finish_global_frame(Some(&main_body_rollback_handle))?;
        (ExecutionResult::Revert { output: &[] }, None)
    }
``` [4](#0-3) 

The native resource budget for a transaction is `nativeLimit = gasLimit × nativePerGas`, where `nativePerGas = gasPrice / nativePrice`. Many EVM opcodes consume native resources at a ratio far exceeding 1 native unit per EVM gas unit. For example:

| Opcode | EVM gas | Native cost | Ratio (native/gas) |
|--------|---------|-------------|---------------------|
| `MLOAD`/`MSTORE` | 3 | 250 | ~83× |
| `MULMOD` | 8 | 4,000 | 500× |
| `DIV` | 5 | 1,500 | 300× |
| Cold SLOAD | 2,100 | ~100,000 | ~47× | [5](#0-4) [6](#0-5) 

A malicious callee receiving `G` EVM gas can consume up to `G/3 × 250 = ~83G` native units via tight `MLOAD` loops. If the transaction's `nativeLimit` is smaller than `83G` (which occurs when `nativePerGas` is low, i.e., gas price is close to `nativePrice`), the callee exhausts the entire native budget using only a fraction of its EVM gas allowance.

---

### Impact Explanation

- **Unconditional top-level revert**: Native exhaustion in any sub-call propagates as `FatalRuntimeError` and reverts the entire transaction, not just the sub-call. The caller has no `try/catch` equivalent that can isolate this failure.
- **Full gas loss**: `exhaust_ergs()` is called before the revert, so the transaction sender loses 100% of their gas fee.
- **No caller-side defense**: Unlike EVM gas (where a caller can pass `CALL(gas_limit, ...)` to cap callee consumption), there is no mechanism to limit native resources given to a callee. The caller cannot protect themselves.

---

### Likelihood Explanation

Any contract that calls an untrusted or upgradeable external contract is exposed. Realistic scenarios include:

1. A DeFi protocol calling an external token or oracle contract that has been upgraded to a malicious implementation.
2. An EIP-7702 delegated account whose delegation target is a malicious contract — any caller of that account triggers the attack.
3. A user calling a contract that internally calls a malicious plugin/hook/strategy contract.

The attack requires no privileged access. Any deployed contract can act as the malicious callee. The attacker only needs to be called with enough EVM gas to exhaust the native budget.

---

### Recommendation

1. **Apply a native resource cap per sub-call**, analogous to the 63/64 rule for EVM gas. For example, pass at most `63/64` of the remaining native resource to each callee, retaining a reserve in the caller frame.
2. Alternatively, **treat native exhaustion in a sub-call as a sub-call failure** (returning 0 on the stack) rather than a fatal top-level revert, consistent with how EVM handles out-of-gas in sub-calls.
3. At minimum, **document this as a known griefing vector** in user-facing developer documentation so protocol authors know not to call untrusted contracts with large gas allowances.

---

### Proof of Concept

**Setup:**
- Transaction gas limit: `100,000`
- `nativePerGas = 1` (i.e., `gasPrice == nativePrice`)
- `nativeLimit = 100,000`

**Malicious callee bytecode** (tight MLOAD loop):
```
; Loop 400 times: PUSH1 0x00, MLOAD, POP
; Each iteration: ~6 EVM gas, 250+250+40 = 540 native
; 400 iterations: ~2,400 EVM gas, ~216,000 native
PUSH1 0x90   ; loop counter = 400 (0x190 = 400)
...
loop:
  PUSH1 0x00
  MLOAD        ; 3 gas, 250 native
  POP          ; 2 gas, 40 native
  ...
  JUMPI loop
```

**Attack flow:**
1. Victim contract calls malicious contract with `50,000` EVM gas.
2. Malicious contract receives `50,000` EVM gas AND all `100,000` native units.
3. Malicious contract executes ~400 `MLOAD`/`POP` iterations: costs ~2,400 EVM gas, consumes ~116,000 native units.
4. Native budget exhausted → `out_of_native_resources!()` → `FatalRuntimeError`.
5. Top-level handler catches it at [7](#0-6) , calls `exhaust_ergs()`, reverts entire transaction.
6. Victim loses all 100,000 gas worth of fees. Their transaction produces no state changes.
