### Title
ETH and ERC-20 Tokens Permanently Lost When `exit_to_near` ft_transfer Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile in Aurora Engine conditionally registers a refund callback only when the `error_refund` compile-time feature is enabled. When this feature is absent (the default), a failed `ft_transfer` promise on the NEAR side causes the user's ETH (credited to the precompile address) or burned ERC-20 tokens to be permanently unrecoverable. This is the direct analog of the Magnetar `burst()` bug: value forwarded to a call that fails is not returned to the sender.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear::run()` function builds the callback arguments for the `exit_to_near_precompile_callback` as follows:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is not enabled, `refund` is hardcoded to `None`. The promise structure then becomes:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // no callback at all
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

For ERC-20 exits (where `transfer_near` is also `None`), `callback_args == default()` is true, so **no callback is registered at all**. For ETH exits where `transfer_near` is `Some`, a callback is registered but with `refund: None`, so the callback handler in `exit_to_near_precompile_callback` falls through to the `else { None }` branch and performs no refund:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // <-- no refund, funds are lost
};
``` [3](#0-2) 

The `error_refund` feature is **not** included in the `default` feature set:

```toml
[features]
default = ["std"]
...
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [4](#0-3) 

The fund loss mechanics differ by exit type:

- **ETH exit**: During EVM execution, ETH is transferred from the user's balance to the `exit_to_near` precompile address (`exit_to_near::ADDRESS`). If the subsequent `ft_transfer` promise fails, the ETH remains at the precompile address with no mechanism to recover it.
- **ERC-20 exit**: The ERC-20 tokens are burned during EVM execution. If the `ft_transfer` promise fails, the tokens are permanently destroyed with no re-mint.

The `refund_on_error` function in `engine/src/engine.rs` correctly handles both cases when called — for ETH it transfers back from the precompile address, for ERC-20 it re-mints — but it is never invoked when `error_refund` is disabled: [5](#0-4) 

The test suite explicitly documents this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) 

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent freezing/loss of user funds.**

When a user calls the `exit_to_near` precompile to bridge ETH or ERC-20 tokens to NEAR and the downstream `ft_transfer` promise fails (e.g., the recipient NEAR account is not registered with the NEP-141 contract, or the Aurora engine's NEP-141 balance is insufficient), the user's funds are irrecoverably lost:

- ETH is stranded at the `exit_to_near` precompile address with no recovery path.
- ERC-20 tokens are permanently burned.

There is no owner rescue function analogous to Magnetar's `rescueEth()` for the precompile address. The funds are frozen permanently.

---

### Likelihood Explanation

**Medium-High.** The `ft_transfer` promise can fail for several realistic, user-triggerable reasons:

1. The recipient NEAR account ID is not registered with the NEP-141 contract (storage deposit not paid).
2. The Aurora engine's NEP-141 balance is insufficient (e.g., drained by a prior transfer).
3. The NEP-141 contract is paused or rejects the transfer for any reason.

Any unprivileged EVM user can trigger this by calling the `exit_to_near` precompile with a recipient that will cause the `ft_transfer` to fail. The entry path is fully reachable via the standard `submit` endpoint.

---

### Recommendation

Ensure the `error_refund` feature is unconditionally enabled in the production WASM build, or remove the feature flag entirely and always include the refund callback args. The `refund_on_error` logic already exists and is correct; the only gap is that it is gated behind a compile-time flag that is absent from the default feature set.

Concretely, in `engine-precompiles/src/native.rs`, replace the conditional compilation with an unconditional call:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    refund: refund_call_args(&exit_to_near_params, &exit_event), // always set
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature (the default build).
2. Bridge ETH to Aurora so a user has an ETH balance.
3. The user calls the `exit_to_near` precompile specifying a NEAR recipient account that is **not registered** with the NEP-141 ETH connector contract.
4. The EVM execution succeeds: ETH is deducted from the user and credited to `exit_to_near::ADDRESS`.
5. The `ft_transfer` promise to the NEP-141 contract fails (unregistered recipient).
6. No callback is registered (or the callback runs with `refund: None`).
7. The user's ETH balance on Aurora is permanently reduced; the ETH at the precompile address is unrecoverable.

This is confirmed by the existing test `test_exit_to_near_eth_refund` which explicitly asserts the balance loss when `error_refund` is not compiled in: [8](#0-7)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
