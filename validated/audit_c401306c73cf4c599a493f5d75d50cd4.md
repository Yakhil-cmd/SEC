### Title
Unbounded Nested Loop in SNS `distribute_rewards` Exhausts Instruction Limit, Permanently Halting Voting Reward Distribution - (File: rs/sns/governance/src/governance.rs)

### Summary
The SNS governance canister's `distribute_rewards` function executes a nested unbounded loop over all unsettled proposals and all their ballots in a single synchronous canister message, with no instruction-limit guard. An unprivileged user can stake SNS tokens to create many neurons, inflating the ballot count per proposal. When the total work (proposals × neurons) exceeds the IC instruction limit, the periodic task traps and voting rewards are permanently halted until a governance upgrade is deployed.

### Finding Description
`distribute_rewards` in `rs/sns/governance/src/governance.rs` is called synchronously from `run_periodic_tasks` (line 5513). Inside it, two unbounded loops execute without any instruction-limit check:

**Loop 1 (lines 5894–5931):** Nested iteration over every `considered_proposal` (up to `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS = 700`) and every ballot entry in each proposal's `ballots` map (one entry per eligible neuron at proposal creation time):

```rust

        for (voter, ballot) in &proposal.ballots {
            // accumulate neuron_id_to_reward_shares

```

**Loop 2 (lines 5954–5997):** Iteration over every entry in `neuron_id_to_reward_shares` to mutate neuron maturity:

```rust
for (neuron_id, neuron_reward_shares) in neuron_id_to_reward_shares {
    let neuron: &mut Neuron = match self.get_neuron_result_mut(&neuron_id) { ... };
    // update maturity
}
```

**Loop 3 (lines 6013–6081):** Iteration over all `considered_proposals` again to settle them and clear ballots.

None of these loops contain a call to `is_message_over_threshold` or any equivalent instruction-limit check. The entire function runs atomically in one message execution.

By contrast, NNS governance was explicitly patched (NNS Proposal 135702) to move reward distribution to a batched `RewardsDistributionStateMachine` with `is_message_over_threshold` checks in `continue_processing`. SNS governance has no equivalent protection.

The attacker-controlled growth vector: any principal can stake SNS tokens to create neurons (subject to `max_number_of_neurons`). With many neurons, every new proposal created carries a large ballot map. With 700 proposals × N neurons, the work in a single `distribute_rewards` call grows as O(proposals × neurons).

### Impact Explanation
When `distribute_rewards` exceeds the IC instruction limit (~40 billion instructions per message), the canister traps mid-execution. Because `run_periodic_tasks` is called on a timer, every subsequent invocation will also trap at the same point. Voting rewards are permanently halted for all SNS neuron holders. The only recovery path is a governance upgrade proposal — which itself requires functioning governance — creating a potential deadlock. Even if an upgrade is possible, all reward rounds that elapsed during the outage are lost (rolled over or skipped), causing permanent financial harm to neuron holders.

### Likelihood Explanation
The `max_number_of_neurons` parameter in `NervousSystemParameters` is SNS-configurable and can be set to large values. The IC instruction limit for a single message is ~40 billion instructions. With 700 proposals and a few thousand neurons, the nested loop work is well within reach of triggering the limit, especially given the `HashMap` insertions, `Decimal` arithmetic, and neuron store lookups per ballot. Any SNS with active governance and a large neuron count is at risk. The attack requires only the ability to stake SNS tokens — an unprivileged, permissionless action.

### Recommendation
Apply the same batched-distribution pattern already used in NNS governance:

1. In `distribute_rewards`, compute `neuron_id_to_reward_shares` and store it in a persistent state machine (analogous to `RewardsDistributionStateMachine`).
2. Process the maturity-increment loop in a separate periodic timer task that checks `is_message_over_threshold` after each neuron and resumes in the next message if the limit is reached.
3. Add an instruction-limit check inside the ballot-accumulation loop (Loop 1) as well, since it also iterates over proposals × neurons.

### Proof of Concept

1. Deploy an SNS with `max_number_of_neurons` set to a large value (e.g., 50,000).
2. Create ~50,000 neurons by staking SNS tokens from many principals (permissionless).
3. Submit and let expire ~700 proposals so that `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS` unsettled proposals accumulate, each with ~50,000 ballot entries.
4. Wait for the next `run_periodic_tasks` invocation to call `distribute_rewards`.
5. The nested loop at lines 5894–5931 must process 700 × 50,000 = 35,000,000 ballot entries with `HashMap` insertions and `Decimal` arithmetic per entry, followed by 50,000 neuron store mutations in Loop 2 — exceeding the IC instruction limit.
6. The canister traps; every subsequent periodic task invocation traps at the same point; voting rewards are permanently halted.
