### Title
Unbacked ERC-20 Token Minting via Direct `ft_on_transfer` Call Without Token Transfer Verification - (`engine/src/contract_methods/connector.rs`)

### Summary

`ft_on_transfer` in the Aurora Engine trusts the NEAR `predecessor_account_id` as proof that a real NEP-141 token transfer occurred, and mints ERC-20 tokens on Aurora accordingly. Because `deploy_erc20_token` is callable by any unprivileged account, an attacker can register their own NEAR account as a NEP-141 token and then call `ft_on_transfer` directly — without ever transferring any tokens — to mint an arbitrary quantity of the corresponding ERC-20 token on Aurora.

### Finding Description

`ft_on_transfer` is the NEP-141 callback that Aurora Engine exposes so that token contracts can notify it after a real `ft_transfer_call`. The function's only guard is a comparison of `predecessor_account_id` against the stored connector account ID:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
``` [1](#0-0) 

When the caller is not the connector, `receive_erc20_tokens` is invoked with `token = predecessor_account_id`. It looks up the registered ERC-20 address for that NEP-141 account and calls `mint(recipient, amount)` on the ERC-20 contract — with the engine's own admin address as the caller, which passes the `onlyAdmin` check in `EvmErc20`:

```rust

let erc20_admin_address = current_address(current_account_id);
let result = self.call(
    &erc20_admin_address,
    &erc20_token,
    Wei::zero(),

There is no check that Aurora's actual NEP-141 balance increased by `amount` before the mint. The function trusts the `amount` field in the caller-supplied JSON args entirely.

### Impact Explanation

An attacker mints an unbounded quantity of ERC-20 tokens on Aurora that have no backing NEP-141 tokens held by the engine. Any user who later tries to exit those tokens back to NEAR via `withdrawToNear` will find that Aurora holds no real NEP-141 balance to transfer, causing the bridge to be insolvent for that token. If the attacker lists the token on an Aurora DEX and attracts liquidity before executing the mint, they can drain the liquidity pool by selling the unbacked tokens for real assets (ETH, USDC, etc.) and exiting those assets to NEAR.

**Impact class:** Critical — direct theft of user funds (DEX liquidity) and insolvency of the ERC-20 bridge for the affected token.

### Likelihood Explanation

The attack requires no special privileges:
- `deploy_erc20_token` is callable by any NEAR account.
- `ft_on_transfer` is a public NEAR contract method with no caller whitelist.
- The attacker only needs a NEAR account and gas.

### Recommendation

Before minting ERC-20 tokens in `receive_erc20_tokens`, record Aurora's NEP-141 balance for the token *before* the call and verify it increased by at least `args.amount` after the call (balance-check pattern). Alternatively, restrict `ft_on_transfer` so that it only accepts calls that arrive as a NEAR cross-contract callback from a registered NEP-141 contract (i.e., enforce that it is called as part of a `ft_transfer_call` promise chain, not as a standalone transaction).

### Proof of Concept

1. Attacker controls NEAR account `attacker.near`.
2. Attacker calls `deploy_erc20_token` on Aurora Engine with `nep141 = "attacker.near"`. Aurora deploys an ERC-20 contract and registers the mapping `attacker.near → ERC20_addr`. No access control prevents this.
3. Attacker calls `ft_on_transfer` on Aurora Engine directly from `attacker.near` with args:
   ```json
   {"sender_id": "attacker.near", "amount": "1000000000000", "msg": "<attacker_evm_address>"}
   ```
4. `ft_on_transfer` sees `predecessor_account_id = "attacker.near" ≠ connector_account_id`, so it calls `receive_erc20_tokens("attacker.near", args, ...)`.
5. `receive_erc20_tokens` resolves `ERC20_addr` from the registry and calls `mint(attacker_evm_address, 1000000000000)` via the engine's admin address — succeeding because the engine is the ERC-20 admin.
6. Attacker now holds 1,000,000,000,000 ERC-20 tokens on Aurora with zero NEP-141 tokens deposited. Attacker sells them on a DEX, draining real liquidity, then exits the proceeds to NEAR. [4](#0-3) [5](#0-4)

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
