### Title
Silent ckETH Loss When `erc20_tx_fee <= CKETH_LEDGER_TRANSACTION_FEE` and ckERC20 Burn Fails — (`rs/ethereum/cketh/minter/src/main.rs`)

### Summary

When a ckERC20 withdrawal fails at the ckERC20 burn step with `InsufficientFunds`, `AmountTooLow`, or `InsufficientAllowance`, the reimbursement amount is computed as `erc20_tx_fee - CKETH_LEDGER_TRANSACTION_FEE`. If this subtraction yields zero (or underflows), the guard `if reimbursed_amount > Wei::ZERO` suppresses the `FailedErc20WithdrawalRequest` event entirely. The ckETH already burned at the prior step is never returned, and no reimbursement request is ever queued.

### Finding Description

In `rs/ethereum/cketh/minter/src/main.rs`, the `withdraw_erc20` function first burns `erc20_tx_fee` ckETH from the caller: [1](#0-0) 

If that burn succeeds but the subsequent ckERC20 burn fails with a user-attributable error, the reimbursement is penalized by one ledger fee: [2](#0-1) 

The reimbursement event is only emitted when the result is strictly positive: [3](#0-2) 

When `erc20_tx_fee <= CKETH_LEDGER_TRANSACTION_FEE`, `checked_sub` returns `None` (underflow) or exactly `Wei::ZERO`, `unwrap_or(Wei::ZERO)` produces zero, the `if` branch is skipped, **no `FailedErc20WithdrawalRequest` event is recorded, no reimbursement request is queued**, and the ckETH burned at line 451 is permanently lost.

`erc20_tx_fee` is computed purely from the live gas oracle with no floor guard relative to `CKETH_LEDGER_TRANSACTION_FEE`: [4](#0-3) 

There is no pre-burn check that asserts `erc20_tx_fee > CKETH_LEDGER_TRANSACTION_FEE` before proceeding.

### Impact Explanation

Any user who calls `withdraw_erc20` when the gas oracle returns a fee estimate at or below `CKETH_LEDGER_TRANSACTION_FEE`, and whose ckERC20 burn then fails (e.g., insufficient ckERC20 balance), permanently loses the ckETH they burned as the gas fee. The loss is bounded by `erc20_tx_fee` (small when gas is cheap), but it is irrecoverable: no event is emitted, no reimbursement is ever minted, and the state machine has no recovery path for this case.

### Likelihood Explanation

The condition requires `gas_price * CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT <= CKETH_LEDGER_TRANSACTION_FEE`. On Ethereum mainnet this is rare but not impossible during extremely low-activity periods. The user-error precondition (insufficient ckERC20 balance) is common. The combination is low-probability but not theoretical — it is directly unit-testable with a mock gas oracle returning a sufficiently small value.

The integration test in `tests/ckerc20.rs` confirms the happy path where `transaction_fee > CKETH_TRANSFER_FEE` and a reimbursement is issued, but it does not test the boundary case where they are equal: [5](#0-4) 

### Recommendation

Add a pre-burn assertion or early return that ensures `erc20_tx_fee > CKETH_LEDGER_TRANSACTION_FEE` before burning ckETH. Alternatively, always emit `FailedErc20WithdrawalRequest` (with `reimbursed_amount = 0`) so the event log is complete, and handle zero-amount reimbursements explicitly rather than silently dropping them.
