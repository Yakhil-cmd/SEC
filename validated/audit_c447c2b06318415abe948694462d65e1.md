### Title
Permanent Freeze of User Funds When `ExitToNear` NEAR-Side Transfer Fails Without `error_refund` Feature - (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` Cargo feature is not compiled in, the `ExitToNear` precompile burns/deducts EVM-side tokens (ETH or ERC-20) before scheduling the NEAR-side `ft_transfer` or `ft_transfer_call` promise, but schedules **no callback** to refund the user if that promise fails. The result is a permanent, irrecoverable freeze of user funds at the `exit_to_near::ADDRESS` precompile address — an EVM address with no associated contract code and no admin recovery function.

---

### Finding Description

In `engine-precompiles/src/native.rs`, `ExitToNear::run` constructs `callback_args` as follows:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,                          // ← always None without the feature
    transfer_near: transfer_near_args,     // ← None for all non-wNEAR paths
};
``` [1](#0-0) 

When both `refund` and `transfer_near` are `None`, `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`, so the code takes the no-callback branch:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback scheduled
} else {
    PromiseArgs::Callback(...)
};
``` [2](#0-1) 

This affects **all non-wNEAR exit paths** when `error_refund` is absent:

- **Base ETH exit (legacy `ft_transfer`)**: `transfer_near_args = None`, `refund = None` → no callback.
- **Base ETH exit (Omni `ft_transfer_call`)**: `transfer_near_args = None`, `refund = None` → no callback.
- **ERC-20 exit (legacy `ft_transfer`)**: `transfer_near_args = None`, `refund = None` → no callback.
- **ERC-20 exit (Omni `ft_transfer_call`)**: `transfer_near_args = None`, `refund = None` → no callback. [3](#0-2) [4](#0-3) 

The EVM-side tokens are consumed **before** the NEAR-side promise executes:

- For ETH exit: ETH is deducted from the caller's EVM balance and credited to `exit_to_near::ADDRESS`.
- For ERC-20 exit: ERC-20 tokens are burned from the caller's balance.

If the NEAR-side `ft_transfer` or `ft_transfer_call` fails (e.g., unregistered account, insufficient storage deposit, rejected `ft_on_transfer`), there is no callback, no `refund_on_error` call, and no recovery path. The ETH sits permanently at `exit_to_near::ADDRESS` — a precompile address with no contract code and no admin rescue function. The ERC-20 tokens are simply gone (burned).

The only recovery path, `refund_on_error`, is gated behind the callback:

```rust
} else if let Some(args) = args.refund {
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
}
``` [5](#0-4) 

Without `error_refund`, `args.refund` is always `None`, so `refund_on_error` is never reached.

The codebase itself documents the fund-lock risk for the ERC-20 path:

> *"In case of withdrawing ERC-20 tokens, the `apparent_value` should be zero. In opposite way the funds will be locked in the address of the precompile without any possibility to withdraw them in the future."* [6](#0-5) 

That guard protects only the `apparent_value != 0` case. The failed-NEAR-transfer case is unguarded when `error_refund` is absent.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any user who calls `ExitToNear` targeting a NEAR account that fails the NEP-141 `ft_transfer` or `ft_transfer_call` permanently loses their EVM-side tokens. For ETH, the funds are frozen at `exit_to_near::ADDRESS` with no admin recovery function. For ERC-20, the tokens are burned with no re-mint. There is no protocol-level mechanism to recover them.

---

### Likelihood Explanation

**High.** The failure condition is easily triggered by any unprivileged EVM user:

- Specifying a NEAR account not registered with the NEP-141 contract (no storage deposit).
- Specifying a NEAR account not registered with the eth-connector.
- Using an Omni message whose `ft_on_transfer` handler rejects the call.

These are realistic user mistakes. The test suite explicitly confirms the loss:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [7](#0-6) 

---

### Recommendation

1. **Make the refund mechanism unconditional**: Remove the `#[cfg(feature = "error_refund")]` / `#[cfg(not(feature = "error_refund"))]` guards and always populate `callback_args.refund` with the appropriate `RefundCallArgs`. The callback is only scheduled when `callback_args != default()`, so the overhead is zero when no refund is needed.

2. **If the feature flag must remain**: Ensure the production WASM binary is always compiled with `error_refund` enabled, and document this as a hard requirement.

3. **Add a guard analogous to the `apparent_value` check**: Before burning ERC-20 tokens or deducting ETH, verify that the destination NEAR account is registered with the relevant NEP-141 contract, so the transaction fails atomically rather than burning tokens against a doomed promise.

---

### Proof of Concept

**Scenario: ERC-20 permanent freeze (no `error_refund`)**

1. User holds 100 units of a bridged ERC-20 token on Aurora.
2. User calls `withdrawToNear(bytes dest, uint256 amount)` on the ERC-20 contract, specifying `dest = "unregistered.near"` and `amount = 100`.
3. The ERC-20 contract burns 100 tokens and calls `ExitToNear` precompile (flag `0x01`).
4. `ExitToNear::run` constructs `callback_args { refund: None, transfer_near: None }` → `callback_args == default()` → schedules `PromiseArgs::Create(ft_transfer(...))` with no callback.
5. The NEAR runtime executes `ft_transfer` on the NEP-141 contract; it fails because `"unregistered.near"` has no storage deposit.
6. No callback fires. `refund_on_error` is never called.
7. The user's 100 ERC-20 tokens are permanently destroyed. The NEP-141 balance remains with Aurora's account. No recovery is possible. [8](#0-7) [9](#0-8)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
