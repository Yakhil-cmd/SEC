### Title
Negative `delta_gas` Not Credited Back to Sender — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

In `compute_gas_refund`, the dual-resource accounting adjustment (`delta_gas`) is applied asymmetrically: when native resource consumption implies *more* gas was used than ergs tracked, `gas_used` is increased and the user pays more. But when native resource consumption implies *less* gas was used than ergs tracked (`delta_gas < 0`), `gas_used` is **not** reduced. The user's "savings" from efficient native resource usage are silently retained by the operator rather than refunded to the sender. A developer `TODO` comment in the code explicitly flags this unresolved asymmetry.

### Finding Description

`compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs` implements ZKsync OS's double-resource accounting reconciliation:

```
delta_gas = (native_used / native_per_gas) - gas_used
``` [1](#0-0) 

Only the `delta_gas > 0` branch is handled — `gas_used` is increased when native consumption exceeds what ergs imply. The `delta_gas < 0` branch (native consumption is *less* than what ergs imply) is silently ignored, with a developer `TODO` comment acknowledging the gap:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [2](#0-1) 

The resulting `gas_used` is then used in `refund_and_commit_fee` to compute both the sender's refund and the operator's payment:

- Sender refund: `(gas_limit - gas_used) * gas_price`
- Operator payment: `gas_used * gas_price` [3](#0-2) [4](#0-3) 

When `delta_gas < 0`, `gas_used` is inflated relative to what native resource consumption justifies. The sender is refunded less than they are owed, and the operator receives more than they are owed. The difference — `|delta_gas| * gas_price` — is effectively transferred from the sender to the operator without justification.

The system documentation confirms only the positive case is intended:

> If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs. [5](#0-4) 

The negative case is not described, consistent with it being an unimplemented path.

### Impact Explanation

A sender whose transaction consumes fewer native resources than their EVM gas usage implies will be overcharged. The excess fee — `|delta_gas| * gas_price` — is paid to the operator instead of being refunded to the sender. This is a direct, quantifiable loss of funds for the sender, analogous to the original report where Party A's realized profits during liquidation were retained by the protocol instead of credited to Party A's balance.

### Likelihood Explanation

Any transaction that performs many EVM opcode operations (consuming ergs) while writing little to storage or producing little pubdata (consuming few native resources) will produce `delta_gas < 0`. This is a realistic and common transaction profile — for example, computation-heavy contracts that read from storage but write minimally. The condition is reachable by any unprivileged transaction sender without any special setup.

### Recommendation

When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (subject to the `minimal_gas_used` floor), so that the sender receives the full refund corresponding to their actual resource consumption:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else {
    // Credit back savings from lower-than-expected native consumption
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

### Proof of Concept

Consider a transaction with:
- `gas_limit = 100_000`
- `native_per_gas = 10`
- EVM execution consumes `80_000` gas (ergs imply `gas_used = 80_000`)
- Native resources consumed: `500_000` units → `native_used / native_per_gas = 50_000`

Then `delta_gas = 50_000 - 80_000 = -30_000`.

Current behavior: `gas_used` stays at `80_000`. Sender refund = `20_000 * gas_price`. Operator gets `80_000 * gas_price`.

Correct behavior: `gas_used` should be reduced to `50_000`. Sender refund = `50_000 * gas_price`. Operator gets `50_000 * gas_price`.

The sender loses `30_000 * gas_price` to the operator with no justification. This loss scales with `gas_price` and the magnitude of the negative delta, making it material for high-fee transactions. [1](#0-0)
