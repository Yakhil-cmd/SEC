Audit Report

## Title
Unbounded Neuron Scan in `maybe_finalize_disburse_maturity` Can Exhaust Instruction Limit and Freeze SNS Disbursements - (File: rs/sns/governance/src/governance.rs)

## Summary

The SNS governance canister's `maybe_finalize_disburse_maturity` function performs a full O(N) scan of every neuron in `self.proto.neurons` on every periodic task invocation, with no instruction-budget guard and no batching. If the neuron count grows large enough that this scan exhausts the per-message instruction limit, the IC runtime traps and rolls back all state, the timer fires again, and the scan traps again — creating a permanent livelock in which no maturity disbursement can ever be finalized. The NNS governance canister has already addressed this exact pattern with a secondary index and single-disbursement-per-invocation design.

## Finding Description

`maybe_finalize_disburse_maturity` is called unconditionally from `run_periodic_tasks` at line 5527 of `rs/sns/governance/src/governance.rs`. The function begins by iterating over the entire `self.proto.neurons` map at lines 4938–4975:

```rust
let neuron_id_and_disbursements: Vec<(NeuronId, DisburseMaturityInProgress)> = self
    .proto
    .neurons
    .values()
    .filter_map(|neuron| { ... })
    .collect();
```

This synchronous scan runs in the first message execution (before any `.await`). There is no call to `is_message_over_threshold` or any equivalent instruction-budget check anywhere in the SNS governance code. If the scan exhausts the 40B-instruction per-message limit, the IC runtime traps the entire message. Because the `is_finalizing_disburse_maturity = Some(true)` assignment at line 4935 is also rolled back on trap, the next timer invocation will attempt the same scan and trap again — a permanent livelock.

The disbursement loop at lines 4976–5081 then processes every collected disbursement in the same execution context, compounding the instruction cost before the first `.await` at line 5046.

The NNS governance canister avoids this by maintaining a secondary index (`get_neuron_ids_ready_to_finalize_maturity_disbursement`) and processing exactly one disbursement per timer invocation, making each execution O(1) in the number of neurons. No equivalent index or batching exists in the SNS governance.

## Impact Explanation

This is a **High** severity application-level DoS. When the instruction limit is exhausted, all pending maturity disbursements for the entire SNS are permanently frozen. Users who have already called `DisburseMaturity` — which deducts maturity from their neuron at call time and places it in `disburse_maturity_in_progress` — have no callable function available to them as unprivileged users to unblock the situation. Their funds remain locked in the pending queue indefinitely. This matches the allowed impact class: *Application/platform-level DoS with concrete user or protocol harm* and *Unauthorized access to neurons/governance assets where exploitation requires meaningful constraints*.

## Likelihood Explanation

Any SNS token holder can call `DisburseMaturity` to add entries to `disburse_maturity_in_progress`. The scan iterates over **all** neurons, not just those with pending disbursements, so the instruction cost scales with total neuron count regardless of how many have active disbursements. A popular SNS can organically accumulate tens of thousands of neurons. The `run_periodic_tasks` function also executes `process_proposals`, upgrade checks, reward distribution, and maturity modulation updates before reaching `maybe_finalize_disburse_maturity`, consuming a non-trivial portion of the instruction budget before the scan begins. The attack requires no special privileges and no victim mistakes — it is triggered by normal organic growth of the SNS.

## Recommendation

1. **Replace the full scan with a secondary index**, analogous to the NNS `get_neuron_ids_ready_to_finalize_maturity_disbursement` index, so only neurons with a ready disbursement are accessed.
2. **Process at most one disbursement per timer invocation**, as the NNS `FinalizeMaturityDisbursementsTask` does, making each execution O(1) in neuron count.
3. **Add an instruction-budget guard** using `is_message_over_threshold` (already used in NNS timer tasks such as `PruneFollowingTask`) so that if the scan does grow large, the function exits gracefully and reschedules rather than trapping.

## Proof of Concept

1. Deploy an SNS with a large number of neurons (e.g., 50,000+) using PocketIC or a local replica.
2. Have a significant fraction of those neurons call `DisburseMaturity`, populating `disburse_maturity_in_progress` entries.
3. Advance the replica time past the 7-day disbursement delay so all entries become eligible.
4. Trigger `run_periodic_tasks` (via the timer or a direct test call) and observe that `maybe_finalize_disburse_maturity` traps due to instruction exhaustion during the `self.proto.neurons.values().filter_map(...)` scan at lines 4938–4975.
5. Confirm via canister state inspection that `disburse_maturity_in_progress` queues remain permanently non-empty across repeated timer invocations, with no callable function available to an unprivileged user to unblock the situation.
