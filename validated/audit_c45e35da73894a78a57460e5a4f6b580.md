Audit Report

## Title
Weak RNG Seeding Enables Prediction of Rule UUIDs, Leaking Confidential Incident Metadata via Public Query - (File: `rs/boundary_node/rate_limits/canister/random.rs`)

## Summary
The rate-limit canister seeds its ChaCha20 RNG with only 8 bytes of publicly observable IC time plus 24 constant bytes (`42`), yielding a fully predictable UUID sequence for all `RuleId` values. Because `get_rule_by_id` is a non-replicated `#[query]` method that bypasses `inspect_message` and is callable by any anonymous principal, an attacker who predicts a `RuleId` can retrieve the `incident_id`, `added_in_version`, and `removed_in_version` of confidential (not-yet-disclosed) rules, directly breaking the canister's confidentiality model.

## Finding Description

**Root cause — `rs/boundary_node/rate_limits/canister/random.rs` lines 8–13:**

The `thread_local!` RNG is seeded with `seed[0..8] = time().to_le_bytes()` and `seed[8..32] = [42u8; 24]`. The IC canister time is recorded on-chain and is publicly readable. The remaining 192 bits are a known constant, giving the seed an effective entropy of 64 bits. [1](#0-0) 

This RNG is registered as the global `getrandom` provider via `getrandom::register_custom_getrandom!`. [2](#0-1) 

**UUID generation — `rs/boundary_node/rate_limits/canister/add_config.rs` lines 186–193:**

Every new `RuleId` is drawn from this weakly-seeded, deterministic RNG via `getrandom::getrandom`. [3](#0-2) 

**Public query access — `rs/boundary_node/rate_limits/canister/canister.rs` lines 123–133:**

`get_rule_by_id` is a non-replicated `#[query]` method. The `inspect_message` hook only gates `get_config` (replicated query), `add_config`, and `disclose_rules`. All other methods, including `get_rule_by_id` and `get_rules_by_incident_id`, are non-replicated queries that bypass `inspect_message` entirely and are callable by any anonymous principal. [4](#0-3) [5](#0-4) 

**Insufficient redaction — `rs/boundary_node/rate_limits/canister/confidentiality_formatting.rs` lines 31–43:**

`RuleConfidentialityFormatter` only redacts `rule_raw` and `description` for non-disclosed rules. The fields `incident_id`, `added_in_version`, `removed_in_version`, and `disclosed_at` are always returned in the response, even for confidential rules. [6](#0-5) 

This is confirmed by the `RuleGetter` implementation, which always populates all fields before applying the formatter. [7](#0-6) 

**Exploit chain:**
1. Read canister install/upgrade timestamp `T` from the IC blockchain.
2. Reconstruct seed: `seed[0..8] = T.to_le_bytes(); seed[8..32] = [42u8; 24]`.
3. Instantiate `ChaCha20Rng::from_seed(seed)` and generate the full UUID sequence.
4. For each predicted UUID, call `get_rule_by_id(<uuid>)` as an anonymous caller.
5. Responses for existing rules return `incident_id`, `added_in_version`, and `removed_in_version` — even for confidential, non-disclosed rules.
6. The first predicted UUID returning `NotFound` is the next `RuleId` to be assigned, enabling pre-registration attacks.

The `get_config` restriction (which would reveal rule counts) is not required for the attack: the attacker determines which predicted UUIDs are already assigned by probing `get_rule_by_id` directly.

## Impact Explanation

An unprivileged anonymous caller can enumerate the `incident_id` and lifecycle metadata of all confidential rate-limit rules before they are officially disclosed. The canister's confidentiality model is explicitly designed to keep active incident rules secret until the authorized principal chooses to disclose them. This attack breaks that model for metadata, constituting a significant boundary/API security impact with concrete harm to the confidentiality guarantees of the rate-limit canister — a production boundary node component. This matches the **High** impact category: significant boundary/API security impact with concrete user or protocol harm.

## Likelihood Explanation

The attack requires no special privileges. The IC canister initialization timestamp is publicly readable from the blockchain. The seed reconstruction is trivial (a few lines of Rust). The UUID enumeration requires only public query calls. The attack is fully deterministic and repeatable after any canister upgrade (which reseeds the RNG with the new upgrade timestamp, resetting the sequence). No social engineering, phishing, or threshold corruption is required.

## Recommendation

Replace the static time-seeded `thread_local!` RNG with per-call invocations of the IC management canister's `raw_rand` endpoint, as already done correctly in the salt-sharing canister: [1](#0-0) 

```rust
pub async fn generate_random_uuid() -> Result<Uuid, anyhow::Error> {
    let (rand_bytes,): ([u8; 16],) =
        ic_cdk::call(Principal::management_canister(), "raw_rand", ())
            .await
            .map_err(|e| anyhow::anyhow!("{:?}: {}", e.0, e.1))?;
    Ok(Uuid::from_slice(&rand_bytes)?)
}
```

Additionally, `incident_id` should be redacted for non-disclosed rules in `RuleConfidentialityFormatter` if it is considered confidential metadata. [8](#0-7) 

## Proof of Concept

1. Read the canister's install/upgrade timestamp `T` from the IC blockchain (e.g., via `dfx canister info` or the IC dashboard).
2. In Rust: `let mut seed = [42u8; 32]; seed[..8].copy_from_slice(&T.to_le_bytes()); let mut rng = ChaCha20Rng::from_seed(seed);`
3. Generate UUID candidates in a loop: `let mut buf = [0u8; 16]; rng.fill_bytes(&mut buf); let uuid = Uuid::from_slice(&buf).unwrap();`
4. For each UUID, call `get_rule_by_id(<uuid>)` as an anonymous caller.
5. Responses returning data (not `NotFound`) will include `incident_id`, `added_in_version`, and `removed_in_version` for confidential rules — confirming the information disclosure.
6. A deterministic integration test using PocketIC can reproduce this by: installing the canister, recording the init time, adding a confidential rule, reconstructing the seed, predicting the UUID, and asserting that `get_rule_by_id` returns the `incident_id` to an anonymous caller.
