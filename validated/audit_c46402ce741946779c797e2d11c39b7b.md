Audit Report

## Title
Spawned NNS Neuron Unconditionally Inherits Parent's Hot Keys When Assigned a New Controller — (`rs/nns/governance/src/governance.rs`)

## Summary
When `spawn_neuron` is called with a `new_controller` that differs from the parent neuron's controller, the child neuron is created with the new controller but unconditionally inherits the parent's `hot_keys` via `.with_hot_keys(parent_neuron.hot_keys.clone())`. These hot keys were authorized by the original controller and remain active on a neuron now controlled by a different principal, granting the original controller's delegates the ability to vote and change followees on the new controller's neuron without their knowledge or consent.

## Finding Description
In `spawn_neuron`, the child controller is resolved at L2651–2655: if `spawn.new_controller` is provided, `child_controller` is set to that new principal; otherwise it defaults to the parent's controller. [1](#0-0) 

The child neuron is then constructed at L2704–2718 with `child_controller` as its controller, but `.with_hot_keys(parent_neuron.hot_keys.clone())` is called unconditionally — no check is made whether `child_controller == parent_neuron.controller()`. [2](#0-1) 

By contrast, `disburse_to_neuron` at L3012–3021 creates the child neuron without copying hot keys at all, and resets followees to system defaults — demonstrating that the codebase handles ownership transfer concerns in at least one analogous path. [3](#0-2) 

The `follow` function at L5725–5748 authorizes hot keys to change followees for all topics except `NeuronManagement`, using `is_caller_authorized_to_vote` which delegates to `is_hotkey_or_controller`. [4](#0-3) 

`is_hotkey_or_controller` at L254–256 returns `true` for any principal in `hot_keys`, regardless of who set those keys or who currently controls the neuron. [5](#0-4) 

Exploit flow:
1. Alice controls neuron N with `hot_keys = [mallory]` and sufficient maturity.
2. Alice calls `Spawn { new_controller: Some(bob), ... }`.
3. Child neuron C is created: `controller = bob`, `hot_keys = [mallory]`.
4. Mallory calls `Follow` on C for any non-`NeuronManagement` topic — succeeds because `is_authorized_to_vote` returns `true`.
5. Mallory calls `RegisterVote` on C — succeeds for the same reason.
6. Bob's neuron votes and follows Mallory's direction without Bob's knowledge or consent.

No existing guard prevents this. The only remediation available to Bob is to proactively inspect the neuron's hot key list and call `RemoveHotKey` — a step most users will not take.

## Impact Explanation
This constitutes unauthorized access to a governance neuron: principals authorized by the previous controller retain the ability to vote on NNS proposals and redirect followees on a neuron now controlled by a different principal. This matches the allowed High impact: **"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds."** The voting power of the child neuron can be silently exercised by the spawner's delegates, undermining the integrity of NNS governance for the affected neuron.

## Likelihood Explanation
`spawn_neuron` with `new_controller` is a standard, publicly reachable NNS governance operation callable by any neuron controller via ingress — no special privileges are required. Realistic scenarios include protocols distributing maturity-derived neurons to users, vesting arrangements where a custodian spawns neurons for beneficiaries, and users gifting spawned neurons. The attack is repeatable, low-cost, and requires no victim action beyond accepting the spawned neuron.

## Recommendation
In `spawn_neuron`, only inherit `hot_keys` when the child controller is the same as the parent controller. When a different `new_controller` is provided, initialize the child neuron with an empty hot key list, mirroring the behavior of `disburse_to_neuron`:

```rust
let child_hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};
// Then use .with_hot_keys(child_hot_keys) in the NeuronBuilder
```

## Proof of Concept
1. Create a test neuron N with `controller = alice`, `hot_keys = [mallory]`, and sufficient maturity.
2. Call `spawn_neuron` with `spawn.new_controller = Some(bob)`.
3. Assert child neuron C has `controller = bob` and `hot_keys = [mallory]`.
4. As Mallory, call `follow` on C for topic `NetworkEconomics` — assert it succeeds.
5. As Mallory, call `register_vote` on C for an open proposal — assert it succeeds.
6. Confirm Bob never authorized Mallory and has no notification of the hot key presence.

This can be implemented as a deterministic unit test in `rs/nns/governance/src/governance.rs` using the existing test infrastructure without any mainnet interaction.
