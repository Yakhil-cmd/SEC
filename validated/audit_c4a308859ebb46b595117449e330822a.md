### Title
Unchecked Precompile Call Return Value After Token Burn Causes Permanent Fund Loss - (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20` and `EvmErc20V2` implement `withdrawToNear` and `withdrawToEthereum` by first burning the caller's tokens via `_burn`, then invoking the Aurora exit precompile via an inline `assembly` `call`. The return value of that `call` is captured in a local variable `res` but is **never checked**. If the precompile call fails for any reason, execution continues without reverting, leaving the user's tokens permanently burned with no corresponding cross-chain transfer initiated.

---

### Finding Description

In both `EvmErc20.sol` and `EvmErc20V2.sol`, the withdrawal functions follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // (1) tokens burned — irreversible

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // (2) res is NEVER checked — silent failure is possible
    }
}
``` [1](#0-0) [2](#0-1) 

The same pattern appears in `withdrawToEthereum`: [3](#0-2) [4](#0-3) 

The EVM `call` opcode returns `0` on failure and `1` on success. The Solidity assembly block assigns the result to `res` but never branches on it (no `if iszero(res) { revert(0,0) }`). Because `_burn` executes unconditionally before the precompile call, any failure of the precompile leaves the user's ERC-20 balance permanently destroyed with no cross-chain transfer queued.

Conditions under which the precompile call can return `0` (fail silently) include:

- The `recipient` bytes do not form a valid NEAR account ID (the precompile returns `ExitError` which the EVM translates to a failed `call`).
- The precompile runs out of the gas forwarded to it.
- The exit precompile address is paused or the connector account is not set in storage.
- Any other internal precompile error path that returns `Err(ExitError::...)`.

In all these cases, `_burn` has already executed and cannot be undone. The user's tokens are gone.

---

### Impact Explanation

**Permanent freezing / destruction of user funds.**

The user's ERC-20 tokens are burned on the Aurora EVM side. Because the precompile call failed silently, no NEAR-side `ft_transfer` or `withdraw` promise is ever scheduled. The tokens cannot be recovered: there is no re-entry point, no refund mechanism, and the burned supply is gone. This matches the "permanent freezing of funds" critical impact category.

---

### Likelihood Explanation

**Medium.** The most common trigger is a user supplying a malformed or too-long `recipient` string. The `ExitToNear` precompile explicitly validates the recipient as a NEAR account ID and returns `ERR_INVALID_RECEIVER_ACCOUNT_ID` on failure: [5](#0-4) 

Any such validation failure causes the `call` to return `0`. Because the Solidity contracts do not check `res`, the burn is not reverted. A user who makes a typo in the recipient account ID, or passes a recipient that is too long, will silently lose all tokens they attempted to withdraw. This is a realistic, user-triggered scenario requiring no special privileges.

---

### Recommendation

Add an explicit revert on precompile call failure inside the assembly block in both `EvmErc20.sol` and `EvmErc20V2.sol`, for all four withdrawal functions:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the precompile call fails for any reason, the entire transaction (including the `_burn`) is reverted, preserving the user's balance.

---

### Proof of Concept

1. Deploy `EvmErc20` with any underlying asset.
2. Mint `1e18` tokens to Alice.
3. Alice calls `withdrawToNear(invalidRecipient, 1e18)` where `invalidRecipient` is not a valid NEAR account ID (e.g., `"!!!"` or an empty byte string).
4. `_burn(alice, 1e18)` executes — Alice's balance drops to `0`.
5. The `call` to the `ExitToNear` precompile at `0xe9217bc7...` fails because `parse_recipient` returns `ERR_INVALID_RECEIVER_ACCOUNT_ID`; the `call` returns `0`.
6. `res` is never checked; the function returns normally.
7. Alice's `1e18` tokens are permanently destroyed. No NEAR-side transfer was initiated. Alice cannot recover her funds.

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
