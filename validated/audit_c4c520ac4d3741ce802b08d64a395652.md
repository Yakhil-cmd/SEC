### Title
Partial `ft_transfer_call` Consumption in Omni Exit Flow Permanently Burns User ERC-20 Tokens Without Refund - (`engine/src/contract_methods/connector.rs`)

### Summary

When a user exits ERC-20 (or base ETH) tokens to NEAR using the Omni message variant of the `ExitToNear` precompile, the engine schedules a `ft_transfer_call` on the NEP-141 contract. Under the NEP-141 standard, the receiver's `ft_on_transfer` may return a non-zero unused amount (partial consumption). The NEP-141 contract then returns those unused tokens to Aurora's own account. However, `exit_to_near_precompile_callback` treats any `PromiseResult::Successful` as a full success and never re-mints the corresponding ERC-20 tokens for the user. The user's ERC-20 tokens are permanently burned while the unspent NEP-141 tokens accumulate in Aurora's account with no recovery path.

### Finding Description

**Step 1 — ERC-20 burn and Omni exit scheduling**

In `engine-precompiles/src/native.rs`, `exit_erc20_token_to_near` handles the `Message::Omni(msg)` branch by selecting `ft_transfer_call` as the method and setting `transfer_near_args = None`: [1](#0-0) 

**Step 2 — Callback construction**

Back in the `ExitToNear::run` method, `callback_args` is built. With the `error_refund` feature enabled, `refund` is populated; `transfer_near` is `None` for the Omni ERC-20 case. A `PromiseWithCallbackArgs` is created whose callback is `exit_to_near_precompile_callback`: [2](#0-1) 

**Step 3 — Callback logic ignores partial return**

`exit_to_near_precompile_callback` branches on whether the promise result is `Successful` or not. On success it only handles the `transfer_near` case (wNEAR unwrap). For the Omni ERC-20 path, `transfer_near` is `None`, so the success branch does **nothing**: [3](#0-2) 

**The gap:** The NEP-141 `ft_transfer_call` standard allows the receiver's `ft_on_transfer` to return a non-zero unused amount. The NEP-141 contract's internal `ft_resolve_transfer` then credits those unused tokens back to Aurora's own NEP-141 balance. The promise still resolves as `PromiseResult::Successful`. The callback receives this success result, does nothing, and returns. The ERC-20 tokens were already burned before the promise was dispatched; there is no code path that re-mints them for the partial unused amount.

**Step 4 — No recovery mechanism**

`refund_on_error` only fires when the promise result is not `Successful`: [4](#0-3) 

There is no analogous `refund_on_partial_success` path. The unused NEP-141 tokens sit in Aurora's account on the NEP-141 contract with no corresponding ERC-20 supply, and no admin or user method exists to recover them as ERC-20 tokens.

### Impact Explanation

A user burns N ERC-20 tokens. The Omni receiver consumes only K < N NEP-141 tokens and returns N−K. The NEP-141 contract credits N−K back to Aurora's account. The user receives value for only K tokens. The N−K NEP-141 tokens are permanently stranded in Aurora's NEP-141 balance with no ERC-20 counterpart and no recovery path. This constitutes **permanent freezing / permanent loss of user funds** (Critical).

### Likelihood Explanation

The Omni exit path is a documented, production-reachable feature. Any unprivileged EVM user can call the ERC-20's `withdrawToNear` function with an Omni-formatted message targeting any NEAR contract that implements `ft_on_transfer`. A receiver that returns a non-zero unused amount (e.g., a DEX router, a lending protocol, or a deliberately crafted contract) triggers the bug. No special privileges are required.

### Recommendation

In `exit_to_near_precompile_callback`, when the promise result is `Successful`, decode the actual amount transferred from the promise result bytes (the NEP-141 `ft_transfer_call` return value is the amount actually used). Compute `unused = requested_amount − actually_used`. If `unused > 0`, call `refund_on_error` (or an equivalent re-mint path) for the unused portion, exactly as is done for the full-failure case. Alternatively, store the requested amount in `ExitToNearPrecompileCallbackArgs` alongside the refund args so the callback has all information needed to issue a partial re-mint.

### Proof of Concept

1. Deploy a NEP-141 token; bridge it to Aurora as an ERC-20.
2. Deploy a NEAR contract `partial_consumer` whose `ft_on_transfer` always returns `amount / 2` (consumes only half).
3. From an EVM account holding 100 ERC-20 units, call `withdrawToNear` with an Omni message targeting `partial_consumer`.
4. Observe: ERC-20 balance drops by 100; `partial_consumer` receives 50 NEP-141 tokens; Aurora's NEP-141 balance increases by 50; the callback fires with `PromiseResult::Successful`; no ERC-20 re-mint occurs.
5. The user has lost 50 ERC-20 tokens permanently. Aurora's NEP-141 balance holds 50 unbacked tokens. [1](#0-0) [5](#0-4)
