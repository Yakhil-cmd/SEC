### Title
Enabling Silo Mode Address Whitelist After ERC-20 Deposits Permanently Freezes Existing Token Holders' Funds - (`engine/src/engine.rs`)

### Summary

When the Aurora Engine is operated without silo mode and users deposit NEP-141 tokens (minting ERC-20 tokens to their addresses), subsequently enabling silo mode with the `Address` whitelist causes those existing token holders to be permanently unable to submit EVM transactions. Since the only ERC-20 withdrawal path (`exit_to_near` precompile) requires submitting an EVM transaction, their tokens are permanently frozen.

### Finding Description

The `receive_erc20_tokens` function in `engine/src/engine.rs` mints ERC-20 tokens to the intended recipient address unconditionally when `erc20_fallback_address` is not set (i.e., silo mode is off):

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

When `erc20_fallback_address` is absent, the `if let Some(...)` guard fails and the redirect is skipped entirely — tokens are minted to the caller-specified recipient regardless of whitelist state.

The owner can later call `set_silo_params` to enable silo mode: [2](#0-1) 

which sets `fixed_gas` and `erc20_fallback_address` in storage: [3](#0-2) 

The owner also enables the `Address` whitelist via `set_whitelist_status`: [4](#0-3) 

After this, `is_allow_submit` — which gates all EVM transaction submission — checks the `Address` whitelist: [5](#0-4) 

```rust
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}
``` [6](#0-5) 

```rust
fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
    !list.is_enabled() || list.is_exist(address)
}
```

Addresses that received ERC-20 tokens while silo mode was off are not in the whitelist. After silo mode is enabled, they cannot submit any EVM transaction, including calls to the `exit_to_near` precompile — the only mechanism to redeem ERC-20 tokens back to NEP-141. There is no alternative NEAR-level withdrawal path for ERC-20 tokens. The `withdraw` function in the connector handles only base ETH tokens, not ERC-20 tokens. [7](#0-6) 

The state inconsistency is structural: ERC-20 balances are written to storage under the "no whitelist" regime, but the only operations that can move those balances require passing the whitelist check that did not exist when the balances were created.

### Impact Explanation

**Critical — Permanent freezing of funds.**

All ERC-20 token holders whose addresses are not explicitly added to the `Address` whitelist before or after silo mode is enabled have their entire ERC-20 balance permanently frozen. There is no recovery path: the `exit_to_near` precompile requires an EVM `submit` call, which is blocked by the whitelist. The ERC-20 contract's `transfer` function is also an EVM call and is equally blocked. The underlying NEP-141 tokens remain locked in the Aurora Engine contract with no reachable burn/release path for those holders.

### Likelihood Explanation

**Medium.** The scenario requires the contract owner to:
1. Initially deploy without silo mode (the default state — whitelists are disabled by default per `is_enabled()` returning `false` when the key is absent).
2. Allow users to deposit NEP-141 tokens during this period.
3. Later enable silo mode with the `Address` whitelist.

This is a realistic operational sequence: a silo operator may start in open mode to bootstrap liquidity, then later restrict access. The code provides no warning, guard, or migration path for this transition. The owner need not be malicious — they may simply be unaware that enabling the whitelist retroactively freezes existing holders. [8](#0-7) 

### Recommendation

1. **Guard the whitelist enable transition**: Before enabling the `Address` whitelist, verify that no ERC-20 token balances exist at addresses not present in the whitelist, or emit a clear warning.
2. **Provide a bypass for `exit_to_near`**: Allow non-whitelisted addresses to call `exit_to_near` even when the `Address` whitelist is enabled, so existing holders can always redeem their tokens.
3. **Atomic migration**: When enabling silo mode, require the caller to supply the full set of existing ERC-20 holders to be added to the whitelist atomically in the same transaction.

### Proof of Concept

```
1. Deploy Aurora Engine (silo mode off, Address whitelist disabled by default).

2. User A calls ft_transfer_call on NEP-141 token contract, routing to Aurora's
   ft_on_transfer. receive_erc20_tokens() is called:
     - get_erc20_fallback_address() returns None  (silo mode off)
     - The `if let Some(fallback_address)` guard fails
     - ERC-20 tokens are minted to User A's address (e.g. 0xAAAA...)

3. Owner calls set_silo_params(Some(SiloParamsArgs { fixed_gas, erc20_fallback_address: 0xFALL... }))
   Owner calls set_whitelist_status({ kind: Address, active: true })
   User A's address 0xAAAA... is NOT added to the whitelist.

4. User A attempts to submit an EVM transaction (e.g. approve + exit_to_near):
     submit_transaction(&user_a_key, tx) → Err(NotAllowed)
   (Confirmed by test test_submit_access_right in engine-tests/src/tests/silo.rs:462-513)

5. User A's ERC-20 balance at 0xAAAA... is permanently inaccessible.
   The NEP-141 tokens backing those ERC-20 tokens remain locked in the Aurora contract.
``` [9](#0-8) [10](#0-9) [11](#0-10)
