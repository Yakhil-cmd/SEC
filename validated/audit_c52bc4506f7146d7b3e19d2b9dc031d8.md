Audit Report

## Title
Byzantine Block Proposer Can Inject Cross-Type Bitcoin Adapter Response, Corrupting ckBTC GetSuccessors Callback — (`rs/replicated_state/src/bitcoin.rs`, `rs/bitcoin/consensus/src/payload_builder.rs`)

## Summary

`push_response` in `rs/replicated_state/src/bitcoin.rs` validates the `callback_id` against the correct context map only for the `GetSuccessorsResponse` branch. The `SendTransactionResponse`, `GetSuccessorsReject`, and `SendTransactionReject` branches push to `consensus_queue` unconditionally with any `callback_id`. Combined with a `validate_payload` implementation that performs only a size and decode check (no semantic validation against state), a Byzantine block proposer can include a `SendTransactionResponse` carrying a `callback_id` belonging to a `bitcoin_get_successors_contexts` entry, causing the ckBTC canister's `GetSuccessors` callback to be consumed with an `EmptyBlob` payload and permanently losing the real `GetSuccessors` response for that `callback_id`.

## Finding Description

**Gap 1 — Asymmetric context validation in `push_response`.**

The `GetSuccessorsResponse` branch guards against mismatched `callback_id` by looking up the context:

```rust
let context = state
    .metadata
    .subnet_call_context_manager
    .bitcoin_get_successors_contexts
    .get_mut(&callback_id)
    .ok_or_else(|| StateError::BitcoinNonMatchingResponse { ... })?;
``` [1](#0-0) 

The `SendTransactionResponse` branch performs no such lookup and pushes unconditionally:

```rust
BitcoinAdapterResponseWrapper::SendTransactionResponse(_) => {
    let callback_id = CallbackId::from(response.callback_id);
    let payload = Payload::Data(EmptyBlob.encode());
    state.consensus_queue.push(ConsensusResponse::new(callback_id, payload));
    Ok(())
}
``` [2](#0-1) 

`GetSuccessorsReject` and `SendTransactionReject` are identical in this regard. [3](#0-2) 

**Gap 2 — `validate_self_validating_payload_impl` is a no-op.**

The entire validation body reduces to an empty-check and a byte count. It never reads state, never checks whether any `callback_id` in the payload exists in either context map, and never checks whether the response variant matches the pending request type:

```rust
fn validate_self_validating_payload_impl(...) -> Result<NumBytes, ...> {
    if *payload == SelfValidatingPayload::default() {
        return Ok(0.into());
    }
    self.metrics.observe_validate_duration(VALIDATION_STATUS_VALID, since);
    Ok(NumBytes::new(payload.count_bytes() as u64))
}
``` [4](#0-3) 

`validate_payload` (the `BatchPayloadBuilder` entry point called by consensus during block validation) delegates entirely to this function plus a size check, so honest replicas accept any well-formed protobuf Bitcoin payload regardless of semantic correctness. [5](#0-4) 

**Exploit flow:**

A Byzantine proposer includes `SendTransactionResponse{callback_id: X}` where `X ∈ bitcoin_get_successors_contexts`:

1. `push_response` pushes `ConsensusResponse{callback_id: X, payload: EmptyBlob}` to `consensus_queue` — no error returned.
2. The execution layer processes the response, calls `retrieve_context(X)` which **removes** X from `bitcoin_get_successors_contexts`. [6](#0-5) 
3. The ckBTC canister's `GetSuccessors` callback at X is delivered `EmptyBlob`; the canister traps attempting to decode it as `BitcoinGetSuccessorsResponse`.
4. Context X is now gone from `bitcoin_get_successors_contexts`.
5. When the next honest proposer includes the legitimate `GetSuccessorsResponse{callback_id: X}`, `push_response` looks up X in `bitcoin_get_successors_contexts`, finds nothing, and returns `StateError::BitcoinNonMatchingResponse` — the real response is rejected. [7](#0-6) 

Net result: the ckBTC canister's `GetSuccessors` callback for X is permanently consumed with wrong data; the real response is irrecoverably lost for that `callback_id`. Bitcoin chain-tip synchronization stalls until the canister issues a new request.

## Impact Explanation

This is a **High** severity finding. A Byzantine block proposer can deterministically disrupt ckBTC Bitcoin chain-tip synchronization by poisoning any in-flight `GetSuccessors` callback. This constitutes a concrete application/platform-level DoS against a Chain Fusion / ck-token component with direct user and protocol harm (ckBTC minting/redemption depends on Bitcoin chain-tip tracking). It maps to: *High ($2,000–$10,000) — Significant Chain Fusion, ck-token security impact with concrete user or protocol harm* and *High — Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS*.

## Likelihood Explanation

The attack requires exactly one Byzantine replica to win a block-proposal slot, which is within the BFT fault tolerance threshold (`f < n/3`) and explicitly within scope as a protocol peer below the consensus fault threshold. No key material, governance majority, or external oracle is needed. The attack is deterministic: the proposer simply crafts a `BitcoinAdapterResponse` protobuf with the wrong response variant and a known `callback_id` (observable from the certified state). The `validate_payload` guard that should catch this is provably absent. The attack is repeatable on every block-proposal slot won by the Byzantine replica.

## Recommendation

1. **In `push_response`**: Add context-existence checks to all three unguarded branches, mirroring the `GetSuccessorsResponse` branch. For `SendTransactionResponse`/`SendTransactionReject`, look up `callback_id` in `bitcoin_send_transaction_internal_contexts` and return `StateError::BitcoinNonMatchingResponse` if absent. For `GetSuccessorsReject`, look up in `bitcoin_get_successors_contexts`. [8](#0-7) 

2. **In `validate_self_validating_payload_impl`**: Load the certified state at `validation_context.certified_height` and verify that each `callback_id` in the payload exists in the correct context map for its response variant. Reject with `InvalidArtifact` on mismatch. [4](#0-3) 

## Proof of Concept

```rust
// Unit test (no network, no keys, single replica)
let mut state = ReplicatedStateBuilder::new().build();
let callback_id = state
    .metadata
    .subnet_call_context_manager
    .push_context(SubnetCallContext::BitcoinGetSuccessors(
        BitcoinGetSuccessorsContext { /* ... */ }
    ));  // stored in bitcoin_get_successors_contexts

// Byzantine proposer crafts a SendTransactionResponse with the GetSuccessors callback_id
let response = BitcoinAdapterResponse {
    response: BitcoinAdapterResponseWrapper::SendTransactionResponse(
        SendTransactionResponse {}
    ),
    callback_id: callback_id.get(),
};

let result = push_response(&mut state, response);
assert!(result.is_ok());           // passes — no error
assert_eq!(state.consensus_queue.len(), 1);  // EmptyBlob pushed for GetSuccessors callback

// After execution processes the queue, bitcoin_get_successors_contexts entry is removed.
// The real GetSuccessorsResponse for the same callback_id will now be rejected
// with StateError::BitcoinNonMatchingResponse, permanently losing the response.
```
