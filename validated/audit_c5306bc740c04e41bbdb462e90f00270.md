Audit Report

## Title
Unbounded Attacker-Controlled Ingress Window Causes OOM / Infinite Loop DoS in ICP Rosetta `construction_payloads` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary

The ICP Rosetta `construction_payloads` handler accepts caller-supplied `ingress_start` and `ingress_end` values from the HTTP request metadata and uses them directly in an unbounded while loop with no validation of the window size. Setting `ingress_start=0` and `ingress_end=u64::MAX` causes the loop to execute approximately 153 billion iterations before exhausting all available memory (OOM crash), or to loop infinitely due to a truncating `as u64` cast in `Time::from_duration` that wraps `now` back to a small value once it approaches `u64::MAX`. Either outcome crashes or permanently hangs the Rosetta node process, denying service to all users of that instance.

## Finding Description

**Root cause — no window-size bound:**

`construction_payloads` computes a fixed step interval:

```
interval = MAX_INGRESS_TTL(300s) − PERMITTED_DRIFT(60s) − 120s = 120s = 120_000_000_000 ns
```
(`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs` L59-60)

It then reads `ingress_start` and `ingress_end` directly from the caller-supplied metadata with no validation of the window size:

```rust
let ingress_start = meta.as_ref().and_then(|meta| meta.ingress_start)
    .map(ic_types::time::Time::from_nanos_since_unix_epoch)
    .unwrap_or_else(ic_types::time::current_time);

let ingress_end = meta.as_ref().and_then(|meta| meta.ingress_end)
    .map(ic_types::time::Time::from_nanos_since_unix_epoch)
    .unwrap_or_else(|| ingress_start + interval);
```
(L74-84)

The loop that follows has no guard on the window size:

```rust
let mut ingress_expiries = vec![];
let mut now = ingress_start;
while now < ingress_end {
    let ingress_expiry = (now + ic_limits::MAX_INGRESS_TTL
        .saturating_sub(ic_limits::PERMITTED_DRIFT))
        .as_nanos_since_unix_epoch();
    ingress_expiries.push(ingress_expiry);
    now += interval;
}
```
(L99-107)

**OOM path:** With `ingress_start=0` and `ingress_end=u64::MAX`, the loop runs `u64::MAX / 120_000_000_000 ≈ 153 billion` iterations, each pushing one `u64` (8 bytes) into `ingress_expiries`. Total allocation ≈ 1.2 TB. The process OOMs and crashes well before the wrap point.

**Infinite-loop path:** `now += interval` calls `Time::AddAssign<Duration>`:

```rust
impl std::ops::AddAssign<Duration> for Time {
    fn add_assign(&mut self, other: Duration) {
        *self = Time::from_duration(Duration::from_nanos(self.0) + other)
    }
}
```
(`rs/types/types/src/time.rs` L55-58)

which calls:

```rust
fn from_duration(t: Duration) -> Self {
    Time(t.as_nanos() as u64)
}
```
(L103-105)

`Duration::as_nanos()` returns `u128`. When `now.0` is near `u64::MAX`, the `Duration` addition succeeds (Duration stores seconds as `u64`, far exceeding `u64::MAX` nanoseconds), but the `as u64` cast silently truncates the `u128` result, wrapping `now` back to approximately `119_999_999_999` ns — still less than `ingress_end = u64::MAX` — making the loop infinite.

**Existing checks are absent:** There is no check anywhere in the ICP Rosetta handler that bounds `ingress_end - ingress_start`. The ICRC1 Rosetta variant (`rs/rosetta-api/icrc1/src/construction_api/services.rs` L148-158) has partial guards (`ingress_start >= ingress_end` and stale-end rejection) but also has no upper bound on the window size, making it independently vulnerable to the same OOM class of attack.

## Impact Explanation

A single unauthenticated HTTP POST to `/construction/payloads` with a crafted JSON body exhausts all available memory of the Rosetta node process, causing it to crash (or hang indefinitely). All ICP transfers routed through that node are blocked for the duration of the outage. If the node is shared infrastructure (e.g., an exchange's Rosetta instance), all users of that instance are affected. This matches the allowed ICP bounty impact: **High ($2,000–$10,000) — Application/platform-level DoS, crash, or significant Rosetta/infrastructure security impact with concrete user or protocol harm.**

## Likelihood Explanation

The Rosetta HTTP API is intentionally exposed to clients. No authentication is required to call `/construction/payloads`. The exploit requires a single HTTP request with a crafted JSON body — no privileged access, no key material, no network-level attack. The only precondition is network reachability to the Rosetta port. The attack is trivially repeatable: after the node restarts, a single new request re-triggers the crash.

## Recommendation

1. **Bound the ingress window before entering the loop.** Add a validation check immediately after computing `ingress_start` and `ingress_end`:

```rust
let max_window = ic_limits::MAX_INGRESS_TTL * 2; // or any reasonable small multiple
if ingress_end > ingress_start + max_window {
    return Err(ApiError::invalid_request(
        "ingress_end - ingress_start exceeds maximum allowed window",
    ));
}
```

Apply the equivalent guard to the ICRC1 Rosetta variant in `rs/rosetta-api/icrc1/src/construction_api/services.rs`.

2. **Fix the truncating cast in `Time::from_duration`.** Replace the silent `as u64` with a checked or saturating conversion to prevent silent wrap-around:

```rust
fn from_duration(t: Duration) -> Self {
    Time(u64::try_from(t.as_nanos()).unwrap_or(u64::MAX))
}
```
