### Title
Unchecked Precompile Return Value in `withdrawToNear` Causes Permanent Token Loss When `Erc20Nep141Map` Entry Is Absent - (`engine-precompiles/src/native.rs`, `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

The `withdrawToNear` function in `EvmErc20.sol` burns ERC-20 tokens **before** calling the `ExitToNear` precompile, and critically **never checks the return value** of the `call` opcode. If the precompile fails (e.g., because `get_nep141_from_erc20` returns `ERR_TARGET_TOKEN_NOT_FOUND`), the EVM treats the precompile invocation as a failed subcall, reverts only the precompile's frame, and returns execution to the ERC-20 contract — which continues and returns successfully. The burn is committed, no NEAR promise is scheduled, and the tokens are permanently destroyed.

---

### Finding Description

**Step 1 — Burn before precompile call, unchecked return value.**

In `EvmErc20.sol`, `withdrawToNear` executes `_burn` first, then calls the precompile via inline assembly without checking `res`: [1](#0-0) 

The variable `res` is assigned but never inspected. Under EVM semantics, a failed `call` opcode returns `0` to the caller; if the caller does not check this and revert, its own state changes (including the burn) are committed.

**Step 2 — Precompile failure path.**

`ExitToNear::run` delegates to `exit_erc20_token_to_near`, which calls `get_nep141_from_erc20`: [2](#0-1) 

`get_nep141_from_erc20` reads `KeyPrefix::Erc20Nep141Map` from storage. If the key is absent, it returns `Err(ExitError::Other("Target token not found"))`: [3](#0-2) 

**Step 3 — Error type mapping.**

`process_precompile` maps any `ExitError` to `PrecompileFailure::Error`: [4](#0-3) 

`PrecompileFailure::Error` causes the EVM to treat the precompile invocation as a failed subcall (returns `0` to caller), **not** as a fatal error that aborts the whole transaction. The ERC-20 contract's call frame is unaffected.

**Step 4 — Burn is committed, no promise scheduled.**

Because `withdrawToNear` does not check `res` and does not revert, the `_burn` executed at line 54 is committed to EVM state. No `PromiseArgs` log is emitted, so no `ft_transfer` or `ft_transfer_call` promise is ever scheduled on NEAR. The burned amount is irrecoverable.

---

### Impact Explanation

Tokens burned in `_burn` are permanently destroyed with no corresponding NEP-141 transfer on NEAR. There is no recovery path: the ERC-20 supply is reduced, the user's balance is zero, and no NEAR-side tokens are released. This is **permanent freezing/destruction of funds**.

---

### Likelihood Explanation

The trigger requires the `Erc20Nep141Map` entry to be absent for a deployed ERC-20. In the standard `deploy_erc20_token` flow, `register_token` writes both directions of the bijection atomically, so under normal operation the mapping is present. However, the vulnerability is unconditional on the code path: **any** precompile failure (paused precompile returning `Fatal` is the exception — that aborts the whole tx — but `Error`-class failures do not). The unchecked return value is a latent design flaw that activates whenever the precompile returns `ExitError`. Realistic triggers include storage migration bugs, a future upgrade altering key layout, or an admin accidentally removing the mapping entry.

---

### Recommendation

1. **Check the precompile return value in `withdrawToNear`** and revert if the call fails:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

2. **Alternatively, restructure the order**: call the precompile first, and only burn if the precompile call succeeds. This is the safer pattern for atomicity.

3. Apply the same fix to `withdrawToEthereum`, which has the identical unchecked-return-value pattern. [5](#0-4) 

---

### Proof of Concept

1. Deploy a bridged ERC-20 token normally (so the ERC-20 contract exists at address `A`).
2. Delete or corrupt the `Erc20Nep141Map` storage key for address `A` (e.g., via a storage migration test harness or mock IO that returns `None` for this key).
3. Call `withdrawToNear(recipient, amount)` from a user holding `amount` tokens.
4. Observe: `_burn` executes, reducing the user's balance by `amount`. The precompile call returns `0` (failure). `withdrawToNear` returns without reverting. No NEAR promise is scheduled. The `amount` tokens are permanently gone.

The existing test `test_exit_to_near_refund` demonstrates the refund path when `ft_transfer` fails at the NEAR level, but there is no test covering the case where the precompile itself fails mid-call while the burn has already been committed — confirming the gap. [6](#0-5)

fn get_nep141_from_erc20<I: IO>(erc20_token: &[u8], io: &I) -> Result<AccountId, ExitError> {
    AccountId::try_from(
        io.read_storage(bytes_to_key(KeyPrefix::Erc20Nep141Map, erc20_token).as_slice())
            .map(|s| s.to_vec())
            .ok_or(ExitError::Other(Cow::Borrowed(ERR_TARGET_TOKEN_NOT_FOUND)))?,
    )
    .map_err(|_| ExitError::Other(Cow::Borrowed("ERR_INVALID_NEP141_ACCOUNT")))
}

fn process_precompile(
    p: &dyn Precompile,
    handle: &impl PrecompileHandle,
) -> Result<PrecompileOutput, PrecompileFailure> {
    let input = handle.input();
    let gas_limit = handle.gas_limit();
    let context = handle.context();
    let is_static = handle.is_static();

    p.run(input, gas_limit.map(EthGas::new), context, is_static)
        .map_err(|exit_status| PrecompileFailure::Error { exit_status })
}
