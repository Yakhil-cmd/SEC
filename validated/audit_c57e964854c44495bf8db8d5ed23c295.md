### Title
SNS Treasury Valuation Hardcodes 8 Decimals, Ignoring Actual SNS Token Decimals - (File: `rs/sns/governance/token_valuation/src/lib.rs`)

### Summary
The SNS governance treasury valuation function `try_get_balance_valuation_factors` unconditionally divides the raw `icrc1_balance_of` response by the hardcoded constant `E8` (10^8) to normalize atomic units into whole tokens. This assumes the SNS token always has exactly 8 decimal places, but the ICRC-1 standard — which the SNS ledger implements — supports configurable decimals. If an SNS token is deployed with fewer than 8 decimals, the treasury valuation is systematically underestimated, causing the 7-day treasury transfer limit to be miscalculated and potentially bypassed.

### Finding Description
In `rs/sns/governance/token_valuation/src/lib.rs`, the private function `try_get_balance_valuation_factors` is the single code path used to compute treasury valuations for both ICP and SNS tokens:

```rust
// Line 177-181
let tokens = Decimal::from(u128::try_from(balance_of_response.0).map_err(|err| {
    ValuationError::new_arithmetic(format!(
        "Balance of {account:?} does not fit in u128: {err:?}"
    ))
})?) / Decimal::from(E8);   // <-- hardcoded 10^8
```

`E8 = 100_000_000` is a compile-time constant. The function never calls `icrc1_decimals` on the ledger canister to discover the actual decimal precision of the token. For ICP this is harmless (ICP is always 8 decimals), but for SNS tokens the ICRC-1 ledger accepts a configurable `decimals` field at initialization.

The resulting `tokens` value feeds directly into `ValuationFactors::to_xdr()`:

```rust
// Line 118-126
fn to_xdr(&self) -> Decimal {
    tokens * icps_per_token * xdrs_per_icp
}
```

This XDR total is then passed to `ProposalsAmountTotalUpperBound::from_valuation_xdr` in `rs/sns/governance/proposals_amount_total_limit/src/lib.rs` to classify the treasury as *small* (< 100,000 XDR → NoLimit), *medium* (100,000–1,200,000 XDR → 25% cap), or *large* (> 1,200,000 XDR → 300,000 XDR cap). The resulting token limit is then compared against the proposal's `amount_e8s / E8` in `treasury_valuation_if_proposal_amount_is_small_enough_or_err` and again at execution time in `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`.

### Impact Explanation
Consider an SNS token with 6 decimals (10^6 atomic units per whole token) and a treasury holding 1,000,000 whole tokens at 0.1 ICP/token and 10 XDR/ICP:

| | Correct (6 decimals) | Code (assumes 8 decimals) |
|---|---|---|
| `balance_atomic` | 10^12 | 10^12 |
| `tokens` | 10^12 / 10^6 = **1,000,000** | 10^12 / 10^8 = **10,000** |
| `to_xdr()` | 1,000,000 × 0.1 × 10 = **1,000,000 XDR** | 10,000 × 0.1 × 10 = **10,000 XDR** |
| Tier | Medium → **25% cap** | Small → **NoLimit** |
| Max transfer/7d | **250,000 tokens** | **10,000 tokens** (code's view) |

A proposer submitting `amount_e8s = 10^12` (= 1,000,000 whole tokens) has `proposal_amount_tokens = 10^12 / 10^8 = 10,000`. The code's `allowance_remainder_tokens` under NoLimit equals `balance_tokens = 10,000`. The check `10,000 ≤ 10,000` passes, allowing the entire treasury to be drained in a single proposal — a governance authorization bypass. The correct limit would have been 250,000 tokens per 7-day window.

The inverse case (SNS token with > 8 decimals) causes the treasury to appear far larger than it is, placing it in the "large" tier and imposing an unnecessarily tight XDR cap, which is a governance liveness issue.

### Likelihood Explanation
The SNS framework's ICRC-1 ledger accepts a configurable `decimals` field at init time. The SNS init YAML convention uses `_e8s` suffixes throughout, implying an 8-decimal assumption, but this is not enforced at the code level — the valuation function never queries `icrc1_decimals`. Any SNS deployed with a non-8-decimal token (whether by design or misconfiguration) triggers the miscalculation. The entry path is an unprivileged SNS governance participant submitting a `TransferSnsTreasuryFunds` or `MintSnsTokens` proposal on an affected SNS.

### Recommendation
`try_get_balance_valuation_factors` should query `icrc1_decimals` on the ledger canister (concurrently with the existing three requests) and use the returned value as the divisor instead of the hardcoded `E8`. The `Icrc1Client` trait should expose a `icrc1_decimals` method (the `ICRC1Client` in `packages/icrc-ledger-client/src/lib.rs` already implements one). The corrected normalization would be:

```rust
let decimals = icrc1_client.icrc1_decimals().await?;
let divisor = Decimal::from(10_u64.pow(decimals as u32));
let tokens = Decimal::from(...balance...) / divisor;
```

### Proof of Concept

1. Deploy an SNS whose ICRC-1 ledger is initialized with `decimals = 6`.
2. Fund the SNS treasury with 1,000,000 tokens (= 10^12 atomic units).
3. Submit a `TransferSnsTreasuryFunds` proposal with `amount_e8s = 10^12`.
4. `try_get_balance_valuation_factors` computes `tokens = 10^12 / 10^8 = 10,000` and `to_xdr() = 10,000 XDR` → tier = Small → NoLimit.
5. `proposal_amount_tokens = 10^12 / 10^8 = 10,000`; `allowance_remainder_tokens = 10,000` (NoLimit = balance).
6. Check `10,000 ≤ 10,000` passes at both submission and execution time.
7. The entire 1,000,000-token treasury is transferred, bypassing the intended 25% (250,000-token) 7-day cap.
