### Title
Permanent Fund Loss in `exitToNear` Precompile When NEP-141 Transfer Fails and `error_refund` Is Disabled — (File: engine-precompiles/src/native.rs)

---

### Summary

The `exitToNear` precompile burns EVM tokens (state committed) before the asynchronous NEP-141 `ft_transfer` promise completes. When the `error_refund` compile-time feature is absent, the attached callback is constructed with `refund: None`. If the NEP-141 transfer promise fails for any reason, the callback silently returns `Ok(None)` without issuing a refund. Because the token burn occurred in a prior NEAR receipt, it is never rolled back, causing permanent fund loss for the user.

---

### Finding Description

**Root cause — conditional `refund: None` in callback args**

In `engine-precompiles/src/native.rs`, the callback arguments for `exit_to_near_precompile_callback` are constructed as:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None when feature is absent
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

**Callback silently swallows the failure**

In `engine/src/contract_methods/connector.rs`, `exit_to_near_precompile_callback` checks the promise result:

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success path
    ...
    None
} else if let Some(args) = args.refund {
    // refund path — only reachable when error_refund is compiled in
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← reached when error_refund is absent AND promise failed
};
Ok(maybe_result)
``` [2](#0-1) 

When `error_refund` is not compiled in and the NEP-141 transfer fails, execution falls into the final `else` branch and returns `Ok(None)` — no refund, no error, no indication to the user.

**State written before the async call**

The EVM token burn happens inside the EVM execution (Receipt 1). The NEP-141 `ft_transfer` promise is a separate NEAR receipt (Receipt 2). The callback is Receipt 3. NEAR only reverts state changes within the receipt that panics; it cannot roll back Receipt 1 from Receipt 3. Therefore, the burned EVM tokens are irrecoverable once the callback completes without refunding. [3](#0-2) 

---

### Impact Explanation

Any EVM user who calls the `exitToNear` precompile and whose NEP-141 `ft_transfer` subsequently fails will permanently lose the bridged tokens. The tokens are burned from the EVM state but never delivered on the NEAR side, and no refund is issued. This constitutes **permanent freezing/loss of user funds**.

---

### Likelihood Explanation

The `exitToNear` precompile is a standard, publicly reachable user-facing feature — any EVM address can invoke it. NEP-141 `ft_transfer` calls can fail for multiple realistic reasons:

- The NEP-141 contract is paused (pausability is a standard NEP-141 feature).
- The Aurora engine account is not registered for storage on the NEP-141 contract.
- The NEP-141 contract enforces additional access controls.
- Transient network or gas issues cause the receipt to fail.

Because the `error_refund` feature is an opt-in compile-time flag (not a default), production builds that omit it expose every `exitToNear` user to this risk whenever the downstream transfer fails.

---

### Recommendation

- **Short term**: Make `error_refund` a default feature, or unconditionally populate the `refund` field in `ExitToNearPrecompileCallbackArgs` so that a failed NEP-141 transfer always triggers a refund path.
- **Long term**: Restructure the `exitToNear` flow so that the EVM token burn is only committed after the NEP-141 transfer is confirmed successful (e.g., by performing the burn inside the callback rather than in the precompile itself).

---

### Proof of Concept

1. User holds EVM-side ETH (or any bridged token) and calls the `exitToNear` precompile targeting a NEP-141 contract that is currently paused.
2. The EVM execution burns the user's tokens and emits a promise log. This state is committed in Receipt 1.
3. The NEAR runtime schedules Receipt 2: `ft_transfer` on the NEP-141 contract. The call fails because the contract is paused.
4. The NEAR runtime schedules Receipt 3: `exit_to_near_precompile_callback`. Because `error_refund` is not compiled in, `args.refund` is `None`. The callback reaches the `else { None }` branch and returns `Ok(None)`.
5. Receipt 1's EVM state (token burn) is never rolled back. The user's tokens are permanently lost. [4](#0-3) [5](#0-4)

#[named]
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
