### Title
Zero EVM Gas Cost for Exit Precompiles Allows ERC-20 Token Burn Without Guaranteed NEAR Promise Execution — (`engine-precompiles/src/native.rs`)

### Summary

Both `ExitToNear` and `ExitToEthereum` precompiles declare their EVM gas cost as `0` with an explicit `TODO` acknowledging the value is undetermined. Because the EVM charges nothing for these operations, the EVM gas limit provided by the caller does not account for the NEAR gas required to schedule and execute the cross-chain promise. If the NEAR transaction's prepaid gas is insufficient to attach gas to the outgoing promise, the ERC-20 tokens are permanently burned while the corresponding NEP-141 transfer or Ethereum withdrawal never occurs.

### Finding Description

In `engine-precompiles/src/native.rs`, the EVM gas costs for both exit precompiles are hardcoded to zero:

```rust
// TODO(#483): Determine the correct amount of gas
pub(super) const EXIT_TO_NEAR_GAS: EthGas = EthGas::new(0);

// TODO(#483): Determine the correct amount of gas
pub(super) const EXIT_TO_ETHEREUM_GAS: EthGas = EthGas::new(0);
``` [1](#0-0) 

These values are returned by `required_gas()` and used as the `cost` field of `PrecompileOutput`: [2](#0-1) [3](#0-2) 

Despite charging 0 EVM gas, the precompiles schedule NEAR promises that require substantial NEAR gas:

- `FT_TRANSFER_GAS = 10 TGas` for `ft_transfer`
- `FT_TRANSFER_CALL_GAS = 70 TGas` for `ft_transfer_call`
- `WITHDRAWAL_GAS = 100 TGas` for `withdraw` (also marked `TODO(#332)`) [4](#0-3) 

The promise is attached with this gas and emitted as a log for the engine to schedule: [5](#0-4) 

The ERC-20 token burn happens **before** the precompile is called. In `EvmErc20V2.sol`, `_burn` is called unconditionally before the precompile call: [6](#0-5) 

If the NEAR transaction's prepaid gas is insufficient to attach the required gas to the outgoing promise, the promise is never created, but the ERC-20 burn is already committed to EVM state.

Furthermore, when the `error_refund` feature is **not** enabled, there is no refund callback at all (`refund: None`), so a failed promise results in permanent token loss with no recovery path: [7](#0-6) 

This is confirmed by the test suite: [8](#0-7) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user's ERC-20 tokens are burned on Aurora (EVM state committed), but the corresponding NEP-141 `ft_transfer` or Ethereum `withdraw` promise is never executed. The tokens are destroyed on the Aurora side and never arrive on the NEAR or Ethereum side. There is no recovery mechanism when `error_refund` is disabled.

### Likelihood Explanation

Any caller of `aurora.submit()` controls the prepaid NEAR gas for that call. A user calling `submit` directly (not through a relayer) with a low NEAR gas limit, or a relayer configured with insufficient gas, can trigger this. Because `EXIT_TO_NEAR_GAS = 0`, the EVM gas limit gives no signal about the NEAR gas requirement, so neither users nor relayers are warned that extra NEAR gas is needed. The CHANGES.md entry "Fixed missing gas costs in `exitToNear` and `exitToEthereum` precompiles" (v2.9.0, [#687]) addressed the NEAR gas attached to promises but left the EVM gas cost at 0, confirming this remains unresolved. [9](#0-8) 

### Recommendation

1. Determine and set correct non-zero values for `EXIT_TO_NEAR_GAS` and `EXIT_TO_ETHEREUM_GAS` that reflect the NEAR gas cost of the outgoing promise (converted via `CROSS_CONTRACT_CALL_NEAR_GAS` ratio or an equivalent methodology).
2. Ensure the `error_refund` feature is enabled in all production builds so that failed promises trigger a refund callback rather than permanent token loss.
3. Resolve `TODO(#332)` for `WITHDRAWAL_GAS` using the same experimental methodology used for `FT_TRANSFER_GAS`.

### Proof of Concept

1. User holds 100 ERC-20 tokens (bridged NEP-141) on Aurora.
2. User calls `withdrawToNear(recipient, 100)` on `EvmErc20V2` — this calls `_burn(sender, 100)` then invokes the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`.
3. The precompile charges **0 EVM gas** and emits a promise log requesting 10 TGas for `ft_transfer`.
4. The user (or relayer) submitted the `aurora.submit()` call with only 15 TGas prepaid — enough for EVM execution but not enough to attach 10 TGas to the outgoing promise after engine overhead.
5. The engine fails to schedule the promise. EVM state (burn) is committed. NEP-141 tokens are never transferred.
6. With `error_refund` disabled, `refund: None` — no callback, no recovery. 100 tokens are permanently destroyed.
