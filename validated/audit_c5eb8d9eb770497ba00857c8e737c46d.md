### Title
Unimplemented `add_batch_actions` in XCC Router Causes Panic and Permanent wNEAR Fund Freeze - (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The `add_batch_actions` function in the XCC router contract is `unimplemented!()` in the default production build (when the `all-promise-actions` feature is not enabled). An unprivileged EVM user can craft a cross-contract call (XCC) containing a `NearPromise::Simple(SimpleNearPromise::Batch(...))` or a `NearPromise::Then { ..., SimpleNearPromise::Batch(...) }` payload. The XCC precompile will first transfer the user's wNEAR to the engine's implicit address (an EVM-level state change that is committed), then dispatch a NEAR promise to the router's `execute` function. The router panics at `add_batch_actions`, the NEAR promise fails, but the wNEAR transfer is never reversed — permanently freezing the user's funds.

---

### Finding Description

In `etc/xcc-router/src/lib.rs`, the `add_batch_actions` function has two conditional implementations: [1](#0-0) 

The `#[cfg(not(feature = "all-promise-actions"))]` variant — which is the default in production — unconditionally calls `unimplemented!()`, causing a panic whenever any `PromiseAction` batch is processed.

This function is called from `recursive_promise_create`: [2](#0-1) 

Which is called from `promise_create`, which is called from both `execute` and `execute_scheduled`: [3](#0-2) 

The attacker-controlled entry path runs through the XCC precompile. In `engine-precompiles/src/xcc.rs`, the precompile first performs an EVM-level `transferFrom` to move wNEAR from the caller to the engine's implicit address: [4](#0-3) 

Only after this EVM state change is committed does the precompile emit a promise log that the Aurora Engine uses to dispatch a NEAR promise to the router's `execute` method: [5](#0-4) 

When the router panics, the NEAR promise fails and its state changes are reverted. However, the wNEAR `transferFrom` was an EVM-level call within the Aurora Engine's own execution context — it is already committed to the NEAR trie and is **not** rolled back when the downstream router promise fails.

The `STORAGE_AMOUNT` charged for a first-time router deployment is 2 NEAR: [6](#0-5) 

There is no callback or refund mechanism in the XCC precompile to recover the transferred wNEAR if the router promise fails.

---

### Impact Explanation

**Permanent freezing of funds.** Any wNEAR transferred by the XCC precompile (covering `attached_near` plus, for first-time users, 2 NEAR of storage staking) is irrecoverably moved to the engine's implicit address. The user receives no XCC execution and no refund. The funds are permanently inaccessible to the user.

---

### Likelihood Explanation

**Medium.** The `PromiseArgs` type is a public, borsh-serialized input accepted by the XCC precompile. Any EVM user who constructs a `CrossContractCallArgs::Eager` or `CrossContractCallArgs::Delayed` payload containing a `NearPromise::Simple(SimpleNearPromise::Batch(...))` or a `NearPromise::Then` with a `Batch` callback will trigger this path. No special privilege is required — only the ability to submit an EVM transaction to Aurora. The `all-promise-actions` feature is not enabled in the production build, making the panic unconditional for any batch-containing promise.

---

### Recommendation

1. **Guard before transferring funds**: Move the wNEAR `transferFrom` call to occur only after validating that the `PromiseArgs` payload does not contain any `SimpleNearPromise::Batch` variant (or any other unsupported action type) when `all-promise-actions` is not enabled.
2. **Return a graceful error**: Replace the `unimplemented!()` in `add_batch_actions` with a proper error return (e.g., `env::panic_str("ERR_BATCH_ACTIONS_NOT_SUPPORTED")`) and propagate it upward so the XCC precompile can revert cleanly before any funds are moved.
3. **Add a refund callback**: Attach a NEAR callback to the router promise that refunds the wNEAR to the caller if the router call fails for any reason.

---

### Proof of Concept

1. Deploy a contract on Aurora that calls the XCC precompile at address `0x516cded1d16af10cad47d6d49128e2eb7d27b372`.
2. Approve the XCC precompile to spend the caller's wNEAR.
3. Encode a `CrossContractCallArgs::Eager` payload (borsh) containing a `PromiseArgs::Recursive(NearPromise::Simple(SimpleNearPromise::Batch({ target_account_id: "any.near", actions: [PromiseAction::Transfer { amount: 1 }] })))`.
4. Submit the EVM transaction. The XCC precompile executes `transferFrom` (wNEAR leaves the user's EVM balance), emits the promise log, and returns success to the EVM.
5. The Aurora Engine dispatches the NEAR promise to `<sender_hex>.aurora`'s `execute` method.
6. The router calls `recursive_promise_create` → `add_batch_actions` → `unimplemented!()` → NEAR panic.
7. The NEAR promise fails. The user's wNEAR (including the 2 NEAR storage deposit if this is their first XCC call) is permanently frozen in the engine's implicit address with no recovery path.
