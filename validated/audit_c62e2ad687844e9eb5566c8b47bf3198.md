Audit Report

## Title
Anonymous Principal Bypass in `icrc152_burn_not_async` Allows Unauthenticated Token Destruction — (File: `rs/ledger_suite/icrc1/ledger/src/main.rs`)

## Summary

`icrc152_burn_not_async` gates execution on `ic_cdk::api::is_controller(&caller)` but never rejects the anonymous principal as the caller. Because the IC protocol permits anonymous ingress update calls and permits the anonymous principal to be a canister controller, any unauthenticated user can call `icrc152_burn` and permanently destroy tokens from any non-anonymous, non-minting account whenever the ledger is deployed with the anonymous principal as a controller. The test harness confirms this exact configuration is used in all happy-path ICRC-152 tests.

## Finding Description

`icrc152_burn_not_async` performs two guards before executing the burn:

1. Feature-flag check: `icrc152` must be enabled.
2. Controller check: `if !ic_cdk::api::is_controller(&caller)`. [1](#0-0) 

The function then burns from the caller-supplied `args.from` account — any account in the ledger — with no ownership relationship between the caller and the target account. [2](#0-1) 

There is a guard against the anonymous principal appearing as the **burn target**: [3](#0-2) 

But there is **no symmetric guard** rejecting the anonymous principal as the **caller**. The IC ingress message validator explicitly accepts anonymous update calls: [4](#0-3) 

The anonymous principal can legitimately be set as a canister controller (the IC protocol does not prohibit it): [5](#0-4) 

The `setup_icrc152` test helper installs the ledger with `None` as the sender, making the anonymous principal the controller, and all happy-path tests use `PrincipalId::new_anonymous()` as the authorized caller: [6](#0-5) [7](#0-6) 

The same gap exists in `icrc152_mint_not_async`: [8](#0-7) 

## Impact Explanation

When the anonymous principal is a controller of the ledger canister, any unauthenticated Internet Computer user can call `icrc152_burn`, supply any non-anonymous non-minting account in `args.from`, and permanently destroy an arbitrary amount of that account's tokens. This constitutes **unauthorized permanent loss of ledger assets** — a direct match for the High impact category: *"Unauthorized access to… ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."* The constraint here is the deployment precondition (anonymous principal as controller), which bounds severity to High rather than Critical.

## Likelihood Explanation

The anonymous principal becomes the effective controller whenever a canister is installed via an anonymous ingress call (no authentication required on the IC) or when a controller explicitly sets it via `UpdateSettings`. The `StateMachine::install_canister(..., None)` pattern used throughout the ICRC-152 test suite demonstrates this is a realistic deployment path. Developer-deployed or bootstrapped ledgers that have not yet transferred control to a governance canister are immediately exploitable. Production SNS ledgers with a governance canister as controller are not affected, but the code provides no defense-in-depth against the misconfiguration. Likelihood is **Medium**.

## Recommendation

Add an explicit anonymous-principal rejection for the **caller** in both `icrc152_burn_not_async` and `icrc152_mint_not_async`, immediately after the feature-flag check and before the `is_controller` check:

```rust
if caller == Principal::anonymous() {
    return Err(Icrc152BurnError::Unauthorized(
        "anonymous principal is not allowed as caller".to_string(),
    ));
}
```

This mirrors the existing guard on `args.from.owner` and closes the bypass regardless of controller configuration. [3](#0-2) 

## Proof of Concept

The existing integration test infrastructure directly demonstrates the exploit:

1. Call `setup_icrc152` — installs the ledger with `None` sender, making `Principal::anonymous()` the controller.
2. Fund victim account `V` with `N` tokens via `icrc152_mint` called as `PrincipalId::new_anonymous()`.
3. As `PrincipalId::new_anonymous()` (no key material required), call `icrc152_burn` with `from: V, amount: N`.
4. `is_controller(anonymous) == true` → controller guard passes; `args.from.owner != anonymous` → `InvalidAccount` guard passes; burn executes; `V`'s balance is reduced to 0 and total supply is decremented.

This is directly exercised by `test_icrc152_mint_and_burn` in `rs/ledger_suite/tests/sm-tests/src/lib.rs`. [9](#0-8)

pub fn test_icrc152_mint_and_burn<T>(ledger_wasm: Vec<u8>, encode_init_args: fn(InitArgs) -> T)
where
    T: CandidType,
{
    let p1 = PrincipalId::new_user_test_id(1);
    let (env, canister_id) = setup_icrc152(ledger_wasm, encode_init_args, vec![]);
    let controller = PrincipalId::new_anonymous();

    let supply_before = total_supply(&env, canister_id);
    assert_eq!(supply_before, 0);

    // --- Mint ---
    let mint_amount = 5_000_000_u64;
    let mint_result = icrc152_mint(
        &env,
        canister_id,
        controller,
        &Icrc152MintArgs {
            to: Account::from(p1.0),
            amount: Nat::from(mint_amount),
            created_at_time: now_nanos(&env),
            reason: Some("test mint".to_string()),
        },
    );
    let mint_block_idx = mint_result.expect("icrc152_mint should succeed");
    assert_eq!(mint_block_idx, Nat::from(0_u64));

    assert_eq!(balance_of(&env, canister_id, p1.0), mint_amount);
    assert_eq!(total_supply(&env, canister_id), mint_amount);

    // Verify block type via icrc3_get_blocks
    let blocks = icrc3_get_blocks(&env, canister_id, 0, 1);
    assert_eq!(blocks.blocks.len(), 1);
    let btype = match &blocks.blocks[0].block {
        ICRC3Value::Map(m) => m
            .iter()
            .find(|(k, _)| k.as_str() == "btype")
            .map(|(_, v)| v.clone()),
        other => panic!("expected Map block, got: {other:?}"),
    };
    assert_eq!(
        btype,
        Some(ICRC3Value::Text("122mint".to_string())),
        "mint block should have btype 122mint"
    );

    // --- Burn ---
    let burn_amount = 2_000_000_u64;
    let burn_result = icrc152_burn(
        &env,
        canister_id,
        controller,
        &Icrc152BurnArgs {
            from: Account::from(p1.0),
            amount: Nat::from(burn_amount),
            created_at_time: now_nanos(&env),
            reason: Some("test burn".to_string()),
        },
    );
    let burn_block_idx = burn_result.expect("icrc152_burn should succeed");
    assert_eq!(burn_block_idx, Nat::from(1_u64));

    assert_eq!(
        balance_of(&env, canister_id, p1.0),
        mint_amount - burn_amount
    );
    assert_eq!(total_supply(&env, canister_id), mint_amount - burn_amount);
