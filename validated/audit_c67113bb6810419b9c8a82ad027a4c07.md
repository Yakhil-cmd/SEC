### Title
Address Whitelist Bypass in `receive_erc20_tokens` When No Fallback Address Is Configured - (`engine/src/engine.rs`)

---

### Summary

In `receive_erc20_tokens`, the Address whitelist check for the ERC-20 recipient is gated behind a `let Some(fallback_address)` guard. When no fallback address is stored, the entire whitelist check is silently skipped and ERC-20 tokens are minted to any arbitrary EVM address the caller specifies — including addresses that are explicitly excluded from the Silo Address whitelist. The consumed NEP-141 tokens cannot be recovered, and the minted ERC-20 tokens are permanently frozen because the non-whitelisted address cannot submit EVM transactions.

---

### Finding Description

In `engine/src/engine.rs`, `receive_erc20_tokens` contains the following guard:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

The compound `if let Some(...) && !check(...)` short-circuits on the first clause. When `get_erc20_fallback_address` returns `None` (i.e., no fallback address is stored), the entire block is skipped — `is_allow_receive_erc20_tokens` is **never called** — and `recipient` remains whatever EVM address the caller supplied in `args.msg`. The ERC-20 mint then proceeds unconditionally against that address. [2](#0-1) 

`is_allow_receive_erc20_tokens` delegates to `is_address_allowed`, which checks the `WhitelistKind::Address` list:

```rust
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}

fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
    !list.is_enabled() || list.is_exist(address)
}
``` [3](#0-2) 

When the Address whitelist is enabled but no fallback address is configured, this check is never reached. The fallback address can be absent in two legitimate operator configurations:

1. `set_silo_params(None)` clears both `fixed_gas` and `erc20_fallback_address` simultaneously.
2. `set_erc20_fallback_address` (added as a standalone entrypoint) can be called with `address: None` to clear it independently. [4](#0-3) [5](#0-4) 

The entry point is `ft_on_transfer`, which is callable by any NEP-141 contract that has been registered with Aurora Engine. It routes to `receive_erc20_tokens` for all non-ETH-connector predecessors: [6](#0-5) 

---

### Impact Explanation

**High — Permanent freezing of funds.**

When a user calls `ft_transfer_call` on a NEP-141 contract targeting Aurora Engine and specifies a non-whitelisted EVM address in `msg`, the following occurs:

1. The NEP-141 tokens are transferred out of the user's NEAR account and into Aurora Engine's custody.
2. `receive_erc20_tokens` mints the equivalent ERC-20 balance to the non-whitelisted EVM address.
3. The non-whitelisted address cannot submit EVM transactions (`is_allow_submit` enforces the same Address whitelist for all transaction submission), so it cannot call `ExitToNear` or transfer the ERC-20 tokens.
4. There is no recovery path: the NEP-141 tokens are consumed and the ERC-20 tokens are stranded.

The funds are permanently frozen. The Silo operator's intent — that only whitelisted addresses may hold bridged ERC-20 balances — is violated.

---

### Likelihood Explanation

**Medium.** The precondition is that the Silo Address whitelist is enabled while no fallback address is stored. This is a reachable and documented configuration: the CHANGES.md explicitly notes `set_erc20_fallback_address` as a standalone transaction, meaning operators can enable the whitelist without setting a fallback. Any NEAR user who then calls `ft_transfer_call` specifying a non-whitelisted EVM address triggers the freeze — no special privilege is required.

---

### Recommendation

Decouple the whitelist check from the fallback address lookup. When the Address whitelist is enabled and the recipient is not allowed, the function should return an error (causing the NEP-141 tokens to be refunded to the sender) regardless of whether a fallback address is configured:

```rust
// Check whitelist independently of fallback address
if silo::is_address_whitelist_enabled(&self.io) {
    if !silo::is_allow_receive_erc20_tokens(&self.io, &recipient) {
        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io) {
            recipient = fallback_address;
        } else {
            return Err(/* AddressNotAllowed error */);
        }
    }
}
```

This ensures that when no fallback is configured, a non-whitelisted recipient causes the transfer to fail and the NEP-141 tokens to be returned to the sender.

---

### Proof of Concept

**Setup:**
- Silo mode is active.
- `WhitelistKind::Address` whitelist is enabled.
- No fallback address is stored (either never set, or cleared via `set_erc20_fallback_address(None)`).
- A NEP-141 token has a corresponding ERC-20 deployed on Aurora.

**Attack steps:**

1. Attacker (or any NEAR user) calls `ft_transfer_call` on the NEP-141 contract:
   - `receiver_id`: Aurora Engine account
   - `amount`: any positive amount
   - `msg`: hex-encoded non-whitelisted EVM address (e.g., `"deadbeef..."`).

2. The NEP-141 contract invokes `ft_on_transfer` on Aurora Engine with `predecessor_account_id` = NEP-141 contract.

3. `ft_on_transfer` routes to `receive_erc20_tokens`. [7](#0-6) 

4. Inside `receive_erc20_tokens`, `get_erc20_fallback_address` returns `None`. The `if let Some(...) && ...` block is skipped entirely. `recipient` remains the non-whitelisted address. [1](#0-0) 

5. The ERC-20 mint call executes successfully, crediting the non-whitelisted address. [8](#0-7) 

6. `ft_on_transfer` returns `Ok`, so `amount_to_return = 0` — the NEP-141 tokens are **not** refunded. [9](#0-8) 

7. The non-whitelisted EVM address holds ERC-20 tokens but cannot submit any EVM transaction to move them. The funds are permanently frozen.

    let key = erc20_fallback_address_key();

    if let Some(address) = address {

        io.remove_storage(&key);
