Audit Report

## Title
Free Ingress Induction for `UpdateSettings` Messages Enables Zero-Cost Subnet Resource Exhaustion - (File: `rs/cycles_account_manager/src/cycles_account_manager.rs`)

## Summary
The `ingress_induction_cost` function returns `IngressInductionCost::Free` for `UpdateSettings` management messages with payloads ≤ 338 bytes. This is intentional to allow unfreezing frozen canisters, but no balance check or cycles charge is enforced at any stage of the ingress pipeline (filter, selector, induction, or post-execution). An attacker who controls a canister with zero cycles can submit an unbounded stream of distinct `UpdateSettings` messages at zero net cost, consuming subnet execution resources and crowding out legitimate ingress traffic up to `max_ingress_messages_per_block` per block.

## Finding Description

**Root cause — `ingress_induction_cost` returns `Free` for small `UpdateSettings` payloads**

In `rs/cycles_account_manager/src/cycles_account_manager.rs`, when the method is `UpdateSettings` and `is_delayed_ingress_induction_cost` returns `true` (payload ≤ `MAX_DELAYED_INGRESS_COST_PAYLOAD_SIZE` = 338 bytes), `paying_canister` is set to `None`, which maps to `IngressInductionCost::Free`: [1](#0-0) 

The threshold check: [2](#0-1) 

**No balance check at the ingress filter (`should_accept_ingress_message`)**

In `rs/execution_environment/src/execution_environment.rs`, the balance check is gated entirely on `IngressInductionCost::Fee`. When the cost is `Free`, the entire check block is skipped: [3](#0-2) 

**No cycles check at the ingress selector (`validate_ingress`)**

In `rs/ingress_manager/src/ingress_selector.rs`, the `Free` arm of the match explicitly does nothing: [4](#0-3) 

**No cycles charged at induction (`enqueue`)**

In `rs/messaging/src/scheduling/valid_set_rule.rs`, free messages are pushed directly into the subnet queue without any cycles withdrawal: [5](#0-4) 

This is confirmed by the existing test `management_message_update_setting_is_inducted_but_not_charged`, which asserts that the canister balance is unchanged after induction: [6](#0-5) 

**Post-execution charge is silently discarded on failure**

After executing `UpdateSettings`, the delayed induction cost is charged via `consume_cycles`. If the canister has no cycles, the error is explicitly discarded with `_ignore_error`: [7](#0-6) 

**Block-level cap does not prevent crowding**

`MAX_INGRESS_MESSAGES_PER_BLOCK` is 1000: [8](#0-7) 

An attacker with multiple canisters can fill all 1000 slots per block with free `UpdateSettings` messages, displacing legitimate paid ingress traffic.

## Impact Explanation

This is a **High** severity finding matching the allowed impact class: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

The attack is not raw volumetric DDoS — it exploits a specific economic bypass in the protocol. By submitting free `UpdateSettings` messages up to `max_ingress_messages_per_block` per block, an attacker can crowd out legitimate user traffic on any application subnet. With multiple canisters (each requiring only a one-time creation fee), the attacker can sustain this indefinitely. The post-execution charge failure is silently discarded, so there is no economic deterrent at any stage.

## Likelihood Explanation

- **Attacker entry point**: Any user with an IC identity can create a canister and become its controller. No privileged access is required.
- **Attack cost**: One-time canister creation fee only. All subsequent `UpdateSettings` messages are free at induction and the post-execution charge is silently dropped when the canister has no cycles.
- **Practical feasibility**: Messages are submitted via the standard `/api/v2/canister/.../call` HTTP endpoint. Each message uses a different nonce to produce a unique `MessageId`, bypassing deduplication. The ingress pool throttler (`exceeds_threshold`) limits per-peer pool size but does not distinguish free-induction messages from paid ones.
- **Mitigating factors**: Per-peer ingress pool limits, `max_ingress_messages_per_block`, round-robin ingress selection, and the 5-minute ingress TTL reduce but do not eliminate the impact. Multiple coordinated attackers with multiple canisters amplify the effect.

## Recommendation

1. **Add a minimum balance check for free-induction messages**: In `should_accept_ingress_message` and `validate_ingress`, check that the target canister has sufficient cycles to cover the post-execution charge even when `IngressInductionCost::Free` is returned.
2. **Rate-limit free-induction messages per canister**: Track and cap the number of `UpdateSettings` messages per canister per block or per time window in the ingress selector.
3. **Do not silently discard the post-execution charge failure**: If `consume_cycles` fails after `UpdateSettings` execution, record the debt or penalize the canister rather than discarding the error with `_ignore_error`.
4. **Charge upfront with a guaranteed refund path**: Charge the induction cost at induction time and refund it if the canister was frozen, eliminating the free window entirely while preserving the unfreeze use case.

## Proof of Concept

```
1. Create canister C with minimal cycles (just enough for creation).
2. Drain C's cycles to zero (or leave it at zero after creation).
3. Construct N distinct UpdateSettings ingress messages targeting C:
   - Each message: canister_id=C, settings={freezing_threshold: X}, payload ≤ 338 bytes
   - Each message uses a different nonce so MessageId is unique
   - Sender = controller of C (the attacker's identity)
4. Submit all N messages via POST /api/v2/canister/IC_00/call
   - ingress_induction_cost() returns Free for each → no balance check at filter
     (rs/execution_environment/src/execution_environment.rs:3351)
   - validate_ingress() hits the Free arm → "Do nothing"
     (rs/ingress_manager/src/ingress_selector.rs:592)
   - enqueue() pushes each message without charging cycles
     (rs/messaging/src/scheduling/valid_set_rule.rs:287)
5. Messages are included in blocks (up to max_ingress_messages_per_block=1000 per block)
   and executed on all replicas.
6. Post-execution charge fails silently (_ignore_error) because C has no cycles
   (rs/execution_environment/src/execution_environment.rs:1045).
7. Attacker has consumed N × (execution cost per UpdateSettings) of subnet resources
   at zero net cost, displacing legitimate ingress traffic.

Reproducible as a deterministic integration test using PocketIC:
- Spin up a subnet, create canister C with zero cycles.
- Submit 1000 UpdateSettings messages with distinct nonces.
- Assert all 1000 are inducted and executed.
- Assert C's balance remains zero throughout.
- Assert legitimate paid ingress messages submitted concurrently are delayed or dropped.
```

#[test]
fn management_message_update_setting_is_inducted_but_not_charged() {
    let ingress_history_writer = MockIngressHistory::new();
    let metrics_registry = MetricsRegistry::new();
    let subnet_id = subnet_test_id(1);
    let valid_set_rule = ValidSetRuleImpl::new(
        Arc::new(ingress_history_writer),
        Arc::new(
            CyclesAccountManagerBuilder::new()
                .with_subnet_id(subnet_id)
                .build(),
        ),
        &metrics_registry,
        no_op_logger(),
    );

    let mut state = ReplicatedStateBuilder::new().build();
    let canister_id = canister_test_id(0);
    let canister = get_running_canister(canister_id);
    let balance_before = canister.system_state.balance();
    state.put_canister_state(canister);

    let payload = UpdateSettingsArgs {
        canister_id: canister_id.get(),
        settings: CanisterSettingsArgsBuilder::new()
            .with_freezing_threshold(1 << 20)
            .build(),
        sender_canister_version: None,
    }
    .encode();
    let ingress = SignedIngressBuilder::new()
        .canister_id(IC_00)
        .method_name("update_settings")
        .method_payload(payload)
        .build();
    assert!(valid_set_rule.enqueue(&mut state, ingress).is_ok());

    let balance_after = state
        .canister_state(&canister_id)
        .unwrap()
        .system_state
        .balance();

    assert_eq!(balance_after, balance_before);
}
