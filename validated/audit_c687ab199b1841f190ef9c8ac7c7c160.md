Audit Report

## Title
`end_timestamp_seconds: None` in SNS `RewardEvent` Causes Permanent DoS of Reward Distribution - (File: `rs/sns/governance/src/governance.rs`)

## Summary

The SNS governance canister's `distribute_rewards` and `should_distribute_rewards` functions use `.unwrap_or_default()` on the `Option<u64>` field `end_timestamp_seconds` of the latest `RewardEvent`. When this field is `None` — which occurs for any SNS canister whose stored proto contains a `RewardEvent` without `end_timestamp_seconds` (e.g., deployed before the field was added) — the fallback silently becomes `0`, causing `new_rounds_count` to be computed as `now / round_duration_seconds` (~19,675 for 1-day rounds). The resulting loop exhausts the per-message instruction budget or overflows `u64`, and because the overflow guard returns without updating `latest_reward_event`, the failure repeats on every periodic task invocation, permanently breaking SNS reward distribution.

## Finding Description

**Root cause — initialization guard bypass:**

In `Governance::new` (`rs/sns/governance/src/governance.rs`, L726–742), the guard that sets `end_timestamp_seconds: Some(now)` only fires when `proto.latest_reward_event.is_none()`:

```rust
if proto.latest_reward_event.is_none() {
    proto.latest_reward_event = Some(RewardEvent {
        end_timestamp_seconds: Some(now),
        ...
    })
}
```

Any SNS canister whose persisted proto already contains a `RewardEvent` (e.g., from a version predating the `end_timestamp_seconds` field) will have `latest_reward_event = Some(RewardEvent { end_timestamp_seconds: None, ... })` after upgrade. The guard does not fire, and `end_timestamp_seconds` remains `None` permanently.

**Trigger path — `should_distribute_rewards` always returns `true`:**

`should_distribute_rewards` (`rs/sns/governance/src/governance.rs`, L5735–5739) computes:

```rust
let seconds_since_last_reward_event = now.saturating_sub(
    self.latest_reward_event()
        .end_timestamp_seconds
        .unwrap_or_default(),   // → 0 when None
);
```

With `end_timestamp_seconds = None`, `seconds_since_last_reward_event = now ≈ 1.7 × 10⁹`, which always exceeds `round_duration_seconds`, so `distribute_rewards` is called on every periodic task tick.

**Unbounded loop in `distribute_rewards`:**

`distribute_rewards` (`rs/sns/governance/src/governance.rs`, L5808–5814) computes:

```rust
let reward_start_timestamp_seconds = self
    .latest_reward_event()
    .end_timestamp_seconds
    .unwrap_or_default();   // → 0
let new_rounds_count = now
    .saturating_sub(reward_start_timestamp_seconds)
    .saturating_div(round_duration_seconds);
```

For `now ≈ 1,700,000,000` and `round_duration_seconds = 86400`, `new_rounds_count ≈ 19,675`. The only guard (`if new_rounds_count == 0 { return; }`, L5815–5820) does not fire.

The loop at L5861–5872 then executes ~19,675 iterations of `Decimal` floating-point arithmetic, exhausting the IC per-message instruction budget.

**Permanent failure — overflow guard does not update state:**

Even if the loop completes, the accumulated `rewards_purse_e8s` overflows `u64`. The overflow guard at L5878–5889 returns early without updating `latest_reward_event`:

```rust
Err(err) => {
    log!(ERROR, "Looks like the rewards purse ({}) overflowed u64: ...");
    return;   // latest_reward_event is NOT updated
}
```

Because `latest_reward_event` is never updated, the next periodic task call recomputes the same `new_rounds_count` and fails identically.

## Impact Explanation

This is a permanent, self-reinforcing DoS of SNS reward distribution for any affected canister. No neuron maturity is ever increased, and the canister wastes its full instruction budget on every timer tick (~every 10 seconds per `RUN_PERIODIC_TASKS_INTERVAL`). The SNS governance canister's core reward function is completely non-functional with no self-recovery path. This matches the **High** impact category: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS" and "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation

The condition is triggered automatically by the IC timer set in `init_timers` — no privileged access or special message is required. Any SNS canister whose stored proto contains a `RewardEvent` without `end_timestamp_seconds` (i.e., deployed before the field was introduced to the protobuf schema) is permanently affected upon upgrade to a version containing this code. The periodic task fires every 10 seconds; any unprivileged user sending any message to the canister can advance the timer queue. No attacker action is required beyond the canister existing in the affected state.

## Recommendation

Replace `unwrap_or_default()` with an explicit fallback to `self.proto.genesis_timestamp_seconds` in both `distribute_rewards` and `should_distribute_rewards`:

```rust
let reward_start_timestamp_seconds = self
    .latest_reward_event()
    .end_timestamp_seconds
    .unwrap_or(self.proto.genesis_timestamp_seconds);
```

Additionally, add a post-upgrade migration in `canister_post_upgrade` that sets `end_timestamp_seconds` to `genesis_timestamp_seconds` for any existing `RewardEvent` where the field is `None`, preventing the issue from manifesting on already-deployed SNS canisters.

## Proof of Concept

1. Deploy an SNS governance canister with an initial `GovernanceProto` where `latest_reward_event = Some(RewardEvent { round: 0, end_timestamp_seconds: None, ..Default::default() })` and `voting_rewards_parameters` configured with `round_duration_seconds = 86400`.
2. Confirm the initialization guard at L726 does not fire (since `latest_reward_event` is `Some`, not `None`).
3. Advance the canister clock to any realistic Unix timestamp (e.g., `1_700_000_000`).
4. Invoke `run_periodic_tasks`. Observe that `should_distribute_rewards` returns `true` (since `now.saturating_sub(0) > 86400`).
5. Observe that `distribute_rewards` computes `new_rounds_count = 1_700_000_000 / 86400 ≈ 19,675` and enters the loop.
6. Observe that the call traps (instruction limit) or the overflow guard fires and returns without updating `latest_reward_event`.
7. Invoke `run_periodic_tasks` again. Observe identical failure — `latest_reward_event` is unchanged, `new_rounds_count` is the same, reward distribution is permanently broken.

A deterministic PocketIC integration test can reproduce this by constructing the described proto, installing the governance canister, and asserting that `latest_reward_event` is never updated across multiple `run_periodic_tasks` calls.
