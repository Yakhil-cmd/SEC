### Title
Hardcoded `E8` Decimal Divisor in SNS Token Treasury Valuation Ignores Actual Token Decimals - (File: `rs/sns/governance/token_valuation/src/lib.rs`)

### Summary

The `try_get_balance_valuation_factors` function in the SNS governance token valuation library unconditionally divides the raw ICRC-1 ledger balance by the hardcoded constant `E8` (= `100_000_000`, i.e., 10^8) when converting the raw balance to a human-readable token count. This is the direct analog of the `redeemDyad()` decimal bug: the code assumes all SNS tokens have exactly 8 decimals, but the ICRC-1 standard allows any number of decimals. An SNS whose ledger is configured with a different decimal count (e.g., 6 or 18) will produce a systematically wrong treasury valuation, which in turn corrupts the 7-day spending limit enforced on `TransferSnsTreasuryFunds` and `MintSnsTokens` governance proposals.

### Finding Description

In `rs/sns/governance/token_valuation/src/lib.rs`, the function `try_get_balance_valuation_factors` fetches the raw ICRC-1 balance (in the token's smallest denomination) and converts it to a `Decimal` number of "whole tokens" by dividing by the hardcoded constant `E8`:

```rust
let tokens = Decimal::from(u128::try_from(balance_of_response.0)...) / Decimal::from(E8);
```

`E8` is defined as `100_000_000` (10^8), which is correct only for tokens with exactly 8 decimal places (like ICP or ckBTC). The ICRC-1 standard (`icrc1_decimals`) allows any `u8` value. An SNS token ledger configured with, say, 6 decimals (like USDC-style tokens) or 18 decimals would cause the balance to be divided by the wrong factor.

This function is called by both `try_get_icp_balance_valuation` (for ICP, which is always 8 decimals — safe) and `try_get_sns_token_balance_valuation` (for the SNS native token — **not** guaranteed to be 8 decimals). The resulting `ValuationFactors.tokens` field is then used directly in:

1. `transfer_sns_treasury_funds_7_day_total_upper_bound_tokens` — to compute the 7-day cap on `TransferSnsTreasuryFunds` proposals.
2. `mint_sns_tokens_7_day_total_upper_bound_tokens` — to compute the 7-day cap on `MintSnsTokens` proposals.
3. `treasury_valuation_amount_e8s` in SNS governance — which calls `tokens_to_e8s` on the already-wrong `tokens` value, compounding the error.

The `proposal_amount_tokens` for `TransferSnsTreasuryFunds` also hardcodes `E8` as the divisor:

```rust
fn proposal_amount_tokens(&self) -> Result<Decimal, String> {
    denominations_to_tokens(self.amount_e8s, E8)
```

So both the limit and the proposal amount use `E8`, but the treasury balance valuation is the critical path that determines whether a proposal is accepted or rejected.

### Impact Explanation

**Ledger conservation / governance authorization bug.**

- **SNS token with fewer than 8 decimals (e.g., 6):** The raw balance is divided by 10^8 instead of 10^6, making the treasury appear **100× smaller** than it actually is. The 7-day spending cap is computed on this deflated valuation, so the cap is 100× too low. Legitimate large treasury transfers or mints are incorrectly blocked. Conversely, the `tokens` field in `ValuationFactors` is wrong, so the XDR valuation used to classify the treasury as "small/medium/large" is also wrong, potentially placing a large treasury in the "small" regime where the most permissive relative limits apply.

- **SNS token with more than 8 decimals (e.g., 18):** The raw balance is divided by 10^8 instead of 10^18, making the treasury appear **10^10× larger** than it actually is. The 7-day spending cap is computed on this massively inflated valuation, so the cap is effectively unlimited. An attacker who can submit governance proposals (any token holder with enough stake) can drain the entire SNS treasury in a single 7-day window without triggering the intended rate-limit protection.

The second scenario (>8 decimals) is the critical security impact: the treasury drain protection is completely bypassed for any SNS whose token ledger uses more than 8 decimals.

### Likelihood Explanation

The ICRC-1 standard does not mandate 8 decimals. Any SNS that initializes its ledger with a non-8-decimal token (which is a valid configuration) is affected. An unprivileged governance participant (any neuron holder) can submit a `TransferSnsTreasuryFunds` proposal. The entry path is fully reachable via normal ingress to the SNS governance canister. No privileged access is required.

### Recommendation

Replace the hardcoded `E8` divisor with a dynamic call to `icrc1_decimals` on the ledger canister. The `LedgerCanister` client already has access to the canister ID; a `decimals()` method can be added to the `Icrc1Client` trait and fetched concurrently alongside the balance. The conversion then becomes:

```rust
let decimals = icrc1_client.icrc1_decimals().await?;
let tokens = Decimal::from(...balance...) / Decimal::from(10_u128.pow(decimals as u32));
```

Similarly, `proposal_amount_tokens` in `TransferSnsTreasuryFunds` and `MintSnsTokens` should use the actual ledger decimals rather than the hardcoded `E8`.

### Proof of Concept

1. Deploy an SNS whose token ledger is configured with `decimals = 18` (valid per ICRC-1).
2. Fund the SNS token treasury with, say, `1_000_000` SNS tokens (= `1_000_000 * 10^18` raw units).
3. Call `try_get_sns_token_balance_valuation` (triggered internally when any `TransferSnsTreasuryFunds` proposal is submitted).
4. The code computes: `tokens = 1_000_000 * 10^18 / 10^8 = 1_000_000 * 10^10` — a value 10^10 times larger than the actual balance.
5. The 7-day cap is computed on this inflated `tokens` value, yielding an astronomically large limit.
6. Submit a `TransferSnsTreasuryFunds` proposal to drain the entire treasury. The proposal passes the amount check because the computed cap far exceeds the actual treasury size.
