Audit Report

## Title
Missing Anonymous Owner Validation in `update_balance` Allows Minting ckDOGE to Anonymous Principal's Account — (`rs/dogecoin/ckdoge/minter/src/main.rs`, `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary

The ckDOGE minter's `update_balance` endpoint guards only against an anonymous *caller* via `check_anonymous_caller()`, but performs no validation that `args.owner` is non-anonymous. A non-anonymous attacker can pass `owner = Some(Principal::anonymous())`, bypass the only guard, and cause ckDOGE to be minted into the anonymous principal's account — an account the protocol explicitly prohibits as evidenced by the parallel check in `get_doge_address`. This constitutes illegal minting in violation of the documented protocol invariant, with the minted tokens either permanently locked or accessible to any unauthenticated caller depending on the ICRC-1 ledger's anonymous-caller policy.

## Finding Description

**Guard in place — what it checks:**

`check_anonymous_caller()` in `rs/dogecoin/ckdoge/minter/src/main.rs` (lines 145–149) inspects only `ic_cdk::api::msg_caller()`:

```rust
fn check_anonymous_caller() {
    if ic_cdk::api::msg_caller() == candid::Principal::anonymous() {
        panic!("anonymous caller not allowed")
    }
}
```

It says nothing about `args.owner`.

**The `update_balance` handler** (lines 89–96) calls `check_anonymous_caller()` and then delegates unconditionally to the shared ckBTC logic:

```rust
#[update]
async fn update_balance(args: UpdateBalanceArgs) -> Result<Vec<UtxoStatus>, UpdateBalanceError> {
    check_anonymous_caller();   // ← only checks msg_caller()
    check_postcondition(
        ic_ckbtc_minter::updates::update_balance::update_balance(args, &DOGECOIN_CANISTER_RUNTIME)
            .await,
    )
}
```

**The shared `update_balance` logic** in `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs` (lines 149–167) contains only a guard against the minter's own principal:

```rust
if args.owner.unwrap_or(caller) == runtime.id() {
    ic_cdk::trap("cannot update minter's balance");
}
```

Then constructs the account to credit without any anonymous check:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),
    subaccount: args.subaccount,
};
```

There is no `assert_ne!(owner, Principal::anonymous())` anywhere in this path. A grep for `Principal::anonymous` in `updates/update_balance.rs` returns zero matches, confirmed by the file read.

**Contrast with `get_doge_address`** in `rs/dogecoin/ckdoge/minter/src/updates/get_doge_address.rs` (lines 14–18), which enforces the invariant:

```rust
assert_ne!(
    owner,
    Principal::anonymous(),
    "the owner must be non-anonymous"
);
```

The Candid interface (`ckdoge_minter.did`) documents "The resolved owner must be a non-anonymous principal" for `get_doge_address` but is silent for `update_balance`, confirming the inconsistency is an oversight.

**Exploit path:**

1. The anonymous principal's deposit address is deterministic — derived from the minter's ECDSA public key + anonymous principal bytes via `derivation_path()` in `get_doge_address.rs` (lines 44–54), which is a public computation requiring no secret.
2. Attacker sends DOGE to that address.
3. Attacker (as a non-anonymous principal) calls `update_balance({ owner: Some(Principal::anonymous()), subaccount: None })`.
4. `check_anonymous_caller()` passes (attacker is non-anonymous).
5. The shared logic constructs `caller_account = Account { owner: anonymous, subaccount: None }` and mints ckDOGE to it.

## Impact Explanation

This constitutes **illegal minting** of a ck-token asset to a protocol-prohibited account. The IC protocol permits the anonymous principal to make update calls; any party calling the ckDOGE ICRC-1 ledger as `Principal::anonymous()` can transfer tokens from the anonymous account, meaning the minted ckDOGE is either accessible to any unauthenticated caller (enabling theft of the deposited DOGE backing) or permanently locked if the ledger rejects anonymous callers (causing permanent loss of the ckDOGE and the corresponding DOGE collateral being stranded in the minter's treasury). Either outcome represents a concrete, in-scope ck-token security impact: illegal minting and potential permanent loss of chain-key ledger assets. This maps to the **High** impact tier — significant Chain Fusion / ck-token security impact with concrete user or protocol harm.

## Likelihood Explanation

The attack requires no privileged role, no key compromise, and no governance majority. It requires only: (a) computing the anonymous principal's deposit address (public derivation, no secret needed), (b) sending DOGE to that address, and (c) calling `update_balance` with `owner = Some(Principal::anonymous())` as any non-anonymous principal. The inconsistency between `get_doge_address` (has the check) and `update_balance` (missing the check) makes this a straightforward, reproducible oversight. It is fully triggerable by an unprivileged external user.

## Recommendation

Add an explicit anonymous-owner guard in the shared `update_balance` logic in `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`, immediately after resolving the effective owner and before constructing `caller_account`:

```rust
let effective_owner = args.owner.unwrap_or(caller);
if effective_owner == Principal::anonymous() {
    ic_cdk::trap("the owner must be non-anonymous");
}
```

This mirrors the check already present in `get_doge_address.rs` and enforces the invariant documented in the Candid interface. Placing it in the shared logic ensures both ckBTC and ckDOGE minters are protected.

## Proof of Concept

```rust
// State-machine test mirroring existing patterns in rs/dogecoin/ckdoge/minter/tests/tests.rs
#[test]
fn update_balance_with_anonymous_owner_should_fail() {
    let setup = Setup::default();
    let minter = setup.minter();

    // Inject a UTXO at the anonymous principal's deposit address
    // (derivation is public: derivation_path(&Account { owner: anonymous, subaccount: None }))
    // into the mock Dogecoin canister state.

    let result = minter.update_balance(
        USER_PRINCIPAL,   // non-anonymous caller → passes check_anonymous_caller
        &UpdateBalanceArgs {
            owner: Some(Principal::anonymous()),
            subaccount: None,
        },
    );
    // Currently: succeeds and mints ckDOGE to anonymous account.
    // Expected:  traps with "the owner must be non-anonymous".
    assert!(result.is_err(), "minter must reject anonymous owner");
}
```

The existing test suite already tests that `get_doge_address` rejects `owner = Some(anonymous)` (tests.rs lines 237–262), but no equivalent test exists for `update_balance`, confirming the gap.
