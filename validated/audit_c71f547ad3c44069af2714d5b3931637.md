### Title
ETH Permanently Burned on `SELFDESTRUCT`-to-Self in Same-Transaction Deployment — (`basic_system/src/system_implementation/ethereum_storage_model/caches/account_cache.rs`, `basic_system/src/system_implementation/flat_storage_model/account_cache.rs`)

---

### Summary

In `mark_for_deconstruction`, when a contract calls `SELFDESTRUCT` with itself as the beneficiary **in the same transaction as its creation**, the contract's ETH balance is unconditionally zeroed (burned) rather than being preserved at the address. This diverges from Ethereum EIP-6780 semantics, where a `SELFDESTRUCT(self)` transfer is a no-op and the ETH remains at the address after the contract is destroyed.

---

### Finding Description

The `SELFDESTRUCT` opcode is dispatched through `selfdestruct()` in `evm_interpreter/src/instructions/host.rs`, which calls `system.io.mark_for_deconstruction(THIS_EE_TYPE, ..., &self.address, &beneficiary)`. [1](#0-0) 

Inside `mark_for_deconstruction` (both storage model implementations), the code branches on two conditions:

```rust
let same_address = at_address == nominal_token_beneficiary;
// ...
let should_be_deconstructed =
    account_data.current().metadata().deployed_in_tx == Some(cur_tx) || in_constructor;
``` [2](#0-1) 

The transfer/zero logic is:

```rust
if !same_address {
    self.transfer_nominal_token_value_inner::<PROOF_ENV>(
        from_ee, resources, at_address, nominal_token_beneficiary, &transfer_amount, ...
    ).map_err(wrap_error!())?;
} else if should_be_deconstructed {
    account_data.update(|cache_record| {
        cache_record.update(|v, _| {
            v.balance = U256::ZERO;   // ← ETH burned here, no transfer
            Ok(())
        })
    })?;
}
``` [3](#0-2) [4](#0-3) 

When `same_address == true` **and** `should_be_deconstructed == true` (contract created in the same transaction, or called from a constructor), the balance is set to `U256::ZERO` without any transfer. The function then returns `transfer_amount` to the caller as if a transfer occurred: [5](#0-4) 

Under EIP-6780, `SELFDESTRUCT(self)` in the same transaction as creation must:
1. Transfer ETH to beneficiary (self → no-op, ETH stays at address)
2. Destroy the contract (code and storage cleared)

The address retains its ETH balance after destruction. ZKsync OS instead zeros the balance, permanently destroying the ETH.

The `finish_tx` function also zeros the balance for contracts marked for deconstruction, compounding the issue: [6](#0-5) 

---

### Impact Explanation

Any ETH sent to a contract that calls `SELFDESTRUCT(address(this))` in its constructor (or in the same transaction as its creation) is permanently burned in ZKsync OS. In Ethereum, that ETH would remain at the address and be recoverable (e.g., by redeploying via CREATE2). This is an unconditional, irreversible loss of user funds caused by an EVM semantic mismatch — directly analogous to the original report's pattern of destroying assets that should be preserved.

    pub fn selfdestruct(
        &mut self,
        system: &mut System<S>,
        tracer: &mut impl Tracer<S>,
    ) -> InstructionResult {
        self.gas
            .spend_gas_and_native(gas_constants::SELFDESTRUCT, SELFDESTRUCT_NATIVE_COST)?;

        if self.is_static_frame() {
            return Err(EvmError::StateChangeDuringStaticCall.into());
        }

        let beneficiary = u256_to_b160(self.stack.pop_1()?);

        let amount_transferred = system
            .io
            .mark_for_deconstruction(
                THIS_EE_TYPE,
                self.gas.resources_mut(),
                &self.address,
                &beneficiary,
            )
            .map_err(wrap_error!())?;

        tracer.evm_tracer().on_selfdestruct(
            beneficiary,
            amount_transferred,
            &InterpreterExternal::new_from(&self, system),
        );

        Err(ExitCode::SelfDestruct)
    }

    pub fn finish_tx<P: StorageAccessPolicy<R, Bytes32>>(
        &mut self,
        storage: &mut EthereumStorageCache<A, SF, N, R, P>,
    ) -> Result<(), InternalError> {
        // Actually deconstructing accounts
        self.cache.apply_to_last_record_of_pending_changes(
            |key, (initial, current), cache_appearance| {
                if current.value.metadata().is_marked_for_deconstruction {
                    // NOTE: initially account had 0 nonce, but it could be "material",
                    // with state root being empty, and bytecode hash being hash of empty string.

                    // NOTE: Balance will be zeroed out if deconstruction happens here
                    let initially_empty = cache_appearance.is_new_element();
                    assert!(cache_appearance.is_value_observed());
                    current.value.update(|x, metadata| {
                        metadata.is_marked_for_deconstruction = false;
                        if initially_empty {
                            debug_assert_eq!(
                                initial.value.value(),
                                &EthereumAccountProperties::EMPTY_ACCOUNT
                            );
                            x.balance = U256::ZERO;
                            x.bytecode_hash = Bytes32::ZERO;
                            x.nonce = 0u64;
                        } else {
                            //
                            debug_assert_eq!(initial.value.value().nonce, 0);
                            debug_assert_eq!(
                                initial.value.value().bytecode_hash,
                                EMPTY_STRING_KECCAK_HASH
                            );
                            debug_assert_eq!(initial.value.value().storage_root, EMPTY_ROOT_HASH);
                            x.balance = U256::ZERO;
                            x.bytecode_hash = EMPTY_STRING_KECCAK_HASH;
                            x.nonce = 0u64;
                        }

                        Ok(())
                    })?;
                    storage
                        .slot_values
                        .clear_state_impl(key)
                        .expect("must clear state for code deconstruction in same TX");
                }
                Ok(())
            },
        )?;
