### Title
`tx_number_in_block` Silently Truncated via Unbounded `current_transaction_number` Cast to `u16` — (File: `zk_ee/src/common_structs/logs_storage.rs`)

---

### Summary

`L2ToL1Log.tx_number_in_block` is a `u16` field serialized into the L2→L1 log Merkle tree. The internal transaction counter `current_transaction_number` is a `u32` that is incremented without an upper-bound check. When a `LogContent` entry is converted to `L2ToL1Log`, the cast `m.tx_number as u16` silently truncates any value above 65535. This corrupts the `tx_number_in_block` field in the encoded log, causing two distinct transactions to share the same identifier in the log tree, and producing an incorrect `l2_logs_tree_root` in the batch public input.

---

### Finding Description

`L2ToL1Log` declares `tx_number_in_block` as a `u16`: [1](#0-0) 

The field is serialized as exactly 2 bytes in the log encoding used for the Merkle tree: [2](#0-1) 

The internal log store holds `tx_number: u32` per entry. The conversion from `LogContent` to `L2ToL1Log` performs an unchecked narrowing cast: [3](#0-2) 

The source of that `u32` value is `current_transaction_number` in `ZKBasicBlockDataKeeper`, which is declared as `u32`: [4](#0-3) 

It is incremented by 1 for every successfully processed transaction, with no upper-bound guard: [5](#0-4) 

There is no check anywhere that `current_transaction_number < u16::MAX` (65535) before the value is stored in a log entry or before the cast occurs.

The `MAX_NUMBER_OF_LOGS` constant (16,384) limits the Merkle tree leaf count, not the transaction counter: [6](#0-5) 

A block with more than 65535 successful transactions but fewer than 16,384 total log emissions (e.g., many transactions that do not emit user messages) would silently wrap `tx_number_in_block`, causing transaction 65536 to appear as transaction 0 in its logs.

---

### Impact Explanation

The `l2_logs_tree_root` is computed over the hashes of all `L2ToL1Log` entries and is committed to in the batch public input: [7](#0-6) [8](#0-7) 

If `tx_number_in_block` is truncated, the leaf hashes are wrong, the Merkle root is wrong, and the `BatchOutput.l2_logs_tree_root` committed on-chain is incorrect. L1 contracts that use `tx_number_in_block` to verify or route L2→L1 messages (e.g., priority queue processing, withdrawal proofs) receive a corrupted identifier, breaking the integrity of cross-layer message attribution. This is a **state-transition / L1 integration correctness bug** with potential for valid-execution unprovability if the forward runner and prover diverge on the truncated value.

---

### Likelihood Explanation

Reaching 65536 successful transactions in a single block requires a very high transaction throughput within one block's gas budget. With ZKsync OS targeting 10,000 TPS and very cheap transactions (e.g., simple ETH transfers), a sufficiently large block gas limit could allow this. The missing check is a latent defect: no attacker needs any privileged role — any unprivileged sender can submit transactions. The risk grows as block gas limits increase or as the system is used for high-frequency, low-gas operations.

---

### Recommendation

Add an explicit check that `current_transaction_number` does not exceed `u16::MAX` before incrementing, or before storing the value in a log entry. For example, in the transaction loop:

```rust
if block_data.current_transaction_number >= u16::MAX as u32 {
    return Err(/* block-level error: too many transactions */);
}
block_data.current_transaction_number += 1;
```

Alternatively, change `tx_number_in_block` to `u32` throughout the log encoding and update the corresponding L1 contract ABI, or enforce the bound at the point of log creation in `push_message` / `push_l1_l2_tx_log`.

---

### Proof of Concept

1. Construct a block containing 65537 minimal valid transactions (e.g., self-transfers with sufficient gas), where each transaction emits exactly one L2→L1 log (e.g., an L1→L2 priority transaction).
2. The 65537th transaction has `current_transaction_number = 65536`. When its log is stored, `tx_number = 65536u32`.
3. At Merkle tree finalization, `From<&LogContent> for L2ToL1Log` executes `tx_number_in_block: 65536u32 as u16 = 0u16`.
4. Transaction 65537's log now has `tx_number_in_block = 0`, identical to transaction 1's log.
5. The resulting `l2_logs_tree_root` differs from the correct value, corrupting the batch public input committed on L1. [9](#0-8)
