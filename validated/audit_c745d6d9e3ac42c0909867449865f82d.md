Audit Report

## Title
Stale Voting Power Snapshot Used for Ballot Creation Allows Inflated Maturity Rewards via Stake Reduction After Snapshot - (File: `rs/nns/governance/src/governance/voting_power_snapshots.rs`)

## Summary

When a voting power spike is detected at proposal creation time, `compute_ballots_for_standard_proposal` initializes ballots from a historical snapshot (up to 7 days old, or up to 3 months old per `MAXIMUM_STALENESS_SECONDS`) rather than the current live neuron state. A neuron owner can reduce their stake after the snapshot is recorded but before a proposal is created during a spike window, causing the proposal to assign that neuron a ballot with the old (higher) voting power. The neuron then votes and earns maturity rewards proportional to the inflated ballot, even though the stake backing that voting power has already been withdrawn. Because maturity is eventually minted as ICP via `DisburseMaturity`, this constitutes illegal ICP minting not backed by current stake.

## Finding Description

**Root cause — stale snapshot selection:**

`VotingPowerSnapshots` stores up to `MAX_VOTING_POWER_SNAPSHOTS = 7` daily snapshots. When `previous_ballots_if_voting_power_spike_detected` is called, it selects the snapshot with the **minimum** `total_potential_voting_power` among all non-stale entries:

```rust
.min_by_key(|(_, snapshot)| snapshot.total_potential_voting_power)?;
``` [1](#0-0) 

This snapshot can be up to 7 days old under normal operation, or up to `MAXIMUM_STALENESS_SECONDS = ONE_MONTH_SECONDS * 3` if the daily task has been failing. [2](#0-1) 

**Root cause — snapshot store freezes during a spike:**

`SnapshotVotingPowerTask::execute` explicitly skips recording a new snapshot when the latest stored snapshot is itself a spike:

```rust
if self.snapshots.with_borrow(|snapshots| snapshots.is_latest_snapshot_a_spike(now_seconds)) {
    return (VOTING_POWER_SNAPSHOT_INTERVAL, self);
}
``` [3](#0-2) 

This means once a spike is triggered, the snapshot store stops updating, keeping the minimum-total snapshot frozen at its old state indefinitely (confirmed by the test at line 216–225 of the same file).

**Root cause — ballots created directly from stale snapshot without current-state validation:**

`create_ballots_and_total_potential_voting_power` converts the snapshot's `voting_power_map` directly into `Ballot` structs with no cross-check against current neuron state:

```rust
let ballots = voting_power_map
    .into_iter()
    .map(|(neuron_id, voting_power)| {
        (neuron_id, Ballot { voting_power, vote: Vote::Unspecified as i32 })
    })
    .collect();
``` [4](#0-3) 

**Root cause — reward distribution does not re-validate current neuron stake:**

`RewardsDistribution::continue_processing` credits maturity to neurons based on pre-computed `reward_e8s` values (derived from ballot `voting_power`) without any check against the neuron's current `cached_neuron_stake_e8s`:

```rust
neuron.maturity_e8s_equivalent =
    neuron.maturity_e8s_equivalent.saturating_add(reward_e8s);
``` [5](#0-4) 

**Exploit flow:**

1. Over 7 days, 7 snapshots are recorded. Attacker's neuron N has stake S (e.g., 10,000 ICP) in all snapshots. Minimum snapshot total = X.
2. Attacker stakes a large amount in a new neuron, pushing current total > 1.5 × X. `SnapshotVotingPowerTask` detects the spike and stops recording new snapshots.
3. Attacker partially disburses neuron N (reduces stake to near zero, keeping the neuron alive). The neuron is now excluded from the current live snapshot but still present in the frozen old snapshot.
4. Any user creates a standard proposal. `compute_ballots_for_standard_proposal` detects the spike and calls `previous_ballots_if_voting_power_spike_detected`, which returns the old frozen snapshot. Neuron N receives a ballot with `voting_power = VP_old` (based on the old 10,000 ICP stake).
5. Attacker votes using neuron N's ballot (the ballot exists in the proposal regardless of current neuron state).
6. At reward distribution time, `distribute_voting_rewards_to_neurons` credits neuron N with maturity proportional to `VP_old`, even though the neuron now holds near-zero stake.
7. Attacker calls `DisburseMaturity` on neuron N, minting ICP from the governance canister's minting account — ICP not backed by current stake.

## Impact Explanation

This is a **ledger conservation violation**: maturity is credited to a neuron based on a stale, inflated ballot voting power, and that maturity is subsequently minted as ICP via `DisburseMaturity`. The attacker withdraws their original stake and also receives ICP rewards as if the stake were still present. This constitutes illegal ICP minting. The impact maps to the **High** bounty tier: significant NNS governance/ledger security impact with concrete user and protocol harm, where exploitation requires meaningful per-target work (capital to trigger the spike, precise timing).

## Likelihood Explanation

**Medium.** The attacker must cause the current total potential voting power to exceed 1.5× the minimum stored snapshot total. On the live NNS, this requires either a large coordinated stake increase or waiting for organic conditions where the threshold is nearly met. The 7-day snapshot window and the freeze behavior (snapshot store stops updating during a spike) extend the exploitation window once the spike is triggered. No privileged access is required — staking, disbursing, and creating proposals are all standard unprivileged operations. The attack is repeatable as long as the spike condition can be re-established.

## Recommendation

1. **Re-validate ballot voting power at reward distribution time**: In `distribute_voting_rewards_to_neurons` / `continue_processing`, cap each neuron's `reward_e8s` at a value proportional to its current `cached_neuron_stake_e8s` before crediting maturity.
2. **Do not freeze the snapshot store during a spike**: Remove the early-return in `SnapshotVotingPowerTask::execute` that skips recording when the latest snapshot is a spike. New snapshots should always be recorded so the store does not become arbitrarily stale.
3. **Reduce `MAXIMUM_STALENESS_SECONDS`**: The current 3-month maximum staleness is far too large; reduce it to match the 7-day operational window.
4. **Invalidate ballots for neurons whose current stake is materially lower than their ballot voting power**: At settlement time, check whether the neuron's current dissolve delay and stake are consistent with the ballot's voting power, and reduce the reward proportionally if not.

## Proof of Concept

**Deterministic integration test plan (PocketIC or local replica):**

1. Initialize governance with neuron N (10,000 ICP, sufficient dissolve delay). Add a second neuron M (1 ICP) to represent the rest of the network. Total VP ≈ 10,001.
2. Advance time by 1 day and call `SnapshotVotingPowerTask::execute` 7 times, recording 7 snapshots. Minimum snapshot total ≈ 10,001.
3. Add a new neuron P with 6,000 ICP (pushing current total to ≈ 16,001 > 1.5 × 10,001 = 15,001.5). Confirm `is_latest_snapshot_a_spike` returns `true` and the next `execute` call skips recording.
4. Reduce neuron N's `cached_neuron_stake_e8s` to 1 (simulating partial disbursal). Confirm neuron N is now excluded from `compute_voting_power_snapshot_for_standard_proposal`.
5. Create a standard proposal. Assert that `previous_ballots_if_voting_power_spike_detected` returns `Some(...)` and that neuron N's ballot has `voting_power` ≈ its old value (≫ 1).
6. Vote with neuron N. Advance time past the reward period. Call `distribute_voting_rewards_to_neurons`.
7. Assert that neuron N's `maturity_e8s_equivalent` has increased by an amount proportional to its old (inflated) ballot voting power, not its current near-zero stake.
8. Call `DisburseMaturity` on neuron N and confirm ICP is minted despite the neuron holding near-zero stake — demonstrating the ledger conservation violation.
