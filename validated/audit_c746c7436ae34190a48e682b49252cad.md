### Title
SNS Swap New-Participant Blocking via Residual-Capacity Griefing — (File: `rs/sns/swap/src/swap.rs`)

---

### Summary

An unprivileged ICP holder can participate in an open SNS swap with a carefully chosen amount that leaves the remaining direct-participation capacity (`available_direct_participation_e8s`) strictly between zero and `min_participant_icp_e8s`. Once in this state, every subsequent call to `refresh_buyer_token_e8s` by a **new** participant is rejected, because the effective amount the swap can accept is capped below the per-participant minimum. If `min_participants` has not yet been reached, the swap will abort at its deadline and all ICP — including the attacker's — is refunded, making the attack nearly free.

---

### Finding Description

Inside `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs`:

1. The maximum increment the swap can still accept is computed:
   ```
   let max_increment_e8s = self.available_direct_participation_e8s();   // line 1177
   ``` [1](#0-0) 

2. The actual increment is capped at that remaining capacity:
   ```
   let actual_increment_e8s = std::cmp::min(max_increment_e8s, requested_increment_e8s);  // line 1224
   let new_balance_e8s = old_amount_icp_e8s.saturating_add(actual_increment_e8s);          // line 1225
   ``` [2](#0-1) 

3. The resulting effective balance is then checked against the per-participant minimum:
   ```
   if new_balance_e8s < params.min_participant_icp_e8s {   // line 1241
       return Err(...)
   }
   ``` [3](#0-2) 

For a **new** participant `old_amount_icp_e8s = 0`, so `new_balance_e8s = actual_increment_e8s = min(available, requested)`. If `available < min_participant_icp_e8s`, the check at line 1241 always fires regardless of how much ICP the new participant transfers.

An attacker achieves this state by calling `refresh_buyer_token_e8s` with an amount equal to:

```
max_direct_participation_e8s − (min_participant_icp_e8s − 1)
```

This leaves exactly `min_participant_icp_e8s − 1` e8s of remaining capacity — one e8 below the threshold — permanently blocking new entrants.

`available_direct_participation_e8s` is simply `max − current`: [4](#0-3) 

The `validate_possibility_of_direct_participation` guard does **not** fire because the ICP target is not yet reached (current < max): [5](#0-4) 

Existing participants are unaffected because their `old_amount_icp_e8s ≥ min_participant_icp_e8s`, so `new_balance_e8s` stays above the minimum even after the tiny increment is added.

---

### Impact Explanation

If `min_participants` has not been satisfied before the attacker acts, no new participant can join. The swap will reach its deadline without meeting the minimum-participant requirement and will abort (`try_abort` succeeds when the deadline passes and `sufficient_participation()` is false). All ICP — including the attacker's — is refunded via `error_refund_icp`. The SNS launch is permanently cancelled for that swap instance.

The test `test_try_commit_or_abort_insufficient_participation_with_max_icp` in `rs/sns/swap/tests/swap.rs` explicitly confirms that a swap aborts when max ICP is reached but `min_participants` is not met: [6](#0-5) 

The test `test_swap_cannot_finalize_via_new_participation_if_remaining_lt_minimal_participation_amount` further confirms that new participants are rejected in exactly this residual-capacity scenario: [7](#0-6) 

---

### Likelihood Explanation

The attack requires only:
- Holding enough ICP to fill the swap to within `min_participant_icp_e8s − 1` e8s of its cap.
- Calling `refresh_buyer_token_e8s` once before `min_participants` is reached.

The attacker bears **no net financial cost**: ICP is refunded on abort. The only cost is the opportunity cost of locking ICP for the swap duration (days to weeks). Any ICP holder with sufficient balance can execute this unilaterally, with no privileged access required. The entry point is the public `refresh_buyer_tokens` ingress method of the SNS swap canister.

The feasibility depends on `max_participant_icp_e8s` being large enough relative to `max_direct_participation_e8s`. When `max_participant_icp_e8s ≈ max_direct_participation_e8s` (a common configuration for smaller SNS launches), a single actor can execute the attack alone. Otherwise, two colluding actors suffice.

---

### Recommendation

After capping `actual_increment_e8s` at `available_direct_participation_e8s`, check whether the remaining capacity after the new participant's contribution would fall below `min_participant_icp_e8s`. If so, either:

1. **Accept the full remaining capacity** for this participant (waiving the minimum for the final slot), or
2. **Reject the participation** with a clear error explaining that the swap can no longer accept new participants (only existing ones may top up), and emit a canister log / metric so the SNS team can observe the condition.

The key invariant to enforce is: a participation should only be accepted if it either (a) fills the swap to its cap, or (b) leaves at least `min_participant_icp_e8s` of remaining capacity for future participants.

### Proof of Concept

Given a swap configured with:
- `max_direct_participation_icp_e8s = 500_000 * E8`
- `min_participant_icp_e8s = 1 * E8`
- `max_participant_icp_e8s = 500_000 * E8`
- `min_participants = 3`

**Step 1 — Attacker participates:**
Attacker transfers `499_999 * E8 + (E8 - 1)` ICP to their swap subaccount and calls `refresh_buyer_token_e8s`.
- `available_direct_participation_e8s()` becomes `1` (one e8).
- Attacker is recorded as buyer #1.

**Step 2 — Legitimate user attempts to join:**
User transfers `1 * E8` ICP and calls `refresh_buyer_token_e8s`.
- `max_increment_e8s = 1` (one e8, the remaining capacity).
- `actual_increment_e8s = 1`.
- `new_balance_e8s = 1 < min_participant_icp_e8s (= E8)`.
- **Rejected**: `"Rejecting participation of effective amount 1; minimum required to participate: 100000000"`.

**Step 3 — Swap deadline passes:**
Only 1 of 3 required participants joined. `try_abort` succeeds. Attacker's ICP is refunded in full. SNS launch fails. [1](#0-0) [8](#0-7)
