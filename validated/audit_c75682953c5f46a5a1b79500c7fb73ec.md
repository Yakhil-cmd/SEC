Audit Report

## Title
Unbounded `SuspendedUtxos` Heap Growth via Dust UTXOs Across Distinct Subaccounts â€” (`rs/bitcoin/ckbtc/minter/src/state.rs`, `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary

An unprivileged depositor can send a sub-minimum-deposit ("dust") UTXO to each of N distinct subaccount-derived Bitcoin addresses and call `update_balance` once per subaccount. Each call unconditionally inserts a new `Account`-keyed entry into `SuspendedUtxos::utxos` with no size bound or eviction mechanism, growing the ckBTC minter's heap proportionally to the number of distinct subaccounts ever targeted. Sufficient entries will exhaust the Wasm heap and trap the minter canister, halting all ckBTC minting and withdrawal operations.

## Finding Description

**Root cause â€” no capacity guard in `SuspendedUtxos::insert`:**

`SuspendedUtxos` holds three unbounded maps with no enforced size limit:

```rust
pub struct SuspendedUtxos {
    utxos_without_account: BTreeMap<Utxo, SuspendedReason>,
    utxos: BTreeMap<Account, BTreeMap<Utxo, SuspendedReason>>,
    last_time_checked_cache: BTreeMap<Utxo, Timestamp>,
}
``` [1](#0-0) 

`SuspendedUtxos::insert` performs a same-account/same-UTXO/same-reason deduplication check but has no per-account cap, no global cap, and no rejection path:

```rust
let utxos = self.utxos.entry(account).or_default();
utxos.insert(utxo, reason);
``` [2](#0-1) 

**Exploit call chain:**

1. **Guard is concurrent-only.** `balance_update_guard` enforces at most 100 *simultaneous* in-flight requests. Sequential calls across distinct subaccounts are entirely unlimited. [3](#0-2) [4](#0-3) 

2. **New dust UTXO enters `processable_utxos`.** `processable_utxos_for_account` classifies a UTXO not yet in `suspended_utxos` as a new processable UTXO: [5](#0-4) 

3. **`satoshis_to_mint == 0` guard does not fire.** The existing memory-exhaustion guard bails out only when the sum of processable UTXO values is zero. A dust UTXO worth 546 satoshis yields `satoshis_to_mint = 546`, so execution continues past the guard: [6](#0-5) 

4. **Dust UTXO is suspended.** The processing loop detects `utxo.value < deposit_btc_min_amount` and calls `audit::ignore_utxo`: [7](#0-6) 

5. **Unbounded insertion.** `audit::ignore_utxo` â†’ `discard_utxo` â†’ `state.suspend_utxo` â†’ `suspended_utxos.insert(account, utxo, ValueTooSmall, now)`. A new `BTreeMap` node keyed by `Account{owner, subaccount=i}` is created with no capacity check: [8](#0-7) [9](#0-8) 

6. **`last_time_checked_cache` also grows.** Each insertion also adds a `Utxo â†’ Timestamp` entry to `last_time_checked_cache` with no bound: [10](#0-9) 

**No removal path exists** for these entries unless the UTXO is spent on-chain or the deposit minimum is lowered so the UTXO becomes processable. Entries persist indefinitely.

## Impact Explanation

This is a **High** severity application-level DoS. At ~358 bytes per subaccount entry, 10,000,000 entries consume ~3.58 GB, approaching the Wasm heap limit and causing the minter canister to trap on allocation. This halts all ckBTC minting and withdrawal operations â€” a significant Chain Fusion / ck-token security impact with concrete user and protocol harm, matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"* and *"Significant Chain Fusion, ck-token, ledger â€¦ security impact with concrete user or protocol harm."*

## Likelihood Explanation

Each entry requires one real on-chain Bitcoin transaction. At ~$0.075 per entry (546-satoshi dust + ~750-satoshi fee at $100k/BTC), causing noticeable memory pressure (100,000 entries, ~35 MB) costs ~$7,500; causing OOM (~10M entries) costs ~$750,000. The barrier is high but not prohibitive for a motivated attacker targeting a protocol managing hundreds of millions of dollars in BTC. The attack requires no special privileges â€” any Bitcoin holder can execute it â€” and can be parallelized by packing many dust outputs into a single Bitcoin transaction, reducing per-entry confirmation wait time.

## Recommendation

1. **Per-account cap:** Enforce a maximum number of suspended UTXOs per `Account` (e.g., 10) inside `SuspendedUtxos::insert`. Silently drop new entries beyond the cap.
2. **Global cap:** Enforce a global cap on `suspended_utxos.num_utxos()` and return an error from `update_balance` if the cap is exceeded.
3. **Extend the `satoshis_to_mint == 0` guard:** Before entering the processing loop, check whether all processable UTXOs are dust (value < `deposit_btc_min_amount`). If so, bail out before calling `audit::ignore_utxo` for first-time dust UTXOs, avoiding the suspension insertion entirely.
4. **Rate-limit `update_balance` per principal per time window** to reduce the throughput of the attack.

pub fn ignore_utxo<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    utxo: Utxo,
    account: Account,
    now: Timestamp,
    runtime: &R,
) {
    discard_utxo(
        state,
        utxo,
        account,
        SuspendedReason::ValueTooSmall,
        now,
        runtime,
    );
}

fn discard_utxo<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    utxo: Utxo,
    account: Account,
    reason: SuspendedReason,
    now: Timestamp,
    runtime: &R,
) {
    // ignored UTXOs are periodically re-evaluated and should not trigger
    // an event if they are still ignored.
    if state.suspend_utxo(utxo.clone(), account, reason, now) {
        record_event(
            EventType::SuspendedUtxo {
                utxo,
                account,
                reason,
            },
            runtime,
        )
    }
}
