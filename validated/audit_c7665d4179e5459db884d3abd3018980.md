Audit Report

## Title
Unauthenticated `buyer` Field in `refresh_buyer_tokens` Enables Confirmation-Text Bypass and Forced SNS Swap Participation — (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` update method accepts a caller-supplied `buyer` principal with no check that it matches the authenticated caller. Because the SNS `confirmation_text` — a consent gate intended to prove the buyer read and accepted swap terms — is stored in publicly readable canister state, any unprivileged third party can read it and submit it on behalf of any victim who has already transferred ICP to their swap subaccount, committing that victim's participation without their knowledge or consent and deleting their open ticket.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 130–134, when `arg.buyer` is non-empty the canister uses the caller-supplied principal directly with no identity assertion:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()   // no caller check
};
```

This `p` is passed directly to `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs`. Inside that function, `validate_confirmation_text` (lines 363–384) performs only a string equality check against `self.init_or_panic().confirmation_text` — it does not verify that the caller is the buyer. The `confirmation_text` is part of the `Init` struct returned by the public `#[query]` endpoint `get_init` (canister.rs line 213), making it trivially readable by anyone.

**Exploit path:**
1. Victim calls `new_sale_ticket` for amount X ICP (authenticated to caller).
2. Victim transfers X ICP to their swap subaccount on the ICP ledger.
3. Attacker queries `get_init` to obtain `confirmation_text`.
4. Attacker calls `refresh_buyer_tokens { buyer: "<victim_principal>", confirmation_text: "<text from step 3>" }`.
5. The swap canister reads the victim's ledger balance (X), passes `validate_confirmation_text` (string matches), reaches the ticket block at lines 1248–1271: since `amount_ticket <= requested_increment_e8s`, the victim's ticket is deleted and participation is committed under the victim's principal.
6. The victim's ICP is now locked into the swap with the confirmation text "accepted" by the attacker, not the victim.

Existing guards are insufficient: `validate_lifecycle_is_open` and `validate_possibility_of_direct_participation` are lifecycle checks only; `validate_confirmation_text` is a pure string comparison with no caller-binding; the minimum-participant check at line 1202 is satisfied because the victim already transferred the required ICP.

## Impact Explanation
This is a **High** severity SNS security impact. The `confirmation_text` mechanism is the protocol's only tool for obtaining affirmative buyer consent to swap terms (e.g., legal disclaimers, risk acknowledgements). An unprivileged attacker can forge that consent for any victim who has deposited ICP, committing their funds to the swap without the victim ever having agreed. If the swap commits, the victim receives SNS tokens they did not explicitly choose to acquire. The victim's open ticket is also deleted, disrupting the ticket-based payment flow. This constitutes concrete user harm within the SNS framework and matches the allowed impact: *"Significant SNS security impact with concrete user or protocol harm."*

## Likelihood Explanation
No special privilege is required. The victim's principal is observable from public query endpoints (`list_direct_participants`, `get_buyer_state`). The `confirmation_text` is readable from the public `get_init` query. The attack requires only that the victim has transferred ICP to their subaccount (a normal step in the participation flow). It is trivially scriptable and repeatable at negligible cost (only ICP ledger query cycles).

## Recommendation
1. **Bind `buyer` to the caller**: in `refresh_buyer_tokens`, if `arg.buyer` is non-empty, assert `PrincipalId::from_str(&arg.buyer) == Ok(caller_principal_id())` and reject mismatches with an explicit error.
2. **Alternatively**, remove the `buyer` field entirely and always derive the target from `caller_principal_id()`, consistent with how `new_sale_ticket` already operates (canister.rs line 234).
3. If the `buyer` override is intentionally kept for automation use-cases, document it explicitly and remove the `confirmation_text` requirement for third-party callers, or require the confirmation text to be signed by the buyer's key rather than supplied as a plain string.

## Proof of Concept
Minimal deterministic integration test using PocketIC:

```rust
// 1. Open swap with confirmation_text = "I agree"
// 2. victim_principal transfers min_participant_icp_e8s to their subaccount
// 3. victim calls new_sale_ticket(amount = min_participant_icp_e8s)
// 4. attacker (different principal) calls:
//      refresh_buyer_tokens(buyer = victim_principal.to_string(),
//                           confirmation_text = Some("I agree"))
// 5. Assert: get_buyer_state(victim_principal).icp_accepted_participation_e8s > 0
//            (participation committed by attacker, not victim)
// 6. Assert: get_open_ticket(victim_principal) returns None
//            (ticket deleted by attacker's call)
// 7. Assert: victim never called refresh_buyer_tokens themselves
```

This can be implemented directly using the existing `pocket_ic_helpers::swap::refresh_buyer_tokens` helper (integration_tests/src/pocket_ic_helpers.rs lines 2762–2774) by passing `attacker` as the agent identity but `victim` as the `buyer` argument.
