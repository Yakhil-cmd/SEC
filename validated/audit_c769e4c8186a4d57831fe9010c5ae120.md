### Title
Unchecked Low-Level Precompile Call Return Value Causes Silent Token Burn Without Bridge Transfer - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20` and `EvmErc20V2` burn the caller's ERC-20 tokens before invoking the Aurora exit precompile via a low-level assembly `call`. The return value of that `call` is captured in a local variable `res` but is **never checked**. If the precompile call fails for any reason, the function returns successfully, the tokens are permanently destroyed, and no NEP-141 tokens are ever transferred to the user.

---

### Finding Description

In both `withdrawToNear` and `withdrawToEthereum` of `EvmErc20.sol` and `EvmErc20V2.sol`, the execution order is:

1. `_burn(_msgSender(), amount)` — ERC-20 tokens are irreversibly destroyed.
2. An inline assembly `call` is made to the Aurora exit precompile address.
3. The return value `res` is assigned but **never read or validated**.

`EvmErc20.sol` `withdrawToNear` (lines 53–63):
```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked
    }
}
```

The identical unchecked pattern appears in `withdrawToEthereum` (lines 65–76) of `EvmErc20.sol`, and in both `withdrawToNear` (lines 53–64) and `withdrawToEthereum` (lines 66–77) of `EvmErc20V2.sol`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

The `ExitToNear` and `ExitToEthereum` precompiles return `Err(ExitError::...)` in multiple documented paths, including `ERR_INVALID_IN_STATIC`, `ERR_INVALID_IN_DELEGATE`, `ERR_ETH_ATTACHED_FOR_ERC20_EXIT`, `ExitError::OutOfGas`, and any failure from `get_nep141_from_erc20`. When a precompile returns an error, the EVM's `CALL` opcode sets the return value to `0` (failure). Because the Solidity code never checks `res`, the outer function does not revert. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing / destruction of user funds.**

When the precompile call fails silently:
- The user's ERC-20 tokens have already been burned by `_burn`. This is irreversible.
- No NEP-141 `ft_transfer` promise is ever scheduled on NEAR.
- The user receives nothing on either side of the bridge.
- There is no refund path because the EVM transaction itself succeeded (no revert).

The burned tokens represent real value locked in the Aurora bridge accounting. The corresponding NEP-141 balance remains on the Aurora engine account with no mechanism to recover it for the affected user. [1](#0-0) 

---

### Likelihood Explanation

**Medium.** The precompile call can fail under realistic, non-adversarial conditions:

1. **Out-of-gas**: If the EVM transaction is submitted with a gas limit that is sufficient to reach the `_burn` but leaves insufficient gas for the precompile's `required_gas` check (`EXIT_TO_NEAR_GAS`), the precompile returns `ExitError::OutOfGas` and the `call` returns 0.
2. **Unregistered ERC-20**: If `get_nep141_from_erc20` cannot find the NEP-141 mapping (e.g., storage corruption, or a token deployed outside the normal factory flow), the precompile returns an error.
3. **Delegatecall context**: Any contract that wraps `withdrawToNear` via `delegatecall` triggers `ERR_INVALID_IN_DELEGATE`, silently failing after the burn.

An unprivileged EVM user calling `withdrawToNear` or `withdrawToEthereum` directly triggers this path with no special preconditions beyond holding a token balance. [7](#0-6) 

---

### Recommendation

Check the return value of the low-level `call` and revert if it is zero. Replace the unchecked assembly block with:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply this fix to all four assembly blocks across `EvmErc20.sol` and `EvmErc20V2.sol`. Alternatively, restructure the functions to call the precompile first and only burn tokens after confirming success, eliminating the burn-before-call ordering risk entirely. [8](#0-7) [9](#0-8) 

---

### Proof of Concept

1. Deploy `EvmErc20` (or `EvmErc20V2`) on Aurora with a valid NEP-141 mapping.
2. Mint tokens to an attacker-controlled address.
3. Submit a call to `withdrawToNear(recipient, amount)` with a gas limit calibrated so that gas is exhausted inside the precompile (after `_burn` executes but before the precompile completes). The `ExitToNear` precompile checks `required_gas > target_gas` and returns `ExitError::OutOfGas`.
4. The assembly `call` returns `0`. `res` is never checked. The function returns without reverting.
5. Observe: the caller's ERC-20 balance is zero (tokens burned), but no NEP-141 `ft_transfer` was ever issued. The tokens are permanently lost. [1](#0-0) [10](#0-9)
