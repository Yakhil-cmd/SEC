## Analysis

Let me trace the full call path to determine if the panic is reachable.

**Step 1: `write_g1`/`write_g2` — the panic site** [1](#0-0) 

Both functions call `.map_err(|_| ()).unwrap()` — a hard panic if `try_extend` returns `Err`. Compare to the pairing precompile, which correctly propagates the error: [2](#0-1) 

**Step 2: `SliceVec::try_extend` — when does it return `Err`?** [3](#0-2) 

It returns `Err(())` the moment capacity is exhausted. The `SliceVec` is backed by the `return_memory` slice passed into the hook.

**Step 3: How `return_memory` is sized when the hook is called** [4](#0-3) 

The hook receives `core::mem::take(&mut self.return_memory)` — the **entire remaining** global return-data buffer. There is no pre-check that the remaining buffer is ≥ 128 bytes before dispatching to the BLS12 hook.

**Step 4: How the buffer is consumed** [5](#0-4) 

Every completed regular call consumes `returndata.len()` bytes from the buffer permanently. The buffer is never reset within a transaction execution.

**Step 5: Error handling in `pure_system_function_hook_impl`** [6](#0-5) 

A `FatalRuntimeError` (e.g., `out_of_return_memory!()`) would be propagated correctly. But `write_g1`/`write_g2` never return an error — they panic unconditionally via `.unwrap()`.

## Verdict

### Title
Unconditional `.unwrap()` in `write_g1`/`write_g2` causes panic when return memory is exhausted — (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/mod.rs`)

### Summary
`write_g1` and `write_g2` call `output.try_extend(buffer).map_err(|_| ()).unwrap()`. If the backing `SliceVec` has fewer than 128 bytes (G1) or 256 bytes (G2) of remaining capacity, `try_extend` returns `Err(())` and `.unwrap()` panics. The capacity is the remaining global return-memory buffer, which is a finite, monotonically-shrinking resource consumed by every call that returns data. An unprivileged caller can exhaust it to below the threshold, then call any G1/G2 BLS12 precompile with valid inputs to trigger the panic.

### Finding Description
All BLS12 G1/G2 output precompiles (G1 addition, G2 addition, G1 MSM, G2 MSM, G1 mapping, G2 mapping) route through `write_g1`/`write_g2`: [7](#0-6) [8](#0-7) 

The pairing precompile — the only BLS12 precompile that does **not** use `write_g1`/`write_g2` — correctly uses `out_of_return_memory!()` and `?` to propagate the error as a `FatalRuntimeError`. The G1/G2 output precompiles do not.

The attack:
1. Deploy a contract that returns a controlled number of bytes.
2. Call it repeatedly to consume the return-memory buffer down to `K` bytes, where `K < 128` (for G1) or `K < 256` (for G2).
3. Call any BLS12 G1 precompile (e.g., `BLS12_G1ADD`) with a valid 256-byte input.
4. The precompile computes successfully, then `write_g1` attempts to extend the 128-byte result into a `SliceVec` of capacity `K < 128`. `try_extend` returns `Err(())`. `.unwrap()` panics.

### Impact Explanation
A panic in the RISC-V ZKsync OS execution environment aborts the entire block execution. The block cannot be finalized or proved. This is a liveness/DoS vulnerability against block production: a single crafted transaction can halt the sequencer's ability to produce a valid block.

### Likelihood Explanation
The attack requires no privileged access. The attacker only needs to:
- Know the initial size of the return-memory buffer (a fixed system constant).
- Spend enough gas to consume the buffer down to below 128 bytes.
- Submit a valid BLS12 G1/G2 precompile call.

All three conditions are achievable by any unprivileged user.

### Recommendation
Change `write_g1` and `write_g2` to return `Result` and propagate the error using `out_of_return_memory!()`, matching the pattern used in the pairing precompile:

```rust
fn write_g1<D: TryExtend<u8> + ?Sized>(
    el: G1Affine,
    output: &mut D,
) -> Result<(), SubsystemError<Bls12PrecompileErrors>> {
    let mut buffer = [0u8; G1_SERIALIZATION_LEN];
    crypto::bls12_381::eip2537::serialize_g1_bytes(el, &mut buffer);
    output.try_extend(buffer).map_err(|_| out_of_return_memory!())
}
```

Update all call sites in `addition.rs`, `msm.rs`, and `mappings.rs` to propagate the error with `?`.

    fn try_extend<I>(&mut self, iter: I) -> Result<(), Self::Error>
    where
        I: IntoIterator<Item = T>,
    {
        let it = iter.into_iter();
        let cap = self.memory.len();
        let mut idx = self.length;

        // Fill until either iterator ends or capacity is exhausted
        for item in it {
            if idx == cap {
                // Ran out of space, partial write!
                return Err(());
            }
            self.memory[idx].write(item);
            idx += 1;
        }

        self.length = idx;
        Ok(())
    }
