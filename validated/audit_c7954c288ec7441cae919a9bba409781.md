### Title
SNS Treasury Manager `deposit` Lacks Slippage Protection, Enabling Sandwich Attacks Against DAO Treasury Deposits into DEX Liquidity Pools - (File: `rs/sns/governance/src/extensions.rs`)

---

### Summary

The SNS Treasury Manager extension framework allows SNS DAOs to deposit treasury assets (SNS tokens + ICP) into external DEX liquidity pools via governance proposals. The `execute_treasury_manager_deposit` function in `rs/sns/governance/src/extensions.rs` approves and forwards token allowances to the treasury manager canister with no price-boundary or slippage check. The `DepositRequest` type carries only token amounts — no minimum LP-token-out, no price ratio bounds. Because SNS governance proposals are public and have a mandatory multi-day voting period before execution, an unprivileged attacker can observe the pending deposit, front-run it by manipulating the DEX pool price, and back-run it to extract value from the SNS treasury. The codebase itself acknowledges this risk in two places but provides no on-chain mitigation.

---

### Finding Description

The `execute_treasury_manager_deposit` function is the on-chain execution path for `TreasuryManagerDeposit` governance proposals:

```
1. governance.approve_treasury_manager(extension_canister_id, sns_e8s, icp_e8s)
2. governance.env.call_canister(extension_canister_id, "deposit", arg_blob)
``` [1](#0-0) 

The `arg_blob` is a `DepositRequest` containing only `allowances` (token amounts and refund accounts): [2](#0-1) 

There are no fields for minimum LP tokens out, maximum acceptable price ratio, or any slippage bound. The validation step (`validate_deposit_operation_impl`) only checks that the requested amounts do not exceed 50% of the current treasury balance — it performs no price or ratio check: [3](#0-2) 

The codebase explicitly acknowledges this gap in two places. In the API contract: [4](#0-3) 

And in the proposal rendering warning shown to voters: [5](#0-4) 

Despite the acknowledgment, no technical mitigation is enforced at the protocol level. The `DepositRequest` type has no slippage fields, and `execute_treasury_manager_deposit` performs no pre- or post-deposit price check.

---

### Impact Explanation

An attacker can extract value directly from the SNS DAO treasury on every `TreasuryManagerDeposit` execution:

1. The SNS DAO submits a `TreasuryManagerDeposit` proposal (e.g., deposit X SNS + Y ICP into a KongSwap pool). The proposal is public on-chain.
2. The proposal passes after the mandatory voting period (minimum 4 days for normal proposals, 5 days for critical ones). [6](#0-5) 
3. The attacker observes the approved proposal and front-runs the execution: they swap a large amount of ICP into the SNS/ICP pool, inflating the SNS spot price.
4. The governance canister executes `execute_treasury_manager_deposit`, which approves and deposits the fixed SNS + ICP amounts at the now-manipulated price. The DAO receives far fewer LP tokens than it should.
5. The attacker back-runs by swapping SNS back to ICP, restoring the price and pocketing the profit extracted from the DAO's deposit.

The financial loss is borne entirely by the SNS treasury (i.e., all SNS token holders). The magnitude scales with the deposit size and the depth of the DEX pool.

---

### Likelihood Explanation

- **High observability**: All SNS governance proposals are public. The deposit amounts, target extension canister, and execution timing are fully visible on-chain before execution.
- **No technical barrier**: Any unprivileged principal can interact with the DEX canister to manipulate pool prices. No privileged role, key, or majority is required.
- **Predictable execution window**: After a proposal passes, execution is triggered by the next heartbeat/timer tick of the governance canister. The attacker can monitor proposal state and act within the same round.
- **Acknowledged but unmitigated**: The risk is documented in the production API contract (`treasury_manager.did`) and in the proposal rendering code (`proposal.rs`), confirming the developers are aware this is a real attack surface.

---

### Recommendation

1. **Add slippage parameters to `DepositRequest`**: Extend the `DepositRequest` type in `treasury_manager.did` with a `min_lp_tokens_out` or `max_price_ratio_deviation_bps` field. The treasury manager implementation must enforce this bound before executing the DEX deposit. [2](#0-1) 

2. **Enforce slippage at the governance layer**: In `execute_treasury_manager_deposit`, require that the `ValidatedDepositOperationArg` includes a price-ratio bound validated at proposal submission time. Reject execution if the current DEX price deviates beyond the bound. [7](#0-6) 

3. **Two-step deposit**: Disable DEX trading (if the DEX supports it), verify the spot price is within range, then execute the deposit, then re-enable trading — analogous to the mitigation described in the original AeraVaultV1 report.

4. **Post-deposit LP token check**: After calling `deposit`, verify the returned `Balances` reflect a minimum expected LP token allocation. Revert (or flag for manual review) if the received LP tokens fall below the threshold encoded in the proposal.

---

### Proof of Concept

**Entry path**: Any unprivileged principal with access to the DEX canister (the `external_custodian`).

**Step-by-step**:

1. Attacker monitors SNS governance for a `TreasuryManagerDeposit` proposal targeting a SNS/ICP DEX pool. The proposal specifies `treasury_allocation_sns_e8s = S` and `treasury_allocation_icp_e8s = I`. [8](#0-7) 

2. Proposal passes after the voting period. Attacker observes `ProposalDecisionStatus::Adopted`.

3. Attacker front-runs: calls the DEX canister to swap a large amount of ICP → SNS, inflating the SNS/ICP price in the pool.

4. Governance canister executes `execute_treasury_manager_deposit`:
   - `approve_treasury_manager` grants the treasury manager an ICRC-2 allowance of exactly `S` SNS and `I` ICP with no price condition. [9](#0-8) 
   - `call_canister(..., "deposit", arg_blob)` forwards a `DepositRequest` with only token amounts — no slippage bound. [10](#0-9) 
   - The treasury manager deposits at the manipulated price; the DAO receives fewer LP tokens than the fair-price equivalent.

5. Attacker back-runs: swaps SNS → ICP, restoring the price and realizing profit at the DAO's expense.

6. The governance canister logs success with no indication of value loss:
   ```
   TreasuryManager.deposit succeeded with response: ...
   ``` [11](#0-10)

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}
