Audit Report

## Title
SNS Governance Proposal Queue Saturation Enables Denial-of-Service Against Legitimate Proposals - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS governance canister enforces a single global ceiling of 700 unsettled proposals (`MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS`) with no per-neuron or per-proposer limit. An attacker holding sufficient governance tokens distributed across neurons can saturate this queue with spam proposals, blocking all non-emergency governance actions for the full voting period. Critically, `ManageNervousSystemParameters` — the proposal type that would allow the community to raise the rejection fee to deter the attack — is not exempt from the cap, creating a self-reinforcing lock.

## Finding Description
In `rs/sns/governance/src/governance.rs` at lines 3532–3547, `make_proposal` enforces only a global count check:

```rust
if self.proto.proposals.values()
    .filter(|data| !data.ballots.is_empty())
    .count()
    >= MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS
    && !proposal.allowed_when_resources_are_low()
{ ... ResourceExhausted ... }
```

The constant is 700, defined in `rs/sns/governance/src/proposal.rs` at line 79. The only exemptions are in `rs/sns/governance/src/types.rs` at lines 1751–1762: `UpgradeSnsControlledCanister`, `UpgradeSnsToNextVersion`, and `ExecuteGenericNervousSystemFunction`. All other action types — including `ManageNervousSystemParameters`, `Motion`, `TransferSnsTreasuryFunds`, `MintSnsTokens`, `RegisterDappCanisters`, `DeregisterDappCanisters`, `ManageSnsMetadata`, and `ManageLedgerParameters` — are blocked when the queue is full.

The stake check at lines 3521–3526 only requires `proposer.stake_e8s() >= reject_cost_e8s`. Since `stake_e8s() = cached_neuron_stake_e8s - neuron_fees_e8s`, a neuron with N tokens can submit N proposals before being blocked. The default `reject_cost_e8s` is 1 governance token (`E8S_PER_TOKEN`), set in `rs/sns/governance/src/types.rs` at line 471. There is no per-neuron open-proposal cap anywhere in the `make_proposal` flow. The comment at lines 3644–3648 claims the fee mechanism "prevents a neuron from having too many proposals outstanding," but this is only true per-neuron; it does not prevent a multi-neuron attacker from filling the global queue.

Ballots are only cleared after the voting period ends and rewards are settled, so the queue remains blocked for the full 4-day default voting period (extendable to 30 days via wait-for-quiet, per `rs/sns/governance/src/types.rs` line 475–476). The attacker can immediately refill cleared slots.

## Impact Explanation
This is a **High** severity finding matching: *"Significant SNS security impact with concrete user or protocol harm."* An attacker can permanently block all non-emergency SNS governance for any SNS instance for as long as they continue to refill the queue. The community cannot pass a `ManageNervousSystemParameters` proposal to raise `reject_cost_e8s` while the queue is full, because that action type is not in `allowed_when_resources_are_low`. This creates a governance deadlock for treasury management, parameter changes, dapp registration, and metadata updates.

## Likelihood Explanation
At default `reject_cost_e8s = 1 token`, the attacker needs 700 governance tokens with dissolve delay ≥ `neuron_minimum_dissolve_delay_to_vote_seconds` (default 6 months). For newly launched SNS instances with low token valuations, this cost can be trivial for a motivated competitor or adversary. The attacker burns tokens on rejection but can repeat the attack indefinitely if the token cost remains low relative to the damage. There is no rate limiting, no per-neuron proposal cap, and no community mechanism to remove spam proposals before their voting period expires.

## Recommendation
- **Short term**: Enforce a per-neuron limit on simultaneously open proposals (e.g., ≤ 5 per neuron) inside `make_proposal`, preventing any single actor from monopolizing the global queue regardless of stake.
- **Short term**: Add `ManageNervousSystemParameters` to `allowed_when_resources_are_low` in `rs/sns/governance/src/types.rs` so the community can always raise `reject_cost_e8s` even when the queue is saturated.
- **Long term**: Implement a supermajority-vote proposal removal mechanism to allow the community to clear spam without waiting for the full voting period.

## Proof of Concept
1. Attacker acquires 700 SNS governance tokens and creates neurons (one or more) each with dissolve delay ≥ `neuron_minimum_dissolve_delay_to_vote_seconds`.
2. Attacker calls `manage_neuron` with `Command::MakeProposal(Motion { motion_text: "spam" })` 700 times across their neurons.
3. Each call passes the stake check (`stake_e8s() >= reject_cost_e8s = 1 token`) and the global count check (count < 700).
4. After 700 submissions, `self.proto.proposals.values().filter(|data| !data.ballots.is_empty()).count() == 700 >= MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS`.
5. Any subsequent `ManageNervousSystemParameters` proposal from a legitimate community member returns `ResourceExhausted` because `ManageNervousSystemParameters` is not in `allowed_when_resources_are_low`.
6. The queue remains full for 4+ days. As proposals expire and are settled, the attacker immediately submits new ones.
7. The community cannot pass a `ManageNervousSystemParameters` proposal to raise `reject_cost_e8s` because that action is not exempt from the cap, completing the governance deadlock.
