Audit Report

## Title
Unbounded `inflight_requests` JoinSet + `read_to_end(128 MiB)` per stream enables per-connection heap exhaustion — (`rs/p2p/quic_transport/src/request_handler.rs`)

## Summary
The `start_stream_acceptor` loop in `request_handler.rs` spawns one unbounded tokio task per accepted QUIC bidirectional stream into an uncapped `JoinSet`. Each task calls `recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)`, accumulating up to 128 MiB into a heap-allocated `Vec<u8>`. A Byzantine subnet peer holding a valid TLS certificate can open all 1,000 permitted concurrent streams and stream 128 MiB on each, driving aggregate heap allocation to ~128 GiB and OOM-crashing the victim replica.

## Finding Description
All constants are confirmed in source:
- `MAX_MESSAGE_SIZE_BYTES = 128 * 1024 * 1024` (`lib.rs:74`)
- `MAX_CONCURRENT_BIDI_STREAMS = VarInt::from_u32(1_000)` (`connection_manager.rs:74`)
- `STREAM_RECEIVE_WINDOW = VarInt::from_u32(4_000_000)` (`connection_manager.rs:73`)
- `RECEIVE_WINDOW = VarInt::from_u32(200_000_000)` (`connection_manager.rs:71`)

The unbounded `JoinSet` is created at `request_handler.rs:50` with no capacity limit or semaphore. The developer comment at lines 52–56 explicitly acknowledges that the stream limit determines how many messages can be buffered, but does not account for the per-stream memory cost. For each accepted stream, a task is unconditionally spawned at lines 79–90. Each task calls `read_request`, which at lines 173–174 calls `recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)`. Quinn's `read_to_end` reads chunks from the QUIC receive buffer into a `Vec<u8>` and issues window updates as data is consumed, allowing the sender to push more data. The `Vec<u8>` accumulates all received bytes without bound up to the 128 MiB cap. QUIC flow control (`STREAM_RECEIVE_WINDOW = 4 MB`, `RECEIVE_WINDOW = 200 MB`) limits instantaneous in-flight bytes inside the QUIC layer but does not cap the total heap allocated across all concurrent `Vec` buffers — it only governs rate, not ceiling. With 1,000 streams each growing to 128 MiB, total heap reaches ~128 GiB.

## Impact Explanation
A single Byzantine subnet member can OOM-crash a victim replica, removing it from consensus participation. The attack is repeatable and can be directed at multiple replicas. This constitutes an application/platform-level crash and subnet availability impact not based on raw volumetric DDoS, matching the **High ($2,000–$10,000)** impact tier: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
The attacker must be a legitimate subnet member with a valid TLS certificate — a Byzantine node within the explicit `f`-fault threshold of the BFT model. No external network access, key compromise, governance majority, or victim mistake is required. Only a single established QUIC connection is needed. The attack is repeatable: after a replica restarts, the Byzantine peer can reconnect and repeat. A single Byzantine node can serially target multiple honest replicas.

## Recommendation
1. **Bound `inflight_requests`**: Replace the unbounded `JoinSet` with a semaphore-gated or bounded-channel design so at most `N` (e.g., 10–50) concurrent stream tasks are active per connection; apply back-pressure by not calling `accept_bi` when the limit is reached.
2. **Reduce `MAX_CONCURRENT_BIDI_STREAMS`**: Lower from 1,000 to a value consistent with actual protocol usage patterns.
3. **Per-connection memory budget**: Track total bytes in-flight across all concurrent `read_to_end` calls for a given peer and reset streams that would exceed a per-peer memory cap.
4. **Streaming deserialization**: Use chunked reads with an explicit size prefix so the allocation can be rejected before the full payload arrives, rather than accumulating into a full `Vec<u8>`.

## Proof of Concept
```
1. Attacker node (valid TLS cert, subnet member) connects to victim replica.
2. Open 1,000 bidi streams via QUIC (within MAX_CONCURRENT_BIDI_STREAMS=1000).
3. On each stream, send a valid protobuf-framed HttpRequest with a 128 MiB body.
4. STREAM_RECEIVE_WINDOW=4 MB throttles each stream, but read_to_end on the
   victim side accumulates received bytes into a Vec<u8> and issues window
   updates, allowing the sender to continue pushing data.
5. After all 1,000 Vecs grow toward 128 MiB each, victim heap usage → ~128 GiB → OOM.
6. Assert: replica process exits (OOM kill) before all streams complete.

Reproducible as a local integration test: spin up a test replica with a
PocketIC or local replica harness, register a test node as a subnet peer,
open 1,000 bidi QUIC streams from the test node, stream 128 MiB on each,
and observe the replica process exit with an OOM signal.
```
