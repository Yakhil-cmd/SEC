Audit Report

## Title
Single-Input Vsize Assumption in `fee_based_minimum_withdrawal_amount` Causes Unreimbursed ckBTC Burn on `AmountTooLow` — (`rs/bitcoin/ckbtc/minter/src/fees/mod.rs`)

## Summary
`fee_based_minimum_withdrawal_amount` computes the minimum withdrawal threshold using a hard-coded single-input vsize of 221 vbytes. When the minter's UTXO pool contains only UTXOs smaller than the withdrawal amount, `greedy` selects N > 1 inputs, and the actual fee computed against the real multi-input transaction vsize can exceed the withdrawal amount. This triggers `BuildTxError::AmountTooLow` in `build_unsigned_transaction_from_inputs`, which is handled in `submit_pending_requests` by finalizing the request as `FinalizedStatus::AmountTooLow` with no reimbursement — permanently burning the user's ckBTC without delivering BTC.

## Finding Description

**Root cause — fee formula uses 1-input vsize:**
`fee_based_minimum_withdrawal_amount` hard-codes `PER_REQUEST_VSIZE_BOUND = 221` vbytes (the vsize of a 1-input/2-output P2WPKH transaction) to compute the minimum withdrawal threshold. It never accounts for N > 1 inputs. [1](#0-0) 

**UTXO selection — `greedy` picks multiple small UTXOs:**
When all UTXOs in the pool are smaller than the withdrawal amount, `find_lower_bound(goal)` returns `None` on every iteration and `last()` (the largest available) is taken repeatedly, selecting `ceil(W/V)` UTXOs for a withdrawal of W sats with all UTXOs worth V sats. [2](#0-1) 

**Actual fee check uses real vsize:**
`evaluate_transaction_fee` calls `fake_sign(tx).vsize()` on the fully-constructed N-input transaction. For N = 20 inputs at 116 sat/vbyte, vsize ≈ 1,432 vbytes → fee ≈ 166,112 sats, which exceeds a 100,000-sat withdrawal amount. [3](#0-2) [4](#0-3) 

**`AmountTooLow` path burns funds without reimbursement:**
The `AmountTooLow` branch in `submit_pending_requests` calls `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow` and emits no `schedule_withdrawal_reimbursement` event. No `reimburse_canceled_requests` call is made. [5](#0-4) [6](#0-5) 

**ckBTC is burned before the request enters the pending queue:**
In `retrieve_btc`, `burn_ckbtcs` is called at line 210 before `accept_retrieve_btc_request` at line 232. The `reimbursement_account` is set on the request, so the infrastructure for reimbursement exists but is never invoked on the `AmountTooLow` path. [7](#0-6) 

**Contrast with `InvalidTransaction` path, which does reimburse:**
`BuildTxError::InvalidTransaction` calls `reimburse_canceled_requests`, correctly returning funds minus a processing fee. [8](#0-7) 

## Impact Explanation
A user who submits a withdrawal at exactly `fee_based_retrieve_btc_min_amount` when the minter's UTXO pool contains only UTXOs smaller than that amount will have their ckBTC permanently burned with no BTC delivery and no reimbursement. The `RetrieveBtcStatusV2::AmountTooLow` terminal state is returned with no recovery path. This constitutes a permanent, irreversible loss of ckBTC (a chain-key ledger asset) for the affected user. This matches the **High** impact class: "Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."

## Likelihood Explanation
The precondition — no single UTXO in the minter's pool ≥ the withdrawal amount — can occur naturally after a period of many small deposits and large withdrawals consuming all large UTXOs. It can also be engineered adversarially: an attacker deposits many small UTXOs (each just above `deposit_btc_min_amount`) to fill the pool, then any user who subsequently withdraws at the minimum amount is affected. The attacker spends real BTC but does not directly profit (griefing attack). The UTXO consolidation mechanism provides partial mitigation but requires `utxo_consolidation_threshold` UTXOs and has a `MIN_CONSOLIDATION_INTERVAL` of 24 hours, leaving a meaningful window of exposure. The attack is repeatable as long as the pool state persists.

## Recommendation
1. **Fix the vsize bound**: Replace `PER_REQUEST_VSIZE_BOUND = 221` with a bound that accounts for multi-input transactions, or compute `fee_based_minimum_withdrawal_amount` using the actual UTXO pool composition (as `estimate_withdrawal_fee` already does for the query endpoint).
2. **Add reimbursement for `AmountTooLow`**: When `BuildTxError::AmountTooLow` is triggered in `submit_pending_requests`, call `reimburse_canceled_requests` (minus a processing fee) rather than silently burning the user's ckBTC. The `InvalidTransaction` path at lines 400–410 of `lib.rs` is the correct model.
3. **Address the TODO**: The comment at line 141 of `fees/mod.rs` (`//TODO DEFI-2187`) acknowledges the formula is known to be imprecise and should be resolved.

## Proof of Concept
State-machine test (Regtest or PocketIC):

```
retrieve_btc_min_amount = 50_000 sats
fee_rate = 116_000 millisats/vbyte
→ fee_based_retrieve_btc_min_amount = 100_000 sats

UTXO pool: 20 UTXOs × 5_000 sats each
(total = 100_000 sats, no single UTXO ≥ 100_000)

1. User calls retrieve_btc(amount=100_000).
2. burn_ckbtcs burns 100_000 ckBTC. Request enters pending queue.
3. submit_pending_requests runs:
   - greedy(100_000) selects all 20 UTXOs (5_000 each).
   - build_unsigned_transaction_from_inputs constructs 20-input tx.
   - evaluate_transaction_fee: vsize ≈ 1,432 vbytes → fee = 166,112 sats.
   - 166,112 + minter_fee > 100,000 → BuildTxError::AmountTooLow.
4. remove_retrieve_btc_request called with FinalizedStatus::AmountTooLow.
5. No reimbursement emitted.

Expected: withdrawal succeeds or user is reimbursed.
Actual: request finalized as AmountTooLow, 100,000 ckBTC burned, no BTC
        delivered, no reimbursement.
```

pub fn remove_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    status: FinalizedStatus,
    runtime: &R,
) {
    record_event(
        EventType::RemovedRetrieveBtcRequest {
            block_index: request.block_index,
        },
        runtime,
    );

    state.push_finalized_request(FinalizedBtcRequest {
        request: request.into(),
        state: status,
    });
}
