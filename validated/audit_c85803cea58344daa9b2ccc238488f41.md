### Title
Forward/Proving Signature Validation Divergence Allows Unprovable Block Construction — (`basic_bootloader/src/bootloader/transaction_flow/`)

### Summary

The forward run uses `BasicBootloaderForwardSimulationConfig` (`VALIDATE_EOA_SIGNATURE = false`) and the proving run uses `BasicBootloaderProvingExecutionConfig` (`VALIDATE_EOA_SIGNATURE = true`). Both `validate_and_compute_fee_for_transaction` implementations gate the entire ecrecover + malleable-signature check behind this flag. An attacker can submit a transaction with a malleable signature (`s > SECP256K1N_HALF`) that the forward run accepts (nonce incremented, fees collected, state committed) but the proving run rejects with `MalleableSignature`, making any block containing that transaction permanently unprovable.

### Finding Description

**Config divergence:**

`BasicBootloaderForwardSimulationConfig::VALIDATE_EOA_SIGNATURE = false` [1](#0-0) 

`BasicBootloaderProvingExecutionConfig::VALIDATE_EOA_SIGNATURE = true` [2](#0-1) 

**Forward run entry point** calls `run_forward::<BasicBootloaderForwardSimulationConfig>`: [3](#0-2) 

**Proving run entry point** calls `run_prepared::<BasicBootloaderProvingExecutionConfig>`: [4](#0-3) 

**The gated check in the ZK flow** (`zk/validation_impl.rs`):

```
if !Config::VALIDATE_EOA_SIGNATURE | Config::SIMULATION {
    // only charge native cost, skip ecrecover entirely
} else {
    if U256::from_be_slice(s) > U256::from_be_bytes(SECP256K1N_HALF) {
        return Err(InvalidTransaction::MalleableSignature.into());
    }
    // ... ecrecover and from-address check
}
``` [5](#0-4) 

The identical gate exists in the Ethereum flow: [6](#0-5) 

The code itself acknowledges the risk with the comment: *"we provided a 'hint' for 'from', so it's sequencer's risks here"* [7](#0-6) 

However, the `run_block` public API — the sequencer's sole block-execution entry point — performs no external signature validation before or after calling the bootloader. The bootloader's forward run is the only validation layer, and it explicitly skips it.

### Impact Explanation

A block that has been forward-executed and committed to state cannot be proven. The proving system panics on `"Tried to prove a failing batch"`: [8](#0-7) 

This halts proof generation for the entire chain until the offending block is identified and discarded, requiring a reorg or manual intervention. The impact matches the stated scope: *"Circuit, node, or program mismatches that make valid ZKsync OS executions unprovable."*

### Likelihood Explanation

The attack requires only:
1. An account with sufficient ETH balance and a correct nonce.
2. A structurally valid transaction with a malleable signature (`s` flipped to `n - s`). For RLP transactions, this still recovers to the same `from` address, so all other checks (nonce, balance, EIP-3607) pass in both runs.
3. The sequencer including the transaction — which it will, because `run_block` accepts it.

No privileged access, no key leakage, no governance attack is required. The attack is trivially constructable by any EOA holder.

### Recommendation

Either:
- Enable signature validation in the forward run by switching `run_block` to use `BasicBootloaderForwardETHLikeConfig` (which already has `VALIDATE_EOA_SIGNATURE = true`), accepting the small performance cost; or
- Add an explicit malleable-signature pre-filter in the sequencer's transaction ingestion path before `run_block` is called, so that transactions with `s > SECP256K1N_HALF` are rejected at the RPC/mempool layer and never reach the bootloader.

The existing `BasicBootloaderForwardETHLikeConfig` already provides the correct config: [9](#0-8) 

### Proof of Concept

```
1. Create EOA with balance B and nonce N.
2. Construct a valid EIP-1559 RLP transaction (correct nonce, sufficient gas, valid fields).
3. Sign it normally → obtain (v, r, s).
4. Flip s to (SECP256K1N - s); adjust v accordingly. The signature still ecrecovers to the same address.
5. Submit the transaction to the sequencer.
6. run_block (BasicBootloaderForwardSimulationConfig):
   - validate_and_compute_fee_for_transaction skips the s > SECP256K1N_HALF check.
   - Nonce incremented, fees collected, state committed. Block finalised.
7. run_proving_inner (BasicBootloaderProvingExecutionConfig):
   - validate_and_compute_fee_for_transaction hits: s > SECP256K1N_HALF → MalleableSignature.
   - run_prepared returns Err → panic!("Tried to prove a failing batch").
8. Block is permanently unprovable.
```
