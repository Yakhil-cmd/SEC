Audit Report

## Title
Missing ingress window validation in ICP Rosetta `/construction/payloads` causes index-out-of-bounds panic in `/construction/parse` — (`rs/rosetta-api/icp/src/request_handler/construction_parse.rs`)

## Summary
The ICP Rosetta API does not validate that `ingress_start < ingress_end` in `/construction/payloads`. Supplying equal timestamps produces an `UnsignedTransaction` with an empty `ingress_expiries` vec but a non-empty `updates` vec. This propagates through `/construction/combine`, which emits a `SignedTransaction` with `requests: [(Send, [])]`. A subsequent `/construction/parse` call with `signed=true` unconditionally indexes `updates[0]` on the empty `Vec<EnvelopePair>`, causing an index-out-of-bounds panic that crashes the request handler.

## Finding Description
In `construction_payloads.rs`, the ingress expiry loop runs only while `now < ingress_end`. [1](#0-0) 
When `ingress_start == ingress_end`, the loop body never executes and `ingress_expiries` remains `vec![]`. No guard rejects this condition. The ICRC1 Rosetta counterpart has an explicit rejection: [2](#0-1) 

`add_payloads` iterates over the empty slice and adds nothing to `payloads`. However, `updates.push((RequestType::Send, update))` at line 377 is unconditional, so the returned `UnsignedTransaction` carries one update entry with zero expiries. [3](#0-2) 

In `construction_combine`, the inner loop over `unsigned_transaction.ingress_expiries` never fires, so `request_envelopes` stays `vec![]`. The outer loop still executes `requests.push((request_type, request_envelopes))` unconditionally at line 170, producing a structurally valid but semantically broken `SignedTransaction { requests: [(Send, [])] }`. [4](#0-3) [5](#0-4) 

In `construction_parse` with `signed=true`, line 42 unconditionally indexes `updates[0]` on the `Vec<EnvelopePair>` for each request entry. With an empty vec, this is an index-out-of-bounds panic. [6](#0-5) 

The `Request` type alias confirms `Vec<EnvelopePair>` can be empty with no type-level protection. [7](#0-6) 

## Impact Explanation
An unprivileged external client can crash the ICP Rosetta API server process with a deterministic, non-volumetric three-request sequence. The Rosetta API is an explicitly in-scope financial integration component. This matches the allowed High impact: **"Application/platform-level DoS, crash... or subnet availability impact not based on raw volumetric DDoS"** and **"Significant... Rosetta... security impact with concrete user or protocol harm"** ($2,000–$10,000).

## Likelihood Explanation
The attack requires only crafting a `ConstructionPayloadsRequest` with `ingress_start == ingress_end` in the metadata. No authentication, no privileged role, no prior state, and no special knowledge beyond the Rosetta API spec is required. The ICRC1 Rosetta already carries the fix, demonstrating the pattern is known. The exploit is deterministic and repeatable by any client.

## Recommendation
Add the same guard present in the ICRC1 handler at the top of the ingress window computation in `construction_payloads.rs`:

```rust
if ingress_start >= ingress_end {
    return Err(ApiError::invalid_request(format!(
        "ingress_start must be before ingress_end: start={}, end={}",
        ingress_start.as_nanos_since_unix_epoch(),
        ingress_end.as_nanos_since_unix_epoch()
    )));
}
```

Additionally, replace the bare `updates[0]` index in `construction_parse.rs` line 42 with a bounds-checked access (`.first().ok_or_else(...)`) as a defense-in-depth measure, and consider asserting non-empty `ingress_expiries` before serializing the `UnsignedTransaction`.

## Proof of Concept
```
POST /construction/payloads
{ "metadata": { "ingress_start": T, "ingress_end": T }, ... }
→ unsigned_transaction: { updates: [(Send, update)], ingress_expiries: [] }

POST /construction/combine
{ "unsigned_transaction": <above>, "signatures": [] }
→ signed_transaction: { requests: [(Send, [])] }

POST /construction/parse
{ "signed": true, "transaction": <above> }
→ PANIC: index out of bounds: the len is 0 but the index is 0
         at construction_parse.rs:42  updates[0]
```
