### Title
Integer Division-Before-Multiplication Precision Loss in `native_per_pubdata` Calculation Causes Systematic Undercharging for Pubdata - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` rate is computed by dividing `pubdata_price` by `native_price` using integer (floor) division before the result is multiplied by actual pubdata bytes consumed. This is the classic div-before-mul precision loss: the truncation error accumulates with every pubdata byte, causing every transaction that generates pubdata to be systematically undercharged for native (proving) resources.

---

### Finding Description

In `validation_impl.rs`, the per-pubdata-byte native cost is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

This truncated rate is then used in two places:

1. **Intrinsic pubdata overhead** in `gas_helpers.rs`:
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   ``` [2](#0-1) 

2. **Execution pubdata charge** in `get_resources_to_charge_for_pubdata`:
   ```rust
   let native = current_pubdata_spent
       .checked_mul(native_per_pubdata)
       .ok_or(out_of_native_resources!())?;
   ``` [3](#0-2) 

The actual charge is:
```
charged = floor(pubdata_price / native_price) * pubdata_bytes
```

The correct charge should be:
```
charged = (pubdata_price * pubdata_bytes) / native_price
```

The maximum undercharge per transaction is:
```
undercharge ≤ (native_price − 1) * pubdata_bytes / native_price  ≈  pubdata_bytes  native units
```

For large `native_price` values, the truncation error approaches 1 native unit per pubdata byte. A transaction generating the maximum allowed pubdata (e.g., thousands of bytes) can underpay by thousands of native units per execution.

A secondary instance of the same pattern exists in `refund_calculation.rs` where `native_used / native_per_gas` (line 72) converts native consumption back to gas units using a truncating division, potentially undercharging gas when native consumption exceeds the gas-equivalent:

```rust
(native_used / native_per_gas) as i64 - (gas_used as i64)
``` [4](#0-3) 

Note that `native_per_gas` itself is computed with `div_ceil` (rounding up), which compounds the asymmetry: the user's native budget is slightly inflated while the conversion back to gas rounds down, both favoring the user at the operator/prover's expense. [5](#0-4) 

---

### Impact Explanation

This is a **resource accounting bug**. The native resource represents the actual ZK proving cost. Systematic undercharging means:

- Every transaction generating pubdata pays less native resource than the true proving cost.
- The operator/prover absorbs the difference on every block.
- An attacker can craft transactions that maximize pubdata generation (e.g., writing to many new storage slots) to maximize the undercharge per transaction.
- The effect is cumulative across all transactions in a block and across all blocks.

The impact matches the Immunefi scope: incorrect resource accounting that causes a systematic loss to the operator/prover, reachable by any unprivileged transaction sender.

---

### Likelihood Explanation

**High.** The condition is triggered whenever `pubdata_price` is not an exact multiple of `native_price`, which is the common case for arbitrary operator-set prices. Any transaction that writes to storage (generating pubdata) triggers this path. No special attacker capability is required beyond submitting a normal transaction.

---

### Recommendation

Replace the div-before-mul pattern with mul-before-div:

```rust
// Instead of:
let native_per_pubdata = pubdata_price.wrapping_div(native_price);
// ...
let native = pubdata_bytes.checked_mul(native_per_pubdata)?;

// Use:
let native = pubdata_bytes_u256
    .checked_mul(pubdata_price)?
    .checked_div(native_price)?;
```

For the `delta_gas` adjustment, use consistent rounding direction (ceiling) when converting native back to gas to avoid systematic undercharging:

```rust
// Instead of:
(native_used / native_per_gas) as i64 - (gas_used as i64)

// Use ceiling division:
native_used.div_ceil(native_per_gas) as i64 - (gas_used as i64)
```

---

### Proof of Concept

**Setup**: Block with `pubdata_price = 10`, `native_price = 3`.

**Current behavior**:
- `native_per_pubdata = floor(10 / 3) = 3`
- Transaction generates 300 pubdata bytes
- Native charged: `3 * 300 = 900`

**Correct behavior**:
- Native charged: `(10 * 300) / 3 = 1000`

**Undercharge**: 100 native units per transaction. At scale (e.g., 1000 transactions/block, each generating 300 pubdata bytes), the operator absorbs 100,000 native units per block in unrecovered proving costs. An attacker maximizing pubdata per transaction (storage writes to new slots) amplifies this loss proportionally.
