Audit Report

## Title
Unprivileged Caller Can DoS Arbitrary Victim Accounts via Attacker-Controlled `owner` in `balance_update_guard` — (`rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary
`update_balance` constructs the guard key from the caller-supplied `args.owner` field rather than the authenticated `caller` principal. An unprivileged attacker can submit 100 concurrent calls each specifying a distinct victim principal as `owner`, filling `update_balance_accounts` to `MAX_CONCURRENT=100` with attacker-chosen accounts. Legitimate owners of those accounts are then denied ckBTC minting service for the duration of the attacker's in-flight calls, and the attacker can re-submit continuously to maintain the DoS.

## Finding Description
In `update_balance`, the account passed to `balance_update_guard` is constructed as:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),  // attacker-controlled
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(caller_account)?;
``` [1](#0-0) 

`args.owner` is an ingress-supplied `Option<Principal>` with no authorization check — any principal can set it to any value. The only pre-guard check is a whitelist mode check on `caller` (not `args.owner`), and a trap if the resolved owner equals the minter's own canister ID. [2](#0-1) 

The guard is then held across the `get_utxos` inter-canister `await` point: [3](#0-2) 

`Guard::new` enforces two conditions against the `update_balance_accounts` set:
- `AlreadyProcessing` — if the exact account is already in the set
- `TooManyConcurrentRequests` — if `accounts.len() >= MAX_CONCURRENT` (100) [4](#0-3) 

Because the IC execution model allows multiple update messages to be in-flight concurrently at `await` points, an attacker submitting 100 calls in rapid succession — each with a distinct `owner = Some(victim_N)` — can hold all 100 guard slots simultaneously while their `get_utxos` calls are pending. Third-party minting (caller ≠ owner) is explicitly an intended design feature, confirmed by integration tests: [5](#0-4) 

Once the set is full, any legitimate victim calling `update_balance` for their own account receives either `AlreadyProcessing` (their specific account is locked) or `TemporarilyUnavailable("too many concurrent requests")` (set is full). [6](#0-5) 

## Impact Explanation
This is a **High** severity application-level DoS. Any unprivileged attacker can block ckBTC minting for up to 100 targeted victim accounts simultaneously. The attacker can cycle through victim sets continuously to maintain persistent DoS, preventing legitimate users from minting ckBTC regardless of their own actions. This matches the allowed impact: *"Application/platform-level DoS … not based on raw volumetric DDoS"* — the attack exploits a specific logic flaw (guard keyed on attacker-controlled input), not raw traffic volume.

## Likelihood Explanation
No privileged access is required — any IC principal can call `update_balance` with arbitrary `owner`. Bitcoin canister `get_utxos` latency is routinely several seconds, easily long enough for 100 sequential ingress submissions to all be in-flight simultaneously. The attacker can re-submit continuously to maintain the DoS. Cost is bounded by cycles for 100 inter-canister calls per rotation, which is economically feasible.

## Recommendation
Key the guard on the authenticated `caller` rather than the attacker-supplied `args.owner`:

```rust
let guard_account = Account {
    owner: caller,  // always use authenticated caller for guard
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(guard_account)?;
// caller_account (with args.owner) is still used for address derivation and minting
```

This preserves third-party minting functionality while ensuring the rate-limiting and DoS-prevention property is tied to the authenticated caller. If per-account deduplication is also desired, a separate check can be added, but it must not be the sole guard against DoS.

## Proof of Concept
Using the existing state-machine test infrastructure in `rs/bitcoin/ckbtc/minter/tests/tests.rs`:

1. Initialize minter with a mock Bitcoin canister that delays `get_utxos` responses.
2. From a single attacker principal, submit 100 `update_balance` calls each with `owner = Some(victim_N)` for N in 0..100, before any complete.
3. Assert `update_balance_accounts.len() == 100` (all guards held).
4. From `victim_0`'s own principal, call `update_balance(owner=None)` (resolves to `victim_0`).
5. Assert the result is `Err(UpdateBalanceError::AlreadyProcessing)` or `Err(TemporarilyUnavailable("too many concurrent requests"))`.

This directly extends the existing test at `guard.rs` lines 141–162, which already demonstrates that filling the set to 100 blocks further entries. [7](#0-6)

    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }
