### Title
`EvmErc20.withdrawToNear` Constructs Malformed Precompile Input When `error_refund` Feature Is Enabled, Causing Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

### Summary

`EvmErc20.sol`'s `withdrawToNear` function burns user tokens and then calls the `ExitToNear` precompile with an input buffer that omits the required 20-byte `refund_address` field expected by the Rust implementation when the `error_refund` feature is compiled in. The precompile call fails or misparsed the amount/recipient, while the token burn is already irreversible, resulting in permanent loss of user funds.

### Finding Description

`EvmErc20.sol`'s `withdrawToNear` constructs the precompile input as:

```
\x01 | amount_b (32 bytes) | recipient (bytes)
``` [1](#0-0) 

However, when the `error_refund` feature is enabled in the Rust `ExitToNear` precompile, `parse_input` expects the layout:

```
flag (1 byte) | refund_address (20 bytes) | amount (32 bytes) | recipient (bytes)
``` [2](#0-1) 

With `error_refund` enabled, `MIN_INPUT_SIZE` is 21 (not 3), and `parse_input` unconditionally reads `input[1..21]` as the `refund_address` and returns `input[21..]` as the remaining payload: [3](#0-2) 

Because `EvmErc20.sol` does not insert a `refund_address` field, the precompile receives:

| Bytes | What `EvmErc20` put there | What Rust reads it as |
|---|---|---|
| `[1..21]` | First 20 bytes of `amount_b` | `refund_address` (wrong) |
| `[21..53]` | Last 12 bytes of `amount_b` + first 20 bytes of `recipient` | `amount` (wrong) |
| `[53..]` | Remainder of `recipient` | `receiver_account_id` (wrong) |

The ERC-20 branch then calls `parse_amount(&input[..32])`: [4](#0-3) 

For any typical NEAR account ID recipient (ASCII bytes), the 20 bytes of the recipient embedded in the "amount" field produce a U256 value that exceeds `u128::MAX`, causing `parse_amount` to return `ERR_INVALID_AMOUNT` and the precompile call to fail. The token burn in `withdrawToNear` has already executed before the precompile call: [1](#0-0) 

The `call` return value is never checked, so the failure is silently swallowed and the user's tokens are permanently destroyed with no withdrawal occurring.

### Impact Explanation

**Critical — Permanent freezing / theft of user funds.**

Any user calling `withdrawToNear` on an `EvmErc20` token contract when the Aurora Engine binary is compiled with `error_refund` will have their ERC-20 tokens burned with no corresponding NEP-141 tokens released on NEAR. The funds are unrecoverable.

### Likelihood Explanation

The `error_refund` feature is a compile-time flag in the Aurora Engine Rust codebase. If the production Aurora Engine deployment enables this feature (which exists specifically as a user-protection mechanism), every call to `EvmErc20.withdrawToNear` is broken. The `EvmErc20` contract is the standard bridge token contract deployed for all NEP-141 ↔ ERC-20 pairs, so the affected surface is the entire token bridge. Any unprivileged EVM user holding a bridged ERC-20 token can trigger this by calling `withdrawToNear`.

### Recommendation

`EvmErc20.sol`'s `withdrawToNear` must include the `refund_address` field in the precompile input when the `error_refund` feature is active. Since Solidity cannot query compile-time Rust features, the recommended fix is to always include the `refund_address` in the input layout (using `msg.sender` or a caller-supplied address as the refund target), and update the Rust `parse_input` to always expect it, removing the feature-gate divergence:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    bytes32 amount_b = bytes32(amount);
    bytes20 refund_addr = bytes20(_msgSender());
    bytes memory input = abi.encodePacked("\x01", refund_addr, amount_b, recipient);
    uint input_size = 1 + 20 + 32 + recipient.length;
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    }
}
```

Additionally, the `call` return value should be checked and the burn should be reverted on failure.

### Proof of Concept

1. Aurora Engine is compiled with the `error_refund` feature enabled.
2. User holds 100 units of a bridged ERC-20 token (e.g., USDC mirror).
3. User calls `withdrawToNear(b"alice.near", 100e6)`.
4. `_burn(msg.sender, 100e6)` executes — tokens destroyed.
5. Input constructed: `\x01 | 0x000000000000000000000000000000000000000000000000000000005F5E100 | alice.near` (no refund_address slot).
6. Rust `parse_input` reads bytes 1–20 of `amount_b` as `refund_address` = `0x0000000000000000000000000000000000000000`.
7. Remaining input = `\x64` (last 12 bytes of amount) + `alice.near` (10 bytes) = 22 bytes.
8. `parse_amount(&input[..32])` — only 22 bytes available, panics or reads garbage; or if padded, the ASCII bytes of `alice.near` in the high positions produce a value > u128::MAX → `ERR_INVALID_AMOUNT`.
9. Precompile call returns failure; `EvmErc20.sol` ignores the return value.
10. User's 100 tokens are permanently lost. [1](#0-0) [5](#0-4) [6](#0-5)

fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);

    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
    }

    Ok(amount)
}
