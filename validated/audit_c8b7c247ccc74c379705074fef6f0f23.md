Audit Report

## Title
Missing `created_at_time` in SNS Swap Ledger Transfers Disables ICRC-1 Deduplication, Enabling Double-Spend on Synthetic Reject - (`rs/nervous_system/clients/src/ledger_client.rs`)

## Summary

`LedgerCanister::transfer_funds` constructs every `TransferArg` with `created_at_time: None`, disabling the ICRC-1 ledger's deduplication window. The swap canister's `transfer_helper` resets `transfer_start_timestamp_seconds` to `0` on any error response — including a system-generated reject delivered after the ledger already executed the transfer. A subsequent `finalize` call then retries the identical transfer, which the ledger accepts as a new transaction, enabling double-spending of ICP or SNS tokens.

## Finding Description

**Root cause — `created_at_time: None`:**

`LedgerCanister::transfer_funds` in `rs/nervous_system/clients/src/ledger_client.rs` always constructs:

```rust
let args = TransferArg {
    from_subaccount,
    to,
    fee: Some(Nat::from(fee_e8s)),
    created_at_time: None,   // deduplication disabled
    amount: Nat::from(amount_e8s),
    memo: Some(Memo::from(memo)),
};
``` [1](#0-0) 

Per ICRC-1, when `created_at_time` is absent the ledger **must not** deduplicate. Two calls with identical `(from, to, amount, memo)` are treated as independent transactions.

**The guard that fails — error path resets the timestamp:**

`TransferableAmount::transfer_helper` in `rs/sns/swap/src/types.rs` sets `transfer_start_timestamp_seconds` before the await and checks it as a skip guard:

```rust
if self.transfer_start_timestamp_seconds > 0 {
    return TransferResult::AlreadyStarted;
}
self.transfer_start_timestamp_seconds = now_fn(false);
// ... await ledger call ...
Err(e) => {
    self.transfer_start_timestamp_seconds = 0;  // guard reset
    self.transfer_success_timestamp_seconds = 0;
``` [2](#0-1) 

If the ledger executes the transfer but the swap canister receives a system-generated reject (e.g., because the calling subnet recovered from a crash checkpoint taken before the response arrived, or because the XNet response was dropped during subnet recovery), the `Err` branch fires, resetting `transfer_start_timestamp_seconds` to `0`. The guard is now bypassed for the next call.

**Retry path — `finalize` is callable by any principal:**

`finalize` is an open update method. After the first `finalize` completes (lock released via `unlock_finalize_swap`), any caller can invoke it again. [3](#0-2) 

`sweep_icp` and `sweep_sns` iterate all buyers/recipes and call `transfer_helper` for each entry where `transfer_start_timestamp_seconds == 0`. [4](#0-3) 

With `created_at_time: None`, the ledger has no basis to reject the retry as a duplicate, so it executes a second transfer.

**`icrc2_approve` has the same issue:** [5](#0-4) 

## Impact Explanation

This matches the allowed impact: **High ($2,000–$10,000) — unauthorized transfer of canister-controlled funds with concrete user/protocol harm.**

| Scenario | Effect |
|---|---|
| `sweep_icp` ABORTED | Buyer receives a second ICP refund; swap canister drained of ICP it does not own |
| `sweep_icp` COMMITTED | SNS governance receives a second ICP deposit; swap canister over-drained |
| `sweep_sns` COMMITTED | Buyer neuron account receives a second SNS token allocation; token supply conservation violated |

The double-spend amount per affected buyer equals their full participation amount (up to `max_participant_icp_e8s`).

## Likelihood Explanation

The trigger requires a cross-subnet call (SNS swap on an application subnet, ICP ledger on the NNS subnet) where the calling subnet crashes and recovers from a checkpoint taken before the XNet response was ingested. This is a documented IC failure mode during subnet recovery. The attacker does not need to cause the crash — they only need to call `finalize` after recovery. Any unprivileged principal can call `finalize_swap` on a committed or aborted swap. The `finalize_swap_in_progress` lock only prevents concurrent calls within a single execution; it does not prevent sequential retries across separate calls. [6](#0-5) 

## Recommendation

Set `created_at_time` to the current IC time in `transfer_funds`:

```rust
created_at_time: Some(ic_cdk::api::time()),
```

Apply the same fix to `icrc2_approve`. With a timestamp set, a retry of the same transfer (same from, to, amount, memo, and timestamp within the 24-hour deduplication window) will be rejected by the ledger as `Duplicate { duplicate_of: N }`, and the swap canister should treat that response as a success and set `transfer_success_timestamp_seconds` accordingly.

## Proof of Concept

1. SNS swap reaches ABORTED state; buyer B contributed 100 ICP.
2. Caller invokes `finalize`; `sweep_icp` calls `transfer_helper` for buyer B.
3. `transfer_start_timestamp_seconds` is set to non-zero; `transfer_funds` is awaited.
4. The ICP ledger (NNS subnet) executes the refund transfer (block N) and sends the response.
5. The swap canister's application subnet crashes and recovers from a checkpoint taken before the XNet response arrived.
6. The IC delivers a synthetic reject to the swap canister's response callback.
7. The `Err` branch fires: `transfer_start_timestamp_seconds = 0`, `transfer_success_timestamp_seconds = 0`.
8. `finalize` completes (with failure), releases the lock.
9. Any caller invokes `finalize` again. `sweep_icp` sees `transfer_start_timestamp_seconds == 0` for buyer B and issues the identical `transfer_funds` call.
10. The ICP ledger, having no `created_at_time` to match against, executes the transfer a second time (block N+k).
11. Buyer B has received 200 ICP for a 100 ICP participation; swap canister conservation is violated.

A deterministic integration test can reproduce this by: (a) mocking the ledger to return `Ok(block_N)` on the first call, then (b) simulating a reject on the response delivery, then (c) calling `finalize` a second time and asserting the ledger receives a second `transfer_funds` call — which it will, because `created_at_time: None` prevents deduplication.     /// Acquires the lock on `finalize_swap`.
    pub fn lock_finalize_swap(&mut self) -> Result<(), String> {
        match self.is_finalize_swap_locked() {
            true => Err("The Swap canister has finalize_swap call already in progress".to_string()),
            false => {
                self.finalize_swap_in_progress = Some(true);
                Ok(())
            }
        }
    }

    /// Releases the lock on `finalize_swap`.
    fn unlock_finalize_swap(&mut self) {
        match self.is_finalize_swap_locked() {
            true => self.finalize_swap_in_progress = Some(false),
            false => {
                log!(
                    ERROR,
                    "Unexpected condition when unlocking finalize_swap_in_progress. \
                    The lock was not held: {:?}.",
                    self.finalize_swap_in_progress
                );
            }
        }
    }

    /// Checks the internal state of `finalize_swap_in_progress` lock.
    pub fn is_finalize_swap_locked(&self) -> bool {
        match self.finalize_swap_in_progress {
            Some(true) => true,
            None | Some(false) => false,
        }
    }

```rust
    pub async fn finalize(
        &mut self,
        now_fn: fn(bool) -> u64,
        environment: &mut impl CanisterEnvironment,
    ) -> FinalizeSwapResponse {
        // Acquire the lock or return a FinalizeSwapResponse with an error message.
        if let Err(error_message) = self.lock_finalize_swap() {
            return FinalizeSwapResponse::with_error(error_message);
        }

        // The lock is now acquired and asynchronous calls to finalize are blocked.
        // Perform all subactions.
        let finalize_swap_response = self.finalize_inner(now_fn, environment).await;

        if finalize_swap_response.has_error_message() {
            log!(
                ERROR,
                "The swap did not finalize successfully. \n\
                finalize_swap_response: {finalize_swap_response:?}"
            );
        } else {
            log!(
                INFO,
                "The swap finalized successfully. \n\
                finalize_swap_response: {finalize_swap_response:?}"
            );
        }

        // Release the lock. Note, if there is a panic, the lock will
        // not be released. In that case, the Swap canister will need
        // to be upgraded to release the lock.
        self.unlock_finalize_swap();

        finalize_swap_response
    }
