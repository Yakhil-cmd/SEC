### Title
Missing `ft_transfer_call` Unused-Token Refund in ERC-20 Omni Exit Path — (`engine-precompiles/src/native.rs`)

### Summary

When an ERC-20 holder exits via `ExitToNear` with flag `0x1` and an Omni message (`recipient:some_msg`), the precompile burns the ERC-20 and schedules an `ft_transfer_call` promise with **no callback** to handle unused tokens returned by the receiver's `ft_on_transfer`. Any tokens returned unused by the receiver accumulate in Aurora's NEP-141 balance with no corresponding ERC-20 re-mint, permanently destroying the user's principal.

---

### Finding Description

**Step 1 — Omni path sets `transfer_near_args = None`**

In `exit_erc20_token_to_near`, the `Some(Message::Omni(msg))` arm explicitly returns `None` for `transfer_near_args`: [1](#0-0) 

**Step 2 — `callback_args` collapses to `default()`**

Back in `ExitToNear::run`, `callback_args` is assembled: [2](#0-1) 

Without the `error_refund` feature (which is **not** in the default feature set): [3](#0-2) [4](#0-3) 

`refund` is hardcoded to `None` and `transfer_near` is `None`, so `callback_args == ExitToNearPrecompileCallbackArgs::default()`.

**Step 3 — No callback is scheduled**

The guard at line 470 fires the bare `PromiseArgs::Create` branch, scheduling only the `ft_transfer_call` with no `exit_to_near_precompile_callback` chained: [5](#0-4) 

**Step 4 — Even with `error_refund` enabled, the callback ignores the "success + unused tokens" case**

`exit_to_near_precompile_callback` handles only two branches: (a) success + `transfer_near` (wNEAR unwrap), and (b) failure + `refund`. There is no branch for "success + unused tokens returned by `ft_on_transfer`": [6](#0-5) 

**Step 5 — NEP-141 `ft_resolve_transfer` silently credits Aurora**

Per the NEP-141 standard, when a receiver returns a non-zero unused amount from `ft_on_transfer`, the NEP-141 contract's internal `ft_resolve_transfer` refunds those tokens to the original sender — Aurora's account. Aurora's NEP-141 balance increases by `unused_amount`, but the ERC-20 was already burned and no re-mint is triggered.

---

### Impact Explanation

The invariant `aurora_nep141_balance == total_erc20_supply` is broken: Aurora holds `unused_amount` more NEP-141 tokens than the ERC-20 supply accounts for. The user who burned `amount` ERC-20 tokens receives only `amount - unused_amount` NEP-141 tokens at the destination, and the remaining `unused_amount` is permanently locked in Aurora's NEP-141 balance with no recovery path. This constitutes **permanent freezing of funds** (matching the "High. Permanent freezing of funds" / "High. Theft of unclaimed yield" allowed impact scopes).

---

### Likelihood Explanation

The Omni exit path is the production path used by the Aurora Omni bridge. Any legitimate receiver contract that implements partial-fill semantics (returning unused tokens) — a standard and expected NEP-141 receiver behavior — will silently trigger this loss. No malicious actor is required; a user routing tokens through any receiver that does not consume the full amount is sufficient.

---

### Recommendation

For the Omni ERC-20 exit path, always schedule a callback regardless of `transfer_near` and `refund` state. The callback should inspect the `ft_transfer_call` promise result and, if unused tokens were returned (detectable via the promise's return value or by comparing pre/post NEP-141 balances), re-mint the corresponding ERC-20 amount to the original sender. Concretely:

- In `exit_erc20_token_to_near`, for `Some(Message::Omni(_))`, populate a `refund` field (analogous to the `error_refund` path) so `callback_args != default()`.
- Extend `exit_to_near_precompile_callback` with a third branch: success + Omni path → parse the `ft_transfer_call` return value (unused token count) and call `engine::refund_on_error` or a new `remint_on_partial_transfer` helper to re-mint the unused amount to the original ERC-20 holder.

---

### Proof of Concept

```
1. Deploy a NEP-141 token T; bridge it to Aurora as ERC-20 E.
2. User holds `amount` of E.
3. Deploy a NEAR contract R implementing ft_on_transfer that returns `amount/2`
   (i.e., accepts only half the tokens).
4. User calls ExitToNear precompile with:
     flag = 0x1
     amount = <amount>
     recipient = "<R_account_id>:<omni_msg>"
   This burns `amount` of E and schedules ft_transfer_call(receiver_id=R, amount=amount, msg=omni_msg)
   with NO callback (callback_args == default()).
5. R.ft_on_transfer returns amount/2.
6. T's ft_resolve_transfer credits amount/2 back to Aurora's NEP-141 balance.
7. Assert:
     - E.balanceOf(user) == 0  (burned, no re-mint)
     - T.ft_balance_of(aurora) increased by amount/2
     - T.ft_balance_of(R) == amount/2
     - User received 0 NEP-141 tokens at their intended destination
```

The `amount/2` tokens are permanently locked in Aurora's NEP-141 balance with no on-chain mechanism to recover them.
