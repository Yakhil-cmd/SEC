Audit Report

## Title
Unvalidated Sandbox-Supplied `PageDeltaSerialization` Fields Cause Replica Panic — (`rs/replicated_state/src/page_map/page_allocator/mmap.rs`)

## Summary
The replica unconditionally processes `PageDeltaSerialization` data received from a canister sandbox process over IPC without validating `file_offset` or `file_len` fields. A compromised sandbox can send a crafted `ExecutionFinishedRequest` that triggers either an `unreachable!()` panic (out-of-bounds `file_offset`) or an `mmap` failure panic (oversized `file_len`), crashing the replica process. The SELinux policy document additionally acknowledges a third crash path via `ftruncate` SIGBUS that is explicitly unmitigated.

## Finding Description

**Data flow:** After execution, the sandbox sends `ExecutionFinishedRequest` → `SandboxExecOutput` → `StateModifications` → `ExecutionStateModifications` → `MemoryModifications.page_delta: PageDeltaSerialization` to the replica over a Unix domain socket. [1](#0-0) [2](#0-1) 

In `update_execution_state`, the delta is applied unconditionally with no field-level validation: [3](#0-2) 

The `verify_size()` check that follows only logs an error metric and does not abort processing or reject the data: [4](#0-3) 

A TODO in the same function explicitly acknowledges that memory bounds from the sandbox are not enforced: [5](#0-4) 

**Crash Path 1 — `unreachable!()` via out-of-bounds `file_offset`:**

`deserialize_page_delta` calls `grow_for_deserialization(page_delta.file_len)` then iterates pages calling `deserialize_page` with the sandbox-supplied `file_offset`. The comment at line 301 states "File offsets of all pages are smaller than `file_len`" as a precondition, but this is never validated against sandbox-supplied data: [6](#0-5) 

If `file_offset >= file_len`, the chunk-search loop in `deserialize_page` exhausts and hits: [7](#0-6) 

**Crash Path 2 — `mmap` OOM panic via oversized `file_len`:**

`grow_for_deserialization` computes `mmap_size = (file_len - self.file_len) as usize` and calls `mmap` with no cap on the size. If `file_len` is near `i64::MAX`, `mmap` fails with `ENOMEM` and the `unwrap_or_else` panics: [8](#0-7) 

**Crash Path 3 — `ftruncate` SIGBUS (acknowledged, unmitigated):**

The SELinux policy document explicitly acknowledges that a sandbox can call `ftruncate` on the shared memory file to reduce its size, causing `SIGBUS` in the replica: [9](#0-8) 

## Impact Explanation

A compromised canister sandbox process can panic the replica process. The replica is the deterministic state machine; a panic causes the node to restart from the last checkpoint. If the malicious execution result is replayed from persisted canister state, the node enters a crash loop. This matches the **High** impact class: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS" ($2,000–$10,000). If only a single replica is affected, it falls to **Medium** ($200–$2,000): "One-time crash of a single replica on an application subnet, limited subnet availability impact."

## Likelihood Explanation

The prerequisite is code execution inside a canister sandbox process, requiring a Wasm sandbox escape (e.g., a Wasmtime memory-safety bug). This is non-trivial but not unrealistic — Wasmtime has had CVEs, and the IC's own threat model explicitly considers this scenario (as evidenced by the SELinux policy document and the TODO comment). Once the sandbox boundary is crossed, the replica has no defense against malformed IPC data. Likelihood is **low-to-medium** given the prerequisite, but the consequence is a deterministic, repeatable replica crash.

## Recommendation

1. **Validate `file_offset` against `file_len`** in `deserialize_page_delta` before calling `deserialize_page`. Return an error or terminate the sandbox if any `file_offset >= file_len` or `file_offset + PAGE_SIZE > file_len`.
2. **Cap `file_len`** in `grow_for_deserialization` to the maximum allowed canister memory size before calling `mmap`, returning an error rather than panicking on oversized values.
3. **Replace `unreachable!()` in `deserialize_page`** with a recoverable `Result`-based error path so a malformed offset kills the sandbox rather than panicking the replica.
4. **Address the `ftruncate` SIGBUS issue** by sealing the memfd against truncation (e.g., `F_SEAL_SHRINK`) before passing the file descriptor to the sandbox.
5. **Resolve the TODO** in `update_execution_state` by enforcing memory size bounds from the sandbox against the canister's allowed memory limit before committing state changes.

## Proof of Concept

A compromised sandbox (post-Wasm-escape) modifies its `execution_finished` IPC call:

```rust
// Crash Path 1: file_offset == file_len → unreachable!() in replica
controller.execution_finished(ExecutionFinishedRequest {
    exec_id: self.exec_id,
    exec_output: SandboxExecOutput {
        state: StateModifications {
            execution_state_modifications: Some(ExecutionStateModifications {
                globals: vec![],
                wasm_memory: MemoryModifications {
                    page_delta: PageDeltaSerialization {
                        file_len: 4096,
                        pages: vec![MmapPageSerialization {
                            page_index: PageIndex::new(0),
                            file_offset: 4096, // >= file_len → unreachable!()
                            validation: PageValidation::default(),
                        }],
                    },
                    size: NumWasmPages::new(1),
                },
                // ...
            }),
            // ...
        },
        // ...
    },
});
```

Expected replica output:
```
thread 'CanisterSandboxIPC' panicked at 'internal error: entered unreachable code:
Couldn't deserialize a page at offset 4096. Current file length 4096.',
rs/replicated_state/src/page_map/page_allocator/mmap.rs:757
```

A unit test can reproduce Crash Path 1 by constructing a `PageDeltaSerialization` with `file_offset == file_len` and calling `deserialize_page_delta` directly. Crash Path 2 can be reproduced by setting `file_len` to a value near `i64::MAX` and observing the `mmap` panic.

```rust
    pub fn deserialize_page_delta(
        page_allocator: &Arc<PageAllocatorInner>,
        page_delta: PageDeltaSerialization,
    ) -> Vec<(PageIndex, Page)> {
        let mut guard = page_allocator.core_allocator.lock().unwrap();
        let core = guard.as_mut().unwrap();
        core.grow_for_deserialization(page_delta.file_len);
        core.deserialized_pages += page_delta.pages.len();
        // Deserialized pages are considered as allocated for the purposes of the metric.
        ALLOCATED_PAGES.inc_by(page_delta.pages.len());
        // File offsets of all pages are smaller than `file_len`, which means
        // that the precondition of `deserialize_page()` is fulfilled after
        // the call to `grow_for_deserialization(file_len)`.
        page_delta
            .pages
            .into_iter()
            .map(|ser| {
                let page = core.deserialize_page(&ser, page_allocator);
                (ser.page_index, Page(Arc::new(page)))
            })
            .collect()
    }
```
