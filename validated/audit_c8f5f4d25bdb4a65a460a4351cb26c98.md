### Title
`pause_precompiles` can be frontrun to exit funds before the freeze takes effect — (File: `engine/src/contract_methods/admin.rs`)

---

### Summary

The `pause_precompiles` function is the Aurora Engine's mechanism for freezing the `EXIT_TO_NEAR` and `EXIT_TO_ETHEREUM` withdrawal precompiles during a security incident. However, because the pause is applied atomically only when its transaction is included in a NEAR block, a user who observes the pending pause transaction can race to submit their own EVM exit transaction first, escaping the intended freeze entirely.

---

### Finding Description

`pause_precompiles` writes a bitmask to storage that marks one or both exit precompiles as paused: [1](#0-0) 

The pause only takes effect once this transaction is finalized in a NEAR block. Until that moment, the `EXIT_TO_NEAR` and `EXIT_TO_ETHEREUM` precompiles remain fully operational. [2](#0-1) 

A user monitoring the NEAR RPC can observe the `pause_precompiles` call in the pending transaction pool. Because NEAR does not use a gas-price auction for intra-block ordering, the user can submit an EVM `submit` transaction that calls the exit precompile and have it land in the same block or an earlier block before the pause is committed. The test suite confirms that once the pause flag is set, the precompile returns `ERR_PAUSED`: [3](#0-2) 

The same race condition applies to `pause_contract`, which sets `state.is_paused = true`: [4](#0-3) 

And to the ETH-connector-level `pa_pause_feature("engine_withdraw")`, which gates the `withdraw` entrypoint: [5](#0-4) 

In all three cases the privileged freeze and the user's exit are independent NEAR transactions with no atomicity guarantee between them.

---

### Impact Explanation

The pause mechanism is the primary emergency brake for the Aurora Engine. It is used to halt withdrawals during bridge exploits or other security incidents. A user who successfully races the pause exits funds that the protocol intended to freeze. This constitutes a bypass of the temporary fund-freeze control, classified as **High — Temporary freezing of funds** (the freeze that should have applied does not).

---

### Likelihood Explanation

- NEAR RPC nodes expose pending transactions before block finalization, making the pending `pause_precompiles` call observable.
- NEAR block time is ~1 second; a user running automated monitoring can react within one block.
- Governance or social-media announcements of an impending pause (common in DeFi incident response) give even more lead time.
- The attacker needs only a standard EVM transaction calling `EXIT_TO_NEAR` or `EXIT_TO_ETHEREUM`; no special privilege is required.
- Sophisticated actors (the most likely targets of an emergency pause) are precisely the ones most likely to run such monitoring.

---

### Recommendation

1. **Introduce a pending-pause delay**: mirror the `upgrade_delay_blocks` pattern — record a "pause-at-block" height and reject exit transactions once that height is staged, before the pause is finalized. [6](#0-5) 

2. **Atomic pause-and-drain**: use a NEAR promise batch to atomically pause the precompile and sweep or lock balances in the same transaction, leaving no window.

3. **Pause-pending flag**: add a `pause_pending` boolean to `EngineState` that is checked by the exit precompiles, set it before the full pause transaction is submitted, and clear it on resume. [7](#0-6) 

---

### Proof of Concept

1. Admin submits `pause_precompiles { paused_mask: EXIT_TO_ETHEREUM_FLAG }` to the Aurora Engine account.
2. Attacker's monitoring bot detects the pending NEAR transaction via `EXPERIMENTAL_tx_status` RPC before it is included in a block.
3. Attacker immediately submits an EVM transaction (via `submit`) that calls the `EXIT_TO_ETHEREUM` precompile to bridge their full balance to an Ethereum address.
4. NEAR includes the attacker's `submit` transaction in the same block or one block earlier than the `pause_precompiles` transaction (no ordering guarantee favors the admin).
5. The attacker's funds are bridged out; the pause flag is then set, but the attacker's balance is already zero on Aurora.
6. The emergency freeze has been fully bypassed. [8](#0-7) [9](#0-8)

#[named]
pub fn pause_precompiles<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        require_running(&state::get_state(&io)?)?;
        let authorizer: EngineAuthorizer = engine::get_authorizer(&io);

        if !authorizer.is_authorized(&env.predecessor_account_id()) {
            return Err(b"ERR_UNAUTHORIZED".into());
        }

        let args: PausePrecompilesCallArgs = io.read_input_borsh()?;
        let flags = PrecompileFlags::from_bits_truncate(args.paused_mask);
        let mut pauser = EnginePrecompilesPauser::from_io(io);
        pauser.pause_precompiles(flags);
        Ok(())
    })
}
