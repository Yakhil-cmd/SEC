### Title
Unbounded `while` Loop in `move_to_block` Can Permanently Freeze All Hashchain-Wrapped Engine Functions - (`engine-hashchain/src/hashchain.rs`)

### Summary

The `Hashchain::move_to_block` function contains an unbounded `while` loop that iterates once per NEAR block between the stored `current_block_height` and the current NEAR block height. This loop is invoked on every call to any hashchain-wrapped engine function (including `submit`, `call`, `deploy_code`, connector deposits/withdrawals, and XCC). If the gap between the stored hashchain height and the live NEAR block height grows large enough, the loop exhausts NEAR's 300 TGas per-transaction gas limit, causing every hashchain-wrapped call to revert. Because the state is rolled back on revert, the hashchain height is never updated, making the freeze self-reinforcing until admin intervention.

Each iteration of the loop performs a `keccak` hash over a ~370-byte concatenation (chain\_id 32 B + account\_id ~15 B + block\_height 8 B + previous hashchain 32 B + merkle root 32 B + bloom 256 B): [7](#0-6) 

The number of iterations equals `env.block_height() − hashchain.current_block_height`. NEAR produces roughly one block per second. If the engine is idle for an extended period (days to weeks on a low-activity deployment), the gap grows into the millions. Once the loop cost exceeds NEAR's 300 TGas transaction gas cap, every call reverts, the hashchain state is never written back, and the gap only widens on subsequent blocks — a self-reinforcing freeze.

### Impact Explanation

All hashchain-wrapped functions — EVM `submit`/`call`/`deploy_code`, ETH connector deposits and withdrawals, and XCC calls — become permanently unreachable until an admin resets the hashchain. User funds locked in the bridge or awaiting withdrawal cannot be moved. This constitutes a **temporary (but potentially prolonged) freezing of funds** requiring privileged admin intervention to resolve.

### Likelihood Explanation

NEAR mainnet produces ~1 block/second. A deployment that is idle for a sustained period (e.g., a silo, testnet, or newly launched instance with low traffic) accumulates a proportionally large gap. No attacker action is required to grow the counter — the NEAR block height is monotonically increasing by protocol design. Any user who submits the first transaction after a sufficiently long idle period will trigger the gas-exhausting loop, after which no further transactions can succeed without admin action. The threshold gap (in blocks) depends on NEAR's exact gas pricing for keccak over ~370 bytes, but the structural unboundedness is unconditional.

### Recommendation

Replace the unbounded `while` loop in `move_to_block` with a bounded catch-up mechanism. Specifically:

1. Add a `max_steps` parameter (or a hard-coded constant) to `move_to_block` so a single call advances the hashchain by at most N blocks.
2. Expose a dedicated `advance_hashchain(steps: u64)` admin method that can be called repeatedly to catch up the hashchain incrementally without touching any user-facing logic.
3. Alternatively, store only the last-seen block height and compute the hashchain lazily (skipping empty blocks in O(1)) rather than iterating through every skipped block.

### Proof of Concept

1. Deploy Aurora Engine with hashchain enabled (`start_hashchain` called at block H).
2. Submit no transactions for a period long enough that `env.block_height()` advances to H + G, where G is large enough that iterating G keccak hashes exceeds 300 TGas.
3. Submit any EVM transaction (e.g., a simple ETH transfer). The NEAR runtime executes `with_logs_hashchain` → `load_hashchain` → `move_to_block(H + G)`, which loops G times. The transaction runs out of gas and reverts.
4. The stored `current_block_height` remains H. The next block is H + G + 1, so the gap is now G + 1. Every subsequent transaction also reverts.
5. All user funds that require `submit`, `call`, or connector withdrawal to move are frozen until the admin pauses the contract and calls `start_hashchain` with `args.block_height = current_block_height − 1` to reset the hashchain to the live block height in a single step.
