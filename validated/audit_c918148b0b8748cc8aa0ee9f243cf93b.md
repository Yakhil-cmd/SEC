Audit Report

## Title
Unprivileged Caller Can Force-Register Any Principal as SNS Swap Buyer via `refresh_buyer_tokens` — (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` canister endpoint accepts an arbitrary `buyer` principal in its request without verifying it matches the caller. Any unprivileged principal can call this endpoint with a victim's principal ID, causing the victim to be registered as a swap buyer. If the victim had ICP sitting in their swap subaccount (a realistic state in the subaccount-based participation model), the attacker's call locks that ICP in escrow and blocks the victim's only self-rescue path (`error_refund_icp`) until `sweep_icp` completes — permanently converting the victim's ICP into SNS neurons if the swap commits.

## Finding Description
**Root cause — missing caller == buyer check:**

`rs/sns/swap/canister/canister.rs` lines 130–133:
```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no caller == buyer assertion
};
```
When `arg.buyer` is non-empty, the parsed principal is used directly with no check that it equals `caller_principal_id()`.

**Downstream effect — BuyerState insertion:**

`refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs` lines 1153–1163) queries the ICP ledger balance of `principal_to_subaccount(&buyer)` on the swap canister. If the balance meets `min_participant_icp_e8s`, it inserts a `BuyerState` entry (lines 1285–1288):
```rust
self.buyers
    .entry(buyer.to_string())
    .or_insert_with(|| BuyerState::new(0))
    .set_amount_icp_e8s(new_balance_e8s);
```
`BuyerState::new(0)` initializes `transfer_success_timestamp_seconds: 0` (`rs/sns/swap/src/types.rs` lines 540–551).

**`error_refund_icp` is blocked:**

`rs/sns/swap/src/swap.rs` lines 1950–1960 show that if a `BuyerState` exists for the victim with `transfer_success_timestamp_seconds == 0`, `error_refund_icp` returns a precondition error:
```rust
if let Some(transfer) = &buyer_state.icp
    && transfer.transfer_success_timestamp_seconds == 0
{
    return ErrorRefundIcpResponse::new_precondition_error(...)
}
```
This blocks the victim's only self-rescue path until `sweep_icp` completes during finalization.

**`confirmation_text` provides no real barrier:**

`validate_confirmation_text` (`rs/sns/swap/src/swap.rs` lines 363–384) compares the caller-supplied text against `self.init_or_panic().confirmation_text`. The `get_init` endpoint is a public query that returns the full init data including this field, so an attacker trivially reads and supplies it.

**Ticket system does not prevent the attack:**

Lines 1249–1272 show the ticket check is skipped entirely when no ticket exists for the buyer: "If there exists no ticket for the buyer, the payment flow will simply ignore the ticket." The attacker does not need a ticket.

## Impact Explanation
**If the swap COMMITS:** After `sweep_icp` runs, the victim's ICP is transferred to SNS governance and the victim receives SNS neurons they never consented to acquire. Their ICP is permanently converted into illiquid governance tokens without authorization.

**If the swap ABORTS:** The victim's ICP is returned after `sweep_icp`, but they are denied the ability to use `error_refund_icp` to self-rescue during the window between swap close and `sweep_icp` completion.

This matches the allowed impact: **High ($2,000–$10,000) — Unauthorized access to ledger/canister-controlled funds with concrete user harm**, specifically forced, unconsented conversion of a victim's ICP into SNS neurons. The existence of `error_refund_icp` as a designed recovery mechanism confirms DFINITY anticipated this exact scenario (ICP in subaccount without registration), making the precondition realistic rather than hypothetical.

## Likelihood Explanation
- The attack requires no special privileges — any principal can send an ingress message to the swap canister.
- The precondition (victim has ICP in their swap subaccount without having called `refresh_buyer_tokens`) is realistic: the subaccount-based participation model requires users to send ICP first, and they may change their mind, experience delays, or make errors before completing registration.
- The `confirmation_text` (if set) is publicly readable via `get_init`, providing no real barrier.
- The attack window is the entire OPEN lifecycle of the swap.
- The attack is a single canister call with no cost beyond transaction fees.

## Recommendation
In `rs/sns/swap/canister/canister.rs`, enforce that the `buyer` field, if provided, must equal the caller:
```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let specified = PrincipalId::from_str(&arg.buyer).unwrap();
    assert_eq!(specified, caller_principal_id(), "buyer field must match the caller");
    specified
};
```
Alternatively, remove the `buyer` field from `RefreshBuyerTokensRequest` entirely and always use `caller_principal_id()`.

## Proof of Concept
1. Victim accidentally sends `>= min_participant_icp_e8s` ICP to `swap_subaccount(victim_principal)` on the ICP ledger, but does not call `refresh_buyer_tokens`.
2. Attacker (any principal) reads `confirmation_text` via `get_init` (if set), then calls:
   ```
   refresh_buyer_tokens(RefreshBuyerTokensRequest {
       buyer: victim_principal.to_string(),
       confirmation_text: <read from get_init>,
   })
   ```
3. The swap canister queries the victim's subaccount balance, finds sufficient ICP, and inserts `BuyerState { icp: TransferableAmount { amount_e8s: X, transfer_success_timestamp_seconds: 0 } }` for the victim.
4. Swap closes (COMMITTED). Victim calls `error_refund_icp` — blocked by the `transfer_success_timestamp_seconds == 0` guard at `rs/sns/swap/src/swap.rs` lines 1951–1959.
5. `sweep_icp` runs, transferring victim's ICP to SNS governance. Victim receives unwanted SNS neurons; ICP is permanently converted.

A deterministic integration test using PocketIC can reproduce this by: (a) opening a swap, (b) funding a victim subaccount, (c) calling `refresh_buyer_tokens` from a different principal with the victim's ID, (d) committing the swap, (e) asserting `error_refund_icp` returns a precondition error, and (f) asserting `sweep_icp` transfers the victim's ICP.
