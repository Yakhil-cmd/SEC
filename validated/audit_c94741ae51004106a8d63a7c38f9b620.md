### Title
Incomplete `check_integrity()` for `CatchUpPackageShare` Allows Crafted Shares to Bypass Cheap Validation and Trigger Expensive Content Checks — (`rs/types/types/src/consensus.rs`)

### Summary

The `check_integrity()` implementation for `CatchUpPackageShare` only verifies the `random_beacon` hash, leaving `block_hash`, `state_hash`, and `oldest_registry_version` unchecked. A Byzantine protocol peer below the fault threshold can craft many shares with a valid `random_beacon`/hash pair but arbitrary values for all other fields. Each such share passes `check_integrity()` and signature verification (the attacker signs with their own legitimate key), then forces the validator to invoke `validate_catch_up_share_content`, which performs expensive state manager and pool lookups before ultimately rejecting the share.

### Finding Description

`CatchUpPackageShare::check_integrity()` at consensus.rs:1737–1741:

```rust
fn check_integrity(&self) -> bool {
    let content = &self.content;
    let random_beacon_hash = content.random_beacon.get_hash();
    &crypto_hash(content.random_beacon.as_ref()) == random_beacon_hash
}
``` [1](#0-0) 

This only recomputes and compares the `random_beacon` hash. The `CatchUpShareContent` type carries `block` (a `CryptoHashOf<Block>`), `state_hash` (`CryptoHashOfState`), and `oldest_registry_version_in_use_by_replicated_state` — all opaque hash/value fields with no embedded counterpart to verify against in the share itself. [2](#0-1) 

By contrast, `CatchUpContent::check_integrity()` (for the full CUP, not the share) verifies the block hash, random beacon hash, and payload hash:

```rust
pub fn check_integrity(&self) -> bool {
    ...
    &crypto_hash(random_beacon) == random_beacon_hash
    && &crypto_hash(block) == block_hash
    && &crypto_hash(block_payload) == payload_hash
}
``` [3](#0-2) 

The share's `check_integrity()` is structurally weaker because the share carries only a `CryptoHashOf<Block>` (no block body to rehash), so the block hash cannot be locally verified. However, `state_hash` and `oldest_registry_version` are also not checked, and there is no guard preventing an attacker from supplying arbitrary values for these fields.

### Impact Explanation

A single Byzantine node (below the fault threshold) can generate an unbounded stream of `CatchUpPackageShare` messages, each with a valid `random_beacon`/hash but random `block_hash`/`state_hash`. Each share:
1. Passes `check_integrity()` (only random_beacon is checked).
2. Passes signature verification (the attacker signs the crafted content with their own legitimate threshold key share).
3. Triggers `validate_catch_up_share_content` in the validator, which performs state manager and pool lookups.
4. Is ultimately rejected as `HandleInvalid`.

This causes CPU exhaustion on honest validators during the period when CUP shares are being processed (near DKG summary heights), potentially delaying or stalling CUP creation and causing constrained subnet availability.

No rate limiting was found in `validator.rs` for `CatchUpPackageShare` processing. [1](#0-0) 

### Likelihood Explanation

- Attacker model is within scope: a single protocol peer below the consensus fault threshold.
- The attack requires only a valid `RandomBeacon` (publicly available from the pool) and the ability to sign with the attacker's own threshold key share.
- No privileged access, key leakage, or majority corruption is required.
- The window of opportunity is near DKG summary heights, which occur on a predictable schedule.

**Mitigating factors** (reducing severity but not eliminating the finding):
- `HandleInvalid` results may trigger peer disconnection, limiting sustained flooding from a single peer.
- P2P-layer per-peer artifact quotas (not visible in the searched files) may bound the rate of ingestion.
- The attacker is limited to one node's throughput.

### Recommendation

Extend `check_integrity()` for `CatchUpPackageShare` to perform any additional cheap, local checks possible. Since `block_hash` cannot be verified locally (no block body in the share), consider:
1. Checking that `state_hash` and `oldest_registry_version` are structurally valid (non-zero, within plausible ranges).
2. Adding a per-sender rate limit or per-height cap on unvalidated `CatchUpPackageShare` entries in the pool before invoking `validate_catch_up_share_content`.
3. Moving the cheapest rejection criteria (e.g., height checks, version checks) to occur before the expensive state manager lookups in `validate_catch_up_share_content`.

### Proof of Concept

1. Obtain a valid `RandomBeacon` and its hash from the public consensus pool at a DKG summary height.
2. Construct N `CatchUpPackageShare` messages, each with the valid `random_beacon`/hash but random `block_hash` and `state_hash` byte arrays.
3. Sign each with the attacker node's threshold signature share key.
4. Submit all N shares to the subnet's P2P layer.
5. Observe that each passes `check_integrity()`, enters the unvalidated pool, and triggers `validate_catch_up_share_content` before being rejected.
6. Benchmark: measure validator CPU time for N crafted shares vs. N shares that fail `check_integrity()` immediately — the former should be significantly higher due to state manager and pool lookups. [1](#0-0) [2](#0-1)
