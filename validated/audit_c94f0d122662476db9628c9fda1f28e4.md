### Title
Unverified `args.amount` in `ft_on_transfer` Allows Malicious NEP-141 Contract to Mint Unbacked ERC-20 Tokens - (`engine/src/contract_methods/connector.rs`)

### Summary

Aurora's `ft_on_transfer` entrypoint unconditionally trusts the `amount` field supplied by the calling NEP-141 contract and mints an equal quantity of ERC-20 mirror tokens inside the EVM, without ever verifying that Aurora's actual NEP-141 balance increased by that amount. Any unprivileged actor can deploy a malicious NEP-141 contract, register it on Aurora via the permissionless `deploy_erc20_token` entrypoint, and then cause Aurora to mint an arbitrary number of unbacked ERC-20 tokens, making the mirrored token insolvent.

**Root cause — `ft_on_transfer` blindly trusts `args.amount`.**

When any registered NEP-141 contract calls `ft_on_transfer` on Aurora, the handler dispatches to `receive_erc20_tokens` with the caller-supplied `args.amount`:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,          // ← amount comes entirely from the NEP-141 contract
        &current_account_id,
        handler,
    )
};
``` [3](#0-2) 

`receive_erc20_tokens` extracts `amount` directly from `args` and calls the ERC-20 `mint` selector with that value, with no balance-before/after check:

```rust
let amount = args.amount.as_u128();
...
let result = self
    .call(
        &erc20_admin_address,
        &erc20_token,
        Wei::zero(),
        setup_receive_erc20_tokens_input(&recipient, amount),
        u64::MAX,
        Vec::new(),
        Vec::new(),
        handler,
    )
    .and_then(submit_result_or_err)?;
``` [4](#0-3) 

`setup_receive_erc20_tokens_input` encodes the `mint(recipient, amount)` call directly from the untrusted value:

```rust

There is no point in the call chain where Aurora queries its own NEP-141 balance to confirm the deposit actually occurred.

### Impact Explanation

A malicious NEP-141 contract can call `ft_on_transfer` on Aurora reporting an `amount` of, say, 10,000,000 tokens while having transferred zero (or far fewer) tokens to Aurora. Aurora will mint 10,000,000 ERC-20 mirror tokens for the attacker. The ERC-20 token's total supply then exceeds the NEP-141 tokens actually held by Aurora, making the token insolvent. Legitimate holders who later try to exit (burn ERC-20 → receive NEP-141) will find Aurora holds insufficient NEP-141 reserves. Additionally, if the attacker swaps the unbacked ERC-20 tokens for real assets (ETH, other ERC-20s) via any DEX deployed on Aurora, real user funds are directly stolen.

This maps to **Critical — Insolvency** and **Critical — Direct theft of user funds** in the allowed impact scope.

### Likelihood Explanation

The attack requires only:
1. Deploying a malicious NEP-141 contract on NEAR (permissionless, costs a small storage deposit).
2. Calling `deploy_erc20_token` on Aurora to register the mapping (permissionless, no owner check).
3. Calling `ft_transfer_call` on the malicious NEP-141, which calls `ft_on_transfer` on Aurora with an inflated amount.

No privileged keys, admin access, or governance capture is needed. The attack is fully reachable by any unprivileged NEAR account.

### Recommendation

After the `ft_transfer_call` completes and before minting ERC-20 tokens, Aurora should verify that its NEP-141 balance actually increased by the claimed amount. Because NEAR is asynchronous, this requires restructuring the flow:

1. Before initiating `ft_transfer_call`, record Aurora's current NEP-141 balance for the token (via a view call or cached storage value).
2. In the `ft_on_transfer` callback, query Aurora's current NEP-141 balance and compute the actual delta.
3. Mint ERC-20 tokens equal to `min(args.amount, actual_delta)` rather than `args.amount` unconditionally.

Alternatively, maintain an internal accounting ledger of NEP-141 deposits per token and cross-check it against the claimed amount on every `ft_on_transfer` call.

### Proof of Concept

1. Deploy a malicious NEAR contract `evil.near` implementing NEP-141 with a `ft_transfer_call` that calls `ft_on_transfer` on Aurora with `amount = 1_000_000` but transfers `0` tokens to Aurora.
2. Call `deploy_erc20_token` on Aurora with `nep141 = "evil.near"` (no access control, succeeds for any caller).
3. Call `evil.near::ft_transfer_call(receiver_id="aurora", amount=1_000_000, msg=<attacker_evm_address>)`.
4. `evil.near` calls `aurora::ft_on_transfer({sender_id: attacker, amount: 1_000_000, msg: <attacker_evm_address>})`.
5. Aurora executes `receive_erc20_tokens` → `setup_receive_erc20_tokens_input(attacker_evm_address, 1_000_000)` → mints 1,000,000 ERC-20-EVIL tokens to the attacker.
6. Aurora holds 0 EVIL NEP-141 tokens; 1,000,000 ERC-20-EVIL tokens exist — the token is insolvent.
7. Attacker swaps ERC-20-EVIL for ETH on any Aurora DEX, draining real user liquidity.
