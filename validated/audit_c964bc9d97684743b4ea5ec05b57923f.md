Audit Report

## Title
Unbounded neuron iteration in SNS `compute_ballots_for_new_proposal` enables permanent governance DoS — (`rs/sns/governance/src/governance.rs`)

## Summary
The SNS Governance canister's `compute_ballots_for_new_proposal` performs a full linear scan of `self.proto.neurons` (a heap-resident `BTreeMap<String, Neuron>`) within a single synchronous message execution, with no instruction-limit guard. Any neuron holder can invoke this path via `make_proposal`. If the SNS neuron population has been grown to its configured `max_number_of_neurons` ceiling — achievable by any actor willing to stake the minimum amount across many accounts — the scan can exhaust the ~40 billion instruction per-message limit, causing every subsequent `make_proposal` call to trap and permanently freezing SNS governance. A secondary unbounded BFS in `cast_vote_and_cascade_follow` compounds the risk on the voting path.

## Finding Description
**Primary path — `compute_ballots_for_new_proposal`:**
In `rs/sns/governance/src/governance.rs`, the function iterates unconditionally over every entry in `self.proto.neurons` to build the electoral roll:

```rust
for (k, v) in self.proto.neurons.iter() {
    // voting power calculation and ballot insertion
}
```

There is no `is_over_instructions_limit()` check, no pagination, and no deferred processing. The entire scan must complete within one message execution. The only guard on neuron population is `check_neuron_population_can_grow`, which prevents adding neurons beyond `max_number_of_neurons` but does nothing to protect the ballot-creation loop once the cap is reached.

**Secondary path — `cast_vote_and_cascade_follow`:**
The SNS version runs a BFS over the follower graph in a plain `while !induction_votes.is_empty()` loop with no instruction-limit pause. The NNS counterpart in `rs/nns/governance/src/voting.rs` was explicitly retrofitted with:

```rust
if is_over_instructions_limit() {
    return;
}
```

…allowing it to pause and resume across heartbeat rounds. The SNS version has no equivalent.

**Contrast with NNS:**
The NNS caps neurons at `MAX_NUMBER_OF_NEURONS = 500_000` in `rs/nns/governance/src/governance.rs`, uses stable storage for neuron access, and employs a voting-power snapshot mechanism so ballot creation does not require a full heap scan in a single message. The SNS stores all neurons in heap memory, has a configurable (and raiseable) `max_number_of_neurons` bounded only by `MAX_NUMBER_OF_NEURONS_CEILING` in `rs/sns/governance/src/types.rs`, and has none of these mitigations.

**Exploit flow:**
1. Attacker stakes minimum `neuron_minimum_stake_e8s` from many accounts, creating neurons up to `max_number_of_neurons`.
2. Any neuron holder (including the attacker or a legitimate user) calls `make_proposal`.
3. `compute_ballots_for_new_proposal` iterates over all neurons; at sufficient count the instruction limit is exceeded.
4. The call traps with `CanisterInstructionLimitExceeded`. The state is not committed.
5. Every subsequent `make_proposal` call also traps — the DoS is persistent.

## Impact Explanation
This matches the High bounty impact: **"Application/platform-level DoS … not based on raw volumetric DDoS."** The SNS governance canister is the sole path for submitting proposals. A permanent trap on `make_proposal` blocks all governance actions: canister upgrades, parameter changes, treasury transfers. The SNS becomes frozen until NNS-level intervention (a slow, high-friction process requiring an NNS proposal to upgrade the SNS governance canister). The secondary BFS DoS on `cast_vote_and_cascade_follow` can additionally leave proposals in a permanently unresolved state.

## Likelihood Explanation
- `make_proposal` is callable by any neuron holder — no privileged role is required.
- Populating the SNS to `max_number_of_neurons` requires only `max_number_of_neurons × neuron_minimum_stake_e8s` in tokens, which is economically feasible for many SNS deployments, especially those with low minimum stake.
- The attacker does not need to be the one who ultimately triggers the trap; once the neuron count is high enough, every legitimate `make_proposal` call will also trap.
- The NNS having survived at scale provides no protection, as each SNS instance has an independent neuron store and independent governance parameters.
- The attack is repeatable and persistent: there is no self-healing mechanism.

## Recommendation
1. **Benchmark and conservatively cap `max_number_of_neurons`** so that `compute_ballots_for_new_proposal` at the cap stays well under the instruction limit, using the NNS benchmark infrastructure in `rs/nns/governance/src/neuron_store/benches.rs` as a template.
2. **Migrate ballot creation to a chunked or background approach** (analogous to the NNS voting-power snapshot) so the full neuron scan is not required in a single message execution.
3. **Add `is_over_instructions_limit()` guards to SNS `cast_vote_and_cascade_follow`** and implement the same pause-and-resume state machine present in `rs/nns/governance/src/voting.rs`.
4. **Move SNS neurons to stable storage** to reduce per-neuron access cost and enable cheaper range queries, as the NNS did.

## Proof of Concept
1. Deploy a local SNS (via PocketIC or the local replica) with `max_number_of_neurons` set to a large value (e.g., 100,000) and `neuron_minimum_stake_e8s` at the minimum.
2. Programmatically stake the minimum from many accounts to create neurons up to the cap.
3. Call `make_proposal` from any neuron holder.
4. Observe the call traps with `CanisterInstructionLimitExceeded` (IC error code 522).
5. Confirm all subsequent `make_proposal` calls also trap, demonstrating the persistent DoS.
6. Use the benchmark template in `rs/nns/governance/src/neuron_store/benches.rs` (specifically `check_projected_instructions`) to measure the exact neuron count at which the instruction limit is hit for the SNS ballot-creation loop.
