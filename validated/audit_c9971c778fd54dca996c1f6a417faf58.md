### Title
`EvmErc20V2.sol` `withdrawToNear` Encodes Sender Address Before Amount, Causing Precompile to Parse Incorrect Amount and Permanently Burn User Tokens Without NEAR Transfer — (`etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20V2.sol`'s `withdrawToNear` function encodes the caller's address (20 bytes) **before** the token amount (32 bytes) in the calldata sent to the `ExitToNear` precompile. The `ExitToNear` precompile, when compiled **without** the `error_refund` feature, expects the amount to begin immediately after the 1-byte flag. This encoding mismatch causes the precompile to parse a U256 value composed of sender address bytes concatenated with the high bytes of the actual amount — a value that always exceeds `u128::MAX` — triggering an `ERR_INVALID_AMOUNT` error. Because the Solidity code does not check the return value of the low-level `call`, the transaction succeeds, the ERC-20 tokens are permanently burned, and no NEP-141 tokens are ever transferred to NEAR.

---

### Finding Description

**`EvmErc20V2.sol` `withdrawToNear` input encoding:**

```solidity
// etc/eth-contracts/contracts/EvmErc20V2.sol, line 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    _burn(sender, amount);                                          // tokens burned first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    //                                    flag(1)  addr(20) amt(32)  recipient(N)
    uint input_size = 1 + 20 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no revert on failure
    }
}

    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    let mut buffer = [0; 20];
    buffer.copy_from_slice(&input[1..21]);
    let refund_address = Address::from_array(buffer);
    Ok((refund_address, &input[21..]))   // strips flag(1) + sender(20) → amount is next
}
``` [5](#0-4) 

**Step-by-step byte layout mismatch:**

| Byte offset | `EvmErc20V2.sol` sends | Precompile (no `error_refund`) reads |
|---|---|---|
| 0 | `0x01` (flag) | flag → stripped |
| 1–20 | `sender` (20 bytes) | **amount[0..20]** ← wrong |
| 21–52 | `amount` (32 bytes) | **amount[20..32] + recipient start** ← wrong |
| 53+ | `recipient` (N bytes) | rest of recipient ← wrong |

The parsed "amount" = `U256::from_big_endian([sender_bytes(20) || amount_bytes[0..12]])`. Since the sender address occupies the high 20 bytes of the 32-byte window, the resulting U256 is always far larger than `u128::MAX` (e.g., for sender `0xABCD...`, the value is `≈ 2^159`). `parse_amount` returns `Err("ERR_INVALID_AMOUNT")`.

The EVM `call` opcode returns `0` (failure), but the Solidity assembly block stores it in `res` and **never checks it**. The outer function returns normally. The `_burn` that already executed is not rolled back. [1](#0-0) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any user who calls `withdrawToNear` on a deployed `EvmErc20V2` contract when the Aurora Engine is compiled without the `error_refund` feature will:

1. Have their ERC-20 tokens permanently burned on Aurora (line 55: `_burn(sender, amount)`).
2. Receive zero NEP-141 tokens on NEAR (precompile silently fails).
3. Have no recourse — there is no refund path without `error_refund`, and the transaction is already finalized.

The tokens are destroyed with no corresponding credit anywhere. This is a direct, irreversible loss of user funds. [6](#0-5) 

---

### Likelihood Explanation

**High.** The entry path is a standard, publicly callable ERC-20 function (`withdrawToNear`) on a production contract (`EvmErc20V2.sol`) in the Aurora Engine repository. No special privileges are required — any token holder can trigger this. The bug fires on every invocation when the `error_refund` feature is absent from the compiled WASM. The `error_refund` feature is a compile-time toggle with a distinct code path, meaning any deployment without it is fully exposed. [1](#0-0) [7](#0-6) 

---

### Recommendation

1. **Check the precompile call result in `EvmErc20V2.sol`**: Revert if the low-level `call` to the precompile returns `0`. This prevents tokens from being burned when the precompile fails:
   ```solidity
   assembly {
       let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
       if iszero(res) { revert(0, 0) }
   }
   ```
2. **Enforce `error_refund` when deploying `EvmErc20V2`**: Document and enforce that `EvmErc20V2.sol` is only compatible with Aurora Engine builds that include the `error_refund` feature, since the input format is designed for that path.
3. **Alternatively**, align `EvmErc20V2.sol`'s encoding with the non-`error_refund` precompile format (omit the sender prefix) and handle refunds at the Solidity level. [8](#0-7) [9](#0-8)

### Proof of Concept

**Setup**: Aurora Engine compiled **without** `error_refund`. `EvmErc20V2` deployed and mapped to a NEP-141 token.

**Attack path** (unprivileged token holder):

1. Alice holds 100 units of an ERC-20 token backed by `EvmErc20V2`.
2. Alice calls `withdrawToNear("alice.near", 100)`.
3. `_burn(alice_address, 100)` executes — Alice's balance drops to 0. ✓
4. Input to precompile: `[0x01][alice_address:20][100_as_bytes32:32]["alice.near":9]`
5. Precompile (no `error_refund`) strips only the flag byte, reads bytes 0–31 as amount:
   - Parsed amount = `U256::from_big_endian([alice_address_bytes || 0x00000000000000000000000000000064[0..12]])`
   - This value ≫ `u128::MAX` → `parse_amount` returns `Err("ERR_INVALID_AMOUNT")`.
6. EVM `call` returns `0` (failure). `res` is set to `0` but never checked.
7. `withdrawToNear` returns normally (no revert).
8. Alice's 100 tokens are gone. `alice.near` receives nothing. Funds permanently frozen. [1](#0-0) [4](#0-3) [3](#0-2)
