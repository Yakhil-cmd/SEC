Audit Report

## Title
Unauthenticated `reset_timers` Allows Any Caller to Repeatedly Delay SNS Periodic Task Execution - (File: rs/sns/governance/canister/canister.rs, rs/sns/swap/canister/canister.rs, rs/sns/root/canister/canister.rs)

## Summary

The `reset_timers` update method in SNS Governance, SNS Swap, and SNS Root canisters is publicly callable by any principal with no caller authentication. The sole protection is a time-based cooldown. Because `init_timers()` cancels the existing running timer and reschedules it from the current moment, an attacker who calls `reset_timers` at each cooldown expiry can continuously push back periodic task execution. When `RUN_PERIODIC_TASKS_INTERVAL >= RESET_TIMERS_COOL_DOWN_INTERVAL`, the periodic task never fires; even when the interval is shorter, the attacker causes repeated, measurable delays to governance-critical and financially-critical operations.

## Finding Description

All three canisters expose `reset_timers` as a bare `#[update]` with no `ic_cdk::api::msg_caller()` / `ic_cdk::api::is_controller()` check:

**SNS Governance** (`rs/sns/governance/canister/canister.rs`, lines 644–661): [1](#0-0) 

**SNS Swap** (`rs/sns/swap/canister/canister.rs`, lines 348–365): [2](#0-1) 

**SNS Root** (`rs/sns/root/canister/canister.rs`, lines 487–506): [3](#0-2) 

In each case the only guard is the cooldown check. `init_timers()` performs two destructive operations: it schedules a fresh `set_timer_interval` starting from now, then calls `clear_timer` on the previously saved timer ID, resetting the countdown to zero:

**SNS Governance `init_timers`** (`rs/sns/governance/canister/canister.rs`, lines 626–642): [4](#0-3) 

**SNS Swap `init_timers`** (`rs/sns/swap/canister/canister.rs`, lines 318–346): [5](#0-4) 

**SNS Root `init_timers`** (`rs/sns/root/canister/canister.rs`, lines 466–485): [6](#0-5) 

**Exploit flow:**
1. Attacker waits until `now - last_reset_timestamp_seconds >= RESET_TIMERS_COOL_DOWN_INTERVAL` (600 s for Governance/Swap; one week for Root).
2. Attacker sends an ingress `reset_timers` call from any principal — no special identity required.
3. `init_timers()` clears the existing timer and starts a new interval from the current moment.
4. If `RUN_PERIODIC_TASKS_INTERVAL >= RESET_TIMERS_COOL_DOWN_INTERVAL`, the attacker repeats step 1–3 every cooldown period, and the periodic task never fires. If the interval is shorter, the attacker still causes a delay of up to `RUN_PERIODIC_TASKS_INTERVAL` on each reset.

The existing cooldown check is insufficient because it only prevents calls within the same cooldown window; it does not prevent an attacker from calling at every cooldown boundary indefinitely.

## Impact Explanation

**SNS Swap** is the highest-impact target: periodic tasks drive swap lifecycle transitions (finalization, token distribution). An attacker calling every 600 seconds can delay or indefinitely defer swap completion, leaving participants' funds locked and potentially causing the swap to expire. This is a concrete financial harm to SNS launch participants.

**SNS Governance**: periodic tasks include neuron reward distribution and proposal processing. Continuous timer resets delay reward payouts and governance actions for all SNS token holders.

**SNS Root**: the one-week cooldown limits the attack to once per week, but each call can delay the next periodic task (canister health checks, archive management) by up to `RUN_PERIODIC_TASKS_INTERVAL`.

This maps to the allowed High impact: *"Significant SNS security impact with concrete user or protocol harm"* and *"Application/platform-level DoS … not based on raw volumetric DDoS"* ($2,000–$10,000).

## Likelihood Explanation

The attack requires only a standard ingress message from any principal — no privileged key, no governance majority, no threshold corruption. The attacker needs only to wait for the cooldown to expire and send a single update call. This is trivially automatable with a cron job or a simple script monitoring `get_timers`. SNS canisters are deployed on mainnet and their Candid interfaces are publicly known.

## Recommendation

Add a caller authorization check at the top of each `reset_timers` implementation, restricting it to controllers or the NNS governance canister, consistent with the pattern already used in `rs/migration_canister/src/privileged.rs`:

```rust
#[update]
fn reset_timers(_request: ResetTimersRequest) -> ResetTimersResponse {
    let caller = ic_cdk::api::caller();
    assert!(
        ic_cdk::api::is_controller(&caller) || caller == GOVERNANCE_CANISTER_ID,
        "Caller is not authorized to reset timers"
    );
    // ... existing cooldown check and init_timers() ...
}
```

## Proof of Concept

1. Identify a live SNS Swap canister ID on mainnet.
2. Call `get_timers` to read `last_reset_timestamp_seconds`.
3. Wait until `now - last_reset_timestamp_seconds >= 600` seconds.
4. Send an ingress `reset_timers {}` call from any principal (e.g., `dfx canister call <swap_id> reset_timers '(record {})'`).
5. Observe via `get_timers` that `last_reset_timestamp_seconds` is updated to now and the internal timer interval restarts from zero.
6. If the periodic task was scheduled to fire imminently, it is now delayed by the full `RUN_PERIODIC_TASKS_INTERVAL`.
7. Automate steps 3–6 to repeat every 600 seconds, continuously deferring periodic task execution.

The integration test `run_canister_reset_timers_cannot_be_spammed_test` in `rs/sns/integration_tests/src/timers.rs` confirms that any caller can invoke `reset_timers` and that the only protection is the cooldown, validating the exploit path in a controlled environment.
