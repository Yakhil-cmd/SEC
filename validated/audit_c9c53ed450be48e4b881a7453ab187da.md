Audit Report

## Title
Stable Memory Dirty Pages Excluded from DTS Yield Threshold — Page Copy Monopolizes Execution Round Without Yielding - (File: `rs/embedders/src/wasm_executor.rs`)

## Summary
The dirty-page yield optimization in `process()` at line 682 counts only `res.wasm_dirty_pages.len()` when deciding whether to yield before copying dirty pages. `res.stable_memory_dirty_pages` is never included. A canister writing exclusively to stable memory can dirty up to 2 GiB (the `STABLE_MEMORY_DIRTY_PAGE_LIMIT_MESSAGE` cap) without triggering `yield_for_dirty_memory_copy()`, causing all stable page copying to execute inline in a single slice, blocking other canisters from being scheduled during that round.

## Finding Description

`InstanceRunResult` carries both fields: [1](#0-0) 

At line 682, only the Wasm heap field is read: [2](#0-1) 

The yield threshold is `DEFAULT_MAX_DIRTY_PAGES_WITHOUT_OPTIMIZATION` = 1 GiB / 4 KiB = **262,144 pages**: [3](#0-2) 

The stable memory dirty page limit for regular messages is `STABLE_MEMORY_DIRTY_PAGE_LIMIT_MESSAGE` = 2 GiB / 4 KiB = **524,288 pages**: [4](#0-3) 

When a canister writes exclusively to stable memory: `res.wasm_dirty_pages.len() == 0` → `dirty_pages == 0` → the yield condition is false → `yield_for_dirty_memory_copy()` is never called. Execution falls through directly to the stable memory page copy in the same slice: [5](#0-4) 

Because `num_dirty_pages` remains 0, the `dirty_page_copy_overhead` (3,000 instructions/page) is also never charged, meaning the page copy work is entirely unaccounted in the instruction budget: [6](#0-5) 

The confirmed DTS yield mechanism — `try_yield_for_dirty_memory_copy()` transitions to `Paused`, fires `pause_callback`, then blocks on `wait_for_resume_or_abort()` — is entirely bypassed for the stable memory path: [7](#0-6) 

The existing test `yield_for_dirty_pages_copy_works` validates the Wasm heap path only; no equivalent test exists for stable memory: [8](#0-7) 

## Impact Explanation
A single unprivileged canister can dirty up to 2 GiB of stable memory in one update message. The resulting page copy (up to 524,288 × 4 KiB) executes inline in the same execution slice without yielding and without instruction overhead, extending the wall-clock duration of that execution round and starving all other canisters on the subnet from being scheduled. This matches the allowed High impact: **Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000).

## Likelihood Explanation
The attack requires only deploying a canister and sending a single update message calling `stable64_write` across a range exceeding 1 GiB. No privileged access, governance majority, or key material is needed. The stable memory dirty page limit (2 GiB) bounds the maximum damage per message, but the gap between the yield threshold (1 GiB) and the stable limit (2 GiB) is the exploitable window. The attack is repeatable at will by any canister owner.

## Recommendation
At line 682, include `res.stable_memory_dirty_pages.len()` in the dirty page count:

```rust
let dirty_pages = NumOsPages::from(
    (res.wasm_dirty_pages.len() + res.stable_memory_dirty_pages.len()) as u64
);
```

This ensures a canister writing exclusively or predominantly to stable memory triggers the same yield-and-resume behavior as one writing to the Wasm heap, bounding page copy work per round to `max_dirty_pages_without_optimization` pages regardless of memory type.

## Proof of Concept
A differential integration test analogous to `yield_for_dirty_pages_copy_works` using `stable64_write` instead of `i32.store`:

```wat
(module
  (import "ic0" "stable64_grow" (func $stable64_grow (param i64) (result i64)))
  (import "ic0" "stable64_write" (func $stable64_write (param i64 i64 i64)))
  (import "ic0" "msg_reply" (func $msg_reply))
  (func (export "canister_update write_stable")
    (drop (call $stable64_grow (i64.const 16385))) ;; grow > 1 GiB
    (local $i i64)
    (local.set $i (i64.const 1073745920))          ;; 1 GiB + 4096
    (loop $loop
      (call $stable64_write (local.get $i) (i64.const 0) (i64.const 1))
      (br_if $loop
        (local.tee $i (i64.sub (local.get $i) (i64.const 4096))))
    )
    (call $msg_reply)
  )
  (memory 1)
)
```

Send two ingress messages. **Expected behavior with fix:** after round 1, message[0] is `Processing` and message[1] is `Received` (yield triggered). **Observed behavior without fix:** message[0] completes in round 1 (no yield), and message[1] also starts in round 1 — demonstrating the page copy monopolized the round.

fn yield_for_dirty_pages_copy_works() {
    let env = ic_state_machine_tests::StateMachineBuilder::new()
        .with_subnet_type(SubnetType::Application)
        .build();

    let wasm = wat::parse_str(WRITE_MORE_THAN_1G_WAT).unwrap();
    let canister_id = env
        .install_canister_with_cycles(wasm, vec![], None, INITIAL_CYCLES_BALANCE)
        .unwrap();

    let mut payload = ic_state_machine_tests::PayloadBuilder::new().with_nonce(0);
    // Send two ingress messages to the same canister.
    for _ in 0..2 {
        payload = payload.ingress(PrincipalId::new_anonymous(), canister_id, "write", vec![]);
    }
    let message_ids = payload.ingress_ids();
    env.execute_payload(payload);

    // Neither of messages should be completed after the first round.
    assert_matches!(
        ingress_state(env.ingress_status(&message_ids[0])),
        Some(IngressState::Processing)
    );
    assert_matches!(
        ingress_state(env.ingress_status(&message_ids[1])),
        Some(IngressState::Received)
    );

    env.tick();

    // Only the first message must be completed after two rounds.
    assert_matches!(
        ingress_state(env.ingress_status(&message_ids[0])),
        Some(IngressState::Completed(_))
    );
    assert_matches!(
        ingress_state(env.ingress_status(&message_ids[1])),
        Some(IngressState::Received)
    );
}
