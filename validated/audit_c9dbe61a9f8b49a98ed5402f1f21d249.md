Audit Report

## Title
ckBTC Minter `finalize_transaction` Hard Panic on Zero `withdrawal_fee` Permanently Halts BTC Withdrawals - (File: rs/bitcoin/ckbtc/minter/src/state.rs)

## Summary

The `finalize_transaction` function in the ckBTC minter contains a hard `assert!(fee > 0, "withdraw_fee is zero")` in the `ToCancel` branch. When a cancellation replacement transaction carries a zero or `None` `withdrawal_fee`, the minter's heartbeat-driven finalization loop traps deterministically on every invocation. This was confirmed to have occurred on mainnet on 2025-06-21, blocking all ckBTC → BTC withdrawals until a canister upgrade was deployed.

## Finding Description

In `rs/bitcoin/ckbtc/minter/src/state.rs`, `finalize_transaction` handles three `SubmittedWithdrawalRequests` variants. The `ToCancel` branch at lines 1049–1053 reads:

```rust
SubmittedWithdrawalRequests::ToCancel { requests, reason } => {
    let requests = requests.into_iter().collect::<BTreeSet<_>>();
    let fee = finalized_tx.withdrawal_fee.unwrap_or_default();
    let fee = fee.bitcoin_fee + fee.minter_fee;
    assert!(fee > 0, "withdraw_fee is zero");
``` [1](#0-0) 

`withdrawal_fee` is typed `Option<WithdrawalFee>`. When it is `None`, `unwrap_or_default()` yields `WithdrawalFee { bitcoin_fee: 0, minter_fee: 0 }`, making `fee = 0` and firing the `assert!`.

The call chain is: heartbeat → `finalize_requests` → `process_maybe_finalized_transactions` → `confirm_transaction` (lib.rs:340) → `state::audit::confirm_transaction` (audit.rs:114) → `state.finalize_transaction(txid)`. [2](#0-1) [3](#0-2) 

The upstream trigger is in `finalize_requests` / `resubmit_transactions` in `lib.rs`, where a stuck transaction is replaced with a `ToCancel` transaction constructed as:

```rust
let new_tx = state::SubmittedBtcTransaction {
    ...
    withdrawal_fee: Some(total_fee),
    ...
};
replace_transaction(old_txid, new_tx, replaced_reason);
``` [4](#0-3) 

If `total_fee` resolves to `WithdrawalFee { bitcoin_fee: 0, minter_fee: 0 }` (e.g., due to an anomalously low fee-per-vbyte estimate), the downstream `assert!` fires when that cancellation transaction is later confirmed on-chain. Because the trap rolls back the heartbeat message but leaves the transaction in state, every subsequent heartbeat re-enters the same path and traps again.

The mainnet upgrade proposal explicitly confirms this sequence:

> "There is a deterministic panic occurring in the minter when it tries to resubmit those transactions, which explains why those transactions are currently stuck. This should be completely fixed by #5713." [5](#0-4) 

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"* and *"Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."*

The heartbeat is the sole mechanism for finalizing submitted BTC transactions. A single trapped heartbeat blocks **all** pending ckBTC → BTC withdrawals for all users, not just the affected request. Funds are not permanently lost but are inaccessible until a governance-approved canister upgrade is deployed. The mainnet incident on 2025-06-21 confirmed three real user withdrawals were stuck.

## Likelihood Explanation

The condition is reachable by any unprivileged user submitting a `retrieve_btc` request. No special privileges are required. The triggering condition (anomalously low fee-per-vbyte from the Bitcoin canister) is an external dependency that occurred on mainnet without any attacker involvement. The sequence is deterministic once triggered: the same heartbeat path traps on every invocation with no self-recovery mechanism. The mainnet incident proves this is not theoretical.

## Recommendation

Replace the hard `assert!` with a graceful error path in the `ToCancel` branch of `finalize_transaction`:

```rust
if fee == 0 {
    log!(Priority::Error,
        "withdraw_fee is zero for txid {txid}; skipping cancellation reimbursement");
    // proceed with cancellation without fee reimbursement, or return None
}
```

Additionally, `resubmit_transactions` should validate `total_fee > 0` before constructing a `ToCancel` replacement transaction, preventing zero-fee cancellation transactions from ever entering state.

## Proof of Concept

1. Submit a `retrieve_btc` request when the minter's fee estimator returns a near-zero fee per vbyte.
2. The transaction is not mined (stuck in mempool or evicted).
3. After `MIN_RESUBMISSION_DELAY`, the minter calls `resubmit_transactions`; a `ToCancel` replacement transaction is constructed with `withdrawal_fee: Some(WithdrawalFee { bitcoin_fee: 0, minter_fee: 0 })`.
4. When the Bitcoin network confirms the cancellation transaction, `finalize_requests` calls `confirm_transaction` → `finalize_transaction`.
5. `assert!(fee > 0, "withdraw_fee is zero")` fires at `state.rs:1053`; the heartbeat traps.
6. All subsequent heartbeats trap on the same path; no withdrawals can be finalized.

A unit test can reproduce this by constructing a `SubmittedBtcTransaction` with `requests: SubmittedWithdrawalRequests::ToCancel { ... }` and `withdrawal_fee: Some(WithdrawalFee { bitcoin_fee: 0, minter_fee: 0 })`, then calling `state.finalize_transaction(&txid)` and observing the panic. The mainnet incident is documented in `rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md` and fixed by PR #5713. [6](#0-5)
