### Title
`native_per_pubdata` Floor-Division Truncation to Zero Allows Free Pubdata Publication — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` conversion ratio is computed with floor (truncating) division via `wrapping_div`, while the analogous `native_per_gas` ratio uses ceiling division (`div_ceil`). When `pubdata_price < native_price`, the floor division silently produces zero, making all pubdata publication free of native-resource cost for the remainder of that transaction. An unprivileged sender can exploit this to publish arbitrarily large state diffs at zero native cost, forcing the operator to absorb the corresponding L1 data fees.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` the two conversion ratios are computed as:

```rust
// native_per_gas — uses ceiling division
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))…;

// native_per_pubdata — uses floor division
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

The same asymmetry is present in the off-chain helper:

```rust
let native_per_gas    = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(())?;
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

`native_per_pubdata` is subsequently used in two places:

1. **Intrinsic pubdata overhead** — deducted from the native limit at resource-creation time:
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   ``` [3](#0-2) 

2. **Post-execution pubdata charge** — the native cost of all pubdata generated during execution:
   ```rust
   let native = current_pubdata_spent.checked_mul(native_per_pubdata)…;
   ``` [4](#0-3) 

When `native_per_pubdata == 0`, both multiplications yield zero, so the transaction pays **no native resources** for any amount of pubdata it generates, regardless of how many storage slots it writes or how many L2→L1 messages it emits.

---

### Impact Explanation

`native_per_pubdata = 0` means:

- The intrinsic pubdata overhead charged at validation time is zero.
- `get_resources_to_charge_for_pubdata` returns zero native cost for every pubdata byte written during execution.
- `check_enough_resources_for_pubdata` always passes.
- The `delta_gas` adjustment in `compute_gas_refund` is also zero, so no extra gas is charged to compensate. [5](#0-4) 

A transaction that writes many storage slots (each slot write produces pubdata) or emits large L2→L1 messages can therefore publish an unbounded amount of L1 data without paying the corresponding native resource cost. The operator must pay the L1 data fees out of pocket, constituting a direct financial loss. This is a **resource accounting bug / public funds-loss path** that is directly reachable by any unprivileged transaction sender.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is realistic in production:

- Both values are independent operator-controlled parameters set in the block metadata oracle.
- The default test configuration already has `native_price = U256::from(10)` and `pubdata_price = U256::from(0)`. [6](#0-5) 
- Any operator configuration where `pubdata_price` is set to a small non-zero value (e.g., to subsidise pubdata) but still less than `native_price` triggers the truncation.
- The attacker needs only to submit a standard L2 transaction; no privileged access is required.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — truncates to 0 when pubdata_price < native_price)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — consistent with native_per_gas)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. [7](#0-6) [8](#0-7)

### Proof of Concept

**Setup**: Operator sets `native_price = 10`, `pubdata_price = 5` (a valid, non-zero pubdata price that is less than the native price).

**Step 1**: Attacker submits a transaction with `gas_price = 1000` (satisfies `gas_price >= native_price`).

**Step 2**: Bootloader computes:
```
native_per_gas     = ceil(1000 / 10) = 100   ✓ (non-zero, correct)
native_per_pubdata = floor(5 / 10)   = 0     ✗ (should be 1 with ceiling)
```

**Step 3**: Transaction writes 10,000 storage slots. Each slot write produces pubdata. Total pubdata = ~320,000 bytes.

**Step 4**: `get_resources_to_charge_for_pubdata` computes:
```
native_cost = 320_000 * 0 = 0
```
The pubdata check passes trivially. The transaction succeeds.

**Step 5**: The operator must publish 320,000 bytes of state diffs to L1 but received zero native resource payment for them. With `pubdata_price = 5` the operator expected to receive `320,000 * 5 = 1,600,000` units of pubdata revenue; they receive nothing. [9](#0-8)

```rust
pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
```
