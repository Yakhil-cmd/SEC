Audit Report

## Title
Silent Integer Overflow in `add_expiration` Produces Epoch-0 Expiration in ICRC-21 Consent Message for `expires_at ≥ i64::MAX + 1` — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary

In the `FieldsDisplayMessage` branch of `add_expiration`, a caller-supplied `u64` nanosecond timestamp is cast directly to `i64` without overflow checking. For `expires_at = u64::MAX`, the cast wraps to `-1`, integer division by `10^9` yields `0`, and the re-cast to `u64` emits `TimestampSeconds { amount: 0 }` — epoch 1970-01-01 — with no error. The `GenericDisplayMessage` branch has the identical arithmetic bug but its `from_unix_timestamp(-1)` call succeeds and formats as "Thu, 31 Dec 1969 23:59:59 +0000", also wrong. The actual on-chain approval stores the correct `u64::MAX` value; only the ICRC-21 consent display is corrupted.

## Finding Description

**Public entrypoint:** `icrc21_canister_call_consent_message` is a `#[update]` endpoint callable by any principal, including anonymous. [1](#0-0) 

**No guard on `expires_at`:** The decoded `expires_at: Option<u64>` from `ApproveArgs` is passed directly to `with_expires_at` and then to `add_expiration` with no range check. [2](#0-1) 

**The overflow — `FieldsDisplayMessage` branch:**
```
expires_at = u64::MAX = 18446744073709551615
step 1: expires_at as i64  →  -1          (wraps; defined in Rust)
step 2: -1_i64 / 10_i64.pow(9)  →  0     (truncates toward zero)
step 3: 0_i64 as u64  →  0               (epoch 0 = 1970-01-01)
``` [3](#0-2) 

**The overflow — `GenericDisplayMessage` branch:** Same `(ts as i64) / 10_i64.pow(9)` arithmetic. For `u64::MAX`, `seconds = -1`; `time::OffsetDateTime::from_unix_timestamp(-1)` succeeds and formats as "Thu, 31 Dec 1969 23:59:59 +0000" — wrong but not caught by the `Err(_)` guard. [4](#0-3) 

**Additional overflow shape** for `expires_at = i64::MAX as u64 + 1 = 9223372036854775808`:
```
as i64  →  i64::MIN = -9223372036854775808
/ 10^9  →  -9223372036
as u64  →  18446744064486375616  (wraps to a spuriously large far-future value)
```

**No existing test covers this range.** All existing tests use realistic near-present timestamps. [5](#0-4) 

## Impact Explanation

ICRC-21 consent messages are the security boundary between a dApp and a hardware-wallet signer (e.g., Ledger). The wallet displays the structured `FieldsDisplayMessage` fields verbatim and asks the user to confirm. If "Approval expiration" shows `TimestampSeconds { amount: 0 }` (epoch 0, year 1970), a user may reasonably conclude the approval is already expired and therefore harmless — while the actual on-chain approval carries `expires_at = u64::MAX` (~year 2554), granting the spender an effectively permanent allowance. This constitutes a **significant ICRC ledger security impact with concrete user harm**: the security system specifically designed to prevent dApp deception is itself producing deceptive output, enabling unauthorized long-term access to user funds. This maps to the **High** impact tier: "Significant ck-token, ledger, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation

The trigger is a single valid Candid-encoded `ApproveArgs` with `expires_at = Some(u64::MAX)` submitted to the public `icrc21_canister_call_consent_message` endpoint. No privileged role, no key material, and no threshold corruption is required. `u64::MAX` is a legal `nat64` in Candid and passes all ledger-side validation. The attack surface is every ICRC-2 ledger deployment that exposes ICRC-21 (all production ICRC-1/ICRC-2 ledgers on the IC). The technical barrier is trivial; the constraint is that a victim must use a hardware wallet relying on the ICRC-21 consent flow with a malicious dApp.

## Recommendation

Replace the unchecked `as i64` cast with arithmetic performed in `u64` space in both branches of `add_expiration`:

```rust
// Instead of: let seconds = (expires_at as i64) / 10_i64.pow(9);
let seconds_u64 = expires_at / 1_000_000_000u64;
let seconds = match i64::try_from(seconds_u64) {
    Ok(s) => s,
    Err(_) => {
        // FieldsDisplayMessage: push an error Value::Text; GenericDisplayMessage: return format!("Invalid timestamp: {expires_at}")
        return;
    }
};
```

Dividing first in `u64` space avoids the overflow for all representable nanosecond timestamps. The `GenericDisplayMessage` branch already has an `Err(_)` fallback path for `from_unix_timestamp`; the `FieldsDisplayMessage` branch needs an equivalent guard.

    expires_at: Some(u64::MAX),

    from_subaccount: None,
    expected_allowance: None,
    created_at_time: None,
    fee: None,
    memo: None,
};
let request = ConsentMessageRequest {
    method: "icrc2_approve".to_owned(),
    arg: Encode!(&approve_args).unwrap(),
    user_preferences: ConsentMessageSpec {
        metadata: ConsentMessageMetadata {
            language: "en".to_string(),
            utc_offset_minutes: None,
        },
        device_spec: Some(DisplayMessageType::FieldsDisplay),
    },
};
let info = icrc21_consent_message(&env, canister_id, Principal::anonymous(), request).unwrap();
// Actual:   TimestampSeconds { amount: 0 }   ← epoch 0, 1970-01-01
// Expected: TimestampSeconds { amount: 18446744073 } or an explicit error
```

The arithmetic is deterministic and fully reproducible in a local state-machine test without any mainnet interaction.
