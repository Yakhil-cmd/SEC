Audit Report

## Title
SNS Governance Voting Rewards Permanently Lost When All Voters Have Zero Voting Power During a Reward Round - (File: `rs/sns/governance/src/governance.rs`)

## Summary
In `distribute_rewards`, when `total_reward_shares == 0` but `considered_proposals` is non-empty, the entire `rewards_purse_e8s` for the round is silently discarded. Proposals are settled (ballots cleared, `reward_event_end_timestamp_seconds` set), causing `rewards_rolled_over()` to return `false` because it checks only `settled_proposals.is_empty()`. As a result, `e8s_equivalent_to_be_rolled_over()` returns `0` and the purse is permanently unrecoverable.

## Finding Description
In `rs/sns/governance/src/governance.rs`, `distribute_rewards` computes `rewards_purse_e8s` from token supply and reward rate, then sums `total_reward_shares` from ballot `voting_power` fields across all `ReadyToSettle` proposals.

When `total_reward_shares == dec!(0)`, the maturity distribution block is skipped entirely, leaving `distributed_e8s_equivalent = 0`: [1](#0-0) 

The function then unconditionally settles all `considered_proposals` — setting `reward_event_end_timestamp_seconds` and clearing ballots — regardless of whether any rewards were distributed: [2](#0-1) 

The new `RewardEvent` is written with `settled_proposals: considered_proposals` (non-empty) and `distributed_e8s_equivalent: 0`: [3](#0-2) 

In the next round, `e8s_equivalent_to_be_rolled_over()` calls `rewards_rolled_over()`, which returns `self.settled_proposals.is_empty()`: [4](#0-3) 

Because `settled_proposals` is non-empty, `rewards_rolled_over()` returns `false`, `e8s_equivalent_to_be_rolled_over()` returns `0`, and the entire purse is gone. The same structural flaw exists in NNS governance's `rs/nns/governance/src/reward/calculation.rs`: [5](#0-4) 

The existing test `zero_total_reward_shares` confirms the code path is reachable and that `distributed_e8s_equivalent == 0` with non-empty `settled_proposals` is the accepted outcome — but the test uses `EmptyLedger` (supply = 0), so no actual purse is lost in the test, masking the real-world impact: [6](#0-5) [7](#0-6) 

## Impact Explanation
The entire `rewards_purse_e8s` for the affected round — computed as `supply * reward_rate * round_duration` plus any rolled-over amount from prior rounds — is permanently unrecoverable. It is never minted as maturity for any neuron and is not carried forward. For an SNS with a meaningful token supply and active reward rate, this represents a significant, irreversible loss of governance token maturity per affected round. This matches: **High ($2,000–$10,000) — Significant SNS security impact with concrete user or protocol harm.**

## Likelihood Explanation
A neuron's `voting_power` is zero when `cached_neuron_stake_e8s <= neuron_fees_e8s`. The `neuron_fees_e8s` field accumulates from rejected governance proposals. An unprivileged participant in a small SNS (few neurons) can engineer this condition by:
1. Submitting proposals that get rejected, accumulating `neuron_fees_e8s` until `neuron_fees_e8s >= cached_neuron_stake_e8s` (voting power = 0).
2. Ensuring the depleted neuron is the only one that cast a ballot on a proposal reaching `ReadyToSettle` (feasible in a small SNS where other neurons do not follow or vote).
3. When `distribute_rewards` runs, `total_reward_shares == 0` with non-empty `considered_proposals`, and the purse is dropped.

The attack is repeatable each reward round under the same conditions and requires no special privileges beyond the ability to submit proposals.

## Recommendation
When `total_reward_shares == 0` but `considered_proposals` is non-empty, the rewards purse should be rolled over rather than discarded. The targeted fix is to extend `rewards_rolled_over()` (or `e8s_equivalent_to_be_rolled_over()`) to also return the purse when `distributed_e8s_equivalent == 0` and `total_available_e8s_equivalent > 0`, regardless of whether proposals were settled. Alternatively, add a third condition: roll over when `distributed_e8s_equivalent == 0` and `total_available_e8s_equivalent > 0`. This fix must be applied symmetrically to both `rs/sns/governance/src/types.rs` and `rs/nns/governance/src/reward/calculation.rs`.

## Proof of Concept
**State setup:**
- SNS has one neuron `N` with `cached_neuron_stake_e8s = 1_000_000` and `neuron_fees_e8s = 1_000_000` (net stake = 0, voting power = 0).
- Token supply is non-zero (e.g., 1_000_000_000 e8s).
- One proposal `P` is created while `N` has zero voting power; `N`'s ballot records `voting_power: 0`.
- Proposal `P` is decided and enters `ReadyToSettle`.

**Execution trace:**
1. `rewards_purse_e8s` is computed as non-zero (e.g., 100_000_000 e8s).
2. `considered_proposals = [P]` (non-empty).
3. `total_reward_shares = 0` (only ballot has `voting_power: 0`).
4. Maturity distribution is skipped; `distributed_e8s_equivalent = 0`.
5. Proposal `P` is settled (ballots cleared, `reward_event_end_timestamp_seconds` set).
6. New `RewardEvent` written: `settled_proposals = [P]`, `distributed_e8s_equivalent = 0`, `total_available_e8s_equivalent = 100_000_000`.
7. Next round: `e8s_equivalent_to_be_rolled_over()` → `rewards_rolled_over()` → `settled_proposals.is_empty()` → `false` → returns `0`. The 100_000_000 e8s purse is permanently lost.

**Reproducible test:** Extend the existing `zero_total_reward_shares` test in `rs/sns/integration_tests/src/neuron.rs` to use a ledger with non-zero supply, then assert that `reward_event.total_available_e8s_equivalent > 0` and that the next round's `e8s_equivalent_to_be_rolled_over()` equals that amount. The test will fail, confirming the bug.
