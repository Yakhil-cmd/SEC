### Title
Permanent Fund Freeze in `ExitToNear` Precompile When `error_refund` Feature Is Disabled — (`engine-precompiles/src/native.rs`)

### Summary

The `ExitToNear` precompile burns ERC-20 tokens (or deducts ETH from the user's EVM balance) before scheduling an `ft_transfer`/`ft_transfer_call` NEAR promise to the destination NEP-141 contract. When the `error_refund` compile-time feature is **not** enabled, no callback is attached to that promise. If the promise fails for any reason (unregistered recipient, paused NEP-141, insufficient storage deposit, etc.), the burned tokens or deducted ETH are permanently unrecoverable — an exact analog of the reported vulnerability class.

### Finding Description

Inside `ExitToNear::run()`, the callback arguments struct is built conditionally:

```rust
// engine-precompiles/src/native.rs  lines 449-455
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← refund path is compiled out
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is absent, `refund` is always `None`. For the common ERC-20 and ETH exit paths `transfer_near` is also `None`, so `callback_args` equals the default value and the promise is emitted **without any callback**:

```rust
// engine-precompiles/src/native.rs  lines 470-483
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // bare promise, no failure handler
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { base: transfer_promise, callback: … })
};
``` [2](#0-1) 

The refund logic that would re-mint burned ERC-20 tokens or return ETH lives in `engine::refund_on_error`, called only from `exit_to_near_precompile_callback` when the callback is present and the promise result is a failure:

```rust
// engine/src/contract_methods/connector.rs  lines 231-239
} else if let Some(args) = args.refund {
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    …
}
``` [3](#0-2) 

Without the callback, this branch is never reached. The `error_refund` feature is **not** listed in the `default` feature set of either `aurora-engine` or `aurora-engine-precompiles`:

```toml
# engine/Cargo.toml  line 43-48
[features]
default = ["std"]
…
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [4](#0-3) 

### Impact Explanation

**Permanent freezing of funds (Critical).**

- **ERC-20 exit path**: The ERC-20 contract burns the user's tokens and calls the precompile. The NEP-141 `ft_transfer` promise is then scheduled. If it fails, the ERC-20 tokens are gone from the EVM and the NEP-141 balance is never credited — the value is destroyed.
- **ETH exit path**: The user's ETH is transferred into the `exitToNear` precompile address (`0xe9217bc7…`) during the EVM call. If the subsequent `ft_transfer` to the ETH connector fails, the ETH is locked inside the precompile address with no recovery path.

The integration tests explicitly confirm this behavior:

```rust
// engine-tests/src/tests/erc20_connector.rs  lines 656-665
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

The same pattern is confirmed for the ETH exit path:

```rust
// engine-tests/src/tests/erc20_connector.rs  lines 771-775
#[cfg(feature = "error_refund")]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE);
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [6](#0-5) 

### Likelihood Explanation

**Medium-High.** The `ft_transfer` promise fails whenever the destination NEAR account is not registered as a storage depositor with the NEP-141 contract — a common situation for new accounts. Any unprivileged EVM user who calls `exitToNear` targeting an unregistered NEAR account (their own or a third party's) will trigger the failure. The user controls the recipient account ID directly through the precompile input, making this trivially reachable. Because `error_refund` is not a default feature, any production WASM built without explicitly enabling it is vulnerable.

### Recommendation

1. Enable `error_refund` unconditionally in the production build (move it into the `default` or `contract` feature set), or
2. Restructure the precompile so that a failure-handling callback is always attached regardless of feature flags, mirroring the existing `refund_on_error` logic in `engine/src/engine.rs`. [7](#0-6) 

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature (the default).
2. Bridge any NEP-141 token to Aurora, receiving ERC-20 tokens.
3. Call the ERC-20 contract's `withdrawToNear` (or equivalent burn+exit function) specifying a NEAR account that is **not** registered with the NEP-141 contract as the recipient.
4. The ERC-20 tokens are burned in the EVM. The `ft_transfer` NEAR promise is scheduled with no callback.
5. The `ft_transfer` fails (unregistered account). No callback fires. No `refund_on_error` is called.
6. Observe: the user's ERC-20 balance is zero; the NEP-141 balance of the recipient is zero; the NEP-141 balance held by Aurora is unchanged. The tokens are permanently destroyed.

The same steps apply to the ETH exit path (`withdrawEthToNear`), where the ETH ends up locked in the `exitToNear` precompile address (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`). [8](#0-7)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
