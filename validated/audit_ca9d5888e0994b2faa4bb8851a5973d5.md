Audit Report

## Title
Cycles Permanently Burned Before Canister-Status Guard in `enqueue`, Enabling Targeted Cycle Drain on Stopping/Stopped Canisters ‚Äî (`rs/messaging/src/scheduling/valid_set_rule.rs`)

## Summary

In `ValidSetRuleImpl::enqueue`, `charge_ingress_induction_cost` is called and permanently burns cycles from the receiving canister before the canister-status guard is checked. When the canister is `Stopping` or `Stopped`, the function returns `Err` with no refund, meaning cycles are consumed for messages that are never inducted. Any unprivileged user can exploit this to drain the cycle balance of any canister passing through `Stopping` or `Stopped` state, ultimately causing the canister to be frozen or deleted.

## Finding Description

In `rs/messaging/src/scheduling/valid_set_rule.rs`, the `enqueue` function (lines 305‚Äì332) charges cycles before checking canister status:

1. **Lines 305‚Äì315**: `charge_ingress_induction_cost` is called. For a canister without a paused execution, this calls `consume_with_threshold::<IngressInduction>` ‚Üí `system_state.consume_cycles(cycles)`, which directly subtracts from `cycles_balance` (line 2057 of `rs/replicated_state/src/canister_state/system_state.rs`). The doc comment on `charge_ingress_induction_cost` is explicit: *"Note that this method reports the cycles withdrawn as consumed (i.e. burnt)."*

2. **Lines 317‚Äì332**: Only after the charge succeeds does the code check whether the canister is `Running`. If it is `Stopping` or `Stopped`, it returns `Err(CanisterStopping)` / `Err(CanisterStopped)` with **no refund call** (`refund_cycles` is never invoked on these paths).

The payer is always the receiving canister (line 584 of `rs/cycles_account_manager/src/cycles_account_manager.rs`: *"A message to a canister is always paid for by the receiving canister."*). The `refund_cycles` mechanism exists in `SystemState` (confirmed by its use in tests at lines 877‚Äì878 of `rs/replicated_state/src/canister_state/tests.rs`) but is not called on the early-return error paths.

The existing tests `ingress_to_stopping_canister_is_rejected` and `ingress_to_stopped_canister_is_rejected` (lines 567‚Äì624 of `rs/messaging/src/scheduling/valid_set_rule/test.rs`) only assert the returned error variant; they never assert that the canister's cycle balance is unchanged after rejection, so the drain goes undetected.

## Impact Explanation

This is a High-severity vulnerability matching the allowed impact: *"Application/platform-level DoS... not based on raw volumetric DDoS"* and *"Significant... security impact with concrete user or protocol harm."* An attacker can drain the entire cycle balance of any canister in `Stopping` or `Stopped` state by submitting repeated ingress messages. Once cycles are exhausted, the canister is frozen (unable to process any messages) or deleted by the system, constituting a targeted canister-level DoS. On application subnets, induction cost scales with message size (base cost plus per-byte cost), so large-payload messages amplify the drain rate significantly. The `Stopped` state persists indefinitely, giving an unbounded attack window.

## Likelihood Explanation

- Any unprivileged user can submit ingress messages to any canister ‚Äî no special role is required.
- The `Stopping` and `Stopped` states are publicly observable via `canister_status` queries, allowing precise attack timing.
- The `Stopped` state persists until the canister is explicitly restarted, providing an unbounded window.
- No per-target-canister rate limiting on ingress submission exists that would prevent bulk submission.
- The attack is cheap for the attacker (ingress submission has no cost to the sender) and expensive for the victim (each message burns `induction_cost` cycles permanently).

## Recommendation

Reorder the checks in `enqueue` so the canister-status guard runs **before** `charge_ingress_induction_cost`. The corrected logic in `rs/messaging/src/scheduling/valid_set_rule.rs` should be:

```
1. Check canister status ‚Üí return Err early if Stopping/Stopped (no cycles touched)
2. Charge cycles via charge_ingress_induction_cost
3. Push ingress to queue
```

Alternatively, if the charge must precede the status check for some reason, add an explicit `system_state.refund_cycles(...)` call on every early-return error path that follows a successful charge. Additionally, add balance-preservation assertions to the existing `ingress_to_stopping_canister_is_rejected` and `ingress_to_stopped_canister_is_rejected` tests to prevent regression.

## Proof of Concept

State-machine test outline:

```rust
// 1. Create an application-subnet canister with known balance B.
// 2. Transition it to Stopping state (stop_canister via management canister).
// 3. Submit N distinct ingress messages (different nonces, max payload size) to the canister.
// 4. Run induct_messages for the batch.
// 5. Assert: ingress_queue_size == 0 (no messages inducted).
// 6. Assert: canister balance == B - N * induction_cost  (cycles drained).
// The delta B - balance equals N * induction_cost even though zero work was done.
// Repeat until balance reaches freeze threshold ‚Üí canister is frozen/deleted.
```

The same test with a `Running` canister would show balance `B - N * induction_cost` AND `ingress_queue_size == N`, confirming the charge is correct only when messages are actually inducted. For `Stopping`/`Stopped`, the charge without induction is the exploitable bug.
