Audit Report

## Title
Minting Rate Limiter Bypass via CMC Balance Inflation in `ensure_balance` - (File: `rs/nns/cmc/src/main.rs`)

## Summary
`ensure_balance` computes `cycles_to_mint` as the saturating difference between the requested `cycles` and the CMC's live `canister_balance128()`. When the CMC's balance has been pre-loaded by an attacker via `deposit_cycles` on the management canister, `cycles_to_mint` collapses to `Cycles(0)`, and the rate limiter is checked against zero — trivially passing. This allows the rate limiter to be silently bypassed for all three minting paths (`notify_top_up`, `notify_create_canister`, `notify_mint_cycles`), and causes `total_cycles_minted` to be permanently understated.

## Finding Description
In `rs/nns/cmc/src/main.rs` at lines 2306–2325, `ensure_balance` reads the live canister balance and computes the mint delta:

```rust
let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
let cycles_to_mint = cycles - current_balance;   // saturating sub
```

`Cycles::Sub` is saturating (`rs/types/cycles/src/cycles.rs`, lines 133–139), so when `current_balance >= cycles`, `cycles_to_mint = Cycles(0)`.

The limiter at `rs/nns/cmc/src/limiter.rs` line 43 checks:
```rust
if count + cycles_to_mint > limit { ... }
```
With `cycles_to_mint = 0`, this reduces to `count > limit`, which passes for any fresh or partially-used limiter window. The limiter records 0 cycles minted, and `state.total_cycles_minted += 0` leaves the accounting metric unchanged.

Any canister on any subnet can call `deposit_cycles` on the management canister targeting the CMC's principal — this is an unprivileged, standard IC operation requiring no special role. The CMC has no guard against receiving cycles. Once the CMC's balance is inflated above the requested `cycles` amount, every subsequent call to `ensure_balance` for amounts up to that balance bypasses the limiter entirely.

All three public minting paths converge on this function: `deposit_cycles` (internal, called by `notify_top_up` and `notify_create_canister`) at lines 2110–2138, and `do_mint_cycles` (called by `notify_mint_cycles`) at lines 2140–2172, both call `ensure_balance` as the sole rate-limiting gate.

## Impact Explanation
This matches the **High** impact class: "Significant NNS security impact with concrete user or protocol harm." The rate limiter is the primary on-chain mechanism bounding ICP-to-cycles conversion per time window. Bypassing it allows more ICP to be burned and more cycles to be dispensed per period than governance has authorized. The `total_cycles_minted` governance metric is permanently corrupted (understated by the amount dispensed from pre-loaded balance), removing the on-chain audit trail for cycle issuance. The bypass is bounded by the attacker's pre-loaded balance and does not create net-new cycles beyond what was pre-loaded, which constrains the impact below the Critical threshold of exorbitant illegal minting or protocol insolvency.

## Likelihood Explanation
The attacker requires no privileged access, no governance approval, and no special role. Any canister on any subnet can call `deposit_cycles` on the management canister targeting the CMC. The cost is real (the attacker must spend cycles to pre-load the CMC) but bounded and repeatable — once the pre-loaded balance is consumed, the attacker can top it up again. XNet messaging makes the CMC reachable from any subnet. The attack is straightforward to execute and requires only standard IC APIs.

## Recommendation
The rate limiter must be checked against `cycles` (the total amount being dispensed), not `cycles_to_mint` (the freshly minted delta). The corrected logic in `ensure_balance`:

```rust
with_state_mut(|state| {
    limiter_to_use.check_and_add_cycles(state, now, cycles)?;  // full dispensed amount
    state.total_cycles_minted += cycles_to_mint;               // only count newly minted
    Ok::<_, String>(())
})?;
```

This ensures the rate limiter correctly bounds total cycle outflow per period regardless of the CMC's pre-existing balance, while keeping `total_cycles_minted` accurate to actual new issuance.

## Proof of Concept
1. Deploy attacker canister `A` with a large cycles balance on any subnet.
2. Canister `A` calls `deposit_cycles` on the management canister with the CMC's principal as target, sending `X` cycles (where `X` exceeds the per-period rate limit).
3. Send `N` ICP to the CMC's subaccount for a target canister, then call `notify_top_up(block_index, target)`.
4. CMC calls `ensure_balance(cycles, limiter)` where `cycles <= X`.
5. `canister_balance128() >= cycles` → `cycles_to_mint = Cycles(0)`.
6. `check_and_add_cycles(state, now, Cycles(0))` passes unconditionally.
7. `ic0_mint_cycles128(0)` — no ICP burned for new cycles; CMC sends `cycles` from pre-loaded balance.
8. `total_cycles_minted` is unchanged.
9. Repeat steps 3–8 for additional conversions; the limiter window never fills.
10. A deterministic integration test or PocketIC test can confirm this by: (a) pre-crediting the CMC's balance to exceed the limit, (b) calling `notify_top_up` for an amount equal to the limit, (c) asserting the call succeeds and `total_cycles_minted()` returns 0.
