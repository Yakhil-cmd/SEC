### Title
Missing Validation on `erc20_fallback_address` Allows Permanent ERC-20/NEP-141 Token Freeze in Silo Mode - (File: `engine/src/contract_methods/silo/mod.rs`)

### Summary
In Silo mode, `set_silo_params` and `set_erc20_fallback_address` accept `Address::zero()` as the `erc20_fallback_address` without any constraint check. When the Address whitelist is active and a user bridges NEP-141 tokens to a non-whitelisted EVM address, the engine silently redirects the mint to the zero address. Because recovering the underlying NEP-141 tokens requires burning the ERC-20 tokens via `exitToNear`, and no one controls `Address::zero()`, the bridged funds are permanently frozen inside Aurora.

### Finding Description
`set_silo_params` in `engine/src/contract_methods/silo/mod.rs` stores `erc20_fallback_address` directly without validating it is non-zero:

```rust
pub fn set_silo_params<I: IO>(io: &mut I, args: Option<SiloParamsArgs>) {
    let (cost, address) = args.map_or((None, None), |params| {
        (Some(params.fixed_gas), Some(params.erc20_fallback_address))
    });
    set_fixed_gas(io, cost);
    set_erc20_fallback_address(io, address);   // no zero-address guard
}
``` [1](#0-0) 

The same absence of validation exists in the standalone `set_erc20_fallback_address` entry point:

```rust
pub extern "C" fn set_erc20_fallback_address() {
    ...
    let args: Erc20FallbackAddressArgs = io.read_input_borsh().sdk_unwrap();
    silo::set_erc20_fallback_address(&mut io, args.address);  // no zero-address guard
}
``` [2](#0-1) 

In `receive_erc20_tokens`, when the Address whitelist is enabled and the intended recipient is not on it, the engine unconditionally replaces the recipient with the stored fallback address:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;   // could be Address::zero()
}
``` [3](#0-2) 

The ERC-20 mint then proceeds to `Address::zero()` with no error or revert: [4](#0-3) 

`SiloParamsArgs` carries no invariant enforcement; `erc20_fallback_address` is a plain `Address` field with `Default` deriving to `[0u8;20]`: [5](#0-4) 

The `ft_on_transfer` entry point that triggers this path is publicly callable by any NEP-141 contract: [6](#0-5) 

### Impact Explanation
When `erc20_fallback_address = Address::zero()` and the Address whitelist is enabled:

1. A user's NEP-141 tokens are transferred to Aurora's contract (consumed).
2. The corresponding ERC-20 tokens are minted to `Address::zero()`.
3. Recovering NEP-141 tokens requires burning the ERC-20 tokens via the `exitToNear` precompile, which requires controlling the EVM address that holds them.
4. No one controls `Address::zero()`, so the ERC-20 tokens can never be burned.
5. The NEP-141 tokens are permanently locked inside Aurora with no recovery path.

This is a **permanent freeze of user funds** (Critical impact class).

### Likelihood Explanation
The likelihood is **low but non-negligible**. The owner must call `set_silo_params` or `set_erc20_fallback_address` with the zero address while also enabling the Address whitelist. `Address::zero()` is the Rust default for `Address`, making it a plausible fat-finger or copy-paste error during deployment scripting. There is no on-chain guard to prevent it, and the engine accepts the call silently. Once misconfigured, every subsequent `ft_transfer_call` from any user to a non-whitelisted address permanently destroys their tokens. The owner can fix the configuration going forward, but cannot recover tokens already minted to the zero address.

### Recommendation
Add a non-zero address check inside `set_silo_params` and `set_erc20_fallback_address` before persisting the value:

```rust
if address.is_zero() {
    return Err(b"ERR_INVALID_FALLBACK_ADDRESS".into());
}
```

Apply the same guard in `set_erc20_fallback_address` in `engine/src/lib.rs` and in `set_silo_params` in `engine/src/contract_methods/silo/mod.rs`.

### Proof of Concept
1. Owner calls `set_silo_params(Some(SiloParamsArgs { fixed_gas: X, erc20_fallback_address: Address::zero() }))` — accepted without error.
2. Owner enables the Address whitelist via `set_whitelist_status({ kind: WhitelistKind::Address, active: true })`.
3. User calls `ft_transfer_call` on any NEP-141 token registered with Aurora, specifying a non-whitelisted EVM recipient address in the `msg` field.
4. Aurora's `ft_on_transfer` → `receive_erc20_tokens` is invoked.
5. `silo::get_erc20_fallback_address` returns `Address::zero()`; `is_allow_receive_erc20_tokens` returns `false`.
6. `recipient` is overwritten with `Address::zero()`.
7. The ERC-20 `mint` call succeeds, crediting `Address::zero()` with the token amount.
8. The NEP-141 tokens remain in Aurora's custody; the ERC-20 tokens at `Address::zero()` are permanently inaccessible.
9. The user's funds are frozen with no recovery mechanism.

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
