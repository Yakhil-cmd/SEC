### Title
`set_eth_connector_contract_account` Change Results in Permanent Freezing of All Bridged User Funds - (File: `engine/src/contract_methods/connector.rs`)

### Summary

`set_eth_connector_contract_account` allows the engine owner to replace the ETH connector contract account ID with no check for existing deposits and no migration of state. All previously bridged ETH and NEP-141 token balances held by the old connector become permanently inaccessible through the engine's interfaces after the update.

### Finding Description

The function `set_eth_connector_contract_account` in `engine/src/contract_methods/connector.rs` unconditionally overwrites the stored connector account ID pointer:

```rust
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}

### Impact Explanation

**Critical â€” Permanent freezing of funds.**

The ETH connector is an external NEAR contract that holds the actual NEP-141 token balances representing all bridged ETH on Aurora. When the owner calls `set_eth_connector_contract_account` with a new account (e.g., to upgrade the connector):

1. The engine's pointer is atomically replaced with the new connector account.
2. All future `withdraw`, `ExitToNear`, `ft_transfer`, and `ft_transfer_call` calls are routed to the new connector, which holds zero balances.
3. The old connector still holds all previously deposited ETH, but the engine will never call it again.
4. Users whose EVM balances were minted against the old connector cannot exit their ETH to NEAR or Ethereum â€” the `withdraw` call reaches the new connector which has no funds to release.
5. There is no recovery path within the engine's interface; the funds in the old connector are permanently stranded.

### Likelihood Explanation

**Medium.** The function is owner-only, so it requires the engine owner to call it. However, this is a legitimate administrative action â€” the owner may wish to upgrade the connector contract (e.g., to add new features or fix bugs) and call `set_eth_connector_contract_account` pointing to a freshly deployed connector, not realizing that all existing deposits are tied to the old account. There is no warning, no balance check, and no migration step enforced by the code. The scenario is realistic during any connector upgrade operation.

### Recommendation

- Before allowing the update, check that the old connector's total supply (all bridged ETH) is zero, preventing the update while funds are at risk.
- Alternatively, implement a migration step that transfers all NEP-141 balances from the old connector to the new one atomically before updating the pointer.
- At minimum, emit a clear warning or require an explicit acknowledgment argument (e.g., `force: bool`) so the caller consciously accepts the risk.

### Proof of Concept

1. Users bridge ETH into Aurora. The ETH connector (`old_connector.near`) holds `N` wei of NEP-141 ETH. The engine's EVM state records corresponding ERC-20 balances for users.
2. The engine owner deploys an upgraded connector at `new_connector.near` (zero balance) and calls:
   ```
   set_eth_connector_contract_account({
       account: "new_connector.near",
       withdraw_serialize_type: Borsh
   })
   ``` [8](#0-7) 
3. The engine's stored connector pointer is now `new_connector.near`.
4. A user calls `withdraw(recipient_eth_address, amount)`. The engine dispatches `engine_withdraw` to `new_connector.near`, which has no ETH â€” the call fails or returns zero. [9](#0-8) 
5. A user calls the `ExitToNear` precompile. It reads `get_eth_connector_contract_account` â†’ `new_connector.near`, calls `ft_transfer` on it â€” which has no balance â€” and the exit fails. [10](#0-9) 
6. All `N` wei of ETH in `old_connector.near` are permanently inaccessible through the engine. Users' EVM balances are non-zero but unexitable â€” a permanent freeze of all bridged ETH.
