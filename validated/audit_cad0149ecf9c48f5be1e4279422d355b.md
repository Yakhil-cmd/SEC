### Title
`exit_to_near_precompile_callback` Blocks Refund When Engine Is Paused, Causing Permanent Fund Loss - (File: `engine/src/contract_methods/connector.rs`)

### Summary

The `exit_to_near_precompile_callback` function enforces `require_running` before executing its refund logic. If the Aurora Engine is paused at the time this callback executes — after a failed `ft_transfer` or `near_withdraw` promise — the refund branch is never reached, and the user's ERC-20 tokens (already burned in the EVM) are permanently lost with no recovery path.

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) allows EVM users to bridge ERC-20 tokens from Aurora to NEAR. The flow is:

1. The ERC-20 contract calls the precompile, which burns the tokens in the EVM state.
2. A NEAR promise (`ft_transfer` or `near_withdraw`) is scheduled to deliver the corresponding NEP-141 tokens to the recipient.
3. When the `error_refund` feature is compiled in, a callback (`exit_to_near_precompile_callback`) is attached to handle failures and re-mint the tokens on Aurora.

`require_running` panics if the engine is paused. Because NEAR receipts are asynchronous, the engine can be paused between the original `submit` call (which burned the tokens) and the execution of this callback receipt. When that happens:

- The `ft_transfer` or `near_withdraw` promise has already failed.
- The callback executes but panics at `require_running`.
- NEAR reverts only the callback's own state changes — the ERC-20 burn from the original receipt is **not** reverted.
- The refund branch (`engine::refund_on_error`) is never reached.
- Tokens are permanently lost.

A secondary, unconditional variant exists: when the `error_refund` feature flag is **not** compiled in, `refund` is hardcoded to `None`: [4](#0-3) 

In that case, for standard ERC-20 exits (no wNEAR unwrap), `callback_args` equals the default value and **no callback is attached at all**: [5](#0-4) 

Any failed `ft_transfer` then causes permanent token loss with zero recovery path. The test suite explicitly acknowledges this: [6](#0-5) 

### Impact Explanation

**Permanent freezing of funds.** ERC-20 tokens are burned in the EVM during the original transaction. If the NEAR-side promise fails and the refund callback cannot execute (due to `require_running` blocking it), those tokens are destroyed on Aurora and never delivered on NEAR. There is no admin recovery function for this state — the EVM balance is gone and the NEP-141 transfer never occurred.

### Likelihood Explanation

**Moderate.** The engine is designed to be pausable for upgrades and emergencies. Any pause window that overlaps with a pending `exit_to_near_precompile_callback` receipt — which can occur if `ft_transfer` fails due to an unregistered recipient, a paused NEP-141 contract, or insufficient storage — triggers the loss. The scenario requires no attacker action beyond initiating a normal exit; the engine pause is a routine admin operation. If `error_refund` is not compiled into the production binary, the likelihood becomes **High** because any failed `ft_transfer` causes loss unconditionally.

### Recommendation

1. **Remove `require_running` from `exit_to_near_precompile_callback`.** Refund callbacks are safety-critical and must execute regardless of the engine's pause state. The callback is already protected by `env.assert_private_call()`, which ensures only the engine itself can invoke it.

2. **Ensure `error_refund` is always enabled in production builds.** The feature should be a default feature in `engine-precompiles/Cargo.toml` and `engine/Cargo.toml`, not an opt-in flag, so that the refund path is never absent.

3. **For the wNEAR unwrap path**, verify that the callback also handles the failure case when `near_withdraw` fails and `error_refund` is disabled — currently the `else { None }` branch silently drops the failure with no refund. [7](#0-6) 

### Proof of Concept

1. User holds ERC-20 tokens on Aurora (bridged from NEP-141 `token.near`).
2. User calls the ERC-20's `withdraw` function, which internally calls the `ExitToNear` precompile. Tokens are burned in the EVM; a `ft_transfer` promise to `token.near` is scheduled.
3. The `ft_transfer` fails because the recipient NEAR account has not paid the NEP-141 storage deposit.
4. The Aurora Engine is paused by an admin for a routine upgrade (a normal, non-malicious operation).
5. The `exit_to_near_precompile_callback` receipt executes. `require_running` returns an error because the engine is paused. The callback panics; NEAR reverts only the callback's state changes.
6. The ERC-20 burn (step 2) is not reverted. The user's tokens are permanently destroyed on Aurora and never received on NEAR. [8](#0-7) [9](#0-8)

#[named]
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
