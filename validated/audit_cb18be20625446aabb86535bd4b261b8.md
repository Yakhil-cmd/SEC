Audit Report

## Title
ckETH Minter Burns User Tokens Without Reimbursement on Persistent InsufficientTransactionFee - (`File: rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary
When `create_transactions_batch` encounters `CreateTransactionError::InsufficientTransactionFee` for a `WithdrawalRequest::CkEth` request, the error handler at `rs/ethereum/cketh/minter/src/withdraw.rs` lines 281â€“291 calls only `reschedule_withdrawal_request`, re-queuing the request indefinitely with no reimbursement path. Because the ckETH burn is irreversible and occurs before the request is enqueued, any user whose withdrawal amount is persistently below the real-time gas fee suffers a permanent, unrecoverable loss of funds.

## Finding Description
The withdrawal flow is:

1. `withdraw_eth` in `rs/ethereum/cketh/minter/src/main.rs` (lines 291â€“336) checks `amount >= cketh_minimum_withdrawal_amount`, then calls `burn_from` on the ckETH ledger. The burn is irreversible and committed on-chain before the function returns.
2. On burn success, an `EventType::AcceptedEthWithdrawalRequest` is emitted and the request enters `pending_withdrawal_requests`.
3. A timer calls `process_retrieve_eth_requests` â†’ `create_transactions_batch`. Inside, `create_transaction` (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, lines 1122â€“1134) computes `max_transaction_fee` from the current gas estimate and returns `Err(CreateTransactionError::InsufficientTransactionFee)` when `withdrawal_amount < max_transaction_fee`.
4. The error arm at `rs/ethereum/cketh/minter/src/withdraw.rs` lines 281â€“291 logs the condition and calls `mutate_state(|s| s.eth_transactions.reschedule_withdrawal_request(request))`. No `ReimbursementRequest` is created, no `EventType::FailedEthWithdrawalRequest` is emitted (this event type does not exist in the codebase â€” confirmed by grep returning zero matches), and `process_reimbursement` is never invoked for this request.

The asymmetry with the ERC20 path is confirmed: `rs/ethereum/cketh/minter/src/main.rs` lines 506â€“531 explicitly schedules `EventType::FailedErc20WithdrawalRequest` when the ckERC20 burn fails, triggering `process_reimbursement` to mint back the burned ckETH fee. No equivalent path exists for the ckETH `InsufficientTransactionFee` case.

The `cketh_minimum_withdrawal_amount` guard (`rs/ethereum/cketh/minter/src/lifecycle/init.rs` line 92) is a static deployment-time value. It is not dynamically adjusted against the live gas fee estimate, so a gas spike after acceptance can cause the condition to fire for legitimately accepted requests.

## Impact Explanation
A user's ckETH is burned on the IC ledger (permanent balance reduction), no ETH is ever sent on Ethereum, and no ckETH is ever minted back. The loss is proportional to the user's withdrawal amount and is permanent. This constitutes a concrete, per-user permanent loss of chain-key ledger assets, matching the **High** impact class: "Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm." Multiple users can be affected simultaneously during a gas spike event, and the minter's internal `eth_balance` accounting becomes inconsistent because no transaction is ever finalized.

## Likelihood Explanation
Any unprivileged IC principal can call `withdraw_eth` with an amount at or near `cketh_minimum_withdrawal_amount`. No privileged access is required. The trigger condition â€” Ethereum gas fees exceeding the withdrawal amount â€” is a well-known, recurring phenomenon (NFT mints, network congestion events). The minimum withdrawal amount is a static buffer that does not track real-time gas. The minter's `lazy_refresh_gas_fee_estimate` fetches live values from Ethereum JSON-RPC providers and can legitimately return values exceeding the minimum. Likelihood is **Medium**: requires a gas spike, but Ethereum gas spikes are historically common and the condition is not hypothetical.

## Recommendation
1. **Immediate**: In the `Err(CreateTransactionError::InsufficientTransactionFee)` arm of `create_transactions_batch` (`rs/ethereum/cketh/minter/src/withdraw.rs` lines 281â€“291), add a retry counter or maximum queue age to the `EthWithdrawalRequest`. Once the threshold is exceeded, emit a new `EventType::FailedEthWithdrawalRequest` event carrying a `ReimbursementRequest` and remove the request from the queue, mirroring the ERC20 path at `rs/ethereum/cketh/minter/src/main.rs` lines 516â€“530.
2. **Structural**: Implement `process_reimbursement` handling for `ReimbursementIndex::CkEth` triggered by the new event type, so `process_reimbursement` mints back the burned ckETH minus any applicable ledger fee.
3. **Preventive**: At `withdraw_eth` acceptance time, compare `amount` against the current gas fee estimate (not just the static minimum) and reject the request before burning if the amount cannot cover fees, preventing the burn from occurring in the first place.
