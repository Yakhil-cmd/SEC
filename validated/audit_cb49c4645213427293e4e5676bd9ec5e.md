### Title
Permanent Loss of ERC-20 / ETH Funds When `ft_transfer` Fails in `exit_to_near` Without `error_refund` Feature — (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `exit_to_near` precompile burns ERC-20 tokens (or transfers ETH to the precompile address) on the Aurora EVM side before dispatching an asynchronous `ft_transfer` / `ft_transfer_call` promise to the NEAR NEP-141 contract. When the `error_refund` compile-time feature is **not** enabled, no failure-handling callback is attached to that promise. If the NEAR-side transfer fails for any reason, the burned tokens are permanently unrecoverable.

---

### Finding Description

The `exit_to_near` precompile in `engine-precompiles/src/native.rs` constructs a `ExitToNearPrecompileCallbackArgs` struct whose `refund` field is unconditionally set to `None` when the `error_refund` Cargo feature is absent: [1](#0-0) 

The promise is then created as a bare `PromiseArgs::Create` (no callback) whenever `callback_args` equals its default value — which happens whenever `refund` is `None` **and** `transfer_near` is `None` (i.e., every non-wNEAR ERC-20 exit and every ETH exit when the feature is off): [2](#0-1) 

Even in the wNEAR-unwrap path where `transfer_near` is `Some` (so a callback *is* attached), the callback handler in `engine/src/contract_methods/connector.rs` only executes a refund when `args.refund` is `Some`: [3](#0-2) 

Because `args.refund` is always `None` without the feature, the `else { None }` branch is taken and no re-mint or ETH credit occurs.

The `refund_on_error` function that *would* re-mint burned ERC-20 tokens or credit ETH back to the sender is therefore dead code in a build without `error_refund`: [4](#0-3) 

The project's own integration tests explicitly document and confirm this behaviour: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing / loss of user funds.**

The ERC-20 `_burn` (or ETH debit) on the Aurora EVM side is committed atomically within the `submit` call. The subsequent `ft_transfer` promise executes asynchronously in a separate NEAR receipt. If that receipt fails:

- The ERC-20 tokens are already gone from the user's EVM balance.
- The NEP-141 balance on the NEAR side is never credited.
- Without a refund callback, no re-mint is scheduled.
- The tokens are permanently destroyed with no recovery path for the user.

---

### Likelihood Explanation

**Medium.**

`ft_transfer` can fail for multiple realistic, user-reachable reasons that are independent of the EVM-side transaction:

1. The recipient NEAR account is not registered with the NEP-141 contract (storage deposit missing).
2. The NEP-141 contract is paused or access-controlled.
3. The Aurora engine's NEP-141 balance is insufficient (e.g., drained by a concurrent exit).
4. The recipient account ID is syntactically valid on the EVM side but rejected by the NEP-141 contract.

Any of these conditions can arise after the EVM burn has already been committed, exactly as described in the external report for moving/time-dependent values.

---

### Recommendation

1. **Enable `error_refund` unconditionally** in the production WASM build, or promote the refund logic out of the feature flag so it is always compiled in.
2. Ensure the callback is always attached to the `ft_transfer` / `ft_transfer_call` promise regardless of feature flags, so that a failed NEAR-side transfer always triggers a re-mint of the burned ERC-20 tokens (or a credit of the ETH) back to the original sender.
3. For the wNEAR-unwrap path, verify that the callback's `refund` branch is reachable even when `transfer_near` is `Some`.

---

### Proof of Concept

**Entry path (unprivileged EVM user):**

1. User holds a bridged ERC-20 (NEP-141 mirror) on Aurora.
2. User calls `withdrawToNear(recipient, amount)` on the ERC-20 contract, which internally calls the `exit_to_near` precompile.
3. The precompile burns `amount` tokens from the user's EVM balance and emits a promise log targeting the NEP-141 contract's `ft_transfer` method.
4. `filter_promises_from_logs` in `engine/src/engine.rs` schedules the promise with **no callback** (because `error_refund` is off and `transfer_near` is `None`). [7](#0-6) 

5. The `ft_transfer` receipt executes on NEAR and fails (e.g., recipient not storage-registered).
6. No callback fires. The burned ERC-20 tokens are permanently lost.

**Confirmed by test `test_exit_to_near_refund`** (workspace integration test): [8](#0-7) 

The test explicitly asserts that without `error_refund`, the user's ERC-20 balance is reduced by `FT_EXIT_AMOUNT` even though the NEAR-side transfer failed — confirming permanent fund loss.

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
