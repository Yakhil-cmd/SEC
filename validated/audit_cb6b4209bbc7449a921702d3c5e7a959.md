Audit Report

## Title
Unbounded Allowance Table Growth via Non-Expiring Approvals Enables Ledger DoS - (`rs/ledger_suite/common/ledger_core/src/approvals.rs`)

## Summary
The `AllowanceTable` in the IC ledger suite stores one entry per `(account, spender)` pair in an unbounded `BTreeMap` with no global or per-account cap. The `prune` function exclusively iterates over the `expiration_queue`, which only contains entries created with a non-`None` `expires_at`. Approvals created with `expires_at: None` are never inserted into the `expiration_queue` and are therefore never pruned, allowing an unprivileged attacker to grow the allowance table without bound by submitting many approvals from distinct accounts, eventually exhausting canister memory and halting all ledger operations.

## Finding Description
In `rs/ledger_suite/common/ledger_core/src/approvals.rs`, `HeapAllowancesData` stores allowances in an unbounded `BTreeMap<(AccountId, AccountId), Allowance<Tokens>>`:

```rust
allowances: BTreeMap<(AccountId, AccountId), Allowance<Tokens>>,
expiration_queue: BTreeSet<(TimeStamp, (AccountId, AccountId))>,
arrival_queue: BTreeSet<(TimeStamp, (AccountId, AccountId))>,
```

The `approve` function (L265–267) only inserts into `expiration_queue` when `expires_at` is `Some`:

```rust
if let Some(expires_at) = expires_at {
    table.allowances_data.insert_expiry(expires_at, key.clone());
}
```

The `prune` function (L372–399) exclusively iterates over `first_expiry()`, which reads from `expiration_queue`. Entries with `expires_at: None` are invisible to it and are never removed. The `arrival_queue` field exists in the struct but the `AllowancesData` trait has no `insert_arrival` method — it is never populated, so arrival-order pruning is not implemented.

In `rs/ledger_suite/common/ledger_canister_core/src/ledger.rs` (L211, L231), `apply_transaction` calls `prune(now, 100)` on every transaction, but this only removes up to 100 expired entries per call and has no effect on non-expiring allowances. Neither `apply_transaction` nor `approve` enforces any cap on the total number of stored allowances.

The exploit path is:
1. Attacker generates N keypairs, derives N IC principals.
2. Each account is funded with enough tokens to pay the approval fee (e.g., 0.0002 ICP per account).
3. Each account calls `icrc2_approve` with `spender = attacker_principal`, `amount = 1`, `expires_at = None`.
4. Each call inserts a permanent `(P_i, P_attacker) -> Allowance { amount: 1, expires_at: None }` entry.
5. `prune(now, 100)` finds nothing to prune (empty `expiration_queue`).
6. After sufficient calls, heap memory (ICP ledger) or stable memory (ICRC-1 ledgers) is exhausted; subsequent update calls trap, halting all ledger operations.

## Impact Explanation
This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

The ICP ledger, ckBTC, ckETH, and ckERC20 ledgers are all explicitly in-scope financial integrations. A successful attack halts all ledger operations — transfers, approvals, burns — for all users, not just the attacker. This is not a raw volumetric DDoS; it is a targeted state-exhaustion attack exploiting a specific missing invariant (no cap, no pruning of non-expiring entries).

## Likelihood Explanation
The attack is reachable by any unprivileged ingress sender. IC principals are derived from public keys and can be generated offline in unlimited quantities. The only cost is the transfer fee per approval (0.0001 ICP on the ICP ledger). Each `HeapAllowancesData` entry occupies roughly 150–250 bytes (two `AccountIdentifier` values plus `Allowance` struct plus `BTreeMap` node overhead). Filling a meaningful fraction of the ICP ledger's heap requires on the order of millions of entries, costing hundreds to low thousands of ICP — a modest cost relative to the value secured by the ledger. The attack is gradual, requires no privileged role, and no governance majority.

## Recommendation
1. **Enforce a global cap** on `len_allowances()` in `AllowanceTable::approve`, rejecting new approvals when the cap is reached.
2. **Implement arrival-order pruning**: the `arrival_queue` field already exists in `HeapAllowancesData` but is never populated. Add an `insert_arrival` method to the `AllowancesData` trait, populate it in `approve`, and extend `prune` to evict the oldest non-expiring entries when the table exceeds a threshold.
3. **Require `expires_at`** for all new approvals, or charge a higher fee for non-expiring approvals to make the attack economically infeasible.
4. **Add a per-account cap** on the number of active allowances to limit the blast radius per principal.

## Proof of Concept
A deterministic integration test using PocketIC or `StateMachine`:

1. Initialize the ICP ledger with a fixed transfer fee.
2. In a loop, create N test principals, fund each with `2 * fee` tokens, and call `icrc2_approve` with `expires_at: None` from each.
3. After each batch, assert `ledger.approvals().get_num_approvals()` grows monotonically.
4. Verify that calling `prune(now, 100)` returns 0 (nothing pruned).
5. At a sufficiently large N, assert that the canister traps or returns an out-of-memory error on the next update call.

The existing test `test_allowance_listing_take` in `rs/ledger_suite/icp/ledger/tests/tests.rs` already demonstrates creating 501 non-expiring allowances from a single account without any rejection, confirming the absence of any cap enforcement. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    /// Prunes allowances that are expired, removes at most `limit` allowances.
    pub fn prune(&mut self, now: TimeStamp, limit: usize) -> usize {
        self.with_postconditions_check(|table| {
            let mut pruned = 0;
            for _ in 0..limit {
                match table.allowances_data.first_expiry() {
                    Some((ts, _key)) => {
                        if ts > now {
                            return pruned;
                        }
                    }
                    None => {
                        return pruned;
                    }
                }
                if let Some((_, (account, spender))) = table.allowances_data.pop_first_expiry() {
                    let key = (account, spender);
                    if let Some(allowance) = table.allowances_data.get_allowance(&key)
                        && allowance.expires_at.unwrap_or_else(remote_future) <= now
                    {
                        table.allowances_data.remove_allowance(&key);
                        pruned += 1;
                    }
                }
            }
            pruned
        })
    }

#[test]
fn test_allowance_listing_take() {
    const INITIAL_BALANCE: u64 = 1_000_000_000;
    const MAX_RESULTS: usize = 500;
    const NUM_SPENDERS: usize = MAX_RESULTS + 1;

    let approver = PrincipalId::new_user_test_id(1);

    let mut spenders = vec![];
    for i in 2..2 + NUM_SPENDERS {
        spenders.push(PrincipalId::new_user_test_id(i as u64));
    }
    spenders.sort_by(|first, second| {
        AccountIdentifier::from(first.0).cmp(&AccountIdentifier::from(second.0))
    });
    assert_eq!(spenders.len(), MAX_RESULTS + 1);

    let (env, canister_id) = setup(
        ledger_wasm_allowance_getter(),
        encode_init_args,
        vec![(Account::from(approver.0), INITIAL_BALANCE)],
    );

    let approve_args = ApproveArgs {
        from_subaccount: None,
        spender: Account::from(spenders[0].0),
        amount: Nat::from(1_u64),
        fee: None,
        memo: None,
        expires_at: None,
        expected_allowance: None,
        created_at_time: None,
    };
    let send_approval = |args: &ApproveArgs| {
        let response = env.execute_ingress_as(
            approver,
            canister_id,
            "icrc2_approve",
            Encode!(args).unwrap(),
        );
        assert!(response.is_ok());
    };

    for spender in &spenders {
        let args = ApproveArgs {
            spender: Account::from(spender.0),
            ..approve_args.clone()
