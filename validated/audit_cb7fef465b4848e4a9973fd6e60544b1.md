### Title
Missing Validation That `total_deposited` Covers Both Max Fee and `value` Causes Block-Halting `TopLevelInsufficientBalance` on Malformed L1 Transactions — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

In `execute_l1_transaction_and_notify_result`, the bootloader mints `to_transfer = total_deposited - max_fee_commitment` to the sender before executing the transaction body with `nominal_token_value = value`. There is no check that `to_transfer >= value` (equivalently, `total_deposited >= max_fee_commitment + value`). If an L1 transaction arrives with `to_mint = gas_price * gas_limit` and `value > 0`, the sender receives zero minted tokens, the subsequent value transfer fails with `TopLevelInsufficientBalance`, and this non-`FatalRuntimeError` propagates out of `process_l1_transaction` — halting block processing entirely.

---

### Finding Description

The bootloader comment at line 70 states:

> "The invariant that the user deposited more than the value needed for the transaction must be enforced on L1, **but we double-check it here**"

The actual double-check at lines 133–137 only verifies:

```rust
require_internal!(
    total_deposited >= tx_internal_cost,  // tx_internal_cost = gas_price * gas_limit
    "Deposited amount too low",
    system
)?;
``` [1](#0-0) 

This check does **not** verify `total_deposited >= max_fee_commitment + value`. Consequently, inside `execute_l1_transaction_and_notify_result`:

```rust
let to_transfer = total_deposited
    .checked_sub(max_fee_commitment)
    .ok_or(internal_error!("td-mfc"))?;
// to_transfer may be 0 or < value

if to_transfer > U256::ZERO || Config::SIMULATION {
    mint_base_token(..., &to_transfer, &from, ...)?;  // mints to_transfer to sender
}

// Then immediately:
run_single_interaction(..., &from, &to, ..., &value, ...)  // tries to transfer `value` from sender
``` [2](#0-1) 

If `to_transfer < value` and the sender has no pre-existing L2 balance, `perform_transfer_if_required` is called with `caller_ee_type = NoEE` (top-level call) and returns `Err(TopLevelInsufficientBalance)`:

```rust
ExecutionEnvironmentType::NoEE => Err(interface_error!(
    BootloaderInterfaceError::TopLevelInsufficientBalance
))
``` [3](#0-2) 

This error propagates back to `process_l1_transaction`. Because `TopLevelInsufficientBalance` is an interface error — not a `FatalRuntimeError` — it falls into the fatal branch:

```rust
_ => {
    system.finish_global_frame(Some(&rollback_handle))?;
    return Err(e);  // halts block processing
}
``` [4](#0-3) 

The analog to the GovernorAlpha bug is exact: in GovernorAlpha, ETH was locked in the Timelock because `execute()` did not forward it from the Timelock's balance to the target. Here, base tokens are locked in the treasury because the bootloader mints an insufficient amount to the sender and then attempts to forward `value` from the sender — who lacks the balance — causing the transfer to fail and the tokens to remain in the treasury, while also halting block processing.

---

### Impact Explanation

- **Funds locked in treasury**: The deposited tokens (`to_transfer`) are minted inside the execution frame. When the value transfer fails and the frame reverts, the minted tokens are rolled back to the treasury. The user's deposit is effectively locked (refunded only via the post-execution refund path, which itself may not execute if the error halts the block).
- **Block processing halted**: `TopLevelInsufficientBalance` is not caught as a recoverable error for L1 transactions. A single malformed L1 transaction with `to_mint = gas_price * gas_limit` and `value > 0` can halt the entire block.

**Impact: 4** — block-halting DoS on L1 transaction processing; deposited funds temporarily locked in treasury.

---

### Likelihood Explanation

The L1 bridge is supposed to enforce `to_mint >= max_fee + value`. However:
1. The bootloader's own comment says it intends to double-check this invariant, but the check is incomplete.
2. Any bug or inconsistency in the L1 bridge validation (e.g., fee calculation mismatch, rounding, or a future bridge upgrade) can produce a transaction that passes the L1 bridge but fails the bootloader's incomplete check.
3. The attacker entry path is an unprivileged L1 transaction submission — no privileged role required.

**Likelihood: 3** — depends on L1 bridge enforcement, but the bootloader's defense-in-depth is broken.

---

### Recommendation

Add a check in `process_l1_transaction` (or `execute_l1_transaction_and_notify_result`) that `total_deposited >= max_fee_commitment + value` before proceeding:

```rust
let value = transaction.value.read();
let total_deposited = transaction.reserved[0].read();
require_internal!(
    total_deposited >= tx_internal_cost,
    "Deposited amount too low",
    system
)?;
require_internal!(
    total_deposited.saturating_sub(tx_internal_cost) >= value,
    "Deposited amount does not cover value",
    system
)?;
```

This closes the gap between the stated intent ("we double-check it here") and the actual validation performed. [1](#0-0) 

---

### Proof of Concept

1. Submit an L1 transaction with:
   - `gas_price = 1000`, `gas_limit = 100_000` → `max_fee = 100_000_000`
   - `to_mint = 100_000_000` (exactly covers max fee, nothing for value)
   - `value = 1_000_000` (1M wei to send to recipient)
   - Sender has zero L2 balance

2. In `process_l1_transaction`, the check `total_deposited >= tx_internal_cost` passes (`100_000_000 >= 100_000_000`).

3. In `execute_l1_transaction_and_notify_result`:
   - `to_transfer = 100_000_000 - 100_000_000 = 0`
   - No minting occurs
   - `run_single_interaction(caller=sender, callee=recipient, value=1_000_000)` is called

4. `perform_transfer_if_required` attempts to transfer `1_000_000` from sender (balance = 0).

5. Returns `Err(TopLevelInsufficientBalance)`.

6. Error propagates through `execute_l1_transaction_and_notify_result` → `process_l1_transaction` → block processing halts. [5](#0-4) [6](#0-5)

        .checked_mul(U256::from(transaction.gas_limit.read()))
        .ok_or(internal_error!("gp*gl"))?;
    let total_deposited = transaction.reserved[0].read();
    let to_transfer = total_deposited
        .checked_sub(max_fee_commitment)
        .ok_or(internal_error!("td-mfc"))?;

    // Transfer value from treasury to sender (the deposit minus max fee).
    // We want to ensure that the simulation of a transaction
    // never underestimates gas/pubdata compared to the actual execution
    // of said transaction.
    // During simulation the gas price is typically set to 0. So we need
    // to be conservative about operations that incur in gas/pubdata depending
    // on the value of the fee. For that reason, we always perform the
    // following transfer on simulation, and avoid compressing the pubdata
    // for the balance changes resulting from it.
    //
    // Mint the value portion of the deposit (total deposited minus max fee)
    // to the sender inside the execution frame, it does not
    // persist if the main tx body reverts.
    //
    // Use with_infinite_ergs so the call cannot fail due to out-of-gas,
    // but native consumption is still tracked against the user's resources.
    if to_transfer > U256::ZERO || Config::SIMULATION {
        resources
            .with_infinite_ergs(|inf_resources| {
                mint_base_token::<S, Config>(
                    system,
                    system_functions,
                    memories.reborrow(),
                    &to_transfer,
                    &from,
                    l1_chain_id,
                    inf_resources,
                    tracer,
                    validator,
                )
            })
            .map_err(|e| match e.root_cause() {
                RootCause::Runtime(RuntimeError::OutOfErgs(_)) => {
                    system_log!(
                        system,
                        "Out of ergs on infinite ergs: inner error was {e:?}"
                    );
                    BootloaderSubsystemError::LeafDefect(internal_error!(
                        "Out of ergs on infinite ergs"
                    ))
                }
                _ => e,
            })?;
    }

    let resources_for_tx = resources.clone();

    // transaction is in managed region, so we can recast it back
    let calldata = transaction.calldata();

    // TODO: add support for deployment transactions,
    // probably unify with execution logic for EOA
    let (reverted, mut returndata) =
        match BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
            system,
            system_functions,
            memories.reborrow(),
            calldata,
            &from,
            &to,
            resources_for_tx,
            &value,
            false,
            tracer,
            validator,
        ) {
            Ok(CompletedExecution {
                resources_returned,
                result,
            }) => {
                let reverted = result.failed();
                // Save the returndata before asset-tracker calls overwrite
                // the runner memory buffer. Use the system allocator (not
                // global) to avoid panics in proving mode.
                let rd = result.return_values().returndata;
                let mut returndata = Vec::with_capacity_in(rd.len(), system.get_allocator());
                returndata.extend_from_slice(rd);
                *resources = resources_returned;
                system.finish_global_frame(reverted.then_some(&rollback_handle))?;
                (reverted, returndata)
            }
            Err(e) => {
                system.finish_global_frame(Some(&rollback_handle))?;
                return Err(e);
            }
        };
