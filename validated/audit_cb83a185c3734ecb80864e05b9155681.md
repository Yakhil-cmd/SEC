### Title
Anyone Can Call `deploy_upgrade()` Once an Upgrade Is Staged — (`engine/src/lib.rs`)

### Summary

The `deploy_upgrade()` function in the Aurora Engine contract is missing a `require_owner_only` authorization check. Once the owner stages an upgrade via `stage_upgrade()` and the timelock block height passes, **any unprivileged NEAR account** can trigger the actual contract deployment. This is the direct analog of the reported `executeMigration()` auth bypass: a privileged state-transition function lacks the caller authorization guard that its companion staging function correctly enforces.

### Finding Description

**Root cause — missing `require_owner_only` in `deploy_upgrade()`**

`stage_upgrade()` correctly gates staging behind `require_owner_only`: [1](#0-0) 

The companion `deploy_upgrade()` function, however, only checks `require_running` and the block-height timelock — it never checks the caller identity: [2](#0-1) 

The `require_running` guard used there: [3](#0-2) 

…only verifies the contract is not paused. It does **not** restrict the caller to the owner.

Compare with the `upgrade()` function (the direct-upgrade path), which correctly calls `require_owner_only`: [4](#0-3) 

**Exploit flow**

1. Owner calls `stage_upgrade()` with new contract bytecode. The code is written to storage under `CODE_KEY`; the unlock block height (`block_height + upgrade_delay_blocks`) is written to `CODE_STAGE_KEY`.
2. The timelock block height passes.
3. Owner discovers a critical bug in the staged bytecode and attempts to cancel by calling `stage_upgrade()` again with safe code (overwriting `CODE_KEY` and `CODE_STAGE_KEY`).
4. An attacker monitors the mempool, sees the owner's cancellation transaction, and front-runs it by calling `deploy_upgrade()` in the same or an earlier block.
5. `deploy_upgrade()` passes both checks (`require_running` and `block_height > index`) and calls `Runtime::self_deploy` with the **buggy** staged code, permanently replacing the Aurora Engine contract. [5](#0-4) 

### Impact Explanation

Aurora Engine is the production EVM contract on NEAR that custodies all bridged ETH and ERC-20 tokens for every user. A successful exploit deploys attacker-timed (potentially buggy or malicious) bytecode as the live contract. Depending on the content of the staged code, this can result in:

- **Permanent freezing of all user funds** — if the deployed code contains a logic error that makes balances unrecoverable.
- **Direct theft of all user funds** — if the deployed code contains a backdoor or accounting error exploitable post-deployment.

Both outcomes fall within the Critical impact tier (permanent fund freeze / direct theft).

### Likelihood Explanation

The precondition is that the owner has staged an upgrade and subsequently wants to cancel it before deployment. This is a realistic operational scenario (e.g., a post-staging security review reveals a bug). The attacker requires no privileged access — only the ability to submit a NEAR transaction and observe the chain. Front-running on NEAR is feasible because transaction ordering within a block is not guaranteed to be FIFO. Likelihood is **Medium**.

### Recommendation

Add `require_owner_only` to `deploy_upgrade()` immediately after `require_running`, mirroring the pattern used in `stage_upgrade()` and `upgrade()`:

```rust
pub extern "C" fn deploy_upgrade() {
    let mut io = Runtime;
    let state = state::get_state(&io).sdk_unwrap();
    require_running(&state)
        .map_err(ContractError::msg)
        .sdk_unwrap();
    // ADD THIS:
    contract_methods::require_owner_only(&state, &io.predecessor_account_id())
        .map_err(ContractError::msg)
        .sdk_unwrap();
    let index = internal_get_upgrade_index();
    if io.block_height() <= index {
        sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
    }
    Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
    io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
}
```

### Proof of Concept

```
1. Alice (owner) calls stage_upgrade(buggy_code).
   → CODE_KEY = buggy_code
   → CODE_STAGE_KEY = current_block + upgrade_delay_blocks

2. upgrade_delay_blocks pass.

3. Alice discovers the bug and submits stage_upgrade(safe_code) to cancel.

4. Attacker observes Alice's pending transaction and submits deploy_upgrade()
   with higher priority (or in the same block before Alice's tx).

5. deploy_upgrade() executes:
   - require_running: passes (contract is running)
   - block_height > CODE_STAGE_KEY: passes (timelock expired)
   - Runtime::self_deploy(CODE_KEY): deploys buggy_code as the live contract

6. Alice's stage_upgrade(safe_code) now executes against the already-replaced
   contract, which may not even have a valid stage_upgrade entrypoint.

Result: Aurora Engine is permanently running buggy_code; all user funds at risk.
```

pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let delay_block_height = env.block_height() + state.upgrade_delay_blocks;

            &delay_block_height.to_le_bytes(),
        );
        Ok(())
    })
}
