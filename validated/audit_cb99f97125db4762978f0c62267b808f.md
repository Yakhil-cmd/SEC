### Title
Truncating Division in `delta_gas` Calculation Causes User to Under-Pay for Native Resource Consumption — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the conversion from native resource consumption back to EVM gas units uses integer division that truncates towards zero. This causes `gas_used` to be understated by up to 1 gas unit per transaction, breaking the invariant that users must fully pay for all native resources consumed.

---

### Finding Description

The vulnerability class from the reference report is **resource accounting bug**: when a total amount is split across multiple parties using integer division, the sum of the parts is less than the whole due to truncation, breaking a conservation invariant.

In ZKsync OS, the analogous root cause is in `compute_gas_refund`:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [1](#0-0) 

The expression `native_used / native_per_gas` performs integer division that truncates towards zero (Rust's default for unsigned integers). This means `delta_gas` is computed as `floor(native_used / native_per_gas) - gas_used` instead of the correct `ceil(native_used / native_per_gas) - gas_used`.

The design intent, documented in `docs/double_resource_accounting.md`, is:

> `deltaGas := (nativeUsed / nativePerGas) - gasUsed`
> If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs. This ensures that gas estimation will include additional gas to cover for native resources. [2](#0-1) 

The purpose of `delta_gas` is to ensure the user pays for native resource consumption. When `native_used % native_per_gas != 0`, the truncation makes `delta_gas` one unit smaller than it should be, so `gas_used` is understated by 1 unit.

Note that `native_per_gas` itself is computed with `div_ceil` (rounding up), which is correct and conservative:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [3](#0-2) 

But the inverse conversion back from native to gas in `delta_gas` uses truncating division, creating an asymmetry.

**Concrete example:**
- `native_per_gas = 7`, `native_used = 1000`, `gas_used = 100`
- Computed: `delta_gas = floor(1000/7) - 100 = 142 - 100 = 42` → `gas_used = 142`
- Correct: `delta_gas = ceil(1000/7) - 100 = 143 - 100 = 43` → `gas_used = 143`
- Result: user is under-charged by 1 gas unit; operator is under-paid by `gas_price * 1`

---

### Impact Explanation

The fee accounting flow is:
1. User pre-pays `gas_price * gas_limit` tokens.
2. After execution, user is refunded `gas_price * (gas_limit - gas_used)`.
3. Operator receives `gas_price * gas_used`.

When `gas_used` is understated by 1 due to truncation, the user receives 1 extra gas unit of refund and the operator receives 1 gas unit less than owed. The conservation invariant `pre_paid = refund + operator_payment` holds in token terms, but the distribution is wrong: the user retains tokens they should not have, at the operator's expense.

The maximum under-payment per transaction is bounded at 1 gas unit (i.e., `gas_price` wei). While small per transaction, this is a real invariant violation that can be triggered on every transaction where `native_used % native_per_gas != 0` and `delta_gas > 0`. [4](#0-3) 

---

### Likelihood Explanation

The condition `native_used % native_per_gas != 0` is satisfied for the vast majority of transactions, since `native_used` is determined by actual computation (RISC-V cycles, pubdata, etc.) and is unlikely to be an exact multiple of `native_per_gas`. The condition `delta_gas > 0` is satisfied whenever native resource consumption is the binding constraint (i.e., the transaction uses more native resources than EVM gas implies). This is the common case for transactions with a low `native_per_gas` ratio, which is the scenario the `delta_gas` mechanism was designed to handle.

Any unprivileged transaction sender can trigger this path by submitting a standard L2 transaction.

---

### Recommendation

Replace the truncating division with ceiling division in the `delta_gas` calculation:

```rust
// Before (truncates towards zero):
(native_used / native_per_gas) as i64 - (gas_used as i64)

// After (rounds up, ensuring user pays for all native consumed):
native_used.div_ceil(native_per_gas) as i64 - (gas_used as i64)
```

This mirrors the conservative rounding already applied when computing `native_per_gas` itself (via `div_ceil`), and ensures the user is charged for the full native resource consumption. [5](#0-4) 

---

### Proof of Concept

Consider a transaction with:
- `gas_price = 7`, `native_price = 1` → `native_per_gas = ceil(7/1) = 7`
- `gas_limit = 200`, `native_limit = 200 * 7 = 1400`
- After execution: EVM `gas_used = 100` (from ergs), `native_used = 1001`

Current behavior:
- `delta_gas = floor(1001 / 7) - 100 = 143 - 100 = 43`
- Final `gas_used = 143`
- User refund: `7 * (200 - 143) = 399`
- Operator payment: `7 * 143 = 1001`

Correct behavior (with `div_ceil`):
- `delta_gas = ceil(1001 / 7) - 100 = 144 - 100 = 44`
- Final `gas_used = 144`
- User refund: `7 * (200 - 144) = 392`
- Operator payment: `7 * 144 = 1008`

The user retains 7 wei (`gas_price * 1`) that should have gone to the operator. The invariant `pre_paid = refund + operator_payment` holds in both cases (`1400 = 399 + 1001` and `1400 = 392 + 1008`), but in the buggy case the user is under-charged for native resource consumption.
