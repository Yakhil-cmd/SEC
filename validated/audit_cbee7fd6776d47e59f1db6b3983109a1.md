Audit Report

## Title
Query and Read-State Routes Lack Per-IP Rate Limiting at Boundary Node - (File: `rs/boundary_node/ic_boundary/src/core.rs`)

## Summary
In `setup_router`, `canister_call_routes` receives optional per-IP and per-subnet rate limiting via `add_ip_rate_limiting` / `add_subnet_rate_limiting`, while `canister_query_routes` and `canister_read_state_routes` receive neither. An unprivileged attacker can flood the query endpoint with syntactically valid but cryptographically invalid requests, forcing expensive `spawn_blocking` signature-verification work on replica nodes with no per-IP throttle at the boundary, potentially saturating the replica's global query concurrency cap and denying service to legitimate callers.

## Finding Description
In `rs/boundary_node/ic_boundary/src/core.rs`, `setup_router` constructs three route groups before merging them under `canister_layers`:

- `canister_query_routes` (lines 856–858): built with no rate-limiting layer.
- `canister_call_routes` (lines 860–879): conditionally wrapped with `add_ip_rate_limiting` and `add_subnet_rate_limiting` when the corresponding CLI flags are set.
- `canister_read_state_routes` (lines 1042–1044): also built with no rate-limiting layer.

All three are merged and then wrapped with `canister_layers` (line 1046–1049), which includes `common_layers`. `common_layers` contains `middleware_bouncer` (line 1005), but this is an `option_layer` — it is only active when a `Bouncer` instance is provided, and it is a coarse aggregate token-bucket, not a fine-grained per-IP governor. The `add_ip_rate_limiting` path (lines 867–868) uses a `GovernorLayer` with `IpKeyExtractor`, which enforces a strict per-source-IP token bucket; this protection is structurally absent from query and read-state routes regardless of CLI configuration.

At the replica, `query()` in `rs/http_endpoints/public/src/query.rs` (lines 277–283) calls `validator.validate_request()` inside `tokio::task::spawn_blocking`. This executes the full signature-verification pipeline: delegation-chain traversal (`validate_delegations`, ingress_validation.rs lines 721–753), per-delegation cryptographic verification, and a final `validate_signature` call (lines 635–703) that performs Ed25519/ECDSA/WebAuthn or canister-sig verification. All of this work occurs before any rate-limiting decision is made for query traffic.

The replica's only self-protection for query is `max_query_concurrent_requests` (http_handler.rs lines 57–59), a global in-flight cap shared across all callers. A single attacker IP can occupy all slots with requests bearing invalid signatures, causing legitimate callers to be shed.

## Impact Explanation
This is an application/platform-level DoS not based on raw volumetric DDoS. The attack is asymmetric: the attacker pays only network egress cost, while each forwarded request consumes a `spawn_blocking` thread slot on the replica for cryptographic work. Once `max_query_concurrent_requests` slots are saturated by attacker-controlled requests, legitimate query traffic receives `429 Too Many Requests`. This matches the High bounty impact class: "Application/platform-level DoS … not based on raw volumetric DDoS."

## Likelihood Explanation
The query endpoint (`/api/v2/canister/{id}/query`, `/api/v3/canister/{id}/query`) is publicly reachable with no prior authentication. Constructing a CBOR-encoded `HttpRequestEnvelope<HttpQueryContent>` with a plausible but invalid signature requires only knowledge of the public IC API specification. The structural asymmetry — call routes rate-limited per-IP, query routes not — is stable across deployments where the optional bouncer is disabled or configured with a permissive threshold. No special privileges, insider access, or victim interaction is required.

## Recommendation
Apply `add_ip_rate_limiting` (and optionally `add_subnet_rate_limiting`) to `canister_query_routes` and `canister_read_state_routes` in `setup_router`, mirroring the treatment of `canister_call_routes`. The rate-limit values may differ from call routes to reflect the higher expected query volume, but a per-IP cap must exist. Additionally, consider incrementing per-IP counters on signature-verification failures so that repeated bad-signature submissions accelerate throttling rather than being invisible to the rate limiter.

## Proof of Concept
1. Construct a valid CBOR-encoded `HttpRequestEnvelope<HttpQueryContent>` targeting any canister, with a non-anonymous sender, a valid-looking `sender_pubkey`, and a random (invalid) `sender_sig`.
2. Send this request in a tight loop (e.g., via `wrk` or a simple async Rust/Python client) to `POST /api/v2/canister/<canister_id>/query` on a boundary node where the bouncer is disabled or permissively configured.
3. Observe that the boundary node forwards each request to a replica with no per-IP check.
4. On the replica, each request spawns a blocking task running `validate_request` → `validate_user_id_and_signature` → `validate_signature` → `verify_basic_sig_by_public_key`, consuming a thread and CPU time before returning `400 Bad Request`.
5. Once `max_query_concurrent_requests` slots are occupied by attacker requests, legitimate users receive `429 Too Many Requests`.
6. A deterministic integration test can be written using `PocketIC` or a local replica: configure `max_query_concurrent_requests = N`, fire `N` concurrent invalid-signature query requests, then assert that a valid query from a second client receives `429`.
