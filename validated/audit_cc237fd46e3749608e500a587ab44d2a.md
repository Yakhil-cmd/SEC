### Title
Unchecked Exit-Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Enables Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` burn a user's ERC-20 tokens before calling the Aurora exit precompile via a low-level assembly `call`. The return value of that `call` is captured in a local variable (`res`) but is **never checked**. If the precompile call fails for any reason, the burn is already committed and the user's tokens are permanently destroyed with no corresponding NEP-141 release on NEAR.

---

### Finding Description

Both `withdrawToNear` and `withdrawToEthereum` in `EvmErc20.sol` follow the same two-step pattern:

1. **Irreversibly burn** the caller's tokens via `_burn`.
2. **Call the exit precompile** via inline assembly, capturing `res` but never branching on it.

`EvmErc20.sol` `withdrawToNear`: [1](#0-0) 

`EvmErc20.sol` `withdrawToEthereum`: [2](#0-1) 

`EvmErc20V2.sol` exhibits the identical pattern: [3](#0-2) 

In all four functions the assembly block is:
```solidity
assembly {
    let res := call(gas(), <precompile_address>, 0, add(input, 32), input_size, 0, 32)
    // res is NEVER checked
}
```

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) can legitimately return a failure (EVM `ExitError`) for several reasons reachable without any privilege: [4](#0-3) 

When the precompile fails, the EVM `call` opcode returns `0` into `res`. Because `res` is never tested, `withdrawToNear`/`withdrawToEthereum` returns **successfully** from the EVM's perspective. The `_burn` state change is already committed; no exit promise is ever scheduled on NEAR.

This is the direct Aurora analog of the external report: a critical token-movement call's return value is silently discarded, allowing a failed operation to be treated as successful.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The user's ERC-20 balance on Aurora is reduced to zero. No NEP-141 tokens are released on NEAR. The tokens are unrecoverable: there is no refund path, no retry mechanism, and no on-chain record that the exit failed. The loss is permanent and proportional to the `amount` passed by the caller.

---

### Likelihood Explanation

**High.** Several unprivileged, realistic triggers exist:

| Trigger | How it reaches the precompile failure path |
|---|---|
| Invalid NEAR account ID as `recipient` | `ExitToNearParams::try_from(input)` returns `ExitError`; precompile reverts |
| Insufficient gas forwarded by the caller | Sub-call runs out of gas; `call` returns `0` |
| Precompile paused by admin | `ExitError` returned; `call` returns `0` |

The first trigger is entirely user-controlled and requires no special conditions. A user who mistypes or programmatically constructs an invalid NEAR account ID will silently lose all bridged tokens.

---

### Recommendation

Check `res` immediately after the assembly `call` and revert if it is zero, so the `_burn` is rolled back:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to all four affected functions across both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. User holds 1000 units of a bridged ERC-20 token on Aurora (backed by NEP-141 on NEAR).
2. User calls `withdrawToNear(invalidRecipient, 1000)` where `invalidRecipient` is not a valid NEAR account ID (e.g., `"!!bad!!"` or an empty byte string).
3. `_burn(msg.sender, 1000)` executes — ERC-20 balance drops to 0, state committed.
4. The assembly `call` invokes the `ExitToNear` precompile. The precompile calls `ExitToNearParams::try_from(input)`, which fails to parse the recipient and returns `ExitError`. [5](#0-4) 
5. The EVM `call` opcode returns `res = 0`. The assembly block exits silently.
6. `withdrawToNear` returns without reverting. The transaction is mined successfully.
7. The user's 1000 ERC-20 tokens are gone. No NEP-141 transfer promise was ever created. Funds are permanently lost.
