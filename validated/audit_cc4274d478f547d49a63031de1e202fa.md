Audit Report

## Title
Blocklist Not Re-Checked at Transaction Processing Time, Enabling Compliance Bypass — (`File: rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary
The ckETH/ckERC20 minter validates the destination Ethereum address against the blocklist only at withdrawal request creation time (`withdraw_eth` / `withdraw_erc20`). When `create_transactions_batch` later dequeues and processes pending requests, no blocklist re-check is performed. Because the blocklist is a static compile-time array updated only via canister upgrades, any withdrawal request accepted before an upgrade that expands the blocklist will still be executed, sending ETH or ERC-20 tokens to the now-blocked address.

## Finding Description
**Creation-time check** — `validate_address_as_destination` is called in both `withdraw_eth` and `withdraw_erc20`, which internally calls `crate::blocklist::is_blocked`:

`rs/ethereum/cketh/minter/src/address.rs` lines 47–56: [1](#0-0) 

`rs/ethereum/cketh/minter/src/main.rs` lines 280–287 (`withdraw_eth`) and 407–414 (`withdraw_erc20`): [2](#0-1) [3](#0-2) 

After passing this check, tokens are burned and the request is pushed into `pending_withdrawal_requests` via `EventType::AcceptedEthWithdrawalRequest`. [4](#0-3) 

**Processing-time — no check** — `create_transactions_batch` dequeues pending requests and calls `create_transaction` with no blocklist validation: [5](#0-4) 

`create_transaction` itself only returns `Err(CreateTransactionError::InsufficientTransactionFee)` or `Ok(...)` — no address validation at all: [6](#0-5) 

**Static blocklist** — `ETH_ADDRESS_BLOCKLIST` is a `const` array compiled into the binary; it is only updated when the minter canister is upgraded via an NNS governance proposal: [7](#0-6) 

**Exploit path:**
1. Attacker observes a public NNS proposal to upgrade the ckETH minter with a new binary that adds address `X` to `ETH_ADDRESS_BLOCKLIST`.
2. Before the proposal executes, attacker calls `withdraw_eth` with `recipient = X`. The address is not yet blocked; tokens are burned; the `EthWithdrawalRequest` enters `pending_withdrawal_requests`.
3. The NNS proposal passes and the minter is upgraded. The new binary now has `X` on the blocklist.
4. The minter timer fires and calls `process_retrieve_eth_requests` → `create_transactions_batch`. The pending request for `X` is dequeued; no blocklist check occurs; `create_transaction` succeeds; the transaction is signed via threshold ECDSA and submitted to Ethereum.
5. ETH is sent to `X`, which `is_address_blocked` now confirms is blocked.

No existing guard intercepts this: the `InsufficientTransactionFee` reschedule path is the only error branch in `create_transactions_batch`, and the reimbursement flow (`process_reimbursement`) is never triggered for this case.

## Impact Explanation
The blocklist is the ckETH/ckERC20 minter's primary sanctions/compliance enforcement mechanism. Bypassing it results in ETH or ERC-20 tokens being sent to a sanctioned Ethereum address in direct violation of the minter's own policy. This constitutes a **significant Chain Fusion / ck-token security impact with concrete protocol harm**, matching the High ($2,000–$10,000) impact class: *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

## Likelihood Explanation
NNS governance proposals are fully public and visible before they pass. The minter processes withdrawal requests asynchronously on a timer, creating a window between request acceptance and execution. An unprivileged user can call `withdraw_eth` or `withdraw_erc20` directly — no special privileges are required. The attacker only needs to monitor the NNS proposal feed and submit a withdrawal to the target address before the upgrade executes. The window is bounded by NNS voting period (days), making the race condition straightforward to win. Likelihood is low in normal operation but non-negligible and reliably exploitable during any active blocklist expansion.

## Recommendation
In `create_transactions_batch` (`rs/ethereum/cketh/minter/src/withdraw.rs`, lines 249–293), add a blocklist check before calling `create_transaction`. If the destination is now blocked, emit a reimbursement event (minting back the burned tokens to the original caller) rather than proceeding with the Ethereum transaction. This mirrors the existing `InsufficientTransactionFee` handling that already reschedules or cancels requests at processing time, and the existing `process_reimbursement` infrastructure can be reused.

## Proof of Concept
1. Deploy a local replica with the ckETH minter canister.
2. Call `withdraw_eth` with `recipient = "0x0330070FD38Ec3bB94F58FA55D40368271E9e54A"` (currently on the blocklist). This will be rejected — confirming the creation-time check works.
3. Temporarily remove that address from `ETH_ADDRESS_BLOCKLIST` in `blocklist.rs`, rebuild, and upgrade the canister. Now call `withdraw_eth` with the same recipient — it succeeds and the request enters `pending_withdrawal_requests`.
4. Upgrade the canister again with the original binary (address back on the blocklist).
5. Advance the timer to trigger `process_retrieve_eth_requests`. Observe that `create_transactions_batch` processes the pending request without error and emits `EventType::CreatedTransaction` for the now-blocked address.
6. Confirm via `is_address_blocked` that the address is blocked, yet the transaction was created and would be sent.

pub fn validate_address_as_destination(address: &str) -> Result<Address, AddressValidationError> {
    let address =
        Address::from_str(address).map_err(|e| AddressValidationError::Invalid { error: e })?;
    if address == Address::ZERO {
        return Err(AddressValidationError::NotSupported(address));
    }
    if crate::blocklist::is_blocked(&address) {
        return Err(AddressValidationError::Blocked(address));
    }
    Ok(address)

fn create_transactions_batch(gas_fee_estimate: GasFeeEstimate) {
    for request in read_state(|s| {
        s.eth_transactions
            .withdrawal_requests_batch(WITHDRAWAL_REQUESTS_BATCH_SIZE)
    }) {
        log!(DEBUG, "[create_transactions_batch]: processing {request:?}",);
        let ethereum_network = read_state(State::ethereum_network);
        let nonce = read_state(|s| s.eth_transactions.next_transaction_nonce());
        let gas_limit = estimate_gas_limit(&request);
        match create_transaction(
            &request,
            nonce,
            gas_fee_estimate.clone(),
            gas_limit,
            ethereum_network,
        ) {
            Ok(transaction) => {
                log!(
                    DEBUG,
                    "[create_transactions_batch]: created transaction {transaction:?}",
                );

                mutate_state(|s| {
                    process_event(
                        s,
                        EventType::CreatedTransaction {
                            withdrawal_id: request.cketh_ledger_burn_index(),
                            transaction,
                        },
                    );
                });

                );
                mutate_state(|s| s.eth_transactions.reschedule_withdrawal_request(request));
            }
        };
    }

pub fn create_transaction(
    withdrawal_request: &WithdrawalRequest,
    nonce: TransactionNonce,
    gas_fee_estimate: GasFeeEstimate,
    gas_limit: GasAmount,
    ethereum_network: EthereumNetwork,
) -> Result<Eip1559TransactionRequest, CreateTransactionError> {
    assert!(
        gas_limit > GasAmount::ZERO,
        "BUG: gas limit should be non-zero"
    );
    match withdrawal_request {
        WithdrawalRequest::CkEth(request) => {
            let transaction_price = gas_fee_estimate.to_price(gas_limit);
            let max_transaction_fee = transaction_price.max_transaction_fee();
            let tx_amount = match request.withdrawal_amount.checked_sub(max_transaction_fee) {
                Some(tx_amount) => tx_amount,
                None => {
                    return Err(CreateTransactionError::InsufficientTransactionFee {
                        cketh_ledger_burn_index: request.ledger_burn_index,
                        allowed_max_transaction_fee: request.withdrawal_amount,
                        actual_max_transaction_fee: max_transaction_fee,
                    });
                }
            };
            Ok(Eip1559TransactionRequest {
                chain_id: ethereum_network.chain_id(),
                nonce,
                max_priority_fee_per_gas: transaction_price.max_priority_fee_per_gas,
                max_fee_per_gas: transaction_price.max_fee_per_gas,
                gas_limit: transaction_price.gas_limit,
                destination: request.destination,
                amount: tx_amount,
                data: Vec::new(),
                access_list: Default::default(),
            })
