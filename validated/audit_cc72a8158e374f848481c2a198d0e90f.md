Audit Report

## Title
SNS Governance Reward Dilution via Instant Neuron Creation Before Proposal — (File: rs/sns/governance/src/governance.rs)

## Summary
`compute_ballots_for_new_proposal` snapshots every neuron's voting power at the moment a proposal is submitted, with no guard against sudden increases in total voting power. Any unprivileged principal can stake a large neuron immediately before a proposal is created, have it auto-vote via following, and receive a share of the reward purse proportional to their freshly-minted stake. The NNS governance canister has an explicit countermeasure (`VotingPowerSnapshots` + spike detection) for exactly this scenario; the SNS canister has none.

## Finding Description
**Root cause — `compute_ballots_for_new_proposal` (SNS)**

`rs/sns/governance/src/governance.rs` lines 5255–5280 iterate over every neuron in `self.proto.neurons` at `now_seconds` and record each neuron's current voting power directly into the proposal's ballot map with no check on neuron age or total-power delta:

```rust
for (k, v) in self.proto.neurons.iter() {
    if v.dissolve_delay_seconds(now_seconds) < min_dissolve_delay_for_vote {
        continue;
    }
    let voting_power = v.voting_power(now_seconds, ...);
    total_power += voting_power as u128;
    electoral_roll.insert(k.clone(), Ballot { vote: Vote::Unspecified, voting_power, ... });
}
```

A neuron created one second before `make_proposal` is called is treated identically to a neuron staked for years.

**Reward distribution uses those ballot weights directly**

`rs/sns/governance/src/governance.rs` lines 5914–5975 sum `ballot.voting_power` for every neuron that voted and divide the reward purse proportionally:

```rust
let reward_shares = i2d(ballot.voting_power);
*neuron_id_to_reward_shares.entry(neuron_id)... += reward_shares;
...
let neuron_reward_e8s = rewards_purse_e8s * (neuron_reward_shares / total_reward_shares);
```

**Missing countermeasure — NNS has it, SNS does not**

`rs/nns/governance/src/governance/voting_power_snapshots.rs` implements `VotingPowerSnapshots` with a rolling window of up to 7 daily snapshots and a 1.5× spike threshold. `rs/nns/governance/src/governance.rs` lines 5506–5512 call `previous_ballots_if_voting_power_spike_detected` inside `compute_ballots_for_standard_proposal` and fall back to the pre-spike snapshot when a spike is detected. A grep across all SNS source files returns zero matches for `VotingPowerSnapshot`, `VOTING_POWER_SNAPSHOTS`, or `snapshot_voting_power` — the mechanism is entirely absent from SNS governance.

**Exploit flow**

1. Attacker acquires a large quantity of SNS tokens.
2. Attacker calls `manage_neuron { Stake / ClaimOrRefresh }` to create a neuron with a large stake and the minimum required dissolve delay, before any proposal is submitted.
3. Attacker sets a following rule so the neuron auto-votes on the relevant proposal type.
4. A proposal is created (by the attacker or any other proposer); the attacker's neuron appears in the ballot map with full voting power.
5. The neuron votes automatically via following.
6. `distribute_rewards` allocates `neuron_reward_e8s` proportional to `ballot.voting_power`, giving the attacker a share equal to `attacker_vp / total_vp`.
7. Attacker begins dissolving the neuron after the reward round, recovering principal minus the lock period cost.

With default SNS parameters (`max_age_bonus_percentage = 25%`, `max_dissolve_delay_bonus_percentage = 100%`), a neuron staked at the minimum dissolve delay with 9× the existing stake captures approximately 90% of the reward purse despite having zero age.

## Impact Explanation
Long-term stakers receive a permanently diluted share of the reward purse for every proposal round in which the attacker participates. The attacker does not need to steal funds — they simply inflate the denominator of the reward calculation. If the reward yield for a single round exceeds the opportunity cost of the minimum dissolve delay lock period (which is plausible for SNS instances with high initial reward rates), the attack is directly profitable and repeatable. This constitutes a significant SNS governance security impact with concrete, measurable harm to legitimate stakers. **Severity: High** — fits "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
- No privileged role is required; any principal can call `manage_neuron` (stake) on any SNS governance canister.
- The attacker does not need to front-run a specific transaction; staking before any future proposal is sufficient.
- The attack is economically rational whenever `reward_per_round × attacker_share > opportunity_cost_of_lock`. For SNS instances with initial reward rates of 10%+ per year and short minimum dissolve delays, this condition is easily met.
- If an SNS sets `max_age_bonus_percentage = 0` (permitted by the parameter schema), the age bonus provides zero protection.
- The attack is repeatable across every reward round.

## Recommendation
Port the NNS voting-power spike detection to SNS governance:

1. Add a `VotingPowerSnapshots` structure (analogous to `rs/nns/governance/src/governance/voting_power_snapshots.rs`) to SNS governance, backed by stable memory, storing up to N daily snapshots of total potential voting power and per-neuron voting power.
2. Add a periodic timer task to SNS governance (analogous to `rs/nns/governance/src/timer_tasks/snapshot_voting_power.rs`) that records a snapshot at regular intervals (e.g., daily).
3. In `compute_ballots_for_new_proposal`, compare the current total voting power against the rolling minimum. If it exceeds `1.5×` the minimum, use the most recent pre-spike snapshot for ballot creation instead of the live neuron state.
4. Store the snapshot timestamp in `ProposalData` for auditability, as NNS does via `previous_ballots_timestamp_seconds`.
