Audit Report

## Title
Global `MAX_PENDING = 100` Withdrawal Queue Exhaustion Enables Sustained DoS on `withdraw_eth` and `withdraw_erc20` - (File: `rs/ethereum/cketh/minter/src/guard/mod.rs`)

## Summary
The ckETH minter enforces a global, canister-wide cap of 100 pending withdrawal requests (`MAX_PENDING = 100`). An unprivileged attacker with 100 IC principals and the minimum ckETH balance can fill this queue, causing every subsequent `withdraw_eth` or `withdraw_erc20` call from any user to trap. Because the queue drains at only 5 requests per timer tick (every 3–6 minutes), the attacker can continuously refill it faster than it drains, sustaining the DoS at the cost of Ethereum gas fees alone.

## Finding Description
In `rs/ethereum/cketh/minter/src/guard/mod.rs` lines 9–10, two global limits are declared:
```rust
pub const MAX_CONCURRENT: usize = 100;
pub const MAX_PENDING: usize = 100;
```
The `Guard::new()` method (lines 50–68) checks the global pending count **before** any per-principal check:
```rust
if PR::pending_requests_count(s) >= MAX_PENDING {
    return Err(GuardError::TooManyPendingRequests);
}
```
`pending_requests_count` resolves via `PendingWithdrawalRequests::pending_requests_count` (lines 32–34) to `state.eth_transactions.withdrawal_requests_len()`, which returns `pending_withdrawal_requests.len()` (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs` lines 929–931). This VecDeque holds every accepted withdrawal request until the minter's timer calls `create_transactions_batch`.

In `withdraw_eth` (`rs/ethereum/cketh/minter/src/main.rs` lines 274–278), a guard failure causes an unconditional trap:
```rust
let _guard = retrieve_withdraw_guard(caller).unwrap_or_else(|e| {
    ic_cdk::trap(format!(
        "Failed retrieving guard for principal {caller}: {e:?}"
    ))
});
```
The same pattern applies to `withdraw_erc20` (lines 401–405). The trap fires **before** any ckETH is burned, so legitimate users are rejected with no funds at risk on their side.

The queue drains via `process_retrieve_eth_requests()` (`rs/ethereum/cketh/minter/src/withdraw.rs` line 150), which processes only `WITHDRAWAL_REQUESTS_BATCH_SIZE = 5` requests per invocation (line 39). The timer fires every `PROCESS_ETH_RETRIEVE_TRANSACTIONS_INTERVAL = 6 minutes` (`rs/ethereum/cketh/minter/src/lib.rs` line 35), with a retry at `PROCESS_ETH_RETRIEVE_TRANSACTIONS_RETRY_INTERVAL = 3 minutes` (line 37) if requests remain. Draining 100 requests therefore takes approximately 60 minutes under ideal conditions. An attacker who resubmits 5 new requests every 3 minutes keeps the queue perpetually saturated.

The existing unit test `should_allow_limited_number_of_pending_requests` (`rs/ethereum/cketh/minter/src/guard/tests.rs` lines 56–83) explicitly confirms that once `MAX_PENDING` requests are enqueued, all further `retrieve_withdraw_guard` calls return `Err(GuardError::TooManyPendingRequests)`.

There is no per-principal pending-request cap, no cooldown, and no economic barrier beyond the minimum withdrawal amount and Ethereum gas fees.

## Impact Explanation
This is a complete liveness failure for the ckETH (and ckERC20) withdrawal path: no user can initiate a new withdrawal while the queue is saturated. This matches the allowed High impact: **"Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS"** ($2,000–$10,000). The attack targets a production in-scope financial integration (ckETH/ckERC20 minter) and causes concrete, measurable harm to all users attempting withdrawals.

## Likelihood Explanation
An attacker needs 100 IC principals (free to create) and 100 × `cketh_minimum_withdrawal_amount` in ckETH. The attacker recovers their ETH after each cycle minus Ethereum gas fees (~21,000 gas × current base fee per transaction). At current Ethereum gas prices this is cents to low dollars per cycle. The attack is fully repeatable: once the minter processes the attacker's batch, the attacker re-deposits and refills the queue. No privileged access, governance vote, or subnet-level compromise is required. The attack is triggerable by any unprivileged external user.

## Recommendation
1. **Per-principal pending cap**: Reject `withdraw_eth`/`withdraw_erc20` if the calling principal already has N (e.g., 3–5) requests in `pending_withdrawal_requests`, preventing any single actor from monopolizing the queue.
2. **Raise `MAX_PENDING`**: Increase the global limit (e.g., to 1,000) to reduce the probability of accidental saturation and raise the capital cost of a deliberate attack.
3. **Increase batch size or timer frequency**: `WITHDRAWAL_REQUESTS_BATCH_SIZE = 5` with a 3–6 minute interval is the primary reason the queue drains slowly. Increasing the batch size or reducing the interval shrinks the window of vulnerability.
4. **Return a typed error instead of trapping**: Change `unwrap_or_else(|e| ic_cdk::trap(...))` to return `Err(WithdrawalError::TemporarilyUnavailable(...))` so callers receive a structured error rather than a trap, improving observability and client-side handling.

## Proof of Concept
The existing test at `rs/ethereum/cketh/minter/src/guard/tests.rs` lines 56–83 already proves the core behavior deterministically:

1. Initialize canister state.
2. For `i` in `0..MAX_PENDING` (100 iterations): acquire `retrieve_withdraw_guard(principal_i)` and call `record_withdrawal_request(LedgerBurnIndex::new(i))` to enqueue a pending request.
3. For any additional principal (index ≥ 100): `retrieve_withdraw_guard(principal)` returns `Err(GuardError::TooManyPendingRequests)`.

For a live integration test:
1. Create 100 IC principals.
2. Each principal deposits `minimum_withdrawal_amount` ETH to the ckETH helper contract and receives ckETH.
3. Each principal calls `withdraw_eth` with `amount = minimum_withdrawal_amount`.
4. After all 100 calls succeed, any new `withdraw_eth` call from any principal traps with `"Failed retrieving guard for principal …: TooManyPendingRequests"`.
5. Observe that the queue remains saturated until the minter's timer drains it (~60 minutes for 100 requests at batch size 5).
6. Re-deposit and repeat to sustain the DoS indefinitely.
