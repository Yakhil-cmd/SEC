### Title
`FatalRuntimeError::OutOfReturnMemory` in Subcall Propagates as Top-Level Transaction Revert, Deviating from EVM Semantics — (`basic_bootloader/src/bootloader/runner.rs`)

### Summary

`copy_into_return_memory` is called for **every** completed call frame, including subcalls. If a callee's returndata exceeds the remaining shared `return_memory` buffer, a `FatalRuntimeError::OutOfReturnMemory` is raised and propagates unconditionally up the entire call stack via `?` operators, causing a top-level transaction failure. EVM does not exhibit this behavior: large returndata from a subcall never causes the parent to revert.

### Finding Description

If the check fails, it returns `Err(out_of_return_memory!().into())` — a `BootloaderSubsystemError` wrapping `FatalRuntimeError::OutOfReturnMemory`.

The `return_memory` buffer is a fixed-size slice passed in from outside and is consumed progressively via `split_at_mut` — each completed call frame consumes a portion: [5](#0-4) 

### Impact Explanation

In EVM, a subcall returning a large buffer (e.g., 64 KB+) does **not** cause the caller to revert. The caller specifies `retOffset`/`retSize` in the CALL opcode; EVM copies only the requested slice and charges memory expansion gas. The full returndata remains accessible via `RETURNDATASIZE`/`RETURNDATACOPY`. No fatal error occurs.

In ZKsync OS, the same scenario triggers `OutOfReturnMemory`, which propagates as a fatal error and reverts the entire transaction. This is an observable EVM deviation: a contract that works correctly on Ethereum (returning large data) can cause a top-level revert on ZKsync OS, enabling DoS of any contract that relies on large returndata from subcalls.

### Likelihood Explanation

The trigger is fully unprivileged: deploy a contract whose bytecode executes `RETURN` with a large memory region (e.g., 64 KB+), then call it from another contract. No special permissions, governance access, or oracle manipulation is required. The attacker controls the returndata size entirely through the deployed bytecode.

### Recommendation

1. `copy_into_return_memory` should not be a fatal error path for subcalls. When a subcall's returndata exceeds the remaining buffer, the system should either truncate the stored returndata (matching EVM's `retSize` semantics) or charge equivalent gas and continue — not abort the transaction.
2. `OutOfReturnMemory` should only be fatal if it occurs at the top-level entry frame where no EVM-equivalent fallback exists.
3. Alternatively, size the `return_memory` buffer to be large enough that it cannot be exhausted by any EVM-valid transaction (bounded by the block gas limit and memory expansion costs).

### Proof of Concept

1. Deploy contract A: bytecode that `MSTORE`s zeros to fill 64 KB of memory, then executes `RETURN(0, 65536)`.
2. Deploy contract B: calls contract A via `CALL`.
3. Submit a transaction calling contract B.
4. **Expected (EVM)**: contract B's call to A succeeds; B can read returndata via `RETURNDATASIZE`; transaction succeeds.
5. **Actual (ZKsync OS)**: `copy_into_return_memory` at runner.rs:97–98 finds `65536 > remaining return_memory`, returns `Err(OutOfReturnMemory)`, which propagates through all `?` operators to `run_till_completion`, causing a top-level transaction revert. [6](#0-5) [7](#0-6)

    fn copy_into_return_memory<'a>(
        &mut self,
        return_values: ReturnValues<'a, S>,
    ) -> Result<ReturnValues<'external, S>, BootloaderSubsystemError> {
        let return_memory = core::mem::take(&mut self.return_memory);
        if return_values.returndata.len() > return_memory.len() {
            return Err(out_of_return_memory!().into());
        }
        let (output, rest) = return_memory.split_at_mut(return_values.returndata.len());
        self.return_memory = rest;

        Ok(ReturnValues {
            returndata: output.write_copy_of_slice(return_values.returndata),
            ..return_values
        })
    }
