### Title
Unbounded Recursive `NearPromise` Deserialization and Execution Causes Stack Overflow, Freezing NEAR in XCC Router Contracts - (File: `engine-types/src/parameters/promise.rs`, `etc/xcc-router/src/lib.rs`)

---

### Summary

The `NearPromise` type implements recursive Borsh deserialization and recursive traversal methods (`promise_count`, `total_gas`, `total_near`) without any depth limit. The XCC router's `recursive_promise_create` function is also unboundedly recursive. An unprivileged EVM user can supply a deeply nested `NearPromise` structure as calldata to the XCC precompile, triggering a WASM stack overflow. In the `Delayed` execution path, this can permanently freeze NEAR funds held in a user's XCC router sub-account.

---

### Finding Description

The `NearPromise` enum is a recursive tree type with three variants: `Simple`, `Then { base: Box<Self>, callback }`, and `And(Vec<Self>)`. Its hand-written Borsh deserializer calls itself recursively without any depth guard:

```rust
// engine-types/src/parameters/promise.rs, line 191
0x01 => {
    let base = Self::deserialize_reader(reader)?;  // unbounded recursion
    ...
}
```

Every method on `NearPromise` follows the same pattern — `promise_count` (line 119), `total_gas` (line 128), and `total_near` (line 143) all recurse into `base` without limit.

The XCC router's `recursive_promise_create` (lines 242–282 of `etc/xcc-router/src/lib.rs`) mirrors this:

```rust
NearPromise::Then { base, callback } => {
    let base_index = Self::recursive_promise_create(base); // unbounded recursion
    ...
}
```

The entry point for an unprivileged user is the XCC precompile (`engine-precompiles/src/xcc.rs`, line 137):

```rust
let args = CrossContractCallArgs::try_from_slice(input)
    .map_err(|_| ExitError::Other(Cow::from(consts::ERR_INVALID_INPUT)))?;
```

This directly deserializes attacker-controlled calldata into a `NearPromise` tree. No depth cap exists anywhere in the codebase — confirmed by the complete absence of any `MAX_DEPTH`, `depth_limit`, or equivalent constant.

**Fund-freeze attack path (`Delayed` variant)**:

1. Attacker crafts a `CrossContractCallArgs::Delayed(PromiseArgs::Recursive(NearPromise))` with nesting depth D — deep enough to overflow the WASM stack in `recursive_promise_create` but shallow enough to survive deserialization (the deserializer's stack frame is larger due to I/O state, so the two thresholds differ).
2. The XCC precompile succeeds: deserialization, `total_near()`, `promise_count()`, and `borsh::to_vec` all complete. The precompile emits a log targeting the router's `schedule` method.
3. The Aurora Engine processes the log and calls `schedule` on the router. The router stores the promise in `scheduled_promises` at a nonce.
4. Any caller invokes `execute_scheduled(nonce)` on the router (this function is intentionally open to anyone, per the comment at line 146).
5. The router removes the promise from storage and calls `Self::promise_create(promise)` → `Self::recursive_promise_create(&p)` → WASM stack overflow → NEAR transaction panics → **state reverts** (the removal is undone, promise stays in storage).
6. The NEAR balance held in the router (deposited during step 2 via wNEAR unwrapping) can never be disbursed. Every subsequent call to `execute_scheduled` for this nonce repeats the overflow and reverts.

---

### Impact Explanation

NEAR funds transferred into the XCC router sub-account as part of the `Delayed` XCC flow are permanently inaccessible. The `execute_scheduled` function is the only mechanism to consume a scheduled promise; if it always panics for a given nonce, the associated NEAR is frozen in the router. The `send_refund` method requires the Aurora Engine (parent) to call it and requires no failed promise results — it is not a general user-accessible recovery path. This constitutes **permanent freezing of user funds**.

Additionally, for the `Eager` path, a sufficiently deep nesting causes a WASM trap during deserialization inside the Aurora Engine itself, making the entire EVM transaction fail — a DoS on the XCC precompile for any user who triggers it.

---

### Likelihood Explanation

The XCC precompile is reachable by any EVM user holding an Aurora account. The `NearPromise::Then` variant requires only a single byte (`0x01`) per nesting level in Borsh encoding, so a 10,000-level deep structure fits in ~10 KB of calldata — well within EVM transaction size limits. No special privilege, governance action, or external dependency is required. The `execute_scheduled` function is explicitly open to any caller.

---

### Recommendation

1. **Cap nesting depth during deserialization**: Add a depth counter to `NearPromise::deserialize_reader` and return an error if it exceeds a safe threshold (e.g., 32 or 64 levels).
2. **Cap in `recursive_promise_create`**: Add a depth parameter and panic with a clear error message if exceeded.
3. **Cap in all recursive traversal methods** (`promise_count`, `total_gas`, `total_near`): Either use an iterative implementation or pass a depth counter.
4. **Validate at precompile entry**: After deserialization, call a bounded `validate_depth` check before any further processing, so malformed inputs are rejected cheaply.

### Proof of Concept

**Crafted calldata** (Borsh-encoded `CrossContractCallArgs::Delayed(PromiseArgs::Recursive(NearPromise))`):

```
[variant: Delayed = 0x01]
[variant: Recursive = 0x02]
[0x01, 0x01, 0x01, ... repeated 10000 times ...]  <- NearPromise::Then nesting
[0x00, ...]  <- NearPromise::Simple base case
[0x00, ...]  <- SimpleNearPromise::Create callback at each level
```

**Step-by-step**:
1. Deploy an EVM contract on Aurora that calls the XCC precompile at `0x516cded1d16af10cad47d6d49128e2eb7d27b372` with the above calldata.
2. The `CrossContractCallArgs::try_from_slice` call in `engine-precompiles/src/xcc.rs` line 137 triggers `NearPromise::deserialize_reader` recursively 10,000 times → WASM stack overflow → Aurora Engine NEAR transaction traps.
3. For the fund-freeze variant: tune depth D so deserialization succeeds (e.g., D = 500), then call `execute_scheduled` on the router → `recursive_promise_create` overflows at D → NEAR frozen.
