Audit Report

## Title
Unbounded `retrieve_btc_account_to_block_indices` Vec Causes Instruction-Limit Exhaustion in `retrieve_btc_status_v2_by_account` — (File: `rs/bitcoin/ckbtc/minter/src/state.rs`)

## Summary
Every accepted `retrieve_btc` request appends a `block_index` to a per-account `Vec<u64>` in `retrieve_btc_account_to_block_indices`, and this Vec is never pruned when requests are finalized. The public query `retrieve_btc_status_v2_by_account` iterates over the entire Vec and invokes `retrieve_btc_status_v2` — which performs three linear scans — for every entry, creating O(m × n) instruction cost that grows without bound as an account accumulates historical withdrawals. For a sufficiently large history, the query will permanently exceed the IC instruction limit for that account.

## Finding Description
**Unbounded accumulation:** In `audit.rs:27-33`, `accept_retrieve_btc_request` conditionally pushes `request.block_index` into the per-account Vec whenever `request.reimbursement_account` is `Some`. The field is declared as `BTreeMap<Account, Vec<u64>>` at `state.rs:462-463` with no size cap. Reviewing all event replay branches in `eventlog.rs:395-660`, no event type ever removes an entry from `retrieve_btc_account_to_block_indices` — not `RemovedRetrieveBtcRequest`, not `ConfirmedBtcTransaction`, not any reimbursement event. By contrast, `finalized_requests` is explicitly capped at `MAX_FINALIZED_REQUESTS = 100` (`state.rs:47`).

**Unbounded iteration in the query:** `retrieve_btc_status_v2_by_account` at `state.rs:807-819` fetches the full Vec and maps `retrieve_btc_status_v2` over every element with no pagination or limit. Each call falls through to `retrieve_btc_status` at `state.rs:872-917`, which performs:
- `pending_retrieve_btc_requests.iter().any(...)` — O(pending)
- `submitted_transactions.iter().find_map(...)` — O(submitted)
- `finalized_requests.iter().find(...)` — O(100)

For entries older than the 100-entry finalized window, all three scans run to completion and return `Unknown`, meaning the full scan cost is paid for every historical entry. Total instruction cost per query: O(m × (pending + submitted + 100)).

**Guard insufficiency:** `MAX_CONCURRENT_PENDING_REQUESTS` at `retrieve_btc.rs:174-179` only limits in-flight requests. Once a batch is confirmed, the slot is freed and new requests can be submitted, allowing unbounded accumulation of historical entries over time.

## Impact Explanation
Once the per-account Vec grows large enough, every invocation of `retrieve_btc_status_v2_by_account` for that account traps with `CanisterInstructionLimitExceeded`. The endpoint becomes permanently unavailable for that account until a canister upgrade prunes the map. Individual statuses remain queryable via `retrieve_btc_status` / `retrieve_btc_status_v2`, so funds are not at risk. This is a permanent application-level DoS on a specific query endpoint, matching the Medium impact tier: "Difficult attack requiring significant... cost... but still with meaningful security impact."

## Likelihood Explanation
Each `retrieve_btc` call requires burning real ckBTC, imposing a direct economic cost per entry. Reaching the instruction limit requires accumulating on the order of hundreds of thousands of entries, which at minimum withdrawal amounts represents a very large economic outlay, making deliberate exploitation expensive. However, the minter has been live on mainnet for years, and high-volume integrators or automated systems processing thousands of withdrawals over the minter's lifetime could approach problematic sizes organically without any malicious intent. No privileged access is required; any authenticated user can trigger accumulation.

## Recommendation
1. **Prune on finalization:** When a request transitions to `Confirmed` or `AmountTooLow` (i.e., in the `RemovedRetrieveBtcRequest` and `ConfirmedBtcTransaction` event replay branches in `eventlog.rs`), remove the corresponding `block_index` from the per-account Vec in `retrieve_btc_account_to_block_indices`, and remove the account key if the Vec becomes empty.
2. **Paginate the query:** Add `start: u64` and `limit: u64` parameters to `retrieve_btc_status_v2_by_account` so callers cannot trigger unbounded iteration in a single query call.
3. **Add an invariant:** Assert that the total number of entries across all accounts in `retrieve_btc_account_to_block_indices` is bounded (e.g., does not exceed `MAX_CONCURRENT_PENDING_REQUESTS` plus a small constant for in-flight entries).

## Proof of Concept
1. Obtain ckBTC on mainnet or a local replica with PocketIC.
2. Repeatedly call `retrieve_btc` (or `retrieve_btc_with_approval`), waiting for each batch to be confirmed before submitting the next to stay under `MAX_CONCURRENT_PENDING_REQUESTS`.
3. After *k* confirmed withdrawals, call `retrieve_btc_status_v2_by_account(null)`.
4. Write a deterministic integration test (using PocketIC) that mocks the instruction counter or measures instruction consumption per call as *k* increases, demonstrating linear growth. At sufficiently large *k* the query traps with `CanisterInstructionLimitExceeded` because the minter iterates over all *k* block indices, each triggering three full linear scans of the live transaction queues.
