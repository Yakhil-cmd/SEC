### Title
Weak RNG Seeding at Canister Initialization Produces Predictable Rule UUIDs, Leaking Confidential Incident IDs - (File: `rs/boundary_node/rate_limits/canister/random.rs`)

---

### Summary

The rate-limit canister's custom `getrandom` backend seeds its ChaCha20 RNG once at canister initialization using only 8 bytes of IC time plus 24 constant bytes (`42`). Every `RuleId` UUID subsequently generated by `generate_random_uuid()` is fully predictable by any observer who knows the canister's initialization timestamp. Because `get_rule_by_id` is a public query that always returns the `incident_id` field even for confidential (not-yet-disclosed) rules, an attacker can enumerate predicted rule IDs and extract the `incident_id` of active security incidents before they are officially disclosed.

---

### Finding Description

**Root cause — `rs/boundary_node/rate_limits/canister/random.rs` lines 8–13:**

```rust
thread_local! {
  static RNG: RefCell<ChaCha20Rng> = {
    let mut seed = [42; 32];
    seed[..8].copy_from_slice(&time().to_le_bytes());
    RefCell::new(ChaCha20Rng::from_seed(seed))
  };
}
```

The seed is constructed from:
- Bytes `[0..8]`: `time()` — the IC canister time in nanoseconds at the moment the thread-local is first accessed (canister install or upgrade).
- Bytes `[8..32]`: the constant `0x2a` (`42`) repeated 24 times.

Effective entropy: **64 bits** (the IC timestamp), with the remaining 192 bits being a known constant. The IC time is publicly observable from the blockchain. [1](#0-0) 

This RNG is registered as the global `getrandom` provider for the canister: [2](#0-1) 

**UUID generation — `rs/boundary_node/rate_limits/canister/add_config.rs` lines 186–193:**

```rust
fn generate_random_uuid() -> Result<Uuid, anyhow::Error> {
    let mut buf = [0_u8; 16];
    getrandom::getrandom(&mut buf)...
    let uuid = Uuid::from_slice(&buf)...
    Ok(uuid)
}
```

Every new `RuleId` is drawn from this weakly-seeded, deterministic RNG. [3](#0-2) 

**Data exposure path — `rs/boundary_node/rate_limits/canister/getter.rs` lines 224–243:**

`get_rule_by_id` is a public `#[query]` method (bypasses `inspect_message`). For `RestrictedRead` callers, `RuleConfidentialityFormatter` redacts only `rule_raw` and `description`. The fields `incident_id`, `added_in_version`, `removed_in_version`, and `disclosed_at` are **always returned**, even for confidential rules: [4](#0-3) 

The formatter confirms only `rule_raw` and `description` are redacted: [5](#0-4) 

---

### Impact Explanation

An unprivileged caller who predicts a `RuleId` can call `get_rule_by_id` and receive the `incident_id` of a confidential rate-limit rule before it is disclosed. The `incident_id` identifies the security incident that triggered the rule (e.g., an ongoing DDoS or exploit campaign). Premature disclosure of `incident_id` values undermines the confidentiality model of the rate-limit canister, which is explicitly designed to keep active incident rules secret until the authorized principal chooses to disclose them.

Additionally, `added_in_version` and `removed_in_version` metadata leak the timing and lifecycle of confidential rules to any observer.

---

### Likelihood Explanation

The IC canister initialization time is recorded on-chain and is publicly readable. The seed construction is trivially reversible: an attacker reads the canister's install/upgrade timestamp from the blockchain, constructs `seed = [time_le_bytes[0..8], 42×24]`, instantiates `ChaCha20Rng::from_seed(seed)`, and advances the RNG by the number of rules already created (observable from `get_config` responses, which return `rule_id` values for all rules including confidential ones). The next UUID output is then the predicted `RuleId`. No privileged access, no threshold attack, and no social engineering is required.

---

### Recommendation

Replace the static, time-seeded `thread_local!` RNG with a per-call invocation of the IC management canister's `raw_rand` endpoint (as already done correctly in the salt-sharing canister at `rs/boundary_node/salt_sharing/canister/helpers.rs`):

```rust
pub async fn generate_random_uuid() -> Result<Uuid, anyhow::Error> {
    let (rand_bytes,): ([u8; 16],) =
        ic_cdk::call(Principal::management_canister(), "raw_rand", ())
            .await
            .map_err(|e| anyhow::anyhow!("{:?}: {}", e.0, e.1))?;
    Ok(Uuid::from_slice(&rand_bytes)?)
}
``` [6](#0-5) 

This ensures each UUID is derived from subnet-consensus-level randomness with full 128-bit entropy, making it unpredictable to any external observer.

---

### Proof of Concept

1. Read the rate-limit canister's install/upgrade timestamp `T` from the IC blockchain (e.g., via `dfx canister info` or the dashboard).
2. Construct the seed: `seed[0..8] = T.to_le_bytes(); seed[8..32] = [42u8; 24]`.
3. Instantiate `ChaCha20Rng::from_seed(seed)`.
4. Call `get_config(None)` (public query) to determine how many rules `N` have already been created and consume `N × 16` bytes from the RNG to advance past already-issued UUIDs.
5. Read the next 16 bytes from the RNG; this is the `RuleId` of the next rule the authorized principal will add.
6. Call `get_rule_by_id(<predicted_uuid>)` as an anonymous caller.
7. Receive a response containing `incident_id`, `added_in_version`, and lifecycle metadata for a confidential rule — before it has been disclosed.
