Audit Report

## Title
Third-Party Caller Can Bypass `confirmation_text` Consent Gate in SNS Swap - (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` endpoint accepts an attacker-supplied `buyer` principal and `confirmation_text` from the request body without verifying that the caller is the buyer. When a `confirmation_text` is configured, the code's own documentation states "the **caller** of this function must accept this confirmation," but no such enforcement exists. Any unprivileged caller can commit a victim's ICP to the swap by supplying the victim's principal and the publicly-readable `confirmation_text`, bypassing the informed-consent gate entirely.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 128–143, the handler resolves the acting principal from `arg.buyer` (attacker-controlled) rather than from `caller_principal_id()` when `arg.buyer` is non-empty:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // attacker-supplied
};
``` [1](#0-0) 

Both `p` and `arg.confirmation_text` are forwarded directly to `refresh_buyer_token_e8s`: [2](#0-1) 

Inside `refresh_buyer_token_e8s`, the only consent gate is `validate_confirmation_text`, which checks only that the supplied string matches the SNS-configured text — it does **not** verify `buyer == caller`: [3](#0-2) 

The function's own docstring at lines 1123–1126 explicitly states the design intent: *"the **caller** of this function must accept this confirmation"* — but the code never enforces `caller == buyer`: [4](#0-3) 

The `confirmation_text` is stored in the public `Init` protobuf field (readable via `get_sale_parameters`), so the attacker has zero-effort access to the required string. After passing the text check, the function reads the ICP balance of the victim's swap subaccount and records the participation: [5](#0-4) 

The victim's ICP is then committed at lines 1285–1288: [6](#0-5) 

**Exploit path:**
1. Victim transfers ICP to the swap subaccount (normal first step) but has not yet called `refresh_buyer_tokens`.
2. Attacker reads `confirmation_text` via `get_sale_parameters` (public query).
3. Attacker calls `refresh_buyer_tokens({ buyer: "<victim>", confirmation_text: Some("<text>") })`.
4. Victim's ICP is committed to the swap without the victim ever explicitly agreeing to the terms.

## Impact Explanation
The `confirmation_text` field exists specifically to enforce informed consent before ICP is committed to an SNS swap. Bypassing it forces a victim into a financial transaction — exchanging ICP for SNS tokens — without their explicit authorization at the time of commitment. This constitutes a significant SNS security impact with concrete user harm: the victim's funds are irreversibly committed to the swap outcome (tokens received or ICP locked until abort) without consent. This maps to **High ($2,000–$10,000)**: "Significant SNS… security impact with concrete user or protocol harm."

## Likelihood Explanation
All preconditions are realistic and observable on-chain: (1) SNS launches commonly set `confirmation_text` as a governance best practice; (2) the victim having transferred ICP but not yet called `refresh_buyer_tokens` is a normal intermediate state in the two-step flow; (3) reading `confirmation_text` requires only a public query call. No privileged access, key compromise, or social engineering is required. Any unprivileged ingress sender can execute this against any victim in the intermediate state.

## Recommendation
When `confirmation_text` is set, enforce that `buyer == caller`. The simplest fix in `rs/sns/swap/canister/canister.rs`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let p = PrincipalId::from_str(&arg.buyer).unwrap();
    if swap().init_or_panic().confirmation_text.is_some()
        && p != caller_principal_id()
    {
        panic!("confirmation_text requires buyer == caller");
    }
    p
};
```

Alternatively, enforce this constraint inside `refresh_buyer_token_e8s` by accepting the caller principal as a separate parameter and rejecting mismatches when `confirmation_text` is configured.

## Proof of Concept
```
// Attacker (any principal A) sends ingress to SNS Swap canister:
refresh_buyer_tokens({
    buyer: "<victim_principal>",   // victim already sent ICP to subaccount
    confirmation_text: Some("<text from get_sale_parameters>")
})
// Result: victim's ICP committed to swap at swap.rs:1285-1288
//         without victim ever calling refresh_buyer_tokens themselves.
```

A deterministic unit test in `rs/sns/swap/tests/swap.rs` can reproduce this by: (1) opening a swap with `confirmation_text` set, (2) crediting a victim subaccount on the mock ICP ledger, (3) calling `refresh_buyer_token_e8s` with `buyer = victim` but a different caller principal and the correct `confirmation_text`, and (4) asserting the victim appears in `self.buyers` with a committed balance.
