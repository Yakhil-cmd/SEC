Audit Report

## Title
Gasless `UpdateSettings` Ingress Execution via Deferred-Charge Bypass Enables Subnet Throughput DoS — (`rs/cycles_account_manager/src/cycles_account_manager.rs`, `rs/execution_environment/src/execution_environment.rs`)

## Summary

`UpdateSettings` ingress messages with payloads ≤ 338 bytes are classified as `IngressInductionCost::Free`, bypassing all cycle checks at induction, block-building, and the HTTP ingress filter's balance gate. The deferred post-execution charge explicitly discards `CanisterOutOfCyclesError` via `let _ignore_error`. An attacker who owns a zero-cycle canister can therefore submit `UpdateSettings` messages targeting that canister at unlimited rate with zero cycle cost, filling block ingress slots and consuming subnet execution instruction budget without payment, degrading throughput for all other canisters on the subnet.

## Finding Description

**Step 1 — Free induction classification.**

`ingress_induction_cost()` sets `paying_canister = None` (yielding `IngressInductionCost::Free`) whenever the method is `UpdateSettings` and `is_delayed_ingress_induction_cost()` returns true, i.e., `arg.len() <= MAX_DELAYED_INGRESS_COST_PAYLOAD_SIZE` (338 bytes). [1](#0-0) [2](#0-1) [3](#0-2) 

**Step 2 — Cycle check skipped at all three enforcement points.**

`valid_set_rule::enqueue()` pushes the message directly into the input queue with no cycle withdrawal when the result is `Free`: [4](#0-3) 

`ingress_selector::validate_ingress()` does nothing for `Free`: [5](#0-4) 

`execution_environment::should_accept_ingress_message()` only enters the balance-check branch for `Fee`; the `Free` path falls through to the controller check without any cycle gate: [6](#0-5) 

**Step 3 — Controller check at the HTTP handler does NOT prevent the attack for own canisters.**

`canister_manager::should_accept_ingress_message()` does verify that the sender is a controller of the target canister before accepting the message: [7](#0-6) 

This means a random user cannot target arbitrary canisters via the HTTP endpoint. However, the attacker only needs to own a canister — which requires a one-time creation cost — and then drain it to zero cycles. After that, they are the controller of a zero-cycle canister and can submit unlimited `UpdateSettings` messages targeting it. The `ingress_selector::validate_ingress()` path used during block validation performs no controller check, so the messages pass consensus-level validation unconditionally once in the pool.

**Step 4 — Deferred charge silently discarded.**

After `update_settings` executes, the post-execution charge is attempted but the error is explicitly thrown away: [8](#0-7) 

For a zero-cycle canister, `consume_cycles` returns `CanisterOutOfCyclesError`, which is silently dropped. No cycles are ever collected.

**Step 5 — Existing test confirms the design and the zero-cost path.**

The test `management_message_update_setting_is_inducted_but_not_charged` explicitly asserts that `balance_after == balance_before` after induction of a small `UpdateSettings` payload, confirming the free-induction path is intentional and tested: [9](#0-8) 

## Impact Explanation

An attacker who controls a zero-cycle canister can continuously submit `UpdateSettings` messages at the maximum ingress rate (`MAX_INGRESS_MESSAGES_PER_BLOCK` = 1000 per block, ~1 block/second) with zero cycle cost. Each message occupies a block ingress slot, an ingress history entry, and consumes subnet execution instruction budget. Sustained flooding displaces legitimate ingress messages from blocks and degrades execution throughput for all other canisters on the subnet. This matches the allowed impact: **High — Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS**, because the attack exploits a specific resource-metering bypass in production execution code rather than raw bandwidth exhaustion. [10](#0-9) 

## Likelihood Explanation

The precondition is owning any canister on the target subnet (a one-time cost to create it) and draining it to zero cycles. No special privileges, governance majority, threshold corruption, or leaked keys are required. The payload is trivially constructable (a valid Candid-encoded `UpdateSettingsArgs` with only `canister_id` and `freezing_threshold` fits well within 338 bytes, as confirmed by the existing unit test). The attack is fully repeatable and sustainable indefinitely at no ongoing cost. [1](#0-0) 

## Recommendation

1. **Decouple the balance check from the charge.** At induction time (both `should_accept_ingress_message` and `valid_set_rule::enqueue`), verify that the paying canister holds sufficient cycles to cover the induction cost even when the actual debit is deferred. The check and the charge can remain temporally separated; only the check needs to move earlier.

2. **Do not silently discard the deferred charge error.** If `consume_cycles` fails post-execution, emit a warning-level log entry and increment a metric so operators can detect and alert on systematic non-payment.

3. **Consider rate-limiting free-induction management messages per sender principal** to bound worst-case block-space consumption even if the balance check is not moved.

## Proof of Concept

```
# One-time setup: create canister C on target subnet, drain to 0 cycles.
# Attacker is the sole controller of C.

payload = Candid.encode(UpdateSettingsArgs {
    canister_id: C,
    settings: CanisterSettingsArgs { freezing_threshold: Some(1) },
    sender_canister_version: None,
})
# assert len(payload) <= 338  → ingress_induction_cost returns Free

# Sustained flood (no cycle cost per iteration):
loop:
    submit_ingress(to=IC_00, method="update_settings", payload=payload)
    # HTTP handler: should_accept_ingress_message → controller check passes (attacker owns C)
    #               cycle check skipped (Free branch)
    # Ingress pool: message admitted
    # Block building: validate_ingress → cycle check skipped (Free branch)
    # Induction: valid_set_rule::enqueue → Free branch, no withdrawal
    # Execution: update_settings runs; consume_cycles fails; _ignore_error discards it
    # Net cycles paid: 0
```

A deterministic integration test using `PocketIC` or `ExecutionTestBuilder` can confirm this by: (1) creating a canister, (2) setting its balance to zero, (3) submitting a small `UpdateSettings` ingress message as the controller, (4) asserting the message is inducted and executed, and (5) asserting the canister balance remains zero after execution. fn management_message_update_setting_is_inducted_but_not_charged() {
```rust
    let ingress_history_writer = MockIngressHistory::new();
    let metrics_registry = MetricsRegistry::new();
    let subnet_id = subnet_test_id(1);
    let valid_set_rule = ValidSetRuleImpl::new(
        Arc::new(ingress_history_writer),
        Arc::new(
            CyclesAccountManagerBuilder::new()
                .with_subnet_id(subnet_id)
                .build(),
        ),
        &metrics_registry,
        no_op_logger(),
    );

    let mut state = ReplicatedStateBuilder::new().build();
    let canister_id = canister_test_id(0);
    let canister = get_running_canister(canister_id);
    let balance_before = canister.system_state.balance();
    state.put_canister_state(canister);

    let payload = UpdateSettingsArgs {
        canister_id: canister_id.get(),
        settings: CanisterSettingsArgsBuilder::new()
            .with_freezing_threshold(1 << 20)
            .build(),
        sender_canister_version: None,
    }
    .encode();
    let ingress = SignedIngressBuilder::new()
        .canister_id(IC_00)
        .method_name("update_settings")
        .method_payload(payload)
        .build();
    assert!(valid_set_rule.enqueue(&mut state, ingress).is_ok());

    let balance_after = state
        .canister_state(&canister_id)
        .unwrap()
        .system_state
        .balance();

    assert_eq!(balance_after, balance_before);
```
