### Title
Cross-Chain Signature Replay via Accepted Pre-EIP-155 Legacy Transactions - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

ZKsync OS accepts pre-EIP-155 legacy transactions (where `v = 27` or `v = 28`) and computes their signing hash **without including the chain ID**. This is structurally identical to the ERC1271Handler vulnerability class: a signature hash that omits chain-binding context allows a valid signature from one chain to be replayed on another. An attacker who observes a pre-EIP-155 transaction on Ethereum mainnet (or any other EVM chain) can submit it verbatim to ZKsync OS and, when the sender's nonce matches, execute an unauthorized state transition — including fund transfers or arbitrary contract calls — on behalf of the original signer.

---

### Finding Description

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, the signing hash for unprotected legacy transactions (those with `v == 27` or `v == 28`, i.e., `is_eip155() == false`) is computed as:

```
keccak256(rlp_list_header || [nonce, gasPrice, gasLimit, to, value, data])
```

No chain ID is included. This is the pre-EIP-155 hash format, intentionally chain-agnostic. [1](#0-0) 

The `is_eip155()` predicate returns `false` precisely when `v` is 27 or 28: [2](#0-1) 

The caller `parse_and_compute_signed_hash` does **not** reject the `Self::Legacy` variant — it is accepted and passed through to signature verification: [3](#0-2) 

Signature verification in the ZK validation path then runs `secp256k1_ec_recover` against this chain-ID-free hash and compares the result to the oracle-supplied `from` address: [4](#0-3) 

Because the hash is identical on every EVM chain for the same `[nonce, gasPrice, gasLimit, to, value, data]` tuple, a signature that is valid on Ethereum mainnet is equally valid on ZKsync OS.

---

### Impact Explanation

An attacker who observes a pre-EIP-155 legacy transaction broadcast on Ethereum mainnet (or any other EVM chain) can submit the identical raw bytes to the ZKsync OS sequencer. If the sender's nonce on ZKsync OS equals the nonce embedded in the replayed transaction, the bootloader will:

1. Recover the correct sender address via `ecrecover` (the hash is the same).
2. Pass nonce validation (nonces match by assumption).
3. Execute the transaction — transferring ETH, calling contracts, or deploying code — without the sender's knowledge or consent.

The impact is **direct, unauthorized loss of user funds** or **unauthorized contract execution** on ZKsync OS, triggered by a transaction the user signed for a different chain.

---

### Likelihood Explanation

Pre-EIP-155 transactions are rare in modern tooling but remain possible. Wallets that default to legacy signing without EIP-155 (e.g., certain hardware wallets in compatibility modes, older tooling, or explicit `chain_id: None` configurations) produce exactly this format. The nonce-alignment requirement reduces the attack window but does not eliminate it: nonce 0 is the most common case (a fresh account on ZKsync OS whose owner has previously signed a pre-EIP-155 transaction on mainnet at nonce 0). The attacker's only requirement is to observe the transaction on a public mempool or block explorer and submit it to the ZKsync OS sequencer before the victim's ZKsync OS nonce advances past the replayed value.

---

### Recommendation

Reject pre-EIP-155 legacy transactions entirely. ZKsync OS is a new L2 chain with its own chain ID; there is no compatibility obligation to accept chain-agnostic signatures. Add an explicit check in `parse_and_compute_signed_hash` (or in `try_parse_and_hash_for_signature_verification`) that returns `Err(InvalidTransaction::InvalidChainId)` when `sig_data.is_eip155()` is `false`, mirroring the chain ID enforcement already applied to all EIP-2718 typed transactions: [5](#0-4) 

---

### Proof of Concept

1. On Ethereum mainnet, Alice signs a legacy transfer with `chain_id: None` (pre-EIP-155), producing `v = 27`, at nonce `N`.
2. Alice also holds funds on ZKsync OS at the same address, with nonce `N`.
3. Attacker observes Alice's Ethereum transaction (raw RLP bytes).
4. Attacker submits the identical raw bytes to the ZKsync OS sequencer as an `RlpEncodedTx`.
5. The bootloader calls `LegacyPayloadParser::try_parse_and_hash_for_signature_verification` with `expected_chain_id = <ZKsync OS chain ID>`.
6. Because `v == 27`, `is_eip155()` returns `false`; the hash is computed **without** chain ID — identical to the Ethereum mainnet hash.
7. `secp256k1_ec_recover` recovers Alice's address; it matches the oracle-supplied `from`.
8. Nonce `N` matches; the transaction executes, draining Alice's ZKsync OS balance to the attacker-controlled `to` address. [6](#0-5) [7](#0-6)
