## Title
ckETH Permanently Lost When `erc20_tx_fee ≤ CKETH_LEDGER_TRANSACTION_FEE` and ckERC20 Burn Fails - (File: rs/ethereum/cketh/minter/src/main.rs)

## Summary
In `withdraw_erc20`, the minter burns ckETH first (for gas fees), then attempts to burn ckERC20. If the ckERC20 burn fails with a user-attributable error and `erc20_tx_fee ≤ CKETH_LEDGER_TRANSACTION_FEE` (2 trillion wei), the computed `reimbursed_amount` becomes `Wei::ZERO` via `checked_sub(...).unwrap_or(Wei::ZERO)`, and the `if reimbursed_amount > Wei::ZERO` guard silently suppresses the `FailedErc20WithdrawalRequest` event entirely. The already-burned ckETH is permanently destroyed with no reimbursement path and no on-chain record of the loss.

## Finding Description
`withdraw_erc20` in `rs/ethereum/cketh/minter/src/main.rs` performs two sequential inter-canister burns:

**Step 1 (lines 448–459):** Burns `erc20_tx_fee` ckETH via `cketh_ledger.burn_from(...)`. On success, `cketh_ledger_burn_index` is obtained.

**Step 2 (lines 468–477):** Burns `ckerc20_withdrawal_amount` ckERC20 via `ckerc20_ledger.burn_from(...)`.

When Step 2 fails with `InsufficientFunds`, `AmountTooLow`, or `InsufficientAllowance`, the error branch at lines 507–513 computes:

```rust
LedgerBurnError::InsufficientFunds { .. }
| LedgerBurnError::AmountTooLow { .. }
| LedgerBurnError::InsufficientAllowance { .. } => erc20_tx_fee
    .checked_sub(CKETH_LEDGER_TRANSACTION_FEE)
    .unwrap_or(Wei::ZERO),
```

When `erc20_tx_fee ≤ CKETH_LEDGER_TRANSACTION_FEE` (2 trillion wei, line 59), `checked_sub` returns `None`, `unwrap_or` yields `Wei::ZERO`, and the guard at line 515:

```rust
if reimbursed_amount > Wei::ZERO {
    // EventType::FailedErc20WithdrawalRequest emitted here
}
```

is never entered. No `FailedErc20WithdrawalRequest` event is emitted, so `process_reimbursement` (which reads exclusively from `reimbursement_requests_iter()` at `withdraw.rs` lines 55–60) never sees this burn index. The ckETH burned in Step 1 is permanently destroyed.

## Impact Explanation
This is a permanent, silent loss of ckETH — an in-scope chain-key token. The ckETH ledger supply is reduced by `erc20_tx_fee` without any corresponding Ethereum-side event or reimbursement. The loss is invisible to auditors and dashboards because no `FailedErc20WithdrawalRequest` event is ever recorded. This constitutes a concrete ledger conservation violation in an in-scope ck-token financial integration, matching the **High** impact class: "Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm." The per-incident loss is bounded by `CKETH_LEDGER_TRANSACTION_FEE` (≤ 0.000002 ETH), but the loss is permanent and unauditable, and the path is reachable by any unprivileged caller.

## Likelihood Explanation
**Low.** The boundary condition requires `erc20_tx_fee ≤ 2_000_000_000_000 wei`. With `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT = 65,000` gas, this implies an effective gas price below ~0.031 gwei — far below typical Ethereum mainnet levels (1–100+ gwei). However, the user-attributable error branch is trivially triggered by any caller who approves ckETH but not ckERC20 (or approves insufficient ckERC20). No privileged access is required. The condition is reachable during extreme low-activity periods or on testnets, and the gas price oracle returning an anomalously low value (e.g., due to a stale read or edge-case in fee estimation) could also trigger it.

## Recommendation
1. **Upfront rejection:** Before burning ckETH, assert `erc20_tx_fee > CKETH_LEDGER_TRANSACTION_FEE`. If not, return an error immediately — no ckETH is burned and no loss occurs.
2. **Remove the guard:** Replace `if reimbursed_amount > Wei::ZERO` with unconditional emission of `FailedErc20WithdrawalRequest` (even with `reimbursed_amount = 0`) so the loss is always auditable.
3. **Stronger invariant:** Add a post-burn assertion or integration test that verifies: for every `cketh_ledger_burn_index` produced in `withdraw_erc20`, either an `AcceptedErc20WithdrawalRequest` or a `FailedErc20WithdrawalRequest` event is always emitted.

## Proof of Concept
1. Deploy minter on a testnet or local replica where the gas price oracle returns a value such that `erc20_tx_fee ≤ 2_000_000_000_000 wei` (e.g., gas price = 1 wei, gas limit = 65,000 → fee = 65,000 wei).
2. Unprivileged user calls `icrc2_approve` on the ckETH ledger for `erc20_tx_fee`, but does **not** approve the ckERC20 ledger (or approves 0).
3. User calls `withdraw_erc20(...)`.
4. **Step 1** succeeds: `cketh_ledger.burn_from(cketh_account, erc20_tx_fee, ...)` burns ckETH and returns `cketh_ledger_burn_index`.
5. **Step 2** fails: `ckerc20_ledger.burn_from(...)` returns `Err(LedgerBurnError::InsufficientAllowance { .. })`.
6. `reimbursed_amount = erc20_tx_fee.checked_sub(CKETH_LEDGER_TRANSACTION_FEE).unwrap_or(Wei::ZERO)` → `Wei::ZERO`.
7. `if reimbursed_amount > Wei::ZERO` is `false` → no event emitted, no reimbursement enqueued.
8. Query `reimbursement_requests_iter()` — the burn index is absent. Query ckETH ledger — the burn is recorded. The ckETH is permanently gone.
9. Verify: total ckETH supply decreased by `erc20_tx_fee` with no corresponding Ethereum withdrawal or reimbursement mint.
