Audit Report

## Title
Hot Keys Not Cleared When NNS Neuron Is Spawned to a New Controller - (File: rs/nns/governance/src/governance.rs)

## Summary
When `spawn_neuron` is called with a `new_controller` different from the parent neuron's controller, the parent's hot keys are unconditionally copied to the child neuron. This allows the original hot key holders to exercise governance operations — including `RegisterVote`, `Follow`, and `JoinCommunityFund` — on the new controller's neuron without their knowledge or consent. No code path clears or filters inherited hot keys when the controller changes.

## Finding Description
In `rs/nns/governance/src/governance.rs` at L2651–2655, `child_controller` is set to `spawn.new_controller` if provided, otherwise falls back to the parent's controller. No conditional logic follows to handle the case where the two differ. At L2704–2718, `NeuronBuilder` constructs the child neuron with `.with_hot_keys(parent_neuron.hot_keys.clone())` unconditionally:

```rust
let child_neuron = NeuronBuilder::new(
    child_nid,
    to_subaccount,
    child_controller,   // may be a completely different principal
    ...
)
.with_hot_keys(parent_neuron.hot_keys.clone())  // ← no filtering
```

In `rs/nns/governance/src/neuron/types.rs` at L253–255, `is_hotkey_or_controller` returns `true` for any principal in `hot_keys`, regardless of who the current controller is:

```rust
fn is_hotkey_or_controller(&self, principal: &PrincipalId) -> bool {
    self.is_controlled_by(principal) || self.hot_keys.contains(principal)
}
```

This means every hot key inherited from the parent neuron immediately has full hot-key-level authorization on the child neuron whose controller is a different, unrelated principal. There are no post-spawn guards, no filtering step, and no notification to the new controller.

## Impact Explanation
This constitutes unauthorized access to a neuron's governance capabilities. The new controller (User B) receives a neuron they believe they fully control, but the spawner's hot key holders retain the ability to: cast votes on NNS proposals on behalf of User B's neuron; change followees for governance topics; and call `JoinCommunityFund`, committing User B's future maturity to the Neurons' Fund — a financially consequential action that is not automatically reversed. This matches the **High** impact class: *Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints.*

## Likelihood Explanation
Spawning to a `new_controller` is the standard on-chain mechanism for transferring maturity-derived ICP to another party. Any neuron controller who has added hot keys for convenience (e.g., a mobile signing key) will inadvertently transfer those authorizations to the recipient's neuron. Deliberate exploitation requires only that the spawner retain a hot key before calling `spawn_neuron` with a different `new_controller` — no special privileges beyond controlling a neuron with maturity are needed. The recipient has no in-protocol notification and may never discover the inherited hot keys. Likelihood of unintentional leakage is **medium-to-high**; deliberate exploitation is **low-to-medium**.

## Recommendation
When `spawn_neuron` is called with a `new_controller` that differs from the parent's controller, the child neuron should be initialized with an empty hot key list. The fix is a one-line conditional at the `NeuronBuilder` call site:

```rust
let child_hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};
// ...
.with_hot_keys(child_hot_keys)
```

If hot key inheritance across controller boundaries is ever desired, it should be an explicit opt-in field in the `Spawn` message, not the default behavior.

## Proof of Concept
1. User A controls neuron N with hot key `H1` (a key User A controls).
2. User A calls `manage_neuron` → `Spawn { new_controller: Some(UserB), nonce: None, percentage_to_spawn: Some(100) }`.
3. `spawn_neuron` executes L2651–2655: `child_controller = UserB`. Then L2714: `.with_hot_keys(parent_neuron.hot_keys.clone())` — child neuron C is created with `controller = UserB` and `hot_keys = [H1]`.
4. User A (via `H1`) calls `manage_neuron` on neuron C with `RegisterVote { proposal_id: X, vote: Yes }`. Governance calls `is_authorized_to_vote` → `is_hotkey_or_controller` → `hot_keys.contains(H1)` → `true`. Vote is accepted.
5. User A (via `H1`) calls `manage_neuron` on neuron C with `JoinCommunityFund {}`. Accepted for the same reason, committing User B's maturity to the Neurons' Fund without User B's consent.
6. A deterministic integration test using `PocketIC` or the existing NNS governance test harness can reproduce this in under 10 lines by: creating a neuron, adding a hot key, spawning to a new controller, then asserting that the hot key can successfully call `RegisterVote` and `JoinCommunityFund` on the child neuron.
