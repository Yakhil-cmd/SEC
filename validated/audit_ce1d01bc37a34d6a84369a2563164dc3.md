## Title
Unvalidated Opening Pool Unbounded Growth via Missing-Complaint Deferral — (`rs/consensus/idkg/src/complaints.rs`)

## Summary
When `validate_openings` processes an opening for an active transcript but finds no matching complaint in the validated pool, it silently defers the opening with no removal action. Combined with a `purge_artifacts` routine that only removes openings for inactive transcripts, and the IDKG P2P channel configured with `SLOT_TABLE_NO_LIMIT`, a single Byzantine subnet peer can flood the unvalidated pool with openings that are never removed as long as the target transcript remains active, causing unbounded memory growth on targeted replicas.

## Finding Description

**Silent-defer branch** (`rs/consensus/idkg/src/complaints.rs`, lines 252–257): When `Action::Process` is returned (transcript is active and fully resolved) but `get_complaint_for_opening` finds no matching complaint in the validated pool, the code takes no action — no `RemoveUnvalidated`, no `HandleInvalid`, no timeout:

```rust
} else {
    // Defer handling the opening in case it was received
    // before the complaint.
    self.metrics
        .complaint_errors_inc("opening_missing_complaint");
}
```

The in-batch duplicate guard (`validated_openings`, lines 215–221) only tracks openings that were **moved to validated** in the current cycle. Deferred openings are never inserted into `validated_openings`, so multiple openings with the same `OpeningKey{transcript_id, dealer_id, opener_id}` but different content hashes are each deferred independently — none are removed. The `has_node_issued_opening` check (lines 631–637) queries only the **validated** pool, providing no protection for unvalidated deferred entries.

**Purge gap** (`rs/consensus/idkg/src/complaints.rs`, lines 663–671): `should_purge` returns `true` only when `!active_transcripts.contains(transcript_id)`. Openings for active transcripts (key transcripts, pre-signature transcripts) are never purged by this path:

```rust
fn should_purge(...) -> bool {
    requested_height <= current_height && !active_transcripts.contains(transcript_id)
}
```

This is confirmed by the test at lines 2012–2018 which explicitly documents "Opening 1: height <= current_height, active transcripts (not purged)."

**Bouncer provides no complaint-existence check** (`rs/consensus/idkg/src/lib.rs`, lines 611–617): The `IDkgBouncer` for openings only checks height against `finalized_height + LOOK_AHEAD`. Any opening within the height window is accepted regardless of whether a complaint exists.

**No per-peer slot table limit** (`rs/replica/setup_ic_network/src/lib.rs`, lines 75 and 291): The IDKG artifact channel is registered with `SLOT_TABLE_NO_LIMIT = usize::MAX`, imposing no cap on how many distinct openings a single peer can advertise.

**Spec divergence** (`rs/consensus/idkg/spec.md`, lines 1385–1391): The formal spec explicitly states that an opening without a matching complaint should return `Some(false)` (invalid → remove). The implementation defers indefinitely instead.

The exploit path is: Byzantine peer crafts N openings for an active transcript (e.g., the long-lived key transcript) with different content hashes → each passes the height-based bouncer → each enters the unvalidated pool → `validate_openings` hits the `opening_missing_complaint` branch for each → no action taken → `purge_artifacts` skips all because the transcript is active → pool grows without bound.

## Impact Explanation

This is a **High** severity finding matching the allowed impact: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

A Byzantine peer can cause unbounded memory growth in the unvalidated IDKG pool on every honest replica it targets. Key transcripts can remain active indefinitely (for the lifetime of a subnet key), meaning the accumulation window is unbounded. If the attacker targets all honest replicas simultaneously (which requires no additional capability beyond sending P2P messages), the result is subnet-wide OOM crashes and loss of availability. Even targeting a subset of replicas degrades subnet performance and can push replicas below the honest-majority threshold.

## Likelihood Explanation

- **Attacker capability**: Must be a registered subnet node (Byzantine peer below the `f` fault threshold). This is the standard Byzantine adversary model explicitly tolerated by the IC protocol.
- **No threshold corruption required**: A single Byzantine peer suffices.
- **No crypto forgery required**: Openings are never cryptographically verified in the `opening_missing_complaint` branch — they only need to pass the height check in the bouncer.
- **Repeatability**: The attack is continuous as long as the target transcript remains active. Key transcripts are long-lived by design.
- **Practical amplification**: The test `create_opening_with_nonce` (lines 1833, 1842) confirms the codebase already supports generating multiple structurally-identical openings with different content hashes, each stored as a distinct pool entry.

## Recommendation

1. **Treat "no complaint" as invalid for active transcripts**: When `Action::Process` is returned but no complaint exists in the validated pool, emit `IDkgChangeAction::HandleInvalid` or `RemoveUnvalidated` rather than silently deferring. This aligns with the spec's `Some(false)` return for this case (`spec.md`, lines 1390–1391).

2. **Add a per-peer slot table limit for IDKG**: Replace `SLOT_TABLE_NO_LIMIT` with a bounded value for the IDKG channel in `rs/replica/setup_ic_network/src/lib.rs` (line 291), analogous to `SLOT_TABLE_LIMIT_INGRESS = 50_000` used for ingress.

3. **Add a complaint-existence check to the bouncer**: `IDkgBouncer::compute_bouncer` for openings (`rs/consensus/idkg/src/lib.rs`, lines 611–617) could return `BouncerValue::Unwanted` if no matching complaint exists in the validated pool, preventing complaint-less openings from entering the unvalidated pool at all.

```rust
// In rs/consensus/idkg/src/complaints.rs tests:
fn test_opening_flood_no_complaint() {
    let key_id = fake_ecdsa_idkg_master_public_key_id();
    let id_active = create_transcript_id_with_height(1, Height::from(20));
    
    // No complaint added to validated pool for id_active
    let block_reader = TestIDkgBlockReader::for_complainer_test(
        &key_id,
        Height::new(100),
        vec![TranscriptRef::new(Height::new(10), id_active)],
    );
    
    // Insert N openings with different nonces for the same active transcript
    for nonce in 0..1000u64 {
        let opening = create_opening_with_nonce(id_active, NODE_2, NODE_3, NODE_4, nonce);
        idkg_pool.insert(UnvalidatedArtifact {
            message: IDkgMessage::Opening(opening),
            peer_id: NODE_4,
            timestamp: UNIX_EPOCH,
        });
    }
    
    let active_transcripts = complaint_handler.active_transcripts(&block_reader, &snapshot);
    let change_set = complaint_handler.validate_openings(&idkg_pool, &block_reader, &active_transcripts);
    
    // BUG: change_set is empty — all 1000 openings are silently deferred
    assert_eq!(change_set.len(), 0);
    
    // BUG: purge_artifacts also removes nothing (transcript is active)
    let purge_set = complaint_handler.purge_artifacts(&idkg_pool, &block_reader, &active_transcripts);
    assert_eq!(purge_set.len(), 0);
    
    // All 1000 openings remain in the unvalidated pool indefinitely
}
```
