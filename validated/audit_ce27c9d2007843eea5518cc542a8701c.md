Audit Report

## Title
Permanent DoS of ICRC-1 Index-NG Sync Timer via `ic_cdk::trap` in `process_balance_changes` - (File: rs/ledger_suite/icrc1/index-ng/src/main.rs)

## Summary
`process_balance_changes` calls `ic_cdk::trap` on several arithmetic and data-integrity error conditions. Because `ic_cdk::trap` is a hard abort that bypasses Rust's `Result` machinery, it propagates out of the one-shot timer callback (`build_index`) before `set_build_index_timer` is ever reached, permanently halting block synchronization. The most concrete trigger — an Approve block whose `fee` and `effective_fee` fields are both `None` and no prior fee has been recorded — is explicitly acknowledged in the source as a known mainnet condition.

## Finding Description

`build_index` (line 706) is the timer callback. It calls `fetch_blocks_via_get_blocks` or `fetch_blocks_via_icrc3`, both of which call `append_blocks` → `append_block` → `process_balance_changes`. `build_index` does handle `Result<_, SyncError>` returns (lines 722–741), rescheduling on retriable errors and stopping cleanly on non-retriable ones. However, `process_balance_changes` (line 1021) returns `()` and never propagates errors through `Result`; instead it calls `ic_cdk::trap` directly:

- **Approve, no fee, no prior fee** (lines 1104–1106): `ic_cdk::trap("bug: index is stuck because block with index {block_index} doesn't contain a fee and no fee has been recorded before")` — self-acknowledged stuck state.
- **Debit underflow** (lines 1141–1143): `ic_cdk::trap(...)` when `balance < amount`.
- **Credit overflow** (lines 1149–1151): `ic_cdk::trap(...)` when `balance + amount` overflows.
- **Transfer, no fee** (lines 1067–1070): `ic_cdk::trap(...)` when neither `fee` nor `effective_fee` is set.
- **Burn/Mint amount+fee overflow** (lines 1032–1035): `ic_cdk::trap(...)`.

`append_block` calls `process_balance_changes` at line 939 with no trap guard; the `Ok(())` at line 941 is unreachable once a trap fires. `ic_cdk::trap` aborts the entire message execution, rolls back all state changes made during the callback, and returns control to the IC runtime — the Rust call stack above the trap point is never resumed. Consequently, `set_build_index_timer` (line 846) is never called, and the one-shot timer (`ic_cdk_timers::set_timer`, not `set_timer_interval`) is never rescheduled.

Recovery via canister upgrade is blocked: `post_upgrade` restarts the timer, but the next tick re-encounters the identical block at the same index (state was rolled back), traps again, and the cycle repeats indefinitely.

## Impact Explanation

The ICRC-1 index-ng canister permanently stops syncing blocks from the ledger. All read-path queries — `icrc1_balance_of` (via index), transaction history, and `list_subaccounts` — return stale or incorrect data with no staleness indicator. Any wallet, DeFi protocol, or integration relying on the index for balance or history information receives wrong answers indefinitely. This matches the allowed High impact: **Application/platform-level DoS** of an in-scope financial integration canister (ICRC-1 ledger suite).

## Likelihood Explanation

The Approve-with-no-fee path is the highest-likelihood trigger. The code comment at lines 1092–1095 explicitly states the ledger bug existed and that affected blocks remain on mainnet. Any index-ng instance deployed against such a ledger that has not yet processed those blocks will trap deterministically on first encounter — no attacker action is required beyond the pre-existing ledger state. The debit-underflow and credit-overflow paths require a balance discrepancy between ledger and index, which is less likely on a correctly functioning ledger but becomes reachable if any indexing inconsistency exists. The Transfer-with-no-fee path mirrors the Approve case for historical malformed blocks.

## Recommendation

1. Change `process_balance_changes` to return `Result<(), SyncError>` and propagate errors through `append_block` and `append_blocks` instead of calling `ic_cdk::trap`.
2. In `build_index`, treat non-retriable `SyncError` from block processing as a clean stop with a logged message (the existing `Err` branch already does this for `SyncError` — extend it to cover these cases).
3. For the Approve-with-no-fee case specifically, treat a missing fee as non-fatal (fall back to `Tokens::zero()`) rather than trapping, consistent with the code's own comment that this is a known historical ledger quirk.
4. Add an admin-callable `restart_sync` update method that invokes `set_build_index_timer` directly, so the controller can recover without a full canister upgrade.

## Proof of Concept

1. Deploy an ICRC-1 ledger that has (or will produce) an Approve block where both `fee` and `effective_fee` are `None`, and this is the first Approve block before any Transfer has set `last_fee`.
2. Deploy an ICRC-1 index-ng canister pointing at that ledger.
3. Wait for the index timer to fire (`build_index` → `fetch_blocks_via_get_blocks` → `append_blocks` → `append_block` → `process_balance_changes`).
4. The canister traps at line 1104–1106 with message `"bug: index is stuck because block with index N doesn't contain a fee and no fee has been recorded before"`.
5. Query `status` on the index: `num_blocks_synced` remains at 0 (or its pre-trap value) while the ledger chain grows.
6. Upgrade the canister: `post_upgrade` restarts the timer; the next tick encounters the same block and traps again. The index is permanently stuck until the code is patched.

A deterministic integration test can reproduce this by constructing a `Block<Tokens>` with `Operation::Approve { fee: None, .. }`, `effective_fee: None`, and calling `append_block(0, encoded_block)` with `last_fee` unset in state — the call will trap.     if with_state(|state| state.is_build_index_running) {
        return None;

    mutate_state(|state| {
        state.is_build_index_running = true;
    });
    let _reset_is_build_index_running_flag_guard = guard((), |_| {
        mutate_state(|state| {
            state.is_build_index_running = false;
        });
    });
    let num_indexed = match find_get_blocks_method().await {
        GetBlocksMethod::GetBlocks => fetch_blocks_via_get_blocks().await,
        GetBlocksMethod::ICRC3GetBlocks => fetch_blocks_via_icrc3().await,
    };
    match num_indexed {
        Ok(num_indexed) => {
            let wait_time = compute_wait_time(num_indexed);
            mutate_state(|state| {
                state.last_wait_time = wait_time;
            });
            log!(P1, "Indexed: {} waiting : {:?}", num_indexed, wait_time);
            set_build_index_timer(wait_time);
        }
        Err(error) => {
            log!(P0, "{}", error.message);
            ic_cdk::eprintln!("{}", error.message);
            if error.retriable {
                let wait_time = with_state(|state| state.last_wait_time);
                set_build_index_timer(wait_time);
            } else {
                log!(P0, "Stopping the indexing timer.");
                ic_cdk::eprintln!("Stopping the indexing timer.");
            }
        }
    };

fn debit(block_index: BlockIndex64, account: Account, amount: Tokens) {
