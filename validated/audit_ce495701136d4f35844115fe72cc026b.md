Audit Report

## Title
Unbounded Per-Peer Slot Table Growth via `SLOT_TABLE_NO_LIMIT` Enables Byzantine Node to OOM Honest Replicas — (`rs/p2p/consensus_manager/src/receiver.rs`)

## Summary
`SLOT_TABLE_NO_LIMIT` is set to `usize::MAX` and is passed as the `slot_limit` for consensus, certification, DKG, IDKG, and HTTPS outcalls channels. The per-peer guard `peer_slot_table_len < self.slot_limit` is effectively disabled, allowing a single authenticated Byzantine subnet member to send arbitrarily many slot updates with unique `SlotNumber` values, causing unbounded growth of the slot table, `active_assembles`, and live async tasks, ultimately leading to OOM and replica crash.

## Finding Description
`SLOT_TABLE_NO_LIMIT` is defined as `usize::MAX` at `rs/replica/setup_ic_network/src/lib.rs:75` and is passed to `abortable_broadcast_channel` for consensus (both `HASHES_IN_BLOCKS_ENABLED` branches at lines 237 and 246), certification (line 268), DKG (line 279), IDKG (line 291), and HTTPS outcalls (line 303). Only ingress uses the finite `SLOT_TABLE_LIMIT_INGRESS = 50_000` at line 257.

In `handle_slot_update_receive` (`receiver.rs:373`), the guard `Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit` is the sole mechanism preventing unbounded per-peer slot table growth. With `slot_limit = usize::MAX`, this condition is always true in practice — a `HashMap` cannot hold `usize::MAX` entries before the process OOMs.

For each new vacant slot entry where `to_add = true`, the code at lines 399–419 inserts into `active_assembles` and spawns a new `process_slot_update` task. Each spawned task holds a `watch::Receiver<PeerCounter>` and blocks at `peer_rx.wait_for(|p| p.is_empty())` (lines 500, 521). A Byzantine node that never sends a deletion keeps all spawned tasks alive indefinitely.

The topology-based pruning in `handle_topology_update` (lines 547–558) only removes nodes that leave the subnet topology. A Byzantine node that remains a subnet member is never pruned, so its accumulated slot table entries and tasks persist.

The inbound channel has capacity 100 (line 43), which limits burst rate but not total volume — the event loop drains it continuously, processing each message and growing state without bound.

## Impact Explanation
A single Byzantine subnet member can exhaust memory on every honest replica it connects to by sending slot updates with monotonically increasing unique `SlotNumber` values. Each update creates a new `SlotEntry`, a new `active_assembles` entry, and a new live async task. Memory grows without bound until the replica process OOMs and crashes. If enough honest replicas crash, consensus finalization halts subnet-wide. This matches the allowed High impact: **Application/platform-level DoS, crash, consensus blocking, or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000).

## Likelihood Explanation
The attacker must be an authenticated subnet member (TLS-authenticated P2P), placing this within the standard BFT fault model (below the consensus fault threshold). No cryptographic break, no majority, and no privileged access beyond subnet membership is required. The attack is deterministic, requires no timing or race conditions, and is repeatable. The attacker simply sends crafted P2P slot update messages with incrementing `SlotNumber` values.

## Recommendation
Replace `SLOT_TABLE_NO_LIMIT` for consensus, certification, DKG, IDKG, and HTTPS outcalls channels with a finite per-peer bound derived from the maximum number of in-flight artifacts a legitimate node would ever advertise per artifact type per round. The existing `SLOT_TABLE_LIMIT_INGRESS = 50_000` pattern at `rs/replica/setup_ic_network/src/lib.rs:74,257` demonstrates the correct approach. Additionally, consider adding a per-peer cap on `active_assembles` / `artifact_processor_tasks` entries to provide defense-in-depth.

## Proof of Concept
```rust
// Unit test using the existing ReceiverManagerBuilder harness in receiver.rs
#[tokio::test]
async fn byzantine_unbounded_slot_table_growth() {
    let (mut mgr, _channels) = ReceiverManagerBuilder::new()
        .with_slot_limit(usize::MAX) // SLOT_TABLE_NO_LIMIT
        .build();
    let cancellation = CancellationToken::new();
    for i in 0..1_000_000u64 {
        mgr.handle_slot_update_receive(
            SlotUpdate {
                slot_number: SlotNumber::from(i),   // unique slot per iteration
                commit_id: CommitId::from(i),
                update: Update::Id(i),              // unique artifact ID per iteration
            },
            NODE_1,
            ConnId::from(1),
            cancellation.clone(),
        );
    }
    // Slot table grows to 1_000_000 entries for NODE_1
    assert_eq!(mgr.slot_table.get(&NODE_1).unwrap().len(), 1_000_000);
    // 1_000_000 live async tasks accumulate, never terminated
    assert_eq!(mgr.artifact_processor_tasks.len(), 1_000_000);
    // On real hardware, memory is exhausted before this point
}
```
The existing `slot_table_limit_exceeded` test at `receiver.rs:847` already confirms the guard works correctly when a finite limit is set (`with_slot_limit(2)`), validating that the fix is straightforward.
