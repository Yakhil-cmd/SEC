Audit Report

## Title
Unbounded `ingress_expiries` Vec Allocation via Attacker-Controlled `ingress_start`/`ingress_end` in `construction_payloads` — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary
The `construction_payloads` function in the ICRC-1 Rosetta node builds a `Vec<u64>` of ingress expiries by looping from `ingress_start` to `ingress_end` in steps of approximately `(MAX_INGRESS_TTL - PERMITTED_DRIFT) - INGRESS_INTERVAL_OVERLAP`. An unauthenticated HTTP client can supply `ingress_start=0` and `ingress_end=u64::MAX` in the request metadata. The two existing guards do not bound the window size, so the loop executes approximately 68 million iterations, allocating a proportionally large `Vec` of `SigningPayload` and `EnvelopeContent` objects, exhausting process memory and crashing the Rosetta node.

## Finding Description
In `services.rs`, `ingress_start` and `ingress_end` are taken directly from attacker-supplied metadata with no window-size cap:

```rust
let mut ingress_start = metadata
    .as_ref()
    .and_then(|meta| meta.ingress_start)
    .unwrap_or(now);

let ingress_end = metadata
    .as_ref()
    .and_then(|meta| meta.ingress_end)
    .unwrap_or(ingress_start + ingress_interval);
``` [1](#0-0) 

The two guards that follow only check ordering and minimum future validity — neither bounds the window size: [2](#0-1) 

With `ingress_start=0` and `ingress_end=u64::MAX`, both guards pass. The unbounded loop then executes:

```rust
let mut ingress_expiries = vec![];
while ingress_start < ingress_end {
    ingress_expiries.push(ingress_start + ingress_interval);
    ingress_start +=
        ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
}
``` [3](#0-2) 

The step per iteration is `ingress_interval - INGRESS_INTERVAL_OVERLAP ≈ 270 billion ns`. With `u64::MAX ≈ 1.84 × 10¹⁹ ns`, this yields approximately 68 million iterations. The resulting `ingress_expiries` vec is then passed to `handle_construction_payloads`, which allocates one `EnvelopeContent::Call` (containing a full CBOR-serialized canister call with cloned `canister_method_args`) and one `SigningPayload` per entry: [4](#0-3) 

`ConstructionPayloadsRequestMetadata` accepts `ingress_start` and `ingress_end` as plain `Option<u64>` with no range validation at the type level: [5](#0-4) 

## Impact Explanation
A single unauthenticated HTTP POST to `/construction/payloads` with a crafted JSON body causes the Rosetta node process to OOM-crash. At a conservative 200 bytes per entry, 68 million entries results in approximately 13 GB of heap allocation. The ICRC-1 Rosetta node is a single-process service with no replica redundancy at the process level, resulting in a complete denial of service until manually restarted. This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS of the Rosetta API with concrete user and protocol harm**, as well as **Significant Rosetta infrastructure security impact**.

## Likelihood Explanation
The Rosetta API is a public HTTP endpoint with no authentication. The malicious request is a single, small JSON payload requiring no special privileges, keys, or network position. The attack is trivially repeatable to prevent recovery after restart. Any external actor can trigger this.

## Recommendation
Add an explicit cap on the ingress window immediately after computing `ingress_start` and `ingress_end`, before the loop:

```rust
const MAX_INGRESS_WINDOW: u64 = 24 * 3600 * 1_000_000_000; // 24 hours in nanoseconds
if ingress_end.saturating_sub(ingress_start) > MAX_INGRESS_WINDOW {
    return Err(Error::processing_construction_failed(
        &"Ingress window exceeds maximum allowed duration of 24 hours"
    ));
}
```

This bounds the loop to at most `MAX_INGRESS_WINDOW / ingress_interval ≈ 320` iterations, matching the intended design (24 h / ~270 s ≈ 320 envelopes) as documented in the comment at `types.rs` line 44. [6](#0-5) 

## Proof of Concept
```rust
#[test]
fn test_unbounded_ingress_window_oom() {
    use std::time::SystemTime;
    use crate::construction_api::types::ConstructionPayloadsRequestMetadata;

    let metadata = ConstructionPayloadsRequestMetadata {
        ingress_start: Some(0),
        ingress_end: Some(u64::MAX),
        created_at_time: None,
        memo: None,
    };

    // This call should return an error, but instead loops ~68M times
    let result = construction_payloads(
        vec![/* valid transfer operation */],
        Some(metadata),
        &candid::Principal::anonymous(),
        vec![/* valid public key */],
        SystemTime::now(),
    );

    assert!(result.is_err(), "Expected error for unbounded ingress window");
}
```

With `ingress_start=0` and `ingress_end=u64::MAX`, both existing guards at lines 148–158 pass, and the process exhausts memory before returning. The fix causes the function to return an error immediately.
