Audit Report

## Title
Unbounded JoinSet Task Spawning and Fatal `AlreadyInserted` Assembly Abort via Crafted `StrippedBlockProposal` Duplicate Entries — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/assembler.rs`)

## Summary
A Byzantine peer below the consensus fault threshold can respond to a stripped-block fetch with a `StrippedBlockProposal` whose `ingress_messages` repeated field contains an arbitrarily large number of duplicate `SignedIngressId` entries. Because `TryFrom<pb::StrippedBlockProposal>` applies no length cap, `BlockProposalAssembler::new` copies the list verbatim, and `assemble_message` spawns one unbounded `JoinSet` task per entry, this causes a burst of Tokio tasks proportional to the duplicate count. Additionally, once the first duplicate task fills a slot, every subsequent task for the same ID returns `InsertionError::AlreadyInserted`, which the caller treats as fatal and returns `AssembleResult::Unwanted`, aborting the entire assembly attempt.

## Finding Description

**No size guard at deserialization:**
`TryFrom<pb::StrippedBlockProposal> for StrippedBlockProposal` in `rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs` lines 100–108 collects `ingress_messages` from the protobuf repeated field with no length check:

```rust
ingress_messages: value
    .ingress_messages
    .into_iter()
    .map(SignedIngressId::try_from)
    .collect::<Result<Vec<_>, _>>()?,
```

The constant `MAX_INGRESS_MESSAGES_PER_BLOCK = 1000` in `rs/limits/src/lib.rs` line 78 is enforced only during ingress payload validation in `ingress_selector.rs`, not here.

**Unbounded task spawning:**
`BlockProposalAssembler::new` (`assembler.rs` lines 572–588) maps the raw list into `Vec<(SignedIngressId, Option<SignedIngress>)>` without deduplication. `missing_stripped_messages()` (lines 591–598) returns every entry whose slot is `None` — all of them on first call. `assemble_message` (lines 214–226) spawns one `get_or_fetch` task per returned ID with no cap. Each task that misses the local pool enters `download_stripped_message` (`download.rs` lines 291–376), which loops indefinitely with exponential backoff (`with_max_elapsed_time(None)`) until the `JoinSet` is dropped.

**Fatal `AlreadyInserted` on duplicate completion:**
`try_insert` (`assembler.rs` lines 435–454) uses a linear `find` that always locates the *first* matching slot. After the first duplicate task fills it, every subsequent task for the same ID hits the `is_some()` branch and returns `InsertionError::AlreadyInserted`. The caller at lines 262–271 treats any `InsertionError` as fatal and returns `AssembleResult::Unwanted`, dropping the `JoinSet` and aborting all remaining tasks — but only after the burst of tasks has already been spawned and has begun consuming resources.

The comment in `try_insert` ("at most 1000 elements") documents the assumed invariant but it is never enforced at the deserialization boundary.

**Reachable exploit path:**
A Byzantine peer observes an honest proposer's `ConsensusMessageId` (which the bouncer will accept), advertises it, and when the victim fetches the stripped body, responds with a crafted `pb::StrippedBlockProposal` containing N duplicate `SignedIngressId` entries. The deserialization succeeds, N tasks are spawned, and the first duplicate completion triggers `AssembleResult::Unwanted` for that assembly attempt.

## Impact Explanation

This matches **High ($2,000–$10,000): Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS.**

With N = 10,000 duplicate entries, N Tokio tasks are spawned simultaneously. Each task that misses the local pool enters an indefinite download loop, holding a thread-pool slot and heap allocation until the `JoinSet` is dropped. The `JoinSet` is only dropped when `AlreadyInserted` fires (after the first duplicate task completes) or when the bouncer fires `Unwanted` (refresh period 3 s). During this window, the Tokio thread pool and heap are saturated, delaying or stalling block assembly at that height for the targeted replica. The attack is repeatable: the Byzantine peer can re-advertise the same or different block IDs to sustain the pressure. The `AlreadyInserted` path additionally aborts the assembly attempt for the targeted block, contributing to finalization delay if the Byzantine peer is the primary advertiser.

## Likelihood Explanation

- Requires one Byzantine peer below the fault threshold — a realistic assumption in the ICP threat model.
- The peer need not be the block proposer; it only needs to advertise a `ConsensusMessageId` observed from an honest proposer and respond with a crafted stripped body.
- No cryptographic material needs to be forged; the `unstripped_consensus_message_id` hash is taken from the honest block and the duplicate list is purely in the stripped metadata.
- The attack is fully local to the targeted replica, leaves no on-chain trace, and is repeatable at every block height.

## Recommendation

1. **Enforce a size cap at deserialization:** In `TryFrom<pb::StrippedBlockProposal>`, reject any `ingress_messages` list whose length exceeds `MAX_INGRESS_MESSAGES_PER_BLOCK` (and similarly `stripped_idkg_dealings` vs. the per-block dealing limit).
2. **Deduplicate before spawning:** In `BlockProposalAssembler::new` or `missing_stripped_messages`, deduplicate the ID list before returning it to the task-spawning loop.
3. **Treat `AlreadyInserted` as non-fatal:** Rather than returning `AssembleResult::Unwanted` on `AlreadyInserted`, log a warning and continue — a duplicate fetch result should not abort a valid assembly.

## Proof of Concept

```rust
// Construct a StrippedBlockProposal proto with 10_000 copies of the same SignedIngressId.
let mut proto = pb::StrippedBlockProposal::default();
proto.pruned_block_proposal = Some(valid_pruned_block_proto());
proto.unstripped_consensus_message_id = Some(valid_consensus_message_id_proto());
let dup_id = fake_signed_ingress_id_proto();
proto.ingress_messages = vec![dup_id; 10_000];

// Deserialize — succeeds with no error.
let stripped = StrippedBlockProposal::try_from(proto).unwrap();

// BlockProposalAssembler::new creates 10_000 slots.
let assembler = BlockProposalAssembler::new(stripped);
// missing_stripped_messages() returns 10_000 IDs.
assert_eq!(assembler.missing_stripped_messages().len(), 10_000);

// assemble_message spawns 10_000 JoinSet tasks.
// The first task to complete fills slot 0; the second returns AlreadyInserted
// → AssembleResult::Unwanted, aborting the assembly and dropping the JoinSet.
// During the window between spawn and drop, 10_000 tasks compete for thread-pool slots.
```

A deterministic unit test can be written by extending the existing `stripped_message_insertion_existing_fails_test` in `assembler.rs` to verify that `try_insert_stripped_message` called twice with the same ID returns `AlreadyInserted`, and a separate integration test can verify that `assemble_message` returns `AssembleResult::Unwanted` when fed a `StrippedBlockProposal` with duplicate entries.
