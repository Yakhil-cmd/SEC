Audit Report

## Title
Unbounded `spawn_blocking` in `state_sync_chunk_handler` Allows Single Byzantine Peer to Exhaust Tokio Blocking Thread Pool — (`rs/p2p/state_sync_manager/src/routes/chunk.rs`)

## Summary
`StateSyncChunkHandler` holds no semaphore or concurrency counter. Every incoming `POST /state-sync/chunk` request unconditionally calls `tokio::task::spawn_blocking` with blocking file I/O. A single TLS-authenticated subnet peer can open up to 1,000 concurrent QUIC bidirectional streams (the configured `MAX_CONCURRENT_BIDI_STREAMS` limit), each triggering one `spawn_blocking` call, saturating Tokio's shared blocking thread pool and stalling other subsystems — including query validation — that also rely on `spawn_blocking`.

## Finding Description

**Root cause — no concurrency guard in `StateSyncChunkHandler`:**

`StateSyncChunkHandler` holds only `_log`, `state_sync`, and `metrics`. There is no `Semaphore`, counter, or rate-limiting field. [1](#0-0) 

Every request unconditionally issues a `spawn_blocking` call wrapping blocking file I/O via `state_sync.chunk()`: [2](#0-1) 

**Contrast with XNet endpoint — which does guard its `spawn_blocking`:**

`XNetEndpoint` uses a `Semaphore` capped at `XNET_ENDPOINT_MAX_CONCURRENT_REQUESTS = 4` and returns `503` on `try_acquire` failure: [3](#0-2) [4](#0-3) 

No equivalent guard exists anywhere in the state sync chunk serving path.

**QUIC stream limit allows 1,000 concurrent streams per peer:**

`MAX_CONCURRENT_BIDI_STREAMS = 1_000` is set per connection: [5](#0-4) 

The stream acceptor loop spawns a new task for every accepted bidirectional stream with no backpressure. The code itself acknowledges this design gap: [6](#0-5) 

**Other subsystems share the same blocking pool:**

Query validation calls `spawn_blocking` on the same Tokio runtime with no semaphore guard: [7](#0-6) 

**Client-side already handles `TOO_MANY_REQUESTS`:**

The response parser in `parse_chunk_handler_response` already maps `StatusCode::TOO_MANY_REQUESTS` to `DownloadChunkError::Overloaded`, confirming the fix path was anticipated: [8](#0-7) 

## Impact Explanation

A single Byzantine subnet peer saturates Tokio's shared blocking thread pool by opening 1,000 concurrent QUIC streams each triggering a `spawn_blocking` disk-I/O call. Once the pool is saturated, query validation `spawn_blocking` calls queue indefinitely, stalling query handling on the target replica. This is a **non-volumetric, application/platform-level DoS** causing subnet availability impact on the targeted replica, matching the High ($2,000–$10,000) impact class: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

The attacker precondition is a valid TLS-authenticated subnet peer — achievable by any Byzantine node below the consensus fault threshold. The exploit requires only opening concurrent QUIC streams using normal state sync protocol behavior. No special protocol knowledge, no external infrastructure, and no victim interaction is required. The missing guard is a straightforward code omission directly reproducible in a unit test.

## Recommendation

Add a bounded `tokio::sync::Semaphore` to `StateSyncChunkHandler`, mirroring the XNet endpoint pattern: [4](#0-3) 

On `try_acquire` failure, return `StatusCode::TOO_MANY_REQUESTS`. The client-side parser already handles this code correctly: [8](#0-7) 

Additionally, consider reducing `MAX_CONCURRENT_BIDI_STREAMS` or enforcing a per-peer application-level concurrency cap in the stream acceptor loop. [5](#0-4) 

## Proof of Concept

1. Implement a mock `StateSyncClient::chunk()` that sleeps for 500 ms (simulating disk I/O latency).
2. From one authenticated peer, open 600 concurrent QUIC streams each sending a valid `StateSyncChunkRequest`.
3. Concurrently, from a second subsystem on the same Tokio runtime, call `tokio::task::spawn_blocking(|| std::thread::sleep(Duration::from_millis(1)))`.
4. Assert the second `spawn_blocking` does not complete within 5 seconds — demonstrating pool starvation.

The `MAX_CONCURRENT_BIDI_STREAMS = 1000` ceiling and the absence of any semaphore in `state_sync_chunk_handler` make this directly reproducible in a unit test without any network infrastructure. [9](#0-8)

pub(crate) async fn state_sync_chunk_handler<T: 'static>(
    State(state): State<Arc<StateSyncChunkHandler<T>>>,
    payload: Bytes,
) -> Result<Bytes, StatusCode> {
    // Parse payload
    let pb::StateSyncChunkRequest { id, chunk_id } =
        pb::StateSyncChunkRequest::decode(payload).map_err(|_| StatusCode::BAD_REQUEST)?;
    let artifact_id: StateSyncArtifactId = id.map(From::from).ok_or(StatusCode::BAD_REQUEST)?;
    let chunk_id = ChunkId::from(chunk_id);

    let jh =
        tokio::task::spawn_blocking(
            move || match state.state_sync.chunk(&artifact_id, chunk_id) {
                Some(data) => {
                    let pb_chunk = pb::StateSyncChunkResponse { data: data.take() };
                    let mut raw = BytesMut::with_capacity(pb_chunk.encoded_len());
                    pb_chunk.encode(&mut raw).expect("Allocated enough memory");
                    let raw = raw.freeze();

                    let compressed = zstd::bulk::compress(&raw, zstd::DEFAULT_COMPRESSION_LEVEL)
                        .expect("Compression failed");
                    state
                        .metrics
                        .compression_ratio
                        .observe(raw.len() as f64 / compressed.len() as f64);
                    Ok(compressed)
                }
                None => Err(StatusCode::NO_CONTENT),
            },
        );
    let data = jh.await.map_err(|_| StatusCode::INTERNAL_SERVER_ERROR)??;

    Ok(data.into())
}
