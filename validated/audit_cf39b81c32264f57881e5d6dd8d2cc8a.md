Audit Report

## Title
Silent ckETH Fund Loss When `erc20_tx_fee <= CKETH_LEDGER_TRANSACTION_FEE` During Failed ckERC20 Withdrawal - (File: `rs/ethereum/cketh/minter/src/main.rs`)

## Summary
In `withdraw_erc20`, after a successful ckETH burn for gas fees, if the subsequent ckERC20 burn fails due to a user error (insufficient funds/allowance/amount), the reimbursement amount is computed as `erc20_tx_fee.checked_sub(CKETH_LEDGER_TRANSACTION_FEE).unwrap_or(Wei::ZERO)`. When `erc20_tx_fee <= CKETH_LEDGER_TRANSACTION_FEE` (2,000 Gwei), this yields `Wei::ZERO`, the `if reimbursed_amount > Wei::ZERO` guard is false, no `FailedErc20WithdrawalRequest` event is emitted, and the user's ckETH is permanently burned with no recovery path.

## Finding Description
`CKETH_LEDGER_TRANSACTION_FEE` is defined as `Wei::new(2_000_000_000_000_u128)` at line 59. The `estimate_erc20_transaction_fee()` function at line 545 derives the fee purely from current Ethereum gas prices with no enforced floor — it calls `gas_fee_estimate.to_price(CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT).max_transaction_fee()` with no minimum bound. During low-gas-price conditions, `erc20_tx_fee` can be at or below 2,000 Gwei.

The two-step burn sequence at lines 448–535:
1. ckETH burn succeeds (line 460), consuming `erc20_tx_fee` from the user.
2. ckERC20 burn fails with `InsufficientFunds`, `AmountTooLow`, or `InsufficientAllowance` (lines 509–513).

The reimbursement path at lines 511–515:
```rust
LedgerBurnError::InsufficientFunds { .. }
| LedgerBurnError::AmountTooLow { .. }
| LedgerBurnError::InsufficientAllowance { .. } => erc20_tx_fee
    .checked_sub(CKETH_LEDGER_TRANSACTION_FEE)
    .unwrap_or(Wei::ZERO),
// ...
if reimbursed_amount > Wei::ZERO { /* schedule reimbursement */ }
```

When `erc20_tx_fee <= 2_000_000_000_000 wei`, `checked_sub` underflows, `unwrap_or(Wei::ZERO)` returns zero, the guard at line 515 is false, no event is emitted, and no reimbursement is ever scheduled. The ckETH burned in step 1 is permanently destroyed. [1](#0-0) [2](#0-1) [3](#0-2) 

## Impact Explanation
This is a **High** severity ck-token ledger conservation bug. ckETH tokens are destroyed on the ledger without the user receiving any value — no ETH is sent on Ethereum, no ckERC20 is burned, and no reimbursement is minted. This constitutes permanent, irreversible loss of user ckETH funds with concrete harm to ck-token accounting integrity. Per-incident loss is bounded by `erc20_tx_fee` (≤ 2,000 Gwei of ckETH), which is small individually but accumulates across users and is fully unrecoverable. This matches: *"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."*

## Likelihood Explanation
Two concurrent conditions are required: (1) Ethereum gas prices low enough that `erc20_tx_fee ≤ 2,000 Gwei` — rare on mainnet but historically observed and common on testnets; (2) the ckERC20 burn fails due to user error (insufficient balance or allowance). An unprivileged user can deliberately trigger condition (2) by calling `withdraw_erc20` with a valid ckETH approval but no ckERC20 allowance. No special privileges are required. The attack surface is the public `withdraw_erc20` update endpoint.

## Recommendation
Remove the `> Wei::ZERO` guard and always schedule a reimbursement when the ckERC20 burn fails. The penalty deduction of `CKETH_LEDGER_TRANSACTION_FEE` should only apply when the result is positive; otherwise the full `erc20_tx_fee` should be returned:

```rust
let reimbursed_amount = match &ckerc20_burn_error {
    LedgerBurnError::TemporarilyUnavailable { .. } => erc20_tx_fee,
    LedgerBurnError::InsufficientFunds { .. }
    | LedgerBurnError::AmountTooLow { .. }
    | LedgerBurnError::InsufficientAllowance { .. } => {
        erc20_tx_fee.checked_sub(CKETH_LEDGER_TRANSACTION_FEE)
            .unwrap_or(erc20_tx_fee) // return full amount if penalty would underflow
    }
};
// Always schedule reimbursement — remove the `> Wei::ZERO` guard
let reimbursement_request = ReimbursementRequest { ... };
mutate_state(|s| process_event(s, EventType::FailedErc20WithdrawalRequest(reimbursement_request)));
``` [4](#0-3) 

## Proof of Concept
1. Set up a local replica or PocketIC test with the ckETH minter.
2. Configure the gas fee oracle to return a near-zero base fee such that `estimate_erc20_transaction_fee()` returns a value ≤ `2_000_000_000_000 wei` (e.g., `1_000_000_000_000 wei`).
3. Approve the minter to burn `1_000_000_000_000 wei` of ckETH from the test user's account, but grant **zero** ckERC20 allowance.
4. Call `withdraw_erc20(amount=X, ckerc20_ledger_id=..., recipient=...)`.
5. Observe: ckETH burn succeeds (ledger block emitted), ckERC20 burn fails with `InsufficientAllowance`.
6. Assert: no `FailedErc20WithdrawalRequest` event is recorded in minter state, user's ckETH balance is reduced by `1_000_000_000_000 wei`, and no reimbursement mint ever occurs.
7. The invariant `total_ckETH_supply_before == total_ckETH_supply_after + ETH_sent_on_chain` is violated. [5](#0-4)
