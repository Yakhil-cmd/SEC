**Bouncer logic** (`rs/https_outcalls/consensus/src/gossip.rs`):

The bouncer returns `MaybeWantsLater` for any artifact whose callback ID exceeds `highest_accepted_request_id`: [1](#0-0) 

`highest_accepted_request_id` is `next_callback_id + MAX_NUMBER_OF_REQUESTS_AHEAD` (345 slots). Any ID above that returns `MaybeWantsLater`.

**`should_download` stash behavior** (`rs/p2p/artifact_downloader/src/fetch_artifact/download.rs`): [2](#0-1) 

When `MaybeWantsLater`, the task drops the artifact payload (`artifact.take()`) but the async task itself remains alive, blocked on `bouncer_watcher.changed()`. It does **not** terminate.

**The slot table guard** (`rs/p2p/consensus_manager/src/receiver.rs`): [3](#0-2) 

New slot entries are only rejected when `peer_slot_table_len >= self.slot_limit`. With `SLOT_TABLE_NO_LIMIT` (effectively `usize::MAX`), this guard is never triggered for https-outcalls.

**Task spawning per unique artifact ID** (`rs/p2p/consensus_manager/src/receiver.rs`): [4](#0-3) 

Each new unique artifact ID (not already in `active_assembles`) spawns a new `process_slot_update` task. The `active_assembles` map deduplicates by ID, but a Byzantine peer using distinct IDs per slot bypasses this.

**Task lifetime**: The `process_slot_update` task races three futures: [5](#0-4) 

The `all_peers_deleted_artifact` branch only fires if the advertising peer retracts. A Byzantine peer that never retracts keeps all tasks alive indefinitely.

## Verdict

### Title
Unbounded Async Task Accumulation via `MaybeWantsLater` Stash with `SLOT_TABLE_NO_LIMIT` for HTTPS Outcalls — (`rs/https_outcalls/consensus/src/gossip.rs`, `rs/p2p/consensus_manager/src/receiver.rs`)

### Summary
A Byzantine protocol peer can exhaust replica memory by advertising arbitrarily many HTTPS-outcall artifacts with callback IDs above `highest_accepted_request_id`, causing unbounded async task accumulation in the `MaybeWantsLater` stash path, because the https-outcalls artifact type is configured with `SLOT_TABLE_NO_LIMIT`.

### Finding Description
The attack proceeds as follows:

1. A Byzantine peer sends N slot updates, each using a distinct `SlotNumber` (e.g., 0, 1, 2, …, N−1) and a distinct `CanisterHttpResponseId` with a callback ID well above `next_callback_id + MAX_NUMBER_OF_REQUESTS_AHEAD`.
2. Because https-outcalls uses `SLOT_TABLE_NO_LIMIT`, the check at receiver.rs line 373 (`peer_slot_table_len < self.slot_limit`) is always satisfied, so all N entries are inserted into the slot table.
3. Each new unique artifact ID triggers a new `process_slot_update` task (receiver.rs line 407).
4. Each task calls `assemble_message` → `download_artifact` → `should_download`. The bouncer returns `MaybeWantsLater` (gossip.rs line 92), so `artifact.take()` frees the payload but the task itself blocks on `bouncer_watcher.changed()` (download.rs line 187).
5. The bouncer refreshes every 3 seconds (gossip.rs line 100). As long as the advertised IDs remain above `highest_accepted_request_id`, every refresh still returns `MaybeWantsLater`, and the tasks remain stashed.
6. The Byzantine peer never retracts the advertisements, so `all_peers_deleted_artifact` never fires.
7. N tasks accumulate in `artifact_processor_tasks` (a `JoinSet`), each holding a stack frame, a `watch::Receiver`, and the artifact ID. Memory grows linearly with N.

The `active_assembles` deduplication map does not help because the attacker uses a distinct `CanisterHttpResponseId` per slot. The inbound channel capacity of 100 (receiver.rs line 43) is only a transient queue; the event loop drains it continuously, spawning tasks faster than they terminate.

### Impact Explanation
Memory exhaustion on the replica node. As tasks accumulate, the replica's heap grows until the OS OOM-kills the process or the replica becomes unresponsive. This terminates consensus participation for the affected node. If enough nodes in a subnet are targeted simultaneously (each by a different Byzantine peer, or one peer targeting all), the subnet halts. Even targeting a single node degrades subnet fault tolerance.

### Likelihood Explanation
The attacker must be a valid protocol peer (i.e., a node in the subnet or able to establish a QUIC transport connection). A single Byzantine subnet node suffices. The attack requires only sending crafted slot update messages — no cryptographic material is needed beyond normal peer authentication. The `SLOT_TABLE_NO_LIMIT` configuration for https-outcalls is the enabling condition; other artifact types with finite slot limits are not affected.

### Recommendation
1. **Apply a finite slot limit for https-outcalls.** Replace `SLOT_TABLE_NO_LIMIT` with a bound derived from the maximum number of concurrent HTTPS outcall requests (e.g., `MAX_NUMBER_OF_REQUESTS_AHEAD` or a configurable registry parameter).
2. **Bound `MaybeWantsLater` stash size.** In `should_download`, track the number of stashed tasks globally or per-peer and drop (return `false`) when the limit is exceeded, rather than waiting indefinitely.
3. **Add a timeout to stashed tasks.** If a task remains in `MaybeWantsLater` for longer than a few bouncer refresh cycles without transitioning to `Wants`, treat it as `Unwanted`.

for slot in 0..N {
    let high_callback_id = next_callback_id + MAX_NUMBER_OF_REQUESTS_AHEAD + slot + 1;

    peer.send_slot_update(SlotUpdate {

// all blocked in should_download awaiting bouncer_watcher.changed(),
// memory grows O(N) with no upper bound enforced.
