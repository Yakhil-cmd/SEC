Audit Report

## Title
Unbounded Tokio Task Spawning via Attacker-Controlled `stripped_idkg_dealings` Count in `assemble_message` — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/assembler.rs`)

## Summary
A Byzantine block maker below the consensus fault threshold can craft a `StrippedBlockProposal` whose `stripped_idkg_dealings` list contains an arbitrarily large number of fake `IDkgArtifactId::Dealing` entries. Because `assemble_message` spawns one unbounded tokio task per entry with no concurrency cap, and each task enters an infinite retry loop against the transport layer, the attacker can transiently exhaust the replica's async task scheduler and QUIC connection pool for the duration of the consensus round, degrading P2P throughput and potentially blocking consensus progress.

## Finding Description

**Root cause — no count guard at deserialization.**

`TryFrom<pb::StrippedBlockProposal> for StrippedBlockProposal` iterates over `value.stripped_idkg_dealings` with no upper-bound check: [1](#0-0) 

The only per-entry validation is that each `IDkgArtifactId` is of variant `Dealing(_, _)` — a check the attacker trivially satisfies by constructing syntactically valid but semantically fake dealing IDs. There is no equivalent of the NiDKG `TooManyDealings` guard for IDKG chain-key dealings.

**Unbounded `JoinSet` spawn.**

After deserialization, `assemble_message` calls `assembler.missing_stripped_messages()` and unconditionally spawns one tokio task per entry with no semaphore or concurrency cap: [2](#0-1) 

**Infinite retry loop per task.**

Each spawned task calls `download_stripped_message`, which is configured with `with_max_elapsed_time(None)` and loops forever until a successful response is received: [3](#0-2) [4](#0-3) 

For fake dealing IDs that exist in no pool, `get_or_fetch` falls through to `download_stripped_message` immediately, and each task issues repeated transport RPCs (initial interval 5 s, max 120 s) until the bouncer fires. [5](#0-4) 

**Bouncer is the only cleanup path.**

Tasks are aborted only when `join_set` is dropped, which happens when the bouncer marks the block `Unwanted` (i.e., when the height is finalized). The bouncer refresh period is 3 seconds: [6](#0-5) [7](#0-6) 

The `try_assemble()` ID-mismatch check fires only after all tasks complete or the bouncer fires — by which time the resource exhaustion has already occurred. [8](#0-7) 

**No transport-layer size limit.**

The axum router serving stripped artifact RPCs explicitly disables the body size limit, removing a potential backstop: [9](#0-8) 

## Impact Explanation

During the window between when the Byzantine block is advertised and when its height is finalized (typically several seconds, up to tens of seconds under slow consensus), N tasks — one per crafted `stripped_idkg_dealings` entry — each hold a tokio task slot and issue repeated transport RPCs. With N in the thousands (protobuf imposes no per-field count limit), this saturates the QUIC connection pool and increases CPU scheduling overhead, degrading the replica's ability to process other P2P messages, consensus artifacts, and ingress. The attack is repeatable every consensus round. This matches the **High** impact class: *Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS* ($2,000–$10,000).

## Likelihood Explanation

The attacker must be a valid block maker for the current round — a Byzantine node below the fault threshold (f < n/3). Such a node can produce a legitimate `BlockProposal` (so `unstripped_consensus_message_id` passes the bouncer), then craft a `StrippedBlockProposal` with the correct block hash but an inflated `stripped_idkg_dealings` list containing thousands of syntactically valid but fake `IDkgArtifactId::Dealing` entries. The attack requires no victim mistakes, no external compromise, and no special tooling beyond the ability to serialize a protobuf message. It is repeatable every round for as long as the Byzantine node remains a block maker.

## Recommendation

Add a maximum-count guard during deserialization of `StrippedBlockProposal`, analogous to the NiDKG `TooManyDealings` check. A reasonable bound is the subnet size (number of nodes), since each transcript can have at most one dealing per dealer. Reject any `StrippedBlockProposal` whose `stripped_idkg_dealings.len()` exceeds this bound before any tasks are spawned. As defense-in-depth, cap the `JoinSet` concurrency in `assemble_message` with a `tokio::sync::Semaphore`.

## Proof of Concept

Construct a `pb::StrippedBlockProposal` protobuf with:
- `pruned_block_proposal`: a valid (empty-ingress) block proposal proto at the current height signed by the Byzantine node.
- `unstripped_consensus_message_id`: the `ConsensusMessageId` of the Byzantine node's own legitimate block proposal at that height (so the bouncer returns `Wants`).
- `stripped_idkg_dealings`: 1,000 entries, each with a syntactically valid `IDkgArtifactId::Dealing` constructed from arbitrary bytes.

Advertise this artifact to a victim replica. Observe via metrics or `tokio-console` that 1,000 tasks are spawned in `assemble_message`, each issuing repeated transport RPCs to random peers, until the height is finalized (~3–10 seconds later). Measure the spike in tokio task count and transport RPC rate. A deterministic integration test using `PocketIC` or a mock transport can verify the task count without requiring a live subnet.
