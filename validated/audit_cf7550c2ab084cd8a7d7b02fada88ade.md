Audit Report

## Title
Unsanitized Memo Bytes Enable Markdown Injection into ICRC-21 `GenericDisplayMessage` — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary

The `add_memo` function in `ConsentMessage` interpolates caller-controlled memo bytes verbatim into a single-backtick inline code span within a markdown-formatted `GenericDisplayMessage`. A memo beginning with a backtick collapses the opening delimiter into an empty inline-code span, causing all subsequent content to render as raw markdown. Because ICRC-21's security guarantee rests on the ledger—not the dApp—generating the consent message, a malicious dApp can use this path to inject visually authentic fake transaction fields into the ledger-produced consent message shown to the user.

## Finding Description

In `packages/icrc-ledger-types/src/icrc21/responses.rs` at line 288–294, `add_memo` decodes memo bytes to UTF-8 (falling back to hex only on invalid UTF-8) and interpolates the result without any escaping:

```rust
let memo_str = match std::str::from_utf8(memo.as_slice()) {
    Ok(valid_str) => valid_str.to_string(),
    Err(_) => hex::encode(memo.as_slice()),
};
// ...
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
```

The template wraps `memo_str` in a single backtick pair. Under CommonMark, if `memo_str` begins with a backtick, the two adjacent backticks form an empty inline-code span and everything that follows is emitted as raw markdown outside any code fence. An attacker crafts a memo whose UTF-8 content is:

```
`\n\n**Amount:** `1 ICP`\n\n**To:** `real_recipient`\n\n**X:**
```

The resulting `GenericDisplayMessage` contains the real ledger-generated fields followed by injected bold-field lines that are visually identical to the real ones. The entry point is `icrc21_canister_call_consent_message`; the memo is decoded directly from the caller-supplied `arg` blob at `lib.rs` lines 392–394 with no sanitization:

```rust
if let Some(memo) = memo {
    display_message_builder =
        display_message_builder.with_memo(GenericMemo::Icrc1Memo(memo.0));
}
```

No existing guard sanitizes or escapes the memo string before it reaches `add_memo`. The `FieldsDisplayMessage` variant is unaffected because it stores the memo as a structured `Value::Text` field rather than interpolating it into a markdown string (lines 296–298).

## Impact Explanation

ICRC-21 exists precisely to protect users from malicious dApps: the wallet displays a consent message generated by the ledger canister, which the user is expected to trust even when the dApp is adversarial. This vulnerability breaks that guarantee. A malicious dApp can inject fake `**Amount:**` and `**To:**` lines into the ledger-produced consent message. A user who approves based on the injected (attacker-controlled) fields rather than the real fields authorizes a transfer with parameters they did not intend. This constitutes a concrete path to unauthorized transfer of user funds via a ledger-level security mechanism bypass — matching the High bounty impact: *Significant ledger security impact with concrete user or protocol harm* ($2,000–$10,000).

## Likelihood Explanation

- The `icrc21_canister_call_consent_message` endpoint is fully unprivileged; any caller may invoke it with an arbitrary `arg` blob.
- The ICRC-1 memo field accepts up to 32 bytes; a backtick + newline + fake bold fields fits within that limit.
- Valid UTF-8 is the common case for memo content, so the hex fallback does not protect against this.
- The attack requires a malicious dApp — which is exactly the adversary ICRC-21 is designed to defend against.
- Any wallet that renders `GenericDisplayMessage` as markdown (the intended rendering per the ICRC-21 spec) is directly affected.

## Recommendation

Escape backtick characters in `memo_str` before interpolation:

```rust
let escaped = memo_str.replace('`', "\\`");
message.push_str(&format!("\n\n**Memo:**\n`{escaped}`"));
```

For more robust protection against multi-line content, use a fenced code block and escape any triple-backtick sequences inside the memo:

```rust
let escaped = memo_str.replace("```", "\\`\\`\\`");
message.push_str(&format!("\n\n**Memo:**\n```\n{escaped}\n```"));
```

## Proof of Concept

Construct a `TransferArg` with a memo whose bytes are the valid UTF-8 string (≤ 32 bytes):

```
`\n\n**Amount:** `999999 ICP`\n\n**To:** `attacker_account`\n\n**X:**
```

Call `icrc21_canister_call_consent_message` with this `TransferArg` encoded as the `arg` blob. Inspect the returned `GenericDisplayMessage` string. The message will contain the real ledger-generated fields followed by the injected `**Amount:**` and `**To:**` lines rendered identically to the real fields by any CommonMark-compliant renderer. A unit test can assert that the returned string contains the injected bold-field text outside any code span, confirming the injection. A PocketIC integration test can further verify that a wallet rendering the message displays two sets of transaction fields.
