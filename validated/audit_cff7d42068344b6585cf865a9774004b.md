Audit Report

## Title
ckERC20 Deposit Helper Contracts Emit Caller-Supplied Amount Instead of Actual Received Amount, Enabling Over-Minting for Fee-on-Transfer Tokens - (File: rs/ethereum/cketh/minter/ERC20DepositHelper.sol, rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol)

## Summary

Both ckERC20 helper smart contracts emit the caller-supplied `amount` parameter in the deposit event rather than the actual balance delta received by the minter address. The IC ckETH minter canister scrapes these events and mints ckERC20 tokens equal to the event-reported `amount` with no on-chain balance verification. For any ERC-20 token that charges a transfer fee, the minter's Ethereum address receives fewer tokens than the event records, but the IC minter mints the full event amount, creating unbacked ckERC20 supply and breaking the 1:1 peg.

## Finding Description

**Root cause in `ERC20DepositHelper.sol`:**

`CkErc20Deposit.deposit` calls `safeTransferFrom(msg.sender, cketh_minter_main_address, amount)` and then emits `ReceivedErc20(erc20_address, msg.sender, amount, principal)` using the caller-supplied `amount` directly, with no balance delta measurement. [1](#0-0) 

**Root cause in `DepositHelperWithSubaccount.sol`:**

`CkDeposit.depositErc20` identically calls `safeTransferFrom(msg.sender, minterAddress, amount)` and emits `ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount)` using the caller-supplied `amount`. [2](#0-1) 

Neither contract measures `balanceOf(minterAddress)` before and after the transfer to compute the actual received amount.

**IC minter log parser reads event data directly as mint value:**

`ReceivedErc20LogParser::parse_log` extracts `value_bytes` from `entry.data` (the raw event data field) and constructs `value: Erc20Value::from_be_bytes(value_bytes)` — this is the event-emitted amount, not a verified on-chain balance. [3](#0-2) 

**IC minter mints exactly `event.value()` with no reconciliation:**

The `mint()` function in `deposit.rs` calls `client.transfer(TransferArg { amount: event.value(), ... })`, minting exactly the event-reported value to the beneficiary. [4](#0-3) 

**`register_deposit_events` has no balance verification step:**

The function only checks the blocklist before calling `process_event(s, event.into_deposit())`. There is no step that queries the minter's actual ERC-20 balance before and after to compute the real received amount. [5](#0-4) 

**Internal balance tracker is also corrupted:**

`update_balance_upon_deposit` adds `event.value` (the event-reported amount, not the actual received amount) to the internal `erc20_balances` tracker, so the minter's own accounting is also inflated. [6](#0-5) 

**No existing guard anywhere in the codebase** checks for fee-on-transfer behavior, measures actual balance deltas, or rejects tokens with transfer fees. The `ckerc20_tokens` whitelist enforced by the minter only filters by contract address — it does not validate token properties. [7](#0-6) 

## Impact Explanation

**Vulnerability class: illegal minting / ck-token protocol insolvency.**

For any whitelisted ERC-20 token that charges a transfer fee (fee-on-transfer), every deposit call causes the IC minter to mint more ckERC20 than the ERC-20 tokens it actually holds. Over time, the total ckERC20 supply exceeds the minter's ERC-20 reserve. When users attempt to withdraw ckERC20 back to ERC-20, the minter cannot fulfill all withdrawal requests — the last withdrawers receive nothing. This is a direct ledger conservation break for the ckERC20 token.

This matches the allowed High impact: **"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm"** — specifically illegal minting and peg break for an in-scope ck-token. At scale (e.g., USDT activating its dormant fee mechanism), the impact could reach Critical thresholds ("illegal minting … involving in-scope chain-key/ledger assets … over $1M").

**Severity: High** ($2,000–$10,000), with potential escalation to Critical depending on which token is affected and the total ckERC20 supply at the time.

## Likelihood Explanation

The minter enforces a whitelist of supported ERC-20 tokens via NNS governance proposals. However:

1. **USDT (already whitelisted)** has a fee mechanism in its contract (`basisPointsRate`, `maximumFee`) that is currently set to zero but can be activated by Tether's admin at any time without any change to the contract address or ABI. [8](#0-7) 

2. **USDC (already whitelisted)** is a proxy-upgradeable contract controlled by Circle; a future upgrade could introduce transfer fees. [9](#0-8) 

3. A future NNS proposal could add a fee-on-transfer token without the community recognizing the protocol-level risk, since the documentation contains no warning about this class of token.

4. Once a fee-on-transfer token is whitelisted, the exploit path is **fully unprivileged** — any user calling `depositErc20` triggers the over-minting. No attacker capability beyond submitting a normal deposit transaction is required.

**Likelihood: Low-Medium** — requires a whitelisted token to have active fee-on-transfer behavior, but the protocol provides zero defense-in-depth against this scenario, and at least one whitelisted token (USDT) has a dormant fee mechanism.

## Recommendation

Both helper contracts should measure the actual balance delta and emit that as the deposit amount:

```solidity
function depositErc20(address erc20Address, uint256 amount, bytes32 principal, bytes32 subaccount) public {
    require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
    IERC20 erc20Token = IERC20(erc20Address);
    uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    uint256 actualAmount = erc20Token.balanceOf(minterAddress) - balanceBefore;
    emit ReceivedEthOrErc20(erc20Address, msg.sender, actualAmount, principal, subaccount);
}
```

Apply the same pattern to `CkErc20Deposit.deposit` in `ERC20DepositHelper.sol`. This ensures the IC minter only mints ckERC20 equal to the tokens it actually received.

## Proof of Concept

1. USDT's admin activates its transfer fee (e.g., 1 basis point), or a new fee-on-transfer token is added via NNS proposal.
2. Attacker calls `depositErc20(usdtAddr, 1_000_000, principal, subaccount)` on `CkDeposit`.
3. The helper calls `safeTransferFrom(attacker, minterAddress, 1_000_000)`. Due to the fee, minter receives `999_900` tokens.
4. The helper emits `ReceivedEthOrErc20(..., 1_000_000, ...)`.
5. The IC minter scrapes the log, parses `value = 1_000_000` from the event data field, and mints `1_000_000` ckUSDT to the attacker's IC account.
6. Attacker holds `1_000_000` ckUSDT backed by only `999_900` USDT.
7. Repeated deposits accumulate unbacked ckUSDT. The final `100` ckUSDT per deposit cycle are permanently unbacked, breaking the 1:1 peg and causing loss for other ckUSDT holders attempting to withdraw.

**Reproducible test plan:** Deploy a mock ERC-20 with a configurable transfer fee (e.g., 1%) in a PocketIC or local replica integration test. Whitelist it as a ckERC20 token. Call `depositErc20` with `amount = 1_000_000`. Assert that the ckERC20 ledger total supply after minting equals `1_000_000` while the minter's actual ERC-20 balance is only `990_000`. The discrepancy of `10_000` unbacked tokens confirms the over-minting.
