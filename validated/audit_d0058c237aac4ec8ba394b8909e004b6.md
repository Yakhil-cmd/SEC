Audit Report

## Title
Incomplete Redaction in Rate-Limit Canister Exposes Confidential Security-Incident Metadata to Unprivileged Callers - (File: `rs/boundary_node/rate_limits/canister/confidentiality_formatting.rs`)

## Summary
The `ConfigConfidentialityFormatter` and `RuleConfidentialityFormatter` only null out `rule_raw` and `description` for non-disclosed rules, leaving `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` intact and returned to any unauthenticated caller. Because `get_config`, `get_rule_by_id`, and `get_rules_by_incident_id` are `#[query]` methods, they bypass `inspect_message` entirely, and the formatter is the sole access-control gate — one that is demonstrably incomplete. Any principal, including the anonymous principal, can enumerate every confidential security incident UUID stored in the canister.

## Finding Description
**Root cause — incomplete formatter:** `ConfigConfidentialityFormatter::format` (`confidentiality_formatting.rs` L17-28) iterates over rules and, for each with `disclosed_at.is_none()`, sets only `rule.description = None` and `rule.rule_raw = None`. The fields `id` (rule UUID) and `incident_id` are never touched. `RuleConfidentialityFormatter::format` (`confidentiality_formatting.rs` L34-42) has the identical gap.

**Why `inspect_message` does not help:** The hook at `canister.rs` L34-68 checks `called_method == REPLICATED_QUERY_METHOD` (the string `"get_config"`) and traps unauthorized callers — but `inspect_message` is only invoked for ingress (update-path) messages. The three affected methods are declared `#[query]` (`canister.rs` L110, L123, L136), so they are served as non-replicated query calls that never pass through `inspect_message`.

**Exploit flow:**
1. Attacker issues a non-replicated query call to `get_config(null)` as the anonymous principal.
2. `ConfigGetter::get` (`getter.rs` L130-132) resolves `AccessLevel::RestrictedRead` for the anonymous caller.
3. Because the caller is not an authorized viewer, `self.formatter.format(config)` is called (`getter.rs` L146).
4. The formatter sets `is_redacted = true` and nulls `rule_raw`/`description` for non-disclosed rules, but returns every `rule_id` and `incident_id` verbatim.
5. The attacker receives a complete enumeration of all confidential security-incident UUIDs and their version timestamps.

**Confirmed by existing test:** The unit test in `getter.rs` L358-384 explicitly asserts that an unauthorized caller receives `rule_id` and `incident_id` for a non-disclosed rule alongside `rule_raw: None` and `description: None`. The integration test (`rate_limit_canister_tests.rs` step 5) likewise asserts `rules[0].incident_id == incident_id_1` for an unauthorized principal.

## Impact Explanation
This is a significant boundary-node infrastructure security impact with concrete protocol harm. The rate-limit canister's stated purpose is to keep security-incident rules confidential until `disclose_rules` is called; that invariant is broken. Any observer can enumerate: (1) the existence and count of all active confidential security incidents, (2) the UUID of each incident (correlatable with forum posts, GitHub issues, or deployment timing), and (3) the exact IC versions at which each confidential rule was introduced and retired. This matches the allowed High impact category: significant boundary/API infrastructure security impact with concrete user or protocol harm.

## Likelihood Explanation
Trivially high. The `get_config` query endpoint requires no authentication, no cycles beyond the standard query fee, and no prior knowledge. A single `dfx canister call ... --query` suffices. The attack is repeatable at any time by any principal on the IC.

## Recommendation
**Short term:** Extend both formatters to also set `rule.incident_id` to an opaque placeholder (e.g., a HMAC of the real UUID keyed by a canister secret) and optionally zero out `rule.id` for non-disclosed rules when `AccessLevel::RestrictedRead` is in effect. Suppress `added_in_version` and `removed_in_version` for non-disclosed rules as well.

**Long term:** Audit every field of `OutputRule` and `OutputRuleMetadata` for information that could reveal the existence or timing of confidential security incidents. Consider whether the `get_rules_by_incident_id` endpoint should return a `NotFound` error (rather than a redacted response) for incidents that have never been disclosed, to prevent incident-existence probing via direct UUID lookup.

## Proof of Concept
```bash
# Any principal (including anonymous) can call:
dfx canister call <rate_limit_canister_id> get_config '(null)' --query

# Response for an unauthorized caller includes, for every non-disclosed rule:
# { rule_id    = "b97730ac-4879-47f2-9fea-daf20b8d4b64";   <- exposed
#   incident_id = "389bbff8-bffa-4430-bb70-8ce1ea399c07";  <- exposed
#   rule_raw    = null;                                     <- redacted
#   description = null;                                     <- redacted
# }
```
The existing unit test (`getter.rs` L358-384) and integration test (`rate_limit_canister_tests.rs` L154-188, step 5) both assert this exact behavior and serve as a deterministic, zero-setup proof of concept.
