### Title
TOCTOU: Stale Controller Snapshot in Migration Canister Enables Unauthorized Controller Restoration — (`File: rs/migration_canister/src/validation.rs`)

---

### Summary

The Migration Canister's `validate_request` function captures the controller list of both `migrated_canister` and `replaced_canister` at validation time via an async `canister_status` call and stores it as `migrated_canister_original_controllers` / `replaced_canister_original_controllers` in the persisted `Request` struct. This snapshot is used much later — after the full multi-step, timer-driven migration pipeline completes — to restore the canisters' controllers. Between the snapshot and the restoration, any controller can legitimately change the controller set. The migration canister blindly restores the stale snapshot, overwriting those changes. A malicious controller can exploit this to restore their own controller access after being legitimately removed.

---

### Finding Description

**Root cause — stale snapshot captured at validation time:**

In `validate_request`, `check_controllers_and_get_status` makes two sequential async inter-canister calls: first `get_canister_info` (used for the authorization check), then `canister_status` (whose `settings.controllers` field is stored as the `original_controllers`):

```
check_controllers_and_get_status:
  1. get_canister_info(canister_id).await   ← authorization check uses this
  2. canister_status(canister_id).await     ← original_controllers captured from this
```

The captured list is stored in the `Request` struct:

```rust
let mut migrated_canister_original_controllers = migrated_canister_status.settings.controllers;
migrated_canister_original_controllers.retain(|e| *e != canister_self());
let request = Request {
    migrated_canister,
    migrated_canister_original_controllers,   // ← stale snapshot
    ...
};
```

This `Request` is immediately persisted to stable memory via `insert_request(RequestState::Accepted { request })`.

**Use of stale data — controller restoration:**

The `CanisterGuard` locks acquired during validation are dropped at the end of the `migrate_canister` handler. The timer-driven processing pipeline then runs asynchronously through states: `Accepted → ControllersChanged → StoppedAndReady → RenamedReplacedCanister → UpdatedRoutingTable → RoutingTableChangeAccepted → MigratedCanisterDeleted → RestoredControllers`. This pipeline can take minutes to hours.

At the final restoration step, `process_migrated_canister_deleted` uses the stale snapshot unconditionally:

```rust
let controllers = request
    .migrated_canister_original_controllers
    .iter()
    .filter(|x| **x != canister_self())
    .cloned()
    .collect::<Vec<Principal>>();
// ...
set_controllers(request.migrated_canister, controllers, ...).await
```

No re-check of the current controller set is performed before restoration. The same stale list is also used in `process_failed` → `controller_recovery` for failure-path restoration.

**The TOCTOU window:**

The window spans from when `validate_request` returns (guards released) to when `process_migrated_canister_deleted` runs. During this window, any controller can call `update_settings` on the management canister to change the controller set. The migration canister has no mechanism to detect or account for such changes.

**Exploit path:**

1. Mallory and Alice are both controllers of `migrated_canister` and `replaced_canister`. The Migration Canister is also a controller (prerequisite for the service).
2. Mallory calls `migrate_canister(migrated_canister, replaced_canister)`. Validation passes; `original_controllers = [Alice, Mallory]` is captured and persisted.
3. Alice, wanting to remove Mallory, calls `update_settings` on the management canister to set controllers to `[Alice]`. This succeeds because Alice is a controller.
4. The migration pipeline proceeds. `process_accepted` calls `set_exclusive_controller(migrated_canister)` — this succeeds because the Migration Canister is still a controller.
5. The pipeline completes. `process_migrated_canister_deleted` calls `set_controllers(migrated_canister, [Alice, Mallory], ...)`.
6. Mallory's controller access is restored. Alice's legitimate removal is overwritten.

The same logic applies to `replaced_canister_original_controllers` and the failure-path recovery in `process_failed`.

---

### Impact Explanation

A malicious controller (Mallory) can initiate a migration and have their controller access to `migrated_canister` (and `replaced_canister`) permanently restored even after being legitimately removed by another controller. This constitutes an unauthorized state change: the canister's controller set is set to a value that no current controller authorized. On the IC, controller access is equivalent to full administrative control over a canister (install code, stop, delete, read state). Restoring unauthorized controller access is a privilege escalation.

The inverse also holds: a controller added to a canister after validation but before `process_accepted` will be permanently removed after migration completes, causing legitimate loss of controller access.

---

### Likelihood Explanation

The attacker (Mallory) must be a legitimate controller of both canisters — a realistic precondition for anyone using the Migration Canister service. The TOCTOU window is not a tight race: it spans the entire multi-step migration pipeline (minutes to hours), giving Alice ample time to remove Mallory. Mallory does not need to time anything precisely; they simply initiate the migration and wait. The attack is deterministic once initiated, requires no privileged access beyond being a controller, and requires no external dependencies.

---

### Recommendation

Re-fetch the current controller list immediately before calling `set_controllers` in `process_migrated_canister_deleted` (and in `controller_recovery` for the failure path), rather than using the snapshot captured at validation time. Specifically, call `canister_status` or `canister_info` at restoration time to obtain the live controller set, and use that as the restoration target. If the goal is to restore the pre-migration controllers, the migration canister should record the controller list at the moment it becomes the exclusive controller (in `process_accepted`), not at validation time, since that is the last authoritative snapshot before the migration canister takes exclusive control.

---

### Proof of Concept

1. Deploy the Migration Canister with Mallory and Alice as controllers of `migrated_canister` and `replaced_canister`.
2. Mallory calls `migrate_canister`. Observe `original_controllers = [Alice, Mallory]` stored in `REQUESTS`.
3. Alice calls `update_settings` on `ic:00` to set `migrated_canister` controllers to `[Alice]` only.
4. Advance time to trigger all timer-based processing steps.
5. After migration completes, query `migrated_canister`'s controllers. Observe `[Alice, Mallory]` — Mallory's access is restored despite Alice's removal.

    canister_id: Principal,

            return Err(ValidationError::CallFailed {
                reason: "Call to management canister (`canister_info`) failed. Try again later."
                    .to_string(),
            });
        }
        ProcessingResult::FatalFailure(()) => {
            return Err(ValidationError::CanisterNotFound {
                canister: canister_id,
            });

        return Err(ValidationError::NotController {
            canister: canister_id,

            canister: canister_id,

            reason: "Call to management canister (`canister_status`) failed. Try again later."
                .to_string(),
        });
    };

# Migration Canister

This canister orchestrates the migration of a canister ID from one subnet to another. 

## Technical overview

Canister migration involves several crossnet calls, which makes the process highly asynchronous and prone to temporary failures. In order to be robust and re-entrant, this canister keeps the state of every request as an explicit record in its stable memory. For every state, there is a transition function that attempts to make progress into the next state. If the transition fails, the original state remains as is, so that it can be retried. Transition functions are scheduled regularly with timers. 

### Lifecycle of a request

A user calls `migrate_canister`, providing the canister ID to migrate and the canister ID to replace. The request is validated using various calls to the registry and management canister. If validation fails, the user receives an error response. If validation suceeds, the result is a `Request` which uniquely identifies a migration request. This data is wrapped in a `RequestState` enum (in the `Accepted` variant) and saved in a stable set `REQUESTS`. A timed function picks up all items in `REQUESTS` that are in state `Accepted` and attempts to make progress into the next state.

As a `Request` progresses through the states, the corresponding `RequestState` variants accumulate data that is necessary to perform some future steps. E.g., `RequestState::StoppedAndReady` contains a `stopped_since` timestamp which will be necessary for a later transition to `RequestState::RestoredControllers`, which may only happen once 5 minutes since `stopped_since` have passed. 

This way, all data necessary to make progress is always available in the `RequestState` itself, making it easy to pick up and retry if it fails for a while. 

Despite the upfront request validation, not all error conditions can be ruled out by it or fixed by retrying. E.g., when the Migration Canister sets itself as the only controller, it might no longer be controller itself (if the user has manipulated the canister between validation and this moment). In such cases, the request enters a fail state, which enables the Migration Canister to clean up any undesirable situations (like the user having no control over their canisters). After cleanup, or after a request has been processed successfully, the `Request` is removed from the REQUESTS set and an event is recorded in `HISTORY`.
