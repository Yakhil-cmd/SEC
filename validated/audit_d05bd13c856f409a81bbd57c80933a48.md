Audit Report

## Title
User-Supplied Nonce in NNS Governance `spawn` Enables Subaccount Griefing via `claim_or_refresh_neuron_by_memo_and_controller` - (File: rs/nns/governance/src/governance.rs)

## Summary
When `spawn` is called with an explicit nonce, the child neuron's subaccount is computed deterministically via `compute_neuron_staking_subaccount(child_controller, nonce)` using the `"neuron-stake"` domain. The public `claim_or_refresh_neuron_by_memo_and_controller` endpoint uses the identical hash domain and accepts an arbitrary `controller` principal from any caller. An attacker who knows the intended `(child_controller, nonce)` pair can pre-stake a neuron at that subaccount, causing the victim's `spawn` to fail with `SubaccountAlreadyExists`. The victim's maturity is not converted and they must retry with a different nonce.

## Finding Description
In `governance.rs` at L2677–2688, when `spawn.nonce` is `Some(nonce_val)`, the subaccount is computed as `ledger::compute_neuron_staking_subaccount(child_controller, nonce_val)` and then passed to `ensure_subaccount_available`, which returns `Err(NeuronStoreError::SubaccountAlreadyExists)` immediately on collision with no retry logic (by design, since the same nonce always produces the same subaccount).

In `governance.rs` at L5852–5871, `claim_or_refresh_neuron_by_memo_and_controller` computes `ledger::compute_neuron_staking_subaccount(controller, memo)` using the same `"neuron-stake"` domain (`ledger.rs` L6–7). Critically, `controller` is set to `memo_and_controller.controller.unwrap_or(*caller)` at L5858 — any caller may supply any `controller` principal. The function then calls `claim_neuron(subaccount, controller, ...)` which creates a neuron owned by the supplied `controller` after verifying ledger balance (L6030–6045). No check verifies that the caller is the controller.

Attack path:
1. Attacker computes `subaccount = compute_neuron_staking_subaccount(child_controller, nonce)`.
2. Attacker transfers ≥ `neuron_minimum_stake_e8s` ICP to `AccountIdentifier::new(GOVERNANCE_CANISTER_ID, Some(subaccount))`.
3. Attacker calls `manage_neuron` → `ClaimOrRefresh { by: MemoAndController { controller: Some(child_controller), memo: nonce } }` from any principal.
4. `claim_neuron` succeeds: a neuron is created at that subaccount with `child_controller` as its controller.
5. Victim calls `spawn` with `nonce = Some(nonce)` → `ensure_subaccount_available` returns `SubaccountAlreadyExists` → spawn fails.

The `disburse_to_neuron` path at L2972–2977 uses `compute_neuron_disburse_subaccount_bytes` (domain `"neuron-split"`), which is distinct from `"neuron-stake"` and is not reachable via the claim flow, so that path is unaffected.

## Impact Explanation
This is a targeted griefing attack against NNS governance's `spawn` operation. The victim's maturity conversion fails and must be retried with a different nonce or without a nonce. No victim funds are permanently lost (maturity remains in the parent neuron), but the attacker can repeatedly block any predictable nonce for a given controller, degrading the usability of the spawn feature. This matches the Medium impact class: a constrained attack with meaningful NNS governance security impact, where the attacker bears a real cost (minimum ICP stake per blocked nonce) and the victim suffers a denial of the intended operation.

## Likelihood Explanation
The IC has no public mempool, so real-time frontrunning is not possible. However, the attack does not require frontrunning — it only requires knowing `(child_controller, nonce)` before the victim's spawn executes. This is realistic when: (a) the nonce is predictable (sequential integers, timestamps, or a fixed constant — the test suite itself uses `NONCE = 12345`); (b) the `child_controller` is the parent neuron's controller, which is public on-chain; or (c) the nonce is disclosed out-of-band. The cost barrier (minimum ICP stake per attempt) limits scale but does not eliminate the attack for a motivated actor.

## Recommendation
1. **Remove the user-supplied nonce path for `spawn` entirely**: when no nonce is provided, a random subaccount is already generated safely via `new_neuron_subaccount` with collision retry. The deterministic nonce path adds no user-facing benefit that cannot be achieved by recording the returned neuron ID.
2. If deterministic subaccounts are required, use a **spawn-specific domain** (e.g., `"neuron-spawn"`) distinct from `"neuron-stake"` so that `claim_or_refresh_neuron_by_memo_and_controller` cannot collide with spawn-reserved subaccounts.
3. Alternatively, derive the child subaccount from `(parent_neuron_id, nonce)` rather than `(child_controller, nonce)`, making the subaccount unguessable without knowledge of the parent neuron ID.

## Proof of Concept
```rust
// Setup: Alice owns a neuron with controller X. She wants to spawn
// a child neuron using nonce N = 0.

// Step 1 (Attacker Bob): Compute the target subaccount.
let subaccount = compute_neuron_staking_subaccount(X, 0);
let destination = AccountIdentifier::new(GOVERNANCE_CANISTER_ID, Some(subaccount));

// Step 2 (Attacker Bob): Fund the subaccount on the ICP ledger.
ledger.send(SendArgs {
    memo: Memo(0),
    amount: Tokens::from_e8s(neuron_minimum_stake_e8s),
    to: destination,
    ..
});

// Step 3 (Attacker Bob): Claim a neuron at that subaccount for controller X.
// Bob can be any principal — no authorization check on the controller field.
governance.manage_neuron(ManageNeuron {
    command: Some(ClaimOrRefresh { by: Some(MemoAndController {
        controller: Some(X),  // victim's controller
        memo: 0,
    })}),
});
// => Succeeds. Neuron created at subaccount with controller = X.

// Step 4 (Victim Alice): Attempt to spawn with nonce = 0.
governance.manage_neuron(ManageNeuron {
    id: alice_neuron_id,
    command: Some(Spawn { new_controller: Some(X), nonce: Some(0), .. }),
});
// => Fails: NeuronStoreError::SubaccountAlreadyExists { subaccount }
// Alice's maturity is not converted; she must retry with a different nonce.
```

A deterministic integration test can reproduce this by: (1) setting up an NNS state machine, (2) having a second principal call `ClaimOrRefresh` with `MemoAndController { controller: victim_principal, memo: 0 }` after funding the subaccount, and (3) asserting that the victim's subsequent `Spawn { nonce: Some(0) }` returns `SubaccountAlreadyExists`.
