### Title
Silent Success on Call to Non-Existent `L2AssetTracker` Skips Base Token Deposit Accounting - (`File: basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

`notify_l2_asset_tracker` calls `L2_ASSET_TRACKER_ADDRESS` via `run_single_interaction` without verifying that a contract is deployed at that address. By ZKsync OS convention, calls to empty (non-existent) accounts return `CallResult::Successful` with empty return data. The `failed` check that follows only catches explicit reverts — it cannot distinguish a genuine successful call from a silent no-op against an empty address. The code itself acknowledges this in a comment but relies on an unenforceable assumption.

---

### Finding Description

In `notify_l2_asset_tracker`, the bootloader calls `handleFinalizeBaseTokenBridgingOnL2` on `L2_ASSET_TRACKER_ADDRESS` for every L1→L2 deposit:

```rust
let failed = resources.with_infinite_ergs(|inf_ergs| {
    let CompletedExecution { result: asset_tracker_result, .. } =
        BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
            system, system_functions, memories,
            &calldata,
            &L2_BASE_TOKEN_ADDRESS,
            &L2_ASSET_TRACKER_ADDRESS,   // ← target
            ...
        )?;
    Ok::<bool, _>(asset_tracker_result.failed())
})?;

if failed { return Err(...); }   // only catches explicit reverts
``` [1](#0-0) 

The bootloader's own call-dispatch logic explicitly states:

```rust
// By convention, calls to empty accounts succeed without any return data
if next_ee_type == ExecutionEnvironmentType::NoEE {
    ...
    return Ok((resources, CallResult::Successful { return_values: ReturnValues::empty() }));
}
``` [2](#0-1) 

So if `L2_ASSET_TRACKER_ADDRESS` has no deployed bytecode, `run_single_interaction` returns `CallResult::Successful`, `asset_tracker_result.failed()` evaluates to `false`, and `notify_l2_asset_tracker` returns `Ok(())` — silently skipping the accounting notification. The code's own doc-comment acknowledges this:

> *"If no contract is deployed at L2AssetTracker, the call succeeds silently (a call to an empty address returns success with no returndata in EVM). However, we are certain that L2AssetTracker is available after the upgrade."* [3](#0-2) 

The assumption is unenforceable in code. Additionally, `read_l1_chain_id` reads `L2_ASSET_TRACKER_ADDRESS` storage slot 154 directly; if the contract is absent, this returns zero, causing `l1_chain_id = 0` to be passed to every `handleFinalizeBaseTokenBridgingOnL2` call even when the contract is later deployed. [4](#0-3) 

---

### Impact Explanation

Every L1→L2 deposit calls `mint_base_token`, which calls `notify_l2_asset_tracker` before `transfer_from_treasury`. If `L2AssetTracker` is absent:

1. The asset tracker records **zero** for the deposit — its cross-chain accounting is permanently incorrect for those deposits.
2. `transfer_from_treasury` still executes, so base tokens **are minted** to the recipient.
3. The mismatch between on-chain token supply and the tracker's recorded state is irreversible for those transactions. [5](#0-4) 

---

### Likelihood Explanation

The vulnerable window is the period between chain genesis (or a protocol upgrade that introduces `L2AssetTracker`) and the moment the contract is actually deployed. Any L1→L2 deposit submitted during that window — by any unprivileged user — triggers the silent-success path. The entry point is a standard L1 priority transaction with `to_mint > 0`, which is fully attacker-controlled. [6](#0-5) 

---

### Recommendation

Add an explicit contract-existence check inside `notify_l2_asset_tracker` before dispatching the call. Read `callee_account_properties.unpadded_code_len` (the same field checked in `call_execute_callee_frame`) for `L2_ASSET_TRACKER_ADDRESS` and return a fatal error if it is zero. This mirrors the pattern already used for CREATE collision detection: [7](#0-6) 

Similarly, `read_l1_chain_id` should assert that the returned slot value is non-zero, or the chain ID should be supplied through a verified configuration path rather than a raw storage read against a potentially absent contract.

---

### Proof of Concept

1. Deploy a ZKsync OS chain **without** pre-deploying `L2AssetTracker` at `0x800b` (or any address matching `L2_ASSET_TRACKER_ADDRESS`).
2. Submit an L1 priority transaction with `to_mint = 1_000_000` (a deposit).
3. Observe: `transfer_from_treasury` mints tokens to the recipient successfully.
4. Observe: `L2AssetTracker` records **zero** deposited amount (the call was a no-op against an empty account).
5. The discrepancy between minted supply and tracked deposits is permanent and cannot be corrected without a chain upgrade. [8](#0-7)

fn notify_l2_asset_tracker<'a, S: EthereumLikeTypes + 'a, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    system_functions: &mut HooksStorage<S, S::Allocator>,
    memories: RunnerMemoryBuffers<'a>,
    amount: U256,
    l1_chain_id: U256,
    resources: &mut S::Resources,
    tracer: &mut impl Tracer<S>,
    validator: &mut impl TxValidator<S>,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
    S::Metadata: ZkSpecificPricingMetadata
        + BasicMetadata<S::IOTypes, TransactionMetadata = TxLevelMetadata<S::IOTypes>>,
{
    if amount > U256::ZERO || Config::SIMULATION {
        // Encode calldata for handleFinalizeBaseTokenBridgingOnL2(uint256,uint256):
        // selector 0x03117c8c + abi-encoded (fromChainId, amount)
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]);
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution {
                resources_returned,
                result: asset_tracker_result,
            } = BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system,
                system_functions,
                memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,
                inf_ergs.clone(),
                &U256::ZERO,
                true, // should_make_frame - isolate state changes
                tracer,
                validator,
            )?;
            // Overwrite resources inside the closure so that
            // with_infinite_ergs correctly restores ergs afterwards.
            *inf_ergs = resources_returned;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
        })?;

        if failed {
            system_log!(
                system,
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 failed for amount {amount:?}\n"
            );
            // A revert here means the chain's token accounting would be inconsistent.
            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
            )
            .into());
        }
    }
    Ok(())
}

fn read_l1_chain_id<S: EthereumLikeTypes>(system: &mut System<S>) -> U256
where
    S::IO: IOSubsystemExt,
{
    // L2AssetTracker storage layout (verified via `forge inspect`):
    //   slots 0-100:   Initializable + OwnableUpgradeable + Ownable2StepUpgradeable
    //   slots 101-150: Ownable2Step __gap
    //   slot 151:      mapping chainBalance
    //   slot 152:      mapping assetMigrationNumber
    //   slot 153:      mapping isAssetRegistered
    //   slot 154:      uint256 L1_CHAIN_ID
    let l1_chain_id_slot = Bytes32::from_u256_be(&U256::from(154));
    let mut inf_resources = S::Resources::FORMAL_INFINITE;
    let chain_id = system
        .io
        .storage_read::<false>(
            ExecutionEnvironmentType::NoEE,
            &mut inf_resources,
            &L2_ASSET_TRACKER_ADDRESS,
            &l1_chain_id_slot,
        )
        .expect("must read L2AssetTracker L1_CHAIN_ID");
    U256::from_be_bytes(chain_id.as_u8_array())
}
