### Title
ERC-20 Tokens Permanently Burned Before NEP-141 Transfer Confirmation When `error_refund` Feature Is Disabled - (`engine-precompiles/src/native.rs`)

### Summary

The `ExitToNear` precompile burns ERC-20 tokens from EVM state before the corresponding NEP-141 `ft_transfer` promise is confirmed. When the `error_refund` compile-time feature is disabled (the default code path), no callback is attached to the outgoing promise, so a failed NEP-141 transfer results in permanent, unrecoverable loss of the user's bridged tokens.

### Finding Description

The ERC-20 withdrawal flow in Aurora Engine follows this sequence:

1. The user calls `withdrawToNear` on `EvmErc20.sol`, which immediately calls `_burn(_msgSender(), amount)` — the EVM-side tokens are destroyed before any cross-chain operation is attempted.

2. The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) constructs a NEAR promise to call `ft_transfer` (or `ft_transfer_call`) on the NEP-141 contract.

3. The `ExitToNearPrecompileCallbackArgs` struct is populated with `refund: None` when the `error_refund` feature is not compiled in:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
```

4. For a standard ERC-20 `ft_transfer` exit (the most common case), `transfer_near_args` is also `None`. When both fields are `None`, `callback_args` equals the default value, and the promise is created **without any callback**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)  // No callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
```

5. If the NEP-141 `ft_transfer` promise fails (e.g., the recipient account is not registered with the NEP-141 contract, the NEP-141 contract has insufficient balance, or any other runtime failure), there is no callback to detect the failure and re-mint the burned ERC-20 tokens. The tokens are permanently destroyed.

This is structurally identical to the Salty.IO M-24 bug: state (ERC-20 token supply) is irrevocably committed before the dependent cross-chain operation (NEP-141 transfer) is confirmed to have succeeded, and there is no compensating rollback path.

### Impact Explanation

**Critical — Permanent freezing/theft of user funds.**

A user who calls `withdrawToNear` with a recipient that is not registered with the NEP-141 contract will have their ERC-20 tokens burned with no corresponding NEP-141 tokens received. The tokens are gone from both sides of the bridge: burned on the EVM side, never transferred on the NEAR side. This constitutes a permanent, unrecoverable loss of bridged assets.

The existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` explicitly confirms this outcome:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

The ERC-20 balance is permanently reduced by `FT_EXIT_AMOUNT` even though the NEP-141 transfer failed.

### Likelihood Explanation

**Medium.** The trigger condition — a failing NEP-141 `ft_transfer` — is reachable by any unprivileged EVM user. Common failure scenarios include:

- Specifying a NEAR recipient account that is not registered (has no storage deposit) with the NEP-141 contract.
- The NEP-141 contract being paused or having transfer restrictions.
- Any transient or permanent failure in the NEP-141 contract's `ft_transfer` logic.

No special privileges or attacker-controlled infrastructure are required. Any user making a withdrawal to an unregistered account triggers the loss.

### Recommendation

The `error_refund` feature should be enabled in production builds, or the refund callback should be unconditionally attached for all ERC-20 exit paths. The `exit_to_near_precompile_callback` function already contains the correct re-mint logic via `engine::refund_on_error`; the issue is that it is never invoked when `error_refund` is disabled and `transfer_near` is `None`.

Specifically, the promise construction in `ExitToNear::run` should always attach a callback for the ERC-20 exit path so that a failed `ft_transfer` triggers `exit_to_near_precompile_callback`, which can then call `refund_on_error` to re-mint the burned tokens.

### Proof of Concept

**Attacker-controlled entry path:**

1. Deploy or use any existing bridged ERC-20 token on Aurora.
2. Call `withdrawToNear(recipient_bytes, amount)` on the ERC-20 contract where `recipient_bytes` encodes a NEAR account that has no storage deposit with the NEP-141 contract.
3. The ERC-20 contract calls `_burn(msg.sender, amount)` — tokens are destroyed from EVM state.
4. The `ExitToNear` precompile schedules `ft_transfer` on the NEP-141 contract with no callback.
5. The NEP-141 `ft_transfer` fails because the recipient is unregistered.
6. No callback fires; no re-mint occurs.
7. The user has lost `amount` ERC-20 tokens permanently.

**Confirmed by existing test** (`engine-tests/src/tests/erc20_connector.rs`, lines 623–666):

```rust
exit_to_near(
    &ft_owner,
    "unregistered.near",  // not registered → ft_transfer fails
    FT_EXIT_AMOUNT,
    &erc20,
    &aurora,
).await.unwrap();

// Without error_refund: tokens are permanently gone
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
assert_eq!(erc20_balance(&erc20, ft_owner_address, &aurora).await, balance);
```
