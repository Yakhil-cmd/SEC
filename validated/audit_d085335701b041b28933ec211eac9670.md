Audit Report

## Title
SNS Neuron Permanently Inaccessible After All Permissions Removed — No Recovery Path - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS governance canister's `remove_neuron_permissions` function permits a caller with `ManagePrincipals` to remove all permissions from all principals on a neuron, leaving it in a zero-permission state. The neuron is not deleted and its staked tokens remain locked in the governance subaccount. Because `add_neuron_permissions` requires the caller to hold `ManagePrincipals` on the target neuron, and no privileged recovery path exists, the lock is permanent and the staked tokens are irretrievably inaccessible.

## Finding Description
`remove_neuron_permissions` in `rs/sns/governance/src/governance.rs` (lines 4659–4715) performs only three checks before delegating to `remove_permissions_for_principal`: the permissions list is non-empty, its length does not exceed the total number of permission types, and the caller currently holds `ManagePrincipals` (or `ManageVotingPermission` for voting-only removals). There is no post-condition check that at least one principal retains `ManagePrincipals` after the operation. [1](#0-0) 

`remove_permissions_for_principal` in `rs/sns/governance/src/neuron.rs` (lines 782–786) explicitly handles the case where all permission types for a principal are removed by deleting that principal's entry from the neuron's permissions list and returning `AllPermissionTypesRemoved`. It does not check whether the neuron as a whole still has any permissions remaining. [2](#0-1) 

The function's own docstring acknowledges this danger: "This is a dangerous operation as it is possible to remove all permissions for a neuron and no longer be able to modify its state, i.e. disbursing the neuron back into the governance token." [3](#0-2) 

The recovery path, `add_neuron_permissions`, calls `check_principal_authorized_to_change_permissions` before making any change. That function requires the caller to already hold `ManagePrincipals` (or `ManageVotingPermission`) on the neuron. [4](#0-3) [5](#0-4) 

Once `neuron.permissions` is empty, `is_authorized` returns `false` for every principal and every permission type, so no ingress call can satisfy the precondition for `add_neuron_permissions`, `disburse`, `configure_dissolve_state`, or any other neuron management command. No SNS proposal action exists to forcibly re-grant permissions to a neuron with an empty permissions list.

The integration test `test_neuron_remove_all_permissions_of_self` in `rs/sns/integration_tests/src/neuron.rs` (lines 2212–2264) confirms the end state is reachable and asserts `neuron.permissions.len() == 0` as the expected outcome. [6](#0-5) 

## Impact Explanation
Staked SNS governance tokens held in the neuron's ledger subaccount become permanently inaccessible. The neuron cannot be dissolved, disbursed, voted with, split, or modified in any way. This constitutes a **High** severity impact under the SNS security category: "Significant SNS infrastructure security impact with concrete user or protocol harm." The affected user permanently loses access to their staked tokens without any burn or legitimate exit mechanism. In a shared-neuron (co-controller) scenario, a single malicious co-controller can permanently lock all other parties' staked tokens without their consent.

## Likelihood Explanation
The trigger requires only that the attacker hold `ManagePrincipals` on the target neuron, which is granted by default to the neuron claimer. No special privileges, key compromise, or majority attack is needed. Two realistic paths exist: (1) accidental self-lock during ACL cleanup, and (2) deliberate griefing by a co-controller of a shared neuron. The operation is a single `manage_neuron` call with `Command::RemoveNeuronPermissions` specifying `NeuronPermissionType::all()`. The existing integration test demonstrates the path is exercised and passes without error.

## Recommendation
Add a post-removal guard in `remove_neuron_permissions` that checks whether any principal still holds `ManagePrincipals` after the removal. If the resulting neuron would have zero total permissions or no principal retaining `ManagePrincipals`, return a `GovernanceError` with type `PreconditionFailed` before committing the change. Alternatively, introduce a privileged SNS root/governance-level proposal action that can forcibly re-grant permissions to a specified principal on a neuron whose permissions list is empty.

## Proof of Concept
1. Stake tokens and claim an SNS neuron configured with `neuron_claimer_permissions: NeuronPermissionType::all()`.
2. Call `manage_neuron` with `Command::RemoveNeuronPermissions { principal_id: self, permissions_to_remove: NeuronPermissionType::all() }`.
3. Observe `neuron.permissions.len() == 0` — confirmed by the existing integration test `test_neuron_remove_all_permissions_of_self`.
4. Attempt any subsequent `manage_neuron` call (disburse, dissolve, add permissions) — all fail with `NotAuthorized` because `is_authorized` returns `false` for every principal.
5. Confirm no SNS proposal type can recover the neuron's permission list.

    pub(crate) fn check_principal_authorized_to_change_permissions(
        &self,
        caller: &PrincipalId,
        permissions_to_change: NeuronPermissionList,
    ) -> Result<(), GovernanceError> {
        // If the permissions to change are exclusively voting related,
        // ManagePrincipals or ManageVotingPermission is sufficient.
        // Otherwise, only ManagePrincipals is sufficient.
        let sufficient_permissions = if permissions_to_change.is_exclusively_voting_related() {
            vec![
                NeuronPermissionType::ManagePrincipals,
                NeuronPermissionType::ManageVotingPermission,
            ]
        } else {
            vec![NeuronPermissionType::ManagePrincipals]
        };

        // The caller is authorized if they have any of the sufficient permissions
        let caller_authorized = sufficient_permissions
            .iter()
            .any(|sufficient_permission| self.is_authorized(caller, *sufficient_permission));

        if caller_authorized {
            Ok(())
        } else {
            let caller_permissions = self.permissions_for_principal(caller);
            Err(GovernanceError::new_with_message(
                ErrorType::NotAuthorized,
                format!(
                    "Caller '{caller:?}' is not authorized to modify permissions {permissions_to_change} for neuron '{}' as it does not have any of {sufficient_permissions:?}. (Caller's permissions are {caller_permissions})",
                    self.id.as_ref().expect("Neuron must have a NeuronId"),
                ),
            ))
        }
    }
