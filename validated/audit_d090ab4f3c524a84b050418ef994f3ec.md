### Title
`gas_per_pubdata_limit` Not Enforced for L2 ZK Transactions Enables Unexpected Full-Gas Loss on Pubdata Price Spike — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

ABI-encoded ZK L2 transactions carry a `gas_per_pubdata_limit` field that is intended to let users cap the pubdata price they are willing to pay. For L1→L2 transactions this field is read and used to compute `native_per_pubdata`. For L2 ZK transactions the field is parsed but **never checked** against the block-level pubdata price during validation. The actual `native_per_pubdata` is computed solely from the sequencer-controlled block parameters `pubdata_price / native_price`. If those parameters spike between transaction submission and execution, the transaction executes, then reverts post-execution with all gas consumed, even though the user set a `gas_per_pubdata_limit` that should have caused an early validation rejection.

---

### Finding Description

In `Transaction::gas_per_pubdata_limit()`, RLP-encoded (standard Ethereum) transactions always return `U256::ZERO`, while ABI-encoded transactions return the user-supplied field value. [1](#0-0) 

For L1→L2 transactions, `process_l1_transaction.rs` reads this field and uses it directly to compute `native_per_pubdata`: [2](#0-1) [3](#0-2) 

For L2 ZK transactions, `validate_and_compute_fee_for_transaction` in `validation_impl.rs` computes `native_per_pubdata` **exclusively from block-level oracle parameters**, with no reference to the user-supplied `gas_per_pubdata_limit`: [4](#0-3) 

The `gas_per_pubdata_limit` field is never read inside this function. A grep of `basic_bootloader/**/*.rs` confirms the field appears only in `abi_encoded/mod.rs` (definition), `transaction/mod.rs` (accessor), and `process_l1_transaction.rs` (L1→L2 path) — never in the L2 validation path.

The `native_per_pubdata` computed here is later used post-execution to check whether the user has enough gas to cover pubdata: [5](#0-4) 

When the check fails, the transaction reverts and **all remaining ergs are exhausted**, meaning the user loses the full `gas_limit × gas_price` amount.

---

### Impact Explanation

A user submits an ABI-encoded ZK L2 transaction with `gas_per_pubdata_limit = X` (a low value, intending to reject execution if pubdata becomes too expensive). Between submission and inclusion:

- The sequencer raises `pubdata_price` (a block-level oracle parameter).
- `native_per_pubdata = pubdata_price / native_price` now exceeds what the user's gas budget can cover.
- Validation passes (the limit is never checked).
- The transaction executes.
- Post-execution pubdata check fails; the transaction reverts and the full `gas_limit` is consumed.

If the limit had been enforced at validation time, the transaction would have been rejected **before execution** with a validation error, costing the user zero gas. Instead, the user loses `gas_limit × effective_gas_price` in native tokens — a direct, quantifiable funds loss caused solely by the missing enforcement.

The test `test_l2_tx_not_enough_native_for_pubdata_uses_full_gas_limit` explicitly demonstrates this outcome: [6](#0-5) 

---

### Likelihood Explanation

`pubdata_price` and `native_price` are sequencer-controlled block-level parameters that change every block. Any user who relies on `gas_per_pubdata_limit` for protection (a field that exists in the transaction format and is documented as a limit) will silently receive no protection on L2 ZK transactions. The scenario requires only a pubdata price increase between mempool submission and block inclusion — a normal operational event, not an attack.

---

### Recommendation

In `validate_and_compute_fee_for_transaction` (`validation_impl.rs`), after computing `native_per_pubdata` from block prices, check it against the user-supplied `gas_per_pubdata_limit` for ABI-encoded transactions (mirroring the L1→L2 path). If the effective pubdata price exceeds the user's limit, return a `TxError::Validation` error before any state changes or gas consumption occur.

---

### Proof of Concept

1. Alice submits an ABI-encoded ZK L2 transaction with `gas_per_pubdata_limit = 100` (gas units per pubdata byte) and `gas_limit = 250_000`.
2. Before inclusion, the sequencer sets `pubdata_price` such that `pubdata_price / native_price = 700_000` native units per pubdata byte (as in the existing test).
3. `validate_and_compute_fee_for_transaction` computes `native_per_pubdata = 700_000` from block parameters; `gas_per_pubdata_limit = 100` is never read.
4. Validation passes; Alice's balance is pre-charged for `gas_limit × gas_price`.
5. The transaction body executes and writes storage (generating pubdata).
6. `check_enough_resources_for_pubdata` finds insufficient gas; the transaction reverts and `exhaust_ergs()` is called.
7. Alice loses `250_000 × gas_price` in native tokens despite having set a `gas_per_pubdata_limit` of 100 that should have caused a zero-cost validation rejection. [4](#0-3) [1](#0-0) [5](#0-4)
