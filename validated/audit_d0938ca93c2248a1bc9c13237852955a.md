### Title
Missing Refund Path for Failed NEAR Transfer After Successful `near_withdraw` in wNEAR Unwrap Flow — (`engine/src/contract_methods/connector.rs`)

---

### Summary

The wNEAR unwrap path in the `ExitToNear` precompile is a two-step process: (1) burn wNEAR ERC-20 tokens and call `near_withdraw` on the wNEAR NEP-141 contract, then (2) transfer the unwrapped NEAR to the user-specified `target_account_id` via `PromiseAction::Transfer`. A refund mechanism exists only for when step 1 (`near_withdraw`) fails. There is no callback or refund path for when step 2 (the `Transfer` action) fails — for example, when the target NEAR account does not exist. In that case, the user's wNEAR ERC-20 tokens are permanently burned with no recovery, and the equivalent NEAR is stranded in the Aurora engine account.

---

### Finding Description

The `ExitToNear` precompile handles wNEAR unwrapping in `exit_erc20_token_to_near`. When the `:unwrap` suffix is detected and the ERC-20 is the registered wNEAR address, it constructs a two-promise chain:

**Step 1 — `near_withdraw`** (base promise): [1](#0-0) 

This burns the wNEAR NEP-141 tokens and credits NEAR to the Aurora engine account. The `TransferNearArgs` struct records the user-supplied `target_account_id` for later delivery.

**Step 2 — `exit_to_near_precompile_callback`** (callback promise): [2](#0-1) 

When `near_withdraw` succeeds, the callback issues a `PromiseAction::Transfer` to `args.target_account_id` and calls `handler.promise_return(promise_id)`. **No further callback is attached to this transfer promise.** If the transfer fails (e.g., the named NEAR account does not exist), the failure is silently dropped.

The existing refund branch only fires when `near_withdraw` itself fails: [3](#0-2) 

This means the refund path is structurally unreachable for the step-2 failure case.

The `error_refund` feature flag only populates `refund` for the step-1 failure case: [4](#0-3) 

Neither branch covers a failed step-2 transfer.

The recipient account ID is validated only for syntactic correctness, not for on-chain existence: [5](#0-4) 

A syntactically valid but non-existent named NEAR account (e.g., `"alice.near"` before it is created) passes all validation and proceeds to burn the ERC-20 tokens.

---

### Impact Explanation

When a user calls `withdrawToNear` on the wNEAR ERC-20 contract with a recipient of the form `"nonexistent.near:unwrap"`:

1. The wNEAR ERC-20 tokens are burned from the user's EVM balance (irreversible).
2. `near_withdraw` succeeds — wNEAR NEP-141 is burned and NEAR is credited to the Aurora engine account.
3. `exit_to_near_precompile_callback` issues `PromiseAction::Transfer` to `"nonexistent.near"`.
4. The transfer fails because the named account does not exist on NEAR.
5. The NEAR remains in the Aurora engine account with no user-accessible recovery path.
6. The user's wNEAR ERC-20 tokens are permanently gone.

This is a **permanent freezing of user funds**. The NEAR accumulates in Aurora's account and is inaccessible to the user. There is no retry mechanism, no alternative claim path, and no admin recovery function visible in the codebase.

---

### Likelihood Explanation

Any user who specifies a non-existent named NEAR account as the unwrap recipient triggers this. This can occur via:

- A typo in the account name (e.g., `"alic.near"` instead of `"alice.near"`).
- Specifying an account that has not yet been created on NEAR.
- A front-end or integration bug that constructs the recipient string incorrectly.

Named NEAR accounts (e.g., `alice.near`) are not auto-created on transfer — unlike implicit accounts (64-char hex), which NEAR creates automatically. The protocol provides no warning or pre-flight check that the account exists before burning the ERC-20 tokens.

---

### Recommendation

Attach a callback to the `PromiseAction::Transfer` promise created in `exit_to_near_precompile_callback`. If the transfer fails, the callback should re-credit the equivalent amount of wNEAR ERC-20 tokens to the original sender's EVM address (analogous to the existing `refund_on_error` path used for step-1 failures). Alternatively, validate that the target NEAR account exists before committing to the burn, though this requires an additional async read and is architecturally more complex.

---

### Proof of Concept

1. Alice holds wNEAR ERC-20 tokens on Aurora.
2. Alice calls `withdrawToNear("typo.near:unwrap", amount)` on the wNEAR ERC-20 contract, where `"typo.near"` does not exist on NEAR.
3. `EvmErc20.withdrawToNear` burns Alice's ERC-20 tokens and calls the `ExitToNear` precompile.
4. The precompile schedules: `near_withdraw` → callback `exit_to_near_precompile_callback`.
5. `near_withdraw` succeeds; NEAR is credited to the Aurora engine account.
6. The callback issues `PromiseAction::Transfer { amount }` targeting `"typo.near"`.
7. The transfer fails (`AccountDoesNotExist`). No callback handles this failure.
8. Alice's wNEAR ERC-20 tokens are permanently burned. The NEAR is stranded in Aurora's account. Alice has no mechanism to recover her funds.
