### Title
Unaccounted Native Proving Cost in EIP-7002/7251 Block Finalization Queue Processing Allows Attacker to Inflate Block Proving Cost Beyond Paid Budget - (`basic_bootloader/src/bootloader/block_flow/ethereum/eip_7002_withdrawal_contract/mod.rs`)

---

### Summary

The `eip7002_system_part` and `eip7251_system_part` functions process withdrawal/consolidation request queues during block finalization using a **local resource pool initialized with `u64::MAX` native resources**, completely disconnected from the block's native resource budget. Any user can cheaply fill these queues by calling the EIP-7002/7251 predeploy contracts (paying only EVM gas), causing the block finalization to consume unaccounted native proving resources. The actual RISC-V proving cost of the block can exceed what was paid for by a bounded but attacker-maximizable amount every block.

---

### Finding Description

In `eip7002_system_part`, a local resource pool is created with `u64::MAX` native: [1](#0-0) 

All storage reads inside the function — including the per-request loop — charge against this local pool, not against any block-level native budget: [2](#0-1) 

The loop iterates up to `MAX_WITHDRAWAL_REQUESTS_PER_BLOCK = 16` times, reading **3 cold storage slots per request** (slot_0, slot_1, slot_2): [3](#0-2) 

Each cold storage read charges `COLD_EXISTING_STORAGE_READ_NATIVE_COST`: [4](#0-3) 

At `100,000` native units per cold read, 16 requests × 3 slots = **4,800,000 native units per block** are consumed from the local `u64::MAX` pool and **never tracked against the block's native resource budget**.

The identical pattern exists in `eip7251_system_part` with `MAX_CONSOLIDATION_REQUESTS_PER_BLOCK = 2` and 4 slots per request: [5](#0-4) [6](#0-5) 

Both functions are called during block finalization in both the sequencing and proving paths, **after** the per-transaction block limit checks have already passed: [7](#0-6) [8](#0-7) 

The block-level native resource checks only apply during the transaction loop: [9](#0-8) 

The `with_infinite_ergs` wrapper used for every storage read bypasses EVM gas but still charges native from the local pool — meaning the native cost is real (it represents actual RISC-V proving cycles) but is silently absorbed by the `u64::MAX` pool rather than being attributed to the block budget: [10](#0-9) 

---

### Impact Explanation

**Resource accounting bug / economic DoS on the prover.** An attacker who fills the EIP-7002 queue to its per-block maximum forces the block finalization to perform up to 48 cold storage reads (16 × 3) consuming ~4,800,000 native units of proving work that is not charged to any user and not bounded by the block's native resource budget. Combined with EIP-7251 (2 × 4 = 8 reads, ~800,000 native units), the total unaccounted proving overhead per block is ~5,600,000 native units.

Because this overhead is applied **after** the block's native resource limit is checked, a block that is already near its native proving budget can be pushed over the prover's actual RISC-V cycle capacity, causing the block to be unprovable — a chain-halting DoS. Even without hitting a hard cycle cap, the prover is forced to perform uncompensated work every block the attacker keeps the queues full, representing a sustained economic attack.

---

### Likelihood Explanation

**Medium-High.** The EIP-7002 and EIP-7251 predeploy contracts are standard Ethereum Pectra-fork contracts that are deployed as part of the chain's genesis/upgrade. Any unprivileged user can call them by sending a transaction with a small ETH value. The attacker pays only EVM gas for the call; the proving overhead is entirely free. The attacker can sustain the attack indefinitely by submitting new requests each block to keep the queue at its maximum depth, since the excess-request throttling mechanism only raises the ETH fee paid to the predeploy contract — it does not charge native proving resources.

---

### Recommendation

1. **Charge the block-level native budget for EIP-7002/7251 finalization work.** Instead of creating a fresh `u64::MAX` local resource pool, pass a reference to the block's shared native resource counter and charge it for each storage read performed during queue processing.
2. **Pre-charge a fixed native overhead per block** proportional to `MAX_WITHDRAWAL_REQUESTS_PER_BLOCK × SLOTS_PER_REQUEST × COLD_EXISTING_STORAGE_READ_NATIVE_COST` as part of the block's intrinsic native cost, so the prover's budget always accounts for worst-case finalization work.
3. **Alternatively**, charge the native proving cost back to the originating transactions that submitted the withdrawal/consolidation requests, similar to how EVM gas is charged for the predeploy call itself.

---

### Proof of Concept

1. The EIP-7002 predeploy contract is deployed at `0x00000961Ef480Eb55e80D19ad83579A64c007002`.
2. An attacker submits 16 withdrawal requests in a single block by calling the predeploy 16 times, paying only EVM gas (~21,000 gas each).
3. At block finalization, `eip7002_system_part` is called. It reads `queue_tail_index - queue_head_index = 16`, computes `num_dequeued = min(16, 16) = 16`, and enters the loop.
4. The loop performs 16 × 3 = 48 cold storage reads, each consuming `COLD_EXISTING_STORAGE_READ_NATIVE_COST ≈ 100,000` native units from the local `u64::MAX` pool — **not from the block's native budget**.
5. The block's actual RISC-V proving cost is ~4,800,000 native units higher than what the block's native resource accounting reports.
6. If the block was already near the prover's cycle capacity (e.g., filled with computationally heavy transactions), this unaccounted overhead causes the block to exceed the prover's hard limit, making it unprovable and halting the chain.
7. The attacker repeats every block to sustain the attack at minimal cost.
