Audit Report

## Title
Unbounded Synchronous BFS in SNS `cast_vote_and_cascade_follow` Causes Instruction-Limit Trap - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS Governance canister's `cast_vote_and_cascade_follow` is a plain synchronous `fn` that performs an unbounded BFS over the entire follower graph with no instruction-limit check of any kind. It is called directly and synchronously from `register_vote`, which is reachable by any neuron holder with the `Vote` permission via the public `manage_neuron` ingress endpoint. On a sufficiently large follower graph, the message exceeds the IC's 40-billion-instruction hard per-message limit, traps, rolls back the vote, and permanently prevents any vote whose cascade would traverse that graph from succeeding.

## Finding Description

`cast_vote_and_cascade_follow` is declared as a plain `fn` (not `async fn`) at [1](#0-0)  and contains an unbounded `while !induction_votes.is_empty()` BFS loop at [2](#0-1)  with no call to any instruction-counter check. A grep across the entire `rs/sns/governance/` tree for `is_message_over_threshold`, `noop_self_call_if_over_instructions`, `HARD_VOTING`, `SOFT_VOTING`, or `instruction_limit` returns zero matches, confirming the absence of any guard.

The call chain is: `manage_neuron` ingress endpoint [3](#0-2)  → `governance.manage_neuron` → `register_vote` → synchronous call to `Governance::cast_vote_and_cascade_follow` at [4](#0-3) .

The IC's hard per-message instruction limit is 40 billion instructions. [5](#0-4)  Deterministic Time Slicing (DTS) allows a message to span multiple rounds with 2B instructions per slice, but the 40B total cap still applies; a large enough BFS will exhaust it.

By contrast, the NNS Governance canister's equivalent function is `async fn cast_vote_and_cascade_follow` with explicit `SOFT_VOTING_INSTRUCTIONS_LIMIT` (1B) and `HARD_VOTING_INSTRUCTIONS_LIMIT` (750B) constants, [6](#0-5)  and uses `noop_self_call_if_over_instructions` inside the processing loop to yield when the soft limit is reached. [7](#0-6)  The SNS version received no analogous refactoring.

## Impact Explanation

When the message traps with `CanisterInstructionLimitExceeded`, the Wasm state is rolled back: the triggering neuron's ballot remains `Unspecified`, and no follower ballots are updated. Because the follower graph is unchanged, every subsequent attempt by the same neuron to vote on the same proposal will re-enter the identical BFS and trap again. This permanently prevents quorum from being reached on any proposal whose cascade would traverse a large follower graph, effectively freezing SNS governance decisions. This matches the allowed High impact: **"Significant SNS security impact with concrete user or protocol harm"** and **"Application/platform-level DoS … not based on raw volumetric DDoS."**

## Likelihood Explanation

The attacker requires only a standard neuron with the `Vote` permission — the most basic, unprivileged role in SNS governance. The `manage_neuron` endpoint is publicly reachable via ingress. No special keys, governance majority, or subnet-level access are needed. The condition is triggered organically as an SNS DAO grows: any SNS with a popular "named neuron" or hub neuron followed by a large number of other neurons is at risk. The threshold is determined by the total instruction cost of the BFS, which scales with the number of followers and the cost of `BTreeMap`/`BTreeSet` operations per neuron.

## Recommendation

Refactor SNS `cast_vote_and_cascade_follow` to mirror the NNS implementation in `rs/nns/governance/src/voting.rs`:
1. Convert it to an `async fn` and introduce a `VotingStateMachine` that checkpoints BFS progress in stable storage.
2. Add soft/hard instruction-limit checks inside the BFS loop using `is_message_over_threshold` / `noop_self_call_if_over_instructions` (from `ic_nervous_system_long_message`).
3. When the soft limit is reached, break out of the current message and offload remaining BFS work to a timer task, as done in NNS governance.

## Proof of Concept

1. Deploy an SNS with N neurons (e.g., N = 50,000) where neurons 2..N all follow neuron 1 on a given proposal action type.
2. Submit a proposal of that action type; all N neurons receive ballots.
3. As neuron 1's controller, submit `manage_neuron { RegisterVote { proposal_id, vote: Yes } }` via ingress.
4. The SNS governance canister enters `cast_vote_and_cascade_follow` at line 3687 and iterates over all N−1 followers in a single synchronous BFS with no instruction guard.
5. The message exhausts the 40B instruction limit, traps with `CanisterInstructionLimitExceeded`, and all state changes are rolled back.
6. Neuron 1's ballot remains `Unspecified`; the follower graph is unchanged; every subsequent vote attempt by neuron 1 on this proposal repeats the trap.
7. The proposal can never reach quorum through the normal cascade path.

A deterministic integration test can reproduce this using PocketIC with a pre-populated neuron store of sufficient size and asserting that `RegisterVote` returns an error or that the ballot remains `Unspecified` after the call.
