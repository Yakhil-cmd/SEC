### Title
Missing Refund Callback on Omni ERC-20 `ft_transfer_call` Complete Rejection — (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` feature is disabled (the default build) and a user exits an ERC-20 token via the Omni path (`Message::Omni`), the precompile schedules a bare `PromiseArgs::Create` (no callback) for the `ft_transfer_call`. If the NEP-141 receiver's `ft_on_transfer` returns the full transferred amount (complete rejection), the NEP-141 tokens are silently returned to aurora's own balance by the NEP-141 contract's internal `ft_resolve_transfer`, while the user's ERC-20 was already burned and no re-mint or recovery path exists.

---

### Finding Description

**Step 1 — Omni ERC-20 path returns `transfer_near_args = None`.**

In `exit_erc20_token_to_near`, the `Some(Message::Omni(msg))` arm hardcodes `transfer_near_args = None` and `method = "ft_transfer_call"`: [1](#0-0) 

**Step 2 — `callback_args` collapses to `default()` when `error_refund` is disabled.**

Back in `ExitToNear::run`, the callback args struct is built as:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    refund: None,          // #[cfg(not(feature = "error_refund"))]
    transfer_near: None,   // Omni path always returns None
};
```

`ExitToNearPrecompileCallbackArgs` derives `Default` with both fields as `Option::None`: [2](#0-1) 

So `callback_args == ExitToNearPrecompileCallbackArgs::default()` is `true`.

**Step 3 — The equality check at line 470 skips the callback entirely.** [3](#0-2) 

Because `callback_args == default()`, the branch takes `PromiseArgs::Create(transfer_promise)` — a fire-and-forget `ft_transfer_call` with **no** `exit_to_near_precompile_callback` scheduled.

**Step 4 — `error_refund` is not a default feature.**

Neither `engine-precompiles` nor `engine` includes `error_refund` in their `default` feature set: [4](#0-3) [5](#0-4) 

**Step 5 — NEP-141 complete rejection is silent from NEAR's promise perspective.**

In NEP-141, `ft_transfer_call` does not panic when `ft_on_transfer` returns the full amount. The NEP-141 contract's internal `ft_resolve_transfer` silently refunds the tokens back to the caller (aurora's account). From NEAR's promise scheduler, the `ft_transfer_call` promise resolves as **Successful**. Without a callback, aurora has no opportunity to detect the rejection and re-mint the burned ERC-20 tokens.

**Step 6 — The callback, even when present, does not handle "successful but fully rejected".**

For completeness: even when `error_refund` is enabled, `exit_to_near_precompile_callback` only triggers the re-mint branch when the promise result is **not** `Successful`. A fully-rejected `ft_transfer_call` resolves as `Successful`, so the `else if let Some(args) = args.refund` re-mint branch is never reached: [6](#0-5) 

The Omni path also sets `transfer_near = None`, so the success branch does nothing either.

---

### Impact Explanation

The user's ERC-20 balance is burned before the precompile returns. The NEP-141 tokens are transferred to the receiver via `ft_transfer_call`, the receiver rejects them, and the NEP-141 contract returns them to aurora's own NEP-141 balance. The user has no ERC-20 tokens and no NEP-141 tokens. Aurora's NEP-141 balance is permanently inflated by the rejected amount. There is no on-chain recovery path for the user.

Impact: **High — Theft of unclaimed yield / permanent loss of user funds** (aurora's NEP-141 balance absorbs the rejected tokens with no user recourse).

---

### Likelihood Explanation

- `error_refund` is **off by default**, making this the standard production code path.
- The Omni exit path is a supported, documented production interface.
- A receiver that returns the full amount from `ft_on_transfer` is a standard NEP-141 rejection pattern (e.g., wrong message format, unsupported token, contract logic).
- A malicious receiver contract can deliberately always reject, reliably triggering this loss for any user who targets it.
- No admin compromise or privileged access is required; any EVM user can trigger this by calling the ERC-20's burn/exit function with an Omni message targeting a rejecting receiver.

Likelihood: **Medium-High** (reachable by any user; receiver rejection is a normal NEP-141 scenario).

---

### Recommendation

1. **Always schedule a callback for `ft_transfer_call` in the Omni ERC-20 path**, regardless of the `error_refund` feature flag. The callback must inspect the NEP-141 `ft_resolve_transfer` result (or the returned amount) to detect partial/full rejection.
2. **In `exit_to_near_precompile_callback`**, add handling for the case where `ft_transfer_call` resolves as `Successful` but the transferred amount was partially or fully returned (i.e., read the promise return value, which is the refunded amount, and re-mint accordingly).
3. Consider removing the `callback_args == default()` short-circuit entirely for `ft_transfer_call` paths, since `ft_transfer_call` always requires a callback to correctly handle rejection.

---

### Proof of Concept

```
1. Deploy a NEP-141 receiver contract whose `ft_on_transfer` always returns
   the full `amount` argument (complete rejection).

2. User holds N units of an ERC-20 token on Aurora (backed by the same NEP-141).

3. User calls the ERC-20's burn/exit function with an Omni message targeting
   the malicious receiver. This invokes ExitToNear::run with:
     - flag = 0x1 (ERC-20)
     - message = Message::Omni("...")
   The ERC-20 contract burns N tokens and calls the precompile.

4. The precompile (error_refund disabled) builds:
     callback_args = { refund: None, transfer_near: None } == default()
   → schedules PromiseArgs::Create(ft_transfer_call) with NO callback.

5. NEAR executes ft_transfer_call on the NEP-141 contract:
   - NEP-141 calls receiver.ft_on_transfer → returns N (full rejection)
   - NEP-141's ft_resolve_transfer refunds N tokens back to aurora's balance
   - Promise resolves Successful

6. Assert:
   - User's ERC-20 balance = 0  (burned, never re-minted)
   - User's NEP-141 balance = 0  (never received)
   - Aurora's NEP-141 balance increased by N
   - No callback was ever scheduled or executed
``` [7](#0-6)
