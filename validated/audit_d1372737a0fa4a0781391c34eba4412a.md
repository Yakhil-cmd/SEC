### Title
Unchecked Exit-Precompile Call Return Value Causes Permanent Token Loss on Withdrawal Failure - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20` and `EvmErc20V2` burn the caller's ERC-20 tokens **before** invoking the Aurora exit precompile via a low-level `call()`. The return value of that `call()` is captured in a local assembly variable `res` but is **never checked**. If the precompile call fails for any reason, the burn is irreversible and the corresponding NEAR or Ethereum transfer is never scheduled — permanently destroying the user's funds with no recovery path.

---

### Finding Description

In `withdrawToNear` and `withdrawToEthereum` of both `EvmErc20.sol` and `EvmErc20V2.sol`, the sequence is:

1. `_burn(_msgSender(), amount)` — tokens are destroyed from the caller's balance.
2. A low-level `call()` is made to the exit precompile address (`ExitToNear` at `0xe9217bc7…` or `ExitToEthereum` at `0xb0bd02f6…`).
3. The result `res` is stored in an assembly local but **never tested**; no `if iszero(res) { revert(0,0) }` guard exists. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

The precompile (`ExitToNear::run` / `ExitToEthereum::run` in `engine-precompiles/src/native.rs`) returns `ExitError` — which the EVM translates to a `call()` return value of `0` — under several reachable conditions:

| Condition | Error string |
|---|---|
| `amount > u128::MAX` (uint256 input accepted by Solidity) | `ERR_INVALID_AMOUNT` |
| Recipient bytes are invalid UTF-8 or invalid NEAR account ID | `ERR_INVALID_RECEIVER_ACCOUNT_ID` |
| ERC-20 → NEP-141 mapping absent from storage | `ERR_TARGET_TOKEN_NOT_FOUND` |
| Input size outside `[MIN_INPUT_SIZE, MAX_INPUT_SIZE]` | `ERR_INVALID_INPUT` | [5](#0-4) [6](#0-5) 

Because the Solidity functions accept `uint256 amount` and `bytes memory recipient` without any pre-validation, all of these failure modes are reachable by an ordinary token holder calling the public `withdrawToNear` / `withdrawToEthereum` interface.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the precompile call fails silently:
- The `_burn` has already reduced the caller's ERC-20 balance to zero for the withdrawn amount.
- No NEAR `ft_transfer` / `ft_transfer_call` promise is ever created.
- No Ethereum `withdraw` promise is ever created.
- There is no refund branch in the Solidity code; the `error_refund` feature in the Rust precompile only handles NEAR-side promise failures, not EVM-side precompile call failures.

The tokens are destroyed on the EVM side and never credited on the destination side. They cannot be recovered by any user action or admin action short of a contract upgrade and manual re-minting.

---

### Likelihood Explanation

**Medium.** The most realistic trigger requires no special privilege:

- A user calls `withdrawToNear(recipient, amount)` where `recipient` encodes an invalid NEAR account ID (e.g., contains uppercase letters, is too long, or is empty after the flag byte). The `_burn` succeeds; the precompile rejects the recipient and returns `ExitError`; `res == 0`; the Solidity function returns normally.
- A user passes `amount` as a `uint256` value exceeding `2^128 - 1`. The Solidity `uint256` parameter accepts it; `_burn` succeeds if the user somehow holds that balance (e.g., via a misconfigured mint); `parse_amount` returns `ERR_INVALID_AMOUNT`; same silent failure.

The first scenario (invalid recipient) is reachable by any token holder making a simple user error or by a malicious contract that constructs the call programmatically.

---

### Recommendation

Add a revert guard immediately after each assembly `call()` block in both contracts:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. This ensures that if the exit precompile rejects the call for any reason, the entire transaction reverts — including the `_burn` — so the user's tokens are never destroyed without a corresponding cross-chain transfer being scheduled.

---

### Proof of Concept

1. Deploy `EvmErc20` (or use an existing Aurora ERC-20 mirror token).
2. Mint `1000` tokens to address `Alice`.
3. Alice calls `withdrawToNear(bytes("\xff\xfe"), 1000)` — `\xff\xfe` is invalid UTF-8, so the precompile will return `ERR_INVALID_RECEIVER_ACCOUNT_ID`.
4. Observe: Alice's balance is now `0` (burn succeeded). No NEAR `ft_transfer` promise was created. The transaction did **not** revert. Alice's 1000 tokens are permanently destroyed.

The root cause is confirmed at:
- `EvmErc20.sol` lines 54 and 61 (burn then unchecked call)
- `EvmErc20V2.sol` lines 55 and 62 (burn then unchecked call)
- `EvmErc20.sol` lines 66 and 74 (burn then unchecked call for Ethereum exit)
- `EvmErc20V2.sol` lines 67 and 75 (burn then unchecked call for Ethereum exit) [1](#0-0) [3](#0-2) [7](#0-6)

fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);

    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
    }

    Ok(amount)
}
