### Title
Unchecked Return Value from Exit Precompile Low-Level Call Causes Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

### Summary
`EvmErc20.sol` and `EvmErc20V2.sol` burn the caller's tokens before making a low-level `call()` to the Aurora exit precompiles. The return value `res` is captured in assembly but never checked. If the precompile call fails, the EVM transaction does not revert, leaving the tokens permanently destroyed with no corresponding exit on NEAR or Ethereum.

### Finding Description
In both `withdrawToNear` and `withdrawToEthereum` in `EvmErc20.sol` and `EvmErc20V2.sol`, the pattern is:

1. `_burn(_msgSender(), amount)` â€” tokens are irreversibly destroyed.
2. A low-level `call()` is made to the exit precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` for NEAR, `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` for Ethereum).
3. The result `res` is stored in a local assembly variable but **never checked**.

`EvmErc20.sol` `withdrawToEthereum`: [2](#0-1) 

`EvmErc20V2.sol` exhibits the identical pattern: [3](#0-2) [4](#0-3) 

The `ExitToNear` precompile can return failure (`ExitError`) in several reachable conditions, including when the ERC-20 contract address is not found in the NEP-141 registry: [5](#0-4) 

It also fails if called in a static context, in a delegate call, or if the input is malformed: [6](#0-5) 

When a precompile call fails in the EVM, the low-level `call()` returns `0` and execution continues in the caller â€” the `_burn` is **not** rolled back. The tokens are gone, and no exit is recorded.

### Impact Explanation
**Critical â€” Permanent freezing of funds.** A user's ERC-20 bridged tokens are burned (supply reduced, balance zeroed) but the corresponding NEP-141 tokens on NEAR (or ETH on Ethereum) are never minted or transferred to the recipient. The funds are permanently destroyed with no recovery path. This affects any holder of any `EvmErc20`/`EvmErc20V2` token who calls `withdrawToNear` or `withdrawToEthereum` when the precompile call fails.

### Likelihood Explanation
**Medium.** The precompile call fails silently whenever:
- The ERC-20 contract address is not registered in the NEP-141 â†” ERC-20 map (e.g., a token deployed outside the normal engine flow, or after a deregistration event).
- The eth connector account key is absent from storage.
- The precompile encounters an out-of-gas condition (the contracts forward `gas()` but the precompile itself schedules NEAR promises that require gas budget).

Any EVM user holding tokens in an affected contract can trigger this by calling `withdrawToNear` or `withdrawToEthereum` directly â€” no special privileges required.

### Recommendation
Check the return value of the precompile call and revert if it is zero:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to both `withdrawToNear` and `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. Alternatively, restructure the functions to perform the precompile call before the burn, reverting on failure before any state change.

### Proof of Concept

1. An `EvmErc20` token contract is deployed at address `A`. For any reason (e.g., manual deployment, deregistration, storage inconsistency), `A` is not present in the engine's NEP-141 â†” ERC-20 map.
2. A user holding `N` tokens calls `withdrawToNear(recipient, N)`.
3. `_burn(msg.sender, N)` executes â€” the user's balance drops to zero, total supply decreases by `N`.
4. The assembly `call()` to `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` is dispatched. Inside the `ExitToNear` precompile, `get_nep141_from_erc20` fails with `ERR_TARGET_TOKEN_NOT_FOUND` because `A` is not registered. The precompile returns `ExitError`, so the low-level call returns `res = 0`.
5. Because `res` is never checked, execution continues normally. The transaction succeeds (no revert).
6. The user has lost `N` tokens permanently: they are burned on the EVM side, and no NEP-141 tokens are transferred on NEAR. [7](#0-6) [1](#0-0)

fn get_nep141_from_erc20<I: IO>(erc20_token: &[u8], io: &I) -> Result<AccountId, ExitError> {
    AccountId::try_from(
        io.read_storage(bytes_to_key(KeyPrefix::Erc20Nep141Map, erc20_token).as_slice())
            .map(|s| s.to_vec())
            .ok_or(ExitError::Other(Cow::Borrowed(ERR_TARGET_TOKEN_NOT_FOUND)))?,
    )
    .map_err(|_| ExitError::Other(Cow::Borrowed("ERR_INVALID_NEP141_ACCOUNT")))
}
