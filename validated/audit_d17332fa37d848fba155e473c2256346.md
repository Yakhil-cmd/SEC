### Title
Cycles Permanently Lost When FlexibleHttpRequest (PayAsYouGo) Is Rejected by Client ‚Äî (`rs/https_outcalls/client/src/client.rs`, `rs/https_outcalls/pricing/src/lib.rs`, `rs/execution_environment/src/execution_environment.rs`)

### Summary

A concrete, multi-file code path exists where cycles attached to a `FlexibleHttpRequest` (which sets `pricing_version=PayAsYouGo`) are taken upfront by the execution environment but never refunded, because the client rejects the request using a `LegacyTracker` that always produces a zero-refund receipt.

### Finding Description

**Step 1 ‚Äî Execution environment takes full payment upfront for PayAsYouGo.**

In `rs/execution_environment/src/execution_environment.rs`, the payment branch for `PayAsYouGo` calls `request.payment.take()`, draining all attached cycles from the request context: [1](#0-0) 

The `refundable_cycles` / `per_replica_allowance` fields are set up to track what should be returned, but the actual refund amount is determined later by the `CanisterHttpPaymentReceipt` produced by the client.

**Step 2 ‚Äî `PricingFactory::new_tracker` ignores `pricing_version` and always creates `LegacyTracker`.**

The TODO comment at line 57 explicitly acknowledges this is incomplete: [2](#0-1) 

So `budget` in the client is always a `LegacyTracker`, regardless of whether the request is `Legacy` or `PayAsYouGo`.

**Step 3 ‚Äî `send()` detects `PayAsYouGo`, rejects with `SysFatal`, and calls `budget.create_payment_receipt()`.** [3](#0-2) 

The `budget.create_payment_receipt()` call at line 175 dispatches to `LegacyTracker::create_payment_receipt()`.

**Step 4 ‚Äî `LegacyTracker::create_payment_receipt()` returns `CanisterHttpPaymentReceipt::default()` (refund = 0).** [4](#0-3) 

The comment explicitly states "no cycles are ever refunded." The default receipt carries zero refund.

**Step 5 ‚Äî Consensus delivers the rejection response with the zero-refund receipt.**

The execution environment processes the `CanisterHttpPaymentReceipt` to determine how many of the `refundable_cycles` to return. Since the receipt reports 0, the canister receives 0 cycles back, while the full payment `P` was already consumed at Step 1.

### Impact Explanation

A canister that submits a `FlexibleHttpRequest` with payment `P > base_fee` loses `P` cycles permanently. The invariant "cycles paid for a rejected request must be refunded to the caller" is violated. At scale (many canisters using `FlexibleHttpRequest` with large payments), this constitutes a material loss of cycles with no recovery path.

### Likelihood Explanation

The precondition is that the `FlexibleHttpRequest` feature flag is enabled. The TODO comment at line 57 of `pricing/src/lib.rs` is a direct acknowledgment that the `PayAsYouGo` path is incomplete ‚Äî the tracker factory was never wired to respect `pricing_version`. The client's rejection of `PayAsYouGo` at lines 155‚Äì178 of `client.rs` was presumably added as a guard, but it was added *after* the execution environment already drained the payment, creating the gap. No additional privilege is required beyond submitting a canister HTTP outcall.

### Recommendation

1. In `PricingFactory::new_tracker`, implement the TODO at line 57: branch on `context.pricing_version` and return a `PayAsYouGoTracker` that correctly computes refunds.
2. In the `send()` early-rejection path (lines 164‚Äì176 of `client.rs`), the `budget.create_payment_receipt()` call must produce a receipt that refunds the full `per_replica_allowance` when the request is rejected before any network resources are consumed.
3. Alternatively, gate the `payment.take()` call in `execution_environment.rs` so it only fires when the `PayAsYouGo` tracker is fully implemented and the client will not immediately reject the request.

### Proof of Concept

A state-machine test would:
1. Enable the `FlexibleHttpRequest` feature flag.
2. Record canister balance `B` before submission.
3. Submit a `FlexibleHttpRequest` with payment `P` (e.g., 10√ó base fee).
4. Mock the client to exercise the `PayAsYouGo` early-rejection path (lines 155‚Äì178).
5. Deliver the consensus response.
6. Assert canister balance equals `B - base_fee` (correct) rather than `B - P` (actual buggy outcome).

The assertion at step 6 will fail, demonstrating permanent cycle loss equal to `P - base_fee`.
