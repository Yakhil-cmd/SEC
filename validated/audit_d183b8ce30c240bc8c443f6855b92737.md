Audit Report

## Title
Unbounded `ingress_expiries` Loop in `construction_payloads` Allows Unauthenticated Single-Request DoS via Memory Exhaustion â€” (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary
The `construction_payloads` function in the ICRC1 Rosetta API iterates from a caller-supplied `ingress_start` to a caller-supplied `ingress_end` with a fixed step of 120 seconds, with no cap on the number of iterations. A single unauthenticated HTTP POST with a crafted `ingress_end` value causes unbounded heap allocation, exhausting the Rosetta process's memory and crashing it. This is confirmed by direct code inspection of the production service.

## Finding Description
In `rs/rosetta-api/icrc1/src/construction_api/services.rs`, `ingress_start` and `ingress_end` are read directly from caller-supplied request metadata:

```rust
// lines 128-136
let mut ingress_start = metadata
    .as_ref()
    .and_then(|meta| meta.ingress_start)
    .unwrap_or(now);

let ingress_end = metadata
    .as_ref()
    .and_then(|meta| meta.ingress_end)
    .unwrap_or(ingress_start + ingress_interval);
``` [1](#0-0) 

Two guards are applied at lines 148â€“158:

```rust
if ingress_start >= ingress_end { return Err(...); }
if ingress_end < now + ingress_interval { return Err(...); }
``` [2](#0-1) 

These enforce `ingress_start < ingress_end` and `ingress_end >= now + ingress_interval`, but place **no upper bound** on `ingress_end - ingress_start`. The loop then runs without any iteration cap:

```rust
// lines 162-167
let mut ingress_expiries = vec![];
while ingress_start < ingress_end {
    ingress_expiries.push(ingress_start + ingress_interval);
    ingress_start +=
        ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
}
``` [3](#0-2) 

The step per iteration is `ingress_interval - INGRESS_INTERVAL_OVERLAP`:
- `ingress_interval` = `MAX_INGRESS_TTL âˆ’ PERMITTED_DRIFT` = 300s âˆ’ 60s = **240s** [4](#0-3) 
- `INGRESS_INTERVAL_OVERLAP` = **120s** [5](#0-4) 
- Step = **120,000,000,000 ns (120 seconds)**

After the loop, `handle_construction_payloads` in `rs/rosetta-api/icrc1/src/construction_api/utils.rs` iterates over `ingress_expiries` and allocates one `EnvelopeContent::Call` and one `SigningPayload` per entry, each containing heap-allocated `Vec<u8>` fields (serialized canister args, hex-encoded message IDs): [6](#0-5) 

The endpoint is fully unauthenticated â€” it accepts any `ConstructionPayloadsRequest` from any HTTP client with no authentication or rate limiting in the service layer: [7](#0-6) 

## Impact Explanation
This matches the allowed ICP bounty impact: **High ($2,000â€“$10,000) â€” Application/platform-level DoS, crash, or availability impact not based on raw volumetric DDoS.**

For `ingress_end = now + 315_360_000_000_000_000` (10 years in nanoseconds):
- Iterations â‰ˆ 315,360,000,000,000,000 / 120,000,000,000 â‰ˆ **2,628,000**
- `ingress_expiries` alone: ~21 MB (8 bytes Ã— 2.6M entries)
- `handle_construction_payloads` then allocates one `EnvelopeContent::Call` (containing cloned `canister_method_args: Vec<u8>`) and one `SigningPayload` (containing a hex string) per entry, multiplying allocation by a large constant factor per entry

For `ingress_end = now + u64::MAX/2` (~292 years):
- Iterations â‰ˆ 76.8 billion â†’ process OOM before completion

A 10-year window is sufficient to cause measurably large heap allocation within a single synchronous request handler, crashing the Rosetta process and causing complete availability loss for all ICRC1 Rosetta API users.

## Likelihood Explanation
The attack requires a single unauthenticated HTTP POST to `/construction/payloads` with a crafted JSON body. No special privileges, credentials, or prior state are required. The endpoint is part of the standard Rosetta Construction API and is publicly accessible. The attack is trivially reproducible, repeatable, and requires no victim interaction.

## Recommendation
Add an explicit cap on the number of ingress expiry entries before the loop:

```rust
const MAX_INGRESS_EXPIRIES: usize = 1000; // ~33 hours at 120s step
let step = ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
if step > 0 {
    let span = ingress_end.saturating_sub(ingress_start);
    let count = (span / step) as usize + 1;
    if count > MAX_INGRESS_EXPIRIES {
        return Err(Error::processing_construction_failed(
            "ingress_end exceeds maximum allowed window"
        ));
    }
}
```

Alternatively, enforce a maximum `ingress_end - ingress_start` window (e.g., 24 hours = 86,400,000,000,000 ns) before entering the loop.

## Proof of Concept
Send a single HTTP POST to the running ICRC1 Rosetta node:

```
POST /construction/payloads
Content-Type: application/json

{
  "network_identifier": { <valid_network_id> },
  "operations": [ <valid_transfer_op> ],
  "public_keys": [ <valid_pubkey> ],
  "metadata": {
    "ingress_start": <now_ns>,
    "ingress_end": <now_ns + 315360000000000000>
  }
}
```

The process enters the `while` loop at line 163 of `services.rs`, allocates ~2.6 million `u64` entries into `ingress_expiries`, then `handle_construction_payloads` allocates one `EnvelopeContent::Call` and one `SigningPayload` per entry. Heap usage grows measurably within a single synchronous call. With `ingress_end = now + u64::MAX/2`, the process is killed by the OS OOM killer before the loop completes.

A deterministic unit test can confirm this without a live node:
```rust
#[test]
fn test_construction_payloads_oom_dos() {
    let now = SystemTime::now();
    let now_ns = now.duration_since(SystemTime::UNIX_EPOCH).unwrap().as_nanos() as u64;
    let ingress_interval = (ic_limits::MAX_INGRESS_TTL - ic_limits::PERMITTED_DRIFT).as_nanos() as u64;
    // 10-year window
    let ingress_end = now_ns + 315_360_000_000_000_000u64;
    let result = construction_payloads(
        vec![<valid_op>],
        Some(ConstructionPayloadsRequestMetadata {
            ingress_start: Some(now_ns),
            ingress_end: Some(ingress_end),
            ..Default::default()
        }),
        &Principal::anonymous(),
        vec![<valid_pubkey>],
        now,
    );
    // Without a fix, this allocates ~2.6M entries and measurably large heap before returning
}
```
