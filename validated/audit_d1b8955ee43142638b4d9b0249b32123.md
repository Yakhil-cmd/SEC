### Title
ckERC20 Minter Does Not Verify ERC20 `transfer` Return Value When Finalizing Withdrawals — (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, `rs/ethereum/cketh/minter/src/state.rs`)

---

### Summary

The ckETH/ckERC20 minter canister sends raw ERC20 `transfer(address,uint256)` transactions to Ethereum for ckERC20 withdrawals. When finalizing these transactions, the minter only checks the Ethereum transaction-level status field (`receipt.status == TransactionStatus::Success`) from the receipt. It never decodes or verifies the boolean return value of the ERC20 `transfer` call, nor does it inspect the transaction logs for a `Transfer` event. For non-reverting ERC20 tokens that return `false` on failure (the exact class described in H-03), the Ethereum transaction succeeds at the protocol level while the token transfer silently fails. The minter then permanently burns the user's ckERC20 tokens without reimbursement, while the ERC20 tokens remain in the minter's Ethereum address.

---

### Finding Description

**Vulnerability class:** Chain-fusion mint/burn/replay bug (ledger conservation bug).

The ckERC20 withdrawal flow works as follows:
1. User calls `withdraw_erc20` on the minter.
2. Minter burns the user's ckERC20 tokens on the IC ledger.
3. Minter constructs and sends an EIP-1559 transaction calling `transfer(to, amount)` on the ERC20 contract address.
4. Minter polls for the transaction receipt via `get_transaction_receipt`.
5. Minter calls `record_finalized_transaction` and `update_balance_upon_withdrawal`.

The `TransactionReceipt` struct used by the minter is defined in `rs/ethereum/cketh/minter/src/eth_rpc_client/responses.rs`:

```rust
pub struct TransactionReceipt {
    pub block_hash: Hash,
    pub block_number: BlockNumber,
    pub effective_gas_price: WeiPerGas,
    pub gas_used: GasAmount,
    pub status: TransactionStatus,
    pub transaction_hash: Hash,
}
``` [1](#0-0) 

The struct contains no `logs` field and no return-data field. The minter has no mechanism to inspect the ERC20 `transfer` return value.

In `record_finalized_transaction`, the ckERC20 reimbursement path is gated exclusively on `receipt.status == TransactionStatus::Failure`:

```rust
WithdrawalRequest::CkErc20(request) => {
    if receipt.status == TransactionStatus::Failure {
        self.record_reimbursement_request(...);
    }
}
``` [2](#0-1) 

In `update_balance_upon_withdrawal`, the minter subtracts from its internal `erc20_balances` whenever `receipt.status == TransactionStatus::Success`:

```rust
if receipt.status == TransactionStatus::Success && !tx.transaction_data().is_empty() {
    let TransactionCallData::Erc20Transfer { to: _, value } = TransactionCallData::decode(
        tx.transaction_data(),
    )...;
    self.erc20_balances.erc20_sub(*tx.destination(), value);
}
``` [3](#0-2) 

For ERC20 tokens that return `false` without reverting (the non-standard behavior described in H-03), the Ethereum transaction receipt carries `status = 1` (Success). The minter therefore:
- Does **not** trigger a reimbursement of the burned ckERC20 tokens.
- Decrements its internal `erc20_balances` as if the transfer succeeded.
- Considers the withdrawal permanently finalized.

The ERC20 `transfer` call data is encoded using the standard `a9059cbb` selector:

```rust
data: TransactionCallData::Erc20Transfer {
    to: request.destination,
    value: request.withdrawal_amount,
}.encode(),
``` [4](#0-3) 

The minter never reads back the return value of this call from the receipt.

---

### Impact Explanation

A user who initiates a ckERC20 withdrawal for a supported non-reverting ERC20 token permanently loses their ckERC20 tokens. The IC-side burn is committed and irreversible. The Ethereum-side transfer silently fails. The minter's internal accounting (`erc20_balances`) is decremented as if the transfer succeeded, causing the minter's tracked balance to diverge from its actual on-chain ERC20 balance. No reimbursement is ever issued. This is a direct ledger conservation violation: ckERC20 supply is destroyed without a corresponding ERC20 transfer on Ethereum.

---

### Likelihood Explanation

The minter supports any ERC20 token added via governance proposal. Several deployed ERC20 tokens (including historical USDT versions, BNB, and others) implement `transfer` as a non-reverting function that returns `false` on failure conditions such as blacklisting, contract pausing, or insufficient balance. If any such token is added as a supported ckERC20 asset, any user whose withdrawal triggers a `false` return (e.g., because the destination address is blacklisted by the token contract) will lose funds. The entry path requires only a standard unprivileged `withdraw_erc20` call — no special access is needed.

---

### Recommendation

1. **Include transaction logs in the receipt:** Extend the `TransactionReceipt` struct and the `eth_getTransactionReceipt` RPC call to capture the `logs` array. After a successful-status ERC20 withdrawal transaction, verify that a `Transfer(from, to, value)` event was emitted by the ERC20 contract with the expected parameters. If no such log is present, treat the withdrawal as failed and trigger reimbursement.

2. **Alternatively, verify via balance check:** After a successful-status ERC20 withdrawal, query the ERC20 balance of the minter's Ethereum address and compare it to the expected post-transfer balance. A discrepancy indicates a silent transfer failure.

3. **Restrict supported tokens:** As a defense-in-depth measure, only allow ERC20 tokens that are known to revert on failure (i.e., standard EIP-20 compliant tokens) to be added as ckERC20 assets.

---

### Proof of Concept

1. Governance adds a non-reverting ERC20 token (e.g., one that returns `false` when the recipient is blacklisted) as a supported ckERC20 asset.
2. User calls `withdraw_erc20` specifying a blacklisted Ethereum destination address.
3. Minter burns the user's ckERC20 tokens on the IC ledger (irreversible).
4. Minter constructs and submits an EIP-1559 transaction calling `transfer(blacklisted_address, amount)` on the ERC20 contract.
5. The ERC20 contract executes, returns `false` (no revert). Ethereum transaction receipt: `status = 1` (Success).
6. Minter calls `record_finalized_transaction`: `receipt.status == TransactionStatus::Success`, so the `if receipt.status == TransactionStatus::Failure` branch at line 734 of `mod.rs` is **not** taken — no reimbursement is recorded.
7. Minter calls `update_balance_upon_withdrawal`: `receipt.status == TransactionStatus::Success` at line 377 of `state.rs` — `erc20_balances.erc20_sub` is called, decrementing the minter's tracked balance.
8. The withdrawal is permanently marked as finalized with `TxFinalizedStatus::Success`.
9. User has lost their ckERC20 tokens. The ERC20 tokens remain in the minter's Ethereum address. The minter's internal balance is now understated relative to its actual on-chain ERC20 holdings. [5](#0-4) [6](#0-5) [1](#0-0)
