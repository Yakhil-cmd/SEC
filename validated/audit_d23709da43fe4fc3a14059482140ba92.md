### Title
Optimistic Scheduled-Promise Deletion Without Failure Callback Causes Permanent Fund Freeze - (`File: etc/xcc-router/src/lib.rs`)

---

### Summary

`Router::execute_scheduled()` unconditionally removes a stored promise from `scheduled_promises` before the dispatched NEAR promise executes, and attaches no failure callback. If the dispatched promise fails, the scheduled data is permanently lost and the EVM-side wNEAR already burned/transferred cannot be recovered.

---

### Finding Description

The XCC router's `execute_scheduled` function is the direct analog of the Timelock `executeTransaction()` bug. [1](#0-0) 

```rust
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

`self.scheduled_promises.remove(&nonce.0)` deletes the stored promise **before** the NEAR promise executes. No callback is attached to `promise_id` to detect failure and restore the entry. If the dispatched promise fails for any reason, the slot is gone permanently.

The `schedule` function stores the promise when the Aurora Engine calls it on behalf of an EVM user who used `CrossContractCallArgs::Delayed`: [2](#0-1) 

The EVM-side entry point is the XCC precompile. When `CrossContractCallArgs::Delayed` is used, the precompile first executes a wNEAR `transferFrom` on the EVM side (burning/locking the user's tokens) and then calls `Router::schedule`: [3](#0-2) [4](#0-3) 

The wNEAR ERC-20 transfer is committed atomically and irreversibly as part of the EVM transaction. The NEAR-side execution of the scheduled promise is deferred to a separate transaction via `execute_scheduled`.

The comment in the code explicitly acknowledges that `execute_scheduled` is open to anyone: [5](#0-4) 

This means any external account can trigger execution of any user's scheduled promise at any time, including under conditions where the target contract is likely to fail (e.g., paused, out of storage, wrong state).

The `execute` path (eager mode) does not have this problem because it fires the promise directly without storing state. The vulnerability is isolated to the `Delayed` / `execute_scheduled` path.

---

### Impact Explanation

**Permanent freezing of funds.**

When a user uses `CrossContractCallArgs::Delayed`, the EVM-side wNEAR tokens are transferred from the user's address to the engine's implicit address as part of the same EVM transaction. This transfer is irreversible once committed. If `execute_scheduled` is called and the dispatched NEAR promise fails:

1. `scheduled_promises` entry is deleted — the promise cannot be retried.
2. The EVM-side wNEAR deduction already happened — the user's EVM balance is reduced.
3. No refund mechanism exists for the `Delayed` path (unlike `exit_to_near_precompile_callback` which has an explicit refund branch for the eager connector path). [6](#0-5) 

The `Delayed` XCC path has no equivalent refund. The user's funds are permanently frozen in the engine's implicit NEAR account with no on-chain path to recover them.

---

### Likelihood Explanation

**High.** `execute_scheduled` is callable by any NEAR account with no access restriction. NEAR promises fail for many realistic reasons: the target contract panics, runs out of gas, is paused, has insufficient storage deposit, or returns an error. Because the caller is arbitrary and the timing is arbitrary, an adversary can deliberately call `execute_scheduled` at a moment when the target is known to be in a failing state (e.g., a paused NEP-141 contract), permanently destroying the scheduled promise and freezing the user's wNEAR.

---

### Recommendation

Attach a failure callback to `promise_id` inside `execute_scheduled`. The callback should:

1. Check `promise_result_check()` — if the promise failed, re-insert the promise back into `scheduled_promises` at the same nonce, or emit an event so the user can re-schedule.
2. Alternatively, do not remove the entry until the callback confirms success (use a "pending" flag or a two-phase commit pattern).

The `factory_update_address_version` callback in the engine already demonstrates the correct pattern for checking promise results before committing state: [7](#0-6) 

Apply the same guard in `execute_scheduled`.

---

### Proof of Concept

**Setup**: EVM user holds wNEAR ERC-20 on Aurora. They call the XCC precompile at `0x516cded1d16af10cad47d6d49128e2eb7d27b372` with `CrossContractCallArgs::Delayed(ft_transfer_promise)` targeting a NEP-141 contract.

**Step 1 — EVM transaction**: The precompile executes `wNEAR.transferFrom(user, engine_implicit, amount)` on the EVM, reducing the user's wNEAR balance. The router's `schedule` stores the promise at nonce `N`. [3](#0-2) 

**Step 2 — Attacker triggers execution at bad time**: Any account calls `router.execute_scheduled({"nonce": "N"})` when the target NEP-141 contract is paused or the receiver is unregistered. [1](#0-0) 

**Step 3 — Promise fails**: The dispatched NEAR promise panics. NEAR rolls back the promise's state changes but does **not** roll back `execute_scheduled`'s storage mutation — `scheduled_promises[N]` is already gone.

**Step 4 — Permanent loss**: `scheduled_promises.get(N)` returns `None`. The user's wNEAR ERC-20 was already burned. No refund path exists. Funds are permanently frozen.

This mirrors the Timelock PoC exactly: state (`queue[txHash]` / `scheduled_promises[N]`) is deleted before confirming the downstream call succeeded, and there is no recovery mechanism.
