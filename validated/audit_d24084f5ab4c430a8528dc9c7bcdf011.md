### Title
Permanent ERC-20 Token Loss When NEAR `ft_transfer` Promise Fails Without `error_refund` Feature — (`etc/eth-contracts/contracts/EvmErc20.sol`, `engine-precompiles/src/native.rs`)

---

### Summary

`EvmErc20.withdrawToNear()` and `EvmErc20.withdrawToEthereum()` burn ERC-20 tokens **before** calling the Aurora bridge precompile, and never check the low-level assembly call return value. When the downstream NEAR promise (`ft_transfer`) fails — e.g., because the recipient account is not registered with the NEP-141 token — and the `error_refund` Cargo feature is absent (it is **not** in the default feature set), the burned tokens are permanently lost with no refund path.

---

### Finding Description

**Step 1 — Burn-before-call with unchecked return value in `EvmErc20.sol`**

`withdrawToNear` calls `_burn` first, then invokes the `ExitToNear` precompile (`0xe921…124f`) via a raw assembly `call`. The return value `res` is captured but never inspected; a precompile failure is silently swallowed and the function returns successfully with the tokens already destroyed. [1](#0-0) 

The same pattern exists in `EvmErc20.withdrawToEthereum` and in both functions of `EvmErc20V2.sol`. [2](#0-1) [3](#0-2) 

**Step 2 — `error_refund` is not in the default feature set**

`engine/Cargo.toml` lists `default = ["std"]`; `error_refund` is a separate opt-in feature. [4](#0-3) 

**Step 3 — Without `error_refund`, the callback sets `refund: None`**

Inside `ExitToNear::run`, the `ExitToNearPrecompileCallbackArgs` is built with `refund: None` when the feature is absent: [5](#0-4) 

**Step 4 — The callback does nothing when `refund` is `None` and the promise fails**

`exit_to_near_precompile_callback` only re-mints tokens when `args.refund` is `Some`. With `None`, the else-branch returns `None` — no re-mint, no recovery: [6](#0-5) 

**Step 5 — The test suite explicitly documents the loss**

The integration test `test_exit_to_near_refund` confirms that without `error_refund`, the ERC-20 balance after a failed `ft_transfer` is `FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT` (tokens gone), not `FT_TRANSFER_AMOUNT` (refunded): [7](#0-6) 

**Step 6 — `EvmErc20.sol` (non-V2) cannot benefit from `error_refund` even if enabled**

`EvmErc20.withdrawToNear` encodes input as `\x01 || amount_b || recipient` — it does **not** embed a refund address. The `error_refund` path requires a 20-byte refund address at bytes `[1..21]` (`MIN_INPUT_SIZE` rises from 3 to 21). So `EvmErc20.sol` is permanently unprotected regardless of the feature flag. [8](#0-7) [9](#0-8) 

---

### Impact Explanation

When a user calls `EvmErc20.withdrawToNear(recipient, amount)` and the downstream NEAR `ft_transfer` promise fails (e.g., recipient account not registered with the NEP-141 token, storage deposit missing, or account does not exist), the ERC-20 tokens are already burned on Aurora and no re-mint occurs. The tokens are **permanently destroyed** — they no longer exist on Aurora EVM and were never delivered on NEAR. This constitutes **permanent freezing / loss of user funds**.

---

### Likelihood Explanation

NEAR NEP-141 tokens require each recipient account to have a storage deposit before they can receive tokens. A user who specifies a NEAR account that has not registered with the specific NEP-141 contract will trigger a failed `ft_transfer` promise. This is a realistic, user-reachable condition requiring no privileged access — any EVM token holder can trigger it by supplying an unregistered recipient address. The `error_refund` feature being off-by-default means the production WASM contract ships without the refund guard.

---

### Recommendation

1. **Enable `error_refund` by default** in `engine/Cargo.toml` (add it to the `default` feature list) so the callback always carries a refund address.
2. **Update `EvmErc20.sol`** to match the `EvmErc20V2.sol` pattern: embed `msg.sender` as the refund address in the precompile input so the callback can re-mint on failure.
3. **Check the assembly `res` value** in both `withdrawToNear` and `withdrawToEthereum` and revert if the precompile call fails, as a defense-in-depth measure.

---

### Proof of Concept

1. Deploy `EvmErc20` on Aurora (default build, no `error_refund`).
2. Mint 100 tokens to `alice` (EVM address).
3. `alice` calls `withdrawToNear(b"unregistered.near", 100)`.
   - `_burn(alice, 100)` executes — alice's ERC-20 balance drops to 0.
   - The `ExitToNear` precompile schedules an `ft_transfer` NEAR promise to `unregistered.near`.
   - The assembly `call` returns `res = 1` (precompile accepted the call synchronously).
   - `withdrawToNear` returns successfully.
4. The NEAR `ft_transfer` promise executes and **fails** (account not registered).
5. `exit_to_near_precompile_callback` fires with `args.refund = None` → no re-mint.
6. Alice has 0 ERC-20 tokens on Aurora and 0 NEP-141 tokens on NEAR. Funds are permanently lost.

This is confirmed by the existing test at: [10](#0-9)
