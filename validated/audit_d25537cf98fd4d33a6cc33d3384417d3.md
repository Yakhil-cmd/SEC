Audit Report

## Title
Unbounded Loops in `distribute_rewards` Can Permanently Halt SNS Reward Distribution — (`rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance canister's `distribute_rewards` function contains three nested, unbounded loops over reward rounds, proposal ballots, and voting neurons with no instruction-limit guard. If the per-message instruction limit is exhausted mid-execution, the entire timer message traps, all state changes (including the `latest_reward_event` update) are rolled back, and every subsequent timer invocation re-enters the same or larger loop — permanently halting SNS reward distribution. The NNS governance canister has already received an equivalent fix via a `DISTRIBUTION_MESSAGE_LIMIT` / `is_message_over_threshold` batching mechanism; the SNS canister has not.

## Finding Description

**Loop 1 — reward-round accumulation** (`rs/sns/governance/src/governance.rs`, lines 5861–5872):

```rust
for i in 1..=new_rounds_count {
    let current_reward_rate = voting_rewards_parameters.reward_rate_at(...);
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
```

`new_rounds_count` is computed at lines 5812–5814 as `(now − reward_start_timestamp_seconds) / round_duration_seconds` with no upper cap. [1](#0-0) 

The only guard on `round_duration_seconds` rejects `0` (lines 5796–5806); a value of `1` is accepted. [2](#0-1) 

After one year at `round_duration_seconds = 1`, `new_rounds_count ≈ 31.5 million`. Each iteration calls `reward_rate_at` (floating-point + `Decimal` arithmetic). At ~1000 instructions/iteration this is ~31.5 billion instructions — far exceeding the 5-billion per-message limit. [3](#0-2) 

**Loop 2 — ballot aggregation** (lines 5894–5930): Iterates over every ballot of every settled proposal with no bound check. [4](#0-3) 

**Loop 3 — neuron maturity update** (lines 5954–5997): Iterates over every neuron that voted, with no instruction-limit check. [5](#0-4) 

A grep search for `is_message_over_threshold` and `DISTRIBUTION_MESSAGE_LIMIT` across all of `rs/sns/**/*.rs` returns **no matches** — confirming the SNS canister has no equivalent guard.

`distribute_rewards` is called synchronously inside `run_periodic_tasks` (lines 5509–5513): [6](#0-5) 

When the instruction limit is exceeded, the IC runtime traps the message; all mutations — including the `latest_reward_event` update written at the end of `distribute_rewards` — are rolled back. On the next timer firing, `should_distribute_rewards()` returns `true` again because `latest_reward_event` was never advanced, and `distribute_rewards` is called again with the same (or larger) `new_rounds_count`. This creates a permanent, self-reinforcing DoS.

**Contrast with the NNS fix.** `rs/nns/governance/src/reward/distribution.rs` introduces `DISTRIBUTION_MESSAGE_LIMIT = 1_000_000_000` (line 17) and breaks the neuron-update loop across multiple timer messages via `is_over_instructions_limit()` (line 184): [7](#0-6) [8](#0-7) [9](#0-8) 

The SNS canister has no equivalent mechanism.

## Impact Explanation

Permanent halting of SNS voting reward distribution: neurons stop accumulating maturity, breaking the economic incentive model of the SNS. The SNS governance canister continues to function for other operations (proposals, voting) but reward distribution is permanently halted with no on-chain recovery path short of a canister upgrade. This matches the allowed impact: **High — Significant SNS security impact with concrete user or protocol harm** ($2,000–$10,000).

## Likelihood Explanation

Two realistic, unprivileged trigger conditions exist:

1. **Small `round_duration_seconds`**: A value of `1` is accepted by the proto validation. This can be set at SNS genesis or via a governance proposal passing with a simple majority of voting power — reachable by any sufficiently large neuron holder coalition without any privileged key. After 60 days at `round_duration_seconds = 1`, `new_rounds_count = 5,184,000`, which is sufficient to exhaust the instruction budget in loop 1 alone.

2. **Large neuron count + many settled proposals**: An SNS with `max_number_of_neurons` set high and many active voters across many proposals accumulates a large `neuron_id_to_reward_shares` map. The per-neuron work in loop 3 (stable-memory lookup + arithmetic) is non-trivial; with tens of thousands of neurons this loop alone can exhaust the 5-billion-instruction budget.

Both conditions are reachable by unprivileged SNS participants (token holders who create neurons and vote).

## Recommendation

1. Apply the same batching mechanism used in NNS governance: move the neuron-maturity update (loop 3) into a persistent state machine stored in stable memory, processed in batches across multiple timer messages using `is_message_over_threshold`.
2. Cap `new_rounds_count` to a safe maximum per invocation (e.g., 1000 rounds), carrying over the remainder to the next timer firing, or add an `is_message_over_threshold` guard inside loop 1.
3. Add a validation lower bound on `round_duration_seconds` (e.g., ≥ 3600 seconds) to prevent pathologically small values from being set via governance proposal.

## Proof of Concept

1. Deploy an SNS with `VotingRewardsParameters { round_duration_seconds: Some(1), ... }`.
2. Wait 60 days. `new_rounds_count = 60 × 86400 = 5,184,000`.
3. Observe the next `run_periodic_tasks` timer fire and call `distribute_rewards`.
4. Loop 1 executes 5.18 million iterations of `reward_rate_at(...)` (floating-point + `Decimal` operations). At ~1000 instructions/iteration this is ~5.18 billion instructions — exceeding the 5-billion per-message limit.
5. The message traps; `latest_reward_event` is not updated.
6. Every subsequent timer firing repeats steps 3–5 with an ever-growing `new_rounds_count`.
7. No rewards are ever distributed; the SNS reward system is permanently halted.

A deterministic integration test using PocketIC can reproduce this by setting `round_duration_seconds = 1`, advancing the replica clock by 60 days, triggering `run_periodic_tasks`, and asserting that `latest_reward_event` remains unchanged across repeated invocations.
