### Title
Permanent Freeze of wNEAR ERC-20 Tokens When NEAR Transfer to Non-Existent Recipient Fails in wNEAR Unwrap Exit Flow - (File: engine/src/contract_methods/connector.rs)

---

### Summary

When a user exits wNEAR ERC-20 tokens from Aurora using the `:unwrap` suffix (triggering the two-step `near_withdraw` → `PromiseAction::Transfer` flow), the ERC-20 tokens are burned and the wNEAR is unwrapped before the final NEAR transfer to the recipient. If the recipient NEAR account does not exist, the `PromiseAction::Transfer` fails and NEAR is refunded to the Aurora contract, but there is no callback to re-mint the burned ERC-20 tokens. The user's funds are permanently frozen.

---

### Finding Description

The `ExitToNear` precompile handles wNEAR unwrapping via a two-step promise chain:

**Step 1 (base promise):** `near_withdraw` is called on the wNEAR NEP-141 contract to unwrap wNEAR into NEAR. This is set up in `exit_erc20_token_to_near` when the `:unwrap` suffix is detected and the caller is the wNEAR ERC-20: [1](#0-0) 

A `TransferNearArgs` is populated with the user-supplied `receiver_account_id` and the amount.

**Step 2 (callback):** `exit_to_near_precompile_callback` is called. If `near_withdraw` succeeded, it issues a `PromiseAction::Transfer` to `args.target_account_id` with **no further callback**: [2](#0-1) 

The `promise_create_batch` call returns a `promise_id` that is returned but has no callback attached. If the NEAR `Transfer` action fails (e.g., recipient account does not exist on NEAR), NEAR is refunded to the Aurora contract by the NEAR runtime, but the ERC-20 tokens that were burned in the EVM before the promise chain started are **never re-minted**.

The `error_refund` feature only guards against failure of the **first** promise (`near_withdraw`). It does not guard against failure of the second `PromiseAction::Transfer`: [3](#0-2) 

The `refund` field in `ExitToNearPrecompileCallbackArgs` is only consumed in the `else if` branch when `promise_result(0)` is **not** `Successful`. Once `near_withdraw` succeeds, the refund path is bypassed entirely: [4](#0-3) 

---

### Impact Explanation

**Impact: Critical — Permanent freezing of funds.**

When the second `PromiseAction::Transfer` fails:
- The user's wNEAR ERC-20 tokens on Aurora are permanently burned (gone from the EVM state).
- The NEAR tokens are refunded to the Aurora contract's NEAR balance, not to the user.
- There is no recovery path: no callback, no re-mint, no admin rescue function for this specific scenario.

The `ExitToNearPrecompileCallbackArgs` struct confirms only one level of refund is modeled: [5](#0-4) 

---

### Likelihood Explanation

**Likelihood: Medium.**

Any EVM user holding wNEAR ERC-20 tokens on Aurora can trigger this by calling `withdrawToNear` with the `:unwrap` suffix and specifying a NEAR account ID that does not exist on-chain (e.g., a typo, a freshly generated account that has not been funded/created, or a deliberately crafted non-existent account). The NEAR account ID format is valid but the account itself need not exist. This is a realistic user error scenario (wrong recipient address) that permanently destroys funds with no recourse.

---

### Recommendation

Attach a second callback to the `PromiseAction::Transfer` promise created in `exit_to_near_precompile_callback`. If the transfer fails, this callback should re-mint the burned ERC-20 tokens to the original sender (the `refund_address` already tracked in `RefundCallArgs` when `error_refund` is enabled). Alternatively, validate that the recipient NEAR account exists before proceeding with the unwrap, or use a two-phase approach where the NEAR is held in escrow until the transfer is confirmed.

---

### Proof of Concept

1. User holds wNEAR ERC-20 tokens on Aurora.
2. User calls `withdrawToNear(bytes,uint256)` on the wNEAR ERC-20 contract with:
   - `recipient = b"nonexistent-account-xyz.near:unwrap"`
   - `amount = N`
3. The ERC-20 contract burns `N` tokens and calls the `ExitToNear` precompile.
4. The precompile detects the `:unwrap` suffix and the wNEAR ERC-20 caller, sets `method = "near_withdraw"` and `transfer_near = Some(TransferNearArgs { target_account_id: "nonexistent-account-xyz.near", amount: N })`.
5. The `near_withdraw` promise executes successfully — wNEAR is unwrapped, NEAR is now held by Aurora.
6. `exit_to_near_precompile_callback` fires, sees `PromiseResult::Successful`, issues `PromiseAction::Transfer { amount: N }` to `"nonexistent-account-xyz.near"`.
7. NEAR runtime rejects the transfer (account does not exist), refunds NEAR to Aurora.
8. No callback fires. ERC-20 tokens remain burned. User has lost `N` wNEAR permanently.

The existing test `test_exit_to_near_refund` confirms the codebase is aware of the unregistered-recipient failure mode for the `ft_transfer` path, but no analogous test or guard exists for the `near_withdraw` + `Transfer` two-step path: [6](#0-5)
