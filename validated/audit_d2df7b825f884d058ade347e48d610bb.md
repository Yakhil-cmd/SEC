### Title
`native_per_pubdata` Truncates to Zero via Floor Division When `pubdata_price < native_price`, Making Pubdata Effectively Free — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio is computed with floor (truncating) integer division. When the operator-set `pubdata_price` is less than `native_price`, the result truncates to zero. This silently zeroes out all native-resource charges for pubdata, allowing any transaction sender to generate maximum pubdata at no native cost and receive an inflated gas refund.

---

### Finding Description

In `validation_impl.rs` line 142, the bootloader computes how many native resource units one byte of pubdata costs:

```rust
// We checked native_price != 0 above
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`wrapping_div` is floor (truncating) integer division. Whenever `pubdata_price < native_price` (e.g., `pubdata_price = 99`, `native_price = 100`), the result is `0`.

By contrast, `native_per_gas` is computed with **ceiling** division:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

The same floor-division pattern is replicated in the public API helper:

```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

Once `native_per_pubdata = 0`, every downstream consumer of this value silently treats pubdata as free:

**1. Intrinsic pubdata overhead becomes zero** in `create_resources_for_tx`:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// = 0 * intrinsic_pubdata = 0
``` [4](#0-3) 

**2. Runtime pubdata charges become zero** in `get_resources_to_charge_for_pubdata`:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // 0 * N = 0
    .ok_or(out_of_native_resources!())?;
``` [5](#0-4) 

**3. `check_enough_resources_for_pubdata` always returns `has_enough = true`** because the required native is 0, so the post-execution pubdata revert path is never triggered. [6](#0-5) 

**4. `delta_gas` in `compute_gas_refund` is reduced**, because `native_used` does not include any pubdata cost, so the user receives a larger gas refund:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
``` [7](#0-6) 

The documentation confirms the intended semantics: `nativePerPubdata := pubdataPrice / nativePrice` is supposed to represent a non-zero cost when `pubdata_price > 0`. [8](#0-7) 

---

### Impact Explanation

When `pubdata_price < native_price`:

- Every transaction pays **zero** native resources for pubdata, regardless of how many storage writes it performs.
- The operator's intended pubdata fee is silently discarded — the operator loses all pubdata revenue.
- A transaction sender can write the maximum number of storage slots (generating tens of kilobytes of pubdata) and pay only the EVM gas cost for the `SSTORE` opcodes, with no native-resource penalty.
- Because `delta_gas` is not inflated by pubdata native usage, the user also receives a larger gas refund than intended.
- Block pubdata capacity can be exhausted cheaply, degrading throughput for all other users.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a realistic operator configuration. `native_price` reflects the cost of a single proving cycle (a compute-heavy resource), while `pubdata_price` reflects the cost of publishing one byte to the DA layer. On chains using cheap DA (e.g., a validium or a low-cost DA layer), `pubdata_price` can legitimately be set below `native_price`. No privileged access is required to exploit this once the condition holds — any unprivileged transaction sender benefits automatically.

---

### Recommendation

Replace the floor division for `native_per_pubdata` with ceiling division, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — truncates to 0 when pubdata_price < native_price)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — matches native_per_gas treatment)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs`. This ensures that any non-zero `pubdata_price` always results in `native_per_pubdata >= 1`, preserving the operator's intent to charge for pubdata.

---

### Proof of Concept

**Setup:**
- `native_price = 100`
- `pubdata_price = 99` (operator intends to charge for pubdata, but less than native)
- `native_per_pubdata = 99 / 100 = 0` (floor division truncates)

**Attack:**
1. Deploy a contract that writes N distinct storage slots (each write produces 32 bytes of pubdata key + compressed value).
2. Send a transaction calling this contract with a normal gas limit.
3. Observe: `get_resources_to_charge_for_pubdata` returns 0 native cost regardless of N.
4. `check_enough_resources_for_pubdata` returns `has_enough = true` for any N.
5. Transaction succeeds; user pays only EVM gas for `SSTORE`, zero native for pubdata.
6. Repeat across multiple transactions to fill the block's pubdata limit at minimal cost, while the operator collects no pubdata fees.

**File:** docs/double_resource_accounting.md (L37-50)
```markdown
First we define the ratio between EVM gas and native resource as:
  `nativePerGas := gasPrice/nativePrice`
Note: for call simulation we use a constant for it, as gasPrice might be set to 0.

Next we define the limit for the native resource as:
  `nativeLimit := gasLimit * nativePerGas`

Then we process the transaction, charging both Ergs for EE execution and native resource for any kind of computation (EE, bootloader or system work).

If execution doesn't run out of native resources, we first charge for pubdata from native resource.
Then we compute the difference between the implicit gas used derived from native resource consumption and the gas used by EEs from the ergs used. We call this value `deltaGas`.
  `deltaGas := (nativeUsed / nativePerGas) - gasUsed`

If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs. This ensures that gas estimation will include additional gas to cover for native resources using just base fee. We expect the base fee to be enough to cover most transactions without the need of additional gas.
