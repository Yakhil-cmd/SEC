Audit Report

## Title
Unbounded neuron iteration in SNS `compute_ballots_for_new_proposal` enables permanent governance DoS — (`rs/sns/governance/src/governance.rs`)

## Summary
The SNS Governance canister's `compute_ballots_for_new_proposal` performs a full linear scan of `self.proto.neurons` in a single synchronous message execution with no instruction-limit guard. Any neuron holder can trigger this path via `make_proposal`. If the SNS neuron population reaches `max_number_of_neurons`, the scan exhausts the IC per-message instruction limit (~40 billion instructions), causing every subsequent `make_proposal` call to trap and permanently blocking all SNS governance proposals, including canister upgrades. The SNS `cast_vote_and_cascade_follow` has the same structural deficiency on the voting path.

## Finding Description
**Primary — `compute_ballots_for_new_proposal`:**

Lines 5255–5280 of `rs/sns/governance/src/governance.rs` iterate unconditionally over every entry in `self.proto.neurons` (a heap-resident `BTreeMap<String, Neuron>`) to build the electoral roll:

```rust
for (k, v) in self.proto.neurons.iter() {
    ...
    electoral_roll.insert(k.clone(), Ballot { ... });
}
```

There is no `is_over_instructions_limit()` check, no pagination, and no deferred processing. The entire scan must complete within one message execution. `check_neuron_population_can_grow` (lines 6363–6379) enforces a configurable `max_number_of_neurons` ceiling but does not bound the per-message instruction cost of iterating that many neurons.

**Secondary — `cast_vote_and_cascade_follow`:**

Line 3749 runs a plain `while !induction_votes.is_empty()` BFS over the follower graph with no instruction-limit pause. A grep over `rs/sns/governance/**` returns zero matches for `is_over_instructions_limit`, confirming no guard exists anywhere in the SNS governance codebase.

**Contrast with NNS:**

The NNS counterpart in `rs/nns/governance/src/voting.rs` was explicitly retrofitted with an `is_over_instructions_limit()` guard that pauses and resumes across heartbeat rounds. The SNS has no equivalent mechanism, despite sharing the same architectural risk.

## Impact Explanation
This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

Once the neuron count is high enough to exhaust the instruction limit, every `make_proposal` call traps with `CanisterInstructionLimitExceeded`. Because SNS governance is the sole path for submitting proposals, this permanently blocks:
- All governance proposals (parameter changes, treasury transfers, upgrade proposals)
- Any SNS canister upgrade (which requires a governance proposal)

The SNS becomes effectively frozen. Recovery requires direct NNS intervention to upgrade the SNS governance canister — a slow, high-friction process that may not be available for all SNS deployments.

## Likelihood Explanation
Any neuron holder can call `make_proposal` — no privileged role is required. An attacker populates the SNS to `max_number_of_neurons` by staking the minimum amount (`neuron_minimum_stake_e8s`) from many accounts. The cost is proportional to `max_number_of_neurons × neuron_minimum_stake_e8s`; for SNS deployments with low minimum stakes this is economically feasible. Once the threshold is crossed, the DoS is persistent and affects all users — the attacker does not need to keep calling `make_proposal`. The attack is non-reversible without external intervention because the frozen governance cannot lower `max_number_of_neurons` via proposal.

## Recommendation
1. Add an `is_over_instructions_limit()` guard inside the `compute_ballots_for_new_proposal` loop and implement a chunked/background ballot-creation mechanism (analogous to the NNS voting-power snapshot) so the full neuron scan is not required in a single message execution.
2. Add the same pause-and-resume instruction-limit guard to SNS `cast_vote_and_cascade_follow` as exists in the NNS `rs/nns/governance/src/voting.rs` implementation.
3. Benchmark `compute_ballots_for_new_proposal` at the current `MAX_NUMBER_OF_NEURONS_CEILING` using the NNS benchmark infrastructure as a template, and enforce a hard cap that keeps the scan well under the instruction limit until a chunked approach is implemented.
4. Consider migrating SNS neurons to stable storage (as the NNS did) to reduce per-neuron access cost.

## Proof of Concept
1. Deploy an SNS with `max_number_of_neurons` set to a large value and `neuron_minimum_stake_e8s` set to the minimum.
2. Create neurons up to the cap by staking the minimum from many accounts.
3. Call `make_proposal` from any neuron holder.
4. Observe the call traps with `CanisterInstructionLimitExceeded` (IC error code 522) because `compute_ballots_for_new_proposal` iterates all neurons in a single message.
5. Confirm all subsequent `make_proposal` calls also trap, freezing SNS governance.

A deterministic integration test can be written using PocketIC: instantiate an SNS, use `claim_swap_neurons` to bulk-create neurons up to the cap, then assert that `make_proposal` returns a trap error. The NNS benchmark at `rs/nns/governance/src/neuron_store/benches.rs` (lines 481–499) provides a template for measuring per-neuron instruction cost to determine the exact threshold.
