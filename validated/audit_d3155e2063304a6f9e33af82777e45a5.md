### Title
Fee-on-Transfer NEP-141 Token Causes ERC-20 Over-Minting and Insolvency - (`engine/src/engine.rs`)

### Summary

The `receive_erc20_tokens` function in Aurora Engine's EVM backend mints ERC-20 tokens equal to the `amount` field reported by the NEP-141 contract in the `ft_on_transfer` callback, without verifying the actual NEP-141 balance change on Aurora's account. For fee-on-transfer NEP-141 tokens, the actual tokens received by Aurora are `amount - fee`, but the ERC-20 minted is `amount`. This inflates the ERC-20 supply beyond its NEP-141 backing, causing insolvency for the last exiters.

### Finding Description

The `ft_on_transfer` entry point in `engine/src/contract_methods/connector.rs` dispatches to `receive_erc20_tokens` for any NEP-141 token that is not the base ETH connector: [1](#0-0) 

Inside `receive_erc20_tokens`, the minted ERC-20 amount is taken directly from `args.amount.as_u128()` — the value reported by the NEP-141 contract in the callback argument — without any independent verification of Aurora's actual NEP-141 balance change: [2](#0-1) 

The NEP-141 `ft_transfer_call` standard does not prohibit fee-on-transfer tokens. If a registered NEP-141 token deducts a fee from the receiver's credited balance (i.e., Aurora receives `amount - fee` but `ft_on_transfer` is called with `amount`), Aurora mints `amount` ERC-20 tokens while holding only `amount - fee` NEP-141 tokens. The same structural flaw exists in `receive_base_tokens`: [3](#0-2) 

The `TransferNearArgs` struct used in the wNEAR unwrap callback also hardcodes `exit_params.amount` without reading the actual result of `near_withdraw`: [4](#0-3) 

And the callback transfers exactly that pre-computed amount, not the actual NEAR received: [5](#0-4) 

### Impact Explanation

Each time a fee-on-transfer NEP-141 token is bridged via `ft_transfer_call`, the ERC-20 supply grows faster than the NEP-141 backing. When users attempt to exit (burn ERC-20 → receive NEP-141 via `ft_transfer`), the last exiters find Aurora's NEP-141 balance insufficient. Their ERC-20 tokens are burned but the NEP-141 transfer fails, permanently freezing their funds. This is **insolvency** of the specific ERC-20 mirror token.

### Likelihood Explanation

`deploy_erc20_token` is a public function with no visible caller restriction in the production code path, meaning any user can register an arbitrary NEP-141 token as an ERC-20 mirror on Aurora. An attacker can deploy a fee-on-transfer NEP-141 token, register it, bridge tokens (receiving over-minted ERC-20), sell those ERC-20 tokens to other users, and leave those users unable to redeem. The entry path is fully unprivileged. [6](#0-5) 

### Recommendation

In `receive_erc20_tokens`, read Aurora's NEP-141 balance of the token before and after the `ft_on_transfer` callback executes (or use the actual balance at callback time) to determine the true received amount, and mint only that amount of ERC-20 tokens. Alternatively, explicitly reject NEP-141 tokens that implement fee-on-transfer semantics during the `deploy_erc20_token` registration step.

### Proof of Concept

1. Deploy a NEP-141 token `fee_token.near` that deducts 10% on every transfer (receiver gets 90% of the specified amount).
2. Call `deploy_erc20_token` on Aurora to register `fee_token.near` → `FEE_ERC20`.
3. Call `ft_transfer_call(receiver_id=aurora, amount=1000, msg=<evm_address>)` on `fee_token.near`. Aurora receives 900 NEP-141 tokens but `ft_on_transfer` is called with `amount=1000`.
4. `receive_erc20_tokens` mints 1000 `FEE_ERC20` to the EVM address.
5. Attacker sells 1000 `FEE_ERC20` to a victim.
6. Victim calls `withdrawToNear` (burns 1000 `FEE_ERC20`), triggering `ft_transfer` of 1000 `fee_token.near` from Aurora. Aurora only holds 900 → transfer fails → victim's funds are permanently frozen. [7](#0-6)

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());
