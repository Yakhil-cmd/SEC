### Title
Inconsistent `minimal_gas_used` Floor Between ZK and Ethereum Transaction Flows Allows Underpayment of Intrinsic Gas - (`basic_bootloader/src/bootloader/transaction_flow/`)

---

### Summary

The ZK transaction flow (`zk/validation_impl.rs`) sets `minimal_gas_used = 0` when the `eip_7623` feature is disabled, while the Ethereum transaction flow (`ethereum/validation_impl.rs`) always sets `minimal_gas_used = TX_INTRINSIC_GAS = 21,000`. Both flows feed this value into the shared `compute_gas_refund` function as the floor for `gas_used`. This inconsistency allows users submitting ZK-type transactions to pay less than the intended 21,000-gas minimum by exploiting EVM storage-clearing refunds.

---

### Finding Description

**Ethereum flow** (`ethereum/validation_impl.rs`, lines 159–162):

```rust
#[cfg(not(feature = "eip_7623"))]
{
    (num_tokens, TX_INTRINSIC_GAS)   // minimal_gas_used = 21,000
}
``` [1](#0-0) 

This value is stored as `minimal_gas_to_charge` in `EthereumTxContext` and passed to `compute_gas_refund` as the floor. [2](#0-1) 

**ZK flow** (`zk/validation_impl.rs`, line 98 + `compute_calldata_tokens`):

```rust
let (calldata_tokens, minimal_gas_used) = compute_calldata_tokens(system, calldata);
``` [3](#0-2) 

`compute_calldata_tokens` returns `(num_tokens, 0)` when `eip_7623` is disabled:

```rust
#[cfg(not(feature = "eip_7623"))]
{
    (num_tokens, 0)   // minimal_gas_used = 0
}
``` [4](#0-3) 

This zero is stored as `minimal_ergs_to_charge = 0` in `TxContextForPreAndPostProcessing`: [5](#0-4) 

In `before_refund`, the ZK flow passes `min_gas_used = 0` to `compute_gas_refund`: [6](#0-5) 

In `compute_gas_refund`, the floor is applied as:

```rust
let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);
``` [7](#0-6) 

With `minimal_gas_used = 0`, there is no floor. EVM refunds (capped at 1/5 of `gas_used` per EIP-3529) can reduce `gas_used` below 21,000.

---

### Impact Explanation

An attacker submitting a ZK-type transaction can pay less than the 21,000-gas intrinsic minimum:

**Concrete scenario:**
1. In a prior transaction, set a storage slot (costs ~20,000 gas).
2. Submit a ZK transaction that clears the slot (warm SSTORE: ~2,900 gas execution).
3. EVM refund counter = 4,800 gas (EIP-3529 SSTORE-clear refund).
4. `gas_used` before refund = 21,000 (intrinsic) + 2,900 (execution) = 23,900.
5. EVM refund cap = 23,900 / 5 = 4,780.
6. Applied refund = min(4,800, 4,780) = 4,780.
7. Final `gas_used` = 23,900 − 4,780 = **19,120 < 21,000**.

The user pays `19,120 × gas_price` instead of `21,000 × gas_price`. The protocol loses the difference (~1,880 gas worth of fees per transaction). At scale, this represents a systematic underpayment of intrinsic gas fees for ZK-type transactions.

---

### Likelihood Explanation

- **Reachable by any unprivileged user** submitting a ZK-type transaction.
- Requires only a prior transaction that set a storage slot (standard contract interaction).
- No privileged access, oracle manipulation, or external dependency required.
- The attack is cheap to execute repeatedly and scales linearly with transaction volume.

---

### Recommendation

In `compute_calldata_tokens`, when `eip_7623` is disabled, return `TX_INTRINSIC_GAS` as the floor (matching the Ethereum flow behavior) instead of `0`:

```rust
#[cfg(not(feature = "eip_7623"))]
{
    (num_tokens, TX_INTRINSIC_GAS)  // enforce 21,000 gas minimum floor
}
``` [4](#0-3) 

This ensures both transaction flows enforce the same minimum gas floor, consistent with EVM semantics.

---

### Proof of Concept

1. Deploy a contract with a storage write in its constructor (sets slot 0 to 1).
2. Submit a ZK-type transaction calling a function that clears slot 0 (`SSTORE(0, 0)`).
3. Observe that `gas_used` reported in the ZK flow is below 21,000 (e.g., ~19,120).
4. Submit the same transaction via the Ethereum flow; observe `gas_used` is floored at 21,000.
5. The difference in fees paid confirms the inconsistency.

The root cause is the divergence between `compute_calldata_tokens` returning `(num_tokens, 0)` for the ZK flow vs. `(num_tokens, TX_INTRINSIC_GAS)` for the Ethereum flow when `eip_7623` is disabled. [1](#0-0) [4](#0-3) [7](#0-6)
