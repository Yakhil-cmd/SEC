Audit Report

## Title
VetKD Share Verification Accepts Cryptographically Invalid Content — (`rs/crypto/src/vetkd/mod.rs`)

## Summary

`verify_encrypted_key_share_internal` verifies only the Ed25519 node signature over the raw bytes of `encrypted_key_share`, but never attempts to deserialize those bytes as an `EncryptedKeyShare`. A single Byzantine subnet node can sign empty or malformed bytes with its legitimate key, pass share verification, enter the validated pool, and permanently block VetKD key derivation for any targeted request because `combine_encrypted_key_shares_internal` aborts on deserialization failure before the per-share filtering fallback is ever reachable.

## Finding Description

**Verification gap.** `verify_encrypted_key_share_internal` performs exactly two operations: it reads the registry version from the threshold-sig data store, then calls `BasicSigVerifierInternal::verify_basic_sig` over `key_share.encrypted_key_share` (the raw bytes). [1](#0-0)  There is no call to `EncryptedKeyShare::deserialize` and no check that the bytes represent a valid BLS12-381 group element triple. A valid Ed25519 signature over empty bytes causes the function to return `Ok(())`.

**Deserialization enforced only at combination time.** `combine_encrypted_key_shares_internal` iterates over every share and calls `EncryptedKeyShare::deserialize` on each one. [2](#0-1)  `EncryptedKeyShare::deserialize` immediately rejects any byte slice whose length differs from `Self::BYTES`. [3](#0-2)  The `.collect::<Result<_, _>>()?` at line 366 means a single deserialization failure causes an early return with `VetKdKeyShareCombinationError::InvalidArgumentEncryptedKeyShare` before `combine_all` is ever called. [4](#0-3) 

**Fallback path is unreachable.** The `combine_valid_shares` fallback is only triggered when `combine_all` returns `EncryptedKeyCombinationError::InvalidShares`. [5](#0-4)  Because the deserialization failure at line 366 returns before `combine_all` is called at line 373, the fallback is structurally unreachable for this failure mode.

**Validated pool accepts the malformed share.** In `signer.rs`, `verify_encrypted_key_share` is called and on success the share is returned as `Ok(IDkgMessage::VetKdKeyShare(share))`. [6](#0-5)  The caller then issues `IDkgChangeAction::MoveToValidated(share)`. [7](#0-6) 

**Combiner uses all validated shares.** `combine_shares` in the chain-key payload builder passes the full validated share map directly to `combine_encrypted_key_shares`. [8](#0-7) 

**Protobuf layer accepts arbitrary bytes.** The `TryFrom<pb::VetKdKeyShare>` implementation wraps `value.encrypted_key_share` in `VetKdEncryptedKeyShareContent` without any length or content check, so the P2P layer accepts any byte slice for this field. [9](#0-8) 

## Impact Explanation

A single Byzantine subnet node below the consensus fault threshold can permanently block VetKD key derivation for any targeted canister request. Once the malformed share is in the validated pool it is included in every combination attempt; because deserialization fails before any per-share filtering occurs, no number of honest shares can rescue the combination. This matches the allowed ICP bounty impact: **High ($2,000–$10,000) — Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS.**

## Likelihood Explanation

Any single compromised or malicious subnet node can execute this attack. The node requires only its own registered Ed25519 signing key (which it legitimately holds) and the ability to emit a `VetKdKeyShare` protobuf with an empty or wrong-length `encrypted_key_share` field. No threshold corruption, no special privileges, and no external dependencies are required. The attack is repeatable for every new VetKD request targeting the subnet.

## Recommendation

Add `EncryptedKeyShare::deserialize` validation inside `verify_encrypted_key_share_internal` immediately after the `BasicSigVerifierInternal::verify_basic_sig` call. If deserialization fails, return a `VetKdKeyShareVerificationError` — either a new `InvalidEncryptedKeyShareContent` variant or reuse `VerificationError` with `CryptoError::InvalidArgument`. This ensures any share that cannot be deserialized is rejected at admission time and never enters the validated pool.

## Proof of Concept

```rust
// Unit test sketch
let byzantine_node_id = /* valid subnet node with registered Ed25519 key */;
let empty_content = VetKdEncryptedKeyShareContent(vec![]);
let sig = ed25519_sign(&empty_content.0, &byzantine_node_sk);
let malformed_share = VetKdEncryptedKeyShare {
    encrypted_key_share: empty_content,
    node_signature: sig,
};

// verify_encrypted_key_share returns Ok(()) — signature is valid over empty bytes
assert_eq!(
    crypto.verify_encrypted_key_share(byzantine_node_id, &malformed_share, &args),
    Ok(())  // BUG: should be Err(...)
);

// combine_encrypted_key_shares fails when this share is included alongside
// threshold-many honest shares
let mut shares = honest_shares;
shares.insert(byzantine_node_id, malformed_share);
assert!(matches!(
    crypto.combine_encrypted_key_shares(&shares, &args),
    Err(VetKdKeyShareCombinationError::InvalidArgumentEncryptedKeyShare)
));
```
