Audit Report

## Title
Unauthenticated `ClaimOrRefresh { by: NeuronIdOrSubaccount }` Allows Any Caller to Dilute Another Neuron's Age - (File: `rs/nns/governance/src/governance.rs`)

## Summary

The NNS Governance canister's `manage_neuron_internal` dispatches `ClaimOrRefresh { by: NeuronIdOrSubaccount }` to `refresh_neuron_by_id_or_subaccount` without passing or checking the caller identity. When a neuron's ledger balance exceeds its cached stake — as happens after a top-up — any non-anonymous ingress caller can trigger `update_stake_adjust_age` on the victim's neuron, permanently diluting its `aging_since_timestamp_seconds` and reducing the age bonus (up to 25% extra voting power) without the owner's consent.

## Finding Description

In `manage_neuron_internal`, the `By::NeuronIdOrSubaccount` branch at line 6132 calls `refresh_neuron_by_id_or_subaccount(id, claim_or_refresh)` without forwarding the `caller` parameter:

```rust
Some(By::NeuronIdOrSubaccount(_)) => {
    let id = mgmt.get_neuron_id_or_subaccount()?.ok_or_else(...)?;
    self.refresh_neuron_by_id_or_subaccount(id, claim_or_refresh)
        .await
        .map(ManageNeuronResponse::claim_or_refresh_neuron_response)
}
```

`refresh_neuron_by_id_or_subaccount` (lines 5875–5896) accepts no `caller` parameter and performs no authorization check before calling `refresh_neuron`. Inside `refresh_neuron`, when `cached_neuron_stake_e8s < balance` (`Ordering::Less`, line 5950–5952), `update_stake_adjust_age` is called unconditionally.

`update_stake_adjust_age` (lines 988–1039 of `rs/nns/governance/src/neuron/types.rs`) uses `combine_aged_stakes` to compute a weighted-average age, then overwrites `aging_since_timestamp_seconds` (lines 1034–1038). The docstring explicitly documents the dilution: a neuron with 10 ICP aged 3 years receiving 5 ICP new stake results in age `(10×3)/(10+5) = 2 years`. This is permanent — the neuron must re-accumulate the lost age.

By contrast, `refresh_voting_power` (lines 5675–5690) enforces `is_authorized_to_vote(caller)` before modifying neuron state, demonstrating that the authorization pattern exists and is applied elsewhere but was omitted here.

The `By::Memo` and `By::MemoAndController` paths both receive the `caller` and enforce ownership via `claim_or_refresh_neuron_by_memo_and_controller`. Only the `By::NeuronIdOrSubaccount` path skips this entirely.

Note: the claim misrepresents `test_refresh_neuron_by_subaccount_by_proxy` (lines 5025–5029) — that test uses `TEST_NEURON_1_OWNER_PRINCIPAL` for both `owner` and `caller`, so it does not actually prove a different principal can call it. However, the function signature itself (no `caller` parameter, no auth check) confirms the absence of authorization independently of the test.

## Impact Explanation

This matches **High ($2,000–$10,000): Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints.**

An attacker can permanently reduce a victim neuron's age bonus without the owner's consent. The age bonus contributes up to 25% additional voting power and proportionally higher staking rewards after 4 years of accumulation. The dilution is irreversible — the neuron must re-accumulate the lost age from scratch. For large, long-lived neurons this represents a meaningful and permanent reduction in governance influence and economic return. The constraint is that the victim's ledger balance must exceed the cached stake at the time of the attack (i.e., a top-up must be pending).

## Likelihood Explanation

The NNS ledger is fully public; any observer can watch for ICP transfers to governance staking subaccounts (derivable from controller principal + memo). The neuron ID is publicly enumerable via `list_neurons`. The `manage_neuron` endpoint accepts any non-anonymous ingress caller. The attacker needs no funds, no privileged access, and no special knowledge. The attack is repeatable: every time a neuron owner tops up their staking subaccount, the window exists until they call `ClaimOrRefresh` themselves. The attacker can also target neurons whose owners never intend to call `ClaimOrRefresh` (e.g., they topped up but changed their mind), forcing an unwanted age dilution.

## Recommendation

Add a caller authorization check in `refresh_neuron_by_id_or_subaccount` (or at the `By::NeuronIdOrSubaccount` dispatch site) mirroring the pattern in `refresh_voting_power`:

```rust
async fn refresh_neuron_by_id_or_subaccount(
    &mut self,
    id: NeuronIdOrSubaccount,
    claim_or_refresh: &ClaimOrRefresh,
    caller: &PrincipalId,  // add caller
) -> Result<NeuronId, GovernanceError> {
    // resolve nid ...
    let is_authorized =
        self.with_neuron(&nid, |neuron| neuron.is_authorized_to_vote(caller))?;
    if !is_authorized {
        return Err(GovernanceError::new_with_message(
            ErrorType::NotAuthorized, ...));
    }
    self.refresh_neuron(nid, subaccount, claim_or_refresh).await
}
```

The `caller` is already available in `manage_neuron_internal` and must be threaded through. This mirrors the existing `is_authorized_to_vote` guard used by `refresh_voting_power`.

## Proof of Concept

1. Identify victim neuron `N` (10 ICP staked, aged 4 years) via `list_neurons`.
2. Observe via the public ICP ledger that the victim transferred 10 ICP to the neuron's staking subaccount (governance subaccount derived from controller + memo).
3. Before the victim calls `ClaimOrRefresh`, submit from any non-anonymous principal:
   ```
   manage_neuron({
     neuron_id_or_subaccount: NeuronId(N),
     command: ClaimOrRefresh { by: NeuronIdOrSubaccount({}) }
   })
   ```
4. Governance calls `refresh_neuron_by_id_or_subaccount` → `refresh_neuron` → ledger balance = 20 ICP > cached 10 ICP → `update_stake_adjust_age(20 ICP, now)`.
5. `combine_aged_stakes(10 ICP, 4 years, 10 ICP, 0)` → new age = `(10×4)/(10+10) = 2 years`. `aging_since_timestamp_seconds` is overwritten to `now - 2_years`.
6. Victim calls `ClaimOrRefresh` — hits `Ordering::Equal`, no further change. Neuron now has 2 years of age instead of 4, permanently losing 2 years of age bonus.

A deterministic unit test can be written using the existing `create_mature_neuron` + `fake::FakeDriver` harness: create a neuron with known age, simulate a top-up by setting ledger balance above cached stake, call `manage_neuron` with `ClaimOrRefresh { by: NeuronIdOrSubaccount }` from a different principal, and assert that `aging_since_timestamp_seconds` was modified.
