Audit Report

## Title
Unbounded CBOR Deserialization Depth in `SignedTransaction::from_str` Enables Stack Overflow DoS — (`rs/rosetta-api/icp/src/models.rs`)

## Summary
`SignedTransaction::from_str` decodes a hex-encoded CBOR payload and passes it directly to `serde_cbor::from_slice` with no recursion depth limit. The `serde_cbor` crate uses recursive descent parsing; a deeply nested CBOR structure exhausts the thread stack and crashes the Rosetta process. The 4 MB JSON body cap is insufficient to prevent this because hex encoding doubles the byte count, allowing ~2 MB of raw CBOR to be delivered in a single request to the unauthenticated `/construction/hash` endpoint.

## Finding Description
`SignedTransaction::from_str` at [1](#0-0)  hex-decodes the input and calls `serde_cbor::from_slice` twice (once for `SignedTransaction`, once for `LegacySignedTransaction` on failure) with no depth guard. The call site is `construction_hash` at [2](#0-1) , which is wired to the public `/construction/hash` endpoint at [3](#0-2)  with no authentication middleware. The only size guard is the 4 MB JSON body limit at [4](#0-3) , but the `signed_transaction` field is a hex string — hex encoding doubles size — so up to ~2 MB of raw CBOR bytes can be delivered. `serde_cbor` is a recursive-descent parser; each CBOR nesting level consumes one stack frame. A CBOR array-of-arrays structure ~100,000 levels deep fits within ~2 MB (each level costs ~20 bytes in CBOR), far exceeding the default 8 MB Rust thread stack. On first parse failure the code retries with `LegacySignedTransaction`, doubling the stack pressure. The identical pattern exists in `ParsedTransaction::try_from` at [5](#0-4)  (reachable via `/construction/parse`) and `UnsignedTransaction::from_str` at [6](#0-5) . The `serde_cbor` dependency is confirmed in [7](#0-6) .

## Impact Explanation
A single malformed HTTP request crashes the Rosetta process with a stack overflow (SIGSEGV). The Rosetta node becomes completely unavailable until manually restarted. This is a concrete, repeatable application-level DoS against the ICP Rosetta API, a financial integration component explicitly listed in scope. This matches the **High ($2,000–$10,000)** impact class: "Application/platform-level DoS, crash" and "Significant Rosetta... security impact with concrete user or protocol harm."

## Likelihood Explanation
The endpoint is publicly reachable with no credentials. The exploit requires only crafting a hex string containing deeply nested CBOR, achievable with a few lines of Python using the `cbor2` library. The attack is repeatable — the process can be crashed again immediately after restart. No special network position, leaked keys, or victim interaction is required.

## Recommendation
- Replace `serde_cbor` (deprecated) with `ciborium`, which supports configurable recursion limits via `ciborium::de::from_reader_with_recursion_limit`.
- Alternatively, enforce a maximum CBOR nesting depth before calling `from_slice` using a non-recursive streaming CBOR scanner.
- Apply the same fix to `ParsedTransaction::try_from` at lines 180–197 and `UnsignedTransaction::from_str` at lines 263–268 of `rs/rosetta-api/icp/src/models.rs`.

## Proof of Concept
```python
import cbor2, binascii, requests

# Build deeply nested CBOR: [[[[...]]]] 100,000 levels deep
payload = b'\x00'
for _ in range(100_000):
    payload = cbor2.dumps([payload])

hex_tx = binascii.hexlify(payload).decode()

body = {
    "network_identifier": {"blockchain": "Internet Computer", "network": "00000000000000020101"},
    "signed_transaction": hex_tx
}
requests.post("http://<rosetta-host>:8080/construction/hash", json=body)
# Rosetta process crashes with stack overflow (SIGSEGV)
```
