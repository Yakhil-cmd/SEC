### Title
Permanent Fund Freeze via `ft_transfer_call` Without Refund Callback in `ExitToNear` Omni Path - (File: engine-precompiles/src/native.rs)

### Summary

When the `ExitToNear` precompile is invoked with an Omni message (triggering `ft_transfer_call` on the NEP-141 contract), and the `error_refund` compile-time feature is not enabled, no refund callback is attached to the NEAR promise. If the recipient NEAR contract does not implement `ft_on_transfer`, the `ft_transfer_call` fails, the NEP-141 tokens are returned to Aurora by the NEP-141 standard's `ft_resolve_transfer`, but the ERC-20 tokens that were already burned on the EVM side are permanently lost with no recovery path.

### Finding Description

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` handles token withdrawals from Aurora EVM to NEAR. When a user uses the Omni path (by providing a message with a `:` separator, e.g. `receiver_id:{"recipient":"eth:...","fee":"0"}`), the function `exit_erc20_token_to_near` selects `ft_transfer_call` as the transfer method and sets `transfer_near_args` to `None`. [1](#0-0) 

This is the NEAR analog of ERC-721's `safeTransferFrom`: `ft_transfer_call` requires the recipient contract to implement `ft_on_transfer`. If it does not, the call fails.

The `callback_args` built in `ExitToNear::run` is: [2](#0-1) 

When the `error_refund` feature is **not** enabled, `refund` is `None` and `transfer_near` is `None` for the Omni path. This equals `ExitToNearPrecompileCallbackArgs::default()`, so the promise is created **without any callback**: [3](#0-2) 

If `ft_transfer_call` subsequently fails (because the recipient does not implement `ft_on_transfer`), there is no `exit_to_near_precompile_callback` invocation to re-mint the burned ERC-20 tokens: [4](#0-3) 

The `error_refund` feature is a compile-time gate. The standard `EvmErc20.sol` contract does **not** include a `refund_address` in its calldata, meaning it is designed for the non-`error_refund` build: [5](#0-4) 

The existing test explicitly confirms that without `error_refund`, there is no EVM-side refund when the NEAR-side transfer fails: [6](#0-5) 

### Impact Explanation

The ERC-20 tokens are burned at the start of the `withdrawToNear` call (inside `EvmErc20.sol`'s `_burn`). If `ft_transfer_call` subsequently fails, the NEP-141 tokens are returned to Aurora by the NEP-141 standard's `ft_resolve_transfer`, but the ERC-20 tokens are permanently destroyed with no recovery path. This constitutes a **permanent freezing of user funds** (Critical).

### Likelihood Explanation

Any EVM user or contract that uses the Omni withdrawal path targeting a NEAR contract that does not implement `ft_on_transfer` will trigger this. This can occur:
- Accidentally, when a user does not know the recipient contract lacks `ft_on_transfer`
- If the recipient contract is upgraded to remove `ft_on_transfer` after the user initiates the withdrawal
- When integrating with NEAR contracts that are not NEP-141 receivers

The Omni path is a production feature reachable by any unprivileged EVM user or contract. No admin access is required.

### Recommendation

Always attach a refund callback when `ft_transfer_call` is selected as the transfer method, regardless of the `error_refund` feature flag. Specifically, in `ExitToNear::run`, the `callback_args` should never be left as `default()` when `ft_transfer_call` is used. A minimal fix is to ensure a callback with a `refund` field is always constructed for the Omni path, re-minting the burned ERC-20 tokens to the original sender if the NEAR-side call fails.

### Proof of Concept

1. User holds ERC-20 tokens on Aurora (bridged from a NEP-141 token).
2. User calls `withdrawToNear` on `EvmErc20.sol` with an Omni-formatted recipient: `receiver_id:{"recipient":"eth:...","fee":"0"}`.
3. `EvmErc20.sol` burns the ERC-20 tokens and calls the `ExitToNear` precompile.
4. `exit_erc20_token_to_near` matches `Message::Omni(msg)` and selects `ft_transfer_call` with no `transfer_near_args`.
5. `callback_args == default()` → `PromiseArgs::Create(transfer_promise)` is emitted with **no callback**.
6. On NEAR, `ft_transfer_call` is dispatched to `receiver_id`. The contract does not implement `ft_on_transfer` → the call panics.
7. NEP-141's `ft_resolve_transfer` returns the NEP-141 tokens to Aurora's account.
8. No `exit_to_near_precompile_callback` is ever invoked; no ERC-20 re-mint occurs.
9. The user's ERC-20 tokens are permanently burned. The NEP-141 tokens sit in Aurora with no owner claim. **Permanent fund freeze.**
