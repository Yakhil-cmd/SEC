Audit Report

## Title
Missing New-Owner Validation in `do_update_node_operator_config_directly_` Allows Unauthorized Node Operator Record Ownership Transfer - (File: `rs/registry/canister/src/mutations/do_update_node_operator_config_directly.rs`)

## Summary
`do_update_node_operator_config_directly_` verifies that the caller equals the current `node_provider_principal_id` of a `NodeOperatorRecord`, but never validates that the new `node_provider_id` in the payload also equals the caller. Any registered node provider can therefore silently transfer ownership of their node operator record — and the ICP node rewards attributed to it — to any arbitrary principal without that principal's consent. The `node_provider_principal_id` field is the direct key used by the reward calculation pipeline to route monthly ICP minting to node providers.

## Finding Description
**Entry point** (`rs/registry/canister/canister/canister.rs`, L809–823): `update_node_operator_config_directly` is explicitly open to any caller with no pre-consensus rejection.

**Authorization check** (`rs/registry/canister/src/mutations/do_update_node_operator_config_directly.rs`, L58–65): The only guard verifies `caller == current node_provider_principal_id`. After passing this check, the function unconditionally writes the caller-supplied `payload.node_provider_id` into the record at L83:

```rust
node_operator_record.node_provider_principal_id = node_provider_id.to_vec();
```

There is no check that `node_provider_id == caller`. The only other guard (L77–81) rejects `node_provider_id == node_operator_id`, which is unrelated to ownership.

**Reward pipeline impact**: `calculate_rewards_v0` in `rs/registry/node_provider_rewards/src/lib.rs` (L38–55) reads `node_operator.node_provider_principal_id` directly from each `NodeOperatorRecord` to build the `BTreeMap<PrincipalId, u64>` of XDR rewards per provider. This map is returned by the Node Rewards Canister and consumed by governance in `get_node_providers_rewards` / `get_monthly_node_provider_rewards` (`rs/nns/governance/src/governance.rs`, L7664–7694). Governance then iterates over `self.heap_data.node_providers`, looks up each provider's XDR reward from this map, converts to ICP, and mints to the provider's `reward_account` or default principal account (`rs/nns/governance/src/governance.rs`, L8248–8271).

**Exploit flow**:
1. Alice (registered node provider) owns `NodeOperatorRecord` for `NO_1` with `node_provider_principal_id = Alice`.
2. Alice calls `update_node_operator_config_directly({ node_operator_id: NO_1, node_provider_id: Bob })`.
3. Check at L59–65 passes: `Alice == current node_provider_principal_id`. No check on `Bob == Alice`.
4. Registry writes `node_provider_principal_id = Bob` for `NO_1`.
5. At next reward cycle, `calculate_rewards_v0` attributes `NO_1`'s XDR rewards to Bob's principal.
6. Governance looks up Bob in the rewards map; if Bob is a registered node provider, Bob's ICP reward is inflated by Alice's nodes' contribution. Alice's rewards are permanently lost.
7. If Bob is near the `maximum_node_provider_rewards_e8s` cap (`rs/nns/governance/src/governance.rs`, L3951–3958), Bob's legitimate rewards are reduced by the cap enforcement.

**Existing checks are insufficient**: The rate-limit check (L67–70) only throttles frequency; it does not prevent the unauthorized transfer. The `do_migrate_node_operator_directly` function (`rs/registry/canister/src/mutations/do_migrate_node_operator_directly.rs`, L132–141) correctly checks `caller.to_vec() == old_node_operator_record.node_provider_principal_id` but does not serve as a guard here.

## Impact Explanation
This is a **Medium** severity finding. A registered node provider can unilaterally redirect their own node operator record's reward attribution to any arbitrary principal without that principal's consent. Concrete impacts:

- **Financial redirection**: Alice's monthly ICP rewards (minted tokens) are permanently routed to Bob's account. Alice loses her rewards; Bob receives unexpected ICP.
- **Reward cap harm**: If Bob is near the `maximum_node_provider_rewards_e8s` cap, the addition of Alice's nodes pushes Bob's computed reward above the cap, causing Bob's legitimate rewards to be silently truncated.
- **Unauthorized obligation transfer**: Alice can exit her node operator obligations by transferring the record to an unsuspecting principal, leaving that principal associated with nodes they do not control and cannot manage.
- **Irreversibility**: Alice loses the ability to call `update_node_operator_config_directly` for `NO_1` after the transfer (since the check now requires `Bob == node_provider_principal_id`). Bob must discover and undo the change themselves.

This matches the allowed impact class: *Unauthorized access to governance assets, wallets, or ledger-adjacent funds with meaningful per-target constraints* (High/Medium tier), constrained here to Medium because the attacker can only redirect their own rewards, not steal from others, and the cap-harm scenario requires Bob to be near the cap.

## Likelihood Explanation
- The entry point is open to any caller; no privileged access, governance majority, or key compromise is required.
- Any existing registered node provider (a real-world role with known principals on mainnet) can trigger this with a single update call.
- The attack is silent: the victim receives no notification and the registry change is not surfaced to them directly.
- The attack is repeatable and cheap (subject only to the rate limit, which throttles frequency but does not prevent the transfer).

## Recommendation
Add a check after the existing caller-authorization check that the new `node_provider_id` equals the caller, mirroring the ownership-transfer pattern used in `do_migrate_node_operator_directly`:

```rust
// After the existing caller == current_node_provider check (L58-65):
if node_provider_id != caller {
    return Err(format!(
        "New node_provider_id {node_provider_id} must equal the caller {caller}. \
         Use a governance proposal to transfer ownership to a different principal."
    ));
}
```

This ensures a node provider can only set themselves as the new provider (a no-op) or that any ownership transfer to a different principal requires a governance proposal via `update_node_operator_config` (which is restricted to the governance canister).

## Proof of Concept
**Minimal unit test** (extend `rs/registry/canister/src/mutations/do_update_node_operator_config_directly.rs` test module):

```rust
#[test]
fn test_update_node_operator_config_directly_cannot_transfer_to_third_party() {
    let mut registry = invariant_compliant_registry(0);
    let now = now_system_time();

    let node_operator_id = PrincipalId::new_user_test_id(1_000);
    let node_provider_id = PrincipalId::new_user_test_id(10_000);
    let victim_id = PrincipalId::new_user_test_id(99_999); // arbitrary third party

    registry.do_add_node_operator(AddNodeOperatorPayload {
        node_operator_principal_id: Some(node_operator_id),
        node_provider_principal_id: Some(node_provider_id),
        node_allowance: 1,
        dc_id: "DC1".to_string(),
        rewardable_nodes: btreemap! { "type1.1".to_string() => 1 },
        ipv6: Some("bar".to_string()),
        max_rewardable_nodes: Some(btreemap! { "type1.2".to_string() => 1 }),
    });

    // Alice (node_provider_id) attempts to transfer ownership to victim_id
    let request = UpdateNodeOperatorConfigDirectlyPayload {
        node_operator_id: Some(node_operator_id),
        node_provider_id: Some(victim_id), // ← should be rejected
    };

    // Currently this SUCCEEDS (demonstrating the bug).
    // After the fix, this should return Err.
    let result = registry.do_update_node_operator_config_directly_(
        request, node_provider_id, now
    );
    assert!(result.is_err(), "Should not allow transfer to third party");
}
```

```rust
pub fn get_node_provider_reward(
    np: &NodeProvider,
    xdr_permyriad_reward: u64,
    xdr_permyriad_per_icp: u64,
) -> Option<RewardNodeProvider> {
    if let Some(np_id) = np.id.as_ref() {
        let amount_e8s = ((xdr_permyriad_reward as u128 * TOKEN_SUBDIVIDABLE_BY as u128)
            / xdr_permyriad_per_icp as u128) as u64;

        let to_account = Some(if let Some(account) = &np.reward_account {
            account.clone()
        } else {
            AccountIdentifier::from(*np_id).into()
        });

        Some(RewardNodeProvider {
            node_provider: Some(np.clone()),
            amount_e8s,
            reward_mode: Some(RewardMode::RewardToAccount(RewardToAccount { to_account })),
        })
    } else {
        None
    }
}
```
