### Title
Engine `pause_contract` Does Not Halt Already-Scheduled XCC Router Promises — (`etc/xcc-router/src/lib.rs`)

---

### Summary

The Aurora Engine's `pause_contract` mechanism sets `is_paused = true` in the engine's own state, blocking EVM execution. However, the XCC Router sub-accounts (child contracts created and owned by the engine) have no corresponding pause mechanism, and their `execute_scheduled` function is intentionally callable by **anyone**. This means that when the engine is paused in an emergency, already-scheduled NEAR cross-contract promises in Router sub-accounts can still be executed by any external actor, bypassing the engine's emergency stop.

---

### Finding Description

The Aurora Engine creates XCC Router sub-accounts via `fund_xcc_sub_account` and `handle_precompile_promise`. During creation, the engine sets itself as the Router's `parent`: [1](#0-0) 

The engine's `pause_contract` function only modifies the engine's own state: [2](#0-1) 

The engine has **no function** to pause individual Router sub-accounts. Reviewing `admin.rs` in full confirms there is no `pause_router`, `stop_xcc`, or equivalent call issued to Router sub-accounts.

Meanwhile, the Router's `execute_scheduled` function is explicitly open to any caller with no parent check and no pause guard: [3](#0-2) 

The comment itself acknowledges this is intentional ("It is intentional that this function can be called by anyone"). However, this design assumption breaks down when the engine is paused for a security emergency: the Router has no awareness of the engine's `is_paused` flag, and no mechanism exists for the engine to propagate a pause to its Router sub-accounts.

The `schedule` function (which creates the stored promises) does enforce the parent-only check: [4](#0-3) 

But `execute_scheduled` does not, meaning any actor can drain the scheduled promise queue at any time, including during an engine pause.

---

### Impact Explanation

**High — Temporary freezing bypass / incomplete emergency stop.**

When the engine is paused (e.g., due to a discovered exploit in the XCC precompile or EVM execution layer), the operator's intent is to halt all protocol activity. However, any NEAR cross-contract calls that were already scheduled in Router sub-accounts before the pause can still be executed by any external account calling `execute_scheduled`. These scheduled promises can include:

- NEAR token transfers to arbitrary accounts (via wNEAR unwrap flows)
- Calls to external NEAR contracts with attacker-controlled arguments

This means the engine's pause does not fully protect users or the protocol during an incident. Funds in transit via the XCC system remain at risk even after the emergency pause is activated.

---

### Likelihood Explanation

**Medium.** This requires: (1) the engine to be paused due to a security incident, (2) one or more scheduled promises to exist in Router sub-accounts at the time of the pause, and (3) an attacker (or anyone) to call `execute_scheduled` on those routers. Condition (2) is realistic in normal operation since `schedule` is a standard XCC flow. Condition (3) requires no special privilege.

---

### Recommendation

Add a pause propagation mechanism. When `pause_contract` is called on the engine, it should either:

1. Issue a NEAR cross-contract call to each known Router sub-account to disable `execute_scheduled` (requires adding a pause flag to the Router), or
2. Have `execute_scheduled` in the Router perform a cross-contract view call to the parent engine to check `is_paused` before executing, or
3. Restrict `execute_scheduled` to be callable only by the parent (engine), removing the open-caller design, so that a paused engine cannot trigger it either.

Option 3 is the simplest but changes the existing open-caller design. Option 1 or 2 preserves the design while closing the gap.

---

### Proof of Concept

1. User's EVM contract calls the XCC precompile, which causes the engine to call `Router::schedule`, storing a promise (e.g., a NEAR token transfer) at nonce `N` in the Router sub-account.
2. Admin detects a security incident and calls `pause_contract` on the Aurora Engine. `is_paused` is set to `true`. New EVM transactions are blocked.
3. Any external NEAR account calls `execute_scheduled(N)` directly on the Router sub-account (`{address}.aurora`).
4. The Router has no pause check; it removes the promise from storage and executes it via `env::promise_return`.
5. The NEAR cross-contract call executes — transferring NEAR or calling an external contract — despite the engine being in a paused state. [5](#0-4) [6](#0-5)

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);

#[named]
pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
