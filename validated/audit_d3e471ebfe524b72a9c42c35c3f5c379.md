### Title
Block L2→L1 Log Capacity Exhaustion via Cheap L2 `sendToL1` Spam - (File: `system_hooks/src/call_hooks/l1_messenger.rs`)

### Summary
An unprivileged L2 transaction sender can exhaust the per-block L2→L1 log capacity (`MAX_NUMBER_OF_LOGS = 16,384`) by repeatedly calling the L1 Messenger system contract (`0x8008`) with `sendToL1(bytes)`. Each call appends one log entry to the block-level `LogsStorage`. Once the limit is reached, every subsequent transaction in the same block is invalidated with `BlockL2ToL1LogsLimitReached`, preventing legitimate users from having their transactions included in that block.

### Finding Description

ZKsync OS enforces a hard per-block limit on L2→L1 logs:

```
pub const MAX_NUMBER_OF_LOGS: u64 = 16_384;
``` [1](#0-0) 

The block-level check is performed in `check_for_block_limits` after every transaction:

```rust
} else if !cfg!(feature = "resources_for_tester") && logs_used > MAX_NUMBER_OF_LOGS {
    Err(InvalidTransaction::BlockL2ToL1LogsLimitReached)
``` [2](#0-1) 

The `logs_used` value is read directly from the live log storage (`system.io.logs_len()`), which counts both user messages (from `sendToL1`) and L1→L2 tx result logs: [3](#0-2) 

Each call to `L1Messenger.sendToL1(bytes)` routes through `l1_messenger_hook` → `send_to_l1_inner` → `system.io.emit_l1_message(...)`, which calls `logs_storage.push_message(...)`, permanently incrementing the block-level log count: [4](#0-3) [5](#0-4) 

The gas cost for a single `sendToL1` call is approximately **9,202 gas** (as measured in the test suite): [6](#0-5) 

With a typical block gas limit (e.g., 80M gas), an attacker can fit approximately `80,000,000 / 9,202 ≈ 8,694` `sendToL1` calls per block. While this does not reach 16,384 in a single block with a standard gas limit, the attacker can combine this with the fact that each L1→L2 priority transaction also unconditionally emits one log entry via `emit_l1_l2_tx_log`: [7](#0-6) 

More critically, a single L2 transaction can call `sendToL1` **multiple times** within one transaction (e.g., via a loop in a smart contract), consuming many log slots at the cost of a single transaction's gas. The gas cost per log entry is dominated by the L1Messenger contract's EVM bytecode (keccak + LOG opcodes), not by a per-log block-level fee. There is no per-transaction cap on the number of `sendToL1` calls, and no per-block fee that scales with log count to deter spam.

Once `logs_used > 16,384`, the block-level check fires and every subsequent transaction is reverted to pre-tx state and reported as `BlockL2ToL1LogsLimitReached`: [8](#0-7) 

### Impact Explanation

An attacker who deploys a contract that loops `sendToL1` calls can exhaust the 16,384-log block capacity within a single transaction or a small number of transactions. Once exhausted, no further transactions can be included in that block. This is a **Denial of Service** against block throughput: legitimate users' transactions are excluded from the block, and the attacker only pays gas proportional to the number of log entries generated — not a block-level penalty. The impact is loss of liveness for the block, analogous to the original report's epoch-level mint limit exhaustion.

### Likelihood Explanation

The attack is cheap and permissionless. Any L2 account with sufficient ETH to pay gas can deploy a looping contract and call it. The L1 Messenger hook is callable by any user via the `L1Messenger` system contract at `0x8008`. No privileged role is required. The attacker recovers no funds but can grief the chain at low cost per block.

### Recommendation

1. **Introduce a per-transaction log cap**: Enforce a maximum number of `sendToL1` calls per transaction (e.g., 512 or 1,024), preventing a single transaction from consuming a disproportionate share of the block log budget.
2. **Charge pubdata-proportional gas for each log**: The pubdata cost per log entry (`L2_TO_L1_LOG_SERIALIZE_SIZE = 88` bytes) should be reflected in the gas charged at the time of the `sendToL1` call, making bulk log spam economically prohibitive.
3. **Reserve log slots for L1→L2 tx result logs**: Since each priority transaction unconditionally consumes one log slot, the effective user-accessible log budget should be reduced by the expected number of L1→L2 transactions in the block.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IL1Messenger {
    function sendToL1(bytes calldata _message) external returns (bytes32);
}

contract LogSpammer {
    IL1Messenger constant L1_MESSENGER = IL1Messenger(0x0000000000000000000000000000000000008008);

    // Call this once with count = 16384 to exhaust the block log limit
    function spam(uint256 count) external {
        for (uint256 i = 0; i < count; i++) {
            L1_MESSENGER.sendToL1(abi.encodePacked(i));
        }
    }
}
```

**Steps:**
1. Deploy `LogSpammer` on ZKsync OS L2.
2. Call `spam(16384)` in a single transaction with sufficient gas.
3. After this transaction is included, `system.io.logs_len()` exceeds `MAX_NUMBER_OF_LOGS`.
4. All subsequent transactions in the same block are rejected with `BlockL2ToL1LogsLimitReached`.
5. Legitimate users cannot get their transactions included until the next block.

The cost per log entry is ~9,202 gas (from `test_l1_messenger_gas_charging`), so exhausting 16,384 slots costs approximately `16,384 × 9,202 ≈ 150M gas` — achievable within a few transactions if the block gas limit is high enough, or across multiple transactions in the same block. fn check_for_block_limits<S: EthereumLikeTypes>(
    system: &mut System<S>,
    gas_used: u64,
    computational_native_used: u64,
    pubdata_used: u64,
    logs_used: u64,
    blob_gas_used: u64,
) -> Result<(), InvalidTransaction>
where
    S::IO: IOSubsystemExt,
    <S as SystemTypes>::Metadata: ZkSpecificPricingMetadata,
{
    if gas_used > system.get_gas_limit() {
        system_log!(
            system,
            "Block gas limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockGasLimitReached)
    } else if blob_gas_used > system.get_blob_gas_limit() {
        system_log!(
            system,
            "Block blob gas limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockBlobGasLimitReached)
    } else if !cfg!(feature = "resources_for_tester")
        && computational_native_used > MAX_NATIVE_COMPUTATIONAL
    {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block native limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockNativeLimitReached)
    } else if !cfg!(feature = "resources_for_tester") && pubdata_used > system.get_pubdata_limit() {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block pubdata limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockPubdataLimitReached)
    } else if !cfg!(feature = "resources_for_tester") && logs_used > MAX_NUMBER_OF_LOGS {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block logs limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockL2ToL1LogsLimitReached)
    } else {
        Ok(())
    }
}

```rust
pub(crate) fn send_to_l1_inner<S: EthereumLikeTypes>(
    calldata: &[u8],
    resources: &mut S::Resources,
    system: &mut System<S>,
) -> Result<Result<(), &'static str>, SystemError> {
    if calldata.len() < 20 {
        return Ok(Err(
            "L1 messenger failure: sendToL1 called with invalid calldata",
        ));
    }

    let address_sender = B160::try_from_be_slice(&calldata[0..20]).ok_or(
        SystemError::LeafDefect(internal_error!("Failed to create B160 from 20 byte array")),
    )?;

    let message = &calldata[20..];

    // emit L1 message (ignore returned hash)
    // TODO(EVM-1190): hash calculation is suboptimal, to be refactored in future
    system.io.emit_l1_message(
        // Gas should be charged by the L1Messenger system contract
        ExecutionEnvironmentType::NoEE,
        resources,
        &address_sender,
        message,
    )?;

    Ok(Ok(()))
}
