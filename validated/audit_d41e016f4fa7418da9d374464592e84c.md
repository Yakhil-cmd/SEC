### Title
Malicious NEP-141 Contract Can Call `ft_on_transfer` Directly to Mint Unbacked ERC-20 Tokens and Drain Deposited NEP-141 Funds - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

Aurora Engine's `ft_on_transfer` entry point mints ERC-20 tokens based solely on the `amount` field supplied by the calling NEP-141 contract, with no verification that the NEP-141 tokens were actually transferred to Aurora's account. A malicious NEP-141 contract operator can call `ft_on_transfer` directly — bypassing the `ft_transfer_call` flow — to mint arbitrary ERC-20 tokens and subsequently drain real NEP-141 deposits made by honest users.

---

### Finding Description

**Root cause — `ft_on_transfer` trusts caller-supplied `amount` without balance verification**

`ft_on_transfer` is a public NEAR contract method. Any NEAR account can call it. The function uses `predecessor_account_id` as the NEP-141 token identity and `args.amount` as the mint quantity: [1](#0-0) 

When `predecessor_account_id` is not the ETH connector, the engine calls `receive_erc20_tokens`: [2](#0-1) 

Inside `receive_erc20_tokens`, `args.amount` is taken at face value and used directly to mint ERC-20 tokens via an internal `call` to the ERC-20 admin address. There is **no check** that Aurora's NEP-141 balance actually increased by `args.amount` before minting.

**No access control on `deploy_erc20_token`**

Any unprivileged NEAR account can register an arbitrary NEP-141 token and obtain a corresponding ERC-20 on Aurora: [3](#0-2) 

There is no `require_owner_only` or similar guard. This means an attacker can register their own malicious NEP-141 contract as a legitimate bridged token.

**Attack path (step-by-step)**

1. Attacker deploys a malicious NEP-141 contract at `attacker-token.near` on NEAR.
2. Attacker calls `deploy_erc20_token` on Aurora with `attacker-token.near` — Aurora deploys an ERC-20 and registers the mapping.
3. Attacker markets the token; honest users call `ft_transfer_call` on `attacker-token.near` with `receiver_id = aurora`, legitimately depositing, say, 1 000 000 tokens. Aurora's NEP-141 balance grows to 1 000 000.
4. Attacker's contract calls `ft_on_transfer` **directly** on Aurora (not via `ft_transfer_call`) with `amount = 1_000_000` and `msg = <attacker_evm_address>`. No tokens are transferred; Aurora's NEP-141 balance is unchanged.
5. Aurora mints 1 000 000 ERC-20 tokens to the attacker's EVM address.
6. Attacker calls `withdrawToNear` on the ERC-20, burning the tokens and triggering `ft_transfer` on `attacker-token.near` for 1 000 000 tokens — draining the honest users' deposits.

---

### Impact Explanation

**Critical — direct theft of user funds.**

Honest users who deposited NEP-141 tokens into Aurora via the legitimate `ft_transfer_call` bridge flow lose their funds. The attacker mints ERC-20 tokens backed by nothing and exits them against the real pool. The ERC-20 supply becomes unbacked, causing insolvency for all other holders of that bridged token.

---

### Likelihood Explanation

**Medium.**

The attacker must:
- Deploy and control a NEAR contract (trivial).
- Register it via `deploy_erc20_token` (no access control, trivial).
- Attract honest users to deposit (requires social engineering or a DeFi integration, non-trivial but realistic).

Once users have deposited, the exploit itself is a single direct NEAR function call. The attacker can also front-run any large deposit.

---

### Recommendation

1. **Balance-check before and after minting.** In `receive_erc20_tokens`, query Aurora's NEP-141 balance of `token` before and after the `ft_on_transfer` invocation and mint only the observed delta, not `args.amount`.
2. **Restrict `deploy_erc20_token` to privileged callers** (owner or a governance-controlled whitelist) so that only vetted NEP-141 tokens can be bridged.
3. **Validate that `ft_on_transfer` is only reachable as a callback** from a legitimate `ft_transfer_call` by checking that the call originates from a promise context (NEAR's `env::predecessor_account_id` inside a callback vs. a direct call can be distinguished by checking `env::promise_results_count() > 0` or similar guards).

```
// Attacker's malicious NEP-141 contract (pseudocode)
fn exploit(aurora: AccountId, attacker_evm: String, amount: u128) {
    // Step 4: call ft_on_transfer directly — no ft_transfer_call, no actual transfer
    aurora.ft_on_transfer(
        sender_id: self.account_id(),
        amount: amount,          // inflated; no tokens moved
        msg: attacker_evm,       // attacker's EVM address
    );
    // Aurora mints `amount` ERC-20 tokens to attacker_evm
}
```

The `predecessor_account_id` seen by Aurora is `attacker-token.near` (registered via `deploy_erc20_token`), so `get_erc20_from_nep141` succeeds and `receive_erc20_tokens` mints the full `amount` without any balance verification. [4](#0-3) [5](#0-4)

```rust
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
```

```rust
    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;
```
