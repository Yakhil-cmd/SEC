Audit Report

## Title
`spawn_neuron` Unconditionally Copies Parent Hot Keys to Child Neuron Assigned to a Different Controller — (`rs/nns/governance/src/governance.rs`)

## Summary
When `spawn_neuron` is called with a `new_controller` that differs from the parent neuron's controller, the child neuron is built with the parent's full `hot_keys` list cloned onto it unconditionally. The new controller (Charlie) receives a neuron pre-authorized for principals (Bob) they did not choose, giving those principals persistent governance access — including voting, following, and Neuron Fund membership changes — without Charlie's knowledge or consent.

## Finding Description
In `rs/nns/governance/src/governance.rs`, `spawn_neuron` resolves `child_controller` from the optional `spawn.new_controller` field (lines 2651–2655). When that field is set, `child_controller` is a completely different principal from the parent's controller. The `NeuronBuilder` call at lines 2704–2718 then unconditionally calls `.with_hot_keys(parent_neuron.hot_keys.clone())` (line 2714), regardless of whether `child_controller` matches the parent's controller. There is no conditional check, no filtering, and no clearing of the inherited hot key list when the controllers differ.

The authorization model in `rs/nns/governance/src/neuron/types.rs` confirms that any principal in `hot_keys` passes `is_hotkey_or_controller` (line 255), which is the gate for `is_authorized_to_vote` (line 243–244) and `is_authorized_to_configure_or_err` for `JoinCommunityFund`/`LeaveCommunityFund` (lines 780–781). Consequently, Bob — a hot key on Alice's parent neuron — retains the ability to cast votes, set follow relationships, and toggle Neuron Fund membership on Charlie's newly spawned neuron, all without Charlie's knowledge.

Exploit path:
1. Alice controls neuron N with hot key Bob.
2. Alice calls `manage_neuron → Spawn { new_controller: Some(Charlie), nonce: Some(42), percentage_to_spawn: Some(100) }`.
3. `spawn_neuron` sets `child_controller = Charlie` and builds the child neuron with `controller = Charlie` but `hot_keys = [Bob]`.
4. Bob sends `manage_neuron → RegisterVote` authenticated as Bob, targeting the child neuron ID.
5. Governance evaluates `is_authorized_to_vote` → `is_hotkey_or_controller` → Bob is in `hot_keys` → authorized. Vote is cast on Charlie's behalf.

No existing guard prevents this. The only mitigation available to Charlie is to discover the inherited hot keys and remove them manually — a step Charlie has no reason to know is necessary.

## Impact Explanation
This constitutes unauthorized access to a neuron's governance capabilities. A third party (Bob) retains the ability to vote, follow, and alter Neuron Fund membership on a neuron controlled by Charlie without Charlie's consent. While hot keys cannot disburse, split, or spawn (so there is no direct financial theft), the governance impact is concrete: Bob can override or dilute Charlie's intended voting behavior on NNS proposals and manipulate Charlie's maturity exposure via Neuron Fund membership. This matches the allowed impact: **High — Unauthorized access to neurons/governance assets where exploitation requires meaningful per-target work** (Alice must specifically target Charlie by spawning for them, constituting per-target effort).

## Likelihood Explanation
Both preconditions — Alice having a hot key (Bob) on her neuron, and Alice spawning for a different principal (Charlie) — are normal, documented operations. The `Spawn` command explicitly supports `new_controller` for gifting maturity. A recipient (Charlie) has no UI-level warning about inherited hot keys and no reason to inspect them. Bob requires no special access beyond having been a hot key on the parent neuron at any point before the spawn. The attack is repeatable and requires no technical sophistication beyond submitting standard `manage_neuron` calls.

## Recommendation
In `spawn_neuron`, conditionally clear the hot key list when `child_controller` differs from the parent's controller:

```rust
let hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};
```

Replace `.with_hot_keys(parent_neuron.hot_keys.clone())` at line 2714 with `.with_hot_keys(hot_keys)`. This ensures a neuron spawned for a third party starts with no pre-authorized hot keys, leaving the new controller in full control from the moment of creation.

## Proof of Concept
Minimal deterministic integration test (PocketIC or local replica):
1. Create neuron N controlled by Alice; add Bob as a hot key via `AddHotKey`.
2. Call `manage_neuron → Spawn { new_controller: Some(Charlie), nonce: Some(1), percentage_to_spawn: Some(100) }` as Alice.
3. Record the returned child neuron ID.
4. Submit a governance proposal to create a test target.
5. As Bob, call `manage_neuron → RegisterVote { neuron_id: child_id, proposal_id: P, vote: Yes }`.
6. Assert the vote is accepted (no `NotAuthorized` error) and recorded against the child neuron.
