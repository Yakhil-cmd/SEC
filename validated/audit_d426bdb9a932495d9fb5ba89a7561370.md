### Title
Ethereum Transaction Flow Hardcodes `native_limit = u64::MAX`, Bypassing Native Resource Accounting - (File: `basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs`)

---

### Summary

The Ethereum transaction validation path in ZKsync OS accepts `gas_limit` and `gas_price` parameters but ignores them when setting the native resource limit, hardcoding `u64::MAX` instead of deriving the limit from the transaction's fee. This is a direct analog of the reported timelock bug: a parameter is accepted and validated, but the critical derived value is hardcoded to an unconstrained sentinel rather than computed from the input.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs`, the private `create_resources_for_tx` function is responsible for constructing the resource budget for every standard Ethereum transaction (EIP-155, EIP-2930, EIP-1559, EIP-4844, EIP-7702):

```rust
fn create_resources_for_tx<S: EthereumLikeTypes>(
    gas_limit: u64,
    is_deployment: bool,
    calldata_len: u64,
    calldata_tokens: u64,
) -> Result<ResourcesForEthereumTx<S>, TxError> {
    // ... intrinsic overhead calculation ...
    let gas_limit_for_tx = gas_limit - intrinsic_overhead;
    let ergs = gas_limit_for_tx.saturating_mul(ERGS_PER_GAS);
    let native_limit =
        <<S as zk_ee::system::SystemTypes>::Resources as Resources>::Native::from_computational(
            u64::MAX,   // <-- hardcoded, gas_limit/gas_price are ignored
        );
    let main_resources = S::Resources::from_ergs_and_native(Ergs(ergs), native_limit);
    Ok(ResourcesForEthereumTx { main_resources })
}
``` [1](#0-0) 

The `gas_limit` parameter is used only to compute the ergs (EVM gas) budget. The native resource limit — which models proving complexity — is unconditionally set to `u64::MAX`, regardless of what the sender paid.

This is confirmed downstream: when `before_refund` is called for Ethereum transactions, `native_per_gas = 0` is passed to `compute_gas_refund`, meaning native consumption never adjusts `gas_used` and the delta-gas mechanism is fully disabled:

```rust
let refund_info = compute_gas_refund(
    system,
    S::Resources::empty(),
    transaction.gas_limit(),
    min_gas_used,
    0u64,   // native_per_gas hardcoded to 0
    &mut context.resources.main_resources,
)?;
``` [2](#0-1) 

**Contrast with the ZK transaction flow**, which correctly derives the native limit from the transaction's fee:

```rust
let native_limit = if cfg!(feature = "unlimited_native") || free_native {
    u64::MAX - 1
} else {
    native_prepaid_from_gas   // = native_per_gas * gas_limit
};
``` [3](#0-2) 

The protocol documentation explicitly states the intended formula:

> `nativePerGas := gasPrice / nativePrice`  
> `nativeLimit := gasLimit * nativePerGas` [4](#0-3) 

The Ethereum flow accepts and validates `gas_limit` and `gas_price` (via `get_gas_prices`) but never feeds them into the native resource budget, leaving it at `u64::MAX`.

---

### Impact Explanation

The native resource models RISC-V proving complexity. If a transaction runs out of native resources, the entire transaction is reverted; if it never runs out (limit = `u64::MAX`), the prover must prove arbitrarily expensive computation regardless of what the sender paid.

An attacker submitting standard Ethereum transactions can:

1. **Consume unbounded native resources** (storage writes, precompile calls, complex EVM computation) without paying the native-resource component of the fee.
2. **Force the prover to process computation far beyond what EVM gas covers**, since the delta-gas adjustment (`native_used / native_per_gas - gas_used`) is disabled (`native_per_gas = 0`).
3. **Cause block-level proving DoS**: if enough Ethereum transactions each consume near-maximum native resources, the aggregate proving cost per block becomes unbounded, potentially making blocks unprovable or economically infeasible to prove.

The ZK transaction flow is protected by this limit; the Ethereum flow — which handles all standard user transactions — is not.

**Impact: 3/5** — Resource accounting bypass enabling proving-cost DoS; does not directly steal funds but can halt the chain's ability to produce proofs.

---

### Likelihood Explanation

Any unprivileged Ethereum transaction sender can trigger this. No special role, leaked key, or governance majority is required. The attacker simply submits a standard EIP-1559 transaction that performs many storage writes or precompile calls. The entry path is the normal transaction submission flow.

**Likelihood: 3/5** — Straightforward to trigger; requires only a funded EOA and knowledge of which operations are native-resource-intensive.

---

### Recommendation

The Ethereum `create_resources_for_tx` function must accept `gas_price` (or `native_per_gas`) as a parameter and compute the native limit the same way the ZK flow does:

```rust
let native_limit = if native_per_gas == 0 {
    u64::MAX - 1
} else {
    gas_limit_for_tx.saturating_mul(native_per_gas)
};
```

Additionally, `before_refund` for Ethereum transactions should pass the actual `native_per_gas` to `compute_gas_refund` so that the delta-gas adjustment is applied, ensuring native-resource-heavy transactions are charged proportionally.

---

### Proof of Concept

1. Compute `native_per_gas = effective_gas_price / native_price` for a typical block (e.g., `1000 / 10 = 100`).
2. Submit an EIP-1559 transaction with `gas_limit = 30_000_000` (block gas limit) that performs 10,000 `SSTORE` operations and calls `sha256` repeatedly.
3. Observe that the transaction is accepted and executed. The native resources consumed (each `SSTORE` and `sha256` round charges native) can far exceed `gas_limit * native_per_gas = 30_000_000 * 100 = 3_000_000_000`.
4. The prover must prove all native work. The sender paid only EVM gas; the native resource budget was `u64::MAX` and was never exhausted.
5. Repeat across multiple transactions in the same block to push total native consumption to levels that make the block unprovable within practical time/cost bounds.

    let mut intrinsic_overhead = TX_INTRINSIC_GAS;
    if is_deployment {
        if calldata_len > MAX_INITCODE_SIZE as u64 {
            return Err(TxError::Validation(CreateInitCodeSizeLimit));
        }
        intrinsic_overhead = intrinsic_overhead.saturating_add(DEPLOYMENT_TX_EXTRA_INTRINSIC_GAS);
        let initcode_gas_cost =
            evm_interpreter::gas_constants::INITCODE_WORD_COST * calldata_len.div_ceil(32);
        intrinsic_overhead = intrinsic_overhead.saturating_add(initcode_gas_cost);
    }
    intrinsic_overhead =
        intrinsic_overhead.saturating_add(calldata_tokens.saturating_mul(CALLDATA_TOKEN_GAS_COST));

    if intrinsic_overhead > gas_limit {
        Err(TxError::Validation(
            InvalidTransaction::OutOfGasDuringValidation,
        ))
    } else {
        let gas_limit_for_tx = gas_limit - intrinsic_overhead;
        let ergs = gas_limit_for_tx.saturating_mul(ERGS_PER_GAS); // we checked at the very start that gas_limit * ERGS_PER_GAS doesn't overflow
        let native_limit =
            <<S as zk_ee::system::SystemTypes>::Resources as Resources>::Native::from_computational(
                u64::MAX,
            );
        let main_resources = S::Resources::from_ergs_and_native(Ergs(ergs), native_limit);

        Ok(ResourcesForEthereumTx { main_resources })
    }
