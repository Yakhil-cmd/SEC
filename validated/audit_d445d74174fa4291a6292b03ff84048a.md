### Title
Stale Submission-Time Valuation Snapshot Used at Execution Allows Treasury Drain Beyond Intended 7-Day Limit — (`File: rs/sns/governance/src/proposal.rs`, `rs/sns/governance/src/governance.rs`)

### Summary

The SNS governance canister snapshots the treasury valuation (token balance × ICP/token price × XDR/ICP rate) at `TransferSnsTreasuryFunds` proposal **submission** time and stores it in `ProposalData.action_auxiliary`. At **execution** time, the same stale snapshot is reused to compute the 7-day transfer allowance. Because the snapshot is never refreshed, an attacker who controls enough voting power can manipulate the effective allowance by timing the proposal submission to coincide with a temporarily inflated treasury valuation, then executing the transfer after the valuation has returned to normal — extracting more tokens than the current treasury state would permit.

### Finding Description

When a `TransferSnsTreasuryFunds` proposal is submitted, `validate_and_render_transfer_sns_treasury_funds` calls `treasury_valuation_if_proposal_amount_is_small_enough_or_err`, which calls `assess_treasury_balance` to fetch the live treasury balance, the CMC 30-day average ICP/XDR rate, and (for SNS tokens) the swap-derived price. The resulting `Valuation` is stored in `ProposalData.action_auxiliary` as `TransferSnsTreasuryFundsActionAuxiliary { valuation }`.

At execution time, `perform_transfer_sns_treasury_funds` retrieves this stored valuation via `get_action_auxiliary` and passes it directly to `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`. That function recomputes only the **spent** tokens (from recently executed proposals) but reuses the **stale** submission-time valuation to derive `allowance_tokens`. The code even documents this explicitly:

> "The upper bound is based on treasury valuation factors at the time of proposal submission."

The `allowance_tokens` bound is therefore fixed at submission time and is never re-evaluated against the current treasury state.

**Concrete manipulation path (ICP treasury variant):**

1. The SNS treasury holds 10,000 ICP. The CMC 30-day average rate is 10 XDR/ICP → treasury = 100,000 XDR (medium regime, 25% cap = 2,500 ICP allowed per 7 days).
2. An attacker with majority voting power waits for a period when the CMC 30-day average rate is temporarily elevated (e.g., 50 XDR/ICP → treasury = 500,000 XDR, large regime, 300,000 XDR cap ≈ 6,000 ICP allowed).
3. Attacker submits a `TransferSnsTreasuryFunds` proposal for 5,000 ICP. Validation passes because the snapshot valuation places the treasury in the large regime.
4. The snapshot `Valuation { xdrs_per_icp: 50, tokens: 10000, ... }` is persisted in `action_auxiliary`.
5. The CMC rate returns to 10 XDR/ICP. Under the current rate, 5,000 ICP would exceed the 25% cap (2,500 ICP).
6. Attacker votes the proposal through and it executes. `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` uses the stale snapshot (50 XDR/ICP), computes `allowance_tokens = 6,000 ICP`, and allows the 5,000 ICP transfer — double the intended limit.

The same logic applies to the SNS token treasury path, where `icps_per_token` is derived from the swap's `sns_tokens_per_icp` field and the current total supply. A temporary deflation of total supply (e.g., a large burn) at submission time inflates `icps_per_token`, inflating the snapshot valuation and thus the allowance.

### Impact Explanation

An SNS governance majority (or a single actor who controls majority voting power, which is common in early-stage SNS deployments) can drain the SNS ICP or SNS-token treasury at a rate significantly exceeding the intended 7-day cap. The cap exists specifically to limit the damage from a compromised or malicious governance majority; bypassing it removes that safety net. The attacker receives real ICP or SNS tokens in excess of what the protocol intends to allow.

**Vulnerability class:** Governance authorization / ledger conservation bug — the rate-limiting invariant on treasury outflows is violated because the price oracle input to the limit calculation is frozen at a manipulable past moment.

### Likelihood Explanation

- The CMC 30-day average ICP/XDR rate fluctuates naturally; no oracle manipulation is required — the attacker only needs to time proposal submission to a high-rate window.
- Any SNS where a single entity or coordinated group holds majority voting power (common at launch, where developer neurons dominate) can execute this unilaterally.
- The voting period is days-long, giving ample time to wait for a favorable rate window before submitting.
- No privileged system access, key compromise, or subnet-majority attack is required.

### Recommendation

Re-fetch the live treasury valuation at execution time instead of reusing the submission-time snapshot. The submission-time snapshot should be used only for the submission-time gate check. At execution, `perform_transfer_sns_treasury_funds` should call `assess_treasury_balance` again (as an async call before the ledger transfer) and pass the fresh valuation to `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`. If a live fetch is undesirable for latency reasons, the snapshot should at minimum be re-validated against a freshly fetched rate and rejected if the current rate would place the treasury in a more restrictive regime.
