### Title
Unchecked Precompile Return Value in `withdrawToNear` Allows Token Burn Without NEAR Credit — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.withdrawToNear` burns the caller's tokens before invoking the exit precompile, and never checks the assembly `call` return value. If the precompile rejects the input for any reason — including an oversized `recipient` that exceeds the precompile's hard `MAX_INPUT_SIZE = 1024` — the burn is not reverted, permanently destroying user tokens without issuing any NEAR-side credit.

---

### Finding Description

In `EvmErc20.sol`, `withdrawToNear` is implemented as:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // (1) burn first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // (2) res is NEVER checked — no revert on failure
    }
}
``` [1](#0-0) 

The precompile (`ExitToNear`) enforces a hard maximum input size:

```rust
const MAX_INPUT_SIZE: usize = 1_024;
``` [2](#0-1) 

This limit is enforced unconditionally in `parse_input`, which is called from `ExitToNearParams::try_from` before any other processing:

```rust
fn parse_input(input: &[u8]) -> Result<&[u8], ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    Ok(&input[1..])
}
``` [3](#0-2) 

```rust
fn validate_input_size(input: &[u8], min: usize, max: usize) -> Result<(), ExitError> {
    if input.len() < min || input.len() > max {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_INPUT")));
    }
    Ok(())
}
``` [4](#0-3) 

When the precompile returns `ExitError`, the EVM `call` opcode returns `0` in `res`. Because `res` is never inspected and no `require`/`revert` follows, the Solidity function returns successfully despite the failed exit.

**Trigger condition**: `recipient.length > 991` causes `input_size = 1 + 32 + recipient.length > 1024`, which the precompile rejects. The same flaw also fires for any other precompile rejection (invalid UTF-8, invalid NEAR account ID format, etc.).

The same pattern exists in `EvmErc20V2.sol` with threshold `recipient.length > 971` (due to the additional 20-byte sender field): [5](#0-4) 

**Note on the overflow sub-question**: Solidity 0.8+ reverts on arithmetic overflow, so `1 + 32 + recipient.length` cannot silently overflow. The actual vulnerability is solely the unchecked `res`.

---

### Impact Explanation

Tokens are permanently burned on the EVM side with no corresponding NEP-141 credit issued on NEAR. This decouples the EVM token supply from its NEAR backing, constituting **Insolvency**: the total EVM supply exceeds the NEAR-side collateral, and the affected user's funds are unrecoverable.

---

### Likelihood Explanation

Any unprivileged token holder can trigger this by calling `withdrawToNear` with a `recipient` bytes array longer than 991 bytes. No special privileges, no admin compromise, and no external dependency failure is required. The attacker only needs a token balance and the ability to submit an EVM transaction.

---

### Recommendation

Check the precompile return value and revert on failure:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. Additionally, add an explicit input-size guard before the burn to fail fast:

```solidity
require(1 + 32 + recipient.length <= 1024, "ERR_RECIPIENT_TOO_LONG");
_burn(_msgSender(), amount);
```

---

### Proof of Concept

1. Deploy `EvmErc20` on a local Aurora sandbox.
2. Mint tokens to `attacker`.
3. Call `withdrawToNear(recipient, amount)` where `recipient` is a 992-byte byte array (any content).
4. Observe: `_burn` succeeds (attacker's balance decreases), the assembly `call` returns `res = 0` (precompile rejected with `ERR_INVALID_INPUT`), but the function does **not** revert.
5. Assert: attacker's EVM balance is reduced by `amount`, but no NEP-141 tokens are credited on NEAR — the invariant is broken.
