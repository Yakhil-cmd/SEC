Audit Report

## Title
NNS/SNS Governance Proposal Queue Exhaustion via Bounded Ballot Retention - (File: rs/nns/governance/src/governance.rs, rs/sns/governance/src/proposal.rs)

## Summary
Both NNS and SNS governance canisters enforce a hard cap on proposals whose ballots have not yet been cleared (`MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS = 200` for NNS, `700` for SNS). Ballots are only cleared during periodic reward events (approximately daily), not at proposal decision time. An attacker with a sufficiently staked neuron can fill this queue, blocking all non-exempt new proposals until the next reward event clears the backlog — a window that can span multiple days when combined with wait-for-quiet extensions.

## Finding Description

**Bounded queue with delayed clearing:**

The NNS enforces the cap at `rs/nns/governance/src/governance.rs` L5254–5269 by counting proposals where `!info.ballots.is_empty() && !info.is_manage_neuron()`. The SNS enforces the equivalent at `rs/sns/governance/src/governance.rs` L3532–3547 by counting `!data.ballots.is_empty()`. The constants are defined at `rs/nns/governance/src/governance.rs` L252 (200) and `rs/sns/governance/src/proposal.rs` L79 (700). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

**Ballots cleared only on reward settlement, not on decision:**

The test at `rs/nns/governance/tests/governance.rs` L8089–8117 explicitly confirms this: after all proposals are rejected and `run_periodic_tasks()` is called, the queue is still full and new proposals are still blocked. Only after advancing time by `REWARD_DISTRIBUTION_PERIOD_SECONDS` and triggering another reward event (L8119–8122) does the queue drain and new proposals succeed. [5](#0-4) 

**Exploit flow:**

1. Attacker creates a neuron with sufficient stake (NNS: enough to cover 200 × rejection cost; SNS: enough to cover `reject_cost_e8s` × up to 700).
2. Attacker submits proposals up to the cap (200 NNS / 700 SNS).
3. Proposals are rejected (attacker lacks majority), but ballots remain populated.
4. All new non-exempt proposals from any participant are blocked with `ResourceExhausted` until the next daily reward event clears the ballots.
5. With wait-for-quiet, the voting window can extend by up to 4 days (NNS) or 2 days (SNS), further delaying ballot clearing.

**Existing mitigations and why they are insufficient:**

- `allowed_when_resources_are_low()`: Only a narrow set of critical proposal types (e.g., `InstallCode` for the Governance canister itself) bypass the limit. All ordinary governance proposals (Motion, parameter changes, treasury transfers, etc.) remain blocked.
- SNS `reject_cost_e8s` check (`rs/sns/governance/src/governance.rs` L3521): The neuron must hold enough stake to cover the rejection cost, but for SNS instances with low configured rejection costs this is a minimal barrier.
- NNS rejection cost (~10 ICP per proposal): Filling 200 slots costs ~2,000 ICP, which is a meaningful but not prohibitive economic barrier for a well-funded attacker seeking to disrupt governance for days. [6](#0-5) [7](#0-6) 

## Impact Explanation

This is an **application/platform-level DoS on NNS and SNS governance** — a concrete allowed High impact. An attacker can prevent any ordinary governance proposal from being submitted for a window of 1–8+ days (voting period + wait-for-quiet extension + time to next reward event). For SNS DAOs, this can freeze all governance activity including treasury operations and parameter changes. For NNS, it blocks all non-exempt proposals affecting the Internet Computer's governance layer. This matches: *"Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS" — High ($2,000–$10,000).*

## Likelihood Explanation

- **NNS:** Requires ~2,000 ICP staked (burned on rejection). Feasible for a well-funded adversary. Attack is repeatable after each reward event if the attacker is willing to keep burning ICP.
- **SNS:** Requires filling 700 slots at the configured `reject_cost_e8s`. For SNS instances with low rejection costs (e.g., near-zero), the attack is cheap and easily repeatable. No special privileges are needed beyond holding a neuron with the minimum dissolve delay.
- The attack is deterministic, requires no timing luck beyond submitting before the daily reward event, and is confirmed by an existing unit test.

## Recommendation

1. **Clear ballots eagerly on final decision** (accepted/rejected/expired) rather than waiting for the reward event. Reward accounting can be preserved by recording the proposal's reward weight at decision time and settling it lazily during the reward event without needing the full ballot map.
2. **Per-neuron proposal rate limiting:** Limit the number of open (ballot-populated) proposals attributable to a single neuron or principal within a rolling window.
3. **Increase the economic barrier:** For NNS, consider raising the rejection cost or requiring a higher minimum stake for proposal submission. For SNS, enforce a minimum `reject_cost_e8s` in `NervousSystemParameters` validation.
4. **Shorter default voting periods for low-stakes proposal types** (e.g., Motion) to reduce the window during which ballots remain populated.

## Proof of Concept

The existing test at `rs/nns/governance/tests/governance.rs` L8089–8117 already constitutes a reproducible PoC: it fills the queue to `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS`, rejects all proposals, calls `run_periodic_tasks()`, and confirms that new proposals are still blocked. A minimal extension of this test would:

1. Submit 200 Motion proposals from a single neuron (NNS) or 700 from a single neuron (SNS).
2. Advance time past the voting deadline and call `run_periodic_tasks()` — confirm all are `Rejected`.
3. Attempt to submit a new proposal from a *different* neuron — confirm `ResourceExhausted`.
4. Advance time by less than `REWARD_DISTRIBUTION_PERIOD_SECONDS` — confirm still blocked.
5. Advance time by `REWARD_DISTRIBUTION_PERIOD_SECONDS` and trigger reward event — confirm unblocked. [8](#0-7)
