Audit Report

## Title
`SenderInfoContent` Canister Signature Does Not Bind to Sender Principal, Enabling Cross-Sender Replay - (File: `rs/types/types/src/messages/http.rs`)

## Summary
`SenderInfoContent` signs only the raw `info` blob under the `"ic-sender-info"` domain separator; the sender's principal is absent from the signed bytes. Any observer of a valid ingress message can copy the `{info, signer, sig}` triple verbatim into a new request under a different principal. `verify_sender_info_canister_sig` accepts the replayed canister signature because it never checks that the signature is bound to the requesting principal, allowing a canister relying on `ic0.msg_caller_info_data()` to receive a replayed attestation as if it were freshly issued for the attacker.

## Finding Description
`SenderInfoContent` is a newtype over a raw byte slice whose `SignedBytesWithoutDomainSeparator` implementation appends only those bytes to the signed buffer:

```rust
// rs/types/types/src/messages/http.rs:342-348
pub struct SenderInfoContent<'a>(pub &'a [u8]);

impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.0);   // only info blob, no sender principal
    }
}
```

The full signed bytes are therefore `"\x0Eic-sender-info" || info_bytes`. The sender's `UserId` and the target canister ID are absent.

`verify_sender_info_canister_sig` (lines 494–544 of `rs/validator/src/ingress_validation.rs`) performs exactly three checks:
1. `sender_pubkey` is a valid canister-signature public key.
2. The canister ID embedded in `sender_pubkey` matches `sender_info.signer`.
3. The canister signature σ is valid over `SenderInfoContent(&sender_info.info)`.

None of these checks bind σ to the principal of the request sender. The message-ID hash (`representation_independent_hash_call_or_query`, lines 43–79 of `rs/types/types/src/messages/http.rs`) does include the `{info, signer, sig}` triple alongside the `sender` field, so the outer envelope signature commits to both the sender principal and the copied σ. However, this does not prevent replay: attacker B can copy the same σ, construct a new message with `sender: P_B` and the identical `sender_info`, compute a fresh message-ID (which now commits to P_B and the same σ), and sign it with B's own key. The envelope signature is valid for B's message, and σ remains a valid canister signature over `SenderInfoContent(info)` regardless of who the sender is. `validate_sender_info` (lines 459–488) has no additional principal-binding check.

## Impact Explanation
Any canister that uses `ic0.msg_caller_info_data()` to make security-sensitive decisions — the explicitly intended use case for Internet Identity attesting KYC status, age verification, or user-specific credentials — can be deceived. An unprivileged attacker with only their own key can present another user's attested attributes to any canister, potentially bypassing access controls, gaining unauthorized privileges, or corrupting per-user state. Internet Identity is explicitly listed in the ICP bounty scope. This matches the **High** impact class: "Significant Internet Identity security impact with concrete user or protocol harm," given that the feature is designed precisely for II to attest caller attributes and the attack requires no privileged access.

## Likelihood Explanation
Ingress messages are publicly observable on the IC. Any observer can extract `{info, signer, sig}` from any accepted ingress message and replay it immediately in a fresh request under their own principal. No privileged access, no key material beyond the attacker's own, and no cryptographic break are required. The only constraint is finding a target canister that acts on `msg_caller_info_data()` without re-checking the sender principal inside the blob — which is the natural and intended usage pattern for the feature.

## Recommendation
Include the sender's principal bytes in the signed content of `SenderInfoContent`:

```rust
pub struct SenderInfoContent<'a> {
    pub sender: &'a [u8],
    pub info: &'a [u8],
}

impl SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.sender);
        bytes.extend_from_slice(self.info);
    }
}
```

`validate_sender_info` must then pass `request.sender().get().as_slice()` when constructing `SenderInfoContent` for verification, ensuring the canister signature is cryptographically bound to the specific principal that submitted the request.

## Proof of Concept
**Setup:** Internet Identity canister `II` issues `σ = II.sign("\x0Eic-sender-info" || b"age_verified=true")` for user A (principal `P_A`). User A submits an ingress call; the `sender_info = {info: b"age_verified=true", signer: II, sig: σ}` is visible in the public ingress pool.

**Replay:** Attacker B (principal `P_B`) constructs:
```rust
HttpCanisterUpdate {
    sender: Blob(P_B.as_slice().to_vec()),
    sender_info: Some(RawSignedSenderInfo {
        info: Blob(b"age_verified=true".to_vec()),
        signer: Blob(II.get().as_slice().to_vec()),
        sig: Blob(sigma.clone()),   // copied verbatim from A's request
    }),
    ...
}
```
B computes the message-ID from `representation_independent_hash_call_or_query` (which includes P_B and the copied σ) and signs it with B's own key. The request passes all validation in `validate_sender_info` / `verify_sender_info_canister_sig` because σ is a valid canister signature over `SenderInfoContent(b"age_verified=true")` regardless of sender. The target canister receives `msg_caller_info_data() = b"age_verified=true"` while `msg_caller() = P_B`.

A deterministic unit test can be written against `verify_sender_info_canister_sig` directly: generate a valid canister signature over `SenderInfoContent(info)` for principal P_A, then call the verifier with the same `sender_info` but a different sender principal P_B and confirm it returns `Ok(())`.
