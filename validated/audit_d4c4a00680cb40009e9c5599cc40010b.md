### Title
Stale ICP/XDR Conversion Rate Used Without Freshness Check Enables Cycles Over-Minting - (File: rs/nns/cmc/src/main.rs)

### Summary
The Cycles Minting Canister (CMC) converts ICP to cycles using a stored `icp_xdr_conversion_rate` that is updated via heartbeat every 5 minutes. The `tokens_to_cycles()` function reads this rate with no staleness check. If the Exchange Rate Canister (XRC) becomes unavailable for an extended period, the stored rate grows arbitrarily stale. Any unprivileged user who calls `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` during a period of significant ICP price decline will receive more cycles than the current market rate justifies, constituting a cycles/resource accounting bug.

### Finding Description
`tokens_to_cycles()` in `rs/nns/cmc/src/main.rs` reads `state.icp_xdr_conversion_rate` and uses only its `xdr_permyriad_per_icp` field, completely ignoring `timestamp_seconds`:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);   // timestamp ignored
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles {
                xdr_permyriad_per_icp,
                cycles_per_xdr: state.cycles_per_xdr,
            }
            .to_cycles(amount)),
            ...
        }
    })
}

This function is called unconditionally by all three public cycle-minting paths:
- `process_top_up` → `notify_top_up`
- `process_create_canister` → `notify_create_canister`
- `process_mint_cycles` → `notify_mint_cycles` [2](#0-1) [3](#0-2) [4](#0-3) 

The rate is refreshed via heartbeat at most every `REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS`: [5](#0-4) 

When the XRC call fails, the error is logged and the old rate is silently retained — there is no circuit-breaker that blocks minting when the rate is too old: [6](#0-5) 

The rate update state machine transitions to `GetRateAt(next_minute)` on failure, meaning the CMC will keep retrying but will continue serving the stale rate to all minting callers in the interim: [7](#0-6) 

### Impact Explanation
Cycles are the fuel of the IC and have a fixed XDR-denominated cost. If the stored ICP/XDR rate is stale at a value higher than the current market rate (i.e., ICP has fallen in XDR terms since the last successful update), every `notify_top_up` / `notify_create_canister` / `notify_mint_cycles` call mints more cycles per ICP than the protocol should issue. This is a direct cycles over-minting / resource accounting bug: the protocol gives away more compute resources than the deposited ICP is worth at current prices, diluting the economic value of cycles for all participants.

### Likelihood Explanation
The XRC is an external system canister. Transient unavailability (canister upgrades, subnet congestion, validation failures such as `StablecoinRateTooFewRates`) is a documented and tested scenario. The integration test explicitly exercises the case where the XRC returns an error and the CMC retains the prior rate: [8](#0-7) 

During any such outage, the rate can drift arbitrarily far from market reality. A sophisticated user monitoring the CMC's certified rate (publicly readable via `get_icp_xdr_conversion_rate`) and the live ICP market price can time their ICP-to-cycles conversion to exploit the deviation. No privileged access is required; the three notify endpoints are open to any principal.

### Recommendation
Add a maximum-age guard in `tokens_to_cycles()`. Before using the stored rate, compare `rate.timestamp_seconds` against `now_seconds()`. If the rate is older than a defined threshold (e.g., `REFRESH_RATE_INTERVAL_SECONDS * 2` or a configurable `max_rate_age_seconds`), return a `NotifyError` indicating the rate is stale rather than minting cycles at a potentially incorrect price. This mirrors the fix described in the external report: prioritize the live feed and refuse to operate on a price that has not been refreshed within an acceptable window.

### Proof of Concept
1. Observe the CMC's current `icp_xdr_conversion_rate` via `get_icp_xdr_conversion_rate` — e.g., `xdr_permyriad_per_icp = 50_000` (5 XDR/ICP).
2. The XRC becomes unavailable (e.g., due to a canister upgrade or sustained validation errors). The CMC heartbeat logs `FailedToRetrieveRate` and retains the old rate.
3. ICP market price drops 20% — the true rate is now ~40,000 XDR/ICP.
4. Attacker calls `notify_top_up` with 1 ICP. `tokens_to_cycles` reads `xdr_permyriad_per_icp = 50_000` (no age check), minting cycles equivalent to 5 XDR instead of the correct 4 XDR — a 25% over-minting.
5. The attacker repeats this for as long as the XRC remains unavailable and the rate deviation persists, extracting excess cycles at the protocol's expense. [9](#0-8) [10](#0-9)
