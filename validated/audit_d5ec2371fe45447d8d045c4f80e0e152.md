Audit Report

## Title
SNS Governance Permanent DoS via Unbounded Ballot Iteration in `distribute_rewards()` Without Instruction-Limit Guard - (`rs/sns/governance/src/governance.rs`)

## Summary
SNS governance enforces a 700-proposal ballot cap, but three action types bypass it via `allowed_when_resources_are_low()`. The synchronous `distribute_rewards()` function iterates over every `ReadyToSettle` proposal and every ballot with no instruction-limit guard. If enough bypass proposals accumulate, every invocation of `distribute_rewards()` traps before clearing ballots, creating a self-reinforcing stuck state where proposals are never settled, the cap is never freed, and voting rewards are never distributed.

## Finding Description

**Confirmed constants and bypass:**

`MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS = 700` at `rs/sns/governance/src/proposal.rs:79` and `MAX_NUMBER_OF_GENERIC_NERVOUS_SYSTEM_FUNCTIONS = 200_000` at `rs/sns/governance/src/proposal.rs:82`.

The cap check at `rs/sns/governance/src/governance.rs:3532-3547` is bypassed for `ExecuteGenericNervousSystemFunction`, `UpgradeSnsControlledCanister`, and `UpgradeSnsToNextVersion` via `allowed_when_resources_are_low()` at `rs/sns/governance/src/types.rs:1751-1762`. Any token holder with sufficient stake can submit `ExecuteGenericNervousSystemFunction` proposals indefinitely past the 700-proposal cap.

**Unbounded synchronous loop in `distribute_rewards()`:**

At `rs/sns/governance/src/governance.rs:5892-5931`, the function performs a nested loop: for every `ReadyToSettle` proposal, it iterates every ballot entry, calling `NeuronId::from_str`, performing `HashMap` insertions, and executing `Decimal` arithmetic. With 700+ proposals each carrying ballots for up to tens of thousands of neurons, this easily exceeds the 40B instruction limit.

`distribute_rewards()` is synchronous (called as `self.distribute_rewards(supply)` at line 5513, no `.await`), so it cannot use `noop_self_call_if_over_instructions`, which requires `await`. No instruction-limit guard exists anywhere in the SNS governance path â€” confirmed by grep returning zero results for instruction-related patterns in `rs/sns/governance/src/governance.rs`.

**Self-reinforcing stuck state:**

`ballots.clear()` at `rs/sns/governance/src/governance.rs:6080` and `reward_event_end_timestamp_seconds` assignment at line 6071 are only reached at the end of `distribute_rewards()`. If the function traps mid-execution, proposals remain in `ReadyToSettle` with full ballot maps, still counting toward `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS`. Every subsequent timer invocation hits the same over-limit loop and traps again. The state is permanent without an emergency canister upgrade.

**Contrast with NNS governance:**

NNS governance's `process_voting_state_machines()` at `rs/nns/governance/src/voting.rs:254-266` uses `noop_self_call_if_over_instructions` with soft and hard limits to break out of the loop before exhausting the instruction budget. SNS governance has no equivalent protection.

## Impact Explanation

This is a High-severity application/platform-level DoS of the SNS governance canister. Once triggered: no new regular proposals can be submitted (cap is full), voting rewards stop being distributed permanently, and the only recovery path is an emergency SNS upgrade proposal â€” which itself requires sufficient voting power to pass. This matches the allowed impact: "Application/platform-level DoS... or subnet availability impact not based on raw volumetric DDoS" (High, $2,000â€“$10,000).

## Likelihood Explanation

An attacker needs enough SNS governance tokens to pay `reject_cost_e8s` (typically ~1 governance token) per proposal submitted. They do not need to pass any proposals â€” submitted proposals that expire or are rejected still retain ballots until settled. The attacker submits enough proposals to push the per-round ballot iteration cost over the instruction limit. This is a targeted, funded attack, not a casual one, but it is reachable by a motivated actor or can occur spontaneously during periods of high governance activity with large neuron counts.

## Recommendation

1. **Add an instruction-limit guard in `process_proposals()`** analogous to NNS governance's `noop_self_call_if_over_instructions` pattern at `rs/nns/governance/src/voting.rs:254-266`. Since `distribute_rewards()` is synchronous, refactor it to be async or introduce a per-round proposal cap so a single invocation processes at most N proposals, carrying the remainder to the next round.
2. **Add a separate, lower cap for bypass proposals** or apply `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS` uniformly to all proposal types to prevent unbounded accumulation past the cap.
3. **Ensure `ballots.clear()` is reached incrementally** (e.g., settle proposals in batches) so partial progress is preserved across timer invocations.

## Proof of Concept

1. Register a `GenericNervousSystemFunction` in the SNS.
2. Submit 700 `Motion` proposals to fill `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS`.
3. Submit additional `ExecuteGenericNervousSystemFunction` proposals past the cap (each accepted due to `allowed_when_resources_are_low()`).
4. Wait for the voting period of the first 700 proposals to expire; they become `ReadyToSettle`.

6. Observe that `ballots.clear()` is never reached, proposals remain `ReadyToSettle`, the cap remains full, and no new regular proposals can be submitted. The governance canister is permanently stuck until an emergency upgrade is executed.
