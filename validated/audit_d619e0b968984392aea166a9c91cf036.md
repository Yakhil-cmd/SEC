Audit Report

## Title
Unbounded Synchronous BFS in SNS Governance `cast_vote_and_cascade_follow` Lacks Instruction-Limit Protection, Enabling Governance DoS — (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance canister's `cast_vote_and_cascade_follow` is a synchronous static function that traverses the entire neuron-following graph via BFS with no instruction-limit check or async continuation. The NNS governance equivalent is `async`, uses explicit soft/hard instruction limits, and can yield across rounds via `noop_self_call_if_over_instructions`. With `MAX_NUMBER_OF_NEURONS_CEILING = 200,000` and up to 15 followees per function/topic per neuron, a sufficiently large following graph can exhaust the per-message instruction budget during a `register_vote` call, causing the canister to trap, rolling back the vote, and permanently blocking governance if the condition persists.

## Finding Description
**Root cause**: `cast_vote_and_cascade_follow` in `rs/sns/governance/src/governance.rs` is declared as a plain synchronous `fn` with no instruction-limit guard anywhere in its BFS loop:

```rust
fn cast_vote_and_cascade_follow(
    proposal_id: &ProposalId,
    voting_neuron_id: &NeuronId,
    ...
    ballots: &mut BTreeMap<String, Ballot>,
    topic: Topic,
) {
    ...
    while !induction_votes.is_empty() {  // ← no instruction check
        ...
    }
}
```

A grep across all SNS governance source files for `is_message_over_threshold`, `instruction_counter`, `over_soft_message_limit`, and `noop_self_call` returns zero matches, confirming no instruction-limit protection exists anywhere in the SNS governance codebase.

**Contrast with NNS**: `rs/nns/governance/src/voting.rs` defines `cast_vote_and_cascade_follow` as `async`, with:
- `SOFT_VOTING_INSTRUCTIONS_LIMIT = 1B` instructions
- `HARD_VOTING_INSTRUCTIONS_LIMIT = 750B` instructions
- A loop that calls `noop_self_call_if_over_instructions(...)` after each BFS tier to yield and continue in a subsequent round

**Exploit path**:
1. Any principal with `Vote` permission on a neuron can call `manage_neuron { command: RegisterVote { ... } }` — an unprivileged ingress call.
2. `register_vote` calls `cast_vote_and_cascade_follow` synchronously and returns immediately after.
3. If the following graph is large enough (e.g., 100,000+ neurons in a chain, each following the previous on every function and topic), the BFS traverses all neurons in a single message.
4. The IC per-message instruction budget (~5 billion instructions) is exhausted; the canister traps.
5. All state changes are rolled back; the vote is never recorded.
6. Every subsequent `register_vote` call on any proposal with the same following graph also traps — no neuron can vote, no proposal can reach quorum, and governance is permanently frozen until a canister upgrade is deployed.

**Existing checks are insufficient**: The only limits enforced are `MAX_NUMBER_OF_NEURONS_CEILING = 200,000` and `MAX_FOLLOWEES_PER_FUNCTION_CEILING = 15` / `MAX_FOLLOWEES_PER_TOPIC = 15`, which bound the graph size but do not prevent instruction exhaustion during traversal. These constants are ceilings on SNS parameters, not runtime instruction guards.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS."* A triggered condition permanently prevents any neuron from registering a vote on any proposal in the affected SNS instance. Proposals cannot reach quorum or be decided, the SNS treasury and upgrade authority are frozen, and the only recovery path is a canister upgrade — which itself requires governance to function, creating a potential deadlock. The impact is scoped to the specific SNS instance, not the entire IC, which places this below Critical.

## Likelihood Explanation
Creating 200,000 neurons requires staking SNS tokens for each, imposing a real economic cost that scales with token price. For SNS instances with high token supply or low token price, this cost may be feasible for a motivated attacker. More importantly, organic growth of a large SNS (e.g., via swap neuron baskets that use topic-based following via `SetFollowing`) can approach this limit without any attacker involvement. The recently added topic-based following multiplies the number of following edges per neuron, increasing BFS cost per neuron visited. The attack is repeatable: once the following graph is large enough, every `register_vote` call traps indefinitely.

## Recommendation
1. **Mirror the NNS pattern**: Refactor SNS `cast_vote_and_cascade_follow` to be `async`, introduce a `VotingStateMachine` equivalent, and add soft/hard instruction-limit checks with timer-based continuation, exactly as implemented in `rs/nns/governance/src/voting.rs` (lines 20–31 and 150–176).
2. **Add a per-BFS-round neuron cap**: As an interim measure, limit the number of neurons processed per `register_vote` call and defer remaining BFS work to a timer.
3. **Benchmark the synchronous path**: Run a canister benchmark (`canbench`) against the maximum neuron count and following depth to confirm the exact instruction budget at which the trap occurs.

## Proof of Concept
1. Deploy an SNS with `max_number_of_neurons = 200_000` and `max_followees_per_function = 15`.
2. Create N neurons (e.g., 100,000+) and configure them in a chain: `neuron[i]` follows `neuron[i-1]` on every registered function ID and every topic via `SetFollowing`.
3. Submit a proposal. All neurons receive ballots.
4. Have `neuron[0]` call `manage_neuron { RegisterVote { proposal, vote: Yes } }`.
5. `register_vote` calls `cast_vote_and_cascade_follow` synchronously; the BFS traverses all N neurons in a single message with no instruction check.
6. Observe the call trap due to instruction exhaustion; the vote is rolled back.
7. Repeat with any neuron: every `register_vote` call traps while the following graph remains this large.

A deterministic integration test using PocketIC can reproduce this by pre-populating the neuron store with a chain following graph and asserting that `register_vote` returns an error or that the canister traps before the ballot is recorded.
