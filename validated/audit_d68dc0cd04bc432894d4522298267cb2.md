Audit Report

## Title
ECDSA `read_state` Envelope Uses Wrong `sender_pubkey` in `construction_combine` — (`rs/rosetta-api/icp/src/request_handler/construction_combine.rs`)

## Summary
In the `SignatureType::Ecdsa` branch of `read_state_envelope` construction within `construction_combine`, `sender_pubkey` is populated from `transaction_signature.public_key.hex_bytes` (line 142) instead of `read_state_signature.public_key.hex_bytes`. The `sender_sig` is correctly taken from `read_state_signature` (line 156). When a caller supplies two distinct ECDSA key pairs — one per signing payload — the resulting read-state envelope carries a mismatched pubkey/signature pair that the IC replica unconditionally rejects with HTTP 400, permanently preventing the Rosetta node from confirming the transaction's finality.

## Finding Description
`signatures_by_sig_data` is keyed by signing-payload hash; `transaction_signature` and `read_state_signature` are looked up independently and can differ freely: [1](#0-0) 

The Ed25519 branch of `read_state_envelope` correctly sources `sender_pubkey` from `read_state_signature`: [2](#0-1) 

The ECDSA branch erroneously sources `sender_pubkey` from `transaction_signature` instead: [3](#0-2) 

While `sender_sig` is correctly taken from `read_state_signature`: [4](#0-3) 

There is no guard anywhere in `construction_combine` asserting that both signatures share the same public key. The IC replica's own regression test confirms that a `read_state` envelope whose `sender_pubkey` does not match `sender_sig` returns HTTP 400: [5](#0-4) 

## Impact Explanation
An attacker (or misconfigured client) submits `POST /construction/combine` with two distinct ECDSA key pairs. The update envelope is well-formed (`sender_pubkey = DER(key_A)`, `sender_sig = sig_A`) and the IC replica accepts it — the ICP transfer executes. The read-state envelope is malformed (`sender_pubkey = DER(key_A)`, `sender_sig = sig_B`) and the IC replica rejects it with HTTP 400. The Rosetta node permanently loses the ability to poll that transaction's status. Any exchange or custodian relying on Rosetta for finality confirmation sees the transaction as perpetually unconfirmed; depending on the operator's retry/refund logic this leads to double-crediting or double-withdrawal of ICP. This matches the allowed High impact: **Significant Rosetta security impact with concrete user or protocol harm** ($2,000–$10,000).

## Likelihood Explanation
The endpoint `POST /construction/combine` is public and unauthenticated. No privilege is required. The attacker generates two ECDSA key pairs and places them as separate entries in the `signatures` array — one keyed to the update payload hash, one to the read-state payload hash. The Rosetta API performs no cross-signature key-equality check. The bug is invisible in the common single-key case, making it easy to miss in normal testing and regression suites. It is trivially repeatable.

## Recommendation
On line 142, replace `transaction_signature.public_key.hex_bytes` with `read_state_signature.public_key.hex_bytes`:

```rust
// ECDSA branch — read_state_envelope (line 142)
Secp256k1KeyPair::hex_decode_pk(
    &read_state_signature.public_key.hex_bytes,  // was: transaction_signature
)
```

Additionally, add an explicit guard after both signatures are resolved that returns `ApiError::InvalidRequest` if `transaction_signature.public_key != read_state_signature.public_key`, enforcing the invariant that both envelopes must share the same IC principal. [6](#0-5) 

## Proof of Concept
```rust
// Unit test outline
let key_a = Secp256k1KeyPair::generate();
let key_b = Secp256k1KeyPair::generate(); // distinct key

let signatures = vec![
    Signature {
        signing_payload: tx_payload,
        public_key: key_a.pubkey(),
        hex_bytes: key_a.sign(tx_payload),
        signature_type: Ecdsa,
    },
    Signature {
        signing_payload: rs_payload,
        public_key: key_b.pubkey(),
        hex_bytes: key_b.sign(rs_payload),
        signature_type: Ecdsa,
    },
];

let response = handler.construction_combine(request_with(signatures));
let signed_tx = decode(response.signed_transaction);

// Demonstrates the bug: read_state envelope carries key_A pubkey with key_B sig
assert_eq!(
    signed_tx.requests[0].1[0].read_state.sender_pubkey,
    Some(Blob(key_a.der_encoded()))  // BUG: should be key_b.der_encoded()
);
// Submitting this to the IC replica returns HTTP 400 on read_state,
// confirmed by rs/tests/crypto/request_signature_test.rs lines 663-683
```
