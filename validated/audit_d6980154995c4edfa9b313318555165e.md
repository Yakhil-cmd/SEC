Audit Report

## Title
Missing `GlobalConcurrencyLimitLayer` on Synchronous Call Endpoints Enables Resource-Exhaustion DoS - (File: `rs/http_endpoints/public/src/lib.rs`)

## Summary
The `make_router` function in `rs/http_endpoints/public/src/lib.rs` applies `GlobalConcurrencyLimitLayer` to every public HTTP endpoint except the three synchronous call routes (`call_v3_router`, `call_v4_router`, `subnet_call_v4_router`). The omission is explicitly acknowledged with a TODO comment. Because the `call_sync` handler holds each tokio task open for up to 10 seconds awaiting ingress certification, an attacker can accumulate an unbounded number of concurrent long-lived tasks, exhausting replica memory and tokio task budget, taking the node offline.

## Finding Description
In `rs/http_endpoints/public/src/lib.rs` at lines 606–609, the three synchronous call routers are merged into the final router without any concurrency-limiting middleware:

```rust
// TODO(CON-1574): see if there is any reasonable explicit concurrency limit we could use here.
.merge(http_handler.call_v3_router)
.merge(http_handler.call_v4_router)
.merge(http_handler.subnet_call_v4_router)
```

Every other endpoint — `call_v2_router` (limit 50), all query routers, all read-state routers, catchup, dashboard, status, pprof, tracing — is wrapped in `service_builder(GlobalConcurrencyLimitLayer::new(...))` before being merged.

The handler backing these three routes is `call_sync` in `rs/http_endpoints/public/src/call/call_sync.rs`. After ingress validation and submission, it blocks the tokio task waiting for certification notification for up to `ingress_message_certificate_timeout_seconds` (default **10 seconds**, confirmed in `rs/config/src/http_handler.rs` line 88):

```rust
match certification_subscriber
    .wait_for_certification()
    .timeout(Duration::from_secs(
        ingress_message_certificate_timeout_seconds,
    ))
    .await
```

The ingress watcher subscription channel has a capacity of `INGRESS_WATCHER_CHANNEL_SIZE = 1000` (`rs/http_endpoints/public/src/call/ingress_watcher.rs` line 22), but this only limits the registration backlog — it does not bound the number of live HTTP tasks already past the channel send. Each accepted request holds a tokio task, a `CancellationToken`/`DropGuard`, a `Notify` arc, and up to 5 MB of request body (`max_request_size_bytes`) for the full timeout duration.

The HTTP/2 `SETTINGS_MAX_CONCURRENT_STREAMS` (default 1000 per connection) provides a per-connection limit, but an attacker can open multiple connections, each saturating their stream limit, with no global cap.

## Impact Explanation
An attacker flooding `/api/v3/canister/.../call` or `/api/v4/...` with valid ingress envelopes causes unbounded accumulation of tokio tasks each held for up to 10 seconds. With enough concurrent requests, the replica process exhausts available memory and is OOM-killed, taking the node offline and reducing subnet fault tolerance below the Byzantine threshold if multiple nodes are targeted. This matches the **High** bounty impact: *Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS* — the attack exploits a specific missing application-level concurrency control, not a network flood.

## Likelihood Explanation
Any unprivileged caller reachable to port 8080 (directly or via boundary node) can trigger this. Boundary-node rate limiting is **opt-in** via CLI flags (`--rate-limit-per-second-per-ip`, `--rate-limit-per-second-per-subnet`) and limits request *rate*, not concurrent in-flight connections. A single IP can open many simultaneous long-lived HTTP/2 streams within the rate window. The TODO comment in the source confirms the gap is known and unresolved. No special privileges, leaked keys, or governance majority are required.

## Recommendation
Wrap the three unguarded routers with `GlobalConcurrencyLimitLayer` in `make_router`, using either the existing `config.max_call_concurrent_requests` (50) or a new dedicated config field (e.g., `max_sync_call_concurrent_requests`) in `rs/config/src/http_handler.rs`. Add a load-shedding integration test analogous to the existing `test_load_shedding_update_call` in `rs/http_endpoints/public/tests/load_shed_test.rs`.

## Proof of Concept
1. Obtain any valid canister ID on an IC subnet.
2. Craft a valid CBOR-encoded `HttpRequestEnvelope<HttpCallContent>` ingress message.
3. Open N concurrent HTTP/2 connections (or streams) to the replica's port 8080, all issuing `POST /api/v3/canister/<canister_id>/call` with the crafted envelope.
4. Each request passes ingress validation, subscribes to the ingress watcher, and blocks for up to 10 seconds.
5. With N in the tens of thousands, the replica process exhausts memory and is OOM-killed.
