Audit Report

## Title
ckERC20 Minter Uses Single Global Gas Limit for All ERC20 Tokens, Causing Irrecoverable ckETH Loss on Out-of-Gas Failures - (File: rs/ethereum/cketh/minter/src/withdraw.rs)

## Summary
The ckETH minter applies a single hardcoded `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT = 65_000` to every ckERC20 withdrawal regardless of the specific ERC20 contract's actual gas requirements. When an ERC20 contract consumes more than 65,000 gas, the Ethereum transaction reverts out-of-gas; the ckERC20 tokens are reimbursed but the ckETH burned for the transaction fee is permanently lost. The documentation explicitly acknowledges both the fixed limit and the non-reimbursement of overcharged fees.

## Finding Description
Two constants are declared in `rs/ethereum/cketh/minter/src/withdraw.rs` at lines 43–44:

```rust
pub const CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT: GasAmount = GasAmount::new(21_000);
pub const CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT: GasAmount = GasAmount::new(65_000);
```

`estimate_gas_limit` at lines 296–301 returns the same constant for every ckERC20 token regardless of which contract is the destination:

```rust
pub fn estimate_gas_limit(withdrawal_request: &WithdrawalRequest) -> GasAmount {
    match withdrawal_request {
        WithdrawalRequest::CkEth(_)   => CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT,
        WithdrawalRequest::CkErc20(_) => CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT,
    }
}
```

`create_transactions_batch` (lines 249–294) calls `estimate_gas_limit` and passes the result to `create_transaction`, which embeds it verbatim into the EIP-1559 transaction at `rs/ethereum/cketh/minter/src/state/transactions/mod.rs` lines 1169–1184.

On finalization, `record_finalized_transaction` (lines 733–745 of `state/transactions/mod.rs`) handles `TransactionStatus::Failure` for ckERC20 by creating a reimbursement request for only `request.withdrawal_amount` (the ckERC20 tokens). The ckETH burned for the transaction fee is not included in the reimbursement. The documentation at `rs/ethereum/cketh/docs/ckerc20.adoc` line 275 confirms: *"Overcharged transaction fees are not reimbursed."*

The `eip_1559_transaction_price` query in `main.rs` lines 173–197 also returns a fee estimate based on the same constant for every ckERC20 token, so users cannot distinguish tokens with higher gas requirements and cannot make an informed decision before withdrawing.

## Impact Explanation
Any user who withdraws a supported ckERC20 token whose underlying ERC20 contract requires more than 65,000 gas suffers a permanent loss of ckETH equal to `max_fee_per_gas × 65,000`. At 30 gwei base fee this is approximately 0.002 ETH (~$5–$10 at current prices) per failed withdrawal. The loss is irreversible: the minter mints back the ckERC20 tokens but never mints back the ckETH fee. This constitutes a concrete, user-facing financial loss in an in-scope ck-token/Chain Fusion component, matching the High impact class: *"Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."*

## Likelihood Explanation
An unprivileged user triggers the path by calling `withdraw_erc20` for any supported token whose contract exceeds the gas limit. USDT (Tether), already supported on mainnet, consumes 63,000–65,000 gas per transfer — right at the boundary. Three realistic triggers exist without any privileged access: (1) a supported token contract is upgraded to consume slightly more gas (USDT has been upgraded before); (2) a new ckERC20 token is added via NNS proposal whose contract has transfer hooks or fee logic exceeding 65,000 gas; (3) storage slot initialization on first transfer to a new address pushes gas above the limit. No attacker capability beyond being a normal user is required.

## Recommendation
1. Store a per-token `gas_limit` field alongside each supported ckERC20 token in the minter state, set at token-addition time via the NNS proposal that adds the token.
2. Expose an NNS-gated update path to revise the gas limit for existing tokens without a full minter upgrade.
3. Raise the default to a safer value (e.g., 100,000) to provide headroom for non-standard contracts.
4. Consider reimbursing the ckETH fee when the transaction fails due to out-of-gas (detectable via `gasUsed == gasLimit` in the receipt) to eliminate the permanent loss scenario.

## Proof of Concept
1. Identify or add a supported ckERC20 token whose underlying ERC20 contract requires >65,000 gas per `transfer` call (e.g., a token with transfer hooks, or USDT after a contract upgrade).
2. Call `withdraw_erc20` for that token; the minter burns ckETH (`max_fee_per_gas × 65,000`) and the ckERC20 tokens.
3. The minter constructs an EIP-1559 transaction with `gas_limit = 65,000` and broadcasts it to Ethereum.
4. Ethereum executes the transaction; it reverts out-of-gas (all 65,000 gas consumed).
5. The minter reads the `Failure` receipt in `record_finalized_transaction`; it creates a reimbursement request for the ckERC20 tokens only (`request.withdrawal_amount`).
6. `process_reimbursement` mints back the ckERC20 tokens; the ckETH burned in step 2 is never minted back.
7. The user's ckETH balance is permanently reduced by the transaction fee.

A deterministic integration test can be written using PocketIC with a mock ERC20 contract that consumes exactly 66,000 gas per transfer, asserting that after a failed withdrawal the ckERC20 balance is restored but the ckETH balance is not.
