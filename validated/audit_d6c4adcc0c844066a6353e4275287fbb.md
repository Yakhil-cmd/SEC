Audit Report

## Title
`inspect_message` Authorization Bypass via Non-Replicated Query Leaks Confidential Incident and Rule UUIDs — (`rs/boundary_node/rate_limits/canister/canister.rs`)

## Summary
The `inspect_message` hook is intended to gate `get_config` so only `FullAccess` or `FullRead` principals can invoke it, but `inspect_message` on ICP fires exclusively for ingress update messages and never for query calls. Since `get_config` is declared `#[query]`, any anonymous caller can invoke it as a non-replicated query, bypassing `inspect_message` entirely. The handler itself does not reject `RestrictedRead` callers; it applies `ConfigConfidentialityFormatter`, which redacts `rule_raw` and `description` but unconditionally preserves `rule_id` and `incident_id` for every rule, including non-disclosed confidential ones, leaking the UUID of every active security incident and its associated rules.

## Finding Description
**Root cause — `inspect_message` does not fire for query calls:**

`inspect_message` is registered at `canister.rs` lines 34–55 and checks that callers of `"get_config"` have `FullAccess` or `FullRead`, trapping otherwise. However, per the ICP execution model, `inspect_message` is invoked only for ingress update messages in the pre-consensus phase. `get_config` is declared `#[query]` at `canister.rs` line 110 and is published as `query` in `interface.did` line 149. Non-replicated query calls never pass through `inspect_message`, so the trap at lines 52–55 is never reached for any query caller.

**`get_config` handler does not reject `RestrictedRead`:**

`ConfigGetter::get()` in `getter.rs` lines 130–147 checks whether the caller is `FullAccess` or `FullRead`. If not, it does not return an error — it proceeds to call `self.formatter.format(config)` and returns a response.

**Formatter does not redact UUIDs:**

`ConfigConfidentialityFormatter::format()` in `confidentiality_formatting.rs` lines 17–28 sets `is_redacted = true` and nulls `description` and `rule_raw` for rules where `disclosed_at.is_none()`. It does not touch `rule_id` or `incident_id`. The `OutputRule` struct in `api/src/lib.rs` lines 109–115 defines `rule_id: RuleId` and `incident_id: IncidentId` as non-optional fields, so they are always serialized in the response.

**Test confirms the leak:**

The unit test in `getter.rs` lines 358–384 explicitly asserts that a `RestrictedRead` caller receives an `OutputRule` with non-null `rule_id` and `incident_id` for a rule where `disclosed_at` is `None`.

## Impact Explanation
An anonymous caller invoking `get_config` as a non-replicated query receives a `ConfigResponse` containing, for every non-disclosed confidential rule: the `rule_id` UUID and the `incident_id` UUID. With the `incident_id`, the caller can further invoke `get_rules_by_incident_id` (also `#[query]`, also bypasses `inspect_message`) to enumerate all rule UUIDs linked to that incident. This reveals that a security incident is actively being rate-limited, the exact incident UUID, and the count and IDs of all associated rules — directly violating the stated confidentiality invariant of the rate-limit canister. This constitutes a significant boundary/API security impact: the confidentiality model protecting active security incident metadata is fully bypassed, matching the High impact category for boundary node security impact with concrete protocol harm.

## Likelihood Explanation
The exploit requires no privileges, no keys, no special tooling, and no protocol-level attack. Any caller with access to the ICP query interface can execute it in a single unauthenticated call using `dfx` or any ICP agent. `get_config` is publicly listed in the Candid interface as a `query` method, making it trivially discoverable. The attack is repeatable at will with zero cost.

## Recommendation
1. **In `get_config` handler** (`canister.rs`): Enforce authorization inside the handler itself. Resolve the access level and return an authorization error (or an empty response) for `RestrictedRead` callers before invoking `ConfigGetter::get()`. Do not rely solely on `inspect_message` for access control on any `#[query]` method.
2. **In `ConfigConfidentialityFormatter::format()`** (`confidentiality_formatting.rs`): For non-disclosed rules, also omit the rule from the response entirely (or null out `rule_id` and `incident_id`) when the caller is `RestrictedRead`, so that even if the handler proceeds, no confidential UUID is exposed.
3. **Architectural note**: The constant `REPLICATED_QUERY_METHOD` and the `inspect_message` logic suggest the developer intended `get_config` to be a replicated query (update call). If replicated execution is required, `get_config` should be changed to `#[update]` or `#[composite_query]` with in-handler authorization, not `#[query]` with `inspect_message`.
