### Title
Truncating Floor Division in `native_per_pubdata` Calculation Causes Systematic Undercharging for Pubdata - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` is computed using floor (truncating) division via `wrapping_div`, while the analogous `native_per_gas` uses `div_ceil`. When `pubdata_price < native_price`, the result truncates to **zero**, meaning users pay no native resources for any pubdata they generate. Even when `pubdata_price >= native_price` but is not an exact multiple, users are systematically undercharged. This causes the operator to bear pubdata L1 costs without full compensation.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (ZK transaction flow), the `native_per_pubdata` ratio is computed as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

The same truncating division appears in the off-chain helper:

```rust
// api/src/helpers.rs:427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

By contrast, `native_per_gas` explicitly uses ceiling division to protect the protocol:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:135
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [3](#0-2) 

The `native_per_pubdata` value flows directly into pubdata charging:

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs:430-431
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [4](#0-3) 

And into the `deltaGas` adjustment that determines how much ETH the user ultimately pays:

```rust
// basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs:72
(native_used / native_per_gas) as i64 - (gas_used as i64)
``` [5](#0-4) 

The documentation confirms the intended formula is `nativePerPubdata := pubdata_price / native_price`, but does not specify rounding direction: [6](#0-5) 

---

### Impact Explanation

**Extreme case (`pubdata_price < native_price`):** `native_per_pubdata = 0`. Every transaction pays **zero** native resources for pubdata, regardless of how many storage writes it performs. The `deltaGas` adjustment never accounts for pubdata costs, so `gas_used` (and thus the ETH fee) does not reflect pubdata consumption. The operator publishes pubdata to L1 at their own cost with no compensation.

**General case (`pubdata_price mod native_price != 0`):** Users are undercharged by `(pubdata_price mod native_price) / native_price` native units per pubdata byte. For a transaction generating `N` pubdata bytes, the operator loses `N * (pubdata_price mod native_price)` wei in uncompensated pubdata costs.

Concrete example with `pubdata_price = 99`, `native_price = 100`, 1000 pubdata bytes:
- `native_per_pubdata = 0` (truncated from 0.99)
- Native charged for pubdata: 0 instead of 990
- `deltaGas` is 99 units smaller than correct
- User underpays by `99 * gas_price` wei; operator absorbs the difference

---

### Likelihood Explanation

The operator sets `pubdata_price` and `native_price` independently. These represent different physical quantities (wei/pubdata-byte vs. wei/native-cycle), so `pubdata_price < native_price` is a realistic configuration. Any unprivileged user submitting a transaction that writes to storage (generating pubdata) automatically benefits from the truncation without any special action. The effect is amplified by transactions that generate large amounts of pubdata (e.g., bulk storage writes).

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before:
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After:
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427 to keep off-chain validation consistent with on-chain behavior.

---

### Proof of Concept

1. Deploy ZKsync OS with `native_price = 100` and `pubdata_price = 99` (or any value where `pubdata_price < native_price`).
2. Submit a transaction that performs 20 SSTORE operations (generating ~640 bytes of pubdata).
3. Observe: `native_per_pubdata = floor(99/100) = 0`.
4. The transaction's native resource consumption does not include any pubdata cost.
5. `deltaGas = native_used / native_per_gas - gas_used` does not include pubdata contribution.
6. The user's `gas_used` (and ETH fee) is computed as if no pubdata was generated.
7. The operator publishes ~640 bytes to L1 at a cost of `640 * 99 = 63,360 wei` with zero compensation from the user's fee.

First we define the ratio between EVM gas and native resource as:
  `nativePerGas := gasPrice/nativePrice`
Note: for call simulation we use a constant for it, as gasPrice might be set to 0.

Next we define the limit for the native resource as:
  `nativeLimit := gasLimit * nativePerGas`

Then we process the transaction, charging both Ergs for EE execution and native resource for any kind of computation (EE, bootloader or system work).

If execution doesn't run out of native resources, we first charge for pubdata from native resource.
Then we compute the difference between the implicit gas used derived from native resource consumption and the gas used by EEs from the ergs used. We call this value `deltaGas`.
  `deltaGas := (nativeUsed / nativePerGas) - gasUsed`
