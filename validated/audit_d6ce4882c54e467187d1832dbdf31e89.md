### Title
`withdraw_erc20` Does Not Accept a User-Specified Maximum ckETH Fee, Enabling Unbounded Burns from Pre-Approved Allowance - (File: `rs/ethereum/cketh/minter/src/main.rs`)

---

### Summary

The `withdraw_erc20` endpoint in the ckETH minter does not allow callers to specify a maximum acceptable ckETH transaction fee. The fee is unilaterally determined by the minter via `estimate_erc20_transaction_fee()` at call time, which fetches the current Ethereum gas fee estimate. The official documentation instructs users to pre-approve a "LARGE_AMOUNT" of ckETH to the minter. Because no `max_transaction_fee` bound is accepted in the call arguments, the minter can burn significantly more ckETH than the user anticipated if gas prices spike between the user's fee check and the actual execution.

---

### Finding Description

The `WithdrawErc20Arg` struct accepted by `withdraw_erc20` contains only:

- `amount` – the ERC-20 withdrawal amount
- `ckerc20_ledger_id` – the token ledger ID
- `recipient` – the Ethereum destination address
- `from_cketh_subaccount` / `from_ckerc20_subaccount` – optional subaccounts

There is no `max_transaction_fee` field. The ckETH fee to pay for the Ethereum transaction is determined entirely by the minter:

```rust
// rs/ethereum/cketh/minter/src/main.rs:430
let erc20_tx_fee = estimate_erc20_transaction_fee().await.ok_or_else(|| {
    WithdrawErc20Error::TemporarilyUnavailable("Failed to retrieve current gas fee".to_string())
})?;
```

`estimate_erc20_transaction_fee()` calls `lazy_refresh_gas_fee_estimate()`, which caches the Ethereum gas fee estimate for up to 60 seconds and refreshes it from EVM RPC providers when stale:

```rust
// rs/ethereum/cketh/minter/src/tx.rs:610-681
pub async fn lazy_refresh_gas_fee_estimate() -> Option<GasFeeEstimate> {
    const MAX_AGE_NS: u64 = 60_000_000_000_u64; //60 seconds
    ...
    match read_state(|s| s.last_transaction_price_estimate.clone()) {
        Some((last_estimate_timestamp_ns, estimate))
            if now_ns < last_estimate_timestamp_ns.saturating_add(MAX_AGE_NS) =>
        { Some(estimate) }
        _ => do_refresh().await,
    }
}
```

The computed `erc20_tx_fee` is then burned from the user's ckETH account via `icrc2_transfer_from` (the `burn_from` call at line 449), and stored as `max_transaction_fee` in the `Erc20WithdrawalRequest`:

```rust
// rs/ethereum/cketh/minter/src/main.rs:480-481
let withdrawal_request = Erc20WithdrawalRequest {
    max_transaction_fee: erc20_tx_fee,
    ...
```

The official documentation (`rs/ethereum/cketh/docs/ckerc20.adoc`, step 1) explicitly instructs users to approve a "LARGE_AMOUNT" of ckETH to the minter (e.g., 1 ETH) to allow multiple withdrawals without re-approving. This large standing allowance is the precondition that makes the missing bound impactful: the minter has permission to burn far more than the user expects.

---

### Impact Explanation

A user who follows the documented workflow (approve a large ckETH allowance, then call `withdraw_erc20`) has no mechanism to cap the ckETH burned as fees. If Ethereum gas prices spike between when the user last queried `eip_1559_transaction_price` and when `withdraw_erc20` executes, the minter will burn the higher fee from the user's account. The user cannot reject the call based on an unacceptable fee because no such parameter exists in the interface. The financial loss is bounded only by the user's pre-approved allowance, which the documentation encourages to be large.

---

### Likelihood Explanation

Ethereum gas prices are volatile and can spike 5–10× within minutes during periods of high network activity. The 60-second cache window in `lazy_refresh_gas_fee_estimate()` means the fee used may reflect conditions up to 60 seconds old, or a freshly fetched spike. Users who follow the documentation's advice to approve a large ckETH allowance are directly exposed. The entry path is reachable by any non-anonymous principal with ckETH and ckERC-20 balances.

---

### Recommendation

Add a `max_transaction_fee : opt nat` field to `WithdrawErc20Arg`. Before burning ckETH, the minter should compare `erc20_tx_fee` against the caller-supplied maximum and return a new error variant (e.g., `TransactionFeeExceeded { max_fee, actual_fee }`) if the estimate exceeds it. This mirrors the protection already present for the `CkErc20` path in `create_transaction`, where `actual_min_max_fee_per_gas > request_max_fee_per_gas` causes an `InsufficientTransactionFee` error rather than silently proceeding.

---

### Proof of Concept

1. User queries `eip_1559_transaction_price` (query call, no cost) and observes a fee of **X ckETH**.
2. User calls `icrc2_approve` on the ckETH ledger, approving the minter for **10X ckETH** (a "large amount" per documentation).
3. Ethereum gas prices spike 6× before the user's `withdraw_erc20` ingress message is executed.
4. `lazy_refresh_gas_fee_estimate()` fetches the new high gas price; `estimate_erc20_transaction_fee()` returns **6X ckETH**.
5. The minter calls `burn_from(cketh_account, 6X, ...)` — this succeeds because the allowance is 10X.
6. The user's `withdraw_erc20` call returns `Ok(...)`, but **6X ckETH** was burned instead of the expected **X ckETH**.
7. The user has no recourse; the `max_transaction_fee` field in `Erc20WithdrawalRequest` is set to **6X** and overcharged fees are explicitly documented as non-reimbursable.

pub async fn lazy_refresh_gas_fee_estimate() -> Option<GasFeeEstimate> {
    const MAX_AGE_NS: u64 = 60_000_000_000_u64; //60 seconds

    async fn do_refresh() -> Option<GasFeeEstimate> {
        let _guard = match TimerGuard::new(TaskType::RefreshGasFeeEstimate) {
            Ok(guard) => guard,
            Err(e) => {
                log!(
                    DEBUG,
                    "[refresh_gas_fee_estimate]: Failed retrieving guard: {e:?}",
                );
                return None;
            }
        };

        let fee_history = match eth_fee_history().await {
            Ok(fee_history) => fee_history,
            Err(e) => {
                log!(
                    INFO,
                    "[refresh_gas_fee_estimate]: Failed retrieving fee history: {e:?}",
                );
                return None;
            }
        };

        let gas_fee_estimate = match estimate_transaction_fee(&fee_history) {
            Ok(estimate) => {
                mutate_state(|s| {
                    s.last_transaction_price_estimate =
                        Some((ic_cdk::api::time(), estimate.clone()));
                });
                estimate
            }
            Err(e) => {
                log!(
                    INFO,
                    "[refresh_gas_fee_estimate]: Failed estimating gas fee: {e:?}",
                );
                return None;
            }
        };
        log!(
            INFO,
            "[refresh_gas_fee_estimate]: Estimated transaction fee: {:?}",
            gas_fee_estimate,
        );
        Some(gas_fee_estimate)
    }

    async fn eth_fee_history() -> Result<FeeHistory, MultiCallError<FeeHistory>> {
        read_state(rpc_client)
            .fee_history((5_u8, BlockTag::Latest))
            .with_reward_percentiles(vec![20])
            .with_cycles(MIN_ATTACHED_CYCLES)
            .try_send()
            .await
            .reduce_with_strategy(StrictMajorityByKey::new(|fee_history: &FeeHistory| {
                Nat::from(fee_history.oldest_block.clone())
            }))
    }

    let now_ns = ic_cdk::api::time();
    match read_state(|s| s.last_transaction_price_estimate.clone()) {
        Some((last_estimate_timestamp_ns, estimate))
            if now_ns < last_estimate_timestamp_ns.saturating_add(MAX_AGE_NS) =>
        {
            Some(estimate)
        }
        _ => do_refresh().await,
    }
}
