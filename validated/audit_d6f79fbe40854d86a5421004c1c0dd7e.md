Audit Report

## Title
Deleted Dapp Canister Permanently Blocks SNS Swap Finalization via Panicking Pre-flight Check - (`rs/sns/root/src/lib.rs`)

## Summary
`SnsRootCanister::set_dapp_controllers` unconditionally panics if `canister_status` fails for any registered dapp canister in its pre-flight loop. Because the swap canister calls this function with `canister_ids: None` (expanding to all registered dapps) during finalization, a single deleted dapp canister permanently blocks SNS swap finalization. No force-removal path exists for dapp canisters, unlike the analogous `clean_up_failed_register_extension` for extensions.

## Finding Description
In `rs/sns/root/src/lib.rs`, the pre-flight loop in `set_dapp_controllers` (lines 796–826) iterates over every target dapp canister and calls `management_canister_client.canister_status(...)`. On error, it unconditionally panics:

```rust
Err(_) => {
    // TODO(NNS1-1993): Remove this panic and return an error type instead.
    panic!(
        "Could not get the status of canister: {dapp_canister_id}.  Root may not be a controller."
    )
}
``` [1](#0-0) 

A second `assert!` also panics if SNS Root is no longer listed as a controller of the canister. [2](#0-1) 

During SNS swap finalization, `take_sole_control_of_dapp_controllers` in `rs/sns/swap/src/swap.rs` calls `set_dapp_controllers` with `canister_ids: None`: [3](#0-2) 

The `None` branch resolves to the full `dapp_canister_ids` list (only the swap canister is authorized to do this): [4](#0-3) 

The only governance path to remove a dapp canister is `perform_deregister_dapp_canisters` in `rs/sns/governance/src/governance.rs`, which encodes a `SetDappControllersRequest` and calls `set_dapp_controllers` on Root — the same panicking function: [5](#0-4) 

A force-removal function (`clean_up_failed_register_extension`) exists for extensions but has no dapp-canister equivalent: [6](#0-5) 

The TODO comments explicitly acknowledge this defect under NNS1-1993 but the fix has not been applied. [7](#0-6) 

## Impact Explanation
This is a **High** severity SNS governance/swap infrastructure impact. A single deleted dapp canister permanently freezes SNS swap finalization: token sale participants cannot receive their tokens, the SNS cannot complete decentralization, and `DeregisterDappCanisters` proposals always fail for the stuck canister. The SNS community has no on-chain recovery path without NNS-level intervention. This matches: *"High ($2,000–$10,000). Significant SNS infrastructure security impact with concrete user or protocol harm."*

## Likelihood Explanation
No attacker privilege is required. Dapp canisters registered with an SNS can run out of cycles through ordinary operation; the IC automatically deletes them when their cycle balance reaches zero. A dapp developer who controls the canister's code can also deliberately drain cycles before or after registration. The registration cap of 100 dapp+extension canisters means a single deleted canister can occupy a slot and block finalization indefinitely. [8](#0-7) 

## Recommendation
1. Replace the two `panic!`/`assert!` calls in the pre-flight loop of `set_dapp_controllers` with graceful error returns, as the existing TODO NNS1-1993 already prescribes.
2. Add a `clean_up_failed_register_dapp` function for dapp canisters analogous to `clean_up_failed_register_extension`, allowing SNS Governance to evict a canister from `dapp_canister_ids` without requiring a successful `canister_status` call on the target.

## Proof of Concept
1. Create an SNS and register dapp canister `D` via `RegisterDappCanisters`. SNS Root becomes sole controller of `D`.
2. Drain `D`'s cycles to zero; the IC deletes `D`.
3. Submit a `DeregisterDappCanisters` proposal listing `D`. SNS Governance calls `Root::set_dapp_controllers({canister_ids: Some([D]), ...})`. The pre-flight loop calls `canister_status(D)`, receives an error, and panics. `D` remains in `dapp_canister_ids`.
4. The SNS swap canister later calls `Root::set_dapp_controllers({canister_ids: None, controller_principal_ids: [sns_root]})` during finalization. The pre-flight loop iterates all registered dapps, hits `D`, panics, and swap finalization is permanently blocked.
5. A deterministic integration test using PocketIC can reproduce this by: registering a canister, deleting it via the management canister, then calling `set_dapp_controllers` and asserting the trap occurs.

    pub async fn clean_up_failed_register_extension(
        self_ref: &'static LocalKey<RefCell<Self>>,
        management_canister_client: &impl ManagementCanisterClient,
        request: CleanUpFailedRegisterExtensionRequest,
    ) -> CleanUpFailedRegisterExtensionResponse {
        let main = async || -> Result<(), CanisterCallError> {
            // Unpack request.
            let CleanUpFailedRegisterExtensionRequest {
                canister_id: extension_canister_id,
            } = request;
            let Some(extension_canister_id) = extension_canister_id else {
                return Err(CanisterCallError {
                    code: Some(RejectCode::CanisterReject as i32),
                    description: "Request lacks canister_id.".to_string(),
                });
            };

            // Remove extension_canister_id from self.extensions.
            //
            // (This might result in no actual changes. In that case, we still
            // power through with the rest of this method.)
            self_ref.with_borrow_mut(|state| {
                let Some(extensions) = state.extensions.as_mut() else {
                    return;
                };

                extensions
                    .extension_canister_ids
                    .retain(|prior_extension_canister_id| {
                        prior_extension_canister_id != &extension_canister_id
                    });
            });

            // Prepare to call stop_canister and delete_canister (by wrapping
            // extension_canister_id in a couple extra layers).
            let extension_canister_id =
                CanisterIdRecord::from(CanisterId::unchecked_from_principal(extension_canister_id));

            // Prepare to delete the canister by stopping it first.
            management_canister_client
                .stop_canister(extension_canister_id)
                .await?;

            // Delete the canister.
            management_canister_client
                .delete_canister(extension_canister_id)
                .await?;

            Ok(())
        };
