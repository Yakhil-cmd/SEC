### Title
When Contract Is Paused, Users Cannot Withdraw Funds or Complete Pending Exit Flows - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

When the Aurora Engine contract is paused via `pause_contract`, the `require_running` guard blocks all user-facing fund-exit paths: `withdraw` (ETH to Ethereum), `ft_transfer`, `ft_transfer_call`, `submit`/`call` (which includes EVM-level `ExitToNear`/`ExitToEthereum` precompile calls), and critically `exit_to_near_precompile_callback`. This means any user funds in Aurora are temporarily frozen for the duration of the pause, and any exit flow that was already in-flight (EVM transaction submitted, callback not yet executed) will fail irrecoverably when the callback fires against a paused contract.

---

### Finding Description

The `EngineState` struct tracks a single boolean `is_paused`: [1](#0-0) 

`pause_contract` sets this flag to `true`: [2](#0-1) 

The helper `require_running` returns `ERR_PAUSED` whenever `is_paused` is `true`: [3](#0-2) 

Every user-facing fund-exit path calls this guard unconditionally:

**`withdraw`** (ETH → Ethereum bridge exit): [4](#0-3) 

**`ft_transfer` / `ft_transfer_call`** (NEP-141 token transfers): [5](#0-4) [6](#0-5) 

**`exit_to_near_precompile_callback`** (completes the ExitToNear precompile flow): [7](#0-6) 

**`submit` / `call`** (all EVM transactions, including those invoking `ExitToNear`/`ExitToEthereum` precompiles): [8](#0-7) [9](#0-8) 

The `exit_to_near_precompile_callback` case is the most severe: a user submits an EVM transaction that calls the `ExitToNear` precompile (moving tokens out of Aurora to NEAR). The EVM transaction succeeds and schedules the NEAR callback. If the contract is paused before that callback executes, the callback hits `require_running` and panics. The refund branch inside the callback (which would return tokens to the user on failure) is also unreachable because the function aborts before reaching it: [10](#0-9) 

The tokens are debited from the EVM state by the precompile but the NEAR-side transfer never completes and the refund path is never reached, leaving the user's tokens in limbo.

---

### Impact Explanation

**High — Temporary freezing of funds.**

- All ETH held in Aurora is unwithdrawable via `withdraw` for the entire pause duration.
- All ERC-20/NEP-141 bridged tokens are non-transferable via `ft_transfer`/`ft_transfer_call`.
- Any in-flight `ExitToNear` flow whose callback fires while the contract is paused will fail without refunding the user, causing a loss of those tokens until the contract is resumed and the state is manually reconciled (if at all).
- `submit`/`call` being blocked means users cannot even attempt to use the `ExitToEthereum` precompile to recover ETH.

---

### Likelihood Explanation

**Medium.** The `pause_contract` function is an explicitly supported emergency mechanism added in v2.10.0. Its intended use is emergency response. Emergency pauses on live DeFi infrastructure are not rare events. The window between an EVM transaction invoking `ExitToNear` and its NEAR callback executing is a normal asynchronous gap in NEAR's execution model, making the in-flight race condition a realistic scenario whenever a pause is triggered during high activity.

---

### Recommendation

1. Remove `require_running` from `withdraw`, `ft_transfer`, `ft_transfer_call`, and `exit_to_near_precompile_callback`. These are fund-exit paths and should remain operational during a pause so users can always recover their assets.
2. Keep `require_running` on `submit`, `call`, `deploy_code`, and `ft_on_transfer` (deposit paths) — blocking new deposits and arbitrary EVM execution during an emergency is reasonable.
3. For `exit_to_near_precompile_callback` specifically, the refund branch must remain reachable regardless of pause state, since it is a recovery path for a previously committed EVM state change.

---

### Proof of Concept

1. User calls `submit` with an EVM transaction that invokes the `ExitToNear` precompile, transferring 1 ETH worth of tokens out of Aurora. The EVM deducts the balance and schedules the NEAR callback `exit_to_near_precompile_callback`.
2. Before the NEAR callback receipt is processed, the Aurora owner calls `pause_contract` (e.g., in response to an unrelated exploit).
3. The NEAR runtime delivers the `exit_to_near_precompile_callback` receipt. The function immediately hits `require_running(&state)?` at line 203 of `connector.rs` and returns `ERR_PAUSED`, causing the NEAR transaction to fail.
4. The refund logic at lines 231–239 of `connector.rs` is never reached. The user's EVM balance was already decremented by the precompile; no NEAR transfer occurred; no refund is issued.
5. The user's funds are frozen until the owner resumes the contract and manually investigates the inconsistency — with no on-chain mechanism to trigger the refund after resumption.

    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
