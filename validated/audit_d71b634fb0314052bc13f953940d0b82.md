### Title
Failed `ExitToNear` Precompile Calls Permanently Destroy User Funds When `error_refund` Feature Is Disabled — (File: `engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

When the `error_refund` Cargo feature is absent from the production binary, the `ExitToNear` precompile sets `refund: None` unconditionally and, for base-token (ETH) and ERC-20 exits, skips scheduling a callback entirely. If the downstream `ft_transfer` / `ft_transfer_call` promise fails, no EVM credit is restored. The EVM balance is permanently reduced (ETH stranded at the precompile address; ERC-20 tokens permanently burned), while the NEP-141 tokens remain in the Aurora contract. The tests in the repository explicitly confirm and accept this outcome when the feature is off.

---

### Finding Description

**Step 1 — Feature gate sets `refund: None`**

In `engine-precompiles/src/native.rs` lines 449–455, the `callback_args` struct is built with a compile-time branch:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None without the feature
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

**Step 2 — No callback is scheduled for ETH/ERC-20 exits**

For base-token (ETH) exits and ERC-20 exits (non-wNEAR), `transfer_near` is also `None`. The struct therefore equals `ExitToNearPrecompileCallbackArgs::default()`, and the promise degrades to a bare `PromiseArgs::Create` with no callback attached:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback registered
} else {
    PromiseArgs::Callback(...)
};
``` [2](#0-1) 

**Step 3 — Even when a callback fires (wNEAR path), `refund: None` silently no-ops**

In `engine/src/contract_methods/connector.rs` lines 231–242, the failure branch checks `args.refund`:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(...)?;
    ...
} else {
    None   // ← silent no-op; no refund issued
};
``` [3](#0-2) 

**Step 4 — `error_refund` is not a default feature**

`engine/Cargo.toml` declares:

```toml
[features]
default = ["std"]
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [4](#0-3) 

The workspace root also pins `aurora-engine` with `default-features = false`:

```toml
aurora-engine = { path = "engine", default-features = false }
``` [5](#0-4) 

A production binary built without explicitly passing `--features error_refund` will have the refund path compiled out entirely.

---

### Impact Explanation

- **ETH exit path**: The user's ETH is moved to the `exit_to_near` precompile address inside the EVM before the NEAR-side `ft_transfer` fires. If `ft_transfer` fails and no callback is scheduled, the ETH is stranded at the precompile address with no recovery path. The user's EVM balance is permanently reduced.
- **ERC-20 exit path**: The ERC-20 tokens are burned before the `ft_transfer` fires. If `ft_transfer` fails and no callback is scheduled, the tokens are permanently destroyed. The NEP-141 balance in the Aurora contract is unchanged (Aurora retains the NEP-141 tokens), while the user's ERC-20 balance is permanently zeroed.

Both cases constitute **permanent, unrecoverable loss of user funds** — classified as **theft of unclaimed yield (High)**.

---

### Likelihood Explanation

- `error_refund` is not in the default feature set; any build that does not explicitly enable it silently disables the refund path.
- The failure trigger (`ft_transfer` to an unregistered NEAR account) is a normal, user-reachable condition — no privileged access or external compromise is required.
- The tests in the repository explicitly encode the broken behavior as the expected outcome when the feature is off, confirming the path is reachable and the loss is real. [6](#0-5) [7](#0-6) 

---

### Recommendation

1. **Promote `error_refund` to a default feature** in `engine/Cargo.toml` and `engine-precompiles/Cargo.toml`, or remove the feature gate entirely and always compile the refund path.
2. If the feature gate must remain for binary-size reasons, add a compile-time `#[cfg]` assertion or a CI check that rejects production builds compiled without `error_refund`.
3. Remove the `#[cfg(not(feature = "error_refund"))] refund: None` branch; the invariant "a failed exit must restore the sender's EVM balance" should be unconditional.
