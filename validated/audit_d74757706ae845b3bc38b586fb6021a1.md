Based on my investigation of the Sei chain codebase, I found a directly analogous vulnerability in the CosmWasm module.

### Title
CosmWasm Contract Instantiation DoS via Pre-funded Deterministic Address - (`File: sei-wasmd/x/wasm/keeper/keeper.go`)

### Summary
An attacker can permanently block any CosmWasm contract instantiation by pre-funding the deterministic contract address before the `MsgInstantiateContract` transaction is processed. The `instantiate` function checks `existingAcct != nil` and returns `ErrAccountExists` if any account (including a plain `BaseAccount` created by a bank send) already exists at the predicted address. Because the address is computed from a publicly readable, monotonically incrementing counter, the next contract address is trivially predictable.

### Finding Description

`generateContractAddress` in `sei-wasmd/x/wasm/keeper/keeper.go` reads the current `KeyLastInstanceID` counter and calls `BuildContractAddress(codeID, instanceID)`:

```go
// generates a contract address from codeID + instanceID
func (k Keeper) generateContractAddress(ctx sdk.Context, codeID uint64) sdk.AccAddress {
    instanceID := k.autoIncrementID(ctx, types.KeyLastInstanceID)
    return BuildContractAddress(codeID, instanceID)
}

func BuildContractAddress(codeID, instanceID uint64) sdk.AccAddress {
    contractID := make([]byte, 16)
    binary.BigEndian.PutUint64(contractID[:8], codeID)
    binary.BigEndian.PutUint64(contractID[8:], instanceID)
    return address.Module(types.ModuleName, contractID)[:types.ContractAddrLen]
}
``` [1](#0-0) 

The current counter value is publicly readable via `PeekAutoIncrementID` or directly from chain state under `KeyLastInstanceID`: [2](#0-1) 

The `instantiate` function then checks for a pre-existing account **before** any deposit logic:

```go
contractAddress := k.generateContractAddress(ctx, codeID)
existingAcct := k.accountKeeper.GetAccount(ctx, contractAddress)
if existingAcct != nil {
    return nil, nil, sdkerrors.Wrap(types.ErrAccountExists, existingAcct.GetAddress().String())
}
``` [3](#0-2) 

The standard Cosmos SDK `bank.SendCoins` (used by `MsgSend`) creates a `BaseAccount` at any address that receives funds if no account exists there:

```go
accExists := k.ak.HasAccount(ctx, toAddr)
if !accExists {
    k.ak.SetAccount(ctx, k.ak.NewAccountWithAddress(ctx, toAddr))
}
``` [4](#0-3) 

**Attack path:**
1. Attacker queries `KeyLastInstanceID` from chain state (public, no permission required).
2. Attacker computes `BuildContractAddress(targetCodeID, currentInstanceID)` — the address of the next contract to be instantiated.
3. Attacker submits a `MsgSend` of 1 usei to that address. The bank module creates a `BaseAccount` there.
4. When the legitimate `MsgInstantiateContract` is processed, `GetAccount` returns the pre-created `BaseAccount`, `existingAcct != nil` is true, and the call returns `ErrAccountExists`.
5. The instantiation is permanently blocked for that `instanceID`. The attacker can repeat for `instanceID+1`, `instanceID+2`, etc., to block all future instantiations.

### Impact Explanation

Any CosmWasm contract instantiation on Sei can be permanently blocked by an unprivileged attacker with minimal cost (1 usei per blocked slot). This is a targeted resource DoS against contract deployment. Critical protocol deployments (DeFi, bridges, governance contracts) can be griefed indefinitely. The attacker's cost is negligible; the victim must wait for a governance/admin intervention or a code fix.

This maps to **Medium** under the Sei bounty scope: resource DoS / unintended contract execution from network bug.

### Likelihood Explanation

- The `instanceID` counter is publicly readable from chain state at any time.
- `BuildContractAddress` is a pure deterministic function — no secret inputs.
- A `MsgSend` to any address is always valid and costs only gas + 1 usei.
- The attacker can pre-fund many consecutive addresses to guarantee success even if other instantiations occur in between.
- No privileged access, validator key, or governance role is required.

### Recommendation

Before the `existingAcct != nil` check, distinguish between a plain `BaseAccount` (created by a bank send) and a proper contract account. Specifically:

1. **Ignore `BaseAccount` with no code**: Only treat an existing account as a conflict if it has contract info stored (i.e., `k.HasContractInfo(ctx, contractAddress)`). A bare `BaseAccount` with no wasm state should not block instantiation.
2. **Alternatively**: Sweep any pre-existing balance to the creator before proceeding, and delete the bare `BaseAccount` if it has no code or nonce, then continue with instantiation.
3. **Or**: Use a nonce/salt in address derivation that is not predictable from public state alone.

### Proof of Concept

```
1. Query: GET /cosmwasm/wasm/v1/contract/sequence  → instanceID = N
2. Compute: addr = BuildContractAddress(codeID=1, instanceID=N)
           → e.g. "sei14hj2tavq8fpesdwxxcu44rty3hh90vhujrvcmstl4zr3txmfvw9sh9m79m"
3. Submit:  MsgSend { from: attacker, to: addr, amount: "1usei" }
           → bank creates BaseAccount at addr
4. Submit:  MsgInstantiateContract { codeID: 1, ... }
           → instantiate() calls GetAccount(addr) → non-nil BaseAccount
           → returns ErrAccountExists
           → instantiation fails permanently for this instanceID
``` [5](#0-4) [6](#0-5)

// BuildContractAddress builds an sdk account address for a contract.
func BuildContractAddress(codeID, instanceID uint64) sdk.AccAddress {
	contractID := make([]byte, 16)
	binary.BigEndian.PutUint64(contractID[:8], codeID)
	binary.BigEndian.PutUint64(contractID[8:], instanceID)
	return address.Module(types.ModuleName, contractID)[:types.ContractAddrLen]
}

func (k Keeper) autoIncrementID(ctx sdk.Context, lastIDKey []byte) uint64 {
	store := ctx.KVStore(k.storeKey)
	bz := store.Get(lastIDKey)
	id := uint64(1)
	if bz != nil {
		id = binary.BigEndian.Uint64(bz)
	}
	bz = sdk.Uint64ToBigEndian(id + 1)
	store.Set(lastIDKey, bz)
	return id
}
