Audit Report

## Title
Unbounded `ingress_messages` deserialization in `TryFrom<pb::StrippedBlockProposal>` enables Byzantine peer to spawn unbounded Tokio tasks and exhaust replica resources — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs`)

## Summary

`TryFrom<pb::StrippedBlockProposal>` performs no upper-bound check on the count of `ingress_messages` entries before collecting them into a `Vec`. A Byzantine subnet peer can craft a `StrippedBlockProposal` with an arbitrarily large `ingress_messages` list while setting `unstripped_consensus_message_id` to a valid, currently-wanted block proposal hash. This causes `BlockProposalAssembler::new` to allocate a proportionally large `Vec`, and `assemble_message` to spawn one indefinitely-looping Tokio task per entry — up to tens of thousands — until the bouncer fires `Unwanted` after its 3-second refresh cycle. The attack is repeatable every block round and can target multiple replicas simultaneously, causing memory exhaustion and task-scheduler saturation sufficient to disrupt consensus participation.

## Finding Description

**Step 1 — No length bound in deserialization.**

`TryFrom<pb::StrippedBlockProposal>` in `stripped.rs` validates that `pruned_block_proposal` is present, that the pruned block's ingress payload is empty, and that `unstripped_consensus_message_id` is a `BlockProposal` hash. It does **not** check `ingress_messages.len()` before collecting:

```rust
stripped_ingress_payload: StrippedIngressPayload {
    ingress_messages: value
        .ingress_messages
        .into_iter()
        .map(SignedIngressId::try_from)
        .collect::<Result<Vec<_>, _>>()?,
},
``` [1](#0-0) 

The developers themselves noted the expected bound in a comment in `assembler.rs`: *"We can have at most 1000 elements in the vector, so it should be reasonably fast to do a linear scan here"* — but this invariant is never enforced at the deserialization boundary. [2](#0-1) 

**Step 2 — `BlockProposalAssembler::new` allocates proportionally.**

`BlockProposalAssembler::new` maps every `SignedIngressId` in the stripped payload into a `Vec<(SignedIngressId, Option<SignedIngress>)>` with no cap: [3](#0-2) 

**Step 3 — One Tokio task spawned per entry.**

`assemble_message` calls `assembler.missing_stripped_messages()` (which returns all N entries since none are in the local pool) and spawns a `get_or_fetch` task for each: [4](#0-3) 

**Step 4 — Each spawned task loops indefinitely.**

`download_stripped_message` uses `with_max_elapsed_time(None)`, meaning each of the N tasks runs forever until the `JoinSet` is dropped: [5](#0-4) 

The loop at line 332 retries with exponential backoff but never terminates on its own. [6](#0-5) 

**Step 5 — The ID check does not constrain `ingress_messages` content.**

`MaybeStrippedConsensusMessage::id()` derives the artifact ID solely from `unstripped_consensus_message_id`. A Byzantine peer sets this field to a valid, currently-wanted block proposal hash while inflating `ingress_messages` to an arbitrary count — the ID check in `FetchArtifact::download_artifact` passes. [7](#0-6) [8](#0-7) 

**Step 6 — Bouncer only checks artifact ID, not payload content.**

`BouncerFactoryWrapper` delegates to the consensus pool bouncer, which evaluates only the `ConsensusMessageId` (height + hash). It has no visibility into the number of ingress message IDs embedded in the stripped payload. [9](#0-8) 

**Step 7 — Bouncer termination window.**

The `JoinSet` is dropped (aborting all tasks) only when the bouncer fires `Unwanted`. The bouncer refresh period is 3 seconds. During this window — and for as long as the block proposal remains wanted — all N tasks are live, consuming memory and scheduler slots. The attack is repeatable every block round (~1 second), allowing continuous resource pressure. [10](#0-9) [11](#0-10) 

## Impact Explanation

A single Byzantine peer can trigger allocation of a `Vec` with up to ~55,000 entries (within a typical 4 MB QUIC message, given ~72 bytes per `pb::StrippedIngressMessage`) and spawn an equal number of Tokio tasks per targeted block proposal. Each task holds a stack frame and loops indefinitely. With multiple concurrent block proposals in flight (the bouncer allows up to `LOOK_AHEAD = 10` heights), the total task count multiplies. The attack is repeatable every block round and can target multiple replicas simultaneously. This matches the **High ($2,000–$10,000)** impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

The attacker needs only to be a subnet peer (below the Byzantine fault threshold) and to observe a block proposal ID that the victim is currently downloading — both are trivially achievable in normal subnet operation. No privileged access, key material, or majority corruption is required. The crafted message passes all existing validation gates (ID check, bouncer check) before the resource allocation occurs. The attack is low-cost and repeatable.

## Recommendation

Add an upper-bound check on `ingress_messages.len()` (and `stripped_idkg_dealings.len()`) inside `TryFrom<pb::StrippedBlockProposal>`, before the `collect()` call, rejecting with `ProxyDecodeError` if the count exceeds the protocol-enforced maximum (`max_ingress_messages_per_block`, default 1000, for ingress; `dkg_dealings_per_block` for IDKG dealings). This enforces the invariant already assumed by the comment at `assembler.rs:440`.

## Proof of Concept

```rust
// Craft a StrippedBlockProposal proto with N fake ingress_messages entries.
let mut proto = pb::StrippedBlockProposal {
    pruned_block_proposal: Some(pb::BlockProposal::default()),
    unstripped_consensus_message_id: Some(/* valid wanted block proposal ID */),
    ingress_messages: (0..55_000)
        .map(|i| pb::StrippedIngressMessage {
            stripped: Some(pb::IngressMessageId {
                expiry: 0,
                message_id: vec![i as u8; 32],
            }),
            ingress_bytes_hash: vec![0u8; 32],
        })
        .collect(),
    stripped_idkg_dealings: vec![],
};
// Serve this proto in response to the victim's RPC for the wanted block proposal ID.
// Observe: BlockProposalAssembler::new allocates a Vec of 55,000 entries,
// assemble_message spawns 55,000 get_or_fetch tasks, each looping indefinitely.
// Assert: memory usage and spawned task count are NOT bounded before the bouncer fires.
```

A deterministic integration test can verify this by: (1) constructing the above proto, (2) serving it from a mock transport, (3) calling `FetchStrippedConsensusArtifact::assemble_message` with a bouncer that stays `Wants` for the duration, and (4) asserting that the `JoinSet` size equals N before the bouncer fires `Unwanted`.
