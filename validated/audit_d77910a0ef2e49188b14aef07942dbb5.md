### Title
Pre-EIP-155 Unprotected Legacy Transactions Accepted Without Chain-ID Binding, Enabling Cross-Chain Replay Attacks - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

ZKsync OS accepts pre-EIP-155 legacy transactions (those with `v = 27` or `v = 28`) without any chain-ID check. Because the signing hash for such transactions is computed over only the six payload fields `[nonce, gasPrice, gasLimit, to, value, data]` — with no chain ID included — a signed transaction that was broadcast on Ethereum mainnet (or any other EVM chain) is cryptographically identical on ZKsync OS. An attacker who observes such a transaction on any public chain can replay it verbatim on ZKsync OS and drain the sender's balance, without knowing the sender's private key.

---

### Finding Description

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, the code branches on whether the signature is EIP-155-protected:

```rust
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    // Unprotected legacy
    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);
    hasher.finalize_reset().into()
} else {
    // EIP-155 protected: v must match 35 + 2*chainId (+ {0,1})
    ...
}
``` [1](#0-0) 

`is_eip155()` returns `false` when `v == 27 || v == 28`: [2](#0-1) 

When the unprotected branch is taken, the signing hash covers only the six-field payload — no chain ID is mixed in. The transaction is then classified as the `Legacy` variant (not `LegacyWithEIP155`) and returned successfully: [3](#0-2) 

There is no subsequent rejection of the `Legacy` variant anywhere in the parsing or validation pipeline. The full validation flow in `validate_and_compute_fee_for_transaction` proceeds normally — it calls `transaction.sig_parity_r_s()`, runs `ecrecover`, checks nonce, balance, etc. — but never checks whether the transaction is chain-ID-bound: [4](#0-3) 

By contrast, all typed transactions (EIP-2930, EIP-1559, EIP-4844, EIP-7702) explicitly reject a mismatched chain ID: [5](#0-4) 

The `docs/not-a-bug.md` file does not list acceptance of unprotected legacy transactions as intentional behavior. [6](#0-5) 

---

### Impact Explanation

An attacker who observes any pre-EIP-155 legacy transaction on Ethereum mainnet (or any other EVM-compatible chain) — identifiable by `v = 27` or `v = 28` — can submit the identical raw bytes to ZKsync OS. If the originating address holds a balance on ZKsync OS, the transaction will pass signature verification (the `ecrecover` output matches `from` because the signing hash is chain-agnostic), pass nonce and balance checks, and execute — transferring the `value` field to the `to` address. The attacker does not need the victim's private key. The impact is **direct loss of funds**.

---

### Likelihood Explanation

Pre-EIP-155 transactions are rare on modern Ethereum but are permanently visible on-chain (e.g., the very first Ethereum transaction, used in the Shardeum PoC, has `v = 28`). Any address that ever sent a pre-EIP-155 transaction on any EVM chain and later received funds on ZKsync OS is vulnerable. The attacker's cost is a single RPC call. No privileged access, oracle manipulation, or key compromise is required.

---

### Recommendation

Reject unprotected legacy transactions at the parsing stage. In `parse_and_compute_signed_hash`, after the `is_eip155()` check, return `Err(InvalidTransaction::InvalidChainId.into())` when the transaction is not EIP-155 protected:

```rust
let tx = if sig_data.is_eip155() {
    Self::LegacyWithEIP155(tx, sig_data)
} else {
    // Reject pre-EIP-155 unprotected transactions to prevent cross-chain replay
    return Err(InvalidTransaction::InvalidChainId.into());
};
``` [7](#0-6)

# Not a Bug

This page lists ZKsync OS behavior that is intentional, protocol-correct, or already documented, but is commonly misreported as a vulnerability.

Use this page as a triage aid, not as a substitute for checking the code. If an observed behavior differs from what is described here, investigate it normally.

## EVM / Cancun Behavior

### Cancun is the supported EVM hardfork

ZKsync OS currently targets Cancun EVM semantics. Reports that assume later hardfork behavior are not valid unless they identify an issue under the supported hardfork.

See [EVM Execution Environment](./execution_environments/evm.md).

### EIP-4844 transactions are disabled in production

EIP-4844 type `0x03` transactions are implemented behind the `basic_bootloader/eip-4844` feature, but this feature is not enabled by the production feature sets. It is enabled for test and Ethereum test-runner configurations.

See [Transaction formats](./bootloader/transaction_format.md).

### `BLOBHASH` returns `0`

This is expected in production. Since EIP-4844 transactions are disabled, transactions do not have blob versioned hashes. Cancun/EIP-4844 specifies that `BLOBHASH(index)` returns a zeroed `bytes32` when `index` is outside `tx.blob_versioned_hashes`.

### `BLOBBASEFEE` returns `1`

This is expected in production. There are no EIP-4844 transactions in production history, so `excess_blob_gas` remains `0`. Cancun/EIP-4844 defines the minimum blob base fee as `1`, and EIP-7516 makes `BLOBBASEFEE` return the current block's blob base fee.

### Opcode `0x44` returns `1`

Opcode `0x44` is `PREVRANDAO` under Cancun semantics, historically named `DIFFICULTY`. ZKsync OS uses the block `mix_hash` value for this opcode. In production, `mix_hash` is mocked to `1`.

This does not contradict the block header `difficulty` field being `0` after the Merge.

See [Bootloader block header](./bootloader/bootloader.md#block-header).

### `SUB` operand order

ZKsync OS follows the standard EVM operand order for `SUB`. The top stack item is the first operand, and the item below it is the second operand. For example, `PUSH1 0x03 PUSH1 0x05 SUB` leaves `0x02` on the stack.

This is not a reversed subtraction bug.
