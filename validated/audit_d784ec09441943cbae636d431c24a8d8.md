### Title
Unbounded DKG Unvalidated Pool Growth via Silent Deferral of Remote-Tagged Dealings — (`rs/consensus/dkg/src/lib.rs`)

### Summary

A Byzantine subnet node (below the fault threshold) can flood the DKG unvalidated pool with crafted messages bearing `NiDkgTargetSubnet::Remote(unknown_id)` at the current `start_block_height`. The validation path silently defers every such message without rejection or removal, and neither the pool nor the P2P slot table enforces a per-peer size limit for DKG artifacts. The pool accumulates unboundedly until the next DKG interval purge, enabling memory and CPU exhaustion.

### Finding Description

**Deferral without rejection — `validate_dealings_for_dealer`** [1](#0-0) 

When `configs.get(message_dkg_id)` returns `None` and `message_dkg_id.target_subnet.is_remote()` is true, the function returns `Mutations::new()` — an empty change set. The message is neither moved to validated nor removed from the unvalidated pool. It stays indefinitely.

**Admission without content filtering — `DkgBouncer`** [2](#0-1) 

The bouncer only compares `id.height` against `start_height`. A message at the current start height always receives `BouncerValue::Wants`, regardless of its `target_subnet`, `dealer_subnet`, or signature validity.

**No capacity limit on the unvalidated pool — `PoolSection`** [3](#0-2) 

`PoolSection` is a plain `BTreeMap<K, V>` with no maximum size. `DkgPoolImpl::insert` calls it unconditionally. [4](#0-3) 

**No per-peer slot limit for DKG — P2P setup** [5](#0-4) 

`SLOT_TABLE_LIMIT_INGRESS = 50_000` is applied only to ingress. DKG uses `SLOT_TABLE_NO_LIMIT = usize::MAX`, so the consensus manager receiver's slot-table guard at `receiver.rs:373` does not bound DKG artifact advertisements per peer.

**`DkgMessageId` keyed by content hash — no deduplication across distinct messages** [6](#0-5) 

Each message with a distinct `NiDkgDealing` payload produces a distinct `DkgMessageId`. An attacker can generate N distinct messages (varying the dealing bytes) that all map to different pool entries.

**O(N) processing on every `on_state_change` call** [7](#0-6) 

Every call to `on_state_change` iterates the entire unvalidated pool, groups by `(signer, dkg_id)`, and calls `validate_dealings_for_dealer` for each group. With N deferred remote messages, this is O(N) work per consensus round.

**Purge only at interval boundary** [8](#0-7) 

The purge fires only when `start_height > dkg_pool.get_current_start_height()`, i.e., at the next DKG summary block. With `DKG_INTERVAL_HEIGHT = 499` rounds, deferred messages accumulate for the entire interval. [9](#0-8) 

### Impact Explanation

A Byzantine subnet node sends N crafted `dkg::Message` artifacts with:
- `content.dkg_id.target_subnet = NiDkgTargetSubnet::Remote(unknown_id)` (any 32-byte ID not in state)
- `content.dkg_id.start_block_height = current_start_height`
- Distinct `dealing` payloads (varying bytes) to produce distinct hashes

Each message passes the bouncer (height matches), is inserted into the unvalidated pool, and is silently deferred by `validate_dealings_for_dealer`. The pool grows without bound. Effects:

1. **Memory exhaustion**: NiDkgDealing objects are large cryptographic structures. N messages consume O(N × dealing_size) bytes.
2. **CPU exhaustion**: `on_state_change` is called every consensus round and iterates all N deferred messages, causing O(N) CPU work per round, potentially starving legitimate local-DKG validation.
3. **DKG stall**: If the replica crashes or falls behind due to resource exhaustion, DKG transcripts are not produced, blocking subnet key generation and subnet creation.

### Likelihood Explanation

- **Attacker model**: A single Byzantine subnet node, explicitly within the IC fault-tolerance model (f < n/3). No governance majority, no key compromise, no external infrastructure attack required.
- **Exploit complexity**: Low. The attacker only needs to craft messages with a remote target ID not present in the replicated state. No valid signature or valid dealing content is required at the admission stage (the bouncer does not verify signatures).
- **Persistence**: The attack persists for the full DKG interval (up to 499 rounds) before purge.
- **Amplification**: Multiple Byzantine nodes (still below fault threshold) multiply the effect linearly.

### Recommendation

1. **Reject remote-tagged messages with unknown target IDs immediately** rather than deferring them. If no `SetupInitialDKG` context exists for the target ID, treat the message as invalid and emit `HandleInvalid`.
2. **Add a per-peer size limit to the DKG unvalidated pool**, analogous to `SLOT_TABLE_LIMIT_INGRESS` for ingress.
3. **Apply a slot-table limit for DKG** in `setup_ic_network` instead of `SLOT_TABLE_NO_LIMIT`.
4. **Verify the signature at admission** (in the bouncer or at insert time) to prevent unsigned garbage from entering the pool.

pub(crate) struct PoolSection<K, V> {
    messages: BTreeMap<K, V>,
    metrics: PoolMetrics,
}

impl<K: Ord, V: HasLabel> PoolSection<K, V> {
    pub(crate) fn new(metrics_registry: MetricsRegistry, pool: &str, pool_type: &str) -> Self {
        Self {
            messages: Default::default(),
            metrics: PoolMetrics::new(metrics_registry, pool, pool_type),
        }
    }

    pub(crate) fn insert(&mut self, key: K, value: V) -> Option<V> {
        self.metrics
            .observe_insert(MESSAGE_SIZE_BYTES, value.label());
        let replaced = self.messages.insert(key, value);
        if let Some(replaced) = &replaced {
            self.metrics
                .observe_duplicate(MESSAGE_SIZE_BYTES, replaced.label());
        }
        replaced
    }
