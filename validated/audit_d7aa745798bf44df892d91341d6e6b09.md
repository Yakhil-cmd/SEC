### Title
Missing Upper-Bound Constraint on `reward_coefficient_percent` in `do_update_node_rewards_table` - (File: rs/registry/canister/src/mutations/do_update_node_rewards_table.rs)

### Summary
The `do_update_node_rewards_table` function in the Registry canister accepts `NodeRewardRate` entries containing a `reward_coefficient_percent` field (`Optional<i32>`) without validating that the value is within the documented valid range [0, 100]. A value above 100 inverts the intended reward-decay logic for type3 nodes, causing exponentially *increasing* rewards per additional node instead of decreasing, inflating ICP node provider rewards paid from the treasury.

### Finding Description
The `NodeRewardRate` protobuf message defines `reward_coefficient_percent` as an `Optional<i32>` representing a percentage (0–100) that controls how rewards decay for each additional type3 node a provider operates:

> "A value of 100 means that the same reward is received for all nodes. A value of 0 means that only the first node gets the rewards, 2nd and later nodes get no reward." [1](#0-0) 

The `do_update_node_rewards_table` function applies the payload directly to the registry with **no validation** of `reward_coefficient_percent`: [2](#0-1) 

The `check_global_state_invariants` function in the Registry canister does not include any check for node rewards table field values — there is no `check_node_rewards_invariants` in the invariant chain: [3](#0-2) 

In the reward calculation, `reward_coefficient_percent` is consumed directly without any bounds guard: [4](#0-3) 

If `reward_coefficient_percent` is set to a value > 100 (e.g., 150), `dc_reward_coefficient_percent` becomes `1.5`, causing `np_coeff` to grow exponentially with each additional node. If set to a negative value (the field is `i32`, so negatives are representable), `dc_reward_coefficient_percent` becomes negative, causing oscillating (alternating-sign) reward accumulation.

### Impact Explanation
**High.** Wrong node provider reward calculations lead to inflated ICP rewards being minted and distributed from the ICP treasury. With a coefficient > 100, each additional type3 node would receive *more* rewards than the previous one, exponentially increasing total rewards paid out. This directly corrupts ICP token supply and node provider economics at the ledger level. [5](#0-4) 

### Likelihood Explanation
**Low.** Exploiting this requires passing an NNS `UpdateNodeRewardsTable` governance proposal with an out-of-range `reward_coefficient_percent` value — either through a malicious NNS governance majority or a significant error in a proposal. The missing validation means there is no safety net to catch such errors at the protocol level, analogous to the original report's "big error on their part" scenario. [2](#0-1) 

### Recommendation
Add validation in `do_update_node_rewards_table` to check that `reward_coefficient_percent`, when `Some`, is within the valid range `[0, 100]`. Since the field is `i32`, also reject negative values. Additionally, add a corresponding invariant check in `check_global_state_invariants` to enforce this constraint on the stored registry state. [6](#0-5) 

### Proof of Concept
1. Submit an NNS `UpdateNodeRewardsTable` proposal with a `NodeRewardRate` entry for a type3 node type where `reward_coefficient_percent = Some(150)`.
2. The proposal passes NNS governance.
3. `do_update_node_rewards_table` applies the mutation without any validation of the coefficient value.
4. When `get_node_providers_monthly_xdr_rewards` is called, the reward calculation computes `dc_reward_coefficient_percent = 150 / 100.0 = 1.5`.
5. For a node provider with N type3 nodes, each subsequent node receives 1.5× the previous node's reward, resulting in exponentially inflated total rewards minted to node providers. [4](#0-3) [7](#0-6)
