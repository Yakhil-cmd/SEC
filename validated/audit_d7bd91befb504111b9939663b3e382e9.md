### Title
Unbounded ingress-window loop in ICP Rosetta `/construction/payloads` allows unauthenticated OOM/CPU DoS — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

### Summary
The ICP Rosetta handler for `POST /construction/payloads` accepts attacker-controlled `ingress_start` and `ingress_end` nanosecond timestamps from the request metadata and uses them directly as the bounds of a `while now < ingress_end` loop with no cap on the window size. Sending `ingress_start=0` and `ingress_end=u64::MAX` causes ~154 million loop iterations that push `u64` values into a `Vec`, allocating over 1 GB of heap memory in a single synchronous request handler, crashing or hanging the Rosetta server process.

### Finding Description
In `construction_payloads()`:

```rust
let interval =
    ic_limits::MAX_INGRESS_TTL - ic_limits::PERMITTED_DRIFT - Duration::from_secs(120);
// interval = 120 seconds = 120_000_000_000 ns

let ingress_start = meta...ingress_start  // attacker-supplied u64, no validation
let ingress_end   = meta...ingress_end    // attacker-supplied u64, no validation

let mut ingress_expiries = vec![];
let mut now = ingress_start;
while now < ingress_end {               // NO BOUND CHECK
    ingress_expiries.push(...);         // grows unboundedly
    now += interval;
}
```

With `ingress_start = 0` and `ingress_end = 18_446_744_073_709_551_615`:

- Iterations: `18_446_744_073_709_551_615 / 120_000_000_000 ≈ 153_722_867`
- Memory for `ingress_expiries` alone: `153_722_867 × 8 bytes ≈ 1.23 GB`
- The `payloads` and `updates` vecs are then populated per-expiry per-transaction, multiplying the allocation further.

The ICRC1 Rosetta implementation at `rs/rosetta-api/icrc1/src/construction_api/services.rs:148–158` has the exact guard that is missing here, confirming this is a known requirement that was not ported to the ICP path.

### Impact Explanation
A single unauthenticated HTTP POST to the Rosetta server's `/construction/payloads` endpoint triggers OOM or prolonged CPU spin in the request-handling thread, crashing or hanging the single Rosetta replica process. No authentication, no rate-limit bypass, and no volumetric traffic is required — one request suffices.

### Likelihood Explanation
The endpoint is publicly reachable, requires no credentials, and the malicious payload is trivially constructed. The missing guard is a one-line omission compared to the sibling ICRC1 implementation.

### Recommendation
Add a window-size cap immediately after computing `ingress_start`/`ingress_end`, mirroring the ICRC1 guard:

```rust
let max_window = interval * MAX_INGRESS_EXPIRY_COUNT; // e.g., cap at ~24 h / interval
if ingress_end.saturating_sub(ingress_start.as_nanos_since_unix_epoch())
    > max_window.as_nanos() as u64
{
    return Err(ApiError::invalid_request(
        "ingress_end - ingress_start exceeds the maximum allowed window",
    ));
}
```
Or simply reject any request where `ingress_end - ingress_start > MAX_INGRESS_TTL` (a single 5-minute window is the legitimate maximum).
