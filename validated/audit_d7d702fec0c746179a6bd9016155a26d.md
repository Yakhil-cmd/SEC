### Title
Unchecked ERC-20 `transferFrom` Boolean Return Value in XCC Precompile Allows Fee-Free Cross-Contract Calls - (`engine-precompiles/src/xcc.rs`)

---

### Summary

The cross-contract call (XCC) precompile in Aurora Engine calls `transferFrom` on the wNEAR ERC-20 contract to collect payment from the caller before scheduling a cross-contract call. The EVM exit reason is checked, but the boolean return value of the `transferFrom` call is never decoded or validated. If the wNEAR token's `transferFrom` returns `false` without reverting (non-standard ERC-20 behavior), the precompile silently proceeds as if payment succeeded, allowing a caller to obtain cross-contract call execution without transferring any wNEAR.

---

### Finding Description

In `engine-precompiles/src/xcc.rs`, when `required_near != ZERO_YOCTO`, the precompile encodes a `transferFrom(sender, engine_implicit_address, required_near)` call and dispatches it via `handle.call()`:

```rust
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
match exit_reason {
    // Transfer successful, nothing to do
    aurora_evm::ExitReason::Succeed(_) => (),   // ← return_value (bool) never checked
    aurora_evm::ExitReason::Revert(r) => { ... }
    aurora_evm::ExitReason::Error(e) => { ... }
    aurora_evm::ExitReason::Fatal(f) => { ... }
}
``` [1](#0-0) 

When `exit_reason` is `Succeed`, the code does nothing (`()`). The `return_value` bytes — which encode the ABI-encoded `bool` returned by `transferFrom` — are never decoded. A non-reverting ERC-20 that returns `false` on failure (the standard alternative behavior described in EIP-20) would cause this branch to be taken while no tokens were actually moved. The precompile then proceeds to emit the promise log and return `Ok(PrecompileOutput { ... })`, scheduling the cross-contract call as if payment was received. [2](#0-1) 

The `transfer_from_args` helper constructs the ABI-encoded calldata for `transferFrom`: [3](#0-2) 

The root cause is the missing decode-and-assert step on `return_value` in the `Succeed` arm. The analogous safe pattern would be to ABI-decode `return_value` as `bool` and return an error if it is `false`.

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield / temporary freezing of funds.**

The engine's implicit address is the account that holds wNEAR used to fund outgoing NEAR cross-contract calls. If `transferFrom` silently returns `false`, the engine's wNEAR balance is not increased, yet the promise is scheduled and the engine is expected to pay for it from its own reserves. An attacker who can trigger this condition repeatedly drains the engine's wNEAR pool without paying, eventually leaving the engine unable to fund legitimate cross-contract calls (temporary freeze) or causing accounting insolvency in the wNEAR reserve.

---

### Likelihood Explanation

**Likelihood: Low-Medium.**

The wNEAR contract address is set by the engine admin and is expected to be a well-behaved ERC-20. Standard OpenZeppelin-based ERC-20 implementations revert on failure rather than returning `false`. However:

1. The wNEAR contract could be upgraded or replaced with a non-reverting variant.
2. Any future ERC-20 set as the wNEAR address that follows the alternative EIP-20 return-`false` convention would immediately expose this path.
3. The missing check is a latent defect that violates the EIP-20 specification regardless of current wNEAR behavior.

The vulnerability class is identical to the external report: the code assumes `Succeed` exit reason implies a successful token transfer, without verifying the token-level boolean result.

---

### Recommendation

After the `Succeed` arm, decode `return_value` as an ABI `bool` and treat `false` as a transfer failure:

```rust
aurora_evm::ExitReason::Succeed(_) => {
    // Decode the bool return value of transferFrom
    let transferred = return_value
        .get(31)  // last byte of the 32-byte ABI bool
        .copied()
        .unwrap_or(0);
    if transferred == 0 {
        return Err(PrecompileFailure::Error {
            exit_status: aurora_evm::ExitError::Other(
                Cow::from("ERR_WNEAR_TRANSFER_FAILED"),
            ),
        });
    }
}
```

Alternatively, use a helper that fully ABI-decodes the `bool` return and errors on `false` or missing data, mirroring OpenZeppelin's `SafeERC20.safeTransferFrom` pattern.

---

### Proof of Concept

1. Deploy or configure a wNEAR-compatible ERC-20 whose `transferFrom` always returns `false` without reverting (a valid EIP-20 implementation).
2. Set this contract as the wNEAR address in the engine state.
3. Call the XCC precompile with any `CrossContractCallArgs` that requires `required_near > 0`.
4. Observe: `exit_reason` is `Succeed`, `return_value` is `0x00…00` (false), the `Succeed` arm does nothing, and the promise log is emitted.
5. The engine's implicit address receives no wNEAR, yet the cross-contract call is scheduled — the caller paid nothing. [4](#0-3)
