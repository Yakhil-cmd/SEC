Audit Report

## Title
O(N²) CPU Exhaustion via `Vec::contains` Linear Scan in `HeightIndex::insert` with Unbounded Consensus Slot Table — (`rs/artifact_pool/src/height_index.rs`)

## Summary

`HeightIndex::insert` uses `Vec::contains` for deduplication, which is O(N) per call against the existing bucket. Because the P2P layer enforces no slot-table limit for consensus artifacts (`SLOT_TABLE_NO_LIMIT`), a single Byzantine peer can inject N cryptographically distinct artifact hashes at the same consensus height, causing O(N²) total CPU work executed synchronously on the consensus thread's unvalidated pool insertion path.

## Finding Description

**Root cause — O(N) deduplication in `HeightIndex::insert`:**

`HeightIndex<T>` stores `BTreeMap<Height, Vec<T>>`. Each call to `insert` at a height with an existing bucket of size K performs a full `Vec::contains` scan of K elements. [1](#0-0) [2](#0-1) 

For N distinct values inserted at the same height, total work is 0+1+2+…+(N-1) = **O(N²)**.

**`InMemoryPoolSection::insert` calls `indexes.insert` unconditionally before BTreeMap deduplication:**

`self.indexes.insert(msg, &hash)` is called first; the O(log N) BTreeMap dedup (`self.artifacts.entry(hash).or_insert(artifact)`) comes after. Every insertion — including those for artifacts already present — triggers the linear scan. [3](#0-2) 

**The P2P layer enforces no slot limit for consensus artifacts:**

Both consensus code paths pass `SLOT_TABLE_NO_LIMIT`, while ingress uses `SLOT_TABLE_LIMIT_INGRESS`. [4](#0-3) [5](#0-4) 

**The per-peer slot table guard is bypassed for consensus:**

The `Entry::Vacant(_) if peer_slot_table_len < self.slot_limit` branch is the only guard against unbounded slot growth. With `slot_limit = SLOT_TABLE_NO_LIMIT` (effectively `usize::MAX`), this condition is always true and the drop branch (`Entry::Vacant(_)` without the guard) is never reached. [6](#0-5) 

**Attack path:**

1. Byzantine peer sends N `SlotUpdate` messages, each on a distinct slot number, each carrying a distinct `NotarizationShare` artifact (different block hash → different `CryptoHashOf<NotarizationShare>`) all at height H. Artifacts are ≤1 KB so they are pushed inline (below `ARTIFACT_PUSH_THRESHOLD_BYTES = 1024`). [7](#0-6) 
2. The `ConsensusManagerReceiver` creates N distinct assemble tasks (one per distinct artifact ID). The bouncer accepts shares at the current height.
3. Each assembled artifact is sent via channel to the consensus thread as `UnvalidatedArtifactMutation::Insert`.
4. The consensus thread calls `MutablePool::insert` → `InMemoryPoolSection::insert` → `Indexes::insert` → `HeightIndex::insert` for the `notarization_share` bucket. [8](#0-7) 
5. The k-th insertion scans k−1 existing entries. For N=10,000 insertions: ~50 million hash comparisons (32 bytes each) executed synchronously on the consensus thread.

## Impact Explanation

The consensus thread is stalled processing O(N²) `Vec::contains` work. This delays or halts the consensus processing loop on the targeted replica, causing it to fall behind the subnet. If the Byzantine peer broadcasts to all replicas simultaneously, the entire subnet's consensus can be stalled. This matches the **High ($2,000–$10,000)** impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

The attack requires only a single Byzantine subnet peer (a compromised node acting maliciously, below the f < n/3 fault threshold). The exploit is deterministic: the attacker controls artifact content and can generate arbitrarily many distinct hashes by varying the block hash field of `NotarizationShare`. No threshold corruption, no volumetric traffic, and no cryptographic break are required. The per-peer slot table is unbounded for consensus, so no existing rate-limiting guard prevents the injection.

## Recommendation

Replace `Vec<T>` with `HashSet<T>` (requiring `T: Hash`) or `BTreeSet<T>` (requiring `T: Ord`) in `HeightIndex` buckets to reduce per-insertion deduplication from O(N) to O(1) amortized or O(log N). `CryptoHashOf<T>` already implements both `Hash` and `Ord`, making this a straightforward substitution with no API changes. Alternatively (or additionally), enforce a finite `slot_limit` for consensus artifacts analogous to `SLOT_TABLE_LIMIT_INGRESS`. [1](#0-0) 

## Proof of Concept

```rust
// Unit benchmark: insert 10_000 distinct CryptoHashOf<NotarizationShare> at height=1
let mut index: HeightIndex<CryptoHashOf<NotarizationShare>> = HeightIndex::new();
let h = Height::from(1);
for i in 0u64..10_000 {
    let hash = CryptoHashOf::from(CryptoHash(i.to_le_bytes().to_vec()));
    index.insert(h, &hash);  // k-th call scans k-1 elements → O(N²) total
}
// Wall-clock time grows quadratically; a HashSet-backed impl is flat O(1) amortized.
```

A deterministic integration test can confirm the quadratic growth by timing `index.insert` at N=1,000, N=5,000, and N=10,000 and verifying the ~25× and ~100× slowdowns relative to N=1,000.
