### Title
Unnecessary Native Resource Charge on Same-Value SSTORE — (`basic_system/src/system_implementation/system/mod.rs`)

### Summary
`charge_storage_write_extra` in `EthereumLikeStorageAccessCostModel` correctly zeroes the EVM-ergs cost when `new_value == current_value`, but unconditionally charges the full native resource cost regardless of whether the value actually changes. This mirrors the Lido-DAO pattern exactly: an expensive side-effect (native resource deduction) is triggered even when the underlying state transition is a no-op.

### Finding Description
In `charge_storage_write_extra` the ergs branch explicitly short-circuits to `0` for a same-value write:

```rust
let total_cost = if new_value == current_value {
    0          // ← ergs correctly waived
} else if ...
``` [1](#0-0) 

However, the native cost block that follows has no such guard:

```rust
let native = if is_new_slot {
    R::Native::from_computational(COLD_NEW_STORAGE_WRITE_EXTRA_NATIVE_COST)
} else {
    R::Native::from_computational(COLD_EXISTING_STORAGE_WRITE_EXTRA_NATIVE_COST)
};
resources.charge(&R::from_ergs_and_native(ergs, native))
``` [2](#0-1) 

The constants involved are substantial:

- `COLD_EXISTING_STORAGE_WRITE_EXTRA_NATIVE_COST` = `native_with_delegations!(40_000, 0, 660)` ≈ 40,660 native units
- `COLD_NEW_STORAGE_WRITE_EXTRA_NATIVE_COST` = `native_with_delegations!(100_000, 0, 1300)` ≈ 101,300 native units [3](#0-2) 

These are charged on every call to `apply_write_impl`, which is the sole write path for both `storage_write` (SSTORE) and `write_special_account_property`: [4](#0-3) 

The write itself is also unconditionally applied to the cache even when the value is unchanged: [5](#0-4) 

### Impact Explanation
Native resources are ZKsync OS's internal computational budget, separate from EVM gas. A transaction that has sufficient EVM gas for a sequence of same-value SSTOREs can still be aborted with an out-of-native-resources error because each such SSTORE drains 40,660–101,300 native units unnecessarily. This creates a class of EVM-valid transactions that are unprovable on ZKsync OS: any contract whose logic writes a storage slot to its current value (reentrancy guards, idempotent setters, flag resets) will consume native budget at the full write rate rather than the read rate, diverging from Ethereum semantics and potentially causing the transaction to fail where it should succeed.

### Likelihood Explanation
Same-value SSTOREs are common in production EVM contracts: reentrancy locks (`locked = true` when already `true`), ERC-20 approval resets, and any setter that does not pre-check the current value. An unprivileged caller can trigger this path by simply calling any such contract function on ZKsync OS. No special privileges, oracle manipulation, or governance access is required.

### Recommendation
Add a same-value guard to the native cost branch inside `charge_storage_write_extra`, mirroring the existing ergs guard:

```rust
let native = if new_value == current_value {
    R::Native::from_computational(0)
} else if is_new_slot {
    R::Native::from_computational(COLD_NEW_STORAGE_WRITE_EXTRA_NATIVE_COST)
} else {
    R::Native::from_computational(COLD_EXISTING_STORAGE_WRITE_EXTRA_NATIVE_COST)
};
```

Alternatively, skip the `addr_data.update(...)` call entirely in `apply_write_impl` when `new_value == current_value`, which would also avoid the unnecessary cache mutation.

### Proof of Concept
1. Deploy a contract on ZKsync OS with a storage slot initialised to `1`.
2. In a loop, execute `SSTORE(slot, 1)` (same-value) N times within a single transaction.
3. Observe that each iteration charges `COLD_EXISTING_STORAGE_WRITE_EXTRA_NATIVE_COST` (≈40,660 native units) despite the EVM ergs cost being 0 for each iteration after the first warm access.
4. With a sufficiently large N, the transaction exhausts native resources and reverts, even though the equivalent Ethereum transaction (paying only 100 gas per warm same-value SSTORE) would succeed.
5. The same transaction submitted to a standard Ethereum node succeeds, confirming the EVM semantic mismatch. [6](#0-5) [7](#0-6)

    fn charge_storage_write_extra(
        &self,
        ee_type: ExecutionEnvironmentType,
        initial_value: &Bytes32,
        current_value: &Bytes32,
        new_value: &Bytes32,
        resources: &mut R,
        is_warm_write: bool,
        is_new_slot: bool,
    ) -> Result<(), SystemError> {
        let ergs = match ee_type {
            ExecutionEnvironmentType::NoEE => Ergs::empty(),
            ExecutionEnvironmentType::EVM => {
                let total_cost = if new_value == current_value {
                    0
                } else if current_value == initial_value {
                    if initial_value.is_zero() {
                        // we do not purge slots, so we use another indicator here
                        SSTORE_SET_EXTRA
                    } else {
                        SSTORE_RESET_EXTRA
                    }
                } else {
                    0
                };

                let total_cost =
                    // In EVM spec there's a discrepancy for cold read and cold write costs. Cold
                    // writes add another 100 from thin air.
                    if is_warm_write == false { total_cost + 100 }
                    else { total_cost };

                Ergs(total_cost * ERGS_PER_GAS)
            }
        };
        let native = if is_new_slot {
            R::Native::from_computational(
                crate::system_implementation::flat_storage_model::cost_constants::COLD_NEW_STORAGE_WRITE_EXTRA_NATIVE_COST,
            )
        } else {
            R::Native::from_computational(
          crate::system_implementation::flat_storage_model::cost_constants::COLD_EXISTING_STORAGE_WRITE_EXTRA_NATIVE_COST,)
        };
        resources.charge(&R::from_ergs_and_native(ergs, native))
    }

    pub fn apply_write_impl(
        &mut self,
        ee_type: ExecutionEnvironmentType,
        key: &K,
        new_value: &V,
        oracle: &mut impl IOOracle,
        resources: &mut R,
    ) -> Result<(V, V), SystemError>
    where
        StorageAddress<EthereumIOTypesConfig>: From<K>,
    {
        let (mut addr_data, is_warm_read) = Self::materialize_element(
            &mut self.cache,
            &mut self.resources_policy,
            self.current_tx_id,
            ee_type,
            resources,
            key,
            oracle,
        )?;

        let val_current = addr_data.current().value();

        // Try to get initial value at the beginning of the tx.
        let val_at_tx_start = addr_data.committed().value().clone();

        let is_new_slot = addr_data.element_properties().is_new_element();
        self.resources_policy.charge_storage_write_extra(
            ee_type,
            &val_at_tx_start,
            val_current,
            new_value,
            resources,
            is_warm_read.0,
            is_new_slot,
        )?;

        let old_value = addr_data.current().value().clone();
        addr_data.update(|cache_record| {
            cache_record.update(|x, _| {
                *x = new_value.clone();
                Ok(())
            })
        })?;

        // Add refund for storage
        let mut refund_counter_value = self.evm_refunds_counter.value().clone();
        self.resources_policy.refund_for_storage_write(
            ee_type,
            &val_at_tx_start,
            &old_value,
            new_value,
            resources,
            &mut refund_counter_value,
        )?;
        self.evm_refunds_counter.update(refund_counter_value);

        Ok((old_value, val_at_tx_start))
    }
