### Title
Byzantine peer forces `AssembleResult::Unwanted` via duplicate `IDkgArtifactId` entries in `StrippedBlockProposal` — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs`)

### Summary
`TryFrom<pb::StrippedBlockProposal> for StrippedBlockProposal` accepts any number of `stripped_idkg_dealings` entries without checking for duplicate `IDkgArtifactId` values. A Byzantine peer can craft a `StrippedBlockProposal` whose `stripped_idkg_dealings` contains two entries sharing the same `IDkgArtifactId` but with different `node_index` values. This causes `BlockProposalAssembler` to create two slots for the same dealing, spawn two fetch tasks, and then fail with `InsertionError::AlreadyInserted` on the second insert — which is treated as a fatal assembly error, returning `AssembleResult::Unwanted` and permanently discarding the block proposal on the victim node.

### Finding Description
The deserialization path in `TryFrom<pb::StrippedBlockProposal>` (stripped.rs lines 110–128) collects all `stripped_idkg_dealings` entries into a plain `Vec` with no uniqueness enforcement on `IDkgArtifactId`. `BlockProposalAssembler::new` (assembler.rs lines 580–585) mirrors this `Vec` directly into `signed_dealings`, creating one `Option<SignedIDkgDealing>` slot per entry. `missing_stripped_messages` (assembler.rs lines 591–597) returns all unfilled slots, so two `get_or_fetch` tasks are spawned for the same `IDkgArtifactId`. When results arrive, `PayloadAssembler::<SignedIDkgDealing>::try_insert` (assembler.rs lines 497–501) searches `signed_dealings` by `dealing_id` alone — `node_index` is silently dropped at the call site (assembler.rs line 609). The first insert fills slot 0; the second finds slot 0 already occupied and returns `InsertionError::AlreadyInserted`. The caller (assembler.rs lines 262–271) treats any insertion error as a bug and returns `AssembleResult::Unwanted`, permanently aborting assembly of that block proposal on the victim node.

### Impact Explanation
A single Byzantine peer (no threshold corruption required) can prevent any victim node from assembling a block proposal that contains IDKG dealings. Because `AssembleResult::Unwanted` signals to the P2P layer that the artifact is not desired, the node will not retry assembly for the same `StrippedConsensusMessageId`. If the attacker races the honest stripped proposal and delivers the malformed one first, the victim node never assembles the block. Scaled to multiple victim nodes, this can stall IDKG transcript completion and threshold key operations for the targeted subnet.

### Likelihood Explanation
The attacker only needs to be a peer on the subnet's P2P overlay (a single Byzantine replica below the fault threshold). Block proposal IDs are observable from the network. The malformed `StrippedBlockProposal` passes all existing deserialization checks. No cryptographic material, admin access, or majority corruption is required.

### Recommendation
In `TryFrom<pb::StrippedBlockProposal>` (stripped.rs lines 110–128), after collecting `stripped_idkg_dealings`, reject the message if any `IDkgArtifactId` appears more than once:

```rust

    if !seen.insert(id) {
        return Err(ProxyDecodeError::Other(
            "Duplicate IDkgArtifactId in stripped_idkg_dealings".into()

    }
}
```

Alternatively, fix `PayloadAssembler::<SignedIDkgDealing>::try_insert` to match on `(node_index, dealing_id)` rather than `dealing_id` alone, and handle `AlreadyInserted` as a non-fatal condition (skip rather than abort).
