### Title
Division by Zero in `get_node_provider_reward` Causes NNS Governance Canister Panic - (File: rs/nns/governance/src/governance.rs)

### Summary
The `get_node_provider_reward` function in the NNS governance canister performs an unchecked integer division by `xdr_permyriad_per_icp`. In Rust, integer division by zero causes a panic (trap), which would abort the node provider reward distribution execution. If `xdr_permyriad_per_icp` reaches zero — possible when `minimum_icp_xdr_rate` is zero (the protobuf default for `u64`) and the CMC oracle returns zero — the governance canister traps during reward calculation.

### Finding Description
In `get_node_provider_reward`, the ICP reward amount is computed as:

```rust
let amount_e8s = ((xdr_permyriad_reward as u128 * TOKEN_SUBDIVIDABLE_BY as u128)
    / xdr_permyriad_per_icp as u128) as u64;
``` [1](#0-0) 

There is no guard against `xdr_permyriad_per_icp == 0`. The callers compute this value as:

```rust
let xdr_permyriad_per_icp = max(avg_xdr_permyriad_per_icp, minimum_xdr_permyriad_per_icp);
``` [2](#0-1) 

where `minimum_xdr_permyriad_per_icp` is derived from:

```rust
let minimum_xdr_permyriad_per_icp = self
    .economics()
    .minimum_icp_xdr_rate
    .saturating_mul(NetworkEconomics::ICP_XDR_RATE_TO_BASIS_POINT_MULTIPLIER);
``` [3](#0-2) 

`minimum_icp_xdr_rate` is a `u64` field in `NetworkEconomics`. Its protobuf default is `0`. If the governance canister is initialized without explicitly setting this field, `minimum_xdr_permyriad_per_icp` evaluates to `0`. If the CMC oracle simultaneously returns `xdr_permyriad_per_icp = 0` (e.g., during early initialization or a transient oracle failure), then `max(0, 0) = 0` is passed to `get_node_provider_reward`, triggering an integer division by zero panic.

The same pattern exists in both `get_node_providers_rewards` and `get_monthly_node_provider_rewards`. [4](#0-3) 

### Impact Explanation
A panic in the NNS governance canister during node provider reward distribution aborts the entire reward calculation execution. Node providers receive no rewards for that period. Depending on how the periodic task is structured, repeated panics could permanently stall the reward distribution pipeline. This is a **governance reward accounting bug** with direct financial impact on node providers.

**Impact: 3** — Reward distribution is disrupted; no funds are lost from the ledger, but node providers are denied earned rewards.

### Likelihood Explanation
The trigger requires two conditions to coincide: (1) `minimum_icp_xdr_rate` is zero (the protobuf default, reachable without any privileged action if the field was never explicitly initialized), and (2) the CMC oracle returns a zero rate (possible during initialization windows or transient oracle states). Neither condition individually requires a privileged actor.

**Likelihood: 2** — Requires a specific combination of default/uninitialized state and an oracle edge case, but no attacker privilege is needed.

### Recommendation
Add an explicit zero-check in `get_node_provider_reward` before performing the division:

```rust
pub fn get_node_provider_reward(
    np: &NodeProvider,
    xdr_permyriad_reward: u64,
    xdr_permyriad_per_icp: u64,
) -> Option<RewardNodeProvider> {
    if xdr_permyriad_per_icp == 0 {
        return None; // or log an error and return None
    }
    // ... existing logic
}
```

Additionally, the callers should validate that `xdr_permyriad_per_icp > 0` before invoking `get_node_provider_reward`, and `NetworkEconomics` should enforce a non-zero minimum for `minimum_icp_xdr_rate` during initialization.

### Proof of Concept

1. Deploy NNS governance canister with `NetworkEconomics` where `minimum_icp_xdr_rate = 0` (protobuf default — no explicit initialization needed).
2. Arrange for the CMC oracle (`get_average_icp_xdr_conversion_rate`) to return `xdr_permyriad_per_icp = 0`.
3. Trigger `get_node_providers_rewards` or `get_monthly_node_provider_rewards` (called periodically by the governance canister's timer tasks).
4. Execution reaches `get_node_provider_reward(np, reward, 0)`.
5. Line 8255 executes `... / 0u128`, causing a Rust integer division by zero panic, trapping the canister message.
6. Node provider rewards for that period are not distributed. [5](#0-4)

pub fn get_node_provider_reward(
    np: &NodeProvider,
    xdr_permyriad_reward: u64,
    xdr_permyriad_per_icp: u64,
) -> Option<RewardNodeProvider> {
    if let Some(np_id) = np.id.as_ref() {
        let amount_e8s = ((xdr_permyriad_reward as u128 * TOKEN_SUBDIVIDABLE_BY as u128)
            / xdr_permyriad_per_icp as u128) as u64;

        let to_account = Some(if let Some(account) = &np.reward_account {
            account.clone()
        } else {
            AccountIdentifier::from(*np_id).into()
        });

        Some(RewardNodeProvider {
            node_provider: Some(np.clone()),
            amount_e8s,
            reward_mode: Some(RewardMode::RewardToAccount(RewardToAccount { to_account })),
        })
    } else {
        None
    }
}
