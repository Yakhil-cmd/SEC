### Title
Unused `gas_per_pubdata_limit` Field Silently Bypasses User-Specified Pubdata Cost Constraint in ZK L2 Transactions — (`basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`)

---

### Summary

The `gas_per_pubdata_limit` field in `AbiEncodedTransaction` is parsed from ZK-format L2 transactions but is annotated `#[allow(dead_code)]` and never enforced during L2 ZK transaction validation. The ZK validation path computes `native_per_pubdata` entirely from system-level oracle prices, never comparing it against the user's stated limit. This is the direct analog of the external report's `sqrtPriceLimitX96: 0` pattern: a user-specified constraint field exists in the struct but is silently ignored, removing the protection it was designed to provide.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`, the `gas_per_pubdata_limit` field is defined with an explicit `#[allow(dead_code)]` suppression:

```rust
/// The maximum amount of gas the user is willing to pay for a byte of pubdata.
#[allow(dead_code)]
pub gas_per_pubdata_limit: ParsedValue<u32>,
``` [1](#0-0) 

The field is parsed and accessible via `transaction.gas_per_pubdata_limit()`, which returns the actual value for ABI-encoded transactions:

```rust
pub fn gas_per_pubdata_limit(&self) -> U256 {
    match self {
        Self::Rlp(_) => U256::ZERO,
        Self::Abi(tx) => U256::from(tx.gas_per_pubdata_limit.read()),
    }
}
``` [2](#0-1) 

For **L1→L2 transactions**, `gas_per_pubdata_limit` is correctly read and used to compute `native_per_pubdata`:

```rust
let gas_per_pubdata = transaction.gas_per_pubdata_limit.read();
``` [3](#0-2) 

However, in the **L2 ZK transaction validation path** (`zk/validation_impl.rs`), `native_per_pubdata` is derived exclusively from system-level oracle prices — `pubdata_price` and `native_price` — with no reference to the transaction's `gas_per_pubdata_limit`:

```rust
let pubdata_price = system.get_pubdata_price();
let native_price = system.get_native_price();
// ...
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [4](#0-3) 

The `gas_per_pubdata_limit` is never compared against `native_per_pubdata` anywhere in this path. The resources are then created using the unconstrained system-derived `native_per_pubdata`:

```rust
let tx_resources = create_resources_for_tx::<S, L2ResourcesPolicy>(
    system,
    tx_gas_limit,
    native_per_gas == 0,
    native_prepaid_from_gas,
    native_per_pubdata,   // ← derived from oracle, not bounded by gas_per_pubdata_limit
    intrinsic_gas,
    intrinsic_computational_native,
    intrinsic_pubdata,
)?;
``` [5](#0-4) 

---

### Impact Explanation

A user signs a ZK L2 transaction with `gas_per_pubdata_limit = X`, intending to cap their pubdata exposure. If the block's pubdata price (set by the operator via oracle) exceeds `X`, the transaction is still accepted and executed, and the user is charged pubdata costs at the full oracle price. The user's stated maximum is never checked, so they are charged more than they consented to. This constitutes a direct fund-loss path for the transaction sender: the user's balance is debited for pubdata at a price they explicitly refused to accept.

---

### Likelihood Explanation

Pubdata prices on ZKsync L2 fluctuate with L1 data availability costs. Any time the operator-set `pubdata_price` exceeds a user's `gas_per_pubdata_limit`, the overcharge silently occurs. Users who set conservative pubdata limits (e.g., pre-signed transactions with a fixed `gas_per_pubdata_limit`) are systematically exposed whenever pubdata prices rise. No special attacker capability is required beyond submitting a standard ZK-format L2 transaction with a non-zero `gas_per_pubdata_limit`.

---

### Recommendation

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, after computing `native_per_pubdata`, compare it against the transaction's `gas_per_pubdata_limit`. If the actual pubdata cost per byte exceeds the user's stated limit, reject the transaction with an appropriate `InvalidTransaction` variant (e.g., `PubdataPriceTooHigh`). Remove the `#[allow(dead_code)]` suppression from the field once it is actively used.

---

### Proof of Concept

1. User constructs a ZK-format L2 transaction with `gas_per_pubdata_limit = 1` (willing to pay at most 1 gas per pubdata byte).
2. Operator sets block `pubdata_price` such that `pubdata_price / native_price` yields `native_per_pubdata = 1000`.
3. `validate_and_compute_fee_for_transaction` in `zk/validation_impl.rs` computes `native_per_pubdata = 1000` from oracle prices and never reads `gas_per_pubdata_limit`.
4. Transaction is accepted and resources are allocated with `native_per_pubdata = 1000`.
5. After execution, pubdata charges are applied at 1000× the user's stated maximum, draining more of the user's balance than they agreed to.
6. The `gas_per_pubdata_limit` field, despite being parsed and stored in the struct, had zero effect on the outcome.
