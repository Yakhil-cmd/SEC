### Title
Missing Validation of `chain_id` in Interop Root Acceptance - (`system_hooks/src/event_hooks/interop_root_reporter.rs`)

### Summary
The `interop_root_reporter_event_hook` function accepts an arbitrary `chain_id` value from event topics and stores it as an `InteropRoot` without any validation. The `InteropRoot` struct itself documents that `chain_id` "must be non-zero," but this invariant is never enforced anywhere in the Rust code path. An interop root with `chain_id = 0` (or any arbitrary, unsupported chain ID) can be stored and included in the `interop_roots_rolling_hash` that becomes part of the batch public input committed to the settlement layer.

### Finding Description

In `system_hooks/src/event_hooks/interop_root_reporter.rs`, the `interop_root_reporter_event_hook` function parses the `chain_id` directly from `topics[1]` and passes it to `system.io.add_interop_root` without any validation:

```rust
let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
system.io.add_interop_root(
    ExecutionEnvironmentType::NoEE,
    resources,
    InteropRoot {
        root,
        block_or_batch_number,
        chain_id,  // ← never validated
    },
)?;
``` [1](#0-0) 

The `InteropRoot` struct in `zk_ee/src/common_structs/interop_root_storage.rs` explicitly documents the invariant "Source chain identifier (must be non-zero)" in a comment, but `push_root` performs no such check:

```rust
pub struct InteropRoot {
    /// The merkle root hash (cannot be zero for valid roots)
    pub root: Bytes32,
    /// Block or batch number from the source chain
    pub block_or_batch_number: U256,
    /// Source chain identifier (must be non-zero)
    pub chain_id: U256,
}
``` [2](#0-1) 

The `push_root` function in `InteropRootStorage` stores the root unconditionally:

```rust
pub fn push_root(&mut self, interop_root: InteropRoot) -> Result<(), SystemError> {
    self.list.push(interop_root, ());
    Ok(())
}
``` [3](#0-2) 

The `add_interop_root` implementation in `basic_system/src/system_implementation/system/io_subsystem.rs` also performs no `chain_id` validation before calling `push_root`: [4](#0-3) 

By contrast, the `root` field IS validated at the Solidity contract level (a zero root causes the transaction to fail, as confirmed by the test `run_fails_if_interop_root_is_incorrect`), but `chain_id` has no equivalent check anywhere. [5](#0-4) 

The stored interop roots are iterated and fed into `calculate_interop_roots_rolling_hash`, which hashes `chain_id` into the batch-level `interop_roots_rolling_hash` that is committed to the settlement layer as part of the batch public input: [6](#0-5) [7](#0-6) 

### Impact Explanation

An interop root with `chain_id = 0` (or any arbitrary, non-existent chain ID) can be stored in `InteropRootStorage` and included in the `interop_roots_rolling_hash` committed to the settlement layer. This corrupts the cross-chain interoperability commitment with data from a non-existent or unsupported source chain. The settlement layer would accept a batch proof that attests to interop roots from chain ID 0, which has no corresponding chain. This undermines the integrity of the cross-chain state verification mechanism and could be used to inject fabricated interop roots into the batch commitment.

### Likelihood Explanation

The `interop_root_reporter_event_hook` fires on any `InteropRootAdded` event emitted by the `L2_INTEROP_ROOT_STORAGE_ADDRESS` contract. The intended trigger is a service transaction (type `0x7d`), which is restricted to `BOOTLOADER_FORMAL_ADDRESS` as sender. [8](#0-7) 

However, the event hook fires on events emitted during any transaction execution against that contract address — not exclusively service transactions. If the deployed `L2_INTEROP_ROOT_STORAGE_ADDRESS` Solidity contract does not enforce caller restrictions on `addInteropRootsInBatch`, any user can call it directly and trigger the hook with an arbitrary `chain_id`. Even if the contract does restrict callers, the operator (who controls service transactions) can submit a `chain_id = 0` root, and the Rust layer will accept it without error.

### Recommendation

Add an explicit non-zero check for `chain_id` in `interop_root_reporter_event_hook` before calling `add_interop_root`, and enforce the documented invariant in `push_root` or `add_interop_root`:

```rust
if chain_id.is_zero() {
    return Err(internal_error!("Interop root reporter: chain_id must be non-zero").into());
}
```

Optionally, maintain a whitelist of supported source chain IDs and reject roots from unknown chains, mirroring the pattern used for transaction chain ID validation in `parse_and_compute_signed_hash`. [9](#0-8) 

### Proof of Concept

A service transaction (or direct call if the Solidity contract permits it) submitting an `InteropRoot` with `chain_id = 0` and a valid non-zero `root`:

```rust
let interop_roots = vec![StoredInteropRoot {
    root: Bytes32::from_u256_be(&U256::ONE), // non-zero root passes Solidity check
    block_or_batch_number: U256::from(42),
    chain_id: U256::ZERO, // zero chain_id — documented as invalid, never enforced
}];
// This succeeds: the Rust hook stores the root without rejecting chain_id = 0.
// The interop_roots_rolling_hash in the batch public input now includes
// a root attributed to the non-existent chain 0.
```

The existing test `run_processes_interop_roots_max_amount` already demonstrates that `chain_id = U256::MAX` (an arbitrary, unsupported value) is accepted without error, confirming no chain ID validation exists. [10](#0-9)

```rust
pub fn calculate_interop_roots_rolling_hash<'a>(
    old_rolling_hash: Bytes32,
    roots: impl Iterator<Item = &'a InteropRoot>,
    hasher: &mut crypto::sha3::Keccak256,
) -> Bytes32 {
    let mut data = [0u8; 96];

    let mut rolling_hash = old_rolling_hash;
    for root in roots {
        data[0..32].copy_from_slice(&rolling_hash.as_u8_ref());
        data[32..64].copy_from_slice(&root.chain_id.to_be_bytes::<{ U256::BYTES }>());
        data[64..96].copy_from_slice(&root.block_or_batch_number.to_be_bytes::<{ U256::BYTES }>());
        hasher.update(data);

        // Note: now we have only one side
        hasher.update(root.root.as_u8_ref());

        rolling_hash = hasher.finalize_reset().into()
    }

    rolling_hash
}
