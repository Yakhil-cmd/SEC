### Title
Silent ICP Burn Failure Breaks ICP/Cycles Conservation Invariant After Successful Cycles Minting - (File: rs/nns/cmc/src/main.rs)

### Summary
In the Cycles Minting Canister (CMC), the `burn_and_log` function is called after cycles are successfully deposited or minted to destroy the corresponding ICP. The function intentionally swallows all errors from the ICP ledger burn call, only logging failures. If the burn fails (e.g., during a ledger upgrade), cycles have already been irreversibly minted while the ICP remains un-burned in the CMC's subaccount, permanently breaking the ICP/cycles conservation invariant with no recovery path.

### Finding Description

The `burn_and_log` function in `rs/nns/cmc/src/main.rs` is the IC analog of the `chargeFees()` / `refundSuccess` pattern in the report. It is called in three critical flows after the primary operation (cycles deposit or mint) has already succeeded:

1. `process_top_up` — after `deposit_cycles` succeeds
2. `process_create_canister` — after `do_create_canister` succeeds
3. `process_mint_cycles` — after `do_mint_cycles` succeeds [1](#0-0) 

The function calls the ICP ledger's `send_pb` to transfer ICP to the minting account (burn). If this call fails, the error is only printed — it is never propagated, never retried, and there is no recovery mechanism: [2](#0-1) 

The explicit design rationale is stated in the comment: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."* The concern is that propagating the error would allow the notification to be retried, potentially double-minting cycles. However, this design leaves no path to recover the un-burned ICP. [3](#0-2) 

After `burn_and_log` returns (regardless of success or failure), the notification is recorded as completed (`NotifiedTopUp`, `NotifiedCreateCanister`, or `NotifiedMint`), permanently preventing any retry: [4](#0-3) 

### Impact Explanation

When `burn_and_log` fails silently:

- Cycles have already been deposited to the target canister or minted to the cycles ledger — this is irreversible.
- The ICP remains in the CMC's subaccount (keyed to the canister ID or controller principal) and is permanently inaccessible — no user can withdraw it, and no retry mechanism exists.
- The ICP/cycles conservation invariant is broken: cycles were created without a corresponding ICP burn, inflating the effective cycles supply relative to the ICP supply.
- The notification is marked complete, so the user cannot trigger a retry.

This is a **ledger conservation bug**: the protocol guarantee that "ICP burned = cycles minted" is violated without any on-chain evidence or alerting mechanism beyond a log print. [5](#0-4) [6](#0-5) 

### Likelihood Explanation

The ICP ledger can be temporarily unavailable during canister upgrades (a routine NNS governance operation). The window between `deposit_cycles`/`do_create_canister`/`do_mint_cycles` completing and `burn_and_log` executing is an async await point — the ledger could be mid-upgrade at exactly that moment. Any unprivileged user calling `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` during such a window triggers the condition. No special privileges are required; the entry path is fully open to any ICP holder. [7](#0-6) 

### Recommendation

Add a deferred retry mechanism for failed burns: record the un-burned subaccount and amount in CMC state, and attempt the burn again in a periodic heartbeat or timer task. Alternatively, expose a privileged `retry_burn` endpoint callable by NNS governance. At minimum, emit a certified metric or stable-memory record of failed burns so that the conservation invariant can be audited and manually corrected via governance proposal.

### Proof of Concept

1. User sends ICP to CMC subaccount for canister top-up and calls `notify_top_up`.
2. CMC calls `deposit_cycles` — succeeds, cycles are credited to the target canister.
3. CMC calls `burn_and_log` — the ICP ledger is mid-upgrade and rejects the call.
4. `burn_and_log` prints the error and returns `()`.
5. CMC records `NotificationStatus::NotifiedTopUp(Ok(cycles))` — notification is permanently closed.
6. The ICP remains in the CMC's subaccount for the canister ID. No user can retrieve it. No retry is possible. The cycles exist on-chain without a corresponding ICP burn. [8](#0-7)

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
