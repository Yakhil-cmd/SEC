Audit Report

## Title
Ingress History Oldest-First Eviction Allows Attacker to Prematurely Destroy Victim's `Completed` Reply Payload — (`rs/replicated_state/src/metadata_state.rs`)

## Summary
`IngressHistoryState::forget_terminal_statuses` evicts `Completed`/`Failed` ingress statuses to `Done` (dropping reply payloads) in oldest-first order when `memory_usage` exceeds `ingress_memory_capacity`. An unprivileged attacker controlling a canister that returns large reply payloads can flood the ingress history with `Completed` entries timestamped after a victim's entry, causing the victim's `Completed` status to be evicted to `Done` before the victim polls `read_state`, permanently destroying the reply payload.

## Finding Description

The eviction loop in `forget_terminal_statuses` iterates `pruning_times` from `next_terminal_time` (oldest) upward:

```rust
for (time, ids) in self
    .pruning_times
    .range((Included(self.next_terminal_time), Unbounded))
{
    self.next_terminal_time = *time;

    if self.memory_usage <= target_size {
        break;
    }
    // ... evict all entries in this bucket to Done
``` [1](#0-0) 

The memory check fires at the **top** of each bucket iteration, **before** processing entries in that bucket. This means: if `memory_usage > target_size` when the oldest bucket is entered, every `Completed`/`Failed` entry in that bucket is unconditionally evicted to `Done`. There is no per-user fairness, no minimum retention window, and no preference for evicting the party that caused the overflow. [2](#0-1) 

The `insert` function inserts the new status into `statuses` and updates `memory_usage` **before** calling `forget_terminal_statuses`:

```rust
self.memory_usage += status.payload_bytes();
let old_status = Arc::make_mut(&mut self.statuses).insert(message_id, Arc::new(status));
// ...
if self.memory_usage > ingress_memory_capacity.get() as usize {
    self.forget_terminal_statuses(ingress_memory_capacity, time, ...);
}
``` [3](#0-2) 

Attack sequence:
1. Victim's ingress message M_v completes at block time T1. `Completed` status inserted with pruning time `T1 + MAX_INGRESS_TTL`.
2. Attacker submits N ingress messages to an attacker-controlled canister returning large reply payloads. These complete at block time T2 > T1, with pruning time `T2 + MAX_INGRESS_TTL`.
3. Each attacker insertion pushes `memory_usage` above `ingress_memory_capacity`.
4. `forget_terminal_statuses` is called. `next_terminal_time ≤ T1 + MAX_INGRESS_TTL`. The victim's bucket is encountered first. Memory > target → victim's entry is evicted to `Done` (payload dropped).
5. Memory drops below target; loop breaks. Attacker's entries at T2 remain `Completed`.
6. Victim polls `read_state` / `get_ingress_status` and receives `Done` — reply payload is permanently gone.

The existing test at lines 1383–1407 explicitly confirms this behavior: with capacity for exactly one entry, the most recently inserted entry always survives while all older entries are evicted to `Done`. [4](#0-3) 

The `set_ingress_status` doc comment at line 803–808 acknowledges oldest-first eviction as the design, but does not account for the cross-user exploitation this enables. [5](#0-4) 

## Impact Explanation

A legitimate user permanently loses their canister reply. The `Done` state carries no payload; there is no retry path — the data is gone from replicated state. The user submitted a valid ingress message, paid fees, the canister executed correctly, but the user never receives the result. This constitutes a targeted application-level denial-of-response against specific users, matching the **High** impact class: *Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS* ($2,000–$10,000).

## Likelihood Explanation

`INGRESS_HISTORY_MEMORY_CAPACITY` is 4 GiB. [6](#0-5) 

With a max response size of ~2 MB per message, filling the history from empty requires ~2,000 attacker messages — a significant but not prohibitive cycles cost. On a busy subnet where the history is already near capacity, far fewer messages suffice. The attacker requires: (1) a canister under their control returning large payloads (trivially deployable), (2) enough cycles to push `memory_usage` above capacity, and (3) the victim's message to have completed in an earlier block than the attacker's flood. No privileged access, consensus corruption, or threshold attack is required.

## Recommendation

1. **Per-user/per-canister eviction fairness**: Track ingress history memory per `user_id` or `receiver`. When eviction is needed, evict from the largest contributor first rather than globally oldest-first.
2. **Evict the inserting party first**: When a new insertion triggers overflow, prefer evicting entries from the same `user_id` before touching other users' entries.
3. **Minimum retention guarantee**: Guarantee that a `Completed` entry is never evicted to `Done` within a minimum window (e.g., 30–60 seconds) after it was set, giving the originating user time to poll.
4. **Per-user ingress history cap**: Reject or immediately evict new `Completed` entries from a user who already holds a disproportionate share of ingress history memory.

## Proof of Concept

State-machine unit test (safe, local, deterministic):

```rust
use ic_replicated_state::metadata_state::IngressHistoryState;
use ic_types::{ingress::{IngressState, IngressStatus}, NumBytes, Time};

let mut history = IngressHistoryState::new();
let capacity = NumBytes::from(2 * 2_000_000u64); // room for ~2 large entries

// Step 1: Insert victim's Completed status at T1
history.insert(
    victim_msg_id(),
    completed_status(T1, large_payload()),
    T1,
    capacity,
    |_| {},
);
assert!(matches!(
    history.get(&victim_msg_id()),
    Some(IngressStatus::Known { state: IngressState::Completed(_), .. })
));

// Step 2: Flood with attacker's Completed statuses at T2 > T1
// Each insertion triggers forget_terminal_statuses when memory > capacity
for i in 0..N {
    history.insert(
        attacker_msg_id(i),
        completed_status(T2, large_payload()),
        T2,
        capacity,
        |_| {},
    );
}

// Step 3: Victim's entry is now Done — payload lost
assert!(matches!(
    history.get(&victim_msg_id()),
    Some(IngressStatus::Known { state: IngressState::Done, .. })
));
// Attacker's most recent entry remains Completed
assert!(matches!(
    history.get(&attacker_msg_id(N - 1)),
    Some(IngressStatus::Known { state: IngressState::Completed(_), .. })
));
```

The existing test `ingress_history_insert_beyond_limit_will_succeed` at lines 1362–1437 already demonstrates the core eviction behavior; extending it with two distinct `user_id` values and verifying cross-user eviction order directly proves the exploit. [7](#0-6)

#[test]
fn ingress_history_insert_beyond_limit_will_succeed() {
    let mut ingress_history = IngressHistoryState::default();

    let insert_status = |ingress_history: &mut IngressHistoryState, i, max_num_entries| {
        let message_id = message_test_id(i);
        let status = test_status_terminal(i);
        let limit = NumBytes::from(max_num_entries * status.payload_bytes() as u64);
        ingress_history.insert(
            message_id.clone(),
            status.clone(),
            Time::from_nanos_since_unix_epoch(i),
            limit,
            |_| {},
        );
        (message_id, status)
    };

    // Inserting with enough space for exactly one entry will always leave the
    // most recently inserted status there while setting everything else to
    // done.
    for i in 1..=100 {
        let (inserted_message_id, inserted_status) = insert_status(&mut ingress_history, i, 1);

        assert_eq!(ingress_history.statuses().count(), i as usize);
        assert_eq!(
            ingress_history.get(&inserted_message_id).unwrap(),
            &inserted_status
        );
        assert_eq!(
            ingress_history
                .statuses()
                .filter(|(_, status)| matches!(
                    status,
                    IngressStatus::Known {
                        state: IngressState::Completed(_),
                        ..
                    } | IngressStatus::Known {
                        state: IngressState::Failed(_),
                        ..
                    }
                ))
                .count(),
            1
        );
    }

    // Inserting without available space will directly transition inserted status
    // to done.
    for i in 101..=200 {
        let (inserted_message_id, _) = insert_status(&mut ingress_history, i, 0);

        assert_eq!(ingress_history.statuses().count(), i as usize);
        assert_eq!(
            ingress_history.get(&inserted_message_id).unwrap(),
            &test_status_done(i),
        );

        assert_eq!(
            ingress_history
                .statuses()
                .filter(|(_, status)| matches!(
                    status,
                    IngressStatus::Known {
                        state: IngressState::Completed(_),
                        ..
                    } | IngressStatus::Known {
                        state: IngressState::Failed(_),
                        ..
                    }
                ))
                .count(),
            0
        );
    }
}
