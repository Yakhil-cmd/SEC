### Title
Unprivileged Caller Can Acquire `balance_update_guard` for Any Victim Account, Blocking ckBTC Minting - (`rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

---

### Summary

The `update_balance` endpoint in the ckBTC minter accepts an optional `owner: Option<Principal>` parameter that any authenticated caller may set to an arbitrary principal. The per-account concurrency guard (`balance_update_guard`) is keyed on the full `Account { owner, subaccount }` derived from this caller-supplied value. An unprivileged attacker can call `update_balance` with a victim's principal as `owner`, acquiring the guard for the victim's account. For the entire duration of the attacker's async execution (which spans multiple inter-canister calls), the legitimate owner's own `update_balance` call returns `AlreadyProcessing`. By continuously repeating this, the attacker sustains a DoS against any specific user's ability to mint ckBTC. With 100 concurrent calls using distinct accounts, the attacker can exhaust the global `MAX_CONCURRENT = 100` slot limit and block all users simultaneously.

---

### Finding Description

**Root cause — caller-supplied `owner` is used as the guard key without authorization:**

`UpdateBalanceArgs` exposes `owner: Option<Principal>` as a free parameter: [1](#0-0) 

Inside `update_balance`, the guard is acquired for the account constructed from the caller-supplied `owner`, not from the actual caller: [2](#0-1) 

The only authorization checks performed before the guard is acquired are: (1) the owner must not equal the minter's own canister ID, and (2) the caller must pass the deposit-mode whitelist check. There is no check that `args.owner == caller`. [3](#0-2) 

**Guard mechanics — `AlreadyProcessing` and `TooManyConcurrentRequests`:**

The guard implementation inserts the account into a `BTreeSet` and returns `AlreadyProcessing` if the same account is already present, or `TooManyConcurrentRequests` if 100 slots are occupied: [4](#0-3) 

The guard is held for the entire async lifetime of `update_balance`, which spans at minimum one call to the Bitcoin canister (`get_utxos`) and potentially calls to the BTC checker and the ckBTC ledger (for minting). The guard is only released via `Drop` when the function returns: [5](#0-4) 

**Exploit path:**

1. Attacker calls `update_balance({ owner = Some(victim_principal), subaccount = victim_subaccount })`.
2. The minter inserts `Account { owner: victim_principal, subaccount: victim_subaccount }` into `update_balance_accounts`.
3. The victim calls `update_balance()` for their own account → receives `AlreadyProcessing`.
4. The attacker's call eventually completes with `NoNewUtxos` (no UTXOs exist at the victim's BTC address from the attacker's perspective, or the attacker's call simply finds nothing new). The guard is released.
5. The attacker immediately re-calls `update_balance({ owner = Some(victim_principal), ... })` before the victim can retry.
6. Repeat indefinitely.

For a global DoS: the attacker opens 100 concurrent `update_balance` calls with 100 distinct accounts (e.g., `Account { owner: attacker_principal, subaccount: Some([i; 32]) }` for `i` in `0..100`). This fills all 100 slots and causes every subsequent `update_balance` call from any user to return `TooManyConcurrentRequests`: [6](#0-5) 

The `MAX_CONCURRENT = 100` constant is the shared ceiling for both `PendingBalanceUpdates` and `RetrieveBtcUpdates` guard types.

---

### Impact Explanation

- **Targeted DoS:** An attacker can continuously block a specific victim from calling `update_balance` for their account, preventing them from minting ckBTC for deposited UTXOs. The victim's BTC is locked on the Bitcoin network with no way to convert it to ckBTC while the attack is sustained.
- **Global DoS:** With 100 concurrent calls using distinct accounts, the attacker can block all users from calling `update_balance` simultaneously, halting all ckBTC minting on the minter canister.
- No funds are directly stolen, but the inability to mint ckBTC for confirmed BTC deposits constitutes a severe service disruption for the chain-fusion bridge.

---

### Likelihood Explanation

The attack is reachable by any non-anonymous ingress caller with no privileged access. The `update_balance` endpoint is publicly callable and the `owner` parameter is explicitly documented as settable to any principal. The attacker's cost is IC cycles per call; each call is cheap relative to the disruption caused. The attack can be automated and sustained indefinitely.

---

### Recommendation

Restrict the guard key to the **caller's** identity rather than the caller-supplied `owner`. Specifically, acquire the guard using `Account { owner: caller, subaccount: args.subaccount }` (or simply `caller`) rather than `Account { owner: args.owner.unwrap_or(caller), subaccount: args.subaccount }`. The guard's purpose is to prevent the same user from submitting duplicate concurrent requests; it should be keyed on who is making the call, not on whose balance is being updated. The actual minting target account can remain `args.owner.unwrap_or(caller)` — only the guard key needs to change.

### Proof of Concept

```
// Attacker principal: ATTACKER
// Victim principal:   VICTIM

// Step 1: Attacker opens a long-running update_balance call for victim's account.
update_balance({ owner = Some(VICTIM), subaccount = None })
// → Minter inserts Account{VICTIM, None} into update_balance_accounts.
// → Async execution begins (get_utxos call to Bitcoin canister in flight).

// Step 2: Victim calls update_balance for their own account (same round or next).
update_balance({ owner = None })  // owner defaults to VICTIM
// → Guard::new finds Account{VICTIM, None} already in set.
// → Returns Err(AlreadyProcessing).

// Step 3: Attacker's call completes (NoNewUtxos). Guard dropped.
// Step 4: Attacker immediately re-calls. Go to Step 1.

// Global variant: attacker opens 100 concurrent calls with distinct accounts.
// All subsequent update_balance calls from any user → TooManyConcurrentRequests.
```

```rust
impl<PR: PendingRequests> Guard<PR> {
    /// Attempts to create a new guard for the current block. Fails if there is
    /// already a pending request for the specified [principal] or if there
    /// are at least [MAX_CONCURRENT] pending requests.
    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }
}
```
