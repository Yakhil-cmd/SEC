### Title
`calculate_attached_gas` Silently Returns Zero Gas, Permanently Locking User's Attached NEAR in Engine Contract — (`engine/src/contract_methods/connector.rs`)

---

### Summary

When `storage_deposit` is called with non-zero NEAR attached and a prepaid gas budget tight enough that `calculate_attached_gas` returns `NearGas::new(0)`, the engine dispatches the connector promise with 0 gas and the full user deposit attached. The promise fails OOG; NEAR protocol refunds the deposit to the engine contract (the receipt's `predecessor_id`), not the original user. No failure callback or refund path exists in the engine, permanently locking the user's NEAR.

---

### Finding Description

**`calculate_attached_gas` silently returns 0** [1](#0-0) 

The function computes `required_gas = used_gas + 2 TGas`. If `required_gas >= prepaid_gas`, it returns `NearGas::new(0)` with no error. The TODO comment on line 587 explicitly acknowledges this is wrong:

```rust
// TODO: Return `Result` with an error about lacking of gas instead.
fn calculate_attached_gas<E: Env>(env: &E) -> NearGas {
    let required_gas = env.used_gas().saturating_add(GAS_FOR_PROMISE_CREATION);
    if required_gas >= env.prepaid_gas() {
        NearGas::new(0)
    } else {
        env.prepaid_gas() - required_gas
    }
}
```

**`return_promise` dispatches the promise unconditionally** [2](#0-1) 

`return_promise` builds a `PromiseCreateArgs` with `attached_gas: calculate_attached_gas(env)` (potentially 0) and `attached_balance: deposit` (the user's full NEAR). It calls `promise_create_call` then `promise_return` with no failure callback.

**`storage_deposit` forwards the full user deposit** [3](#0-2) 

```rust
return_promise(
    io, env,
    "engine_storage_deposit",
    args,
    Yocto::new(env.attached_deposit()),  // full user deposit forwarded
)
```

**`promise_create_call` passes 0 gas directly to the NEAR host** [4](#0-3) 

No validation that `gas > 0` is performed before calling `exports::promise_create`. The 0 value is passed verbatim to the NEAR runtime.

**`GAS_FOR_PROMISE_CREATION` is 2 TGas** [5](#0-4) 

This is the constant subtracted from `prepaid_gas`. Any user who submits with `prepaid_gas` in the range `(X, X + 2 TGas]` — where X is the gas consumed by `storage_deposit` before `calculate_attached_gas` is evaluated — will trigger the 0-gas path.

**NEAR protocol behavior on 0-gas receipt failure:**
NEAR protocol allows `promise_create` with 0 gas at the API level (no host-side validation). When the receipt executes, it fails immediately (OOG). On failure, NEAR refunds the attached deposit to the `predecessor_id` of the receipt — which is the engine contract, not the original user. The engine contract has no mechanism to track or forward this refund.

---

### Impact Explanation

The user's attached NEAR (e.g., 1 NEAR for a storage deposit) is permanently locked in the engine contract's balance. There is no callback, no refund promise, and no admin method to recover it. This is **permanent freezing of funds**.

---

### Likelihood Explanation

The attack is fully user-controlled. The attacker:
1. Measures (or calculates deterministically) the gas X consumed by `storage_deposit` before `calculate_attached_gas` is called.
2. Submits a `storage_deposit` call with non-zero NEAR attached and `prepaid_gas` set to any value in `(X, X + 2 TGas]`.

NEAR protocol imposes no minimum prepaid gas floor that would prevent this. The window is 2 TGas wide, which is easily targeted. This can also be triggered accidentally by users who submit with low gas.

---

### Recommendation

Implement the acknowledged TODO: change `calculate_attached_gas` to return `Result<NearGas, ContractError>` and propagate the error up through `return_promise` and `storage_deposit`. When there is insufficient gas, the engine's execution should fail before the promise is dispatched, causing NEAR protocol to refund the attached deposit to the original caller automatically.

---

### Proof of Concept

```
1. Determine X = gas consumed by storage_deposit before calculate_attached_gas is called
   (deterministic; can be measured with a dry-run or estimated from NEAR gas tables)

2. Submit: storage_deposit(account_id: "victim.near")
           attached_deposit = 1_000_000_000_000_000_000_000_000 yoctoNEAR (1 NEAR)
           prepaid_gas = X + 1_000_000_000_000 (X + 1 TGas)

3. Inside calculate_attached_gas:
   required_gas = X + 2_000_000_000_000
   X + 2 TGas > X + 1 TGas  →  returns NearGas::new(0)

4. return_promise dispatches engine_storage_deposit with:
   attached_gas     = 0
   attached_balance = 1 NEAR

5. Receipt executes on connector; fails OOG immediately.

6. NEAR protocol refunds 1 NEAR to engine contract (predecessor_id of receipt).

7. User's 1 NEAR is permanently locked in engine contract.
   No callback fires. No refund path exists.
```
