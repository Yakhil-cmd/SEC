### Title
Unprivileged Client Can Trigger Unconditional `todo!()` Panic via Hash-Only `/block` Request in `RosettaBlocksMode::Enabled` — (`rs/rosetta-api/icp/src/request_handler.rs`)

### Summary

When the ICP Rosetta node is running with `RosettaBlocksMode::Enabled`, any unauthenticated client can send a `/block` POST request containing only a block hash (no index). If `get_verified_block_by_hash` returns `Err(ApiError::InvalidBlockId(_, _))` — which happens whenever the hash does not correspond to a verified (non-Rosetta) block — the code unconditionally hits `todo!("Fetching Rosetta Blocks by hash is not supported yet")` at line 359, causing a Rust panic.

### Finding Description

The `get_block` function in `RosettaRequestHandler` has three match arms for `PartialBlockIdentifier`. The hash-only arm (`index: None, hash: Some(hash)`) at lines 349–366 contains incomplete logic:

```rust

    if self.is_rosetta_blocks_mode_enabled().await {
        match self.get_verified_block_by_hash(&hash).await {
            Ok(block) => Ok(block),
            Err(ApiError::InvalidBlockId(_, _)) => {
                todo!("Fetching Rosetta Blocks by hash is not supported yet")  // line 359
            }
            e => e,
        }
    } else {
        self.get_verified_block_by_hash(&hash).await

}
```

`get_verified_block_by_hash` returns `Err(ApiError::InvalidBlockId(true, Default::default()))` at line 459 whenever the hash is not found as a verified block. This is the normal outcome for any hash belonging to a Rosetta block (the entire purpose of the mode). The `todo!()` macro unconditionally panics with no recovery path.

The trigger condition is straightforward:
1. Rosetta node started with `--enable-rosetta-blocks` (sets `RosettaBlocksMode::Enabled`)
2. Client POSTs to `/block` with `{ "block_identifier": { "hash": "<any_hash>" } }` (no `index` field)
3. The hash does not match a pre-Rosetta verified block (e.g., it is a Rosetta block hash, or simply any unknown hash that resolves to `InvalidBlockId`) [5](#0-4) 

### Impact Explanation

`todo!()` in Rust expands to `panic!("not yet implemented: ...")`. In actix-web (the framework used here), panics in async handlers propagate to the worker thread. Depending on actix-web's internal panic handling:

- **Minimum impact**: The request panics, actix-web catches it via `catch_unwind`, and returns HTTP 500. Every hash-only block request in `RosettaBlocksMode::Enabled` that doesn't match a verified block returns 500 instead of a proper response — making the hash-only lookup feature completely broken and unusable.
- **Maximum impact**: The worker thread crashes and is restarted by actix-web. Repeated requests cause repeated worker restarts, degrading or denying service to all Rosetta API users.

The Rosetta node is used for ICP ledger operations (balance queries, transfers via Coinbase/exchange integrations). Disrupting it affects ledger accessibility for downstream users and integrators. [6](#0-5) 

### Likelihood Explanation

- **Condition**: `RosettaBlocksMode::Enabled` requires the `--enable-rosetta-blocks` feature flag at startup. This is non-default but is a documented, production-intended feature.
- **Trigger**: Any client sending a syntactically valid `/block` request with only a hash field. No authentication, no special knowledge required. The hash can be arbitrary (unknown hashes also produce `InvalidBlockId` via `BlockStoreError::Other("Hash Not Found")` propagation).
- **Reproducibility**: 100% deterministic once the mode is enabled. [7](#0-6) 

### Recommendation

Replace the `todo!()` with a proper implementation or a graceful error return:

```rust
Err(ApiError::InvalidBlockId(_, _)) => {
    // Attempt to look up as a Rosetta block by hash
    // For now, return a proper error instead of panicking:
    Err(ApiError::InvalidBlockId(false, "Fetching Rosetta Blocks by hash is not yet supported".into()))
}
```

Long-term, implement the Rosetta block hash lookup using the existing `rosetta_blocks` table.

    async fn get_verified_block_by_hash(
        &self,
        block_hash: &str,
    ) -> Result<rosetta_core::objects::Block, ApiError> {
        let hash = convert::to_hash::<ic_ledger_core::block::EncodedBlock>(block_hash)?;
        let block = {
            let blocks = self.ledger.read_blocks().await;
            if !blocks.is_verified_by_hash(&hash)? {
                return Err(ApiError::InvalidBlockId(true, Default::default()));
            }
            let block_index = blocks.get_block_idx_by_block_hash(&hash)?;
            blocks.get_hashed_block(&block_index)
        }?;
        self.hashed_block_to_rosetta_core_block(block).await
    }

    pub fn get_block_idx_by_block_hash(
        connection: &mut Connection,
        hash: &HashOf<EncodedBlock>,
    ) -> Result<u64, BlockStoreError> {
        let mut stmt = connection
            .prepare_cached("SELECT block_idx from blocks where block_hash = ?")
            .map_err(|e| BlockStoreError::Other(e.to_string()))
            .unwrap();
        let block_idx = stmt
            .query_map(params![&hash.into_bytes().to_vec()], |row| {
                Ok(row.get(0).unwrap())
            })
            .unwrap()
            .next()

            .map(|block| block.unwrap())?;
        Ok(block_idx)
