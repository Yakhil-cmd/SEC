### Title
Unchecked Precompile Return Value After Token Burn Causes Permanent Fund Loss — (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` burn a user's ERC-20 tokens **before** calling the `ExitToNear` or `ExitToEthereum` precompile, and **never check the return value** of that call. If the precompile call fails for any reason, the tokens are permanently destroyed on the EVM side with no corresponding NEP-141 release on NEAR. A concrete, systemic trigger exists: `EvmErc20V2.sol` encodes a `sender` address into the precompile input that the `ExitToNear` precompile does not expect when the `error_refund` feature is disabled, causing every `withdrawToNear` call on V2 to fail silently.

---

### Finding Description

**Root cause 1 — Unchecked call return value (both contracts, both withdraw functions)**

In `EvmErc20.sol` lines 60–62 and `EvmErc20V2.sol` lines 61–63, the precompile is invoked via inline assembly:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
}
```

`res` is assigned but never read. If the precompile returns an error (`ExitError`), the EVM `call` opcode returns `0` (failure) but does **not** revert the caller's state. The `_burn` executed on the line immediately before has already modified the ERC-20 balance, and that change is committed. The function returns normally, giving the user no indication that the withdrawal failed.

The same pattern appears in `withdrawToEthereum` in both contracts.

**Root cause 2 — Input format mismatch in `EvmErc20V2.withdrawToNear`**

`EvmErc20V2.sol` lines 57–59 encode the sender address into the precompile input:

```solidity
bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
uint input_size = 1 + 20 + 32 + recipient.length;
```

Format sent: `[0x01][sender_20bytes][amount_32bytes][recipient_bytes]`

The `ExitToNear` precompile's `TryFrom` implementation in `engine-precompiles/src/native.rs` lines 787–791 (when the `error_refund` feature is **not** compiled in) strips only the flag byte:

```rust
#[cfg(not(feature = "error_refund"))]
fn parse_input(input: &[u8]) -> Result<&[u8], ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    Ok(&input[1..])
}
```

Then for flag `0x1` (lines 758–763), it reads `input[..32]` as the amount:

```rust
0x1 => {
    let amount = parse_amount(&input[..32])?;
    let Recipient { receiver_account_id, message } = parse_recipient(&input[32..])?;
```

When V2's input is processed without `error_refund`, `input[..32]` is `[sender_20bytes || first_12_bytes_of_amount]`. This 256-bit value has the sender address in its high 160 bits, making it far exceed `u128::MAX`. `parse_amount` (lines 337–344) enforces:

```rust
if amount > U256::from(u128::MAX) {
    return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
}
```

The precompile returns `ERR_INVALID_AMOUNT`. The EVM `call` returns `0`. The Solidity code ignores it. The user's tokens are gone.

By contrast, `EvmErc20.sol` (V1) sends `[0x01][amount_32bytes][recipient]` — the correct format without `error_refund`. V2's format is only correct when `error_refund` **is** compiled in (lines 778–785), where the precompile reads `input[1..21]` as a refund address before the amount.

---

### Impact Explanation

**Permanent freezing of funds (Critical).**

When `EvmErc20V2.withdrawToNear` is called on a deployment where the `error_refund` feature is not enabled:

1. `_burn(sender, amount)` destroys the user's ERC-20 tokens.
2. The precompile call fails with `ERR_INVALID_AMOUNT`.
3. No NEAR promise is created; the NEP-141 tokens remain locked in the connector.
4. The user's ERC-20 balance is zero; the NEAR balance is never credited.
5. The tokens are irrecoverably lost — there is no refund path, no retry, and no revert.

This affects every user who calls `withdrawToNear` on a V2 deployment without `error_refund`, making it a systemic, not edge-case, loss.

---

### Likelihood Explanation

**High.** The mismatch is structural: V2 unconditionally includes the sender address in the precompile input regardless of the feature flag. Any production deployment of V2 without `error_refund` will silently destroy tokens on every `withdrawToNear` call. The unchecked return value means there is no on-chain signal of failure; users and operators have no way to detect the loss until they notice the NEAR balance was never credited.

---

### Recommendation

1. **Check the precompile call return value and revert on failure** in both `withdrawToNear` and `withdrawToEthereum` in both contracts:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures the burn is atomically rolled back if the precompile fails.

2. **Align the input format with the compiled feature flag.** Either:
   - Remove the `sender` field from V2's `withdrawToNear` input when `error_refund` is not enabled, or
   - Gate the V2 contract's deployment on the `error_refund` feature being present, or
   - Unify the format so both contracts always include the sender (and update the precompile to always expect it).

---

### Proof of Concept

**Step 1.** Deploy `EvmErc20V2` for a registered NEP-141 token on an Aurora instance compiled **without** the `error_refund` feature.

**Step 2.** Call `withdrawToNear(recipient_bytes, amount)` from an account holding tokens.

**Step 3.** Observe:
- `_burn(sender, amount)` executes; the caller's ERC-20 balance decreases by `amount`.
- The precompile at `0xe9217bc7...` receives input `[0x01][sender_20bytes][amount_32bytes][recipient]`.
- `parse_input` strips the flag byte; `parse_amount` reads `[sender_20bytes || first_12_bytes_of_amount]` as a U256 value >> u128::MAX.
- Precompile returns `ERR_INVALID_AMOUNT`; EVM `call` returns `0`.
- The assembly block assigns `res = 0` and falls through; no revert.
- No NEAR promise is emitted; the NEP-141 tokens are never transferred.
- The user has lost `amount` tokens permanently.
