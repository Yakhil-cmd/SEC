### Title
SNS Governance `reject_cost_e8s` Accepts Zero — Proposer Can Self-Pass Proposals at Negligible Cost, Crowding Out Legitimate Governance - (File: rs/sns/governance/src/types.rs)

### Summary
The SNS governance `NervousSystemParameters.reject_cost_e8s` field has no minimum value enforced in its validation logic. An SNS community member who controls a majority of voting power can set `reject_cost_e8s` to `0` via a `ManageNervousSystemParameters` proposal, then continuously submit self-passing proposals (the proposer auto-votes yes) at zero cost. This is the IC analog of the `voterPercent = 0` self-challenge exploit: a dominant token holder can occupy the proposal slot queue at negligible cost, preventing other participants from submitting meaningful proposals, and can pass arbitrary governance actions unilaterally without any economic deterrent.

### Finding Description

In `rs/sns/governance/src/types.rs`, the `validate_reject_cost_e8s` function only checks that the field is `Some(...)` — it does not enforce any minimum value:

```rust
fn validate_reject_cost_e8s(&self) -> Result<u64, String> {
    self.reject_cost_e8s
        .ok_or_else(|| "NervousSystemParameters.reject_cost_e8s must be set".to_string())
}
```

This means `reject_cost_e8s = 0` is a valid parameter. The integration test `test_init_with_sys_params` in `rs/sns/integration_tests/src/nervous_system_parameters.rs` explicitly initializes an SNS with `reject_cost_e8s: Some(0)` and asserts it succeeds.

The `make_proposal` flow in `rs/sns/governance/src/governance.rs` charges `neuron_fees_e8s += proposal_data.reject_cost_e8s` upfront, and the proposer automatically casts a `yes` vote. With `reject_cost_e8s = 0`, the proposer:
1. Pays nothing to submit proposals.
2. Automatically votes yes.
3. If they hold >50% of voting power (or have sufficient following), the proposal passes immediately via `early_decision()`.
4. The `max_number_of_proposals_with_ballots` cap (default 700) is the only throttle, but with zero cost, a dominant holder can continuously cycle through proposals.

The `ManageNervousSystemParameters` proposal action allows any neuron with sufficient voting power to set `reject_cost_e8s` to `0`. The validation in `validate_and_render_manage_nervous_system_parameters` calls `new_parameters.inherit_from(current_parameters).validate()`, which only checks that `reject_cost_e8s` is `Some`, not that it is above any minimum.

### Impact Explanation

**Governance authorization bug / resource accounting bug.** An SNS token holder controlling a majority of voting power can:
1. Submit a `ManageNervousSystemParameters` proposal setting `reject_cost_e8s = 0`.
2. Since they hold majority, the proposal passes immediately (proposer auto-votes yes, triggering `early_decision`).
3. Subsequently submit an unlimited stream of proposals at zero cost, each auto-passing.
4. This occupies the `max_number_of_proposals_with_ballots` queue, preventing other participants from submitting proposals (`ResourceExhausted` error).
5. The dominant holder can pass arbitrary governance actions (treasury transfers, canister upgrades, parameter changes) with no economic friction, undermining the deterrent purpose of `reject_cost_e8s`.

The economic deterrent for spam and governance capture is completely removed. Other token holders cannot submit competing proposals when the queue is full.

### Likelihood Explanation

Any SNS where a single principal or coordinated group holds >50% of voting power (common in early-stage SNS deployments with developer neurons) can exploit this. The attack requires only a standard `ManageNervousSystemParameters` proposal — a normal, unprivileged governance action available to any neuron meeting the dissolve delay requirement. The entry path is fully reachable via the public `manage_neuron` ingress endpoint of the SNS governance canister.

### Recommendation

Enforce a minimum value for `reject_cost_e8s` in `validate_reject_cost_e8s`. The minimum should be at least `transaction_fee_e8s` (already validated for `neuron_minimum_stake_e8s`) or a protocol-defined floor constant, analogous to how `max_proposals_to_keep_per_action` enforces `> 0`. For example:

```rust
fn validate_reject_cost_e8s(&self) -> Result<u64, String> {
    let reject_cost_e8s = self.reject_cost_e8s
        .ok_or_else(|| "NervousSystemParameters.reject_cost_e8s must be set".to_string())?;
    let transaction_fee_e8s = self.transaction_fee_e8s.unwrap_or(0);
    if reject_cost_e8s < transaction_fee_e8s {
        return Err(format!(
            "reject_cost_e8s ({reject_cost_e8s}) must be >= transaction_fee_e8s ({transaction_fee_e8s})"
        ));
    }
    Ok(reject_cost_e8s)
}
```

Additionally, consider a hard floor constant (e.g., `MIN_REJECT_COST_E8S`) independent of `transaction_fee_e8s`.

### Proof of Concept

**Step 1:** Deploy an SNS where the attacker controls >50% of voting power (e.g., developer neurons at genesis).

**Step 2:** Submit a `ManageNervousSystemParameters` proposal via `manage_neuron` ingress:
```
reject_cost_e8s: Some(0)
```
The proposer auto-votes yes; with >50% voting power, `early_decision()` returns `Vote::Yes` immediately. The proposal executes, setting `reject_cost_e8s = 0`.

**Step 3:** Continuously submit proposals (e.g., `Motion` or `TransferSnsTreasuryFunds`) at zero cost. Each proposal auto-votes yes and passes immediately if the attacker holds majority. The `neuron_fees_e8s` charged is `0`, so the attacker's stake is never reduced.

**Step 4:** The `max_number_of_proposals_with_ballots` queue fills. Other participants receive `ResourceExhausted` when attempting to submit proposals.

**Root cause:** `validate_reject_cost_e8s` in `rs/sns/governance/src/types.rs` line 597–600 only checks `is_some()`, not a minimum value. The `ManageNervousSystemParameters` validation path at `rs/sns/governance/src/proposal.rs` line 536 calls `validate()` which delegates to this insufficient check.
