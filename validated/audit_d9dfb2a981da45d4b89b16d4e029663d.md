### Title
Truncating Division in `native_per_pubdata` Computation Undercharges Users for Pubdata — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` is computed using floor (truncating) integer division, while the analogous `native_per_gas` uses ceiling division. This systematic rounding in the user's favor causes every transaction with pubdata to pay less native resource — and therefore less gas fee — than the actual proving cost of that pubdata. The operator/protocol silently absorbs the shortfall.

---

### Finding Description

In `validate_and_compute_fee_for_transaction`, the two key resource-pricing ratios are computed as follows:

**`native_per_gas`** — uses `div_ceil` (rounds **up**, protocol's favor):

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [1](#0-0) 

**`native_per_pubdata`** — uses `wrapping_div` (rounds **down**, user's favor):

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [2](#0-1) 

The same truncating division appears in the public API helper:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

`native_per_pubdata` is then used in two places:

1. **Intrinsic pubdata overhead** — `native_per_pubdata_byte.saturating_mul(intrinsic_pubdata)` in `create_resources_for_tx`. [4](#0-3) 

2. **Post-execution pubdata charge** — `current_pubdata_spent.checked_mul(native_per_pubdata)` in `get_resources_to_charge_for_pubdata`. [5](#0-4) 

Because `native_per_pubdata` is truncated down, the native resource charged for pubdata is lower than the true cost. This flows into `compute_gas_refund`:

```rust
let native_used = full_native_limit.saturating_sub(resources.native().remaining().as_u64());
// ...
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
``` [6](#0-5) 

A lower `native_used` (from the truncated pubdata charge) produces a lower `delta_gas`, which produces a lower final `gas_used`, which means the user pays less fee to the operator.

---

### Impact Explanation

For every pubdata byte, the user is undercharged by up to `(native_price − 1)` native units. Concretely:

- If `pubdata_price = 10`, `native_price = 3`: correct `native_per_pubdata = ceil(10/3) = 4`, actual = `floor(10/3) = 3`.
- Per pubdata byte, 1 native unit is not charged.
- With `native_per_gas = 100` and 100,000 pubdata bytes: `100,000 / 100 = 1,000` gas units of fee are silently waived per transaction.

Across many transactions with large pubdata (e.g., storage-heavy contracts), the cumulative underpayment to the operator is material. The operator bears the proving cost of pubdata without receiving the corresponding fee. This is a direct resource-accounting bug in the protocol's favor of the user, analogous to the HomoraBank debt truncation.

---

### Likelihood Explanation

`pubdata_price` and `native_price` are operator-set block-level parameters. They are almost never exactly divisible. The truncation fires on virtually every transaction that writes storage (i.e., generates pubdata). The attacker entry path requires no special privilege: any unprivileged L2 transaction sender submitting storage-writing transactions triggers this path.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates in user's favor):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, protocol's favor):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427.

---

### Proof of Concept

**Setup:**
- `pubdata_price = 10`, `native_price = 3`
- `native_per_pubdata` (actual) = `10 / 3 = 3` (truncated)
- `native_per_pubdata` (correct) = `ceil(10/3) = 4`

**Transaction with 90,000 pubdata bytes:**
- Actual native charged for pubdata: `90,000 × 3 = 270,000`
- Correct native charged for pubdata: `90,000 × 4 = 360,000`
- Shortfall: `90,000` native units

**With `native_per_gas = 100`:**
- Gas underpaid: `90,000 / 100 = 900` gas units per transaction
- At `gas_price = 1,000 wei/gas`: `900,000 wei` fee waived per transaction

A user submitting many storage-heavy transactions (e.g., a contract that writes 90,000 bytes of pubdata per call) systematically underpays the operator for proving costs. The operator receives less fee than the actual cost of proving the pubdata, with the shortfall growing linearly with pubdata volume and transaction count.
