Audit Report

## Title
Silently Discarded `burn_and_log` Result in `process_create_canister` Allows Cycles Minting Without ICP Burn — (File: `rs/nns/cmc/src/main.rs`)

## Summary
In the Cycles Minting Canister, `process_create_canister` calls `burn_and_log` after successfully creating a canister, but `burn_and_log` is explicitly designed to swallow all errors and return `()`. If the ICP ledger call inside `burn_and_log` fails (e.g., during a ledger upgrade), cycles are minted and the canister is created, but the corresponding ICP is never burned. The notification block index is simultaneously recorded as permanently processed (`NotifiedCreateCanister(Ok(...))`), preventing any future retry. The ICP/cycles conservation invariant is permanently broken for that transaction. The same pattern also affects `process_top_up` and `process_mint_cycles`.

## Finding Description
In `rs/nns/cmc/src/main.rs`, `process_create_canister` (L1925–1956) follows this sequence:

1. `tokens_to_cycles(amount)` — converts ICP to cycles.
2. `do_create_canister(...)` — calls the IC management canister to create a new canister and credit it with cycles. This call is **independent of the ICP ledger** and succeeds during a ledger upgrade.
3. On `Ok(canister_id)`: calls `burn_and_log(sub, amount).await` and discards the result, then returns `Ok(canister_id)`.

`burn_and_log` (L2014–2048) is explicitly designed to swallow errors. The comment at L2014–2016 reads: *"Burning doesn't return errors — we don't want to reject the transaction notification because then it could be retried."* When `call_protobuf(ledger_canister_id, "send_pb", send_args)` fails, the error is only printed; the caller receives `()`.

Back in `notify_create_canister` (L1411–1418), the result `Ok(canister_id)` is not a transient error, so `is_transient_error` returns false and the block index is permanently written as `NotificationStatus::NotifiedCreateCanister(Ok(canister_id))`. Any subsequent call with the same block index returns the cached `Ok` result without re-attempting the burn.

The ICP remains in CMC's subaccount for the controller's principal — not burned, not refunded, and not recoverable by the user (the notification is closed). No pending-burns queue or recovery mechanism exists in the codebase.

## Impact Explanation
Cycles are minted and a canister is created without the backing ICP being destroyed. This breaks the fundamental ICP/cycles conservation invariant: every unit of cycles must correspond to burned ICP. The ICP is permanently stranded in CMC's subaccount. Repeated occurrences (e.g., across multiple ledger upgrade windows) accumulate unburned ICP while the cycles supply grows unbacked. This constitutes illegal minting and a ledger conservation violation — a significant NNS/CMC financial integrity impact with concrete protocol harm, matching the High impact criterion for significant ledger/NNS security impact with concrete user or protocol harm.

## Likelihood Explanation
ICP ledger upgrades are publicly observable NNS governance events. During the upgrade window (typically seconds to a few minutes), inter-canister calls to the ledger are rejected with a transient error. An unprivileged user can:
- Pre-stage the ICP transfer to CMC's subaccount before the upgrade.
- Submit `notify_create_canister` timed to the upgrade window (observable via NNS proposal execution).

No privileged access, key material, or majority corruption is required. The user does not need to profit — the conservation invariant is violated regardless of attacker motivation. The same window affects `notify_top_up` and `notify_mint_cycles` identically via the same `burn_and_log` pattern.

## Recommendation
`burn_and_log` should be refactored to return a `Result`. On burn failure after a successful canister creation/top-up/mint, `process_create_canister` (and its siblings) should either:
1. Store the failed burn in a persistent pending-burns queue that a heartbeat or retry mechanism can drain, or
2. Treat a burn failure as a transient error (by removing the block from `blocks_notified`), allowing the notification to be retried — accepting the risk of a duplicate-notification attempt, which can be mitigated by idempotency checks on the ledger side.

The current design conflates two separate concerns — preventing notification replay and ensuring the burn completes — and resolves the tension by silently sacrificing conservation correctness.

## Proof of Concept
1. Observe an upcoming ICP ledger upgrade proposal on the NNS dashboard.
2. Send N ICP (e.g., 10 ICP) to `AccountIdentifier(CMC_principal, Subaccount::from(&user_principal))` with memo `MEMO_CREATE_CANISTER`.
3. Note the resulting block index B.
4. At the moment the ledger upgrade executes (calls to ledger are rejected), submit `notify_create_canister { block_index: B, controller: user_principal, ... }`.
5. CMC executes `do_create_canister` — succeeds (calls IC management canister, not ledger).
6. CMC executes `burn_and_log` — `call_protobuf` to `send_pb` returns `Err((code, msg))`; error is printed; `()` is returned.
7. `process_create_canister` returns `Ok(canister_id)`.
8. `blocks_notified[B]` is set to `NotifiedCreateCanister(Ok(canister_id))` permanently.
9. **Observable result**: canister exists with full cycles allocation; 10 ICP sits unburned in CMC's subaccount; total cycles supply is inflated relative to burned ICP. A deterministic PocketIC or StateMachine integration test can reproduce this by intercepting or failing the ledger `send_pb` call after `do_create_canister` succeeds.
