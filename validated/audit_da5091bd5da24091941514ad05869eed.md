Audit Report

## Title
Integer Overflow in `construction_payloads` While-Loop Causes Unbounded Iteration / DoS - (File: `rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary
The `construction_payloads` function in the ICRC1 Rosetta API performs raw `u64` arithmetic on user-controlled `ingress_start` and `ingress_end` values. When an attacker supplies values near `u64::MAX`, both input validation checks pass, and the while-loop that builds `ingress_expiries` overflows and wraps on the first iteration, causing `ingress_start` to reset to a small value. The loop then runs for approximately 154 million iterations, allocating over 1 GB of memory and consuming 100% CPU, effectively denying service to the ICRC1 Rosetta API process.

## Finding Description

**Constants** (confirmed from source):
- `MAX_INGRESS_TTL = 300s`, `PERMITTED_DRIFT = 60s` вЖТ `ingress_interval = 240 √Ч 10вБє ns`
- `INGRESS_INTERVAL_OVERLAP = 120s` вЖТ step per iteration = `120 √Ч 10вБє ns`

**Vulnerable loop** at `rs/rosetta-api/icrc1/src/construction_api/services.rs` lines 163вАУ167:
```rust
while ingress_start < ingress_end {
    ingress_expiries.push(ingress_start + ingress_interval);   // raw u64 add
    ingress_start +=
        ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
    // ^^^ += on u64 also wraps in release builds
}
```
Both additions are bare `u64` ops with no overflow guard.

**Exploit inputs**: `ingress_start = u64::MAX вИТ 1`, `ingress_end = u64::MAX`

**Check 1** (line 148): `ingress_start >= ingress_end` вЖТ `u64::MAXвИТ1 >= u64::MAX` вЖТ **false** вЖТ passes.

**Check 2** (line 154): `ingress_end < now + ingress_interval` вЖТ `u64::MAX < ~1.75√Ч10¬євБЄ + 2.4√Ч10¬є¬є` вЖТ `1.844√Ч10¬євБє < ~1.75√Ч10¬євБЄ` вЖТ **false** вЖТ passes.

**Loop iteration 1** (release build, wrapping arithmetic):
- `push((u64::MAXвИТ1) + 240_000_000_000)` вЖТ wraps to `239_999_999_998`
- `ingress_start += 120_000_000_000` вЖТ `(u64::MAXвИТ1) + 120_000_000_000` wraps to `119_999_999_998`

**Loop condition**: `119_999_999_998 < u64::MAX` вЖТ **true** вЖТ continues.

From this point `ingress_start вЙИ 1.2√Ч10¬є¬є` and increments by `1.2√Ч10¬є¬є` per iteration until it reaches `u64::MAX вЙИ 1.844√Ч10¬євБє`. Total iterations вЙИ `1.844√Ч10¬євБє / 1.2√Ч10¬є¬є вЙИ 154 million`. Each iteration pushes 8 bytes to the `ingress_expiries` vector, consuming **~1.23 GB** of heap memory.

The ICP Rosetta path wraps `ingress_start` in `ic_types::time::Time`, whose `Add<Duration>` delegates to `std::time::Duration` addition, which **panics** on overflow in both debug and release builds rather than silently wrapping. The ICRC1 path has no equivalent protection.

## Impact Explanation
A single unauthenticated HTTP POST to `/construction/payloads` causes the ICRC1 Rosetta handler thread to allocate ~1.23 GB and spin for ~154 million iterations before returning. During this time the process is unresponsive to all other requests, completely blocking ICRC1 token transfer operations routed through this node. This is a concrete, non-volumetric application-level DoS of a Rosetta financial integration component, matching the **High** bounty impact: *"Application/platform-level DoS вА¶ or subnet availability impact not based on raw volumetric DDoS"* and *"Significant вА¶ Rosetta вА¶ security impact with concrete user or protocol harm."*

## Likelihood Explanation
The `/construction/payloads` endpoint is unauthenticated and publicly reachable. The attack requires a single HTTP request with two crafted integer fields (`ingress_start`, `ingress_end`). No keys, credentials, or prior state are needed. Any external party aware of the Rosetta API specification can trigger this. The attack is repeatable: after the process recovers (OOM kill or restart), a new request immediately re-triggers it.

## Recommendation
Replace the raw `u64` additions inside the loop with checked or saturating operations, and add an explicit upper-bound guard before entering the loop:

```rust
// Guard before the loop
if ingress_start > u64::MAX - ingress_interval {
    return Err(Error::processing_construction_failed(
        "ingress_start too large: would overflow when adding ingress_interval",
    ));
}

while ingress_start < ingress_end {
    let expiry = ingress_start
        .checked_add(ingress_interval)
        .ok_or_else(|| Error::processing_construction_failed("ingress expiry overflow"))?;
    ingress_expiries.push(expiry);
    ingress_start = ingress_start
        .saturating_add(ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64));
}
```

Additionally, add an explicit upper-bound validation on `ingress_end` (e.g., reject values above a reasonable future horizon such as `now + 24 * ingress_interval`) to prevent the loop from running an unreasonably large but non-overflowing number of iterations.
