Audit Report

## Title
Unbounded `neuron_ids` and `neuron_subaccounts` in `list_neurons` Eagerly Materializes All Pages, Enabling Instruction Exhaustion DoS - (File: `rs/nns/governance/src/governance.rs`)

## Summary
The NNS governance `list_neurons` endpoint accepts caller-supplied `neuron_ids: Vec<u64>` and `neuron_subaccounts: Option<Vec<NeuronSubaccount>>` with no enforced size cap. Before returning a single page, the implementation eagerly materializes all pages into a `Vec<Vec<NeuronId>>`. An unprivileged caller can supply the maximum-sized vector allowed by the 2 MB ingress limit (~262,000 entries) to force O(N log N) BTreeSet construction and O(N) full-chunk materialization per call, with the `neuron_subaccounts` path additionally triggering up to ~62,500 stable-memory lookups per call.

## Finding Description
In `rs/nns/governance/src/governance.rs`, the `list_neurons` function (line 1614) destructures the caller-supplied `ListNeurons` struct with no prior size validation on `neuron_ids` or `neuron_subaccounts`.

`page_size` is capped at `MAX_LIST_NEURONS_RESULTS` (500), but no corresponding cap exists for `neuron_ids`: [1](#0-0) 

The `neuron_subaccounts` field is processed without any size limit, with each entry triggering a `get_neuron_id_for_subaccount` stable-memory lookup: [2](#0-1) 

All supplied `neuron_ids` are inserted into a `BTreeSet` (O(N log N)), and then **all** chunks are eagerly materialized into a `Vec<Vec<NeuronId>>` before the single requested page is accessed: [3](#0-2) 

Only after all chunks are materialized is the single page accessed via `.get(page_number as usize)`. The `ListNeurons` type definition confirms `neuron_ids` is an unbounded `Vec<u64>`: [4](#0-3) 

The Candid interface exposes both fields without any declared length bound: [5](#0-4) 

## Impact Explanation
This matches **High ($2,000–$10,000): Application/platform-level DoS, or subnet availability impact not based on raw volumetric DDoS**. The NNS governance canister is a critical system canister. Forcing it to perform O(N log N) BTreeSet construction plus O(N) chunk materialization for every query call — and additionally O(N) stable-memory lookups via the `neuron_subaccounts` path — wastes instruction budget proportional to the total input size rather than the requested page size. Stable-memory reads are significantly more expensive than heap reads; at realistic per-lookup costs, 62,500 stable-memory lookups can approach or exceed the 5-billion-instruction query limit, causing the call to trap with `CanisterInstructionLimitExceeded`. Even below the trap threshold, repeated concurrent calls degrade governance canister responsiveness for all legitimate users.

## Likelihood Explanation
`list_neurons` is a publicly callable query and update endpoint on the NNS governance canister — no special role, neuron ownership, or privileged key is required. Any boundary-node user or canister can craft a maximum-size ingress message with a large `neuron_ids` or `neuron_subaccounts` vector and submit it repeatedly. The cost to the attacker is only the ingress fee; the cost to the canister scales with N.

## Recommendation
1. **Cap `neuron_ids` before processing** — reject or truncate requests where `neuron_ids.len()` exceeds `MAX_LIST_NEURONS_RESULTS` (or a separately defined ingress cap).
2. **Lazy page computation** — replace the eager `.collect()` of all chunks with a lazy skip-and-take so only the requested page is ever materialized:
   ```rust
   let current_page: Vec<NeuronId> = requested_neuron_ids
       .into_iter()
       .skip(page_number as usize * page_size as usize)
       .take(page_size as usize)
       .collect();
   ```
3. **Cap `neuron_subaccounts`** — apply the same size limit to the `neuron_subaccounts` field to bound the number of stable-memory lookups per call.

## Proof of Concept
Send the following ingress call to the NNS governance canister from any principal:

```
list_neurons(record {
    neuron_ids       = vec { 1; 2; 3; … /* 262 000 distinct u64 values */ };
    include_neurons_readable_by_caller = false;
    page_number      = opt 0;
    page_size        = opt 500;
    neuron_subaccounts = opt vec {};
    include_empty_neurons_readable_by_caller = opt false;
    include_public_neurons_in_full_neurons   = opt false;
})
```

The canister will insert all 262,000 IDs into a `BTreeSet`, then materialize ~524 `Vec<NeuronId>` chunks, and finally return only the first 500-entry page. For the `neuron_subaccounts` variant, replace `neuron_ids` with an empty vec and supply ~62,500 `NeuronSubaccount` entries (each 32 bytes), triggering 62,500 stable-memory lookups before the same chunking path runs. A deterministic integration test using PocketIC can confirm instruction counts by measuring the cycles consumed per call as N scales from 1 to 262,000.
