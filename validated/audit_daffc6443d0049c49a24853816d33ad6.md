Audit Report

## Title
Cache Stampede / Thundering-Herd in `subnet_read_state_cache_middleware` Allows Upstream Replica Connection Pool Exhaustion — (`rs/boundary_node/ic_boundary/src/http/middleware/subnet_read_state_cache.rs`)

## Summary

`subnet_read_state_cache_middleware` performs a non-atomic `cache.get()` + `cache.insert()` sequence with no in-flight deduplication. N concurrent requests sharing the same `(subnet_id, paths)` cache key that all arrive before any response is stored will each observe a cache miss and independently forward to the upstream replica, defeating the cache's protective purpose. The subnet read_state routes carry no per-IP or per-subnet rate limiting, removing the primary mitigation that exists for call routes.

## Finding Description

The middleware at `rs/boundary_node/ic_boundary/src/http/middleware/subnet_read_state_cache.rs` performs:

1. **Line 122** — `state.cache.get(&cache_key)`: non-blocking read from `moka::sync::Cache`. On miss, execution falls through.
2. **Line 130** — `next.run(request).await`: request forwarded to upstream replica (full round-trip latency).
3. **Line 143** — `state.cache.insert(cache_key, cached.clone())`: entry stored only after the upstream round-trip completes.

Between steps 1 and 3 there is a window equal to the full upstream latency. Any concurrent request calling `cache.get()` during this window also sees a miss and independently calls `next.run()`. There is no in-flight deduplication, no mutex, and no compute-once primitive.

`moka::sync::Cache` (used here) does not provide an async-safe `get_with`. The correct fix requires switching to `moka::future::Cache`, which exposes an async `get_with` / `entry_by_ref().or_insert_with_if()` API that serializes concurrent fetches for the same key. This pattern is already used in the same codebase at `rs/boundary_node/ic_boundary/src/http/handlers.rs`, confirming developer awareness.

The `subnet_layers` stack (core.rs lines 1033–1040) wraps subnet read_state routes but does **not** include the IP or subnet rate-limiting layers that are conditionally applied to `canister_call_routes` (core.rs lines 867–876). The default TTL is 30 seconds (cli.rs line 352), so each 30-second window can be targeted with a fresh stampede burst.

## Impact Explanation

Every boundary-node async task handling one of the N concurrent requests independently opens a connection to the upstream replica and awaits a full response. This exhausts the upstream replica's connection pool or triggers its internal rate limiter, degrading or blocking service for all users of that subnet. The cache is specifically designed to shield replicas from repeated identical requests; the stampede nullifies that shield entirely. Because subnet read_state paths (`canister_ranges`, `subnet`) are well-known and identical across all boundary-node clients, a single attacker can target the highest-traffic cache key with maximum effect.

This matches the allowed High impact: **Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS** — the attack exploits a specific application-level code flaw (missing in-flight deduplication) to bypass a protective control, causing disproportionate upstream load from a small, timed burst rather than requiring sustained volumetric traffic.

## Likelihood Explanation

The endpoint is fully public and unauthenticated. The attack requires only sending N HTTP requests concurrently timed to arrive before the first response is cached — trivially achievable with any HTTP client library. No privileged access, key material, governance majority, or network-level attack is required. The absence of per-IP rate limiting on subnet read_state routes removes the primary mitigation present on call routes. The attack is repeatable every TTL window (default 30 seconds).

## Recommendation

Switch `SubnetReadStateCacheState` from `moka::sync::Cache` to `moka::future::Cache` and replace the `get` + `insert` pattern with `moka::future::Cache::get_with` (or `entry_by_ref().or_insert_with_if()`), which guarantees that only one in-flight upstream request is made per cache key at a time:

```rust
// Switch field type:
cache: moka::future::Cache<CacheKey, Response<Bytes>>,

// Replace get+insert with:
let cached = state.cache.get_with(cache_key, async {
    // forward to upstream exactly once per key
    next.run(request).await
    // ... buffer body, construct Response<Bytes> ...
}).await;
```

Additionally, apply per-IP rate limiting to subnet read_state routes consistent with the protection already applied to `canister_call_routes`.
