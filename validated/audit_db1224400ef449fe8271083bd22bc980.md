### Title
Silent U256→u128 Truncation in Legacy Exit Precompile Paths Causes Permanent Fund Loss - (File: engine-precompiles/src/native.rs)

### Summary

The `ExitToNear` and `ExitToEth` precompiles silently truncate ERC-20 token amounts from `U256` to `u128` via `.as_u128()` in their legacy JSON-encoding paths, without any overflow guard. An EVM user who burns ERC-20 tokens with an amount exceeding `u128::MAX` will have their tokens permanently destroyed on Aurora while only the low-128-bit remainder is forwarded to the NEAR-side NEP-141 contract. The inconsistency is confirmed by the fact that the Omni path (`ft_transfer_call_args`) already contains an explicit overflow check that the legacy paths lack.

---

### Finding Description

In `engine-precompiles/src/native.rs`, three distinct legacy encoding sites call `U256::as_u128()` on an attacker-supplied amount without first verifying that the value fits in 128 bits:

**Site 1 – `json_args` (ExitToEth, JSON serialization path)** [1](#0-0) 

```rust
fn json_args(address: Address, amount: U256) -> Result<Vec<u8>, ExitError> {
    Ok(format!(
        r#"{{"amount":"{}","recipient":"{}"}}"#,
        amount.as_u128(),   // ← silent truncation, no overflow check
        address.encode(),
    )
    .into_bytes())
}
```

**Site 2 – `borsh_args` (ExitToEth, Borsh serialization path)** [2](#0-1) 

```rust
fn borsh_args(address: Address, amount: U256) -> Result<Vec<u8>, ExitError> {
    borsh::to_vec(&WithdrawCallArgs {
        recipient_address: address,
        amount: NEP141Wei::new(amount.as_u128()),  // ← silent truncation
    })
    ...
}
```

**Site 3 – `exit_erc20_token_to_near`, wNEAR-unwrap branch** [3](#0-2) 

```rust
format!(r#"{{"amount":"{}"}}"#, exit_params.amount.as_u128()),  // ← silent truncation
```

**Site 4 – `exit_erc20_token_to_near`, legacy `ft_transfer` branch** [4](#0-3) 

```rust
format!(
    r#"{{"receiver_id":"{}","amount":"{}"}}"#,
    exit_params.receiver_account_id,
    exit_params.amount.as_u128()   // ← silent truncation
),
```

**Site 5 – `exit_base_token_to_near`, legacy `ft_transfer` branch** [5](#0-4) 

```rust
format!(
    r#"{{"receiver_id":"{}","amount":"{}"}}"#,
    exit_params.receiver_account_id,
    context.apparent_value.as_u128()   // ← silent truncation
),
```

**Contrast with the protected Omni path** – `ft_transfer_call_args` explicitly guards against this: [6](#0-5) 

```rust
fn ft_transfer_call_args(receiver_id: &AccountId, amount: U256, msg: &str) -> Result<String, ExitError> {
    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
    }
    ...
}
```

The ERC-20 amount is parsed from 32 bytes of EVM calldata as a full `U256` and stored in `Erc20TokenParams.amount`: [7](#0-6) 

No u128 bound is enforced at parse time, so any value up to `2^256 - 1` can reach the encoding sites.

The `FtOnTransferArgs.amount` field is typed as `Balance` (a `u128` wrapper) and its serde deserializer correctly rejects non-string JSON values: [8](#0-7) 

But the precompile encoding path bypasses that deserializer entirely; it constructs the JSON string directly via `format!`.

---

### Impact Explanation

`U256::as_u128()` in `primitive-types` returns the **low 128 bits** of the value with no error signal. For an amount `A > u128::MAX`:

- Aurora burns `A` ERC-20 tokens from the user's balance (irreversible).
- The NEAR-side `ft_transfer` / `near_withdraw` call receives only `A mod 2^128`.
- The difference `A - (A mod 2^128)` is permanently unrecoverable: the ERC-20 tokens no longer exist on Aurora, and the NEP-141 contract never received them.
- The `error_refund` callback only triggers on NEAR-side call failure; a successful transfer of the truncated amount does not trigger any refund.

This constitutes **permanent freezing (destruction) of funds** — Critical impact.

---

### Likelihood Explanation

`u128::MAX ≈ 3.4 × 10^38`. With 18 decimals this is `≈ 3.4 × 10^20` whole tokens. ERC-20 tokens with very large supplies (e.g., meme tokens, tokens with many decimals, or tokens intentionally minted with large supplies) can have individual balances exceeding this threshold. Any such token bridged to Aurora via the NEP-141 ↔ ERC-20 mechanism is vulnerable. The entry path requires no special privilege: any EVM user holding a sufficiently large ERC-20 balance can trigger the loss by calling the exit precompile. Likelihood: **2 / 5** (requires an unusual but valid token configuration; no admin access needed).

---

### Recommendation

Add an explicit overflow guard at every site that converts a `U256` amount to `u128`, mirroring the check already present in `ft_transfer_call_args`:

```rust
if amount > U256::from(u128::MAX) {
    return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
}
```

Apply this guard in `json_args`, `borsh_args`, `exit_erc20_token_to_near` (both branches), and `exit_base_token_to_near` before any `.as_u128()` call.

---

### Proof of Concept

1. Deploy an ERC-20 token on Aurora whose total supply is `2^128 + 1_000` (achievable by minting via the admin address after bridging a NEP-141 with a large supply).
2. Hold `2^128 + 1_000` tokens in address `VICTIM`.
3. From `VICTIM`, call the `ExitToNear` precompile (legacy path, no `:omni` suffix) with the full balance.
4. The ERC-20 contract burns `2^128 + 1_000` tokens and calls the precompile with `amount = 2^128 + 1_000`.
5. `exit_erc20_token_to_near` encodes `format!(..., exit_params.amount.as_u128())` → `"1000"`.
6. The NEAR-side `ft_transfer` transfers only `1_000` NEP-141 tokens to the recipient.
7. `2^128` tokens are permanently destroyed; no refund is issued because the NEAR call succeeded.
