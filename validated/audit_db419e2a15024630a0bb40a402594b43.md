### Title
Unchecked Precompile Call Return Value in `withdrawToNear` / `withdrawToEthereum` Causes Permanent Token Burn Without Withdrawal - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

In `EvmErc20.sol` and `EvmErc20V2.sol`, both `withdrawToNear` and `withdrawToEthereum` first irreversibly burn the caller's ERC-20 tokens, then invoke the Aurora exit precompile via inline assembly. The assembly `call` return value (`res`) is captured but **never checked**. If the precompile call fails, the transaction does not revert, the tokens are permanently destroyed, and no corresponding withdrawal is initiated on NEAR or Ethereum.

---

### Finding Description

In `EvmErc20.sol`, `withdrawToNear` executes:

```solidity
_burn(_msgSender(), amount);   // irreversible
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    // res is never checked — no `if iszero(res) { revert(0,0) }`
}
``` [1](#0-0) 

The same pattern appears in `withdrawToEthereum`: [2](#0-1) 

And identically in `EvmErc20V2.sol`: [3](#0-2) [4](#0-3) 

In EVM assembly, `call` returns `0` on failure and `1` on success. Without checking `res` and reverting on `0`, a failed precompile invocation silently succeeds from the EVM's perspective. The `_burn` that precedes the assembly block has already permanently reduced the caller's balance and total supply with no possibility of rollback.

The exit precompile (`engine-precompiles/src/native.rs`) has numerous failure paths returning `Err(ExitError::Other(...))` — for example, invalid recipient address encoding, mismatched input size, unregistered NEP-141 token, or insufficient gas forwarded to the precompile. Any of these causes `call` to return `0`, which the contract ignores. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing / destruction of funds.**

A user calling `withdrawToNear` or `withdrawToEthereum` with any input that causes the precompile to fail will have their ERC-20 tokens burned on Aurora with no corresponding NEP-141 release on NEAR and no Ethereum-side withdrawal. The bridged value is permanently destroyed. There is no recovery path: the burn is committed, the precompile failure is silent, and the transaction is marked successful.

---

### Likelihood Explanation

**Medium.** Any unprivileged token holder can trigger this by:
- Supplying a malformed or unregistered NEAR recipient account ID in `withdrawToNear`.
- Supplying a recipient address that causes the precompile to reject the call.
- Calling with insufficient gas forwarded to the precompile (the precompile costs are noted as `EthGas::new(0)` in costs but the underlying NEAR cross-contract call requires real gas).

The function is externally callable by any token holder with no access control, making the attack surface broad.

---

### Recommendation

Check the return value of the assembly `call` and revert if it is zero. For example:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply this fix to both `withdrawToNear` and `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. Alternatively, restructure the functions to perform the precompile call first and only burn tokens after confirming success.

---

### Proof of Concept

1. Deploy `EvmErc20` with a registered NEP-141 backing token.
2. Mint tokens to `attacker` address.
3. `attacker` calls `withdrawToNear` with a recipient string that is syntactically invalid or too long for the precompile to accept (e.g., an account ID exceeding NEAR's 64-character limit, or a zero-length byte string that fails the precompile's input validation).
4. The precompile's `call` returns `0` (failure).
5. Because `res` is never checked, no revert occurs.
6. `attacker`'s ERC-20 balance is now zero (burned), no NEP-141 tokens are released on NEAR, and the transaction receipt shows success.
7. The bridged value is permanently lost. [1](#0-0) [3](#0-2)
