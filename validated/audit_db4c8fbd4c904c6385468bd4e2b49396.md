### Title
Missing Deposit Refund on Failed `AddFunctionCallKey` Batch Action — (`engine/src/contract_methods/admin.rs`)

---

### Summary

When `add_relayer_key` is called with an attached NEAR deposit and the target public key already exists on the engine's NEAR account, the batch promise's `AddFunctionCallKey` action fails, the entire batch (including `store_relayer_key_callback`) is rolled back, but the attached deposit is never refunded. The deposit is permanently absorbed into the engine contract's balance with no recovery path.

---

### Finding Description

`add_relayer_key` reads `env.attached_deposit()` as the key `allowance`, validates it is ≥ 100 yoctoNEAR, then constructs a single `PromiseBatchAction` containing two actions in sequence:

1. `PromiseAction::AddFunctionCallKey { allowance, … }`
2. `PromiseAction::FunctionCall { name: "store_relayer_key_callback", attached_yocto: ZERO_YOCTO, … }` [1](#0-0) 

The function returns `Ok(())` after scheduling the batch. In NEAR Protocol, once a function call returns without panicking, the attached deposit is irrevocably transferred to the callee's balance. If the subsequently-scheduled promise fails, NEAR does **not** automatically refund the original deposit.

If the supplied public key already exists on the engine NEAR account (e.g., it is the engine's own deployment key, or a previously registered function-call key), the `AddFunctionCallKey` action fails at the NEAR runtime level. NEAR's batch-promise atomicity guarantee then rolls back the entire batch receipt — meaning `store_relayer_key_callback` is never executed — but the deposit that was already transferred to the engine contract during the `add_relayer_key` call is **not** returned.

This exact failure path is acknowledged and tested in the existing test suite: [2](#0-1) 

The test confirms the transaction fails and that `store_relayer_key_callback` is rolled back, but it does not assert any refund of the attached deposit. No refund logic exists anywhere in `add_relayer_key`, `store_relayer_key_callback`, or any error-handling callback. [3](#0-2) 

---

### Impact Explanation

The key manager's attached NEAR deposit (the `allowance` amount, which in tests is `NearToken::from_millinear(500)` = 0.5 NEAR) is absorbed into the engine contract's balance with no refund path. Because there is no admin function to sweep or return stuck deposits, and no error callback is registered on the batch, the funds are effectively permanently frozen inside the engine contract. The impact is **permanent freezing of funds** for the key manager, which maps to Critical severity, though the question scopes it as High (temporary freezing). [4](#0-3) 

---

### Likelihood Explanation

The trigger condition — a public key that already exists on the engine NEAR account — is realistic and non-adversarial. The engine's own deployment key is a full-access key present from genesis. Any key manager who attempts to register that key (or any previously-added function-call key) as a relayer key will silently lose their deposit. The key manager role is privileged but not infallible, and the contract provides no pre-flight check to detect duplicate keys before scheduling the batch. [5](#0-4) 

---

### Recommendation

Two complementary fixes:

1. **Use a callback pattern instead of a flat batch.** Replace the single `PromiseBatchAction` with a `promise_create_batch` (for `AddFunctionCallKey` only) followed by `promise_attach_batch_callback` whose callback checks `promise_result_check()`. If the result is `Some(false)`, issue a `PromiseAction::Transfer` back to `env.predecessor_account_id()` for the full `allowance` amount.

2. **Pre-flight duplicate-key check.** Before scheduling the batch, verify the key is not already registered in the engine's internal storage (via `engine::get_function_call_keys`) and return an error early (causing an automatic deposit refund via panic) if it is already present.

---

### Proof of Concept

The existing test `test_delete_fak_via_relayer_key` already demonstrates the failure path on unmodified code. To prove the deposit loss, extend it as follows (sandbox integration test):

```rust
#[tokio::test]
async fn test_deposit_lost_on_duplicate_key() {
    let aurora = deploy_engine().await;
    let public_key = aurora.public_key(); // already exists as FAK

    let manager = aurora.root()
        .create_subaccount("key_manager", NearToken::from_near(10))
        .await.unwrap();

    aurora.set_key_manager(RelayerKeyManagerArgs {
        key_manager: Some(manager.id().clone()),
    }).max_gas().transact().await.unwrap();

    let balance_before = manager.view_account().await.unwrap().balance;

    let result = manager
        .call(&aurora.id(), "add_relayer_key")
        .args_json(RelayerKeyArgs { public_key })
        .max_gas()
        .deposit(NearToken::from_millinear(500)) // 0.5 NEAR deposit
        .transact().await.unwrap();

    assert!(result.is_failure()); // batch fails: key already exists

    let balance_after = manager.view_account().await.unwrap().balance;

    // Assert: balance decreased by ~0.5 NEAR (deposit not refunded)
    // Gas costs are small; the 0.5 NEAR deposit is the dominant loss.
    let lost = balance_before.as_yoctonear() - balance_after.as_yoctonear();
    assert!(lost > NearToken::from_millinear(490).as_yoctonear(),
        "Deposit was not refunded: lost {} yoctoNEAR", lost);
}

pub fn store_relayer_key_callback<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;

        require_running(&state)?;
        env.assert_private_call()?;

        let public_key = serde_json::from_slice::<RelayerKeyArgs>(&io.read_input().to_vec())
            .map(|args| args.public_key)
            .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;

        aurora_engine_sdk::log!("add relayer public key: {public_key}");

        engine::add_function_call_key(&mut io, &public_key);

        Ok(())
    })
}

async fn test_delete_fak_via_relayer_key() {

        .await

        .call(&aurora.id(), "add_relayer_key")
        .args_json(RelayerKeyArgs { public_key })
        .max_gas()
        .deposit(DEPOSIT)
        .transact()
        .await
        .unwrap();
    // Should be failed because the key is already added.
    assert!(result.is_failure());

    let result = manager
        .call(&aurora.id(), "remove_relayer_key")
        .args_json(RelayerKeyArgs { public_key })
        .max_gas()
        .transact()
        .await
        .unwrap();
    // Should be failed because the key hasn't been added by the `store_relayer_key_callback`
    // triggered by the `add_relayer_key` transaction. The changes made by
    // `store_relayer_key_callback` are rollbacked in this case because it’s called with
    // `AddFunctionCallKey` action in one batch, and the action `AddFunctionCallKey`
    // failed because the key is already added.
    assert!(result.is_failure());
}
