### Title
Divide-Before-Multiply Precision Loss in `native_per_pubdata` Causes Systematic Undercharge of Pubdata Native Resources - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` is computed as `floor(pubdata_price / native_price)` and then multiplied by the number of pubdata bytes consumed. This divide-before-multiply ordering causes a systematic truncation loss, meaning every transaction that uses pubdata pays fewer native resources than it should. The truncated rate is then used both to deduct the intrinsic pubdata overhead from the native limit and to charge for execution-time pubdata, so the undercharge propagates through the entire transaction lifecycle.

---

### Finding Description

In `validation_impl.rs` line 142, `native_per_pubdata` is derived by integer (floor) division:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

This truncated rate is then used in two places:

1. **Intrinsic pubdata deduction** (`gas_helpers.rs:352`):
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   let native_limit = native_limit.checked_sub(intrinsic_pubdata_overhead)...
   ```

2. **Execution pubdata charging** (`gas_helpers.rs:430`):
   ```rust
   let native = current_pubdata_spent
       .checked_mul(native_per_pubdata)
       .ok_or(out_of_native_resources!())?;
   ```

The correct formula is `(pubdata_price × pubdata_bytes) / native_price`, but the code computes `floor(pubdata_price / native_price) × pubdata_bytes`. The difference per transaction is:

```
loss = (pubdata_price % native_price) × pubdata_bytes / native_price
```

This is bounded by `pubdata_bytes` native units per transaction but is always non-negative — it is a one-directional systematic undercharge.

The asymmetry with `native_per_gas` makes this clearly unintentional: `native_per_gas` uses `div_ceil` (rounds **up**, protecting the protocol), while `native_per_pubdata` uses `wrapping_div` (rounds **down**, benefiting the user):

```rust
// validation_impl.rs:135 — rounds UP (correct, protocol-protective)
u256_try_to_u64(&gas_price.div_ceil(native_price))

// validation_impl.rs:142 — rounds DOWN (incorrect, user-favorable)
u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
```

The same truncated rate is mirrored in `api/src/helpers.rs:427` for the off-chain intrinsic resource validator, meaning the off-chain check also underestimates the required native resources.

---

### Impact Explanation

Every transaction that writes pubdata pays fewer native resources than the operator priced. The undercharge per transaction scales with pubdata usage:

- Example: `pubdata_price = 1000`, `native_price = 7`, `pubdata_bytes = 1000`
  - Charged: `floor(1000/7) × 1000 = 142 × 1000 = 142,000` native units
  - Correct: `1000 × 1000 / 7 = 142,857` native units
  - Loss: **857 native units (~0.6%)** per 1000 bytes of pubdata

A user submitting a transaction with maximum pubdata (e.g., 100 KB) can systematically underpay for native resources on every transaction. The operator bears the proving/publishing cost for the underpaid pubdata. Over many transactions, the cumulative loss to the operator is proportional to total pubdata volume × `(pubdata_price % native_price) / native_price`.

Additionally, because the intrinsic pubdata overhead is also undercharged, the user receives slightly more native computational budget than they paid for, which could allow marginally more computation per transaction.

---

### Likelihood Explanation

This affects **every L2 transaction** that uses pubdata whenever `pubdata_price % native_price ≠ 0`. Since `pubdata_price` and `native_price` are independent operator-set parameters, the remainder is non-zero in the general case. No special transaction crafting is required — any normal ERC-20 transfer or storage write triggers this path. The entry point is the standard L2 transaction validation flow, reachable by any unprivileged user.

---

### Recommendation

Compute the native cost for pubdata without pre-dividing. Instead of storing a truncated `native_per_pubdata` rate and multiplying later, either:

1. **Use ceiling division** for `native_per_pubdata` (consistent with `native_per_gas`):
   ```rust
   let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
       .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
   ```

2. **Or defer the division**: pass `pubdata_price` and `native_price` separately and compute `(pubdata_bytes × pubdata_price) / native_price` at charge time to avoid any intermediate truncation.

Option 1 is the minimal fix and is consistent with the existing `div_ceil` pattern used for `native_per_gas`.

---

### Proof of Concept

**Concrete numeric example:**

Let `pubdata_price = 15`, `native_price = 10`, `pubdata_bytes = 100`.

Current code:
```
native_per_pubdata = floor(15 / 10) = 1
native_charged     = 1 × 100 = 100
```

Correct calculation:
```
native_charged = (15 × 100) / 10 = 150
```

The user pays **100** native units instead of **150** — a **33% undercharge** for pubdata. A user who repeatedly submits transactions with 100 bytes of pubdata pays only 2/3 of the correct native cost for pubdata on every transaction.
