Audit Report

## Title
Stale ICP/XDR Conversion Rate Used in `tokens_to_cycles` Enables Over-Minting During 5-Minute Cache Window — (`rs/nns/cmc/src/main.rs`)

## Summary

The Cycles Minting Canister caches the ICP/XDR conversion rate and refreshes it at most once every 5 minutes via `update_exchange_rate`. The function `tokens_to_cycles` reads this cached rate with no staleness check. Because both the CMC's cached rate and the live XRC rate are publicly queryable, an unprivileged attacker can detect when the cached rate is materially higher than the current market rate (i.e., ICP has just dropped in value) and immediately call `notify_top_up` or `notify_mint_cycles` to receive more cycles per ICP than the current market warrants. The ICP is burned at the stale rate, and the excess cycles are permanently credited.

## Finding Description

`tokens_to_cycles` in `rs/nns/cmc/src/main.rs` reads `state.icp_xdr_conversion_rate` unconditionally, with no check on `rate.timestamp_seconds`:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);
        ...
    })
}
``` [1](#0-0) 

This function is called unconditionally from `process_top_up` and `process_mint_cycles`, both of which are reachable from the permissionless `notify_top_up` and `notify_mint_cycles` update endpoints: [2](#0-1) [3](#0-2) 

The rate is refreshed by `update_exchange_rate` (called from the CMC heartbeat) at most once every `REFRESH_RATE_INTERVAL_SECONDS`: [4](#0-3) 

`do_set_icp_xdr_conversion_rate` only validates that the incoming rate has a strictly greater timestamp than the current one — it does not enforce any maximum age on the rate used for minting: [5](#0-4) 

`notify_top_up` has no caller restriction — any principal can invoke it: [6](#0-5) 

The CMC's cached rate is publicly queryable via `get_icp_xdr_conversion_rate` (a certified query). The live XRC rate is also publicly queryable. An attacker can therefore monitor both, detect a divergence caused by a rapid ICP price drop, and call `notify_top_up` before the next heartbeat fires to mint cycles at the stale (higher) rate.

The hourly minting cap (`DEFAULT_CYCLES_LIMIT = 150e15`) bounds the per-window gain but does not prevent the attack: [7](#0-6) 

## Impact Explanation

The attacker receives more cycles than the current ICP market value justifies. Cycles represent real compute resources on the IC. At a 10% ICP price drop within a 5-minute window, the attacker can mint up to 150,000 XDR worth of cycles (the hourly cap) while paying ICP worth only ~135,000 XDR at the current rate — a gain of ~15,000 XDR (~$19,500) per exploitation window. The ICP is burned, so the protocol absorbs the loss in the form of under-priced compute. This constitutes a significant CMC/XRC security impact with concrete, repeatable protocol harm, qualifying as **High** severity: "Significant XRC, NNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation

Both the CMC cached rate and the live XRC rate are publicly observable with no authentication. ICP price volatility of several percent within 5 minutes is not uncommon during market events. The attack requires only a standard ICP ledger transfer followed by a `notify_top_up` call — both are permissionless ingress messages requiring no special tooling beyond a price feed and a canister call. The attack is repeatable once per hourly minting window.

## Recommendation

1. **Staleness guard in `tokens_to_cycles`**: Compare `env.now_timestamp_seconds()` against `rate.timestamp_seconds`; if the difference exceeds a configurable threshold (e.g., 10 minutes), return a retriable `NotifyError` so callers can retry after the next heartbeat.
2. **Reduce `REFRESH_RATE_INTERVAL_SECONDS`**: Lower from 5 minutes to 1–2 minutes to shrink the exploitation window.
3. **Bound rate age at minting time**: Record the rate's `timestamp_seconds` alongside the minting event and refuse to mint if the rate is older than N seconds at the moment `process_top_up` / `process_mint_cycles` executes.

## Proof of Concept

1. Query `get_icp_xdr_conversion_rate` on the CMC → observe cached rate R_cmc (e.g., 50,000 XDR/ICP).
2. Query the XRC canister directly → observe live rate R_xrc (e.g., 45,000 XDR/ICP, a 10% drop).
3. Transfer N ICP to the CMC's subaccount for a target canister on the ICP ledger.
4. Call `notify_top_up` before the next CMC heartbeat fires (within the 5-minute window).
5. `process_top_up` calls `tokens_to_cycles(amount)`, which reads `state.icp_xdr_conversion_rate.xdr_permyriad_per_icp = 50,000` (stale) and mints cycles at that rate.
6. Attacker receives ~11% more cycles than the current market rate would give.
7. After the heartbeat fires and the rate corrects to 45,000, the attacker's cycles are already permanently credited.

A deterministic integration test can reproduce this using `state_machine_builder_for_nns_tests`, setting the CMC rate to 50,000 XDR/ICP, advancing time by less than 5 minutes without triggering a heartbeat, then calling `notify_top_up` and asserting the minted cycles exceed what the current XRC rate would produce.
