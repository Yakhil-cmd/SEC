### Title
Missing Validation for Anonymous `authorized_principal` in Rate-Limit Canister Initialization - (File: rs/boundary_node/rate_limits/canister/canister.rs)

### Summary
The `init` (and `post_upgrade`) function of the rate-limit canister accepts `Principal::anonymous()` as a valid `authorized_principal` without any validation. If the canister is initialized or upgraded with `authorized_principal = Some(Principal::anonymous())`, any anonymous ingress sender gains full write access to the privileged `add_config` and `disclose_rules` update methods, bypassing the intended access-control model.

### Finding Description
In `rs/boundary_node/rate_limits/canister/canister.rs`, the `init` function stores the caller-supplied `authorized_principal` directly into stable state without checking whether it equals `Principal::anonymous()`:

```rust
#[init]
fn init(init_arg: InitArg) {
    with_canister_state(|state| {
        if let Some(principal) = init_arg.authorized_principal {
            state.set_authorized_principal(principal);   // no anonymous check
        }
        ...
    });
}
``` [1](#0-0) 

`post_upgrade` delegates unconditionally to `init`, so the same path is reachable on every upgrade:

```rust
#[post_upgrade]
fn post_upgrade(init_arg: InitArg) {
    init(init_arg);
}
``` [2](#0-1) 

The `inspect_message` hook grants `has_full_access` by comparing `Some(caller_id) == authorized_principal`. When `authorized_principal` is `Principal::anonymous()`, every unsigned (anonymous) ingress message satisfies this equality:

```rust
let (has_full_access, _) = with_canister_state(|state| {
    let authorized_principal = state.get_authorized_principal();
    (
        Some(caller_id) == authorized_principal,   // true for anonymous caller
        ...
    )
});
``` [3](#0-2) 

The same equality is re-evaluated inside `AccessLevelResolver::get_access_level`, which gates the actual `add_config` and `disclose_rules` update handlers:

```rust
if let Some(authorized_principal) = self.canister_api.get_authorized_principal()
    && self.caller_id == authorized_principal
{
    return AccessLevel::FullAccess;
}
``` [4](#0-3) 

`set_authorized_principal` performs no validation before writing to stable storage:

```rust
fn set_authorized_principal(&self, principal: Principal) {
    self.authorized_principal
        .with(|cell| cell.borrow_mut().insert((), principal));
}
``` [5](#0-4) 

### Impact Explanation
If the canister controller initializes or upgrades the rate-limit canister with `authorized_principal = Some(Principal::anonymous())`, any anonymous ingress sender can:

1. Call `add_config` — injecting arbitrary rate-limit rules that are immediately enforced by all API boundary nodes polling the canister.
2. Call `disclose_rules` — publicly revealing confidential incident-response rules that are intentionally kept hidden from the public.

The rate-limit canister is a production boundary-node infrastructure canister. Injecting malicious rate-limit rules could be used to selectively block or throttle legitimate traffic to any canister on the IC, constituting a boundary/API validation bypass with real denial-of-service impact.

### Likelihood Explanation
`Principal::anonymous()` is a syntactically valid Candid `principal` value. A deployment script, CI pipeline, or operator that constructs the `InitArg` programmatically and passes a default/zero-value `Principal` (e.g., from an uninitialized variable or a Candid default) would silently set the authorized principal to anonymous. The `post_upgrade` path means this can be triggered on any future upgrade, not only at initial install. The likelihood is low-to-medium: it requires a controller-level misconfiguration, but the absence of a guard makes accidental misconfiguration realistic.

### Recommendation
Add an explicit anonymous-principal check in `init` before calling `set_authorized_principal`:

```rust
if let Some(principal) = init_arg.authorized_principal {
    if principal == Principal::anonymous() {
        ic_cdk::trap("authorized_principal must not be the anonymous principal");
    }
    state.set_authorized_principal(principal);
}
```

Alternatively, enforce the check inside `set_authorized_principal` in `state.rs` so that no code path can store an anonymous principal as the authorized writer.

### Proof of Concept
1. Deploy the rate-limit canister with `InitArg { authorized_principal: Some(Principal::anonymous()), registry_polling_period_secs: 5 }`.
2. Send an unsigned (anonymous) ingress `add_config` update call with an arbitrary `InputConfig`.
3. Observe that `inspect_message` accepts the call (`has_full_access = true`) and `WithAuthorization::add_config` also passes (`AccessLevel::FullAccess`), allowing the anonymous caller to inject rate-limit rules into the canister's stable state.
4. All API boundary nodes polling the canister will subsequently enforce the injected rules. [6](#0-5) [7](#0-6)
