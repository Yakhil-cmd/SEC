### Title
Pre-EIP-155 Legacy Transactions Bypass Chain ID Domain Binding in `submit_with_alt_modexp` — (File: engine/src/engine.rs)

---

### Summary

The `submit` entrypoint explicitly documents that it "Must match `CHAIN_ID` to make sure it's signed for given chain vs replayed from another chain." However, the chain ID check is conditional on the field being present. Legacy (pre-EIP-155) transactions with `v ∈ {27, 28}` carry no chain ID and silently bypass the check. Any pre-EIP-155 Ethereum transaction can be submitted verbatim to Aurora's `submit` function and will execute against Aurora user balances without any chain domain binding.

---

### Finding Description

In `submit_with_alt_modexp` the chain ID validation is:

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [1](#0-0) 

The guard `if let Some(chain_id) = transaction.chain_id` means the check is **only entered when a chain ID is present**. For legacy transactions, `LegacyEthSignedTransaction::chain_id()` returns `None` whenever `v ∈ {27, 28}`:

```rust
pub const fn chain_id(&self) -> Option<u64> {
    match self.v {
        0..=34 => None,
        _ => Some((self.v - 35) / 2),
    }
}
``` [2](#0-1) 

For these transactions, `sender()` still performs ecrecover and recovers a valid address — the signature is cryptographically verified — but the signed message is not bound to Aurora's chain ID. The signed bytes are `rlp([nonce, gasPrice, gasLimit, to, value, data])` with no chain ID component, identical across every EVM chain.

The contract entrypoint comment explicitly states the intent to prevent cross-chain replay: [3](#0-2) 

The implementation fails to enforce this for the pre-EIP-155 transaction class.

---

### Impact Explanation

An attacker who locates a pre-EIP-155 transaction on Ethereum's public history where:

1. The sender (`ecrecover`-recovered address) holds a balance on Aurora,
2. The sender's Aurora nonce equals the transaction's nonce field, and
3. The transaction's `to` field is an address the attacker controls (or the attacker is the original recipient),

can submit the raw RLP bytes directly to Aurora's `submit` function. The transaction passes ecrecover, skips the chain ID check (chain_id is `None`), passes the nonce check, and executes — transferring the sender's Aurora ETH to the attacker. No private key is needed; the transaction data is already public on Ethereum.

**Impact class**: Critical — direct theft of user funds at rest on Aurora.

---

### Likelihood Explanation

**Low-Medium.** Pre-EIP-155 transactions are rare in modern wallets but exist throughout Ethereum's history (pre-2016 era and some tooling). The nonce matching requirement is the primary constraint: the victim's Aurora nonce must equal the nonce in the historical Ethereum transaction. For users who have been active on both Ethereum (pre-EIP-155 era) and Aurora with coincidentally matching nonce states, the attack is directly executable with no additional prerequisites. An attacker can scan Ethereum's transaction history for pre-EIP-155 transactions where the recipient is an address they control, then check whether the sender has Aurora funds and a matching nonce.

---

### Recommendation

Reject transactions where `chain_id` is `None` before the existing conditional check. Add an explicit guard immediately before the current check in `submit_with_alt_modexp`:

```rust
// Reject pre-EIP-155 transactions: they carry no chain ID and are
// replayable across all EVM chains.
if transaction.chain_id.is_none() {
    return Err(EngineErrorKind::InvalidChainId.into());
}

// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [1](#0-0) 

---

### Proof of Concept

1. Find any pre-EIP-155 transaction on Ethereum mainnet (v=27 or v=28) where the sender has a non-zero Aurora balance and the sender's Aurora nonce equals the transaction nonce. The recipient must be an address the attacker controls.
2. Take the raw RLP-encoded bytes of that signed transaction.
3. Submit them to Aurora's `submit` WASM entrypoint (or `submit_with_args`).
4. `NormalizedEthTransaction::try_from` parses the transaction; `sender()` runs ecrecover and recovers the victim's address — signature check passes.
5. `transaction.chain_id` is `None`; the `if let Some(chain_id)` guard is not entered — chain ID check is skipped entirely.
6. `check_nonce` passes because the nonce matches.
7. The EVM executes the transfer, moving the victim's Aurora ETH to the attacker's address.

The attacker-controlled entry path is the public `submit` WASM function, callable by any NEAR account (including unprivileged relayers). No admin access, no private key, and no governance action is required. [4](#0-3) [5](#0-4)

impl LegacyEthSignedTransaction {
    /// Returns sender of given signed transaction by doing ecrecover on the signature.
    pub fn sender(&self) -> Result<Address, Error> {
        let mut rlp_stream = RlpStream::new();
        // See details of CHAIN_ID computation here - https://github.com/ethereum/EIPs/blob/master/EIPS/eip-155.md#specification
        let (chain_id, rec_id) = match self.v {
            0..=26 | 29..=34 => return Err(Error::InvalidV),
            27..=28 => (
                None,
                u8::try_from(self.v - 27).map_err(|_e| Error::InvalidV)?,
            ),
            _ => (
                Some((self.v - 35) / 2),
                u8::try_from((self.v - 35) % 2).map_err(|_e| Error::InvalidV)?,
            ),
        };
        self.transaction
            .rlp_append_unsigned(&mut rlp_stream, chain_id);

        sdk::ecrecover(message_hash, &super::vrs_to_arr(rec_id, self.r, self.s))
