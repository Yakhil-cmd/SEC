### Title
EIP-7610 Storage Emptiness Check Bypassed via Non-Zero Slot Storage — (File: `engine/src/engine.rs`)

---

### Summary

`is_empty_storage` and `is_account_empty` both check only storage slot `H256::zero()` (slot 0) as a proxy for "has any storage." A contract that writes exclusively to non-zero storage slots (e.g., via a mapping whose keys hash to non-zero slots) will cause both functions to incorrectly report the address as having empty storage. This bypasses the EIP-7610 CREATE/CREATE2 collision guard, allowing a new contract to be deployed to an address that already holds attacker-controlled storage values.

---

### Finding Description

`is_empty_storage` is the backend method called by SputnikVM to enforce EIP-7610 before executing CREATE/CREATE2. Its implementation contains an explicit, documented false assumption:

```
// As we can't read all storage data for account we assuming that if storage exists
// `index = 0` always true
``` [1](#0-0) 

The function checks only whether `storage_has_key` returns true for `H256::zero()` (slot 0). If an address has storage exclusively at non-zero slots, the function returns `true` — incorrectly declaring storage empty. [2](#0-1) 

The same flaw is present in `is_account_empty`, which is used in `apply()` to decide whether to delete an account after EVM execution: [3](#0-2) 

An account with zero balance, zero nonce, no code, but storage at non-zero slots passes `is_account_empty` as "empty" and is deleted — potentially without incrementing the generation counter, leaving the non-zero-slot storage accessible at the current generation. [4](#0-3) 

---

### Impact Explanation

EIP-7610 exists to prevent a contract from being deployed to an address that already has storage, because the new contract would inherit those storage values as its initial state. If `is_empty_storage` returns `true` when it should return `false`, the EIP-7610 guard is silently bypassed. A newly deployed contract (e.g., a token vault, an AMM pool, an access-control contract) inherits attacker-controlled storage values. Depending on the contract's logic, this can enable direct theft of user funds deposited into the contract (e.g., pre-set balance mappings, pre-set owner slots).

**Impact class**: Critical — direct theft of user funds.

---

### Likelihood Explanation

The attack requires:
1. The attacker to predict a CREATE2 target address (feasible for any public factory contract with a known salt scheme).
2. The attacker to deploy a contract to that address first, write storage only to non-zero slots, then remove the code/balance/nonce (e.g., via SELFDESTRUCT, subject to the generation-counter behavior of `remove_account`).
3. The victim factory to subsequently CREATE2 to the same address.

Factories with deterministic addresses (e.g., Uniswap-style pool factories) are common targets. The exploitability is conditional on whether `remove_account` increments the generation counter on SELFDESTRUCT; if it does not, the non-zero-slot storage remains visible at the current generation and the attack is directly executable.

**Likelihood**: Medium — requires address prediction and a two-step setup, but is fully attacker-controlled with no admin involvement.

---

### Recommendation

Aurora Engine cannot enumerate all NEAR storage keys for an address, so a full scan is not feasible. The correct mitigation is to ensure the generation counter is **always** incremented when an account is deleted (via SELFDESTRUCT or `delete_empty`), making all prior storage — at any slot — inaccessible to a subsequently deployed contract at the same address. Additionally, the comment in `is_empty_storage` should be elevated to a security note, and the EIP-7610 check should be documented as relying on the generation mechanism for correctness rather than on the slot-0 heuristic alone.

---

### Proof of Concept

1. Identify a CREATE2 factory and compute the deterministic target address `A` for a known salt `S` and init code `I`.
2. Deploy a contract to `A` using CREATE2 with salt `S` and init code `I` (first deployment). In the constructor, execute `SSTORE(1, attacker_address)` — writing only to slot 1 (non-zero).
3. Call SELFDESTRUCT on the deployed contract. Address `A` now has: no code, no balance, no nonce, storage at slot 1 = `attacker_address`.
4. Trigger the factory to CREATE2 to `A` again (second deployment, e.g., a token vault).
5. SputnikVM calls `is_empty_storage(A)`. The function checks `storage_has_key(A, slot_0, generation)` → `false` → returns `true` (storage "empty"). EIP-7610 check passes.
6. The new vault contract is deployed to `A` and inherits `storage[1] = attacker_address`.
7. If slot 1 is the vault's `owner` or a balance mapping key, the attacker has pre-authorized control over the vault and can drain deposited funds. [1](#0-0) [3](#0-2)

/// EIP-7610: balance, nonce, code, storage should be empty
pub fn is_account_empty<I: IO>(io: &I, address: &Address) -> bool {
    get_balance(io, address).is_zero()
        && get_nonce(io, address).is_zero()
        && get_code_size(io, address) == 0
        && !storage_has_key(io, address, &H256::zero(), get_generation(io, address))
}

    /// Check if the storage of the address is empty.
    /// Related to EIP-7610: non-empty storage
    fn is_empty_storage(&self, address: H160) -> bool {
        // As we can't read all storage data for account we assuming that if storage exists
        // `index = 0` always true
        let index = H256::zero();
        // First we're checking cache to not produce read-storage operation
        let address = Address::new(address);
        if self
            .contract_storage_cache
            .borrow()
            .contains_key(&(address, index))
        {
            return false;
        }
        let generation = *self
            .generation_cache
            .borrow_mut()
            .entry(address)
            .or_insert_with(|| get_generation(&self.io, &address));
        !storage_has_key(&self.io, &address, &index, generation)
    }
