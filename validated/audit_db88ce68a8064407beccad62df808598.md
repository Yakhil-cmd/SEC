### Title
Users Cannot Withdraw Bridged ETH When Contract Is Paused — (File: `engine/src/contract_methods/connector.rs`)

### Summary
The `withdraw()` function in the Aurora Engine enforces a `require_running()` guard that reverts when the contract is paused. The contract owner can pause the engine via `pause_contract()`, setting `state.is_paused = true`. Any user who has deposited ETH to the NEAR side of Aurora is then unable to call `withdraw()` to reclaim their funds until the owner calls `resume_contract()`, causing a temporary fund freeze.

### Finding Description
In `engine/src/contract_methods/connector.rs`, the `withdraw()` function is the primary mechanism for NEAR-side users to burn their bridged ETH (NEP-141 representation) and receive ETH back on Ethereum. It begins with an unconditional `require_running()` check:

```rust
pub fn withdraw<I: IO + Copy + PromiseHandler, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    require_running(&state::get_state(&io)?)?;   // <-- blocks all withdrawals when paused
    env.assert_one_yocto()?;
    ...
    return_promise(io, env, "engine_withdraw", args, ONE_YOCTO)
}
``` [1](#0-0) 

`require_running()` is defined as:

```rust
pub fn require_running(state: &state::EngineState) -> Result<(), ContractError> {
    if state.is_paused {
        return Err(errors::ERR_PAUSED.into());
    }
    Ok(())
}
``` [2](#0-1) 

The owner can set `state.is_paused = true` at any time by calling `pause_contract()`:

```rust
pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
``` [3](#0-2) 

This is exposed as a public NEAR contract method: [4](#0-3) 

The existing test `test_withdraw_from_near_pausability` explicitly confirms that when the `engine_withdraw` feature is paused, user calls to `withdraw` on the engine contract fail with `"Pausable: Method is paused"`, while the owner retains the ability to withdraw directly from the ETH connector: [5](#0-4) 

### Impact Explanation
**High — Temporary freezing of funds.**

Any user who has bridged ETH to the NEAR side of Aurora (holding NEP-141 ETH tokens) is completely unable to call `withdraw()` to reclaim their ETH on Ethereum while the contract is paused. The funds remain locked until the owner calls `resume_contract()`. There is no emergency withdrawal path for users that bypasses the pause.

### Likelihood Explanation
**Low.**

Triggering this requires the contract owner (`state.owner_id`) to call `pause_contract()`. This is an intentional administrative action, not something an unprivileged attacker can trigger. However, the owner is a single account with no time-lock or multisig requirement enforced at the contract level, making the risk non-negligible in adversarial or operational-error scenarios.

### Recommendation
Remove the `require_running()` check from the `withdraw()` function in `engine/src/contract_methods/connector.rs`, or add an explicit carve-out so that user fund withdrawals remain executable regardless of the contract's paused state. Administrative operations and new deposits can remain gated behind `require_running()`, but the exit path for user funds should always be available.

### Proof of Concept
1. User deposits ETH to Aurora via the Rainbow Bridge; they now hold NEP-141 ETH tokens on the NEAR side.
2. The contract owner calls `pause_contract()` on the Aurora Engine NEAR contract.
3. `state.is_paused` is set to `true`.
4. User calls `withdraw(recipient_address, amount)` on the Aurora Engine contract.
5. `require_running()` returns `Err(ERR_PAUSED)` at line 47 of `connector.rs`.
6. The transaction reverts; the user's bridged ETH is frozen.
7. Funds remain inaccessible until the owner calls `resume_contract()`.
