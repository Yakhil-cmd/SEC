Audit Report

## Title
Unauthorized Third-Party Griefing via `refresh_buyer_tokens` Caller/Buyer Mismatch — (`rs/sns/swap/canister/canister.rs`)

## Summary
The SNS Swap canister's `refresh_buyer_tokens` endpoint resolves the effective buyer from the caller-supplied `arg.buyer` field without verifying it matches the authenticated ingress sender. Any unprivileged principal can call this endpoint on behalf of any victim who has already transferred ICP to their swap subaccount, forcibly registering the victim as a committed participant and — when a `confirmation_text` is configured — forging the victim's acceptance of the swap's terms of service. If the swap commits, the victim's ICP is swept to the SNS governance canister and they receive SNS tokens without ever having consented to participate.

## Finding Description
In `rs/sns/swap/canister/canister.rs` at L127–143, the `refresh_buyer_tokens` update method accepts an arbitrary `buyer` principal from the request body:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()   // ← any principal accepted, no caller check
};
```

There is no assertion that `p == caller_principal_id()` when `arg.buyer` is non-empty. The resolved principal `p` is passed directly to `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs` at L1134.

Inside `refresh_buyer_token_e8s`, `validate_confirmation_text` (L1150, implemented at L363–384 of `swap.rs`) performs only a content-equality check against the swap-configured text — it is entirely decoupled from the authenticated caller identity. The confirmation text is publicly readable from swap state, so any attacker can supply it.

Once the function succeeds, the victim is inserted into `self.buyers` (L1285–1288) with their full subaccount balance recorded. The `error_refund_icp` function (L1925–1959 of `swap.rs`) explicitly blocks refunds for any principal present in `self.buyers` with `transfer_success_timestamp_seconds == 0`, returning a precondition error: `"ICP cannot be refunded as principal X has Y ICP (e8s) in escrow"`. This lock persists until the swap finalizes and `sweep_icp` runs.

If the swap commits, `sweep_icp` transfers the victim's ICP to the SNS governance canister and mints SNS tokens to the victim — a forced, unconsented participation. If the swap aborts, `sweep_icp` returns the ICP to the victim, but the victim's ToS acceptance was still forged and their funds were locked for the duration of the open phase without consent.

No existing guard prevents this. The proto definition at `rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto` L843–851 documents `buyer` as optional (defaulting to caller) but imposes no restriction on who may supply it.

## Impact Explanation
This is a **High** severity finding. A victim who transferred ICP to the swap subaccount intending to reconsider (or by mistake) can be forcibly committed to the swap by any unprivileged attacker. If the swap commits, the victim's ICP is permanently converted to SNS tokens without their knowledge or consent — a concrete, unauthorized transfer of canister-controlled user funds. When `confirmation_text` is set, the victim's legal/ToS acceptance is forged. This matches the allowed impact: *"Significant SNS security impact with concrete user or protocol harm"* and *"Unauthorized access to canister-controlled funds."*

## Likelihood Explanation
The attack requires no special privileges. The only precondition is that the victim has a non-zero ICP balance in their swap subaccount — a normal intermediate state in the payment flow. The victim's principal is observable on-chain. The `confirmation_text` is readable from public swap state via `get_init`. The attack window spans the entire `Open` lifecycle (potentially days to weeks). A motivated attacker (e.g., one trying to push a borderline swap over its minimum participation threshold) has clear incentive. The attack is repeatable for every victim who has transferred ICP but not yet called `refresh_buyer_tokens`.

## Recommendation
Enforce that the resolved buyer principal equals the authenticated caller when a non-empty `buyer` field is supplied:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let requested = PrincipalId::from_str(&arg.buyer).unwrap();
        assert_eq!(requested, caller, "buyer field must match the caller");
        requested
    };
    ...
}
```

If third-party notification (calling on behalf of another buyer) is intentionally desired for the no-`confirmation_text` case, it must be explicitly gated: allow third-party calls only when `init.confirmation_text.is_none()`, since accepting terms on behalf of another principal is never safe.

## Proof of Concept
1. Deploy an SNS swap with `confirmation_text = "I agree to the SNS terms"` in `Open` state.
2. Victim Alice transfers 10 ICP to `Account { owner: swap_canister_id, subaccount: Some(principal_to_subaccount(alice)) }` on the ICP ledger but does not call `refresh_buyer_tokens`.
3. Attacker Eve (any principal) submits an ingress call:
   ```
   refresh_buyer_tokens({
     buyer: alice_principal.to_text(),
     confirmation_text: Some("I agree to the SNS terms")
   })
   ```
4. The swap canister resolves `p = alice_principal`, reads Alice's 10 ICP balance from the ledger, validates the confirmation text (matches), and records Alice in `self.buyers` with `amount_icp_e8s = 10_0000_0000`.
5. Alice's ICP is now in escrow. Calling `error_refund_icp` for Alice returns: `"ICP cannot be refunded as principal alice has 1000000000 ICP (e8s) in escrow"`.
6. If the swap commits, `sweep_icp` transfers Alice's ICP to SNS governance and mints SNS tokens to Alice — forced, unconsented participation.

This can be reproduced as a unit test by calling `swap.refresh_buyer_token_e8s(alice, Some("I agree..."), ...)` from a different caller context, then asserting Alice appears in `swap.buyers` and that `error_refund_icp` returns a precondition error.
