### Title
Unchecked `transferFrom` Boolean Return Value in XCC Precompile Allows Bypassing wNEAR Payment â€” (`engine-precompiles/src/xcc.rs`)

---

### Summary

The Cross-Contract Call (XCC) precompile calls `transferFrom` on the wNEAR ERC-20 contract to collect payment from the caller before creating a NEAR promise. The EVM-level `exit_reason` is checked, but the ABI-encoded boolean return value of `transferFrom` is captured and silently discarded. If the wNEAR ERC-20 follows the non-reverting ERC-20 pattern (returning `false` on failure instead of reverting), an attacker can bypass payment entirely, causing the engine to fund the attacker's router account from its own wNEAR balance.

---

### Finding Description

In `CrossContractCall::run_with_handle`, when `required_near != ZERO_YOCTO`, the precompile calls `transferFrom` on the wNEAR ERC-20 to move wNEAR from the caller to the engine's implicit address:

```rust
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
match exit_reason {
    // Transfer successful, nothing to do
    aurora_evm::ExitReason::Succeed(_) => (),
    aurora_evm::ExitReason::Revert(r) => { ... }
    aurora_evm::ExitReason::Error(e) => { ... }
    aurora_evm::ExitReason::Fatal(f) => { ... }
}
``` [1](#0-0) 

`return_value` is bound but never decoded or inspected. For ERC-20 `transferFrom`, the return value is `abi.encode(bool)`. A `Succeed` exit reason only means the EVM call frame completed without reverting â€” it does not guarantee the transfer actually occurred. If the wNEAR ERC-20 returns `false` (insufficient balance or allowance) while the EVM call succeeds, the precompile falls through the `Succeed` arm and continues to emit the promise log as if payment was collected.

The `required_near` amount (which includes `STORAGE_AMOUNT = 2 NEAR` for new router accounts) is then passed to `handle_precompile_promise` in `engine/src/xcc.rs`, which calls `withdraw_wnear_to_router`. This function calls the wNEAR exit precompile on the **engine's implicit address**, not the attacker's address. [2](#0-1) 

If the attacker's `transferFrom` silently failed, the engine's implicit address was never credited with the attacker's wNEAR, yet the engine proceeds to withdraw from its own wNEAR balance to fund the attacker's router.

---

### Impact Explanation

**Critical â€” Direct theft of engine-held wNEAR/NEAR funds.**

An attacker with zero wNEAR (or zero allowance granted to the XCC precompile address) can repeatedly invoke the XCC precompile. Each invocation causes the engine to pay `STORAGE_AMOUNT` (2 NEAR) + any `attached_near` from the call out of its own wNEAR balance to deploy and fund the attacker's router sub-account. The engine's wNEAR reserve is drained without any corresponding payment from the attacker. [3](#0-2) 

---

### Likelihood Explanation

**Medium.** The exploitability depends on whether the deployed wNEAR ERC-20 on Aurora follows the non-reverting ERC-20 pattern. Standard OpenZeppelin ERC-20 reverts on `transferFrom` failure, which would be caught by the `Revert` arm. However, many ERC-20 implementations (including some bridged token wrappers) return `false` instead of reverting. The code is structurally wrong regardless: the return value is never checked, and any future wNEAR contract upgrade to a non-reverting pattern would immediately open this attack. The entry path is fully unprivileged â€” any EVM user can call the XCC precompile.

---

### Recommendation

After confirming `exit_reason` is `Succeed`, decode and assert the boolean return value:

```rust
aurora_evm::ExitReason::Succeed(_) => {
    // Decode the bool return value of transferFrom
    let transfer_ok = return_value
        .get(31)
        .copied()
        .unwrap_or(0) == 1;
    if !transfer_ok {
        return Err(PrecompileFailure::Revert {
            exit_status: aurora_evm::ExitRevert::Reverted,
            output: b"ERR_WNEAR_TRANSFER_FAILED".to_vec(),
        });
    }
}
``` [4](#0-3)

### Proof of Concept

1. Attacker deploys or uses an existing EVM account with zero wNEAR balance and zero allowance granted to the XCC precompile address (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`).
2. Attacker submits an Aurora EVM transaction calling the XCC precompile with a `CrossContractCallArgs::Eager` payload targeting any NEAR contract. The `required_near` computed by the precompile will be `STORAGE_AMOUNT` (2 NEAR) because no router exists for the attacker's address yet.
3. The precompile calls `transferFrom(attacker, engine_implicit, 2e24)` on wNEAR. If wNEAR returns `false` (non-reverting pattern), `exit_reason` is `Succeed` and `return_value` is `abi.encode(false)`.
4. The precompile does not inspect `return_value`, emits the promise log with `required_near = 2e24`.
5. The engine's `apply_log` processes the promise log, calls `handle_precompile_promise`, which calls `withdraw_wnear_to_router` â€” withdrawing 2 NEAR from the **engine's** wNEAR balance to the attacker's router sub-account.
6. Attacker repeats until the engine's wNEAR reserve is exhausted. [5](#0-4) [6](#0-5)

pub fn withdraw_wnear_to_router<I: IO + Copy, E: Env, M: ModExpAlgorithm, H: PromiseHandler>(
    recipient: &AccountId,
    amount: Yocto,
    wnear_address: Address,
    engine: &mut Engine<I, E, M>,
    handler: &mut H,

    let mut interceptor = PromiseInterceptor::new(handler);
    let withdraw_call_args = withdraw_wnear_call_args(recipient, amount, wnear_address);

}
