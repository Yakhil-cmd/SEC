### Title
Unauthenticated `ft_on_transfer` Allows Minting Unbacked ERC-20 Tokens Without Depositing NEP-141 Tokens - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `ft_on_transfer` entry point on Aurora Engine can be called directly by any unprivileged NEAR account. Because the function uses `predecessor_account_id` as the NEP-141 token identifier with no check that actual tokens were transferred, and because `deploy_erc20_token` is permissionless, an attacker can register their own NEAR account as a NEP-141 token and then call `ft_on_transfer` directly to mint arbitrary amounts of ERC-20 tokens without depositing any real NEP-141 tokens.

---

### Finding Description

**Root cause — `ft_on_transfer` has no caller validation:** [1](#0-0) 

The function reads `predecessor_account_id` from the NEAR environment and immediately uses it as the NEP-141 token identifier:

```rust
let predecessor_account_id = env.predecessor_account_id();
...
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,   // ← arbitrary caller account used as token
        &args,
        &current_account_id,
        handler,
    )
};
```

There is no check that the caller is a legitimate NEP-141 token contract, and no check that any tokens were actually transferred to Aurora Engine before this call.

**Root cause — `deploy_erc20_token` is permissionless:** [2](#0-1) 

The only guard is `require_running`. Any NEAR account can register any NEP-141 account ID and get an ERC-20 token deployed for it.

**Minting path — `receive_erc20_tokens` mints unconditionally:** [3](#0-2) 

Once `get_erc20_from_nep141` succeeds (i.e., the caller's account ID is registered), `setup_receive_erc20_tokens_input` is called and the ERC-20 `mint` is executed with `erc20_admin_address` (Aurora Engine itself) as the caller — which is the admin of every deployed ERC-20 token. No token balance check, no transfer receipt, no proof of deposit.

---

### Impact Explanation

**Critical — Insolvency / Direct theft of funds.**

An attacker mints ERC-20 tokens on Aurora that have no NEP-141 backing. These tokens are indistinguishable from legitimately bridged tokens within the EVM. The attacker can:

- Swap them for other ERC-20 tokens in Aurora DeFi protocols, draining liquidity pools.
- Transfer them to other users who later attempt to exit to NEAR, causing those exits to fail because Aurora Engine holds no corresponding NEP-141 balance.

The result is insolvency of the Aurora bridge: legitimate depositors cannot withdraw their NEP-141 tokens because the attacker has extracted the backing.

---

### Likelihood Explanation

**High.** The attack requires:
1. A NEAR account (free to create).
2. One call to `deploy_erc20_token` (permissionless, no deposit required).
3. One direct call to `ft_on_transfer` with an arbitrary `amount` and a recipient EVM address.

No privileged access, no leaked keys, no governance capture. The entire attack is executable by any NEAR user in two transactions.

---

### Recommendation

1. **Validate the caller in `ft_on_transfer`:** Maintain an allowlist of registered NEP-141 account IDs and reject calls from accounts not on the list, or verify that the caller is a known NEP-141 contract by checking the `nep141_erc20_map` before minting.
2. **Gate `deploy_erc20_token`:** Restrict who can register new NEP-141 ↔ ERC-20 mappings (e.g., owner-only or via a governance mechanism), so an attacker cannot self-register an arbitrary account.
3. **Defense-in-depth:** Consider requiring a 1-yoctoNEAR deposit on `ft_on_transfer` (analogous to `assert_one_yocto`) to at least raise the cost of spam, though this alone does not fix the root cause.

---

### Proof of Concept

```
1. Attacker creates NEAR account `attacker.near`.

2. Attacker calls `deploy_erc20_token` on Aurora Engine with
   nep141 = "attacker.near".
   → Aurora Engine deploys an ERC-20 token and stores the mapping
     attacker.near → <erc20_address> in nep141_erc20_map.

3. Attacker calls `ft_on_transfer` directly on Aurora Engine
   (as a NEAR transaction signed by attacker.near) with:
     sender_id = "attacker.near"
     amount    = 1_000_000_000_000   (arbitrary large amount)
     msg       = hex(victim_evm_address)

4. Inside ft_on_transfer:
   - predecessor_account_id = "attacker.near"
   - predecessor_account_id ≠ connector_account_id  →  receive_erc20_tokens is called
   - get_erc20_from_nep141("attacker.near") succeeds (registered in step 2)
   - setup_receive_erc20_tokens_input mints 1_000_000_000_000 ERC-20 tokens
     to victim_evm_address (or any address the attacker controls)

5. Attacker now holds 1_000_000_000_000 unbacked ERC-20 tokens.
   They swap them in Aurora DeFi for legitimately backed tokens,
   then exit those tokens to NEAR, draining the bridge.
```

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;
