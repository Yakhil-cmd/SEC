### Title
Permanent Loss of ERC-20 Tokens on Failed `exit_to_near` When `error_refund` Feature Is Disabled — (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is absent, a failed `ft_transfer` promise during an ERC-20 exit-to-NEAR operation results in permanent, unrecoverable loss of user tokens. The ERC-20 tokens are burned inside the EVM, but the corresponding NEP-141 tokens are never delivered to the recipient and remain locked in Aurora's account with no escape hatch.

---

### Finding Description

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` handles bridging ERC-20 tokens back to NEAR by:

1. Burning the ERC-20 tokens in the EVM.
2. Scheduling a `ft_transfer` (or `ft_transfer_call`) NEAR promise to the NEP-141 contract.
3. Optionally attaching a callback (`exit_to_near_precompile_callback`) that re-mints the burned tokens if the transfer fails.

The critical branch is at lines 449–483:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None when feature is absent
    transfer_near: transfer_near_args,
};

let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { base: transfer_promise, callback: … })
};
``` [1](#0-0) 

For a regular ERC-20 exit (not a wNEAR unwrap), `transfer_near_args` is `None`. When `error_refund` is also absent, `callback_args.refund` is `None`, making `callback_args` equal to its `Default` value. The condition on line 470 therefore selects `PromiseArgs::Create` — a bare promise with **no callback**. [2](#0-1) 

If the `ft_transfer` promise fails (e.g., the recipient account is not registered with the NEP-141 contract, the NEP-141 contract is paused, or storage deposit is insufficient), execution ends with no recovery path:

- The ERC-20 tokens are already burned in the EVM — they are gone.
- The NEP-141 tokens remain in Aurora's account — permanently inaccessible to the user.
- There is no admin function, no invalidation flag, and no user-callable escape hatch to reclaim them.

The callback handler `exit_to_near_precompile_callback` in `engine/src/contract_methods/connector.rs` (lines 231–239) calls `engine::refund_on_error` to re-mint burned tokens, but this handler is **never scheduled** when `error_refund` is absent. [3](#0-2) 

The engine's own integration tests explicitly document this behavior:

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

The same permanent-loss scenario applies to ETH exits, confirmed by the parallel test `test_exit_to_near_eth_refund`: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any EVM user who calls the `ExitToNear` precompile (directly or via an ERC-20 wrapper contract) with a NEAR recipient that causes the `ft_transfer` promise to fail will permanently lose their tokens. The ERC-20 supply is reduced (tokens burned) while the NEP-141 supply held by Aurora is unchanged — a permanent accounting divergence with no recovery mechanism. The loss is 100% of the exited amount.

---

### Likelihood Explanation

**Medium.** The failure condition — a rejected `ft_transfer` — is realistic and user-triggerable:

- Sending to a NEAR account not registered with the NEP-141 contract (no storage deposit).
- Sending to a non-existent NEAR account.
- The NEP-141 contract being paused or blacklisting Aurora.

Any of these conditions, combined with a production binary compiled without `error_refund`, produces permanent fund loss. The condition is reachable by any unprivileged EVM user without any admin involvement.

---

### Recommendation

1. **Enable `error_refund` as a default feature** in the production WASM build so that `callback_args.refund` is always populated and a recovery callback is always scheduled.
2. Alternatively, unconditionally attach the `exit_to_near_precompile_callback` even when `error_refund` is disabled, and have the callback perform a no-op refund in that case, so the promise chain always has a failure handler.
3. Add a protocol-level invariant check that rejects any `exit_to_near` call that would result in a bare `PromiseArgs::Create` with no failure handler.

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature flag.
2. Bridge a NEP-141 token to Aurora, minting ERC-20 tokens to address `A`.
3. From address `A`, call the ERC-20 exit function targeting a NEAR account `unregistered.near` that has no storage deposit with the NEP-141 contract.
4. The ERC-20 tokens are burned (EVM balance of `A` drops to zero).
5. The `ft_transfer` promise to the NEP-141 contract fails (account not registered).
6. No callback fires; no re-mint occurs.
7. Observe: ERC-20 balance of `A` = 0; NEP-141 balance of `unregistered.near` = 0; NEP-141 balance of Aurora = unchanged (tokens locked forever).

This is directly confirmed by the existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` lines 623–666, which shows the balance discrepancy when the feature is absent. [6](#0-5)
