Audit Report

## Title
DKG Pool Hit Bypass Allows Stale Remote Dealings Into Finalized Blocks — (`rs/consensus/dkg/src/payload_validator.rs`)

## Summary

In `validate_dealings_payload`, the `validated_contains` fast-path at line 213 unconditionally skips both the config-existence check and `crypto_validate_dealing` for any dealing already present in the local validated DKG pool. Because the DKG pool is only purged at DKG interval boundaries (not on remote context cancellation), a dealing for a cancelled remote DKG config remains in every honest node's validated pool for the rest of the interval. A single Byzantine block proposer can exploit this to include those stale dealings in finalized blocks, consuming all `max_dealings_per_payload` slots and delaying or preventing DKG transcript completion for the affected interval.

## Finding Description

`validate_dealings_payload` (`rs/consensus/dkg/src/payload_validator.rs`, lines 209–224) iterates over dealings in the proposed payload:

```rust
for message in &dealings.messages {
    if dkg_pool.validated_contains(message) {   // line 213 — fast-path
        continue;                               // skips config check AND crypto
    }
    let Some(config) = configs.get(&message.content.dkg_id) else {
        return Err(InvalidDkgPayloadReason::MissingDkgConfigForDealing.into());
    };
    crypto_validate_dealing(crypto, config, message)?;
}
```

The `configs` map is rebuilt from certified replicated state on every call (lines 193–205). When a remote DKG context is cancelled or times out, its entry is removed from replicated state, so `configs` no longer contains the corresponding `dkg_id`. However, the DKG pool's `purge` method (`rs/artifact_pool/src/dkg_pool.rs`, lines 59–82) only removes entries whose `id.height < height` (i.e., from a previous DKG interval). A dealing validated during the current interval for a now-cancelled remote config therefore remains in every honest node's validated pool for the rest of the interval.

The precondition is established by `validate_dealings_for_dealer` in `rs/consensus/dkg/src/lib.rs` (lines 207–219): remote DKG dealings are explicitly **deferred** (not rejected) when no config is found yet, so they accumulate in the unvalidated pool and are moved to validated once the config appears. After context cancellation, those validated entries are never evicted mid-interval.

Attack sequence:
1. Remote DKG context exists → honest node's dealing is validated into the pool (`AddToValidated`).
2. Remote DKG context is cancelled/timed-out → removed from replicated state.
3. `configs` no longer contains the dealing's `dkg_id`.
4. Byzantine proposer crafts a `DkgDataPayload` containing that stale dealing (already gossiped to all peers).
5. Every honest validator calls `dkg_pool.validated_contains(message)` → `true` → `continue` → the config-existence check at line 218 is never reached → `Ok(())`.
6. The block is finalized with a dealing for a non-existent config occupying a payload slot.

The existing test `test_validate_payload_dealings_registry_version` (lines 856–874) already demonstrates `validate_payload` returns `Ok(())` when the dealing is in the validated pool, but does not remove the config from state first, confirming the bypass is untested.

## Impact Explanation

This is a **liveness** impact. `max_dealings_per_payload` is a hard cap checked before the per-message loop (lines 164–170). A Byzantine proposer can fill all slots with stale dealings across consecutive rounds it proposes, preventing legitimate dealings from being included. Repeated over a DKG interval this delays or prevents transcript completion, which in turn delays subnet creation or resharing operations. The DKG summary creator ignores dealings for absent configs, so no incorrect transcript is produced and safety is not broken. This maps to the **High** bounty impact: *Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS* ($2,000–$10,000).

## Likelihood Explanation

Requires only one Byzantine replica in the normal block proposer rotation — no threshold corruption. Remote DKG context cancellation (subnet creation timeout, reshare cancellation) is a normal protocol event. The stale dealing is already present in every honest node's validated pool via gossip — no forgery or key material is needed. The exploitable window is the remainder of the current DKG interval after context removal, which can span many blocks.

## Recommendation

Move the `validated_contains` fast-path **after** the config-existence check:

```rust
for message in &dealings.messages {
    let Some(config) = configs.get(&message.content.dkg_id) else {
        return Err(InvalidDkgPayloadReason::MissingDkgConfigForDealing.into());
    };
    if dkg_pool.validated_contains(message) {
        continue;   // config still active; safe to skip expensive crypto re-validation
    }
    crypto_validate_dealing(crypto, config, message)?;
}
```

This preserves the performance optimisation (skip `crypto_validate_dealing` for already-validated dealings) while enforcing that the config must still be active in replicated state.

## Proof of Concept

Extend the existing test `test_validate_payload_dealings_registry_version` (`rs/consensus/dkg/src/payload_validator.rs`, lines 856–874):

1. Set up a subnet with a live remote DKG context; let `DkgImpl::on_state_change` produce and validate a dealing into the pool (`AddToValidated`).
2. Remove the remote DKG context from the mock `StateManager` (simulate cancellation — return a state snapshot without the context).
3. Construct a `DkgDataPayload` containing that dealing.
4. Call `validate_payload` with the updated state (context absent) and the populated DKG pool.
5. Assert the result is `Ok(())` — demonstrating the bypass.
6. Assert `MissingDkgConfigForDealing` is **not** returned despite the config being absent from `configs`.

After applying the fix, step 5 should instead return `Err(MissingDkgConfigForDealing)`.
