### Title
Silo Whitelist Validates Sender But Not Recipient, Enabling Permanent Fund Freeze - (`engine/src/engine.rs`)

### Summary

In Aurora Engine's Silo mode, the `assert_access` function validates only the **sender** EVM address against the Address whitelist before executing a transaction. The **recipient** address (`transaction.to`) is never checked. A whitelisted sender can therefore transfer ETH or ERC-20 tokens to any non-whitelisted address. Because the whitelist also gates all outbound transaction submission, the recipient is permanently unable to move those funds.

### Finding Description

`assert_access` in `engine/src/engine.rs` is the sole pre-execution access gate for all EVM transactions submitted via `submit()` / `submit_with_args()`:

```rust

`is_allow_submit` checks only the **sender** (`transaction.address`) and the NEAR predecessor account:

```rust
// engine/src/contract_methods/silo/mod.rs  lines 136-138
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)

``` [2](#0-1) 

`transaction.to` — the recipient — is never passed to any whitelist check. The EVM then executes the transfer unconditionally, crediting the recipient's balance or ERC-20 balance regardless of whitelist status.

The same whitelist that blocks the recipient from **receiving** also blocks the recipient from **sending**: any subsequent `submit()` call from the recipient's address will fail `is_allow_submit` because the recipient's address is not in the `WhitelistKind::Address` list. The funds are therefore irrecoverable without admin intervention.

**Contrast with the NEP-141 bridge path**: `receive_erc20_tokens` explicitly checks the recipient and redirects to a fallback address when the recipient is not whitelisted:

```rust

    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [3](#0-2) 

No equivalent protection exists for the EVM transaction path. Additionally, `receive_base_tokens` (the ETH bridge path) also lacks any recipient whitelist check: [4](#0-3) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any ETH balance or ERC-20 token balance transferred to a non-whitelisted EVM address in Silo mode is permanently inaccessible to the recipient. The recipient cannot submit any EVM transaction (blocked by `is_allow_submit`), so they cannot transfer, approve, or otherwise move the assets. Recovery requires the Silo admin to add the recipient to the whitelist — an out-of-band action that may never occur and is not guaranteed by the protocol.

### Likelihood Explanation

**Medium.** Silo mode is an explicitly supported Aurora deployment configuration used by Silo operators to restrict access. Any Silo deployment with the `WhitelistKind::Address` list enabled is affected. A whitelisted user can trigger the freeze accidentally (sending to a friend's address not yet on the whitelist) or deliberately (griefing another user). No special privileges beyond being a whitelisted sender are required.

### Recommendation

In `assert_access`, when `transaction.to` is `Some(recipient)`, additionally verify that the recipient address is permitted to receive funds. The simplest fix mirrors the existing `is_allow_receive_erc20_tokens` logic:

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if let Some(recipient) = transaction.to {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
            && silo::is_allow_receive_erc20_tokens(io, &recipient)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };
    ...
}
```

Apply the same recipient check in `receive_base_tokens` (the ETH bridge path), consistent with the protection already present in `receive_erc20_tokens`.

### Proof of Concept

**Setup**: Silo mode active, `WhitelistKind::Address` whitelist enabled. Alice (`0xAAAA…`) is whitelisted. Bob (`0xBBBB…`) is not whitelisted.

1. Alice calls `submit()` with a signed EVM transaction: `to = 0xBBBB…`, `value = 1 ETH` (or an ERC-20 `transfer(0xBBBB…, amount)` call).
2. `assert_access` is called. It checks `is_allow_submit(…, alice_account, 0xAAAA…)` → **passes** (Alice is whitelisted). `0xBBBB…` is never checked.
3. The EVM executes. Bob's balance at `0xBBBB…` is credited with 1 ETH (or ERC-20 tokens).
4. Bob attempts to call `submit()` to move his funds. `assert_access` checks `is_allow_submit(…, bob_account, 0xBBBB…)` → **fails** (`NotAllowed`). Bob cannot submit any transaction.
5. Funds at `0xBBBB…` are permanently frozen.

The call chain is: `lib.rs::submit()` → `evm_transactions::submit()` → `engine::submit_with_alt_modexp()` → `assert_access()` (sender-only check, line 1052) → EVM execution → balance credited to non-whitelisted recipient. [5](#0-4) [6](#0-5) [7](#0-6)
