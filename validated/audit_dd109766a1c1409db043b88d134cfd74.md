### Title
Missing `http2_max_header_list_size` on SOCKS Proxy Client Enables Adapter Crash via Oversized HTTP/2 Headers — (`rs/https_outcalls/adapter/src/rpc_server.rs`)

---

### Summary

The HTTPS outcalls adapter builds two separate hyper clients: a direct client and a SOCKS proxy client. The direct client is constructed with `.http2_max_header_list_size(MAX_HEADER_LIST_SIZE)` (52 KB), but the SOCKS proxy client omits this call entirely, leaving h2's default of `u32::MAX`. An unprivileged canister developer can route a request through the SOCKS path by targeting an IPv4 address (direct connection fails on IPv6-only nodes), then have their controlled server respond with an HTTP/2 HEADERS frame exceeding the limit. Because `abort_on_panic()` is installed in `main()`, any resulting hyper panic terminates the entire adapter process, dropping all in-flight HTTP outcalls.

---

### Finding Description

**Direct client construction** (protected):

```rust
let client = Client::builder(TokioExecutor::new())
    .http2_max_header_list_size(MAX_HEADER_LIST_SIZE)   // 52 * 1024
    .build::<_, Full<Bytes>>(direct_https_connector);
``` [1](#0-0) 

**SOCKS proxy client construction** (unprotected):

```rust
Client::builder(TokioExecutor::new()).build::<_, Full<Bytes>>(
    builder
        .enable_all_versions()
        .wrap_connector(SocksConnector { proxy_addr, auth: None, connector: http_connector }),
)
``` [2](#0-1) 

The constant and the developer's own comment explain why the limit exists:

```rust
/// Hyper only supports a maximum of 32768 headers … and it panics if we try to allocate more headers.
const HEADERS_LIMIT: usize = 1_024;
const MAX_HEADER_LIST_SIZE: u32 = 52 * 1024;
``` [3](#0-2) 

The SOCKS path is reached automatically when the direct connection fails:

```rust
self.client.request(http_req)
    .or_else(|direct_err| async move {
        self.do_https_outcall_socks_proxy(req.socks_proxy_addrs, http_req_clone).await
        ...
    })
``` [4](#0-3) 

The post-receive header size check (lines 402–417) only runs **after** `http_resp` is already returned from hyper — i.e., after hyper has already parsed and inserted all headers into its `HeaderMap`. It provides no protection against the panic. [5](#0-4) 

`abort_on_panic()` is installed unconditionally at adapter startup, so any hyper `HeaderMap` panic terminates the process:

```rust
abort_on_panic();
``` [6](#0-5) 

---

### Impact Explanation

A crash of the adapter process drops every in-flight HTTP outcall on the affected replica node. Canisters relying on HTTP outcalls receive transient errors for all pending requests. The adapter is a per-node process; crashing it on multiple nodes (by issuing concurrent requests from multiple canisters) degrades subnet-wide HTTP outcall availability until the adapter is restarted by systemd.

---

### Likelihood Explanation

The SOCKS path is the normal production fallback for IPv4 destinations (IC nodes are IPv6-only). Any canister can issue an `http_request` to an IPv4 address it controls. The SOCKS proxy addresses are populated from the registry (boundary nodes) and are not attacker-controlled, but the **destination server** is fully attacker-controlled. Triggering the path requires only a valid canister and an IPv4 HTTPS endpoint that returns an oversized HTTP/2 HEADERS frame. No privileged access, key material, or network-level attack is required.

---

### Recommendation

Add `.http2_max_header_list_size(MAX_HEADER_LIST_SIZE)` to the `Client::builder` call inside `create_socks_proxy_client`, mirroring the direct client:

```rust
Client::builder(TokioExecutor::new())
    .http2_max_header_list_size(MAX_HEADER_LIST_SIZE)   // add this line
    .build::<_, Full<Bytes>>(
        builder.enable_all_versions().wrap_connector(SocksConnector { ... })
    )
``` [2](#0-1) 

---

### Proof of Concept

1. Deploy a canister that calls `http_request` targeting `https://<attacker-IPv4>/`.
2. The adapter's direct connection fails (node has no IPv4 route); `do_https_outcall_socks_proxy` is invoked.
3. The attacker's server, on receiving the TLS-tunnelled HTTP/2 connection via the SOCKS proxy, responds with a HEADERS frame containing either >32 768 individual header fields or a total header block exceeding 52 KB.
4. Because `create_socks_proxy_client` omits `http2_max_header_list_size`, h2 does not reject the frame at the protocol level; hyper attempts to insert all headers into its `HeaderMap`.
5. `HeaderMap` panics; `abort_on_panic()` kills the adapter process.
6. Differential test: send the identical oversized response to the direct client — h2 rejects it with a `COMPRESSION_ERROR` before hyper touches the `HeaderMap`, returning a graceful connection error to the canister. The SOCKS client panics instead.
