Audit Report

## Title
`get_allowances` Break Condition Checks Only `owner`, Not Full `Account`, Leaking Allowances Across Subaccounts — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

## Summary

The `get_allowances` function performs a `StableBTreeMap` range scan over `AccountSpender` keys ordered by `(account, spender)`. Its only loop-termination guard compares `account_spender.account.owner != from.owner`, ignoring the subaccount field. Because `Account` ordering includes the subaccount, after exhausting all spenders for `{A, Some([1;32])}` the iterator advances to `{A, Some([2;32])}` — same owner, different subaccount — and the guard never fires, causing allowances belonging to unintended subaccounts to be appended to the result. The public `#[query]` entrypoint `icrc103_get_allowances` has no access-control check, so any unprivileged caller can trigger this against any principal.

## Finding Description

`get_allowances` in `rs/ledger_suite/icrc1/ledger/src/lib.rs` (L1174–1223) opens a range scan on `ALLOWANCES_MEMORY` (a `StableBTreeMap<AccountSpender, StorableAllowance, ...>`) starting at `start_account_spender` and iterates forward. The `AccountSpender` ordering is defined at L372–378:

```rust
impl std::cmp::Ord for AccountSpender {
    fn cmp(&self, other: &Self) -> std::cmp::Ordering {
        self.account
            .cmp(&other.account)
            .then_with(|| self.spender.cmp(&other.spender))
    }
}
```

`Account::cmp` includes both `owner` and `subaccount`. After all spenders for `{A, Some([1;32])}` are exhausted, the next key in the map is `{A, Some([2;32])}` (same owner, lexicographically larger subaccount). The termination guard at L1204–1206 is:

```rust
if account_spender.account.owner != from.owner {
    break;
}
```

Since `account_spender.account.owner == from.owner` still holds, the loop continues and pushes allowances for `{A, Some([2;32])}` into the result. The public entrypoint at `rs/ledger_suite/icrc1/ledger/src/main.rs` L1214–1232 is a `#[query]` method with no caller identity check — `from_account` is taken directly from the argument.

## Impact Explanation

An unprivileged caller can enumerate allowances (spender identity, approved amount, expiration timestamp) for every subaccount of any target principal `A` by issuing a single `icrc103_get_allowances` call with `from_account = {A, Some([0;32])}`. Up to `max_take_allowances` (default 500) entries are returned per call, and the caller can page through all subaccounts. This constitutes unauthorized disclosure of financial data (approved amounts and counterparty identities) stored in the ICRC-1 ledger, a production in-scope financial integration. This matches the **High** impact class: "Significant ledger security impact with concrete user or protocol harm."

## Likelihood Explanation

The exploit requires no privileges, no cycles beyond a normal query call, and no special conditions. It is reproducible on any replica running the affected ledger canister with two `icrc2_approve` calls from different subaccounts of the same principal. The endpoint is callable by any IC principal.

## Recommendation

Replace the owner-only comparison with a full `Account` equality check:

```rust
// Before (buggy):
if account_spender.account.owner != from.owner {
    break;
}

// After (correct):
if account_spender.account != from {
    break;
}
```

This ensures the scan terminates as soon as the iterated key's account diverges from the exact requested `from` account (owner **and** subaccount).

## Proof of Concept

```
Setup (state-machine test or local replica):
  principal A = <test principal>
  icrc2_approve(from={A, Some([1;32])}, spender=spender_X, amount=100)
  icrc2_approve(from={A, Some([2;32])}, spender=spender_Y, amount=200)

Call (any unprivileged principal):
  icrc103_get_allowances({
    from_account: Some({owner: A, subaccount: Some([1;32])}),
    prev_spender: None,
    take: None
  })

Expected:
  [ {from_account: {A, Some([1;32])}, to_spender: spender_X, allowance: 100} ]

Actual (buggy):
  [ {from_account: {A, Some([1;32])}, to_spender: spender_X, allowance: 100},
    {from_account: {A, Some([2;32])}, to_spender: spender_Y, allowance: 200} ]
```

The second entry is returned because the break guard at L1204 only checks `owner`, not the full `Account`. A deterministic integration test using `PocketIC` or the `ic-state-machine-tests` framework can confirm this with two approvals and one query call.
