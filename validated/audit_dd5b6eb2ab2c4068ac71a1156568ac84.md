Audit Report

## Title
Permanent `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` Exhaustion via Unauthenticated `canister_status` Panic-on-Unwrap After IC Callback Rollback — (`rs/nns/handlers/root/impl/canister/canister.rs`)

## Summary

The publicly accessible `canister_status` update method on the NNS root canister decrements `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` before making an inter-canister call to the management canister, then calls `.unwrap()` on the result in the callback. When the management canister rejects (root is not a controller of the target), the `.unwrap()` panics, causing a Wasm trap. The IC rolls back all callback-phase state changes — including the `SlotLoan::drop` increment — while the pre-call decrement remains committed. Any unauthenticated caller can repeat this 167 times to drive `available_slot_count` to zero, permanently blocking `update_canister_settings`, `change_canister_controllers`, `take_canister_snapshot`, and `load_canister_snapshot` through the NNS root canister until a governance-initiated root canister upgrade is executed.

## Finding Description

**Root cause — slot decrement committed before await, increment rolled back on panic:**

In `LimitedOutstandingCallsManagementCanisterClient::canister_status`:

```rust
async fn canister_status(&self, ...) -> Result<...> {
    let _loan = self.try_borrow_slot()?;   // decrements available_slot_count
    self.inner.canister_status(canister_id_record).await
}
```

`try_borrow_slot` at [1](#0-0)  decrements `available_slot_count` synchronously before the `.await`. On the IC, state is committed at every inter-canister call boundary. The decremented value is durable once the outbound call is enqueued.

`SlotLoan::drop` at [2](#0-1)  increments `available_slot_count` when the loan is dropped. This drop occurs during the callback execution.

The outer handler at [3](#0-2)  calls `.unwrap()` on the result after the await. When the management canister rejects, the callback execution sequence is:
1. `_loan` drops → `available_slot_count` incremented (callback-phase state change)
2. `.unwrap()` panics → Wasm trap
3. IC rolls back all callback-phase state changes → the increment is undone
4. `available_slot_count` remains at the committed (decremented) value

**No authorization check:**

The `canister_status` endpoint at [3](#0-2)  has no `check_caller_is_*` guard. The `.did` file at [4](#0-3)  confirms it is a public method callable by any principal.

**Slot counter initialized to 167:**

`AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` at [5](#0-4)  is initialized to 167 in a `thread_local!` that is reset only on canister upgrade/reinstall, not on `post_upgrade` from stable memory.

**VIPs also blocked at zero:**

The zero-check in `try_borrow_slot` at [6](#0-5)  returns `Err` for all callers when `available_slot_count == 0`, including VIP callers (NNS canisters) who have `used_slot_count = 0` but still hit the guard.

**Affected governance-gated operations:**

`update_canister_settings`, `change_canister_controllers`, `take_canister_snapshot`, and `load_canister_snapshot` all call `new_management_canister_client()` at [7](#0-6) , which wraps the client with `LimitedOutstandingCallsManagementCanisterClient` backed by the same `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT`. These operations fail once the counter reaches zero.

**Existing test does not assert slot invariant:**

`test_canister_status_call_tracking` at [8](#0-7)  exercises the exact failure path (root not a controller) and confirms the call returns an error, but only asserts on the `ProxiedCanisterCallsTracker` metric — it never checks that `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` is unchanged after the failed call.

## Impact Explanation

After 167 unauthenticated calls to `canister_status` with any canister ID that root does not control, `available_slot_count` reaches 0. All subsequent calls to `update_canister_settings`, `change_canister_controllers`, `take_canister_snapshot`, and `load_canister_snapshot` through the NNS root canister return `Err(SysTransient)`, blocking NNS governance proposals that exercise these paths. Recovery requires a governance proposal to upgrade the root canister (which resets the `thread_local!`); the upgrade path itself goes through the lifeline canister directly and is not affected. This matches: **High — Application/platform-level DoS with concrete NNS governance impact, not based on raw volumetric DDoS.**

## Likelihood Explanation

The `canister_status` endpoint is callable by any anonymous principal at zero privilege. The management canister deterministically rejects `canister_status` for any canister root does not control, making the panic path fully reliable. The attack requires exactly 167 update calls, each costing a small amount of cycles. It is fully automatable with no victim interaction required.

## Recommendation

1. **Immediate**: Replace `.unwrap()` with proper error propagation in the `canister_status` handler:
   ```rust
   canister_status_response.map_err(|(code, msg)| ic_cdk::trap(&format!("canister_status failed ({code}): {msg}")))?
   ```
   Or change the return type to `Result<CanisterStatusResult, ...>`. The same applies to `stop_or_start_nns_canister` at [9](#0-8)  (governance-gated, lower urgency).

2. **Structural**: Persist `available_slot_count` to stable memory and restore it in `post_upgrade`, so the counter survives upgrades without requiring a full reinstall. Alternatively, move slot accounting to a pre-call hook that survives callback rollback.

3. **Test**: Extend `test_canister_status_call_tracking` to assert that `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` is unchanged after a failed `canister_status` call, and add a test that calls it 167 times and verifies the counter does not reach zero.

## Proof of Concept

```
# Using any canister ID not controlled by NNS root (e.g., a freshly created canister)
for i in 1..=167:
    dfx canister call nns-root canister_status '(record { canister_id = principal "<non-controlled-id>" })'
    # Each call returns an error but permanently decrements available_slot_count by 1

# Now available_slot_count == 0
dfx canister call nns-root canister_status '(record { canister_id = principal "<any-id>" })'
# Returns: Err(SysTransient, "Unavailable. Maybe, try again later?")

# Governance-initiated update_canister_settings also now fails:
# submit NNS proposal -> root.update_canister_settings(...) -> Err(SysTransient)
```

Reproducible as a deterministic integration test using `StateMachine` (same framework as `test_canister_status_call_tracking`): call `canister_status` 167 times with a non-controlled canister ID, then assert `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` equals 0 and that a subsequent `update_canister_settings` governance call fails.

fn test_canister_status_call_tracking() {
