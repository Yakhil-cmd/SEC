### Title
`require_running` Guard in `factory_update_address_version` Callback Permanently Breaks XCC Router Version Accounting, Draining User wNEAR on Every Subsequent Call — (`engine/src/contract_methods/xcc.rs`)

### Summary

A race condition between a legitimate `pause_contract` admin action and an in-flight XCC router-deploy callback causes `factory_update_address_version` to abort before writing the router's version to engine storage. Because the wNEAR `STORAGE_AMOUNT` charge is an irreversible EVM-level transfer that already occurred during the originating EVM transaction, and because the version key remains `None`, every subsequent XCC call from the same address re-enters `DeployNeeded { create_needed: true }`, charges the user another `STORAGE_AMOUNT` (2 NEAR), and fails again — indefinitely.

### Finding Description

**Step 1 — EVM execution charges wNEAR atomically.**

In `engine-precompiles/src/xcc.rs`, when `get_code_version_of_address` returns `None` (no router deployed yet), the precompile computes `required_near = attached_near + STORAGE_AMOUNT` and immediately executes an EVM-level `transferFrom` of that amount from the caller's EVM address to the engine's implicit EVM address. [1](#0-0) 

This transfer is committed as part of the EVM state transition. It is not rolled back if subsequent NEAR promises fail.

**Step 2 — Deploy batch + callback are scheduled.**

`handle_precompile_promise` in `engine/src/xcc.rs` schedules a NEAR promise batch (`CreateAccount` + `Transfer(STORAGE_AMOUNT)` + `DeployContract` + `initialize`) and attaches `factory_update_address_version` as a callback on that batch. [2](#0-1) 

**Step 3 — Deploy batch succeeds; admin pauses the engine.**

The deploy batch executes successfully: the router sub-account is created on NEAR. Before the callback receipt is processed, the owner calls `pause_contract`, setting `state.is_paused = true`. [3](#0-2) 

**Step 4 — `factory_update_address_version` callback aborts on `require_running`.**

The callback fires. The very first statement inside `with_hashchain` is `require_running(&state::get_state(&io)?)`, which returns `Err(ERR_PAUSED)` because the engine is now paused. The function exits before reaching `xcc::set_code_version_of_address`. [4](#0-3) 

The version key for the address is never written.

**Step 5 — Downstream callbacks cascade-fail.**

`withdraw_wnear_to_router` is attached as a callback to `factory_update_address_version`. It checks `promise_result_check()` and returns `ERR_CALLBACK_OF_FAILED_PROMISE` when the predecessor failed. [5](#0-4) 

`send_refund` (attached to `withdraw_wnear_to_router`) and the user's actual XCC call also fail. The `STORAGE_AMOUNT` wNEAR transferred in Step 1 is now stranded in the engine's implicit EVM address.

**Step 6 — Engine is resumed; state is permanently broken.**

After `resume_contract`, `get_code_version_of_address` still returns `None` for the address. [6](#0-5) 

`AddressVersionStatus::new` maps `None` → `DeployNeeded { create_needed: true }`. [7](#0-6) 

**Step 7 — Every subsequent XCC call drains 2 NEAR.**

On the next XCC call the precompile again charges `STORAGE_AMOUNT`. The deploy batch now tries `CreateAccount` on an account that already exists — NEAR rejects this, the batch fails, `factory_update_address_version` receives `Some(false)` and returns `ERR_ROUTER_DEPLOY_FAILED`, the version remains `None`, and the cycle repeats.

There is no admin escape hatch: `factory_update_address_version` requires both `assert_private_call()` (predecessor == current account) and `promise_result_check() == Some(true)`, so it cannot be invoked directly to repair the version. [8](#0-7) 

### Impact Explanation

Every XCC call from the affected address permanently loses `STORAGE_AMOUNT` = 2 NEAR worth of wNEAR (transferred to the engine's implicit EVM address with no recovery path). The user's XCC functionality is permanently broken until a contract upgrade adds a repair function. This constitutes repeated, compounding loss of user funds — **High: Temporary (escalating to permanent) freezing of funds**.

### Likelihood Explanation

NEAR's asynchronous receipt model means there are typically one or more blocks between the deploy batch execution and the `factory_update_address_version` callback receipt. Any `pause_contract` call landing in that window triggers the bug. Pausing is a routine owner operation (security incidents, upgrades, maintenance). No malicious intent is required; the owner need not even be aware of in-flight XCC callbacks. The window is narrow per individual call but widens under load.

### Recommendation

Remove the `require_running` guard from `factory_update_address_version`. This callback is purely a bookkeeping write that records a deployment that has already succeeded on-chain; it carries no security risk when the engine is paused and must not be blocked by pause state. Alternatively, gate only the `require_running` check behind a flag that exempts self-callbacks originating from deploy batches.

### Proof of Concept

1. Deploy Aurora engine locally (sandbox).
2. Initiate an XCC call from a fresh EVM address (no router deployed). Confirm `STORAGE_AMOUNT` wNEAR is deducted from the caller.
3. After the deploy batch receipt executes but before the `factory_update_address_version` callback receipt, inject a `pause_contract` call from the owner account.
4. Allow the callback receipt to execute. Assert it panics with `ERR_PAUSED`.
5. Assert `get_code_version_of_address(address) == None`.
6. Call `resume_contract`.
7. Initiate a second XCC call from the same address. Assert another `STORAGE_AMOUNT` is deducted and the call fails.
8. Repeat step 7 to confirm the loop is unbounded.
