Audit Report

## Title
Unbounded tokio task spawning via crafted `StrippedBlockProposal` ingress/IDKG ID vectors — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/assembler.rs`)

## Summary

A Byzantine peer below the consensus fault threshold can serve a crafted `StrippedBlockProposal` whose `unstripped_consensus_message_id` matches a legitimately advertised block ID while carrying an arbitrarily large `ingress_messages` or `stripped_idkg_dealings` vector. `assemble_message` spawns one unbounded tokio task per entry with no count cap, and each task enters an infinite retry loop. The `JoinSet` is only dropped when the bouncer fires (up to 3 s later, or longer if the block remains "wanted"), during which all N tasks are live and consuming heap. The attack is repeatable every block interval, sustaining memory pressure that can degrade or halt consensus liveness on the targeted replica.

## Finding Description

**Root cause — no vector length cap at deserialization or task-spawn time.**

`TryFrom<pb::StrippedBlockProposal> for StrippedBlockProposal` iterates `value.ingress_messages` and `value.stripped_idkg_dealings` with no length guard: [1](#0-0) 

The axum router serving these artifacts explicitly disables body-size limits: [2](#0-1) 

**ID check is insufficient.** For a `StrippedBlockProposal`, `id()` is derived solely from `unstripped_consensus_message_id`, not from the ingress/IDKG vectors: [3](#0-2) 

A Byzantine peer sets `unstripped_consensus_message_id` to a legitimately advertised block ID and stuffs `ingress_messages` with N fake entries. The ID check passes; the vector length is never checked.

**One task spawned per entry, no cap.** `assemble_message` calls `missing_stripped_messages()` and immediately spawns one `JoinSet` task per returned ID: [4](#0-3) 

`missing_stripped_messages()` returns every entry from both vectors unconditionally: [5](#0-4) 

The comment at line 440 notes "at most 1000 elements" as an expected invariant, but this is never enforced: [6](#0-5) 

**Each spawned task runs an infinite retry loop.** `download_stripped_message` uses `with_max_elapsed_time(None)` and loops forever until it successfully downloads the artifact: [7](#0-6) [8](#0-7) 

For fake IDs that don't exist in any peer's pool, `get_or_fetch` finds nothing in the local pool and falls through to `download_stripped_message`, which loops indefinitely: [9](#0-8) 

**Bouncer cancellation is post-spawn.** The bouncer check runs inside the join loop, after all N tasks are already live. The bouncer refresh period is 3 seconds, and if the block remains "wanted" by consensus (which it will, since `unstripped_consensus_message_id` is legitimate), the window extends further: [10](#0-9) [11](#0-10) 

**Final hash validation is post-hoc.** `try_assemble` validates the reconstructed block hash only after all tasks complete — after the resource damage is done: [12](#0-11) 

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

At N = 100,000 fake ingress IDs, the victim replica spawns 100,000 tokio tasks, each holding its own async stack frame, a cloned `StrippedMessageId`, `ConsensusMessageId`, transport handle, logger, metrics arc, and peer receiver. Heap consumption reaches hundreds of MB. At N = 1,000,000, OOM is plausible on typical replica hardware. Because the attack is repeatable every block interval, memory pressure is sustained, degrading or halting consensus participation on the targeted replica.

## Likelihood Explanation

The attacker requires only a single Byzantine node in the same subnet — below the f-fault threshold. No key material, governance majority, or network-level attack is required. The crafted payload is a valid protobuf message with a legitimate `unstripped_consensus_message_id` and an inflated ID vector. The attack is repeatable every block interval (~1 second), making sustained impact straightforward.

## Recommendation

1. **Cap vector length before spawning tasks**: After `missing_stripped_messages()` returns, assert the count is within a protocol-defined bound (e.g., `MAX_INGRESS_MESSAGES_PER_BLOCK + MAX_IDKG_DEALINGS_PER_BLOCK`). Return `AssembleResult::Unwanted` if exceeded.
2. **Enforce the cap at deserialization**: Add a length check inside `TryFrom<pb::StrippedBlockProposal>` so malformed payloads are rejected before reaching `assemble_message`.
3. **Re-enable body size limits**: Remove or scope `DefaultBodyLimit::disable()` to avoid accepting arbitrarily large artifact payloads from peers.

// Heap grows by hundreds of MB; consensus stalls until bouncer fires (>=3s).
