### Title
`L1_TX_NATIVE_PRICE` Hardcoded Too Low Causes L1→L2 Transactions to Bypass Native Resource Accounting — (`basic_bootloader/src/bootloader/constants.rs`)

---

### Summary

`L1_TX_NATIVE_PRICE` is hardcoded to `10` with an explicit TODO acknowledging the value is a placeholder. This is orders of magnitude lower than the operator-set `native_price` used for L2 transactions (500–911,615 in tests). Because `nativePerGas = gasPrice / L1_TX_NATIVE_PRICE`, L1→L2 priority transactions receive a `nativePerGas` ratio that is 50×–91,000× higher than equivalent L2 transactions, effectively making the `deltaGas` adjustment mechanism irrelevant for L1 transactions and allowing them to consume RISC-V proving cycles without paying proportionally.

---

### Finding Description

The constant is defined at: [1](#0-0) 

```rust
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
```

It is consumed in `prepare_and_check_resources` for every L1→L2 priority transaction: [2](#0-1) 

```rust
let native_price = L1_TX_NATIVE_PRICE;          // = 10
let native_per_gas = if is_priority_op {
    ...
    u256_try_to_u64(&gas_price.div_ceil(native_price))  // = gasPrice / 10
    ...
};
```

The `nativeLimit` is then `gasLimit * nativePerGas`: [3](#0-2) 

After execution, the `deltaGas` correction is applied: [4](#0-3) 

```rust
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
```

**The math with `L1_TX_NATIVE_PRICE = 10`:**

| Parameter | L1 tx (`native_price=10`) | L2 tx (`native_price=500,000`) |
|---|---|---|
| `gas_price` | 1,500 | 1,500 |
| `nativePerGas` | **150** | **≈ 1** |
| `nativeLimit` (70k gas) | 10,500,000 | 70,000 |
| `nativeUsed / nativePerGas` (for 10M native used) | 66,667 | 10,000,000 |
| `deltaGas` (gasUsed=21k) | **−0** (no charge) | **+9,979,000** (massive charge) |

With `nativePerGas = 150`, the term `nativeUsed / nativePerGas` is always small relative to `gasUsed`, so `deltaGas ≤ 0` and no extra gas is ever charged. The `deltaGas` mechanism — which exists precisely to ensure native resource consumption is reflected in fees — is completely neutralized for L1→L2 transactions.

The test file itself confirms the value is intentionally hardcoded and not derived from the operator: [5](#0-4) 

```rust
// L1 Txs have a hard-coded native price of 10
let native_price = 10;
```

The `MAX_NATIVE_COMPUTATIONAL` cap (`2^35 ≈ 34 billion cycles`) is defined here: [6](#0-5) 

---

### Impact Explanation

Any L1→L2 priority transaction sender can submit transactions that consume up to `MAX_NATIVE_COMPUTATIONAL = 2^35` RISC-V proving cycles while paying fees as if they consumed `nativeUsed / 150` cycles. For L2 transactions under a realistic operator `native_price`, the same proving work would trigger a `deltaGas` surcharge that could be millions of gas units. L1 transactions pay none of this surcharge. This breaks the economic model of the double-resource accounting system: proving costs are not properly reflected in L1→L2 fees, enabling sustained consumption of prover capacity at a fraction of the intended cost.

---

### Likelihood Explanation

The entry path is fully permissionless — any user can submit L1→L2 priority transactions. The misconfiguration is live in the production codebase with an explicit `TODO (EVM-1157)` acknowledging the value is wrong. No privileged access, oracle manipulation, or external dependency is required.

---

### Recommendation

Resolve `TODO (EVM-1157)` before deployment. `L1_TX_NATIVE_PRICE` must be set to a value consistent with the operator-set `native_price` for L2 transactions so that `nativePerGas` for L1 transactions is in the same order of magnitude as for L2 transactions. This ensures the `deltaGas` mechanism correctly charges L1→L2 senders for native resource consumption. [1](#0-0) 

---

### Proof of Concept

1. Submit an L1→L2 priority transaction: `gas_price = 1500`, `gas_limit = 70,000`, calldata that triggers a native-heavy computation (e.g., many `EXP` opcodes, each costing `EXP_PER_BYTE_NATIVE_COST = 5,000` native cycles).
2. `nativePerGas = ceil(1500 / 10) = 150`; `nativeLimit = 70,000 × 150 = 10,500,000`.
3. Transaction executes, consuming e.g. 8,000,000 native cycles and 30,000 EVM gas.
4. `deltaGas = 8,000,000 / 150 − 30,000 = 53,333 − 30,000 = 23,333` → a small positive delta, charged.
5. Now submit the same transaction as an L2 transaction with operator `native_price = 500,000`:
   - `nativePerGas = ceil(1500 / 500,000) = 1`; `nativeLimit = 70,000`.
   - Transaction runs out of native resources immediately and reverts.
6. The L1 transaction succeeds and pays ~53,333 gas total. The equivalent L2 transaction cannot even execute. The L1 sender consumed 8,000,000 native cycles for the cost of 53,333 gas; an L2 sender would need `gas_price × 8,000,000` native budget, requiring a much higher gas price or gas limit.
