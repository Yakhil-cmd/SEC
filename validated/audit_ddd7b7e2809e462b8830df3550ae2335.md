Audit Report

## Title
SNS Token Burn Inflates Treasury Valuation, Pushing Medium-Regime Treasury Into Large-Regime Rate Limit - (File: `rs/sns/governance/token_valuation/src/lib.rs`)

## Summary

`fetch_icps_per_sns_token()` derives the current SNS token price by dividing the initial price by `current_supply / initial_supply`. Because `current_supply` is read live from `icrc1_total_supply`, any SNS token holder can burn their own tokens via a standard `icrc1_transfer` to the minting account, reducing `current_supply`, inflating `icps_per_sns_token`, and inflating the treasury's XDR valuation. If the treasury is in the "medium" rate-limit regime (100,000–1,200,000 XDR), a targeted burn can push the valuation above 1,200,000 XDR into the "large" regime, where the 7-day transfer cap drops from 25% of treasury tokens to a fixed 300,000 XDR equivalent — blocking legitimate `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals.

## Finding Description

**Root cause — live supply read with no ceiling clamp:**

`IcpsPerSnsTokenClient::fetch_icps_per_sns_token()` concurrently fetches three values and computes:

```
total_inflation     = current_supply_e8s / initial_supply_e8s
icps_per_sns_token  = initial_icps_per_sns_token / total_inflation
```

`current_supply_e8s` comes from a live `icrc1_total_supply` call that any actor can influence by burning tokens. [1](#0-0) [2](#0-1) 

**Burn mechanism — unprivileged, standard ICRC-1:**

The ICRC-1 ledger treats any `icrc1_transfer` whose `to` field equals the minting account as a burn, reducing `total_supply`. This path requires no special privilege — any token holder can execute it. [3](#0-2) 

**Rate-limit regime transition:**

`ProposalsAmountTotalUpperBound::in_tokens()` computes the 7-day cap:
- **Medium** (≤ 1,200,000 XDR): cap = `0.25 × balance_tokens` (constant in token terms)
- **Large** (> 1,200,000 XDR): cap = `300,000 XDR / xdrs_per_token` = `300,000 × T / V` tokens

At the boundary V = 1,200,000 XDR both caps equal 0.25 × T. Above the boundary, the large cap = `300,000 × T / V < 0.25 × T`. Therefore, burning tokens to push V from just below 1,200,000 to significantly above 1,200,000 **decreases** the effective token cap, causing proposals that were valid under the medium cap to be rejected. [4](#0-3) [5](#0-4) 

**No ceiling guard on `icps_per_sns_token`:**

`clamp_xdrs_per_icp` enforces a `MIN_XDRS_PER_ICP` floor to prevent artificially low ICP prices from pushing valuations into the permissive "small" regime, but there is no symmetric `MAX_ICPS_PER_SNS_TOKEN` ceiling to prevent artificially high token prices from pushing valuations into the restrictive "large" regime. [6](#0-5) [7](#0-6) 

**Valuation computed at submission time:**

The valuation is computed and stored in `ActionAuxiliary` during proposal validation, so a burn executed before submission permanently affects the outcome for that proposal. [8](#0-7) [9](#0-8) 

## Impact Explanation

An unprivileged SNS token holder can permanently suppress the 7-day treasury transfer allowance for `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals. Legitimate SNS governance operations are blocked without any privileged access. This matches the allowed impact: **"Application/platform-level DoS … or SNS … security impact with concrete user or protocol harm"** — severity **Medium** given the economic cost and narrow precondition window. [10](#0-9) 

## Likelihood Explanation

**Medium.** The attacker must permanently burn their own SNS tokens (real economic cost). The attack is most effective when the treasury valuation is near the 1,200,000 XDR boundary — a condition that is fully observable on-chain. The attacker does not need to front-run any specific transaction; burning before a proposal is submitted is sufficient. SNS tokens are publicly acquirable via the SNS swap. The `icrc1_transfer`-to-minting-account burn path requires no special privilege. [11](#0-10) 

## Recommendation

Apply a symmetric clamp on `icps_per_sns_token` analogous to the existing `MIN_XDRS_PER_ICP` floor. Define a `MAX_ICPS_PER_SNS_TOKEN` constant and clamp the computed value in `fetch_icps_per_sns_token()` or in `ProposalsAmountTotalUpperBound::in_tokens()` before the regime is determined. This prevents supply manipulation (in either direction) from moving the treasury across regime boundaries. Alternatively, use the initial supply directly as the reference supply (i.e., fix `total_inflation = 1`) or use a governance-set reference supply, so that post-genesis supply changes do not affect the treasury rate limit. [6](#0-5) [2](#0-1) 

## Proof of Concept

**Setup (corrected from the submitted PoC, which incorrectly started in the large regime):**

1. Treasury holds T = 100,000 SNS tokens. Initial swap price: 1 SNS = 0.5 ICP. ICP = 10 XDR/ICP. Initial supply S₀ = 10,000,000 SNS.
2. Current supply S = 8,500,000 SNS (some deflation already). `total_inflation = 0.85`. `icps_per_token = 0.5 / 0.85 ≈ 0.588`. Treasury valuation V = 100,000 × 0.588 × 10 ≈ 588,000 XDR → **medium regime**. 7-day cap = 0.25 × 100,000 = **25,000 SNS tokens**.
3. SNS governance submits a proposal to transfer 24,500 SNS tokens. This is within the 25,000-token cap → **proposal accepted**.
4. Attacker calls `icrc1_transfer` with `to = minting_account` for 4,500,000 of their own SNS tokens. New supply S = 4,000,000. `total_inflation = 0.4`. `icps_per_token = 0.5 / 0.4 = 1.25`. V = 100,000 × 1.25 × 10 = **1,250,000 XDR → large regime**. 7-day cap = 300,000 / (1.25 × 10) = **24,000 SNS tokens**.
5. The same 24,500-token proposal is now submitted. 24,500 > 24,000 → **proposal rejected** with "Amount is too large."

A deterministic PocketIC integration test can reproduce this by: initializing an SNS with the above parameters, calling `icrc1_transfer` to the minting account from an attacker principal, then attempting to submit a `TransferSnsTreasuryFunds` proposal and asserting it is rejected.     fn in_tokens(mut valuation: Valuation) -> Result<Decimal, ProposalsAmountTotalLimitError> {
        Self::clamp_xdrs_per_icp(&mut valuation);

        let ValuationFactors {
            tokens: balance_tokens,
            icps_per_token,
            xdrs_per_icp,
        } = valuation.valuation_factors;

        let self_ = Self::from_valuation_xdr(valuation.to_xdr());
        let result_tokens = match self_ {
            Self::NoLimit => balance_tokens,

            Self::Fraction(fraction) => balance_tokens
                .checked_mul(fraction)
                // Overflow should not be possible, since fraction is supposed to be at most 1.0.
                .ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "Unable to perform {balance_tokens} * {fraction}.",
                    ))
                })?,

            Self::Xdr(max_xdr) => {
                let xdrs_per_token = xdrs_per_icp.checked_mul(icps_per_token).ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "XDRs per token could not be calculated from valuation: {valuation:?}"
                    ))
                })?;

                // Calculate the inverse conversion rate.
                if xdrs_per_token == Decimal::from(0) {
                    // This is not reachable, because in this case, valuation.to_xdr() would return
                    // 0, and in that case, we would have taken the NoLimit branch.
                    return Err(ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "It appears that the tokens have zero value in XDR. valuation = {valuation:?}"
                    )));
                }
                let tokens_per_xdr = xdrs_per_token.inv();

                max_xdr.checked_mul(tokens_per_xdr).ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "Max tokens could not be calculated with valuation: {valuation:?}",
                    ))
                })?
            }
        };

        Ok(result_tokens)
    }

    fn from_valuation_xdr(valuation_xdr: Decimal) -> Self {
        // Ideally, this would be checked at compile time. In principal, this should be possible,
        // since all the inputs are const, but I'm not sure how to do that. Therefore,
        // debug_assert_eq is used instead, and should be very nearly as good, because this will be
        // run during CI.
        debug_assert_eq!(
            Self::MAX_MEDIUM_TREASURY_SIZE_XDR.checked_mul(ONE_QUARTER),
            Some(Self::MAX_XDR),
        );

        if valuation_xdr <= Self::MAX_SMALL_TREASURY_SIZE_XDR {
            return Self::NoLimit;
        }

        if valuation_xdr <= Self::MAX_MEDIUM_TREASURY_SIZE_XDR {
            return Self::Fraction(ONE_QUARTER);
        }

        Self::Xdr(Self::MAX_XDR)
    }
