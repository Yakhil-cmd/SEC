### Title
`ExitToEthereum` Precompile Burns Tokens Without a Failure-Refund Callback - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile burns user tokens (ETH or ERC-20) inside the EVM and then schedules a NEAR promise to call `withdraw` on the eth-connector. Unlike the `ExitToNear` precompile, **no failure callback is attached**. If the `withdraw` promise fails for any reason after the EVM state is committed, the burned tokens are permanently unrecoverable.

---

### Finding Description

In `ExitToEthereum::run()`, after computing the `withdraw_promise`, the code wraps it as a bare `PromiseArgs::Create` with no callback:

```rust
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [1](#0-0) 

This promise is emitted as a log and later scheduled by the NEAR runtime. If the `withdraw` call on the eth-connector fails (e.g., the connector is paused, gas is exhausted, or any other runtime error), the NEAR promise reverts — but the EVM state (the token burn) was already committed in the prior receipt and **cannot be rolled back**.

Contrast this with `ExitToNear::run()`, which conditionally wraps the transfer promise in a `PromiseArgs::Callback` that calls `exit_to_near_precompile_callback` on failure to re-mint the burned tokens:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            method: "exit_to_near_precompile_callback".to_string(),
            ...
        },
    })
};
``` [2](#0-1) 

The `exit_to_near_precompile_callback` handler explicitly calls `engine::refund_on_error` to re-mint burned ERC-20 tokens or return ETH when the base promise fails:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
``` [3](#0-2) 

No equivalent mechanism exists for `ExitToEthereum`. The `withdraw` promise is fire-and-forget.

The ERC-20 contract (`EvmErc20`) burns tokens **before** calling the precompile, so the burn is part of the same EVM execution that succeeds:

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    // ... calls ExitToEthereum precompile
}
``` [4](#0-3) 

Once the EVM transaction is finalized and the promise is scheduled, there is no path to recover the burned tokens if the promise fails.

---

### Impact Explanation

**Impact: High — Permanent freezing/loss of user funds.**

If the `withdraw` promise on the eth-connector fails after the EVM transaction commits, the user's ERC-20 tokens (or ETH) are permanently destroyed with no recovery path. The NEP-141 tokens on the NEAR side remain in Aurora's account (since the `withdraw` failed), but the corresponding EVM-side tokens are gone. The user loses their bridged assets entirely.

---

### Likelihood Explanation

**Likelihood: Medium.**

The `withdraw` call on the eth-connector can fail in realistic scenarios:

1. **Connector paused**: The eth-connector supports pausability. The `engine_withdraw` feature can be paused by an admin (demonstrated in `test_withdraw_from_near_pausability`). Any user who calls `withdrawToEthereum` while the connector is paused will permanently lose their tokens. [5](#0-4) 

2. **Gas exhaustion**: If `costs::WITHDRAWAL_GAS` is insufficient for the connector's `withdraw` logic in a given runtime state, the promise fails silently. [6](#0-5) 

3. **Connector accounting mismatch**: Any edge case causing the connector's `withdraw` to revert (e.g., insufficient NEP-141 balance due to an accounting discrepancy) results in permanent token loss.

The `ExitToNear` refund test explicitly acknowledges this class of failure and tests the recovery path. No equivalent test or recovery path exists for `ExitToEthereum`. [7](#0-6) 

---

### Recommendation

Add a failure callback to the `ExitToEthereum` promise, analogous to `exit_to_near_precompile_callback`. The callback should:

1. Check `handler.promise_result(0)` for failure.
2. On failure, call `engine::refund_on_error` to re-mint the burned ERC-20 tokens (or restore ETH balance) to the original sender.

The `RefundCallArgs` struct already supports both ETH and ERC-20 refund paths:

```rust
pub struct RefundCallArgs {
    pub recipient_address: Address,
    pub erc20_address: Option<Address>,
    pub amount: RawU256,
}
``` [8](#0-7) 

The `ExitToEthereum` promise should be changed from `PromiseArgs::Create` to `PromiseArgs::Callback` with a new `exit_to_ethereum_precompile_callback` method that mirrors the existing `exit_to_near_precompile_callback` logic.

---

### Proof of Concept

1. User holds 1000 bridged USDC (ERC-20) on Aurora.
2. User calls `EvmErc20.withdrawToEthereum(eth_recipient, 1000)`.
3. `_burn(msg.sender, 1000)` executes — ERC-20 balance goes to 0.
4. `ExitToEthereum` precompile runs, creates `PromiseArgs::Create(withdraw_promise)` targeting the eth-connector.
5. EVM transaction succeeds and is committed. ERC-20 tokens are gone.
6. The eth-connector's `withdraw` promise fails (e.g., `engine_withdraw` is paused).
7. No callback exists to re-mint the tokens.
8. User has 0 ERC-20 tokens on Aurora and 0 USDC on Ethereum. Funds are permanently lost.
