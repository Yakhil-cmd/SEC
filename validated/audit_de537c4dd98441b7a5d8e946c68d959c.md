Audit Report

## Title
Hot Keys Unconditionally Inherited by Child Neuron When Spawned to a New Controller - (File: rs/nns/governance/src/governance.rs)

## Summary
When `spawn_neuron` is called with a `new_controller` different from the parent neuron's controller, the child neuron is built with `with_hot_keys(parent_neuron.hot_keys.clone())` unconditionally, transferring all of the spawner's hot keys to a neuron now controlled by a different principal. The new controller receives a neuron they believe they fully own, but the spawner's hot key holders retain the ability to vote, change followees, and join the Neurons' Fund on that neuron without the new controller's knowledge or consent.

## Finding Description
In `rs/nns/governance/src/governance.rs`, `spawn_neuron` resolves `child_controller` from `spawn.new_controller` (lines 2651–2655): if a new controller is provided it is used directly, otherwise the parent's controller is used. The child neuron is then constructed at lines 2704–2718:

```rust
let child_neuron = NeuronBuilder::new(
    child_nid,
    to_subaccount,
    child_controller,   // may be a completely different principal
    ...
)
.with_hot_keys(parent_neuron.hot_keys.clone())  // ← unconditional copy
.with_followees(parent_neuron.followees.clone())
...
.build();
```

There is no conditional branch that clears or filters `hot_keys` when `child_controller != parent_neuron.controller()`. The `Spawn` protobuf message explicitly supports `new_controller` as an optional field (governance.proto lines 862–871), making cross-controller spawning a first-class, documented use case.

Authorization for non-privileged neuron operations is checked via `is_authorized_to_vote` → `is_hotkey_or_controller` in `rs/nns/governance/src/neuron/types.rs` (lines 243–255):

```rust
fn is_hotkey_or_controller(&self, principal: &PrincipalId) -> bool {
    self.is_controlled_by(principal) || self.hot_keys.contains(principal)
}
```

Because the inherited hot keys are present in the child neuron's `hot_keys` list, any principal that was a hot key on the parent neuron passes this check on the child neuron, regardless of who the child's controller is. No existing guard removes or filters inherited hot keys post-spawn.

Exploit path:
1. User A (controller of neuron N) adds hot key `H1` to neuron N.
2. User A calls `manage_neuron` → `Spawn { new_controller: Some(UserB), ... }`.
3. Governance creates child neuron C with `controller = UserB` and `hot_keys = [H1]`.
4. User A (via `H1`) calls `manage_neuron` on C with `RegisterVote`, `Follow`, or `JoinCommunityFund`.
5. Governance accepts because `is_hotkey_or_controller` returns `true` for `H1` on C.
6. User B, unaware of `H1`, never removes it; User A retains persistent governance influence over User B's neuron.

## Impact Explanation
The concrete impact is unauthorized access to a neuron's governance capabilities: an unprivileged ingress sender (hot key holder `H1`) can vote on NNS proposals, change followees, and join the Neurons' Fund on behalf of a neuron they do not control. `JoinCommunityFund` is financially consequential and not automatically reversible. This matches the allowed High impact: *"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."* Severity: **High ($2,000–$10,000)**.

## Likelihood Explanation
Adding hot keys for convenience (e.g., a mobile signing key) is common practice. Spawning to a different controller is the primary on-chain mechanism for transferring maturity-derived ICP to another party. The combination is therefore a routine, non-adversarial scenario that silently produces the vulnerability. Deliberate exploitation requires only that the spawner retain a hot key before spawning; the recipient has no in-protocol notification of inherited hot keys and no reason to audit them. Likelihood of unintentional authorization leakage: **medium-to-high**. Likelihood of deliberate exploitation: **low-to-medium**.

## Recommendation
In `spawn_neuron`, when `child_controller` differs from `parent_neuron.controller()`, build the child neuron with an empty hot key list instead of cloning the parent's:

```rust
let child_hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};
// ...
.with_hot_keys(child_hot_keys)
```

If hot key inheritance across controller boundaries is ever desired, it should be an explicit opt-in field in the `Spawn` message, not the default behavior.

## Proof of Concept
Minimal deterministic integration test (PocketIC or existing NNS integration test harness):

1. Create governance with a funded neuron N owned by `principal_a`.
2. Call `manage_neuron(N, AddHotKey { hot_key_principal: hot_key_h1 })` as `principal_a`.
3. Call `manage_neuron(N, Spawn { new_controller: Some(principal_b), nonce: None, percentage_to_spawn: Some(100) })` as `principal_a`; record returned child neuron ID `C`.
4. Assert `child_neuron(C).controller == principal_b`.
5. Assert `child_neuron(C).hot_keys.contains(hot_key_h1)` — this will pass, confirming the bug.
6. Call `manage_neuron(C, RegisterVote { proposal_id: P, vote: Yes })` as `hot_key_h1`; assert the call succeeds.
7. Call `manage_neuron(C, JoinCommunityFund {})` as `hot_key_h1`; assert the call succeeds and `C` is now in the Neurons' Fund.
