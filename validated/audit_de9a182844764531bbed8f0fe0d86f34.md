### Title
Any Unprivileged NEAR Account Can Trigger `deploy_upgrade()`, Bypassing Owner-Only Restriction - (File: engine/src/lib.rs)

### Summary

The `deploy_upgrade()` contract entrypoint in `engine/src/lib.rs` is callable by any NEAR account. It only checks that the contract is running (`require_running`), but never verifies that the caller is the contract owner. The companion function `stage_upgrade()` — which stores the code to be deployed — correctly enforces `require_owner_only`. The missing check on `deploy_upgrade()` allows any unprivileged NEAR account to force the deployment of staged contract code at the earliest possible moment after the delay expires, preventing the owner from canceling or replacing a staged upgrade.

---

### Finding Description

`deploy_upgrade()` is the NEAR contract method that finalizes a two-step upgrade process. The first step, `stage_upgrade()`, stores new WASM bytecode and records the earliest block height at which it may be deployed. The second step, `deploy_upgrade()`, reads that stored bytecode and calls `Runtime::self_deploy` to replace the running contract.

`stage_upgrade()` in `engine/src/contract_methods/admin.rs` correctly enforces owner-only access: [1](#0-0) 

However, `deploy_upgrade()` in `engine/src/lib.rs` only calls `require_running` and omits any caller identity check: [2](#0-1) 

The `require_running` and `require_owner_and_running` helpers are defined in `engine/src/contract_methods/mod.rs`: [3](#0-2) 

`require_owner_and_running` checks both that the contract is not paused **and** that `predecessor_account_id == state.owner_id`. `deploy_upgrade()` calls only `require_running`, skipping the owner identity check entirely.

---

### Impact Explanation

Once the upgrade delay has elapsed, any NEAR account can call `deploy_upgrade()` and atomically:

1. Replace the running Aurora Engine WASM with the staged bytecode (`Runtime::self_deploy`).
2. Erase the staged-code storage key, preventing any further cancellation.

If the owner staged an upgrade and subsequently discovered a critical bug in the staged code — intending to cancel by re-staging corrected code — an attacker monitoring the chain can call `deploy_upgrade()` the moment the delay expires, before the owner's cancellation transaction is processed. The buggy contract is then live and irrevocable. Depending on the nature of the bug in the staged code, this can result in:

- **Permanent freezing of all user funds** held in the Aurora EVM (if the deployed code breaks execution or storage).
- **Direct theft of user funds** (if the deployed code contains a vulnerability the attacker can exploit immediately after deployment).

The impact maps to **Critical** (permanent fund freeze / direct fund theft) because the Aurora Engine holds ETH and ERC-20 balances for all users bridged to Aurora.

---

### Likelihood Explanation

The attack is straightforward and requires no special privileges:

1. Attacker watches the NEAR chain for a `stage_upgrade` call by the owner.
2. Attacker reads `upgrade_delay_blocks` from contract state to know when deployment becomes eligible.
3. At the target block, attacker submits a `deploy_upgrade()` transaction.

No private keys, governance capture, or social engineering is required. The attacker only needs a funded NEAR account and knowledge of the block height. The window of opportunity is every block after the delay expires until the owner cancels — which could be indefinitely long if the owner is unaware of the race condition.

---

### Recommendation

Add an owner check to `deploy_upgrade()` in `engine/src/lib.rs`, consistent with every other mutative administrative entrypoint:

```rust
pub extern "C" fn deploy_upgrade() {
    let mut io = Runtime;
    let state = state::get_state(&io).sdk_unwrap();
    // Add the missing owner check:
    require_owner_and_running(&state, &io.predecessor_account_id())
        .map_err(ContractError::msg)
        .sdk_unwrap();
    let index = internal_get_upgrade_index();
    if io.block_height() <= index {
        sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
    }
    Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
    io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
}
```

### Proof of Concept

1. Owner calls `stage_upgrade(code_bytes)` — stores buggy WASM and sets `upgrade_index = current_block + upgrade_delay_blocks`.
2. Owner discovers the bug and prepares a corrected `stage_upgrade` call.
3. Attacker monitors the chain. At block `upgrade_index + 1`, attacker submits `deploy_upgrade()` with any NEAR account.
4. `require_running` passes (contract is running). Block height check passes. `Runtime::self_deploy` executes, replacing the contract with the buggy WASM. The staged-code key is erased.
5. Owner's corrective `stage_upgrade` transaction arrives too late — the buggy contract is already live.
6. Attacker exploits the vulnerability in the newly deployed buggy contract to drain user funds or freeze them permanently.

The root cause is the missing `require_owner_only` (or `require_owner_and_running`) call in `deploy_upgrade()`: [2](#0-1)

pub fn require_running(state: &state::EngineState) -> Result<(), ContractError> {
    if state.is_paused {
        return Err(errors::ERR_PAUSED.into());
    }
    Ok(())
}

pub fn require_paused(state: &state::EngineState) -> Result<(), ContractError> {
    if !state.is_paused {
        return Err(errors::ERR_RUNNING.into());
    }
    Ok(())
}

pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}

pub fn require_owner_and_running(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    require_running(state)?;
    require_owner_only(state, predecessor_account_id)?;

    Ok(())
}
