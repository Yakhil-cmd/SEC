### Title
Silo Mode `WhitelistKind::Address` Bypass via ERC-20 `transfer()` Causes Token Freeze for Non-Whitelisted Recipients - (`engine/src/engine.rs`, `engine/src/contract_methods/silo/mod.rs`)

---

### Summary

In Aurora Engine's Silo Mode, the `WhitelistKind::Address` whitelist is enforced at the bridge entry point (`receive_erc20_tokens`) to prevent non-whitelisted EVM addresses from receiving ERC-20 tokens. However, the access check in `assert_access` only validates the **sender** of an EVM transaction, not the recipient of any ERC-20 `transfer()` call executed within it. A whitelisted address can therefore call `erc20.transfer(non_whitelisted_address, amount)` and successfully deliver tokens to a non-whitelisted address. Because non-whitelisted addresses are also blocked from submitting any transactions by `is_allow_submit`, those tokens become frozen and unspendable until an admin intervenes.

---

### Finding Description

Aurora Engine's Silo Mode uses a `WhitelistKind::Address` list to control two things simultaneously:

1. **Who can submit EVM transactions** — enforced by `is_allow_submit`, which checks `transaction.address` (the signer).
2. **Who can receive ERC-20 tokens via the bridge** — enforced by `is_allow_receive_erc20_tokens`, which also checks the `WhitelistKind::Address` list.

The bridge-level guard is in `receive_erc20_tokens`:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

This correctly redirects bridge mints away from non-whitelisted addresses. However, the EVM-level access check in `assert_access` only inspects the **transaction signer**:

```rust
let allowed = if transaction.to.is_some() {
    silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
} else {
    silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
};
``` [2](#0-1) 

And `is_allow_submit` checks only the sender's address:

```rust
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}
``` [3](#0-2) 

There is no check on the **recipient** of an ERC-20 `transfer()` call that executes inside the EVM. A whitelisted address can therefore submit a transaction calling `EvmErc20.transfer(non_whitelisted_addr, amount)`. The `assert_access` gate passes (sender is whitelisted), the EVM executes the standard OpenZeppelin `transfer`, and the non-whitelisted address receives the tokens with no further validation.

The `WhitelistKind::Address` enum documents that addresses in the list "can submit transactions": [4](#0-3) 

Because the non-whitelisted recipient cannot submit any transaction, it cannot call `withdrawToNear`, `withdrawToEthereum`, or any other function on the ERC-20 contract. The tokens are stranded.

---

### Impact Explanation

**High — Temporary (potentially permanent) freezing of funds.**

Any ERC-20 tokens transferred to a non-whitelisted address in Silo Mode are immediately unspendable. The non-whitelisted address is blocked from submitting transactions by `is_allow_submit`, so it cannot call `transfer`, `withdrawToNear`, or `withdrawToEthereum`. The tokens remain frozen until a Silo admin explicitly adds the recipient address to the `WhitelistKind::Address` list. If the admin does not act (e.g., the recipient is unknown or the admin is unresponsive), the freeze is permanent.

---

### Likelihood Explanation

**High.** The precondition is only that the sender holds ERC-20 tokens and is whitelisted. Any whitelisted participant can trigger this for any non-whitelisted address — including by accident (wrong address) or by a malicious third party who tricks a whitelisted user into sending tokens to a controlled but non-whitelisted address. No special privileges, key compromise, or governance capture is required.

---

### Recommendation

Enforce the `WhitelistKind::Address` check on the **recipient** of ERC-20 transfers, not only on the transaction sender. One approach is to override the `transfer` and `transferFrom` functions in `EvmErc20.sol` to call a precompile that validates the recipient against the Silo whitelist before executing the transfer, reverting if the recipient is not whitelisted. Alternatively, the engine can intercept ERC-20 transfer calls at the Rust level and validate the ABI-decoded recipient address against `is_allow_receive_erc20_tokens` before allowing execution to proceed. [5](#0-4) 

---

### Proof of Concept

**Setup:** Silo Mode is active. `WhitelistKind::Address` whitelist is enabled. Address `A` is whitelisted. Address `B` is not whitelisted. The Silo has a configured `erc20_fallback_address`.

1. Bridge 1000 NEP-141 tokens to address `A` via `ft_on_transfer`. `receive_erc20_tokens` checks `is_allow_receive_erc20_tokens(A)` → passes → `A` receives 1000 ERC-20 tokens. [1](#0-0) 

2. Address `A` submits an EVM transaction: `erc20.transfer(B, 1000)`. `assert_access` checks `is_allow_submit(predecessor, A)` → passes (A is whitelisted). The EVM executes the standard ERC-20 `transfer`. Address `B` now holds 1000 tokens. [2](#0-1) 

3. Address `B` attempts to submit any transaction (e.g., `erc20.withdrawToNear(...)`). `assert_access` checks `is_allow_submit(predecessor, B)` → `is_address_allowed(B)` → `B` is not in the `WhitelistKind::Address` list → **`EngineErrorKind::NotAllowed`**. The 1000 tokens are frozen. [6](#0-5)
