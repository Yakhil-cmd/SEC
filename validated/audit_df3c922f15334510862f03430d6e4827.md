Audit Report

## Title
`SenderInfoContent` Signed Digest Does Not Bind to the Request Sender, Enabling Cross-User Replay of Canister-Signed Attestations - (File: `rs/types/types/src/messages/http.rs`)

## Summary
`SenderInfoContent` signs only the raw `info` bytes under the `"ic-sender-info"` domain separator. The `sender` principal, `canister_id`, and `ingress_expiry` are absent from the attesting canister's signed payload. Any party who obtains a valid `(info, signer, sig)` triple can attach it verbatim to their own ingress message, and `verify_sender_info_canister_sig` will accept it because none of its three checks bind the attestation to the originating sender. A target canister that uses `msg_caller_info_data()` for authorization decisions will then grant the attacker the privileges attested for a different principal.

## Finding Description

`SenderInfoContent` is a newtype over the raw `info` bytes:

```
rs/types/types/src/messages/http.rs L342
pub struct SenderInfoContent<'a>(pub &'a [u8]);
```

Its `write_signed_bytes_without_domain_separator` appends only those bytes:

```
L344-348
impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.0);
    }
}
```

The full signed payload is therefore `\x0Eic-sender-info || info_bytes`. The `sender` principal, `canister_id`, and `ingress_expiry` are present in the outer message-ID hash (`representation_independent_hash_call_or_query`, L54-78), but that hash is signed by the **user**, not by the attesting canister.

`verify_sender_info_canister_sig` (L494-544) performs exactly three checks:
1. `sender_pubkey` is a valid canister-signature public key.
2. The canister ID encoded in `sender_pubkey` matches `sender_info.signer`.
3. The canister signature over `SenderInfoContent(&sender_info.info)` is valid.

None of these checks compare the `sender` field of the ingress message against anything inside `info_bytes`. The attacker constructs a new ingress message with their own `sender` principal, copies the victim's `(info, signer, sig)` triple verbatim into `sender_info`, signs the outer envelope with their own key (valid, because the message-ID hash includes `sender_info` as opaque bytes), and submits. All three checks in `verify_sender_info_canister_sig` pass.

## Impact Explanation

A target canister that calls `ic0::msg_caller_info_data_copy` / `ic0::msg_caller_info_signer_copy` to make authorization decisions (e.g., "this session has admin role", "this user passed KYC") will receive `caller = attacker_principal` alongside `sender_info` attesting to attributes that were signed for a different principal. The attacker inherits those privileges without the attesting canister ever having approved them for the attacker's principal. This constitutes unauthorized access to canister-controlled resources and matches the High bounty impact class: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."

## Likelihood Explanation

Medium. The attacker must obtain a valid `(info, signer, sig)` triple. Realistic acquisition paths include: (a) any application where the attesting canister signs the same `info_bytes` for a role or attribute that is not user-specific (e.g., `b"role=admin"` or a KYC-passed blob), allowing any holder of that blob to replay it under a different principal; (b) observation of another user's HTTPS request at a compromised or colluding boundary node. Path (a) requires no network access and no privilege — it is available to any user who legitimately receives a role attestation and wishes to replay it under a second identity. The replay requires no threshold corruption, no governance action, and no privileged access.

## Recommendation

Include the `sender` principal (and optionally `canister_id`) in the `SenderInfoContent` signed digest, mirroring how `Delegation` binds `pubkey`, `expiration`, and optionally `targets` into its signed bytes (L595-611). For example:

```rust
pub struct SenderInfoContent<'a> {
    pub info: &'a [u8],
    pub sender: &'a [u8],
    pub canister_id: &'a [u8],
}

impl SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        // representation-independent hash of {info, sender, canister_id}
        use RawHttpRequestVal::*;
        let map = btreemap! {
            "info"       => Bytes(self.info),
            "sender"     => Bytes(self.sender),
            "canister_id"=> Bytes(self.canister_id),
        };
        bytes.extend_from_slice(&hash_of_map(&map, |k, v| hash_key_val(k, v)));
    }
}
```

`verify_sender_info_canister_sig` must then pass the request's `sender` and `canister_id` into `SenderInfoContent` so the verification covers those fields.

## Proof of Concept

**Unit test plan (safe, local, no mainnet):**

```rust
#[test]
fn cross_user_replay_of_sender_info_succeeds() {
    let rng = &mut reproducible_rng();
    let root_of_trust = RootOfTrust::new_random(rng);
    let builder = default_verifier().with_root_of_trust(root_of_trust.public_key);

    // Attesting canister C signs info_bytes for user A.
    let signer = CanisterSigner { seed: b"seed".to_vec(), canister_id: CANISTER_ID_SIGNER,
        root_public_key: root_of_trust.public_key, root_secret_key: root_of_trust.secret_key };
    let info_bytes = b"role=admin".to_vec();
    let sig_a = signer.sign(&SenderInfoContent(&info_bytes));
    let sender_info = RawSignedSenderInfo {
        info: Blob(info_bytes), signer: Blob(CANISTER_ID_SIGNER.get().into_vec()),
        sig: Blob(sig_a.0),
    };

    // Attacker (user B) uses a DIFFERENT CanisterSigner (different seed/identity)
    // but reuses the SAME sender_info triple originally produced for user A.
    let attacker_signer = CanisterSigner { seed: b"attacker_seed".to_vec(), ... };
    let request = HttpRequestBuilder::new_update_call()
        .with_ingress_expiry_at(CURRENT_TIME)
        .with_authentication(Direct(CanisterSignature(attacker_signer)))
        .with_sender_info(sender_info)   // victim's triple, attacker's envelope key
        .build();

    // This should FAIL (sender not bound) but currently PASSES:
    assert_eq!(builder.build().validate_request(&request), Ok(()));
}
```

This test directly demonstrates that `verify_sender_info_canister_sig` accepts a `sender_info` triple produced for one identity when presented under a completely different identity's envelope key, because `SenderInfoContent` does not include the sender principal in the signed digest.
