Audit Report

## Title
Unbounded `limit` parameter in `search_transactions` enables unauthenticated OOM crash of ICRC-1 Rosetta node — (`rs/rosetta-api/icrc1/src/data_api/services.rs`)

## Summary
The `search_transactions` function in the ICRC-1 Rosetta API uses `MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST` (10,000) only as a default when `limit` is absent, never as a server-side cap. An attacker supplying `limit=9223372036854775807` (`i64::MAX`) causes the value to pass `try_into::<u64>()` without error, become the SQL `LIMIT` clause verbatim, and force the process to materialize every matching block into a heap-allocated `Vec<RosettaBlock>`, crashing the Rosetta node with OOM. No authentication is required.

## Finding Description
In `rs/rosetta-api/icrc1/src/common/constants.rs` line 18, `MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST` is defined as `10000`. [1](#0-0) 

In `rs/rosetta-api/icrc1/src/data_api/services.rs` lines 385–391, the effective limit is resolved as:

```rust
let limit: u64 = request
    .limit
    .unwrap_or(MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST as i64)
    .try_into()
    ...?;
```

The constant is used only as the `unwrap_or` fallback. When a caller supplies an explicit value, it is accepted as-is. `i64::MAX` (9223372036854775807) converts to `u64` without error, and no subsequent `.min(MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST)` guard exists. [2](#0-1) 

The uncapped value is then cast back to `i64` and injected directly as the SQL `LIMIT` clause:

```rust
command.push_str("LIMIT :limit ");
parameters.push((":limit".to_string(), rusqlite::types::Value::Integer(limit as i64)));
``` [3](#0-2) 

SQLite receives `LIMIT 9223372036854775807`, effectively no limit. All matching rows are fetched and pushed into a `Vec<RosettaBlock>` before the response is serialized: [4](#0-3) 

## Impact Explanation
The Rosetta node is a standalone OS process serving the public Rosetta API. A single malformed request exhausts process heap memory and triggers an OOM kill, taking down the entire Rosetta service. This matches the allowed High impact: **"Application/platform-level DoS, crash, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"** — a single crafted HTTP request, not a volumetric flood, causes the crash.

## Likelihood Explanation
The `/search/transactions` endpoint is publicly reachable with no authentication. The payload is a standard Rosetta `SearchTransactionsRequest` JSON body with `limit` set to `i64::MAX`. A single request against a node with a sufficiently large block database is sufficient to trigger OOM. The attack is trivially repeatable after each restart.

## Recommendation
Enforce a hard server-side cap immediately after parsing the limit:

```rust
let limit: u64 = request
    .limit
    .unwrap_or(MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST as i64)
    .try_into()
    .map_err(|err| Error::request_processing_error(...))?;

let limit = limit.min(MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST);
```

Apply the same pattern to `offset` and `max_block` for defense-in-depth.

## Proof of Concept
```rust
// Unit test sketch (safe, local, in-memory)
let storage = StorageClient::new_in_memory().await.unwrap();
// Insert a large number of blocks into storage...
let request = SearchTransactionsRequest {
    limit: Some(i64::MAX),
    ..Default::default()
};
// Without the fix, this materializes all blocks into memory.
// With a sufficiently large DB, this OOMs the process.
let _ = services::search_transactions(&storage, request, "ICP".into(), 8).await;
```
