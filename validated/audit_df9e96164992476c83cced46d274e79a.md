### Title
Hardcoded `chain_id = 1` in `HeaderAndHistory::new()` Causes Incorrect EIP-155 Transaction Validation on Non-Mainnet Settlement Layers - (`File: basic_bootloader/src/bootloader/block_flow/ethereum/block_header.rs`)

---

### Summary

`HeaderAndHistory::new()` unconditionally assigns `chain_id = 1u64` instead of reading the actual chain ID from the oracle-supplied block header. This value is then exposed through the `BasicBlockMetadata::chain_id()` implementation and consumed by the Ethereum transaction validation path to enforce EIP-155 replay protection. On any settlement layer other than Ethereum mainnet (chain ID ≠ 1), the check will either silently accept mainnet-signed transactions (replay attack) or reject all legitimately-signed local transactions.

---

### Finding Description

In `HeaderAndHistory::new()`, the settlement-layer chain ID is hardcoded:

```rust
// basic_bootloader/src/bootloader/block_flow/ethereum/block_header.rs, line 198
let chain_id = 1u64;
```

The struct then implements `BasicBlockMetadata<EthereumIOTypesConfig>`, returning this constant from `chain_id()`:

```rust
impl BasicBlockMetadata<EthereumIOTypesConfig> for HeaderAndHistory {
    fn chain_id(&self) -> u64 {
        self.chain_id   // always 1
    }
    ...
}
```

`HeaderAndHistory` is the block-level metadata type for the Ethereum execution environment:

```rust
// basic_bootloader/src/bootloader/block_flow/ethereum/metadata_op.rs, line 19-23
pub type EthereumBlockMetadata = SystemMetadata<
    EthereumIOTypesConfig,
    HeaderAndHistory,
    EthereumTransactionMetadata<{ MAX_BLOBS_PER_BLOCK }>,
>;
```

During transaction parsing, `parse_and_compute_signed_hash` receives `expected_chain_id` from `system.metadata.chain_id()` and enforces it for every typed transaction:

```rust
// basic_bootloader/src/bootloader/transaction/rlp_encoded/mod.rs, lines 65-67, 75-77, 86-88, 98-100
if tx.chain_id != expected_chain_id {
    return Err(InvalidTransaction::InvalidChainId.into());
}
```

And for EIP-155 legacy transactions:

```rust
// basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs, lines 97-99
let min_v = U256::from(35) + U256::from(expected_chain_id) * U256::from(2);
if !(legacy_signature.v == min_v || legacy_signature.v == min_v + U256::ONE) {
    return Err(InvalidTransaction::InvalidEncoding.into());
}
```

Because `expected_chain_id` is always `1`, the system will:
1. Accept any transaction signed for Ethereum mainnet (chain ID 1), regardless of the actual settlement layer.
2. Reject every EIP-155 transaction signed for the real settlement layer chain ID (e.g., Sepolia = 11155111, Holesky = 17000, or any custom chain).

The actual Ethereum block header parsed from the oracle **does contain** the correct chain ID implicitly (via the block's EIP-155 context), but the code never reads it — it discards the oracle-supplied value and substitutes the literal `1`.

---

### Impact Explanation

**Replay attack / incorrect state transition**: An adversary holding a signed Ethereum mainnet transaction can submit it verbatim to a ZKsync OS instance whose settlement layer is any chain with ID ≠ 1. The hardcoded `expected_chain_id = 1` causes the signature check to pass, crediting the attacker's mainnet-signed transfer or contract call on the wrong chain. Conversely, all legitimate users of that chain whose wallets sign with the correct chain ID will have every EIP-155 transaction rejected with `InvalidChainId`, making the chain unusable for standard EVM tooling.

In proving mode the same hardcoded value is used, so the prover generates a valid proof for a state transition that accepted replayed mainnet transactions — the settlement layer contract will verify and finalize an incorrect state root.

---

### Likelihood Explanation

ZKsync OS is explicitly designed to support multiple settlement layers and secondary chains. The `settlement_layer_chain_id` concept is already present throughout the codebase (stored in `SystemContext`, updated via service transactions, included in `BatchOutput`). The Ethereum execution environment is the path used when ZKsync OS settles on an Ethereum-compatible chain. Any testnet deployment (Sepolia, Holesky) or custom EVM chain used as a settlement layer triggers this bug immediately for every EIP-155 transaction.

---

### Recommendation

Read the chain ID from the oracle-supplied block header rather than hardcoding it. The `PectraForkHeader` struct already contains all necessary fields; the chain ID should be passed in from the oracle query or derived from a configurable constant that matches the actual deployment target. At minimum, add a compile-time or runtime assertion that the hardcoded value matches the expected deployment chain ID, and expose the chain ID as a configurable parameter to `HeaderAndHistory::new()`.

---

### Proof of Concept

1. Deploy ZKsync OS targeting Sepolia (chain ID 11155111) as the settlement layer.
2. Construct an EIP-1559 transaction signed for chain ID 11155111 (the correct chain).
3. Submit it through the Ethereum execution environment.
4. `parse_and_compute_signed_hash` calls `system.metadata.chain_id()` → returns `1`.
5. The check `if tx.chain_id != expected_chain_id` fires (`11155111 != 1`) → `InvalidChainId` error.
6. The transaction is rejected despite being correctly signed.

Conversely:
1. Take any existing Ethereum mainnet EIP-1559 transaction (chain ID 1).
2. Submit it to the same Sepolia-targeted ZKsync OS instance.
3. `tx.chain_id == 1 == expected_chain_id` → check passes.
4. The mainnet transaction is executed on Sepolia, producing an incorrect state transition that the prover will certify as valid.
