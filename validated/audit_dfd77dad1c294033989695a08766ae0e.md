Audit Report

## Title
Missing `owner` anonymity check in `update_balance` allows minting ckDOGE to the anonymous principal's uncontrolled account - (File: rs/dogecoin/ckdoge/minter/src/main.rs)

## Summary

The ckDOGE minter's `update_balance` endpoint guards against anonymous **callers** via `check_anonymous_caller()`, but never validates that the `owner` field in `UpdateBalanceArgs` is non-anonymous. A non-anonymous attacker can pass `owner = Some(Principal::anonymous())`, causing the minter to mint ckDOGE to `Account { owner: anonymous, subaccount: None }` — an account no one controls. The deposited DOGE is permanently locked in the minter's UTXO set and the minted ckDOGE is permanently inaccessible, breaking the 1:1 backing invariant.

## Finding Description

**Root cause — `check_anonymous_caller` only inspects `msg_caller`, not `args.owner`:**

`rs/dogecoin/ckdoge/minter/src/main.rs` lines 89–96 call `check_anonymous_caller()` before delegating to the shared logic:

```rust
#[update]
async fn update_balance(args: UpdateBalanceArgs) -> Result<Vec<UtxoStatus>, UpdateBalanceError> {
    check_anonymous_caller();
    check_postcondition(
        ic_ckbtc_minter::updates::update_balance::update_balance(args, &DOGECOIN_CANISTER_RUNTIME)
            .await,
    )
}
```

`check_anonymous_caller` (lines 145–149) only checks `ic_cdk::api::msg_caller()`. It never inspects `args.owner`. A non-anonymous caller passes this guard unconditionally regardless of what `args.owner` contains.

**Shared `update_balance` logic resolves `caller_account` from `args.owner` without an anonymous check:**

In `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs` lines 148–167, the only guard on the resolved owner is the minter self-deposit check (`args.owner.unwrap_or(caller) == runtime.id()`). There is no `assert_ne!(owner, Principal::anonymous())`:

```rust
let caller = runtime.caller();
if args.owner.unwrap_or(caller) == runtime.id() {
    ic_cdk::trap("cannot update minter's balance");
}
// ...
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),
    subaccount: args.subaccount,
};
```

This contrasts directly with `get_doge_address` in `rs/dogecoin/ckdoge/minter/src/updates/get_doge_address.rs` lines 12–18, which **does** have the anonymous check:

```rust
let owner = owner.unwrap_or_else(ic_cdk::api::msg_caller);
let account = Account { owner, subaccount };
assert_ne!(
    owner,
    Principal::anonymous(),
    "the owner must be non-anonymous"
);
```

**Minting path uses `icrc1_transfer`, which accepts the anonymous principal as a recipient:**

`DogeCanisterRuntime::mint_ckbtc` (in `rs/dogecoin/ckdoge/minter/src/lib.rs` lines 105–112) delegates to `ic_ckbtc_minter::updates::update_balance::mint`, which calls `icrc1_transfer` (lines 458–480 of `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`). The ICRC-1 ledger does not block transfers to the anonymous principal — confirmed by `test_anonymous_transfers` in `rs/ledger_suite/tests/sm-tests/src/lib.rs` lines 673–679, which explicitly verifies that `transfer(..., anon.0, ...)` succeeds.

**Exploit path:**

1. Attacker computes the Dogecoin address for `Account { owner: anonymous, subaccount: None }` off-chain using the minter's public ECDSA key and the deterministic derivation path defined in `rs/dogecoin/ckdoge/minter/src/updates/get_doge_address.rs` lines 44–53 (the `get_doge_address` canister endpoint cannot be used because it traps on anonymous owner, but the derivation is fully deterministic and public).
2. Attacker deposits DOGE to that address and waits for `min_confirmations`.
3. Attacker calls `update_balance(UpdateBalanceArgs { owner: Some(Principal::anonymous()), subaccount: None })` from any non-anonymous principal.
4. `check_anonymous_caller()` passes (caller is non-anonymous).
5. Shared logic resolves `caller_account = Account { owner: anonymous, subaccount: None }`.
6. Minter mints ckDOGE to the anonymous account via `icrc1_transfer` — succeeds.
7. The anonymous account's ckDOGE is permanently inaccessible; `retrieve_doge_with_approval` also calls `check_anonymous_caller()` so no one can burn it.
8. The DOGE UTXO is permanently absorbed into the minter's UTXO set with no corresponding redeemable ckDOGE.

## Impact Explanation

This is a **High** severity finding. An unprivileged attacker can permanently break the ckDOGE 1:1 backing invariant by sacrificing their own DOGE. The minted ckDOGE is permanently inaccessible (no one controls the anonymous principal), and the underlying DOGE is permanently locked in the minter's UTXO set. This constitutes illegal minting and permanent loss of in-scope chain-key/ledger assets (ckDOGE and DOGE), with concrete and irreversible protocol harm. The attack is self-funded (no financial gain for the attacker), which limits scale but does not eliminate the impact — repeated small attacks permanently degrade the protocol's solvency guarantee. This matches the allowed impact: *"Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."*

## Likelihood Explanation

Any non-anonymous principal can trigger this with no special privileges. The attacker must sacrifice their own DOGE, making it a deliberate griefing attack with no financial return. Accidental triggering is unlikely; deliberate exploitation is straightforward. The attack is repeatable and each instance permanently worsens the invariant violation.

## Recommendation

Add an explicit non-anonymous check on the resolved `owner` inside the shared `update_balance` function in `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`, immediately after resolving `caller_account`, mirroring the check already present in `get_doge_address`:

```rust
let owner = args.owner.unwrap_or(caller);
if owner == Principal::anonymous() {
    ic_cdk::trap("owner must be non-anonymous");
}
let caller_account = Account {
    owner,
    subaccount: args.subaccount,
};
```

Alternatively, add the check in the ckDOGE-specific wrapper in `rs/dogecoin/ckdoge/minter/src/main.rs` before delegating to the shared logic. Placing it in the shared logic is preferable as it also protects the ckBTC minter from the same issue.
