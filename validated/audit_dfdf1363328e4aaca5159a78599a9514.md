Audit Report

## Title
ICRC-21 `add_expiration` Silently Truncates `expires_at > i64::MAX` to Epoch 0 in `FieldsDisplayMessage` — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary

In `ConsentMessage::add_expiration`, the `FieldsDisplayMessage` arm casts a `u64` nanosecond timestamp to `i64` before dividing by `10^9`. For any `expires_at > i64::MAX` (e.g., `u64::MAX`), the cast wraps to a negative value; integer division toward zero yields `0`; the final `as u64` cast produces `0`. The hardware-wallet-facing `FieldsDisplayMessage` therefore stores `TimestampSeconds { amount: 0 }` (Unix epoch, Jan 1 1970) while the on-chain approval retains the original far-future timestamp unchanged, breaking the invariant that the consent message accurately reflects what the user is signing.

## Finding Description

**Root cause — `FieldsDisplayMessage` arm, lines 265 and 269:**

```rust
// packages/icrc-ledger-types/src/icrc21/responses.rs  L265, L269
let seconds = (expires_at as i64) / 10_i64.pow(9);
// ...
Value::TimestampSeconds { amount: seconds as u64 }
```

Arithmetic for `expires_at = u64::MAX`:
- `u64::MAX as i64` → `-1` (wrapping cast, defined Rust behavior)
- `-1_i64 / 1_000_000_000_i64` → `0` (truncation toward zero)
- `0_i64 as u64` → `0`

Result: `TimestampSeconds { amount: 0 }` — epoch 0.

**Why the `GenericDisplayMessage` arm does not silently corrupt (lines 232–233):**

```rust
let seconds = (ts as i64) / 10_i64.pow(9);          // 0
let nanos   = ((ts as i64) % 10_i64.pow(9)) as u32; // (-1 % 10^9) = -1 → wraps to 4_294_967_295
```

`replace_nanosecond(4_294_967_295)` fails (valid range 0–999_999_999) and returns the string `"Invalid nanosecond: 4294967295"` — visibly broken, not silently wrong.

**Why the ledger accepts `u64::MAX` (line 247 of `approvals.rs`):**

```rust
if expires_at.unwrap_or_else(remote_future) <= now {
    return Err(ApproveError::ExpiredApproval { now });
}
```

`u64::MAX <= now` is always `false` for any realistic `now` (~1.7 × 10¹⁸ ns), so the ledger stores the original `u64::MAX` expiration unchanged.

**`ApproveArgs` places no upper bound on `expires_at` (line 20 of `approve.rs`):**

```rust
pub expires_at: Option<u64>,
```

**Exploit flow:**
1. Attacker operates a dApp and constructs `icrc2_approve` args with `expires_at = u64::MAX`.
2. Attacker calls `icrc21_canister_call_consent_message` (public query, no privilege required).
3. The ledger returns a `FieldsDisplayMessage` with `TimestampSeconds { amount: 0 }`.
4. A hardware wallet implementing the ICRC-21 `FieldsDisplay` spec renders "Approval expiration: Jan 1 1970 00:00:00 UTC."
5. A user who interprets this as "already expired / harmless" signs the transaction.
6. The on-chain approval carries `expires_at = u64::MAX` (year 2554), granting the spender a permanent allowance.

## Impact Explanation

The ICRC-21 `FieldsDisplayMessage` path is the designated security mechanism for constrained hardware wallets (Ledger, Keystone, etc.) that cannot parse free-form text. The bug causes this mechanism to silently misrepresent the expiration of an ICRC-2 approval, enabling a malicious dApp to obtain a permanent spending allowance over a victim's account while the hardware wallet displays a date that has already elapsed. This constitutes a **High** severity impact: significant ICRC ledger security impact with concrete user-funds harm — unauthorized access to ledger funds where exploitation requires the attacker to operate a dApp and the victim to use a hardware wallet with ICRC-21 support.

## Likelihood Explanation

- `icrc21_canister_call_consent_message` is a public query call; no privilege, key, or governance action is required.
- Any dApp operator can craft the malicious args.
- The `FieldsDisplay` path is specifically designed for hardware wallets, which are the devices most likely to render the field value literally without additional validation.
- The displayed date (Jan 1 1970) is plausibly misread as "no expiration" or "already expired / safe to sign" by non-expert users.
- The attack is repeatable and deterministic.

## Recommendation

Replace the lossy `as i64` cast with pure `u64` arithmetic in both arms of `add_expiration`:

```rust
// FieldsDisplayMessage arm (line 265)
let seconds = expires_at / 1_000_000_000_u64;
fields_display.fields.push((
    "Approval expiration".to_string(),
    Value::TimestampSeconds { amount: seconds },
));
```

For the `GenericDisplayMessage` arm (lines 232–233), use `u64` division for `seconds` and derive `nanos` from the remainder before casting to `u32`, ensuring the value is always in `[0, 999_999_999]`:

```rust
let seconds = (expires_at / 1_000_000_000_u64) as i64; // safe: max ~1.8×10^10 < i64::MAX
let nanos   = (expires_at % 1_000_000_000_u64) as u32;
```

## Proof of Concept

```rust
// Differential invariant — no runtime required
let expires_at: u64 = u64::MAX;

// Buggy path (current code)
let seconds_buggy = (expires_at as i64) / 10_i64.pow(9); // = 0
let displayed     = seconds_buggy as u64;                 // = 0
assert_eq!(displayed, 0); // epoch 0 — WRONG

// Correct path
let seconds_correct = expires_at / 1_000_000_000_u64;    // = 18_446_744_073
assert_ne!(displayed, seconds_correct);                   // invariant violated
```
