### Title
Missing Expiry Parameter on Scheduled XCC Promises Allows Indefinite Stale Execution - (File: `etc/xcc-router/src/lib.rs`)

### Summary
The `execute_scheduled` function in the XCC Router contract has no expiry or deadline parameter. A promise scheduled via `CrossContractCallArgs::Delayed` is stored indefinitely and can be triggered by any caller at any future time, with no time-bound validity check. This is the direct analog of the missing `expires` parameter in the LOB `placeOrder` report: a time-sensitive operation (a NEAR cross-contract call carrying attached NEAR tokens) can be executed long after the user intended, under entirely different market or protocol conditions.

### Finding Description

The XCC Router contract exposes two relevant entry points:

**`schedule`** — callable only by the parent Aurora Engine, stores a `PromiseArgs` (including `attached_balance` in yoctoNEAR) keyed by a monotonically increasing nonce. No timestamp or deadline is recorded alongside the promise. [1](#0-0) 

**`execute_scheduled`** — callable by **anyone**, retrieves the stored promise by nonce and executes it immediately. There is no check against any expiry timestamp, block height, or deadline. [2](#0-1) 

The `PromiseArgs` type (and its sub-types `PromiseCreateArgs`, `PromiseWithCallbackArgs`, `NearPromise`) carry no expiry field: [3](#0-2) 

The EVM-level entry path is `CrossContractCallArgs::Delayed`, which causes the XCC precompile to call `schedule` on the router sub-account: [4](#0-3) 

There is no cancel function anywhere in the router. Once a promise is scheduled, the only way to remove it from storage is to execute it. [5](#0-4) 

### Impact Explanation

**High — Temporary freezing of funds / stale execution causing financial loss.**

When a user schedules a Delayed XCC call, the NEAR tokens required for the cross-contract call (`attached_balance`) are sourced from the router sub-account's NEAR balance, which was funded at sub-account creation time. Those funds remain locked in the router until `execute_scheduled` is called. Because:

1. There is no expiry on the stored promise.
2. There is no cancel/withdraw mechanism for the user.
3. `execute_scheduled` is open to any caller.

The user's NEAR is effectively frozen in the router for an unbounded period. Additionally, when the promise is eventually triggered (possibly by a third party), it executes with the original parameters (target account, method, args, attached balance) regardless of how much time has elapsed. If the target contract's state, token prices, or protocol parameters have changed, the user suffers an unintended financial outcome — the exact stale-order scenario described in the reference report.

### Likelihood Explanation

**Medium.** The `CrossContractCallArgs::Delayed` path is an intentional, documented feature used when an EVM call consumes most of the 300 Tgas budget and the user needs a fresh gas budget for the NEAR call. Any EVM user interacting with NEAR DeFi protocols via XCC is exposed. A griefing actor or MEV-equivalent on NEAR can monitor the router's `scheduled_promises` storage and delay or front-run execution to maximise the staleness of the call.

### Recommendation

Add an optional `expires_at` block-height or timestamp field to `PromiseArgs` (or as a separate parameter to `execute_scheduled`). In `execute_scheduled`, revert if `env::block_timestamp()` or `env::block_index()` exceeds the deadline:

```rust
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    // Example: store deadline alongside promise and check here
    if let Some(deadline) = promise.expires_at {
        if env::block_timestamp() > deadline {
            env::panic_str("ERR_PROMISE_EXPIRED");
        }
    }
    let promise_id = Self::promise_create(promise.args);
    env::promise_return(promise_id);
}
```

Additionally, add a `cancel_scheduled` function callable only by the parent (Aurora Engine, on behalf of the originating EVM address) so users can reclaim their locked NEAR if they change their mind.

### Proof of Concept

1. Alice (EVM address `0xAlice`) calls the XCC precompile with `CrossContractCallArgs::Delayed(PromiseArgs::Create(ft_transfer_promise))`, where `ft_transfer_promise` transfers 1000 wNEAR to Bob at the current exchange rate.
2. The Aurora Engine calls `schedule` on Alice's router sub-account. The promise is stored at nonce `0`. Alice's router NEAR balance is committed to the call.
3. Market conditions shift significantly (e.g., the target DeFi protocol changes its rate, or Alice's intended recipient changes).
4. Alice has no way to cancel the scheduled promise.
5. One week later, any third party calls `execute_scheduled({"nonce": "0"})` on Alice's router. The stale `ft_transfer` executes with the original parameters, transferring Alice's NEAR under conditions she no longer accepts. [6](#0-5) [4](#0-3)

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

    /// It is intentional that this function can be called by anyone (not just the parent).
    /// There is no security risk to allowing this function to be open because it can only
    /// act on promises that were created via `schedule`.

            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
