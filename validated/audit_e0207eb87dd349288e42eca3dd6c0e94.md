Audit Report

## Title
Missing On-Chain Recovery Path for Quarantined ckBTC Withdrawal Reimbursements Causes Permanent Fund Loss — (`rs/bitcoin/ckbtc/minter/src/reimbursement/mod.rs`)

## Summary

The ckBTC minter's `reimburse_withdrawals` function installs a `scopeguard` before the `mint_ckbtc` inter-canister call that, on any panic, permanently moves the reimbursement entry into a `Err(ReimbursedError::Quarantined)` terminal state. No canister endpoint exists to retry or release a quarantined entry. A user whose cancelled withdrawal lands in this state has their ckBTC permanently destroyed — the tokens were burned at withdrawal time and the minter will never re-mint them without an NNS governance upgrade proposal.

## Finding Description

**Entry into the vulnerable state (user-triggered):** A user calls `retrieve_btc_with_approval` with an amount requiring more UTXOs than `max_num_inputs_in_transaction`. The minter accepts the burn, then during batch processing detects `TooManyInputs` and calls `schedule_withdrawal_reimbursement`, placing the entry in `pending_withdrawal_reimbursements`.

**Quarantine trigger:** The periodic `reimburse_withdrawals` task (`rs/bitcoin/ckbtc/minter/src/reimbursement/mod.rs`, L67–71) installs a scopeguard before the `mint_ckbtc` await:

```rust
let prevent_double_minting_guard = scopeguard::guard(burn_index, |index| {
    state::mutate_state(|s| {
        state::audit::quarantine_withdrawal_reimbursement(s, index, runtime)
    });
});
```

If a panic occurs at any point during or after the `mint_ckbtc` call (L75–81) — before `ScopeGuard::into_inner` is reached at L107 — the guard fires and calls `quarantine_withdrawal_reimbursement` (`rs/bitcoin/ckbtc/minter/src/state.rs`, L1773–1777):

```rust
pub fn quarantine_withdrawal_reimbursement(&mut self, burn_index: LedgerBurnIndex) {
    self.pending_withdrawal_reimbursements.remove(&burn_index);
    self.reimbursed_withdrawals
        .insert(burn_index, Err(ReimbursedError::Quarantined));
}
```

This is a one-way transition: the entry is removed from `pending_withdrawal_reimbursements` and inserted as `Err(Quarantined)` into `reimbursed_withdrawals`. The status query (`rs/bitcoin/ckbtc/minter/src/state.rs`, L859–861) then returns `RetrieveBtcStatusV2::Unknown` to the user:

```rust
Err(err) => match err {
    ReimbursedError::Quarantined => RetrieveBtcStatusV2::Unknown,
},
```

**No recovery endpoint:** The minter's update methods are limited to `get_btc_address`, `get_withdrawal_account`, `retrieve_btc`, `update_balance`, and `icrc21`. A grep across all update files confirms zero references to `Quarantined` outside of tests. There is no endpoint — user-callable or admin-callable — to re-queue or release a quarantined reimbursement. The code comment at L34–35 of `reimbursement/mod.rs` explicitly states: *"will not be further processed without manual intervention."*

**Existing guard is insufficient:** The scopeguard correctly prevents double-minting, but the design provides no compensating recovery mechanism. The `QuarantinedWithdrawalReimbursement` event is recorded in the event log (`rs/bitcoin/ckbtc/minter/src/state/eventlog.rs`, L270–274), but this is observability only — it does not trigger any automated recovery.

## Impact Explanation

This is a concrete ledger conservation violation in an in-scope chain-key asset: ckBTC tokens are burned from the ICRC-1 ledger at withdrawal time, and the minter holds the accounting obligation to re-mint them on cancellation. Once quarantined, the minter permanently abandons that obligation with no automated or user-accessible remedy. The affected user's ckBTC is effectively destroyed. This matches the allowed High impact: *"Significant Chain Fusion, ck-token, ledger… security impact with concrete user or protocol harm."*

## Likelihood Explanation

Two conditions must coincide: (1) a withdrawal is cancelled and enters `pending_withdrawal_reimbursements` — a real production path confirmed by the `TooManyInputs` code path in `rs/bitcoin/ckbtc/minter/src/reimbursement/mod.rs` (L50–54) and the `should_cancel_and_reimburse_large_withdrawal` test; and (2) an unexpected panic occurs in the `reimburse_withdrawals` callback. Condition 2 is low-probability under normal operation, but the guard exists precisely because it is a real scenario. The existence of `minter_upgrade_2025_06_27.md` and `minter_upgrade_2026_03_20.md` confirms that the ckBTC minter has experienced production panics requiring emergency NNS upgrades. The combination is low-to-medium likelihood, but the impact when it occurs is total and permanent for the affected user.

## Recommendation

1. **Add a retry endpoint:** Expose an update method (callable by any user, gated on the caller matching the original `reimbursement_account`) that accepts a `burn_block_index`, checks whether `reimbursed_withdrawals[index] == Err(Quarantined)`, queries the ckBTC ledger for an existing mint with memo `ReimburseWithdrawal { withdrawal_id: burn_index }` to rule out a completed-but-unrecorded mint, and if none is found, re-inserts the entry into `pending_withdrawal_reimbursements` for the next `reimburse_withdrawals` run.
2. **Two-phase commit:** Record a `MintAttempted` event before the `mint_ckbtc` call. On recovery from quarantine, query the ledger to determine whether the mint succeeded before deciding to quarantine vs. complete.
3. **Metric/alert:** Emit a counter metric when `quarantine_withdrawal_reimbursement` is called so operators can detect affected users promptly without waiting for an NNS upgrade cycle.

## Proof of Concept

1. Deploy the ckBTC minter locally (PocketIC or local replica).
2. Configure `max_num_inputs_in_transaction` to a small value (e.g., 2).
3. Fund a test account with enough UTXOs to exceed the limit.
4. Call `retrieve_btc_with_approval` — the minter accepts the burn and schedules a reimbursement via `TooManyInputs` cancellation.
5. Patch the minter's `reimburse_withdrawals` to panic after the `mint_ckbtc` await but before `ScopeGuard::into_inner` (simulating a production panic).
6. Trigger the periodic task.
7. Observe: `retrieve_btc_status_v2(burn_index)` returns `Unknown`; `pending_withdrawal_reimbursements` is empty; `reimbursed_withdrawals[burn_index] == Err(Quarantined)`.
8. Confirm no endpoint exists to recover the entry — the user's ckBTC is permanently lost.

This is directly reproducible as a deterministic integration test using PocketIC with an injected panic, matching the existing test pattern in `rs/bitcoin/ckbtc/minter/src/updates/tests.rs`.
