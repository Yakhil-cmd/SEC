Audit Report

## Title
Single Byzantine Node Forces O(n·pairing) `combine_valid_shares` Fallback on Every VetKD Request — (`rs/crypto/src/vetkd/mod.rs`, `rs/crypto/internal/crypto_lib/bls12_381/vetkd/src/lib.rs`)

## Summary
`verify_encrypted_key_share_internal` admits shares to the validated pool after only an Ed25519 signature check, without verifying the BLS12-381 pairing equations of the share's curve points. A single Byzantine subnet node can submit a share with valid-but-unrelated curve points that passes pool admission, corrupts the `combine_all` Lagrange interpolation, and forces every VetKD combiner to fall back to `combine_valid_shares`, which performs up to O(n) individual pairing checks instead of O(1). The system remains correct but VetKD throughput is degraded by up to ~34× per request for the lifetime of the Byzantine node's committee membership.

## Finding Description
**Pool admission gate (`verify_encrypted_key_share_internal`, `rs/crypto/src/vetkd/mod.rs` L272–301):** Only `BasicSigVerifierInternal::verify_basic_sig` is called — an Ed25519 check over the raw bytes of `encrypted_key_share`. There is no call to `EncryptedKeyShare::is_valid`. The pairing relationship among `c1 ∈ G1`, `c2 ∈ G2`, `c3 ∈ G1` is never verified at admission time.

**Deserialization (`EncryptedKeyShare::deserialize`, `lib.rs` L446–464):** Only checks that each component is a valid curve point on BLS12-381. Three random valid-but-unrelated points pass deserialization and pool admission.

**`combine_all` failure (`lib.rs` L252–265):** Performs Lagrange interpolation over all admitted shares (cheap), then calls `is_valid` once on the combined result. One bad share corrupts the interpolated `(c1, c2, c3)` triple, causing `is_valid` to return `false` and `combine_all` to return `EncryptedKeyCombinationError::InvalidShares`.

**Unconditional fallback (`mod.rs` L388–430):** The `InvalidShares` arm in `combine_encrypted_key_shares_internal` unconditionally invokes `EncryptedKey::combine_valid_shares`, fetching individual node public keys from the store and constructing the full per-node map.

**O(n) pairing cost (`lib.rs` L286–298, L157–187):** `combine_valid_shares` iterates all `n` shares, calling `node_eks.is_valid(...)` on each, which internally calls `check_validity` with two `Gt::multipairing` operations per share. For a 34-node subnet with threshold 23, this is up to 34 `is_valid` calls (stopping after 23 valid ones) versus the single `is_valid` call in the `combine_all` fast path — a ~23–34× increase in pairing work per request.

## Impact Explanation
Every `vetkd_derive_key` request processed by an honest combiner on the affected subnet incurs O(n·pairing\_cost) cryptographic work instead of O(1·pairing\_cost). For a 34-node subnet this is a ~34× slowdown in the dominant cryptographic operation per request. Under concurrent VetKD load this degrades VetKD throughput for the entire subnet for as long as the Byzantine node remains in the committee. This constitutes a **subnet availability impact for VetKD operations not based on raw volumetric DDoS**, matching the **High ($2,000–$10,000)** impact class.

## Likelihood Explanation
The attack requires exactly one Byzantine node holding a valid subnet committee membership — well below the BFT fault threshold. The node needs only to: generate three random valid BLS12-381 curve points, concatenate them into a 192-byte blob, sign with its Ed25519 node key, and broadcast the share. No special capability beyond subnet membership is required. The attack is persistent and affects every concurrent VetKD request as long as the node remains in the committee. The Byzantine node's share passes all existing admission checks (`verify_encrypted_key_share` returns `Ok`).

## Recommendation
Add a cryptographic validity check inside `verify_encrypted_key_share_internal` (`rs/crypto/src/vetkd/mod.rs`). After the Ed25519 check passes, deserialize the share and call `EncryptedKeyShare::is_valid` with the signer's individual node public key (derivable from the NI-DKG transcript data already present in `lockable_threshold_sig_data_store`). Shares that fail this pairing check should be rejected at pool admission with a `VetKdKeyShareVerificationError`, preventing them from ever reaching `combine_encrypted_key_shares_internal`. This makes the fast `combine_all` path the normal case even in the presence of Byzantine nodes.

## Proof of Concept
```
1. In a 13-node test committee (threshold = 9):
   - Generate 12 honest shares via EncryptedKeyShare::create
   - Generate 1 Byzantine share: pick three random valid BLS12-381 points
     (G1, G2, G1), concatenate (192 bytes), sign with the Byzantine node's
     Ed25519 key
   - Call verify_encrypted_key_share for the Byzantine share → returns Ok(())
     (Ed25519 check passes, no pairing check performed)
   - Call combine_encrypted_key_shares with all 13 shares

2. Observe:
   - combine_all returns InvalidShares (combined result fails is_valid)
   - combine_valid_shares is invoked, performing up to 13 is_valid calls
   - Each is_valid call executes 2 Gt::multipairing operations

3. Benchmark comparison using the existing bench harness at
   rs/crypto/internal/crypto_lib/bls12_381/vetkd/benches/vetkd.rs:
   - Baseline (0 bad shares): combine_all path, 1 pairing check total
   - Attack  (1 bad share):   combine_valid_shares path, 13 pairing checks
   - Assert measured latency ratio ≈ 13× (scales with n, not threshold)
```

    c2: &G2Affine,

    msg: &G1Affine,
) -> bool {
    let neg_g2_g = G2Prepared::neg_generator();
    let c2_prepared = G2Prepared::from(c2);

    // check e(c1,g2) == e(g1, c2)
    let c1_c2 = Gt::multipairing(&[(c1, neg_g2_g), (G1Affine::generator(), &c2_prepared)]);

        (tpk.point(), &c2_prepared),
        (msg, &verification_key_prepared),
    ]);

    if !c3_c2_msg.is_identity() {
        return false;
    }
