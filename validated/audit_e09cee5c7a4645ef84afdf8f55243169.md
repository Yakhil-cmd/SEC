### Title
Fee-on-Transfer NEP-141 Token Mints Unbacked ERC-20 Tokens on Aurora — (`engine/src/engine.rs`)

### Summary

Aurora Engine's `ft_on_transfer` callback unconditionally trusts the `amount` field supplied by the calling NEP-141 token contract and mints an equal quantity of ERC-20 tokens (or credits ETH) inside the EVM. If the NEP-141 token charges a transfer fee, Aurora's actual on-chain balance of that token is less than `amount`, yet the full `amount` is minted. This creates unbacked ERC-20 tokens and makes the bridge insolvent for that token.

### Finding Description

When a NEP-141 token is bridged into Aurora, the standard NEAR `ft_transfer_call` flow causes the NEP-141 contract to invoke `ft_on_transfer` on Aurora. Aurora's handler reads the caller-supplied `amount` field and immediately mints that many ERC-20 tokens (or credits that much ETH) with no balance-before/after check.

The `args.amount` value originates entirely from the NEP-141 contract's callback invocation. Aurora never queries its own NEP-141 balance to confirm the actual credit.

**ERC-20 minting path — `receive_erc20_tokens` in `engine/src/engine.rs`:** [2](#0-1)

### Impact Explanation

A fee-on-transfer NEP-141 token charges a fee when tokens are transferred. If a user calls `ft_transfer_call(receiver_id=aurora, amount=X)` on such a token:

1. The NEP-141 contract credits Aurora's balance with `X − fee` tokens.
2. The NEP-141 contract calls `ft_on_transfer(amount=X)` on Aurora.
3. Aurora mints `X` ERC-20 tokens for the user.

Aurora now holds `X − fee` NEP-141 tokens but has issued `X` ERC-20 tokens. Repeating this inflates the ERC-20 supply beyond the NEP-141 backing. When users later burn ERC-20 tokens to exit via `withdrawToNear`, the last users to exit will find Aurora's NEP-141 balance exhausted — **permanent insolvency / fund theft for those users**.

**Impact: Critical — insolvency / permanent freezing of funds for late-exiting users.**

### Likelihood Explanation

Any unprivileged user can trigger this by calling `ft_transfer_call` on any fee-on-transfer NEP-141 token that has been registered with Aurora (via `deploy_erc20_token`). No admin access or special privilege is required. The attacker can repeat the operation to widen the insolvency gap. The likelihood is **medium** — it requires a fee-on-transfer NEP-141 token to be bridged, which is an uncommon but valid token design permitted by the NEP-141 standard.

### Recommendation

Before minting, Aurora should verify the actual balance change on the NEP-141 token. Because NEAR cross-contract calls are asynchronous, this requires restructuring the deposit flow:

1. Before accepting the transfer, record Aurora's current NEP-141 balance via a view call.
2. In a callback after `ft_on_transfer`, compare the new balance to the recorded value.
3. Mint only the **actual difference** (new balance − old balance), not `args.amount`.

Alternatively, document and enforce that only non-fee-on-transfer NEP-141 tokens may be registered via `deploy_erc20_token`, and add a runtime guard that rejects `ft_on_transfer` calls from tokens not on an explicit allowlist.

### Proof of Concept

1. Deploy a fee-on-transfer NEP-141 token `fee_token.near` that deducts 10% on every transfer.
2. Call `deploy_erc20_token` on Aurora to register `fee_token.near` → `EvmErc20` at address `0xABC`.
3. Call `ft_transfer_call(receiver_id="aurora", amount=1000, msg=<evm_address>)` on `fee_token.near`.
   - `fee_token.near` credits Aurora with 900 tokens (10% fee deducted).
   - `fee_token.near` calls `ft_on_transfer(sender_id=attacker, amount=1000, msg=<evm_address>)` on Aurora.
4. Aurora executes `receive_erc20_tokens` → `setup_receive_erc20_tokens_input(recipient, 1000)` → `EvmErc20.mint(recipient, 1000)`.
5. The attacker now holds 1000 ERC-20 tokens backed by only 900 NEP-141 tokens.
6. Repeat until Aurora's NEP-141 balance is exhausted relative to outstanding ERC-20 supply.
7. Honest users who bridged legitimately and try to exit last receive fewer NEP-141 tokens than they deposited, or none at all. [6](#0-5) [7](#0-6)

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
