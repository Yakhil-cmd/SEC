### Title
Inconsistent `MAX_BLOBS_PER_BLOCK` Constant Between ZK STF and Ethereum Paths Causes Blob Transaction Rejection - (`zk_ee/src/system/constants.rs` vs `basic_bootloader/src/bootloader/block_flow/ethereum/mod.rs`)

---

### Summary

`MAX_BLOBS_PER_BLOCK` is defined as **6** in `zk_ee/src/system/constants.rs` but as **9** in `basic_bootloader/src/bootloader/block_flow/ethereum/mod.rs` and `basic_bootloader/src/bootloader/block_flow/ethereum/block_header.rs`. The ZK STF transaction validation path and ZK block metadata both consume the `zk_ee` constant (6), while the Ethereum-native path uses the local constant (9). This is a direct pre-defined limit mismatch: the Ethereum Pectra fork (EIP-7691) raised the maximum blobs per block from 6 to 9, and the Ethereum path was updated, but the `zk_ee` constant was not.

---

### Finding Description

Three separate definitions of `MAX_BLOBS_PER_BLOCK` exist in the codebase:

**`zk_ee/src/system/constants.rs` line 31 — value: 6** [1](#0-0) 

**`basic_bootloader/src/bootloader/block_flow/ethereum/mod.rs` line 9 — value: 9** [2](#0-1) 

**`basic_bootloader/src/bootloader/block_flow/ethereum/block_header.rs` line 30 — value: 9** [3](#0-2) 

The ZK STF metadata (`BlockMetadataFromOracle`) imports from `zk_ee::system::constants::*` and therefore returns `max_blobs() = 6` and `blobs_gas_limit() = 6 * GAS_PER_BLOB`: [4](#0-3) [5](#0-4) 

The ZK transaction validation path calls `parse_blobs_list::<MAX_BLOBS_PER_BLOCK>` using the `zk_ee` constant (6): [6](#0-5) 

The ZK STF `TxLevelMetadata` blob array is also sized by the `zk_ee` constant (6): [7](#0-6) 

In contrast, the Ethereum path's `HeaderAndHistory` returns `max_blobs() = 9` and `blobs_gas_limit() = 9 * GAS_PER_BLOB`: [8](#0-7) 

The Ethereum transaction validation path calls `parse_blobs_list::<MAX_BLOBS_PER_BLOCK>` using the local constant (9): [9](#0-8) 

The `EthereumBlockMetadata` type is parameterized with the Ethereum-path `MAX_BLOBS_PER_BLOCK` (9) via `metadata_op.rs`: [10](#0-9) 

The block-level blob gas limit check in the ZK post-tx loop uses `system.get_blob_gas_limit()`, which resolves to `max_blobs() * GAS_PER_BLOB`. For the ZK STF path this is `6 * GAS_PER_BLOB`, while for the Ethereum path it is `9 * GAS_PER_BLOB`: [11](#0-10) 

---

### Impact Explanation

Any ZKsync-native (ZK STF) transaction carrying 7, 8, or 9 blobs will be rejected at the `parse_blobs_list::<6>` call with `BlobListTooLong`, even though the Ethereum Pectra specification and the Ethereum execution path both permit up to 9 blobs per block. Additionally, the ZK STF block-level blob gas limit is `6 * GAS_PER_BLOB` instead of `9 * GAS_PER_BLOB`, so a block whose total blob gas usage falls between those two values would be accepted by the Ethereum path but rejected by the ZK STF path. This constitutes a forward/proving divergence: a sequencer running the Ethereum path could produce a valid block that the ZK STF prover path would then reject as exceeding the blob gas limit, rendering the block unprovable and stalling the chain.

---

### Likelihood Explanation

The Ethereum Pectra fork (EIP-7691) is already live on mainnet. Any user submitting a ZKsync-native EIP-4844 transaction with more than 6 blobs triggers this path unconditionally. No special privileges are required; a standard unprivileged transaction sender is sufficient.

---

### Recommendation

Update `zk_ee/src/system/constants.rs` to match the Pectra value:

```rust
// was: pub const MAX_BLOBS_PER_BLOCK: usize = 6;
pub const MAX_BLOBS_PER_BLOCK: usize = 9;
```

Alternatively, remove the duplicate definitions in `basic_bootloader/src/bootloader/block_flow/ethereum/mod.rs` and `block_header.rs` and have all paths import from the single authoritative `zk_ee::system::constants` source, ensuring a single point of truth for this limit.

---

### Proof of Concept

1. Construct a ZKsync-native (ZK STF) EIP-4844 transaction carrying 7 blob versioned hashes.
2. Submit it to the ZKsync OS sequencer running the ZK STF path.
3. The call reaches `parse_blobs_list::<MAX_BLOBS_PER_BLOCK>(blobs_list)` in `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` line 411, where `MAX_BLOBS_PER_BLOCK = 6` (from `zk_ee::system::constants`).
4. Inside `parse_blobs_list`, the check `blobs_list.count > MAX_BLOBS_IN_TX` (i.e., `7 > 6`) is true, returning `Err(TxError::Validation(InvalidTransaction::BlobListTooLong))`.
5. The same transaction submitted through the Ethereum-native path would pass, since that path uses `MAX_BLOBS_PER_BLOCK = 9`. [12](#0-11)
