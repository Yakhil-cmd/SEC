Audit Report

## Title
SNS Swap Finalization Permanently Bricked via Pre-existing Neuron ID Collision in Memo Range - (`rs/sns/swap/src/types.rs`, `rs/sns/swap/src/swap.rs`)

## Summary
The SNS Swap canister maps `ClaimedSwapNeuronStatus::AlreadyExists` to `ClaimedStatus::Invalid`, a terminal non-retryable state. Any swap participant who pre-creates an SNS neuron using memo `NEURON_BASKET_MEMO_RANGE_START` (1,000,000) before the swap commits will cause `claim_swap_neurons` to return `AlreadyExists` for their basket entry, which permanently halts every subsequent `finalize` call. SNS Governance is never set to normal mode, dapp controllers are never transferred, and SNS tokens transferred during `sweep_sns` are permanently locked in the governance canister's subaccount.

## Finding Description
**Root cause:** `rs/sns/swap/src/types.rs` L1032 maps `ClaimedSwapNeuronStatus::AlreadyExists` to `ClaimedStatus::Invalid`:
```rust
ClaimedSwapNeuronStatus::AlreadyExists => ClaimedStatus::Invalid,
```
`ClaimedStatus::Invalid` is a terminal state: `to_neuron_recipe` in `rs/sns/swap/src/swap.rs` L3501–3511 returns `ConversionError::Invalid` for any recipe already marked `Invalid`, and the recipe is never retried. `set_claim_neuron_result` in `rs/sns/swap/src/types.rs` L921–928 calls `is_successful_sweep`, which returns `false` if `invalid > 0` (L977), setting a permanent error message and halting finalization.

**Exploit path:**
1. Attacker calls `refresh_buyer_tokens` with the minimum amount to participate in the swap.
2. Before the swap commits, attacker stakes tokens to SNS governance at subaccount `compute_neuron_staking_subaccount_bytes(attacker_principal, 1_000_000)` and calls `ManageNeuron::ClaimOrRefresh`. `new_neuron_id` in `rs/sns/governance/src/governance.rs` L836–848 only checks for duplicate IDs — it has no memo-range restriction for user-created neurons.
3. Swap reaches end time and commits.
4. `finalize` is called: `sweep_sns` succeeds (ICRC1 ledger is unaware of neurons), transferring SNS tokens to the attacker's neuron staking subaccount.
5. `claim_swap_neurons` is called. Governance finds the neuron already exists at the attacker's ID and returns `ClaimedSwapNeuronStatus::AlreadyExists` (governance.rs L4498–4504).
6. Swap maps this to `ClaimedStatus::Invalid`. `invalid > 0` → `set_claim_neuron_result` sets error message → finalization halts.
7. Every subsequent `finalize` call re-encounters the `Invalid` recipe and permanently fails.

**Why existing guards are insufficient:** The memo-range restriction in `rs/sns/init/src/distributions.rs` L219–228 applies only to developer neurons at SNS initialization, not to user-created neurons via `ManageNeuron::ClaimOrRefresh`. There is no runtime guard preventing a user from claiming a neuron with memo `1_000_000` in SNS Governance.

## Impact Explanation
This is a **High** severity finding matching: *"Significant SNS infrastructure security impact with concrete user or protocol harm."* A single unprivileged attacker permanently bricks the entire SNS swap finalization: SNS Governance is never set to normal mode (the SNS is permanently non-functional), dapp controllers are never transferred to the SNS, and SNS tokens transferred during `sweep_sns` are permanently locked in the governance canister's subaccount with no associated neuron and no recovery mechanism. All legitimate swap participants lose their SNS token allocations. Depending on swap size, token losses could reach Critical thresholds (>$1M).

## Likelihood Explanation
The attack requires only two unprivileged ingress calls available to any principal: `refresh_buyer_tokens` (with minimum participation) and `ManageNeuron::ClaimOrRefresh` with memo `1_000_000`. The constant `NEURON_BASKET_MEMO_RANGE_START = 1_000_000` is publicly documented. The attack can also occur accidentally if a legitimate user happens to create a neuron with this memo while also participating in a swap. No special privileges, social engineering, or external dependencies are required.

## Recommendation
1. **Primary fix (`rs/sns/swap/src/types.rs` L1032):** Map `ClaimedSwapNeuronStatus::AlreadyExists` to `ClaimedStatus::Success` instead of `ClaimedStatus::Invalid`. The neuron already exists at the correct ID, so the swap's goal is effectively achieved. The existing protobuf comment already states this intent: *"Future attempts to claim the same Neuron will result in `ClaimedSwapNeuronStatus::AlreadyExists`"* — implying prior success.
2. **Defense-in-depth (`rs/sns/governance/src/governance.rs`, `new_neuron_id`):** Reject memo values in `[NEURON_BASKET_MEMO_RANGE_START, SALE_NEURON_MEMO_RANGE_END]` for user-created neurons via `ManageNeuron::ClaimOrRefresh` when a swap is active, analogous to the existing developer neuron restriction.

## Proof of Concept
```
1. Deploy SNS with swap configured with basket_count >= 1.
2. Attacker calls refresh_buyer_tokens with minimum participation amount.
3. Attacker stakes tokens to SNS governance at:
       compute_neuron_staking_subaccount_bytes(attacker_principal, 1_000_000)
   and calls ManageNeuron::ClaimOrRefresh { memo: 1_000_000 }.
   → SNS Governance creates neuron with ID = hash(attacker_principal, 1_000_000).
4. Swap reaches end time and commits (Lifecycle::Committed).
5. Anyone calls finalize_swap.
6. sweep_sns succeeds — SNS tokens transferred to attacker's neuron staking subaccount.
7. claim_swap_neurons: Governance returns AlreadyExists for attacker's basket entry.
8. Swap maps AlreadyExists → ClaimedStatus::Invalid.
9. is_successful_sweep() returns false (invalid > 0).
10. set_claim_neuron_result sets error_message → finalize returns error.
11. SNS Governance never set to normal mode; dapp never transferred.
12. Every subsequent finalize call re-encounters Invalid recipe → permanent failure.
13. SNS tokens from step 6 are permanently locked (no neuron, no recovery path).

Verifiable via PocketIC integration test: assert that after step 3,
finalize_swap always returns an error and claim_neuron_result.invalid == 1.
```
