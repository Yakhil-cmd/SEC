Audit Report

## Title
Unbounded `environment_variables` Deserialization in `CanisterSettingsArgs` Enables CPU Exhaustion DoS - (File: rs/types/management_canister_types/src/lib.rs)

## Summary
`CanisterSettingsArgs.environment_variables` is typed as `Option<Vec<EnvironmentVariable>>` — a plain, unbounded `Vec` — while the analogous `controllers` and `allowed_viewers` fields use `BoundedVec` wrappers that enforce their limits at Candid deserialization time. An attacker can submit an `update_settings` or `create_canister` call containing far more than 32 environment variables; the replica will fully deserialize the payload and construct a `BTreeMap` over all entries in native Rust (outside Wasm instruction metering) before the count limit is checked and the call is rejected.

## Finding Description
`CanisterSettingsArgs` at line 2382 of `rs/types/management_canister_types/src/lib.rs` declares:

```rust
pub environment_variables: Option<Vec<EnvironmentVariable>>,
``` [1](#0-0) 

By contrast, `controllers` uses `BoundedControllers` (a `BoundedVec` with `MAX_ALLOWED_CONTROLLERS_COUNT = 10`) and `LogVisibilityV2::AllowedViewers` uses `BoundedAllowedViewers` (also max 10), both enforced at deserialization time: [2](#0-1) [3](#0-2) 

`BoundedVec`'s `Deserialize` implementation rejects oversized sequences immediately, element-by-element, before the full sequence is materialized: [4](#0-3) 

For `environment_variables`, no such guard exists. In `TryFrom<CanisterSettingsArgs> for CanisterSettings`, the full `Vec` is iterated and every `(name, value)` pair is inserted into a `BTreeMap<String, String>` — an O(n log n) native Rust operation — before any count check occurs: [5](#0-4) 

The count limit (`MAX_ENVIRONMENT_VARIABLES = 32`) is only enforced later, inside `validate_environment_variables()` in the canister manager: [6](#0-5) [7](#0-6) 

The attack sequence is: (1) Candid deserializes the full `Vec<EnvironmentVariable>` into heap memory; (2) `TryFrom` iterates all entries and builds a `BTreeMap`; (3) only then is the count checked and the call rejected. Steps 1 and 2 are entirely unmetered native Rust execution.

## Impact Explanation
This matches the **High** severity impact: *Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS.* An attacker controlling a single canister can repeatedly call `update_settings` with payloads containing thousands of environment variables (well within the 2 MB ingress limit). Each call forces the replica to perform unbounded heap allocation and O(n log n) `BTreeMap` construction in native code before rejection. Sustained at the ingress rate, this can saturate replica CPU across all nodes of a subnet, degrading or halting execution of legitimate canister calls. The attack exploits a specific unmetered code path, not raw packet volume.

## Likelihood Explanation
`update_settings` requires the caller to be a controller of a canister, so the attacker must have deployed at least one canister (a one-time cost). Once that precondition is met, the attack is trivially repeatable: any Candid encoder can construct the malicious payload, no special knowledge is required, and the per-message cost is bounded only by ingress fees. `create_canister` is also affected but requires cycles to be attached. The constraint of needing one canister is minimal and does not meaningfully limit a motivated attacker.

## Recommendation
Apply the same `BoundedVec` pattern already used for `controllers` and `allowed_viewers`. Define a bounded type in `rs/types/management_canister_types/src/lib.rs`:

```rust
const MAX_ALLOWED_ENV_VARS_COUNT: usize = 32;
pub type BoundedEnvironmentVariables =
    BoundedVec<MAX_ALLOWED_ENV_VARS_COUNT, UNBOUNDED, UNBOUNDED, EnvironmentVariable>;
```

Replace the `environment_variables` field type in `CanisterSettingsArgs` from `Option<Vec<EnvironmentVariable>>` to `Option<BoundedEnvironmentVariables>`. This enforces the limit at Candid deserialization time, before any heap allocation or `BTreeMap` construction, consistent with how `controllers` and `allowed_viewers` are already protected.

## Proof of Concept
Using a local replica or PocketIC:

1. Deploy a canister and record its ID and the controller identity.
2. Construct a `update_settings` Candid payload with `environment_variables` set to a `vec` of 10,000 `{name=""; value=""}` records (well under 2 MB).
3. Submit the call as the controller.
4. Observe via replica metrics or profiling that native CPU time is consumed deserializing and building the `BTreeMap` before the call is rejected with a count-exceeded error.
5. Repeat at high frequency and observe degradation of other canister execution on the same subnet.

A deterministic integration test can assert that the call is rejected *at deserialization time* (before `TryFrom` is invoked) once the fix is applied, mirroring the existing tests for `BoundedControllers`.
