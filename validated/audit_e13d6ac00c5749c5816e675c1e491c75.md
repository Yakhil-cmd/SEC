### Title
Unchecked Precompile Return Value After `_burn` in `withdrawToNear`/`withdrawToEthereum` Causes Permanent ERC-20 Supply Discrepancy and NEP-141 Fund Freeze — (`File: etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

In `EvmErc20.sol` (and `EvmErc20V2.sol`), both `withdrawToNear` and `withdrawToEthereum` call `_burn` on the caller's ERC-20 balance **before** invoking the exit precompile via inline assembly, and **never check the return value** of that precompile call. If the precompile call fails (returns 0), the ERC-20 tokens are permanently burned while no NEAR promise is scheduled, leaving the corresponding NEP-141 tokens permanently locked in Aurora's account. This creates a permanent discrepancy between the ERC-20 `totalSupply` and the NEP-141 tokens held by Aurora — the direct analog of the reported `_totalSupply` accounting bug.

---

### Finding Description

In `EvmErc20.sol`, both exit functions follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ERC-20 totalSupply decreases

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // `res` is NEVER CHECKED
    }
}
``` [1](#0-0) 

The same pattern exists in `withdrawToEthereum`: [2](#0-1) 

And identically in `EvmErc20V2.sol`: [3](#0-2) 

The precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` is `ExitToNear`, implemented in `engine-precompiles/src/native.rs`. It can return `Err(ExitError)` in several reachable cases:

- `parse_recipient` fails if `recipient` bytes are not valid UTF-8 or not a valid NEAR account ID (e.g., a user passes raw Ethereum address bytes).
- `get_nep141_from_erc20` fails if the ERC-20 is not registered in the NEP-141 map.
- `get_eth_connector_contract_account` fails if the connector account key is absent. [4](#0-3) 

When the precompile returns `Err(ExitError)`, the EVM `call` opcode returns `0` to the caller. Because `res` is never checked, `withdrawToNear` returns normally. The `_burn` state change (which happened in the **caller's** EVM context, before the `call`) is **not reverted**. No NEAR promise is scheduled. The NEP-141 tokens remain permanently locked in Aurora's account.

The `error_refund` feature only handles the case where the NEAR promise fails **after** the EVM transaction succeeds. It does not protect against the precompile itself failing: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the precompile call fails silently:

1. The ERC-20 `totalSupply` is permanently reduced (tokens burned from the user's balance).
2. No NEP-141 `ft_transfer` promise is ever scheduled.
3. The NEP-141 tokens remain locked in Aurora's account with no mechanism to recover them — the ERC-20 tokens that represent them have been destroyed.
4. The bridge accounting is permanently broken: ERC-20 `totalSupply` < NEP-141 tokens held by Aurora.

This is the direct analog of the reported bug: a permanent discrepancy between the ERC-20 supply accounting and the actual underlying token supply, leading to irrecoverable locked funds.

---

### Likelihood Explanation

**Medium.**

The most realistic trigger is a user passing a `recipient` value that is not a valid NEAR account ID. The function signature accepts `bytes memory recipient`, which is permissive. A user could accidentally pass:
- Raw Ethereum address bytes (20 bytes, not valid UTF-8 as a NEAR account ID).
- Any byte sequence that fails `parse_recipient`'s UTF-8 or account ID validation. [6](#0-5) 

The function returns successfully in all cases (no revert, no event indicating failure), so the user receives no on-chain signal that their tokens are permanently lost. This makes accidental triggering realistic.

---

### Recommendation

Check the return value of the precompile `call` in both `withdrawToNear` and `withdrawToEthereum`, and revert if it returns 0:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the precompile fails, the entire transaction (including `_burn`) is reverted, preserving the invariant that ERC-20 `totalSupply` always equals the NEP-141 tokens held by Aurora.

Apply the same fix to both `EvmErc20.sol` and `EvmErc20V2.sol`, and to both `withdrawToNear` and `withdrawToEthereum`.

---

### Proof of Concept

1. Deploy `EvmErc20` for a NEP-141 token. Bridge `1000` NEP-141 tokens to Aurora; user receives `1000` ERC-20 tokens. ERC-20 `totalSupply` = 1000, Aurora holds 1000 NEP-141 tokens.
2. User calls `withdrawToNear(invalidRecipientBytes, 500)` where `invalidRecipientBytes` is 20 raw bytes (e.g., an Ethereum address) that fail NEAR account ID validation.
3. `_burn(user, 500)` executes: ERC-20 `totalSupply` = 500, user's ERC-20 balance = 500.
4. `call(gas(), exitToNear, ...)` executes. Inside `ExitToNear::run`, `parse_recipient` fails on the invalid bytes, returning `Err(ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID"))`. The `call` opcode returns `0`.
5. `res` is never checked. `withdrawToNear` returns normally. Transaction succeeds.
6. **Result**: ERC-20 `totalSupply` = 500, but Aurora still holds 1000 NEP-141 tokens. The 500 NEP-141 tokens corresponding to the burned ERC-20 are permanently locked. No recovery path exists. [1](#0-0) [7](#0-6)

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
