Audit Report

## Title
Wrong Field in `From<ValidatedNeuronsFundParticipationConstraints>` Serializes Minimum Threshold as Neurons' Fund Hard Cap — (File: rs/sns/swap/src/neurons_fund.rs)

## Summary
In `rs/sns/swap/src/neurons_fund.rs`, the `From<ValidatedNeuronsFundParticipationConstraints<F>> for NeuronsFundParticipationConstraintsPb` implementation assigns `value.min_direct_participation_threshold_icp_e8s` to the `max_neurons_fund_participation_icp_e8s` protobuf field instead of `value.max_neurons_fund_participation_icp_e8s`. The SNS swap canister deserializes this field and uses it as the hard cap on Neurons' Fund participation, causing every SNS swap with Neurons' Fund matched funding to enforce a cap orders of magnitude lower than governance intended.

## Finding Description
At `rs/sns/swap/src/neurons_fund.rs` lines 512–514, the `From` conversion contains a copy-paste error:

```rust
max_neurons_fund_participation_icp_e8s: Some(
    value.min_direct_participation_threshold_icp_e8s,  // wrong field
),
``` [1](#0-0) 

The correct source field is `value.max_neurons_fund_participation_icp_e8s`. The serialized protobuf is forwarded from NNS governance to the SNS swap canister. The swap canister's `MatchedParticipationFunction::apply` reads `self.max_neurons_fund_participation_icp_e8s` in two places:

1. As the direct return value when direct participation exceeds the last interval's upper bound (`rs/nervous_system/neurons_fund/src/lib.rs` line 1005). [2](#0-1) 

2. As `hard_cap_icp` that clamps the computed effective participation (`rs/nervous_system/neurons_fund/src/lib.rs` line 1025). [3](#0-2) 

Because the serialized value is `min_direct_participation_threshold_icp_e8s` (e.g., 50,000 ICP) rather than `max_neurons_fund_participation_icp_e8s` (e.g., 1,000,000 ICP), the hard cap enforced by the swap canister is far below what governance approved. No existing validation rejects this because the minimum threshold is itself a valid `u64`; the substitution is semantically wrong but structurally well-formed.

## Impact Explanation
Every SNS swap that opts into Neurons' Fund matched funding will have its Neurons' Fund participation silently capped at `min_direct_participation_threshold_icp_e8s` instead of `max_neurons_fund_participation_icp_e8s`. These values differ by orders of magnitude in practice. The result is that governance-approved ICP participation is suppressed without any error, SNS swaps may fail to reach their minimum direct participation targets due to suppressed matched funding, and SNS projects may fail to decentralize. This constitutes significant SNS framework financial harm with concrete user and protocol impact, matching the **High ($2,000–$10,000)** bounty tier: "Significant … NNS, SNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
No attacker action is required. The bug is triggered unconditionally by any SNS swap proposal that includes Neurons' Fund participation — a standard, permissionless action available to any SNS project. The `From` conversion is called during normal swap initialization. The bug is deterministic and affects 100% of qualifying swaps.

## Recommendation
Fix line 513 in `rs/sns/swap/src/neurons_fund.rs`:

```rust
max_neurons_fund_participation_icp_e8s: Some(
    value.max_neurons_fund_participation_icp_e8s, // corrected
),
```

Add a round-trip unit test asserting that after the `From` conversion, the output's `max_neurons_fund_participation_icp_e8s` equals the input's `max_neurons_fund_participation_icp_e8s` and is strictly greater than `min_direct_participation_threshold_icp_e8s`.

## Proof of Concept
Construct a `ValidatedNeuronsFundParticipationConstraints` with `min_direct_participation_threshold_icp_e8s = 50_000 * E8` and `max_neurons_fund_participation_icp_e8s = 1_000_000 * E8`. Call `NeuronsFundParticipationConstraintsPb::from(value)`. Assert that the resulting `max_neurons_fund_participation_icp_e8s` field equals `1_000_000 * E8`. Under the current code it will equal `50_000 * E8`, confirming the bug. A subsequent call to `MatchedParticipationFunction::apply` with any direct participation above the last interval's upper bound will return `50_000 * E8` instead of `1_000_000 * E8`, and the `hard_cap_icp` path will clamp all intermediate results to `50_000 * E8`.
