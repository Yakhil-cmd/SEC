### Title
Stale ICP/XDR Exchange Rate Used for Cycles Minting Without Freshness Check — (`File: rs/nns/cmc/src/main.rs`)

---

### Summary

The Cycles Minting Canister (CMC) converts ICP to cycles using a cached `icp_xdr_conversion_rate` that is updated asynchronously via heartbeat. When the Exchange Rate Canister (XRC) fails or returns an invalid rate, the CMC silently continues using an arbitrarily stale rate for all cycle-minting operations (`notify_top_up`, `notify_create_canister`, `notify_mint_cycles`) with no staleness check and no user-visible warning. This is the direct IC analog of the Chainlink oracle going offline while the system continues operating on a frozen price.

---

### Finding Description

The CMC's `tokens_to_cycles` function reads `state.icp_xdr_conversion_rate` and uses whatever value is cached, regardless of how old it is:

```rust
// rs/nns/cmc/src/main.rs:1900-1922
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles { ... }.to_cycles(amount)),
            None => Err(NotifyError::Other { ... }),
        }
    })
}
```

The only check is whether the rate is `None` (never set). There is no check on `timestamp_seconds` to determine whether the rate is fresh.

The rate is updated via `canister_heartbeat` → `update_exchange_rate` → XRC call. When XRC fails, the CMC logs the error and schedules a retry one minute later, but the stale cached rate remains in use:

```rust
// rs/nns/cmc/src/main.rs:2397-2428
#[heartbeat]
async fn canister_heartbeat() {
    if with_state(|state| state.exchange_rate_canister_id.is_some()) {
        update_exchange_rate().await
    }
}
```

On XRC failure, `UpdateExchangeRateError::FailedToRetrieveRate` or `InvalidRate` is logged but the cached `icp_xdr_conversion_rate` is left unchanged:

```rust
// rs/nns/cmc/src/exchange_rate_canister.rs:149-161
Err(error) => match error {
    UpdateExchangeRateError::FailedToRetrieveRate(_)
    | UpdateExchangeRateError::FailedToSetRate(_)
    | UpdateExchangeRateError::InvalidRate(_) => {
        state.update_exchange_rate_canister_state.replace(
            UpdateExchangeRateState::get_rate_at_next_minute(...),
        );
    }
},
```

Additionally, when a `DivergedRate` governance proposal is submitted, the automatic XRC update is permanently `Disabled` until an `EnableAutomaticExchangeRateUpdates` proposal re-enables it. During this entire window, the frozen rate is used for all minting:

```rust
// rs/nns/cmc/src/exchange_rate_canister.rs:311-314
UpdateIcpXdrConversionRatePayloadReason::DivergedRate => {
    state.update_exchange_rate_canister_state
        .replace(UpdateExchangeRateState::Disabled);
}
```

The `icp_xdr_conversion_rate` field carries a `timestamp_seconds` that is never compared against the current time before minting.

---

### Impact Explanation

**Impact: Medium-High**

The ICP/XDR rate directly controls how many cycles are minted per ICP. If the XRC goes offline or returns invalid data for an extended period (hours to days), the CMC continues minting cycles at the last known rate. If the real ICP price has dropped significantly since the cached rate was set, users receive more cycles per ICP than they should — effectively minting cycles at a subsidized rate at the expense of the network's economic model. Conversely, if the price has risen sharply, users receive fewer cycles than the market rate warrants, causing economic harm to users. In the `Disabled` state (triggered by a `DivergedRate` proposal), the rate can be frozen indefinitely until a new governance proposal passes, which takes days on the NNS.

The three affected public entry points are:
- `notify_top_up` — tops up a canister with cycles
- `notify_create_canister` — creates a canister using ICP
- `notify_mint_cycles` — mints cycles to the cycles ledger

All three call `tokens_to_cycles` without any rate-freshness guard.

---

### Likelihood Explanation

**Likelihood: Low-Medium**

The XRC itself is a decentralized canister that aggregates rates from multiple HTTP outcall sources. Transient failures are expected and handled by retry logic. However:
- The `Disabled` state can persist for days (NNS governance cycle).
- The XRC can return `InconsistentRatesReceived` or `StablecoinRateTooFewRates` during market stress, causing repeated failures and an increasingly stale rate.
- The CMC's integration test explicitly demonstrates this scenario: when XRC returns `StablecoinRateTooFewRates`, the old rate is preserved and used.

The scenario is not theoretical — the test at `rs/nns/integration_tests/src/cycles_minting_canister_with_exchange_rate_canister.rs:162-185` confirms the stale-rate behavior is the intended design, but no staleness bound is enforced.

---

### Recommendation

1. Add a staleness check in `tokens_to_cycles`: compare `icp_xdr_conversion_rate.timestamp_seconds` against the current canister time. If the rate is older than a configurable threshold (e.g., 24 hours), return a retriable `NotifyError` instead of minting at a stale rate.
2. Expose the rate age in the `get_icp_xdr_conversion_rate` certified response so callers can independently verify freshness.
3. Consider a secondary fallback: if the XRC is unavailable for more than N minutes, fall back to the NNS governance-provided rate (which is updated via proposal) rather than the last XRC-polled value.

---

### Proof of Concept

**Entry path (unprivileged ingress sender):**

1. User sends ICP to the CMC subaccount for their canister (standard ledger transfer).
2. User calls `notify_top_up` with the block index.
3. CMC calls `tokens_to_cycles(amount)`.
4. `tokens_to_cycles` reads `state.icp_xdr_conversion_rate` — which may be hours or days old if XRC has been failing.
5. Cycles are minted at the stale rate with no error or warning to the user.

**Staleness scenario:**
- XRC returns `StablecoinRateTooFewRates` repeatedly for 6 hours.
- CMC retries every minute, fails each time, keeps the rate from 6 hours ago.
- ICP price has dropped 30% in those 6 hours.
- Every `notify_top_up` call during this window mints ~43% more cycles than the current market rate warrants (since cycles ∝ `xdr_permyriad_per_icp`).
