### Title
Permanent Loss of ERC-20 Tokens When `ExitToNear` NEAR Promise Fails Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`)

---

### Summary

When a user calls `withdrawToNear` on an Aurora ERC-20 contract, the ERC-20 tokens are **burned first** in the EVM, and then a NEAR `ft_transfer` promise is scheduled asynchronously. If that NEAR promise fails and the `error_refund` compile-time feature is **not** enabled in the production build, there is no refund path — the burned ERC-20 tokens are permanently lost. This is a direct structural analog to the reported vulnerability: a state-destroying action (burn) occurs before the dependent claim (NEAR-side token receipt) is confirmed, and the absence of a recovery path causes permanent fund loss.

---

### Finding Description

**Step 1 — ERC-20 burn before NEAR promise**

In `etc/eth-contracts/contracts/EvmErc20.sol`, `withdrawToNear` burns the caller's tokens unconditionally before invoking the `ExitToNear` precompile: [1](#0-0) 

The burn is irreversible at the EVM level. The precompile call that follows only *schedules* a NEAR promise — it does not execute it synchronously.

**Step 2 — Precompile schedules promise with conditional refund**

Inside `ExitToNear::run` in `engine-precompiles/src/native.rs`, the `callback_args.refund` field is populated **only** when the `error_refund` Cargo feature is compiled in: [2](#0-1) 

When `error_refund` is absent, `refund` is hardcoded to `None`. The promise is then emitted as a plain `PromiseArgs::Create` (no callback) if both `refund` and `transfer_near` are `None`: [3](#0-2) 

**Step 3 — Callback does nothing on failure without refund args**

In `engine/src/contract_methods/connector.rs`, `exit_to_near_precompile_callback` handles the NEAR promise result. When the promise fails and `args.refund` is `None`, the else-branch is taken and the function returns `Ok(None)` — no re-mint, no recovery: [4](#0-3) 

**Step 4 — Refund path exists but is feature-gated**

The `refund_on_error` function in `engine/src/engine.rs` *can* re-mint burned ERC-20 tokens, but it is only reachable when `args.refund` is `Some`, which requires `error_refund` to have been compiled in: [5](#0-4) 

**Step 5 — Test confirms permanent loss without the feature**

The integration test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` explicitly documents the two outcomes: [6](#0-5) 

When `error_refund` is absent, the ERC-20 balance after a failed exit is `FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT` — the burned amount is gone.

---

### Impact Explanation

**Impact: Critical — Permanent freezing of funds.**

A user who calls `withdrawToNear` (or `withdrawToEthereum` via the analogous `ExitToEthereum` path) has their ERC-20 tokens burned in the EVM. If the subsequent NEAR-side `ft_transfer` promise fails for any reason (recipient account not registered with the NEP-141 contract, NEP-141 contract paused, insufficient attached gas, etc.) and the `error_refund` feature is not compiled into the production binary, those tokens are permanently unrecoverable. There is no on-chain mechanism to re-mint them or retry the transfer. The user's funds are frozen at address(0) with no owner.

---

### Likelihood Explanation

**Likelihood: Medium.**

The failure condition is reachable by any unprivileged EVM user without admin involvement:

- Sending tokens to a NEAR account that has not called `storage_deposit` on the NEP-141 contract is a common user error and causes `ft_transfer` to fail.
- The `error_refund` feature is a compile-time opt-in, not a default. Any production deployment built without it (e.g., for binary size or compatibility reasons) is permanently exposed.
- `EvmErc20.sol` (the non-V2 contract) does not encode a refund address in its precompile input at all, meaning even if `error_refund` were enabled engine-side, the old ERC-20 contract format would not supply a valid refund target. [1](#0-0) [7](#0-6) 

---

### Recommendation

1. **Make `error_refund` a default feature** in `engine-precompiles/Cargo.toml` and `engine/Cargo.toml` so that the refund callback is always compiled into production builds.
2. **Update `EvmErc20.sol`** to encode the caller's address as the refund target (matching the `EvmErc20V2.sol` format) so that the engine-side refund path has a valid recipient even for legacy ERC-20 contracts.
3. **Add a guard** in `ExitToNear::run` that refuses to emit a promise without a callback when the `error_refund` feature is absent, making the unsafe configuration a compile-time error rather than a silent runtime risk.

---

### Proof of Concept

The existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` already demonstrates the vulnerability. Compile the engine **without** the `error_refund` feature and run:

```
cargo test test_exit_to_near_refund
```

The test asserts:
```rust
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
assert_eq!(erc20_balance(&erc20, ft_owner_address, &aurora).await, balance);
```

This confirms that `FT_EXIT_AMOUNT` tokens are burned from the ERC-20 balance and never returned, even though the NEAR-side `ft_transfer` to `"unregistered.near"` failed. The user has lost their tokens with no recovery path. [8](#0-7)
