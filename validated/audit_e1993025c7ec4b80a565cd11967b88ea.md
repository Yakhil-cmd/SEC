### Title
Unchecked Integer Arithmetic in SNS Governance Neuron Maturity Accumulation During Reward Distribution - (File: rs/sns/governance/src/governance.rs)

### Summary
The SNS governance `distribute_rewards` function uses plain Rust `+` and `+=` operators when accumulating voting rewards into neuron maturity fields. Unlike the analogous NNS governance code which correctly uses `saturating_add`, the SNS path has no overflow protection. Depending on compilation flags, this causes either a silent wrap-around (corrupting maturity) or a panic/trap (losing the entire reward distribution round).

### Finding Description
In `rs/sns/governance/src/governance.rs`, the `distribute_rewards` function accumulates per-neuron reward amounts using unchecked arithmetic: [1](#0-0) 

Both `neuron.staked_maturity_e8s_equivalent.unwrap_or(0) + neuron_reward_e8s` (line 5991) and `neuron.maturity_e8s_equivalent += neuron_reward_e8s` (line 5994) are plain `u64` additions with no overflow guard. The accumulator `distributed_e8s_equivalent += neuron_reward_e8s` at line 5996 is also unguarded.

The only overflow check present is a `debug_assert!` at lines 6004–6006, which is compiled out in production (release) builds: [2](#0-1) 

By direct contrast, the NNS governance reward distribution path — which performs the identical logical operation — explicitly uses `saturating_add` on both the `staked_maturity_e8s_equivalent` and `maturity_e8s_equivalent` fields: [3](#0-2) 

The SNS `add_maturity` test-only helper also uses `saturating_add`, confirming the intent: [4](#0-3) 

The `rewards_purse_e8s` is correctly guarded against `u64` overflow before the per-neuron loop (lines 5878–5890), but the per-neuron accumulation into existing neuron state is not guarded. [5](#0-4) 

### Impact Explanation
- **If overflow-checks are enabled** (common for IC canister release builds): the addition panics, trapping the canister mid-reward-distribution. The entire reward round is lost — no neuron receives maturity — and the governance canister's periodic task fails silently until the next round. This is a **governance conservation bug**: legitimately earned voting rewards are permanently destroyed.
- **If overflow-checks are disabled**: the maturity field wraps around to a small value, silently corrupting the neuron's accumulated maturity. A neuron that should have a large maturity balance instead shows near-zero, making it unable to spawn child neurons or disburse maturity. This is also a **governance conservation bug**.

In either case, the impact is loss or corruption of SNS neuron maturity — a financial asset representing earned voting rewards.

### Likelihood Explanation
An SNS can be deployed with a token supply up to `u64::MAX` e8s (~1.84 × 10¹⁹). A neuron holding a large fraction of voting power in a high-supply SNS accumulates maturity proportional to the supply each reward round. Over many rounds, `maturity_e8s_equivalent` can approach `u64::MAX`. No special attacker action is required — the overflow is triggered by the normal periodic reward distribution timer, reachable by any governance participant who votes on proposals. The condition is more likely in SNS deployments with large token supplies and long-running neurons with auto-stake-maturity disabled.

### Recommendation
Replace the plain `+` and `+=` operators in `distribute_rewards` with `saturating_add`, matching the NNS governance implementation:

```rust
// staked maturity path
neuron.staked_maturity_e8s_equivalent = Some(
    neuron.staked_maturity_e8s_equivalent.unwrap_or(0)
        .saturating_add(neuron_reward_e8s),
);

// unstaked maturity path
neuron.maturity_e8s_equivalent =
    neuron.maturity_e8s_equivalent.saturating_add(neuron_reward_e8s);

// accumulator
distributed_e8s_equivalent =
    distributed_e8s_equivalent.saturating_add(neuron_reward_e8s);
```

Additionally, promote the `debug_assert!` at line 6004 to a hard `assert!` or a logged error with graceful handling, so overflow conditions are observable in production.

### Proof of Concept
1. Deploy an SNS with a token supply of `u64::MAX / 2` e8s and a high initial reward rate.
2. Create a single neuron holding 100% of the voting power with `auto_stake_maturity = false`.
3. Vote on proposals across enough reward rounds so that the cumulative `neuron_reward_e8s` additions cause `maturity_e8s_equivalent` to exceed `u64::MAX`.
4. On the overflow round, `distribute_rewards` either traps (if overflow-checks are on) — causing the entire reward event to be silently skipped — or wraps the maturity to a near-zero value (if overflow-checks are off).
5. Observe that `latest_reward_event().distributed_e8s_equivalent` reports a non-zero amount distributed, but the neuron's `maturity_e8s_equivalent` is either unchanged (trap case) or corrupted (wrap case).
