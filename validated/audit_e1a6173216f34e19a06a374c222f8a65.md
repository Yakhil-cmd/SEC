Audit Report

## Title
Missing Staleness Check on ICP/XDR Conversion Rate Allows Excess Cycles Minting - (File: rs/nns/cmc/src/main.rs)

## Summary

`tokens_to_cycles` in `rs/nns/cmc/src/main.rs` reads only the numeric `xdr_permyriad_per_icp` field from the cached `IcpXdrConversionRate` and silently discards `timestamp_seconds`, performing no freshness check before minting cycles. When the Exchange Rate Canister becomes unavailable or the `Disabled` state is active, the frozen rate is accepted indefinitely, allowing any unprivileged caller of `notify_top_up`, `notify_mint_cycles`, or `notify_create_canister` to receive more cycles per ICP than the current market rate justifies.

## Finding Description

`tokens_to_cycles` (lines 1900ã1922 of `rs/nns/cmc/src/main.rs`) is the single conversion point for all three public minting entry-points:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);  // timestamp_seconds silently dropped
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles { ... }.to_cycles(amount)),
            None => Err(...),  // only None is rejected; a stale rate is accepted
        }
    })
}
``` [1](#0-0) 

The `IcpXdrConversionRate` struct carries a `timestamp_seconds` field (documented at line 217: "How many XDR 1 ICP is worth, along with a timestamp"), but it is never compared against the current time before the rate is used. [2](#0-1) 

Grep searches for `MAX_RATE_AGE`, `stale`, and `rate_age` across all CMC source files return zero matches, confirming no staleness guard exists anywhere in the codebase.

Two distinct staleness windows exist:

**Window 1 ã XRC failure retry loop:** When `get_icp_to_xdr_exchange_rate` fails, `schedule_next_attempt` reschedules a retry at the next minute but leaves the cached rate unchanged. The normal refresh interval is `REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS`. [3](#0-2) [4](#0-3) 

**Window 2 ã `Disabled` state:** When a governance proposal arrives with `DivergedRate` reason, `set_update_exchange_rate_state` sets `UpdateExchangeRateState::Disabled`. The guard in `UpdateExchangeRateGuard::new` returns `Err(UpdateExchangeRateError::Disabled)` immediately, halting all XRC polling until a subsequent governance proposal re-enables it. This window is unbounded. [5](#0-4) [6](#0-5) 

`notify_top_up` is an `#[update]` method with no caller authentication; it calls `process_top_up`, which calls `tokens_to_cycles` directly. [7](#0-6) [8](#0-7) 

## Impact Explanation

Cycles are backed by burned ICP. Issuing cycles at a stale (higher) rate when the ICP market price has fallen means the protocol burns less ICP than the cycles issued are worth, constituting illegal minting and a direct resource-accounting loss. The magnitude scales with the staleness duration and the size of the ICP price decline. During a `Disabled` window spanning hours or days combined with a significant ICP price drop, the aggregate excess cycles minted across all callers can represent substantial protocol-level economic harm. This matches the allowed High impact: "Significant XRC, NNS, or infrastructure security impact with concrete user or protocol harm," and approaches Critical ("illegal mintingãÎ involving exorbitant ICP/Cycles") under worst-case conditions.

## Likelihood Explanation

`notify_top_up` requires no special privileges ã any principal can call it after transferring ICP to the CMC subaccount. The `Disabled` state is a legitimate protocol state triggered by a governance proposal with `DivergedRate` reason; once set, it persists until another governance proposal re-enables polling, a window that can span hours. The XRC failure path (Window 1) requires only that the XRC system canister be temporarily unavailable, which can occur during upgrades or sustained errors. Neither precondition requires attacker control of any privileged key or node. The exploit is repeatable for the entire duration of the staleness window.

## Recommendation

In `tokens_to_cycles`, compare `rate.timestamp_seconds` against the current time and return a retriable error when the age exceeds a defined threshold (e.g., 30 minutes, chosen to be larger than the normal 5-minute refresh interval but small enough to bound worst-case price divergence):

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        match state.icp_xdr_conversion_rate.as_ref() {
            Some(rate) => {
                let age = now_seconds().saturating_sub(rate.timestamp_seconds);
                if age > MAX_RATE_AGE_SECONDS {
                    return Err(NotifyError::Other {
                        error_code: NotifyErrorCode::Internal as u64,
                        error_message: format!("ICP/XDR rate is stale ({age}s old)"),
                    });
                }
                Ok(TokensToCycles {
                    xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
                    cycles_per_xdr: state.cycles_per_xdr,
                }.to_cycles(amount))
            }
            None => Err(...),
        }
    })
}
```

Additionally, consider emitting a metric or log warning when the rate age exceeds the refresh interval, to surface staleness before it becomes exploitable.

## Proof of Concept

1. Submit a governance proposal with `DivergedRate` reason to set `UpdateExchangeRateState::Disabled` on the CMC. The cached rate freezes at its current value (e.g., `xdr_permyriad_per_icp = 100_000`, representing 10 XDR/ICP).
2. Allow the ICP market price to decline (e.g., to 8 XDR/ICP) while the `Disabled` state persists.
3. Transfer 1 ICP to the CMC subaccount for a target canister and call `notify_top_up` as an unprivileged principal.
4. Observe that `tokens_to_cycles` reads `xdr_permyriad_per_icp = 100_000` without checking `timestamp_seconds` and mints cycles as if ICP is worth 10 XDR ã approximately 25% more than the current 8 XDR market rate justifies.
5. Repeat until the governance re-enables XRC polling and a fresh rate is set.

A deterministic integration test can reproduce this by: (a) initializing the CMC state with a rate timestamped in the past, (b) setting `UpdateExchangeRateState::Disabled`, (c) calling `notify_top_up` via PocketIC, and (d) asserting that the returned `Cycles` value exceeds what the current market rate would produce.
