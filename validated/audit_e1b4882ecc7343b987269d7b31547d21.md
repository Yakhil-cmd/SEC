### Title
Unverified Oracle Advice in Modexp Delegation Allows Malicious Prover to Produce Arbitrary Modexp Results — (`basic_system/src/system_functions/modexp/delegation/bigint.rs`)

---

### Summary

In the proving execution path, the `OracleAdvisor` for modular exponentiation accepts quotient and remainder values from the oracle (prover-controlled non-deterministic input) and uses them directly in the `modpow` computation without verifying the fundamental arithmetic identity `quotient × modulus + remainder == dividend`. Only a loose upper-bound length check is performed. A malicious prover can supply arbitrary quotient/remainder values that pass the length check but produce a wrong modexp output, and the ZK proof will still be valid for that wrong output.

---

### Finding Description

The modexp precompile (EVM address `0x05`) uses a delegation pattern in proving mode: instead of computing big-integer division natively on RISC-V (which is expensive to prove), the RISC-V program asks the oracle for the quotient and remainder of each intermediate reduction step, then uses those values directly.

The oracle advice is consumed in `OracleAdvisor::get_reduction_op_advice` in `basic_system/src/system_functions/modexp/delegation/bigint.rs`:

```rust
// check that hint is "sane" in upper bound
assert!(q_len.next_multiple_of(8) / 8 <= max_quotient_digits);
assert!(r_len.next_multiple_of(8) / 8 <= max_remainder_digits);

write_bigint(&mut it, q_len, quotient_dst);
write_bigint(&mut it, r_len, remainder_dst);
``` [1](#0-0) 

The two `assert!` calls only verify that the returned lengths fit within the expected digit-count upper bounds. They do **not** verify the arithmetic relationship `quotient * m + remainder == a`. After writing the oracle-supplied values into `quotient_dst` and `remainder_dst`, the function returns and the caller (`modpow`) uses those values as if they were correct. [2](#0-1) 

This code path is active only in the proving environment:

```rust
#[cfg(any(all(target_arch = "riscv32", feature = "proving"), test))]
impl<'a, O: IOOracle> ModexpAdvisor for OracleAdvisor<'a, O> {
``` [3](#0-2) 

In the forward (sequencer) path, the native `::modexp::modexp` crate is used instead, which computes the correct result:

```rust
#[cfg(not(any(all(target_arch = "riscv32", feature = "proving"), test)))]
let output = ::modexp::modexp(base.as_slice(), exponent.as_slice(), modulus.as_slice(), allocator);
``` [4](#0-3) 

The oracle documentation explicitly states that oracle responses are untrusted and **must** be validated by the calling code:

> "Oracle responses are treated as untrusted input and MUST be validated" [5](#0-4) 

The modexp delegation violates this contract.

---

### Impact Explanation

A malicious prover controlling the oracle advice can supply any quotient/remainder pair that satisfies only the length bounds. Because the RISC-V program does not verify the arithmetic identity, the ZK circuit (which constrains the RISC-V execution trace) also does not constrain the oracle-supplied values. The prover can therefore generate a valid ZK proof for a state transition in which the modexp precompile returned an attacker-chosen output.

Concrete consequences:
- Any EVM contract that uses the modexp precompile (e.g., RSA signature verification, Diffie-Hellman key exchange, identity-based cryptography) can be made to return an attacker-chosen result.
- This creates a **forward/proving divergence**: the sequencer computes the correct modexp result, but the prover proves a different (attacker-chosen) result. The settlement layer accepts the proof as valid.
- Downstream effects include bypassing on-chain signature checks, manipulating cryptographic accumulators, or corrupting any state that depends on modexp output.

---

### Likelihood Explanation

The attacker is the prover/operator who constructs the ZK witness. In a ZK rollup, the prover is a privileged but potentially adversarial role (e.g., a rogue sequencer, a compromised proving service, or a future permissionless prover). The attack requires no special on-chain access — only the ability to supply oracle responses to the RISC-V execution. The `ArithmeticQuery` oracle processor in `callable_oracles/src/arithmetic/mod.rs` is the forward-mode counterpart that correctly computes division, but in proving mode the prover supplies the values directly with no cross-check. [6](#0-5) 

---

### Recommendation

After receiving the oracle-supplied quotient `q` and remainder `r` for the division `a / m`, add an arithmetic verification step before using the values:

```rust
// Verify: q * m + r == a  (and r < m)
let mut reconstructed = q.clone();
reconstructed.mul_assign(&m, advisor, allocator.clone());
reconstructed.add_assign(&r);
assert_eq!(reconstructed, *a, "modexp oracle advice failed arithmetic check");
assert!(r < *m, "modexp oracle remainder >= modulus");
```

This ensures the ZK circuit constrains the oracle advice to be arithmetically correct, closing the gap between forward and proving execution.

---

### Proof of Concept

1. Deploy an EVM contract that calls the modexp precompile with known inputs, e.g. `base=2, exp=10, mod=1000` (expected result: `24`).
2. In the proving oracle (`ArithmeticQuery` or equivalent), intercept the `MODEXP_ADVICE_QUERY_ID` query and return a crafted quotient/remainder pair that satisfies the length bounds but encodes a wrong division result (e.g., one that will cause the final `modpow` to return `42` instead of `24`).
3. Run the RISC-V prover. The `assert!` length checks pass. The `modpow` function computes `42` using the wrong intermediate values.
4. The prover generates a valid ZK proof for the state transition where modexp returned `42`.
5. The settlement layer verifies the proof and accepts the incorrect state transition. [7](#0-6) [8](#0-7)

#[cfg(any(all(target_arch = "riscv32", feature = "proving"), test))]
impl<'a, O: IOOracle> ModexpAdvisor for OracleAdvisor<'a, O> {
    fn get_reduction_op_advice<A: Allocator + Clone>(
        &mut self,
        a: &BigintRepr<A>,
        m: &BigintRepr<A>,
        quotient_dst: &mut BigintRepr<A>,
        remainder_dst: &mut BigintRepr<A>,
    ) {
        let arg: ModExpAdviceParams = {
            let a_len = a.digits;
            let a_ptr = a.backing.as_ptr();

            let modulus_len = m.digits;
            let modulus_ptr = m.backing.as_ptr();

            assert!(modulus_len > 0);

            ModExpAdviceParams {
                op: 0,
                a_ptr: a_ptr.addr() as u32,
                a_len: a_len as u32,
                b_ptr: 0,
                b_len: 0,
                modulus_ptr: modulus_ptr.addr() as u32,
                modulus_len: modulus_len as u32,
            }
        };

        // We assume that oracle's response is well-formed lengths-wise, and we will check value-wise separately
        let mut it = self
            .inner
            .raw_query(
                MODEXP_ADVICE_QUERY_ID,
                &((&arg as *const ModExpAdviceParams).addr() as u32),
            )
            .unwrap();

        let q_len = it.next().expect("quotient length");
        let r_len = it.next().expect("remainder length");

        let max_quotient_digits = if a.digits < m.digits {
            0
        } else if a.digits == m.digits {
            1
        } else {
            a.digits + 1 - m.digits
        };

        let max_remainder_digits = m.digits;

        const {
            assert!(core::mem::size_of::<usize>() == core::mem::size_of::<u32>());
        }

        // check that hint is "sane" in upper bound

        assert!(q_len.next_multiple_of(8) / 8 <= max_quotient_digits);
        assert!(r_len.next_multiple_of(8) / 8 <= max_remainder_digits);

        write_bigint(&mut it, q_len, quotient_dst);
        write_bigint(&mut it, r_len, remainder_dst);

        assert!(it.next().is_none());
    }
