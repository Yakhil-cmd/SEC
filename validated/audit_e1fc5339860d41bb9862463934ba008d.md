Audit Report

## Title
ckERC20 Minter Does Not Verify ERC20 `transfer()` Return Value, Causing Silent Withdrawal Failure and Permanent Fund Loss - (File: `rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

## Summary
The ckETH minter finalizes ckERC20 withdrawals by inspecting only the top-level Ethereum transaction receipt status (`TransactionStatus::Success` / `TransactionStatus::Failure`). It never inspects the ABI-decoded return value of the ERC20 `transfer()` call. For non-conforming ERC20 tokens that signal failure by returning `false` without reverting, the receipt status is `Success`, the minter marks the withdrawal as complete, burns the user's ckERC20 tokens permanently, and never triggers reimbursement — resulting in permanent loss of user funds with no protocol-level recovery path.

## Finding Description
The reimbursement gate in `record_finalized_transaction` (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, lines 733–746) only schedules a reimbursement for `CkErc20` withdrawals when `receipt.status == TransactionStatus::Failure`:

```rust
WithdrawalRequest::CkErc20(request) => {
    if receipt.status == TransactionStatus::Failure {
        self.record_reimbursement_request(...)
    }
}
```

Simultaneously, `update_balance_upon_withdrawal` (`rs/ethereum/cketh/minter/src/state.rs`, lines 377–383) decrements the minter's internal ERC20 balance whenever `receipt.status == TransactionStatus::Success`:

```rust
if receipt.status == TransactionStatus::Success && !tx.transaction_data().is_empty() {
    let TransactionCallData::Erc20Transfer { to: _, value } = TransactionCallData::decode(
        tx.transaction_data(),
    ).expect("BUG: ...");
    self.erc20_balances.erc20_sub(*tx.destination(), value);
}
```

The minter constructs a raw EIP-1559 transaction encoding `transfer(recipient, amount)` (selector `0xa9059cbb`) and submits it to Ethereum. For ERC20 tokens that return `false` on failure without reverting (a well-documented non-conforming pattern present in tokens such as ZRX and BAT, and also triggerable by token-internal conditions like blacklists or paused state), the Ethereum transaction receipt carries `status = 1` (Success) even though no tokens moved. The minter's finalization path (`rs/ethereum/cketh/minter/src/withdraw.rs`, lines 451–461) emits a `FinalizedTransaction` event with this receipt, causing the state machine to:

1. Skip the reimbursement branch (because `status != Failure`).
2. Decrement `erc20_balances` as if the transfer succeeded.
3. Leave the user's ckERC20 burn on the IC ledger as permanent.

The return value `success: bool` declared in the `transfer` ABI (confirmed in `rs/ethereum/cketh/minter/src/state/transactions/tests.rs`, lines 548–572) is never read from the receipt's return data. No guard exists anywhere in the finalization path to detect a `false` return.

By contrast, the deposit-side helper contract (`rs/ethereum/cketh/minter/ERC20DepositHelper.sol`, lines 498–503) uses OpenZeppelin's `safeTransferFrom`, which reverts on a `false` return — the withdrawal path has no equivalent safeguard.

## Impact Explanation
This is a **permanent, unrecoverable loss of user funds** in the ckERC20 system, which is an explicitly in-scope Chain Fusion / ck-token financial integration. For any supported ckERC20 token whose `transfer()` returns `false` without reverting:

- The user's ckERC20 tokens are permanently burned on the IC ledger.
- No ERC20 tokens are received on Ethereum.
- The minter's internal `erc20_balances` diverges from the actual on-chain balance.
- No reimbursement is ever triggered; the user has no recovery path.

This matches the allowed High impact: **"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."** Severity is High rather than Critical because exploitation requires a supported token to exhibit the non-reverting `false` return behavior, which constrains the affected token set.

## Likelihood Explanation
An unprivileged user triggers the bug with no special access once a qualifying token is supported. Two realistic paths exist:

1. **Future NNS governance addition**: Any NNS proposal that adds a token using the non-reverting `false` return pattern (ZRX, BAT, and others from the pre-EIP-20 era are real examples) immediately exposes all users of that token.
2. **Token-internal conditions on an already-supported token**: If a currently supported ckERC20 token has a blacklist, pause mechanism, or per-address transfer cap that causes `transfer()` to return `false` without reverting, any user who calls `withdraw_erc20` with an affected recipient triggers the bug without any governance action. This path requires no privileged access whatsoever.

Neither path requires phishing, social engineering, leaked keys, or governance-majority corruption. The trigger is a normal user withdrawal call.

## Recommendation
The withdrawal path must be hardened to treat a `false` ERC20 `transfer()` return as a failure, consistent with how the deposit side already uses `safeTransferFrom`. Two concrete approaches:

1. **Ethereum-side wrapper contract**: Deploy a thin proxy that calls `transfer()`, checks the return value, and reverts if it is `false` or absent. This causes the Ethereum transaction to revert (`status = Failure`), which the existing reimbursement logic already handles correctly.
2. **Receipt return-data inspection**: If `eth_getTransactionReceipt` provides the call return data, ABI-decode it and treat a `false` bool return as equivalent to `TransactionStatus::Failure`, routing the withdrawal into the existing reimbursement path.

Option 1 is preferred because it requires no changes to the receipt-parsing logic and mirrors the deposit-side design.

## Proof of Concept
**Minimal deterministic integration test plan:**

1. Deploy a mock ERC20 contract on a local Anvil/Hardhat fork whose `transfer(address, uint256)` always returns `false` without reverting (no `revert` opcode, EVM exits with status `1`).
2. Register this token as a supported ckERC20 token in a local minter instance (via `record_add_ckerc20_token`).
3. Simulate a user deposit to mint ckERC20 tokens on the IC ledger.
4. Call `withdraw_erc20` on the minter canister, specifying any recipient address.
5. Advance the mock Ethereum chain; the transaction receipt has `status = 1`.
6. Run `finalize_transactions_batch`; observe that `record_finalized_transaction` takes the `CkErc20` branch and does **not** call `record_reimbursement_request` (because `receipt.status != Failure`).
7. Assert: `reimbursement_requests` is empty, the user's ckERC20 burn index is finalized with no reimbursement, and `erc20_balances` has been decremented despite no tokens moving on-chain.

This test is fully reproducible in the existing PocketIC / integration-test harness used by the ckETH minter test suite without any mainnet interaction.
