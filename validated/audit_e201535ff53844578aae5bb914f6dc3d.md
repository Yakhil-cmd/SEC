### Title
`ExitToEthereum` Precompile Has No Error-Refund Callback, Permanently Freezing Burned ERC-20 Tokens on Promise Failure - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile schedules a `withdraw` promise on the NEP-141 eth-connector contract after ERC-20 tokens have already been burned in the EVM. Unlike `ExitToNear`, which attaches an `exit_to_near_precompile_callback` to handle promise failures and re-mint burned tokens, `ExitToEthereum` creates only a bare `PromiseArgs::Create` with no error callback. If the asynchronous `withdraw` promise fails for any reason, the ERC-20 tokens are permanently destroyed with no recovery path.

---

### Finding Description

The `ExitToEthereum::run()` function in `engine-precompiles/src/native.rs` constructs and emits a single `withdraw` promise with no attached callback:

```rust
let withdraw_promise = PromiseCreateArgs { ... method: "withdraw" ... };
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [1](#0-0) 

There is no `PromiseArgs::Callback` wrapping this promise. Compare this to `ExitToNear::run()`, which (when the `error_refund` feature is enabled) wraps the transfer promise in a `PromiseArgs::Callback` pointing to `exit_to_near_precompile_callback`:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            method: "exit_to_near_precompile_callback".to_string(), ...
        },
    })
};
``` [2](#0-1) 

The `exit_to_near_precompile_callback` handler explicitly re-mints burned ERC-20 tokens or refunds ETH when the base promise fails:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
``` [3](#0-2) 

`refund_on_error` re-mints the burned ERC-20 tokens to the original sender:

```rust
if let Some(erc20_address) = args.erc20_address {
    // ERC-20 exit; re-mint burned tokens
``` [4](#0-3) 

No equivalent mechanism exists for `ExitToEthereum`. The `ExitToEthereum` struct has no `current_account_id` field and no callback infrastructure. [5](#0-4) 

The user-facing entry points are `EvmErc20.withdrawToEthereum` and `EvmErc20V2.withdrawToEthereum`. Both burn ERC-20 tokens **before** calling the precompile:

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    // ...
    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, ...)
    }
}
``` [6](#0-5) [7](#0-6) 

The EVM transaction commits the burn and the promise is scheduled. If the NEP-141 `withdraw` promise then fails asynchronously (e.g., eth-connector is paused, insufficient NEP-141 balance in the engine account, or any other runtime failure), the ERC-20 tokens are gone with no refund path.

The codebase itself acknowledges this asymmetry in test comments for `ExitToNear`:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [8](#0-7) 

For `ExitToEthereum`, there is no refund even when `error_refund` is enabled.

---

### Impact Explanation

Any user who calls `withdrawToEthereum` on a bridged ERC-20 token (`EvmErc20` or `EvmErc20V2`) and whose asynchronous `withdraw` promise fails will have their ERC-20 tokens permanently destroyed. The NEP-141 tokens remain in the Aurora engine account (not burned), but the user's ERC-20 representation is gone with no recovery mechanism. This constitutes a **permanent freezing of user funds** (the ERC-20 value is destroyed; the underlying NEP-141 value is stranded in the engine).

---

### Likelihood Explanation

The `withdraw` promise on the eth-connector can fail in realistic scenarios:
- The eth-connector contract is paused (the `pause_precompiles` admin function exists and is callable by authorized accounts).
- The Aurora engine's NEP-141 balance in the eth-connector is insufficient due to an accounting discrepancy.
- The eth-connector undergoes a migration or upgrade that temporarily rejects `withdraw` calls.

Any user who initiates a `withdrawToEthereum` during such a window loses their tokens permanently. The `pause_precompiles` mechanism is a documented operational feature, making this a realistic trigger. [9](#0-8) 

---

### Recommendation

Add a `current_account_id` field to `ExitToEthereum` (mirroring `ExitToNear`) and wrap the `withdraw` promise in a `PromiseArgs::Callback` that calls a new `exit_to_ethereum_precompile_callback` method on the engine. That callback should check the promise result and, on failure, re-mint the burned ERC-20 tokens to the original sender using `refund_on_error`, exactly as `exit_to_near_precompile_callback` does.

---

### Proof of Concept

1. User holds 100 units of a bridged ERC-20 token on Aurora (backed by NEP-141 in the eth-connector).
2. User calls `EvmErc20.withdrawToEthereum(recipientEthAddress, 100)`.
3. `_burn(msg.sender, 100)` executes — ERC-20 balance drops to 0.
4. The `ExitToEthereum` precompile succeeds synchronously and schedules `withdraw` on the eth-connector.
5. Between the EVM transaction and the async promise execution, the eth-connector is paused by an authorized account (or any other failure condition occurs).
6. The `withdraw` promise fails. No callback exists to re-mint the 100 ERC-20 tokens.
7. User has 0 ERC-20 tokens on Aurora and 0 ETH received on Ethereum. Funds are permanently lost. [10](#0-9) [6](#0-5)
