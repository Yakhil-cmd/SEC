### Title
Permanent ERC-20 Token Freeze on Failed `exit_to_near` When `error_refund` Feature Is Disabled — (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns ERC-20 tokens before scheduling an `ft_transfer` promise to the NEP-141 contract. When the `error_refund` compile-time feature is absent — which is the case in the production build — no callback is attached to the outgoing promise. If the `ft_transfer` fails (e.g., the recipient is not registered with the NEP-141 contract), the burned ERC-20 tokens are never re-minted and are permanently destroyed.

---

### Finding Description

The production WASM is built with `CARGO_FEATURES_BUILD = "contract"` and does not include `error_refund`. [1](#0-0) 

The `error_refund` feature is defined but is not part of the `default` feature set in either the precompiles crate or the engine crate. [2](#0-1) [3](#0-2) 

Inside `ExitToNear::run`, the `callback_args.refund` field is unconditionally set to `None` when the feature is absent: [4](#0-3) 

Because `transfer_near` is also `None` for the standard ERC-20 exit path (only set for the wNEAR-unwrap path), `callback_args` equals its default value. The branch at line 470 therefore schedules only a bare `PromiseArgs::Create` — no callback is attached: [5](#0-4) 

The `exit_to_near_precompile_callback` that would re-mint the burned tokens is therefore never called on failure: [6](#0-5) 

The test suite explicitly documents this behaviour: without `error_refund`, the ERC-20 balance after a failed exit is `FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT` (i.e., the tokens are gone): [7](#0-6) 

The same loss applies to the ETH (base-token) exit path: [8](#0-7) 

---

### Impact Explanation

When `ft_transfer` fails, the ERC-20 tokens have already been burned inside the EVM state transition. Without a callback to detect the failure and call `refund_on_error` (which re-mints them), those tokens are permanently destroyed. The corresponding NEP-141 tokens remain locked in the Aurora engine account with no mechanism to release them to the user. This constitutes a **permanent freeze of user funds**.

---

### Likelihood Explanation

The trigger condition is straightforward and requires no special privilege:

1. A user calls `withdrawToNear` on any bridged ERC-20 token, specifying a NEAR account ID that is not registered with the NEP-141 contract (e.g., a freshly created account, a typo, or any account that has never called `storage_deposit` on that token).
2. The EVM burns the ERC-20 tokens and schedules `ft_transfer`.
3. `ft_transfer` panics because the recipient has no storage deposit.
4. No callback exists to detect the failure; the tokens are gone.

This is a realistic, low-effort mistake any ordinary user can make. The NEP-141 storage-registration requirement is a well-known footgun on NEAR, and unregistered recipients are common.

---

### Recommendation

Enable the `error_refund` feature unconditionally in the production build by adding it to `CARGO_FEATURES_BUILD` in `Makefile.toml`, or move the refund-callback logic out from behind the feature flag so it is always compiled in. The callback infrastructure (`exit_to_near_precompile_callback`, `refund_on_error`) already exists and is correct; it simply needs to be reachable from the production binary.

---

### Proof of Concept

1. Deploy Aurora engine **without** the `error_refund` feature (current production configuration).
2. Bridge any NEP-141 token to Aurora, obtaining an ERC-20 balance.
3. Call `withdrawToNear(bytes,uint256)` on the ERC-20 contract, passing a NEAR account ID that has never called `storage_deposit` on the NEP-141 contract.
4. Observe: the ERC-20 balance decreases by the exit amount; the NEP-141 balance of the recipient remains zero; the NEP-141 balance of the Aurora engine account is unchanged (tokens locked); no re-mint occurs.
5. The test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` (lines 623–665) already encodes this exact scenario and confirms the loss when compiled without `error_refund`. [9](#0-8)
