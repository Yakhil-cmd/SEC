### Title
Unchecked Precompile Call Return Value Causes Permanent Token Burn Without Withdrawal - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

In `EvmErc20.sol`, both `withdrawToNear()` and `withdrawToEthereum()` burn the caller's ERC-20 tokens first, then invoke the Aurora exit precompile via a low-level assembly `call`. The return value of that `call` is captured into a local variable `res` but is **never checked**. If the precompile call fails (returns 0), the tokens are permanently destroyed with no corresponding NEP-141 release on NEAR or ETH release on Ethereum.

---

### Finding Description

`withdrawToNear` and `withdrawToEthereum` follow this pattern:

1. `_burn(_msgSender(), amount)` — irreversibly destroys the user's ERC-20 tokens.
2. An inline assembly `call` is made to the Aurora exit precompile address.
3. The return value `res` is assigned but **never read or branched on**.

```solidity
// withdrawToNear (lines 53–63)
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked
    }
}

// withdrawToEthereum (lines 65–76)
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
        // res is never checked
    }
}
``` [1](#0-0) [2](#0-1) 

The Aurora exit precompiles (`exitToNear` at `0xe921...` and `exitToEthereum` at `0xb0bd...`) are implemented in `engine-precompiles/src/native.rs` and can return failure (EVM `call` returns 0) under several conditions — for example, if the recipient account ID is malformed, if the precompile is paused, or if the cross-chain promise cannot be scheduled. [3](#0-2) 

Because the Solidity code does not revert on a failed `call`, the outer transaction succeeds: the ERC-20 tokens are burned, the event is emitted, but no cross-chain withdrawal is ever initiated.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user who calls `withdrawToNear` or `withdrawToEthereum` under conditions that cause the precompile to fail will have their ERC-20 tokens permanently destroyed. The corresponding NEP-141 tokens held by the Aurora engine contract on NEAR are never released, and the user receives nothing on either side of the bridge. The funds are frozen forever with no recovery path.

---

### Likelihood Explanation

**Medium.** The precompile call can fail in realistic scenarios:

- A user supplies a malformed or too-long NEAR recipient account ID in `withdrawToNear`.
- The exit precompile is paused by the Aurora admin (a supported operational state).
- Insufficient gas is forwarded to the precompile (though `gas()` is passed, edge cases exist).

Any of these conditions, which are reachable by an ordinary EVM user without any privileged access, trigger the silent failure path.

---

### Recommendation

Check the return value of the assembly `call` and revert if it is zero, so that the `_burn` is also reverted:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum`. This ensures atomicity: either both the burn and the cross-chain exit succeed, or neither does.

---

### Proof of Concept

1. Deploy `EvmErc20` on Aurora with a NEP-141 token that has been bridged.
2. Pause the `exitToNear` precompile via the Aurora admin interface.
3. Call `withdrawToNear(recipient, amount)` as a normal token holder.
4. Observe: the transaction succeeds, `Transfer` event is emitted with `to = address(0)` (burn), the caller's balance is reduced to zero.
5. Observe: no NEP-141 tokens are released on NEAR; the precompile call silently returned 0 and was ignored.
6. The user's tokens are permanently lost with no recourse. [1](#0-0) [2](#0-1)
