Audit Report

## Title
Unbounded While-Loop DoS via Attacker-Controlled `ingress_end = u64::MAX` in `construction_payloads` — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary
The `construction_payloads` function accepts `ingress_end` directly from an unauthenticated HTTP request body with no upper-bound validation. Supplying `ingress_end = u64::MAX` bypasses both existing guards and causes the subsequent while-loop to execute approximately 139 million iterations, allocating ~1.1 GB of memory before an OOM kill (debug build) or entering an infinite loop due to integer wrap-around (release build). This results in a complete denial-of-service of the ICRC1 Rosetta server.

## Finding Description
In `services.rs`, `ingress_end` is read directly from the caller-supplied metadata with no cap: [1](#0-0) 

Two guards exist. Guard 1 rejects `ingress_start >= ingress_end`: [2](#0-1) 

Guard 2 rejects `ingress_end < now + ingress_interval`: [3](#0-2) 

Neither guard imposes an upper bound on `ingress_end`. With `ingress_end = u64::MAX ≈ 1.844×10¹⁹`, both checks pass trivially (any `ingress_start < u64::MAX` satisfies guard 1; `u64::MAX` is never less than `now + ingress_interval`). The loop then runs uncapped: [4](#0-3) 

The step size is `ingress_interval - INGRESS_INTERVAL_OVERLAP`:
- `ingress_interval = (MAX_INGRESS_TTL - PERMITTED_DRIFT) = 300s - 60s = 240s = 240,000,000,000 ns` [5](#0-4) 

- `INGRESS_INTERVAL_OVERLAP = 120s = 120,000,000,000 ns` [6](#0-5) 

Step = `240,000,000,000 - 120,000,000,000 = 120,000,000,000 ns`. With `ingress_start ≈ 1.75×10¹⁸ ns` (current epoch) and `ingress_end = u64::MAX`:

```
iterations ≈ (1.844×10¹⁹ - 1.75×10¹⁸) / 1.2×10¹¹ ≈ 139,000,000
memory     ≈ 139,000,000 × 8 bytes ≈ 1.1 GB
```

In release mode, when `ingress_start` nears `u64::MAX`, the `+=` at line 165 wraps to a small value, making `ingress_start < u64::MAX` true again — producing an infinite loop. In debug mode, the `ingress_start + ingress_interval` addition at line 164 panics on overflow after OOM.

## Impact Explanation
A single unauthenticated HTTP POST to `/construction/payloads` either exhausts all server memory (~1.1 GB minimum) causing an OOM kill, or locks the server in an infinite CPU loop in release builds. This is a complete, persistent denial-of-service of the ICRC1 Rosetta server — an explicitly in-scope financial integration component. This matches the allowed High impact: *"Significant Rosetta … security impact with concrete user or protocol harm"* and *"Application/platform-level DoS … not based on raw volumetric DDoS"* ($2,000–$10,000).

## Likelihood Explanation
No authentication is required for `/construction/payloads` by Rosetta API design. The malicious payload is a single valid JSON body with `ingress_end` set to `18446744073709551615`. The attack requires no prior state, no special privileges, and is immediately and repeatably triggerable by any external user.

## Recommendation
Add an explicit upper-bound check on the `ingress_end - ingress_start` range before entering the loop. For example:

```rust
let max_range = ingress_interval * MAX_INGRESS_EXPIRY_COUNT as u64; // e.g., cap at 24h
if ingress_end.saturating_sub(ingress_start) > max_range {
    return Err(Error::processing_construction_failed(
        &"Ingress window exceeds maximum allowed range"
    ));
}
```

Alternatively, break out of the loop after a hard cap (e.g., 1440 entries for a 24-hour window at 1-minute steps), or use `ingress_start.checked_add(ingress_interval)` inside the loop body to detect and reject overflow.

## Proof of Concept

```rust
use std::time::SystemTime;

let now = SystemTime::now();
let result = construction_payloads(
    valid_operations(),
    Some(ConstructionPayloadsRequestMetadata {
        ingress_start: Some(
            now.duration_since(SystemTime::UNIX_EPOCH).unwrap().as_nanos() as u64
        ),
        ingress_end: Some(u64::MAX),  // attacker-controlled
        created_at_time: None,
        memo: None,
    }),
    &some_principal,
    vec![valid_public_key()],
    now,
);
// Release build: infinite loop, server permanently unresponsive
// Debug build: OOM panic after ~139M iterations / ~1.1 GB allocation
```

A unit test can be added directly to the existing `#[cfg(test)]` module in `services.rs` calling `construction_payloads` with `ingress_end = Some(u64::MAX)` and asserting it returns an error — currently it will hang or panic instead.
