### Title
SNS Treasury Manager Deposit Lacks Execution-Time Balance Re-Validation, Enabling Full Treasury Drain into External Custodian - (File: rs/sns/governance/src/extensions.rs)

### Summary

The SNS Treasury Manager extension framework mirrors the ChainportMainBridge vulnerability class exactly: treasury funds are transferred to an `external_custodian` (e.g., a DEX canister), but the SNS Governance canister maintains no on-chain accounting of how much is currently deposited there. The 50% balance guard is checked only at proposal submission time, not at execution time. Two concurrent deposit proposals can each independently pass the 50% check and together drain up to 100% of the treasury into the external custodian. Once funds are in the external custodian, if that canister becomes unavailable, there is no on-chain recovery path.

### Finding Description

The deposit flow in the SNS Treasury Manager extension is:

```
treasury_owner (SNS Governance) → treasury_manager canister → external_custodian (DEX)
```

In `validate_deposit_operation_impl` (`rs/sns/governance/src/extensions.rs`, lines 276–321), the governance canister checks that the requested deposit does not exceed 50% of the current treasury balance:

```rust
// Fail if either is asking for more than 50% of current balance.  The balance could have changed
// since the proposal was created, and we don't assume that the proposal should work
if sns_requested > sns_balance.checked_div(2).unwrap() {
    return Err(...)
}
```

This check is performed **only at proposal submission/validation time**. The execution function `execute_treasury_manager_deposit` (lines 1545–1610) performs no equivalent re-validation:

1. It calls `approve_treasury_manager` to grant an ICRC-2 allowance.
2. It calls `deposit` on the extension canister.
3. The extension canister pulls funds and forwards them to the external custodian.

There is no on-chain state in the governance canister tracking how much of the treasury is currently held by any external custodian. The `balances` query on the treasury manager canister returns a locally cached view refreshed periodically by `refresh_balances`, not a live on-chain invariant enforced by governance.

Contrast this with `TransferSnsTreasuryFunds`, which has an explicit execution-time re-check (`transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`, `rs/sns/governance/src/proposal.rs`, lines 2600–2658). No equivalent guard exists for treasury manager deposits.

Additionally, the `withdraw` path (`execute_treasury_manager_withdraw`, lines 1612–1661) simply calls the extension canister's `withdraw` method. If the external custodian is unavailable (stopped, upgraded, or failed), the call fails with no fallback or on-chain recovery mechanism.

### Impact Explanation

**Scenario 1 – Concurrent proposal drain**: Two deposit proposals are submitted simultaneously against the same treasury balance. Both pass the 50% check independently (e.g., each requests 40%). Both are approved by governance voting. Both execute sequentially. The second execution finds no re-validation gate and drains an additional 40%, for a combined 80% of the treasury deposited into the external custodian. With three proposals, 100% can be drained.

**Scenario 2 – External custodian unavailability**: After funds are deposited into the external custodian (DEX), if that canister is stopped, upgraded with breaking changes, or its liquidity pool is drained by its own governance, the SNS treasury funds are locked. The governance canister has no on-chain mechanism to recover them — it can only call `withdraw` on the extension canister, which in turn calls the DEX. If the DEX call fails, the proposal fails, and the funds remain stranded.

**Impact**: Permanent or prolonged loss of SNS treasury funds (ICP and SNS tokens) with no on-chain recovery path. This is a ledger conservation bug: the invariant `managed_assets[k] == treasury_manager[k] + treasury_owner[k] + external_custodian[k]` documented in the DID file is not enforced by the governance canister.

### Likelihood Explanation

Any SNS neuron holder with sufficient voting power can submit deposit proposals. The concurrent proposal scenario requires no special access — two proposals can be submitted in the same voting period. The external custodian unavailability scenario depends on the DEX's own governance or operational failures, which are outside the SNS's control. The KongSwap extension was already removed from the allowed list (`ALLOWED_EXTENSIONS` in `rs/sns/governance/src/extensions.rs`, lines 48–53) after KongSwap ceased operations on April 6, 2026, demonstrating that external custodian failure is a realistic, already-materialized risk.

### Recommendation

1. **Add execution-time re-validation**: In `execute_treasury_manager_deposit`, re-query the current treasury balance and re-apply the 50% limit before executing, mirroring the pattern used in `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`.

2. **Track deposited amounts on-chain**: The governance canister should maintain a per-extension-canister accounting of how much has been deposited but not yet withdrawn, and enforce that the sum of all pending deposits does not exceed the configured limit.

3. **Add a withdrawal pre-check**: Before executing a deposit, verify that the extension canister's `withdraw` path is functional (e.g., by querying `balances`) to ensure funds can be recovered.

4. **Enforce a minimum retained balance**: Require that a configurable minimum fraction of the treasury always remains in the governance canister's direct control, not delegated to any extension.

### Proof of Concept

1. SNS treasury holds 1,000,000 SNS tokens and 500,000 ICP.
2. Neuron A submits `ExecuteExtensionOperation { operation_name: "deposit", treasury_allocation_sns_e8s: 400_000_000_000, treasury_allocation_icp_e8s: 200_000_000_000 }`. Validation passes (40% ≤ 50%).
3. Neuron B simultaneously submits an identical proposal. Validation passes against the same current balance (40% ≤ 50%).
4. Both proposals pass governance voting.
5. Proposal A executes: `execute_treasury_manager_deposit` calls `approve_treasury_manager` then `deposit`. 400,000 SNS and 200,000 ICP move to the DEX. No execution-time re-check occurs.
6. Proposal B executes: same code path, no re-check. Another 400,000 SNS and 200,000 ICP move to the DEX.
7. 80% of the SNS treasury and 80% of the ICP treasury are now in the external custodian. The governance canister has no on-chain record of this.
8. If the DEX canister is stopped, `execute_treasury_manager_withdraw` fails, and the funds are permanently inaccessible.

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

/// Execute a treasury manager deposit operation
async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())
}
