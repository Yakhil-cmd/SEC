### Title
Unprotected Legacy Transactions (v=27/28) Accepted Without Chain-ID Enforcement, Enabling Cross-Chain Replay - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

ZKsync OS explicitly accepts pre-EIP-155 legacy transactions (those with `v = 27` or `v = 28`) and computes their signing hash **without** including the chain ID. Any legacy transaction originally signed on Ethereum mainnet or any other EVM-compatible chain before EIP-155 was adopted can be replayed verbatim on ZKsync OS, provided the sender's nonce on ZKsync OS matches the nonce in the original transaction.

---

### Finding Description

`LegacySignatureData::is_eip155()` classifies a transaction as unprotected when `v == 27 || v == 28`: [1](#0-0) 

`LegacyPayloadParser::try_parse_and_hash_for_signature_verification()` then takes the `is_eip155() == false` branch and hashes only the 6-field payload (nonce, gasPrice, gasLimit, to, value, data) — **no chain ID is mixed in**: [2](#0-1) 

The caller in `RlpEncodedTxInner::parse_and_compute_signed_hash()` accepts this result without any rejection: [3](#0-2) 

The resulting `RlpEncodedTxInner::Legacy` variant propagates through the full validation pipeline in `validate_and_compute_fee_for_transaction()`. The ecrecover check in the ZK validation path verifies the recovered address against `from`, but since the signing hash was computed without chain ID, a signature valid on Ethereum mainnet is equally valid here: [4](#0-3) 

The `chain_id()` accessor explicitly returns `None` for the unprotected variant, confirming no chain-ID binding exists at any later stage: [5](#0-4) 

---

### Impact Explanation

An attacker who obtains a pre-EIP-155 legacy transaction (v=27 or v=28) originally broadcast on Ethereum mainnet or any other EVM chain can replay it on ZKsync OS if:

1. The victim's ZKsync OS account nonce equals the nonce in the original transaction.
2. The victim holds funds on ZKsync OS at the same address.

Because the signing hash is chain-agnostic, the ECDSA signature passes `ecrecover` verification unchanged. The transaction then executes normally — transferring ETH or invoking contracts — resulting in direct, irreversible loss of user funds. The nonce constraint limits the attack surface but does not eliminate it: a user whose ZKsync OS nonce is 0 and who sent a pre-EIP-155 transaction with nonce 0 on mainnet is immediately vulnerable.

---

### Likelihood Explanation

Pre-EIP-155 transactions (v=27/28) exist on Ethereum mainnet from before the Spurious Dragon hard fork (block 2,675,000, late 2016). Any user who:
- Held ETH on mainnet before EIP-155 and sent at least one transaction, **and**
- Later bridged funds to ZKsync OS at the same address

is a potential victim if their ZKsync OS nonce matches the nonce of any of their old pre-EIP-155 transactions. For accounts that are new to ZKsync OS (nonce = 0), the first pre-EIP-155 transaction they ever sent on mainnet is directly replayable. The attack requires no privileged access: the attacker only needs to find the historical raw transaction bytes (publicly available on-chain) and submit them to ZKsync OS.

---

### Recommendation

Reject unprotected legacy transactions at the bootloader level. In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, return `Err(InvalidTransaction::InvalidEncoding)` when `legacy_signature.is_eip155() == false`. This mirrors the enforcement already applied to all typed transaction formats (EIP-2930, EIP-1559, EIP-4844, EIP-7702), each of which hard-rejects a chain-ID mismatch: [6](#0-5) 

If backward compatibility with pre-EIP-155 tooling is required, the rejection should at minimum be enforced by default and opt-out only through an explicit operator configuration flag, not silently accepted for all transactions.

---

### Proof of Concept

**Setup:**
- Alice sent a pre-EIP-155 ETH transfer on Ethereum mainnet at nonce 0 (v=27). Raw RLP bytes are publicly available on-chain.
- Alice later bridges 10 ETH to ZKsync OS. Her ZKsync OS nonce is 0.

**Attack:**
1. Attacker retrieves Alice's raw pre-EIP-155 transaction bytes from Ethereum mainnet.
2. Attacker submits the identical bytes as an RLP-encoded L2 transaction to ZKsync OS.
3. `LegacyPayloadParser` detects `v == 27`, takes the unprotected branch, and computes `sig_hash = keccak256(rlp([nonce, gasPrice, gasLimit, to, value, data]))` — identical to the Ethereum mainnet signing hash.
4. `ecrecover(sig_hash, v=27, r, s)` recovers Alice's address. The `recovered_from == from` check passes.
5. Nonce check: Alice's ZKsync OS nonce is 0, matching the transaction nonce. Nonce is incremented.
6. The transaction executes, transferring Alice's ZKsync OS ETH to the original `to` address (which the attacker may control or monitor).
