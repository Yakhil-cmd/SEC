Audit Report

## Title
Missing Anonymous Principal Validation in `error_refund_icp` Allows Draining of Swap Canister's Anonymous Subaccount - (File: rs/sns/swap/src/swap.rs)

## Summary
The `error_refund_icp` function in the SNS Swap canister accepts a caller-controlled `source_principal_id` without validating that it is not the anonymous principal. Any unprivileged ingress sender can trigger a transfer of ICP from the swap canister's anonymous subaccount to the anonymous principal's main ICP ledger account. Because the ICP ledger's `can_send` guard does not block the anonymous principal, a subsequent `icrc1_transfer` call as anonymous completes the theft.

## Finding Description
`error_refund_icp` at `rs/sns/swap/src/swap.rs` L1938–1948 unpacks `source_principal_id` from the request and proceeds directly to balance query and transfer with no anonymous principal check. [1](#0-0) 

The function computes `source_subaccount = principal_to_subaccount(source_principal_id)`, queries the balance of `swap_canister[anonymous_subaccount]`, and transfers `balance - DEFAULT_TRANSFER_FEE` to `Account { owner: anonymous, subaccount: None }` on the ICP ledger. [2](#0-1) 

The canister-level handler at `rs/sns/swap/canister/canister.rs` L161–167 passes the request directly to the core logic with no caller restriction or additional validation. [3](#0-2) 

By contrast, `new_sale_ticket` at L2537–2539 explicitly rejects anonymous callers, confirming the pattern is known and intentionally applied elsewhere but omitted here. [4](#0-3) 

The ICP ledger's `icrc1_transfer` endpoint calls `can_send` to gate transfers. A grep for `is_anonymous` in `rs/ledger_suite/icp/ledger/src/lib.rs` returns no matches, confirming `can_send` does not block the anonymous principal from sending tokens it holds. [5](#0-4) 

Exploit flow:
1. ICP arrives in `swap_canister[principal_to_subaccount(anonymous)]` (accidental user transfer or deliberate attacker deposit racing another depositor).
2. Swap reaches `Lifecycle::Aborted` or `Lifecycle::Committed`.
3. Attacker calls `error_refund_icp({ source_principal_id: Some(anonymous) })` from any ingress identity.
4. ICP is transferred to `Account { owner: anonymous, subaccount: None }` on the ICP ledger.
5. Attacker calls `icrc1_transfer` on the ICP ledger as the anonymous principal, moving funds to the attacker's own account.

## Impact Explanation
This constitutes unauthorized access to canister-controlled ICP funds. The funds at risk are those held in the swap canister's anonymous subaccount. The exploit is fully executable by any unprivileged ingress sender once the preconditions are met, with no authentication or privilege required. This matches the **High** bounty impact class: "Unauthorized access to … canister-controlled funds where exploitation requires meaningful per-target work or other constraints" ($2,000–$10,000). The constraint is that ICP must be present in the anonymous subaccount, which limits the maximum loss per swap instance but does not prevent the theft.

## Likelihood Explanation
Medium-low. The anonymous subaccount of a swap canister can receive ICP via accidental user transfer (a user mistypes or uses the wrong principal). Every SNS swap eventually reaches `Aborted` or `Committed`, so the lifecycle precondition is always satisfied in the long run. The `error_refund_icp` endpoint is publicly callable with no authentication, making step 3 trivially reachable. The ICP ledger does not block anonymous-principal transfers, making step 5 executable. The scenario is not purely theoretical but requires the accidental-deposit precondition.

## Recommendation
Add an anonymous principal guard immediately after unpacking `source_principal_id`, consistent with the pattern in `new_sale_ticket`:

```rust
if source_principal_id.is_anonymous() {
    return ErrorRefundIcpResponse::new_invalid_request_error(
        "source_principal_id must not be the anonymous principal.",
    );
}
```

This should be inserted at `rs/sns/swap/src/swap.rs` after line 1948, before the buyer-state lookup. [6](#0-5) 

## Proof of Concept
1. Deploy an SNS swap canister and advance it to `Committed` or `Aborted` state.
2. Transfer any amount of ICP (e.g., 1 ICP) to `AccountIdentifier::new(swap_canister_id, Some(Subaccount::from(&PrincipalId::new_anonymous())))` on the ICP ledger.
3. Call `error_refund_icp({ source_principal_id: Some(PrincipalId::new_anonymous()) })` from any ingress identity.
4. Query `icrc1_balance_of` for `Account { owner: anonymous, subaccount: None }` on the ICP ledger — balance increases by `deposited_amount - DEFAULT_TRANSFER_FEE`.
5. Call `icrc1_transfer` on the ICP ledger as the anonymous principal with the attacker's account as destination — funds are moved to the attacker.

This can be implemented as a PocketIC integration test using the existing SNS swap test infrastructure in `rs/sns/swap/tests/`.
