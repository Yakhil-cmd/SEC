Audit Report

## Title
Zero-Cost `UpdateSettings` Ingress Induction Enables Execution Slot Exhaustion DoS - (`rs/cycles_account_manager/src/cycles_account_manager.rs` / `rs/execution_environment/src/execution_environment.rs`)

## Summary

`UpdateSettings` ingress messages with a payload ≤ 338 bytes are classified as `IngressInductionCost::Free` at every pre-execution checkpoint. Every cycle-balance gate in the ingress filter, ingress selector, and valid-set rule silently passes `Free`-classified messages without any balance check. After execution, the deferred fee collection explicitly discards the `CanisterOutOfCyclesError`. A canister drained to zero cycles can therefore submit `UpdateSettings` messages at zero ongoing cost, consuming execution slots and starving legitimate traffic.

## Finding Description

**Root cause – deferred fee returns `Free`:**
In `ingress_induction_cost()` (`rs/cycles_account_manager/src/cycles_account_manager.rs`, L570–595), when the method is `UpdateSettings` and `is_delayed_ingress_induction_cost(ingress.arg())` is true (payload ≤ `MAX_DELAYED_INGRESS_COST_PAYLOAD_SIZE` = 338 bytes, L43–45), `paying_canister` is set to `None`, which maps to `IngressInductionCost::Free`.

**All pre-execution gates pass `Free` unconditionally:**

- **Ingress filter** (`should_accept_ingress_message`, `rs/execution_environment/src/execution_environment.rs` L3351–3373): only the `IngressInductionCost::Fee` branch calls `can_withdraw_cycles_with_threshold`; `Free` is never checked.
- **Ingress selector** (`validate_ingress`, `rs/ingress_manager/src/ingress_selector.rs` L592–594): `IngressInductionCost::Free => { // Do nothing. }`.
- **Valid-set rule** (`enqueue`, `rs/messaging/src/scheduling/valid_set_rule.rs` L287–291): `IngressInductionCost::Free` pushes the message directly with no balance check.

**Post-execution fee collection silently discards the error:**
After `update_settings` executes, `rs/execution_environment/src/execution_environment.rs` L1043–1052 calls `consume_cycles` and assigns the result to `_ignore_error`, with the comment `// we ignore the error anyway`. A zero-balance canister pays nothing.

**Existing rate limits do not prevent the attack:**
`MAX_INGRESS_MESSAGES_PER_BLOCK = 1000` (`rs/limits/src/lib.rs` L78) is a global per-block cap shared by all canisters. The ingress selector uses a round-robin quota that limits a *single* canister's share per block, but an attacker controlling N canisters (each created once and drained to zero) can fill N quota slots per block. The `INGRESS_HISTORY_MAX_MESSAGES` cap (L37: `2 * 1000 * 300 = 600,000`) bounds the pool size but does not prevent sustained slot consumption at the per-block rate.

## Impact Explanation

An attacker with one or more zero-balance canisters can continuously saturate the subnet's ingress execution pipeline at zero ongoing cost. Legitimate canister calls compete for the same fixed per-block execution slots and experience increased latency or starvation. This is a concrete application/platform-level DoS with subnet availability impact, not based on raw volumetric DDoS — matching the **High ($2,000–$10,000)** bounty impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

The attack requires only: (1) creating one or more canisters on the target subnet (one-time cost, minimal cycles); (2) draining their balances to zero; (3) submitting `UpdateSettings` ingress messages in a loop with a payload ≤ 338 bytes. No privileged access, no governance majority, no threshold corruption, and no external dependency is required. Any unprivileged user can execute this. The ongoing cost is zero once the canister is created and drained.

## Recommendation

1. **Require a minimum balance check before inducting deferred-fee messages.** Before classifying an `UpdateSettings` message as `Free`, verify that the target canister holds at least the induction fee (or a small fixed deposit). This preserves the unfreeze use case while preventing zero-cost flooding.
2. **Apply a per-canister rate limit on `UpdateSettings` ingress messages**, independent of the fee mechanism, to bound worst-case throughput from a single canister or a small set of attacker-controlled canisters.
3. **Do not silently discard the fee collection error.** At minimum, increment a metric when `consume_cycles` fails for the deferred `UpdateSettings` fee so operators can detect and respond to abuse.

## Proof of Concept

```
1. Create canister C on an application subnet with minimal cycles.
2. Drain C's cycles balance to zero.
3. In a loop, submit ingress messages:
     destination: IC_00 (management canister)
     method:      update_settings
     payload:     UpdateSettingsArgs { canister_id: C,
                    settings: { freezing_threshold: k } }
                  (encoded size ≤ 338 bytes, confirmed by the test at
                   rs/cycles_account_manager/src/cycles_account_manager/tests.rs L22–37)
4. Observe:
   - ingress_filter accepts each message (IngressInductionCost::Free, no balance check).
   - ingress_selector accepts each message (Free arm, no balance check).
   - valid_set_rule enqueues each message directly.
   - execution_environment executes each message and silently drops the fee error.
   - C's balance remains zero throughout.
   - Legitimate canister calls on the subnet experience increased latency or starvation.
5. Scale by creating N additional zero-balance canisters to bypass the per-canister
   round-robin quota in the ingress selector and fill more slots per block.
