### Title
`Router::send_refund` Is Permanently Inaccessible — Aurora Engine Exposes No Mechanism to Invoke It (`etc/xcc-router/src/lib.rs`)

---

### Summary

The `send_refund` function in the XCC Router contract is gated so that only the Aurora Engine (the "parent") can call it. Its sole purpose is to transfer the 2 NEAR storage deposit back to the engine when a router is decommissioned. However, the Aurora Engine exposes no entry point — no wrapper, no callback, no promise chain — that ever calls `send_refund` on any router sub-account. The result is that 2 NEAR per XCC router sub-account is permanently frozen with no recovery path.

---

### Finding Description

Every EVM address that uses the XCC precompile gets a dedicated router sub-account (`{address}.aurora`) created by the engine. At creation time, the engine deposits `STORAGE_AMOUNT` (2 NEAR) to fund the router's storage staking. [1](#0-0) 

The XCC Router exposes `send_refund` as the only mechanism to return this 2 NEAR to the parent engine: [2](#0-1) 

`send_refund` is gated by `require_caller(&parent)`, meaning only the Aurora Engine account (`aurora`) can call it: [3](#0-2) 

Now examine every engine-side XCC method. The engine creates promises targeting only `execute`, `schedule`, `deploy_upgrade`, and `factory_update_address_version` on router contracts: [4](#0-3) 

The engine's exposed XCC contract methods are `withdraw_wnear_to_router`, `factory_update`, `factory_update_address_version`, `factory_set_wnear_address`, `factory_get_wnear_address`, and `fund_xcc_sub_account`: [5](#0-4) 

None of these methods create a promise that calls `send_refund` on any router sub-account. The Aurora Engine has no `#[no_mangle]` entry point in `lib.rs` that triggers `send_refund`: [6](#0-5) 

This is structurally identical to the reported M-09 pattern: a function (`send_refund`) is restricted to the owner/parent contract (Aurora Engine), but the owner/parent contract has no wrapper to invoke it, making it permanently unreachable.

---

### Impact Explanation

**Permanent freezing of funds.** The 2 NEAR deposited into each XCC router sub-account at creation time can never be reclaimed. `send_refund` is the only on-chain path to return those funds to the engine, and it is permanently inaccessible because the engine never calls it. For every EVM address that has ever used the XCC precompile, 2 NEAR is irreversibly locked in their router sub-account.

---

### Likelihood Explanation

**High.** The XCC precompile is a core Aurora feature. Any EVM contract or user that has ever triggered a cross-contract call to NEAR has a router sub-account with 2 NEAR locked. The issue is not conditional on any edge case — it is a structural omission that affects every XCC user unconditionally.

---

### Recommendation

Add a new engine-level entry point (e.g., `reclaim_xcc_storage`) that constructs a promise calling `send_refund` on a specified router sub-account. This method should be callable by the owner or key manager. Example:

```rust
// In engine/src/lib.rs
#[unsafe(no_mangle)]
pub extern "C" fn reclaim_xcc_storage() {
    // require_owner_only or require_key_manager_only
    // then promise_create_call targeting {address}.aurora::send_refund
}
```

This mirrors the recommended fix in M-09: the parent contract must expose a wrapper that forwards the call to the child contract's restricted function.

---

### Proof of Concept

1. User calls the XCC precompile from an EVM contract, triggering router creation for their address.
2. The engine deposits 2 NEAR to `{evm_address}.aurora` via a `Transfer` batch action. [7](#0-6) 
3. The router is initialized with `parent = aurora` (the engine account). [8](#0-7) 
4. `send_refund` on the router can only be called by `aurora`. [2](#0-1) 
5. The engine has no method that calls `send_refund` on any router — confirmed by exhaustive inspection of all XCC-related engine entry points. [9](#0-8) 
6. The 2 NEAR is permanently locked in the router sub-account with no recovery path.

#[named]
pub fn withdraw_wnear_to_router<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;
        if matches!(handler.promise_result_check(), Some(false)) {
            return Err(b"ERR_CALLBACK_OF_FAILED_PROMISE".into());
        }
        let args: WithdrawWnearToRouterArgs = io.read_input_borsh()?;
        let current_account_id = env.current_account_id();
        let recipient = AccountId::try_from(format!(
            "{}.{}",
            args.target.encode(),
            current_account_id.as_ref()
        ))?;
        let wnear_address = aurora_engine_precompiles::xcc::state::get_wnear_address(&io);
        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&current_account_id),
            current_account_id,
            io,
            env,
        );
        let (result, ids) = xcc::withdraw_wnear_to_router(
            &recipient,
            args.amount,
            wnear_address,
            &mut engine,
            handler,
        )?;
        if !result.status.is_ok() {
            return Err(b"ERR_WITHDRAW_FAILED".into());
        }
        let id = ids.last().ok_or(b"ERR_NO_PROMISE_CREATED")?;
        handler.promise_return(*id);
        Ok(result)
    })
}

#[named]
pub fn factory_update<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        let bytes = io.read_input().to_vec();
        let router_bytecode = xcc::RouterCode::new(bytes);
        xcc::update_router_code(&mut io, &router_bytecode);
        Ok(())
    })
}

#[named]
pub fn factory_update_address_version<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &H,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        // The function is only set to be private, otherwise callback error will happen.
        env.assert_private_call()?;
        let check_deploy: Result<(), &[u8]> = match handler.promise_result_check() {
            Some(true) => Ok(()),
            Some(false) => Err(b"ERR_ROUTER_DEPLOY_FAILED"),
            None => Err(b"ERR_ROUTER_UPDATE_NOT_CALLBACK"),
        };
        check_deploy?;
        let args: xcc::AddressVersionUpdateArgs = io.read_input_borsh()?;
        xcc::set_code_version_of_address(&mut io, &args.address, args.version);
        Ok(())
    })
}

pub fn factory_get_wnear_address<I: IO + Copy>(mut io: I) -> Result<(), ContractError> {
    let address = aurora_engine_precompiles::xcc::state::get_wnear_address(&io);
    let bytes = borsh::to_vec(&address).map_err(|_| errors::ERR_SERIALIZE)?;
    io.return_output(&bytes);
    Ok(())
}

#[named]
pub fn fund_xcc_sub_account<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;

        let args: xcc::FundXccArgs = io.read_input_borsh()?;

        // If a specific wNEAR account is specified then this transaction must
        // come from a trusted user. The wNEAR account must be accurate for the
        // XCC sub-account to work properly.
        // This method can be public when `args.wnear_account_id.is_none()`
        // because then the Engine figures out the correct wNEAR account on
        // its own.
        if args.wnear_account_id.is_some() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        xcc::fund_xcc_sub_account(&io, handler, env, args)?;
        Ok(())
    })
}
