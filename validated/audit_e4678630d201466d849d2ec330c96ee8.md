### Title
ERC-20 Tokens Permanently Burned With No Refund When `ft_transfer` Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

### Summary

When the `error_refund` compile-time feature is disabled, a user who calls the `ExitToNear` precompile to bridge ERC-20 tokens back to NEAR will have their EVM-side tokens permanently burned if the downstream `ft_transfer` promise fails (e.g., because the receiver account is not registered with the NEP-141 contract). No callback is scheduled, so no re-mint occurs. The user loses both their ERC-20 balance and their NEP-141 tokens.

---

### Finding Description

**Step 1 — Compile-time hardcoding of `refund: None`**

In `native.rs`, the `ExitToNear::run()` method builds the callback args struct:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None without the feature
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

**Step 2 — No callback is attached for the standard ERC-20 exit path**

Immediately after, the code decides whether to attach a callback at all:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

`ExitToNearPrecompileCallbackArgs` derives `Default`, so its default is `{ refund: None, transfer_near: None }`. [3](#0-2) 

For the standard ERC-20 exit (flag `0x1`, non-wNEAR), `exit_erc20_token_to_near` returns `transfer_near_args = None`: [4](#0-3) 

Therefore `callback_args == default()` is `true`, and **no callback is ever scheduled**. The promise chain is just `PromiseArgs::Create(ft_transfer_promise)` with no error handler.

**Step 3 — Even when a callback exists, `refund: None` causes silent no-op**

For the wNEAR unwrap path, `transfer_near_args` is `Some(...)`, so a callback IS attached. But in `exit_to_near_precompile_callback`:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← falls through here when refund is None
};
``` [5](#0-4) 

With `refund: None`, the callback returns `Ok(None)` without calling `refund_on_error`, so no ERC-20 re-mint occurs.

**Step 4 — `refund_on_error` is the only recovery path**

`refund_on_error` is the sole mechanism that re-mints burned ERC-20 tokens:

```rust
pub fn refund_on_error(...) -> EngineResult<SubmitResult> {
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        ...
        engine.call(&erc20_admin_address, &erc20_address, Wei::zero(), input, ...)
    }
}
``` [6](#0-5) 

Without `error_refund`, this function is never reached.

**Step 5 — `error_refund` is not a default feature** [7](#0-6) 

The `default` feature set is `["std"]`. `error_refund` must be explicitly opted into. Any deployment that omits it is vulnerable.

**Step 6 — The test suite explicitly confirms the broken invariant**

The integration test `test_exit_to_near_refund` directly documents the behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [8](#0-7) 

The test asserts that without `error_refund`, the ERC-20 balance is reduced by `FT_EXIT_AMOUNT` with no recovery — confirming the burn is permanent.

---

### Impact Explanation

- ERC-20 tokens are burned in the EVM before the precompile returns (the burn is part of the ERC-20 `withdrawToNear` call that triggers the precompile).
- The NEP-141 `ft_transfer` fails on the NEAR side (e.g., unregistered receiver), so the NEP-141 tokens remain on the Aurora connector account.
- No callback is scheduled (or the callback is a no-op), so no re-mint occurs.
- The user ends up with zero ERC-20 tokens and zero NEP-141 tokens. The NEP-141 tokens are stranded on the connector with no automatic recovery path.
- This matches **High — Temporary freezing of funds** (the NEP-141 tokens are not destroyed, but the user cannot access them without admin intervention, which may never come).

---

### Likelihood Explanation

- Any user who attempts to exit ERC-20 tokens to an unregistered NEAR account triggers this path.
- Unregistered accounts are a common user error (e.g., a new NEAR account that has never interacted with the NEP-141 contract).
- The `error_refund` feature is not in the default feature set, so any deployment that does not explicitly enable it is affected.
- The path is reachable via a standard EVM transaction calling the ERC-20 `withdrawToNear` function — no special privileges required.

---

### Recommendation

1. Add `error_refund` to the `default` features in `engine/Cargo.toml` and `engine-precompiles/Cargo.toml`, or remove the feature gate entirely and always populate `refund`.
2. Alternatively, validate at the precompile level that the receiver account is registered before burning tokens (though this is difficult cross-contract in a single EVM call).
3. At minimum, document clearly in the deployment checklist that `error_refund` must be enabled for production builds.

---

### Proof of Concept

1. Compile Aurora Engine **without** `--features error_refund`.
2. Deploy a NEP-141 token; bridge it to an ERC-20 on Aurora.
3. Call the ERC-20's `withdrawToNear` targeting an account that is **not registered** with the NEP-141 contract.
4. Observe: ERC-20 balance decreases by the exit amount; NEP-141 balance of the target account remains 0; NEP-141 balance of the Aurora connector account is unchanged (tokens stranded); no re-mint callback fires.
5. Compare with a build that **includes** `error_refund`: the ERC-20 balance is restored to the sender.

This is exactly what `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` demonstrates at lines 623–665. [9](#0-8)
