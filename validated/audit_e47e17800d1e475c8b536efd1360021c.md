Audit Report

## Title
Unbounded Ingress Window in `construction_payloads` Enables Memory/CPU Exhaustion of Rosetta Node — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The `/construction/payloads` endpoint accepts attacker-controlled `ingress_start` and `ingress_end` metadata fields with no validation of the window size. The handler unconditionally builds a `Vec` of expiry slots spanning the entire window, then allocates two `SigningPayload` objects per slot per operation. A single unauthenticated HTTP POST with `ingress_end = u64::MAX` causes the Rosetta node to loop ~1.5×10¹¹ times, exhausting all available memory and CPU until the OS kills the process.

## Finding Description
`ingress_start` and `ingress_end` are read directly from the request metadata with no bounds check on the window size: [1](#0-0) 

The `interval` is fixed at exactly 120 seconds (`MAX_INGRESS_TTL` 300 s − `PERMITTED_DRIFT` 60 s − 120 s): [2](#0-1) [3](#0-2) 

The while loop at lines 99–107 steps by `interval` with no cap on iterations: [4](#0-3) 

For every transaction, `add_payloads` pushes **2 `SigningPayload` objects** per entry in `ingress_expiries`: [5](#0-4) 

The only server-side guard is a 4 MB JSON body limit, which constrains the number of operations (N) but does nothing to bound the window size, since `ingress_start`/`ingress_end` are two `u64` fields occupying ~40 bytes: [6](#0-5) 

The identical unbounded loop exists in the ICRC-1 Rosetta implementation: [7](#0-6) 

## Impact Explanation
This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

With `ingress_end = u64::MAX` and `interval = 120 s` (1.2×10¹¹ ns), the loop executes ≈1.5×10¹¹ iterations — the process never returns and RSS climbs until OOM kill. Even a bounded 1-year window (86 400 s / 120 s = 720 slots × 2 payloads × N operations) allocates hundreds of MB per request and is trivially parallelisable to sustain a denial of service against any deployed ICP Rosetta node.

## Likelihood Explanation
The `/construction/payloads` endpoint is unauthenticated and publicly reachable on any deployed ICP Rosetta node. The exploit requires a single HTTP POST with a crafted JSON body under 1 KB. No credentials, prior state, or privileged access are needed. The attack is repeatable and trivially scriptable.

## Recommendation
1. **Cap the ingress window** before the loop: reject requests where `ingress_end − ingress_start` exceeds `MAX_INGRESS_TTL` (or a small configured multiple, e.g. 24 h).
2. **Cap the `ingress_expiries` vector length** with an explicit early return (e.g. `> 720` slots → `ApiError::invalid_request`).
3. Apply the same fix to the ICRC-1 Rosetta `construction_payloads` in `rs/rosetta-api/icrc1/src/construction_api/services.rs`, which has an identical unbounded loop.
4. Consider capping the operations count in `operations_to_requests` independently of the body-size limit.

## Proof of Concept
```http
POST /construction/payloads HTTP/1.1
Host: <rosetta-node>:8080
Content-Type: application/json

{
  "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
  "operations": [
    {"operation_identifier":{"index":0},"type":"STAKE",
     "account":{"address":"<valid-account>"},
     "metadata":{"neuron_index":0}}
  ],
  "public_keys": [{"hex_bytes":"<valid-pubkey>","curve_type":"edwards25519"}],
  "metadata": {
    "ingress_start": 0,
    "ingress_end": 18446744073709551615
  }
}
```
The server enters the `while now < ingress_end` loop and never exits. Peak RSS climbs until OOM. A bounded variant (`ingress_end − ingress_start = 86400 × 1e9` ns, i.e. 1 day) allocates ~150 MB per request and is trivially parallelisable to sustain the DoS.
