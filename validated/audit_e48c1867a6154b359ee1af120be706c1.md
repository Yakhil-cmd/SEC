Audit Report

## Title
Sequential Per-Input `sign_with_ecdsa` Calls Stall ckBTC Withdrawal Pipeline — (File: `rs/bitcoin/ckbtc/minter/src/tx.rs`)

## Summary
`BitcoinTransactionSigner::sign_transaction` issues one `sign_with_ecdsa` management-canister call per transaction input inside a sequential `for` loop, with no parallelism. Because a global `TimerLogicGuard` prevents any concurrent timer execution, the entire withdrawal pipeline is serialized behind this signing loop. An attacker who floods the minter with many small UTXO deposits can force a transaction requiring up to `DEFAULT_MAX_NUM_INPUTS_IN_TRANSACTION` (1,000) sequential threshold-ECDSA round-trips, stalling all other pending withdrawals for the duration.

## Finding Description
In `rs/bitcoin/ckbtc/minter/src/tx.rs` at lines 688–711, `BitcoinTransactionSigner::sign_transaction` iterates over every input and `await`s a separate `sign_with_ecdsa` call for each one:

```rust
for (input, account) in unsigned_tx.inputs.iter().zip(accounts) {
    // ...
    let sec1_signature =
        management::sign_with_ecdsa(self.key_name.clone(), path, sighash, runtime).await?;
    // ...
}
``` [1](#0-0) 

There is no `join_all` or equivalent — calls are strictly sequential. The total signing latency scales linearly with the number of inputs.

The `TimerLogicGuard` in `rs/bitcoin/ckbtc/minter/src/guard.rs` enforces that only one timer task runs at a time:

```rust
pub fn new() -> Option<Self> {
    mutate_state(|s| {
        if s.is_timer_running { return None; }
        s.is_timer_running = true;
        Some(TimerLogicGuard(()))
    })
}
``` [2](#0-1) 

This guard is acquired at the top of `run_task` before calling `submit_pending_requests`, meaning no other `ProcessLogic` task (which handles all withdrawals) can run while signing is in progress. [3](#0-2) 

`submit_pending_requests` builds exactly one batch per invocation and calls `sign_and_submit_request` for it: [4](#0-3) 

The default cap is `DEFAULT_MAX_NUM_INPUTS_IN_TRANSACTION = 1_000`: [5](#0-4) 

**Why existing checks are insufficient:** `max_num_inputs_in_transaction` bounds the worst case per transaction but does not prevent the stall — it only caps how long the stall lasts. The `MIN_PENDING_REQUESTS = 20` threshold is easily satisfied by normal queue accumulation or by the attacker's own requests. [6](#0-5) 

## Impact Explanation
This is an **application/platform-level DoS** on the ckBTC withdrawal pipeline — a High ($2,000–$10,000) impact. While signing up to 1,000 sequential threshold-ECDSA calls (each requiring subnet-wide coordination), every other pending withdrawal request is blocked. Legitimate users experience unbounded delays proportional to the number of inputs the attacker forced into the transaction. The attack is repeatable: after one transaction is signed and submitted, the attacker can trigger another batch with more small UTXOs, sustaining the stall indefinitely. This directly harms users of a production in-scope financial integration (ckBTC).

## Likelihood Explanation
The attack entry point is the public `retrieve_btc` / `retrieve_btc_with_approval` endpoint, callable by any unprivileged principal. The attacker's cost is the BTC required to create many small UTXOs above `deposit_btc_min_amount`. No privileged access, key material, or subnet-majority corruption is required. The `MIN_PENDING_REQUESTS = 20` condition is easily met through normal queue accumulation. The attack is repeatable and low-complexity once the UTXO pool is seeded.

## Recommendation
Replace the sequential `for` loop in `BitcoinTransactionSigner::sign_transaction` with parallel signing using `futures::future::join_all` (or `try_join_all`), issuing all `sign_with_ecdsa` calls concurrently and collecting results. This mirrors the approach already used in the ckETH minter's `sign_transactions_batch` in `rs/ethereum/cketh/minter/src/withdraw.rs`, which uses `join_all` to sign multiple transactions in parallel. Additionally, enforce a tighter, well-reasoned bound on `max_num_inputs_in_transaction` to limit worst-case signing latency even before parallelism is deployed.

## Proof of Concept
1. Attacker calls `retrieve_btc_with_approval` many times with the minimum deposit amount, seeding the minter's available UTXO pool with N small UTXOs (up to `DEFAULT_MAX_NUM_INPUTS_IN_TRANSACTION = 1_000`).
2. Any user (or the attacker) submits a withdrawal large enough that the UTXO selection algorithm must consume many of those small UTXOs as inputs.
3. The minter's `ProcessLogic` timer fires, acquires `TimerLogicGuard`, calls `submit_pending_requests`, builds the batch, and calls `sign_and_submit_request`.
4. `BitcoinTransactionSigner::sign_transaction` enters the sequential `for` loop and issues up to 1,000 sequential `sign_with_ecdsa` calls, each awaiting a full threshold-ECDSA round-trip.
5. All other withdrawal requests queued by legitimate users remain unprocessed until all N signing calls complete, because `TimerLogicGuard` blocks any concurrent `ProcessLogic` execution.
6. Attacker repeats from step 1 to sustain the stall across multiple timer cycles.

A deterministic integration test using PocketIC can reproduce this by: initializing the minter with many small UTXOs in `available_utxos`, submitting a withdrawal that selects all of them, advancing the timer, and asserting that no other withdrawal advances until the signing loop completes.
