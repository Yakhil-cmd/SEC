### Title
Unprotected `new()` Initialization Allows Front-Running by Any Unprivileged Caller — (File: `engine/src/contract_methods/admin.rs`)

---

### Summary

The Aurora Engine's `new()` initialization function contains no caller access control. Any unprivileged NEAR account can call it before the legitimate deployer does, setting themselves as `owner_id` and taking full administrative control of the engine.

---

### Finding Description

The `new()` function in `engine/src/contract_methods/admin.rs` is the sole initializer for the `EngineState`, which stores `owner_id`, `chain_id`, `upgrade_delay_blocks`, and `key_manager`. It guards against *re-initialization* by checking whether state already exists: [1](#0-0) 

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
```

However, there is **no check on `env.predecessor_account_id()`** — no assertion that the caller is the contract account itself, the deployer, or any trusted party. The function accepts arbitrary `NewCallArgs` from any caller and writes them directly into state: [2](#0-1) 

The `owner_id` field written into `EngineState` is taken verbatim from the attacker-supplied input: [3](#0-2) 

This is exposed as a public WASM export with no additional guard: [4](#0-3) 

All other state-mutating functions (`set_owner`, `upgrade`, `pause_contract`, `factory_update`, etc.) correctly call `state::get_state(&io)?` first, which fails with `ERR_STATE_NOT_FOUND` if uninitialized: [5](#0-4) 

So the only write path reachable before initialization is `new()` itself — and it is open to any caller.

---

### Impact Explanation

If an attacker front-runs `new()` before the legitimate deployer:

1. **Attacker becomes `owner_id`** of the Aurora Engine.
2. As owner, they can call `upgrade()` to deploy arbitrary contract code — enabling direct theft of all bridged ETH and ERC-20 balances held in the engine's EVM state.
3. They can call `pause_contract()` to permanently freeze all user funds.
4. They can call `factory_update()` to poison XCC router bytecode, redirecting cross-contract call funds.
5. The legitimate deployer's subsequent call to `new()` fails with `ERR_ALREADY_INITIALIZED`, and the attacker's state persists.

Impact classification: **Critical** — direct theft of user funds and/or permanent fund freeze.

---

### Likelihood Explanation

NEAR Protocol supports atomic batch transactions combining `DeployContract` + `FunctionCall` actions, which is the standard mitigation. However:

- The production contract code itself contains **zero enforcement** of atomic deployment.
- Any deployment process that separates the deploy step from the `new()` call (e.g., a two-step script, a failed batch, a deployment pipeline bug, or a future re-deployment after an upgrade) opens the window.
- An attacker monitoring the NEAR chain for a newly deployed Aurora Engine contract (account ID `aurora`) can immediately call `new()` with attacker-controlled arguments in the same block.

Likelihood: **Low-to-Medium** — depends entirely on deployment discipline, with no on-chain enforcement.

---

### Recommendation

Add a caller check inside `new()` to restrict initialization to the contract account itself (self-call) or a pre-committed deployer address:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // Ensure only the contract account itself can initialize
    if env.predecessor_account_id() != env.current_account_id() {
        return Err(b"ERR_NOT_ALLOWED".into());
    }
    ...
}
```

Alternatively, enforce atomic deployment + initialization at the protocol level by documenting and enforcing a single-batch deploy pattern, and add an integration test that verifies `new()` reverts when called by a non-self account.

---

### Proof of Concept

1. Attacker watches the NEAR chain for a `DeployContract` action targeting the `aurora` account.
2. In the same or next block (before the deployer's `new()` call), attacker submits a transaction calling `new()` on `aurora` with `owner_id = attacker.near`, arbitrary `chain_id`, and `upgrade_delay_blocks = 0`.
3. `state::get_state(&io)` returns `Err(NotFound)`, so the guard passes.
4. `state::set_state` writes attacker-controlled `EngineState` to storage.
5. Deployer's `new()` call returns `ERR_ALREADY_INITIALIZED`.
6. Attacker calls `upgrade()` (as owner) to deploy a malicious contract, draining all EVM balances.

pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())
}
