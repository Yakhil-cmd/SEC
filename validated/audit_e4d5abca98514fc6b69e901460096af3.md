### Title
EIP-7702 Failed Authorization Entries Overcharged `NEWACCOUNT` Instead of `PER_AUTH_BASE_COST` — EVM Semantic Mismatch - (File: `basic_bootloader/src/bootloader/transaction/authorization_list.rs`)

---

### Summary

In `validate_and_apply_delegation`, every authorization entry is pre-charged `NEWACCOUNT` (25,000 gas) unconditionally at step 0, but the compensating refund of `NEWACCOUNT − PER_AUTH_BASE_COST` (12,500 gas) is only issued at step 7 when the delegation **succeeds and the authority account is non-empty**. Authorization entries that fail any of the intermediate checks (chain_id mismatch, nonce overflow, bad signature, authority is a contract, nonce mismatch) return `Ok(false)` without issuing any refund. This means each failed authorization entry costs 25,000 gas instead of the 12,500 gas required by EIP-7702, a 12,500-gas overcharge per failed entry.

---

### Finding Description

`validate_and_apply_delegation` in `authorization_list.rs` implements the EIP-7702 authorization processing loop. The function's own docstring labels step 0 as "Pre-charge for intrinsic gas cost of delegation (PER_AUTH_BASE_COST)", but the actual charge is `NEWACCOUNT * ERGS_PER_GAS` (25,000 gas), not `PER_AUTH_BASE_COST` (12,500 gas): [1](#0-0) 

The refund that brings the net cost down to `PER_AUTH_BASE_COST` is only issued at step 7, and only when the authority account is non-empty: [2](#0-1) 

All early-exit paths — chain_id mismatch (step 1), nonce overflow (step 2), high-s signature (step 3), ecrecover failure (step 3), authority is a contract (step 5), nonce mismatch (step 6) — return `Ok(false)` without issuing any refund: [3](#0-2) 

The ZK transaction path pre-charges `NEWACCOUNT` per authorization entry in `calculate_tx_intrinsic_gas`: [4](#0-3) 

The Ethereum transaction path charges directly from `tx_resources.main_resources` (not infinite ergs) inside `validate_and_apply_delegation`: [5](#0-4) 

In both paths, the net gas cost for a **failed** authorization entry is `NEWACCOUNT` = 25,000 gas. EIP-7702 specifies the intrinsic cost as `PER_AUTH_BASE_COST` = 12,500 gas per entry regardless of success or failure: [6](#0-5) [7](#0-6) 

**Gas accounting per entry (ZKsync OS vs. EIP-7702 spec):**

| Outcome | ZKsync OS | EIP-7702 spec | Delta |
|---|---|---|---|
| Failed auth (any check) | 25,000 | 12,500 | +12,500 overcharge |
| Succeeded, empty authority | 25,000 | 25,000 | correct |
| Succeeded, non-empty authority | 12,500 | 12,500 | correct |

---

### Impact Explanation

1. **EVM semantic mismatch / resource accounting bug**: ZKsync OS claims EVM equivalence but charges 2× the specified intrinsic cost for every failed EIP-7702 authorization entry. A transaction with N failed entries is overcharged by N × 12,500 gas.

2. **Transaction failure when Ethereum mainnet would succeed**: A user who sizes their gas limit to the EIP-7702 spec cost will have their transaction run out of gas on ZKsync OS if any authorization entries fail. This is a correctness divergence between ZKsync OS and Ethereum mainnet.

3. **Forward/proving divergence risk**: The overcharge is applied identically in both the ZK path (via `calculate_tx_intrinsic_gas` + `with_infinite_ergs`) and the Ethereum path (direct charge), so there is no sequencer/prover split here — but the divergence from the EIP-7702 spec means any cross-client comparison (e.g., against revm) will produce different gas-used values, which can surface as a state-transition mismatch.

---

### Likelihood Explanation

Any unprivileged transaction sender can submit an EIP-7702 transaction (type `0x04`) with authorization entries that will fail. Realistic failure scenarios include:

- **Nonce mismatch**: the authority's nonce was incremented by a prior transaction between the time the user signed the authorization and the time the block is processed (step 6 check).
- **Authority is a contract**: the authority address has deployed code (step 5 check).
- **Cross-chain authorization with chain_id = 0**: the entry is valid on any chain but the authority's nonce has already advanced.

These are not adversarial inputs — they are normal operational conditions. The overcharge is deterministic and reproducible.

---

### Recommendation

Issue the `NEWACCOUNT − PER_AUTH_BASE_COST` refund for **all** authorization entries that pass the chain_id check (step 1), not only for non-empty successful ones. Alternatively, restructure the pre-charge to use `PER_AUTH_BASE_COST` at step 0 and charge the additional `NEWACCOUNT − PER_AUTH_BASE_COST` only when the authority account is confirmed empty at step 7. Either approach aligns with the EIP-7702 specification.

---

### Proof of Concept

1. Deploy an EIP-7702 transaction with two authorization entries:
   - Entry A: valid chain_id, correct nonce for authority `addr_A` (will succeed, authority non-empty → net 12,500 gas).
   - Entry B: valid chain_id, but `nonce = current_nonce_of_addr_B + 1` (will fail at step 6 → net 25,000 gas instead of 12,500).

2. Set `gas_limit = TX_INTRINSIC_GAS + 2 * PER_AUTH_BASE_COST + execution_gas` (the correct EIP-7702 budget).

3. On Ethereum mainnet (or revm), the transaction succeeds.

4. On ZKsync OS, the transaction runs out of gas during validation because entry B consumes 25,000 gas instead of 12,500, exhausting the budget before execution begins.
