Audit Report

## Title
Unbounded SCM_RIGHTS FD Accumulation in `socket_read_messages` Allows Compromised Sandbox to Exhaust Replica FD Table — (`rs/canister_sandbox/src/transport.rs`)

## Summary

In `socket_read_messages`, the `fds: Vec<RawFd>` accumulator is only drained by `install_file_descriptors` when `decoder.decode` returns a complete frame. A compromised sandbox can send an unbounded stream of partial frames each carrying ancillary SCM_RIGHTS file descriptors; because no complete frame is ever assembled, `fds` grows without bound and every entry is a live, open file descriptor in the replica process. Exhausting the replica's FD table causes it to fail to open new sockets, files, or pipes, breaking consensus participation and subnet availability.

## Finding Description

In `socket_read_messages` (lines 434–484), the outer loop calls `reader.receive_message(&mut buf, &mut fds, ...)` unconditionally on every iteration. Inside `receive_message` → `receive_message` (lines 658–686), every `SCM_RIGHTS` control message received via `recvmsg` is parsed and each FD is appended to `fds` via `fds.push(fd)` (line 680) with no upper-bound check.

`install_file_descriptors` (lines 488–506) is the only drain point for `fds`, and it is only reached when `decoder.decode(&mut buf)` returns `Some(frame)` (line 447). If the sender never completes a frame — by sending only partial payload bytes — `decoder.decode` never returns `Some`, the inner `while` loop body never executes, and `fds` is never drained.

The sender-side constant `MAX_NUM_FD_PER_MESSAGE = 16` (line 18) is enforced only in `send_message` (line 546: `let fds_to_send = std::cmp::min(MAX_NUM_FD_PER_MESSAGE, fds.len())`). The receiver imposes no equivalent cap. The receiver's `CONTROL_MESSAGE_SIZE = 4096` (line 21) allows up to ~1000 FDs per single `recvmsg` call (4096 bytes / 4 bytes per FD minus CMSG header). There is no check anywhere in the receive path that limits total accumulated FDs.

Additionally, `install_file_descriptors` itself has a secondary leak: when `fd_locs.len() < fds.len()` (line 501), it drains only the consumed prefix and leaves the remainder in `fds` — those leftover FDs are real open descriptors in the replica that are never closed until the next frame consumes them.

## Impact Explanation

Each FD received via `SCM_RIGHTS` is a real, open file descriptor in the **replica (controller) process**. Accumulating thousands of them exhausts the per-process FD limit. Once exhausted, the replica cannot accept new connections, open state files, or communicate with other subsystems — causing an application/platform-level DoS and subnet availability impact. This matches the allowed High impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

The precondition is a compromised canister sandbox process. The IC's security architecture is explicitly defense-in-depth: the sandbox is isolated via seccomp, Linux namespaces, and SELinux (`ic_canister_sandbox_t` domain) precisely because canister Wasm execution is untrusted. The SELinux policy documentation explicitly acknowledges that a compromised sandbox can affect the replica through the permitted IPC channel, and even documents a known analogous issue (ftruncate-induced SIGBUS). A canister exploiting a memory-safety bug in the Wasm runtime or sandbox binary gains code execution in the sandbox and can immediately trigger this path. The exploit is trivial once the sandbox is controlled: send a stream of 1-byte partial frames each with the maximum number of SCM_RIGHTS FDs. No complete frame is ever assembled; `fds` grows without bound.

## Recommendation

1. **Enforce a hard cap on `fds.len()`** inside the `socket_read_messages` loop immediately after `receive_message` returns. If `fds.len()` exceeds a reasonable maximum (e.g., `MAX_NUM_FD_PER_MESSAGE * 2`), close all excess FDs with `libc::close()` and terminate the connection.
2. **Close unconsumed FDs** in `install_file_descriptors` when `fd_locs.len() < fds.len()`: the current code at lines 501–502 drains the used prefix but leaves the remainder open. The remainder should be explicitly closed before draining.
3. **Add a maximum accumulated-bytes limit** on `buf` to prevent a parallel memory-exhaustion attack via partial frames.

## Proof of Concept

```rust
// Pseudocode for compromised sandbox process
// Each sendmsg sends 1 byte of payload (partial frame, never completes)
// with up to ~1000 SCM_RIGHTS FDs (bounded only by CONTROL_MESSAGE_SIZE=4096)
loop {
    let devnull_fds: Vec<RawFd> = (0..16).map(|_| open("/dev/null")).collect();
    sendmsg(replica_socket,
        iov = [b'\x00'],                    // 1 byte: never completes a frame
        cmsg = SCM_RIGHTS(devnull_fds));    // 16 FDs appended to replica's fds vec
    // After ~64 iterations: 1024 FDs accumulated, replica FD table exhausted
    // replica.open(), replica.accept(), replica.connect() all return EMFILE
}
```

A deterministic integration test can be written using a `UnixStream::pair()`, a writer thread that sends only partial frame bytes with SCM_RIGHTS FDs in a loop, and an assertion that `socket_read_messages` does not accumulate more than `MAX_NUM_FD_PER_MESSAGE * 2` open FDs in the replica-side process (verified via `/proc/self/fd` count).
