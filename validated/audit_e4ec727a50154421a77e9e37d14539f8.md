### Title
Unsigned `max_gas_price` in `SubmitArgs` Allows Relayer to Drain User ETH via Gas Overcharge - (`engine-types/src/parameters/engine.rs`, `engine/src/engine.rs`)

---

### Summary

`submit_with_args` accepts a `SubmitArgs` struct containing a signed Ethereum transaction (`tx_data`) plus two unsigned fields: `max_gas_price` and `gas_token_address`. Because neither field is covered by the Ethereum ECDSA signature in `tx_data`, a malicious relayer or front-running attacker can resubmit the user's signed transaction with `max_gas_price` stripped or altered, forcing the user to pay the full gas price encoded in the signed transaction rather than the lower cap they intended.

---

### Finding Description

`SubmitArgs` is defined as: [1](#0-0) 

```rust
pub struct SubmitArgs {
    pub tx_data: Vec<u8>,
    pub max_gas_price: Option<u128>,
    pub gas_token_address: Option<Address>,
}
```

`tx_data` is a standard RLP-encoded, ECDSA-signed Ethereum transaction. The signature inside `tx_data` covers only the Ethereum transaction fields (`nonce`, `gas_price`/`max_fee_per_gas`, `gas_limit`, `to`, `value`, `data`, `chain_id`, etc.). It does **not** cover `max_gas_price` or `gas_token_address`.

In `submit_with_alt_modexp`, `max_gas_price` is extracted directly from `args` and passed to `charge_gas`: [2](#0-1) 

Inside `charge_gas`, `max_gas_price` acts as a downward cap on `priority_fee_per_gas`: [3](#0-2) 

```rust
let priority_fee_per_gas = max_gas_price.map_or(priority_fee_per_gas, |price| {
    price.min(priority_fee_per_gas)
});
let effective_gas_price = priority_fee_per_gas + block_base_fee_per_gas;
```

The `prepaid_amount` is then deducted from the user's ETH balance: [4](#0-3) 

After execution, the relayer (the NEAR `predecessor_account_id` of whoever calls `submit_with_args`) receives `priority_fee_per_gas * gas_used` as a reward: [5](#0-4) 

Because `max_gas_price` is not part of the Ethereum signature, any party who possesses the signed Ethereum transaction bytes can call `submit_with_args` with `max_gas_price: None`, bypassing the user's intended cap and collecting the full priority fee.

---

### Impact Explanation

A user who signs an Ethereum transaction with `max_fee_per_gas = 100` but intends to submit it via `SubmitArgs { max_gas_price: Some(5), ... }` expects to pay at most `gas_used × 5` wei. A malicious relayer submitting with `max_gas_price: None` causes the user to pay `gas_used × 100` wei instead. The difference — `gas_used × 95` wei — is credited to the attacker's NEAR-derived EVM address as relayer reward. This is a direct, quantifiable theft of user ETH funds from their Aurora EVM balance.

**Impact class: Critical — direct theft of user funds.**

---

### Likelihood Explanation

The attack requires only that:
1. The attacker observes the user's signed Ethereum transaction bytes (available in any NEAR mempool or off-chain relay channel).
2. The attacker calls `submit_with_args` before the legitimate relayer, or the attacker IS the relayer.

`submit_with_args` is a public NEAR contract method with no access control. NEAR has a public mempool. Any party who sees the `SubmitArgs` payload (or just the `tx_data` bytes) can reconstruct and resubmit with altered `max_gas_price`. No privileged keys or special permissions are required.

---

### Recommendation

Include `max_gas_price` and `gas_token_address` in the data that the user signs. One approach is to define a typed structured message (analogous to EIP-712) that commits to all fields of `SubmitArgs`, and require the Ethereum transaction's `data` field or a separate signature to cover these parameters. Alternatively, encode `max_gas_price` inside the Ethereum transaction itself (e.g., as part of `data` or as a dedicated field in a new transaction type) so it is protected by the existing ECDSA signature.

---

### Proof of Concept

```
1. Alice signs an Ethereum tx: gas_price=100, gas_limit=100_000, nonce=0, to=Bob, value=1 ETH
   → tx_bytes = rlp_encode(signed_tx)

2. Alice sends SubmitArgs { tx_data: tx_bytes, max_gas_price: Some(5), gas_token_address: None }
   to a relayer, expecting to pay at most 100_000 × 5 = 500_000 wei in gas.

3. Attacker (or malicious relayer) observes tx_bytes in the NEAR mempool.

4. Attacker calls submit_with_args on Aurora with:
   SubmitArgs { tx_data: tx_bytes, max_gas_price: None, gas_token_address: None }
   using their own NEAR account as predecessor.

5. charge_gas computes effective_gas_price = 100 (uncapped).
   Alice's balance is debited 100_000 × 100 = 10_000_000 wei.
   Attacker's EVM address receives 100_000 × 100 = 10_000_000 wei as relayer reward.
   (Alice intended to pay only 500_000 wei; 9_500_000 wei is stolen.)

6. Alice's nonce is consumed; she cannot replay her original SubmitArgs.
```
