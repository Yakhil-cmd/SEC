### Title
Charge-Before-Validate Ordering in `submit_with_alt_modexp` Permanently Freezes Prepaid ETH When `gas_limit > u64::MAX` — (`engine/src/engine.rs`)

---

### Summary

In `submit_with_alt_modexp`, `charge_gas` deducts `gas_limit * gas_price` from the sender's balance **before** the `gas_limit.try_into::<u64>()` conversion is attempted. When a transaction carries a `gas_limit` that is a valid `U256` value exceeding `u64::MAX`, the conversion fails and the function returns early via `?`, bypassing `refund_unused_gas` entirely. The prepaid ETH is permanently lost with no recovery path.

---

### Finding Description

`Transaction2930.gas_limit` is typed as `U256` and is decoded from RLP with no upper-bound check: [1](#0-0) [2](#0-1) 

The value flows unchanged into `NormalizedEthTransaction.gas_limit` (also `U256`): [3](#0-2) 

Inside `submit_with_alt_modexp`, the intrinsic-gas guard compares `transaction.gas_limit` (U256) against `intrinsic_gas` (u64 widened to U256). Any value above `u64::MAX` trivially passes: [4](#0-3) 

`charge_gas` then uses `transaction.gas_limit` directly as a U256 multiplier. With `gas_limit = u64::MAX + 1` and `gas_price = 1`, the product fits in U256, `checked_mul` succeeds, and `set_balance` permanently reduces the sender's balance: [5](#0-4) [6](#0-5) 

Immediately after, the narrowing conversion is attempted: [7](#0-6) 

Because `u64::MAX + 1` does not fit in `u64`, `try_into` returns `Err`, and the `?` operator causes an early return. The function never reaches `refund_unused_gas`: [8](#0-7) 

`refund_unused_gas` is the sole mechanism for returning unspent prepayment to the sender: [9](#0-8) 

---

### Impact Explanation

The prepaid Wei amount (`(u64::MAX + 1) * gas_price`) is subtracted from the sender's balance and never returned. There is no fallback, no admin recovery function, and no on-chain mechanism to reclaim it. The ETH is permanently frozen inside the engine contract. Impact: **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

The exploit requires only a validly signed EIP-2930 transaction with `gas_limit` set to any U256 value above `u64::MAX` and a non-zero `gas_price`. The sender must hold sufficient balance to cover the prepayment. No privileged access, no oracle, no governance action is required. The transaction is accepted through the standard `submit` public entry point. Any user or automated tool that constructs such a transaction (deliberately or by bug) triggers the loss.

---

### Recommendation

Move the `gas_limit` narrowing check to **before** `charge_gas` is called, so that an oversized `gas_limit` is rejected before any balance mutation occurs:

```rust
// Validate gas_limit fits in u64 BEFORE charging
let gas_limit: u64 = transaction
    .gas_limit
    .try_into()
    .map_err(|_| EngineErrorKind::GasOverflow)?;

// Now safe to charge
let prepaid_amount = match engine.charge_gas(...) { ... };
```

This mirrors the existing pattern for `fixed_gas` overflow detection at line 1066, which also guards before any state mutation. [10](#0-9)

```rust
pub fn refund_unused_gas<I: IO>(
    io: &mut I,
    sender: &Address,
    gas_used: u64,
    gas_result: &GasPaymentResult,
    relayer: &Address,
    fixed_gas: Option<EthGas>,
) -> Result<(), GasPaymentError> {
    if gas_result.effective_gas_price.is_zero() {
        return Ok(());
    }

    let (refund, relayer_reward) = {
        let gas_to_wei = |price: U256| {
            fixed_gas
                .map_or_else(|| gas_used.into(), EthGas::as_u256)
                .checked_mul(price)
                .map(Wei::new)
                .ok_or(GasPaymentError::EthAmountOverflow)
        };

        let spent_amount = gas_to_wei(gas_result.effective_gas_price)?;
        let reward_amount = gas_to_wei(gas_result.priority_fee_per_gas)?;

        let refund = gas_result
            .prepaid_amount
            .checked_sub(spent_amount)
            .ok_or(GasPaymentError::EthAmountOverflow)?;

        (refund, reward_amount)
    };

    if !refund.is_zero() {
        add_balance(io, sender, refund)?;
    }

    if !relayer_reward.is_zero() {
        add_balance(io, relayer, relayer_reward)?;
    }

    Ok(())
}
```
