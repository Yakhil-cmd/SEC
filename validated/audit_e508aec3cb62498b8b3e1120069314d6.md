### Title
Silent Permanent ERC-20 Token Loss on Failed `ft_transfer` Promise When `error_refund` Feature Is Disabled — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

When the `error_refund` compile-time feature is not enabled, the `exit_to_near_precompile_callback` function silently returns `Ok(None)` on a failed `ft_transfer` NEAR promise, with no refund issued. Because the ERC-20 tokens were already burned during the precompile's EVM execution phase, the user's funds are permanently destroyed with no recovery path.

---

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) is the bridge exit path for ERC-20 tokens. Its execution flow is:

**Step 1 — EVM execution (synchronous):** The precompile burns the user's ERC-20 tokens and emits a log encoding a NEAR promise.

**Step 2 — Promise construction:** In `native.rs`, the `ExitToNearPrecompileCallbackArgs` struct is built. The `refund` field is conditionally set:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,   // ← always None when feature is absent
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is absent, `refund` is hardcoded to `None` regardless of the token type or amount.

**Step 3 — Async callback (NEAR promise result):** `exit_to_near_precompile_callback` in `connector.rs` is invoked after the `ft_transfer` / `ft_transfer_call` promise settles:

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success path
    None
} else if let Some(args) = args.refund {
    // refund path — only reachable when error_refund is compiled in
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent success return; tokens already burned, no refund
};
``` [2](#0-1) 

The `else { None }` branch is reached whenever:
1. The `ft_transfer` promise **failed** (any reason: unregistered recipient, paused NEP-141, insufficient storage, etc.), **AND**
2. `args.refund` is `None` — which is always the case when `error_refund` is not compiled in.

The function returns `Ok(None)` — no error, no refund, no event. The ERC-20 tokens burned in Step 1 are gone.

**The integration test explicitly confirms this behavior:**

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [3](#0-2) 

The test acknowledges that without `error_refund`, the user's ERC-20 balance is permanently reduced by `FT_EXIT_AMOUNT` even when the NEAR-side transfer fails.

---

### Impact Explanation

**Permanent freezing / destruction of funds.** Any ERC-20 token holder who calls the `ExitToNear` precompile and whose `ft_transfer` promise subsequently fails will have their tokens burned on the EVM side with no corresponding credit on NEAR and no refund to the EVM. The tokens cease to exist in any accessible form. This matches the **Critical — Permanent freezing of funds** impact category.

---

### Likelihood Explanation

`ft_transfer` on a NEP-141 contract fails in several ordinary, non-adversarial conditions:

- The recipient NEAR account is not registered with the NEP-141 contract (storage deposit not paid).
- The NEP-141 contract is paused or has transfer restrictions.
- The Aurora engine account has insufficient NEP-141 balance due to accounting drift.

Any of these conditions — which a user cannot always predict or control — triggers the silent loss path. The likelihood is **Medium**: the failure condition is not exotic, and the user has no on-chain signal that their tokens are gone (the Aurora transaction itself succeeds).

---

### Recommendation

1. **Make `error_refund` a default feature** in the production build of `aurora-engine-precompiles`, or enforce it at the engine crate level so it cannot be accidentally omitted.
2. **Add a hard error** in the `else { None }` branch of `exit_to_near_precompile_callback` when `args.refund` is `None` and the promise failed, so the callback panics rather than silently succeeding. A panic reverts the NEAR receipt and surfaces the failure to operators.
3. **Decouple refund capability from a feature flag**: the refund address should always be encoded in the callback args for ERC-20 exits, making the refund path unconditional.

---

### Proof of Concept

1. Build Aurora Engine **without** the `error_refund` feature (the compile-time default path).
2. Bridge a NEP-141 token to Aurora, obtaining an ERC-20 balance of `N` tokens.
3. Call the `ExitToNear` precompile specifying a NEAR recipient account that has **not** paid the storage deposit for the NEP-141 contract (e.g., a freshly created account).
4. The EVM transaction succeeds; ERC-20 balance is reduced by `N` (tokens burned).
5. The NEAR `ft_transfer` promise executes and **fails** (storage not paid).
6. `exit_to_near_precompile_callback` is invoked: `handler.promise_result(0)` is not `Successful`; `args.refund` is `None`; the function returns `Ok(None)`.
7. Observe: ERC-20 balance on Aurora = original − N; NEP-141 balance on NEAR = unchanged; no refund anywhere. The `N` tokens are permanently destroyed.

This is confirmed by the existing integration test `test_exit_to_near_refund` which explicitly asserts the reduced balance under `#[cfg(not(feature = "error_refund"))]`. [4](#0-3)

            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);

            None
        } else if let Some(args) = args.refund {

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
