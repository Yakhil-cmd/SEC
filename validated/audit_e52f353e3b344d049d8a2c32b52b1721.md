### Title
Malicious Dealer Can Force Hours-Long BSGS Decryption via Out-of-Range Chunks Passing `verify_chunking` — (`rs/crypto/internal/crypto_lib/threshold_sig/bls12_381/src/ni_dkg/fs_ni_dkg/forward_secure.rs`)

---

### Summary

A Byzantine dealer below the fault threshold can craft a NI-DKG dealing whose plaintext chunks lie outside `[CHUNK_MIN, CHUNK_MAX]` while still producing a `verify_chunking` proof that passes. When honest nodes later call `dec_chunks` during `load_transcript`, the `HonestDealerDlogLookupTable` fails to find the discrete log, and the code falls through to `CheatingDealerDlogSolver`, which the codebase itself documents as potentially taking **hours** per chunk. This stalls every subnet node during key loading.

---

### Finding Description

**Step 1 — The range check in `verify_chunking` does not bound individual chunks.** [1](#0-0) 

The only range enforcement is `z_sk < zz_big` on the proof response `z_s`, where `zz = 2 * NUM_ZK_REPETITIONS * n * m * (CHUNK_SIZE-1) * CHALLENGE_MASK`. This is a statistical bound on a *linear combination* of chunks, not a direct bound on each `s_ij`. A cheating dealer can choose chunks in `[CHUNK_SIZE, zz-1]` — far outside `[0, 65535]` — and still satisfy the check by rejection-sampling `sigma` until `z_s` lands below `zz_big`, exactly as the honest prover does.

**Step 2 — `dec_chunks` falls through to the expensive BSGS solver.** [2](#0-1) 

`HonestDealerDlogLookupTable` only covers `[CHUNK_MIN..=CHUNK_MAX]`: [3](#0-2) 

When any chunk is out of that range, `solve_several` returns `None` for that slot, and `dec_chunks` unconditionally constructs `CheatingDealerDlogSolver::new(n, m)` and calls `solve` on every unresolved chunk.

**Step 3 — `CheatingDealerDlogSolver` is explicitly documented to take hours.** [4](#0-3) 

The BSGS table is up to 2 GiB (`MAX_TABLE_MBYTES = 2 * 1024`), the search range is `2 * zz - 1` (enormous for production subnet sizes), and the outer loop iterates over `scale_range = 2^CHALLENGE_BITS` values of `delta`. The `#[ignore]` test `print_time_for_cheating_dlog_solver_to_run` exists precisely to measure this cost and acknowledges it can be hours. [5](#0-4) 

**Step 4 — The existing test confirms out-of-range chunks pass ciphertext integrity and trigger the cheating path.** [6](#0-5) 

`should_decrypt_correctly_for_cheating_dealer` explicitly creates chunks with `chunk > CHUNK_MAX`, asserts `verify_ciphertext_integrity` passes, and then calls `dec_chunks` — confirming the full path is reachable.

---

### Impact Explanation

Every honest node in the subnet must call `dec_chunks` on every dealing included in the transcript. A single malicious dealer's dealing with one out-of-range chunk forces **all** nodes to run `CheatingDealerDlogSolver` for that chunk. With production subnet sizes (28–40 nodes), the BSGS range and `scale_range` outer loop make this a multi-hour computation per node. During this time, `load_transcript` blocks, preventing the subnet from completing the DKG round, loading the threshold key, and resuming threshold signing — a complete liveness failure for the subnet.

---

### Likelihood Explanation

A single Byzantine dealer (one node, below any fault threshold) suffices. The dealer only needs to:
1. Choose one chunk value slightly above `CHUNK_MAX` (e.g., `CHUNK_SIZE + 1`).
2. Run the standard `prove_chunking` rejection-sampling loop — the probability of `z_s` landing in range is still non-negligible for small overflows, so the loop terminates quickly.
3. Submit the dealing through the normal DKG protocol flow.

No privileged access, no key material, no majority collusion is required.

---

### Recommendation

Add an explicit per-chunk range check inside `verify_chunking` (or in `verify_dealing` before accepting a dealing into the transcript pool) that rejects any dealing whose ciphertext chunks, when combined with the ZK witness, imply plaintext values outside `[CHUNK_MIN, CHUNK_MAX]`. Alternatively, enforce the range at the `ChunkingWitness` construction boundary so that out-of-range witnesses are rejected before a proof is even attempted. The `CheatingDealerDlogSolver` should remain as a correctness fallback but should never be reachable from a dealing that passed `verify_dealing`.

---

let _ = dec_chunks(&secret_key, 0, &crsz, epoch, &[]);

#[test]
fn should_decrypt_correctly_for_cheating_dealer() {
    let epoch = Epoch::from(0);
    let rng = &mut reproducible_rng();
    let associated_data = rng.r#gen::<[u8; 10]>();

    let sys = SysParam::global();
    let key_gen_assoc_data = rng.r#gen::<[u8; 32]>();

    let nodes = 3;

    let mut keys = Vec::new();
    for _ in 0..nodes {
        let key_pair = kgen(&key_gen_assoc_data, sys, rng);
        keys.push(key_pair);
    }
    let pks: Vec<_> = keys.iter().map(|key| key.0.public_key().clone()).collect();

    let mut sij = {
        let mut sij = Vec::with_capacity(nodes);

        for _ in 0..nodes {
            let mut chunks = [0; NUM_CHUNKS];

            for i in 0..NUM_CHUNKS {
                // ensure that multiplying by delta pushes us out of Chunk range
                let chunk = (0x8000 | rng.r#gen::<u16>()) as isize;
                chunks[i] = chunk;
            }
            // this ensures that chunks is the encoding of a scalar less
            // than the group order:
            chunks[0] %= 0x73ee;

            sij.push(chunks);
        }

        sij
    };

    let cheating_i = rng.r#gen::<usize>() % nodes;
    let cheating_j = std::cmp::max(1, rng.r#gen::<usize>() % NUM_CHUNKS);

    let delta = (2 + rng.r#gen::<usize>() % 10) as isize;
    sij[cheating_i][cheating_j] *= delta; // doesn't overflow as delta is small and isize >> u16

    // however the new sij *is* larger than the maximum "legal" chunk
    assert!(sij[cheating_i][cheating_j] > CHUNK_MAX);

    let cheating_chunks = sij
        .iter()
        .map(|c| PlaintextChunks::new_unchecked(*c))
        .collect::<Vec<_>>();

    let pks_and_chunks = pks
        .iter()
        .cloned()
        .zip(cheating_chunks.iter().cloned())
        .collect::<Vec<_>>();

    let (crsz, _witness) = enc_chunks(&pks_and_chunks, epoch, &associated_data, sys, rng);

    // still a valid ciphertext
    assert!(verify_ciphertext_integrity(&crsz, epoch, &associated_data, sys).is_ok());

    for i in 0..keys.len() {
        let secret_key = &keys[i].1;
        let out = dec_chunks(secret_key, i, &crsz, epoch, &associated_data);
        assert_eq!(
            out.unwrap(),
            cheating_chunks[i].recombine_to_scalar(),
            "Message decrypted wrongly"
        );
    }
