### Title
Integer Division Truncation Causes `native_per_pubdata` to Be Zero, Making Pubdata Free - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary

When `pubdata_price < native_price`, the floor-division used to compute `native_per_pubdata` truncates to `0`. This silently makes all pubdata free in terms of native resources, allowing any transaction sender to write unlimited pubdata without paying the native resource cost the operator intended to charge.

### Finding Description

In `validation_impl.rs`, the `native_per_pubdata` ratio is computed using plain integer floor division (`wrapping_div`):

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

By contrast, `native_per_gas` is computed with ceiling division (`div_ceil`), which guarantees a non-zero result whenever `gas_price > 0`:

```rust
// line 135
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

When `pubdata_price < native_price` (e.g., `pubdata_price = 500`, `native_price = 1000`), `wrapping_div` yields `0`. The same truncation exists in the public API helper:

```rust
// api/src/helpers.rs, line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

With `native_per_pubdata = 0`, every downstream pubdata charge becomes zero:

1. **Intrinsic pubdata overhead** in `create_resources_for_tx`:
   `intrinsic_pubdata_overhead = 0.saturating_mul(intrinsic_pubdata) = 0` [4](#0-3) 

2. **Execution pubdata charge** in `get_resources_to_charge_for_pubdata`:
   `native = current_pubdata_spent.checked_mul(0) = 0` [5](#0-4) 

The `native_per_gas == 0` case is explicitly handled as "unlimited native" (free), but `native_per_pubdata == 0` has no such guard — it silently makes pubdata free rather than being rejected or rounded up. [6](#0-5) 

The same `native_per_pubdata` value flows into the `before_refund` step for L2 transactions: [7](#0-6) 

### Impact Explanation

When `pubdata_price < native_price`, any transaction sender can write the maximum allowed pubdata per transaction at zero native resource cost. Pubdata is the dominant proving cost (it drives L1 data availability fees). The operator set a non-zero `pubdata_price` intending to charge for it, but the truncation silently zeroes out the charge. The result is:

- **Resource accounting bug**: Users underpay for proving costs; the operator/protocol absorbs the proving cost without compensation.
- **Pubdata spam**: An attacker can fill every transaction with storage writes up to the pubdata limit, maximising proving cost while paying only EVM gas (which does not cover pubdata proving).

The pubdata limit (`get_pubdata_limit`) bounds the per-transaction damage, but the attacker can submit many such transactions. [8](#0-7) 

### Likelihood Explanation

The condition `pubdata_price < native_price` is a realistic operator configuration. For example, if `native_price` is set high to reflect expensive proving cycles and `pubdata_price` is set to a moderate value (e.g., reflecting current L1 calldata costs), the ratio truncates to 0. No special attacker capability is required beyond submitting a normal L2 transaction; the attacker simply writes storage slots to maximise pubdata.

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, mirroring the treatment of `native_per_gas`:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. This ensures that any non-zero `pubdata_price` always results in at least 1 native unit charged per pubdata byte, preventing silent free pubdata.

### Proof of Concept

**Setup:**
- `native_price = 1000` (operator-set block parameter)
- `pubdata_price = 999` (operator-set block parameter, non-zero but less than `native_price`)
- `gas_price = 2000`, `gas_limit = 10_000_000`

**Trace through validation:**

1. `native_per_gas = div_ceil(2000, 1000) = 2` ✓ (non-zero, correct)
2. `native_per_pubdata = wrapping_div(999, 1000) = 0` ← **truncated to zero**
3. `native_prepaid_from_gas = 2 * 10_000_000 = 20_000_000`
4. `intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0` → no native deducted for intrinsic pubdata
5. During execution, attacker writes 20 storage slots → ~640 bytes pubdata
6. `get_resources_to_charge_for_pubdata(640, 0)` → charges `640 * 0 = 0` native
7. `check_enough_resources_for_pubdata` passes (0 ≤ remaining native)
8. Transaction succeeds; attacker paid 0 native for 640 bytes of pubdata

The operator intended to charge `999 * 640 = 639,360` base-token units worth of native resources for this pubdata, but charged 0. [9](#0-8) [10](#0-9)

pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent

    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
