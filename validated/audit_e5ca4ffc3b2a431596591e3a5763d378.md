### Title
SNS Governance `perform_manage_nervous_system_parameters` Applies New `round_duration_seconds` Retroactively Without First Distributing Pending Rewards - (File: rs/sns/governance/src/governance.rs)

---

### Summary

When a `ManageNervousSystemParameters` proposal changes `voting_rewards_parameters.round_duration_seconds`, the new duration is immediately stored and then used by `distribute_rewards` to compute `new_rounds_count` for the **entire elapsed period since the last reward event**, including time that elapsed under the old duration. This is the direct IC analog of the Ion Protocol bug: a rate-governing parameter is changed without first settling the accrued state, causing the new parameter to be applied retroactively.

---

### Finding Description

`perform_manage_nervous_system_parameters` in SNS Governance simply validates and stores the new parameters with no reward settlement step:

```rust
fn perform_manage_nervous_system_parameters(
    &mut self,
    proposed_params: NervousSystemParameters,
) -> Result<(), GovernanceError> {
    let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());
    match new_params.validate() {
        Ok(()) => {
            self.proto.parameters = Some(new_params);  // ← no distribute_rewards() call
            Ok(())
        }
        ...
    }
}
``` [1](#0-0) 

When `distribute_rewards` is subsequently called by the periodic task, it reads the **current** (already-changed) `round_duration_seconds` and divides the entire elapsed wall-clock time since `latest_reward_event.end_timestamp_seconds` by it:

```rust
let round_duration_seconds = match voting_rewards_parameters.round_duration_seconds { ... };

let reward_start_timestamp_seconds = self
    .latest_reward_event()
    .end_timestamp_seconds
    .unwrap_or_default();
let new_rounds_count = now
    .saturating_sub(reward_start_timestamp_seconds)
    .saturating_div(round_duration_seconds);   // ← uses NEW duration for OLD elapsed time
``` [2](#0-1) 

The reward purse loop then multiplies by the new `round_duration()` for every round:

```rust
for i in 1..=new_rounds_count {
    let current_reward_rate = voting_rewards_parameters.reward_rate_at(...);
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
``` [3](#0-2) 

The proto definition itself acknowledges the hazard but does not prevent it:

> "When this field is not populated, voting rewards are 'disabled'. Once this is set, it probably should not be changed, because the results would probably be pretty confusing." [4](#0-3) 

---

### Impact Explanation

**Case A — `round_duration_seconds` decreased (e.g., 7 days → 1 day):**
If 6 days have elapsed since the last reward event, the old duration would yield `6d / 7d = 0` rounds (no distribution yet). After the change, `6d / 1d = 6` rounds are computed. Six reward purses are minted immediately, each using the new 1-day `round_duration()` in the formula. Rewards are distributed prematurely and in incorrect amounts relative to what token holders expected.

**Case B — `round_duration_seconds` increased (e.g., 1 day → 7 days):**
If 3 days have elapsed, the old duration would yield 3 rounds. After the change, `3d / 7d = 0` rounds — no distribution occurs. The 3 days of accrued rewards are silently delayed until 7 full days have elapsed from the last event. Neuron maturity growth is unexpectedly stalled.

In both cases the `reward_rate_at` time-argument is also shifted because it is computed relative to the new `round_duration_seconds`, further distorting the per-round rate used in the purse calculation.

**Impact: Medium** — Incorrect neuron maturity accrual (over- or under-distribution) for all staked neurons in the affected SNS. This is a ledger conservation / governance reward accounting bug.

---

### Likelihood Explanation

**Likelihood: Low** — Exploiting this requires a `ManageNervousSystemParameters` proposal that changes `voting_rewards_parameters.round_duration_seconds` to pass with a governance majority. This is a standard, non-privileged governance action available to any SNS token holder with sufficient stake. The effect is triggered automatically by the next periodic task execution after the proposal executes — no further attacker action is needed. The proto comment warns against changing this field, but the code does not enforce that warning.

---

### Recommendation

In `perform_manage_nervous_system_parameters`, before applying new `voting_rewards_parameters`, call `distribute_rewards` (or an equivalent settlement step) using the **current** parameters. This ensures the new `round_duration_seconds` only governs future reward periods, not the already-elapsed time since the last reward event. Alternatively, after the parameter change, update `latest_reward_event.end_timestamp_seconds` to `now` so that the elapsed-time calculation starts fresh from the moment of the change.

---

### Proof of Concept

1. SNS is initialized with `round_duration_seconds = 604800` (7 days).
2. 6 days pass; `distribute_rewards` is not triggered (threshold not yet met under old duration).
3. A `ManageNervousSystemParameters` proposal is submitted and passes, setting `round_duration_seconds = 86400` (1 day).
4. `perform_manage_nervous_system_parameters` stores the new parameters immediately with no reward settlement. [5](#0-4) 
5. On the next periodic task tick, `distribute_rewards` computes:
   `new_rounds_count = (6 * 86400) / 86400 = 6`
   instead of the expected `0` under the old 7-day duration. [6](#0-5) 
6. Six reward purses are minted, each using `round_duration() = 1 day`, distributing 6× the expected single-round reward amount to all voting neurons.
7. `latest_reward_event.end_timestamp_seconds` is advanced by `6 * 86400`, consuming the 6-day window under the wrong duration. [7](#0-6)

    fn perform_manage_nervous_system_parameters(
        &mut self,
        proposed_params: NervousSystemParameters,
    ) -> Result<(), GovernanceError> {
        // Only set `self.proto.parameters` if "applying" the proposed params to the
        // current params results in valid params
        let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());

        log!(
            INFO,
            "Setting Governance nervous system params to: {:?}",
            &new_params
        );

        match new_params.validate() {
            Ok(()) => {
                self.proto.parameters = Some(new_params);
                Ok(())
            }

            // Even though proposals are validated when they are first made, this is still
            // possible, because the inner value of a ManageNervousSystemParameters
            // proposal is only valid with respect to the current
            // nervous_system_parameters() at the time when the proposal was first
            // made. If nervous_system_parameters() changed (by another proposal) since
            // the current proposal was first made, the current proposal might have become
            // invalid. Basically, this might occur if there are conflicting (concurrent)
            // proposals, but we expect this to be highly unusual in practice.
            Err(msg) => Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!(
                    "Failed to perform ManageNervousSystemParameters action, proposed \
                        parameters would lead to invalid NervousSystemParameters: {msg}"
                ),
            )),
        }
    }
