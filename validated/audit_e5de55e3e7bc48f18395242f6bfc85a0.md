Audit Report

## Title
Unbounded `inbound_connecting` JoinSet Allows Byzantine Subnet Peer to Exhaust Replica Resources via QUIC Connection Flood — (`rs/p2p/quic_transport/src/connection_manager.rs`)

## Summary

`handle_inbound_conn_attemp` unconditionally spawns a Tokio task into the `inbound_connecting` `JoinSet` for every accepted QUIC `Incoming` with no per-peer cap, no global cap, and no deduplication. A single Byzantine subnet peer holding a valid NNS-issued TLS certificate can open arbitrarily many concurrent QUIC connections, accumulating unbounded tasks in the `JoinSet` — each alive for up to `CONNECT_TIMEOUT` (10 s) — exhausting Tokio task heap, QUIC connection state, and OS file descriptors, potentially crashing or stalling the victim replica's event loop.

## Finding Description

**Root cause — `handle_inbound_conn_attemp` (L538–588):**

Every iteration of the `run` `select!` loop calls `self.endpoint.accept()`. Each accepted `Incoming` is passed directly to `handle_inbound_conn_attemp`: [1](#0-0) 

Inside that function, a timeout-wrapped future is spawned with no guard of any kind: [2](#0-1) 

The field itself is a plain, unbounded `JoinSet`: [3](#0-2) 

**Contrast with outbound path:** `outbound_connecting` is a `JoinMap<NodeId, ...>`, so at most one outbound task per peer exists at any time. `inbound_connecting` is a plain `JoinSet` — no key, no deduplication, no cap: [4](#0-3) 

**TLS does not prevent the flood:** On topology change, the server config is updated to only allow subnet members: [5](#0-4) 

This blocks non-subnet nodes from completing TLS. However, a Byzantine peer *is* a subnet member and holds a valid NNS-issued certificate. Its connections complete TLS successfully, so every spawned task runs to completion — it is not rejected early. The TLS guard filters outsiders only, not misbehaving insiders.

**Event loop processes one completion per iteration:** The `select!` loop handles exactly one branch per iteration. If the attacker opens connections faster than the loop drains `inbound_connecting.join_next()`, the `JoinSet` grows without bound: [6](#0-5) 

**`CONNECT_TIMEOUT` = 10 s** means each task lives for up to 10 seconds, giving the attacker a sustained window to accumulate tasks: [7](#0-6) 

## Impact Explanation

Each spawned task holds a Tokio task stack allocation, a `quinn::Incoming`/`quinn::Connection` object (QUIC connection state, crypto context, stream buffers), and a reference to the endpoint's internal connection table entry. With N concurrent connections from one Byzantine peer, the replica accumulates N tasks × (task overhead + QUIC connection state). At scale this exhausts Tokio task heap memory or OS-level resources (file descriptors, UDP socket receive-buffer entries), stalling or OOM-crashing the replica's event loop. A crashed replica loses its role in consensus. This matches the **High** bounty impact: *Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS.*

## Likelihood Explanation

The attacker is a Byzantine subnet peer — a realistic fault model explicitly within the IC's threat model (up to `f` Byzantine nodes in a subnet of `3f+1`). The peer holds a valid node TLS certificate issued by the NNS. Opening many concurrent QUIC connections from a single process is trivial using the `quinn` client library in a loop. No privileged access beyond subnet membership, no key compromise, and no majority corruption is required. The attack is local-testable and reproducible.

## Recommendation

1. **Add a global cap** on `inbound_connecting.len()` (e.g., `2 × subnet_size`) before spawning; call `incoming.refuse()` when the cap is reached.
2. **Add a per-peer inbound connection cap** by tracking in-flight inbound tasks per source IP/`NodeId` (the remote address is available on `Incoming` before the handshake) and dropping `Incoming` objects that exceed the limit (e.g., 2–3 concurrent attempts per peer).
3. **Consider migrating `inbound_connecting` to a `JoinMap`** keyed by source IP or peer identity once extractable pre-handshake, mirroring the deduplication already present in `outbound_connecting`.

## Proof of Concept

```rust
// Byzantine peer opens N concurrent QUIC connections to victim replica
let client_config = make_client_config_with_valid_node_cert(); // valid NNS-issued cert
let mut handles = Vec::new();
for _ in 0..N {
    let cfg = client_config.clone();
    handles.push(tokio::spawn(async move {
        let endpoint = quinn::Endpoint::client("0.0.0.0:0".parse().unwrap()).unwrap();
        let _ = endpoint
            .connect_with(cfg, victim_addr, "irrelevant")
            .unwrap()
            .await; // completes TLS, task stays in inbound_connecting for up to 10s
    }));
}
// Assert: victim's inbound_connecting.len() grows to N
// Assert: victim RSS grows proportionally to N
// Assert: victim event loop latency degrades as N increases
```

A deterministic integration test can be written using `PocketIC` or a local `quinn` endpoint pair: spawn a test replica, connect N times from a peer with a valid test certificate, and assert that `connecting_connections` metric (which tracks `inbound_connecting.len()`) reaches N and that the event loop's select iteration latency degrades.
