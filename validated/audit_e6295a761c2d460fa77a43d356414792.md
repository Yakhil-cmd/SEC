### Title
Unbounded `ingress_expiries` Vec Allocation via Attacker-Controlled `ingress_start=0` / `ingress_end=u64::MAX` Causes OOM DoS — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

---

### Summary

The `construction_payloads` function in the ICRC1 Rosetta server accepts attacker-controlled `ingress_start` and `ingress_end` values from the HTTP request body, applies two guards that are both bypassable with extreme values, and then enters an unbounded `while` loop that pushes one `u64` per iteration into a `Vec`. Supplying `ingress_start=0` and `ingress_end=u64::MAX` causes ~154 billion loop iterations allocating ~1.23 TB of heap memory, crashing the Rosetta server process with OOM.

---

### Finding Description

The vulnerable function is `construction_payloads` in: [1](#0-0) 

The step values are derived from protocol constants:
- `MAX_INGRESS_TTL` = 300 s, `PERMITTED_DRIFT` = 60 s → `ingress_interval` = **240,000,000,000 ns**
- `INGRESS_INTERVAL_OVERLAP` = 120 s → **120,000,000,000 ns** [2](#0-1) [3](#0-2) 

The two guards before the loop are:

**Guard 1** — `ingress_start >= ingress_end`: [4](#0-3) 

With `ingress_start=0` and `ingress_end=u64::MAX`, `0 >= u64::MAX` is **false** → guard does not fire.

**Guard 2** — `ingress_end < now + ingress_interval`: [5](#0-4) 

`now` in 2026 ≈ 1.75×10¹⁸ ns. `now + ingress_interval` ≈ 1.75×10¹⁸. `u64::MAX` = 1.844×10¹⁹, which is **not less than** `now + ingress_interval` → guard does not fire.

The loop then executes without any bound: [6](#0-5) 

**Iteration count** = `u64::MAX / 120_000_000_000` ≈ **153,722,867,280** (~154 billion).  
**Memory allocated** = 154 billion × 8 bytes ≈ **1.23 TB** → OOM kill.

The same structural issue exists in the ICP Rosetta variant: [7](#0-6) 

---

### Impact Explanation

An unauthenticated HTTP client can POST a single `ConstructionPayloadsRequest` to the Rosetta server's `/construction/payloads` endpoint. The server allocates memory proportional to `(ingress_end - ingress_start) / step` with no cap. The process is killed by the OS OOM killer, denying service to all concurrent users of that Rosetta instance, including exchange integrations relying on it for ICP/ICRC1 transfers.

---

### Likelihood Explanation

The endpoint is publicly reachable with no authentication. The payload is a standard JSON body. A single request is sufficient to trigger the crash. No special knowledge, keys, or privileges are required.

---

### Recommendation

Add an explicit upper bound on the number of ingress expiries before entering the loop:

```rust
const MAX_INGRESS_EXPIRIES: usize = 1_000; // ~33 hours at 2-min intervals

if ingress_start >= ingress_end {
    return Err(...);
}

let window_ns = ingress_end.saturating_sub(ingress_start);
let step = ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
let estimated_count = window_ns.checked_div(step).unwrap_or(u64::MAX);
if estimated_count as usize > MAX_INGRESS_EXPIRIES {
    return Err(Error::processing_construction_failed(
        "Ingress window too large"
    ));
}
```

Apply the same fix to `rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`.
