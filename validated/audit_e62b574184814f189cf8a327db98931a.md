### Title
ETH Permanently Locked at `ExitToNear` Precompile Address on Failed NEAR Promise Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

---

### Summary

When the `ExitToNear` precompile is invoked with ETH (base token) value and the downstream NEAR `ft_transfer` promise fails, the ETH is permanently locked at the precompile address. This occurs because the `error_refund` Cargo feature — which enables the refund path — is **not** included in the production `contract` feature set, leaving `callback_args.refund` unconditionally `None` in production builds.

---

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) handles ETH bridge-out by:

1. Accepting ETH value from the EVM caller (deducted from caller's EVM balance, credited to the precompile address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`).
2. Scheduling a NEAR `ft_transfer` (or `ft_transfer_call`) promise to the ETH connector contract.
3. Optionally attaching a callback `exit_to_near_precompile_callback` to handle failure.

The callback args are constructed as:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None in production
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is absent, `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`, so **no callback is attached at all** — the promise is created as `PromiseArgs::Create` with no failure handler:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback, no refund
} else {
    PromiseArgs::Callback(...)
};
``` [2](#0-1) 

If the `ft_transfer` promise fails (e.g., recipient NEAR account not registered with the ETH connector, insufficient gas, connector paused), the ETH remains at the precompile address with no recovery path. The `refund_on_error` function — which transfers ETH back from `exit_to_near::ADDRESS` to the user — is never called:

```rust
} else if let Some(args) = args.refund {
    // Only reached when error_refund is enabled
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← production path: silent loss
};
``` [3](#0-2) 

The `error_refund` feature is **not** in the `default` or `contract` feature sets of either crate:

```toml
[features]
default = ["std"]
contract = ["log", "aurora-engine-sdk/contract", "aurora-engine-precompiles/contract"]
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [4](#0-3) 

The `refund_on_error` function confirms that ETH is held at the precompile address and must be explicitly transferred back:

```rust
// ETH exit; transfer ETH back from precompile address
let exit_address = exit_to_near::ADDRESS;
engine.call(&exit_address, &refund_address, amount, ...)
``` [5](#0-4) 

---

### Impact Explanation

ETH sent to the `ExitToNear` precompile is credited to the precompile address in Aurora's EVM state. The precompile address has no private key and no code that can initiate a withdrawal. If the NEAR-side promise fails and `error_refund` is not enabled, the ETH is **permanently frozen** at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` with no recovery mechanism available to any party.

This is confirmed by the integration test, which explicitly documents the loss:

```rust
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
// "If the refund feature is not enabled, then there is no refund in the EVM"
``` [6](#0-5) 

**Impact: Critical — Permanent freezing of user ETH funds.**

---

### Likelihood Explanation

The trigger conditions are realistic and reachable by any unprivileged EVM user:

- Calling `ExitToNear` with a NEAR recipient account that is not registered with the ETH connector (a common user error).
- Calling `ExitToNear` when the ETH connector's `ft_transfer` is paused.
- Any transient NEAR-side failure (gas exhaustion in the promise, connector contract upgrade mid-flight).

The production WASM build (`contract` feature) does not include `error_refund`, so every production deployment is affected. No admin compromise or special privilege is required — a standard EVM transaction suffices.

---

### Recommendation

Enable `error_refund` in the `contract` feature set so that the refund path is always active in production:

```toml
contract = ["log", "aurora-engine-sdk/contract", "aurora-engine-precompiles/contract", "error_refund"]
```

Alternatively, if the input format change is a concern, implement an unconditional refund path that does not depend on a compile-time feature flag, ensuring that any failed `ft_transfer` promise always triggers `refund_on_error` for the ETH base-token exit case.

---

### Proof of Concept

1. Deploy Aurora Engine with the `contract` feature (no `error_refund`).
2. Fund EVM address `Alice` with 1 ETH on Aurora.
3. Alice calls the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with `value = 1 ETH` and `recipient = "unregistered.near"` (an account not registered with the ETH connector).
4. The EVM deducts 1 ETH from Alice's balance and credits it to the precompile address.
5. The NEAR `ft_transfer` promise to the ETH connector fails (unregistered account).
6. No callback is attached; `exit_to_near_precompile_callback` is never invoked.
7. Alice's EVM balance is 0 ETH. The precompile address holds 1 ETH. No recovery is possible.

The test `test_exit_to_near_eth_refund` in `engine-tests/src/tests/erc20_connector.rs` reproduces this exact scenario and confirms the ETH loss under `#[cfg(not(feature = "error_refund"))]`. [7](#0-6)
