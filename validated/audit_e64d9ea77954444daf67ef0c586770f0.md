### Title
EIP-2929 Warm Access Status Incorrectly Rolled Back on Subcall Revert — (`basic_system/src/system_implementation/caches/generic_pubdata_aware_plain_storage.rs`)

### Summary
`StorageElementMetadata::last_touched_in_tx` (the warm/cold flag) is stored in the rollbackable value chain of `ElementWithHistory`. When a subcall reverts, `finish_frame_impl` rolls back the cache snapshot, resetting `last_touched_in_tx` to its pre-subcall state (cold). EIP-2929 mandates that `accessed_storage_keys` are **never** rolled back on revert — once warm, always warm for the duration of the transaction.

### Finding Description

`materialize_element` warms a slot by writing `last_touched_in_tx = Some(current_tx_id)` via `x.update(...)`, which creates a new history record in the rollbackable chain. [2](#0-1) 

`finish_frame_impl` with a `rollback_handle` calls `self.cache.rollback(x.cache)`, which traverses the history linked list and removes all records newer than the snapshot, including the warm-up record. [6](#0-5) 

The developer comment at line 180 explicitly acknowledges this: *"in case of revert it should become cold again"* — but this is incorrect per EIP-2929. [7](#0-6) 

The fix would require storing `last_touched_in_tx` in `element_properties` (the non-rollbackable tier of `ElementWithHistory`) rather than in the rollbackable `CacheRecord` value. [8](#0-7) 

### Impact Explanation
Any EVM contract that:
1. Makes a subcall that SLOADs/SSTOREs slot `S` and then reverts
2. Then SLOADs slot `S` in the outer call

...will be charged 2100 gas (cold) instead of 100 gas (warm). Contracts designed with EIP-2929 warm-access assumptions (e.g., reentrancy guards, gas-bounded callbacks, ERC-4337 paymasters) may hit unexpected out-of-gas reverts on ZKsync OS that would not occur on mainnet Ethereum. This breaks EVM equivalence and can cause silent behavioral divergence.

### Likelihood Explanation
The path is unconditional and reachable by any unprivileged EVM contract. No special permissions, governance, or oracle manipulation required. The pattern (subcall reverts, outer call re-accesses same slot) is common in production contracts.

### Recommendation
Move `last_touched_in_tx` out of `CacheRecord<V, StorageElementMetadata>` (the rollbackable value) and into the `element_properties` field of `ElementWithHistory` (the non-rollbackable tier). The `element_properties` field is explicitly documented as persisting across rollbacks. [9](#0-8)  This matches EIP-2929 semantics: the warm flag is transaction-scoped, not frame-scoped.

### Proof of Concept
Deploy two contracts on ZKsync OS and a reference EVM (revm/geth):

```solidity
// Inner contract
function touchAndRevert(address target, uint slot) external {
    assembly { pop(sload(slot)) }  // warms slot
    revert();
}

// Outer contract
function test(address inner, uint slot) external returns (uint gasBefore, uint gasAfter) {
    try IInner(inner).touchAndRevert(address(this), slot) {} catch {}
    gasBefore = gasleft();
    assembly { pop(sload(slot)) }  // should be warm (100 gas), is cold (2100 gas) on ZKsync OS
    gasAfter = gasleft();
}
```

Assert: `gasBefore - gasAfter == 100` on reference EVM. On ZKsync OS, this will be `2100`, confirming the deviation.

    pub fn rollback(
        &mut self,
        records_memory_pool: &mut ElementPool<V, A>,
        snapshot_id: CacheSnapshotId,
    ) {
        // Caller should guarantee that snapshot_id is correct

        if unsafe { self.head.as_ref() }.touch_ss_id <= snapshot_id {
            return;
        }

        let mut first_removed_record = self.head;
        // Find first elem such that elem.touch_ss_id > snapshot_id and set previous as first_removed_record
        loop {
            let n_lnk = unsafe {
                first_removed_record
                    .as_mut()
                    .previous
                    .as_mut()
                    .expect("Every history is terminated with a 0'th snapshot")
            };

            let n = unsafe { n_lnk.as_mut() };

            if n.touch_ss_id <= snapshot_id {
                // This is guaranteed to happen by encountering the terminator snapshot.
                break;
            }

            first_removed_record = *n_lnk;
        }

        let last_removed_record = self.head;

        let new_head = unsafe { first_removed_record.as_mut() }
            .previous
            .take()
            .unwrap();

        if first_removed_record == self.first {
            self.first = new_head;
        }

        self.head = new_head;

        // Return subchain to the pool to be reused later
        records_memory_pool.reuse_memory(last_removed_record, first_removed_record);
    }
