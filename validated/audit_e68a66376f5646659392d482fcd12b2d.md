Audit Report

## Title
Unprivileged Caller Can Force Arbitrary Victim Into SNS Swap, Bypassing Confirmation-Text Consent Gate — (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` update endpoint on the SNS Swap canister accepts an arbitrary `buyer` principal in its request payload and performs no check that the caller equals that principal. Any unprivileged ingress sender can therefore call the endpoint on behalf of any victim, committing the victim's ICP to the swap and satisfying the swap's `confirmation_text` consent gate on their behalf — even though the victim never agreed. The confirmation text is publicly readable via `get_init`, making the gate entirely ineffective against a third-party caller.

## Finding Description
In `rs/sns/swap/canister/canister.rs` at L130–134, the effective buyer principal is resolved as:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no caller == buyer check
};
```

When `arg.buyer` is non-empty, `p` is passed directly to `refresh_buyer_token_e8s` with no assertion that `caller_principal_id() == p`. The proto definition at `rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto` L844 explicitly documents this as "If not specified, the caller is used", confirming the field is intentionally caller-overridable with no access control.

Inside `refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs` L1150), the only caller-supplied gate is:

```rust
self.validate_confirmation_text(confirmation_text)?;
```

This validates the attacker-supplied text against the swap's `Init.confirmation_text`. Because `confirmation_text` is part of the public `Init` state (readable by anyone via `get_init`), the attacker can read it and supply it verbatim. After passing this check, the function reads the ICP ledger balance of the victim's subaccount (L1154–1163), registers it as the victim's participation (L1285–1288), and deletes the victim's open ticket (L1270) — all keyed on the attacker-supplied `buyer` principal, not the actual caller.

There is no guard anywhere in the call path that requires the caller to be the buyer when a `confirmation_text` is configured.

## Impact Explanation
This is a **High** severity finding matching: *"Significant SNS infrastructure security impact with concrete user or protocol harm."* Concretely:

1. **Consent bypass**: SNS projects use `confirmation_text` as an explicit legal/consent gate. Because any caller can supply the public text on behalf of any buyer, the gate is entirely ineffective. A victim is recorded as having consented to terms they never read or accepted.
2. **Forced premature participation**: A user who has transferred ICP to their swap subaccount but has not yet decided to participate can be forced into the swap. Their ICP is locked until the swap concludes (committed or aborted).
3. **Ticket destruction**: The victim's open ticket is silently deleted, breaking the payment-flow state machine for that user and preventing future use of the ticket-based flow.

## Likelihood Explanation
The attack requires only: (1) the victim's principal (public on-chain information), (2) the swap's `confirmation_text` (readable from the public `get_init` endpoint), and (3) the victim having already transferred ICP to their swap subaccount (a normal first step in the payment flow). No privileged access, no key material, and no majority corruption is required. The attacker pays only the cycles cost of a single update call. The attack is repeatable against any victim who has funded their subaccount.

## Recommendation
Require that the effective buyer principal equals the caller when a `confirmation_text` is present, or unconditionally enforce `caller == buyer`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let specified = PrincipalId::from_str(&arg.buyer).unwrap();
    if specified != caller_principal_id() {
        if swap().init_or_panic().confirmation_text.is_some() {
            panic!("Caller must be the buyer when a confirmation text is required");
        }
    }
    specified
};
```

Alternatively, remove the ability to specify an arbitrary `buyer` entirely and always use `caller_principal_id()`.

## Proof of Concept
1. Deploy an SNS swap with `confirmation_text = "I confirm I have read the terms."`.
2. Victim transfers 10 ICP to `swap_canister[principal_to_subaccount(victim)]` on the ICP ledger (normal first step).
3. Victim has not yet called `refresh_buyer_tokens` — they are still deciding.
4. Attacker calls `get_init` on the swap canister to read the public `confirmation_text`.
5. Attacker sends an ingress update call to `refresh_buyer_tokens` with `buyer = victim_principal_text` and `confirmation_text = "I confirm I have read the terms."`.
6. The swap canister resolves `p = victim_principal` (L130–134), reads the victim's 10 ICP balance (L1154–1163), validates the public confirmation text as if the victim supplied it (L1150), commits 10 ICP as the victim's participation (L1285–1288), and deletes the victim's open ticket (L1270).
7. The victim is now a registered swap participant who has "agreed" to the terms, with their ICP locked in the swap — without ever having called the endpoint themselves.

A deterministic integration test or PocketIC test can reproduce this by: creating a swap with `confirmation_text`, funding a victim subaccount, then calling `refresh_buyer_tokens` from a different principal with the victim's principal as `buyer` and the public confirmation text, and asserting the victim appears in `buyers` and their ticket is gone.
