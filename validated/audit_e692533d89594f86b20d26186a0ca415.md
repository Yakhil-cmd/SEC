Audit Report

## Title
Missing `callback_id` validation in `push_response` reject arms and `validate_self_validating_payload_impl` allows a malicious block proposer to forge Bitcoin adapter reject responses and deliver them to any canister — (`rs/replicated_state/src/bitcoin.rs`, `rs/bitcoin/consensus/src/payload_builder.rs`)

## Summary
`validate_self_validating_payload_impl` accepts any non-empty `SelfValidatingPayload` that fits within the byte limit without verifying that any `callback_id` corresponds to a pending context. The `GetSuccessorsReject` and `SendTransactionReject` arms of `push_response` push a `ConsensusResponse` to `consensus_queue` with no context lookup, unlike the `GetSuccessorsResponse` arm which enforces a mandatory lookup. A single malicious block proposer can craft a `BitcoinAdapterResponse` with an arbitrary `callback_id`, have it notarized by honest nodes, and cause the forged reject payload to be delivered to whichever canister owns that `callback_id` — permanently consuming the context.

## Finding Description
**Root cause 1 — `push_response` reject arms lack context validation.**
In `rs/replicated_state/src/bitcoin.rs`, the `GetSuccessorsResponse` arm (lines 32–39) performs a mandatory lookup into `bitcoin_get_successors_contexts` and returns `StateError::BitcoinNonMatchingResponse` if the `callback_id` is unknown. The `GetSuccessorsReject` arm (lines 77–89) and `SendTransactionReject` arm (lines 90–102) skip this lookup entirely: they take `callback_id` directly from the attacker-supplied response and push a `ConsensusResponse` to `consensus_queue` unconditionally.

**Root cause 2 — `validate_self_validating_payload_impl` performs no state check.**
In `rs/bitcoin/consensus/src/payload_builder.rs` lines 234–251, the entire validation body is: return `Ok(0)` for an empty payload, otherwise return `Ok(byte_size)`. It does not load the certified state, does not check that any `callback_id` in the payload exists in `bitcoin_get_successors_contexts` or `bitcoin_send_transaction_internal_contexts`, and does not verify that the response variant matches the request type. Both `validate_self_validating_payload` (the `SelfValidatingPayloadBuilder` impl, line 300) and `validate_payload` (the `BatchPayloadBuilder` impl, line 379) delegate to this same function.

**Exploit flow:**
1. Attacker (a single malicious block proposer) crafts a `BitcoinAdapterResponse { response: GetSuccessorsReject(BitcoinReject { reject_code: SysFatal, message: "forged" }), callback_id: <target_id> }` and includes it in a `SelfValidatingPayload`.
2. `validate_payload` is called on honest nodes; it decodes the payload, calls `validate_self_validating_payload_impl`, which returns `Ok(byte_size)`. Honest nodes notarize and finalize the block.
3. During block execution, `push_response` is called. The `GetSuccessorsReject` arm pushes `ConsensusResponse::new(callback_id, Payload::Reject(...))` to `consensus_queue` with no lookup.
4. The scheduler (lines 1299–1325 of `rs/execution_environment/src/scheduler.rs`) drains `consensus_queue` unconditionally, wrapping each entry into a `SubnetMessage::Response` and passing it to `execute_subnet_message`.
5. `execute_subnet_message` calls `retrieve_context(callback_id)` (lines 699–702 of `rs/execution_environment/src/execution_environment.rs`), which searches all six context maps in order: `setup_initial_dkg_contexts`, `sign_with_threshold_contexts`, `reshare_chain_key_contexts`, `canister_http_request_contexts`, `bitcoin_get_successors_contexts`, `bitcoin_send_transaction_internal_contexts`.
6. If the attacker-chosen `callback_id` matches any pending context, that context is removed and `push_subnet_output_response` is called (line 771) with the attacker-controlled `Payload::Reject`, delivering the forged rejection to the originating canister.

## Impact Explanation
This is a **High** severity finding. A malicious block proposer can:
- Force any canister on a bitcoin-enabled subnet with a pending `BitcoinGetSuccessors` or `BitcoinSendTransactionInternal` call to receive a forged reject with an attacker-chosen `RejectCode` and message, permanently consuming the context so the legitimate adapter response is silently dropped.
- Because `retrieve_context` searches all six context maps, the same technique can forge rejections for threshold-signing (`SignWithECDSA`/`SignWithSchnorr`) or canister-HTTP contexts whose `callback_id` the attacker can observe from the replicated state. This directly threatens Chain Fusion and ckBTC/ckETH signing operations.
- The consumed context is permanently removed; the pending call never completes legitimately, constituting a targeted, permanent DoS of in-flight subnet-level calls.

This matches the allowed impact: **High ($2,000–$10,000) — Significant Chain Fusion, ck-token, ledger, boundary/API, or infrastructure security impact with concrete user or protocol harm**, and **High — Application/platform-level DoS or subnet availability impact**.

## Likelihood Explanation
A single malicious node below the Byzantine fault threshold can trigger this in any consensus round where it is selected as block maker. `callback_id` values are sequential integers observable from the public replicated state. The attack is deterministic, requires no brute-force, and can be repeated every time the attacker is selected as block proposer. No special privileges beyond being a subnet node are required.

## Recommendation
1. **In `push_response`** (`rs/replicated_state/src/bitcoin.rs`): Add a context lookup for `GetSuccessorsReject` and `SendTransactionReject` identical to the one already present for `GetSuccessorsResponse` — look up the `callback_id` in `bitcoin_get_successors_contexts` (respectively `bitcoin_send_transaction_internal_contexts`) and return `StateError::BitcoinNonMatchingResponse` if not found. Also apply the same fix to the `SendTransactionResponse` arm (lines 64–76), which similarly lacks a context lookup.
2. **In `validate_self_validating_payload_impl`** (`rs/bitcoin/consensus/src/payload_builder.rs`): Load the certified state at `validation_context.certified_height` and verify that each `BitcoinAdapterResponse.callback_id` in the payload corresponds to a pending entry in the appropriate context map (`bitcoin_get_successors_contexts` for `GetSuccessors*` variants, `bitcoin_send_transaction_internal_contexts` for `SendTransaction*` variants), and that the response variant matches the request type. Return `SelfValidatingPayloadValidationError` if any `callback_id` is unknown or mismatched.

## Proof of Concept
State-machine test outline:
1. Create a canister that issues a `BitcoinGetSuccessors` call; record the assigned `callback_id` (e.g., `0`).
2. Directly construct a `BitcoinAdapterResponse { response: GetSuccessorsReject(BitcoinReject { reject_code: SysFatal, message: "forged" }), callback_id: 0 }` and place it in a `SelfValidatingPayload`.
3. Call `validate_payload` on the payload — assert it returns `Ok(())` (demonstrating the missing guard).
4. Deliver the batch containing this payload; tick the state machine.
5. Assert the canister's reject handler fired with `reject_code = SysFatal` and `message = "forged"`.
6. Assert `bitcoin_get_successors_contexts` is now empty (context permanently consumed).
7. Optionally: repeat with a `callback_id` matching a pending `SignWithECDSA` context to demonstrate cross-context impact.
