Based on my thorough code analysis, here is my assessment:

### Title
Loose NIZK Chunking Proof Bounds Allow Byzantine Dealer to Trigger 2 GiB BSGS Allocation and Hours-Long CPU Exhaustion on All Receiver Replicas via `dec_chunks` → `CheatingDealerDlogSolver` — (`rs/crypto/internal/crypto_lib/threshold_sig/bls12_381/src/ni_dkg/fs_ni_dkg/forward_secure.rs`)

---

### Summary

A single Byzantine dealer node (below the fault threshold) can craft a NiDKG dealing whose encrypted plaintext chunks are out-of-range (chunk value × delta > `CHUNK_MAX`) while still producing a valid `ZKProofDec` (chunking proof). This is possible because `verify_chunking` only bounds the proof response `z_s < zz`, where `zz = 2 * NUM_ZK_REPETITIONS * ss`, which is approximately **64× larger** than the honest chunk range. When receivers call `load_transcript`, `dec_chunks` fails the fast `HonestDealerDlogLookupTable` lookup and falls through to `CheatingDealerDlogSolver::new(n, m)`, which allocates up to **2 GiB** and runs for **hours** per chunk. This violates the invariant that a single below-threshold Byzantine dealer must not cause unbounded computation on honest receivers.

---

### Finding Description

**Step 1 — The NIZK chunking proof range is loose.**

`verify_chunking` in `nizk_chunking.rs` only checks:

```rust
for z_sk in nizk.z_s.iter() {
    if z_sk >= &zz_big {
        return Err(ZkProofChunkingError::InvalidProof);
    }
}
``` [1](#0-0) 

where `zz = 2 * NUM_ZK_REPETITIONS * ss` and `ss = n * m * (CHUNK_SIZE - 1) * CHALLENGE_MASK`. [2](#0-1) 

This bounds `z_s`, not the chunk values directly. The implied bound on individual chunk values is approximately `zz / (n * m * CHALLENGE_MASK) = 2 * NUM_ZK_REPETITIONS * (CHUNK_SIZE - 1) ≈ 64 * CHUNK_MAX`. The codebase itself documents this:

> "The bounds of the NIZK chunking proof are loose, so a malicious DKG participant can force us to search around 2^40 candidates for a discrete log." [3](#0-2) 

**Step 2 — A Byzantine dealer can produce a valid chunking proof for out-of-range chunks.**

For chunks multiplied by delta ∈ [2, 64], the prover samples `sigma_k ∈ [-ss, zz-1]` and retries until `z_s = sum[e_ijk * s_ij] + sigma_k ∈ [0, zz)`. The probability of success per attempt for delta = 2 is approximately `(63/65)^32 ≈ 37%` (expected ~2.7 retries). For delta = 10, it is ~0.4% (expected ~250 retries). Both are computationally feasible for the attacker.

**Step 3 — The dealing passes `verify_dealing` / `verify_zk_proofs`.**

`verify_zk_proofs` calls `verify_ciphertext_integrity` and `verify_chunking` — both pass for out-of-range chunks with small delta. The test `should_decrypt_correctly_for_cheating_dealer` explicitly confirms `verify_ciphertext_integrity` passes:

```rust
// still a valid ciphertext
assert!(verify_ciphertext_integrity(&crsz, epoch, &associated_data, sys).is_ok());
``` [4](#0-3) 

`verify_zk_proofs` is called from `verify_dealing`: [5](#0-4) 

**Step 4 — `dec_chunks` falls through to `CheatingDealerDlogSolver::new(n, m)`.**

When `HonestDealerDlogLookupTable` fails to find the discrete log (because the chunk is out of range), `dec_chunks` constructs a `CheatingDealerDlogSolver`:

```rust
if dlogs.iter().any(|x| x.is_none()) {
    // Cheating dealer case
    let cheating_solver = CheatingDealerDlogSolver::new(n, m);
    // ...
    // It may take hours to brute force a cheater's discrete log.
    dlogs[i] = cheating_solver.solve(&powers[i]);
}
``` [6](#0-5) 

**Step 5 — `CheatingDealerDlogSolver::new` allocates 2 GiB.**

```rust
const MAX_TABLE_MBYTES: usize = 2 * 1024; // 2 GiB
``` [7](#0-6) 

The BSGS table is built with up to `MAX_TABLE_MBYTES` and `LARGEST_TABLE_MUL = 20` in release mode: [8](#0-7) 

**Step 6 — `load_transcript` → `compute_threshold_signing_key` iterates over ALL dealers.**

`compute_threshold_signing_key` decrypts shares from every dealer in the transcript, including Byzantine ones: [9](#0-8) 

`load_transcript` is called in a background thread with no timeout: [10](#0-9) 

---

### Impact Explanation

Every receiver replica that calls `load_transcript` for a transcript containing a Byzantine dealing will:
- Allocate up to **2 GiB** for the BSGS table in `CheatingDealerDlogSolver::new`
- Spend **hours** of CPU in `CheatingDealerDlogSolver::solve` (iterating up to 255 delta values × BSGS search over ~2^40 candidates)
- Repeat this for each of the `NUM_CHUNKS = 16` chunks in the Byzantine dealing

With multiple Byzantine dealers (still below the fault threshold f), the memory pressure multiplies: f × 2 GiB per receiver, potentially causing OOM on replica nodes.

---

### Likelihood Explanation

A Byzantine subnet member (a valid protocol peer below the consensus fault threshold) can execute this attack during any DKG interval. The attacker only needs to:
1. Be a registered dealer in the DKG config
2. Craft a dealing with chunks multiplied by delta ∈ [2, 64]
3. Produce a valid chunking proof (feasible with ~3–250 retries depending on delta)

The dealing will be validated and included in the transcript by honest nodes, triggering the expensive computation on all receivers.

---

### Recommendation

1. **Tighten the NIZK chunking proof range**: Reduce the sigma sampling range so that `z_s < zz` implies chunk values ≤ `CHUNK_MAX`. This requires adjusting the proof parameters to make the range tight.
2. **Add a chunk range check before invoking `CheatingDealerDlogSolver`**: After decryption, verify that the recovered discrete log is within `[0, CHUNK_MAX]` before proceeding. If not, immediately return `Err(DecErr::InvalidChunk)` without running the expensive BSGS.
3. **Resource-limit the background thread**: Apply a timeout or memory cap to the `load_transcript` background thread.
4. **Reject transcripts with out-of-range chunks at the dealing verification stage**: Add an explicit range check on the decrypted chunk values during `verify_dealing`.

---
