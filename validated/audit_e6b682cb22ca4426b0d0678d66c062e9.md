### Title
Unvalidated NEAR Recipient Account in `ExitToNear` Precompile Enables Permanent Fund Freeze — (`File: engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile accepts a user-supplied NEAR account ID as the withdrawal recipient. It validates only the syntactic format of the account ID, not whether the account exists on NEAR or has NEP-141 storage registered. When the downstream `ft_transfer` promise fails (e.g., unregistered recipient), the ERC-20 tokens have already been burned on Aurora with no on-chain refund path unless the compile-time `error_refund` feature is active. This is a direct structural analog to the ThorSwap unvalidated vault address: an unprivileged EVM user supplies an arbitrary destination, funds leave the protocol, and no recovery is possible.

---

### Finding Description

**Root cause — `parse_recipient` performs format-only validation:** [1](#0-0) 

`parse_recipient` calls `.parse()` on the raw bytes, which checks NEAR account ID character rules but makes no on-chain existence or storage-registration check. Any syntactically valid but non-existent (or unregistered) account ID passes through.

**Funds are committed before the promise is dispatched:**

For the base-token (ETH) path, `context.apparent_value` is deducted from the caller's EVM balance during execution. For the ERC-20 path, the ERC-20 contract burns tokens before calling the precompile. Both happen inside the EVM execution phase, before any NEAR promise is scheduled. [2](#0-1) 

**The callback (refund) is conditionally absent:** [3](#0-2) 

The callback that would refund the user is only appended when `callback_args != ExitToNearPrecompileCallbackArgs::default()`. For the base-token and standard ERC-20 paths, `transfer_near_args` is `None`: [4](#0-3) 

Without the `error_refund` compile-time feature, `refund` is also `None`: [5](#0-4) 

So `callback_args == default()` → `PromiseArgs::Create(transfer_promise)` is used → no callback → if `ft_transfer` fails on NEAR, there is no on-chain path to return the burned tokens.

**The scheduled promise targets the NEP-141 contract with `ft_transfer`:** [6](#0-5) 

On NEAR, `ft_transfer` panics if the recipient has no storage registered (NEP-141 requirement). The tokens remain with the NEP-141 contract; Aurora has no record of them and no callback to reclaim them.

---

### Impact Explanation

**Permanent freezing of funds (Critical).**

ERC-20 tokens (or ETH) are irreversibly burned/deducted on Aurora. The corresponding NEP-141 `ft_transfer` fails silently from Aurora's perspective (no callback). The tokens are stranded in the NEP-141 contract with no mechanism for the user or the protocol to recover them. This matches the "permanent freezing of funds" impact category.

---

### Likelihood Explanation

**Medium.**

- The `ExitToNear` precompile is a primary bridge exit path used by every EVM user withdrawing tokens to NEAR.
- NEAR account IDs are human-readable strings (e.g., `alice.near`). Typos, copy-paste errors, or directing funds to an account that has not yet registered storage with the NEP-141 contract are realistic user mistakes.
- No privileged access is required; any EVM transaction can trigger this path.
- The `error_refund` feature is a compile-time opt-in, not a guaranteed runtime safeguard, leaving deployments without it fully exposed.

---

### Recommendation

1. **Always attach a callback** (`exit_to_near_precompile_callback`) for every `ExitToNear` invocation, regardless of the `error_refund` feature flag, so that a failed `ft_transfer` on NEAR triggers an on-chain refund to the original sender on Aurora.
2. **Make `error_refund` a default-on feature** or remove the conditional compilation entirely, ensuring the refund path is always present in production builds.
3. Optionally, perform a pre-flight storage-registration check via a view call before committing the burn, though this is harder to implement atomically in the NEAR async model.

---

### Proof of Concept

1. Alice holds 100 units of an ERC-20 token on Aurora backed by NEP-141 `token.near`.
2. Alice calls `withdraw(amount=100, recipient="nonexistent.near")` on the ERC-20 contract.
3. The ERC-20 contract burns 100 tokens and calls the `ExitToNear` precompile with `flag=0x1`, `amount=100`, `recipient_account_id="nonexistent.near"`.
4. `parse_recipient` accepts `"nonexistent.near"` as syntactically valid.
5. The precompile emits a promise log: `ft_transfer(receiver_id="nonexistent.near", amount="100")` targeting `token.near`.
6. Aurora schedules `PromiseArgs::Create(transfer_promise)` — no callback, because `callback_args == default()` (no `error_refund`, no wNEAR unwrap).
7. On NEAR, `token.near::ft_transfer` panics: `"The account nonexistent.near is not registered"`.
8. The promise failure is unobserved by Aurora (no callback registered).
9. Alice's 100 ERC-20 tokens are permanently gone from Aurora; `token.near` retains the NEP-141 balance with no path for recovery. [1](#0-0) [3](#0-2) [7](#0-6)

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}

fn exit_erc20_token_to_near<I: IO>(
    context: &Context,
    exit_params: &Erc20TokenParams,
    io: &I,
) -> Result<
    (
        AccountId,
        String,
        events::ExitToNear,
        String,
        Option<TransferNearArgs>,
    ),
    ExitError,
> {
    // In case of withdrawing ERC-20 tokens, the `apparent_value` should be zero. In opposite way
    // the funds will be locked in the address of the precompile without any possibility
    // to withdraw them in the future. So, in case if the `apparent_value` is not zero, the error
    // will be returned to prevent that.
    if context.apparent_value != U256::zero() {
        return Err(ExitError::Other(Cow::from(
            "ERR_ETH_ATTACHED_FOR_ERC20_EXIT",
        )));
    }

    let erc20_address = context.caller; // because ERC-20 contract calls the precompile.
    let nep141_account_id = get_nep141_from_erc20(erc20_address.as_bytes(), io)?;

    let (nep141_account_id, args, method, transfer_near_args, event) = match exit_params.message {
        // wNEAR address should be set via the `factory_set_wnear_address` transaction first.
        Some(Message::UnwrapWnear) if erc20_address == get_wnear_address(io).raw() =>
        // The flow is following here:
        // 1. We call `near_withdraw` on wNEAR account id on `aurora` behalf.
        // In such way we unwrap wNEAR to NEAR.
        // 2. After that, we call callback `exit_to_near_precompile_callback` on the `aurora`
        // in which make transfer of unwrapped NEAR to the `target_account_id`.
        {
            (
                nep141_account_id,
                format!(r#"{{"amount":"{}"}}"#, exit_params.amount.as_u128()),
                "near_withdraw",
                Some(TransferNearArgs {
                    target_account_id: exit_params.receiver_account_id.clone(),
                    amount: exit_params.amount.as_u128(),
                }),
                events::ExitToNear::Legacy(ExitToNearLegacy {
                    sender: Address::new(erc20_address),
                    erc20_address: Address::new(erc20_address),
                    dest: exit_params.receiver_account_id.to_string(),
                    amount: exit_params.amount,
                }),
            )
        }
        // In this flow, we're just forwarding the `msg` to the `ft_transfer_call` transaction.
        Some(Message::Omni(msg)) => (
            nep141_account_id,
            ft_transfer_call_args(&exit_params.receiver_account_id, exit_params.amount, msg)?,
            "ft_transfer_call",
            None,
            events::ExitToNear::Omni(ExitToNearOmni {
                sender: Address::new(erc20_address),
                erc20_address: Address::new(erc20_address),
                dest: exit_params.receiver_account_id.to_string(),
                amount: exit_params.amount,
                msg: msg.to_string(),
            }),
        ),
        // The legacy flow. Just withdraw the tokens to the NEAR account id.
        // P.S. We use underscore here instead of `None` to handle the case when a user
        // could add the `unwrap` suffix for non wNEAR ERC-20 token by mistake.
        _ => {
            // There is no way to inject json, given the encoding of both arguments
            // as decimal and valid account id respectively.
            (
                nep141_account_id,
                format!(
                    r#"{{"receiver_id":"{}","amount":"{}"}}"#,
                    exit_params.receiver_account_id,
                    exit_params.amount.as_u128()
                ),
                "ft_transfer",
                None,
                events::ExitToNear::Legacy(ExitToNearLegacy {
                    sender: Address::new(erc20_address),
                    erc20_address: Address::new(erc20_address),
                    dest: exit_params.receiver_account_id.to_string(),
                    amount: exit_params.amount,
                }),
            )
        }
    };

    Ok((
        nep141_account_id,
        args,
        event,
        method.to_string(),
        transfer_near_args,
    ))
}
