Audit Report

## Title
SNS Governance `cast_vote_and_cascade_follow` Performs Unbounded Synchronous BFS Over Follower Graph, Enabling Instruction-Exhaustion DoS Against Any Voting Neuron - (File: `rs/sns/governance/src/governance.rs`)

## Summary

The SNS governance canister's `cast_vote_and_cascade_follow` is a plain synchronous function with no instruction-limit guards. Because there is no cap on the number of neurons that may follow a given neuron (fan-in is unbounded), an attacker who stakes K neurons and sets each to follow a victim neuron can cause every subsequent `register_vote` call by that victim to exhaust the IC's 5-billion-instruction per-message limit, trapping the message and rolling back the victim's ballot. NNS governance already fixed this exact issue by converting its equivalent function to `async` with soft/hard instruction limits and timer-based deferral; SNS governance has no equivalent protection.

## Finding Description

**Root cause — synchronous unbounded BFS:**

`Governance::cast_vote_and_cascade_follow` in `rs/sns/governance/src/governance.rs` is declared as a plain `fn` (not `async`):

```rust
fn cast_vote_and_cascade_follow(
    proposal_id: &ProposalId,
    voting_neuron_id: &NeuronId,
    ...
) {
    while !induction_votes.is_empty() {
        // inner loops — no instruction check
        for follower_neuron_id in follower_neuron_ids {
            neurons.get(&follower_neuron_id.to_string()); // BTreeMap O(log n)
            follower_neuron.vote_from_ballots_following(...);
        }
    }
}

It is called synchronously from `register_vote` (also a plain `fn`):

```rust
Governance::cast_vote_and_cascade_follow(
    proposal_id, neuron_id, vote, function_id,
    &self.function_followee_index, ...

``` [3](#0-2) 

**Root cause — unbounded fan-in:**

The `follow` endpoint enforces `max_followees_per_function` (the number of neurons a single neuron may follow), but places **no cap** on the number of neurons that may follow a given neuron: [4](#0-3) 

Any principal can stake K neurons and call `Follow { followees: [victim_id] }` on each, making the victim's entry in `function_followee_index` contain K attacker-controlled neuron IDs with no protocol barrier.

**Contrast with NNS governance (already fixed):**

NNS governance's equivalent function is `async`, checks `over_soft_message_limit()` after each BFS tier, and calls `noop_self_call_if_over_instructions` with a 1B soft limit and 750B hard limit, deferring remaining work to a timer job: [5](#0-4) [6](#0-5) 

SNS governance has none of these protections.

**Exploit flow:**

1. Attacker stakes K SNS neurons via repeated `ClaimOrRefresh` calls.
2. For each attacker neuron, calls `manage_neuron → Follow { function_id: X, followees: [victim_id] }`. After K calls, `function_followee_index[X][victim_id]` contains K attacker neuron IDs.
3. Victim calls `manage_neuron → RegisterVote { proposal_id, vote: Yes }`.
4. `register_vote` calls `cast_vote_and_cascade_follow` synchronously. The BFS second loop iterates over all K follower neurons, performing a `neurons.get(...)` (BTreeMap O(log n)) and `vote_from_ballots_following(...)` for each.
5. With K large enough, the message exhausts 5B instructions and traps. All state changes are rolled back — including the victim's own ballot. The victim's vote is never recorded.
6. The attacker's follower neurons remain in the index permanently, so every future `register_vote` by the victim on any proposal traps identically.

**Why existing checks are insufficient:**

- `max_followees_per_function` limits outgoing edges per neuron, not incoming edges (fan-in). It does not bound the size of `function_followee_index[X][victim_id]`.
- There is no per-message instruction counter check anywhere in `cast_vote_and_cascade_follow` or its callers in SNS governance.
- The BFS terminates correctly (monotonicity argument in comments), but termination does not prevent instruction exhaustion within a single synchronous message.

## Impact Explanation

This is a targeted, persistent DoS against any SNS neuron's ability to vote. The victim neuron's ballot is permanently `Unspecified` on every proposal created after the attacker's neurons are staked, as long as the follower neurons remain. This constitutes a significant SNS governance security impact with concrete user and protocol harm: a neuron with substantial voting power is silenced indefinitely, potentially affecting proposal outcomes. This matches the **High ($2,000–$10,000)** impact class: "Significant SNS or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation

Any unprivileged principal can execute this attack. The only economic barrier is the cost of staking SNS tokens across K neurons. On SNS instances where the governance token trades at low prices, the cost to create enough follower neurons to exhaust 5B instructions (empirically, tens of thousands given the cost of BTreeMap lookups and `vote_from_ballots_following`) is modest. The attack requires no victim mistakes, no social engineering, no privileged access, and no ongoing attacker action after setup — it is fully persistent.

## Recommendation

1. Convert `cast_vote_and_cascade_follow` in `rs/sns/governance/src/governance.rs` to an `async` function, mirroring the NNS governance implementation.
2. Insert an instruction-limit check (using `ic_cdk::api::instruction_counter()` or `ic_nervous_system_long_message::is_message_over_threshold`) after each BFS tier, deferring remaining work to a timer job analogous to `process_voting_state_machines` in NNS governance.
3. Alternatively (or additionally), enforce a maximum fan-in per neuron per function in the `follow` endpoint: reject a `Follow` call if the target neuron already has more than `MAX_FOLLOWERS` followers for that function ID.

## Proof of Concept

**Minimal deterministic integration test using PocketIC or `StateMachine`:**

1. Deploy an SNS governance canister with a low-price token.
2. Create a victim neuron `V` with voting power.
3. Stake K attacker neurons (e.g., K = 50,000) via `ClaimOrRefresh`.
4. For each attacker neuron, call `manage_neuron → Follow { function_id: <motion_function_id>, followees: [V] }`.
5. Create a proposal of the corresponding function type.
6. Call `manage_neuron → RegisterVote { proposal_id, vote: Yes }` as the controller of `V`.
7. **Expected (vulnerable) outcome**: the call traps with an instruction limit error; `proposal.ballots[V].vote` remains `Unspecified`.
8. **Expected (fixed) outcome**: the call succeeds; `proposal.ballots[V].vote` is `Yes`; remaining follower cascade is deferred to a timer.

The existing `sns_cast_vote` test helper in `rs/sns/test_utils/src/state_test_helpers.rs` can be used as the basis for this test. [7](#0-6)

    pub async fn cast_vote_and_cascade_follow(
        &mut self,
        proposal_id: ProposalId,
        voting_neuron_id: NeuronId,
        vote_of_neuron: Vote,
        topic: Topic,
    ) {
        let voting_started = self.env.now();

        if !self.heap_data.proposals.contains_key(&proposal_id.id) {
            // This is a critical error, but there is nothing that can be done about it
            // at this place.  We somehow have a vote for a proposal that doesn't exist.
            eprintln!(
                "error in cast_vote_and_cascade_follow: Proposal not found: {}",
                proposal_id.id
            );
            return;
        }

        // First we cast the ballot.
        self.record_neuron_vote(proposal_id, voting_neuron_id, vote_of_neuron, topic);

        // We process until voting is finished, and then do any other work that fits into the soft
        // limit of the current message.  Votes are guaranteed to be recorded before the function
        // returns, but recent_ballots for neurons might be recorded later in a timer job.  This
        // ensures we return to the caller in a reasonable amount of time.
        let mut is_voting_finished = false;

        while !is_voting_finished {
            // Now we process until we are done or we are over a limit and need to
            // make a self-call.
            with_voting_state_machines_mut(|voting_state_machines| {
                voting_state_machines.with_machine(proposal_id, topic, |machine| {
                    self.process_machine_until_soft_limit(machine, over_soft_message_limit);
                    is_voting_finished = machine.is_voting_finished();
                });
            });

            // This returns an error if we hit the hard limit, which should basically never happen
            // in production, but we need a way out of this loop in the worst case to prevent
            // the canister from being unable to upgrade.
            if let Err(e) = noop_self_call_if_over_instructions(
                SOFT_VOTING_INSTRUCTIONS_LIMIT,
                Some(HARD_VOTING_INSTRUCTIONS_LIMIT),
            )
            .await
            {
                println!(
                    "Error in cast_vote_and_cascade_follow, \
                        voting will be processed in timers: {}",
                    e
                );
                break;
            }
        }
        // We use the time from the beginning of the function to retain the behaviors needed
        // for wait for quiet even when votes can be processed asynchronously.
        self.recompute_proposal_tally(proposal_id, voting_started);
    }
