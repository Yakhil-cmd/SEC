Audit Report

## Title
Permanent State Sync Stall via Persistent HTTP 429 With No Peer Eviction — (`rs/p2p/state_sync_manager/src/ongoing.rs`)

## Summary
In `OngoingStateSync::handle_downloaded_chunk_result`, the `DownloadChunkError::Overloaded` arm (triggered by HTTP 429) only re-queues the failed chunk and never removes the peer from `active_downloads` or decrements `allowed_downloads`. Because the only self-termination condition of the `run` loop is `active_downloads.is_empty()`, a Byzantine peer that persistently returns 429 keeps the loop alive indefinitely while the state sync never progresses, permanently preventing the victim replica from rejoining consensus.

## Finding Description

**Root cause — `Overloaded` has no eviction path.**

`parse_chunk_handler_response` maps `StatusCode::TOO_MANY_REQUESTS` to `DownloadChunkError::Overloaded`: [1](#0-0) 

In `handle_downloaded_chunk_result`, the `Overloaded | Timeout | Cancelled` arm only calls `chunks_to_download.download_failed(chunk_id)`. It never calls `active_downloads.remove(&peer_id)` and never decrements `allowed_downloads`: [2](#0-1) 

Contrast with `NoContent` and `RequestError`, which both remove the peer and decrement the budget: [3](#0-2) 

**Loop never exits.**

The only self-termination condition (besides external cancellation) is `active_downloads.is_empty()`. Since `Overloaded` never removes the peer, this condition is never satisfied when the Byzantine peer is the only advertiser: [4](#0-3) 

**Cycle repeats indefinitely.**

After each 429 response, `active_downloads[peer]` is decremented to 0 via `saturating_sub`, the chunk is re-queued, and `spawn_chunk_downloads` immediately re-selects the same peer (weight = `max - 0 + 1`, the highest possible) for the next download: [5](#0-4) [6](#0-5) 

**No higher-level timeout in `StateSyncManager`.**

`StateSyncManager::run` in `lib.rs` only cleans up an ongoing sync when `ongoing.shutdown.completed()` (inner loop exited) or `cancel_if_running` returns true (triggered by a newer advert for a different artifact). If the Byzantine peer is the only advertiser of the target state, neither condition is ever triggered: [7](#0-6) 

**Existing test confirms the design intent.**

`test_cancel_if_running` uses a mock transport that always returns `StatusCode::TOO_MANY_REQUESTS` and only asserts that the sync can be externally cancelled — it does not assert peer eviction. This confirms the current design treats `Overloaded` as "retry forever": [8](#0-7) 

## Impact Explanation
A lagging replica performing state sync is permanently stalled as long as the Byzantine peer is the only (or first and sole) advertiser of the target state. The replica cannot complete state sync, cannot rejoin consensus, and is effectively removed from the subnet's active validator set. This reduces the subnet's effective fault tolerance — a subnet tolerating `f` Byzantine nodes now tolerates `f-1` after the attack. This matches the **High** impact class: *Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS* ($2,000–$10,000).

## Likelihood Explanation
Any node operator running a malicious replica can implement this by returning HTTP 429 for all chunk requests. No threshold corruption, key compromise, or privileged access is required. The attacker must be the peer that first (or exclusively) advertises the target state to the victim replica — a realistic condition when the victim is catching up and the Byzantine node is the first to send an advert. The attack is repeatable across every state sync attempt by the victim.

## Recommendation
In the `Overloaded` arm of `handle_downloaded_chunk_result`, maintain a per-peer consecutive-overload counter. After a configurable threshold (e.g., `N` consecutive `Overloaded` responses), treat the peer identically to `RequestError`: call `active_downloads.remove(&peer_id)` and decrement `allowed_downloads` by `PARALLEL_CHUNK_DOWNLOADS`. Alternatively, apply exponential backoff that effectively starves the peer of download slots over time, preventing it from monopolizing the download budget.

## Proof of Concept
Construct an `OngoingStateSync` with a single mock peer whose transport always returns `StatusCode::TOO_MANY_REQUESTS` (exactly as `test_cancel_if_running` does). After `N * PARALLEL_CHUNK_DOWNLOADS` download cycles, assert either: (a) the peer is absent from `active_downloads`, or (b) the sync has self-terminated. Under the current code, neither assertion holds — the loop runs indefinitely until the external cancellation token fires, which the existing test already demonstrates implicitly.
