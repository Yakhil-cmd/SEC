### Title
Silent Precompile Failure After `_burn` Permanently Freezes Bridged ERC-20 Funds — (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first burning the caller's ERC-20 tokens and then invoking the exit precompile via inline assembly. The assembly `call` return value is **never checked**. If the precompile sub-call fails for any reason (most concretely: insufficient gas due to EIP-150 gas retention), the burn is irrevocably committed while the corresponding NEP-141 tokens remain permanently locked inside the Aurora contract. This is a direct analog to the WERC721 `totalSupply` accounting bug: a supply counter decreases without the underlying asset actually moving.

---

### Finding Description

In `EvmErc20.sol`, `withdrawToNear` executes:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // (1) burn committed

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // (2) res is captured but NEVER checked — no revert if res == 0
    }
}

The identical pattern appears in `withdrawToEthereum` (calling the `exitToEthereum` precompile at `0xb0bd02f6...`): [2](#0-1)

**Why the precompile sub-call can fail while the parent call succeeds:**

EIP-150 mandates that a `call` opcode retains at least `gas / 64` in the parent frame. When `call(gas(), precompile, ...)` is executed, the parent retains `G/64` gas regardless of what happens inside the sub-call. If the precompile returns `ExitError::OutOfGas` (because the forwarded gas is below `costs::EXIT_TO_NEAR_GAS` or `costs::EXIT_TO_ETHEREUM_GAS`), the sub-call reverts, the parent continues with its retained `G/64` gas, and the function returns normally — with `_burn` already committed.

The precompile enforces its gas requirement explicitly:

```rust
fn required_gas(_input: &[u8]) -> Result<EthGas, ExitError> {
    Ok(costs::EXIT_TO_NEAR_GAS)
}
// ...
if let Some(target_gas) = target_gas && required_gas > target_gas {
    return Err(ExitError::OutOfGas);
}
``` [5](#0-4) 

Additional failure modes that also leave `res == 0` unchecked:
- `ERR_INVALID_IN_STATIC` — precompile called in a static context
- `ERR_INVALID_IN_DELEGATE` — precompile called via `delegatecall`
- `get_nep141_from_erc20` lookup failure (ERC-20 not registered)
- `ERR_ETH_ATTACHED_FOR_ERC20_EXIT` — non-zero `apparent_value` [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the precompile sub-call fails silently:
- The ERC-20 `totalSupply` is permanently reduced (tokens burned from the user's balance).
- The corresponding NEP-141 tokens remain locked inside the Aurora engine contract with no recovery path.
- There is no admin function, no refund mechanism, and no retry: the NEP-141 balance is stranded forever.

This is structurally identical to the WERC721 bug: a supply/balance counter decreases without the underlying asset actually exiting the wrapper/connector.

---

### Likelihood Explanation

**High.** The gas-manipulation path requires no special privilege. Any ERC-20 holder can call `withdrawToNear` or `withdrawToEthereum`. A transaction submitted with a gas limit carefully chosen so that the remaining gas at the `call` site is below `EXIT_TO_NEAR_GAS` (or `EXIT_TO_ETHEREUM_GAS`) will trigger the silent failure. This can happen accidentally (users underestimating gas) or deliberately (an attacker griefing a specific address by front-running with a gas-exhausting transaction, or a user self-triggering to test behavior). The pattern affects every ERC-20 token bridged via `EvmErc20` or `EvmErc20V2`.

---

### Recommendation

Add an explicit revert on precompile call failure in both `withdrawToNear` and `withdrawToEthereum` in both contract versions:

```solidity
assembly {
    let res := call(gas(), EXIT_TO_NEAR_PRECOMPILE, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the precompile sub-call fails for any reason, the entire transaction reverts, rolling back the `_burn` and leaving the user's balance intact.

---

### Proof of Concept

1. Deploy an ERC-20 token bridged via `EvmErc20` (or `EvmErc20V2`) on Aurora.
2. Mint tokens to `alice`.
3. `alice` calls `withdrawToNear(recipient, amount)` with a gas limit `G` such that the gas remaining at the `call` opcode is less than `EXIT_TO_NEAR_GAS` but `G/64` is sufficient to return from the function.
4. Observe:
   - `balanceOf(alice)` decreased by `amount`.
   - `totalSupply()` decreased by `amount`.
   - The NEP-141 balance of `recipient` on NEAR is **unchanged**.
   - The NEP-141 balance held by the Aurora contract is **unchanged** (tokens locked forever).
5. No revert occurred; the transaction receipt shows success.

The accounting invariant `ERC-20 totalSupply == NEP-141 balance held by Aurora` is broken: `totalSupply` decreased without the NEP-141 tokens moving — the exact analog of the WERC721 `totalSupply` depletion without NFT release.

        Ok(costs::EXIT_TO_NEAR_GAS)

    #[allow(clippy::too_many_lines)]
    fn run(
        &self,
        input: &[u8],
        target_gas: Option<EthGas>,
        context: &Context,
        is_static: bool,
    ) -> EvmPrecompileResult {
        // ETH (base) transfer input format: (85 bytes)
        //  - flag (1 byte)
        //  - refund_address (20 bytes), present if the feature "error_refund" is enabled
        //  - recipient_account_id (max MAX_INPUT_SIZE - 20 - 1 bytes)
        // ERC-20 transfer input format: (124 bytes)
        //  - flag (1 byte)
        //  - refund_address (20 bytes), present if the feature "error_refund" is enabled.
        //  - amount (32 bytes)
        //  - recipient_account_id (max MAX_INPUT_SIZE - 1 - (20) - 32 bytes)
        //  - `:unwrap` suffix in a case of wNEAR (7 bytes)
        let required_gas = Self::required_gas(input)?;

        if let Some(target_gas) = target_gas
            && required_gas > target_gas
        {
            return Err(ExitError::OutOfGas);
        }

        // It's not allowed to call exit precompiles in static mode
        if is_static {
            return Err(ExitError::Other(Cow::from("ERR_INVALID_IN_STATIC")));
        } else if context.address != exit_to_near::ADDRESS.raw() {
            return Err(ExitError::Other(Cow::from("ERR_INVALID_IN_DELEGATE")));
        }
