### Title
Fatal Block Halt via `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` Revert Permanently DoSes All Pending L1→L2 Deposits - (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The `notify_l2_asset_tracker` function unconditionally treats any revert from `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` as a **fatal system error** that terminates block processing entirely. Because L1→L2 (priority) transactions cannot be invalidated or skipped, any L1 transaction whose deposit amount triggers a revert in `L2AssetTracker` will permanently halt the chain's ability to seal new blocks, freezing all subsequent pending L1→L2 deposits in the priority queue with no recovery path.

---

### Finding Description

During L1→L2 transaction processing, the bootloader calls `notify_l2_asset_tracker` up to three times per transaction — once for the value mint (inside the execution frame), once for the operator fee payment, and once for the refund (both outside the execution frame, using `FORMAL_INFINITE` resources). [1](#0-0) 

If any of these calls returns a failed execution result, `notify_l2_asset_tracker` returns an `InternalError`:

```rust
if failed {
    return Err(internal_error!(
        "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
    ).into());
}
```

This `InternalError` is **not** a `RuntimeError::FatalRuntimeError`, so it does not fall into the graceful out-of-native handler in `process_l1_transaction`. Instead it hits the catch-all branch: [2](#0-1) 

```rust
_ => {
    system.finish_global_frame(Some(&rollback_handle))?;
    return Err(e);
}
```

This propagates as `TxError::Internal` to the ZK transaction loop, which immediately terminates block processing: [3](#0-2) 

```rust
Err(TxError::Internal(err)) => {
    system.finish_global_frame(None)?;
    return Err(err);
}
```

The critical asymmetry is that L1→L2 transactions **cannot be invalidated**. The test suite and documentation explicitly acknowledge this: [4](#0-3) 

> "L1 transactions cannot be invalidated. Therefore, the following function makes sure L1 transactions are processable even when some checks that should be performed by the L1 don't hold."

The `amount` values passed to `handleFinalizeBaseTokenBridgingOnL2` are derived from L1-transaction-sender-controlled fields:
- Value-mint amount: `total_deposited - max_fee_commitment` (controlled via `to_mint`, `gas_price`, `gas_limit`)
- Operator fee: `gas_used * gas_price`
- Refund: `total_deposited - pay_to_operator` [5](#0-4) 

The `L2AssetTracker` contract (`handleFinalizeBaseTokenBridgingOnL2`) has state-dependent logic including accumulator arithmetic (`totalSuccessfulDepositsFromL1 += amount`) and asset-registration checks. Any revert condition reachable via a specific `(fromChainId, amount)` pair — including arithmetic overflow in the accumulator, an unregistered asset state, or any future bug — will trigger the fatal path.

Unlike the EigenLayer M-02 scenario where users could re-queue without the malicious strategy, there is **no recovery mechanism** here: the failing L1 transaction stays at the head of the priority queue, and every subsequent block attempt will re-encounter the same fatal error.

---

### Impact Explanation

- **Permanent chain halt**: Block processing terminates on every attempt to include the offending L1 transaction. Since L1 transactions cannot be skipped or invalidated, the chain cannot seal any further blocks.
- **All pending L1→L2 deposits frozen**: Every user who submitted an L1→L2 deposit transaction after the offending one has their funds locked in the L1 bridge contract with no L2 credit and no refund path.
- **No operator workaround**: Unlike M-02 (where re-queuing without the bad strategy was possible), the operator has no mechanism to bypass or skip the failing priority transaction.

---

### Likelihood Explanation

- The `L2AssetTracker` is a real Solidity 0.8+ contract with checked arithmetic. The `totalSuccessfulDepositsFromL1` accumulator is a `uint256` that grows with every successful deposit. An attacker who can observe the current accumulator value and submit an L1 transaction with `to_mint` set to `U256::MAX - current_accumulator` would cause an overflow revert on the next call.
- Additional revert conditions exist in the asset-registration and chain-balance logic of `L2AssetTracker` that may be reachable with crafted `(fromChainId, amount)` inputs.
- The `amount` for all three notification calls is directly or indirectly derived from L1-transaction-sender-controlled fields (`to_mint`, `gas_price`, `gas_limit`), giving an unprivileged L1 sender meaningful control over the inputs.

---

### Recommendation

1. **Degrade gracefully**: Change `notify_l2_asset_tracker` to log the failure and continue rather than returning a fatal `InternalError`. Emit an L2→L1 log recording the accounting discrepancy so it can be reconciled off-chain.
2. **Separate fatal from non-fatal errors**: Distinguish between `L2AssetTracker` reverts (which should be non-fatal, similar to how out-of-gas is handled) and true system invariant violations.
3. **Add a skip/invalidation mechanism for L1 transactions**: Introduce a governance-controlled list of L1 transaction hashes that can be skipped in the priority queue, analogous to the `indicesToSkip` mechanism in EigenLayer's `slashQueuedWithdrawal`.
4. **Cap or validate `amount` before calling `L2AssetTracker`**: Perform a pre-call check that the accumulator will not overflow before invoking `handleFinalizeBaseTokenBridgingOnL2`.

---

### Proof of Concept

1. Read the current value of `L2AssetTracker.interopInfo[BASE_TOKEN_ASSET_ID].totalSuccessfulDepositsFromL1` on L2.
2. Compute `attack_amount = U256::MAX - current_accumulator + 1` (the amount that would overflow the accumulator).
3. Submit an L1→L2 priority transaction with `to_mint = attack_amount + max_fee_commitment` so that `to_transfer = attack_amount`.
4. When the bootloader processes this transaction, `notify_l2_asset_tracker` calls `handleFinalizeBaseTokenBridgingOnL2(l1_chain_id, attack_amount)`, which reverts due to overflow in the Solidity 0.8+ checked arithmetic.
5. `notify_l2_asset_tracker` returns `Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"))`.
6. This propagates as `TxError::Internal` to the ZK tx loop, which calls `return Err(err)`, terminating block processing.
7. Every subsequent block attempt re-encounters the same L1 transaction at the head of the priority queue and fails identically.
8. All L1→L2 deposits submitted after step 3 are permanently frozen. [6](#0-5) [7](#0-6) [8](#0-7)

fn notify_l2_asset_tracker<'a, S: EthereumLikeTypes + 'a, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    system_functions: &mut HooksStorage<S, S::Allocator>,
    memories: RunnerMemoryBuffers<'a>,
    amount: U256,
    l1_chain_id: U256,
    resources: &mut S::Resources,
    tracer: &mut impl Tracer<S>,
    validator: &mut impl TxValidator<S>,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
    S::Metadata: ZkSpecificPricingMetadata
        + BasicMetadata<S::IOTypes, TransactionMetadata = TxLevelMetadata<S::IOTypes>>,
{
    if amount > U256::ZERO || Config::SIMULATION {
        // Encode calldata for handleFinalizeBaseTokenBridgingOnL2(uint256,uint256):
        // selector 0x03117c8c + abi-encoded (fromChainId, amount)
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]);
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution {
                resources_returned,
                result: asset_tracker_result,
            } = BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system,
                system_functions,
                memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,
                inf_ergs.clone(),
                &U256::ZERO,
                true, // should_make_frame - isolate state changes
                tracer,
                validator,
            )?;
            // Overwrite resources inside the closure so that
            // with_infinite_ergs correctly restores ergs afterwards.
            *inf_ergs = resources_returned;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
        })?;

        if failed {
            system_log!(
                system,
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 failed for amount {amount:?}\n"
            );
            // A revert here means the chain's token accounting would be inconsistent.
            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
            )
            .into());
        }
    }
    Ok(())
}
