Audit Report

## Title
Unbounded Concurrent Connection Hold in `call_sync` with No Concurrency Cap — (File: `rs/http_endpoints/public/src/call/call_sync.rs`)

## Summary

The synchronous call handler (`/api/v3/canister/.../call`, `/api/v4/canister/.../call`, `/api/v4/subnet/.../call`) implemented in `call_sync` holds each HTTP connection open while awaiting `wait_for_certification()` with no concurrency limit on simultaneous in-flight requests. The configuration explicitly caps `/api/v2/call` at `max_call_concurrent_requests: 50`, but no equivalent cap exists for the v3/v4 sync endpoints. The `IngressWatcher` backing these endpoints stores per-message state in an unbounded `HashMap` and `JoinMap`, bounded only indirectly by the ingress pool throttle. An unprivileged attacker with a valid identity can flood the sync endpoint with valid ingress messages, exhausting HTTP connection resources and growing `IngressWatcher` internal state up to the ingress pool limit.

## Finding Description

**Root cause 1 — No concurrency cap on `call_sync`:**

`call_sync` holds the HTTP connection open by awaiting `wait_for_certification()` with a timeout of `ingress_message_certificate_timeout_seconds` (default 10 seconds):

```rust
// rs/http_endpoints/public/src/call/call_sync.rs L310-327
match certification_subscriber
    .wait_for_certification()
    .timeout(Duration::from_secs(
        ingress_message_certificate_timeout_seconds,
    ))
    .await
```

`SynchronousCallHandlerState` contains no semaphore or concurrency guard. Compare to the async handler, which has an explicit `ingress_tracking_semaphore: Arc<Semaphore>` initialized at `MAX_CONCURRENT_TRACKING_TASKS = 10_000`:

```rust
// rs/http_endpoints/public/src/call/call_async.rs L33, L39, L50
const MAX_CONCURRENT_TRACKING_TASKS: usize = 10_000;
ingress_tracking_semaphore: Arc<Semaphore>,
Arc::new(Semaphore::new(MAX_CONCURRENT_TRACKING_TASKS))
```

The HTTP handler configuration (`rs/config/src/http_handler.rs`) explicitly caps `/api/v2/call` at `max_call_concurrent_requests: 50` but defines no equivalent field for the v3/v4 sync endpoints:

```rust
// rs/config/src/http_handler.rs L54-55, L85
/// Serving at most `max_call_concurrent_requests` requests concurrently for endpoint `/api/v2/call`.
pub max_call_concurrent_requests: usize,
// default: 50
```

No `GlobalConcurrencyLimitLayer` or semaphore is applied to the sync call router in `call_sync.rs`.

**Root cause 2 — Unbounded `IngressWatcher` internal state:**

`IngressWatcher` stores per-message state in a plain `HashMap` and `JoinMap` with no size cap:

```rust
// rs/http_endpoints/public/src/call/ingress_watcher.rs L128-132
cancellations: JoinMap<MessageId, ()>,
message_statuses: HashMap<MessageId, (MessageExecutionStatus, Arc<Notify>)>,
completed_execution_heights: BTreeMap<Height, HashSet<MessageId>>,
```

Each unique message subscribed via `handle_ingress_message` inserts into `message_statuses` and spawns a task in `cancellations` with no cap:

```rust
// rs/http_endpoints/public/src/call/ingress_watcher.rs L289-302
Entry::Vacant(vacant_entry) => {
    self.cancellations.spawn_on(
        message.clone(),
        cancellation_token.cancelled_owned(),
        &self.rt_handle,
    );
    let certification_notifier = Arc::new(tokio::sync::Notify::new());
    vacant_entry.insert((MessageExecutionStatus::InProgress, certification_notifier.clone()));
    Ok(certification_notifier)
}
```

The subscription channel is bounded at 1000 (`INGRESS_WATCHER_CHANNEL_SIZE`), but this only limits the pending queue; once processed, entries live in the unbounded `HashMap` until certified or the subscriber drops.

**Existing guard and its insufficiency:**

`validate_ingress_message` checks `ingress_throttler.exceeds_threshold()` before creating a subscription:

```rust
// rs/http_endpoints/public/src/call.rs L230-236
let ingress_pool_is_full = ingress_throttler.read().unwrap().exceeds_threshold();
if ingress_pool_is_full {
    Err(HttpError { status: StatusCode::SERVICE_UNAVAILABLE, ... })?;
}
```

This provides a natural upper bound on `message_statuses` size (roughly the ingress pool capacity, ~10,000 messages). However, this does not cap the number of simultaneously open HTTP connections — it only prevents new subscriptions once the pool is full. An attacker who fills the pool holds all previously accepted connections open for the full 10-second timeout, exhausting file descriptors and connection-state memory. The ingress pool is designed to limit message throughput, not to protect HTTP connection resources.

**Exploit flow:**

1. Attacker generates N valid ingress messages (unique nonce/expiry per message, valid signature).
2. Each request passes `validate_ingress_message` (pool not yet full), subscribes to `IngressWatcher`, and submits to the pool.
3. Each request blocks at `wait_for_certification()` for up to 10 seconds, holding the TCP connection open.
4. Once the pool fills, new requests are rejected with 503, but all previously accepted connections remain open.
5. `message_statuses` and `cancellations` grow to pool capacity; the single-threaded `select!` event loop processes one event per iteration with a large `cancellations` JoinMap, degrading certification notification latency for all legitimate subscribers.

## Impact Explanation

**High — Application/platform-level DoS on the replica's public API.**

An attacker can hold up to ~10,000 simultaneous TCP connections open for 10 seconds each, consuming file descriptors and per-connection memory on the replica. The `IngressWatcher` event loop, running as a single-threaded `select!` loop, degrades under a large `cancellations` JoinMap, delaying certification notifications for all legitimate users. This constitutes an application-level DoS on the replica's public API — not raw volumetric DDoS — exploiting a specific design gap (missing concurrency cap on the sync endpoint). The impact matches: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

The `/api/v3/canister/.../call` and `/api/v4/canister/.../call` endpoints are publicly reachable by any unprivileged sender. The attacker needs only a valid identity (key pair) and the ability to generate unique ingress messages (varying nonce or expiry). No privileged access, key compromise, or threshold corruption is required. The ingress pool (~10,000 messages) provides friction but not prevention — a single attacker with a scripted client can fill the pool and hold connections open within seconds. The attack is repeatable: after the timeout expires and connections close, the attacker can immediately repeat.

## Recommendation

1. **Add a `GlobalConcurrencyLimitLayer` to the sync call router** in `new_router` in `call_sync.rs`, analogous to the existing `max_call_concurrent_requests` applied to `/api/v2/call`. Add a corresponding config field (e.g., `max_sync_call_concurrent_requests`) with a conservative default (e.g., 100–500).
2. **Cap `IngressWatcher.message_statuses`**: Enforce a maximum size. When the cap is reached, return `SubscriptionError` so `call_sync` falls back to `202 Accepted`.
3. **Reduce `ingress_message_certificate_timeout_seconds`**: A shorter default reduces the window during which connections are held open.
