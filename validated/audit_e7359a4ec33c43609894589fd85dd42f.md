### Title
`gas_per_pubdata_limit` User-Provided Pubdata Price Cap Is Silently Ignored for L2 ZK Transactions — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

ZKsync OS accepts a `gas_per_pubdata_limit` field in ABI-encoded (EIP-712 / ZK-type) L2 transactions, which is intended to let users cap the maximum gas they are willing to pay per byte of pubdata. However, this field is never read or enforced during L2 ZK transaction validation. The actual pubdata cost rate is computed exclusively from the block-level `pubdata_price` and `native_price` oracle values, making the user's protection parameter void. This is the direct analog of the Ajna `limitIndex_` bug: a user-provided limit argument is accepted but silently ignored in a specific code path.

---

### Finding Description

The `Transaction` wrapper exposes `gas_per_pubdata_limit()` for ABI-encoded transactions:

```rust
pub fn gas_per_pubdata_limit(&self) -> U256 {
    match self {
        Self::Rlp(_) => U256::ZERO,
        Self::Abi(tx) => U256::from(tx.gas_per_pubdata_limit.read()),
    }
}
``` [1](#0-0) 

For **L1→L2 transactions**, `gas_per_pubdata_limit` is read and used as the actual pubdata charging rate:

```rust
let gas_per_pubdata = transaction.gas_per_pubdata_limit.read();
// ...
let native_per_pubdata = (gas_per_pubdata as u64).checked_mul(native_per_gas)...;
``` [2](#0-1) [3](#0-2) 

For **L2 ZK transactions**, however, `validate_and_compute_fee_for_transaction` in `validation_impl.rs` never calls `transaction.gas_per_pubdata_limit()`. Instead, `native_per_pubdata` is derived solely from block-level oracle values:

```rust
let pubdata_price = system.get_pubdata_price();
let native_price = system.get_native_price();
// ...
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [4](#0-3) 

There is no check of the form `require!(gas_per_pubdata_limit >= actual_pubdata_cost_in_gas)` anywhere in the L2 ZK validation path. The field is structurally present, signed over by the user, but never enforced.

---

### Impact Explanation

A user submits a ZK L2 transaction with `gas_per_pubdata_limit = X` (e.g., 100 gas/byte), intending to protect against pubdata price spikes. If the block's pubdata price is equivalent to Y gas/byte where Y > X, the transaction is still executed and the user is charged at rate Y. Two concrete harms follow:

1. **Silent overpayment**: If the user's `gas_limit` is large enough, the transaction succeeds but the user pays more for pubdata than they consented to.
2. **Unexpected revert with full gas burn**: If the user sized their `gas_limit` based on the expected pubdata cost at rate X, the higher rate Y exhausts the gas budget mid-execution, causing a revert. The user loses the full gas prepayment (the post-execution pubdata check burns all remaining ergs on failure). [5](#0-4) 

---

### Likelihood Explanation

- **No prerequisites**: Any ZK L2 transaction with a non-zero `gas_per_pubdata_limit` is affected.
- **Trigger condition**: The block's pubdata price (driven by L1 gas price) exceeds the user's stated limit. L1 gas price spikes are common and unpredictable.
- **Attacker-controlled entry**: An unprivileged user submitting a standard ZK transaction is the victim. No privileged role is required. The sequencer/oracle controls the block pubdata price, which is the necessary condition.

---

### Recommendation

In `validate_and_compute_fee_for_transaction` (`validation_impl.rs`), after computing the actual pubdata cost in gas units, add a check against the user-provided `gas_per_pubdata_limit` for non-service ZK transactions:

```rust
// After computing native_per_pubdata from block context:
let gas_per_pubdata_limit = transaction.gas_per_pubdata_limit();
if !gas_per_pubdata_limit.is_zero() && !transaction.is_service() {
    // actual cost in gas per pubdata byte ≈ pubdata_price / gas_price
    let actual_gas_per_pubdata = pubdata_price
        .checked_div(gas_price)
        .unwrap_or(U256::MAX);
    require!(
        actual_gas_per_pubdata <= gas_per_pubdata_limit,
        InvalidTransaction::GasPerPubdataLimitExceeded,
        system
    )?;
}
```

This mirrors the fix applied in Ajna: the user-provided limit argument must be enforced in all relevant execution paths, not just in one branch (L1 transactions).

---

### Proof of Concept

1. User constructs a ZK EIP-712 transaction with `gas_per_pubdata_limit = 50` (50 gas/pubdata byte) and `gas_limit = 100_000`.
2. Block is produced with `pubdata_price = 500_000` and `native_price = 1_000`, making the effective pubdata cost `500` gas/byte (10× the user's limit).
3. `validate_and_compute_fee_for_transaction` reads `pubdata_price` and `native_price` from the block context and computes `native_per_pubdata` without ever consulting `transaction.gas_per_pubdata_limit()`.
4. The transaction is accepted and executed. The user's `gas_per_pubdata_limit = 50` is never compared against the actual rate of `500`.
5. If the transaction writes to storage (generating pubdata), the post-execution pubdata check at line 880 of `zk/mod.rs` finds insufficient gas, reverts the transaction, and burns the full `gas_limit` — all while the user's stated protection was silently ignored. [4](#0-3) [1](#0-0) [6](#0-5)
