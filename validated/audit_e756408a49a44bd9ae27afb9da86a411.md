Audit Report

## Title
Cross-Subaccount Allowance Disclosure via Incomplete Break Condition in `get_allowances` — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

## Summary
The `get_allowances` function performs a `StableBTreeMap` range scan starting at a caller-supplied `Account` (owner + subaccount) but breaks only when the iterated entry's **owner** differs from the queried owner. Because `Account` ordering sorts by owner first and then by subaccount, the scan silently continues across every subaccount of the same owner that sorts after the requested one. Combined with the absence of any caller-authorization check in `icrc103_get_allowances`, any unprivileged principal can enumerate the full set of ICRC-2 allowances — amounts, spenders, and expiry timestamps — for every subaccount of any victim principal.

## Finding Description

**Entrypoint** — `icrc103_get_allowances` (`rs/ledger_suite/icrc1/ledger/src/main.rs`, L1214–1232):

```rust
#[query]
fn icrc103_get_allowances(arg: GetAllowancesArgs) -> Result<Allowances, GetAllowancesError> {
    let from_account = arg.from_account.unwrap_or_else(|| Account {
        owner: ic_cdk::api::msg_caller(),
        subaccount: None,
    });
    // No check: from_account.owner == msg_caller()
    Ok(get_allowances(from_account, arg.prev_spender, max_results, ic_cdk::api::time()))
}
```

Any caller may supply an arbitrary `from_account`; there is no guard that `from_account.owner` equals `msg_caller()`.

**Root cause** — break condition in `get_allowances` (`rs/ledger_suite/icrc1/ledger/src/lib.rs`, L1204–1206):

```rust
if account_spender.account.owner != from.owner {
    break;
}
```

This checks only the `owner` field. It does not check `subaccount`.

**Why the scan crosses subaccounts** — `Account::Ord` (`packages/icrc-ledger-types/src/icrc1/account.rs`, L54–60) sorts by `owner` first, then by `effective_subaccount()`. A range scan starting at `{owner: X, subaccount: [0x01;32]}` therefore advances through:

```
{owner:X, sub:[0x01;32]} → spender_A   ← returned (correct)
{owner:X, sub:[0x01;32]} → spender_B   ← returned (correct)
{owner:X, sub:[0x02;32]} → spender_C   ← returned (BUG: wrong subaccount)
{owner:X, sub:[0x03;32]} → spender_D   ← returned (BUG: wrong subaccount)
{owner:Y, sub:...}                      ← break fires here
```

The break fires only on owner mismatch, so every entry for `owner: X` with any subaccount `>= [0x01;32]` is included in the response.

**Existing guards reviewed and found insufficient**: The only guard present is the `spender.is_some() && account_spender == start_account_spender` skip at L1198–1200, which handles pagination cursor exclusion only. There is no subaccount equality check anywhere in the loop.

## Impact Explanation

An unprivileged attacker can enumerate, for any victim principal:
- Which subaccounts have active ICRC-2 approvals
- The identity of every approved spender per subaccount
- The exact allowance amounts and expiration timestamps

This constitutes unauthorized disclosure of financial relationships across subaccount isolation boundaries in the ICRC-1/ICRC-2 ledger — a production in-scope financial integration component. This matches the allowed impact: **High ($2,000–$10,000) — Significant ledger security impact with concrete user harm**, specifically the exposure of sensitive financial data that subaccount isolation is designed to protect.

## Likelihood Explanation

- The endpoint is a public `#[query]` callable by any principal with no fee and no authentication.
- No privileged role, key, or social engineering is required.
- The exploit is deterministic and locally reproducible with PocketIC or a local replica.
- The only precondition is that the victim principal has ICRC-2 approvals from more than one subaccount.

## Recommendation

Change the break condition from an owner-only check to a full `Account` equality check. `Account::PartialEq` already compares both `owner` and `effective_subaccount()`:

```rust
// Before (buggy):
if account_spender.account.owner != from.owner {
    break;
}

// After (correct):
if account_spender.account != from {
    break;
}
```

Additionally, consider adding a caller-authorization check in `icrc103_get_allowances` so that only `from_account.owner == msg_caller()` (or a delegated principal) may list allowances, using the `AccessDenied` error variant already defined in `packages/icrc-ledger-types/src/icrc103/get_allowances.rs`.

## Proof of Concept

```rust
// Setup: victim principal X has approvals from two subaccounts
icrc2_approve(caller=X_sub1, spender=S1, amount=100);  // sub1 = [0x01;32]
icrc2_approve(caller=X_sub2, spender=S2, amount=200);  // sub2 = [0x02;32]

// Attacker (any principal) queries:
let result = icrc103_get_allowances(GetAllowancesArgs {
    from_account: Some(Account { owner: X, subaccount: Some([0x01; 32]) }),
    prev_spender: None,
    take: None,
});

// Expected (correct): only [{from: X/sub1, to: S1, amount: 100}]
// Actual (buggy):     [{from: X/sub1, to: S1, amount: 100},
//                      {from: X/sub2, to: S2, amount: 200}]
assert_eq!(result.unwrap().len(), 2);  // passes — demonstrates the leak
```

This is reproducible as a PocketIC integration test or a unit test against the `get_allowances` function directly with a mock `StableBTreeMap`.
