Audit Report

## Title
Cache Stampede / Thundering-Herd in `subnet_read_state_cache_middleware` Allows Upstream Replica Connection Pool Exhaustion — (`rs/boundary_node/ic_boundary/src/http/middleware/subnet_read_state_cache.rs`)

## Summary

`subnet_read_state_cache_middleware` uses a non-atomic `cache.get()` + `cache.insert()` pattern with no in-flight deduplication. N concurrent requests sharing the same `(subnet_id, paths)` cache key that all arrive before any response is stored will all observe a cache miss and all be independently forwarded to the upstream replica, defeating the cache's protective purpose and exhausting the replica's connection pool.

## Finding Description

The middleware at lines 122–143 of `subnet_read_state_cache.rs` performs a two-step, non-atomic cache interaction:

1. **Line 122** — `state.cache.get(&cache_key)`: non-blocking read from `moka::sync::Cache`. On miss, execution falls through.
2. **Line 130** — `next.run(request).await`: the request is forwarded to the upstream replica.
3. **Line 143** — `state.cache.insert(cache_key, cached.clone())`: the response is stored only after the full upstream round-trip completes.

Between steps 1 and 3 there is a window equal to the entire upstream latency during which the cache entry does not exist. Any concurrent request calling `cache.get()` during this window also sees a miss and independently calls `next.run()`. There is no in-flight deduplication, no mutex, and no `get_with`/compute-once primitive.

`moka::sync::Cache::get_with` — which serializes concurrent initializations for the same key — is already used in the same codebase at `handlers.rs` line 99 (`state.ip_cache.get_with((ip, canister_id), || ...)`), confirming the developers are aware of the pattern.

The `subnet_layers` stack (lines 1033–1040 of `core.rs`) wraps subnet read_state routes but does **not** include the IP or subnet rate-limiting guards that are applied to `canister_call_routes` (lines 860–879 of `core.rs`). There is therefore no secondary mitigation to limit the number of concurrent in-flight upstream requests per attacker IP.

The default TTL is 30 seconds (`cli.rs` lines 351–353). At the start of each 30-second window, an attacker can open a new stampede by sending a burst of concurrent requests timed to arrive before the first response is cached.

## Impact Explanation

Every boundary-node worker task handling one of the N concurrent requests independently opens a connection to the upstream replica and awaits a full response. This exhausts the upstream replica's connection pool or triggers its internal rate limiter, degrading or denying service for all users of that subnet. The cache is specifically designed to shield replicas from repeated identical requests; the stampede nullifies that shield entirely. Because subnet read_state paths (`canister_ranges`, `subnet`) are well-known and identical across all boundary-node clients, a single attacker can target the highest-traffic cache key. This constitutes an **application/platform-level DoS against subnet availability not based on raw volumetric DDoS** — matching the High ($2,000–$10,000) impact tier.

## Likelihood Explanation

The endpoint is fully public and unauthenticated. The attack requires only sending N HTTP requests concurrently to the same well-known path — trivially achievable with any HTTP client library. No privileged access, key material, governance majority, or network-level attack is required. The absence of per-IP rate limiting on subnet read_state routes removes the primary mitigation that exists for call routes. The attack is repeatable every TTL window (default 30 seconds).

## Recommendation

Replace the `get` + `insert` pattern with `moka`'s compute-once API so that only one in-flight upstream request is made per cache key at a time:

```rust
// Instead of:
if let Some(cached) = state.cache.get(&cache_key) { ... }
// ... forward to upstream ...
state.cache.insert(cache_key, cached.clone());

// Use moka's async get_with (or entry().or_insert_with_async()):
let cached = state.cache.get_with_by_ref(&cache_key, async {
    // forward to upstream exactly once per key
    ...
}).await;
```

Additionally, add per-IP rate limiting to subnet read_state routes, consistent with the protection already applied to `canister_call_routes`.
