Audit Report

## Title
`API_BOUNDARY_NODE_PRINCIPALS` Lost on Upgrade Causes Confidential Rate-Limit Rule Bypass Window — (`rs/boundary_node/rate_limits/canister/storage.rs`)

## Summary
`API_BOUNDARY_NODE_PRINCIPALS` is stored in a plain heap-resident `RefCell<HashSet<Principal>>` with no stable memory backing. On every canister upgrade, this set is reset to empty and is only repopulated after `registry_polling_period_secs` seconds when the periodic timer fires. During this window, all API boundary nodes are demoted to `RestrictedRead`, causing confidential (not-yet-disclosed) rate-limit rules to be redacted from their `get_config` responses and update-style `get_config` calls to be rejected outright by `inspect_message`.

## Finding Description
In `storage.rs` L21–24, four persistent stores (`CONFIGS`, `RULES`, `INCIDENTS`, `AUTHORIZED_PRINCIPAL`) are backed by `StableBTreeMap` over `MemoryId`-allocated virtual memory and survive upgrades. [1](#0-0) 

However, `API_BOUNDARY_NODE_PRINCIPALS` at L149 is a plain `RefCell<HashSet<Principal>>` with no stable memory slot: [2](#0-1) 

`post_upgrade` at L100–103 simply delegates to `init`: [3](#0-2) 

`init` at L85–88 calls `periodically_poll_api_boundary_nodes`, which registers a `set_timer_interval` — it does **not** fire immediately: [4](#0-3) [5](#0-4) 

Until the first interval fires, `is_api_boundary_node_principal` returns `false` for every caller: [6](#0-5) 

`AccessLevelResolver::get_access_level` uses this check to grant `FullRead`; with an empty set every boundary node falls through to `RestrictedRead`: [7](#0-6) 

`inspect_message` uses the same check to gate update-style `get_config` calls, trapping boundary node callers during the window: [8](#0-7) 

The existing integration test upgrades the canister but never asserts boundary node access immediately post-upgrade before the timer fires, leaving the gap untested: [9](#0-8) 

## Impact Explanation
During the post-upgrade window, API boundary nodes receive redacted `get_config` responses (confidential, not-yet-disclosed rate-limit rules are hidden) and their update-style `get_config` calls are rejected. Boundary nodes operating on incomplete rule sets cannot enforce confidential rate-limit rules, allowing traffic that should be rate-limited to pass through unimpeded. This is a concrete **boundary/API security impact** matching the High tier: "Significant boundary/API infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
NNS upgrade proposals are public and their execution time is observable on-chain. Any unprivileged attacker can monitor governance proposals, determine the exact upgrade execution block, and time a burst of traffic to coincide with the window. No special privileges are required. The window is nonzero on every upgrade and is bounded only by `registry_polling_period_secs` (configurable at upgrade time via `InitArg`).

## Recommendation
Persist `API_BOUNDARY_NODE_PRINCIPALS` in stable memory using a `StableBTreeMap` with a new `MemoryId` (e.g., `MemoryId::new(4)`), mirroring the pattern used for `AUTHORIZED_PRINCIPAL` at L137–141 of `storage.rs`. [10](#0-9) 

Alternatively, in `post_upgrade` (or at the end of `init` when called from `post_upgrade`), schedule a one-shot timer with zero delay via `ic_cdk_timers::set_timer(Duration::ZERO, ...)` to immediately trigger a registry poll before the interval timer is registered, reducing the window to a single async round-trip.

## Proof of Concept
1. Install the canister with a known API boundary node principal `BN` and `registry_polling_period_secs = 60`.
2. Advance PocketIC time by 60+ seconds so the first timer fires and `API_BOUNDARY_NODE_PRINCIPALS` is populated with `BN`.
3. Trigger a canister upgrade via `pocket_ic.upgrade_canister(...)`.
4. Immediately after upgrade (without advancing time), call `get_config` as `BN` using an update call.
5. Observe that `inspect_message` traps with `"message_inspection_failed"` — `BN` is no longer recognized.
6. Call `get_config` as `BN` using a query call; observe the response has `is_redacted: true` and confidential rules are `None`, confirming `RestrictedRead` access instead of `FullRead`.
