### Title
Integer Division Truncation of `native_per_pubdata` to Zero Causes Free Pubdata Generation - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary

When the operator-set `pubdata_price` is less than `native_price`, the integer division `pubdata_price.wrapping_div(native_price)` silently truncates to `0`. This sets `native_per_pubdata = 0`, causing every subsequent pubdata-cost check to charge zero native resources for all pubdata generated by the transaction. Any unprivileged user can then generate pubdata up to the block-level `pubdata_limit` without consuming any native resource budget for it.

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the per-pubdata-byte native cost is computed as:

```rust
// We checked native_price != 0 above
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is Rust's standard integer (floor) division. When `pubdata_price < native_price` — for example `pubdata_price = 500`, `native_price = 1000` — the result is `0`. No error is returned; the transaction proceeds with `native_per_pubdata = 0`.

This value is then threaded through the entire pubdata-charging pipeline:

1. **Intrinsic pubdata overhead** in `create_resources_for_tx`:
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata); // = 0
   ```
   No native budget is withheld for intrinsic pubdata.

2. **Post-execution pubdata check** in `get_resources_to_charge_for_pubdata`:
   ```rust
   let native = current_pubdata_spent.checked_mul(native_per_pubdata); // = 0
   ```
   The native cost of all pubdata generated during execution is zero.

3. **Sufficiency check** in `check_enough_resources_for_pubdata`:
   ```rust
   let enough = resources.has_enough(&resources_for_pubdata); // always true (0 needed)
   ```
   The transaction is never reverted for pubdata costs.

The same truncation is present in the public API helper `api/src/helpers.rs`:
```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

### Impact Explanation

When `native_per_pubdata = 0`, a transaction's entire native resource budget is available for computation alone. The transaction can write to as many storage slots as its gas limit allows (up to the block-level `pubdata_limit`) without any of those writes consuming native resources. The protocol publishes this pubdata to L1 at real cost, but the user paid zero native resources for it. This breaks the double-accounting invariant: the native resource limit is supposed to cover both computation and pubdata, but pubdata is silently excluded when the truncation occurs.

**Impact: High** — the protocol absorbs the L1 data-availability cost for pubdata that users should have paid for in native resources. A user can craft a transaction that maximally spams storage writes, generating the maximum allowed pubdata per block at no native resource cost beyond the EVM gas for the SSTORE opcodes themselves.

### Likelihood Explanation

**Likelihood: Medium** — the condition `pubdata_price < native_price` is a realistic operator configuration. Both values are U256 quantities in base-token units but measure different things (cost per pubdata byte vs. cost per native cycle). There is no protocol-level enforcement that `pubdata_price >= native_price`. The default test configuration even sets `pubdata_price = 0`, confirming the code path is exercised. Any block where the operator sets these values such that `pubdata_price < native_price` exposes every transaction in that block to this undercharge.

### Recommendation

Replace `wrapping_div` with `div_ceil` (already used for `native_per_gas`) so that a non-zero `pubdata_price` always produces at least `1` native unit per pubdata byte:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs`. Additionally, consider adding a validation that `native_per_pubdata > 0` whenever `pubdata_price > 0`, to make the invariant explicit.

### Proof of Concept

**Setup:** operator sets `native_price = 1000`, `pubdata_price = 999`, `eip1559_basefee = 1000`.

**Step 1 — truncation occurs:**
```
native_per_pubdata = 999 / 1000 = 0  (integer division)
```

**Step 2 — intrinsic pubdata overhead is zero:**
```rust
// create_resources_for_tx, gas_helpers.rs:352
intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0
// native_limit is NOT reduced for intrinsic pubdata
```

**Step 3 — user submits a transaction that writes to N storage slots:**
Each SSTORE generates ~66 bytes of pubdata. With `native_per_pubdata = 0`:
```rust
// get_resources_to_charge_for_pubdata, gas_helpers.rs:430-431
let native = current_pubdata_spent.checked_mul(0) = Some(0)
```

**Step 4 — pubdata sufficiency check always passes:**
```rust
// check_enough_resources_for_pubdata, gas_helpers.rs:454
resources.has_enough(&Resources::from_native(0)) == true  // always
```

**Result:** The transaction succeeds, all N storage writes are committed and published to L1, but the user paid zero native resources for the pubdata. The full native budget was available for EVM computation. The protocol bears the L1 data-availability cost.
