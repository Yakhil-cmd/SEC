### Title
Unverified `native_price` / `pubdata_price` Oracle Values Enable Forward/Proving Divergence in Resource Accounting - (`basic_bootloader/src/bootloader/block_flow/zk/metadata_op.rs`)

### Summary

`BlockMetadataFromOracle` fields `native_price` and `pubdata_price` are fetched from the oracle and used directly for all L2 transaction native-resource accounting, but neither field is committed to in the ZK public inputs (`ChainStateCommitment` or `BatchOutput`). A malicious prover can supply different pricing values during proving than the sequencer used during forward execution, causing the proven state transition to diverge from the actually-executed state — transactions that succeeded can be proven as reverted, and vice versa.

### Finding Description

In `metadata_op.rs`, the entire `BlockMetadataFromOracle` blob — including `native_price` and `pubdata_price` — is accepted from the oracle with no validation beyond gas-limit bounds:

```rust
let block_level_metadata: BlockMetadataFromOracle =
    oracle.query_with_empty_input(BLOCK_METADATA_QUERY_ID)?;
// Only check: block_gas_limit and individual_tx_gas_limit
``` [1](#0-0) 

These two fields drive every L2 transaction's native-resource budget:

```
nativePerGas      = ceil(gasPrice / native_price)
nativePerPubdata  = pubdata_price / native_price
nativeLimit       = gasLimit * nativePerGas
``` [2](#0-1) 

Neither `native_price` nor `pubdata_price` appears in `ChainStateCommitment`: [3](#0-2) 

Nor in `BatchOutput` (the other component of the public input hash): [4](#0-3) 

The project's own documentation explicitly acknowledges this gap:

> "Currently it misses `gas_per_pubdata` and `native_price`, but we already working on design and implementation to solve this issue." [5](#0-4) 

The `BlockMetadataFromOracle` struct confirms both fields are present and fully oracle-supplied: [6](#0-5) 

Note: L1 transactions are **not** affected — they use the hardcoded constant `L1_TX_NATIVE_PRICE` instead of the oracle value. [7](#0-6) 

### Impact Explanation

Because `native_price` and `pubdata_price` are absent from the public inputs, the settlement-layer verifier cannot enforce that the prover used the same pricing as the sequencer. A malicious prover can:

1. **Inflate `native_price`** → `nativePerGas` collapses → every L2 transaction that consumed significant native resources during forward execution is proven as out-of-native and reverted. Users lose funds (fees charged, execution reverted) for transactions that actually succeeded.
2. **Deflate `native_price`** → `nativePerGas` inflates → transactions that ran out of native resources during forward execution are proven as successful, writing state that was never actually produced.
3. **Inflate `pubdata_price`** → `nativePerPubdata` spikes → any transaction that wrote storage is proven as reverted due to post-execution pubdata charging, even if it succeeded during sequencing.

The resulting proof is cryptographically valid (the RISC-V execution is self-consistent) but the proven state transition does not correspond to the state the sequencer actually produced. This breaks the core ZK-rollup guarantee that the settlement layer can trust the proven state.

### Likelihood Explanation

The scope explicitly lists **"prover/forward execution input"** as a valid attacker entry point. The prover supplies oracle data to the RISC-V binary via CSR reads; substituting different `native_price`/`pubdata_price` values requires only that the prover's oracle implementation returns different bytes for `BLOCK_METADATA_QUERY_ID` — a trivial modification. No cryptographic key compromise or governance attack is needed; the prover simply provides a different serialized `BlockMetadataFromOracle` blob.

### Recommendation

1. **Commit `native_price` and `pubdata_price` to the public inputs.** Include them in `ChainStateCommitment` or a dedicated block-parameters commitment that is hashed into the public input, so the settlement layer can enforce consistency between sequencing and proving.
2. **Add range checks on `native_price` and `pubdata_price`** in `metadata_op.rs` (e.g., enforce non-zero, enforce a maximum ratio) as a defense-in-depth measure even before the commitment fix lands.
3. As an interim measure, consider deriving `native_price` deterministically from on-chain data (e.g., L1 gas price) rather than accepting it as a free oracle input.

### Proof of Concept

**Forward execution (sequencer):** runs block with `native_price = 100`, `pubdata_price = 1000`. All L2 transactions succeed; `nativePerGas = gasPrice / 100`.

**Proving (malicious prover):** supplies `BLOCK_METADATA_QUERY_ID` response with `native_price = 1_000_000`. Now `nativePerGas = gasPrice / 1_000_000` — 10,000× smaller. Every transaction that consumed more than `gasLimit * (gasPrice / 1_000_000)` native units is proven as out-of-native and reverted.

The resulting `state_after` hash differs from the sequencer's output. The proof is valid because `native_price` is not constrained by the public inputs. The settlement layer accepts the proof, committing a state that never existed.

    fn metadata_op<Config: BasicBootloaderExecutionConfig>(
        oracle: &mut impl IOOracle,
        _allocator: S::Allocator,
    ) -> Result<<S as SystemTypes>::Metadata, InternalError> {
        let block_level_metadata: BlockMetadataFromOracle =
            oracle.query_with_empty_input(BLOCK_METADATA_QUERY_ID)?;

        let metadata = ZkMetadata {
            tx_level: TxLevelMetadata::default(),
            block_level: block_level_metadata,
            _marker: core::marker::PhantomData,
        };

        if metadata.block_gas_limit() > MAX_BLOCK_GAS_LIMIT
            || metadata.individual_tx_gas_limit() > MAX_TX_GAS_LIMIT
        {
            return Err(internal_error!("block or tx gas limit is too high"));
        }

        Ok(metadata)
    }

impl ChainStateCommitment {
    ///
    /// Calculate blake2s hash of chain state commitment.
    ///
    /// We are using proving friendly blake2s because this commitment will be generated and opened during proving,
    /// but we don't need to open it on the settlement layer.
    ///
    pub fn hash(&self) -> [u8; 32] {

        hasher.update(self.state_root.as_u8_ref());

        hasher.update(&self.block_number.to_be_bytes());
        hasher.update(self.last_256_block_hashes_blake.as_u8_ref());
        hasher.update(&self.last_block_timestamp.to_be_bytes());
        hasher.finalize()
    }
