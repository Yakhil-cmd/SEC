### Title
Insufficient Treasury Balance During L1→L2 Transaction Processing Causes Unrecoverable Block Halt - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

### Summary

When the `BASE_TOKEN_HOLDER_ADDRESS` treasury has insufficient balance to cover an L1→L2 priority transaction's value transfer, fee payment, or refund, the `TreasuryTransferFailed` error is not handled gracefully. It propagates as a fatal block-level error rather than a per-transaction revert. Because L1 priority transactions cannot be skipped or invalidated, this permanently halts chain progression.

### Finding Description

The `mint_base_token` function is called up to three times per L1→L2 transaction: once to transfer the deposit value to the sender, once to pay the operator fee, and once to issue the gas refund. Each call internally invokes `transfer_from_treasury`, which returns `TreasuryTransferFailed` on `InsufficientBalance`. [1](#0-0) 

`transfer_from_treasury` maps any `SubsystemError::LeafUsage` (i.e., insufficient balance) to `BootloaderInterfaceError::TreasuryTransferFailed`: [2](#0-1) 

This error is a `LeafUsage` variant, **not** a `Runtime(FatalRuntimeError)`. The error-handling branch in `process_l1_transaction` only converts `FatalRuntimeError` into a graceful per-transaction revert; all other errors — including `TreasuryTransferFailed` — fall through to `return Err(e)`, which propagates the error out of the block processor entirely: [3](#0-2) 

The same unguarded propagation occurs in the post-execution operator-fee and refund `mint_base_token` calls: [4](#0-3) [5](#0-4) 

The existing test suite explicitly confirms this behavior — an L1 transaction against an underfunded treasury causes the entire block execution to return `Err(TreasuryTransferFailed)`: [6](#0-5) 

The resilience documentation for L1 transactions states the reason this matters: [7](#0-6) 

### Impact Explanation

**Impact: High.** A single L1→L2 priority transaction that cannot be fulfilled by the treasury causes the entire block to fail. Because L1 transactions come from the priority queue and cannot be invalidated or skipped, the chain is permanently halted at that block height. All user funds on L2 become inaccessible. This is strictly worse than the external report's "funds temporarily frozen for one user" — here, all users' funds are frozen indefinitely.

### Likelihood Explanation

**Likelihood: Medium.** The treasury (`BASE_TOKEN_HOLDER_ADDRESS`) is expected to be pre-funded by the L1 bridge with exactly `total_deposited` per transaction. However, any discrepancy — a bug in L1 bridge accounting, a race condition between multiple L1 transactions in the same block collectively exceeding the treasury balance, or a deliberate drain of the treasury via a separate vulnerability — would trigger this halt. The code itself acknowledges the invariant is enforced on L1 but only double-checked on L2: [8](#0-7) 

### Recommendation

`TreasuryTransferFailed` should be treated as a per-transaction failure for L1 priority transactions, not a block-level fatal error. The fix mirrors how `FatalRuntimeError` is already handled: catch `TreasuryTransferFailed` in the same match arm, exhaust ergs, roll back the execution frame, and continue block processing with `is_success = false`. The refund logic already handles the `!is_success` case by returning `total_deposited - pay_to_operator` to the refund recipient, so the accounting path exists. The operator-fee and refund `mint_base_token` calls in the post-execution phase should similarly degrade gracefully rather than propagating a block-fatal error.

### Proof of Concept

1. Deploy a ZKsync OS chain with treasury balance set to a value lower than `total_deposited` of a pending L1→L2 priority transaction (e.g., treasury = 1000 wei, transaction `total_deposited` = 500,000 wei).
2. Submit the L1→L2 transaction to the priority queue.
3. Attempt to seal a block containing that transaction.
4. `execute_l1_transaction_and_notify_result` calls `mint_base_token` → `transfer_from_treasury` → `update_account_nominal_token_balance` returns `InsufficientBalance` → mapped to `TreasuryTransferFailed`.
5. The error propagates through the `_ => { return Err(e); }` branch in `process_l1_transaction`.
6. Block sealing returns `Err(TreasuryTransferFailed)`.
7. The priority transaction cannot be skipped; no subsequent block can be sealed. Chain is permanently halted.

This is directly confirmed by the existing test `test_treasury_insufficient_balance_failure` at: [9](#0-8)

/// Test treasury transfer failure when treasury has insufficient balance
#[test]
fn test_treasury_insufficient_balance_failure() {
    use rig::system_hooks::addresses_constants::BASE_TOKEN_HOLDER_ADDRESS;

    // Manually set very low treasury balance instead of using default
    let low_treasury_balance = U256::from(1000u64);
    let mut tester = TestingFramework::new()
        .with_balance(BASE_TOKEN_HOLDER_ADDRESS.into_alloy(), low_treasury_balance);

    // Create L1→L2 transaction that requires more tokens than treasury has
    let l1_sender = address!("1234000000000000000000000000000000000000");
    let l1_recipient = address!("5678000000000000000000000000000000000000");

    let gas_price = 1000u64;
    let gas_limit = 100_000u64;
    let value_to_transfer = U256::from(500_000u64); // More than treasury can cover

    let l1_tx: ZKsyncTxEnvelope = L1TxBuilder::new()
        .from(l1_sender)
        .to(l1_recipient)
        .gas_price(gas_price.into())
        .gas_limit(gas_limit.into())
        .value(value_to_transfer)
        .build()
        .into();

    // Ensure we rely on treasury balance, not auto-minting.
    tester = tester.without_minting_tokens_to_treasury();

    // This should fail due to insufficient treasury balance
    let result = tester.execute_block_no_panic(vec![l1_tx]);

    // Verify transaction fails due to treasury insufficient balance
    assert!(
        result.is_err(),
        "L1→L2 transaction should fail when treasury has insufficient balance"
    );

    // Verify the specific error is treasury transfer failed
    let error_debug = format!("{:?}", result.unwrap_err());
    assert!(
        error_debug.contains("TreasuryTransferFailed"),
        "Error should indicate treasury transfer failed, got: {}",
        error_debug
    );
}
