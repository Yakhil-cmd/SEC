Audit Report

## Title
ICP Burn Fire-and-Forget After Cycles Mint Creates Permanent ICP Supply Inflation on Ledger Unavailability - (File: rs/nns/cmc/src/main.rs)

## Summary
In `process_mint_cycles`, `process_top_up`, and `process_create_canister`, the CMC credits cycles to the recipient before burning the backing ICP. The `burn_and_log` function is explicitly designed to never propagate errors — it logs failures silently and returns `()`. If the ICP ledger is temporarily unavailable (e.g., during a governance-triggered upgrade), the burn silently fails, the notification is permanently marked as completed (`NotifiedMint`/`NotifiedTopUp`/`NotifiedCreateCanister`), and the ICP remains stranded in the CMC's subaccount with no automated recovery path, permanently inflating the ICP total supply while the corresponding cycles remain in circulation.

## Finding Description

**Exact code path:**

In `process_mint_cycles` (lines 1966–1968), `process_top_up` (lines 1999–2001), and `process_create_canister` (lines 1943–1945), the operation order is:
1. Mint/deposit cycles to recipient (`do_mint_cycles` → cycles ledger `deposit`, or `deposit_cycles` → management canister, or `do_create_canister`).
2. Call `burn_and_log(sub, amount)` to destroy the backing ICP.

`burn_and_log` (lines 2014–2049) is explicitly designed to swallow all errors. The comment reads: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."* On any failure of the `send_pb` call to the ICP ledger, it logs the error and returns `()` — the caller receives no signal that the burn failed.

After `process_mint_cycles` returns `Ok(...)`, the caller (`notify_mint_cycles`) stores `NotificationStatus::NotifiedMint(result)` in `blocks_notified` (line 1308). The `is_transient_error` check (line 1310) only removes the entry for retriable errors; a successful `Ok` result is permanently stored. The block is therefore non-retryable regardless of whether the burn succeeded.

**Why existing checks fail:**

The `is_transient_error` guard (line 1125–1130) only removes the notification entry when the overall result is a retriable `NotifyError`. Since `burn_and_log` failure does not propagate to the result — the function returns `Ok(NotifyMintCyclesSuccess{...})` even when the burn fails — the block is permanently marked as processed. There is no retry queue, no reconciliation job, and no mechanism to recover the stranded ICP from the CMC subaccount.

**Root cause:** The ICP burn is a fire-and-forget side effect after the state-changing cross-canister call has already succeeded and been committed to `blocks_notified`.

## Impact Explanation

When `burn_and_log` fails, the ICP amount remains in the CMC's subaccount (`Subaccount::from(&caller())` for `notify_mint_cycles`, `Subaccount::from(&canister_id)` for `notify_top_up`). It is not burned and not refunded. The ICP total supply is permanently inflated by the unburned amount, while the corresponding cycles are in circulation — breaking the ICP↔cycles conservation invariant that underpins the economic model of the Internet Computer. This constitutes a concrete, permanent ledger accounting violation: cycles exist in circulation without the corresponding ICP destruction. This matches the **Medium** impact tier: a difficult attack requiring strict target conditions (ledger upgrade timing) with meaningful security impact on the ICP ledger's conservation invariant.

## Likelihood Explanation

The ICP ledger is a separate canister. NNS governance proposals to upgrade it are publicly visible and their execution time is deterministic after the voting period ends. An attacker can:
1. Pre-position ICP in the CMC subaccount before the upgrade window.
2. Call `notify_mint_cycles` at the moment the ledger canister is stopped for upgrade (causing `send_pb` inside `burn_and_log` to fail with `CanisterStopped`).
3. Receive cycles; the ICP is never burned.

The attack requires no special privileges — any ICP holder can execute it. The constraint is the short ledger upgrade window, making this a constrained but realistic and repeatable attack (once per ledger upgrade event). The CMC's own comment acknowledges the failure path as a known design trade-off.

## Recommendation

Reorder the operations so the ICP burn is confirmed before cycles are minted:
1. Call the ICP ledger `send_pb` to burn the ICP from the CMC subaccount.
2. Only if the burn succeeds, mint/deposit cycles to the recipient.

If the burn fails, return a retriable error so the user can retry after the ledger recovers. This eliminates the window where cycles exist without backing ICP destruction. Alternatively, implement a persistent retry queue for failed burns with an automated reconciliation mechanism, so stranded ICP in CMC subaccounts is eventually burned. The current comment justifying the fire-and-forget approach ("we don't want to reject the transaction notification because then it could be retried") applies to the mint step, not the burn step — a failed burn should be retried by the CMC internally, not silently dropped.

## Proof of Concept

**Preconditions:** Attacker holds ICP. An NNS proposal to upgrade the ICP ledger canister is pending with a known execution timestamp.

1. Attacker transfers X ICP to the CMC subaccount `Subaccount::from(&attacker_principal)` with `MEMO_MINT_CYCLES`.
2. Attacker waits for the NNS proposal execution time (deterministic).
3. At the moment the ICP ledger canister is stopped for upgrade, attacker calls `notify_mint_cycles` on the CMC.
4. CMC executes `process_mint_cycles` → `do_mint_cycles` → cycles ledger `deposit` → cycles credited to attacker. ✓
5. CMC calls `burn_and_log(sub, X)` → `send_pb` to ledger → ledger is stopped → call fails with `CanisterStopped`.
6. `burn_and_log` logs the error, returns `()`.
7. `notify_mint_cycles` stores `NotificationStatus::NotifiedMint(Ok(...))` — permanently non-retryable.
8. `notify_mint_cycles` returns `Ok(NotifyMintCyclesSuccess{...})` to attacker.
9. **Result:** Attacker holds cycles worth X ICP. X ICP sits in CMC subaccount, never burned. ICP total supply permanently inflated by X. No recovery path exists.

A deterministic integration test using PocketIC can reproduce this by: deploying CMC + ICP ledger + cycles ledger locally, stopping the ICP ledger canister after the cycles deposit call returns but before `burn_and_log` completes (using PocketIC's canister stop API timed to the inter-canister call boundary), and asserting that `icrc1_total_supply` on the ICP ledger exceeds the expected post-burn value while the CMC's `total_cycles_minted` reflects the minted cycles.
