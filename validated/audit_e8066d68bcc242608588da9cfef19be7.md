Audit Report

## Title
Permanent ckBTC Minting Denial for UTXOs from Transactions Exceeding 400 kB Due to Missing Suspension Logic - (File: `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary
When a Bitcoin transaction exceeds 400 kB, the btc_checker canister returns `CheckTransactionIrrecoverableError::ResponseTooLarge`, which the ckBTC minter's `check_utxo` function converts to `Err(UpdateBalanceError::GenericError)` and propagates via `?` without suspending the affected UTXO. Because `SuspendedReason` has no variant for size-limit failures, the UTXO remains in the processable pool indefinitely. Every subsequent `update_balance` call re-encounters the same UTXO, re-triggers the same failing check, and returns the same `GenericError`, permanently blocking ckBTC minting for that deposit with no user-accessible recovery path.

## Finding Description

**Step 1 — Size ceiling.** `rs/bitcoin/checker/lib/lib.rs` lines 12–25 define `RETRY_MAX_RESPONSE_BYTES = 400 * 1024` and document that transactions between 400 kB and 2 MiB are unhandled.

**Step 2 — Fetch logic.** In `rs/bitcoin/checker/src/fetch.rs` lines 148–168, when `http_get_tx` returns `Err(HttpGetTxError::ResponseTooLarge)` and `max_response_bytes >= RETRY_MAX_RESPONSE_BYTES`, the code falls through to the generic `Err(err)` branch, sets `FetchTxStatus::Error`, and returns `FetchResult::Error(HttpGetTxError::ResponseTooLarge)`.

**Step 3 — Error mapping.** `rs/bitcoin/checker/src/fetch.rs` lines 28–30 convert `HttpGetTxError::ResponseTooLarge` to `CheckTransactionIrrecoverableError::ResponseTooLarge { txid }`. Per `rs/bitcoin/checker/lib/types.rs` lines 97–101, this maps to `CheckTransactionResponse::Unknown(CheckTransactionStatus::Error(...))`.

**Step 4 — Retry loop in btc_checker.** On subsequent `check_transaction` calls for the same txid, `try_fetch_tx` (fetch.rs lines 85–90) sees `FetchTxStatus::Error` and retries with the next provider at the same 400 kB buffer. Since the transaction is genuinely too large, every provider also returns `ResponseTooLarge`, cycling through providers indefinitely and consuming cycles on each call.

**Step 5 — Minter propagates without suspending.** In `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs` lines 442–448, `check_utxo` matches `CheckTransactionStatus::Error` and returns `Err(UpdateBalanceError::GenericError { error_code: KytError })`. Line 302 propagates this via `?`, returning immediately without calling `ignore_utxo`, `quarantine_utxo`, or any suspension function.

**Step 6 — No suspension variant exists.** `rs/bitcoin/ckbtc/minter/src/state.rs` lines 1911–1917 show `SuspendedReason` has only `ValueTooSmall` and `Quarantined`. There is no variant for irrecoverable fetch failures. The UTXO therefore remains in the processable pool returned by `processable_utxos_for_account`, and every future `update_balance` call repeats steps 2–5 identically.

## Impact Explanation

A legitimate depositor whose BTC arrives via a transaction exceeding 400 kB (e.g., a large Taproot consolidation) permanently loses the ability to mint ckBTC for that deposit through the normal protocol flow. The BTC is not stolen, but it is effectively locked: the minter will never advance the UTXO past the check stage, and the user receives a `GenericError` on every `update_balance` attempt with no indication of the root cause or any self-service remedy. Recovery requires a governance-approved canister upgrade. This constitutes a significant ck-token impact with concrete, repeatable user harm, matching the High ($2,000–$10,000) impact class: *Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm.*

## Likelihood Explanation

No special privileges are required. Any user who deposits BTC via a transaction larger than 400 kB triggers the path. Standard non-Taproot transactions rarely reach this size, but Taproot transactions with many inputs/outputs can exceed it. The code comment in `lib.rs` explicitly acknowledges the gap ("Transaction size between 400kB and 2MiB are also uncommon, we could handle them in the future if required"), confirming the scenario is real and anticipated. As Taproot adoption grows, the probability increases. The attack is free to trigger and repeatable.

## Recommendation

1. Add a `ResponseTooLarge` (or `IrrecoverableCheckError`) variant to `SuspendedReason` in `rs/bitcoin/ckbtc/minter/src/state.rs`.
2. In `check_utxo` (`update_balance.rs`), match `CheckTransactionStatus::Error(CheckTransactionIrrecoverableError::ResponseTooLarge { .. })` specifically and call `ignore_utxo` / `quarantine_utxo` (or a new suspension function) so the UTXO is moved out of the processable pool.
3. Increase `RETRY_MAX_RESPONSE_BYTES` toward the 2 MiB subnet ceiling to reduce the set of unhandleable transactions.
4. Return a distinct, user-visible error (not a generic `KytError`) so depositors understand why their UTXO is suspended.

## Proof of Concept

1. Deploy a local ckBTC stack (minter + btc_checker) in regtest mode using PocketIC or the existing integration test harness.
2. Construct a Bitcoin transaction exceeding 400 kB (e.g., a Taproot transaction with hundreds of inputs) and broadcast it to the regtest node.
3. Mine enough blocks to satisfy `min_confirmations`.
4. Call `update_balance` for the depositor's account.
5. Observe `UpdateBalanceError::GenericError { error_code: 2, error_message: "Bitcoin checker error: ResponseTooLarge { txid: ... }" }`.
6. Call `update_balance` again (any number of times) and observe the identical error each time, confirming the UTXO is never suspended and the loop is permanent.
7. Inspect minter state to confirm the UTXO is absent from `checked_utxos`, `suspended_utxos`, and `minted_outpoints`.
