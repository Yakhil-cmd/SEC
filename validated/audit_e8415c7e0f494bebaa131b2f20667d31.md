Audit Report

## Title
ckETH Minter Mints Non-Withdrawable Dust Balances Due to Missing Minimum Deposit Check - (File: rs/ethereum/cketh/minter/src/deposit.rs)

## Summary
The ckETH minter enforces a `minimum_withdrawal_amount` on the `withdraw_eth` endpoint but applies no equivalent minimum on the deposit path. Any ETH deposited via the Ethereum helper contract below `minimum_withdrawal_amount` is minted as ckETH that can never be converted back to ETH, permanently locking the underlying ETH in the minter's Ethereum address. The ckBTC minter correctly implements the analogous `deposit_btc_min_amount` guard; ckETH does not.

## Finding Description
The `mint()` function in `rs/ethereum/cketh/minter/src/deposit.rs` iterates over all `events_to_mint` and unconditionally calls `client.transfer(amount: event.value())` on the ckETH ledger with no comparison against `cketh_minimum_withdrawal_amount` or any deposit floor. The upstream `register_deposit_events()` function only filters for blocked Ethereum addresses; it performs no amount check. The `State` struct contains `cketh_minimum_withdrawal_amount: Wei` but no `deposit_minimum_amount` field. `validate_config()` validates only the withdrawal minimum. The Ethereum helper contract `depositEth()` and `depositErc20()` accept any non-zero `msg.value`/`amount` with no minimum enforcement. The result is that a deposit of X wei where X < `minimum_withdrawal_amount` (currently 5,000,000,000,000,000 wei on mainnet) is irreversibly transferred to the minter's Ethereum address, minted as ckETH, and permanently unrecoverable: `withdraw_eth` rejects it as below minimum, and if X < ledger transfer fee (2,000,000,000,000 wei), even `icrc1_transfer` fails. The ckBTC minter avoids this exact scenario by checking `utxo.value < deposit_btc_min_amount` and returning `UtxoStatus::ValueTooSmall` before any minting occurs.

## Impact Explanation
This matches the allowed High impact: "Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm." Any ETH deposited below the withdrawal minimum is permanently locked in the minter's Ethereum address with no admin recovery path. An unprivileged attacker can also grief arbitrary users by calling `depositEth(victim_principal, subaccount)` with dust amounts, causing the victim to receive non-withdrawable (and potentially non-transferable) ckETH dust in their IC account at low cost to the attacker. Additionally, a governance upgrade raising `minimum_withdrawal_amount` retroactively converts previously-borderline deposits into permanently locked funds, broadening the affected population without any user action.

## Likelihood Explanation
Medium. The Ethereum helper contract surfaces no IC-side minimum to depositors, making accidental sub-minimum deposits realistic for users unfamiliar with the minter's parameters. The griefing vector requires no victim cooperation: an attacker specifies any victim's principal in `depositEth()` and pays only gas plus the dust amount. The `minimum_withdrawal_amount` is a mutable governance parameter, meaning the locked-funds population can grow after upgrades.

## Recommendation
1. Add a `deposit_minimum_amount: Wei` field to the `State` struct in `rs/ethereum/cketh/minter/src/state.rs`, initialized to at least `cketh_minimum_withdrawal_amount`.
2. In `register_deposit_events()` in `rs/ethereum/cketh/minter/src/deposit.rs`, check `event.value() < deposit_minimum_amount` and emit `EventType::InvalidDeposit` (with reason `"deposit amount below minimum"`) instead of queuing the event for minting, mirroring the ckBTC `UtxoStatus::ValueTooSmall` pattern.
3. Validate in `validate_config()` that `deposit_minimum_amount >= cketh_minimum_withdrawal_amount`.
4. Consider enforcing the minimum in `DepositHelperWithSubaccount.sol` to prevent ETH from being transferred to the minter in the first place.

## Proof of Concept
1. Call `depositEth(victim_principal, subaccount)` on the deployed Ethereum helper contract with `msg.value = 1` (1 wei). No revert occurs.
2. The helper emits `ReceivedEthOrErc20(0x0, attacker, 1, victim_principal, subaccount)` and transfers 1 wei to the minter address.
3. The ckETH minter's `scrape_logs()` timer picks up the event; `register_deposit_events()` passes it (not a blocked address, no amount check) and queues it via `process_event(s, event.into_deposit())`.
4. `mint()` calls `client.transfer(amount: 1)` on the ckETH ledger — succeeds, minting 1 wei ckETH to the victim's IC account.
5. Victim calls `withdraw_eth(amount: 1)` → `WithdrawalError::AmountTooLow { min_withdrawal_amount: 5_000_000_000_000_000 }`.
6. Victim calls `icrc1_transfer(amount: 1)` → `BadBurn` / fee error (amount < 2,000,000,000,000 wei ledger fee).
7. 1 wei ETH is permanently locked in the minter's Ethereum address; 1 wei ckETH is permanently frozen in the victim's IC account.
A deterministic integration test or PocketIC test can reproduce this by initializing the minter with `minimum_withdrawal_amount = 5_000_000_000_000_000`, injecting a `ReceivedEthEvent` with `value = 1`, running `mint()`, and asserting that `withdraw_eth(1)` returns `AmountTooLow`.
