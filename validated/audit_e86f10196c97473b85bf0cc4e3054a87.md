### Title
Missing Removal Function for Registered NEP-141 Token Mapping Prevents Emergency Response to Compromised Token Contracts - (File: engine/src/engine.rs)

### Summary

The Aurora Engine allows NEP-141 tokens to be permanently registered into the `nep141_erc20_map` via `deploy_erc20_token`, but provides no corresponding function to remove a registered NEP-141 token from this mapping. If a registered NEP-141 token contract becomes compromised and is actively exploited, the Aurora Engine owner has no targeted mechanism to revoke its minting rights, leading to potential insolvency of the corresponding ERC-20 token on Aurora.

### Finding Description

The `nep141_erc20_map` bijection map in `engine/src/engine.rs` permanently maps NEP-141 token account IDs to ERC-20 token addresses on Aurora. Tokens are registered via `deploy_erc20_token` in `engine/src/contract_methods/connector.rs`, which calls `engine.register_token(address, nep141)`, inserting into the map: [1](#0-0) 

The `ft_on_transfer` function in `engine/src/contract_methods/connector.rs` uses this mapping to mint ERC-20 tokens when a NEP-141 token is transferred to Aurora. It routes any call whose `predecessor_account_id` is a registered NEP-141 token to `receive_erc20_tokens`: [2](#0-1) 

Inside `receive_erc20_tokens`, the engine looks up the ERC-20 address from the NEP-141 account ID and mints tokens: [3](#0-2) 

The `register_token` function only supports insertion — there is no `unregister_token`, `remove_nep141`, or equivalent function anywhere in the contract API: [4](#0-3) 

The public contract entrypoints expose `deploy_erc20_token` and `mirror_erc20_token` for registration, but no removal counterpart: [5](#0-4) 

The only emergency mechanism available to the owner is `pause_contract`, which halts the entire engine — a blunt instrument that stops all user operations, not just the compromised token's minting path: [6](#0-5) 

### Impact Explanation

**Critical — Insolvency.** If a registered NEP-141 token contract is compromised (e.g., contains a bug allowing arbitrary `ft_transfer_call` invocations without real token backing), the attacker can repeatedly call `ft_on_transfer` on Aurora with the compromised NEP-141 as the `predecessor_account_id`. Aurora will mint the corresponding ERC-20 tokens without any real NEP-141 backing. This inflates the ERC-20 supply beyond the actual NEP-141 reserves held by Aurora, making the ERC-20 token insolvent. Legitimate holders of the ERC-20 token on Aurora cannot fully redeem their tokens for the underlying NEP-141 asset.

The owner's only recourse — pausing the entire contract — stops all bridge deposits, withdrawals, and EVM transactions for all users, introducing significant collateral damage and delay during a time-critical incident.

### Likelihood Explanation

Aurora has many NEP-141 tokens registered. Any one of them could contain a vulnerability (reentrancy, access control bug, or upgrade-key compromise) that allows an attacker to trigger `ft_on_transfer` on Aurora with inflated amounts. The NEAR ecosystem has seen NEP-141 contract bugs historically. The missing removal function means the window of exploitation is unbounded once a token is registered.

### Recommendation

Add an owner-only function to remove a NEP-141 token from the `nep141_erc20_map`:

```rust
#[named]
pub fn remove_erc20_token<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        let nep141: AccountId = io.read_input_borsh()?;
        // Remove both directions of the bijection map
        let erc20 = get_erc20_from_nep141(&io, &nep141)
            .map_err(|_| errors::ERR_NEP141_NOT_FOUND)?;
        nep141_erc20_map(io).remove(&NEP141Account(nep141), &ERC20Address(erc20));
        Ok(())
    })
}
```

This mirrors the pattern already used for the Silo whitelist, which correctly provides both `add_entry_to_whitelist` and `remove_entry_from_whitelist`: [7](#0-6) 

### Proof of Concept

1. Owner calls `deploy_erc20_token` for `vuln-token.near`, registering it in `nep141_erc20_map` and deploying a corresponding ERC-20 on Aurora.
2. A vulnerability is discovered in `vuln-token.near` that allows an attacker to invoke `ft_transfer_call` targeting Aurora with an arbitrary `amount` without holding real tokens.
3. The attacker exploits this to call `ft_on_transfer` on Aurora with `predecessor_account_id = "vuln-token.near"` and `amount = 1_000_000_000`.
4. Aurora's `ft_on_transfer` routes to `receive_erc20_tokens`, looks up the ERC-20 address via `get_erc20_from_nep141`, and mints 1,000,000,000 unbacked ERC-20 tokens to the attacker's address.
5. The attacker uses the inflated ERC-20 balance to drain liquidity pools or exit to NEAR via the `exit_to_near` precompile.
6. The owner attempts to stop the attack. There is no `remove_erc20_token` function. The only option is `pause_contract`, which halts the entire Aurora Engine for all users.
7. Every second of delay while the owner decides to pause the whole engine allows the attacker to drain more value. [8](#0-7) [9](#0-8)

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;
