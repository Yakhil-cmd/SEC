### Title
Silent Success on Call to Codeless `L2AssetTracker` Skips Token Accounting - (`File: basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The `notify_l2_asset_tracker` function in the L1 transaction processing flow calls `handleFinalizeBaseTokenBridgingOnL2` on `L2_ASSET_TRACKER_ADDRESS` without verifying that the address has deployed code. Per EVM semantics — explicitly implemented in ZKsync OS's own `call_execute_callee_frame` — calls to accounts with no code succeed silently with empty return data. The `failed` check that follows only catches explicit reverts, not silent success from an empty account. The code itself acknowledges this behavior in a comment but relies on an assumption rather than an enforcement. As a result, if `L2AssetTracker` has no code at the time of an L1→L2 deposit, the treasury token transfer still executes while the asset tracker's accounting is never updated.

---

### Finding Description

In `notify_l2_asset_tracker`, the bootloader calls `run_single_interaction` targeting `L2_ASSET_TRACKER_ADDRESS`: [1](#0-0) 

```rust
/// If no contract is deployed at L2AssetTracker, the call succeeds silently
/// (a call to an empty address returns success with no returndata in EVM).
/// However, we are certain that L2AssetTracker is available after the upgrade.
``` [2](#0-1) 

The result is checked only for explicit failure:

```rust
if failed {
    return Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted").into());
}
```

The ZKsync OS bootloader runner itself confirms that calls to empty accounts (`ExecutionEnvironmentType::NoEE`) return `CallResult::Successful` with empty return data: [3](#0-2) 

```rust
// By convention, calls to empty accounts succeed without any return data
if next_ee_type == ExecutionEnvironmentType::NoEE {
    ...
    return Ok((
        ...,
        CallResult::Successful { return_values: ReturnValues::empty() },
    ));
}
```

Therefore `asset_tracker_result.failed()` returns `false` when `L2AssetTracker` has no code, `notify_l2_asset_tracker` returns `Ok(())`, and execution continues into `transfer_from_treasury`, which transfers real tokens from the treasury: [4](#0-3) 

The `notify_l2_asset_tracker` function is called up to three times per L1 transaction — for the value mint, operator fee, and refund: [5](#0-4) 

---

### Impact Explanation

If `L2AssetTracker` has no code deployed at its canonical address (`0x800b` / `L2_ASSET_TRACKER_ADDRESS`), every L1→L2 deposit transaction will:

1. Silently skip the `handleFinalizeBaseTokenBridgingOnL2` accounting call.
2. Still execute `transfer_from_treasury`, moving real base tokens from the treasury to the recipient, operator, and refund recipient.
3. Leave `interopInfo[BASE_TOKEN_ASSET_ID].totalSuccessfulDepositsFromL1` permanently unupdated.

This produces a permanent divergence between the on-chain token supply and the asset tracker's recorded deposit accounting. Any downstream protocol logic (cross-chain settlement, migration checks using `totalSuccessfulDepositsFromL1`, `_needToForceSetAssetMigrationOnL2`) will operate on stale/incorrect state. The treasury balance is correctly drained, but the accounting layer is silently corrupted.

---

### Likelihood Explanation

The scenario is reachable during any window where `L2AssetTracker` is not yet deployed — for example, before the protocol upgrade that installs it, during a chain migration, or if the bytecode is cleared by a buggy upgrade transaction. Any unprivileged user who submits an L1→L2 transaction with `to_mint > 0` during such a window triggers the path. The code comment at line 852–854 explicitly acknowledges the silent-success behavior and defers to an assumption ("we are certain that L2AssetTracker is available after the upgrade") rather than a runtime check, confirming the gap is known but unguarded.

---

### Recommendation

Add an explicit code-existence check inside `notify_l2_asset_tracker` before dispatching the call. Read the account properties of `L2_ASSET_TRACKER_ADDRESS` and assert `unpadded_code_len != 0`. If the contract is absent, treat it as a fatal error rather than silently succeeding. This mirrors the recommendation from the analogous FeeWrapper report: verify the target has code before trusting a successful low-level call result.

---

### Proof of Concept

1. Deploy a ZKsync OS chain **without** pre-installing `L2AssetTracker` bytecode at `L2_ASSET_TRACKER_ADDRESS` (i.e., skip `install_default_predeployed_contracts` or clear the bytecode).
2. Submit an L1→L2 transaction with `to_mint > 0` (a deposit).
3. Observe: the transaction succeeds, treasury balance decreases, recipient receives tokens.
4. Observe: `interopInfo[BASE_TOKEN_ASSET_ID].totalSuccessfulDepositsFromL1` in the `L2AssetTracker` storage slot remains zero — the accounting call was silently swallowed.

The `call_execute_callee_frame` path at `runner.rs:422–442` is the necessary vulnerable step: it returns `CallResult::Successful` for any call to an account with `next_ee_type == NoEE` (no code), which is exactly what `L2_ASSET_TRACKER_ADDRESS` resolves to when undeployed. [3](#0-2) [6](#0-5)

fn notify_l2_asset_tracker<'a, S: EthereumLikeTypes + 'a, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    system_functions: &mut HooksStorage<S, S::Allocator>,
    memories: RunnerMemoryBuffers<'a>,
    amount: U256,
    l1_chain_id: U256,
    resources: &mut S::Resources,
    tracer: &mut impl Tracer<S>,
    validator: &mut impl TxValidator<S>,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
    S::Metadata: ZkSpecificPricingMetadata
        + BasicMetadata<S::IOTypes, TransactionMetadata = TxLevelMetadata<S::IOTypes>>,
{
    if amount > U256::ZERO || Config::SIMULATION {
        // Encode calldata for handleFinalizeBaseTokenBridgingOnL2(uint256,uint256):
        // selector 0x03117c8c + abi-encoded (fromChainId, amount)
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]);
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution {
                resources_returned,
                result: asset_tracker_result,
            } = BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system,
                system_functions,
                memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,
                inf_ergs.clone(),
                &U256::ZERO,
                true, // should_make_frame - isolate state changes
                tracer,
                validator,
            )?;
            // Overwrite resources inside the closure so that
            // with_infinite_ergs correctly restores ergs afterwards.
            *inf_ergs = resources_returned;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
        })?;

        if failed {
            system_log!(
                system,
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 failed for amount {amount:?}\n"
            );
            // A revert here means the chain's token accounting would be inconsistent.
            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
            )
            .into());
        }
    }
    Ok(())
