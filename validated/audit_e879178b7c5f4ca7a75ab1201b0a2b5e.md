### Title
Aurora Engine `new()` Initialization Frontrun Allows Attacker to Seize Engine Ownership - (File: `engine/src/contract_methods/admin.rs`)

---

### Summary

The Aurora Engine's `new()` initialization function has no caller restriction and is invoked in a separate transaction from the WASM deployment. An attacker who observes the deploy transaction on-chain can frontrun the `new()` call, set themselves as `owner_id`, and permanently seize control of the engine instance before the legitimate deployer can initialize it.

---

### Finding Description

The `new()` function in `engine/src/contract_methods/admin.rs` is the sole initialization entry point for the Aurora Engine contract. It accepts the first call unconditionally — the only guard is a check that state does not already exist:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // ... no predecessor_account_id check ...
    state::set_state(&mut io, &state)?;
    Ok(())
}
``` [1](#0-0) 

There is no check that `env.predecessor_account_id()` equals the contract account or any trusted deployer. Any NEAR account can call `new()` on a freshly deployed but uninitialized engine.

The deployment flow in `engine-workspace/src/lib.rs` performs two separate async transactions — one to deploy the WASM and one to call `new()`:

```rust
let contract = account.deploy(&self.code...).await?;
// ... separate transaction ...
engine.new(self.chain_id, self.owner_id, self.upgrade_delay_blocks).transact().await...
``` [2](#0-1) 

Between these two transactions there is a window in which any NEAR account can call `new()` with attacker-controlled arguments, including a malicious `owner_id`.

The `owner_id` field written by `new()` is subsequently used by every privileged administrative function — `set_owner`, `set_eth_connector_contract_account`, `pause_contract`, `factory_set_wnear_address`, etc. — all of which gate access via `require_owner_only`:

```rust
pub fn require_owner_only(state: &state::EngineState, predecessor_account_id: &AccountId) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}
``` [3](#0-2) 

---

### Impact Explanation

**Critical — Direct theft of user funds.**

Once the attacker controls `owner_id`, they can call `set_eth_connector_contract_account` to redirect the engine to a malicious ETH connector contract. All subsequent bridge deposits (ETH and ERC-20 tokens) flow through the attacker-controlled connector. The attacker can drain all bridged assets at will. Additionally, the attacker can call `attach_full_access_key` to add a full-access key to the engine account, giving them unrestricted on-chain control. [4](#0-3) 

---

### Likelihood Explanation

**Medium.**

NEAR Protocol does not have a traditional public mempool, but transactions are visible once included in a block. If the WASM deploy and `new()` are in separate blocks (as the two-`await` pattern in the workspace implies), an attacker monitoring the NEAR RPC can observe the deploy receipt and submit a `new()` call targeting the freshly deployed, uninitialized contract account before the legitimate deployer's `new()` is processed. This requires no special privilege — only the ability to submit a NEAR transaction.

---

### Recommendation

Ensure `new()` is called atomically in the same batch transaction as the WASM deployment, using NEAR's `promise_batch_action_deploy_contract` followed immediately by `promise_batch_action_function_call` for `new`. This is the same pattern already used correctly for the XCC router sub-accounts: [5](#0-4) 

Alternatively, add a caller restriction inside `new()` requiring `env.predecessor_account_id() == env.current_account_id()`, so only the contract itself (via a self-batch) can initialize.

---

### Proof of Concept

1. Attacker monitors NEAR RPC for a `DeployContract` action targeting a new Aurora Engine account (e.g., `aurora.near`).
2. Upon seeing the deploy receipt confirmed, attacker immediately submits a call to `aurora.near::new` with `owner_id = attacker.near` and arbitrary `chain_id`.
3. Attacker's `new()` succeeds — `get_state` returns `Err` (no state yet), so the guard passes, and `state::set_state` writes `owner_id = attacker.near`.
4. The legitimate deployer's `new()` call arrives next and fails with `ERR_ALREADY_INITIALIZED`.
5. Attacker calls `set_eth_connector_contract_account` pointing to a malicious connector contract they control.
6. Users bridge ETH/ERC-20 tokens into the engine; all deposits are processed by the malicious connector.
7. Attacker calls `attach_full_access_key` to add their own key to the engine account, cementing permanent control.
8. Attacker drains all bridged funds. [6](#0-5) [7](#0-6)

#[named]
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
