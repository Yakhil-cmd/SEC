Audit Report

## Title
Single Principal Monopolizes All `MAX_CONCURRENT` Guard Slots via Distinct Subaccounts in `retrieve_btc_with_approval` — (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary

`Guard<RetrieveBtcUpdates>` is keyed on the full ICRC-1 `Account` (principal + subaccount), not on the principal alone. Because `retrieve_btc_with_approval` constructs the guard key using the caller-controlled `args.from_subaccount`, a single unprivileged principal can submit 100 concurrent calls each with a distinct subaccount, occupying all 100 `MAX_CONCURRENT` slots simultaneously and returning `TemporarilyUnavailable` to every other user for the duration of those in-flight async operations. Repeated waves can also exhaust the global `MAX_CONCURRENT_PENDING_REQUESTS = 5000` queue.

## Finding Description

**Guard struct is keyed on full `Account`:**
`Guard<PR>` stores an `Account` (owner + subaccount). [1](#0-0) 

The admission logic checks `accounts.contains(&account)` (exact `Account` match) and `accounts.len() >= MAX_CONCURRENT` (global cap of 100). [2](#0-1) 

`AlreadyProcessing` fires only on an identical `Account`; different subaccounts for the same principal are treated as distinct entries.

**`retrieve_btc_with_approval` passes caller-controlled `from_subaccount` as the guard key:** [3](#0-2) 

The attacker trivially supplies 100 distinct `[i; 32]` subaccount values, each acquiring a separate slot without ever triggering `AlreadyProcessing`.

**Contrast with `retrieve_btc` (non-approval path):** it always passes `subaccount: None`, so one principal can hold at most one slot there. [4](#0-3) 

**Queue exhaustion check is after guard acquisition:** the `MAX_CONCURRENT_PENDING_REQUESTS` check at line 274 is reached only after the guard is already held, so each successful wave of 100 calls appends 100 entries to the pending queue. [5](#0-4) 

**The existing test explicitly confirms the bypass:** it creates `MAX_CONCURRENT / 2` guards for the same principal (id=0) using distinct subaccounts and asserts they all succeed, proving one principal can saturate a large fraction (and by extension all 100) of the global limit. [6](#0-5) 

## Impact Explanation

**Phase 1 — Concurrent DoS (near-zero cost):** While 100 in-flight calls are executing the async BTC-checker inter-canister call and ICRC-2 `transfer_from`, the guard set is full. Every other user's `retrieve_btc_with_approval` call returns `TemporarilyUnavailable("too many concurrent requests")`. The attacker only needs ICRC-2 approvals in place; the ckBTC burn happens inside the guarded async section and is not required to succeed for the guard to be held. This is a concrete, repeatable application/platform-level DoS on the ckBTC minter's withdrawal path, matching the **High ($2,000–$10,000)** impact class: *Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS* and *Significant Chain Fusion, ck-token … security impact with concrete user or protocol harm.*

**Phase 2 — Queue exhaustion (capital-recoverable cost):** Each completed wave of 100 successful calls adds 100 entries to `retrieve_btc_accounts`. After 50 waves the queue reaches `MAX_CONCURRENT_PENDING_REQUESTS = 5000`, causing every user (including legitimate ones) to receive `TemporarilyUnavailable("too many pending retrieve_btc requests")` until the Bitcoin network drains the backlog. The attacker recovers capital as BTC; net cost is miner fees and ledger fees only.

## Likelihood Explanation

Phase 1 requires no privileged access, no key material, and no victim interaction. Any ICRC-2 account holder can execute it. The only precondition is having ICRC-2 approvals set up for 100 subaccounts on the ckBTC ledger, which is a trivial on-chain operation. The attack is repeatable indefinitely. Phase 2 additionally requires holding enough ckBTC for 5000 minimum-amount withdrawals, which is non-trivial but the capital is fully recovered, making it economically feasible for a motivated attacker.

## Recommendation

1. **Key the guard on `caller` (principal) only**, not on the full `Account`. One in-flight `retrieve_btc_with_approval` per principal is sufficient for replay protection, matching the behavior of `retrieve_btc`.
2. **Add a per-principal cap on pending queue entries** before accepting a new request — count how many `RetrieveBtcRequest` entries in the pending queue belong to a given principal and reject if it exceeds a reasonable per-user limit.
3. Alternatively, track in-flight calls as `BTreeMap<Principal, usize>` and cap per-principal concurrency at a small value (e.g., 1 or 5).

```rust
    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }
```
