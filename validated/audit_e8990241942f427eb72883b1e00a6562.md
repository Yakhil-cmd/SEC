### Title
SNS Governance `default_followees` Feature Is Permanently Broken — (`rs/sns/governance/src/types.rs`, `rs/sns/governance/src/governance.rs`)

---

### Summary

The SNS Governance canister exposes a `default_followees` field in `NervousSystemParameters` that is supposed to configure the default following behavior for every newly claimed neuron. However, the validation layer enforces that this field must always be empty, making the feature permanently non-functional. Any SNS community that attempts to configure default followees via a `ManageNervousSystemParameters` proposal will have the proposal rejected. Newly claimed neurons always receive an empty followees map, regardless of what the SNS community intended.

---

### Finding Description

`NervousSystemParameters.default_followees` is defined in the SNS governance proto and Rust structs as the set of followees that every newly created neuron will follow per proposal function. The field is explicitly documented as having no effect:

```
// TODO NNS1-2169: This field currently has no effect.
// TODO NNS1-2169: Design and implement this feature.
optional DefaultFollowees default_followees = 6;
``` [1](#0-0) 

The validation function `validate_default_followees` in `NervousSystemParameters` enforces that the field must always be empty:

```rust
fn validate_default_followees(&self) -> Result<(), String> {
    self.default_followees
        .as_ref()
        .ok_or_else(|| "NervousSystemParameters.default_followees must be set".to_string())
        .and_then(|default_followees| {
            if default_followees.followees.is_empty() {
                Ok(())
            } else {
                Err(format!(
                    "DefaultFollowees.default_followees must be empty, but found {:?}",
                    default_followees.followees
                ))
            }
        })
}
``` [2](#0-1) 

Despite this, the `claim_neuron` function reads `default_followees_or_panic()` and assigns it to every newly created neuron:

```rust
let neuron = Neuron {
    ...
    followees: self.default_followees_or_panic().followees,
    ...
};
``` [3](#0-2) 

The getter `default_followees_or_panic` is also annotated with the same TODO:

---

### Impact Explanation

Every SNS instance on the Internet Computer is affected. The `default_followees` feature is permanently non-functional:

1. Any `ManageNervousSystemParameters` proposal that attempts to set non-empty `default_followees` will be **rejected** by the validation layer.
2. Every neuron claimed via `manage_neuron` → `ClaimOrRefresh` will always receive an **empty followees map**, regardless of what the SNS community configured.
3. SNS communities have no mechanism to configure default following behavior for newly claimed neurons, even though the parameter is publicly documented and exposed in the governance API.

The consequence is that newly claimed neurons in any SNS have no default following behavior, which can reduce governance participation and voting power utilization. SNS communities that rely on this feature to bootstrap governance participation are silently denied the ability to use it.

---

### Likelihood Explanation

The likelihood is **high**. The broken state is permanent and affects every SNS canister deployed on the IC mainnet. Any SNS community that reads the documentation for `NervousSystemParameters.default_followees` and attempts to configure it will encounter a rejected proposal. The feature has been broken since SNS governance was deployed, as evidenced by the long-standing `TODO NNS1-2169` comment present across multiple files. [7](#0-6) 

The entry path is fully reachable by any unprivileged user: calling `manage_neuron` with `ClaimOrRefresh` triggers `claim_neuron`, which reads the always-empty `default_followees` and assigns it to the new neuron.

---

### Recommendation

Implement the `default_followees` feature as documented, or remove the field from `NervousSystemParameters` entirely if the feature is not intended to be supported. If implementing:

1. Remove the enforcement in `validate_default_followees` that rejects non-empty values.
2. Add validation that each followee neuron ID in `default_followees` exists in the governance state.
3. Ensure the `claim_neuron` path correctly applies the configured followees to newly created neurons.
4. Add a corresponding entry in the `ManageNervousSystemParameters` proposal handling to allow updating this field.

If the feature is intentionally deferred, the field should be removed from the public API to avoid misleading SNS communities.

---

### Proof of Concept

1. Deploy an SNS canister (or use an existing one on mainnet).
2. Submit a `ManageNervousSystemParameters` proposal with a non-empty `default_followees` map (e.g., mapping proposal function ID `0` to a valid neuron ID).
3. Observe that the proposal **fails validation** with: `"DefaultFollowees.default_followees must be empty, but found ..."`.
4. Alternatively, claim a new neuron via `manage_neuron` → `ClaimOrRefresh` and inspect the resulting neuron's `followees` field — it will always be an empty map, regardless of any `default_followees` configuration attempt. [8](#0-7) [9](#0-8)
