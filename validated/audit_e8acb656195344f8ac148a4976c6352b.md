### Title
Anyone Can Force SNS Swap Participation on Behalf of Any Buyer, Bypassing Confirmation Text Consent - (`rs/sns/swap/canister/canister.rs`)

### Summary

The `refresh_buyer_tokens` update method in the SNS Swap canister accepts an arbitrary `buyer` principal in its request body and performs no check that the caller matches the specified buyer. When an SNS swap requires a `confirmation_text` (a consent mechanism requiring participants to explicitly agree to terms), any unprivileged caller can supply another user's principal as `buyer` along with the publicly readable confirmation text, forcing that user's ICP into the swap without their explicit consent.

### Finding Description

The canister entry point `refresh_buyer_tokens` in `rs/sns/swap/canister/canister.rs` resolves the buyer principal from the request body rather than from the IC caller:

```rust

        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()  // ← attacker-controlled
    };
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await

The inner function `refresh_buyer_token_e8s` validates the `confirmation_text` against the SNS-configured text stored in the swap's `Init` state:

The `confirmation_text` is set during SNS initialization and is publicly readable from the swap canister's state. The proto documentation explicitly describes it as a consent mechanism: *"An optional text that swap participants should confirm before they may participate in the swap."* [3](#0-2) 

The code comment on `refresh_buyer_token_e8s` states *"the caller of this function must accept this confirmation"*, but because the `buyer` field decouples the acting principal from the IC caller, the confirmation is never actually verified to come from the buyer. [4](#0-3) 

### Impact Explanation

An attacker can force any user who has pre-funded the swap canister's subaccount (a required preparatory step before calling `refresh_buyer_tokens`) into the swap without that user's explicit consent. The confirmation text — intended as a legal/terms-of-service gate — is rendered meaningless because it is public and the caller identity is never verified against the buyer identity. If the swap commits, the victim receives SNS tokens instead of their ICP, with no recourse to reclaim ICP via `error_refund_icp` (which only operates before finalization). [5](#0-4) 

### Likelihood Explanation

The attack requires no privileged access. The attacker only needs to:
1. Read the public `confirmation_text` from the swap canister's init state.
2. Observe that a victim has transferred ICP to the swap canister's subaccount (visible on the ICP ledger).
3. Submit a single `refresh_buyer_tokens` call with `buyer = victim_principal` and the correct `confirmation_text`.

All three steps are trivially achievable by any unprivileged ingress sender. The swap is in the `Open` state for a defined window, giving the attacker ample time. [6](#0-5) 

### Recommendation

Enforce that the `buyer` field, when non-empty, must equal the IC caller's principal. If third-party notification is intentionally desired (e.g., for automation), the confirmation text check must be removed or the consent must be recorded separately per-buyer via a signed message. The minimal fix:

```rust
#[update]
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let requested = PrincipalId::from_str(&arg.buyer).unwrap();
        // Enforce caller == buyer when confirmation_text is required
        if swap().init_or_panic().confirmation_text.is_some() {
            assert_eq!(caller, requested, "caller must match buyer when confirmation_text is set");
        }
        requested
    };
    ...
}
```

### Proof of Concept

1. SNS swap is deployed with `confirmation_text = "I agree to the terms"` and is in `Open` state.
2. Victim transfers 10 ICP to `subaccount(swap_canister, victim_principal)` on the ICP ledger as a preparatory step, but has not yet called `refresh_buyer_tokens`.
3. Attacker queries the swap canister's `get_init` (public query) to read `confirmation_text = "I agree to the terms"`.
4. Attacker submits:
   ```
   refresh_buyer_tokens({
     buyer: "<victim_principal>",
     confirmation_text: "I agree to the terms"
   })
   ```
   from their own principal (no special privileges needed).
5. The swap canister registers the victim's 10 ICP as committed participation. The victim's consent was never obtained; the confirmation text was provided by the attacker. If the swap commits, the victim receives SNS tokens and cannot reclaim their ICP. [7](#0-6) [8](#0-7)

        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

            let account = Account {
                owner: this_canister.get().0,
                subaccount: Some(principal_to_subaccount(&buyer)),
            };
            icp_ledger
                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
        };
