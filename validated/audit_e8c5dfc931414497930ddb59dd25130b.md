### Title
Unbounded Iteration Over All Block Events Without Native Resource Charging in `eip6110_events_parser` — (`basic_bootloader/src/bootloader/block_flow/ethereum/eip_6110_deposit_events_parser/mod.rs`)

---

### Summary

`eip6110_events_parser` iterates over every event emitted in the entire block to find EIP-6110 deposit events. Unlike the analogous EIP-7002 and EIP-7251 post-op handlers — which enforce hard caps (`MAX_WITHDRAWAL_REQUESTS_PER_BLOCK = 16` and `MAX_CONSOLIDATION_REQUESTS_PER_BLOCK = 2`) and use bounded storage reads — the EIP-6110 parser has no cap on the number of deposit events processed and charges zero native resources for the SHA-256 hashing work it performs per deposit event. An unprivileged caller can fill a block with deposit-contract calls, causing the post-op phase to perform unbounded, unaccounted proving work.

---

### Impact Explanation

**Vulnerability class:** Resource accounting bug / unbounded iteration with unaccounted proving cost.

The post-op phase runs outside any user-controlled resource budget. Because the proving cost of the EIP-6110 scan is not charged to any resource, the sequencer's resource model underestimates the true proving cost of a block containing many deposit events. In the proving mode (RISC-V / Airbender), the prover has a fixed circuit capacity. A block that the sequencer accepts as valid (within resource limits) may exceed the prover's cycle budget, causing:

- **Block unprovability / liveness failure**: The block cannot be proved, stalling the chain.
- **Resource accounting divergence between forward and proving modes**: The forward system accepts the block; the proving system cannot complete it.

With a 30M gas block limit and a minimum deposit transaction cost of ~25,000–30,000 gas, an attacker can pack ~1,000 deposit events into a single block, generating ~39,400,000 unaccounted native units of SHA-256 work in the post-op phase alone — on top of the normal block proving cost.

---

### Likelihood Explanation

**Medium.** The attack requires:
1. The Ethereum deposit contract (`0x00000000219ab540356cBB839Cbe05303d7705Fa`) to be deployed and callable in the ZKsync OS Ethereum-mode environment.
2. An attacker to submit many transactions calling the deposit contract in a single block.

Both conditions are realistic in a live Ethereum-equivalent deployment. The attacker pays EVM gas for the deposit calls but does not pay for the post-op proving overhead. The attack is repeatable every block.

---

### Recommendation

1. **Enforce a `MAX_DEPOSIT_REQUESTS_PER_BLOCK` cap** in `eip6110_events_parser`, consistent with EIP-7002 (`MAX_WITHDRAWAL_REQUESTS_PER_BLOCK = 16`) and EIP-7251 (`MAX_CONSOLIDATION_REQUESTS_PER_BLOCK = 2`). The Ethereum spec does not mandate a cap for EIP-6110, but ZKsync OS must enforce one to bound proving cost.

2. **Charge native resources** for the SHA-256 hashing work performed per deposit event inside `validate_and_write_event_data`, using the existing `SHA256_BASE_NATIVE_COST` / `SHA256_ROUND_NATIVE_COST` constants.

3. **Charge native resources for the full-block event scan** (the `events_iterator()` loop), proportional to the total number of events iterated, to prevent the scan itself from being a free operation.

---

### Proof of Concept

1. Deploy a contract on the ZKsync OS Ethereum-mode chain that calls the EIP-6110 deposit contract (`0x00000000219ab540356cBB839Cbe05303d7705Fa`) in a loop, emitting the maximum number of valid deposit events per transaction.
2. Submit enough such transactions to fill a block (e.g., ~1,000 deposit events at ~25,000 gas each in a 30M gas block).
3. The sequencer accepts the block (all transactions are within their individual gas/native limits).
4. `eip6110_events_parser` is called in `post_op` with no resource budget. It iterates all ~1,000 deposit events and performs ~9,000 SHA-256 rounds (~39,400,000 native units) of unaccounted work.
5. In the proving mode, the RISC-V binary exceeds its circuit cycle budget during the post-op phase, causing the block to be unprovable.
