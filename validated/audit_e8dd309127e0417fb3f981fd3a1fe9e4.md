Audit Report

## Title
Unbounded Header Count in `read_request` Enables Memory Amplification via `http::HeaderMap` — (`rs/p2p/quic_transport/src/request_handler.rs`)

## Summary
The `read_request` function caps raw stream bytes at `MAX_MESSAGE_SIZE_BYTES` (128 MiB) but places no limit on the number of protobuf-encoded headers. A Byzantine subnet peer can craft a 128 MiB protobuf message containing millions of minimal headers; deserializing them into an `http::HeaderMap` causes per-entry heap allocations that amplify memory consumption well beyond the raw byte cap. With up to 1,000 concurrent bidirectional streams per connection, a single compromised node can exhaust the victim replica's memory and cause an OOM crash.

## Finding Description
In `read_request` (`rs/p2p/quic_transport/src/request_handler.rs`, lines 173–207):

```rust
// Raw bytes capped at 128 MiB
let request_bytes = recv_stream
    .read_to_end(MAX_MESSAGE_SIZE_BYTES)   // lib.rs:74 — 128 * 1024 * 1024
    .await ...?;

// Protobuf decoded — total header payload ≤ 128 MiB, but count is unbounded
let request_proto = pb::HttpRequest::decode(request_bytes.as_slice())?;

// NO header count guard
for h in request_proto.headers {
    let pb::HttpHeader { key, value } = h;
    request_builder = request_builder.header(key, value);  // heap-allocates per entry
}
```

A `pb::HttpHeader` with a 1-byte key and 1-byte value encodes to approximately 8 bytes in protobuf wire format. A 128 MiB message therefore fits ≈16.8 million such headers. Each call to `request_builder.header(key, value)` causes the `http` crate to:
1. Heap-allocate a `HeaderName` (custom name → `Bytes::copy_from_slice`, minimum allocator block ≈32 bytes).
2. Heap-allocate a `HeaderValue` (`Bytes::copy_from_slice`, minimum allocator block ≈32 bytes).
3. Insert a `Bucket<HeaderValue>` struct (≈60 bytes) into the `HeaderMap`'s internal `Vec<Bucket>`.

Per-header overhead ≈124 bytes. 16.8 M × 124 bytes ≈ **2.08 GB per stream** from a 128 MiB input (~16× amplification). The raw-byte cap in `read_to_end` does not constrain this amplification because it only limits the serialized protobuf size, not the number of decoded objects or their expanded in-memory representation.

The stream concurrency limit is `MAX_CONCURRENT_BIDI_STREAMS = 1,000` (`connection_manager.rs`, line 74). Each accepted stream is immediately spawned as an independent tokio task with no backpressure (`request_handler.rs`, lines 79–90). All 1,000 tasks can be reading and expanding headers concurrently: 1,000 × 2.08 GB ≈ **2 TB** → OOM → replica crash.

## Impact Explanation
A single Byzantine subnet node can crash any honest replica it is connected to by exhausting its memory. Crashing enough honest replicas on a subnet disrupts consensus and subnet availability. This matches the allowed High impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation
The attack requires exactly one compromised subnet node holding a valid TLS node certificate. The QUIC transport layer is explicitly designed to operate with Byzantine peers (mutual TLS authentication is enforced, but a single Byzantine peer is within the expected threat model). The crafted message is valid protobuf, passes all existing checks, and requires no race conditions or special timing. The attacker freely controls header count and sizes within the 128 MiB raw-byte budget. The attack is repeatable and deterministic.

## Recommendation
Add a header count guard immediately after protobuf decode, before the loop:

```rust
const MAX_HEADERS_PER_REQUEST: usize = 100;

if request_proto.headers.len() > MAX_HEADERS_PER_REQUEST {
    return Err(P2PError::from("too many headers".to_string()));
}
```

Additionally, bound individual header key and value lengths independently of total message size to prevent analogous amplification through large individual values.

## Proof of Concept
1. Construct a `pb::HttpRequest` with `N = floor(128 MiB / 8)` ≈ 16.8 M entries in the `headers` repeated field, each `HttpHeader` having `key = b"a"` and `value = b"b"`.
2. Encode to a byte buffer (total ≤ 128 MiB).
3. From a compromised subnet node, open 1,000 QUIC bidirectional streams to the victim replica and send this message on each stream simultaneously.
4. Monitor victim RSS — it will grow toward ~2 TB before the OOM killer terminates the replica process.

A deterministic integration test can reproduce this with a single stream and a smaller `N` (e.g., 1 M headers) to confirm memory growth proportional to header count without triggering a full OOM.
