### Title
Pubdata Fee Systematically Undercharged Due to Floor Division in `native_per_pubdata` Calculation - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS's L2 transaction validation, the `native_per_pubdata` ratio is computed using floor (truncating) division, while the analogous `native_per_gas` ratio uses ceiling division. This asymmetry causes pubdata fees to be systematically undercharged. When `pubdata_price < native_price`, `native_per_pubdata` truncates to zero, making pubdata completely free. The operator (fee beneficiary) receives less than the actual cost of publishing data to L1, directly analogous to the M-10 dilution pattern where the fee beneficiary receives less value than the nominal fee amount.

---

### Finding Description

In `validation_impl.rs`, the two critical resource-pricing ratios are computed as follows:

**`native_per_gas`** — uses **ceiling division** (`div_ceil`): [1](#0-0) 

**`native_per_pubdata`** — uses **floor division** (`wrapping_div`): [2](#0-1) 

The ceiling division for `native_per_gas` is intentional: it ensures the operator is never underpaid for computational resources. However, `native_per_pubdata` uses `wrapping_div` (Rust's standard integer division, which truncates toward zero), creating a systematic asymmetry.

`native_per_pubdata` is then passed into `get_resources_to_charge_for_pubdata`, which multiplies it by the number of pubdata bytes consumed: [3](#0-2) 

If `native_per_pubdata = 0` (which occurs whenever `pubdata_price < native_price`), the product is zero and the user pays **nothing** for pubdata. The same zero value propagates into `create_resources_for_tx` for the intrinsic pubdata overhead: [4](#0-3) 

The undercharged native resource consumption flows into `compute_gas_refund`, where `native_used` is lower than it should be, `delta_gas` is correspondingly smaller, and the final `gas_used` charged to the user is reduced: [5](#0-4) 

Because the operator is paid `gas_used * gas_price_for_operator`, a lower `gas_used` directly reduces the operator's token revenue.

The same floor-division path exists in the L2 ZK transaction validation flow: [6](#0-5) 

The documentation confirms the intended formula is `nativePerGas := gasPrice / nativePrice` and `deltaGas := (nativeUsed / nativePerGas) - gasUsed`, but the rounding direction for pubdata is never specified as intentionally downward: [7](#0-6) 

---

### Impact Explanation

**Scenario A — Partial underpayment** (`pubdata_price >= native_price`):  
For every pubdata byte consumed, the user is charged `floor(pubdata_price / native_price)` native units instead of `ceil(pubdata_price / native_price)`. The per-byte shortfall is `(pubdata_price % native_price)` native units. Over a transaction generating `P` pubdata bytes, the operator is underpaid by up to `P * (native_price - 1)` native units, which translates to reduced `gas_used` and reduced token payment.

**Scenario B — Complete pubdata fee bypass** (`pubdata_price < native_price`):  
`native_per_pubdata = floor(pubdata_price / native_price) = 0`. Pubdata is entirely free. A user can write to as many storage slots as their EVM gas budget allows, generating arbitrarily large pubdata with zero additional cost. The operator bears the full L1 publication cost with no compensation. This is the direct analog to M-10: the nominal fee (pubdata_price > 0) implies the operator should be compensated, but the actual amount received is zero.

---

### Likelihood Explanation

- Every L2 transaction that writes to storage generates pubdata and goes through this code path.
- The operator sets both `pubdata_price` and `native_price` as block context parameters. If `pubdata_price` is set to any value less than `native_price` (e.g., `pubdata_price = 9`, `native_price = 10`), the bypass is active for the entire block.
- The asymmetry with `native_per_gas` (which uses `div_ceil`) is subtle; an operator configuring these parameters may not realize that `pubdata_price < native_price` silently zeroes out all pubdata charges.
- Any unprivileged transaction sender can trigger this path with a standard EIP-1559 transaction containing storage writes.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata` to match the rounding direction used for `native_per_gas`:

```rust
// Before (floor division — undercharges pubdata):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `prepare_and_check_resources` for L1 transactions if a similar floor-division path exists there.

### Proof of Concept

**Setup**: Block context with `native_price = 10`, `pubdata_price = 9`, `eip1559_basefee = 9`.

**Step 1**: Validation computes:
```
native_per_gas    = ceil(9 / 10) = 1
native_per_pubdata = floor(9 / 10) = 0   ← zero
```

**Step 2**: User submits a transaction that writes to 100 storage slots, generating ~3,200 pubdata bytes.

**Step 3**: `get_resources_to_charge_for_pubdata` computes:
```
native_for_pubdata = 3200 * 0 = 0
```
The user's native resource budget is not reduced by pubdata at all.

**Step 4**: `compute_gas_refund` computes `native_used` as if no pubdata was consumed. `delta_gas = 0`. The user pays only for EVM computation gas.

**Step 5**: The operator receives `gas_used * gas_price_for_operator` tokens, which reflects only computational cost — zero compensation for 3,200 bytes of L1 pubdata publication.

**Expected behavior**: With `div_ceil`, `native_per_pubdata = ceil(9/10) = 1`, and the user would be charged `3200 * 1 = 3200` native units for pubdata, correctly compensating the operator.

**File:** docs/double_resource_accounting.md (L37-50)
```markdown
First we define the ratio between EVM gas and native resource as:
  `nativePerGas := gasPrice/nativePrice`
Note: for call simulation we use a constant for it, as gasPrice might be set to 0.

Next we define the limit for the native resource as:
  `nativeLimit := gasLimit * nativePerGas`

Then we process the transaction, charging both Ergs for EE execution and native resource for any kind of computation (EE, bootloader or system work).

If execution doesn't run out of native resources, we first charge for pubdata from native resource.
Then we compute the difference between the implicit gas used derived from native resource consumption and the gas used by EEs from the ergs used. We call this value `deltaGas`.
  `deltaGas := (nativeUsed / nativePerGas) - gasUsed`

If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs. This ensures that gas estimation will include additional gas to cover for native resources using just base fee. We expect the base fee to be enough to cover most transactions without the need of additional gas.
```
