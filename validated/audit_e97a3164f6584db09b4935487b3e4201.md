Audit Report

## Title
NI-DKG FS Encryption PoP Accepts G1 Identity Element, Enabling Trivial Share Recovery - (File: rs/crypto/internal/crypto_lib/threshold_sig/bls12_381/src/ni_dkg/fs_ni_dkg/encryption_key_pop.rs)

## Summary
The Schnorr Proof of Possession (PoP) for NI-DKG forward-secure encryption keys does not reject the G1 identity element (point at infinity) as a public key. A malicious node operator can register the identity as their FS encryption public key, pass all validation including `verify_pop`, and cause any DKG dealing encrypted to that key to produce ciphertext chunks that are trivially recoverable by any observer via a small-range discrete log.

## Finding Description

**Root cause — `prove_pop` accepts `(pk=identity, witness=0)`:**

The only instance validity check in `prove_pop` is:
```rust
if instance.public_key != (&instance.g1_gen * witness).to_affine() {
    return Err(EncryptionKeyPopError::InvalidInstance);
}
```
With `witness = Scalar::zero()` and `public_key = G1Affine::identity()`, the RHS evaluates to `(g1 * 0).to_affine() = identity`, so the condition is `false` and no error is returned. The prover then computes `pop_key = pop_base * 0 = identity` and `response = challenge * 0 + random_scalar = random_scalar`, producing a structurally valid PoP.

**`verify_pop` passes for the identity key:**

During verification, `blinder_public_key = identity * (-challenge) + g1 * response = g1 * random_scalar`. This is identical to what the prover committed to, so the recomputed challenge matches `pop.challenge` exactly and `verify_pop` returns `Ok(())`.

**`G1Affine::deserialize` accepts the identity:**

`G1Affine::deserialize` calls `ic_bls12_381::G1Affine::from_compressed`, which accepts the standard compressed encoding of the point at infinity (`0xc0` followed by 47 zero bytes). The test suite at line 791 explicitly confirms this round-trips successfully.

**Full validation chain passes:**

`ValidDkgDealingEncryptionPublicKey::try_from` calls `fs_ni_dkg_pubkey_from_proto` → `PublicKeyWithPop::deserialize` (accepts identity via `G1Affine::deserialize`) → `fs_ni_dkg_pubkey.verify` (calls `verify_pop`, which passes). No check in the chain rejects the identity element.

**`enc_chunks` leaks the plaintext:**

When encrypting to the identity public key, the ciphertext chunk computation is:
```rust
let pk_g1_tbl = G1Projective::compute_mul2_affine_tbl(pk, g1);
let enc_chunks = G1Projective::batch_normalize_array(&pk_g1_tbl.mul2_array(&r, &chunks));
```
With `pk = identity`, this yields `identity * r[i] + g1 * chunk[i] = g1 * chunk[i]`. Each ciphertext chunk is simply the plaintext chunk encoded as a G1 point. Since `CHUNK_BYTES = 2` and `NUM_CHUNKS = 16`, each chunk is a 16-bit value in `[0, 65535]`, recoverable by any observer via a baby-step giant-step discrete log over this small range.

## Impact Explanation

A malicious node operator can register the G1 identity as their FS encryption public key, pass all cryptographic validation, and cause every DKG dealing encrypted to their key to have their secret share trivially recoverable from the public ciphertext. This constitutes disclosure of subnet key shares — the PoP mechanism, which is the sole cryptographic guard against this attack, fails to enforce the discrete-log relation. This matches the allowed Critical impact: *Disclosure of subnet key shares or compromise of threshold signing/key material.*

## Likelihood Explanation

Exploitation requires a governance-approved node operator, which is a Byzantine protocol peer below the consensus fault threshold — explicitly within the IC threat model. The attack is deterministic and repeatable: once the identity key is registered, every subsequent DKG dealing leaks that node's share. No special timing, victim interaction, or additional compromise is required. The PoP is the only cryptographic barrier and it fails unconditionally for this input.

## Recommendation

Add an explicit identity-element rejection in both `prove_pop` and `verify_pop` in `encryption_key_pop.rs`:
```rust
if instance.public_key.is_identity() {
    return Err(EncryptionKeyPopError::InvalidInstance);
}
```
Additionally, add the same check in `ValidDkgDealingEncryptionPublicKey::try_from` in `rs/crypto/node_key_validation/src/lib.rs` before calling `verify`, and in `PublicKeyWithPop::deserialize` in `forward_secure.rs` after deserializing `key_value`.

// prove_pop succeeds with witness=0 and pk=identity
let pop = prove_pop(&instance, &witness, rng).expect("should succeed");

// verify_pop passes
assert_eq!(verify_pop(&instance, &pop), Ok(()));

// Confirm the full validation chain accepts the identity key
// (serialize identity + pop into FsEncryptionPublicKey/FsEncryptionPop proto,

let chunk = Scalar::from_u64(chunk_val);
