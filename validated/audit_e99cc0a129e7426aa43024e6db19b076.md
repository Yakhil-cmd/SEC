### Title
Oracle-Provided Block Metadata Fields Uncommitted to Public Input Allow Malicious Prover to Manipulate Execution - (File: `basic_bootloader/src/bootloader/block_flow/zk/metadata_op.rs`)

---

### Summary

`BlockMetadataFromOracle` is fetched entirely from the oracle and used to drive execution, but only a small subset of its fields (`block_number`, `timestamp`, `block_hashes`) are committed to the ZK proof public input. Critical execution-affecting fields — `eip1559_basefee`, `pubdata_price`, `native_price`, `coinbase`, `gas_limit`, `pubdata_limit`, `mix_hash`, and `blob_fee` — are never committed. A malicious prover can supply arbitrary values for these fields, manipulating fee deductions, transaction validity, PREVRANDAO results, and block reward routing, all while producing a valid proof that L1 accepts.

---

### Finding Description

In `metadata_op.rs`, the entire `BlockMetadataFromOracle` struct is fetched from the oracle with only a gas-limit bounds check:

```rust
let block_level_metadata: BlockMetadataFromOracle =
    oracle.query_with_empty_input(BLOCK_METADATA_QUERY_ID)?;
// Only check: block_gas_limit > MAX_BLOCK_GAS_LIMIT
``` [1](#0-0) 

The `BlockMetadataFromOracle` struct contains twelve fields: [2](#0-1) 

At proof finalization, the `ChainStateCommitment` (which becomes `state_before`/`state_after` in the public input) commits only to `state_root`, `next_free_slot`, `block_number`, `last_256_block_hashes_blake`, and `last_block_timestamp`: [3](#0-2) 

The `BatchOutput` (which becomes `batch_output` in the public input) commits only to `chain_id`, timestamps, DA commitment, L1 tx hashes, L2 logs root, and upgrade tx hash: [4](#0-3) 

The fields `eip1559_basefee`, `pubdata_price`, `native_price`, `coinbase`, `gas_limit`, `pubdata_limit`, `mix_hash`, and `blob_fee` are consumed during execution but appear in neither commitment. The only runtime check on oracle-provided temporal data is a non-decreasing timestamp assertion: [5](#0-4) 

The L1 integration documentation confirms this design: only `chain_id` and `block_timestamp(s)` are validated on the settlement layer. [6](#0-5) 

Because the ZK proof proves "given these oracle inputs, the state transition is correct" but does not prove "the oracle inputs are correct," the prover can freely choose any value for the uncommitted fields. The resulting proof is valid and L1 accepts it.

---

### Impact Explanation

A malicious prover can manipulate the uncommitted oracle fields to:

1. **`eip1559_basefee = 0`**: All EIP-1559 transactions become valid regardless of their `max_fee_per_gas`. The effective gas price drops to `max_priority_fee_per_gas`, eliminating base-fee burns and allowing transactions that should be rejected.
2. **`pubdata_price = 0` / `native_price = 0`**: Users are undercharged for pubdata and native resources. Transactions that would fail due to insufficient funds for these costs succeed instead, draining protocol revenue.
3. **`coinbase = prover_address`**: All priority fees (EIP-1559 tips) are routed to the prover's address rather than the intended beneficiary, constituting direct fund theft.
4. **`mix_hash` = arbitrary value**: PREVRANDAO-dependent smart contracts (lotteries, randomness consumers) receive a prover-chosen value, enabling manipulation of randomness-dependent outcomes.
5. **`gas_limit` / `pubdata_limit` manipulation**: The prover can selectively include or exclude transactions by adjusting block resource limits.

The resulting state root (reflecting wrong balances and fee deductions) is committed to L1 via the proof, making the manipulation permanent and undetectable by the settlement layer.

---

### Likelihood Explanation

The attacker is the prover/sequencer providing oracle inputs in proving mode. The Immunefi scope explicitly lists "prover/forward execution input" as a valid attacker entry point. In ZKsync OS's proving mode, the oracle is CSR-based and all data is provided by the prover as witness input. No external dependency or key compromise is required — the prover simply provides crafted `BlockMetadataFromOracle` data. The oracle security model documentation itself warns: "Oracle responses are treated as untrusted input and MUST be validated." [7](#0-6) 

---

### Recommendation

Commit the execution-critical oracle metadata fields to the public input, or verify them against the previous `ChainStateCommitment`. Specifically:

- Add `eip1559_basefee`, `coinbase`, `gas_limit`, `pubdata_limit`, `pubdata_price`, `native_price`, `mix_hash`, and `blob_fee` to either `ChainStateCommitment` or `BatchOutput` so the L1 contract can verify them.
- Alternatively, derive `eip1559_basefee` deterministically from the previous block's gas usage (as Ethereum does) and assert the derivation inside the proof, eliminating the need for oracle-provided basefee.

---

### Proof of Concept

1. Prover constructs `BlockMetadataFromOracle` with `eip1559_basefee = 0`, `pubdata_price = 0`, `coinbase = prover_address`.
2. `metadata_op` accepts this — only gas limit is checked.
3. Transactions execute: users pay zero basefee and zero pubdata cost; all priority fees go to `prover_address`.
4. State root reflects these wrong balances (users retain ETH they should have paid; prover gains ETH).
5. `post_op` builds `ChainStateCommitment` using only `block_number`, `block_hashes`, `timestamp` — none of the manipulated fields appear.
6. ZK proof is generated over the RISC-V execution; it is valid because the computation was internally consistent.
7. L1 contract verifies the proof: `state_before` matches stored commitment (block_number/timestamp/hashes are correct), `state_after` is accepted as the new canonical state.
8. The manipulated state (wrong fee deductions, stolen priority fees) is permanently committed on L1. [8](#0-7) [9](#0-8)

pub struct ChainStateCommitment {
    pub state_root: Bytes32,
    pub next_free_slot: u64,
    pub block_number: u64,
    pub last_256_block_hashes_blake: Bytes32,
    pub last_block_timestamp: u64,
}

impl ChainStateCommitment {
    ///
    /// Calculate blake2s hash of chain state commitment.
    ///
    /// We are using proving friendly blake2s because this commitment will be generated and opened during proving,
    /// but we don't need to open it on the settlement layer.
    ///
    pub fn hash(&self) -> [u8; 32] {
        let mut hasher = crypto::blake2s::Blake2s256::new();
        hasher.update(self.state_root.as_u8_ref());
        hasher.update(&self.next_free_slot.to_be_bytes());
        hasher.update(&self.block_number.to_be_bytes());
        hasher.update(self.last_256_block_hashes_blake.as_u8_ref());
        hasher.update(&self.last_block_timestamp.to_be_bytes());
        hasher.finalize()
    }

impl BatchOutput {
    ///
    /// Calculate keccak256 hash of public input
    ///
    pub fn hash(&self) -> [u8; 32] {
        let mut hasher = Keccak256::new();
        hasher.update(self.chain_id.to_be_bytes::<32>());
        hasher.update(&self.first_block_timestamp.to_be_bytes());
        hasher.update(&self.last_block_timestamp.to_be_bytes());
