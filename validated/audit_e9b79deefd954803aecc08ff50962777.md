### Title
Unchecked Zero Address for Recipient in `receive_base_tokens` — (File: `engine/src/engine.rs`)

### Summary
`receive_base_tokens` mints ETH base tokens to a recipient address parsed from user-controlled calldata (`args.msg`) without any zero-address check. An unprivileged user can supply the zero address as the recipient, permanently freezing their deposited ETH at `0x0000000000000000000000000000000000000000` on Aurora with no recovery path.

### Finding Description
The `ft_on_transfer` entry point in `engine/src/contract_methods/connector.rs` dispatches to `engine.receive_base_tokens(&args)` when the predecessor is the eth-connector account. [1](#0-0) 

`receive_base_tokens` in `engine/src/engine.rs` parses the recipient from `args.msg` via `FtTransferMessageData::try_from`: [2](#0-1) 

The parser in `engine-types/src/parameters/connector.rs` accepts any valid 40-character hex string — including 40 zeros — and returns the zero address without error: [3](#0-2) 

After parsing, `set_balance` is called directly on the returned address with no zero-address guard: [4](#0-3) 

No check of the form `if recipient == Address::zero() { return Err(...) }` exists anywhere in this call chain.

### Impact Explanation
ETH base tokens minted to `Address::zero()` on Aurora are permanently inaccessible. No private key controls the zero address; the balance can never be spent or recovered. This constitutes a **permanent freezing of funds** (Critical).

### Likelihood Explanation
Any user holding ETH on NEAR (NEP-141 tokens from the eth-connector) can trigger this by calling `ft_transfer_call` on the eth-connector with `receiver_id = <aurora_engine_account>` and `msg = "0000000000000000000000000000000000000000"`. No special privileges are required. The operation succeeds silently — the contract returns `"0"` (no refund), confirming the tokens were accepted and minted to the zero address.

### Recommendation
Add a zero-address guard in `receive_base_tokens` before crediting the balance:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let receipient = message_data.recipient;
    // ADD: reject zero address
    if receipient == Address::zero() {
        return Err(errors::ERR_ZERO_ADDRESS.into());
    }
    // ...
}
```

Alternatively, add the check inside `FtTransferMessageData::try_from` so all callers benefit.

### Proof of Concept

**Attacker-controlled entry path:**

1. Attacker holds ETH on NEAR (NEP-141 from the eth-connector).
2. Attacker calls `ft_transfer_call` on the eth-connector contract:
   - `receiver_id`: Aurora engine account ID
   - `amount`: any non-zero amount (e.g., `"1000"`)
   - `msg`: `"0000000000000000000000000000000000000000"` (40 hex zeros = zero address)
3. The eth-connector calls `ft_on_transfer` on the Aurora engine.
4. `ft_on_transfer` confirms `predecessor_account_id == eth_connector_account_id` and calls `receive_base_tokens`.
5. `FtTransferMessageData::try_from("0000000000000000000000000000000000000000")` succeeds, returning `recipient = Address::zero()`.
6. `set_balance(&mut self.io, &Address::zero(), &new_balance)` is called — ETH is credited to the zero address.
7. `ft_on_transfer` returns `"0"` (zero refund), confirming the tokens were consumed.
8. The deposited ETH is permanently frozen at the zero address on Aurora.

    pub fn receive_base_tokens(
        &mut self,
        args: &FtOnTransferArgs,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
        let amount = Wei::new_u128(args.amount.as_u128());
        let receipient = message_data.recipient;
        let balance = get_balance(&self.io, &receipient);
        let new_balance = balance
            .checked_add(amount)
            .ok_or(errors::ERR_BALANCE_OVERFLOW)?;

        set_balance(&mut self.io, &receipient, &new_balance);

        sdk::log!("Mint {amount} base tokens for: {}", receipient.encode());

        Ok(None)
    }
