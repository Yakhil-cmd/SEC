Audit Report

## Title
Precision Loss in ICRC-21 Consent Message Token Amount Display via `f64` Conversion - (File: `packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
The `convert_tokens_to_string_representation()` function converts an arbitrary-precision `Nat` token amount to `f64` before formatting it for display in ICRC-21 `GenericDisplayMessage` consent messages. Because `f64` has only 53 bits of mantissa, any integer above 2^53 (≈ 9.007 × 10^15) is silently rounded. This causes the consent message shown to a user on a hardware wallet or signing device to display a different amount than what they are actually authorizing on-chain.

## Finding Description
The vulnerable function is at `packages/icrc-ledger-types/src/icrc21/responses.rs` lines 318–327:

```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let tokens = tokens.0.to_f64().ok_or(Icrc21Error::GenericError {
        error_code: Nat::from(500_u64),
        description: "Failed to convert tokens to u64".to_owned(),
    })?;
    Ok(format!("{}", tokens / 10_f64.pow(decimals)))
}
```

`Nat` wraps a `BigUint` (arbitrary precision). The call `tokens.0.to_f64()` silently rounds any value above 2^53 to the nearest representable `f64`. The rounded value is then divided by `10^decimals` and formatted as the displayed amount. No error is returned — the function succeeds and returns a subtly wrong string.

This function is called from four public methods on `ConsentMessage`, all of which populate the `GenericDisplayMessage` path:
- `add_amount()` (L113)
- `add_fee()` (L141)
- `add_allowance()` (L186)
- `add_existing_allowance()` (L212)

The `GenericDisplayMessage` path is the default when no `device_spec` is specified in the consent message request (lib.rs L173–175), meaning it is the path exercised by hardware wallets that do not specify a structured display type.

The `FieldsDisplayMessage` path uses `nat_to_u64()` instead, which is a separate (also lossy, but differently bounded) conversion and is not the subject of this finding.

No existing guard checks whether the `Nat` value exceeds 2^53 before the conversion. The `to_f64()` call from `num_traits::ToPrimitive` returns `None` only for values that cannot be represented as any finite `f64` (i.e., values exceeding ~1.8 × 10^308), not for values that lose integer precision. All realistic token amounts succeed silently with a rounded result.

## Impact Explanation
ICRC-21 consent messages are the security-critical display layer that hardware wallets and signing devices use to show users exactly what they are signing. A discrepancy between the displayed amount and the actual on-chain amount directly undermines this trust guarantee.

Concrete precision loss examples:
- **ckETH (18 decimals):** Any transfer above ~9,007 ETH (2^53 wei) has its displayed amount rounded. `9007199254740993 wei` displays as `9007199254740992 wei`.
- **ICP (8 decimals):** Any transfer above ~90 million ICP (2^53 e8s) is affected.
- **ICRC-1 tokens with 0 decimals:** Any amount above 2^53 ≈ 9 × 10^15 tokens is affected; the rounding error grows with magnitude (ULP at 2^63 is 2048 units, at 2^73 is ~2 million units).

A malicious dApp can craft a `TransferArg` with an amount X > 2^53 such that `to_f64()` rounds it down to Y < X. The consent message displays Y; the user approves; the actual ledger transfer executes for X. The attacker receives X while the user believed they authorized Y. This matches the **"Significant ck-token, ledger security impact with concrete user harm"** category at **Medium** severity, given that the per-transaction financial delta is small for common tokens at typical amounts but the integrity of the consent display mechanism is concretely broken for any amount above the 2^53 threshold.
