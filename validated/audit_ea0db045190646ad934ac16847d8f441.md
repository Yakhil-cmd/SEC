Audit Report

## Title
Unbounded Neuron Iteration in SNS `compute_ballots_for_new_proposal` Can Exhaust Instruction Limit, Permanently Blocking Proposal Submission - (File: rs/sns/governance/src/governance.rs)

## Summary
`make_proposal` in the SNS Governance canister unconditionally calls `compute_ballots_for_new_proposal`, which performs a synchronous, unbounded iteration over every neuron in `self.proto.neurons` with no instruction-limit guard. When the neuron population is large enough, every call to `make_proposal` traps with `CanisterInstructionLimitExceeded`, permanently preventing new governance proposals from being submitted to that SNS instance.

## Finding Description
`make_proposal` calls `compute_ballots_for_new_proposal` at [1](#0-0)  before any proposal is inserted. `compute_ballots_for_new_proposal` contains a plain `for (k, v) in self.proto.neurons.iter()` loop at [2](#0-1)  that, per iteration, calls `v.dissolve_delay_seconds()`, `v.voting_power()` (multi-factor arithmetic), and `BTreeMap::insert`. There is no call to `ic_cdk::api::instruction_counter()`, no `over_soft_message_limit()` check, and no `noop_self_call_if_over_instructions` escape hatch anywhere in the SNS governance code path — confirmed by a search returning zero matches for those symbols across `rs/sns/governance/**`.

By contrast, the NNS Governance canister defines explicit hard and soft instruction limits [3](#0-2)  and uses `noop_self_call_if_over_instructions` inside its voting loop to yield before the limit is hit. [4](#0-3)  The SNS has neither protection.

SNS canisters run on application subnets, where the update-message instruction limit is 5 × 10⁹. With per-neuron work estimated at tens of thousands of instructions (dissolve-delay read + voting-power arithmetic + `BTreeMap::insert` at O(log n)), the threshold is reachable well within the `max_number_of_neurons` ceiling defined in `NervousSystemParameters`. Once the threshold is crossed, the trap is deterministic and repeatable on every subsequent `make_proposal` call because the neuron map size is unchanged.

## Impact Explanation
This is an application/platform-level DoS on SNS governance proposal submission. No new proposals can be submitted to the affected SNS — including any remediation proposal that would itself require `make_proposal` to succeed. Ongoing proposals are unaffected, but the governance system is permanently frozen for new actions. This matches the allowed High impact: "Application/platform-level DoS… or subnet availability impact not based on raw volumetric DDoS" and "Significant SNS… security impact with concrete user or protocol harm." Severity: **High ($2,000–$10,000)**.

## Likelihood Explanation
Any neuron holder with `SubmitProposal` permission and sufficient stake to cover `reject_cost_e8s` can trigger the trap — no privileged access is required. An adversary can accelerate neuron population growth by splitting existing neurons or staking small amounts. Organic SNS growth can reach the threshold without adversarial action. The condition is permanent once triggered: no subsequent caller can succeed regardless of their identity or stake.

## Recommendation
1. **Adopt the NNS snapshot pattern**: replace the live neuron iteration with a pre-computed, periodically refreshed voting-power snapshot updated in a timer job, analogous to `compute_voting_power_snapshot_for_standard_proposal` in the NNS neuron store.
2. **Add an instruction-limit guard**: if live iteration must be kept, check `ic_cdk::api::instruction_counter()` inside the loop and return a graceful error (not a trap) when approaching the soft limit.
3. **Enforce a safe neuron cap**: lower `max_number_of_neurons` in `NervousSystemParameters` to a value provably safe given the per-neuron instruction cost, and add a benchmark test that enforces the bound.

## Proof of Concept
1. Deploy an SNS with default `NervousSystemParameters`.
2. Create N neurons (via staking and claiming, or by splitting) where N × per-neuron instruction cost > 5 × 10⁹.
3. Call `manage_neuron` → `MakeProposal` from any neuron with `SubmitProposal` permission.
4. Observe the call trap with `CanisterInstructionLimitExceeded`.
5. Confirm that all subsequent `make_proposal` calls trap identically, regardless of the proposer, because the neuron map size is unchanged.

A deterministic integration test using PocketIC can reproduce this by populating the neuron map to the threshold count and asserting that `make_proposal` returns a trap rather than a governance error.
