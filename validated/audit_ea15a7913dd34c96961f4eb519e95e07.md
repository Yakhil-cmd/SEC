### Title
Silo Whitelist Bypass via `deploy_code` Entrypoint — (`engine/src/contract_methods/evm_transactions.rs`)

### Summary

In Aurora Engine's Silo mode, the `deploy_code` NEAR contract entrypoint deploys EVM bytecode without enforcing the `Admin`/`EvmAdmin` whitelist. Any unprivileged NEAR account can call `deploy_code` and deploy arbitrary EVM contracts into a Silo instance even when the whitelist is fully enabled, bypassing the intended access control.

### Finding Description

The Silo whitelisting system defines four lists. The `Admin` (NEAR account) and `EvmAdmin` (EVM address) lists are specifically designed to restrict who may deploy EVM code. The enforcement point is `assert_access` in `engine/src/engine.rs`:

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };
    if !allowed {
        return Err(EngineError { kind: EngineErrorKind::NotAllowed, gas_used: 0 });
    }
    Ok(())
}
``` [1](#0-0) 

`assert_access` is called inside `submit_with_alt_modexp`, which is the path taken by the `submit` and `submit_with_args` entrypoints: [2](#0-1) 

However, the `deploy_code` entrypoint takes a completely different code path:

```rust
pub fn deploy_code<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;          // ← only checks "not paused"
        let input = io.read_input().to_vec();
        let current_account_id = env.current_account_id();
        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&env.predecessor_account_id()),
            current_account_id,
            io,
            env,
        );
        let result = engine.deploy_code_with_input(input, None, handler)?;
        ...
    })
}
``` [3](#0-2) 

`deploy_code` calls only `require_running` (contract not paused) and then directly invokes `engine.deploy_code_with_input`. It never calls `assert_access`, `silo::is_allow_deploy`, or any whitelist check. The `Admin` and `EvmAdmin` whitelists are completely skipped.

The whitelist logic that should be enforced is:

```rust
pub fn is_allow_deploy<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_account_allowed_deploy(io, account) && is_address_allowed_deploy(io, address)
}

fn is_account_allowed_deploy<I: IO + Copy>(io: &I, account_id: &AccountId) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Admin);
    !list.is_enabled() || list.is_exist(account_id)
}

fn is_address_allowed_deploy<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::EvmAdmin);
    !list.is_enabled() || list.is_exist(address)
}
``` [4](#0-3) 

This is the direct analog of the external report: the `submit` path correctly gates deployment through `assert_access`, but the `deploy_code` path silently omits the same gate, allowing non-whitelisted callers to deploy EVM code.

The `deploy_code` method is confirmed as a public NEAR contract entrypoint via the `TransactionKindTag` enum: [5](#0-4) 

### Impact Explanation

In a Silo deployment with `Admin` and `EvmAdmin` whitelists enabled, any NEAR account — including completely unauthorized ones — can call `deploy_code` and deploy arbitrary EVM bytecode. The attacker can:

1. Deploy a malicious ERC-20 contract that mimics a legitimate Silo token, then trick users into approving or transferring funds to it → **direct theft of user funds**.
2. Deploy a contract that permanently locks ETH or ERC-20 balances → **permanent freezing of funds**.

The Silo's entire access control model for contract deployment is rendered ineffective.

**Impact: Critical — direct theft of user funds / permanent freezing of funds.**

### Likelihood Explanation

- The `deploy_code` method is a standard public NEAR contract entrypoint, callable by any NEAR account with no special privileges.
- No leaked keys, governance capture, or social engineering is required.
- The attacker only needs to know the Aurora Silo contract account ID and submit a NEAR transaction calling `deploy_code` with malicious EVM bytecode.
- Silo deployments are production configurations explicitly designed to restrict access; operators enabling the whitelist have a reasonable expectation that `deploy_code` is also gated.

**Likelihood: High.**

### Recommendation

Add a whitelist check at the start of `deploy_code` in `engine/src/contract_methods/evm_transactions.rs`, mirroring the check performed inside `submit_with_alt_modexp`:

```rust
pub fn deploy_code<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;

        // Add silo whitelist enforcement:
        let predecessor = env.predecessor_account_id();
        let sender = predecessor_address(&predecessor);
        if !silo::is_allow_deploy(&io, &predecessor, &sender) {
            return Err(errors::ERR_NOT_ALLOWED.into());
        }

        let input = io.read_input().to_vec();
        ...
    })
}
```

### Proof of Concept

1. Operator deploys Aurora Engine in Silo mode and enables the `Admin` and `EvmAdmin` whitelists (no entries added, so no one is authorized to deploy).
2. Attacker (any NEAR account, e.g. `attacker.near`) calls the `deploy_code` NEAR method on the Silo contract with arbitrary EVM bytecode as input.
3. The call succeeds: `require_running` passes (contract is not paused), `assert_access` is never invoked, and `engine.deploy_code_with_input` executes the deployment.
4. The attacker's contract is now live in the Silo EVM at a deterministic address, despite the `Admin`/`EvmAdmin` whitelists being enabled and the attacker not being listed.
5. The attacker can now interact with the deployed contract to steal funds from users or freeze balances.
