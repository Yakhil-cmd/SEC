### Title
EIP-7702 Chain-Agnostic Authorization (`chain_id = 0`) Enables Cross-Chain Replay to Steal User Funds - (File: `engine-transactions/src/eip_7702.rs`)

### Summary
The Aurora Engine's EIP-7702 implementation accepts authorization tuples with `chain_id = 0` as valid on Aurora regardless of which chain the user originally signed the authorization for. An attacker who observes a victim's chain-agnostic EIP-7702 authorization on any other EVM chain can replay it on Aurora, installing arbitrary delegation code on the victim's Aurora account and draining their ETH balance.

### Finding Description
In `engine-transactions/src/eip_7702.rs`, the `authorization_list()` method on `SignedTransaction7702` validates each authorization tuple's chain ID as follows:

```rust
let mut is_valid = if auth.s > SECP256K1N_HALF {
    false
} else {
    (auth.chain_id.is_zero() || auth.chain_id == current_tx_chain_id)
        && auth.parity <= U256::one()
};
```

When `auth.chain_id` is zero, the authorization is accepted as valid on any chain — including Aurora (chain ID `1313161554`). The comment in the code explicitly acknowledges this: `"chain_id = 0 for chain-agnostic authorization"`. [1](#0-0) 

The outer EIP-7702 transaction's chain ID is separately validated against the engine's configured state chain ID in `engine/src/engine.rs`:

```rust
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [2](#0-1) 

This means the **outer transaction** must carry Aurora's chain ID (correctly enforced), but the **authorization tuple** inside it can carry `chain_id = 0`, which was signed by the victim on a completely different chain (e.g., Ethereum mainnet). The attacker wraps the victim's pre-existing chain-agnostic authorization inside a new EIP-7702 transaction they sign themselves, targeting Aurora.

The analog to the LifeBuoy bug is direct: LifeBuoy assumes that a deployer identity on one chain cannot be replicated on another chain, but shared factories break this assumption. Here, Aurora assumes that a `chain_id = 0` authorization is safe to accept, but it was signed by the user with the intent of delegating on a specific chain — not on Aurora. The user's intent is violated cross-chain.

The attack proceeds as follows:

1. Victim signs an EIP-7702 authorization with `chain_id = 0` on Ethereum mainnet, delegating their account to contract `C` (e.g., a batch-call helper they trust on mainnet).
2. This authorization is publicly visible on-chain.
3. Attacker deploys a malicious contract at the same address `C` on Aurora (possible via CREATE2 with the same factory and salt, or by front-running if the factory exists on both chains).
4. Attacker constructs a new EIP-7702 transaction with Aurora's chain ID, embedding the victim's `chain_id = 0` authorization in the `authorization_list`.
5. Attacker submits this transaction to Aurora via the `submit` entrypoint.
6. `authorization_list()` accepts the authorization (chain_id = 0 passes the check).
7. The EVM installs the delegation designator `EF0100 || C` on the victim's Aurora account.
8. Attacker calls the malicious contract at `C` in the context of the victim's account, draining the victim's ETH balance on Aurora. [3](#0-2) 

### Impact Explanation
An attacker can steal the entire ETH balance of any Aurora user who has ever signed a `chain_id = 0` EIP-7702 authorization on any EVM chain. The delegation code installed on the victim's account executes in the victim's context, giving the attacker full control over the victim's funds. This is direct theft of user funds at rest.

### Likelihood Explanation
EIP-7702 with `chain_id = 0` is explicitly documented as a "chain-agnostic" authorization pattern and is used in practice for account abstraction wallets and batch-call helpers that want a single authorization to work across multiple chains. An attacker can monitor public EVM chains for `chain_id = 0` authorizations, identify victims with ETH balances on Aurora, and replay the authorization. The attacker only needs to ensure a malicious contract exists at the delegated address on Aurora, which is achievable via CREATE2 with a shared factory (e.g., the deterministic deployment proxy at `0x4e59b44847b379578588920cA78FbF26c0B4956C`, which exists on both Ethereum and Aurora).

### Recommendation
Aurora should reject EIP-7702 authorization tuples with `chain_id = 0` entirely, since Aurora is a distinct chain and chain-agnostic authorizations create a cross-chain replay surface. Alternatively, Aurora should treat `chain_id = 0` authorizations as valid only if `chain_id = 0` is also the engine's own configured chain ID (which it never is, since Aurora uses `1313161554`). The fix is a one-line change in `authorization_list()`:

```rust
// Change:
(auth.chain_id.is_zero() || auth.chain_id == current_tx_chain_id)
// To:
auth.chain_id == current_tx_chain_id
``` [4](#0-3) 

### Proof of Concept

1. On Ethereum mainnet, victim signs and submits an EIP-7702 transaction with an authorization tuple `{chain_id: 0, address: C, nonce: N}`. This is publicly visible.

2. Attacker deploys a malicious contract at address `C` on Aurora using the same CREATE2 factory and salt.

3. Attacker constructs:
   ```
   Transaction7702 {
       chain_id: 1313161554,  // Aurora mainnet
       to: victim_address,
       authorization_list: [
           AuthorizationTuple {
               chain_id: 0,   // victim's original chain-agnostic auth
               address: C,
               nonce: N,
               parity, r, s   // victim's original signature
           }
       ],
       ...
   }
   ```

4. Attacker signs the outer transaction with their own key and submits to Aurora via `submit`.

5. In `authorization_list()`, line 181: `auth.chain_id.is_zero()` is `true` → `is_valid = true`.

6. EVM installs `EF0100 || C` as the code of `victim_address` on Aurora.

7. Attacker calls `C.drain(victim_address)` — the malicious contract executes in victim's context and transfers all ETH to the attacker. [5](#0-4) [6](#0-5)

    pub fn authorization_list(&self) -> Result<Vec<Authorization>, Error> {
        if self.transaction.authorization_list.is_empty() {
            return Err(Error::EmptyAuthorizationList);
        }
        let current_tx_chain_id = U256::from(self.transaction.chain_id);
        let mut authorization_list = Vec::with_capacity(self.transaction.authorization_list.len());
        let mut rlp_stream = RlpStream::new();
        let mut message_bytes = vec![MAGIC; 1];
        // According to EIP-7702, we should validate each authorization. We shouldn't skip any of them.
        // And just put the `is_valid` flag to `false` if any of them is invalid. It's related to
        // gas calculation, as each `authorization_list` must be charged, even if it's invalid.
        for auth in &self.transaction.authorization_list {
            // According to EIP-7702 step 1. validation, we should verify it as
            // `chain_id = 0 || current_chain_id`.
            // AS `current_chain_id` we used `transaction.chain_id` as we will validate `chain_id` in
            // Engine `submit_transaction` method.

            // Step 2 - validation logic inside EVM itself.
            // Step 3. Checking: authority = ecrecover(keccak(MAGIC || rlp([chain_id, address, nonce])), y_parity, r, s])
            // Validate the signature, as in tests it is possible to have invalid signature values.
            // Value `v` shouldn't be greater than 1
            let mut is_valid = if auth.s > SECP256K1N_HALF {
                false
            } else {
                (auth.chain_id.is_zero() || auth.chain_id == current_tx_chain_id)
                    && auth.parity <= U256::one()
            };

            let auth_address = if is_valid {
                rlp_stream.begin_list(3);
                rlp_stream.append(&auth.chain_id);
                rlp_stream.append(&auth.address);
                rlp_stream.append(&auth.nonce);

                message_bytes.extend_from_slice(rlp_stream.as_raw());

                let signature_hash = aurora_engine_sdk::keccak(&message_bytes);
                // U256::as_u32() is safe because here we're sure that the parity <= 1.
                let v = u8::try_from(auth.parity.as_u32()).unwrap_or(u8::MAX);
                let auth_address = ecrecover(signature_hash, &super::vrs_to_arr(v, auth.r, auth.s))
                    .unwrap_or_else(|_| {
                        is_valid = false;
                        Address::default()
                    });

                message_bytes.truncate(1);
                rlp_stream.clear();

                auth_address
            } else {
                Address::default()
            };

            // Validations steps 2,4-9 0f EIP-7702 provided by EVM itself.
            authorization_list.push(Authorization {
                authority: auth_address.raw(),
                address: auth.address,
                nonce: auth.nonce,
                is_valid,
            });
        }
        Ok(authorization_list)
    }
