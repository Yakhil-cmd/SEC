### Title
Incorrect `continue` Target in Nested Loop Causes Miller-Rabin Primality Test to Return False Negatives — (`File: callable_oracles/src/hash_to_prime/common.rs`)

---

### Summary

The `miller_rabin_u32` function in `callable_oracles/src/hash_to_prime/common.rs` contains a misuse of `continue` inside a nested loop. When the inner loop finds `result == minus_one` (the witness-passes condition), `continue` advances the **inner** loop instead of the **outer** witness loop. Execution then falls through to `return false`, incorrectly declaring a prime number composite. This is the direct Rust analog of the Solidity `continue` bug in the original report: in both cases, `continue` is applied to the wrong loop scope, causing the loop to behave incorrectly.

---

### Finding Description

In `miller_rabin_u32`, the Miller-Rabin primality test iterates over three fixed witnesses (`[2, 7, 61]`). For each witness `a`, if `a^d mod N` is neither `1` nor `-1`, the code enters an inner loop to check whether any squaring produces `-1`: [1](#0-0) 

```rust
for _r in 1..s {
    result = mont_square_u32(result, candidate, inv);
    if result == minus_one {
        continue;   // ← BUG: continues inner loop, not outer witness loop
    }
}

// no luck
return false;
```

In Rust, an unlabeled `continue` applies to the **innermost** enclosing loop. Here the innermost loop is `for _r in 1..s`, so when `result == minus_one` is found, the code merely advances `_r` and keeps squaring — it does **not** skip to the next witness. After the inner loop exhausts all `r` values (regardless of whether `-1` was ever found), execution unconditionally falls through to `return false`.

The correct behavior requires breaking out of the inner loop and continuing the outer `for a in MILLER_RABIN_BASES` loop (i.e., `continue 'outer` with a loop label, or restructuring with a flag/`break`). [2](#0-1) 

---

### Impact Explanation

`miller_rabin_u32` is the primality oracle for 32-bit prime candidates used in the hash-to-prime construction. The bug causes **false negatives**: primes whose witnesses satisfy the condition `a^(d·2^r) ≡ −1 (mod N)` for some `r ∈ [1, s)` (rather than at `r = 0`) are incorrectly reported as composite.

For the deterministic bases `[2, 7, 61]`, a non-trivial fraction of 32-bit primes will be misclassified. This breaks the hash-to-prime oracle: the oracle will fail to certify valid prime candidates, causing the Pocklington-chain construction to stall or produce an incorrect result. Downstream, any proof system component relying on a correctly-certified prime (e.g., accumulator membership proofs) will either fail to generate a proof or generate an unsound one. [3](#0-2) 

---

### Likelihood Explanation

The bug is triggered whenever a prime candidate's witness satisfies the Miller-Rabin condition through the inner-loop path (i.e., `a^(d·2^r) ≡ −1` for `r ≥ 1`). This is a common case for primes with `s ≥ 2` (i.e., `N − 1` divisible by 4 or higher powers of 2), which is the majority of odd primes. Any transaction that causes the hash-to-prime oracle to be invoked with such a candidate will trigger the bug deterministically. [4](#0-3) 

---

### Recommendation

Add a loop label to the outer witness loop and use `continue 'outer` when `minus_one` is found in the inner loop:

```rust
'witness: for a in MILLER_RABIN_BASES.iter().copied() {
    // ... compute result = a^d mod N ...

    if result == mont_r   { continue 'witness; }
    if result == minus_one { continue 'witness; }

    for _r in 1..s {
        result = mont_square_u32(result, candidate, inv);
        if result == minus_one {
            continue 'witness;  // ← correct: skip to next witness
        }
    }

    return false;
}
true
``` [2](#0-1)

### Proof of Concept

Take the prime `N = 5` (trivial example with `s = 2`, `d = 1`):
- `N − 1 = 4 = 2^2 · 1`, so `s = 2`, `d = 1`.
- Witness `a = 2`: `a^d = 2`. Neither `1` nor `−1 mod 5 = 4`.
- Inner loop `r = 1`: `result = 2^2 mod 5 = 4 = minus_one`. The buggy `continue` advances `_r` to end of range.
- After inner loop: falls through to `return false` — incorrectly reports `5` as composite.

The same pattern applies to any prime where the witness satisfies the condition at `r ≥ 1`, which is the majority of primes in the 32-bit range used by the hash-to-prime oracle. [1](#0-0)

pub fn miller_rabin_u32(candidate: u32) -> bool {
    if candidate & 1 == 0 {
        return false;
    }
    // prime is considered N = 2^s * d + 1;
    let tmp = candidate - 1;
    let s = tmp.trailing_zeros();
    debug_assert!(s > 0);
    let d = tmp >> s;

    // since N is odd, we can compute the Montgomery representation constant
    let mont_r = (1u64 << 32) % (candidate as u64);
    let mont_r2 = (mont_r * mont_r) % (candidate as u64);
    let mont_r = mont_r as u32;
    let mont_r2 = mont_r2 as u32;
    let minus_one = candidate - mont_r;

    let mut inv = 1u32;
    for _ in 0..31 {
        inv = inv.wrapping_mul(inv);
        inv = inv.wrapping_mul(candidate);
    }
    inv = inv.wrapping_neg();

    for a in MILLER_RABIN_BASES.iter().copied() {
        let a = mont_mul_u32(a, mont_r2, candidate, inv);
        // first check that a^d == 1 mod N

        // top bit is set
        let mut result = a;
        let bits = 32 - d.leading_zeros() - 1;
        for i in (0..bits).rev() {
            result = mont_square_u32(result, candidate, inv);
            if d & (1 << i) > 0 {
                result = mont_mul_u32(result, a, candidate, inv);
            }
        }

        if result == mont_r {
            continue;
        }

        // then {a^d)^{2^r} == -1 mod N for 0 < r < s
        if result == minus_one {
            continue;
        }

        for _r in 1..s {
            result = mont_square_u32(result, candidate, inv);
            if result == minus_one {
                continue;
            }
        }

        // no luck
        return false;
    }

    true
}
