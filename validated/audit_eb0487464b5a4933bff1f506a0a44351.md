Audit Report

## Title
SNS Governance `distribute_rewards` Unbounded Neuron Iteration Causes Permanent Reward Distribution Liveness Failure - (`rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance canister's `distribute_rewards` function iterates over all ballots of all `ReadyToSettle` proposals and then over all rewarded neurons in a single synchronous message execution with no instruction-limit guard. Unlike NNS governance, which was explicitly refactored to use a batched, instruction-aware distribution mechanism backed by stable memory, SNS governance performs this work atomically. If the instruction limit is exceeded, the message traps, `latest_reward_event` is not updated, and every subsequent heartbeat re-attempts the same unbounded work — permanently halting reward distribution.

## Finding Description
**Unbounded loops with no instruction guard:**

`distribute_rewards` first builds `neuron_id_to_reward_shares` by iterating over every ballot of every `ReadyToSettle` proposal with no instruction check: [1](#0-0) 

It then iterates over every entry in that map to update neuron maturity, again with no instruction check: [2](#0-1) 

A search for `is_message_over_threshold` in the entire `rs/sns/governance/` tree returns zero matches, confirming there is no instruction guard anywhere in SNS governance.

**Call path — heartbeat-triggered, no privilege required:**

`run_periodic_tasks` calls `distribute_rewards` synchronously after fetching the token supply: [3](#0-2) 

If the call traps, `self.proto.latest_reward_event` is never updated, so `should_distribute_rewards()` returns `true` on the next heartbeat, re-triggering the same unbounded work.

**Contrast with NNS governance (the fixed version):**

NNS governance defines a 1-billion-instruction per-message limit: [4](#0-3) 

`distribute_pending_rewards` uses `is_message_over_threshold(DISTRIBUTION_MESSAGE_LIMIT)` and persists partial progress to a `StableBTreeMap` so the next timer tick can resume: [5](#0-4) 

`continue_processing` breaks after each neuron update when the limit is reached: [6](#0-5) 

The NNS CHANGELOG explicitly documents this fix as a response to the exact same class of bug: [7](#0-6) 

SNS governance has no equivalent mechanism.

## Impact Explanation
If the instruction limit is exceeded during `distribute_rewards`:
- The heartbeat message traps and all state changes are rolled back.
- `latest_reward_event` is not advanced, so the same `ReadyToSettle` proposals remain.
- Every subsequent heartbeat re-attempts the same unbounded work and traps again.
- Reward distribution is **permanently halted** for the affected SNS with no on-chain recovery path.
- All SNS neuron holders lose staking rewards indefinitely, breaking the DAO's core economic incentive mechanism.

This matches the allowed High impact: **Application/platform-level DoS** with concrete, permanent user and protocol harm to an in-scope SNS governance canister.

## Likelihood Explanation
Any unprivileged token holder can trigger this by staking tokens to create many neurons, using neuron following to cascade votes across all neurons on every proposal, and allowing proposals to accumulate in `ReadyToSettle` state. The instruction cost scales as O(neurons × settled_proposals). With 10,000 neurons and 100 settled proposals, the ballot scan alone (~5,000 instructions/ballot) plus neuron updates (~50,000 instructions/neuron) reaches approximately 5.5 × 10⁹ instructions, exceeding the 5 × 10⁹ application-subnet per-message limit. This threshold is reachable without malicious intent as an SNS DAO matures, and is deterministically triggerable by a deliberate attacker. The NNS team already identified and fixed this exact class of bug for NNS governance, confirming the realistic likelihood.

## Recommendation
Port the NNS batched reward distribution pattern to SNS governance:
1. After computing `neuron_id_to_reward_shares`, persist it to stable memory (analogous to `RewardsDistributionStateMachine` backed by `StableBTreeMap`).
2. Replace the single-message neuron-update loop with a timer-driven task that calls `is_message_over_threshold` after each neuron update and breaks when the limit is reached, resuming on the next tick.
3. Only advance `latest_reward_event` after the distribution map is fully drained.
4. Mirror the `DISTRIBUTION_MESSAGE_LIMIT` constant (1 billion instructions) used by NNS governance.

## Proof of Concept
**Deterministic reasoning:**
- IC application-subnet per-message instruction limit: 5 × 10⁹ instructions.
- Cost per ballot entry iteration: ~5,000 instructions.
- Cost per neuron maturity update (HashMap lookup + arithmetic): ~50,000 instructions.
- 10,000 neurons × 100 settled proposals × 5,000 (ballot scan) = 5 × 10⁹ instructions from ballot iteration alone.
- Adding 10,000 × 50,000 (neuron updates) = 5.5 × 10⁹ total → exceeds limit.

**Reproducible test plan:** Write a PocketIC integration test that: (1) creates an SNS with 10,000 neurons all following a root neuron, (2) submits and passes 100+ proposals so they enter `ReadyToSettle`, (3) advances the clock past the reward period, (4) triggers `run_periodic_tasks`, and (5) asserts that `latest_reward_event` is not updated (trap occurred) and that subsequent heartbeats also fail to advance it.

```rust
        &mut self,
        neuron_store: &mut NeuronStore,
        is_over_instructions_limit: fn() -> bool,
    ) {
        while let Some((id, reward_e8s)) = self.rewards.pop_first() {
            match neuron_store.with_neuron_mut(&id, |neuron| {
                let auto_stake = neuron.auto_stake_maturity.unwrap_or(false);

                        neuron

                            .unwrap_or_default()

                    rewards are calculated: {}",

                }
            };
```

```markdown
## Changed

* Voting Rewards will be scheduled by a timer instead of by heartbeats.
* Unstaking maturity task will be processing up to 100 neurons in a single message, to avoid
  exceeding the instruction limit in a single execution.
* Voting Rewards will be distributed asynchronously in the background after being calculated.
```
