### Title
Incorrect Accounting for NEP-141 Tokens with Transfer Fees in `ft_on_transfer` - (`engine/src/engine.rs`)

### Summary

Aurora Engine's `ft_on_transfer` handler mints ERC-20 tokens equal to `args.amount` — the amount reported by the calling NEP-141 contract — without verifying the actual amount credited to Aurora's NEP-141 balance. If the NEP-141 token charges a transfer fee (fee-on-transfer), Aurora mints more ERC-20 tokens than it has NEP-141 backing, creating insolvency and enabling fund theft.

### Finding Description

When a user bridges a NEP-141 token to Aurora via `ft_transfer_call`, the NEP-141 contract calls Aurora's `ft_on_transfer` callback with `args.amount` equal to the amount the sender specified. Aurora then mints exactly `args.amount` ERC-20 tokens for the recipient.

In `receive_erc20_tokens`: [3](#0-2) 

The `setup_receive_erc20_tokens_input` function encodes `amount` (from `args.amount`) directly into the ERC-20 `mint` call: [4](#0-3) 

If the NEP-141 token deducts a fee during transfer (e.g., a 1% fee-on-transfer), Aurora's actual NEP-141 balance increases by only `amount - fee`, but `amount` ERC-20 tokens are minted. The ERC-20 supply becomes unbacked by the corresponding NEP-141 reserve.

### Impact Explanation

**Critical — Insolvency and direct theft of funds.**

- The attacker deposits `N` NEP-141 tokens with a fee-on-transfer. Aurora receives `N - fee` NEP-141 tokens but mints `N` ERC-20 tokens.
- The attacker can immediately exit with `N` ERC-20 tokens, triggering an `ft_transfer` of `N` NEP-141 tokens from Aurora's reserve. If Aurora has enough reserve from other users, the attacker extracts `fee` tokens that belong to other depositors.
- Repeated exploitation drains the NEP-141 reserve. The last users to exit cannot withdraw their full balance — permanent partial fund freeze and insolvency.

### Likelihood Explanation

**Medium.** Fee-on-transfer NEP-141 tokens are a known token design pattern. Any such token can be deployed on NEAR and bridged to Aurora via the standard `deploy_erc20_token` + `ft_transfer_call` flow, which is permissionless. No admin access or special privileges are required. The attacker only needs to hold a fee-on-transfer NEP-141 token and call `ft_transfer_call` targeting Aurora.

### Recommendation

In `receive_base_tokens` and `receive_erc20_tokens`, do not trust `args.amount` as the mint quantity. Instead, measure the actual NEP-141 balance change by querying Aurora's NEP-141 balance before and after the transfer, and mint only the net received amount. Alternatively, return `args.amount` (reject the transfer) for any NEP-141 token whose `ft_transfer_call` results in a balance increase less than `args.amount`, forcing the NEP-141 contract to refund the full amount to the sender.

### Proof of Concept

1. Deploy a NEP-141 token on NEAR that charges a 1% fee on every transfer (deducted from the credited amount).
2. Call `deploy_erc20_token` on Aurora to register the NEP-141 → ERC-20 mapping.
3. Call `ft_transfer_call(receiver_id=aurora, amount=1000, msg=<attacker_evm_address>)` on the NEP-141 contract.
   - NEP-141 credits Aurora with 990 tokens (1% fee = 10 tokens burned/sent to fee collector).
   - NEP-141 calls `ft_on_transfer(sender=attacker, amount=1000, msg=<attacker_evm_address>)` on Aurora.
   - Aurora's `receive_erc20_tokens` mints **1000** ERC-20 tokens for the attacker.
4. Attacker calls `withdrawToNear` (burns 1000 ERC-20), triggering `ft_transfer(attacker_near, 1000)` on the NEP-141 contract.
   - Aurora only holds 990 NEP-141 tokens but attempts to send 1000.
   - If other users have deposited, the attacker successfully withdraws 1000, stealing 10 tokens from the pool.
5. Repeated over many users, Aurora's NEP-141 reserve is drained; the last depositors cannot withdraw. [5](#0-4) [6](#0-5)

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
