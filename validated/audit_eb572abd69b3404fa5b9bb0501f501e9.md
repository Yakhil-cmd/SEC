Audit Report

## Title
SNS Governance Lacks Voting Power Spike Detection, Enabling Stake-Inflation Ballot Manipulation - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS governance canister's `compute_ballots_for_new_proposal` function computes ballot voting power using only the live neuron state at proposal creation time, with no historical snapshot or spike-detection mechanism. An attacker who controls a neuron with sufficient dissolve delay can temporarily inflate their stake immediately before calling `make_proposal`, obtain a ballot with inflated voting power that is permanently fixed at creation time, then begin recovering their tokens. Because the proposer auto-votes `Yes` at creation time, if the inflated stake exceeds 50% of total voting power the proposal is adopted in the same transaction. The NNS governance canister has an explicit defense against this attack (`VotingPowerSnapshots` / `SnapshotVotingPowerTask`); SNS governance has no equivalent.

## Finding Description
**Root cause:** `compute_ballots_for_new_proposal` in `rs/sns/governance/src/governance.rs` (L5225–5294) iterates over `self.proto.neurons` and calls `v.voting_power(now_seconds, ...)` for every eligible neuron, recording the live `cached_neuron_stake_e8s` at the instant `make_proposal` is called. There is no snapshot store, no historical comparison, and no spike-detection guard anywhere in the SNS codebase (confirmed: zero matches for `VotingPowerSnapshot`, `snapshot_voting_power`, or `SnapshotVotingPower` under `rs/sns/`).

**Contrast with NNS:** `compute_ballots_for_standard_proposal` in `rs/nns/governance/src/governance.rs` (L5497–5524) first computes the current snapshot, then calls `VOTING_POWER_SNAPSHOTS.previous_ballots_if_voting_power_spike_detected(...)`. If the current total exceeds 1.5× the historical minimum across up to 7 daily snapshots, it falls back to the snapshot with the minimum total. The `SnapshotVotingPowerTask` (L9–57 of `rs/nns/governance/src/timer_tasks/snapshot_voting_power.rs`) runs daily to maintain this window. SNS has no timer task, no `VotingPowerSnapshots` structure, and no fallback path.

**Exploit flow:**
1. Attacker acquires SNS tokens and creates a neuron with dissolve delay ≥ `neuron_minimum_dissolve_delay_to_vote_seconds`.
2. Immediately before calling `make_proposal`, attacker calls `manage_neuron { ClaimOrRefresh }` to top up `cached_neuron_stake_e8s` to a very large value.
3. `make_proposal` calls `compute_ballots_for_new_proposal` (L3557–3559), which records the inflated voting power in the `ballots` map.
4. `cast_vote_and_cascade_follow` is called with `Vote::Yes` for the proposer (L3658–3669), immediately exercising the inflated ballot.
5. If inflated power > 50% of total, `tally` crosses the adoption threshold and the proposal executes in the same call.
6. Attacker begins dissolving the neuron to recover tokens after the dissolve delay.

**Why existing checks fail:** The only guards in `compute_ballots_for_new_proposal` are an overflow check (`total_power >= u64::MAX`) and an empty-roll check. Neither detects nor prevents a sudden stake inflation. The `neuron_fees_e8s` deduction applied after ballot creation (L3649–3653) reduces the proposer's *future* voting power but does not retroactively alter the already-fixed ballot.

## Impact Explanation
An attacker can pass arbitrary SNS governance proposals — including `UpgradeSnsControlledCanister`, `TransferSnsTreasuryFunds`, or `ExecuteGenericNervousSystemFunction` — with voting power that does not reflect the genuine long-term stake distribution. This enables unauthorized transfer of SNS treasury funds, malicious canister upgrades, or parameter changes. This matches the **High** impact category: unauthorized access to governance assets and canister-controlled funds, where exploitation requires meaningful capital commitment (acquiring tokens and locking them for the dissolve delay period).

## Likelihood Explanation
The attack is fully permissionless — no privileged role is required. SNS tokens are publicly tradeable. The only friction is the minimum dissolve delay, which varies per SNS but is typically days to months; the attacker must commit tokens for that period but recovers them afterward. The stake inflation step (`ClaimOrRefresh`) is a standard, unrestricted operation. The attack is repeatable across any SNS deployment that has not independently implemented spike detection. An attacker with sufficient capital can execute this against any SNS whose token supply is concentrated or whose total staked voting power is low relative to available liquidity.

## Recommendation
Implement a voting power snapshot mechanism for SNS governance analogous to the NNS system:

1. Add a `VotingPowerSnapshots`-equivalent structure to SNS governance stable memory.
2. Add a recurring timer task (analogous to `SnapshotVotingPowerTask`) that records a daily snapshot of total voting power across all eligible SNS neurons.
3. In `compute_ballots_for_new_proposal`, compare the current total voting power against the historical minimum in the snapshot window. If current > 1.5× historical minimum, use the snapshot with the minimum total to construct ballots.
4. As a lower-cost interim measure, reject proposals where the submitting neuron's voting power exceeds a configurable fraction (e.g., 33%) of the total voting power at proposal creation time.

## Proof of Concept
**Minimal deterministic test plan (PocketIC or SNS integration test):**

1. Initialize an SNS with two neurons: `neuron_A` (attacker, dissolve delay = `min_dissolve_delay_for_vote`, stake = 1 token) and `neuron_B` (honest, stake = 2 tokens). Total voting power ≈ 3 units; attacker holds ~33%.
2. Transfer 10 tokens to the SNS ledger account controlled by `neuron_A`'s owner.
3. Call `manage_neuron { ClaimOrRefresh }` on `neuron_A` to update `cached_neuron_stake_e8s` to 11 tokens. Total voting power is now ≈ 13 units; attacker holds ~85%.
4. Call `manage_neuron { MakeProposal }` from `neuron_A` with a `TransferSnsTreasuryFunds` action.
5. Assert: the proposal is immediately adopted (tally shows `yes > 50%` of total) without any vote from `neuron_B`.
6. Assert: `neuron_B` had no opportunity to prevent adoption.
7. Contrast: repeat steps 1–5 against NNS governance with the same stake ratio — the spike-detection guard prevents immediate adoption.
