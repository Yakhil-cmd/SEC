### Title
Unhandled Failed `ft_transfer` Promise in `exit_to_near_precompile_callback` Causes Permanent ERC-20 Token Loss When `error_refund` Feature Is Disabled - (File: `engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

When the compile-time feature `error_refund` is not enabled, a failed `ft_transfer` or `ft_transfer_call` promise triggered by the `ExitToNear` precompile results in no refund being issued. The user's ERC-20 tokens are permanently burned in the EVM with no corresponding NEP-141 transfer completing on NEAR, causing a permanent loss of funds.

---

### Finding Description

The `ExitToNear` precompile burns a user's ERC-20 tokens (or ETH) and schedules a `ft_transfer` / `ft_transfer_call` promise to the NEP-141 contract. Whether a callback (`exit_to_near_precompile_callback`) is attached to handle failure depends on the `error_refund` compile-time feature.

In `engine-precompiles/src/native.rs`, the `refund` field of `ExitToNearPrecompileCallbackArgs` is unconditionally set to `None` when `error_refund` is not enabled:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For the standard ERC-20 exit path (`ft_transfer`, `ft_transfer_call`), `transfer_near` is also `None`. This makes `callback_args` equal to `ExitToNearPrecompileCallbackArgs::default()`, so the promise is created **without any callback at all**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // No callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

For the wNEAR unwrap path (`near_withdraw`), a callback IS attached (because `transfer_near` is `Some`), but in `exit_to_near_precompile_callback`, the failure branch checks `args.refund`, which is `None`, and silently does nothing:

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success path
    None
} else if let Some(args) = args.refund {
    // refund path — never reached when error_refund is disabled
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None  // silent no-op on failure
};
``` [3](#0-2) 

The test suite explicitly acknowledges this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

---

### Impact Explanation

When `error_refund` is not enabled and the NEP-141 `ft_transfer` promise fails (e.g., the recipient NEAR account is not registered with the NEP-141 contract), the ERC-20 tokens are permanently burned in the EVM with no refund. The user suffers a **permanent, irrecoverable loss of funds**. This maps to **Critical — Permanent freezing/loss of funds**.

---

### Likelihood Explanation

The `error_refund` feature is an opt-in compile-time flag. Any production deployment that does not explicitly enable it is vulnerable. A failure of `ft_transfer` is easily triggered by any user who specifies a NEAR recipient account that is not registered with the NEP-141 token contract — a common real-world condition. The attacker path requires no special privileges: any EVM user holding ERC-20 tokens can trigger this by calling the ERC-20 burn/exit function with an unregistered recipient.

---

### Recommendation

- Make the refund logic unconditional (not gated behind `error_refund`), or document clearly that deployments without `error_refund` expose users to permanent token loss.
- In `exit_to_near_precompile_callback`, always attach a callback for the `ft_transfer` path and always check the promise result, issuing a re-mint (refund) of the burned ERC-20 tokens when the promise fails.
- The `refund_on_error` function already implements the correct re-mint logic; it simply needs to be reachable on the failure path unconditionally. [5](#0-4) 

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Bridge a NEP-141 token to Aurora, creating an ERC-20 mirror.
3. As an EVM user holding the ERC-20, call the ERC-20 burn/exit function targeting a NEAR account that is **not registered** with the NEP-141 contract.
4. The `ExitToNear` precompile burns the ERC-20 tokens and schedules `ft_transfer` with no callback.
5. The `ft_transfer` promise fails (unregistered recipient).
6. No callback fires; no refund is issued.
7. The user's ERC-20 balance is permanently zero; the NEP-141 balance of the recipient is also zero. Funds are destroyed.

This is confirmed by the existing test `test_exit_to_near_refund` which explicitly asserts the reduced balance `(FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT)` when `error_refund` is not compiled in. [6](#0-5)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
