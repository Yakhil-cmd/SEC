### Title
Unchecked Precompile Return Value in `EvmErc20.withdrawToNear` / `withdrawToEthereum` Burns Tokens Without Guaranteeing NEP-141 Transfer - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.withdrawToNear` and `EvmErc20.withdrawToEthereum` (and their `EvmErc20V2` counterparts) burn the caller's ERC-20 tokens unconditionally before invoking the Aurora exit precompile via an inline assembly `call`. The return value of that `call` is captured in a local variable `res` but is **never checked**. If the precompile call fails for any reason, the ERC-20 tokens are permanently destroyed while no NEP-141 transfer promise is ever scheduled, leaving the underlying NEAR-side tokens locked in the Aurora engine contract forever.

---

### Finding Description

In `etc/eth-contracts/contracts/EvmErc20.sol`, the `withdrawToNear` function:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked — no revert on failure
    }
}
```

The identical pattern appears in `withdrawToEthereum` (calling `0xb0bd02f6...`) and in both functions of `EvmErc20V2`.

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) can return an `ExitError` (causing the EVM `call` to return `0`) under several reachable conditions:

1. **Input size validation failure** — `validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)` rejects inputs outside `[3, 1024]` bytes. A `recipient` byte array that is empty, malformed, or exceeds ~991 bytes causes the precompile to return an error.
2. **Amount overflow** — `parse_amount` rejects any `U256` value exceeding `u128::MAX`. A user who holds ERC-20 tokens minted with an amount above `u128::MAX` (possible if the admin minted directly via the `mint(address, uint256)` function) triggers this path.
3. **Invalid recipient format** — `parse_recipient` validates the NEAR account ID format. Passing raw Ethereum address bytes or any non-UTF-8 / non-valid-NEAR-account-ID bytes causes a parse failure.

In all these cases, the EVM `call` returns `0`, the assembly block silently ignores it, `_burn` has already committed, and the function returns successfully. No NEAR-side promise is ever created. The NEP-141 tokens that were locked in the Aurora engine contract when the ERC-20 was originally minted remain locked permanently.

This is the direct structural analog of the reported `_addInvestor` bug: **state is mutated (tokens burned / allocation recorded) without verifying that the corresponding asset movement (NEP-141 transfer / `transferFrom`) was actually initiated**.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the precompile call fails silently:
- The user's ERC-20 balance is reduced to zero (burned).
- No `ft_transfer` or `ft_transfer_call` promise is scheduled on the NEP-141 contract.
- The NEP-141 tokens that back the burned ERC-20 remain locked inside the Aurora engine contract with no recovery path.
- There is no refund mechanism in `EvmErc20` (only `EvmErc20V2` with the `error_refund` feature has a refund path, but even there the refund depends on the callback receiving a failed promise result — which never fires if the precompile itself never scheduled a promise).

---

### Likelihood Explanation

**Medium.** The failure modes are reachable by any unprivileged EVM user:

- A user calling `withdrawToNear` with a recipient that is not a valid NEAR account ID (e.g., passing raw Ethereum address bytes, an empty byte array, or a string > 991 bytes) triggers condition 1 or 3.
- A user or contract that received ERC-20 tokens minted with `amount > type(uint128).max` via the admin `mint` function triggers condition 2.
- No special privileges, governance capture, or key compromise is required — the entry point is the standard ERC-20 `withdrawToNear` / `withdrawToEthereum` interface callable by any token holder.

---

### Recommendation

Check the return value of the precompile `call` and revert if it fails, so that the `_burn` is rolled back atomically:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    bool success;
    assembly {
        success := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    }
    require(success, "ExitToNear precompile call failed");
}
```

Alternatively, restructure the function to call the precompile first and only burn on success (requires a two-step design or a precompile that supports this ordering).

Apply the same fix to `withdrawToEthereum` in both `EvmErc20` and `EvmErc20V2`.

---

### Proof of Concept

**Step 1 — Obtain bridged ERC-20 tokens.**
Bridge any NEP-141 token to Aurora via `ft_transfer_call` → `ft_on_transfer` → `receive_erc20_tokens`. The user now holds ERC-20 tokens on Aurora.

**Step 2 — Call `withdrawToNear` with an invalid recipient.**
```solidity
// Pass an empty bytes array as recipient — parse_recipient will fail
evmErc20.withdrawToNear(bytes(""), amount);
```

**Step 3 — Observe the outcome.**
- `_burn(msg.sender, amount)` executes: the user's ERC-20 balance drops to zero.
- The assembly `call` to `0xe9217bc7...` returns `0` (precompile rejected the empty recipient).
- `res` is never checked; the function returns without reverting.
- No NEAR-side promise is created; the NEP-141 tokens remain locked in the Aurora engine contract.
- The user has lost their tokens permanently with no recovery path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

        is_static: bool,

        //  - flag (1 byte)
        //  - refund_address (20 bytes), present if the feature "error_refund" is enabled
        //  - recipient_account_id (max MAX_INPUT_SIZE - 20 - 1 bytes)
        // ERC-20 transfer input format: (124 bytes)
        //  - flag (1 byte)
        //  - refund_address (20 bytes), present if the feature "error_refund" is enabled.
        //  - amount (32 bytes)
        //  - recipient_account_id (max MAX_INPUT_SIZE - 1 - (20) - 32 bytes)
        //  - `:unwrap` suffix in a case of wNEAR (7 bytes)
        let required_gas = Self::required_gas(input)?;

        if let Some(target_gas) = target_gas
            && required_gas > target_gas
        {
            return Err(ExitError::OutOfGas);
        }

        // It's not allowed to call exit precompiles in static mode
        if is_static {
            return Err(ExitError::Other(Cow::from("ERR_INVALID_IN_STATIC")));
        } else if context.address != exit_to_near::ADDRESS.raw() {
            return Err(ExitError::Other(Cow::from("ERR_INVALID_IN_DELEGATE")));
        }

        let exit_to_near_params = ExitToNearParams::try_from(input)?;

        let (nep141_address, args, exit_event, method, transfer_near_args) =
            match exit_to_near_params {
                // ETH(base) token transfer
                //
                // Input slice format:
                //  recipient_account_id (bytes) - the NEAR recipient account which will receive
                //  NEP-141 (base) tokens, or also can contain the `:unwrap` suffix in case of
                //  withdrawing wNEAR, or another message of JSON in case of OMNI, or address of
                //  receiver in case of transfer tokens to another engine contract.
                ExitToNearParams::BaseToken(ref exit_params) => {
                    let eth_connector_account_id = self.get_eth_connector_contract_account()?;
                    exit_base_token_to_near(eth_connector_account_id, context, exit_params)?

                // ERC-20 token transfer
                //
                // This precompile branch is expected to be called from the ERC-20 burn function.
                //
                // Input slice format:
                //  amount (U256 big-endian bytes) - the amount that was burned
                //  recipient_account_id (bytes) - the NEAR recipient account which will receive
                //  NEP-141 tokens, or also can contain the `:unwrap` suffix in case of withdrawing
                //  wNEAR, or another message of JSON in case of OMNI, or address of receiver in case
                //  of transfer tokens to another engine contract.
                ExitToNearParams::Erc20TokenParams(ref exit_params) => {
                    exit_erc20_token_to_near(context, exit_params, &self.io)?

            refund: refund_call_args(&exit_to_near_params, &exit_event),

            refund: None,
            transfer_near: transfer_near_args,
        };

            costs::FT_TRANSFER_CALL_GAS
        } else {
            costs::FT_TRANSFER_GAS
        };

        let transfer_promise = PromiseCreateArgs {
            target_account_id: nep141_address,
            method,
            args: args.into_bytes(),
            attached_balance: Yocto::new(1),
            attached_gas,
        };

        let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
            PromiseArgs::Create(transfer_promise)

                callback: PromiseCreateArgs {

                    method: "exit_to_near_precompile_callback".to_string(),
                    args: borsh::to_vec(&callback_args).unwrap(),

                    attached_gas: costs::EXIT_TO_NEAR_CALLBACK_GAS,

            data,
        };

            logs: vec![promise_log, exit_event_log],
            cost: required_gas,
            output: Vec::new(),
        })
