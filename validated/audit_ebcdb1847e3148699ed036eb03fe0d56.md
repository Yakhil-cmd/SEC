Audit Report

## Title
Predictable ChaCha20Rng Seed Enables UUID Prediction and Undisclosed Rule Metadata Leakage — (`rs/boundary_node/rate_limits/canister/random.rs`)

## Summary

The `ChaCha20Rng` in `random.rs` is seeded with only 8 bytes of publicly observable IC block time, with the remaining 24 bytes fixed at the constant `42`. Because `get_rule_by_id` is a `#[query]` method that bypasses `inspect_message` entirely, any anonymous caller can query it. An attacker who observes the block timestamp of the first `add_config` call after install/upgrade can reconstruct the exact seed, predict all subsequent rule UUIDs, and call `get_rule_by_id` to confirm rule existence and retrieve `incident_id`, `added_in_version`, and `removed_in_version` for undisclosed rules.

## Finding Description

**Weak seed construction** (`random.rs` lines 8–14): The 32-byte seed is constructed as `time().to_le_bytes() ++ [42u8; 24]`. Only 8 bytes vary; the remaining 24 bytes are the constant `42`. `ic_cdk::api::time()` returns the IC consensus block time in nanoseconds, which is publicly observable via certified state reads. The `thread_local!` is initialized lazily on first access to `RNG`, which occurs during the first `add_config` call after install/upgrade, at which point `time()` captures that block's timestamp.

**Query method bypasses `inspect_message`** (`canister.rs` lines 34–68, 123–133): `get_rule_by_id` is annotated `#[query]`. The `inspect_message` hook on the IC applies exclusively to ingress update messages; non-replicated query calls bypass it entirely. The hook's catch-all rejection (`"method call is prohibited in the current context"`) therefore never fires for `get_rule_by_id`. Any anonymous or unprivileged caller can invoke it freely.

**Partial metadata leakage for undisclosed rules** (`getter.rs` lines 224–246, `confidentiality_formatting.rs` lines 31–42): For a `RestrictedRead` caller, `RuleConfidentialityFormatter::format` only redacts `rule_raw` and `description` when `disclosed_at.is_none()`. The fields `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` are always returned. A `NotFound` error is returned only when the UUID does not exist, giving the attacker a clean existence oracle.

**Full exploit path:**
1. Attacker monitors the IC state tree for the first `add_config` ingress message after canister install/upgrade and reads its block timestamp `T`.
2. Attacker constructs `seed[..8] = T.to_le_bytes(); seed[8..] = [42u8; 24]` and instantiates `ChaCha20Rng::from_seed(seed)` locally.
3. Attacker replays `rng.fill_bytes(&mut buf[..16])` for each subsequent `add_config` call to predict the corresponding `rule_id` UUID.
4. Attacker calls `get_rule_by_id(predicted_uuid)` as an anonymous query; a non-`NotFound` response confirms the rule exists and returns `incident_id`, `added_in_version`, `removed_in_version`.

## Impact Explanation

This breaks the stated confidentiality invariant of the rate-limits canister: undisclosed rules are intended to be inaccessible to unprivileged callers. The `incident_id` field directly links a confidential rate-limit rule to a security incident before public disclosure, enabling an attacker to enumerate ongoing security incidents at the boundary node layer. This constitutes a significant boundary/API infrastructure security impact with concrete harm — premature disclosure of security incident associations — mapping to the **High** bounty tier ($2,000–$10,000): "Significant boundary/API … infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation

The attack requires no privileges, no brute force, and no key material. IC block timestamps are publicly available with nanosecond precision via certified state reads. The only prerequisite is observing the block time of the first `add_config` call after an install/upgrade, which is trivially detectable by monitoring the canister's ingress history. The attack is fully deterministic and repeatable.

## Recommendation

Replace the `time()`-seeded `thread_local!` initialization entirely. Use `ic_cdk::api::management_canister::main::raw_rand()` (threshold-BLS randomness) in the `#[init]` and `#[post_upgrade]` hooks to obtain 32 unpredictable bytes, store the seed in stable memory or a canister-held variable, and initialize the RNG from that stored seed. The lazy `thread_local!` pattern with `time()` must be removed.

## Proof of Concept

```rust
// Deterministic replay — run locally against a PocketIC instance
let block_time_ns: u64 = /* read from IC state tree at first add_config block */;
let mut seed = [42u8; 32];
seed[..8].copy_from_slice(&block_time_ns.to_le_bytes());
let mut rng = ChaCha20Rng::from_seed(seed);

for _ in 0..N {
    let mut buf = [0u8; 16];
    rng.fill_bytes(&mut buf);
    let predicted_uuid = Uuid::from_slice(&buf).unwrap().to_string();
    // Anonymous query call:
    let response = canister.query("get_rule_by_id", predicted_uuid);
    // Assert: response is Ok (not NotFound) — rule exists
    // Assert: incident_id, added_in_version, removed_in_version are present in response
    // Assert: rule_raw and description are None (redacted but existence confirmed)
}
```

A PocketIC integration test can install the canister, call `add_config` N times, capture the mock block time, replay the RNG locally, and assert all N predicted UUIDs return `Ok` with non-empty `incident_id` from an anonymous query caller.
