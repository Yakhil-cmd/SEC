### Title
Truncating Division in `native_per_pubdata` Causes Pubdata to Be Completely Free When `pubdata_price < native_price` - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary

The `native_per_pubdata` ratio used to charge native resources for pubdata consumption is computed with floor (truncating) integer division. When `pubdata_price < native_price`, this ratio truncates to zero, making all pubdata completely free in native resources. Any unprivileged transaction sender can then generate arbitrarily large amounts of pubdata without consuming any native resource budget, bypassing the intended pubdata cost enforcement.

### Finding Description

**Root cause — floor division to zero:**

In `validate_and_compute_fee_for_transaction` (ZK L2 transaction path):

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is Rust's truncating integer division. When `pubdata_price < native_price` (e.g., `pubdata_price = 99`, `native_price = 100`), the result is `0`. The same truncating division appears in the off-chain helper:

```rust
// api/src/helpers.rs, line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

Note the asymmetry: `native_per_gas` uses `div_ceil` (rounds up, favoring the protocol), while `native_per_pubdata` uses `wrapping_div` (rounds down, favoring the user).

**Downstream effect — pubdata check always passes:**

`native_per_pubdata` flows into `create_resources_for_tx` (intrinsic pubdata overhead) and `get_resources_to_charge_for_pubdata` (execution pubdata):

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs, line 352
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// → 0 * intrinsic_pubdata = 0 native reserved for pubdata

// gas_helpers.rs, line 430-432
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // → 0 for any pubdata_spent
    .ok_or(out_of_native_resources!())?;
```

When `native_per_pubdata = 0`, `get_resources_to_charge_for_pubdata` returns zero native resources regardless of how many pubdata bytes the transaction writes. Consequently, `check_enough_resources_for_pubdata` always returns `true`:

```rust
// gas_helpers.rs, line 454
let enough = resources.has_enough(&resources_for_pubdata); // resources_for_pubdata = 0 → always true
```

The post-execution pubdata revert path in the ZK transaction flow is also bypassed:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs, line 880-888
let (has_enough, to_charge_for_pubdata, pubdata_used) = check_enough_resources_for_pubdata(...)?;
if !has_enough {
    execution_result = execution_result.to_reverted(); // never reached when native_per_pubdata = 0
```

**Attacker-controlled entry path:**

1. Operator sets `pubdata_price < native_price` (a realistic configuration when proving is expensive relative to data availability costs, or when market conditions shift).
2. Attacker submits an L2 transaction (no privilege required) that writes to many storage slots, generating large amounts of pubdata.
3. Since `native_per_pubdata = 0`, the native resource budget is not reduced by pubdata at all — the full native budget is available for computation, and the pubdata check always passes.
4. The transaction succeeds and forces the operator to publish large amounts of pubdata without the sender paying any native resource cost for it.

### Impact Explanation

- **Pubdata cost enforcement is completely bypassed** when `pubdata_price < native_price`. The native resource accounting — which is the primary per-transaction mechanism for limiting pubdata — is rendered inoperative.
- Attackers can generate pubdata up to the block-level `pubdata_limit` without consuming any native resource budget, effectively getting pubdata for free in native resource terms.
- The operator must publish this pubdata (for data availability) but receives no native resource compensation for it. Over many transactions, this creates a systematic undercharge for pubdata costs.
- The off-chain `validate_l2_tx_intrinsic_native_resources` helper in `api/src/helpers.rs` has the identical truncation, so pre-submission validation also fails to detect the undercharge.

### Likelihood Explanation

`pubdata_price` and `native_price` are independent operator-set parameters. `native_price` reflects the cost of a RISC-V proving cycle; `pubdata_price` reflects the cost of one byte of L1 data availability. These can easily diverge: if proving becomes expensive (high `native_price`) while L1 calldata/blob fees are low (low `pubdata_price`), the condition `pubdata_price < native_price` holds naturally. No attacker action is needed to create the vulnerable configuration — the attacker only needs to observe the block context and submit storage-writing transactions.

### Recommendation

Replace the truncating division with ceiling division for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// validation_impl.rs line 142 — change from:
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...;
// to:
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))...;
```

Apply the same fix in `api/src/helpers.rs` line 427. Additionally, consider enforcing a minimum `native_per_pubdata` of 1 when `pubdata_price > 0`, to ensure pubdata always consumes at least one native resource unit per byte.

### Proof of Concept

Scenario: `native_price = 100`, `pubdata_price = 99`.

```
native_per_pubdata = 99 / 100 = 0  (truncating division)
```

A transaction writing 10,000 storage slots (generating ~320,000 bytes of pubdata):

```
native cost for pubdata = 320,000 * 0 = 0
```

`check_enough_resources_for_pubdata` returns `true` with `resources_for_pubdata = 0`. The transaction succeeds. The operator publishes 320,000 bytes of pubdata and receives zero native resource compensation for it.

With correct ceiling division: `native_per_pubdata = ceil(99/100) = 1`, and the same transaction would consume `320,000 * 1 = 320,000` native units, properly reducing the transaction's native budget. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)
