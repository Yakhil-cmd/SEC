Audit Report

## Title
SNS Governance Allows Concurrent Identical `ExecuteGenericNervousSystemFunction` Proposals Without Deduplication Guard - (File: `rs/sns/governance/src/governance.rs`)

## Summary
SNS governance enforces a concurrency guard (`check_no_upgrades_in_progress`) for upgrade-class proposals but applies no equivalent guard for `ExecuteGenericNervousSystemFunction` proposals. Two proposals with identical `function_id` and `payload` can be submitted, voted through via the liquid-democracy following mechanism without explicit re-review, and both executed against the same external target canister. If the registered function is non-idempotent (e.g., a treasury transfer), the double-execution causes concrete fund loss.

## Finding Description
`perform_action` in `rs/sns/governance/src/governance.rs` (L2139–2175) dispatches `Action::ExecuteGenericNervousSystemFunction` directly to `perform_execute_generic_nervous_system_function` with no concurrency guard. By contrast, `Action::UpgradeSnsControlledCanister` and `Action::UpgradeSnsToNextVersion` both route through `check_no_upgrades_in_progress` (L2755–2788), which blocks a second concurrent upgrade from proceeding.

The `make_proposal` path (L3457–3674) performs authorization, stake, and dissolve-delay checks but contains no deduplication check for `ExecuteGenericNervousSystemFunction`. The validation function `validate_and_render_execute_nervous_system_function` in `rs/sns/governance/src/proposal.rs` (L1431–1481) only verifies that the referenced `NervousSystemFunction` exists and calls the validator canister; it does not inspect open or adopted proposals for a matching `function_id`/`payload`.

NNS governance demonstrates the correct pattern: `select_nonfinal_proposal_ids` is used in `validate_subnet_rental_proposal` (L4977–4991) and `validate_create_service_nervous_system` (L5063–5082) to block concurrent identical proposals at submission time. No analogous mechanism exists in SNS governance for generic function proposals.

The exploit path:
1. Attacker registers or targets an existing `NervousSystemFunction` (e.g., id=1000) whose `target_method` transfers SNS treasury tokens.
2. Attacker submits Proposal A: `ExecuteGenericNervousSystemFunction { function_id: 1000, payload: <transfer 500 tokens> }`.
3. Proposal A reaches majority; `start_proposal_execution` (L2118–2134) spawns `perform_action` asynchronously.
4. Before Proposal A's async execution completes, attacker submits Proposal B with identical `function_id` and `payload`.
5. At proposal creation (L3658–3669), `cast_vote_and_cascade_follow` is called for the proposer's Yes vote, cascading to all neurons that follow the attacker's neuron on that `function_id` or via catch-all following — without those neurons explicitly reviewing the duplicate.
6. Proposal B reaches majority through the same following graph that passed Proposal A.
7. `perform_action` is called a second time for the same `function_id`/`payload`; the target canister's transfer method executes twice, draining 1000 tokens instead of 500.

## Impact Explanation
The concrete impact is unauthorized double-execution of a governance-callable external function, which can result in double-transfer of SNS treasury funds or double-minting of tokens. This maps to the High bounty impact: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints." The SNS governance framework provides no mechanism to prevent this, and the `NervousSystemFunction` registration interface does not document the idempotency requirement.

## Likelihood Explanation
The attacker requires: (1) enough SNS stake to submit proposals (the `reject_cost_e8s` threshold, which is a low barrier in many SNS DAOs); (2) sufficient following power so that neurons following the attacker's neuron constitute a voting majority — a realistic condition in SNS DAOs where a small number of large neurons dominate following relationships; (3) correct timing to submit Proposal B before Proposal A's async execution completes, which is feasible given typical voting periods. Critically, the neurons that auto-vote Yes on Proposal B do so through the following mechanism without explicitly reviewing the duplicate, making the attack viable without a malicious majority.

## Recommendation
1. Add a deduplication check in `validate_and_render_execute_nervous_system_function` (or in `make_proposal` before proposal insertion) that calls an equivalent of `select_nonfinal_proposal_ids` filtering for open/adopted proposals with the same `function_id` (and optionally the same `payload` hash), analogous to the NNS pattern at `rs/nns/governance/src/governance.rs` L4977–4991.
2. Alternatively, extend `upgrade_proposals_in_progress` to a general `proposals_in_progress_by_action` map so any action type, including `ExecuteGenericNervousSystemFunction`, can opt into single-inflight semantics.
3. Document in the `NervousSystemFunction` registration interface that governance-callable functions may be invoked multiple times concurrently via duplicate proposals and must be designed to be idempotent or protected by their own deduplication logic.

## Proof of Concept
1. Deploy a local SNS (PocketIC or local replica) with a registered `NervousSystemFunction` (id=1000) whose `target_method` transfers 500 SNS tokens to a specified account and is intentionally non-idempotent.
2. Submit Proposal A: `ExecuteGenericNervousSystemFunction { function_id: 1000, payload: <transfer 500 tokens to attacker> }`.
3. Vote Yes with enough neurons to reach majority; observe Proposal A enters `Adopted` state and `start_proposal_execution` is called.
4. Before the async execution of Proposal A completes (use a mock environment that delays the inter-canister call), submit Proposal B with identical `function_id` and `payload`.
5. Observe that `cast_vote_and_cascade_follow` auto-votes Yes on Proposal B for all neurons following the proposer, passing Proposal B without explicit re-review.
6. Allow both async executions to complete; verify the target canister's transfer method was called twice and the attacker received 1000 tokens instead of 500.
7. A deterministic integration test can mock the environment to pause execution between `start_proposal_execution` calls and assert that both proposals reach `Executed` status with two calls to the target method.
