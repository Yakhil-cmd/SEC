### Title
Unchecked `transferFrom` Boolean Return Value in XCC Precompile Allows Free Cross-Contract Calls Without NEAR Payment - (File: `engine-precompiles/src/xcc.rs`)

### Summary
The `CrossContractCall` precompile calls `transferFrom` on the wNEAR ERC-20 contract to collect required NEAR payment from the caller before scheduling a cross-contract call. The call's EVM exit reason is checked, but the ABI-encoded boolean return value of `transferFrom` is never decoded or verified. If the wNEAR contract returns `false` (a valid ERC-20 failure signal) instead of reverting, the precompile treats the call as successful and proceeds to schedule the XCC promise — without the NEAR payment having been transferred.

### Finding Description
In `engine-precompiles/src/xcc.rs`, `CrossContractCall::run_with_handle` performs a sub-call to the wNEAR ERC-20 contract's `transferFrom` function to move wNEAR from the caller to the engine's implicit address:

```rust
// Lines 199-216
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
match exit_reason {
    // Transfer successful, nothing to do
    aurora_evm::ExitReason::Succeed(_) => (),
    aurora_evm::ExitReason::Revert(r) => { ... }
    aurora_evm::ExitReason::Error(e) => { ... }
    aurora_evm::ExitReason::Fatal(f) => { ... }
}
```

The `return_value` variable holds the ABI-encoded `bool` returned by `transferFrom`. The ERC-20 standard (EIP-20) explicitly permits returning `false` to signal failure without reverting. When this happens, `exit_reason` is still `ExitReason::Succeed(_)`, so the match arm at line 203 ("Transfer successful, nothing to do") is taken. The code then falls through to emit the XCC promise log at lines 226–231, scheduling the cross-contract call as if payment succeeded.

The `return_value` is captured but never inspected anywhere in the function. [1](#0-0) 

The promise log is unconditionally emitted after the match block, regardless of whether the `transferFrom` actually moved tokens: [2](#0-1) 

The `transfer_from_args` helper correctly encodes the `transferFrom(address,address,uint256)` selector and arguments, and the test in the same file even documents that `transferFrom` returns a `bool` — yet the return value is never checked at the call site: [3](#0-2) 

### Impact Explanation
**Connector/bridge accounting bug → Insolvency.**

If the wNEAR ERC-20 contract returns `false` from `transferFrom` (e.g., insufficient allowance, insufficient balance, or any non-reverting failure path), the XCC precompile schedules the NEAR cross-contract call without the required wNEAR having been debited from the caller. The engine's implicit address never receives the wNEAR, yet the XCC router executes the call and consumes real NEAR gas. Over repeated invocations this drains the engine's NEAR balance without corresponding wNEAR inflows, causing insolvency of the NEAR payment pool backing XCC operations.

### Likelihood Explanation
The wNEAR address is set by the admin via `factory_set_wnear_address`. The currently deployed wNEAR on Aurora uses OpenZeppelin's ERC-20, which reverts on failure rather than returning `false`, so the bug is not exploitable against the current deployment. However:

1. The ERC-20 standard explicitly permits returning `false` — this is not anomalous behavior.
2. Any future upgrade or replacement of the wNEAR contract that follows the ERC-20 spec's non-reverting failure path would immediately expose this bug to any unprivileged EVM caller.
3. The code comment "Transfer successful, nothing to do" at line 202–203 demonstrates the developer's intent was to treat `Succeed` as a confirmed transfer, which is incorrect without decoding the return value.

Likelihood is **medium** given the current wNEAR implementation, but the code-level defect is unconditional.

### Recommendation
After the `ExitReason::Succeed(_)` arm, decode `return_value` and revert if the boolean is `false`:

```rust
aurora_evm::ExitReason::Succeed(_) => {
    // Decode the bool return value of transferFrom
    let transfer_ok = return_value
        .get(31)  // last byte of the 32-byte ABI bool
        .map_or(false, |&b| b == 1);
    if !transfer_ok {
        return Err(PrecompileFailure::Revert {
            exit_status: aurora_evm::ExitRevert::Reverted,
            output: b"ERR_WNEAR_TRANSFER_FAILED".to_vec(),
        });
    }
}
```

Alternatively, use `SafeERC20`-style logic or require the wNEAR contract to always revert on failure and document this as an invariant enforced at deployment time.

### Proof of Concept
1. Deploy a wNEAR-compatible ERC-20 contract whose `transferFrom` returns `false` instead of reverting when the caller has insufficient allowance.
2. Set this contract as the wNEAR address via `factory_set_wnear_address` (admin step, or assume a future upgrade).
3. As an unprivileged EVM user, call the XCC precompile (`cross_contract_call::ADDRESS`) with a `CrossContractCallArgs::Eager` payload that specifies a non-zero `attached_near`.
4. The precompile calls `transferFrom` on the wNEAR contract; it returns `false` with `ExitReason::Succeed`.
5. The match arm at line 203 is taken; no error is returned.
6. The XCC promise log is emitted at lines 226–231; the NEAR cross-contract call is scheduled and executes.
7. The engine's implicit address received zero wNEAR; the caller paid nothing; the XCC executed for free. [4](#0-3) [5](#0-4)
