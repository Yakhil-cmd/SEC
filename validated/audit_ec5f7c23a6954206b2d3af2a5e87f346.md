### Title
Staged Upgrade Persists After Emergency Owner Replacement, Allowing Malicious Code Deployment - (File: `engine/src/lib.rs`)

### Summary
The Aurora Engine's `deploy_upgrade` entrypoint has no owner check and can be called by anyone once the timelock expires. When a malicious owner stages a malicious upgrade and is then replaced via `set_owner`, the staged code and its timelock remain in storage untouched. After the delay passes, any caller — including the displaced attacker — can invoke `deploy_upgrade` to overwrite the engine contract with the malicious binary, achieving full control of the Aurora Engine account and all funds it holds.

### Finding Description

The engine exposes a two-step upgrade path:

**Step 1 — `stage_upgrade`** (owner-only): writes arbitrary WASM code to `CODE_KEY` and records `current_block + upgrade_delay_blocks` at `CODE_STAGE_KEY`. [1](#0-0) 

**Step 2 — `deploy_upgrade`** (no access control): reads the staged block index and, if the current block height exceeds it, calls `Runtime::self_deploy` with the stored code. The only guard is `require_running` (engine not paused) and the block-height check. [2](#0-1) 

The emergency owner-replacement path, `set_owner`, only updates `state.owner_id` in the `EngineState` struct. It does not touch `CODE_KEY` or `CODE_STAGE_KEY`. [3](#0-2) 

Because `EngineState` holds only the owner field and the staged upgrade lives in separate storage slots under `KeyPrefix::Config`, replacing the owner leaves the malicious staged code fully intact. [4](#0-3) 

### Impact Explanation

The Aurora Engine account holds all bridged ETH and ERC-20 tokens. Deploying an arbitrary WASM binary as the engine contract gives the attacker unrestricted control over every function the contract exposes, enabling direct theft of all user funds at rest, permanent freezing of funds, or insolvency. This is a **Critical** impact.

### Likelihood Explanation

The attack requires only that the malicious owner call `stage_upgrade` before (or in the same block as) the admin's `set_owner` call — a straightforward frontrun on NEAR. No specialized knowledge is needed. After the timelock, `deploy_upgrade` is permissionless, so the attacker does not even need to be the one to trigger it; they can wait or have any account do so.

### Recommendation

`set_owner` should clear the staged upgrade by removing both `CODE_KEY` and `CODE_STAGE_KEY` from storage whenever the owner changes. Additionally, consider adding an owner check to `deploy_upgrade` so that only the current owner can finalize a staged upgrade, consistent with the access control on `stage_upgrade` itself.

### Proof of Concept

1. Malicious owner calls `stage_upgrade` with a backdoored WASM binary. Storage now holds:
   - `KeyPrefix::Config ++ "CODE"` → malicious WASM
   - `KeyPrefix::Config ++ "CODE_STAGE"` → `current_block + upgrade_delay_blocks` [5](#0-4) 

2. Admin calls `set_owner` with a trusted replacement account. `state.owner_id` is updated; `CODE_KEY` and `CODE_STAGE_KEY` are untouched. [6](#0-5) 

3. After `upgrade_delay_blocks` blocks pass, any account calls `deploy_upgrade`. The check `io.block_height() <= index` passes, and `Runtime::self_deploy` overwrites the live engine contract with the malicious binary. [7](#0-6) 

4. The malicious contract now controls the Aurora Engine account and can drain all bridged assets.

pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;

        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;

        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }

        state.owner_id = args.new_owner;
        state::set_state(&mut io, &state)?;

        Ok(())
    })
}

#[named]
pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let delay_block_height = env.block_height() + state.upgrade_delay_blocks;
        require_owner_only(&state, &env.predecessor_account_id())?;
        io.read_input_and_store(&storage::bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.write_storage(
            &storage::bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY),
            &delay_block_height.to_le_bytes(),
        );
        Ok(())
    })
}

    pub extern "C" fn deploy_upgrade() {
        // This function is intentionally not implemented in `contract_methods`
        // because it only makes sense in the context of the NEAR runtime.
        let mut io = Runtime;
        let state = state::get_state(&io).sdk_unwrap();
        require_running(&state)
            .map_err(ContractError::msg)
            .sdk_unwrap();
        let index = internal_get_upgrade_index();
        if io.block_height() <= index {
            sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
        }
        Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
    }
