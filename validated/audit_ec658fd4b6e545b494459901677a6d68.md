### Title
Missing Refund Callback for Omni `ft_transfer_call` Exits When `error_refund` Feature Is Disabled — (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is disabled (the default in production builds), the `ExitToNear::run` precompile unconditionally sets `refund = None`. For all Omni exit paths, `transfer_near_args` is also `None`. This causes `callback_args` to equal `ExitToNearPrecompileCallbackArgs::default()`, which triggers the no-callback branch. The `ft_transfer_call` promise is scheduled without any `exit_to_near_precompile_callback`, so if the receiver rejects the tokens, the user's already-burned ERC-20 tokens are never re-minted and the returned NEP-141 tokens accumulate permanently in Aurora's balance.

---

### Finding Description

**Step 1 — Feature flag is off by default.**

Neither `engine-precompiles/Cargo.toml` nor `engine/Cargo.toml` lists `error_refund` in `default`. The `contract` feature used for WASM production builds does not pull it in either. [1](#0-0) [2](#0-1) 

**Step 2 — `refund` is hardcoded to `None` without the feature.**

In `ExitToNear::run`, the `callback_args` struct is built with a compile-time conditional:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None in production
    transfer_near: transfer_near_args,
};
``` [3](#0-2) 

**Step 3 — Omni paths always return `transfer_near_args = None`.**

Both Omni branches (`Message::Omni`) in `exit_base_token_to_near` and `exit_erc20_token_to_near` explicitly return `None` for `transfer_near_args`: [4](#0-3) [5](#0-4) 

**Step 4 — `callback_args == default()` suppresses the callback.**

`ExitToNearPrecompileCallbackArgs` derives `Default`, so `default()` = `{ refund: None, transfer_near: None }`. With `error_refund` disabled and an Omni path, `callback_args` always equals `default()`, causing the engine to emit a bare `PromiseArgs::Create` with no callback:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback ever scheduled
} else {
    PromiseArgs::Callback(...)
};
``` [6](#0-5) [7](#0-6) 

**Step 5 — Without the callback, rejected tokens are never re-minted.**

`exit_to_near_precompile_callback` is the only place that calls `engine::refund_on_error`, which re-mints burned ERC-20 tokens or returns ETH. If no callback is scheduled, this function is never invoked. The NEP-141 `ft_resolve_transfer` mechanism returns the rejected tokens to Aurora's NEP-141 balance, but nothing re-mints the corresponding ERC-20 tokens on the EVM side. [8](#0-7) [9](#0-8) 

---

### Impact Explanation

A user who calls the Omni exit precompile (ERC-20 or base-token with a `:msg` suffix) has their EVM tokens burned at the point the precompile executes. If the NEAR-side receiver rejects the `ft_transfer_call` (e.g., insufficient storage, invalid message, contract logic), the NEP-141 tokens are returned to Aurora's account via `ft_resolve_transfer`, but the user's EVM balance is never restored. The tokens are permanently lost to the user and silently accumulate in Aurora's NEP-141 balance — matching the "theft of unclaimed yield" scope.

---

### Likelihood Explanation

The `error_refund` feature is not in the default feature set and is not included in the `contract` feature used for WASM production builds. Any Omni `ft_transfer_call` exit where the receiver rejects the transfer triggers the loss. Receivers can legitimately reject for many reasons (storage not registered, invalid msg format, contract-side validation). The path is reachable by any EVM user calling the exit precompile with an Omni-format recipient string.

The existing test suite explicitly encodes this behavior as the expected outcome when `error_refund` is disabled: [10](#0-9) [11](#0-10) 

---

### Recommendation

Either:
1. Enable `error_refund` in the `contract` feature (and thus in production WASM builds), or
2. Decouple the callback-scheduling decision from the `error_refund` flag so that `ft_transfer_call` Omni exits always schedule a callback capable of handling rejection, even if the refund address is not provided.

The callback itself already handles the `refund = None` case gracefully (line 240–242 of `connector.rs`), so scheduling it unconditionally for `ft_transfer_call` paths would not break existing behavior.

---

### Proof of Concept

1. Build the engine WASM **without** `error_refund` (the default).
2. Deploy a NEAR receiver contract that rejects all `ft_on_transfer` calls (returns the full amount).
3. From an EVM account, call the exit precompile with an Omni-format input: `0x01 || amount(32) || "receiver.near:{msg}"`.
4. Observe that only a `PromiseArgs::Create` (no callback) is emitted in the promise log.
5. After the NEAR receipt settles, assert:
   - The user's ERC-20 balance on Aurora is `original - exit_amount` (tokens burned, not refunded).
   - Aurora's NEP-141 balance for that token is `original + exit_amount` (tokens returned to Aurora, not re-minted).
   - `exit_to_near_precompile_callback` was never called.

This matches the behavior already documented in `test_exit_to_near_refund` and `test_exit_to_near_eth_refund` under `#[cfg(not(feature = "error_refund"))]`.
