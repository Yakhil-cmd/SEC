Audit Report

## Title
SNS Governance Voting Rewards Permanently Lost When All Voters Have Zero Voting Power During a Reward Round - (File: `rs/sns/governance/src/governance.rs`)

## Summary
In `distribute_rewards`, when all ballots on `ReadyToSettle` proposals carry `voting_power: 0`, the entire `rewards_purse_e8s` is silently discarded. Proposals are settled and ballots cleared, but `distributed_e8s_equivalent` remains 0. Because `rewards_rolled_over()` gates on `settled_proposals.is_empty()`, the non-empty settled list prevents rollover, and the purse is permanently unrecoverable.

## Finding Description
`distribute_rewards` (SNS: `rs/sns/governance/src/governance.rs`) collects `considered_proposals` from `ready_to_be_settled_proposal_ids()` and computes `rewards_purse_e8s` from token supply and reward rate. [1](#0-0) [2](#0-1) 

It then sums `total_reward_shares` from each ballot's `voting_power`. When `total_reward_shares == dec!(0)`, the maturity distribution loop is skipped entirely and `distributed_e8s_equivalent` stays at `0`. [3](#0-2) 

Critically, the code unconditionally proceeds to settle all `considered_proposals` — setting `reward_event_end_timestamp_seconds`, updating `reward_event_round`, and clearing ballots — regardless of whether any rewards were distributed. [4](#0-3) 

The final `RewardEvent` is written with `settled_proposals: considered_proposals` (non-empty) and `distributed_e8s_equivalent: 0`. [5](#0-4) 

In the next round, `e8s_equivalent_to_be_rolled_over()` calls `rewards_rolled_over()`, which returns `true` only when `settled_proposals.is_empty()`. [6](#0-5) 

Since `settled_proposals` is non-empty, `rewards_rolled_over()` returns `false`, `e8s_equivalent_to_be_rolled_over()` returns `0`, and the entire `rewards_purse_e8s` is gone. The same structural flaw exists in NNS governance. [7](#0-6) 

The existing test `zero_total_reward_shares` confirms this code path executes without error but uses `EmptyLedger` (supply = 0), so `rewards_purse_e8s` is also 0 in the test — the test does not catch the loss when supply is non-zero. [8](#0-7) [9](#0-8) 

## Impact Explanation
The entire `rewards_purse_e8s` for the affected round — computed as `supply × reward_rate × round_duration` plus any previously rolled-over amount — is permanently unrecoverable. It is never minted as maturity for any neuron and is not carried forward. For an SNS with a meaningful token supply and active reward rate, this represents a concrete, irreversible loss of SNS governance token maturity per affected round. This matches the allowed High impact: **Significant SNS governance security impact with concrete user or protocol harm** ($2,000–$10,000).

## Likelihood Explanation
The trigger condition — all neurons that voted on all `ReadyToSettle` proposals having `voting_power: 0` at proposal creation time — is constrained but reachable by an unprivileged participant in a small SNS. A neuron's voting power is zero when `cached_neuron_stake_e8s <= neuron_fees_e8s`; fees accumulate from rejected proposals. An attacker can deliberately exhaust their own neuron's net stake via rejected proposals, then ensure that depleted neuron is the sole voter on a proposal that reaches `ReadyToSettle`. No privileged access, governance majority, or external compromise is required. The attack is repeatable each reward round under the same conditions.

## Recommendation
When `total_reward_shares == 0` but `considered_proposals` is non-empty, the rewards purse must be rolled over rather than discarded. The minimal targeted fix is to modify `rewards_rolled_over()` in `rs/sns/governance/src/types.rs` (and the NNS equivalent in `rs/nns/governance/src/reward/calculation.rs`) to also return `true` when `distributed_e8s_equivalent == 0` regardless of `settled_proposals`. Alternatively, add a guard in `distribute_rewards`: if `total_reward_shares == dec!(0)` and `considered_proposals` is non-empty, skip settling proposals and return early so the purse is naturally rolled over in the next round.

## Proof of Concept
Extend the existing `zero_total_reward_shares` test with a non-zero ledger supply (e.g., `total_supply = 1_000_000_000_000`). After `run_periodic_tasks()`:
1. Assert `reward_event.distributed_e8s_equivalent == 0` (already confirmed).
2. Assert `reward_event.settled_proposals` is non-empty (already confirmed).
3. Assert `reward_event.e8s_equivalent_to_be_rolled_over() > 0` — **this assertion will fail**, proving the purse is lost.
4. Run a second `run_periodic_tasks()` and assert the new `rewards_purse_e8s` includes the prior round's purse — **this will also fail**, confirming permanent loss.
