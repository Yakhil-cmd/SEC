### Title
Forward/Proving Divergence via Malleable Signature Bypass — (`forward_system/src/run/mod.rs`, `basic_bootloader/src/bootloader/config.rs`)

---

### Summary

The production forward block-building function uses `BasicBootloaderForwardSimulationConfig` (`VALIDATE_EOA_SIGNATURE=false`), which skips the malleable-signature check. The proving run uses `BasicBootloaderProvingExecutionConfig` (`VALIDATE_EOA_SIGNATURE=true`), which enforces it. A legacy transaction with `s > secp256k1n/2` is accepted by the forward run and rejected by the proving run, producing an unprovable block.

---

### Finding Description

**Config divergence** is explicit in the codebase:

`BasicBootloaderForwardSimulationConfig` sets `VALIDATE_EOA_SIGNATURE=false`: [1](#0-0) 

`BasicBootloaderProvingExecutionConfig` sets `VALIDATE_EOA_SIGNATURE=true`: [2](#0-1) 

The malleable-signature guard in both `zk/validation_impl.rs` and `ethereum/validation_impl.rs` is gated on this flag:

```rust
if !Config::VALIDATE_EOA_SIGNATURE | Config::SIMULATION {
    // skip — forward simulation path
} else {
    if U256::from_be_slice(s) > U256::from_be_bytes(SECP256K1N_HALF) {
        return Err(InvalidTransaction::MalleableSignature.into());
    }
    // ecrecover ...
}
``` [3](#0-2) [4](#0-3) 

The **production** forward block-building entry point hardcodes `BasicBootloaderForwardSimulationConfig`:

```rust
run_forward::<BasicBootloaderForwardSimulationConfig>(oracle, &mut result_keeper, ...);
``` [5](#0-4) 

The **production** proving entry point hardcodes `BasicBootloaderProvingExecutionConfig`:

```rust
ProvingBootloader::<O, L>::run_prepared::<BasicBootloaderProvingExecutionConfig>(...)
    .expect("Tried to prove a failing batch");
``` [6](#0-5) 

The `LegacySignatureData::decode_list_body` parser accepts any `s` value up to 32 bytes with no upper-bound check — it only rejects leading zeroes and oversized fields: [7](#0-6) 

So a well-formed legacy transaction with `s` in the upper half of the secp256k1 order passes RLP parsing in both runs, but diverges at the signature-validation gate.

**Note**: The test rig's `run_inner` deliberately uses `BasicBootloaderProvingExecutionConfig` for its forward run too (with the comment *"we use proving config here for benchmarking, although sequencer can have extra optimizations"*), so this divergence is invisible in the test suite: [8](#0-7) 

---

### Impact Explanation

- Forward run accepts the tx, applies state changes, and commits a block header.
- Proving run rejects the same tx with `MalleableSignature`, producing different state transitions and a different state root.
- The prover panics (`expect("Tried to prove a failing batch")`), or the committed state root cannot be reproduced, making the block permanently unprovable.
- Chain liveness is broken for that batch; operator intervention is required to recover.

---

### Likelihood Explanation

- Any unprivileged user can craft a valid ECDSA signature with `s > secp256k1n/2` (simply negate `s` and flip `v` on any existing signature). No key compromise or privileged access is needed.
- The attacker only needs to submit one such legacy transaction to a sequencer running `run_block`.
- The divergence is deterministic and 100% reproducible.

---

### Recommendation

Replace `BasicBootloaderForwardSimulationConfig` with `BasicBootloaderProvingExecutionConfig` (or a new config with `VALIDATE_EOA_SIGNATURE=true`) in the production `run_block` entry point, so the forward and proving runs apply identical validation rules. The comment in `config.rs` acknowledges this is an optimization shortcut; the shortcut must not skip checks that the prover enforces. [9](#0-8) 

---

### Proof of Concept

1. Take any valid secp256k1 signature `(v, r, s)` with `s ≤ secp256k1n/2`.
2. Compute `s' = secp256k1n - s` (upper-half `s`), flip `v` accordingly.
3. Encode a legacy EIP-155 transaction with this `(v', r, s')`.
4. Submit to a sequencer running `run_block` (uses `BasicBootloaderForwardSimulationConfig`): transaction is accepted, block is built.
5. Feed the same block to the prover (uses `BasicBootloaderProvingExecutionConfig`): `MalleableSignature` error fires at `validation_impl.rs:260`, state root diverges, block is unprovable.

impl<'a> RlpListDecode<'a> for LegacySignatureData<'a> {
    fn decode_list_body(r: &mut Rlp<'a>) -> Result<Self, InvalidTransaction> {
        let v = r.u256()?;
        let r_bytes = r.bytes()?;
        let s = r.bytes()?;
        // Check that r and s are at most 32 bytes each, and are not
        // non-canonically encoded (no leading zeroes).
        if r_bytes.len() > 32
            || s.len() > 32
            || (!r_bytes.is_empty() && r_bytes[0] == 0)
            || (!s.is_empty() && s[0] == 0)
        {
            return Err(InvalidTransaction::InvalidStructure);
        }
        let new = Self { v, r: r_bytes, s };
        Ok(new)
    }
}
