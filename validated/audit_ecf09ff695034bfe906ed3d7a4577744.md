### Title
Unordered Execution of Scheduled XCC Promises Allows Any Actor to Permanently Freeze User Funds - (File: etc/xcc-router/src/lib.rs)

---

### Summary

The `execute_scheduled` function in the XCC router contract is intentionally callable by **any** NEAR account with no access control. When a user schedules multiple dependent XCC promises (e.g., approve → swap → withdraw), any external actor can execute them out of order. Because the promise is **removed from storage before execution**, a failed out-of-order execution permanently destroys the promise, leaving user funds stranded in intermediate contracts with no XCC recovery path.

---

### Finding Description

The XCC router's `execute_scheduled` function at `etc/xcc-router/src/lib.rs` lines 149–156 is explicitly open to any caller:

```rust
/// It is intentional that this function can be called by anyone (not just the parent).
/// There is no security risk to allowing this function to be open because it can only
/// act on promises that were created via `schedule`.
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
``` [1](#0-0) 

The comment's safety claim — "it can only act on promises that were created via `schedule`" — is correct in isolation, but it ignores the **ordering dependency** between multiple scheduled promises. The `schedule` function assigns monotonically increasing nonces:

```rust
pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
    self.assert_preconditions();
    let nonce = self.nonce.get().unwrap_or_default();
    self.scheduled_promises.insert(nonce, promise);
    self.nonce.set(&(nonce + 1));
    near_sdk::log!("Promise scheduled at nonce {}", nonce);
}
``` [2](#0-1) 

The router stores multiple independent promises keyed by nonce in `scheduled_promises: LookupMap<u64, PromiseArgs>`. [3](#0-2) 

The critical flaw: `execute_scheduled` calls `self.scheduled_promises.remove(&nonce.0)` **before** the promise executes. In NEAR, this state mutation is committed atomically with the function call receipt. If the subsequently created promise receipt fails (because it depended on a prior promise's state change), the promise is **permanently gone** from storage — the user cannot retry it.

The XCC precompile in `engine-precompiles/src/xcc.rs` confirms that `CrossContractCallArgs::Delayed` schedules promises for later execution via this exact path:

```rust
CrossContractCallArgs::Delayed(call) => {
    ...
    let promise = PromiseCreateArgs {
        target_account_id,
        method: consts::ROUTER_SCHEDULE_NAME.into(),
        ...
    };
    (promise, attached_near)
}
``` [4](#0-3) 

---

### Impact Explanation

**High → Critical: Temporary or permanent freezing of user funds.**

When a user schedules two dependent XCC promises:
- **Nonce 0**: `ft_transfer` of 1000 USDC to a DEX contract
- **Nonce 1**: `swap` call on the DEX that requires the 1000 USDC to already be present

An adversary calls `execute_scheduled({"nonce": "1"})` first. The swap fails (no USDC in DEX yet), but promise 1 is **permanently removed** from storage. The user then calls `execute_scheduled({"nonce": "0"})`, transferring 1000 USDC to the DEX. There is now no XCC mechanism to trigger the swap — the USDC is stranded in the DEX contract. Recovery depends entirely on whether the DEX has a manual withdrawal path, which is not guaranteed.

This maps directly to the external report's vulnerability class: **sequencing of approved operations can be exploited by an unprivileged third party**, with the Aurora analog being strictly worse because the attacker need not be a co-owner — they can be any NEAR account.

---

### Likelihood Explanation

**Medium.** The XCC `Delayed` mode is a production feature explicitly designed for complex multi-step NEAR interactions. Any user performing a multi-step DeFi operation (approve + swap, deposit + borrow, etc.) across separate EVM transactions is exposed. The attack requires only:
1. Observing the NEAR mempool or on-chain state for a router with multiple scheduled promises.
2. Calling `execute_scheduled` with a later nonce before the earlier one.

No special privileges, keys, or governance access are required. The function is public and `#[payable]` (no deposit required to call it).

---

### Recommendation

1. **Enforce sequential execution**: Require that `execute_scheduled(n)` can only be called if nonce `n-1` has already been executed (i.e., is no longer in `scheduled_promises`). This mirrors the nonce-ordering guarantee that EVM transactions provide via `check_nonce`.
2. **Alternatively, restrict the caller**: Allow only the parent Aurora engine (or the EVM address owner) to call `execute_scheduled`, removing the "anyone can call" property. The comment's safety argument only holds when promises are independent; for dependent promises, open access is unsafe.
3. **Or, restore on failure**: Use a NEAR callback on `execute_scheduled` that re-inserts the promise if the downstream call fails, preventing permanent loss.

### Proof of Concept

```
1. Alice (EVM address 0xABC) calls XCC precompile with CrossContractCallArgs::Delayed:
   - Promise A (nonce 0): ft_transfer(1000 USDC → DEX)
   - Promise B (nonce 1): swap(USDC → ETH) [depends on A completing first]

2. Both are stored in router at {0xabc...}.aurora:
   scheduled_promises[0] = Promise A
   scheduled_promises[1] = Promise B

3. Adversary (any NEAR account) calls:
   {0xabc...}.aurora.execute_scheduled({"nonce": "1"})

4. Promise B is removed from storage, executed → FAILS (no USDC in DEX yet).
   Promise B is permanently gone.

5. Alice calls:
   {0xabc...}.aurora.execute_scheduled({"nonce": "0"})

6. Promise A executes → 1000 USDC transferred to DEX.
   No Promise B exists to trigger the swap.
   1000 USDC is permanently stranded in the DEX.
```

The root cause is at `etc/xcc-router/src/lib.rs` line 151 (`self.scheduled_promises.remove`) combined with the absence of any caller restriction or sequential ordering enforcement in `execute_scheduled`. [5](#0-4)
