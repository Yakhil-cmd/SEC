### Title
Integer Underflow in EIP-2935 System Hook on Genesis Block (block 0) — (`basic_bootloader/src/bootloader/block_flow/ethereum/eip_2935_historical_block_hash/mod.rs`)

---

### Summary

`eip2935_system_part` performs `block_number - 1` on a `u64` without guarding against `block_number == 0`. When the genesis block (block 0) is processed and the EIP-2935 history-storage contract is deployed, this subtraction underflows: it panics in debug/test mode and wraps to `u64::MAX` in release mode, causing the parent-hash to be stored at the wrong history-storage slot.

---

### Finding Description

`eip2935_system_part` is called as a system-level hook at the end of every block to record the parent block hash into the EIP-2935 history-storage contract. The function unconditionally computes `block_number - 1` on two separate lines:

```rust
let block_number = system.get_block_number();
let parent_hash = system.get_blockhash(block_number - 1)?;   // line 51
// ...
let slot_idx = (block_number - 1) % HISTORY_SERVE_WINDOW;    // line 55
``` [1](#0-0) 

There is no guard for `block_number == 0`. The only early-exit guard is the `is_contract` check (whether the EIP-2935 contract is deployed), which is unrelated to the block number:

```rust
let is_contract = props.nonce.0 == 1 && props.observable_bytecode_len.0 > 0;
if is_contract == false {
    return Ok(());
}
``` [2](#0-1) 

By contrast, the `form_block_header` helper in the same crate correctly guards the identical subtraction:

```rust
let previous_block_hash = if block_number == 0 {
    Bytes32::ZERO
} else {
    system.get_blockhash(block_number - 1)?
};
``` [3](#0-2) 

The `eip2935_system_part` function lacks this identical guard.

**Execution paths:**

| Mode | `block_number - 1` result | `get_blockhash` result | `slot_idx` | Net effect |
|---|---|---|---|---|
| Debug (overflow-checks on) | **panic** | — | — | Block 0 processing aborts |
| Release (RISC-V proving binary) | `u64::MAX` (wraps) | `Bytes32::ZERO` (out-of-range → zero) | `u64::MAX % 8191 = 4095` | Writes `Bytes32::ZERO` to wrong slot 4095 |

`get_blockhash` safely returns `Bytes32::ZERO` for `u64::MAX` because the out-of-range check fires:

```rust
if block_number >= current_block_number
    || block_number < current_block_number.saturating_sub(256)
{
    Ok(Bytes32::ZERO)
}
``` [4](#0-3) 

So in release mode the write value is accidentally correct (`Bytes32::ZERO`), but the destination slot (`4095`) is wrong. If slot 4095 was already zero (fresh chain), this is a no-op state write. However, the storage model may still record it as a dirty entry, polluting pubdata / DA commitment for the genesis block.

---

### Impact Explanation

**Debug / test mode (overflow-checks enabled):** Processing block 0 with the EIP-2935 contract deployed causes an immediate panic. This prevents genesis-block execution in any test or development environment that pre-deploys the EIP-2935 contract, blocking the entire chain from starting.

**Release mode (RISC-V proving binary):** The subtraction wraps to `u64::MAX`. The parent hash written is `Bytes32::ZERO` (correct for genesis), but it is stored at slot `4095` instead of no slot at all. If the storage model records this as a net diff, the genesis-block pubdata / DA commitment will contain a spurious zero-write to slot 4095 of the EIP-2935 contract. This creates a state-transition divergence: the settlement layer's view of the genesis state will include an unexpected storage entry, which may cause proof verification to fail or produce an incorrect `chain_state_commitment_before` for block 1.

---

### Likelihood Explanation

This triggers on the very first block (block 0) whenever the EIP-2935 history-storage contract is deployed. For any EVM-equivalent ZKsync OS chain that pre-deploys EIP-2935 (as required for EIP-2935 compliance), this fires unconditionally on genesis. No attacker action is required — it is a deterministic first-block bug identical in structure to the external report's first-batch bug.

---

### Recommendation

Add a `block_number == 0` guard at the top of `eip2935_system_part`, mirroring the pattern already used in `form_block_header`:

```rust
let block_number = system.get_block_number();
if block_number == 0 {
    // Genesis block has no parent; nothing to record.
    return Ok(());
}
let parent_hash = system.get_blockhash(block_number - 1)?;
let slot_idx = (block_number - 1) % HISTORY_SERVE_WINDOW;
```

---

### Proof of Concept

1. Pre-deploy the EIP-2935 contract at `HISTORY_STORAGE_ADDRESS` with `nonce = 1` and non-zero bytecode length (satisfying the `is_contract` check).
2. Execute block 0 (genesis block, `block_number = 0`).
3. **Debug mode:** execution panics at `block_number - 1` (u64 overflow).
4. **Release mode:** `slot_idx = u64::MAX % 8191 = 4095`; `Bytes32::ZERO` is written to slot 4095 of the history-storage contract instead of being skipped entirely. The genesis-block state diff / pubdata contains a spurious storage write. [5](#0-4)
