### Title
Unprivileged Caller Can Inject Arbitrary Interop Roots via Crafted Event Emission — (File: `system_hooks/src/event_hooks/interop_root_reporter.rs`)

---

### Summary

The `interop_root_reporter_event_hook` processes any event whose first topic matches `INTEROP_ROOT_ADDED_EVENT_SIG`, without verifying that the event was emitted by the authoritative `L2_INTEROP_ROOT_STORAGE_ADDRESS` contract. Any unprivileged EVM contract can emit a log with that exact topic and well-formed data, causing the bootloader to unconditionally append an attacker-controlled `InteropRoot` to the in-memory `InteropRootStorage`. This corrupts the `interop_roots_rolling_hash` that is committed to in the batch public inputs, producing an invalid or manipulated state-transition output.

---

### Finding Description

`interop_root_reporter_event_hook` is registered as a system-level event hook and is invoked for every event emitted during block execution. Its only guard is a topic-signature check:

```rust
if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
    return Ok(());
}
``` [1](#0-0) 

There is no check on the address that emitted the event. The `_caller_ee` parameter (execution-environment type) is explicitly ignored (underscore prefix), and no emitter-address parameter is present in the function signature at all. [2](#0-1) 

After passing the topic check, the hook unconditionally parses `chain_id` and `block_or_batch_number` from the topics and `root` from the data, then calls `system.io.add_interop_root(...)`: [3](#0-2) 

`add_interop_root` charges native resources and calls `push_root`, which performs no content validation: [4](#0-3) 

The only zero-root validation that exists lives inside the `L2_INTEROP_ROOT_STORAGE_ADDRESS` Solidity contract (enforced by the service-transaction path). An attacker who bypasses that contract by emitting the event directly from their own EVM contract completely sidesteps that check.

The injected roots are later folded into the `interop_roots_rolling_hash` during batch finalization: [5](#0-4) 

This hash is part of the public inputs committed to by the prover, so injecting fake roots produces a provably incorrect or manipulated batch commitment.

---

### Impact Explanation

**Vulnerability class:** state-transition bug / oracle IO mismatch.

An attacker can:
1. Deploy an EVM contract that emits a log with topic `0x6b451b8422636e45b93bf7f594fa2c1769d039766c4254a6e7f9c0ee1715cdb0` and 96 bytes of attacker-controlled data (offset=32, len=1, arbitrary root hash).
2. Call that contract from a normal L2 transaction.
3. The hook fires, appending an `InteropRoot` with arbitrary `chain_id`, `block_or_batch_number`, and `root` to the in-memory storage.
4. At batch finalization the rolling hash is computed over the legitimate roots **plus** the injected ones, producing a wrong `interop_roots_rolling_hash` in the public inputs.

Consequences:
- **Batch commitment corruption**: the settlement layer receives a public-input hash that does not correspond to the true set of imported interop roots, breaking cross-chain state verification.
- **Proof/forward divergence**: the forward run and the proving run will both include the injected root (since the hook fires in both modes), so the divergence is not caught by consistency checks, but the committed value is semantically wrong.
- **Denial of valid interop messages**: downstream consumers that verify the rolling hash against an expected value will reject the batch.

---

### Likelihood Explanation

The attack requires only a standard L2 transaction that calls a deployed EVM contract. No privileged keys, no governance access, no oracle manipulation. The `INTEROP_ROOT_ADDED_EVENT_SIG` value is a public constant. The required calldata format (96-byte ABI-encoded payload with offset=32, len=1, 32-byte root) is fully derivable from the open-source hook code. Any user with enough gas to deploy a contract and call it can execute this attack in a single block.

---

### Recommendation

Add an emitter-address check at the top of `interop_root_reporter_event_hook`. The hook should only process events whose emitter is `L2_INTEROP_ROOT_STORAGE_ADDRESS`:

```rust
// Reject events not emitted by the authoritative interop root storage contract
if emitter != L2_INTEROP_ROOT_STORAGE_ADDRESS {
    return Ok(());
}
```

This requires threading the emitter address through the event-hook dispatch call site (adding it as a parameter alongside `_caller_ee`). Alternatively, the hook can be restructured so it is only reachable from the service-transaction execution path, not from arbitrary EVM execution.

---

### Proof of Concept

1. Deploy the following minimal EVM contract on the ZKsync OS chain:

```solidity
contract FakeInteropEmitter {
    // INTEROP_ROOT_ADDED_EVENT_SIG
    bytes32 constant SIG =
        0x6b451b8422636e45b93bf7f594fa2c1769d039766c4254a6e7f9c0ee1715cdb0;

    function inject(uint256 chainId, uint256 blockNum, bytes32 root) external {
        // topics: [SIG, chainId, blockNum]
        // data:   abi.encode(uint256(32), uint256(1), root)
        bytes memory data = abi.encode(uint256(32), uint256(1), root);
        assembly {
            log3(add(data, 32), mload(data),
                 0x6b451b8422636e45b93bf7f594fa2c1769d039766c4254a6e7f9c0ee1715cdb0,
                 chainId, blockNum)
        }
    }
}
```

2. Call `inject(1, 999, 0xdeadbeef...)` from a normal L2 transaction.
3. Observe that `interop_root_reporter_event_hook` fires (topic matches, data length = 96, offset = 32, len = 1).
4. The `InteropRoot { chain_id: 1, block_or_batch_number: 999, root: 0xdeadbeef... }` is appended to `InteropRootStorage`.
5. At batch finalization, `calculate_interop_roots_rolling_hash` folds this fake root into the rolling hash, producing a corrupted public input. [6](#0-5) [5](#0-4)

```rust
pub fn calculate_interop_roots_rolling_hash<'a>(
    old_rolling_hash: Bytes32,
    roots: impl Iterator<Item = &'a InteropRoot>,
    hasher: &mut crypto::sha3::Keccak256,
) -> Bytes32 {
    let mut data = [0u8; 96];

    let mut rolling_hash = old_rolling_hash;
    for root in roots {
        data[0..32].copy_from_slice(&rolling_hash.as_u8_ref());
        data[32..64].copy_from_slice(&root.chain_id.to_be_bytes::<{ U256::BYTES }>());
        data[64..96].copy_from_slice(&root.block_or_batch_number.to_be_bytes::<{ U256::BYTES }>());
        hasher.update(data);

        // Note: now we have only one side
        hasher.update(root.root.as_u8_ref());

        rolling_hash = hasher.finalize_reset().into()
    }

    rolling_hash
}
