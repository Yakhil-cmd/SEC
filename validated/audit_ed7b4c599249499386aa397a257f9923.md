### Title
`native_per_pubdata` Computed with Floor Division Instead of Ceiling Division Causes Systematic Underpayment for Pubdata - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS, the `native_per_pubdata` ratio — which determines how many native (proving) resources a user must pay per byte of pubdata — is computed using floor division (`wrapping_div`). In contrast, the analogous `native_per_gas` ratio is explicitly computed using ceiling division (`div_ceil`). This asymmetric rounding direction means every transaction that uses pubdata pays slightly less native resource than the actual cost, causing the protocol to systematically absorb the difference. The error accumulates across all transactions, analogous to the vault rounding bug in the reference report.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the two key resource ratios are computed as follows:

```rust
// native_per_gas: rounds UP (ceil) — favors the protocol
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...
``` [1](#0-0) 

```rust
// native_per_pubdata: rounds DOWN (floor) — favors the user
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [2](#0-1) 

The same asymmetry is present in the off-chain validation helper:

```rust
// native_per_gas = ceil(gas_price / native_price)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(())?;

// native_per_pubdata = pubdata_price / native_price  (floor, no comment)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

The `native_per_pubdata` value is then used in two places to charge users:

1. **Intrinsic pubdata overhead** during resource creation:
   `intrinsic_pubdata_overhead = native_per_pubdata * intrinsic_pubdata` [4](#0-3) 

2. **Execution pubdata charge** at refund time:
   `native = current_pubdata_spent * native_per_pubdata` [5](#0-4) 

When `pubdata_price` is not perfectly divisible by `native_price`, `wrapping_div` truncates the result. The true cost per pubdata byte is `ceil(pubdata_price / native_price)`, but users are charged `floor(pubdata_price / native_price)`. The difference is between 0 and `native_price - 1` native units per pubdata byte.

The double-resource accounting model is documented explicitly: [6](#0-5) 

The design intent is `nativePerGas := gasPrice / nativePrice`, but the document does not specify rounding direction for pubdata. The explicit use of `div_ceil` for `native_per_gas` (with a comment) and the absence of any such comment for `native_per_pubdata` strongly suggests the floor rounding is unintentional.

---

### Impact Explanation

For every transaction that uses pubdata, the user pays up to `(native_price - 1)` fewer native units per pubdata byte than the actual cost. This means:

- The operator/protocol absorbs the rounding shortfall for every transaction.
- A transaction with `N` pubdata bytes underpays by up to `N * (native_price - 1)` native units.
- Over many transactions, this accumulates into a systematic resource deficit for the protocol.
- An attacker can maximize the underpayment by submitting transactions that generate large amounts of pubdata (e.g., many storage writes), paying less native resource than the actual proving cost.

This is a resource accounting bug where rounding favors users over the protocol — the direct analog of the vault rounding bug in the reference report.

---

### Likelihood Explanation

- Every L2 transaction that uses pubdata triggers this path. No special privileges are required.
- The entry point is any unprivileged EVM transaction that writes to storage or emits logs (both generate pubdata).
- The rounding error is deterministic and occurs on every such transaction.
- The magnitude per transaction is small but non-zero whenever `pubdata_price % native_price != 0`, which is the common case for arbitrary operator-set prices.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for the `native_per_pubdata` calculation, consistent with how `native_per_gas` is computed:

```rust
// Before (rounds down, favors user):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, favors protocol):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. [7](#0-6) 

---

### Proof of Concept

**Setup:**
- `native_price = 10`
- `pubdata_price = 19` (not divisible by 10)
- `native_per_pubdata = floor(19 / 10) = 1` (actual cost = `ceil(19/10) = 2`)

**Per transaction with 1000 pubdata bytes:**
- User is charged: `1000 * 1 = 1000` native units
- Actual cost: `1000 * 2 = 2000` native units
- Protocol shortfall: `1000` native units per transaction

**Over 1,000,000 transactions:**
- Total shortfall: `1,000,000,000` native units absorbed by the protocol

Any unprivileged user submitting a transaction that writes to storage (e.g., a simple `SSTORE` opcode) will trigger this path. The attacker does not need to do anything special — the rounding error is structural and occurs on every pubdata-generating transaction.
