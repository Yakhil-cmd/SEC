### Title
Funds Permanently Frozen When `gas_limit > u64::MAX` Causes `GasOverflow` After `charge_gas` Deduction — (`engine/src/engine.rs`)

### Summary

In `submit_with_alt_modexp`, `charge_gas` successfully deducts `gas_limit * effective_gas_price` from the sender's balance using `U256` arithmetic. Immediately after, `transaction.gas_limit.try_into::<u64>()` fails for any `gas_limit > u64::MAX`, returning early via `?` and **bypassing the `refund_unused_gas` call entirely**. The deducted funds are never returned.

### Finding Description

The execution order in `submit_with_alt_modexp` is:

1. **`charge_gas` succeeds** — uses `transaction.gas_limit` as `U256` to compute `prepaid_amount = gas_limit * effective_gas_price`, deducts it from sender balance via `set_balance`.
2. **`gas_limit.try_into::<u64>()` fails** — returns `Err(EngineErrorKind::GasOverflow)` via `?`, causing an **early return** from the function.
3. **`refund_unused_gas` is never reached** — it sits after the `try_into` call and is skipped entirely.

```
// Step 1: balance deducted here
let prepaid_amount = match engine.charge_gas(&sender, &transaction, max_gas_price, fixed_gas) {
    Ok(gas_result) => gas_result,
    Err(err) => { return Err(...); }
};

// Step 2: early return here if gas_limit > u64::MAX — skips refund
let gas_limit = transaction
    .gas_limit
    .try_into()
    .map_err(|_| EngineErrorKind::GasOverflow)?;  // <-- early return

// ... EVM execution ...

// Step 3: NEVER REACHED when GasOverflow fires above
refund_unused_gas(&mut io, &sender, gas_used, &prepaid_amount, &relayer_address, fixed_gas)
    .map_err(...)?;
``` [1](#0-0) [2](#0-1) 

The `charge_gas` function computes `prepaid_amount` using `transaction.gas_limit` as a raw `U256` — no `u64` bound is enforced there: [3](#0-2) 

The `gas_limit` field in `Transaction1559` and `NormalizedEthTransaction` is typed as `U256`, so any value up to `2^256 - 1` can be RLP-encoded and decoded without error: [4](#0-3) [5](#0-4) 

The normalization path (`TryFrom<EthTransactionKind>`) passes `gas_limit` through verbatim with no range check: [6](#0-5) 

The intrinsic-gas check at line 1079 compares `transaction.gas_limit` (U256) against `u64` intrinsic gas cast to `U256`, so `u64::MAX + 1 > 21000` passes trivially: [7](#0-6) 

### Impact Explanation

A sender with balance ≥ `(u64::MAX + 1) * effective_gas_price` wei who submits a transaction with `gas_limit = u64::MAX + 1` will have that entire prepaid amount permanently removed from their balance with no refund. `u64::MAX` wei ≈ 18.4 ETH, so this is a realistic balance threshold on Aurora. The funds are written to storage via `set_balance` and never restored, constituting **permanent freezing of funds**.

### Likelihood Explanation

The path is reachable by any user who submits a raw EIP-1559 transaction directly to the Aurora engine (via the `submit` or `submit_with_args` NEAR method). No admin access or special privilege is required. The scenario can be triggered accidentally (a wallet or tool that sets an absurdly high gas limit) or deliberately by a griefing attacker targeting their own funds. The only precondition is a balance ≥ `u64::MAX + 1` wei and a crafted transaction byte string.

### Recommendation

Move the `gas_limit.try_into::<u64>()` conversion **before** the `charge_gas` call, or add an explicit guard rejecting `gas_limit > u64::MAX` before any balance deduction occurs. This ensures `GasOverflow` is returned before any funds are touched.
