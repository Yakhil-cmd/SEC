### Title
Missing Failure-Recovery Callback in XCC Promise Chain Permanently Freezes User wNEAR — (`engine/src/xcc.rs`)

---

### Summary

When a user initiates a cross-contract call (XCC) that requires attached NEAR (funded via wNEAR), the Aurora Engine burns the user's wNEAR ERC-20 balance and sends real NEAR to the router sub-account before the target call executes. If the target cross-contract call subsequently fails, the NEAR is returned by the NEAR runtime to the router sub-account, but there is **no failure-handling callback** in the promise chain to re-wrap that NEAR back into wNEAR and credit it to the user. The user's wNEAR is permanently destroyed with no recourse.

---

### Finding Description

The XCC promise chain is constructed in `handle_precompile_promise` in `engine/src/xcc.rs`. When `required_near > ZERO_YOCTO`, the chain is:

1. **[Optional]** Deploy/upgrade router sub-account batch.
2. **[Optional]** `factory_update_address_version` callback on the engine.
3. `withdraw_wnear_to_router` callback on the engine — this burns the user's wNEAR ERC-20 tokens and sends real NEAR to the router.
4. **[Optional]** `send_refund` callback on the router (only when `create_needed == true`).
5. The user's actual target cross-contract call, attached as a callback to the result of step 3/4. [1](#0-0) 

The `withdraw_wnear_to_router` engine method (step 3) checks whether the *prior* promise (setup) failed and aborts early if so — correctly preventing the wNEAR burn in that case: [2](#0-1) 

However, once `withdraw_wnear_to_router` succeeds (wNEAR burned, NEAR delivered to router), the user's target call in step 5 is attached with no corresponding **failure callback**. If the target call panics or returns an error, the NEAR protocol returns the unused attached NEAR to the router sub-account. The promise chain ends there — no callback exists to detect this failure, re-wrap the NEAR via the wNEAR contract, and restore the user's ERC-20 balance.

The router sub-account is a NEAR account of the form `{evm_address}.aurora`: [3](#0-2) 

The router exposes only `execute`, `execute_scheduled`, `schedule`, `send_refund`, and `deploy_upgrade`. There is no `withdraw_near` or equivalent function that would allow the user or the engine to recover stranded NEAR from the router back into the wNEAR/ERC-20 accounting system. [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Every XCC call that attaches NEAR (via wNEAR) and whose target call fails results in:
- The user's wNEAR ERC-20 balance permanently reduced (burned).
- An equivalent amount of NEAR stranded in the router sub-account with no on-chain path to recover it into the ERC-20 accounting layer.

The NEAR is not lost to an attacker; it accumulates in the router. But from the user's perspective the wNEAR is gone and cannot be recovered, satisfying the **permanent fund freeze** impact criterion.

---

### Likelihood Explanation

**High.** Any XCC call that attaches NEAR and whose target function reverts triggers this. Target-side failures are routine (wrong arguments, insufficient gas forwarded, target contract logic errors, target account not existing). No special attacker capability is required — an ordinary EVM user calling the XCC precompile with any non-zero wNEAR attachment is sufficient. [5](#0-4) 

---

### Recommendation

Attach a **failure callback** after the user's target call (step 5). This callback should:
1. Check `promise_result_check()` for failure.
2. If failed, call `wrap_near` (or equivalent) on the wNEAR contract to re-wrap the NEAR that was returned to the router, then transfer the resulting wNEAR back to the engine and credit the user's ERC-20 balance.

This mirrors the TBTC fix pattern: add an explicit "close/release" step at the end of the lifecycle to return locked resources to their rightful owner when the happy path does not complete.

---

### Proof of Concept

1. User holds 1 wNEAR as ERC-20 on Aurora.
2. User calls the XCC precompile with `CrossContractCallArgs::Eager(PromiseArgs::Create(...))` targeting a non-existent NEAR account, with `attached_balance = 1_000_000_000_000_000_000_000_000` yoctoNEAR.
3. Engine executes `handle_precompile_promise`: burns 1 wNEAR from user's ERC-20 balance, sends 1 NEAR to `{user_address}.aurora` router, then attaches the target call.
4. Target call fails (account does not exist). NEAR runtime returns the 1 NEAR to the router.
5. Promise chain ends. No failure callback fires.
6. User's wNEAR ERC-20 balance: 0. NEAR in router: 1 NEAR. No on-chain path to recover. [6](#0-5) [7](#0-6)

pub fn withdraw_wnear_to_router<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;
        if matches!(handler.promise_result_check(), Some(false)) {
            return Err(b"ERR_CALLBACK_OF_FAILED_PROMISE".into());
        }
        let args: WithdrawWnearToRouterArgs = io.read_input_borsh()?;
        let current_account_id = env.current_account_id();
        let recipient = AccountId::try_from(format!(
            "{}.{}",
            args.target.encode(),
            current_account_id.as_ref()
        ))?;
        let wnear_address = aurora_engine_precompiles::xcc::state::get_wnear_address(&io);
        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&current_account_id),
            current_account_id,
            io,
            env,
        );
        let (result, ids) = xcc::withdraw_wnear_to_router(
            &recipient,
            args.amount,
            wnear_address,
            &mut engine,
            handler,
        )?;
        if !result.status.is_ok() {
            return Err(b"ERR_WITHDRAW_FAILED".into());
        }
        let id = ids.last().ok_or(b"ERR_NO_PROMISE_CREATED")?;
        handler.promise_return(*id);
        Ok(result)
    })
