### Title
Unbounded `CANISTER_DISPLAY_NAMES` HashMap Growth via Attacker-Controlled `network_identifier.network` Causes OOM DoS — (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`)

---

### Summary

The `RosettaMetricsMiddleware` unconditionally inserts every unique value of `network_identifier.network` from any incoming POST body into the process-global `CANISTER_DISPLAY_NAMES: Mutex<HashMap<String,String>>`. There is no size cap, no eviction, and no validation. An unauthenticated attacker can exhaust the Rosetta process's heap by sending requests with distinct strings in that field.

---

### Finding Description

`CANISTER_DISPLAY_NAMES` is declared as a process-global `lazy_static`: [1](#0-0) 

`RosettaMetrics::new` unconditionally inserts into it on every call: [2](#0-1) 

`RosettaMetricsMiddleware::call` calls `RosettaMetrics::new` for every request that carries a `network_identifier.network` field, using the attacker-supplied string as both key and value (because `get_display_name_from_canister_id` returns the ID itself when it is not already registered): [3](#0-2) 

`extract_canister_id` performs no length or character validation — it accepts any JSON string: [4](#0-3) 

The ICRC-1 Rosetta server applies this middleware to **all** routes and binds to `0.0.0.0:8080` with no authentication: [5](#0-4) 

A secondary, compounding vector exists: every unique `token_display_name` passed to `inc_api_status_count` and `observe_request_duration` creates a new Prometheus time-series label set in the default registry, which also has no eviction: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

Each unique string sent in `network_identifier.network` permanently occupies heap memory in two places: the `CANISTER_DISPLAY_NAMES` HashMap and the Prometheus registry. With arbitrarily long strings (the body limit in the ICP Rosetta server is 4 MB; the ICRC-1 server has no explicit body limit — `usize::MAX` is passed to `to_bytes`) and arbitrarily many requests, the Rosetta process heap grows without bound until the OS OOM-killer terminates it, causing a complete denial of service of the Rosetta node. [8](#0-7) 

---

### Likelihood Explanation

The attack requires only unauthenticated HTTP POST access to any Rosetta endpoint (e.g., `/network/status`). No credentials, no privileged role, and no protocol-level access are needed. The default bind address is `0.0.0.0`, making the port reachable from any network-adjacent host. The attack is trivially scriptable.

---

### Recommendation

1. **Cap `CANISTER_DISPLAY_NAMES`**: enforce a maximum entry count (e.g., 1 000) and reject or ignore insertions beyond that limit.
2. **Validate the extracted canister ID**: accept only strings that match the canonical IC principal text format (e.g., `[a-z0-9]{5}(-[a-z0-9]{5}){4}`) before inserting into the map or using as a Prometheus label.
3. **Do not create new `RosettaMetrics` instances per request**: the middleware should look up an existing, pre-registered `RosettaMetrics` by canister ID and fall back to the default rather than constructing a new one (and thus a new Prometheus label set) for every unrecognised ID.
4. **Apply a request body size limit** in the ICRC-1 Axum server consistent with the 4 MB limit already present in the ICP Actix server.

---

### Proof of Concept

```python
import requests, string, random

TARGET = "http://<rosetta-host>:8080/network/status"

for i in range(100_000):
    unique_id = ''.join(random.choices(string.ascii_lowercase, k=64)) + str(i)
    requests.post(TARGET, json={
        "network_identifier": {"blockchain": "Internet Computer", "network": unique_id},
        "metadata": {}
    })
    if i % 1000 == 0:
        print(f"Sent {i} requests")
# Monitor RSS of the rosetta process: grows proportionally to i
```

```rust
for i in 0..N {
    send_post("/network/status", json!({"network_identifier":{"network": format!("fake-{i}")}}));
}
assert_eq!(CANISTER_DISPLAY_NAMES.lock().unwrap().len(), N); // holds for arbitrary N
```
