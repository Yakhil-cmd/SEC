### Title
Unbounded `distribute_rewards` Loop Due to Insufficient `round_duration_seconds` Floor Validation Causes Permanent Instruction-Exhaustion DOS - (`rs/sns/governance/src/governance.rs`)

---

### Summary

`VotingRewardsParameters.round_duration_seconds` is validated to a minimum of 1 second. When an SNS sets this to a very small value and the canister subsequently misses any significant period of reward distribution (e.g., after an upgrade), the `distribute_rewards` function iterates an unbounded number of rounds in a single message, exhausting the instruction limit and permanently DOSing SNS reward distribution.

---

### Finding Description

`VotingRewardsParameters.round_duration_seconds` controls the length of each reward round. Its validation in `round_duration_seconds_defects` enforces only `1..=MAX_REWARD_ROUND_DURATION_SECONDS` (~1 year): [1](#0-0) 

The default value is `ONE_DAY_SECONDS` (86400 s): [2](#0-1) 

Inside `distribute_rewards`, the number of missed rounds is computed without any upper bound: [3](#0-2) 

This value is then used directly as the loop bound: [4](#0-3) 

Each iteration performs floating-point arithmetic via `reward_rate_at`. There is no guard on `new_rounds_count` being too large before entering the loop. If `round_duration_seconds = 1` and the canister misses even 1 hour of heartbeats (e.g., during an upgrade), `new_rounds_count = 3600`. If it misses 1 day, `new_rounds_count = 86400`. If it misses 1 week, `new_rounds_count = 604800`. At 1 second per round and any multi-day gap, the loop exhausts the IC instruction limit (~40B instructions for update calls), causing the canister to trap.

Because the trap rolls back state without advancing `latest_reward_event`, every subsequent call to `run_periodic_tasks` recomputes the same (or larger) `new_rounds_count` and traps again — a permanent DOS.

The validation only checks presence for `max_dissolve_delay_seconds` and `max_neuron_age_for_age_bonus` (no ceiling or floor): [5](#0-4) 

In contrast, parameters like `initial_voting_period_seconds` and `max_proposals_to_keep_per_action` have explicit floor and ceiling constants: [6](#0-5) 

---

### Impact Explanation

If `round_duration_seconds` is set to 1 second and the SNS governance canister misses any significant period of reward distribution (routine upgrade, subnet maintenance, etc.), `distribute_rewards` will attempt to iterate millions of rounds in a single message, exhausting the instruction limit. The canister traps, state rolls back, and the same condition repeats on every subsequent heartbeat. SNS reward distribution is permanently halted. Neurons accumulate no maturity, and the governance canister's periodic task loop is effectively bricked for reward purposes.

---

### Likelihood Explanation

An SNS community can legitimately set `round_duration_seconds = 1` via a `ManageNervousSystemParameters` proposal — the validation explicitly permits it. Any subsequent canister upgrade (a routine operation) introduces a gap during which no heartbeats fire. Even a gap of a few hours at 1 s/round produces tens of thousands of iterations. A gap of days produces hundreds of thousands. The DOS is triggered by a combination of a governance-approved parameter value and any normal operational event, not by a targeted attack.

---

### Recommendation

1. Establish a safe minimum for `round_duration_seconds` that prevents `new_rounds_count` from exceeding a safe instruction budget. For example, a minimum of 3600 seconds (1 hour) limits `new_rounds_count` to at most ~8760 per year of missed rounds.
2. Add an explicit cap on `new_rounds_count` inside `distribute_rewards` before the loop, processing at most N rounds per call and deferring the rest to the next heartbeat.
3. Apply the same analysis to `max_dissolve_delay_seconds` and `max_neuron_age_for_age_bonus`, which currently have no floor or ceiling validation at all.

---

### Proof of Concept

1. SNS governance passes a `ManageNervousSystemParameters` proposal setting `voting_rewards_parameters.round_duration_seconds = Some(1)`.
2. The SNS governance canister is upgraded (a normal operation). During the upgrade, no heartbeats fire for ~10 minutes (600 seconds).
3. After upgrade, `run_periodic_tasks` calls `distribute_rewards`. `new_rounds_count = 600 / 1 = 600`. The loop runs 600 iterations — manageable.
4. Now repeat with a longer gap: a subnet upgrade takes 30 minutes → `new_rounds_count = 1800`. Still manageable.
5. A subnet maintenance window of 6 hours → `new_rounds_count = 21600`. Floating-point arithmetic per iteration × 21600 iterations approaches the instruction limit.
6. Any gap exceeding ~12–24 hours at `round_duration_seconds = 1` causes the loop to exhaust the 40B instruction limit, trap, and permanently DOS reward distribution. [7](#0-6) [8](#0-7)
