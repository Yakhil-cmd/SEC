### Title
Paused NEP-141 Token Causes Permanent ERC-20 Fund Loss on ExitToNear Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`)

### Summary

The `ExitToNear` precompile burns ERC-20 tokens in the Aurora EVM before scheduling a NEAR `ft_transfer` promise. When compiled without the `error_refund` feature (the default), no callback is attached for regular ERC-20 exits. If the downstream `ft_transfer` fails — for example because the NEP-141 contract is paused — the burned ERC-20 tokens are permanently lost with no recovery path.

### Finding Description

The `ExitToNear::run()` function in `engine-precompiles/src/native.rs` constructs a `callback_args` struct and only attaches a callback promise when `callback_args != default()`:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For a regular ERC-20 exit (not wNEAR), `transfer_near_args` is `None`. Without `error_refund`, `callback_args` equals the default value, so the promise is created without any callback:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

The `exit_to_near_precompile_callback` function, which is the only place where a refund can be issued, is never scheduled in this path:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None  // ← no refund, tokens are gone
};
``` [3](#0-2) 

The `error_refund` feature is opt-in (non-default in Rust), meaning the production binary compiled without `--features error_refund` has no refund path. The existing test explicitly acknowledges this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a user calls `withdrawToNear` on an `EvmErc20` contract:
1. The ERC-20 contract burns the user's tokens in the Aurora EVM.
2. The `ExitToNear` precompile schedules `ft_transfer` on the NEP-141 contract.
3. If `ft_transfer` fails (NEP-141 paused, recipient unregistered, contract deleted, etc.), the NEAR-side transfer never completes.
4. Without `error_refund`, no callback exists to re-mint the tokens in the EVM.
5. The tokens are burned in Aurora and never arrive on NEAR — permanently destroyed. [5](#0-4) 

### Likelihood Explanation

**Medium.** Many NEP-141 token contracts implement pause functionality (analogous to the ERC-721 pause in the external report). Any NEP-141 that is paused, has its `ft_transfer` restricted, or whose recipient account is not storage-registered will trigger this failure. The `ExitToNear` path is a core user-facing operation. The `error_refund` feature being opt-in means production deployments that omit it are silently vulnerable. [6](#0-5) 

### Recommendation

1. **Enable `error_refund` unconditionally** or promote it to a default feature so the refund callback is always attached for ERC-20 exits.
2. Alternatively, restructure the code so that a failure-handling callback is always attached regardless of the feature flag, minting tokens back to the user on any `ft_transfer` failure.
3. Consider a pull-based withdrawal pattern: record the intent to withdraw in Aurora state first, then allow the user to finalize after the NEP-141 transfer succeeds, rather than burning tokens speculatively.

### Proof of Concept

1. User holds 100 units of `TOKEN` (ERC-20 on Aurora, backed by `token.near` NEP-141).
2. `token.near` NEP-141 contract is paused by its owner.
3. User calls `TOKEN.withdrawToNear("user.near", 100)` on Aurora.
4. ERC-20 contract burns 100 tokens from user's EVM balance.
5. `ExitToNear` precompile schedules `token.near::ft_transfer({"receiver_id":"user.near","amount":"100"})`.
6. `ft_transfer` panics because `token.near` is paused.
7. Without `error_refund`: no callback fires, user's 100 tokens are gone from both Aurora EVM and NEAR.
8. User's EVM balance: 0. `token.near` balance of `user.near`: 0. Funds permanently lost. [7](#0-6) [8](#0-7)

```rust
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
