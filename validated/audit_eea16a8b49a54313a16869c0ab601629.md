### Title
Unvalidated `native_price` and `pubdata_price` Oracle Values Not Committed to ZK Public Input Enable Prover-Driven State Divergence — (`basic_bootloader/src/bootloader/block_flow/zk/metadata_op.rs`)

---

### Summary

`BlockMetadataFromOracle` is read from the oracle at block start and used throughout the block for all native-resource and pubdata fee calculations. The fields `native_price` and `pubdata_price` are never committed to in the ZK proof's public input (`BatchPublicInput` / `ChainStateCommitment`). A prover controlling the oracle response can supply arbitrary values for these fields, re-execute the block with different fee parameters, generate a valid proof, and have the settlement layer accept a state transition that diverges from the sequencer's execution.

---

### Finding Description

In `metadata_op.rs`, the ZK-mode bootloader reads the entire `BlockMetadataFromOracle` struct from the oracle and performs only a gas-limit bounds check:

```rust
// basic_bootloader/src/bootloader/block_flow/zk/metadata_op.rs
let block_level_metadata: BlockMetadataFromOracle =
    oracle.query_with_empty_input(BLOCK_METADATA_QUERY_ID)?;   // line 18-19

if metadata.block_gas_limit() > MAX_BLOCK_GAS_LIMIT
    || metadata.individual_tx_gas_limit() > MAX_TX_GAS_LIMIT   // lines 27-31
{
    return Err(internal_error!("block or tx gas limit is too high"));
}
``` [1](#0-0) 

`BlockMetadataFromOracle` carries `native_price` and `pubdata_price` as full `U256` fields: [2](#0-1) 

These values drive every native-resource and pubdata fee calculation in transaction validation:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs
let pubdata_price = system.get_pubdata_price();   // line 106
let native_price  = system.get_native_price();    // line 107
...
if native_price.is_zero() {
    return Err(internal_error!("Native price cannot be 0").into());  // line 123
}
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...  // line 135
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...  // line 142
``` [3](#0-2) 

The only guard is a per-transaction zero-check on `native_price`; there is no block-level validation of either field.

The ZK proof's public input is assembled in `post_tx_op_proving_singleblock_batch.rs` as a `BatchPublicInput` whose preimage is `ChainStateCommitment` (state root, next free slot, block number, last 256 block hashes, last block timestamp) and `BatchOutput` (chain id, timestamps, DA commitment, L1 tx hashes, L2 logs root, upgrade tx hash): [4](#0-3) [5](#0-4) 

Neither `native_price` nor `pubdata_price` appears anywhere in this commitment chain. The oracle documentation acknowledges this gap:

> "Currently it misses `gas_per_pubdata` and `native_price`, but we are already working on design and implementation to solve this issue." [6](#0-5) 

---

### Impact Explanation

Because `native_price` and `pubdata_price` are not committed to in the public input, a prover controlling the oracle can:

1. **Set `native_price` = 1**: `native_per_gas = gas_price / 1 = gas_price`. Every transaction receives a native resource limit equal to its full gas price, making transactions that would normally fail with `NativeResourcesAreTooExpensive` or out-of-native-resources succeed.
2. **Set `pubdata_price` = 0**: `native_per_pubdata = 0`. All pubdata is free; transactions that would normally revert due to pubdata cost succeed.
3. **Set `eip1559_basefee` = 0**: All transactions pass the base-fee check regardless of their `max_fee_per_gas`.

The prover re-executes the block with these manipulated values, obtains a different (but internally consistent) state root, generates a valid ZK proof for it, and submits it to the settlement layer. The settlement layer verifies the proof and updates the on-chain state to this diverged root. The sequencer's state and the proven state are now permanently inconsistent: user balances, transaction outcomes, and fee accounting all differ from what the sequencer computed.

---

### Likelihood Explanation

The attacker-controlled entry path is the prover/oracle input, which the prompt explicitly lists as in-scope. The prover supplies the `BlockMetadataFromOracle` response via `BLOCK_METADATA_QUERY_ID`. No leaked key or governance attack is required — only the ability to control the oracle response fed to the proving execution, which is the prover's normal role. The absence of any commitment to `native_price`/`pubdata_price` in the public input means the manipulation is undetectable by the settlement layer verifier.

---

### Recommendation

1. Include `native_price`, `pubdata_price`, and `eip1559_basefee` in the `BatchOutput` or `ChainStateCommitment` so they become part of the ZK proof's public input and are verified on the settlement layer.
2. Add block-level validation of `native_price` (non-zero, within operator-defined bounds) and `pubdata_price` (within bounds) in `metadata_op.rs`, analogous to the existing `MAX_BLOCK_GAS_LIMIT` check.
3. Ensure the Ethereum block header used in the Ethereum execution mode also commits to these ZKsync-specific pricing fields (the docs already note this is a known gap).

---

### Proof of Concept

**Step 1**: Prover intercepts the `BLOCK_METADATA_QUERY_ID` oracle response and replaces `native_price` with `U256::from(1)` and `pubdata_price` with `U256::ZERO`.

**Step 2**: The bootloader reads these values in `metadata_op.rs` line 18–19 with no further validation.

**Step 3**: In `validation_impl.rs` line 135, `native_per_gas = gas_price.div_ceil(1) = gas_price`. A transaction with `max_fee_per_gas = 1` and `gas_limit = 100_000` now receives `native_limit = 100_000` native units instead of the ~1 unit it would receive at the sequencer's `native_price = 100_000`.

**Step 4**: Transactions that the sequencer rejected with `NativeResourcesAreTooExpensive` or that ran out of native resources mid-execution now succeed. The resulting state root differs from the sequencer's.

**Step 5**: The prover generates a valid ZK proof for this diverged state root. The `BatchPublicInput` contains no field derived from `native_price` or `pubdata_price`, so the proof verifies correctly on the settlement layer.

**Step 6**: The settlement layer updates the on-chain state to the diverged root, permanently separating the sequencer's view of the chain from the proven on-chain state.

            tx_level: TxLevelMetadata::default(),
            block_level: block_level_metadata,

        }

        Ok(metadata)
    }

#[derive(Debug)]
pub struct ChainStateCommitment {
    pub state_root: Bytes32,
    pub next_free_slot: u64,
    pub block_number: u64,
    pub last_256_block_hashes_blake: Bytes32,
    pub last_block_timestamp: u64,
}

impl ChainStateCommitment {
    ///
    /// Calculate blake2s hash of chain state commitment.
    ///
    /// We are using proving friendly blake2s because this commitment will be generated and opened during proving,
    /// but we don't need to open it on the settlement layer.
    ///
    pub fn hash(&self) -> [u8; 32] {
        let mut hasher = crypto::blake2s::Blake2s256::new();
        hasher.update(self.state_root.as_u8_ref());
        hasher.update(&self.next_free_slot.to_be_bytes());
        hasher.update(&self.block_number.to_be_bytes());
        hasher.update(self.last_256_block_hashes_blake.as_u8_ref());
        hasher.update(&self.last_block_timestamp.to_be_bytes());
        hasher.finalize()
    }
