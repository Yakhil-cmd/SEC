Audit Report

## Title
SNS Governance Reward Purse Inflation via Zero `end_timestamp_seconds` Fallback in `distribute_rewards` - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance canister's `distribute_rewards()` and `should_distribute_rewards()` functions both call `.unwrap_or_default()` on the `optional` proto field `end_timestamp_seconds` of `latest_reward_event`. When that field is `None` — as it would be for any SNS whose persisted state predates the introduction of `end_timestamp_seconds` (proto tag 5) — the fallback is `0` (Unix epoch), causing the reward purse to be computed for every round since epoch 0 and distributing massively inflated maturity to all voting neurons. No migration guard exists for the case where `latest_reward_event` is `Some` but `end_timestamp_seconds` is `None`.

## Finding Description

**Root cause — `should_distribute_rewards` (L5735–5738):** [1](#0-0) 

When `end_timestamp_seconds` is `None`, `seconds_since_last_reward_event = now - 0 = now` (~1.7×10⁹ s), which is always greater than `round_duration_seconds`, so the function immediately returns `true`.

**Root cause — `distribute_rewards` (L5808–5814):** [2](#0-1) 

`reward_start_timestamp_seconds = 0`, so `new_rounds_count = now / round_duration_seconds`. For a daily-round SNS (~86400 s/round) running since mainnet launch, this yields ~19,675 rounds instead of 1.

**Rewards purse loop (L5861–5871):** [3](#0-2) 

The loop accumulates interest for all ~19,675 rounds with no cap, producing a purse ~19,675× the expected single-round value.

**Insufficient initialization guard (L726–742):** [4](#0-3) 

The guard only fires when `proto.latest_reward_event.is_none()`. If stable memory contains `Some(RewardEvent { end_timestamp_seconds: None, ... })` — the exact state of any SNS deployed before proto tag 5 was introduced — the guard is skipped entirely and `end_timestamp_seconds` remains `None`.

**Proto field is `optional` (L1383):** [5](#0-4) 

Old serialized state deserializes with `end_timestamp_seconds = None` by proto3 semantics. The `round` field is explicitly marked `DEPRECATED` in favor of `end_timestamp_seconds` (L1336), confirming the field was added post-deployment. [6](#0-5) 

No migration code was found anywhere in `rs/sns/governance/src/` that back-fills `end_timestamp_seconds` when `latest_reward_event` is `Some` but the field is `None`.

**Note on PoC citations:** The integration tests cited at lines 1293–1301 and 1535–1543 both have `end_timestamp_seconds: Some(1)`, not `None`. [7](#0-6) [8](#0-7) 

These tests do not demonstrate the `None` case; a dedicated unit test with `end_timestamp_seconds: None` is required to reproduce the vulnerability.

## Impact Explanation

On the first `run_periodic_tasks()` call after an upgrade on an affected SNS:
1. `should_distribute_rewards()` returns `true` (elapsed time appears ~54 years).
2. `distribute_rewards()` computes a purse for ~19,675 rounds instead of 1.
3. All neurons that voted on any `ReadyToSettle` proposal receive ~19,675× the expected maturity.
4. Maturity is convertible to SNS governance tokens via `disburse_maturity`, constituting illegal minting and direct financial loss to all token holders of the affected SNS.

This matches the **High** impact class: "Significant SNS security impact with concrete user or protocol harm" — specifically unauthorized inflation of SNS token supply and dilution of all holders. The impact is bounded to individual SNS instances rather than the entire IC, constraining it below Critical.

## Likelihood Explanation

The precondition — `latest_reward_event` being `Some` with `end_timestamp_seconds: None` — is not attacker-controlled; it is a structural property of any SNS deployed before proto tag 5 was introduced. Once that precondition exists, no privileged access is required: the periodic task fires automatically on every heartbeat cycle, and any neuron holder who voted on a proposal receives the inflated maturity. The trigger is fully automatic and requires no attacker action beyond holding a neuron in an affected SNS.

## Recommendation

Extend the guard in `Governance::new()` to also handle the case where `latest_reward_event` is `Some` but `end_timestamp_seconds` is `None`:

```rust
if proto.latest_reward_event.is_none()
    || proto.latest_reward_event
           .as_ref()
           .map_or(true, |e| e.end_timestamp_seconds.is_none())
{
    // back-fill end_timestamp_seconds from actual_timestamp_seconds or now
    ...
}
```

Alternatively, replace `.unwrap_or_default()` in both `should_distribute_rewards()` and `distribute_rewards()` with `.unwrap_or(self.proto.genesis_timestamp_seconds)` so that a missing field cannot produce an epoch-0 start time.

## Proof of Concept

1. Construct a `GovernanceProto` with `latest_reward_event: Some(RewardEvent { end_timestamp_seconds: None, round: 1, actual_timestamp_seconds: T, ... })` where `T` is a past timestamp.
2. Set at least one proposal to `ReadyToSettle`.
3. Instantiate `Governance::new()` with this proto — the guard at L726 is skipped because `latest_reward_event` is `Some`.
4. Call `run_periodic_tasks()` (or directly call `should_distribute_rewards()` then `distribute_rewards()`).
5. Assert that `new_rounds_count = now / round_duration_seconds` (~19,675 for daily cadence) and that voting neurons receive ~19,675× the expected maturity increment.
6. This is reproducible as a deterministic unit test in `rs/sns/governance/src/governance.rs` using the existing mock environment, requiring no mainnet access.
