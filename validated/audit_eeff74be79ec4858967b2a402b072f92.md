### Title
EVM SSTORE Gas Refund Counter Not Reset Between Transactions — (`basic_system/src/system_implementation/caches/generic_pubdata_aware_plain_storage.rs`)

---

### Summary

The `evm_refunds_counter` in `GenericPubdataAwarePlainStorage` is never reset to zero after a transaction commits. `finish_frame_impl` only rolls the counter back on revert; on commit it is silently retained. As a result, SSTORE gas refunds accumulated during one transaction leak into every subsequent transaction in the same block, allowing an unprivileged sender to obtain inflated gas refunds they did not earn, causing the operator to receive less fee than owed.

---

### Finding Description

`compute_gas_refund` reads the refund counter to compute the EVM-level gas refund for the current transaction:

```rust
let evm_refund = {
    let full_refund_ergs = system.io.get_refund_counter().ergs();
    let full_refund_gas = full_refund_ergs.0.div_floor(ERGS_PER_GAS);
    let max_refund = gas_used / 5;
    core::cmp::min(full_refund_gas, max_refund)
};
gas_used -= evm_refund;
``` [1](#0-0) 

After reading the counter, the function does **not** reset it. The counter is a history-tracked value inside `GenericPubdataAwarePlainStorage`. Its rollback path is:

```rust
pub fn finish_frame_impl(
    &mut self,
    rollback_handle: Option<&StorageSnapshotId>,
) -> Result<(), InternalError> {
    if let Some(x) = rollback_handle {
        self.evm_refunds_counter.rollback(x.evm_refunds_counter);
        self.cache.rollback(x.cache)
    } else {
        Ok(())   // ← commit path: counter is NOT cleared
    }
}
``` [2](#0-1) 

When a transaction commits (`finish_global_frame(None)`), the counter retains whatever value it accumulated during that transaction. The next transaction in the same block therefore starts with a non-zero counter baseline. `compute_gas_refund` for that next transaction reads the stale counter and applies up to `gas_used / 5` of it as a free refund — refunds the second transaction never earned.

The counter is populated per SSTORE write via `apply_write_impl` → `refund_for_storage_write`: [3](#0-2) [4](#0-3) 

The `get_refund_counter_impl` accessor simply returns the current accumulated value without any per-transaction scoping: [5](#0-4) 

Both the ZK and Ethereum transaction flows call `compute_gas_refund` and rely on the counter being transaction-scoped: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**Impact: High.** An attacker who controls two consecutive transactions in the same block can:

1. In TX1, clear N previously-written storage slots. Each clear adds 4 800 gas to the refund counter (EIP-3529 logic). TX1 legitimately receives `min(N×4800, gas_used₁/5)` gas back.
2. The counter is **not** reset after TX1 commits.
3. In TX2 (any transaction), `compute_gas_refund` reads the counter — still showing N×4800 gas — and grants TX2 up to `gas_used₂/5` gas refund for free.

The operator receives `gas_used₂/5 × gas_price` less fee than owed. For a 1 000 000-gas transaction at 1 000 wei/gas this is up to 200 000 000 wei per exploited transaction. The attack is repeatable across every transaction pair in a block.

---

### Likelihood Explanation

**Likelihood: Medium.** Any unprivileged sender can submit two transactions to the same block. No privileged role, oracle manipulation, or external dependency is required. The attacker only needs the operator to include both transactions in the same block in order, which is the normal sequencing behaviour.

---

### Recommendation

Reset `evm_refunds_counter` to `R::empty()` at the start of each transaction. The cleanest place is inside `start_global_frame` (or equivalently at the top of the per-transaction processing loop), so that every new transaction begins with a zero refund counter regardless of what the previous transaction accumulated. Alternatively, explicitly zero the counter inside `compute_gas_refund` after reading it, before returning.

---

### Proof of Concept

```
Block N:
  TX1 (attacker):
    - Writes 0 to 100 previously-non-zero storage slots
    - Each SSTORE: refund_counter += 4800 gas  →  counter = 480 000 gas
    - compute_gas_refund: evm_refund = min(480000, gas_used₁/5)
      e.g. gas_used₁ = 600 000  →  evm_refund = 120 000 gas (legitimate)
    - finish_global_frame(None): counter NOT reset, stays at 480 000 gas

  TX2 (attacker, any calldata):
    - No SSTORE operations
    - compute_gas_refund reads counter = 480 000 gas (leftover from TX1)
    - evm_refund = min(480000, gas_used₂/5)
      e.g. gas_used₂ = 100 000  →  evm_refund = 20 000 gas (FREE, unearned)
    - Attacker pays 20 000 × gas_price less; operator receives that much less
```
