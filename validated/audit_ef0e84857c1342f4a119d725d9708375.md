Audit Report

## Title
Unauthorized Third-Party Can Bypass SNS Swap Confirmation Text and Force Participation for Arbitrary Buyers via `refresh_buyer_tokens` - (File: `rs/sns/swap/canister/canister.rs`)

## Summary
The `refresh_buyer_tokens` update endpoint in the SNS Swap canister accepts a caller-supplied `buyer` principal with no check that the caller equals the specified buyer. Because the SNS confirmation text is publicly readable from canister state, any unprivileged ingress sender can call `refresh_buyer_tokens` on behalf of any victim who has already transferred ICP to the swap subaccount, bypassing the confirmation-text gate and forcing the victim's participation to be committed and their open ticket to be deleted — without the victim's explicit agreement.

## Finding Description
In `rs/sns/swap/canister/canister.rs` (lines 130–134), the effective buyer principal is resolved as:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()
};
```

There is no assertion that `caller == p` when `arg.buyer` is non-empty. The resolved `p` is passed directly into `refresh_buyer_token_e8s`. Inside that function, `validate_confirmation_text` (swap.rs lines 363–384) only performs a string equality check between the caller-supplied text and the expected text stored in init state — it does not verify the identity of who is supplying the text. The confirmation text is set at SNS initialization and is publicly readable via `get_init`, so any caller can trivially obtain and supply it. After the ledger balance check, the function updates `self.buyers[p]` and unconditionally deletes `p`'s open ticket from `OPEN_TICKETS_MEMORY` (swap.rs lines 1248–1272) if the balance satisfies the ticket amount. There is no existing guard that prevents a third party from executing this full flow on behalf of an arbitrary victim.

## Impact Explanation
The confirmation text feature exists specifically to ensure that each participant explicitly agrees to terms (legal disclaimers, risk warnings) before their ICP is committed. Bypassing it constitutes a significant SNS security impact with concrete user harm: a victim's ICP is committed to the swap without their explicit agreement, their open ticket is deleted (removing the ability to cancel through the normal ticket-based flow), and if the swap commits, the victim receives SNS tokens they never consented to purchase. This maps to the High impact class: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
The only precondition is that the victim has transferred ICP to their swap subaccount — a normal, expected step in the participation flow. The confirmation text is publicly readable with no privileged access required. The victim's principal is observable on-chain. No key material, network-level attack, or social engineering is needed. Any unprivileged ingress sender can execute this against any in-progress buyer at negligible cost and can repeat it for every active participant in the swap.

## Recommendation
Add a caller-identity check inside `refresh_buyer_tokens` when a non-empty `buyer` is specified and a confirmation text is required:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let specified = PrincipalId::from_str(&arg.buyer).unwrap();
    if swap().init_or_panic().confirmation_text.is_some()
        && specified != caller_principal_id()
    {
        panic!("Caller must match buyer when a confirmation text is required");
    }
    specified
};
```

Alternatively, restrict the `buyer` override to the buyer themselves unconditionally, matching the security model of the confirmation-text feature.

## Proof of Concept
1. Deploy an SNS with `confirmation_text = "I accept the risk"` (readable via `get_init`).
2. Victim calls `new_sale_ticket(amount_icp_e8s = X)` — a ticket is created in `OPEN_TICKETS_MEMORY`.
3. Victim transfers `X` ICP to `swap_canister_subaccount(victim_principal)` on the ICP ledger, intending to review the confirmation text before committing.
4. Attacker (any principal) calls:
   ```
   refresh_buyer_tokens({
     buyer: "<victim_principal>",
     confirmation_text: Some("I accept the risk")
   })
   ```
5. `refresh_buyer_token_e8s` reads the victim's ICP balance, validates the attacker-supplied confirmation text (passes, since the text matches), registers the victim's participation in `self.buyers`, and deletes the victim's open ticket.
6. The victim's ICP is now committed to the swap without their explicit agreement. If the swap commits, the victim receives SNS tokens without consent.

Reproducible as a unit test in `rs/sns/swap/tests/swap.rs` by calling `refresh_buyer_token_e8s` with a buyer principal distinct from the caller, with `confirmation_text` set in init, and verifying that participation is registered and the ticket is deleted — mirroring the pattern already established in `test_swap_participation_confirmation`.
