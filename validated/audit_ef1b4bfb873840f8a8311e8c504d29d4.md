### Title
Missing Zero-Address Check in `receive_base_tokens` Allows Permanent Freezing of Bridged ETH — (File: `engine/src/engine.rs`)

---

### Summary

The `receive_base_tokens` function in `engine/src/engine.rs` mints bridged ETH (base tokens) to a user-supplied recipient address without validating that the recipient is not the zero address. A user who specifies `0x0000000000000000000000000000000000000000` as the recipient when bridging ETH from NEAR to Aurora will have their ETH permanently frozen at address(0), while Aurora's internal ETH balance accounting is inflated — creating an insolvency discrepancy between tracked supply and accessible supply.

---

### Finding Description

The `receive_base_tokens` function is the handler for bridging ETH from NEAR into Aurora. It parses the recipient EVM address from the user-controlled `args.msg` field and directly credits the balance: [1](#0-0) 

The recipient is extracted from `FtTransferMessageData::try_from(args.msg.as_str())` and immediately used in `set_balance` with no check that `recipient != Address::zero()`. [2](#0-1) 

By contrast, the ERC-20 token minting path (`receive_erc20_tokens`) delegates to the `EvmErc20` / `EvmErc20V2` Solidity contracts: [3](#0-2) 

Those contracts call OpenZeppelin's `_mint`, which **does** include a zero-address guard. The base-token path has no equivalent protection.

---

### Impact Explanation

- ETH credited to `address(0)` is permanently inaccessible — no private key controls it, and no EVM mechanism can recover it.
- The ETH was already locked on the NEAR side when `ft_on_transfer` was triggered, so it cannot be refunded.
- Aurora's internal ETH balance sum (across all accounts including address(0)) exceeds the sum of all reachable balances, creating a permanent insolvency: the connector holds more ETH than users can ever withdraw.
- **Impact: Permanent freezing of funds / insolvency.**

---

### Likelihood Explanation

The `msg` field of `ft_on_transfer` is entirely user-controlled. Any NEAR account that holds wETH or bridged ETH can trigger this path by specifying the zero address as the recipient. No special privileges are required. The scenario is reachable by any unprivileged user, either accidentally (mistyped address) or deliberately (self-burn that inflates supply accounting).

---

### Recommendation

Add a zero-address guard in `receive_base_tokens` immediately after parsing the recipient, analogous to OpenZeppelin's check in `_mint`:

```rust
if recipient == Address::zero() {
    return Err(errors::ERR_ZERO_ADDRESS_RECIPIENT.into());
}
```

This mirrors the protection already present in the ERC-20 path via OpenZeppelin and closes the gap for base-token minting.

---

### Proof of Concept

1. On NEAR, call `ft_transfer_call` on the ETH connector, setting `msg` to `"0000000000000000000000000000000000000000"` (40 hex chars of zero).
2. The connector invokes `ft_on_transfer` on Aurora with `args.msg = "0000000000000000000000000000000000000000"`.
3. `receive_base_tokens` parses `recipient = Address::zero()`.
4. `set_balance(&mut self.io, &Address::zero(), &new_balance)` executes without error.
5. The ETH is credited to address(0) on Aurora — permanently inaccessible.
6. The NEAR-side ETH is already locked; no refund path exists.
7. Aurora's total tracked ETH supply is now greater than the sum of all reachable balances. [4](#0-3)

```rust
    pub fn receive_base_tokens(
        &mut self,
        args: &FtOnTransferArgs,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
        let amount = Wei::new_u128(args.amount.as_u128());
        let receipient = message_data.recipient;
        let balance = get_balance(&self.io, &receipient);
        let new_balance = balance
            .checked_add(amount)
            .ok_or(errors::ERR_BALANCE_OVERFLOW)?;

        set_balance(&mut self.io, &receipient, &new_balance);

        sdk::log!("Mint {amount} base tokens for: {}", receipient.encode());

        Ok(None)
    }
```
