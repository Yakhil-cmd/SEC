### Title
Unvalidated `ft_on_transfer` Caller Allows Minting ERC-20 Tokens Without Actual NEP-141 Deposit — (`engine/src/contract_methods/connector.rs`)

### Summary
The `ft_on_transfer` entry point in Aurora Engine accepts and acts on attacker-controlled `amount` values without verifying that the calling NEAR account is a legitimate NEP-141 token contract that has actually transferred tokens to Aurora. Any NEAR account that controls a registered NEP-141 token can call `ft_on_transfer` directly with a fabricated `amount`, causing Aurora to mint ERC-20 tokens without a corresponding deposit, enabling theft of other users' deposited tokens.

### Finding Description

`ft_on_transfer` is a public NEAR contract method. In the NEP-141 standard it is intended to be called only by a token contract as the second step of `ft_transfer_call`, after the token contract has already moved tokens into Aurora's custody. Aurora's implementation performs no such verification:

```rust
// engine/src/contract_methods/connector.rs, lines 62–109
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();   // ← attacker-controlled
        ...
        let args: FtOnTransferArgs = read_json_args(&io)?;           // ← amount is attacker-controlled
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(                             // ← mints ERC-20 tokens
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };
        ...
    })
}
```

The only guard is whether `predecessor_account_id` equals the stored ETH connector account. For any other caller, `receive_erc20_tokens` is invoked with the caller's account treated as the NEP-141 token contract and `args.amount` (fully attacker-controlled JSON) as the deposit amount. No Merkle proof, no cross-check against Aurora's actual token balance change, and no verification that a real `ft_transfer_call` initiated the call chain.

The `FtOnTransferArgs` struct confirms the amount is a plain caller-supplied field:

```rust
// engine-types/src/parameters/connector.rs, lines 194–199
pub struct FtOnTransferArgs {
    pub sender_id: AccountId,
    pub amount: Balance,   // ← no on-chain proof this was actually transferred
    pub msg: String,
}
```

Additionally, `deploy_erc20_token` carries no owner-only restriction, so any NEAR account can register an arbitrary NEP-141 token in Aurora:

```rust
// engine/src/contract_methods/connector.rs, lines 112–159
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(...) {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        // No require_owner_only check
        ...
    })
}
```

### Impact Explanation

**Critical — Direct theft of user funds.**

An attacker who controls a NEP-141 token contract registered in Aurora can call `ft_on_transfer` directly with an arbitrarily large `amount`. Aurora mints the corresponding ERC-20 tokens to the attacker's EVM address. The attacker then calls the exit precompile (`ExitToNear`) or the ERC-20 `withdraw` path to redeem those ERC-20 tokens for real NEP-141 tokens held by Aurora on behalf of legitimate depositors. Every NEP-141 token of that type deposited by honest users is drainable in a single transaction.

### Likelihood Explanation

**Medium.** The attacker must:
1. Control a NEAR account that is registered as a NEP-141 token in Aurora (trivially achievable via the open `deploy_erc20_token` call).
2. Wait for or socially engineer legitimate users to deposit that token into Aurora.

Step 1 requires no special privilege. Step 2 is the limiting factor, but any token project that integrates with Aurora satisfies it automatically. The attack requires no leaked keys, no governance capture, and no admin compromise — only a direct NEAR function call.

### Recommendation

1. **Validate the call chain**: Inside `ft_on_transfer`, verify that the call originates from a cross-contract callback initiated by a legitimate `ft_transfer_call`. NEAR's `env::predecessor_account_id` alone is insufficient; the function should also assert it is being called as a promise callback (e.g., via `env.assert_private_call()` or an equivalent promise-result check) so that direct external invocations are rejected.
2. **Cross-check the actual balance change**: Before minting ERC-20 tokens, query Aurora's NEP-141 balance on the token contract and compare it to the pre-call balance stored in state, accepting only the real delta as the mintable amount.
3. **Restrict `deploy_erc20_token`**: Add an allowlist or owner-only guard so that only vetted NEP-141 tokens can be registered, reducing the attack surface for step 1.

### Proof of Concept

1. Attacker deploys a NEAR contract `attacker-token.near` implementing the NEP-141 interface.
2. Attacker calls `aurora.deploy_erc20_token(attacker-token.near)` — succeeds with no access check.
3. Legitimate users call `attacker-token.near.ft_transfer_call(receiver_id: "aurora", amount: 1000, msg: "")`, depositing 1 000 tokens into Aurora. Aurora now holds 1 000 `attacker-token.near` tokens.
4. Attacker calls `aurora.ft_on_transfer` **directly** (not via `ft_transfer_call`) with JSON args `{"sender_id": "attacker.near", "amount": "1000000000", "msg": ""}`. `predecessor_account_id` is `attacker-token.near` (the attacker's contract).
5. Aurora's `receive_erc20_tokens` looks up the ERC-20 for `attacker-token.near`, finds it, and mints 1 000 000 000 ERC-20 tokens to the attacker's EVM address — no actual transfer occurred.
6. Attacker calls the `ExitToNear` precompile to redeem ERC-20 tokens for the 1 000 real `attacker-token.near` tokens held by Aurora, draining all legitimate depositors. [1](#0-0) [2](#0-1) [3](#0-2)

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })
}
