Audit Report

## Title
Unbounded Recursive CBOR Deserialization Causes Stack Overflow and Process Crash in `/_/catch_up_package` Handler â€” (`rs/http_endpoints/public/src/catch_up_package.rs`)

## Summary

The `cup` handler at line 71 of `rs/http_endpoints/public/src/catch_up_package.rs` calls `serde_cbor::from_slice::<CatchUpPackageParam>(&body)` synchronously on a Tokio worker thread with no recursion depth guard. `CatchUpPackageParam` lacks `#[serde(deny_unknown_fields)]`, so unknown CBOR map keys trigger recursive `IgnoredAny` deserialization. A single unauthenticated 5 MB request containing maximally nested CBOR maps exhausts the worker thread's stack, causing a SIGSEGV that terminates the replica process.

## Finding Description

**Confirmed code path:**

1. `RequestBodyLimitLayer::new(config.max_request_size_bytes as usize)` admits up to 5 MB (`max_request_size_bytes: 5 * 1024 * 1024`). [1](#0-0) [2](#0-1) 

2. `verify_cbor_content_header` middleware passes for `Content-Type: application/cbor`. [3](#0-2) 

3. The `cup` async handler receives the raw `Bytes` and calls `serde_cbor::from_slice::<CatchUpPackageParam>(&body)` **synchronously** on the Tokio worker thread â€” no `spawn_blocking`, no yield point. [4](#0-3) 

4. `CatchUpPackageParam` is a two-field flat struct with **no** `#[serde(deny_unknown_fields)]`. Unknown keys are silently skipped via `serde::de::IgnoredAny`. [5](#0-4) 

5. `serde_cbor` uses recursive-descent parsing. For each unknown key whose value is a nested CBOR map, the deserializer calls `deserialize_any â†’ visit_map â†’ next_value::<IgnoredAny> â†’ deserialize_any â†’ â€¦`, one Rust call-stack frame per nesting level. At â‰ˆ3 bytes per level (`0xA1 0x61 0x61` = map(1) with 1-byte key), a 5 MB payload yields â‰ˆ1.7 M nesting levels. Even at a conservative 50 bytes per frame, this requires â‰ˆ85 MB of stack â€” far exceeding the 8 MB OS default for Tokio worker threads.

**Why existing mitigations fail:**

- The `timeout(Duration::from_secs(config.request_timeout_seconds))` tower layer wraps the async future but cannot preempt a synchronous blocking call occupying the thread. The stack overflow occurs before any async yield point. [6](#0-5) 

- `max_catch_up_package_concurrent_requests: 100` limits simultaneous in-flight requests but does not prevent a single request from overflowing the stack. [7](#0-6) 

- No authentication is required on this endpoint; the route is publicly reachable. [8](#0-7) 

## Impact Explanation

A stack overflow on a Tokio worker thread triggers Rust's stack overflow handler, which calls `abort()`, terminating the entire replica process with SIGSEGV. The orchestrator will restart the replica, but the attacker can immediately re-send the same payload, keeping the replica in a crash-restart loop. Targeting multiple replicas in a subnet simultaneously can reduce the number of live replicas below the consensus fault threshold, halting the subnet.

This matches the **High** impact category: *Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS* ($2,000â€“$10,000). It does not reach Critical because it is a DoS rather than consensus integrity compromise or RCE.

## Likelihood Explanation

The endpoint is unauthenticated and publicly reachable on port 8080. The payload is trivial to construct (a CBOR one-element map nested inside itself, repeated to fill 5 MB). No special privileges, cryptographic keys, network position, or victim interaction are required. The attack is repeatable with a single HTTP request and can be automated.

## Recommendation

1. **Add `#[serde(deny_unknown_fields)]` to `CatchUpPackageParam`** (`rs/types/types/src/consensus/catchup.rs`, line 318). This causes `serde_cbor` to return an error immediately on the first unknown key rather than recursively skipping its value, eliminating the recursion path entirely. [5](#0-4) 

2. **Add a CBOR recursion depth pre-scan** before passing the body to `serde_cbor`. Reject any payload whose CBOR nesting depth exceeds a small fixed limit (e.g., 10).

3. **Move deserialization into `spawn_blocking`** to isolate the blocking call from the async executor, preventing starvation of other requests (though this alone does not prevent the crash).

## Proof of Concept

# Build a 5 MB deeply nested CBOR one-element map:
# 0xA1 = map(1), 0x61 0x61 = text("a"), value = next level

    level = b'\xa1\x61\x61' + level  # map(1) {"a": <level>}

    "http://<replica-ip>:8080/_/catch_up_package",
    data=level[:5 * 1024 * 1024],
    headers={"Content-Type": "application/cbor"},
    timeout=30,
)

# orchestrator restarts it; repeat to maintain crash-restart loop.
