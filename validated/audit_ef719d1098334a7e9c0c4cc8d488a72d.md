Audit Report

## Title
Unbounded `ingress_messages` Deserialization in `TryFrom<pb::StrippedBlockProposal>` Enables Byzantine Peer to Exhaust Replica Memory and Saturate Tokio Task Scheduler — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs`)

## Summary
`TryFrom<pb::StrippedBlockProposal>` performs no upper-bound check on the count of `ingress_messages` entries before collecting them into a `Vec`. A Byzantine peer below the consensus fault threshold can serve a crafted `StrippedBlockProposal` with an arbitrarily large `ingress_messages` list under a valid, currently-wanted block proposal hash, causing `BlockProposalAssembler::new` to allocate an unbounded `Vec` and `assemble_message` to spawn one indefinitely-looping Tokio task per entry — exhausting replica memory and saturating the task scheduler.

## Finding Description

**Root cause — no count bound in deserialization.**
`TryFrom<pb::StrippedBlockProposal>` at `stripped.rs` lines 100–108 validates that `pruned_block_proposal` is present, that its ingress payload is empty, and that `unstripped_consensus_message_id` is a `BlockProposal` hash. It does **not** check `ingress_messages.len()` before collecting:

```rust
stripped_ingress_payload: StrippedIngressPayload {
    ingress_messages: value
        .ingress_messages
        .into_iter()
        .map(SignedIngressId::try_from)
        .collect::<Result<Vec<_>, _>>()?,
},
``` [1](#0-0) 

The same absence of a bound applies to `stripped_idkg_dealings`. [2](#0-1) 

**Proportional allocation in `BlockProposalAssembler::new`.**
Every deserialized `SignedIngressId` is mapped into a `Vec<(SignedIngressId, Option<SignedIngress>)>` with no cap. The comment at line 440–441 says "We can have at most 1000 elements" — this is an unenforced assumption, not a runtime check. [3](#0-2) [4](#0-3) 

**One Tokio task spawned per entry.**
`assemble_message` calls `assembler.missing_stripped_messages()` (which returns all N entries since none are in the local pool for fake IDs) and spawns a `get_or_fetch` task for each without any concurrency cap: [5](#0-4) 

**Each spawned task loops indefinitely.**
`download_stripped_message` uses `with_max_elapsed_time(None)`, meaning the exponential backoff has no maximum elapsed time. Each of the N tasks loops forever until the `JoinSet` is dropped: [6](#0-5) [7](#0-6) 

**ID check does not constrain content.**
`MaybeStrippedConsensusMessage::id()` derives the artifact ID solely from `unstripped_consensus_message_id`, ignoring `ingress_messages` entirely. The `FetchArtifact::download_artifact` check `message.id() == id` therefore passes for any crafted payload that carries a valid block proposal hash in that field: [8](#0-7) [9](#0-8) 

**Bouncer does not inspect content.**
`BouncerFactoryWrapper::new_bouncer` delegates to the consensus pool bouncer, which evaluates only the `ConsensusMessageId` (height + hash). It has no visibility into the number of ingress message IDs embedded in the stripped payload: [10](#0-9) 

**Exploit flow:**
1. Attacker is a subnet peer (below Byzantine fault threshold) and observes a `StrippedConsensusMessageId` the victim is currently downloading.
2. Attacker crafts a `pb::StrippedBlockProposal` with a valid `unstripped_consensus_message_id` (the wanted block proposal hash) and ~62,500 fake `ingress_messages` entries (each ~64 bytes, fitting within a 4 MB QUIC message).
3. Attacker serves this proto in response to the victim's RPC.
4. `TryFrom` deserialization succeeds; `message.id() == id` passes; bouncer returns `Wanted`.
5. `BlockProposalAssembler::new` allocates a `Vec` of 62,500 entries.
6. `assemble_message` spawns 62,500 `get_or_fetch` tasks, each looping indefinitely.
7. With `LOOK_AHEAD = 10` concurrent block proposals, total task count reaches ~625,000.
8. The `JoinSet` is only dropped when the bouncer fires `Unwanted` — but by then all tasks are already consuming memory and scheduler slots. The attacker repeats for each new block proposal.

## Impact Explanation
A single below-threshold Byzantine peer can trigger allocation of a `Vec` with up to ~62,500 entries and spawn an equal number of indefinitely-looping Tokio tasks per targeted block proposal. With up to 10 concurrent proposals in flight, this can exhaust replica memory and flood the Tokio thread pool, starving legitimate consensus work and causing the replica to crash or become unresponsive. This matches the **High** impact category: *Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS* ($2,000–$10,000).

## Likelihood Explanation
The attacker needs only to be a subnet peer (below the Byzantine fault threshold) and to observe a block proposal ID the victim is currently downloading — both are trivially achievable in normal subnet operation. No privileged access, key material, or majority corruption is required. The crafted message passes all existing validation gates (`TryFrom`, `message.id() == id`, bouncer) before any resource allocation occurs. The attack is repeatable for every new block proposal.

## Recommendation
Add an upper-bound check on `ingress_messages.len()` (and `stripped_idkg_dealings.len()`) inside `TryFrom<pb::StrippedBlockProposal>`, before the `collect()` call. The bound should be the protocol-enforced maximum: `max_ingress_messages_per_block` (registry parameter, default 1000) for ingress, and the subnet's maximum IDKG dealings per block for dealings. Return `ProxyDecodeError` if either count exceeds its respective limit. This enforces the invariant already assumed by the comment at `assembler.rs` line 440.
