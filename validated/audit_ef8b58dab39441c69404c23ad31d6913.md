### Title
Treasury Assets Permanently Locked: `validate_execute_extension_operation` Re-validates WASM Hash Against Empty `ALLOWED_EXTENSIONS`, Blocking All Withdraw/Deposit Proposals — (`rs/sns/governance/src/extensions.rs`)

---

### Summary

`ALLOWED_EXTENSIONS` is intentionally empty in production as of April 6, 2026. `get_extension_spec_and_update_cache`, called unconditionally from `validate_execute_extension_operation`, always re-validates the installed WASM hash against this empty map. There is no fallback to the stable-memory cache. Any SNS that had a KongSwap extension registered and treasury assets deposited into it can never pass proposal validation for a withdraw (or deposit) operation. The assets are permanently locked with no governance recovery path.

---

### Finding Description

**`ALLOWED_EXTENSIONS` is empty in production:** [1](#0-0) 

In non-test builds, `validate_extension_wasm` reads from this empty map: [2](#0-1) 

`get_extension_spec_and_update_cache` always calls `validate_extension_wasm` against the live WASM hash fetched from `canister_module_hash`. It does **not** fall back to the stable-memory cache. The cache is only written on success; on failure the error is returned directly: [3](#0-2) 

`validate_execute_extension_operation` calls `get_extension_spec_and_update_cache` unconditionally and propagates any error immediately, before any execution logic is reached: [4](#0-3) 

**Correction to the question's framing:** The proposal does not "pass validation but fail at execution time." It fails at **validation time** — `validate_execute_extension_operation` returns `Err(InvalidProposal)` before execution is ever attempted. The locked-asset outcome is the same, but the failure point is earlier.

---

### Impact Explanation

Any SNS that:
1. Registered a KongSwap extension before April 6, 2026, and
2. Deposited treasury assets (SNS tokens or ICP) into that extension canister

…now has those assets permanently locked. Every `ExecuteExtensionOperation` proposal (withdraw or deposit) will be rejected at validation with `"Wasm module with hash X is not allowed as an extension."` There is no alternative governance proposal type that can call `withdraw` on the extension canister, and no bypass in the validation path. The impact is **ledger loss** — treasury assets are irrecoverable through any on-chain governance mechanism.

---

### Likelihood Explanation

The precondition is concrete and already satisfied on mainnet for any SNS that used KongSwap. The governance canister upgrade that cleared `ALLOWED_EXTENSIONS` has already been deployed. No attacker action is required; the condition is triggered by any legitimate governance participant submitting a withdraw proposal.

---

### Recommendation

`get_extension_spec_and_update_cache` should fall back to the stable-memory cache when `validate_extension_wasm` fails, rather than propagating the error. Alternatively, a separate `UnregisterExtension` or `EmergencyWithdraw` proposal type should be introduced that bypasses the WASM-hash check for already-registered extensions, allowing asset recovery even after the allowed list changes. The invariant to enforce: **a registered extension with deposited assets must always have a governance-accessible withdrawal path, independent of the current state of `ALLOWED_EXTENSIONS`.**

---

### Proof of Concept

State-machine test outline:
1. Start an SNS with a KongSwap extension registered (WASM hash in stable memory cache).
2. Clear `ALLOWED_EXTENSIONS` (simulating the April 6, 2026 production state).
3. Submit an `ExecuteExtensionOperation` proposal with `operation_name = "withdraw"`.
4. Assert that `validate_execute_extension_operation` returns `Err(InvalidProposal, "Wasm module with hash … is not allowed as an extension.")`.
5. Assert that no withdraw call is ever made to the extension canister.
6. Assert that the treasury balance in the extension canister is unchanged (assets locked).
