### Title
NEP-141 Rebasing Token Bridge Accounting Invariant Broken: 1:1 ERC-20 Mirror Supply vs. Escrowed NEP-141 Balance - (`engine/src/engine.rs`, `engine-precompiles/src/native.rs`)

---

### Summary

Aurora Engine's NEP-141 ↔ ERC-20 bridge assumes a permanent 1:1 relationship between the ERC-20 tokens minted on Aurora and the NEP-141 tokens held (escrowed) by the Aurora Engine contract on NEAR. This invariant is violated by rebasing NEP-141 tokens, whose balances change outside of explicit transfers. A negative rebase causes withdrawal failures (temporary or permanent fund freeze depending on the `error_refund` feature flag), while a positive rebase permanently traps the accrued yield inside the Aurora Engine account with no recovery path.

---

### Finding Description

**Deposit path** (`receive_erc20_tokens`, `engine/src/engine.rs:796–844`):

When a user sends `N` NEP-141 tokens to the Aurora Engine account via `ft_transfer`, the `ft_on_transfer` callback fires and calls `receive_erc20_tokens`. The function reads `amount = args.amount.as_u128()` — the exact transfer amount — and mints exactly that many ERC-20 tokens to the recipient:

```rust
let amount = args.amount.as_u128();
// ...
setup_receive_erc20_tokens_input(&recipient, amount)  // mints `amount` ERC-20
```

**Withdrawal path** (`exit_erc20_token_to_near`, `engine-precompiles/src/native.rs:558–656`):

When a user calls `withdrawToNear(recipient, amount)` on the `EvmErc20` / `EvmErc20V2` contract, the ERC-20 burns exactly `amount` tokens and calls the `ExitToNear` precompile. The precompile constructs an `ft_transfer` promise requesting exactly `exit_params.amount` NEP-141 tokens from Aurora Engine's NEAR account:

```rust
format!(
    r#"{{"receiver_id":"{}","amount":"{}"}}"#,
    exit_params.receiver_account_id,
    exit_params.amount.as_u128()   // ← burned ERC-20 amount, not actual NEP-141 balance
)
```

**The broken invariant:**

The system assumes:

```
ERC-20 total supply on Aurora == NEP-141 balance held by Aurora Engine on NEAR
```

For a rebasing NEP-141 token, the Aurora Engine's NEP-141 balance changes autonomously between deposit and withdrawal, silently diverging from the ERC-20 supply.

---

### Impact Explanation

**Scenario 1 — Negative rebase (balance decreases):**

- User deposits 1000 NEP-141 → Aurora Engine holds 1000 NEP-141, user holds 1000 ERC-20.
- Rebase occurs: Aurora Engine's NEP-141 balance drops to 900.
- User burns 1000 ERC-20 and requests 1000 NEP-141 → `ft_transfer` fails (only 900 available).
- **Without `error_refund` feature**: ERC-20 tokens are permanently burned, NEP-141 never received → **Critical: permanent loss of user funds**.
- **With `error_refund` feature**: ERC-20 tokens are refunded via `exit_to_near_precompile_callback`, but the user can never successfully withdraw while the deficit persists → **High: temporary/permanent freeze of funds**.

**Scenario 2 — Positive rebase (balance increases):**

- User deposits 1000 NEP-141 → Aurora Engine holds 1000 NEP-141, user holds 1000 ERC-20.
- Rebase occurs: Aurora Engine's NEP-141 balance grows to 1100.
- User burns 1000 ERC-20 and receives 1000 NEP-141 → succeeds, but 100 NEP-141 remain permanently trapped in Aurora Engine's account.
- There is no administrative function to sweep or redistribute these excess NEP-141 tokens.
- **High: permanent freeze of accrued yield / theft of unclaimed yield**.

---

### Likelihood Explanation

Any NEP-141 token whose balance can change outside of explicit `ft_transfer` calls (rebasing, airdrop-style distributions, fee-on-transfer with redistribution) triggers this bug the moment it is bridged via `deploy_erc20_token` + `ft_on_transfer`. The entry path is fully unprivileged: any token holder can initiate a deposit via the standard NEP-141 `ft_transfer` to the Aurora Engine account, and any ERC-20 holder can initiate a withdrawal via `withdrawToNear`. No admin action is required to reach the vulnerable state.

---

### Recommendation

1. **Disallow rebasing NEP-141 tokens** from being registered via `deploy_erc20_token` by documenting and enforcing this restriction.
2. **Alternatively**, track the actual NEP-141 balance held by Aurora Engine per token (not the minted ERC-20 supply) and use that as the withdrawal cap, issuing proportional shares rather than absolute amounts — analogous to a vault share model.
3. **For positive rebase surplus**, add an administrative sweep function that can redistribute or burn excess NEP-141 tokens that exceed the outstanding ERC-20 supply.
