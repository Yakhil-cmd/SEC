### Title
EIP-7702 Authorization List Applied Unconditionally in Simulation Mode, Enabling Persistent Delegation State Manipulation - (File: `basic_bootloader/src/bootloader/transaction/authorization_list.rs`)

---

### Summary

In ZKsync OS, the `BasicBootloaderCallSimulationConfig` (used for `eth_call` / `eth_estimateGas`) skips the originator's nonce check and signature validation, but **does not skip EIP-7702 authorization list processing**. The function `parse_authorization_list_and_apply_delegations` is called unconditionally regardless of the `Config::SIMULATION` flag, meaning a simulation call can permanently write delegation state (`set_delegation`) and bump authority nonces on-chain — state changes that are supposed to be ephemeral in a simulation context.

---

### Finding Description

The ZK validation path (`validation_impl.rs` for the ZK flow) guards the originator nonce check behind `if !Config::SIMULATION`:

```rust
if !Config::SIMULATION {
    // Nonce validation - skipped for service transactions
    ...
    require!(old_nonce == originator_expected_nonce, err, system)?;
}
```

However, the authorization list processing block that follows has **no such guard**:

```rust
#[cfg(feature = "eip-7702")]
{
    if let Some(authorization_list) = transaction.authorization_list() {
        intrinsic_resources.with_infinite_ergs(|inf_resources| {
            crate::bootloader::transaction::authorization_list::parse_authorization_list_and_apply_delegations(
                system,
                inf_resources,
                authorization_list,
            )
        })?;
    }
}
```

Inside `parse_authorization_list_and_apply_delegations`, each valid authorization entry causes two permanent state mutations:

1. **`system.io.set_delegation(...)`** — writes the `0xef0100 || address` delegation bytecode to the authority account.
2. **`system.io.increment_nonce(...)`** — bumps the authority's nonce by 1.

These are real storage writes, not read-only operations. The nonce check inside `validate_and_apply_delegation` (step 6: `account_properties.nonce.0 != auth_nonce`) uses the **live on-chain nonce**, not a simulated one, so a valid pre-signed authorization tuple will pass and apply.

The `simulate_tx` entry point in `forward_system/src/run/mod.rs` explicitly documents that it skips "signature check, nonce check and EIP-3607 check" — but EIP-7702 delegation application is not listed and is not skipped.

---

### Impact Explanation

An unprivileged attacker who has obtained (or can construct) a valid EIP-7702 authorization signature from a victim EOA can:

1. Submit an `eth_call` / `eth_estimateGas` simulation containing an EIP-7702 type-4 transaction with the victim's signed authorization tuple.
2. The simulation applies `set_delegation` and increments the victim's nonce as a **permanent state change**, even though the transaction is never included in a block.
3. The victim's account is now delegated to an attacker-controlled contract without the victim ever submitting a real transaction.
4. The victim's nonce is incremented, which can invalidate pending legitimate transactions.

This is a direct analog to the reported MEV/unordered-nonce issue: signed authorization tuples (like signed permits) can be replayed in an unexpected context (simulation) to produce state changes the signer never intended, and the ordering/context of execution is attacker-controlled.

**Concrete impact:**
- Unauthorized EIP-7702 delegation of a victim EOA to a malicious contract, enabling the attacker to execute arbitrary code in the victim's context.
- Nonce increment invalidates victim's pending transactions (griefing / forced replay failure).
- Permanent state corruption from what should be a read-only simulation call.

---

### Likelihood Explanation

- EIP-7702 authorization signatures are broadcast publicly (e.g., in mempools, off-chain coordination channels).
- The `simulate_tx` / `eth_call` path is a standard, publicly accessible RPC endpoint — no privileged access required.
- The attacker only needs a valid signed authorization tuple, which is the normal precondition for any EIP-7702 use.
- The `eip-7702` feature flag must be enabled, but it is present and tested in the codebase.

---

### Recommendation

Guard the authorization list application with the same `!Config::SIMULATION` check used for nonce validation:

```rust
#[cfg(feature = "eip-7702")]
{
    if !Config::SIMULATION {
        if let Some(authorization_list) = transaction.authorization_list() {
            intrinsic_resources.with_infinite_ergs(|inf_resources| {
                crate::bootloader::transaction::authorization_list::parse_authorization_list_and_apply_delegations(
                    system,
                    inf_resources,
                    authorization_list,
                )
            })?;
        }
    }
}
```

Apply the same fix to the Ethereum validation path in `basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs` at the `parse_authorization_list_and_apply_delegations` call site (line 347–353), which also lacks a simulation guard.

---

### Proof of Concept

1. Victim EOA `V` signs an EIP-7702 authorization: `Authorization { chain_id: X, address: attacker_contract, nonce: V.nonce }`.
2. Attacker calls `eth_call` (or `eth_estimateGas`) with a type-4 EIP-7702 transaction containing this authorization tuple. The transaction sender can be any address (signature check is skipped in simulation).
3. ZKsync OS processes the call via `simulate_tx` → `BasicBootloaderCallSimulationConfig` → `validation_impl.rs`.
4. The originator nonce check is skipped (`Config::SIMULATION == true`), but `parse_authorization_list_and_apply_delegations` is called unconditionally.
5. `validate_and_apply_delegation` passes all checks (valid sig, nonce matches live state), calls `set_delegation(V, attacker_contract)` and `increment_nonce(V, 1)`.
6. After the simulation returns, `V`'s account has `is_delegated = true` pointing to `attacker_contract`, and `V.nonce` has been incremented — permanent state changes from a read-only call.

```rust
pub fn parse_authorization_list_and_apply_delegations<S: EthereumLikeTypes>(
    system: &mut System<S>,
    resources: &mut S::Resources,
    auth_list: AuthorizationList<'_>,
) -> Result<(), TxError>
where
    S::IO: IOSubsystemExt,
{
    use crate::bootloader::transaction::rlp_encoded::AuthorizationEntry;
    let mut hasher = crypto::sha3::Keccak256::new();

    for entry in auth_list.iter() {
        let AuthorizationEntry {
            chain_id,
            address,
            nonce,
            y_parity,
            r,
            s,
        } = entry;
        let success = validate_and_apply_delegation(
            system,
            resources,
            &chain_id,
            nonce,
            address,
            (y_parity, r, s),
            &mut hasher,
        )?;
        system_log!(system, "Delegation success: {success}\n");

        if !success {}
    }
    Ok(())
}
```
