### Title
`L2AssetTracker` Assumed Deployed at Hardcoded Address Without Existence Check — Silent Accounting Divergence on New Networks - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

### Summary

The bootloader's `notify_l2_asset_tracker` function unconditionally calls `handleFinalizeBaseTokenBridgingOnL2` on the hardcoded `L2_ASSET_TRACKER_ADDRESS` (`0x1000f`) without verifying that a contract is actually deployed there. If `L2AssetTracker` is absent (new network, pre-upgrade state), the call silently succeeds as a call to an empty account, tokens are minted from treasury, but the asset tracker's deposit accounting is never updated — permanently diverging the chain's interop state.

### Finding Description

In `notify_l2_asset_tracker`, the bootloader calls `handleFinalizeBaseTokenBridgingOnL2(uint256,uint256)` on `L2_ASSET_TRACKER_ADDRESS` for every L1 deposit transaction with `amount > 0`. The code itself documents the assumption:

> "If no contract is deployed at L2AssetTracker, the call succeeds silently (a call to an empty address returns success with no returndata in EVM). However, we are certain that L2AssetTracker is available after the upgrade." [1](#0-0) 

The call is made to `L2_ASSET_TRACKER_ADDRESS` (`0x1000f`) with no bytecode-existence check: [2](#0-1) 

The address constant is hardcoded: [3](#0-2) 

Additionally, `read_l1_chain_id` reads `L1_CHAIN_ID` from storage slot 154 of `L2_ASSET_TRACKER_ADDRESS`. If the contract is not deployed, this returns `U256::ZERO`, so `notify_l2_asset_tracker` is called with `_fromChainId = 0`: [4](#0-3) 

The `failed` check only catches an explicit EVM revert. A call to an empty account returns `CallResult::Successful` with empty returndata — `failed()` returns `false` — so the error path is never triggered: [5](#0-4) 

After the silent no-op call, `transfer_from_treasury` still executes and mints tokens to the recipient: [6](#0-5) 

### Impact Explanation

On any network where `L2AssetTracker` has not yet been deployed at `0x1000f` (new chain, pre-upgrade state), every L1 deposit transaction with `to_mint > 0` will:

1. Call `notify_l2_asset_tracker` → silently succeeds (empty account, no state change).
2. Proceed to `transfer_from_treasury` → tokens are minted and transferred correctly.
3. `L2AssetTracker.totalSuccessfulDepositsFromL1` is **never incremented**.

The result is a permanent, unrecoverable divergence between the actual minted token supply and the asset tracker's recorded deposit accounting. Any downstream interop logic (cross-chain settlement, asset migration checks using `totalSupply() == 0`, `_needToForceSetAssetMigrationOnL2`) will operate on stale/zero data, potentially breaking bridge invariants and cross-chain token accounting for the entire lifetime of the chain.

### Likelihood Explanation

This is triggered by any unprivileged user submitting an L1 deposit transaction (`to_mint > 0`) on a network where `L2AssetTracker` has not been deployed. This is a realistic scenario for any new ZKsync OS chain deployment before the upgrade that installs `L2AssetTracker`. The attacker does not need any special privileges — a standard L1→L2 deposit transaction is sufficient.

### Recommendation

Add an explicit bytecode-existence check before calling `notify_l2_asset_tracker`. If `L2AssetTracker` has no deployed code at `L2_ASSET_TRACKER_ADDRESS`, either:
- Revert the entire L1 transaction (fail-safe), or
- Skip the notification and emit a warning log, but only if this is an explicitly supported pre-upgrade mode.

The check should use `system.io.get_account_properties` (or equivalent) to verify `bytecode_hash != Bytes32::zero()` before dispatching the call. The current comment-only assumption is insufficient for a safety-critical accounting path.

### Proof of Concept

1. Deploy a fresh ZKsync OS chain without running the upgrade that installs `L2AssetTracker` at `0x1000f`.
2. Submit an L1 transaction with `to_mint = 1_000_000` (any nonzero deposit amount).
3. Observe: the transaction succeeds, tokens are minted to the recipient via `transfer_from_treasury`.
4. Query `L2AssetTracker` storage slot 156 (interop info mapping for `totalSuccessfulDepositsFromL1`): value is `0`.
5. The asset tracker accounting is permanently wrong — all future interop operations that depend on `totalSuccessfulDepositsFromL1` will see incorrect data.

The test infrastructure itself confirms this behavior: `install_default_predeployed_contracts` must explicitly deploy `L2AssetTracker` bytecode and seed its storage for tests to pass, demonstrating that the contract's presence is a required precondition that is not enforced at runtime. [7](#0-6)
