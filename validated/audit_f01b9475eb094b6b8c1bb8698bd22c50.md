### Title
Missing Storage Collision Check in CREATE/CREATE2 Allows Attacker to Pre-Populate Storage at Deployment Address - (`evm_interpreter/src/ee_trait_impl.rs`)

---

### Summary

ZKsync OS's EVM interpreter omits the EVM-specified storage non-emptiness check during CREATE/CREATE2 collision detection. Per EIP-161 (Spurious Dragon), a deployment must fail if the target address has non-zero nonce, non-empty code, **or non-empty storage**. ZKsync OS only checks nonce and code length. An attacker can exploit this by deploying a contract to a deterministic CREATE2 address, writing attacker-controlled values into storage, then self-destructing (which clears nonce and code but leaves storage intact). When the victim subsequently deploys to the same address, ZKsync OS allows the deployment to succeed, and the newly deployed contract inherits the attacker-controlled storage. This is a direct analog to H-08: just as an attacker front-runs `initialize()` to gain ownership of a proxy, here an attacker pre-populates storage to corrupt or DOS the initialization of any contract deployed to a predictable CREATE2 address.

---

### Finding Description

In `evm_interpreter/src/ee_trait_impl.rs`, the `before_executing_frame` hook performs the CREATE/CREATE2 collision check:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
``` [1](#0-0) 

The comment explicitly acknowledges the missing storage check. The `can_deploy_into` helper in `zk_ee/src/system/io.rs` mirrors this gap:

```rust
pub fn can_deploy_into(&self) -> bool {
    self.nonce.0 == 0 && self.has_bytecode() == false
}
``` [2](#0-1) 

The EVM test suite confirms this is a known deviation — the `create2collisionStorageParis` and `RevertInCreateInInitCreate2Paris` test cases are explicitly disabled across all test index files with comments:

> `comment: We do not check storage for collision`
> `comment: We do not check for storage collisions` [3](#0-2) [4](#0-3) 

**Attack flow**:
1. Attacker computes the CREATE2 address of a victim's planned proxy deployment (e.g., using a public CREATE2 factory like `0x4e59b44847b379578588920cA78FbF26c0B4956C`).
2. Attacker deploys a contract to that address using the same factory, salt, and initcode hash.
3. The attacker's contract writes attacker-controlled values to storage (e.g., slot 0 = 1, which is the `initialized` flag in OpenZeppelin-style proxies).
4. The attacker's contract calls `SELFDESTRUCT`, clearing nonce and code but leaving storage intact.
5. Victim deploys their proxy contract to the same CREATE2 address.
6. ZKsync OS: `deployee_code_len == 0` and `deployee_nonce == 0` → collision check passes → deployment succeeds.
7. The victim's proxy contract is deployed with `initialized = 1` already set in storage.
8. The victim's `initialize()` call reverts because the contract believes it is already initialized.
9. The proxy is permanently bricked — DOS of the deployment.

---

### Impact Explanation

- **Deployed contracts inherit attacker-controlled storage.** Any contract deployed to a CREATE2 address where an attacker previously wrote storage will start with corrupted state.
- **Initialization DOS.** Contracts using storage-based initialization guards (e.g., OpenZeppelin's `Initializable`, which stores an `initialized` flag in slot 0) will have their `initialize()` permanently blocked.
- **State manipulation.** Beyond DOS, the attacker can set arbitrary storage values — e.g., pre-setting an `owner` slot to their own address — giving them control over the deployed contract's logic.
- This is a direct analog to H-08: instead of front-running `initialize()` after deployment, the attacker corrupts the storage *before* deployment, achieving the same or worse outcome.

---

### Likelihood Explanation

- **Moderate-to-high.** The attack requires knowing the target CREATE2 address in advance, which is trivially computable from public on-chain data (deployer address, salt, initcode).
- Public CREATE2 factories (widely used in production) allow any caller to deploy with any salt, making the pre-population step straightforward.
- The attacker only needs to act before the victim's deployment transaction is included — a standard front-running window on any sequencer.
- The self-destruct + storage persistence pattern is well-known and has been used in prior EVM exploits.

---

### Recommendation

Implement the storage non-emptiness check as part of the CREATE/CREATE2 collision detection in `before_executing_frame`. Before allowing a constructor to execute, verify that the target address has no non-zero storage slots. Since a full storage scan is expensive, the standard approach is to track whether any storage was ever written to an address (a "dirty" flag) and use that as a proxy for non-empty storage. Alternatively, maintain a per-address "has been written" bit in the account cache metadata.

The `can_deploy_into` function in `zk_ee/src/system/io.rs` should also be updated to reflect the full EIP-161 collision condition once the storage check is implemented. [1](#0-0) [5](#0-4)

### Proof of Concept

```
// Attacker contract (deployed to victim's CREATE2 address first)
contract Poisoner {
    constructor() {
        // Slot 0 = 1 (marks contract as "initialized" in OZ Initializable)
        assembly { sstore(0, 1) }
        selfdestruct(payable(msg.sender));
    }
}

// Step 1: Attacker computes victim's CREATE2 address
// address = keccak256(0xff ++ factory ++ salt ++ keccak256(victimInitcode))[12:]

// Step 2: Attacker deploys Poisoner to that address using the same factory
// factory.deploy(poisonerInitcode, salt)
// → Poisoner sets slot 0 = 1, then self-destructs
// → Address now has: nonce=0, code=empty, storage[0]=1

// Step 3: Victim deploys their proxy
// factory.deploy(victimInitcode, salt)
// → ZKsync OS: nonce==0 && code_len==0 → collision check passes → deployment succeeds
// → Proxy deployed with storage[0]=1 (initialized=true)

// Step 4: Victim calls initialize()
// proxy.initialize(...)
// → Reverts: "Initializable: contract is already initialized"
// → Proxy is permanently bricked
```
