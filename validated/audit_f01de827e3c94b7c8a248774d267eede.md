### Title
ERC-20 Mirror Token Accounting Diverges from Actual NEP-141 Balance for Fee-on-Transfer Tokens - (File: engine/src/engine.rs)

### Summary
When a NEP-141 token with a transfer fee (fee-on-transfer / deflationary token) is bridged to Aurora via `ft_on_transfer`, the engine mints ERC-20 tokens equal to `args.amount` — the amount *sent* by the caller — rather than the actual balance increase Aurora received. Because the NEP-141 contract deducts a fee during transfer, Aurora's actual NEP-141 balance is less than the ERC-20 tokens minted. This causes the ERC-20 mirror token's total supply to exceed the NEP-141 tokens Aurora actually holds, making the bridge insolvent for that token and preventing the last users from redeeming their ERC-20 tokens.

### Finding Description
The `ft_on_transfer` entry point in `engine/src/contract_methods/connector.rs` dispatches to `Engine::receive_erc20_tokens` for any NEP-141 token that is not the base ETH connector. [1](#0-0) 

Inside `receive_erc20_tokens`, the amount to mint is taken directly from `args.amount`: [2](#0-1) 

This value is passed unchanged into `setup_receive_erc20_tokens_input`, which encodes it as the ERC-20 mint amount: [3](#0-2) 

The ERC-20 `mint` function in `EvmErc20.sol` then mints exactly this amount to the recipient: [4](#0-3) 

The NEP-141 `ft_transfer_call` protocol guarantees that `ft_on_transfer` is called with the `amount` field set to the amount the *sender* specified, not the amount Aurora's account actually received. For a fee-on-transfer NEP-141 token, the NEP-141 contract deducts a fee during the transfer, so Aurora's actual balance increase is `amount - fee`. However, `args.amount` is still the full pre-fee value. Aurora mints `amount` ERC-20 tokens while holding only `amount - fee` NEP-141 tokens.

There is no post-transfer balance check anywhere in the `receive_erc20_tokens` path to detect this discrepancy.

### Impact Explanation
After bridging a fee-on-transfer NEP-141 token, the ERC-20 mirror token's total supply exceeds the NEP-141 tokens Aurora holds. When users call `withdrawToNear` on the ERC-20 (burning ERC-20 tokens), Aurora attempts to `ft_transfer` the corresponding NEP-141 tokens back. The last users to exit will find Aurora's NEP-141 balance insufficient to cover their ERC-20 holdings. This constitutes bridge insolvency for the affected token and results in a permanent freeze of funds for those users.

**Impact: High — Permanent freezing of funds / Insolvency of the ERC-20 bridge for fee-on-transfer NEP-141 tokens.**

### Likelihood Explanation
Fee-on-transfer tokens are a well-known and deployed pattern in the NEP-141 ecosystem (analogous to deflationary ERC-20s on Ethereum). Any unprivileged user can:
1. Deploy or use an existing fee-on-transfer NEP-141 token.
2. Call `deploy_erc20_token` on Aurora to register the ERC-20 mirror.
3. Call `ft_transfer_call` on the NEP-141 contract, routing tokens to Aurora.

Each such bridge operation silently inflates the ERC-20 supply relative to Aurora's actual NEP-141 holdings. The discrepancy accumulates with every bridge-in transaction and is never corrected because there is no reconciliation mechanism analogous to the interest-collection path described in the original report.

**Likelihood: Medium** — Requires a fee-on-transfer NEP-141 token, but such tokens are a standard pattern and the attack path is fully permissionless.

### Recommendation
In `receive_erc20_tokens`, instead of using `args.amount` directly as the mint amount, query Aurora's NEP-141 balance before and after the `ft_transfer_call` completes and use the actual balance increase as the mint amount. Alternatively, reject `ft_on_transfer` calls from NEP-141 tokens that are known to charge transfer fees, or add an explicit check that the amount to mint does not exceed the actual balance increase observed on-chain.

### Proof of Concept
1. Deploy a NEP-141 token `FeeToken` that charges a 1% fee on every transfer (i.e., a transfer of 1000 units results in the recipient receiving 990 units, with 10 burned or sent to a fee collector).
2. Call `deploy_erc20_token` on Aurora to create the ERC-20 mirror `FeeTokenERC20`.
3. Alice calls `ft_transfer_call` on `FeeToken` with `amount = 1000`, `receiver_id = aurora`, `msg = alice_evm_address`.
4. `FeeToken` transfers 990 units to Aurora (after 1% fee) and calls `ft_on_transfer` on Aurora with `amount = 1000`.
5. Aurora's `receive_erc20_tokens` mints 1000 `FeeTokenERC20` to Alice's EVM address.
6. Aurora now holds 990 `FeeToken` NEP-141 units but has issued 1000 `FeeTokenERC20` units.
7. Alice calls `withdrawToNear` on `FeeTokenERC20` to burn 1000 ERC-20 and reclaim 1000 NEP-141. Aurora attempts `ft_transfer(alice, 1000)` on `FeeToken` but only holds 990 — the call fails, permanently freezing Alice's funds. [5](#0-4) [6](#0-5)

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
