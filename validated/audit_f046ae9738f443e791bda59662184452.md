Audit Report

## Title
Silent Capping of Oversized `icrc2_approve` Amounts to `u64::MAX` Instead of Returning an Error - (`rs/ledger_suite/icrc1/ledger/src/main.rs`, `rs/ledger_suite/icp/ledger/src/main.rs`)

## Summary
Both the ICRC-1 and ICP ledgers accept `icrc2_approve` calls with `amount` values exceeding `u64::MAX` (valid as `nat` on the wire) and silently substitute `Tokens::max_value()` / `u64::MAX` instead of returning an error. The approval fee is charged, a block is committed, and `Ok(block_index)` is returned — with no indication that the stored allowance differs from the submitted value. Any programmatic caller that submits an oversized amount (e.g., due to an arithmetic overflow or unit-conversion error) unknowingly grants the spender an effectively unlimited allowance.

## Finding Description
`ApproveArgs.amount` is typed as `Nat` (arbitrary-precision) in both the Candid interface and the Rust `ApproveArgs` struct (`packages/icrc-ledger-types/src/icrc2/approve.rs`, line 16). Internally, both ledgers store token amounts as `u64`.

**ICRC-1 ledger** (`rs/ledger_suite/icrc1/ledger/src/main.rs`, line 840):
```rust
let amount = Tokens::try_from(arg.amount).unwrap_or_else(|_| Tokens::max_value());
```
`Tokens::try_from(Nat)` (in `rs/ledger_suite/common/ledger_core/src/tokens.rs`, lines 288–296) calls `value.0.to_u64()` and returns `Err` for any value exceeding `u64::MAX`. The `unwrap_or_else` silently substitutes `Tokens::MAX` (`u64::MAX`).

**ICP ledger** (`rs/ledger_suite/icp/ledger/src/main.rs`, line 1350):
```rust
let allowance = Tokens::from_e8s(arg.amount.0.to_u64().unwrap_or(u64::MAX));
```
`to_u64()` returns `None` for values above `u64::MAX`; `unwrap_or(u64::MAX)` silently substitutes the maximum.

In both cases, execution continues: the fee is deducted, the block is written, and `Ok(block_index)` is returned (ICRC-1: line 890; ICP: line 1415). The caller receives a success response with no indication that the stored allowance differs from the submitted value.

The `expected_allowance` guard only protects callers who explicitly set it. When `expected_allowance` is `None` (the common case), there is no protection. Notably, when `expected_allowance` itself overflows, both ledgers correctly return `ApproveError::AllowanceChanged` — making the asymmetric treatment of `amount` inconsistent and surprising.

## Impact Explanation
This is a concrete, in-scope ledger security impact. Any caller — including SNS swap canisters, DeFi aggregators, or wallet SDKs — that submits `amount = u64::MAX + 1` (a valid `nat`) pays the approval fee and receives `Ok(block_index)`, while the spender silently holds a `u64::MAX` allowance. A programmatic caller that expected the call to fail (e.g., as a boundary check) or that intended to set a specific large-but-bounded amount instead grants an effectively unlimited allowance. This constitutes unauthorized/unintended fund delegation on in-scope ICP and ICRC-1 financial ledgers, with concrete user and protocol harm. Severity: **High** — significant ledger security impact with concrete user harm, matching the "Significant... ledger... security impact with concrete user or protocol harm" allowed impact class.

## Likelihood Explanation
Moderate. The `amount` field is `nat` and programmatic callers routinely compute allowance amounts arithmetically. An off-by-one or unit-conversion error producing a value just above `u64::MAX` silently becomes an unlimited approval. The ckBTC and ckETH withdrawal flows encourage approving large amounts to avoid repeated approvals, increasing the probability of oversized values being submitted. No special privileges are required; any unprivileged ingress sender or canister caller can trigger this.

## Recommendation
Return `ApproveError::GenericError` (or a new `AmountTooLarge` variant) when the submitted `amount` cannot be represented in the ledger's internal token type, instead of silently capping it. Apply the same fix to the `expected_allowance` path for consistency (though that path already returns an error, the error type is `AllowanceChanged` rather than a dedicated overflow error).

```rust
// ICRC-1 ledger fix (rs/ledger_suite/icrc1/ledger/src/main.rs, line 840)
let amount = Tokens::try_from(arg.amount)
    .map_err(|_| ApproveError::GenericError {
        error_code: Nat::from(1u64),
        message: "amount exceeds maximum token value".to_string(),
    })?;

// ICP ledger fix (rs/ledger_suite/icp/ledger/src/main.rs, line 1350)
let allowance = arg.amount.0.to_u64()
    .map(Tokens::from_e8s)
    .ok_or_else(|| ApproveError::GenericError {
        error_code: Nat::from(1u64),
        message: "amount exceeds maximum token value".to_string(),
    })?;
```

## Proof of Concept
Submit via ingress or canister call:
```
icrc2_approve(record {
  spender  = record { owner = principal "<SPENDER>" };
  amount   = 18_446_744_073_709_551_616;  // u64::MAX + 1, valid nat
  fee      = null;
  expires_at = null;
  expected_allowance = null;
  from_subaccount = null;
  memo = null;
  created_at_time = null;
})
```
Expected (correct) behavior: `Err(ApproveError::GenericError { ... })`, fee not charged.
Actual behavior: `Ok(block_index)`, fee deducted, `icrc2_allowance` returns `u64::MAX`.

A deterministic integration test using `PocketIC` or `StateMachine` can confirm this by:
1. Minting tokens to an account.
2. Calling `icrc2_approve` with `amount = Nat::from(u64::MAX) + Nat::from(1u64)`.
3. Asserting the return is `Ok(_)` (demonstrating the bug).
4. Querying `icrc2_allowance` and asserting the stored allowance equals `u64::MAX`.
5. Asserting the caller's balance decreased by the approval fee.
