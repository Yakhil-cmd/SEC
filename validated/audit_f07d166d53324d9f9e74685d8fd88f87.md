Audit Report

## Title
Block Proposer Controls Ingress Insertion Timestamp, Enabling Intra-Block Message Ordering Manipulation - (File: `rs/ingress_manager/src/ingress_selector.rs`)

## Summary
The ingress selector in `get_ingress_payload` sorts per-canister message queues by pool insertion timestamp to enforce fairness, but that timestamp is assigned by the node's own clock inside the artifact processor loop — not by the sender. A node operator who is the block proposer can ensure its own ingress messages receive an earlier timestamp than competing messages by submitting them before victims' messages are processed, then constructing a block where its message executes first. Against the SNS swap canister near its ICP participation cap, this allows the proposer to capture the last allocation slot and cause the victim's call to be rejected.

## Finding Description

**Timestamp assignment — `rs/p2p/artifact_manager/src/lib.rs` L382–388**

`IngressProcessor::process_changes` stamps every ingress message with the node's local clock at the instant the artifact-processor loop drains its input channel:

```rust
UnvalidatedArtifactMutation::Insert((message, peer_id)) => {
    let unvalidated_artifact = UnvalidatedArtifact {
        message,
        peer_id,
        timestamp: time_source.get_relative_time(),  // node-local, not sender-set
    };
    ingress_pool.insert(unvalidated_artifact);
}
```

The processing loop (`process_messages`, L183–224) polls with a 200 ms timeout (`ARTIFACT_MANAGER_TIMER_DURATION_MSEC = 200`, L227). Each drain cycle calls `time_source.get_relative_time()` individually per message inside the `for artifact_event in artifact_events` loop (L380), so messages processed in different drain cycles receive meaningfully different timestamps.

**Ordering in the ingress selector — `rs/ingress_manager/src/ingress_selector.rs` L146–154**

`get_ingress_payload` explicitly sorts per-canister queues by this timestamp in descending order and `pop()`s from the back, so the message with the **smallest** (oldest) timestamp is selected first:

```rust
// In order to prevent malicious users from putting their messages ahead of others
// by carefully crafting the expiry times, we sort the ingress messages by the time
// they were delivered to the pool.
for v in canister_queues.values_mut() {
    v.msgs.sort_unstable_by_key(|artifact| {
        std::cmp::Reverse(artifact.timestamp.as_nanos_since_unix_epoch())
    });
}
```

The comment acknowledges the fairness intent, but the fix is incomplete: it moves the manipulation surface from the user-controlled expiry field to the node-controlled insertion timestamp.

**Exploit path**

A node operator running the block-proposing replica can:
1. Monitor the SNS swap canister state via query calls until the remaining headroom is within one `max_participant_icp_e8s`.
2. Submit its own `refresh_buyer_tokens` call immediately; the artifact processor drains it in the current cycle, assigning timestamp `T_self`.
3. A victim's `refresh_buyer_tokens` call arrives and is processed in a subsequent drain cycle, receiving timestamp `T_victim > T_self`.
4. When `get_ingress_payload` runs, the proposer's message has the smaller (older) timestamp and is popped first.
5. The proposer's call executes first inside the block.

Alternatively, with modified replica software, the proposer can delay draining the channel for victim messages (the 200 ms poll interval provides a natural window), submit its own message, then drain — ensuring its message is processed in an earlier cycle with a smaller timestamp.

**SNS swap impact — `rs/sns/swap/src/swap.rs`**

`refresh_buyer_token_e8s` computes available headroom at call time (L1177) and caps the accepted amount to whatever remains (L1224). When the proposer's call executes first and consumes all remaining headroom, the victim's subsequent call computes `actual_increment_e8s = 0`, and if the resulting `new_balance_e8s` falls below `min_participant_icp_e8s`, the call is rejected (L1241–1245). The swap code itself acknowledges message re-ordering as a known concern (L1215–1216: `// Strict inequality can happen if messages are re-ordered`), confirming the scenario is realistic.

**Why existing checks are insufficient**

The timestamp-based sort was introduced specifically to prevent ordering manipulation, but it replaces one unverifiable key (user-set expiry) with another (node-set insertion time). No other replica can independently verify the insertion timestamp; it is not part of the signed ingress message and is not consensus-certified.

## Impact Explanation

A malicious block proposer can guarantee its own `refresh_buyer_tokens` call executes before any competing call in the same block. When the SNS swap is near its ICP cap, this lets the proposer capture the last participation slot, causing the victim's call — submitted earlier — to be rejected or receive a zero allocation. The victim's ICP is locked in the swap subaccount until `error_refund_icp` is called after the swap closes. This constitutes a concrete, targeted financial loss for the victim and an equivalent gain for the proposer. This matches the **Medium** impact class: meaningful security impact with significant constraints (node operator control, correct round selection, near-cap timing).

## Likelihood Explanation

Every subnet node is eligible to be block proposer in any given round. Block proposer selection is deterministic and publicly observable via the random beacon, so an attacker can prepare in advance. The attacker needs only to operate a single node on the application subnet hosting the SNS swap canister — no threshold corruption, governance majority, or privileged key is required. The attacker can monitor the swap's participation level via query calls and wait for the cap to come within range. The attack is repeatable across multiple SNS swaps and any other canister with first-come-first-served semantics.

## Recommendation

1. **Remove the timestamp-based ordering dependency.** The pool insertion timestamp is node-local and unverifiable by other replicas; it cannot provide a consensus-safe fairness guarantee.
2. **Use a verifiable, sender-controlled ordering key.** The ingress expiry time is user-set and cryptographically signed; rounding it to the nearest second prevents fine-grained manipulation while providing a canonical order all replicas can independently verify.
3. **Alternatively, randomize intra-block ordering** using the round's random tape, which is unpredictable to the proposer at payload-construction time.
4. **For the SNS swap specifically**, consider making `refresh_buyer_tokens` idempotent with respect to ordering by snapshotting the ICP ledger balance at a certified height rather than relying on first-writer-wins execution order.

## Proof of Concept

1. Deploy an SNS swap canister on an application subnet with `max_direct_participation_icp_e8s = 100 ICP` and `max_participant_icp_e8s = 10 ICP`.
2. Fill the swap to 91 ICP (9 ICP remaining headroom, below `max_participant_icp_e8s`).
3. Victim transfers 9 ICP to their swap subaccount and sends `refresh_buyer_tokens`; the message enters the proposer node's artifact-processor channel.
4. The proposer node (running modified software or simply acting within the 200 ms poll window) delays draining the channel, then submits its own `refresh_buyer_tokens` with 9 ICP deposited; the artifact processor assigns the proposer's message timestamp `T` and the victim's message timestamp `T + delay`.
5. The proposer calls `get_ingress_payload`; the selector pops the proposer's message first (timestamp `T < T + delay`).
6. The proposer's call executes first: `available_direct_participation_e8s()` returns 9 ICP, the proposer receives the full allocation, swap reaches cap.
7. The victim's call executes second: `available_direct_participation_e8s()` returns 0, `actual_increment_e8s = 0`, `new_balance_e8s = 0 < min_participant_icp_e8s`, call rejected with "Rejecting participation of effective amount 0".
8. Victim must reclaim ICP via `error_refund_icp` after swap closes.

A deterministic integration test can reproduce this by injecting two ingress messages into the same block with controlled timestamps (earlier for the proposer, later for the victim) and asserting that the victim's `refresh_buyer_tokens` call returns an error while the proposer's succeeds.
