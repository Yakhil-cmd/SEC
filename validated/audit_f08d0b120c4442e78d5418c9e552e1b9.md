### Title
SNS-WASM Canister Uses Deprecated 64-bit `msg_cycles_accept` / `call_with_payment` Instead of 128-bit Equivalents - (`rs/nns/sns-wasm/canister/canister.rs`)

---

### Summary

The SNS-WASM canister's `CanisterApiImpl` uses the deprecated 64-bit cycles system APIs (`msg_cycles_accept` and `call_with_payment`) instead of their 128-bit counterparts (`msg_cycles_accept128` / `call_with_payment128`). This is the direct IC analog of the Solidity `transfer()` vs `call()` issue: a hardcoded resource-width limit that silently truncates or fails when cycles amounts exceed `u64::MAX`, and that can break integrations if the IC's cycles economy evolves.

---

### Finding Description

In `rs/nns/sns-wasm/canister/canister.rs`, the `CanisterApiImpl` trait implementation contains two affected functions:

**1. `accept_message_cycles` (line 178–184):**
```rust
fn accept_message_cycles(&self, cycles: Option<u64>) -> Result<u64, String> {
    let cycles = cycles.unwrap_or_else(ic_cdk::api::call::msg_cycles_available);
    self.message_has_enough_cycles(cycles)?;
    let accepted = ic_cdk::api::call::msg_cycles_accept(cycles);
    Ok(accepted)
}
```
`ic_cdk::api::call::msg_cycles_available()` wraps the deprecated `ic0.msg_cycles_available` (64-bit). If more than `u64::MAX` cycles are attached to the incoming call, the available amount is silently capped at `u64::MAX`. `msg_cycles_accept` is likewise the deprecated 64-bit syscall, explicitly documented as deprecated in `rs/interfaces/src/execution_environment.rs` with the note *"Please use `ic0_msg_cycles_accept128` instead. This API supports only 64-bit values."*

**2. `send_cycles_to_canister` (line 186–198):**
```rust
async fn send_cycles_to_canister(&self, target: CanisterId, cycles: u64) -> Result<(), String> {
    let response: CallResult<()> = ic_cdk::api::call::call_with_payment(
        CanisterId::ic_00().get().0,
        "deposit_cycles",
        (CanisterIdRecord::from(target),),
        cycles,
    ).await;
    ...
}
```
`call_with_payment` is the deprecated 64-bit variant. The 128-bit replacement is `call_with_payment128`. The function signature itself is `u64`, meaning any caller passing a cycles amount above `u64::MAX` would be silently truncated before the management canister call is made.

The IC protocol explicitly marks these APIs as deprecated in favour of 128-bit equivalents: [1](#0-0) [2](#0-1) 

The SNS-WASM production canister uses the deprecated 64-bit path: [3](#0-2) 

---

### Impact Explanation

If cycles amounts attached to an SNS deployment call grow beyond `u64::MAX` (~18.4 × 10¹⁸), or if the IC's cycles cost schedule is updated such that SNS deployment requires more than `u64::MAX` cycles:

- `msg_cycles_available()` silently returns `u64::MAX` instead of the true available amount.
- `msg_cycles_accept` accepts at most `u64::MAX` cycles; the remainder is refunded to the caller.
- `send_cycles_to_canister` can only forward at most `u64::MAX` cycles to the management canister's `deposit_cycles`, silently truncating any excess.

This means `deploy_new_sns` could silently under-fund the newly created SNS canisters, causing them to be created with insufficient cycles and immediately freeze or fail to operate — without any error being surfaced to the caller. The caller's excess cycles are refunded, but the deployment proceeds with a truncated budget.

---

### Likelihood Explanation

The IC's `DEFAULT_CYCLES_LIMIT` was already raised from `50e15` to `150e15` in December 2024 due to price increases. The `SUBNET_RENTAL_DEFAULT_CYCLES_LIMIT` is `500e15`. These are currently well below `u64::MAX` (~18.4 × 10¹⁸), but the trend is upward. Any future protocol change that raises per-operation costs or requires large cycles deposits for SNS deployment would push amounts closer to the 64-bit ceiling. The entry path is fully unprivileged: any user can call `deploy_new_sns` on the SNS-WASM canister with attached cycles. [4](#0-3) 

---

### Recommendation

Replace all deprecated 64-bit cycles APIs in `rs/nns/sns-wasm/canister/canister.rs` with their 128-bit equivalents:

- Replace `ic_cdk::api::call::msg_cycles_available()` → `ic_cdk::api::call::msg_cycles_available128()`
- Replace `ic_cdk::api::call::msg_cycles_accept(cycles)` → `ic_cdk::api::call::msg_cycles_accept128(cycles as u128)`
- Change `send_cycles_to_canister` parameter from `u64` to `u128` and use `call_with_payment128`
- Update `accept_message_cycles` and `message_has_enough_cycles` signatures to use `u128`

This mirrors the recommendation in the original report: use the more capable API so that future cost changes do not silently break the integration.

---

### Proof of Concept

1. Deploy the SNS-WASM canister on a local replica.
2. Call `deploy_new_sns` attaching exactly `u64::MAX + 1` cycles (requires a test harness that can attach 128-bit cycle amounts).
3. Observe that `msg_cycles_available()` returns `u64::MAX` (truncated), `msg_cycles_accept` accepts only `u64::MAX`, and the SNS canisters are created with `u64::MAX` cycles instead of the full amount.
4. If the SNS deployment cost schedule is set above `u64::MAX`, the deployment silently under-funds the new canisters. [5](#0-4)

    fn message_has_enough_cycles(&self, required_cycles: u64) -> Result<u64, String> {
        let available = ic_cdk::api::call::msg_cycles_available();

        if available < required_cycles {
            return Err(format!(
                "Message execution requires at least {required_cycles} cycles, but only {available} cycles were sent.",
            ));
        }
        Ok(available)
    }

    fn accept_message_cycles(&self, cycles: Option<u64>) -> Result<u64, String> {

        let accepted = ic_cdk::api::call::msg_cycles_accept(cycles);
