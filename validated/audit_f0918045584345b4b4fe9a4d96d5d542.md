### Title
Scheduled XCC Promises Have No Deadline/Expiry, Enabling Stale Execution by Anyone - (`etc/xcc-router/src/lib.rs`)

---

### Summary

The `execute_scheduled` function in the XCC router contract has no deadline or expiry check on stored promises. A user who schedules a cross-contract call via `CrossContractCallArgs::Delayed` has their promise stored indefinitely with no timestamp bound. Because `execute_scheduled` is intentionally callable by anyone, a third party can trigger execution at any future time — including after market conditions have changed — and the user has no mechanism to cancel the scheduled promise.

---

### Finding Description

When a user calls the XCC precompile with `CrossContractCallArgs::Delayed`, the Aurora Engine calls `schedule()` on the user's router sub-account, which stores the `PromiseArgs` in `scheduled_promises` keyed by a monotonic nonce: [1](#0-0) 

The stored `PromiseArgs` type contains no deadline or expiry field: [2](#0-1) 

The `execute_scheduled` function is explicitly open to any caller, performs no timestamp check, and has no cancellation counterpart anywhere in the contract: [3](#0-2) 

A `grep` across the entire repository confirms there is no `cancel_scheduled`, `remove_scheduled`, or equivalent function. Once a promise is scheduled, it is irrevocable and executable by anyone at any future block.

The `CrossContractCallArgs::Delayed` variant is documented as the mechanism for expanding available NEAR gas by deferring execution to a separate transaction: [4](#0-3) 

---

### Impact Explanation

A user scheduling a Delayed XCC call (e.g., a token swap, a DeFi deposit, or an `ft_transfer` on a NEAR-native protocol) commits their NEAR/wNEAR to the router at the time of scheduling. The `attached_balance` field in `PromiseCreateArgs` specifies NEAR to be forwarded from the router's balance: [5](#0-4) 

Because the promise has no expiry and cannot be cancelled, the user's funds are effectively frozen in the router until `execute_scheduled` is called. A malicious actor can:

1. Monitor the chain for scheduled promises.
2. Wait until market conditions are maximally unfavorable for the user (e.g., after a price crash that makes a swap execute at a loss, or after a DeFi protocol's rates have deteriorated).
3. Call `execute_scheduled(nonce)` to trigger execution with the user's original, now-stale parameters.

This constitutes **temporary freezing of funds** (the user cannot recover or redirect their NEAR until execution occurs) and **theft of unclaimed yield** (execution at stale conditions produces worse-than-expected returns that a deadline would have prevented).

---

### Likelihood Explanation

The `execute_scheduled` function is intentionally permissionless — the contract comment explicitly states "anyone can call this function." This means no special privilege is required. Any unprivileged NEAR account can call `execute_scheduled` on any router sub-account at any time. The attacker's only requirement is knowing the nonce of the scheduled promise, which is logged on-chain via `near_sdk::log!` at scheduling time: [6](#0-5) 

This makes the attack trivially discoverable and executable by any on-chain observer.

---

### Recommendation

1. Add an optional `deadline` field to `PromiseArgs` (or as a wrapper in `scheduled_promises`). In `execute_scheduled`, revert if `env::block_timestamp() > deadline`.
2. Add a `cancel_scheduled(nonce: U64)` function callable only by the parent (Aurora Engine), so users can cancel stale promises via the EVM.
3. Alternatively, restrict `execute_scheduled` to only the parent account, removing the open-caller design, and require the user to explicitly re-trigger execution.

---

### Proof of Concept

1. Alice (EVM user) calls the XCC precompile with `CrossContractCallArgs::Delayed(PromiseArgs::Create(swap_call))`, where `swap_call` targets a NEAR DEX with `attached_balance = 10 NEAR` and swap args encoding `min_amount_out = 95 TOKEN` (based on current price of 10 TOKEN/NEAR).
2. Aurora Engine calls `schedule()` on Alice's router. The promise is stored at nonce `0`. The log `"Promise scheduled at nonce 0"` is emitted on-chain.
3. The TOKEN price crashes. Alice wants to cancel but has no mechanism to do so.
4. Bob (attacker) calls `execute_scheduled({"nonce": "0"})` on Alice's router. The swap executes with Alice's 10 NEAR at the crashed price, returning only 96 TOKEN (just above `min_amount_out`). Alice loses the yield she would have captured by cancelling and re-entering at a better time.
5. Alice had no recourse: no deadline check in `execute_scheduled`, no cancel function, and the function is open to any caller. [7](#0-6)

```rust
    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }
```

```rust
    /// It is intentional that this function can be called by anyone (not just the parent).
    /// There is no security risk to allowing this function to be open because it can only
    /// act on promises that were created via `schedule`.
    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }
```
