### Title
Unchecked Exit Precompile Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

### Summary

In `EvmErc20.sol` and `EvmErc20V2.sol`, both `withdrawToNear` and `withdrawToEthereum` burn the caller's ERC-20 tokens **before** calling the exit precompile via inline assembly. The return value of the assembly `call` is captured in `res` but never checked. If the precompile call fails (returns `0`), execution continues silently, the tokens are permanently destroyed, and no NEAR-side withdrawal is ever scheduled. The user's funds are irrecoverably lost with no emergency exit path.

### Finding Description

In `EvmErc20.sol`, both withdrawal functions follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}
``` [1](#0-0) [2](#0-1) 

The identical pattern exists in `EvmErc20V2.sol`: [3](#0-2) [4](#0-3) 

The exit precompile (`ExitToNear` at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`, `ExitToEthereum` at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`) can return failure (`0`) under several reachable conditions:

1. `get_nep141_from_erc20` fails — the ERC-20 contract address is not found in the NEP-141 registry (e.g., token deployed but not registered, or registry corrupted).
2. `get_eth_connector_contract_account` fails — the eth connector account ID is not set in engine storage.
3. Out-of-gas — the caller forwards insufficient gas to the precompile call.
4. Input validation errors — e.g., `validate_input_size` rejects the input, or `parse_amount` fails if amount overflows `u128`. [5](#0-4) [6](#0-5) 

In all these cases, the Solidity `call` opcode returns `0` (failure), but since `res` is never checked and no `revert` is issued, the EVM transaction completes successfully. The `_burn` is already committed; no promise is ever scheduled on the NEAR side.

The `error_refund` feature provides a callback-based refund only when the NEAR-side `ft_transfer` promise fails *after* the precompile has already succeeded and scheduled the promise. It does not cover the case where the precompile itself fails and never schedules a promise. [7](#0-6) 

### Impact Explanation

**Critical — Permanent freezing/loss of user funds.**

When the precompile call fails silently:
- The user's ERC-20 tokens are permanently burned (irreversible `_burn`).
- No NEAR-side NEP-141 `ft_transfer` or `withdraw` promise is ever created.
- The underlying NEP-141 tokens remain locked in the Aurora engine contract forever.
- There is no emergency exit, no retry mechanism, and no refund path.

This is a direct, permanent loss of bridged user assets.

### Likelihood Explanation

**Medium.** The most realistic trigger is an out-of-gas condition: a caller who forwards insufficient gas to the assembly `call` (e.g., a contract that calls `withdrawToNear` with a gas stipend) will silently lose their tokens. Additionally, any future state inconsistency in the NEP-141 registry (e.g., a token deployed via a non-standard path) would trigger the `get_nep141_from_erc20` failure path. Both are reachable by an unprivileged EVM user without any admin involvement.

### Recommendation

1. **Check the return value** of the assembly `call` and revert if it returns `0`:
   ```solidity
   assembly {
       let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
       if iszero(res) { revert(0, 0) }
   }
   ```
   This ensures the `_burn` is atomically rolled back if the precompile fails.

2. **Invert the order of operations**: call the precompile first (or use a check-effects-interactions pattern) so that a precompile failure reverts before any tokens are burned.

3. Apply the same fix to both `withdrawToNear` and `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

### Proof of Concept

1. Deploy `EvmErc20` for a NEP-141 token that is **not** registered in the Aurora engine's ERC-20/NEP-141 map (or use a token where the registry entry has been removed).
2. Mint some tokens to `alice`.
3. `alice` calls `withdrawToNear("alice.near", 100)`.
4. `_burn(alice, 100)` executes — alice's balance drops to 0.
5. The assembly `call` to `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` fails because `get_nep141_from_erc20` returns `ERR_ERC20_NOT_FOUND`; `res = 0`.
6. No revert is issued; the transaction succeeds.
7. Alice's 100 ERC-20 tokens are gone. No NEP-141 tokens are transferred. Funds are permanently lost. [1](#0-0) [6](#0-5)
