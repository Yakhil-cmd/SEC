### Title
ckETH Burned Without Receiving ERC-20 Withdrawal When ckERC20 Burn Fails and `erc20_tx_fee <= CKETH_LEDGER_TRANSACTION_FEE` - (File: rs/ethereum/cketh/minter/src/main.rs)

### Summary

In the `withdraw_erc20` function of the ckETH minter, ckETH tokens are burned first as a gas fee deposit, and then ckERC20 tokens are burned in a second step. If the ckERC20 burn fails due to user error (insufficient funds or insufficient allowance), the minter attempts to reimburse the ckETH — but only if `reimbursed_amount > Wei::ZERO`. When `erc20_tx_fee` (the burned ckETH amount) is exactly equal to `CKETH_LEDGER_TRANSACTION_FEE` (2,000,000,000,000 Wei), the penalty subtraction yields `Wei::ZERO`, and **no reimbursement is scheduled**. The user's ckETH is permanently burned with no ERC-20 withdrawal occurring and no refund.

### Finding Description

The `withdraw_erc20` function in `rs/ethereum/cketh/minter/src/main.rs` performs two sequential burns:

1. Burns ckETH from the user's account as a gas fee deposit (line 448–458).
2. Burns ckERC20 tokens from the user's account (line 468–477).

If step 2 fails due to `InsufficientFunds`, `AmountTooLow`, or `InsufficientAllowance`, the code computes a reimbursement amount with a penalty:

```rust
LedgerBurnError::InsufficientFunds { .. }
| LedgerBurnError::AmountTooLow { .. }
| LedgerBurnError::InsufficientAllowance { .. } => erc20_tx_fee
    .checked_sub(CKETH_LEDGER_TRANSACTION_FEE)
    .unwrap_or(Wei::ZERO),
```

Then it only schedules a reimbursement if `reimbursed_amount > Wei::ZERO`:

```rust
if reimbursed_amount > Wei::ZERO {
    // schedule reimbursement
}
```

`CKETH_LEDGER_TRANSACTION_FEE` is defined as `Wei::new(2_000_000_000_000_u128)` (2 trillion Wei = 0.000002 ETH). [1](#0-0) 

The `erc20_tx_fee` is computed from `estimate_erc20_transaction_fee()`, which calls `lazy_refresh_gas_fee_estimate()` and multiplies by `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` (65,000 gas). [2](#0-1) 

When Ethereum gas prices are extremely low (e.g., near-zero base fee and near-minimum priority fee), the computed `erc20_tx_fee` can be at or near `CKETH_LEDGER_TRANSACTION_FEE`. In that case:

- `erc20_tx_fee.checked_sub(CKETH_LEDGER_TRANSACTION_FEE)` returns `Some(Wei::ZERO)` (or `None` → `unwrap_or(Wei::ZERO)`).
- `reimbursed_amount == Wei::ZERO`, so the `if reimbursed_amount > Wei::ZERO` guard is **false**.
- No `FailedErc20WithdrawalRequest` event is emitted, no reimbursement is queued.
- The user's ckETH is permanently burned with no ERC-20 withdrawal and no refund. [3](#0-2) 

This is the direct IC analog of the Illuminate H-4 bug: a token burn (ckETH) proceeds and is finalized, but the corresponding value transfer (ERC-20 withdrawal) fails, and the user receives nothing back.

### Impact Explanation

A user who calls `withdraw_erc20` when:
- Ethereum gas prices are very low (making `erc20_tx_fee` ≤ `CKETH_LEDGER_TRANSACTION_FEE`), AND
- The ckERC20 burn fails (e.g., they forgot to call `icrc2_approve` on the ckERC20 ledger, or have insufficient ckERC20 balance)

will have their ckETH permanently burned with no ERC-20 withdrawal executed and no reimbursement. This is a **ledger conservation bug**: ckETH supply is reduced without any corresponding Ethereum transaction or user benefit.

The burned ckETH amount equals `erc20_tx_fee` (up to ~0.000002 ETH at minimum gas prices), which is small but non-zero and represents a real loss to the user.

### Likelihood Explanation

The condition requires two simultaneous factors:
1. Ethereum gas prices near their historical minimum (base fee ≈ 0, priority fee ≈ 1.5 gwei × 65,000 gas = ~97,500,000,000,000 Wei ≈ 97.5 trillion Wei). In practice, the minimum `erc20_tx_fee` is well above `CKETH_LEDGER_TRANSACTION_FEE` under normal conditions. However, during periods of extremely low Ethereum network activity (e.g., post-merge periods with near-zero base fees), the computed fee could approach the threshold.
2. The user's ckERC20 burn fails — this is a realistic user error (forgetting to approve, or having insufficient ckERC20 balance).

The more impactful scenario is when `erc20_tx_fee < CKETH_LEDGER_TRANSACTION_FEE` (i.e., `checked_sub` returns `None` → `unwrap_or(Wei::ZERO)`), which would require gas prices even lower than the minimum enforced by `MIN_MAX_PRIORITY_FEE_PER_GAS` (1.5 gwei). This is currently prevented by the `MIN_MAX_PRIORITY_FEE_PER_GAS` floor in `estimate_transaction_fee`. The exact-equality case (`erc20_tx_fee == CKETH_LEDGER_TRANSACTION_FEE`) is the realistic trigger.

### Recommendation

Change the reimbursement guard from `> Wei::ZERO` to `>= Wei::ZERO` (or simply always schedule the reimbursement when the ckETH burn succeeded but the ckERC20 burn failed), so that even a zero-amount reimbursement is recorded. Alternatively, ensure the reimbursement is always scheduled regardless of amount when the ckETH burn has already been committed:

```rust
// Always schedule reimbursement when ckETH was already burned
let reimbursement_request = ReimbursementRequest {
    ledger_burn_index: cketh_ledger_burn_index,
    reimbursed_amount: reimbursed_amount.change_units(),
    to: cketh_account.owner,
    to_subaccount: cketh_account.subaccount.and_then(LedgerSubaccount::from_bytes),
    transaction_hash: None,
};
mutate_state(|s| {
    process_event(s, EventType::FailedErc20WithdrawalRequest(reimbursement_request));
});
```

Additionally, consider whether the penalty (`CKETH_LEDGER_TRANSACTION_FEE` deduction) should be capped so that `reimbursed_amount` is never negative or zero when the user's ckERC20 burn failure was due to a user error unrelated to the ckETH burn.

### Proof of Concept

1. Ethereum gas prices drop to near-minimum: `base_fee_per_gas ≈ 0`, `max_priority_fee_per_gas = 1_500_000_000` (1.5 gwei, the `MIN_MAX_PRIORITY_FEE_PER_GAS` floor).
2. `erc20_tx_fee = (2 * 0 + 1_500_000_000) * 65_000 = 97_500_000_000_000` Wei ≈ 97.5 trillion Wei. This is still above `CKETH_LEDGER_TRANSACTION_FEE` (2 trillion Wei), so the exact-zero case requires even lower gas.
3. More concretely: if `base_fee_per_gas` rounds such that `erc20_tx_fee` equals exactly `CKETH_LEDGER_TRANSACTION_FEE = 2_000_000_000_000` Wei, then `checked_sub` returns `Some(Wei::ZERO)`.
4. User calls `withdraw_erc20` without having approved the ckERC20 ledger (`icrc2_approve` not called).
5. ckETH burn succeeds (line 460): `cketh_ledger_burn_index` is obtained.
6. ckERC20 burn fails with `InsufficientAllowance` (line 506).
7. `reimbursed_amount = erc20_tx_fee.checked_sub(CKETH_LEDGER_TRANSACTION_FEE).unwrap_or(Wei::ZERO) = Wei::ZERO`.
8. `if reimbursed_amount > Wei::ZERO` is `false` — no reimbursement is scheduled.
9. User's ckETH is permanently burned. No ERC-20 withdrawal occurs. No refund. [4](#0-3) [1](#0-0)
