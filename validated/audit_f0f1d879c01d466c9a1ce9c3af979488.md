Audit Report

## Title
Mint-Before-Burn Ordering in CMC Allows Cycles to Be Minted Without ICP Being Burned if `burn_and_log` Fails - (File: rs/nns/cmc/src/main.rs)

## Summary

In `process_mint_cycles`, `process_top_up`, and `process_create_canister`, the CMC delivers the output (cycles deposited or canister created) before attempting to burn the backing ICP via `burn_and_log`. The `burn_and_log` function is explicitly designed to swallow all errors and never block the success path. If the ICP ledger's `send_pb` call fails after a successful mint/deposit, cycles are permanently credited to the user while the ICP remains unburned in the CMC's subaccount, and the block notification is permanently consumed preventing any retry. This violates the ICP ↔ cycles conservation invariant.

## Finding Description

In `process_mint_cycles` ( [1](#0-0) ), `do_mint_cycles` is called first, crediting the user's cycles ledger account via the cycles ledger `deposit` endpoint. Only on success does `burn_and_log` execute. The same pattern exists in `process_top_up` ( [2](#0-1) ) and `process_create_canister` ( [3](#0-2) ).

The `burn_and_log` function is explicitly documented to never propagate errors ( [4](#0-3) ). It contains an early-return guard that silently skips the burn if `amount < DEFAULT_TRANSFER_FEE` ( [5](#0-4) ), and on `send_pb` failure it only logs the error ( [6](#0-5) ).

After `process_mint_cycles` returns `Ok(...)`, the block is recorded as `NotifiedMint(Ok(...))` in `blocks_notified`. The `is_transient_error` guard only removes the entry for transient errors — a successful result is never removed, permanently consuming the notification and preventing any retry ( [7](#0-6) ).

The `do_mint_cycles` function calls the cycles ledger's `deposit` endpoint with payment, irreversibly crediting the user before the burn is attempted ( [8](#0-7) ).

The result when `burn_and_log` fails after a successful `do_mint_cycles`:
- Cycles are credited to the user's cycles ledger account (irreversible).
- The ICP remains in the CMC's per-user subaccount (not burned, not refunded).
- The block notification is permanently consumed — no retry is possible.
- The ICP total supply is not reduced as required by the ICP ↔ cycles conservation invariant.

## Impact Explanation

This constitutes illegal minting of cycles without a corresponding ICP burn. The attacker receives cycles at zero net cost: the ICP they deposited remains stranded in the CMC's subaccount rather than being destroyed, inflating the effective ICP supply relative to the cycles in circulation. This directly matches the allowed impact: **Theft, permanent loss, illegal minting, or protocol insolvency involving ICP/Cycles**. Severity is **High** given the constraint that the ICP ledger must be temporarily unavailable, limiting the attack window but not eliminating it.

## Likelihood Explanation

The `burn_and_log` failure requires the ICP ledger's `send_pb` call to fail. NNS upgrade proposals for the ICP ledger are public and observable on-chain. During a ledger canister upgrade, there is a brief window (seconds) where calls to the ledger fail. An unprivileged attacker can:
1. Pre-fund the CMC subaccount in advance.
2. Monitor NNS proposals for a scheduled ICP ledger upgrade.
3. Submit `notify_mint_cycles` timed to the upgrade window.

The cycles ledger is a separate canister and is not upgraded simultaneously, so `do_mint_cycles` can succeed while `burn_and_log` fails. The attack window is narrow but deterministically predictable from public NNS data, making it feasible for a motivated attacker. Accidental occurrence during unplanned ledger disruptions is also non-negligible given CMC operation volume.

## Recommendation

Reverse the order of operations to follow Checks-Effects-Interactions:

1. In `process_mint_cycles`: perform the ICP burn (failable) **before** calling `do_mint_cycles`.
2. In `process_top_up`: perform the ICP burn **before** calling `deposit_cycles`.
3. In `process_create_canister`: perform the ICP burn **before** calling `do_create_canister`.

If the concern is that a failable burn enables infinite retries, use a two-phase commit: record a "burn-in-progress" state with the burn block index before minting, and only proceed with minting after confirming the burn succeeded. The ckBTC minter's `reimburse_withdrawals` demonstrates the correct pattern using `scopeguard` to quarantine on panic and prevent double-minting.

## Proof of Concept

1. Attacker transfers `N` ICP to the CMC subaccount keyed by their principal with memo `MEMO_MINT_CYCLES`.
2. Attacker monitors NNS for a scheduled ICP ledger upgrade proposal.
3. Attacker submits `notify_mint_cycles` timed to the ICP ledger upgrade window.
4. CMC calls `do_mint_cycles` → cycles ledger `deposit` succeeds → attacker's cycles ledger balance increases.
5. CMC calls `burn_and_log` → `send_pb` to ICP ledger fails (ledger upgrading) → error is logged and swallowed.
6. `notify_mint_cycles` returns `Ok(NotifyMintCyclesSuccess { ... })`.
7. Block is marked `NotifiedMint(Ok(...))` — permanently consumed.
8. Result: attacker holds cycles; `N` ICP remains in CMC subaccount, not burned; ICP total supply is inflated by `N`.

A deterministic integration test can reproduce this by: deploying CMC + ICP ledger + cycles ledger in a PocketIC environment, funding the CMC subaccount, stopping the ICP ledger canister to simulate an upgrade, calling `notify_mint_cycles`, and asserting that the cycles ledger balance increased while the ICP ledger burn count remains zero and the CMC subaccount balance is unchanged.
