### Title
Hardcoded `E8` Divisor in SNS Token Valuation Ignores Actual ICRC-1 Token Decimals - (`rs/sns/governance/token_valuation/src/lib.rs`)

---

### Summary

`try_get_balance_valuation_factors` in `rs/sns/governance/token_valuation/src/lib.rs` always divides the raw ICRC-1 balance by the hardcoded constant `E8` (10^8) to produce a "tokens" quantity, regardless of the actual `icrc1_decimals` of the SNS ledger. The ICRC-1 standard allows any `decimals` value (0–255). When an SNS token has decimals ≠ 8, the resulting `ValuationFactors.tokens` is wrong by a factor of `10^(8 - actual_decimals)`, which propagates into `to_xdr()` and ultimately into the `ProposalsAmountTotalUpperBound` that guards SNS treasury transfers and token minting.

---

### Finding Description

In `rs/sns/governance/token_valuation/src/lib.rs` at line 177–181:

```rust
let tokens = Decimal::from(u128::try_from(balance_of_response.0).map_err(|err| {
    ValuationError::new_arithmetic(format!(
        "Balance of {account:?} does not fit in u128: {err:?}"
    ))
})?) / Decimal::from(E8);   // ← hardcoded 10^8
```

`balance_of_response.0` is the raw value returned by `icrc1_balance_of`, which is denominated in the token's smallest unit. The correct divisor is `10^(token.icrc1_decimals())`, not the fixed `E8 = 10^8`.

The ICRC-1 ledger explicitly supports configurable decimals:

```
// rs/ledger_suite/icrc1/ledger/ledger.did line 104
decimals : opt nat8;
```

and the ledger stores and exposes whatever value was set at init:

```rust
// rs/ledger_suite/icrc1/ledger/src/lib.rs line 715
decimals: decimals.unwrap_or_else(default_decimals),
```

The erroneous `tokens` value flows into `ValuationFactors::to_xdr()`:

```rust
pub fn to_xdr(&self) -> Decimal {
    tokens * icps_per_token * xdrs_per_icp   // tokens is wrong
}
```

`to_xdr()` is consumed by `ProposalsAmountTotalUpperBound::in_tokens()` to decide which regime applies (NoLimit / Fraction / Xdr cap) and, in the `NoLimit` and `Fraction` branches, to compute the actual token limit:

```rust
Self::NoLimit => balance_tokens,                          // wrong
Self::Fraction(fraction) => balance_tokens * fraction,    // wrong
Self::Xdr(max_xdr) => max_xdr * tokens_per_xdr,          // correct but wrong regime
```

This limit is enforced for every `TransferSnsTreasuryFunds` and `MintSnsTokens` governance proposal.

---

### Impact Explanation

**SNS token with decimals < 8 (e.g., 6):**
`balance_tokens` is underestimated by `10^(8-6) = 100×`. A treasury that is actually "large" (> 1,200,000 XDR) appears "small" (< 100,000 XDR), so `NoLimit` applies and the returned limit is `balance_tokens_wrong = actual_balance / 100`. When `actual_balance / 100 > 300,000 / xdrs_per_token` (i.e., actual treasury > 30 M XDR), the limit is **more permissive** than the intended 300,000 XDR/7-day cap, allowing a governance majority to drain the treasury faster than the protocol intends.

**SNS token with decimals > 8 (e.g., 18):**
`balance_tokens` is overestimated by `10^(18-8) = 10^10×`. A treasury that is actually "small" (< 100,000 XDR) appears "large" (> 1,200,000 XDR), so the `Xdr(300,000)` cap applies. The returned limit is `300,000 / xdrs_per_token`. If the token price is very low (e.g., 10^-6 XDR/token), this evaluates to `3×10^11` tokens — far exceeding the actual treasury — making the limit effectively **uncapped** while the correct regime would have been `NoLimit` returning only the actual balance. Conversely, for higher-priced tokens the limit becomes more restrictive than intended, blocking legitimate governance.

In both cases the `ProposalsAmountTotalUpperBound` safety mechanism — the only protocol-level guard against rapid treasury drain — produces incorrect results.

---

### Likelihood Explanation

The ICRC-1 standard and the IC SNS framework do not enforce 8 decimals for SNS tokens. Any SNS launched with a non-8-decimal token (e.g., a ckETH-style 18-decimal token, or a 6-decimal stablecoin) triggers this bug automatically. The attacker-controlled entry path is:

1. Submit a `CreateServiceNervousSystem` NNS proposal that initialises the SNS ledger with `decimals ≠ 8`.
2. Once the SNS is live, accumulate voting power (or exploit an existing SNS whose token happens to have non-8 decimals).
3. Submit a `TransferSnsTreasuryFunds` or `MintSnsTokens` proposal. The governance canister calls `try_get_sns_token_balance_valuation`, which calls `try_get_balance_valuation_factors`, which computes the wrong limit.
4. The proposal passes the (incorrect) limit check and executes.

No privileged access, no key compromise, and no threshold attack is required — only the ability to pass an SNS governance proposal, which is the normal operation of any SNS.

---

### Recommendation

Replace the hardcoded `E8` with the actual token decimals fetched from the ledger via `icrc1_decimals`:

```rust
// Fetch decimals alongside balance
let decimals_request = icrc1_client.icrc1_decimals();
let (balance_of_response, decimals_response, icps_per_token_response, xdrs_per_icp_response) =
    join!(balance_of_request, decimals_request, icps_per_token_request, xdrs_per_icp_request);

let decimals = decimals_response.map_err(|err| {
    ValuationError::new_external(format!("Unable to obtain token decimals: {err:?}"))
})?;
let divisor = Decimal::from(10_u128.pow(decimals as u32));

let tokens = Decimal::from(u128::try_from(balance_of_response.0)?) / divisor;
```

The `Icrc1Client` trait should expose `icrc1_decimals()` (the method already exists on the real ICRC-1 ledger interface).

### Proof of Concept

Consider an SNS token with `decimals = 6` and a treasury of 10,000,000 SNS tokens (= 10^13 smallest units), where 1 SNS token = 5 XDR.

**Correct calculation:**
- `tokens = 10^13 / 10^6 = 10,000,000`
- `to_xdr() = 10,000,000 × 5 = 50,000,000 XDR` → large treasury → `Xdr(300,000)` cap
- `in_tokens = 300,000 / 5 = 60,000 SNS tokens` per 7 days

**Buggy calculation (current code):**
- `tokens = 10^13 / 10^8 = 100,000`
- `to_xdr() = 100,000 × 5 = 500,000 XDR` → still large → `Xdr(300,000)` cap
- `in_tokens = 300,000 / 5 = 60,000 SNS tokens` per 7 days ← same result here

Now with `decimals = 6` and treasury = 10,000,000,000 SNS tokens (= 10^15 smallest units), 1 SNS token = 0.001 XDR:

**Correct:**
- `tokens = 10^15 / 10^6 = 10^9`
- `to_xdr() = 10^9 × 0.001 = 10^6 XDR` → large → `Xdr(300,000)` cap
- `in_tokens = 300,000 / 0.001 = 3×10^8` tokens per 7 days

**Buggy:**
- `tokens = 10^15 / 10^8 = 10^7`
- `to_xdr() = 10^7 × 0.001 = 10,000 XDR` → **small** (< 100,000 XDR) → `NoLimit`
- `in_tokens = balance_tokens = 10^7` tokens per 7 days

The correct limit is 3×10^8 tokens; the buggy limit is 10^7 tokens — 30× more restrictive, blocking legitimate governance. Reversing the direction (18-decimal token, low price), the limit becomes orders of magnitude more permissive, defeating the treasury-drain protection entirely.

        Self::clamp_xdrs_per_icp(&mut valuation);

        let ValuationFactors {
            tokens: balance_tokens,
            icps_per_token,
            xdrs_per_icp,
        } = valuation.valuation_factors;

        let self_ = Self::from_valuation_xdr(valuation.to_xdr());
        let result_tokens = match self_ {
            Self::NoLimit => balance_tokens,

            Self::Fraction(fraction) => balance_tokens
                .checked_mul(fraction)
                // Overflow should not be possible, since fraction is supposed to be at most 1.0.
                .ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "Unable to perform {balance_tokens} * {fraction}.",
                    ))
                })?,

            Self::Xdr(max_xdr) => {
                let xdrs_per_token = xdrs_per_icp.checked_mul(icps_per_token).ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "XDRs per token could not be calculated from valuation: {valuation:?}"
                    ))

                // Calculate the inverse conversion rate.

                        "It appears that the tokens have zero value in XDR. valuation = {valuation:?}"
                    )));
