### Title
Missing Rejection of Attached NEAR Deposits in `execute_scheduled` Causes Permanent Fund Freeze — (File: `etc/xcc-router/src/lib.rs`)

### Summary
The `execute_scheduled` function in the XCC Router contract is marked `#[payable]`, meaning it silently accepts attached NEAR tokens from any caller. However, the function never uses, forwards, or refunds the attached deposit. Because the router exposes no general withdrawal mechanism, any NEAR tokens attached to a call of `execute_scheduled` are permanently locked in the router sub-account with no recovery path for the depositor.

### Finding Description
The XCC Router is a NEAR contract deployed as a sub-account for each Aurora user address. Its `execute_scheduled` function is intentionally callable by anyone (not just the Aurora engine parent): [1](#0-0) 

The function is decorated with `#[payable]`, which in the NEAR SDK means the runtime will not panic if NEAR tokens are attached to the call. However, the function body never calls `env::attached_deposit()`, never forwards the deposit into the scheduled promise, and never refunds it to the caller: [2](#0-1) 

The only outbound NEAR transfer in the entire contract is `send_refund`, which unconditionally sends a hardcoded `REFUND_AMOUNT` (2 NEAR) to the parent account and is restricted to the parent caller: [3](#0-2) 

There is no function that allows recovery of an arbitrary NEAR balance, so any deposit attached to `execute_scheduled` is irrecoverable.

### Impact Explanation
Any NEAR tokens attached to a call of `execute_scheduled` are transferred to the router sub-account's balance and cannot be retrieved by the depositor, the Aurora engine, or any other party. This constitutes **permanent freezing of funds**. The router has no `withdraw`, `refund`, or general balance-recovery method. The only transfer-out path (`send_refund`) is fixed at 2 NEAR and restricted to the parent.

### Likelihood Explanation
`execute_scheduled` is explicitly designed to be callable by any unprivileged account: [4](#0-3) 

Because the function is `#[payable]` and publicly reachable, any external caller — including an EVM contract interacting with the XCC precompile flow, a NEAR wallet user, or a bot — can attach NEAR to the call. The NEAR SDK does not warn callers that the deposit will be silently consumed. The likelihood is low-to-medium: it requires an explicit deposit, but the open access surface and the absence of any on-chain warning make accidental or mistaken deposits realistic.

### Recommendation
Remove `#[payable]` from `execute_scheduled` if attached deposits are not required. If NEAR must sometimes be forwarded into the scheduled promise, add explicit handling:

```rust
// Option A: reject any attached deposit
pub fn execute_scheduled(&mut self, nonce: U64) { ... }  // remove #[payable]

// Option B: forward the deposit into the promise
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let deposit = env::attached_deposit();
    // pass deposit into the promise or refund it
    ...
}
```

Apply the same review to `deploy_upgrade`, which is also `#[payable]` but does not explicitly account for the attached deposit beyond the batch action it creates. [5](#0-4) 

### Proof of Concept
1. A router sub-account `user.aurora` is deployed by the Aurora engine.
2. A scheduled promise exists at nonce `N` in the router's storage.
3. An unprivileged attacker (or a mistaken user) calls:
   ```
   near call user.aurora execute_scheduled '{"nonce": "0"}' --deposit 10
   ```
4. The NEAR SDK accepts the 10 NEAR deposit (function is `#[payable]`).
5. The function removes the promise from storage and executes it — the 10 NEAR deposit is never touched.
6. The 10 NEAR now sits in `user.aurora`'s balance.
7. No function in the router allows the depositor or any third party to recover those 10 NEAR. `send_refund` only transfers the fixed 2 NEAR `REFUND_AMOUNT` to the parent and is restricted to the parent caller.
8. The 10 NEAR is permanently frozen. [6](#0-5)
