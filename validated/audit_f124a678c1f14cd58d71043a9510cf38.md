Audit Report

## Title
ckBTC Minter `reimburse_canceled_requests` Hard `assert!` Causes Deterministic Panic Permanently DoSing Withdrawal Processing — (`File: rs/bitcoin/ckbtc/minter/src/lib.rs`)

## Summary
The `reimburse_canceled_requests` function in `rs/bitcoin/ckbtc/minter/src/lib.rs` contains a hard `assert!` at line 302 that panics if the computed per-request reimbursement fee exceeds `state.retrieve_btc_min_amount`. This assert is reachable from `submit_pending_requests` whenever `build_unsigned_transaction` returns `BuildTxError::InvalidTransaction`. Because the minter's periodic task re-enters this code path on every tick, the panic is deterministic and permanent, halting all ckBTC withdrawal processing until an NNS upgrade is deployed. PR #5713 fixed a structurally identical panic in `resubmit_transactions` but left this assert in `reimburse_canceled_requests` unfixed.

## Finding Description

**Root cause:** In `rs/bitcoin/ckbtc/minter/src/lib.rs` at lines 292–329, `reimburse_canceled_requests` computes per-request fees via `distribute(total_fee, requests.len())` and then asserts:

```rust
assert!(
    fees[0] <= state.retrieve_btc_min_amount,
    "BUG: fees {fees:?} for {} withdrawal requests are larger than `retrieve_btc_min_amount` {}",
    requests.len(),
    state.retrieve_btc_min_amount
);
``` [1](#0-0) 

This function is called from `submit_pending_requests` at line 409 when `build_unsigned_transaction` returns `BuildTxError::InvalidTransaction` (e.g., `TooManyInputs`):

```rust
let reimbursement_fee = fee_estimator
    .reimbursement_fee_for_pending_withdrawal_requests(batch.len() as u64);
reimburse_canceled_requests(s, batch, reason, reimbursement_fee, runtime);
``` [2](#0-1) 

The reimbursement fee is computed as `batch.len() * COST_OF_ONE_BILLION_CYCLES` (satoshis per request): [3](#0-2) 

If `COST_OF_ONE_BILLION_CYCLES` (the per-request fee in satoshis, derived from the ICP/BTC exchange rate) exceeds `retrieve_btc_min_amount` (configurable, as low as 50,000 satoshis), the assert fires. Since `submit_pending_requests` is called on every periodic task tick and the state that triggers `BuildTxError::InvalidTransaction` (e.g., a UTXO set with too many small inputs) persists across ticks, the panic is deterministic and repeating.

**Why existing checks fail:** There is no guard before calling `reimburse_canceled_requests` to verify that the reimbursement fee is within bounds. The assert itself is the only check, and it panics rather than handling the error gracefully.

**Distinction from PR #5713:** The June 2025 upgrade (`minter_upgrade_2025_06_27.md`) confirms PR #5713 fixed a deterministic panic in `resubmit_transactions` (the path for already-submitted stuck transactions): [4](#0-3) 

The `resubmit_transactions` function now handles `BuildTxError::InvalidTransaction` gracefully at lines 900–937 by constructing a cancellation transaction instead of panicking. [5](#0-4) 

However, the hard `assert!` in `reimburse_canceled_requests` — called from the separate `submit_pending_requests` path for new (not yet submitted) transactions — was not addressed by PR #5713 and remains in the current codebase. [1](#0-0) 

## Impact Explanation

A deterministic panic in the minter's periodic task permanently halts all ckBTC → BTC withdrawal processing. All pending `retrieve_btc` requests remain stuck indefinitely. Users who have already burned ckBTC cannot receive their BTC until an NNS governance proposal upgrades the minter canister — a process taking multiple days. This constitutes a significant Chain Fusion / ck-token DoS with concrete, direct financial harm to users holding pending withdrawal requests, matching the **High** impact class: "Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm" ($2,000–$10,000).

## Likelihood Explanation

The trigger condition requires two concurrent factors: (1) `build_unsigned_transaction` returning `BuildTxError::InvalidTransaction` for a new batch (e.g., `TooManyInputs` due to UTXO fragmentation), and (2) `COST_OF_ONE_BILLION_CYCLES` exceeding `retrieve_btc_min_amount`. The June 2025 mainnet incident confirms this class of vulnerability is not theoretical — anomalous Bitcoin fee data caused a structurally identical deterministic panic in production. The `retrieve_btc_min_amount` is configurable and has been reduced over time, lowering the threshold at which the assert fires. [6](#0-5) 

The `submit_pending_requests` → `BuildTxError::InvalidTransaction` → `reimburse_canceled_requests` path is reachable without any special privileges: it is triggered by normal UTXO set conditions (fragmentation) combined with exchange-rate-driven fee dynamics, both of which are external inputs the minter cannot control.

## Recommendation

Replace the hard `assert!` in `reimburse_canceled_requests` with a graceful error path. If `fees[0] > state.retrieve_btc_min_amount`, the minter should log the anomaly and either: (a) cap the reimbursement fee at `retrieve_btc_min_amount`, (b) skip reimbursement and place requests in a recoverable error state, or (c) reimburse the full request amount with zero fee deduction. The assert should never be a panic boundary in a periodic task. Additionally, the minimum fee per vbyte floor introduced by PR #5742 reduces the upstream likelihood but does not eliminate the assert's reachability via UTXO fragmentation.

## Proof of Concept

1. Configure a local PocketIC or replica with `retrieve_btc_min_amount` set to 50,000 satoshis.
2. Fragment the minter's UTXO set into many small UTXOs (each below the per-request amount) so that `build_unsigned_transaction` for a batch returns `BuildTxError::InvalidTransaction(TooManyInputs)`.
3. Set `COST_OF_ONE_BILLION_CYCLES` (or mock the fee estimator) such that `reimbursement_fee_for_pending_withdrawal_requests(1) > 50_000`.
4. Trigger the minter's periodic task (`submit_pending_requests`).
5. Observe the canister trap at `rs/bitcoin/ckbtc/minter/src/lib.rs:302` with message `"BUG: fees [...] are larger than retrieve_btc_min_amount"`.
6. Trigger the periodic task again — observe the same trap, confirming the permanent DoS.

Alternatively, write a unit test in `rs/bitcoin/ckbtc/minter/tests/tests.rs` that directly calls `reimburse_canceled_requests` with a `total_fee` exceeding `retrieve_btc_min_amount` and asserts the function panics — confirming the assert fires deterministically. [7](#0-6)

fn reimburse_canceled_requests<R: CanisterRuntime>(
    state: &mut state::CkBtcMinterState,
    requests: BTreeSet<state::RetrieveBtcRequest>,
    reason: WithdrawalReimbursementReason,
    total_fee: u64,
    runtime: &R,
) {
    assert!(!requests.is_empty());
    let fees = distribute(total_fee, requests.len() as u64);
    // This assertion makes sure the fee is smaller than each request amount
    assert!(
        fees[0] <= state.retrieve_btc_min_amount,
        "BUG: fees {fees:?} for {} withdrawal requests are larger than `retrieve_btc_min_amount` {}",
        requests.len(),
        state.retrieve_btc_min_amount
    );
    for (request, fee) in requests.into_iter().zip(fees.into_iter()) {
        if let Some(account) = request.reimbursement_account {
            let amount = request.amount.saturating_sub(fee);
            if amount > 0 {
                state::audit::reimburse_withdrawal(
                    state,
                    request.block_index,
                    amount,
                    account,
                    reason.clone(),
                    runtime,
                );
            }
        } else {
            log!(
                Priority::Info,
                "[reimburse_canceled_requests]: account is not found for retrieve_btc request ({:?})",
                request
            );
        }
    }
}
