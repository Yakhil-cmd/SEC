### Title
Burn-on-Transfer ERC-20 Deposit Over-Mints ckERC20 Due to Event Amount vs. Actual Received Amount Discrepancy - (File: `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`, `rs/ethereum/cketh/minter/ERC20DepositHelper.sol`, `rs/ethereum/cketh/minter/src/deposit.rs`)

---

### Summary

Both ckERC20 deposit helper smart contracts emit the caller-supplied `amount` parameter in the deposit event rather than the actual token balance received by the minter address. The IC ckETH minter unconditionally trusts this event value and mints an equal quantity of ckERC20 tokens. For any deflationary or burn-on-transfer ERC-20 token, the minter mints more ckERC20 than it holds in ERC-20 collateral, breaking the 1:1 peg and causing subsequent withdrawals to fail on Ethereum.

---

### Finding Description

The ERC-20 deposit flow works as follows:

1. A user calls `depositErc20(erc20Address, amount, principal, subaccount)` on the helper contract.
2. The helper calls `safeTransferFrom(msg.sender, minterAddress, amount)` on the ERC-20 contract.
3. The helper emits `ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount)` — using the **input parameter `amount`**, not the actual balance delta received.

In `DepositHelperWithSubaccount.sol`:

```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);
``` [1](#0-0) 

In `ERC20DepositHelper.sol`:

```solidity
erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
emit ReceivedErc20(erc20_address, msg.sender, amount, principal);
``` [2](#0-1) 

The IC minter scrapes these logs and parses the `value` field directly from the event data:

```rust
let [value_bytes] = parse_hex_into_32_byte_words(entry.data, event_source)?;
// ...
value: Erc20Value::from_be_bytes(value_bytes),
``` [3](#0-2) 

The minter then mints exactly `event.value()` ckERC20 tokens to the beneficiary with no balance-before/balance-after verification:

```rust
amount: event.value(),
``` [4](#0-3) 

For a burn-on-transfer ERC-20 token (e.g., one that burns 1% on every transfer), if a user deposits `1000` tokens:
- The minter's actual ERC-20 balance increases by `990`.
- The event emits `amount = 1000`.
- The minter mints `1000` ckERC20 tokens.
- The minter is now `10` tokens short of full collateralization.

This discrepancy accumulates with every deposit, and when users attempt to withdraw ckERC20 back to ERC-20, the minter's Ethereum transaction will fail because it lacks sufficient ERC-20 balance to fulfill the transfer.

---

### Impact Explanation

**Ledger conservation bug / chain-fusion mint/burn/replay bug.**

- The ckERC20 total supply on the IC ledger exceeds the actual ERC-20 collateral held by the minter's Ethereum address.
- Withdrawals (`withdraw_erc20`) will fail on Ethereum for the over-minted portion, as the minter cannot send tokens it does not hold.
- Users who deposited legitimately may be unable to withdraw their full balance, as the minter's ERC-20 reserves are insufficient.
- The minter's internal `erc20_balances` state (tracking expected holdings) will diverge from the actual on-chain balance, corrupting the minter's accounting permanently.

The `erc20_balances` state is updated from the accepted deposit event value: [5](#0-4) 

---

### Likelihood Explanation

**Low-to-Medium.** The minter only processes deposit events for tokens explicitly added to `ckerc20_tokens` via NNS governance proposal. Currently supported tokens (USDC, USDT, etc.) are not burn-on-transfer. However:

- The governance process for adding new ckERC20 tokens is open and permissionless in terms of proposal submission.
- The minter code contains no guard against burn-on-transfer tokens being added.
- If any burn-on-transfer token is ever added as a supported ckERC20 token, every deposit by any unprivileged user will silently over-mint, with no error or warning.
- The attacker-controlled entry path is simply calling `depositErc20` on the helper contract with a supported burn-on-transfer ERC-20 token — no special privileges required.

---

### Recommendation

1. **In the Solidity helper contracts**: Record the minter's ERC-20 balance before and after the `safeTransferFrom` call, and emit the actual received amount (balance delta) in the event rather than the input `amount` parameter:

```solidity
uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
```

2. **In the IC minter governance process**: Add an explicit check or documentation requirement that burn-on-transfer tokens must not be added as supported ckERC20 tokens until the helper contracts are updated.

---

### Proof of Concept

1. A burn-on-transfer ERC-20 token `BurnToken` (burns 5% on every transfer) is added as a supported ckERC20 token via NNS proposal.
2. Attacker (or any user) calls `depositErc20(BurnToken, 1_000_000, principal, subaccount)` on `CkDeposit`.
3. `safeTransferFrom` executes: minter receives `950_000` tokens (5% burned).
4. Event emits: `ReceivedEthOrErc20(BurnToken, sender, 1_000_000, principal, subaccount)`.
5. IC minter scrapes the log, reads `value = 1_000_000`, mints `1_000_000` ckBurnToken to the user.
6. Minter's actual ERC-20 balance: `950_000`. ckBurnToken total supply: `1_000_000`.
7. User calls `withdraw_erc20` for `1_000_000` ckBurnToken.
8. Minter burns `1_000_000` ckBurnToken on the IC ledger and submits an Ethereum transaction to transfer `1_000_000` BurnToken to the user.
9. The Ethereum transaction reverts — minter only holds `950_000` BurnToken.
10. The minter reimburses the ckBurnToken burn, but the accounting is now permanently corrupted for all subsequent depositors. [6](#0-5) [4](#0-3) [7](#0-6)
