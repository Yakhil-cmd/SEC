Audit Report

## Title
Wasm Trap in `canister_status` Callback Permanently Leaks Slot Due to IC State Rollback — (`rs/nns/handlers/root/impl/canister/canister.rs`)

## Summary
The NNS root canister's public `canister_status` update endpoint calls `.unwrap()` on a management canister `Result` after the `SlotLoan` RAII guard has already been dropped within the same callback execution. Because the IC rolls back all heap mutations when a response callback traps, the slot restoration performed by `SlotLoan::drop` is undone. An unprivileged caller can exhaust all 167 slots by repeatedly triggering this trap, permanently disabling the public `canister_status` endpoint for all non-NNS principals until the canister is upgraded.

## Finding Description

**No access control on `canister_status`:** [1](#0-0) 
The endpoint is a plain `#[update]` with no `check_caller_is_*` guard. Any principal can invoke it with any `CanisterIdRecord`.

**Slot borrow in `LimitedOutstandingCallsManagementCanisterClient::canister_status`:** [2](#0-1) 
`_loan` is a local variable scoped to this async function. In Rust, it is dropped when the function returns — i.e., after `self.inner.canister_status(...).await` completes and before the `Result` is returned to the caller in `canister.rs`.

**`SlotLoan::drop` restores the slot:** [3](#0-2) 

**`.unwrap()` traps after `_loan` has already been dropped:** [4](#0-3) 

The execution sequence in Message 2 (the response callback):
1. `LimitedOutstandingCallsManagementCanisterClient::canister_status` resumes; inner call returns `Err(...)`.
2. `_loan` goes out of scope → `available_slot_count` incremented from N-1 to N (heap mutation).
3. `Err` propagates back to `canister.rs::canister_status`.
4. `.map(CanisterStatusResult::from)` → still `Err`.
5. `.unwrap()` panics → **Wasm TRAP**.
6. IC rolls back Message 2 heap → `available_slot_count` reverts to N-1. Step 2's increment is undone. Slot permanently leaked.

This is confirmed by the codebase itself: a comment in the interface client explicitly acknowledges that `canister_status` uses a trap to propagate errors: [5](#0-4) 

**Slot capacity and exhaustion check:** [6](#0-5) [7](#0-6) 

Once `available_slot_count == 0`, all non-VIP calls return `Err(SysTransient, "Unavailable. Maybe, try again later?")` immediately.

**VIP bypass (NNS canisters unaffected):** [8](#0-7) 
VIP callers consume 0 slots, so governance and SNS-W operations are unaffected. Only the public `canister_status` endpoint is impacted.

## Impact Explanation
After 167 trapped callbacks, `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` reaches 0 and the public `canister_status` endpoint returns `SysTransient` errors immediately for all non-NNS principals, without making any inter-canister call. The endpoint is permanently non-functional until the canister is upgraded via a governance proposal. This constitutes an **application-level DoS on a public NNS endpoint** — matching the High ($2,000–$10,000) impact class: "Application/platform-level DoS... or subnet availability impact not based on raw volumetric DDoS" and "Significant NNS... security impact with concrete user or protocol harm."

## Likelihood Explanation
- The `canister_status` endpoint is public with no access control.
- Triggering a management canister rejection is trivial: pass any canister ID that NNS root does not control (e.g., any user-owned canister). The management canister rejects the call because NNS root is not a controller.
- Only 167 update calls are required. Each costs only standard ingress fees.
- No privileged access, key material, or subnet-majority corruption is required.
- The existing test `test_canister_status_call_tracking` confirms the failure path is reachable and that NNS root is not a controller of arbitrary canisters. [9](#0-8) 

## Recommendation
Replace `.unwrap()` with proper error propagation so the endpoint returns a rejection to the caller instead of trapping. This ensures `SlotLoan::drop` and the subsequent `.unwrap()` occur within the same atomic message execution, or that no trap occurs after the slot is restored:

```rust
// Instead of:
canister_status_response.unwrap()

// Use (CDK trap encodes a rejection without rolling back committed state):
canister_status_response.map_err(|(_, msg)| ic_cdk::trap(&msg))
// or return Result<CanisterStatusResult, String> and let the CDK encode the rejection
```

Alternatively, restructure so that `.unwrap()` is called before `_loan` is dropped (i.e., inside `LimitedOutstandingCallsManagementCanisterClient::canister_status`), but the correct fix is to never trap after releasing borrowed resources.

## Proof of Concept
1. Obtain any canister ID that NNS root does not control (e.g., a freshly created user canister).
2. Send 167 update calls to NNS root's `canister_status` with that canister ID from any non-NNS principal.
3. Each call: management canister rejects (NNS root is not a controller) → `Err` returned → `_loan` dropped (slot restored in heap) → `.unwrap()` traps → IC rolls back callback heap → slot leaked.
4. On the 168th call from any non-VIP principal, observe `Err(SysTransient, "Unavailable. Maybe, try again later?")` returned immediately.

A deterministic integration test using `StateMachine` (as in `test_open_calls_tracking.rs`) can reproduce this by looping 167 `canister_status` calls against a non-controlled canister ID and asserting the 168th call fails with the exhaustion error.
