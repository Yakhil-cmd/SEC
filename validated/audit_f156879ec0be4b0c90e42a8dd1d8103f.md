Audit Report

## Title
Hot Keys Not Cleared When NNS Neuron Is Spawned to a Different Controller - (File: `rs/nns/governance/src/governance.rs`)

## Summary
`spawn_neuron` unconditionally copies the parent neuron's `hot_keys` list to the child neuron even when `spawn.new_controller` designates a completely different principal as the child's controller. The inherited hot keys remain authorized to vote on NNS proposals and change followees on the new controller's neuron without the new controller's knowledge or consent.

## Finding Description
In `rs/nns/governance/src/governance.rs`, `spawn_neuron` resolves `child_controller` at lines 2651–2655: when `spawn.new_controller` is `Some(p)`, `child_controller = p`; otherwise it falls back to `parent_neuron.controller()`. No subsequent check compares `child_controller` against `parent_neuron.controller()`. The child neuron is then built at lines 2704–2718 with `.with_hot_keys(parent_neuron.hot_keys.clone())` unconditionally, regardless of whether the controller changed. There is no guard that clears or filters `hot_keys` when `child_controller != parent_neuron.controller()`.

`is_authorized_to_vote` in `rs/nns/governance/src/neuron/types.rs` (lines 243–244) accepts either the controller or any hot key. `register_vote` (lines 5594–5595) and `follow` (lines 5725–5748) both gate on `is_authorized_to_vote`, so any inherited hot key immediately has the ability to vote on proposals and change followees (except the `NeuronManagement` topic, which requires the controller). Existing guards only check that the caller is the parent's controller before spawning; they do not sanitize the child's hot key list.

The spawned neuron is created in dissolving state with `when_dissolved_timestamp_seconds` set to `now + neuron_spawn_dissolve_delay_seconds`. After the spawn delay elapses, the neuron is dissolved (dissolve delay = 0), below the minimum required to vote. However, once the new controller increases the dissolve delay — a standard, expected action to use the neuron for governance — the inherited hot keys immediately gain voting power over the new controller's neuron.

## Impact Explanation
This is a **High** severity finding matching: *"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."*

Once the new controller increases the dissolve delay, the spawning party's hot keys can: (1) cast votes on NNS governance proposals using the new controller's neuron's voting power, against the new controller's wishes; (2) redirect the new controller's liquid-democracy delegation by changing followees on all non-`NeuronManagement` topics. The new controller has no notification mechanism and must proactively query `get_full_neuron` and issue `RemoveHotKey` commands to remediate.

## Likelihood Explanation
The attack path is reachable by any unprivileged ingress sender who holds a neuron with accumulated maturity and at least one registered hot key. No privileged role, key compromise, or social engineering is required. `spawn_neuron` is a standard, publicly documented NNS operation exposed via the `manage_neuron` update endpoint. The constraint is that the new controller must increase the dissolve delay for voting power to materialize, which is a normal and expected governance action.

## Recommendation
When `spawn.new_controller` is set to a principal different from `parent_neuron.controller()`, initialize the child neuron with an empty hot keys list:

```rust
let inherited_hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};
// ...
.with_hot_keys(inherited_hot_keys)
```

The new controller can add their own hot keys after taking ownership. The same logic should be reviewed for `split_neuron`, though impact there is lower because the child always shares the parent's controller.

## Proof of Concept
1. Alice controls neuron `N` with hot key `H` and sufficient maturity.
2. Alice calls `manage_neuron` with `Command::Spawn(Spawn { new_controller: Some(Bob), nonce: Some(42), percentage_to_spawn: Some(100) })`.
3. Governance creates child neuron `C` with `controller = Bob` and `hot_keys = [H]` (confirmed at line 2714).
4. After the spawn delay, `C` is dissolved with 0 dissolve delay.
5. Bob calls `manage_neuron` on `C` with `Command::Configure(IncreaseDissolveDelay { ... })` to gain voting power — a standard action.
6. `H` immediately calls `manage_neuron` on `C` with `Command::RegisterVote(...)` or `Command::Follow(...)`, exercising Bob's voting power on any open proposal.
7. A deterministic unit test can be written by extending the existing `test_neuron_spawn` test in `rs/nns/governance/tests/governance.rs` (around line 5744): add a hot key to the parent neuron before spawning with `new_controller = child_controller`, then assert that the hot key is present on the child neuron and can successfully call `register_vote` after the dissolve delay is increased.
