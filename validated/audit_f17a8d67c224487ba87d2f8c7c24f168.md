### Title
Unbounded `while` Loop in `move_to_block` Causes NEAR Gas Exhaustion DoS on Every Aurora Transaction - (`engine-hashchain/src/hashchain.rs`)

---

### Summary

The `Hashchain::move_to_block` function contains an unbounded `while` loop that iterates once per NEAR block skipped since the last Aurora transaction. This loop is executed on every call to `submit`, `submit_with_args`, `call`, and `deploy_code`. If the Aurora Engine is idle for a sufficient number of NEAR blocks, the loop will exhaust the NEAR gas limit (300 TGas), causing every subsequent Aurora transaction to fail — a complete, self-reinforcing DoS of the protocol.

---

### Finding Description

`Hashchain::move_to_block` in `engine-hashchain/src/hashchain.rs` iterates from the stored `current_block_height` to the current NEAR block height, performing a `keccak` hash computation on every iteration: [1](#0-0) 

This function is called unconditionally from `load_hashchain` in `engine/src/hashchain.rs`: [2](#0-1) 

`load_hashchain` is called by both `with_hashchain` and `with_logs_hashchain`: [3](#0-2) [4](#0-3) 

These wrappers are used by every state-mutating EVM entry point: [5](#0-4) [6](#0-5) 

The number of loop iterations equals `current_NEAR_block_height − last_Aurora_tx_block_height`. Each iteration calls `compute_block_hashchain`, which invokes `keccak` on a ~100-byte payload: [7](#0-6) 

There is no cap on the number of iterations and no check before entering the loop.

---

### Impact Explanation

**High — Temporary freezing of funds / DoS of all Aurora transactions.**

When the gap between NEAR blocks grows large enough (proportional to NEAR's ~1 block/second cadence), the `while` loop exhausts the 300 TGas NEAR gas limit. Every call to `submit`, `submit_with_args`, `call`, and `deploy_code` fails. Users cannot move funds, execute contracts, or interact with the EVM in any way. Because the hashchain state is never updated on a failed transaction, the condition is self-reinforcing: once the threshold is crossed, no ordinary transaction can recover the state. The only recovery path, `start_hashchain`, also calls `move_to_block` with the same unbounded loop: [8](#0-7) 

making even the administrative recovery mechanism vulnerable to the same failure.

---

### Likelihood Explanation

NEAR produces approximately one block per second. The NEAR gas limit per transaction is 300 TGas. Each loop iteration performs at least one `keccak` call plus memory allocations. Even at a conservative estimate of a few GGas per iteration, tens of thousands of idle NEAR blocks (a few hours of Aurora inactivity) are sufficient to trigger the failure. Aurora has historically experienced periods of low or zero transaction activity. Any unprivileged user can trigger the DoS simply by submitting a transaction after such a gap — no special privileges or funds are required.

---

### Recommendation

Cap the number of blocks that `move_to_block` will process in a single call. If the gap exceeds the cap, either:
1. Return an error and require the operator to call a dedicated "catch-up" administrative function that processes the gap in bounded chunks across multiple transactions, or
2. Skip intermediate empty blocks entirely and only record the hashchain for blocks that actually contain Aurora transactions, eliminating the need to iterate over idle blocks.

---

### Proof of Concept

1. Deploy Aurora Engine with hashchain enabled via `start_hashchain`.
2. Allow the NEAR chain to produce N blocks with no Aurora transactions (e.g., N = 50,000, representing ~14 hours of inactivity).
3. Submit any EVM transaction via `submit`.
4. `with_logs_hashchain` → `load_hashchain` → `hashchain.move_to_block(current_block_height)` enters the `while` loop at `engine-hashchain/src/hashchain.rs:75`.
5. The loop executes N iterations, each calling `compute_block_hashchain` (one `keccak` call) and `clear_txs`.
6. NEAR gas is exhausted before the loop completes; the transaction fails with a gas-exceeded error.
7. All subsequent `submit`/`call`/`deploy_code` calls fail identically — the protocol is fully DoS'd.
8. Attempting recovery via `start_hashchain` also fails because it calls `move_to_block` at `engine/src/contract_methods/admin.rs:448` with the same unbounded loop.
