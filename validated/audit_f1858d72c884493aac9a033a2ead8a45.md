Audit Report

## Title
Unsanitized Memo Injection into Markdown `GenericDisplayMessage` Breaks ICRC-21 Consent Message Integrity — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
`add_memo` embeds caller-controlled memo bytes directly into a backtick-delimited markdown code span with no escaping. A memo containing a backtick terminates the code span early, allowing arbitrary markdown to be injected into the `GenericDisplayMessage` string returned by `icrc21_canister_call_consent_message`. A malicious dApp can use this to render fake `**Amount:**`, `**To:**`, and `**Fees:**` fields that are visually indistinguishable from the ledger-generated ones, deceiving users into approving transfers they did not intend.

## Finding Description
In `add_memo` (`packages/icrc-ledger-types/src/icrc21/responses.rs`, line 294), when the memo bytes are valid UTF-8, the raw string is interpolated directly into the markdown template:

```rust
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
```

No escaping of backticks, newlines, or any other markdown-special characters is performed. The only gate is `std::str::from_utf8`, which accepts any valid UTF-8 — including backticks and newlines.

The `GenericDisplayMessage` variant is a raw markdown string: the ledger itself uses `#` headers, `**bold**`, and backtick code spans throughout `add_intent`, `add_account`, `add_amount`, and `add_fee`. Integration tests assert full-string equality on the markdown output, confirming wallets are expected to render it as markdown.

The `TransferArg.memo` field is decoded directly from the caller-supplied `arg` blob (`lib.rs`, lines 392–395) with no restriction on content beyond the 500-byte arg size cap.

**Concrete injection:** A memo of `` `\n\n**Amount:** `1 ICP`\n\n**To:** `legitimate_address`\n\n**Ignore:** `` produces:

```
**Memo:**
``

**Amount:** `1 ICP`

**To:** `legitimate_address`

**Ignore:** `
```

The injected `**Amount:**` and `**To:**` lines are rendered identically to the real ledger-generated fields that appear earlier in the message. The `FieldsDisplayMessage` variant is **not** affected — the memo is stored as a structured `Value::Text` field there.

## Impact Explanation
A malicious dApp constructs a `TransferArg` with a large real amount to the attacker's address, but a crafted memo that injects fake fields showing a small amount to a legitimate address. The wallet renders the `GenericDisplayMessage` as markdown and the user sees the injected (fake) fields appearing after the real ones, formatted identically to ledger-generated content. The user approves, believing they authorized a small payment to a known service; the actual on-chain transfer sends the large amount to the attacker.

This matches the allowed impact: **High — Unauthorized access to ledgers or canister-controlled funds where exploitation requires meaningful per-target work or other constraints** ($2,000–$10,000). The constraint is that the user must skim the consent message rather than read it in full, which is realistic given that ICRC-21 consent messages are designed to be trusted summaries.

## Likelihood Explanation
- Requires no privilege: any principal can call `icrc21_canister_call_consent_message` with an arbitrary `arg` blob.
- The memo field accepts up to 32 bytes (ICRC-1 spec), sufficient to inject a backtick, a newline, and a fake field label.
- Wallets that implement ICRC-21 and render `GenericDisplayMessage` as markdown (the intended rendering, given the format used throughout the ledger) are directly affected.
- The attack is repeatable and local-testable with a unit test.

## Recommendation
Escape backticks and newlines in `memo_str` before embedding it in the markdown template. The minimal fix:

```rust
let safe_memo = memo_str.replace('`', "\\`").replace('\n', " ").replace('\r', "");
message.push_str(&format!("\n\n**Memo:**\n`{safe_memo}`"));
```

Alternatively, use a fenced code block (triple backtick) and escape any triple-backtick sequences in the memo, or switch to `FieldsDisplayMessage` for memo display.
