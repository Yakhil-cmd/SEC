### Title
Unbounded DKG Unvalidated Pool Growth via Remote-Dealing Deferral with No Slot Limit — (`rs/replica/setup_ic_network/src/lib.rs`, `rs/consensus/dkg/src/lib.rs`)

### Summary
A Byzantine subnet peer can flood the unvalidated `DkgPool` with an unbounded number of remote DKG dealings referencing non-existent target subnet IDs. The deferral logic in `validate_dealings_for_dealer` silently keeps these in the pool, and the P2P layer imposes no slot-table limit on DKG artifacts, allowing the pool to grow without bound until the next DKG interval purge.

### Finding Description

The attack exploits three independent gaps that compose into a complete exploit path:

**Gap 1 — Deferral without rejection.**
In `DkgImpl::validate_dealings_for_dealer`, when a dealing's `NiDkgId` references a `NiDkgTargetSubnet::Remote(_)` with no matching config, the code returns `Mutations::new()` — the artifact is neither validated nor invalidated and stays in the unvalidated pool indefinitely. [7](#0-6) 

**Gap 2 — No P2P slot-table limit for DKG.**
In `AbortableBroadcastChannels::new`, the DKG channel is registered with `SLOT_TABLE_NO_LIMIT` (`usize::MAX`). The slot-table enforces a per-peer cap on how many artifact IDs a peer may advertise simultaneously; with `usize::MAX` this cap is effectively absent. Ingress, by contrast, uses `SLOT_TABLE_LIMIT_INGRESS = 50_000`. [3](#0-2) 

The slot-table enforcement in `ConsensusManagerReceiver::handle_slot_update_receive` drops new entries only when `peer_slot_table_len >= self.slot_limit`. With `slot_limit = usize::MAX` this branch is never taken. [8](#0-7) 

**Gap 3 — Content-blind bouncer.**
`DkgBouncer::new_bouncer` returns `BouncerValue::Wants` for any artifact whose `DkgMessageId.height` equals the current start height, with no inspection of the target subnet or any other content field. Every crafted dealing at the current height is therefore downloaded. [9](#0-8) 

**Attack sequence:**
1. Byzantine peer (legitimate subnet member, below fault threshold) crafts N distinct `dkg::Message` values, each with `NiDkgTargetSubnet::Remote(random_id_i)` and `start_block_height = current_start_height`. Each has a unique hash → unique `DkgMessageId`.
2. Peer advertises each via a fresh slot number (slot 1, 2, …, N). Because `slot_limit = usize::MAX`, all N slot entries are accepted.
3. Each artifact is downloaded and delivered to `DkgPoolImpl::insert`, which performs an unconditional BTreeMap insert with no size guard. [5](#0-4) 
4. `on_state_change` → `validate_dealings_for_dealer` → config not found, `is_remote() = true` → `Mutations::new()`. All N entries remain in the unvalidated pool.
5. Repeat until the DKG interval ends (~500 blocks, ~8 minutes on mainnet). Purge removes entries with `id.height < new_height`, clearing the pool — but the attacker immediately restarts for the next interval.

### Impact Explanation
NiDKG dealings contain large cryptographic payloads (polynomial commitments, encrypted shares). Flooding with O(N) such artifacts can exhaust replica heap memory, causing an OOM crash and halting consensus participation for the targeted replica. If enough replicas on a subnet are targeted simultaneously (each by a different Byzantine peer, or one peer targeting all), subnet liveness is lost.

### Likelihood Explanation
The attacker must be a legitimate subnet node (authenticated via TLS). This is a realistic Byzantine fault scenario — the protocol tolerates up to f faults, and a single Byzantine node suffices to target individual replicas. No external infrastructure is required; the attack is purely P2P artifact flooding. The 8-minute window per DKG interval is sufficient to exhaust memory given the size of NiDKG dealings.

### Recommendation
1. **Apply a finite slot-table limit to DKG**, analogous to `SLOT_TABLE_LIMIT_INGRESS`. A bound of `O(subnet_size × configs_per_interval)` is sufficient for honest operation.
2. **Reject (not defer) remote dealings with unknown configs** when the target subnet ID is not plausibly pending (e.g., not referenced in any recent state). At minimum, add a per-peer cap on deferred remote dealings in the unvalidated pool.
3. **Extend `DkgBouncer`** to return `BouncerValue::Unwanted` for artifacts whose `NiDkgId.target_subnet` is `Remote(_)` and whose target ID is not present in the current replicated state, preventing download before pool insertion.
