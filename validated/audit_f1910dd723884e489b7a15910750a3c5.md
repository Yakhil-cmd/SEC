Audit Report

## Title
Fragmented UTXO Set Causes `BuildTxError::AmountTooLow` With No Reimbursement When Withdrawal Is At Minimum Amount — (`rs/dogecoin/ckdoge/minter/src/fees/mod.rs`)

## Summary

`fee_based_minimum_withdrawal_amount` in the ckDOGE minter computes the minimum withdrawal amount using `PER_REQUEST_MINTER_FEE_BOUND = 326_000` koinu — a value 1000x smaller than the actual minter fee for even a 2-input transaction (326,000,000 koinu). When the minter's UTXO set is fragmented and the real transaction requires many inputs, `build_unsigned_transaction` returns `BuildTxError::AmountTooLow`, which finalizes the request with `FinalizedStatus::AmountTooLow` and **no reimbursement**, permanently burning the user's ckDOGE.

## Finding Description

**Step 1 — Minimum amount is computed with a severely underestimated minter fee bound.**

`fee_based_minimum_withdrawal_amount` in `rs/dogecoin/ckdoge/minter/src/fees/mod.rs` uses `PER_REQUEST_MINTER_FEE_BOUND = 326_000` koinu (L104). The actual minter fee for 2 inputs, 2 outputs computed by `evaluate_minter_fee` is `146_000_000 * 2 + 4_000_000 * 2 + 26_000_000 = 326_000_000` koinu (L80–91) — a factor of 1000 larger. At low fee rates, the formula reduces to simply `retrieve_doge_min_amount` (default 50 DOGE = 5,000,000,000 koinu), with the minter fee bound contributing negligibly.

**Step 2 — Actual transaction building uses the real input count.**

`build_unsigned_transaction` in `rs/bitcoin/ckbtc/minter/src/lib.rs` calls `fee_estimator.evaluate_minter_fee(input_utxos.len() as u64, ...)` (L1264–1265) with the actual number of selected UTXOs. For N inputs, the minter fee is `146_000_000 * N + 34_000_000` koinu. The check at L1306–1307 is:
```rust
if fee + minter_fee > amount {
    return Err(BuildTxError::AmountTooLow);
}
```
For N = 35 inputs: `146_000_000 * 35 + 34_000_000 = 5,144,000,000 koinu > 5,000,000,000 koinu` → `AmountTooLow`.

**Step 3 — `AmountTooLow` finalizes with no reimbursement.**

The `AmountTooLow` branch in `submit_pending_requests` (L412–434) calls `state::audit::remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow` for each request in the batch, issuing no reimbursement. By contrast, `BuildTxError::InvalidTransaction` (L400–411) calls `reimburse_canceled_requests`.

**Step 4 — ckDOGE uses the same code path.**

`retrieve_doge_with_approval` delegates to `ic_ckbtc_minter::updates::retrieve_btc::retrieve_btc_with_approval` (main.rs L135), which burns ckDOGE before transaction building (L314–319 in retrieve_btc.rs). The timer calls `ic_ckbtc_minter::timer(DOGECOIN_CANISTER_RUNTIME)` (main.rs L60), which invokes `submit_pending_requests`.

**Step 5 — Ingress check only validates against the pre-computed minimum.**

`retrieve_btc_with_approval` checks `args.amount < s.fee_based_retrieve_btc_min_amount` (retrieve_btc.rs L265–270). This minimum is derived from `fee_based_minimum_withdrawal_amount` with its underestimated minter fee bound. No simulation of the actual transaction is performed before burning.

## Impact Explanation

A user who submits a withdrawal for exactly the advertised minimum amount (e.g., 50 DOGE) has their ckDOGE burned at request acceptance. If the minter's UTXO set requires ≥35 inputs to cover that amount, the minter fee alone exceeds the withdrawal amount, `AmountTooLow` is returned, and the user's ckDOGE is permanently lost with no reimbursement. This is a concrete, permanent loss of user funds in an in-scope ck-token (ckDOGE), matching the **High** impact class: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation

UTXO fragmentation occurs naturally from many small deposits. An unprivileged attacker can accelerate it by depositing ~35 UTXOs of ≤1.43 DOGE each (total ≤50 DOGE). Any user who queries `get_minter_info()` to obtain `retrieve_doge_min_amount` and submits exactly that amount is at risk when the UTXO set is fragmented. The consolidation mechanism (`utxo_consolidation_threshold`) is asynchronous and not guaranteed to prevent the condition. The attack requires no special privileges and is repeatable.

## Recommendation

1. **Fix `PER_REQUEST_MINTER_FEE_BOUND`** in `rs/dogecoin/ckdoge/minter/src/fees/mod.rs`: change `326_000` to `326_000_000` to correctly reflect the 2-input minter fee, and use a conservative upper-bound input count (e.g., configurable `max_expected_inputs`) rather than a fixed 2-input assumption.
2. **Add reimbursement on `AmountTooLow`** in `rs/bitcoin/ckbtc/minter/src/lib.rs` (L412–434): when `BuildTxError::AmountTooLow` occurs, call `reimburse_canceled_requests` (minus a small processing fee) analogous to the `InvalidTransaction` path, so users are not permanently penalized for a minter-side UTXO fragmentation condition.
3. **Alternatively**: before finalizing with `AmountTooLow`, attempt UTXO re-selection to reduce input count, or split the batch.

## Proof of Concept

```
1. Deposit 40 UTXOs of 1.25 DOGE each into the ckDOGE minter (total: 50 DOGE in minter's available UTXO set).
2. Query get_minter_info() → retrieve_doge_min_amount = 50 DOGE (at low fee rate).
3. Call retrieve_doge_with_approval({ amount: 50_DOGE, address: "..." }).
   → Ingress check passes: 50 DOGE >= 50 DOGE minimum (fee_based_retrieve_btc_min_amount).
   → ckDOGE burned from user account via burn_ckbtcs_icrc2.
4. Timer fires → submit_pending_requests → build_unsigned_transaction:
   - utxo_selection selects all 40 UTXOs to cover 50 DOGE.
   - minter_fee = 146_000_000 * 40 + 4_000_000 * 2 + 26_000_000 = 5,874,000,000 koinu.
   - 5,874,000,000 > 5,000,000,000 = amount → BuildTxError::AmountTooLow.
5. AmountTooLow branch: remove_retrieve_btc_request called with FinalizedStatus::AmountTooLow, no reimbursement.
6. User has permanently lost 50 DOGE worth of ckDOGE.
```

Verifiable via a PocketIC integration test: initialize ckDOGE minter on Regtest, inject 40 UTXOs of 1.25 DOGE each into `available_utxos`, call `retrieve_doge_with_approval` for 50 DOGE, advance the timer, and assert the request is finalized with `AmountTooLow` and no reimbursement event is emitted.     fn fee_based_minimum_withdrawal_amount(&self, median_fee_rate: FeeRate) -> u64 {
        match self.network {
            Network::Mainnet => {
                //in Koinu
                const PER_REQUEST_RBF_BOUND: u64 = 374_000;
                // in Bytes
                // Size of a typical transaction made by the minter,
                // which is a P2PKH transaction with 2 inputs and 2 outputs
                const PER_REQUEST_SIZE_BOUND: u64 = 374;
                //in Koinu
                const PER_REQUEST_MINTER_FEE_BOUND: u64 = 326_000;

                let min_withdrawal_amount_increment = self.retrieve_doge_min_amount >> 1;
                ((PER_REQUEST_RBF_BOUND
                    + median_fee_rate.fee_ceil(PER_REQUEST_SIZE_BOUND)
                    + PER_REQUEST_MINTER_FEE_BOUND)
                    / min_withdrawal_amount_increment)
                    * min_withdrawal_amount_increment
                    + self.retrieve_doge_min_amount
            }
            Network::Regtest => self.retrieve_doge_min_amount,
        }
    }
