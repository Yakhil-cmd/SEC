### Title
Unchecked Arithmetic Operators on ETH Connector Balance Type `NEP141Wei` Enable Silent Underflow/Overflow in Release Mode - (`engine-types/src/types/wei.rs`)

### Summary
The `NEP141Wei` type, which represents bridged ETH balances in Aurora's ETH connector, implements the standard Rust `Sub`, `Add`, and `SubAssign` operators using raw, unchecked `u128` arithmetic. In Rust release mode (the WASM production build), these operations silently wrap on underflow or overflow instead of panicking. The `Yocto` and `Wei` types share the same defect. This is a direct analog to the Uniswap/Superposition finding: checked variants (`checked_sub`, `checked_add`) exist on the type but are bypassed whenever the plain `-`, `+`, or `-=` operators are used.

### Finding Description

In `engine-types/src/types/wei.rs`, the `NEP141Wei` type (the canonical balance unit for the ETH connector's NEP-141 accounting) implements three unchecked arithmetic operators:

```rust
impl Sub for NEP141Wei {
    fn sub(self, rhs: Self) -> Self::Output {
        Self(self.0 - rhs.0)   // raw u128 subtraction — wraps on underflow in release
    }
}

impl Add for NEP141Wei {
    fn add(self, rhs: Self) -> Self::Output {
        Self(self.0 + rhs.0)   // raw u128 addition — wraps on overflow in release
    }
}

impl SubAssign for NEP141Wei {
    fn sub_assign(&mut self, rhs: Self) {
        *self = *self - rhs;   // delegates to unchecked Sub above
    }
}
```

The `SubAssign` implementation is only defined when `-=` is expected to be used in production code. The type simultaneously exposes `checked_sub` and `checked_add` methods, confirming the developers were aware of the need for safe arithmetic, yet the operator overloads remain unchecked.

The same defect is present in `engine-types/src/types/balance.rs` for `Yocto` (NEAR balance type) and in `engine-types/src/types/wei.rs` for `Wei` (EVM account balance type):

```rust
// Yocto (balance.rs lines 135-141)
impl Sub for Yocto {
    fn sub(self, rhs: Self) -> Self::Output {
        Self(self.0 - rhs.0)   // raw u128 subtraction
    }
}

// Wei (wei.rs lines 181-187)
impl Sub for Wei {
    fn sub(self, rhs: Self) -> Self::Output {
        Self(self.0 - rhs.0)   // raw U256 subtraction
    }
}
```

In Rust debug builds these panic on overflow/underflow; in release builds (the WASM production artifact) they silently wrap.

### Impact Explanation

`NEP141Wei` is the balance type for the ETH connector's NEP-141 accounting — it tracks how much bridged ETH each account holds and the total supply. If a withdrawal or transfer path uses the `-=` operator (via `SubAssign`) on a `NEP141Wei` balance without a prior sufficiency check, and the subtrahend exceeds the minuend, the result wraps to `u128::MAX - (rhs - lhs) + 1`. This inflates the stored balance to near-maximum, allowing the affected account to withdraw far more ETH than it deposited — a direct insolvency of the ETH connector bridge. Similarly, an unchecked `+` on `NEP141Wei` total supply can overflow to a small number, causing legitimate deposits to be rejected or balances to be understated.

**Impact: Critical — Insolvency / Direct theft of bridged ETH funds.**

### Likelihood Explanation

The `SubAssign` implementation is only useful if `-=` is called somewhere in production. The ETH connector must subtract balances on every withdrawal. Any withdrawal code path that uses the `-=` operator rather than `checked_sub` on a `NEP141Wei` value is exploitable by any user who can trigger a withdrawal with a crafted amount. The WASM release build is the only production artifact, and it is the one that silently wraps.

### Recommendation

Replace the unchecked operator implementations with checked variants that panic or return an error on overflow/underflow:

```rust
// engine-types/src/types/wei.rs

impl Sub for NEP141Wei {
    type Output = Self;
    fn sub(self, rhs: Self) -> Self::Output {
        self.checked_sub(rhs).expect("NEP141Wei subtraction underflow")
    }
}

impl Add for NEP141Wei {
    type Output = Self;
    fn add(self, rhs: Self) -> Self::Output {
        self.checked_add(rhs).expect("NEP141Wei addition overflow")
    }
}
```

Apply the same fix to `Yocto` and `Wei`. Alternatively, remove the operator overloads entirely and force all call sites to use `checked_sub` / `checked_add` explicitly, returning a proper error to the caller rather than panicking.

### Proof of Concept

Root cause is in the operator definitions: [1](#0-0) 

The `Sub` at line 81 performs `self.0 - rhs.0` on a `u128`. In Rust release mode, if `rhs.0 > self.0`, the result wraps to `u128::MAX - (rhs.0 - self.0 - 1)` instead of panicking. The `SubAssign` at lines 93–96 delegates directly to this unchecked `Sub`, so any `-=` on a `NEP141Wei` balance in the ETH connector inherits the defect. [2](#0-1) 

The same pattern exists for `Yocto` (NEAR balance type), and for `Wei`: [3](#0-2) 

The `checked_sub` and `checked_add` safe alternatives exist on `NEP141Wei` but are not used by the operator overloads: [4](#0-3)
