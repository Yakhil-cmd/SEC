### Title
Unprotected Initializer Allows Anyone to Seize Engine Ownership Before `new` Is Called - (File: `engine/src/contract_methods/admin.rs`)

### Summary

The `new` function in `engine/src/contract_methods/admin.rs` performs no caller-identity check. Its only guard is `ERR_ALREADY_INITIALIZED`, which fires only after state has been written. Any external account that calls `new` before the legitimate deployer does so will write an attacker-controlled `owner_id` into `EngineState`. Because every privileged administrative action (including `attach_full_access_key`, `stage_upgrade`, `deploy_upgrade`, `set_key_manager`) is gated solely on `require_owner_only(&state, &predecessor_account_id)`, the attacker immediately inherits full administrative authority over the Aurora Engine NEAR account and all bridged funds it holds.

### Finding Description

`engine/src/contract_methods/admin.rs` lines 55-88:

```rust
#[named]
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {          // ← only guard: already initialized?
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input)  // ← fully attacker-controlled
        .map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;
    // ...
    state::set_state(&mut io, &state)?;          // ← writes attacker's owner_id
    Ok(())
}
```

There is no check on `env.predecessor_account_id()` or `env.current_account_id()`. The `NewCallArgs` struct carries `owner_id`, `chain_id`, `upgrade_delay_blocks`, and `key_manager`, all of which are written verbatim into `EngineState`.

The workspace deployment helper (`engine-workspace/src/lib.rs` lines 107-125) issues the WASM `deploy` and the `new` call as **two separate transactions** with an `await` between them, creating an observable window on-chain:

```rust
let contract = account.deploy(&self.code...).await?;   // tx 1
// ← attacker window here
engine.new(self.chain_id, self.owner_id, ...).transact().await  // tx 2
```

### Impact Explanation

Once the attacker's `owner_id` is stored in `EngineState`, they can call:

- `attach_full_access_key` (`admin.rs` lines 483-512) — adds an attacker-controlled full-access key to the Aurora Engine NEAR account, granting unrestricted control over the account and all assets it holds.
- `stage_upgrade` / `deploy_upgrade` — replaces the engine WASM with arbitrary code, enabling any state manipulation or fund drain.
- `set_key_manager` — installs an attacker-controlled key manager, enabling relayer key injection.

All of these are gated only on `require_owner_only`, which compares `state.owner_id` against `predecessor_account_id`. With the attacker's account stored as `owner_id`, every check passes.

**Impact: Critical — direct theft of all user funds held by the Aurora Engine contract.**

### Likelihood Explanation

NEAR Protocol allows any account to call any public method on any deployed contract. The only barrier is the `ERR_ALREADY_INITIALIZED` guard, which is absent before the first `new` call. The workspace code confirms that deployment and initialization are separate transactions. An attacker monitoring the NEAR mempool or block stream for a freshly deployed Aurora Engine WASM can submit a `new` call with their own parameters in the same block or the next block, before the legitimate `new` transaction is included. No privileged access, leaked keys, or social engineering is required — only the ability to submit a NEAR transaction.

### Recommendation

Add a caller-identity check at the top of `new` that restricts initialization to the contract's own account (i.e., `env.current_account_id() == env.predecessor_account_id()`). In NEAR, a contract can only be deployed by its own account, so this check ensures only the account owner can initialize the contract:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if env.predecessor_account_id() != env.current_account_id() {
        return Err(b"ERR_NOT_ALLOWED".into());
    }
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // ...
}
```

Alternatively, use NEAR's batch-transaction pattern to atomically deploy and call `new` in a single transaction, eliminating the window entirely.

### Proof of Concept

1. Attacker watches the NEAR blockchain for a `DeployContract` action targeting the Aurora Engine account.
2. In the next transaction (or same block), attacker calls `new` on the Aurora Engine account with:
   ```json
   {
     "chain_id": 1313161554,
     "owner_id": "attacker.near",
     "upgrade_delay_blocks": 0,
     "key_manager": "attacker.near",
     "initial_hashchain": null
   }
   ```
3. `state::get_state` returns `Err(NotFound)`, so the `ERR_ALREADY_INITIALIZED` guard does not fire.
4. `state::set_state` writes `owner_id = "attacker.near"` into storage.
5. The legitimate deployer's subsequent `new` call returns `ERR_ALREADY_INITIALIZED` and fails.
6. Attacker calls `attach_full_access_key` with their own public key. `require_owner_only` passes because `state.owner_id == "attacker.near" == predecessor_account_id`.

#[named]
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
