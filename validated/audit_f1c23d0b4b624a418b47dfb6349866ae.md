Audit Report

## Title
Unused `delivered_ids` in `validate_payload` Allows Byzantine Proposer to Inject Duplicate Bitcoin Adapter Responses — (`rs/bitcoin/consensus/src/payload_builder.rs`)

## Summary

`validate_payload` in the `BatchPayloadBuilder` impl for `BitcoinPayloadBuilder` computes `delivered_ids` from `past_payloads` at line 369 but never consults it when validating the proposed payload. A single Byzantine block proposer can include a `BitcoinAdapterResponse` whose `callback_id` already appears in a past payload; all honest validators accept the block, and `push_response` processes the duplicate, overwriting `bitcoin_get_successors_follow_up_responses` in replicated state and injecting a second `ConsensusResponse` into `consensus_queue`.

## Finding Description

**Root cause — unused `delivered_ids`:**

`validate_payload` computes `delivered_ids` at line 369 and then never references it again. The only downstream call is `validate_self_validating_payload_impl`, which receives only the payload and validation context: [1](#0-0) 

`validate_self_validating_payload_impl` itself only short-circuits on an empty payload and returns the byte count — no content or deduplication check: [2](#0-1) 

The `SelfValidatingPayloadBuilder` impl compounds this by explicitly ignoring `_past_payloads`: [3](#0-2) 

By contrast, `build_payload` correctly passes `delivered_ids` into `get_self_validating_payload_impl` and skips already-delivered IDs at line 129: [4](#0-3) 

**Exploit path:**

1. A Byzantine block proposer observes a finalized block containing a `BitcoinAdapterResponse` with `callback_id = N`.
2. It crafts a new block whose bitcoin payload contains a response with the same `callback_id = N` (and arbitrary content).
3. Honest validators call `validate_payload`, compute `delivered_ids` (which contains `N`), and then ignore it — the block passes validation and is finalized.
4. `demux.process_payload` iterates `batch_messages.bitcoin_adapter_responses` and calls `push_response` for each entry, including the duplicate.

**State corruption in `push_response`:**

For `GetSuccessorsResponse`, `push_response` uses `get_mut` (not `remove`) on `bitcoin_get_successors_contexts`, so the context survives the first call and is found again on the duplicate call: [5](#0-4) 

The comment on line 43 explicitly notes the overwrite: `// Store the follow-ups for later (overwrites previous ones).` The duplicate call replaces `bitcoin_get_successors_follow_up_responses` with the attacker-controlled follow-up blocks, corrupting the paginated block data the bitcoin canister will serve. A second `ConsensusResponse` is also pushed to `consensus_queue`.

For `SendTransactionResponse`, `GetSuccessorsReject`, and `SendTransactionReject`, there is no context lookup at all — both the original and duplicate `ConsensusResponse` are unconditionally pushed to `consensus_queue`: [6](#0-5) 

## Impact Explanation

This is a **High** severity finding. A Byzantine proposer can corrupt `bitcoin_get_successors_follow_up_responses` in replicated state with attacker-controlled block data. The bitcoin canister reads these follow-up responses to serve paginated `get_successors` results to callers; corrupted follow-ups mean the canister delivers an incorrect view of the Bitcoin chain. This directly affects the Chain Fusion / ckBTC infrastructure, which relies on accurate Bitcoin chain data for minting and redemption decisions. This matches the allowed impact: *"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm"* (High, $2,000–$10,000).

## Likelihood Explanation

Exploitation requires exactly one Byzantine replica acting as block proposer during its normal turn — no threshold corruption, no key compromise, no external infrastructure attack. The proposer can observe any past finalized payload to obtain a valid `callback_id`. The attack is repeatable every time the malicious node is scheduled as proposer. All honest validators will notarize and finalize the block because `validate_payload` performs no duplicate check.

## Recommendation

After computing `delivered_ids` in `validate_payload`, iterate over the decoded responses and reject any whose `callback_id` is already delivered:

```rust
let delivered_ids = parse::parse_past_payload_ids(past_payloads, &self.log);
let payload = parse::bytes_to_payload(payload)...?;
for response in &payload {
    if delivered_ids.contains(&response.callback_id) {
        return Err(ValidationError::InvalidArtifact(
            consensus::InvalidPayloadReason::InvalidSelfValidatingPayload(
                InvalidSelfValidatingPayloadReason::DuplicateResponse,
            ),
        ));
    }
}
```

Apply the same fix to `validate_self_validating_payload` — stop ignoring `_past_payloads` and check for duplicates there as well. Additionally, `push_response` for `GetSuccessorsResponse` should use `remove` instead of `get_mut` so that any duplicate reaching the state layer is caught by `BitcoinNonMatchingResponse` rather than silently overwriting follow-up data.

```rust
// 2. Pass that encoding as a PastPayload entry.

// 4. Assert Ok(()) — demonstrating no duplicate check occurs today.
let response = BitcoinAdapterResponse { callback_id: 0, response: /* any valid wrapper */ };
let payload_bytes = encode_payload(&[response]);
let past = PastPayload { height: Height::new(1), time: mock_time(), payload: &payload_bytes, ... };

    Height::new(2), &proposal_context, &payload_bytes, &[past]
);
assert_eq!(result, Ok(()));  // passes today; must return Err after fix
```
