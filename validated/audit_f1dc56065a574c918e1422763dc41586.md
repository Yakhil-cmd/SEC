### Title
Asymmetric Access Control on `resume_precompiles` Causes Latency-Induced Temporary Fund Freeze - (File: `engine/src/contract_methods/admin.rs`)

### Summary
`pause_precompiles` is callable by any account in the `EngineAuthorizer` ACL, enabling fast emergency response. However, `resume_precompiles` is gated exclusively behind `require_owner_only`. If the owner account is a DAO or multisig with a governance voting/timelock delay, the bridge exit precompiles (`EXIT_TO_NEAR`, `EXIT_TO_ETHEREUM`) can remain paused for the full duration of that delay, temporarily freezing all user funds that depend on those precompiles to exit Aurora.

### Finding Description

In `engine/src/contract_methods/admin.rs`, the two precompile control functions have deliberately asymmetric access control:

**`pause_precompiles`** (line 225–241) uses the `EngineAuthorizer` ACL:

```rust
let authorizer: EngineAuthorizer = engine::get_authorizer(&io);
if !authorizer.is_authorized(&env.predecessor_account_id()) {
    return Err(b"ERR_UNAUTHORIZED".into());
}
``` [1](#0-0) 

Any account pre-listed in the ACL can pause precompiles immediately — this is the intended fast-path for emergencies.

**`resume_precompiles`** (line 208–223) uses `require_owner_only`:

```rust
require_owner_only(&state, &predecessor_account_id)?;
``` [2](#0-1) 

Only the single `owner_id` stored in `EngineState` can resume precompiles. [3](#0-2) 

The `EngineAuthorizer` ACL is a `BTreeSet<AccountId>` that can contain multiple accounts: [4](#0-3) 

But the `EngineState.owner_id` is a single account. If that account is a DAO or multisig contract with a governance voting period or timelock, every call to `resume_precompiles` must wait for that governance process to complete before the precompiles can be re-enabled.

The two precompiles controlled by this mechanism are `EXIT_TO_NEAR` and `EXIT_TO_ETHEREUM`: [5](#0-4) 

These are the only on-chain exit paths for users to move their ETH and ERC-20 tokens out of Aurora back to NEAR or Ethereum.

### Impact Explanation

When `EXIT_TO_NEAR` or `EXIT_TO_ETHEREUM` is paused, users cannot bridge their assets out of Aurora. If the owner is a DAO with a multi-day governance delay (e.g., 7 days, as in the referenced report), the precompiles remain paused for that entire period. During this window, all user funds that can only exit via these precompiles are **temporarily frozen** — users cannot withdraw to NEAR or Ethereum regardless of their urgency.

This matches the **High — Temporary freezing of funds** impact class.

### Likelihood Explanation

Aurora is a production protocol where the `owner_id` is realistically a DAO or multisig (standard governance practice for large DeFi protocols). The `EngineAuthorizer` ACL already demonstrates that the design anticipates multiple fast-response accounts for pausing. The absence of a symmetric fast-path for resuming is a structural gap. An authorized account (or the owner itself) may legitimately trigger an emergency pause, after which the governance delay prevents timely resumption. This is a realistic operational scenario, not a theoretical one.

### Recommendation

Extend the `EngineAuthorizer` ACL to also authorize `resume_precompiles`, mirroring the access model used for `pause_precompiles`. Alternatively, introduce a dedicated `resume_authorizer` ACL (separate from the pause ACL) so that a set of trusted fast-response accounts can resume precompiles without waiting for owner-level governance. The owner should retain the ability to manage both ACLs.

### Proof of Concept

1. Owner (a DAO with a 7-day voting period) pre-authorizes account `sentinel.near` in the `EngineAuthorizer` ACL.
2. An anomaly is detected; `sentinel.near` calls `pause_precompiles` with `EXIT_TO_NEAR | EXIT_TO_ETHEREUM` — executes immediately, no delay.
3. The anomaly is resolved within hours. `sentinel.near` attempts to call `resume_precompiles` — **reverts** with `ERR_NOT_ALLOWED` because `sentinel.near` is not the `owner_id`.
4. A DAO proposal to call `resume_precompiles` is submitted. The voting period runs for 7 days.
5. For 7 days, all Aurora users are unable to exit their ETH or ERC-20 tokens to NEAR or Ethereum via the `EXIT_TO_NEAR` / `EXIT_TO_ETHEREUM` precompiles — funds are temporarily frozen.

pub fn resume_precompiles<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let predecessor_account_id = env.predecessor_account_id();

        require_owner_only(&state, &predecessor_account_id)?;

        let args: PausePrecompilesCallArgs = io.read_input_borsh()?;
        let flags = PrecompileFlags::from_bits_truncate(args.paused_mask);
        let mut pauser = EnginePrecompilesPauser::from_io(io);
        pauser.resume_precompiles(flags);
        Ok(())
    })
}

#[named]
pub fn pause_precompiles<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        require_running(&state::get_state(&io)?)?;
        let authorizer: EngineAuthorizer = engine::get_authorizer(&io);

        if !authorizer.is_authorized(&env.predecessor_account_id()) {
            return Err(b"ERR_UNAUTHORIZED".into());
        }

        let args: PausePrecompilesCallArgs = io.read_input_borsh()?;
        let flags = PrecompileFlags::from_bits_truncate(args.paused_mask);
        let mut pauser = EnginePrecompilesPauser::from_io(io);
        pauser.pause_precompiles(flags);
        Ok(())
    })
}
