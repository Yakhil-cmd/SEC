### Title
Multiple Aurora-Specific Precompiles Charge Zero EVM Gas for Real NEAR Work, Enabling Unmetered NEAR Gas Consumption - (File: engine-precompiles/src/native.rs, engine-precompiles/src/random.rs, engine-precompiles/src/prepaid_gas.rs, engine-precompiles/src/account_ids.rs)

---

### Summary

Six Aurora-specific precompiles are hardcoded to charge **zero EVM gas** despite performing real NEAR host function calls, storage reads, and promise creation. Any unprivileged EVM user can call these precompiles in a tight loop, consuming NEAR gas without paying proportional EVM gas. The Aurora contract bears the NEAR gas cost, creating a gas accounting mismatch that allows an attacker to fill NEAR blocks cheaply and temporarily prevent other users' transactions from being processed.

---

### Finding Description

The following Aurora-specific precompiles all return `EthGas::new(0)` from `required_gas()`, explicitly marked as unresolved TODOs:

**`ExitToNear`** and **`ExitToEthereum`** (bridge withdrawal precompiles): [1](#0-0) 

**`PrepaidGas`**: [2](#0-1) 

**`RandomSeed`**: [3](#0-2) 

**`PredecessorAccount`** and **`CurrentAccount`**: [4](#0-3) 

The gas cost returned by `required_gas()` is what gets recorded against the EVM gas budget via `post_process → handle.record_cost(output.cost.as_u64())`: [5](#0-4) 

Because `output.cost` is `EthGas::new(0)` for all six precompiles, `record_cost(0)` is called — the EVM gasometer is never debited for the real NEAR work performed.

The `ExitToNear` and `ExitToEthereum` precompiles are the most impactful: they perform storage reads (`get_eth_connector_contract_account`, `get_nep141_from_erc20`) and create NEAR cross-contract promises, all of which consume significant NEAR gas: [6](#0-5) [7](#0-6) 

`PrepaidGas`, `RandomSeed`, and `PredecessorAccount`/`CurrentAccount` invoke NEAR host functions (`prepaid_gas()`, `random_seed`, `predecessor_account_id()`) that also consume NEAR gas, but are callable with no token requirement whatsoever.

The CHANGES.md confirms this class of bug was previously identified: *"Fixed missing gas costs in `exitToNear` and `exitToEthereum` precompiles by [@lempire123]. ([#687])"* — yet the current codebase still carries `EthGas::new(0)` for both. [8](#0-7) 

---

### Impact Explanation

In Aurora's architecture, the user pays EVM gas; the Aurora NEAR contract pays NEAR gas. EVM gas is supposed to be proportional to NEAR gas consumed. When a precompile charges 0 EVM gas, the attacker pays nothing for the NEAR gas the precompile consumes — the Aurora contract absorbs that cost.

An attacker deploys a Solidity contract with a loop calling `randomSeed` (address `0xc104f484...`) or `prepaidGas` (address `0x536822d2...`) repeatedly. Each loop iteration costs only the EVM opcodes for the loop body (~10 gas), while the precompile call itself costs 0 EVM gas. The attacker can therefore make hundreds of NEAR host function calls per transaction for a trivially small EVM gas expenditure.

By submitting many such transactions, the attacker fills NEAR blocks with NEAR gas consumption, starving legitimate user transactions (withdrawals, transfers) of block space. This constitutes **temporary freezing of funds**: users cannot execute bridge withdrawals or ERC-20 transfers while the attacker sustains the block-filling campaign.

**Impact: High — Temporary freezing of funds.**

---

### Likelihood Explanation

- No special privileges required: any EVM address can call `randomSeed`, `prepaidGas`, `predecessorAccountId`, or `currentAccountId` directly.
- `ExitToNear` and `ExitToEthereum` can be called with 0 ETH value (base-token path uses `context.apparent_value`, which can be 0), requiring no token balance.
- The attack is cheap: the only EVM gas cost is the CALL opcode (~700 gas) plus loop overhead per iteration.
- The CHANGES.md shows this exact class of bug was previously reported and "fixed," indicating it is a known, realistic attack vector.

**Likelihood: High.**

---

### Recommendation

Assign a non-zero `required_gas` to each affected precompile, calibrated to the actual NEAR gas consumed by the underlying host function calls and promise creation. The existing `CROSS_CONTRACT_CALL_BASE`/`CROSS_CONTRACT_CALL_BYTE` methodology in `xcc.rs` (which converts NEAR gas to EVM gas using `CROSS_CONTRACT_CALL_NEAR_GAS = 175_000_000`) provides a template: [9](#0-8) 

For `ExitToNear` and `ExitToEthereum`, the gas cost should account for at minimum the storage reads and promise scheduling. For `PrepaidGas`, `RandomSeed`, `PredecessorAccount`, and `CurrentAccount`, the cost should reflect the NEAR host function call overhead. The TODO issue `#483` already tracks this work.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract ZeroGasAbuse {
    // randomSeed precompile: 0xc104f4840573bed437190daf5d2898c2bdf928ac
    address constant RANDOM_SEED = 0xc104f4840573bed437190daf5d2898c2bdf928ac;

    // Call randomSeed 10,000 times — costs only loop EVM gas (~10 gas/iter),
    // but each call invokes a NEAR host function consuming real NEAR gas.
    // The precompile charges 0 EVM gas, so record_cost(0) is called each time.
    function drain() external {
        for (uint256 i = 0; i < 10000; i++) {
            (bool ok,) = RANDOM_SEED.call("");
            require(ok);
        }
    }
}
```

The attacker deploys this contract and calls `drain()` repeatedly across many transactions. Each transaction consumes significant NEAR gas (bounded by the 300 Tgas NEAR limit) while paying only minimal EVM gas (loop opcodes + 700 gas per CALL). The Aurora contract absorbs the NEAR gas cost. Sustained across multiple blocks, this starves legitimate user transactions of NEAR block space, temporarily preventing fund withdrawals and transfers.

        // ETH (base) transfer input format: (85 bytes)

        //  - recipient_account_id (max MAX_INPUT_SIZE - 20 - 1 bytes)
        // ERC-20 transfer input format: (124 bytes)
        //  - flag (1 byte)

        //  - amount (32 bytes)

        //  - `:unwrap` suffix in a case of wNEAR (7 bytes)

        if let Some(target_gas) = target_gas

fn post_process(
    output: PrecompileOutput,
    handle: &mut impl PrecompileHandle,
) -> Result<executor::stack::PrecompileOutput, PrecompileFailure> {
    handle.record_cost(output.cost.as_u64())?;
    for log in output.logs {
        handle.log(log.address, log.topics, log.data)?;
    }
    Ok(executor::stack::PrecompileOutput {
        exit_status: ExitSucceed::Returned,
        output: output.output,
    })
}
