### Title
Silent Precompile Failure Allows Permanent Freezing of ERC-20 Funds via Oversized `recipient` in `withdrawToNear` — (`etc/eth-contracts/contracts/EvmErc20.sol`, `EvmErc20V2.sol`)

### Summary
`EvmErc20.withdrawToNear` and `EvmErc20V2.withdrawToNear` burn the caller's ERC-20 tokens before calling the `ExitToNear` precompile, and never check the return value of that call. If the `recipient` argument is long enough to push the total precompile input past `MAX_INPUT_SIZE` (1 024 bytes), the precompile rejects the call silently, no NEP-141 `ft_transfer` promise is scheduled, and the burned tokens are permanently lost.

### Finding Description
Both `EvmErc20` and `EvmErc20V2` follow the same pattern:

1. `_burn(msg.sender, amount)` — irreversible.
2. Construct input: `\x01 || [sender (V2 only)] || amount_b || recipient`.
3. `assembly { let res := call(...) }` — `res` is never read.

The `ExitToNear` precompile calls `validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)` as its very first action. If `input.len() > 1_024`, it returns `Err(ExitError::Other("ERR_INVALID_INPUT"))`. The EVM translates this into a call return of `0`. Because the Solidity assembly block discards `res`, the outer transaction succeeds, the burn is committed, and no promise is ever created.

### Impact Explanation
**Critical — Permanent freezing of funds.** The ERC-20 tokens are burned on Aurora (supply reduced, balance zeroed). No NEP-141 `ft_transfer` is scheduled on NEAR. The underlying NEP-141 tokens remain locked in the connector contract forever with no recovery path for the user.

### Likelihood Explanation
**High.** The attack requires no privilege. Any holder of a nonzero ERC-20 balance can trigger it by calling `withdrawToNear` with `recipient.length >= 992` (EvmErc20) or `>= 972` (EvmErc20V2). This can happen accidentally (a user passing a long NEAR account ID or message) or deliberately (a griefing attack against another user's funds if the caller is a contract that accepts arbitrary recipient bytes).

### Recommendation
Add an explicit revert on precompile call failure in both contracts:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This must be applied to **both** `withdrawToNear` and `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`, since the same unchecked-return pattern exists in all four functions.

Alternatively, add a Solidity-level length guard before constructing the input:

```solidity
require(1 + 32 + recipient.length <= 1024, "recipient too long");
```

The precompile-side `MAX_INPUT_SIZE` guard is correct and should be kept as a defense-in-depth measure.

        // Craft a recipient that pushes total input over 1024 bytes:

        assert(before - afterBal == amount); // burn succeeded

    }
