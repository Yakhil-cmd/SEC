### Title
ERC-20 Tokens Permanently Burned When Exit Precompile Call Fails Silently — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` burn the caller's ERC-20 tokens **before** calling the `ExitToNear` or `ExitToEthereum` precompile via inline assembly. The assembly `call` return value (`res`) is captured but **never checked**. If the precompile returns failure for any reason, the EVM transaction still succeeds, the burn is committed, and the user permanently loses their tokens with no corresponding NEP-141 or Ethereum-side transfer.

---

### Finding Description

Both `withdrawToNear` and `withdrawToEthereum` in `EvmErc20.sol` follow the same unsafe pattern:

```solidity
// EvmErc20.sol – withdrawToNear (lines 53–63)
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — silent failure swallowed
    }
}
``` [1](#0-0) 

The identical unchecked-call pattern appears in `withdrawToEthereum`: [2](#0-1) 

And is replicated verbatim in `EvmErc20V2.sol`: [3](#0-2) [4](#0-3) 

The called precompile (`ExitToNear`) can return `ExitError` — causing the EVM `call` opcode to return `0` — under several conditions reachable by an unprivileged user:

1. **`ERR_INVALID_RECEIVER_ACCOUNT_ID`** — `parse_recipient` rejects any recipient bytes that are not valid UTF-8 or not a valid NEAR account ID. [5](#0-4) 

2. **`ERR_INVALID_AMOUNT`** — `parse_amount` rejects any `U256` value exceeding `u128::MAX`. Because `EvmErc20` uses Solidity `uint256`, a token with supply or balance above `u128::MAX` can trigger this path. [6](#0-5) 

3. **`ERR_TARGET_TOKEN_NOT_FOUND`** — `get_nep141_from_erc20` fails if the ERC-20 → NEP-141 mapping is absent or corrupted in engine storage. [7](#0-6) 

In every case, the `_burn` on line 54 has already executed and is not rolled back, because the assembly `call` failure is silently swallowed.

The `error_refund` feature provides a callback-based re-mint path (`refund_on_error`) for NEAR-side promise failures, but it is **not triggered** when the precompile itself returns an error before a promise is ever created: [8](#0-7) 

The test suite explicitly acknowledges that without `error_refund`, a failed `ft_transfer` promise results in permanent token loss: [9](#0-8) 

---

### Impact Explanation

**Critical — Permanent destruction of user funds.**

A user calling `withdrawToNear` or `withdrawToEthereum` on a deployed `EvmErc20` or `EvmErc20V2` contract loses their ERC-20 tokens with no recovery path when the precompile call fails silently. The EVM transaction reports success (no revert), so the user has no on-chain signal that the withdrawal failed. The burned tokens are gone; no NEP-141 or ETH is ever delivered.

---

### Likelihood Explanation

**Medium.**

- Any user who passes a recipient string that is not a valid NEAR account ID (e.g., contains `@`, uppercase letters, or invalid UTF-8) triggers the silent burn immediately.
- Any token whose per-account balance can exceed `u128::MAX` (≈ 3.4 × 10³⁸) triggers `ERR_INVALID_AMOUNT` after the burn.
- The `EvmErc20` contract is the standard bridge token deployed for every NEP-141 bridged to Aurora, making the attack surface broad.
- No special privilege is required; any token holder can call `withdrawToNear` or `withdrawToEthereum`.

---

### Recommendation

**Short term:** Add a `require(res != 0, "EXIT_PRECOMPILE_FAILED")` check immediately after each assembly `call` block in both `withdrawToNear` and `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. This causes the entire transaction to revert if the precompile fails, so `_burn` is also rolled back and the user's tokens are preserved.

**Long term:** Restructure the exit flow to call the precompile first (in a read-only simulation or with a pre-validation step) and only burn tokens after confirming the precompile will succeed, mirroring the recommendation in the original report to transfer rather than destroy tokens on failure.

---

### Proof of Concept

1. Alice holds 1000 units of a bridged NEP-141 token represented as `EvmErc20` on Aurora.
2. Alice calls `withdrawToNear(bytes("\xff\xfe"), 1000)` — the recipient bytes are invalid UTF-8.
3. `_burn(Alice, 1000)` executes; Alice's ERC-20 balance drops to 0.
4. The assembly `call` to `ExitToNear` precompile executes; `parse_recipient` returns `Err(ERR_INVALID_RECEIVER_ACCOUNT_ID)`; the precompile returns `ExitError`; the EVM `call` opcode returns `res = 0`.
5. The assembly block exits without checking `res`; `withdrawToNear` returns normally.
6. The EVM transaction is marked **successful**. Alice's 1000 ERC-20 tokens are permanently burned. No NEP-141 transfer is ever initiated. Alice has lost her funds with no on-chain indication of failure. [1](#0-0) [10](#0-9) [11](#0-10)

fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);

    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
    }

    Ok(amount)
}

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
