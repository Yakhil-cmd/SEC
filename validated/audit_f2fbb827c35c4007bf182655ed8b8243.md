### Title
Base Token Deposits to Non-Whitelisted Addresses in Silo Mode Permanently Freeze Funds — (File: `engine/src/engine.rs`)

---

### Summary

In silo mode with an active `Address` whitelist, `receive_base_tokens` mints ETH (base tokens) to any caller-specified recipient address without checking whether that address is whitelisted. Because non-whitelisted addresses cannot submit transactions in silo mode, any ETH minted to such an address is permanently frozen. The ERC-20 token path (`receive_erc20_tokens`) correctly implements a fallback-redirect guard for this exact scenario; the base token path has no equivalent protection.

---

### Finding Description

`receive_base_tokens` is called from `ft_on_transfer` when the predecessor is the authorized ETH connector account. It parses the recipient address from the caller-supplied `msg` field and unconditionally mints ETH to that address:

```rust
// engine/src/engine.rs L773-L790
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let receipient = message_data.recipient;
    let balance = get_balance(&self.io, &receipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &receipient, &new_balance);
    Ok(None)
}
```

There is no call to `silo::is_allow_receive_erc20_tokens` (which wraps `is_address_allowed`) and no fallback redirect.

Contrast this with `receive_erc20_tokens`, which explicitly guards against this case:

```rust
// engine/src/engine.rs L818-L822
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
```

The silo `Address` whitelist governs both receiving and submitting. `is_allow_submit` requires the sender address to be whitelisted:

```rust
// engine/src/contract_methods/silo/mod.rs L136-L138
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}
```

`is_address_allowed` returns `false` for any address not present in the enabled `WhitelistKind::Address` list:

```rust
// engine/src/contract_methods/silo/mod.rs L155-L158
fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
    !list.is_enabled() || list.is_exist(address)
}
```

A non-whitelisted address that receives ETH via `receive_base_tokens` therefore holds a balance it can never spend: it cannot call `submit`, cannot transfer ETH, and cannot invoke any EVM function. The ETH is irrecoverably locked unless the silo operator explicitly whitelists the address after the fact.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any ETH bridged to a non-whitelisted address in a silo instance is permanently inaccessible to the recipient. The balance exists in the EVM state but the address has no mechanism to move it. There is no on-chain recovery path that does not require operator intervention (adding the address to the whitelist), and the operator may be unaware of the deposit.

---

### Likelihood Explanation

**Medium.** Silo mode is a production deployment configuration. Any user who bridges ETH to an address that is not yet whitelisted — whether by mistake, because the whitelist was updated after the bridge call was submitted, or because the silo operator has a restricted whitelist — will have their funds frozen. The entry point (`ft_on_transfer` → `receive_base_tokens`) is a standard, publicly reachable NEP-141 receiver interface. No special privilege is required to trigger it; the caller only needs to send ETH through the authorized ETH connector.

---

### Recommendation

Apply the same fallback-redirect guard used in `receive_erc20_tokens` to `receive_base_tokens`. Before minting, check whether the recipient is allowed in the silo address whitelist and, if not, redirect to the configured `erc20_fallback_address` (or a dedicated `base_token_fallback_address`):

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let mut recipient = message_data.recipient;

    // Mirror the guard present in receive_erc20_tokens
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
        && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
    {
        recipient = fallback_address;
    }

    let balance = get_balance(&self.io, &recipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &recipient, &new_balance);
    Ok(None)
}
```

### Proof of Concept

1. Deploy Aurora Engine in silo mode with `WhitelistKind::Address` enabled and a non-empty whitelist. Configure `erc20_fallback_address` (so the ERC-20 path is protected).
2. Choose `victim_address` — an EVM address **not** present in the `Address` whitelist.
3. On the NEAR side, call `ft_transfer_call` on the ETH connector contract with `receiver_id = "aurora"` and `msg = "0x<victim_address>"`.
4. The ETH connector calls `ft_on_transfer` on Aurora. Because the predecessor is the connector account, `receive_base_tokens` is invoked.
5. `receive_base_tokens` parses `victim_address` from `msg` and calls `set_balance`, crediting ETH to `victim_address` with no whitelist check.
6. Any subsequent attempt by `victim_address` to call `submit` is rejected by `is_allow_submit` → `is_address_allowed` returning `false`.
7. The ETH balance at `victim_address` is permanently frozen. The ERC-20 path for the same address would have redirected to `fallback_address`; the base token path does not.
