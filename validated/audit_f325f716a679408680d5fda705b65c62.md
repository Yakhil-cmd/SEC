### Title
Integer Division Truncation in `native_per_pubdata` Calculation Allows Free Pubdata When `pubdata_price < native_price` - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In `validate_and_compute_fee_for_transaction`, the `native_per_pubdata` rate is computed via plain integer division `pubdata_price / native_price`. When `pubdata_price < native_price`, this truncates to `0`. A `native_per_pubdata` of `0` means every pubdata byte costs zero native resources, so any transaction can write unlimited state diffs to L1 without paying for them. The operator bears the full L1 data-availability cost.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` at line 142:

```rust
// We checked native_price != 0 above
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`pubdata_price` and `native_price` are independent U256 block-level parameters set by the operator. `wrapping_div` is plain integer division — no scaling, no rounding-up. When `pubdata_price < native_price`, the quotient is `0`, and `native_per_pubdata` is silently set to `0`.

This value is then passed into `create_resources_for_tx` and later into `get_resources_to_charge_for_pubdata`:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // = 0 when native_per_pubdata == 0
    .ok_or(out_of_native_resources!())?;
``` [2](#0-1) 

With `native_per_pubdata = 0`, the native cost for any amount of pubdata is `0`. The check `check_enough_resources_for_pubdata` always returns `enough = true`, and no native resources are ever charged for pubdata. [3](#0-2) 

The same truncation is replicated in the public API helper:

```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [4](#0-3) 

Note that `native_per_gas` is computed with `div_ceil` (rounds up), which correctly avoids truncation to zero:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [5](#0-4) 

But `native_per_pubdata` uses `wrapping_div` (floor division) with no analogous protection.

---

### Impact Explanation

`pubdata_price` is the operator-set price per byte of pubdata (state diffs published to L1). `native_price` is the operator-set cost per RISC-V proving cycle. These are economically independent quantities.

When `pubdata_price < native_price` (a realistic configuration — e.g., cheap L1 gas relative to proving cost), `native_per_pubdata` becomes `0`. Every transaction then has its pubdata cost zeroed out. An attacker can write many storage slots (each generating ~32 bytes of pubdata) and pay only EVM gas for `SSTORE` operations, while the operator must pay the full L1 data-availability cost for the resulting state diffs.

**Concrete example**:
- `native_price = 1000`, `pubdata_price = 999`
- `native_per_pubdata = 999 / 1000 = 0`
- Attacker writes 1000 storage slots → ~32,000 bytes of pubdata
- Pubdata cost charged to attacker: `32000 * 0 = 0` native
- Actual L1 cost borne by operator: `32000 * 999` wei

This is a direct operator fund-loss path.

---

### Likelihood Explanation

The `BlockContext` struct shows `pubdata_price` and `native_price` are independently configurable per block:

```rust
pub struct BlockContext {
    pub pubdata_price: U256,
    pub native_price: U256,
    ...
}
``` [6](#0-5) 

The default test configuration already sets `pubdata_price = U256::default() = 0` and `native_price = U256::from(10)`, meaning `native_per_pubdata = 0` by default in tests. [7](#0-6) 

In production, any block where the operator sets `pubdata_price < native_price` triggers the bug. This is a normal operating condition when L1 gas is cheap relative to proving costs. No attacker privilege is required — any unprivileged L2 transaction sender can exploit this.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` (ceiling division) for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, never silently zeroes):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix to `api/src/helpers.rs` line 427. [4](#0-3) 

---

### Proof of Concept

1. Operator sets block context: `native_price = 1000`, `pubdata_price = 999`, `eip1559_basefee = 1000`.
2. `native_per_pubdata = 999 / 1000 = 0` (integer truncation at line 142 of `validation_impl.rs`).
3. Attacker submits a transaction calling a contract that executes 100 `SSTORE` operations writing non-zero values to fresh storage slots.
4. Each `SSTORE` generates ~32 bytes of pubdata (state diff). Total: ~3200 bytes.
5. `get_resources_to_charge_for_pubdata` computes `3200 * 0 = 0` native cost.
6. `check_enough_resources_for_pubdata` returns `enough = true`.
7. Transaction succeeds. Attacker paid only EVM gas for `SSTORE`. Operator pays L1 cost for 3200 bytes of pubdata at `pubdata_price = 999` per byte.
8. Repeat with many transactions to drain operator funds. [1](#0-0) [8](#0-7)
