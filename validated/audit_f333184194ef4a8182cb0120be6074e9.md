### Title
Missing `chain_id` Validation in `interop_root_reporter_event_hook` Allows Zero-Chain-ID Interop Root to Corrupt Batch Public Input — (`File: system_hooks/src/event_hooks/interop_root_reporter.rs`)

---

### Summary

`interop_root_reporter_event_hook` validates the event signature and ABI-encoded data format of an `InteropRootAdded` event, but does **not** validate that `chain_id != 0`. The `InteropRoot` struct explicitly documents `chain_id` as "must be non-zero", yet no enforcement exists anywhere in the Rust system layer. A zero `chain_id` stored in `InteropRootStorage` propagates directly into the `interop_roots_rolling_hash` field of `BatchPublicInput`, corrupting the ZK proof public input committed to the settlement layer.

---

### Finding Description

The event hook is registered exclusively for events emitted from `L2_INTEROP_ROOT_STORAGE_ADDRESS` (0x10008):

```rust
pub fn add_interop_root_reporter<S: EthereumLikeTypes, A: Allocator + Clone>(
    hooks: &mut HooksStorage<S, A>,
) -> Result<(), InternalError> {
    hooks.add_event_hook(
        L2_INTEROP_ROOT_STORAGE_ADDRESS_LOW,
        SystemEventHook::new(interop_root_reporter_event_hook),
    )
}
``` [1](#0-0) 

Inside the hook, the following parameters are validated: event signature (topic[0]), data length (96 bytes), ABI offset (must be 32), array length (must be 1), and topics count (must be 3). However, after parsing `chain_id` from `topics[1]`, it is passed directly to `add_interop_root` with **no zero-check**:

```rust
let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
system.io.add_interop_root(
    ExecutionEnvironmentType::NoEE,
    resources,
    InteropRoot {
        root,
        block_or_batch_number,
        chain_id,   // ← never checked for zero
    },
)?;
``` [2](#0-1) 

The `InteropRoot` struct explicitly documents the invariant that is never enforced:

```rust
/// Source chain identifier (must be non-zero)
pub chain_id: U256,
``` [3](#0-2) 

Neither `push_root` in `InteropRootStorage` nor `add_interop_root` in the IO subsystem performs any validation: [4](#0-3) [5](#0-4) 

The stored roots are then consumed by `calculate_interop_roots_rolling_hash`, which encodes `chain_id` directly into the rolling hash:

```rust
data[32..64].copy_from_slice(&root.chain_id.to_be_bytes::<{ U256::BYTES }>());
``` [6](#0-5) 

This rolling hash is committed into `BatchOutput.interop_roots_rolling_hash`, which is hashed into the final `BatchPublicInput` verified on the settlement layer: [7](#0-6) 

---

### Impact Explanation

A zero `chain_id` in the rolling hash produces a `BatchPublicInput` that encodes a semantically invalid interop root (chain 0 does not exist). The settlement layer contract, which validates the batch public input, is expected to reject such a commitment — causing a **valid-execution unprovability** scenario: the forward system accepts and finalizes the block, but the resulting proof cannot be verified on-chain. This breaks the rollup's ability to settle, constituting a liveness/integrity failure for the interop subsystem.

Additionally, if the settlement layer does not independently validate `chain_id != 0`, a zero-chain-ID root could be used to forge cross-chain state proofs against a non-existent source chain, enabling false interop message verification.

---

### Likelihood Explanation

The hook fires for any `InteropRootAdded` event emitted by the contract at `L2_INTEROP_ROOT_STORAGE_ADDRESS` (0x10008). This contract is a regular EVM contract callable via L2 transactions. If the contract's `addInteropRootsInBatch` function does not restrict callers or does not validate `chainId != 0` on its own, any unprivileged user can submit a transaction with `chainId = 0` to trigger the path. The existing test suite validates that `root = 0` causes a revert (enforced in the contract), but there is **no corresponding test for `chain_id = 0`**, suggesting this invariant is not enforced at the contract level either. [8](#0-7) 

---

### Recommendation

Add an explicit `chain_id != 0` guard in `interop_root_reporter_event_hook` immediately after parsing `chain_id`, mirroring the existing `root != 0` enforcement in the contract:

```rust
let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
if chain_id.is_zero() {
    return Err(internal_error!(
        "Interop root reporter event hook: chain_id must be non-zero"
    ).into());
}
```

Additionally, enforce the same invariant in `push_root` / `add_interop_root` as a defense-in-depth measure, consistent with the documented contract of `InteropRoot`.

---

### Proof of Concept

1. Deploy or interact with the contract at `L2_INTEROP_ROOT_STORAGE_ADDRESS` (0x10008).
2. Call `addInteropRootsInBatch` with an `InteropRoot` where `chainId = 0`, `blockOrBatchNumber = 1`, `sides = [nonzero_bytes32]`.
3. The contract emits `InteropRootAdded(chainId=0, blockOrBatchNumber=1, sides=[...])`.
4. `try_intercept_event` dispatches to `interop_root_reporter_event_hook`.
5. The hook passes all format checks and calls `add_interop_root` with `chain_id = U256::ZERO`.
6. `push_root` stores `InteropRoot { chain_id: 0, ... }` without error.
7. At batch finalization, `calculate_interop_roots_rolling_hash` encodes `chain_id = 0` into the rolling hash.
8. The resulting `BatchPublicInput` contains a corrupted `interop_roots_rolling_hash` that the settlement layer rejects, making the batch unprovable. [9](#0-8) [10](#0-9)

```rust
pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // First, ensure we're capturing the InteropRootAdded event
    if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
        return Ok(());
    }
    // Internal error if the data supplied doesn't match the expected value
    if data.len() != 96 {
        return Err(internal_error!("Interop root reporter event hook received bad data").into());
    }

    // Parse data
    let offset: u32 = match U256::from_be_slice(&data[..32]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad offset").into(),
            );
        }
    };
    // This event is part of the system, but we check it anyways
    if offset != 32 {
        return Err(internal_error!("Interop root reporter event hook received bad offset").into());
    }

    let len: u32 = match U256::from_be_slice(&data[32..64]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad length").into(),
            );
        }
    };
    // It should have exactly one side
    if len != 1 {
        return Err(internal_error!("Interop root reporter event hook received bad length").into());
    }
    // Validate topics length
    if topics.len() != 3 {
        return Err(internal_error!("Interop root reporter event hook received bad topics").into());
    }

    let root = Bytes32::from_array(data[64..96].try_into().unwrap());
    let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
    system.io.add_interop_root(
        ExecutionEnvironmentType::NoEE,
        resources,
        InteropRoot {
            root,
            block_or_batch_number,
            chain_id,
        },
    )?;

    Ok(())
}

```rust
pub fn calculate_interop_roots_rolling_hash<'a>(
    old_rolling_hash: Bytes32,
    roots: impl Iterator<Item = &'a InteropRoot>,
    hasher: &mut crypto::sha3::Keccak256,
) -> Bytes32 {
    let mut data = [0u8; 96];

    let mut rolling_hash = old_rolling_hash;
    for root in roots {
        data[0..32].copy_from_slice(&rolling_hash.as_u8_ref());
        data[32..64].copy_from_slice(&root.chain_id.to_be_bytes::<{ U256::BYTES }>());
        data[64..96].copy_from_slice(&root.block_or_batch_number.to_be_bytes::<{ U256::BYTES }>());
        hasher.update(data);

        // Note: now we have only one side
        hasher.update(root.root.as_u8_ref());

        rolling_hash = hasher.finalize_reset().into()
    }

    rolling_hash
}

    ///

    pub fn hash(&self) -> [u8; 32] {
        let mut hasher = Keccak256::new();
        hasher.update(self.chain_id.to_be_bytes::<32>());
        hasher.update(&self.first_block_timestamp.to_be_bytes());
        hasher.update(&self.last_block_timestamp.to_be_bytes());

        hasher.update(self.pubdata_commitment.as_u8_ref());
