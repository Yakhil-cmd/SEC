### Title
SNS Swap `Init` Stores Immutable Copies of `neuron_minimum_stake_e8s` / `transaction_fee_e8s` That Diverge from Updatable `NervousSystemParameters` in SNS Governance — (`rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto`)

---

### Summary

The SNS Swap canister's `Init` struct stores `neuron_minimum_stake_e8s` and `transaction_fee_e8s` at creation time and **cannot modify them afterwards**. The same two values are independently stored in SNS Governance's `NervousSystemParameters`, which **can be updated** at any time via a `ManageNervousSystemParameters` governance proposal. The protocol code itself acknowledges the risk with the comment: *"Whether the values match is not checked. If they don't match things will break."* When the values diverge, swap finalization fails because the swap computes neuron recipes using its stale copy while governance validates those recipes against its updated copy.

---

### Finding Description

**Location 1 — Immutable swap-side copy:**

The swap canister's `Init` message is explicitly documented as unmodifiable after creation:

> "The initialisation data of the canister. Always specified on canister creation, and cannot be modified afterwards." [1](#0-0) 

The proto comment for both fields explicitly warns of the divergence risk:

```
// Same as SNS ledger. Must hold the same value as SNS ledger. Whether the
// values match is not checked. If they don't match things will break.
optional uint64 transaction_fee_e8s = 13;

// Same as SNS governance. Must hold the same value as SNS governance. Whether
// the values match is not checked. If they don't match things will break.
optional uint64 neuron_minimum_stake_e8s = 14;
``` [2](#0-1) 

The `Init::validate()` function only checks that the fields are *present*, not that they match governance: [3](#0-2) 

**Location 2 — Mutable governance-side copy:**

`NervousSystemParameters` in SNS Governance stores the same two fields and is updated in-place by `perform_manage_nervous_system_parameters` whenever a `ManageNervousSystemParameters` proposal executes: [4](#0-3) 

The integration test confirms this is a live, exercised code path: [5](#0-4) 

**The divergence path during finalization:**

When the swap finalizes, it calls `claim_swap_neurons` on SNS Governance. Governance validates each neuron recipe's `stake_e8s` against its own (potentially updated) `neuron_minimum_stake_e8s`: [6](#0-5) 

The swap computed those recipes using the stale value from its `Init`. If governance's `neuron_minimum_stake_e8s` was raised after swap initialization, the recipes will be rejected, and finalization fails.

The `swap_init_args` function in `SnsInitPayload` copies the values into the swap's `Init` at creation time with no subsequent synchronization mechanism: [7](#0-6) 

---

### Impact Explanation

If `neuron_minimum_stake_e8s` or `transaction_fee_e8s` in SNS Governance diverges from the swap's `Init` copy:

1. **Swap finalization fails permanently.** The swap computes neuron recipes using the stale minimum stake. Governance rejects them. The swap cannot commit, and participants' ICP contributions are locked until the swap aborts.
2. **Incorrect participation validation.** The swap uses its stale `transaction_fee_e8s` to validate that `min_participant_icp_e8s` is large enough to form neurons. If the ledger fee was raised, the swap's validation passes but actual neuron creation fails at the ledger transfer step.
3. **No recovery path.** The swap `Init` is immutable. There is no admin function to re-sync the values. The only recovery is swap abortion, which refunds ICP but destroys the decentralization event.

---

### Likelihood Explanation

Any SNS token holder with sufficient voting power can submit a `ManageNervousSystemParameters` proposal. The SNS governance starts in `PreInitializationSwap` mode during the swap, but the code does not explicitly block `ManageNervousSystemParameters` proposals in that mode — developer neurons created at SNS launch have voting power and can pass such proposals. A malicious developer neuron holder could intentionally raise `neuron_minimum_stake_e8s` to a value that breaks finalization, causing the swap to abort and refunding ICP while preventing decentralization. The code comment *"If they don't match things will break"* confirms the developers are aware of the risk but have not added a guard.

---

### Recommendation

1. **Remove the duplicate.** During swap finalization, have the swap canister query `NervousSystemParameters` from governance at finalization time rather than relying on its stale `Init` copy.
2. **Or add a sync function.** Add an admin-callable (NNS governance) function on the swap canister to update `transaction_fee_e8s` and `neuron_minimum_stake_e8s` from the live governance/ledger values.
3. **Or add a guard.** In `claim_swap_neurons`, cross-check the incoming recipe stakes against the swap canister's `Init` values rather than governance's live parameters, so the two copies are used consistently.
4. **At minimum**, block `ManageNervousSystemParameters` proposals that change `neuron_minimum_stake_e8s` or `transaction_fee_e8s` while the SNS is in `PreInitializationSwap` mode.

---

### Proof of Concept

1. Deploy an SNS with `neuron_minimum_stake_e8s = 100_000_000` (1 token). The swap `Init` records this value.
2. Developer neurons (created at SNS launch) submit and pass a `ManageNervousSystemParameters` proposal setting `neuron_minimum_stake_e8s = 500_000_000` (5 tokens) in governance.
3. The swap reaches its end date with sufficient participation. Each participant contributed enough ICP to receive neuron baskets of 1-token neurons (valid under the old minimum).
4. The swap calls `claim_swap_neurons` with recipes having `stake_e8s = 100_000_000`.
5. Governance's `claim_swap_neurons` validates `stake_e8s >= neuron_minimum_stake_e8s_or_panic()` which now returns `500_000_000`. The recipes fail validation.
6. Swap finalization returns an error. The swap cannot commit. Participants' ICP is locked until the swap aborts. [8](#0-7) [9](#0-8) [10](#0-9)

    /// Executes a ManageNervousSystemParameters proposal by updating Governance's
    /// NervousSystemParameters
    fn perform_manage_nervous_system_parameters(
        &mut self,
        proposed_params: NervousSystemParameters,
    ) -> Result<(), GovernanceError> {
        // Only set `self.proto.parameters` if "applying" the proposed params to the
        // current params results in valid params
        let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());

        log!(
            INFO,
            "Setting Governance nervous system params to: {:?}",
            &new_params
        );

        match new_params.validate() {
            Ok(()) => {
                self.proto.parameters = Some(new_params);
                Ok(())
            }

            // Even though proposals are validated when they are first made, this is still
            // possible, because the inner value of a ManageNervousSystemParameters
            // proposal is only valid with respect to the current
            // nervous_system_parameters() at the time when the proposal was first
            // made. If nervous_system_parameters() changed (by another proposal) since
            // the current proposal was first made, the current proposal might have become
            // invalid. Basically, this might occur if there are conflicting (concurrent)
            // proposals, but we expect this to be highly unusual in practice.
            Err(msg) => Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!(
                    "Failed to perform ManageNervousSystemParameters action, proposed \
                        parameters would lead to invalid NervousSystemParameters: {msg}"
                ),
            )),
        }
    }
