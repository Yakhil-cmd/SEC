### Title
Unbounded `ingress_expiries` Loop in `construction_payloads` Causes OOM via Attacker-Controlled `ingress_start`/`ingress_end` — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

---

### Summary

An unprivileged HTTP client can supply `ingress_start=0` and `ingress_end=u64::MAX` in a `POST /construction/payloads` request to the ICRC-1 Rosetta server. Both input-validation guards pass for these values, and the subsequent `while ingress_start < ingress_end` loop runs for approximately **153 billion iterations**, allocating ~1.23 TB of heap memory before the process is killed by the OS OOM killer.

---

### Finding Description

The vulnerable function is `construction_payloads` in: [1](#0-0) 

**Step 1 — Attacker-controlled inputs accepted as `u64`:**

`ingress_start` and `ingress_end` are `Option<u64>` fields in `ConstructionPayloadsRequestMetadata`, deserialized directly from the JSON body with no range restriction. [2](#0-1) 

**Step 2 — Both guards are bypassable:**

Guard 1 (`ingress_start >= ingress_end`): with `ingress_start=0` and `ingress_end=u64::MAX`, `0 >= u64::MAX` is `false` → passes. [3](#0-2) 

Guard 2 (`ingress_end < now + ingress_interval`): `now` ≈ 1.7×10¹⁸ ns (current epoch time), `ingress_interval` = 240×10⁹ ns, so `now + ingress_interval` ≈ 1.7×10¹⁸. `u64::MAX` ≈ 1.84×10¹⁹ >> 1.7×10¹⁸, so `u64::MAX < now + ingress_interval` is `false` → passes. [4](#0-3) 

**Step 3 — The unbounded loop:**

```rust
while ingress_start < ingress_end {
    ingress_expiries.push(ingress_start + ingress_interval);
    ingress_start +=
        ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
}
``` [5](#0-4) 

**Concrete arithmetic:**

| Constant | Value |
|---|---|
| `MAX_INGRESS_TTL` | 300 s = 300,000,000,000 ns |
| `PERMITTED_DRIFT` | 60 s = 60,000,000,000 ns |
| `ingress_interval` | 240,000,000,000 ns |
| `INGRESS_INTERVAL_OVERLAP` | 120 s = 120,000,000,000 ns |
| **Loop step** | 240,000,000,000 − 120,000,000,000 = **120,000,000,000 ns** | [6](#0-5) [7](#0-6) 

With `ingress_start=0` and `ingress_end=u64::MAX = 18,446,744,073,709,551,615`:

- **Iterations before `ingress_start` wraps**: `u64::MAX / 120,000,000,000 ≈ 153,722,867,280`
- **Heap allocated** (each push is 8 bytes): `153,722,867,280 × 8 ≈ 1.23 TB`
- In Rust release builds, `u64` addition wraps on overflow, so `ingress_start` wraps back to a small value and the loop **never terminates** — it becomes an infinite loop with unbounded allocation.

The `ingress_expiries` vector is then passed directly into `handle_construction_payloads`, which iterates it again to build `signing_payloads` and `envelope_contents`: [8](#0-7) 

---

### Impact Explanation

A single malicious HTTP POST to `/construction/payloads` causes the Rosetta server process to exhaust all available memory and be killed by the OOM killer. This denies service to all legitimate ledger users relying on the Rosetta API for transfers, approvals, and balance queries. No authentication or privileged access is required.

---

### Likelihood Explanation

The attack is trivially reproducible with a single `curl` command. The two guards are both bypassable with boundary values (`0` and `u64::MAX`) that are valid `u64` integers. There is no rate limiting, request size cap, or envelope count cap anywhere in the call path. Any public-facing ICRC-1 Rosetta deployment is affected.

---

### Recommendation

Add an explicit cap on the number of generated ingress expiries immediately after the guards, before the loop:

```rust
const MAX_INGRESS_EXPIRIES: usize = 300; // e.g. 24h / 5min = 288, round up

if ingress_end.saturating_sub(ingress_start) / step > MAX_INGRESS_EXPIRIES as u64 {
    return Err(Error::processing_construction_failed(
        "ingress window too large: would generate too many signing payloads",
    ));
}
```

Alternatively, break out of the loop once `ingress_expiries.len() >= MAX_INGRESS_EXPIRIES`. Also consider using `checked_add` for `ingress_start` inside the loop to prevent silent wrapping.

---

### Proof of Concept

```bash
curl -X POST http://<rosetta-host>/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"<ledger-id>"},
    "operations": [<valid_transfer_op>],
    "public_keys": [<valid_pubkey>],
    "metadata": {
      "ingress_start": 0,
      "ingress_end": 18446744073709551615
    }
  }'
```

The server process will exhaust memory and be killed. The assertion `envelope_count <= 288` will fail with a count orders of magnitude larger.
