### Title
Silent Precompile Failure Burns ERC-20 Tokens Without Bridging Them to NEAR — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.withdrawToNear` and `EvmErc20.withdrawToEthereum` burn the caller's tokens unconditionally before invoking the exit precompile via inline assembly. The return value of the `call` opcode is captured in a local variable but **never inspected**. When the precompile returns failure (which it does for several reachable input conditions), the burn is already committed and the tokens are permanently destroyed with no corresponding credit on NEAR or Ethereum.

---

### Finding Description

In both `EvmErc20.sol` and `EvmErc20V2.sol`, the withdrawal functions follow this pattern:

```solidity
// etc/eth-contracts/contracts/EvmErc20.sol  lines 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // ← burn is unconditional

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is never read — no revert on failure
    }
}
```

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) returns `Err(ExitError)` — causing the EVM `call` to return `0` — in at least the following reachable cases:

| Condition | Error |
|---|---|
| Recipient bytes are not valid UTF-8 | `ERR_INVALID_RECEIVER_ACCOUNT_ID` |
| Recipient string is not a valid NEAR account ID | `ERR_INVALID_RECEIVER_ACCOUNT_ID` |
| Amount exceeds `u128::MAX` | `ERR_INVALID_AMOUNT` |
| Called in a static context | `ERR_INVALID_IN_STATIC` |
| Out of gas | `OutOfGas` | [1](#0-0) [2](#0-1) [3](#0-2) 

Because `res` is never checked, the Solidity function returns `success` to the caller even when the precompile has rejected the exit. The ERC-20 burn is already finalised; there is no rollback path.

The same pattern is present in `withdrawToEthereum` (lines 65–76 of `EvmErc20.sol`) and identically in `EvmErc20V2.sol` (lines 53–77). [4](#0-3) [5](#0-4) 

The engine test suite explicitly acknowledges that without the optional `error_refund` feature flag the lost balance is **not recovered**:

```rust
// engine-tests/src/tests/erc20_connector.rs  lines 771-775
#[cfg(feature = "error_refund")]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE);
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [6](#0-5) 

---

### Impact Explanation

Any user who calls `withdrawToNear` or `withdrawToEthereum` with a recipient that the precompile rejects (e.g. a byte string that is not a valid NEAR account ID) will have their ERC-20 tokens permanently burned with zero credit on the destination chain. This is **Critical — Permanent freezing / destruction of user funds**.

---

### Likelihood Explanation

The `recipient` parameter is a raw `bytes` argument with no on-chain validation before the burn. A user who mistypes a NEAR account ID, passes an empty byte string, or interacts through a third-party contract that supplies a malformed recipient will silently lose their tokens. The path is reachable by any unprivileged EVM user with a token balance.

---

### Recommendation

Check the return value of the assembly `call` and revert if it indicates failure, so the burn is rolled back atomically:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures the burn and the bridge exit are atomic: either both succeed or neither does.

---

### Proof of Concept

1. Deploy `EvmErc20` with a NEP-141 mirror (standard setup).
2. Mint tokens to `alice`.
3. `alice` calls `withdrawToNear(bytes("not!!a!!valid.near.account"), 100)`.
4. `_burn(alice, 100)` executes — alice's balance drops by 100.
5. The precompile rejects the recipient (`ERR_INVALID_RECEIVER_ACCOUNT_ID`), the `call` returns `0`.
6. The assembly block ignores `res`; the function returns without reverting.
7. Alice's ERC-20 balance is 100 less; she receives nothing on NEAR.
8. The tokens are permanently destroyed. [3](#0-2) [1](#0-0) [7](#0-6)

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
