Audit Report

## Title
Wasm64 Fee Overflow in `charge_direct_fee` Skips Instruction Counter Decrement, Enabling Near-Free Execution — (`rs/embedders/src/wasmtime_embedder/linker.rs`)

## Summary

In Wasm64 mode, any canister can call a byte-charging system API (e.g., `ic0_msg_arg_data_copy`) with `size = i64::MAX`. The computed fee exceeds `i64::MAX`, causing `fee.get().try_into::<i64>()` to fail at line 179 of `linker.rs`. The early return via `?` exits `charge_direct_fee` before `store_value` is called at line 185, leaving the Wasm instruction counter global unchanged. After execution terminates, `message_instructions_left` is computed from this un-decremented counter and is nearly equal to `message_instruction_limit`, causing `refund_unused_execution_cycles` to issue a near-full refund of prepaid execution cycles. The canister pays almost nothing for the message execution.

## Finding Description

**Root cause — `charge_direct_fee` early return without counter update:**

In `rs/embedders/src/wasmtime_embedder/linker.rs`, `charge_direct_fee` loads the instruction counter, then attempts to convert the `u64` fee to `i64`:

```rust
let fee = fee.get().try_into().map_err(|_| {
    HypervisorError::InstructionLimitExceeded(NumInstructions::from(instruction_limit as u64))
})?;
// Now we can subtract the fee and store the new instruction counter.
instruction_counter = instruction_counter.saturating_sub(fee);
store_value(&num_instructions_global, instruction_counter, caller)?;
```

When `fee > i64::MAX`, the `?` operator returns early. `store_value` at line 185 is never reached. The Wasm global holding the instruction counter retains its pre-call value. [1](#0-0) 

**Trigger path — Wasm64 `msg_arg_data_copy` with `size = i64::MAX`:**

In Wasm64 mode, `syscalls::<u64>` is used. The `size` parameter is `u64`. With `size = 9223372036854775807` (i64::MAX), the conversion to `usize` succeeds on 64-bit systems. `charge_for_system_api_call` computes `fee = overhead + size = ~500 + 9223372036854775807 = 9223372036854776307`, which exceeds `i64::MAX`. The overflow path is triggered. [2](#0-1) [3](#0-2) [4](#0-3) 

**Inflated `instructions_left` causes over-refund:**

After execution, `slice_instructions_executed` computes `current_slice_instruction_limit - instruction_counter`. Since the counter was not decremented, this value is nearly zero (only the few Wasm instructions before the system API call). `message_instructions_left ≈ message_instruction_limit`. [5](#0-4) [6](#0-5) 

`refund_unused_execution_cycles` then refunds cycles proportional to `instructions_left`, which is nearly the full prepaid amount: [7](#0-6) 

**Wasm64 is unconditionally enabled:**

The embedder detects Wasm64 by checking if the heap memory is 64-bit and selects `syscalls::<u64>` accordingly. No feature flag or governance gate blocks deployment. [8](#0-7) 

**Existing test confirms the overflow path is reachable but does not verify instruction accounting:**

The test `ic0_debug_print_with_large_memory_wasm64` confirms `InstructionLimitExceeded` is returned for a large Wasm64 size argument, but does not assert `instructions_left == 0` or that cycles are fully charged. [9](#0-8) 

The `TODO: RUN-841: Cover with tests` comments on both `charge_for_system_api_call` and `charge_direct_fee` confirm this code path has no accounting correctness tests. [10](#0-9) [11](#0-10) 

## Impact Explanation

A Wasm64 canister calling `ic0_msg_arg_data_copy(0, 0, i64::MAX)` as its first instruction will prepay for `message_instruction_limit` (e.g., 40B) instructions worth of cycles, trigger the overflow, and receive a refund for nearly all 40B instructions. Net cost per call: only the handful of Wasm instructions executed before the system API call. This is repeatable indefinitely, allowing a canister to consume subnet computation resources without paying for them — constituting ongoing cycles theft from the subnet. This matches the **Critical** impact class: theft or protocol insolvency involving exorbitant cycles, easy to perform without special privileges, low cost, and severe global impact across all subnets running Wasm64 canisters.

## Likelihood Explanation

- Requires only deploying a Wasm64 canister (no privileged access, no governance, no key material)
- The exploit is a single system API call with a crafted `size` argument
- Wasm64 is production-enabled and already used by real canisters
- The attack is repeatable indefinitely with no per-call cost increase
- The existing test `ic0_debug_print_with_large_memory_wasm64` demonstrates the overflow path is reachable in the production test suite

## Recommendation

1. **Immediate fix**: In `charge_direct_fee`, when `fee > i64::MAX`, set the instruction counter to `i64::MIN` (or 0) and call `out_of_instructions` before returning the error. This ensures `instructions_left = 0` regardless of the overflow path:
   ```rust
   let fee: i64 = fee.get().try_into().unwrap_or_else(|_| {
       // Fee exceeds i64::MAX; treat as consuming all remaining instructions.
       instruction_counter = i64::MIN;
       i64::MAX
   });
   ```
   Or equivalently, cap the fee at `instruction_limit` before the conversion, since any fee exceeding the limit is equivalent to consuming all remaining instructions.

2. **Add tests** asserting `instructions_left == 0` when `fee > i64::MAX`, as noted by the existing `TODO: RUN-841` comments.

## Proof of Concept

```wat
(module
  (import "ic0" "msg_arg_data_copy"
    (func $ic0_msg_arg_data_copy (param i64 i64 i64)))
  (memory i64 1)
  (func (export "canister_update exploit")
    ;; size = i64::MAX = 9223372036854775807
    (call $ic0_msg_arg_data_copy
      (i64.const 0)
      (i64.const 0)
      (i64.const 9223372036854775807))
  )
)
```

Deploy this Wasm64 canister and call `exploit` repeatedly. Each call prepays ~40B instructions worth of cycles but receives a near-full refund. A unit test asserting `instructions_left == 0` after this call would confirm the bug. The existing test infrastructure in `rs/embedders/tests/wasmtime_embedder.rs` and `rs/execution_environment/tests/hypervisor.rs` provides the scaffolding needed to write such a test.

fn charge_for_system_api_call(
    caller: &mut Caller<'_, StoreData>,
    overhead: NumInstructions,
    num_bytes: usize,
) -> HypervisorResult<()> {
    let system_api = caller.data_mut().system_api()?;
    let bytes_charge = system_api.get_num_instructions_from_bytes(NumBytes::from(num_bytes as u64));
    let overhead = overhead.saturating_add(&bytes_charge);

fn ic0_debug_print_with_large_memory_wasm64() {

        .with_execution_config(config)
        .build();

        (module
            (import "ic0" "debug_print" (func $ic0_debug_print (param i64) (param i64)))
            (import "ic0" "msg_reply" (func $msg_reply))
            (func $test (export "canister_update test")
                (local $i i64)
                ;; Call debug_print with offset 0 and large negative size
                ;; (when interpreted as unsigned, this is a huge number)

                ;; Loop to a large count
                (loop $my_loop
                    local.get $i
                    i64.const 1
                    i64.add
                    local.tee $i
                    i64.const 9223372034854775815
                    i64.lt_s
                    br_if $my_loop
                )
                (call $msg_reply)
            )
            (memory i64 49000)
        )"#;
    // Give the canister a trillion cycles to ensure it doesn't run out

        .canister_from_cycles_and_wat(initial_cycles, wat)
        .unwrap();

    let result = test.ingress(canister_id, "test", vec![]);

        ErrorCode::CanisterInstructionLimitExceeded,

    );
