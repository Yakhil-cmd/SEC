### Title
XRC Unavailability Permanently Blocks NNS Neuron Maturity Disbursement Finalization - (`File: rs/nns/governance/src/governance/disburse_maturity.rs`)

---

### Summary

The NNS Governance canister's `finalize_maturity_disbursement` timer task requires a valid `maturity_modulation.current_value_permyriad` to be present in state before it will finalize any pending maturity disbursement. This value is populated by a separate daily timer task (`UpdateIcpXdrRateRelatedData`) that calls the Exchange Rate Canister (XRC). If the XRC is persistently unavailable or returns only zero-rate responses during the 365-day backfill window, `maturity_modulation.current_value_permyriad` remains `None`, and every call to `finalize_maturity_disbursement` returns `FinalizeMaturityDisbursementError::NoMaturityModulation` — permanently blocking all neuron owners from receiving their maturity disbursements, even though the disbursement delay has fully elapsed.

---

### Finding Description

When a neuron owner calls `DisburseMaturity`, the maturity is immediately deducted from the neuron and a `MaturityDisbursement` record is queued with a `finalize_disbursement_timestamp_seconds` set 7 days in the future. [1](#0-0) 

After 7 days, the timer task `finalize_maturity_disbursement` is supposed to mint ICP to the destination account. However, the very first check inside `next_maturity_disbursement_to_finalize` is: [2](#0-1) 

This reads `maturity_modulation.current_value_permyriad` from governance state: [3](#0-2) 

The `current_value_permyriad` field is populated by the `UpdateIcpXdrRateRelatedData` timer task, which calls the XRC canister to build a 365-day ICP price history and then computes the modulation. If the XRC is unavailable or returns zero rates, the `update_maturity_modulation` function leaves `current_value_permyriad` unchanged (i.e., `None` on a fresh canister): [4](#0-3) 

The CHANGELOG confirms this was a known issue and was partially fixed (for the XRC backfill stall), but the fix only addresses the case where the cursor advances past failed days — it does not address the case where `current_value_permyriad` is still `None` when disbursements are due: [5](#0-4) 

The test `test_finalize_maturity_disbursement_no_maturity_modulation` explicitly confirms the blocking behavior: [6](#0-5) 

The error is treated as a retryable failure with a 60-second retry interval, but since the root cause (missing modulation) is not resolved by retrying, the disbursement is permanently blocked until the XRC becomes available and the modulation is computed. [7](#0-6) 

---

### Impact Explanation

A neuron owner who calls `DisburseMaturity` has their maturity **immediately and irrevocably deducted** from the neuron. The maturity is gone from the neuron's balance. If the XRC is unavailable for an extended period (or if the 365-day backfill has not yet completed on a fresh governance canister), the ICP minting step is permanently blocked. The neuron owner loses access to their maturity — it is neither in the neuron nor minted to their account — for the entire duration of the XRC outage. This is a direct financial loss of user funds during the outage window.

---

### Likelihood Explanation

The XRC is an on-chain canister (`rkp4c-7iaaa-aaaaa-aaaca-cai`) that itself depends on canister HTTP outcalls to external price feeds. Any period of XRC unavailability (e.g., XRC canister upgrade, HTTP outcall consensus failures, or the XRC returning only zero rates) will prevent `current_value_permyriad` from being set. Additionally, on a freshly deployed or upgraded governance canister, the 365-day backfill takes approximately 30 minutes at 5-second intervals — during this window, any disbursements that come due will be blocked. The CHANGELOG entry for Proposal 141771 confirms this scenario has already occurred in practice. [8](#0-7) 

---

### Recommendation

When `current_value_permyriad` is `None` and disbursements are past their `finalize_disbursement_timestamp_seconds`, the finalization should fall back to a neutral modulation value (0 permyriad, i.e., no adjustment) rather than blocking indefinitely. This is analogous to the fix applied in the Y2K Finance report: the price data is used to *adjust* the disbursement amount, not to gate whether disbursement happens at all. A neuron owner who waited 7 days should not be denied their maturity because an external oracle is temporarily unavailable.

Concretely, in `next_maturity_disbursement_to_finalize`, replace the hard `ok_or(NoMaturityModulation)?` with a fallback to `0i32` (neutral modulation) when the value is absent, and log a warning. Alternatively, gate the `NoMaturityModulation` error only on disbursements that are not yet overdue, and use a neutral fallback for overdue disbursements.

---

### Proof of Concept

1. Neuron owner calls `DisburseMaturity` — maturity is deducted from neuron immediately.
2. XRC canister becomes unavailable (e.g., upgrade, HTTP outcall failures).
3. `UpdateIcpXdrRateRelatedData` timer fails to fetch rates; `current_value_permyriad` remains `None`.
4. After 7 days, `finalize_maturity_disbursement` timer fires.
5. `try_finalize_maturity_disbursement` reads `maturity_modulation.current_value_permyriad` → `None`.
6. `next_maturity_disbursement_to_finalize` returns `Err(NoMaturityModulation)`.
7. `finalize_maturity_disbursement` logs the error and schedules a retry in 60 seconds.
8. Steps 4–7 repeat indefinitely while XRC is unavailable.
9. Neuron owner's maturity is neither in the neuron nor minted — funds are inaccessible.
