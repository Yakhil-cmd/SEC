### Title
Temporary `mint_base_token` System Hook Explicitly Marked for Pre-Release Removal Remains Registered in Production Bootloader — (`File: basic_bootloader/src/bootloader/block_flow/zk/post_init_op.rs` + `system_hooks/src/call_hooks/mod.rs`)

---

### Summary

A system hook module (`mint_base_token`) is explicitly annotated with `// TODO: temporary solution, should be removed before the release` in two production files, yet it is unconditionally registered as a live system call hook during bootloader initialization. Any EVM transaction that calls address `0x7100` (`MINT_HOOK_ADDRESS_LOW`) will invoke this hook. If the hook's internal caller-restriction logic is insufficient (a realistic risk for code explicitly flagged as temporary/incomplete), an unprivileged sender can trigger unauthorized base-token minting, directly draining or inflating the L2 base-token supply.

---

### Finding Description

In `system_hooks/src/call_hooks/mod.rs`, the `mint_base_token` module is declared with an explicit pre-release removal notice:

```rust
// TODO: temporary solution, should be removed before the release
pub mod mint_base_token;
```

In `basic_bootloader/src/bootloader/block_flow/zk/post_init_op.rs`, the corresponding hook is unconditionally registered during every block's system initialization:

```rust
// TODO(EVM-1191): temporary solution, should be removed before the release
system_hooks::add_base_token_mint(system_functions)?;
```

`add_base_token_mint` registers `mint_base_token_hook` at `MINT_HOOK_ADDRESS_LOW = 0x7100`:

```rust
pub fn add_base_token_mint<S: EthereumLikeTypes, A: Allocator + Clone>(
    hooks: &mut HooksStorage<S, A>,
) -> Result<(), InternalError>
where
    S::IO: IOSubsystemExt,
{
    hooks.add_call_hook(
        MINT_HOOK_ADDRESS_LOW,
        SystemCallHook::new(mint_base_token_hook),
    )
}
```

The address constant comment states the hook is "Only callable by L2_BASE_TOKEN_ADDRESS (0x800a)", but this restriction lives inside the temporary hook implementation itself — code that is explicitly flagged as not production-ready. The hook is reachable by any EVM `CALL` to `0x7100` from any sender.

The analog to the reported issue is exact: just as `freezerAddress` and `rescuerAddress` were silently assigned to `_projectAddress` (a placeholder) with TODO comments, here a temporary minting hook is silently wired into the live system hook dispatch table with TODO comments marking it for removal.

---

### Impact Explanation

- **Vulnerability class:** Test/placeholder code left in production — temporary system hook with unverified access controls registered in the live bootloader.
- **Direct impact:** If `mint_base_token_hook` does not correctly enforce that `msg.sender == L2_BASE_TOKEN_ADDRESS (0x800a)` (a realistic risk for explicitly temporary code), any unprivileged transaction sender can call address `0x7100` to mint arbitrary amounts of the L2 base token. This constitutes unauthorized inflation of the token supply and direct loss of funds for all token holders.
- **Secondary impact:** Even if the access control is currently correct, the presence of explicitly-flagged temporary code in the production hook dispatch path means the hook may be modified, weakened, or left in place past its intended removal window, expanding the attack surface over time.

---

### Likelihood Explanation

- The hook is registered unconditionally on every block (no feature flag, no compile-time guard beyond `#[cfg(not(feature = "disable_system_contracts"))]`).
- Address `0x7100` is in the system hook address space and is reachable by any EVM `CALL` opcode from any EOA or contract.
- The TODO comments explicitly acknowledge the code is not production-ready, making it likely that the access-control logic inside the hook has not been hardened to production standards.
- An attacker needs only to send a transaction calling `0x7100` — no privileged access, no leaked keys, no governance attack required.

---

### Recommendation

1. Remove the `mint_base_token` hook and its registration from all production code paths before release, as the TODO comments themselves demand.
2. If the hook must remain temporarily, gate it behind a compile-time feature flag (e.g., `#[cfg(feature = "temp_mint_hook")]`) so it cannot be included in production builds.
3. Audit `mint_base_token_hook`'s caller-restriction logic independently to confirm it correctly rejects any caller other than `L2_BASE_TOKEN_ADDRESS (0x800a)`.
4. Add a CI check that fails if any `TODO.*temporary.*release` or `TODO.*remove.*release` annotation exists in non-test production source files.

---

### Proof of Concept

1. Deploy ZKsync OS with the current codebase (no special flags needed).
2. Submit an EVM transaction with `to = 0x0000000000000000000000000000000000007100` and calldata encoding a large mint amount.
3. The bootloader's hook dispatch in `runner.rs` routes the call to `mint_base_token_hook` because `0x7100 < SPECIAL_ADDRESS_BOUND` and the hook is registered.
4. If `mint_base_token_hook` does not enforce `caller == 0x800a`, the call succeeds and mints tokens to an attacker-controlled address.
