Audit Report

## Title
Unbounded `limit` in `POST /search/transactions` bypasses `MAX_SEARCH_LIMIT`, enabling OOM crash of the Rosetta process â€” (`rs/rosetta-api/icp/src/request_handler.rs`)

## Summary
The `search_transactions` handler uses `MAX_SEARCH_LIMIT` only as the default value when `request.limit` is absent. When a client supplies an explicit value, the sole validation is a `try_into::<u64>()` conversion that rejects negatives but accepts `i64::MAX` (9,223,372,036,854,775,807). This value is passed verbatim to a SQLite `LIMIT` clause, causing the handler to attempt loading the entire `blocks` table into a `Vec<HashedBlock>` in process memory, exhausting RAM and crashing the Rosetta process.

## Finding Description
`MAX_SEARCH_LIMIT` is defined as `const MAX_SEARCH_LIMIT: usize = 10_000` at line 57. At lines 762â€“768, the limit is parsed as:

```rust
let limit: u64 = request
    .limit
    .unwrap_or(MAX_SEARCH_LIMIT as i64)  // cap only applied when field is absent
    .try_into()
    .map_err(|err| {
        ApiError::invalid_request(format!("Limit has to be a valid u64: {err}"))
    })?;
```

No `.min(MAX_SEARCH_LIMIT as u64)` is applied after the conversion. The resulting `limit` is bound directly to the query at lines 837â€“838:

```rust
command.push_str("LIMIT :limit ");
parameters.push((":limit", Box::new(limit)));
```

The HTTP handler at lines 196â€“206 of `rosetta_server.rs` delegates directly to `req_handler.search_transactions` with no pre-validation. The only body-size guard is a 4 MB JSON payload cap at line 298, which does not constrain the numeric value of `limit`. With `i64::MAX` supplied, SQLite returns all rows in the `blocks` table, which are then deserialized into a `Vec<HashedBlock>` at lines 854â€“863, exhausting process memory.

## Impact Explanation
On a production Rosetta node synced to the ICP ledger (tens of millions of blocks), loading the entire `blocks` table into a `Vec<HashedBlock>` exhausts available RAM and triggers an OOM kill. This crashes the Rosetta process, making the ICP Rosetta API unavailable until manually restarted. This matches the allowed impact: **High ($2,000â€“$10,000) â€” Significant Rosetta/ledger/infrastructure security impact with concrete user or protocol harm**, as exchanges and integrators relying on the Rosetta API lose access to ICP transaction data and submission capabilities.

## Likelihood Explanation
The `/search/transactions` endpoint is unauthenticated and publicly reachable. The exploit payload is a single JSON object with one oversized integer field. No rate limiting or authentication is visible in the server setup. Any network-reachable attacker can trigger this repeatedly to keep the process down indefinitely.

## Recommendation
After the `try_into()` conversion, enforce the cap unconditionally:

```rust
let limit: u64 = request
    .limit
    .unwrap_or(MAX_SEARCH_LIMIT as i64)
    .try_into()
    .map_err(|err| ApiError::invalid_request(format!("Limit has to be a valid u64: {err}")))?;

let limit = limit.min(MAX_SEARCH_LIMIT as u64);
```

Alternatively, reject requests where the provided `limit` exceeds `MAX_SEARCH_LIMIT` with an explicit `ApiError::invalid_request`.
