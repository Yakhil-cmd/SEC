Audit Report

## Title
Byzantine Dealer Out-of-Range Chunk DoS via `CheatingDealerDlogSolver` in `dec_chunks` — (`rs/crypto/internal/crypto_lib/threshold_sig/bls12_381/src/ni_dkg/fs_ni_dkg/forward_secure.rs`)

## Summary

The NiDKG chunking NIZK (`verify_chunking`) bounds only the weighted-sum response scalars `z_s[k] < zz ≈ 4.8×10¹¹`, not individual plaintext chunk values. A single Byzantine dealer can therefore produce a dealing whose ciphertext chunks lie above `CHUNK_MAX = 65535` while still passing `verify_dealing`. When any receiver calls `load_transcript`, `dec_chunks` falls through to `CheatingDealerDlogSolver`, which the code itself annotates as potentially taking **hours** per chunk. When the DKG deadline arrives, `enforce_transcript_loading` calls `handle.recv()` on the blocked background thread, stalling the consensus thread for the same duration and preventing the subnet from advancing.

## Finding Description

**Root cause — `verify_chunking` does not bound individual chunks.**

`verify_chunking` in `nizk_chunking.rs` computes:

```
ss  = n * m * (CHUNK_SIZE - 1) * CHALLENGE_MASK  ≈ 7.5 × 10⁹
zz  = 2 * NUM_ZK_REPETITIONS * ss                ≈ 4.8 × 10¹¹
```

and rejects only if any `z_s[k] >= zz`. [1](#0-0) 

Because `zz ≫ CHUNK_MAX`, the prover's slack is large enough to keep `z_s[k]` within `zz` even when individual chunks are set to multiples of `CHUNK_MAX`. No check on individual chunk values exists anywhere in `verify_zk_proofs` or `verify_ciphertext_integrity`. [2](#0-1) 

**`verify_dealing` passes for out-of-range chunks.**

The existing test `should_decrypt_correctly_for_cheating_dealer` constructs chunks with `PlaintextChunks::new_unchecked` where values exceed `CHUNK_MAX`, calls `enc_chunks`, and asserts `verify_ciphertext_integrity` returns `Ok`. [3](#0-2) 

**`dec_chunks` unconditionally invokes `CheatingDealerDlogSolver` for out-of-range chunks.**

`HonestDealerDlogLookupTable::solve_several` scans only `[0, CHUNK_MAX]`; any chunk above that returns `None`. The fallback constructs `CheatingDealerDlogSolver::new(n, m)` (allocating up to 2 GiB BSGS table) and calls `solve()` for every failing chunk. The code comment reads: *"It may take hours to brute force a cheater's discrete log."* [4](#0-3) 

`CheatingDealerDlogSolver::solve` iterates up to `scale_range = 256` BSGS lookups per chunk, over a search range of `bsgs_range ≈ 9.6×10¹¹`. [5](#0-4) 

**`enforce_transcript_loading` blocks the consensus thread.**

`DkgKeyManager` spawns a background thread per transcript that loops on `NiDkgAlgorithm::load_transcript` with no timeout. [6](#0-5) 

When the DKG deadline height is reached, `enforce_transcript_loading` calls `handle.recv()` — a blocking call — on the still-running background thread. [7](#0-6) 

This blocks the consensus `on_state_change` loop for the entire duration of the BSGS computation (hours), preventing the node from producing or validating blocks.

**The `#[ignore]`-tagged benchmark confirms the runtime is prohibitive.**

`print_time_for_cheating_dlog_solver_to_run` is marked `#[ignore]` precisely because the runtime exceeds test infrastructure timeouts. [8](#0-7) 

## Impact Explanation

Every receiver in the subnet calls `load_transcript` for the new epoch's transcript. A single Byzantine dealer's dealing — which passes `verify_dealing` — is included in the transcript. Each receiver's DKG key-manager background thread is blocked for hours in `CheatingDealerDlogSolver::solve`. When the DKG deadline arrives, `enforce_transcript_loading` blocks the consensus thread itself via `handle.recv()`, halting block production and validation on every affected node. With all or a threshold-breaking number of nodes stalled, the subnet loses liveness, stalling chain-key operations (tECDSA, tSchnorr, vetKD) and subnet key refresh. This matches the allowed impact: **High — Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS.**

## Likelihood Explanation

A single Byzantine dealer — one node below the consensus fault threshold — is sufficient. The dealer must be a registered subnet node (requires governance onboarding), but no majority corruption is needed. The attack is deterministic: any chunk value above `CHUNK_MAX` reliably triggers the slow path. The dealing passes all existing verification checks. The attack is repeatable every DKG interval.

## Recommendation

1. **Add a per-chunk range check in `verify_chunking` or `verify_zk_proofs`**: After the NIZK equations pass, verify that each ciphertext chunk decodes to a value in `[0, CHUNK_MAX]` using the public ciphertext commitments. The ciphertext chunks are public, so this check is feasible without secret material.

2. **Add a timeout / cancellation token to `CheatingDealerDlogSolver::solve`**: Even if the NIZK check is strengthened, a defense-in-depth timeout prevents indefinite blocking of the consensus thread.

3. **Return `DecErr::InvalidChunk` immediately on out-of-range detection**: If `dec_chunks` detects that `HonestDealerDlogLookupTable` returns `None`, it should immediately return an error and issue a complaint rather than invoking `CheatingDealerDlogSolver`.

## Proof of Concept

The existing test `should_decrypt_correctly_for_cheating_dealer` already exercises the full path with small out-of-range values. To reproduce the DoS:

```rust
// Byzantine dealer side:
let mut chunks = [0isize; NUM_CHUNKS];
for i in 0..NUM_CHUNKS {
    chunks[i] = CHUNK_MAX as isize * 200; // above CHUNK_MAX, within zz
}
let cheating_plaintext = PlaintextChunks::new_unchecked(chunks);
let (crsz, _witness) = enc_chunks(&[(receiver_pk, cheating_plaintext)], epoch, ad, sys, rng);
assert!(verify_ciphertext_integrity(&crsz, epoch, ad, sys).is_ok()); // passes
// verify_chunking also passes because z_s[k] < zz

// Receiver side during load_transcript:
// dec_chunks → HonestDealerDlogLookupTable returns None for all 16 chunks
//            → CheatingDealerDlogSolver::new(28, 16) allocates ~2 GiB table
//            → solve() iterates 255 × 16 BSGS searches
//            → wall-clock time: hours
// enforce_transcript_loading → handle.recv() blocks consensus thread
```

Run `print_time_for_cheating_dlog_solver_to_run` (remove `#[ignore]`) to observe the prohibitive runtime directly. [8](#0-7)

        let ss = n * m * (CHUNK_SIZE - 1) * (scale_range - 1);
        let zz = 2 * NUM_ZK_REPETITIONS * ss;

        let bsgs_lo = 1 - zz as isize;
        let bsgs_range = 2 * zz - 1;

        let baby_giant = BabyStepGiantStep::new(
            Gt::generator(),
            bsgs_lo,
            bsgs_range,
            Self::MAX_TABLE_MBYTES,
            Self::LARGEST_TABLE_MUL,
        );
        Self {
            baby_giant,
            scale_range,
        }
    }

    /// Searches for discrete log for a malicious DKG participant whose NIZK
    /// chunking proof checks out, which implies certain bounds on the search
    ///
    /// This function is not constant time, but it only leaks information in the
    /// event that a dealer is already dishonest, and the only information it
    /// leaks is the value that the dishonest dealer sent.
    pub fn solve(&self, target: &Gt) -> Option<Scalar> {
        /*
        For some Delta in [1..E - 1] the answer s satisfies (Delta * s) in
        [1 - Z..Z - 1].

        For each delta in [1..E - 1] we compute target*delta and use
        baby-step-giant-step to find `scaled_answer` such that:
           base*scaled_answer = target*delta

         Then `base * (scaled_answer / delta) = target`
          (here division is modulo the group order
         That is, the discrete log of target is `scaled_answer / delta`.
        */
        let mut target_power = Gt::identity();
        for delta in 1..self.scale_range {
            target_power += target;

            if let Some(scaled_answer) = self.baby_giant.solve(&target_power) {
                let inv_delta = Scalar::from_usize(delta)
                    .inverse()
                    .expect("Delta is always invertible");
                let result = scaled_answer * inv_delta;
                return Some(result);
            }
        }
