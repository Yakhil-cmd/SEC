Audit Report

## Title
Unbounded Neuron Ballot Loop in SNS Governance `distribute_rewards` Has No Instruction-Limit Guard, Risking Permanent Reward Distribution Halt - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS `distribute_rewards` function iterates over all ballots of all settled proposals and then over all rewarded neurons in a single synchronous message with no instruction-limit check or batching mechanism. If the combined neuron × proposal count grows large enough to exhaust the per-message instruction limit, the message traps and is rolled back, and every subsequent reward round will fail identically, permanently halting SNS voting rewards. NNS Governance encountered this exact failure in production and added a `RewardsDistributionStateMachine` with an explicit `DISTRIBUTION_MESSAGE_LIMIT` guard; SNS Governance has no equivalent protection.

## Finding Description
`distribute_rewards` is called synchronously from `run_periodic_tasks` at [1](#0-0)  with no instruction budget check before or after the call.

Inside `distribute_rewards` (function signature at [2](#0-1) ), two unbounded loops execute:

1. An outer loop over `considered_proposals` with an inner loop over every `ballot` entry (one per voting neuron) at [3](#0-2) .
2. A second loop over every entry in `neuron_id_to_reward_shares`, calling `get_neuron_result_mut` per neuron to credit maturity at [4](#0-3) .

A grep for `is_message_over_threshold`, `instruction_limit`, and `DISTRIBUTION_MESSAGE_LIMIT` in `rs/sns/governance/src/governance.rs` returns zero matches, confirming no guard exists anywhere in this code path.

By contrast, NNS Governance uses `DISTRIBUTION_MESSAGE_LIMIT = 1_000_000_000` instructions and calls `is_message_over_threshold` after each neuron credit, breaking out and persisting remaining work to stable storage for the next timer invocation at [5](#0-4) . The NNS CHANGELOG explicitly documents that this batching was added because the unbounded loop caused a production failure requiring a hotfix at [6](#0-5) .

## Impact Explanation
If the instruction limit is exceeded, the entire `distribute_rewards` message traps and is rolled back. Because `run_periodic_tasks` is driven by a heartbeat/timer, the next invocation re-attempts the identical full computation and fails again. The result is that `latest_reward_event.end_timestamp_seconds` stops advancing and `maturity_e8s_equivalent` stops increasing for all SNS neurons — voting rewards are permanently frozen for the affected SNS. This matches the allowed ICP bounty impact: **significant SNS security impact with concrete user and protocol harm** (High, $2,000–$10,000), specifically a governance liveness failure that is not based on raw volumetric DDoS.

## Likelihood Explanation
The IC per-message instruction limit is 40 billion instructions on application subnets. An SNS with 10,000 neurons and 10 proposals settling in one round produces 100,000 ballot iterations plus 10,000 `get_neuron_result_mut` calls. Neuron counts of this scale are realistic for active SNS deployments (e.g., OpenChat, Kinic). An attacker can accelerate the condition by acquiring many neurons via swap participation or `ManageNeuron::ClaimOrRefresh` — both are permissionless operations available to any principal. The NNS Governance team documented this exact failure mode occurring in production, establishing that the threshold is reachable in practice, not merely in theory.

## Recommendation
1. Adopt the NNS pattern: separate reward calculation (synchronous, produces a `neuron_id → reward_e8s` map) from reward disbursement (batched across multiple timer-driven messages with an instruction-limit guard), as implemented in `rs/nns/governance/src/reward/distribution.rs`.
2. Add an `is_message_over_threshold` check inside the neuron maturity-credit loop, persisting remaining work to stable storage for the next timer invocation.
3. Add canbench benchmarks measuring `distribute_rewards` instruction consumption at 10K, 50K, and 100K neuron counts to establish a regression baseline.
4. As a short-term mitigation, cap the number of proposals settled per reward round to bound worst-case instruction consumption.

## Proof of Concept
1. Deploy an SNS with `voting_rewards_parameters` enabled and a short `reward_round_duration_seconds`.
2. Create or acquire ~10,000–50,000 neurons via swap participation or `ManageNeuron::ClaimOrRefresh` (permissionless).
3. Let several proposals reach `ReadyToSettle` state simultaneously.
4. Observe that `run_periodic_tasks` → `distribute_rewards` traps on every invocation.
5. Verify: `latest_reward_event.end_timestamp_seconds` stops advancing; `maturity_e8s_equivalent` stops increasing for all neurons.

A deterministic integration test can reproduce this by constructing a `Governance` instance with N neurons, creating M settled proposals each with N ballots, calling `distribute_rewards` directly, and asserting that the function completes without trapping — then measuring instruction consumption via `canbench` at realistic scale to confirm the limit is approached.

```markdown
        * Distribute rewards is moved to timer, and has a mechanism to distribute in batches in
          multiple messages.
        * Unstaking maturity task has a limit of 100 neurons per message, which prevents it from

        * The execution of `ApproveGenesisKyc` proposals have a limit of 1000 neurons, above which
          the proposal will fail.
        * More benchmarks were added.
* Enable timer task metrics for better observability.

## Changed

* Voting Rewards will be scheduled by a timer instead of by heartbeats.
* Unstaking maturity task will be processing up to 100 neurons in a single message, to avoid
  exceeding the instruction limit in a single execution.
* Voting Rewards will be distributed asynchronously in the background after being calculated.
    * This will allow rewards to be compatible with neurons being stored in Stable Memory.
```
