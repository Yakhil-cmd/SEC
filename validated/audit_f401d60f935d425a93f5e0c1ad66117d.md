### Title
Unchecked Decreasing Cumulative Block Counters Cause u64 Underflow in Node Provider Reward Accounting - (File: rs/node_rewards/canister/src/metrics.rs)

---

### Summary

The `metrics_by_subnet()` function in the Node Rewards canister computes daily block metrics by subtracting the previous day's cumulative totals from the current day's cumulative totals using plain `u64` arithmetic. It does not guard against the case where the current cumulative total is **less than** the previous day's stored total. This can occur when a node is pruned from `BlockmakerMetricsTimeSeries` due to inactivity and then reappears with counters restarted from zero. The resulting integer underflow silently produces astronomically large values in Wasm (wrapping arithmetic), permanently corrupting the failure-rate and reward calculations for that node provider.

---

### Finding Description

The `MetricsManager::metrics_by_subnet()` function fetches stored cumulative block totals (`num_blocks_proposed_total`, `num_blocks_failed_total`) from the management canister and derives daily activity by subtraction:

```rust
// rs/node_rewards/canister/src/metrics.rs L236-L241
NodeMetricsDailyRaw {
    node_id: NodeId::from(node.node_id.unwrap()),
    num_blocks_proposed: node.num_blocks_proposed_total
        - initial_proposed_total,          // plain u64 subtraction
    num_blocks_failed: node.num_blocks_failed_total - initial_failed_total,
}
``` [1](#0-0) 

The code implicitly assumes `num_blocks_proposed_total` is monotonically non-decreasing. This assumption is violated by the pruning logic in `BlockmakerMetricsTimeSeries::observe()`:

```rust
// rs/replicated_state/src/metadata_state.rs L1887-L1894
running_stats.node_stats.retain(|node_id, stats| {
    match last_snapshot.get(node_id) {
        None => true,
        Some(last_stats) => last_stats != stats,
    }
});
``` [2](#0-1) 

Per the documented invariant, a node whose stats are unchanged between two consecutive daily snapshots is **pruned** from the running stats map. When that node reappears, `or_default()` restarts its counters from zero:

```rust
// rs/replicated_state/src/metadata_state.rs L1854-L1857
self.node_stats
    .entry(metrics.blockmaker)
    .or_default()
    .blocks_proposed_total += 1;
``` [3](#0-2) 

The management canister's `node_metrics_history` endpoint then reports the restarted (lower) totals. The `MetricsManager` has already stored the pre-pruning totals as the baseline. The subtraction underflows.

In Wasm (IC production execution), integer subtraction wraps silently — there is no trap. The result is a value near `u64::MAX`, which is then fed into the failure-rate and reward calculations.

---

### Impact Explanation

The corrupted `num_blocks_proposed` value (≈ `u64::MAX`) is passed to `calculate_daily_node_failure_rate`:

```rust
// rs/node_rewards/rewards_calculation/src/performance_based_algorithm/mod.rs L248-L254
fn calculate_daily_node_failure_rate(proposed: u64, failed: u64) -> Decimal {
    let total = Decimal::from(proposed + failed);
    if total.is_zero() {
        Decimal::ZERO
    } else {
        Decimal::from(failed) / total
    }
}
``` [4](#0-3) 

`proposed + failed` itself overflows `u64` (wrapping to a small number), producing a nonsensical failure rate (potentially > 1.0). This distorts the subnet-level 75th-percentile failure rate used to penalize or reward all nodes in the subnet, and corrupts the `adjusted_rewards_xdr_permyriad` for the affected node provider. The accounting divergence is **permanent** — the stored baseline in `last_timestamp_per_subnet` advances past the pruning event, so subsequent days continue computing deltas from the wrong baseline. [5](#0-4) 

---

### Likelihood Explanation

A node operator can deliberately trigger this by keeping their node offline for exactly one full day (causing its stats to be unchanged between two consecutive snapshots, satisfying the pruning condition), then bringing it back online. This is a realistic operational scenario — nodes go offline for maintenance, upgrades, or network issues. No privileged access beyond operating a node is required. The pruning behavior is documented and deterministic. [6](#0-5) 

---

### Recommendation

Replace the plain `u64` subtraction with a saturating or checked subtraction, and treat a decrease in cumulative totals as a counter reset (baseline = 0 for that day):

```rust
num_blocks_proposed: node.num_blocks_proposed_total
    .saturating_sub(initial_proposed_total),
num_blocks_failed: node.num_blocks_failed_total
    .saturating_sub(initial_failed_total),
```

Alternatively, detect the decrease explicitly and treat the full current-day total as the daily delta (analogous to how the oracle bug fix recommended directly controlling for decreases).

---

### Proof of Concept

**Day 1:** Node A proposes 100 blocks. `MetricsManager` stores `num_blocks_proposed_total = 100` for Node A at timestamp T1.

**Day 2:** Node A is offline. Its stats remain at (100, 0). At the start of Day 3, `BlockmakerMetricsTimeSeries` prunes Node A because its stats equal the Day 1 snapshot.

**Day 3:** Node A comes back online and proposes 1 block. `node_metrics_history` now reports `num_blocks_proposed_total = 1` for Node A (restarted from 0).

**`metrics_by_subnet()` for Day 3:**
```
initial_proposed_total = 100   (stored from Day 1)
node.num_blocks_proposed_total = 1
num_blocks_proposed = 1u64 - 100u64
                    = 18446744073709551517  (wrapping underflow in Wasm)
```

This value is passed to `calculate_daily_node_failure_rate(18446744073709551517, 0)`, producing `total = 18446744073709551517`, `failure_rate = 0`. The subnet's 75th-percentile failure rate is distorted, and all node providers in that subnet receive incorrect rewards for Day 3 and all subsequent days until the stored baseline is corrected. /// Metrics for a time series aggregated from the `BlockmakerMetrics` present in each batch.
///
/// Blockmaker stats are continuously accumulated for each node ID. On the first
/// observation of each day (as determined by `days_since_unix_epoch()`) a snapshot along
/// with a timestamp of the last observation on the previous day is stored in the metrics.
/// For each day metrics were aggregated since the first observation, there is exactly one
/// snapshot in the series (including 'today').
///
/// The number of snapshots is capped at `BLOCKMAKER_METRICS_TIME_SERIES_NUM_SNAPSHOTS` by
/// discarding the oldest snapshot(s) once this limit is exceeded.
///
/// To ensure the roster of node IDs does not grow indefinitely, Node IDs whose stats are
/// equal in two consecutive snapshots, are pruned from the metrics such that they are missing
/// from that point on. If such a node reappears later on, it will be added as a new node with
/// restarted stats.
///
/// There is a runtime invariant (excluding checks at deserialization):
/// - There are at most `BLOCKMAKER_METRICS_TIME_SERIES_NUM_SNAPSHOTS` snapshots.
///
/// There is an invariant (including checks at deserialization):
/// - Each timestamp corresponding to a snapshot maps onto a unique day (as determined
///   by `days_since_unix_epoch()`).
#[derive(Clone, Eq, PartialEq, Debug, Default)]
pub struct BlockmakerMetricsTimeSeries(BTreeMap<Time, BlockmakerStatsMap>);
