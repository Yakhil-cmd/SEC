### Title
`floor_gas` Uses Wrong Base for Contract-Creation Transactions, Causing Spurious `IntrinsicGasNotMet` Rejections — (`engine-transactions/src/lib.rs`)

### Summary
`NormalizedEthTransaction::floor_gas` always uses `gas_transaction_call` (21000) as its base cost, even when `self.to.is_none()` (contract creation). For contract-creation transactions with sufficient non-zero calldata under Prague config, `floor_gas` exceeds `intrinsic_gas`, causing `submit_with_alt_modexp` to reject the transaction with `IntrinsicGasNotMet` even when the user supplies `gas_limit == intrinsic_gas`.

### Finding Description
`floor_gas` is defined in `engine-transactions/src/lib.rs`: [1](#0-0) 

It unconditionally sets `base_gas = config.gas_transaction_call` (21000). It never checks `self.to.is_none()` to use `config.gas_transaction_create` (53000) for contract creation, unlike `intrinsic_gas`: [2](#0-1) 

In `submit_with_alt_modexp`, the engine enforces `max(intrinsic_gas, floor_gas)` as the minimum gas: [3](#0-2) 

When `floor_gas > intrinsic_gas`, the effective minimum required gas is `floor_gas`, which is higher than what the user computed as the intrinsic gas. The user's transaction is rejected with `IntrinsicGasNotMet` even though `gas_limit >= intrinsic_gas`.

### Impact Explanation
A user deploying a contract with heavy non-zero calldata (≥ ~1334 bytes) under Prague config will have their deployment transaction rejected if they set `gas_limit` to the standard intrinsic gas. The error `IntrinsicGasNotMet` is misleading — the user did meet the intrinsic gas. Their ETH remains locked in their account, unable to deploy the contract, until they discover the undocumented floor gas threshold and manually raise `gas_limit` above it. This constitutes **temporary freezing of funds** (High impact).

### Likelihood Explanation
Prague config is the active production config. Any contract deployment with ≥ ~1334 non-zero calldata bytes triggers the condition. Large contract deployments (e.g., ERC-20, proxy factories, complex DeFi contracts) routinely exceed this threshold. The threshold is not documented anywhere in the public interface, making it a silent footgun for any user or tooling that computes gas limits from the standard intrinsic gas formula.

### Recommendation
In `floor_gas`, apply the same `is_contract_creation` branch as `intrinsic_gas`:

```rust
let base_gas = if self.to.is_none() {
    config.gas_transaction_create
} else {
    config.gas_transaction_call
};
``` [4](#0-3) 

Note: `init_code_cost` should **not** be added to `floor_gas` (EIP-7623 floor does not include initcode cost), but the base must be `gas_transaction_create` for contract creation.

    #[allow(clippy::naive_bytecount)]
    pub fn floor_gas(&self, config: &aurora_evm::Config) -> Result<u64, Error> {
        if config.has_floor_gas {
            let num_zero_bytes = u64::try_from(self.data.iter().filter(|b| **b == 0).count())
                .map_err(|_e| Error::IntegerConversion)?;
            let data_len = u64::try_from(self.data.len()).map_err(|_e| Error::IntegerConversion)?;
            let num_non_zero_bytes = data_len
                .checked_sub(num_zero_bytes)
                .ok_or(Error::GasOverflow)?;

            let base_gas = config.gas_transaction_call;
            let tokens_in_calldata = num_non_zero_bytes
                .checked_mul(4)
                .and_then(|gas| gas.checked_add(num_zero_bytes))
                .ok_or(Error::GasOverflow)?;

            tokens_in_calldata
                .checked_mul(config.total_cost_floor_per_token)
                .and_then(|gas| gas.checked_add(base_gas))
                .ok_or(Error::GasOverflow)
        } else {
            Ok(0)
        }
    }
```
