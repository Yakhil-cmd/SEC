Audit Report

## Title
VIP Caller Bypass Broken in `try_borrow_slot`: Slot Exhaustion Blocks NNS Governance Management Operations — (`rs/nervous_system/clients/src/management_canister_client.rs`)

## Summary
In `LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot`, the `available_slot_count == 0` guard fires unconditionally for all callers before the VIP-specific `used_slot_count = 0` path has any effect. An unprivileged user can exhaust all 167 slots via the public `canister_status` endpoint on the NNS root canister, causing subsequent calls from VIP callers (NNS governance and other NNS canisters) to receive `SysTransient` errors, blocking critical NNS canister management operations.

## Finding Description
In `try_borrow_slot` at `rs/nervous_system/clients/src/management_canister_client.rs` lines 264–287:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if *available_slot_count == 0 {   // ← fires for ALL callers
                let code = RejectCode::SysTransient as i32;
                let message = "Unavailable. Maybe, try again later?".to_string();
                return Err((code, message));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

The VIP path sets `used_slot_count = 0` so that `saturating_sub(0)` leaves the counter unchanged — the intent being that VIP callers consume no slots and are never rate-limited. However, the `== 0` guard is evaluated **before** the subtraction and does not inspect `is_caller_vip`. When `available_slot_count` reaches zero, the guard returns `Err` for every caller, VIP or not.

The slot pool is initialized to 167 in `rs/nns/handlers/root/impl/canister/canister.rs` (line 50). The `canister_status` endpoint (lines 88–98) is explicitly documented as public with no access control. `new_management_canister_client()` (lines 53–67) derives `is_caller_vip` from the caller per-request, correctly marking NNS canisters as VIP — but the guard in `try_borrow_slot` ignores this flag entirely when the counter is zero.

The `SlotLoan::drop` implementation (lines 365–372) adds back `used_slot_count` on drop. For VIP callers `used_slot_count = 0`, so their loans restore nothing — correct in isolation, but irrelevant once the guard blocks them before a loan is ever issued.

Exploit flow:
1. Attacker submits 167 concurrent update calls to the public `canister_status` endpoint from an unprivileged principal.
2. Each call borrows one slot and suspends at `await management_canister.canister_status()` (IC async model: canister processes subsequent messages while awaiting inter-canister responses; responses arrive in a later round).
3. `available_slot_count` reaches 0.
4. Any subsequent call — including from NNS governance (`is_caller_vip = true`) — hits the `== 0` guard and receives `Err((SysTransient, "Unavailable..."))`.
5. Operations routed through root that depend on `try_borrow_slot` — `stop_canister`, `update_settings`, `delete_canister`, `take_canister_snapshot`, `load_canister_snapshot` — all fail for VIP callers.

## Impact Explanation
This is an application/platform-level DoS against NNS governance operations. The NNS root canister is the controller of all NNS canisters; blocking its management canister client prevents governance-initiated canister upgrades, stops, settings changes, and snapshot operations. This matches the allowed High impact: **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"** and **"Significant NNS security impact with concrete user or protocol harm."** Severity: High ($2,000–$10,000).

## Likelihood Explanation
- The `canister_status` endpoint has no access control and is intentionally public (documented in source).
- The IC message queue limit is 500; holding 167 concurrent in-flight calls is well within reach.
- The attack requires no privileged access, key material, or consensus manipulation.
- The bug is deterministic: once `available_slot_count == 0`, every subsequent call fails regardless of VIP status.
- The attacker must continuously replenish calls (management canister responds in subsequent rounds), but this is feasible via repeated ingress submissions and is a sustained, low-cost DoS.

## Recommendation
Add a VIP bypass to the `== 0` guard in `try_borrow_slot`:

```rust
if !self.is_caller_vip && *available_slot_count == 0 {
    let code = RejectCode::SysTransient as i32;
    let message = "Unavailable. Maybe, try again later?".to_string();
    return Err((code, message));
}
```

This ensures VIP callers always proceed regardless of slot exhaustion, matching the documented intent ("VIP = is an NNS canister") in `new_management_canister_client`.

## Proof of Concept
Using the existing test infrastructure in `rs/nns/handlers/root/impl/tests/test.rs` with `state_machine_test_on_nns_subnet` and `update_with_sender`:

1. Install the NNS root canister with `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT = 167`.
2. Submit 167 concurrent `canister_status` update calls from an anonymous principal targeting a canister whose management canister response is delayed (or use a mock that holds responses).
3. Assert `available_slot_count == 0`.
4. Submit a `canister_status` call from a known NNS canister principal (e.g., governance canister ID), which sets `is_caller_vip = true`.
5. Assert the response is `Err((SysTransient, "Unavailable. Maybe, try again later?"))` — demonstrating VIP callers are blocked despite `used_slot_count = 0`.
