### Title
Unchecked Integer Multiplication Overflow in XCC Precompile Gas Calculation Causes Router Execution Failure and Fund Freeze - (File: engine-precompiles/src/xcc.rs)

### Summary
In the `CrossContractCall` precompile's `run_with_handle` method, the expression `callback_count * costs::ROUTER_EXEC_PER_CALLBACK.as_u64()` is an unchecked `u64` multiplication. A user can craft a `CrossContractCallArgs::Eager` payload with a large `promise_count()` (achievable with exponentially nested `NearPromise::And` structures) to silently wrap this multiplication in Rust release mode. The wrapped result produces a drastically underestimated `router_exec_cost`, causing the NEAR-level router `execute` call to receive far less gas than required. The router fails, while the user's wNEAR — already burned in the EVM and converted to NEAR sent to the router — becomes frozen in the router account with no automatic refund path.

### Finding Description

In `engine-precompiles/src/xcc.rs`, the `CrossContractCall::run_with_handle` method computes the gas to attach to the router's `execute` call:

```rust
let callback_count = call
    .promise_count()
    .checked_sub(1)
    .ok_or_else(|| ExitError::Other(Cow::from(consts::ERR_INVALID_INPUT)))?;
let router_exec_cost = costs::ROUTER_EXEC_BASE
    + NearGas::new(callback_count * costs::ROUTER_EXEC_PER_CALLBACK.as_u64());
```

`ROUTER_EXEC_PER_CALLBACK` is `12_000_000_000_000` (12 TGas). The multiplication `callback_count * 12_000_000_000_000` is a plain `u64 * u64` with no overflow guard. Rust release builds use wrapping semantics for integer overflow, so when `callback_count ≥ ⌈2^64 / 12_000_000_000_000⌉ ≈ 1_537_229`, the product silently wraps to a small value.

`NearGas::add` is also unchecked:

```rust
impl Add for NearGas {
    type Output = Self;
    fn add(self, rhs: Self) -> Self::Output {
        Self(self.0 + rhs.0)
    }
}
```

The resulting `router_exec_cost` is far below what the router needs. The EVM-level gas cost check also passes because it is derived from the (now tiny) `promise.attached_gas`:

```rust
cost += EthGas::new(promise.attached_gas.as_u64() / costs::CROSS_CONTRACT_CALL_NEAR_GAS);
```

The XCC precompile call succeeds in the EVM. The engine then processes the promise log via `handle_precompile_promise`, which calls `withdraw_wnear_to_router` to unwrap the user's wNEAR and send real NEAR to the router, then dispatches the router's `execute` with the underestimated gas. The router runs out of gas and fails. The NEAR already deposited into the router account is not automatically refunded (the `send_refund` callback is only attached when `refund_needed` is true, i.e., only on first-time account creation).

### Impact Explanation

The user's wNEAR is irreversibly burned in the EVM (transferred to the engine's implicit address via `transferFrom`). The corresponding NEAR is sent to the router and becomes frozen there when `execute` fails. There is no automatic on-chain recovery path for the deposited NEAR in the non-creation case. This constitutes a **high-severity temporary (potentially permanent) freezing of user funds**.

### Likelihood Explanation

The attack is reachable by any unprivileged EVM user who can call the XCC precompile. Crafting a `NearPromise::And` tree with exponential promise count requires only polynomial input size: 21 levels of `And([prev, prev])` yields `2^21 = 2_097_152 > 1_537_229` promises with roughly 4–5 KB of serialized input. No governance or admin access is required. The attacker can target their own funds or any user who calls the XCC precompile with a crafted payload.

### Recommendation

Replace the unchecked multiplication with a checked or saturating variant and propagate the error:

```rust
let per_callback = costs::ROUTER_EXEC_PER_CALLBACK.as_u64();
let callbacks_gas = callback_count
    .checked_mul(per_callback)
    .ok_or_else(|| ExitError::Other(Cow::from("ERR_GAS_OVERFLOW")))?;
let router_exec_cost = costs::ROUTER_EXEC_BASE
    .saturating_add(NearGas::new(callbacks_gas));
```

Additionally, enforce an upper bound on `promise_count()` before performing any arithmetic, and make `NearGas::add` use `saturating_add` or return a `Result`.

### Proof of Concept

1. Construct a `NearPromise::And` tree with depth 21, where each node is `And([left, right])` and leaves are `Simple(PromiseCreateArgs { attached_gas: NearGas::new(0), ... })`. This yields `promise_count() = 2^21 = 2_097_152`.
2. Wrap it in `PromiseArgs::Recursive(tree)` and then `CrossContractCallArgs::Eager(args)`.
3. Serialize with Borsh and call the XCC precompile address (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`) from an EVM contract.
4. `callback_count = 2_097_151`. Multiplication: `2_097_151 * 12_000_000_000_000 = 25_165_812_000_000_000_000`, which wraps mod `2^64` to `6_707_938_926,290,448,384` — still large but the key point is that for `callback_count = 1_537_229`, the wrap produces `≈ 3.9 TGas`, far below the `≈ 18.4 PGas` needed.
5. The XCC precompile succeeds in the EVM. The engine dispatches the router `execute` with `≈ 10.9 TGas` instead of `≈ 18.4 PGas`.
