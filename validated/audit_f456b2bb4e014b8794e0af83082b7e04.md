Audit Report

## Title
Unconditional `neuron_ids.retain()` Before Result Check in `AccountState::transfer` Permanently Orphans Failed-Transfer Neurons — (File: rs/nns/gtc/src/lib.rs)

## Summary
In `AccountState::transfer`, `self.neuron_ids.retain(|id| id != &neuron_id)` at line 192 executes unconditionally after each inter-canister call to governance, regardless of whether that call succeeded or failed. A neuron whose transfer returns `Err` is evicted from `neuron_ids` and appended to `failed_transferred_neurons`, which has no associated retry or recovery path anywhere in the GTC canister. Those neurons remain under GTC controller in governance but become permanently unreachable through the GTC, constituting an irrecoverable loss of ICP stake for the affected account owners.

## Finding Description
`AccountState::transfer` (rs/nns/gtc/src/lib.rs, lines 188–210) clones `self.neuron_ids` into a local `neuron_ids` vector, then iterates over it. For each element it:

1. Awaits `GovernanceCanister::transfer_gtc_neuron(neuron_id, custodian_neuron_id)`.
2. **Unconditionally** calls `self.neuron_ids.retain(|id| id != &neuron_id)` (line 192), removing the ID from the live tracking list before inspecting the result.
3. Matches on `result`: on `Ok` pushes to `successfully_transferred_neurons`; on `Err` pushes to `failed_transferred_neurons`.

Because step 2 precedes step 3, a governance `Err` still removes the neuron ID from `neuron_ids`. After the loop, `neuron_ids` is empty regardless of how many transfers failed. A subsequent call to `transfer` iterates over an empty clone and is a no-op. The `failed_transferred_neurons` field is a write-only audit log; no GTC method reads it to re-attempt or recover the listed neurons. The governance canister still holds those neurons with the GTC as controller, but no GTC entry point can interact with them.

The two callers are:
- `donate_account` (lib.rs line 89): requires secp256k1 public-key authentication; only the legitimate account owner can reach this path.
- `forward_whitelisted_unclaimed_accounts` (lib.rs lines 102–131, canister.rs line 181): **no authentication**; callable by any principal after `SECONDS_UNTIL_FORWARD_WHITELISTED_UNCLAIMED_ACCOUNTS_CAN_BE_CALLED` (188 days post-genesis). The outer function swallows per-account errors (lines 120–125) and returns `Ok(())`, so a partial failure leaves no externally visible signal beyond the canister log.

## Impact Explanation
Neurons orphaned by this bug are permanently inaccessible: the GTC no longer tracks them in `neuron_ids`, `has_donated`/`has_forwarded` are never set (so the account appears re-enterable but iterates over nothing), and no GTC method can recover them. The ICP stake locked in those neurons is effectively frozen under a controller (the GTC) that has lost its own reference to them. This constitutes a permanent, irrecoverable loss of NNS neuron assets for the affected account owners — matching the High impact class: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints," and "Significant NNS/SNS or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
The unprivileged `forward_whitelisted_unclaimed_accounts` path is the primary risk surface. Any caller can invoke it after 188 days; no key, privilege, or social engineering is required. The attacker does not need to cause the governance failure — they only need to invoke the function during any transient governance degradation (canister upgrade window, message queue saturation, governance bug, etc.). Such conditions are realistic on a live NNS subnet. The `donate_account` path additionally exposes authenticated account owners to the same data loss if governance is degraded at the moment they donate. The bug is deterministic once triggered: a single `Err` from governance is sufficient to orphan the corresponding neuron permanently.

## Recommendation
Move `self.neuron_ids.retain(...)` inside the `Ok(_)` branch so it executes only on confirmed success:

```rust
match result {
    Ok(_) => {
        self.neuron_ids.retain(|id| id != &neuron_id);
        self.successfully_transferred_neurons.push(donated_neuron);
    }
    Err(e) => {
        donated_neuron.error = Some(e.to_string());
        self.failed_transferred_neurons.push(donated_neuron);
    }
}
```

Additionally, add a privileged recovery entry point (e.g., callable by the GTC controller or NNS governance) that re-reads `failed_transferred_neurons` and retries the transfer for each listed neuron ID, removing it from `failed_transferred_neurons` on success.

## Proof of Concept
Minimal deterministic unit test (mock governance):

```
1. Construct AccountState { neuron_ids: [N1, N2], has_claimed: false,
   has_donated: false, has_forwarded: false, ... }.
2. Mock GovernanceCanister::transfer_gtc_neuron:
     N1 → Err("transient reject")
     N2 → Ok(())
3. Call account.transfer(Some(custodian_id)).await.
4. Assert:
     account.neuron_ids                       == []   // BUG: N1 removed despite Err
     account.failed_transferred_neurons       == [N1]
     account.successfully_transferred_neurons == [N2]
5. Call account.transfer(Some(custodian_id)).await again.
6. Assert: neuron_ids still []; N1 is permanently unrecoverable.
```

This test requires no mainnet interaction and can be executed as a standard Rust async unit test with a mock `GovernanceCanister`. The same scenario is reproducible via a PocketIC integration test by pausing the governance canister between the GTC call and its response to simulate a reject.
