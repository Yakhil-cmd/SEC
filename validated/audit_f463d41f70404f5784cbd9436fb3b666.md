### Title
SNS Governance `disburse_maturity` Panics on u64 Overflow for High-Maturity Neurons — (`rs/sns/governance/src/governance.rs`)

---

### Summary

The `disburse_maturity` function in the SNS governance canister performs a `u64 × u64` multiplication using `checked_mul` but immediately calls `.expect()` on the result. When `maturity_e8s_equivalent * percentage_to_disburse` exceeds `u64::MAX`, the multiplication returns `None` and the `.expect()` call panics, trapping the canister. Any neuron holder with `DisburseMaturity` permission and sufficiently large accumulated maturity can trigger this unconditionally.

---

### Finding Description

At lines 1643–1649 of `rs/sns/governance/src/governance.rs`, the percentage-of-maturity calculation is:

```rust
let maturity_to_deduct = neuron
    .maturity_e8s_equivalent
    .checked_mul(disburse_maturity.percentage_to_disburse as u64)
    .expect("Overflow while processing maturity to disburse.")   // PANICS
    .checked_div(100)
    .expect("Error when processing maturity to disburse.")
    as u128;
``` [1](#0-0) 

Both operands are `u64`. The product overflows when:

```
maturity_e8s_equivalent > u64::MAX / percentage_to_disburse
```

With `percentage_to_disburse = 100` (the most easily triggered case), the threshold is:

```
u64::MAX / 100 ≈ 1.844 × 10^17 e8s  ≈  1.844 billion tokens
```

Any neuron whose `maturity_e8s_equivalent` exceeds this value will cause `.expect()` to panic on every `DisburseMaturity` call regardless of the percentage chosen (1–100), because even `percentage=1` overflows at `maturity > u64::MAX` which is unreachable, but `percentage=2` overflows at `maturity > ~9.22 × 10^18` e8s, and `percentage=100` overflows at `maturity > ~1.84 × 10^17` e8s.

The NNS governance codebase already has the correct fix in its equivalent helper `percentage_of_maturity`: it casts to `u128` before multiplying and propagates `None` as an `Err` rather than panicking:

```rust
(total_maturity_e8s as u128)
    .checked_mul(percentage_to_disburse as u128)
    ...
    .ok_or_else(|| InitiateMaturityDisbursementError::Unknown { ... })
``` [2](#0-1) 

The SNS path never received this fix.

---

### Impact Explanation

A panic in an IC canister update call causes the entire canister execution to **trap**. On the Internet Computer, a trap in an update message rolls back state changes for that message but the canister itself remains alive for future calls. However, if the neuron's maturity is not reduced before the trap (state is rolled back), the condition persists permanently: every subsequent `DisburseMaturity` call on that neuron will also trap. The neuron's maturity cannot be reduced through this path, and the governance canister will trap on every such call until an upgrade patches the code. Depending on the SNS's upgrade cadence and the number of affected neurons, this can cause sustained partial unavailability of the SNS governance canister's update path.

---

### Likelihood Explanation

The overflow threshold for `percentage=100` is ~1.844 × 10^17 e8s. For an SNS with a total token supply in the billions (common for large SNS deployments), a whale neuron staked for years accumulating voting rewards could reach this threshold. Additionally, the `maturity_e8s_equivalent` field can be set directly via privileged `AddMaturity` proposals (used in testing and SNS initialization), making the precondition reachable in practice. The attacker only needs `NeuronPermissionType::DisburseMaturity` on their own neuron — a standard, non-privileged permission granted to neuron controllers by default.

---

### Recommendation

Replace the `u64 × u64` multiplication with `u128` arithmetic, mirroring the NNS fix:

```rust
// In rs/sns/governance/src/governance.rs, disburse_maturity()
let maturity_to_deduct = (neuron.maturity_e8s_equivalent as u128)
    .checked_mul(disburse_maturity.percentage_to_disburse as u128)
    .and_then(|v| v.checked_div(100))
    .and_then(|v| u64::try_from(v).ok())
    .ok_or_else(|| GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        "Overflow while calculating maturity to disburse.",
    ))?;
```

This matches the pattern already used in `rs/nns/governance/src/governance/disburse_maturity.rs` and eliminates the panic path entirely.
