### Title
Missing Refund Callback on Failed `ft_transfer` in `ExitToNear` Precompile When `error_refund` Feature Is Disabled - (File: `engine-precompiles/src/native.rs`)

---

### Summary

When a user bridges ERC-20 tokens back to NEAR via the `ExitToNear` precompile, the ERC-20 tokens are burned first, then a NEAR `ft_transfer` promise is scheduled. If that promise fails and the `error_refund` compile-time feature is not enabled (which is the case in the production build), no refund callback is attached and the user's tokens are permanently destroyed with no recovery path.

---

### Finding Description

The `ExitToNear` precompile's `run` function constructs a `ExitToNearPrecompileCallbackArgs` struct whose `refund` field is unconditionally set to `None` when the `error_refund` Cargo feature is absent: [1](#0-0) 

When `refund` is `None` and `transfer_near` is also `None` (the common ERC-20 exit case), `callback_args` equals the default value, so the code takes the no-callback branch: [2](#0-1) 

This means only a bare `PromiseArgs::Create(transfer_promise)` is emitted — no `exit_to_near_precompile_callback` is ever scheduled. If the downstream `ft_transfer` or `ft_transfer_call` promise fails, there is nothing to re-mint the burned ERC-20 tokens.

The refund logic that *would* re-mint tokens lives exclusively in `exit_to_near_precompile_callback`: [3](#0-2) 

But that callback is never registered in the no-`error_refund` build.

The production `engine` crate's feature list confirms `error_refund` is **not** a default feature and is **not** pulled in by the `contract` feature used for the production WASM: [4](#0-3) 

Likewise in the precompiles crate: [5](#0-4) 

The ERC-20 Solidity contracts burn tokens before calling the precompile, making the burn unconditional: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When `ft_transfer` fails (e.g., the recipient NEAR account is not registered with the NEP-141 contract, or the Aurora engine's NEP-141 balance is insufficient), the sequence is:

1. ERC-20 tokens are burned from the user's EVM balance — irreversible.
2. The NEP-141 `ft_transfer` promise fails — tokens are not delivered.
3. No refund callback exists — ERC-20 tokens are never re-minted.

The user loses the full bridged amount with no on-chain recovery path. The test suite explicitly documents this outcome: [8](#0-7) [9](#0-8) 

---

### Likelihood Explanation

**High.** The `ft_transfer` promise can fail under several realistic, user-reachable conditions:

- The recipient NEAR account is not registered (has no storage deposit) with the NEP-141 contract — a common mistake for new users.
- The Aurora engine's NEP-141 custodial balance is temporarily insufficient.
- The NEP-141 contract is paused or has been upgraded with breaking changes.

None of these conditions require admin compromise or privileged access. Any unprivileged EVM user who calls `withdrawToNear` on a bridged ERC-20 token is exposed.

---

### Recommendation

1. **Enable `error_refund` in the production build** by adding it to the `contract` feature in `engine/Cargo.toml`, or make it unconditionally compiled in.
2. **Alternatively**, always attach the `exit_to_near_precompile_callback` regardless of the feature flag, passing the refund address from the calldata (as `EvmErc20V2.sol` already does).
3. **Validate** that the recipient is registered with the NEP-141 contract before burning ERC-20 tokens, reverting the EVM transaction if registration is absent.

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature (the default production build).
2. Bridge a NEP-141 token to Aurora, receiving ERC-20 tokens.
3. Call `withdrawToNear("unregistered.near", amount)` on the ERC-20 contract.
4. The ERC-20 tokens are burned; the `ft_transfer` to `unregistered.near` fails because that account has no storage deposit with the NEP-141 contract.
5. Observe: ERC-20 balance decreases by `amount`; NEP-141 balance of `unregistered.near` remains 0; no refund is issued.

The test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` (lines 623–666) reproduces this exact scenario and explicitly asserts the token loss when `error_refund` is absent. [10](#0-9)
