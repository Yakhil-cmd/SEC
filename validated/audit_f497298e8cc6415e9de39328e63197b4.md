### Title
External `account_balance` Query Used for Participation Accounting Allows Attacker to Manipulate SNS Swap State - (File: rs/sns/swap/src/swap.rs)

### Summary

The SNS Swap canister's `refresh_buyer_token_e8s` function determines a buyer's ICP contribution by querying the external ICP ledger's `account_balance` rather than using internal accounting. Because `refresh_buyer_tokens` is a public update endpoint that accepts any `buyer` principal, an unprivileged attacker can send ICP to a victim's subaccount of the swap canister and then call `refresh_buyer_tokens` on behalf of the victim, inflating the victim's recorded participation. This can consume the swap's available participation capacity, force premature swap commitment, and prevent legitimate buyers from participating.

### Finding Description

`refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs` queries the ICP ledger for the raw on-chain balance of a buyer's subaccount and uses that value directly to compute the participation increment:

```rust
let e8s = {
    let account = Account {
        owner: this_canister.get().0,
        subaccount: Some(principal_to_subaccount(&buyer)),
    };
    icp_ledger
        .account_balance(account)
        .await
        ...
        .get_e8s()
};
``` [1](#0-0) 

The increment is then computed as `e8s - old_amount_icp_e8s`, capped by `available_direct_participation_e8s()`: [2](#0-1) 

The public canister endpoint `refresh_buyer_tokens` accepts an arbitrary `buyer` field — if non-empty, it parses and uses that principal directly without any caller-identity check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()
};
``` [3](#0-2) 

This means any ingress sender can trigger a balance refresh for any victim principal.

### Impact Explanation

An attacker can:

1. Transfer ICP directly to `victim_principal`'s subaccount of the swap canister on the ICP ledger (subaccount is deterministically derived from the principal via `principal_to_subaccount`).
2. Call `refresh_buyer_tokens { buyer: "victim_principal", ... }` from any identity.
3. The swap canister queries the inflated subaccount balance and credits the victim with participation up to `max_participant_icp_e8s`.

Concrete consequences:
- **Swap capacity exhaustion**: By repeating this across many sybil victim principals (each funded to `max_participant_icp_e8s`), the attacker can fill `available_direct_participation_e8s` to zero, blocking all legitimate buyers. [4](#0-3) 
- **Premature swap commitment**: Once `current_direct_participation_e8s >= max_direct_participation_icp_e8s`, `can_commit` returns true and the swap can be committed immediately before the intended participant set has joined. [5](#0-4) 
- **Forced participation of unwilling users**: A victim who never intended to participate is credited with ICP participation; their ICP (sent by the attacker) is locked until swap finalization.

The attacker loses the ICP they send (it goes to SNS governance on commit, or is refunded to the victim on abort), but the disruption to the swap's integrity is real and the cost to execute is bounded by `max_participant_icp_e8s` per victim slot consumed.

### Likelihood Explanation

- `refresh_buyer_tokens` is a standard public `#[update]` endpoint reachable by any ingress sender with no access control. [6](#0-5) 
- The `buyer` field is explicitly designed to accept third-party principals (to allow bots/agents to call on behalf of users).
- ICP transfers to arbitrary subaccounts of the swap canister are a normal part of the participation flow, so the ledger will accept them without restriction.
- No special privilege, key, or majority is required.

### Recommendation

1. **Restrict `refresh_buyer_tokens` to self-refresh only**: Require `arg.buyer` to be empty or equal to `ic_cdk::api::msg_caller()`. If third-party refresh is needed, add an explicit allowlist or signed authorization from the buyer.
2. **Use internal accounting for increments**: Track the total ICP deposited per buyer internally (e.g., record the last-seen balance and only accept increments that match a corresponding transfer memo/block index), rather than trusting the raw ledger balance which can be inflated by anyone sending ICP to the subaccount.

### Proof of Concept

```
// Attacker setup:
// 1. Derive victim's subaccount on the swap canister
let victim_subaccount = principal_to_subaccount(&victim_principal);
let victim_swap_account = Account {
    owner: swap_canister_id.into(),
    subaccount: Some(victim_subaccount),
};

// 2. Transfer ICP to victim's subaccount (fills it to max_participant_icp_e8s)
icp_ledger.transfer(TransferArg {
    to: victim_swap_account,
    amount: Nat::from(max_participant_icp_e8s),
    ...
}).await;

// 3. Call refresh_buyer_tokens for the victim from any identity
swap_canister.refresh_buyer_tokens(RefreshBuyerTokensRequest {
    buyer: victim_principal.to_string(),
    confirmation_text: None,
}).await;
// -> victim is now credited with max_participant_icp_e8s of participation
// -> available_direct_participation_e8s decreases by max_participant_icp_e8s
// Repeat with more sybil victims to exhaust the swap's capacity.
```

The swap's `available_direct_participation_e8s` is reduced by each such call, and once it reaches zero, legitimate buyers are rejected with "ICP target" errors, as demonstrated by the existing test `test_refresh_buyer_tokens_token_limit`. [7](#0-6)

        let current_direct_participation_e8s = self.current_direct_participation_e8s();

                    < current_direct_participation_e8s ({current_direct_participation_e8s})"
                );
                0
            })
    }
