### Title
Unauthenticated ERC-20 Minting via Open `deploy_erc20_token` Registration and Unverified `ft_on_transfer` Caller - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

`deploy_erc20_token` is callable by any NEAR account with no access control. Once an attacker registers their own NEAR contract as a NEP-141 token, they can directly call `ft_on_transfer` on Aurora from that contract with an arbitrary `amount` field. Aurora's `ft_on_transfer` trusts any registered NEP-141 predecessor as a legitimate minting authority without verifying that real tokens were actually transferred. This allows an unprivileged attacker to mint arbitrary ERC-20 tokens on Aurora without depositing any real NEP-141 tokens.

---

### Finding Description

**Step 1 — Open registration (`deploy_erc20_token`)**

`deploy_erc20_token` in `engine/src/contract_methods/connector.rs` only checks `require_running`; there is no `require_owner_only` guard:

```rust

                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

Any NEAR account can call this and register any `AccountId` as a NEP-141 token, creating a new ERC-20 mirror for it. [1](#0-0) 

The comment in the `WithMetadata` branch says "this transaction could be executed by the owner of the contract only," but no such enforcement exists in either branch. [2](#0-1) 

**Step 2 — Unverified minting in `ft_on_transfer`**

`ft_on_transfer` dispatches to `receive_erc20_tokens` for any predecessor that is not the ETH connector account:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,   // ← trusted solely because it is registered
        &args,
        &current_account_id,
        handler,
    )
};
``` [3](#0-2) 

`receive_erc20_tokens` looks up the registered ERC-20 address for the predecessor and calls `mint` on it as the engine admin:

```rust
let erc20_token = get_erc20_from_nep141(&self.io, token)?;
let erc20_admin_address = current_address(current_account_id);
let result = self.call(
    &erc20_admin_address,
    &erc20_token,
    Wei::zero(),
    setup_receive_erc20_tokens_input(&recipient, amount),
    u64::MAX, ...
``` [4](#0-3) 

There is no check that the NEAR runtime actually transferred tokens from the predecessor to Aurora before this call. The `amount` field comes entirely from the JSON args supplied by the caller: [5](#0-4) 

The `setup_receive_erc20_tokens_input` function encodes a `mint(address, uint256)` call using the attacker-controlled `amount`: [6](#0-5) 

---

### Impact Explanation

An attacker can mint an unbounded quantity of any bridged ERC-20 token on Aurora without depositing the corresponding NEP-141 tokens. These minted tokens can be:

- Swapped in Aurora-native AMMs for ETH or other ERC-20 tokens, draining liquidity pools.
- Used as collateral in lending protocols to borrow real assets.
- Exited back to NEAR via the bridge, making the bridge insolvent (it cannot release NEP-141 tokens it never received).

**Impact: Critical — direct theft of user funds and bridge insolvency.**

---

### Likelihood Explanation

The entry path requires only:
1. Deploying a NEAR contract (permissionless, costs a small amount of NEAR for storage).
2. Calling `deploy_erc20_token` on Aurora (permissionless, no owner check).
3. Calling `ft_on_transfer` directly from the attacker's contract (permissionless NEAR cross-contract call).

No privileged keys, governance access, or social engineering are required. Any NEAR account holder can execute this. **Likelihood: High.**

---

### Recommendation

1. **Restrict `deploy_erc20_token`**: Add `require_owner_only` (or a dedicated allowlist check) to `deploy_erc20_token` in `engine/src/contract_methods/connector.rs` so only the engine owner can register new NEP-141 ↔ ERC-20 mappings.

2. **Verify actual token receipt in `ft_on_transfer`**: The NEP-141 standard guarantees that `ft_on_transfer` is called by the token contract only as part of `ft_transfer_call`, after tokens are transferred. Aurora should not rely solely on the predecessor being registered; it should also verify that the NEAR runtime's attached deposit or promise result confirms the transfer occurred. At minimum, document and enforce that `ft_on_transfer` must only be reachable via `ft_transfer_call` (which the NEAR runtime enforces when called correctly, but Aurora currently does not validate).

---

### Proof of Concept

```
1. Attacker deploys a NEAR contract at `evil.near`.

2. Attacker calls aurora.near::deploy_erc20_token(
       DeployErc20TokenArgs::Legacy("evil.near")
   )
   → Aurora registers evil.near → ERC-20 address 0xABCD...
   → No owner check; succeeds for any caller.

3. evil.near directly calls aurora.near::ft_on_transfer with:
   {
     "sender_id": "evil.near",
     "amount":    "1000000000000000000000000",
     "msg":       "<attacker_evm_address_hex>"
   }
   predecessor_account_id = "evil.near"

4. ft_on_transfer:
   - predecessor != connector_account_id  → goes to receive_erc20_tokens
   - get_erc20_from_nep141("evil.near")   → returns 0xABCD... (registered in step 2)
   - calls mint(attacker_evm_address, 1000000000000000000000000) as engine admin

5. Attacker now holds 1e24 ERC-20 tokens (backed by zero real NEP-141 tokens).
   Attacker swaps them in an Aurora AMM for ETH, draining the pool.
```

                );
                Ok(PromiseOrValue::Value(address))
            }
