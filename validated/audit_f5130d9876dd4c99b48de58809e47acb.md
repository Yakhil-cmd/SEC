Audit Report

## Title
Attacker-Controlled Memory Pressure Causes Premature Eviction of Victim's `Completed` Ingress Status to `Done` — (`rs/replicated_state/src/metadata_state.rs`)

## Summary
`IngressHistoryState::insert` triggers `forget_terminal_statuses` whenever `memory_usage > ingress_memory_capacity`. That function evicts `Completed`/`Failed` entries to `Done` in strict chronological order (oldest pruning time first), with no per-user isolation. An unprivileged attacker who submits ingress messages to a canister returning large reply payloads — completing after the victim's message — will have later pruning times, causing the victim's earlier-pruning-time entry to be evicted first, permanently destroying the victim's reply payload before they can poll `read_state`.

## Finding Description
In `IngressHistoryState::insert` (lines 1583–1610), when a terminal status is inserted, its pruning time is set to `time + MAX_INGRESS_TTL`. The `next_terminal_time` reset guard at line 1590 only resets to an earlier time: `if self.next_terminal_time > timeout && state.is_terminal_with_payload()`. Attacker entries completing at T2 > T1 produce a timeout of T2 + MAX_INGRESS_TTL, which is greater than the current `next_terminal_time` (T1 + MAX_INGRESS_TTL), so the guard condition is false and `next_terminal_time` remains pointing at the victim's bucket.

When `forget_terminal_statuses` is called (lines 1693–1733), it iterates `pruning_times` starting from `next_terminal_time`. The memory check at line 1699 (`if self.memory_usage <= target_size { break; }`) fires at the top of each bucket iteration, before processing entries. Since memory is still over capacity when the victim's bucket is reached, the victim's `Completed` status is transitioned to `Done` (lines 1717–1729), zeroing its `payload_bytes`. The loop may then break after the victim's bucket if memory drops below target, leaving attacker entries intact as `Completed`.

There is no per-user memory quota, no minimum retention window for recently-completed entries, and no protection preventing one user's insertions from evicting another user's `Completed` status.

## Impact Explanation
The victim's `Completed` ingress status is irreversibly transitioned to `Done`. The reply payload (return value of the canister call) is permanently lost. The victim calling `read_state` after eviction receives `Done` with no data and no way to recover the response. The canister's state change is committed, but the user cannot observe the result. This constitutes a targeted denial-of-response attack, fitting the **High** impact class: "Application/platform-level DoS, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
The attacker requires: (a) a canister (their own or any canister returning large responses) that returns large reply payloads, (b) enough cycles to submit the required number of ingress messages, and (c) timing such that their messages complete after the victim's. The number of messages required is approximately `ingress_history_memory_capacity / max_reply_size`. The attack is repeatable, automatable, and requires no privileged access, governance majority, or subnet compromise. The timing constraint (attacker messages must complete after victim's) is achievable by monitoring the subnet for victim message submission and responding quickly, or by continuously flooding the ingress history to keep it over capacity.

## Recommendation
- Introduce per-user or per-message minimum retention guarantees: a `Completed` entry should not be evicted until it has been resident for at least a configurable minimum duration.
- Account memory pressure per-originating-user so that one user's large payloads cannot evict another user's entries.
- Consider evicting the largest entries first within the same pruning bucket rather than strictly oldest-first, reducing the ability to target specific victims.
- Enforce per-user caps on total ingress history memory to prevent a single attacker from consuming the shared budget.

## Proof of Concept
State-machine test outline using `IngressHistoryState` directly:
1. Insert victim's `Completed` status with a small payload at time T; confirm `ingress_history.get(victim_id)` returns `Completed`.
2. Insert N attacker `Completed` statuses with large payloads (each near max reply size) at time T + 1ns, where N × payload_size > `ingress_memory_capacity`.
3. Assert that `ingress_history.get(victim_id)` now returns `Done` (evicted).
4. Assert that at least some attacker entries remain `Completed` (they have later pruning times and are processed after the victim's entry, with the loop breaking once memory drops below target).

This directly demonstrates that the victim's reply is lost due to attacker-controlled memory pressure, with no privileged access required. The existing test `ingress_history_insert_beyond_limit_will_succeed` (metadata_state/tests.rs lines 1362–1437) already confirms the eviction-to-`Done` behavior; the PoC extends it to the cross-user scenario.
