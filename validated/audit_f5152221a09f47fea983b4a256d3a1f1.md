### Title
Integer Overflow in EIP-1559 Base Fee Delta Computation Causes Prover Liveness Failure - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/block_header.rs`)

### Summary

The `verify_chain` function in `PectraForkHeader`'s `ChainChecker` implementation computes the expected EIP-1559 base fee using unchecked `u64` multiplication. When Ethereum base fees are high and blocks are near-full, `parent_base_fee_per_gas * gas_used_delta` overflows `u64`, producing a silently wrapped (incorrect) delta. The subsequent `assert_eq!` comparing the computed expected base fee against the actual header value then panics, making valid Ethereum blocks unprovable by ZKsync OS.

### Finding Description

In `verify_chain`, the EIP-1559 base fee transition is verified at `depth == 0` (the parent block). Two branches compute `base_fee_per_gas_delta` using plain `u64` multiplication:

**Upward branch (block above target):**
```rust
let base_fee_per_gas_delta = core::cmp::max(
    parent_base_fee_per_gas * gas_used_delta   // ← unchecked u64 * u64
        / parent_gas_target
        / EIP_1559_BASE_FEE_MAX_CHANGE_DENOMINATOR,
    1,
);
```

**Downward branch (block below target):**
```rust
let base_fee_per_gas_delta = parent_base_fee_per_gas * gas_used_delta  // ← unchecked u64 * u64
    / parent_gas_target
    / EIP_1559_BASE_FEE_MAX_CHANGE_DENOMINATOR;
```

Both are `u64 * u64` with no overflow guard. In Rust release mode (the proving environment), overflow wraps silently. The wrapped result produces a wrong `expected_base_fee_per_gas`, which then fails the hard assertion:

```rust
assert_eq!(expected_base_fee_per_gas, self.base_fee_per_gas);
```

This assertion panic aborts the proving run. [1](#0-0) 

The overflow threshold is:

```
parent_base_fee_per_gas * gas_used_delta > u64::MAX (≈ 1.844e19)
```

With Ethereum's current gas limit (~36M), `gas_used_delta` can reach up to `gas_limit / 2 = 18M`. Overflow occurs when:

```
parent_base_fee_per_gas > 1.844e19 / 18e6 ≈ 1024 gwei
```

Ethereum has historically reached 500–600 gwei during peak congestion (May 2021, November 2021 NFT/DeFi events). With a rising gas limit, the threshold decreases further.

The forward execution path explicitly skips this check (noted in `metadata_op.rs`), but the proving path invokes `verify_chain` to enforce cross-block pricing invariants. [2](#0-1) 

### Impact Explanation

When Ethereum base fees exceed ~1000 gwei with a near-full block, the ZKsync OS prover panics inside `verify_chain` while verifying the Ethereum L1 settlement-layer header chain. No ZKsync OS batch can be finalized on Ethereum during this window. Users cannot complete L2→L1 withdrawals, and the rollup's liveness is broken until Ethereum fees drop below the overflow threshold.

**Vulnerability class**: Valid-execution unprovability / liveness failure.

### Likelihood Explanation

Ethereum base fees above 1000 gwei have not yet occurred but are plausible during extreme congestion events (large NFT drops, protocol launches, market crashes). As Ethereum's gas limit continues to increase (currently ~36M, trending upward), the overflow threshold decreases proportionally, making the condition easier to reach. The condition is also reachable by a well-funded attacker who spam-fills Ethereum blocks to drive up the base fee.

### Recommendation

Replace the unchecked `u64` multiplications with overflow-safe arithmetic. Use `u128` for the intermediate product, or use `checked_mul` and propagate an error:

```rust
// Safe version using u128 intermediate
let product = (parent_base_fee_per_gas as u128) * (gas_used_delta as u128);
let base_fee_per_gas_delta = core::cmp::max(
    (product / parent_gas_target as u128 / EIP_1559_BASE_FEE_MAX_CHANGE_DENOMINATOR as u128) as u64,
    1,
);
```

This matches how Ethereum's reference clients (go-ethereum, reth) handle the computation — using wider integer types to avoid overflow.

### Proof of Concept

Given:
- `parent_gas_limit = 36_000_000` (current Ethereum gas limit)
- `parent_gas_target = 18_000_000` (half of gas limit)
- `parent_gas_used = 36_000_000` (full block)
- `gas_used_delta = 36_000_000 - 18_000_000 = 18_000_000`
- `parent_base_fee_per_gas = 1_100_000_000_000` (1100 gwei, plausible during extreme congestion)

Computation:
```
parent_base_fee_per_gas * gas_used_delta
= 1_100_000_000_000 * 18_000_000
= 19_800_000_000_000_000_000_000
```

`u64::MAX = 18_446_744_073_709_551_615`

The product `19.8e21 > u64::MAX`, so it wraps to:
```
19_800_000_000_000_000_000_000 mod 2^64
= 19_800_000_000_000_000_000_000 - 18_446_744_073_709_551_616
= 1_353_255_926_290_448_384_000  (wrong, much smaller)
```

The wrapped delta divided by `parent_gas_target * 8` gives a wrong `expected_base_fee_per_gas`. The `assert_eq!` at line 423 then panics because the actual `self.base_fee_per_gas` (correctly computed by Ethereum) does not match the overflowed value, aborting the proving run. [3](#0-2)
