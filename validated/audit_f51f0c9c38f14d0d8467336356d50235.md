Audit Report

## Title
`FieldsDisplay` Branch of `add_expiration` Silently Wraps `expires_at > i64::MAX`, Displaying a Fabricated Timestamp to Hardware Wallet Users — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary

The `FieldsDisplay` arm of `add_expiration` performs an unchecked `u64 → i64` cast on the nanosecond timestamp at line 265, producing a silently wrong `TimestampSeconds` value for any `expires_at > i64::MAX`. The `GenericDisplay` branch incidentally catches the same overflow via a nanosecond range check and returns an explicit error string; the `FieldsDisplay` branch has no equivalent guard. A malicious dApp can exploit this to make a hardware wallet display a fabricated past expiration date (e.g., Unix epoch) while the actual on-chain approval is permanent.

## Finding Description

In `packages/icrc-ledger-types/src/icrc21/responses.rs`, the `FieldsDisplay` arm at lines 264–271:

```rust
Some(expires_at) => {
    let seconds = (expires_at as i64) / 10_i64.pow(9);
    fields_display.fields.push((
        "Approval expiration".to_string(),
        Value::TimestampSeconds { amount: seconds as u64 },
    ))
}
```

The cast `expires_at as i64` is a Rust wrapping (bit-reinterpretation) cast. For `expires_at = u64::MAX`:
- `u64::MAX as i64 = -1`
- `-1 / 1_000_000_000 = 0`
- `0 as u64 = 0`
- Result: `TimestampSeconds { amount: 0 }` → hardware wallet renders **"Jan 1, 1970"**

The `GenericDisplay` branch at lines 232–243 performs the same initial cast but incidentally catches the overflow: for `u64::MAX`, `nanos` wraps to `4294967295`, which exceeds the valid nanosecond range (0–999999999), causing `replace_nanosecond` to fail and return `"Invalid nanosecond: 4294967295"`. The `FieldsDisplay` branch has no such check.

The `expires_at` value flows from the Candid-decoded `ApproveArgs` directly into `add_expiration` with no range check, as confirmed at `packages/icrc-ledger-types/src/icrc21/lib.rs` lines 439–463: the `arg` blob is fully attacker-controlled, decoded via `Decode!(&consent_msg_request.arg, ApproveArgs)`, and `expires_at` is passed to `display_message_builder.with_expires_at(expires_at)` without validation.

## Impact Explanation

The ICRC-21 consent message is the sole security signal a hardware wallet has before the user signs a transaction. The `FieldsDisplay` variant is specifically designed for structured rendering on constrained devices (Ledger hardware wallets, etc.). A malicious dApp can cause the hardware wallet to display **"Approval expiration: Jan 1, 1970"** for an `icrc2_approve` with `expires_at = u64::MAX`. The user interprets this as a harmless, already-expired approval and signs. The on-chain approval is permanent (`u64::MAX` nanoseconds ≈ year 2554), granting the spender an indefinite allowance to drain the user's tokens. This is a concrete financial loss impact on ICRC ledger users.

**Severity: High ($2,000–$10,000)** — Significant ICRC ledger security impact with concrete user funds harm; exploitation requires per-target interaction with a malicious dApp but no special privileges.

## Likelihood Explanation

- `icrc21_consent_message` is a public query endpoint; no authentication is required.
- The `arg` blob is fully attacker-controlled (Candid encoding of `ApproveArgs`).
- `u64::MAX` is a valid `nat64` value; the ledger imposes no upper bound on `expires_at` beyond "must be in the future," and `u64::MAX` (year 2554) passes that check.
- Hardware wallets implementing `FieldsDisplay` are the intended production consumers of this code path.
- The attack is repeatable and requires no special on-chain state.

## Recommendation

Replace the unchecked cast with an explicit range check before converting, mirroring the error-handling already present in the `GenericDisplay` branch:

```rust
Some(expires_at) => {
    if expires_at > i64::MAX as u64 {
        fields_display.fields.push((
            "Approval expiration".to_string(),
            Value::Text {
                content: format!("Invalid timestamp: {expires_at}"),
            },
        ));
    } else {
        let seconds = (expires_at as i64) / 1_000_000_000_i64;
        fields_display.fields.push((
            "Approval expiration".to_string(),
            Value::TimestampSeconds { amount: seconds as u64 },
        ));
    }
}
```

## Proof of Concept

```rust
// Unit test
let expires_at: u64 = u64::MAX;
let mut msg = ConsentMessage::FieldsDisplayMessage(Default::default());
msg.add_expiration(Some(expires_at), None);

if let ConsentMessage::FieldsDisplayMessage(fd) = &msg {
    let (_, value) = fd.fields.iter()
        .find(|(k, _)| k == "Approval expiration").unwrap();
    // BUG: displays Unix epoch (0) instead of an error
    assert_eq!(*value, Value::TimestampSeconds { amount: 0 });
    // Hardware wallet renders this as "Approval expiration: Jan 1, 1970"
}
```
