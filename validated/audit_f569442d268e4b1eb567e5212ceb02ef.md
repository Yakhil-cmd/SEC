### Title
`set_owner` Accepts Empty `AccountId`, Causing Irrevocable Loss of Contract Ownership — (`File: engine/src/contract_methods/admin.rs`)

---

### Summary

The `set_owner` admin function in the Aurora Engine does not validate that the incoming `new_owner` `AccountId` is non-empty. The `BorshDeserialize` implementation for `AccountId` explicitly accepts an empty string and returns `AccountId::default()` (an empty `AccountId`) for backward compatibility. If the owner accidentally passes a borsh-encoded empty string, `state.owner_id` is permanently set to `""`, which no real NEAR account can ever match, irrevocably locking out all owner-only administrative functions.

---

### Finding Description

The `set_owner` function in `engine/src/contract_methods/admin.rs` reads the new owner from borsh-encoded input and applies a single guard — that the new owner is not the same as the current owner:

```rust
let args: SetOwnerArgs = io.read_input_borsh()?;
if state.owner_id == args.new_owner {
    return Err(errors::ERR_SAME_OWNER.into());
}
state.owner_id = args.new_owner;
state::set_state(&mut io, &state)?;
``` [1](#0-0) 

There is no check that `args.new_owner` is a valid, non-empty account ID.

The `BorshDeserialize` implementation for `AccountId` contains an explicit backward-compatibility escape hatch: if the deserialized string is empty, it silently returns `Self::default()` — an `AccountId` wrapping an empty `Box<str>` — without returning an error:

```rust
if account.is_empty() {
    return Ok(Self::default());
}
``` [2](#0-1) 

`AccountId::default()` is `AccountId("")`. The `AccountId::validate` function enforces a minimum length of 2 characters for any real NEAR account: [3](#0-2) 

This means no real NEAR account can ever have an empty ID. Once `state.owner_id` is set to `""`, the `require_owner_only` guard will permanently reject every caller:

```rust
if &state.owner_id != predecessor_account_id {
    return Err(errors::ERR_NOT_ALLOWED.into());
}
``` [4](#0-3) 

All owner-gated functions — `set_owner`, `set_upgrade_delay_blocks`, `pause_contract`, `resume_contract`, `set_key_manager`, `attach_full_access_key`, `stage_upgrade`, `upgrade`, `resume_precompiles` — become permanently inaccessible. [5](#0-4) 

---

### Impact Explanation

Once `owner_id` is set to the empty `AccountId`:

1. The contract can never be upgraded (`stage_upgrade` and `upgrade` are owner-only).
2. The contract can never be paused in an emergency (`pause_contract` is owner-only).
3. The key manager cannot be changed (`set_key_manager` is owner-only).
4. No new full-access keys can be attached (`attach_full_access_key` is owner-only).

The Aurora Engine holds all user ETH balances and ERC-20 mirror token state. If a critical vulnerability is subsequently discovered in the EVM execution path, the contract cannot be upgraded or paused to protect user funds. This constitutes **permanent freezing of funds** — the administrative capability required to protect or recover funds is irrevocably destroyed.

---

### Likelihood Explanation

The trigger is an accidental operator error: the owner calls `set_owner` with a borsh-encoded empty string (four zero bytes `[0x00, 0x00, 0x00, 0x00]` as the length-prefixed borsh encoding of `""`). This is a realistic mistake when constructing borsh arguments programmatically with an uninitialized or default-valued struct. The `BorshDeserialize` path for `AccountId` explicitly accepts this input without error, making the misconfiguration silent and unrecoverable.

---

### Recommendation

Add an explicit non-empty validation in `set_owner` before writing the new owner to state:

```rust
let args: SetOwnerArgs = io.read_input_borsh()?;
if args.new_owner.as_bytes().is_empty() {
    return Err(b"ERR_INVALID_OWNER".into());
}
if state.owner_id == args.new_owner {
    return Err(errors::ERR_SAME_OWNER.into());
}
```

Additionally, consider removing or guarding the empty-string escape hatch in `AccountId::BorshDeserialize` so that it no longer silently accepts invalid account IDs in new contexts, or at minimum document that callers must not rely on this behavior for security-sensitive fields.

---

### Proof of Concept

1. The Aurora Engine is deployed. `state.owner_id` is set to `"aurora"` (a valid NEAR account).
2. The owner constructs a `SetOwnerArgs` with `new_owner` borsh-encoded as an empty string: input bytes = `[0x00, 0x00, 0x00, 0x00]`.
3. The owner calls `set_owner` with this input.
4. `io.read_input_borsh::<SetOwnerArgs>()` deserializes successfully. The `BorshDeserialize` for `AccountId` sees an empty string and returns `AccountId::default()` = `AccountId("")`.
5. The guard `state.owner_id == args.new_owner` evaluates `"aurora" == ""` → `false`, so no error is returned.
6. `state.owner_id` is written as `""` and persisted.
7. Any subsequent call to `set_owner` (or any other owner-only function) by any real NEAR account fails with `ERR_NOT_ALLOWED`, because `predecessor_account_id` (minimum 2 chars) can never equal `""`.
8. Administrative control is permanently and irrevocably lost. [6](#0-5) [7](#0-6) [8](#0-7)

impl BorshDeserialize for AccountId {
    fn deserialize_reader<R: io::Read>(reader: &mut R) -> io::Result<Self> {
        let account: String = BorshDeserialize::deserialize_reader(reader)?;

        // It's for saving backward compatibility.
        if account.is_empty() {
            return Ok(Self::default());
        }

        Self::try_from(account)
            .map_err(|e| io::Error::new(io::ErrorKind::InvalidData, e.to_string()))
    }
