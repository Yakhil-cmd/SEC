### Title
EvmAdmin Whitelist Bypass via Factory Contract CREATE — (`engine/src/engine.rs`, `engine/src/contract_methods/silo/mod.rs`)

### Summary

`assert_access` gates contract deployment using `transaction.to.is_none()` as the sole signal that a CREATE is occurring. A CALL transaction (`to` is `Some`) to a factory contract that internally executes `CREATE`/`CREATE2` passes only the `is_allow_submit` check (Account + Address whitelists), never `is_allow_deploy` (Admin + EvmAdmin whitelists). The EvmAdmin whitelist is therefore completely bypassable.

### Finding Description

`assert_access` in `engine/src/engine.rs` dispatches between two whitelist checks based solely on whether the top-level transaction has a `to` field: [1](#0-0) 

```
transaction.to.is_some()  →  is_allow_submit  (Account + Address whitelists)
transaction.to.is_none()  →  is_allow_deploy  (Admin   + EvmAdmin whitelists)
```

`is_allow_deploy` checks the Admin and EvmAdmin whitelists: [2](#0-1) 

`is_allow_submit` checks only Account and Address whitelists: [3](#0-2) 

`assert_access` is called exactly once, before EVM execution begins: [4](#0-3) 

There is no hook, callback, or re-invocation of `is_allow_deploy` anywhere inside the EVM execution loop for internal `CREATE`/`CREATE2` opcodes. A grep across all of `engine/src/` confirms `is_allow_deploy` and `assert_access` appear only in `engine.rs` and `silo/mod.rs`, never inside the EVM `Backend` trait implementation. [5](#0-4) 

The `Backend` trait implementation for `Engine` does not intercept CREATE opcodes with any whitelist check. Once the EVM begins executing, it can freely issue `CREATE`/`CREATE2` sub-calls.

### Impact Explanation

An attacker whitelisted only for Account + Address (submit) can:

1. Pre-deploy a factory contract (requires a separate Admin/EvmAdmin-whitelisted account, or the factory was already deployed before whitelists were enabled).
2. Send a CALL transaction (`to = factory_address`) — passes `is_allow_submit`.
3. The factory executes `CREATE`/`CREATE2` internally — no whitelist check occurs.
4. A new contract is deployed under the attacker's control without Admin/EvmAdmin membership.

The deployed contract can contain arbitrary logic: `selfdestruct` to drain ETH balances, re-entrancy attacks against other EVM contracts holding protocol funds, or malicious ERC-20 logic. This satisfies the Insolvency impact scope.

### Likelihood Explanation

- Preconditions are realistic: silo mode with all whitelists enabled is the intended production configuration.
- The attacker only needs Account + Address whitelist membership, which is the lower-privilege tier.
- A factory contract can be pre-deployed by any Admin/EvmAdmin account (including the attacker's accomplice or a previously deployed public factory).
- No admin compromise, key leak, or governance capture is required.

### Recommendation

The whitelist check must be enforced not only at the transaction boundary but also for any internal `CREATE`/`CREATE2` operation. Options:

1. **Hook the EVM `Backend`**: Override the `create` method in the `Backend` implementation for `Engine` to call `is_allow_deploy` before allowing any internal contract creation.
2. **Disable CREATE in submit path**: When silo mode is active and EvmAdmin whitelist is enabled, reject any EVM execution that results in a `CREATE`/`CREATE2` unless the originating address is in the EvmAdmin whitelist — detectable post-execution via the returned `Apply` state diff.
3. **Separate submit and deploy whitelists at the EVM level**: Intercept `CREATE`/`CREATE2` opcodes in the executor and check the EvmAdmin whitelist against `transaction.address` (the origin) before allowing the sub-create to proceed.

### Proof of Concept

```
1. Enable all four whitelists (Admin, EvmAdmin, Account, Address).
2. Add attacker NEAR account to Account whitelist only.
3. Add attacker EVM address to Address whitelist only.
4. Using an Admin/EvmAdmin account, deploy a factory contract:
     contract Factory {
         function deploy(bytes memory code) external returns (address) {
             address child;
             assembly { child := create(0, add(code, 0x20), mload(code)) }
             return child;
         }
     }
5. Attacker submits a CALL transaction: to=factory_address, data=Factory.deploy(maliciousCode).
6. assert_access sees transaction.to.is_some() → calls is_allow_submit → passes (attacker is in Account+Address).
7. EVM executes Factory.deploy → CREATE opcode fires → no whitelist check → malicious contract deployed.
8. Assert: new contract address exists in EVM state with malicious bytecode.
9. Attacker calls malicious contract to drain protocol ETH balances.
```

The invariant "no EVM contract deployment must occur unless the originating NEAR account and EVM address are in Admin and EvmAdmin whitelists respectively" is broken by a single CALL transaction.

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
```

```rust
/// Check if a user has the right to submit transactions.
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}

/// Check if a user has the right to receive erc20 tokens.
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}

fn is_account_allowed_deploy<I: IO + Copy>(io: &I, account_id: &AccountId) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Admin);
    !list.is_enabled() || list.is_exist(account_id)
}

fn is_address_allowed_deploy<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::EvmAdmin);
    !list.is_enabled() || list.is_exist(address)
}
```
