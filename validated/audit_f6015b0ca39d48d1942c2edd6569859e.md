### Title
Cycles Minting Rate Limit Exhausted at Near-Zero Cost via Failed `notify_top_up` - (`File: rs/nns/cmc/src/main.rs`)

---

### Summary

The Cycles Minting Canister (CMC) charges its hourly rate limiter **before** attempting to deposit cycles to the target canister. When the deposit fails (e.g., the target canister does not exist), the ICP is refunded to the caller but the rate-limit quota is permanently consumed. An unprivileged user can repeatedly exploit this to exhaust the 150P/hour global minting cap at the cost of only ledger transfer fees, denying legitimate users the ability to mint cycles for up to one hour per attack window.

---

### Finding Description

`notify_top_up` calls `process_top_up`, which calls `deposit_cycles` with `mint_cycles = true`: [1](#0-0) 

Inside `deposit_cycles`, `ensure_balance` is called **first**, which both charges the rate limiter and physically mints cycles into the CMC's own balance via `ic0_mint_cycles128`: [2](#0-1) 

`ensure_balance` records the minting event in the `Limiter` and increments `total_cycles_minted` before the management-canister call is even attempted: [3](#0-2) 

The `Limiter::check_and_add_cycles` method has no rollback path; once `add` is called the window count is permanently incremented: [4](#0-3) 

If the subsequent `deposit_cycles` management-canister call fails (non-existing canister, stopped canister, etc.), `deposit_cycles` returns `Err`, `process_top_up` issues a full ICP refund to the caller, and the block is marked `NotifiedTopUp(Err(...))`. The rate-limit quota is **not** restored.

The existing integration test `cmc_notify_top_up_invalid` explicitly confirms that 100 T cycles are counted as minted even when the top-up fails and the ICP is refunded: [5](#0-4) 

The test `cmc_notify_top_up_not_rate_limited_by_invalid_top_up` models the attack but only verifies that a small residual (0.96 P) remains after consuming 149.04 P; it does **not** verify that the rate limiter is unaffected by failed top-ups: [6](#0-5) 

The global base limit is 150 P/hour: [7](#0-6) 

---

### Impact Explanation

An attacker who exhausts the 150 P/hour base limit blocks **all** `notify_top_up`, `notify_create_canister`, and `notify_mint_cycles` calls for up to one hour. Canisters that run out of cycles during this window will be frozen or deleted. The attack is repeatable every hour indefinitely.

---

### Likelihood Explanation

`notify_top_up` is a public, permissionless `#[update]` endpoint callable by any principal. No staking, governance role, or privileged key is required. The only cost to the attacker is the ICP ledger transfer fee (~0.0001 ICP per block) plus `TOP_UP_CANISTER_REFUND_FEE`; the principal ICP is returned. A single ICP is sufficient to sustain the attack across many hourly windows. [8](#0-7) 

---

### Recommendation

Move the `ensure_balance` / rate-limiter charge to **after** a successful management-canister `deposit_cycles` call, or verify that the target canister exists and is running before charging the limiter. Alternatively, do not count cycles toward the rate limit when the deposit call fails and the cycles are returned to the CMC's balance.

---

### Proof of Concept

1. Attacker holds 2 ICP (enough to cover fees across many iterations).
2. Attacker transfers 1500 ICP worth of ICP to the CMC subaccount for a non-existing `canister_id` (e.g., `CanisterId::from_u64(999_999_999)`), obtaining `block_index`.
3. Attacker calls `notify_top_up { block_index, canister_id }`.
4. CMC calls `ensure_balance(150_000_000_000_000 cycles)` → rate limiter records 150 P, cycles minted into CMC balance.
5. CMC calls management canister `deposit_cycles` for the non-existing canister → call fails, cycles returned to CMC balance.
6. CMC refunds ~1500 ICP minus fees to attacker.
7. The `base_limiter` now shows 150 P consumed; all subsequent `notify_top_up` / `notify_create_canister` / `notify_mint_cycles` calls for the next hour are rejected with `"More than 150000000000000000 cycles have been minted in the last 3600 seconds, please try again later."`. [9](#0-8)

async fn deposit_cycles(
    canister_id: CanisterId,
    cycles: Cycles,
    mint_cycles: bool,
    limiter_to_use: CyclesMintingLimiterSelector,
) -> Result<(), String> {
    if mint_cycles {
        ensure_balance(cycles, limiter_to_use)?;
    }

    let res: CallResult<()> = ic_cdk::api::call::call_with_payment128(
        candid::Principal::management_canister(),
        METHOD_DEPOSIT_CYCLES,
        (CanisterIdRecord {
            canister_id: canister_id.get().0,
        },),
        u128::from(cycles),
    )
    .await;

    res.map_err(|(code, msg)| {
        format!(
            "Depositing cycles failed with code {}: {:?}",
            code as i32, msg
        )
    })?;

    Ok(())
}

fn ensure_balance(
    cycles: Cycles,
    limiter_to_use: CyclesMintingLimiterSelector,
) -> Result<(), String> {
    let now = now_system_time();

    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    let cycles_to_mint = cycles - current_balance;

    with_state_mut(|state| {
        limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;
        state.total_cycles_minted += cycles_to_mint;
        Ok::<_, String>(())
    })?;

    // unused because of check above
    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())

    pub fn check_and_add_cycles(
        &mut self,
        now: SystemTime,
        cycles_to_mint: Cycles,
        limit: Cycles,
    ) -> Result<(), String> {
        self.purge_old(now);
        let count = self.get_count();

        if count + cycles_to_mint > limit {
            LIMITER_REJECT_COUNT.with(|count| {
                count.set(count.get().saturating_add(1));
            });

            return Err(format!(
                "More than {} cycles have been minted in the last {} seconds, please try again later.",
                limit,
                self.get_max_age().as_secs(),
            ));
        }
        self.add(now, cycles_to_mint);
        Ok(())
    }

#[test]
fn cmc_notify_top_up_not_rate_limited_by_invalid_top_up() {
    let good_account = AccountIdentifier::new(*TEST_USER1_PRINCIPAL, None);
    let good_account_icpts = Tokens::new(10, 0).unwrap();
    let bad_account = AccountIdentifier::new(*TEST_USER2_PRINCIPAL, None);
    let bad_account_icpts = Tokens::new(1, 0).unwrap();

    let state_machine = state_machine_builder_for_nns_tests().build();
    let nns_init_payloads = NnsInitPayloadsBuilder::new()
        .with_test_neurons()
        .with_ledger_accounts(vec![
            (good_account, good_account_icpts),
            (bad_account, bad_account_icpts),
        ])
        .build();
    setup_nns_canisters(&state_machine, nns_init_payloads);
    let non_existing_canister_id = CanisterId::from_u64(123_456_789);

    // First make sure topping up 400T cycles on a valid canister works.
    let cycles = notify_top_up(
        &state_machine,
        GOVERNANCE_CANISTER_ID,
        Tokens::new(4, 0).unwrap(),
    )
    .unwrap();
    assert_eq!(cycles, Cycles::new(400_000_000_000_000_u128));

    // Advance time by 1 hour to make sure the rate limit is reset.
    state_machine.advance_time(Duration::from_secs(60 * 60));

    // Now the attack begins - the bad account sends 0.69 tokens per 5 seconds to the non-existing
    // canister, which makes it 207T * 12 * 60 = 149.04P cycles per hour, close to the 150P limit,
    // while getting the 0.69 tokens back each time (the account only has 1 token in the
    // beginning).
    for _ in 0..(12 * 60) {
        let error = notify_top_up(
            &state_machine,
            non_existing_canister_id,
            Tokens::from_e8s(207_000_000),
        )
        .unwrap_err();
        assert_matches!(error, NotifyError::Refunded { .. });
        state_machine.advance_time(Duration::from_secs(5));
    }

    // Now the good account tries to top up 400T cycles on the governance canister again, which
    // should still succeed.
    let cycles = notify_top_up(
        &state_machine,
        GOVERNANCE_CANISTER_ID,
        Tokens::new(4, 0).unwrap(),
    )
    .unwrap();
    assert_eq!(cycles, Cycles::new(400_000_000_000_000_u128));
}
