Audit Report

## Title
Cross-Operator Rate-Limit Drain via Misattributed Bucket Charge in `do_remove_node_directly_` — (`rs/registry/canister/src/mutations/node_management/do_remove_node_directly.rs`)

## Summary
`do_remove_node_directly_` reserves rate-limit capacity against the **node's registered operator** (`node_operator_id`) rather than the **caller** (`caller_id`). Because the authorization logic intentionally permits a sibling operator (same DC + same node provider) to remove another operator's nodes, a sibling operator can exhaust the victim operator's rate-limit bucket without consuming any of its own. Once the bucket is drained, the victim operator is blocked from both `add_node` and `remove_node_directly` for up to 7 days.

## Finding Description

**Root cause — rate-limit charged to the wrong principal.**

In `do_remove_node_directly_`, the rate-limit reservation is made against `node_operator_id` (the node's owner), not `caller_id`:

```rust
let node_operator_id = get_node_operator_id_for_node(self, payload.node_id)?;
let reservation =
    self.try_reserve_capacity_for_node_operator_operation(now, node_operator_id, 1)?;
``` [1](#0-0) 

The authorization check that determines *who is allowed to call* happens only afterwards, inside `make_remove_or_replace_node_mutations`: [2](#0-1) 

**Authorization explicitly permits sibling operators.**

Inside `make_remove_or_replace_node_mutations`, when `caller_id != node_operator_id`, the code falls back to comparing DC and node-provider IDs. If both match, the removal is allowed: [3](#0-2) 

**Existing test confirms the cross-operator removal path.**

`should_succeed_remove_node_compare_dc_and_node_provider` sets up operator1 and operator2 under the same provider and same DC, then has operator2 successfully remove operator1's node: [4](#0-3) 

**Rate-limit parameters.**

The per-operator bucket has a maximum spike of `NODE_OPERATOR_MAX_SPIKE = 140` operations (20 ops/day × 7 days), shared with the node-provider bucket: [5](#0-4) 

`try_reserve_capacity_for_node_operator_operation` charges both the operator bucket (keyed on the passed `node_operator_id`) and the provider bucket derived from it: [6](#0-5) 

**`add_node` shares the same per-operator bucket.**

`do_add_node_` calls `try_reserve_capacity_for_node_operator_operation(now, caller_id, 1)`, so an exhausted operator bucket also blocks node additions by the victim.

**Existing test confirms exhausted bucket blocks operations.**

`test_do_remove_node_directly_fails_when_rate_limits_exceeded` proves that once the bucket is at zero, `do_remove_node_directly_` returns `"Rate Limit Capacity exceeded. Please wait and try again later."`: [7](#0-6) 

**Combined exploit path:** operator2 (same DC + same provider as operator1) submits up to 140 `remove_node_directly` calls targeting operator1's nodes. Each call passes authorization (sibling check) and charges 1 unit to operator1's bucket. After 140 calls, operator1's bucket is at zero. operator1's subsequent `add_node` and `remove_node_directly` calls all fail with the rate-limit error for up to 7 days.

## Impact Explanation

A sibling node operator can lock out a victim node operator from all `add_node` and `remove_node_directly` operations for up to 7 days. This prevents the victim from re-registering replacement hardware after the attacker has removed their nodes, causing a sustained operational disruption to a legitimate IC node operator. This matches the allowed High impact: **Application/platform-level DoS with concrete user or protocol harm** ($2,000–$10,000).

## Likelihood Explanation

The precondition — two operator records sharing the same DC and node provider — is a normal production configuration (a node provider running multiple operator records in the same data center). The `remove_node_directly` endpoint is callable by any authenticated principal with no governance proposal required. The attack requires only N sequential ingress messages to the registry canister (one per victim node), which is trivially automatable. The attack is repeatable: after the 7-day refill window, the attacker can drain the bucket again.

## Recommendation

Charge the rate-limit cost against the **caller** (`caller_id`), not the node's registered operator. In `do_remove_node_directly_`, replace:

```rust
let node_operator_id = get_node_operator_id_for_node(self, payload.node_id)?;
let reservation =
    self.try_reserve_capacity_for_node_operator_operation(now, node_operator_id, 1)?;
```

with:

```rust
let node_operator_id = get_node_operator_id_for_node(self, payload.node_id)?;
let reservation =
    self.try_reserve_capacity_for_node_operator_operation(now, caller_id, 1)?;
```

This ensures that operator2's actions consume operator2's budget, leaving operator1's budget unaffected. This is consistent with how `do_add_node_` already charges the caller's bucket.

## Proof of Concept

The two existing unit tests, combined, prove the full attack path without any new code:

1. `should_succeed_remove_node_compare_dc_and_node_provider` — proves operator2 (same DC + same provider) can remove operator1's node, and that the rate-limit is charged to operator1's bucket (operator2's `node_allowance` is unchanged, confirming operator1's record is the one modified).
2. `test_do_remove_node_directly_fails_when_rate_limits_exceeded` — proves that once the operator bucket is exhausted, `do_remove_node_directly_` returns the rate-limit error.

A minimal combined test to confirm the cross-operator drain:

```rust
// Setup: operator1 owns 140 nodes; operator2 shares DC and provider
// Attack: operator2 calls do_remove_node_directly_ 140 times
for node_id in operator1_nodes {
    registry.do_remove_node_directly_(
        RemoveNodeDirectlyPayload { node_id },
        operator2_id,
        now,
    ).unwrap(); // each succeeds, charges operator1's bucket
}
// Verify: operator1 is now locked out
let err = registry.do_remove_node_directly_(
    RemoveNodeDirectlyPayload { node_id: any_new_node },
    operator1_id,
    now,
).unwrap_err();
assert_eq!(err, "Rate Limit Capacity exceeded. Please wait and try again later.");
```

    fn test_do_remove_node_directly_fails_when_rate_limits_exceeded() {
        let (mut registry, node_ids, node_operator_id, node_provider_id) =
            setup_registry_for_test();
        let node_id = node_ids[0];

        let now = now_system_time();

        let payload = RemoveNodeDirectlyPayload { node_id };

        // Exhaust the rate limit capacity
        let available_operator =
            registry.get_available_node_operator_op_capacity(node_operator_id, now);
        let available_provider =
            registry.get_available_node_provider_op_capacity(node_provider_id, now);
        let available = available_operator.min(available_provider);
        let reservation = registry
            .try_reserve_capacity_for_node_operator_operation(now, node_operator_id, available)
            .unwrap();
        registry
            .commit_used_capacity_for_node_operator_operation(now, reservation)
            .unwrap();

        let error = registry
            .do_remove_node_directly_(payload, node_operator_id, now)
            .unwrap_err();
        assert_eq!(
            error,
            "Rate Limit Capacity exceeded. Please wait and try again later."
        );
    }

```rust
    pub fn try_reserve_capacity_for_node_operator_operation(
        &self,
        now: SystemTime,
        node_operator_id: PrincipalId,
        requested_capacity: u64,
    ) -> Result<RateLimitReservation, RateLimiterError> {
        // Find the associated node provider ID for this node operator
        let node_provider_id = get_node_provider_id_for_operator_id(self, node_operator_id)
            .map_err(RateLimiterError::InvalidArguments)?;

        // First reserve from node operator rate limiter (primary)
        let operator_reservation = with_node_operator_rate_limiter(|rate_limiter| {
            rate_limiter.try_reserve(now, node_operator_key(node_operator_id), requested_capacity)
        })?;

        // Then reserve from node provider rate limiter (secondary)
        let provider_reservation = with_node_provider_rate_limiter(|rate_limiter| {
            rate_limiter.try_reserve(now, node_provider_key(node_provider_id), requested_capacity)
        })?;

        Ok(RateLimitReservation {
            operator_reservation,
            provider_reservation,
        })
    }
