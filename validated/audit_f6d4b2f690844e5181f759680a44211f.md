L1→L2 Priority Transaction Processing Halts Block if `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` Reverts — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The bootloader's L1→L2 transaction processing calls `notify_l2_asset_tracker` up to three times per deposit transaction (once for the value mint, once for the operator fee, once for the refund). Each call invokes `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` as a live EVM execution. If that contract reverts for any reason, the bootloader escalates the failure to a fatal `InternalError`, explicitly halting block processing. Because L1→L2 priority transactions cannot be skipped or censored, a single revert in the asset tracker bricks the entire chain until the contract is fixed via a governance upgrade.

---

### Finding Description

In `process_l1_transaction`, after the main transaction body executes, the bootloader calls `mint_base_token` (which internally calls `notify_l2_asset_tracker`) up to three times:

1. **Value mint** — inside `execute_l1_transaction_and_notify_result` (line 634)
2. **Operator fee** — at line 290
3. **Refund** — at line 338

Each of these calls `notify_l2_asset_tracker`, which executes `handleFinalizeBaseTokenBridgingOnL2(fromChainId, amount)` on the live `L2AssetTracker` EVM contract via `run_single_interaction`.

The critical path is in `notify_l2_asset_tracker` (lines 901–911):

```rust
if failed {
    // A revert here means the chain's token accounting would be inconsistent.
    // Treated as a fatal system error — block processing cannot continue.
    return Err(internal_error!(
        "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
    )
    .into());
}
```

A revert from the `L2AssetTracker` contract is unconditionally converted into a `BootloaderSubsystemError::LeafDefect` (internal error), which propagates up through `process_l1_transaction` and out of the transaction loop, halting the entire block.

The `l1_chain_id` passed to the call is read directly from storage slot 154 of the `L2AssetTracker` contract at `read_l1_chain_id`. If the contract is deployed but not yet initialized (slot 154 = 0), the call is made with `fromChainId = 0`, which may itself cause a revert inside the contract's validation logic. ### Impact Explanation

**Chain halt.** L1→L2 priority transactions are drawn from a priority queue enforced on L1. The bootloader cannot skip or reorder them. If any such transaction triggers a revert in `handleFinalizeBaseTokenBridgingOnL2`, the fatal error propagates out of the transaction loop and block processing stops. No further blocks can be produced until the `L2AssetTracker` contract is repaired via a governance upgrade transaction — which itself is an L1→L2 transaction that would also call `notify_l2_asset_tracker`, potentially creating a deadlock.

The code comment at line 906–907 explicitly acknowledges this: *"Treated as a fatal system error — block processing cannot continue."* [5](#0-4) 

---

### Likelihood Explanation

The `L2AssetTracker` is a real EVM contract (not a system hook) executed through the full EVM interpreter. Conditions under which `handleFinalizeBaseTokenBridgingOnL2` can revert include:

- The contract is deployed but `L1_CHAIN_ID` (slot 154) is zero or uninitialized, causing the call to be made with `fromChainId = 0`, which the contract may reject.
- The base token asset is not yet registered in the contract's `isAssetRegistered` mapping (slot 153), causing an access-control revert.
- An incomplete or partially-applied upgrade leaves the contract in an inconsistent state.
- Any future bug introduced in the contract during an upgrade.

Any L1→L2 transaction with `total_deposited > 0` (i.e., any deposit) triggers this path. An attacker who can submit an L1→L2 transaction during a window when the `L2AssetTracker` is in a revert-prone state can halt the chain. [6](#0-5) 

---

### Recommendation

1. **Decouple the asset-tracker notification from the fatal error path.** If `handleFinalizeBaseTokenBridgingOnL2` reverts, log the failure and emit a system log, but do not convert it into a fatal `InternalError`. Instead, continue block processing and flag the accounting discrepancy for off-chain reconciliation or a subsequent governance action.

2. **Alternatively**, introduce a separate, skippable notification path analogous to the Telcoin recommendation: provide a fallback `process_l1_transaction` variant that skips the asset-tracker notification when the contract is known to be in a broken state, allowing the priority queue to drain.

3. **Guard the `l1_chain_id` read**: if slot 154 returns zero, skip the `notify_l2_asset_tracker` call entirely rather than passing `fromChainId = 0` to the contract.

---

### Proof of Concept

**Setup**: Deploy `L2AssetTracker` at the expected address but leave `L1_CHAIN_ID` (slot 154) uninitialized (= 0), or deploy a version whose `handleFinalizeBaseTokenBridgingOnL2` reverts when called with `fromChainId = 0` or when the base token asset is not registered.

**Attack**:
1. Submit an L1→L2 priority transaction with `total_deposited > 0` (any standard deposit).
2. The bootloader calls `read_l1_chain_id`, which reads slot 154 and returns `0`.
3. `notify_l2_asset_tracker` is called with `amount = to_transfer` and `l1_chain_id = 0`.
4. `handleFinalizeBaseTokenBridgingOnL2(0, amount)` reverts inside the `L2AssetTracker`.
5. `notify_l2_asset_tracker` returns `Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"))`.
6. This propagates as a `BootloaderSubsystemError::LeafDefect` out of `process_l1_transaction`.
7. The transaction loop receives a fatal error and block processing halts.
8. All subsequent L1→L2 transactions in the priority queue are also blocked, as they follow the same code path. [7](#0-6) [8](#0-7)

```rust
fn read_l1_chain_id<S: EthereumLikeTypes>(system: &mut System<S>) -> U256
where
    S::IO: IOSubsystemExt,
{
    // L2AssetTracker storage layout (verified via `forge inspect`):
    //   slots 0-100:   Initializable + OwnableUpgradeable + Ownable2StepUpgradeable
    //   slots 101-150: Ownable2Step __gap
    //   slot 151:      mapping chainBalance
    //   slot 152:      mapping assetMigrationNumber
    //   slot 153:      mapping isAssetRegistered
    //   slot 154:      uint256 L1_CHAIN_ID
    let l1_chain_id_slot = Bytes32::from_u256_be(&U256::from(154));
    let mut inf_resources = S::Resources::FORMAL_INFINITE;
    let chain_id = system
        .io
        .storage_read::<false>(
            ExecutionEnvironmentType::NoEE,
            &mut inf_resources,
            &L2_ASSET_TRACKER_ADDRESS,
            &l1_chain_id_slot,
        )
        .expect("must read L2AssetTracker L1_CHAIN_ID");
    U256::from_be_bytes(chain_id.as_u8_array())
```
