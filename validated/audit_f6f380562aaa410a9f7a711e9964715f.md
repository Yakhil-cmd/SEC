Audit Report

## Title
Voting Rewards Permanently Lost When No Neurons Vote During a Reward Period With Settled Proposals - (`rs/sns/governance/src/governance.rs`, `rs/nns/governance/src/governance.rs`)

## Summary

In both SNS and NNS governance, when `total_reward_shares == 0` (SNS) or `total_voting_rights < 0.001` (NNS) during a reward period that contains `ReadyToSettle` proposals, the entire reward purse for that period is permanently destroyed. The zero-voting-power guard skips maturity distribution but does not prevent proposal settlement; once proposals are settled, `rewards_rolled_over()` returns `false` and `e8s_equivalent_to_be_rolled_over()` returns `0`, silently discarding the accumulated reward budget with no recovery path.

## Finding Description

**SNS root cause (`rs/sns/governance/src/governance.rs`):**

`distribute_rewards` accumulates `total_reward_shares` by summing `ballot.voting_power` for every neuron that cast an eligible vote across all `ReadyToSettle` proposals. [1](#0-0) 

When `total_reward_shares == dec!(0)`, the function logs a warning and skips the maturity-increment loop. Critically, execution falls through unconditionally into the proposal-settlement loop: [2](#0-1) 

The settlement loop at L6013–6081 runs regardless of `total_reward_shares`, setting `reward_event_end_timestamp_seconds`, `reward_event_round`, and clearing ballots on every proposal in `considered_proposals`. [3](#0-2) 

The resulting `RewardEvent` is written with a non-empty `settled_proposals` list and `distributed_e8s_equivalent = 0`: [4](#0-3) 

**SNS rollover gate (`rs/sns/governance/src/types.rs`):**

`rewards_rolled_over()` returns `true` only when `settled_proposals.is_empty()`. Because proposals were just settled above, it returns `false`. Consequently, `e8s_equivalent_to_be_rolled_over()` returns `0`: [5](#0-4) 

On the next reward round, `rewards_purse_e8s` is initialized from `e8s_equivalent_to_be_rolled_over()` of the previous event, which is `0`. The entire reward purse from the zero-voter period is silently discarded: [6](#0-5) 

**NNS root cause (`rs/nns/governance/src/governance.rs`):**

The identical structural flaw exists in `calculate_voting_rewards`. When `total_voting_rights < 0.001`, `reward_distribution` is set to `None` (no maturity incremented), but `settled_proposals: considered_proposals` is still populated in the returned `RewardEvent`: [7](#0-6) 

The NNS rollover gate is identical in logic — `rewards_rolled_over()` checks `settled_proposals.is_empty()`: [8](#0-7) 

The NNS code explicitly acknowledges the unresolved risk in the guard comment: *"Not sure if that is theoretically possible, but even if it isn't, it might occur due to some bug."* [9](#0-8) 

## Impact Explanation

Any SNS token maturity rewards calculated for a reward round in which no neuron cast an eligible vote are permanently destroyed. The `rewards_purse_e8s` computed for that round — proportional to the total SNS token supply and the configured reward rate — is never added to any neuron's `maturity_e8s_equivalent` or `staked_maturity_e8s_equivalent`, and is never carried forward. Neuron holders lose maturity they are entitled to, with no recovery path. Because maturity is the precursor to minting new governance tokens, this constitutes a ledger conservation violation: the protocol promises a reward rate but silently fails to honour it. This matches the allowed impact: **High — Significant SNS/NNS security impact with concrete user or protocol harm.**

## Likelihood Explanation

**SNS:** Realistic for small or declining SNS DAOs. If all neurons have dissolved (dissolve delay expired, voting power = 0), their ballots carry `voting_power = 0`, so `total_reward_shares` sums to zero even if every neuron "voted." Alternatively, if no neuron voted (directly or via following) on any proposal that reached `ReadyToSettle`, the same condition is triggered. An unprivileged actor who holds a neuron can submit a proposal; if the SNS community is inactive, the reward purse for that round is lost. No privileged access is required.

**NNS:** The guard comment concedes the condition "might occur due to some bug." A bug in `sum_weighted_voting_power` or in vote propagation that zeroes out `total_voting_rights` while leaving `considered_proposals` non-empty would silently destroy the day's reward pot. Likelihood is low under normal operation but non-zero under fault conditions.

## Recommendation

When `total_reward_shares == dec!(0)` (SNS) or `total_voting_rights < 0.001` (NNS) and there are proposals to settle, the reward purse must not be silently discarded. Two complementary fixes:

1. **Do not settle proposals when no rewards can be distributed.** Skip the settlement loop when `total_reward_shares == 0`, leaving the proposals in `ReadyToSettle` so they are reconsidered in the next round.

2. **Alternatively, decouple rollover eligibility from proposal settlement.** Change `rewards_rolled_over()` (and thus `e8s_equivalent_to_be_rolled_over()`) to return `true` whenever `distributed_e8s_equivalent == 0`, regardless of whether `settled_proposals` is empty. This ensures the purse is always carried forward when nothing was actually paid out.

## Proof of Concept

**SNS scenario (all neurons dissolved, voting power = 0):**

1. Deploy an SNS with `voting_rewards_parameters` configured (non-zero reward rate).
2. Create a proposal via a neuron before its dissolve delay expires.
3. Allow all neurons to dissolve (dissolve delay reaches 0, voting power = 0 in all ballots).
4. Advance time past one reward round so `distribute_rewards` is called with the proposal in `ReadyToSettle`.
5. Observe: `total_reward_shares == dec!(0)` → the `if` branch logs a warning and skips maturity increments.
6. Observe: the proposal's `reward_event_end_timestamp_seconds` is set and ballots are cleared — it is permanently settled.
7. Advance time past a second reward round.
8. Observe: `rolling_over_from_previous_reward_event_e8s_equivalent` is `0` because `e8s_equivalent_to_be_rolled_over()` returned `0` (`settled_proposals` was non-empty).
9. The reward purse from step 4 is gone; no neuron ever received it.

This can be reproduced as a deterministic unit test using the existing `distribute_rewards` test harness in `rs/sns/governance/src/governance/assorted_governance_tests.rs`, by constructing a scenario where all neuron ballots carry `voting_power = 0` and asserting that `total_available_e8s_equivalent` in the subsequent `RewardEvent` does not include the prior round's purse.
