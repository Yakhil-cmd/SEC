### Title
Missing Block Timestamp Monotonicity Check in `PectraForkHeader::verify_chain` - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/block_header.rs`)

---

### Summary

`PectraForkHeader::verify_chain` validates cross-block invariants (EIP-1559 base fee, EIP-4844 excess blob gas) against the parent block header but omits the Ethereum consensus requirement that `block.timestamp > parent.timestamp`. A prover/operator supplying oracle data can submit a block with a non-monotonically-increasing timestamp that passes all ZKsync OS validation and is proven valid.

---

### Finding Description

The `ChainChecker` implementation for `PectraForkHeader` in `verify_chain` fetches the parent block header from the oracle at `depth == 0` and performs two cross-block invariant checks:

1. **EIP-1559**: verifies `base_fee_per_gas` is correctly derived from parent gas usage.
2. **EIP-4844**: verifies `excess_blob_gas` is correctly derived from parent blob gas. [1](#0-0) 

The parent block's `timestamp` field is parsed and available as `historical_header.timestamp`, but **no assertion is made that `self.timestamp > historical_header.timestamp`**. This is a mandatory Ethereum consensus rule (Yellow Paper §4.3.4: `Hs > Hp.Hs`). The check is structurally absent — the code comment even says *"we should check cross-blocks pricing invariants"* but only lists EIP-1559 and EIP-4844, omitting the timestamp invariant entirely. [2](#0-1) 

For comparison, the ZKsync-native proving path does enforce a weaker non-decreasing check (`>=`) against its own chain state commitment's `last_block_timestamp`: [3](#0-2) 

But the Ethereum STF path has no equivalent check at all.

---

### Impact Explanation

**Type**: EVM semantic mismatch / state-transition bug.

An accepted block with `timestamp <= parent.timestamp` violates Ethereum consensus. Any smart contract relying on `block.timestamp` for time-based logic — time-locks, auctions, vesting schedules, deadline enforcement (`require(block.timestamp > deadline)`) — can be manipulated. A prover can replay a block with a past or equal timestamp, causing time-sensitive contract logic to behave incorrectly. Because the proof is generated over the invalid state transition, the settlement layer accepts it as valid, making the manipulation permanent and irreversible.

---

### Likelihood Explanation

The oracle provides block header bytes, and the prover controls oracle inputs. The `verify_chain` function is the sole enforcement point for cross-block header invariants in the Ethereum STF path. [4](#0-3) 

Since the STF does not enforce timestamp monotonicity, a prover supplying crafted oracle data (a valid-hash-chain block with a manipulated timestamp) will pass all existing checks. The entry path is the prover/forward execution input, which is explicitly in scope.

---

### Recommendation

Inside the `depth == 0` branch of `PectraForkHeader::verify_chain`, add a strict timestamp monotonicity assertion alongside the existing EIP-1559 and EIP-4844 checks:

```rust
// Timestamp must be strictly greater than parent (Ethereum Yellow Paper §4.3.4)
assert!(self.timestamp > historical_header.timestamp,
    "block timestamp must be strictly greater than parent timestamp");
``` [1](#0-0) 

---

### Proof of Concept

1. Prover constructs a valid Ethereum block header where `timestamp == parent.timestamp` (or any value ≤ parent timestamp), with all other fields (parent hash, number, EIP-1559 base fee, EIP-4844 excess blob gas) correctly derived so existing assertions pass.
2. Prover feeds this header via the oracle (`ETHEREUM_TARGET_HEADER_BUFFER_*` query IDs).
3. `verify_chain` is called: it verifies the hash chain, block number, EIP-1559 base fee, and EIP-4844 excess blob gas — all pass.
4. No timestamp check exists; the block is accepted.
5. The proving run completes and produces a valid proof over a state transition where `block.timestamp` did not advance.
6. Any contract in that block that checks `block.timestamp > some_past_deadline` behaves as if time has not moved, enabling bypass of time-based guards. [5](#0-4)
