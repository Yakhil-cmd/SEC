### Title
Pre-EIP-155 Legacy Transactions Bypass Chain ID Validation, Enabling Cross-Chain Replay Attacks - (`engine/src/engine.rs`)

---

### Summary

The `submit_with_alt_modexp` function in `engine/src/engine.rs` validates the chain ID only when it is present in the transaction signature (`if let Some(chain_id)`). Pre-EIP-155 legacy transactions (signed with `v = 27` or `v = 28`) produce `chain_id: None` and bypass this check entirely. Because Aurora Engine is deployed on multiple networks with distinct chain IDs (mainnet `1313161554`, testnet `1313161555`, local `1313161556`), any pre-EIP-155 transaction accepted on one deployment can be replayed verbatim on any other, draining the victim's funds on the second chain.

---

### Finding Description

**Root cause — conditional chain ID check:**

In `submit_with_alt_modexp`, the chain ID guard is:

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [1](#0-0) 

The guard fires only when `transaction.chain_id` is `Some(...)`. When it is `None`, the block is skipped and execution continues unconditionally.

**How `chain_id` becomes `None` — legacy transaction parsing:**

`LegacyEthSignedTransaction::chain_id()` returns `None` for any transaction whose `v` field is 27 or 28 (the pre-EIP-155 encoding):

```rust
pub const fn chain_id(&self) -> Option<u64> {
    match self.v {
        0..=34 => None,
        _ => Some((self.v - 35) / 2),
    }
}
``` [2](#0-1) 

`NormalizedEthTransaction` propagates this `None` directly:

```rust
Legacy(tx) => Self {
    address: tx.sender()?,
    chain_id: tx.chain_id(),   // None for pre-EIP-155
    ...
``` [3](#0-2) 

The engine's own comment in `lib.rs` acknowledges the intent to prevent cross-chain replay:

> "Must match `CHAIN_ID` to make sure it's signed for given chain vs replayed from another chain." [4](#0-3) 

Yet the implementation leaves a gap: pre-EIP-155 transactions are accepted without any chain-binding check.

**Multiple Aurora deployments share the same address space:**

| Network | Chain ID |
|---|---|
| Mainnet | 1313161554 |
| Testnet | 1313161555 |
| Local | 1313161556 |

A user's NEAR-derived EVM address is identical across all deployments. Nonces start at 0 on each chain independently. A pre-EIP-155 transaction with nonce N that executes on mainnet will also execute on testnet if the victim's nonce there is also N.

---

### Impact Explanation

An attacker who observes a pre-EIP-155 transaction on one Aurora deployment can submit the identical RLP-encoded bytes to any other Aurora deployment via the public `submit` NEAR method. The engine will:

1. Parse the transaction successfully (v = 27/28 is valid).
2. Recover the correct sender via `ecrecover` (the signature is valid on all chains).
3. Find `chain_id: None`, skip the chain ID guard.
4. Execute the transaction — transferring ETH, calling a contract, or deploying code — on the victim's account on the second chain.

This constitutes **direct theft of user funds** (Critical impact).

---

### Likelihood Explanation

- The `submit` entry point is publicly callable by any NEAR account — no privilege is required.
- Pre-EIP-155 transactions are still produced by some tooling, scripts, and older wallets.
- Aurora Engine explicitly handles `v = 27/28` without rejecting them, so the attack surface is intentionally open.
- The attacker only needs to observe a single on-chain transaction; no key material or off-chain coordination is required.

---

### Recommendation

Reject pre-EIP-155 legacy transactions outright by treating an absent chain ID as an error, making the chain ID check unconditional:

```rust
// Require chain ID to be present and match this deployment.
let chain_id = transaction.chain_id
    .ok_or(EngineErrorKind::InvalidChainId)?;
if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

This eliminates the replay surface entirely. If backward compatibility with pre-EIP-155 transactions is required, a strict allowlist of accepted chain IDs (including `0` as a sentinel) should be enforced.

---

### Proof of Concept

1. **Victim** signs a legacy ETH transfer on Aurora mainnet using a tool that omits EIP-155 (`v = 27` or `v = 28`). The transaction is submitted and mined.
2. **Attacker** reads the raw RLP bytes of the victim's transaction from the NEAR blockchain.
3. **Attacker** calls `submit` on the Aurora **testnet** contract, passing the identical bytes.
4. The testnet engine parses the transaction, recovers the same sender address, finds `chain_id: None`, skips the chain ID guard at `engine/src/engine.rs:1055`, and executes the transfer.
5. The victim's ETH balance on testnet is drained to the same recipient, with no action required from the victim.

The same attack applies in the reverse direction (testnet → mainnet) and to any other Aurora Engine deployment sharing the same codebase. [1](#0-0) [5](#0-4) [6](#0-5)

impl LegacyEthSignedTransaction {
    /// Returns sender of given signed transaction by doing ecrecover on the signature.
    pub fn sender(&self) -> Result<Address, Error> {
        let mut rlp_stream = RlpStream::new();
        // See details of CHAIN_ID computation here - https://github.com/ethereum/EIPs/blob/master/EIPS/eip-155.md#specification
        let (chain_id, rec_id) = match self.v {
            0..=26 | 29..=34 => return Err(Error::InvalidV),
            27..=28 => (
                None,
                u8::try_from(self.v - 27).map_err(|_e| Error::InvalidV)?,
            ),
            _ => (
                Some((self.v - 35) / 2),
                u8::try_from((self.v - 35) % 2).map_err(|_e| Error::InvalidV)?,
            ),
        };
        self.transaction
            .rlp_append_unsigned(&mut rlp_stream, chain_id);
        let message_hash = sdk::keccak(rlp_stream.as_raw());
        sdk::ecrecover(message_hash, &super::vrs_to_arr(rec_id, self.r, self.s))
            .map_err(|_| Error::EcRecover)
    }

    /// Returns chain id encoded in `v` parameter of the signature if that was done, otherwise None.
