Audit Report

## Title
Unbounded DKG Unvalidated Pool Growth via Remote-Tagged Artifact Flooding — (`rs/artifact_pool/src/dkg_pool.rs`, `rs/consensus/dkg/src/lib.rs`, `rs/replica/setup_ic_network/src/lib.rs`)

## Summary
A Byzantine subnet member below the fault threshold can flood the DKG unvalidated pool with arbitrarily many distinct artifacts by advertising them over the authenticated P2P channel using distinct slot numbers. Because the DKG broadcast channel is registered with `SLOT_TABLE_NO_LIMIT` (`usize::MAX`), the slot table imposes no per-peer cap; the `DkgBouncer` accepts every message at `current_start_height`; `DkgPoolImpl::insert` has no admission control; and `validate_dealings_for_dealer` silently defers every remote-tagged message whose `NiDkgId` has no matching config, leaving it in the unvalidated pool until the next DKG interval boundary (~500 blocks). The result is unbounded heap growth and OOM crash of honest replicas.

## Finding Description

**Step 1 — No slot table limit for DKG.**
The DKG broadcast channel is registered with `SLOT_TABLE_NO_LIMIT = usize::MAX`: [1](#0-0) [8](#0-7) 

The slot table enforces a per-peer cap only when `peer_slot_table_len >= self.slot_limit`: [2](#0-1) 

With `slot_limit = usize::MAX`, the `Entry::Vacant(_)` drop branch is never reached for DKG, so a Byzantine peer can advertise an unbounded number of distinct slot numbers simultaneously.

**Step 2 — `DkgBouncer` accepts everything at `current_start_height`.**
The bouncer checks only the height field of `DkgMessageId`, not content, signer, or `target_subnet`: [3](#0-2) 

Any message with `id.height == start_height` returns `BouncerValue::Wants`, so all N crafted messages are fetched and forwarded to the pool.

**Step 3 — `DkgPoolImpl::insert` has no admission control.**
The insert path is an unconditional `HashMap` insert with no size check, no per-peer quota, and no content validation: [4](#0-3) 

The key is `DkgMessageId = (crypto_hash(msg), start_block_height)`: [7](#0-6) 

Each message with distinct content (different `NiDkgTargetId`, different dealing bytes, or different signature bytes) produces a distinct key, so N distinct messages produce N distinct pool entries.

**Step 4 — `validate_dealings_for_dealer` silently defers remote-tagged messages with unknown configs.**
When `on_state_change` processes the unvalidated pool, it groups messages by `(signer, dkg_id)` and calls `validate_dealings_for_dealer` for each group: [9](#0-8) 

Inside `validate_dealings_for_dealer`, if the message's `dkg_id` is not in the current configs map AND `target_subnet.is_remote()`, the function returns `Mutations::new()` — an empty change set — leaving the artifact in the unvalidated pool with no removal action: [5](#0-4) 

**Step 5 — Purge only fires at DKG interval boundaries.**
The purge action is only emitted when `start_height > dkg_pool.get_current_start_height()`, i.e., when a new DKG summary block is finalized: [6](#0-5) 

The purge removes entries with `id.height < height`: [10](#0-9) 

Messages at `current_start_height` survive until the next interval. DKG intervals are typically 500 blocks (~8 minutes). Flooded messages accumulate for the entire interval.

## Impact Explanation
Each `dkg::Message` contains an `NiDkgDealing` with a large cryptographic payload. With N=10,000 messages at ~100 KB each, the unvalidated pool grows by ~1 GB. With N=100,000 messages, it reaches ~10 GB. The replica process is killed by the OS OOM killer, halting consensus on the affected node. If enough honest nodes are targeted simultaneously (still below the Byzantine fault threshold), subnet liveness is violated.

This matches the allowed ICP bounty impact: **High ($2,000–$10,000) — Application/platform-level DoS, crash, consensus blocking, or subnet availability impact not based on raw volumetric DDoS.**

## Likelihood Explanation
The attacker must be a subnet member with a valid TLS node certificate — a Byzantine node below the fault threshold, which is explicitly within the IC threat model. No governance majority, no key compromise, and no external infrastructure attack is required. The attack is fully local: craft messages with distinct `NiDkgTargetId` values, send them over the authenticated P2P channel using distinct slot numbers, and wait for the victim's heap to exhaust. The attack is repeatable every DKG interval.

## Recommendation
1. **Add a slot table limit for DKG.** Replace `SLOT_TABLE_NO_LIMIT` with a small constant (e.g., `~2 * subnet_size`) for the DKG broadcast channel, analogous to `SLOT_TABLE_LIMIT_INGRESS` for ingress.
2. **Add admission control in `DkgPoolImpl::insert`.** Enforce a maximum unvalidated pool size (e.g., `MAX_DEALERS * MAX_DKG_CONFIGS_PER_INTERVAL`) and drop artifacts that exceed it.
3. **Eagerly reject remote-tagged messages with unrecognized `dealer_subnet`.** In `validate_dealings_for_dealer`, if `target_subnet.is_remote()` and the `dealer_subnet` does not match the local subnet ID, emit `HandleInvalid` rather than `Mutations::new()`.
4. **Bound the deferred remote DKG set.** Track the number of deferred remote messages per `NiDkgTargetId` and cap it at the maximum number of legitimate dealers.

## Proof of Concept
```rust
// Byzantine node: craft N messages with distinct NiDkgTargetId values
for i in 0..N {
    let target_id = NiDkgTargetId::new([i as u8; 32]); // distinct per iteration
    let dkg_id = NiDkgId {
        start_block_height: current_start_height, // accepted by bouncer
        dealer_subnet: subnet_test_id(999),       // arbitrary
        dkg_tag: NiDkgTag::HighThreshold,
        target_subnet: NiDkgTargetSubnet::Remote(target_id), // triggers deferral
    };
    let msg = craft_dkg_message(dkg_id); // arbitrary dealing bytes, arbitrary signature
    send_via_p2p(victim_node, slot_number=i, msg);
}
// Each message: bouncer returns Wants (height matches), slot table has no limit,
// insert() adds to unvalidated pool, validate_dealings_for_dealer returns Mutations::new().
// After N iterations: pool.get_unvalidated().count() == N, heap grows unboundedly.
// No purge until next DKG summary block (~500 blocks later).
```

A deterministic integration test can be written using `DkgPoolImpl` directly: insert N messages with distinct `NiDkgTargetId` values at `current_start_height`, call `on_state_change` with a config map that does not contain any of the remote `NiDkgId`s, and assert that `pool.get_unvalidated().count() == N` after the call — confirming zero removals.
