### Title
Silent `u16` Truncation of `tx_number_in_block` in L2→L1 Log Encoding Produces Incorrect Pubdata — (File: `zk_ee/src/common_structs/logs_storage.rs`)

---

### Summary

`L2ToL1Log.tx_number_in_block` is declared as `u16`, capping it at 65,535. The conversion from `GenericLogContent` (which stores `tx_number: u32`) uses a bare Rust `as u16` cast — a silent, non-panicking truncation. If a block ever contains more than 65,535 transactions, every L2→L1 log emitted by transactions beyond that threshold will carry a silently wrong `tx_number_in_block`, corrupting the log hash, the Merkle tree root, and the pubdata committed to L1.

---

### Finding Description

In `zk_ee/src/common_structs/logs_storage.rs`, the on-wire log structure is:

```rust
pub struct L2ToL1Log {
    pub l2_shard_id: u8,
    pub is_service: bool,
    pub tx_number_in_block: u16,   // ← only 16 bits; max 65 535
    pub sender: B160,
    pub key: Bytes32,
    pub value: Bytes32,
}
``` [1](#0-0) 

The in-memory log content stores the transaction number as `u32`:

```rust
pub struct GenericLogContent<IOTypes: SystemIOTypesConfig, A: Allocator = Global> {
    pub tx_number: u32,   // ← 32 bits
    ...
}
``` [2](#0-1) 

The `From<&LogContent<A>> for L2ToL1Log` conversion narrows the value with a bare `as u16` cast:

```rust
tx_number_in_block: m.tx_number as u16,
``` [3](#0-2) 

Rust's `as` cast on integers is defined to truncate the high bits silently — it never panics and never returns an error. Transaction 65,536 would produce `tx_number_in_block = 0`, transaction 65,537 would produce `1`, and so on, silently aliasing with earlier transactions.

The corrupted field is then serialised byte-for-byte into the 88-byte log encoding:

```rust
buffer[2..4].copy_from_slice(&self.tx_number_in_block.to_be_bytes());
``` [4](#0-3) 

and fed into the keccak hash that becomes the Merkle-tree leaf:

```rust
fn hash(&self) -> Bytes32 {
    let mut hasher = crypto::sha3::Keccak256::new();
    self.write_encoding(&mut hasher);
    hasher.finalize().into()
}
``` [5](#0-4) 

The Merkle root over all leaves is the value committed to L1 as part of the block's pubdata output. [6](#0-5) 

---

### Impact Explanation

Any block that contains more than 65,535 transactions will produce L2→L1 log entries with a silently wrong `tx_number_in_block`. Because this field is hashed into each Merkle-tree leaf, the resulting tree root committed to L1 will be incorrect. L1 contracts that rely on Merkle inclusion proofs to verify L2 transaction execution results (e.g., to release funds or confirm L1→L2 message delivery) will receive a root that does not match the actual execution, breaking cross-layer finality guarantees. The corruption is undetectable at the ZKsync OS level because no bounds check or error is raised.

---

### Likelihood Explanation

Reaching 65,535 transactions in a single block is constrained today by gas and native-resource limits. However:

1. The type mismatch is a latent defect that grows more reachable as throughput targets increase (the stated goal is 10,000 TPS).
2. An adversary who can cheaply fill blocks with minimal-cost transactions (e.g., simple ETH transfers) could deliberately push a block past the threshold.
3. The silent truncation means the system gives no warning as the limit is approached; the first observable symptom is a wrong Merkle root on L1.

---

### Recommendation

Change `tx_number_in_block` in `L2ToL1Log` from `u16` to `u32` to match the source type in `GenericLogContent`. Update the 88-byte serialisation layout accordingly, or add an explicit checked conversion (`u16::try_from(m.tx_number).expect(...)`) that panics loudly rather than silently corrupting data.

---

### Proof of Concept

1. Construct a block containing 65,537 minimal transactions, each emitting one L2→L1 log.
2. Transaction index 65,536 (0-based) has `tx_number = 65536 = 0x10000`; the `as u16` cast yields `0x0000`.
3. Transaction index 65,537 has `tx_number = 65537 = 0x10001`; the cast yields `0x0001` — identical to transaction 1.
4. The keccak hash of the log for transaction 65,536 is therefore identical to the hash of a log from transaction 0 with the same `sender`/`key`/`value`, producing a wrong Merkle leaf.
5. The Merkle root committed to L1 diverges from the root that would be computed from the true transaction numbers, breaking any L1-side Merkle proof that references those logs.
