Audit Report

## Title
Global Pending-Queue Saturation Griefing Blocks All ckETH and ckERC20 Withdrawals - (File: `rs/ethereum/cketh/minter/src/guard/mod.rs`)

## Summary
The ckETH minter enforces a single global `MAX_PENDING = 100` limit on pending withdrawal requests shared across all principals and all token types. An unprivileged attacker can make 100 sequential `withdraw_eth` calls to saturate `pending_withdrawal_requests`, causing every subsequent `withdraw_eth` and `withdraw_erc20` call from any principal to fail with `TooManyPendingRequests`. Because the attacker's principal is removed from `guarded_principals` on each guard drop while the withdrawal request persists in the queue, sequential calls from the same principal are not blocked, enabling indefinite queue monopolization at the cost of only Ethereum gas fees.

## Finding Description
`Guard::new` in `rs/ethereum/cketh/minter/src/guard/mod.rs` (L50–68) checks `PR::pending_requests_count(s) >= MAX_PENDING` as a global gate before allowing any new withdrawal. `PendingWithdrawalRequests::pending_requests_count` (L32–34) returns `state.eth_transactions.withdrawal_requests_len()`, which is the raw length of the shared `pending_withdrawal_requests` deque (L929–931 of `transactions/mod.rs`) — a global count, not per-principal.

The `Drop` impl (L71–75 of `guard/mod.rs`) removes the principal from `guarded_principals` when the guard is dropped at function return, but the `WithdrawalRequest` appended via `record_withdrawal_request` (L453–466 of `transactions/mod.rs`) remains in `pending_withdrawal_requests` until the minter's timer task processes it. The `AlreadyProcessing` check (L56–58) only prevents concurrent in-flight calls from the same principal; it does not prevent sequential calls that each leave a request in the queue.

Both `withdraw_eth` (L274 of `main.rs`) and `withdraw_erc20` (L401 of `main.rs`) call `retrieve_withdraw_guard(caller)`, which resolves to `Guard::<PendingWithdrawalRequests>::new`, sharing the same 100-slot global queue.

**Exploit flow:**
1. Attacker calls `withdraw_eth` 100 times sequentially with the minimum withdrawal amount (0.03 ETH each).
2. Each call: guard acquired → principal inserted into `pending_withdrawal_principals` → ckETH burned → request pushed to `pending_withdrawal_requests` → guard dropped → principal removed from `pending_withdrawal_principals`.
3. After 100 calls, `pending_withdrawal_requests.len() == 100`.
4. Every subsequent `withdraw_eth` or `withdraw_erc20` call from any principal hits `pending_requests_count >= MAX_PENDING` and is rejected.
5. Attacker re-fills the queue as the minter drains it, sustaining the DoS indefinitely.

## Impact Explanation
All ckETH and ckERC20 withdrawals are blocked for every user on the minter until the attacker's queued requests are processed. Because the attacker can continuously re-submit after each batch drains, the disruption can be sustained indefinitely. This matches the **High ($2,000–$10,000)** impact: "Application/platform-level DoS… or subnet availability impact not based on raw volumetric DDoS" and "Significant Chain Fusion, ck-token… security impact with concrete user or protocol harm." Users holding ckETH or ckERC20 tokens cannot convert them back to native assets, breaking the chain-fusion bridge's core redeemability guarantee.

## Likelihood Explanation
The attack requires ~3 ETH of capital (fully recoverable minus gas fees) and 100 sequential canister calls — both trivially achievable by any motivated adversary. No privileged role, governance majority, or key compromise is needed. The entry path is the public `withdraw_eth` update method, reachable by any non-anonymous ingress sender. The net cost (gas fees only, roughly $500–$5,000 at typical gas prices) is low relative to the disruption caused to all minter users.

## Recommendation
1. **Add a per-principal pending-request cap** alongside the global cap. Reject a new request if the calling principal already has `N` (e.g., 5) requests in `pending_withdrawal_requests`, preventing a single principal from monopolizing the queue.
2. **Separate ckETH and ckERC20 pending queues** so that flooding one token type does not block the other.
3. **Raise `MAX_PENDING`** to a value that reflects realistic throughput, reducing the feasibility of saturation.

## Proof of Concept
```rust
// Minimal integration test using PocketIC or local replica:
// 1. Deploy ckETH minter canister.
// 2. Fund attacker principal with 3+ ETH worth of ckETH.
// 3. Execute 100 sequential update calls:
for i in 0..100 {
    minter.withdraw_eth(WithdrawalArg {
        amount: 30_000_000_000_000_000u64.into(), // minimum
        recipient: attacker_eth_address.clone(),
        from_subaccount: None,
    }).await.expect("call should succeed");
}
// 4. Assert pending_withdrawal_requests.len() == 100.
// 5. Attempt withdrawal from a victim principal:
let result = minter_as_victim.withdraw_eth(WithdrawalArg {
    amount: 1_000_000_000_000_000_000u64.into(),
    recipient: victim_eth_address.clone(),
    from_subaccount: None,
}).await;
// 6. Assert result is a trap with TooManyPendingRequests — confirming DoS.
```
The test is deterministic, requires no mainnet interaction, and directly exercises the `Guard::new` path in `rs/ethereum/cketh/minter/src/guard/mod.rs` L52–53.
