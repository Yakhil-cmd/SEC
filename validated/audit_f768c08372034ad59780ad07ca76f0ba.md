Audit Report

## Title
Missing Globals Length/Type Validation in `update_execution_state` Allows Compromised Sandbox to Corrupt Replicated State and Trigger Replica Crash Loop — (`rs/canister_sandbox/src/replica_controller/sandboxed_execution_controller.rs`)

## Summary
The `update_execution_state` function in the sandboxed execution controller accepts the `globals` vector returned by the sandbox over IPC without validating its length or element types against the original `exported_globals`. A compromised sandbox can return a malformed globals vector that is persisted unconditionally to replicated state. On the next execution, the corrupted globals are fed back to a new sandbox instance, which calls `fatal!()` and aborts. This sandbox abort propagates through the launcher process (which also panics) to the replica process, which crashes. Because the corrupted globals are in the checkpoint, the replica enters a permanent crash loop requiring subnet recovery.

## Finding Description

**Step 1 — No globals validation in `update_execution_state`:**

At `rs/canister_sandbox/src/replica_controller/sandboxed_execution_controller.rs:1713–1724`, the function already guards `num_instructions_left` with an explicit comment "If sandbox is compromised this value could be larger than the initial limit." No equivalent guard exists for the globals vector. At lines 1817–1822, `execution_state_modifications.globals` is wrapped directly into `ExecutionStateChanges` without any comparison against `execution_state.exported_globals`.

**Step 2 — Corrupted globals committed to replicated state:**

`apply_canister_state_changes` in `rs/execution_environment/src/execution/common.rs:590` unconditionally assigns `execution_state.exported_globals = globals` when `ExecutionStateChanges` is present. The `ExecutionStateModifications` struct in `rs/canister_sandbox/src/protocol/structs.rs:76–86` has no invariant enforcement. The corrupted globals are checkpointed.

**Step 3 — Fatal panic on next execution:**

On the next execution, the replica sends the now-corrupted `exported_globals` (wrong length or wrong types) to a fresh sandbox instance. In `rs/embedders/src/wasmtime_embedder.rs:512–518`, a length mismatch triggers `fatal!()`. At lines 539–553, a type mismatch triggers a second `fatal!()` path. Both are confirmed by `#[should_panic]` tests in `rs/embedders/tests/wasmtime_embedder.rs:465–497`.

**Step 4 — Crash propagation to replica:**

The sandbox binary installs `abort_on_panic()` via `child_process_initialization()` in `rs/canister_sandbox/src/lib.rs:136`, so the `fatal!()` panic causes `std::process::abort()` in the sandbox. The launcher's child-watcher thread in `rs/canister_sandbox/src/launcher.rs:143–156` detects the unexpected sandbox exit and calls `panic!("Launcher detected sandbox exit")`, which also aborts the launcher (same `abort_on_panic` hook). The replica's monitoring thread at `rs/canister_sandbox/src/replica_controller/sandboxed_execution_controller.rs:1344–1349` calls `panic_due_to_exit()` at line 2175–2183, panicking the replica. Because the corrupted globals are in the checkpoint, every restart replays the same sequence — a permanent crash loop requiring subnet recovery.

**Existing checks are insufficient:**

Memory size is validated (`verify_size()` at lines 1784–1815) and instruction counts are clamped (lines 1713–1724), but the globals vector has no analogous guard.

## Impact Explanation

A compromised sandbox can corrupt replicated state in a way that causes every subsequent execution attempt for the affected canister to crash the sandbox, the launcher, and the replica. Because the IC is a deterministic replicated state machine, all replicas on the subnet restart from the same checkpoint and hit the same crash. This constitutes a **subnet-level availability impact** — a crash loop requiring manual subnet recovery — matching: **High ($2,000–$10,000): Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS.**

## Likelihood Explanation

**Low.** The attacker must first achieve arbitrary code execution within the sandbox process (e.g., via a Wasmtime memory-safety vulnerability), then craft a malformed `ExecutionFinishedRequest` with a globals vector of wrong length or wrong element types. This is a two-step exploit. However, the IC's own threat model explicitly anticipates sandbox compromise (the `num_instructions_left` guard with its "If sandbox is compromised" comment demonstrates this), making the missing globals guard a concrete gap in the existing defense-in-depth posture.

## Recommendation

In `update_execution_state`, after receiving `execution_state_modifications`, add a guard analogous to the existing `num_instructions_left` check:

```rust
// Guard: sandbox cannot change the number or types of exported globals.
if execution_state_modifications.globals.len() != execution_state.exported_globals.len() {
    error!(self.logger,
        "[EXC-BUG] Canister {} sandbox returned {} globals, expected {}",
        canister_id,
        execution_state_modifications.globals.len(),
        execution_state.exported_globals.len());
    self.metrics.sandboxed_execution_critical_error_invalid_globals.inc();
    // Reject the modification rather than persist corrupted state.
    // Return the original globals unchanged.
    execution_state_modifications.globals = execution_state.exported_globals.clone();
}
```

Element-by-element type validation (matching `Global::I32` to `Global::I32`, etc.) should also be enforced before committing. This mirrors the pattern already used for memory size validation at lines 1784–1815.

## Proof of Concept

1. Deploy a canister with N exported globals.
2. Intercept the sandbox IPC channel; inject an `ExecutionFinishedRequest` where `execution_state_modifications.globals` has N+K entries (or entries with wrong types).
3. Observe that `update_execution_state` accepts the vector without rejection and `apply_canister_state_changes` persists it (`execution_state.exported_globals = globals`).
4. Trigger any subsequent execution for the canister.
5. Observe the sandbox calls `fatal!()` at `rs/embedders/src/wasmtime_embedder.rs:512–518` ("Given number of exported globals N+K is not equal to the number of instance exported globals N"), confirmed by the `#[should_panic]` test at `rs/embedders/tests/wasmtime_embedder.rs:465–479`.
6. Observe the launcher panics ("Launcher detected sandbox exit") and the replica crashes.
7. Restart the replica; observe the crash loop repeats from checkpoint.
