### Title
Missing Caller Validation in `ft_on_transfer` Allows Minting ERC-20 Tokens Without Transferring NEP-141 Tokens - (File: `engine/src/contract_methods/connector.rs`)

### Summary
The `ft_on_transfer` function is a NEP-141 callback that should only be invoked by a legitimate token contract as part of the `ft_transfer_call` flow. It has no access control, allowing any NEAR account to call it directly. An attacker who controls a NEAR account registered as a NEP-141 token on Aurora can call `ft_on_transfer` directly with arbitrary `sender_id` and `amount`, causing Aurora to mint ERC-20 tokens without receiving the corresponding NEP-141 tokens.

### Finding Description

The `ft_on_transfer` function is exposed as a public NEAR contract entry point with no caller restriction: [1](#0-0) 

Inside the implementation, the function reads `predecessor_account_id` and uses it to route to either `receive_base_tokens` (if the caller is the connector) or `receive_erc20_tokens` (for any other caller): [2](#0-1) 

The only guard present is `require_running`. There is no assertion that `predecessor_account_id` is a legitimately registered NEP-141 token contract that has already transferred tokens to Aurora as part of a `ft_transfer_call`. The `args.sender_id` and `args.amount` fields are fully attacker-controlled inputs read from JSON: [3](#0-2) 

Additionally, `deploy_erc20_token` — the function that registers a NEP-141 account as having an ERC-20 mirror on Aurora — has no access control beyond `require_running`: [4](#0-3) 

**Attack path:**
1. Attacker creates a NEAR account `attacker.near`.
2. Attacker calls `deploy_erc20_token` on Aurora with `nep141 = "attacker.near"` — succeeds because there is no caller restriction.
3. Attacker calls `ft_on_transfer` directly on Aurora from `attacker.near`, supplying `sender_id = <attacker EVM address>` and `amount = <arbitrary large value>`.
4. Aurora routes to `receive_erc20_tokens(&predecessor_account_id, &args, ...)`, which looks up the ERC-20 mirror for `attacker.near` and mints the specified amount of ERC-20 tokens to the attacker's EVM address.
5. No NEP-141 tokens were ever transferred to Aurora. The ERC-20 supply is now unbacked.

The analog to the reported bug is exact: just as `sponsor` was documented as "called by Factory" but lacked the check, `ft_on_transfer` is documented as a NEP-141 callback invoked by a token contract after transfer, but the engine never verifies the caller is such a contract.

### Impact Explanation
**Critical — Insolvency.** The attacker mints ERC-20 tokens on Aurora that are not backed by any NEP-141 tokens held by the Aurora contract. The ERC-20 token's total supply on Aurora exceeds the actual NEP-141 balance Aurora holds, making the bridge insolvent for that token. Any legitimate holder of the same ERC-20 token who attempts to exit via `ExitToNear` may find Aurora unable to fulfill the withdrawal because the NEP-141 reserves are insufficient.

### Likelihood Explanation
**High.** The attack requires only:
1. Creating a NEAR account (trivial, costs fractions of a cent).
2. Calling `deploy_erc20_token` to register it (no access control).
3. Calling `ft_on_transfer` directly (no access control).

No privileged access, leaked keys, or governance capture is required. The entire attack is executable by any unprivileged NEAR account holder.

### Recommendation
Add a caller check inside `ft_on_transfer` to assert that `predecessor_account_id` is a known, registered NEP-141 token contract before minting any ERC-20 tokens. Concretely, before calling `receive_erc20_tokens`, verify that a mapping from `predecessor_account_id` to an ERC-20 address already exists in storage **and** that the call arrived as a genuine NEP-141 callback (i.e., the predecessor is the token contract, not an arbitrary account). A stricter approach is to maintain an explicit allowlist of approved NEP-141 token contracts that are permitted to invoke `ft_on_transfer`.

### Proof of Concept

```
# Step 1 – register attacker's NEAR account as a NEP-141 token on Aurora
near call aurora deploy_erc20_token \
  '{"nep141": "attacker.near"}' \
  --accountId attacker.near --gas 100000000000000

# Step 2 – call ft_on_transfer directly (no ft_transfer_call involved)
near call aurora ft_on_transfer \
  '{"sender_id": "attacker.near", "amount": "1000000000000000000000000", "msg": "<attacker_evm_hex>"}' \
  --accountId attacker.near --gas 100000000000000

# Result: Aurora mints 1e24 ERC-20 tokens (for attacker.near's mirror)
# to the attacker's EVM address with zero NEP-141 tokens transferred.
# Aurora's ERC-20 supply for that token now exceeds its NEP-141 reserves.

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
