### Title
`new_settlement_layer_chain_id_storage` Excluded from IO Frame Snapshot Prevents Rollback on Transaction Revert — (`basic_system/src/system_implementation/system/io_subsystem.rs`)

---

### Summary

`FullIO` maintains six sub-storages, but only five of them are snapshotted and rolled back when an IO frame reverts. `new_settlement_layer_chain_id_storage` is silently omitted from `FullIOStateSnapshot`, from `start_io_frame`, and from `finish_io_frame`. Any transaction that triggers a settlement-layer chain-ID update and then reverts leaves a permanent, non-rolled-back mutation in that storage for the rest of the block, corrupting the block commitment and permanently blocking any subsequent legitimate chain-ID update in the same block.

---

### Finding Description

`FullIO` holds six sub-storages:

```
storage, transient_storage, logs_storage,
events_storage, interop_root_storage,
new_settlement_layer_chain_id_storage   ← never snapshotted
```

`FullIOStateSnapshot` captures only five of them: [1](#0-0) 

`start_io_frame` calls `start_frame()` on every sub-storage **except** `new_settlement_layer_chain_id_storage`: [2](#0-1) 

`finish_io_frame` rolls back every sub-storage **except** `new_settlement_layer_chain_id_storage`: [3](#0-2) 

`NewSettlementLayerChainIdStorage` does expose `start_frame` / `finish_frame` methods that are fully capable of rollback: [4](#0-3) 

The mutation path is:

1. An EVM `LOG` opcode emits a `SettlementLayerChainIdUpdated` event.
2. `system_context_event_hook` fires synchronously and calls `system.io.update_settlement_layer_chain_id(...)`.
3. `new_settlement_layer_chain_id_storage.update()` writes the new chain ID. [5](#0-4) 

4. The transaction later reverts (out-of-gas, explicit `REVERT`, or block-limit exceeded).
5. `finish_io_frame(Some(&pre_tx_rollback_handle))` is called — all five snapshotted sub-storages are rolled back, but `new_settlement_layer_chain_id_storage` retains the written value. [6](#0-5) 

Furthermore, `begin_next_tx` does not reset `new_settlement_layer_chain_id_storage`, so the stale value persists across all subsequent transactions in the block: [7](#0-6) 

`update()` enforces a once-per-block invariant: if `value().is_some()` it returns an internal error: [8](#0-7) 

Once the stale value is set by a reverted transaction, every subsequent call to `update_settlement_layer_chain_id` in the same block panics with an internal error, permanently blocking legitimate governance operations for that block.

---

### Impact Explanation

**State-transition / storage rollback bug.**

- The block output/commitment includes a settlement-layer chain-ID update that originated from a reverted transaction. The committed state diverges from what the EVM execution model guarantees (reverted side-effects must not persist).
- In proving mode the prover must reproduce the same state transition; the forward sequencer and the prover will disagree on whether the chain-ID update is present, causing a forward/proving divergence.
- Any subsequent legitimate service transaction that attempts to update the settlement-layer chain ID in the same block will hit the once-per-block guard and fail with an internal error, constituting a denial-of-service against that governance operation.

---

### Likelihood Explanation

The `system_context_event_hook` checks only the event signature (`SL_CHAIN_ID_UPDATED_EVENT_SIG`) and not the emitting contract address: [9](#0-8) 

If the hook is registered globally for all emitters, any unprivileged user can deploy a contract that emits the matching 4-topic log and then reverts, triggering the bug with a single transaction. Even if the hook is restricted to the SystemContext contract address, a transaction that calls SystemContext and runs out of gas after the event emission triggers the same path. The revert path is exercised in normal block processing (block-limit exceeded, out-of-gas, explicit revert), making accidental triggering realistic and deliberate triggering trivial.

---

### Recommendation

1. Add `new_settlement_layer_chain_id_storage` to `FullIOStateSnapshot`:

```rust
pub struct FullIOStateSnapshot<M: StorageModel> {
    io: M::StateSnapshot,
    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,
    new_sl_chain_id: NewSettlementLayerChainIdSnapshotId, // add this
}
```

2. Call `start_frame()` on it inside `start_io_frame` and store the handle in the snapshot.

3. Call `finish_frame(rollback_handle.map(|x| x.new_sl_chain_id))` on it inside `finish_io_frame`.

4. Verify whether the hook should be restricted to events emitted only by the canonical SystemContext address to prevent unprivileged callers from triggering the update path.

---

### Proof of Concept

```
Block with two transactions:

Tx 1 (attacker):
  - Calls a contract that emits LOG2(SL_CHAIN_ID_UPDATED_EVENT_SIG, attacker_chain_id)
    with empty data, then executes REVERT.
  - system_context_event_hook fires → new_settlement_layer_chain_id_storage.update(attacker_chain_id)
  - finish_io_frame(Some(pre_tx_snapshot)) rolls back storage/events/transient/logs/interop_roots
    but NOT new_settlement_layer_chain_id_storage.
  - Block output now contains attacker_chain_id as the new settlement-layer chain ID.

Tx 2 (legitimate governance):
  - Calls SystemContext to set the real new chain ID.
  - system_context_event_hook fires → new_settlement_layer_chain_id_storage.update(real_chain_id)
  - update() sees value().is_some() → returns InternalError → transaction aborts.
  - Legitimate chain-ID update is blocked for the entire block.
```

```rust
fn new_sl_chain_id_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // Internal error if the data supplied isn't empty
    if !data.is_empty() {
        return Err(
            internal_error!("New SL chain id reporter event hook received bad data").into(),
        );
    }
    // Same if there's a mismatch in expected topics
    if topics.len() != 2 {
        return Err(
            internal_error!("New SL chain id reporter event hook received bad topics").into(),
        );
    }

    let new_sl_chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    system.io.update_settlement_layer_chain_id(
        ExecutionEnvironmentType::NoEE,
        resources,
        new_sl_chain_id,
    )?;

    Ok(())
```
