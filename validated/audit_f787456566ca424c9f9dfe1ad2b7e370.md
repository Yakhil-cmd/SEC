Audit Report

## Title
Unprivileged Neuron Can Grief Subnet Rental Lifecycle via Spoofed `SubnetRentalRequest` Proposal — (`rs/nns/governance/src/governance.rs`)

## Summary

`validate_subnet_rental_proposal` performs no check that the proposal submitter is the `user` named in the payload, and no check that `user` has deposited ICP. The concurrent-proposal guard is global (not per-user), so any neuron holder can submit a `SubnetRentalRequest{user=victim}` that blocks the legitimate renter from submitting their own proposal for the entire NNS voting period. The attack is repeatable at the cost of the NNS proposal rejection fee.

## Finding Description

`validate_subnet_rental_proposal` in `rs/nns/governance/src/governance.rs` performs exactly two checks:

1. The payload deserializes as a valid `SubnetRentalRequest`.
2. No other non-final `SubnetRentalRequest` proposal exists (global, not per-user). [1](#0-0) 

The function signature receives only `&self` and `payload: &[u8]` — the proposer's identity is never passed in and never checked. The `make_proposal` caller/neuron identity is available at the call site but is not threaded into this validation. [2](#0-1) 

The `SubnetRentalRequest` struct exposes `user` as a free `PrincipalId` field with no ownership constraint: [3](#0-2) 

The ICP deposit check is deferred entirely to execution time inside the Subnet Rental Canister. The existing integration test `test_renting_a_subnet_without_paying_fails` confirms that a proposal with no ICP deposit passes `validate_subnet_rental_proposal` and only fails at execution: [4](#0-3) 

`select_nonfinal_proposal_ids` blocks on any non-final `SubnetRentalRequest` proposal regardless of which `user` is named: [5](#0-4) 

**Exploit flow:**
1. Attacker (any NNS neuron holder) submits `SubnetRentalRequest{user=victim_principal, rental_condition_id=App13CH}` with no ICP deposited.
2. `validate_subnet_rental_proposal` passes: payload is valid, no concurrent proposal exists.
3. While this proposal is in `Open` or `Adopted` state, any subsequent `SubnetRentalRequest` — including the legitimate one from the victim — is rejected with `"There is another open SubnetRentalRequest proposal: [attacker_proposal_id]"`.
4. If the attacker's proposal is adopted and executed, it fails with `InsufficientFunds` and reaches `Failed` state, unblocking the victim — but the attacker can immediately repeat.
5. If the attacker's proposal stays `Open` (no large neuron votes YES), the block lasts the full NNS voting period (days).

## Impact Explanation

This is a targeted, repeatable application-level DoS against the NNS subnet rental process. A legitimate renter who has already deposited ICP to the Subnet Rental Canister is indefinitely blocked from submitting their proposal. Their ICP deposit is held in the Subnet Rental Canister while the attack is active (they can request a refund, but the rental process is stalled). The attacker pays only the NNS proposal rejection fee per cycle. This constitutes a significant NNS governance security impact with concrete user harm, matching the **High ($2,000–$10,000)** impact class: "Application/platform-level DoS... or subnet availability impact not based on raw volumetric DDoS" and "Significant... NNS... security impact with concrete user or protocol harm."

## Likelihood Explanation

- Requires only a standard NNS neuron — no privileged role, no governance majority.
- The proposal passes `validate_subnet_rental_proposal` unconditionally for any valid `PrincipalId` as `user`.
- The attack is cheap relative to the cost of renting a subnet (rejection fee only).
- The victim has no recourse within the protocol; they must wait for the blocking proposal to finalize before submitting their own.
- The attack is fully repeatable: the attacker can re-submit immediately after each blocking proposal finalizes.

## Recommendation

Add a caller-equals-user check inside `validate_subnet_rental_proposal`. The `make_proposal` function already has access to the `caller: &PrincipalId` and `proposer_id: &NeuronId`; thread the neuron's controller/hotkey into `validate_subnet_rental_proposal` and verify it matches the `user` field in the decoded payload. This eliminates the spoofing vector entirely without requiring any async ICP deposit pre-check. [6](#0-5) 

## Proof of Concept

Extend the existing state-machine integration test infrastructure in `rs/nns/integration_tests/src/subnet_rental_canister.rs`:

1. `attacker_neuron` (any neuron other than `large_neuron`) submits `SubnetRentalRequest{user=victim_principal, rental_condition_id=App13CH}` — victim has NOT deposited ICP.
2. Assert the proposal is accepted (passes validation).
3. While the proposal is non-final (Open), `victim_neuron` submits `SubnetRentalRequest{user=victim_principal}` (victim HAS deposited ICP).
4. Assert the victim's proposal is rejected with `"There is another open SubnetRentalRequest proposal"`.
5. Wait for the attacker's proposal to reach `Failed` state (via `nns_wait_for_proposal_failure`).
6. Assert the victim can now successfully submit their proposal.

The existing `test_renting_a_subnet_without_paying_fails` test already demonstrates steps 1–2 and the `InsufficientFunds` failure path in isolation. [7](#0-6)

        // Must be able to parse the payload.
        if let Err(e) = Decode!([decoder_config()]; &payload, SubnetRentalRequest) {
            return Err(format!("Invalid SubnetRentalRequest: {e}"));
        }

        // No concurrent subnet rental requests are allowed.
        let other_proposal_ids = self.select_nonfinal_proposal_ids(|action| {
            let Action::ExecuteNnsFunction(execute_nns_function) = action else {
                return false;
            };

            execute_nns_function.nns_function == NnsFunction::SubnetRentalRequest as i32
        });
        if !other_proposal_ids.is_empty() {
            return Err(format!(
                "There is another open SubnetRentalRequest proposal: {other_proposal_ids:?}",
            ));

fn test_renting_a_subnet_without_paying_fails() {
