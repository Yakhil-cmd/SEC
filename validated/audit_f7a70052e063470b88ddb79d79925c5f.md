Audit Report

## Title
Unauthorized Third-Party Can Force SNS Swap Participation for Any User - (`rs/sns/swap/canister/canister.rs`)

## Summary
The `refresh_buyer_tokens` endpoint accepts an arbitrary `buyer` principal in the request body with no check that the caller matches the specified buyer. Any unprivileged user can call this function with a victim's principal after the victim has transferred ICP to their swap subaccount, registering the victim as a committed participant without their explicit consent. Once registered, the victim's ICP is locked in escrow and cannot be refunded until the swap finalizes and `sweep_icp` runs, at which point the victim receives SNS tokens instead of ICP.

## Finding Description

In `rs/sns/swap/canister/canister.rs` (lines 127–143), the `refresh_buyer_tokens` update method resolves the buyer principal from the caller-supplied `arg.buyer` field with no authorization check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // ← attacker-controlled
};
```

The resolved principal `p` is passed directly to `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs` (lines 1134–1163), which reads the ICP balance of `swap_canister_subaccount[p]` on the ICP ledger and registers that amount as the buyer's committed participation.

The only other check is `validate_confirmation_text` (lines 363–384 of `swap.rs`), which compares the caller-supplied text against the SNS init payload's `confirmation_text` field. This field is publicly readable on-chain via the `get_init` query endpoint, so it provides no real access control — any attacker can read and replay it.

The ticket check at lines 1248–1272 of `swap.rs` is also insufficient: if the victim has no open ticket, the check is entirely skipped (`// If there exists no ticket for the buyer, the payment flow will simply ignore the ticket`). If a ticket exists, the attacker can still proceed as long as the subaccount balance meets the ticket amount, and the ticket is then deleted.

Once `refresh_buyer_tokens` succeeds for the victim, `error_refund_icp` is blocked. At lines 1931–1936 of `swap.rs`, `error_refund_icp` requires lifecycle ABORTED or COMMITTED. Even after the swap closes, lines 1950–1959 block the refund while the buyer's ICP has `transfer_success_timestamp_seconds == 0` (i.e., not yet swept), returning: `"ICP cannot be refunded as principal X has Y ICP in escrow"`. The victim's ICP is locked until `sweep_icp` runs during finalization.

The integration test at `rs/tests/nns/sns/lib/src/sns_deployment.rs` (lines 919–926) explicitly demonstrates and validates this cross-identity call path: `default_sns_agent` calls `refresh_buyer_tokens(Some(wealthy_user_identity.principal_id), None)` and the test asserts it succeeds identically to the user calling for themselves.

## Impact Explanation

This matches the **High** severity impact: "Significant SNS infrastructure security impact with concrete user or protocol harm." A victim who transferred ICP to the swap subaccount but has not yet called `refresh_buyer_tokens` — whether they changed their mind, transferred accidentally, or are mid-flow — can be forcibly committed to the swap by any unprivileged attacker. The victim receives SNS tokens instead of ICP (potentially at a loss), and cannot reclaim their ICP until finalization completes. The `confirmation_text` mechanism, which exists specifically to enforce explicit participant consent to swap terms, is rendered ineffective.

## Likelihood Explanation

**Medium.** The precondition is that the victim has transferred ICP to the swap subaccount but has not yet called `refresh_buyer_tokens`. This window exists in normal usage. An attacker monitoring the ICP ledger for transfers to swap subaccounts (all subaccount addresses are deterministically derived from principal IDs) can detect this window and immediately call `refresh_buyer_tokens` on the victim's behalf. The attack requires no privileged access, no key material, and no on-chain majority. The confirmation text, if set, is publicly readable on-chain. The attack is repeatable across any number of victims during any open SNS swap.

## Recommendation

Add a caller authorization check in `refresh_buyer_tokens` in `rs/sns/swap/canister/canister.rs`. When `arg.buyer` is non-empty, verify the caller matches the specified buyer:

```rust
#[update]
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let specified = PrincipalId::from_str(&arg.buyer).unwrap();
        if specified != caller {
            panic!("Caller is not authorized to refresh tokens for a different buyer");
        }
        specified
    };
    ...
}
```

If the open-caller design is intentional for operational reasons (e.g., allowing bots to assist users), the `confirmation_text` check must be moved to the canister layer and verified against the caller's identity rather than the supplied text, or an explicit opt-in mechanism must be added.

## Proof of Concept

1. Alice calls `new_sale_ticket` on the SNS swap canister.
2. Alice transfers `min_participant_icp_e8s` ICP to `swap_canister_subaccount[alice]` on the ICP ledger.
3. Alice decides not to participate and does **not** call `refresh_buyer_tokens`.
4. Bob (attacker), monitoring the ICP ledger, observes Alice's transfer. Bob reads the SNS init payload via `get_init` to obtain the `confirmation_text` (if set). Bob calls:
   ```
   refresh_buyer_tokens({ buyer = alice_principal; confirmation_text = <text_from_get_init> })
   ```
   from his own identity.
5. The swap canister reads Alice's subaccount balance, finds the ICP, and registers Alice as a committed participant.
6. Alice's ICP is now in escrow. `error_refund_icp` returns `"ICP cannot be refunded as principal alice has N ICP (e8s) in escrow"` until `sweep_icp` runs.
7. When the swap commits and `sweep_icp` runs, Alice's ICP is transferred to SNS governance and Alice receives SNS tokens — forced into a swap she did not consent to.

This exact cross-identity call pattern is already exercised and asserted to succeed in `rs/tests/nns/sns/lib/src/sns_deployment.rs` lines 919–948, confirming the attack path is reachable on mainnet.
