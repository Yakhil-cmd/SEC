Audit Report

## Title
SNS Governance `distribute_rewards` Unbounded Neuron Iteration Causes Permanent Reward Liveness Failure - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS Governance canister's `distribute_rewards` function iterates over all voting neurons in a single synchronous message with no instruction-limit guard. If the neuron count grows large enough to exhaust the IC per-message instruction limit, the message traps and rolls back all state changes. Because `latest_reward_event` and proposal settlement are only committed after the loop, the canister enters a permanent liveness failure where every subsequent timer invocation re-attempts and traps on the same unbounded loop.

## Finding Description
`distribute_rewards` is called synchronously from `run_periodic_tasks` at [1](#0-0)  after `should_distribute_rewards()` returns true.

Inside `distribute_rewards`, the function builds `neuron_id_to_reward_shares` from all proposal ballots, then iterates over every entry to credit maturity at [2](#0-1) . There is no call to `is_message_over_threshold` or any equivalent instruction-limit guard anywhere in this loop — confirmed by the absence of any such pattern in the entire `rs/sns/` tree.

Only after the loop completes does the function settle proposals (lines 6013–6081) and write the new `latest_reward_event` at [3](#0-2) . If the loop traps due to instruction exhaustion, all state changes roll back atomically: `latest_reward_event` is unchanged, proposals remain in `ReadyToSettle`, and `should_distribute_rewards()` returns `true` again on the next timer tick, re-entering the identical (or larger) loop.

By contrast, NNS Governance uses a batched, resumable state machine with an explicit instruction-limit check at [4](#0-3) , breaking out of the loop and rescheduling via a periodic timer. The SNS canister has no equivalent mechanism.

## Impact Explanation
Once the voting-neuron count in an SNS DAO grows large enough to exhaust the ~40 billion instruction per-message limit, every invocation of `run_periodic_tasks` traps at the same point. Voting rewards are permanently halted: neuron maturity stops accruing, breaking the economic incentive for governance participation. This is a **governance liveness failure** matching the allowed impact: *"Significant SNS security impact with concrete user or protocol harm"* — **High severity**.

## Likelihood Explanation
This is an organic, non-adversarial failure requiring no special privileges. Any SNS DAO that grows to a sufficiently large number of active voting neurons will hit it. The NNS governance team already acknowledged and fixed the identical bug in NNS Governance (Proposal 135702), explicitly noting batched distribution and per-message instruction limits were added. The SNS canister did not receive the same remediation, leaving any sufficiently popular SNS DAO at risk.

## Recommendation
Apply the same batched-distribution pattern used in NNS Governance to SNS Governance:
1. After computing `neuron_id_to_reward_shares`, store the pending distribution in stable memory (analogous to `RewardsDistributionStateMachine`).
2. Advance `latest_reward_event` and settle proposals immediately so they are not re-processed on the next timer tick.
3. Distribute maturity to neurons in a separate recurring timer task that checks `is_message_over_threshold` after each neuron and resumes in the next message if the limit is reached, mirroring the NNS implementation at [5](#0-4) .

## Proof of Concept
1. An SNS DAO accumulates a large number of neurons (e.g., 50,000+), all of which vote on proposals.
2. The recurring timer fires `run_periodic_tasks` → `distribute_rewards`.
3. The loop at line 5954 iterates over all `neuron_id_to_reward_shares` entries, performing a `get_neuron_result_mut` lookup and maturity update for each.
4. After enough neurons, the IC instruction counter exceeds the per-message limit; the message traps.
5. All state changes roll back. `latest_reward_event` is unchanged. `considered_proposals` remain in `ReadyToSettle`.
6. The next timer fires and `should_distribute_rewards()` returns `true` again. The identical loop runs and traps again.
7. Voting rewards are permanently halted.

A deterministic integration test can reproduce this by creating an SNS governance canister in PocketIC, staking enough neurons to exceed the instruction limit, submitting and closing a proposal, then asserting that `latest_reward_event` never advances across multiple timer ticks.
