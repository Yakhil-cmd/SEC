Audit Report

## Title
Archive `icrc3_supported_block_types` Omits ICRC-122 Block Types When ICRC-152 Is Enabled — (`rs/ledger_suite/icrc1/archive/src/main.rs`)

## Summary
The ICRC-1 archive canister's `icrc3_supported_block_types` endpoint returns a hardcoded static list that permanently excludes `"122mint"` and `"122burn"`. When the paired ledger has ICRC-152 enabled, it produces and archives blocks with exactly those `btype` values. Any ICRC-3-compliant client that uses `icrc3_supported_block_types` for discovery will never learn the archive holds ICRC-122 blocks, causing those blocks to be silently skipped during history reconstruction once they leave the ledger's in-memory window.

## Finding Description
The archive's `icrc3_supported_block_types` at `rs/ledger_suite/icrc1/archive/src/main.rs` L324–356 returns a compile-time-fixed `vec![]` containing only `"1burn"`, `"1mint"`, `"1xfer"`, `"2approve"`, `"2xfer"`, and `BTYPE_107`. There is no runtime state, no feature-flag check, and no reference to ICRC-122 types anywhere in the archive canister (confirmed by grep returning zero matches for `icrc152` in that file).

The ledger, by contrast, conditionally appends `"122mint"` and `"122burn"` to its own `icrc3_supported_block_types` response when the `icrc152` feature flag is set, as verified by the integration test at `rs/ledger_suite/tests/sm-tests/src/lib.rs` L6594–6644.

When `icrc152_mint` is called (`rs/ledger_suite/icrc1/ledger/src/main.rs` L990–996), it records a block with `Operation::AuthorizedMint` (which serialises to `btype = "122mint"`, defined in `packages/icrc-ledger-types/src/icrc122/schema.rs` L10) and immediately calls `archive_blocks`. Once the ledger's in-memory window fills, those blocks are moved to the archive. The archive stores them faithfully but its `icrc3_supported_block_types` response never mentions them.

The result is a split view: while blocks reside on the ledger, the ledger's discovery endpoint advertises `"122mint"`/`"122burn"` and clients see them. Once archived, the archive's discovery endpoint omits those types and standard-compliant clients stop querying for them.

## Impact Explanation
An ICRC-3-compliant client (index canister, chain-fusion bridge, block explorer, or auditor) that follows the standard discovery flow — call `icrc3_supported_block_types`, then issue `icrc3_get_blocks` only for advertised types — will silently omit every archived ICRC-152 mint and burn. For a chain-fusion bridge that reconstructs token supply or verifies authorized mints from the block log, this produces incorrect accounting without any on-chain error signal. Missed burns mean the bridge may credit assets that have already been destroyed; missed mints mean authorized issuances are invisible post-archival. This constitutes a significant ledger and Chain Fusion security impact with concrete user and protocol harm, qualifying as High severity.

## Likelihood Explanation
ICRC-152 is a deployable production feature flag on the ICRC-1 ledger. Any operator that enables it and accumulates enough transactions to trigger archiving exposes this discrepancy to every ICRC-3-compliant client. The ICRC-3 standard explicitly designates `icrc3_supported_block_types` as the authoritative discovery mechanism, so well-behaved clients are expected to rely on it. The defect is deterministic and reproducible on every archive canister paired with an ICRC-152-enabled ledger; no attacker action is required — normal operation triggers it.

## Recommendation
The archive's advertised block-type set must be a superset of every `btype` value it can store. Two concrete approaches:

1. **Dynamic propagation**: When the ledger spawns or upgrades an archive, include the set of block types the ledger can produce (derived from its feature flags) in the archive's init/upgrade argument. The archive stores this set and returns it from `icrc3_supported_block_types`.
2. **Static inclusion with feature awareness**: Pass the ledger's feature flags to the archive at spawn time and conditionally include `BTYPE_122_MINT` / `BTYPE_122_BURN` (from `packages/icrc-ledger-types/src/icrc122/schema.rs`) in the archive's response when ICRC-152 is enabled.

An integration test should assert that after archiving ICRC-152 blocks, the archive's `icrc3_supported_block_types` includes `"122mint"` and `"122burn"`.

## Proof of Concept
1. Deploy an ICRC-1 ledger with `icrc152: true` in the feature flags.
2. Call `icrc152_mint` enough times to exhaust the ledger's in-memory block window and trigger `archive_blocks`.
3. Query `icrc3_supported_block_types` on the **archive** canister — observe that `"122mint"` and `"122burn"` are absent (only `"1burn"`, `"1mint"`, `"1xfer"`, `"2approve"`, `"2xfer"`, `BTYPE_107` are returned).
4. Query `icrc3_get_blocks` on the archive for the archived range — observe that blocks with `btype = "122mint"` are present and fully retrievable.
5. A client following the standard discovery flow (step 3 before step 4) would never issue the query in step 4 for those block types, silently missing all archived ICRC-152 mints and burns.
