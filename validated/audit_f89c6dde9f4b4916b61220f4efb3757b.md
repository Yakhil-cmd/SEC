### Title
`native_per_pubdata` Floor-Division Truncates to Zero When `pubdata_price < native_price`, Zeroing All Pubdata Native Charges - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In the ZK L2 transaction validation path, `native_per_pubdata` is computed with **floor division** (`wrapping_div`). When the operator-supplied `pubdata_price` is less than `native_price`, the result truncates to exactly zero. Every subsequent pubdata charge in that block — both the intrinsic overhead deducted from the native limit and the post-execution pubdata charge — becomes zero. Any unprivileged sender can then submit pubdata-heavy transactions and consume the full block pubdata budget without spending a single unit of native resource on it.

---

### Finding Description

**Root cause — floor division in `native_per_pubdata`**

In `validation_impl.rs` line 142:

```rust
// We checked native_price != 0 above
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is Rust's standard integer floor division. When `pubdata_price < native_price`, the quotient is `0`. There is no guard that rejects or corrects this result.

Compare with `native_per_gas`, computed on line 135 with **ceiling** division:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

`div_ceil` guarantees `native_per_gas ≥ 1` whenever `gas_price > 0`, preventing gas from being free. No equivalent protection exists for pubdata.

**Propagation through the resource pipeline**

`native_per_pubdata = 0` silently zeroes every downstream pubdata charge:

1. **Intrinsic pubdata overhead** (`gas_helpers.rs` line 352):
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   // = 0 × intrinsic_pubdata = 0  → native_limit unchanged
   ```

2. **Post-execution pubdata charge** (`gas_helpers.rs` line 430–432):
   ```rust
   let native = current_pubdata_spent
       .checked_mul(native_per_pubdata)   // = pubdata_bytes × 0 = 0
       .ok_or(out_of_native_resources!())?;
   ```
   `check_enough_resources_for_pubdata` always returns `enough = true` and charges zero native, regardless of how many pubdata bytes the transaction wrote.

3. **`deltaGas` correction** (`refund_calculation.rs` line 72):
   ```rust
   (native_used / native_per_gas) as i64 - (gas_used as i64)
   ```
   Because pubdata consumed zero native, `native_used` is artificially low, so `delta_gas` is also suppressed and no compensating gas is added.

The same floor-division path is mirrored in the public API helper (`api/src/helpers.rs` line 427):
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```
confirming this is a systemic pattern, not an isolated typo.

**Triggering condition**

`pubdata_price < native_price` is a reachable state. `pubdata_price` tracks the cost of posting calldata/state-diff bytes to L1; `native_price` tracks the cost of a single RISC-V proving cycle. During periods of low L1 gas prices (or if the operator misconfigures the ratio), `pubdata_price` can legitimately fall below `native_price`. The protocol imposes no invariant preventing this ordering.

---

### Impact Explanation

When `native_per_pubdata = 0`:

- Every transaction in the block can write storage slots, emit events, and generate state-diff bytes up to the block pubdata limit (`pubdata_limit`) without consuming any native resource for those bytes.
- The operator still pays L1 data-availability costs for every byte posted, but receives zero native-resource compensation from senders.
- A single attacker can fill the entire block pubdata budget (e.g., by writing to many distinct storage slots) at the cost of only EVM gas, with no native resource expenditure. This is the direct analog of the Surge "borrow at the current stale rate" impact: the resource-accounting mechanism is frozen at zero, and the attacker exploits the gap between the real cost (L1 posting) and the charged cost (zero native).

---

### Likelihood Explanation

- `pubdata_price` and `native_price` are independent block-level parameters supplied by the operator oracle. Their relative magnitude is not constrained by the protocol.
- On ZKsync (an L2), L1 gas prices fluctuate. A sustained drop in L1 calldata costs can push `pubdata_price` below `native_price` without any malicious intent.
- The inconsistency between `div_ceil` (for `native_per_gas`) and `wrapping_div` (for `native_per_pubdata`) is subtle; an operator reviewing the pricing model would not obviously predict that a ratio slightly below 1 collapses to zero rather than rounding up to 1.
- No existing validation rejects or warns when `pubdata_price < native_price`.

---

### Recommendation

Replace the floor division with ceiling division for `native_per_pubdata`, mirroring the treatment of `native_per_gas`:

```rust
// Before (floor division — can produce 0 when pubdata_price < native_price)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — guarantees ≥ 1 whenever pubdata_price > 0)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix to the mirrored calculation in `api/src/helpers.rs` line 427. Optionally, add a protocol-level assertion that `pubdata_price >= native_price` when `pubdata_price > 0`, or document the intended behavior if zero is acceptable.

---

### Proof of Concept

1. Configure a block context with `native_price = 100` and `pubdata_price = 99` (one unit below `native_price`).
2. `native_per_pubdata = 99 / 100 = 0` (floor division).
3. Deploy a contract that writes to 20 distinct storage slots (generating ~640 bytes of pubdata).
4. Submit the transaction with a gas limit sufficient for EVM execution but a native limit that would be exhausted if pubdata were charged at the correct rate.
5. Observe: the transaction succeeds and `native_used` reflects only computational native, not pubdata native. The operator posts 640 bytes to L1 and receives no native-resource compensation.
6. Repeat with `pubdata_price = 100` (equal to `native_price`): `native_per_pubdata = 1`, pubdata is charged correctly, and the same transaction may revert due to insufficient native resources — confirming the cliff-edge behavior introduced by floor division.
