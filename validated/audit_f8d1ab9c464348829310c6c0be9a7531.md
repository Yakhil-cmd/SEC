### Title
Single-Step `set_owner` Transfer Permanently Locks All Owner-Gated Engine Functions — (`engine/src/contract_methods/admin.rs`)

---

### Summary

The Aurora Engine's `set_owner` function performs an immediate, single-step ownership transfer with no confirmation mechanism. If the current owner supplies a valid NEAR `AccountId` string that they do not actually control, `owner_id` is overwritten instantly and irrecoverably. Every function gated by `require_owner_only` — including `upgrade`, `resume_contract`, `pause_contract`, `attach_full_access_key`, and others — becomes permanently inaccessible with no on-chain recovery path.

---

### Finding Description

`set_owner` in `engine/src/contract_methods/admin.rs` reads a `SetOwnerArgs` struct from input and writes `args.new_owner` directly into `EngineState.owner_id` in a single atomic step:

```rust
// engine/src/contract_methods/admin.rs  lines 103-121
pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;

        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }

        state.owner_id = args.new_owner;   // ← immediate, irreversible write
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
```

The only guards are:
1. Contract must not be paused (`require_running`)
2. Caller must be the current owner (`require_owner_only`)
3. New owner must differ from current owner (`ERR_SAME_OWNER`)

There is **no check** that `args.new_owner` corresponds to an account the caller actually controls, and **no pending-owner / claim step** that would require the new owner to prove key possession before the transfer is finalized.

`require_owner_only` is a strict equality check against `state.owner_id`:

```rust
// engine/src/contract_methods/mod.rs  lines 79-87
pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}
```

Once `owner_id` is set to an uncontrolled account, this check can never pass for any legitimate caller again.

### Impact Explanation

**Permanent freezing of funds (Critical).**

The most direct path to fund loss:

1. Owner calls `pause_contract` (legitimately, e.g., during an incident response). This sets `state.is_paused = true`.
2. Owner then calls `set_owner` with an incorrect `new_owner` address (typo, clipboard error, wrong environment).
3. `owner_id` is now an uncontrolled account.
4. `resume_contract` requires `require_owner_only` — it can never be called again.
5. All user funds held in the Aurora Engine (bridged ETH, ERC-20 mirror balances) are permanently frozen because every EVM transaction path checks `require_running`, which rejects calls when `is_paused = true`.

Even without a prior pause, the loss of `upgrade` and `attach_full_access_key` means:
- No contract bug fixes are possible.
- No full-access-key recovery is possible.
- The engine is effectively bricked for all administrative operations.

---

### Likelihood Explanation

**Low probability, critical impact.**

The scenario requires the current owner to supply a syntactically valid NEAR `AccountId` that they do not control — a realistic operational error (clipboard mistake, wrong environment, transposed characters in a long account name). NEAR `AccountId` validation only enforces format rules, not account existence or key ownership. The probability is low but non-zero for any production system that has ever needed to rotate its owner key.

---

### Recommendation

Replace the single-step transfer with a two-step commit-and-claim pattern:

1. **Step 1 — Propose**: Current owner calls `set_owner(new_owner)`, which writes `new_owner` into a `pending_owner_id` field in `EngineState` without touching `owner_id`.
2. **Step 2 — Claim**: The `pending_owner_id` account calls a new `claim_ownership()` function, which moves `pending_owner_id` into `owner_id` and clears the pending field.

If the proposed address is wrong, the current owner can overwrite `pending_owner_id` with the correct address before any claim is made. Only a successful claim — which requires the new owner to sign a transaction — finalizes the transfer.

---

### Proof of Concept

**State before attack:**
- `state.owner_id = "aurora"` (the legitimate owner)
- Contract is running normally

**Step 1:** Legitimate owner intends to transfer ownership to `"aurora-multisig.near"` but mistakenly submits `"aurora-mu1tisig.near"` (digit `1` instead of letter `l`).

```
near call aurora set_owner \
  '{"new_owner": "aurora-mu1tisig.near"}' \
  --accountId aurora
```

**Step 2:** `set_owner` executes. The only checks pass:
- `require_running` → OK (not paused)
- `require_owner_only` → OK (caller is `"aurora"`, current `owner_id`)
- `state.owner_id != args.new_owner` → OK (different strings)

`state.owner_id` is now `"aurora-mu1tisig.near"`.

**Step 3:** Any subsequent call to an owner-gated function from the real owner fails:

```
near call aurora resume_contract '' --accountId aurora
# → ERR_NOT_ALLOWED
```

**Step 4:** If the contract was paused before or after this event, `resume_contract` can never succeed. All EVM `submit` calls are rejected by `require_running`. All bridged user funds are permanently frozen. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)

pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())
}

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
}
