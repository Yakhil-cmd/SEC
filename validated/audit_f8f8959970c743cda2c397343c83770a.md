### Title
Authorization Credentials Sent Over Plaintext HTTP in Bitcoin Checker Canister - (File: rs/bitcoin/checker/src/providers.rs)

### Summary
The `parse_authorization_header_from_url` function in the Bitcoin checker canister extracts credentials from a URL and constructs an `Authorization: Basic` HTTP header without validating that the target URL uses the `https://` scheme. This is a direct analog to KZG-02, where authorization tokens were transmitted over unencrypted HTTP.

### Finding Description
The function `parse_authorization_header_from_url` in `rs/bitcoin/checker/src/providers.rs` accepts any URL (including `http://`), extracts embedded username/password credentials, base64-encodes them, and places them into an `Authorization: Basic <credentials>` header. No HTTPS scheme check is performed at the canister level before the header is constructed and the `HttpRequestArgs` is submitted. [1](#0-0) 

This function is called by `make_post_request`, which is invoked for `BtcNetwork::Regtest { json_rpc_url }`: [2](#0-1) 

The test suite explicitly validates that `http://guest:pass@localhost:3030` is accepted and produces a valid `Authorization: Basic` header, confirming the function is designed to accept plaintext HTTP URLs: [3](#0-2) 

The HTTPS enforcement exists only at the adapter layer (`rs/https_outcalls/adapter/src/rpc_server.rs`) and is gated behind a compile-time `#[cfg(not(feature = "http"))]` flag: [4](#0-3) 

When the `http` feature is enabled (as it is in certain test and development deployments), this adapter-level check is entirely absent, and the request with the `Authorization: Basic` header is forwarded over plaintext HTTP.

### Impact Explanation
A network observer positioned between the IC node and the configured `json_rpc_url` endpoint can intercept the plaintext HTTP request and read the `Authorization: Basic` header. Since Basic auth credentials are only base64-encoded (not encrypted), they are trivially decoded to recover the username and password. An attacker with these credentials can authenticate to the Bitcoin JSON-RPC endpoint and issue arbitrary RPC calls, including those that could manipulate transaction data returned to the checker canister.

### Likelihood Explanation
Likelihood is **Low** in production mainnet because the HTTPS outcall adapter enforces HTTPS by default (without the `http` feature). However, likelihood is **Medium** in any deployment where the `http` feature flag is enabled, which is explicitly supported by the codebase and tested. The `BtcNetwork::Regtest` path is the entry point, and its `json_rpc_url` is canister-configuration-controlled. A canister developer or operator who configures an `http://` URL with embedded credentials in a non-production or development IC environment would expose those credentials in plaintext.

### Recommendation
Add an HTTPS scheme validation inside `parse_authorization_header_from_url` before constructing the `Authorization` header:

```rust
if url.scheme() != "https" {
    return Err(format!(
        "JSON RPC URL must use https scheme, got: {}", url.scheme()
    ));
}
```

This mirrors the defense-in-depth approach already applied at the adapter layer and ensures credentials are never placed into headers destined for plaintext HTTP connections, regardless of which feature flags are active.

### Proof of Concept
1. Configure the Bitcoin checker canister with `BtcNetwork::Regtest { json_rpc_url: "http://user:secret@attacker-controlled-host:8332" }`.
2. Build the adapter with the `http` feature enabled (bypassing the adapter-level HTTPS check at line 267–282 of `rpc_server.rs`).
3. Trigger a transaction check that calls `Provider::create_request` → `make_post_request` → `parse_authorization_header_from_url`.
4. Observe the outgoing HTTP request contains `Authorization: Basic dXNlcjpzZWNyZXQ=` (base64 of `user:secret`) in plaintext.
5. Decode: `echo dXNlcjpzZWNyZXQ= | base64 -d` → `user:secret`. [5](#0-4)

pub(crate) fn parse_authorization_header_from_url(
    json_rpc_url: &str,
) -> Result<(url::Url, HttpHeader), String> {
    let mut url = url::Url::parse(json_rpc_url).map_err(|err| err.to_string())?;
    let username = url.username();
    let password = url.password().unwrap_or_default();
    let authorization = base64::encode(format!(
        "{}:{}",
        url::form_urlencoded::parse(username.as_bytes())
            .next()
            .ok_or("Missing username or error in url_decode".to_string())?
            .0,
        url::form_urlencoded::parse(password.as_bytes())
            .next()
            .ok_or("Missing password or error in url_decode".to_string())?
            .0,
    ));
    url.set_username("")
        .map_err(|()| format!("Invalid JSON RPC URL {json_rpc_url}"))?;
    url.set_password(None)
        .map_err(|()| format!("Invalid JSON RPC URL {json_rpc_url}"))?;
    let header = HttpHeader {
        name: "Authorization".to_string(),
        value: format!("Basic {authorization}"),
    };
    Ok((url, header))
}
