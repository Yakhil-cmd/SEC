### Title
EVM Refund Counter Silently Suppressed by `minimal_gas_used` Floor in `compute_gas_refund` — (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` applies a `max(gas_used, minimal_gas_used)` floor **after** the EVM refund counter has already been subtracted from `gas_used`. In the Ethereum transaction flow, `minimal_gas_used` is unconditionally set to `TX_INTRINSIC_GAS` (21 000). When a transaction earns a large SSTORE-based EVM refund, the floor silently discards the refund: the user is overcharged and the operator receives excess fees. Standard Ethereum has no such floor; refunds can reduce final gas below the intrinsic cost.

---

### Finding Description

**Refund accumulation** — `refund_for_storage_write` in `basic_system/src/system_implementation/system/mod.rs` correctly accumulates EVM refunds per EIP-3529 semantics. Clearing a storage slot (original non-zero → zero) adds `4800 * ERGS_PER_GAS` to the refund counter: [1](#0-0) 

**Refund suppression** — `compute_gas_refund` in `refund_calculation.rs` follows this sequence:

1. `gas_used = gas_limit − remaining_ergs / ERGS_PER_GAS` (total gas consumed, intrinsic included).
2. `evm_refund = min(refund_counter, gas_used / 5)` — correct EIP-3529 cap.
3. `gas_used -= evm_refund` — refund applied.
4. **`gas_used = max(gas_used, minimal_gas_used)`** — floor applied, potentially erasing the refund. [2](#0-1) 

**Floor source in the Ethereum flow** — `validate_and_compute_fee_for_transaction` in `ethereum/validation_impl.rs` sets `minimal_gas_used = TX_INTRINSIC_GAS` (21 000) when EIP-7623 is disabled (the default): [3](#0-2) 

This value is stored in `EthereumTxContext.minimal_gas_to_charge` and passed directly to `compute_gas_refund` in `before_refund`: [4](#0-3) 

**Concrete example:**

| Step | Value |
|---|---|
| Intrinsic gas | 21 000 |
| Execution gas (one warm SSTORE clear) | 100 |
| `gas_used` before refund | 21 100 |
| `evm_refund` = min(4 800, 21 100/5) | 4 220 |
| `gas_used` after refund | 16 880 |
| Floor applied: max(16 880, 21 000) | **21 000** |
| User overcharged | **4 120 gas** |

The `RefundInfo.evm_refund` field still records 4 220, creating a discrepancy between the reported refund and the refund actually credited to the user: [5](#0-4) 

The operator is then paid for `gas_used = 21 000` instead of the correct `16 880`: [6](#0-5) 

---

### Impact Explanation

Any unprivileged user whose EVM transaction clears one or more storage slots with low execution gas relative to the intrinsic cost will be overcharged. The excess fee flows directly to the operator (coinbase). The maximum overcharge per transaction is bounded by `gas_used / 5 × gas_price`. For a 21 000-gas transaction at 1 Gwei this is ~4 200 gas × 1 Gwei = 4 200 Gwei per transaction. Across many transactions this is a systematic, silent transfer of funds from users to the operator. Standard Ethereum has no such floor; this is an EVM semantic mismatch and a resource accounting bug.

---

### Likelihood Explanation

Clearing storage slots is a routine operation: resetting ERC-20 allowances, zeroing state variables, deleting mappings. Any contract that does so with minimal surrounding execution gas will trigger the suppression. No special attacker capability is required — a normal user submitting a standard EVM transaction is sufficient. The condition `execution_gas < minimal_gas_used / 4` (≈ 5 250 for a simple call) is easily satisfied by contracts that do a single warm SSTORE clear.

---

### Recommendation

Apply the `minimal_gas_used` floor **before** computing the EVM refund, not after. The floor is semantically correct only as an EIP-7623 calldata floor (preventing underpriced calldata-heavy transactions); it should not be applied to the post-refund gas figure. Alternatively, remove the floor from the post-refund path entirely and rely solely on the EIP-3529 `gas_used / 5` cap.

---

### Proof of Concept

1. Deploy a contract that stores a non-zero value in slot 0 during construction.
2. Submit an EVM transaction calling a function that executes `SSTORE(0, 0)` (warm clear: 100 gas execution, 4 800 gas refund credit).
3. Observe `gas_used` reported = 21 000 (the floor), not ~16 880 (the correct post-refund value).
4. The user is charged for 21 000 gas instead of ~16 880 gas; the operator receives the ~4 120-gas surplus.
5. The `gas_refunded` field in the transaction result reports 4 220 (the computed refund), but the user's balance reflects only the 100-gas reduction — confirming the discrepancy between reported and actual refund.
