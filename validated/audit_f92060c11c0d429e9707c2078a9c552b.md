Audit Report

## Title
Unauthorized Third-Party Confirmation-Text Forgery and Forced Participation via `refresh_buyer_tokens` Caller/Buyer Mismatch — (`rs/sns/swap/canister/canister.rs`)

## Summary
The `refresh_buyer_tokens` endpoint resolves the effective buyer from the caller-supplied `arg.buyer` field without verifying it matches the authenticated ingress sender. When a swap is configured with a `confirmation_text`, any unprivileged caller who knows the victim's principal and the publicly readable confirmation text can forge the victim's acceptance of the swap's terms of service. Additionally, once a victim's ICP is force-registered into `self.buyers`, the victim loses the ability to recover their ICP via `error_refund_icp` if the swap commits, because the funds are held in escrow until `sweep_icp` runs and then transferred to SNS governance.

## Finding Description

**Root cause — no caller/buyer binding in `refresh_buyer_tokens`:**

In `rs/sns/swap/canister/canister.rs` L127–143, the effective buyer principal is taken directly from `arg.buyer` with no check that it equals `caller_principal_id()`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()   // any principal accepted
};
```

The inner `refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs` L1134–1163) then:
1. Calls `validate_confirmation_text` (L1150) — checks only that the supplied text equals the swap-configured text; it is not bound to the authenticated caller.
2. Reads the ICP ledger balance of `buyer`'s subaccount (L1153–1163).
3. Permanently records the buyer in `self.buyers` (L1285–1288).

**Ticket system does not block the attack:**

The ticket check at L1250–1272 is explicitly skipped when no ticket exists for the buyer: *"If there exists no ticket for the buyer, the payment flow will simply ignore the ticket."* An attacker calling on behalf of a victim who has not created a ticket bypasses this check entirely.

**`error_refund_icp` is blocked once the buyer is registered:**

In `rs/sns/swap/src/swap.rs` L1950–1959, `error_refund_icp` returns a precondition error for any buyer present in `self.buyers` whose `transfer_success_timestamp_seconds == 0` (i.e., not yet swept):

```rust
if let Some(transfer) = &buyer_state.icp
    && transfer.transfer_success_timestamp_seconds == 0
{
    return ErrorRefundIcpResponse::new_precondition_error(format!(
        "ICP cannot be refunded as principal {} has {} ICP (e8s) in escrow", ...
    ));
}
```

Once `sweep_icp` runs on a committed swap, the victim's ICP is transferred to SNS governance — it cannot be recovered.

**Third-party calling without `confirmation_text` is intentional design** (confirmed by the integration test at `rs/tests/nns/sns/lib/src/sns_deployment.rs` L755–756, which explicitly tests and expects a third party to successfully call `refresh_buyer_tokens` for another user after ICP is transferred). However, the proto documentation at `rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto` L847–850 states *"a participant should send the confirmation text"*, making clear that supplying `confirmation_text` on behalf of another principal is not the intended design.

## Impact Explanation

**Confirmation-text forgery (High):** When `confirmation_text` is configured, an attacker can permanently record any victim as having accepted the swap's legal terms without their knowledge or consent. This is a significant SNS governance/participation integrity impact with concrete user harm.

**Forced financial participation (High):** A victim who transferred ICP to the swap subaccount by mistake (or who changed their mind before calling `refresh_buyer_tokens`) loses the ability to recover their ICP via `error_refund_icp` once an attacker force-registers their participation. If the swap commits, `sweep_icp` transfers the victim's ICP to SNS governance; the victim receives SNS tokens instead of recovering their ICP. Without the attack, the victim (not present in `self.buyers`) could call `error_refund_icp` post-commit to recover their funds.

This matches the allowed High impact: *"Significant SNS security impact with concrete user or protocol harm"* and *"Unauthorized access to canister-controlled funds where exploitation requires meaningful per-target work or other constraints."*

## Likelihood Explanation

- The endpoint is publicly reachable by any unprivileged ingress sender.
- The only precondition is that the victim has a non-zero ICP balance in their swap subaccount (observable on-chain).
- The `confirmation_text` is readable from public swap state — no secret knowledge is required.
- The attack window spans the entire `Open` lifecycle (potentially days to weeks).
- A motivated attacker (e.g., a competing participant wanting to push a borderline swap over its minimum commitment threshold) has clear incentive.

## Recommendation

Enforce that when `confirmation_text` is supplied, the resolved buyer principal must equal the authenticated caller:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let requested = PrincipalId::from_str(&arg.buyer).unwrap();
        if arg.confirmation_text.is_some() && requested != caller {
            panic!("confirmation_text may only be supplied by the buyer themselves");
        }
        requested
    };
    ...
}
```

This preserves the intentional third-party notification path (no `confirmation_text`) while preventing consent forgery.

## Proof of Concept

1. Deploy an SNS swap with `confirmation_text = "I agree to the SNS terms"` in `Open` state.
2. Victim Alice transfers 10 ICP to `swap_canister_subaccount(alice_principal)` on the ICP ledger but does not call `refresh_buyer_tokens`.
3. Attacker Eve (any principal) submits:
   ```
   refresh_buyer_tokens({
     buyer: alice_principal.to_text(),
     confirmation_text: Some("I agree to the SNS terms")
   })
   ```
4. The swap canister resolves `p = alice_principal`, reads Alice's 10 ICP balance, validates the confirmation text (matches), and records Alice in `self.buyers`.
5. The swap reaches its end time and commits.
6. Alice calls `error_refund_icp` → returns precondition error: *"ICP cannot be refunded as principal alice has N ICP (e8s) in escrow"*.
7. `sweep_icp` runs, transferring Alice's ICP to SNS governance. Alice receives SNS tokens; her ICP is unrecoverable.
8. Without the attack, Alice (absent from `self.buyers`) could have called `error_refund_icp` post-commit to recover her ICP.

A deterministic unit test can be written against `Swap::refresh_buyer_token_e8s` and `Swap::error_refund_icp` using `MockLedger`, following the pattern in `rs/sns/swap/tests/common/mod.rs` `buy_token` / `try_error_refund_err`, with a separate caller principal and a non-empty `confirmation_text`.
