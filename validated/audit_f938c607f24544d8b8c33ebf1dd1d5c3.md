### Title
Caller-Supplied `wnear_account_id` in `fund_xcc_sub_account` Initializes XCC Router with Unvalidated wNEAR Dependency — (`engine/src/xcc.rs`)

---

### Summary

The `fund_xcc_sub_account` function in `engine/src/xcc.rs` accepts a caller-supplied `wnear_account_id` that is passed directly into the XCC router's `initialize` call without any validation that it is the authentic wNEAR NEAR contract. Because the engine tracks deployed router versions (acting as a factory), a malicious actor can front-run the deployment of a victim's XCC router sub-account with a malicious wNEAR NEAR account. The engine's version-tracking then marks the router as up-to-date, preventing redeployment. When the victim subsequently uses XCC with NEAR payment, the router interacts with the malicious wNEAR contract, enabling theft or permanent freezing of the victim's NEAR.

---

### Finding Description

In `engine/src/xcc.rs`, `fund_xcc_sub_account` accepts `FundXccArgs` which contains an optional `wnear_account_id`:

```rust
let wnear_account = if let Some(wnear_account) = args.wnear_account_id {
    wnear_account   // ← used directly, no authenticity check
} else {
    let wnear_address = get_wnear_address(io);
    crate::engine::nep141_erc20_map(*io)
        .lookup_right(&crate::engine::ERC20Address(wnear_address))
        .ok_or(FundXccError::MissingWNearAddress)?
        .0
};
``` [1](#0-0) 

This `wnear_account` is then embedded into the router's `initialize` call arguments:

```rust
let init_args = format!(
    r#"{{"wnear_account": "{}", "must_register": {}}}"#,
    wnear_account.as_ref(),
    create_needed,
);
``` [2](#0-1) 

The router is then deployed and the engine records the version for `args.target`:

```rust
let args = AddressVersionUpdateArgs {
    address: args.target,
    version: latest_code_version,
};
let callback = PromiseCreateArgs {
    target_account_id: current_account_id,
    method: "factory_update_address_version".into(),
    ...
};
``` [3](#0-2) 

Once the engine records the router version for `args.target`, the `AddressVersionStatus::new` check in both `fund_xcc_sub_account` and `handle_precompile_promise` will return `UpToDate`, preventing any redeployment:

```rust
Some(version) if version < latest_code_version => Self::DeployNeeded { create_needed: false },
Some(_version) => Self::UpToDate,
``` [4](#0-3) 

By contrast, the `handle_precompile_promise` path (triggered by the XCC precompile during EVM execution) always derives the wNEAR account from the engine's internal NEP-141/ERC-20 mapping — never from user input:

```rust
let wnear_account = crate::engine::nep141_erc20_map(*io)
    .lookup_right(&crate::engine::ERC20Address(wnear_address))
    .expect("wnear account not found");
``` [5](#0-4) 

But this safe path is only reached when the router does **not** already exist. If the attacker pre-deploys the victim's router via `fund_xcc_sub_account` with a malicious `wnear_account_id`, the `handle_precompile_promise` path skips redeployment entirely and the malicious wNEAR persists in the router's state.

---

### Impact Explanation

When the victim later uses the XCC precompile with a NEAR payment, the engine calls `withdraw_wnear_to_router`, which sends wNEAR to the victim's router sub-account. The router then calls its stored `wnear_account` to unwrap the wNEAR into NEAR. Because the router was initialized with a malicious NEAR account, the malicious contract receives the unwrap call and can:

- **Steal** the NEAR by not forwarding it to the intended recipient.
- **Permanently freeze** the NEAR by reverting or silently dropping the transfer.

This constitutes **direct theft or permanent freezing of user funds** — Critical impact.

---

### Likelihood Explanation

- `fund_xcc_sub_account` is a public NEAR contract method (it consumes `env.attached_deposit()` and accepts an arbitrary `target` EVM address), callable by any unprivileged NEAR account.
- The `args.target` field allows specifying any EVM address, enabling an attacker to front-run any victim who has not yet deployed their router.
- The attack requires only a small NEAR deposit (`STORAGE_AMOUNT = 2 NEAR`) and knowledge of the victim's EVM address, both of which are trivially obtainable on-chain.
- Once the malicious router is deployed and the version is recorded, there is no recovery path for the victim without admin intervention.

---

### Recommendation

In `fund_xcc_sub_account`, when `args.wnear_account_id` is `Some(...)`, validate that the provided account ID matches the engine's canonical wNEAR NEAR account (derived from the internal NEP-141/ERC-20 mapping, the same source used by `handle_precompile_promise`). If they do not match, return an error. Alternatively, remove the `wnear_account_id` field from `FundXccArgs` entirely and always derive the wNEAR account from the engine's internal mapping, as `handle_precompile_promise` already does.

---

### Proof of Concept

1. Victim `0xALICE` has never used XCC; no router exists for her address.
2. Attacker deploys a malicious NEAR contract `evil-wnear.near` that accepts `near_withdraw` calls but does not forward NEAR to the caller.
3. Attacker calls the engine's `fund_xcc_sub_account` NEAR method with:
   - `target = 0xALICE`
   - `wnear_account_id = Some("evil-wnear.near")`
   - Attached deposit ≥ `STORAGE_AMOUNT` (2 NEAR)
4. The engine deploys `{alice_hex}.aurora` with `initialize({"wnear_account": "evil-wnear.near", ...})` and records version `v` for Alice's address.
5. Alice calls the XCC precompile from her EVM contract, attaching wNEAR for NEAR payment.
6. `handle_precompile_promise` sees Alice's router is at version `v` (up-to-date) and skips redeployment.
7. `withdraw_wnear_to_router` sends wNEAR to Alice's router.
8. Alice's router calls `near_withdraw` on `evil-wnear.near`.
9. `evil-wnear.near` absorbs the NEAR without forwarding it — Alice's NEAR is stolen/frozen. [6](#0-5)

```rust
pub fn fund_xcc_sub_account<I, P, E>(
    io: &I,
    handler: &mut P,
    env: &E,
    args: FundXccArgs,
) -> Result<(), FundXccError>
where
    P: PromiseHandler,
    I: IO + Copy,
    E: Env,
{
    let current_account_id = env.current_account_id();
    let target_account_id = AccountId::try_from(format!(
        "{}.{}",
        args.target.encode(),
        current_account_id.as_ref()
    ))?;

    let latest_code_version = get_latest_code_version(io);
    let target_code_version = get_code_version_of_address(io, &args.target);
    let deploy_needed = AddressVersionStatus::new(io, latest_code_version, target_code_version);

    let fund_amount = Yocto::new(env.attached_deposit());

    let mut promise_actions = Vec::with_capacity(4);

    // If account needs to be created and/or updated then include those actions.
    if let AddressVersionStatus::DeployNeeded { create_needed } = deploy_needed {
        let code = get_router_code(io).0.into_owned();
        // wnear_account is needed for initialization so we must assume it is set
        // in the Engine, or we need to accept it as input.
        let wnear_account = if let Some(wnear_account) = args.wnear_account_id {
            wnear_account
        } else {
            // If the wnear account is not specified then we must look it up based on the
            // bridged token registry for the engine.
            let wnear_address = get_wnear_address(io);
            crate::engine::nep141_erc20_map(*io)
                .lookup_right(&crate::engine::ERC20Address(wnear_address))
                .ok_or(FundXccError::MissingWNearAddress)?
                .0
        };
        let init_args = format!(
            r#"{{"wnear_account": "{}", "must_register": {}}}"#,
            wnear_account.as_ref(),
            create_needed,
        );
        if create_needed {
            if fund_amount < STORAGE_AMOUNT {
                return Err(FundXccError::InsufficientBalance);
            }

            promise_actions.push(PromiseAction::CreateAccount);
            promise_actions.push(PromiseAction::Transfer {
                amount: fund_amount,
            });
            promise_actions.push(PromiseAction::DeployContract { code });
            promise_actions.push(PromiseAction::FunctionCall {
                name: "initialize".into(),
                args: init_args.into_bytes(),
                attached_yocto: ZERO_YOCTO,
                gas: INITIALIZE_GAS,
            });
        } else {
            let deploy_args = DeployUpgradeParams {
                code,
                initialize_args: init_args.into_bytes(),
            };
            promise_actions.push(PromiseAction::FunctionCall {
                name: "deploy_upgrade".into(),
                args: borsh::to_vec(&deploy_args).expect(ERR_UPGRADE_ARG_SERIALIZATION),
                attached_yocto: fund_amount,
                gas: UPGRADE_GAS + INITIALIZE_GAS,
            });
        }
    } else {
        // No matter what include the transfer of the funding amount
        promise_actions.push(PromiseAction::Transfer {
            amount: fund_amount,
        });
    }

    let batch = PromiseBatchAction {
        target_account_id,
        actions: promise_actions,
    };
    // Safety: same as safety in `handle_precompile_promise`
    let promise_id = handler.promise_create_batch(&batch);

    if let AddressVersionStatus::DeployNeeded { .. } = deploy_needed {
        // If a creation and/or deployment were needed, then we must attach a callback to update
        // the Engine's record of the account.

        let args = AddressVersionUpdateArgs {
            address: args.target,
            version: latest_code_version,
        };
        let callback = PromiseCreateArgs {
            target_account_id: current_account_id,
            method: "factory_update_address_version".into(),
            args: borsh::to_vec(&args).map_err(|_| FundXccError::SerializationFailure)?,
            attached_balance: ZERO_YOCTO,
            attached_gas: VERSION_UPDATE_GAS,
        };
        let _promise_id = handler.promise_attach_callback(promise_id, &callback);
    }

    Ok(())
}
