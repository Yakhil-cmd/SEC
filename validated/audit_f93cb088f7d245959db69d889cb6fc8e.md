### Title
Truncating Integer Division in `native_per_pubdata` Yields Zero, Making Pubdata Free — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` is computed with floor (integer) division `pubdata_price.wrapping_div(native_price)`. When `pubdata_price < native_price`, the result truncates to `0`. A `native_per_pubdata` of `0` means every pubdata byte costs zero native resources, so an unprivileged sender can write unbounded pubdata in a single transaction without consuming any native budget for it.

---

### Finding Description

In `validation_impl.rs` the two conversion ratios are computed back-to-back:

```rust
// line 135 — uses ceiling division ✓
u256_try_to_u64(&gas_price.div_ceil(native_price))

// line 142 — uses floor division ✗
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`native_per_gas` correctly uses `div_ceil` so that a non-zero gas price always produces a non-zero ratio. `native_per_pubdata` uses `wrapping_div` (floor division) and has no analogous guard. Whenever `pubdata_price < native_price` the quotient truncates to `0`.

`native_per_pubdata` is then threaded through the entire resource pipeline:

- `create_resources_for_tx` charges `native_per_pubdata_byte * intrinsic_pubdata` for intrinsic pubdata overhead — product is `0`.
- `get_resources_to_charge_for_pubdata` charges `current_pubdata_spent * native_per_pubdata` — product is `0`.
- `check_enough_resources_for_pubdata` always reports "enough" because the required resources are `0`.

The same floor-division defect is replicated in the off-chain helper `api/src/helpers.rs` line 427, which mirrors the bootloader logic for pre-flight validation. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

`native_per_pubdata = 0` disables the entire pubdata-cost dimension of native resource accounting. A sender can issue a transaction that writes the maximum number of new storage slots (each slot write emits pubdata) and pay only the EVM gas cost for those `SSTORE` operations — zero native resources are charged for the resulting pubdata. Because pubdata is ultimately published to L1, the protocol bears the L1 data-availability cost while the sender pays nothing for it. This is a direct funds-loss path for the protocol: the operator/sequencer subsidises L1 publication costs that should have been collected from the sender. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a normal operating state, not an edge case. `native_price` reflects the cost of a single RISC-V proving cycle; `pubdata_price` reflects the cost of one pubdata byte. These are measured in different units and their relative magnitude is determined by the operator's configuration. Any deployment where the operator sets `pubdata_price` to a value smaller than `native_price` (e.g., `pubdata_price = 500`, `native_price = 1000`) silently zeroes out pubdata charging for every L2 transaction. No special privilege is required from the attacker — any standard transaction sender can exploit the resulting `native_per_pubdata = 0` condition. [6](#0-5) 

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, exactly as is already done for `native_per_gas`:

```rust
// Before (floor division — can produce 0)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — mirrors native_per_gas treatment)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix to `api/src/helpers.rs` line 427. Optionally add an explicit guard that rejects or saturates to `1` when `pubdata_price > 0` but the quotient would be `0`, to make the invariant explicit. [7](#0-6) [3](#0-2)

### Proof of Concept

**Setup:** operator sets `native_price = 1000`, `pubdata_price = 999`.

**Step 1 — `native_per_pubdata` truncates to 0:**
```
native_per_pubdata = 999 / 1000 = 0   (integer division)
```

**Step 2 — intrinsic pubdata overhead is 0:**
```rust
// gas_helpers.rs line 352
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// = 0 * L2_TX_INTRINSIC_PUBDATA = 0
```

**Step 3 — execution pubdata charge is 0 regardless of bytes written:**
```rust
// gas_helpers.rs line 430-432
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // = N * 0 = 0
    .ok_or(out_of_native_resources!())?;
```

**Step 4 — attacker submits a transaction that writes 1000 new storage slots.** Each new slot emits ~65 bytes of pubdata → ~65 000 pubdata bytes. With `native_per_pubdata = 0`, the native cost for this pubdata is `0`. The attacker pays only the EVM gas for 1000 `SSTORE` operations (~20 000 gas each = 20 M gas), but the protocol must publish 65 000 bytes to L1 at its own expense.

**Contrast with correct behaviour:** with `div_ceil`, `native_per_pubdata = ceil(999/1000) = 1`, and the 65 000 pubdata bytes would cost `65 000` native units, correctly charging the sender. [7](#0-6) [8](#0-7) [9](#0-8)
