### Title
Spurious `ReportedToExecution` Entry Bypasses `validate_reshare_dealings`, Permanently Suppressing XNet Reshare Requests — (`rs/consensus/idkg/src/payload_verifier.rs`)

### Summary

`validate_reshare_dealings` silently skips all `CompletedReshareRequest::ReportedToExecution` entries in `curr_payload.xnet_reshare_agreements`. A Byzantine block proposer can inject a fabricated `ReportedToExecution` entry for any request that has an active `idkg_dealings_context` in execution state, causing the entry to survive the honest proposer's cleanup pass and permanently block `initiate_reshare_requests` from ever initiating the legitimate reshare.

### Finding Description

The validation loop in `validate_reshare_dealings` is:

```rust
for (request, dealings) in curr_payload.xnet_reshare_agreements.iter() {
    if let idkg::CompletedReshareRequest::Unreported(dealings) = &dealings
        && !prev_payload.xnet_reshare_agreements.contains_key(request)
    {
        // only Unreported+new entries are validated
    }
}
``` [1](#0-0) 

The `if let` pattern only matches `Unreported`. A `ReportedToExecution` variant falls through with zero validation — no check that it was present in `prev_payload.xnet_reshare_agreements`, no check that it was ever in `prev_payload.ongoing_xnet_reshares`. A Byzantine proposer can freely insert any `ReportedToExecution` entry for any request key.

The second loop in the same function enforces that requests disappearing from `ongoing_xnet_reshares` must have a corresponding `Unreported` agreement: [2](#0-1) 

This only guards requests that were already in `prev_payload.ongoing_xnet_reshares`. A fabricated entry for a request that was never there is invisible to both loops.

Once the malicious block is finalized, the honest block proposer's `update_completed_reshare_requests` runs its cleanup:

```rust
payload.xnet_reshare_agreements = payload
    .xnet_reshare_agreements
    .keys()
    .filter(|&request| {
        idkg_dealings_contexts.values()
            .any(|context| *request == reshare_request_from_dealings_context(context))
    })
    .cloned()
    .map(|request| (request, idkg::CompletedReshareRequest::ReportedToExecution))
    .collect();
``` [3](#0-2) 

If the fabricated request matches an active `idkg_dealings_context` (which the Byzantine node can observe from replicated state), the spurious entry **survives** the cleanup and is re-emitted as `ReportedToExecution`.

Subsequently, `initiate_reshare_requests` deduplicates against `xnet_reshare_agreements`:

```rust
if payload.ongoing_xnet_reshares.contains_key(&request)
    || payload.xnet_reshare_agreements.contains_key(&request)
{
    continue;
}
``` [4](#0-3) 

The legitimate reshare is never initiated. Because the context remains in execution state (waiting for a response that never comes), the spurious entry is never purged, making the suppression permanent.

### Impact Explanation

The target subnet never receives the reshared key material. Any chain-fusion operation (e.g., threshold ECDSA/Schnorr signing for Bitcoin/Ethereum) that depends on that reshare is permanently blocked on the target subnet. This is a targeted, persistent denial-of-service against cross-subnet key distribution, not a direct theft — but it renders chain-fusion assets on the target subnet inaccessible.

### Likelihood Explanation

- The attacker only needs to be a subnet node selected as block proposer (no privileged role required; block proposer selection is round-robin/random among all subnet nodes).
- The active `idkg_dealings_contexts` are part of replicated state visible to all subnet nodes.
- The attack requires crafting a single malicious block payload; it is deterministic and requires no threshold corruption.
- Other nodes will notarize the block because `validate_reshare_dealings` passes cleanly.

### Recommendation

In `validate_reshare_dealings`, add an explicit check that every `ReportedToExecution` entry in `curr_payload.xnet_reshare_agreements` was already present in `prev_payload.xnet_reshare_agreements`. Any `ReportedToExecution` entry that is new (absent from `prev_payload`) must be rejected with an error analogous to `XNetReshareAgreementWithoutRequest`. [1](#0-0) 

### Proof of Concept

1. Observe an active `idkg_dealings_context` for request `R` in execution state; confirm `R` is absent from `prev_payload.ongoing_xnet_reshares` and `prev_payload.xnet_reshare_agreements`.
2. Construct `curr_payload` identical to `prev_payload` except `curr_payload.xnet_reshare_agreements` contains `(R, CompletedReshareRequest::ReportedToExecution)`.
3. Call `validate_reshare_dealings(crypto, block_reader, &prev_payload, &curr_payload)`.
4. Assert: the function returns `Ok(...)` — no error is raised for the spurious entry.
5. Finalize the block; observe that subsequent honest blocks produced by `update_completed_reshare_requests` + `initiate_reshare_requests` never add `R` to `ongoing_xnet_reshares`.
