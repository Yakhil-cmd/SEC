Audit Report

## Title
Unbounded Tokio Task and Memory Growth via `SLOT_TABLE_NO_LIMIT` in P2P Consensus Manager Receiver — (`rs/p2p/consensus_manager/src/receiver.rs`)

## Summary

All non-ingress artifact channels (consensus, certifier, DKG, IDKG, HTTPS outcalls) are configured with `slot_limit = usize::MAX`, rendering the per-peer slot table guard permanently ineffective. A single Byzantine subnet peer can send an unbounded number of slot updates with distinct slot numbers and distinct artifact IDs, causing unbounded growth of `slot_table`, `active_assembles`, and `artifact_processor_tasks`, exhausting heap memory and Tokio scheduler capacity on honest replicas.

## Finding Description

**Root cause — `SLOT_TABLE_NO_LIMIT` assignment:**

In `rs/replica/setup_ic_network/src/lib.rs`, `SLOT_TABLE_NO_LIMIT = usize::MAX` is defined and passed as `slot_limit` to every non-ingress channel: [1](#0-0) [2](#0-1) [3](#0-2) 

Only ingress uses the finite `SLOT_TABLE_LIMIT_INGRESS = 50_000`: [4](#0-3) 

**Guard rendered ineffective:**

In `handle_slot_update_receive`, the guard is: [5](#0-4) 

`peer_slot_table_len` is a `usize` counter. The condition `peer_slot_table_len < usize::MAX` is always true — a `usize` counter cannot reach `usize::MAX` before the process runs out of memory. The `Entry::Vacant(_)` drop branch at line 381 is therefore unreachable for any non-ingress channel.

**Unbounded task spawning:**

Every new `(peer_id, slot_number)` pair with a previously-unseen artifact ID that passes the guard causes a new entry in `slot_table[peer_id]`, a new entry in `active_assembles`, and a new Tokio task spawned in `artifact_processor_tasks`: [6](#0-5) 

**Tasks do not self-terminate for fake IDs:**

Each spawned `process_slot_update` task terminates only on: (1) cancellation token, (2) `assemble_message` completion, or (3) `all_peers_deleted_artifact` (peer counter goes to zero): [7](#0-6) 

The `all_peers_deleted_artifact` branch fires only when `sender.send_if_modified(|h| h.remove(peer_id))` is called, which only happens in the `to_remove` path triggered by slot overwrite: [8](#0-7) 

A Byzantine peer using monotonically increasing slot numbers never overwrites any slot, so `to_remove` is always `None`, the peer counter for each artifact ID stays non-empty indefinitely, and all N spawned tasks persist until replica shutdown.

**Topology removal does not help:**

`handle_topology_update` removes nodes leaving the subnet topology, but a Byzantine peer that remains a subnet member is never removed: [9](#0-8) 

## Impact Explanation

A single Byzantine subnet peer (below the f < n/3 fault threshold) can grow `active_assembles` and `artifact_processor_tasks` to an arbitrary size, exhausting process heap memory and Tokio task scheduler capacity. This causes honest replicas to crash or become unresponsive, halting subnet progress. This matches the allowed High impact: **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"** ($2,000–$10,000).

## Likelihood Explanation

The attack requires only subnet membership — any Byzantine node can send `SlotUpdate` protobuf messages over the existing QUIC transport connection. No special privileges, leaked keys, or governance majority are required. The attack is repeatable, low-cost, and straightforward: send N messages with distinct `slot_id` values and distinct artifact IDs. The `update_handler` decodes and enqueues them without any rate limiting at the application layer. The event loop processes `handle_slot_update_receive` synchronously and quickly, so the attacker can sustain the attack at the channel's processing rate.

## Recommendation

Apply a finite, protocol-appropriate per-peer slot limit to all artifact channels, not just ingress. The existing guard infrastructure in `handle_slot_update_receive` is correct; only the limit value needs to change. `SLOT_TABLE_NO_LIMIT` should be removed or restricted to contexts where it is genuinely safe. For consensus, DKG, IDKG, certifier, and HTTPS outcalls, the natural protocol bounds on concurrent artifacts per peer should inform the limit (e.g., a few thousand at most, similar to the ingress limit of 50,000 or tighter based on protocol semantics).

## Proof of Concept

Using the existing test infrastructure in `rs/p2p/consensus_manager/src/receiver.rs`:

```rust
#[tokio::test]
async fn unbounded_slot_growth_with_no_limit() {
    // Default builder uses slot_limit = usize::MAX (line 662)
    let (mut mgr, _channels) = ReceiverManagerBuilder::new().build();
    let cancellation = CancellationToken::new();
    const N: u64 = 100_000;
    for i in 0..N {
        mgr.handle_slot_update_receive(
            SlotUpdate {
                slot_number: SlotNumber::from(i),
                commit_id: CommitId::from(i),
                update: Update::Id(i as usize),
            },
            NODE_1,
            ConnId::from(1),
            cancellation.clone(),
        );
    }
    // All N entries accepted — guard never fired
    assert_eq!(mgr.slot_table.get(&NODE_1).unwrap().len(), N as usize);
    assert_eq!(mgr.active_assembles.len(), N as usize);
    assert_eq!(mgr.artifact_processor_tasks.len(), N as usize);
}
```

The existing `slot_table_limit_exceeded` test at line 846 confirms the guard works correctly with a finite limit of 2: [10](#0-9) 

The same test with `ReceiverManagerBuilder::new()` (no `.with_slot_limit(2)`) would show unbounded growth, confirming the vulnerability.
