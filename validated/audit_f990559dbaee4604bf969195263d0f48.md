### Title
Sequential Nonce Blocking in ckETH Minter Resubmission Pipeline Permanently Stalls All Pending Withdrawals - (File: `rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

---

### Summary

The ckETH minter's transaction resubmission pipeline uses Ethereum's sequential nonce system. When a sent transaction at nonce N cannot be resubmitted because its withdrawal amount no longer covers the spiked gas fee, `create_resubmit_transactions` immediately returns early and the caller `resubmit_transactions_batch` only logs the error. The stuck transaction is never removed from `sent_tx`, and because Ethereum enforces sequential nonces, every transaction at nonces N+1, N+2, … is also permanently stalled. There is no on-chain recovery path.

---

### Finding Description

The `EthTransactions` state machine in `rs/ethereum/cketh/minter/src/state/transactions/mod.rs` processes withdrawal requests in strict FIFO order, assigning each a monotonically increasing `TransactionNonce`. [1](#0-0) 

When a transaction is created, `next_nonce` is incremented immediately: [2](#0-1) 

During the periodic resubmission cycle, `create_resubmit_transactions` iterates over all unconfirmed sent transactions in nonce order. If the transaction at nonce N cannot be resubmitted because the new gas fee exceeds the withdrawal amount's budget, the function **immediately returns**, explicitly stopping all further resubmission attempts: [3](#0-2) 

The caller `resubmit_transactions_batch` in `rs/ethereum/cketh/minter/src/withdraw.rs` receives this early-return result and only logs the error — no recovery, no cancellation, no nonce skip: [4](#0-3) 

The stuck transaction remains in `sent_tx` indefinitely. On every subsequent timer tick, `create_resubmit_transactions` hits the same nonce N, fails again, and returns early again. Because Ethereum enforces sequential nonces, every transaction at nonces N+1, N+2, … is also unmined and cannot be replaced. New withdrawal requests that arrive after the stuck one receive nonces > N and are equally blocked on Ethereum.

The code itself acknowledges the blocking property in its comment: [5](#0-4) 

There is no mechanism to cancel the stuck transaction, reimburse the user, and free the nonce slot so that subsequent withdrawals can proceed.

---

### Impact Explanation

All pending ckETH and ckERC20 withdrawal requests whose nonces are greater than the stuck nonce are permanently frozen until Ethereum gas fees drop below the stuck transaction's fee cap. Users who submitted withdrawals after the stuck one cannot receive their funds. The minter's `pending_withdrawal_requests` queue continues to accept new requests (users burn ckETH/ckERC20 tokens), but none of those requests can be finalized. This constitutes a **ledger conservation bug**: tokens are burned on the IC ledger but the corresponding ETH/ERC-20 is never delivered. [6](#0-5) 

---

### Likelihood Explanation

Any withdrawal submitted with an amount close to the minimum threshold is at risk during a gas fee spike. Ethereum gas fees can increase by orders of magnitude during network congestion events. The minimum withdrawal amount is set conservatively to handle moderate fee increases, but extreme spikes (as seen historically during major NFT mints or DeFi events) can exceed the buffer. No privileged access or external oracle manipulation is required — the condition arises from normal protocol operation. An unprivileged user calling `withdraw_eth` with a near-minimum amount is the only prerequisite. [7](#0-6) 

---

### Recommendation

1. **Skip-and-continue instead of early return**: When a transaction at nonce N fails resubmission due to `InsufficientTransactionFee`, do not stop the loop. Log the error and continue attempting to resubmit transactions at nonces N+1, N+2, … (they may still be resubmittable if their fee budgets are larger). The Ethereum mempool will hold them pending nonce N's resolution.

2. **Introduce a stuck-transaction cancellation path**: Add a governance-controlled or time-triggered mechanism that, after a configurable timeout, reimburses the user for the stuck withdrawal (minting back the burned ckETH/ckERC20) and removes the stuck entry from `sent_tx`, freeing the nonce slot.

3. **Expose a `cancel_withdrawal` endpoint**: Allow the original requester to cancel a pending withdrawal that has been stuck for a configurable period, triggering reimbursement and nonce release.

---

### Proof of Concept

1. Alice calls `withdraw_eth` with `withdrawal_amount = MINIMUM_WITHDRAWAL_AMOUNT + ε`. Her request is enqueued and assigned nonce 5. A transaction is created and sent to Ethereum.

2. Bob calls `withdraw_eth` with a large amount. His request is assigned nonce 6.

3. Ethereum gas fees spike dramatically (e.g., 10×). Alice's transaction at nonce 5 cannot be resubmitted because `new_tx_price.max_transaction_fee() > alice.withdrawal_amount`.

4. `create_resubmit_transactions` hits nonce 5, receives `InsufficientTransactionFee`, pushes the error, and **returns early** at line 630.

5. Bob's transaction at nonce 6 is never resubmitted. Ethereum will not mine nonce 6 until nonce 5 is resolved.

6. On every subsequent timer tick, step 4 repeats. Bob's withdrawal — and every withdrawal submitted after Alice's — is permanently frozen. [8](#0-7) [9](#0-8)

    /// Create transactions to resubmit corresponding to already sent transactions
    /// with nonces greater than the latest mined transaction nonce:
    /// * the resubmitted transaction will need to be re-signed if its transaction fee was increased
    /// * the resubmitted transaction can be resent as is if its transaction fee was not increased
    ///
    /// We stop on the first error since if a transaction with nonce n could not be resubmitted
    /// (e.g., the transaction amount does not cover the new fees),
    /// then the next transactions with nonces n+1, n+2, ... are blocked anyway
    /// and trying to resubmit them would only artificially increase their transaction fees.
    pub fn create_resubmit_transactions(
        &self,
        latest_transaction_count: TransactionCount,
        current_gas_fee: GasFeeEstimate,
    ) -> Vec<Result<(LedgerBurnIndex, Eip1559TransactionRequest), ResubmitTransactionError>> {
        // If transaction count at block height H is c > 0, then transactions with nonces
        // 0, 1, ..., c - 1 were mined. If transaction count is 0, then no transactions were mined.
        // The nonce of the first pending transaction is then exactly c.
        let first_pending_tx_nonce: TransactionNonce = latest_transaction_count.change_units();
        let mut transactions_to_resubmit = Vec::new();
        for (nonce, burn_index, signed_tx) in self
            .sent_tx
            .iter()
            .filter(|(nonce, _burn_index, _signed_tx)| *nonce >= &first_pending_tx_nonce)
        {
            let last_signed_tx = signed_tx.last().expect("BUG: empty sent transactions list");
            match last_signed_tx.resubmit(current_gas_fee.clone()) {
                Ok(Some(new_tx)) => {
                    transactions_to_resubmit.push(Ok((*burn_index, new_tx)));
                }
                Ok(None) => {
                    // the transaction fee is still up-to-date but because the transaction did not get included,
                    // we re-send it as is to be sure that it remains known to the mempool and hopefully be included at some point.
                    // Since we always re-send the last non-included transactions in sent_tx, there is nothing to do.
                }
                Err(crate::tx::ResubmitTransactionError::InsufficientTransactionFee {
                    allowed_max_transaction_fee,
                    actual_max_transaction_fee,
                }) => {
                    transactions_to_resubmit.push(Err(
                        ResubmitTransactionError::InsufficientTransactionFee {
                            ledger_burn_index: *burn_index,
                            transaction_nonce: *nonce,
                            allowed_max_transaction_fee,
                            max_transaction_fee: actual_max_transaction_fee,
                        },
                    ));
                    return transactions_to_resubmit;
                }

fn create_transactions_batch(gas_fee_estimate: GasFeeEstimate) {
    for request in read_state(|s| {
        s.eth_transactions
            .withdrawal_requests_batch(WITHDRAWAL_REQUESTS_BATCH_SIZE)
    }) {
        log!(DEBUG, "[create_transactions_batch]: processing {request:?}",);
        let ethereum_network = read_state(State::ethereum_network);
        let nonce = read_state(|s| s.eth_transactions.next_transaction_nonce());
        let gas_limit = estimate_gas_limit(&request);
        match create_transaction(
            &request,
            nonce,
            gas_fee_estimate.clone(),
            gas_limit,
            ethereum_network,
        ) {
            Ok(transaction) => {
                log!(
                    DEBUG,
                    "[create_transactions_batch]: created transaction {transaction:?}",
                );

                mutate_state(|s| {
                    process_event(
                        s,
                        EventType::CreatedTransaction {
                            withdrawal_id: request.cketh_ledger_burn_index(),
                            transaction,
                        },
                    );
                });
            }
            Err(CreateTransactionError::InsufficientTransactionFee {
                cketh_ledger_burn_index: ledger_burn_index,
                allowed_max_transaction_fee: withdrawal_amount,
                actual_max_transaction_fee: max_transaction_fee,
            }) => {
                log!(
                    INFO,
                    "[create_transactions_batch]: Withdrawal request with burn index {ledger_burn_index} has insufficient amount {withdrawal_amount:?} to cover transaction fees: {max_transaction_fee:?}. Request moved back to end of queue."
                );
                mutate_state(|s| s.eth_transactions.reschedule_withdrawal_request(request));
            }
        };
    }
