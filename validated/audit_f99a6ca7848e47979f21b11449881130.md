### Title
Unsolicited Pong Disconnects Any HandshakeComplete Connection â€” (`rs/bitcoin/adapter/src/connectionmanager.rs`)

---

### Summary

`process_pong_message` unconditionally calls `conn.disconnect()` when the connection is in `PingState::Idle`. Because the normal resting state of every post-handshake connection is `Idle`, any Bitcoin peer that has completed the version handshake can tear down its own connection with a single unsolicited `Pong` message. There is no rate-limiting, grace period, or state guard.

---

### Finding Description

In `connectionmanager.rs`, `process_pong_message` matches on `conn.ping_state()`: [1](#0-0) 

The `PingState::Idle` arm unconditionally evaluates to `valid_pong = false`, which falls through to `conn.disconnect()`. The connection's state is set to `NodeDisconnected` and its task is aborted: [2](#0-1) 

`reap_disconnected()` then removes the entry from `self.connections` and calls `address_book.remove_from_active()`: [3](#0-2) 

The message dispatch in `process_bitcoin_network_message` routes `Pong` directly to `process_pong_message` with no prior connection-state guard: [4](#0-3) 

The normal post-handshake ping state is `PingState::Idle` (set at construction and after every valid pong): [5](#0-4) 

So the invariant "a connection in `Idle` state is safe from peer-initiated disconnection" is broken.

---

### Impact Explanation

An attacker who controls or can spoof any Bitcoin peer that has completed the version handshake can immediately disconnect that connection by sending one `Pong` message. If the attacker controls or can intercept traffic to enough peers (or repeatedly reconnects and sends `Pong` to each new connection), the Bitcoin adapter can be held at zero available connections. This prevents the IC's Bitcoin integration (ckBTC chain-fusion) from relaying Bitcoin data to the subnet â€” a constrained subnet availability issue for the Bitcoin adapter component.

---

### Likelihood Explanation

The precondition is realistic: any peer that completes the Bitcoin version handshake (a public, unauthenticated protocol) can trigger this. A network-positioned attacker (e.g., operating a malicious Bitcoin node or performing BGP-level interception) can complete the handshake and immediately send `Pong`. No privileged access, key material, or majority corruption is required.

---

### Recommendation

In `process_pong_message`, treat an unsolicited `Pong` as a no-op (or log and ignore) rather than disconnecting. The `PingState::Idle` branch should simply return without calling `conn.disconnect()`:

```rust
PingState::Idle { last_pong_at: _ } => {
    // Unsolicited pong â€” ignore silently.
}
```

Alternatively, use `conn.discard()` (which blacklists the address) instead of `conn.disconnect()` if the intent is to penalize misbehaving peers, but the primary fix is to not disconnect on an unsolicited `Pong`.

---

### Proof of Concept

State-machine test (no network required):

1. Create a `ConnectionManager` with a connection in `HandshakeComplete` + `PingState::Idle`.
2. Call `process_bitcoin_network_message(addr, &NetworkMessage::Pong(any_nonce))`.
3. Assert `conn.state()` is `ConnectionState::NodeDisconnected`.
4. Call `reap_disconnected()`.
5. Assert `self.connections` is empty.

The `PingState::Idle` arm at [6](#0-5)  is the single root cause line. Any nonce value triggers it.

    fn reap_disconnected(&mut self) {
        let mut disconnects = vec![];
        for (addr, conn) in self.connections.iter() {
            match conn.state() {
                ConnectionState::AdapterDiscarded => {
                    self.address_book.discard(conn.address_entry());
                }
                ConnectionState::NodeDisconnected => {
                    self.address_book.remove_from_active(conn.address_entry());
                }
                _ => {}
            }

            if conn.is_disconnected() {
                disconnects.push(*addr);
            }
        }

        for addr in disconnects {
            self.connections.remove(&addr);
        }
