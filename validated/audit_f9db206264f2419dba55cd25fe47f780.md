Audit Report

## Title
Missing Concurrency Limit on Synchronous Call Endpoints (`/api/v3`, `/api/v4`) Allows Resource Exhaustion - (File: `rs/http_endpoints/public/src/lib.rs`)

## Summary
The `make_router` function in `rs/http_endpoints/public/src/lib.rs` applies a `GlobalConcurrencyLimitLayer` to every API route except `call_v3_router`, `call_v4_router`, and `subnet_call_v4_router`. These synchronous call endpoints hold each HTTP connection open for up to 10 seconds awaiting ingress certification, and with no concurrency cap, an unprivileged attacker can saturate the replica's Tokio async runtime with an unbounded number of long-lived tasks, degrading or denying service to all other endpoints on that node.

## Finding Description
In `rs/http_endpoints/public/src/lib.rs` at lines 603–609, `call_v2_router` receives a `GlobalConcurrencyLimitLayer(max_call_concurrent_requests=50)`, while `call_v3_router`, `call_v4_router`, and `subnet_call_v4_router` are merged without any layer. The TODO comment at line 606 explicitly acknowledges this gap. Each request that reaches the synchronous call handler (`call_sync`) subscribes to the `IngressWatcher` and then blocks in `wait_for_certification().timeout(Duration::from_secs(ingress_message_certificate_timeout_seconds))` at `rs/http_endpoints/public/src/call/call_sync.rs` lines 310–315, holding a Tokio task alive for up to 10 seconds (default `ingress_message_certificate_timeout_seconds = 10`).

The existing guards are insufficient to bound concurrent long-lived connections:
- The ingress pool throttler (`call.rs` lines 229–236) returns 503 when the pool is full, but only prevents new messages from entering the pool — connections already in the waiting-for-certification phase are unaffected.
- `INGRESS_WATCHER_CHANNEL_SIZE = 1000` (`ingress_watcher.rs` line 22) bounds the subscription registration queue, not the number of active waiting connections; when the channel is full, `call_sync` returns early with 202 rather than shedding the HTTP connection before the wait phase.
- `http_max_concurrent_streams = 1000` is a per-connection HTTP/2 limit; an attacker using multiple TCP connections bypasses it entirely.
- Boundary node rate limiting (`ic_boundary/src/cli.rs` lines 272–277) is `Option<u32>` and not enforced by default.

An attacker can therefore maintain a sustained pool of concurrent connections all blocked in the 10-second certification wait, exhausting Tokio worker threads and OS connection resources on the targeted replica node.

## Impact Explanation
This is an application-level DoS against a single replica node, not raw volumetric DDoS — it exploits a specific missing concurrency control to hold async tasks open efficiently. The result is degraded or denied service for all endpoints (query, read_state, v2 call, status) on the targeted node. This matches the **High** bounty impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation
The attack requires only the ability to send valid HTTP POST requests with well-formed CBOR ingress envelopes to a publicly reachable replica endpoint. No authentication, cycles, or special privileges are needed — anonymous sender is accepted at the HTTP layer. The attack is trivially automatable with any HTTP client supporting concurrent connections. The TODO comment (`CON-1574`) in the source confirms the gap is known and unresolved. The attack is repeatable and sustainable: as old connections time out after 10 seconds, new batches can be submitted.

## Recommendation
Apply a `GlobalConcurrencyLimitLayer` to `call_v3_router`, `call_v4_router`, and `subnet_call_v4_router` in `make_router`, using a dedicated config field (e.g., `max_sync_call_concurrent_requests`) in `rs/config/src/http_handler.rs`. Because these endpoints hold connections open far longer than the fire-and-forget v2 endpoint, the concurrency cap should be set conservatively (e.g., ≤50). Additionally, enforce boundary node rate limiting (`rate_limit_per_second_per_ip`, `rate_limit_per_second_per_subnet`) by default rather than as opt-in `Option<u32>` fields.

## Proof of Concept
A minimal local test using `pocket-ic` or a local replica:
1. Start a local replica with default `Config`.
2. Spawn 500+ concurrent async HTTP clients each posting a valid CBOR ingress envelope to `/api/v3/canister/{id}/call`.
3. Observe that all 500 requests enter `call_sync` and block for up to 10 seconds with no 503 load-shedding from the v3 router.
4. Simultaneously send requests to `/api/v2/query` and observe severe latency increase or timeouts, confirming Tokio runtime saturation.
