### Title
Permanent Fund Loss in `ExitToNear` Precompile When NEP-141 Transfer Fails Without Refund Callback — (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns EVM-side tokens before the corresponding NEAR-side NEP-141 `ft_transfer` promise is confirmed. When the `error_refund` compile-time feature is disabled, no callback is attached to handle a failed promise. A recipient can intentionally unregister from the NEP-141 token contract (removing their storage deposit) to cause the `ft_transfer` promise to fail, permanently destroying the sender's EVM tokens with no refund path.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear::run` method constructs a NEAR promise to call `ft_transfer` (or `ft_transfer_call`) on the NEP-141 contract. The callback wrapper is only attached when `callback_args` differs from its default value: [1](#0-0) 

Without the `error_refund` feature, `refund` is always `None`: [1](#0-0) 

And for a standard ERC-20 exit (non-wNEAR), `transfer_near` is also `None`, so `callback_args` equals `default()`. The result is a bare `PromiseArgs::Create` with no callback: [2](#0-1) 

The `exit_to_near_precompile_callback` in `engine/src/contract_methods/connector.rs` confirms: when the promise fails and `args.refund` is `None`, the function silently returns `Ok(None)` — no refund, no error: [3](#0-2) 

The refund path is gated entirely behind the `error_refund` feature: [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing / destruction of user funds.**

When the `ft_transfer` NEAR promise fails and no callback is registered, the EVM-side tokens have already been burned (the ERC-20 `burn` call precedes the precompile invocation). There is no on-chain mechanism to recover them. The funds are permanently destroyed.

---

### Likelihood Explanation

**Medium.** The attack requires a recipient to:
1. Be the target of an `ExitToNear` transfer from another user.
2. Call `storage_unregister` on the NEP-141 contract before the NEAR promise executes (NEAR transactions are asynchronous; the promise executes in a subsequent block).

This is a realistic griefing vector: a malicious counterparty receiving tokens via `ExitToNear` can unregister from the NEP-141 contract in the same or next block, causing the sender's EVM tokens to be permanently lost. The sender has no recourse.

---

### Recommendation

1. **Always attach the refund callback** for `ExitToNear`, regardless of the `error_refund` feature flag. The callback in `exit_to_near_precompile_callback` already handles the failure case correctly when `args.refund` is populated.
2. Alternatively, ensure the `error_refund` feature is unconditionally enabled in all production builds and document this as a hard requirement.
3. Consider a two-phase commit pattern: lock EVM tokens rather than burning them until the NEAR-side transfer is confirmed successful.

---

### Proof of Concept

1. Alice holds 100 ERC-20 tokens (mapped to NEP-141 `token.near`) on Aurora EVM.
2. Alice calls the ERC-20 `withdraw` function, which burns her tokens and invokes the `ExitToNear` precompile with Bob's NEAR account as recipient.
3. The precompile emits a `PromiseArgs::Create` log targeting `ft_transfer` on `token.near` — **no callback is attached** (production build without `error_refund`). [5](#0-4) 

4. Bob, observing the pending NEAR transaction in the mempool, calls `storage_unregister` on `token.near` to remove his storage deposit.
5. The `ft_transfer` promise executes and fails: Bob is no longer registered.
6. Because no callback exists, the failure is silently dropped. Alice's 100 ERC-20 tokens are permanently destroyed; Bob receives nothing.

The root cause is the conditional callback attachment at: [2](#0-1) 

combined with the compile-time suppression of the refund argument at: [1](#0-0)

#[cfg(feature = "error_refund")]
#[allow(clippy::unnecessary_wraps)]
fn refund_call_args(
    params: &ExitToNearParams,
    event: &events::ExitToNear,
) -> Option<RefundCallArgs> {
    Some(RefundCallArgs {
        recipient_address: match params {
            ExitToNearParams::BaseToken(params) => params.refund_address,
            ExitToNearParams::Erc20TokenParams(params) => params.refund_address,
        },
        erc20_address: match params {
            ExitToNearParams::BaseToken(_) => None,
            ExitToNearParams::Erc20TokenParams(_) => {
                let erc20_address = match event {
                    events::ExitToNear::Legacy(legacy) => legacy.erc20_address,
                    events::ExitToNear::Omni(omni) => omni.erc20_address,
                };
                Some(erc20_address)
            }
        },
        amount: types::u256_to_arr(&match event {
            events::ExitToNear::Legacy(legacy) => legacy.amount,
            events::ExitToNear::Omni(omni) => omni.amount,
        }),
    })
}
