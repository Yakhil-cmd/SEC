### Title
`receive_erc20_tokens` and `receive_base_tokens` Trust Caller-Supplied `amount` Without Verifying Actual Balance Received for Fee-on-Transfer NEP-141 Tokens - (`engine/src/engine.rs`)

---

### Summary

Aurora Engine's `ft_on_transfer` handler mints ERC-20 (or base ETH) tokens equal to `args.amount` — the value supplied by the calling NEP-141 contract — without verifying the actual balance change in Aurora's account. For fee-on-transfer or share-based NEP-141 tokens, the actual amount received by Aurora can be less than `args.amount`, causing Aurora to mint more ERC-20 tokens than it holds in NEP-141 backing, leading to insolvency.

---

### Finding Description

When a NEP-141 token is bridged into Aurora via `ft_transfer_call`, the NEP-141 contract calls Aurora's `ft_on_transfer` with the original requested `amount`. Aurora then mints exactly that `amount` of ERC-20 tokens (or ETH) to the recipient.

In `engine/src/contract_methods/connector.rs`, `ft_on_transfer` dispatches to one of two minting paths based on whether the caller is the ETH connector: [1](#0-0) 

For the ERC-20 path, `receive_erc20_tokens` reads the amount directly from `args`: [2](#0-1) 

And mints exactly that amount via an EVM call: [3](#0-2) 

For the base token (ETH) path, `receive_base_tokens` does the same: [4](#0-3) 

Neither path checks the actual NEP-141 balance of Aurora before and after the transfer. The `args.amount` field in `FtOnTransferArgs` is set by the NEP-141 contract itself: [5](#0-4) 

For a fee-on-transfer or share-based NEP-141 token, the NEP-141 contract may transfer `args.amount - delta` to Aurora while still calling `ft_on_transfer` with the original `args.amount`. Aurora mints the full `args.amount` of ERC-20 tokens, but only holds `args.amount - delta` NEP-141 tokens. This discrepancy accumulates with every deposit, eventually making Aurora insolvent for that token.

---

### Impact Explanation

**Insolvency (Critical)**: Aurora's ERC-20 mirror token supply exceeds the actual NEP-141 tokens held by Aurora. When users attempt to exit via the `ExitToNear` precompile (which calls `ft_transfer` on the NEP-141 contract with the full ERC-20 amount), the last users to exit will find Aurora has insufficient NEP-141 balance to honor their withdrawal. Their ERC-20 tokens are burned but no NEP-141 tokens are received, resulting in permanent loss of funds. [6](#0-5) 

---

### Likelihood Explanation

Any NEP-141 token with fee-on-transfer or share-based transfer mechanics can be registered with Aurora via `deploy_erc20_token` — this is an open, permissionless operation. Once registered, any user can trigger the vulnerability by calling `ft_transfer_call` on the NEP-141 contract targeting Aurora. No special privileges are required. The vulnerability is triggered on every deposit, so the insolvency gap grows monotonically.

---

### Recommendation

In both `receive_base_tokens` and `receive_erc20_tokens`, record the relevant balance before the transfer and use the actual balance delta as the mint amount, rather than trusting `args.amount`:

```rust
// In receive_erc20_tokens (conceptual fix):
let balance_before = query_nep141_balance_of_aurora(token);
// ... (ft_on_transfer is called after the transfer has already occurred)
let balance_after = query_nep141_balance_of_aurora(token);
let actual_amount = balance_after - balance_before;
// mint actual_amount instead of args.amount
```

Alternatively, document and enforce that only non-fee-on-transfer NEP-141 tokens may be registered with Aurora, and add a validation check in `deploy_erc20_token` or `ft_on_transfer` to reject tokens that do not transfer the exact declared amount.

---

### Proof of Concept

1. Deploy a NEP-141 token `fee_token.near` that deducts a 1% fee on every transfer (i.e., transferring 1000 tokens results in the recipient receiving 990 tokens, while `ft_on_transfer` is called with `amount = 1000`).
2. Register `fee_token.near` with Aurora via `deploy_erc20_token`.
3. Call `ft_transfer_call` on `fee_token.near` with `receiver_id = aurora`, `amount = 1000`.
4. Aurora's `ft_on_transfer` is called with `args.amount = 1000`. Aurora mints 1000 ERC-20 mirror tokens to the recipient.
5. Aurora's actual NEP-141 balance of `fee_token.near` is only 990.
6. Repeat 100 times: Aurora has minted 100,000 ERC-20 tokens but holds only 99,000 NEP-141 tokens.
7. The last ~1% of ERC-20 holders cannot exit — their ERC-20 tokens are burned by the `ExitToNear` precompile but the `ft_transfer` call on `fee_token.near` fails due to insufficient balance, permanently freezing their funds. [7](#0-6) [8](#0-7)

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
