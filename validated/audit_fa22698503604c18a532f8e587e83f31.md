Audit Report

## Title
ckERC20 Minter Mints Based on Event-Logged Amount Rather Than Actual Received Tokens, Breaking 1:1 Backing Invariant for Fee-on-Transfer ERC20 Tokens — (`rs/ethereum/cketh/minter/ERC20DepositHelper.sol`)

## Summary
The `deposit()` function in `ERC20DepositHelper.sol` emits the `ReceivedErc20` event with the caller-supplied `amount` rather than the actual tokens received by the minter's address after `safeTransferFrom`. The IC minter canister reads this event value and mints exactly that many ckERC20 tokens. For fee-on-transfer ERC20 tokens, the minter's actual ERC20 balance is less than the minted ckERC20 supply, permanently breaking the 1:1 backing invariant and causing withdrawal transactions to fail on Ethereum.

## Finding Description

**Root cause — helper contract emits requested amount, not received amount:**

`CkErc20Deposit.deposit()` at `rs/ethereum/cketh/minter/ERC20DepositHelper.sol` lines 498–503 calls `safeTransferFrom` and immediately emits the event with the caller-supplied `amount`:

```solidity
function deposit(address erc20_address, uint256 amount, bytes32 principal) public {
    IERC20 erc20Token = IERC20(erc20_address);
    erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
    emit ReceivedErc20(erc20_address, msg.sender, amount, principal);
}
```

For a fee-on-transfer ERC20 token, `safeTransferFrom` succeeds but the minter's address receives `amount - fee`, not `amount`. The emitted event still records `amount`.

**Minter mints based on the event value:**

In `rs/ethereum/cketh/minter/src/deposit.rs` lines 73–81, the minter scrapes this log and mints exactly `event.value()` ckERC20 tokens — the value taken directly from the log event, not from any on-chain balance check:

```rust
let block_index = match client
    .transfer(TransferArg {
        ...
        amount: event.value(),
    })
    .await
```

**Internal balance tracking is also inflated:**

`update_balance_upon_deposit` in `rs/ethereum/cketh/minter/src/state.rs` lines 332–338 adds the event's `value` to `erc20_balances`, so the minter's internal accounting records `amount` while the actual Ethereum balance is `amount - fee`:

```rust
fn update_balance_upon_deposit(&mut self, event: &ReceivedEvent) {
    match event {
        ReceivedEvent::Erc20(event) => self
            .erc20_balances
            .erc20_add(event.erc20_contract_address, event.value),
    };
}
```

**Withdrawal encodes the full ckERC20 burn amount:**

In `rs/ethereum/cketh/minter/src/state/transactions/mod.rs` lines 1177–1181, the minter constructs an Ethereum ERC20 `transfer` call encoding the full `withdrawal_amount`:

```rust
data: TransactionCallData::Erc20Transfer {
    to: request.destination,
    value: request.withdrawal_amount,
}.encode(),
```

If the minter's actual ERC20 balance is less than `withdrawal_amount` (because fee-on-transfer reduced the deposited amount), the ERC20 `transfer` call reverts on Ethereum. The minter has already burned the user's ckERC20 tokens on the IC ledger. The reimbursement path (`rs/ethereum/cketh/minter/src/state.rs` lines 377–383) only subtracts from `erc20_balances` on a successful receipt, so the inflated internal balance persists even after failed withdrawals.

**No existing guard prevents this:**

There is no on-chain whitelist in the helper contract — the documentation at `rs/ethereum/cketh/docs/ckerc20.adoc` lines 182–191 explicitly states the helper contract does not enforce any whitelist of allowed ERC20 tokens. The minter only filters by supported token address, not by token behavior. There is no fee-on-transfer detection anywhere in the deposit or minting path.

## Impact Explanation

This is a **Critical** impact matching: *Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant ICP/Cycles or in-scope chain-key/ledger assets, especially over $1M.*

Every deposit of a fee-on-transfer ERC20 token mints more ckERC20 than the minter holds in ERC20 backing. The total ckERC20 supply exceeds the minter's actual ERC20 holdings by the cumulative fee amount. Users who hold the excess ckERC20 can never redeem it because the backing ERC20 tokens do not exist at the minter's address. This constitutes illegal minting and protocol insolvency for any fee-on-transfer token added as a supported ckERC20 token. Given that the NNS can add any ERC20 token and the currently supported list includes tokens like SHIB and PEPE (which have had deflationary/fee mechanics in related tokens), the potential value at risk is substantial.

## Likelihood Explanation

The attack requires only that the NNS governance process adds a fee-on-transfer ERC20 token as a supported ckERC20 token. Once added, any unprivileged user can trigger the vulnerability by calling the normal deposit flow — no special access is required. The NNS governance process for adding new ckERC20 tokens (`rs/ethereum/ledger-suite-orchestrator/README.adoc` lines 82–110) specifies only chain ID, contract address, and ledger init args; there is no requirement to verify that the token is not fee-on-transfer. The exploit is repeatable on every deposit and scales linearly with deposit volume.

## Recommendation

The helper contract should measure the actual received amount by querying the minter's balance before and after `safeTransferFrom`, and emit that delta as the event value:

```solidity
function deposit(address erc20_address, uint256 amount, bytes32 principal) public {
    IERC20 erc20Token = IERC20(erc20_address);
    uint256 balanceBefore = erc20Token.balanceOf(cketh_minter_main_address);
    erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
    uint256 actualReceived = erc20Token.balanceOf(cketh_minter_main_address) - balanceBefore;
    emit ReceivedErc20(erc20_address, msg.sender, actualReceived, principal);
}
```

Alternatively, the NNS governance process for adding new ckERC20 tokens should explicitly screen out fee-on-transfer tokens, and the minter documentation should state this restriction clearly.

## Proof of Concept

1. An NNS proposal adds a fee-on-transfer ERC20 token (e.g., 1% fee per transfer) as a supported ckERC20 token via the ledger suite orchestrator.
2. Alice calls `CkErc20Deposit.deposit(token, 1_000_000, alice_principal)` on the helper contract at `rs/ethereum/cketh/minter/ERC20DepositHelper.sol` line 498.
3. `safeTransferFrom(alice, minter_address, 1_000_000)` executes. Due to the 1% fee, the minter receives `990_000` tokens.
4. The helper contract emits `ReceivedErc20(token, alice, 1_000_000, alice_principal)` at line 502 — recording `1_000_000`, not `990_000`.
5. The IC minter scrapes the log, reads `value = 1_000_000` via `event.value()` at `rs/ethereum/cketh/minter/src/deposit.rs` line 80, and mints `1_000_000` ckERC20 to Alice.
6. `update_balance_upon_deposit` at `rs/ethereum/cketh/minter/src/state.rs` line 337 records `erc20_balances[token] += 1_000_000`, but the minter only holds `990_000`.
7. Alice calls `withdraw_erc20` for `1_000_000` ckERC20. The minter burns `1_000_000` ckERC20 and constructs an Ethereum transaction calling `transfer(alice_eth_addr, 1_000_000)` at `rs/ethereum/cketh/minter/src/state/transactions/mod.rs` line 1179.
8. The Ethereum transaction reverts because the minter only holds `990_000` tokens. Alice's ckERC20 is reimbursed but her ckETH gas fee is lost.
9. After many deposits, the total ckERC20 supply far exceeds the minter's actual ERC20 holdings. The last users to withdraw find their transactions permanently failing and the excess ckERC20 in circulation can never be redeemed.

A deterministic integration test can be written using the existing `CkErc20Setup` test harness in `rs/ethereum/cketh/minter/tests/ckerc20.rs` by deploying a mock ERC20 token that deducts a 1% fee on every `transferFrom`, running the deposit flow, and asserting that `erc20_balances` exceeds the mock token's recorded balance at the minter address.
