### Title
Unchecked Return Value of Precompile `call` in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Burn Without Withdrawal - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first burning the caller's ERC-20 tokens via `_burn`, then making a low-level assembly `call` to the exit precompile. The return value `res` of that assembly `call` is captured but **never checked**. If the precompile call fails for any reason, the tokens are permanently destroyed on the EVM side with no corresponding NEP-141 release on the NEAR side — a direct analog to the unchecked `transferFrom` return value in the DODO report.

---

### Finding Description

In `EvmErc20.sol`, both withdrawal functions follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — if call returns 0 (failure), execution continues
    }
}
``` [1](#0-0) 

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    ...
    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}
``` [2](#0-1) 

The same pattern is present in `EvmErc20V2.sol`: [3](#0-2) [4](#0-3) 

In EVM assembly, `call` returns `0` on failure and `1` on success. The variable `res` is assigned but never used in a conditional or `revert`. This means:

1. `_burn` executes and permanently destroys the user's ERC-20 balance.
2. The low-level `call` to the exit precompile is made.
3. If the precompile call fails (returns `0`), the EVM transaction still **succeeds** from the caller's perspective.
4. No NEP-141 tokens are released on the NEAR side.
5. The user's funds are permanently lost.

The exit precompile addresses are hardcoded. The precompile call can fail due to: insufficient gas forwarded, malformed input, or any internal precompile error path (e.g., `ERR_INVALID_RECIPIENT_ADDRESS`, `ERR_ETH_ATTACHED_FOR_ERC20_EXIT`, etc. as seen in `engine-precompiles/src/native.rs`). [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing/destruction of user funds.**

A user calling `withdrawToNear` or `withdrawToEthereum` on any `EvmErc20`/`EvmErc20V2` token (all NEP-141-backed ERC-20 mirrors on Aurora) will have their ERC-20 tokens burned. If the precompile call silently fails, those tokens are gone with no recourse: the NEP-141 balance held by the Aurora contract is never released, and the ERC-20 balance is already zero. The user suffers a total, permanent loss of the withdrawn amount.

---

### Likelihood Explanation

**Medium.** Every user of any bridged NEP-141 token on Aurora who calls `withdrawToNear` or `withdrawToEthereum` is exposed. The failure condition requires the precompile call to return `0`. This can occur if:
- The gas forwarded to the precompile is insufficient (the `gas()` opcode forwards all remaining gas, but if the outer transaction is called with tight gas limits, the inner call may run out).
- The precompile encounters an internal error (e.g., invalid recipient encoding, missing wNEAR address configuration).
- A future upgrade changes precompile behavior.

The burn-before-call ordering makes any such failure irreversible.

---

### Recommendation

Check the return value of the assembly `call` and revert if it is `0`:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This must be applied to all four affected functions across both `EvmErc20.sol` and `EvmErc20V2.sol`. Alternatively, restructure the functions to call the precompile **before** burning, and only burn on success.

---

### Proof of Concept

1. User holds 1000 units of a NEP-141-backed ERC-20 token on Aurora (e.g., a token deployed via `deploy_erc20_token`).
2. User calls `withdrawToNear(recipient_bytes, 1000)` on the `EvmErc20` contract.
3. `_burn(msg.sender, 1000)` executes — user's ERC-20 balance drops to 0.
4. The assembly `call` to the ExitToNear precompile (`0xe921...`) fails (e.g., due to a gas edge case or precompile error) and returns `res = 0`.
5. Since `res` is never checked, no `revert` is triggered. The transaction succeeds.
6. The NEP-141 balance held by the Aurora contract is never transferred to the recipient.
7. User has lost 1000 tokens permanently — ERC-20 balance is 0, NEP-141 balance is unchanged on Aurora's side.
