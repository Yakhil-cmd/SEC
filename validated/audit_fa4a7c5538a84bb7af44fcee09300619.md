Audit Report

## Title
Unauthenticated `buyer` Parameter in `refresh_buyer_tokens` Allows Anyone to Lock Victim's ICP in SNS Swap Escrow - (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` update endpoint on the SNS Swap canister accepts an arbitrary `buyer` principal in its request payload without verifying it matches the authenticated caller. Any unprivileged ingress sender can supply a victim's principal as the `buyer` field, causing the victim's ICP (already transferred to the swap subaccount) to be committed into the swap's escrow and their open ticket to be deleted — all without the victim's consent.

## Finding Description
In `rs/sns/swap/canister/canister.rs` at lines 128–143, the handler reads the effective buyer principal from the request argument rather than from the authenticated caller:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // ← no caller check
};
``` [1](#0-0) 

When `arg.buyer` is non-empty, the attacker-supplied principal `p` is passed directly to `refresh_buyer_token_e8s`. That function:

1. Reads the ICP balance of `principal_to_subaccount(&buyer)` on the swap canister — the victim's subaccount. [2](#0-1) 
2. Validates `confirmation_text` — a publicly readable value from `get_init`. [3](#0-2) 
3. Deletes the victim's open ticket from `OPEN_TICKETS_MEMORY`. [4](#0-3) 
4. Inserts the victim into `self.buyers` with their committed ICP amount. [5](#0-4) 

Once a buyer is present in `self.buyers` with `transfer_success_timestamp_seconds == 0`, `error_refund_icp` explicitly blocks any refund:

```rust
if let Some(transfer) = &buyer_state.icp
    && transfer.transfer_success_timestamp_seconds == 0
{
    return ErrorRefundIcpResponse::new_precondition_error(...)
}
``` [6](#0-5) 

Furthermore, `error_refund_icp` is only callable after the swap reaches `Lifecycle::Aborted` or `Lifecycle::Committed`, so the victim has no recourse while the swap is open. [7](#0-6) 

By contrast, `new_sale_ticket` always uses `caller_principal_id()` with no override, demonstrating the intended authentication pattern. [8](#0-7) 

The `RefreshBuyerTokensRequest.buyer` field is documented as "If not specified, the caller is used," but no authorization check enforces that a non-empty value matches the caller. [9](#0-8) 

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Unauthorized access to… canister-controlled funds where exploitation requires meaningful per-target work or other constraints."* An attacker can lock a victim's ICP in escrow for the entire remaining swap duration (up to 90 days) without the victim's consent, destroy the victim's open ticket, and bypass the mandatory `confirmation_text` requirement. If the swap commits, the victim's ICP is swept to the SNS governance treasury and the victim receives SNS tokens they never agreed to purchase — constituting unauthorized conversion of ICP assets. The SNS Swap canister is explicitly in-scope as a core SNS framework canister.

## Likelihood Explanation
The attack requires no special privileges, no governance majority, and no cryptographic material. The victim's principal is observable on-chain; the `confirmation_text` is readable via the public `get_init` query. The only precondition is that the victim has transferred ICP to their swap subaccount (the normal first step of participation). The attack is therefore trivially executable by any IC user who monitors on-chain state.

## Recommendation
Remove the unauthenticated `buyer` override. The effective buyer should always be derived from the authenticated caller:

```rust
#[update]
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p = caller_principal_id();
    let icp_ledger = create_real_icp_ledger(swap().init_or_panic().icp_ledger_or_panic());
    match swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
    {
        Ok(r) => r,
        Err(msg) => panic!("{}", msg),
    }
}
```

If third-party notification of another user's participation is a desired feature, it must be gated by an explicit allowlist or require the victim's principal to match the caller, analogous to how `new_sale_ticket` always uses `caller_principal_id()` with no override.

## Proof of Concept
1. **Setup**: SNS swap is in `Lifecycle::Open`. Alice calls `new_sale_ticket` (which correctly uses `caller_principal_id()`) to create a ticket for `amount = min_participant_icp_e8s`. Alice then transfers that ICP to `Account { owner: swap_canister_id, subaccount: principal_to_subaccount(alice) }` on the ICP ledger.

2. **Attack**: Before Alice calls `refresh_buyer_tokens`, attacker Bob submits:
   ```
   refresh_buyer_tokens({
       buyer: alice_principal.to_text(),
       confirmation_text: Some("<public confirmation text from get_init>"),
   })
   ```

3. **Effect**: The swap canister reads Alice's subaccount balance (≥ `min_participant_icp_e8s`), validates the public confirmation text, inserts Alice into `self.buyers` with her ICP amount, and deletes Alice's open ticket from `OPEN_TICKETS_MEMORY`.

4. **Consequence**: Alice's ICP is now in escrow. If she calls `error_refund_icp` while the swap is open, it is rejected with "Error refunds can only be performed when the swap is ABORTED or COMMITTED." If the swap commits, Alice's ICP is sent to the SNS treasury and she receives SNS tokens she did not explicitly agree to purchase.

A deterministic integration test using PocketIC can reproduce this by: deploying a local SNS swap canister, having Alice create a ticket and transfer ICP, then having a separate identity (Bob) call `refresh_buyer_tokens` with Alice's principal, and asserting that Alice's ticket is deleted and her ICP is locked in `self.buyers`.
