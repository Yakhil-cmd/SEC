### Title
Missing Concurrency Limit on Synchronous Call Endpoints (`/api/v3` and `/api/v4` call) Enables Resource-Exhaustion DoS - (File: `rs/http_endpoints/public/src/lib.rs`)

### Summary

The IC replica's public HTTP endpoint applies `GlobalConcurrencyLimitLayer` to every route except the three synchronous call endpoints introduced in v3/v4. An unprivileged caller can open an unbounded number of concurrent long-lived connections to these endpoints, exhausting the replica process's memory and tokio task budget, causing a denial-of-service. The gap is explicitly acknowledged in the source with a TODO comment.

### Finding Description

In `rs/http_endpoints/public/src/lib.rs`, the function `make_router` assembles the final Axum router by merging per-route sub-routers, each wrapped in a `GlobalConcurrencyLimitLayer`. Every endpoint receives this protection — except three:

```rust
// TODO(CON-1574): see if there is any reasonable explicit concurrency limit we could use here.
.merge(http_handler.call_v3_router)
.merge(http_handler.call_v4_router)
.merge(http_handler.subnet_call_v4_router)
``` [1](#0-0) 

By contrast, every other endpoint is guarded:

- `call_v2_router` → `GlobalConcurrencyLimitLayer::new(config.max_call_concurrent_requests)` (default 50)
- `query_v2/v3/subnet_v3` → `GlobalConcurrencyLimitLayer::new(config.max_query_concurrent_requests)`
- `read_state`, `catchup`, `dashboard`, `status`, `pprof`, `tracing` → all have explicit limits [2](#0-1) 

The three unguarded routes map to the synchronous call handler (`call_sync`):

- `POST /api/v3/canister/{effective_canister_id}/call`
- `POST /api/v4/canister/{effective_canister_id}/call`
- `POST /api/v4/subnet/{effective_subnet_id}/call` [3](#0-2) 

The `call_sync` handler is long-lived by design. After validating the ingress message, it subscribes to the ingress watcher and then **blocks the tokio task** waiting for certification for up to `ingress_message_certificate_timeout_seconds` (default **10 seconds**):

```rust
match certification_subscriber
    .wait_for_certification()
    .timeout(Duration::from_secs(
        ingress_message_certificate_timeout_seconds,
    ))
    .await
``` [4](#0-3) 

The default timeout is 10 seconds: [5](#0-4) 

Each in-flight request holds:
- A tokio task pinned for up to 10 seconds
- A subscription slot in the ingress watcher
- Up to 5 MB of request body memory (`max_request_size_bytes`)
- A TLS connection and associated buffers

With no upper bound on concurrent requests, an attacker can accumulate thousands of these simultaneously.

### Impact Explanation

An unprivileged attacker who can reach the replica's HTTP endpoint (port 8080, accessible from boundary nodes and other registered nodes) can send a continuous flood of POST requests to `/api/v3/canister/.../call`. Each request is accepted, validated, and then held open for up to 10 seconds awaiting certification. Because there is no concurrency cap, the number of simultaneously held tokio tasks, ingress watcher subscriptions, and 5 MB request buffers grows without bound. This leads to memory exhaustion and potential OOM-kill of the replica process, taking the node offline and reducing subnet fault tolerance. The impact is equivalent to the Staking API DoS: the ingress submission service becomes unavailable to legitimate users.

### Likelihood Explanation

The boundary node (`ic_boundary`) proxies all public API traffic to the replica. The boundary node's IP-based and subnet-based rate limiting is **optional** — it is only activated when the CLI flags `--rate-limit-per-second-per-ip` and `--rate-limit-per-second-per-subnet` are set:

```rust
if let Some(rl) = cli.rate_limiting.rate_limit_per_second_per_ip {
    route = RateLimit::try_from(rl).unwrap().add_ip_rate_limiting(route);
}
``` [6](#0-5) 

Even when boundary-node rate limiting is active, it limits the **rate** (requests/second), not the number of **concurrent in-flight** requests. A single IP can open many simultaneous long-lived connections within the rate limit window. The replica itself has no defense-in-depth for these three endpoints. The TODO comment in the source confirms the gap is known and unresolved.

### Recommendation

Apply `GlobalConcurrencyLimitLayer` to `call_v3_router`, `call_v4_router`, and `subnet_call_v4_router` in `make_router`, using the existing `config.max_call_concurrent_requests` (or a new dedicated config field). Add a corresponding entry to `rs/config/src/http_handler.rs` (e.g., `max_sync_call_concurrent_requests`) and a load-shedding test analogous to the existing `test_load_shedding_update_call` test in `rs/http_endpoints/public/tests/load_shed_test.rs`.

### Proof of Concept

1. Obtain a valid canister ID on any IC subnet.
2. Craft a valid CBOR-encoded `HttpCallContent` ingress envelope (any well-formed call, e.g., to `aaaaa-aa` management canister).
3. Open N concurrent HTTP/2 streams (or HTTP/1.1 connections) to the replica's port 8080 (or via a boundary node without rate limiting), all targeting `POST /api/v3/canister/aaaaa-aa/call`.
4. Each request passes ingress validation, subscribes to the ingress watcher, and blocks for up to 10 seconds.
5. With N large enough (e.g., tens of thousands), the replica process exhausts available memory and is OOM-killed.

The absence of any `GlobalConcurrencyLimitLayer` on these three routes — confirmed at: [1](#0-0) 

— is the necessary and sufficient root cause. All other routes are protected; these three are not.
