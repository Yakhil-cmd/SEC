Audit Report

## Title
Unbounded `active_assembles` / `artifact_processor_tasks` growth via `SLOT_TABLE_NO_LIMIT` allows single Byzantine subnet peer to OOM a replica — (`rs/p2p/consensus_manager/src/receiver.rs` + `rs/replica/setup_ic_network/src/lib.rs`)

## Summary
`SLOT_TABLE_NO_LIMIT` is set to `usize::MAX` and is used as the `slot_limit` for consensus, certification, DKG, IDKG, and HTTPS-outcalls artifact managers. The sole per-peer admission guard in `handle_slot_update_receive` is `peer_slot_table_len < self.slot_limit`, which never fires when `slot_limit = usize::MAX`. A single TLS-authenticated Byzantine subnet peer can send arbitrarily many slot updates with distinct slot numbers and artifact IDs, causing `active_assembles` and `artifact_processor_tasks` to grow without bound until the replica process is OOM-killed.

## Finding Description
`SLOT_TABLE_NO_LIMIT` is defined as `usize::MAX` at `rs/replica/setup_ic_network/src/lib.rs:75` and is passed as `slot_limit` for consensus (lines 237, 246), certification (line 268), DKG (line 279), IDKG (line 291), and HTTPS-outcalls (line 303). Only ingress uses a concrete bound (`SLOT_TABLE_LIMIT_INGRESS = 50_000`, line 74).

In `handle_slot_update_receive` (`rs/p2p/consensus_manager/src/receiver.rs:373`), the guard is:
```rust
Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit => { ... }
```
With `slot_limit = usize::MAX`, this condition is always true. For each new `(slot_number, artifact_id)` pair where the artifact ID has not been seen before, the code at lines 393–420:
1. Inserts a new `SlotEntry` into `slot_table[peer_id]`
2. Inserts a new `watch::Sender` into `active_assembles` (line 405)
3. Spawns a new async task into `artifact_processor_tasks` (lines 407–419)

There is no secondary cap on `active_assembles.len()` or `artifact_processor_tasks.len()`. The `handle_topology_update` path (lines 547–558) only prunes entries when a peer *leaves* the subnet topology — a still-connected Byzantine peer is never pruned.

## Impact Explanation
Each spawned `process_slot_update` task allocates a Tokio task stack, a `watch` channel pair, and holds a clone of the `artifact_assembler`. Sending O(millions) of slot updates from one peer exhausts heap memory, causing an OOM kill of the replica process. A crashed replica drops out of consensus participation. A single compromised node — well below the Byzantine fault threshold `f < n/3` — is sufficient to crash all honest replicas simultaneously, halting subnet liveness. This matches the allowed High impact: **Application/platform-level DoS, crash, consensus blocking, or subnet availability impact not based on raw volumetric DDoS**.

## Likelihood Explanation
The attacker must be a valid subnet member (TLS-authenticated via QUIC transport). A single compromised node below the fault threshold is sufficient. The attack requires only sending crafted P2P slot-update messages in a tight loop, which is trivially scriptable using the node's own TLS key. No additional cryptographic material or victim interaction is needed. The attack is repeatable and deterministic.

## Recommendation
Replace `SLOT_TABLE_NO_LIMIT` with a concrete, protocol-justified per-peer cap for each artifact type, analogous to `SLOT_TABLE_LIMIT_INGRESS = 50_000`. The correct bound for each type is the maximum number of in-flight artifacts a single honest peer would legitimately advertise simultaneously (e.g., bounded by the consensus round window for consensus artifacts, DKG transcript count for DKG, etc.). Alternatively, add a global cap on `active_assembles.len()` with back-pressure or LRU eviction to bound total memory usage regardless of per-peer slot counts.

## Proof of Concept
```rust
// Unit test using the existing ReceiverManagerBuilder test harness in receiver.rs:
#[tokio::test]
async fn unbounded_active_assembles_growth() {
    let (mut mgr, _channels) = ReceiverManagerBuilder::new()
        .with_slot_limit(usize::MAX) // matches SLOT_TABLE_NO_LIMIT
        .build();
    let cancellation = CancellationToken::new();
    let n = 100_000usize;
    for i in 0..n {
        mgr.handle_slot_update_receive(
            SlotUpdate {
                slot_number: SlotNumber::from(i as u64),
                commit_id: CommitId::from(1),
                update: Update::Id(i as u64), // distinct ID each time
            },
            NODE_1,
            ConnId::from(1),
            cancellation.clone(),
        );
    }
    // Both grow to n with no bound:
    assert_eq!(mgr.active_assembles.len(), n);
    assert_eq!(mgr.artifact_processor_tasks.len(), n);
    // In production, process OOMs long before n reaches usize::MAX.
}
```
This test requires no external infrastructure and directly exercises the vulnerable code path using the existing `ReceiverManagerBuilder` harness already present in the test module at `rs/p2p/consensus_manager/src/receiver.rs:621`.
