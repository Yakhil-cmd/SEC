### Title
Compounded Rounding Error in L1 Transaction `native_per_pubdata` Derivation Systematically Overcharges Pubdata Fees — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

In `prepare_and_check_resources`, the `native_per_pubdata` rate for L1→L2 transactions is derived by multiplying `gas_per_pubdata` by the already-rounded `native_per_gas`. Because `native_per_gas` is computed with `div_ceil` (rounding up), the rounding error is amplified by `gas_per_pubdata`. This causes every L1 transaction sender to be systematically overcharged for pubdata native resources, which propagates into the `delta_gas` refund calculation and reduces the gas refund the sender receives.

---

### Finding Description

In `prepare_and_check_resources` the derivation of `native_per_pubdata` for L1 transactions is a two-step calculation:

**Step 1** — `native_per_gas` is computed with ceiling division: [1](#0-0) 

```
native_per_gas = ceil(gas_price / L1_TX_NATIVE_PRICE)
```

`L1_TX_NATIVE_PRICE` is the constant `10`. [2](#0-1) 

**Step 2** — `native_per_pubdata` is derived by multiplying the already-rounded `native_per_gas`: [3](#0-2) 

```
native_per_pubdata = gas_per_pubdata * ceil(gas_price / 10)
```

The mathematically correct formula is:

```
native_per_pubdata = ceil(gas_per_pubdata * gas_price / 10)
```

The rounding error introduced by `div_ceil` in step 1 (at most +1) is multiplied by `gas_per_pubdata` in step 2. Since `gas_per_pubdata` is a `u32` (up to ~4 billion), the amplified overestimation per pubdata byte can be enormous.

**Contrast with L2 transactions**, where `native_per_pubdata` is computed in a single independent division from the block-level `pubdata_price`: [4](#0-3) 

The L1 path has no equivalent single-step formula because `pubdata_price` is not a block-level parameter for L1 transactions — it is derived from the per-transaction `gas_per_pubdata` field, making the two-step derivation the root cause.

**Propagation into the refund calculation:**

The overestimated `native_per_pubdata` is passed to `get_resources_to_charge_for_pubdata`, which charges `pubdata_used * native_per_pubdata` native units: [5](#0-4) 

The inflated pubdata charge increases `native_used`. In `compute_gas_refund`, `delta_gas` is then computed as: [6](#0-5) 

```
delta_gas = (native_used / native_per_gas) - gas_used
```

Because `native_used` is inflated by the overestimated pubdata charge, `delta_gas` is inflated, `gas_used` is increased, and the sender's gas refund (`gas_limit - gas_used`) is reduced.

---

### Impact Explanation

Every L1→L2 transaction sender whose `gas_price` is not an exact multiple of `L1_TX_NATIVE_PRICE = 10` is overcharged. The overcharge per pubdata byte is:

```
overcharge_per_byte = gas_per_pubdata * (ceil(gas_price/10) - gas_price/10)
```

For `gas_price = 11`, `gas_per_pubdata = 800`:
- `native_per_gas = ceil(11/10) = 2` (true value: 1.1)
- `native_per_pubdata = 800 * 2 = 1600` (correct: `ceil(800*11/10) = 880`)
- Overcharge per pubdata byte: `720` native units
- For 1000 pubdata bytes: `720,000` extra native units charged
- `delta_gas = 720,000 / 2 = 360,000` extra gas charged to the sender
- Financial loss: `360,000 * 11 = 3,960,000` wei per 1000 pubdata bytes

The sender receives a smaller gas refund than they are entitled to. In extreme cases (high `gas_per_pubdata`, high pubdata usage), the transaction may fail with "out of native resources" even when the sender provided sufficient gas.

---

### Likelihood Explanation

- Any `gas_price` not divisible by `10` triggers the rounding error. Since `L1_TX_NATIVE_PRICE = 10` is a hardcoded constant, virtually all real-world gas prices (e.g., 1 gwei = 1,000,000,000) will not be exact multiples of 10 in the relevant unit.
- The entry path is an unprivileged L1→L2 priority transaction — no special role or access is required.
- The effect is deterministic and reproducible for every affected transaction.

---

### Recommendation

Compute `native_per_pubdata` directly from the original values without using the intermediate rounded `native_per_gas`:

```rust
// Instead of:
let native_per_pubdata = (gas_per_pubdata as u64).checked_mul(native_per_gas)...;

// Use:
let native_per_pubdata = u256_try_to_u64(
    &(U256::from(gas_per_pubdata) * gas_price).div_ceil(native_price)
).unwrap_or_else(|| { ... u64::MAX });
```

This mirrors the L2 transaction path where `native_per_pubdata` is computed in a single division from the source values, avoiding amplification of the rounding error.

---

### Proof of Concept

**Setup:**
- `gas_price = 11` (not divisible by `L1_TX_NATIVE_PRICE = 10`)
- `gas_per_pubdata = 800`
- Transaction generates 1000 bytes of pubdata

**Current behavior (buggy):**
```
native_per_gas      = ceil(11 / 10)       = 2        // rounding error: +0.9
native_per_pubdata  = 800 * 2             = 1600      // amplified error: +720/byte
pubdata_charge      = 1000 * 1600         = 1,600,000 native units
delta_gas           = 1,600,000 / 2       = 800,000
gas_used            += 800,000            (inflated)
refund              = gas_limit - gas_used (reduced by 800,000 gas)
```

**Correct behavior:**
```
native_per_pubdata  = ceil(800 * 11 / 10) = 880       // single-step
pubdata_charge      = 1000 * 880          = 880,000 native units
delta_gas           = 880,000 / 1.1       ≈ 800,000   // correct
```

The sender is overcharged `800,000 - 800,000 = 0` gas in the correct path, but `800,000` extra gas in the buggy path — a direct financial loss of `800,000 * 11 = 8,800,000` wei for this example transaction.
