### Title
Unchecked precompile call return value in `withdrawToNear` allows permanent token destruction — (`etc/eth-contracts/contracts/EvmErc20.sol`)

### Summary
`EvmErc20.withdrawToNear` burns the caller's ERC-20 tokens before invoking the `ExitToNear` precompile, and never checks the return value of the assembly `call`. Supplying an invalid (zero-length or non-UTF-8) `recipient` causes the precompile to revert internally (returning `res=0`), but the Solidity function does not revert, leaving the tokens permanently destroyed with no corresponding NEP-141 credit on NEAR.

### Finding Description
In `EvmErc20.sol`, `withdrawToNear` follows this sequence:

1. `_burn(_msgSender(), amount)` — irreversibly destroys the caller's ERC-20 tokens.
2. Constructs input as `\x01 ++ amount_b ++ recipient` and calls the `ExitToNear` precompile at `0xe9217bc7...`.
3. The assembly block captures `res` but **never checks it**; there is no `require(res != 0, ...)`. [1](#0-0) 

Inside the precompile, `ExitToNearParams::try_from` calls `parse_recipient` on the recipient slice. For `b""`, `"".parse::<AccountId>()` fails. For `[0xc2]`, `str::from_utf8` fails. Both return `Err(ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID"))`, causing the precompile `run` to return `Err(...)`, which the EVM translates to `res=0`. [2](#0-1) [5](#0-4) 

Because `res` is never checked, execution continues after the failed precompile call. The burn is not rolled back, and no NEAR promise is emitted.

### Impact Explanation
Any user who calls `withdrawToNear` with a zero-length or non-UTF-8 `recipient` will have their ERC-20 tokens permanently burned with no NEP-141 tokens credited on NEAR. This is a direct, permanent loss of user funds matching the Critical scope: *"Direct theft of any user funds, whether at-rest or in-motion."*

### Likelihood Explanation
The function is publicly callable by any ERC-20 token holder. The invalid input can be supplied accidentally (e.g., a frontend bug passing an empty bytes argument) or deliberately by an attacker targeting their own tokens or tricking another user into calling the function. No special privileges are required.

### Recommendation
Add a `require` check on the precompile call result immediately after the assembly block:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Alternatively, restructure the function to validate the recipient before burning: call the precompile first (or perform recipient validation in Solidity), and only burn on success.
