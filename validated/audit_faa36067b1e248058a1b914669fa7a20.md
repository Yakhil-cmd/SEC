### Title
EVM Storage Refund (`evm_refund`) Cancelled by Native-Resource `delta_gas` Adjustment — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the EVM storage refund (`evm_refund`, from EIP-3529 SSTORE refunds) is subtracted from `gas_used` **before** the native-resource `delta_gas` adjustment is computed. When native resource consumption is high enough to dominate, `delta_gas` is computed against the already-reduced `gas_used`, effectively adding the `evm_refund` back and cancelling it entirely. The user does not receive the EVM refund they are entitled to, and the operator collects excess fees.

---

### Finding Description

In `compute_gas_refund` (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`):

```rust
// Step 1: evm_refund subtracted from gas_used
gas_used -= evm_refund;                                          // line 48

// Step 2: max with minimal_gas_used
let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);  // line 56

// Step 3: native_used computed from remaining native budget
let native_used = full_native_limit
    .saturating_sub(resources.native().remaining().as_u64());   // line 64

// Step 4: delta_gas computed against already-reduced gas_used
let delta_gas = (native_used / native_per_gas) as i64
    - (gas_used as i64);                                         // line 72

if delta_gas > 0 {
    gas_used += delta_gas as u64;                                // line 78
}
``` [1](#0-0) 

The `evm_refund` is subtracted from `gas_used` at line 48. Then at line 72, `delta_gas` is computed as `(native_used / native_per_gas) - gas_used`, where `gas_used` is already the post-refund value. When `delta_gas > 0` (native resource consumption dominates), `gas_used` is raised back to `native_used / native_per_gas`, which is exactly `gas_used_before_refund - evm_refund + delta_gas`. The final `gas_used` equals `native_used / native_per_gas`, and the `evm_refund` is completely cancelled.

**Concrete example:**
- `gas_limit = 1000`, `gas_used_before_refund = 800`, `evm_refund = 100`
- `native_used / native_per_gas = 750`

Current (buggy) path:
1. `gas_used = 800 - 100 = 700`
2. `delta_gas = 750 - 700 = 50 > 0`
3. `gas_used = 700 + 50 = 750`
4. User refund: `(1000 - 750) * gas_price = 250 * gas_price`

Correct path (evm_refund applied after delta_gas):
1. `gas_used = 800`
2. `delta_gas = 750 - 800 = -50 ≤ 0` → no adjustment
3. `gas_used = 800 - 100 = 700`
4. User refund: `(1000 - 700) * gas_price = 300 * gas_price`

The user loses `evm_refund * gas_price = 100 * gas_price` tokens.

The `evm_refund` is stored in `context.gas_refunded` and reported in `ZkTxResult`, but it is never used to compensate the user — the actual token refund is computed solely from `context.tx_gas_limit - context.gas_used`. [2](#0-1) [3](#0-2) 

---

### Impact Explanation

Users who perform SSTORE operations that generate EVM storage refunds (e.g., clearing storage slots to zero) AND whose transactions also have significant native resource consumption (pubdata writes) will not receive the EVM refund they are entitled to under EIP-3529. The operator receives excess fees equal to `evm_refund * gas_price`. This is a direct, quantifiable loss of user funds on every affected transaction. The maximum loss per transaction is `(gas_used / 5) * gas_price` (EIP-3529 cap).

---

### Likelihood Explanation

The bug manifests whenever `native_used / native_per_gas > gas_used_before_refund - evm_refund`. This is realistic for DeFi transactions that:
1. Clear storage slots (common in token approvals, DEX swaps resetting allowances, etc.)
2. Also write significant pubdata (any state-changing transaction on ZKsync)

The condition is `native_used / native_per_gas > gas_used * 0.8` (since `evm_refund ≤ gas_used / 5`). This is a common scenario when pubdata prices are non-trivial. Any unprivileged L2 transaction sender can trigger this path.

---

### Recommendation

Apply the `evm_refund` **after** the `delta_gas` adjustment, not before. The corrected order:

```rust
// 1. Compute gas_used from ergs
let mut gas_used = gas_limit
    .checked_sub(resources.ergs().0.div_floor(ERGS_PER_GAS))
    .ok_or(internal_error!("gas remaining > gas limit"))?;
resources.exhaust_ergs();

// 2. Apply minimal_gas_used floor
let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);

// 3. Compute and apply delta_gas from native resource consumption
let native_used = full_native_limit.saturating_sub(resources.native().remaining().as_u64());
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}

// 4. Apply evm_refund AFTER delta_gas (so it is not cancelled by native adjustment)
let evm_refund = {
    let full_refund_ergs = system.io.get_refund_counter().ergs();
    let full_refund_gas = full_refund_ergs.0.div_floor(ERGS_PER_GAS);
    let max_refund = gas_used / 5;
    core::cmp::min(full_refund_gas, max_refund)
};
gas_used = gas_used.saturating_sub(evm_refund);
```

### Proof of Concept

1. Deploy a contract that writes to 5 storage slots and then clears them (generating SSTORE refunds).
2. Submit an L2 transaction calling this contract with a gas price high enough that `native_per_gas` is non-trivial and pubdata from the storage writes causes `native_used / native_per_gas > gas_used * 0.8`.
3. Observe that `gas_refunded` in the `ZkTxResult` is non-zero (confirming EVM refunds were computed), but the actual token refund to the sender equals `(gas_limit - gas_used) * gas_price` where `gas_used = native_used / native_per_gas` — the `evm_refund` is not reflected in the token refund.
4. Compare the sender's balance change against `gas_limit * gas_price - (gas_used_before_refund - evm_refund) * gas_price` to confirm the shortfall of `evm_refund * gas_price`. [1](#0-0) [4](#0-3) [3](#0-2)
