Audit Report

## Title
Unbounded `ingress_expiries` Vec Allocation via Attacker-Controlled `ingress_end` Causes OOM/Infinite Loop DoS — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary

The `construction_payloads` function in the ICRC1 Rosetta server accepts attacker-controlled `ingress_start` and `ingress_end` values from the HTTP request body and enters an unbounded `while` loop that pushes one `u64` per iteration into a `Vec`. No guard caps the window size. Supplying `ingress_start=0` and `ingress_end=u64::MAX` causes the server to attempt allocating hundreds of gigabytes of memory (OOM), and because `ingress_start +=` wraps in Rust release mode, the loop becomes infinite after the wrap, permanently hanging the handler task. The same pattern exists in the ICP Rosetta server.

## Finding Description

**Entry point:** `POST /construction/payloads` → `endpoints::construction_payloads` → `services::construction_payloads`.

The endpoint handler passes raw metadata fields directly to the service with no pre-validation of the window size: [1](#0-0) 

Inside `services::construction_payloads`, the step size is computed as:
- `ingress_interval = (MAX_INGRESS_TTL − PERMITTED_DRIFT).as_nanos() as u64 = (300s − 60s) = 240_000_000_000 ns`
- `INGRESS_INTERVAL_OVERLAP = 120s = 120_000_000_000 ns`
- Effective step = `240_000_000_000 − 120_000_000_000 = 120_000_000_000 ns` [2](#0-1) [3](#0-2) 

The two existing guards only check ordering and minimum future reach — neither bounds the window size: [4](#0-3) 

With `ingress_start=0` and `ingress_end=u64::MAX`, both guards pass (`0 < u64::MAX` and `u64::MAX >= now + ingress_interval`). Execution falls into the unbounded loop: [5](#0-4) 

**Iteration count:** `u64::MAX / 120_000_000_000 ≈ 153,722,867,280` iterations, each pushing 8 bytes → ~1.2 TB of heap allocation attempted. The server OOMs long before completion. Additionally, because `ingress_start +=` is a plain `u64` addition in Rust release mode (wrapping semantics), once `ingress_start` wraps past `u64::MAX` it becomes a small value again, still less than `ingress_end = u64::MAX`, converting the OOM into a **permanent infinite loop** on the handler task.

The same unbounded loop pattern exists in the ICP Rosetta server with no window-size guard: [6](#0-5) 

## Impact Explanation

A single unauthenticated HTTP POST to `/construction/payloads` causes the Rosetta server process to exhaust all available heap memory and be killed by the OS OOM killer, or to spin indefinitely on the handler task. All concurrent and subsequent requests are denied service. This matches the allowed ICP bounty impact: **High ($2,000–$10,000) — Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS**, specifically targeting the Rosetta API, a listed in-scope financial integration component.

## Likelihood Explanation

The endpoint is publicly reachable with no authentication. The payload is a standard JSON body. The attack requires a single HTTP request and is trivially reproducible by any external party. No privileged access, key material, or protocol-level attack is needed. The attack is repeatable: if the process is restarted, a single new request re-triggers the condition.

## Recommendation

Add an explicit upper bound on the ingress window before entering the loop in both `rs/rosetta-api/icrc1/src/construction_api/services.rs` and `rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`. For example, in the ICRC1 service:

```rust
const MAX_INGRESS_WINDOW_ENTRIES: usize = 100; // ~200 minutes at 120s step

let step = ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
let window_size = ingress_end.saturating_sub(ingress_start);
if step == 0 || window_size / step > MAX_INGRESS_WINDOW_ENTRIES as u64 {
    return Err(Error::processing_construction_failed(
        "Ingress window exceeds maximum allowed size",
    ));
}
```

Apply the equivalent guard in the ICP Rosetta server before the `while now < ingress_end` loop.

## Proof of Concept

```bash
curl -X POST http://<rosetta-host>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"<network_id>"},
    "operations": [<valid_transfer_op>],
    "public_keys": [<valid_pubkey>],
    "metadata": {
      "ingress_start": 0,
      "ingress_end": 18446744073709551615
    }
  }'
```

Expected result: the Rosetta server process is killed by the OS OOM killer or the handler task spins indefinitely; all subsequent requests to the Rosetta server time out or are refused. A unit-level proof can be written by calling `services::construction_payloads` directly with `ingress_start=0`, `ingress_end=u64::MAX`, and a valid public key, and asserting that it returns an error rather than hanging.
