### Title
Silent ICP Burn Failure in CMC Allows ICP to Become Permanently Trapped - (`rs/nns/cmc/src/main.rs`)

### Summary
The Cycles Minting Canister (CMC) contains a `burn_and_log` function that intentionally silences all ICP burn failures after cycles have already been minted or deposited. If the ICP ledger rejects the burn call for any reason (e.g., transient unavailability during an upgrade), the cycles are irreversibly minted while the ICP remains permanently trapped in the CMC's subaccount. The notification is already marked as processed, so no retry is possible.

### Finding Description
In `rs/nns/cmc/src/main.rs`, the function `burn_and_log` is called after cycles have been successfully minted or deposited to a canister. Its explicit design is to swallow all errors:

The CMC uses a deduplication map to prevent re-notification. Once a notification is processed (regardless of whether the burn succeeded), it cannot be retried. The ICP sent to the CMC's subaccount is therefore permanently stranded.

### Impact Explanation
When `burn_and_log` fails silently:
1. Cycles have already been irreversibly minted or deposited to the target canister.
2. The ICP in the CMC's subaccount is **not** burned — it remains there permanently.
3. The notification block index is marked as already processed, preventing any retry.
4. The ICP total supply is not reduced as expected, violating the ICP conservation invariant: cycles were created without destroying the corresponding ICP.

This is a **ledger conservation bug**: the ICP supply is inflated relative to the cycles minted, and the stranded ICP in the CMC's subaccount can only be recovered via an NNS governance proposal — an out-of-band, privileged action not available to the affected user.

### Likelihood Explanation
The ICP ledger can transiently reject calls during canister upgrades, under heavy load, or if the ledger's message queue is full. These are realistic, non-adversarial conditions on mainnet. Any user who calls `notify_top_up` or `notify_mint_cycles` at the moment the ICP ledger is temporarily unavailable will trigger this condition. The user receives their cycles but the ICP is permanently trapped. The likelihood is low-to-medium per individual transaction but non-negligible at scale given the CMC's high transaction volume.

### Recommendation
`burn_and_log` should propagate burn failures rather than silencing them. The concern about retries is valid, but the correct fix is to use a persistent retry queue: record the pending burn in stable state before minting cycles, and retry the burn in a background task. This is the same pattern used by the ckBTC minter for its own retry logic. Alternatively, the burn should be attempted **before** minting cycles, so that a burn failure prevents the cycle mint rather than leaving ICP stranded.

### Proof of Concept
1. User sends 10 ICP to the CMC's subaccount via the ICP ledger.
2. User calls `notify_top_up` (or `notify_mint_cycles`) on the CMC.
3. CMC successfully calls `deposit_cycles` (or `do_mint_cycles`) — cycles are irreversibly deposited.
4. CMC calls `burn_and_log` to burn the 10 ICP. The ICP ledger is temporarily unavailable (e.g., mid-upgrade) and returns a reject code.
5. `burn_and_log` logs the error and returns `()` — no error is propagated.
6. `process_top_up` returns `Ok(cycles)` to the user.
7. The notification block index is recorded as processed in the CMC's `already_notified` map.
8. The 10 ICP remain in the CMC's subaccount indefinitely. The user has their cycles; the ICP supply is not reduced. [4](#0-3)

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
