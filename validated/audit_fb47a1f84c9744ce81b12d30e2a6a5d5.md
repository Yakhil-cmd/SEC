### Title
Unguarded `execute_scheduled` Allows Any Caller to Front-Run Scheduled XCC Promises, Causing Temporary Fund Freeze - (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router's `execute_scheduled` function is intentionally callable by any NEAR account. This creates a state-transition race condition: an attacker can front-run a user's intended execution of a scheduled promise at a moment when the target contract would reject the call. Because the promise is removed from storage before the cross-contract call executes, a failed promise is permanently consumed and cannot be retried. NEAR that was pre-funded to the router during scheduling is then frozen at the router with no built-in recovery path.

---

### Finding Description

**Root cause — `execute_scheduled` has no caller restriction:**

```rust
/// It is intentional that this function can be called by anyone (not just the parent).
/// There is no security risk to allowing this function to be open because it can only
/// act on promises that were created via `schedule`.
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
``` [1](#0-0) 

The comment's safety claim — "it can only act on promises that were created via `schedule`" — is correct about *what* executes, but ignores *when* it executes. The timing of execution is entirely uncontrolled.

**State-removal happens before the async call resolves:**

`self.scheduled_promises.remove(&nonce.0)` commits in the current NEAR transaction. The actual cross-contract call (`promise_create`) executes in a subsequent receipt. If that receipt fails (target contract panics, insufficient storage, wrong state), the promise entry is already gone from storage — it cannot be retried. [2](#0-1) 

**NEAR is pre-funded to the router during scheduling, not during execution:**

When a user uses `CrossContractCallArgs::Delayed` with `required_near > 0`, the engine's `handle_precompile_promise` burns the user's wNEAR ERC-20 tokens and sends NEAR to the router as part of the scheduling transaction — before `execute_scheduled` is ever called. [3](#0-2) 

**No recovery path exists in the router:**

The router has no general-purpose withdrawal function. The only egress for NEAR is through scheduled promises or `send_refund`, which is restricted to the parent (engine) and limited to a fixed `REFUND_AMOUNT` of 2 NEAR. [4](#0-3) 

**`schedule` and `execute` are parent-only; `execute_scheduled` is not:**

```rust
pub fn execute(&self, #[serializer(borsh)] promise: PromiseArgs) {
    self.assert_preconditions();   // parent-only
    ...
}
pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
    self.assert_preconditions();   // parent-only
    ...
}
// execute_scheduled: NO assert_preconditions()
``` [5](#0-4) 

---

### Impact Explanation

**High — Temporary freezing of funds.**

Attack outcome:
1. User's wNEAR ERC-20 tokens are burned during scheduling (irreversible at that point).
2. NEAR is sent to the router and held there.
3. Attacker front-runs `execute_scheduled`; the promise executes against a target contract in a state that causes it to fail (e.g., contract paused, storage not registered, wrong precondition).
4. The promise is consumed from storage; NEAR is refunded by NEAR runtime back to the router (not to the user).
5. The user's NEAR is frozen at the router with no direct recovery mechanism.
6. Recovery requires the user to schedule a new promise via the XCC precompile, burning additional wNEAR ERC-20 tokens.

In the edge case where the user has no remaining wNEAR to fund a recovery promise, the freeze becomes permanent — escalating to **Critical: Permanent freezing of funds**.

---

### Likelihood Explanation

**Medium.** An attacker needs to:
1. Monitor the NEAR chain for `schedule` calls on XCC router sub-accounts (all sub-accounts follow the deterministic pattern `{evm_address}.aurora`).
2. Identify a scheduled promise whose target contract is transiently in a rejecting state (paused, unregistered storage, etc.).
3. Call `execute_scheduled` before the user does.

No privileged access is required. The attacker is an ordinary NEAR account. The NEAR chain is public, so scheduled promises and their target contracts are fully observable. This is a realistic MEV/griefing vector for any protocol that uses the XCC `Delayed` mode with NEAR-attached promises.

---

### Recommendation

1. **Restrict `execute_scheduled` to the parent (engine) or the scheduling EVM address.** The engine knows which EVM address owns each router sub-account and can enforce that only the owner triggers execution.
2. **Alternatively, add a recovery function** that allows the parent to return stranded NEAR to the user's router balance or back to the engine, callable when a scheduled promise has been consumed but the NEAR remains.
3. **At minimum, document the timing risk** so integrators using `Delayed` mode understand that their promises can be executed by any third party at any time.

### Proof of Concept

```
1. User (EVM address 0xABCD) calls XCC precompile with:
     CrossContractCallArgs::Delayed(PromiseArgs::Create(PromiseCreateArgs {
         target_account_id: "some_dex.near",
         method: "swap",
         args: ...,
         attached_balance: Yocto::new(1_000_000_000_000_000_000_000_000), // 1 NEAR
         attached_gas: NearGas::new(50_000_000_000_000),
     }))

2. Engine burns 1 wNEAR ERC-20 from 0xABCD's balance.
   Engine sends 1 NEAR to router sub-account "abcd...abcd.aurora".
   Router stores promise at nonce=0.

3. "some_dex.near" is temporarily paused by its admin.

4. Attacker (any NEAR account) calls:
     "abcd...abcd.aurora".execute_scheduled({"nonce": "0"})

5. Router removes nonce=0 from scheduled_promises (committed immediately).
   Router creates NEAR promise to call "some_dex.near".swap(...).

6. "some_dex.near" panics: "ERR_CONTRACT_PAUSED".
   NEAR runtime refunds 1 NEAR back to "abcd...abcd.aurora" (the router).

7. Result:
   - 0xABCD's wNEAR ERC-20 balance: burned (irreversible)
   - Router's scheduled_promises: empty (nonce=0 consumed)
   - Router's NEAR balance: +1 NEAR (frozen, no recovery path)
   - User cannot retry the swap without burning more wNEAR.
```

The entry path is fully reachable by an unprivileged NEAR account with no special keys or permissions.
