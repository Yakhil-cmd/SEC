### Title
Silent ERC-20 Burn Without NEP-141 Transfer in `EvmErc20V2.withdrawToNear` Due to Unchecked Precompile Return Value — (File: `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20V2.sol` encodes the caller's address into the `withdrawToNear` precompile calldata in a format that is incompatible with the production precompile (compiled without the `error_refund` feature). The precompile rejects the call with `ERR_INVALID_AMOUNT`, but the Solidity assembly does not check the return value. The ERC-20 tokens are already burned at this point and the transaction succeeds, permanently destroying the user's funds with no corresponding NEP-141 transfer on NEAR.

---

### Finding Description

**Root cause — calldata format mismatch between `EvmErc20V2.sol` and the non-`error_refund` precompile:**

`EvmErc20V2.withdrawToNear` encodes:

```
\x01 | sender (20 bytes) | amount_b (32 bytes) | recipient (variable)
``` [1](#0-0) 

`EvmErc20.sol` (v1) encodes:

```
\x01 | amount_b (32 bytes) | recipient (variable)
``` [2](#0-1) 

The `ExitToNear` precompile has two parsing paths gated by the `error_refund` compile-time feature:

- **With `error_refund`**: reads 20 bytes after the flag as `refund_address`, then parses `amount` from the next 32 bytes. This is compatible with `EvmErc20V2.sol`.
- **Without `error_refund`** (production default): skips only the flag byte, then reads the next 32 bytes as `amount`. This is compatible with `EvmErc20.sol` (v1) but **not** with `EvmErc20V2.sol`. [3](#0-2) 

When `EvmErc20V2.sol` is used with the non-`error_refund` precompile, the 32 bytes parsed as `amount` are:

```
sender_address (20 bytes) || first 12 bytes of amount_b
```

This value is always far greater than `u128::MAX`, so `parse_amount` returns `ERR_INVALID_AMOUNT`: [4](#0-3) 

**Root cause — unchecked precompile return value:**

The assembly in `EvmErc20V2.withdrawToNear` captures the call result in `res` but never checks it:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc7..., 0, add(input, 32), input_size, 0, 32)
}
``` [5](#0-4) 

Because `_burn` executes before the precompile call and the failed call does not revert the transaction, the ERC-20 tokens are destroyed with no NEP-141 transfer ever scheduled.

**`error_refund` is NOT a production default:**

`engine/Cargo.toml` shows `error_refund` is an optional feature absent from both `default` and `contract` feature sets: [6](#0-5) [7](#0-6) 

The same unchecked-return pattern exists in `EvmErc20.sol` (v1), but v1's calldata layout is correct for the non-`error_refund` precompile, so v1 only silently fails for other error conditions (e.g., unregistered token, invalid recipient). `EvmErc20V2.sol` fails on **every** `withdrawToNear` call.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Every invocation of `EvmErc20V2.withdrawToNear` by any unprivileged user results in:
1. The caller's ERC-20 mirror tokens being burned on Aurora (irreversible within the transaction).
2. No NEP-141 `ft_transfer` promise being created on NEAR.
3. The underlying NEP-141 tokens remaining locked in the bridge with no recovery path for the caller.

The user loses the full `amount` of bridged tokens permanently.

---

### Likelihood Explanation

**High.** The failure is deterministic and affects 100% of `withdrawToNear` calls on `EvmErc20V2.sol` when the engine is compiled without `error_refund` (the production default). Any user who holds a NEP-141-backed ERC-20 token deployed via the V2 contract and attempts to bridge back to NEAR will silently lose their funds. No special attacker setup is required; ordinary user behavior triggers the loss.

---

### Recommendation

1. **Check the precompile return value and revert on failure** in both `EvmErc20.sol` and `EvmErc20V2.sol`:

```solidity
assembly {
    let res := call(gas(), PRECOMPILE_ADDR, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

2. **Align calldata formats**: `EvmErc20V2.sol` should only include the sender prefix when the engine is compiled with `error_refund`, or the precompile should be updated to handle both formats.

3. **Enable `error_refund` by default** in the `contract` feature, or document clearly which contract version is compatible with which precompile build.

---

### Proof of Concept

**Setup**: Aurora Engine compiled without `error_refund` (production default). `EvmErc20V2.sol` deployed as the ERC-20 mirror for some NEP-141 token.

**Steps**:

1. Alice holds 1000 units of the ERC-20 mirror token on Aurora.
2. Alice calls `EvmErc20V2.withdrawToNear("alice.near", 1000e6)`.
3. `_burn(alice, 1000e6)` executes — Alice's ERC-20 balance drops to 0.
4. Precompile calldata assembled: `\x01 || alice_evm_address (20 bytes) || bytes32(1000e6) (32 bytes) || "alice.near"`.
5. Precompile (non-`error_refund`) reads bytes `[1..33]` as amount: `alice_evm_address (20 bytes) || first 12 bytes of bytes32(1000e6)`. This value ≫ `u128::MAX`.
6. `parse_amount` returns `Err(ExitError::Other("ERR_INVALID_AMOUNT"))`.
7. EVM call to precompile returns `res = 0`; assembly does not check `res`.
8. Transaction succeeds. Alice's 1000 ERC-20 tokens are gone. No NEP-141 transfer is ever created. The NEP-141 tokens remain locked in the bridge forever. [1](#0-0) [4](#0-3) [3](#0-2) [8](#0-7)

fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);

    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
