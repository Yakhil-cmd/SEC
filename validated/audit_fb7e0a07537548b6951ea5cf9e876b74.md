### Title
L1→L2 Priority Transaction Can Permanently Halt Block Processing via `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` Revert - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The `notify_l2_asset_tracker` function in the ZKsync OS bootloader explicitly treats any revert from `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` as a **fatal, unrecoverable system error** that halts block processing entirely. This call is made for every L1→L2 priority transaction that deposits base tokens. Because L1 priority transactions cannot be skipped or invalidated once committed to the L1 priority queue, a crafted or edge-case L1 transaction that causes the `L2AssetTracker` to revert will permanently stall the ZKsync OS chain — no further blocks can be produced until a governance upgrade replaces the `L2AssetTracker` bytecode.

---

### Finding Description

In `process_l1_transaction`, after the main transaction body executes, the bootloader calls `mint_base_token` up to three times (value mint, operator fee, refund) to notify the `L2AssetTracker` contract and transfer tokens from the treasury. Each call routes through `notify_l2_asset_tracker`:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs:901-911
if failed {
    // A revert here means the chain's token accounting would be inconsistent.
    // Treated as a fatal system error — block processing cannot continue.
    return Err(internal_error!(
        "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
    )
    .into());
}
```

The code comment itself acknowledges this is a **fatal, unrecoverable halt**. The `internal_error!` propagates as a `BootloaderSubsystemError::LeafDefect`, which is not caught at the transaction level — it propagates up through `process_l1_transaction` → the ZK tx loop → and terminates block execution entirely. [1](#0-0) 

The call is made with `with_infinite_ergs`, meaning it cannot fail due to gas exhaustion — only an EVM-level revert from the `L2AssetTracker` contract logic itself triggers the fatal path: [2](#0-1) 

The `L2AssetTracker` is called for every L1 transaction with `amount > U256::ZERO`: [3](#0-2) 

This call is made three separate times per L1 deposit transaction — once for the value mint (inside the execution frame), once for the operator fee, and once for the refund: [4](#0-3) [5](#0-4) 

The `L2AssetTracker` contract at address `0x1000f` is a real EVM contract with complex logic (ownership checks, asset registration checks, migration number tracking, cross-chain settlement layer checks). Any of these internal checks reverting — due to uninitialized state, a bug in the contract, or a state manipulation — will trigger the fatal halt. [6](#0-5) 

The `L1_TX_INTRINSIC_NATIVE_COST` comment explicitly acknowledges that "first mint / call to L2AssetTracker can fail due to out-of-native": [7](#0-6) 

The ZK tx loop propagates `TxError::Internal` as a block-terminating error: [8](#0-7) 

---

### Impact Explanation

**Severity: Critical — Chain Halt / Permanent Denial of Service**

If any L1→L2 priority transaction with a non-zero deposit causes `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` to revert, the entire block processing halts with a fatal internal error. Because L1 priority transactions are committed to the L1 priority queue and **cannot be skipped or invalidated** (the code explicitly notes this), the chain becomes permanently stuck:

- The failing L1 transaction will be replayed at the head of every new block attempt.
- Every block attempt will hit the same fatal error.
- No L2 transactions can be processed.
- The chain is halted until a governance upgrade replaces the `L2AssetTracker` bytecode — which itself requires a protocol upgrade transaction, which is also an L1 transaction that must pass through the same broken path.

This is a **complete chain halt** analogous to the dForce `_updateInterest` halt: a mandatory system call that cannot be bypassed, whose revert terminates all further execution with no recovery path available to the operator.

---

### Likelihood Explanation

**Medium likelihood.** The `L2AssetTracker` contract has multiple internal checks that could revert:

1. **Asset not registered**: If `isAssetRegistered[assetId]` is `false`, the contract may revert.
2. **Migration number mismatch**: The `assetMigrationNumber` check could fail if state is inconsistent.
3. **Settlement layer chain ID mismatch**: The `SystemContext.currentSettlementLayerChainId()` call could return an unexpected value.
4. **Contract upgrade bug**: A future upgrade to `L2AssetTracker` that introduces a revert path in `handleFinalizeBaseTokenBridgingOnL2` would immediately halt the chain on the next L1 deposit.
5. **Out-of-native during the call**: Although `with_infinite_ergs` is used for ergs, native resources are still tracked. The comment at line 194 of `constants.rs` explicitly warns "first mint / call to L2AssetTracker can fail due to out-of-native." If native resources are exhausted inside the `L2AssetTracker` call, this propagates as a `FatalRuntimeError`, which is caught and re-wrapped as an internal error at lines 305-307, also halting the block.

An attacker who can influence the `L2AssetTracker` state (e.g., by manipulating storage slots via a separate L2 transaction before the L1 deposit is processed) could deliberately trigger this condition.

---

### Recommendation

1. **Remove the fatal halt on `L2AssetTracker` revert.** Instead of treating a revert as a block-terminating error, log the failure and continue block processing. Token accounting inconsistency is a lesser evil than a permanent chain halt.

2. **Add a recovery escape hatch.** Similar to the dForce fix (removing `settleInterest` from `_setInterestRateModel`), add a privileged operator path that can process L1 transactions while skipping the `L2AssetTracker` notification, allowing the chain to drain the priority queue even when the asset tracker is broken.

3. **Bound native resource consumption.** The `with_infinite_ergs` call still consumes native resources. Add an explicit native resource cap on the `L2AssetTracker` call to prevent out-of-native from propagating as a fatal error.

4. **Decouple the asset tracker notification from the mandatory L1 tx processing path.** The accounting update could be queued and retried rather than being a synchronous prerequisite for block finalization.

---

### Proof of Concept

1. Attacker submits an L1→L2 priority transaction with `to_mint > 0` (any non-zero deposit) to the ZKsync L1 diamond contract. This enqueues the transaction in the L1 priority queue.

2. Before the sequencer processes this transaction, the attacker (or a separate mechanism) manipulates the `L2AssetTracker` storage at `0x1000f` to put it in a state where `handleFinalizeBaseTokenBridgingOnL2` will revert (e.g., clearing `isAssetRegistered` for the base token asset ID, or corrupting the `assetMigrationNumber`).

3. The sequencer attempts to produce a new block. The L1 priority transaction is at the head of the queue and must be processed first.

4. `process_l1_transaction` calls `mint_base_token` → `notify_l2_asset_tracker` → `run_single_interaction` on `L2AssetTracker`.

5. `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` reverts due to the manipulated state.

6. `notify_l2_asset_tracker` returns `Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"))`.

7. This propagates as `BootloaderSubsystemError::LeafDefect` → `TxError::Internal` → the ZK tx loop returns `Err(err)` at line 112, terminating block execution.

8. Every subsequent block attempt replays the same L1 transaction and hits the same fatal error. The chain is permanently halted. [9](#0-8) [10](#0-9) [8](#0-7)

/// Notify L2AssetTracker about base token bridging from L1.
///
/// Calls handleFinalizeBaseTokenBridgingOnL2(uint256 _fromChainId, uint256 _amount)
/// as L2_BASE_TOKEN_ADDRESS (0x800a) to pass the onlyBaseTokenHolderOrL2BaseToken modifier.
///
/// This is called separately for each token movement (value mint, operator
/// payment, refund) so that the asset tracker's accounting stays correct even
/// if the main transaction body reverts.
///
/// Resource usage depends on the caller — value-mint tracks native against user resources;
/// operator-fee and refund use FORMAL_INFINITE.
///
/// Failure halts block processing — if the asset tracker reverts, the
/// chain's token accounting would be inconsistent, so we treat it as
/// fatal rather than silently continuing with incorrect bookkeeping.
///
/// If no contract is deployed at L2AssetTracker, the call succeeds silently
/// (a call to an empty address returns success with no returndata in EVM).
/// However, we are certain that L2AssetTracker is available after the upgrade.
fn notify_l2_asset_tracker<'a, S: EthereumLikeTypes + 'a, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    system_functions: &mut HooksStorage<S, S::Allocator>,
    memories: RunnerMemoryBuffers<'a>,
    amount: U256,
    l1_chain_id: U256,
    resources: &mut S::Resources,
    tracer: &mut impl Tracer<S>,
    validator: &mut impl TxValidator<S>,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
    S::Metadata: ZkSpecificPricingMetadata
        + BasicMetadata<S::IOTypes, TransactionMetadata = TxLevelMetadata<S::IOTypes>>,
{
    if amount > U256::ZERO || Config::SIMULATION {
        // Encode calldata for handleFinalizeBaseTokenBridgingOnL2(uint256,uint256):
        // selector 0x03117c8c + abi-encoded (fromChainId, amount)
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]);
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution {
                resources_returned,
                result: asset_tracker_result,
            } = BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system,
                system_functions,
                memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,
                inf_ergs.clone(),
                &U256::ZERO,
                true, // should_make_frame - isolate state changes
                tracer,
                validator,
            )?;
            // Overwrite resources inside the closure so that
            // with_infinite_ergs correctly restores ergs afterwards.
            *inf_ergs = resources_returned;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
        })?;

        if failed {
            system_log!(
                system,
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 failed for amount {amount:?}\n"
            );
            // A revert here means the chain's token accounting would be inconsistent.
            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
            )
            .into());
        }
    }
    Ok(())
}
