### Title
TOCTOU Race: `count_incomplete_retrieve_btc_requests` Check Bypassed Across Async Awaits, Allowing `pending_retrieve_btc_requests` to Exceed `MAX_CONCURRENT_PENDING_REQUESTS` — (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

---

### Summary

In both `retrieve_btc` and `retrieve_btc_with_approval`, the `MAX_CONCURRENT_PENDING_REQUESTS` (5000) guard is a synchronous read-only snapshot check that is not atomic with the eventual `mutate_state(accept_retrieve_btc_request)` call. Three inter-canister `.await` points separate the check from the mutation. Under the IC execution model, each `.await` is a yield point where other ingress messages — including other concurrent `retrieve_btc` calls — can be scheduled and processed. Up to `MAX_CONCURRENT = 100` concurrent calls (the `retrieve_btc_guard` ceiling) can all pass the count check simultaneously before any of them commits to state, allowing the pending queue to grow to at most `5000 + 100 = 5100` entries.

---

### Finding Description

**Entrypoint:** Any unprivileged principal calls `retrieve_btc` (or `retrieve_btc_with_approval`) via ingress.

**Execution flow in `retrieve_btc`:**

```
line 162: retrieve_btc_guard acquired (atomic, MAX_CONCURRENT=100 cap)
line 174: read_state(count_incomplete_retrieve_btc_requests() >= 5000)  ← CHECK
line 181: balance_of(caller).await   ← YIELD POINT 1
line 187: check_address(...).await   ← YIELD POINT 2
line 210: burn_ckbtcs(...).await     ← YIELD POINT 3
line 232: mutate_state(accept_retrieve_btc_request)  ← MUTATION
```

The check at line 174 reads a snapshot of the count. At each `.await`, the IC scheduler can interleave other `retrieve_btc` messages. If N callers all read the count as `< 5000` before any of them reaches line 232, all N will eventually call `mutate_state`, pushing the count to `existing_count + N`.

The `retrieve_btc_guard` (line 162) is the only mechanism that bounds N. It uses a `BTreeSet<Account>` capped at `MAX_CONCURRENT = 100`. [1](#0-0) 

The guard is acquired atomically before the count check, so at most 100 distinct accounts can be simultaneously in-flight. This bounds the maximum overshoot to 100 extra entries. [2](#0-1) 

The same pattern exists identically in `retrieve_btc_with_approval`: [3](#0-2) 

`accept_retrieve_btc_request` unconditionally pushes to `pending_retrieve_btc_requests` with no re-check of the limit: [4](#0-3) 

`count_incomplete_retrieve_btc_requests` sums three collections: [5](#0-4) 

---

### Impact Explanation

- The invariant `count_incomplete_retrieve_btc_requests() <= MAX_CONCURRENT_PENDING_REQUESTS` is violated.
- Maximum reachable count: `5000 + 100 = 5100` (bounded by `MAX_CONCURRENT = 100` in the guard).
- Each extra entry is a `RetrieveBtcRequest` struct persisted in heap state, causing bounded heap growth.
- The minter's `build_batch` and `submit_pending_requests` logic iterates over `pending_retrieve_btc_requests`, so a larger-than-expected queue increases per-round processing cost and degrades latency.
- No fund loss or double-spend is possible: each request requires a successful `burn_ckbtcs` ledger call, so each entry corresponds to real ckBTC already burned.

---

### Likelihood Explanation

- Requires 100 distinct principals each holding sufficient ckBTC — a real but non-trivial cost.
- The IC message scheduler naturally interleaves concurrent ingress calls at `.await` points; no special tooling is needed.
- The overshoot is bounded (2% above the limit) and self-correcting once the guard slots free up.
- Likelihood is **low-to-medium**: the precondition of ~5000 existing pending requests is itself unusual in normal operation, but the race is mechanically sound and locally reproducible.

---

### Recommendation

Re-check the count inside `mutate_state` immediately before pushing, so the check and the push are atomic:

```rust
mutate_state(|s| {
    if s.count_incomplete_retrieve_btc_requests() >= MAX_CONCURRENT_PENDING_REQUESTS {
        return Err(RetrieveBtcError::TemporarilyUnavailable(...));
    }
    state::audit::accept_retrieve_btc_request(s, request, &IC_CANISTER_RUNTIME);
    Ok(())
})?;
```

This eliminates the TOCTOU window entirely because `mutate_state` holds exclusive mutable access to the state for the duration of the closure, and no `.await` can occur inside it.

---

### Proof of Concept

1. Pre-populate the minter with 4999 pending `retrieve_btc` requests (via 4999 successful prior calls).
2. Spawn 100 concurrent ingress calls to `retrieve_btc` from 100 distinct principals, each with sufficient ckBTC balance.
3. All 100 calls read `count = 4999 < 5000` at line 174 before any reaches line 232.
4. All 100 calls complete `balance_of`, `check_address`, and `burn_ckbtcs` successfully.
5. All 100 calls execute `mutate_state(accept_retrieve_btc_request)`.
6. Assert: `read_state(|s| s.count_incomplete_retrieve_btc_requests())` returns `5099`, violating the `<= 5000` invariant.

pub fn accept_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    runtime: &R,
) {
    record_event(
        EventType::AcceptedRetrieveBtcRequest(request.clone()),
        runtime,
    );
    state.pending_retrieve_btc_requests.push(request.clone());
    if let Some(account) = request.reimbursement_account {
        state
            .retrieve_btc_account_to_block_indices
            .entry(account)
            .and_modify(|entry| entry.push(request.block_index))
            .or_insert(vec![request.block_index]);
    }
    if let Some(kyt_provider) = request.kyt_provider {
        *state.owed_kyt_amount.entry(kyt_provider).or_insert(0) += state.check_fee;
    }
}
