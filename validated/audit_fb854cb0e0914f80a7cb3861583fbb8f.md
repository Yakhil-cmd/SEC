### Title
Unbounded Recursion in `recursive_promise_create` Can Cause OOG Panic and Permanent Fund Freeze — (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router's `recursive_promise_create` function processes user-supplied `NearPromise` trees without any depth or size bound. A sufficiently deep `NearPromise::Then` nesting or a large `NearPromise::And` array causes a stack overflow or out-of-gas (OOG) panic inside `execute` or `execute_scheduled`. Because NEAR funds are transferred to the router in an earlier promise step (wNEAR unwrapping), a panic in the subsequent `execute` call leaves those funds permanently stranded in the router with no recovery path.

---

### Finding Description

`recursive_promise_create` in `etc/xcc-router/src/lib.rs` is a mutually recursive function that processes the `NearPromise` enum:

```
NearPromise::Then { base, callback } => {
    let base_index = Self::recursive_promise_create(base);   // unbounded recursion
    ...
}
NearPromise::And(promises) => {
    let indices: Vec<PromiseIndex> = promises
        .iter()
        .map(Self::recursive_promise_create)   // unbounded iteration + recursion
        .collect();
    env::promise_and(&indices)
}
``` [1](#0-0) 

There is no depth counter, no maximum nesting limit, and no maximum `And`-branch count enforced anywhere before or inside this function. A `NearPromise::Then { base: NearPromise::Then { base: … } }` chain of depth *N* causes *N* recursive Rust stack frames. A `NearPromise::And(vec![…; M])` causes *M* recursive calls per element.

The same unbounded pattern appears in `add_batch_actions` (enabled under `all-promise-actions`):

```rust
fn add_batch_actions(id: PromiseIndex, actions: &[PromiseAction]) {
    for action in actions.iter() { … }
}
``` [2](#0-1) 

Both `execute` and `execute_scheduled` call `Self::promise_create(promise)` which dispatches to `recursive_promise_create` for the `PromiseArgs::Recursive` variant:

```rust
fn promise_create(promise: PromiseArgs) -> PromiseIndex {
    match promise {
        PromiseArgs::Create(call) => Self::base_promise_create(&call),
        PromiseArgs::Callback(cb) => Self::cb_promise_create(&cb),
        PromiseArgs::Recursive(p) => Self::recursive_promise_create(&p),
    }
}
``` [3](#0-2) 

`execute_scheduled` is explicitly open to **any caller** (not just the parent engine):

```rust
/// It is intentional that this function can be called by anyone (not just the parent).
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
``` [4](#0-3) 

The fund-freeze mechanism is rooted in the XCC promise chain constructed in `engine/src/xcc.rs`. The engine first withdraws wNEAR and converts it to real NEAR (sending it to the router), then attaches `execute` as a callback:

```rust

    None => handler.promise_create_call(&withdraw_call),
    Some(setup_id) => handler.promise_attach_callback(setup_id, &withdraw_call),
};
``` [5](#0-4) 

```rust
match withdraw_id {
    None => handler.promise_create_call(promise),
    Some(withdraw_id) => handler.promise_attach_callback(withdraw_id, promise),
}
``` [6](#0-5) 

The NEAR is already resident in the router when `execute` runs. If `execute` panics (OOG or stack overflow from unbounded recursion), NEAR's promise model reverts only the callback's own state changes — it does **not** automatically return NEAR that was deposited in a prior promise step. The router has no sweep or recovery function for stranded NEAR.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any NEAR that was unwrapped from wNEAR and sent to the router in the preceding promise step is irrecoverably stranded if `execute` or `execute_scheduled` panics. The router exposes no admin withdrawal or recovery method for arbitrary NEAR balances. The only refund path (`send_refund`) returns a fixed `REFUND_AMOUNT` (2 NEAR storage stake) and is unrelated to user-deposited NEAR. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** The `PromiseArgs::Recursive` variant and `NearPromise` type are part of the public Borsh-serialized interface consumed by `execute` and `schedule`. The Aurora Engine's XCC precompile constructs `PromiseArgs` from user-supplied EVM calldata. A user who can invoke the XCC precompile (any EVM address) can supply a deeply nested `NearPromise::Then` chain or a large `NearPromise::And` array. Once scheduled, `execute_scheduled` is callable by anyone, so even a third party can trigger the panic after the engine has scheduled the promise and the NEAR is in the router.

---

### Recommendation

1. **Add a depth limit** to `recursive_promise_create`. Maintain a counter and panic with a clear error if depth exceeds a safe constant (e.g., 10).
2. **Add a size limit** to `NearPromise::And` branches and `PromiseBatchAction.actions` slices, analogous to how `_distributeFunds` in the referenced report caps `MAX_ROYALTY_RECIPIENTS_INDEX`.
3. **Enforce the same limits** at the XCC precompile layer in the engine before the promise is serialized and sent to the router, so malformed inputs are rejected before any NEAR is moved.

---

### Proof of Concept

1. Attacker constructs EVM calldata encoding a `PromiseArgs::Recursive(NearPromise::Then { base: NearPromise::Then { base: … (depth 500) … } })`.
2. Attacker calls the Aurora XCC precompile with sufficient wNEAR attached.
3. Engine executes `handle_precompile_promise`: wNEAR is burned, NEAR is transferred to the router sub-account, and `execute` is scheduled as a callback. [8](#0-7) 
4. NEAR runtime invokes `execute` on the router. `promise_create` dispatches to `recursive_promise_create`. [9](#0-8) 
5. After ~500 recursive Rust frames, the NEAR Wasm stack overflows or gas is exhausted; the call panics.
6. NEAR reverts the callback's state changes but the NEAR already transferred to the router in step 3 remains there permanently — no recovery path exists.
