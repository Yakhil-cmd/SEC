### Title
wNEAR Permanently Stuck in Engine's Implicit EVM Address When XCC Router Deployment Promise Fails — (`engine-precompiles/src/xcc.rs`)

---

### Summary

The XCC precompile synchronously transfers wNEAR from the calling EVM address to the engine's implicit EVM address before any NEAR promise is scheduled. If the subsequent asynchronous NEAR promise chain (router deployment → `withdraw_wnear_to_router`) fails, no failure-path callback exists to refund the wNEAR. The engine's implicit address has no private key and no user-accessible withdrawal function, so the wNEAR is permanently frozen.

---

### Finding Description

**Step 1 — Synchronous wNEAR transfer to an intermediary address**

Inside `CrossContractCall::run_with_handle` in `engine-precompiles/src/xcc.rs`, when `required_near != ZERO_YOCTO`, the precompile immediately executes an EVM `transferFrom` that moves wNEAR from the caller (`sender`) to `engine_implicit_address`:

```rust
let engine_implicit_address = aurora_engine_sdk::types::near_account_to_evm_address(
    self.engine_account_id.as_bytes(),
);
let tx_data = transfer_from_args(
    sender.0.into(),
    engine_implicit_address.raw().0.into(),
    required_near.as_u128().into(),
);
// ...
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
```

This EVM state change is committed immediately, before any NEAR promise is created. [1](#0-0) 

**Step 2 — Asynchronous promise chain with no failure callback**

`handle_precompile_promise` in `engine/src/xcc.rs` then builds the NEAR promise chain:

1. **Setup batch** (router deployment, if needed) — `promise_create_batch` / `promise_attach_batch_callback`
2. **`factory_update_address_version` callback** — updates the engine's record of the router version
3. **`withdraw_wnear_to_router` callback** — only runs if step 1+2 succeed

```rust
let setup_id = match &deploy_needed {
    AddressVersionStatus::DeployNeeded { create_needed } => {
        // ... build deployment batch ...
        let promise_id = match base_id {
            Some(id) => handler.promise_attach_batch_callback(id, &batch),
            None => handler.promise_create_batch(&batch),
        };
        Some(handler.promise_attach_callback(promise_id, &callback))
    }
    AddressVersionStatus::UpToDate => base_id,
};
// ...
let withdraw_id = if required_near == ZERO_YOCTO {
    setup_id
} else {
    // withdraw_wnear_to_router is chained as a callback of setup_id
    let id = match setup_id {
        None => handler.promise_create_call(&withdraw_call),
        Some(setup_id) => handler.promise_attach_callback(setup_id, &withdraw_call),
    };
```

There is **no failure-path callback** on the setup batch. If the router deployment fails, `withdraw_wnear_to_router` never executes, and the wNEAR remains in `engine_implicit_address`. [2](#0-1) 

**Step 3 — The intermediary address has no withdrawal mechanism**

`engine_implicit_address` is `near_account_to_evm_address(engine_account_id.as_bytes())` — a deterministic EVM address derived from the NEAR account ID. It has no EVM contract code, no private key, and no user-accessible engine method to recover ERC-20 tokens held there. There is no `recover_wnear` or equivalent function anywhere in the contract methods. [3](#0-2) 

---

### Impact Explanation

**Permanent freezing of funds.** Any wNEAR transferred to `engine_implicit_address` during a failed XCC router deployment is irrecoverable by the user. The amount equals `attached_near + STORAGE_AMOUNT` (up to `STORAGE_AMOUNT = 2 NEAR` for a new router account, plus any user-attached NEAR for the call itself). [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

Two realistic failure paths exist:

1. **Race condition**: Two XCC calls from the same EVM address submitted in the same NEAR block both observe `target_code_version = None` (the `factory_update_address_version` callback from the first call has not yet executed). Both attempt `CreateAccount` on the same router sub-account. The second deployment batch fails because the account already exists, leaving the second call's wNEAR stuck.

2. **Inconsistent router state**: If a previous deployment attempt failed after `CreateAccount` but before `factory_update_address_version` ran, the NEAR account exists but the engine's storage still shows `None` for that address. The next XCC call again attempts `CreateAccount`, which fails, stranding the wNEAR. [6](#0-5) [7](#0-6) 

---

### Recommendation

Add a failure-path callback after the router deployment batch. On failure, this callback should call back into the engine to re-credit the wNEAR (or equivalent NEAR) to the original EVM caller's address. Concretely:

- Attach a `promise_attach_callback` to the setup batch that fires on failure.
- In that callback, transfer the wNEAR from `engine_implicit_address` back to `sender` (or mint the equivalent amount directly to `sender`'s EVM balance).

Alternatively, perform the wNEAR transfer from `sender` to `engine_implicit_address` only inside a NEAR promise (not synchronously in the EVM), so that the EVM state change is atomic with the NEAR promise outcome.

---

### Proof of Concept

1. Alice (EVM address `0xAlice`) calls the XCC precompile for the first time, attaching 2.5 NEAR worth of wNEAR.
2. The precompile transfers 2.5 NEAR of wNEAR from `0xAlice` to `engine_implicit_address` (EVM state committed).
3. Bob (same EVM address, different NEAR transaction in the same block) also calls the XCC precompile, attaching 2.5 NEAR of wNEAR. The engine still shows `target_code_version = None` for `0xAlice`'s router (the callback from step 2 hasn't run yet). Bob's wNEAR is also transferred to `engine_implicit_address`.
4. Block N+1: Alice's router deployment succeeds; `factory_update_address_version` runs; router version is set.
5. Block N+1: Bob's router deployment fails (`CreateAccount` rejected — account already exists). The `withdraw_wnear_to_router` callback is never scheduled.
6. Bob's 2.5 NEAR of wNEAR is permanently frozen in `engine_implicit_address`, which has no private key and no recovery function.

impl AddressVersionStatus {
    fn new<I: IO>(
        io: &I,
        latest_code_version: CodeVersion,
        target_code_version: Option<CodeVersion>,
    ) -> Self {
        let first_upgradable_version =
            get_first_upgradable_version(io).unwrap_or(CodeVersion::ZERO);
        match target_code_version {
            None => Self::DeployNeeded {
                create_needed: true,
            },
            Some(version) if version < first_upgradable_version => {
                // It is impossible to upgrade the initial XCC routers because
                // they lack the upgrade method.
                Self::UpToDate
            }
            Some(version) if version < latest_code_version => Self::DeployNeeded {
                create_needed: false,
            },
            Some(_version) => Self::UpToDate,
        }
    }
