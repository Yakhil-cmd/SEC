### Title
Silent Fund Loss on Failed `ExitToNear` Cross-Contract Call When `error_refund` Feature Is Disabled - (File: `engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` Cargo feature is not enabled (which is the case in the default build), the `ExitToNear` precompile does not attach any refund callback to the NEAR-side `ft_transfer` / `ft_transfer_call` promise. If that promise fails, the user's ETH (or burned ERC-20 tokens) is permanently lost with no recovery path. This is the direct Aurora Engine analog of the TON `SendIgnoreErrors` silent-drop bug: funds are debited from the user before the cross-chain transfer, and a failure of that transfer is silently ignored.

---

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) handles two exit flows:

**ETH (base token) exit:** During EVM execution the caller's ETH is transferred to the `exit_to_near::ADDRESS` precompile address. A NEAR cross-contract call (`ft_transfer` or `ft_transfer_call`) is then scheduled to move the corresponding NEP-141 tokens to the recipient.

**ERC-20 exit:** The ERC-20 contract burns the caller's tokens, then calls the precompile, which schedules an `ft_transfer` on the NEP-141 contract.

In both cases the EVM-side debit happens **before** the NEAR-side promise is dispatched. Whether a refund callback is attached depends entirely on the `error_refund` compile-time feature:

```rust
// engine-precompiles/src/native.rs  lines 449-453
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None in the default build
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `refund` is `None` and `transfer_near` is also `None`, `callback_args` equals `Default::default()`, so the promise is created **without any callback**:

```rust
// lines 470-483
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

Even when a callback **is** attached (e.g., for the wNEAR-unwrap path where `transfer_near` is `Some`), the callback handler only issues a refund if `args.refund` is `Some`:

```rust
// engine/src/contract_methods/connector.rs  lines 231-242
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← no refund; funds are silently lost
};
``` [3](#0-2) 

The `error_refund` feature is **not** listed in the `default` feature set of either `aurora-engine` or `aurora-engine-precompiles`:

```toml

default = ["std"]

---

### Impact Explanation

**ETH exit failure:** The user's ETH is moved to `exit_to_near::ADDRESS` during EVM execution. If the NEAR-side `ft_transfer` fails, that ETH remains in the precompile address permanently — there is no privileged recovery function and no refund path without the `error_refund` feature. This constitutes **permanent freezing of user funds**.

**ERC-20 exit failure:** The ERC-20 contract burns the user's tokens before calling the precompile. If the NEAR-side `ft_transfer` fails, the burned tokens are gone with no re-mint. This constitutes **direct theft / permanent destruction of user funds**.

Impact: **Critical — permanent freezing / destruction of user funds.**

---

### Likelihood Explanation

The NEAR-side `ft_transfer` can fail for several realistic, non-admin reasons:

1. The Aurora contract's NEP-141 balance is insufficient (e.g., drained by a prior sequence of exits or an accounting discrepancy).
2. The recipient NEAR account is not registered with the NEP-141 contract (storage deposit missing).
3. The NEP-141 contract is paused.

The test `test_exit_to_near_eth_refund` demonstrates scenario 1 explicitly by draining Aurora's balance before calling `exitToNear`. [7](#0-6) 

Any EVM user can trigger this path by calling `exitToNear` with ETH value or by burning ERC-20 tokens via the standard ERC-20 `withdraw` interface. No special privilege is required.

---

### Recommendation

1. **Short term:** Enable the `error_refund` feature in the production WASM build (add it to the `contract` feature set or the default features of `aurora-engine`). This ensures that a refund callback is always attached to the `ft_transfer` promise, so a NEAR-side failure triggers re-minting of burned ERC-20 tokens or return of ETH to the caller.

2. **Long term:** Decouple the refund mechanism from a compile-time feature flag. The refund callback should be unconditionally attached; the `error_refund` flag was introduced as a temporary measure and should be promoted to always-on behavior. Additionally, add integration tests that verify fund conservation across the full exit flow under low-balance and unregistered-recipient conditions.

---

### Proof of Concept

1. Deploy Aurora without the `error_refund` feature (default build).
2. Drain Aurora's NEP-141 balance so that `ft_transfer` will fail (or use an unregistered recipient account).
3. As an unprivileged EVM user, call `exitToNear` sending `N` wei of ETH.
4. The EVM deducts `N` wei from the caller's balance and credits `exit_to_near::ADDRESS`.
5. The NEAR-side `ft_transfer` promise fails.
6. Because `refund: None` is compiled in, no callback fires and no refund is issued.
7. The caller's EVM balance is permanently reduced by `N` wei; the ETH is frozen in `exit_to_near::ADDRESS` with no recovery path.

The same sequence applies to ERC-20 exits: tokens are burned in step 4 and permanently destroyed in step 7. [8](#0-7) [9](#0-8) [3](#0-2)
