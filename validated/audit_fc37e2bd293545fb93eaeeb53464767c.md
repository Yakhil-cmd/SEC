Audit Report

## Title
`read_state` envelope embeds transaction signer's public key instead of read_state signer's public key for secp256k1 ‚Äî (`rs/rosetta-api/icp/src/request_handler/construction_combine.rs`)

## Summary
In `construction_combine`, the `SignatureType::Ecdsa` branch that constructs the `read_state_envelope` sources `sender_pubkey` from `transaction_signature.public_key.hex_bytes` (line 142) while `sender_sig` is correctly sourced from `read_state_signature.hex_bytes` (line 156). The Ed25519 branch for the same envelope correctly uses `read_state_signature.public_key.hex_bytes` (line 121). When a Rosetta client supplies distinct secp256k1 keypairs for the update and read_state payloads, the resulting `read_state` envelope carries a public key that does not match its signature, causing the IC replica to reject every read_state poll with an authentication error and leaving the client unable to confirm transaction finality.

## Finding Description
Two signatures are looked up from the client-supplied map:

- `transaction_signature` ‚Äî keyed by `make_sig_data(&update.id())` (lines 50‚Äì56)
- `read_state_signature` ‚Äî keyed by `make_sig_data(&MessageId::from(read_state.representation_independent_hash()))` (lines 57‚Äì65)

The `read_state_envelope` is built in a `match` on `read_state_signature.signature_type`. In the `Ed25519` arm (lines 116‚Äì136), `sender_pubkey` is correctly derived from `read_state_signature.public_key.hex_bytes` (line 121) and `sender_sig` from `read_state_signature.hex_bytes` (line 134) ‚Äî both consistent.

In the `Ecdsa` arm (lines 137‚Äì158), `sender_pubkey` is derived from `transaction_signature.public_key.hex_bytes` (line 142) ‚Äî the wrong variable ‚Äî while `sender_sig` is correctly derived from `read_state_signature.hex_bytes` (line 156). The IC replica authenticates a read_state request by verifying that `sender_sig` is a valid signature over the request content under `sender_pubkey`; when the two originate from different keypairs, this check fails unconditionally. No existing guard in `construction_combine` detects or prevents this mismatch.

## Impact Explanation
This is a significant Rosetta API security impact with concrete user harm, qualifying as **High** under the allowed ICP bounty impacts. The ICP transfer executes on-chain (the update envelope is correctly formed), but the Rosetta node's polling loop in `/construction/submit` never receives a certified read_state response. From the client's perspective the transaction is permanently unconfirmable. A client following standard retry logic may resubmit the transfer, resulting in duplicate ICP transfers and direct financial loss to the user.

## Likelihood Explanation
The Rosetta Construction API specification explicitly permits distinct signing keys per payload type. Any secp256k1 client that exercises this ‚Äî a valid, spec-compliant pattern ‚Äî triggers the bug deterministically on every call. No special privileges, no victim interaction, and no race condition are required. The Ed25519 path is unaffected; only secp256k1 users are impacted. The bug is reachable by any unprivileged external API caller.

## Recommendation
On line 142, replace `transaction_signature.public_key.hex_bytes` with `read_state_signature.public_key.hex_bytes`, mirroring the correct Ed25519 branch at line 121:

```rust
// lines 141‚Äì143 ‚Äî fix:
Secp256k1KeyPair::der_encode_pk(
    Secp256k1KeyPair::hex_decode_pk(
        &read_state_signature.public_key.hex_bytes,  // was: transaction_signature
    )
```

## Proof of Concept
Construct a `ConstructionCombineRequest` with two distinct secp256k1 keypairs (key A for the update payload, key B for the read_state payload). Call `construction_combine`. Inspect the returned `read_state_envelope`: `sender_pubkey` will DER-decode to key A's public key while `sender_sig` is a signature produced by key B. Submit the envelope pair to a local replica or PocketIC instance; the update call is accepted, but the read_state poll returns an authentication error because the signature (key B) does not verify under the embedded public key (key A). A deterministic integration test can assert `read_state_envelope.sender_pubkey == der_encode(key_B_public)` after calling `construction_combine` with distinct keypairs.
