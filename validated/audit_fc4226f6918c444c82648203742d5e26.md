### Title
Voting Reward Purse Permanently Lost When Settled Proposals Receive Zero Votes — (`rs/sns/governance/src/governance.rs`, `rs/nns/governance/src/governance.rs`)

---

### Summary

Both NNS and SNS governance compute a reward purse each round based on the total token supply and the configured reward rate. The rollover mechanism that carries unused rewards forward to the next round is gated exclusively on whether `settled_proposals` is empty. When one or more proposals are settled in a round but **no neuron cast a vote** on any of them, the entire reward purse for that round is silently discarded — it is neither distributed nor rolled over.

---

### Impact Explanation

Every reward round in which at least one proposal is settled but zero votes are cast causes the entire computed reward purse (`supply × reward_rate × round_duration`, plus any previously rolled-over amount) to be permanently discarded. The maturity that would have been minted for that round is never created. For SNS instances this can represent a meaningful fraction of the annual inflation budget. The loss compounds if multiple consecutive rounds are affected, because each round's purse is computed fresh from the supply without recovering the prior round's lost amount.

---

### Likelihood Explanation

**SNS (higher likelihood):** SNS instances often launch with a small number of neurons and no mandatory default-followee graph. A round in which all neurons abstain — or in which the only settled proposals were created before any neuron existed with sufficient dissolve delay — is realistic, especially in the early life of an SNS. Any governance participant (canister developer, SNS token holder) can trigger this by making a proposal and ensuring no vote is cast before the reward round closes.

**NNS (lower likelihood):** The NNS has a well-established default-followee system that causes most proposals to receive votes automatically. However, the code path is identical and the condition is reachable in principle.

---

### Recommendation

Change the rollover condition to account for whether rewards were actually distributed, not merely whether proposals were settled. Concretely:

```rust
pub(crate) fn rewards_rolled_over(&self) -> bool {
    // Roll over if no proposals were settled OR if proposals were settled
    // but nothing was actually distributed (e.g., zero votes cast).
    self.settled_proposals.is_empty() || self.distributed_e8s_equivalent == 0
}
```

This mirrors the intent of the original StakingRewards mitigation: unused reward budget should always carry forward rather than be silently discarded.

### Proof of Concept

1. Deploy an SNS with `initial_reward_rate_basis_points > 0` and `round_duration_seconds = T`.
2. Create a proposal (requires a neuron with sufficient dissolve delay and stake).
3. Allow the proposal's voting period to expire without any neuron casting a vote.
4. Wait for the reward round to close (`T` seconds after the last `end_timestamp_seconds`).
5. `run_periodic_tasks` calls `distribute_rewards`:
   - `considered_proposals` = `[proposal_id]` (non-empty).
   - `neuron_id_to_reward_shares` = `{}` (no votes).
   - `total_reward_shares == dec!(0)` → warning logged, no maturity incremented.
   - `distributed_e8s_equivalent = 0`.
   - Proposal is marked settled → `settled_proposals = [proposal_id]`.
   - `rewards_rolled_over()` = `false` → `e8s_equivalent_to_be_rolled_over()` = `0`.
6. Next round's purse starts from 0 rollover; the current round's purse is gone.
7. Observe `latest_reward_event.total_available_e8s_equivalent > 0` but `distributed_e8s_equivalent == 0`, and the following round's `total_available_e8s_equivalent` does **not** include the lost amount. [9](#0-8) [10](#0-9) [3](#0-2)
