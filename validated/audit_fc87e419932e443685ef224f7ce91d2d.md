### Title
Permanent ERC-20 / ETH Fund Loss When `ft_transfer` Promise Fails Without `error_refund` Feature — (`File: engine-precompiles/src/native.rs`)

### Summary

The `ExitToNear` precompile burns ERC-20 tokens (or transfers ETH to the precompile address) before scheduling an asynchronous `ft_transfer` promise on NEAR. When the `error_refund` Cargo feature is not compiled in, no refund callback is attached to that promise. If the `ft_transfer` call fails on-chain, the burned tokens are permanently lost with no recovery path for the user.

### Finding Description

The `ExitToNear` precompile's `run()` function constructs `callback_args` with a compile-time conditional:

```rust
// engine-precompiles/src/native.rs  lines 449-453
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← no refund callback when feature is absent
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `refund` is `None` and `transfer_near` is also `None`, `callback_args` equals its default value, so the promise is created as a bare `PromiseArgs::Create` with **no callback**:

```rust
// lines 470-483
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

The refund logic that would re-mint burned ERC-20 tokens or return ETH lives exclusively in `exit_to_near_precompile_callback`, which is only reachable when a `PromiseArgs::Callback` is scheduled:

```rust
// engine/src/contract_methods/connector.rs  lines 231-239
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
}
``` [3](#0-2) 

Without the `error_refund` feature, this callback is never scheduled, so a failed `ft_transfer` leaves the EVM state permanently inconsistent: tokens are burned / ETH is held by the precompile address, but nothing arrives on the NEAR side and nothing is returned to the user.

The engine-level `refund_on_error` function that would re-mint ERC-20 tokens or transfer ETH back is never invoked: [4](#0-3) 

The test suite explicitly documents this loss:

```rust
// engine-tests/src/tests/erc20_connector.rs  lines 656-660
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

And for ETH: [6](#0-5) 

### Impact Explanation

- **ERC-20 path**: tokens are burned in the EVM before the promise is dispatched. If `ft_transfer` fails (e.g., recipient not registered with the NEP-141 contract), the burned tokens are gone — no re-mint occurs. This is a **permanent freeze / destruction of user funds**.
- **ETH path**: ETH is transferred to the `exit_to_near` precompile address before the promise. If `ft_transfer` fails, the ETH is stranded in the precompile address with no recovery path. This is also a **permanent freeze of user funds**.

Impact classification: **Critical — permanent freezing / loss of funds**.

### Likelihood Explanation

The trigger condition — `ft_transfer` failing — is realistic and attacker-controllable:
1. A user (or attacker on behalf of a victim) calls `ExitToNear` targeting a NEAR account that is not registered with the NEP-141 contract.
2. The ERC-20 tokens are burned synchronously in the EVM.
3. The `ft_transfer` promise executes asynchronously and fails because the recipient is unregistered.
4. Without the `error_refund` feature, no callback fires and the tokens are gone.

An attacker can also front-run a legitimate exit by deregistering the recipient from the NEP-141 contract between simulation and settlement, mirroring the SUI attack path exactly.

### Recommendation

The `error_refund` feature should be made a **default** Cargo feature (or unconditionally compiled in) so that the refund callback is always attached to the `ft_transfer` promise. The conditional compilation guard should be removed from the production code path:

```rust
// Remove the cfg guards; always populate refund args
let callback_args = ExitToNearPrecompileCallbackArgs {
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    transfer_near: transfer_near_args,
};
```

This ensures `exit_to_near_precompile_callback` is always scheduled and can re-mint tokens or return ETH whenever the downstream `ft_transfer` fails.

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` Cargo feature.
2. Bridge a NEP-141 token to Aurora as an ERC-20.
3. Call the `ExitToNear` precompile targeting a NEAR account that is **not** registered with the NEP-141 contract.
4. Observe: ERC-20 tokens are burned in the EVM (balance decreases), `ft_transfer` fails on NEAR, no callback fires, no re-mint occurs.
5. The user's ERC-20 balance is permanently reduced with no corresponding NEP-141 credit — confirmed by the existing test `test_exit_to_near_refund` at: [7](#0-6)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )

        let exit_address = exit_to_near::ADDRESS;

            &exit_address,
            &refund_address,
            amount,

            u64::MAX,

                (refund_address.raw(), Vec::new()),
            ],
