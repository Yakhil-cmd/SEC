Audit Report

## Title
SNS Token Burn Inflates Treasury Valuation, Blocking Legitimate `TransferSnsTreasuryFunds` Proposals - (File: rs/sns/governance/token_valuation/src/lib.rs)

## Summary

`IcpsPerSnsTokenClient::fetch_icps_per_sns_token` computes the current SNS token price by dividing the initial swap price by `total_inflation = current_supply / initial_supply`. Because any unprivileged token holder can burn SNS tokens via a standard ICRC-1 transfer to the minting account, an attacker can reduce `current_supply` below `initial_supply`, making `total_inflation < 1` and inflating `icps_per_token` without bound. The inflated price propagates into the treasury's XDR valuation, which governs the `ProposalsAmountTotalUpperBound` regime, potentially blocking legitimate `TransferSnsTreasuryFunds` governance proposals.

## Finding Description

In `rs/sns/governance/token_valuation/src/lib.rs`, `fetch_icps_per_sns_token` computes:

```rust
let total_inflation = current_supply_e8s.checked_div(initial_supply_e8s)...;
initial_icps_per_sns_token.checked_div(total_inflation)
``` [1](#0-0) 

`current_supply_e8s` is fetched live via `icrc1_total_supply`: [2](#0-1) 

`initial_supply_e8s` is determined by scanning ledger mints from block 0 sharing the same timestamp as the first transaction — a fixed historical value: [3](#0-2) 

The resulting `icps_per_token` feeds directly into the treasury XDR valuation: [4](#0-3) 

In `rs/sns/governance/proposals_amount_total_limit/src/lib.rs`, `ProposalsAmountTotalUpperBound::in_tokens` applies `clamp_xdrs_per_icp` (enforcing a `MIN_XDRS_PER_ICP = 1`) but applies **no analogous clamp on `icps_per_token`**. The code comment explicitly acknowledges the asymmetry — no `MAX_XDRS_PER_ICP` is enforced because high valuations push into the more restrictive "large" regime — but the same reasoning was not applied to `icps_per_token`: [5](#0-4) 

The three regimes are:
- **Small** (≤ 100,000 XDR): `NoLimit`
- **Medium** (100,000–1,200,000 XDR): `Fraction(0.25)`
- **Large** (> 1,200,000 XDR): `Xdr(300,000)` — at most 300,000 XDR worth per 7 days [6](#0-5) 

The valuation check is enforced both at proposal submission: [7](#0-6) 

and at execution time using the valuation snapshot stored at submission: [8](#0-7) 

**Exploit flow:**
1. Attacker holds SNS tokens and observes a treasury near a regime boundary.
2. Attacker calls `icrc1_transfer` on the SNS ledger with `to = minting_account` and a chosen `amount` — a standard ICRC-1 burn requiring no special permissions.
3. `current_supply` drops; `total_inflation < 1`; `icps_per_token` inflates.
4. Treasury XDR valuation crosses a regime boundary (e.g., Small → Medium, or Medium → Large).
5. Legitimate `TransferSnsTreasuryFunds` proposals that were previously within the `NoLimit` or `Fraction(0.25)` allowance are now rejected at submission or execution.

No existing guard prevents this: `clamp_xdrs_per_icp` does not touch `icps_per_token`, and there is no floor or ceiling on the computed token price.

## Impact Explanation

An unprivileged SNS token holder can block legitimate `TransferSnsTreasuryFunds` governance proposals by burning their own tokens to inflate the computed treasury valuation. This constitutes a targeted, application-level governance DoS against a specific SNS, with concrete harm to token holders who depend on treasury disbursements. This matches the **High** impact category: *"Significant SNS security impact with concrete user or protocol harm"* and *"Application/platform-level DoS... not based on raw volumetric DDoS."*

## Likelihood Explanation

Any SNS token holder can execute this attack by issuing a standard ICRC-1 burn — no privileged access, admin key, or governance majority is required. The required burn size scales with the distance of the treasury valuation from the nearest regime boundary; for a treasury already near 100,000 XDR, a small burn suffices. The attacker has a direct economic incentive: preventing treasury disbursements that would dilute the value of their remaining holdings. The effect persists until staking rewards or new minting restores the supply ratio, which may take a long time. The attack is repeatable.

## Recommendation

1. **Clamp `icps_per_token`**: Introduce a `MAX_ICPS_PER_TOKEN` analogous to the existing `MIN_XDRS_PER_ICP` floor in `ProposalsAmountTotalUpperBound::in_tokens`, so that supply deflation (whether natural or attacker-induced) cannot produce unbounded price inflation.
2. **Use a smoothed or snapshotted supply**: Replace the instantaneous `icrc1_total_supply` call with a moving average or a supply snapshot at a fixed recent block, making single-block burns insufficient to manipulate the valuation.
3. **Decouple security-critical limits from the inflation approximation**: The code comment at line 356 acknowledges the valuation "isn't an exact science." Security-critical treasury limits should not rely solely on this approximation without additional safeguards such as the clamps described above.

## Proof of Concept

**Setup:**
- `initial_supply = 1,000,000` tokens
- `initial_sns_tokens_per_icp = 100` → `initial_icps_per_sns_token = 0.01`
- `xdrs_per_icp = 10`
- Treasury balance = 100,000 tokens

**Before attack:**
- `current_supply = 1,000,000`; `total_inflation = 1.0`; `icps_per_token = 0.01`
- `treasury_value_xdr = 100,000 × 0.01 × 10 = 10,000 XDR` → **Small** → `NoLimit`

**Attacker burns 900,000 tokens** (`icrc1_transfer` with `to = minting_account`, `amount = 900_000 * E8`):
- `current_supply = 100,000`; `total_inflation = 0.1`; `icps_per_token = 0.1`
- `treasury_value_xdr = 100,000 × 0.1 × 10 = 100,000 XDR` → **Medium** → `Fraction(0.25)` → only 25,000 tokens transferable per 7 days

**Attacker burns 990,000 tokens:**
- `current_supply = 10,000`; `total_inflation = 0.01`; `icps_per_token = 1.0`
- `treasury_value_xdr = 100,000 × 1.0 × 10 = 1,000,000 XDR` → **Large** → `Xdr(300,000)` → at most `300,000 / (1.0 × 10) = 30,000` tokens per 7 days

A deterministic integration test or PocketIC test can reproduce this by: (1) initializing an SNS with the above parameters, (2) submitting a `TransferSnsTreasuryFunds` proposal for the full treasury balance (succeeds before burn), (3) executing the burn via `icrc1_transfer` to the minting account, and (4) verifying that an identical proposal is now rejected at submission.

pub(crate) fn transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err<'a>(
    transfer: &TransferSnsTreasuryFunds,
    valuation: Valuation,
    proposals: impl Iterator<Item = &'a ProposalData>,
    now_timestamp_seconds: u64,
) -> Result<(), GovernanceError> {
    let allowance_tokens = transfer_sns_treasury_funds_7_day_total_upper_bound_tokens(valuation)
        .map_err(|err| {
            // This should not be possible, because valuation was already used the same way during
            // proposal submission/creation/validation.
            GovernanceError::new_with_message(
                ErrorType::InconsistentInternalData,
                format!(
                    "Unable to determined upper bound on the amount of \
                     TransferSnsTreasuryFunds proposals: {err:?}\nvaluation:{valuation:?}",
                ),
            )
        })?;

    // The total calculated here _could_ be different from what was calculated at proposal
    // submission/creation time. A difference would result from the execution of (another)
    // TransferSnsTreasuryFunds proposal between now and then.
    let spent_tokens = total_treasury_transfer_amount_tokens(
        proposals,
        transfer.from_treasury(),
        now_timestamp_seconds - 7 * ONE_DAY_SECONDS,
    )
    .map_err(|message| {
        GovernanceError::new_with_message(ErrorType::InconsistentInternalData, message)
    })?;

    let remainder_tokens = allowance_tokens - spent_tokens;
    let transfer_amount_tokens = denominations_to_tokens(transfer.amount_e8s, E8)
        // This Err cannot be provoked, because we are dividing a u64 (amount_e8s) by a positive
        // integer (E8).
        .ok_or_else(|| {
            GovernanceError::new_with_message(
                ErrorType::UnreachableCode,
                format!(
                    "Unable to convert proposals amount {} e8s to tokens.",
                    transfer.amount_e8s,
                ),
            )
        })?;
    if transfer_amount_tokens > remainder_tokens {
        return Err(GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!(
                "Executing this proposal is not allowed at this time, because doing \
                 so would cause the 7 day upper bound of {allowance_tokens} tokens to be exceeded. \
                 Maybe, try again later? The total amount transferred in the past \
                 7 days stands at {spent_tokens} tokens, and the amount in this proposal is {transfer_amount_tokens} \
                 tokens. The upper bound is based on treasury valuation factors at \
                 the time of proposal submission: {valuation:?}",
            ),
        ));
    }

    Ok(())
