Audit Report

## Title
Unbounded DKG Unvalidated Pool Growth via Silent Deferral of Remote-Tagged Dealings — (`rs/consensus/dkg/src/lib.rs`)

## Summary
A Byzantine subnet node (below the fault threshold) can flood the DKG unvalidated pool by sending crafted messages with `NiDkgTargetSubnet::Remote(unknown_id)` at the current `start_block_height`. The `DkgBouncer` admits these messages based solely on height, `validate_dealings_for_dealer` silently defers them without rejection or removal, and neither the pool nor the P2P slot table enforces a per-peer size limit for DKG artifacts. The pool accumulates unboundedly until the next DKG interval purge (~499 rounds), enabling memory and CPU exhaustion that can stall DKG transcript production and subnet availability.

## Finding Description

**Silent deferral without rejection — `validate_dealings_for_dealer`**

At `rs/consensus/dkg/src/lib.rs` L207–211, when `configs.get(message_dkg_id)` returns `None` and `message_dkg_id.target_subnet.is_remote()` is true, the function returns `Mutations::new()` — an empty change set. The message is neither moved to validated nor removed from the unvalidated pool. It remains indefinitely until the next purge. [1](#0-0) 

**Admission without content filtering — `DkgBouncer`**

The bouncer at L396–402 only compares `id.height` against `start_height`. A message at the current start height always receives `BouncerValue::Wants`, regardless of `target_subnet`, `dealer_subnet`, or signature validity. [2](#0-1) 

**No capacity limit on the unvalidated pool — `PoolSection`**

`PoolSection` is a plain `BTreeMap<K, V>` with no maximum size. `DkgPoolImpl::insert` calls it unconditionally. [3](#0-2) [4](#0-3) 

**No per-peer slot limit for DKG — P2P setup**

DKG uses `SLOT_TABLE_NO_LIMIT = usize::MAX` at L279, while only ingress uses `SLOT_TABLE_LIMIT_INGRESS = 50_000`. The comment at L72–73 explicitly acknowledges the ingress limit exists to protect against malicious peers, but no equivalent protection is applied to DKG. [5](#0-4) [6](#0-5) 

**`DkgMessageId` keyed by content hash — no deduplication across distinct messages**

Each message with a distinct `NiDkgDealing` payload produces a distinct `DkgMessageId`. An attacker varying dealing bytes or `unknown_target_id` values generates N distinct pool entries. [7](#0-6) 

**O(N) processing on every `on_state_change` call**

Every call to `on_state_change` iterates the entire unvalidated pool, groups by `(signer, dkg_id)`, and calls `validate_dealings_for_dealer` for each group. With N deferred remote messages using distinct `dkg_id` values, this produces N groups and N calls — O(N) work per consensus round. [8](#0-7) 

**Purge only at interval boundary**

The purge fires only when `start_height > dkg_pool.get_current_start_height()`, i.e., at the next DKG summary block. With `DKG_INTERVAL_HEIGHT = 499`, deferred messages accumulate for the entire interval. [9](#0-8) [10](#0-9) 

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

Concrete effects:
1. **Memory exhaustion**: NiDkgDealing objects are large cryptographic structures. N messages consume O(N × dealing_size) bytes with no eviction until the next summary block.
2. **CPU exhaustion**: `on_state_change` is called every consensus round and iterates all N deferred messages, causing O(N) CPU work per round, starving legitimate local-DKG validation.
3. **DKG stall**: If the replica crashes or falls behind due to resource exhaustion, DKG transcripts are not produced, blocking subnet key generation and subnet creation.

## Likelihood Explanation

- **Attacker model**: A single Byzantine subnet node, explicitly within the IC fault-tolerance model (f < n/3). No governance majority, no key compromise, no external infrastructure required.
- **Exploit complexity**: Low. The attacker only needs to craft messages with a remote target ID not present in replicated state. No valid signature or valid dealing content is required at the admission stage.
- **Persistence**: The attack persists for the full DKG interval (up to 499 rounds) before purge.
- **Amplification**: Multiple Byzantine nodes (still below fault threshold) multiply the effect linearly.

## Recommendation

1. **Reject remote-tagged messages with unknown target IDs immediately** rather than deferring them. If no `SetupInitialDKG` context exists for the target ID at validation time, emit `HandleInvalid` instead of returning `Mutations::new()`.
2. **Add a per-peer size limit to the DKG unvalidated pool**, analogous to `SLOT_TABLE_LIMIT_INGRESS` for ingress, in `setup_ic_network`.
3. **Apply a slot-table limit for DKG** at L279 in `setup_ic_network/src/lib.rs` instead of `SLOT_TABLE_NO_LIMIT`.
4. **Verify the signature at admission** (in the bouncer or at insert time) to prevent unsigned garbage from entering the pool.

pub(crate) struct PoolSection<K, V> {
    messages: BTreeMap<K, V>,
    metrics: PoolMetrics,
}

impl<K: Ord, V: HasLabel> PoolSection<K, V> {
    pub(crate) fn new(metrics_registry: MetricsRegistry, pool: &str, pool_type: &str) -> Self {
        Self {
            messages: Default::default(),
            metrics: PoolMetrics::new(metrics_registry, pool, pool_type),
        }
    }

    pub(crate) fn insert(&mut self, key: K, value: V) -> Option<V> {
        self.metrics
            .observe_insert(MESSAGE_SIZE_BYTES, value.label());
        let replaced = self.messages.insert(key, value);
        if let Some(replaced) = &replaced {
            self.metrics
                .observe_duplicate(MESSAGE_SIZE_BYTES, replaced.label());
        }
        replaced
    }
