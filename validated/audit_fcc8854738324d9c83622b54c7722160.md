### Title
Missing Zero Address Validation for L1 Transaction `refund_recipient` Causes Permanent Loss of Refund Funds - (File: `basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`)

### Summary

The `validate_structure()` function in `AbiEncodedTransaction` explicitly skips address validation for `reserved[1]` (the `refund_recipient` field) in L1→L2 and upgrade transactions, marked with a `// TODO: validate address?` comment. When `refund_recipient` is zero, the bootloader mints the refund amount to `address(0)`, permanently destroying those tokens.

### Finding Description

In `basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`, the `validate_structure()` function validates most fields of ABI-encoded L1 transactions but explicitly leaves the `reserved[1]` (refund recipient) address unvalidated:

```rust
// reserved[1] = refund recipient for l1 to l2 and upgrade txs
match tx_type {
    Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
        // TODO: validate address?   // <-- validation intentionally skipped
    }
    _ => unreachable!(),
}
``` [1](#0-0) 

This `reserved[1]` value is later consumed in `process_l1_transaction.rs` as the refund recipient address, and tokens are unconditionally minted to it:

```rust
let refund_recipient = u256_to_b160_checked(transaction.reserved[1].read());
mint_base_token::<S, Config>(
    system, system_functions, memories.reborrow(),
    &to_refund_recipient,
    &refund_recipient,   // <-- could be address(0)
    ...
)
``` [2](#0-1) 

There is no guard preventing `refund_recipient == B160::ZERO` before the mint call. The `L1TxBuilder` in the test framework also defaults `refund_recipient` to `Default::default()` (address zero), confirming this path is reachable: [3](#0-2) 

The test suite even explicitly uses `address!("0000000000000000000000000000000000000000")` as a refund recipient and tracks the balance at address(0) growing, confirming the protocol silently accepts and processes zero-address refund recipients: [4](#0-3) 

### Impact Explanation

When `refund_recipient` is `address(0)`, the unused-gas refund amount (`(gas_limit - gas_used) * gas_price` in base tokens) is minted to `address(0)` and is permanently inaccessible. This is a direct, irreversible loss of user funds. The amount can be substantial — up to `gas_limit * gas_price` base tokens per transaction.

### Likelihood Explanation

The `L1TxBuilder` defaults `refund_recipient` to `address(0)` when not explicitly set. Any L1 bridge integration, tooling, or user that omits the refund recipient field (relying on the default) will silently burn their refund. The `// TODO: validate address?` comment confirms the developers identified this gap but left it unresolved. The path is exercised in existing tests without triggering any error.

### Recommendation

In `validate_structure()`, add an explicit zero-address check for `reserved[1]` when `tx_type` is `L1_L2_TX_TYPE` or `UPGRADE_TX_TYPE`:

```rust
Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
    let refund_recipient = u256_to_b160_checked(self.reserved[1].read());
    if refund_recipient == B160::ZERO {
        return Err(());
    }
}
```

Alternatively, if zero-address refund recipients are intentionally allowed (to burn the refund), this should be explicitly documented and the `// TODO` comment resolved. The `L1TxBuilder` default should also be changed to a non-zero sentinel or require explicit opt-in.

### Proof of Concept

1. Submit an L1→L2 transaction with `refund_recipient = address(0)`, `gas_limit = 100_000`, `gas_price = 1000`.
2. The transaction executes and uses, say, 21,000 gas.
3. `to_refund_recipient = (100_000 - 21_000) * 1000 = 79,000,000` base tokens.
4. `mint_base_token` is called with `recipient = B160::ZERO`.
5. 79,000,000 base tokens are credited to `address(0)` — permanently inaccessible.
6. The existing test `test_treasury_based_token_distribution_regression` in `tests/instances/transactions/src/lib.rs` already demonstrates this exact scenario, using `refund_recipient = address!("0000000000000000000000000000000000000000")` and tracking the balance increase at address(0). [5](#0-4)
