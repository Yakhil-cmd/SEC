Audit Report

## Title
Off-by-One Index Out of Bounds in `get_key_family` When Search Key Exceeds All Registry Records - (File: rs/registry/client/src/client.rs)

## Summary
`RegistryClientImpl::get_key_family` performs a binary search over the sorted `records` vector and, in the `Err(idx)` branch, unconditionally indexes `records[idx]` without verifying `idx < records.len()`. When the search key is lexicographically greater than every stored record, `binary_search_by_key` returns `Err(records.len())`, and the subsequent index access panics. The identical defect exists in `rs/registry/fake/src/lib.rs`.

## Finding Description
In `rs/registry/client/src/client.rs` at line 290–296, the `Err(idx)` arm of the binary search directly accesses `cache_state.records[idx]`:

```rust
Err(idx) => {
    if !cache_state.records[idx].key.starts_with(key_prefix) {
        return Ok(vec![]);
    }
    idx
}
``` [1](#0-0) 

`binary_search_by_key` documents that `Err(p)` returns an insertion point `p` in the range `[0, len]` inclusive. When `key_prefix` is lexicographically greater than every key in `records`, `p == records.len()`, and `records[records.len()]` is an out-of-bounds access that panics at runtime in Rust's debug and release builds alike (Rust's slice indexing always bounds-checks).

The inline comment ("The entry at idx cannot be lexicographically less than key_prefix") correctly describes an ordering property but incorrectly implies a bounds property; `Err(len)` satisfies the ordering invariant while still being out of bounds.

The same pattern appears verbatim in `rs/registry/fake/src/lib.rs` at line 143–150:

```rust
Err(idx) => {
    if !records[idx].key.starts_with(key_prefix) {
        return Ok(vec![]);
    }
    idx
}
``` [2](#0-1) 

By contrast, `get_versioned_value` in the same file correctly guards its analogous `Err(idx)` arm with `idx > 0` before accessing `records[idx - 1]`: [3](#0-2) 

No existing guard prevents the out-of-bounds access in `get_key_family`.

## Impact Explanation
`get_key_family` is on the hot path for consensus and execution: it is called by `get_chain_key_enabled_subnets` (chain-key routing), node-record enumeration, API boundary-node discovery, and other registry helpers. A panic in this function unwinds the calling replica thread. Because replica code does not wrap registry calls in `std::panic::catch_unwind`, the thread terminates and the replica loses the ability to participate in consensus. If the triggering registry state is present on all nodes of a subnet simultaneously (which it would be, since all nodes share the same registry snapshot), every replica on the subnet panics on the same call, causing a full subnet stall. This matches the allowed High impact: **application/platform-level DoS, consensus blocking, or subnet availability impact** ($2,000–$10,000).

## Likelihood Explanation
The trigger requires a registry state where all stored keys are lexicographically less than the queried prefix. Two realistic operational paths produce this:

1. **Freshly bootstrapped subnet**: early registry state may contain only keys with prefixes earlier in the alphabet (e.g., `api_boundary_node_record_*`). Any `get_key_family` call with a later prefix (e.g., `subnet_record_*`) hits the bug before those keys are populated.
2. **New key-family prefix introduced in a replica upgrade**: the first call with the new prefix against an older registry state that has no such keys — and whose highest key is lexicographically less than the new prefix — triggers the panic.

Both scenarios arise from normal operational sequencing without any attacker action. Because all replicas on a subnet share the same registry state, the panic is simultaneous across all nodes, making recovery impossible without a code fix or registry intervention.

## Recommendation
Add an explicit upper-bounds check before indexing in both files:

```rust
Err(idx) => {
    if idx >= cache_state.records.len()
        || !cache_state.records[idx].key.starts_with(key_prefix)
    {
        return Ok(vec![]);
    }
    idx
}
```

Apply the same fix to `rs/registry/fake/src/lib.rs`. This mirrors the defensive pattern already used in `get_versioned_value`.

## Proof of Concept
```
// Minimal unit test (no network, no replica needed)
let records = vec![
    RegistryRecord { key: "aaa".to_string(), version: v(1), value: Some(...) },
    RegistryRecord { key: "bbb".to_string(), version: v(1), value: Some(...) },
];
// All keys < "zzz_prefix"
// binary_search_by_key(("zzz_prefix", v(1)), ...) => Err(2)
// records[2] => ** index out of bounds: len is 2 but index is 2 ** PANIC
get_key_family("zzz_prefix", v(1));
```

A deterministic unit test constructing a `CacheState` with two records whose keys are both lexicographically less than the queried prefix and calling `get_key_family` will reproduce the panic unconditionally on the current code.
