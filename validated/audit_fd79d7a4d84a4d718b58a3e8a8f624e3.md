Audit Report

## Title
Caller-Controlled `buyer` Parameter in SNS Swap `refresh_buyer_tokens` Bypasses Confirmation-Text Consent Requirement - (File: `rs/sns/swap/canister/canister.rs`)

## Summary
The `refresh_buyer_tokens` update handler in the SNS Swap canister resolves the buyer identity from a caller-supplied string without verifying it matches `msg_caller`. Because the confirmation text is also caller-supplied and publicly readable from `get_init`, any unprivileged principal can register another user's ICP participation—including supplying the required consent text—without that user ever calling the endpoint. If the swap commits, the victim's ICP is irreversibly converted to SNS tokens they never agreed to receive.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 130–134, the buyer principal is resolved as:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no caller == buyer check
};
``` [1](#0-0) 

The resolved `p` is passed directly to `refresh_buyer_token_e8s`, which calls `validate_confirmation_text` — a function that only checks whether the supplied text matches the SNS-configured text, with no check that the caller is the buyer:

```rust
self.validate_confirmation_text(confirmation_text)?;
``` [2](#0-1) 

The `validate_confirmation_text` implementation confirms this — it only compares strings: [3](#0-2) 

The confirmation text is stored in `Init.confirmation_text` and is publicly readable via the `get_init` query endpoint: [4](#0-3) 

The existing system test in `rs/tests/nns/sns/lib/src/sns_deployment.rs` explicitly demonstrates the exploit path: after the wealthy user transfers ICP, the **default user** (a different principal) calls `refresh_buyer_tokens` with `buyer = wealthy_user_principal` and the call **succeeds** (`.unwrap()` is called, not `assert!(res.is_err())`): [5](#0-4) 

No authorization check anywhere in the call path enforces `caller == buyer`. The `authorize_caller_to_call_notify_create_canister_on_behalf_of_creator` pattern used elsewhere in the IC codebase (e.g., `rs/nns/cmc/src/main.rs`) is absent here.

## Impact Explanation
This is a **High** severity SNS security impact. The confirmation text is the sole mechanism by which an SNS can require buyers to affirmatively consent to swap terms (risk disclosures, legal notices). A third party can bypass this gate entirely by supplying `buyer = <victim>` and the publicly known confirmation text. If the swap commits, the victim's ICP is irreversibly converted to SNS tokens without their consent — a concrete unauthorized financial state change. This matches the allowed impact: *"Significant SNS security impact with concrete user or protocol harm"* and *"Unauthorized access to ledger or canister-controlled funds."*

## Likelihood Explanation
**Medium-High.** The attacker requires only: (1) an unprivileged ingress identity, (2) observation that a victim has sent ICP to the swap subaccount (visible on the public ICP ledger), and (3) the confirmation text (readable via `get_init` query). No keys, governance majority, or threshold assumptions are needed. The attack is repeatable against any open SNS swap that uses a confirmation text, and the system test already demonstrates the exact call succeeds.

## Recommendation
Add a caller-equals-buyer check in the canister handler before passing the resolved principal to `refresh_buyer_token_e8s`:

```rust
#[update]
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let requested = PrincipalId::from_str(&arg.buyer).unwrap();
        if requested != caller {
            panic!("Caller {} is not authorized to refresh tokens for buyer {}", caller, requested);
        }
        requested
    };
    // ...
}
```

This mirrors the `authorize_caller_to_call_notify_create_canister_on_behalf_of_creator` pattern already used in the IC codebase. [6](#0-5) 

## Proof of Concept
The system test at `rs/tests/nns/sns/lib/src/sns_deployment.rs` (`add_one_participant`) already constitutes a working proof of concept:

1. Wealthy user transfers ICP to the swap subaccount.
2. Default user (different principal) calls `refresh_buyer_tokens(Some(wealthy_user_principal_id), None)`.
3. The call succeeds and records the wealthy user's participation.
4. `assert_eq!(res_4, res_5)` confirms both calls produce identical results. [7](#0-6) [8](#0-7) 

A minimal unit test can be constructed using the existing `SwapBuilder` and `mock_stub` infrastructure in `rs/sns/swap/tests/swap.rs`, calling `refresh_buyer_token_e8s` with `buyer = user_A` from a context where the caller is `user_B`, with a matching confirmation text, and asserting `Ok` is returned.
