### Title
`payable.transfer()` in `WSEI.withdraw()` Permanently Blocks Smart Contract Withdrawals — (`contracts/src/WSEI.sol`)

### Summary
The Sei chain's deployed Wrapped SEI (`WSEI`) contract uses Solidity's `payable(msg.sender).transfer(wad)` in its `withdraw()` function. Since EIP-1884 (Istanbul), `.transfer()` is hardcoded to forward only 2300 gas. Any smart contract whose `receive()` or `fallback()` function consumes more than 2300 gas will have every `withdraw()` call revert, permanently blocking that contract from unwrapping its WSEI back to native SEI.

### Finding Description
In `contracts/src/WSEI.sol`, the `withdraw()` function decrements the caller's balance and then attempts to push native SEI back to `msg.sender` using `.transfer()`:

```solidity
// contracts/src/WSEI.sol line 27-31
function withdraw(uint wad) public {
    require(balanceOf[msg.sender] >= wad);
    balanceOf[msg.sender] -= wad;
    payable(msg.sender).transfer(wad);   // ← 2300 gas cap
    emit Withdrawal(msg.sender, wad);
}
``` [1](#0-0) 

The `.transfer()` call will revert (rolling back the balance decrement) whenever `msg.sender` is a smart contract whose receive/fallback logic exceeds 2300 gas. This includes:
- Contracts that emit events in their `receive()` hook
- Contracts that write to storage in their `receive()` hook
- Any contract called through a proxy that adds overhead above 2300 gas

The contract is confirmed deployed: compiled artifacts exist at `x/evm/artifacts/wsei/WSEI.abi` and `x/evm/artifacts/wsei/WSEI.bin`. [2](#0-1) 

### Impact Explanation
A smart contract that has deposited native SEI into WSEI (via `deposit()` or the `fallback`) and later calls `withdraw()` will have the call permanently revert. The WSEI balance remains credited to the contract, but the contract cannot convert it back to native SEI through the normal path. If the contract has no alternative mechanism to move the WSEI tokens to an EOA, the native SEI equivalent is effectively frozen inside the WSEI contract. This maps to **Medium** under the Sei bounty scope (fund freeze). [3](#0-2) 

### Likelihood Explanation
DeFi integrations on Sei EVM routinely wrap/unwrap native tokens from within smart contracts (e.g., routers, vaults, yield aggregators). Any such contract whose `receive()` function does more than a bare ETH acceptance (e.g., emits an event, updates a state variable) will trigger this failure. The pattern is common and the failure is silent from the depositor's perspective until withdrawal is attempted. [4](#0-3) 

### Recommendation
Replace `.transfer()` with a low-level `.call{value: wad}("")` and check the return value, following the checks-effects-interactions pattern (balance is already decremented before the call, so re-entrancy is mitigated):

```solidity
function withdraw(uint wad) public {
    require(balanceOf[msg.sender] >= wad);
    balanceOf[msg.sender] -= wad;
    (bool success, ) = payable(msg.sender).call{value: wad}("");
    require(success, "WSEI: ETH transfer failed");
    emit Withdrawal(msg.sender, wad);
}
```

### Proof of Concept
1. Deploy a vault contract with a `receive()` that emits an event (costs ~750 gas, total > 2300 with proxy overhead).
2. Call `WSEI.deposit{value: 1 ether}()` from the vault.
3. Call `WSEI.withdraw(1 ether)` from the vault.
4. Transaction reverts at the `.transfer()` call; the vault's WSEI balance remains but native SEI cannot be recovered via `withdraw()`. [5](#0-4)
