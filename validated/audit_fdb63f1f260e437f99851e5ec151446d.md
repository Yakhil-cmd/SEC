Audit Report

## Title
Unbounded Ingress Window Causes Uncapped Memory Allocation in `construction_payloads` - (File: `rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary
The `construction_payloads` function in the ICRC1 Rosetta API accepts attacker-controlled `ingress_start` and `ingress_end` values with no upper bound on the window size. The while loop that populates `ingress_expiries` runs one iteration per ~150-second step across the entire window, and `handle_construction_payloads` allocates one `EnvelopeContent::Call` with a full clone of `canister_method_args` per entry. An unprivileged HTTP client can trigger unbounded heap allocation by supplying a multi-year window, exhausting server memory and crashing the Rosetta sidecar process.

## Finding Description
In `construction_payloads` (`services.rs` lines 148–167), the only validations on the ingress window are:

- `ingress_start >= ingress_end` → reject [1](#0-0) 
- `ingress_end < now + ingress_interval` → reject [2](#0-1) 

There is no maximum window size check. The while loop then runs unboundedly: [3](#0-2) 

The step size is `ingress_interval − INGRESS_INTERVAL_OVERLAP`. With `MAX_INGRESS_TTL = 5 min`, `PERMITTED_DRIFT = 30 s`, and `INGRESS_INTERVAL_OVERLAP = 120 s`: step = **150 seconds**. A 10-year window yields ~2,102,400 iterations.

`handle_construction_payloads` then allocates one `EnvelopeContent::Call` per entry, cloning `canister_method_args` each time: [4](#0-3) 

The `UnsignedTransaction` is then CBOR-serialized and hex-encoded (doubling byte size) before being returned in the HTTP response: [5](#0-4) 

The endpoint handler passes `ingress_start`/`ingress_end` directly from the JSON request body with no additional guards: [6](#0-5) 

The `ConstructionPayloadsRequestMetadata` struct accepts these as plain `Option<u64>` with no domain-level constraints at deserialization time: [7](#0-6) 

The documented design intent (comment at `types.rs` line 44) explicitly states the intended window is 24 hours, yielding ~576 entries — confirming the absence of a cap is unintentional: [8](#0-7) 

## Impact Explanation
An unprivileged HTTP client can crash the ICRC1 Rosetta sidecar process via OOM by sending a single small JSON request. The Rosetta API is an explicitly in-scope financial integration component. This matches the **High** bounty impact: *"Application/platform-level DoS, crash... or subnet availability impact not based on raw volumetric DDoS"* and *"Significant... Rosetta... security impact with concrete user or protocol harm."* The crash disrupts all ICRC1 Rosetta operations (transaction construction, submission, and status queries) for any user of the affected sidecar instance.

## Likelihood Explanation
The endpoint requires no authentication. The request payload is trivially small (a single JSON object). The attack is repeatable — the process can be crashed again immediately after restart. No special privileges, credentials, or network position are required. Any external HTTP client that can reach the Rosetta port can trigger this.

## Recommendation
Add an explicit upper bound on the ingress window before the while loop, matching the documented 24-hour design intent:

```rust
const MAX_INGRESS_WINDOW: u64 = 24 * 3600 * 1_000_000_000; // 24 hours in nanoseconds
if ingress_end - ingress_start > MAX_INGRESS_WINDOW {
    return Err(Error::processing_construction_failed(
        "Ingress window exceeds maximum allowed 24-hour range"
    ));
}
```

This caps the vector at ~576 entries and aligns with the stated design intent in `types.rs` line 44. [9](#0-8) 

## Proof of Concept
```bash
# Set ingress_start to current time in nanoseconds, ingress_end to now + 10 years
NOW_NS=$(date +%s)000000000
END_NS=$((NOW_NS + 315576000000000000))  # +10 years in ns

curl -X POST http://<rosetta-host>:8081/construction/payloads \
  -H 'Content-Type: application/json' \
  -d "{
    \"network_identifier\": {\"blockchain\":\"Internet Computer\",\"network\":\"<ledger_id>\"},
    \"operations\": [/* valid ICRC1 transfer ops */],
    \"public_keys\": [{\"hex_bytes\":\"<pubkey>\",\"curve_type\":\"edwards25519\"}],
    \"metadata\": {
      \"ingress_start\": $NOW_NS,
      \"ingress_end\":   $END_NS
    }
  }"
```

With a 10-year window (~3.15×10¹⁷ ns), the loop runs ~2,102,400 iterations, each cloning the transfer args and building an `EnvelopeContent`. The CBOR+hex response will be several gigabytes, exhausting server memory before the response is written. A local integration test can reproduce this deterministically by calling `services::construction_payloads` directly with `ingress_end = ingress_start + 10_years_ns` and asserting that memory usage grows proportionally.
