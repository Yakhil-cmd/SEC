### Title
Unchecked Arithmetic in `NEP141Wei` Balance Type Enables Silent Overflow/Underflow in ETH Connector Accounting - (File: `engine-types/src/types/wei.rs`)

### Summary
The `NEP141Wei` type, which represents ETH connector balances denominated in Wei, implements `Add` and `Sub` using plain Rust arithmetic operators (`+`, `-`) without overflow/underflow protection. In Rust release builds (the WASM production target), integer overflow wraps silently. This is the direct analog of the reported Solidity safe-math omission: unchecked arithmetic on financial balance types in production code.

### Finding Description
In `engine-types/src/types/wei.rs`, the `NEP141Wei` type implements `Sub`, `Add`, and `SubAssign` using bare arithmetic on its inner `u128` field:

```rust
impl Sub for NEP141Wei {
    fn sub(self, rhs: Self) -> Self::Output {
        Self(self.0 - rhs.0)   // ← unchecked subtraction, wraps on underflow
    }
}
impl Add for NEP141Wei {
    fn add(self, rhs: Self) -> Self::Output {
        Self(self.0 + rhs.0)   // ← unchecked addition, wraps on overflow
    }
}
impl SubAssign for NEP141Wei {
    fn sub_assign(&mut self, rhs: Self) {
        *self = *self - rhs;   // ← delegates to unchecked Sub
    }
}
``` [1](#0-0) 

This type is used in `engine-precompiles/src/native.rs` (the `ExitToNear`/`ExitToEthereum` precompiles) and `engine/src/lib.rs` for ETH connector balance accounting. The `Yocto` type in `engine-types/src/types/balance.rs` has the same defect in its `Sub` implementation:

```rust
impl Sub for Yocto {
    fn sub(self, rhs: Self) -> Self::Output {
        Self(self.0 - rhs.0)   // ← unchecked subtraction
    }
}
``` [2](#0-1) 

By contrast, the `Wei` type (used for EVM-side balances) correctly provides `checked_sub` and `checked_add`, and `add_balance` in `engine/src/engine.rs` uses `checked_add` with an error return: [3](#0-2) [4](#0-3) 

The asymmetry is the root cause: the NEP-141 connector-side balance type lacks the same protection the EVM-side type has.

### Impact Explanation
In Rust release/WASM builds, integer overflow on primitive types wraps silently (no panic, no error). If `NEP141Wei::sub()` is called with `rhs > self`, the result wraps to `u128::MAX - (rhs - self) + 1`, a near-maximum value. If this wrapped value is stored as a connector balance or total supply, the connector's accounting becomes permanently incorrect. Depending on the call site, this can result in:

- **Insolvency**: The connector believes it holds far more ETH than it actually does, allowing subsequent withdrawals that drain real funds.
- **Permanent fund freeze**: If the wrapped value is used as a guard condition, legitimate withdrawals may be blocked.

The CHANGES.md confirms this class of bug is real and exploitable in this codebase: [5](#0-4) 

Those fixes addressed the XCC module; the `NEP141Wei` and `Yocto` types remain unpatched.

### Likelihood Explanation
The `ExitToNear` and `ExitToEthereum` precompiles are reachable by any EVM user by calling the precompile address from a deployed contract or a direct transaction. No admin privilege is required. The attacker only needs to craft a transaction that causes a subtraction where `rhs > self` on a `NEP141Wei` value — a condition that may arise naturally from a race condition, a crafted sequence of deposits/withdrawals, or a direct call with a larger-than-balance exit amount if the call site does not independently guard the subtraction.

### Recommendation
Replace the unchecked `Sub`/`Add`/`SubAssign` implementations on `NEP141Wei` and `Yocto` with checked variants that return `Option` or `Result`, mirroring the pattern already used by `Wei::checked_sub` and `Wei::checked_add`. All call sites must propagate the error rather than silently accepting a wrapped value. Alternatively, use `u128::checked_sub`/`u128::checked_add` inline at every call site and treat `None` as a hard error.

### Proof of Concept
1. Deploy a contract on Aurora that calls the `ExitToNear` precompile requesting an exit of amount `A`.
2. Arrange (e.g., via a prior sequence of operations) that the `NEP141Wei` connector balance for the relevant account is `B < A`.
3. If the call site invokes `balance - exit_amount` using `NEP141Wei::sub()` without a prior guard, the result wraps to `u128::MAX - (A - B) + 1`.
4. The connector stores this wrapped value as the new balance, making the accounting permanently incorrect and enabling subsequent over-withdrawal.

The exact exploitability depends on whether each call site in `native.rs` independently guards the subtraction — which cannot be confirmed without reading that file — but the type itself provides no safety net, and the precedent of identical bugs being fixed in the XCC module (PR #888, #889) confirms the pattern is exploitable.
