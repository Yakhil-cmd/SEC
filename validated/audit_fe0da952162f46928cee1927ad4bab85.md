Audit Report

## Title
Unbounded Synchronous BFS in SNS `cast_vote_and_cascade_follow` Enables Instruction-Exhaustion DoS on Victim Neuron Voting - (File: rs/sns/governance/src/governance.rs)

## Summary

The SNS governance canister's `cast_vote_and_cascade_follow` function is a plain synchronous function that runs an unbounded BFS loop over the follower graph with no instruction-limit checkpoints, no async self-call mechanism, and no timer-based continuation. Because the `follow` function enforces only an outbound followee cap (max 15) but imposes no cap on how many neurons may follow a given neuron, an unprivileged attacker can pre-populate arbitrarily many follower neurons targeting a victim. When the victim calls `register_vote`, the synchronous BFS exhausts the IC per-message instruction limit, the call traps, and the victim's vote is permanently lost for that proposal. The attack repeats automatically for every subsequent proposal where the attacker's neurons hold ballots.

## Finding Description

`register_vote` calls `cast_vote_and_cascade_follow` as a plain synchronous function call with no `await` point: [1](#0-0) 

`cast_vote_and_cascade_follow` is declared as a plain `fn` (not `async fn`) and its BFS loop contains no instruction-limit checks, no `is_over_instructions_limit()` guards, and no self-call or timer continuation: [2](#0-1) [3](#0-2) 

The `follow` function enforces only the outbound followee count (`max_followees_per_function`, ceiling = 15), but performs no check on how many neurons may follow a given target neuron. The `function_followee_index` is updated with an unbounded `insert` per follower: [4](#0-3) [5](#0-4) 

The ceiling on neurons per SNS is 200,000: [6](#0-5) 

By contrast, NNS governance uses a `ProposalVotingStateMachine` with explicit `is_over_instructions_limit()` checks after each follower processed and an async `noop_self_call_if_over_instructions` escape hatch to continue processing across multiple messages: [7](#0-6) [8](#0-7) 

SNS governance has none of these safeguards.

**Exploit path:**
1. Attacker creates `N` neurons in the target SNS (each with minimum stake) before a proposal is submitted.
2. Each attacker neuron calls `Follow { function_id: X, followees: [victim_neuron_id] }` — permitted with no inbound follower-count check.
3. A proposal of function type `X` is submitted. All `N` attacker neurons and the victim neuron receive ballots.
4. Victim calls `RegisterVote { proposal: P, vote: Yes }`.
5. `register_vote` → `cast_vote_and_cascade_follow` enters the BFS, finds `N` followers of the victim in `function_followee_index`, iterates over all of them synchronously.
6. The call exhausts the IC per-message instruction limit (~20 billion instructions) and traps. The victim's vote is not recorded.
7. The attacker's neurons persist; the attack repeats for every subsequent proposal where they hold ballots, requiring no further action from the attacker.

## Impact Explanation

This is a concrete application/platform-level DoS against a specific neuron's governance participation. The victim permanently loses the ability to vote on any proposal where the attacker's neurons hold ballots. Because the attacker's follower setup persists across proposals, the DoS is not a one-time event but an ongoing suppression of the victim's voting rights. This matches the allowed ICP bounty impact: **High ($2,000–$10,000) — Application/platform-level DoS with concrete user and protocol harm in SNS governance.**

## Likelihood Explanation

The attack requires no privileged role; `follow` and `claim_or_refresh` are open to any principal. The primary constraint is economic: the attacker must stake the minimum neuron stake (`neuron_minimum_stake_e8s`) in each of the `N` neurons, and all neurons must be created before the targeted proposal is submitted. For SNS instances with low minimum stake or high token liquidity, the cost is feasible for a motivated adversary. The attack is technically straightforward, deterministic, and persistent once set up. **Likelihood: Low** due to capital cost, but not negligible for high-value SNS instances.

## Recommendation

1. **Add instruction-limit checkpoints to SNS `cast_vote_and_cascade_follow`**, mirroring the NNS `ProposalVotingStateMachine` pattern: check `is_over_instructions_limit()` after each follower is processed and store intermediate BFS state for continuation via a timer job.
2. **Enforce a per-followee inbound follower cap** in the `follow` function: reject a `Follow` call if the target followee already has more than a configured maximum number of followers for that function/topic, analogous to how `max_followees_per_function` caps the outbound direction.
3. **Alternatively**, adopt the NNS async voting state machine architecture for SNS governance so that large follower graphs are processed incrementally across multiple messages.

## Proof of Concept

1. Deploy a local SNS instance (via `pocket-ic` or `dfx` with SNS test fixtures) with a low `neuron_minimum_stake_e8s`.
2. Create `N` attacker neurons (start with N = 10,000; tune empirically) and have each call `Follow { function_id: 0, followees: [victim_neuron_id] }`.
3. Submit a proposal of function type 0. Confirm all `N + 1` neurons receive ballots.
4. Have the victim call `RegisterVote { proposal: P, vote: Yes }`.
5. Observe the call trap due to instruction exhaustion; confirm the victim's ballot remains `Unspecified`.
6. Repeat for the next proposal to confirm the attack is persistent with no additional attacker action.

A deterministic integration test using `pocket-ic` with instruction counting can establish the exact `N` threshold for a given SNS configuration.
