### Title
Missing Rollback for `new_settlement_layer_chain_id_storage` in IO Frame Snapshot Causes Persistent State on Transaction Revert - (`basic_system/src/system_implementation/system/io_subsystem.rs`)

### Summary

`FullIOStateSnapshot` and the `start_io_frame` / `finish_io_frame` pair in `io_subsystem.rs` snapshot and roll back every IO state component — persistent storage, transient storage, logs, events, and interop roots — but silently omit `new_settlement_layer_chain_id_storage`. As a result, any write to that storage that occurs inside a frame which is subsequently rolled back (reverted transaction, failed inner call, etc.) is **never undone**, leaving the settlement-layer chain-ID state permanently mutated even though the transaction reverted.

### Finding Description

`FullIO` owns six rollback-capable state components:

```
storage
transient_storage
logs_storage
events_storage
interop_root_storage
new_settlement_layer_chain_id_storage   ← present in FullIO, absent from snapshot
```

`FullIOStateSnapshot` captures only five of them:

```rust
pub struct FullIOStateSnapshot<M: StorageModel> {
    io: M::StateSnapshot,
    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,
    // new_settlement_layer_chain_id_storage: MISSING
}
```

`start_io_frame` takes no snapshot of `new_settlement_layer_chain_id_storage`:

```rust
fn start_io_frame(&mut self) -> Result<Self::StateSnapshot, InternalError> {
    let io        = self.storage.start_frame();
    let transient = self.transient_storage.start_frame();
    let messages  = self.logs_storage.start_frame();
    let events    = self.events_storage.start_frame();
    let interop_roots = self.interop_root_storage.start_frame();
    // self.new_settlement_layer_chain_id_storage.start_frame() — NEVER CALLED
    Ok(FullIOStateSnapshot { io, transient, messages, events, interop_roots })
}
```

`finish_io_frame` never rolls it back:

```rust
fn finish_io_frame(&mut self, rollback_handle: Option<&Self::StateSnapshot>) -> Result<(), InternalError> {
    self.storage.finish_frame(rollback_handle.map(|x| &x.io))?;
    self.transient_storage.finish_frame(rollback_handle.map(|x| &x.transient))?;
    self.logs_storage.finish_frame(rollback_handle.map(|x| x.messages));
    self.events_storage.finish_frame(rollback_handle.map(|x| x.events));
    self.interop_root_storage.finish_frame(rollback_handle.map(|x| x.interop_roots));
    // self.new_settlement_layer_chain_id_storage.finish_frame(...) — NEVER CALLED
    Ok(())
}
```

The parallel with `interop_root_storage` — which is structurally identical in purpose and IS correctly snapshotted — makes the omission clearly unintentional rather than a design choice. [1](#0-0) [2](#0-1) 

### Impact Explanation

The protocol enforces a one-update-per-block invariant on the settlement-layer chain ID (confirmed by the test `test_new_sl_chain_id_two_updates_fail`). Because the storage is never rolled back on revert:

1. **DoS on legitimate chain-ID updates**: An attacker submits a transaction that calls `update_settlement_layer_chain_id` and then deliberately reverts (e.g., via an explicit `REVERT` or by exhausting gas after the write). The write to `new_settlement_layer_chain_id_storage` survives the revert. Any subsequent legitimate update in the same block is rejected by the one-update guard, effectively blocking the protocol-level chain-ID migration for that entire block.

2. **State divergence between forward and proving modes**: The forward system (sequencer) and the proving system (RISC-V prover) both use the same `FullIO` / `FullIOStateSnapshot` path. A reverted transaction that mutates `new_settlement_layer_chain_id_storage` will leave the sequencer and prover in a state where the chain-ID storage reflects a write that the EVM state root does not account for, creating a forward/proving divergence. [3](#0-2) 

### Likelihood Explanation

The settlement-layer chain-ID update is triggered through a regular L2 transaction calling a system contract (as shown in the test harness). An unprivileged sender can submit such a transaction with a gas limit calibrated to exhaust gas immediately after the `update_settlement_layer_chain_id` system call completes but before the transaction returns successfully. The revert path in the bootloader calls `finish_global_frame(Some(&rollback_handle))`, which propagates to `finish_io_frame` with a rollback handle — but because `new_settlement_layer_chain_id_storage` is absent from the snapshot, the write is never undone. This is a single-transaction, zero-privilege attack requiring only the ability to submit a transaction. [4](#0-3) 

### Recommendation

Add `new_settlement_layer_chain_id_storage` to `FullIOStateSnapshot`, take its snapshot in `start_io_frame`, and roll it back in `finish_io_frame`, mirroring the existing pattern for `interop_root_storage`:

```rust
pub struct FullIOStateSnapshot<M: StorageModel> {
    io: M::StateSnapshot,
    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,
    new_sl_chain_id: <NewSettlementLayerChainIdStorage as Snapshottable>::SnapshotId, // add
}
```

```rust
fn start_io_frame(&mut self) -> Result<Self::StateSnapshot, InternalError> {
    // ... existing fields ...
    let new_sl_chain_id = self.new_settlement_layer_chain_id_storage.start_frame(); // add
    Ok(FullIOStateSnapshot { ..., new_sl_chain_id })
}

fn finish_io_frame(&mut self, rollback_handle: Option<&Self::StateSnapshot>) -> Result<(), InternalError> {
    // ... existing rollbacks ...
    self.new_settlement_layer_chain_id_storage
        .finish_frame(rollback_handle.map(|x| x.new_sl_chain_id)); // add
    Ok(())
}
```

### Proof of Concept

1. Deploy a contract whose fallback: (a) calls `update_settlement_layer_chain_id` via the system hook, then (b) executes a tight loop until out-of-gas.
2. Submit a transaction calling that contract with a gas limit just sufficient to complete step (a) but not step (b).
3. The transaction reverts with `OutOfGas`.
4. Observe that `new_settlement_layer_chain_id_storage` still records the update (the one-update guard is now tripped).
5. Submit a legitimate `update_settlement_layer_chain_id` transaction in the same block — it is rejected because the guard sees a prior update, even though the prior transaction reverted. [5](#0-4) [6](#0-5)
