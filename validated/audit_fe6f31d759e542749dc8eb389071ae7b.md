### Title
Scheduled Promise Permanently Deleted Before Execution Completes, Causing Irreversible Loss of Pending Operations — (`etc/xcc-router/src/lib.rs`)

### Summary

The XCC router's `execute_scheduled` function removes a user's scheduled promise from persistent storage **before** the promise executes. In NEAR's async execution model, the storage mutation is committed in the current receipt while the promise runs in a subsequent receipt. If the promise fails for any reason, the promise data is permanently erased and the user cannot retry the operation, potentially freezing any funds that were contingent on that call.

### Finding Description

`execute_scheduled` is the public entry point for executing promises that were previously stored via `schedule`. The function unconditionally removes the promise from the `scheduled_promises` map, then schedules it for execution: [1](#0-0) 

```rust
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

In NEAR's execution model, the state write (`scheduled_promises.remove`) is committed when the current function call receipt succeeds. The promise returned via `promise_return` executes in a **separate, subsequent receipt**. If that receipt fails (gas exhaustion, target contract panic, insufficient balance, etc.), NEAR reverts only the state changes from that receipt — the deletion of the promise from storage is **not** rolled back.

The `schedule` function that inserts promises is gated behind `assert_preconditions`, which requires the caller to be the Aurora Engine parent: [2](#0-1) 

This means a user cannot re-insert the same promise themselves. Once deleted, the scheduled operation is gone forever.

Critically, `execute_scheduled` is intentionally open to **any caller**: [3](#0-2) 

A malicious actor can call `execute_scheduled` with deliberately insufficient gas, or at a moment when the target contract is in a state that guarantees failure (paused, unregistered, etc.), causing the promise to fail and be permanently erased.

### Impact Explanation

The XCC router is the execution layer for Aurora EVM cross-contract calls. Scheduled promises represent pending financial operations: token claims from external DeFi protocols, reward withdrawals, or any other NEAR cross-contract call initiated from an EVM address. If such a promise is deleted before it executes successfully, the funds that were supposed to flow from that call remain locked in the external contract. Because only the Aurora Engine can re-schedule promises (via `schedule`), and the Engine only does so in response to a new EVM transaction invoking the XCC precompile, the user would need to construct an entirely new call — which may not be possible if the external protocol's state has already advanced (e.g., a one-time reward claim that was already "consumed" by the failed receipt).

**Impact: High — Permanent freezing of funds / theft of unclaimed yield.**

### Likelihood Explanation

`execute_scheduled` is callable by any NEAR account with no access restriction. An adversary who observes a scheduled promise (all NEAR state is public) can:

1. Identify a high-value scheduled promise.
2. Call `execute_scheduled` with a gas amount insufficient for the target call to succeed.
3. The promise is deleted; the user's pending operation is permanently lost.

No privileged access, leaked keys, or governance capture is required. The attacker needs only to submit a NEAR transaction.

**Likelihood: Medium** — requires an adversary to monitor router state and time the call, but the mechanism is straightforward and the router state is fully public on-chain.

### Recommendation

Attach a self-callback to `execute_scheduled` that checks the promise result and **re-inserts** the promise into `scheduled_promises` if the execution failed. This mirrors the pattern used elsewhere in the engine (e.g., `exit_to_near_precompile_callback` re-mints tokens on failure): [4](#0-3) 

Alternatively, only remove the promise from storage inside a success-only callback, keeping it in storage until confirmed execution.

### Proof of Concept

1. Alice uses the Aurora XCC precompile to schedule a promise that calls `ft_transfer` on an external NEP-141 contract to claim her accrued rewards. The promise is stored at nonce `N` in her router's `scheduled_promises`.
2. Bob (adversary) observes the scheduled promise on-chain.
3. Bob calls `execute_scheduled({nonce: N})` on Alice's router contract, attaching only enough gas for the `execute_scheduled` function itself but not enough for the `ft_transfer` promise to complete.
4. `scheduled_promises.remove(&N)` executes and is committed — Alice's promise is deleted from storage.
5. The `ft_transfer` promise receipt fails due to gas exhaustion; NEAR reverts only that receipt's state changes.
6. Alice's rewards remain in the external NEP-141 contract. Her router no longer holds the scheduled promise. She cannot retry through the router (only the Aurora Engine parent can call `schedule`), and her rewards are permanently inaccessible through the XCC mechanism. [1](#0-0) [2](#0-1) [5](#0-4)

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }
