Audit Report

## Title
`spawn_neuron` Unconditionally Copies Parent Hot Keys to Child Neuron Assigned to a Different Controller — (`rs/nns/governance/src/governance.rs`)

## Summary
When `spawn_neuron` is called with a `new_controller` that differs from the parent neuron's controller, the child neuron is built with the new controller but inherits the parent's full `hot_keys` list verbatim. Any principal that was a hot key on the parent neuron retains the ability to vote, follow, and modify Neuron Fund membership on the child neuron without the new controller's knowledge or consent.

## Finding Description
In `spawn_neuron`, the child controller is resolved at lines 2651–2655: if `spawn.new_controller` is set, `child_controller` becomes that third-party principal (Charlie); otherwise it defaults to the parent's controller. [1](#0-0) 

The `NeuronBuilder` call at lines 2704–2718 then constructs the child neuron with `child_controller` as its controller but unconditionally calls `.with_hot_keys(parent_neuron.hot_keys.clone())`, copying every hot key from the parent regardless of whether `child_controller` differs from the parent's controller. [2](#0-1) 

There is no conditional check or clearing of hot keys when `child_controller != parent_neuron.controller()`. The child neuron is stored in governance state with Charlie as controller and Bob (parent's hot key) in `hot_keys`.

`is_authorized_to_vote` delegates to `is_hotkey_or_controller`, which returns `true` if the caller is in `hot_keys`: [3](#0-2) 

`is_authorized_to_configure_or_err` also grants hot keys the ability to `JoinCommunityFund` / `LeaveCommunityFund`: [4](#0-3) 

By contrast, `split_neuron` always uses `*caller` as the child controller (line 2244), so the child controller is always the same principal who can immediately manage their own hot keys — the cross-controller scenario does not arise there. [5](#0-4) 

## Impact Explanation
This constitutes unauthorized access to a governance asset (neuron). Bob, a former hot key on Alice's parent neuron, retains the ability to:
- Cast `RegisterVote` on NNS proposals on Charlie's behalf.
- Set `Follow` relationships on Charlie's neuron, causing it to auto-vote in ways Charlie did not choose.
- Call `JoinCommunityFund` / `LeaveCommunityFund` on Charlie's neuron, affecting Charlie's maturity exposure.

Charlie receives a neuron with pre-authorized principals they did not select and has no UI-level warning. This matches the **High** bounty impact: *Unauthorized access to neurons, governance assets … where exploitation requires meaningful per-target work or other constraints* — the constraints here are the specific setup conditions (Alice having hot keys and spawning to a different controller), and the governance impact on Charlie's neuron is concrete and persistent.

## Likelihood Explanation
Both preconditions are normal, documented operations: adding a hot key to a neuron and spawning maturity to a third party via `new_controller`. The `Spawn` proto message explicitly documents `new_controller` as the mechanism for gifting maturity to another principal. Bob requires no special privileges beyond having been a hot key on the parent neuron at any point before the spawn. The exploit is repeatable for every neuron spawned to a different controller while the parent has hot keys.

## Recommendation
In `spawn_neuron`, clear the hot keys when the child controller differs from the parent controller. The fix is localized to the `NeuronBuilder` call:

```rust
let hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};
// ...
.with_hot_keys(hot_keys)
``` [6](#0-5) 

## Proof of Concept
1. Alice controls neuron N with hot key Bob.
2. Alice calls `manage_neuron` → `Spawn { new_controller: Some(Charlie), nonce: Some(42), percentage_to_spawn: Some(100) }`.
3. `spawn_neuron` sets `child_controller = Charlie` (line 2651–2655) and builds the child neuron with `.with_hot_keys(parent_neuron.hot_keys.clone())` (line 2714), storing `hot_keys = [Bob]` on the child.
4. Bob sends `manage_neuron` → `RegisterVote { proposal: P, vote: Yes }` authenticated as Bob, targeting the child neuron ID.
5. Governance calls `is_authorized_to_vote` → `is_hotkey_or_controller` → Bob is in `hot_keys` → authorized → vote cast on Charlie's neuron without Charlie's knowledge.

A deterministic unit test can be written against the existing `governance.rs` test harness: create a parent neuron with a hot key, call `spawn_neuron` with a different `new_controller`, retrieve the child neuron, and assert that the inherited hot key can successfully call `RegisterVote` on the child — confirming the unauthorized access path.
