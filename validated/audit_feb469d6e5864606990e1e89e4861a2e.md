Audit Report

## Title
Unbounded Prometheus Label Cardinality via Attacker-Controlled `subnet_id` in Subnet API Routes — (`rs/boundary_node/ic_boundary/src/metrics.rs`)

## Summary
An unprivileged attacker can send HTTP requests to `/api/v2/subnet/{subnet_id}/read_state` (and v3/v4 equivalents) with an unbounded number of syntactically valid but non-existent `subnet_id` Principal values. Each unique value is unconditionally recorded as a distinct Prometheus label combination in the in-process registry before subnet existence is checked, causing unbounded memory growth and eventual OOM crash of the boundary node process.

## Finding Description
`validate_subnet_request` in `rs/boundary_node/ic_boundary/src/http/middleware/validate.rs` (lines 81–89) parses the URL path `{subnet_id}` as a `PrincipalId` using `Principal::from_text` — a format-only check — and inserts the resulting `SubnetId` into request extensions with no existence verification.

The `subnet_layers` middleware stack in `rs/boundary_node/ic_boundary/src/core.rs` (lines 1033–1040) is ordered as: `validate_subnet_request` → `common_layers` (which contains `middleware_metrics` at line 1008) → `middleware_subnet_lookup`. Because `ServiceBuilder` layers wrap top-down, `metrics_middleware` executes entirely outside and before `subnet_lookup`.

In `metrics_middleware` (`rs/boundary_node/ic_boundary/src/metrics.rs`, lines 493–495), the attacker-controlled `subnet_id` is read from request extensions and captured into `subnet_id_str` **before** `next.run(request).await` is called. After `next.run()` returns — even when `subnet_lookup` fails with `SubnetNotFound` — the code at lines 596–603 unconditionally calls `with_label_values(labels)` on all four metric objects (`counter`, `durationer`, `request_sizer`, `response_sizer`), creating a new in-memory time series for each novel `subnet_id` string.

`remove_stale_metrics` (lines 81–135) only filters the `Vec<MetricFamily>` snapshot passed to the Prometheus text encoder. It calls `mf.set_metric(metrics)` on the export snapshot but never calls `remove_label_values` on the underlying `HistogramVec`/`IntCounterVec` objects. The Prometheus registry's internal hash maps therefore retain every label combination ever seen and grow without bound.

## Impact Explanation
Each unique `subnet_id` label value creates 4 new metric series. Each `HistogramVec` series allocates bucket arrays (8 duration buckets, 5 request-size buckets, 4 response-size buckets defined). At ~1 KB per series, 1 million unique subnet IDs ≈ ~4 GB of registry memory. The boundary node process will OOM and crash, causing a denial of service for all IC API traffic routed through that node. This matches the allowed High impact: **Application/platform-level DoS, crash, or boundary/API availability impact not based on raw volumetric DDoS** ($2,000–$10,000). This is a code-level memory exhaustion vulnerability, not a raw volumetric DDoS.

## Likelihood Explanation
The attack requires only syntactically valid Principal IDs (29-byte values with correct CRC32 checksum), which are trivially generated programmatically. No authentication is required. A per-IP rate limit slows accumulation but does not prevent it — at 10 req/s, an attacker accumulates 864,000 unique label combinations per day from a single IP. Using multiple source IPs removes even this constraint. The per-subnet rate limiter is ineffective because each request uses a *different* subnet ID.

## Recommendation
1. **Move `subnet_lookup` before `metrics_middleware`** in the layer stack, or record `subnet_id` only from the `Arc<Subnet>` response extension (which is only set on successful lookup), so unknown IDs never become label values.
2. **Alternatively**, in `metrics_middleware`, replace `subnet_id_str` with `SUBNET_ID_UNKNOWN` when the response contains `ErrorCause::SubnetNotFound`, preventing unknown IDs from ever entering the label set.
3. **Defense-in-depth**: Call `remove_label_values` on the underlying metric objects (not just filter the export snapshot) when stale subnet IDs are detected in `remove_stale_metrics`.

## Proof of Concept
```python
import struct, base64, requests

def crc32(data):
    import binascii
    return binascii.crc32(data) & 0xFFFFFFFF

def make_principal(n: int) -> str:
    raw = b'\x04' + n.to_bytes(28, 'big')
    crc = struct.pack('>I', crc32(raw))
    encoded = base64.b32encode(crc + raw).decode().lower().rstrip('=')
    return '-'.join(encoded[i:i+5] for i in range(0, len(encoded), 5))

for i in range(1_000_000):
    sid = make_principal(i)
    requests.post(
        f"https://<boundary-node>/api/v2/subnet/{sid}/read_state",
        data=b'\xa0',
        headers={"Content-Type": "application/cbor"}
    )
# Monitor boundary node RSS: it grows proportionally to i
# Scrape /metrics: http_request_total cardinality is bounded (stale filtering),
# but process RSS grows unboundedly, confirming the registry leak
```

A local integration test can confirm the bug without mainnet access: instantiate `ic_boundary` with a mock registry containing zero subnets, send N requests with N distinct valid-format Principal IDs to `/api/v2/subnet/{id}/read_state`, then assert that `registry.gather()` returns a growing number of metric series proportional to N.
