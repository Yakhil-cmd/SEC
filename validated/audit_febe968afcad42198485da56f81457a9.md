### Title
Missing `pausable` Modifier on Exit Functions Allows Withdrawals During Emergency Pause — (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`withdrawToNear` and `withdrawToEthereum` in both `EvmErc20` and `EvmErc20V2` are not decorated with the `pausable` modifier. After `adminPause(type(uint).max)` is called, any unprivileged user can still burn tokens and invoke the exit precompiles, defeating the entire purpose of the pause mechanism for the most critical state-changing operations.

---

### Finding Description

`AdminControlled` defines a `pausable(uint flag)` modifier: [1](#0-0) 

And `adminPause` sets the bitmask: [2](#0-1) 

In `EvmErc20`, both exit functions carry **no** `pausable` modifier: [3](#0-2) [4](#0-3) 

The same is true in `EvmErc20V2`: [5](#0-4) [6](#0-5) 

A compounding factor: in both contracts the assembly `call` result (`res`) is captured but never checked. If the precompile call fails silently, `_burn` has already executed and the tokens are gone with no corresponding unlock on the NEAR side.

---

### Impact Explanation

**Critical — Insolvency.**

1. Admin calls `adminPause(type(uint).max)` during an emergency (e.g., the NEAR-side connector is in an inconsistent state, a bridge exploit is being patched, or a migration is in progress).
2. Any user calls `withdrawToNear` or `withdrawToEthereum` — the call succeeds because there is no pause gate.
3. `_burn` destroys the user's ERC-20 tokens on the Aurora/EVM side.
4. The precompile at `0xe921...` or `0xb0bd...` is invoked. If the NEAR-side is in the inconsistent state that motivated the pause, the precompile call can fail silently (return value is never checked).
5. Tokens are permanently destroyed on the EVM side with no corresponding release on NEAR → total supply on NEAR and total supply on Aurora diverge → insolvency.

Even if the precompile does not fail, allowing exits during a pause window (e.g., while a migration or emergency fix is applied) can drain the backing reserve before the fix is complete.

---

### Likelihood Explanation

The admin pause is a standard emergency mechanism. Any time it is invoked, the window is open for this exploit. A user does not need any special privilege — only a token balance. The path is a single direct EVM call with no preconditions beyond holding tokens.

---

### Recommendation

Apply the `pausable` modifier to both exit functions in both contracts, using a dedicated pause bit (e.g., bit `1` for `withdrawToNear`, bit `2` for `withdrawToEthereum`):

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount)
    external
    override
    pausable(PAUSE_WITHDRAW_TO_NEAR)   // add this
{
    ...
}

function withdrawToEthereum(address recipient, uint256 amount)
    external
    override
    pausable(PAUSE_WITHDRAW_TO_ETHEREUM)  // add this
{
    ...
}
```

Additionally, check the return value of the precompile `call` and revert on failure so that a failed cross-chain message never silently burns tokens.

### Proof of Concept

```solidity
// SPDX-License-Identifier: CC0-1.0
pragma solidity ^0.8.0;

// Preconditions:
//   - deploy EvmErc20 with `admin = ADMIN`
//   - mint 100e18 tokens to ATTACKER

// Step 1: admin pauses everything
vm.prank(ADMIN);
token.adminPause(type(uint256).max);

// Step 2: verify paused
assertEq(token.paused(), type(uint256).max);

// Step 3: non-admin calls withdrawToNear — should revert if pause worked, but it does NOT
vm.prank(ATTACKER);
token.withdrawToNear(abi.encodePacked(nearRecipient), 100e18);

// Step 4: tokens are burned; ATTACKER balance is now 0
assertEq(token.balanceOf(ATTACKER), 0);
// Precompile call may have failed silently → supply/backing desync confirmed
```

The call at Step 3 succeeds on unmodified code, confirming the invariant is broken.
