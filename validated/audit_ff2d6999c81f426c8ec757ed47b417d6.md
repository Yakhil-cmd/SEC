Audit Report

## Title
Nodes Without iDKG Key Timestamps Bypass Subnet-Wide Cooldown and Reset the Rotation Timer, Delaying Peer Key Rotations - (File: `rs/registry/canister/src/mutations/do_update_node_directly.rs`)

## Summary
In `do_update_node`, the subnet-wide serialization guard (step 4) is gated entirely on `previous_timestamp_set`, which is `false` whenever a node has no prior timestamp. A node operator whose node lacks a timestamp can call `update_node_directly` immediately, skip the gamma-cooldown check, and write a fresh timestamp that becomes the new `last_key_update_on_subnet` maximum — imposing a full gamma-length cooldown on every other node in the subnet. Chaining this across multiple timestamp-less nodes extends the blocking window proportionally, degrading the forward-secrecy guarantee of iDKG dealing encryption.

## Finding Description
`do_update_node` in `rs/registry/canister/src/mutations/do_update_node_directly.rs` sets `previous_timestamp_set = false` when the node's stored key has no timestamp (line 100) or no key record exists at all (line 103). The subnet-wide guard at lines 108–123 is then skipped entirely:

```rust
if previous_timestamp_set                          // false → entire block skipped
    && let Some(last_key_update_timestamp) = self.last_key_update_on_subnet(subnet_record)
{ ... return Err("the signing subnet had a key update recently"); }
```

After the bypass, line 137 unconditionally stamps the new key with the current time:
```rust
pk.timestamp = Some(duration_since_unix_epoch.as_millis() as u64);
```

`last_key_update_on_subnet` (lines 167–184) returns the `.max()` of all node timestamps, so this new value immediately becomes the subnet-wide high-water mark. Every subsequent node with `previous_timestamp_set = true` must now wait `gamma = delta / subnet_size * 0.85` before it can rotate. The orchestrator mirrors this bypass at `rs/orchestrator/src/registration.rs` lines 485–488: `if own_key_timestamp.is_none() { return true; }`, bypassing `is_time_to_rotate_in_subnet` entirely.

The code comment at line 86–87 acknowledges the intent ("we accept the update as this is a new node joining the signing subnet"), but does not account for the side-effect of resetting the subnet-wide timer and blocking peers.

## Impact Explanation
Each bypass by a timestamp-less node imposes a gamma ≈ `delta / N * 0.85` cooldown on all other nodes. For a 13-node subnet with `delta = 2 weeks`, gamma ≈ 1.1 days per bypass. A node operator controlling K nodes without timestamps (e.g., when chain-key signing is first enabled on an existing subnet, or when a batch of new nodes is added) can chain K consecutive bypasses, blocking all remaining nodes from rotating their iDKG dealing encryption keys for up to K × gamma. Nodes unable to rotate continue using older keys beyond the configured security window, degrading the forward-secrecy property of the iDKG dealing encryption scheme. This matches the Medium allowed impact: node/boundary-node control required, meaningful security impact on chain-key signing infrastructure.

## Likelihood Explanation
The precondition — a node in a signing subnet with no iDKG key timestamp — arises naturally in two routine NNS-governed operations: (1) enabling chain-key signing on an existing subnet (all existing nodes have no timestamp), and (2) adding new nodes to a signing subnet. The call to `update_node_directly` is authenticated only by the node's own signing key; no additional privilege beyond being a registered node is required. A single node operator controlling even one such node can trigger the bypass at will during these windows.

## Recommendation
Decouple first-time key registration from the subnet-wide cooldown update. When `previous_timestamp_set == false` (first-time registration), allow the key write to proceed but do not let the new timestamp participate in `last_key_update_on_subnet`'s maximum computation — either by excluding timestamp-less-to-timestamped transitions from the max, or by tracking first-time registrations separately and not updating the subnet-wide high-water mark for them. This preserves the ability of new nodes to register immediately while preventing them from imposing a gamma-length cooldown on all other nodes.

## Proof of Concept
Using the existing unit-test harness in `rs/registry/canister/src/mutations/do_update_node_directly.rs`:

1. Create a 13-node signing subnet with `idkg_key_rotation_period_ms = 2 weeks`. Do **not** pre-populate any node's iDKG key (leave all timestamps absent).
2. Call `registry.do_update_node(now, node_A, payload_A)` — succeeds (`previous_timestamp_set = false`, subnet check skipped). `last_key_update_on_subnet` = `T_now`.
3. Immediately call `registry.do_update_node(now + ε, node_B, payload_B)` (node B also has no timestamp) — succeeds again. `last_key_update_on_subnet` = `T_now + ε`.
4. Now call `registry.do_update_node(now + ε, node_C, payload_C)` where node C **has** a timestamp older than delta — expect `Err("the signing subnet had a key update recently")` because `T_now + ε + gamma > now + ε`.
5. Verify that nodes C through M are blocked for approximately `gamma` after each successive timestamp-less registration, confirming the chained blocking effect.
