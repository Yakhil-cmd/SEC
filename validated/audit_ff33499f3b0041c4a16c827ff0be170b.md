Audit Report

## Title
Free Ingress Induction for `UpdateSettings` with Small Payload Enables Zero-Cost Subnet Ingress Starvation - (`rs/cycles_account_manager/src/cycles_account_manager.rs` / `rs/execution_environment/src/execution_environment.rs`)

## Summary
`UpdateSettings` ingress messages with a payload ≤ 338 bytes (`MAX_DELAYED_INGRESS_COST_PAYLOAD_SIZE`) are classified as `IngressInductionCost::Free` at every pre-execution checkpoint. The deferred post-execution fee collection explicitly discards the `CanisterOutOfCyclesError`. An attacker who controls one or more zero-balance canisters can therefore submit `UpdateSettings` messages at zero ongoing economic cost, consuming ingress block slots and execution round capacity, and starving legitimate canister calls.

## Finding Description

**Root cause — fee deferred and silently dropped:**

In `ingress_induction_cost()`, when the method is `UpdateSettings` and `is_delayed_ingress_induction_cost(ingress.arg())` returns `true` (payload ≤ 338 bytes), `paying_canister` is set to `None`, which maps to `IngressInductionCost::Free`: [1](#0-0) 

The constant is: [2](#0-1) 

**All pre-execution cycle gates pass `Free` messages unconditionally:**

- `should_accept_ingress_message` (ingress filter): the `IngressInductionCost::Fee` branch that calls `can_withdraw_cycles_with_threshold` is never entered: [3](#0-2) 

- `validate_ingress` (ingress selector): the `Free` arm does nothing: [4](#0-3) 

- `enqueue` (valid-set rule): the `Free` arm pushes the message directly without any balance check: [5](#0-4) 

**Post-execution fee collection silently discards the error:**

After `update_settings` executes, the deferred fee attempt explicitly ignores `CanisterOutOfCyclesError`: [6](#0-5) 

**The subnet ingress queue has no upper bound:**

The `IngressQueue` implementation explicitly documents that there is no capacity limit: [7](#0-6) 

**Block-level limit is the only constraint, and it is shared:**

The hard cap is `MAX_INGRESS_MESSAGES_PER_BLOCK = 1000` messages per block: [8](#0-7) 

The ingress selector uses round-robin scheduling across canisters with per-canister byte quotas: [9](#0-8) 

An attacker who creates N zero-balance canisters (one-time cost) and submits `UpdateSettings` messages from each can claim N/(N+legitimate) of every block's 1000-message budget at zero ongoing cost, proportionally starving legitimate traffic.

## Impact Explanation

This is a platform-level application subnet DoS. By creating a modest number of zero-balance canisters (one-time cost) and continuously submitting small `UpdateSettings` messages, an attacker can occupy a dominant fraction of every block's ingress budget. Legitimate update calls experience proportionally increased latency or are indefinitely deferred. This matches the allowed impact: **Application/platform-level DoS, subnet availability impact not based on raw volumetric DDoS** (High, $2,000–$10,000). The execution overhead per message is low (settings update, no Wasm), but the ingress slot consumption is real and bounded only by the block message count limit shared across all users.

## Likelihood Explanation

The attack requires only:
1. Creating one or more canisters on the target subnet (one-time cycles cost, minimal).
2. Draining those canisters to zero cycles.
3. Submitting `UpdateSettings` ingress messages in a loop with a payload ≤ 338 bytes.

No privileged access, no governance majority, no threshold corruption, and no external dependency is required. Any unprivileged user can execute this. The ongoing cost is zero because the fee collection error is explicitly discarded. The attack is repeatable indefinitely.

## Recommendation

1. **Require a minimum balance check even for deferred-fee messages.** Before inducting a `Free`-classified `UpdateSettings` message, verify that the target canister holds at least the induction fee (or a small fixed deposit). This preserves the unfreeze use case while preventing zero-cost flooding.
2. **Rate-limit `UpdateSettings` ingress messages per canister per block**, independent of the fee mechanism, to bound worst-case throughput of this path.
3. **Do not silently ignore the fee collection error.** At minimum, emit a metric when `consume_cycles` fails for the deferred `UpdateSettings` fee so operators can detect and respond to abuse.

## Proof of Concept

```
1. Create canister C on an application subnet with minimal cycles.
2. Drain C's cycles balance to zero.
3. In a loop, submit ingress messages:
     destination: IC_00 (management canister)
     method:      update_settings
     payload:     UpdateSettingsArgs { canister_id: C, settings: { freezing_threshold: k } }
                  (encoded size ≤ 338 bytes — confirmed by the test at
                   rs/cycles_account_manager/src/cycles_account_manager/tests.rs)
4. Observe:
   - ingress_filter accepts each message (IngressInductionCost::Free, no balance check).
   - ingress_selector accepts each message (Free arm, no balance check).
   - valid_set_rule enqueues each message directly (no balance check, unbounded IngressQueue).
   - execution_environment executes each message and silently drops the fee error.
   - C's balance remains zero throughout.
5. Repeat with N canisters to claim N/(N+k) of every block's 1000-message budget,
   proportionally starving legitimate calls.
```

A deterministic integration test can be written using `StateMachine` or `ValidSetRuleImpl` directly: create N zero-balance canisters, submit N×M `UpdateSettings` messages, submit M legitimate update calls, execute rounds, and assert that legitimate call latency grows linearly with N.
