Audit Report

## Title
Reentrancy via Checks-Effects-Interactions Violation in `claim_neurons` Allows Double-Invocation of Governance - (File: `rs/nns/gtc/src/lib.rs`)

## Summary
The `claim_neurons` function in the Genesis Token Canister sets `has_claimed = true` **after** an inter-canister `await` to NNS Governance, violating the checks-effects-interactions pattern. On the Internet Computer, a canister may process additional ingress messages at every `await` point, so a GTC account holder can submit two concurrent `claim_neurons` calls that both pass the `has_claimed` guard before either writes it, causing `GovernanceCanister::claim_gtc_neurons` to be invoked twice with the same neuron IDs. The same structural flaw exists in `donate_account` / `has_donated`.

## Finding Description
In `rs/nns/gtc/src/lib.rs`, `claim_neurons` (lines 40–70):

```
Line 62-64: guard read  — if account.has_claimed { return Ok(...) }
Line 66:    await point — GovernanceCanister::claim_gtc_neurons(...).await?;
Line 68:    guard write — account.has_claimed = true;
```

The GTC canister stores all state in a single `static mut GTC` global, accessed via `gtc_mut()` (`rs/nns/gtc/canister/canister.rs`, lines 19–34). The `claim_neurons` endpoint is registered as an `over_async` update call (canister.rs lines 144–153), reachable by any external ingress sender who controls the corresponding secp256k1 private key.

IC execution guarantees single-threaded message processing **within** a turn, but suspends the current execution at every `await` and allows other queued messages to run. The interleaving is:

1. Ingress A arrives: `has_claimed = false` → passes guard → sends `claim_gtc_neurons` to governance → suspends at `await`.
2. Ingress B arrives for the same account: `has_claimed` is **still** `false` → passes guard → sends `claim_gtc_neurons` to governance a second time → suspends at `await`.
3. A's response callback resumes: sets `has_claimed = true`.
4. B's response callback resumes: sets `has_claimed = true` (already set, but the second governance call already fired).

The same pattern exists in `donate_account` (lines 75–93): `account.transfer(...).await?` is called before `account.has_donated = true`. Inside `AccountState::transfer` (lines 174–210), the loop over neuron IDs contains an `await` on each iteration (`GovernanceCanister::transfer_gtc_neuron(...).await`), and the outer `has_donated` flag is only set by the caller after the entire future resolves.

## Impact Explanation
A GTC account holder can cause `GovernanceCanister::claim_gtc_neurons` to be invoked twice with the same neuron IDs. If governance's `claim_gtc_neurons` is not strictly idempotent, neurons are transferred to the caller's principal twice — unauthorized minting/duplication of NNS governance neurons. Even if governance is idempotent and rejects the second call, the `?` on line 66 causes the second `claim_neurons` invocation to return `Err(...)` before reaching `account.has_claimed = true`; however, since the first callback already set `has_claimed = true`, the net GTC state is correct in that scenario. The primary risk is the double-governance-call path when governance does not deduplicate. This matches the **High** impact category: unauthorized access to / duplication of NNS governance neurons.

## Likelihood Explanation
Exploitation requires only that the legitimate GTC account holder (who controls the secp256k1 private key for the account) submits two ingress `claim_neurons` update calls before the first call's response is processed. This is trivially achievable with any standard IC agent by firing two calls in rapid succession. No privileged access, subnet-majority corruption, or external compromise is required. The note in the claim that "anyone who knows the public key" can exploit this is inaccurate — the IC authenticates the caller's principal against the public key, so the private key must be held — but the account holder themselves can trivially perform the attack.

## Recommendation
Apply the checks-effects-interactions pattern: set `has_claimed = true` **before** the `await`, and revert on failure:

```rust
// claim_neurons:
account.has_claimed = true;
let result = GovernanceCanister::claim_gtc_neurons(caller, account.neuron_ids.clone()).await;
if result.is_err() {
    account.has_claimed = false;
    return Err(...);
}
Ok(account.neuron_ids.clone())
```

Apply the same fix to `donate_account` (`has_donated`) and `forward_whitelisted_unclaimed_accounts` (`has_forwarded`): set the guard flag before the inter-canister call and revert it if the call fails.

## Proof of Concept
1. Obtain a GTC account's secp256k1 private key (the account holder's own key).
2. Using an IC agent (e.g., `ic-agent` in Rust or `@dfinity/agent` in JS), construct two `claim_neurons(hex_pubkey)` update call requests signed with that key.
3. Submit both calls to the GTC canister without waiting for the first response (fire-and-forget both before polling for replies).
4. Both calls read `has_claimed = false` synchronously, both pass the guard, and both enqueue `claim_gtc_neurons` messages to the NNS Governance canister.
5. Observe via governance query or replica logs that `claim_gtc_neurons` is invoked twice for the same neuron IDs, confirming the double-invocation. If governance is not idempotent, both calls succeed and the neurons are transferred twice.
