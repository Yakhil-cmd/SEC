Audit Report

## Title
ICP Burn Silently Ignored After Successful Cycles Minting, Breaking Conservation Invariant - (File: `rs/nns/cmc/src/main.rs`)

## Summary
The `burn_and_log` function in the Cycles Minting Canister unconditionally swallows all ICP ledger burn errors after cycles have already been deposited to the caller. If the burn call fails for any reason, cycles are permanently credited to the recipient while the corresponding ICP remains unburned in the CMC's subaccount, and the notification is permanently closed — preventing any retry or recovery. This violates the ICP/cycles conservation invariant: cycles are created without the corresponding ICP being destroyed.

## Finding Description
In `rs/nns/cmc/src/main.rs`, three processing functions call `burn_and_log` **after** the irreversible action has already succeeded:

- `process_top_up` (L1999–2002): calls `deposit_cycles`, then `burn_and_log`
- `process_mint_cycles` (L1966–1973): calls `do_mint_cycles`, then `burn_and_log`
- `process_create_canister` (L1943–1946): calls `do_create_canister`, then `burn_and_log`

The `burn_and_log` function (L2014–2049) is explicitly annotated: *"Burning doesn't return errors — we don't want to reject the transaction notification because then it could be retried."* It returns `()` regardless of outcome, silently no-opping on three distinct failure paths: `minting_account_id` is `None` (L2020–2023), `amount < DEFAULT_TRANSFER_FEE` (L2027–2030), or the `call_protobuf` ledger call returns `Err` (L2044–2047).

After `burn_and_log` returns, the caller stores a terminal `NotificationStatus` (`NotifiedTopUp`, `NotifiedMint`, or `NotifiedCreateCanister`) at L1214–1222, L1305–1313, and L1411–1419. The `is_transient_error` check at L1219/1310/1416 only removes the entry if `process_*` itself returned a retriable `NotifyError` — but `process_*` returns `Ok(...)` when the cycles deposit succeeded, so the notification is permanently finalized regardless of whether the burn succeeded.

There is no heartbeat, recovery path, or operator mechanism to retroactively burn ICP that was left in a CMC subaccount after a failed `burn_and_log`. The ICP is permanently stranded.

The design rationale is documented: if the notification were rejected on burn failure, the user could retry, and on retry the cycles would be deposited again (double-spend). The code deliberately accepts the conservation violation to prevent double-spending. However, this means any transient ledger failure during the burn window permanently breaks the invariant.

## Impact Explanation
This constitutes **illegal minting** under the NNS/CMC financial integrity impact class: cycles are created and credited to a recipient without the corresponding ICP being destroyed. The ICP remains in the CMC's subaccount indefinitely with no recovery path. Over time, repeated transient ledger failures (e.g., during ledger upgrades) accumulate unburned ICP in CMC subaccounts while the cycles supply grows unchecked relative to the ICP supply. This is a concrete, permanent protocol-level conservation invariant violation in an in-scope NNS canister with direct financial impact on the ICP/cycles exchange integrity.

Severity: **High** — the impact is significant and concrete (cycles minted without ICP burn, permanent ICP stranding), but the attacker cannot directly control the trigger timing and does not extract the stranded ICP, constraining it below Critical.

## Likelihood Explanation
No privileged access is required. Any user calling `notify_top_up`, `notify_mint_cycles`, or `notify_create_canister` during a window when the ICP ledger is transiently unavailable (mid-upgrade, message queue full, subnet reject) will trigger this. ICP ledger upgrades are routine NNS governance operations occurring multiple times per year; during the upgrade window (typically seconds to minutes), all inter-canister calls to the ledger fail. Multiple users transacting during this window will each silently fail their burn. This is not a theoretical scenario — transient inter-canister call failures to NNS canisters have occurred on mainnet.

## Recommendation
1. **Do not finalize the notification if the burn fails.** If `burn_and_log` returns a ledger error, store a distinct `BurnFailed` status rather than `NotifiedTopUp`/`NotifiedMint`/`NotifiedCreateCanister`, allowing a privileged operator or automated retry to complete the burn later.
2. **Alternatively, implement a retry queue.** On burn failure, enqueue `(subaccount, amount)` in stable state and retry in `heartbeat` until the burn succeeds, only then transitioning to the terminal notification status.
3. **Add an invariant monitor** in `heartbeat` that sums all CMC subaccount balances and alerts (or auto-retries burns) when residual ICP is detected.
4. The current design comment acknowledges the tradeoff but the correct resolution is a `BurnFailed` terminal state (not retriable by the user, but recoverable by the system), not silent discard.

## Proof of Concept
1. User sends `N` ICP to the CMC subaccount keyed by their canister ID with memo `MEMO_TOP_UP_CANISTER`.
2. User calls `notify_top_up { block_index, canister_id }`.
3. CMC fetches the transaction, sets `blocks_notified[block_index] = Processing`, calls `process_top_up`.
4. `deposit_cycles` succeeds — target canister receives cycles.
5. `burn_and_log` is called. The ICP ledger is mid-upgrade; `call_protobuf(..., "send_pb", ...)` returns `Err(RejectionCode::SysTransient, "...")`. `burn_and_log` logs the error and returns `()`.
6. `process_top_up` returns `Ok(cycles)`. CMC stores `NotifiedTopUp(Ok(cycles))` — permanent terminal state.
7. Any subsequent `notify_top_up` call for the same block returns `NotifiedTopUp(Ok(cycles))` immediately (duplicate detection at L1187).
8. Result: target canister has received `N * rate` cycles; `N` ICP remains unburned in CMC's subaccount; conservation invariant broken.

**Deterministic integration test plan:** Use PocketIC with a mock ICP ledger that returns `SysTransient` on `send_pb`. Call `notify_top_up`, assert cycles were deposited, assert `blocks_notified[block_index]` is `NotifiedTopUp(Ok(...))`, assert CMC subaccount balance on the mock ledger is still `N` (not zero). This directly demonstrates the invariant violation without mainnet interaction.
