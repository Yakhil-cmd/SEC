Audit Report

## Title
Unprivileged Client Can Trigger Unconditional `todo!()` Panic via Hash-Only `/block` Request in `RosettaBlocksMode::Enabled` — (File: `rs/rosetta-api/icp/src/request_handler.rs`)

## Summary

When the ICP Rosetta node runs with `RosettaBlocksMode::Enabled`, any unauthenticated client can POST to `/block` with only a block hash (no index). If the hash does not correspond to a verified non-Rosetta block, `get_verified_block_by_hash` returns `Err(ApiError::InvalidBlockId(_, _))`, which unconditionally hits `todo!("Fetching Rosetta Blocks by hash is not supported yet")` at line 359, causing a Rust panic. This makes hash-based block lookup completely broken in this mode and exposes the server to repeated panic-induced worker crashes.

## Finding Description

The `get_block` function in `RosettaRequestHandler` at `rs/rosetta-api/icp/src/request_handler.rs` lines 349–366 contains the following incomplete branch:

```rust
Some(PartialBlockIdentifier { index: None, hash: Some(hash) }) => {
    if self.is_rosetta_blocks_mode_enabled().await {
        match self.get_verified_block_by_hash(&hash).await {
            Ok(block) => Ok(block),
            Err(ApiError::InvalidBlockId(_, _)) => {
                todo!("Fetching Rosetta Blocks by hash is not supported yet") // line 359
            }
            e => e,
        }
    } else {
        self.get_verified_block_by_hash(&hash).await
    }
}
```

`get_verified_block_by_hash` (lines 451–465) returns `Err(ApiError::InvalidBlockId(true, Default::default()))` at line 459 whenever `is_verified_by_hash` returns false — which is the normal outcome for any Rosetta block hash or any unknown hash. The `todo!()` macro unconditionally panics with no recovery path. The `/block` HTTP handler at `rs/rosetta-api/icp/src/rosetta_server.rs` lines 58–68 is unauthenticated and directly calls `req_handler.block(...)`, which reaches this code path. No existing guard prevents the panic: the only check is whether `RosettaBlocksMode` is enabled, which is the exact precondition that routes into the broken arm.

## Impact Explanation

`todo!()` expands to `panic!("not yet implemented: ...")`. In actix-web async handlers, this panic propagates through the tokio task, crashing the worker. Actix-web restarts workers after crashes, but repeated requests cause repeated crashes, degrading or denying service to all Rosetta API users. At minimum, every hash-only block lookup in `RosettaBlocksMode::Enabled` returns an unhandled panic instead of a structured error, making the feature entirely non-functional. The Rosetta API is a financial integration component used for ICP ledger operations by exchanges and integrators. This matches the allowed High impact: **Application/platform-level DoS or crash** and **Significant Rosetta/ledger security impact with concrete user or protocol harm**.

## Likelihood Explanation

- `RosettaBlocksMode::Enabled` requires `--enable-rosetta-blocks` at startup — non-default but a documented, production-intended feature.
- Trigger requires only a syntactically valid unauthenticated POST to `/block` with a hash-only `block_identifier`. No credentials, no special knowledge, no prior state required.
- Any hash value that is not a verified non-Rosetta block (including all Rosetta block hashes and arbitrary unknown hashes) triggers the panic.
- 100% deterministic and reproducible once the mode is enabled.

## Recommendation

Replace the `todo!()` with a graceful error return or a proper Rosetta block hash lookup:

```rust
Err(ApiError::InvalidBlockId(_, _)) => {
    // Rosetta block hash lookup not yet implemented; return structured error
    Err(ApiError::InvalidBlockId(
        false,
        "Fetching Rosetta Blocks by hash is not yet supported".into(),
    ))
}
```

Long-term, implement hash-based lookup against the `rosetta_blocks` table, consistent with the index-based path at lines 343–344.

## Proof of Concept

```bash
# Prerequisites: Rosetta node started with --enable-rosetta-blocks

curl -X POST http://localhost:8081/block \
  -H "Content-Type: application/json" \
  -d '{
    "network_identifier": {
      "blockchain": "Internet Computer",
      "network": "00000000000000020101"
    },
    "block_identifier": {
      "hash": "aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa"
    }
  }'
# Result: Rosetta worker panics with
# "not yet implemented: Fetching Rosetta Blocks by hash is not supported yet"
# Worker crashes; actix-web restarts it; repeating the request repeats the crash.
```

A deterministic integration test can be written by starting a local Rosetta instance with `--enable-rosetta-blocks`, syncing at least one Rosetta block, then issuing the above request and asserting the server does not panic.
