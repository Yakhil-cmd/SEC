### Title
Missing `error_refund` Recovery Mechanism in `ExitToNear` Precompile Causes Permanent Fund Freeze - (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns ERC-20 tokens (or locks ETH at the precompile address) before scheduling a NEAR-side `ft_transfer` promise. A refund mechanism (`error_refund`) exists in the codebase to re-mint burned tokens or return locked ETH when the NEAR-side promise fails, but this feature is **not enabled in the default production build**. The `refund` field in `ExitToNearPrecompileCallbackArgs` is hardcoded to `None` without the feature flag, so any failed NEAR-side transfer results in permanent, irrecoverable loss of user funds.

---

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) handles two token exit paths:

**ERC-20 exit (flag `0x1`):** The calling Solidity contract (`EvmErc20.withdrawToNear`) first burns the user's ERC-20 tokens via `_burn`, then calls the precompile. The precompile schedules a `ft_transfer` or `ft_transfer_call` promise on the NEP-141 contract.

**ETH exit (flag `0x0`):** ETH is transferred from the caller to the precompile address during EVM execution, then a `ft_transfer` promise is scheduled.

In both cases, the precompile constructs `ExitToNearPrecompileCallbackArgs`: [1](#0-0) 

Without the `error_refund` feature, `refund` is always `None`. The promise structure check at line 470 means that for regular ERC-20 and ETH exits (where `transfer_near_args` is also `None`), **no callback is scheduled at all**: [2](#0-1) 

When the NEAR-side `ft_transfer` fails (e.g., recipient not registered with the NEP-141 contract), there is no callback to invoke `engine::refund_on_error`. The `error_refund` feature, which would populate `refund` with a `RefundCallArgs` and trigger re-minting of burned ERC-20 tokens or return of locked ETH, is **not in the `default` feature set**: [3](#0-2) [4](#0-3) 

The `ExitToNearPrecompileCallbackArgs` struct and the `refund_on_error` function both exist and are fully implemented: [5](#0-4) [6](#0-5) 

But the recovery path is unreachable in the default build because `refund` is always `None`.

The callback handler confirms: when `args.refund` is `None` and the promise failed, the function returns `None` — doing nothing: [7](#0-6) 

The test suite explicitly documents this behavior: [8](#0-7) [9](#0-8) 

---

### Impact Explanation

**Permanent freezing of funds (Critical).** When a user calls `ExitToNear` (directly or via `EvmErc20.withdrawToNear`) and the NEAR-side `ft_transfer` fails:

- **ERC-20 case**: Tokens are burned from the user's EVM balance. The NEP-141 tokens remain in Aurora's custody. The user loses their ERC-20 tokens with no recovery path.
- **ETH case**: ETH is transferred to the `exit_to_near` precompile address during EVM execution. If the NEAR-side transfer fails, the ETH is permanently locked at that address.

In both cases, the funds are irrecoverable without the `error_refund` feature enabled.

---

### Likelihood Explanation

This is realistically triggerable by any unprivileged EVM user. The NEAR-side `ft_transfer` fails whenever the recipient account is not registered with the NEP-141 token (a common requirement in the NEAR fungible token standard). A user who provides an unregistered NEAR account ID as the exit recipient — a natural mistake — will permanently lose their tokens. No special privileges or coordination are required.

---

### Recommendation

Enable `error_refund` as a default feature in the production build of `aurora-engine` and `aurora-engine-precompiles`, or unconditionally populate the `refund` field in `ExitToNearPrecompileCallbackArgs` without gating it behind a compile-time feature. The refund logic in `engine::refund_on_error` and `exit_to_near_precompile_callback` is already fully implemented and only needs to be activated.

---

### Proof of Concept

1. Deploy an ERC-20 token on Aurora backed by a NEP-141.
2. Bridge NEP-141 tokens to ERC-20 on Aurora (user holds ERC-20 balance).
3. Call `EvmErc20.withdrawToNear("unregistered.near", amount)` — `unregistered.near` is not registered with the NEP-141 contract.
4. The ERC-20 tokens are burned from the user's balance (EVM state committed).
5. The NEAR-side `ft_transfer` to `unregistered.near` fails (account not registered).
6. Without `error_refund`, no callback fires, no re-mint occurs.
7. User's ERC-20 balance is `original - amount`; NEP-141 balance of `unregistered.near` is `0`. Funds are permanently lost.

This is confirmed by the existing test `test_exit_to_near_refund` which explicitly shows the balance discrepancy when `error_refund` is not enabled: [10](#0-9)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
