Fee-on-Transfer NEP-141 Token Mints Unbacked ERC-20 Tokens on Aurora — (`engine/src/engine.rs`)

---

### Summary

`receive_erc20_tokens` mints ERC-20 tokens on Aurora using `args.amount` — the amount *stated* in the `ft_on_transfer` callback — rather than the amount Aurora *actually received*. For any NEP-141 token that charges a fee on transfer, these two values differ. Aurora mints the full stated amount while holding fewer backing tokens, creating insolvency in the bridge.

---

### Finding Description

The NEP-141 `ft_transfer_call` flow works as follows:

1. A user calls `ft_transfer_call(receiver_id="aurora", amount=X, msg=<evm_address>)` on a NEP-141 token contract.
2. The NEP-141 contract deducts `X` from the sender and credits Aurora. If the token charges a fee on transfer, Aurora receives `X - fee`, not `X`.
3. The NEP-141 contract then calls `aurora.ft_on_transfer(sender_id=..., amount=X, msg=...)` — `amount` here is the *sent* amount, not the *received* amount.
4. Aurora's `ft_on_transfer` dispatches to `receive_erc20_tokens`.

Inside `receive_erc20_tokens`:

```rust
// engine/src/engine.rs:803
let amount = args.amount.as_u128();   // ← stated amount, not actual received amount
```

This `amount` is then passed directly to `setup_receive_erc20_tokens_input`, which encodes a call to `mint(recipient, amount)` on the ERC-20 contract:

```rust
// engine/src/engine.rs:831
setup_receive_erc20_tokens_input(&recipient, amount),
```

```rust
// engine/src/engine.rs:1306-1313
pub fn setup_receive_erc20_tokens_input(recipient: &Address, amount: u128) -> Vec<u8> {
    let selector = ERC20_MINT_SELECTOR;
    let tail = ethabi::encode(&[
        ethabi::Token::Address(recipient.raw().0.into()),
        ethabi::Token::Uint(amount.into()),
    ]);
    [selector, tail.as_slice()].concat()
}
```

And `EvmErc20.mint` unconditionally mints the full stated amount:

```solidity
// etc/eth-contracts/contracts/EvmErc20.sol:49-51
function mint(address account, uint256 amount) public onlyAdmin {
    _mint(account, amount);
}
```

There is no balance snapshot of Aurora's NEP-141 holdings before and after the transfer to verify what was actually received. The engine trusts `args.amount` entirely.

---

### Impact Explanation

**Critical — Insolvency.**

Each deposit of a fee-on-transfer NEP-141 token mints more ERC-20 tokens on Aurora than the NEP-141 tokens held in custody. The ERC-20 supply on Aurora exceeds the NEP-141 reserve. When users later withdraw (burn ERC-20 → receive NEP-141 via `exit_to_near`), the last withdrawers cannot be satisfied because the NEP-141 reserve is insufficient. This is a direct bridge insolvency: honest depositors of non-fee tokens lose their backing because an attacker (or even an innocent user of a fee token) inflated the ERC-20 supply.

An attacker who controls a fee-on-transfer NEP-141 token that is registered with Aurora can:
1. Repeatedly call `ft_transfer_call` with `amount=X`, paying only `X - fee` NEP-141 tokens but receiving `X` ERC-20 tokens.
2. Withdraw the excess ERC-20 tokens via `exit_to_near`, draining NEP-141 tokens deposited by other users.

---

### Likelihood Explanation

**Medium.** Any NEP-141 token registered with Aurora (via `deploy_erc20_token`) that implements a transfer fee triggers this bug. The registration step is permissioned, but once a fee-on-transfer token is registered — whether by mistake or by a malicious deployer — the insolvency path is fully open to any unprivileged user who simply calls `ft_transfer_call` on that token. No special privileges are required after registration.

---

### Recommendation

In `receive_erc20_tokens`, snapshot Aurora's NEP-141 balance of the bridged token *before* the `ft_on_transfer` callback is processed, and compare it to the balance *after* to determine the actual received amount. Use the delta — not `args.amount` — as the mint quantity. Concretely:

```
actual_received = aurora_nep141_balance_after - aurora_nep141_balance_before
mint(recipient, actual_received)
```

This requires a cross-contract read of the NEP-141 token's `ft_balance_of(aurora)` before and after, which can be structured as a promise chain, or alternatively the engine can reject registration of tokens that are known to charge transfer fees.

---

### Proof of Concept

1. Deploy a NEP-141 token `fee_token.near` that charges a 10% fee on every `ft_transfer` / `ft_transfer_call`.
2. Register it with Aurora: call `deploy_erc20_token` for `fee_token.near`. An ERC-20 mirror is deployed at address `ERC20_ADDR`.
3. Call `fee_token.near.ft_transfer_call(receiver_id="aurora", amount=1000, msg=<attacker_evm_addr>)`.
   - `fee_token.near` transfers 900 tokens to Aurora (1000 − 10% fee) and calls `aurora.ft_on_transfer(amount=1000, ...)`.
   - `receive_erc20_tokens` reads `args.amount = 1000` and calls `ERC20_ADDR.mint(attacker_evm_addr, 1000)`.
   - Aurora now holds 900 NEP-141 tokens but has minted 1000 ERC-20 tokens. **100 tokens are unbacked.**
4. Attacker calls `ERC20_ADDR.withdrawToNear(attacker_near_addr, 1000)` (exit_to_near precompile), triggering `ft_transfer(receiver_id=attacker_near_addr, amount=1000)` on `fee_token.near`.
   - Aurora's NEP-141 balance drops to 0 (or goes negative if other users' deposits are consumed).
   - Attacker received 900 NEP-141 tokens net (after the exit fee), having deposited only 900 — but drained 100 tokens from the shared reserve.
5. Repeat to drain the entire NEP-141 reserve held by Aurora for that token, leaving honest depositors unable to withdraw.
