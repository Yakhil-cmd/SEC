Audit Report

## Title
ICRC-21 Consent Message Hard Failure for u256-Token Amounts Exceeding u64::MAX — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
The `FieldsDisplay` path in the ICRC-21 consent message library unconditionally converts token amounts to `u64` via `nat_to_u64`. Because ckETH and all ckERC20 ledgers are deployed as u256-token variants, any amount exceeding `u64::MAX` (≈ 18.4 ETH in wei) causes `nat_to_u64` to return `Icrc21Error::GenericError`, aborting consent message generation entirely. Additionally, the `GenericDisplay` path converts amounts via `to_f64()`, silently losing precision for amounts above ~9×10¹⁵ wei.

## Finding Description
`Value::TokenAmount` stores `amount: u64`: [1](#0-0) 

`nat_to_u64` returns `GenericError` for any `Nat` whose inner `BigUint` exceeds `u64::MAX`: [2](#0-1) 

This is called unconditionally in `add_amount`, `add_fee`, `add_allowance`, and `add_existing_allowance` for the `FieldsDisplay` branch: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

For `GenericDisplay`, `convert_tokens_to_string_representation` uses `to_f64()`, which silently rounds for amounts above ~9×10¹⁵: [7](#0-6) 

The single public entry point `build_icrc21_consent_info_for_icrc1_and_icrc2_endpoints` is used by all ICRC-1/ICRC-2 ledgers: [8](#0-7) 

ckERC20 ledgers are deployed as the u256 variant (`ic-icrc1-ledger-u256.wasm.gz`), confirmed by the orchestrator upgrade proposal: [9](#0-8) 

The u256 ledger's `Tokens::MAX` is confirmed to exceed `u64::MAX` by the test suite: [10](#0-9) 

ckETH amounts are denominated in wei (18 decimals), so `u64::MAX` ≈ 18.4 ETH: [11](#0-10) 

There are no guards or fallback paths in the `FieldsDisplay` branch that handle amounts exceeding `u64::MAX`. The `?` operator propagates the `GenericError` immediately, aborting the entire consent message build.

## Impact Explanation
**FieldsDisplay (hard failure):** Any ckETH or ckERC20 user whose transfer, approval, or transfer-from amount exceeds `u64::MAX` receives `Icrc21Error::GenericError` instead of a valid consent message. Hardware wallets and ICRC-21-compliant wallets that request `FieldsDisplay` cannot render a consent screen, blocking those users from signing transactions through compliant wallet software. This is a concrete, user-facing DoS on a security-critical endpoint for ck-token ledgers.

**GenericDisplay (silent precision loss):** Amounts above ~9×10¹⁵ wei are displayed with rounding error due to `f64` conversion. A user approving `u256::MAX` as an infinite allowance would see a wildly incorrect number, undermining the security purpose of the consent message.

This matches: **High ($2,000–$10,000) — Significant Chain Fusion / ck-token security impact with concrete user or protocol harm.**

## Likelihood Explanation
The threshold is `u64::MAX` wei ≈ 18.4 ETH. At current ETH prices, this is a routine balance for retail and institutional ckETH holders. The exploit path is a standard unprivileged `icrc21_canister_call_consent_message` update call with no special role or precondition. It is deterministically reproducible for any amount exceeding the threshold. The `GenericDisplay` precision-loss path is triggered for any user with more than ~9,000 ETH or using `u256::MAX` as an infinite approval.

## Recommendation
1. Replace `Value::TokenAmount { amount: u64 }` with a `Nat` or `String` field, or add a `Text` fallback for amounts exceeding `u64::MAX`.
2. In `convert_tokens_to_string_representation`, replace `to_f64()` with arbitrary-precision integer arithmetic (e.g., integer division with remainder formatting) to avoid precision loss for large amounts.
3. Add test coverage for amounts exceeding `u64::MAX` on u256-token ledgers in the ICRC-21 consent message test suite.

## Proof of Concept
1. Deploy the u256-variant ICRC-1 ledger (as used for ckETH/ckERC20).
2. Mint `u64::MAX + 1` tokens to a test account.
3. Call `icrc21_canister_call_consent_message` with:
   - `method = "icrc1_transfer"`
   - `amount = u64::MAX + 1`
   - `device_spec = FieldsDisplay`
4. Observe `Icrc21Error::GenericError { description: "Failed to convert tokens to u64" }` instead of a valid consent message.

The failure is deterministic: `nat_to_u64` at line 330 calls `BigUint::to_u64()`, which returns `None` for any value requiring more than one 64-bit limb, and the `?` operator propagates the error immediately. [2](#0-1)

fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let tokens = tokens.0.to_f64().ok_or(Icrc21Error::GenericError {
        error_code: Nat::from(500_u64),
        description: "Failed to convert tokens to u64".to_owned(),
    })?;
    Ok(format!("{}", tokens / 10_f64.pow(decimals)))
}
