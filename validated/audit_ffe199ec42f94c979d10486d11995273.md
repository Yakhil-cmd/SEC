Audit Report

## Title
Unprivileged Canister Can Monopolize the Per-Key Threshold Signature Queue — (`rs/execution_environment/src/execution_environment.rs`)

## Summary

The `sign_with_threshold` function enforces only a global per-key queue size limit with no per-canister sub-limit. A single canister with sufficient cycles can occupy all `dynamic_queue_size` slots for a given `key_id`, causing every subsequent signing request from any other canister to be rejected with a queue-full error until the attacker's requests drain. Because the attacker receives valid signatures in return, the attack is economically self-sustaining.

## Finding Description

**Admission guard (no per-canister check):**

In `rs/execution_environment/src/execution_environment.rs` lines 3844–3858, the only queue-full guard is:

```rust
if state
    .metadata
    .subnet_call_context_manager
    .sign_with_threshold_contexts_count(&threshold_key)
    >= dynamic_queue_size
{
    return Err(UserError::new(...));
}
```

`sign_with_threshold_contexts_count` (lines 437–453 of `rs/replicated_state/src/metadata_state/subnet_call_context_manager.rs`) counts **all** pending contexts for the key regardless of `request.sender`:

```rust
pub fn sign_with_threshold_contexts_count(&self, key_id: &MasterPublicKeyId) -> usize {
    self.sign_with_threshold_contexts
        .iter()
        .filter(|(_, context)| match (key_id, &context.args) { ... })
        .count()
}
```

The backing store (`sign_with_threshold_contexts: BTreeMap<CallbackId, SignWithThresholdContext>` at line 218) is keyed only by `CallbackId`; there is no per-canister accounting anywhere in `SubnetCallContextManager` or in `sign_with_threshold`.

**Exploit path:**

1. Attacker deploys a canister on the target subnet.
2. Canister sends `dynamic_queue_size` (typically 20) signing requests for the target `key_id`, each with sufficient cycles.
3. All 20 slots are occupied by the attacker's canister.
4. Any subsequent `sign_with_ecdsa` / `sign_with_schnorr` / `vetkd_derive_key` call from any other canister is immediately rejected with `"request queue for key … is full."`.
5. As the attacker's requests are processed and slots free up, the attacker re-submits to maintain the blockade.
6. The attacker receives valid signatures for each request, recovering economic value.

**Existing mitigations are insufficient:**

- The cycles fee check (lines 3790–3797) only ensures the attacker pays per request; it does not limit how many requests one canister may have in-flight.
- `signature_request_timeout_ns` is an optional registry parameter; it is not universally configured and does not prevent the initial monopolization.
- `get_dynamic_signature_queue_size` (lines 3838–3842) may expand the queue based on pre-signature stash size, but this only raises the cost of the attack proportionally — it does not introduce fairness.

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS … subnet availability impact not based on raw volumetric DDoS."*

For subnets that host chain-key Bitcoin or Ethereum signing (ckBTC, ckETH, ckERC20), a sustained blockade of the threshold signature queue halts all cross-chain operations for legitimate users. The attacker does not need to corrupt consensus or hold any privileged role; the attack is entirely within the normal execution path.

## Likelihood Explanation

Requirements:
1. A deployed canister on the target subnet (no special privileges).
2. `max_queue_size × signature_fee` cycles upfront (e.g., 20 × 10 T = 200 T cycles for ECDSA on mainnet).
3. Continuous re-submission as slots free up.

The cycles cost is a real economic barrier but not a technical one. Because the attacker receives valid signatures in return, the net cost per round is only the opportunity cost of the cycles, not their full value. A well-funded attacker can sustain the DoS indefinitely.

## Recommendation

Introduce a per-canister sub-limit at the admission point in `sign_with_threshold`. Concretely, add a variant of `sign_with_threshold_contexts_count` that also filters by `request.sender`, and reject if the per-canister count for the given `key_id` exceeds a configurable cap (e.g., `max(1, max_queue_size / fairness_divisor)`). Alternatively, enforce a hard per-canister cap of 1–3 concurrent requests per canister per key. The cap should be enforced before the global queue-full check so that a single canister cannot consume all available slots.

## Proof of Concept

The existing test `test_sign_with_threshold_key_queue_fills_up` at `rs/execution_environment/tests/threshold_signatures.rs:848–905` already constitutes a local-reproducible proof: a single canister sends 20 requests, and the 21st is rejected with the queue-full error. To confirm the cross-canister monopolization invariant, extend the test as follows:

1. Create `canister_a` and `canister_b` on the same `StateMachine`.
2. Have `canister_a` send `max_queue_size` requests (filling the queue).
3. Have `canister_b` send one request.
4. Assert `canister_b`'s request is rejected with `"request queue for key … is full."`.

This requires no mainnet access and is deterministic under the existing `StateMachineBuilder` harness with signing disabled.
