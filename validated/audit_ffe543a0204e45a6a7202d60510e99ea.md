### Title
Systematic Pubdata Fee Undercharging Due to Asymmetric Rounding in `native_per_pubdata` Calculation — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` conversion rate (native resource units charged per pubdata byte) is computed using **floor/truncating division** (`wrapping_div`), while `native_per_gas` is computed using **ceiling division** (`div_ceil`). This asymmetry causes the system to systematically undercharge for pubdata on every L2 transaction that writes state, allowing users to generate pubdata at a lower cost than the protocol intends and reducing operator fee revenue proportionally to pubdata usage.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the two key conversion rates are computed as:

```rust
// native_per_gas: ceiling division — rounds UP (conservative for protocol)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...;

// native_per_pubdata: floor division — rounds DOWN (favorable for user)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

The same pattern appears in the public API helper:

```rust
// native_per_gas = ceil(gas_price / native_price)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(())?;

// native_per_pubdata = pubdata_price / native_price  ← floor
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

`native_per_pubdata` flows into two critical accounting paths:

1. **`create_resources_for_tx`** — computes the intrinsic pubdata overhead deducted from the native budget at transaction start.
2. **`get_resources_to_charge_for_pubdata`** — computes the actual native charge for pubdata bytes written during execution: `native = pubdata_used * native_per_pubdata`. [3](#0-2) 

The underestimated charge flows into `compute_gas_refund`, where it reduces `native_used`, which in turn reduces `delta_gas`, which reduces `gas_used`, which reduces the operator's fee payment:

```rust
resources.charge_unchecked(&to_charge_for_pubdata);   // underestimated
let native_used = full_native_limit
    .saturating_sub(resources.native().remaining().as_u64()); // too low
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);
if delta_gas > 0 { gas_used += delta_gas as u64; }   // gas_used too low
``` [4](#0-3) 

The operator is paid `gas_used * gas_price`, so the underestimated `gas_used` directly reduces operator revenue. [5](#0-4) 

---

### Impact Explanation

**Deficit per pubdata byte** (in native units) = `pubdata_price mod native_price`

**Deficit per pubdata byte** (in gas units) = `(pubdata_price mod native_price) / native_per_gas`

**Maximum deficit fraction** per pubdata byte = `(native_price − 1) / pubdata_price`

**Concrete example:**
- `pubdata_price = 10`, `native_price = 3`, `gas_price = 3`, `gas_limit = 100`, `pubdata_used = 10`
- `native_per_gas = ceil(3/3) = 1`
- Actual `native_per_pubdata = floor(10/3) = 3`; correct value = `ceil(10/3) = 4`
- Actual pubdata charge: `10 × 3 = 30` native → `30` gas → `90` tokens paid to operator
- Correct pubdata charge: `10 × 4 = 40` native → `40` gas → `120` tokens
- **Deficit: 30 tokens = 25% of expected pubdata fee**

The deficit grows **linearly** with pubdata usage. For high-pubdata transactions (many `SSTORE` operations), the cumulative deficit is significant. In the worst case (`pubdata_price` just above `native_price`), the deficit reaches ~50% of the expected pubdata fee.

---

### Likelihood Explanation

Every L2 transaction that writes state (any `SSTORE`) generates pubdata and is affected. The undercharging is automatic and requires no special attacker action beyond submitting normal transactions. An attacker maximizing the deficit would craft transactions with maximum pubdata (e.g., writing many distinct storage slots), which is a standard EVM operation pattern. The vulnerability is present in both the bootloader execution path and the public `validate_l2_tx_intrinsic_native_resources` API. [6](#0-5) 

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for the `native_per_pubdata` calculation to match the rounding direction used for `native_per_gas`:

```rust
// Before (floor division — undercharges pubdata):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. [7](#0-6) 

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Attacker submits an L2 EIP-1559 transaction calling a contract that performs many `SSTORE` operations (e.g., writing 20 distinct storage slots as in the existing test contract at `tests/instances/transactions/src/lib.rs`).
2. During validation, `native_per_pubdata = floor(pubdata_price / native_price)` is computed — one unit less than the correct ceiling value whenever `pubdata_price mod native_price ≠ 0`.
3. After execution, `get_resources_to_charge_for_pubdata` charges `pubdata_used × floor(pubdata_price / native_price)` native units instead of `pubdata_used × ceil(pubdata_price / native_price)`.
4. `compute_gas_refund` converts the underestimated native usage to a smaller `gas_used`.
5. The operator receives `gas_used × gas_price` tokens — systematically less than the pubdata price implies.

**Numerical verification** (using values from the existing test infrastructure):
- `pubdata_price = 7`, `native_price = 3`, `gas_price = 3`
- `native_per_pubdata` (actual) = `floor(7/3) = 2`; (correct) = `ceil(7/3) = 3`
- For 100 pubdata bytes: actual charge = 200 native = 200 gas; correct = 300 native = 300 gas
- Fee deficit = 100 gas × 3 = **300 tokens per 100 pubdata bytes** (33% deficit) [8](#0-7) [9](#0-8) [10](#0-9)

pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
